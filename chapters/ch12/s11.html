Emerging Patterns — MoE, SLMs, Multimodal & Beyond

Chapter 12: Large Language Models 8 subtopics
TL;DR

The LLM landscape is shifting fast. Mixture of Experts (MoE) gives you dense-model quality at a fraction of the inference cost by routing each token to only 2 of 8 expert networks. Small Language Models like Phi and Gemma prove that data quality beats raw parameter count. Multimodal models process images, audio, and text natively. Context windows have exploded from 2K to 1M+ tokens. Compound AI systems combine models, tools, and retrieval into pipelines that outperform any single model call. Synthetic data is training the next generation of models. Open-source is closing the gap on closed APIs. And edge AI is putting capable models on phones and laptops. This section surveys all of it — not as hype, but as patterns you’ll encounter in real production systems.

Mixture of Experts (MoE) — How Mixtral Works

Here’s a question that should bother you: if a 70B dense model uses all 70 billion parameters for every single token — including the word “the” — isn’t that absurdly wasteful? Does predicting a comma really need the same compute as reasoning about quantum mechanics?

Mixture of Experts says: no, it doesn’t. Not all parameters need to be active for every token. Instead, you split the feed-forward network (FFN) into multiple “expert” sub-networks and route each token to only the top-k experts. The rest stay dormant.

The Architecture

In a standard transformer, each layer has an attention block followed by a feed-forward block. In an MoE transformer, the attention block stays the same, but the single FFN is replaced by N expert FFNs plus a gating (router) network:

# Standard Transformer Layer:
#   x → Attention → FFN → output
#
# MoE Transformer Layer:
#   x → Attention → Router → [Expert₁, Expert₂, ..., Expert₈] → weighted sum → output
#
# The router decides which experts process each token

def moe_layer(x, experts, router, top_k=2):
    """
    x: (batch, seq_len, d_model) — hidden states
    experts: list of N FFN modules
    router: linear layer (d_model → N)
    """
    # Router produces logits over experts
    router_logits = router(x)                  # (batch, seq_len, num_experts)
    router_probs = softmax(router_logits, dim=-1)

    # Select top-k experts per token
    top_k_probs, top_k_indices = topk(router_probs, k=top_k)

    # Normalize the selected probabilities
    top_k_probs = top_k_probs / top_k_probs.sum(dim=-1, keepdim=True)

    # Run selected experts and combine
    output = zeros_like(x)
    for i in range(top_k):
        expert_idx = top_k_indices[:, :, i]     # which expert
        weight = top_k_probs[:, :, i:i+1]       # how much to weight it
        expert_out = experts[expert_idx](x)      # run that expert
        output += weight * expert_out

    return output

The key insight: the router is a learned linear layer. During training, it learns to specialize experts — one might handle code, another might handle reasoning, another might handle multilingual text. Nobody programs this. It emerges from gradient descent.

Mixtral 8x7B — The Numbers

Mixtral 8x7B became the poster child for MoE done right. Let’s look at what the name actually means:

Property Value Why It Matters
Total parameters 46.7B Full model size on disk/memory
Active parameters per token ~12.9B Actual compute cost per forward pass
Number of experts per layer 8 Each a ~7B-sized FFN
Top-k routing 2 Each token uses 2 of 8 experts
Shared parameters Attention layers All tokens share the same attention weights
Quality comparison ≈ LLaMA 2 70B Dense 70B quality at ~13B inference cost

Read that last row again. You get 70B-class quality while only running 13B parameters per token. That’s the MoE value proposition: dense quality at sparse cost.

The Load Balancing Problem

There’s a nasty failure mode with MoE: expert collapse. If the router starts sending most tokens to one or two experts, those experts get better (more training signal), which causes the router to send even more tokens to them, which starves the other experts. Positive feedback loop. You end up with 2 overloaded experts and 6 dead ones — effectively a much smaller model.

The fix is an auxiliary load-balancing loss added to the training objective:

# Auxiliary load balancing loss
# Goal: ensure each expert gets roughly equal traffic

def load_balance_loss(router_probs, top_k_indices, num_experts):
    """
    Encourages uniform expert utilization.
    Added to main loss: total_loss = LM_loss + α * balance_loss
    α is typically 0.01 - 0.1
    """
    # Fraction of tokens routed to each expert
    # (how much traffic each expert gets)
    tokens_per_expert = one_hot(top_k_indices, num_experts).mean(dim=[0, 1])

    # Average router probability assigned to each expert
    mean_prob_per_expert = router_probs.mean(dim=[0, 1])

    # Loss = N * Σ(fraction_i * mean_prob_i)
    # Minimized when both are uniform (1/N each)
    return num_experts * (tokens_per_expert * mean_prob_per_expert).sum()
Switch Transformer (Google, 2022) simplified MoE even further: top-1 routing instead of top-2. Each token goes to exactly one expert. Simpler routing, less communication overhead, and surprisingly competitive quality. The lesson: you don’t always need complexity. Sometimes top-1 is enough.

MoE Training and Inference Challenges

MoE isn’t free. The tradeoffs are real:

  • Memory: You still need all 47B parameters in memory (or on disk) even though only 13B are active. Memory savings only come if you can offload inactive experts.
  • Routing instability: Early in training, the router hasn’t learned anything useful, so expert assignment is nearly random. This can lead to training instability until the routing stabilizes.
  • Communication overhead: In distributed training, tokens routed to different experts may live on different GPUs. All-to-all communication to shuffle tokens to their assigned experts is expensive.
  • Inference advantage: At inference time, you only need to load the 2 active experts into fast GPU memory. With expert offloading, you can serve a 47B MoE model on hardware that couldn’t handle a dense 47B model.

Small Language Models — The SLM Revolution

For years the mantra was “bigger is better.” Then Microsoft released Phi-2 at 2.7 billion parameters and it matched models 10–25x its size on reasoning benchmarks. That wasn’t a fluke. It was a thesis: data quality matters more than model size.

The SLM revolution didn’t come from architecture breakthroughs. It came from being brutally selective about training data. Instead of crawling the entire internet and hoping the model figures it out, teams like Microsoft Research curated “textbook-quality” synthetic data — carefully generated examples designed to teach reasoning, not just pattern-match web text.

The Landscape

Phi-3 Mini (Microsoft) 3.8B Mixtral 8x7B, GPT-3.5 Runs on-device, strong reasoning
Phi-3 Medium (Microsoft) 14B GPT-4 (on some benchmarks) Competitive with frontier at fraction of cost
Gemma 2B (Google) 2B LLaMA 2 7B (on several tasks) Lightweight, open weights, strong base
Gemma 7B (Google) 7B LLaMA 2 13B Great fine-tuning foundation

SmolLM (HuggingFace) 135M–1.7B Varies by size Purpose-built for on-device

When SLMs Win

You don’t always need GPT-4. In fact, for many production use cases, a well-tuned small model is strictly better:

  • Edge deployment: A 3B model quantized to 4-bit fits in ~1.5GB of RAM. That runs on a phone.
  • Cost-sensitive applications: If you’re processing millions of documents, the difference between $0.01/1K tokens (API) and $0.0001/1K tokens (self-hosted SLM) is the difference between viable and bankrupt.
  • Privacy: Data never leaves the device. No API calls, no third-party data processing agreements, no compliance headaches.
  • Latency: Local inference eliminates network round-trips. Sub-10ms TTFT is achievable.
  • Specialized tasks: A 3B model fine-tuned on your domain can outperform a 70B general model on your specific task.
SLMs aren’t magic. They excel at focused tasks — classification, extraction, short-form generation, simple Q&A. For complex multi-step reasoning, long-form creative writing, or tasks requiring broad world knowledge, larger models still dominate. Don’t try to replace GPT-4 with a 2B model for agentic workflows. Match model size to task complexity.

Multimodal Foundation Models

Text-only LLMs were always an intermediate step. The real world is multimodal — we see images, hear audio, read text, watch video — and the most capable AI systems are now multimodal too. The shift happened fast: in early 2023, GPT-4 was text-only. By late 2023, GPT-4V processed images natively. By mid-2024, GPT-4o handled text, images, and audio in a single model.

The Architecture Patterns

There are two main approaches to building multimodal models:

Approach 1: Encoder + Projection + LLM (modular)

# The "bolted-on" multimodal architecture
# Used by: LLaVA, early GPT-4V, many open-source models

Image → Visual Encoder (ViT) → Image Embeddings (N patches × d_vision)
                                        ↓
                               Projection Layer (linear or MLP)
                                        ↓
                               Aligned Embeddings (N × d_model)
                                        ↓
            Text Tokens → Token Embeddings ──┐
                                              ├──→ LLM → Output Tokens
            Image Embeddings ────────────────┘

# The visual encoder (typically a ViT like CLIP ViT-L/14) is pretrained separately.
# The projection layer maps vision embeddings into the LLM's token space.
# The LLM "sees" image patches as if they were extra tokens in the sequence.

Approach 2: Natively multimodal (unified)

# The "born multimodal" architecture
# Used by: Gemini, GPT-4o

# All modalities are tokenized into a shared vocabulary from scratch.
# Images → visual tokens, audio → audio tokens, text → text tokens
# Single transformer processes all modality tokens together.
#
# ┌──────────────────────────────────────────────────┐
# │  [img_1] [img_2] ... [img_N] [text_1] [text_2]  │
# │         Single unified transformer               │
# │  All tokens attend to all other tokens            │
# └──────────────────────────────────────────────────┘
#
# Advantage: cross-modal attention from the start
# Disadvantage: extremely expensive to train from scratch

The Multimodal Landscape

Model Modalities Architecture Notable Capability
GPT-4V / GPT-4o Text, image, audio Natively multimodal (4o) Real-time voice conversation
Gemini 1.5 Pro Text, image, audio, video Natively multimodal 1M+ token context, video understanding
Claude 3 / 3.5 Text, image Encoder + LLM Strong document/chart understanding
LLaVA 1.6 Text, image CLIP ViT + projection + LLM Open-source, good for research

Why multimodal matters for practitioners: Real-world tasks are rarely text-only. Invoice processing needs to read scanned documents. Customer support needs to understand screenshots. Accessibility tools need to describe images. Medical AI needs to interpret scans alongside clinical notes. If your production system only handles text, you’re leaving value on the table.

Long-Context Windows

The maximum context length of LLMs has grown by three orders of magnitude in just a few years:

Context Window Evolution:
═══════════════════════════════════════════════════════════════
BERT (2018)         ████ 512 tokens
GPT-2 (2019)        ████████ 1,024 tokens
GPT-3 (2020)        ████████████████ 2,048 → 4,096 tokens
Claude 2 (2023)     ████████████████████████████████████████ 100K tokens
GPT-4 Turbo (2023)  █████████████████████████████████████████████ 128K tokens
Gemini 1.5 (2024)   ████████████████████████████████████████████████████████████ 1M+ tokens
                    └────┴────┴────┴────┴────┴────┴────┴────┘
                    0    128K  256K  384K  512K  640K  768K  1M

This isn’t just a numbers game. Going from 4K to 128K tokens means you can fit an entire codebase, a full legal contract, or a book-length document into a single prompt. That changes what’s architecturally possible.

How They Did It

Standard positional encodings (like the original sinusoidal or learned embeddings) were trained on fixed lengths. If you trained on 2K tokens, the model has never seen position 2,001 and has no idea what to do with it. RoPE (Rotary Position Embeddings) offered a path forward because its structure can be mathematically extended:

  • NTK-aware scaling: Instead of naively linearly interpolating positions (which degrades high-frequency features), NTK-aware scaling adjusts the base frequency of RoPE. This preserves local position information while extending the range. Think of it as changing the “resolution” of positional encoding rather than stretching it.
  • YaRN (Yet another RoPE extensioN): Combines NTK-aware scaling with attention scaling and a temperature factor. Currently the most effective RoPE extension method.
  • Ring Attention: For truly massive contexts, distribute the sequence across multiple GPUs. Each GPU holds a chunk of the KV cache and passes attention computations in a ring topology. This scales to millions of tokens by distributing the memory burden.

Long Context vs RAG — When Each Wins

This is one of the most common architectural decisions you’ll face. Both long context and RAG solve the same fundamental problem — giving the model access to more information than fits in a single prompt — but they have very different characteristics:

Dimension Long Context RAG
Corpus size Up to ~1M tokens (~750K words) Unlimited (millions of documents)
Latency Higher (process everything upfront) Lower (retrieve only what’s needed)
Cost per query Expensive (pay for all input tokens) Cheaper (only retrieved chunks)
Cross-document reasoning Excellent (sees everything at once) Poor (only sees retrieved chunks)
Setup complexity Simple (just stuff it in the prompt) Complex (embeddings, vector DB, chunking)
Data freshness Whatever’s in the prompt Update index anytime
Best for Single-document analysis, codebases Large knowledge bases, FAQ, enterprise search

The “Lost in the Middle” Problem

Here’s something that should worry you if you’re building long-context applications: models don’t attend equally to all positions. Research from Stanford and others showed that LLMs reliably find information placed at the beginning or end of the context window, but performance degrades significantly for information buried in the middle.

# "Lost in the Middle" effect (Liu et al., 2023)
#
# Task: find a specific fact placed at different positions in a long context
#
# Accuracy by position:
#
# 90% ██████████████████░░░░░░░░░░░░░░░░░░░░░░░██████████████████
# 70% ░░░░░░░░░░░░░░░░██████████████████████████░░░░░░░░░░░░░░░░░

#      ▲               ▲                          ▲
#      Beginning       Middle (worst)              End
#
# Practical implication: put important information at the

Don’t assume “longer context = better.” More context means more noise, higher cost, and the lost-in-the-middle problem. If your task only needs 3 relevant paragraphs, retrieving just those 3 paragraphs (RAG) will often outperform stuffing 100K tokens of context. Long context is a capability, not always the right strategy.

Compound AI Systems — Beyond Single Model Calls

Here’s an uncomfortable truth about modern AI engineering: the era of “call one model, return the result” is over. The best-performing production AI systems are not a single model — they’re systems that orchestrate multiple models, tools, retrieval pipelines, and verification steps.

The Berkeley AI Research (BAIR) blog coined the term Compound AI Systems to describe this shift: moving from individual model improvements to system-level design.

Common Compound Patterns

# Pattern 1: RAG (Retrieval-Augmented Generation)
User Query → Retriever → Top-K Documents → LLM → Answer
# The retriever and LLM are separate components with different optimization targets

# Pattern 2: Agent Loops
User Task → LLM (plan) → Tool Call → Observation → LLM (reason) → Tool Call → ... → Answer
# The LLM calls tools iteratively until the task is complete

# Pattern 3: Model Cascade
User Query → Small/Cheap Model → [confident?] → Return Answer
                                → [uncertain?] → Large/Expensive Model → Return Answer
# Route easy queries to cheap models, hard ones to expensive models

# Pattern 4: Judge-Generator
User Query → Generator LLM → Candidate Answer → Judge LLM → [good?] → Return
                                                            → [bad?]  → Regenerate
# One model generates, another verifies — catches hallucinations

# Pattern 5: Multi-Agent Collaboration
Task → Planner Agent → [subtask₁] → Coder Agent → Code
                     → [subtask₂] → Researcher Agent → Facts
                     → [subtask₃] → Reviewer Agent → Feedback
       → Planner Agent (synthesize) → Final Output

DSPy — Programmatic Prompt Optimization

DSPy (Stanford) is one of the most interesting frameworks in this space. Instead of hand-writing prompts, you define your pipeline as a program with typed signatures, and DSPy compiles the prompts for you by optimizing them against a metric on your data:

# DSPy: prompts as compiled programs, not hand-written strings

import dspy

# Define signatures (what the module does, not how)
class GenerateAnswer(dspy.Signature):
    """Answer questions using provided context."""
    context = dspy.InputField(desc="relevant passages")
    question = dspy.InputField()
    answer = dspy.OutputField(desc="concise factual answer")

# Build a RAG pipeline
class RAG(dspy.Module):
    def __init__(self, num_passages=3):
        self.retrieve = dspy.Retrieve(k=num_passages)
        self.generate = dspy.ChainOfThought(GenerateAnswer)

    def forward(self, question):
        context = self.retrieve(question).passages
        answer = self.generate(context=context, question=question)
        return answer

# Compile — DSPy optimizes the prompts automatically
from dspy.teleprompt import BootstrapFewShot
optimizer = BootstrapFewShot(metric=my_accuracy_metric)
compiled_rag = optimizer.compile(RAG(), trainset=my_examples)

# The compiled program has optimized few-shot examples
# baked into the prompts — no manual prompt engineering

The Reliability–Autonomy Tradeoff

Every compound system sits somewhere on a spectrum:

More Reliable                                        More Autonomous
◄──────────────────────────────────────────────────────────────────►
│                    │                    │                         │
Hardcoded           RAG with            Agent with              Fully autonomous
pipeline            fixed retrieval     tool access             multi-agent system
│                    │                    │                         │
99.9% predictable   95% reliable         80-90% reliable          60-80% reliable
No flexibility      Some flexibility     High flexibility          Maximum flexibility
The practical rule: Start with the simplest compound system that works (often just RAG). Add autonomy only when you have clear metrics showing it helps. Every additional model call, tool, or agent loop is another failure point. The goal is the minimum viable compound system, not the most impressive architecture diagram.

Synthetic Data — Training AI with AI-Generated Data

We’re running out of internet. Not literally, but practically: high-quality text data for training LLMs is a finite resource, and multiple estimates suggest we’ll exhaust the supply of unique, high-quality web text by 2026–2028. The solution the industry has converged on? Use AI to generate training data for AI.

That sounds like it should be circular and broken. Sometimes it is. But done carefully, synthetic data is becoming one of the most important techniques in modern ML.

Approaches

1. Distillation: Use a large, expensive model (GPT-4, Claude) to generate training examples for a smaller model. This is the most common approach and it works remarkably well — you’re essentially compressing the large model’s capability into a smaller one through its outputs.

2. Self-play / self-improvement: The model generates candidate solutions, verifies them (via tests, proofs, or another model), and trains on the verified correct ones. This is how models like AlphaCode and DeepSeek improve at code and math.

3. Persona-driven generation: Generate diverse training data by prompting the teacher model with different personas, writing styles, difficulty levels, and domains. This prevents the synthetic data from being monotonously similar.

# Synthetic data pipeline for instruction tuning
# (simplified version of what Phi, Orca, etc. use)

personas = [
    "You are a patient math tutor explaining to a high school student",
    "You are a concise technical writer for an API reference",
    "You are a curious scientist thinking through a problem step-by-step",
    "You are a senior engineer doing a code review",
]

topics = load_topic_taxonomy()  # thousands of topics

for persona in personas:
    for topic in topics:
        for difficulty in ["beginner", "intermediate", "advanced"]:
            prompt = f"""
            {persona}.
            Generate a {difficulty}-level question about {topic},
            then provide a detailed, correct answer.
            Show your reasoning step by step.
            """
            response = teacher_model.generate(prompt)  # GPT-4, Claude, etc.

            # Quality filtering — critical step
            if passes_quality_check(response):
                training_data.append({
                    "instruction": extract_question(response),
                    "response": extract_answer(response)
                })

When Synthetic Data Works (and Doesn’t)

Works Well Why
Math problems Answers are verifiable — you can check if 2+2 actually equals 4
Code generation Code can be executed and tested — verifiable correctness
Instruction following Format compliance is checkable, and diversity is the main need
Data augmentation Paraphrasing, translation, reformatting existing data
Risky / Doesn’t Work Why
Training on own outputs (recursive) Model collapse — errors compound, diversity shrinks, quality degrades over generations
Factual knowledge LLMs hallucinate, so synthetic “facts” may be wrong. No easy verification.
Replacing real distribution Synthetic data has distributional biases from the generator. Real data captures edge cases synthetic data misses.
Model collapse is real. Shumailov et al. (2023) showed that recursively training models on their own outputs leads to progressive quality degradation. The tails of the distribution get cut off — rare but important patterns disappear. After a few generations, the model converges to a narrow, repetitive output distribution. Always mix synthetic data with real data, and always filter for quality.

Open vs Closed Models — The Ecosystem Divide

This isn’t a technical distinction — it’s a strategic one that affects everything from cost structure to data privacy to long-term vendor risk. And the landscape is shifting fast.

The Current State

Dimension Open Models Closed Models
Examples LLaMA 3, Mistral, Qwen 2, Gemma, Command-R GPT-4, Claude 3.5, Gemini Ultra
Quality (frontier) Close but behind on hardest tasks Best available quality
Fine-tuning Full control: LoRA, full FT, RLHF Limited (API fine-tuning only, if available)
Self-hosting Complete control over infrastructure API only — vendor controls everything
Data privacy Data never leaves your servers Data sent to third-party API
Cost at scale Lower (after infra investment) Higher (per-token pricing adds up)
Time to production Weeks (need serving infra) Hours (API key and go)
Vendor lock-in None Significant — prompt engineering is provider-specific

Decision Framework

# When to use closed (API) models:
# ─────────────────────────────────────────────
# ✓ Prototyping and MVP — fastest path to working product
# ✓ Best-quality-at-any-cost tasks — complex reasoning, creative
# ✓ Small-to-medium scale — under ~1M tokens/day
# ✓ No ML infrastructure team
# ✓ Multimodal (vision, audio) — open models lag here

# When to use open models:
# ─────────────────────────────────────────────
# ✓ High volume — >10M tokens/day (cost crossover point)
# ✓ Data privacy requirements — healthcare, finance, legal
# ✓ Custom fine-tuning needed — domain-specific behavior
# ✓ Low latency / offline — edge deployment, air-gapped
# ✓ Long-term stability — can't afford API deprecation risk
# ✓ Regulatory compliance — need to explain and audit the model

# The hybrid approach (increasingly common):
# ─────────────────────────────────────────────
# Use closed models for complex tasks (planning, hard reasoning)
# Use open models for high-volume simple tasks (classification, extraction)
# Route between them based on query complexity
The gap is closing fast. LLaMA 3 70B and Qwen 2 72B are competitive with GPT-3.5 Turbo and approaching early GPT-4 on many benchmarks. Mistral Large competes with Claude 3 Sonnet. The open-source community moves at an incredible pace — what was frontier-only 6 months ago often has an open alternative today. Plan your architecture to swap between open and closed models without rewriting your entire pipeline.

Edge AI & On-Device Models

Running a language model on a phone sounded absurd in 2022. By 2024, Apple had on-device models powering Siri features, Google had Gemini Nano on Pixel phones, and hobbyists were running Mistral 7B on M1 MacBooks at 30+ tokens per second. The stack that made this possible is the convergence of three trends: smaller models, better quantization, and dedicated hardware.

The On-Device Stack

# What makes on-device LLMs possible:
#
# ┌──────────────────────────────────────────────────────┐
# │  1. Small Models (1B–7B parameters)                  │
# │     Phi-3 Mini, Gemma 2B, Gemini Nano               │
# │     Quality that was 70B-only in 2023                │
# ├──────────────────────────────────────────────────────┤
# │  2. Aggressive Quantization (4-bit, GGUF)            │

Why Edge AI Matters

  • Privacy: Data never leaves the device. No API calls, no data processing agreements, no risk of data breaches through third parties. For healthcare, finance, and personal AI assistants, this is often a hard requirement.
  • Latency: Zero network round-trip. On-device inference can achieve sub-10ms time-to-first-token for small models — faster than the speed of light to the nearest data center.
  • Offline operation: Works on airplanes, in tunnels, in areas with poor connectivity. Critical for industrial IoT, military, and field applications.
  • Cost: After the hardware is purchased, inference is free. No per-token API charges. For high-volume personal-device use cases, the economics are unbeatable.

The Practical Reality

On-device models are real, but let’s be honest about the constraints:

Desktop + RTX 4090 (24GB) 24 GB VRAM 13B–30B parameters 40–80 tok/s
On-device ≠ on-par. A 3B model running on your phone is impressive engineering, but it’s not replacing GPT-4 for complex reasoning. On-device models excel at specific, well-scoped tasks: autocomplete, summarization of short texts, simple Q&A, voice commands. Design your system to use on-device for what it’s good at and route harder tasks to the cloud. The hybrid approach — on-device for speed and privacy, cloud for capability — is where the industry is heading.

The Big Picture — Where It’s All Heading

Step back and look at these eight trends together. They’re not independent — they’re converging:

# The convergence
#
# MoE + SLMs            → Efficient models that punch above their weight
# SLMs + Edge AI        → Capable models on every device
# Multimodal + Agents   → Systems that see, hear, and act
# Long Context + RAG    → Complementary retrieval strategies
# Synthetic Data + Open → Open models trained on synthetic data close gap with closed
# Compound Systems      → The architecture pattern that ties it all together
#
# The trajectory:
# 2022: "One big model behind an API"
# 2023: "Fine-tuned models + RAG"
# 2024: "Compound systems with specialized models"
# 2025+: "Autonomous AI systems with on-device + cloud hybrid"

The most important takeaway: the winning strategy is not picking one model — it’s building systems. A compound AI system that routes between a tiny on-device model, a self-hosted 7B, and a frontier API — with RAG, caching, and verification — will outperform any single model call at any scale. The best engineers in this space aren’t prompt whisperers. They’re system designers.
