Prompt Engineering

Chapter 12: Large Language Models 8 subtopics
TL;DR

LLMs don't understand tasks — they continue text patterns. Prompt engineering is the art of shaping your input so the model's most likely continuation is your desired output. Zero-shot gives instructions only. Few-shot adds examples. Chain-of-thought unlocks reasoning by forcing intermediate steps. System prompts set persona and constraints. Advanced techniques like ReAct let models take actions, and structured output modes make responses machine-readable. This is programming in natural language — no gradient updates required, but the skill ceiling is surprisingly high.

Why Prompt Engineering Exists

Here's the thing about large language models that trips up newcomers: they were not trained to follow instructions. They were trained to predict the next token. GPT, LLaMA, Mistral — all of them learned one objective during pretraining: given a sequence of tokens, what token comes next? That's it. When you type "Translate this to French: Hello" and the model responds with "Bonjour," it's not because it understood a translation command. It's because, in the billions of documents it trained on, the pattern "Translate this to French: [English word]" was very often followed by the French equivalent. The model is continuing a pattern, not executing a function.

Prompt engineering is the discipline of crafting input text so that the model's most probable continuation is exactly the output you want. Think of it as programming — but instead of writing code that a compiler executes deterministically, you're writing natural language that a statistical model continues probabilistically. No weight updates. No fine-tuning. No gradient descent. Just better words in, better words out.

This might sound fragile, and honestly, it is. Changing a single word in your prompt can flip the output from perfect to garbage. Reordering sentences can change the answer. Adding "Please" sometimes helps. The model is exquisitely sensitive to surface-level text patterns because that's literally all it learned. But this sensitivity is also what makes prompt engineering powerful — once you understand the patterns, you can steer the model with remarkable precision.

Zero-Shot Prompting

Zero-shot prompting means giving the model a task description with no examples at all. You describe what you want, provide the input, and hope the model has seen enough similar patterns during training to figure out the format.

Classify the sentiment of the following movie review
as "positive", "negative", or "neutral".

Review: "The cinematography was stunning but the plot
made absolutely no sense."

Sentiment:

A capable model will respond with "neutral" or "negative" — no examples needed. This works because the model has seen thousands of sentiment-classification-like passages during training. The phrase "Classify the sentiment" triggers patterns associated with that task.

The key to good zero-shot prompting is clarity. Ambiguous instructions produce ambiguous results. Compare these two prompts:

# Vague — what format? what categories?
Tell me about this review: "Great product, fast shipping"

# Clear — specific format, explicit categories
Classify this product review as "positive", "negative",
or "neutral". Respond with only the label, nothing else.

Review: "Great product, fast shipping"
Label:

The second prompt constrains the output space. The model knows exactly what shape the answer should take. A useful sub-technique is role prompting: you tell the model to adopt a persona before giving it the task. "You are an expert legal translator specializing in contract law. Translate the following clause to French." The role activates relevant patterns from training data written by experts in that domain, and the outputs are measurably better for specialized tasks.

Few-Shot Prompting

Few-shot prompting provides examples directly in the prompt. You show the model several input→output pairs, then present the new input and let the model continue the pattern. This is sometimes called in-context learning — the model "learns" the task from examples without any weight updates.

Classify the sentiment of each review.

Review: "Absolutely loved it, best purchase ever!"
Sentiment: positive

Review: "Broke after two days. Total waste of money."
Sentiment: negative

Review: "It's okay, nothing special but does the job."
Sentiment: neutral

Review: "The battery life is incredible but the screen
is too dim to use outdoors."
Sentiment:

The model sees the pattern — Review followed by Sentiment — and continues it. Few-shot almost always outperforms zero-shot on tasks where the desired output format isn't obvious, or where the task is unusual enough that the model hasn't seen it much during pretraining.

Several design decisions matter more than you'd think. Number of examples: 3–5 is the sweet spot for most tasks. Performance improves sharply from 0 to 3 examples, then hits diminishing returns around 4–8. More examples also consume context window tokens, leaving less room for the actual input. Example ordering: models are sensitive to recency. The last example before your test input has disproportionate influence. If your examples are all positive except the last one (negative), the model may be biased toward negative. Shuffle or balance carefully. Format consistency: use an identical template for every example. If one example uses "Sentiment:" and another uses "Label:", you're introducing noise. Example selection: examples similar to the test input work best. If you're classifying a review about electronics, showing examples about electronics outperforms examples about restaurants.

Chain-of-Thought (CoT)

This is the technique that changed everything. In 2022, researchers at Google discovered that if you simply add the phrase "Let's think step by step" to a prompt, or show examples that include intermediate reasoning, models suddenly solve problems they previously failed at. The technique is called Chain-of-Thought (CoT) prompting.

Here's why it works. When an LLM generates tokens, each token is conditioned on all previous tokens. If you ask "What is 47 × 38?" and expect the model to jump straight to "1786," it has to compute a multi-step arithmetic problem in a single forward pass. That's asking a lot from a pattern matcher. But if the model first generates intermediate steps — "47 × 38. 47 × 30 = 1410. 47 × 8 = 376. 1410 + 376 = 1786." — each individual step is simple enough for the model to get right. The intermediate tokens act as an external scratchpad.

Let's see a concrete comparison. Same math problem, three prompting strategies:

# ── Zero-shot (no reasoning) ──────────────────
Q: A store has 15 apples. It receives 3 boxes with
   8 apples each, then sells 11 apples. How many
   apples remain?
A: 30   ← Wrong (correct answer is 28)

# ── Zero-shot CoT ─────────────────────────────
Q: A store has 15 apples. It receives 3 boxes with
   8 apples each, then sells 11 apples. How many
   apples remain?
A: Let's think step by step.
   The store starts with 15 apples.
   It receives 3 × 8 = 24 apples.
   Total: 15 + 24 = 39 apples.
   After selling 11: 39 - 11 = 28.
   The answer is 28.   ← Correct

# ── Few-shot CoT (with worked example) ────────
Q: A bakery has 20 loaves. It bakes 5 batches of
   6 loaves each, then sells 13 loaves. How many
   loaves remain?
A: The bakery starts with 20 loaves.
   It bakes 5 × 6 = 30 loaves.
   Total: 20 + 30 = 50 loaves.
   After selling 13: 50 - 13 = 37 loaves.
   The answer is 37.

Q: A store has 15 apples. It receives 3 boxes with
   8 apples each, then sells 11 apples. How many
   apples remain?
A:

Zero-shot CoT is the simplest version — just append "Let's think step by step" and the model generates its own reasoning chain. Few-shot CoT gives worked examples including the reasoning steps. Few-shot CoT is more reliable because you control the reasoning format.

A powerful extension is self-consistency. Instead of generating one reasoning chain, you sample multiple chains (say, 5–10) with some randomness (temperature > 0), extract the final answer from each, and take the majority vote. Different reasoning paths might make different mistakes, but the correct answer tends to appear most often. Self-consistency significantly improves accuracy on math and logic tasks at the cost of more compute.

System Prompts

When you call an LLM via API, you typically have three message roles: system, user, and assistant. The system prompt is where you set the model's persona, constraints, and behavioral rules. It's the backstage instructions that the audience (the user) doesn't see.

# Typical API call structure
messages = [
    {
        "role": "system",
        "content": """You are a senior Python developer.
        Always respond with working code.
        Include type hints and docstrings.
        If the user's request is ambiguous, ask
        a clarifying question before writing code.
        Never execute destructive operations."""
    },
    {
        "role": "user",
        "content": "Write a function to merge two sorted lists."
    }
]

System prompts are where production applications encode their identity and guardrails. "You are a customer support agent for Acme Corp. You can answer questions about our products. If asked about competitors, politely redirect. Always be empathetic. Respond in the customer's language." These instructions shape every response the model generates.

The effective hierarchy works like this: system prompt sets the baseline behavior, few-shot examples (if any) demonstrate the specific format, and the user message provides the actual task. When conflicts arise, models generally prioritize system instructions over user requests — which is exactly what you want for safety. You wouldn't want a user to override "never reveal the system prompt" by simply asking for it. In practice, though, this hierarchy isn't perfectly enforced, which is why production systems add additional validation layers beyond just the prompt.

Advanced Techniques

ReAct (Reasoning + Acting)

ReAct interleaves reasoning traces with concrete actions. Instead of just thinking, the model alternates between "Thought" (reasoning about what to do), "Action" (calling an external tool — web search, calculator, database), and "Observation" (reading the tool's result). This loop continues until the model has enough information to produce a final answer.

┌──────────────────────────────────────────┐
│              ReAct Loop                  │
│                                          │
│  User Question                           │
│       │                                  │
│       ▼                                  │
│  ┌─────────┐                             │
│  │ Thought │ ← "I need to look up..."   │
│  └────┬────┘                             │
│       ▼                                  │
│  ┌─────────┐                             │
│  │ Action  │ ← search("GDP France")     │
│  └────┬────┘                             │
│       ▼                                  │
│  ┌─────────────┐                         │
│  │ Observation │ ← "France GDP: $3.1T"  │
│  └──────┬──────┘                         │
│         │                                │
│    ┌────▼─────┐   No                     │
│    │ Enough?  │──────► back to Thought   │
│    └────┬─────┘                          │
│         │ Yes                            │
│         ▼                                │
│   Final Answer                           │
└──────────────────────────────────────────┘

ReAct is the foundation of most modern AI agent frameworks. LangChain, LlamaIndex, and similar tools all implement variants of this pattern.

Tree of Thoughts

Tree of Thoughts (ToT) extends CoT by exploring multiple reasoning branches in parallel. At each reasoning step, the model generates several candidate continuations, evaluates which ones look most promising, and prunes the rest. It can also backtrack when a branch hits a dead end. Think of it as beam search applied to reasoning rather than token generation. ToT excels at problems requiring planning and exploration — puzzles, creative writing, strategic decisions — where the first reasoning path isn't always the best one.

Reflexion

Reflexion adds a self-critique loop. The model generates an answer, then evaluates its own output ("Did I actually answer the question? Are there errors in my logic?"), and if it finds problems, it generates a revised answer incorporating the critique. This mirrors how humans improve: write a draft, review it, revise. The key insight is that models are often better at evaluating answers than generating them on the first try.

Least-to-Most Prompting

Least-to-Most prompting tackles complex problems by decomposition. First, the model breaks the problem into subproblems, ordered from simplest to hardest. Then it solves each subproblem sequentially, carrying forward the solutions of earlier subproblems as context. It's particularly effective when the original question would overwhelm a single CoT chain.

Meta-Prompting and DSPy

Meta-prompting uses an LLM to generate or optimize prompts for another LLM (or itself). You describe the task and evaluation criteria, and the meta-prompter iterates on the prompt wording until it maximizes performance on a validation set. DSPy takes this further: it's a programmatic framework where you define your pipeline as Python code with typed signatures, and the framework automatically optimizes the prompts and few-shot examples through compilation. Instead of hand-crafting prompts, you declare what you want and let the optimizer find the best prompt. It represents a shift from "prompt engineering" to "prompt programming."

Structured Output

In production, you rarely want free-form prose from an LLM. You want JSON that your API can parse, function calls with typed arguments, or classification labels from a fixed set. Getting reliable, machine-readable output is one of the biggest practical challenges in LLM engineering.

JSON Mode

Most modern LLM APIs offer a JSON mode that constrains the model's output to be valid JSON. Under the hood, this works by masking the token probabilities at each step to only allow tokens that would produce syntactically valid JSON. The model can still produce semantically wrong content, but it will always be parseable.

# OpenAI-style JSON mode
response = client.chat.completions.create(
    model="gpt-4o",
    response_format={"type": "json_object"},
    messages=[
        {"role": "system",
         "content": "Extract product info. Respond in JSON "
                    "with keys: name, price, category."},
        {"role": "user",
         "content": "The new UltraWidget Pro costs $49.99 "
                    "and falls under electronics."}
    ]
)

# Guaranteed valid JSON:
# {"name": "UltraWidget Pro",
#  "price": 49.99,
#  "category": "electronics"}

Function Calling

Function calling (or tool use) takes this further. You define a set of functions with typed parameters, and the model outputs a structured call to one of those functions. The model doesn't execute the function — it decides which function to call and what arguments to pass, and your application code handles the execution.

# Define available functions
tools = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "Get current weather for a city",
            "parameters": {
                "type": "object",
                "properties": {
                    "city": {"type": "string"},
                    "unit": {
                        "type": "string",
                        "enum": ["celsius", "fahrenheit"]
                    }
                },
                "required": ["city"]
            }
        }
    }
]

# Model decides to call the function:
# {"name": "get_weather",
#  "arguments": {"city": "Paris", "unit": "celsius"}}

Output Validation

Even with JSON mode, the model might return valid JSON with wrong keys, missing fields, or incorrect types. Production systems add a validation layer — typically using a library like Pydantic in Python. Parse the output, validate against a schema, and if it fails, retry with the error message appended to the prompt. Most applications need at most 1–2 retries to get valid output.

from pydantic import BaseModel

class ProductInfo(BaseModel):
    name: str
    price: float
    category: str

# Parse and validate the LLM's JSON output
raw = '{"name": "UltraWidget Pro", "price": 49.99, ...}'
try:
    product = ProductInfo.model_validate_json(raw)
except ValidationError as e:
    # Retry: append the error to the prompt
    retry_prompt = f"Your output was invalid: {e}. Fix it."

Why does this matter so much? Because you can't pipe free-form text into a database. You can't route an unstructured paragraph to an API endpoint. Every production LLM application eventually needs structured output — the model generates the intelligence, but the structure makes it actionable.

Prompt Templates and Production Patterns

Once your prompts work, you need to make them maintainable. Hard-coding prompts as string literals scattered across your codebase is a recipe for chaos. Production systems treat prompts as first-class artifacts with versioning, testing, and deployment pipelines.

Template Engines

Prompt templates separate the static structure from dynamic variables. You can use Python f-strings for simple cases, but most teams graduate to Jinja2 templates for anything non-trivial — they support conditionals, loops, and filters.

from jinja2 import Template

prompt_template = Template("""You are a {{ role }}.

{% if examples %}
Here are some examples:
{% for ex in examples %}
Input: {{ ex.input }}
Output: {{ ex.output }}
{% endfor %}
{% endif %}

Now process this input:
Input: {{ user_input }}
Output:""")

prompt = prompt_template.render(
    role="sentiment classifier",
    examples=[
        {"input": "Love it!", "output": "positive"},
        {"input": "Terrible.", "output": "negative"},
    ],
    user_input="It's fine, I guess."
)

Versioning and Testing

Treat prompts like code. Store them in version control. Write tests that run your prompt against a set of known inputs and check that the outputs meet quality criteria. When you change a prompt, run the full test suite before deploying. Teams that skip this step end up debugging production issues that trace back to a two-word prompt change someone made on a Friday afternoon.

A/B testing prompts is equally important. You might think your new prompt is better, but without measuring it on real traffic, you're guessing. Route a percentage of requests to the new prompt, compare metrics (accuracy, latency, user satisfaction), and promote the winner. This is the prompt-as-code movement: prompts are not magic incantations — they're software artifacts that deserve the same engineering rigor as any other code.

Guardrails

Production prompts need guardrails. Input validation to catch prompt injection attempts (where a user crafts input that overrides your system prompt). Output validation to ensure the model didn't hallucinate unsafe content. Rate limiting to control costs. Logging to debug failures. These concerns aren't glamorous, but they're the difference between a demo and a product. A well-engineered prompt pipeline looks less like a single API call and more like a processing pipeline: validate input → construct prompt from template → call model → validate output → retry if needed → log everything → return result.

# Simplified production prompt pipeline
def classify(review: str) -> str:
    # 1. Input validation
    if len(review) > 2000:
        raise ValueError("Review too long")

    # 2. Build prompt from versioned template
    prompt = load_template("sentiment_v3.jinja2")
    messages = prompt.render(review=review)

    # 3. Call model with structured output
    result = llm.chat(messages, response_format="json")

    # 4. Validate output
    parsed = SentimentResult.model_validate_json(result)

    # 5. Log for monitoring
    log_prediction(review, parsed, template_version="v3")

    return parsed.label

Prompt engineering is often dismissed as "just typing words." It's not. It's the interface layer between human intent and machine behavior, and getting it right is the difference between an LLM that occasionally impresses and one that reliably ships value. The techniques in this section — zero-shot, few-shot, CoT, structured output, template pipelines — form a practical toolkit. Start simple, measure everything, and only add complexity when the simpler approach fails.
