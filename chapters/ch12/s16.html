Parameter-Efficient Fine-Tuning

Chapter 12: Large Language Models 10 subtopics
TL;DR

Full fine-tuning of a large language model means updating billions of parameters — requiring enormous GPU memory for weights, gradients, and optimizer states. Parameter-efficient fine-tuning (PEFT) methods update only 0.1–1% of parameters while achieving 90–99% of full fine-tuning quality. LoRA decomposes weight updates into low-rank matrices and is the default choice. QLoRA adds 4-bit quantization so you can fine-tune a 65B model on a single 48GB GPU. Other methods — adapters, prefix tuning, prompt tuning, IA3 — trade off simplicity, parameter count, and inference overhead differently. Data quality matters more than data quantity.

Why Parameter-Efficient Fine-Tuning

Let's start with the problem. You have a 70-billion-parameter model. You want to teach it to do something specific — answer medical questions, write legal contracts, generate SQL. The straightforward approach is to fine-tune every parameter. But a 70B model in FP16 takes about 140 GB just to store the weights. The Adam optimizer keeps two additional copies of every parameter (momentum and variance), tripling the memory footprint. Add gradients and activations, and you need 400+ GB of GPU memory. That's multiple A100s. For most teams, that's either impossibly expensive or simply unavailable.

Parameter-Efficient Fine-Tuning (PEFT) is the family of techniques that sidestep this cost. The core idea: don't update all the parameters. Update a tiny fraction — sometimes as few as 0.1% — while keeping the rest of the model frozen. The frozen parameters still contribute their knowledge. The small trainable piece learns the new task. You get most of the benefit of full fine-tuning at a fraction of the cost. And you get a bonus: you can keep one base model in memory and swap lightweight adapters for different tasks, like changing hats on the same person.

Full Fine-Tuning Baseline

Before we talk about being efficient, let's understand what we're being efficient about. Full fine-tuning means unfreezing every parameter in a pretrained model and continuing gradient descent on your task-specific data. The most common form for LLMs is supervised fine-tuning (SFT), sometimes called instruction tuning — you train the model on (instruction, response) pairs so it learns to follow directions rather than just predict the next token.

Full fine-tuning works. When you have enough compute, it gives the best performance because the model can reshape its entire weight space for your task. But it has real downsides beyond cost. The biggest is catastrophic forgetting. Fine-tune a general-purpose model on medical data, and it might lose its ability to write code. The parameters that encoded coding knowledge get overwritten by medical patterns. You end up with a specialist that forgot how to be a generalist. You could mix in general data to mitigate this, but that adds complexity and cost.

The memory cost is the main motivator for PEFT. For a 7B model in FP16, full fine-tuning needs roughly 7B × 2 bytes (weights) + 7B × 2 bytes (gradients) + 7B × 8 bytes (Adam states) = about 84 GB. Scale to 70B and you're at 840 GB. PEFT methods cut the trainable parameters by 100–1000×, which slashes the gradient and optimizer memory proportionally.

LoRA (Low-Rank Adaptation)

LoRA is the dominant PEFT method, and for good reason. The insight behind it is elegant. When you fine-tune a pretrained model, the weight updates tend to live in a low-dimensional subspace. You don't need a full-rank update matrix. Most of the "directions" in weight space don't change meaningfully during fine-tuning.

Here's the math. Suppose you have a weight matrix W of shape d × d. In full fine-tuning, you'd learn an update ΔW of the same shape — that's d² trainable parameters. LoRA instead decomposes the update as ΔW = B × A, where B is d × r and A is r × d. The rank r is tiny — typically 8 to 64. This reduces trainable parameters from d² to 2 × d × r. For d = 4096 and r = 16, that's a reduction from 16.7 million to 131 thousand parameters per matrix — a 128× saving.

A is initialized with a small random Gaussian, and B is initialized to zero, so the adapter starts as an identity (no change to the pretrained weights). During training, only A and B are updated. The frozen W never needs gradients or optimizer states. At inference time, you compute W' = W + B × A and merge the adapter directly into the weights. The merged model has exactly the same architecture and latency as the original — zero overhead.

Which layers should you apply LoRA to? The standard choice is the attention projections: Q, K, V, and the output projection O. Some practitioners also apply it to the MLP layers for more capacity. The rank r controls the tradeoff: higher rank means more expressive adapters but more parameters. For most tasks, r = 16 or r = 32 is a good starting point. Research shows diminishing returns beyond r = 64.

# LoRA: conceptual implementation
import torch
import torch.nn as nn

class LoRALinear(nn.Module):
    """Wraps a frozen linear layer with a low-rank adapter."""
    def __init__(self, original: nn.Linear, r: int = 16, alpha: int = 16):
        super().__init__()
        d_in, d_out = original.in_features, original.out_features
        self.original = original
        self.original.weight.requires_grad_(False)  # freeze base weights

        # Low-rank decomposition: ΔW = B @ A
        self.A = nn.Parameter(torch.randn(r, d_in) * 0.01)   # r × d_in
        self.B = nn.Parameter(torch.zeros(d_out, r))          # d_out × r
        self.scaling = alpha / r  # controls update magnitude

    def forward(self, x):
        base_out = self.original(x)              # frozen forward pass
        lora_out = (x @ self.A.T @ self.B.T) * self.scaling  # low-rank path
        return base_out + lora_out

    def merge(self):
        """Merge adapter into base weights for zero-overhead inference."""
        self.original.weight.data += (self.B @ self.A * self.scaling)

The alpha parameter is a scaling factor that controls how much the adapter contributes relative to the base weights. A common practice is to set alpha = r, which gives a scaling of 1.0. If you increase r but keep alpha fixed, each adapter "direction" contributes less, which can stabilize training.

QLoRA

QLoRA was the practical breakthrough that democratized LLM fine-tuning. The idea: quantize the base model to 4-bit precision, then attach LoRA adapters in higher precision (BF16). You train only the adapters. The base model sits in 4-bit memory, slashing the footprint dramatically.

Three innovations make QLoRA work. First, NormalFloat4 (NF4) — a 4-bit data type designed specifically for neural network weights, which tend to be normally distributed. NF4 places its quantization levels to match this distribution, minimizing information loss compared to uniform 4-bit quantization. Second, double quantization — the quantization constants themselves (one per block of 64 weights) are quantized from FP32 to 8-bit, saving an additional ~0.4 bits per parameter. Third, paged optimizers — when GPU memory spikes during training (from gradient checkpointing or long sequences), optimizer states are automatically offloaded to CPU RAM and paged back in, like virtual memory for GPU.

The numbers are compelling. A 65B model in FP16 needs ~130 GB. In NF4 with double quantization, it fits in ~33 GB. Add the LoRA adapters and optimizer states for those adapters, and total training memory is around 40–48 GB — a single A100 or A6000. The Guanaco models, trained with QLoRA, matched the performance of full fine-tuning baselines on the Vicuna benchmark, proving that the quantization doesn't meaningfully hurt quality.

# Practical QLoRA fine-tuning with PEFT + bitsandbytes
from transformers import AutoModelForCausalLM, AutoTokenizer, BitsAndBytesConfig
from peft import LoraConfig, get_peft_model, prepare_model_for_kbit_training
from trl import SFTTrainer, SFTConfig

# Step 1: Load base model in 4-bit NF4
bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",           # NormalFloat4
    bnb_4bit_compute_dtype="bfloat16",    # compute in BF16
    bnb_4bit_use_double_quant=True,       # double quantization
)

model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-2-7b-hf",
    quantization_config=bnb_config,
    device_map="auto",
)
tokenizer = AutoTokenizer.from_pretrained("meta-llama/Llama-2-7b-hf")

# Step 2: Prepare model and attach LoRA adapters
model = prepare_model_for_kbit_training(model)

lora_config = LoraConfig(
    r=16,                                  # rank
    lora_alpha=32,                         # scaling factor
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj"],
    lora_dropout=0.05,
    bias="none",
    task_type="CAUSAL_LM",
)
model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
# → trainable params: 13.1M || all params: 6,738.4M || trainable%: 0.19%

# Step 3: Fine-tune with SFTTrainer
training_config = SFTConfig(
    output_dir="./qlora-output",
    per_device_train_batch_size=4,
    gradient_accumulation_steps=4,
    learning_rate=2e-4,
    num_train_epochs=3,
    bf16=True,
    logging_steps=10,
    optim="paged_adamw_8bit",              # paged optimizer
    max_seq_length=2048,
)

Adapters

Adapter layers were the original PEFT method, introduced by Houlsby et al. in 2019. The idea: insert small bottleneck modules between the existing transformer sublayers. Each adapter has a down-projection that compresses the hidden state to a small dimension, a nonlinearity (typically ReLU or GELU), and an up-projection back to the original dimension. A residual connection around the whole thing ensures the adapter can learn to do nothing (pass through) at initialization.

The original design places adapters sequentially — after the attention sublayer and after the feed-forward sublayer. This means every token must pass through the adapter during inference, adding latency. Parallel adapters address this by running the adapter alongside the existing sublayer rather than after it. The outputs are summed. This is more GPU-friendly because the adapter computation can overlap with the main computation.

Adapters typically have fewer trainable parameters than LoRA for a given capacity, but they add permanent inference overhead unless you prune or distill them away. This is the key tradeoff: LoRA merges into the weights cleanly, adapters do not. In practice, LoRA has largely displaced adapters for LLM fine-tuning, though adapters remain popular in vision transformers and cross-lingual transfer settings.

Prefix Tuning

Prefix tuning takes a completely different approach. Instead of modifying weights or adding new layers, it prepends learnable continuous vectors to the key and value sequences in every attention layer. Think of these as "virtual tokens" that the model attends to. They steer the model's behavior by shaping what the attention mechanism can retrieve, without changing any of the model's actual parameters.

The prefix vectors are learned through standard backpropagation. In practice, a small reparameterization MLP generates the prefix embeddings during training (for stability), and after training you discard the MLP and keep just the learned vectors. The number of prefix tokens is a hyperparameter — typically 10 to 100. More prefix tokens give more capacity but also consume part of the context window.

Prefix tuning works especially well for generation tasks like summarization and table-to-text, where the prefix can encode the "style" or "format" of the desired output. It's related to prompt tuning but operates deeper — in the attention key-value space of every layer, not just the input embedding layer. This gives it more expressiveness at the cost of slightly more parameters.

Prompt Tuning (Soft Prompts)

Prompt tuning is the simplest PEFT method. You prepend a small number of learnable embedding vectors — called soft prompts — to the input sequence. Only these vectors are optimized. Every single parameter in the model stays frozen. The soft prompts live in the same embedding space as real tokens, but they aren't constrained to correspond to any actual word. The optimizer finds whatever embedding vectors best steer the model for your task.

The parameter count is minimal: if you use 20 soft prompt tokens with an embedding dimension of 4096, that's just 81,920 parameters. For a 7B model, that's 0.001% of the total. The remarkable finding from the original Lester et al. paper is that as model size increases, prompt tuning closes the gap with full fine-tuning. For models above 10B parameters, prompt tuning can match full fine-tuning on many benchmarks.

The killer feature is multitask serving. Keep one frozen model in GPU memory. For each task, load a different set of soft prompt vectors — a few hundred kilobytes. No weight merging, no model reloading. This is ideal for serving platforms where you want to support dozens of tasks without multiplying your GPU fleet. The limitation is that for complex adaptations — heavily changing the model's style, learning new factual knowledge — prompt tuning falls short compared to LoRA. The soft prompts can only influence the model through the input; they can't reshape internal representations.

IA3

IA3 (Infused Adapter by Inhibiting and Amplifying Inner Activations) pushes parameter efficiency even further. Instead of learning low-rank matrices like LoRA or bottleneck layers like adapters, IA3 learns simple scaling vectors. It rescales the keys, values, and intermediate FFN activations element-wise. That's it — just learned multipliers on existing activations.

The parameter count is remarkably small. For each layer, IA3 adds three vectors of dimension d instead of two matrices of shape d × r. For a 7B model, this can mean as few as 500K trainable parameters — roughly 10× fewer than a typical LoRA setup. Performance is competitive on many tasks, especially classification and short-form generation. For more complex tasks requiring deeper behavioral changes, LoRA tends to win. IA3 is a good choice when you need to fine-tune under extreme memory constraints or want to store thousands of task-specific adapters cheaply.

Fine-Tuning Data Considerations

The method you choose matters less than the data you feed it. A mediocre dataset with LoRA will underperform a great dataset with simple prompt tuning. Data quality trumps data quantity, consistently.

Most instruction fine-tuning datasets use a standard format: each example is a triple of (instruction, input, output). The instruction describes the task. The input provides the context (sometimes empty). The output is the desired response. Common formatting standards include the Alpaca format (simple instruction/input/output JSON) and ShareGPT format (multi-turn conversations). Your training framework needs to know where to apply loss — typically only on the output tokens, not the instruction or input.

A landmark result is the LIMA paper ("Less Is More for Alignment"). The authors showed that just 1,000 carefully curated examples — hand-written by researchers — produced a model competitive with those trained on 50,000+ examples. The takeaway: a smaller dataset where every example is high-quality and diverse beats a massive dataset full of noise and redundancy. Spend your time filtering and curating, not scraping more data.

Synthetic data generation has become standard practice. Use a stronger model (GPT-4, Claude) to generate training examples, then fine-tune a smaller model on those examples. This is how many open-source instruction models are built. The risk is "model collapse" — if the synthetic data doesn't cover enough diversity, the fine-tuned model inherits the biases and blind spots of the teacher. Mix synthetic data with some human-written examples for robustness.

Practical Guide: When to Use What

LoRA is the default choice. Start here unless you have a specific reason not to. It's well-supported in every major framework (PEFT, Unsloth, Axolotl), merges cleanly into the base model for deployment, and gives strong performance across tasks. Use r = 16 to 32, target the attention projections, and you'll be fine for most use cases.

QLoRA is what you reach for when GPU memory is the bottleneck — which is most of the time for models above 13B. The 4-bit base model plus BF16 adapters is the best cost-to-performance ratio available today. The slight training speed overhead from dequantization is a small price for fitting a 70B model on hardware that would otherwise require full fine-tuning across 8+ GPUs.

Full fine-tuning still has its place. If you have the compute budget and need maximum performance — say, you're building a flagship product model and quality is paramount — full fine-tuning gives the optimizer the most freedom. Combine it with careful data mixing to avoid catastrophic forgetting.

Prompt tuning makes sense when you're serving many tasks from one model and want minimal overhead per task. The adapter files are tiny (kilobytes), and swapping them is nearly free. The quality tradeoff is acceptable for lighter tasks like classification or simple routing.

For deployment, adapter merging is the cleanest path. Train with LoRA, merge the adapter into the base weights, and deploy a standard model. No adapter logic in your serving infrastructure. If you need to serve multiple adapters simultaneously, frameworks like LoRAX and S-LoRA can batch requests across different adapters efficiently, sharing the base model computation.

The field moves fast, but the principles are stable: freeze what you can, train what you must, and let the data do the heavy lifting.
