Multimodal papers love two-box diagrams, arrow, and words "shared space" — like that explains thing. But what happens inside boxes? How do images and text reach same representation? Dive starts here.

Multimodal models process more than one input type — text and images, text and audio, or all at once. Idea old (humans are multimodal by default). Practical ability exploded around 2021 with CLIP, reshaping search engines, chatbots, and document processing.

The Single-Modality Ceiling

Start with concrete problem. Build smart museum guide. Visitor snaps painting photo and wants info. System must identify artwork and return description.

With only text, search artwork-description database by keyword. Visitor types "oil painting, woman, blue background." Maybe lucky. Usually not — visitor lacks right words, painting may be abstract, or hundred descriptions may match.

With only images, do reverse image search — match photo against database of known paintings. Better. But visitor cannot ask follow-up questions. "Who painted this?" needs language. "Is there anything similar in next gallery?" needs visual style and museum layout.

Ceiling clear: system stuck in one modality answers only questions that modality can express. Real world not like that. Doctor reads an X-ray while reviewing patient notes. Driver watches road while listening for sirens. Child learns word "dog" by hearing it while seeing one. Senses do not work alone. Models should not either.

Observation sounds obvious. But building system that truly combines modalities raises deep question: how make number for image and number for sentence mean same thing?

A Shared Space for Images and Text

Core idea. Imagine 512-dimensional space — coordinate system with 512 axes. Train two separate neural networks: one takes images and outputs point in that 512-D space, and one takes text and outputs point in same space. Train them so photo of golden retriever and sentence "a photo of a golden retriever" land near each other, while photo of car and same sentence land far apart. Then you built something powerful.

You built translation layer. Not between English and French, but between seeing and reading.

Think UN translation desk. Speech in Mandarin and speech in Arabic can express same idea. Interpreters convert both into shared representation — internal understanding — that preserves meaning. Two neural networks do same. Image encoder is visual interpreter. Text encoder is language interpreter. Shared 512-D space is internal understanding both map into.

This is foundation of CLIP (Contrastive Language-Image Pre-training), released by OpenAI in 2021. Rosetta Stone of modern multimodal AI. Almost everything after builds on this idea. Next: how training works.

How Contrastive Learning Actually Trains CLIP

Back to museum guide. Suppose we have three artwork photos, each with caption:

Photo 1: landscape painting → "Rolling hills under cloudy sky"
Photo 2: abstract sculpture → "Twisted metal reaching upward"
Photo 3: portrait → "Woman with pearl earring"

Feed three images through image encoder, get three vectors. Feed three captions through text encoder, get three more. Then compute similarity — cosine similarity, fancy way to say "how much these two arrows point in same direction" — between every image vector and every text vector. Result: 3×3 grid.

Diagonal entries are correct pairs: Photo 1 with Caption 1, Photo 2 with Caption 2, Photo 3 with Caption 3. Everything off diagonal is wrong pair. Training pushes diagonal similarities up and off-diagonal similarities down. That is entire training signal.

Real version uses batches of 32,768 pairs, not 3. Not typo. In contrastive learning, every other sample in batch is negative example — wrong pairing that teaches model what doesn't match. Batch of 3 gives 2 negatives per sample. Batch of 32,768 gives 32,767. More negatives means harder task. Harder task means model must learn more nuanced representations to succeed. If contrastive model fails, check batch size first.

One more detail matters: temperature parameter. Similarity scores get divided by learned temperature τ before softmax. Lower temperature makes distribution more peaked — model must be more confident about matching pairs. Higher temperature makes it more permissive. CLIP learns temperature during training instead of fixing it, so model calibrates own confidence.

Loss function is symmetric: compute cross-entropy from image→text (which image matches which caption?) and from text→image (which caption matches which image?), then average. Both directions matter because relationship is bidirectional — understanding goes both ways at UN translation desk.

import torch
import torch.nn as nn
import torch.nn.functional as F

class CLIPContrastiveLoss(nn.Module):
    """The actual training objective behind CLIP.
    This is the mechanism that creates a shared image-text space."""

    def __init__(self, initial_temperature=0.07):
        super().__init__()
        self.log_temp = nn.Parameter(
            torch.log(torch.tensor(1.0 / initial_temperature))
        )

    def forward(self, image_embeddings, text_embeddings):
        # Both encoders output vectors. We normalize them to unit length
        # so the dot product becomes cosine similarity.
        image_embeddings = F.normalize(image_embeddings, dim=-1)
        text_embeddings = F.normalize(text_embeddings, dim=-1)

        temperature = self.log_temp.exp().clamp(max=100)

        # The 3×3 grid (or 32768×32768 in real training).
        # Each row: one image compared against ALL texts.
        # Each column: one text compared against ALL images.
        similarity = (image_embeddings @ text_embeddings.T) * temperature

        # The correct answer: pair 0 matches pair 0, pair 1 matches pair 1...
        targets = torch.arange(similarity.shape[0], device=similarity.device)

        # Image→text: for each image, which text is the right one?
        loss_i2t = F.cross_entropy(similarity, targets)
        # Text→image: for each text, which image is the right one?
        loss_t2i = F.cross_entropy(similarity.T, targets)

        return (loss_i2t + loss_t2i) / 2

Code block is engine behind shared space. similarity matrix is 3×3 grid (or N×N grid). targets diagonal says "pair i should match pair i." Cross-entropy loss does pushing — diagonal up, off-diagonal down. Temperature controls how sharp model opinions must be.

Zero-Shot Classification from a Shared Space

Payoff of shared space hits here. After CLIP training, two encoders map images and text into same coordinates. No more fine-tuning needed.

Back at museum. Visitor photographs painting never catalogued. We have no classifier trained for this painting. But we can write possible category descriptions — "a landscape painting," "a portrait," "an abstract sculpture," "a still life" — and encode each one. Then encode visitor photo and check which text description lands closest in shared space.

This is zero-shot classification. No training data for specific task. No fine-tuning. Model never saw these categories as classification labels — it learned from natural language captions, and that is enough. CLIP matched or beat fully supervised ImageNet models on benchmarks it never trained for. Field got shocked.

Subtle detail: prompt engineering matters here too. "dog" works worse than "a photo of a dog" because CLIP trained on natural captions, not single words. Original paper tested 80 prompt templates and found ensembling improved accuracy by 3-5%. Lesson: text encoder learned distribution of how people describe things, not dictionary of labels.

Limits show up here too. "Three cats on a sofa" and "one cat on a sofa" produce nearly identical embeddings. "A cup to the left of a plate" and "a plate to the left of a cup" are hard to distinguish. CLIP does not count. CLIP does not understand spatial relationships. Makes sense — model trained on whole-image, whole-caption pairs. It learns what's in the image, not where things are or how many there are.

Limit pulls us toward stronger system. We can match images and text. But we cannot answer questions. We cannot say why something is in image, or generate new text describing what we see. For that, wire vision into language model.

The Fusion Problem: How Do Modalities Combine?

Every multimodal system faces this question: when do you combine information from different senses? Answer shapes what model can do, how expensive it is, and how hard it is to build.

Use museum guide. Three scenarios.

Scenario 1: Late fusion. Visitor photo goes through image model and produces summary: "19th century oil painting, impressionist style, water lilies." Separately, visitor spoken question goes through speech model and produces: "Who painted this?" At end, simple logic matches two summaries. This is late fusion — each modality gets its own specialist, and they talk only at end. Like two experts writing separate reports and manager combining them.

Late fusion is simplest approach. Plug in any pre-trained image model and any pre-trained text model. Each one is debuggable on its own. CLIP is late fusion — two separate encoders, combined only in final similarity computation. Limit: text never looks at specific image patches, and image never attends to specific words. Modalities exchange one message at end.

Scenario 2: Early fusion. Convert photo into sequence of visual tokens (small image patches, each as vector) and question into text tokens. Then concatenate into one long sequence and run through single transformer. Every visual token can attend to every text token from first layer. This is early fusion — throw everything into one pot from start.

Early fusion gives richest cross-modal interaction. Model can learn that word "who" should look at painter's signature in corner of image. But expensive — hard to reuse pre-trained unimodal models, you need aligned tokenization across modalities, and training from scratch here needs enormous compute. Google's Gemini and OpenAI's GPT-4o use early fusion. Most teams cannot.

Scenario 3: Cross-attention fusion. Keep separate encoders, but at intermediate layers let text model peek into image model's internal representations. Text model generates queries (what info do I need?), and image model provides keys and values (here is what I have). This is cross-attention — mechanism from original transformer translation, reused so one modality can interrogate another.

Cross-attention sits in sweet spot. You keep pre-trained models (image encoder stays frozen, language model stays frozen) and add learnable cross-attention layers between them. This is how Flamingo works, how BLIP-2 works, and how most production vision-language models are built.

Heuristic I use: start with late fusion for prototyping. Move to cross-attention when you need fine-grained reasoning ("what color is object left of cat?"). Reach for early fusion only if you have Google-scale compute and Google-scale ambitions.

Teaching a Language Model to See: Cross-Attention in Practice

Get concrete. Here is how cross-attention works when text reads image. This mechanism sits inside Flamingo, BLIP-2, and most production vision-language models, so mechanical understanding matters.

Say we have 20 text tokens (visitor question: "What art movement does this painting belong to?") and 196 image patches (14×14 grid from Vision Transformer processing painting). Each text token becomes query: "I am word 'movement' — what visual info matters to me?" Each image patch becomes key (my address) and value (my content). Text query computes similarity against all 196 image keys, gets attention-weight distribution, and reads weighted combination of image values.

Result: each text token gets visual information from specific image patches relevant to it. Word "movement" might attend strongly to brushstroke textures. Word "painting" might attend to frame and canvas. Word "this" might attend broadly to whole image.

import torch
import torch.nn as nn
import torch.nn.functional as F

class CrossModalAttention(nn.Module):
    """Text tokens query image patches — the core of Flamingo, BLIP-2, etc.
    This is where 'the language model learns to see.'"""

    def __init__(self, text_dim=768, vision_dim=1024, num_heads=8):
        super().__init__()
        self.num_heads = num_heads
        self.head_dim = text_dim // num_heads

        # Queries come from text. Keys and values come from the image.
        self.q_proj = nn.Linear(text_dim, text_dim)
        self.k_proj = nn.Linear(vision_dim, text_dim)
        self.v_proj = nn.Linear(vision_dim, text_dim)
        self.out_proj = nn.Linear(text_dim, text_dim)

        # The gate: starts at zero, meaning the model initially ignores
        # visual information entirely. During training, it gradually learns
        # how much vision to mix in. This preserves the LLM's existing
        # language abilities while adding sight.
        self.gate = nn.Parameter(torch.zeros(1))
        self.norm = nn.LayerNorm(text_dim)

    def forward(self, text_tokens, image_patches):
        residual = text_tokens
        B, T, _ = text_tokens.shape
        P = image_patches.shape[1]

        Q = self.q_proj(text_tokens).view(
            B, T, self.num_heads, self.head_dim
        ).transpose(1, 2)
        K = self.k_proj(image_patches).view(
            B, P, self.num_heads, self.head_dim
        ).transpose(1, 2)
        V = self.v_proj(image_patches).view(
            B, P, self.num_heads, self.head_dim
        ).transpose(1, 2)

        # Each text token attends to all image patches
        attn_weights = (Q @ K.transpose(-2, -1)) * (self.head_dim ** -0.5)
        attn_weights = F.softmax(attn_weights, dim=-1)

        visual_context = (attn_weights @ V).transpose(1, 2).reshape(B, T, -1)
        visual_context = self.out_proj(visual_context)

        # Gated residual: tanh(gate) starts near 0, grows during training
        return self.norm(residual + torch.tanh(self.gate) * visual_context)

Gate in that code matters. It starts at zero, so tanh(0) = 0, so visual context gets multiplied by zero and discarded. Model starts as pure language model. During training, gate slowly opens and visual information flows in. This is DeepMind's Flamingo trick. Elegant because random initial outputs from visual encoder cannot corrupt good language model. Model decides how much to trust new sense of sight.

Flamingo: The Careful Architecture

DeepMind's Flamingo (2022) was first model that could handle interleaved image-text sequences — like showing two paintings and asking "How are these different?" Architecture has three key pieces.

First, frozen vision encoder extracts features from each image. Output is grid of 196+ tokens per image — one for each patch Vision Transformer divides image into.

Second, Perceiver Resampler. Important part. 196 tokens per image is lot to feed into language model at every layer. If visitor shows three paintings, that is nearly 600 visual tokens competing with text tokens for attention. Perceiver Resampler compresses them. Start with small set of learned query tokens (usually 64). These queries cross-attend to all 196 image patches, distilling image into compact representation. Like asking 64 targeted questions about image and getting 64 focused answers, instead of dumping raw visual field into language model's lap.

Third, gated cross-attention layers from code above, inserted at regular intervals into frozen LLM. LLM stays frozen — language abilities survive. Only cross-attention layers and Perceiver Resampler get trained.

Flamingo proved you can add vision to powerful language model without destroying it. But architecture is complex. Resampler, gated cross-attention, careful freezing strategy. Someone saw this complexity and had radical thought.

LLaVA: The Embarrassingly Simple Approach That Worked

LLaVA (Large Language and Vision Assistant) asked: what if we skip Perceiver Resampler, skip gated cross-attention, and do dumbest possible thing? Take image features from CLIP's Vision Transformer. Pass them through small MLP — two linear layers with GELU between — to project them into LLM's embedding dimension. Then concatenate projected visual tokens with text tokens and feed whole thing into LLM like all of it is text.

That is all. No special attention mechanisms. No gates. No frozen components in second training stage. Visual tokens sit beside text tokens, and LLM's existing self-attention treats them all same.

Training happens in two stages. First, freeze LLM and train only MLP projection on image-caption pairs — teaching projection layer to map visual features into right neighborhood of LLM's embedding space. Second, unfreeze everything and fine-tune on visual instruction-following data — conversations where user asks questions about image and gets detailed answers.

My favorite part of LLaVA: aside from high-level explanations for why simple components should work together, nobody is fully sure why it works this well. Flamingo's careful design — resampler, gating, freezing — looked necessary. LLaVA showed that two-layer MLP plus brute-force concatenation gets you surprisingly far. Authors used GPT-4 to generate 150,000 visual instruction-following examples, and that training data mattered more than architecture.

Open-source community jumped on this. LLaVA variants now rival commercial systems, built by researchers without Flamingo-scale compute budgets. Lesson: when in doubt, try simple thing first.

Visual Question Answering: Where Fusion Gets Tested

Visual Question Answering (VQA) tests most directly whether model truly understands both modalities. Show image. Ask question. Get answer.

Back to museum guide. Visitor photographs painting and asks: "What art movement is this?" Useful answer needs visual understanding (brushstroke style, color palette, composition) and language (question itself, plus knowledge of art movements). Neither modality alone is enough.

Early VQA systems (2015-2019) used pipeline approach: extract image features with CNN, encode question with LSTM, combine them with element-wise multiplication or small fusion network, and predict answer from fixed vocabulary. Worked for simple questions ("What color is the car?") but failed on reasoning questions ("Is there enough food on the table for everyone?").

Modern VQA uses vision-language models we built toward. LLaVA, GPT-4V, and Gemini do not have fixed answer vocabulary — they generate free-form text answers. So they can say "This painting shows characteristics of Post-Impressionism, particularly in its use of bold, non-naturalistic color and visible brushstrokes, reminiscent of Van Gogh's later work." Big leap from picking "post-impressionism" from dropdown.

But VQA also exposes limits. Model might answer "What sport is being played?" correctly by recognizing uniform colors, not by understanding action. It might answer "How many people are in the photo?" wrong because — as we saw — counting is hard. Right answer for wrong reasons is persistent evaluation problem in multimodal systems, and one reason human evaluation remains gold standard for anything you deploy to real users.

Image Captioning and the Text-to-Image Connection

Image captioning — generating natural-language description of image — is VQA's cousin. Instead of answering specific question, model describes what it sees. Same vision-language architectures handle both tasks: encode image, feed visual features into language model, let it generate text.

Captioning matters here because of link to text-to-image generation. Captioning goes image → text. Text-to-image goes text → image. They are inverse problems, and systems like Stable Diffusion use same CLIP text encoder that powers captioning and VQA to condition image generation. Text encoder converts prompt into embedding vectors, and those vectors guide diffusion model through cross-attention — same cross-attention mechanism we saw in vision-language models, now inside U-Net denoiser. Each denoising step "reads" prompt to decide how to refine each image region.

I will not go deeper into diffusion here (own chapter), but connection matters: same shared embedding space that enables understanding also enables generation. UN translation desk works both directions.

Audio-Visual Learning: Beyond Text and Images

So far we focused on text-image pair because field matured there first. But humans live in multimodal world with sound, touch, and motion. Model that watches speaking video and lip-reads when audio is noisy — that is genuinely useful multimodal capability requiring vision and audio together.

Audio follows same pattern. Convert audio signal into spectrogram (visual representation of sound frequencies over time), treat it like image, and encode it with vision-like model. Or learn audio tokens directly, similar to BPE tokenization for text. Either way, you get sequence of vectors that can use same fusion strategies — late fusion, cross-attention, or early fusion — as any other modality.

More interesting development: ImageBind from Meta (2023). It showed something surprising. You can build shared embedding space across six modalities — images, text, audio, depth, thermal, and IMU (motion sensor data) — by anchoring everything to one modality: images.

Here is trick. You do not need paired data for every modality combination. You do not need audio-thermal pairs or depth-text pairs. You need image-text pairs (CLIP already has them), image-audio pairs (videos give them free), image-depth pairs (RGB-D cameras), and so on. Images act as hub modality. Because every other modality trains to align with images, they end up aligned with each other too — even without explicit pairings. Audio-to-text retrieval works, even though model never saw audio-text pairs during training.

This emergent alignment surprised me when I first read it. It feels wrong — aligning A with C and B with C does not guarantee A and B align well with each other. But in high-dimensional spaces with enough training data, transitive property holds well enough to be useful. UN translation desk analogy stretches here too: if every interpreter translates into same internal representation, speakers of any two languages can communicate, even if those interpreters never worked together directly.

Reading Documents with Layout: LayoutLM

One multimodal understanding type gets overlooked and turns out very valuable in production: document understanding. Form, invoice, receipt — these are visual documents where spatial layout carries meaning. Number "1,234.56" means very different things depending on whether it sits next to "Total Due" or "Item Count."

Microsoft's LayoutLM family tackled this by extending BERT in clever way. Insight: OCR (optical character recognition) gives not only page text, but also bounding box coordinates — where each word sits physically. LayoutLM takes each word's text embedding (same as regular BERT) and adds 2D position embedding encoding word's spatial location: x and y coordinates of top-left corner and bottom-right corner of each word's bounding box.

Walk through concrete case. On invoice, word "Total" appears at coordinates (100, 500) and number "$4,259.00" appears at (350, 500) — same y-coordinate, so same line. Word "Date" appears at (100, 50) with "March 15, 2024" at (350, 50) — also same line, but top of page. Without spatial info, model sees jumbled bag of words: "Total", "$4,259.00", "Date", "March 15, 2024". With spatial embeddings, model understands which values belong together — things on same line and near each other are related.

Later versions (LayoutLMv2, LayoutLMv3) added actual image pixels as third modality — so model sees text, knows where text sits, and also sees visual appearance of document (lines, boxes, logos, signatures). This was critical for handling things like checkboxes (is it checked or not?) and tables (cell boundaries matter more than text proximity).

My intuition here is still forming: why do visual pixels on top of text + layout help so much? You might think bounding boxes capture most layout information. Best guess: visual cues like separator lines, bold headers, and table borders add structural information bounding boxes miss. Pixels tell model "there's a horizontal line here separating the items from the total" in way coordinates cannot.

The Modality Gap: Shared Spaces Aren't Really Shared

This bothered me when I first learned it, and more people should know it. Remember CLIP's "shared" embedding space for images and text? If you inspect where embeddings land, they do not overlap.

Image embeddings cluster in one region. Text embeddings cluster in another. Persistent gap sits between them — systematic offset. Photo of dog and caption "a photo of a dog" are closer to each other than random image-text pair, so CLIP works. But they are not in same neighborhood. Images form one cone in high-dimensional space. Text forms another cone. Cones point in roughly similar directions, but do not intermingle.

This is modality gap, and 2024 research traces it to two sources. First, neural networks have inductive biases that collapse embeddings for each modality into narrow clusters — "cone effect." Second, contrastive loss preserves and even reinforces this separation. Loss cares only about relative ordering — is correct pair more similar than wrong pairs? — not about absolute proximity. So model can satisfy training objective while keeping modalities in separate neighborhoods.

What does this mean in practice? Direct comparison between image embedding and text embedding always has baseline offset. Retrieval systems mixing both modalities in one index must account for gap — text query may be systematically closer to all text items than to any image item, regardless of semantic relevance. And methods like ImageBind's emergent cross-modal alignment have more nuanced story than "everything lives in same space."

Some researchers say gap should be minimized (AlignCLIP uses parameter sharing and specialized losses to close it). Others say accept it — forcing modalities to overlap can destroy useful semantic structure within each modality. I find "accept it" camp more convincing, but debate is active. Honest answer: we are still figuring out geometry of multimodal embedding spaces.

Unified Architectures: GPT-4o and Gemini

Everything so far uses stitching: take pre-trained vision model, take pre-trained language model, wire them together with projections or cross-attention. Adapter approach. Works. Practical. But ceiling exists on how deeply modalities can interact.

Alternative: build natively multimodal model from scratch. Instead of training separate encoders and then figuring out connection, design one architecture that processes text, images, audio, and video as different token types in one unified sequence. Every token can attend to every other token from first layer. This is early fusion taken to end point.

Google's Gemini (2023) was built this way. Text, images, audio, and video are tokenized and interleaved in one sequence. Model applies standard transformer attention across whole mixed-modality context. So audio token can attend to image patch can attend to text token — orchestra conductor coordinates all instruments from first measure, not finale.

GPT-4o ("o" means "omni") from OpenAI pushed this further in 2024, adding real-time audio input and output. One model can see images, read text, hear speech, and answer with speech — all processed natively in one architecture. Practical result: no more speech-to-text → text processing → text-to-speech pipeline. Model processes audio directly, so it catches tone, emphasis, and emotion that transcription would lose.

Conductor analogy returns. Adapter-based models are like orchestra where string section rehearses separately, brass section rehearses separately, then they meet for performance. Unified architectures are like orchestra rehearsing together from day one — every section learns to listen to and complement others from start. Music is more integrated, but rehearsal time grows (read: training compute) and conductor must be skilled (read: careful architecture design).

For practitioners, question is whether you need that much integration. If task needs tight cross-modal reasoning — understanding diagram where text labels, arrows, and visual elements all interact — natively multimodal models have clear edge. If you need image-description matching or basic visual question answering, adapter-based approach like LLaVA gets you 90% there at fraction of cost.

Multimodal Embeddings in Practice

With theory in hand, build something. Most immediately useful multimodal capability for most engineers is CLIP embeddings as search and retrieval infrastructure.

Back to museum guide one last time. We have catalog of 10,000 paintings with descriptions. We want visitor to snap photo and find closest match, or type "impressionist landscapes with water" and get relevant results. We also want reverse queries: given painting, find related descriptions or exhibition notes.

Pattern: precompute CLIP embeddings for all images and all text descriptions. Store them in vector database. At query time, encode input (image or text) and find nearest neighbors. Because images and text live in same space (modality gap still there), text query can retrieve images and vice versa.

import torch
import numpy as np
from transformers import CLIPModel, CLIPProcessor

class MuseumSearchEngine:
    """A stripped-down multimodal search index.
    In production, you'd replace the numpy search with FAISS or a
    vector database like Pinecone, Weaviate, or Qdrant."""

    def __init__(self, model_name="openai/clip-vit-base-patch32"):
        self.model = CLIPModel.from_pretrained(model_name)
        self.processor = CLIPProcessor.from_pretrained(model_name)
        self.embeddings = []
        self.metadata = []

    @torch.no_grad()
    def index_image(self, image, info):
        inputs = self.processor(images=image, return_tensors="pt")
        emb = self.model.get_image_features(**inputs)
        emb = emb / emb.norm(dim=-1, keepdim=True)
        self.embeddings.append(emb.squeeze().numpy())
        self.metadata.append(info)

    @torch.no_grad()
    def search_by_text(self, query, top_k=5):
        inputs = self.processor(
            text=[query], return_tensors="pt", padding=True
        )
        query_emb = self.model.get_text_features(**inputs)
        query_emb = query_emb / query_emb.norm(dim=-1, keepdim=True)

        corpus = np.stack(self.embeddings)
        scores = (query_emb.numpy() @ corpus.T).squeeze()
        top_idx = np.argsort(scores)[::-1][:top_k]
        return [(self.metadata[i], float(scores[i])) for i in top_idx]

That is entire search engine. Fifty lines. index_image method encodes paintings into vectors. search_by_text method encodes text query into same space and finds nearest painting vectors. L2 normalization (dividing by norm) ensures we compare directions, not magnitudes — cosine similarity in disguise.

In production, swap numpy search for FAISS (speed), add vector database (persistence and scaling), and maybe add re-ranking step with stronger VLM. But core pattern — embed everything into shared space, retrieve by nearest neighbor — is foundation of every multimodal search system running today.

The Challenges That Remain

We have come long way. Hard parts still remain.

Modality gap means cross-modal retrieval always carries systematic bias. Fixes exist, but they add complexity. Compositional understanding — "a red cube on top of a blue sphere" versus "a blue cube on top of a red sphere" — stays fragile across current models, whether matching, generating, or answering questions. Counting is unreliable. Spatial reasoning is inconsistent.

Evaluation is its own problem. Generated image can be beautiful and still miss prompt completely. VQA model can get right answer for wrong reason. CLIPScore (cosine similarity between CLIP embeddings of generated image and prompt) became standard automated metric, but it inherits CLIP's limits. Human evaluation remains gold standard, and it does not scale.

In production, messy inputs are normal. Camera drops frame. User uploads image with no description. Audio cuts out. Robust multimodal systems use modality dropout during training — randomly zeroing whole modality inputs 20-30% of time — so model learns to predict from whatever subset is available. Some systems use learned null embeddings: when modality is missing, substitute special "no information" token instead of zero vector, so model can distinguish "nothing here" from "I see something that encodes to zero."

Field moves fast enough that specific architecture details may be outdated by time you read this. But trajectory is clear: separate unimodal models are being replaced by unified architectures. "Any-to-any" model — one system taking any input combination and producing any output combination — is no longer theoretical. Remaining challenges: efficient tokenization for each modality, managing enormous context lengths required by video and audio, and training at sufficient scale.
