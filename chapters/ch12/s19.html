Production LLM Systems

Chapter 12: Large Language Models 5 subtopics
TL;DR

Training an LLM is the easy part — serving it to millions of users without burning money or dropping requests is where the real engineering lives. Production LLM systems revolve around three pillars: smart batching (continuous batching beats naive static batching by 10–20×), efficient memory management (the KV cache is your main bottleneck, not compute), and cost-aware routing (speculative decoding and model cascading let you spend GPU dollars only where they matter). Master these and you can serve a 70B-parameter model on hardware that would otherwise choke on a 7B one.

You've fine-tuned your model. The eval numbers look great. Now a product manager says "ship it" and you discover a brutal truth: inference at scale is a completely different discipline than training. Training is a batch job — you throw GPUs at it, wait, and collect a checkpoint. Serving is real-time. Every millisecond of latency costs user patience. Every idle GPU cycle costs money. And the autoregressive nature of LLMs — generating one token at a time, sequentially — makes this fundamentally harder than serving a BERT classifier or an image model. Let's break down how production systems solve this.

LLM Serving Architecture

The core challenge is this: an LLM generates tokens one at a time. Each forward pass through a 70-billion-parameter model produces exactly one token. A typical response might be 500 tokens. That's 500 sequential forward passes. You cannot parallelize this across tokens because each token depends on the previous one. So the question becomes: how do you keep the GPU busy while it's stuck in this sequential loop?

Why Naive Serving Fails

The simplest approach is static batching: collect N requests, pad them to the same length, run them together, and return all results when the longest one finishes. This is how we batch image classification — and it's terrible for LLMs. Here's why. Suppose you batch 8 requests. Request 1 needs 20 tokens, request 8 needs 500 tokens. With static batching, all 8 slots stay occupied until request 8 finishes. Requests 1 through 7 are done but their GPU memory sits wasted. Meanwhile, new incoming requests queue up behind the batch. Throughput craters.

Continuous Batching

Continuous batching (also called iteration-level batching) fixes this by operating at the granularity of individual decode steps rather than entire requests. After every single token generation step, the scheduler checks: did any request just finish? If so, eject it and immediately slot in a waiting request. The GPU never idles on completed sequences.

# Conceptual continuous batching loop
class ContinuousBatchScheduler:
    def __init__(self, model, max_batch_size):
        self.model = model
        self.max_batch = max_batch_size
        self.active = []       # currently generating
        self.waiting = []      # queued requests

    def step(self):
        # Remove finished requests
        self.active = [r for r in self.active if not r.is_done()]

        # Fill empty slots from waiting queue
        while len(self.active) < self.max_batch and self.waiting:
            new_req = self.waiting.pop(0)
            new_req.prefill()   # compute initial KV cache
            self.active.append(new_req)

        # One decode step for all active requests
        if self.active:
            tokens = self.model.decode_step(self.active)
            for req, tok in zip(self.active, tokens):
                req.append_token(tok)

The key insight: the batch composition changes at every iteration. This is fundamentally different from training, where your batch is fixed for the entire forward-backward pass. Systems like vLLM, TensorRT-LLM, and TGI all implement continuous batching. In benchmarks, continuous batching typically achieves 10–23× higher throughput than static batching on the same hardware. That's not a small optimization — it's the difference between needing 20 GPUs and needing 1.

💡 Prefill vs. Decode

LLM inference has two distinct phases. Prefill processes the entire input prompt in parallel — this is compute-bound and fast. Decode generates output tokens one by one — this is memory-bandwidth-bound and slow. Production schedulers treat these phases differently. Some systems (like Sarathi-Serve) even split prefill and decode across separate GPU pools to avoid interference.

KV Cache Management

Here's the real bottleneck nobody warns you about. During autoregressive decoding, the model needs access to the key and value tensors from every previous token at every attention layer. Recomputing them each step would be wasteful, so we cache them — that's the KV cache. The problem? It's enormous. For a 70B model with 80 layers, 64 heads, and a 128-dimensional head size, the KV cache for a single sequence of 2048 tokens is roughly:

# KV cache size per sequence
layers = 80
kv_heads = 64   # could be fewer with GQA
head_dim = 128
seq_len = 2048
bytes_per_param = 2  # FP16

kv_cache_bytes = 2 * layers * kv_heads * head_dim * seq_len * bytes_per_param
print(f"{kv_cache_bytes / 1e9:.1f} GB")  # ~5.4 GB per sequence!

5.4 GB for a single sequence. If you want to serve a batch of 8 concurrent users, that's 43 GB just for KV cache — more than an entire A100's memory. This is why KV cache management is the central engineering challenge of LLM serving.

The breakthrough insight behind vLLM is PagedAttention. Instead of pre-allocating a contiguous block of memory for each sequence's maximum possible length, PagedAttention borrows the concept of virtual memory paging from operating systems. The KV cache is divided into fixed-size blocks (pages). Sequences get pages allocated on demand, and pages can be non-contiguous in physical GPU memory. A page table maps logical token positions to physical memory locations.

# PagedAttention: KV cache as non-contiguous pages
class PagedKVCache:
    def __init__(self, block_size=16, num_blocks=1024):
        self.block_size = block_size
        # Physical KV blocks on GPU
        self.kv_blocks = torch.zeros(num_blocks, 2, num_heads, block_size, head_dim)
        self.free_blocks = list(range(num_blocks))
        self.page_tables = {}  # request_id -> [block_indices]

    def allocate(self, request_id):
        block_idx = self.free_blocks.pop()
        self.page_tables.setdefault(request_id, []).append(block_idx)
        return block_idx

    def free(self, request_id):
        blocks = self.page_tables.pop(request_id)
        self.free_blocks.extend(blocks)

Why does paging matter so much? Without it, you must reserve memory for the maximum sequence length up front, even if most requests use far less. A 2048-max-length reservation that only generates 200 tokens wastes 90% of its allocated memory. PagedAttention eliminates this waste. In practice, vLLM achieves 2–4× higher throughput than static-allocation systems like HuggingFace's text-generation-inference (before it adopted similar techniques) purely through better memory utilization.

⚠️ KV Cache Is Why Context Length Is Expensive

When a provider charges 10× more for a 128K-context model vs. an 8K model, it's not because the model is bigger. It's because the KV cache grows linearly with context length. Doubling the context window roughly halves the number of concurrent requests you can serve on the same GPU. This is the direct economic pressure behind techniques like GQA (Grouped Query Attention), MQA, and KV cache quantization.

Speculative Decoding

Autoregressive decoding is fundamentally sequential: one token per forward pass. Or is it? Speculative decoding is a clever trick that breaks this bottleneck without changing the model or its outputs.

The idea: use a small, fast "draft" model to speculatively generate K tokens ahead. Then verify all K tokens in a single forward pass of the large "target" model. If the target model agrees with the draft tokens, you just generated K tokens for the cost of one large forward pass plus one small forward pass. If some tokens are rejected, you fall back to the first rejected position and try again.

# Speculative decoding: draft, then verify
def speculative_decode(draft_model, target_model, prompt, K=5):
    tokens = list(prompt)

    while not is_done(tokens):
        # Step 1: Draft K tokens quickly
        draft_tokens = []
        draft_probs = []
        draft_input = tokens[:]
        for _ in range(K):
            p = draft_model.get_probs(draft_input)
            t = sample(p)
            draft_tokens.append(t)
            draft_probs.append(p[t])
            draft_input.append(t)

        # Step 2: Verify all K tokens in one target forward pass
        target_probs = target_model.get_probs_batch(tokens, draft_tokens)

        # Step 3: Accept or reject each token
        for i, dt in enumerate(draft_tokens):
            target_p = target_probs[i][dt]
            draft_p = draft_probs[i]
            # Accept with probability min(1, target_p / draft_p)
            if random.random() < min(1.0, target_p / draft_p):
                tokens.append(dt)
            else:
                # Reject: sample from adjusted distribution
                adjusted = max(0, target_probs[i] - draft_probs_full[i])
                tokens.append(sample(normalize(adjusted)))
                break

    return tokens

The mathematical beauty here: the acceptance-rejection scheme guarantees that the output distribution is identical to what the target model would have produced on its own. You get a 2–3× speedup with zero quality loss. The acceptance rate depends on how well the draft model approximates the target. A 7B draft model paired with a 70B target might see 70–85% acceptance on typical text. Codex-style code generation, where tokens are more predictable, often sees even higher acceptance rates.

💡 Draft Model Selection

The ideal draft model is 10–20× smaller than the target but trained on similar data. Some systems use an earlier layer of the same model (self-speculative decoding) or a separate head attached to an intermediate layer (Medusa). The key tradeoff: a better draft model means higher acceptance rates but slower drafting. You want the sweet spot where verification savings outweigh drafting costs.

Cost Optimization

GPU inference is expensive. A single A100 runs around $2–3/hour on cloud providers. Serving a 70B model needs at least 2 A100s (for FP16 weights alone), so you're looking at $4–6/hour before even considering the KV cache overhead or redundancy. At scale, inference costs dominate — often 90% of the total ML budget. Here's how production systems fight back.

Model Cascading

Model cascading routes easy queries to small, cheap models and only escalates to large, expensive models when needed. Think of it like a hospital triage system. A simple "What's the capital of France?" doesn't need GPT-4 — a 7B model handles it fine. A complex multi-step reasoning question might need the full 70B model.

# Simple model cascading with confidence routing
class ModelCascade:
    def __init__(self):
        self.small = load_model("7B")    # $0.001 per request
        self.large = load_model("70B")   # $0.02 per request

    def route(self, prompt):
        # Try small model first
        response, confidence = self.small.generate_with_confidence(prompt)

        if confidence > 0.85:
            return response, "small"
        else:
            # Escalate to large model
            response = self.large.generate(prompt)
            return response, "large"

# If 70% of traffic is "easy," you save:
# 0.7 * ($0.02 - $0.001) = $0.0133 per request
# At 1M requests/day: ~$13,300/day saved

The confidence estimation is the tricky part. Common approaches: use the average token log-probability from the small model, train a lightweight classifier on the prompt, or use the small model's self-assessed uncertainty. Some systems use the length and complexity of the prompt as a routing signal — simple factual queries go small, long multi-turn conversations go large.

KV Cache Quantization

Your model weights might be quantized to INT4 or INT8, but what about the KV cache? It's consuming more memory than the model itself for long sequences. KV cache quantization compresses cached keys and values from FP16 to INT8 or even INT4. This roughly halves or quarters your memory usage, letting you serve 2–4× more concurrent users. The quality impact is surprisingly small — attention patterns are robust to moderate quantization noise, typically degrading perplexity by less than 1%.

Prompt Caching

Many production workloads share common prefixes. A customer support bot might prepend the same 2000-token system prompt to every request. Prompt caching (also called prefix caching) computes the KV cache for shared prefixes once and reuses it across requests. If 1000 requests share the same system prompt, you compute 2000 tokens of prefill once instead of 1000 times. vLLM implements this through its block-sharing mechanism in PagedAttention — identical prefix blocks share the same physical memory pages via reference counting, just like copy-on-write in operating systems.

🔥 The Real Cost Equation

Don't just measure cost per token. Measure cost per useful token. If your system retries 30% of requests due to quality issues with a cheap model, or if a slow model causes users to abandon sessions, the "savings" evaporate. Track end-to-end metrics: cost per successful task completion, not cost per generated token. A $0.02 request that works is cheaper than three $0.005 requests that don't.

Application Design Patterns

Prompt Versioning

Your prompt is code. Treat it like code. Prompt versioning means tracking every change to your system prompts, few-shot examples, and template strings in version control with the same rigor as application code. Why? Because a one-word change in a system prompt can swing task accuracy by 15%. You need to know exactly which prompt version produced which outputs, especially for debugging regressions and A/B testing.

# Prompt registry with versioning
class PromptRegistry:
    def __init__(self, storage_path="prompts/"):
        self.path = storage_path

    def get(self, name, version="latest"):
        # Load versioned prompt template
        path = f"{self.path}/{name}/v{version}.txt"
        with open(path) as f:
            return f.read()

    def log_usage(self, name, version, request_id, response_quality):
        # Track which prompt version produced which results
        log_entry = {
            "prompt": name, "version": version,
            "request_id": request_id,
            "quality_score": response_quality,
            "timestamp": datetime.now().isoformat()
        }
        append_to_log("prompt_usage.jsonl", log_entry)

Monitoring and Observability

LLM systems fail in ways traditional software doesn't. The model doesn't crash — it confidently returns garbage. You need monitoring that catches semantic failures, not just HTTP 500s. Track these metrics: time to first token (TTFT) measures user-perceived latency for streaming responses. tokens per second (TPS) measures generation throughput. time per output token (TPOT) measures the decode phase speed. And beyond latency, track output quality signals: refusal rates, average response length, user feedback scores, and downstream task success rates.

Fallback Strategies

Production systems need graceful degradation. If your primary model provider goes down (and they will), what happens? A fallback chain might look like: try the primary 70B model → if timeout or error, try a secondary provider's equivalent model → if that fails, try a local 7B model → if that fails, return a cached response or a human-readable error. Each fallback step trades quality for availability. The key is making this transparent to your monitoring system so you know how often you're falling back and can react accordingly.

Specialized LLM Applications

Code Generation Models

Code LLMs like StarCoder, DeepSeek-Coder, and CodeLlama are pre-trained on massive code corpora with a crucial twist: they support fill-in-the-middle (FIM). Standard left-to-right generation can only complete code after the cursor. FIM training teaches the model to generate code that fits between a prefix and a suffix — exactly what you need for IDE autocomplete where the cursor is in the middle of a function. The training reformats code as [PREFIX]code_before[SUFFIX]code_after[MIDDLE]code_to_generate, teaching the model to condition on both surrounding context.

Speech Models

Whisper, OpenAI's speech-to-text model, is essentially a sequence-to-sequence Transformer trained on 680,000 hours of labeled audio. It takes log-mel spectrogram features as input and autoregressively generates text tokens — the same architecture as a language model, just with audio features instead of text embeddings as the encoder input. Production speech pipelines often chain Whisper (audio → text) with an LLM (text → response) with a TTS model (response → audio), creating a voice assistant stack where each component can be independently scaled and optimized.

Model Merging

Model merging combines multiple fine-tuned models into a single model without additional training. Think of it as mixing paint: blend a model fine-tuned on code with one fine-tuned on math, and you might get a model that does both well. SLERP (Spherical Linear Interpolation) interpolates between two models along the surface of a hypersphere in weight space. TIES merging resolves conflicts when merging multiple models by trimming small-magnitude changes and resolving sign disagreements. DARE (Drop And REscale) randomly drops most delta parameters before merging, based on the insight that fine-tuning changes are highly redundant. These techniques power the open-source model ecosystem — many top models on the Open LLM Leaderboard are merges, not independently trained models.

💡 The Production Stack in Practice

A typical production LLM deployment looks like: an API gateway handling auth and rate limiting → a router deciding which model to use (cascade) → a serving engine (vLLM/TGI) with continuous batching and PagedAttention → a monitoring layer tracking latency, cost, and quality → a prompt management system with versioning and A/B testing. Each layer solves a different problem. Skip any one and you'll feel the pain within weeks of launching.
