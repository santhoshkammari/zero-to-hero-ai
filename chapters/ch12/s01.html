LLM Training — Pretraining, Scaling & Fundamentals

Chapter 12: Large Language Models 14 subtopics
TL;DR

This section covers the three pillars of LLM training: pretraining objectives (CLM, MLM, span corruption, ELECTRA), scaling laws (Kaplan vs Chinchilla and why it changes everything about how you budget compute), and the practical engineering of actually training these models (data prep, AdamW, learning rate schedules, mixed precision, distributed training, and stability tricks). The pretraining objective you choose determines what the model can do. Scaling laws tell you how big to make it and how much data to feed it. And the training fundamentals determine whether your run actually converges or burns money. Master all three and you can plan, launch, and debug a real LLM training run.

Part I — Pretraining Objectives

Before an LLM can chat, summarize, or write code, it has to be pretrained — trained on a massive corpus with a self-supervised objective. No labels, no human annotation. Just raw text and a clever loss function. The choice of pretraining objective is arguably the single most important design decision you make, because it determines what the model learns to represent and what it can later be fine-tuned to do.

There are four major objectives worth knowing cold. Let’s go through each.

Causal Language Modeling (Next-Token Prediction)

This is the one that powers GPT, LLaMA, Mistral, Claude, and basically every model you’ve interacted with through a chatbot. Causal language modeling (CLM) trains a model to predict the next token given all previous tokens. That’s the entire objective. The most capable AI systems on the planet are built on “read left to right, guess what comes next.”

Formally, given a sequence of tokens x₁, x₂, ..., xₙ, we factorize the joint probability autoregressively:

P(x₁, x₂, ..., xₙ) = ∏ₜ P(xₜ | x₁, x₂, ..., xₜ₋₁)

At each position t, the model predicts:
  P(xₜ | x₁, x₂, ..., xₜ₋₁)

The model never looks ahead. Position 5 can attend to positions 1–4 but never to 6, 7, or 8. This is enforced by a causal attention mask — a lower-triangular matrix that blocks attention to future positions:

Causal Mask (seq_len = 5):

         Key positions →
Query  ┌                     ┐
  ↓    │  1  0  0  0  0      │   position 1 sees only itself
       │  1  1  0  0  0      │   position 2 sees 1-2
       │  1  1  1  0  0      │   position 3 sees 1-3
       │  1  1  1  1  0      │   position 4 sees 1-4
       │  1  1  1  1  1      │   position 5 sees 1-5
       └                     ┘
  1 = can attend, 0 = masked (set to -∞ before softmax)

The loss is cross-entropy computed at every position. For each token, the model outputs a probability distribution over the entire vocabulary (typically 32k–128k tokens), and you penalize it for assigning low probability to the actual next token. Sum or average across all positions.

Here’s the critical implementation detail — the shift trick:

# CLM training step — the shift trick
logits = model(input_ids)              # (batch, seq_len, vocab_size)

# The model predicts position t+1 from position t
# So we align: logits at position t → label at position t+1
shift_logits = logits[:, :-1, :]       # all but last position
shift_labels = input_ids[:, 1:]        # all but first token

loss = F.cross_entropy(
    shift_logits.reshape(-1, vocab_size),
    shift_labels.reshape(-1)
)
loss.backward()

Why the shift? Because logits[t] is the model’s prediction for position t+1, made using context from positions 1..t. So you compare logits[:, :-1] (predictions for positions 2 through N) against labels[:, 1:] (actual tokens at positions 2 through N). Off-by-one errors here are one of the most common bugs in training code.

During training, we use teacher forcing: we feed the model the actual ground-truth tokens at every position, not its own predictions. This stabilizes learning enormously. At inference time, you don’t have ground truth, so you feed back the model’s own outputs autoregressively — which is why generation quality can degrade over very long sequences.

Why does next-token prediction produce such capable models? Because language encodes everything. To predict the next word accurately, the model must learn syntax, semantics, world knowledge, reasoning patterns, social conventions, code logic, and mathematical structure. The objective is trivially simple; the knowledge required to minimize it is not. This is the deep insight behind GPT-scale models.

Masked Language Modeling (MLM)

BERT took a completely different approach. Instead of predicting left-to-right, masked language modeling randomly hides some tokens and asks the model to fill them back in — using context from both sides. Bidirectional context is the superpower here. When predicting a masked token, the model sees everything to the left and right, giving it a much richer representation of each position.

The masking protocol uses an 80/10/10 strategy. You randomly select 15% of tokens for prediction, then:

  • 80% of selected tokens → replaced with [MASK]
  • 10% → replaced with a random vocabulary token
  • 10% → left unchanged (but still predicted)
Input:    The cat [MASK] on the [MASK]
Target:          sat           mat

               ┌──────────────────────┐
Context:  ◄─── The cat ??? on the ??? ───►
               └──────────────────────┘
          Bidirectional: each [MASK] sees ALL other tokens

Why the 80/10/10 split? If you always used [MASK], the model would learn to only care when it sees that special token — but [MASK] never appears during fine-tuning or inference. The random replacements force the model to maintain good representations for every token, because any token might be corrupted. The unchanged tokens ensure the model doesn’t assume non-mask tokens are always correct.

The critical tradeoff: MLM produces excellent representations for understanding tasks (classification, NER, question answering) because each token’s representation incorporates the full bidirectional context. But MLM models can’t generate text natively. There’s no left-to-right factorization — you can’t autoregressively sample from a model that was trained to fill in blanks at random positions.

The 15% bottleneck. MLM only computes loss on the 15% of tokens that were selected for masking. That means 85% of the input tokens contribute nothing to the gradient signal in each training step. This is a massive sample inefficiency — and it’s exactly what ELECTRA was designed to fix.

Span Corruption (T5)

Google’s T5 generalized MLM in an elegant way. Instead of masking individual tokens, span corruption masks contiguous spans of variable length. Each span is replaced with a single sentinel token (<extra_id_0>, <extra_id_1>, etc.), and the decoder generates the missing spans:

Original:    "The quick brown fox jumps over the lazy dog"

Corrupted input (encoder sees this):
  "The <extra_id_0> fox jumps <extra_id_1> dog"

Target output (decoder generates this):
  "<extra_id_0> quick brown <extra_id_1> over the lazy"

           spans removed              decoder reconstructs
           ┌──────────┐              ┌──────────────────────┐
Input: The │  <X0>    │ fox jumps   │ <X0> quick brown     │
           └──────────┘              │ <X1> over the lazy   │
                                     └──────────────────────┘

T5 typically corrupts 15% of tokens total, with a mean span length of 3. This has several advantages over token-level MLM:

  • Multi-token reconstruction: The model learns to reconstruct phrases, not just isolated words.
  • Compressed input: A 5-token span becomes 1 sentinel, so the encoder processes shorter sequences — computationally cheaper.
  • Compact targets: The decoder output only contains the missing spans, not the full original sequence.

T5 casts everything as text-to-text. Pretraining: corrupted text → original spans. Fine-tuning: “translate English to German: The house is blue” → “Das Haus ist blau.” Same architecture, same loss, every task. This uniformity is both elegant and practical — no task-specific heads needed.

Replaced Token Detection (ELECTRA)

ELECTRA is the cleverest pretraining objective and possibly the most underrated. The insight: instead of masking tokens and predicting them (which only learns from 15% of tokens), train a discriminator that identifies which tokens have been replaced — learning from every single token in the sequence.

The setup involves two models:

  1. Generator (small): A small MLM model that proposes plausible replacements for masked positions.
  2. Discriminator (main model): Takes the generator’s output and classifies each token as “original” or “replaced.”
Original:      The chef cooked a delicious meal
Masked:        The chef [MASK] a [MASK] meal
Generator:     The chef prepared a wonderful meal    ← plausible fakes
                         ↓            ↓
Discriminator: The  chef  prepared  a  wonderful  meal
               orig orig  REPLACED  orig REPLACED  orig
               ✓    ✓     ✗         ✓    ✗         ✓

Loss: binary cross-entropy on EVERY token position

The key advantage: the discriminator computes loss on all tokens, not just the 15% that were masked. Every position contributes gradient signal. This makes ELECTRA dramatically more sample-efficient — a small ELECTRA model can match the performance of a BERT model trained with 4× the compute.

The generator is intentionally small (typically 1/4 to 1/3 the size of the discriminator). You don’t want the generator to be too good — if it produced perfect replacements, the discrimination task would be impossible. You also don’t want it too bad — if replacements are obviously wrong, the task is trivially easy. The sweet spot is plausible-but-detectable fakes.

After pretraining, you throw away the generator and fine-tune only the discriminator. The discriminator has learned rich token-level representations from the all-token signal, and these transfer beautifully to downstream tasks.

Architecture ↔ Objective Mapping

Different architectures pair naturally with different objectives. This table is worth memorizing:

Architecture Typical Objective Attention Pattern Best For Examples
Encoder-only MLM, Replaced Token Detection Bidirectional (full) Classification, NER, retrieval, embeddings BERT, RoBERTa, ELECTRA, DeBERTa
Decoder-only Causal LM (next-token) Causal (left-to-right) Text generation, chat, code, reasoning GPT, LLaMA, Mistral, PaLM
Encoder-Decoder Span corruption, denoising Bidirectional encoder + causal decoder Translation, summarization, structured generation T5, BART, UL2, Flan-T5

The industry has converged heavily on decoder-only + CLM. Why? Because generation is the killer app, and decoder-only models scale most cleanly — one architecture, one objective, one training pipeline. Encoder-only models are still king for embedding and classification tasks. Encoder-decoder models remain strong for structured sequence-to-sequence problems, but they’re increasingly replaced by decoder-only models that are simply big enough to do everything.

UL2: the best-of-all-worlds attempt. Google’s UL2 (Unified Language Learner) trains a single model with a mixture of objectives — causal LM, prefix LM, and span corruption — distinguished by special mode-switching tokens. The idea is that each objective teaches the model different capabilities. UL2 showed strong results, but the complexity of managing multiple objectives during training has limited adoption.

Part II — Scaling Laws

Scaling laws are what transformed LLM training from expensive gambling into engineering. They’re empirical relationships that predict how a model’s loss decreases as you increase compute, data, or parameters. If you train large models — or even decide whether to — scaling laws are the most important planning tool you have.

Kaplan et al. (2020) — The Original Power Laws

The landmark paper came from OpenAI in January 2020. Kaplan and colleagues trained hundreds of transformer language models ranging from 768 parameters to 1.5B parameters and discovered three remarkably clean power-law relationships. On log-log plots, loss vs. each resource formed a straight line.

The three power laws:

1. Loss vs Parameters (unlimited data/compute):
   L(N) ≈ (Nc / N)^αN      αN ≈ 0.076

2. Loss vs Dataset Size (unlimited model/compute):
   L(D) ≈ (Dc / D)^αD      αD ≈ 0.095

3. Loss vs Compute (optimally allocated):
   L(C) ≈ (Cc / C)^αC      αC ≈ 0.050

  Loss (log)                         Loss (log)
   │                                  │
   │╲                                 │╲
   │  ╲                               │  ╲
   │    ╲  slope ≈ -0.076            │    ╲  slope ≈ -0.095
   │      ╲                           │      ╲
   │        ╲                         │        ╲
   │──────────── Parameters (log)     │──────────── Tokens (log)

The crucial interpretation: every time you 10× your parameters, you get a fixed reduction in log-loss. The returns never stop — they just get more expensive. This is profoundly important. There’s no ceiling where bigger models stop helping (at least, none that anyone has found yet).

Kaplan’s key conclusion: given a fixed compute budget, you should prioritize model size over data. Make the model bigger rather than training on more data. The recommendation was roughly: if you 10× your compute, 5× your parameters but only 2× your data.

This directly shaped the “make it bigger” era. GPT-3 was 175B parameters trained on only 300B tokens — far fewer tokens per parameter than what we’d later learn is optimal. The thinking was: parameters matter most. This turned out to be wrong.

Chinchilla (Hoffmann et al., 2022) — The Correction

In March 2022, DeepMind published a paper that reshaped the field overnight: “Training Compute-Optimal Large Language Models.” The finding was simple and devastating: almost every existing large model was undertrained.

The problem with Kaplan’s analysis was subtle. Their experiments didn’t vary model size and training tokens jointly in a fully balanced way. When Hoffmann’s team ran a proper sweep — over 400 models from 70M to 16B parameters on varying amounts of data, with fixed compute budgets — a different picture emerged.

The core finding: for compute-optimal training, scale parameters and data equally. The optimal ratio is roughly:

Chinchilla Optimal Ratio:

  D ≈ 20 × N

  N = number of parameters
  D = number of training tokens

Given compute budget C ≈ 6 × N × D:
  Optimal N ∝ √C
  Optimal D ∝ √C

Both scale as the square root of compute — equally.

To prove it, DeepMind ran the most convincing experiment possible. They had recently trained Gopher: 280B parameters, 300B tokens. Classic Kaplan-era design — huge model, modest data. Then they trained Chinchilla: only 70B parameters (4× smaller) but on 1.4T tokens (4.7× more data). Same compute budget.

┌─────────────┬──────────────┬──────────────┬─────────────┐
│   Model     │  Parameters  │  Tokens      │  Result     │
├─────────────┼──────────────┼──────────────┼─────────────┤
│  Gopher     │  280B        │  300B        │  Baseline   │
│  Chinchilla │  70B (4× ↓)  │  1.4T (4.7× ↑) │  Better!  │
└─────────────┴──────────────┴──────────────┴─────────────┘

Same compute, 4× smaller model, but dramatically more data → wins.

Chinchilla beat Gopher on nearly every benchmark. A model 4× smaller, trained on more data, was flat-out better. And because it was smaller, it was cheaper to serve at inference time too.

Here’s the practical lookup table for compute-optimal training based on the Chinchilla ratio:

Parameters (N) Optimal Tokens (≈20N) Approx FLOPs (6ND)
400M8B2 × 10¹⁹
1B20B1.2 × 10²⁰
7B140B5.9 × 10²¹
13B260B2 × 10²²
70B1.4T5.9 × 10²³
175B3.5T3.7 × 10²⁴

Notice: GPT-3 (175B parameters, 300B tokens) was trained on roughly 12× fewer tokens than the Chinchilla-optimal amount. It was massively undertrained by modern standards.

Why LLaMA exists. Meta’s LLaMA (2023) was explicitly designed around Chinchilla scaling. The LLaMA-7B model was trained on 1T tokens — roughly 7× the Chinchilla-optimal amount. Why overtrain? Because inference cost depends on model size, not training cost. A smaller model trained on more data than “optimal” is cheaper to deploy. LLaMA proved that you can trade extra training compute for cheaper inference, which matters enormously when you’re serving millions of users.

The Data Wall

Here’s the uncomfortable truth: we’re running out of high-quality text data.

Chinchilla says a 1T-parameter model needs 20T tokens. The entire publicly available internet (Common Crawl, books, code, Wikipedia, etc.) is estimated at roughly 5–15T tokens of reasonable quality. We’re already scraping the barrel. Several implications follow:

  • Data quality > quantity: Deduplication and quality filtering can improve model quality more than adding more low-quality data. The Phi models from Microsoft demonstrated that carefully curated “textbook quality” data can produce surprisingly capable small models.
  • Synthetic data: Models generating training data for other models (or themselves). This works better than you’d expect, especially for code and math, but carries risks of “model collapse” — where distributional biases compound over generations.
  • Multi-epoch training: Training on the same data multiple times. Chinchilla assumed single-epoch. Recent work shows 4–8 epochs can work with appropriate regularization, but returns diminish quickly.
  • Multimodal data: Images, video, and audio contain enormous amounts of information. Training on multimodal data is partly a strategy to escape the text data wall.

Emergent Abilities

One of the most fascinating (and debated) aspects of scaling is emergence — capabilities that appear to materialize suddenly once a model crosses a certain scale threshold, with near-zero performance below and strong performance above.

Examples that have been claimed as emergent:

  • Few-shot arithmetic: Small models can’t do 3-digit addition, but models above ~10B parameters suddenly can.
  • Chain-of-thought reasoning: Prompting “let’s think step by step” only helps above a certain scale.
  • Multi-step logical reasoning: Tasks requiring multiple inference steps show sharp transitions.
  • Code generation: Models below a threshold produce syntactically incorrect code; above it, they write working programs.
  Accuracy
   │
   │                          ●●●●● (large models)
   │                        ●
   │                      ●
   │                    ●         ← sharp transition
   │  ●●●●●●●●●●●●●●●
   │  (small models — near zero)
   │──────────────────────────── Model Scale (log)

   Emergent ability: near-zero below threshold, strong above
The emergence debate. A 2023 paper by Schaeffer et al. argued that emergence might be a mirage caused by metric choice. When you measure with discontinuous metrics (like exact-match accuracy), you see sharp transitions. When you measure with continuous metrics (like token-level log-likelihood), improvement is smooth and predictable. The model is getting better gradually — it’s just that accuracy jumps from 0% to high once the model crosses the “good enough” threshold. This is still debated, but the lesson is: be careful about what you measure and how.

Practical Implications of Scaling Laws

Scaling laws aren’t just academic curiosities. They’re engineering tools. Here’s how practitioners use them:

  1. Train small, predict large. Train a series of small models (100M, 300M, 1B) and plot their losses. Fit a power law. Extrapolate to predict what a 70B model would achieve. This can save millions of dollars by catching bad architectural decisions early.
  2. Compute-optimal budgeting. Given a GPU budget, use the Chinchilla ratio to determine the right model size and data volume. Don’t just train the biggest model you can fit in memory.
  3. Inference-aware training. If you’re deploying to millions of users, a smaller model trained longer (like LLaMA) may be better than a compute-optimal model. Training cost is one-time; inference cost is forever.
  4. Know when to stop scaling. If your loss curve is flattening faster than the power law predicts, something is wrong — bad data, bugs, or you’ve hit a bottleneck. If it’s tracking the predicted curve, your run is healthy.
  5. Architecture comparisons. Compare architectures at small scale. If architecture A has 5% lower loss than B at 1B parameters, the power law says A will still be better at 70B (assuming the exponents are similar). You don’t need to train the full-scale model to decide.
# Quick compute-optimal planning
def chinchilla_optimal(flops_budget):
    """Given a FLOPs budget, compute optimal model size and tokens."""
    # C ≈ 6 * N * D, and D ≈ 20 * N
    # So C ≈ 6 * N * 20N = 120 * N²
    # N = sqrt(C / 120)
    import math
    N = math.sqrt(flops_budget / 120)
    D = 20 * N
    return int(N), int(D)

# Example: 10^22 FLOPs budget
params, tokens = chinchilla_optimal(1e22)
print(f"Optimal: {params/1e9:.1f}B params, {tokens/1e9:.0f}B tokens")
# → Optimal: ~9.1B params, ~182B tokens

Part III — Training Fundamentals

You know the objective (CLM) and you know the budget (Chinchilla). Now comes the hard part: actually training the thing. LLM training is an engineering discipline with a thousand ways to fail. This section covers the practical knowledge you need to plan, execute, and debug a training run.

Data Preparation

Your model is only as good as your data. This isn’t a platitude — it’s a scaling law. Bad data shifts your loss curve up, and no amount of compute compensates.

Deduplication

Duplicate data is shockingly common in web crawls. Common Crawl contains huge amounts of near-duplicate content — boilerplate, scraped pages, copy-pasted text. Training on duplicates wastes compute and can cause memorization (the model regurgitates training data verbatim, which is bad for privacy and generalization).

The standard approach is MinHash + LSH (Locality-Sensitive Hashing):

  1. For each document, generate a set of n-grams (e.g., 5-grams of words).
  2. Compute a MinHash signature: apply k hash functions to the n-gram set, take the minimum hash value for each function. This gives a compact fingerprint.
  3. Use LSH to bucket documents with similar fingerprints.
  4. Within each bucket, do pairwise Jaccard similarity. Remove near-duplicates above a threshold (e.g., 0.8).

This reduces training data by 30–50% while improving model quality. The LLaMA paper reports exact deduplication at the line level plus fuzzy deduplication with MinHash at the document level.

Quality Filtering

Not all text is created equal. A Wikipedia article is more useful training data than a spam page or a cookie consent banner. Common filtering approaches:

  • Perplexity-based filtering: Train a small language model on high-quality data (e.g., Wikipedia). Score web-crawled documents with it. High perplexity = low quality. The CCNet pipeline uses a KenLM model for this.
  • Heuristic rules: Remove documents that are too short, have too many special characters, have low word-to-symbol ratios, or contain too many repeated lines. The RefinedWeb paper lists dozens of such rules.
  • Classifier-based: Train a binary classifier (e.g., fastText) to distinguish high-quality from low-quality text. Score and filter.

Data Mix

Modern LLMs train on a deliberate mixture of data sources. Getting the mix right matters a lot:

Source Typical Share Why It Matters
Web crawl (filtered)60–70%Breadth of knowledge, conversational text
Code (GitHub)5–15%Reasoning, structured output, code generation
Books5–10%Long-form coherent text, deep knowledge
Wikipedia3–5%Factual knowledge, structured writing
Academic papers3–5%Scientific reasoning, technical writing
Conversational data2–5%Dialog, Q&A patterns

There’s growing evidence that code data improves reasoning ability even on non-code tasks. Code is highly structured, logically precise, and forces the model to learn sequential reasoning. This is why models like Codex-descendants (GPT-3.5, GPT-4) are better at reasoning than models trained only on text.

The AdamW Optimizer

SGD doesn’t cut it for transformers. The loss landscape is too complex, with sharp valleys and saddle points. Every serious LLM training run uses AdamW — Adam with decoupled weight decay.

Quick refresher on Adam: it maintains two exponential moving averages per parameter — the first moment (mean of gradients, controlled by β₁) and the second moment (mean of squared gradients, controlled by β₂). This gives each parameter its own adaptive learning rate. Parameters with noisy gradients get smaller effective learning rates; parameters with consistent gradients get larger ones.

The “W” in AdamW is about decoupled weight decay. Standard L2 regularization adds the weight penalty to the loss, which means the penalty interacts with Adam’s adaptive learning rate in weird ways. Decoupled weight decay applies the penalty directly to the weights, bypassing the optimizer’s momentum and scaling. This works better in practice and is now the universal default.

# Typical AdamW settings for LLM training
optimizer = torch.optim.AdamW(
    model.parameters(),
    lr=3e-4,           # peak learning rate (see schedule below)
    betas=(0.9, 0.95), # β1, β2 — note β2=0.95, not the default 0.999
    weight_decay=0.1,  # aggressive weight decay
    eps=1e-8
)

# Why β2=0.95 instead of 0.999?
# 0.999 means the second moment adapts very slowly — it takes ~1000 steps
# to forget old gradient magnitudes. At 0.95, it adapts in ~20 steps.
# LLM training data is highly non-stationary (different domains, topics),
# so you want faster adaptation. Most LLaMA-class models use 0.95.
Why not SGD? SGD works great for vision models (ResNets with SGD+momentum is still competitive). But transformers have attention layers with very different gradient scales across parameters — query/key/value projections, layer norms, embeddings all live in different regimes. Adam’s per-parameter learning rate handles this naturally. SGD would require extremely careful per-layer LR tuning, which doesn’t scale.

Learning Rate Schedules

The learning rate schedule is one of the most critical hyperparameters in LLM training. Get it wrong and your run either diverges or converges too slowly. The standard approach has two phases:

1. Linear Warmup

Start with a very small learning rate and linearly increase to the peak over the first 1–5% of training steps. Why? At initialization, the model’s activations and gradients are unpredictable. A large learning rate at this stage can push the model into a bad region of the loss landscape that it never recovers from. Warmup lets the optimizer “feel out” the landscape before committing to large steps.

2. Cosine Decay

After warmup, decay the learning rate following a cosine curve down to some minimum (typically 10% of the peak, or sometimes 0). Cosine decay is smooth, doesn’t require step schedule tuning, and empirically works as well as or better than step decay for LLMs.

   │   ╱                ╲

   │╱
   │──────────────────────────────── Steps
    warmup          main training
    (1-5%)          (95-99%)

# PyTorch implementation
from torch.optim.lr_scheduler import CosineAnnealingLR, LinearLR, SequentialLR

warmup = LinearLR(optimizer, start_factor=1e-8/3e-4, total_iters=2000)
cosine = CosineAnnealingLR(optimizer, T_max=total_steps - 2000, eta_min=3e-5)
scheduler = SequentialLR(optimizer, [warmup, cosine], milestones=[2000])

Batch Size Scaling

Larger batch sizes let you take advantage of GPU parallelism, but they also change the optimization dynamics. The linear scaling rule (from Goyal et al., 2017): when you multiply the batch size by k, multiply the learning rate by k as well to maintain similar training dynamics.

In practice, LLM training starts with a smaller batch size (to improve gradient signal-to-noise ratio early in training when the model is most sensitive) and ramps up the batch size during training. LLaMA-2 used batch size 4M tokens initially, ramping to 4M tokens throughout.

Mixed Precision Training

Training a 7B model in FP32 requires ~28GB just for the parameters, plus 2–3× that for optimizer states and gradients. At 70B it’s physically impossible on any single GPU. Mixed precision training cuts memory usage roughly in half by storing most tensors in 16-bit floating point while maintaining FP32 where it matters.

BF16 vs FP16

There are two 16-bit formats, and the distinction matters:

Format Sign Exponent Mantissa Dynamic Range Precision
FP16 1 bit 5 bits 10 bits ±65,504 ~3.3 decimal digits
BF16 1 bit 8 bits 7 bits ±3.4 × 10³⁸ (same as FP32!) ~2.4 decimal digits

BF16 won. The reason is simple: LLM training produces activations and gradients with a huge dynamic range. Values can span many orders of magnitude. FP16 has only 5 exponent bits, so it overflows or underflows frequently, requiring careful loss scaling. BF16 has 8 exponent bits — the same dynamic range as FP32 — so values almost never overflow. You lose some mantissa precision, but that turns out to matter much less than dynamic range for gradient-based optimization.

# BF16 mixed precision with PyTorch AMP
from torch.cuda.amp import autocast, GradScaler

# BF16 — no scaler needed (dynamic range handles it)
with autocast(dtype=torch.bfloat16):
    logits = model(input_ids)
    loss = compute_loss(logits, labels)
loss.backward()
optimizer.step()

# FP16 — needs loss scaling to prevent gradient underflow
scaler = GradScaler()
with autocast(dtype=torch.float16):
    logits = model(input_ids)
    loss = compute_loss(logits, labels)
scaler.scale(loss).backward()      # scale up loss to prevent underflow
scaler.step(optimizer)              # unscale gradients, then step
scaler.update()                     # adjust scale factor dynamically

The key insight: with BF16, you can drop the GradScaler entirely. No loss scaling, no scale factor tuning, no NaN headaches from overflow. This is why every modern training framework defaults to BF16 on hardware that supports it (A100, H100, and newer).

Master weights in FP32. Even with BF16 forward/backward passes, you should maintain a FP32 copy of the weights for the optimizer update. Accumulating many small gradient updates in BF16 causes rounding errors that compound over thousands of steps. The optimizer reads BF16 gradients, updates the FP32 master weights, then casts back to BF16 for the next forward pass. This is what “mixed” precision means — you mix precisions strategically, not uniformly.

Training Instabilities

LLM training runs are long (days to months) and expensive. A single instability event can waste enormous compute. Here are the common failure modes and their fixes.

Loss Spikes

The most common instability: loss is decreasing smoothly, then suddenly spikes upward before (hopefully) recovering. Causes include:

  • Bad data batches: A batch with highly unusual content (very long sequences, strange Unicode, code with extreme nesting) can produce extreme gradients. Fix: better data filtering, or skip batches with anomalous loss.
  • Learning rate too high: The optimizer takes a step that’s too large and lands in a bad region. Fix: reduce peak LR, use longer warmup.
  • Numerical instability: FP16 overflow, division by very small numbers in LayerNorm, or softmax overflow. Fix: switch to BF16, add epsilon to denominators, use numerically stable softmax.

Google’s PaLM paper documented a practical approach: when a loss spike occurs, roll back to a checkpoint from ~100 steps before the spike and skip the data batches that triggered it. This is crude but effective.

Gradient Clipping

The universal safety net: clip the global gradient norm to a maximum value (typically 1.0). This prevents any single batch from causing an outsized parameter update.

# Gradient clipping — do this BEFORE optimizer.step()
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
optimizer.step()

Why global norm rather than per-parameter clipping? Because the direction of the gradient update matters. Per-parameter clipping distorts the gradient direction; global norm clipping preserves the direction while capping the magnitude. It’s like saying “go in the same direction, just don’t go as far.”

Other Stability Tricks

  • QK-Norm: Apply LayerNorm to queries and keys before computing attention scores. Prevents attention logits from growing unboundedly as training progresses. Used in PaLM, LLaMA-2.
  • Independent weight decay for embeddings: Embedding layers sometimes benefit from different (often zero) weight decay compared to other layers.
  • z-loss: Add a small penalty on the log-partition function of the output softmax to prevent logits from growing too large. Used in PaLM training.

Distributed Training

No one trains an LLM on a single GPU. Even a 7B model requires distributed training across multiple GPUs. The question is: how do you distribute?

Data Parallelism (DDP)

The simplest approach: replicate the entire model on each GPU, feed different data batches to each replica, compute gradients locally, then all-reduce (average) the gradients across GPUs before the optimizer step. Every GPU ends up with identical weights.

GPU 0: model copy → batch A → grads_A ─┐
GPU 1: model copy → batch B → grads_B ─┼→ all-reduce → avg_grads → update
GPU 2: model copy → batch C → grads_C ─┤
GPU 3: model copy → batch D → grads_D ─┘

Effective batch size = per_gpu_batch × num_gpus

Problem: each GPU must hold the entire model plus optimizer states. For a 7B model with AdamW in FP32, that’s ~28GB (params) + 28GB (gradients) + 56GB (optimizer states: m and v) = ~112GB. That doesn’t fit on a single 80GB A100.

FSDP / ZeRO — Sharded Data Parallelism

The breakthrough insight from DeepSpeed’s ZeRO (Zero Redundancy Optimizer): in standard data parallelism, every GPU stores a full copy of optimizer states, gradients, and parameters — massive redundancy. What if we shard these across GPUs?

ZeRO has three stages:

Stage What’s Sharded Memory per GPU (7B model, 8 GPUs)
ZeRO-1 Optimizer states only ~49GB (down from 112GB)
ZeRO-2 Optimizer states + gradients ~35GB
ZeRO-3 / FSDP Optimizer states + gradients + parameters ~14GB

PyTorch’s FSDP (Fully Sharded Data Parallel) implements ZeRO-3 natively. Each GPU holds only a shard of parameters. During the forward pass, parameters are all-gathered just before they’re needed, used, then discarded. During the backward pass, the same thing happens, plus gradients are reduce-scattered back to their owning GPU.

# FSDP in PyTorch (simplified)
from torch.distributed.fsdp import FullyShardedDataParallel as FSDP
from torch.distributed.fsdp import MixedPrecision

mp_policy = MixedPrecision(
    param_dtype=torch.bfloat16,
    reduce_dtype=torch.bfloat16,
    buffer_dtype=torch.bfloat16,
)

model = FSDP(
    model,
    mixed_precision=mp_policy,
    sharding_strategy=ShardingStrategy.FULL_SHARD,  # ZeRO-3
)
# Now train as normal — FSDP handles communication

Gradient Accumulation

Sometimes you want a very large effective batch size but can’t fit it in GPU memory. Gradient accumulation is the simple trick: run multiple forward/backward passes, accumulating gradients, and only step the optimizer every N micro-batches.

# Gradient accumulation (accumulate_steps = 4)
optimizer.zero_grad()
for i, batch in enumerate(dataloader):
    loss = model(batch) / accumulate_steps   # normalize
    loss.backward()                           # accumulate grads
    if (i + 1) % accumulate_steps == 0:
        torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
        optimizer.step()
        optimizer.zero_grad()

# Effective batch size = micro_batch × accumulate_steps × num_gpus

Tensor Parallelism and Pipeline Parallelism

For the largest models (100B+), even FSDP isn’t enough. You need to split individual layers across GPUs:

  • Tensor Parallelism (TP): Split individual weight matrices across GPUs. For example, a 4-way TP splits each attention head’s Q/K/V projections across 4 GPUs. Each GPU computes a slice of the attention, then they all-reduce. Very fast (within a single node connected by NVLink) but requires tight GPU interconnect.
  • Pipeline Parallelism (PP): Assign different layers to different GPUs. GPU 0 runs layers 1–8, GPU 1 runs layers 9–16, etc. The input flows through GPUs like a pipeline. The problem is “pipeline bubbles” — GPUs sit idle while waiting for activations from earlier stages. Microbatching (1F1B schedule) reduces but doesn’t eliminate these bubbles.
Real-world parallelism strategy (e.g., LLaMA-70B on 256 GPUs):

  TP=8 (within node, NVLink)
  × PP=4 (across nodes)
  × DP=8 (across groups, FSDP)
  = 256 GPUs total

  Each node (8 GPUs): tensor-parallel group
  4 nodes: pipeline stages
  8 such pipeline groups: data-parallel replicas
The 3D parallelism hierarchy. Modern large-scale training combines all three types: TP within a node (requires fast NVLink), PP across a few nodes (tolerates moderate latency), and DP/FSDP across many nodes (tolerates higher latency). The ordering reflects the communication bandwidth requirements: TP needs the most bandwidth, DP the least.

Checkpointing Strategies

Training runs crash. GPUs fail, nodes go down, NFS hiccups. If you don’t checkpoint, you lose everything since the last save.

Training Checkpoints

Save every N steps (e.g., every 1000 steps) and keep the last K checkpoints (e.g., last 5). Each checkpoint includes:

  • Model weights
  • Optimizer states (both moments for AdamW)
  • Learning rate scheduler state
  • Data loader state (which samples you’ve seen)
  • Random number generator states (for exact reproducibility)

For a 7B model with AdamW, a full checkpoint is ~84GB (28GB params + 56GB optimizer). With FSDP, each GPU saves its shard, which is more efficient but requires all GPUs to be present for loading.

Activation Checkpointing (Gradient Checkpointing)

This is a different kind of checkpointing — it’s a memory optimization during training, not a durability strategy.

During the forward pass, you normally store all intermediate activations because they’re needed for the backward pass. For a 7B model with a sequence length of 4096, this can eat 50–100GB of GPU memory. Activation checkpointing trades compute for memory: only store activations at certain layer boundaries, and recompute the rest during the backward pass.

# Activation checkpointing in PyTorch
from torch.utils.checkpoint import checkpoint

class TransformerBlock(nn.Module):
    def forward(self, x):
        # Without activation checkpointing:
        # x = self.attention(x) + x
        # x = self.ffn(x) + x

        # With activation checkpointing:
        x = checkpoint(self.attention, x, use_reentrant=False) + x
        x = checkpoint(self.ffn, x, use_reentrant=False) + x
        return x

# Roughly halves activation memory at the cost of ~33% more compute
# (you do the forward pass twice for checkpointed regions)

The tradeoff: roughly 33% more compute (you recompute forward activations during backward) in exchange for drastically less memory. This is almost always worth it for large models because memory is the bottleneck, not compute.

Some practical notes from running this in the real world:

  • Monitoring: Log loss, gradient norm, learning rate, and throughput (tokens/second) to Weights & Biases or TensorBoard. Loss spikes are your early warning system.
  • Throughput target: On 64 A100-80GB GPUs, expect ~150k–300k tokens/second for a 7B model, depending on your sequence length and parallelism strategy. That’s roughly 13–26B tokens/day, so training on 2T tokens takes 77–154 days.
  • Cost estimate: At ~$2/GPU-hour for A100s, 64 GPUs for 100 days = ~$307k. This is why people care about Chinchilla-optimal training — every wasted FLOP is real money.
  • Debugging runs: Always do a short sanity check first — 100 steps on a small subset — to verify the loss decreases, gradients are reasonable, and checkpointing works. Don’t start a 3-month run without this.
The most expensive bug. The single most costly bug in LLM training is a subtle data issue that doesn’t crash the run but degrades quality. For example: tokenizer mismatch (training on data tokenized with a different vocab), incorrect padding/masking (loss computed on padding tokens), or data leakage (evaluation data in the training set). These bugs can waste weeks of compute before anyone notices the resulting model is worse than expected.
