Pre-training Objectives

Chapter 12: Large Language Models 8 subtopics
TL;DR

Before a language model can chat, summarize, or write code, it must be pre-trained — and the choice of pre-training objective shapes everything the model can later do. Causal language modeling (next-token prediction) powers GPT-style generators. Masked language modeling gives BERT bidirectional context at the cost of generation ability. Span corruption (T5) generalizes masking to variable-length spans. ELECTRA flips the script with a discriminator that learns from every token, not just masked ones. Encoder-only, decoder-only, and encoder-decoder architectures each pair naturally with different objectives. The objective you choose is the single most consequential design decision in LLM pre-training.

Causal Language Modeling (Next-Token Prediction)

Let's start with the objective that powers GPT, LLaMA, Mistral, and every model you've used in a chatbot. Causal language modeling (CLM) trains a model to predict the next token given all previous tokens. That's it. The entire premise of the most capable AI systems on Earth is: read left to right, guess what comes next.

Formally, given a sequence of tokens x₁, x₂, ..., xₙ, the model learns the joint probability by factoring it autoregressively:

P(x₁, x₂, ..., xₙ) = P(x₁) × P(x₂|x₁) × P(x₃|x₁,x₂) × ... × P(xₙ|x₁,...,xₙ₋₁)

At each position t, the model predicts:
  P(xₜ | x₁, x₂, ..., xₜ₋₁)

This left-to-right factorization is called autoregressive because each prediction feeds into the next. The model never looks ahead — position 5 can attend to positions 1–4 but not 6, 7, or 8. This is enforced by a causal attention mask: a triangular matrix that blocks attention to future tokens.

The loss function is cross-entropy computed at every position. For each token in the sequence, the model outputs a probability distribution over the entire vocabulary (often 32k–128k tokens), and you penalize it for assigning low probability to the actual next token. You sum (or average) this loss across all positions in the sequence.

# Simplified CLM training step
logits = model(input_ids)              # shape: (batch, seq_len, vocab_size)
# Shift: predict position t+1 from position t
shift_logits = logits[:, :-1, :]       # all but last position
shift_labels = input_ids[:, 1:]        # all but first token

loss = cross_entropy(
    shift_logits.reshape(-1, vocab_size),
    shift_labels.reshape(-1)
)
loss.backward()

During training, we use a trick called teacher forcing: instead of feeding the model its own previous predictions (which are initially garbage), we feed it the actual ground-truth tokens at every position. This means the model trains on correct context, which stabilizes learning enormously. At inference time, of course, you don't have ground truth — you feed back the model's own outputs, which is why generation quality can degrade over long sequences.

Why does predicting the next token produce such powerful models? Because language is deeply structured. To predict the next word, the model must learn syntax, semantics, facts about the world, reasoning patterns, and even social conventions. The objective is simple; the knowledge required to minimize it is not.

Masked Language Modeling (MLM)

BERT took a different path. Instead of predicting left to right, masked language modeling (MLM) randomly hides some tokens and asks the model to fill them back in — using context from both sides. This is the key advantage: bidirectional context. When predicting a masked token, the model can look left and right, which gives it a much richer understanding of each position.

Here's how BERT's masking works. You randomly select 15% of the tokens in a sequence for prediction. But you don't just replace all of them with a [MASK] token. Instead, BERT uses an 80-10-10 strategy:

  • 80% of selected tokens → replaced with [MASK]
  • 10% → replaced with a random vocabulary token
  • 10% → left unchanged (but still predicted)

Why this mix? If you always used [MASK], the model would learn to only pay attention when it sees that special token — and [MASK] never appears during fine-tuning or inference. The random replacements force the model to maintain good representations of every token, because any token might secretly be corrupted. The unchanged tokens ensure the model doesn't assume non-mask tokens are always correct.

Input:    The cat [MASK] on the [MASK]
Target:          sat           mat

                 ┌──────────────────┐
Context:   ◄──── The cat ??? on the ??? ────►
                 └──────────────────┘
           Bidirectional: each [MASK] sees ALL other tokens

A later refinement is Whole Word Masking. Standard MLM might mask individual subword tokens, breaking a word like "playing" into "play" + "##ing" and only masking "##ing." That's too easy — the model just learns subword assembly. Whole word masking ensures that if any subword token of a word is selected, all subword tokens of that word are masked together. This pushes the model toward genuine semantic understanding.

The critical tradeoff: MLM produces excellent representations for understanding tasks (classification, NER, question answering) because each token's representation is informed by the full context. But MLM models can't easily generate text. There's no natural left-to-right mechanism — you can't autoregressively sample from a model that was trained to fill in blanks at random positions.

Prefix Language Modeling

What if you want a model that can both understand and generate? Prefix language modeling is a hybrid that tries to get the best of both worlds. You split the input into a prefix and a continuation. The prefix portion uses bidirectional attention (every token can attend to every other token in the prefix, like BERT). The continuation uses causal attention (left-to-right only, like GPT).

Prefix (bidirectional)       Continuation (causal)
┌─────────────────────┐     ┌──────────────────────┐
│ Translate English    │ ──► │ Traduire l'anglais   │
│ to French:           │     │ en français:          │
│ ◄───── all-to-all ──►│     │ ──── left-to-right ─► │
└─────────────────────┘     └──────────────────────┘

Models like UniLM and certain configurations of PaLM use this approach. The prefix captures the "understanding" portion — the task description, the source text, the context. The continuation is where the model actually generates. During training, you compute the CLM loss only on the continuation tokens (the prefix is fully visible and doesn't need prediction).

The trade is flexibility for complexity. You must decide where to split prefix from continuation, and the optimal split varies by task. For translation, the prefix is the source sentence. For summarization, it's the document. For open-ended generation, you might use no prefix at all, collapsing back to standard CLM. In practice, pure decoder models (GPT-style) have won the scaling race, but prefix LM remains a useful conceptual bridge.

Span Corruption (T5 Style)

Google's T5 introduced span corruption, which generalizes MLM in an important way. Instead of masking individual tokens, you mask contiguous spans of varying length. Each span is replaced with a single unique sentinel token (<extra_id_0>, <extra_id_1>, etc.), and the model must output the missing spans one after another.

Original:    "The quick brown fox jumps over the lazy dog"

Corrupted input:
  "The <extra_id_0> fox jumps <extra_id_1> dog"

Target output:
  "<extra_id_0> quick brown <extra_id_1> over the lazy"

             spans removed          model predicts them back
             ┌──────────┐          ┌──────────────────────┐
Input:  The  │ <X0>     │ fox ...  │ <X0> quick brown     │
             └──────────┘          │ <X1> over the lazy   │
                                   └──────────────────────┘

This design has several advantages over single-token MLM. First, spans can be any length (T5 uses a mean span length of 3 tokens with 15% of tokens corrupted total), so the model learns to reconstruct multi-token phrases, not just isolated words. Second, the sentinel tokens compress the input — a span of 5 tokens becomes 1 sentinel — so the model processes shorter sequences, which is computationally cheaper. Third, the target is also compact: it only contains the missing spans, not the full original sequence.

T5 casts everything as a text-to-text problem. Pre-training: corrupted text → original spans. Fine-tuning: "translate English to German: The house is blue" → "Das Haus ist blau." Same architecture, same loss, every task. This uniformity is elegant and practical — you don't need task-specific heads or output formats.

Other Denoising Objectives

BART pushed denoising further by combining multiple corruption strategies. The philosophy: if each corruption teaches the model a different skill, using all of them together produces a more robust model. Here are BART's five corruption schemes:

Text infilling replaces a span of tokens with a single [MASK] token, regardless of the span's length. Unlike T5's span corruption, the model must figure out how many tokens are missing. A span of 4 tokens and a span of 1 token both become a single [MASK]. This forces the model to learn about natural phrase lengths.

Sentence permutation shuffles the order of sentences in a document. The model must reconstruct the original order, learning about discourse structure and logical flow. Document rotation picks a random token and rotates the document so it starts at that token. The model must identify the true beginning — learning what makes a coherent opening.

Token deletion randomly removes tokens without leaving any placeholder. The model must figure out that something is missing and what it was. This is harder than masking because there's no signal about where the deletion happened. Finally, token masking is standard MLM included for completeness.

Each corruption targets a different linguistic skill. Infilling teaches phrase structure. Permutation teaches discourse. Rotation teaches document organization. Deletion teaches gap detection. Combined, they produce a model with deep text-reconstruction abilities, which is why BART excels at summarization — summarization is, in a sense, reconstructing the key content from a noisy (verbose) source.

Encoder vs. Decoder vs. Encoder-Decoder Pre-training

The choice of pre-training objective is intimately tied to the model architecture. This pairing isn't arbitrary — each architecture's attention pattern naturally suits certain objectives. Let's map them out.

┌─────────────────────────────────────────────────────────────────────┐
│              ARCHITECTURE – OBJECTIVE – STRENGTH MAP                │
├──────────────────┬──────────────────┬───────────────────────────────┤
│  ENCODER-ONLY    │  DECODER-ONLY    │  ENCODER-DECODER              │
│  (BERT, RoBERTa) │  (GPT, LLaMA)    │  (T5, BART, UL2)             │
├──────────────────┼──────────────────┼───────────────────────────────┤
│  Attention:      │  Attention:      │  Attention:                   │
│  Bidirectional   │  Causal (L→R)    │  Bidir encoder + Causal dec   │
│  (all-to-all)    │  (triangular     │  + Cross-attention between    │
│                  │   mask)          │    encoder and decoder        │
├──────────────────┼──────────────────┼───────────────────────────────┤
│  Objective:      │  Objective:      │  Objective:                   │
│  MLM, ELECTRA    │  CLM (next-token │  Span corruption, denoising,  │
│                  │  prediction)     │  prefix LM                    │
├──────────────────┼──────────────────┼───────────────────────────────┤
│  Best for:       │  Best for:       │  Best for:                    │
│  Classification  │  Text generation │  Seq-to-seq tasks             │
│  NER, QA         │  Code generation │  Translation, summarization   │
│  Retrieval       │  Chat, reasoning │  Question answering           │
├──────────────────┼──────────────────┼───────────────────────────────┤
│  Examples:       │  Examples:       │  Examples:                    │
│  BERT, RoBERTa,  │  GPT-2/3/4,      │  T5, BART, mBART, Flan-T5,   │
│  DeBERTa, ALBERT │  LLaMA, Mistral  │  UL2, PEGASUS                │
└──────────────────┴──────────────────┴───────────────────────────────┘

Encoder-only models process the full input with bidirectional attention, producing rich token-level representations. MLM is their natural objective — it leverages bidirectional context perfectly. The output is a representation per token, not a generated sequence. You slap a classification head on top for downstream tasks. These models dominate when you need to understand text: sentiment analysis, named entity recognition, semantic similarity, retrieval.

Decoder-only models use causal attention and are trained with CLM. Every token can only see tokens before it. This makes them natural generators — at inference time, you just keep sampling the next token. The surprising finding of recent years is that decoder-only models, scaled up enough, also become excellent at understanding tasks. GPT-4 can do NER, classification, and QA — not because its architecture is ideal for those tasks, but because scale compensates. This is why decoder-only models have dominated the LLM landscape.

Encoder-decoder models get the best of both: the encoder processes the full input with bidirectional attention, and the decoder generates the output autoregressively while cross-attending to the encoder's representations. Span corruption and denoising objectives fit naturally here — the encoder sees the corrupted input, the decoder produces the clean output. These models are architecturally ideal for translation, summarization, and any task with a distinct input and output. However, they have roughly twice the parameter count of a decoder-only model at similar capacity, which complicates scaling comparisons.

Replaced Token Detection (ELECTRA)

ELECTRA introduces a clever twist on MLM that dramatically improves sample efficiency. The core insight: in standard MLM, you only learn from the 15% of tokens that were masked. The other 85% contribute nothing to the loss. That's wasteful. ELECTRA makes every single token a training signal.

Here's the setup. You train two models: a small generator (typically a compact MLM model) and a larger discriminator. The generator replaces masked tokens with plausible alternatives. The discriminator then sees the full sequence — with some tokens secretly replaced — and must classify each token as "original" or "replaced."

Step 1: Generator (small MLM) fills in masked tokens
  Input:    "The chef [MASK] a delicious [MASK]"
  Output:   "The chef cooked a delicious cake"
                    ↑ generated           ↑ generated

Step 2: Discriminator classifies EVERY token
  Input:    "The chef cooked a delicious cake"
  Labels:    orig orig replaced orig orig replaced
                       ↑                   ↑
              (original was "made")  (original was "meal")

  Loss: binary cross-entropy at EVERY position
        → 6 training signals instead of 2

The discriminator's job is binary classification at every position: real or fake? This means every token in the sequence contributes to the loss, not just the masked ones. ELECTRA's paper showed that this approach matches BERT's performance with roughly one quarter of the compute. A small ELECTRA model trained on one GPU for 4 days matched a GPT model trained on significantly more resources.

The tradeoff is architectural complexity. You need to train two models, manage the generator's quality (if the generator is too good, the discrimination task becomes impossible; too bad, and it's trivial), and the generator is discarded after pre-training — only the discriminator is used downstream. But the efficiency gains are compelling, especially for resource-constrained settings.

Pre-training Data Engineering

The objective tells the model how to learn. The data determines what it learns. And frankly, data engineering is where most pre-training projects succeed or fail. A perfect objective trained on garbage data produces a garbage model.

Data Deduplication

Web crawls are full of duplicates. The same boilerplate text appears on millions of pages. Exact-match deduplication is straightforward: hash each document (or paragraph) and remove duplicates. But near-duplicates — pages that differ by a date, a username, or a few words — require fuzzy deduplication. The standard tool is MinHash with locality-sensitive hashing (LSH). You convert each document to a set of n-gram shingles, compute a MinHash signature (a compact fingerprint), and use LSH to efficiently find documents whose signatures are suspiciously similar. Documents above a Jaccard similarity threshold (typically 0.8) are treated as duplicates, and all but one copy are removed.

Why does deduplication matter? Repeated data causes the model to memorize those passages — it overfits on duplicates and can regurgitate them verbatim at inference time. Studies have shown that deduplicated datasets produce better models at the same compute budget.

Quality Filtering

Not all text is created equal. A random web crawl includes spam, SEO garbage, auto-generated content, and nonsensical text. Perplexity-based filtering uses a lightweight language model (often a KenLM n-gram model) trained on high-quality text (like Wikipedia). Documents with very high perplexity — meaning the quality model finds them surprising — are likely garbage and get removed. Toxicity filters use classifiers to flag and remove hateful, violent, or otherwise harmful content. Heuristic filters catch documents that are too short, have too many special characters, or contain excessive repetition.

Multilingual Balancing

If you're training a multilingual model, you face a balancing problem. English dominates the web. If you train proportionally, your model will be excellent at English and terrible at Swahili. A common fix is temperature-based sampling: raise the probability of low-resource languages using a temperature parameter, so the model sees them more often than their natural frequency. This hurts English performance slightly but dramatically improves low-resource languages.

Optimizers and Training Stability

AdamW is the default optimizer for LLM pre-training. It decouples weight decay from the adaptive learning rate, which prevents the subtle regularization bug in vanilla Adam. For very large models where Adam's optimizer states blow up memory, Adafactor factorizes the second-moment matrix into row and column factors, cutting optimizer memory by roughly half.

Training instability is the silent killer of LLM pre-training runs. At scale, loss spikes — sudden jumps in the loss curve — are common and can derail a multi-week run. Mitigations include gradient clipping (typically max-norm of 1.0), learning rate warmup (linearly increasing from near-zero over the first few thousand steps), careful initialization (smaller init scales for deeper layers), and sometimes reducing the learning rate mid-training if spikes persist. Some teams use z-loss regularization, which penalizes large logit magnitudes to prevent numerical instability in the softmax. Monitoring is essential — loss spikes caught early can be recovered from by rolling back to a recent checkpoint and lowering the learning rate.

The bottom line: pre-training objectives are only half the story. The data pipeline — deduplication, filtering, balancing — and training stability engineering are equally critical. The most elegant objective in the world won't save you if your data is dirty or your training run diverges at hour 300 of a 1,000-hour job.
