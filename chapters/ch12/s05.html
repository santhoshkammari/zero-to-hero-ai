RAG & Semantic Search

Chapter 12: Large Language Models 11 subtopics
TL;DR

Retrieval-Augmented Generation is the single most important production pattern for LLM systems in 2024–2025. The idea: instead of hoping the model “knows” the answer, you retrieve relevant documents and stuff them into the context window before generating. This kills hallucination, sidesteps knowledge cutoffs, and lets you ground answers in private data. The pipeline looks simple — chunk, embed, index, retrieve, generate — but every stage has sharp edges. Chunking strategy makes or breaks relevance. Embedding model choice determines your semantic ceiling. Vector stores range from in-memory FAISS for prototyping to managed Pinecone for production. Hybrid retrieval (dense + sparse) consistently outperforms either alone. Rerankers are the cheapest accuracy boost you’ll find. And evaluating RAG is its own discipline — retrieval recall, faithfulness, and relevance each need separate metrics. This section covers all of it end-to-end, from embedding theory through production deployment.

Why RAG Exists

LLMs have three fundamental problems that no amount of training data fixes:

1. They hallucinate. Ask GPT-4 for a specific API call and it’ll confidently invent one that looks correct, compiles in your head, and doesn’t exist. The model has no concept of “I don’t know” — it always generates something.

2. Knowledge cutoffs. The model’s world froze when training ended. Ask about last week’s security patch and you get silence or stale information. Fine-tuning to update knowledge is expensive and slow.

3. No access to private data. Your company’s internal docs, Confluence pages, Slack threads, proprietary databases — none of that was in the training set. The model literally cannot answer questions about your domain-specific data.

RAG solves all three in one architecture. Retrieval-Augmented Generation means: before asking the LLM to generate an answer, you first retrieve the relevant documents from an external knowledge base and inject them into the prompt. Now the model isn’t guessing from parametric memory — it’s reading the source material and synthesizing an answer.

Why not just fine-tune? Fine-tuning bakes knowledge into weights. It’s expensive, slow to update, and the model can still hallucinate because it “sort of remembers” rather than directly reading a source. RAG keeps the knowledge external — you can update documents instantly, cite sources, and the model reasons over actual text rather than vague parameterized memories. Fine-tuning is for teaching behavior. RAG is for providing knowledge.

RAG is the #1 architecture pattern deployed in production LLM systems today. If you’re building anything that answers questions over documents, internal knowledge bases, codebases, or any dataset that changes — you’re building RAG.

RAG Architecture

Every RAG system has two pipelines: offline (indexing) and online (querying). Getting this separation clear in your head is essential.

Offline Pipeline (Indexing)

Documents → Load → Chunk → Embed → Store in Vector DB
     │         │        │        │            │
  PDFs,     Parse    Split    Convert      FAISS,
  HTML,     into     into     chunks to    Pinecone,
  Markdown  clean    smaller  vector       Qdrant,
  CSVs...   text     pieces   embeddings   pgvector...

This runs once (or on a schedule). You take your raw documents, parse them into clean text, chunk them into retrieval-sized pieces, convert each chunk to an embedding vector, and store those vectors in a searchable index. The quality of every step here determines your retrieval ceiling.

Online Pipeline (Querying)

User Query → Embed Query → Search Vector DB → Retrieve Top-K
     │                              │                │
     │                        Similarity          Rerank
     │                         search            (optional)
     │                                              │
     └─────────────── Augment Prompt ───────────────┘
                          │
                    LLM generates answer
                    grounded in retrieved docs

At query time: embed the user’s question using the same embedding model, search the vector index for the most similar chunks, optionally rerank them for higher precision, stuff the top results into the prompt alongside the user’s question, and let the LLM generate an answer grounded in real documents.

Same embedding model. Your query embeddings and document embeddings must come from the same model. Mixing models (e.g., embedding docs with OpenAI but querying with Cohere) produces vectors in different spaces. Similarity scores become meaningless. This sounds obvious, but it’s a real production bug.

Embeddings — The Foundation

Embeddings are the bridge between human language and machine math. You take a piece of text — a word, sentence, paragraph — and map it to a dense vector in continuous space where semantic similarity corresponds to geometric proximity. “Dog” and “puppy” end up near each other. “Dog” and “quantum mechanics” end up far apart. That’s the entire game.

Brief History: Word2Vec to Modern Sentence Embeddings

Word2Vec (2013) started the revolution. Mikolov et al. showed that training a shallow neural net on “predict neighboring words” produced vectors where king - man + woman ≈ queen. Beautiful, but limited: one vector per word, no context sensitivity (“bank” gets the same vector whether it’s a river bank or a financial bank).

GloVe (2014) approached the same goal via matrix factorization of word co-occurrence statistics. Similar quality, different math, same limitation.

ELMo (2018) introduced contextualized word embeddings via bidirectional LSTMs. Now “bank” gets different vectors depending on context. Big leap.

BERT (2019) gave us transformer-based contextual embeddings. You could pool BERT’s token outputs to get sentence embeddings, but they were mediocre out-of-the-box — BERT wasn’t trained to make whole-sentence representations useful.

Sentence-BERT / SBERT (2019) fixed this by fine-tuning BERT with a siamese architecture on sentence pairs. Finally — good sentence embeddings. This is where modern embedding models begin.

2023–2024 brought a new wave: models trained specifically for retrieval with contrastive objectives over massive datasets. OpenAI’s text-embedding-3, BGE, E5, GTE, Jina — these are purpose-built retrieval embedding models, and they’re dramatically better than SBERT-era models.

How Embeddings Are Trained: Contrastive Learning

Modern embedding models are trained with contrastive learning. The core idea: given a query, push the embedding of the relevant document (positive) closer, and push irrelevant documents (negatives) farther away.

# Contrastive loss (simplified InfoNCE)
# q = query embedding, p = positive doc embedding
# n_1, n_2, ... = negative doc embeddings

similarity(a, b) = dot(a, b) / (||a|| * ||b||)  # cosine similarity

loss = -log(
    exp(similarity(q, p) / τ) /
    (exp(similarity(q, p) / τ) + Σ exp(similarity(q, nᵢ) / τ))
)

# τ (temperature) controls how "sharp" the distribution is
# Lower τ → harder distinctions → model learns finer-grained similarity

The secret sauce is hard negatives — documents that are topically related but not the correct answer. “What is the capital of France?” paired with a passage about French cuisine is a hard negative. Easy negatives (random documents) teach the model almost nothing after the first few epochs. Mining good hard negatives is often the difference between a mediocre and an excellent embedding model.

Similarity Metrics

Three ways to measure how close two vectors are:

Metric Formula Range When to Use
Cosine Similarity dot(a,b) / (||a|| · ||b||) [-1, 1] Default for most embedding models. Ignores magnitude, compares direction only.
Dot Product Σ aᵢbᵢ (-∞, ∞) When embeddings are already normalized (then equivalent to cosine). Slightly faster.
L2 (Euclidean) Distance √(Σ(aᵢ - bᵢ)²) [0, ∞) Lower = more similar. Useful when magnitude matters. Less common for text.

Rule of thumb: use cosine similarity unless the embedding model documentation says otherwise. Most modern models normalize their output vectors, in which case cosine similarity and dot product give identical rankings.

Code: Generating Embeddings

# === OpenAI embeddings ===
from openai import OpenAI

client = OpenAI()

def get_openai_embeddings(texts: list[str], model="text-embedding-3-small"):
    response = client.embeddings.create(input=texts, model=model)
    return [item.embedding for item in response.data]

query_embedding = get_openai_embeddings(["How does photosynthesis work?"])[0]
doc_embeddings = get_openai_embeddings([
    "Photosynthesis converts sunlight into chemical energy...",
    "The stock market rallied on Tuesday...",
    "Plants use chlorophyll to absorb light energy...",
])

# Cosine similarity
import numpy as np

def cosine_similarity(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

for i, doc_emb in enumerate(doc_embeddings):
    sim = cosine_similarity(query_embedding, doc_emb)
    print(f"Doc {i}: similarity = {sim:.4f}")
# Doc 0: similarity = 0.8234  ← relevant
# Doc 1: similarity = 0.1102  ← irrelevant
# Doc 2: similarity = 0.7891  ← relevant

# === Open-source with sentence-transformers ===
from sentence_transformers import SentenceTransformer

model = SentenceTransformer("BAAI/bge-large-en-v1.5")

# BGE models use a query instruction prefix for asymmetric retrieval
query = "Represent this sentence for searching relevant passages: How does photosynthesis work?"
docs = [
    "Photosynthesis converts sunlight into chemical energy...",
    "The stock market rallied on Tuesday...",
]

query_emb = model.encode(query, normalize_embeddings=True)
doc_embs = model.encode(docs, normalize_embeddings=True)

# With normalized embeddings, dot product == cosine similarity
similarities = doc_embs @ query_emb
print(similarities)  # [0.8156, 0.0987]

Chunking Strategies — This Makes or Breaks Your RAG

Here’s a harsh truth: the most common reason RAG fails isn’t the LLM, isn’t the embedding model, and isn’t the vector store. It’s bad chunking. If your chunks don’t contain coherent, self-contained pieces of information, your retrieval will return garbage and your LLM will hallucinate on top of that garbage.

Chunking is the process of splitting your documents into pieces small enough to embed and retrieve, but large enough to carry meaningful context. Every strategy involves tradeoffs.

Fixed-Size Chunks

Split every N characters or tokens. Simple. Fast. Context-unaware. A chunk boundary might land in the middle of a sentence, splitting a key fact across two chunks where neither chunk alone is useful.

# Fixed-size chunking — naive but sometimes sufficient
def fixed_size_chunks(text: str, chunk_size: int = 500, overlap: int = 50):
    chunks = []
    start = 0
    while start < len(text):
        end = start + chunk_size
        chunks.append(text[start:end])
        start = end - overlap  # overlap to reduce info loss at boundaries
    return chunks

Recursive Character Splitting

This is LangChain’s default and what most people should start with. The idea: try to split on natural boundaries in priority order — first double newlines (paragraphs), then single newlines, then sentences, then words, then characters. You get chunks that respect document structure as much as possible while staying under your size limit.

from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,
    chunk_overlap=64,        # ~12% overlap
    separators=[
        "\n\n",              # paragraph breaks first
        "\n",                # then line breaks
        ". ",                # then sentences
        ", ",                # then clauses
        " ",                 # then words
        ""                   # last resort: character-level
    ],
    length_function=len,     # or use tiktoken for token-based sizing
)

chunks = splitter.split_text(document_text)

Semantic Chunking

Instead of splitting on syntactic boundaries, split on semantic boundaries — where the topic changes. Embed each sentence, compute similarity between consecutive sentences, and split where similarity drops below a threshold. More expensive, but produces chunks that are topically coherent.

from langchain_experimental.text_splitter import SemanticChunker
from langchain_openai import OpenAIEmbeddings

chunker = SemanticChunker(
    OpenAIEmbeddings(),
    breakpoint_threshold_type="percentile",
    breakpoint_threshold_amount=80,  # split at largest drops in similarity
)

semantic_chunks = chunker.split_text(document_text)

Document-Aware Chunking

Respect the structure of the document. Markdown headers, HTML sections, code blocks, table rows — these are natural chunk boundaries that the author already defined for you. Don’t split a code example in half. Don’t cut a table between rows 3 and 4.

from langchain.text_splitter import MarkdownHeaderTextSplitter

headers_to_split_on = [
    ("#", "Header 1"),
    ("##", "Header 2"),
    ("###", "Header 3"),
]

splitter = MarkdownHeaderTextSplitter(headers_to_split_on=headers_to_split_on)
chunks = splitter.split_text(markdown_text)

# Each chunk gets metadata with the header hierarchy:
# chunk.metadata = {"Header 1": "RAG Guide", "Header 2": "Chunking"}

Chunk Size and Overlap

Chunk size: 256–512 tokens is the sweet spot for most use cases. Smaller chunks (128 tokens) are more precise but lose context. Larger chunks (1024+) carry more context but dilute the signal — a 2000-token chunk might have one relevant sentence buried in irrelevant text, and retrieval will surface the whole thing.

Overlap: 10–20% of chunk size. Overlap ensures that if a key fact sits near a chunk boundary, it appears in both adjacent chunks. Without overlap, you get “boundary blindness” — questions about information at chunk edges fail silently.

Chunking Comparison

Strategy Best For Pros Cons
Fixed-size Uniform text (transcripts, logs) Simple, fast, predictable sizes Breaks mid-sentence, loses context
Recursive character General-purpose (start here) Respects paragraphs/sentences, configurable Still syntactic, not semantic
Semantic Mixed-topic documents Topically coherent chunks Slow (requires embedding each sentence), variable size
Document-aware Structured docs (Markdown, HTML, code) Respects author’s structure, preserves metadata Requires format-specific parsers, sections may exceed size limits
The #1 chunking mistake: using token counts but measuring in characters (or vice versa). A “512 character” chunk is roughly 100–130 tokens — far smaller than you probably intended. Always be explicit about your unit. Use tiktoken to count tokens accurately if your embedding model has a token limit.

Vector Stores — Where Embeddings Live

You have a million 1536-dimensional vectors. A user query comes in, you embed it, and you need the 10 most similar vectors. Brute-force search — computing cosine similarity against all million vectors — takes ~100ms for a million vectors. Sounds fine, right? Now scale to 100 million vectors. Now it’s 10 seconds per query. Unacceptable.

This is where Approximate Nearest Neighbor (ANN) algorithms come in. They trade a tiny amount of accuracy for massive speed gains — O(log N) instead of O(N).

HNSW: The Default Algorithm

Hierarchical Navigable Small World graphs are what most vector databases use under the hood. The intuition: build a multi-layer graph where the top layer has few, widely-spaced nodes (for coarse navigation) and the bottom layer has all nodes (for fine-grained search). Searching starts at the top and “zooms in” layer by layer — like finding a city on a map by first looking at the continent, then the country, then the region.

HNSW Search — Conceptual View:

Layer 2 (sparse):   A ─────────── B ─────────── C
                         (big jumps, coarse search)

Layer 1 (medium):   A ── D ── E ── B ── F ── G ── C
                         (medium jumps)

Layer 0 (all nodes): A D H D I E J B K F L G M C
                         (fine-grained, exact neighbors)

Search: start at Layer 2, greedily walk to nearest node,
        drop to Layer 1, refine, drop to Layer 0, find exact neighbors.

Complexity: O(log N) search time
Tradeoff:   Higher M (connections per node) → better recall, more memory
            Higher ef_construction → better graph quality, slower build

IVF + Product Quantization: Billion-Scale

For truly massive datasets (billions of vectors), HNSW’s memory requirements become prohibitive. IVF (Inverted File Index) partitions the space into clusters (Voronoi cells), and at query time you only search the nearest clusters. Product Quantization (PQ) compresses each vector by splitting it into sub-vectors and quantizing each independently, reducing memory by 10–100x.

IVF+PQ is what powers billion-scale systems at companies like Spotify and Meta. The recall is lower than HNSW, but it fits in memory.

Code: FAISS Index

import faiss
import numpy as np

# Create embeddings (normally from your embedding model)
dimension = 1536
num_vectors = 100_000
embeddings = np.random.randn(num_vectors, dimension).astype("float32")

# Normalize for cosine similarity (FAISS inner product on normalized = cosine)
faiss.normalize_L2(embeddings)

# Build HNSW index
index = faiss.IndexHNSWFlat(dimension, 32)  # 32 = M (connections per node)
index.hnsw.efConstruction = 200              # higher = better quality, slower build
index.add(embeddings)

# Search
query = np.random.randn(1, dimension).astype("float32")
faiss.normalize_L2(query)

index.hnsw.efSearch = 128  # higher = more accurate, slower search
distances, indices = index.search(query, k=10)

print(f"Top 10 nearest neighbor indices: {indices[0]}")
print(f"Distances (inner product): {distances[0]}")

Retrieval: Dense, Sparse, and Hybrid

Retrieval is where most people stop thinking. “Embed the query, find the nearest vectors, done.” But there are three fundamentally different retrieval paradigms, and the best production systems combine them.

Sparse Retrieval: BM25 and TF-IDF

BM25 is the workhorse of traditional information retrieval. It represents documents and queries as sparse vectors (one dimension per vocabulary term), weights terms by frequency and inverse document frequency, and ranks documents by lexical overlap. It’s been the backbone of search engines for 30 years and it’s still really good at keyword matching.

Strengths: exact keyword matching, rare term handling, zero training needed, fast, interpretable. If the user searches for “error code E-4021”, BM25 will find it instantly. A dense model might map it to vaguely related error codes because it doesn’t understand that the exact string matters.

Weakness: no semantic understanding. “How to fix a broken pipe” won’t match a document about “SIGPIPE error handling” because the words don’t overlap.

Dense Retrieval: Embedding Similarity

Everything we covered in the embeddings section. Encode both query and documents as dense vectors, measure cosine similarity. Captures meaning regardless of exact words. “How to fix a broken pipe” will match “handling SIGPIPE errors in Python” because the embeddings capture the semantic overlap.

Weakness: can miss exact matches. If a user searches for a specific product ID or error code, dense retrieval might return semantically similar but wrong results. It also struggles with negation and quantitative comparisons.

Hybrid Retrieval: Best of Both Worlds

Combine sparse and dense. Run both retrievers, merge the results. The standard approach is Reciprocal Rank Fusion (RRF):

# Reciprocal Rank Fusion (RRF)
# Combines rankings from multiple retrievers without needing score calibration

def reciprocal_rank_fusion(rankings: list[list[str]], k: int = 60) -> list[str]:
    """
    rankings: list of ranked document ID lists from different retrievers
    k: constant to prevent high-ranked docs from dominating (default 60)
    Returns: fused ranking as list of document IDs
    """
    scores = {}
    for ranking in rankings:
        for rank, doc_id in enumerate(ranking):
            if doc_id not in scores:
                scores[doc_id] = 0.0
            scores[doc_id] += 1.0 / (k + rank + 1)

    # Sort by fused score, descending
    fused = sorted(scores.keys(), key=lambda d: scores[d], reverse=True)
    return fused

# Example: combine BM25 and dense retrieval results
bm25_results = ["doc_3", "doc_7", "doc_1", "doc_15", "doc_8"]
dense_results = ["doc_7", "doc_3", "doc_22", "doc_1", "doc_5"]

hybrid_results = reciprocal_rank_fusion([bm25_results, dense_results])
# doc_3 and doc_7 appear high in both → ranked highest
# doc_22 (dense only) and doc_15 (BM25 only) → ranked lower

When Dense Beats Sparse (and Vice Versa)

Scenario Winner Why
User asks a natural-language question Dense Semantic understanding captures intent beyond keywords
User searches for exact ID, code, or name Sparse Exact lexical matching — embeddings don’t respect string identity
Domain with specialized jargon Sparse (unless fine-tuned) General embedding models don’t know your acronyms
Multilingual queries Dense Multilingual embeddings bridge language gaps; BM25 can’t
Mixed use cases Hybrid When you don’t know what the user will search for, hedge with both
Production reality: Hybrid retrieval with RRF consistently outperforms either dense or sparse alone on real-world benchmarks. If you’re building a production system and can only pick one architecture, pick hybrid. The implementation cost is minimal (you’re just running two retrievers and merging results), and the quality uplift is meaningful.

Reranking — The Accuracy Booster

Here’s the retrieval dilemma: you want high recall (don’t miss relevant docs) and high precision (don’t include irrelevant ones). Embedding search is great for recall but mediocre for precision. The solution: retrieve broadly, then rerank precisely.

Retrieve the top 50–100 candidates cheaply with embedding similarity (fast bi-encoder), then rerank them with a more powerful model to get the best 5–10.

Bi-Encoders vs Cross-Encoders

Bi-encoders (what embedding models are) encode query and document independently. Fast because you can pre-compute document embeddings. But the query and document never “see” each other during encoding — the model can’t do token-level cross-attention between them.

Cross-encoders take the query and document as a single concatenated input: [query] [SEP] [document]. Full cross-attention between every query token and every document token. Much more accurate for relevance scoring, but you can’t pre-compute anything — you must run the model for every (query, document) pair at query time. That’s why you only rerank a small candidate set.

Bi-encoder (embedding model):
  Query  → Encoder → query_vec   ──┐
                                    ├── cosine_similarity → score
  Doc    → Encoder → doc_vec     ──┘
  ✓ Pre-compute doc embeddings. O(1) per query-doc pair.
  ✗ No cross-attention between query and doc.

Cross-encoder (reranker):
  [Query + Doc] → Encoder → relevance_score
  ✓ Full cross-attention. Much more accurate.
  ✗ Must run model for EVERY query-doc pair. O(N) per query.

Reranker Models

Model Type Open Source Notes

BGE-reranker-v2-m3 Cross-encoder Yes Strong open-source. Multilingual.

Code: Reranking Pipeline

# Reranking with a cross-encoder using sentence-transformers
from sentence_transformers import CrossEncoder

reranker = CrossEncoder("BAAI/bge-reranker-v2-m3", max_length=512)

query = "How do I handle database connection timeouts?"
candidates = [
    "Configure the connection pool timeout parameter in your database driver settings...",
    "Database normalization involves organizing data to reduce redundancy...",
    "When a connection times out, the client should implement retry logic with...",
    "PostgreSQL supports multiple authentication methods including...",
    "Connection timeout handling requires setting both connect_timeout and...",
]

# Score each (query, candidate) pair
pairs = [(query, doc) for doc in candidates]
scores = reranker.predict(pairs)

# Sort by relevance score
ranked = sorted(zip(candidates, scores), key=lambda x: x[1], reverse=True)
for doc, score in ranked:
    print(f"[{score:.4f}] {doc[:70]}...")

# [0.9632] Configure the connection pool timeout parameter in your databas...
# [0.9215] When a connection times out, the client should implement retry...
# [0.0234] PostgreSQL supports multiple authentication methods including...
# [0.0018] Database normalization involves organizing data to reduce redun...
The latency tradeoff: Reranking adds 50–200ms per query (depending on candidate count and model size). For most production applications, this is acceptable. If you’re in a sub-100ms latency budget, consider ColBERT (late interaction model) — it pre-computes document token embeddings and only does lightweight interaction at query time. It’s the sweet spot between bi-encoder speed and cross-encoder accuracy.

Complete RAG Pipeline — End to End

Enough components. Let’s wire the whole thing together. Here’s a production-style RAG pipeline in raw Python — no framework magic, every step explicit so you understand what’s happening.

import os
import faiss
import numpy as np
from openai import OpenAI
from sentence_transformers import CrossEncoder
from langchain.text_splitter import RecursiveCharacterTextSplitter

client = OpenAI()

# ── Step 1: Load documents ──────────────────────────────────────────
def load_documents(directory: str) -> list[dict]:
    """Load text files from a directory."""
    docs = []
    for filename in os.listdir(directory):
        if filename.endswith(".txt") or filename.endswith(".md"):
            with open(os.path.join(directory, filename)) as f:
                docs.append({
                    "content": f.read(),
                    "source": filename,
                })
    return docs

# ── Step 2: Chunk ───────────────────────────────────────────────────
def chunk_documents(docs: list[dict], chunk_size=512, chunk_overlap=64):
    """Split documents into retrieval-sized chunks with metadata."""
    splitter = RecursiveCharacterTextSplitter(
        chunk_size=chunk_size,
        chunk_overlap=chunk_overlap,
    )
    chunks = []
    for doc in docs:
        splits = splitter.split_text(doc["content"])
        for i, text in enumerate(splits):
            chunks.append({
                "text": text,
                "source": doc["source"],
                "chunk_index": i,
            })
    return chunks

# ── Step 3: Embed ──────────────────────────────────────────────────
def embed_texts(texts: list[str], model="text-embedding-3-small") -> np.ndarray:
    """Embed a list of texts using OpenAI. Returns normalized numpy array."""
    response = client.embeddings.create(input=texts, model=model)
    embeddings = np.array([item.embedding for item in response.data], dtype="float32")
    faiss.normalize_L2(embeddings)
    return embeddings

# ── Step 4: Index ──────────────────────────────────────────────────
def build_index(embeddings: np.ndarray) -> faiss.IndexHNSWFlat:
    """Build a FAISS HNSW index."""
    dim = embeddings.shape[1]
    index = faiss.IndexHNSWFlat(dim, 32)
    index.hnsw.efConstruction = 200
    index.add(embeddings)
    return index

# ── Step 5: Retrieve ───────────────────────────────────────────────
def retrieve(query: str, index, chunks, k=20):
    """Embed query and retrieve top-k chunks."""
    query_emb = embed_texts([query])
    index.hnsw.efSearch = 128
    distances, indices = index.search(query_emb, k)
    results = []
    for dist, idx in zip(distances[0], indices[0]):
        if idx >= 0:  # FAISS returns -1 for missing results
            results.append({**chunks[idx], "score": float(dist)})
    return results

# ── Step 6: Rerank ─────────────────────────────────────────────────
reranker = CrossEncoder("BAAI/bge-reranker-v2-m3", max_length=512)

def rerank(query: str, candidates: list[dict], top_k=5) -> list[dict]:
    """Rerank candidates using a cross-encoder."""
    pairs = [(query, c["text"]) for c in candidates]
    scores = reranker.predict(pairs)
    for c, s in zip(candidates, scores):
        c["rerank_score"] = float(s)
    ranked = sorted(candidates, key=lambda x: x["rerank_score"], reverse=True)
    return ranked[:top_k]

# ── Step 7: Generate ───────────────────────────────────────────────
RAG_PROMPT = """Answer the user's question based ONLY on the provided context.
If the context doesn't contain enough information, say "I don't have enough
information to answer this question." Do not make up information.

Cite your sources using [Source: filename] after each claim.

Context:
{context}

Question: {question}

Answer:"""

def generate(query: str, context_chunks: list[dict]) -> str:
    """Generate an answer grounded in retrieved context."""
    context = "\n\n---\n\n".join(
        f"[Source: {c['source']}]\n{c['text']}" for c in context_chunks
    )
    prompt = RAG_PROMPT.format(context=context, question=query)

    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[{"role": "user", "content": prompt}],
        temperature=0.1,  # low temp for factual accuracy
    )
    return response.choices[0].message.content

# ── Full Pipeline ──────────────────────────────────────────────────
def rag_pipeline(query: str, index, chunks):
    """End-to-end: retrieve → rerank → generate."""
    candidates = retrieve(query, index, chunks, k=20)
    top_chunks = rerank(query, candidates, top_k=5)
    answer = generate(query, top_chunks)
    return answer, top_chunks

# ── Usage ──────────────────────────────────────────────────────────
docs = load_documents("./knowledge_base")
chunks = chunk_documents(docs)
embeddings = embed_texts([c["text"] for c in chunks])
index = build_index(embeddings)

answer, sources = rag_pipeline("How do I configure SSL for PostgreSQL?", index, chunks)
print(answer)
print("\nSources used:")
for s in sources:
    print(f"  - {s['source']} (chunk {s['chunk_index']}, score: {s['rerank_score']:.4f})")
The RAG prompt matters enormously. Three things your prompt must do: (1) instruct the model to use only the provided context, (2) tell it to admit when it doesn’t know, and (3) require source citations. Without (1), the model will happily hallucinate using its parametric memory. Without (2), it invents answers when the context is insufficient. Without (3), you can’t debug retrieval failures. These three instructions are non-negotiable in production RAG.

RAG Evaluation — How to Know If Your RAG Works

Building RAG is easy. Knowing whether it works is hard. You need to evaluate both the retrieval stage and the generation stage independently, because a failure in either one sinks the whole system.

Retrieval Metrics

Metric What It Measures Formula (Simplified)
Recall@K Of all relevant docs, how many are in the top K? |relevant ∩ retrieved@K| / |relevant|
Precision@K Of the top K retrieved, how many are relevant? |relevant ∩ retrieved@K| / K
MRR (Mean Reciprocal Rank) How high is the first relevant result? 1/rank_of_first_relevant, averaged over queries
NDCG@K Are the most relevant results ranked highest? Discounted cumulative gain, normalized. Considers graded relevance.

Recall@K is king for RAG. If the relevant document isn’t in your retrieved set, the LLM can’t use it — no amount of prompt engineering fixes a retrieval miss. Target recall@10 ≥ 0.9 for production systems. If you’re below 0.8, fix retrieval before touching anything else.

Generation Metrics

Even with perfect retrieval, the LLM can still mess up:

Faithfulness: Does the generated answer accurately reflect the retrieved documents? Or did the model hallucinate additional “facts” not present in the context? This is the most critical generation metric.

Answer Relevance: Does the answer actually address the user’s question? A faithful answer that doesn’t address the query is useless.

Context Relevance: Were the retrieved documents actually relevant to the question? Low context relevance means your retrieval is surfacing noise.

RAGAS Framework

RAGAS (Retrieval Augmented Generation Assessment) is the standard open-source framework for RAG evaluation. It uses LLMs as judges to compute four metrics:

# RAGAS evaluation — the 4 core metrics
from ragas import evaluate
from ragas.metrics import (
    faithfulness,        # Is the answer grounded in the context?
    answer_relevancy,    # Does the answer address the question?
    context_precision,   # Are retrieved docs relevant to the question?
    context_recall,      # Do retrieved docs cover the ground truth?
)
from datasets import Dataset

# Prepare evaluation data
eval_data = {
    "question": ["How do I configure SSL for PostgreSQL?"],
    "answer": ["To configure SSL, edit postgresql.conf and set ssl = on..."],
    "contexts": [["PostgreSQL SSL can be enabled by setting ssl = on in postgresql.conf..."]],
    "ground_truth": ["Enable SSL by setting ssl = on in postgresql.conf, provide server.crt..."],
}

dataset = Dataset.from_dict(eval_data)

results = evaluate(
    dataset,
    metrics=[faithfulness, answer_relevancy, context_precision, context_recall],
)
print(results)
# {'faithfulness': 0.95, 'answer_relevancy': 0.88,
#  'context_precision': 0.90, 'context_recall': 0.85}
The human evaluation loop: LLM-as-judge metrics are useful for automated testing, but they’re not ground truth. Set up a human eval pipeline: sample 50–100 queries per week, have domain experts rate retrieval relevance and answer quality on a 1–5 scale. Track these scores over time. Automated metrics tell you when something breaks; human evaluation tells you how well you’re actually doing.

Advanced RAG Patterns

Vanilla RAG (single retrieval → single generation) works surprisingly well. But there’s a rich ecosystem of techniques that push quality further for specific failure modes.

Multi-Step RAG / Iterative Retrieval

Complex questions often can’t be answered with a single retrieval. “Compare the security models of PostgreSQL and MongoDB” requires information about both databases — a single query might only surface docs about one. Multi-step RAG decomposes the question, retrieves for each sub-question, and synthesizes:

# Multi-step RAG — decompose, retrieve per step, synthesize
def multi_step_rag(question: str, index, chunks):
    # Step 1: LLM decomposes the question
    decomposition_prompt = f"""Break this question into 2-4 simpler sub-questions
that can each be answered independently:

Question: {question}

Return as a JSON list of strings."""

    sub_questions = llm_call(decomposition_prompt)  # ["What is PostgreSQL's security model?",
                                                     #  "What is MongoDB's security model?"]

    # Step 2: Retrieve and answer each sub-question
    sub_answers = []
    for sq in sub_questions:
        candidates = retrieve(sq, index, chunks, k=10)
        top = rerank(sq, candidates, top_k=3)
        answer = generate(sq, top)
        sub_answers.append({"question": sq, "answer": answer})

    # Step 3: Synthesize into final answer
    synthesis_prompt = f"""Based on these sub-answers, provide a comprehensive
answer to the original question: {question}

Sub-answers:
{format_sub_answers(sub_answers)}"""

    return llm_call(synthesis_prompt)

HyDE — Hypothetical Document Embeddings

The query–document asymmetry problem: user queries are short and vague (“SSL postgres”), but documents are long and detailed. Their embeddings live in different regions of the space. HyDE bridges this gap: ask the LLM to generate a hypothetical answer (even if it’s wrong), then embed that instead of the raw query. The hypothetical answer looks more like a document, so its embedding is closer to the actual relevant documents.

# HyDE: embed a hypothetical answer instead of the raw query
def hyde_retrieve(query: str, index, chunks, k=20):
    # Generate a hypothetical answer (may contain hallucinations — that's OK)
    hypothesis = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{
            "role": "user",
            "content": f"Write a short paragraph answering: {query}"
        }],
        temperature=0.7,
    ).choices[0].message.content

    # Embed the hypothesis (not the query)
    hypo_emb = embed_texts([hypothesis])
    distances, indices = index.search(hypo_emb, k)

    return [chunks[idx] for idx in indices[0] if idx >= 0]
When HyDE helps: Short, vague queries where the embedding model struggles to bridge the query–document gap. Doesn’t help (and can hurt) for precise, well-formed questions or keyword searches. It adds one LLM call of latency, so only use it when vanilla retrieval recall is poor.

Parent Document Retrieval

The chunking dilemma: small chunks are precise for retrieval, but lack context for generation. Large chunks have context but are imprecise. Solution: embed small chunks for retrieval, but return the parent document (or a larger chunk) for generation. You index 256-token chunks but when one is retrieved, you pass the full 2048-token parent section to the LLM.

# Parent document retrieval strategy
# Index: small chunks (256 tokens) → precise retrieval
# Return: parent section (2048 tokens) → rich context for generation

def parent_document_retrieval(query, index, small_chunks, parent_map):
    """
    small_chunks: list of small chunks used for embedding/indexing
    parent_map: dict mapping small_chunk_id → parent_chunk
    """
    candidates = retrieve(query, index, small_chunks, k=20)
    reranked = rerank(query, candidates, top_k=5)

    # Map back to parent documents (deduplicate)
    seen_parents = set()
    parent_chunks = []
    for chunk in reranked:
        parent_id = chunk["parent_id"]
        if parent_id not in seen_parents:
            seen_parents.add(parent_id)
            parent_chunks.append(parent_map[parent_id])

    return generate(query, parent_chunks)

Metadata Filtering

Not all retrieval is purely semantic. If the user asks about “Python SSL configuration”, you should filter documents to only those tagged with language: python before doing semantic search. This is a simple metadata pre-filter that dramatically improves precision.

# Most vector DBs support metadata filtering
# Qdrant example:
from qdrant_client import QdrantClient
from qdrant_client.models import Filter, FieldCondition, MatchValue

client = QdrantClient("localhost", port=6333)

results = client.search(
    collection_name="docs",
    query_vector=query_embedding,
    query_filter=Filter(
        must=[
            FieldCondition(key="language", match=MatchValue(value="python")),
            FieldCondition(key="doc_type", match=MatchValue(value="tutorial")),
        ]
    ),
    limit=10,
)

Long-Context Models vs RAG

Gemini 1.5 can handle 1M tokens. GPT-4o supports 128K. Why not just dump all your documents into the context window and skip RAG entirely?

Factor Long Context RAG
Cost Expensive (pay per token for entire context every query) Cheap (only embed once, retrieve K small chunks per query)
Latency Slow (processing 100K+ tokens takes seconds) Fast (retrieve + generate with small context)
Accuracy on needle-in-haystack Degrades with context length (“lost in the middle” problem) Consistent (retrieval narrows to relevant chunks)
Data freshness Must reload entire context each time Update index incrementally
Simplicity Simpler (no indexing pipeline) More complex (chunking, embedding, vector store)
Scale Capped at context window size Scales to millions of documents

Rule of thumb: If your total corpus fits in ~50K tokens and doesn’t change often, long context might be simpler. Beyond that, or if cost/latency matter, RAG wins. For most production systems, the answer is RAG — often with long-context models used for the generation step (stuff more retrieved chunks in) rather than replacing retrieval entirely.

Production Considerations

Getting RAG to work in a notebook is a weekend project. Getting it to work reliably in production is a multi-month effort. Here’s what you need to think about.

Embedding Model Fine-Tuning

Off-the-shelf embedding models are trained on general web data. Your domain has different vocabulary, different notions of “similarity,” and different document structures. Fine-tuning the embedding model on your domain data is the highest-leverage improvement you can make.

# Fine-tune embeddings with sentence-transformers
from sentence_transformers import SentenceTransformer, InputExample, losses
from torch.utils.data import DataLoader

model = SentenceTransformer("BAAI/bge-base-en-v1.5")

# Training data: (query, positive_passage) pairs from your domain
train_examples = [
    InputExample(texts=["SSL handshake timeout", "Configure ssl_timeout in pg_hba.conf..."]),
    InputExample(texts=["connection refused error", "Check listen_addresses and pg_hba.conf..."]),
    # ... hundreds to thousands of domain-specific pairs
]

train_dataloader = DataLoader(train_examples, shuffle=True, batch_size=16)
train_loss = losses.MultipleNegativesRankingLoss(model)

model.fit(
    train_objectives=[(train_dataloader, train_loss)],
    epochs=3,
    warmup_steps=100,
    output_path="./fine-tuned-embeddings",
)
How many training pairs? You can see improvements with as few as 100–500 high-quality (query, relevant passage) pairs. 1000+ is better. Mine them from search logs, user feedback, or have domain experts create them. Quality matters far more than quantity.

Index Updates and Freshness

Documents change. New docs are added, old ones are updated or deleted. Your index needs to stay in sync. Options:

Full rebuild: Re-embed and re-index everything on a schedule (daily, hourly). Simple, but expensive for large corpora. Good for <100K documents.

Incremental updates: Track document changes, embed only new/modified documents, add to index, remove deleted documents. More complex but necessary at scale. Most vector databases support upsert and delete operations.

Versioning: Keep the old index live while building a new one. Swap atomically when the new index is ready. Prevents serving stale or partial indexes during rebuilds.

Cost Optimization

RAG costs add up. The main cost drivers:

Embedding API calls: text-embedding-3-small costs ~$0.02 per 1M tokens. For 100K documents averaging 500 tokens each, that’s $1 to embed your entire corpus. Re-embedding on every minor update is wasteful — track hashes to skip unchanged documents.

LLM generation: The big cost. GPT-4o at $2.50/1M input tokens — if you stuff 5 chunks of 512 tokens each, that’s ~2,500 tokens per query just for context. At 10K queries/day, that’s ~$6.25/day for context alone. Consider using GPT-4o-mini ($0.15/1M) for simpler queries and routing hard queries to the full model.

Vector storage: Pinecone charges by vector count and queries. At scale, self-hosted options (Qdrant on your own infra) become cheaper. pgvector is “free” if you already run Postgres.

Monitoring Retrieval Quality

The silent failure mode of RAG: retrieval quality degrades slowly, answers get slightly worse, and nobody notices until a customer complains. What to monitor:

Retrieval score distributions: Track the average similarity score of retrieved chunks over time. A sudden drop means your embedding model or index has a problem.

Empty retrieval rate: How often does the retriever return no relevant results (all scores below threshold)? Rising empty retrieval rate means your knowledge base has coverage gaps.

“I don’t know” rate: If your RAG prompt instructs the model to admit uncertainty, track how often it does. Increasing “I don’t know” responses signal retrieval problems.

User feedback signals: Thumbs up/down, follow-up questions (user wasn’t satisfied), copy events (user found the answer useful). These are your ground truth.

Failure Modes and Debugging

Symptom Likely Cause Fix
Irrelevant chunks retrieved Bad chunking or poor embedding model Inspect chunks manually. Try semantic chunking. Fine-tune embeddings.
Relevant doc exists but isn’t retrieved Chunk too large (info diluted) or query-doc vocabulary mismatch Smaller chunks. Add hybrid retrieval (BM25). Try HyDE.
Retrieved docs are relevant but answer is wrong LLM ignoring context or hallucinating Strengthen RAG prompt. Lower temperature. Use a more capable model.
Answer is correct but missing key details Top-K too small or reranker filtering out important chunks Increase K. Inspect reranker scores. Try without reranking.
Good for common queries, bad for rare ones Embedding model doesn’t understand domain terminology Fine-tune embeddings. Add keyword search. Expand the knowledge base.
Performance degrades over time Stale index, new docs not indexed, drift in query patterns Set up index refresh schedule. Monitor retrieval metrics weekly.
Debug retrieval before generation. When a RAG system gives bad answers, 80% of the time the problem is retrieval, not generation. Before tweaking your prompt or switching LLM models, inspect what chunks are actually being retrieved. Print them out. Read them. Ask yourself: “If I were given only these chunks, could I answer this question correctly?” If not, the LLM can’t either. Fix retrieval first.
