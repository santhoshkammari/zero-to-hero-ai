LLM Evaluation

Chapter 12: Large Language Models 8 subtopics
TL;DR

Evaluating LLMs is fundamentally harder than evaluating classifiers — there’s no single number that tells you if a summary is “good.” Perplexity measures language modeling ability but not usefulness. Classical NLP metrics (BLEU, ROUGE, BERTScore) work for narrow tasks but break on open-ended generation. Benchmarks like MMLU and HumanEval provide standardized comparisons, but suffer from data contamination and saturation. The practical workhorse is LLM-as-judge — using a strong model (GPT-4, Claude) to evaluate weaker ones — which correlates surprisingly well with human ratings when done right. Human evaluation remains the gold standard for safety-critical and subjective tasks but is expensive and slow. The winning strategy: build your own eval set, iterate with LLM-as-judge, validate with humans on edge cases, and treat every production failure as a new test case.

Why LLM Evaluation Is Hard

In traditional ML, evaluation is (relatively) straightforward. You have labels. You have predictions. You compute accuracy, precision, recall, F1, AUC — pick your metric, run the math, get a number. A classifier that scores 0.94 F1 is better than one scoring 0.91. Done.

LLMs don’t work that way. When you ask a model to summarize a document, there are dozens of equally valid summaries. When you ask it to write code, there are multiple correct implementations. When you ask “explain quantum entanglement simply,” there’s no single ground truth. The output space is enormous, and quality is multidimensional: is the response accurate? Helpful? Concise? Safe? On-topic? Well-structured? A response can nail accuracy while being verbose and poorly organized.

This makes evaluation the weakest link in most LLM projects. Teams spend months building RAG pipelines and fine-tuning models, then evaluate by … vibes. Someone reads a few outputs, says “looks good,” and ships it. Then production users find all the edge cases you missed.

The eval gap is real. Most LLM failures in production aren’t model failures — they’re evaluation failures. The model was always doing the wrong thing; you just didn’t have the eval to catch it. Invest in evaluation before you invest in fancier models.

The evaluation landscape looks like this:

Evaluation Method Speed Cost Reliability Best For
Perplexity Fast Low Narrow (only measures prediction) Comparing base models on same data
Classical metrics (BLEU, ROUGE) Fast Low Moderate (string overlap) Translation, summarization
Benchmarks (MMLU, HumanEval) Moderate Moderate Good for broad comparison Model selection, public comparison
LLM-as-Judge Moderate Moderate Good (with debiasing) Rapid iteration, production eval
Human evaluation Slow High Highest Safety, subjective quality, final sign-off

Let’s go through each one.

Perplexity — The Fundamental LM Metric

Before we get to the fancy stuff, there’s the original language model metric: perplexity (PPL). It directly measures what language models were originally designed to do — predict the next token.

Mathematically:

# Perplexity = exponentiated average cross-entropy loss
#
# Given a sequence of tokens x₁, x₂, ..., xₙ:
#
#   PPL = exp( -1/N × Σ log P(xᵢ | x₁, ..., xᵢ₋₁) )
#
# Intuition: if the model assigns probability 1.0 to every token,
# cross-entropy = 0, PPL = exp(0) = 1.0 (perfect).
#
# If the model is "confused" and spreads probability across many tokens,
# PPL goes up. A PPL of 20 means the model is, on average, as uncertain
# as if it were choosing uniformly among 20 options at each step.
#
# Lower PPL = better language modeling.

What perplexity tells you: how well the model predicts held-out text. If you’re comparing two models trained on the same data distribution, lower perplexity generally means a better model. It’s useful for tracking training progress and for comparing base models head-to-head.

What perplexity does not tell you:

  • Whether the model generates useful text (a model can have great PPL while generating bland, generic completions)
  • Whether the model is helpful or safe as a chat assistant
  • Whether it follows instructions well
  • Anything about factual accuracy or hallucination
Perplexity is domain-sensitive. A model with PPL 8 on Wikipedia might have PPL 40 on legal contracts. Always evaluate perplexity on text from your target domain. And you can only compare perplexity across models that use the same tokenizer — different tokenizations produce different sequence lengths, making PPL values incomparable.

Here’s how to compute it in practice:

import torch
from transformers import AutoModelForCausalLM, AutoTokenizer

def compute_perplexity(model_name, text, max_length=512):
    """Compute perplexity of a model on a given text."""
    tokenizer = AutoTokenizer.from_pretrained(model_name)
    model = AutoModelForCausalLM.from_pretrained(
        model_name, torch_dtype=torch.float16, device_map="auto"
    )

    encodings = tokenizer(text, return_tensors="pt", truncation=True,
                          max_length=max_length)
    input_ids = encodings.input_ids.to(model.device)

    with torch.no_grad():
        outputs = model(input_ids, labels=input_ids)
        # outputs.loss is the average cross-entropy per token
        neg_log_likelihood = outputs.loss

    ppl = torch.exp(neg_log_likelihood).item()
    return ppl

# Example usage
text = """The transformer architecture has revolutionized natural language
processing. Self-attention allows each token to attend to every other token
in the sequence, enabling the model to capture long-range dependencies."""

ppl = compute_perplexity("gpt2", text)
print(f"GPT-2 Perplexity: {ppl:.2f}")
# GPT-2 Perplexity: ~25-35 (typical for general text)

# For long texts, use a sliding window approach to handle
# sequences longer than the model's context window
def compute_perplexity_sliding_window(model_name, text, stride=256):
    """Sliding window PPL for texts longer than context window."""
    tokenizer = AutoTokenizer.from_pretrained(model_name)
    model = AutoModelForCausalLM.from_pretrained(
        model_name, torch_dtype=torch.float16, device_map="auto"
    )
    max_length = model.config.max_position_embeddings

    encodings = tokenizer(text, return_tensors="pt")
    input_ids = encodings.input_ids
    seq_len = input_ids.size(1)

    nlls = []
    prev_end = 0
    for begin in range(0, seq_len, stride):
        end = min(begin + max_length, seq_len)
        target_len = end - prev_end  # only score the new tokens

        input_chunk = input_ids[:, begin:end].to(model.device)
        target_chunk = input_chunk.clone()
        target_chunk[:, :-target_len] = -100  # mask already-scored tokens

        with torch.no_grad():
            outputs = model(input_chunk, labels=target_chunk)
            nlls.append(outputs.loss * target_len)

        prev_end = end
        if end == seq_len:
            break

    ppl = torch.exp(torch.stack(nlls).sum() / prev_end).item()
    return ppl

Classical NLP Metrics

Before LLMs, NLP tasks like translation and summarization had reference-based metrics. These still get used — but you need to know their limitations.

BLEU (Bilingual Evaluation Understudy)

Counts n-gram overlaps between the generated text and one or more reference translations. BLEU-4 uses unigrams through 4-grams, with a brevity penalty to discourage overly short outputs. Scores range from 0 to 1 (often reported as 0–100).

Good for: machine translation, where phrasing matters and references are relatively constrained.

Bad for: open-ended generation. “The cat sat on the mat” and “A feline rested upon the rug” would score nearly zero despite meaning the same thing.

ROUGE (Recall-Oriented Understudy for Gisting Evaluation)

Similar idea, but recall-oriented (what fraction of the reference n-grams appear in the output). ROUGE-1 (unigrams), ROUGE-2 (bigrams), and ROUGE-L (longest common subsequence) are the standard variants.

Good for: summarization, where you care about coverage of key content.

Bad for: anything where paraphrasing matters. Same problem as BLEU — it’s string matching.

BERTScore

Computes semantic similarity using contextual BERT embeddings. Each token in the candidate is matched to the most similar token in the reference (and vice versa), then precision, recall, and F1 are computed over these matches.

Good for: capturing paraphrases and semantic equivalence. “The cat sat on the mat” vs. “A feline rested on the rug” would score high with BERTScore.

Bad for: it’s still reference-based. You need a “gold” answer to compare against.

Metric Type Best For Weakness Needs Reference?
BLEU Precision-oriented n-gram overlap Machine translation No semantic understanding; punishes valid paraphrases Yes
ROUGE-L Recall-oriented (LCS) Summarization String matching; misses meaning-preserving rewrites Yes
BERTScore Embedding similarity Any reference-based task Computationally heavier; still needs a reference Yes
METEOR Alignment with stemming + synonyms Translation Language-specific; slower than BLEU Yes
Perplexity Probability of held-out text LM comparison Says nothing about generation quality No (uses model’s own probabilities)
The reference problem. All classical metrics require reference outputs. For many real-world LLM tasks (chatbots, creative writing, complex reasoning), there simply is no single correct reference. This is why the field has moved toward reference-free evaluation methods like LLM-as-judge.

Benchmarks — The Leaderboard Landscape

When someone claims their model is “state of the art,” they’re usually pointing at benchmark numbers. These standardized tests let you compare models on a level playing field — or at least, that’s the idea. In practice, the benchmark landscape is a mix of genuinely useful signal and misleading noise.

The Major Benchmarks

MMLU (Massive Multitask Language Understanding): 57 subjects, 14,000+ multiple-choice questions spanning STEM, humanities, social sciences, and more. Think of it as the SAT for LLMs. It was the go-to benchmark for years, but top models now score >90%, and it’s arguably saturating.

HellaSwag: Commonsense reasoning through sentence completion. Given a scenario, pick the most plausible continuation. Models found it trivially easy once they got big enough — GPT-4 scores >95%.

HumanEval / MBPP: Code generation. Given a function signature and docstring, generate the implementation. Evaluated with pass@k — generate k solutions, pass if any one works. HumanEval has 164 problems; MBPP has ~1000. These are more resistant to saturation than text benchmarks because code either runs or it doesn’t.

GSM8K: 8,500 grade-school math word problems requiring multi-step reasoning. Tests chain-of-thought ability. Important because it measures reasoning, not just pattern matching. Top models now score >90% with CoT prompting.

TruthfulQA: 817 questions designed to trigger common misconceptions. Measures the model’s tendency to hallucinate or repeat popular myths. “What happens if you crack your knuckles?” — a truthful model says “nothing harmful,” not “you get arthritis.”

MT-Bench: 80 multi-turn questions judged by GPT-4. Covers writing, reasoning, math, coding, extraction, STEM, humanities, and roleplay. Produces a 1–10 score. Good for measuring conversational ability.

LMSYS Chatbot Arena: Crowd-sourced, blind head-to-head comparisons. Users chat with two anonymous models simultaneously and pick the better one. ELO ratings are computed from thousands of battles. Arguably the most reliable overall evaluation because it captures real user preferences with no benchmark gaming possible.

Benchmark What It Measures Format Status
MMLU Broad knowledge & reasoning Multiple choice (57 subjects) Saturating (>90% for top models)
HellaSwag Commonsense reasoning Sentence completion Largely saturated (>95%)
HumanEval Code generation Function implementation, pass@k Still useful; harder variants exist (HumanEval+)
MBPP Code generation (basic) Short Python problems, pass@k Active
GSM8K Math reasoning Word problems, chain-of-thought Saturating; MATH benchmark is harder
TruthfulQA Hallucination tendency Questions designed to trigger misconceptions Active, but small (817 questions)
MT-Bench Multi-turn chat quality GPT-4 judged, 1–10 score Active
Chatbot Arena Overall chat preference Crowd-sourced ELO from blind battles Gold standard for chat models
GPQA Graduate-level Q&A Expert-written hard questions Active; very hard even for experts
ARC-AGI Abstract reasoning Visual pattern puzzles Active; frontier models still struggle
Don’t benchmark-shop. It’s trivially easy to find the benchmark where your model looks best and only report that one. Look at aggregate performance across multiple benchmarks, and weight the benchmarks that matter for your use case. A model that crushes MMLU but fails at code generation is useless if you’re building a coding assistant.

LLM-as-Judge — The Production Evaluation Workhorse

Here’s the method that’s actually changed how teams evaluate LLMs in practice: use a stronger LLM to judge the outputs of a weaker one. It sounds circular, but it works surprisingly well.

The core idea: take a powerful model (GPT-4, Claude, Gemini) and give it a rubric-based prompt that asks it to evaluate another model’s response. The judge rates quality on specified dimensions (accuracy, helpfulness, relevance, safety) and produces a structured score.

Why It Works

Multiple studies show >80% agreement between GPT-4 judgments and expert human annotators — comparable to inter-annotator agreement between humans. For many tasks, the strong model genuinely “knows” what a good answer looks like, even if it can’t always produce one itself.

The economics are compelling: human evaluation costs $10–50+ per sample (annotator time, management, quality control). LLM-as-judge costs $0.01–0.10 per sample and runs in seconds instead of days. This means you can evaluate hundreds of examples on every code push instead of waiting for a weekly human review.

Three Evaluation Patterns

# Pattern 1: SINGLE-SCORE — Rate one response on a scale
# Simple but prone to absolute calibration issues
#
# Pattern 2: PAIRWISE COMPARISON — Which of two responses is better?
# More reliable. Humans are better at comparisons than absolute ratings.
# Used by MT-Bench, Chatbot Arena.
#
# Pattern 3: REFERENCE-GUIDED — Compare response to a gold answer
# Best when you have reference answers. Reduces ambiguity for the judge.

The Biases You Need to Know

LLM judges aren’t objective. They have systematic biases:

Bias What Happens Mitigation
Position bias Judges favor the first (or last) response in pairwise comparisons Randomize order; run each comparison twice (A/B and B/A) and check consistency
Verbosity bias Longer responses rated higher, even when padding adds no value Explicitly instruct judge to penalize unnecessary verbosity; include conciseness in rubric
Self-preference bias Models rate their own outputs higher than others’ Use a different model family as judge than the one being evaluated
Authority/format bias Well-formatted responses (bullet points, headers) score higher regardless of content Rubric should emphasize content accuracy over formatting

A Full LLM-as-Judge Implementation

import json
from openai import OpenAI

client = OpenAI()

# ── Rubric-based single-score evaluation ──────────────────────

JUDGE_SYSTEM_PROMPT = """You are an expert evaluator. Given a question and
a model's response, rate the response on these dimensions:

1. **Accuracy** (1-5): Is the information factually correct?
2. **Completeness** (1-5): Does it address all parts of the question?
3. **Clarity** (1-5): Is it well-organized and easy to understand?
4. **Conciseness** (1-5): Does it avoid unnecessary filler?

Return ONLY valid JSON:
{
  "accuracy": {"score": N, "reason": "..."},
  "completeness": {"score": N, "reason": "..."},
  "clarity": {"score": N, "reason": "..."},
  "conciseness": {"score": N, "reason": "..."},
  "overall": N
}

Be strict. A score of 3 means acceptable. 5 means exceptional.
Do NOT give high scores by default."""

def judge_single(question: str, response: str, model="gpt-4o") -> dict:
    """Rate a single response using LLM-as-judge."""
    result = client.chat.completions.create(
        model=model,
        temperature=0.0,  # deterministic judging
        response_format={"type": "json_object"},
        messages=[
            {"role": "system", "content": JUDGE_SYSTEM_PROMPT},
            {"role": "user", "content": f"Question: {question}\n\n"
                                        f"Response: {response}"}
        ]
    )
    return json.loads(result.choices[0].message.content)

# ── Pairwise comparison (more reliable) ──────────────────────

PAIRWISE_PROMPT = """You are an expert evaluator. Given a question and two
responses (A and B), decide which response is better overall.

Consider: accuracy, completeness, clarity, and helpfulness.
Do NOT favor longer responses unless the extra length adds value.

Return ONLY valid JSON:
{
  "winner": "A" or "B" or "tie",
  "reason": "Brief explanation of your decision"
}"""

def judge_pairwise(question: str, response_a: str, response_b: str,
                   model="gpt-4o") -> dict:
    """Pairwise comparison with position-bias mitigation."""
    import random

    # Randomize order to mitigate position bias
    if random.random() > 0.5:
        first, second = response_a, response_b
        order = "original"
    else:
        first, second = response_b, response_a
        order = "swapped"

    result = client.chat.completions.create(
        model=model,
        temperature=0.0,
        response_format={"type": "json_object"},
        messages=[
            {"role": "system", "content": PAIRWISE_PROMPT},
            {"role": "user", "content":
                f"Question: {question}\n\n"
                f"Response A:\n{first}\n\n"
                f"Response B:\n{second}"}
        ]
    )
    verdict = json.loads(result.choices[0].message.content)

    # Correct for swapped order
    if order == "swapped" and verdict["winner"] in ("A", "B"):
        verdict["winner"] = "B" if verdict["winner"] == "A" else "A"

    return verdict

# ── Run evaluation over a test set ───────────────────────────

def evaluate_model(test_cases: list[dict], generate_fn, judge_model="gpt-4o"):
    """
    Evaluate a model on a test set using LLM-as-judge.

    test_cases: [{"question": "...", "reference": "..." (optional)}]
    generate_fn: function(question) -> response string
    """
    results = []
    for case in test_cases:
        response = generate_fn(case["question"])
        score = judge_single(case["question"], response, model=judge_model)
        results.append({
            "question": case["question"],
            "response": response,
            "scores": score
        })

    # Aggregate
    avg_overall = sum(r["scores"]["overall"] for r in results) / len(results)
    print(f"Average overall score: {avg_overall:.2f} / 5.0")
    print(f"Evaluated {len(results)} examples")
    return results
When to trust LLM-as-judge: It’s excellent for relative comparisons (is model A better than model B on this task?) and for detecting regressions (did the new prompt make things worse?). It’s less reliable for absolute quality assessment (is this response “good enough” for production?). Always validate your judge against human ratings on a sample before trusting it at scale.

Human Evaluation

LLM-as-judge is great for iteration speed, but some things still need a human in the loop. Safety assessments, nuanced quality judgments, detecting subtle hallucinations in domain-specific content, evaluating cultural sensitivity — these are areas where even GPT-4 can be confidently wrong.

When You Need Humans

  • Safety-critical applications: medical, legal, financial advice
  • Subjective quality: creative writing, brand voice, emotional tone
  • Final validation: before major launches or model switches
  • Calibrating your LLM judge: you need human labels to know if your judge is reliable
  • Edge cases: the long tail of weird inputs that automated evals miss

Measuring Agreement: Cohen’s Kappa

When two humans evaluate the same output, they won’t always agree. The question is: do they agree more than chance? Cohen’s kappa (κ) adjusts for chance agreement:

# Cohen's Kappa
#
#   κ = (p_observed - p_chance) / (1 - p_chance)
#
#   κ = 1.0  → perfect agreement
#   κ = 0.0  → agreement no better than random
#   κ < 0.0  → agreement worse than random (shouldn't happen)
#
# Rules of thumb:
#   κ > 0.8  → excellent agreement (publishable)
#   κ 0.6-0.8 → substantial agreement (usable)
#   κ 0.4-0.6 → moderate agreement (questionable)
#   κ < 0.4  → poor agreement (fix your rubric)

from sklearn.metrics import cohen_kappa_score

annotator_1 = [3, 4, 5, 2, 4, 3, 5, 4, 2, 3]  # ratings 1-5
annotator_2 = [3, 5, 5, 2, 3, 3, 4, 4, 2, 4]

kappa = cohen_kappa_score(annotator_1, annotator_2, weights="quadratic")
print(f"Cohen's Kappa: {kappa:.3f}")
# With quadratic weighting, disagreeing by 1 point is penalized
# less than disagreeing by 3 points. Use this for ordinal scales.

The Hybrid Approach

Most production teams use a layered strategy:

  1. Fast iteration: LLM-as-judge on every commit. Catches regressions in minutes.
  2. Weekly spot-check: Human review of a random sample (20–50 examples). Catches judge blind spots.
  3. Pre-launch validation: Full human eval on the complete test set. The final gate.
  4. Continuous calibration: Compare judge scores to human scores monthly. Retune the rubric when they diverge.
The cost math. A typical human eval setup: 3 annotators, 200 examples, 5 minutes per example = 50 person-hours. At $25/hour, that’s $1,250 per eval round. LLM-as-judge on the same 200 examples: ~$2–20 and 5 minutes wall time. Use LLM-as-judge to run 100 evals so you can spend your human eval budget on the 2–3 that actually matter.

Evaluation Challenges

Even if you pick the right metrics and build a great eval set, there are meta-problems that can undermine your entire evaluation:

Data Contamination

The elephant in the room. LLMs are trained on massive web crawls that may include the test data you’re evaluating on. If MMLU questions appear in the training data, the model isn’t “reasoning” — it’s remembering. This is a documented problem: multiple studies have found benchmark data in LLM training sets.

Mitigation: use held-out test sets you create yourself. Time-based splits (evaluate on data from after the training cutoff). Canary strings in your test data to detect if they appear in model outputs.

Task Saturation

Benchmarks have a shelf life. When the best models score >95% on a task, it loses discriminative power. HellaSwag, ARC-Easy, and increasingly MMLU are entering this territory. The response has been a continual arms race: harder benchmarks (GPQA, ARC-AGI, MATH, SWE-Bench) that push the frontier.

Prompt Sensitivity

The same model can score dramatically differently based on the prompt template used for evaluation. “Answer the following question” vs. “Let’s think step by step” can swing scores by 10–20 points on reasoning benchmarks. This makes cross-paper comparisons tricky when each team uses slightly different evaluation prompts.

# Same model, same question, different prompts, different results
#
# Prompt A: "What is the capital of Australia?"
# Model answer: "Canberra" ✓
#
# Prompt B: "Answer with just the city name. Capital of Australia?"
# Model answer: "Canberra" ✓
#
# Prompt C: "The capital of Australia is"
# Model answer: "Sydney" ✗ (common misconception, model completes the pattern)
#
# For benchmarks, standardize your prompt template
# and NEVER change it between experiments.

Goodhart’s Law

“When a measure becomes a target, it ceases to be a good measure.” Models (and teams) optimize for benchmark scores. RLHF can be tuned to boost benchmark numbers without improving real-world usefulness. A model that scores 90% on MMLU but can’t handle your specific customer support queries is failing at your task, regardless of the leaderboard position.

Benchmarks measure models. Evals measure products. Public benchmarks tell you about general capability. Your own eval set — built from your actual use cases, with your actual edge cases — tells you if the model works for your users. Never skip the second part just because the first looks good.

Practical Evaluation Strategy

Enough theory. Here’s how to actually build an evaluation pipeline for an LLM-powered product.

Step 1: Build Your Own Eval Set

This is the single highest-leverage thing you can do. Create 50–200 examples that represent your specific task. Each example should have:

  • An input (the question, document, or context your model receives)
  • An expected output or rubric (what a good response looks like)
  • Edge cases and failure modes you’ve seen in practice
# Example eval set structure
eval_set = [
    {
        "id": "summarize-001",
        "input": "Summarize this earnings call transcript...",
        "reference": "Q3 revenue grew 12% YoY to $4.2B, driven by cloud...",
        "rubric": {
            "must_include": ["revenue figure", "YoY growth", "key driver"],
            "must_not_include": ["speculation", "opinions not in source"],
            "max_length": 200
        },
        "difficulty": "easy",
        "category": "summarization"
    },
    {
        "id": "summarize-002",
        "input": "Summarize this earnings call with contradictory guidance...",
        "reference": "Management raised Q4 guidance to $4.5B but warned...",
        "rubric": {
            "must_include": ["both the positive and negative guidance"],
            "must_not_include": ["one-sided summary"],
            "max_length": 200
        },
        "difficulty": "hard",
        "category": "summarization"
    },
    # ... 50-200 examples covering your task distribution
]

Step 2: Automate with LLM-as-Judge

Wire up LLM-as-judge to run on every model change, prompt update, or pipeline modification. This is your CI/CD for LLM quality.

# Minimal eval pipeline — runs in CI
import json
from pathlib import Path

def run_eval_pipeline(eval_set_path: str, generate_fn, judge_model="gpt-4o"):
    """Automated eval pipeline for CI integration."""
    eval_set = json.loads(Path(eval_set_path).read_text())
    results = []

    for case in eval_set:
        response = generate_fn(case["input"])
        scores = judge_single(case["input"], response, model=judge_model)

        results.append({
            "id": case["id"],
            "category": case.get("category", "general"),
            "scores": scores,
            "passed": scores["overall"] >= 3  # minimum quality bar
        })

    # Summary
    pass_rate = sum(r["passed"] for r in results) / len(results)
    by_category = {}
    for r in results:
        cat = r["category"]
        if cat not in by_category:
            by_category[cat] = {"total": 0, "passed": 0}
        by_category[cat]["total"] += 1
        by_category[cat]["passed"] += int(r["passed"])

    print(f"\nOverall pass rate: {pass_rate:.1%}")
    for cat, stats in by_category.items():
        cat_rate = stats["passed"] / stats["total"]
        print(f"  {cat}: {cat_rate:.1%} ({stats['passed']}/{stats['total']})")

    # Fail CI if quality regresses
    if pass_rate < 0.85:  # your threshold
        print("\n❌ EVAL FAILED: quality below threshold")
        exit(1)
    else:
        print("\n✅ EVAL PASSED")

    return results

Step 3: Track Regression Across Updates

Every time you change the model, prompt, or pipeline, you should know if quality went up or down. Store eval results with timestamps and version info.

# Simple regression tracking
import sqlite3
from datetime import datetime

def log_eval_results(db_path, version, results):
    """Store eval results for regression tracking."""
    conn = sqlite3.connect(db_path)
    conn.execute("""
        CREATE TABLE IF NOT EXISTS eval_runs (
            run_id TEXT, version TEXT, timestamp TEXT,
            case_id TEXT, category TEXT, overall_score REAL, passed INTEGER
        )
    """)

    run_id = f"{version}_{datetime.now().isoformat()}"
    for r in results:
        conn.execute(
            "INSERT INTO eval_runs VALUES (?, ?, ?, ?, ?, ?, ?)",
            (run_id, version, datetime.now().isoformat(),
             r["id"], r["category"], r["scores"]["overall"], int(r["passed"]))
        )
    conn.commit()

    # Compare with previous version
    prev = conn.execute("""
        SELECT AVG(overall_score) FROM eval_runs
        WHERE version != ? ORDER BY timestamp DESC LIMIT 200
    """, (version,)).fetchone()

    curr = conn.execute("""
        SELECT AVG(overall_score) FROM eval_runs WHERE version = ?
    """, (version,)).fetchone()

    if prev[0] and curr[0]:
        delta = curr[0] - prev[0]
        emoji = "📈" if delta > 0 else "📉" if delta < 0 else "➡️"
        print(f"{emoji} Score delta: {delta:+.3f} "
              f"(prev: {prev[0]:.3f}, curr: {curr[0]:.3f})")

    conn.close()

Step 4: A/B Test in Production

Offline evals don’t capture everything. Real users behave differently from your test cases. Run A/B tests: serve the old model/prompt to 50% of users and the new one to 50%. Measure engagement, satisfaction, task completion, and complaint rate.

Step 5: The Eval Flywheel

Here’s the most important pattern: every production failure becomes a new eval case.

# The eval flywheel:
#
#   ┌──────────────┐
#   │  Deploy model │
#   └──────┬───────┘
#          │
#          ▼
#   ┌──────────────┐
#   │  Users find   │
#   │  failures     │──── user reports, flagged responses,
#   └──────┬───────┘     low satisfaction scores
#          │
#          ▼
#   ┌──────────────┐
#   │  Add failure  │
#   │  to eval set  │──── input + expected output + rubric
#   └──────┬───────┘
#          │
#          ▼
#   ┌──────────────┐
#   │  Improve      │
#   │  model/prompt │──── iterate until the new eval case passes
#   └──────┬───────┘
#          │
#          ▼
#   ┌──────────────┐
#   │  Run full     │
#   │  eval suite   │──── ensure no regressions
#   └──────┬───────┘
#          │
#          └───────────────► back to Deploy
#
# Your eval set grows organically from real failures.
# After 6 months, it's worth more than any public benchmark.
Start with 50 examples. You don’t need 10,000 eval cases to start. Fifty well-chosen examples covering your core use cases and known edge cases will catch more regressions than you’d expect. Grow the set over time using the flywheel. The goal isn’t completeness — it’s catching the failures that matter most to your users.
