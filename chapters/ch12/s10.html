Evaluation

Chapter 12: Large Language Models 4 subtopics
TL;DR

Evaluating LLMs is fundamentally harder than evaluating classifiers because the output space is enormous and multiple correct answers exist. Perplexity tells you how surprised the model is but nothing about generation quality. Classical metrics like BLEU and ROUGE rely on surface overlap and miss semantic meaning. Benchmarks like MMLU, HumanEval, and GSM8K test specific capabilities but suffer from data contamination and saturation. LLM-as-judge uses a strong model to score a weaker one — surprisingly effective but biased. Human evaluation remains the gold standard for subjective quality but is expensive and slow. In practice, you need a layered strategy: automated metrics for fast iteration, LLM-as-judge for nuance, and targeted human eval for high-stakes decisions.

Why Evaluation is Hard for LLMs

If you train an image classifier, evaluation is simple. The model says "cat" or "dog," and you check if it's right. You get a clean accuracy number. You sleep well at night. LLMs don't give you that luxury.

Ask an LLM to write a poem about autumn. There are millions of valid poems it could produce. Which one is "correct"? Ask it to explain quantum entanglement. A physics professor and a curious ten-year-old need different explanations — both can be right. Ask it to write a Python function that sorts a list. There are dozens of valid implementations with different tradeoffs. The output space — the set of all possible valid responses — is astronomically large, and "correct" is often a spectrum rather than a binary label.

This is the core problem. Traditional metrics assume there's a single right answer, or at least a small set of right answers. When you're dealing with open-ended text generation, that assumption collapses. You can't just compare the model's output to a reference string and call it a day. Two completely different sentences can express the same idea perfectly well, while two nearly identical sentences might differ on a critical factual detail.

There's another wrinkle. LLMs are used for dozens of different tasks — summarization, translation, question answering, code generation, creative writing, reasoning, conversation. A model that excels at coding might be mediocre at creative writing. No single metric captures everything. Evaluation for LLMs is inherently multi-dimensional, and pretending otherwise leads to misleading leaderboards and bad decisions.

Perplexity

Let's start with the oldest and simplest metric. Perplexity measures how "surprised" a language model is by a piece of text. The intuition is direct: if the model thinks each word in a sentence is highly likely, perplexity is low. If the model keeps getting surprised by what comes next, perplexity is high.

Mathematically, perplexity is the exponentiated average negative log-likelihood per token. For a sequence of N tokens:

# Perplexity formula
# PPL = exp( -(1/N) * Σ log P(token_i | token_1, ..., token_{i-1}) )
#
# In plain English:
# 1. For each token, compute the model's predicted probability
# 2. Take the log of each probability
# 3. Average those log-probabilities
# 4. Negate it (because log-probs are negative)
# 5. Exponentiate the result
#
# Lower perplexity = model assigns higher probability = less surprised
# A perplexity of 1 means perfect prediction
# GPT-2: ~30 on WikiText-103
# GPT-3: ~20 on similar benchmarks

Here's how you compute it in practice:

import torch
from transformers import AutoModelForCausalLM, AutoTokenizer

model_name = "gpt2"
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForCausalLM.from_pretrained(model_name).eval()

text = "The transformer architecture revolutionized natural language processing."
inputs = tokenizer(text, return_tensors="pt")

with torch.no_grad():
    outputs = model(**inputs, labels=inputs["input_ids"])
    # outputs.loss = average cross-entropy loss per token
    perplexity = torch.exp(outputs.loss)

print(f"Perplexity: {perplexity.item():.2f}")
# Lower = model found the text more predictable

Perplexity is useful for comparing two models on the same dataset. If Model A has perplexity 15 and Model B has perplexity 22 on the same test set, Model A is a better language model in the statistical sense — it assigns higher probabilities to real text. This is valuable during pretraining, where you want to track whether your model is learning language patterns effectively.

Perplexity Gotchas

Perplexity depends on the tokenizer. A model with a 100K-token vocabulary might get lower perplexity than one with a 32K vocabulary simply because it splits text into fewer tokens — not because it understands language better. Always compare models that use the same tokenizer, or at least acknowledge this confound. Also, perplexity says nothing about whether the model generates coherent, helpful, or truthful text. A model can have excellent perplexity and still hallucinate confidently.

Classical NLP Metrics

Before LLMs dominated, the NLP community developed a handful of automatic metrics for specific generation tasks. They're still widely used, so you need to understand them — and their limitations.

BLEU

BLEU (Bilingual Evaluation Understudy) was designed for machine translation. It measures n-gram overlap — how many short phrases in the model's output also appear in a reference translation. BLEU computes precision: of all the n-grams in the candidate, what fraction appears in the reference? It uses a brevity penalty so the model can't cheat by outputting just one correct word. BLEU scores range from 0 to 1, with 1 being a perfect match to the reference.

The problem is that BLEU only cares about exact word matches. "The cat sat on the mat" and "A feline rested upon the rug" mean roughly the same thing, but BLEU gives a low score because the words are different. BLEU also can't handle paraphrasing, creative rewording, or any linguistic flexibility. For machine translation between similar languages, it works okay. For open-ended generation, it's deeply misleading.

ROUGE

ROUGE (Recall-Oriented Understudy for Gisting Evaluation) was designed for summarization. Where BLEU focuses on precision (what fraction of the candidate's n-grams appear in the reference), ROUGE focuses on recall (what fraction of the reference's n-grams appear in the candidate). ROUGE-1 measures unigram overlap, ROUGE-2 measures bigram overlap, and ROUGE-L measures the longest common subsequence. It tells you how much of the reference content the model captured.

ROUGE has the same fundamental weakness as BLEU: it's a surface-level comparison. Two summaries can capture identical information using completely different phrasing, and ROUGE will penalize the one that doesn't match the reference wording.

BERTScore

BERTScore takes a different approach. Instead of comparing exact words, it computes contextual embeddings for each token in both the candidate and the reference, then matches tokens by cosine similarity. This captures semantic similarity — "cat" and "feline" have similar embeddings, so they'll match well even though they're different strings. BERTScore correlates much better with human judgments than BLEU or ROUGE.

But even BERTScore has limits. It still needs a reference to compare against. For open-ended tasks where there's no single correct answer, you either need multiple references (expensive to create) or you need to abandon reference-based metrics entirely. That's exactly where the field has moved.

Benchmarks

Because no single metric captures LLM quality, the community has developed a suite of benchmarks — standardized test sets that probe specific capabilities. Think of them as subject-specific exams for LLMs. Each benchmark tests a different skill, and together they paint a more complete picture.

MMLU

MMLU (Massive Multitask Language Understanding) is the SAT of LLM benchmarks. It contains around 16,000 multiple-choice questions spanning 57 academic subjects — everything from abstract algebra and anatomy to world religions and US history. Each question has four choices. The model picks one. You measure accuracy.

What makes MMLU powerful is its breadth. A model can't game it by being good at one narrow skill. It has to demonstrate broad factual knowledge and reasoning across fields that require genuinely different expertise. GPT-3.5 scored around 70% on MMLU. GPT-4 pushed past 86%. Human expert performance varies by subject but averages around 90%. When you see MMLU scores on a model card, you're seeing a rough proxy for "how much stuff does this model know?"

HellaSwag

HellaSwag tests commonsense reasoning through sentence completions. You get a situation — like "A woman is sitting at a piano. She places her hands on the keys." — and four possible continuations. Only one is plausible; the others are adversarially crafted to sound grammatically correct but be logically absurd. Humans score 95%+ easily. Early language models struggled badly because the wrong answers were designed using an adversarial filtering process that exploited statistical patterns models rely on. Modern LLMs have largely saturated this benchmark, which is itself an interesting signal about progress.

HumanEval

HumanEval is the benchmark that matters most to software engineers. It contains 164 Python programming problems with function signatures, docstrings, and test cases. The model generates a function body, and you run the tests to see if it passes. The metric is Pass@k: generate k candidate solutions and check if at least one passes all tests. Pass@1 is the strictest — does the first attempt work? Pass@10 allows ten tries, which better captures the model's capability if you're willing to use sampling and filtering.

HumanEval problems range from simple string manipulation to tricky algorithm questions. GPT-4 achieves around 67% Pass@1; Claude 3.5 Sonnet reaches similar territory. These numbers are remarkable — and also reveal how far we are from models that reliably write correct code on the first try.

GSM8K

GSM8K contains 8,500 grade-school-level math word problems. The problems require 2 to 8 steps of arithmetic reasoning. For example: "Janet's ducks lay 16 eggs per day. She eats three for breakfast and bakes four into muffins. She sells the rest at the farmers' market at $2 per egg. How much does she make per day?" The answer requires multiple steps: 16 − 3 − 4 = 9 eggs, then 9 × $2 = $18. This benchmark became famous because it demonstrated that chain-of-thought prompting dramatically improves LLM math performance. Without chain-of-thought, models often fail on multi-step problems they'd solve if they "showed their work."

TruthfulQA

TruthfulQA is designed to catch models in the act of hallucinating. It contains 817 questions that humans frequently answer incorrectly because of common misconceptions, superstitions, or conspiracy theories. For example: "Can you teach an old dog new tricks?" Many people say no, but the truthful answer is yes. Models tend to reproduce the popular misconception because they've seen it more often in training data. A model's TruthfulQA score measures its ability to be truthful rather than merely agreeable or fluent.

MT-Bench and Chatbot Arena

MT-Bench evaluates multi-turn conversation ability. It contains 80 carefully crafted two-turn questions across eight categories: writing, roleplay, extraction, reasoning, math, coding, knowledge, and STEM. A strong LLM (typically GPT-4) acts as the judge, scoring responses from 1 to 10. The two-turn design tests whether the model can follow up coherently — a skill that single-turn benchmarks completely miss.

LMSYS Chatbot Arena takes the crowd-sourcing approach. Real users submit prompts. Two anonymous models generate responses. The user picks the better one. Over hundreds of thousands of such votes, an Elo rating emerges — the same system used in chess rankings. As of now, Chatbot Arena is arguably the most trusted ranking of conversational LLMs because it reflects actual user preferences on real-world prompts rather than curated academic tests.

LLM-as-Judge

Here's a surprising development from the past two years. If you ask a strong LLM to evaluate the output of another LLM, the judgments correlate remarkably well with human preferences — often 80–85% agreement, which is close to human-human agreement rates. This technique, called LLM-as-judge, has become the workhorse of modern LLM evaluation because it's fast, cheap, and scalable.

There are two main approaches. In pairwise comparison, the judge sees two responses side by side and picks the better one. This is easier than absolute scoring because relative judgments are more consistent — even for humans, "Is A better than B?" is easier than "Rate A on a scale of 1–10." In Likert scoring, the judge rates a single response on a scale (say 1 to 5 or 1 to 10) along specific dimensions like helpfulness, accuracy, and coherence.

import openai

def llm_judge_pairwise(question: str, response_a: str, response_b: str) -> str:
    """Use a strong LLM to compare two responses."""
    prompt = f"""You are an expert judge evaluating AI assistant responses.

Question: {question}

Response A:
{response_a}

Response B:
{response_b}

Compare both responses on these criteria:
1. Accuracy: Are the facts correct?
2. Helpfulness: Does it answer the question fully?
3. Clarity: Is it well-written and easy to follow?

First, analyze each response briefly. Then output your verdict as
exactly one of: "A is better", "B is better", or "Tie".
"""
    result = openai.chat.completions.create(
        model="gpt-4",
        messages=[{"role": "user", "content": prompt}],
        temperature=0.0,  # deterministic for reproducibility
    )
    return result.choices[0].message.content

# Example usage
verdict = llm_judge_pairwise(
    question="Explain gradient descent in simple terms.",
    response_a="Gradient descent is an optimization algorithm...",
    response_b="Imagine you're blindfolded on a hill..."
)
print(verdict)

The method has a known weakness: position bias. LLM judges tend to favor whichever response appears first in the prompt. The fix is simple but important — run each comparison twice, swapping the order of A and B. If the judge picks A both times, you're confident. If it flips, call it a tie. This doubles your cost but dramatically improves reliability.

When LLM-as-Judge Fails

LLM judges struggle with domains where they themselves are weak. If you use GPT-4 to judge responses about cutting-edge physics research, and GPT-4 doesn't understand that research, its judgments will be unreliable. LLM-as-judge works best when the judge model is substantially stronger than the models being evaluated, or when the task is well within the judge's competence. For highly specialized domains, you still need human experts.

Human Evaluation

For all the progress in automated metrics, human evaluation remains the gold standard when the task involves subjective quality — creativity, tone, helpfulness, trustworthiness. Humans can judge things that no metric captures: whether a response feels condescending, whether it misses the subtext of a question, whether the humor lands.

The standard approach uses Likert scales. Annotators rate each response on dimensions like fluency (1–5), relevance (1–5), and factual accuracy (1–5). Alternatively, comparative ranking asks annotators to rank multiple responses from best to worst, which is often easier and more reliable than absolute scoring. The key quality metric is inter-annotator agreement — if three annotators disagree wildly about which response is better, your evaluation criteria are ambiguous and your results are noise.

The downside is cost and speed. Hiring qualified annotators, training them on your rubric, getting enough data points for statistical significance — this takes weeks and real money. A single round of human evaluation across a few hundred examples might cost thousands of dollars and take days to complete. Compare that to running an LLM judge in minutes for a few dollars. This is why human eval is typically reserved for the final, high-stakes comparison — not for daily iteration during development.

When do you absolutely need human evaluation? When the task has no clear "right answer" (creative writing, open-ended advice). When safety and sensitivity matter (medical, legal, or politically charged topics). When you're deploying to real users and need confidence that automated metrics aren't misleading you. Think of human eval as audit-level scrutiny — you don't run it every commit, but you run it before every major release.

Evaluation Challenges

Data Contamination

Here's a dirty secret of LLM benchmarking: the models may have seen the test data during training. LLMs train on massive internet crawls. Benchmark datasets live on the internet. If GSM8K problems appeared in the training data, the model isn't reasoning about math — it's retrieving answers from memory. This is data contamination, and it's extremely hard to detect or prevent. Some models show suspiciously high scores on specific benchmarks that drop sharply when you rephrase the same questions. That's a contamination red flag.

Task Saturation

When top models score 95%+ on a benchmark, the benchmark stops being useful. This is task saturation. HellaSwag is nearly saturated — most strong models score above 85%. MMLU is approaching saturation at the top end. When this happens, score differences between models are within the noise margin and don't tell you much. The community responds by creating harder benchmarks (MMLU-Pro, GPQA), but it's a constant arms race.

Prompt Sensitivity

This is perhaps the most underappreciated problem. Small changes to the prompt — adding "Let's think step by step," changing "Answer:" to "The answer is:", or even adding a newline — can swing benchmark scores by 5–15 percentage points. This means that reported benchmark numbers are implicitly conditioned on specific prompt templates, and comparing two models evaluated with different prompts is borderline meaningless. Reproducible evaluation requires fixing and reporting the exact prompt template, which many papers and model cards fail to do.

The Leaderboard Trap

Don't pick models based solely on leaderboard rankings. Models can be optimized specifically for benchmark performance through data contamination, prompt engineering, or training directly on benchmark-style data. A model that tops the leaderboard might perform worse on your actual use case than a model ranked lower. Always test on your own data with your own prompts. Benchmarks are directionally useful — they're not a substitute for task-specific evaluation.

Practical Evaluation Strategy

So what do you actually do? Here's the strategy that works in practice, whether you're fine-tuning a model, choosing between providers, or validating a production deployment.

First, build a task-specific evaluation set. Collect 100–500 examples that represent your actual use case. If you're building a customer support bot, gather real customer questions with expert-written reference answers. If you're building a code assistant, gather real coding tasks with test cases. This dataset is your ground truth — more valuable than any public benchmark because it directly measures what you care about.

# Practical evaluation pipeline structure
eval_results = []

for example in eval_dataset:
    # Layer 1: Automated metrics (fast, cheap, run on every change)
    response = model.generate(example["prompt"])
    metrics = {
        "exact_match": response.strip() == example["reference"].strip(),
        "bleu": compute_bleu(response, example["reference"]),
        "contains_answer": example["key_fact"] in response,
    }

    # Layer 2: LLM-as-judge (moderate cost, run daily or per PR)
    judge_score = llm_judge_score(
        question=example["prompt"],
        response=response,
        rubric="Rate 1-5 on accuracy, helpfulness, and clarity"
    )
    metrics["judge_score"] = judge_score

    eval_results.append(metrics)

# Layer 3: Human evaluation (expensive, run before major releases)
# Sample 50-100 examples stratified by judge_score
# Have 3 annotators rate on your rubric
# Compute inter-annotator agreement (Krippendorff's alpha > 0.7)

# Aggregate and report
avg_judge = sum(r["judge_score"] for r in eval_results) / len(eval_results)
pass_rate = sum(r["exact_match"] for r in eval_results) / len(eval_results)
print(f"Judge avg: {avg_judge:.2f}, Pass rate: {pass_rate:.1%}")

Second, layer your evaluation approach. Use cheap automated metrics (exact match, BLEU, keyword presence) for fast feedback during development. These catch regressions quickly. Add LLM-as-judge evaluation for nuanced quality assessment — run this daily or per pull request. Reserve human evaluation for major decisions: choosing between model providers, validating before launch, or investigating cases where automated metrics and LLM judges disagree.

Third, track trends rather than absolute numbers. If your LLM-judge score was 3.8 last week and dropped to 3.4 after a prompt change, that's a meaningful signal even if you don't know what "3.8" means in absolute terms. Relative comparisons within your evaluation framework are more reliable than cross-framework comparisons. Run the same evaluation pipeline consistently, and you'll catch regressions early.

Fourth, actively test for failure modes. Don't just measure average performance. Deliberately test edge cases: very long inputs, adversarial prompts, questions about your model's known weak spots, prompts in multiple languages if that matters. Create a "red team" subset of your eval set specifically designed to probe for hallucinations, refusal failures, and safety issues. Average performance can hide catastrophic failures on specific input types.

Finally, accept that evaluation is never "solved." As your use case evolves, your eval set needs to evolve too. As models improve, your benchmarks need to get harder. As new failure modes emerge, you need new test cases. Evaluation is a continuous practice, not a one-time setup. The teams that build the best LLM products are invariably the ones with the most rigorous and frequently updated evaluation pipelines.
