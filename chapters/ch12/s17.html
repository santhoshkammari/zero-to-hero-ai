Modern LLM Architecture Decisions

Chapter 12: Large Language Models 10 subtopics
TL;DR

The original 2017 Transformer has been refined into a largely converged modern recipe: Pre-Norm with RMSNorm, SwiGLU activations, Rotary Position Embeddings, Grouped-Query Attention, and BPE tokenizers with 32K–128K vocabularies. Each choice exists because someone tried the alternatives and measured the tradeoffs. Understanding why these decisions were made—not just copying configs—is what lets you adapt when the next breakthrough lands.

Why Architecture Decisions Matter

Building a modern LLM is like designing a race car. Every component choice ripples through the whole system. You cannot change the positional encoding without rethinking your attention mechanism. You cannot pick a vocabulary size without considering inference speed.

The 2017 "Attention Is All You Need" Transformer was a starting point. Since then, hundreds of experiments have refined every component. Today's best open models—Llama 3, Mistral, DeepSeek—share a converged set of choices. That convergence isn't coincidence. It's the result of brutal empirical selection.

If you just copy those configs without understanding the reasoning, you're stuck the moment something changes. The engineer who knows why SwiGLU beat ReLU can evaluate the next activation function in an afternoon. The practitioner who copied a config file cannot. Let's walk through each decision, bottom-up.

Pre-Norm vs Post-Norm

The original Transformer used what we now call Post-Norm: you run the sublayer (attention or feedforward), add the residual connection, and then apply LayerNorm. This ordering causes a subtle problem. The gradients have to flow through the normalization layer on the backward pass, and at scale—hundreds of layers, billions of parameters—this leads to unstable training. Gradients either explode or vanish, and you spend weeks tuning learning rates.

Pre-Norm flips the order: apply LayerNorm before the sublayer. The residual connection now carries a clean, unnormalized signal straight through the network. Gradients flow more easily. Training is dramatically more stable, converges faster, and is far less sensitive to hyperparameters. Virtually every modern LLM uses Pre-Norm.

There's a deeper variant called DeepNorm that scales the residual connections by a factor that depends on the network depth. This allows training extremely deep networks—1000+ layers—without the instability that would normally creep in. It's a compromise between Pre-Norm's stability and Post-Norm's slightly better final quality at moderate depths.

Now, about the normalization itself. Standard LayerNorm computes the mean and variance of activations, subtracts the mean, and divides by the standard deviation. RMSNorm drops the mean-centering step entirely. It only divides by the root mean square of the activations. That sounds like a minor simplification, but it saves compute on every single token at every single layer. In practice, the quality difference is negligible. Llama, Mistral, and most modern models use RMSNorm. Faster is better when the quality is the same.

SwiGLU Activation

The feedforward network (FFN) inside each Transformer layer is deceptively important. It's where the model stores most of its factual knowledge. The original Transformer used ReLU: max(0, x). Simple, fast, and it works. But we can do better.

SwiGLU is a gated linear unit combined with the Swish activation function. Here's the intuition. The input to the FFN gets split into two paths. One path goes through the Swish function—a smooth, non-monotonic activation defined as x · sigmoid(βx). The other path stays linear. Then the two paths are multiplied together element-wise. That multiplication is the "gating": the Swish path decides how much of the linear path gets through.

Why does this work better than plain ReLU? The gating mechanism gives the network a richer set of transformations to learn. Empirically, SwiGLU consistently outperforms ReLU, GELU, and the related GeGLU on language modeling benchmarks. It's not a dramatic improvement on any single test, but it's consistent, and at the scale of modern LLMs, consistent small improvements compound.

One practical detail: when you switch from a plain ReLU FFN to SwiGLU, the gating mechanism adds parameters because you're projecting the input into three matrices instead of two. To keep the total parameter count comparable, the hidden dimension of the FFN is typically set to 8/3 × d_model instead of the traditional 4 × d_model. This is why you'll see odd hidden dimensions like 11008 in Llama's config—it's 8/3 × 4096, rounded to a convenient multiple.

Rotary Position Embedding (RoPE)

Positional encoding is arguably the single most impactful architecture choice in modern LLMs. The original Transformer used fixed sinusoidal functions to inject absolute position. That approach has a hard ceiling: it doesn't generalize well beyond the training length, and it treats position as an add-on rather than something intrinsic to attention.

Rotary Position Embedding (RoPE) takes a fundamentally different approach. Instead of adding a position vector to the token embedding, it rotates the query and key vectors. Each pair of dimensions gets rotated by an angle proportional to the token's position. When you compute the dot product of a rotated query and key, the angles subtract—so the attention score depends only on the relative distance between tokens, not their absolute positions.

This has beautiful properties. Nearby tokens naturally attend more strongly because the rotation difference is small. Distant tokens attend weakly because their rotations diverge. This decay emerges from the geometry, not from a forced constraint. The rotation is applied to consecutive pairs of dimensions. For a head dimension of 128, you get 64 rotation planes, each with a different frequency—low frequencies capture long-range patterns, high frequencies encode local word order.

import torch

def apply_rope(x, positions, dim, base=10000):
    """Apply Rotary Position Embedding to queries or keys.
    x: (batch, seq_len, n_heads, head_dim)
    positions: (batch, seq_len) integer positions
    """
    head_dim = x.shape[-1]
    # Each pair of dims gets a different frequency
    freqs = 1.0 / (base ** (torch.arange(0, head_dim, 2).float() / head_dim))
    # Angles = position × frequency
    angles = positions.unsqueeze(-1).float() * freqs  # (batch, seq, head_dim/2)
    angles = angles.unsqueeze(2)  # broadcast over heads

    # Split into pairs and rotate
    x1, x2 = x[..., ::2], x[..., 1::2]
    cos_a, sin_a = angles.cos(), angles.sin()
    rotated_x1 = x1 * cos_a - x2 * sin_a
    rotated_x2 = x1 * sin_a + x2 * cos_a

    # Interleave back
    return torch.stack([rotated_x1, rotated_x2], dim=-1).flatten(-2)

The key line is the rotation: [x1·cos θ − x2·sin θ, x1·sin θ + x2·cos θ]. That's a standard 2D rotation matrix applied to each pair of dimensions. Simple linear algebra, enormous practical impact.

Context Length Extension

RoPE models are trained at a fixed context length—say, 8192 tokens. But users want 128K or more. The rotation angles at position 8193 were never seen during training. What happens? The model degrades, often catastrophically.

Position Interpolation is the simplest fix: divide all positions by a scaling factor so they fit within the trained range. Training to 8K and want 32K? Divide every position by 4. The model sees familiar rotation angles, and quality degrades gracefully. A brief fine-tune at the new length sharpens things up.

YaRN (Yet Another RoPE extensioN) is more sophisticated. It recognizes that different frequency components should be scaled differently. High-frequency components encoding local word order shouldn't be scaled—they're already within range. Low-frequency components encoding global position need the most interpolation. YaRN applies frequency-dependent scaling with measurably better results. NTK-Aware Scaling adjusts the base frequency of RoPE itself rather than scaling positions, implicitly redistributing frequencies to improve extrapolation.

Even with these tricks, long-context models hit two known failure modes. The attention sink phenomenon: first tokens in the sequence receive disproportionately high attention regardless of their content. Models seem to use the first position as a "default" attention target. The lost-in-the-middle problem: models are measurably worse at using information placed in the middle of long contexts compared to the beginning or end. Both are active research areas with no clean solutions yet. Be honest with yourself about these limitations when deploying long-context models.

Grouped-Query Attention (GQA)

Standard Multi-Head Attention (MHA) gives each attention head its own query, key, and value projections. With 32 heads, that's 32 separate K and V matrices. During inference, you cache the K and V tensors for every token you've generated so far—the infamous KV cache. For long sequences, this cache dominates GPU memory.

Multi-Query Attention (MQA) was the first attempt to fix this: all heads share a single K and V projection. The KV cache shrinks by a factor of 32. Inference is dramatically faster. But there's a quality penalty—the heads lose expressiveness when they're all attending with the same keys and values.

Grouped-Query Attention (GQA) splits the difference. You group the query heads and give each group its own K and V projection. Llama 3 uses 32 query heads with 8 KV heads, meaning every 4 query heads share one set of keys and values. The KV cache is 4× smaller than MHA, but each group retains its own expressive power. In practice, GQA nearly matches MHA quality while approaching MQA speed. It's the current standard.

# MHA: every head has its own K, V
# 32 heads → 32 K projections, 32 V projections
q = linear_q(x).view(B, T, 32, D)    # 32 query heads
k = linear_k(x).view(B, T, 32, D)    # 32 key heads
v = linear_v(x).view(B, T, 32, D)    # 32 value heads

# GQA: groups of query heads share K, V
# 32 query heads, 8 KV groups → 4 queries per group
q = linear_q(x).view(B, T, 32, D)          # 32 query heads
k = linear_k(x).view(B, T, 8, D)           # 8 key heads
v = linear_v(x).view(B, T, 8, D)           # 8 value heads
# Expand: repeat each KV head for its 4 query partners
k = k.unsqueeze(3).expand(B, T, 8, 4, D).reshape(B, T, 32, D)
v = v.unsqueeze(3).expand(B, T, 8, 4, D).reshape(B, T, 32, D)
# Now compute attention as usual — same math, 4× less cache

DeepSeek goes further with Multi-Head Latent Attention (MLA). Instead of caching full K and V tensors, it compresses them into a low-rank latent space. The KV cache stores these compressed representations, and the full K and V are reconstructed on the fly during attention. This squeezes even more memory savings out of the system at the cost of additional compute per token. When memory is your bottleneck—and during inference, it usually is—this tradeoff makes sense.

Sliding Window Attention

Full self-attention scales as O(n²) with context length. Double the context, quadruple the compute. For 128K tokens, that's brutal.

Sliding window attention restricts each token to attend only to the last W tokens—typically W=4096. This makes attention O(n·W) instead of O(n²). But doesn't that lose long-range information? Not entirely. Information propagates through layers. If layer 1 sees tokens 0–4096, and layer 2 sees tokens shifted by one layer's worth of propagation, then by layer 32, information from token 0 has had 32 chances to propagate forward through the residual stream.

Mistral uses this design cleverly: most layers use sliding window attention for speed, but a few layers use full attention to directly capture global dependencies. The combination gives you most of the quality of full attention at a fraction of the cost. The key insight is pragmatic: most useful context is local. You rarely need to attend to a token 50,000 positions ago. When you do, a few full-attention layers handle it.

Tokenizer Design & Vocabulary Size

Byte-Pair Encoding (BPE) is the dominant tokenization algorithm. It starts with individual bytes or characters, then iteratively merges the most frequent pair into a new token. After enough merges, you get a vocabulary of subword units that balances between whole words (efficient) and individual characters (flexible). SentencePiece is a popular library that implements BPE in a language-agnostic way—it treats the input as a raw byte stream, no language-specific preprocessing needed.

GPT models use byte-level BPE, which guarantees that any byte sequence can be tokenized. You'll never hit an "unknown token" error. tiktoken is OpenAI's optimized implementation—it's written in Rust and is significantly faster than the Python alternatives.

Vocabulary size is a genuine tradeoff. A larger vocabulary (100K–128K tokens) means common words and phrases get single tokens. Fewer tokens per text means faster inference—you generate fewer steps. But the embedding table grows linearly with vocabulary size, and rare tokens get less training signal. A smaller vocabulary (32K) produces more tokens per text but each token is better trained and the embedding table is smaller. Most modern models settle between 32K and 128K. Llama 3 uses 128K, betting that the inference speedup is worth the larger embeddings.

Tokenizer-free models are an active research direction. The idea: work directly with bytes or characters, eliminating the tokenizer entirely. This removes a whole class of problems (multilingual fairness, tokenization artifacts, the mismatch between tokens and words), but current byte-level models are slower and harder to train. Watch this space.

Training Decisions

The Chinchilla scaling laws showed that the optimal way to spend a compute budget is to scale model parameters and training tokens roughly equally. A 70B model should see about 1.4 trillion tokens. But Meta's Llama approach deliberately violates Chinchilla: they train smaller models for much longer. Why? Because training is a one-time cost, but inference happens millions of times. A 7B model trained on 2T tokens is "over-trained" by Chinchilla standards, but it's cheaper to serve than a Chinchilla-optimal 13B model—and it performs nearly as well.

BF16 (bfloat16) has become the default training dtype. It uses 8 exponent bits (same range as FP32) but only 7 mantissa bits (much less precision). That sounds lossy, but neural network training is surprisingly tolerant of reduced precision. The memory savings—half of FP32—let you train larger models or use larger batch sizes. FP16 is riskier because its smaller exponent range causes overflow during training.

For learning rate scheduling, the classic choice is a cosine schedule: warm up, then decay following a cosine curve to near zero. A newer alternative is WSD (Warmup-Stable-Decay): warm up, hold constant for most of training, then decay sharply at the end. WSD is simpler to tune and allows you to decide the total training duration later—you just start the decay phase when you're ready to stop. Gradient clipping (typically at norm 1.0) prevents rare batches of unusual data from causing catastrophic parameter updates.

Data mixing is more art than science. Modern models train on a blend of web text, code, books, academic papers, and conversation data. The ratios matter enormously. Too much code makes the model overly literal. Too much web text makes it shallow. MuP (Maximal Update Parameterization) helps here: it's a principled way to transfer hyperparameters from small test runs to full-scale training, so you can experiment with data mixes cheaply on a small model and trust the results will transfer.

Tied vs Untied Embeddings

Tied embeddings means the input embedding matrix and the output projection (the "LM head" that predicts the next token) share the same weights. This was common in earlier models. The logic: both layers map between token IDs and vector space, so sharing weights acts as regularization and saves parameters.

Most large modern models untie these matrices. The input embedding converts a token ID into a rich contextual representation. The output head converts a contextual representation into a probability distribution over tokens. These are related but not identical tasks. At scale, separate parameters consistently improve quality, and the cost is modest relative to a 70B-parameter model.

When training models this large, a single GPU isn't enough. Tensor parallelism splits individual matrix multiplications across GPUs—each GPU computes a slice of the output and they combine results. Pipeline parallelism assigns different layers to different GPUs, passing activations forward through a pipeline. In practice, you use both simultaneously: tensor parallelism within a node (fast NVLink connections between GPUs), pipeline parallelism across nodes (slower network connections). The tied-vs-untied decision interacts with parallelism—untied embeddings are easier to shard because the input and output matrices can live on different GPU groups.

These ten decisions—normalization, activation, position encoding, context extension, attention grouping, window attention, tokenizer, training strategy, and embedding tying—form the modern LLM recipe. None are permanent. Every one was different five years ago, and every one might change again. But the reasoning behind each choice transfers. Learn the tradeoffs, not just the answers.
