Agentic AI Systems

Chapter 12: Large Language Models 15 subtopics
TL;DR

A chatbot answers questions. An agent takes actions. The difference is a loop: perceive the environment, reason about what to do, call a tool, observe the result, repeat. The ReAct pattern interleaves thinking and acting. Planning strategies let agents tackle multi-step tasks. Memory systems give agents persistence across sessions. Multi-agent architectures let specialists collaborate. Code-generation agents can read issues and ship patches. Browser agents can navigate the web. All of this is powerful and all of it is fragile — agents today are 60–80% reliable on complex tasks, not 99%. Safety, sandboxing, and cost management aren't afterthoughts. They're the whole engineering challenge.

Why Agents

Ask a chatbot to book you a flight. It gives you a helpful paragraph about how to book flights. That's the gap. A language model alone is an oracle — it knows things, it can reason about things, but it can't do things. It lives inside a text box. The real world — your calendar, your browser, your codebase, your email — is on the other side of the glass.

An agent is what happens when you give a language model hands. Instead of just generating text, the model can call functions, execute code, click buttons, read files, and interact with APIs. The shift is from "language model as oracle" to "language model as actor." The model doesn't just tell you the answer — it goes and gets the answer, or does the thing, or builds the thing. This is the most consequential change in how we use LLMs since fine-tuning.

Why now? Two capabilities converged. First, models got good enough at reasoning to decompose tasks into steps. Second, function calling gave models a structured way to invoke external tools. Put those together and you get a system that can plan, act, observe, and adapt. That's an agent.

Core Agent Architecture

Every agent follows the same fundamental loop: perceive, reason, act. The agent loop is just that cycle repeated until the task is done or a stop condition is met.

Perception is how the agent observes its environment. It reads the user's message, views the current state of a browser, reads file contents, or receives the output from a previous tool call. Everything the agent knows about the world comes through perception.

Reasoning is where the LLM does its work. Given what it perceives, it thinks about what to do next. This is often chain-of-thought reasoning — the model writes out its thinking step by step before deciding on an action. The quality of this reasoning is what separates a good agent from a flailing one.

Action is the agent executing something in the world. It calls a function, writes a line of code, sends an API request, clicks a button in a browser. The action produces an observation — the function's return value, the compiler's output, the new state of the page — and that observation feeds back into the next perception step.

The loop repeats. Perceive the result of your last action. Reason about what to do next. Act. The agent keeps going until it decides the task is complete, hits a maximum step limit, or encounters an error it can't recover from. Every agent system you'll ever see — from a simple chatbot with tools to a fully autonomous coding agent — is a variation on this loop.

Tool Use & Function Calling

Function calling is the mechanism that makes agents possible. Without it, the model's only output is text. With it, the model can output structured JSON that the runtime interprets as a command. The model says {"function": "search_web", "args": {"query": "flights to Tokyo May 2025"}}, and the system actually calls that function and returns the results.

How does a model learn to do this? During training, models see special tokens that delimit tool calls. The supervised fine-tuning (SFT) data includes thousands of examples where the correct response is a tool call rather than a text response. The model learns when to call tools, which tool to pick, and how to format the arguments. OpenAI popularized a specific format where you declare tools in the system prompt with JSON schemas, and the model responds with a tool_calls field containing the function name and arguments.

The runtime side is straightforward. You maintain a loop: send messages to the model, check if the response contains tool calls, execute those calls, append the results as tool-response messages, and send the updated conversation back to the model. The model then decides whether to make another tool call or respond to the user.

import openai, json

tools = [
    {"type": "function", "function": {
        "name": "search_web",
        "description": "Search the web for current information",
        "parameters": {
            "type": "object",
            "properties": {"query": {"type": "string"}},
            "required": ["query"]
        }
    }}
]

def search_web(query: str) -> str:
    # In production, call a real search API
    return f"Top result for '{query}': ..."

tool_map = {"search_web": search_web}

messages = [{"role": "user", "content": "What's the weather in Tokyo?"}]

# The tool-use loop
while True:
    response = openai.chat.completions.create(
        model="gpt-4o", messages=messages, tools=tools
    )
    msg = response.choices[0].message
    messages.append(msg)

    if not msg.tool_calls:
        print(msg.content)  # Final text response
        break

    for call in msg.tool_calls:
        fn = tool_map[call.function.name]
        result = fn(**json.loads(call.function.arguments))
        messages.append({
            "role": "tool",
            "tool_call_id": call.id,
            "content": result
        })

That's the entire skeleton. Every agent framework — LangChain, LlamaIndex, AutoGen — wraps this loop with convenience methods, error handling, and state management. But underneath, it's always this: model generates a call, runtime executes it, result goes back to the model.

The ReAct Pattern

The ReAct pattern (Reasoning + Acting) is the foundational agent design. The idea, introduced by Yao et al. in 2022, is simple: let the model interleave thinking and acting. Instead of blindly calling tools or silently reasoning, the model explicitly writes out its thought process, then takes an action, then observes the result, then thinks again.

A ReAct trace looks like this:

Thought: The user wants to know who won the 2024 Nobel Prize
         in Physics. I don't have this information. I should
         search the web.
Action:  search_web("2024 Nobel Prize in Physics winner")
Observation: The 2024 Nobel Prize in Physics was awarded to
         John Hopfield and Geoffrey Hinton for foundational
         discoveries in machine learning with artificial
         neural networks.
Thought: I now have the answer. Hopfield and Hinton won for
         machine learning and neural network contributions.
         I can respond to the user.
Answer:  The 2024 Nobel Prize in Physics was awarded to John
         Hopfield and Geoffrey Hinton for their foundational
         work on machine learning with artificial neural
         networks.

Why does this beat the alternatives? If you only act without thinking, the agent makes impulsive tool calls with no strategy — it might search for the wrong thing three times before stumbling on the answer. If you only think without acting, the model hallucinates facts it doesn't have. ReAct gets the best of both: the reasoning grounds the tool choice, and the tool results ground the reasoning. You also get interpretability for free — you can read the thought steps to understand why the agent did what it did.

def react_agent(question: str, tools: dict, max_steps: int = 10):
    messages = [
        {"role": "system", "content": (
            "You are a ReAct agent. For each step, output:\n"
            "Thought: your reasoning\n"
            "Action: tool_name(args) OR Answer: final response"
        )},
        {"role": "user", "content": question}
    ]

    for step in range(max_steps):
        response = llm_call(messages)
        messages.append({"role": "assistant", "content": response})

        if "Answer:" in response:
            return parse_answer(response)

        action = parse_action(response)  # extract tool + args
        observation = tools[action.name](**action.args)
        messages.append({
            "role": "user",
            "content": f"Observation: {observation}"
        })

    return "Max steps reached without answer."

Planning: Plan-and-Execute

ReAct works one step at a time. For complex tasks — "research this topic, write a report, format it as a PDF" — one-step-at-a-time thinking often loses the thread. The plan-and-execute pattern addresses this by splitting the work into two phases. First, the agent creates a full plan. Then it executes each step, potentially revising the plan if something goes wrong.

The planning phase produces a structured list of steps. The execution phase works through them sequentially, feeding the result of each step into the next. If step 3 fails, the agent can re-plan from that point rather than starting over. This is fundamentally more robust for multi-step workflows.

Hierarchical Task Networks (HTN) take this further. A high-level task like "deploy the application" gets decomposed into subtasks: "run tests," "build the container," "push to registry," "update the deployment." Each subtask can be further decomposed. The agent works at the right level of abstraction, delegating details to sub-plans. This mirrors how humans tackle complex projects — you don't write every line of code in your head before starting. You plan at a high level and fill in details as you go.

The tradeoff is latency. Creating a plan requires an extra LLM call upfront. For simple tasks — "what's the weather?" — planning is overkill. For tasks with five or more steps, it's almost always worth it.

Reflexion

Agents fail. They write buggy code, call the wrong API, misinterpret instructions. Reflexion is a pattern where the agent learns from its own failures within a single task. After a failed attempt, the agent writes a reflection — a short analysis of what went wrong and what to try differently. That reflection gets added to the context for the next attempt.

It's exactly what you do when debugging. You run the code, see a traceback, think "oh, I used the wrong argument order," and fix it. Reflexion formalizes that self-correction loop. The agent might write: "My previous code failed because I assumed the API returns a list, but it returns a dictionary. I need to access the 'results' key." The next attempt succeeds because the agent has that explicit lesson in context.

The results are dramatic. On HumanEval (a code generation benchmark), reflexion improved pass rates from around 67% to over 88% by giving the agent up to three attempts with reflection. The cost is more LLM calls, but the success rate jump usually justifies it. The broader lesson: letting agents retry with self-feedback is almost always better than one-shot attempts.

Memory Systems

An agent without memory is like a goldfish with tools. It can act, but it can't learn or remember. Real agents need multiple kinds of memory.

Short-term memory is the conversation context window. Everything the agent has seen in the current session — user messages, tool results, its own thoughts — lives here. The problem: context windows are finite. Even a 128K token window fills up fast when you're stuffing in file contents and API responses.

Working memory (or scratchpad) is an explicit note-taking space. The agent writes structured notes to itself — key findings, intermediate results, current plan status. This is more efficient than relying on the raw conversation history because the agent curates what matters. Think of it as the agent's equivalent of scribbling on a whiteboard.

Long-term memory persists across sessions. This typically uses a vector store where the agent can save and retrieve information using semantic similarity search. Two flavors matter. Episodic memory records what happened — "last time the user asked about the billing API, the endpoint was /api/v2/billing." Semantic memory stores general knowledge — "this codebase uses pytest for testing and follows the repository pattern."

MemGPT is an elegant approach that treats the context window like virtual memory in an operating system. There's a main context (like RAM) that the agent actively uses, and an overflow storage (like disk) for older information. The agent can explicitly page information in and out, deciding what to keep in its limited "working set." This lets agents handle tasks that span far more information than fits in a single context window.

The Model Context Protocol (MCP) standardizes how LLMs connect to external data sources and tools. Instead of each agent framework inventing its own integration layer, MCP defines a common protocol. A tool provider implements the MCP server interface, and any MCP-compatible agent can use it. Think of it like USB for AI tools — a universal plug that lets any agent connect to any data source without custom wiring.

Multi-Agent Systems

A single agent has limits. Its context window constrains how much it can hold in mind. Its system prompt constrains its persona and expertise. Multi-agent systems break complex workflows into pieces handled by specialized agents.

The supervisor-worker pattern is the most common. One agent (the supervisor) receives the task, creates a plan, and delegates subtasks to specialist agents. A coding agent handles code changes. A research agent handles information gathering. A review agent checks the work. The supervisor orchestrates the flow and assembles the final result.

def supervisor_agent(task: str, specialists: dict):
    """A supervisor that delegates to specialist agents."""
    # Step 1: Plan the work
    plan = llm_call(
        f"Break this task into subtasks for these specialists: "
        f"{list(specialists.keys())}.\nTask: {task}"
    )
    subtasks = parse_plan(plan)  # [{agent, task}, ...]

    results = {}
    for subtask in subtasks:
        agent_name = subtask["agent"]
        agent_fn = specialists[agent_name]
        # Each specialist runs its own ReAct loop
        result = agent_fn(
            subtask["task"],
            context=results  # pass prior results as context
        )
        results[subtask["id"]] = result

    # Step 3: Synthesize final answer
    summary = llm_call(
        f"Combine these results into a final response:\n"
        f"{json.dumps(results, indent=2)}"
    )
    return summary

# Usage
specialists = {
    "researcher": research_agent,
    "coder": coding_agent,
    "reviewer": review_agent,
}
result = supervisor_agent(
    "Add input validation to the /users endpoint", specialists
)

The agent debate pattern has multiple agents argue about a problem. Agent A proposes an answer. Agent B critiques it. Agent A refines. This adversarial dynamic catches errors that a single agent might miss, similar to how code review catches bugs the author overlooked.

The swarm pattern uses handoffs instead of a central supervisor. Agent A handles the conversation until it detects the topic is outside its expertise, then hands off to Agent B. This is natural for customer service workflows where a general-purpose agent routes to billing, technical support, or sales specialists.

Frameworks like CrewAI and AutoGen make these patterns easy to implement. CrewAI uses a role-based metaphor — you define agents with roles, goals, and backstories, then assemble them into a crew. AutoGen focuses on multi-agent conversations where agents message each other in structured patterns.

The honest tradeoff: multi-agent adds latency, cost, and coordination complexity. Two agents means at least twice the LLM calls. The supervisor adds more. Use multi-agent when the task genuinely requires different expertise or when the context for a single agent would overflow. For simpler tasks, one well-prompted agent is faster, cheaper, and more predictable.

Code Generation Agents

This is where agents get real. A code-generation agent reads a GitHub issue, navigates a codebase, writes a patch, runs the tests, and iterates until the tests pass. The autonomous coding loop is: understand the problem, locate the relevant code, edit it, test it, fix failures, repeat.

SWE-bench is the gold-standard benchmark. It contains 2,294 real GitHub issues from popular Python repositories like Django and scikit-learn, paired with the human-written pull requests that fixed them. To solve a SWE-bench task, an agent must understand the issue, navigate a large unfamiliar codebase, write the correct fix, and make the existing tests pass. Top agents solve around 40–50% of tasks as of early 2025 — impressive but far from human-level.

Real-world examples are maturing fast. GitHub's Copilot coding agent can be assigned an issue and autonomously open a pull request. Devin (by Cognition) was one of the first "AI software engineer" demos. OpenHands (formerly OpenDevin) is an open-source agent that operates inside a sandboxed development environment. These systems all use variations of the same loop: read the issue, explore the codebase with grep and file reads, form a hypothesis, edit files, run tests, fix errors based on test output.

The skill library pattern is a powerful extension. As the agent solves problems, it saves reusable tools — scripts, templates, common fix patterns — into a library. Next time a similar task comes up, the agent can retrieve and apply a previously learned skill instead of solving from scratch. This is how agents get faster over time.

Browser & Computer Use Agents

A browser agent controls a web browser. It sees a webpage (either as a screenshot or as the accessibility tree / DOM), decides what to do (click a link, fill a form, scroll down), takes that action, and observes the new page state. A computer use agent extends this to the entire desktop — it can open applications, move files, use terminal commands.

This is brutally hard. The visual state space is enormous — every webpage looks different, and layouts change between visits. Actions are often irreversible — you can't un-submit a form. Latency is high — every action requires rendering a page and waiting for network requests. And the agent needs to handle pop-ups, CAPTCHAs, cookie banners, and all the messy reality of the web.

Two approaches exist. GUI agents work with screenshots and pixel coordinates. The model sees a screenshot, identifies UI elements visually, and outputs click coordinates. This is general-purpose (it works on any application) but slow and error-prone. API-first agents skip the browser entirely and call APIs directly. This is faster and more reliable, but only works when APIs exist and you have access.

Benchmarks like WebArena test browser agents on realistic web tasks — shopping on e-commerce sites, managing content in a CMS, navigating forums. Current agents score 15–35% on these benchmarks. The gap between "demo that works on a curated example" and "system that handles the messy real web" is still wide.

Agent Frameworks

You don't have to build agents from scratch. Several frameworks handle the plumbing. Here's the landscape.

LangChain is the most widely used. It started with the concept of chains — linear sequences of LLM calls — and evolved to support agents with tool use. LCEL (LangChain Expression Language) lets you compose components declaratively. The ecosystem is huge, with integrations for hundreds of tools and LLM providers. The downside is complexity — LangChain has a lot of abstraction layers, and debugging can feel like archaeology.

LangGraph extends LangChain with stateful, graph-based workflows. Instead of a linear chain, you define nodes (agent steps) and edges (transitions), including conditional routing. This is ideal for complex agents that need to branch, loop, or run steps in parallel. State is explicitly managed and checkpointed.

LlamaIndex focuses on retrieval. If your agent's primary job is answering questions over documents, LlamaIndex provides excellent indexing, chunking, and retrieval abstractions. It's less general than LangChain for tool-use agents, but superior for RAG-heavy workflows.

AutoGen (from Microsoft) is built around multi-agent conversations. Agents message each other in turn-based or asynchronous patterns. It's the natural choice when your design involves multiple agents collaborating.

CrewAI takes a role-based approach. You define agents with roles ("Senior Researcher," "Technical Writer"), assign them tasks, and let them collaborate. The metaphor makes complex workflows intuitive to set up.

How do you choose? If you need maximum flexibility and ecosystem support, LangChain or LangGraph. If your problem is document Q&A, LlamaIndex. If you're building multi-agent systems, AutoGen or CrewAI. If you want to learn how agents work under the hood, build one from scratch first — it's surprisingly little code.

Orchestration & Production

Getting an agent to work in a demo is easy. Getting it to work in production is an engineering problem. Several concerns emerge.

State management across agent steps is the first challenge. An agent might execute fifteen steps to complete a task. Each step changes the state of the world — files are edited, APIs are called, databases are updated. You need to track this state explicitly so you can resume, retry, or roll back.

Checkpointing and recovery handle the inevitable failures. What if the agent crashes mid-task? What if the LLM API times out on step 8 of 12? Without checkpointing, you start over from scratch. With it, you resume from the last successful step. LangGraph bakes this in — every node transition is automatically checkpointed.

Observability means seeing what the agent is doing, in detail, after the fact. Distributed tracing tools like LangSmith or Arize Phoenix record every LLM call, tool invocation, and decision point. When an agent produces a wrong answer, you can trace back through its reasoning to find exactly where it went off the rails. This is non-negotiable for production systems.

Cost tracking per step matters because agent workflows consume many LLM calls. A single user request might trigger 5–20 model invocations. Without per-step cost tracking, you'll be surprised by your bill. Log token usage, model name, and latency for every call.

Semantic caching avoids redundant LLM calls. If the agent asks the same question (or a semantically similar one) twice, serve the cached result instead of making another API call. This is especially valuable for research agents that might search for similar queries across different tasks.

Safety & Sandboxing

Here's the part that keeps me up at night. An agent has real-world side effects. It can delete files, send emails, make purchases, push code. A chatbot that hallucinates gives you wrong text. An agent that hallucinates performs wrong actions. The stakes are categorically different.

Sandboxing is the first layer. Run agent-generated code inside containers with restricted file system access, no network access (or an allowlist), and resource limits (CPU, memory, time). Docker containers or microVMs (like Firecracker) are the standard approach. The agent can do whatever it wants inside its sandbox — it just can't touch anything outside it.

Human-in-the-loop is the second layer. Before executing a destructive action — deleting a file, sending an email, making a purchase, merging a PR — the agent pauses and asks for human approval. The classification of what counts as "destructive" is itself a design decision. A conservative system asks for approval on everything. A permissive system only gates truly irreversible actions.

Action gating formalizes this. Every action the agent wants to take gets classified as safe (proceed automatically), sensitive (require approval), or forbidden (never execute). A separate classifier — which can be a simpler, faster model — makes this decision before the action reaches the runtime. This provides defense in depth: even if the agent's reasoning goes wrong, the safety layer catches dangerous actions.

Rate limiting and budget controls prevent runaway agents. Set a maximum number of LLM calls per task, a maximum dollar spend per task, and a maximum wall-clock time. An agent stuck in a loop burning tokens at $0.10 per call will rack up real costs fast if you don't set a ceiling.

Defense in depth is the principle. No single safety mechanism is enough. You layer sandboxing, human-in-the-loop, action gating, rate limiting, and monitoring on top of each other. Any one of them might have gaps. Together, they form a robust barrier.

Evaluation & Benchmarks

How do you know if your agent is any good? This is harder than evaluating a classifier. A classifier either gets the label right or wrong. An agent might take a weird path but arrive at the correct answer, or take a sensible path but produce a wrong result.

Trajectory evaluation examines the steps the agent took. Did it call reasonable tools? Did it avoid unnecessary actions? Did its reasoning make sense? This requires either human reviewers or an LLM-as-judge that scores the trajectory.

Task completion is the bottom line: did the agent achieve the goal? For coding agents, this means: do the tests pass? For research agents: is the answer correct? For browser agents: did it end up on the right page with the right form filled in? Task completion is the metric that matters most, but it's binary and doesn't tell you why the agent failed.

Key benchmarks include SWE-bench (coding), WebArena (browser tasks), and GAIA (general AI assistants, requiring multi-step reasoning with tools). These benchmarks are hard. State-of-the-art agents don't crack 50% on most of them. That's the honest state of the field.

The reliability problem is real. Run the same agent on the same task ten times and you'll get different results. Agents are non-deterministic — different reasoning paths, different tool call orders, different failures. Any serious evaluation must run multiple trials and report confidence intervals, not single-run numbers. A claim of "85% success rate" based on one trial per task is noise, not signal.

Reliability & Cost Management

Let's talk about money and reality. Agents are expensive. A single task might require 5–20 LLM calls. At $0.01–$0.10 per call (depending on model and token count), one task can cost $0.05–$2.00. Scale that to thousands of tasks per day and the bill adds up fast.

The first strategy is model routing. Use a smaller, cheaper model (like GPT-4o-mini or Claude Haiku) for easy decisions — "which file should I look at?" — and a larger model (like GPT-4o or Claude Sonnet) for hard decisions — "what's the correct fix for this bug?" A routing layer (which can itself be a small model or a classifier) decides which model to invoke for each step. This can cut costs by 50–70% with minimal quality loss.

Caching helps a lot. If the agent runs similar workflows repeatedly, many intermediate queries will be identical or near-identical. A semantic cache (using embedding similarity) can serve cached results for queries that are close enough, avoiding the LLM call entirely.

Timeouts and retry budgets prevent waste. If the agent hasn't made progress in five steps, stop and report failure rather than burning tokens in a loop. Set a maximum retry count for reflexion-style self-correction — three attempts is usually enough. Diminishing returns hit hard after that.

The cold reality: most agent systems today achieve 60–80% reliability on complex, real-world tasks. Not 99%. Not 95%. Building with agents means building with this uncertainty. Design for graceful failure. Have fallback paths. Make it easy for humans to pick up where the agent left off. The systems that work in production aren't the ones with the smartest agents — they're the ones with the best safety nets around imperfect agents.

That said, the trajectory is clear. Two years ago, autonomous coding agents barely existed. Today they're shipping real pull requests to real codebases. The reliability will improve. The costs will drop. The architectures described in this section — ReAct, planning, reflexion, multi-agent collaboration, memory, safety layers — are the foundation that everything else is being built on. Learn them now. You'll need them.
