Agentic AI

Chapter 12: Large Language Models 10 subtopics
TL;DR

LLMs alone answer questions. Agents take actions. An agent is an LLM wired into a perceive-reason-act loop with tool access and memory. Function calling lets models output structured JSON that your code dispatches to real tools. The ReAct pattern interleaves reasoning and action. Planning strategies range from one-step-at-a-time (ReAct) to full upfront plans (Plan-and-Execute) to self-correcting loops (Reflexion). Memory systems give agents persistence beyond the context window. Multi-agent architectures let specialized agents collaborate on complex tasks. Frameworks like LangGraph, CrewAI, and AutoGen accelerate development, but production reliability remains the hard problem — agents compound errors across tool calls, and every loop iteration costs tokens. This section covers the architecture, patterns, code, and the sharp edges you’ll hit when you move from chatbot to autonomous agent.

Why Agents

Here’s the difference in one sentence: a chatbot tells you what to do, an agent does it.

Ask an LLM “what’s the current price of AAPL?” and it’ll give you a stale number from its training data, or honestly say it doesn’t know. Ask an agent the same question, and it calls a stock API, parses the response, and gives you the live price. The LLM is the same. The difference is the loop around it.

# Plain LLM: gives you advice
response = llm("How do I find rows with null emails in the users table?")
# → "You can run: SELECT * FROM users WHERE email IS NULL;"

# Agent: actually does it
response = agent("Find rows with null emails in the users table")
# → Thinks: "I need to query the database"
# → Calls: execute_sql("SELECT * FROM users WHERE email IS NULL;")
# → Observes: 47 rows returned
# → Responds: "Found 47 users with null emails. Here are the first 10..."

The formula is simple:

Agent = LLM + Tools + Planning Loop

The LLM provides reasoning. Tools provide capability — search, code execution, API calls, database access. The planning loop is the glue: it decides when to think, when to act, and when to stop. Without tools, the LLM is stuck in its own head. Without the planning loop, tool access is just a fancier API wrapper. You need all three.

Why now? Agents aren’t a new idea — early attempts go back to AutoGPT in 2023. What changed is model capability. GPT-4-class models are reliable enough at function calling and multi-step reasoning that agents now actually work often enough to be useful in production. The jump from “works 30% of the time” to “works 85% of the time” is the difference between a demo and a product.

Core Agent Architecture

Every agent, no matter the framework, follows the same fundamental loop: perceive → reason → act. The LLM is the “brain,” tools are the “hands.”

# The universal agent loop
#
#   ┌──────────────────────────────────────────┐
#   │              SYSTEM PROMPT               │
#   │  "You are an agent with these tools..."  │
#   └──────────────┬───────────────────────────┘
#                  │
#                  ▼
#   ┌──────────────────────────┐
#   │   PERCEIVE               │ ◄── user input, tool results,
#   │   (read current state)   │     environment observations
#   └──────────┬───────────────┘
#              │
#              ▼
#   ┌──────────────────────────┐
#   │   REASON                 │ ◄── LLM processes context,
#   │   (think about action)   │     decides what to do next
#   └──────────┬───────────────┘
#              │
#              ▼
#   ┌──────────────────────────┐
#   │   ACT                    │ ◄── call a tool, or respond
#   │   (execute decision)     │     to the user
#   └──────────┬───────────────┘
#              │
#              ▼
#         Done? ──No──► back to PERCEIVE
#           │
#          Yes
#           │
#           ▼
#     Return final answer

The key components that make this loop work:

System prompt. This is the agent’s identity and instruction set. It defines what tools are available, how to use them, what the agent should and shouldn’t do, and the format for tool calls. A well-written system prompt is the difference between an agent that reliably completes tasks and one that flails around randomly calling tools.

Tool definitions. Structured descriptions of each tool — name, parameters with types, description of what it does. The LLM uses these to decide which tool to call and how to call it. Poor tool descriptions lead to poor tool selection.

Memory. At minimum, the conversation history. In more sophisticated agents, this includes a working scratchpad for the current task and long-term storage of past interactions. We’ll go deep on memory systems later.

Planning strategy. How the agent decides what to do next. Does it plan the entire task upfront? Take it one step at a time? Reflect on failures? This is where most of the design variation lives.

Tool Use & Function Calling

Function calling is the mechanism that turns an LLM from a text generator into an agent. Instead of outputting natural language, the model outputs structured JSON specifying which function to call and with what arguments. Your application code then executes that function and feeds the result back to the model.

The flow looks like this:

# Function calling flow
#
# 1. You send: message + tool definitions
#    ┌─────────────────────────────────┐
#    │  messages: "What's the weather   │
#    │            in Tokyo?"            │
#    │  tools: [get_weather, ...]       │
#    └──────────────┬──────────────────┘
#                   │
#                   ▼
# 2. Model responds with a tool call (NOT natural language)
#    ┌─────────────────────────────────┐
#    │  tool_call:                      │
#    │    name: "get_weather"           │
#    │    arguments: {"city": "Tokyo"}  │
#    └──────────────┬──────────────────┘
#                   │
# 3. YOUR CODE executes the function
#                   │
#                   ▼
# 4. You send the result back to the model
#    ┌─────────────────────────────────┐
#    │  tool_result: {"temp": 22,       │
#    │    "condition": "partly cloudy"} │
#    └──────────────┬──────────────────┘
#                   │
#                   ▼
# 5. Model generates final natural language response
#    "It's currently 22°C and partly cloudy in Tokyo."

OpenAI Function Calling Format

OpenAI pioneered the function calling API. You define tools as JSON schemas, and the model decides when and how to invoke them:

import openai

tools = [
    {
        "type": "function",
        "function": {
            "name": "get_stock_price",
            "description": "Get the current stock price for a given ticker symbol",
            "parameters": {
                "type": "object",
                "properties": {
                    "ticker": {
                        "type": "string",
                        "description": "Stock ticker symbol, e.g. 'AAPL', 'GOOGL'"
                    }
                },
                "required": ["ticker"]
            }
        }
    },
    {
        "type": "function",
        "function": {
            "name": "execute_sql",
            "description": "Execute a read-only SQL query against the database",
            "parameters": {
                "type": "object",
                "properties": {
                    "query": {
                        "type": "string",
                        "description": "SQL SELECT query to execute"
                    }
                },
                "required": ["query"]
            }
        }
    }
]

response = openai.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "What's AAPL's stock price?"}],
    tools=tools,
    tool_choice="auto"  # let the model decide whether to call a tool
)

# Check if the model wants to call a function
message = response.choices[0].message
if message.tool_calls:
    tool_call = message.tool_calls[0]
    print(f"Function: {tool_call.function.name}")
    print(f"Args: {tool_call.function.arguments}")
    # → Function: get_stock_price
    # → Args: {"ticker": "AAPL"}

Anthropic Tool Use Format

Anthropic’s format is conceptually identical but structurally different. Tools are defined at the top level of the request, and results come back as tool_use content blocks:

import anthropic

client = anthropic.Anthropic()

response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=1024,
    tools=[
        {
            "name": "get_stock_price",
            "description": "Get the current stock price for a ticker symbol",
            "input_schema": {
                "type": "object",
                "properties": {
                    "ticker": {
                        "type": "string",
                        "description": "Stock ticker symbol"
                    }
                },
                "required": ["ticker"]
            }
        }
    ],
    messages=[{"role": "user", "content": "What's AAPL trading at?"}]
)

# Response contains a tool_use content block
for block in response.content:
    if block.type == "tool_use":
        print(f"Tool: {block.name}")      # → get_stock_price
        print(f"Input: {block.input}")     # → {"ticker": "AAPL"}
        print(f"Call ID: {block.id}")      # → toolu_abc123 (use this in result)

Complete Tool-Calling Loop

Here’s the full loop — send a message, execute the tool call, return the result, get the final answer:

import openai
import json

client = openai.OpenAI()

# Define tools
tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "Get current weather for a city",
        "parameters": {
            "type": "object",
            "properties": {
                "city": {"type": "string", "description": "City name"},
                "units": {"type": "string", "enum": ["celsius", "fahrenheit"],
                          "description": "Temperature units"}
            },
            "required": ["city"]
        }
    }
}]

# Actual function implementation
def get_weather(city: str, units: str = "celsius") -> dict:
    # In production, call a real weather API
    return {"city": city, "temp": 22, "units": units, "condition": "sunny"}

# Map function names to implementations
available_functions = {"get_weather": get_weather}

# Step 1: Send user message with tools
messages = [{"role": "user", "content": "What's the weather in Paris?"}]
response = client.chat.completions.create(
    model="gpt-4o", messages=messages, tools=tools
)
assistant_msg = response.choices[0].message
messages.append(assistant_msg)

# Step 2: If model made tool calls, execute them
if assistant_msg.tool_calls:
    for tool_call in assistant_msg.tool_calls:
        func_name = tool_call.function.name
        func_args = json.loads(tool_call.function.arguments)

        # Execute the function
        result = available_functions[func_name](**func_args)

        # Append the tool result to conversation
        messages.append({
            "role": "tool",
            "tool_call_id": tool_call.id,
            "content": json.dumps(result)
        })

    # Step 3: Get final response with tool results in context
    final_response = client.chat.completions.create(
        model="gpt-4o", messages=messages, tools=tools
    )
    print(final_response.choices[0].message.content)
    # → "It's currently 22°C and sunny in Paris!"

Parallel Function Calling

Models can request multiple tool calls in a single response. If a user asks “compare weather in Tokyo and London,” the model will emit two get_weather calls simultaneously. Your code should execute them in parallel:

import asyncio, json

async def execute_tool_calls(tool_calls, available_functions):
    """Execute multiple tool calls concurrently."""
    async def run_one(tc):
        func = available_functions[tc.function.name]
        args = json.loads(tc.function.arguments)
        result = await asyncio.to_thread(func, **args)  # run sync func in thread
        return {
            "role": "tool",
            "tool_call_id": tc.id,
            "content": json.dumps(result)
        }
    return await asyncio.gather(*[run_one(tc) for tc in tool_calls])

The Trust Boundary

Not all tools are created equal. Some are safe to auto-execute; others need a human in the loop.

Risk Level Examples Policy
Read-only Search, weather lookup, database SELECT Auto-execute freely
Limited write Create draft email, add calendar event Auto-execute with logging
Significant write Send email, make purchase, database INSERT/UPDATE Require human approval
Dangerous Delete records, execute arbitrary code, transfer funds Strict approval + confirmation
Never auto-execute destructive tools. An agent with unrestricted access to execute_sql can and will run DROP TABLE if it hallucinates that as the right action. Always enforce tool-level permissions. Read-only tools should be genuinely read-only at the infrastructure level, not just at the prompt level. Prompt-level restrictions are suggestions; code-level restrictions are guarantees.

The ReAct Pattern

ReAct (Reasoning + Acting) is the most widely used agent pattern, and for good reason: it’s simple, effective, and easy to debug. The idea is that the model alternates between thinking (reasoning in natural language) and acting (calling tools), with each observation feeding into the next thought.

# ReAct trace example:
#
# User: "How many employees joined after the company's last funding round?"
#
# Thought 1: I need to find when the last funding round was, then count
#            employees who joined after that date. Let me check the
#            funding_rounds table first.
#
# Action 1:  execute_sql("SELECT MAX(date) FROM funding_rounds")
# Observation 1: 2024-03-15
#
# Thought 2: Last funding was March 15, 2024. Now I need to count
#            employees with a start_date after that.
#
# Action 2:  execute_sql("SELECT COUNT(*) FROM employees
#                         WHERE start_date > '2024-03-15'")
# Observation 2: 23
#
# Thought 3: 23 employees joined after the last funding round. I have
#            the answer.
#
# Final Answer: 23 employees joined after the company's last funding
#               round (March 15, 2024).

Why does this work? Two reasons. First, forcing the model to write out its reasoning before each action reduces errors — it’s chain-of-thought prompting applied to action selection. Second, each observation grounds the model in reality. It’s not hallucinating a database schema; it’s reading real results.

Simple ReAct Implementation

import openai, json

client = openai.OpenAI()

SYSTEM_PROMPT = """You are a helpful agent with access to tools.

For each step, you MUST follow this format:
Thought: [your reasoning about what to do next]
Action: [tool call]

After receiving tool results, reason about them before taking the next action.
When you have enough information, provide your final answer."""

def react_loop(user_query: str, tools: list, available_functions: dict,
               max_steps: int = 10) -> str:
    """Execute a ReAct loop until the agent produces a final answer."""
    messages = [
        {"role": "system", "content": SYSTEM_PROMPT},
        {"role": "user", "content": user_query}
    ]

    for step in range(max_steps):
        response = client.chat.completions.create(
            model="gpt-4o",
            messages=messages,
            tools=tools,
            tool_choice="auto"
        )

        assistant_msg = response.choices[0].message
        messages.append(assistant_msg)

        # If no tool calls, the agent is done reasoning → return answer
        if not assistant_msg.tool_calls:
            return assistant_msg.content

        # Execute each tool call
        for tool_call in assistant_msg.tool_calls:
            func_name = tool_call.function.name
            func_args = json.loads(tool_call.function.arguments)

            print(f"  Step {step+1}: {func_name}({func_args})")

            try:
                result = available_functions[func_name](**func_args)
                result_str = json.dumps(result)
            except Exception as e:
                result_str = json.dumps({"error": str(e)})

            messages.append({
                "role": "tool",
                "tool_call_id": tool_call.id,
                "content": result_str
            })

    return "Agent reached maximum steps without a final answer."
The max_steps guard is non-negotiable. Without it, an agent stuck in a loop will keep calling tools forever — burning tokens and money. Always set a maximum iteration count, and consider adding a token budget limit too. In production, 5–15 steps covers most tasks. If an agent consistently needs more, the task may be better decomposed into sub-tasks.

Planning Strategies

How should an agent decide what to do? There’s no single answer. Different tasks call for different planning strategies.

Plan-and-Execute

Make a full plan upfront, then execute each step. A “planner” LLM call creates the plan, then an “executor” handles each step, potentially re-planning if something goes wrong.

# Plan-and-Execute flow:
#
# User: "Create a report on Q3 sales by region"
#
# PLANNING PHASE (one LLM call):
#   Plan:
#     1. Query total Q3 sales grouped by region
#     2. Query Q2 sales for comparison
#     3. Calculate quarter-over-quarter growth
#     4. Generate summary with key findings
#     5. Format as markdown report
#
# EXECUTION PHASE (one step at a time):
#   Execute step 1 → observe result → execute step 2 → ...
#   If step fails → re-plan remaining steps

ReAct (One Step at a Time)

No upfront plan. The agent decides the next action based on everything it’s seen so far. More flexible, better for tasks where you don’t know what you’ll find.

Reflexion

Try, fail, reflect on why you failed, then retry with the lesson learned. This adds a self-critique step that dramatically improves performance on tasks where the first attempt often isn’t perfect.

# Reflexion flow:
#
# Attempt 1:
#   Agent tries to solve the task → produces result
#   Evaluator checks result → FAIL (wrong output)
#
# Reflection:
#   "I failed because I used the wrong column name.
#    The table has 'revenue' not 'sales'. I should check
#    the schema first next time."
#
# Attempt 2:
#   Agent retries WITH the reflection in context
#   → checks schema first → uses correct column → SUCCESS

When to Use Which

Strategy Best For Weakness Steps Visibility
ReAct Exploratory tasks, unknown structure, Q&A Can lose track of long-term goal One step ahead
Plan-and-Execute Well-defined multi-step tasks, reports, pipelines Upfront plan can be wrong; replanning adds latency Full plan upfront
Reflexion Code generation, tasks with verifiable output Expensive (multiple full attempts); needs evaluator Full attempt + retry
ReAct + Plan Complex tasks needing both structure and flexibility Higher implementation complexity Plan, but flexible execution
Start with ReAct. It’s the simplest to implement, the easiest to debug (just read the trace), and handles 80% of agent tasks well enough. Move to Plan-and-Execute when your agents keep losing track of multi-step goals. Add Reflexion when you have a way to verify outputs and the task is hard enough that first attempts regularly fail.

Memory Systems

An agent without memory forgets everything between turns. That’s fine for single-shot tasks, but real-world agents need to remember context, learn from past interactions, and track progress on long tasks. Memory in agentic systems typically splits into three types:

Short-Term Memory: The Context Window

This is the conversation history — every message, tool call, and tool result that fits within the model’s context window. It’s “free” in the sense that you don’t need any extra infrastructure, but it’s bounded. A 128k-token context window sounds huge until an agent has made 15 tool calls, each returning a few thousand tokens of database results.

Working Memory: The Scratchpad

A dedicated space for the agent to track its current task state. Think of it like a notepad the agent writes to and reads from. Common implementations use a special “memory” tool or a system message that gets updated:

# Working memory as a tool
working_memory_tool = {
    "type": "function",
    "function": {
        "name": "update_scratchpad",
        "description": "Save important findings or intermediate results. "
                       "Use this to remember key facts across many steps.",
        "parameters": {
            "type": "object",
            "properties": {
                "key": {"type": "string", "description": "Label for this note"},
                "value": {"type": "string", "description": "The information to save"}
            },
            "required": ["key", "value"]
        }
    }
}

# The scratchpad contents get injected into the system prompt
scratchpad = {}

def update_scratchpad(key: str, value: str) -> str:
    scratchpad[key] = value
    return f"Saved: {key}"

def build_system_prompt():
    base = "You are a helpful agent..."
    if scratchpad:
        notes = "\n".join(f"- {k}: {v}" for k, v in scratchpad.items())
        base += f"\n\nYour working notes:\n{notes}"
    return base

Long-Term Memory: Retrieval-Augmented

For agents that operate across sessions — remember a user’s preferences, learn from past successes and failures, recall facts from weeks-old conversations — you need a persistent memory store. The standard approach is a vector database:

# Long-term memory with vector search
#
# STORING memories:
#   1. Agent completes a task or learns something important
#   2. Create a text summary of the memory
#   3. Embed it with an embedding model
#   4. Store embedding + text + metadata in vector DB
#
# RETRIEVING memories:
#   1. New user query arrives
#   2. Embed the query
#   3. Vector search for similar past memories
#   4. Inject top-k results into the agent's context
#
# ┌─────────────┐    embed    ┌──────────────┐
# │ "User prefers│ ─────────► │ Vector DB    │
# │  metric units│            │ (ChromaDB,   │
# │  for weather"│            │  Pinecone,   │
# └─────────────┘            │  Weaviate)   │
#                             └──────┬───────┘
#                                    │ similarity search
#                                    ▼
#                             ┌──────────────┐
#                             │ Top-k matches│
#                             │ injected into│
#                             │ agent context│
#                             └──────────────┘

The Memory Challenge

Memory in agents is an unsolved problem. Context windows are finite. Summarization loses detail. Vector search can miss relevant memories or retrieve irrelevant ones. And every token of memory you inject is a token you can’t use for reasoning.

Memory Type Persistence Capacity Retrieval Limitation
Short-term (context) Single session 128k tokens (model-dependent) Always available Finite, expensive per token
Working (scratchpad) Single task A few KB Always available Eats into context window
Long-term (vector DB) Persistent Unlimited Requires search query Retrieval quality varies
Practical tip: Most production agents work fine with just short-term memory (conversation context) plus a simple scratchpad. Long-term memory is powerful but adds significant complexity — embedding pipelines, vector DB ops, retrieval tuning. Add it when you have a clear use case (e.g., a personal assistant that needs to remember user preferences across sessions), not as a default.

Multi-Agent Systems

Sometimes one agent isn’t enough. Complex tasks benefit from multiple specialized agents working together, each with its own tools, system prompt, and expertise. Think of it like a team: you wouldn’t ask one person to be the researcher, writer, reviewer, and publisher all at once.

Supervisor Pattern

One “manager” agent receives the user’s request, breaks it into subtasks, and delegates to specialized worker agents. The supervisor decides which worker to call, how to combine results, and when the task is complete.

# Supervisor pattern
#
#                  ┌──────────────┐
#     User ──────► │  Supervisor  │ ◄── decides who to call
#                  └──┬───┬───┬──┘
#                     │   │   │
#            ┌────────┘   │   └────────┐
#            ▼            ▼            ▼
#     ┌────────────┐ ┌─────────┐ ┌──────────┐
#     │ Researcher │ │ Analyst │ │  Writer  │
#     │ (search,   │ │ (SQL,   │ │ (format, │
#     │  browse)   │ │  calc)  │ │  draft)  │
#     └────────────┘ └─────────┘ └──────────┘

Debate Pattern

Multiple agents tackle the same problem independently, then “debate” their solutions. Useful for tasks where you want high confidence — if three agents independently reach the same conclusion, it’s probably right.

Assembly Line (Sequential Pipeline)

Each agent handles one stage, passing its output to the next. Like a manufacturing pipeline: researcher → drafter → reviewer → editor. Each stage is focused and well-defined.

# Assembly line pattern
#
# User input
#     │
#     ▼
# ┌───────────────┐   ┌───────────────┐   ┌───────────────┐
# │  Researcher   │──►│    Drafter    │──►│   Reviewer    │──► Final output
# │  (gathers     │   │  (writes      │   │  (critiques,  │
# │   facts)      │   │   content)    │   │   improves)   │
# └───────────────┘   └───────────────┘   └───────────────┘

When Multi-Agent Helps vs. Single Agent

Use Multi-Agent When Stick to Single Agent When
Task requires diverse specialized skills Task is straightforward, even if multi-step
You need checks and balances (reviewer, critic) Speed matters more than quality
Different stages need different tools/models Context sharing between stages is critical
You want parallelizable subtasks You want simpler debugging and lower cost
Multi-agent complexity is real. Every agent hop adds latency, cost, and potential points of failure. A supervisor that delegates to three workers makes at least four LLM calls minimum. Inter-agent communication is lossy — summaries lose detail. Start with a single well-prompted agent. Only go multi-agent when you’ve hit a wall that can’t be solved by better prompting or tool design.

Agent Frameworks

You can build agents from scratch — the ReAct loop above is maybe 40 lines of code. But as you add memory, multi-agent coordination, error recovery, streaming, and observability, framework support helps. Here’s the landscape:

Framework Best For Complexity Maturity Key Feature
LangGraph Complex stateful agents, custom workflows High Mature Graph-based workflows with state machines
LangChain Rapid prototyping, RAG chains Medium Mature Extensive integrations ecosystem
CrewAI Multi-agent role-based collaboration Low Growing Role/goal/backstory per agent
AutoGen (Microsoft) Multi-agent conversations, research Medium Growing Flexible conversation patterns
OpenAI Assistants API Simple tool-using agents, managed hosting Low Mature Managed threads, retrieval, code interpreter
Smolagents (HuggingFace) Lightweight agents, open-source models Low Early Minimal abstraction, code agents
The honest take. If your agent is simple (single model, few tools, ReAct loop), build it yourself — it’s 50–100 lines and you understand every line. Use LangGraph when you need complex control flow (branching, cycles, human-in-the-loop gates). Use CrewAI when you want multi-agent with minimal code. Use the OpenAI Assistants API when you want zero infrastructure and are already on OpenAI. The framework choice matters less than your tool definitions and system prompt quality.

Production & Reliability

Here’s the uncomfortable truth about agents in production: they fail more than simple LLM calls. A lot more. And the failure modes are different — instead of a wrong answer, you get an agent stuck in a loop, or calling the wrong tool, or hallucinating an API that doesn’t exist.

The Compounding Error Problem

Every step in an agent’s execution is a potential failure point. If each step has a 95% success rate (which is good for LLMs), a 10-step agent has a 0.9510 = 60% success rate. That’s the math that makes agent reliability hard.

# Compounding error rates
#
# Steps    95% per step    90% per step    99% per step
# ─────    ────────────    ────────────    ────────────
#    3        85.7%           72.9%           97.0%
#    5        77.4%           59.0%           95.1%
#   10        59.9%           34.9%           90.4%
#   15        46.3%           20.6%           86.0%
#   20        35.8%           12.2%           81.8%
#
# Takeaway: make each step as reliable as possible, and
# minimize the number of steps needed.

Cost Management

Agent loops are expensive. Each iteration makes at least one LLM call (often with a large context), and tool results add tokens. A complex agent task can easily consume 50k–100k tokens — that’s $0.50–$2.00 per task with GPT-4o. Multiply by thousands of users and the bill gets real.

# Cost control strategies:
#
# 1. Max step limits (hard stop after N iterations)
# 2. Token budgets (track cumulative usage, stop if exceeded)
# 3. Model cascading (use cheaper model for simple steps)
# 4. Caching (same tool call with same args → cached result)
# 5. Prompt compression (summarize long tool results before
#    feeding back to the model)

class BudgetedAgent:
    def __init__(self, max_steps=10, max_tokens=50_000):
        self.max_steps = max_steps
        self.max_tokens = max_tokens
        self.total_tokens = 0

    def run_step(self, messages, tools):
        if self.total_tokens >= self.max_tokens:
            return "Budget exceeded. Returning best answer so far."

        response = client.chat.completions.create(
            model="gpt-4o", messages=messages, tools=tools
        )
        self.total_tokens += response.usage.total_tokens
        return response

Sandboxing

Agents that execute code or run shell commands need strict isolation. A code-executing agent should run in a sandboxed environment — a Docker container, a Firecracker microVM, or at minimum a restricted subprocess. Never give an agent direct access to your production database or filesystem.

Human-in-the-Loop

For high-stakes actions, insert approval gates into the agent loop. The agent proposes an action, a human reviews and approves (or rejects with feedback), and execution continues.

# Human-in-the-loop gate
REQUIRES_APPROVAL = {"send_email", "delete_record", "make_purchase",
                     "deploy_code", "modify_permissions"}

def execute_with_approval(tool_call, available_functions):
    func_name = tool_call.function.name

    if func_name in REQUIRES_APPROVAL:
        print(f"\n⚠️  Agent wants to: {func_name}")
        print(f"   Args: {tool_call.function.arguments}")
        approval = input("   Approve? (y/n): ")
        if approval.lower() != 'y':
            return {"status": "rejected", "reason": "Human denied this action"}

    return available_functions[func_name](
        **json.loads(tool_call.function.arguments)
    )

Monitoring & Observability

In production, you need to see everything an agent does. Log every tool call, every tool result, every reasoning step. Track success rates per tool, per task type, per agent configuration. When an agent fails, the trace should tell you exactly where and why.

# Essential agent metrics to track:
#
# Per-task metrics:
#   - Total steps taken
#   - Total tokens consumed
#   - Total cost
#   - Task success / failure
#   - Time to completion
#   - Tools called (with args and results)
#
# Aggregate metrics:
#   - Success rate by task type
#   - Average steps per task
#   - Average cost per task
#   - Most common failure modes
#   - Tool reliability rates
#   - Median loop iterations before answer
#
# Tools: LangSmith, Arize Phoenix, Braintrust, custom logging
Agent observability isn’t optional. Without traces, debugging a failing agent is like debugging a distributed system with no logs. You won’t know if the failure was a bad LLM response, a bad tool result, or a bad tool choice. Instrument from day one. Tools like LangSmith and Arize Phoenix give you trace visualization out of the box. At minimum, log the full message history for every failed task.

The Autonomy-Reliability Tradeoff

There’s a fundamental tension in agent design: the more autonomy you give an agent, the more it can do — but also the more it can get wrong. This isn’t a problem to solve; it’s a tradeoff to manage.

# The autonomy-reliability spectrum
#
# ◄──── Less Autonomy                    More Autonomy ────►
# ◄──── Higher Reliability          Lower Reliability ────►
#
# ┌──────────┬──────────────┬──────────────┬───────────────┐
# │ Copilot  │ Tool-using   │ Autonomous   │ Fully         │
# │ mode     │ assistant    │ agent        │ autonomous    │
# ├──────────┼──────────────┼──────────────┼───────────────┤
# │ Suggests │ Calls tools  │ Plans and    │ Operates      │
# │ actions, │ with human   │ executes     │ independently │
# │ human    │ approval for │ multi-step   │ for hours,    │
# │ executes │ writes       │ tasks with   │ manages own   │
# │          │              │ some guardrails│ errors       │
# ├──────────┼──────────────┼──────────────┼───────────────┤
# │ ~99%     │ ~95%         │ ~80%         │ ~50-60%       │
# │ reliable │ reliable     │ reliable     │ reliable      │
# └──────────┴──────────────┴──────────────┴───────────────┘
#
# Most production systems today sit in the middle two columns.
# The right column is where research is headed.
Benchmark reality check. Agent benchmarks are still young and somewhat noisy. SWE-bench scores vary depending on the subset, evaluation harness, and whether you count “assisted” vs. fully autonomous resolution. Use benchmarks to track progress and compare approaches, but validate on your tasks. A model that scores 50% on SWE-bench might score 90% on your specific workflow — or 10%.
