LLM Production — Serving & Inference

Chapter 12: Large Language Models 9 subtopics
TL;DR

Serving LLMs is nothing like serving a ResNet. Autoregressive generation is memory-bandwidth-bound, not compute-bound. The KV cache dominates your memory budget for long sequences. Continuous batching gives you 10-23x throughput over naive batching. Speculative decoding lets a small model draft tokens verified by the big model for 2-3x speedups. vLLM’s PagedAttention manages KV cache like OS virtual memory pages. Your serving framework choice (vLLM, TensorRT-LLM, TGI, SGLang) depends on your performance vs. ease-of-use tradeoff. And the business side — cost per token, when to self-host, model cascading, prompt caching — often matters more than raw inference speed. This section covers everything from the KV cache math to production infrastructure to your cloud bill.

Why LLM Serving Is Different

If you’re coming from traditional ML model serving — a classification model behind a REST endpoint, a recommendation model doing batch scoring — forget most of what you know. LLM serving is a fundamentally different beast, and the reasons all trace back to one thing: autoregressive generation.

A BERT model or a vision model processes the entire input in one forward pass and returns a result. Done. An LLM generating a 500-token response makes 500 sequential forward passes, each producing exactly one token, each depending on every token that came before it. You can’t parallelize this across the sequence dimension. Token 247 cannot be computed until tokens 1–246 exist.

This has two consequences that shape everything about LLM serving:

1. The bottleneck is memory bandwidth, not compute. During generation, each forward pass is relatively small (one new token), but it needs to read the entire model weights and the entire KV cache from GPU memory. On an A100, you have 312 TFLOPS of compute but only 2 TB/s of memory bandwidth. For the generation phase of a large model, you’re spending most of your time waiting for data to arrive at the compute cores, not actually doing math. This is the “memory-bound” regime, and it’s why simply buying more FLOPs doesn’t help as much as you’d expect.

2. Latency has two distinct components.

Metric What It Measures Determined By Typical Target
TTFT (Time to First Token) How long until the first output token appears Prompt processing (prefill phase) < 500ms for chat
TPOT (Time Per Output Token) Time between consecutive output tokens Decode phase, memory bandwidth < 50ms (~20 tokens/sec)
Total Latency TTFT + (num_tokens × TPOT) Both phases Application-dependent

The prefill phase processes the entire prompt in one forward pass — it’s compute-bound and parallelizable. The decode phase generates tokens one at a time — it’s memory-bandwidth-bound and sequential. Optimizing these two phases requires different strategies, and many production tricks target one or the other.

# Two-phase nature of LLM inference:
#
# Phase 1: PREFILL (compute-bound)
# ─────────────────────────────────
# Input:  entire prompt (e.g., 2000 tokens)
# Output: KV cache + first output token
# Cost:   one big forward pass, all tokens processed in parallel
# Like:   traditional model inference — embarrassingly parallel
#
# Phase 2: DECODE (memory-bandwidth-bound)
# ─────────────────────────────────────────
# Input:  one new token + entire KV cache
# Output: next token
# Cost:   repeated N times for N output tokens
# Like:   sequential, each step reads entire model + cache from memory
#
# TTFT = prefill_time
# TPOT = single_decode_step_time
# Total = TTFT + output_length × TPOT
Why GPU utilization looks terrible during generation. During decode, a 70B model on 8 GPUs might show only 5-10% compute utilization. That’s not a bug — it’s the physics of memory-bound workloads. The arithmetic units are idle waiting for memory reads. This is exactly why batching (processing multiple requests at once) is critical: it lets you reuse the same model weight reads across many requests, amortizing the memory bandwidth cost.

The KV Cache — Deep Dive

Self-attention computes Q × KT for every query against every key. During generation, each new token needs to attend to all previous tokens. Without any optimization, this means recomputing K and V projections for the entire sequence at every decoding step — O(n²) work that grows quadratically with sequence length.

The KV cache eliminates this recomputation. At each decoding step, you compute K and V only for the new token, then concatenate them with the cached K and V from all previous tokens. The attention for the new token then attends over the full cached K/V, but you only do the projection work for one new token.

# Without KV cache (naive):
for step in range(output_length):
    all_tokens = prompt + generated_so_far
    Q, K, V = project(all_tokens)       # re-project EVERYTHING each step
    output = attention(Q, K, V)
    next_token = output[-1]

# With KV cache:
# Step 1 (prefill): process entire prompt
Q, K, V = project(prompt)
kv_cache = (K, V)
output = attention(Q, K, V)
next_token = output[-1]

# Step 2+ (decode): process only the new token
for step in range(output_length - 1):
    q, k, v = project(next_token)        # project ONE token
    kv_cache = concat(kv_cache, (k, v))  # append to cache
    output = attention(q, kv_cache.K, kv_cache.V)  # attend over full cache
    next_token = output

KV Cache Memory Math

Here’s where things get uncomfortable. Let’s do the math for a 7B parameter model (LLaMA-2 7B architecture):

# KV Cache size calculation
# ═══════════════════════════════════════════
# Model: LLaMA-2 7B
# Layers:         32
# Attention heads: 32
# Head dimension:  128
# Sequence length: 4096
# Dtype:          FP16 (2 bytes per element)

# Per token, per layer, we store K and V vectors:
per_token_per_layer = 2 × 32 × 128 × 2   # 2 (K+V) × heads × dim × bytes
                    = 16,384 bytes         # 16 KB per token per layer

# Per token across all layers:
per_token = 16,384 × 32                   # × num_layers
          = 524,288 bytes                  # 512 KB per token

# For a full 4096-token sequence:
per_request = 524,288 × 4096
            = 2,147,483,648 bytes
            ≈ 2 GB per request             # !

# Model weights (7B params in FP16): ~14 GB
# So 7 concurrent requests with 4096-length sequences
# need 14 GB KV cache — the SAME as the model weights!
KV cache can exceed model weights. For long sequences, the KV cache easily dominates your memory budget. A 7B model in FP16 is ~14 GB. Serve 10 concurrent requests at 4096 tokens each and you need ~20 GB just for KV cache. At 32K sequence length, a single request’s KV cache balloons to ~16 GB. This is why KV cache management is the central problem of LLM serving — not model weights, not compute.

KV Cache Quantization

A simple and effective optimization: store the KV cache in INT8 instead of FP16. This halves the cache memory with minimal quality loss.

KV Cache Dtype Bytes per Element Cache Size (7B, 4096 seq) Quality Impact
FP16 2 ~2 GB Baseline
INT8 1 ~1 GB Negligible (<0.5% perplexity increase)
FP8 (E4M3) 1 ~1 GB Near-zero on H100/Ada GPUs
INT4 0.5 ~0.5 GB Noticeable for long sequences

INT8 KV cache is essentially free quality-wise and should be your default for production. Most serving frameworks support it out of the box. FP8 is even better on newer hardware (H100, L40S) since the native FP8 compute pipeline avoids dequantization overhead.

Grouped-Query Attention Reduces KV Cache

GQA (from Section 1) isn’t just a training optimization — it directly shrinks the KV cache. LLaMA-2 7B uses 32 KV heads. Mistral 7B uses 8 KV heads with GQA. That’s a 4x reduction in KV cache size for free. This is a big part of why GQA became standard in all modern architectures: it was designed with inference serving in mind.

# GQA impact on KV cache:
# LLaMA-2 7B: 32 KV heads → 2 GB per request @ 4096 tokens
# Mistral 7B:  8 KV heads → 0.5 GB per request @ 4096 tokens
#
# Same model quality class, 4x more concurrent requests possible

Continuous Batching — The Efficiency Breakthrough

Batching is obvious: process multiple requests at once to amortize the cost of reading model weights from memory. The problem is how you batch in an autoregressive setting where different requests have different lengths and finish at different times.

Naive (Static) Batching

The simplest approach: collect N requests, pad them to the same length, process them as a batch, wait for all of them to finish before returning any results.

# Naive batching — the waste problem
#
# Request A: prompt=100 tokens, generates 20 tokens   → done at step 20
# Request B: prompt=100 tokens, generates 200 tokens  → done at step 200
# Request C: prompt=100 tokens, generates 50 tokens   → done at step 50
#
# With naive batching:
# - All three are batched together
# - A finishes at step 20 but WAITS until step 200
# - C finishes at step 50 but WAITS until step 200
# - GPU computes padding tokens for A from step 21-200 → pure waste
# - Request A's latency is 10x what it should be
# - GPU is 50-90% wasted on padding tokens

This is terrible. Short requests are held hostage by long ones, and the GPU burns cycles generating pad tokens that nobody wants.

Continuous Batching (Iteration-Level Scheduling)

The fix, introduced by Orca (Yu et al., 2022): schedule at the iteration level, not the request level. After each decode step, check which requests are done (hit EOS or max length) and immediately replace them with waiting requests from the queue.

# Continuous batching — iteration-level scheduling
#
# Step 1:  [A, B, C] → all three generate one token
# Step 20: A hits EOS → remove A, slot in new request D
#          [D, B, C] → continue
# Step 50: C hits EOS → remove C, slot in new request E
#          [D, B, E] → continue
# Step 75: D hits EOS → remove D, slot in new request F
#          [F, B, E] → continue
# ...
#
# GPU is always full. No request waits for unrelated requests.
# New requests start immediately when slots open.

The results are dramatic:

Metric Naive Batching Continuous Batching
Throughput Baseline 10-23x improvement
GPU utilization 10-30% 70-95%
Short request latency Penalized by long requests Returns as soon as finished
Queue wait time Waits for batch formation Joins next iteration

Every modern serving framework uses continuous batching. If your serving setup doesn’t have it, you’re leaving 10-20x throughput on the table.

Decoding Strategies — How Tokens Are Selected

The model outputs a probability distribution over the vocabulary at each step. The decoding strategy decides how to pick a token from that distribution. This choice dramatically affects output quality and character.

Greedy Decoding

Pick the highest-probability token every time. Deterministic — same input always gives the same output. Fast. But often repetitive and boring. The model gets stuck in loops because the locally optimal choice at each step doesn’t produce a globally optimal sequence.

# Greedy decoding
next_token = argmax(logits)  # always pick the most likely token

# Problem: "The cat sat on the mat. The cat sat on the mat. The cat..."

Beam Search

Maintain the top-k most probable sequences (beams) at each step, expanding each by one token and keeping the best k overall. Finds higher-probability sequences than greedy but is more expensive (k forward passes per step). Good for translation and summarization where you want the “best” output, not a creative one.

# Beam search with k=3
# Step 1: top 3 tokens → 3 beams
# Step 2: expand each beam by all vocab → 3 × V candidates → keep top 3
# Step 3: repeat until EOS
# Return the beam with highest total probability

Temperature Sampling

Divide the logits by a temperature T before applying softmax. This reshapes the distribution:

# Temperature sampling
scaled_logits = logits / temperature
probs = softmax(scaled_logits)
next_token = sample(probs)

# T = 1.0  → standard distribution (model's natural confidence)
# T < 1.0  → sharper: high-prob tokens get even higher probability
#             (more deterministic, more "focused")
# T > 1.0  → flatter: probabilities spread more evenly
#             (more random, more "creative")
# T → 0    → approaches greedy decoding
# T → ∞    → approaches uniform random sampling

Top-k Sampling

Sample from only the k highest-probability tokens. Everything else gets zero probability. Prevents the model from picking wildly unlikely tokens that occasionally ruin generation.

# Top-k sampling (k=50)
top_k_logits = top_k(logits, k=50)      # zero out everything below rank 50
probs = softmax(top_k_logits)
next_token = sample(probs)

Top-p (Nucleus) Sampling

Sort tokens by probability, take the smallest set whose cumulative probability ≥ p. This adapts the candidate pool size to the model’s confidence: when the model is very confident, only a few tokens pass the threshold; when it’s uncertain, more tokens are included.

# Top-p (nucleus) sampling (p=0.9)
sorted_probs = sort_descending(softmax(logits))
cumulative = cumsum(sorted_probs)
# Keep tokens until cumulative prob ≥ 0.9
cutoff_idx = first_index_where(cumulative >= 0.9)
nucleus = sorted_probs[:cutoff_idx + 1]
next_token = sample(nucleus)

Top-p is generally preferred over top-k because it adapts to the distribution shape. When the model is confident (one token has 95% probability), top-p naturally restricts to ~1 token. Top-k with k=50 would still sample from 50 tokens including garbage.

Penalty Mechanisms

Repetition penalty: Reduce logits for tokens that already appeared in the output. Prevents loops.

Frequency penalty: Reduce logits proportional to how many times the token has appeared. More occurrences = stronger penalty.

Presence penalty: Binary: reduce logits for any token that has appeared at all, regardless of count. Encourages topic diversity.

Strategy Comparison

Strategy Best For Typical Settings Trade-off
Greedy Factual Q&A, extraction T=0 (or argmax) Deterministic but repetitive
Beam search Translation, summarization k=4–8 beams Higher quality, expensive
Temp + top-p Creative writing, brainstorming T=0.7–1.0, p=0.9–0.95 Diverse but less predictable
Low temp + top-p Code generation T=0.2–0.4, p=0.95 Mostly deterministic, avoids worst tokens
Top-k General chat k=40–100, T=0.7 Simple, good default
Greedy + rep penalty Long-form factual content T=0, rep_penalty=1.1–1.3 Deterministic without loops
The practitioner’s default. For most production chat applications: temperature=0.7, top_p=0.9, repetition_penalty=1.05. For code: temperature=0.2, top_p=0.95. For factual extraction: temperature=0 (greedy). Start with these and tune based on user feedback, not benchmarks.

Speculative Decoding — Deep Dive

Here’s the core insight: verifying a sequence of tokens is much cheaper than generating them one at a time. The large model can check N candidate tokens in a single forward pass (because they can all be processed in parallel during prefill), but generating those N tokens one-by-one would take N sequential forward passes. Speculative decoding exploits this asymmetry.

How It Works

# Speculative Decoding Algorithm
# ═══════════════════════════════════════════
# Players:
#   Draft model (Mq): small, fast (e.g., 1B params)
#   Target model (Mp): large, accurate (e.g., 70B params)
#
# Step 1: Draft model generates N candidate tokens autoregressively
#   draft_tokens = Mq.generate(prompt, n_tokens=N)  # fast, N sequential steps
#
# Step 2: Target model verifies ALL N tokens in ONE forward pass
#   target_probs = Mp.forward([prompt + draft_tokens])  # one parallel pass
#
# Step 3: Accept/reject each token left-to-right
#   for i in range(N):
#       p = target_probs[i]   # target's probability for token i
#       q = draft_probs[i]    # draft's probability for token i
#       if random() < min(1, p/q):
#           accept token i
#       else:
#           reject token i, sample correction from (p - q)+
#           discard all tokens after i
#           break
#
# Step 4: Target model also generates one additional token after the last accepted
#
# Best case: all N accepted → N+1 tokens for ~1 large model forward pass
# Worst case: first token rejected → 1 token (same as normal decoding)

The acceptance criterion is carefully designed to produce exactly the same output distribution as the target model alone. This isn’t an approximation — speculative decoding is a lossless speedup technique. You get the exact same quality as running the target model; you just get there faster.

What Determines the Speedup?

Factor Impact
Acceptance rate Higher agreement between draft and target = more tokens accepted = bigger speedup
Draft model speed The draft model must be fast enough that generating N tokens takes < 1 target model forward pass
Speculation length N Longer speculation = more potential gain but lower acceptance probability for the full sequence
Task difficulty Easy/predictable text (boilerplate, common phrases) = high acceptance. Novel/creative text = lower acceptance

In practice, you see 2–3x speedup for typical workloads. The sweet spot for speculation length is usually N=3–5 tokens.

Medusa: Draft Heads Instead of a Draft Model

Medusa (Cai et al., 2024) takes a different approach: instead of a separate draft model, add multiple prediction heads to the target model itself. Each head predicts a different future token position (head 1 predicts token t+1, head 2 predicts t+2, etc.). The candidates from all heads form a tree of possible continuations, which can be verified in one forward pass using a tree attention mask.

# Medusa architecture
#
# Standard LLM:
#   hidden_state → lm_head → next_token (position t+1)
#
# Medusa LLM:
#   hidden_state → lm_head   → token at t+1
#               → medusa_head_1 → token at t+2   (added)
#               → medusa_head_2 → token at t+3   (added)
#               → medusa_head_3 → token at t+4   (added)
#
# Medusa heads are small MLPs trained while target model is frozen
# No separate draft model needed = simpler deployment
When to use speculative decoding. It shines when: (1) latency matters more than throughput, (2) you have a good draft model available (or can add Medusa heads), and (3) you have enough GPU memory for both models. It’s less useful for high-throughput batch processing where you’re already GPU-saturated with continuous batching — speculative decoding adds complexity without helping when the GPU is already busy with a large batch.

Serving Frameworks — The Landscape

You don’t build an LLM serving stack from scratch. You pick a framework. Here’s the real landscape as of 2024–2025, with honest assessments from someone who’s deployed with most of these.

vLLM

The default choice for most teams, and for good reason. vLLM introduced PagedAttention (more on this below), has first-class continuous batching, supports AWQ/GPTQ/FP8 quantized models, and runs with a simple API. If you don’t have a strong reason to pick something else, start here.

# Serving a model with vLLM — it's this simple
pip install vllm

# Start an OpenAI-compatible API server
python -m vllm.entrypoints.openai.api_server \
    --model meta-llama/Llama-2-7b-chat-hf \
    --dtype float16 \
    --max-model-len 4096 \
    --gpu-memory-utilization 0.90 \
    --enable-prefix-caching

# Use it like OpenAI's API
curl http://localhost:8000/v1/chat/completions \
    -H "Content-Type: application/json" \
    -d '{
        "model": "meta-llama/Llama-2-7b-chat-hf",
        "messages": [{"role": "user", "content": "Hello!"}],
        "temperature": 0.7,
        "max_tokens": 256
    }'

TensorRT-LLM

NVIDIA’s runtime squeezes every last drop of performance from NVIDIA GPUs. Custom CUDA kernels, kernel fusion, in-flight batching, FP8 on Hopper — it’s the fastest option if you’re on NVIDIA hardware and willing to deal with a more complex build process. The compilation step converts your model to an optimized engine, which can take 30+ minutes but pays off in throughput.

Text Generation Inference (TGI)

Hugging Face’s production server. Written in Rust for the hot path. Easiest setup if you’re already in the HF ecosystem. Good performance, excellent documentation, Docker-first deployment.

Ollama / llama.cpp

For local and edge deployment. llama.cpp is a C/C++ implementation that runs LLMs on CPUs and consumer GPUs using the GGUF quantization format. Ollama wraps it with a friendly CLI and model registry. Perfect for developer machines, offline use, and privacy-sensitive deployments.

SGLang

The newest contender with a killer feature: RadixAttention for automatic prefix caching. Uses a radix tree to share KV cache across requests that share common prefixes, without any manual configuration. Also has a programming-language-style frontend for complex LLM programs (multi-turn, branching, tool use).

Framework Comparison

Framework Performance Ease of Use Key Feature Best For
vLLM ★★★★☆ ★★★★★ PagedAttention, OpenAI-compatible API General production serving
TensorRT-LLM ★★★★★ ★★★☆☆ Kernel fusion, FP8, custom CUDA kernels Max perf on NVIDIA GPUs
TGI ★★★★☆ ★★★★★ HF ecosystem integration, Rust hot path HF-native teams, Docker deploys

The honest recommendation. Start with vLLM. It has the best community, the simplest API, and good-enough performance for 90% of workloads. If you’re hitting latency limits and you’re on NVIDIA GPUs, evaluate TensorRT-LLM. If you’re building complex multi-turn LLM programs with heavy prefix sharing, look at SGLang. If you need local/edge, use Ollama.

PagedAttention — vLLM’s Key Innovation

Before PagedAttention, serving frameworks pre-allocated contiguous blocks of GPU memory for each request’s KV cache — sized for the maximum possible sequence length. If your max sequence length was 4096 tokens but a request only generated 50 tokens, you wasted 99% of that allocation. And because the blocks were contiguous, you couldn’t reclaim the unused space without moving memory around.

This is the exact same problem operating systems faced with physical memory allocation in the 1960s. And PagedAttention uses the exact same solution: virtual memory with paging.

# Traditional KV cache allocation (wasteful):
# ═══════════════════════════════════════════
# GPU Memory Layout:
# ┌───────────────────────────────────┐
# │ Request A: [KV KV KV KV _ _ _ _ _ _ _ _]  ← 4 tokens used, 8 wasted
# │ Request B: [KV KV KV KV KV KV _ _ _ _ _]  ← 6 tokens used, 5 wasted
# │ Request C: [KV KV _ _ _ _ _ _ _ _ _ _ _]  ← 2 tokens used, 11 wasted
# │ ████████ FRAGMENTED FREE SPACE ████████
# └───────────────────────────────────┘
# Can't fit Request D even though there's enough total free memory
# because no single contiguous block is large enough

# PagedAttention (efficient):
# ═══════════════════════════════════════════
# GPU Memory split into fixed-size pages (e.g., 16 tokens each):
# ┌──────────────────────────────────────┐
# │ Page 0: [A:KV A:KV A:KV A:KV]  ← Request A, page 1 of 1
# │ Page 1: [B:KV B:KV B:KV B:KV]  ← Request B, page 1 of 2
# │ Page 2: [B:KV B:KV _ _ ]        ← Request B, page 2 of 2
# │ Page 3: [C:KV C:KV _ _ ]        ← Request C, page 1 of 1
# │ Page 4: [FREE FREE FREE FREE]   ← available for new requests
# │ Page 5: [FREE FREE FREE FREE]
# └──────────────────────────────────────┘
# Each request has a page table mapping logical → physical pages
# Pages allocated on demand, freed immediately when request completes
# Near-zero waste, no fragmentation

Why This Matters in Practice

PagedAttention reduces KV cache memory waste from 60–80% (typical with pre-allocation) to under 4%. This directly translates to serving more concurrent requests on the same hardware:

Metric Pre-allocated (Traditional) PagedAttention (vLLM)
Memory waste 60–80% < 4%
Concurrent requests (7B model, A100 80GB) ~8–12 ~30–50+
Memory fragmentation Severe over time Eliminated

Prefix Caching via Shared Pages

PagedAttention enables another powerful optimization: prefix caching. When multiple requests share the same prefix (e.g., a system prompt), their KV cache pages for that prefix are identical. Instead of computing and storing the KV cache for “You are a helpful assistant...” once per request, compute it once and share the physical pages across all requests via their page tables — exactly like copy-on-write pages in an OS.

# Prefix caching example:
#
# 100 requests all start with the same 500-token system prompt
#
# Without prefix caching:
#   KV cache for system prompt = 500 tokens × 512 KB = 256 MB per request
#   Total: 100 × 256 MB = 25.6 GB just for system prompts
#
# With prefix caching:
#   Shared KV pages: 256 MB (computed once)
#   Total: 256 MB + per-request unique portions
#   Savings: ~25 GB of GPU memory
#
# Enable in vLLM:
# python -m vllm.entrypoints.openai.api_server \
#     --model your-model \
#     --enable-prefix-caching    # ← this flag
When prefix caching is a game-changer. If your application uses long system prompts (RAG contexts, few-shot examples, tool definitions), prefix caching can reduce memory usage by 50%+ and improve TTFT dramatically because the prefill for the shared prefix is computed once and reused. This is especially impactful for agent frameworks that inject large tool schemas into every request.

Production Infrastructure

Frameworks handle the inference engine. But serving in production means dealing with load balancing, autoscaling, monitoring, failover, and cost — the stuff that turns a demo into a product.

Latency Budgets

Set concrete targets and work backwards:

Application TTFT Target TPOT Target Notes
Real-time chat < 500ms < 50ms Users expect streaming at readable speed
Code completion < 200ms < 30ms Must feel instant; latency kills adoption
Document summarization < 2s < 100ms Users expect to wait a moment for long docs
Batch processing N/A Maximize throughput Latency doesn’t matter; cost per token does

Autoscaling

Don’t scale on CPU utilization — LLM workloads are GPU-bound. Scale on:

  • Queue depth: Number of pending requests. Most reliable signal. Scale up when queue > threshold for > 30s.
  • GPU utilization: Scale up above 80%, scale down below 30%. But beware: decode-phase GPU utilization is naturally low even when fully loaded (memory-bound).
  • TPOT / TTFT percentiles: Scale up when p95 latency exceeds your SLA. The most user-centric signal.
# Example: Kubernetes HPA for LLM serving
# Scale on custom metric: request queue depth
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: llm-serving-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: llm-serving
  minReplicas: 2
  maxReplicas: 16
  metrics:
  - type: Pods
    pods:
      metric:
        name: llm_request_queue_depth
      target:
        type: AverageValue
        averageValue: "10"     # scale up when avg queue > 10
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60   # wait 60s before scaling up
    scaleDown:
      stabilizationWindowSeconds: 300  # wait 5min before scaling down (GPU startup is slow)

Model Cascading

Not every query needs your biggest model. Route simple queries to a small, fast, cheap model and only escalate to the expensive model when needed.

# Model cascading pseudocode
def route_request(query):
    # Step 1: Classify query complexity (use a tiny classifier or heuristics)
    complexity = classify_complexity(query)

    if complexity == "simple":
        # FAQ, simple factual, short answers
        return small_model.generate(query)    # 7B model, $0.0001/request

    elif complexity == "medium":
        # Standard chat, summarization
        return medium_model.generate(query)   # 70B model, $0.002/request

    else:
        # Complex reasoning, code generation, ambiguous queries
        return large_model.generate(query)    # 405B model, $0.01/request

# Typical result: 60-70% of queries handled by small model
# Cost savings: 5-10x compared to routing everything to the largest model

Fallback Strategies

Production systems fail. Plan for it:

  • Model failover: If primary model is down, route to backup (same model on different GPU, or a smaller fallback model)
  • Provider failover: If self-hosted is down, fall back to an API provider (OpenAI, Anthropic)
  • Graceful degradation: If all LLMs are unavailable, return cached responses for common queries, or a friendly “service degraded” message with a retry option
  • Circuit breaker: If error rate exceeds threshold, stop sending requests to the failing endpoint and redirect to healthy ones

Batch Processing

For offline workloads (document processing, data labeling, evaluation), batch processing is dramatically cheaper than real-time serving:

  • API batch endpoints: OpenAI offers 50% discount for batch API (results within 24 hours). Anthropic has similar batch pricing.
  • Self-hosted batching: Run on spot/preemptible instances (60–80% cheaper), maximize batch size for throughput over latency, run during off-peak hours.
  • Async queues: Decouple submission from completion. Use a message queue (SQS, RabbitMQ) with GPU workers pulling jobs. Scale workers to zero during idle periods.

Caching Repeated Queries

LLM outputs are expensive to compute and often repetitive. Cache aggressively:

import hashlib
from functools import lru_cache

# Level 1: Exact match cache (hash the full prompt)
def exact_cache_key(messages, params):
    content = str(messages) + str(params)
    return hashlib.sha256(content.encode()).hexdigest()

def semantic_cache_lookup(query, threshold=0.95):

Model Distillation

The nuclear option for cost reduction: train a smaller, task-specific model using outputs from a large model. This is not general fine-tuning — it’s using the large model as a teacher to create training data for the student.

Distillation licensing. Some model providers explicitly prohibit using their API outputs to train competing models. OpenAI’s terms, for instance, restrict this for certain use cases. Always check the terms of service before building a distillation pipeline. Open-source models (LLaMA, Mistral) have more permissive licenses for this.
