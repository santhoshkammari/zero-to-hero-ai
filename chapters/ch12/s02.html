LLM Families & Architecture

Chapter 12: Large Language Models 6 subtopics
TL;DR

Three transformer paradigms emerged — encoder-only (BERT), decoder-only (GPT), and encoder-decoder (T5) — but decoder-only won the generation wars through sheer scaling. The BERT family still owns understanding tasks: classification, NER, embeddings. The GPT lineage bet everything on "bigger is better" and was right. Meanwhile, open models (LLaMA, Mistral, Qwen) democratized access, and modern architecture decisions — RoPE, GQA, SwiGLU, pre-norm — became the standard recipe that every 2024 LLM shares regardless of who built it. This section is your field guide to who built what, why it matters, and which architectural choices actually stuck.

The Three Paradigms

Every transformer-based language model falls into one of three camps, defined by which parts of the original transformer architecture they keep:

Paradigm Architecture Attention Pattern Good At Flagship Models
Encoder-only Transformer encoder stack Bidirectional (sees all tokens) Understanding: classification, NER, similarity BERT, RoBERTa, DeBERTa
Decoder-only Transformer decoder stack Causal (sees only past tokens) Generation: text completion, chat, code GPT, LLaMA, Mistral
Encoder-Decoder Full transformer (both stacks) Bidirectional encoder + causal decoder Seq2seq: translation, summarization T5, BART, FLAN-T5

Why decoder-only won for generation. It comes down to three things. First, simplicity: one stack is easier to scale, shard, and optimize than two. Second, the autoregressive objective (predict next token) naturally handles any text-to-text task — you just format the input right. Third, decoder-only models get a KV cache that makes inference efficient: you compute each token's key-value once and reuse it for all future tokens. Encoder-decoder models need to process the full input through the encoder for every output sequence, which is wasteful when both input and output are long.

That said, encoder-only models aren't dead — they're just specialized. If you need to classify a document, extract entities, or compute sentence embeddings, a 110M-parameter BERT variant will outperform a 7B decoder model at 1/60th the cost. Right tool, right job.

📌 A common interview question

"Why can't you just use BERT for generation?" Because BERT sees all tokens simultaneously during training — it has no notion of sequential token prediction. You can force it to generate iteratively by masking one position at a time and filling it in, but each fill is conditioned on the original surrounding tokens, not on previously generated ones. The generations are incoherent because there's no autoregressive chain. Decoder-only models generate one token at a time, each conditioned on all previous tokens — that causal structure is what makes coherent generation possible.

BERT Family

BERT (2018) was the model that made pre-training + fine-tuning the default NLP workflow. The core ideas: Masked Language Modeling (MLM) — randomly mask 15% of tokens and predict them using bidirectional context — and Next Sentence Prediction (NSP) — a binary task predicting whether sentence B actually follows sentence A. It shipped in two sizes: Base (110M params, 12 layers) and Large (340M params, 24 layers), trained on BookCorpus + Wikipedia (~3.3B words).

What made BERT revolutionary wasn't any single clever trick — it was the paradigm shift. Before BERT, NLP practitioners trained task-specific architectures from scratch: one model for sentiment, another for NER, another for QA. After BERT, you pre-trained once on massive unlabeled text, then fine-tuned with a thin head on top. Transfer learning went from "nice idea" to "only sane approach."

# BERT's MLM training objective in pseudocode:
# 1. Take a sentence: "The cat sat on the mat"
# 2. Randomly select 15% of tokens
# 3. Of selected: 80% → [MASK], 10% → random word, 10% → unchanged
#    Result: "The [MASK] sat on the mat"  (or "The dog sat on the mat")
# 4. Model sees ALL tokens simultaneously (bidirectional)
# 5. Predict the original token at masked positions
#
# NSP objective (later shown to be unnecessary):
# Input:  [CLS] Sentence A [SEP] Sentence B [SEP]
# Output: Binary — does B actually follow A in the corpus?

Then the community spent years systematically fixing BERT's rough edges:

Model Year Params Key Change Practical Impact
RoBERTa 2019 125M / 355M Dropped NSP, dynamic masking, 10× more data, bigger batches Proved BERT was undertrained, not under-designed. Same arch, much better scores.
ALBERT 2019 12M–235M Factorized embeddings (V×E + E×H), cross-layer parameter sharing, SOP replaces NSP 87% fewer embedding params. Better regularization. But same inference speed — you still run all layers.
DistilBERT 2019 66M Knowledge distillation: 6-layer student mimics 12-layer teacher via KL divergence + cosine loss 40% smaller, 60% faster, retains 97% of BERT's GLUE score. The production workhorse.
DeBERTa 2020 134M–1.5B Disentangled attention (separate content + position vectors), enhanced mask decoder First model to surpass human performance on SuperGLUE. DeBERTa-v3 is still a top choice.
ELECTRA 2020 14M–335M Replaced token detection instead of masked prediction. Every token is a training signal, not just 15%. ~4× more sample-efficient than MLM. Great when you have limited compute.
ModernBERT 2024 149M / 395M RoPE, Flash Attention, unpadding, 8192 context, trained on 2T tokens State-of-the-art encoder model. Handles long documents. Modern hardware optimized.
📌 When to use BERT-family models today

Use them for: text classification, named entity recognition, sentence embeddings (via sentence-transformers), semantic search, reranking. Don't use them for: text generation, chatbots, summarization — that's decoder territory. The sweet spot is any task where you need to understand a piece of text, not produce new text. A fine-tuned DeBERTa-v3-base will beat GPT-4 on your specific classification task at 1/1000th the inference cost.

GPT Family

The GPT story is really a story about one bet: that scaling a simple autoregressive language model is all you need. OpenAI made that bet early, stuck with it when the rest of the field was chasing architectural novelty, and it paid off spectacularly.

Model Year Params Key Insight What Changed
GPT-1 2018 117M Unsupervised pre-training works Pre-train on books with language modeling, then fine-tune on downstream tasks. First proof that generative pre-training transfers.
GPT-2 2019 1.5B Zero-shot via scale No fine-tuning needed for many tasks — just prompt the model. "Language models are unsupervised multitask learners." Initially withheld for safety concerns.
GPT-3 2020 175B Few-shot in-context learning Give the model a few examples in the prompt and it figures out the task. No gradient updates. Training cost estimated at $4.6M. Created the API-as-a-product model.
InstructGPT / GPT-3.5 2022 ~175B RLHF alignment Fine-tuned GPT-3 with Reinforcement Learning from Human Feedback. The model that powered early ChatGPT. Proved that alignment makes models more useful, not just safer.
GPT-4 2023 Undisclosed (rumored ~1.8T MoE) Multimodal + reliability at scale Accepts images and text. Passes bar exam, medical licensing, coding interviews. Rumored Mixture-of-Experts architecture. Massive jump in reasoning.
GPT-4o 2024 Undisclosed Omni-modal efficiency Natively handles text, vision, and audio in a single model. 2× faster, 50% cheaper than GPT-4 Turbo. Optimized for real-time interaction.

The throughline here is worth noticing. GPT-1 showed pre-training transfers. GPT-2 showed you can skip fine-tuning. GPT-3 showed you can skip even zero-shot with few-shot prompting. InstructGPT showed you can align models to be helpful. Each generation didn't invent a fundamentally new architecture — it removed a constraint by adding scale or training signal. OpenAI's core thesis was always: a good enough architecture, scaled far enough, will develop emergent capabilities that no amount of architectural cleverness at small scale can match.

Whether that thesis continues to hold is one of the most debated questions in the field. But from 2018 to 2024, the scaling camp won every round.

# Quick GPT-family usage via OpenAI API
from openai import OpenAI
client = OpenAI()

# Few-shot in-context learning — the GPT-3 breakthrough
response = client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {"role": "system", "content": "Classify the sentiment as positive/negative/neutral."},
        {"role": "user", "content": "I love this product! → positive"},
        {"role": "user", "content": "It broke on day one. → negative"},
        {"role": "user", "content": "The package arrived on Tuesday. →"},
    ]
)
print(response.choices[0].message.content)  # "neutral"
# No fine-tuning, no gradient updates. Just examples in the prompt.

Other Major Model Families

The LLM landscape exploded after LLaMA showed you didn't need OpenAI's budget to build capable models. Here's the field:

T5 / FLAN-T5

T5 (Text-to-Text Transfer Transformer, Google 2019) reframed every NLP task as text-to-text: translation becomes "translate English to German: [text]", classification becomes "classify: [text]" → "positive". Elegant unification, but the encoder-decoder architecture made it harder to scale for pure generation. FLAN-T5 (2022) applied instruction tuning across 1,800+ tasks — the model that proved instruction tuning is a general capability amplifier, not just a fine-tuning trick.

LLaMA Family

LLaMA-1 (Meta, Feb 2023) was the shot heard around the open-source world. A 65B model trained on 1.4T tokens of publicly available data, competitive with GPT-3. The key insight: smaller models trained on more tokens beat larger models trained on fewer tokens (the Chinchilla scaling law, applied aggressively). When the weights leaked, the open-source explosion began.

LLaMA-2 (Jul 2023) shipped with a commercial license, chat-tuned variants, and RLHF. LLaMA-3 (Apr 2024) jumped to 15T training tokens — 10× LLaMA-1 — and introduced 8B, 70B, and 405B parameter variants. The 8B model rivals LLaMA-2 70B. The 405B model is competitive with GPT-4 class systems. All with open weights.

Mistral / Mixtral

Mistral-7B (Sep 2023) punched way above its weight class. Key ingredients: sliding window attention (4096 token window), grouped-query attention, and high-quality training data curation. It beat LLaMA-2 13B on every benchmark — at half the size. Mixtral 8x7B brought Mixture-of-Experts to the open-source world: 8 expert FFNs per layer, 2 active at a time, giving 46.7B total parameters but only ~13B active per token. Sparse MoE gives you large-model quality at small-model inference cost.

PaLM / Gemini

Google's path went from PaLM (540B, 2022) to PaLM-2 (2023, undisclosed size, better multilingual) to Gemini (2023–2024). Gemini is natively multimodal — trained from the ground up on interleaved text, images, audio, and video, not bolted-on vision like GPT-4. Gemini 1.5 Pro introduced a 1M-token context window using mixture-of-experts, pushing the boundaries of what "long context" means.

Claude

Anthropic's Claude series is built on Constitutional AI (CAI): instead of pure RLHF with human labelers, the model critiques its own outputs against a set of principles ("be helpful, harmless, honest") and revises them. Claude 3 (2024) came in three sizes — Haiku (fast), Sonnet (balanced), Opus (max capability) — with 200K token context and strong reasoning. The safety-first design philosophy means Claude tends to be more cautious but also more reliable on sensitive tasks.

Phi Series

Microsoft's Phi models (2023–2024) challenged the "scale is all you need" narrative. Phi-1 (1.3B) was trained primarily on textbook-quality data and synthetic examples, matching models 10× its size on coding benchmarks. Phi-2 (2.7B) and Phi-3 (3.8B) continued the thesis: data quality matters more than data quantity. Phi-3-mini rivals Mixtral 8x7B on many benchmarks — at 1/12th the total parameter count. If you're deploying on-device or at the edge, the Phi family is where to look.

Chinese Open Models

Qwen (Alibaba), DeepSeek, and Yi (01.AI) are the leading Chinese open-weight model families. Qwen-2 (2024) comes in sizes from 0.5B to 72B, with strong multilingual and coding capabilities. DeepSeek-V2 introduced Multi-head Latent Attention (MLA) — compressing KV caches by projecting keys and values into a low-dimensional latent space — achieving massive efficiency gains. These models are important because they demonstrate that frontier capabilities aren't geographically locked to the US.

📌 The open vs. closed divide

The LLM landscape has a fundamental split. Closed models (GPT-4, Claude, Gemini) push the frontier but give you API access only — no fine-tuning control, no deployment flexibility, and you're subject to usage policies and pricing changes. Open-weight models (LLaMA, Mistral, Qwen) let you fine-tune, quantize, deploy anywhere, and modify freely. The capability gap between the two camps has been shrinking fast — LLaMA-3 405B and Qwen-2 72B approach GPT-4 class performance on many benchmarks. For most production use cases, the question isn't "which model is best?" but "what are my constraints?" — latency, cost, data privacy, and deployment control often matter more than the last few benchmark points.

Modern Architecture Decisions

Here's the thing — if you look at the source code of LLaMA-2, Mistral, Qwen-2, and Phi-3, the architectures are shockingly similar. The field has converged on a standard recipe. Understanding these shared design decisions matters more than memorizing any single model, because they tell you what actually works when you're building or fine-tuning an LLM in 2024.

Pre-Norm vs Post-Norm

The original transformer paper placed LayerNorm after the attention and FFN blocks (post-norm). Virtually every modern LLM uses pre-norm — LayerNorm before each sub-layer:

# Post-Norm (original transformer, GPT-2)
x = x + Attention(x)
x = LayerNorm(x)
x = x + FFN(x)
x = LayerNorm(x)

# Pre-Norm (LLaMA, Mistral, most modern LLMs)
x = x + Attention(LayerNorm(x))
x = x + FFN(LayerNorm(x))

Why the switch? Training stability. With post-norm, gradients in the residual stream accumulate layer-norm operations, making deep networks (60+ layers) brittle to train — you need careful learning rate warmup and often get training collapses. Pre-norm ensures the residual path is a clean addition, so gradients flow unimpeded through the skip connections. The tradeoff: pre-norm can slightly degrade final performance compared to a perfectly-tuned post-norm setup, but "perfectly-tuned" is the operative phrase. Pre-norm just works, so everyone uses it.

Most modern models specifically use RMSNorm instead of standard LayerNorm — it drops the mean-centering step and only normalizes by the root-mean-square, saving ~10% of the normalization compute for essentially the same training dynamics.

SwiGLU Activation

The feed-forward network (FFN) in the original transformer was simple: two linear projections with a ReLU in between. Modern LLMs almost universally use SwiGLU (Shazeer, 2020), which adds a gating mechanism:

# Original FFN (GPT-2 era)
FFN(x) = ReLU(xW₁ + b₁)W₂ + b₂

# SwiGLU FFN (LLaMA, Mistral, Qwen, Phi, etc.)
SwiGLU(x) = (Swish(xW_gate) ⊙ xW₁) W₂

# Where:
#   W_gate, W₁ ∈ ℝ^(d_model × d_ff)   ← note: THREE weight matrices now
#   W₂ ∈ ℝ^(d_ff × d_model)
#   Swish(z) = z · σ(z)                 ← smooth, non-monotonic activation
#   ⊙ = element-wise multiplication     ← the "gating"

The gating mechanism lets the network learn to selectively pass information through the FFN — the sigmoid-gated path controls how much of each dimension gets through. Empirically, SwiGLU gives ~1-3% better performance across benchmarks for the same compute budget. The cost is a third weight matrix (W_gate), which is why modern LLMs typically use d_ff = (8/3)d_model instead of the original 4×d_model — it keeps the total FFN parameter count roughly the same while accommodating the extra projection. No bias terms, either — another modern convention.

Rotary Position Embeddings (RoPE)

Position encoding was a solved problem — until models needed to generalize to longer sequences than they trained on. Learned absolute position embeddings (BERT, GPT-2) memorize specific positions and fall off a cliff beyond the training length. Sinusoidal embeddings (original transformer) are fixed but don't capture relative positions well.

RoPE (Su et al., 2021) encodes position by rotating the query and key vectors in pairs of dimensions. Here's the core idea:

# For a 2D example, position m rotates the vector by angle m·θ:
#
#   RoPE(x, m) = [x₁·cos(mθ) - x₂·sin(mθ),
#                  x₁·sin(mθ) + x₂·cos(mθ)]
#
# In matrix form for a pair of dimensions:
#
#   ┌ cos(mθ)  -sin(mθ) ┐   ┌ x₁ ┐
#   │                    │ × │    │
#   └ sin(mθ)   cos(mθ) ┘   └ x₂ ┘
#
# Applied to d-dimensional vectors: pair up dimensions (1,2), (3,4), ..., (d-1,d)
# Each pair gets a different frequency θᵢ = 10000^(-2i/d)
# 
# The magic: q_m · k_n depends only on (m - n), giving you
# relative position encoding for free.

Why this won: when query at position m and key at position n compute their dot product, the rotation matrices combine so that the attention score depends only on the relative distance (m − n), not the absolute positions. This is exactly what you want — "the word 3 positions back" matters, "the word at position 847" doesn't.

The real superpower of RoPE is context length extension. Because positions are encoded as rotation frequencies, you can scale them to handle longer sequences than training saw:

  • Linear interpolation: divide all position indices by a factor (e.g., 4× to go from 4K to 16K context). Simple but loses resolution.
  • NTK-aware scaling: adjust the base frequency (10000 → higher values) to spread rotation angles across the extended range. Better resolution preservation.
  • YaRN (Yet another RoPE extensioN): combines NTK scaling with attention temperature adjustment and per-dimension interpolation ratios. Currently the state-of-the-art for context extension with minimal fine-tuning.

This is why models like LLaMA-3 can train at 8K context but extend to 128K with continued pre-training — RoPE's mathematical structure supports it gracefully.

Grouped-Query Attention (GQA)

Standard multi-head attention (MHA) gives every attention head its own query, key, and value projections. During inference, the KV cache — which stores the key-value pairs for all previous tokens — grows linearly with both sequence length and number of heads. At 70B parameters with 64 heads and a 128K context, that KV cache alone is tens of gigabytes.

Multi-Query Attention (MQA) was the extreme fix: all heads share a single set of KV projections. Massive memory savings, but quality drops because the keys and values lose expressiveness.

Grouped-Query Attention (GQA) is the sweet spot. Instead of 1 KV head (MQA) or N KV heads (MHA), you use G KV head groups, where each group serves N/G query heads:

# Multi-Head Attention (MHA):       64 query heads, 64 KV heads
# Multi-Query Attention (MQA):      64 query heads,  1 KV head
# Grouped-Query Attention (GQA):    64 query heads,  8 KV heads (LLaMA-2 70B)
#
# KV cache reduction: 64/8 = 8× smaller than MHA
#
# Visualization:
# 
# MHA:  Q₁→KV₁  Q₂→KV₂  Q₃→KV₃  Q₄→KV₄  ...  Q₆₄→KV₆₄
#
# GQA:  Q₁─┐         Q₉──┐
#       Q₂─┤         Q₁₀─┤
#       Q₃─┼→ KV₁    Q₁₁─┼→ KV₂    ...   Q₅₇─┐
#       Q₄─┤         Q₁₂─┤                Q₅₈─┤
#       ...┤         ... ─┤                Q₅₉─┼→ KV₈
#       Q₈─┘         Q₁₆─┘                ... ─┤
#                                          Q₆₄─┘
#
# MQA:  Q₁─┐
#       Q₂─┤
#       Q₃─┤
#       ...┼→ KV₁   (all queries share one KV)
#       Q₆₄┘

LLaMA-2 70B uses 8 KV heads for 64 query heads. Mistral-7B uses 8 KV heads for 32 query heads. The quality impact is negligible — within noise of full MHA on benchmarks — but the KV cache savings make long-context inference actually feasible on real hardware.

To put concrete numbers on this: for a 70B model with d_model=8192, 64 heads, head_dim=128, at sequence length 32K with float16:

# KV cache memory comparison at 32K context, 70B model:
#
# MHA (64 KV heads):
#   2 (K+V) × 64 heads × 128 dim × 32768 seq × 80 layers × 2 bytes
#   = ~84 GB  ← doesn't fit on a single A100 (80GB)
#
# GQA (8 KV heads):
#   2 (K+V) × 8 heads × 128 dim × 32768 seq × 80 layers × 2 bytes
#   = ~10.5 GB  ← fits comfortably, leaves room for weights
#
# That's the difference between "needs 2 GPUs just for cache"
# and "runs on one GPU with room to spare."

Sliding Window Attention

Standard causal attention has O(n²) complexity — every token attends to all previous tokens. Sliding window attention (Mistral's approach) restricts each token to a local window of W previous tokens (W = 4096 for Mistral-7B).

# Standard causal attention at position 8000:
#   Token 8000 attends to tokens 0, 1, 2, ..., 7999  (8000 entries)
#
# Sliding window attention (W=4096) at position 8000:
#   Token 8000 attends to tokens 3904, 3905, ..., 7999  (4096 entries)
#
# But information still propagates!
# Layer 1: token 8000 sees tokens [3904, 7999]
# Layer 2: those tokens already encoded info from [0, 7999] via layer 1
# After L layers with window W: effective receptive field = L × W
# Mistral-7B: 32 layers × 4096 window = 131K effective context

The clever bit: information from early tokens still reaches late tokens — it just travels through the layer stack rather than through direct attention. After L layers with window W, the effective receptive field is L × W. Mistral pairs this with a rolling KV cache: the cache is a fixed-size circular buffer of length W, so memory usage is constant regardless of sequence length. You never run out of KV cache memory during inference.

⚠️ Sliding window ≠ lost information

A common misconception: "sliding window attention means the model forgets tokens outside the window." Not exactly. Each layer compresses information from its window into its output representations. Layer 2 sees layer 1's compressed outputs, which already incorporate information from a wider span. By layer 32, each position's representation has been influenced by tokens up to 32 × 4096 = 131K positions away — just indirectly, through the residual stream. It's lossy compared to full attention, but empirically the quality difference is small for most tasks, and the memory savings are enormous.

Mixture-of-Experts (MoE)

MoE keeps showing up in this chapter — Mixtral, GPT-4 (rumored), Gemini — so let's nail down what it actually is. In a standard transformer, every token passes through the same FFN in each layer. In a Mixture-of-Experts layer, there are N parallel FFNs (the "experts"), and a lightweight router network selects the top-K experts for each token:

# MoE FFN layer (Mixtral-style):
# 
# Standard FFN:  y = FFN(x)           — one FFN, all tokens use it
#
# MoE FFN:       router_logits = x @ W_router     (d_model → N_experts)
#                top_k_indices = top_k(router_logits, k=2)
#                weights = softmax(router_logits[top_k_indices])
#                y = Σ weights[i] × Expert_i(x)   — only K experts computed
#
# Mixtral 8x7B:
#   N = 8 experts per layer, K = 2 active per token
#   Total params: ~46.7B (8 expert FFNs × 7B-worth of FFN params)
#   Active params: ~13B per token (only 2 of 8 experts fire)
#   Result: 46.7B-model quality at 13B-model inference cost

The router learns to specialize experts — one might handle code, another mathematical reasoning, another conversational text. The key engineering challenges are load balancing (preventing the router from always picking the same 2 experts, leaving others untrained — solved with auxiliary load-balancing losses) and communication overhead in distributed training (experts on different GPUs need token routing across the network).

MoE is how you scale total knowledge without scaling inference cost proportionally. Expect to see more of it — it's the architecture behind the "how is this model so good for its inference speed?" surprises.

Vocabulary Size & Embedding Choices

A few more decisions that show up in every modern architecture:

Decision Old Default Modern Default Why
Vocab size 30K–50K 32K–128K+ Larger vocab = fewer tokens per text = faster inference + better multilingual coverage. LLaMA-3 uses 128K vocab. Tradeoff: larger embedding matrix.
Tied embeddings Tied (input = output) Often untied Tying saves parameters (one matrix serves both input embed and output projection). Untying lets input and output representations specialize. LLaMA-2 ties them; some models don't.
Bias terms Yes (in attention + FFN) No bias terms Removing bias saves parameters and marginally improves training stability. Every modern LLM has removed them.
Tokenizer WordPiece (BERT), BPE (GPT-2) SentencePiece BPE / Tiktoken Language-agnostic byte-level BPE handles any script. No unknown tokens. Better compression for code and multilingual text.

The Modern LLM Recipe

If you're reading a new model's paper in 2024 and it doesn't explain a specific design choice, it's almost certainly using this default stack:

The 2024 LLM Standard Recipe:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Architecture:    Decoder-only transformer
Normalization:   Pre-norm with RMSNorm
Activation:      SwiGLU in FFN (d_ff ≈ 8/3 × d_model)
Position Enc:    RoPE (Rotary Position Embeddings)
Attention:       GQA (Grouped-Query Attention)
Biases:          None (in attention and FFN)
Vocab:           32K–128K BPE tokens
Context:         8K–128K (RoPE enables extension)
Training:        AdamW, cosine LR schedule, pre-norm
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Deviations from this recipe are the interesting parts of any new paper.
Everything else is assumed.
💡 Why this matters for practitioners

When you understand the standard recipe, reading new model papers becomes much faster. You only need to look for what's different. DeepSeek-V2's MLA? That replaces GQA — interesting, read that section carefully. Mamba's state-space layers? That replaces the entire attention mechanism — major departure. Jamba mixing Mamba + Transformer layers? Hybrid approach. The recipe is your baseline; deviations are the signal.
