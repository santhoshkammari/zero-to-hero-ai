Nice to Know

Chapter 12: Large Language Models Interesting but not essential
TL;DR

Things you’ll encounter in the wild but don’t need to master on day one. Each topic gets a brief explanation and a pointer for when you’d actually need to dig deeper.

How to use this section: Skim the headings. Read any topic that sounds familiar from something you’ve seen in the wild. Skip the rest — you can always come back. The “When you’d care” line at the end of each topic tells you exactly when it becomes relevant.

1. Tokenizer Training Internals

BPE starts with individual characters, then repeatedly merges the most frequent adjacent pair into a new token — thousands of merges later, you have a vocabulary. WordPiece is similar but picks merges by maximizing likelihood rather than raw frequency. Unigram works in reverse: start with a huge vocab and prune tokens that hurt the overall likelihood the least.

Vocab sizes typically land between 32K and 128K — smaller is more compute-efficient, larger handles multilingual and code better. GPT-4 uses ~100K tokens; Llama 2 used 32K; Llama 3 bumped to 128K partly for better multilingual coverage.

When you’d care: building a tokenizer for a new language, a domain-specific corpus (legal, biomedical), or training a model from scratch rather than fine-tuning.

2. MoE Routing Deep Dives

Beyond basic top-k gating, there’s a whole zoo of routing strategies. Expert-choice routing flips the paradigm — experts pick tokens instead of tokens picking experts, giving perfect load balance. Hash routing eliminates the learned router entirely with a fixed hash function. Soft MoE blends expert outputs with soft assignments rather than hard top-k selection. The tricky part is always balancing: auxiliary losses nudge the router toward even expert usage, while router z-loss stabilizes training by penalizing large logits.

When you’d care: training or debugging MoE models from scratch, or diagnosing expert collapse during pretraining.

3. Prefix Language Modeling

Some architectures apply bidirectional attention over a prefix (like an encoder) and then switch to autoregressive (causal) generation for the rest of the sequence. This gives the model a richer understanding of the input context before it starts generating. Google’s UL2 and some T5 variants used this approach.

When you’d care: rarely in practice — modern decoder-only models with long contexts have largely made this pattern obsolete. You might encounter it in older papers or niche encoder-decoder setups.

4. Speculative Decoding Variants

Standard speculative decoding uses a small draft model verified by the big model. The variants get creative. Medusa adds multiple prediction heads to the model itself — no separate draft model needed, each head predicts a different future position. EAGLE works at the feature level rather than token level, drafting from internal representations for higher acceptance rates. Lookahead decoding uses Jacobi iteration to generate and verify multiple tokens simultaneously without any draft model at all.

When you’d care: squeezing maximum inference speed from self-hosted models, especially when you can’t afford the latency of standard autoregressive decoding.

5. Watermarking & AI Text Detection

Statistical watermarking works by partitioning the vocabulary into “green” and “red” lists at each generation step (using a hash of previous tokens as the seed). The model biases toward green tokens. Detection checks whether a text has a statistically improbable number of green tokens. Humans can’t see the watermark, but it’s mathematically detectable. Without watermarking, AI text detection is unreliable — classifiers like GPTZero have high false-positive rates and are easily fooled by paraphrasing.

When you’d care: content provenance, academic integrity systems, or regulatory compliance requiring generated-content labeling.

The detection arms race: Without watermarking, distinguishing AI text from human text is fundamentally unreliable. Any detection method based purely on style or perplexity can be defeated by simple paraphrasing, and false-positive rates are unacceptably high for consequential decisions. If you need reliable detection, you need watermarking at generation time.

6. RWKV & Linear Attention

RWKV is a hybrid that trains like a transformer (parallelizable) but infers like an RNN (constant memory per token). It replaces quadratic attention with a linear recurrence — meaning inference cost scales O(N) with sequence length, not O(N²). Mamba and other state-space models follow a similar philosophy: keep transformer-quality language modeling while making inference tractable for extremely long sequences.

When you’d care: working with extremely long sequences (100K+ tokens), resource-constrained inference on edge devices, or when you need constant-memory streaming generation.

7. Model Merging

Why fine-tune when you can frankenstein? SLERP (Spherical Linear Interpolation) smoothly blends two models’ weights along a curved path in weight space. TIES resolves sign conflicts between task vectors before merging, keeping only the parameters that agree on direction. DARE randomly drops most delta parameters (differences from the base model) and rescales the survivors.

Tools like mergekit make this a one-command operation:

# mergekit-yaml config (simplified)
models:
  - model: coding-llama-7b
    parameters: { weight: 0.5 }
  - model: chat-llama-7b
    parameters: { weight: 0.5 }
merge_method: slerp
base_model: meta-llama/Llama-2-7b

It’s wildly popular on the HuggingFace open-model leaderboard — many top models are merges, not fresh fine-tunes.

When you’d care: combining capabilities from different fine-tunes (say, a coding model + a chat model) without the compute cost of retraining.

8. Activation Sparsity

Here’s a fun fact: in ReLU-based models, 90%+ of neuron activations are zero or near-zero for any given input. That means you’re doing a ton of math that multiplies by zero. If you can predict which neurons will fire, you skip the rest. PowerInfer exploits this for consumer GPUs — it preloads “hot” neurons on the GPU and fetches cold ones from CPU RAM on demand, achieving 10x+ speedups over naive offloading.

When you’d care: running large models on consumer hardware (gaming GPUs, laptops) where you can’t fit the full model in VRAM.

9. Multi-Token Prediction

Standard LLM training predicts one next token per position. Multi-token prediction trains the model to predict the next 2, 4, or even 8 tokens simultaneously using multiple prediction heads. Meta’s research showed this improves reasoning (the model has to “think further ahead”) and code generation quality. Bonus: at inference time, those extra heads can serve as a built-in draft model for self-speculative decoding, giving you faster generation for free.

When you’d care: understanding next-gen pretraining objectives, or evaluating models that use multi-token prediction (like DeepSeek or Meta’s research models).

Why this matters beyond speed: Multi-token prediction forces the model to develop better internal representations. To predict token +4, you implicitly need to “plan” the intervening tokens. This planning pressure is believed to improve reasoning and reduce the “greedy one-step” failure mode of standard next-token training.

10. Decontamination

If your training data accidentally contains benchmark test questions, your model’s scores are meaningless — it’s memorized the answers, not learned the skills. Decontamination detects this. N-gram overlap finds verbatim or near-verbatim matches between training data and benchmark examples. Min-k% prob checks if the model assigns suspiciously high probability to benchmark text (suggesting memorization). Serious benchmark papers now include a decontamination section or their results get dismissed.

When you’d care: publishing fair benchmark results, validating claims from open-source model releases, or curating training datasets.

11. Constitutional AI Details

Anthropic’s Constitutional AI replaces human labelers with a principled self-improvement loop. Step 1: the model generates responses. Step 2: the model critiques its own responses against a written constitution (“Is this harmful? Is this honest?”). Step 3: it revises based on the critique. Step 4: the revised pairs become training data for RLAIF (RL from AI Feedback) — same as RLHF but the preference labels come from the AI, not humans. This scales safety training without an army of human annotators.

When you’d care: building safety-aware training pipelines, or designing self-improving alignment systems where human labeling is a bottleneck.

12. Flash Attention

Standard attention materializes the full N×N attention matrix in GPU HBM (High Bandwidth Memory) — that’s O(N²) memory and a brutal bottleneck for long sequences. Flash Attention never materializes the full matrix. Instead, it tiles the computation into blocks that fit in fast SRAM, computes attention block by block, and accumulates results on the fly.

This is an IO-aware algorithm — it’s not changing the math, just where the math happens. The key insight: moving data between HBM and SRAM is the bottleneck, not the arithmetic itself.

# Conceptual pseudocode (massively simplified)
for block_q in tiled_queries:        # outer loop over query blocks
    for block_k, block_v in tiled_kv:  # inner loop over key-value blocks
        local_attn = block_q @ block_k.T   # computed in SRAM
        local_out  = softmax(local_attn) @ block_v
        accumulate(output, local_out)  # running softmax trick

Result: O(N) memory, 2–4x faster training, and longer context lengths. Flash Attention is now the default in PyTorch, HuggingFace, and every serious training framework. Flash Attention 2 and 3 added further optimizations for newer GPU architectures.

When you’d care: understanding why modern models train faster than they “should,” debugging training performance, or implementing custom attention patterns.

13. Full-Duplex Voice AI

Traditional voice assistants pipeline speech-to-text → LLM → text-to-speech, introducing 1–3 seconds of latency:

Traditional: Audio → STT → Text → LLM → Text → TTS → Audio  (1-3s latency)
Native:      Audio tokens → Multimodal LLM → Audio tokens     (<300ms latency)

GPT-4o-style models handle speech natively — audio tokens go in, audio tokens come out. “Full-duplex” means the model can listen while it speaks, enabling natural interruptions and backchannels (“uh-huh,” “right”). The model also captures tone, pacing, and emotion directly from the audio signal rather than flattening everything to text first.

When you’d care: building voice-first AI products, real-time conversation agents, or any application where sub-second response time matters.

14. Tensor Parallelism Details

Tensor parallelism splits individual layers across GPUs, unlike pipeline parallelism which puts different layers on different GPUs. For attention, the heads are naturally sharded — each GPU computes a subset of heads. For FFN layers, you split the weight matrix by columns on the first linear layer (each GPU computes a slice of the hidden dimension) and by rows on the second (so the outputs can be summed via all-reduce).

This requires fast inter-GPU communication (NVLink, not PCIe) because every layer needs a synchronization step. The communication overhead means you rarely get perfect linear scaling — 4 GPUs won’t give you 4x the throughput, but they will let you run a model that doesn’t fit on one card.

When you’d care: serving or fine-tuning models too large for a single GPU, configuring multi-GPU inference in vLLM/TensorRT-LLM, or understanding why your multi-GPU setup isn’t scaling linearly.

The point of “Nice to Know”: None of these topics are required to use LLMs effectively. They’re the things that show up in research papers, conference talks, and HuggingFace model cards. Skim them now so you recognize the terms when they appear — then come back and dig deeper when a specific project demands it.
