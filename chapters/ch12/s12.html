MoE in Large Language Models

Chapter 12: Large Language Models 13 subtopics
TL;DR

Dense models activate every parameter for every token — brutally expensive at scale. Mixture of Experts (MoE) replaces the feed-forward layer with N parallel expert networks and a learned router that picks just top-k experts per token. You get a model with trillions of total parameters but only billions of active parameters per forward pass. Switch Transformer simplified this to top-1 routing and scaled to 1.6T parameters. Mixtral proved open MoE models can match GPT-3.5-class performance. The tradeoffs are real — routing instability, load imbalance, and communication overhead during training — but the economics are compelling: more capacity for the same inference cost.

Why MoE?

Here's the problem. You train a 70-billion-parameter dense model. Every single token — whether it's the word "the" or a complex calculus equation — activates all 70 billion parameters. That's wasteful. The word "the" doesn't need the same computational firepower as a multi-step reasoning chain. You're paying for a full hospital visit when you just need a bandaid.

That hospital analogy is actually perfect. A real hospital doesn't send every patient to every doctor. A receptionist (the router) quickly triages you and sends you to the right specialist. The cardiologist handles heart issues. The dermatologist handles skin problems. The hospital has dozens of specialists on staff (huge total capacity), but each patient only sees one or two (small active capacity). This is exactly how Mixture of Experts works.

The core insight: decouple total parameter count from compute cost per token. A dense 70B model uses 70B parameters per token. An MoE model might have 400B total parameters but only activate 50B per token. You get the knowledge capacity of a much larger model at the inference cost of a much smaller one. This isn't a free lunch — training is harder, memory is higher, and the engineering complexity is real — but the economics are so compelling that almost every frontier lab is now building MoE models. GPT-4 is widely believed to be MoE. Mixtral proved the approach works for open models. DeepSeek-V2 pushed it further with novel routing. The trend is clear.

Sparse MoE Architecture

In a standard Transformer block, you have attention followed by a feed-forward network (FFN). The FFN is two linear layers with an activation function in between. In an MoE Transformer, you replace that single FFN with N parallel FFN networks — the experts — plus a small gating network called the router. Everything else stays the same. Attention is unchanged. Layer norms are unchanged. Only the FFN is swapped out.

Here's what happens during a forward pass. A token's hidden state comes out of the attention layer. It hits the router, which is just a linear layer mapping the hidden dimension to N scores (one per expert). The router applies softmax and picks the top-k experts. The token's hidden state is then sent to those k experts, each of which processes it independently through their own FFN. The outputs are weighted by the router's scores and summed together. That weighted sum replaces what the original single FFN would have produced.

The beautiful part: this is a drop-in replacement. From the perspective of the rest of the Transformer, the MoE layer takes in a hidden state and outputs a hidden state of the same shape. It's modular.

import torch
import torch.nn as nn
import torch.nn.functional as F

class Expert(nn.Module):
    """A single expert: just a standard FFN."""
    def __init__(self, d_model, d_ff):
        super().__init__()
        self.w1 = nn.Linear(d_model, d_ff)
        self.w2 = nn.Linear(d_ff, d_model)
        self.act = nn.SiLU()

    def forward(self, x):
        return self.w2(self.act(self.w1(x)))

class MoELayer(nn.Module):
    """Mixture of Experts layer with top-k routing."""
    def __init__(self, d_model, d_ff, num_experts=8, top_k=2):
        super().__init__()
        self.num_experts = num_experts
        self.top_k = top_k
        self.experts = nn.ModuleList(
            [Expert(d_model, d_ff) for _ in range(num_experts)]
        )
        # Router: linear layer mapping hidden state to expert scores
        self.router = nn.Linear(d_model, num_experts, bias=False)

    def forward(self, x):
        # x shape: (batch, seq_len, d_model)
        batch, seq_len, d_model = x.shape
        x_flat = x.view(-1, d_model)  # (batch * seq_len, d_model)

        # Compute router logits and pick top-k experts
        router_logits = self.router(x_flat)           # (tokens, num_experts)
        top_k_logits, top_k_indices = torch.topk(
            router_logits, self.top_k, dim=-1
        )
        top_k_weights = F.softmax(top_k_logits, dim=-1)  # normalize

        # Dispatch tokens to selected experts
        output = torch.zeros_like(x_flat)
        for i in range(self.top_k):
            expert_idx = top_k_indices[:, i]       # which expert per token
            weight = top_k_weights[:, i].unsqueeze(-1)
            for e in range(self.num_experts):
                mask = (expert_idx == e)
                if mask.any():
                    expert_input = x_flat[mask]
                    expert_output = self.experts[e](expert_input)
                    output[mask] += weight[mask] * expert_output

        return output.view(batch, seq_len, d_model)
Why Only the FFN?

Attention layers share information across tokens — they need a global view. But the FFN operates independently on each token's hidden state. This token-level independence is what makes it natural to route different tokens to different expert FFNs. Some research has explored MoE attention heads, but MoE-on-FFN is the dominant paradigm because it's simpler and more stable.

Gating Networks and Routing

The router is the brain of the MoE layer. It decides which experts see which tokens. Getting this right is the single most important design choice in MoE, and getting it wrong leads to catastrophic failure modes.

Top-k Routing

The simplest approach: take the router's logits, pick the k experts with the highest scores, normalize their scores to sum to 1, and use those as weights. When k=2, each token visits two experts and the outputs are blended. When k=1, each token visits exactly one expert (no blending). Top-2 gives better quality because experts can complement each other. Top-1 is faster because you only run one expert per token. Most production systems use k=1 or k=2.

Noisy Top-k Gating

The original MoE paper by Shazeer et al. (2017) introduced noisy top-k gating. Before selecting the top-k experts, you add tunable Gaussian noise to the router logits. This noise encourages exploration during training — without it, the router quickly converges to always sending tokens to the same two experts and the other experts never get trained. The noise acts like epsilon-greedy exploration in reinforcement learning. During inference, you turn off the noise and route deterministically.

def noisy_top_k_gating(x, router, num_experts, top_k, noise_std=1.0):
    """Noisy top-k gating from Shazeer et al. (2017)."""
    logits = router(x)  # (batch, num_experts)

    if self.training:
        noise = torch.randn_like(logits) * noise_std
        logits = logits + noise

    top_k_logits, top_k_indices = torch.topk(logits, top_k, dim=-1)
    # Zero out non-selected experts (for gradient flow)
    mask = torch.full_like(logits, float('-inf'))
    mask.scatter_(1, top_k_indices, top_k_logits)
    weights = F.softmax(mask, dim=-1)

    return weights, top_k_indices

Expert Choice Routing

Standard routing lets tokens choose experts. Expert Choice routing (Zhou et al., 2022) flips this: each expert chooses its top-k tokens from the batch. The expert looks at all router scores for its slot and picks the tokens it's most confident about. This elegantly solves load balancing — every expert processes exactly the same number of tokens by construction. The downside is that some tokens might be selected by many experts (wasting compute) while others might not be selected at all (losing information). In practice, Expert Choice works surprisingly well and eliminates the need for auxiliary balancing losses entirely.

Switch Transformer's Top-1 Routing

Google's Switch Transformer (Fedus et al., 2022) argued that simpler is better. Use k=1. Each token goes to exactly one expert. No weighted combination, no blending. This halves the expert compute per token and simplifies the implementation enormously. The quality tradeoff is small, and the speed advantage is significant. Most modern MoE systems follow Switch Transformer's lead and use top-1 or top-2 routing.

Load Balancing

This is where MoE gets tricky. Without constraints, the router will collapse. Here's why: early in training, one expert randomly performs slightly better than others. The router sends more tokens to it. That expert gets more gradient signal. It improves faster. The router sends even more tokens to it. This is a positive feedback loop that ends with one or two experts handling everything while the rest are dead weight. This phenomenon is called expert collapse, and it's the central failure mode of MoE training.

Auxiliary Load-Balancing Loss

The standard fix is to add an auxiliary loss that penalizes uneven load distribution. You compute two things for each expert: the fraction of tokens routed to it (f_i) and the average router probability assigned to it (P_i). The auxiliary loss is their dot product, scaled by the number of experts. When load is perfectly balanced, each expert gets 1/N of the tokens, and the loss is minimized. When load is skewed, the loss is high.

Formally, for N experts across a batch of T tokens:

# Load-balancing auxiliary loss
# f_i = fraction of tokens dispatched to expert i
# P_i = mean router probability for expert i across all tokens

def load_balancing_loss(router_logits, expert_indices, num_experts):
    """
    Auxiliary loss from Switch Transformer.
    Encourages equal token distribution across experts.
    """
    router_probs = F.softmax(router_logits, dim=-1)  # (T, num_experts)
    num_tokens = router_logits.shape[0]

    # f_i: fraction of tokens assigned to each expert
    # Using one-hot encoding of chosen experts
    expert_mask = F.one_hot(expert_indices, num_experts).float()  # (T, N)
    f = expert_mask.mean(dim=0)  # (N,) — fraction per expert

    # P_i: average router probability per expert
    P = router_probs.mean(dim=0)  # (N,) — mean prob per expert

    # Loss = N * sum(f_i * P_i) — minimized when uniform
    aux_loss = num_experts * (f * P).sum()
    return aux_loss

The coefficient on this auxiliary loss matters. Too small and experts still collapse. Too large and the router is forced to distribute uniformly, which defeats the purpose of routing (you want the router to make smart choices, not random ones). Typical values range from 0.01 to 0.1.

Router z-Loss

Router z-loss (introduced in ST-MoE) prevents the router logits from growing too large in magnitude. Large logits lead to very peaked softmax distributions, which means small perturbations cause drastic routing changes — instability during training. The z-loss is simply the log of the sum of squared exponentials of the logits. It gently pushes logits toward zero, keeping the softmax distribution smoother.

Auxiliary-Loss-Free Balancing

Recent work (DeepSeek-V2, for instance) explores balancing without any auxiliary loss. One approach: add a per-expert bias term that is adjusted dynamically based on expert load. If an expert is overloaded, reduce its bias. If underloaded, increase it. This is simpler to tune than an auxiliary loss coefficient and adapts during training rather than relying on a fixed penalty strength. Another approach is sequence-level balancing, where you enforce balance within each sequence rather than across the entire batch, giving the router more freedom while still preventing collapse.

The Balancing Tightrope

Load balancing is a tension between two goals. You want the router to make intelligent routing decisions (some tokens genuinely benefit from specific experts). But you also need all experts to be utilized — dead experts are wasted parameters. Push too hard on balance and you get random routing. Push too little and experts collapse. Every MoE system walks this tightrope, and there's no universally optimal balance coefficient.

Expert Capacity and Token Dropping

In practice, you can't let a single expert handle an unlimited number of tokens per batch. GPU kernels work best with fixed-size tensors. If expert 3 gets 80% of the tokens and expert 7 gets 1%, you can't efficiently parallelize that. So MoE systems define an expert capacity: the maximum number of tokens an expert can process per batch.

The capacity factor (CF) controls this. If you have T tokens and N experts with top-1 routing, the ideal load per expert is T/N. The capacity is CF × (T/N). A capacity factor of 1.0 means each expert can handle exactly its fair share — no slack. A factor of 1.5 means 50% buffer for imbalanced routing. When an expert's buffer is full, excess tokens are dropped — their hidden states pass through unchanged (via the residual connection) and skip the MoE layer entirely.

Token dropping is a lossy operation. Those dropped tokens get no expert processing, which degrades quality. But it's necessary for computational efficiency. The tradeoff: higher capacity factor means fewer dropped tokens but more wasted compute (empty expert slots). Lower capacity factor means tighter packing but more token drops. In practice, CF = 1.25 is a common default — a 25% buffer handles most routing imbalances without too much waste.

Switch Transformer

Google's Switch Transformer (2022) was a landmark paper that dramatically simplified MoE and showed it scales beautifully. The key simplifications were brutal in their elegance: use top-1 routing (each token goes to exactly one expert), use a simple linear router with no noise, and rely entirely on the auxiliary load-balancing loss for expert utilization.

The results were striking. Switch Transformer scaled to 1.6 trillion parameters (with 2048 experts) and achieved a 4–7× speedup in pre-training compared to the dense T5-XXL model with equivalent compute. The model trained on the same data in a fraction of the time because each token only activated a tiny fraction of the total parameters.

Switch Transformer also introduced the capacity factor concept and showed that even with moderate token dropping, the model quality remained strong. The key insight was that simplicity wins at scale: top-1 routing with a simple auxiliary loss outperformed more complex routing schemes once you have enough experts.

Mixtral

Mixtral 8x7B (Mistral AI, 2024) brought MoE to the open-source world in a big way. The architecture is straightforward: 8 expert FFNs per MoE layer, top-2 routing, and a total of roughly 47 billion parameters with approximately 13 billion active per token. It's based on the Mistral 7B architecture — each expert is roughly a 7B-class FFN.

The results shocked people. Mixtral matched or exceeded LLaMA 2 70B on most benchmarks while being dramatically faster at inference (because it only activates 13B parameters per token instead of 70B). It showed that a well-designed MoE model with less than 50B total parameters could compete with dense models 1.5× its size. Mixtral also demonstrated excellent multilingual capabilities, with experts showing some degree of language-specific specialization.

The "8×7B" Naming Trap

Mixtral 8×7B does NOT have 8 × 7B = 56B parameters. The name is misleading. Only the FFN layers are replicated across experts. Attention layers, embeddings, and layer norms are shared. The total is about 47B. Always distinguish between total parameters (what you need in memory) and active parameters (what determines inference speed).

Expert Specialization

What do experts actually learn? This question has fascinated researchers since MoE was introduced. The answer is nuanced and somewhat surprising.

Early analyses of Switch Transformer found weak but detectable specialization. Some experts preferred tokens related to specific topics (science, law, code). Others preferred specific syntactic positions (tokens at the beginning of sentences, tokens after punctuation). But the specialization was noisy — no expert was a pure "code expert" or a pure "math expert." There was significant overlap.

Mixtral's experts showed clearer patterns. Analysis revealed that certain experts were consistently chosen for code tokens, others for mathematical notation, and others for specific languages in multilingual text. But even here, every expert handled a broad mix of tokens. The specialization is more like "preference" than "exclusive domain." Think of it less as specialist doctors and more as generalist doctors who happen to be a bit better at certain conditions.

Some research has explored expert merging: after training, identify experts that have converged to similar functions and merge them to reduce model size. This works surprisingly well when two experts are near-duplicates, but you need careful analysis to avoid merging experts that look similar on average but specialize differently on tail distributions.

Routing Strategies Deep Dive

Hash Routing

Hash routing takes the simplest possible approach: route tokens based on a hash function of their input. No learned router, no gating network, no training instability. You hash the token embedding (or even just the token ID) and map it to an expert deterministically. The Roller et al. (2021) paper showed that hash routing performs surprisingly close to learned routing, especially when the number of experts is large. The advantage is rock-solid stability and zero auxiliary losses. The disadvantage is that the routing can't adapt to context — the same token always goes to the same expert regardless of what came before it.

Soft MoE

Soft MoE (Puigcerver et al., 2023) eliminates hard token-to-expert assignment entirely. Instead of routing discrete tokens to discrete experts, every expert processes a weighted combination of all tokens. The router produces a full attention-style weight matrix between tokens and experts. Each expert gets a soft blend of all tokens, weighted by relevance. This removes token dropping entirely and makes the whole system differentiable end-to-end. The cost is that every expert processes every token (at different weights), which reduces the sparsity advantage. Soft MoE works well for vision (V-MoE) where batch sizes are large enough to amortize the cost.

Topology-Aware Routing

In distributed systems where experts live on different GPUs or even different machines, routing decisions have network implications. Sending a token to an expert on a remote GPU requires all-to-all communication. Topology-aware routing considers network topology when making routing decisions. If two experts have equal scores, prefer the one on the local GPU. This reduces communication overhead at the cost of slightly suboptimal routing quality. DeepSeek-V2 and Megatron-LM implement versions of this optimization.

Expert Offloading

For inference on consumer hardware, you can't fit all experts in GPU memory. Expert offloading keeps only the active experts on the GPU and stores the rest on CPU RAM or even disk. When a token needs a different expert, you swap it in. Because MoE is sparse (each token only needs k experts), you only need k experts in GPU memory at any moment. This makes trillion-parameter models runnable on a single consumer GPU — slowly, but it works. Libraries like llama.cpp and ExLlamaV2 implement expert offloading for Mixtral.

Training Challenges

MoE models are harder to train than dense models. Period. There are three major categories of difficulty, and you need solutions for all of them.

First, routing instability. Early in training, the router hasn't learned anything meaningful. Its initial random decisions create feedback loops (expert collapse, as discussed above). Even with auxiliary losses, training can be unstable for the first few thousand steps. Warmup strategies help: start with a higher load-balancing coefficient and anneal it down, or use dropout on the router during early training.

Second, communication overhead. In distributed training, MoE requires all-to-all communication. Each GPU has a subset of experts. Each GPU has a subset of tokens. After routing, each token needs to be sent to the GPU that holds its assigned expert, processed, and the result sent back. This all-to-all pattern is fundamentally different from the all-reduce pattern used in dense model parallelism, and it doesn't overlap with computation as cleanly. At large scale (hundreds of GPUs), all-to-all communication can become the bottleneck.

Third, memory management. Even though each token only uses k experts, during training you still need all N experts in memory (for backpropagation). With 128 experts, each being a multi-billion-parameter FFN, memory requirements are enormous. Expert parallelism helps (spread experts across GPUs), but you're now juggling data parallelism, tensor parallelism, pipeline parallelism, AND expert parallelism simultaneously. The engineering complexity is substantial.

Training Stability Checklist

If your MoE model is training poorly, check these in order: (1) Are experts collapsing? Monitor the percentage of tokens going to each expert — it should be roughly uniform. (2) Are router logits exploding? Add router z-loss. (3) Is auxiliary loss too high or too low? Try 0.01 as a starting point. (4) Is the capacity factor too tight? Try increasing it to 1.5. (5) Is all-to-all communication a bottleneck? Profile your communication-to-computation ratio.

Inference Optimization

Training MoE is hard, but inference is where MoE shines — if you optimize it correctly. The key insight: at inference time, each token only needs k experts. You have several strategies to exploit this.

Expert parallelism distributes experts across GPUs. If you have 8 experts and 8 GPUs, each GPU holds one expert. Tokens are routed to the appropriate GPU, processed, and results sent back. This is the simplest strategy for multi-GPU inference. The communication pattern is predictable and can be overlapped with computation.

Expert offloading (mentioned earlier) is the key technique for consumer hardware. In the extreme case, you keep only 1–2 experts on the GPU and swap the rest from CPU RAM. With Mixtral's top-2 routing, you need at most 2 experts at a time out of 8. That means you need roughly 25% of the total expert memory on the GPU. Combined with quantization (4-bit or 8-bit weights), this makes even 47B-parameter Mixtral runnable on a 24GB GPU.

Speculative expert loading goes further: predict which experts the next few tokens will need (based on the input pattern) and start loading them before the router makes its decision. If the prediction is right, you eliminate the swap latency entirely. If wrong, you fall back to on-demand loading. This is analogous to CPU branch prediction and, like branch prediction, it works surprisingly well because expert routing patterns have locality — consecutive tokens often route to the same experts.

Batch routing optimization groups tokens by their assigned expert. Instead of processing tokens one at a time (each needing a potentially different expert), you batch all tokens assigned to expert 3, process them together, then batch all tokens for expert 5, and so on. This maximizes GPU utilization because each expert processes a full batch rather than individual tokens.

# Expert offloading: pseudocode for consumer GPU inference
class MoEOffloadEngine:
    def __init__(self, experts, device='cuda'):
        self.experts = experts          # all experts on CPU
        self.cache = {}                 # GPU cache: expert_id -> module
        self.cache_size = 2             # keep top-2 experts on GPU

    def get_expert(self, expert_id):
        if expert_id in self.cache:
            return self.cache[expert_id]

        # Evict least recently used if cache full
        if len(self.cache) >= self.cache_size:
            lru_id = min(self.cache, key=lambda k: self.cache[k]._last_used)
            self.cache[lru_id].cpu()    # move back to CPU
            del self.cache[lru_id]

        # Load requested expert to GPU
        self.experts[expert_id].cuda()
        self.cache[expert_id] = self.experts[expert_id]
        return self.cache[expert_id]

    def forward(self, hidden_states, router_logits):
        top2_ids = router_logits.topk(2).indices
        output = torch.zeros_like(hidden_states)
        weights = F.softmax(router_logits.topk(2).values, dim=-1)

        for i in range(2):
            expert = self.get_expert(top2_ids[0, i].item())
            expert._last_used = time.time()
            output += weights[0, i] * expert(hidden_states)
        return output

Advanced Topics

Vision MoE (V-MoE)

MoE isn't limited to language. Vision MoE (Riquelme et al., 2021) applies the same idea to Vision Transformers (ViTs). Image patches replace tokens, and experts specialize in processing different types of visual content — some patches contain textures, others contain edges or objects. V-MoE scaled to 15B parameters and matched the performance of much larger dense ViTs. Interestingly, expert specialization was clearer in vision than in language: some experts clearly preferred patches with specific visual characteristics.

Dense-MoE Hybrids

Not every Transformer layer needs to be MoE. Many architectures alternate between dense FFN layers and MoE layers. Mixtral, for instance, uses MoE in every layer. But you could use MoE only in every other layer, or only in the middle layers of the network. The tradeoff is parameter count vs. routing complexity. DeepSeek-V2 uses a hybrid approach with some shared experts that process every token (like dense layers) plus sparse routed experts. This gives you a stable baseline from the shared experts while getting extra capacity from the routed ones.

MoE Distillation

A trained MoE model has enormous knowledge capacity, but it's expensive to serve because of memory requirements (all experts must be stored). MoE distillation trains a smaller dense student model to mimic the MoE teacher's outputs. The student can't match the teacher's full performance, but it can capture most of the knowledge at a fraction of the memory cost. This is especially useful for edge deployment where you can't store 8 expert networks but can store one good dense network that learned from them.

Multi-Head Latent Attention (MLA)

DeepSeek-V2 introduced Multi-Head Latent Attention (MLA), which compresses the key-value cache into a low-rank latent space. This isn't strictly MoE, but it's a critical companion technique. MoE models often have large hidden dimensions (to give each expert meaningful capacity), which means KV caches grow quickly. MLA projects keys and values into a much smaller latent dimension, reducing KV cache memory by 90%+ while maintaining attention quality. Combined with MoE's sparse activation, you get a model that's efficient in both computation and memory.

Multi-Token Prediction

Multi-token prediction trains the model to predict several future tokens at once rather than just the next one. In MoE models, this is particularly powerful because the router can use the multi-token prediction heads to anticipate which experts future tokens will need, enabling speculative expert loading. Meta's research showed that multi-token prediction objectives improve sample efficiency during training and can be combined with MoE to further reduce per-token compute.

Grouped Query Attention (GQA)

Grouped Query Attention groups multiple query heads to share the same key-value head. Standard multi-head attention has separate K, V projections per head. GQA uses, say, 8 KV heads for 32 query heads. Mixtral uses GQA to keep inference memory manageable. In MoE models, where you're already memory-constrained by storing multiple experts, GQA's KV cache savings are critical. Without GQA, a large MoE model's attention KV cache could exceed the memory saved by sparse expert activation.

Fine-Grained Expert Segmentation

Instead of 8 large experts, use 64 smaller experts and route to top-8. Fine-grained segmentation (used in DeepSeek-V2) increases routing flexibility. With more experts, the router can make more precise assignments. Each expert is smaller, so the capacity per expert is lower, but the combination of top-k small experts can match or exceed a single large expert. The tradeoff is increased routing overhead and more complex load balancing.

Shared Experts

Shared experts (DeepSeek-V2, Snowflake Arctic) are experts that process every token regardless of routing decisions. They capture common knowledge that all tokens need, while routed experts capture specialized knowledge. Think of it as: shared experts handle the grammar and common patterns, routed experts handle domain-specific knowledge. This stabilizes training because the model always has a baseline (the shared expert) even if routing fails.

DeepSeek-V2's Architecture

DeepSeek-V2 combines MLA, fine-grained expert segmentation (160 routed experts + 2 shared experts per layer), auxiliary-loss-free balancing, and multi-token prediction. With 236B total parameters and only 21B active, it achieves performance competitive with much larger models. It's arguably the most architecturally ambitious open MoE model as of 2024.

Scaling to Trillions

Where is MoE heading? The honest answer: everywhere. The economic argument is too strong to ignore. Training a dense 1T-parameter model would require an astronomical compute budget and inference would be cripplingly expensive. But an MoE model with 1T total parameters and 50B active parameters? That's trainable with current hardware and servable at reasonable cost.

The scaling laws support this. Research from Google, Meta, and Mistral consistently shows that MoE models achieve better loss per FLOP than dense models. If you have a fixed compute budget, you're better off training a large MoE model than a smaller dense model. The gap widens as you scale: at 100B active parameters, MoE might be 10% better per FLOP. At 1T total parameters, the advantage grows significantly.

But the challenges scale too. Communication overhead grows superlinearly with expert count. Load balancing becomes harder with thousands of experts across thousands of GPUs. Memory management becomes a distributed systems problem. You need expert parallelism, pipeline parallelism, data parallelism, and tensor parallelism all working together, and the interaction between these parallelism strategies creates complex optimization problems.

The industry consensus seems to be converging on a practical sweet spot: 8–128 experts per layer, top-1 or top-2 routing, with shared experts for stability, and fine-grained segmentation for flexibility. Models like Mixtral (8 experts), DeepSeek-V2 (160 routed + 2 shared), and the rumored architecture of GPT-4 (speculated 8–16 experts) all fit this pattern. The extreme end — Switch Transformer's 2048 experts — was impressive as a research demonstration but hasn't been replicated in production systems because the communication overhead is too high.

What's exciting is that MoE makes trillion-parameter models economically viable. The cost equation changes from "can we afford to run this?" to "can we engineer the routing and communication well enough?" That's a solvable engineering problem, not a fundamental limitation. The next generation of frontier models will almost certainly be MoE, and the techniques in this section — routing strategies, load balancing, expert specialization, inference optimization — are the building blocks that make them possible.
