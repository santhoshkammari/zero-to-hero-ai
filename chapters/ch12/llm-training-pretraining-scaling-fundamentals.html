Technically correct, practically useless: “Next-token prediction on lots of text.” Like saying skyscraper comes from “stacking stuff up.” Real story between “download Common Crawl” and “you have a working chatbot” is enormous engineering pipeline. Here is dive.

LLM pretraining teaches randomly initialized neural network to understand and generate language by exposing it to enormous text. Model learns no specific task — learns language itself, building internal representations of syntax, semantics, world knowledge, and reasoning patterns. GPT (2018) and BERT (2018) popularized pretraining. Since then, pretraining became foundation of every major language AI system.

Where Does the Training Data Come From?

Imagine training small language model for recipe website. Want completions like “Preheat the oven to” or “Stir the mixture until.” Need training text source.

For tiny recipe model, maybe scrape three cooking blogs. Get maybe 10,000 sentences. Real LLM uses same answer — scrape web — but at hard-to-grasp scale. Primary source for most open LLMs is Common Crawl, nonprofit crawling web monthly since 2008. Each monthly snapshot is tens of terabytes of raw HTML. All of it. News articles, blog posts, forum threads, product listings, cookie consent banners, spam, astonishing number of pages made only of phrase “lorem ipsum.”

Common Crawl alone gives hundreds of trillions of words. Raw web text is flood river: water, driftwood, plastic bags, occasional shopping cart. First look at raw Common Crawl surprised me. Huge amount is garbage. Not edge-case garbage. Majority garbage.

Major open pretraining datasets use different ways to tame flood:

The Pile, built by EleutherAI, used curation-first approach. Instead of one giant web crawl, team assembled 22 distinct subsets: Wikipedia, ArXiv papers, PubMed biomedical literature, GitHub code, Stack Exchange Q&A, Project Gutenberg books, YouTube subtitles, even Enron emails and Ubuntu IRC chat logs. Each subset represented different kind of language model should learn. Total: 825 GB of text. Deliberately diverse.

RedPajama, from Together AI, tried to reproduce data recipe behind LLaMA. It combined filtered Common Crawl with Wikipedia, books, ArXiv, Stack Exchange, PubMed, and GitHub code — roughly 1.2 trillion tokens. Key word is “reproduce” — goal was showing open dataset could match what Meta used behind closed doors.

RefinedWeb, built for Falcon models, used opposite philosophy. Instead of curating many sources, team focused entirely on Common Crawl and applied brutally aggressive filtering. Thesis: clean web data well enough, curated sources not needed. Bold claim. Falcon’s strong performance backed it up.

Back to recipe model. We scraped three blogs and got 10,000 sentences. That is our “Common Crawl.” Now need cleaning. Hardest part of data preparation starts here — part that gets too little credit.

Cleaning the Mess

Recipe scrape has duplicates. Lots. Same chocolate chip cookie recipe appeared on all three blogs (all copied from same source, naturally). Also navigation menus, ad text, comment sections full of “great recipe!!!”, one blog auto-generated 500 pages of keyword-stuffed SEO nonsense about “best recipe for recipe recipes.”

Cleaning has two stages: deduplication and quality filtering. Both matter more than most people think.

Deduplication

Assumed web data mostly unique. Wrong by embarrassing margin. Studies find 30–50% of web-crawled text is near-duplicate content. Not vaguely similar — near-identical. Boilerplate footers, syndicated articles, copy-pasted forum posts, scraped mirrors of same sites.

Training on duplicates is worse than wasteful. It makes model memorize passages instead of generalizing from them. Same paragraph appears thousand times, model learns to recite it verbatim. Bad for generalization and privacy.

Standard deduplication approach uses MinHash with Locality-Sensitive Hashing (LSH). Trace it through recipe example. Suppose we have three documents:

Doc A: "Preheat oven to 350°F. Mix flour, sugar, and butter."
Doc B: "Preheat oven to 350°F. Mix flour, sugar, and butter. Add eggs."
Doc C: "Sauté onions in olive oil until translucent."

First, break each document into overlapping word chunks called n-grams (typically 5-grams). Doc A becomes set {“Preheat oven to 350°F Mix”, “oven to 350°F Mix flour”, ...}. Doc B gives very similar set with few extra pieces at end.

Next, apply multiple hash functions to each n-gram set and keep minimum hash value for each function. This makes compact fingerprint called MinHash signature — maybe 128 numbers summarizing entire document. Key math property: probability two documents share MinHash value equals their Jaccard similarity (overlap of n-gram sets divided by union).

Then LSH groups documents with similar fingerprints into buckets. Inside each bucket, compute exact Jaccard similarity. Documents above threshold (commonly 0.8) get flagged as near-duplicates, and all but one copy gets removed.

In recipe example, Docs A and B land in same LSH bucket. Their Jaccard similarity is high because most n-grams overlap, so score exceeds 0.8. Keep one, discard other. Doc C has different n-grams, so it stays.

At production scale, LLaMA team used exact deduplication at line level (removing identical lines across corpus) plus fuzzy deduplication with MinHash at document level. This alone cut data by roughly one third.

Quality Filtering

After deduplication, lot of low-quality text still remains. Goal: keep Wikipedia articles and cooking instructions, remove SEO spam and cookie consent banners.

Three approaches. Often combined:

Perplexity-based filtering trains small, cheap language model (often KenLM n-gram model) on reference corpus of high-quality text such as Wikipedia. Then each crawl document gets scored: how surprised is quality model by this text? Well-written article gets low perplexity. Garbled text or keyword spam gets high perplexity. Documents above threshold get removed. This is core of CCNet pipeline, developed by Facebook for processing Common Crawl.

Heuristic rules catch patterns statistical models miss. Remove documents shorter than 200 characters. Remove documents where more than 30% of characters are special symbols. Remove documents with word-to-symbol ratio below threshold. Remove documents with more than 3 consecutive repeated lines. RefinedWeb paper documents dozens of such rules, each found by manually inspecting failure cases.

Classifier-based filtering trains fast classifier (often fastText) on small set of human-rated examples. “High quality or no?” Then it scores entire crawl. More expensive than heuristics. Catches subtler quality problems.

Recipe-model version: read 10,000 scraped sentences, throw out navigation menus and comment spam, keep actual recipes. At web scale, manual reading fails. Build automated versions of yourself.

Key lesson: data quality is scaling law. Bad data does not get diluted by more compute. It shifts whole loss curve upward. Well-curated 1T token dataset can beat poorly filtered 5T token dataset. Microsoft’s Phi models showed this hard — small models trained on “textbook quality” data rivaled models 10× larger trained on noisier corpora.

Building a Vocabulary from Scratch

We have clean text. Before training model, need to convert words into numbers. But which numbers? And what counts as “word”?

Recipe corpus has words like “preheat,” “350°F,” “tablespoon,” and “sauté.” If each unique word becomes token, vocabulary maybe 5,000 entries. Fine for tiny model. Real English has hundreds of thousands of words, plus every language we want to support, plus code, plus mathematical notation. Word-level vocabulary becomes enormous and still fails on unseen words.

Solution is subword tokenization. Dominant algorithm is Byte Pair Encoding (BPE). Core idea: start with individual characters (or bytes), then repeatedly merge most frequent adjacent token pair into new token. After enough merges, common words become single tokens while rare words split into familiar pieces.

Walk through BPE on tiny vocabulary. Suppose our entire corpus is:

Corpus: "heat heat heat preheat preheat"

Start with character-level tokens:
  h e a t   h e a t   h e a t   p r e h e a t   p r e h e a t

Frequency of adjacent pairs:
  (h, e): 5    (e, a): 5    (a, t): 5    (p, r): 2    (r, e): 2

Merge #1: (h, e) → "he"  (most frequent pair, tied — pick one)
  he a t   he a t   he a t   p r e he a t   p r e he a t

Merge #2: (a, t) → "at"
  he at   he at   he at   p r e he at   p r e he at

Merge #3: (he, at) → "heat"
  heat   heat   heat   p r e heat   p r e heat

Merge #4: (e, heat) → "eheat"  ... and so on

After enough merges, “heat” becomes single token and “preheat” becomes two tokens: “pre” + “heat.” Unseen word like “reheat,” splits into “re” + “heat” — meaningful pieces model knows.

Vocabulary size — number of merges — is design choice with real tradeoffs. LLaMA-2 uses 32,000 tokens (via SentencePiece). GPT-4 uses about 100,000 tokens (via tiktoken). Larger vocabulary means fewer tokens per word, saving sequence length and attention compute, but it also means larger embedding matrix and larger output layer.

At LLM scale, tokenizer usually trains on representative sample of training data, not full corpus. Tokenizer should match text distribution model will see. Train tokenizer on English Wikipedia and model struggles with Python code. Train it on balanced mix, it handles both.

Easy mistake: tokenizer trains separately from model, before model training starts. Once set, it stays frozen. Vocabulary cannot change mid-training. Tokenizer design choices made months earlier have permanent consequences. Mismatch between tokenizer and training data is silent bug that hurts quality without throwing error.

The Pretraining Objective: What Does the Model Actually Learn?

We have clean data and tokenizer. Now core question: what task does model get during training? No labels. No human annotations. Model must learn from text itself. Trick is designing self-supervised objective — task data gives for free.

Back to recipe corpus. Sentence is: “Preheat the oven to 350°F and grease the pan.” What task can model do with this, without human labeling?

Causal Language Modeling: Guess What Comes Next

Most powerful idea in modern AI may also sound most underwhelming: predict the next token. That’s it. Given everything model has read so far, what token comes next?

For recipe sentence, model sees “Preheat” and predicts “the.” Then sees “Preheat the” and predicts “oven.” Then “Preheat the oven” → “to.” Keep sliding one token at time. At every position, model outputs probability distribution over whole vocabulary — all 32,000 tokens — and gets penalized for giving low probability to actual next token.

Input tokens:     Preheat  the   oven   to   350   °F   and
                    ↓       ↓      ↓     ↓     ↓    ↓     ↓
Model predicts:   the     oven    to    350   °F   and  grease
                    ↕       ↕      ↕     ↕     ↕    ↕     ↕
Actual next:      the     oven    to    350   °F   and  grease  ✓ ✓ ✓ ✓ ✓ ✓ ✓

Loss = cross-entropy at EVERY position, averaged across the sequence

This is called causal language modeling (CLM) because each prediction is causal: position 5 can attend to positions 1 through 4 but never to position 6 or beyond. Model reads left to right, like us. Causal attention mask enforces this — triangular matrix blocking each position from seeing future:

Causal Mask (5 tokens):

         Preheat  the  oven  to  350
Preheat  [  1      0    0    0    0  ]   sees only itself
the      [  1      1    0    0    0  ]   sees Preheat, the
oven     [  1      1    1    0    0  ]   sees Preheat, the, oven
to       [  1      1    1    1    0  ]   sees positions 1-4
350      [  1      1    1    1    1  ]   sees everything so far

1 = can attend, 0 = masked (set to -∞ before softmax)

Implementation has critical detail called shift trick. Model’s output at position t is prediction for position t+1. So loss compares logits at positions 1 through N−1 against actual tokens at positions 2 through N:

logits = model(input_ids)              # shape: (batch, seq_len, vocab_size)

shift_logits = logits[:, :-1, :]       # predictions for positions 2..N
shift_labels = input_ids[:, 1:]        # actual tokens at positions 2..N

loss = cross_entropy(shift_logits, shift_labels)

Off-by-one errors in this shift are common training-code bugs. Even experienced engineers lose days to them.

During training, use teacher forcing: model always receives actual ground-truth tokens as input, not its own predictions. At inference time, no ground truth exists, so model feeds back its own outputs and generates autoregressively, one token at time.

Why does this tiny objective produce such capable models? Because language encodes everything. To predict next word in Wikipedia article about quantum mechanics, model must understand physics. To predict next line of Python function, it must understand programming logic. To predict next sentence in novel, it must understand human psychology. Objective is trivial. Knowledge needed to minimize it is not. That is deep insight behind GPT and every decoder-only model since.

CLM powers GPT, LLaMA, Mistral, Claude, effectively every chatbot you’ve used. Industry converged on it because generation is killer app, and decoder-only models with CLM scale cleanly: one architecture, one objective, one pipeline.

Three Flavors of Prediction

CLM is not only pretraining objective. Two important alternatives exist. Understanding all three shows why field converged on CLM — and what got sacrificed.

Masked Language Modeling: Fill in the Blanks

BERT (2018) took different path. Instead of reading left to right and predicting next word, it randomly hides some tokens and asks model to fill them back in — using context from both sides.

Recipe sentence becomes:

Original: "Preheat the oven to 350°F and grease the pan"
Masked:   "Preheat the [MASK] to 350°F and [MASK] the pan"

The model must predict:
  [MASK] → "oven"   (using left context "Preheat the" AND right context "to 350°F...")
  [MASK] → "grease" (using "...and" AND "the pan")

Superpower here is bidirectional context. When predicting “oven,” model sees both “Preheat the” and “to 350°F.” CLM sees only left side. This makes masked language modeling (MLM) produce richer representations for understanding tasks — classification, named entity recognition, question answering.

BERT masks 15% of tokens using 80/10/10 protocol: 80% become special [MASK] token, 10% become random vocabulary word, 10% stay unchanged but still get predicted. Random replacements and unchanged tokens stop model from learning shortcut that activates only when it sees [MASK] — token that never appears during fine-tuning or real use.

Tradeoff is severe: MLM models cannot generate text. No left-to-right factorization. No autoregressive sampling. Also sample-efficiency problem: only 15% of tokens contribute to loss each step. Other 85% provide context but generate zero gradient signal. Big compute waste.

Prefix Language Modeling and Span Corruption

Between CLM and MLM sits family of hybrid objectives. Prefix language modeling, used in models like UniLM, lets model attend bidirectionally inside “prefix” section, then autoregressively predict what follows. Think of it as reading whole question bidirectionally, then generating answer left-to-right.

Span corruption, used by T5, generalizes MLM by masking contiguous token spans instead of individual words. Each span is replaced with single sentinel token, decoder reconstructs missing spans:

Original:  "Preheat the oven to 350°F and grease the pan"
Corrupted: "Preheat <X0> 350°F and <X1> pan"
Target:    "<X0> the oven to <X1> grease the"

The encoder sees the corrupted input (shorter — saves compute).
The decoder generates only the missing pieces (compact targets).

T5 cast every NLP task as text-to-text: same architecture, same loss, every task. Pretraining used span corruption. Fine-tuning looked like “translate English to German: The house is blue” → “Das Haus ist blau.” Elegant and practical. But encoder-decoder architecture is harder to scale than pure decoder.

Mapping worth internalizing:

Architecture Objective Attention Strengths Examples
Encoder-only MLM Bidirectional Classification, retrieval, embeddings BERT, RoBERTa, ELECTRA
Decoder-only CLM Causal (left-to-right) Generation, chat, code, reasoning GPT, LLaMA, Mistral
Encoder-Decoder Span corruption / prefix LM Bidirectional encoder + causal decoder Translation, summarization T5, BART, UL2

Industry converged on decoder-only CLM not because it is theoretically best for every task, but because it is simplest to scale and generation became killer capability. One architecture. One objective. One training pipeline. Encoder-only models still rule embeddings and classification. Encoder-decoder models stay strong for structured sequence-to-sequence problems. But once model is big enough, decoder-only does everything well enough, and “well enough at everything” beat “best at one thing.”

This still is not full story. It does not answer “how big should the model be?” or “how many GPUs do you need?” or “what happens when the training run blows up at 3 AM?” Those questions separate someone who understands pretraining from someone who can plan and execute one.

Short version: mathematical laws tell optimal model size for budget, training needs hundreds of GPUs tied together by specialized networking, training runs are fragile beasts needing careful babysitting. There. You’re 60% there.

If missing details still bother you, read on.

How Big Should the Model Be? Scaling Laws

For years, training LLMs was expensive gambling. Pick model size. Pick dataset size. Spend millions. Hope it works. Then in 2020, OpenAI team found something that turned gambling into engineering.

Kaplan and colleagues trained hundreds of transformer models ranging from few hundred parameters to 1.5 billion parameters. When they plotted loss against model size on log-log chart, they got straight line. Same for loss against dataset size. Same for loss against total compute.

Three clean power-law relationships:

Loss vs. Parameters:  L(N) ∝ N^(-0.076)
Loss vs. Data:        L(D) ∝ D^(-0.095)
Loss vs. Compute:     L(C) ∝ C^(-0.050)

        Loss (log scale)
         │
         │╲
         │  ╲
         │    ╲  ← straight line on log-log
         │      ╲     (this is a power law)
         │        ╲
         │──────────── Parameters or Tokens (log scale)

Profound implication: every 10× increase in parameters gives fixed reduction in loss. Returns never stop. They get more expensive, but they do not stop. No ceiling found yet.

Kaplan’s key conclusion: with fixed compute budget, prioritize model size over data. Make model bigger instead of training on more data. Recommendation: if compute grows 10×, grow parameters 5× but data only 2×.

This drove “make it bigger” era. GPT-3 had 175 billion parameters trained on 300 billion tokens. Massive model. Modest data. Thinking was simple: parameters matter most.

That turned out wrong.

The Chinchilla Correction

In March 2022, DeepMind published paper that reshuffled field overnight. Title was dry: “Training Compute-Optimal Large Language Models.” Finding was brutal: almost every existing large model was undertrained.

Hoffmann’s team ran proper sweep: over 400 models from 70 million to 16 billion parameters, trained on varying data amounts, all with fixed compute budgets. Picture differed from Kaplan’s.

Core finding: for compute-optimal training, scale parameters and data equally. Optimal ratio is roughly 20 tokens per parameter:

Chinchilla Rule:   D ≈ 20 × N

  N = number of model parameters
  D = number of training tokens

Given compute C ≈ 6 × N × D:
  Optimal N ∝ √C
  Optimal D ∝ √C     (both scale equally with compute)

To prove it, team ran strongest experiment possible. DeepMind had recently trained Gopher: 280B parameters, 300B tokens. Classic Kaplan era — huge model, modest data. Then team trained Chinchilla: 70B parameters (4× smaller) on 1.4 trillion tokens (4.7× more data). Same compute budget. Result: Chinchilla won on nearly every benchmark. Model 4× smaller, trained on more data, was simply better. And because it was smaller, deployment was cheaper.

I still look up exact Chinchilla ratio whenever I do back-of-envelope calculation. Here is lookup table I keep using:

Parameters Optimal Tokens (≈20N) Approx FLOPs
400M8B2 × 10¹&sup9;
1B20B1.2 × 10²&sup0;
7B140B5.9 × 10²¹
13B260B2 × 10²²
70B1.4T5.9 × 10²³

GPT-3 (175B parameters, 300B tokens) trained on roughly 12× fewer tokens than Chinchilla would recommend. Massively undertrained by modern standards.

Quick Python function for budget planning:

import math

def chinchilla_optimal(flops_budget):
    """Given a FLOPs budget, estimate compute-optimal model size and tokens."""
    # C ≈ 6 * N * D, with D ≈ 20 * N
    # So C ≈ 120 * N², therefore N = sqrt(C / 120)
    N = math.sqrt(flops_budget / 120)
    D = 20 * N
    return N, D

# Example: 10^22 FLOPs
n, d = chinchilla_optimal(1e22)
print(f"≈{n/1e9:.1f}B params, ≈{d/1e9:.0f}B tokens")
# → ≈9.1B params, ≈182B tokens

Compute-Optimal Training and the Data Wall

Chinchilla changed how labs allocate compute. It also exposed looming problem: training data may run out.

A 1-trillion-parameter model, by Chinchilla rule, needs 20 trillion tokens. Entire publicly available internet — Common Crawl, books, code, Wikipedia, everything — is estimated at 5 to 15 trillion tokens of reasonable quality. We are already scraping edges.

This “data wall” drove several responses. Meta’s LLaMA deliberately over-trains: LLaMA-7B trained on 1 trillion tokens, roughly 7× Chinchilla-optimal amount. Reason is pragmatic. Chinchilla optimizes training compute, but inference cost depends on model size, not training duration. Smaller model trained longer is cheaper to deploy. When serving millions of users, few extra training weeks pay back many times in cheaper inference.

Synthetic data — models generating training data for other models — is another escape route. It works surprisingly well for code and mathematics, where generated outputs can be verified for correctness. But risk is model collapse: distributional biases compound over generations, like photocopying photocopy until only darkest marks survive.

Multi-epoch training (reusing data multiple times) works with proper regularization, but returns diminish after 4–8 passes. Multimodal data — images, video, audio — is partly escape plan for text data wall, tapping huge information content in visual and auditory signals.

No one knows right path with certainty. Questions here stay wide open. That is part of what makes this corner of field exciting.

The Physical Machine: Clusters and Interconnects

Put theory on hardware. Training 7B-parameter model requires multiple GPUs working in lockstep. Training 70B model requires hundreds. Physical infrastructure matters as much as algorithms.

Modern training cluster is built from nodes, each with 8 high-end GPUs (typically NVIDIA H100s). Inside node, GPUs communicate through NVLink, proprietary interconnect providing up to 900 GB/s bidirectional bandwidth per GPU. Fast enough that GPUs inside node can share data almost like own memory.

Between nodes, interconnect is InfiniBand, providing 400 Gbps with latencies around 1–2 microseconds. Fast by networking standards, but roughly 18× slower than NVLink. This asymmetry shapes distributed training strategy.

Intuition: if NVLink were fire hose, InfiniBand would be garden hose. Both move water. Planning changes a lot depending on which hose sits between two GPUs.

Distributed training uses three kinds of parallelism, layered by bandwidth need:

Tensor Parallelism (TP) splits individual weight matrices across GPUs inside node. Each GPU computes slice of attention or feedforward layer, then combines results. This needs constant GPU-to-GPU communication, so it uses fastest link: NVLink inside node.

Pipeline Parallelism (PP) assigns different transformer layers to different node groups. GPU group 0 runs layers 1–8, group 1 runs layers 9–16, and so on. Data flows through groups like assembly line. Downside is “pipeline bubbles” — GPUs sit idle waiting for activations from earlier stages. This tolerates moderate latency, so it spans few nodes connected by InfiniBand.

Data Parallelism (DP / FSDP) replicates training across many groups, each processing different batches. Gradients are averaged across groups after each step. This needs least frequent communication, so it spans widest network distances.

Example: LLaMA-70B on 256 GPUs

  TP = 8  (within each node, NVLink — splits weight matrices)
  × PP = 4  (across 4 node groups — splits layers)
  × DP = 8  (across 8 pipeline replicas — splits data)
  = 8 × 4 × 8 = 256 GPUs

  Think of it as 8 identical assembly lines (DP),
  each with 4 stages (PP),
  each stage using 8 GPUs in lockstep (TP).

Fire-hose/garden-hose analogy comes back here. TP needs fire hose (NVLink, inside node). PP can work with garden hose (InfiniBand, across few nodes). DP can work with trickle (InfiniBand across wider network) because it syncs gradients only periodically.

Key memory optimization making this practical is FSDP (Fully Sharded Data Parallel), PyTorch’s implementation of ZeRO-3 from DeepSpeed. Standard data parallelism makes every GPU hold complete copy of model, optimizer states, gradients — massive redundancy. FSDP shards everything: each GPU stores only 1/N of parameters, optimizer states, gradients. During forward pass, parameters are gathered from other GPUs right before use, used, then discarded. This can cut per-GPU memory from over 100 GB to under 15 GB for 7B model across 8 GPUs.

When Training Goes Wrong: Loss Spikes and Stability

First time I saw loss spike in training run, I thought run was dead. Loss had fallen smoothly for days, then — sudden upward jump, like heart rate monitor in horror movie. Twenty minutes of panic. Then loss recovered like nothing happened.

Loss spikes are most common instability in LLM training. Loss declines peacefully, then shoots upward before usually recovering over next few hundred steps. Causes include:

Bad data batch — batch with highly unusual content (extremely long documents, garbled Unicode, code with pathological nesting) can produce extreme gradient values that shove model into bad region of parameter space. Fix: better data filtering before training, or detect and skip batches with anomalous loss during training.

Learning rate too high — optimizer takes step that is too large and overshoots into steep valley. Fix: reduce peak learning rate, use longer warmup, or both.

Numerical instability — FP16 overflow, division by very small numbers in layer normalization, or softmax overflow. Fix: use BF16 (same dynamic range as FP32, so overflow becomes extremely rare), add small epsilon values to denominators, use numerically stable softmax implementations.

Google’s PaLM team documented practical recovery: when loss spike happens, roll back to checkpoint from roughly 100 steps before spike started, skip data batches that triggered it. Crude, but effective. Some teams also use checkpoint averaging — averaging weights from last 5–20 checkpoints to smooth catastrophic updates and return model to stable region.

Universal safety net is gradient clipping: before each optimizer step, compute global norm of all gradients and, if it exceeds threshold (typically 1.0), scale all gradients down proportionally. This preserves direction of update while capping magnitude. Same direction, smaller step. Global norm clipping, not per-parameter clipping, matters because per-parameter clipping distorts gradient direction.

Other stability techniques proven essential at scale:

BF16 mixed precision — store most tensors in 16-bit bfloat16 (trading mantissa precision for FP32-level dynamic range), but keep FP32 copy of weights for optimizer update. Optimizer accumulates many small gradient updates. Doing this in BF16 would create rounding errors that compound over thousands of steps. That precision mix is why name is “mixed precision.”

QK-Norm — apply layer normalization to queries and keys before computing attention scores. Without this, attention logits can grow without bound as training progresses, eventually causing instability. Used in PaLM and LLaMA-2.

z-loss — small penalty on log-partition function of output softmax, preventing output logits from growing too large. Used in PaLM training.

Learning rate warmup — start with near-zero learning rate and linearly increase to peak over first 1–5% of training. At initialization, model’s gradients are wildly unpredictable. Full-strength learning rate here can launch model into bad region it never escapes. Warmup lets optimizer cautiously explore landscape before taking big steps. After warmup, learning rate follows cosine decay down to roughly 10% of peak.

  Learning Rate

Feeding the Beast: Curriculum and Data Mixing

So far, training data looked like one homogeneous token stream. In practice, it is careful blend of many sources, and source proportions strongly affect resulting model.

Back to recipe model. Suppose we have three sources: recipe instructions (60%), food science articles (30%), restaurant reviews (10%). Oversample restaurant reviews and model gets good at “the ambiance was delightful” but worse at “preheat the oven to 350.” Mix matters.

Real LLMs face this at enormous scale:

Source Typical Share What It Teaches
Filtered web crawl60–70%Breadth: conversational text, general knowledge
Code (GitHub)5–15%Structured reasoning, logic, programming
Books5–10%Long-form coherence, deep knowledge
Wikipedia3–5%Factual knowledge, structured writing
Academic papers3–5%Scientific reasoning, technical language
Conversational data2–5%Dialog patterns, Q&A structure

Growing evidence says code data improves reasoning even on non-code tasks. Code is highly structured, logically precise, demands sequential thinking. Models with significant code exposure (like GPT-3.5 and GPT-4, descendants of Codex) consistently beat text-only models on reasoning benchmarks. Field did not expect this.

Getting data mix right is part art, part science. DoReMi, method from Google, tries to learn optimal mixing proportions automatically. It frames problem as minimax optimization: data mixture and small proxy model are co-trained, adjusting proportions to maximize downstream performance. Result often differs from hand-tuned proportions in hard-to-predict ways — for example, DoReMi might upweight small, high-quality source humans would undersample.

Curriculum learning goes further by changing data order during training. One approach: start with simpler, cleaner text (Wikipedia, well-edited books) and gradually introduce noisier, more complex data (web crawl, code). Intuition: early training builds foundational representations, and noisy data too early is like learning calculus before arithmetic. In practice, benefits for LLMs are mixed — some teams report improvements, others find random shuffling works equally well. Intuition here is still developing.

Honest truth about data mixing: no one knows optimal proportions with certainty. Labs guard data recipes closely, few ablation studies that exist show small mix changes can create surprisingly large downstream behavior changes. Practical knowledge is ahead of theory here.
