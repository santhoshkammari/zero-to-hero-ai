Data Visualization

Chapter 1: Python & Programming Foundations 4 subtopics
TL;DR

Matplotlib gives you pixel-level control over every plot through its Figure/Axes object model — learn this instead of the pyplot shortcut and you'll never get confused by subplots again. Seaborn sits on top of matplotlib and makes statistical plots (heatmaps, pairplots, distributions) a one-liner. For dashboards and presentations, Plotly and Bokeh add interactivity. And for ML-specific work, you'll live in confusion matrices, ROC curves, and learning curves — the diagnostic plots that tell you what's actually wrong with your model.

You've cleaned your data, engineered your features, and trained a model. Now someone asks: "How does the model perform?" If your answer is a wall of numbers, you've already lost the room. A single well-chosen plot communicates in two seconds what a table of metrics takes two minutes to parse. Visualization isn't decoration — it's your primary debugging tool before modeling and your primary communication tool after. You'll plot feature distributions to catch skew, correlation heatmaps to spot redundant features, and learning curves to diagnose whether your model needs more data or more capacity. Let's build those skills from the ground up.

Matplotlib

Matplotlib is the grandfather of Python plotting. Nearly every other visualization library — Seaborn, Pandas plotting, even parts of scikit-learn's display utilities — builds on top of it. You'll see two styles of matplotlib code in the wild: the pyplot state-machine style (plt.plot(), plt.title()) and the object-oriented style using Figure and Axes objects. We're going to focus on the object-oriented approach because it's the one that actually scales. The pyplot style works fine for a quick one-off plot, but the moment you need subplots, shared axes, or any kind of layout control, the state machine turns into a guessing game about which "current axes" you're modifying.

Figure and Axes Objects

Here's the mental model that clears up all the confusion. A Figure is the entire window or image — think of it as the canvas. An Axes is a single plot living inside that Figure. One Figure can hold many Axes objects (that's what subplots are). When you call plt.subplots(), you get back both: the Figure and one or more Axes. From that point on, you call methods on the Axes object — ax.plot(), ax.set_title(), ax.set_xlabel() — and you always know exactly which plot you're talking to.

import matplotlib.pyplot as plt
import numpy as np

# Create a Figure with one Axes
fig, ax = plt.subplots(figsize=(8, 5))

# Everything goes through the ax object — no ambiguity
x = np.linspace(0, 10, 100)
ax.plot(x, np.sin(x))
ax.set_title("A Simple Sine Wave")
ax.set_xlabel("x")
ax.set_ylabel("sin(x)")

plt.tight_layout()
plt.savefig("sine_wave.png", dpi=150)
plt.close()

The key insight: fig controls figure-level things (size, saving, overall title), while ax controls what's actually drawn. Once this clicks, every matplotlib task becomes "get the right Axes object and call methods on it."

Line Plots

Line plots are your go-to for anything that changes over a sequence — training loss over epochs, stock prices over time, sensor readings over samples. The ax.plot() method connects your data points with lines.

fig, ax = plt.subplots(figsize=(8, 5))

epochs = range(1, 51)
train_loss = 2.5 * np.exp(-0.08 * np.array(epochs)) + 0.1 + np.random.normal(0, 0.03, 50)
val_loss = 2.5 * np.exp(-0.06 * np.array(epochs)) + 0.2 + np.random.normal(0, 0.05, 50)

ax.plot(epochs, train_loss, label="Train Loss", color="#2196F3", linewidth=2)
ax.plot(epochs, val_loss, label="Val Loss", color="#FF5722", linewidth=2, linestyle="--")

ax.set_xlabel("Epoch")
ax.set_ylabel("Loss")
ax.set_title("Training vs Validation Loss")
ax.legend()
ax.grid(True, alpha=0.3)

plt.tight_layout()
plt.savefig("loss_curves.png", dpi=150)
plt.close()

Notice we passed label to each plot call and then called ax.legend() — matplotlib builds the legend automatically from those labels. The linestyle="--" makes the validation line dashed, which is a common convention so readers can distinguish the two even in black-and-white prints.

Scatter Plots

Scatter plots reveal relationships between two continuous variables. In ML, you'll use them constantly — plotting predicted vs. actual values, visualizing cluster assignments, or showing 2D embeddings of high-dimensional data.

fig, ax = plt.subplots(figsize=(7, 6))

# Simulate two clusters (like a classification dataset)
np.random.seed(42)
class_0 = np.random.randn(100, 2) + np.array([2, 2])
class_1 = np.random.randn(100, 2) + np.array([-1, -1])

ax.scatter(class_0[:, 0], class_0[:, 1], label="Class 0", alpha=0.6, s=40, c="#4CAF50")
ax.scatter(class_1[:, 0], class_1[:, 1], label="Class 1", alpha=0.6, s=40, c="#E91E63")

ax.set_xlabel("Feature 1")
ax.set_ylabel("Feature 2")
ax.set_title("Two-Class Scatter Plot")
ax.legend()
plt.tight_layout()
plt.savefig("scatter_classes.png", dpi=150)
plt.close()

The alpha=0.6 parameter adds transparency so you can see where points overlap. The s=40 controls marker size. These small tweaks make the difference between a plot that communicates and one that's a blob of ink.

Bar Charts

Bar charts compare discrete categories. You'll reach for these when showing feature importances, per-class accuracy, or comparing model performance across experiments.

fig, ax = plt.subplots(figsize=(8, 5))

models = ["Logistic Reg", "Random Forest", "XGBoost", "Neural Net"]
accuracies = [0.82, 0.89, 0.91, 0.88]
colors = ["#42A5F5", "#66BB6A", "#FFA726", "#AB47BC"]

bars = ax.bar(models, accuracies, color=colors, edgecolor="white", linewidth=1.5)

# Add value labels on top of each bar
for bar, acc in zip(bars, accuracies):
    ax.text(bar.get_x() + bar.get_width() / 2, bar.get_height() + 0.005,
            f"{acc:.2f}", ha="center", va="bottom", fontweight="bold")

ax.set_ylabel("Accuracy")
ax.set_title("Model Comparison on Test Set")
ax.set_ylim(0.75, 0.95)
plt.tight_layout()
plt.savefig("model_comparison.png", dpi=150)
plt.close()

Histograms

Before you train any model, you need to understand your feature distributions. Are they normal? Skewed? Bimodal? Histograms answer these questions at a glance. A histogram bins your continuous data into intervals and counts how many values fall into each bin.

fig, ax = plt.subplots(figsize=(8, 5))

# Simulate a skewed feature (like income or house prices)
np.random.seed(42)
feature = np.random.lognormal(mean=3, sigma=0.8, size=2000)

ax.hist(feature, bins=50, color="#26A69A", edgecolor="white", alpha=0.8)
ax.axvline(np.median(feature), color="#D32F2F", linestyle="--", label=f"Median: {np.median(feature):.1f}")
ax.axvline(np.mean(feature), color="#1565C0", linestyle="--", label=f"Mean: {np.mean(feature):.1f}")

ax.set_xlabel("Feature Value")
ax.set_ylabel("Count")
ax.set_title("Distribution of a Skewed Feature")
ax.legend()
plt.tight_layout()
plt.savefig("histogram.png", dpi=150)
plt.close()

When the mean is far from the median, you've got skew — and that tells you something actionable. You might apply a log transform before feeding this feature into a model that assumes roughly normal inputs (like linear regression or SVMs). One plot saved you from a silent modeling mistake.

Subplots

Real analysis rarely involves a single plot. You want to see multiple features side by side, or compare distributions across classes. That's where subplots shine. The plt.subplots(nrows, ncols) call creates a grid of Axes objects.

fig, axes = plt.subplots(2, 2, figsize=(12, 10))

np.random.seed(42)
features = {
    "Age": np.random.normal(35, 10, 500),
    "Income": np.random.lognormal(10, 0.8, 500),
    "Score": np.random.beta(2, 5, 500) * 100,
    "Tenure": np.random.exponential(5, 500),
}

for ax, (name, data) in zip(axes.flat, features.items()):
    ax.hist(data, bins=30, color="#5C6BC0", edgecolor="white", alpha=0.8)
    ax.set_title(f"Distribution of {name}")
    ax.set_ylabel("Count")

plt.suptitle("Feature Distributions Overview", fontsize=14, fontweight="bold")
plt.tight_layout()
plt.savefig("subplots_grid.png", dpi=150)
plt.close()

The trick here is axes.flat, which turns the 2D array of Axes into a flat iterator. You can loop over it alongside your data using zip. This pattern — creating a grid and iterating — is one you'll use hundreds of times.

Customizing Plots (Labels, Titles, Legends, Colors)

A plot without labels is a plot that tells no story. Always label your axes with units where applicable, give your plot a descriptive title, and use legends whenever you have multiple series. For colors, matplotlib accepts hex codes ("#FF5722"), named colors ("steelblue"), and colormaps ("viridis", "plasma"). Stick with colorblind-friendly palettes when publishing — "viridis" was designed for exactly this purpose.

fig, ax = plt.subplots(figsize=(8, 5))

x = np.linspace(0, 10, 100)
ax.plot(x, np.sin(x), color="#1E88E5", linewidth=2, label="sin(x)")
ax.plot(x, np.cos(x), color="#E53935", linewidth=2, linestyle="--", label="cos(x)")

ax.set_xlabel("Radians", fontsize=12)
ax.set_ylabel("Amplitude", fontsize=12)
ax.set_title("Trigonometric Functions", fontsize=14, fontweight="bold")
ax.legend(loc="upper right", fontsize=11, framealpha=0.9)
ax.grid(True, linestyle=":", alpha=0.5)
ax.set_xlim(0, 10)
ax.set_ylim(-1.5, 1.5)

plt.tight_layout()
plt.savefig("customized_plot.png", dpi=150)
plt.close()

Saving Figures

Use fig.savefig() or plt.savefig() to export. The dpi parameter controls resolution — 150 is fine for screens, 300 for print. Use bbox_inches="tight" to trim whitespace. Supported formats include PNG, PDF, SVG, and EPS. For papers, SVG or PDF gives you vector graphics that scale to any size without pixelation.

# Save in multiple formats
fig.savefig("plot.png", dpi=300, bbox_inches="tight")
fig.savefig("plot.pdf", bbox_inches="tight")         # Vector format for papers
fig.savefig("plot.svg", bbox_inches="tight")         # Vector format for web

# Always close after saving to free memory
plt.close(fig)
⚠️ Memory Leak Alert

If you create plots in a loop (say, generating one per experiment) and forget to call plt.close(), matplotlib keeps every Figure in memory. In a long training run, this will eat your RAM. Always close figures you're done with.

Seaborn

Seaborn is what you reach for when you want beautiful statistical plots without wrestling with matplotlib's low-level API. It's built on top of matplotlib, so everything you learned above still applies — a Seaborn plot returns matplotlib Axes objects you can customize further. But Seaborn's real power is that it understands DataFrames natively. Pass a DataFrame and column names, and it handles binning, aggregation, error bars, and color coding for you.

Statistical Plots Overview

Seaborn organizes its plotting functions into three families: relational (how do two variables relate?), distributional (what does a variable look like?), and categorical (how does a variable differ across groups?). Each family has a figure-level function for quick multi-panel figures and axes-level functions for fine control. We'll walk through the most important ones.

import seaborn as sns
import pandas as pd
import numpy as np

# Create a realistic-looking dataset
np.random.seed(42)
n = 300
df = pd.DataFrame({
    "age": np.random.normal(40, 12, n).clip(18, 80).astype(int),
    "income": np.random.lognormal(10.5, 0.6, n).astype(int),
    "spending_score": np.random.normal(50, 20, n).clip(1, 100).astype(int),
    "segment": np.random.choice(["Budget", "Mid-Range", "Premium"], n, p=[0.4, 0.35, 0.25]),
})

Distribution Plots (histplot, kdeplot, boxplot, violinplot)

Before modeling, your first question should always be: "What do my features look like?" Distribution plots answer that. sns.histplot() gives you a histogram, optionally with a KDE (kernel density estimate) curve — a smooth approximation of the distribution.

fig, axes = plt.subplots(2, 2, figsize=(12, 10))

# Histogram with KDE
sns.histplot(data=df, x="income", kde=True, ax=axes[0, 0], color="#5C6BC0")
axes[0, 0].set_title("Income Distribution")

# Pure KDE comparison across groups
sns.kdeplot(data=df, x="spending_score", hue="segment", ax=axes[0, 1], fill=True, alpha=0.4)
axes[0, 1].set_title("Spending Score by Segment")

# Boxplot — shows median, quartiles, and outliers
sns.boxplot(data=df, x="segment", y="income", ax=axes[1, 0], palette="Set2")
axes[1, 0].set_title("Income by Segment (Boxplot)")

# Violinplot — like a boxplot, but shows the full distribution shape
sns.violinplot(data=df, x="segment", y="spending_score", ax=axes[1, 1], palette="Set2")
axes[1, 1].set_title("Spending Score by Segment (Violin)")

plt.tight_layout()
plt.savefig("seaborn_distributions.png", dpi=150)
plt.close()

A boxplot shows you the median (center line), interquartile range (the box), and outliers (the dots beyond the whiskers). It's a quick way to spot if one group has extreme values. A violinplot is a boxplot's more informative cousin — it shows the full distribution shape, so you can see if data is bimodal or skewed, which a boxplot hides.

Categorical Plots (countplot, barplot, stripplot)

When one of your variables is categorical (like customer segment or product type), Seaborn's categorical plots make comparison effortless.

fig, axes = plt.subplots(1, 3, figsize=(16, 5))

# Count of samples per category
sns.countplot(data=df, x="segment", ax=axes[0], palette="pastel", order=["Budget", "Mid-Range", "Premium"])
axes[0].set_title("Samples per Segment")

# Mean income per segment (with confidence intervals)
sns.barplot(data=df, x="segment", y="income", ax=axes[1], palette="muted", ci=95)
axes[1].set_title("Mean Income by Segment")

# Individual points — see the actual data, not just a summary
sns.stripplot(data=df, x="segment", y="spending_score", ax=axes[2], alpha=0.4, jitter=True, palette="deep")
axes[2].set_title("Spending Scores (Individual Points)")

plt.tight_layout()
plt.savefig("seaborn_categorical.png", dpi=150)
plt.close()

The sns.barplot() here doesn't plot raw values — it computes the mean of income for each segment and adds a confidence interval. That black line on each bar? That's a 95% CI by default. A stripplot shows every individual data point, which is crucial for small datasets where a mean can be misleading.

Relational Plots (scatterplot, lineplot)

Relational plots answer: "How do two numeric variables relate, and does that relationship change across groups?"

fig, axes = plt.subplots(1, 2, figsize=(14, 5))

sns.scatterplot(data=df, x="age", y="income", hue="segment", ax=axes[0], alpha=0.6, palette="deep")
axes[0].set_title("Age vs Income by Segment")

# Lineplot with aggregation (useful for time-series-like data)
df_sorted = df.sort_values("age")
sns.lineplot(data=df_sorted, x="age", y="spending_score", hue="segment", ax=axes[1], ci=68)
axes[1].set_title("Spending Score Trend by Age")

plt.tight_layout()
plt.savefig("seaborn_relational.png", dpi=150)
plt.close()

Notice how the hue parameter does all the grouping and coloring automatically. Pass a column name and Seaborn splits the data, assigns colors, and builds a legend. This is the biggest productivity gain over raw matplotlib.

Heatmaps and Correlation Matrices

Heatmaps are one of the most important plots in ML feature engineering. You compute the correlation between every pair of numeric features and display it as a color grid. Highly correlated features are redundant — they add noise without information. This one plot can save you hours of debugging multicollinearity issues.

fig, ax = plt.subplots(figsize=(8, 6))

# Compute pairwise correlations
numeric_cols = df.select_dtypes(include=[np.number])
corr_matrix = numeric_cols.corr()

# Plot heatmap with annotations
sns.heatmap(corr_matrix, annot=True, fmt=".2f", cmap="coolwarm",
            center=0, vmin=-1, vmax=1, square=True,
            linewidths=0.5, ax=ax)

ax.set_title("Feature Correlation Matrix")
plt.tight_layout()
plt.savefig("correlation_heatmap.png", dpi=150)
plt.close()

The center=0 ensures zero correlation is white, positive is red, and negative is blue. The annot=True prints the actual correlation values inside each cell. If you see two features with a correlation above 0.9, consider dropping one — your model likely won't benefit from both.

Pairplot

A pairplot is a grid of scatter plots for every pair of numeric features, with histograms on the diagonal. It's the fastest way to get a comprehensive view of your dataset's structure.

# Pairplot — one line gives you a complete overview
g = sns.pairplot(df, hue="segment", palette="deep", diag_kind="kde",
                 plot_kws={"alpha": 0.5, "s": 30})
g.fig.suptitle("Pairwise Feature Relationships", y=1.02, fontsize=14)
g.savefig("pairplot.png", dpi=150)
plt.close()

For a dataset with 5 numeric features, this produces a 5×5 grid — 25 plots in one call. It's heavy for large feature sets, but for exploratory analysis on a subset of your most important features, nothing beats it.

Styling and Themes

Seaborn ships with built-in themes that instantly make your plots publication-ready. Call sns.set_theme() once at the top of your notebook and every subsequent plot (including raw matplotlib) picks up the style.

# Set a clean theme for all plots
sns.set_theme(style="whitegrid", palette="deep", font_scale=1.1)

# Available styles: "darkgrid", "whitegrid", "dark", "white", "ticks"
# You can also set context for different output sizes:
sns.set_context("notebook")  # Options: "paper", "notebook", "talk", "poster"

Use "talk" context for slides (larger fonts and thicker lines) and "paper" for journal submissions (compact and dense). This one setting change adapts every plot for the right medium.

FacetGrid for Multi-panel Figures

A FacetGrid creates a grid of plots where each panel corresponds to a subset of your data — split by one or two categorical variables. It's like running a for-loop over groups and plotting each one, but Seaborn handles the layout, axis sharing, and legends.

g = sns.FacetGrid(df, col="segment", height=4, aspect=1.2)
g.map_dataframe(sns.histplot, x="income", kde=True, color="#5C6BC0")
g.set_titles(col_template="{col_name} Segment")
g.set_axis_labels("Income", "Count")
g.fig.suptitle("Income Distributions by Segment", y=1.02, fontsize=13)
g.savefig("facetgrid.png", dpi=150)
plt.close()

FacetGrid is particularly useful when you want to compare distributions across groups while keeping the scales consistent, so visual comparisons are fair.

Interactive Visualization (Plotly, Bokeh)

Static plots are perfect for papers, reports, and notebooks. But when you're building a dashboard, presenting to stakeholders who want to zoom into specific data points, or exploring a dataset with millions of rows, you need interactivity. That's where Plotly and Bokeh come in.

Plotly produces HTML-based interactive plots with hover tooltips, zoom, pan, and export — all without writing any JavaScript. Its API mirrors matplotlib's closely enough that the learning curve is gentle.

import plotly.express as px

# Interactive scatter with hover info
fig = px.scatter(df, x="age", y="income", color="segment",
                 hover_data=["spending_score"],
                 title="Customer Segments (Interactive)",
                 opacity=0.6)
fig.update_layout(template="plotly_white")
fig.write_html("interactive_scatter.html")  # Open in any browser
fig.show()                                  # Renders inline in Jupyter

Bokeh is the other major option. Its strength is building full web-application dashboards with widgets (sliders, dropdowns, buttons) using pure Python. If you need a live dashboard that updates from a data stream, Bokeh is the tool. For one-off exploration and quick interactivity, Plotly is usually faster to set up.

from bokeh.plotting import figure, output_file, save

p = figure(title="Income vs Age", x_axis_label="Age", y_axis_label="Income",
           width=700, height=450, tools="pan,wheel_zoom,box_zoom,reset,hover")
p.circle(df["age"], df["income"], size=6, alpha=0.5, color="#2196F3")

output_file("bokeh_scatter.html")
save(p)
ℹ️ When to Use What

Matplotlib/Seaborn: notebooks, papers, any static output. Plotly: presentations, exploratory analysis, quick interactive dashboards. Bokeh: production dashboards with widgets and streaming data. Don't over-engineer — 90% of your ML visualization work will be matplotlib and Seaborn.

ML-Specific Plots (Confusion Matrix, ROC, Learning Curves)

These are the plots you'll make every single day as an ML engineer. They aren't about understanding your data — they're about understanding your model. Each one answers a specific diagnostic question, and together they tell you exactly where your model is succeeding and where it's failing.

Confusion Matrix

A confusion matrix shows you, for each true class, how many samples your model assigned to each predicted class. The diagonal cells are correct predictions. Everything off-diagonal is an error — and the pattern of errors tells you what's going wrong. If your spam detector is flagging 15% of legitimate emails as spam (high false positives), the confusion matrix shows that immediately.

from sklearn.metrics import confusion_matrix, ConfusionMatrixDisplay
from sklearn.datasets import make_classification
from sklearn.model_selection import train_test_split
from sklearn.ensemble import RandomForestClassifier

# Train a quick classifier
X, y = make_classification(n_samples=1000, n_features=20, n_classes=3,
                           n_informative=10, random_state=42)
X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.3, random_state=42)

clf = RandomForestClassifier(n_estimators=100, random_state=42)
clf.fit(X_train, y_train)
y_pred = clf.predict(X_test)

# Plot the confusion matrix
fig, ax = plt.subplots(figsize=(7, 6))
cm = confusion_matrix(y_test, y_pred)
disp = ConfusionMatrixDisplay(confusion_matrix=cm, display_labels=["Class 0", "Class 1", "Class 2"])
disp.plot(cmap="Blues", ax=ax, values_format="d")
ax.set_title("Confusion Matrix — Random Forest")
plt.tight_layout()
plt.savefig("confusion_matrix.png", dpi=150)
plt.close()

Read it row by row. Each row is a true class. The numbers in that row tell you where those samples ended up. If row "Class 1" has 85 in the "Class 1" column and 10 in the "Class 2" column, your model is confusing Class 1 with Class 2 ten times. That tells you those two classes might need more training data or better features to distinguish them.

ROC Curve and AUC

The ROC curve (Receiver Operating Characteristic) plots the true positive rate against the false positive rate at every classification threshold. A perfect model hugs the top-left corner. A random coin flip gives you the diagonal. The AUC (Area Under the Curve) collapses this into a single number between 0.5 (random) and 1.0 (perfect).

from sklearn.metrics import roc_curve, auc
from sklearn.preprocessing import label_binarize

# For multiclass, we compute one-vs-rest ROC curves
classes = [0, 1, 2]
y_test_bin = label_binarize(y_test, classes=classes)
y_score = clf.predict_proba(X_test)

fig, ax = plt.subplots(figsize=(8, 6))
colors = ["#1E88E5", "#43A047", "#E53935"]

for i, (cls, color) in enumerate(zip(classes, colors)):
    fpr, tpr, _ = roc_curve(y_test_bin[:, i], y_score[:, i])
    roc_auc = auc(fpr, tpr)
    ax.plot(fpr, tpr, color=color, linewidth=2, label=f"Class {cls} (AUC = {roc_auc:.3f})")

ax.plot([0, 1], [0, 1], "k--", alpha=0.4, label="Random (AUC = 0.500)")
ax.set_xlabel("False Positive Rate")
ax.set_ylabel("True Positive Rate")
ax.set_title("ROC Curves — One-vs-Rest")
ax.legend(loc="lower right")
ax.grid(True, alpha=0.3)
plt.tight_layout()
plt.savefig("roc_curves.png", dpi=150)
plt.close()

ROC curves are most useful for binary classification, but the one-vs-rest approach shown here extends it to multiclass problems. When you're comparing two models, the one with the higher AUC is generally better at ranking positive examples above negative ones — regardless of which threshold you choose.

Precision-Recall Curve

When your dataset is imbalanced (say, 95% negative, 5% positive), ROC curves can be misleadingly optimistic. A model that flags everything as negative gets a very low false positive rate on the huge negative class, making the ROC look decent. The precision-recall curve is the antidote — it focuses on how well you find the positives.

from sklearn.metrics import precision_recall_curve, average_precision_score

fig, ax = plt.subplots(figsize=(8, 6))

for i, (cls, color) in enumerate(zip(classes, colors)):
    precision, recall, _ = precision_recall_curve(y_test_bin[:, i], y_score[:, i])
    ap = average_precision_score(y_test_bin[:, i], y_score[:, i])
    ax.plot(recall, precision, color=color, linewidth=2, label=f"Class {cls} (AP = {ap:.3f})")

ax.set_xlabel("Recall")
ax.set_ylabel("Precision")
ax.set_title("Precision-Recall Curves")
ax.legend(loc="lower left")
ax.grid(True, alpha=0.3)
plt.tight_layout()
plt.savefig("precision_recall.png", dpi=150)
plt.close()
🚨 Imbalanced Datasets

If your positive class is rare (fraud detection, disease diagnosis, anomaly detection), always look at the precision-recall curve, not the ROC. A model with 0.98 ROC-AUC might still have terrible precision at reasonable recall thresholds. The precision-recall curve reveals this; the ROC hides it.

Learning Curves

Learning curves plot model performance (train and validation scores) against the amount of training data. They answer one of the most practical questions in ML: "Will getting more data help?" If the validation curve is still climbing when you run out of data, more data will help. If both curves have flattened and there's a gap between them, you've got a variance problem — your model is too complex. If both have flattened and converged at a low score, you've got a bias problem — your model isn't expressive enough.

from sklearn.model_selection import learning_curve

train_sizes, train_scores, val_scores = learning_curve(
    clf, X, y,
    train_sizes=np.linspace(0.1, 1.0, 10),
    cv=5,
    scoring="accuracy",
    n_jobs=-1,
    random_state=42,
)

train_mean = train_scores.mean(axis=1)
train_std = train_scores.std(axis=1)
val_mean = val_scores.mean(axis=1)
val_std = val_scores.std(axis=1)

fig, ax = plt.subplots(figsize=(8, 6))

ax.fill_between(train_sizes, train_mean - train_std, train_mean + train_std, alpha=0.15, color="#1E88E5")
ax.fill_between(train_sizes, val_mean - val_std, val_mean + val_std, alpha=0.15, color="#E53935")
ax.plot(train_sizes, train_mean, "o-", color="#1E88E5", linewidth=2, label="Training Score")
ax.plot(train_sizes, val_mean, "o-", color="#E53935", linewidth=2, label="Validation Score")

ax.set_xlabel("Training Set Size")
ax.set_ylabel("Accuracy")
ax.set_title("Learning Curves — Random Forest")
ax.legend(loc="lower right")
ax.grid(True, alpha=0.3)
ax.set_ylim(0.5, 1.05)
plt.tight_layout()
plt.savefig("learning_curves.png", dpi=150)
plt.close()

The shaded bands show ±1 standard deviation across the cross-validation folds, giving you a sense of how stable the scores are. This is one of the first plots you should make after any initial model training — it tells you whether to invest in more data, a different model, or better feature engineering.

ℹ️ Your Visualization Workflow

In a typical ML project, the visualization flow goes: (1) histograms and pairplots to understand raw features, (2) correlation heatmaps to catch redundancy, (3) boxplots to spot outliers, (4) learning curves after first model to diagnose bias/variance, (5) confusion matrix and ROC/PR curves to understand final model errors. Master these five steps and you'll catch problems that pure metrics miss.
