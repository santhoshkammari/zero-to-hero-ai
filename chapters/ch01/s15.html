Chapter 1: Python & Programming Foundations 10 topics

These are features you'll bump into in other people's code, library internals, or very specific edge cases. You don't need to master any of them — but recognizing them on sight saves you a "what the heck is this?" moment and an hour of confused Googling.

The Topics

Type Parameter Syntax

Before 3.12, writing a generic function meant creating a TypeVar on a separate line, naming it, then using it. The new syntax bakes it right in:

# old way
from typing import TypeVar
T = TypeVar("T")
def first(items: list[T]) -> T: ...

# new way (3.12+)
def first[T](items: list[T]) -> T: ...

You'll see this in newer libraries that dropped <3.12 support. Same meaning, less boilerplate.

Metaclasses

Metaclasses are one of those things where you think "I'll never need this" until you're reading Django source code and suddenly everything makes sense. A metaclass is a class that creates classes — it controls what happens at class-definition time.

# when you see this in Django:
class User(models.Model):
    name = models.CharField(max_length=100)
    class Meta:
        db_table = "users"
# ...that's metaclass magic wiring up the ORM.

ORMs like SQLAlchemy and Django use them heavily. You'll read code that uses them, rarely write your own. When you think you need a metaclass, you usually want a class decorator instead.

Descriptors

Ever wonder how @property actually works? Descriptors. They're objects that implement __get__, __set__, or __delete__, and Python calls those methods automatically when you access an attribute.

# @property is secretly a descriptor
class Temperature:
    @property
    def fahrenheit(self):
        return self._celsius * 9/5 + 32

# under the hood, Python is calling:
#   Temperature.fahrenheit.__get__(instance, owner)

@classmethod, @staticmethod, and method binding itself all run on this protocol. Understanding it helps when debugging framework internals that seem like magic.

dataclasses Advanced

Basic @dataclass is great. But when your config objects get complex — mutable defaults, computed fields, inheritance — you need the deeper tools:

from dataclasses import dataclass, field

@dataclass
class ExperimentConfig:
    name: str
    tags: list[str] = field(default_factory=list)  # mutable default done right
    lr: float = 0.001

    def __post_init__(self):
        # runs after __init__ — validate, transform, compute derived fields
        self.name = self.name.strip().lower()

field(default_factory=...) for mutable defaults, __post_init__ for validation, and dataclass inheritance all show up once your projects get past the toy stage.

functools — The Rest You'll See in the Wild

You probably know lru_cache. Here are three more from the same module that pop up in well-written codebases:

from functools import singledispatch, total_ordering, cached_property

@singledispatch          # function overloading by type
def process(data): ...

@process.register(list)
def _(data): ...         # called when data is a list

@total_ordering          # just define __eq__ + __lt__, get all 6 comparisons
class Version:
    def __eq__(self, other): ...
    def __lt__(self, other): ...

class Dataset:
    @cached_property     # lazy — computed once, then cached as attribute
    def stats(self): return expensive_computation(self.data)

singledispatch is function overloading. total_ordering saves you from writing six comparison methods. cached_property is lazy evaluation for attributes.

itertools — The Rest You'll See in the Wild

Beyond chain and product, a few lesser-known tools that save real work:

from itertools import tee, zip_longest, starmap, pairwise

a, b = tee(iterator)          # duplicate an iterator (consume independently)
zip_longest([1,2], [3,4,5])   # fills missing with None instead of truncating
starmap(pow, [(2,3), (3,2)])   # like map but unpacks argument tuples → [8, 9]
pairwise([1,2,3,4])           # sliding window of 2 → (1,2), (2,3), (3,4)

pairwise is Python 3.10+. All of these shine in data pipelines where you're chaining transformations on sequences.

tomllib

Since Python 3.11, you can read TOML files with zero dependencies. Read-only — no writing.

import tomllib

with open("pyproject.toml", "rb") as f:
    config = tomllib.load(f)

print(config["project"]["name"])

Useful whenever you're parsing pyproject.toml, experiment configs, or any TOML-based settings. Need to write TOML? You'll still need third-party tomli-w.

f-string Debug Mode

Tiny feature, genuine time-saver. Add = after the variable name inside the f-string and Python prints both the name and the value:

batch_size = 64
lr = 0.001
print(f"{batch_size=}, {lr=}")
# batch_size=64, lr=0.001

Works with expressions too: f"{len(data)=}" prints len(data)=1000. Beats typing the variable name twice during quick debugging.

The Point of This Section

You don't need to master any of these right now. When you encounter them in someone else's code or a library's source, you'll know what's going on — and that awareness is what separates "stuck" from "I can figure this out."

What You Should Now Be Able To Do

  • Recognize the walrus operator, pattern matching, and f-string debug syntax when they show up in a code review
  • Know when someone's using metaclasses vs decorators — and why you'd almost always pick decorators
  • Spot where functools or itertools would simplify a pipeline instead of hand-rolling loops
  • Read framework internals that use descriptors without feeling lost
