Added second model variant to training pipeline. Whole thing collapsed. Not math — no structure. Functions mutating shared state. One change broke three things. Learned why OOP exists.

OOP = skeleton beneath every ML framework — scikit-learn, PyTorch, HuggingFace, LangChain, OpenAI SDK. Every model.fit(X, y) or model(input) uses it. Real reason it matters in production: ten engineers extend same system without breaking each other's work.

Building tiny training framework from scratch. Adding concepts one at a time — encapsulation, contracts, inheritance, composition, patterns, Python internals.

What is Class in OOP

Class = namespace + behavior binder.

Group data + functions under one type. Defines structure (attributes) + operations (methods). Objects = instances of this type.

Key: class creates new type, not just container.

class User:
    def __init__(self, name):
        self.name = name

    def greet(self):
        return f"Hi {self.name}"

User → type. User("A") → object. greet → behavior bound to data.

Core idea: bind state + behavior → enforce rules + reuse + polymorphism.

Spaghetti Script

Where most ML projects start — single Python script. Building minitrainer: loads data, trains model, evaluates. Code most people write first:

import numpy as np

# globals everywhere
train_data = None
targets = None
model_weights = None
learning_rate = 0.01
history = []

def load_data(path):
    global train_data
    train_data = np.load(path)

def train(epochs):
    global model_weights, history
    model_weights = np.random.randn(train_data.shape[1])
    for epoch in range(epochs):
        preds = train_data @ model_weights
        loss = np.mean((preds - targets) ** 2)
        grad = 2 * train_data.T @ (preds - targets) / len(targets)
        model_weights -= learning_rate * grad
        history.append(loss)

def evaluate():
    preds = train_data @ model_weights
    return np.mean((preds - targets) ** 2)

Works for single experiment, single person. Pain begins when extending. Two models with different learning rates? Share same globals. Colleague adding different optimizer? Must read every function to understand state. Testing without real data? Everything wired to everything.

First Class — Encapsulation as Self-Defense

Problem with spaghetti: all state — data, weights, history, learning rate — floats in module-level variables. Any function reads or modifies any of them. Something goes wrong → trace entire file to find which function mutated what.

Class = bundle related state + functions that operate on it into one unit. Each experiment gets own sealed envelope. Data, weights, history for experiment A live inside one envelope. Experiment B gets own. Can't accidentally reach into each other's pockets.

class MiniTrainer:
    def __init__(self, learning_rate=0.01):
        self.learning_rate = learning_rate
        self.weights = None
        self.history = []

    def fit(self, X, y, epochs=100):
        self.weights = np.random.randn(X.shape[1])
        for epoch in range(epochs):
            preds = X @ self.weights
            loss = np.mean((preds - y) ** 2)
            grad = 2 * X.T @ (preds - y) / len(y)
            self.weights -= self.learning_rate * grad
            self.history.append(loss)
        return self

    def predict(self, X):
        return X @ self.weights

Two experiments run simultaneously without interference:

trainer_a = MiniTrainer(learning_rate=0.01)
trainer_b = MiniTrainer(learning_rate=0.001)
trainer_a.fit(X_train, y_train)
trainer_b.fit(X_train, y_train)

Each trainer carries own weights, own history, own learning rate. Sealed envelope idea = encapsulation. Data and methods live in same place. Bug in trainer_a's history? Problem is in MiniTrainer's methods, not random function three files away.

Entire argument for classes, stripped to bone. Not inheritance, not polymorphism. Argument: state and behavior travel together → bugs have fewer places to hide.

Mutable class attribute trap. class Model: layers = [] means every instance shares same list. Append in one instance → shows up in all. Always initialize mutable state in __init__ as self.layers = []. Same trap with mutable default arguments — def __init__(self, items=[]) reuses same list object across calls. Fix: items=None, then self.items = items or [].

MiniTrainer works, but optimization logic hardcoded. Want Adam instead of vanilla gradient descent? Rewrite fit method. Someone wants RMSprop? Copy entire class, change few lines. Not scalable.

Contract — Why Interfaces Matter

Scenario at every ML team: you build trainer. Colleague builds different trainer. Third person writes evaluation script. Evaluation script calls .predict() on whatever model it receives. But colleague called their method .run_inference(). Evaluation breaks.

Pain isn't naming inconsistency — no agreement about what "model" should look like. No contract. Python gives two ways to express contracts.

Abstract Base Class (ABC): "Want to be valid model? Must implement these methods. Forget one → Python refuses to create instance." Fails loudly at construction time — not silently two hours into training run.

from abc import ABC, abstractmethod

class BaseModel(ABC):
    @abstractmethod
    def fit(self, X, y):
        ...

    @abstractmethod
    def predict(self, X):
        ...

class BrokenModel(BaseModel):
    def fit(self, X, y):
        pass
    # forgot predict!

# BrokenModel()
# → TypeError: Can't instantiate abstract class BrokenModel
#   with abstract method predict
# Fails NOW, not when predict() is called hours later.

Protocol (Python 3.8, PEP 544): same idea — "valid model must have fit and predict" — but no inheritance required. Works through structural subtyping. Class has right methods with right signatures → satisfies protocol. Type checker (mypy, pyright) verifies at analysis time, not runtime.

from typing import Protocol
import numpy as np

class Predictor(Protocol):
    def fit(self, X: np.ndarray, y: np.ndarray) -> "Predictor": ...
    def predict(self, X: np.ndarray) -> np.ndarray: ...

class LinearModel:
    def fit(self, X, y):
        self.w = np.linalg.lstsq(X, y, rcond=None)[0]
        return self
    def predict(self, X):
        return X @ self.w

def evaluate(model: Predictor, X_test, y_test):
    preds = model.predict(X_test)
    return np.mean((preds - y_test) ** 2)

# LinearModel satisfies Predictor without inheriting from it.
# mypy will catch it if LinearModel is missing predict().

ABC vs Protocol: ABC → runtime isinstance() checks or base class with default implementations. Protocol → max flexibility, especially third-party classes you can't modify. Newer codebases → Protocol is default.

Inheritance: When Framework Says So

Want minitrainer to work with sklearn's Pipeline, GridSearchCV, cross_val_score? Follow sklearn's contract. Inherit from BaseEstimator + mixin.

Inheritance = framework architect built kitchen (gas, plumbing, ventilation). You bring recipe. Subclass nn.Module or BaseEstimator → you bring recipe. Framework already built kitchen — parameter registration, device movement, gradient tracking, hyperparameter introspection, grid search integration. Get all that by inheriting + filling blanks.

from sklearn.base import BaseEstimator, RegressorMixin
from sklearn.utils.validation import check_is_fitted
import numpy as np

class MiniRegressor(BaseEstimator, RegressorMixin):
    def __init__(self, learning_rate=0.01, epochs=100):
        self.learning_rate = learning_rate
        self.epochs = epochs

    def fit(self, X, y):
        self.coef_ = np.random.randn(X.shape[1])
        for _ in range(self.epochs):
            preds = X @ self.coef_
            grad = 2 * X.T @ (preds - y) / len(y)
            self.coef_ -= self.learning_rate * grad
        return self

    def predict(self, X):
        check_is_fitted(self)
        return X @ self.coef_

Three conventions make this work. First, __init__ stores hyperparameters only — BaseEstimator introspects __init__ signature to power get_params()/set_params(), how GridSearchCV clones estimator with different hyperparameters. Second, fit() returns self — enables method chaining in pipelines. Third, fitted attributes end with underscore (coef_) — check_is_fitted() detects these to verify fit() was called before predict().

RegressorMixin gives score() for free — calls predict(), computes R². That's a mixin: small class adding single slice of behavior without defining __init__. Sklearn uses mixins everywhere — ClassifierMixin, TransformerMixin, ClusterMixin. Each assumes host class has certain methods, gives something back. Fair trade. Right use of inheritance.

Same applies to PyTorch. Subclass nn.Module → implement forward(). Framework's __call__ wraps your forward() with pre-hooks, post-hooks, autograd setup. Write model(x), never model.forward(x) directly — forward() bypasses hooks, silently breaks gradient tracking.

import torch
import torch.nn as nn

class TinyNet(nn.Module):
    def __init__(self, in_dim, hidden, out_dim):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(in_dim, hidden),
            nn.ReLU(),
            nn.Linear(hidden, out_dim),
        )

    def forward(self, x):
        return self.net(x)

model = TinyNet(784, 256, 10)
# model(x) → runs hooks, autograd, THEN forward()
# model.parameters() → auto-discovered from self.net
# model.to('cuda') → recursively moves all parameters

Behind scenes, nn.Module uses custom __setattr__ to intercept every attribute assignment in __init__. Write self.net = nn.Sequential(...) → detects Module, registers as child. That's how model.parameters() recurses into nested modules, finds every learnable weight without manual listing.

Diamond Problem and MRO

Class inherits from multiple parents → which path does super().__init__() take? Python resolves with Method Resolution Order (MRO), computed via C3 linearization.

Key insight: super() ≠ "call my parent." It means "call next class in MRO chain."

class A:
    def __init__(self):
        print("A")

class B(A):
    def __init__(self):
        print("B")
        super().__init__()    # calls C, NOT A

class C(A):
    def __init__(self):
        print("C")
        super().__init__()    # calls A

class D(B, C):
    def __init__(self):
        print("D")
        super().__init__()    # calls B

D()   # prints: D → B → C → A
print(D.__mro__)
# (D, B, C, A, object)

Every class appears once in MRO, always before its parents. class SGDClassifier(BaseSGD, ClassifierMixin) in sklearn — this machinery makes it work without ambiguity.

Cooperative super() Correctly

MRO works only if every class cooperates — accepting and forwarding args it doesn't recognize. Pattern: accept **kwargs, use what you need, forward rest.

class Loggable:
    def __init__(self, verbose=False, **kwargs):
        super().__init__(**kwargs)   # forward remaining kwargs
        self.verbose = verbose

class Serializable:
    def __init__(self, save_path=None, **kwargs):
        super().__init__(**kwargs)
        self.save_path = save_path

class MyModel(Loggable, Serializable):
    def __init__(self, hidden_size, **kwargs):
        super().__init__(**kwargs)   # passes verbose, save_path through chain
        self.hidden_size = hidden_size

# All kwargs find their target class in the MRO
model = MyModel(hidden_size=256, verbose=True, save_path="/tmp/model.pt")

Without **kwargs forwarding, adding mixin to existing class breaks __init__ chains. Most common cooperative multi-inheritance bug — class swallows keyword arguments instead of forwarding.

Composition: Building Your Own Thing

Inheritance = right tool when framework gives kitchen. Building own system — training pipeline, experiment manager, evaluation suite — composition wins almost every time.

Optimization logic baked into fit(). Want different optimizers? Inheritance approach: create MiniTrainerSGD, MiniTrainerAdam, MiniTrainerRMSprop. Three classes identical except few lines. Add new data loading strategy → MiniTrainerSGDWithCSV? Hierarchy explodes.

Composition = opposite. Instead of "trainer is kind of SGD trainer" → "trainer has optimizer." Optimizer is separate object, passed in from outside.

class SGDOptimizer:
    def __init__(self, lr=0.01):
        self.lr = lr

    def step(self, weights, grad):
        return weights - self.lr * grad

class AdamOptimizer:
    def __init__(self, lr=0.001, beta1=0.9, beta2=0.999, eps=1e-8):
        self.lr = lr
        self.beta1, self.beta2, self.eps = beta1, beta2, eps
        self.m = None
        self.v = None
        self.t = 0

    def step(self, weights, grad):
        if self.m is None:
            self.m = np.zeros_like(weights)
            self.v = np.zeros_like(weights)
        self.t += 1
        self.m = self.beta1 * self.m + (1 - self.beta1) * grad
        self.v = self.beta2 * self.v + (1 - self.beta2) * grad ** 2
        m_hat = self.m / (1 - self.beta1 ** self.t)
        v_hat = self.v / (1 - self.beta2 ** self.t)
        return weights - self.lr * m_hat / (np.sqrt(v_hat) + self.eps)

class Trainer:
    def __init__(self, optimizer):
        self.optimizer = optimizer
        self.history = []

    def fit(self, X, y, epochs=100):
        weights = np.random.randn(X.shape[1])
        for _ in range(epochs):
            preds = X @ weights
            loss = np.mean((preds - y) ** 2)
            grad = 2 * X.T @ (preds - y) / len(y)
            weights = self.optimizer.step(weights, grad)
            self.history.append(loss)
        self.weights = weights
        return self

Swapping optimizers = one-line change:

trainer_sgd = Trainer(SGDOptimizer(lr=0.01))
trainer_adam = Trainer(AdamOptimizer(lr=0.001))

No new subclass. No copy-paste. Trainer doesn't know or care which optimizer — calls self.optimizer.step(), trusts result. "has-a" relationship: trainer has optimizer. Isn't optimizer.

Sklearn's Pipeline = masterclass. Pipeline([('scaler', StandardScaler()), ('clf', SVC())]) composes scaler + classifier into one unit. Doesn't inherit from either. Has them. Swap SVC() for RandomForestClassifier() without changing pipeline class.

Rule: inherit when framework gives base class with real functionality. Compose when building own system needing flexibility. Production ratio: ~20% inheritance, ~80% composition.

Patterns You'll See in Production

Design patterns sound like enterprise Java, but you already use them — every nn.Module subclass = Template Method, every optimizer you pass in = Strategy. Naming what you know.

Template Method

Pattern behind every framework that says "implement this one method, we handle rest." PyTorch nn.Module: framework defines __call__ → runs pre-forward hooks, calls your forward(), runs post-forward hooks, manages autograd. You only write forward(). Template (skeleton of algorithm) = fixed. You fill variable part.

Sklearn same. TransformerMixin.fit_transform() calls your fit() then transform(). You implement pieces; framework orchestrates.

Value: framework author guarantees hooks always run, gradients always tracked, fitted state always verified — regardless of what you put in forward() or fit(). Can't accidentally skip safety rails.

Strategy

Already built this in composition section. Trainer accepts optimizer object, calls self.optimizer.step() → optimizer = strategy. Different strategies (SGD, Adam, RMSprop) interchangeable because same interface. Trainer doesn't know which one running.

PyTorch uses everywhere. Loss functions, activation functions, LR schedulers — all strategies. Pass them in, training loop calls them, specific behavior determined by injected object.

Observer (Hooks)

System broadcasts events to interested listeners without knowing who they are. PyTorch = hooks — functions registered on module, fire automatically during forward/backward passes.

def log_activations(module, input, output):
    print(f"{module.__class__.__name__}: "
          f"output shape {output.shape}, "
          f"mean {output.mean():.4f}")

model = TinyNet(784, 256, 10)
hook = model.net[0].register_forward_hook(log_activations)
# now every forward pass through the first Linear layer
# triggers our logging function
# hook.remove() when done

Gradient clipping, activation logging, feature extraction from intermediate layers — all hooks. Model doesn't know it's being observed. Add monitoring/debugging/profiling to any model without modifying code.

Registry / Factory

Pattern HuggingFace built empire on. AutoModel.from_pretrained("bert-base-uncased") → loads config, reads model_type field ("bert"), looks up in dictionary mapping strings to classes: {"bert": BertModel, "gpt2": GPT2Model, "t5": T5Model, ...}. Instantiates correct class, loads weights.

Dictionary = registry. from_pretrained() classmethod = factory. Together → load any of hundreds of architectures through single uniform interface. Same pattern powers AutoTokenizer, AutoConfig. See "Auto" in HuggingFace → think "registry mapping config strings to concrete classes."

Dunder Methods That Earn Their Keep

Python has 100+ dunder methods. Most never touched. Handful shows up in ML code constantly — difference between reading framework source fluently and staring in confusion.

__call__ — makes instance callable like function. model(x) → model.__call__(x). PyTorch nn.Module defines __call__ to run hooks + autograd before calling forward(). Why model(x) ≠ model.forward(x) — forward() skips entire hook and gradient machinery.

__len__ + __getitem__ — dataset pattern. PyTorch DataLoader needs __len__ (how many samples) and __getitem__ (fetch by index). Implement two methods → works with shuffling, batching, multi-worker loading.

class TextDataset:
    def __init__(self, texts, labels):
        self.texts = texts
        self.labels = labels

    def __len__(self):
        return len(self.texts)

    def __getitem__(self, idx):
        return self.texts[idx], self.labels[idx]

# DataLoader(TextDataset(texts, labels), batch_size=32, shuffle=True)
# Batching, shuffling, multiprocess prefetch — all from two methods.

__enter__ + __exit__ — context managers. Every with torch.no_grad(): or with open(path) as f:. __enter__ sets up resource, __exit__ tears down — even if exception fires. Anything needing guaranteed cleanup: timers, gradient disabling, DB connections.

import time

class Timer:
    def __enter__(self):
        self.start = time.perf_counter()
        return self

    def __exit__(self, *exc):
        self.elapsed = time.perf_counter() - self.start
        print(f"{self.elapsed:.3f}s")
        return False   # don't suppress exceptions

# with Timer():
#     model.fit(X, y)
# prints elapsed time even if fit() crashes

__repr__ — debugging. Controls what you see in REPL/print. Sklearn auto-generates from get_params() → shows RandomForestClassifier(n_estimators=100, ...) instead of memory address. Define one string method → make it __repr__.

__iter__ + __next__ — makes objects iterable. Essential for streaming datasets that don't fit in memory.

class BatchIterator:
    def __init__(self, X, y, batch_size=32):
        self.X, self.y = X, y
        self.batch_size = batch_size
        self.idx = 0

    def __iter__(self):
        self.idx = 0
        return self

    def __next__(self):
        if self.idx >= len(self.X):
            raise StopIteration
        sl = slice(self.idx, self.idx + self.batch_size)
        self.idx += self.batch_size
        return self.X[sl], self.y[sl]

# for X_batch, y_batch in BatchIterator(X, y, batch_size=64):
#     train_step(X_batch, y_batch)

Modern Toolkit — Dataclasses and Protocols

Config class that's nothing but __init__ storing dozen parameters? @dataclass kills boilerplate dead.

from dataclasses import dataclass, field

@dataclass
class TrainConfig:
    lr: float = 0.001
    epochs: int = 100
    batch_size: int = 32
    tags: list = field(default_factory=list)

cfg = TrainConfig(lr=0.01)
print(cfg)   # TrainConfig(lr=0.01, epochs=100, batch_size=32, tags=[])

cfg2 = TrainConfig(lr=0.01)
cfg == cfg2  # True — __eq__ compares all fields automatically

@dataclass auto-generates __init__, __repr__, __eq__ from field annotations. Add frozen=True → immutable config object, FrozenInstanceError on modification, __hash__ auto-generated → works as dict keys and set members.

field(default_factory=list) solves mutable default problem. Creates fresh list for each instance. Dataclass equivalent of items=None; self.items = items or [] pattern, but cleaner.

HuggingFace uses dataclasses extensively for training arguments. TrainingArguments = dataclass with dozens of fields — works with serialization, comparison, printing for free.

Protocols (covered earlier) = typing-era answer to duck typing. Python traditionally: "has fit method → it's model." Protocol lets you express contract in code, mypy verifies statically. No inheritance. No runtime overhead. Dataclasses for data-holding classes + Protocols for behavioral contracts = most of what you need for clean modern Python ML.

Anti-Patterns That Kill Codebases

  • God Class — does everything (load, train, evaluate, save, notify). Fix: extract separate objects, wire in thin orchestrator.
  • Deep Inheritance — 3+ levels = can't trace which method runs. Fix: flatten to 1-2 levels into framework base, compose the rest.
  • Global Mutable State — singleton configs, module-level dicts mutated during training. Spaghetti script wearing class costume. Fix: pass dependencies explicitly.
  • Premature Abstraction — building BaseModel with factory pattern before second implementation exists. Rule: wait for 2-3 concrete implementations before extracting shared interface.
