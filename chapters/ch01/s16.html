Jupyter Notebooks

Chapter 1: Python & Programming Foundations 2 subtopics
TL;DR

Jupyter Notebooks let you mix runnable code, visualizations, and prose in a single document — making them the default workspace for data science and ML experimentation. They run in your browser, execute Python through a kernel process, and save everything as a JSON .ipynb file. The power is real, but so are the pitfalls: out-of-order execution is the number-one source of "works on my machine" bugs. Learn the environment, learn the discipline, and notebooks become your most productive tool.

Jupyter Environment

Here is a scenario you have probably lived through. You write a Python script, run it, stare at a wall of terminal output, tweak one line, re-run the whole thing, wait again. For a ten-line utility that is fine. For exploratory work — loading a dataset, cleaning columns, trying three different plots, fitting a model, tweaking hyperparameters — that loop is painfully slow. You want to run code in small pieces, see results immediately, keep the good parts, and throw away the bad parts without re-executing everything from scratch.

That is the problem Jupyter Notebooks solve. A notebook is an interactive document that lives in your browser. It contains cells — small blocks of code or text — that you can run one at a time. Each cell's output appears right below it: a number, a table, a chart, an error traceback. You edit, run, inspect, iterate. The feedback loop shrinks from minutes to seconds.

The name "Jupyter" comes from three languages — Julia, Python, and R — but Python is by far the most common kernel. Under the hood, Jupyter is a client-server architecture: your browser is the client, a Python process called the kernel is the server, and they talk over a messaging protocol. Your code does not run in the browser. It runs in a real Python process on your machine (or a remote server), and the results get sent back to the browser for display.

Installing Jupyter

Installation is one command. If you have Python and pip, you are five seconds away from a working notebook environment.

# Install the classic notebook interface
pip install notebook

# Or install JupyterLab (the newer, more powerful interface)
pip install jupyterlab

If you are using a virtual environment (and you should be), activate it first. Jupyter will automatically use the Python interpreter from that environment, which means it sees whatever packages you have installed there. No extra configuration needed.

# Best practice: install inside a virtual environment
python -m venv myproject-env
source myproject-env/bin/activate   # Linux/Mac
pip install jupyterlab pandas matplotlib

Launching the Notebook Server

Once installed, you start the server from your terminal:

# Launch classic notebook
jupyter notebook

# Or launch JupyterLab
jupyter lab

This does two things. First, it starts a local web server — typically on http://localhost:8888. Second, it opens that URL in your default browser. You will see a file browser showing the directory where you ran the command. From there, you can create new notebooks, open existing ones, or navigate folders.

The terminal where you launched Jupyter will show log output. Do not close that terminal — it is running the server process. If you close it, your notebooks lose their connection to the kernel and you cannot run cells anymore.

Token Authentication

By default, Jupyter generates a random token for security. You will see a URL like http://localhost:8888/?token=abc123... in the terminal output. If your browser does not open automatically, copy that full URL. The token prevents anyone on your local network from accessing your notebook server without authorization.

JupyterLab vs Jupyter Notebook

You will hear both names thrown around. They are different interfaces to the same underlying system.

Jupyter Notebook (the classic interface) is a single-document view. One notebook per browser tab. It is simple, loads fast, and has been around since 2014. If you are following an older tutorial, this is what they are using.

JupyterLab is the next-generation interface, released in 2018. It gives you a full IDE-like layout: a file browser on the left, multiple tabs, a terminal emulator, a text editor, and the ability to arrange notebooks side by side. It supports extensions for things like Git integration, table of contents, and variable inspectors.

For day-to-day work, JupyterLab is the better choice. It does everything the classic interface does, plus more. The notebook file format (.ipynb) is identical between the two — a notebook created in one opens fine in the other. Think of it as the difference between Notepad and VS Code: same files, different editing experience.

Cell Types

Every notebook is a sequence of cells. Each cell has a type that determines how Jupyter treats it.

Code cells contain Python (or whatever language your kernel supports). When you run a code cell, the kernel executes it and displays the output below. This is where your actual programming happens.

Markdown cells contain formatted text written in Markdown syntax. When you run a Markdown cell, it renders into HTML — headings, bold text, links, images, LaTeX math equations, even tables. These cells are how you document your work, explain your reasoning, and make the notebook readable by humans.

Raw cells are unprocessed text. Jupyter does not execute or render them. They exist mostly for edge cases — like when you are exporting to a format that needs literal content passed through. In practice, you will rarely use raw cells.

# This is a code cell. When you run it, Python executes it.
import pandas as pd

df = pd.read_csv("sales.csv")
df.head()  # The output (a table) appears right below this cell

To change a cell's type in JupyterLab, use the dropdown in the toolbar or keyboard shortcuts: Y for code, M for markdown, R for raw (press Escape first to enter command mode).

Running Cells and Execution Order

This is where notebooks get powerful — and dangerous.

You run a cell by pressing Shift+Enter (run and move to next cell) or Ctrl+Enter (run and stay on current cell). Each code cell that runs gets a number in square brackets to its left: [1], [2], [3], and so on. That number tells you the order of execution, not the position in the notebook.

Here is the critical thing to understand: cells do not have to run top-to-bottom. You can run cell 5, then cell 2, then cell 5 again. The kernel maintains a single global state — all variables, imports, and function definitions live in one namespace. When you run a cell, it modifies that shared state. When you run another cell, it sees those modifications.

# Cell [1]
x = 10

# Cell [2]
print(x)  # Output: 10

# Now go back and change Cell [1] to x = 99, re-run it:
# Cell [3] (re-running the first cell)
x = 99

# Cell [4] (re-running the second cell)
print(x)  # Output: 99

This flexibility is what makes notebooks great for exploration. But it is also the number-one footgun.

The Execution Order Trap

Imagine you define df in cell 3, use it in cell 7, then delete cell 3. Cell 7 still works — for now — because df is still in kernel memory. But if you restart the kernel and run all cells top-to-bottom, cell 7 will crash with a NameError. This is the most common notebook bug. The notebook file looks fine, the cells ran fine during your session, but the document is not reproducible. Always test by doing Kernel → Restart & Run All before sharing.

Kernel Management

The kernel is the Python process that runs your code. It starts when you open a notebook and stays alive until you shut it down. Understanding kernel management saves you from a lot of confusion.

Restart Kernel kills the Python process and starts a fresh one. All variables, imports, and in-memory data are gone. The notebook cells still have their code and displayed outputs, but those outputs are stale snapshots — they do not reflect the current kernel state (which is now empty). Use this when your environment gets into a weird state, or when you want to verify your notebook runs cleanly from scratch.

Interrupt Kernel sends a signal (like Ctrl+C in a terminal) to stop the currently running cell. Use this when a cell is taking too long — maybe you accidentally started an infinite loop or kicked off a computation on a dataset ten times larger than intended.

Restart & Run All is your best friend. It restarts the kernel and then executes every cell from top to bottom in order. This is the gold standard test for reproducibility. If it fails, your notebook has hidden dependencies on execution order.

# Quick way to check what is in kernel memory
%who       # Lists all variables in the namespace
%whos      # Lists variables with type and size info
%reset     # Clears everything (asks for confirmation)

Effective Notebook Usage

Knowing how notebooks work is step one. Using them well is step two. A sloppy notebook is worse than a script — it gives the illusion of documentation while actually being a maze of out-of-order cells, mystery variables, and unreproducible outputs. A disciplined notebook is a beautiful thing: a self-contained narrative that mixes explanation, code, and results into something a colleague can read and understand without you sitting next to them.

Markdown for Documentation

Markdown cells are not decoration. They are the difference between a notebook that communicates and one that confuses.

Use headings to create structure. Use prose to explain why you are doing something, not just what. The code already shows the "what." Write the thought process: why did you choose this approach? What did you try that did not work? What does the output mean?

# In a Markdown cell, you can write:

## Data Cleaning

# The raw dataset has 15,000 rows but includes test entries
# (order IDs starting with "TEST-") and rows with negative
# quantities that represent returns. We remove both.

# You can also use LaTeX for math:
# The loss function is $L = -\sum_{i} y_i \log(\hat{y}_i)$

# And tables:
# | Metric   | Before Cleaning | After Cleaning |
# |----------|----------------|----------------|
# | Rows     | 15,000         | 12,847         |
# | Nulls    | 342            | 0              |

Magic Commands

Magic commands are special Jupyter instructions that start with % (line magics) or %% (cell magics). They are not Python — they are Jupyter extensions that do things Python cannot do on its own.

%timeit is probably the one you will use most. It benchmarks a single line by running it many times and reporting the average execution time.

# Time a single line
%timeit sum(range(1000000))
# Output: 12.3 ms ± 456 µs per loop (mean ± std. dev. of 7 runs, 100 loops each)

# Time an entire cell with %%timeit
%%timeit
total = 0
for i in range(1000000):
    total += i
# Output: 78.2 ms ± 1.23 ms per loop (mean ± std. dev. of 7 runs, 10 loops each)

%matplotlib inline tells matplotlib to render plots directly in the notebook output, instead of opening a separate window. You typically put this near the top of your notebook, right after your imports.

%matplotlib inline
import matplotlib.pyplot as plt
import numpy as np

x = np.linspace(0, 10, 100)
plt.plot(x, np.sin(x))
plt.title("Sine Wave")
plt.xlabel("x")
plt.ylabel("sin(x)")
plt.show()  # The plot appears right below this cell

%%writefile writes the contents of a cell to a file. This is handy when you prototype a function in a notebook and then want to save it as a standalone .py module.

%%writefile utils.py
def clean_dataframe(df):
    """Remove test entries and negative quantities."""
    df = df[~df["order_id"].str.startswith("TEST-")]
    df = df[df["quantity"] > 0]
    return df.dropna()

# Output: Writing utils.py

A few other useful magics:

%pwd               # Print working directory
%cd /path/to/dir   # Change directory
%ls                # List files
%env               # Show environment variables
%load script.py    # Load a file's contents into a cell
%%html             # Render the cell as HTML
%history           # Show command history

Inline Plotting

One of the main reasons data scientists love notebooks is seeing plots right next to the code that generated them. With %matplotlib inline set, any matplotlib figure renders directly in the output area. But matplotlib is not the only option.

# Seaborn for statistical plots (built on matplotlib)
import seaborn as sns
tips = sns.load_dataset("tips")
sns.scatterplot(data=tips, x="total_bill", y="tip", hue="time")

# Plotly for interactive plots
import plotly.express as px
fig = px.scatter(tips, x="total_bill", y="tip", color="time")
fig.show()  # Renders an interactive, zoomable chart in the notebook

Interactive plotting libraries like Plotly are especially powerful in notebooks because the interactivity (hover, zoom, pan) works right in the browser. In a script, you would need to save the chart to an HTML file and open it separately.

Exporting Notebooks

A .ipynb file is a JSON document. It is great for Jupyter but not great for sharing with people who do not have Jupyter installed. Fortunately, you can export notebooks to other formats.

# Export to HTML (anyone with a browser can view it)
jupyter nbconvert --to html my_notebook.ipynb

# Export to PDF (requires LaTeX installed)
jupyter nbconvert --to pdf my_notebook.ipynb

# Export to a plain Python script (strips markdown, keeps code)
jupyter nbconvert --to script my_notebook.ipynb

# Export to Markdown
jupyter nbconvert --to markdown my_notebook.ipynb

The HTML export is the most universally useful. It creates a self-contained file that preserves all outputs, plots, and formatting. Send it to a colleague, host it on a website, or attach it to a report. The Python script export is useful when you have finished experimenting and want to turn your notebook into production code — though you will almost certainly need to clean it up afterwards.

Version Control Tip

Notebook files are JSON, which makes them technically text files — but they are noisy in Git diffs. Cell outputs (especially base64-encoded images) create enormous diffs. Before committing, clear all outputs: Kernel → Restart & Clear Output, then save. Alternatively, use nbstripout as a Git filter to automatically strip outputs on commit. Your diffs become clean and reviewable.

Best Practices for Reproducibility

Notebooks have a bad reputation in some engineering circles, and honestly, it is deserved — when they are used carelessly. The problem is not the tool. The problem is that notebooks make it easy to write code that works during your interactive session but breaks when anyone else (or future-you) tries to run it. Here is how to avoid that.

First: put all imports at the top. Create a single cell at the very beginning of the notebook that imports everything. Do not scatter import pandas as pd halfway down the notebook. When someone reads your notebook, they should see every dependency in the first cell.

# Cell 1: All imports
import pandas as pd
import numpy as np
import matplotlib.pyplot as plt
from sklearn.model_selection import train_test_split
from sklearn.ensemble import RandomForestClassifier

%matplotlib inline

Second: run top-to-bottom before sharing. Do Kernel → Restart & Run All. If any cell fails, fix it. This is the single most important habit. If you do nothing else from this section, do this.

Third: avoid hidden state. If you delete a cell, restart the kernel and run all cells to make sure nothing depended on it. If you rename a variable, search the rest of the notebook for the old name. The kernel's memory is not visible in the notebook file — your cells are the only source of truth.

Fourth: pin your dependencies. Include a cell that records the exact package versions you used.

# Last cell: record environment for reproducibility
!pip freeze > requirements.txt

# Or, more selectively:
import pandas, sklearn, matplotlib
print(f"pandas:     {pandas.__version__}")
print(f"sklearn:    {sklearn.__version__}")
print(f"matplotlib: {matplotlib.__version__}")
# Output:
# pandas:     2.1.4
# sklearn:    1.3.2
# matplotlib: 3.8.2

Fifth: keep notebooks focused. One notebook per task — data cleaning, exploration, model training, evaluation. A 200-cell mega-notebook that does everything is impossible to debug, impossible to review, and impossible to reproduce. If your notebook has more than 30-40 cells, consider splitting it.

The Reproducibility Checklist

Before sharing or committing any notebook, run through this list: (1) All imports in the first cell? (2) Restart & Run All succeeds without errors? (3) No cells depend on deleted or reordered cells? (4) Package versions recorded? (5) Data file paths are relative, not absolute? If you answer yes to all five, your notebook is in good shape.

Google Colab Overview

Everything we have discussed so far runs on your local machine. But what if you need a GPU for training a neural network and you do not have one? Or what if you want to share a notebook with a colleague so they can run it without installing anything? That is where Google Colab (short for Colaboratory) comes in.

Colab is a free, cloud-hosted Jupyter environment provided by Google. You open it in your browser, write and run code in notebook cells, and the computation happens on Google's servers. It uses a modified version of JupyterLab under the hood, so the experience is familiar — cells, kernel, same keyboard shortcuts.

The killer feature is free GPU access. Colab provides NVIDIA GPUs (typically T4s) and even TPUs at no cost. There are usage limits — you get a few hours of GPU time per day, and your session can be disconnected after idle periods — but for learning, prototyping, and running experiments that would take forever on a laptop CPU, it is invaluable.

# In Colab: check what GPU you have
# First, set runtime type: Runtime → Change runtime type → GPU
import torch
print(torch.cuda.is_available())        # True (if GPU runtime is active)
print(torch.cuda.get_device_name(0))    # e.g., "Tesla T4"

Colab notebooks are stored in your Google Drive. Sharing is as easy as sharing a Google Doc — click "Share," add an email, done. The recipient can view, comment, or make a copy to edit. This makes Colab the de facto standard for ML tutorials, course assignments, and research paper code. When you see a "Open in Colab" badge in a GitHub README, that is someone giving you a one-click way to run their code without any local setup.

There are a few things to know about Colab that differ from a local Jupyter setup:

Files are ephemeral. When your Colab session ends, any files you created (downloaded datasets, trained models) disappear. To persist them, save to Google Drive:

# Mount Google Drive in Colab
from google.colab import drive
drive.mount('/content/drive')

# Now you can read/write files that persist:
df = pd.read_csv('/content/drive/MyDrive/data/sales.csv')
model.save('/content/drive/MyDrive/models/my_model.h5')

Pre-installed packages are generous. Colab comes with most data science libraries already installed — NumPy, Pandas, Matplotlib, Scikit-learn, TensorFlow, PyTorch. If you need something extra, !pip install some-package works, but you will need to re-install it each session since the environment resets.

Colab also offers paid tiers (Colab Pro, Colab Pro+) with longer runtimes, more memory, and access to faster GPUs like the A100. For serious training jobs, these tiers or a dedicated cloud instance make more sense. But for learning and experimentation, the free tier is more than enough.

Local vs Colab — When to Use Which

Use a local Jupyter setup when you are working with sensitive data, need full control over your environment, or have a powerful machine. Use Colab when you need a GPU you do not own, want to share runnable notebooks effortlessly, or are following along with someone else's tutorial. Many practitioners use both: prototype in Colab for quick GPU access, then move to local (or a cloud VM) for longer, more controlled experiments.

Notebooks are a tool, and like any tool, they reward skill. The interactive loop — write, run, see, iterate — is unmatched for exploration. The document format — code plus narrative plus output — is unmatched for communication. The pitfalls — hidden state, execution order bugs, non-reproducibility — are real but avoidable with the discipline we covered here. Master the environment, adopt the practices, and you will understand why Jupyter became the default workspace for an entire field.
