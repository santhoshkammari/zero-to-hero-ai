Async Programming

Chapter 1: Python & Programming Foundations 6 subtopics
TL;DR

Async programming lets a single thread handle thousands of concurrent I/O operations — network calls, database queries, file reads — without sitting idle waiting for responses. Python's asyncio module provides coroutines (async def / await), an event loop, and tools like gather() and TaskGroup for orchestrating concurrent work. For ML services that serve predictions over HTTP or fetch training data from multiple sources, async code can be 10–50× faster than sequential requests. The key insight: async is about concurrency (juggling many tasks), not parallelism (doing many things simultaneously on multiple CPUs).

Why Async Matters: The Waiting Problem

Picture this. Your ML inference service gets a request. It needs to fetch a user profile from a database, pull the user's recent activity from a cache, grab the latest model weights from a registry, and then run prediction. Each of those I/O operations takes maybe 50 milliseconds. Done sequentially, that's 200ms of waiting before any computation happens. And during each wait, your thread is doing absolutely nothing — it's blocked, staring at the ceiling, waiting for bytes to arrive over the network.

Now imagine 100 requests arrive per second. With synchronous code, you'd need 100 threads just to keep up, and most of those threads are sleeping. That's a massive waste of memory and operating system resources. This is the problem async programming solves. Instead of one thread per request, a single thread can juggle hundreds of requests by switching to another task whenever one is waiting on I/O.

Think of it like a chef in a kitchen. A synchronous chef puts bread in the toaster and stands there staring at it until it pops. An async chef puts bread in the toaster, starts chopping vegetables, puts water on to boil, and only comes back to the toaster when it dings. Same chef, same kitchen, dramatically more throughput.

Coroutines: async def and await

A coroutine is a function that can be paused and resumed. In Python, you create one by writing async def instead of def. Inside a coroutine, you use await to pause execution until some I/O operation completes. When you hit an await, control returns to the event loop, which can run other coroutines while you wait.

import asyncio
import aiohttp

async def fetch_user_profile(user_id: int) -> dict:
    """A coroutine — note 'async def' instead of 'def'."""
    async with aiohttp.ClientSession() as session:
        # 'await' pauses this coroutine until the HTTP response arrives.
        # The event loop runs other coroutines during this pause.
        async with session.get(f"https://api.example.com/users/{user_id}") as resp:
            return await resp.json()

async def fetch_model_metadata(model_name: str) -> dict:
    async with aiohttp.ClientSession() as session:
        async with session.get(f"https://registry.example.com/models/{model_name}") as resp:
            return await resp.json()

Calling a coroutine function doesn't run it. It returns a coroutine object — think of it as a recipe card, not a cooked meal. You need to await it or hand it to the event loop to actually execute.

# This does NOT run the coroutine — it creates a coroutine object
coro = fetch_user_profile(42)
print(type(coro))  # <class 'coroutine'>

# This actually runs it (inside another coroutine)
async def main():
    profile = await fetch_user_profile(42)
    print(profile)
Common Mistake: Forgetting await

If you call a coroutine without await, Python won't raise an error — it will silently return a coroutine object instead of the result you expected. You'll get a RuntimeWarning: coroutine was never awaited in the logs. This is the #1 async debugging headache. Whenever a function returns an unexpected coroutine object, check for a missing await.

The Event Loop: asyncio

The event loop is the brain behind async programming. It's a single-threaded scheduler that keeps a list of tasks (coroutines) and runs them cooperatively. When one task hits an await, the loop switches to the next ready task. When an I/O operation completes, the loop marks the waiting task as ready and resumes it on the next cycle.

In modern Python (3.7+), you almost never interact with the event loop directly. You use asyncio.run() as your entry point:

import asyncio

async def main():
    print("Starting...")
    await asyncio.sleep(1)  # Simulates an I/O wait
    print("Done!")

# This creates an event loop, runs main(), then cleans up
asyncio.run(main())

Under the hood, asyncio.run() creates a new event loop, schedules your coroutine, runs until it completes, and then closes the loop. In Jupyter notebooks, a loop is already running, so you'd use await main() directly instead.

How the Event Loop Actually Works

The event loop uses your OS's I/O multiplexing mechanism — epoll on Linux, kqueue on macOS, IOCP on Windows. These system calls efficiently monitor thousands of file descriptors (sockets, files, pipes) simultaneously and tell the loop which ones are ready. This is why a single thread can handle thousands of connections — the OS does the heavy lifting of watching for I/O readiness.

asyncio.gather(), wait(), and TaskGroup()

Running one coroutine at a time with await is sequential — you're just writing synchronous code with extra steps. The real power of async comes from running multiple coroutines concurrently. Python gives you three tools for this.

asyncio.gather() — Fire and Collect

asyncio.gather() takes multiple coroutines and runs them all concurrently. It returns the results in the same order you passed the coroutines, regardless of which one finished first.

import asyncio
import time

async def fetch_from_service(name: str, delay: float) -> str:
    await asyncio.sleep(delay)  # Simulate network I/O
    return f"{name} responded"

async def main():
    start = time.perf_counter()

    # Sequential — takes 3 seconds total
    r1 = await fetch_from_service("DB", 1.0)
    r2 = await fetch_from_service("Cache", 0.5)
    r3 = await fetch_from_service("Registry", 1.5)
    print(f"Sequential: {time.perf_counter() - start:.1f}s")  # ~3.0s

    start = time.perf_counter()

    # Concurrent with gather — takes 1.5 seconds (the longest single call)
    r1, r2, r3 = await asyncio.gather(
        fetch_from_service("DB", 1.0),
        fetch_from_service("Cache", 0.5),
        fetch_from_service("Registry", 1.5),
    )
    print(f"Concurrent: {time.perf_counter() - start:.1f}s")  # ~1.5s

asyncio.run(main())

That's a 2× speedup with zero extra threads. The three network calls overlap in time because while one is waiting for bytes, the others are making progress.

asyncio.wait() — Fine-Grained Control

asyncio.wait() gives you more control. It returns two sets: completed tasks and pending tasks. You can use it with different strategies — wait for the first task to complete, wait for all tasks, or wait with a timeout.

async def main():
    tasks = [
        asyncio.create_task(fetch_from_service("DB", 1.0)),
        asyncio.create_task(fetch_from_service("Cache", 0.5)),
        asyncio.create_task(fetch_from_service("Registry", 1.5)),
    ]

    # Process results as they complete (first-come, first-served)
    done, pending = await asyncio.wait(tasks, return_when=asyncio.FIRST_COMPLETED)
    for task in done:
        print(f"First result: {task.result()}")

    # Wait for the rest
    done, pending = await asyncio.wait(pending)
    for task in done:
        print(f"Remaining: {task.result()}")

TaskGroup — Structured Concurrency (Python 3.11+)

TaskGroup is the modern, safer way to manage concurrent tasks. It guarantees that if any task raises an exception, all other tasks in the group are cancelled. This is called structured concurrency — tasks have a clear lifetime tied to a scope.

async def main():
    results = {}

    async with asyncio.TaskGroup() as tg:
        task1 = tg.create_task(fetch_from_service("DB", 1.0))
        task2 = tg.create_task(fetch_from_service("Cache", 0.5))
        task3 = tg.create_task(fetch_from_service("Registry", 1.5))

    # If we get here, ALL tasks completed successfully
    results = {
        "db": task1.result(),
        "cache": task2.result(),
        "registry": task3.result(),
    }
gather() vs TaskGroup

Use gather() when you want a quick fire-and-collect for a known set of coroutines. Use TaskGroup when error handling matters — if one service is down, you probably want to cancel all the other calls instead of waiting for them. In production ML services, prefer TaskGroup for its clean cancellation semantics.

Mixing Async with Threads: run_in_executor

Here's a real-world problem. You've written a beautiful async API server, but your model inference uses NumPy, PyTorch, or scikit-learn — all of which are synchronous, CPU-bound libraries. You can't await a synchronous function. If you call model.predict(x) inside a coroutine, it blocks the entire event loop, freezing all other requests.

The solution is run_in_executor(), which runs a synchronous function in a thread pool (or process pool) without blocking the event loop.

import asyncio
from concurrent.futures import ThreadPoolExecutor
import numpy as np

# Pretend this is a heavy ML model
def sync_predict(features: np.ndarray) -> np.ndarray:
    """Synchronous, CPU-bound prediction — can't use await here."""
    import time
    time.sleep(0.1)  # Simulate computation
    return features * 2.0

async def handle_prediction_request(features: np.ndarray):
    loop = asyncio.get_running_loop()

    # Run the blocking function in a thread pool
    result = await loop.run_in_executor(
        None,  # None = default ThreadPoolExecutor
        sync_predict,
        features
    )
    return result

async def main():
    # Handle 10 prediction requests concurrently
    requests = [np.array([i, i+1, i+2]) for i in range(10)]
    results = await asyncio.gather(
        *[handle_prediction_request(r) for r in requests]
    )
    print(f"Got {len(results)} predictions")

asyncio.run(main())
Thread Pool vs Process Pool for ML

The default ThreadPoolExecutor works fine for I/O-bound tasks and for CPU-bound code that releases the GIL (NumPy, PyTorch operations). But for pure Python CPU-bound code, threads won't give you true parallelism because of the GIL. In that case, use ProcessPoolExecutor — but be aware that it serializes data between processes, which is expensive for large arrays. For most ML inference, ThreadPoolExecutor is the right choice because NumPy and PyTorch release the GIL during computation.

Async Patterns for ML Services

Let's put it all together with patterns you'll actually use in production ML systems.

Pattern 1: Concurrent Feature Fetching

Before running inference, you often need to gather features from multiple sources. Async makes this fast:

import asyncio
import aiohttp
from dataclasses import dataclass

@dataclass
class FeatureSet:
    user_features: dict
    item_features: dict
    context_features: dict

async def fetch_features(user_id: int, item_id: int) -> FeatureSet:
    async with aiohttp.ClientSession() as session:
        user_coro = session.get(f"http://feature-store/users/{user_id}")
        item_coro = session.get(f"http://feature-store/items/{item_id}")
        ctx_coro = session.get("http://feature-store/context")

        user_resp, item_resp, ctx_resp = await asyncio.gather(
            user_coro, item_coro, ctx_coro
        )

        return FeatureSet(
            user_features=await user_resp.json(),
            item_features=await item_resp.json(),
            context_features=await ctx_resp.json(),
        )

Pattern 2: Async Batching with Timeout

In serving, you might want to collect multiple incoming requests into a batch for efficient GPU inference:

import asyncio
from collections import deque

class AsyncBatcher:
    def __init__(self, max_batch_size: int = 32, max_wait_ms: float = 10.0):
        self.max_batch_size = max_batch_size
        self.max_wait_ms = max_wait_ms
        self.queue: asyncio.Queue = asyncio.Queue()
        self._running = False

    async def predict(self, features: dict) -> dict:
        """Called by each request handler. Waits for batch result."""
        future = asyncio.get_running_loop().create_future()
        await self.queue.put((features, future))
        return await future  # Suspends until batch is processed

    async def batch_loop(self):
        """Background loop that collects requests into batches."""
        self._running = True
        while self._running:
            batch = []
            # Collect items up to max_batch_size or max_wait_ms
            try:
                item = await asyncio.wait_for(
                    self.queue.get(), timeout=self.max_wait_ms / 1000
                )
                batch.append(item)
            except asyncio.TimeoutError:
                continue

            # Drain the queue up to max_batch_size
            while len(batch) < self.max_batch_size and not self.queue.empty():
                batch.append(self.queue.get_nowait())

            # Run batch inference (offload to thread pool)
            features_batch = [item[0] for item in batch]
            futures = [item[1] for item in batch]

            loop = asyncio.get_running_loop()
            results = await loop.run_in_executor(
                None, run_model_batch, features_batch
            )

            # Deliver results back to waiting coroutines
            for future, result in zip(futures, results):
                future.set_result(result)

Pattern 3: Rate-Limited API Calls with Semaphore

When scraping training data or calling external APIs, you need to limit concurrency to avoid being throttled:

import asyncio
import aiohttp

async def fetch_with_limit(
    urls: list[str], max_concurrent: int = 10
) -> list[str]:
    semaphore = asyncio.Semaphore(max_concurrent)
    results = []

    async def fetch_one(url: str) -> str:
        async with semaphore:  # At most max_concurrent at a time
            async with aiohttp.ClientSession() as session:
                async with session.get(url) as resp:
                    return await resp.text()

    results = await asyncio.gather(*[fetch_one(u) for u in urls])
    return results

# Fetch 1000 pages, but only 10 at a time
asyncio.run(fetch_with_limit([f"https://data.example.com/page/{i}" for i in range(1000)]))

Concurrency vs Parallelism

These two words get confused constantly. Let's clear this up once and for all.

Concurrency is about dealing with many things at once. It's structural — your code is organized to handle multiple tasks that make progress in overlapping time periods. A single-core machine can be concurrent. Async programming is concurrency.

Parallelism is about doing many things at once. It's physical — multiple CPU cores execute code simultaneously. multiprocessing is parallelism.

# Concurrency (async) — one thread, overlapping I/O waits
async def concurrent_fetches():
    await asyncio.gather(
        fetch("http://a.com"),  # These three overlap in time
        fetch("http://b.com"),  # but only ONE runs at any given instant
        fetch("http://c.com"),
    )

# Parallelism (multiprocessing) — multiple threads/processes, simultaneous CPU work
from multiprocessing import Pool

def parallel_training():
    with Pool(4) as pool:
        # These four ACTUALLY run at the same time on 4 CPU cores
        results = pool.map(train_model, [fold1, fold2, fold3, fold4])

For ML work, the rule of thumb is: use async for I/O-bound work (network calls, database queries, loading files from remote storage), use multiprocessing for CPU-bound work (training, heavy preprocessing, data transforms). Many production systems use both — an async web server that dispatches inference to a process pool.

Don't Use Async for CPU-Bound ML Code

A common mistake is wrapping CPU-bound model training in async def and expecting speedup. Async only helps when your code is waiting on external resources. If your code is crunching numbers on the CPU, async adds overhead without benefit. Worse, a CPU-intensive coroutine blocks the event loop, starving all other tasks. Use run_in_executor() with a process pool for CPU work, or keep the CPU work in a separate service entirely.
