Used NumPy every day — np.array, .reshape(), broadcasting with prayer — but whenever something broke, tried random transposes until it worked. Every bug I couldn't explain, every one-liner someone wrote that would've taken me twenty lines — discomfort of not understanding grew. Here is that dive.

NumPy (Numerical Python) — created by Travis Oliphant 2005, unifying two earlier array libraries. Single data structure: ndarray. Stores numbers in contiguous memory block, operates in compiled C. Every major ML library — Pandas, scikit-learn, PyTorch, TensorFlow — built on top. model.fit(X), computing loss, normalizing feature matrix → using NumPy whether you realize or not.

Why Arrays Exist

To see why NumPy exists, feel pain of not having it. Three users, four movies, ratings 1-5. Pure Python → list of lists:

ratings = [
    [5, 3, 1, 4],   # Alice
    [4, 5, 3, 2],   # Bob
    [1, 2, 5, 4],   # Carol
]

Looks fine. Want each user's average rating? Write loop, iterate inner list, sum, divide. Three users? Fine. Three million users with ten thousand movies? Loop hurts.

Reason: architectural. Python list ≠ block of numbers. It's collection of pointers, each pointing to separate Python object in memory. Like filing cabinet of sticky notes where each says "actual number in room 4,207 shelf B." To do math: follow pointer, find object, check type, unwrap value, do arithmetic, wrap result into new object, store pointer. For. every. single. element.

NumPy: stores all numbers side by side in one continuous memory block — books spine-to-spine on shelf. No pointers. No wrapping. No type-checking per element. "Add 1 to every element" → single C function walks through memory, adds 1 to each raw number in place. No Python interpreter involved.

import numpy as np

ratings = np.array([
    [5, 3, 1, 4],
    [4, 5, 3, 2],
    [1, 2, 5, 4],
], dtype=np.float32)

# Average rating per user — one line, no loop
avg = ratings.mean(axis=1)
# array([3.25, 3.5 , 3.  ])

axis=1 = "compute mean along axis 1" (columns). Key: no Python loop. Entire computation in compiled C, on contiguous memory, one call. That's entire promise of NumPy.

Four Ingredients of an ndarray

Every NumPy array = exactly four things. Understanding these separates people who use NumPy from people who understand it.

Inspect ratings array:

print(ratings.shape)    # (3, 4)
print(ratings.dtype)    # float32
print(ratings.strides)  # (16, 4)
print(ratings.data)     # <memory at 0x...>

Data pointer — memory address. "Numbers start here." Street address of contiguous byte block.

Shape — tuple telling NumPy how to interpret flat bytes as multi-dimensional structure. (3, 4) = "3 rows, 4 columns." Shape doesn't live in data. It's metadata — label on block.

Dtype — how many bytes each element occupies + how to interpret them. float32 = 4 bytes each. float64 (NumPy default) = 8 bytes — double memory for precision almost never needed in ML.

And strides. This unlocks everything.

Strides — Trick That Makes Everything Work

Strides = the central idea making views, broadcasting, transpose, reshape all work without copying data.

Ratings array stored as 12 numbers side by side in memory:

# In memory (float32 = 4 bytes each):
# [5.0][3.0][1.0][4.0][4.0][5.0][3.0][2.0][1.0][2.0][5.0][4.0]
#  ^--- byte 0          ^--- byte 16         ^--- byte 32

Strides tuple (16, 4): to move to next row → jump 16 bytes. Next column → jump 4 bytes. NumPy uses strides to compute where any element lives.

Element [1, 2] — Bob's rating of third movie? Start at base address, jump 1 × 16 + 2 × 4 = 24 bytes forward. No search. No traversal. Direct computation.

print(ratings[1, 2])  # 3.0 — Bob's rating of movie #3
# NumPy computed: base_address + 1*16 + 2*4 = base + 24 bytes

Why 16 bytes per row? 4 elements × 4 bytes (float32) = 16. Stride = "how many bytes until next element along this axis."

Bookshelf analogy: numbers arranged spine-to-spine on one long shelf. Strides tell you how far to reach for next book in each direction. "Every 4 inches for next column, every 16 inches for next row." Books don't move. You change the rule for which book to grab.

Massive consequence: change strides without changing data → change array shape without copying anything.

Views and Copies — Consequence of Strides

Slice array with basic indexing — ratings[0], ratings[:, 1:3], ratings[::2] — NumPy doesn't copy data. Creates new array object pointing to same memory, with different shape/strides. This = view.

alice = ratings[0]        # Alice's ratings
print(alice.shape)         # (4,)
print(alice.strides)       # (4,)
print(np.shares_memory(alice, ratings))  # True — same data!

Alice's ratings live inside same memory block as full array. NumPy adjusted starting pointer + shape. Modifying alice[0] modifies ratings[0, 0]. Same bytes.

Common bug source. Slice array, modify slice thinking you have own copy, original array changes. Fix: .copy() when you need independence.

alice_copy = ratings[0].copy()   # independent data
alice_copy[0] = 999
print(ratings[0, 0])              # still 5.0 — original untouched

Rule to memorize: Basic slicing (colons, integers) → view. Fancy indexing (list of indices like arr[[0, 2]]) and boolean masking (arr[arr > 3]) → copies. Reason: basic slicing = always expressible as stride change. Fancy indexing picks arbitrary elements, no regular memory pattern.

Unsure? np.shares_memory(a, b) tells truth.

Transpose: ratings.T — NumPy doesn't rearrange data. Swaps strides.

print(ratings.strides)     # (16, 4) — rows are 16 bytes apart, columns 4
print(ratings.T.strides)   # (4, 16) — now columns are 4 bytes apart, rows 16

print(np.shares_memory(ratings, ratings.T))  # True — same data, different strides

Transpose = free. Zero cost. Metadata change. Subtle consequence: transposed array no longer C-contiguous — rows not laid out consecutively. Some operations slower because CPU cache can't predict memory access pattern.

Broadcasting from Scratch

Understanding strides → see how broadcasting works without copying data.

Recommendation engine: 3×4 ratings matrix. Want to center each user's ratings — subtract their mean. Common preprocessing step.

user_means = ratings.mean(axis=1, keepdims=True)
print(user_means.shape)  # (3, 1)
print(user_means)
# [[3.25],
#  [3.5 ],
#  [3.  ]]

centered = ratings - user_means
print(centered.shape)  # (3, 4) — it worked. But how?

Subtracted (3, 1) from (3, 4) → got (3, 4). NumPy "broadcast" means across columns. Behaved as if (3, 1) means repeated 4 times to become (3, 4), but never copied data. Internally, set stride along axis 1 to zero — "move to next column → don't move in memory." Same number, read four times.

Broadcasting rules: applied from rightmost dimension, working left. Compare shapes dim by dim from right. Two dims compatible if: equal, or one is 1. Neither → ValueError. Fewer dimensions → pad shape with 1s on left.

Trace our example:

# ratings shape:     (3, 4)
# user_means shape:  (3, 1)
#
# Rightmost dim: 4 vs 1 → 1 stretches to 4 ✓
# Next dim:      3 vs 3 → equal ✓
# Result shape:  (3, 4)

Another example — add "movie popularity bonus" to each column. 1D array of four bonuses, one per movie:

bonus = np.array([0.5, 0.0, 0.2, 0.1])   # shape (4,)
adjusted = ratings + bonus

# ratings shape:  (3, 4)
# bonus shape:    (4,)    → padded to (1, 4)
#
# Rightmost dim: 4 vs 4 → equal ✓
# Next dim:      3 vs 1 → 1 stretches to 3 ✓
# Result shape:  (3, 4)

Bonus array treated as if leading dimension of 1, stretched to 3 rows. No loop. No copy. One C operation.

Danger: shapes (1000, 1) and (1, 5000) are broadcast-compatible. Both dimensions are 1, both stretch. Result = (1000, 5000) — five million elements from two small arrays. Silently allocates gigabytes when someone forgets keepdims=True.

Print shapes before any binary operation: print(a.shape, b.shape). Single most effective NumPy debugging habit.

Why NumPy Is Fast — Assembly Line Under the Hood

"Runs in C" and "no Python loop" — let's make concrete. Speed difference isn't 2× or 5× — often 100×+. Reason goes deeper than "C faster than Python."

Python for loop over list = craftsman hand-building each item. Every number: look up variable in dict, check type, unbox raw value from Python object, do arithmetic, box result into new object, update reference count for GC. Six-seven operations overhead per one addition.

NumPy = assembly line. a + b on two arrays → Python makes one call into C. C function knows type (no checking — same dtype), knows memory layout (contiguous, predictable), blasts through data in tight loop. Modern CPUs use SIMD (Single Instruction, Multiple Data) — add 4-8 float32 values in single CPU cycle. Assembly line doesn't inspect each item. Processes in bulk.

Matrix operations — A @ B, np.linalg.solve, np.linalg.svd — NumPy delegates to BLAS (Basic Linear Algebra Subprograms) and LAPACK. Libraries optimized for decades, hand-tuned assembly for specific CPU architectures. OpenBLAS and Intel MKL use cache-blocking, multi-threaded execution, instruction-level parallelism no Python code could touch.

Scale up: 10,000 users, 5,000 movies. Want each user's average:

big_ratings = np.random.randn(10000, 5000).astype(np.float32)

# The Python loop way
def mean_loop(X):
    result = np.empty(X.shape[0])
    for i in range(X.shape[0]):
        total = 0.0
        for j in range(X.shape[1]):
            total += X[i, j]
        result[i] = total / X.shape[1]
    return result

# The NumPy way
means = big_ratings.mean(axis=1)

# The loop takes ~15 seconds. NumPy takes ~5 milliseconds.
# That's a 3000× difference. Not a typo.

Gap from all three layers stacking: no interpreter overhead, cache-friendly memory access on contiguous block, SIMD processing multiple values at once. Python loop hits none. NumPy hits all three.

Practical design principle: for loop over array elements in NumPy = almost certainly wrong. Express computation as operations on whole arrays — vectorization. Not because elegant (though is), because only way to access fast path.

Reshaping and Stride Connection

Understanding strides → reshaping becomes transparent. .reshape() tries to find new strides making same flat memory look like different shape. Succeeds → returns view (zero cost). Can't (data not contiguous right way) → copies.

# Our ratings: shape (3, 4), strides (16, 4)
flat = ratings.reshape(12)      # shape (12,), strides (4,)
print(np.shares_memory(flat, ratings))  # True — same data, new strides

# Reshape to (4, 3)
reshaped = ratings.reshape(4, 3)  # shape (4, 3), strides (12, 4)
print(np.shares_memory(reshaped, ratings))  # True — still a view

-1 shorthand: NumPy computes one dimension for you. ratings.reshape(-1, 2) = "give 2 columns, figure out rows." Divides total elements (12) by 2 → 6 rows. Essential when you know one dimension but other depends on data size — batching.

np.newaxis (literally None) adds dimension of size 1 by inserting stride of 0. Tool for making broadcasting work when shapes don't line up.

# user_means has shape (3,) — we need (3, 1) for broadcasting
user_means = ratings.mean(axis=1)          # shape (3,)
user_means_col = user_means[:, np.newaxis]  # shape (3, 1)

# Now we can subtract: (3, 4) - (3, 1) → (3, 4)
centered = ratings - user_means_col

keepdims=True does same thing inline — ratings.mean(axis=1, keepdims=True) directly gives shape (3, 1) instead of (3,). Use keepdims almost always. Makes broadcasting intention explicit, avoids newaxis dance.

Subtlety: .reshape() returns view. .flatten() always returns copy. .ravel() returns view when can, copy when must. Want guaranteed independence → .flatten(). Want speed → .ravel().

.T — transpose — swaps strides. 2D = rows-to-columns. Higher dims: .transpose(2, 0, 1) specifies axis order. Converting image formats: (height, width, channels) → (channels, height, width) = single .transpose(2, 0, 1) — zero copy, instant.

einsum — Universal Tensor Language

np.einsum expresses transpose, batch matrix multiply, trace, outer product — each in single call.

Einstein summation: label each axis of each input tensor with letter. Letters in output → kept. Letters only in input → summed over. That's entire language.

Dot product between Alice's and Bob's ratings — similarity measure:

alice = ratings[0]   # shape (4,)
bob = ratings[1]      # shape (4,)

# The familiar way
similarity = np.dot(alice, bob)   # 39.0

# The einsum way
similarity = np.einsum('i,i->', alice, bob)   # 39.0

Read subscript: both inputs have axis i. Output (after ->) has no axes → i gets summed over. Multiply corresponding elements, sum. = dot product.

All pairwise similarities — 3×3 matrix where [a, b] = dot product of user a with user b:

# The matmul way
sim_matrix = ratings @ ratings.T   # (3, 4) @ (4, 3) → (3, 3)

# The einsum way
sim_matrix = np.einsum('ik,jk->ij', ratings, ratings)   # (3, 3)

String: first array axes i, k. Second array axes j, k. Output keeps i, j, sums over k. = matrix multiplication — einsum makes contraction axis explicit. No mental tracking of transpose dimensions.

Where einsum becomes essential: batch operations. Embeddings for batch of sequences, need attention scores — dot product of every query with every key, separately per batch item:

Q = np.random.randn(8, 10, 64).astype(np.float32)   # (batch, seq_len, d_model)
K = np.random.randn(8, 10, 64).astype(np.float32)

# Without einsum: transpose K, then batch matmul
scores = np.matmul(Q, K.transpose(0, 2, 1))   # (8, 10, 10)

# With einsum
scores = np.einsum('bid,bjd->bij', Q, K)   # (8, 10, 10)

Einsum says exactly what we mean: for each batch b, dot product of query position i with key position j, summing over model dimension d. No transposing, no worrying about axis swaps. Letters are documentation.

When einsum faster vs equivalent to @: simple matrix multiplies → @ delegates to BLAS more directly. Complex multi-axis contractions with batch dims → einsum can be faster (avoids intermediate arrays). optimize=True flag helps find efficient contraction orderings.

Production Patterns and Gotchas

Gap between "can use NumPy" and "can use NumPy in production without burning compute money." Patterns learned hard way.

Cast dtypes early. NumPy defaults to float64. ML frameworks use float32. (100000, 768) embedding in float64 = 587 MB; float32 = 293 MB. Compounds across every operation, intermediate array, gradient. Cast on ingest: X = np.array(data, dtype=np.float32).

# Don't:
X = np.array(data)               # float64 by default
X = X.astype(np.float32)          # wasteful — allocated float64 first, then copied

# Do:
X = np.array(data, dtype=np.float32)   # right from the start

Never grow arrays in loop. Every np.append() or np.concatenate() inside loop allocates new array, copies everything. For n iterations = O(n²). Collect in Python list, convert once at end.

# This is O(n²) and will destroy your performance:
result = np.array([])
for x in stream:
    result = np.append(result, process(x))

# This is O(n):
results = []
for x in stream:
    results.append(process(x))
result = np.array(results, dtype=np.float32)

In-place operations when possible. a += b modifies in place. a = a + b creates new array, points a at it, old one → GC. Large arrays in tight loops → allocation overhead adds up.

Boolean masks compose. Combine masks with & (and), | (or), ~ (not) before applying — rather than filtering multiple times.

# Filter to confident positive predictions
mask = (predictions > 0.9) & (labels == 1)
selected = X[mask]   # one copy, one filter

# Not this — two intermediate arrays:
step1 = X[predictions > 0.9]      # copy
step2 = step1[labels_subset == 1]   # another copy

Understand contiguity. After transpose or non-trivial slice → array may not be C-contiguous. BLAS expects contiguous data; NumPy silently copies before calling BLAS. Repeated matrix ops on transposed data → call np.ascontiguousarray() once up front.

Random numbers: modern API. Legacy np.random.seed(42) sets global state — any imported library could read/write it, making experiments non-reproducible. Modern: create local generator.

rng = np.random.default_rng(seed=42)
batch = rng.choice(dataset, size=256, replace=False)
noise = rng.normal(0, 0.1, size=(256, 768))

default_rng uses PCG64 algorithm — statistically better than old Mersenne Twister, state is local to that object. Pass rng through code → randomness fully controlled and reproducible.

Memory-mapped arrays for datasets exceeding RAM. np.memmap creates array backed by file on disk. Only accessed pages loaded into memory. 50 GB dataset → work as if normal array, OS handles paging.

mmap = np.memmap('embeddings.dat', dtype=np.float32,
                  mode='r', shape=(10_000_000, 768))
batch = mmap[1000:2000]   # only loads these 1000 rows from disk
