CPython internals — memory management, GIL, concurrency — explain why code fast or slow, leaks memory or not, scales or chokes. Dive here.

CPython — interpreter you almost certainly use — is C program. Compiles Python source to bytecode, then runs bytecode on virtual machine. Along way manages memory with reference counting, coordinates threads with global lock, provides event loop for concurrent I/O. Not obscure details. Reason code fast or slow, leaks memory or not, scales or chokes.

What Happens When You Run python script.py

Tiny script. Three lines. Goal: build log ingestion pipeline — read log files, count errors, report. First version tiny on purpose:

logs = ["ERROR: disk full", "INFO: started", "ERROR: timeout"]
errors = [line for line in logs if line.startswith("ERROR")]
print(len(errors))

Hit enter. CPython not run text direct. Pipeline time. First, parser reads source, builds Abstract Syntax Tree — structured representation of code meaning. Then compiler walks tree, produces bytecode — low-level instructions for CPython virtual machine. Finally, VM executes instructions one by one in giant C loop (file called ceval.c, beating heart of interpreter).

Can see bytecode. Python ships module dis — short for disassemble — that shows exact instructions VM executes:

import dis

def count_errors(logs):
    errors = [line for line in logs if line.startswith("ERROR")]
    return len(errors)

dis.dis(count_errors)
#  2   0 LOAD_CONST        1 (<code object <listcomp>>)
#      2 MAKE_FUNCTION     0
#      4 LOAD_FAST         0 (logs)
#      6 GET_ITER
#      8 CALL_FUNCTION     1
#     10 STORE_FAST        1 (errors)
#  3  12 LOAD_GLOBAL       0 (len)
#     14 LOAD_FAST         1 (errors)
#     16 CALL_FUNCTION     1
#     18 RETURN_VALUE

Each line = one bytecode instruction. LOAD_FAST pushes local variable onto stack. CALL_FUNCTION pops arguments from stack, calls callable on top. STORE_FAST takes result, binds it to local name. VM does not know variable names at execution time — works with indices into arrays of names and constants, stored in code object.

That code object gets saved to .pyc file in __pycache__/. Next import of same module skips parsing and compiling, loads cached bytecode direct. Reason first import slower than later ones.

Performance reason: each iteration of Python for loop is several bytecode instructions — load iterator, call __next__, check StopIteration, branch back. Call np.sum(array), and one bytecode instruction (CALL_FUNCTION) drops into C code, rips through millions of elements without touching bytecode loop. Gap between "Python loop over million items" and "one C call over million items" not 2x. It is 100x. Now you know why.

Everything Is a PyObject

C level — where CPython lives — every Python value is C struct called PyObject. Integers, lists, functions, classes, modules, None — all start with same two fields:

typedef struct _object {
    Py_ssize_t ob_refcnt;       // how many references point here
    struct _typeobject *ob_type; // what type am I?
} PyObject;

ob_refcnt is reference count — more on that soon. ob_type is pointer to another object (PyTypeObject) that says what thing is and what operations it supports. Integer extends base struct with field for actual numeric value. List extends it with pointer to resizable array of other PyObject pointers. Top level: all look same to interpreter.

People say "everything in Python is an object." Not philosophy. Literal C struct. Every variable you create in Python is pointer to one of these structs on heap — memory region managed by allocator, not stack. Write a = [1, 2, 3], list struct gets allocated on heap and a points to it. Write b = a, you do not copy list. You create second pointer to same struct. One list, two names.

Key insight. df2 = df does not copy DataFrame because assignment never copies. It creates new pointer to same heap object. Want real copy? Call df.copy(). Same with NumPy: slicing returns view — new object sharing same underlying memory buffer. Modify view, original changes.

Reference Counting — The First Line of Defense

Back to ob_refcnt. Every PyObject carries integer tracking how many things point to it. Create new name pointing to object — count up. Delete name or reassign it — count down. Moment count hits zero, CPython frees memory immediately. No waiting. No background process. Instant reclamation.

Trace log pipeline. Watch it happen:

import sys

logs = ["ERROR: disk full", "INFO: started", "ERROR: timeout"]
print(sys.getrefcount(logs))  # 2 (one for 'logs', one for the getrefcount argument itself)

backup = logs
print(sys.getrefcount(logs))  # 3 (logs + backup + getrefcount arg)

del backup
print(sys.getrefcount(logs))  # back to 2

del logs
# refcount hits 0 → list is freed immediately, along with any strings
# only it was holding alive

Elegant in common case. Most objects get created, used, discarded inside single function. Reference counting handles that with zero garbage-collector delay — memory reclaimed moment object no longer needed. Reason del df on huge DataFrame often frees gigabytes instantly. Count hits zero, memory goes back to allocator. Done.

But system has crack. Imagine two objects pointing to each other: object A references B, B references A. Even if nothing else in program uses either one, both reference counts stay at one. They never hit zero. They sit in memory forever.

The Garbage Collector — Catching What Refcounting Misses

Here CPython's cyclic garbage collector steps in. One job: find and destroy reference cycles reference counting cannot handle.

Collector divides objects into three generations. Generation 0 holds new objects. When collector runs and object survives (still reachable), object moves to Generation 1. Survive again, move to Generation 2. Key insight: most objects die young. Temporary list inside function, string built for log message — created and destroyed in milliseconds. So Generation 0 gets collected often (every ~700 allocations by default), while Generation 2 gets collected rarely.

Mailroom picture. Letters arriving today get checked tonight. Still unclaimed after week, move to shelf checked monthly. Survive month, move to storage room checked yearly. Most letters get picked up same day.

For ML work, know this pattern: some training frameworks call gc.disable() during training to avoid small periodic garbage-collection overhead. Fine if you call gc.collect() between epochs. If not, reference cycles formed during training — and frameworks with complex callback systems can create them — accumulate silently until OOM reaper kills process.

Real bug pattern: training run crashes around epoch 15 — always epoch 15, no matter batch size or model. Memory profiler shows slow steady climb unrelated to model or data. Cause: circular references between callback objects that disabled garbage collector never cleaned up. Fix: gc.collect() at top of each epoch.

CPython's Secret Weapon: The Memory Allocator

Reference counting and garbage collector decide when memory gets freed. Allocator decides how underlying memory blocks get managed. CPython does not use system malloc for most objects. It uses allocator called pymalloc, tuned for Python pattern: millions of tiny objects created and destroyed fast.

Architecture has three levels. Bottom: CPython asks operating system for large chunks called arenas — each 256 KB. Each arena splits into pools of 4 KB. Each pool splits into fixed-size blocks tuned for one object size class. Objects up to 512 bytes go through pymalloc. Bigger ones go straight to system allocator.

Why matter? Explains behavior that confuses everybody first time: Python processes often do not return memory to operating system, even after you del everything. Memory gets freed back to pymalloc pools and arenas, but pymalloc keeps those arenas for future allocations. OS sees process using same RAM. Python sees memory ready for new objects. Not leak. Optimization — OS reallocation is slow.

CPython also uses clever tricks for common objects. Integers from -5 to 256 get pre-allocated at startup and reused forever — common enough that creating fresh heap objects for every 0 and 1 would waste memory. None, True, False are singletons, alive for program lifetime. Variable names and short string literals get interned — stored once in global table and reused by identity, making dictionary lookups faster.

Subtle trap: sys.getsizeof() lies. It reports shallow size of container object itself — internal pointer array for list, hash table for dict — not things inside. sys.getsizeof(my_list) on list of million dictionaries returns maybe 8 MB (pointer array), not hundreds of megabytes dictionaries consume. For real memory numbers, use memory_profiler or df.info(memory_usage='deep') for DataFrames.

The GIL — One Chef, One Kitchen

Kitchen with one stove. No matter how many chefs you hire, only one chef uses stove at time. Others wait. That is Global Interpreter Lock, or GIL — mutex inside CPython ensuring only one thread executes Python bytecode at time.

Reason goes straight back to ob_refcnt. Every object has reference count that changes constantly — every assignment, every function call, every scope exit. If two threads decrement same ob_refcnt at same time without coordination, count can go wrong. Object freed while still in use. Segfault. Memory corruption. GIL prevents this by serializing all bytecode execution through one lock.

C level: GIL lives in ceval.c beside bytecode evaluation loop. It is mutex with condition variable. Thread holds GIL, executes bytecode, and after configurable interval — about 5 milliseconds by default, tunable via sys.setswitchinterval() — checks whether another thread waits. If yes, thread releases lock, signals other thread, waits to reacquire. Handoff controlled by gil_drop_request flag so unnecessary context switches do not happen when no other thread cares.

Could CPython use fine-grained per-object locks instead? Theory: yes. Practice: adding lock to every PyObject slows single-threaded case — overwhelmingly common case — by 30-40%. Tradeoff got rejected decades ago and re-checked several times. Single coarse lock keeps common path fast.

Kitchen analogy goes further. Chef can step away from stove to check delivery (I/O), and while chef waits, another chef uses stove. Same thing here: GIL gets released during I/O operations. Network calls, disk reads, database queries — thread doing I/O drops GIL, waits for OS response, reacquires GIL when data arrives. Not loophole. Design. I/O-bound multithreaded Python works well because stove stays free while chefs wait for deliveries.

Where the GIL Doesn't Matter (and Where It Bites)

Thing most people miss about GIL: for most ML work, it does not matter. NumPy, PyTorch, TensorFlow, XGBoost all release GIL when they enter C/C++/CUDA kernels. When NumPy crunches matrix multiplication, GIL gets released. When PyTorch runs forward pass on GPU, GIL gets released. model.fit() in scikit-learn with n_jobs=-1 uses multiprocessing internally. GIL is not training bottleneck.

Where it does bite: pure Python loops doing computation. If you wrote for loop that processes ten million rows in Python — not vectorized through NumPy or Pandas — threading will not help. Threads take turns on GIL, one at time, maybe slower than single thread because lock acquisition and context switching cost extra. Like hiring three chefs for one stove — one stove throughput, minus time spent deciding who goes next.

Practical test: simple. If you profile multi-threaded Python program and CPU usage never goes above 100% (one core), you hit GIL. If it goes above 100%, you are either in C extension code that releases GIL or using multiprocessing.

Threading, Multiprocessing, and Picking the Right Tool

Expand log ingestion pipeline. Now need to process 10,000 log files: download from remote server, parse each file for error patterns (CPU-intensive regex work), push results to database. Three stages. Three bottleneck profiles. Here picking right concurrency tool stops being theory.

Stage 1: Downloading — I/O-bound. GIL gets released while waiting for network responses, so threads work perfectly. Twenty threads. Twenty concurrent downloads. Each thread spends 95% of time waiting for data, and GIL stays free during wait.

from concurrent.futures import ThreadPoolExecutor
import requests

def download_log(url: str) -> str:
    """I/O-bound: network wait dominates. GIL is released during .get()."""
    return requests.get(url, timeout=30).text

urls = [f"https://logs.example.com/files/{i}.log" for i in range(10_000)]

with ThreadPoolExecutor(max_workers=20) as pool:
    raw_logs = list(pool.map(download_log, urls))
# 20 downloads happening concurrently. Wall time: ~500x faster than sequential.

Stage 2: Parsing — CPU-bound. Regex over megabytes of text is pure computation. Threads would take turns on GIL. Need separate processes, each with own interpreter and own GIL.

from concurrent.futures import ProcessPoolExecutor, as_completed
import re

ERROR_PATTERN = re.compile(r"^(ERROR|CRITICAL):.*", re.MULTILINE)

def parse_log(text: str) -> list[str]:
    """CPU-bound: regex matching across the full text."""
    return ERROR_PATTERN.findall(text)

with ProcessPoolExecutor(max_workers=8) as pool:
    futures = {pool.submit(parse_log, log): i for i, log in enumerate(raw_logs)}
    results = {}
    for future in as_completed(futures):
        idx = futures[future]
        try:
            results[idx] = future.result()
        except Exception as e:
            results[idx] = f"FAILED: {e}"
# 8 processes, true parallelism. One crashed file doesn't take down the others.

Key design choice: submit() with as_completed() instead of map(). With map(), you wait for results in order — one slow file blocks everything behind it. With as_completed(), you handle results as they finish, and one failure does not sink batch.

Beautiful part of concurrent.futures: ThreadPoolExecutor and ProcessPoolExecutor share exact same interface. Start with threads. If profiling shows CPU bottleneck, swap Thread for Process — one-word change. Good standard-library design pays off.

Pickle Trap

Multiprocessing sends data between processes using pickle serialization. Send 500MB DataFrame to 8 workers, and you burn 8 copies × 500MB = 4GB of memory on serialization alone. Fix: do not send data. Send file paths or row indices. Let each worker load own slice. Difference between pipeline that scales and pipeline that gets OOM-killed at 16 workers.

Asyncio — Cooperative Multitasking from Scratch

Threads work well for tens or hundreds of concurrent I/O tasks. But what if you need thousands? Web server handling 5,000 concurrent inference requests. Scraper hitting 10,000 endpoints. Each OS thread uses memory (typically ~8 MB of stack space), and context-switching across thousands of threads gets expensive. Need different approach.

Asyncio is Python's answer: cooperative multitasking on single thread. Instead of operating system preemptively switching threads, coroutines voluntarily yield control at await points. One thread runs event loop that juggles thousands of tasks by resuming whichever one has data ready.

To see under hood, know this: coroutines grew out of generators. Generator pauses at yield and resumes when you call next(). Coroutine (defined with async def) pauses at await and resumes when event loop decides it is ready. Under C hood, both are state machines — function local variables and instruction pointer get saved on heap (not stack), so resumption is cheap.

Event loop itself sits on operating system I/O multiplexing primitives — epoll on Linux, kqueue on macOS, IOCP on Windows. These let one thread ask OS: "wake me when any of these 5,000 sockets has data." Loop calls selector.select(), blocks until events arrive, then resumes matching coroutines. No polling. No spinning. No wasted CPU cycles.

Rule of thumb: if you manage more than few hundred concurrent I/O operations, or need fine-grained control over cancellation and timeouts, async is worth cognitive overhead. For simpler workloads, threads are easier to reason about and work fine.

Production pattern in ML serving: async for HTTP traffic, process pool for actual computation:

import asyncio
from concurrent.futures import ProcessPoolExecutor
from fastapi import FastAPI
import numpy as np

app = FastAPI()
executor = ProcessPoolExecutor(max_workers=4)

def predict_sync(features: list[float]) -> float:
    """CPU-bound inference — cannot be awaited directly."""
    X = np.array(features).reshape(1, -1)
    return float(X.sum() * 0.42)  # stand-in for model.predict(X)[0]

@app.post("/predict")
async def predict(features: list[float]):
    loop = asyncio.get_event_loop()
    result = await loop.run_in_executor(executor, predict_sync, features)
    return {"prediction": result}

# The event loop handles thousands of concurrent HTTP connections.
# run_in_executor offloads each prediction to a separate process.
# The loop is never blocked. That's the whole trick.

run_in_executor bridges two worlds: async event loop stays responsive to incoming requests, while CPU-bound work happens in separate processes where GIL does not matter. Without this, one slow prediction blocks every other request in server. With it, chef keeps taking orders while prep cooks do heavy work in their own kitchens.

The Future: Free-Threaded Python

Python 3.13, released in October 2024, ships with experimental free-threaded build — version of CPython where GIL can be disabled entirely. Result of PEP 703: years of work making reference counting thread-safe without global lock, using biased reference counting, where each object tracks its "owning" thread separately from other threads.

Tradeoffs are real. Single-threaded code runs about 10% slower in free-threaded build, because more complex reference counting operations — atomic increments, deferred decrements — add overhead GIL previously avoided. Multi-threaded CPU-bound code, on other hand, can finally achieve true parallelism, with dramatic speedups on multi-core machines.

As of mid-2025, ecosystem is catching up. NumPy and pandas are being updated. Many C extensions that relied on GIL for thread safety need auditing and modification. Build is opt-in, and if extension has not been updated, importing it silently re-enables GIL with warning.

Honest assessment: do not redesign production systems around it yet. Use multiprocessing for CPU-bound parallelism today — it works, battle-tested, and keeps working regardless of what happens with GIL. Watch ecosystem tracking pages. When libraries you depend on are certified free-threading-safe, then switch. Probably late 2025 earliest for adventurous teams, 2026 for most teams.
