UTF-8 — How Text Becomes Bytes

Integers, floats: CPU handles natively. Text is different. Character isn't a number in hardware sense; CPU has no idea what "A" is. Everything is bytes. Every character must map to byte sequence. Encoding that rules modern internet — appears constantly in ML preprocessing, file I/O, tokenizer code — is UTF-8.

The Problem: 256 Slots Aren't Enough

ASCII (1963) solved original problem: map English alphabet, digits, control chars to 7-bit codes. 128 slots — enough for teletype. Extended ASCII stretched to 8 bits (256 slots) for Western European accented chars. Then rest of world showed up: Arabic, Chinese, Japanese, Korean, Cyrillic, emoji, math symbols. Over 140,000 chars in use today. One byte cannot hold them.

Solution: Unicode — universal catalog assigning every character a code point: integer in range U+0000 to U+10FFFF (~1.1M slots, ~150K assigned). Unicode answers "what number does this character get?" Not "how to store in memory?" That's where UTF-8 comes in.

UTF-8: Variable-Width Encoding

UTF-8 (designed by Ken Thompson and Rob Pike in 1992, on a placemat in a New Jersey diner) encodes each code point as 1–4 bytes using self-describing bit pattern. Key insight: first byte tells you total byte count.

Bit pattern is self-synchronizing by design. Land mid-stream in UTF-8 file — always tell start byte vs continuation byte: start bytes begin with 0 (ASCII) or 11; continuation bytes always begin with 10. Scan backwards from any position to find character boundary. UTF-16 lacks this for two-unit sequences — one of many reasons UTF-8 won.

Concrete example: Euro sign € = code point U+20AC (decimal 8364). Binary: 0010 0000 1010 1100 — 14 significant bits → 3-byte range. Packing into 1110xxxx 10xxxxxx 10xxxxxx template:

Code point:  U+20AC = 0010 0000 1010 1100
Split:       xxxx = 0010 | xxxxxx = 000010 | xxxxxx = 101100
Bytes:       1110 0010 | 10 000010 | 10 101100
             = 0xE2     | 0x82      | 0xAC

Peel off framing bits (1110, 10, 10) and you recover original code point. That's all text editor, HTTP server, Python runtime do when handling non-ASCII text.

Python: str Is Not Bytes

Python 3 made clean separation Python 2 bungled: str = sequence of Unicode code points. bytes = sequence of raw octets. Completely different types. Move between them: encode (str → bytes) or decode (bytes → str), always specify encoding.

s = "café"          # str: 4 code points — U+0063 U+0061 U+0066 U+00E9
b = s.encode("utf-8")
print(b)            # b'caf\xc3\xa9'   — 5 bytes (é is 2 bytes in UTF-8)
print(len(s))       # 4  (code points)
print(len(b))       # 5  (bytes)

# Decode back
print(b.decode("utf-8"))   # 'café'

# Wrong encoding → garbage or error
b.decode("latin-1")        # 'cafÃ©'  — bit pattern reinterpreted incorrectly
b.decode("ascii")          # UnicodeDecodeError: byte 0xc3 out of range

len() discrepancy — 4 chars but 5 bytes — source of ML preprocessing bugs. Naive text[:max_len] truncation operates on bytes, not chars. Can cut multi-byte sequence in half → invalid UTF-8. Always decode to str first, operate on code points, encode back if needed.

BOM and UTF-16

Files sometimes have BOM (Byte Order Mark) — 2–3 byte header signaling encoding and byte order. UTF-8 BOM is 0xEF 0xBB 0xBF — almost never needed (UTF-8 has no endianness issue) but Windows tools emit it. Use encoding='utf-8-sig' in Python to strip silently. UTF-16 requires BOM (or specify utf-16-le / utf-16-be) because each char is 2 bytes and byte order matters. UTF-8 is right default for all new work.

Why This Matters for ML

UTF-8 appears in three places. First, reading data: always open text files with open(path, encoding='utf-8') (Python's default is locale-dependent on Windows — silent disaster on cloud servers). Second, tokenizers: BPE tokenizers like GPT-2 operate on UTF-8 bytes, not chars. Vocabulary maps byte sequences to token IDs. "byte_level": true config = UTF-8 byte level — 256 byte values become base vocabulary. Third, string length heuristics: comparing lengths, measuring tokens, slicing context windows — always know whether counting code points or bytes.

import json

# Reading a dataset — always specify encoding
with open("dataset.jsonl", encoding="utf-8") as f:
    records = [json.loads(line) for line in f]

# Counting characters correctly
text = "こんにちは"   # 5 Japanese characters
print(len(text))                    # 5  (correct: code points)
print(len(text.encode("utf-8")))    # 15 (3 bytes each in UTF-8)

# Byte-level BPE: each byte is a valid "token" slot
vocab_base = {bytes([i]): i for i in range(256)}   # 256-entry base vocab

UTF-8 works invisibly until user submits emoji, or you concatenate str and bytes, or 512-token context window silently truncates Japanese mid-character. Understand bit layout → nothing mysterious. Just bytes with well-defined framing scheme.

Floating-Point — Where Things Get Weird

Integers are clean — every integer has exact binary representation. Floats are different. Most ML precision bugs come from here.

Floating-point follows IEEE 754 (1985, updated 2008). Idea borrowed from scientific notation: instead of writing 6,022,000,000,000,000,000,000,000, scientists write 6.022 × 1023. Computer does same in binary.

32-bit float (float32) has three fields:

Sign bit (1 bit): positive or negative. Exponent (8 bits): scale. Mantissa (aka significand/fraction, 23 bits): precision digits. Hidden trick: leading 1 before decimal is implicit (not stored) → effectively 24 bits of precision. Value: (-1)sign × 2(exponent - 127) × 1.mantissa.

Concrete example: 6.5 in float32:

6.5 in binary: 110.1 (4 + 2 + 0.5). Normalized: 1.101 × 22. Sign bit: 0 (positive). Exponent: 2 + 127 = 129 = 10000001 (127 bias lets negative exponents work without separate sign). Mantissa: 10100000000000000000000 (101 after implicit leading 1, padded with zeros).

This has fundamental consequence that bites people daily:

>>> 0.1 + 0.2
0.30000000000000004

0.1 in decimal = infinitely repeating fraction in binary (0.0001100110011...), like 1/3 = 0.333... in decimal. Mantissa stores only 23 bits (float32) or 52 bits (float64) → gets rounded. Error tiny but real. In ML, rounding errors accumulate over millions of gradient updates.

Common float types in ML:

TypeBitsExponentMantissaPrecision (decimal digits)Range
float16 (half)16510~3.3±65,504
bfloat161687~2.4±3.4×1038
float32 (single)32823~7.2±3.4×1038
float64 (double)641152~15.9±1.8×10308

bfloat16 — Google Brain invented for deep learning. Same exponent range as float32 (same magnitudes, critical for gradient spikes), but fewer mantissa bits (less precision). Tradeoff: lose small-digit precision but almost never overflow. Standard float16 overflows at 65,504 — gradient spike pushes past that, loss becomes inf, then NaN, training run dead.

Three IEEE 754 special values: inf (exponent all 1s, mantissa all 0s), -inf, NaN (exponent all 1s, any non-zero mantissa). NaN is only value in computing not equal to itself: NaN != NaN is True. By design — signals something went wrong.

import torch
x = torch.tensor(1e38, dtype=torch.float16)
print(x)  # tensor(inf) — overflowed

y = torch.tensor(1e38, dtype=torch.bfloat16)
print(y)  # tensor(1.0000e+38) — same range as float32, no overflow

General rule in modern ML: train in bfloat16 or mixed precision (float32 master weights, bfloat16 forward/backward), accumulate gradients in float32, quantize to int8 or int4 for inference. Optimal recipe changes with each GPU architecture.

How Bytes Sit in Memory

Computer memory: long flat array of bytes, each with numeric address. 4-byte float32 occupies four consecutive addresses. Quirk: which byte goes first?

Big-endian: most significant byte first. Little-endian: least significant byte first. Most modern CPUs (x86, ARM) use little-endian. Network protocols use big-endian. Rarely matters in Python — until reading binary model weight files from different architecture or interfacing with CUDA memory directly.

import struct
val = 6.5
packed = struct.pack('f', val)    # pack as float32
print([f'0x{b:02x}' for b in packed])
# ['0x00', '0x00', '0xd0', '0x40'] — little-endian on x86

Raw material: bits grouped into bytes, interpreted as integers or floats by specific rules, at addresses in flat memory. Python doesn't expose this. No malloc(4), no byte order in normal Python code. Python wraps every value in an object. Understanding what object is at C level — key to understanding Python behavior.

What Is an Object, Really? — From C Structs to Python Names

Variables-as-boxes model is wrong in Python. Source of persistent production bugs. Need to go one level deeper — into C code Python is written in.

CPython — The Interpreter You're Actually Running

Type python in terminal → running CPython — reference implementation, written in C. Other implementations exist (PyPy, Jython, GraalPy) but CPython ships by default, runs pip packages and PyTorch loops. Every Python object is C struct allocated on heap.

Root of everything: PyObject, C struct in Include/object.h. Every Python object — integers, strings, lists, custom classes, None — starts with this structure:

// Simplified from CPython source (Include/object.h)
typedef struct _object {
    Py_ssize_t ob_refcnt;    // reference count
    PyTypeObject *ob_type;   // pointer to the type object
} PyObject;

Two fields. Every Python object begins with reference count (how many names point to it) and type pointer (tells Python what operations are supported). Python integer isn't raw 42 in register. It's PyLongObject — struct with PyObject header plus fields for actual numeric value:

// Simplified from CPython source
typedef struct {
    PyObject ob_base;        // refcount + type pointer
    Py_ssize_t ob_size;      // number of "digits"
    uint32_t ob_digit[1];    // flexible array of 30-bit digits
} PyLongObject;

Python integers have no size limit — stored as arrays of "digits" that grow. 42 uses one digit. Thousands of decimal digits use hundreds. Tradeoff: 4-byte C integer takes ≥28 bytes in Python (PyObject header + size + digit + allocator overhead). That's cost of flexibility.

Names Are Post-It Notes, Not Boxes

Here's how it all connects. When you write:

a = [1, 2, 3]

Python does three things. Creates PyListObject on heap — C struct with PyObject header, pointer to array of PyObject* pointers, and length. Creates (or finds) PyLongObjects for 1, 2, 3. Binds name a — in current namespace dict (strings → object pointers) — to list object's address.

Name a doesn't contain list. It points to it. Post-it note on object, not box holding it. If you write:

b = a

No copy. No new PyListObject. Name b points to same object as a. List's ob_refcnt goes from 1 to 2. Two names, one object.

Concrete experiment tracker example:

best_run = results[1]       # {"lr": 0.001, "epochs": 20, "accuracy": 0.91}
alias = best_run

One dictionary exists. Single PyDictObject on heap. best_run, alias, results[1] — all store same pointer to that one dict. Modify through any name → every other name sees change:

alias["accuracy"] = 0.95
print(best_run["accuracy"])  # 0.95 — same object, remember

This is aliasing — multiple names referring to same object. Verify with id(): in CPython returns memory address of underlying PyObject:

print(id(best_run) == id(alias))  # True — same PyObject address
print(best_run is alias)          # True — 'is' checks identity, not equality

is checks if two names point to same C struct — identity. == checks same value (calls __eq__). Different questions. Separate dicts with identical contents: == True, is False.

Why This Design? Why Not Boxes?

In C, variable is box — named memory region, fixed address, fixed size. int x = 5; allocates 4 stack bytes, writes 5 directly. y = x copies 4 bytes. Fast and memory-efficient, but type and size frozen at compile time.

Python chose differently. Everything heap-allocated PyObject referenced by name → dynamic typing (name x can point to int now, string later — type stored in object), automatic memory management (refcount hits zero → object freed, no free() calls), everything first-class (functions, classes, modules all passable objects). Cost: every integer carries 28+ byte wrapper. For orchestration language delegating heavy work to C/CUDA, tradeoff is worth it.

Mutability — The Other Half of the Story

Not all objects allow content changes. Hard line: mutable types change in-place; immutable types create new object on any change.

Lists, dicts, sets: mutable. Integers, floats, strings, tuples: immutable. x = 5; x = x + 1 — not modifying integer 5. Creating new integer 6, rebinding x. Old object 5 still exists (refcount drops by 1), freed when nothing points to it.

a = [1, 2, 3]
print(id(a))       # 0x7f3... — address of the list object
a.append(4)
print(id(a))       # 0x7f3... — SAME address! Mutated in place.

s = "hello"
print(id(s))       # 0x7f4... — address of the string object
s = s + " world"
print(id(s))       # 0x7f5... — DIFFERENT address. New string created.

x is None is correct — never x == None. One None in all of Python — singleton PyObject. Identity check: pointer comparison (one CPU instruction). Equality check: calls __eq__, could do anything on custom class.

Complete foundation: every Python value is PyObject on heap, names are namespace dict entries storing pointers, mutability determines in-place change vs new object. Choice of list vs set vs dict isn't stylistic — it's mechanical. What's actually inside these containers?

Data Structures — What's Actually Happening Inside the Four Containers

Four core container types. Choice isn't stylistic — real performance consequences rooted in C-level construction. 14-hour pipeline vs 20-minute one: list membership check vs set check. Not magic. Architecture.

List — A Dynamic Array of Pointers

Python list is not linked list. It's dynamic array — contiguous memory block holding pointers to PyObjects. C struct:

// Simplified from CPython (Include/cpython/listobject.h)
typedef struct {
    PyObject ob_base;
    Py_ssize_t ob_size;       // current number of elements
    PyObject **ob_item;       // pointer to array of PyObject pointers
    Py_ssize_t allocated;     // capacity of the array
} PyListObject;

Key field: ob_item — pointer to C array of PyObject* pointers. Each pointer 8 bytes (64-bit). Elements live elsewhere on heap; list stores only addresses. Indexing O(1): compute ob_item + (i × 8), dereference. One multiply, one memory access.

Appending is clever. Array fills → Python over-allocates: 0, 4, 8, 16, 24, 32, 40, 52, 64, ... Each resize allocates more than needed → future appends skip allocation. append is amortized O(1) — O(1) on average, occasional resize copies entire array.

Searching — if x in my_list — is O(n). Walk every pointer, dereference, call __eq__. Million elements = million dereferences and comparisons. That 14-hour deduplication pipeline: if item in seen_list inside loop over millions of items = O(n) × O(n) = O(n²).

Set — A Hash Table That Changes Everything

Python set is hash table — same structure that makes dicts fast, but keys only, no values. Why if x in my_set is O(1) while if x in my_list is O(n): hashing.

Hash function maps hashable object to integer. Python's hash(): hash("hello") → large integer, hash(42) → 42, hash((1, 2)) → integer. Hash computed from content. Equal values → same hash (required). Different objects can share hash = collision (pigeonhole principle: infinite values mapped to finite integers).

Set uses hashing for O(1) lookup. Maintains array of slots (8 to start). Add element: compute hash(element) % num_slots → slot index → store pointer. Check element in my_set: same hash → same slot → check equality. No scanning. One hash computation, one (or few, on collision) slot checks.

# What "x in my_set" actually does under the hood:
# 1. Compute hash(x) → some integer, say 738291
# 2. Compute 738291 % table_size → slot index, say 3
# 3. Look at slot 3:
#    - Empty? → x is not in the set. Done.
#    - Occupied? → Compare stored element with x using ==
#      - Equal? → x IS in the set. Done.
#      - Not equal? → Collision. Probe next slot. Repeat.

Collision (two elements same slot): Python uses open addressing — probes next slots until empty found. Table stays under 2/3 full (CPython resizes then). Expected probes constant → average lookup O(1).

Tradeoff: only hashable objects in set — immutable types (integers, strings, tuples of hashable elements). Lists: mutable → unhashable. Mutate list after adding → hash changes → set loses it. Python prevents this.

# The 14-hour pipeline fix:
seen_list = []                    # O(n) membership check
seen_set = set()                  # O(1) membership check

for item in million_items:
    if item not in seen_set:      # O(1) — hash, jump, done
        seen_set.add(item)
        process(item)

O(n²) vs O(n). 14 hours vs 20 minutes.

Dict — A Hash Table with Values Attached

Python dict = set storing key-value pairs. Since Python 3.7: preserves insertion order. CPython achieves this with compact dictionary design (Python 3.6+):

Two arrays: hash table (sparse array of indices) and dense array of (hash, key, value) entries in insertion order. d["lr"]: compute hash("lr") → sparse table → dense array index → key-value pair. O(1) average.

# What d["lr"] = 0.01 does internally:
# 1. hash("lr") → 2847291
# 2. 2847291 % sparse_size → slot 5 in hash table
# 3. sparse[5] = 0  (index into dense array)
# 4. dense[0] = (2847291, "lr", 0.01)
#
# Lookup d["lr"]:
# 1. hash("lr") → 2847291
# 2. 2847291 % sparse_size → slot 5
# 3. sparse[5] = 0 → check dense[0].key == "lr" → yes!
# 4. Return dense[0].value → 0.01

Compact design uses ~25% less memory than pre-3.6: sparse table stores small integers (indices) not full entries, dense array has no gaps.

Dicts: natural choice for experiment records — fast key lookup, add/remove fields freely.

Tuple — The Immutable, Cached, Hashable Record

tuple is simplest container: fixed-size PyObject* pointer array, allocated once, never resized. No over-allocation, no growth, no hash table. Lean C struct:

// Simplified
typedef struct {
    PyObject ob_base;
    Py_ssize_t ob_size;
    PyObject *ob_item[1];    // fixed array, inline
} PyTupleObject;

Immutable → Python optimizes aggressively. Small tuples (length 0–1) cached and reused. () is singleton. Tuples of immutable elements are hashable → usable as dict keys or set elements. Useful for compound keys:

# Track which hyperparameter combos you've tried
tried = set()
tried.add((0.01, 10))      # (lr, epochs) as tuple — hashable ✓
tried.add((0.001, 20))
print((0.01, 10) in tried)  # True, O(1) lookup

List won't work: tried.add([0.01, 10]) → TypeError: unhashable type: 'list'.

Tuples use less memory than lists — no allocated field, no separate pointer array indirection. Elements stored inline in struct. For returning multiple values — return loss, accuracy — tuples are idiomatic. Python creates tuple behind scenes.

Containers: no longer black boxes. But so far data fits in memory. 50GB dataset? Can't load all at once. Need piece-by-piece, on-demand loading. Lazy iteration meets real-world ML infrastructure.

PyTorch DataLoader — How Your Training Loop Actually Loads Data

Most use DataLoader with surface-level mental model: pass dataset, get batches, num_workers > 0 = faster. Understanding internals crucial for debugging production deadlocks and bottlenecks.

Core idea: lazy iteration. 50GB dataset → load one batch at a time. Each training iteration: fetch 32 (or 64, or 128) samples, process, discard, fetch next. Memory stays O(batch_size), not O(dataset_size).

The Two Contracts: Dataset and DataLoader

Two abstractions: Dataset defines what data is and how to access samples. DataLoader defines how to iterate — batching, shuffling, parallelism.

Map-style Dataset implements two methods:

import torch
from torch.utils.data import Dataset, DataLoader

class ExperimentDataset(Dataset):
    def __init__(self, records):
        self.records = records

    def __len__(self):
        return len(self.records)

    def __getitem__(self, idx):
        r = self.records[idx]
        features = torch.tensor([r["lr"], r["epochs"]], dtype=torch.float32)
        target = torch.tensor(r["accuracy"], dtype=torch.float32)
        return features, target

__len__: tells DataLoader sample count. __getitem__: fetches one sample by index. That's entire contract. DataLoader calls nothing else.

What Happens Inside the DataLoader

for batch in loader — what actually happens:

dataset = ExperimentDataset(results)
loader = DataLoader(dataset, batch_size=2, shuffle=True)

for features, targets in loader:
    print(features.shape, targets.shape)
    # torch.Size([2, 2]) torch.Size([2])

for loop calls iter(loader) → creates _SingleProcessDataLoaderIter (or _MultiProcessingDataLoaderIter if num_workers > 0). Iterator holds state for one pass through dataset.

Four stages per batch: Sampler generates index permutation (shuffled for training, sequential for validation). BatchSampler chunks into batch_size groups. Fetcher calls dataset[idx] for each index. Collate (default_collate) stacks samples into batched tensors.

The num_workers Machinery

num_workers=0 (default): everything in main process. Fetching calls dataset[idx] synchronously — GPU sits idle while CPU loads/preprocesses. Common bottleneck: GPU waiting for data.

num_workers > 0: PyTorch spawns worker processes (not threads — GIL negates benefit for CPU-bound work). Each worker has own process with own dataset copy. Main process sends batch indices via multiprocessing.Queue, workers fetch and preprocess, send results back. Main process consumes from results queue.

# What num_workers=4 looks like internally:
#
# Main process:   [send indices] → Queue → [receive batches] → GPU
#                                    ↑           ↑
# Worker 0: dataset[idx] → preprocess → result queue
# Worker 1: dataset[idx] → preprocess → result queue
# Worker 2: dataset[idx] → preprocess → result queue
# Worker 3: dataset[idx] → preprocess → result queue
#
# Workers prefetch: while GPU processes batch N,
# workers prepare batch N+1, N+2, ...

prefetch_factor (default: 2): batches each worker prepares ahead. 4 workers + prefetch_factor=2 → up to 8 batches queued while GPU processes current. Called double buffering (multi-buffering). Key to hiding data loading latency behind GPU computation.

Gotchas: each worker gets fork/spawn of main process + dataset copy. Large in-memory dataset × num_workers = multiplied memory. Fix: memory-mapped files or lazy loading in __getitem__. Random state not automatically different across workers — data augmentation with random/numpy.random needs worker_init_fn seeding each worker differently, or identical augmentations across workers.

def worker_init_fn(worker_id):
    seed = torch.initial_seed() % 2**32
    import numpy as np
    np.random.seed(seed + worker_id)

loader = DataLoader(dataset, batch_size=32, num_workers=4,
                    worker_init_fn=worker_init_fn)

pin_memory=True allocates batch tensors in page-locked (pinned) memory — faster CPU-to-GPU transfers via DMA. Almost always worth enabling on GPU, but increases host memory. Can cause OOM if batches large and many prefetched batches in flight.

DataLoader's lazy batched iteration: reason training on datasets larger than RAM is possible. Containers and iteration patterns share one untouched thing — redefining what +, [], len() mean for custom objects. That mechanism has a name.

Operator Overloading — Teaching Objects New Tricks

Every a + b in Python is function call. Python translates to a.__add__(b). + isn't hardcoded to "add numbers" — calls __add__ on left operand with right as argument. What that method does: entirely up to object.

Operator overloading — define what operators mean for custom types. Python uses dunder methods ("double underscore"). Seen some already: __len__ for len(), __getitem__ for []. Full set is large; here are ones you'll encounter in ML:

Concrete example — experiment results with comparison and arithmetic:

class ExperimentResult:
    def __init__(self, lr, epochs, accuracy):
        self.lr = lr
        self.epochs = epochs
        self.accuracy = accuracy

    def __repr__(self):
        return f"Exp(lr={self.lr}, epochs={self.epochs}, acc={self.accuracy:.3f})"

    def __gt__(self, other):
        return self.accuracy > other.accuracy

    def __eq__(self, other):
        return self.accuracy == other.accuracy

    def __add__(self, other):
        # Average of two experiments
        return ExperimentResult(
            lr=(self.lr + other.lr) / 2,
            epochs=(self.epochs + other.epochs) // 2,
            accuracy=(self.accuracy + other.accuracy) / 2
        )

exp1 = ExperimentResult(0.01, 10, 0.82)
exp2 = ExperimentResult(0.001, 20, 0.91)

print(exp2 > exp1)      # True — __gt__ compares accuracy
print(sorted([exp1, exp2]))  # [Exp(lr=0.01, epochs=10, acc=0.820), ...]
print(exp1 + exp2)       # Exp(lr=0.006, epochs=15, acc=0.865)

sorted() works on custom objects because we defined __gt__. Python sort needs only pair comparison — dunders provide that interface.

The __call__ Method — Why model(x) Works

Most important dunder in PyTorch. output = model(input) → model.__call__(input). In nn.Module, __call__ runs hooks, handles autograd setup, manages train/eval mode, then calls forward. Never call model.forward(input) directly — skips all that machinery.

import torch.nn as nn

class SimpleModel(nn.Module):
    def __init__(self, input_dim, output_dim):
        super().__init__()
        self.linear = nn.Linear(input_dim, output_dim)

    def forward(self, x):
        return self.linear(x)

model = SimpleModel(2, 1)
x = torch.tensor([[0.01, 10.0]])
output = model(x)      # calls model.__call__(x) → runs hooks → calls forward(x)
print(output)           # tensor([[...]], grad_fn=<AddmmBackward0>)

__matmul__ (@) added in Python 3.5 for scientific computing. Before: np.dot(A, B) or A.dot(B) — unreadable in chains. Now: output = W @ x + b. Linear layer in one line.

The Reverse Operators

Subtlety: 2 * tensor → first tries int.__mul__(2, tensor). Integer can't multiply tensor → returns NotImplemented. Python tries reflected operator: tensor.__rmul__(2). PyTorch defines __rmul__ → works. Without this: always write tensor * 2, never 2 * tensor.

x = torch.tensor([1.0, 2.0, 3.0])
print(2 * x)    # tensor([2., 4., 6.]) — int.__mul__ fails → tensor.__rmul__ succeeds
print(x * 2)    # tensor([2., 4., 6.]) — tensor.__mul__ succeeds directly

Operator overloading makes PyTorch feel like math. loss = (predictions - targets) ** 2 works because tensors define __sub__ and __pow__. Every operator: method call that builds computation graph for autograd. Dunders all the way down.

Objects define own behavior for built-in operations. Passed around, stored, combined. Most fundamental unit: function. In Python, functions are objects.

Functions Are Objects

In Python, function is object — instance of function type, heap-allocated PyObject. Has attributes (__name__, __doc__, __defaults__). Assign to variable, put in list, pass as argument, return from function.

Property: first-class citizen. Foundation of closures, higher-order functions, callbacks.

def square(x):
    return x * x

def apply(func, value):
    return func(value)

print(apply(square, 5))   # 25 — we passed a function as an argument

Used this in sorted(results, key=lambda r: r["accuracy"]). key accepts function. lambda — anonymous function expression — passed as sorting criterion. sorted calls it on each element.

lambda = shorthand for single-expression function. lambda r: r["accuracy"] exactly equivalent to:

def get_accuracy(r):
    return r["accuracy"]

Only difference: lambda doesn't bind name. Creates function object, hands it over unnamed. Use lambda for short throwaway functions. Use def for anything more — readability over cleverness.

Functions-as-objects is useful alone. Real power: function remembers variables from creation scope. That's a closure.

Closures — Captured State

Closure: function that remembers enclosing scope variables even after that scope finishes. "Closes over" those variables, carries them wherever it goes.

def make_threshold_filter(min_accuracy):
    def passes(result):
        return result["accuracy"] >= min_accuracy
    return passes

is_good = make_threshold_filter(0.85)
print(is_good(results[0]))  # False (0.82 < 0.85)
print(is_good(results[1]))  # True (0.91 >= 0.85)

make_threshold_filter(0.85) called → creates inner passes and returns it. Outer function finished — local scope gone. passes still accesses min_accuracy because Python stores it in cell object attached to function. Inner function carries environment.

Gotcha: closures capture variable, not value. Called late binding:

filters = []
for threshold in [0.7, 0.8, 0.9]:
    filters.append(lambda r: r["accuracy"] >= threshold)

# You'd expect three different filters. But:
print(filters[0]({"accuracy": 0.75}))  # False — threshold is 0.9, not 0.7!

All three lambdas close over same variable threshold, not value at each iteration. By call time: threshold = final loop value 0.9. Fix: capture current value as default argument:

filters = []
for threshold in [0.7, 0.8, 0.9]:
    filters.append(lambda r, t=threshold: r["accuracy"] >= t)
    # t=threshold evaluates at definition time, freezing the value

Default arguments evaluated at definition time — same behavior that caused mutable default bug fixes late binding here. Same rule, opposite consequence. Python is consistent. Mental models are not.

Closures power many ML patterns: LR schedulers closing over optimizer, callbacks closing over training state. And context managers. Speaking of which — "set up, do work, tear down" pattern appears constantly in PyTorch training loops.

Context Managers — PyTorch's Train, Eval, and No-Grad

PyTorch training loop rhythm: train epoch, evaluate on validation, repeat. Switching train→eval requires state changes that must happen and must be undone. Getting wrong: insidious bugs — validation accuracy looks fine but model still in training mode with dropout active, or gradients silently computed eating GPU memory during inference.

Python's with statement and context manager protocol: set up, do work, guarantee teardown even if crash. Three most important context managers in PyTorch — here's how they work.

torch.no_grad() — The Memory Saver

During training, PyTorch builds computation graph — every tensor operation recorded so .backward() can compute gradients. Graph consumes memory. During inference: no gradients needed. Forward pass should run without recording.

model = SimpleModel(2, 1)
x = torch.tensor([[0.01, 10.0]])

# WITHOUT no_grad — computation graph is built (wasteful during inference)
output = model(x)
print(output.requires_grad)  # True — graph was recorded, memory allocated

# WITH no_grad — no graph, no gradient memory
with torch.no_grad():
    output = model(x)
    print(output.requires_grad)  # False — clean, no graph overhead

torch.no_grad() internals: entering with block sets thread-local flag → autograd engine stops recording. Every tensor op inside skips graph construction. Exit — normal or exception — restores flag. Guarantee: if inference code throws, no-grad flag doesn't stay stuck for rest of program.

model.train() and model.eval() — Behavioral Switches

Some layers behave differently in train vs inference. Dropout: randomly zeros neurons during training, passes everything during inference. BatchNorm: batch stats during training, running-average stats during inference. Layers check model's training flag.

model.train()   # sets self.training = True on model and ALL submodules
# Dropout is active, BatchNorm uses batch stats

model.eval()    # sets self.training = False on model and ALL submodules
# Dropout is disabled, BatchNorm uses running stats

Common mistake: model.eval() without torch.no_grad(). Different things. model.eval(): changes layer behavior (dropout, batchnorm). torch.no_grad(): disables gradient computation. Almost always want both during validation:

def validate(model, val_loader, criterion):
    model.eval()                          # change layer behavior
    total_loss = 0.0
    with torch.no_grad():                 # stop recording gradients
        for features, targets in val_loader:
            output = model(features)
            loss = criterion(output, targets)
            total_loss += loss.item()
    model.train()                         # restore training behavior
    return total_loss / len(val_loader)

Must remember model.train() at end. Forget — or exception interrupts — model stays in eval mode. Dropout won't fire, BatchNorm uses stale running stats, training silently degrades.

How Context Managers Work Under the Hood

Context manager: any object implementing __enter__ and __exit__. with obj as x → calls x = obj.__enter__(). Block ends (normal or exception) → obj.__exit__(exc_type, exc_val, exc_tb).

torch.no_grad() simplified:

# Simplified from torch/autograd/grad_mode.py
class no_grad:
    def __enter__(self):
        self.prev = torch.is_grad_enabled()   # save previous state
        torch.set_grad_enabled(False)          # disable gradients
        return self

    def __exit__(self, exc_type, exc_val, exc_tb):
        torch.set_grad_enabled(self.prev)      # restore previous state
        return False                           # don't suppress exceptions

__exit__ restores grad state regardless of how block ended. Safety guarantee: inference code raises error → gradients still re-enabled.

Build own context managers for ML patterns. Here's one handling train/eval switching:

from contextlib import contextmanager

@contextmanager
def eval_mode(model):
    """Context manager that handles model.eval() + no_grad safely."""
    was_training = model.training
    model.eval()
    try:
        with torch.no_grad():
            yield model
    finally:
        if was_training:
            model.train()

# Now validation is safe even if something crashes:
with eval_mode(model) as m:
    for batch in val_loader:
        output = m(batch)
        # ... if an exception occurs here, model.train() is still called

@contextmanager from contextlib turns generator function into context manager. Code before yield = __enter__, code after yield in finally = __exit__. Closures, dunder methods, generators — all working together.

Another production context manager: torch.cuda.amp.autocast() for mixed-precision training. Temporarily changes default dtype to float16 or bfloat16 → faster computation on modern GPUs:

from torch.amp import autocast, GradScaler

# Assumes model, optimizer, criterion, and train_loader are already defined
scaler = GradScaler()

for features, targets in train_loader:
    optimizer.zero_grad()

    with autocast(device_type='cuda'):         # forward pass in float16/bfloat16
        output = model(features)
        loss = criterion(output, targets)

    scaler.scale(loss).backward()              # backward in float32
    scaler.step(optimizer)
    scaler.update()

Pattern always same: set up temporary state, yield control, guarantee cleanup. Most reliable way to manage complex state transitions in ML training.
