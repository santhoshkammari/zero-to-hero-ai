Version Control with Git

Chapter 1: Python & Programming Foundations 3 subtopics
TL;DR

Git tracks every change you make to your code so you can rewind mistakes, work on features without breaking what already works, and collaborate with others without overwriting each other's work. You stage changes, commit them with a message, branch off for experiments, merge back when ready, and push to a remote like GitHub so the whole team stays in sync. Master these basics and you'll never lose work again.

Git Basics

What is Version Control

Picture this. You're training a model. It works. You tweak the preprocessing code to try something new. Now the model is broken, and you can't remember what you changed. You dig through files named model_v2_final.py, model_v2_final_REAL.py, and model_v2_final_REAL_USE_THIS.py. Sound familiar?

Version control is a system that records changes to files over time so you can recall any previous version. Think of it as an infinite undo button for your entire project. Every change is tracked: what changed, who changed it, when, and why. Git is the most widely used version control system in the world, and for good reason. It's fast, it works offline, and it handles projects of any size — from a weekend script to the Linux kernel (which is where Git was born, by the way — Linus Torvalds built it because nothing else was good enough).

Git doesn't store copies of your whole project every time you save. Instead, it stores snapshots — compressed records of what every file looks like at each save point. If a file hasn't changed, Git references the previous version instead of duplicating it. This makes it incredibly efficient.

Installing Git

Before anything, you need Git on your machine. On most Linux distributions it's already there. On macOS it ships with the Xcode command line tools. On Windows, download it from the official site.

# Check if Git is installed
git --version
# git version 2.43.0

# Ubuntu/Debian
sudo apt-get install git

# macOS (if not already installed)
xcode-select --install

# After installing, tell Git who you are
git config --global user.name "Ada Lovelace"
git config --global user.email "ada@example.com"

That name and email get stamped on every change you make. They're not login credentials — they're your signature in the project history.

git init

Every Git project starts with git init. This command creates a hidden .git folder inside your project directory. That folder is Git's brain — it holds the entire history of your project. Delete it, and the history is gone. Leave it alone, and Git handles the rest.

mkdir ml-experiment && cd ml-experiment
git init
# Initialized empty Git repository in /home/ada/ml-experiment/.git/

ls -la
# drwxr-xr-x  7 ada ada 4096 Jan 15 10:00 .git
# Nothing else yet — just an empty repo waiting for your first file.

You run git init once per project. After that, you're working inside a repository (or "repo" for short) — a directory whose changes Git is watching.

git add (Staging)

Here's where most beginners get confused. Git doesn't automatically track every change you make. You have to explicitly tell it which changes matter. This happens in two steps: staging, then committing.

Think of a staging area as a loading dock at a warehouse. You're packing a shipment (a commit). You don't throw everything from the warehouse onto the truck at once. You walk around, pick the specific boxes you want on this particular truck, carry them to the loading dock, check the manifest, and then send the truck off. The loading dock is the staging area. git add carries boxes to the dock. git commit sends the truck.

# Create some files
echo "# ML Experiment" > README.md
echo "import pandas as pd" > preprocess.py
echo "secret_key=abc123" > .env

# Stage one file
git add README.md

# Stage multiple files
git add README.md preprocess.py

# Stage everything in the current directory
git add .

Why not stage everything automatically? Because sometimes you change five files but only three of those changes are related. You want your commits to be logical units — "added data preprocessing" or "fixed the learning rate bug" — not "changed a bunch of stuff." Staging lets you curate exactly what goes into each commit.

The Three States

Every file in a Git repo lives in one of three states: modified (changed but not staged), staged (marked for the next commit), or committed (safely stored in history). Understanding this flow — modify → stage → commit — is the mental model that makes everything else click.

git commit

A commit is a permanent snapshot of your staged changes. Each commit gets a unique ID (a long hexadecimal hash), records who made the change and when, and requires a message explaining why.

git commit -m "Add README and preprocessing script"
# [main (root-commit) a3b8f2c] Add README and preprocessing script
#  2 files changed, 2 insertions(+)
#  create mode 100644 README.md
#  create mode 100644 preprocess.py

That hash a3b8f2c is the first seven characters of a SHA-1 hash. It uniquely identifies this exact snapshot. You can always return to it later.

Write commit messages that your future self will thank you for. "Fixed stuff" tells you nothing six months later. "Fix off-by-one error in batch data loader" tells you everything. A good pattern is: start with a verb in the imperative mood ("Add", "Fix", "Refactor", "Remove") and keep the first line under 72 characters.

# Quick commit for small changes
git commit -m "Fix learning rate default from 0.1 to 0.001"

# For longer messages, drop the -m flag to open your editor
git commit
# This opens vim/nano/your configured editor for a multi-line message

git status and git log

git status is your dashboard. Run it constantly. It tells you what's modified, what's staged, what's untracked, and what branch you're on. When in doubt, git status.

echo "from sklearn.ensemble import RandomForestClassifier" > train.py
git status
# On branch main
# Untracked files:
#   (use "git add ..." to include in what will be committed)
#         .env
#         train.py
#
# nothing added to commit but untracked files present

git add train.py
git commit -m "Add model training script"

git log
# commit f7e2d1a (HEAD -> main)
# Author: Ada Lovelace 
# Date:   Mon Jan 15 10:15:00 2025
#
#     Add model training script
#
# commit a3b8f2c
# Author: Ada Lovelace 
# Date:   Mon Jan 15 10:00:00 2025
#
#     Add README and preprocessing script

git log shows the history — most recent commit first. For a compact view, try git log --oneline which squeezes each commit onto a single line. When your history grows long, this becomes essential.

git log --oneline
# f7e2d1a (HEAD -> main) Add model training script
# a3b8f2c Add README and preprocessing script

git diff

git diff shows you exactly what changed, line by line. This is how you review your own work before staging it.

# Edit the training script
echo "clf = RandomForestClassifier(n_estimators=200)" >> train.py

git diff
# diff --git a/train.py b/train.py
# index 8a3c5e1..b2d7f4a 100644
# --- a/train.py
# +++ b/train.py
# @@ -1 +1,2 @@
#  from sklearn.ensemble import RandomForestClassifier
# +clf = RandomForestClassifier(n_estimators=200)

Lines starting with + are additions. Lines starting with - are deletions. The @@ markers tell you where in the file the change happened. Once you stage a file, git diff won't show it anymore — use git diff --staged to see what's queued up for the next commit.

.gitignore File

Not everything belongs in version control. Datasets that are gigabytes in size, trained model weights, API keys, cached files — these should stay out of your repo. A .gitignore file tells Git which files and folders to pretend don't exist.

# Create .gitignore in the root of your project
cat > .gitignore << 'EOF'
# Python
__pycache__/
*.pyc
*.pyo
*.egg-info/
dist/
build/

# Virtual environments
venv/
.venv/
env/

# Secrets and config
.env
*.secret

# Data files (too large for Git)
data/raw/
data/processed/
*.csv
*.parquet
*.h5
*.hdf5

# Model checkpoints and artifacts
models/
checkpoints/
*.pkl
*.joblib
*.pt
*.pth
*.onnx
*.safetensors

# Jupyter notebook checkpoints
.ipynb_checkpoints/

# OS files
.DS_Store
Thumbs.db

# IDE settings
.vscode/
.idea/
EOF

git add .gitignore
git commit -m "Add .gitignore for ML project"
gitignore Timing

A .gitignore only prevents untracked files from being added. If you already committed a file and then add it to .gitignore, Git still tracks it. You'd need to run git rm --cached filename to untrack it while keeping it on disk. Set up your .gitignore early — ideally as your first or second commit.

Notice the .env file from earlier? It never got committed because we staged our files selectively. But once .gitignore is in place, even git add . will skip it. That's the safety net.

Branching and Merging

Creating Branches

Here's the scenario. Your preprocessing pipeline works. Your teammate needs it to stay stable while they build the training loop. But you want to try a completely different approach to feature engineering. How do you experiment without breaking what works?

Branches. A branch is an independent line of development. Think of your project history as a tree trunk. A branch sprouts off from a point on that trunk, letting you make changes in isolation. The trunk — usually called main — stays untouched while you experiment on your branch.

# See what branches exist
git branch
# * main

# Create a new branch
git branch feature/new-preprocessing

The * next to main means that's where you are right now. Creating a branch doesn't move you to it — it's like putting up a signpost without walking down the path yet.

Switching Branches (checkout/switch)

To actually start working on a branch, you switch to it.

# Modern way (Git 2.23+)
git switch feature/new-preprocessing
# Switched to branch 'feature/new-preprocessing'

# Older way (still works everywhere)
git checkout feature/new-preprocessing

# Create and switch in one step
git switch -c feature/data-augmentation
# or
git checkout -b feature/data-augmentation

Now every commit you make lands on this branch, not on main. Your main branch stays frozen at the exact state it was in when you branched off. You can switch back to main at any time and see the "old" version of your code, completely untouched by your experiment.

# On feature/new-preprocessing branch
echo "from sklearn.preprocessing import StandardScaler" >> preprocess.py
git add preprocess.py
git commit -m "Add StandardScaler to preprocessing pipeline"

# Switch back to main
git switch main
cat preprocess.py
# import pandas as pd
# (The StandardScaler line is NOT here — it only exists on the feature branch)

This feels like magic the first time you see it. Git literally swaps out the files on your disk when you switch branches. Your editor might ask you to reload files. That's normal.

Merging Branches

Your experiment worked. The new preprocessing is better. Now you want to bring those changes back into main. That's a merge.

# First, switch to the branch you want to merge INTO
git switch main

# Then merge the feature branch into it
git merge feature/new-preprocessing
# Updating a3b8f2c..c9d4e5f
# Fast-forward
#  preprocess.py | 1 +
#  1 file changed, 1 insertion(+)

# The feature branch's commits are now part of main
git log --oneline
# c9d4e5f (HEAD -> main, feature/new-preprocessing) Add StandardScaler to preprocessing pipeline
# f7e2d1a Add model training script
# a3b8f2c Add README and preprocessing script

When the merge is clean — meaning nobody else changed the same parts of the same files — Git handles it automatically. The term "Fast-forward" in the output means main hadn't moved since you branched off, so Git could apply your commits directly on top without any additional work.

Once merged, you can delete the feature branch. It served its purpose.

git branch -d feature/new-preprocessing
# Deleted branch feature/new-preprocessing (was c9d4e5f).

Resolving Merge Conflicts

Merges aren't always clean. When two branches modify the same line in the same file, Git doesn't know which version to keep. It stops and asks you to decide. This is a merge conflict.

Let's walk through one. Suppose you and a teammate both edited train.py — you changed the number of estimators on main, and they changed the same line on their branch.

# On main: update estimators to 300
git switch main
# (edit train.py so the line reads n_estimators=300)
git add train.py && git commit -m "Increase estimators to 300"

# Create a branch that diverged earlier with a different value
git switch -c experiment/more-trees
# (edit train.py so the line reads n_estimators=500)
git add train.py && git commit -m "Try 500 estimators"

# Now try to merge back into main
git switch main
git merge experiment/more-trees
# Auto-merging train.py
# CONFLICT (content): Merge conflict in train.py
# Automatic merge failed; fix conflicts and then commit the result.

Open train.py and you'll see Git's conflict markers:

from sklearn.ensemble import RandomForestClassifier
<<<<<<< HEAD
clf = RandomForestClassifier(n_estimators=300)
=======
clf = RandomForestClassifier(n_estimators=500)
>>>>>>> experiment/more-trees

Everything between <<<<<<< HEAD and ======= is what's on your current branch (main). Everything between ======= and >>>>>>> is what's on the incoming branch. Your job: pick one, combine them, or write something entirely new. Then delete the conflict markers.

from sklearn.ensemble import RandomForestClassifier
clf = RandomForestClassifier(n_estimators=500)  # Went with 500 after testing

After editing, tell Git the conflict is resolved:

git add train.py
git commit -m "Resolve conflict: use 500 estimators based on validation results"
Conflict Prevention

Merge conflicts aren't failures — they're a natural part of collaboration. But you can minimize them: keep branches short-lived, merge main into your feature branch regularly to stay up to date, and communicate with your team about who's working on what.

Remote Repositories

GitHub/GitLab Overview

Everything so far has been local — on your machine only. If your hard drive dies, your project dies with it. And nobody else can see your work. Remote repositories solve both problems.

GitHub, GitLab, and Bitbucket are hosting platforms for Git repositories. They store a copy of your repo on their servers, giving you a backup and a place for others to access the code. GitHub is the most popular, especially for open-source and ML projects. GitLab is common in enterprise settings because companies can host it on their own servers.

These platforms add features on top of Git itself: issue tracking, project boards, code review tools, CI/CD pipelines, and wikis. But at their core, they're hosting a Git repo that you push to and pull from.

git clone

When you want to work on a project that already exists on a remote, you clone it. This downloads the full repository — every file, every commit, every branch — to your machine.

git clone https://github.com/ada/ml-experiment.git
# Cloning into 'ml-experiment'...
# remote: Enumerating objects: 42, done.
# remote: Total 42 (delta 12), reused 38 (delta 10)
# Receiving objects: 100% (42/42), done.

cd ml-experiment
git log --oneline
# All the history is here, ready to work with

Cloning automatically sets up a remote called origin — a bookmark pointing to the URL you cloned from. You don't need to configure anything else to start pushing and pulling.

git push and git pull

git push sends your local commits to the remote. git pull fetches commits from the remote and merges them into your local branch. Push to share your work. Pull to get everyone else's.

# Push your local main branch to the remote
git push origin main
# Enumerating objects: 5, done.
# Counting objects: 100% (5/5), done.
# Writing objects: 100% (3/3), 312 bytes | 312.00 KiB/s, done.
# To https://github.com/ada/ml-experiment.git
#    a3b8f2c..f7e2d1a  main -> main

# Pull the latest changes from the remote
git pull origin main
# Already up to date.
# (or it'll download and merge new commits)

If you're pushing a branch that doesn't exist on the remote yet, Git will tell you:

git switch -c feature/better-metrics
# (make some commits)
git push origin feature/better-metrics
# This creates the branch on the remote
Push Rejected?

If someone else pushed changes to the same branch since your last pull, your push will be rejected. Don't panic. Run git pull first to incorporate their changes, resolve any conflicts, then push again. Never use git push --force on shared branches — it rewrites history and can destroy your teammates' work.

git remote

A remote is a named reference to a repository hosted somewhere else. When you clone, Git creates one called origin automatically. You can have multiple remotes — useful when contributing to open source.

# List remotes
git remote -v
# origin  https://github.com/ada/ml-experiment.git (fetch)
# origin  https://github.com/ada/ml-experiment.git (push)

# Add another remote
git remote add upstream https://github.com/original-author/ml-experiment.git

# Now you have two remotes
git remote -v
# origin    https://github.com/ada/ml-experiment.git (fetch)
# origin    https://github.com/ada/ml-experiment.git (push)
# upstream  https://github.com/original-author/ml-experiment.git (fetch)
# upstream  https://github.com/original-author/ml-experiment.git (push)

The convention: origin is your copy. upstream is the original project you forked from. This matters a lot in open-source workflows, which we'll get to shortly.

Pull Requests (Concept)

A pull request (PR) isn't a Git feature — it's a platform feature added by GitHub, GitLab (where they're called "merge requests"), and similar services. But it's become so central to how teams work that you need to understand it.

Here's the workflow. You create a branch for your feature. You push it to the remote. Then, instead of merging it yourself, you open a pull request — a proposal that says "I'd like to merge these changes into main. Please review them." Your teammates look at the diff, leave comments, maybe request changes. You push additional commits to address their feedback. When everyone's satisfied, someone clicks "Merge" and the branch is folded into main.

This is how professional software teams work. No code reaches the main branch without being reviewed. In ML projects, this is especially important because a small bug in data preprocessing can silently corrupt your entire training pipeline without throwing any errors.

The typical PR flow looks like this:

# 1. Create a feature branch
git switch -c feature/add-cross-validation

# 2. Make changes and commit
echo "from sklearn.model_selection import cross_val_score" >> train.py
git add train.py
git commit -m "Add cross-validation to training pipeline"

# 3. Push the branch to the remote
git push origin feature/add-cross-validation

# 4. Open a pull request on GitHub (done in the browser)
#    GitHub will often show a prompt: "Compare & pull request"

# 5. After review and approval, merge via the GitHub UI

# 6. Locally, switch back to main and pull the merged changes
git switch main
git pull origin main

# 7. Clean up
git branch -d feature/add-cross-validation

Some teams require that automated tests pass before a PR can be merged. Some require approval from at least two reviewers. These guardrails exist because the main branch should always be in a working state.

Forking a Repository

What if you want to contribute to a project you don't have write access to? You can't push branches to someone else's repo. This is where forking comes in.

A fork is your personal copy of someone else's repository, hosted under your own GitHub account. You have full control over your fork — you can push branches, break things, experiment freely. When you've got something worth contributing back, you open a pull request from your fork to the original repo.

Here's the complete open-source contribution workflow:

# 1. Fork the repo on GitHub (click the "Fork" button in the browser)
#    This creates github.com/ada/cool-ml-library (your copy)
#    from github.com/original-author/cool-ml-library

# 2. Clone YOUR fork
git clone https://github.com/ada/cool-ml-library.git
cd cool-ml-library

# 3. Add the original repo as "upstream"
git remote add upstream https://github.com/original-author/cool-ml-library.git

# 4. Create a feature branch (never work directly on main)
git switch -c fix/typo-in-readme

# 5. Make your changes, commit, and push to YOUR fork
git add README.md
git commit -m "Fix typo in installation instructions"
git push origin fix/typo-in-readme

# 6. Open a pull request on GitHub:
#    from ada/cool-ml-library:fix/typo-in-readme
#    into original-author/cool-ml-library:main

# 7. Keep your fork up to date with the original
git switch main
git pull upstream main
git push origin main

Step 7 is one that people often forget. The original project keeps moving forward. If you don't regularly sync your fork, your branches will be based on stale code and your PRs will have conflicts. Make it a habit: before starting any new branch, pull from upstream first.

Fork vs. Clone vs. Branch

These three concepts confuse everyone at first. A clone downloads a repo to your machine. A branch is a parallel line of work within a single repo. A fork is a full copy of a repo under your own account on GitHub. You fork to get your own remote copy, clone to get it locally, and branch to organize your work within it.

Contributing to open-source ML projects — fixing a bug in scikit-learn, adding a feature to Hugging Face Transformers, improving documentation — follows this exact pattern. It's one of the best ways to learn, build a reputation, and give back to the tools you depend on every day.

Git has a learning curve. There's no getting around it. The command names aren't always intuitive, the error messages can be cryptic, and the first time you get a merge conflict it feels like the world is ending. But here's the thing: every professional developer uses Git. Every ML team uses Git. Every open-source project uses Git. The time you invest now pays dividends for your entire career. Start with init, add, commit, push, and pull. Build from there. Before long, it'll be second nature.
