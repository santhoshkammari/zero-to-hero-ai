Modern Python Features

Chapter 1: Python & Programming Foundations 7 subtopics
TL;DR

Python evolves fast. Recent versions (3.8–3.12) introduced features that make code shorter, safer, and faster — from the walrus operator and structural pattern matching to modern type hints and exception groups. The Faster CPython project has delivered 25–60% speedups across releases. Understanding these features matters for ML work because they improve code clarity in complex pipelines, catch type errors before runtime, and reduce the overhead of Python's interpreted nature.

Walrus Operator (:=, Python 3.8+)

You've written this code a hundred times: call a function, assign the result to a variable, then check the variable. It works, but it's clunky. The walrus operator (:=), officially called an "assignment expression," lets you assign and test in a single expression.

# Before: Separate assignment and test
line = input_file.readline()
while line:
    process(line)
    line = input_file.readline()

# After: Walrus operator — assign and test in one shot
while (line := input_file.readline()):
    process(line)

The name comes from the operator's visual resemblance to a walrus on its side — the colon is the eyes and the equals sign is the tusks. Beyond loops, it's particularly useful in list comprehensions where you'd otherwise compute a value twice:

# Without walrus: compute_metric is called twice for matching items
results = []
for batch in data_batches:
    metric = compute_metric(batch)
    if metric > threshold:
        results.append(metric)

# With walrus: compute once, filter and keep
results = [
    m for batch in data_batches
    if (m := compute_metric(batch)) > threshold
]

In ML work, this pattern shows up when filtering datasets — compute a feature, check if it meets criteria, and keep it, all without redundant computation.

# Practical ML use: filter batches where loss exceeds threshold
hard_examples = [
    (batch, loss)
    for batch in training_data
    if (loss := compute_loss(model, batch)) > hard_example_threshold
]
Don't Overuse the Walrus

The walrus operator is powerful but can hurt readability if overused. Use it when it eliminates a genuine redundancy — duplicate function calls or awkward loop priming. Don't use it to cram complex logic into a single line. If your coworker needs to read it three times to understand, use a regular assignment.

Type Parameter Syntax (Python 3.12)

Before Python 3.12, writing generic types required importing TypeVar and doing awkward gymnastics:

# Old way (Python < 3.12) — verbose and error-prone
from typing import TypeVar, Generic

T = TypeVar("T")          # Have to repeat the name as a string
U = TypeVar("U", bound=int)  # Constraints are separate from usage

class Stack(Generic[T]):
    def __init__(self) -> None:
        self._items: list[T] = []

    def push(self, item: T) -> None:
        self._items.append(item)

    def pop(self) -> T:
        return self._items.pop()

Python 3.12 introduced a type parameter syntax that reads like how you think about generics. Type parameters are declared inline, right where they're used:

# New way (Python 3.12+) — clean and intuitive
class Stack[T]:
    def __init__(self) -> None:
        self._items: list[T] = []

    def push(self, item: T) -> None:
        self._items.append(item)

    def pop(self) -> T:
        return self._items.pop()

# Generic functions — no separate TypeVar declaration
def first[T](items: list[T]) -> T:
    return items[0]

# With constraints
def maximum[T: (int, float)](a: T, b: T) -> T:
    return a if a > b else b

For ML code, this makes generic data pipeline components much cleaner:

# A type-safe data pipeline component
class Transform[InputT, OutputT]:
    def __init__(self, fn: Callable[[InputT], OutputT]):
        self.fn = fn

    def __call__(self, data: InputT) -> OutputT:
        return self.fn(data)

    def chain[NewOutputT](
        self, next_fn: Callable[[OutputT], NewOutputT]
    ) -> "Transform[InputT, NewOutputT]":
        return Transform(lambda x: next_fn(self.fn(x)))

TypeAlias, TypeGuard, and Self Type Hints

Modern Python's type system has grown beyond basic int and str annotations. Three additions are especially useful for ML codebases.

TypeAlias (Python 3.10+)

When you create complex type aliases, it's ambiguous whether you're creating a type alias or a regular variable. TypeAlias makes the intent explicit:

from typing import TypeAlias
import numpy as np

# Explicit type aliases — reader knows these are types, not values
Vector: TypeAlias = np.ndarray
Matrix: TypeAlias = np.ndarray
BatchedTensor: TypeAlias = np.ndarray  # shape: (batch, features)

# Python 3.12 syntax is even cleaner
type Vector = np.ndarray
type Matrix = np.ndarray
type ModelConfig = dict[str, int | float | str]

TypeGuard (Python 3.10+)

TypeGuard lets you write functions that narrow types — useful when validating ML inputs:

from typing import TypeGuard
import numpy as np

def is_valid_feature_vector(data: object) -> TypeGuard[np.ndarray]:
    """Type checker now knows: if this returns True, data is np.ndarray."""
    return (
        isinstance(data, np.ndarray)
        and data.ndim == 1
        and data.dtype == np.float32
        and not np.any(np.isnan(data))
    )

def predict(data: object) -> float:
    if is_valid_feature_vector(data):
        # Type checker knows 'data' is np.ndarray here
        return float(model.predict(data.reshape(1, -1))[0])
    raise ValueError("Invalid feature vector")

Self Type (Python 3.11+)

Before Self, returning self from a method required awkward workarounds to get the type right in subclasses:

from typing import Self

class ModelBuilder:
    def set_learning_rate(self, lr: float) -> Self:
        self.lr = lr
        return self  # Fluent interface — allows chaining

    def set_epochs(self, epochs: int) -> Self:
        self.epochs = epochs
        return self

    def build(self) -> "Model":
        return Model(self.lr, self.epochs)

# Works correctly with subclasses too
class AdvancedModelBuilder(ModelBuilder):
    def set_regularization(self, reg: float) -> Self:
        self.reg = reg
        return self

# Type checker knows this returns AdvancedModelBuilder, not ModelBuilder
builder = AdvancedModelBuilder().set_learning_rate(0.01).set_regularization(0.1)

Improved Error Messages (Python 3.11+)

Debugging ML code often means staring at tracebacks trying to figure out which part of a complex expression failed. Python 3.11 dramatically improved error messages with precise error locations.

# In Python 3.10, this error message says:
#   TypeError: unsupported operand type(s) for +: 'int' and 'str'
# But WHICH '+' on a complex line? No idea.

# In Python 3.11+, you get:
#   result = scores[0] + weights[1] * features["age"] + bias
#                                     ~~~~~~~~~~~~~~~~^~~~~~
#   TypeError: unsupported operand type(s) for +: 'float' and 'str'

The caret (^) and tildes (~) point to the exact subexpression that failed. This is enormous for ML code where you often chain operations on complex data structures:

# Before 3.11: Good luck figuring out which key is missing
loss = (predictions["cls_head"]["logits"] - targets["labels"]).mean()

# Python 3.11: The traceback shows exactly which access failed
#   loss = (predictions["cls_head"]["logits"] - targets["labels"]).mean()
#                                                ~~~~~~~~~~~~~~~
#   KeyError: 'labels'
Worth the Upgrade

If you're still on Python 3.9 or 3.10, the improved error messages alone are worth upgrading. In ML code with deeply nested dict/tensor operations, the difference between "KeyError somewhere on line 47" and "KeyError at exactly this bracket" can save hours of debugging.

ExceptionGroup and except* (Python 3.11)

Traditional try/except handles one exception at a time. But what happens when you run five concurrent tasks and three of them fail? With asyncio.gather() or TaskGroup, you can get multiple exceptions simultaneously. ExceptionGroup bundles them together, and except* lets you handle them selectively.

import asyncio

async def fetch_data(source: str) -> dict:
    if source == "broken_api":
        raise ConnectionError(f"Cannot reach {source}")
    if source == "bad_format":
        raise ValueError(f"Invalid data from {source}")
    return {"source": source, "data": [1, 2, 3]}

async def main():
    try:
        async with asyncio.TaskGroup() as tg:
            tg.create_task(fetch_data("good_api"))
            tg.create_task(fetch_data("broken_api"))
            tg.create_task(fetch_data("bad_format"))
    except* ConnectionError as eg:
        # Handles ALL ConnectionErrors from the group
        print(f"Connection failures: {len(eg.exceptions)}")
        for exc in eg.exceptions:
            print(f"  - {exc}")
    except* ValueError as eg:
        # Handles ALL ValueErrors from the group
        print(f"Data format errors: {len(eg.exceptions)}")
        for exc in eg.exceptions:
            print(f"  - {exc}")

asyncio.run(main())

The except* syntax (note the asterisk) matches against the types inside the group. Multiple except* clauses can each handle different exception types from the same group — they don't short-circuit like regular except chains. This is crucial for ML pipelines that fetch data from multiple sources concurrently.

tomllib Module (Python 3.11)

ML projects are drowning in config files. YAML is ambiguous (is no a string or a boolean?), JSON doesn't support comments, and INI is too flat. TOML — Tom's Obvious Minimal Language — is the sweet spot. Python 3.11 added tomllib as a built-in module for parsing TOML files.

import tomllib

# config.toml
"""
[model]
name = "resnet50"
learning_rate = 0.001
epochs = 100
batch_size = 32

[model.optimizer]
type = "adam"
weight_decay = 1e-4

[data]
train_path = "/data/train"
val_path = "/data/val"
augmentation = true

[[data.transforms]]
name = "resize"
size = [224, 224]

[[data.transforms]]
name = "normalize"
mean = [0.485, 0.456, 0.406]
std = [0.229, 0.224, 0.225]
"""

with open("config.toml", "rb") as f:  # Note: binary mode required
    config = tomllib.load(f)

print(config["model"]["learning_rate"])  # 0.001
print(config["data"]["transforms"][0])   # {'name': 'resize', 'size': [224, 224]}
Read-Only by Design

tomllib only reads TOML — it can't write it. This is intentional. For writing TOML, use the third-party tomli-w package. The reasoning: reading TOML is straightforward, but writing it while preserving comments and formatting is complex, and the stdlib maintainers didn't want to ship a half-baked writer.

Faster CPython Project (Performance Improvements)

Python has a reputation for being slow. The Faster CPython project, led by Mark Shannon with support from Microsoft, is changing that. The goal: make CPython 5× faster over several releases. They're not there yet, but the progress is real.

Python 3.11: The Big Jump

Python 3.11 was 10–60% faster than 3.10, depending on the workload. The key optimizations:

Specializing adaptive interpreter — CPython now watches what types flow through each bytecode instruction at runtime. If it sees that a particular + operation always adds two integers, it replaces the generic BINARY_ADD with a specialized BINARY_ADD_INT that skips type checking. If the pattern changes, it de-specializes back to the generic version.

import dis

def add_numbers(a, b):
    return a + b

# After a few calls with ints, the bytecode gets specialized:
# BINARY_OP (generic) → BINARY_OP_ADD_INT (specialized for ints)
# This skips the type dispatch overhead on each call
for i in range(100):
    add_numbers(i, i + 1)

Faster startup — frozen modules and lazy imports mean Python starts up faster, which matters for serverless ML deployments where cold start time is critical.

Zero-cost exceptions — try/except blocks now have zero overhead when no exception is raised. Before 3.11, entering a try block had a cost even on the happy path. This matters for ML code that wraps numerical operations in try/except to catch NaN or overflow.

Python 3.12: Per-Interpreter GIL and More

Python 3.12 continued the trend with comprehension inlining (list comprehensions no longer create a separate stack frame), more specialized bytecodes, and the groundwork for per-interpreter GIL — a step toward true multi-threaded Python.

Python 3.13: Free-Threaded Python (Experimental)

Python 3.13 ships with an experimental no-GIL build. This is the big one — the Global Interpreter Lock has been the single biggest performance limitation of Python for multi-threaded CPU-bound code. The no-GIL build allows true parallel execution of Python threads.

# With free-threaded Python (3.13+ experimental, no-GIL build),
# this actually runs on 4 cores simultaneously:
from threading import Thread

def train_fold(fold_data):
    """CPU-bound training — benefits from true thread parallelism."""
    model = fit_model(fold_data)
    return evaluate(model)

threads = [Thread(target=train_fold, args=(fold,)) for fold in folds]
for t in threads:
    t.start()
for t in threads:
    t.join()
Free-Threaded Python Isn't Ready for Production

The no-GIL build in Python 3.13 is experimental. Many C extensions (including NumPy and PyTorch) need updates to work correctly without the GIL. The GIL protected C extensions from thread-safety bugs they never had to worry about. Expect 1–2 more releases before the ecosystem catches up. For now, stick with multiprocessing for CPU parallelism in production ML code.

The broader trend is clear: Python is getting faster with each release, and the gap between "Python is too slow" and "Python is fast enough" is closing. For most ML work, the bottleneck is I/O or GPU compute, not Python speed. But for data preprocessing, feature engineering, and serving, these speedups compound into real time and cost savings.
