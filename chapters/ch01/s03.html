Object-Oriented Programming

Chapter 1: Python & Programming Foundations 6 tables · 2 code blocks · Deep

Every ML framework is built on OOP — model.fit(X, y) is a method call on a BaseEstimator subclass, model(input) invokes __call__ on nn.Module, and a HuggingFace Dataset requires __len__ + __getitem__. If you cannot read and extend these class hierarchies, you cannot do production ML work.

Core OOP Concepts

Not "what is a class" — focus on the parts that matter when you read framework source code.

ConceptWhatWhy It Matters in MLExample Pattern
class Blueprint that bundles data (attributes) and behavior (methods) into one unit Every sklearn estimator, every PyTorch module, every HuggingFace tokenizer is a class — the unit of reuse in all frameworks class MyModel(nn.Module):
__init__ Initializer — runs when you create an instance. Sets up all state the object needs Where you declare layers (self.fc1 = nn.Linear(…)), store hyperparams, init weights def __init__(self, n_estimators=100):
self Explicit reference to the current instance — Python passes it automatically as first arg Lets each model instance carry its own weights, config, and fitted state independently self.weights = None
Instance attrs Set on self in __init__ — each instance gets its own copy Model weights, fitted flag, training history — must be per-instance self.is_fitted_ = False
Class attrs Set on the class body — shared across ALL instances Default configs, registry dicts, version strings. Gotcha: mutable class attrs (lists/dicts) are shared — mutating one instance's copy mutates all _estimator_type = "classifier"
Methods Functions bound to instances — receive self so they can read/write instance state fit(), predict(), forward() — the entire API surface of ML objects def fit(self, X, y):
Mutable class attribute trap: If you write class Model: layers = [], every instance shares the same list. Appending in one instance appends for all. Always initialize mutable state in __init__ as self.layers = [].

Dunder Methods — The Protocol System

Dunder (double-underscore) methods are Python's protocol hooks. You don't call them directly — Python calls them when you use operators, built-in functions, or language constructs. This table is the backbone of understanding how ML frameworks work internally.

MethodTriggers WhenML Framework UsageQuick Example
__init__(self, ...) obj = MyClass() Declare layers, store hyperparams, set defaults self.lr = lr
__repr__(self) repr(obj), REPL display sklearn's estimator repr: RandomForestClassifier(n_estimators=100) — auto-generated from get_params() return f"Model(lr={self.lr})"
__str__(self) str(obj), print(obj) Human-readable summary in logs. Rule: if you define only one, define __repr__ — it's used as fallback for str() return "Trained model (acc=0.95)"
__len__(self) len(obj) PyTorch Dataset.__len__ — DataLoader needs this to know total samples return len(self.data)
__getitem__(self, idx) obj[idx], obj[start:stop] PyTorch Dataset.__getitem__ — DataLoader indexes into your data with this. Also enables iteration and slicing return self.X[idx], self.y[idx]
__call__(self, ...) obj(...) — calling instance like a function The big one. PyTorch's nn.Module.__call__ does hooks + gradient tracking + calls forward(). You NEVER call forward() directly output = model(input)
__enter__ / __exit__ with obj as x: torch.no_grad(), file handles, timer contexts, database connections with torch.no_grad():
__iter__(self) for x in obj: DataLoader iteration, streaming datasets, batch generators yield from self.batches
__next__(self) next(obj) Manual iteration control — fetching next batch from a data stream return self.buffer.pop(0)
__eq__(self, other) obj == other Comparing configs, checking if two pipeline steps are identical return self.params == other.params
__hash__(self) hash(obj), using as dict key or in set Caching results by model config. Rule: if you define __eq__, Python sets __hash__ to None — you must define it explicitly to use objects in sets/dicts return hash(tuple(sorted(self.params.items())))
__add__(self, other) obj + other Tensor operations, combining datasets, merging configs return Tensor(self.data + other.data)
Why __call__ matters so much: When you write output = model(x) in PyTorch, you're NOT calling forward(). You're calling __call__, which: (1) runs registered forward hooks, (2) sets up autograd context, (3) calls forward(), (4) runs post-forward hooks. Calling model.forward(x) directly skips all of this and will silently break gradient computation.

Inheritance — Decision Table

Inheritance is how frameworks provide default behavior you can customize. The question is always: inherit or compose?

ApproachWhen to UseProsConsML Example
Single inheritance Extending one framework base class Simple, clear hierarchy, framework expects it Tight coupling to parent class MyModel(nn.Module)
Multiple inheritance Mixing capabilities from >1 parent Reuse across orthogonal features MRO complexity, diamond problem, debugging nightmares sklearn's class SGDClassifier(BaseSGD, ClassifierMixin)
Mixins Adding a slice of behavior without full parent coupling Composable, each mixin does one thing Method name collisions, implicit dependencies on host class sklearn's ClassifierMixin adds score() method
Abstract Base Classes (ABC) Enforcing an interface — subclasses MUST implement certain methods Fail-fast if contract not met, clear documentation of required API Can't instantiate directly, slight overhead sklearn's BaseEstimator with @abstractmethod fit
Composition "Has-a" not "is-a" — object contains other objects as attributes Loose coupling, easy to swap parts, testable More boilerplate, no automatic method delegation Pipeline([('scaler', StandardScaler()), ('clf', SVC())])
Default to composition. Use inheritance only when a framework requires it (PyTorch modules, sklearn estimators) or when you genuinely need polymorphism. Most "clever" inheritance hierarchies in application code become maintenance nightmares.

Decorators on Classes — The Decision Triangle

Three decorators, three different use cases. The confusion is always "which one do I use?" — this table resolves it permanently.

DecoratorWhat It DoesReceivesWhen to Use Over OthersML Example
@property Makes a method look like an attribute — computed on access, no () needed self When the value depends on instance state and should feel like an attribute. Validation on set. Lazy computation model.num_params — computed from layer sizes
@classmethod Bound to the class, not the instance — receives cls as first arg cls Alternative constructors (factory methods). When you need to create instances differently from __init__ Model.from_pretrained("bert-base")
@staticmethod Just a function namespaced inside the class — no access to self or cls Nothing Utility that logically belongs to the class but doesn't need instance/class state. Often a code smell — consider a module-level function instead Tokenizer.is_valid_token(t)
Decision rule: Need instance state? → regular method. Need to compute a "virtual attribute"? → @property. Need an alternative constructor? → @classmethod. Need neither instance nor class? → probably a module function, but @staticmethod if you want namespacing.

Design Patterns in ML Frameworks

You don't need to memorize GoF patterns. But you do need to recognize these — they explain why framework APIs look the way they do.

PatternWhere You See ItWhy It Exists
Template Method sklearn's fit()/predict(): base class defines the skeleton, subclass overrides specific steps Consistent API across 100+ estimators — users always call .fit(X, y) regardless of algorithm
Strategy Passing loss_fn, optimizer, scheduler as objects to a trainer Swap algorithms without changing the training loop — open/closed principle
Observer / Callback Keras callbacks=[EarlyStopping(), ModelCheckpoint()], PyTorch Lightning callbacks Decouple training logic from monitoring/logging/saving — add behaviors without editing the loop
Factory Method Model.from_pretrained(), Dataset.from_dict(), Tokenizer.from_file() Multiple ways to construct an object — each @classmethod handles a different source format
Registry HuggingFace model registry, BACKBONE_REGISTRY["resnet50"], LOSS_REGISTRY Map string names to classes — enables config-driven instantiation without massive if-else chains
Composite nn.Sequential, sklearn Pipeline, nn.ModuleList Treat a group of components as a single component — uniform interface for one or many

Code: sklearn-Style Estimator

The sklearn API contract: __init__ stores hyperparams, fit() learns from data and sets attributes ending in _, predict() uses them. Inheriting BaseEstimator gives you get_params()/set_params() for free.

from sklearn.base import BaseEstimator, ClassifierMixin
from sklearn.utils.validation import check_X_y, check_is_fitted
import numpy as np

class MeanClassifier(BaseEstimator, ClassifierMixin):
    """Predicts based on distance to class means. Minimal but sklearn-compatible."""

    def __init__(self, threshold=0.5):
        self.threshold = threshold       # __init__ stores hyperparams ONLY

    def fit(self, X, y):
        X, y = check_X_y(X, y)          # validation — sklearn convention
        self.classes_ = np.unique(y)
        self.means_ = {c: X[y == c].mean(axis=0) for c in self.classes_}
        return self                      # fit() MUST return self for pipelines

    def predict(self, X):
        check_is_fitted(self)            # raises if fit() not called
        dists = {c: np.linalg.norm(X - m, axis=1) for c, m in self.means_.items()}
        return self.classes_[np.argmin(list(dists.values()), axis=0)]

    def __repr__(self):
        return f"MeanClassifier(threshold={self.threshold})"
Convention: Fitted attributes end with _ (e.g., means_, classes_). This lets check_is_fitted() auto-detect whether fit() was called.

Code: PyTorch-Style Module

PyTorch's contract: subclass nn.Module, define layers in __init__, implement forward(). Never call forward() directly — call the instance.

import torch
import torch.nn as nn

class SimpleClassifier(nn.Module):
    def __init__(self, input_dim, hidden_dim, num_classes):
        super().__init__()                         # MUST call — registers params
        self.net = nn.Sequential(
            nn.Linear(input_dim, hidden_dim),
            nn.ReLU(),
            nn.Dropout(0.3),
            nn.Linear(hidden_dim, num_classes),
        )

    def forward(self, x):                          # called by __call__
        return self.net(x)

# Usage — note: model(x), NOT model.forward(x)
model = SimpleClassifier(784, 256, 10)
print(model)                                       # __repr__ auto-generated by nn.Module
print(sum(p.numel() for p in model.parameters()))  # parameter count

Interview Crux Table

The questions interviewers love because they expose shallow understanding. After this table, they should feel trivial.

Tricky QuestionAnswerWhy It Trips People
What is MRO? Method Resolution Order — the order Python searches classes for a method. Uses C3 linearization. Check with MyClass.__mro__ With multiple inheritance, the order isn't obvious. Diamond inheritance makes it worse. Python's C3 algorithm guarantees a consistent, predictable order — but you must know it exists
Mutable default in __init__? def __init__(self, items=[]) — all instances share the SAME list. Fix: items=None, then self.items = items or [] Default args are evaluated ONCE at function definition, not at each call. The same list object is reused
Mutable class attribute? class C: data = [] — shared across all instances. C().data.append(1) affects every instance People confuse "class attribute" with "default value." Class attrs live on the class object, not on instances
__new__ vs __init__ __new__ creates the instance (allocates memory, returns the object). __init__ initializes it (sets attributes). __new__ runs FIRST __new__ is rarely overridden — mainly for immutable types (int, str), singletons, or metaclass tricks. 99% of the time you only need __init__
is vs == is checks identity (same object in memory). == checks equality (same value via __eq__) CPython caches small ints (-5 to 256) and short strings, so a is b sometimes works by accident for those values, then fails for larger numbers
Shallow vs deep copy copy(): new outer container, same inner objects. deepcopy(): recursively copies everything. Assignment (b = a): no copy at all — alias Shallow copy of a model config with nested dicts means changing an inner dict changes both copies. Use deepcopy for configs with nested structures
super() in multiple inheritance super() follows MRO, not "parent class." In a diamond, super().__init__() calls the NEXT class in MRO, which may not be the direct parent People assume super() = "call my parent." It actually means "call the next class in the MRO chain"
Why can't you use a list as a dict key? Lists are mutable → unhashable. Dict keys must be hashable (immutable). Use tuples instead The connection between hashability, mutability, and dict/set membership isn't taught together
