Object-Oriented Programming

Chapter 1: Python & Programming Foundations 7 subtopics

Every ML framework you'll touch runs on OOP. model.fit(X, y) is a method call on a BaseEstimator subclass. model(input) triggers __call__ on nn.Module. Custom Dataset classes require you to implement __len__ + __getitem__. This isn't "what is a class" — it's how to read and extend class hierarchies fluently, because that's the daily work of production ML.

The Basics You Already Know

Classes, __init__, self, instance vs class attributes, methods — you've seen these. If you're solid, skip ahead. One gotcha worth pausing on:

Mutable class attributes are shared across all instances. class Model: layers = [] means every instance appends to the same list. Always initialize mutable state in __init__: self.layers = []. Same trap with mutable default arguments — def __init__(self, items=[]) reuses the same list object across calls. Fix: items=None, then self.items = items or [].

Inheritance vs Composition — The Actual Decision

Here's the thing about inheritance — in a textbook, it's the star of OOP. In production, composition wins 80% of the time. The exceptions are when you're extending frameworks that require you to subclass: nn.Module, BaseEstimator, PreTrainedModel. Those are "inherit because the framework says so" situations, and they're well-designed. The trouble starts when you build your own three-level deep class hierarchy because it felt clean at the time.

When to inherit: when a framework gives you a base class with hooks you need to fill in. Sklearn's BaseEstimator gives you get_params()/set_params() for free. PyTorch's nn.Module handles parameter registration, device movement, and gradient tracking. HuggingFace's Dataset expects __len__ and __getitem__. In all these cases, the framework is doing real work for you in exchange for a subclass contract.

When to compose: when you're building your own system. "Has-a" beats "is-a" almost always. A training pipeline has a model, a loss function, and an optimizer — it isn't any of those things. Composition means loose coupling, easy swapping, straightforward testing. The sklearn Pipeline is itself a masterclass in composition: Pipeline([('scaler', StandardScaler()), ('clf', SVC())]).

The Diamond Problem and MRO

Multiple inheritance introduces the diamond problem: class D inherits from B and C, which both inherit from A. When D calls super().__init__(), which path does it take? Python solves this with the Method Resolution Order (MRO), computed via C3 linearization. The key insight: super() doesn't mean "call my parent." It means "call the next class in the MRO chain."

class A:
    def __init__(self):
        print("A")

class B(A):
    def __init__(self):
        print("B")
        super().__init__()    # calls C, NOT A — surprise

class C(A):
    def __init__(self):
        print("C")
        super().__init__()    # calls A

class D(B, C):
    def __init__(self):
        print("D")
        super().__init__()    # calls B

D()   # prints: D → B → C → A
print(D.__mro__)
# (D, B, C, A, object) — this IS the resolution order

The rule: every class in the MRO appears exactly once, and a class always appears before its parents. When you see class SGDClassifier(BaseSGD, ClassifierMixin, RegressorMixin) in sklearn's source, this is the machinery that makes it work without ambiguity.

Mixins

Mixins are small classes that add a single slice of behavior — they're not meant to be instantiated alone. Sklearn uses them everywhere: ClassifierMixin adds a score() method, TransformerMixin adds fit_transform(). The convention: a mixin provides methods that assume the host class has certain attributes, but it doesn't define __init__ itself. When you need to add logging, serialization, or a standard interface to multiple unrelated classes, mixins are the right pattern.

Abstract Base Classes

When you're building a system where multiple people write plugins or extensions, abc.ABC enforces the contract. If a subclass forgets to implement a required method, it fails at instantiation — not silently at runtime two hours into a training run.

from abc import ABC, abstractmethod

class BasePreprocessor(ABC):
    @abstractmethod
    def transform(self, X):
        """Subclasses MUST implement this."""
        ...

    def fit_transform(self, X, y=None):
        return self.fit(X, y).transform(X)    # template method

class MissingImpl(BasePreprocessor):
    pass

# MissingImpl()  → TypeError: Can't instantiate abstract class
# Fails NOW, not when .transform() is called 3 hours later

The Pattern You'll Actually Write

This is the single most common OOP pattern in ML codebases — extending sklearn's BaseEstimator with a mixin to create a custom transformer that plugs into any pipeline:

from sklearn.base import BaseEstimator, TransformerMixin
from sklearn.utils.validation import check_is_fitted
import numpy as np

class ColumnDropper(BaseEstimator, TransformerMixin):
    """Drops columns by index. Pipeline-compatible, clone-safe, grid-searchable."""

    def __init__(self, columns=None):
        self.columns = columns               # hyperparams ONLY in __init__

    def fit(self, X, y=None):
        self.n_features_in_ = X.shape[1]     # fitted attrs end with _
        return self                           # MUST return self

    def transform(self, X):
        check_is_fitted(self)
        mask = [i for i in range(X.shape[1]) if i not in (self.columns or [])]
        return X[:, mask] if hasattr(X, '__array__') else X.iloc[:, mask]

# drops right into a pipeline:
# Pipeline([('drop', ColumnDropper(columns=[0, 3])), ('clf', SVC())])
# works with GridSearchCV: param_grid={'drop__columns': [[0], [0, 3]]}
Why this works: BaseEstimator gives you get_params()/set_params() by introspecting __init__'s signature. TransformerMixin gives you fit_transform() for free. Fitted attributes ending in _ let check_is_fitted() auto-detect whether fit() was called. Follow these three conventions and your transformer works with Pipeline, GridSearchCV, clone(), and cross_val_score — zero extra effort.

Dunder Methods — Only The Ones That Matter

Python has 100+ dunder methods. You'll implement maybe 8 of them in your career. Here's which ones, when, and why — skip the rest until you need them.

__init__(self, ...) — object setup. You already know this. Store hyperparams, declare layers, set defaults. The one rule that matters: don't do heavy computation here. Sklearn expects __init__ to only store parameters, nothing else.

__repr__(self) — debugging representation. Called by repr() and the REPL. If you define only one string method, define this one — Python uses it as fallback for str() too. Sklearn auto-generates these from get_params(), which is why printing a model shows RandomForestClassifier(n_estimators=100, ...).

def __repr__(self):
    return f"Config(lr={self.lr}, epochs={self.epochs})"

__len__(self) + __getitem__(self, idx) — the Dataset pattern. PyTorch's DataLoader needs __len__ to know how many samples exist and __getitem__ to fetch one by index. Implement these two and your class works with shuffling, batching, and multi-worker loading for free.

class TextDataset:
    def __init__(self, texts, labels):
        self.texts = texts
        self.labels = labels

    def __len__(self):
        return len(self.texts)

    def __getitem__(self, idx):
        return self.texts[idx], self.labels[idx]

# DataLoader(TextDataset(texts, labels), batch_size=32, shuffle=True)
# Just works. Batching, shuffling, multiprocess — all from two methods.

__call__(self, ...) — the big one. Makes an instance callable like a function. PyTorch's nn.Module.__call__ runs forward hooks, sets up autograd, calls your forward(), then runs post-forward hooks. This is why you write model(x) and never model.forward(x) — calling forward() directly skips gradient tracking and silently breaks training.

__enter__(self) / __exit__(self, ...) — context managers. torch.no_grad(), file handles, database connections, timers. The pattern: __enter__ sets up a resource, __exit__ tears it down — even if an exception fires. For simple cases, contextlib.contextmanager with a generator is less boilerplate.

class Timer:
    def __enter__(self):
        self.start = time.perf_counter()
        return self

    def __exit__(self, *exc):
        self.elapsed = time.perf_counter() - self.start
        print(f"{self.elapsed:.3f}s")
        return False              # don't suppress exceptions

# with Timer():
#     model.fit(X, y)             # prints elapsed time even if fit() crashes

__eq__(self, other) / __hash__(self) — only when you need objects as dict keys or set members. Implement __eq__ for config comparison. But beware: defining __eq__ sets __hash__ to None, making your objects unhashable. If you need them in sets or as dict keys, define both. Most of the time, you don't need either — dataclass(frozen=True) generates both automatically.

Properties and Descriptors

@property makes a method behave like an attribute — computed on access, validated on set. Use it for derived values (model.num_params computed from layer sizes) or when setting an attribute needs validation (self.learning_rate must be positive). That's the 90% use case.

class Config:
    def __init__(self, lr=0.001):
        self.lr = lr                # triggers the setter below

    @property
    def lr(self):
        return self._lr

    @lr.setter
    def lr(self, value):
        if value <= 0:
            raise ValueError(f"lr must be positive, got {value}")
        self._lr = value

Descriptors are the mechanism that powers @property under the hood — any object with __get__, __set__, or __delete__ methods. You'll almost never write one directly, but knowing they exist explains why @property, @classmethod, and @staticmethod all work the way they do.

Dataclasses

@dataclass is the modern way to create classes that mainly hold data. It auto-generates __init__, __repr__, and __eq__ from your field annotations — eliminating the boilerplate that makes simple config or record classes tedious to write.

from dataclasses import dataclass, field

@dataclass
class TrainConfig:
    lr: float = 0.001
    epochs: int = 100
    batch_size: int = 32
    tags: list = field(default_factory=list)    # mutable default done right

cfg = TrainConfig(lr=0.01)
print(cfg)          # TrainConfig(lr=0.01, epochs=100, batch_size=32, tags=[])
cfg2 = TrainConfig(lr=0.01)
cfg == cfg2          # True — __eq__ compares all fields

Add frozen=True for immutable config objects — attempts to modify fields raise FrozenInstanceError. This also auto-generates __hash__, so frozen dataclasses work as dict keys and set members out of the box.

ChoiceWhenTrade-off
@dataclass Config objects, records, anything with multiple named fields Full class power — methods, inheritance, mutability control
NamedTuple Lightweight immutable records, function return types Immutable by default, tuple unpacking, less overhead
Plain class Significant custom logic in __init__, complex inheritance Full control, but you write all the boilerplate yourself

OOP Patterns in ML Frameworks

You don't need to memorize design patterns from a textbook. But you do need to recognize the patterns that ML frameworks are built on — so when you read their source code, you know what's happening.

sklearn: fit / transform / predict

Every sklearn estimator follows the same contract: __init__ stores hyperparams (nothing else), fit() learns from data and sets attributes ending in _, predict() or transform() uses them. BaseEstimator gives you get_params()/set_params() by introspecting your __init__ signature — this is what makes GridSearchCV and clone() work. check_is_fitted() verifies that fit() was called by looking for trailing-underscore attributes. Break any of these conventions and your estimator silently fails in pipelines.

PyTorch: nn.Module

Subclass nn.Module, define layers in __init__ (calling super().__init__() first), implement forward(). The module tracks every nn.Parameter and sub-module you assign as attributes — that's how model.parameters(), model.to(device), and model.state_dict() work automatically. Hooks let you inject behavior before or after forward() without modifying it — gradient clipping, activation logging, feature extraction from intermediate layers.

import torch.nn as nn

class SimpleNet(nn.Module):
    def __init__(self, in_dim, hidden, out_dim):
        super().__init__()                      # registers params
        self.net = nn.Sequential(
            nn.Linear(in_dim, hidden),
            nn.ReLU(),
            nn.Dropout(0.3),
            nn.Linear(hidden, out_dim),
        )

    def forward(self, x):                       # called by __call__
        return self.net(x)

model = SimpleNet(784, 256, 10)
# model(x) — NOT model.forward(x)
# model.parameters() — auto-discovered from self.net
# model.to('cuda') — moves everything, recursively

HuggingFace: PreTrainedModel and AutoModel

HuggingFace builds on two patterns: factory methods and a registry. Model.from_pretrained("bert-base-uncased") is a @classmethod that downloads weights, instantiates the right architecture, and loads the state dict. AutoModel.from_pretrained() adds a registry layer — it reads the config to figure out which class to instantiate, so you don't need to import BertModel vs GPT2Model vs T5Model yourself. The same pattern appears with AutoTokenizer, AutoConfig, and AutoModelForSequenceClassification. When you see "Auto" in HuggingFace, think "registry that maps config strings to classes."
