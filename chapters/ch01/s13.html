Python Under the Hood

Chapter 1: Python & Programming Foundations 1 table · 3 code blocks

Your training script is slow. Your inference server leaks memory. Your pipeline uses one core on an 8-core machine. Understanding what's happening underneath — memory management, the GIL, concurrency — is how you diagnose these. This isn't academic. This is Tuesday.

The Decision Matrix — Check This Before Optimizing Anything

Using the wrong concurrency tool doesn't just waste time — it makes things worse. Threading a CPU-bound workload adds overhead with zero speedup. Multiprocessing an I/O-bound task burns memory for nothing. Identify the bottleneck first, then pick.

BottleneckExampleUseWhyGotcha
CPU-bound Feature engineering, tokenization, image resizing multiprocessing / ProcessPoolExecutor Separate process = separate GIL = true parallelism Data must be serialized (pickled) across process boundaries. Keep payloads small.
I/O-bound API calls, file downloads, database queries, S3 reads threading / ThreadPoolExecutor or asyncio GIL is released during I/O — threads genuinely overlap Don't use threads for CPU work. The GIL means they'll take turns, not run in parallel.
Memory-bound Large DataFrames, huge arrays, training data that barely fits in RAM Memory profiling + optimization Find the leak or bloat first — can't fix what you can't see Premature optimization wastes time. Profile, then optimize.
Quick Test

Not sure what type? Profile with cProfile. If inserting time.sleep(1) in your inner loop barely changes wall-clock time, you're I/O-bound. If it adds exactly N × 1s, you're CPU-bound.

Memory Model — How Python Manages Objects

Here's the mental model that makes everything else click: everything in Python is an object, and every variable is a pointer to an object. When you write a = [1, 2, 3], the list lives somewhere on the heap, and a is just a name tag pointing to it. When you write b = a, you don't copy the list — you create a second name tag pointing to the same list. One list, two names.

This matters enormously for ML work. df2 = df doesn't copy your 2GB DataFrame — it gives you two variables pointing at the same data. Mutate through df2 and you've changed df too. If you want an actual copy, you need df.copy(). Same story with NumPy: slicing returns a view (shared memory), not a copy. Modifying a view modifies the original.

Reference Counting

Python tracks how many names point to each object. Every object has a hidden counter. Every time a new variable points to it, the count goes up. Every time a variable is deleted or reassigned, it goes down. The moment the count hits zero, the memory is freed — immediately, no delay.

This is why del df often frees memory instantly. The reference count drops to zero, Python reclaims the memory, done. But there's a catch: if two objects point to each other (A references B, B references A), neither count ever reaches zero, even if nothing else in your program uses them.

The Garbage Collector

That's where the cyclic garbage collector comes in. It periodically scans for these reference cycles and cleans them up. It runs in three generations — young objects get checked frequently, old objects rarely. For most code you never think about it. The exception: if you've called gc.disable() to speed up a training loop (a real pattern), you need to call gc.collect() between epochs or those circular references pile up.

Why This Matters for ML

The single most common memory bug in ML pipelines: accidentally keeping references to old DataFrames. You load a dataset, transform it three times, and now you have df_raw, df_cleaned, df_featured, and df_final all alive in memory. The earlier versions aren't freed because you still have names pointing to them. Delete or overwrite them explicitly.

And here's a subtlety that trips everyone up at least once: sys.getsizeof() lies. It reports the shallow size of the container object itself — not the things inside it. sys.getsizeof(my_list) on a list of a million dicts returns the size of the list's internal pointer array (~8MB), not the dicts themselves (potentially hundreds of MB). For real numbers, use memory_profiler or df.info(memory_usage='deep') for DataFrames.

Production Pattern

Convert low-cardinality object columns to category dtype: df['country'] = df['country'].astype('category'). A column with 50 unique country codes stored as strings across 10M rows drops from ~600MB to ~10MB. Check with df.info(memory_usage='deep').

The GIL — What It Actually Is

Python has a global lock — the Global Interpreter Lock — that says only one thread can execute Python bytecode at a time. This sounds terrible, and for CPU-bound code it is. But there's a crucial exception: the GIL is released during I/O operations. Network calls, disk reads, database queries — while one thread is waiting on I/O, another thread can run Python code. That's not a loophole. It's the design.

Why does it exist? CPython's memory management uses reference counting — every object tracks how many names point to it. Without the GIL, two threads decrementing a reference count simultaneously could corrupt memory. The lock is the price of simple, fast single-threaded performance and safe C extension integration.

Here's the thing about the GIL that everyone complains about but most of the time doesn't matter: your ML code is calling NumPy and PyTorch, which release the GIL during their C-level computation anyway. When NumPy is crunching a matrix multiplication, it's running C code with the GIL released. Your model.fit() in sklearn or XGBoost already uses multiprocessing internally via n_jobs. The GIL is not your bottleneck during training.

Where the GIL does bite: pure Python loops doing computation. If you've written a for loop that processes 10 million rows in Python (not vectorized through NumPy/Pandas), threading won't help — threads will take turns on that GIL, not run in parallel. Two options: vectorize the operation (first choice), or use multiprocessing — each process gets its own interpreter with its own GIL.

The future: Python 3.13+ has an experimental free-threaded mode (PEP 703) that removes the GIL entirely. It's opt-in, it breaks many C extensions as of early 2025, and production readiness is years away. Don't plan around it. Use multiprocessing for CPU-bound work today.

Don't fear the GIL. Understand it, and you'll know exactly when threading works and when it doesn't.

Concurrency Patterns — Threading vs Multiprocessing vs Async

Three tools, three use cases. No need to memorize a table — the patterns are the teaching.

Threading — concurrent.futures.ThreadPoolExecutor. Use for parallel I/O: downloading files, calling APIs, reading from databases. Threads share memory, so there's no serialization overhead.

from concurrent.futures import ThreadPoolExecutor
import requests

def fetch(url: str) -> str:
    return requests.get(url, timeout=10).text

urls = ["https://api.example.com/data/1", "https://api.example.com/data/2", ...]

# 20 threads, 20 concurrent downloads. GIL doesn't matter — they're waiting on network.
with ThreadPoolExecutor(max_workers=20) as executor:
    results = list(executor.map(fetch, urls))

Multiprocessing — concurrent.futures.ProcessPoolExecutor. Use for CPU-bound work: feature engineering, image preprocessing, tokenization. Each process gets its own GIL.

from concurrent.futures import ProcessPoolExecutor, as_completed
from pathlib import Path
import json

def preprocess_file(path: str) -> dict:
    """CPU-bound: parse JSON, extract features, return summary."""
    with open(path) as f:
        raw = json.load(f)
    return {
        "n_tokens": len(raw["text"].split()),
        "has_label": raw.get("label") is not None,
        "path": path,
    }

paths = [str(p) for p in Path("data/raw").glob("*.json")]

with ProcessPoolExecutor(max_workers=8) as executor:
    # submit() + as_completed() > map() — one slow file doesn't block everything
    futures = {executor.submit(preprocess_file, p): p for p in paths}
    for future in as_completed(futures):
        try:
            result = future.result()
        except Exception as e:
            print(f"Failed {futures[future]}: {e}")  # one file fails, rest continues

The key design detail: ThreadPoolExecutor and ProcessPoolExecutor share the exact same interface. Start with threads. If profiling shows CPU is the bottleneck, swap to ProcessPoolExecutor — one import change.

Async — asyncio. Use when you need thousands of concurrent I/O operations. Web scraping, ML API serving, streaming. Threads work fine for tens to hundreds of concurrent tasks, but at thousands, thread overhead adds up and async's single-threaded event loop wins.

Gotcha

Multiprocessing serializes data between processes using pickle. A 500MB DataFrame being sent to 8 workers means 8 copies × 500MB = 4GB of memory just for IPC. Keep payloads small — send file paths or indices, not data.

Async/Await — Just Enough to Read Async Code

Async is cooperative multitasking. Functions voluntarily yield control with await, letting other tasks run during I/O waits. No threads, no locks, no race conditions. One thread juggles thousands of tasks by switching between them at await points.

When to use: high-concurrency I/O — serving ML models via FastAPI, scraping hundreds of endpoints, streaming responses. When NOT to use: CPU-bound work. Async doesn't magically parallelize computation. It's still one thread — it just uses idle I/O time efficiently.

import asyncio
from concurrent.futures import ProcessPoolExecutor
from fastapi import FastAPI
import numpy as np

app = FastAPI()
executor = ProcessPoolExecutor(max_workers=4)

def predict_sync(features: list[float]) -> float:
    """Sync, CPU-bound — cannot be awaited directly."""
    X = np.array(features).reshape(1, -1)
    return float(X.sum() * 0.42)  # replace with model.predict(X)[0]

@app.post("/predict")
async def predict(features: list[float]):
    loop = asyncio.get_event_loop()
    # Offload CPU work to process pool — event loop stays free for other requests
    result = await loop.run_in_executor(executor, predict_sync, features)
    return {"prediction": result}

# FastAPI handles 1000+ concurrent requests.
# Without run_in_executor, one slow prediction blocks ALL of them.

This is the production pattern: async handles the HTTP traffic, run_in_executor offloads the actual computation. The event loop is never blocked.

What You Should Now Be Able To Do

  • Look at a slow Python workload and classify it as CPU-bound, I/O-bound, or memory-bound — then pick the right concurrency tool without guessing.
  • Explain why b = a doesn't copy and why df2 = df gives you two names for the same data — and when that causes bugs.
  • Explain what the GIL is, why it exists, and correctly predict whether threads or processes will help a given workload.
  • Use ProcessPoolExecutor with as_completed() to parallelize CPU-bound work with per-item error handling.
  • Use ThreadPoolExecutor to overlap I/O-bound tasks like downloading datasets or calling APIs.
  • Write an async FastAPI endpoint that offloads sync ML inference to a process pool via run_in_executor().
  • Spot the most common ML memory bug — dangling references to old DataFrames — and know why sys.getsizeof() won't help you find it.
