Pandas

Chapter 1: Python & Programming Foundations 8 subtopics
TL;DR

Pandas gives you a spreadsheet-on-steroids inside Python — the DataFrame is your universal container for tabular data. You select rows and columns with loc/iloc, clean missing values with fillna/dropna, reshape data with groupby and pivot tables, merge datasets with SQL-style joins, and read/write every format from CSV to SQL. Master these operations and you'll spend less time wrestling data and more time building models.

Every machine learning project starts the same way: you get a messy CSV, an Excel dump from a stakeholder, or a database export full of nulls and inconsistent column names. Before any model touches that data, someone has to load it, inspect it, clean it, merge it with other sources, and reshape it into the format the model expects. That someone is you, and your primary weapon is Pandas — a Python library that turns tabular data into something you can slice, dice, aggregate, and transform with a few lines of code. If NumPy is for raw numerical arrays, Pandas is for labeled, heterogeneous, real-world data. It's not optional. It's foundational.

Core Data Structures

Series

A Series is a one-dimensional labeled array. Think of it as a single column from a spreadsheet — it has values and an index that labels each value. Under the hood, a Series wraps a NumPy array, so you get vectorized operations for free. But unlike a raw array, a Series carries a name and an index, which means you can align data by label, not position.

import pandas as pd
import numpy as np

# A Series of house prices, labeled by address
prices = pd.Series(
    [350_000, 420_000, 289_000, 510_000],
    index=["101 Oak St", "202 Elm Ave", "303 Pine Rd", "404 Maple Dr"],
    name="sale_price"
)

print(prices)
# 101 Oak St     350000
# 202 Elm Ave    420000
# 303 Pine Rd    289000
# 404 Maple Dr   510000
# Name: sale_price, dtype: int64

# Vectorized operations work like NumPy
print(prices.mean())          # 392250.0
print(prices > 400_000)       # Boolean Series

The key insight: when you pull a column out of a DataFrame, you get a Series. When you apply an aggregation inside a groupby, you often get a Series back. It's the atomic unit of Pandas.

DataFrame

A DataFrame is a two-dimensional labeled data structure — rows and columns, each with labels. It's the workhorse. Every dataset you load becomes a DataFrame. Each column is a Series, and they can hold different types: integers, floats, strings, dates, booleans, even categorical data. This is the critical difference from NumPy arrays, which demand a single dtype.

# A small housing dataset — the kind you'd actually work with
df = pd.DataFrame({
    "address": ["101 Oak St", "202 Elm Ave", "303 Pine Rd", "404 Maple Dr"],
    "sqft": [1500, 2100, 1200, 2800],
    "bedrooms": [3, 4, 2, 5],
    "sale_price": [350_000, 420_000, 289_000, 510_000],
    "has_garage": [True, True, False, True]
})

print(df.shape)    # (4, 5)
print(df.dtypes)
# address        object
# sqft            int64
# bedrooms        int64
# sale_price      int64
# has_garage       bool

Index Objects

The Index is what makes Pandas different from a plain 2D array. Every DataFrame has a row index and column labels. By default, rows get integer indices (0, 1, 2, ...), but you can set any column as the index. The index is immutable and hashable, which makes label-based lookups fast. It also drives automatic alignment — when you add two DataFrames, Pandas matches rows by index, not position. This prevents a whole class of bugs where data gets silently misaligned.

Creating DataFrames

You'll create DataFrames from four main sources in practice. From a dictionary of lists (as shown above) for quick prototyping. From a list of dictionaries when each record is a dict. From CSV files — by far the most common in ML work. And from databases or Excel when dealing with enterprise data.

# From a list of dicts (common when parsing API responses)
records = [
    {"patient_id": "P001", "age": 45, "bp_systolic": 130},
    {"patient_id": "P002", "age": 62, "bp_systolic": None},  # missing!
    {"patient_id": "P003", "age": 38, "bp_systolic": 118},
]
df_patients = pd.DataFrame(records)

# From CSV (the bread-and-butter of ML)
# df = pd.read_csv("house_prices.csv")

# From Excel
# df = pd.read_excel("clinical_trial.xlsx", sheet_name="results")

Data Selection and Indexing

You have a DataFrame with 50 columns and 100,000 rows. Now what? You need to grab specific slices. This is where most beginners stumble, because Pandas offers multiple ways to select data and the differences matter.

Column Selection

Grab a single column with bracket notation — you get a Series. Grab multiple columns with a list of names — you get a DataFrame. That's it.

# Single column → Series
prices = df["sale_price"]

# Multiple columns → DataFrame
subset = df[["address", "sale_price", "sqft"]]

# Dot notation works too, but breaks on column names with spaces
df.sqft  # same as df["sqft"], but less flexible

Row Selection with loc and iloc

This is the single most important distinction in Pandas indexing. loc selects by label. iloc selects by integer position. Confuse them and you'll get wrong results with no error — the worst kind of bug.

# Set address as index to illustrate the difference
df_indexed = df.set_index("address")

# loc: selection by LABEL
df_indexed.loc["101 Oak St"]              # single row by label
df_indexed.loc["101 Oak St":"303 Pine Rd"]  # label slice (INCLUSIVE on both ends!)
df_indexed.loc["101 Oak St", "sqft"]      # single value: row label, column label

# iloc: selection by INTEGER POSITION
df_indexed.iloc[0]           # first row (position 0)
df_indexed.iloc[0:2]         # positions 0 and 1 (EXCLUSIVE end, like Python)
df_indexed.iloc[0, 1]        # row 0, column 1
⚠️ The loc/iloc Trap

loc slices are inclusive on both ends: loc["a":"c"] includes "c". iloc slices are exclusive on the end, like standard Python: iloc[0:2] gives positions 0 and 1 only. Mixing these up is one of the most common Pandas bugs in data pipelines.

Boolean Indexing

This is how you filter rows by condition — the Pandas equivalent of a SQL WHERE clause. You create a boolean Series and pass it into the brackets.

# Houses over $400k
expensive = df[df["sale_price"] > 400_000]

# Combine conditions with & (and), | (or), ~ (not)
# Parentheses are REQUIRED around each condition
big_and_expensive = df[(df["sqft"] > 2000) & (df["sale_price"] > 400_000)]

# Filter using .isin() for categorical matching
target_streets = df[df["address"].isin(["101 Oak St", "404 Maple Dr"])]

Setting and Resetting Index

Use set_index() to promote a column to the index. Use reset_index() to push the index back into a column. You'll do this constantly — after a groupby aggregation, the grouping column becomes the index, and you often want it back as a regular column.

df_indexed = df.set_index("address")
df_flat = df_indexed.reset_index()  # address is a column again

MultiIndex

A MultiIndex (hierarchical index) lets you have multiple levels of row or column labels. It's powerful for representing structured data — like measurements across multiple patients over multiple visits.

# Create MultiIndex from columns
visits = pd.DataFrame({
    "patient": ["P001", "P001", "P002", "P002"],
    "visit": [1, 2, 1, 2],
    "bp_systolic": [130, 125, 145, 140],
    "bp_diastolic": [85, 80, 92, 88],
})

visits_mi = visits.set_index(["patient", "visit"])
print(visits_mi.loc["P001"])         # all visits for P001
print(visits_mi.loc[("P001", 2)])    # specific patient + visit

Data Manipulation

Now we reshape, transform, and engineer features. This is where you turn raw data into something a model can use.

Adding and Removing Columns

# Add a price-per-sqft feature (useful for ML)
df["price_per_sqft"] = df["sale_price"] / df["sqft"]

# Add a column with a constant
df["country"] = "US"

# Drop columns you don't need
df_clean = df.drop(columns=["has_garage", "country"])

Renaming Columns

Messy column names from CSVs are the norm. Rename them early to keep your code readable.

# Rename specific columns
df = df.rename(columns={"sqft": "square_feet", "sale_price": "price_usd"})

# Bulk rename: lowercase everything, replace spaces
df.columns = df.columns.str.lower().str.replace(" ", "_")

Sorting

# Sort by values — most expensive houses first
df_sorted = df.sort_values("sale_price", ascending=False)

# Sort by multiple columns
df_sorted = df.sort_values(["bedrooms", "sale_price"], ascending=[True, False])

# Sort by index
df_indexed.sort_index(inplace=True)

Applying Functions

Three tools here: apply runs a function along an axis (row or column). map applies element-wise on a Series. applymap (or map on DataFrame in newer Pandas) applies element-wise on an entire DataFrame. Use apply most often — it's the Swiss army knife.

# apply on a column (Series) — classify price tier
def price_tier(price):
    if price > 450_000:
        return "premium"
    elif price > 300_000:
        return "mid"
    return "affordable"

df["tier"] = df["sale_price"].apply(price_tier)

# apply on rows — combine multiple columns
df["description"] = df.apply(
    lambda row: f"{row['bedrooms']}BR, {row['sqft']}sqft", axis=1
)

# map for simple value replacement
df["has_garage_str"] = df["has_garage"].map({True: "Yes", False: "No"})

String Methods (.str accessor)

When a column contains text, the .str accessor gives you vectorized string operations — no loops needed.

# Extract street type from address
df["street_type"] = df["address"].str.split().str[-1]  # "St", "Ave", etc.

# Check if address contains "Oak"
df["is_oak"] = df["address"].str.contains("Oak")

# Clean up messy strings
df["address"] = df["address"].str.strip().str.title()

DateTime Methods (.dt accessor)

Time-series data is everywhere in ML — user activity logs, stock prices, sensor readings. The .dt accessor lets you extract components from datetime columns.

# Parse dates and extract features
logs = pd.DataFrame({
    "timestamp": ["2024-01-15 08:30:00", "2024-03-22 14:45:00", "2024-07-04 23:10:00"],
    "event": ["login", "purchase", "logout"]
})
logs["timestamp"] = pd.to_datetime(logs["timestamp"])

logs["month"] = logs["timestamp"].dt.month
logs["day_of_week"] = logs["timestamp"].dt.day_name()
logs["hour"] = logs["timestamp"].dt.hour
logs["is_weekend"] = logs["timestamp"].dt.dayofweek >= 5

Handling Missing Data

Real datasets have holes. A patient skipped a blood test. A user didn't fill in their age. A sensor went offline for an hour. How you handle missing values directly affects model quality. Get this wrong and you either throw away good data or introduce bias.

Detecting Missing Values

medical = pd.DataFrame({
    "patient_id": ["P001", "P002", "P003", "P004", "P005"],
    "age": [45, 62, None, 38, 55],
    "bp_systolic": [130, None, 118, 145, None],
    "diagnosis": ["healthy", "diabetes", "healthy", None, "diabetes"]
})

# Detect nulls
print(medical.isnull().sum())   # count of missing per column
# age             1
# bp_systolic     2
# diagnosis       1

# Percentage missing — useful for deciding strategy
print(medical.isnull().mean() * 100)

Dropping Missing Values

The nuclear option. Use it when the percentage of missing data is tiny and dropping rows won't bias your sample.

# Drop any row with at least one null
medical_clean = medical.dropna()

# Drop only rows where specific columns are null
medical_clean = medical.dropna(subset=["bp_systolic"])

# Drop columns with too many nulls (threshold: keep columns with at least 3 non-nulls)
medical_clean = medical.dropna(axis=1, thresh=3)

Filling Missing Values

Usually a better choice than dropping. Fill with a constant, the mean, the median, or the most recent known value (forward fill).

# Fill numeric columns with median (robust to outliers)
medical["age"] = medical["age"].fillna(medical["age"].median())

# Fill with column mean
medical["bp_systolic"] = medical["bp_systolic"].fillna(medical["bp_systolic"].mean())

# Fill categorical with mode (most frequent value)
medical["diagnosis"] = medical["diagnosis"].fillna(medical["diagnosis"].mode()[0])

# Forward fill — useful for time series (carry last known value forward)
sensor_data = pd.Series([22.1, None, None, 23.5, None, 24.0])
sensor_filled = sensor_data.ffill()
# [22.1, 22.1, 22.1, 23.5, 23.5, 24.0]
🚨 Don't Fill Before Splitting

If you fill missing values using statistics (mean, median) computed on the entire dataset before your train/test split, you've leaked test information into training. Compute fill values on the training set only, then apply those same values to the test set.

Interpolation

Interpolation estimates missing values based on surrounding data points. Linear interpolation draws a straight line between known neighbors. It's particularly useful for time-series data where values change smoothly.

# Temperature readings with gaps
temps = pd.Series([20.0, None, None, 26.0, None, 30.0])

# Linear interpolation: fills proportionally between neighbors
print(temps.interpolate())
# 0    20.0
# 1    22.0  ← estimated
# 2    24.0  ← estimated
# 3    26.0
# 4    28.0  ← estimated
# 5    30.0

Grouping and Aggregation

You have 100,000 rows of user behavior data. You don't care about individual rows — you want the average session duration per user, the total revenue per region, the max temperature per day. This is the split-apply-combine pattern, and groupby is the tool.

groupby Operations

sales = pd.DataFrame({
    "region": ["West", "East", "West", "East", "West", "East"],
    "product": ["Widget", "Widget", "Gadget", "Gadget", "Widget", "Widget"],
    "revenue": [1200, 900, 1800, 1500, 1100, 950],
    "units": [10, 8, 12, 10, 9, 7]
})

# Basic groupby: split by region, compute mean of each numeric column
print(sales.groupby("region").mean(numeric_only=True))
#         revenue  units
# East    1116.67   8.33
# West    1366.67  10.33

# Group by multiple columns
print(sales.groupby(["region", "product"]).sum(numeric_only=True))

Aggregation Functions (agg)

The agg method gives you fine-grained control — different aggregation functions per column, or multiple aggregations at once.

# Different functions per column
summary = sales.groupby("region").agg(
    total_revenue=("revenue", "sum"),
    avg_revenue=("revenue", "mean"),
    total_units=("units", "sum"),
    num_transactions=("revenue", "count")
)
print(summary)

Transform

transform returns a result the same shape as the input — no aggregation. It's perfect for group-level normalization, a common preprocessing step in ML.

# Normalize revenue within each region (z-score per group)
sales["revenue_zscore"] = sales.groupby("region")["revenue"].transform(
    lambda x: (x - x.mean()) / x.std()
)

# Percentage of regional total
sales["pct_of_region"] = sales.groupby("region")["revenue"].transform(
    lambda x: x / x.sum()
)

Pivot Tables

A pivot table restructures your data — rows become one dimension, columns become another, and values are aggregated. It's the same concept as Excel pivot tables.

pivot = sales.pivot_table(
    values="revenue",
    index="region",
    columns="product",
    aggfunc="sum",
    fill_value=0
)
print(pivot)
# product  Gadget  Widget
# region
# East       1500    1850
# West       1800    2300

Crosstab

crosstab is a shortcut for frequency tables. Useful for quick exploration of how two categorical features relate.

print(pd.crosstab(sales["region"], sales["product"]))
# product  Gadget  Widget
# region
# East          1       2
# West          1       2

Melt and Stack/Unstack

melt transforms wide-format data into long-format — a common requirement for visualization libraries and certain ML pipelines. stack/unstack do similar reshaping using the index hierarchy.

# Wide format: each metric is a column
wide = pd.DataFrame({
    "patient": ["P001", "P002"],
    "visit1_bp": [130, 145],
    "visit2_bp": [125, 140]
})

# Melt to long format: one row per measurement
long = wide.melt(
    id_vars=["patient"],
    value_vars=["visit1_bp", "visit2_bp"],
    var_name="visit",
    value_name="blood_pressure"
)
print(long)
#   patient       visit  blood_pressure
# 0    P001  visit1_bp             130
# 1    P002  visit1_bp             145
# 2    P001  visit2_bp             125
# 3    P002  visit2_bp             140

Merging and Joining

Real ML datasets don't live in a single table. You have a users table, an orders table, a products table. Or you have training features in one CSV and labels in another. You need to combine them. Pandas gives you SQL-style joins through merge.

merge (SQL-style Joins)

# Features table
features = pd.DataFrame({
    "house_id": [1, 2, 3, 4],
    "sqft": [1500, 2100, 1200, 2800],
    "bedrooms": [3, 4, 2, 5]
})

# Labels table (some houses don't have sale data yet)
labels = pd.DataFrame({
    "house_id": [1, 2, 4, 5],
    "sale_price": [350_000, 420_000, 510_000, 380_000]
})

# Inner join: only rows present in BOTH tables
inner = pd.merge(features, labels, on="house_id", how="inner")
# house_id 1, 2, 4 — dropped house 3 (no price) and 5 (no features)

# Left join: keep ALL rows from features, fill missing labels with NaN
left = pd.merge(features, labels, on="house_id", how="left")
# house 3 has NaN for sale_price

# Outer join: keep ALL rows from both, NaN where data is missing
outer = pd.merge(features, labels, on="house_id", how="outer")
# all 5 houses, with NaN where applicable

Inner, Outer, Left, Right Joins

Think of it this way. Inner: only the intersection — rows that have matching keys in both tables. Left: everything from the left table, matched rows from the right, NaN otherwise. Right: mirror of left. Outer: the union — everything from both, NaN where there's no match. In ML, left joins are the most common — you have a feature table and you want to enrich it with additional data without losing any rows.

ℹ️ Watch for Duplicated Rows

If your join key isn't unique in one of the tables, merge produces a Cartesian product for those keys — your row count can explode. Always check df.shape after a merge. Use validate="one_to_one" or validate="many_to_one" to catch unexpected duplicates early.

concat (Stacking DataFrames)

Use concat when you want to stack DataFrames vertically (append rows) or horizontally (append columns). Common use case: you have monthly CSVs and need to combine them into one dataset.

# Stack monthly data files
jan = pd.DataFrame({"date": ["2024-01-01"], "revenue": [1000]})
feb = pd.DataFrame({"date": ["2024-02-01"], "revenue": [1200]})
mar = pd.DataFrame({"date": ["2024-03-01"], "revenue": [1150]})

full_year = pd.concat([jan, feb, mar], ignore_index=True)

join Method

The join method is a convenience wrapper around merge that joins on the index by default. It's handy when your DataFrames already have meaningful indices.

# Join on index
features_idx = features.set_index("house_id")
labels_idx = labels.set_index("house_id")
combined = features_idx.join(labels_idx, how="left")

Data I/O

Pandas can read from and write to almost everything. Here are the formats you'll actually use in ML work.

Reading Data

# CSV — the lingua franca of datasets
df = pd.read_csv("train.csv")

# Useful read_csv parameters
df = pd.read_csv(
    "train.csv",
    usecols=["id", "feature1", "target"],  # only load needed columns
    dtype={"id": str, "feature1": float},   # explicit types save memory
    nrows=10_000,                           # sample for exploration
    parse_dates=["timestamp"],              # auto-parse date columns
    na_values=["N/A", "missing", "-1"],     # treat these as NaN
)

# Excel — when stakeholders send you "the data"
df = pd.read_excel("report.xlsx", sheet_name="Q4_Results")

# JSON — common for API data and nested structures
df = pd.read_json("events.json", orient="records")

# SQL — pull directly from a database
import sqlite3
conn = sqlite3.connect("app.db")
df = pd.read_sql("SELECT * FROM users WHERE active = 1", conn)

Writing Data

# Save processed features
df.to_csv("features_clean.csv", index=False)

# Save to Excel (requires openpyxl)
df.to_excel("report.xlsx", sheet_name="features", index=False)

# Save to JSON
df.to_json("output.json", orient="records", indent=2)

# Save to Parquet — fast, compressed, preserves types (best for large ML datasets)
df.to_parquet("features.parquet", engine="pyarrow")
ℹ️ Use Parquet for ML Pipelines

CSV is human-readable but slow and wasteful for large datasets. Parquet is columnar, compressed, and preserves dtypes — no more re-parsing dates every time you load. For datasets over 100MB, switching from CSV to Parquet can cut load times by 10x and storage by 5x.

Advanced Topics

query() Method

The query() method lets you filter DataFrames using a string expression, which can be more readable than bracket notation for complex conditions.

# Instead of: df[(df["sqft"] > 1500) & (df["bedrooms"] >= 3)]
filtered = df.query("sqft > 1500 and bedrooms >= 3")

# You can reference variables with @
min_price = 300_000
filtered = df.query("sale_price >= @min_price")

pipe() for Method Chaining

Method chaining lets you build readable data transformation pipelines. The pipe() method lets you plug your own functions into a chain — each function takes a DataFrame in and returns a DataFrame out.

def load_data(path):
    return pd.read_csv(path)

def clean_columns(df):
    df.columns = df.columns.str.lower().str.replace(" ", "_")
    return df

def remove_outliers(df, col, z_thresh=3):
    z_scores = (df[col] - df[col].mean()) / df[col].std()
    return df[z_scores.abs() < z_thresh]

def add_features(df):
    df["price_per_sqft"] = df["sale_price"] / df["sqft"]
    df["is_large"] = df["sqft"] > 2000
    return df

# The full pipeline — reads top to bottom like a recipe
result = (
    load_data("houses.csv")
    .pipe(clean_columns)
    .pipe(remove_outliers, col="sale_price")
    .pipe(add_features)
    .query("bedrooms >= 2")
    .sort_values("price_per_sqft", ascending=False)
)

This style is far easier to debug, test, and extend than a tangled mess of intermediate variables. Each function does one thing. You can test them in isolation. You can swap steps in and out. This is how production data pipelines should look.

Time Series (resample, rolling, shift)

When your index is a datetime, Pandas unlocks time-series-specific operations. resample is like groupby for time — it buckets data into time intervals. rolling computes sliding window statistics. shift offsets data in time, which is essential for creating lag features in time-series ML models.

# Daily stock data
dates = pd.date_range("2024-01-01", periods=90, freq="D")
stock = pd.DataFrame({
    "close": np.random.uniform(100, 200, 90)
}, index=dates)

# Resample to weekly average
weekly_avg = stock.resample("W").mean()

# 7-day rolling mean (smoothing)
stock["rolling_7d"] = stock["close"].rolling(window=7).mean()

# Lag features — yesterday's close as a feature for today's prediction
stock["close_lag1"] = stock["close"].shift(1)
stock["close_lag7"] = stock["close"].shift(7)

# Percent change — another common feature
stock["pct_change"] = stock["close"].pct_change()

Categorical Dtype for Memory Optimization

If a column has low cardinality — a small number of unique values relative to total rows, like "region" with 4 values across 1 million rows — converting it to categorical dtype saves huge amounts of memory. Instead of storing the string "West" a million times, Pandas stores integer codes that map to the string values.

# Before: region column uses ~8MB for 1M rows of strings
df["region"] = df["region"].astype("category")
# After: ~1MB — integer codes + a small lookup table

# Check the memory savings
print(df["region"].cat.codes)       # integer codes
print(df["region"].cat.categories)  # the unique values
print(df.memory_usage(deep=True))   # compare before/after

Copy-on-Write (CoW)

Copy-on-Write is a behavior change introduced in Pandas 2.0 (and the default in Pandas 3.0) that eliminates a long-standing class of bugs. Previously, when you sliced a DataFrame, sometimes you got a view (shares memory with the original) and sometimes a copy. Modifying a view could silently modify the original — or trigger the dreaded SettingWithCopyWarning. With CoW enabled, every indexing operation behaves as if it returns a copy. No accidental mutation. No warnings.

# Enable Copy-on-Write (default in Pandas 3.0+)
pd.set_option("mode.copy_on_write", True)

# Now this is always safe — no SettingWithCopyWarning
subset = df[df["bedrooms"] > 3]
subset["new_col"] = 1  # modifies subset only, never touches df

# The actual copy only happens when you write — reads still share memory
# This means CoW is both safer AND memory-efficient
ℹ️ CoW Is the Future

If you're starting a new project, enable Copy-on-Write immediately. It makes Pandas behavior consistent and eliminates an entire category of subtle bugs. The performance cost is negligible because copies are deferred until a write actually happens.

Polars as Alternative to Pandas

Polars is a newer DataFrame library written in Rust that's gaining serious traction. It's designed for speed: Polars uses lazy evaluation (builds a query plan before executing), multi-threaded execution by default, and avoids the GIL entirely. On large datasets, Polars can be 5-20x faster than Pandas. Its API is different — more expression-based, closer to SQL in spirit — but the concepts are the same.

import polars as pl

# Polars equivalent of the Pandas pipeline above
result = (
    pl.read_csv("houses.csv")
    .rename(lambda col: col.lower().replace(" ", "_"))
    .filter(pl.col("bedrooms") >= 2)
    .with_columns(
        (pl.col("sale_price") / pl.col("sqft")).alias("price_per_sqft"),
        (pl.col("sqft") > 2000).alias("is_large"),
    )
    .sort("price_per_sqft", descending=True)
)

Should you switch? Not yet if you're learning. Pandas has a massive ecosystem — scikit-learn, most visualization libraries, and most tutorials assume Pandas DataFrames. Learn Pandas first, because the concepts transfer directly. But for production workloads on datasets that make Pandas sweat (10M+ rows), Polars is worth evaluating. Many teams use both: Polars for heavy lifting, convert to Pandas for the final step into scikit-learn or plotting.
