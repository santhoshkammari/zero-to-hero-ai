Python Under the Hood

Chapter 1: Python & Programming Foundations 6 tables · 2 code blocks
TL;DR

Your training script is slow, your inference server leaks memory, and your data pipeline uses one core on an 8-core machine. These are runtime bugs, not code bugs. This section covers three things: how CPython manages memory (and why your DataFrame uses 3× expected RAM), how the GIL blocks CPU-bound threads (and the patterns that bypass it), and when async I/O is the right tool (and when it absolutely is not). Every row in every table maps to a real production scenario.

If you don't know how CPython allocates memory, enforces the GIL, and schedules I/O, you cannot diagnose why things are slow, leaking, or underutilizing hardware. This section gives you that mental model — memory first, then concurrency, then async.

The Decision Matrix — Which Tool for Which Bottleneck

This is the single most important table in this section. Before optimizing anything, identify your bottleneck type, then pick the right tool. Using the wrong one wastes time or makes things worse.

Bottleneck TypeExample ScenarioUseWhy This ToolGotcha
CPU-bound (compute-heavy) Feature engineering on 10M rows, image resizing, text tokenization multiprocessing / ProcessPoolExecutor Each process gets its own Python interpreter + GIL — true parallel execution on separate cores Data must be serialized (pickled) to cross process boundaries. Large objects = slow IPC. Keep payloads small or use shared memory.
I/O-bound, blocking (sync calls) Downloading 500 dataset files, querying a feature store, reading from S3 threading / ThreadPoolExecutor GIL is released during I/O waits, so threads genuinely overlap. No serialization overhead — threads share memory. Threads share state → race conditions. Use queue.Queue or concurrent.futures to avoid manual locking.
I/O-bound, async (non-blocking) ML API server handling 1000 concurrent requests, async DB queries, streaming responses asyncio / aiohttp / httpx.AsyncClient Single thread, no context-switching overhead. Event loop multiplexes thousands of connections efficiently. ONE blocking call (e.g., model.predict()) freezes the entire event loop. Must use run_in_executor() for sync code.
Parallel data loading Scikit-learn cross-validation, parallel grid search, batch preprocessing joblib.Parallel Built-in to sklearn. Handles chunking, progress bars, backend selection. Uses loky (fork+exec) by default — safer than raw multiprocessing. n_jobs=-1 uses all cores but each process copies memory. On a 32GB machine with a 10GB DataFrame, 8 workers = OOM.
GPU-bound Model training, batch inference on GPU PyTorch DataLoader(num_workers=N) Spawns worker processes that prefetch + preprocess batches while GPU computes. Keeps GPU fed. Workers use fork by default on Linux → CUDA context issues. Set multiprocessing_context='spawn' if using GPU in workers.
Mixed (I/O + CPU) Download images → resize → upload results Async I/O + ProcessPoolExecutor Async handles the network; process pool handles the compute. Best of both worlds. Architectural complexity. Only worth it for pipelines processing 10K+ items. For smaller jobs, sequential is fine.
Decision Rule of Thumb

Waiting on network/disk? → threads or async. Burning CPU cycles? → multiprocessing. Not sure? → profile first with cProfile. If time.sleep(1) inserted into your inner loop doesn't change wall-clock time, you're I/O-bound.

Python Memory Model — Why Your DataFrame Uses 3× Expected RAM

Every Python object lives on a private heap managed by CPython. You never call malloc() directly. Understanding these four mechanisms explains every "why is my process using so much memory?" question.

ConceptWhat It DoesWhy You Care (ML Context)
Reference counting Every object has a hidden counter. When count → 0, object is freed immediately. No delay. This is why del df frees memory instantly for non-circular objects. But if two objects reference each other (circular ref), refcount never reaches 0 — that's where the GC steps in.
Cyclic garbage collector Periodically scans for reference cycles (objects pointing at each other). Runs in 3 generations — Gen 0 (frequent), Gen 1, Gen 2 (rare). Custom dataset classes with parent↔child refs can leak if GC is disabled. Training loops that call gc.disable() for speed must call gc.collect() between epochs.
pymalloc (small object allocator) Objects ≤ 512 bytes use CPython's own allocator: 256 KB arenas → 4 KB pools → fixed-size blocks. Fast allocation, but arenas can only return to OS when every block is free. A list of 1M small dicts → millions of small allocations scattered across arenas. After del, RSS may not drop because one surviving object per arena pins the entire 256 KB. This is not a leak — it's allocator behavior.
Private heap All Python objects live here. OS sees one big allocation; Python subdivides internally. Large objects (> 512 bytes) go straight to the system allocator (malloc). NumPy arrays store their data buffer outside the Python heap (via malloc), so they return memory to OS cleanly. But the Python wrapper object (~100 bytes) stays on the private heap. A list of 100K tiny NumPy arrays = 100K wrapper objects on the Python heap.
Object overhead Every Python object has a header: 16 bytes (refcount + type pointer) on 64-bit. An empty dict is 64 bytes. An int is 28 bytes. A float is 24 bytes. A Pandas DataFrame of 1M floats stored as Python objects: 1M × 24 bytes = 24 MB for values + 16 bytes overhead each = 40 MB total. Same data as float64 NumPy: 8 MB. That's your 3–5× factor.
Production Pattern

Use df.info(memory_usage='deep') to see actual memory. Convert object columns to category dtype for columns with low cardinality (e.g., country codes) — often reduces memory 10×.

The GIL — Myths vs. Reality

The Global Interpreter Lock (GIL) is a mutex that allows only one thread to execute Python bytecode at a time. It exists because CPython's reference counting is not thread-safe. Here's what that actually means in practice:

MythRealityImplication for ML
"Python can't do parallel processing" The GIL only affects threads within one process. multiprocessing spawns separate processes, each with its own GIL → true parallelism. NumPy/PyTorch release the GIL during C-level computation. Your model.fit() in sklearn/XGBoost already uses multiprocessing internally via n_jobs. The GIL is not your bottleneck during training.
"Threading is useless in Python" The GIL is released during I/O operations (network, disk, sleep). Threads are excellent for I/O-bound work — downloading data, API calls, database queries all run truly concurrently. A data pipeline that downloads 500 files using 20 threads runs ~20× faster than sequential. No multiprocessing overhead needed.
"async replaces threading" Async and threading solve the same problem (I/O concurrency) differently. Async: cooperative scheduling, single thread, no race conditions. Threading: preemptive, multiple threads, shared state risks. Async does NOT help with CPU-bound work. For ML API servers (FastAPI), async handles thousands of concurrent HTTP requests. For batch file downloads in a script, threads are simpler and equally fast.
"The GIL will be removed soon" PEP 703 (free-threaded CPython) is experimental in Python 3.13. It's opt-in, breaks many C extensions (including NumPy as of early 2025), and production readiness is years away. Don't plan around free-threaded Python. Use multiprocessing for CPU-bound work today. Re-evaluate when NumPy/PyTorch/pandas officially support no-GIL builds.

Concurrency Tools — Quick Reference

Python has five concurrency tools you'll actually use. Each has a sweet spot.

ToolModuleParallelism TypeBest ForOne-Line Example
Thread threading Concurrent (shared memory, GIL-limited) Simple I/O overlap: downloading files, polling APIs Thread(target=download, args=(url,)).start()
ThreadPoolExecutor concurrent.futures Concurrent (thread pool) Batch I/O: download 100 files with 10 threads, collect results executor.map(download, urls)
ProcessPoolExecutor concurrent.futures Truly parallel (separate processes) CPU-bound: feature engineering, image preprocessing, tokenization executor.map(preprocess, image_paths)
joblib.Parallel joblib Truly parallel (loky backend) Sklearn-adjacent work: parallel CV, preprocessing pipelines Parallel(n_jobs=4)(delayed(fn)(x) for x in data)
asyncio.gather asyncio Concurrent (single thread, event loop) Async servers, concurrent API calls, streaming I/O await asyncio.gather(*[fetch(u) for u in urls])
concurrent.futures: The Unified API

ThreadPoolExecutor and ProcessPoolExecutor share the same interface. Start with threads. If profiling shows CPU is the bottleneck, swap to ProcessPoolExecutor — one line change. That's why concurrent.futures is the default choice for production code.

Code: Parallel Data Preprocessing with ProcessPoolExecutor

from concurrent.futures import ProcessPoolExecutor, as_completed
from pathlib import Path
import json

def preprocess_file(path: str) -> dict:
    """CPU-bound: parse JSON, extract features, return summary."""
    with open(path) as f:
        raw = json.load(f)
    # Simulate heavy feature extraction
    features = {
        "n_tokens": len(raw["text"].split()),
        "has_label": raw.get("label") is not None,
        "path": path,
    }
    return features

def parallel_preprocess(file_paths: list[str], max_workers: int = 8) -> list[dict]:
    """Process files across all cores. Returns results in completion order."""
    results = []
    with ProcessPoolExecutor(max_workers=max_workers) as executor:
        # submit() gives you Future objects — better error handling than map()
        future_to_path = {
            executor.submit(preprocess_file, p): p for p in file_paths
        }
        for future in as_completed(future_to_path):
            path = future_to_path[future]
            try:
                results.append(future.result())
            except Exception as e:
                print(f"Failed {path}: {e}")  # Don't let one file kill the pipeline
    return results

# Usage
paths = [str(p) for p in Path("data/raw").glob("*.json")]
features = parallel_preprocess(paths, max_workers=8)
# 8 cores → ~8x speedup for CPU-bound work

Why as_completed over map? — map() returns results in input order, so one slow file blocks everything. as_completed() yields results as they finish + lets you handle per-file exceptions without killing the whole batch.

Memory Profiling Tools

When something uses too much memory, you need to measure before guessing. Use these in order: start with the cheap ones, escalate only if needed.

ToolWhat It ShowsWhen to UseKey Command
sys.getsizeof() Shallow size of ONE object in bytes (does not follow references) Quick sanity check: "how big is this dict/list itself?" sys.getsizeof(my_dict) — remember this misses nested objects
tracemalloc Per-line memory allocation tracking. Take snapshots, diff them to find where memory grew. Finding memory leaks in training loops. "Memory grew 2GB between epoch 1 and 5 — which line?" tracemalloc.start() → run code → snapshot.compare_to(old, 'lineno')
memory_profiler Line-by-line memory usage of a function (like cProfile but for memory) Profiling a specific function: "which line in my load_data() allocates the most?" Decorate with @profile, run with python -m memory_profiler script.py
objgraph Object reference graphs. Shows what's keeping objects alive. Debugging why something isn't being garbage collected. "What's referencing my old model?" objgraph.show_backrefs(obj, max_depth=3)
pympler Deep size of object graphs (follows references, unlike sys.getsizeof) "What's the TOTAL memory of this nested dict of lists of dicts?" from pympler.asizeof import asizeof; asizeof(obj)
The tracemalloc Pattern for Training Loops

tracemalloc.start() before training. Take a snapshot after epoch 1. Take another after epoch 5. Diff them: snapshot2.compare_to(snapshot1, 'lineno')[:10]. The top 10 lines are where your memory grew. This catches 90% of training-loop memory leaks in under a minute.

Async Patterns for ML — When Yes, When No

Async is powerful for I/O multiplexing but actively harmful if misapplied. This table draws the line.

PatternWhen to UseWhen NOT to UseWhy
Async API serving (FastAPI, Starlette) ML inference endpoints receiving 100+ concurrent requests where each request waits on I/O (feature store, database, downstream services) If each request is pure CPU (model prediction takes 500ms of compute) — async adds overhead without benefit Async shines when the server spends most time waiting. 1 thread can handle 1000 waiting connections. But 1 thread can only compute 1 prediction at a time.
Async data fetching Downloading training data from multiple URLs, fetching embeddings from a vector DB, calling multiple APIs in parallel Reading local files (disk I/O is fast enough that async overhead > benefit for typical dataset sizes) Network latency (10-200ms per call) dwarfs async overhead (~0.01ms per task switch). 100 network calls go from 10s sequential to ~0.2s concurrent.
Async batch inference Sending batches to a remote inference API (e.g., calling OpenAI, Hugging Face Inference Endpoints) Local batch inference (just use model.predict(batch) — no I/O involved) Remote inference is I/O-bound (network round-trip). Local inference is CPU/GPU-bound. Async helps the former, not the latter.
Async + CPU hybrid (via run_in_executor) FastAPI endpoint that needs to call a sync ML model: await loop.run_in_executor(pool, model.predict, input) Don't wrap everything in executors — if most of your code is sync/CPU, just use a sync framework (Flask + Gunicorn workers) run_in_executor offloads blocking calls to a thread/process pool so the event loop stays responsive. Essential for mixing async serving with sync ML code.

Code: Async Inference API with Sync Model

import asyncio
from concurrent.futures import ProcessPoolExecutor
from fastapi import FastAPI
import numpy as np

app = FastAPI()
# Process pool: keeps CPU-bound prediction off the event loop
executor = ProcessPoolExecutor(max_workers=4)

def predict_sync(features: list[float]) -> float:
    """Sync, CPU-bound prediction — cannot be awaited directly."""
    X = np.array(features).reshape(1, -1)
    # Simulate heavy model inference
    return float(X.sum() * 0.42)  # replace with model.predict(X)[0]

@app.post("/predict")
async def predict(features: list[float]):
    loop = asyncio.get_event_loop()
    # Offload sync prediction to process pool → event loop stays free
    result = await loop.run_in_executor(executor, predict_sync, features)
    return {"prediction": result}

# This pattern lets FastAPI handle 1000+ concurrent requests
# while each prediction runs in a separate process.
# Without run_in_executor, one slow prediction blocks ALL requests.

Key insight: The async def endpoint receives requests concurrently. The run_in_executor call sends CPU work to a process pool. The event loop is never blocked. This is the standard pattern for ML APIs in production.
