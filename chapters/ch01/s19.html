Memory Management and CPython Internals

Chapter 1: Python & Programming Foundations 10 subtopics
TL;DR

CPython manages memory through a private heap, using reference counting as its primary cleanup mechanism and a generational garbage collector to handle circular references. Understanding these internals matters because ML workloads routinely push memory limits — a leaked tensor or a forgotten reference to a training batch can silently consume gigabytes. This section walks through how Python allocates, tracks, and frees memory, and gives you concrete tools to find and fix leaks before they crash your production inference server at 3 AM.

Why Memory Management Matters for ML Engineers

Picture this: your model training script runs fine for 20 epochs, then dies on epoch 21 with a cryptic MemoryError. Or your Flask-based inference API gradually eats more and more RAM until the container gets OOM-killed every few hours. These aren't exotic problems. They're Monday.

ML workloads are memory-hungry by nature. You load entire datasets into RAM. You create massive NumPy arrays and PyTorch tensors. You run long-lived services that process thousands of requests. In this world, understanding how CPython manages memory isn't academic trivia — it's survival knowledge. If you don't know how Python decides when to free memory, you can't reason about why it's not freeing memory.

Let's start at the foundation and build up.

Python Memory Model: The Private Heap

Every Python object lives on a private heap — a region of memory that the Python runtime manages internally. You never allocate or free memory on this heap directly. The Python memory manager handles all of it behind the scenes.

This is different from languages like C, where you call malloc() and free() yourself. Python's philosophy is: the runtime manages memory so you can focus on logic. But "manages" doesn't mean "manages perfectly." The runtime follows specific rules, and those rules have edge cases that bite.

The heap is organized in layers. At the bottom, CPython asks the operating system for large chunks of memory. On top of that sits Python's own allocator. On top of that sit the object-specific allocators. Think of it as a three-tier cake:

# Conceptual layers (not real code — a mental model)
#
# Layer 3: Object-specific allocators (int, list, dict, etc.)
# Layer 2: Python's memory allocator (pymalloc)
# Layer 1: OS-level allocator (malloc/free from libc)
# Layer 0: Virtual memory from the OS kernel
#
# You interact with Layer 3. Layers below are invisible to you.

When you write x = [1, 2, 3], Python allocates memory for a list object and three integer objects on this private heap. You never see the heap address. You never call free. But understanding this layered structure helps you reason about why memory behaves the way it does — especially why Python sometimes doesn't return memory to the OS even after you delete objects.

Reference Counting: Python's Primary Cleanup Mechanism

Every Python object carries a hidden counter called its reference count. This counter tracks how many things in your program currently point to that object. When the count drops to zero, CPython immediately frees the object's memory. No delay. No background thread. Immediate.

Let's see this in action:

import sys

a = [1, 2, 3]
print(sys.getrefcount(a))  # 2 — one for 'a', one for the argument to getrefcount()

b = a
print(sys.getrefcount(a))  # 3 — 'a', 'b', and the getrefcount argument

del b
print(sys.getrefcount(a))  # 2 — back to 'a' and the argument

c = [a, a, a]
print(sys.getrefcount(a))  # 5 — 'a', three slots in 'c', and the argument
Why getrefcount() Shows +1

You'll always see one more reference than you expect. That's because passing the object to sys.getrefcount() temporarily creates an extra reference. It's not a bug — it's the function holding a reference to the thing it's counting.

Reference counting is elegant. It's deterministic — you know exactly when an object will be freed. It's fast for the common case. But it has one fatal flaw: it can't handle cycles.

Circular References and the gc Module

A circular reference happens when two or more objects point at each other. Their reference counts never reach zero, even after you stop using them. This is a genuine memory leak if left unchecked.

import gc

class Node:
    def __init__(self, name):
        self.name = name
        self.neighbor = None

    def __repr__(self):
        return f"Node({self.name})"

# Create a cycle
a = Node("A")
b = Node("B")
a.neighbor = b
b.neighbor = a  # a -> b -> a -> b -> ...

# Delete our references
del a
del b

# The nodes still exist! Each holds a reference to the other.
# Reference counting alone can't free them.

# But the garbage collector can:
collected = gc.collect()
print(f"Collected {collected} objects")  # Collected 2 objects (or more)

This is where generational garbage collection enters the picture. CPython's garbage collector exists for one purpose: finding and cleaning up circular references that reference counting misses.

How Generational GC Works

The garbage collector divides objects into three generations: generation 0 (youngest), generation 1, and generation 2 (oldest). The core insight is: most objects die young. A temporary variable inside a function, a string you built for a log message, an intermediate calculation — these live for microseconds. So the collector checks young objects frequently and old objects rarely.

import gc

# See the current thresholds
print(gc.get_threshold())  # (700, 10, 10) by default

# This means:
# - Generation 0 is collected after 700 new allocations (minus deallocations)
# - Generation 1 is collected every 10 generation-0 collections
# - Generation 2 is collected every 10 generation-1 collections

# You can inspect what's in each generation
print(gc.get_count())  # (number_in_gen0, number_in_gen1, number_in_gen2)

# Tune thresholds if needed (rare, but useful for ML pipelines)
gc.set_threshold(1000, 15, 15)  # Less aggressive collection

When the collector runs on a generation, it uses a cycle detection algorithm. It temporarily decrements reference counts for all internal references between objects in that generation. If an object's adjusted count hits zero, it's unreachable — part of a cycle with no external references. The collector frees it.

Objects that survive a collection get promoted to the next generation. This is why long-lived objects (like your model weights loaded at startup) eventually end up in generation 2 and are checked very rarely.

GC Pauses in ML Training Loops

Garbage collection pauses can cause unexpected latency spikes. If you're doing real-time inference and notice occasional slow responses, GC might be the culprit. Some teams disable GC during latency-sensitive sections with gc.disable() and run gc.collect() manually between batches. Do this carefully — disabling GC means circular references accumulate until you collect manually.

pymalloc: The Small Object Allocator

Most Python objects are tiny. An integer is a few dozen bytes. A short string, maybe 50 bytes. Calling the OS-level malloc() for each of these would be absurdly slow — system calls have overhead.

So CPython has its own allocator for small objects (512 bytes or less) called pymalloc. It works in layers:

Arenas are 256 KB chunks requested from the OS. Each arena is divided into pools of 4 KB (one virtual memory page). Each pool is dedicated to a specific size class — objects of 8 bytes, 16 bytes, 24 bytes, up to 512 bytes. Within each pool, memory is carved into fixed-size blocks.

# Conceptual diagram of pymalloc's structure:
#
# Arena (256 KB)
# ├── Pool (4 KB) — size class: 32 bytes
# │   ├── Block (32 bytes) — holds one small object
# │   ├── Block (32 bytes) — holds one small object
# │   └── ... (about 126 blocks per pool)
# ├── Pool (4 KB) — size class: 64 bytes
# │   ├── Block (64 bytes)
# │   └── ...
# └── ... (64 pools per arena)
#
# Objects > 512 bytes bypass pymalloc and go straight to malloc().

This design is fast because allocating a small object means finding a free block in an existing pool — no system call needed. It's also why Python can sometimes appear to "hold onto" memory: an arena can only be returned to the OS when every block in every pool in that arena is free. One lingering object in an arena pins the entire 256 KB.

String Interning and Small Integer Caching

CPython plays some clever tricks to save memory and speed up comparisons for common values.

Small Integer Caching

Integers from -5 to 256 are pre-allocated when the interpreter starts. Every time your code uses the number 42, it gets the same object in memory. This avoids creating thousands of duplicate integer objects for common values.

# Same object — cached by CPython
a = 256
b = 256
print(a is b)       # True
print(id(a) == id(b))  # True — exact same memory address

# Different objects — outside the cache range
a = 257
b = 257
print(a is b)       # False (in most contexts)
print(id(a) == id(b))  # False — different memory addresses

# The boundary is sharp
a = -5
b = -5
print(a is b)  # True — -5 is the lower boundary

a = -6
b = -6
print(a is b)  # False — -6 is outside the range
Never Use is to Compare Values

The caching behavior is an implementation detail of CPython, not part of the Python language spec. Code that relies on is for integer or string comparison will break across Python implementations (PyPy, Jython) and sometimes even between CPython versions. Use == for value comparison. Always.

String Interning

String interning means that certain strings are stored only once in memory, and every variable holding that string value points to the same object. CPython automatically interns strings that look like identifiers (letters, digits, underscores) because these are commonly used as dictionary keys and attribute names.

import sys

# Automatically interned — looks like an identifier
a = "hello"
b = "hello"
print(a is b)  # True — same object in memory

# Not automatically interned — contains a space
a = "hello world"
b = "hello world"
print(a is b)  # False (usually) — different objects

# You can force interning
a = sys.intern("hello world")
b = sys.intern("hello world")
print(a is b)  # True — now they share the same object

# Why bother? Dictionary lookups become pointer comparisons
# instead of character-by-character string comparisons.
# This matters when you have millions of lookups.

In ML code, you might see interning help with feature name lookups in large dictionaries, or column name comparisons when processing DataFrames with millions of rows.

The weakref Module: References That Don't Count

Sometimes you want to reference an object without preventing it from being garbage collected. This is exactly what the weakref module provides. A weak reference points to an object but doesn't increment its reference count.

The classic use case: caches. You want to cache expensive-to-compute objects, but you don't want the cache itself to keep objects alive when nothing else needs them.

import weakref

class LargeModel:
    def __init__(self, name, size_mb):
        self.name = name
        self.data = bytearray(size_mb * 1024 * 1024)  # Simulate large data

    def __repr__(self):
        return f"LargeModel({self.name})"

# Without weakref — the cache keeps the model alive forever
bad_cache = {}

model = LargeModel("resnet50", 100)
bad_cache["resnet50"] = model
del model
# Model is still alive — bad_cache holds a strong reference
print(bad_cache["resnet50"])  # LargeModel(resnet50) — still here, 100 MB pinned

# With weakref — the cache lets the model be collected
good_cache = weakref.WeakValueDictionary()

model = LargeModel("resnet50", 100)
good_cache["resnet50"] = model
print(good_cache["resnet50"])  # LargeModel(resnet50) — still here

del model
# Now the model can be garbage collected — good_cache doesn't prevent it
try:
    print(good_cache["resnet50"])
except KeyError:
    print("Model was garbage collected!")  # This prints

WeakValueDictionary automatically removes entries when the referenced objects are collected. There's also WeakSet and the lower-level weakref.ref() for building your own patterns. In long-running ML services, weak references are your tool for caches that adapt to memory pressure instead of fighting it.

The __del__ Method and Its Pitfalls

Python lets you define a __del__ method on your class — a finalizer that runs when the object is about to be destroyed. It sounds useful. It's mostly a trap.

class ResourceHolder:
    def __init__(self, name):
        self.name = name
        print(f"{self.name}: acquired resource")

    def __del__(self):
        print(f"{self.name}: releasing resource")

# Normal case works fine
obj = ResourceHolder("A")
del obj  # Prints: A: releasing resource

That looks clean. Here's where it falls apart:

Pitfall 1: Exception Swallowing

class BadFinalizer:
    def __del__(self):
        raise ValueError("Something went wrong during cleanup!")

obj = BadFinalizer()
del obj
# Python prints a warning to stderr but SWALLOWS the exception:
# Exception ignored in: 
# Traceback (most recent call last):
#   ...
# ValueError: Something went wrong during cleanup!
#
# Your program keeps running. The error is silently ignored.
# If your __del__ was supposed to close a database connection
# or flush a file, you'll never know it failed.

Pitfall 2: Object Resurrection

zombie_list = []

class Zombie:
    def __del__(self):
        # Resurrecting self by adding to a global list
        zombie_list.append(self)
        print("I'm not dead yet!")

z = Zombie()
del z       # Prints: I'm not dead yet!
print(len(zombie_list))  # 1 — the object escaped destruction

# The object is alive again, but __del__ won't be called
# a second time (in CPython). You've now got an object that
# can never be finalized properly.

Pitfall 3: Circular References with __del__

In older Python versions (before 3.4), objects with __del__ that were part of a reference cycle couldn't be collected at all — they'd end up in gc.garbage forever. Python 3.4+ (PEP 442) improved this, but the interaction between finalizers and cycles is still tricky. The safe alternative is weakref.finalize():

import weakref

class SafeResource:
    def __init__(self, name):
        self.name = name
        # Register cleanup with weakref.finalize — much safer
        weakref.finalize(self, SafeResource._cleanup, name)

    @staticmethod
    def _cleanup(name):
        print(f"{name}: safely cleaned up")

obj = SafeResource("DB-conn")
del obj  # Prints: DB-conn: safely cleaned up
Prefer Context Managers Over __del__

If you need deterministic cleanup (closing files, releasing GPU memory, disconnecting from databases), use a context manager (with statement) instead of __del__. Context managers give you guaranteed, timely cleanup regardless of how the object's reference count behaves.

tracemalloc: Memory Profiling from the Inside

When you suspect a memory leak, guessing doesn't work. You need data. The tracemalloc module, built into Python since 3.4, traces every memory allocation back to the line of code that caused it.

import tracemalloc

tracemalloc.start()

# Simulate a memory-heavy operation
data = []
for i in range(100_000):
    data.append({
        "id": i,
        "features": [float(x) for x in range(50)],
        "label": i % 10
    })

# Take a snapshot
snapshot = tracemalloc.take_snapshot()

# Show top 5 memory consumers by line
top_stats = snapshot.statistics("lineno")
print("[ Top 5 memory consumers ]")
for stat in top_stats[:5]:
    print(stat)

That's the basic pattern. But the real power is in comparing snapshots — taking one before an operation, another after, and looking at the difference:

import tracemalloc

tracemalloc.start()

# --- Snapshot BEFORE ---
snapshot1 = tracemalloc.take_snapshot()

# Do some work that might leak
cache = {}
for i in range(50_000):
    key = f"item_{i}"
    cache[key] = [0.0] * 100  # 100 floats per entry

# --- Snapshot AFTER ---
snapshot2 = tracemalloc.take_snapshot()

# Compare the two snapshots
top_stats = snapshot2.compare_to(snapshot1, "lineno")
print("[ Top memory growth ]")
for stat in top_stats[:5]:
    print(stat)

# Output shows exactly which lines allocated how much NEW memory
# between the two snapshots. This pinpoints leaks with surgical precision.

For ML pipelines, wrap your training loop or inference batch in snapshot comparisons. If memory grows between iterations and the growth points to the same lines, you've found your leak.

Debugging a Memory Leak: A Practical Walkthrough

Let's put it all together. Here's a scenario that mirrors real production bugs — a prediction service that slowly leaks memory.

import gc
import tracemalloc
import sys

tracemalloc.start()

class PredictionService:
    def __init__(self):
        self._history = []  # Bug: unbounded history

    def predict(self, input_data):
        result = {"input": input_data, "score": 0.95}
        self._history.append(result)  # Leak: history grows forever
        return result

service = PredictionService()

# Simulate 10,000 requests
snapshot1 = tracemalloc.take_snapshot()

for i in range(10_000):
    service.predict({"features": list(range(20))})

snapshot2 = tracemalloc.take_snapshot()

# Step 1: Check overall memory growth
current, peak = tracemalloc.get_traced_memory()
print(f"Current memory: {current / 1024:.1f} KB")
print(f"Peak memory:    {peak / 1024:.1f} KB")

# Step 2: Find where the growth happened
top_stats = snapshot2.compare_to(snapshot1, "lineno")
print("\n[ Memory growth by line ]")
for stat in top_stats[:3]:
    print(stat)

# Step 3: Check if objects are accumulating
print(f"\nHistory length: {len(service._history)}")  # 10,000!
print(f"History memory: {sys.getsizeof(service._history)} bytes (list overhead only)")

# Step 4: The fix — use a bounded collection
from collections import deque
service._history = deque(maxlen=100)  # Keep only last 100 predictions

The investigation pattern is always the same. First, notice memory growth (monitoring, alerts, or OOM kills). Second, use tracemalloc snapshots to find which lines are responsible. Third, examine the objects at those lines. Fourth, fix the root cause — usually an unbounded collection, a forgotten callback registration, or a circular reference.

Quick Diagnostic Toolkit

For rapid memory debugging, keep these in your toolbox: sys.getsizeof(obj) for shallow size of a single object, tracemalloc for tracing allocations to source lines, gc.get_referrers(obj) to find what's holding a reference to an object, and objgraph (third-party) for visualizing reference graphs. Start with tracemalloc — it solves 80% of leak investigations.

Memory Leak Detection: gc Module Deep Dive

Beyond its collection duties, the gc module gives you tools to inspect what the garbage collector is doing. This is invaluable when debugging cyclic reference leaks.

import gc

# Enable GC debug output
gc.set_debug(gc.DEBUG_LEAK)  # Logs uncollectable objects

class Cycle:
    def __init__(self, name):
        self.name = name
        self.ref = None

# Create a cycle
a = Cycle("A")
b = Cycle("B")
a.ref = b
b.ref = a

# Drop external references
del a, b

# Force collection and check results
unreachable = gc.collect()
print(f"Unreachable objects found: {unreachable}")

# Inspect what the GC knows about
print(f"Objects tracked by GC: {len(gc.get_objects())}")

# Find specific object types
cycle_objects = [obj for obj in gc.get_objects()
                 if isinstance(obj, Cycle)]
print(f"Cycle objects still alive: {len(cycle_objects)}")

# For each, find what's referencing it
for obj in cycle_objects:
    referrers = gc.get_referrers(obj)
    print(f"  {obj.name} has {len(referrers)} referrers")

gc.get_referrers() is the "who's pointing at this?" function. When you find an object that should have been freed, call this to discover what's keeping it alive. Follow the chain of referrers until you find the root — that's your leak.

Putting It All Together

Memory management in CPython isn't one mechanism — it's a layered system where each layer handles a different concern:

Reference counting handles the common case: objects that are created, used, and discarded without cycles. It's immediate and deterministic. Generational garbage collection handles the uncommon but dangerous case: circular references that reference counting can't detect. pymalloc makes small allocations fast by avoiding system calls. String interning and integer caching reduce memory use and speed up comparisons for frequently-used values. weakref gives you a way to observe objects without keeping them alive. And tracemalloc gives you the X-ray vision to see where every byte went when things go wrong.

For ML work, the practical takeaways are:

Watch out for unbounded collections in long-running services — prediction histories, feature caches, logging buffers. Use deque(maxlen=N) or WeakValueDictionary instead of plain dicts and lists when the data is transient. Profile with tracemalloc before and after your training loops to catch leaks early. Be cautious with gc.disable() — it speeds things up but only if you manually collect at safe points. Avoid __del__ for cleanup; use context managers or weakref.finalize(). And never use is to compare values, no matter how tempting the integer caching trick makes it look.

Memory bugs are the quietest kind of bug. Your code runs perfectly — until it doesn't. The tools in this section turn invisible memory behavior into visible, debuggable data. Use them proactively, not just when something's already on fire.
