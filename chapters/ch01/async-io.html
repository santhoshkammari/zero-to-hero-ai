Async IO

Chapter 1: Python & Programming Foundations 12 code blocks

Asyncio is Python's built-in machinery for concurrent code without wasted waiting. Formalized in Python 3.4, refined across releases, now backbone of modern Python web frameworks, scrapers, bots, and anything that talks to outside world.

The Waiting Problem

Programs spend shocking amount of time doing nothing.

When code asks operating system to read file, fetch web page, or query database, CPU sends request then waits. It waits for hard drive, network packets, or database server. These waits are called I/O operations. They are slow. Not slow like "this loop has too many iterations." Slow like "fiber optic trip across Atlantic."

Rough scale: if one CPU cycle took one second, reading RAM would take about 4 minutes. Reading SSD would take 1–3 days. Network round trip to server across world? About 4 years.

CPU is extraordinarily fast. Outside world is extraordinarily slow. Asyncio exists to exploit that gap.

Blocking Code

Simple problem example. Imagine small tool checking whether three websites are up.

import time

def check_site(url):
    print(f"Checking {url}...")
    time.sleep(2)  # simulating a network request
    print(f"{url} is up!")

start = time.time()
check_site("site-a.com")
check_site("site-b.com")
check_site("site-c.com")
print(f"Done in {time.time() - start:.1f} seconds")

Checking site-a.com...
site-a.com is up!
Checking site-b.com...
site-b.com is up!
Checking site-c.com...
site-c.com is up!
Done in 6.0 seconds

Six seconds. Each check_site call blocks. It takes control and gives nothing back until wait ends. Second site cannot start until first ends, even though CPU does nothing during those two-second waits.

"Blocking" does heavy lifting. It does not mean hard computation. It means function sits idle and lets nothing else happen until wait ends.

Three sites, two seconds each, six seconds total. Hundred sites means two hundred seconds. Waste: waits could overlap. Start all three at once, finish in two seconds. Goal.

Doing Things While You Wait

Core asyncio insight is not technical. You already do it daily.

Put bread in toaster, you do not stare. You pour coffee. While coffee brews, you check phone. One person. Several tasks. Switch when one waits.

Not doing three things simultaneously. Doing one thing at time, but choosing next work while something else is blocked.

That is concurrency. One worker, many tasks, smart switching.

Concurrency Is Not Parallelism

These two get tangled constantly. Be explicit.

Parallelism means multiple workers doing work at same time. Two cooks, different dishes. In Python, multiprocessing gives this — separate processes, each with its own CPU core.

Concurrency means one worker juggling multiple tasks during waits. One cook: pasta boils, vegetables get chopped. In Python, asyncio gives this — one thread switching tasks when one waits on I/O.

Threading sits between them. It gives multiple workers (threads), but in CPython GIL — Global Interpreter Lock — makes them take turns on CPU. Rule: only one thread executes Python bytecode at time, even with 16 idle CPU cores. For I/O-bound work, threading still helps because threads mostly wait, not compute. But threading does not give true parallelism for Python code.

Practical rule: I/O-bound work means asyncio. CPU-bound work means multiprocessing. Use asyncio for CPU-bound work, expect disappointment. Not magic. Smart wait-time scheduling.

Website checker spends time waiting on network. Perfect asyncio candidate. But path from "I want concurrency" to async def has few stops. Each stop fixes failure from step before.

Callbacks: The First Attempt

Before current asyncio, standard concurrent I/O style was callbacks. Idea: instead of waiting for result, hand off function to call when result is ready, then move on.

def on_site_checked(url):
    print(f"{url} is up!")

def check_site(url, callback):
    print(f"Checking {url}...")
    # imagine this registers the check and returns immediately
    # when the response arrives later, callback gets called
    register_network_request(url, callback=lambda: callback(url))

This works. You can start all three checks without waiting, and each callback runs when its check finishes. Problem appears when flow gets more complex.

def step1(callback):
    fetch_user(user_id, lambda user:
        fetch_orders(user.id, lambda orders:
            fetch_details(orders[0].id, lambda details:
                callback(details))))

This is callback hell. Each step depends on previous step, so nesting gets deeper. Code reads inside-out and bottom-up. Error handling becomes nightmare. Debugging hurts because call stack no longer matches program logic.

Callbacks are powerful but brutal ergonomically. They turn sequential logic into tangled web of nested functions. People tolerated this for years — early Node.js ecosystem ran on it — but better way was needed.

The Event Loop

Center of asyncio is event loop. Less mysterious than it sounds.

First time I read event loop was while True, I did not believe it. Felt too simple. More plumbing exists, but core idea really is that plain. Loop runs forever, or until you stop it. Each iteration asks: "Any tasks ready to make progress?" If yes, run task until task says "I'm waiting on something." Then move to next ready task.

Cartoon version:

while there are tasks:
    for each task:
        if task is ready:
            run task until it says "I'm waiting"
            note what it's waiting for
    check if any waited-on things are now ready

Whole idea. Event loop is cook in kitchen. It does not perform actual I/O. It orchestrates. It decides who runs when.

Switching happens at defined points — only when task explicitly says "I'm waiting." This is cooperative multitasking. Tasks yield control voluntarily when they have nothing to do. Compare threads: operating system can yank control at any moment, even between two lines of code. That is preemptive multitasking. That is source of race conditions that make multithreaded programming feel like bomb defusal.

With cooperative multitasking, cook checks toaster only after finishing current vegetable. No mid-chop knife drop. You know where switch can happen.

Rest Stop and an Off Ramp

Made it this far. You can stop now if you want.

Mental model: one cook juggling tasks during waits, coordinated by event loop that is glorified while-True. Useful model. Programs waste time waiting. Instead of wasting it, switch to other work. Loop manages switching. Tasks yield voluntarily.

Practical cheat sheet: write async def instead of def, put await before anything that waits, use asyncio.run() to start. That gets you 80% there.

But if not knowing what sits underneath still nags — if you want how Python implements this — keep reading. Next detour: generators. They were original mechanical foundation.

Generators: A Detour Worth Taking

Normal function runs top to bottom and returns once.

def count_up():
    return [1, 2, 3]

Generator function can pause midway and resume later.

def count_up():
    yield 1
    yield 2
    yield 3

yield is magic word. When generator hits yield, it hands value back to caller and freezes in place. Local variables, code position, everything stays preserved. Ask for next value, it wakes exactly where it stopped and continues until next yield.

gen = count_up()
print(next(gen))  # 1
print(next(gen))  # 2
print(next(gen))  # 3

Between next() calls, you can do anything. Generator stays suspended, holding state, waiting to resume. Function freezing in place and resuming later with locals intact still feels slightly magical.

That pause-and-resume ability is exactly what concurrent I/O needs. See connection. Two generators pretend to work with waits between steps, and while loop interleaves them manually:

def task_a():
    print("A: start")
    yield  # "I'm waiting"
    print("A: middle")
    yield  # "I'm waiting again"
    print("A: done")

def task_b():
    print("B: start")
    yield
    print("B: done")

tasks = [task_a(), task_b()]
while tasks:
    task = tasks.pop(0)
    try:
        next(task)
        tasks.append(task)
    except StopIteration:
        pass

A: start
B: start
A: middle
B: done
A: done

Two functions, interleaved, driven by while loop. yield is each function saying "Nothing to do now. Help someone else." while loop is event loop. This is asyncio in 12 lines. Rest is refinement.

Key insight behind asyncio: instead of yielding nothing, function could yield description of what it waits for. "Waiting on network response. Take this token. Wake me when ready."

Coroutines

Coroutine is generator that yields control instead of values. That is conceptual leap.

Early asyncio (Python 3.4) used coroutines that literally were generators with decorator slapped on:

import asyncio

@asyncio.coroutine
def check_site(url):
    print(f"Checking {url}...")
    yield from asyncio.sleep(2)
    print(f"{url} is up!")

yield from asyncio.sleep(2) means "Need to wait 2 seconds. Yielding control back to event loop. Wake me when time ends." Event loop records wake-up time, runs other tasks, comes back after 2 seconds.

This worked, but yield from was confusing. It looked like generators but meant something else. I spent embarrassing time staring at yield from asyncio.sleep(2) trying to find what value sleep yielded. Answer: irrelevant. It yields control, not data. Confusion is real. Not your fault.

Python 3.5 fixed this with dedicated syntax.

async and await

async def and await are syntactic sugar over coroutines. No new capability. More clarity.

import asyncio

async def check_site(url):
    print(f"Checking {url}...")
    await asyncio.sleep(2)
    print(f"{url} is up!")

async def declares coroutine function. Calling it does not run it — it creates coroutine object, like generator function call creates generator object.

await is new yield from. It means "run this other coroutine and suspend me until it finishes." When event loop sees await, it knows task handed back control. Time to do something else.

Critical point: await is only place event loop can switch tasks. Between await points, code runs uninterrupted. Remember cook: check toaster after finishing current vegetable. No mid-chop knife drop. That is why asyncio avoids threading-style race condition problems. You know exactly where task switches can happen.

Running the Loop

Coroutine alone does nothing. It needs event loop to drive it.

import asyncio

async def main():
    print("hello")
    await asyncio.sleep(1)
    print("world")

asyncio.run(main())

asyncio.run() creates event loop, runs coroutine to completion, then shuts loop down. Standard entry point since Python 3.7.

But this still runs one coroutine. Where is concurrency? All this machinery, website checker still takes 6 seconds. Not satisfying. We need way to run multiple coroutines at same time.

Awaitables

Before concurrency, be precise about what you can await. Three kinds. Coroutines: objects from calling async def. Tasks: coroutines handed to event loop and scheduled to run. Futures: low-level objects for results that do not exist yet; plumbing you rarely touch directly.

Difference between coroutines and tasks is where concurrency starts.

Tasks: Fire and (Almost) Forget

If you await coroutine directly, you wait for it to finish before moving on. Sequential, not concurrent.

async def main():
    await check_site("site-a.com")   # waits 2 seconds
    await check_site("site-b.com")   # waits 2 more seconds
    await check_site("site-c.com")   # waits 2 more seconds
    # total: 6 seconds. No better than blocking.

To run coroutines concurrently, wrap them in Tasks. Task tells event loop "schedule this coroutine, but do not wait for it right now."

async def main():
    task_a = asyncio.create_task(check_site("site-a.com"))
    task_b = asyncio.create_task(check_site("site-b.com"))
    task_c = asyncio.create_task(check_site("site-c.com"))

    await task_a
    await task_b
    await task_c

Now all three tasks get scheduled immediately. When task_a hits its await asyncio.sleep(2), it yields control. Event loop picks up task_b, which starts and yields too. Same for task_c. All three wait simultaneously. After 2 seconds, they finish in rapid succession.

Step-by-step event loop trace made this click for me:

Iteration 1: task_a runs → prints "Checking site-a.com..." → hits await sleep(2) → yields
Iteration 2: task_b runs → prints "Checking site-b.com..." → hits await sleep(2) → yields
Iteration 3: task_c runs → prints "Checking site-c.com..." → hits await sleep(2) → yields
[all three are now sleeping, ~2 seconds pass]
Iteration 4: task_a wakes up → prints "site-a.com is up!" → done
Iteration 5: task_b wakes up → prints "site-b.com is up!" → done
Iteration 6: task_c wakes up → prints "site-c.com is up!" → done

Checking site-a.com...
Checking site-b.com...
Checking site-c.com...
site-a.com is up!
site-b.com is up!
site-c.com is up!
Done in 2.0 seconds

Two seconds. Down from six. CPU did same amount of work. We stopped wasting time. Cook started all three pots boiling at once instead of watching each one separately.

gather: Running Things Together

Creating tasks and awaiting them one by one works, but verbose. asyncio.gather is shortcut that runs multiple coroutines concurrently and collects results.

async def check_site(url):
    print(f"Checking {url}...")
    await asyncio.sleep(2)
    return f"{url} is up!"

async def main():
    results = await asyncio.gather(
        check_site("site-a.com"),
        check_site("site-b.com"),
        check_site("site-c.com"),
    )
    for r in results:
        print(r)

gather creates tasks for each coroutine, runs them concurrently, and returns results list in input order. Workhorse of asyncio concurrency.

There is also asyncio.as_completed() if you want results as they finish instead of waiting for all:

async def main():
    coros = [check_site(url) for url in urls]
    for future in asyncio.as_completed(coros):
        result = await future
        print(result)  # prints in completion order, not submission order

Since Python 3.11, asyncio.TaskGroup adds structured concurrency. Idea: concurrent task group should have clear lifetime. Tasks start together, and if one fails, rest get cancelled instead of dangling.

async def main():
    async with asyncio.TaskGroup() as tg:
        tg.create_task(check_site("site-a.com"))
        tg.create_task(check_site("site-b.com"))
        tg.create_task(check_site("site-c.com"))
    # all done by the time we get here

async with here is async context manager. It works like regular with, but setup and teardown are awaitable. You will see async with often in asyncio code, especially for connections and sessions needing async cleanup.

Real Example: Checking Multiple Websites

Running example used asyncio.sleep as stand-in for network requests. Now build complete runnable version with real network I/O.

import asyncio
import aiohttp
import time

async def check_site(session, url):
    try:
        async with session.get(url, timeout=aiohttp.ClientTimeout(total=5)) as response:
            return f"{url} — {response.status}"
    except Exception as e:
        return f"{url} — Error: {e}"

async def main():
    urls = [
        "https://www.google.com",
        "https://www.github.com",
        "https://www.python.org",
        "https://www.stackoverflow.com",
        "https://httpbin.org/delay/2",
    ]

    start = time.time()

    async with aiohttp.ClientSession() as session:
        results = await asyncio.gather(
            *[check_site(session, url) for url in urls]
        )

    for r in results:
        print(r)

    print(f"\nDone in {time.time() - start:.1f} seconds")

asyncio.run(main())

Five websites, checked concurrently. Total time is roughly slowest single response, not sum of all responses. Payoff.

Notice aiohttp, not requests. requests is blocking — it cannot yield control to event loop. Asyncio needs async-aware libraries. Biggest practical gotcha. Cook can juggle only tasks that know how to say "I'm waiting." If task refuses to put down knife — blocks without yielding — whole kitchen freezes.

asyncio.Queue and Producer-Consumer

So far website checker fires all requests at once. Fine for 5 URLs. What about 10,000? Ten thousand concurrent connections can trigger rate limits, bans, or exhaust your machine's file descriptors.

Need bounded concurrency — say 3 workers checking URLs from shared queue. That is producer-consumer pattern.

import asyncio

async def producer(queue, urls):
    for url in urls:
        await queue.put(url)
    for _ in range(3):  # one sentinel per consumer
        await queue.put(None)

async def consumer(queue, name):
    while True:
        url = await queue.get()
        if url is None:
            break
        print(f"{name}: checking {url}")
        await asyncio.sleep(1)  # simulate network request
        print(f"{name}: {url} done!")
        queue.task_done()

async def main():
    queue = asyncio.Queue()
    urls = [f"site-{i}.com" for i in range(9)]

    await asyncio.gather(
        producer(queue, urls),
        consumer(queue, "worker-1"),
        consumer(queue, "worker-2"),
        consumer(queue, "worker-3"),
    )

asyncio.run(main())

Three consumers pull URLs from queue and check them concurrently. Queue handles coordination. No locks. No race conditions. No shared mutable state. Nine sites, three workers, one second each — done in 3 seconds instead of 9. Pattern scales cleanly.

Common Pitfalls

After some asyncio time, same mistakes keep appearing. First one still gets me sometimes.

Calling a coroutine without awaiting it. Write check_site("example.com") without await, you get coroutine object and no work happens. If lucky, console warns quietly. Fix: await check_site("example.com") or asyncio.create_task(check_site("example.com")).

async def main():
    check_site("example.com")  # BUG: does nothing, returns a coroutine object
    await check_site("example.com")  # correct

Using blocking code inside async functions. time.sleep(5) is not async-aware. It freezes entire event loop for 5 seconds. Every other task stops. Cook fell asleep at counter — toaster dings, coffee finishes, nobody notices. Use await asyncio.sleep(5) instead, or use asyncio.to_thread() (Python 3.9+) for blocking libraries you cannot avoid:

async def bad():
    time.sleep(5)  # blocks the ENTIRE event loop

async def good():
    await asyncio.to_thread(time.sleep, 5)  # runs in a thread, loop stays free

CPU-bound work starving the event loop. Tight computation with no await means loop gets no chance to switch tasks. Everything else freezes. For CPU-bound work, use asyncio.to_thread() or concurrent.futures.ProcessPoolExecutor.

gather swallowing exceptions. By default, gather raises first exception and propagates it. Remaining tasks keep running in background, and later exceptions may disappear silently. return_exceptions=True is often right when gathered tasks can fail independently:

results = await asyncio.gather(
    might_fail_1(),
    might_fail_2(),
    return_exceptions=True,  # exceptions returned as values, not raised
)
# inspect each result individually

Trying to run asyncio inside asyncio. If you already sit inside event loop and call asyncio.run(), you get RuntimeError. Already in kitchen. No need to build second kitchen. Await coroutine directly.

When NOT to Use asyncio

Asyncio is not always better than synchronous code. It adds complexity — async and await spread up entire call stack. Make one function async, every caller must become async too. This is function coloring problem: code splits into two colors — sync and async — and they mix poorly.

Do not reach for asyncio when I/O is minimal. Reading one file and processing it does not justify event-loop overhead. Do not use it for CPU-bound work — asyncio cannot parallelize computation. Do not use it when dependencies are all blocking — if you must keep requests, psycopg2, or similar blocking libraries and cannot switch, you may spend more time fighting asyncio than gaining from it. Do not use it for script that already finishes in 3 seconds. Sometimes sequential code is feature, not compromise.

Asyncio shines with many network requests, web servers, APIs, bots, scrapers, or hundreds of overlapping I/O-bound tasks. If code spends most time waiting on world outside CPU, asyncio is almost certainly right call. Gray area exists — threading may be simpler for modest concurrency — but above a few dozen concurrent I/O operations, asyncio usually wins cleanly.
