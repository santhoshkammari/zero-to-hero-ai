Things like __get__, metaclass=, or triple-nested decorator factory look impossible until you see machinery. Not everyday features. Spot them fast, get X-ray vision for framework source. "Magic" in Django, SQLAlchemy, and PyTorch stops being magic. Becomes machinery you can understand.

We use small mental scenario: you write tiny ML experiment tracker — logs runs, stores configs, wraps training functions. Each feature shows up because we need it. Not because syllabus says next.

The Gotchas That Bite First

Before build, see traps. These bugs throw no errors. They silently give wrong results, and you blame model for hours when problem was Python quirk.

Mutable Default Arguments

Suppose experiment tracker has function that collects tags for run.

def log_run(name, tags=[]):
    tags.append("v1")
    return {"name": name, "tags": tags}

print(log_run("run_a"))  # {'name': 'run_a', 'tags': ['v1']}
print(log_run("run_b"))  # {'name': 'run_b', 'tags': ['v1', 'v1']}  ← wait, what?

Second call inherits tags from first. Default list [] gets created once — when Python reads function definition, not each call. Every call without tags shares same list object in memory. Like giving everyone in restaurant same plate — first person adds food, next person gets it too.

This burned me in production. Logging function accumulated metadata across requests, and bug showed only under load because sequential test calls hid it. Fix: use None as sentinel.

def log_run(name, tags=None):
    if tags is None:
        tags = []
    tags.append("v1")
    return {"name": name, "tags": tags}

Now each call gets own fresh list. None sentinel pattern appears everywhere in Python codebases. When you see it, you know why.

Late Binding Closures

Say we want batch of callback functions — one for each metric tracker records.

metrics = ["loss", "accuracy", "f1"]
callbacks = []
for m in metrics:
    callbacks.append(lambda: print(f"Recording {m}"))

callbacks[0]()  # Recording f1
callbacks[1]()  # Recording f1
callbacks[2]()  # Recording f1

All three print "f1." All. Python closures capture variables by reference, not value. When callback runs, loop already finished and m points to last value. Like phone number on Post-it that always shows last number written on whiteboard, not value from when you stuck it.

# Fix: bind the current value as a default argument
callbacks = []
for m in metrics:
    callbacks.append(lambda m=m: print(f"Recording {m}"))

callbacks[0]()  # Recording loss ✓

m=m trick captures value at that moment by binding it to default parameter. Looks odd. Standard pattern.

is vs ==

One more trap. == checks equal values. is checks same object in memory. Most times, you want ==. But CPython caches small integers (-5 to 256) and some strings, so is can seem right for wrong reason.

a = 256
b = 256
print(a is b)   # True — CPython caches this integer

a = 257
b = 257
print(a is b)   # False — not cached, different objects
print(a == b)   # True — same value

Rule: use is for None checks (if x is None). Use == for everything else. Code that uses is for value comparison may pass tests with small numbers and fail in production with large ones. I still catch this in code reviews, sometimes mine.

Shallow vs Deep Copy

Tracker stores nested config dicts. Copy config to tweak for new run. Last gotcha hits here.

import copy

config = {"model": "resnet", "params": {"lr": 0.001, "layers": [64, 128]}}
config_v2 = copy.copy(config)          # shallow copy
config_v2["params"]["lr"] = 0.01

print(config["params"]["lr"])           # 0.01 ← original is mutated!

copy.copy makes new outer dict, but inner objects stay shared references. Like photocopying folder — new folder, same physical pages inside. Mutate one, mutate both.

config_v2 = copy.deepcopy(config)       # recursively copies everything
config_v2["params"]["lr"] = 0.01
print(config["params"]["lr"])           # 0.001 ✓ original untouched

deepcopy walks whole object tree and makes independent copies at every level. Slower, but only safe choice for nested mutable structures.

Decorators Under the Hood

Tracker needs timing and logging on every model-training function. We could paste timing code everywhere. Bad path. Decorators exist for this — and understanding how they work turns them from "syntax I copy" into "machinery I control."

Decorator is not special syntax. It is function that takes function and returns function. That is all. When you write this:

@timer
def train(epochs):
    ...

Python does exactly this:

def train(epochs):
    ...
train = timer(train)

@ symbol is syntactic sugar. Nothing more. Real function train gets replaced by whatever timer returns. If timer returns wrapper function, that wrapper runs instead of original whenever anyone calls train().

Why functools.wraps Matters

Here is decorator without it.

def timer(func):
    def wrapper(*args, **kwargs):
        import time
        start = time.time()
        result = func(*args, **kwargs)
        print(f"{func.__name__} took {time.time() - start:.2f}s")
        return result
    return wrapper

@timer
def train(epochs): ...

print(train.__name__)  # "wrapper" — not "train"

Original function's name, docstring, and module vanish. Stack traces show "wrapper." Debug tools show "wrapper." Introspection breaks. In production, when you read logs at 2 AM trying to find slow function, "wrapper" tells you nothing.

from functools import wraps

def timer(func):
    @wraps(func)          # copies __name__, __doc__, __module__ from func
    def wrapper(*args, **kwargs):
        import time
        start = time.time()
        result = func(*args, **kwargs)
        print(f"{func.__name__} took {time.time() - start:.2f}s")
        return result
    return wrapper

print(train.__name__)  # "train" ✓

@wraps(func) is one-line fix that preserves decorated function's identity. Every decorator you write should include it. Every library decorator missing it has small bug.

Decorator Factories — When You Need Arguments

Want timer to print only when execution passes threshold? We need decorator that takes parameters. That means one more nesting layer.

def timer(threshold=1.0):
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            import time
            start = time.time()
            result = func(*args, **kwargs)
            elapsed = time.time() - start
            if elapsed > threshold:
                print(f"⚠ {func.__name__} took {elapsed:.2f}s (threshold: {threshold}s)")
            return result
        return wrapper
    return decorator

@timer(threshold=0.5)
def train(epochs): ...

Three levels deep. Outermost function (timer) takes arguments and returns real decorator. Middle function (decorator) takes function. Innermost (wrapper) does work. I still double-check this nesting every time. Makes sense once you trace it. Feels confusing every time you write from scratch.

Stacking Decorators

Stack decorators. They apply bottom-up, execute top-down.

@authenticate    # outer: runs first on call
@timer           # inner: applied first at definition
def train(epochs): ...

Equivalent to train = authenticate(timer(train)). At definition time, timer wraps train first, then authenticate wraps result. On call, authenticate's wrapper runs first (outermost layer), then timer's, then original function. Like Russian dolls — build inside out, open outside in.

Context Managers — Cleanup You Can't Forget

Tracker opens files for logging, acquires database connections, temporarily changes working directories. Same pattern: acquire resource, use resource, guarantee cleanup even if something explodes. Context managers formalize that.

with statement does nothing fancy. It calls __enter__ when block starts and __exit__ when block ends — including on exceptions. That guarantee is whole point.

class DatabaseConnection:
    def __enter__(self):
        self.conn = connect_to_db()
        return self.conn

    def __exit__(self, exc_type, exc_val, exc_tb):
        self.conn.close()             # always runs, even on exception
        return False                  # don't suppress the exception

with DatabaseConnection() as conn:
    conn.execute("INSERT ...")

If INSERT fails, __exit__ still runs and closes connection. No leaked handles. No finally blocks to remember. Cleanup baked into structure.

But writing class with __enter__ and __exit__ for every small cleanup task gets tedious. contextlib helps.

from contextlib import contextmanager

@contextmanager
def temp_directory(path):
    original = os.getcwd()
    os.chdir(path)
    try:
        yield path          # everything before yield = __enter__
    finally:
        os.chdir(original)  # everything after yield = __exit__

with temp_directory("/data/experiment_42"):
    # we're in the temp dir here
    save_results()
# back to original directory, guaranteed

Code before yield is setup. Code after is teardown. try/finally guarantees cleanup even on exceptions. Most library authors write context managers this way — more readable than full class.

One more tool: ExitStack. When you do not know beforehand how many resources you need — say, variable number of log files — ExitStack manages them dynamically.

from contextlib import ExitStack

with ExitStack() as stack:
    files = [stack.enter_context(open(f)) for f in log_paths]
    # all files open, all will be closed when we leave this block

Descriptors — The Machinery Behind @property

When you put @property on method, it feels like special language feature. Not so. It is descriptor — object that implements __get__, __set__, or __delete__. Python calls these methods automatically on attribute access. Whole attribute access system runs on this protocol.

class Celsius:
    def __init__(self):
        self._temp = 0

    @property
    def fahrenheit(self):
        return self._temp * 9/5 + 32

t = Celsius()
print(t.fahrenheit)   # 32.0

Under hood, property is class that defines __get__. When you access t.fahrenheit, Python does not find regular attribute — it finds descriptor object on class and calls its __get__ method, which runs your getter. @classmethod and @staticmethod work same way — descriptor objects intercepting attribute access.

Subtle point. Python separates data descriptors (define __set__ or __delete__) from non-data descriptors (only __get__). Data descriptors beat instance attributes. Non-data descriptors do not. That is why you cannot accidentally override property by assigning same name, but you can shadow regular method.

Lookup order (data descriptor → instance dict → non-data descriptor) takes time to internalize. I still draw it on paper when debugging weird behavior. No need to memorize it. But know it exists. Next time attribute access acts strange, you know where to look.

Metaclasses — The Nuclear Option

Metaclass is class whose instances are classes. If that sentence made you stop, good. Same here. Analogy: if class is blueprint for objects, metaclass is blueprint for blueprints. It controls what happens when class gets defined, not when instances get created.

Where do you see them? ORMs. When you write this in Django:

class Article(models.Model):
    title = models.CharField(max_length=200)
    published = models.BooleanField(default=False)

You are not writing normal class. Metaclass ModelBase intercepts class definition and does big work: inspects every attribute, builds database column mappings from field objects, registers model with Django app registry, sets up query managers, wires signal handlers. All before single instance exists. That is why Article.objects.filter(published=True) works — metaclass created objects for you.

But here is key point — you will read metaclass code far more often than you will write it. And for most cases where you think you need metaclass, Python 3.6+ has simpler alternative.

class Plugin:
    registry = {}

    def __init_subclass__(cls, **kwargs):
        super().__init_subclass__(**kwargs)
        Plugin.registry[cls.__name__] = cls

class AudioPlugin(Plugin): ...
class VideoPlugin(Plugin): ...

print(Plugin.registry)  # {'AudioPlugin': , 'VideoPlugin': }

__init_subclass__ runs every time subclass gets created. It handles most common metaclass use case — automatic registration — without metaclass machinery. Covers about 80% of cases that needed metaclass in older Python. Remaining 20% involves changing class namespace before class body executes. That truly needs metaclass, and you mainly meet it while reading ORM and framework source.

Modern Python Worth Knowing

Python has changed faster in recent versions than most people think. Few features crossed line from "cool new thing" to "shows up in production code."

The Walrus Operator (:=)

Assign and use value in single expression. Looks strange first time. Then becomes indispensable in specific patterns.

# without walrus — compute len twice or use a temp variable
data = get_batch()
if len(data) > 0:
    process(data)

# with walrus — assign and test in one shot
if (n := len(data := get_batch())) > 0:
    print(f"Processing {n} items")
    process(data)

Best use: while loops and comprehension filters. Alternative is awkward repeated computation or clunky pre-assignment.

Structural Pattern Matching

Python 3.10 added match/case. Much stronger than switch statement. It destructures data.

def handle_event(event):
    match event:
        case {"type": "metric", "name": name, "value": float(v)}:
            record_metric(name, v)
        case {"type": "error", "message": msg}:
            log_error(msg)
        case [x, y, *rest]:
            print(f"Sequence starting with {x}, {y}")
        case _:
            print("Unknown event")

Not string matching. Structural matching — checks data shape and binds variables on way through. For parsing API responses, handling command-line arguments, routing messages — anywhere you write if isinstance(...) elif isinstance(...) chains — pattern matching is cleaner.

f-string Debug Mode

Tiny 3.8 feature. Add = after expression, Python prints expression and value.

batch_size = 64
lr = 3e-4
print(f"{batch_size=}, {lr=}")
# batch_size=64, lr=0.0003

print(f"{len(train_data)=}")
# len(train_data)=50000

Saves typing variable name twice while debugging. Small feature. Hard to give up once you use it.

Dataclasses Beyond the Basics

Basic @dataclass gives __init__, __repr__, and __eq__ for free. Advanced options make it far more powerful.

from dataclasses import dataclass, field

@dataclass(frozen=True, slots=True)
class ExperimentConfig:
    model: str
    lr: float = 0.001
    tags: list[str] = field(default_factory=list)

    def __post_init__(self):
        if self.lr <= 0:
            raise ValueError(f"Learning rate must be positive, got {self.lr}")

frozen=True makes instances immutable — changing field raises error. Great for configs that must not change mid-experiment. slots=True (Python 3.10+) removes instance __dict__, cuts memory use, speeds attribute access. field(default_factory=list) fixes mutable default gotcha from earlier — each instance gets own list. __post_init__ runs after auto-generated __init__, giving place for validation without writing whole __init__ yourself.

For configs, data transfer objects, and anything that is "bag of named values with validation," dataclasses with these options have mostly replaced manual __init__ methods in modern Python.

Type Hints That Actually Help

Beyond basic int and str annotations, two typing features are genuinely useful in production.

Protocol is structural subtyping — compile-time duck typing. Instead of requiring inheritance, you define methods type must have, and any class with them qualifies.

from typing import Protocol

class Trainable(Protocol):
    def fit(self, X, y) -> None: ...
    def predict(self, X) -> list: ...

def evaluate(model: Trainable, test_X, test_y):
    preds = model.predict(test_X)
    ...

Any class with fit and predict satisfies Trainable. No inheritance needed. This is how you type-hint "anything that looks like scikit-learn model" without coupling to scikit-learn.

New type parameter syntax in Python 3.12 cleans up generics a lot.

# old way — separate TypeVar declaration
from typing import TypeVar
T = TypeVar("T")
def first(items: list[T]) -> T: ...

# new way (3.12+) — inline, clean
def first[T](items: list[T]) -> T: ...

Same meaning. Less boilerplate. You will see this in newer libraries that dropped pre-3.12 support.

The GIL — What It Actually Means

Global Interpreter Lock is Python's most misunderstood feature. In CPython (standard interpreter), only one thread can execute Python bytecode at time, even on multi-core machine. This does not make threading useless — it means you must know when it helps and when it does not.

For I/O-bound work (network requests, file reads, database queries), threading works fine. GIL releases while waiting for I/O, so threads can truly run concurrently. For CPU-bound work (matrix math, data processing), threads give zero speedup because GIL blocks parallel execution. Use multiprocessing instead — separate processes, separate GILs.

Then asyncio. It handles I/O concurrency with single thread using cooperative scheduling. No GIL contention, no threading overhead, scales to thousands of concurrent connections. Most modern Python web frameworks (FastAPI, newer Django) build on it.

The Point of This Section

You will not use these every day. They are recognition tools — X-ray vision for framework source, so you see machinery instead of magic. Next time you hit triple-nested decorator, descriptor protocol method, or metaclass in someone's ORM, you will not back out of room. You will know what happens under hood. That changes everything.
