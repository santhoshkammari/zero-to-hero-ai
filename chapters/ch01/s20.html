Concurrency and Parallelism

Chapter 1: Python & Programming Foundations 5 subtopics
TL;DR

Your ML pipeline spends most of its time waiting—waiting on disk reads, API responses, or number-crunching across millions of rows. Concurrency lets you overlap that waiting; parallelism lets you do multiple computations at once on separate CPU cores. Python's Global Interpreter Lock (GIL) means threads share one core for CPU work, so you reach for multiprocessing or concurrent.futures when you need real speed. Pick threads for I/O-bound tasks, processes for CPU-bound tasks, and joblib when scikit-learn already wired it up for you.

Why This Matters for ML

Picture this: you have a training pipeline that downloads 50,000 images from S3, preprocesses each one (resize, normalize, augment), then feeds them into a model. The download step is waiting on the network. The preprocessing step is crunching numbers on your CPU. If you do these one at a time, sequentially, your eight-core machine sits mostly idle. That is money and time going to waste.

Or consider hyperparameter search. You need to train the same model 200 times with different configurations. Each training run is independent. If you run them one after another, you are using one core out of eight. That is 12.5% utilization. Your cloud bill stays the same whether you use one core or all eight.

Concurrency means dealing with multiple things at once—overlapping tasks so that while one waits, another makes progress. Parallelism means doing multiple things at the exact same instant—literally running computations on different CPU cores simultaneously. Concurrency is about structure. Parallelism is about execution. You need both in real ML work.

The Global Interpreter Lock (GIL)

What the GIL Is and Why It Exists

Here is the thing that trips up every Python developer eventually. CPython—the standard Python interpreter you almost certainly use—has a Global Interpreter Lock (GIL). It is a mutex (a lock) that allows only one thread to execute Python bytecode at a time, even on a machine with 16 cores.

Think of it like this. You live in an apartment with four roommates, but there is only one bathroom. Everyone can plan their morning routine concurrently—pick out clothes, make coffee, check their phone—but only one person can use the bathroom at a time. The bathroom is the Python interpreter. The GIL is the lock on the door.

Why does CPython have this? Memory safety. Python's memory management uses reference counting. Every object tracks how many variables point to it. When that count hits zero, the object is freed. If two threads modified the reference count of the same object simultaneously without a lock, you would get corrupted memory—objects freed while still in use, or memory that never gets freed at all. The GIL was a pragmatic engineering choice made in the 1990s that kept CPython's internals safe and made writing C extensions much easier.

import sys

# Every Python object has a reference count
x = [1, 2, 3]
print(sys.getrefcount(x))  # Usually 2: 'x' + the temporary reference from getrefcount()

# The GIL protects this counter from corruption during multi-threaded access

Impact on Multi-threaded Programs

The GIL does not mean threads are useless. It means threads cannot speed up CPU-bound Python code. When your code is doing math, string processing, or list manipulation—pure Python computation—only one thread runs at a time regardless of how many you create.

But when your code is waiting for a file to download, a database query to return, or a socket to receive data, the GIL is released. The thread says "I am going to wait for the network, someone else can use the interpreter." That is why threads still shine for I/O-bound work.

import threading
import time

def cpu_work(n):
    """Pure CPU-bound: summing numbers."""
    total = 0
    for i in range(n):
        total += i
    return total

# Sequential
start = time.perf_counter()
cpu_work(10_000_000)
cpu_work(10_000_000)
seq_time = time.perf_counter() - start

# Threaded (won't be faster due to GIL)
start = time.perf_counter()
t1 = threading.Thread(target=cpu_work, args=(10_000_000,))
t2 = threading.Thread(target=cpu_work, args=(10_000_000,))
t1.start(); t2.start()
t1.join(); t2.join()
thread_time = time.perf_counter() - start

print(f"Sequential: {seq_time:.2f}s")
print(f"Threaded:   {thread_time:.2f}s")
# Both are roughly the same. Threads did NOT help here.

PEP 703 and Free-Threaded Python (3.13+)

The landscape is changing. PEP 703 introduced an experimental "free-threaded" build of CPython starting with Python 3.13. This build removes the GIL entirely, allowing true parallel execution of threads on multiple cores. As of 3.13, you opt in with a special build flag (--disable-gil). It is experimental—some C extensions break, performance of single-threaded code takes a small hit, and the ecosystem needs time to adapt. But the direction is clear: Python's future is GIL-free. For now, know it exists, watch the progress, and use the battle-tested approaches below.

The GIL Decision Rule

If your task is I/O-bound (downloading, reading files, querying APIs), use threads. The GIL releases during I/O waits, so threads overlap the waiting efficiently. If your task is CPU-bound (math, preprocessing, encoding), use processes. Each process has its own interpreter and its own GIL, so they truly run in parallel.

Threading

The threading Module

Python's threading module lets you run functions in separate threads within a single process. All threads share the same memory space—they can read and write the same variables. That is both the strength (easy data sharing) and the danger (race conditions).

import threading
import urllib.request
import time

def download(url, dest):
    """Download a file from a URL."""
    urllib.request.urlretrieve(url, dest)
    print(f"Downloaded {dest}")

urls = [
    ("https://example.com/data1.csv", "data1.csv"),
    ("https://example.com/data2.csv", "data2.csv"),
    ("https://example.com/data3.csv", "data3.csv"),
]

# Without threads: each download waits for the previous one
start = time.perf_counter()
for url, dest in urls:
    download(url, dest)
print(f"Sequential: {time.perf_counter() - start:.2f}s")

# With threads: downloads overlap
start = time.perf_counter()
threads = []
for url, dest in urls:
    t = threading.Thread(target=download, args=(url, dest))
    t.start()
    threads.append(t)

for t in threads:
    t.join()  # Wait for all threads to finish
print(f"Threaded: {time.perf_counter() - start:.2f}s")
# Threaded version is roughly 3x faster (limited by the slowest download)

I/O-Bound vs CPU-Bound Tasks

This distinction is the single most important concept in this entire section. Get it wrong and you will write code that is slower with concurrency than without.

I/O-bound means your program spends most of its time waiting for input/output operations: network requests, file reads, database queries. The CPU is idle. Threads fix this because while one thread waits on I/O, another can do useful work.

CPU-bound means your program spends most of its time doing computation: transforming data, training a model, encoding features. The CPU is the bottleneck. Threads do not help here (the GIL blocks parallel execution). You need separate processes, each with its own CPU core.

In ML, you hit both constantly. Loading a dataset from disk or an API? I/O-bound. Tokenizing text, resizing images, computing features? CPU-bound. A good pipeline uses threads for the loading stage and processes for the transformation stage.

Thread Safety and Locks

When two threads read and write the same variable, you get a race condition. The result depends on which thread runs first—it is nondeterministic and the bugs are brutal to track down.

import threading

# BROKEN: Race condition
counter = 0

def increment(n):
    global counter
    for _ in range(n):
        counter += 1  # Read counter, add 1, write back — NOT atomic

threads = [threading.Thread(target=increment, args=(100_000,)) for _ in range(5)]
for t in threads:
    t.start()
for t in threads:
    t.join()

print(f"Expected: 500000, Got: {counter}")
# You'll often get something less than 500000!

The problem: counter += 1 looks like one operation, but it is three—read the value, add one, write back. Thread A reads 42, then Thread B reads 42 (before A writes), both write 43. You lost an increment.

The fix: a Lock. Only the thread holding the lock can enter the critical section.

import threading

counter = 0
lock = threading.Lock()

def safe_increment(n):
    global counter
    for _ in range(n):
        with lock:          # Acquire lock, release automatically when done
            counter += 1    # Only one thread runs this at a time

threads = [threading.Thread(target=safe_increment, args=(100_000,)) for _ in range(5)]
for t in threads:
    t.start()
for t in threads:
    t.join()

print(f"Expected: 500000, Got: {counter}")
# Now it's always 500000
Lock Overhead

Locks serialize access—they force threads to take turns. If your critical section is large or frequently entered, the lock becomes a bottleneck and you lose most of the concurrency benefit. Keep locked sections as short as possible. If you need heavy shared-state manipulation, consider using a Queue instead—it handles locking internally.

Multiprocessing

The multiprocessing Module

When you need to bypass the GIL and use all your CPU cores, you spawn separate processes. Each process has its own Python interpreter, its own GIL, and its own memory space. They truly run in parallel.

import multiprocessing
import time
import math

def heavy_computation(n):
    """CPU-bound: compute sum of square roots."""
    return sum(math.sqrt(i) for i in range(n))

if __name__ == "__main__":
    N = 5_000_000

    # Sequential
    start = time.perf_counter()
    results = [heavy_computation(N) for _ in range(4)]
    print(f"Sequential: {time.perf_counter() - start:.2f}s")

    # Multiprocessing
    start = time.perf_counter()
    processes = []
    for _ in range(4):
        p = multiprocessing.Process(target=heavy_computation, args=(N,))
        p.start()
        processes.append(p)
    for p in processes:
        p.join()
    print(f"Multiprocessing: {time.perf_counter() - start:.2f}s")
    # On a 4+ core machine, roughly 3-4x faster
The if __name__ == "__main__" Guard

On Windows and macOS (with the "spawn" start method), new processes import your script to set themselves up. Without the if __name__ == "__main__" guard, each new process would try to spawn more processes, creating an infinite recursion bomb. Always wrap multiprocessing code in this guard. Even on Linux (which uses "fork"), it is good practice.

Process Pools

Manually creating and managing processes is tedious. A Pool gives you a fixed number of worker processes and distributes tasks to them automatically.

import multiprocessing
import math

def process_chunk(data_chunk):
    """Simulate CPU-heavy preprocessing on a chunk of data."""
    return [math.sqrt(abs(x)) * math.log1p(abs(x)) for x in data_chunk]

if __name__ == "__main__":
    # Simulate a large dataset split into chunks
    data = list(range(1, 2_000_001))
    chunk_size = len(data) // 4
    chunks = [data[i:i + chunk_size] for i in range(0, len(data), chunk_size)]

    with multiprocessing.Pool(processes=4) as pool:
        results = pool.map(process_chunk, chunks)

    # Flatten results
    flat_results = [x for chunk in results for x in chunk]
    print(f"Processed {len(flat_results)} items across 4 workers")

pool.map() works like the built-in map() but distributes the calls across worker processes. It blocks until all results are ready. For more control, there is pool.apply_async() which returns a result object you can check later.

Shared State: Value, Array, Manager, and Queue

Because each process has its own memory, sharing data between processes requires special tools. This is the hardest part of multiprocessing.

multiprocessing.Value and multiprocessing.Array use shared memory—memory that multiple processes can access directly. They are fast but limited to simple C types (ints, floats, arrays of them).

import multiprocessing

def worker(shared_counter, lock):
    for _ in range(100_000):
        with lock:
            shared_counter.value += 1

if __name__ == "__main__":
    counter = multiprocessing.Value('i', 0)  # 'i' = signed int, initial value 0
    lock = multiprocessing.Lock()

    procs = [multiprocessing.Process(target=worker, args=(counter, lock)) for _ in range(4)]
    for p in procs:
        p.start()
    for p in procs:
        p.join()

    print(f"Counter: {counter.value}")  # 400000

For more complex data, multiprocessing.Manager provides shared dictionaries, lists, and other Python objects. Under the hood, a server process holds the real data and other processes communicate with it via proxies. It is more flexible but slower than shared memory.

The most practical tool for inter-process communication is the Queue. It is thread-safe and process-safe. One process puts items in; another takes them out. This is the producer-consumer pattern, and it is everywhere in ML pipelines.

import multiprocessing
import time

def producer(queue, items):
    """Simulate loading data and pushing to a queue."""
    for item in items:
        time.sleep(0.01)  # Simulate I/O delay
        queue.put(item)
    queue.put(None)  # Sentinel: signals "I'm done"

def consumer(queue):
    """Pull items from queue and process them."""
    processed = 0
    while True:
        item = queue.get()
        if item is None:
            break
        _ = item ** 2  # Simulate processing
        processed += 1
    print(f"Consumer processed {processed} items")

if __name__ == "__main__":
    queue = multiprocessing.Queue(maxsize=50)  # Bounded queue: backpressure!
    items = list(range(200))

    prod = multiprocessing.Process(target=producer, args=(queue, items))
    cons = multiprocessing.Process(target=consumer, args=(queue,))

    prod.start()
    cons.start()
    prod.join()
    cons.join()

Serialization Challenges (pickle)

When Python sends data between processes, it uses pickle—Python's serialization protocol. The data is converted to bytes, sent across, and reconstructed on the other side. This has real consequences.

First, not everything is picklable. Lambda functions, inner functions, open file handles, database connections—these all fail. If your worker function is a lambda, multiprocessing will throw a PicklingError. Second, pickling large objects (like a 2 GB DataFrame) is slow and memory-hungry—you are essentially copying the data. For large datasets, consider memory-mapped files or shared memory (multiprocessing.shared_memory in Python 3.8+) instead.

import multiprocessing

# This FAILS: lambdas can't be pickled
# pool.map(lambda x: x ** 2, range(10))  # PicklingError!

# This works: top-level named function
def square(x):
    return x ** 2

if __name__ == "__main__":
    with multiprocessing.Pool(2) as pool:
        results = pool.map(square, range(10))
    print(results)  # [0, 1, 4, 9, 16, 25, 36, 49, 64, 81]

concurrent.futures — The Modern API

The concurrent.futures module, introduced in Python 3.2, provides a clean, unified interface for both threading and multiprocessing. It is the API you should reach for first in most cases. Same interface, same patterns—you swap one executor for another depending on whether your work is I/O-bound or CPU-bound.

ThreadPoolExecutor

from concurrent.futures import ThreadPoolExecutor
import urllib.request
import time

def fetch_url(url):
    """Download a URL and return its size."""
    with urllib.request.urlopen(url) as response:
        data = response.read()
    return len(data)

urls = [
    "https://example.com",
    "https://httpbin.org/get",
    "https://jsonplaceholder.typicode.com/posts",
    "https://api.github.com",
]

# Submit tasks and collect results
start = time.perf_counter()
with ThreadPoolExecutor(max_workers=4) as executor:
    # executor.map preserves order, returns an iterator of results
    sizes = list(executor.map(fetch_url, urls))

elapsed = time.perf_counter() - start
for url, size in zip(urls, sizes):
    print(f"{url}: {size} bytes")
print(f"Completed in {elapsed:.2f}s")

ProcessPoolExecutor

Same interface, different engine. Swap ThreadPoolExecutor for ProcessPoolExecutor and your CPU-bound work now runs across multiple cores.

from concurrent.futures import ProcessPoolExecutor
import math

def compute_primes(limit):
    """Count primes up to limit using trial division."""
    count = 0
    for num in range(2, limit):
        if all(num % d != 0 for d in range(2, int(math.sqrt(num)) + 1)):
            count += 1
    return count

if __name__ == "__main__":
    ranges = [100_000, 120_000, 110_000, 130_000]

    # Parallel prime counting across 4 processes
    with ProcessPoolExecutor(max_workers=4) as executor:
        results = list(executor.map(compute_primes, ranges))

    for limit, count in zip(ranges, results):
        print(f"Primes below {limit}: {count}")

Future Objects and Callbacks

When you need more control than executor.map() provides, use executor.submit(). It returns a Future object—a placeholder for a result that does not exist yet. You can check if it is done, wait for it, get its result, or attach a callback that fires when it completes.

from concurrent.futures import ThreadPoolExecutor, as_completed
import time
import random

def simulate_api_call(endpoint):
    """Simulate an API call with variable latency."""
    delay = random.uniform(0.5, 2.0)
    time.sleep(delay)
    return {"endpoint": endpoint, "latency": round(delay, 2), "status": 200}

endpoints = ["/users", "/products", "/orders", "/analytics", "/health"]

with ThreadPoolExecutor(max_workers=3) as executor:
    # Submit tasks individually — get Future objects back
    future_to_endpoint = {
        executor.submit(simulate_api_call, ep): ep for ep in endpoints
    }

    # as_completed yields futures as they finish (not in submission order)
    for future in as_completed(future_to_endpoint):
        endpoint = future_to_endpoint[future]
        try:
            result = future.result()  # Blocks only if this future isn't done yet
            print(f"{result['endpoint']}: {result['latency']}s")
        except Exception as exc:
            print(f"{endpoint} raised {exc}")

The as_completed() function is powerful for ML work. Imagine launching 20 model training runs with different hyperparameters. Instead of waiting for all 20 to finish, you process results as they arrive—log metrics, update a leaderboard, even cancel remaining runs if you found a good enough result.

You can also attach callbacks that run the moment a future completes:

from concurrent.futures import ThreadPoolExecutor

def fetch_data(source):
    # ... imagine downloading data here ...
    return f"data_from_{source}"

def on_complete(future):
    """Called automatically when the future finishes."""
    result = future.result()
    print(f"Callback fired: received {result}")

with ThreadPoolExecutor(max_workers=2) as executor:
    future = executor.submit(fetch_data, "database")
    future.add_done_callback(on_complete)
    # Callback runs in the executor thread — no need to poll
executor.map() vs executor.submit()

map() is the quick path: give it a function and an iterable, get results back in order. submit() is the flexible path: you get Future objects, can attach callbacks, handle exceptions per-task, and use as_completed() to process results as they arrive. Use map() when all tasks are identical. Use submit() when tasks vary or you want fine-grained control.

joblib for Scikit-learn Parallelization

If you use scikit-learn, you have probably seen the n_jobs parameter on estimators and utilities. That parameter is powered by joblib, a library purpose-built for embarrassingly parallel loops and transparent disk-caching of results.

The beauty of joblib in scikit-learn is that you do not need to think about processes, pools, or serialization. You set n_jobs=-1 (use all cores) and scikit-learn handles the rest.

from sklearn.model_selection import cross_val_score, GridSearchCV
from sklearn.ensemble import RandomForestClassifier
from sklearn.datasets import make_classification

# Generate synthetic data
X, y = make_classification(n_samples=10_000, n_features=20, random_state=42)
model = RandomForestClassifier(n_estimators=100, random_state=42)

# Cross-validation: 5 folds, each evaluated in a separate process
scores = cross_val_score(model, X, y, cv=5, n_jobs=-1)  # -1 = all cores
print(f"CV scores: {scores.mean():.3f} ± {scores.std():.3f}")

# Grid search: every parameter combination runs in parallel
param_grid = {
    "n_estimators": [50, 100, 200],
    "max_depth": [5, 10, 20, None],
    "min_samples_split": [2, 5, 10],
}

grid = GridSearchCV(model, param_grid, cv=3, n_jobs=-1, verbose=1)
grid.fit(X, y)
print(f"Best params: {grid.best_params_}")
print(f"Best score:  {grid.best_score_:.3f}")

Under the hood, joblib uses loky—a robust process pool executor that handles the messy edge cases of multiprocessing (worker crashes, memory leaks, pickling issues). You can also use joblib directly for your own parallel loops:

from joblib import Parallel, delayed
import numpy as np

def preprocess_image(image_path):
    """Simulate heavy image preprocessing."""
    # In real code: load, resize, normalize, augment
    return np.random.rand(224, 224, 3)  # Placeholder

image_paths = [f"img_{i}.jpg" for i in range(1000)]

# Process 1000 images across all CPU cores
results = Parallel(n_jobs=-1, verbose=5)(
    delayed(preprocess_image)(path) for path in image_paths
)
print(f"Preprocessed {len(results)} images")

The delayed() function is a wrapper that captures the function and its arguments without calling it. Parallel then distributes those captured calls across worker processes. The verbose parameter gives you a progress indicator—handy for long-running jobs.

joblib Backends

By default, joblib uses the loky backend (multiprocessing). You can switch to threading for I/O-bound work: Parallel(n_jobs=4, backend="threading"). Some libraries like NumPy release the GIL during heavy computations, so threading can actually speed up NumPy-heavy workloads without the overhead of process spawning.

Putting It All Together: The Decision Framework

Here is how to think about it. Start by profiling your bottleneck. Is your program spending time waiting or computing?

If your task is I/O-bound—downloading data, reading files, calling APIs—use ThreadPoolExecutor. Threads are lightweight, share memory, and the GIL releases during I/O waits. Start with a thread count of 2–4 times your core count for network-heavy work.

If your task is CPU-bound—transforming data, computing features, training models—use ProcessPoolExecutor or joblib.Parallel. Each process gets its own core and its own GIL. Set worker count equal to your core count (n_jobs=-1 in joblib).

If you are using scikit-learn and the function already has an n_jobs parameter, use it. Do not reinvent the wheel. Set n_jobs=-1 and move on.

If you need to share state between workers, prefer multiprocessing.Queue for producer-consumer patterns. Avoid shared memory unless you are sure you need it—the synchronization complexity is not worth it for most ML code.

If you are on Python 3.13+ and feeling adventurous, try the free-threaded build for CPU-bound threaded code. But for production ML pipelines today, processes remain the safe bet for CPU-bound parallelism.

# Quick reference: choosing the right tool

# I/O-bound (downloads, API calls, file reads)
from concurrent.futures import ThreadPoolExecutor
with ThreadPoolExecutor(max_workers=8) as executor:
    results = list(executor.map(fetch, urls))

# CPU-bound (math, preprocessing, feature engineering)
from concurrent.futures import ProcessPoolExecutor
with ProcessPoolExecutor(max_workers=4) as executor:
    results = list(executor.map(transform, data_chunks))

Concurrency and parallelism are not about clever tricks. They are about not leaving hardware idle. Every core sitting unused during a training pipeline is wasted time and money. The tools are there. The patterns are well-established. The hardest part is correctly diagnosing whether your bottleneck is I/O or CPU—get that right, and picking the tool is the easy part.
