Spotting honest vs. dishonest plots is learnable skill. When figure lies, data usually not wrong; visual encoding is. Here comes dive.

Data visualization means encoding data into visual form — positions, lengths, colors, shapes — so human perception finds patterns raw numbers hide. In Python ecosystem, this centers on matplotlib (created by John Hunter in 2003, modeled after MATLAB plotting), with seaborn, Plotly, and Altair layered on top. Field draws on decades of perceptual psychology and statistical graphics theory, especially Edward Tufte's work from 1980s.

Why We Visualize

You visualize data for two reasons. First: understand data yourself — exploration. Second: communicate findings to someone else — presentation. Different jobs. Different tools. Different standards.

Think napkin sketch vs. mural. Napkin sketch is for you. Messy, unlabeled, three overlapping plots crammed together. Goal: find surprises — is distribution bimodal? Are features correlated? Is cluster here that should not be? Speed matters. Polish does not.

Mural is for everyone else. Every axis needs label. Every color needs legend. Title must tell story, not describe variable. Chart must be honest — not exaggerate, hide, or mislead. We come back to what "honest" means later. More subtle than it looks.

The Anatomy of a Plot — Figure, Axes, and Artists

Before making plots, understand what matplotlib builds. This matters because many confusing matplotlib errors trace back to not understanding these three layers.

Imagine artist studio. Figure is canvas — blank rectangular surface everything lives on. You can have one canvas or many, but each is self-contained world. Axes is picture frame nailed to canvas — rectangular region where data gets drawn, with its own coordinate system, tick marks, and labels. One Figure can hold multiple Axes, same way one canvas can hold multiple framed pictures.

Everything you see — every line, dot, text element, tick mark — is an Artist. That is matplotlib's name for any visual element. Line2D object is Artist. Text object is Artist. Even Axes itself is Artist, because it has visible border. Figure is Artist too. Artists all way down.

Hierarchy in practice:

import matplotlib.pyplot as plt

fig = plt.figure(figsize=(8, 5))      # the canvas
ax = fig.add_subplot(111)              # a picture frame on the canvas
line, = ax.plot([1, 2, 3], [1, 4, 9]) # an Artist (Line2D) inside the frame

print(type(fig))   # matplotlib.figure.Figure
print(type(ax))    # matplotlib.axes._subplots.AxesSubplot
print(type(line))  # matplotlib.lines.Line2D

When matplotlib renders to screen or file, it walks this tree top-down. Figure tells each Axes to draw itself. Each Axes tells its Artists to draw themselves. Actual pixel work happens through Renderer (turns abstract shapes into pixels) and Canvas (surface pixels land on — screen, PNG, PDF). Different backends supply different Canvas-Renderer pairs, so same code can produce desktop window, PNG on server, or vector PDF for paper.

Reason for hammering this: Figure-Axes-Artist distinction is skeleton key to matplotlib. Once you see it, arbitrary things start making sense. Why both fig.suptitle() and ax.set_title()? Different hierarchy levels — one labels canvas, other labels frame. Why does fig.savefig() live on Figure? Because saving is canvas-level operation.

We return to this canvas-and-frame analogy often. It keeps paying off.

The State Machine Trap

Matplotlib offers two interfaces. One will betray you.

First interface is one most tutorials teach — pyplot state machine. You call plt.plot(), plt.title(), plt.xlabel(), and matplotlib keeps hidden "current figure" and "current axes" behind scenes. It works like drawing program with one active layer, and every command hits that layer.

plt.plot([1, 2, 3], [1, 4, 9])
plt.title("My Plot")
plt.xlabel("x")
plt.show()

This is fine for single one-off plot. Moment you need two plots, or function that creates plot, or loop that generates figures, hidden state becomes trap. Which figure is "current"? Which axes? You get plots on wrong figure, titles on wrong subplot, and no error message explaining it.

Second interface is object-oriented. You create Figure and Axes explicitly, and every operation runs on specific object you mean.

fig, ax = plt.subplots()
ax.plot([1, 2, 3], [1, 4, 9])
ax.set_title("My Plot")
ax.set_xlabel("x")
fig.savefig("my_plot.png")
plt.close(fig)

Same result, no ambiguity. ax object IS specific picture frame you draw on. You can pass it to function, return it, store it in list. It has clear identity, not side-channel to hidden global state.

Back to studio analogy: pyplot style is yelling commands into room and hoping right assistant hears. OO style is handing instructions directly to person doing work. One scales. Other does not.

For rest of this section, we use OO interface only. If you keep one thing, keep this: fig, ax = plt.subplots() is line separating code that works from code that works until it does not.

Our Running Example — Predicting House Prices

Need concrete scenario. Imagine building model to predict house prices from dataset of 500 homes. Dataset has five features: square footage, number of bedrooms, year built, distance to city center, and target — sale price. Data is loaded into pandas DataFrame called df.

First instinct: understand data. Not by staring at rows of numbers — by visualizing.

import matplotlib.pyplot as plt

fig, ax = plt.subplots(figsize=(8, 5))
ax.scatter(df["sqft"], df["price"], alpha=0.5, s=15)
ax.set_xlabel("Square Footage")
ax.set_ylabel("Sale Price ($)")
ax.set_title("House Prices vs. Size")
plt.close(fig)

Scatter plot. Square footage on x-axis, price on y-axis. Each dot is house. We immediately see what summary statistic would miss — cluster of expensive small homes near city center, and few suspiciously cheap large homes that may be data errors. Two seconds of looking teaches more than five minutes of df.describe().

That is power of visualization. Not chart itself — speed of perception. Visual cortex processes spatial patterns in milliseconds; verbal brain may take minutes. Visualization outsources computation to fastest processor you have: your eyes.

We keep returning to this house price dataset while exploring chart types and tools.

Choosing the Right Chart

Real data visualization skill has nothing to do with code. Ask right question first. Then pick chart that answers it.

When we stare at house price data, we should not think "I want histogram." We should think "How is price distributed? Is it normal? Is it skewed?" Chart type follows question. Sounds obvious, but people grab scatter plots by reflex, like carpenter grabbing hammer. Not everything is nail.

Back to houses. We want price distribution. Histogram chops price range into bins and counts how many houses fall into each. It answers "what is common and what is rare?"

fig, ax = plt.subplots(figsize=(8, 5))
ax.hist(df["price"], bins=30, edgecolor="white", alpha=0.8)
ax.set_xlabel("Sale Price ($)")
ax.set_ylabel("Count")
ax.set_title("Distribution of House Prices")
plt.close(fig)

Now we see right skew — most houses cluster in $200k–$400k range, with long tail of expensive outliers. That skew matters. It suggests log-transforming target before feeding it to linear model.

What about relationship between every pair of features? We could make scatter plots one at a time, but with five features that means ten combinations. Better question: "are any features strongly correlated?" Right answer: correlation heatmap.

import seaborn as sns

fig, ax = plt.subplots(figsize=(8, 6))
sns.heatmap(df.corr(), annot=True, fmt=".2f", cmap="coolwarm",
            center=0, ax=ax)
ax.set_title("Feature Correlations")
plt.close(fig)

Heatmap gives matrix of pairwise correlations. Warm colors for positive correlations, cool for negative, white for zero. At glance, we see square footage and price are strongly correlated (0.85), but year built and price are weakly correlated (0.12). If two features are highly correlated with each other but not target, one may be redundant.

What if we want to see how price differs between houses with 2, 3, and 4 bedrooms? Then we need box plot — it shows median, quartiles, and outliers for numeric variable split by categorical one.

fig, ax = plt.subplots(figsize=(8, 5))
sns.boxplot(data=df, x="bedrooms", y="price", ax=ax)
ax.set_title("Price Distribution by Bedroom Count")
plt.close(fig)

Box plot reveals what average would hide: 3-bedroom homes have widest price spread, suggesting bedroom count alone is weak predictor once you are in 3-bedroom category. Outlier dots above whiskers? Those are houses worth investigating one by one.

Mental framework: question determines chart.

Your QuestionChart TypeWhy This One
How is a single variable distributed?Histogram / KDEShows shape, skew, modes, outliers
How do two numeric variables relate?Scatter plotReveals correlation, clusters, outliers
How does a numeric variable differ across groups?Box plot / ViolinShows median, spread, and outliers per group
Which features are correlated?HeatmapMatrix view of all pairwise correlations
How does something change over time or epochs?Line plotEmphasizes trends and changes in order
How do categories compare in magnitude?Bar chartPosition on a common axis; humans compare lengths well
What do all pairwise relationships look like?Pair plotGrid of scatter plots and histograms for every combination

That is seven chart types. They cover roughly 95% of what you need in ML workflow. Remaining 5% — radar charts, treemaps, Sankey diagrams — are niche enough that you will know when you need them, and not worth memorizing early.

Limitation: this framework only helps pick chart type. It says nothing about whether chart is honest. Histogram with 5 bins tells very different story than one with 50 bins, and both are "correct." We face that later.

Seaborn — Statistical Visualization Without the Suffering

You may ask: if matplotlib can do everything, why does seaborn exist? Because matplotlib is like workshop full of power tools — you can build anything, but simple bookshelf takes fifty operations. Seaborn is bookshelf kit. It knows what you probably want, and gives it in one function call.

Seaborn is built directly on matplotlib. Every seaborn plot is matplotlib Figure with matplotlib Axes and matplotlib Artists. Seaborn does not replace workshop — it provides pre-assembled components you can still customize with raw tools when needed.

Seaborn adds three things. First, it understands pandas DataFrames natively — you pass column names as strings instead of extracting arrays. Second, it computes statistics on fly — confidence intervals on bar plots, kernel density estimates, regression lines. Third, defaults look good. Not "spend hour tweaking" good, but "presentable in meeting" good.

Back to house price data with seaborn. Most useful exploration tool is pair plot:

import seaborn as sns

sns.pairplot(df, hue="bedrooms", height=2.5)
plt.close("all")

One line. That produces grid of scatter plots for every pair of features, with histograms on diagonal, colored by bedroom count. Fastest way to get holistic view of dataset. I use it on first day of every project.

hue parameter is key insight — it maps categorical variable to color, turning regular scatter plot into one that reveals class structure. In our case, if 4-bedroom houses cluster differently from 2-bedroom houses in sqft-price space, pair plot makes that visible immediately.

Seaborn also has concept borrowed from grammar of graphics: FacetGrid. Instead of cramming everything into one plot with color coding, FacetGrid creates grid of separate panels — one per category. Each panel shows same plot type for different subset of data.

g = sns.FacetGrid(df, col="bedrooms", col_wrap=3, height=4)
g.map_dataframe(sns.scatterplot, x="sqft", y="price")
g.set_axis_labels("Square Footage", "Sale Price ($)")
plt.close("all")

This gives one scatter plot per bedroom count, side by side. Patterns within each category stay visible without clutter from overlapping colors. I still look up FacetGrid API about once per month — syntax for map_dataframe versus map trips me every time. Fine. API is consistent enough that documentation gets you unstuck fast.

Under hood, every seaborn function returns matplotlib Axes object (or FacetGrid that contains them). So when seaborn does not give exactly what you want, you can reach through to matplotlib layer:

fig, ax = plt.subplots(figsize=(8, 5))
sns.histplot(df, x="price", kde=True, ax=ax)
ax.axvline(df["price"].median(), color="red", linestyle="--",
           label=f"Median: ${df['price'].median():,.0f}")
ax.legend()
plt.close(fig)

Seaborn draws histogram and KDE curve. Then we reach into matplotlib Axes to add vertical median line — something seaborn does not do alone. Two libraries compose naturally because they share same underlying objects. Canvas-and-frame model still holds: seaborn paints picture, and we add brushstroke to same frame.

Limitation of seaborn is performance. pairplot on DataFrame with 100,000 rows and 20 features takes minutes and produces nearly unreadable grid. For large datasets, sample first or switch to aggregated views. Seaborn optimizes for clarity on moderate data, not scalability.

Short version: use viridis for colormaps, Plotly when someone else needs to explore data, and plt.close(fig) in every loop. That gets you 80% of way there.

The Color Problem Nobody Talks About

When you map numbers to colors, equal data steps must look like equal color steps. If going from 0.1 to 0.2 looks tiny but going from 0.7 to 0.8 looks dramatic, your eyes will see patterns that are not in data.

Old default colormap in matplotlib was jet — rainbow spectrum from blue through green and yellow to red. Jet is perceptually non-uniform. Some bands make color change look faster (around yellow-green); others make it stall (around cyan). Think ruler with unevenly spaced markings. You can still measure things, but measurements get biased toward noticing some ranges and missing others.

In 2015, matplotlib switched default to viridis — colormap designed so equal numerical steps produce equal perceptual steps in color. Change was not cosmetic. It corrected systematic source of visual error.

Viridis also solves second problem: colorblind accessibility. About 8% of men and 0.5% of women have some form of color vision deficiency, most often red-green. Jet fails these users — large parts of rainbow merge into muddy bands. Viridis, plus plasma, inferno, magma, and cividis, was designed to stay distinguishable under every common form of color vision deficiency.

Practical rule: for sequential data (values going low to high), use viridis or siblings. For diverging data (values going both directions from center, like correlation coefficients from -1 to +1), use coolwarm or RdBu centered at zero. For categorical data, use distinct hues — seaborn's default palette handles this well.

fig, axes = plt.subplots(1, 2, figsize=(14, 5))

# Sequential: viridis (perceptually uniform)
sns.heatmap(df.corr().abs(), cmap="viridis", annot=True,
            fmt=".2f", ax=axes[0])
axes[0].set_title("Absolute Correlations (viridis)")

# Diverging: coolwarm centered at zero
sns.heatmap(df.corr(), cmap="coolwarm", center=0, annot=True,
            fmt=".2f", ax=axes[1])
axes[1].set_title("Correlations (coolwarm, centered)")

plt.tight_layout()
plt.close(fig)

Back to ruler analogy: viridis is ruler with evenly spaced markings. Jet is ruler made by someone in hurry eyeballing spacing. Both "work," but one gives trustworthy measurements.

Interactive Visualization — Plotly and When You Actually Need It

Everything we made so far is static. Image. PNG. It cannot be interrogated. If someone sees scatter plot and asks "which specific house is outlier at $1.2M?", they cannot click to find out. For your own EDA in notebook, this is fine — you made plot and can rerun with filter. But once audience is product manager, client, or cross-functional team, static plots hit wall.

Plotly renders charts as interactive HTML. Hover over point to see values. Zoom into region. Click legend items to toggle series on and off. Pan, select, export. Chart lives in browser, so it can be embedded in notebooks, dashboards, or standalone HTML files you email to someone without Python installed.

import plotly.express as px

fig = px.scatter(df, x="sqft", y="price", color="bedrooms",
                 hover_data=["year_built", "distance_center"],
                 title="House Prices: Interactive Explorer")
fig.write_html("house_prices_interactive.html")

hover_data parameter is key. It attaches extra columns to each point's tooltip. Now product manager can hover over outlier and see "Oh, it is 1920s home 0.2 miles from downtown — historical property, not data error." That one interaction replaces back-and-forth email thread.

Plotly Express (plotly.express) is high-level API — it mirrors seaborn's one-function-one-plot philosophy. For full control, there is plotly.graph_objects, Plotly's equivalent of matplotlib's OO interface. For full web applications with dropdowns, sliders, and dynamic data loading, there is Dash — framework built on Plotly that turns Python into dashboard-building language.

When should you use Plotly instead of matplotlib? When audience is not you. When person looking at chart needs to ask their own questions about data, not only see answers you pre-selected. When chart goes into web page, Slack thread, or client report. For your own EDA and publication figures, matplotlib and seaborn remain right tools.

Quick note on Altair, another option worth knowing. Altair takes declarative approach — you describe what you want to see ("map sqft to x, price to y, color by bedrooms") and it figures out rendering. It is built on Vega-Lite specification, so charts are defined as JSON and can be rendered anywhere that supports Vega. Mental model differs from matplotlib's imperative "draw this line, then this label" approach. If grammar-of-graphics style clicks for you, Altair is worth exploring. I still develop intuition for when to pick it over Plotly.

Production Patterns — Viz That Doesn't Break at 3am

Visualization in notebook is one thing. Visualization in production pipeline — generating figures automatically, logging them to experiment trackers, running on server with no display — is different discipline.

First production lesson, learned hard way: memory leaks. Each matplotlib Figure object allocates memory. In notebook, plt.show() renders and garbage collector can clean up. In script or training loop, if you create figures without closing them, each one stays around. Generate thousand figures — one per training epoch, say — and process bloats until crash. Fix: always close what you open.

for epoch in range(1000):
    fig, ax = plt.subplots()
    try:
        ax.plot(train_losses[:epoch+1])
        ax.set_title(f"Training Loss — Epoch {epoch}")
        fig.savefig(f"plots/loss_epoch_{epoch:04d}.png",
                    dpi=100, bbox_inches="tight")
    finally:
        plt.close(fig)

try/finally ensures figure closes even if plotting code throws error. bbox_inches="tight" prevents axis labels from getting clipped at edges — universal problem that should be default, but is not. PDF and SVG for papers (vector graphics, infinite zoom). PNG for dashboards and Slack (rasterized, universally supported).

On headless servers — CI/CD pipeline, cloud training job, Docker container — there is no display for matplotlib to draw to. You need Agg backend, which renders to in-memory buffer without screen:

import matplotlib
matplotlib.use("Agg")  # must be before importing pyplot
import matplotlib.pyplot as plt

Second production lesson: log your figures to experiment trackers. If you use Weights & Biases, MLflow, or TensorBoard, do not take screenshots. Log programmatically.

# Weights & Biases
import wandb
wandb.log({"confusion_matrix": wandb.Image(fig)})

# MLflow
import mlflow
mlflow.log_figure(fig, "confusion_matrix.png")

# TensorBoard (PyTorch)
from torch.utils.tensorboard import SummaryWriter
writer = SummaryWriter()
writer.add_figure("confusion_matrix", fig, global_step=epoch)

Six months later, someone asks "can you regenerate that chart from experiment we ran in March?" If it is logged, you point them to run. If it is screenshot in Slack thread, you spend day recreating it from memory. Manual screenshots do not survive first team rotation.

Third production lesson: reproducibility. Any plot involving randomness — t-SNE, UMAP, sampled data, shuffled train/val splits — needs random seed set before operation. Code that produced each figure should be versioned alongside figure itself. Plot is output. Code is recipe. You need both.

Visualization as Communication (and Deception)

Edward Tufte, godfather of data visualization theory, introduced data-ink ratio: proportion of ink on page representing actual data vs. ink spent on decoration. His argument: maximize data-ink and remove everything else — heavy gridlines, 3D effects, ornate borders, background images. He called decorative excess chartjunk.

Tufte's principle matters now more than ever, because default output from most visualization tools includes fair amount of chartjunk. Matplotlib's default grid, for example, is heavier than needed. Seaborn's sns.set_theme() improves this a lot — rare case where one function call genuinely makes output more honest.

But dishonest charts are not always about decoration. More dangerous forms are structural. Truncated y-axis — starting at 95% instead of 0% — makes 2% improvement look like 40% jump. Cherry-picked time ranges can turn flat trend into hockey stick. Dual y-axes make unrelated metrics look correlated because scales can be adjusted independently to create any visual relationship you want. I have seen all of these in ML papers and production dashboards.

Truncated axis problem is especially common in ML. When reporting accuracy improvement from 96.2% to 97.1%, starting y-axis at 0% makes improvement invisible. Starting it at 95% makes it dramatic. Neither is absolutely "wrong," but second is misleading if audience misses axis. Honest approach: show full axis AND zoomed inset, or explicitly annotate scale.

For house price model, imagine plotting predicted vs. actual prices. If we show only range where model performs well and crop out tail where it falls apart, chart looks great. Senior interviewer or senior engineer will ask: "What happens outside this range?" If you do not have answer — or worse, cropped it deliberately — conversation ends.

Rule I try to follow: every chart should be hard to misread even by someone glancing for five seconds. If understanding chart requires reading fine print about axis ranges, log scales, or cherry-picked subsets, it failed as communication tool.
