Object-Oriented Programming

Chapter 1: Python & Programming Foundations 6 subtopics

TL;DR

Object-Oriented Programming (OOP) lets you bundle data and the functions that operate on that data into a single unit called a class. Python's OOP model gives you encapsulation, inheritance, polymorphism, and a rich set of special "dunder" methods that make your objects behave like built-in types. If you plan to work with frameworks like PyTorch, scikit-learn, or Django, OOP is not optional — their entire API surface is class-based.

Why OOP?

Picture building a machine learning library. You have models, datasets, optimizers, loss functions. Each has data (weights, learning rate, rows) and behavior (train, predict, shuffle). Without OOP you end up with a tangled web of functions passing dictionaries around, hoping someone remembers which key holds the learning rate.

OOP lets you create a Model class that owns its weights and knows how to train itself. A Dataset class that holds rows and knows how to iterate them. Everything is organized, discoverable, and harder to break.

Classes and Objects

Defining Classes and Creating Instances

A class is a blueprint. An object (or instance) is a concrete thing built from that blueprint. You define a class with the class keyword:

class Dog:
    # Class attribute — shared by ALL instances
    species = "Canis familiaris"

    def __init__(self, name, age):
        # Instance attributes — unique to each object
        self.name = name
        self.age = age

    def bark(self):
        return f"{self.name} says Woof!"

# Create instances (objects)
rex = Dog("Rex", 5)
luna = Dog("Luna", 3)

print(rex.bark())       # Rex says Woof!
print(luna.age)         # 3
print(rex.species)      # Canis familiaris
print(Dog.species)      # Canis familiaris — same thing

The __init__ method is the constructor — it runs automatically when you create a new instance. The self parameter is a reference to the instance being created. Python passes it implicitly; you never write rex.bark(rex).

Class attributes live on the class itself and are shared across all instances. Instance attributes are set on self and belong to one specific object.

# Class vs Instance attributes
rex.species = "Canis lupus"  # Creates an instance attribute, shadows the class one
print(rex.species)   # Canis lupus (instance)
print(luna.species)  # Canis familiaris (still class attribute)
print(Dog.species)   # Canis familiaris (class attribute unchanged)

Encapsulation

Encapsulation means hiding internal details and exposing a clean interface. Python uses naming conventions rather than strict access modifiers:

class BankAccount:
    def __init__(self, owner, balance=0):
        self.owner = owner          # Public
        self._currency = "USD"      # Protected (convention: "don't touch")
        self.__balance = balance    # Private (name-mangled)

    @property
    def balance(self):
        """Read-only access to balance."""
        return self.__balance

    @balance.setter
    def balance(self, amount):
        if amount < 0:
            raise ValueError("Balance cannot be negative")
        self.__balance = amount

    def deposit(self, amount):
        if amount <= 0:
            raise ValueError("Deposit must be positive")
        self.__balance += amount
        return self.__balance

    def withdraw(self, amount):
        if amount > self.__balance:
            raise ValueError("Insufficient funds")
        self.__balance -= amount
        return self.__balance

acct = BankAccount("Alice", 1000)
print(acct.balance)       # 1000 (uses @property getter)
acct.deposit(500)
print(acct.balance)       # 1500

# acct.__balance          # AttributeError!
# But Python just mangles the name:
print(acct._BankAccount__balance)  # 1500 — not truly private

The @property decorator turns a method into something that looks like an attribute access. The setter lets you add validation. This is Python's version of getters and setters — no boilerplate, but you keep control.

Key Insight: Python's philosophy is "we're all consenting adults." The underscore conventions (_protected, __private) signal intent, but nothing truly prevents access. Name mangling (__attr becomes _ClassName__attr) makes accidental collisions less likely, especially in inheritance chains.

Inheritance

Single Inheritance and super()

Inheritance lets a child class reuse and extend a parent class. The super() function gives you access to the parent's methods:

class Animal:
    def __init__(self, name, sound):
        self.name = name
        self.sound = sound

    def speak(self):
        return f"{self.name} says {self.sound}!"

class Dog(Animal):
    def __init__(self, name, breed):
        super().__init__(name, sound="Woof")
        self.breed = breed

    def fetch(self, item):
        return f"{self.name} fetches the {item}"

class Cat(Animal):
    def __init__(self, name):
        super().__init__(name, sound="Meow")

    def speak(self):
        # Override parent's method
        return f"{self.name} purrs softly..."

dog = Dog("Rex", "Labrador")
cat = Cat("Whiskers")

print(dog.speak())   # Rex says Woof!
print(cat.speak())   # Whiskers purrs softly...
print(dog.fetch("ball"))  # Rex fetches the ball

Multiple Inheritance and MRO

Python supports multiple inheritance — a class can have more than one parent. The Method Resolution Order (MRO) determines which parent's method gets called:

class Flyable:
    def move(self):
        return "Flying through the air"

class Swimmable:
    def move(self):
        return "Swimming through water"

class Duck(Flyable, Swimmable):
    pass

donald = Duck()
print(donald.move())  # Flying through the air (Flyable is first)

# Inspect the MRO
print(Duck.__mro__)
# (Duck, Flyable, Swimmable, object)

Python uses the C3 linearization algorithm to build the MRO. The rule is roughly: the child comes first, then parents left-to-right, then grandparents. When in doubt, print ClassName.__mro__.

Abstract Base Classes

Sometimes you want to define an interface — "any subclass must implement these methods." The abc module handles this:

from abc import ABC, abstractmethod

class Model(ABC):
    @abstractmethod
    def fit(self, X, y):
        """Train the model."""
        pass

    @abstractmethod
    def predict(self, X):
        """Make predictions."""
        pass

    def summary(self):
        return f"{self.__class__.__name__} model"

# Cannot instantiate directly:
# model = Model()  # TypeError!

class LinearRegression(Model):
    def __init__(self):
        self.weights = None

    def fit(self, X, y):
        # Simplified: just store means
        self.weights = [sum(col)/len(col) for col in zip(*X)]
        return self

    def predict(self, X):
        return [sum(w * x for w, x in zip(self.weights, row)) for row in X]

lr = LinearRegression()
print(lr.summary())  # LinearRegression model
Key Insight: This is exactly how scikit-learn works. Every estimator inherits from BaseEstimator and implements fit() and predict(). When you write model.fit(X, y), you are using this pattern. Understanding ABCs helps you read framework source code and build your own plug-in components.

Polymorphism

Polymorphism means "many forms" — different classes can respond to the same method call in their own way. Python uses duck typing: if an object has a fit() method, you can call it. You do not need to declare an interface or inherit from a specific class.

class RandomForest:
    def fit(self, X, y):
        print("Training random forest...")
        return self
    def predict(self, X):
        return [0] * len(X)

class NeuralNetwork:
    def fit(self, X, y):
        print("Training neural network...")
        return self
    def predict(self, X):
        return [1] * len(X)

def train_and_evaluate(model, X_train, y_train, X_test):
    """Works with ANY object that has fit() and predict()."""
    model.fit(X_train, y_train)
    predictions = model.predict(X_test)
    return predictions

# Both work — polymorphism via duck typing
for cls in [RandomForest, NeuralNetwork]:
    model = cls()
    preds = train_and_evaluate(model, [[1],[2]], [0,1], [[3]])
    print(f"  {cls.__name__} predicted: {preds}")

Operator Overloading

Python lets you define how operators (+, ==, <, etc.) behave for your classes using special dunder methods:

class Vector:
    def __init__(self, x, y):
        self.x = x
        self.y = y

    def __add__(self, other):
        return Vector(self.x + other.x, self.y + other.y)

    def __mul__(self, scalar):
        return Vector(self.x * scalar, self.y * scalar)

    def __eq__(self, other):
        return self.x == other.x and self.y == other.y

    def __repr__(self):
        return f"Vector({self.x}, {self.y})"

    def __len__(self):
        return 2  # 2D vector

v1 = Vector(1, 2)
v2 = Vector(3, 4)
print(v1 + v2)       # Vector(4, 6)
print(v1 * 3)        # Vector(3, 6)
print(v1 == v2)      # False
print(len(v1))       # 2

Special (Dunder) Methods

Dunder methods (double-underscore methods like __init__, __str__) are Python's protocol hooks. Implement them and your object integrates seamlessly with built-in functions and syntax.

class Dataset:
    def __init__(self, data):
        self._data = list(data)

    def __len__(self):
        return len(self._data)

    def __getitem__(self, index):
        return self._data[index]

    def __setitem__(self, index, value):
        self._data[index] = value

    def __str__(self):
        return f"Dataset({len(self)} items)"

    def __repr__(self):
        return f"Dataset({self._data!r})"

    def __iter__(self):
        return iter(self._data)

    def __contains__(self, item):
        return item in self._data

ds = Dataset([10, 20, 30, 40, 50])
print(len(ds))        # 5
print(ds[2])          # 30
print(ds[-1])         # 50
ds[0] = 99
print(99 in ds)       # True
print(str(ds))        # Dataset(5 items)

# Works with for loops because we defined __iter__
for item in ds:
    print(item, end=" ")  # 99 20 30 40 50

Context Managers: __enter__ and __exit__

class Timer:
    """Context manager that times a block of code."""
    import time

    def __enter__(self):
        self.start = self.time.time()
        return self

    def __exit__(self, exc_type, exc_val, exc_tb):
        self.elapsed = self.time.time() - self.start
        print(f"Elapsed: {self.elapsed:.4f}s")
        return False  # Don't suppress exceptions

import time
with Timer():
    time.sleep(0.5)  # Elapsed: 0.50xxs

Making Objects Callable with __call__

class Scaler:
    """Scale values by a fixed factor. Callable like a function."""
    def __init__(self, factor):
        self.factor = factor

    def __call__(self, value):
        return value * self.factor

double = Scaler(2)
triple = Scaler(3)

print(double(5))   # 10
print(triple(5))   # 15

# This pattern is everywhere in PyTorch:
# loss_fn = nn.CrossEntropyLoss()
# loss = loss_fn(predictions, targets)  # __call__
Key Insight: In PyTorch, every layer and loss function is a callable class. When you write model(input), you are invoking __call__, which internally calls forward(). Understanding __call__ is essential for reading deep learning code.

Advanced OOP

Class Methods and Static Methods

class Config:
    _instances = {}

    def __init__(self, env, debug=False):
        self.env = env
        self.debug = debug

    @classmethod
    def from_dict(cls, d):
        """Alternative constructor from a dictionary."""
        return cls(env=d["env"], debug=d.get("debug", False))

    @classmethod
    def from_env(cls, env_name):
        """Factory that caches instances (Singleton-like)."""
        if env_name not in cls._instances:
            cls._instances[env_name] = cls(env=env_name)
        return cls._instances[env_name]

    @staticmethod
    def is_valid_env(name):
        """Utility — no access to cls or self needed."""
        return name in ("dev", "staging", "production")

# Usage
cfg = Config.from_dict({"env": "production", "debug": False})
print(Config.is_valid_env("production"))  # True

# Singleton behavior
c1 = Config.from_env("dev")
c2 = Config.from_env("dev")
print(c1 is c2)  # True — same instance

A @classmethod receives the class itself (cls) as the first argument. Use it for alternative constructors. A @staticmethod receives neither self nor cls — it is a plain function that lives in the class namespace for organizational reasons.

Dataclasses

Python 3.7 introduced dataclasses to eliminate boilerplate for classes that primarily hold data:

from dataclasses import dataclass, field

@dataclass
class Experiment:
    name: str
    learning_rate: float = 0.001
    epochs: int = 10
    metrics: list = field(default_factory=list)

    @property
    def description(self):
        return f"{self.name} (lr={self.learning_rate}, epochs={self.epochs})"

exp1 = Experiment("baseline", learning_rate=0.01)
exp2 = Experiment("baseline", learning_rate=0.01)

print(exp1)            # Experiment(name='baseline', ...)
print(exp1 == exp2)    # True — auto-generated __eq__
print(exp1.description)  # baseline (lr=0.01, epochs=10)

The @dataclass decorator auto-generates __init__, __repr__, and __eq__ based on your type annotations. Use field(default_factory=list) for mutable defaults.

Composition vs Inheritance

# Inheritance: "is-a" relationship
class Animal:
    def breathe(self):
        return "breathing"

class Dog(Animal):
    pass  # Dog IS an Animal

# Composition: "has-a" relationship
class Engine:
    def start(self):
        return "vroom"

class Car:
    def __init__(self):
        self.engine = Engine()  # Car HAS an Engine

    def start(self):
        return self.engine.start()
Watch Out: Favor composition over inheritance. Deep inheritance trees become fragile — a change in the base class can break everything downstream. Compose small, focused objects instead. Most design patterns (Strategy, Observer, Decorator) use composition.

Mixins

A mixin is a small class designed to be combined with other classes via multiple inheritance. It adds a single capability:

import json

class JsonMixin:
    """Add JSON serialization to any class."""
    def to_json(self):
        return json.dumps(self.__dict__, indent=2)

    @classmethod
    def from_json(cls, json_str):
        data = json.loads(json_str)
        return cls(**data)

class LogMixin:
    """Add logging to any class."""
    def log(self, message):
        print(f"[{self.__class__.__name__}] {message}")

@dataclass
class User(JsonMixin, LogMixin):
    name: str
    email: str

user = User("Alice", "alice@example.com")
user.log("Created")  # [User] Created
print(user.to_json())
# {"name": "Alice", "email": "alice@example.com"}

Metaclasses

A metaclass is a class whose instances are themselves classes. In Python, the default metaclass is type. You rarely need custom metaclasses, but understanding them demystifies how Python classes work under the hood:

# type is the metaclass of all classes
print(type(int))      # <class 'type'>
print(type(str))      # <class 'type'>
print(type(type))     # <class 'type'> — it's turtles all the way

# Creating a class dynamically with type()
MyClass = type("MyClass", (object,), {"x": 10, "greet": lambda self: "hi"})
obj = MyClass()
print(obj.x, obj.greet())  # 10 hi

# Custom metaclass: auto-register all subclasses
class PluginMeta(type):
    registry = {}
    def __init_subclass__(cls, **kwargs):
        super().__init_subclass__(**kwargs)

class Plugin(metaclass=PluginMeta):
    pass

# Simpler approach with __init_subclass__
class BasePlugin:
    _plugins = []

    def __init_subclass__(cls, **kwargs):
        super().__init_subclass__(**kwargs)
        BasePlugin._plugins.append(cls)

class CSVLoader(BasePlugin):
    pass

class JSONLoader(BasePlugin):
    pass

print(BasePlugin._plugins)  # [CSVLoader, JSONLoader]

The Descriptor Protocol

A descriptor is any object that defines __get__, __set__, or __delete__. Descriptors are how @property, @classmethod, and @staticmethod work under the hood:

class Validated:
    """Descriptor that enforces a type and range."""
    def __init__(self, min_val=None, max_val=None):
        self.min_val = min_val
        self.max_val = max_val

    def __set_name__(self, owner, name):
        self.name = name

    def __get__(self, obj, objtype=None):
        if obj is None:
            return self
        return getattr(obj, f"_{self.name}", None)

    def __set__(self, obj, value):
        if not isinstance(value, (int, float)):
            raise TypeError(f"{self.name} must be numeric")
        if self.min_val is not None and value < self.min_val:
            raise ValueError(f"{self.name} must be >= {self.min_val}")
        if self.max_val is not None and value > self.max_val:
            raise ValueError(f"{self.name} must be <= {self.max_val}")
        setattr(obj, f"_{self.name}", value)

class HyperParams:
    learning_rate = Validated(min_val=0, max_val=1)
    epochs = Validated(min_val=1, max_val=10000)

hp = HyperParams()
hp.learning_rate = 0.01  # OK
hp.epochs = 100          # OK
# hp.learning_rate = -1  # ValueError: learning_rate must be >= 0
# hp.epochs = "ten"      # TypeError: epochs must be numeric

__slots__

By default, Python stores instance attributes in a dictionary (__dict__), which is flexible but uses extra memory. __slots__ tells Python to allocate a fixed set of attribute slots, saving ~40% memory per instance:

class PointRegular:
    def __init__(self, x, y):
        self.x = x
        self.y = y

class PointSlotted:
    __slots__ = ("x", "y")
    def __init__(self, x, y):
        self.x = x
        self.y = y

import sys
r = PointRegular(1, 2)
s = PointSlotted(1, 2)
print(sys.getsizeof(r.__dict__))  # ~104 bytes for the dict
# s.__dict__  # AttributeError — no dict!
# s.z = 3    # AttributeError — can't add new attributes
Key Insight: Use __slots__ when you will create millions of instances (like tokens in an NLP pipeline, or nodes in a graph). The memory savings add up. But be aware: slotted classes cannot have arbitrary attributes, and multiple inheritance with slots requires care.

Design Patterns

# Singleton — ensure only one instance
class Singleton:
    _instance = None

    def __new__(cls, *args, **kwargs):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
        return cls._instance

a = Singleton()
b = Singleton()
print(a is b)  # True

# Factory — create objects without specifying exact class
class ModelFactory:
    _models = {}

    @classmethod
    def register(cls, name, model_cls):
        cls._models[name] = model_cls

    @classmethod
    def create(cls, name, **kwargs):
        if name not in cls._models:
            raise ValueError(f"Unknown model: {name}")
        return cls._models[name](**kwargs)

# Register models
ModelFactory.register("linear", LinearRegression)
# model = ModelFactory.create("linear")

# Observer — notify subscribers of events
class EventEmitter:
    def __init__(self):
        self._listeners = {}

    def on(self, event, callback):
        self._listeners.setdefault(event, []).append(callback)

    def emit(self, event, *args):
        for cb in self._listeners.get(event, []):
            cb(*args)

emitter = EventEmitter()
emitter.on("epoch_end", lambda e: print(f"Epoch {e} done"))
emitter.emit("epoch_end", 5)  # Epoch 5 done
