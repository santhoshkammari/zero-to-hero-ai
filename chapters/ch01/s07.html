Pandas

Chapter 1: Python & Programming Foundations 4 code blocks · 1 table · 1 checklist
TL;DR

Pandas is where 80% of your ML time actually goes — not modeling, but cleaning, transforming, and exploring. The leverage comes from three things: method chaining (write readable pipelines, not 15 temp variables), groupby (the pattern behind every per-group feature and cohort analysis), and knowing the merge gotchas that silently corrupt your data. Master those, and you cut hours off every project.

Here's the uncomfortable truth about ML work: the difference between an hour and a day on a data task usually isn't algorithmic cleverness. It's knowing the right Pandas pattern. The person who knows transform builds a per-group feature in one line. The person who doesn't writes a loop, gets it wrong twice, and spends an hour debugging index alignment. This section is about becoming the first person.

DataFrame Basics — Quick Pass

A DataFrame is a table. A Series is a column. You know this. What matters is the muscle memory for any new dataset:

import pandas as pd

df = pd.read_csv("data.csv", usecols=["id", "age", "label"], dtype={"label": "category"})
df.head()          # eyeball the data
df.info()          # dtypes, non-null counts, memory
df.describe()      # stats for numerics; add include="all" for categoricals too
df.isnull().sum()  # missing values per column — always check

Those four commands are how you start every dataset. No exceptions.

The one distinction that trips people up: .loc[] is label-based (inclusive slices), .iloc[] is position-based (exclusive end, like normal Python). df.loc[0:3] gives you 4 rows. df.iloc[0:3] gives you 3. Burn that in and move on — use .loc as your default, .query() for complex filters, and you'll avoid 90% of selection bugs.

Method Chaining — The Superpower

Here's the thing about method chaining — once you see code written this way, the old style with 15 intermediate DataFrames looks painful. This is how experienced Pandas users write code:

result = (
    df
    .query("age > 18")
    .assign(bmi=lambda d: d.weight / (d.height / 100) ** 2)
    .groupby("category")
    .agg(mean_bmi=("bmi", "mean"), count=("bmi", "size"))
    .sort_values("mean_bmi", ascending=False)
)

Compare that to the alternative:

# The "intermediate variable" style — works, but painful to read and maintain
df2 = df[df["age"] > 18]
df2 = df2.copy()
df2["bmi"] = df2["weight"] / (df2["height"] / 100) ** 2
grouped = df2.groupby("category")
result = grouped.agg(mean_bmi=("bmi", "mean"), count=("bmi", "size"))
result = result.sort_values("mean_bmi", ascending=False)

Same result. The chained version is half the lines, reads top-to-bottom, and you can add or remove a step without touching anything else. Three methods make chaining work:

.assign() creates new columns without breaking the chain. Each keyword becomes a column. Use lambda d: to reference the DataFrame as it exists at that point in the chain — not some earlier version sitting in a variable somewhere:

df = (
    df
    .assign(
        name_clean=lambda d: d["name"].str.strip().str.lower(),
        los_days=lambda d: d["los_days"].clip(lower=0),
        admit_year=lambda d: d["admit_date"].dt.year,
        # can reference columns created in the same assign (Pandas 0.23.2+)
        is_recent=lambda d: d["admit_date"].dt.year >= 2023,
    )
)

.query() filters rows with a string expression. More readable than boolean indexing for complex conditions, and it avoids the &/|/parentheses dance:

# Boolean indexing — works, but gets ugly fast
df[(df["age"] > 30) & (df["city"] == "NYC") & (~df["status"].isin(["inactive"]))]

# query — same thing, readable
df.query("age > 30 and city == 'NYC' and status != 'inactive'")

.pipe() is for custom functions in the chain. Each function takes a DataFrame, returns a DataFrame. This is how production pipelines get written — each step is a named, testable function:

def remove_outliers(df, col, n_std=3):
    mean, std = df[col].mean(), df[col].std()
    return df.query(f"{col} > {mean - n_std * std} and {col} < {mean + n_std * std}")

def add_time_features(df, date_col="admit_date"):
    return df.assign(
        month=lambda d: d[date_col].dt.month,
        dayofweek=lambda d: d[date_col].dt.dayofweek,
        is_weekend=lambda d: d[date_col].dt.dayofweek.ge(5).astype(int),
    )

# Compose — reads like a recipe
features = (
    df
    .pipe(remove_outliers, "cost")
    .pipe(add_time_features)
    .drop(columns=["admit_date", "patient_id"])
)
ℹ️ Why pipe() over just calling functions?

You could write add_time_features(remove_outliers(df, "cost")) — but that reads inside-out and gets unreadable past two functions. pipe() keeps the top-to-bottom flow, makes each step independently testable, and is trivial to reorder or disable during experimentation.

GroupBy — Where The Real Work Happens

df.groupby("col").agg(...) is the pattern you'll use 10 times a day. Every "per-category average," every "normalize within each group," every cohort analysis flows through groupby. The mechanics are simple — split rows into groups, do something to each group, combine the results. What matters is knowing which "something" to reach for.

Named Aggregation — The Clean Way

Named aggregation is the modern syntax. Each keyword becomes a column name, and the tuple tells Pandas which source column and which function:

summary = (
    df.groupby("department")
    .agg(
        avg_salary=("salary", "mean"),
        headcount=("employee_id", "nunique"),
        max_tenure=("years", "max"),
        p95_salary=("salary", lambda x: x.quantile(0.95)),
    )
    .sort_values("avg_salary", ascending=False)
)

You can also pass a list for quick exploration — .agg(["mean", "std", "count"]) — but the named style is what you want in production code because the output columns have meaningful names instead of a MultiIndex.

transform vs agg — The KEY Distinction

This is the concept that separates "I know Pandas" from "I'm productive in Pandas." Both operate on groups, but they return fundamentally different shapes:

agg reduces each group to one row — you get a summary table. transform returns a value for every row in the original DataFrame, broadcasting the group result back. Same number of rows in, same number out.

# agg: one row per department → summary table
dept_stats = df.groupby("department")["salary"].agg(["mean", "std"])

# transform: group mean broadcast back to every row → feature engineering
df["salary_vs_dept"] = (
    df.groupby("department")["salary"]
    .transform(lambda x: (x - x.mean()) / x.std())
)
# every employee now has a z-score relative to their own department

transform is the workhorse for feature engineering in ML. Anytime you need "this row's value relative to its group" — group-normalized features, within-group rankings, group-level fill values — that's transform:

# fill missing values with the group median (not the global median)
df["age"] = df.groupby("category")["age"].transform(lambda x: x.fillna(x.median()))

# rank within each group
df["salary_rank"] = df.groupby("department")["salary"].transform("rank", ascending=False)

apply — Last Resort

apply is the escape hatch when agg and transform can't express what you need. It's also slow — it runs a Python function on each group, which means no vectorization. Before reaching for apply, ask: can I do this with agg, transform, or a vectorized operation? Almost always, yes.

# filter: keep only groups with enough data (no apply needed)
df_filtered = df.groupby("category").filter(lambda g: len(g) >= 30)

# apply: when you genuinely need custom logic per group
# (e.g., top-3 rows per group by some column)
top3 = df.groupby("category").apply(lambda g: g.nlargest(3, "score")).reset_index(drop=True)

Merge & Join — The Gotchas

Three tools, three jobs: merge for SQL-like joins on columns, join for index-based joins (rarely needed), concat for stacking rows or columns. In practice you'll use merge and concat almost exclusively.

Merge TypeKeepsSQL EquivalentWhen
innerOnly matching keys in bothINNER JOINDefault — safest starting point
leftAll left rows, matched rightLEFT JOINEnrich a main table with lookup data
outerAll rows from bothFULL OUTER JOINWhen you need the complete picture (expect NaNs)

(right join exists but just swap the table order and use left — less confusing.)

For concat: pd.concat([df1, df2]) stacks rows (like SQL UNION ALL). Always pass ignore_index=True or you'll get duplicate indexes. pd.concat([df1, df2], axis=1) stacks columns side-by-side — but it aligns on the index, so mismatched indexes produce NaN.

🚨 The Row Explosion Problem

This is the merge gotcha that silently corrupts data. If the left table has 3 rows with key "A" and the right table has 4 rows with key "A", you get 12 rows — a many-to-many join. Your DataFrame just quadrupled in size and you might not notice until the model trains on garbage. Always defend against this:

# validate= catches row explosion at merge time
merged = pd.merge(orders, customers, on="customer_id", validate="many_to_one")

# indicator=True adds a column showing which rows matched
merged = pd.merge(left, right, on="key", how="outer", indicator=True)
print(merged["_merge"].value_counts())
# both          850
# left_only      23   ← these didn't match — investigate
# right_only     12

Memory & Performance

Pandas is fast enough for most ML work — until it isn't. The bottlenecks are almost always the same few antipatterns.

Check memory first: df.memory_usage(deep=True).sum() / 1e6 gives you megabytes. If your 100K-row DataFrame is eating 500MB, you have a dtype problem, not a Pandas problem.

Categorical dtype is the single biggest memory win. A string column with 10 unique values across 1M rows stores 1M strings as object. As category, it stores 10 strings plus 1M tiny integer codes. That's 90%+ memory savings, and groupby/merge on categoricals is faster too:

df["status"] = df["status"].astype("category")       # after loading
# or at load time:
df = pd.read_csv("data.csv", dtype={"status": "category", "region": "category"})

Downcast numerics when you know the range: pd.to_numeric(df["age"], downcast="integer") picks int8 instead of int64 — 87% less memory per column.

Why apply() is slow — it runs a Python function on each row (or group), which means the GIL, no vectorization, and Python-level overhead per element. The fix is almost always a vectorized alternative:

# SLOW: apply with axis=1 (Python loop in disguise)
df["full_name"] = df.apply(lambda r: r["first"] + " " + r["last"], axis=1)

# FAST: vectorized string concatenation (100x faster)
df["full_name"] = df["first"] + " " + df["last"]

# SLOW: apply for string operations
df["clean"] = df["name"].apply(lambda x: x.strip().lower())

# FAST: .str accessor (5-50x faster)
df["clean"] = df["name"].str.strip().str.lower()

When you absolutely must iterate (rare), use df.itertuples() over df.iterrows() — it's 5-10x faster because it returns namedtuples instead of Series objects.

Missing Data

The trio: isna() to find it, fillna() to replace it, dropna() to remove it. But before you reach for any of these — understand why the data is missing. Is it random? Is it structural (e.g., "treatment_end_date" is null because treatment is ongoing)? The answer changes your strategy entirely. Not everything should be filled.

The gotcha that bites everyone once: NaN != NaN in Python. You cannot check for missing values with ==. Use df["col"].isna(), never df["col"] == np.nan.

🚨 fillna Leakage in ML

Never compute fill statistics on the full dataset. Compute on train, apply to both: median_val = X_train["age"].median(), then X_train["age"].fillna(median_val) and X_test["age"].fillna(median_val). Using X.fillna(X.median()) on the combined set leaks test information into training.

What You Should Now Be Able To Do

  • ✅ Write a method-chained pipeline using query, assign, groupby, agg, and sort_values — no intermediate variables
  • ✅ Use .pipe() to compose named functions into a testable, reorderable data pipeline
  • ✅ Explain the difference between agg (reduces rows) and transform (broadcasts back) — and use transform for per-group feature engineering
  • ✅ Merge tables with validate= to catch row explosions, and use indicator=True to debug unmatched rows
  • ✅ Cut DataFrame memory by 50-90% using category dtype and numeric downcasting
  • ✅ Replace apply(axis=1) with vectorized operations or .str/.dt accessors for 10-1000x speedups
  • ✅ Handle missing data correctly: check why it's missing, use isna() (not ==), and never compute fill stats on the test set
