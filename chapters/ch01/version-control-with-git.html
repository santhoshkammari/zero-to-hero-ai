Long time, I used Git like many people: memorize few spells. git add ., git commit -m "stuff", git push. Something broke—a merge conflict I did not understand, branch somehow diverged from reality—I stared at terminal, tried Stack Overflow fixes, then deleted folder and cloned again. Not proud. Probably not alone.

Thing that broke pattern: nagging discomfort. I did not know what was happening inside. I was operating machine I could not see, and that pushed me to look. What I found surprised me: Git's internals are simple. Not easy, but simple. Few elegant ideas build everything Git does.

Git came from Linus Torvalds in 2005, built in about two weeks after Linux kernel team lost free license for previous tool and needed replacement fast. It's distributed version control system—every clone contains full history, not pointer to central server. ML projects, open-source libraries, production systems: all live in Git.

We use SHA hashes and internal data structures in this section. No prior knowledge needed. Build every concept from scratch. Start with how Git stores things. End with commands you run every day.

Snapshots, Not Diffs

Common misconception. People think Git tracks changes, stores diffs, then replays patch history to rebuild file versions. CVS and Subversion worked that way. Git does something different.

Git takes snapshot of whole project at every commit. Not list of changes—full photo of every file at that moment.

Suppose tiny ML project has three files: train.py, config.yaml, and data/dataset.csv. First commit records all three. Edit only train.py and commit again, Git takes new snapshot—but for config.yaml and data/dataset.csv, which did not change, it does not store them again. It stores references to previous versions. Same content, same reference.

Mental model that unlocks everything: Git is photo album, not film strip. Each commit is complete photo of project at that moment. File that did not change gets pointer to old photo, not new one. Keep this model in mind. Everything else clicks.

Full snapshots raise obvious question: how does Git avoid eating your disk alive? Object model answers that.

The Object Model

Inside .git/objects/, Git stores everything as four object types: blobs, trees, commits, and tags. Understand these four. Understand Git.

Blobs

Run git add train.py. Git compresses file contents and stores blob object. Blob is pure content—it does not know own filename, directory, or project context. Only bytes.

Git hashes content with SHA-1. Cryptographic hash function. Any input in, consistent 40-character hexadecimal string out. Hash becomes blob's address in object store. You can verify this:

echo "print('hello')" | git hash-object --stdin

Run that command twice on any machine. Same hash. Change one character in input and hash changes completely. This property—same content always maps to same address—is content-addressable storage. Reason Git never stores same content twice: two files with identical content hash to same blob. One copy. Two names.

Big surprise for me: blob does not know own filename. That blew my mind. If blob has no name, something else must handle names. Tree does that.

Trees

Tree object is Git version of directory listing. It maps filenames and permissions to blob hashes, and subdirectories to other tree hashes. Run git cat-file -p <tree-hash>. You see something like this:

git cat-file -p HEAD^{tree}
100644 blob a8c3f2e1...  config.yaml
100644 blob 4d9f7a12...  train.py
040000 tree 7b2e1c8a...  data

Tree gives blobs names and place in directory hierarchy. Root tree is snapshot of whole filesystem at that commit.

Commits

Commit object ties everything together. It points to tree, stores author name, email, timestamp, commit message, and parent commit or commits. Look at one:

git cat-file -p HEAD
tree 7b2e1c8a4d3e9f1c2b5a8d7e6f0c1a3b4d5e6f7a
parent 3a1b2c4d5e6f7a8b9c0d1e2f3a4b5c6d7e8f9a0b
author Ada Lovelace <ada@example.com> 1710000000 +0000
committer Ada Lovelace <ada@example.com> 1710000000 +0000

Add balanced sampling to training loop

That is commit. Small text file with four fields. Content determines hash, and hash becomes commit identity. Change message by one character and hash changes—it becomes different commit. This immutability is fundamental to Git's integrity. Every commit forms cryptographic chain back to first one—tamper with middle commit and hash changes cascade forward, making damage visible.

Objects give us content-addressed snapshot store. But how do we navigate it—how do we know which commit is current, and what is branch, really?

The Three Trees

Git manages three distinct "trees" at same time. Word "tree" is loose metaphor here—not all are literal tree data structures—but name stuck. Understand them, and git status, git add, and git commit stop feeling magical.

First is working directory: actual files on disk, files you open in editor, files you run directly. Changes here stay untracked until you tell Git about them.

Second is staging area, also called index. Draft of next commit—a prepared snapshot between working directory and permanent record. Run git add train.py, and current state of train.py moves from working directory into staging area.

Third is repository: committed history stored in .git/, chain of commit objects we just saw.

Why does staging area exist? Because you often change five files while solving problem, but only three changes belong together as coherent unit. Staging area lets you choose exactly what goes into each commit. You are not forced to commit everything at once. Powerful feature. Annoying until day it saves you.

Walk through commands with this model. Edit train.py: working directory differs from staging area. Run git add train.py: staging area differs from last commit. Run git commit: repository matches staging area.

git diff           # working directory vs. staging area
git diff --staged  # staging area vs. last commit
git status         # shows the state of all three trees at once

If working directory is workshop, staging area is box where you pack items before shipping. You choose what goes in box. Commit ships box and seals it permanently. Once sealed, nothing changes—ever.

We know objects and three trees. But Git still needs way to track which commit is current and what branches actually are. That is refs.

Refs — Where Branches Actually Live

Branch in Git is text file. Whole secret. Open .git/refs/heads/main in any repository and you find one line: 40-character SHA-1 hash pointing to commit.

cat .git/refs/heads/main
# a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0

Make new commit on main, and Git writes new commit hash into that file. That is branching—a file update. Branches are incredibly cheap because they are only 41 bytes on disk.

HEAD is another reference, stored at .git/HEAD. Usually it contains symbolic reference to current branch:

cat .git/HEAD
# ref: refs/heads/main

Switch branches with git checkout feature or git switch feature. Git updates .git/HEAD to point to new branch. Then commit happens. Git follows HEAD, finds current branch, and updates that branch's ref file to point to new commit. Mechanism is simple.

Detached HEAD happens when HEAD points directly to commit hash instead of branch name. You are looking at specific snapshot but not on named branch. Commits you make become orphaned—not reachable from any branch—unless you create new branch before leaving. First time I saw "detached HEAD state" in terminal, panic. Now I run git switch -c new-branch-name and move on.

Tags work similarly. Lightweight tags are ref files in .git/refs/tags/—like branches, but they do not move when you commit. Annotated tags are full objects with their own hash, author, and message, pointed to by ref. Both are stable named bookmarks to specific commits—usually release versions.

Merge vs. Rebase — Two Ways to Combine Work

You have been on feature branch for two days. Main moved forward while you worked. Now work must combine with theirs. Git gives two fundamentally different ways to do this, and difference is worth understanding.

Fast-Forward Merge

If main has not diverged—your branch is ahead of main, but main has no new commits since branch point—Git can do fast-forward merge. Nothing to reconcile. Git moves main pointer forward to your latest commit. No new commit. History stays linear.

git switch main
git merge feature   # fast-forward if possible

Three-Way Merge

If both branches have new commits since they diverged, Git needs three-way merge. It finds common ancestor, compares both branches against it, and creates new merge commit with two parents. If same part of same file changed on both branches, Git cannot resolve automatically. That is merge conflict, and you resolve it manually.

Result is historically accurate but visually noisy. git log --graph shows true branch-and-merge topology, but busy projects get hard to read.

Rebase

Rebase is alternative. Instead of creating merge commit, rebase takes your feature-branch commits and replays them one by one on top of latest commit on main. Result looks like you started branch from main's current tip—linear history, no merge commit, no graph branching.

git switch feature
git rebase main     # replay feature commits on top of latest main

But catch exists. Replaying commits creates new commits—new parent, new hash—even if diff is identical. Old commits still exist in object store, but branch now points to new ones. If someone else pulled your feature branch and based work on old commits, they now have commits that no longer exist on any branch. Their history diverged from yours in painful way.

Golden rule: never rebase commits that other people have already pulled. Local branch you have not pushed? Rebase away. Shared branch? Do not. I rebased shared branch once while colleague was actively working on it. Team lost half day figuring out what happened and reconstructing work. Bad afternoon.

When Everything Goes Wrong

This section earns its keep at 2 AM. Not list of commands—a set of tools that make sense once you understand why they work. They work because of object model. Commits do not get deleted; they get unreferenced. As long as something points to commit—a branch, tag, or reflog—it is recoverable.

The Reflog — Git's Flight Recorder

Every time HEAD moves—commit, checkout, reset, rebase—Git records it in reflog. Run git reflog and you see last 90 days of HEAD movements, each with short hash and cause.

git reflog
a1b2c3d HEAD@{0}: commit: Add regularization
f4e5d6c HEAD@{1}: rebase: fast-forward
9g8h7i6 HEAD@{2}: checkout: moving from main to feature
...

Even after hard reset seems to destroy last three commits, those commits still sit in object store. Reflog shows their hashes. You can recover them with git checkout <hash> and then create new branch from that point. Nothing is truly gone until Git garbage-collects it, and that takes at least two weeks by default.

Bisect — Binary Search for Bugs

Bisect is one of Git's most underused superpowers. Code worked at commit A and is broken at commit Z, with 200 commits in between. Git bisect performs binary search: it checks out midpoint, you test it, tell Git good or bad, and Git cuts range in half. Repeat until exact commit that introduced bug appears. For 200 commits, that is at most 8 tests.

git bisect start
git bisect bad              # current commit is broken
git bisect good a1b2c3d     # this commit was known-good
# Git checks out the midpoint — test your code, then:
git bisect good             # or: git bisect bad
# Repeat until Git announces the first bad commit
git bisect reset            # return to where you started

For ML work, this is invaluable. Validation accuracy drops from 0.94 to 0.87 between last week and this week. Bisect finds exact commit that caused it—subtle data processing change, hyperparameter tweak, or normalization bug.

Stash — A Clipboard for Work in Progress

Stash temporarily shelves uncommitted changes. You have been editing train.py when teammate asks you to look at something on different branch. You are not ready to commit. git stash saves changes, cleans working directory to match HEAD, and lets you switch context. git stash pop brings everything back.

git stash           # save changes, clean working directory
git switch main     # do other work...
git switch feature
git stash pop       # restore your changes

You can accumulate multiple stashes (git stash list shows them all) and apply specific ones with git stash apply stash@{2}. I treat stash as stack of temporary saves, not long-term storage strategy—if you need something permanently, commit it to branch.

Reset — Three Flavors of Undo

Reset moves current branch pointer to different commit. It comes in three modes. Know which one to use, because two are recoverable and one is not.

git reset --soft HEAD~1    # undo last commit; changes stay staged
git reset --mixed HEAD~1   # undo last commit; changes unstaged (this is the default)
git reset --hard HEAD~1    # undo last commit; changes DESTROYED

Soft reset is most useful. Made commit with bad message? git reset --soft HEAD~1, fix message, recommit. Need to break big commit into smaller ones? Soft reset, re-add carefully. Mixed reset is default and fine for most cases. Hard reset needs caution—it discards working directory changes too. But even then: commit object still exists. Reflog still points to it. You can get it back within 90-day window.

Reason these rescue tools exist—the reason reflog can recover from hard reset, reason bisect can navigate hundreds of commits efficiently—is object model. Commits are immutable and content-addressed. They do not get deleted because branch stops pointing to them. Understanding internals is not academic. It changes how mistakes feel.

Git for Machine Learning Teams

Here Git for ML projects diverges from regular software development. Core problem has two parts: code is small, text-based, and diffable. Data and model weights are huge, binary, and not meaningfully diffable. Get separation wrong and repository grows to gigabytes, cloning takes twenty minutes, and model checkpoint gets embedded in history forever.

The .gitignore — Your First Commit

Before first line of code in new ML project, create .gitignore. File tells Git which files and directories to ignore completely. It affects only untracked files—if you committed something first and added it to .gitignore later, Git keeps tracking it. Fix that with git rm --cached filename to stop tracking file without deleting it from disk.

# Data — never in git
data/
*.csv
*.parquet
*.json.gz

# Model artifacts
models/
*.pt
*.pth
*.pkl
*.safetensors
*.ckpt
*.onnx

# Python artifacts
__pycache__/
*.pyc
*.pyo
.venv/
venv/
*.egg-info/

# Environment and secrets
.env
.env.*
*.key

# Experiment tracking noise
wandb/
mlruns/
.ipynb_checkpoints/
*.log

Commit this file. Push it. Make it first thing anyone who clones repo sees.

DVC — Git for Data and Models

DVC (Data Version Control) solves large-file problem for ML. Idea: Git tracks code, DVC tracks data and models. Use both together. They integrate cleanly.

dvc add data/train.csv

This does two things. First, it adds data/train.csv to .gitignore, so Git ignores actual file. Second, it creates data/train.csv.dvc—small text file containing file's hash and size, which does go in Git. When collaborator clones repo and runs dvc pull, DVC fetches actual data from remote storage (S3, GCS, Azure, SSH—DVC supports all of them). Git history stays lean. Data stays versioned.

DVC also defines pipelines. dvc.yaml describes ML workflow as DAG of stages—preprocess, train, evaluate—each with defined inputs and outputs. Run dvc repro and DVC reruns only stages whose inputs changed. This is reproducibility infrastructure: given commit and data version, you can reproduce exact model that produced result.

dvc repro          # reproduce the pipeline, skipping unchanged stages
dvc push           # upload tracked data/models to remote storage
dvc pull           # download tracked data/models from remote storage

I once committed 2GB model checkpoint directly to Git. Not to branch—to main. Repository was ruined for everyone. Even after removing file, it lived in Git's history, and every clone pulled that 2GB forever. We had to rewrite history with git filter-branch, force-push everything, and make every team member re-clone. Nobody was happy. Use DVC.

Git LFS — The Simpler Alternative

Git LFS (Large File Storage) is simpler alternative if you already know Git. Install LFS extension, tell it which patterns to track, and it handles rest transparently.

git lfs install
git lfs track "*.pt"       # track all .pt files with LFS
git add .gitattributes     # commit the tracking rules
git add model.pt           # now stored in LFS, not in git
git commit -m "Add model checkpoint"

Large files are stored on separate LFS server. Git objects contain small pointer files instead. Cloning is fast because LFS downloads only what you need. It is less powerful than DVC—no pipelines, no experiment tracking, no Python API—but if your only problem is large model files and you do not want new tooling, LFS solves it.

Commit Messages That Matter

This sounds like soft skill. It is not. "Updated model" is commit message. "Retrain with balanced sampling — accuracy 0.92 but minority-class recall was 0.31; switched to weighted cross-entropy" is commit message. When model accuracy regresses three weeks from now and you are bisecting through 40 commits at 11 PM, difference is finding cause in five minutes or two hours.

Convention: one short summary line under 72 characters, then blank line, then as much context as you want. Include what you changed, why you changed it, and result. If you ran experiment, put key metric in message. Future you benefits. Teammates benefit. Code review gets faster.

Branching for Experiments

Because branches are text files, they are essentially free. Pattern that works well for ML experimentation: one short-lived branch per experiment. Branch from main, change hyperparameter or preprocessing step, train, evaluate, push results to commit message or linked experiment tracker. Merge winner. Delete losers—or keep them around, since disk is cheap and history is useful.

This keeps main in known-good state. It lets multiple experiments run in parallel without clobbering changes. It makes comparison easy: two branches, two result sets, pick better one. Branches are cheap in Git—41-byte text files—so make new one for every experiment. I worked on teams where people committed directly to main. First bad experiment broke everyone's baseline. Branching habit stuck for good.
