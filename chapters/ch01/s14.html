Algorithms and Data Structures for ML

Chapter 1: Python & Programming Foundations 1 table
TL;DR

You don't need to implement red-black trees. But you do need to know why your nearest-neighbor search is slow, why hash maps power your feature stores, and which data structure sits under every ML library you use. This is the "where do these show up in real ML" guide.

Data Structures in the Wild

Most DS&A courses teach you how to implement these. That's backwards for ML work. You need to know which structure is already running underneath the tools you use, so you can reason about performance when things get slow. Let's walk through the ones that actually matter.

Arrays (NumPy ndarrays, Tensors)

Every matrix multiply, every gradient update, every batch of images flowing through a model — it's all arrays. NumPy's ndarray and PyTorch's Tensor are contiguous blocks of memory with a fixed data type, and that contiguity is the whole game. CPUs can blast through contiguous floats with SIMD instructions; GPUs can process thousands of elements in parallel precisely because they know where every element lives in memory.

This is why np.dot(a, b) is 100x faster than a Python for-loop doing the same math. The algorithm is identical — the data layout is what changed.

import numpy as np

# Python list: each element is a pointer to an object somewhere in memory
py_list = [1.0, 2.0, 3.0, 4.0]

# NumPy array: one contiguous block of 64-bit floats
arr = np.array(py_list, dtype=np.float64)

# This single call triggers optimized C code operating on contiguous memory
result = arr @ arr  # dot product — no Python loop involved

Hash Maps (Python dict)

Every feature store you'll use is backed by hash maps. When you call features[user_id], that O(1) lookup is why your serving latency stays flat whether you have 1,000 or 10 million users. Switch that to a list scan and your latency grows linearly with your user base — fine at prototype scale, catastrophic in production.

Beyond feature stores: vocabulary mappings (token_to_index), prediction caching, deduplication, config lookups. If you need "give me the thing for this key, instantly" — it's a dict.

# Feature store pattern: O(1) per lookup regardless of store size
feature_store = {
    "user_42": {"age": 31, "signup_days": 400, "avg_session": 12.5},
    "user_99": {"age": 24, "signup_days": 50, "avg_session": 8.1},
}

# This is O(1) — same speed with 100 or 100 million users
features = feature_store["user_42"]

Trees

Trees show up in three very different ways in ML, and conflating them causes confusion. First, the obvious one: decision trees are the model in Random Forest, XGBoost, and LightGBM. Tree-based models still dominate tabular ML in production because they handle mixed feature types, require less preprocessing, and are fast at inference.

Second: B-trees power every database index. When your training pipeline queries PostgreSQL for 50 million rows and it comes back in seconds instead of hours, that's a B-tree index turning an O(n) scan into O(log n) lookups.

Third: KD-trees and Ball trees are what sklearn.neighbors uses under the hood for nearest-neighbor search. And tries (prefix trees) show up in tokenizer vocabularies and autocomplete systems — they let you find all words starting with "un" without scanning the entire vocabulary.

Heaps / Priority Queues

Anytime you need "the top K items from a massive collection," a heap is the right tool. Full sort costs O(n log n). A heap gives you top-K in O(n log k) — and when n is 10 million candidates and k is 10, that difference is enormous.

Where this appears: beam search in sequence models (LLMs keep the top-k most promising sequences at each step), top-K retrieval in recommendation systems, and priority scheduling in training job queues. Python's heapq module handles this cleanly:

import heapq

scores = [0.92, 0.15, 0.87, 0.33, 0.99, 0.44, 0.76, 0.61]

# Top 3 without sorting the entire list
top_3 = heapq.nlargest(3, scores)
# [0.99, 0.92, 0.87]

Graphs

Here's something that clicks once and never un-clicks: when you call loss.backward() in PyTorch, you're running a reverse traversal over a directed acyclic graph. Every operation you performed on tensors built up a computation graph, and backpropagation walks it backward to compute gradients. That's not a metaphor — it's literally a graph algorithm (reverse topological sort).

Graphs also power ML pipeline orchestration (Airflow DAGs, Kubeflow), knowledge graphs for entity relationships, and the entire field of graph neural networks. If something has "nodes connected by relationships," there's a graph underneath.

Stacks / Queues

Queues are the connective tissue of production ML. When your model receives inference requests through RabbitMQ or SQS, that's a queue decoupling your API from your model server. Stacks and queues also underlie DFS/BFS graph traversals — which means they're indirectly powering the computation graph operations we just discussed.

You won't implement these from scratch. But recognizing "this is a queue problem" helps you pick the right architecture when designing ML systems.

Algorithmic Complexity — The Intuition

If you don't know why your code is O(n²), you can't know why it takes 100x longer when your data doubles. That's the entire pitch for understanding Big-O: it tells you whether your approach will survive contact with real data.

The practical test is simple: "Will this scale to 10x the data?" If your operation is O(n), 10x data means 10x time — probably fine. If it's O(n²), 10x data means 100x time. That's the difference between a 10-minute job and a 16-hour job.

ComplexityNameML ExampleWill It Scale?
O(1) Constant Dict lookup in a feature store Always — same speed at any size
O(log n) Logarithmic Binary search with bisect, balanced tree lookup Always — 30 steps covers 1 billion items
O(n) Linear Single pass over a dataset, vectorized NumPy op Offline: yes. Online: only if n is small
O(n log n) Linearithmic Sorting predictions for ranking, AUC-ROC computation Batch/offline: fine. Real-time on millions: watch it
O(n²) Quadratic Pairwise distance matrix, full self-attention Up to ~50K offline. At 1M: pain. At 10M: impossible
Offline vs Online Rule of Thumb

Offline (training, batch jobs): O(n log n) is almost always fine. O(n²) works up to roughly 50K items. You have time to wait. Online (inference, serving): you need O(1) or O(log n) per request. O(n) per request only if n is small (hundreds, not millions). If your serving path has an O(n²) operation, you have a scaling problem waiting to happen.

Sorting & Searching — What You Actually Need

Python's built-in sort() uses Timsort — O(n log n), stable, and highly optimized for partially-sorted real-world data. You will almost never need to implement your own sorting algorithm. Know that it exists, know it's O(n log n), and move on.

Binary search is worth knowing a bit more about, because it comes up when you need fast lookups into sorted data. Python's bisect module handles this and it's surprisingly useful for things like finding classification thresholds or inserting into sorted sequences:

import bisect

# Sorted thresholds for bucketing a continuous score
thresholds = [0.2, 0.4, 0.6, 0.8]
labels = ["very_low", "low", "medium", "high", "very_high"]

score = 0.55
bucket = bisect.bisect(thresholds, score)
print(labels[bucket])  # "medium" — found in O(log n), not O(n)
The Practical Stance

Know the complexities, not the implementations. You'll use library functions for sorting and searching — sorted(), bisect.bisect(), heapq.nlargest(). Your job is to pick the right one and know when it won't scale, not to hand-code quicksort.

What You Should Now Be Able To Do

  • Name the data structure behind any ML system component — dict for feature stores, heap for beam search, graph for autograd, array for every tensor operation
  • Estimate whether an approach will scale by checking its Big-O against your data size — and know the offline vs online thresholds
  • Reach for bisect when you need fast lookup in sorted data, and heapq when you need top-K without a full sort
  • Explain why loss.backward() is a graph traversal and why contiguous memory layout is what makes NumPy fast
