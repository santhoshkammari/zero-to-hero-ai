SciPy

Chapter 1: Python & Programming Foundations 5 subtopics
TL;DR

SciPy is the scientific toolkit that turns NumPy arrays into real engineering muscle. It gives you battle-tested statistical tests for validating whether your model actually improved or you got lucky (scipy.stats), optimization routines that are the industrial-strength version of the gradient descent you write by hand (scipy.optimize), sparse matrix formats that let you handle million-feature NLP datasets without melting your RAM (scipy.sparse), matrix decomposition algorithms like SVD that power PCA and dimensionality reduction (scipy.linalg), and spatial data structures like KD-trees that make nearest-neighbor search fast enough to be practical (scipy.spatial).

You've got NumPy. You can create arrays, slice them, do matrix multiplication. That's like having a well-stocked kitchen with knives and cutting boards. But you still need recipes. SciPy is that recipe book — it builds on top of NumPy and provides the actual algorithms scientists and engineers use every day. Statistical tests, optimization solvers, sparse data structures, decomposition routines, spatial search. These aren't toys. They're the same LAPACK, ARPACK, and MINPACK Fortran libraries that NASA uses, wrapped in a clean Python API so you don't have to read Fortran from the 1970s.

The mental model is straightforward: NumPy gives you the data structure (the ndarray) and basic operations. SciPy gives you the algorithms that operate on those structures. When you need to test whether two groups of users behave differently, you reach for scipy.stats. When you need to fit a curve or minimize a loss function, you reach for scipy.optimize. When your feature matrix has 500,000 columns but only 0.1% are non-zero, you reach for scipy.sparse. Let's dig into each module and see exactly how they plug into your ML workflow.

scipy.stats — Statistical Tests & Distributions

Why You Need This

You ran an A/B test. Version B's conversion rate is 4.2% versus Version A's 3.8%. Is that a real difference or random noise? Your gut says "higher number wins." Your gut is wrong — without a statistical test, you have no idea. This is where scipy.stats earns its keep. It gives you the machinery to answer: "Given the data I collected, how likely is it that the difference I see happened by chance?"

The module packs two big capabilities. First, probability distributions — objects that let you generate random samples, compute probabilities, and fit distributions to data. Second, hypothesis tests — functions that take your data and return a test statistic and a p-value telling you whether to trust what you're seeing.

Working with Distributions

Every distribution in SciPy follows the same interface. You get a frozen distribution by calling the distribution with its parameters, and then you can sample from it, compute its PDF, CDF, or quantiles. This consistency is a gift — once you learn one distribution, you know them all.

from scipy import stats
import numpy as np

# Create a normal distribution: mean=0, std=1
normal = stats.norm(loc=0, scale=1)

# Sample 1000 points
samples = normal.rvs(size=1000, random_state=42)

# Probability density at x=0 (the peak)
print(f"PDF at x=0: {normal.pdf(0):.4f}")        # 0.3989

# Cumulative probability: P(X <= 1.96)
print(f"CDF at x=1.96: {normal.cdf(1.96):.4f}")  # 0.9750

# Inverse CDF: what x gives us P(X <= x) = 0.95?
print(f"95th percentile: {normal.ppf(0.95):.4f}") # 1.6449

# Fit a normal distribution to observed data
mu_hat, sigma_hat = stats.norm.fit(samples)
print(f"Fitted: mu={mu_hat:.3f}, sigma={sigma_hat:.3f}")

The loc and scale parameters are SciPy's universal way of shifting and stretching any distribution. For the normal distribution, loc is the mean and scale is the standard deviation. For the exponential distribution, scale is 1/λ. Every distribution follows this pattern, so the API stays uniform even when the underlying math changes.

Hypothesis Testing in Practice

Here's a realistic scenario: you've built a new feature engineering pipeline and you want to know if model accuracy actually improved. You ran both pipelines 30 times each (different random seeds) and collected accuracy scores. A t-test tells you whether the difference in means is statistically significant.

from scipy import stats
import numpy as np

np.random.seed(42)

# Simulated accuracy scores from 30 runs of each pipeline
old_pipeline = np.random.normal(loc=0.82, scale=0.03, size=30)
new_pipeline = np.random.normal(loc=0.84, scale=0.03, size=30)

# Two-sample independent t-test
t_stat, p_value = stats.ttest_ind(old_pipeline, new_pipeline)
print(f"t-statistic: {t_stat:.4f}")
print(f"p-value: {p_value:.6f}")

# Interpret the result
alpha = 0.05
if p_value < alpha:
    print("Reject null hypothesis — the improvement is statistically significant.")
else:
    print("Fail to reject — the difference could be due to random variation.")

The p-value answers one specific question: "If the two pipelines were actually identical in performance, what's the probability of seeing a difference at least this large?" A small p-value (typically below 0.05) means it's very unlikely the difference is just noise. A large p-value means you can't rule out chance. Note the wording carefully — you never "prove" anything. You either reject the null hypothesis or you don't.

For categorical data — like whether users who saw a certain UI element clicked more often — you'd use the chi-squared test:

from scipy.stats import chi2_contingency
import numpy as np

# Contingency table: rows = [control, treatment], cols = [no_click, click]
observed = np.array([[450, 50],    # control: 450 didn't click, 50 clicked
                     [420, 80]])   # treatment: 420 didn't click, 80 clicked

chi2, p_value, dof, expected = chi2_contingency(observed)
print(f"Chi-squared: {chi2:.4f}, p-value: {p_value:.6f}, dof: {dof}")
print(f"Expected frequencies:\n{expected}")
⚠️ p-value ≠ Effect Size

A tiny p-value tells you the effect is real, not that it's big. With enough data, even a 0.01% improvement becomes "statistically significant." Always pair your p-value with an effect size measure (like Cohen's d) or a confidence interval. In ML, ask: "Is the improvement large enough to justify the added complexity?"

scipy.optimize — Optimization Routines

Why You Need This

Training a machine learning model means one thing at its core: finding the set of parameters that minimizes a loss function. Gradient descent is how most ML frameworks do it. But gradient descent is one specific optimization algorithm, and scipy.optimize gives you a toolbox full of them — including methods that work when you don't have gradients, methods with convergence guarantees, and methods for constrained problems where parameters must stay within bounds.

Minimizing a Function

Let's start with the workhorse: minimize(). Say you want to find the minimum of a simple loss function. Here's how SciPy's optimizer and hand-rolled gradient descent solve the exact same problem:

from scipy.optimize import minimize
import numpy as np

# A simple loss function: L(w) = (w - 3)^2 + 2*sin(w)
def loss(w):
    return (w - 3)**2 + 2 * np.sin(w)

# Its gradient (derivative)
def gradient(w):
    return 2 * (w - 3) + 2 * np.cos(w)

# --- SciPy's optimizer ---
result = minimize(loss, x0=0.0, jac=gradient, method='L-BFGS-B')
print(f"SciPy found minimum at w = {result.x[0]:.6f}, loss = {result.fun:.6f}")

# --- Manual gradient descent (for comparison) ---
w = 0.0
lr = 0.1
for step in range(200):
    w = w - lr * gradient(w)
print(f"Manual GD found minimum at w = {w:.6f}, loss = {loss(w):.6f}")

Both arrive at the same answer. The difference? SciPy's L-BFGS-B method uses second-order information (an approximation of the Hessian) to take smarter steps. It typically converges in far fewer iterations than vanilla gradient descent. When you pass jac=gradient, you're giving the optimizer the analytical gradient. If you skip that, SciPy will approximate the gradient numerically using finite differences — slower but still works.

Curve Fitting

Curve fitting is optimization in disguise. You hypothesize a functional form (say, exponential decay) and let SciPy find the parameters that best match your data. This is exactly what happens when you fit a regression model — you're optimizing parameters to minimize the residuals.

from scipy.optimize import curve_fit
import numpy as np

# Model: exponential decay y = a * exp(-b * x) + c
def exp_decay(x, a, b, c):
    return a * np.exp(-b * x) + c

# Simulated data: training loss over epochs (with noise)
np.random.seed(42)
epochs = np.arange(0, 50, 1)
true_loss = 2.5 * np.exp(-0.08 * epochs) + 0.3
noisy_loss = true_loss + np.random.normal(0, 0.05, size=len(epochs))

# Fit the curve
params, covariance = curve_fit(exp_decay, epochs, noisy_loss, p0=[2.0, 0.1, 0.5])
a_fit, b_fit, c_fit = params
print(f"Fitted: a={a_fit:.3f}, b={b_fit:.4f}, c={c_fit:.3f}")
print(f"Predicted final loss plateau: {c_fit:.3f}")

This is genuinely useful. If your training loss follows an exponential decay curve, fitting it lets you predict where the loss will plateau — so you can decide whether it's worth training for 100 more epochs or if you've already squeezed out most of the improvement. The covariance matrix returned tells you how confident the fit is for each parameter.

ℹ️ SciPy vs. PyTorch Optimizers

PyTorch's torch.optim.SGD and Adam are designed for stochastic, mini-batch optimization on massive datasets with GPU acceleration. SciPy's optimizers work on the full dataset in CPU memory. Use SciPy for small-scale optimization, curve fitting, hyperparameter tuning of simple models, and scientific computing. Use PyTorch/TensorFlow optimizers for training neural networks.

scipy.sparse — Sparse Matrices

Why You Need This

Imagine you're building a text classifier. Your vocabulary has 100,000 words. Each document is represented as a vector of 100,000 numbers — one per word — counting how many times each word appears. Most of those counts are zero. A typical document uses maybe 200 unique words out of 100,000. That means 99.8% of every row in your feature matrix is zeros. Storing all those zeros in a dense NumPy array is pure waste.

With 50,000 documents, your dense matrix would be 50,000 × 100,000 = 5 billion entries. At 8 bytes each (float64), that's 40 GB of RAM. A sparse matrix stores only the non-zero values and their positions. If each document averages 200 non-zero entries, you're storing 10 million values instead of 5 billion — roughly 240 MB. That's a 160× reduction.

Sparse Formats: CSR and CSC

SciPy offers several sparse formats, but two dominate in practice. CSR (Compressed Sparse Row) is optimized for fast row access and matrix-vector multiplication — perfect when you're iterating over data points (rows). CSC (Compressed Sparse Column) is optimized for fast column access — perfect when you're iterating over features (columns). Most ML libraries expect CSR.

from scipy.sparse import csr_matrix, csc_matrix, random as sparse_random
import numpy as np

# Create a sparse matrix from dense data
dense = np.array([
    [0, 0, 3, 0, 0],
    [1, 0, 0, 0, 7],
    [0, 2, 0, 0, 0],
    [0, 0, 0, 4, 0]
])

sparse_csr = csr_matrix(dense)
print(f"Dense shape: {dense.shape}")
print(f"Dense memory: {dense.nbytes} bytes")
print(f"Sparse stored entries: {sparse_csr.nnz}")
print(f"Sparse memory: {sparse_csr.data.nbytes + sparse_csr.indices.nbytes + sparse_csr.indptr.nbytes} bytes")
print(f"Sparsity: {1 - sparse_csr.nnz / np.prod(dense.shape):.1%}")

# Access row 1 (fast in CSR)
print(f"\nRow 1: {sparse_csr[1].toarray()}")

Real-World Memory Savings

Let's see the savings at ML scale:

from scipy.sparse import random as sparse_random
import numpy as np

# Simulate a TF-IDF matrix: 50,000 documents, 100,000 vocabulary
n_docs, n_vocab = 50_000, 100_000
density = 0.002  # each doc uses ~200 words out of 100k

sparse_tfidf = sparse_random(n_docs, n_vocab, density=density, format='csr',
                              random_state=42)

# Compare memory usage
dense_bytes = n_docs * n_vocab * 8  # float64
sparse_bytes = (sparse_tfidf.data.nbytes +
                sparse_tfidf.indices.nbytes +
                sparse_tfidf.indptr.nbytes)

print(f"Dense would need:  {dense_bytes / 1e9:.1f} GB")
print(f"Sparse needs:      {sparse_bytes / 1e6:.0f} MB")
print(f"Compression ratio: {dense_bytes / sparse_bytes:.0f}x")
print(f"Non-zero entries:  {sparse_tfidf.nnz:,}")

This isn't a contrived example. Every time you use TfidfVectorizer from scikit-learn, it returns a CSR sparse matrix. Recommender systems are the same story — a user-item interaction matrix for a platform with 1 million users and 500,000 products is almost entirely zeros. Without sparse matrices, these problems don't fit in memory at all.

Arithmetic with Sparse Matrices

Sparse matrices support the same operations as dense arrays — addition, multiplication, slicing — but they're smart about it. When you multiply two sparse matrices, the algorithm only touches the non-zero entries. A matrix-vector product that would take O(n²) on a dense matrix takes O(nnz) on a sparse one, where nnz is the number of non-zero entries.

from scipy.sparse import csr_matrix
import numpy as np

# Sparse matrix-vector multiply (like computing predictions)
X_sparse = csr_matrix(np.random.binomial(1, 0.01, size=(10000, 50000)))
weights = np.random.randn(50000)

# This runs in O(nnz) time, not O(10000 * 50000)
predictions = X_sparse.dot(weights)
print(f"Predictions shape: {predictions.shape}")
print(f"Non-zero entries in X: {X_sparse.nnz:,} out of {10000 * 50000:,}")
🚨 Don't Call .toarray() on Large Sparse Matrices

Calling .toarray() or .todense() on a large sparse matrix defeats the entire purpose — it allocates the full dense array in memory. If your sparse matrix fits in 200 MB but the dense version would be 40 GB, calling .toarray() will crash your process. Keep operations in sparse land as long as possible. Scikit-learn, XGBoost, and LightGBM all accept sparse inputs natively.

scipy.linalg — Linear Algebra

Why You Need This

NumPy has numpy.linalg. SciPy has scipy.linalg. What's the difference? SciPy's version is a superset — it includes everything NumPy has plus additional decompositions, matrix functions, and specialized solvers. Under the hood, it links against optimized LAPACK and BLAS implementations. For ML, the star of this module is Singular Value Decomposition (SVD), which is the engine behind PCA, latent semantic analysis, and low-rank approximations.

SVD — The Swiss Army Knife of Matrix Decomposition

SVD factors any matrix A into three matrices: A = UΣVᵀ. U contains the left singular vectors (patterns in the rows), Σ is a diagonal matrix of singular values (importance of each pattern), and Vᵀ contains the right singular vectors (patterns in the columns). The singular values are sorted in decreasing order, so the first few capture the most important structure in your data.

from scipy.linalg import svd
import numpy as np

np.random.seed(42)

# Simulate a dataset: 100 samples, 50 features
# The data actually lives in a ~5-dimensional subspace + noise
n_samples, n_features, true_rank = 100, 50, 5
low_rank = np.random.randn(n_samples, true_rank) @ np.random.randn(true_rank, n_features)
noise = 0.1 * np.random.randn(n_samples, n_features)
X = low_rank + noise

# Center the data (mean-subtract) — necessary before PCA
X_centered = X - X.mean(axis=0)

# Full SVD
U, sigma, Vt = svd(X_centered, full_matrices=False)

print(f"U shape: {U.shape}")       # (100, 50) — left singular vectors
print(f"sigma shape: {sigma.shape}")  # (50,)    — singular values
print(f"Vt shape: {Vt.shape}")     # (50, 50) — right singular vectors

# Variance explained by each component
variance_explained = sigma**2 / np.sum(sigma**2)
cumulative = np.cumsum(variance_explained)
print(f"\nTop 5 components explain {cumulative[4]:.1%} of variance")
print(f"Top 10 components explain {cumulative[9]:.1%} of variance")

This is PCA. When you call sklearn.decomposition.PCA, it's running SVD under the hood. The principal components are the rows of Vᵀ, and the transformed data is U × Σ (or equivalently, X_centered @ Vᵀ.T). By keeping only the top k components, you reduce your 50 features down to k features while preserving most of the variance.

Low-Rank Approximation

You can reconstruct an approximation of the original matrix using only the top k singular values. This is how latent semantic analysis compresses a term-document matrix, and how recommender systems approximate a user-item matrix:

from scipy.linalg import svd
import numpy as np

# Keep only top k components for a low-rank approximation
k = 5
X_approx = U[:, :k] @ np.diag(sigma[:k]) @ Vt[:k, :]

# Measure reconstruction error
error = np.linalg.norm(X_centered - X_approx) / np.linalg.norm(X_centered)
print(f"Reconstruction error with k={k}: {error:.4f}")
print(f"Original matrix: {X_centered.shape} = {np.prod(X_centered.shape)} values")
print(f"Compressed: {n_samples*k + k + k*n_features} values")
print(f"Compression ratio: {np.prod(X_centered.shape) / (n_samples*k + k + k*n_features):.1f}x")

Other Useful Decompositions

Beyond SVD, scipy.linalg gives you LU decomposition for solving linear systems efficiently, Cholesky decomposition for symmetric positive-definite matrices (which show up in Gaussian processes and kernel methods), and QR decomposition for least-squares problems. Each has its niche, but SVD is the one you'll reach for most in ML.

from scipy.linalg import cho_factor, cho_solve
import numpy as np

# Cholesky solve: useful in Gaussian Processes
# Solve Kx = y where K is a positive-definite covariance matrix
K = np.array([[2.0, 0.5, 0.3],
              [0.5, 1.5, 0.2],
              [0.3, 0.2, 1.0]])
y = np.array([1.0, 2.0, 3.0])

# Cholesky is 2x faster than generic solve for positive-definite matrices
c, low = cho_factor(K)
x = cho_solve((c, low), y)
print(f"Solution: {x}")
print(f"Verify K @ x ≈ y: {np.allclose(K @ x, y)}")

scipy.spatial — Distance & Spatial Data Structures

Why You Need This

K-nearest neighbors. K-means clustering. DBSCAN. Anomaly detection. All of these algorithms rely on one fundamental operation: measuring how far apart two data points are. And when your dataset has millions of points, computing distances naively (every point against every other point) is O(n²) — unbearably slow. scipy.spatial gives you two things: efficient distance computation and smart data structures that make nearest-neighbor queries logarithmic instead of linear.

Distance Computation

The distance submodule computes pairwise distances between collections of points. The two workhorses are cdist (distances between two sets of points) and pdist (all pairwise distances within one set).

from scipy.spatial.distance import cdist, pdist, squareform
import numpy as np

np.random.seed(42)

# 5 data points in 3 dimensions
points = np.random.randn(5, 3)

# All pairwise Euclidean distances (condensed form)
pairwise = pdist(points, metric='euclidean')
print(f"Condensed distance vector: {pairwise.shape}")  # (10,) — 5 choose 2

# Convert to a full distance matrix
dist_matrix = squareform(pairwise)
print(f"Distance matrix:\n{np.round(dist_matrix, 2)}")

# Distances between two different sets of points
query = np.random.randn(2, 3)
distances = cdist(query, points, metric='euclidean')
print(f"\nDistances from 2 queries to 5 points:\n{np.round(distances, 2)}")

# Find nearest neighbor for each query
nearest_idx = distances.argmin(axis=1)
print(f"Nearest point indices: {nearest_idx}")

The metric parameter accepts over 20 distance metrics — Euclidean, Manhattan (cityblock), cosine, Mahalanobis, Hamming, and more. Cosine distance is particularly important in NLP, where documents are compared by the angle between their TF-IDF vectors rather than their magnitude. Manhattan distance (L1) is often preferred in high-dimensional spaces where Euclidean distances become less meaningful — a phenomenon called the curse of dimensionality.

KD-Trees for Fast Nearest-Neighbor Search

Computing cdist between your query and every point in the dataset is O(n) per query. If your dataset has 10 million points and you need to answer thousands of queries, that's brutal. A KD-tree is a binary tree that partitions space, letting you find nearest neighbors in O(log n) time on average.

from scipy.spatial import KDTree
import numpy as np
import time

np.random.seed(42)

# Build a KD-tree with 100,000 points in 10 dimensions
data = np.random.randn(100_000, 10)
tree = KDTree(data)

# Query: find the 5 nearest neighbors of a single point
query_point = np.random.randn(10)

start = time.perf_counter()
distances, indices = tree.query(query_point, k=5)
tree_time = time.perf_counter() - start

print(f"KD-tree query time: {tree_time*1000:.2f} ms")
print(f"Nearest 5 indices: {indices}")
print(f"Distances: {np.round(distances, 3)}")

# Compare with brute-force
start = time.perf_counter()
all_dists = np.linalg.norm(data - query_point, axis=1)
brute_indices = np.argsort(all_dists)[:5]
brute_time = time.perf_counter() - start

print(f"\nBrute-force query time: {brute_time*1000:.2f} ms")
print(f"Same result: {np.array_equal(np.sort(indices), np.sort(brute_indices))}")

The KD-tree shines when you have many queries to answer against the same dataset. Building the tree takes O(n log n), but each subsequent query is O(log n) instead of O(n). For a KNN classifier making predictions on a test set of 10,000 points against a training set of 1,000,000 points, that's the difference between hours and seconds.

⚠️ KD-Trees Struggle in High Dimensions

KD-trees work well up to about 20-30 dimensions. Beyond that, the partitioning becomes ineffective and query time degrades toward brute-force. For high-dimensional data (like 768-dim embeddings), look into approximate nearest-neighbor libraries like FAISS, Annoy, or ScaNN. They sacrifice a tiny bit of accuracy for massive speed gains. SciPy's cKDTree (the C-optimized version) helps, but it can't beat the curse of dimensionality.

Radius Queries and Spatial Clustering

Sometimes you don't want the k closest points — you want all points within a certain distance. This is exactly what DBSCAN does internally to find clusters. KD-trees handle this too:

from scipy.spatial import KDTree
import numpy as np

np.random.seed(42)

# 2D data — easy to visualize
points = np.random.randn(1000, 2)
tree = KDTree(points)

# Find all points within radius 0.5 of the origin
center = np.array([0.0, 0.0])
indices = tree.query_ball_point(center, r=0.5)
print(f"Points within radius 0.5 of origin: {len(indices)}")

# Count neighbors for every point (local density estimation)
# This is essentially what DBSCAN computes
neighbor_counts = tree.query_ball_point(points, r=0.3)
densities = np.array([len(n) for n in neighbor_counts])
print(f"Average neighborhood size: {densities.mean():.1f}")
print(f"Densest point has {densities.max()} neighbors")

The query_ball_point method returns all points within a given radius — a core operation in density-based clustering algorithms. Combined with the distance matrix from pdist, you have everything you need to implement clustering algorithms from scratch or understand what scikit-learn does under the hood.

Tying It All Together

These five modules aren't isolated — they feed into each other in real ML workflows. You use scipy.sparse to represent your text data. You pass that sparse matrix to an SVD from scipy.linalg (or its sparse variant scipy.sparse.linalg.svds) to reduce dimensions. You use scipy.spatial to build a KD-tree on the reduced representations for fast neighbor search. You evaluate your model's improvement with a statistical test from scipy.stats. And when you need to tune a simple model or fit a parametric curve, scipy.optimize handles the math.

SciPy isn't glamorous. It doesn't generate images or chat with you. But it's the bedrock — the layer of proven, battle-tested algorithms that every higher-level library builds on. Scikit-learn uses SciPy's sparse matrices, distance functions, and optimization routines internally. Understanding SciPy means understanding what's happening below the surface of your ML pipeline, and that understanding is what separates someone who can use tools from someone who can debug them when they break.
