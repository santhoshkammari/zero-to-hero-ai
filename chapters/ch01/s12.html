Scikit-learn

Chapter 1: Python & Programming Foundations Concise — Leverage Tool

The beautiful thing about scikit-learn is that you learn the API once and you can use any of its 200+ algorithms without reading separate documentation for each one. Every estimator — classifier, regressor, scaler, encoder, decomposition — follows the exact same pattern. That consistency is the real superpower.

The Universal API Pattern

Every scikit-learn estimator works the same way:

  • Models: .fit(X, y) → .predict(X)
  • Transformers: .fit(X) → .transform(X) (or .fit_transform(X) as a shortcut)

That's it. Whether it's a linear regression, a random forest, a PCA, or a k-means — the interface is identical. Watch what happens when you swap models:

from sklearn.linear_model import LinearRegression
from sklearn.ensemble import RandomForestRegressor

# Version 1: linear regression
model = LinearRegression()
model.fit(X_train, y_train)
predictions = model.predict(X_test)

# Version 2: random forest — literally change one line
model = RandomForestRegressor(n_estimators=100)
model.fit(X_train, y_train)
predictions = model.predict(X_test)

Nothing else changes. The rest of your code — evaluation, pipeline, cross-validation — all works identically. This is why scikit-learn won.

The Golden Rule

fit() on training data only. transform()/predict() on both train and test. If you fit a scaler on your test data, it learned test-set statistics and your metrics are lying to you. This is data leakage — the most common beginner mistake in ML.

Pipelines — The Production Pattern

A Pipeline chains preprocessing steps and a model into a single estimator. Why bother? Three reasons:

  • Prevents data leakage — when you pass a pipeline to cross_val_score, preprocessing gets refit on each fold's training data automatically. No accidental fitting on test data.
  • Clean code — one object does everything. No scattered scaler/encoder variables floating around.
  • Easy serialization — joblib.dump(pipe, 'model.pkl') saves the entire pipeline. Deploy it, and preprocessing + prediction happen in one call.

The real-world pattern uses ColumnTransformer because numeric and categorical columns need different preprocessing:

from sklearn.pipeline import Pipeline, make_pipeline
from sklearn.compose import ColumnTransformer
from sklearn.preprocessing import StandardScaler, OneHotEncoder
from sklearn.impute import SimpleImputer
from sklearn.ensemble import RandomForestClassifier

# Different preprocessing for different column types
num_pipe = make_pipeline(SimpleImputer(strategy='median'), StandardScaler())
cat_pipe = make_pipeline(SimpleImputer(strategy='most_frequent'),
                         OneHotEncoder(drop='first', handle_unknown='ignore'))

preprocessor = ColumnTransformer([
    ('num', num_pipe, ['age', 'salary', 'years_exp']),
    ('cat', cat_pipe, ['department', 'city'])
])

# One object: impute → scale/encode → classify
pipe = Pipeline([('prep', preprocessor), ('clf', RandomForestClassifier())])

pipe.fit(X_train, y_train)          # all preprocessing fits on train only
pipe.predict(X_test)                # transforms test, then predicts

make_pipeline() is a shorthand that auto-names steps — use it for simple chains where you don't need to reference steps by name.

Preprocessing — Which One When

ScalerWhat It DoesUse When
StandardScalerZero mean, unit varianceDefault choice — SVM, logistic regression, KNN
MinMaxScalerSqueezes into [0, 1]Bounded values needed — neural nets, image data
RobustScalerUses median/IQR instead of mean/stdData has outliers — the other two get wrecked
EncoderWhat It DoesUse When
OneHotEncoderBinary column per categoryNominal features — color, city, department
OrdinalEncoderMaps categories to integersOrdinal features — S < M < L < XL
LabelEncoderMaps labels to integersTarget variable y only — never for input features

Model Selection

train_test_split — always do this first. Use stratify=y for classification so class proportions are preserved in both sets. Hold the test set sacred — evaluate on it exactly once, at the very end.

cross_val_score — the honest way to evaluate. Splits training data into folds, trains and evaluates on each. Pass your pipeline (not just the model) so preprocessing gets refit per fold:

from sklearn.model_selection import cross_val_score

scores = cross_val_score(pipe, X_train, y_train, cv=5, scoring='f1')
print(f"F1: {scores.mean():.3f} ± {scores.std():.3f}")

GridSearchCV / RandomizedSearchCV — hyperparameter tuning with cross-validation built in. GridSearchCV tries every combination (exhaustive, slow). RandomizedSearchCV samples randomly (faster, usually good enough). Both follow the estimator API — fit(), predict(), best_params_.

The workflow: split → build pipeline → cross-validate → tune hyperparameters → evaluate on test set (once). In that order.

What You Should Now Be Able To Do

  • Explain the fit() / predict() / transform() contract and swap any algorithm without changing surrounding code
  • Build a Pipeline with ColumnTransformer that handles numeric and categorical features with zero data leakage
  • Choose the right scaler (Standard vs MinMax vs Robust) and encoder (OneHot vs Ordinal vs Label) for the situation
  • Use cross_val_score with a pipeline for honest evaluation
  • Tune hyperparameters with GridSearchCV or RandomizedSearchCV
  • Follow the correct workflow: split → pipeline → cross-validate → tune → test set once
