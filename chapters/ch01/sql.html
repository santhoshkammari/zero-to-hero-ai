SQL — Structured Query Language — language for talking to relational databases. Born at IBM in 1970s, standardized in 1986, now powers nearly every system storing structured data at scale. ML pipelines pulling training data, feature stores serving real-time predictions, analytics dashboards stakeholders watch — SQL sits underneath.

Why SQL Matters for ML

First year of data journey, I avoided SQL. Had Pandas. Could load CSVs. Why need ancient 1970s language that looked like half-finished English-as-programming?

Then team had training data in warehouse: 400 million rows, 80 tables. Notebook loaded everything into DataFrame, crashed. Senior engineer said, "Write a query." Avoidance ended. Dive starts here.

Start small. Tiny three-row table first, production-grade patterns for real ML feature pipelines later. No prior database experience needed. Add concepts one at time, with explanation.

Journey long. Hope trip worth it.

Tables: The Atomic Unit

Relational database stores data in tables. Table is grid — rows and columns, like spreadsheet, but with rules. Each column has name and type. Each row is one record. That's it.

Build tiny one. Imagine tracking ML experiments:

CREATE TABLE experiments (
    id        INTEGER PRIMARY KEY,
    model     TEXT NOT NULL,
    accuracy  REAL,
    train_date DATE
);

INSERT INTO experiments VALUES (1, 'logistic_reg', 0.82, '2024-01-15');
INSERT INTO experiments VALUES (2, 'random_forest', 0.91, '2024-01-16');
INSERT INTO experiments VALUES (3, 'xgboost', 0.94, '2024-01-17');

Now table experiments has four columns and three rows. INTEGER PRIMARY KEY gives each row unique identifier — no two experiments share id. NOT NULL on model means no row without model name. These are constraints — rules keeping data sane.

DataFrame is table in memory. Database table is table on disk — with guarantees for correctness, concurrency, durability that DataFrame never gives.

Your First Query: SELECT

Reading data from table is query. Command is SELECT.

SELECT model, accuracy FROM experiments;

model          | accuracy
---------------+---------
logistic_reg   | 0.82
random_forest  | 0.91
xgboost        | 0.94

Asked for two columns. Got all three rows. Asterisk * means "everything":

SELECT * FROM experiments;

That gives every column. In production, you almost never want * — pulling data across network, dragging 50 columns when you need 3 wastes bandwidth. For exploration, fine.

Mental model: SELECT changes nothing. Read operation. Run it thousand times, data stays same. Like flashlight on filing cabinet — you look, not move.

Filtering with WHERE

WHERE clause is filter. It returns only rows satisfying condition.

SELECT model, accuracy
FROM experiments
WHERE accuracy > 0.90;

model          | accuracy
---------------+---------
random_forest  | 0.91
xgboost        | 0.94

Database engine scans rows, checks condition on each one, returns only rows that pass. You can combine conditions with AND and OR:

SELECT * FROM experiments
WHERE accuracy > 0.85 AND train_date >= '2024-01-16';

This gives models that are accurate and recent. Pattern stays same: SELECT what you want, FROM where it lives, WHERE conditions hold. That three-part structure is spine of almost every SQL query you write.

Sorting and Limiting

Tables have no inherent order. Rows come back in whatever order engine chooses unless you specify one.

SELECT model, accuracy
FROM experiments
ORDER BY accuracy DESC;

DESC means descending — best accuracy first. ASC (default) goes other way.

Have millions of rows and want top 10?

SELECT model, accuracy
FROM experiments
ORDER BY accuracy DESC
LIMIT 10;

You use this constantly in ML work — "give me best-performing experiments" or "show me most recent 100 training runs." Without LIMIT, production query can return millions of rows and crash notebook. Learned that hard way.

Aggregations: GROUP BY

So far, individual rows. Often you want summaries — averages, counts, maximums across groups. This is where GROUP BY comes in.

Expand table with more experiments:

INSERT INTO experiments VALUES (4, 'logistic_reg', 0.84, '2024-02-01');
INSERT INTO experiments VALUES (5, 'xgboost', 0.95, '2024-02-02');
INSERT INTO experiments VALUES (6, 'random_forest', 0.89, '2024-02-03');

Now multiple runs per model. What is average accuracy for each?

SELECT model, AVG(accuracy) AS avg_acc, COUNT(*) AS num_runs
FROM experiments
GROUP BY model;

model          | avg_acc | num_runs
---------------+---------+---------
logistic_reg   | 0.83    | 2
random_forest  | 0.90    | 2
xgboost        | 0.945   | 2

GROUP BY model collapses rows with same model name into one group. Then AVG(accuracy) computes mean inside each group. COUNT(*) tells how many rows fell in each bucket. AS names computed column — without it, header becomes ugly like AVG(accuracy).

Aggregate functions — COUNT, SUM, AVG, MIN, MAX — are five you use 95% of time. They take column with many values and reduce it to one value per group.

Want to filter after grouping? Use HAVING:

SELECT model, AVG(accuracy) AS avg_acc
FROM experiments
GROUP BY model
HAVING AVG(accuracy) > 0.85;

WHERE filters individual rows before grouping. HAVING filters groups after aggregation. I mixed them up for months. Key rule: if condition uses aggregate function, put it in HAVING.

Rest Stop

Made it this far. You can stop here if you want.

Now you have mental model for maybe 70% of SQL in wild: select columns, filter rows, sort, limit, aggregate into groups. Enough to pull training data, compute basic feature statistics, explore new dataset handed to you.

Story not complete. Real data lives across multiple tables — users in one, orders in another, products in third. To combine them, you need JOINs. For complex logic, you need subqueries and CTEs. For things like "running averages" or "rank within group," you need window functions. These separate using SQL from thinking in SQL.

Short version: JOINs combine tables by matching rows on shared key. CTEs name intermediate results. Window functions compute aggregates without collapsing rows. There. 70% done.

If not knowing underneath still nags, read on.

JOINs: Connecting Tables

Real databases do not keep everything in one giant table. They split data across tables and connect them with shared keys. This is normalization — reason is avoiding redundancy. If user's name changes, you update one place, not every row in every table mentioning user.

Add second table:

CREATE TABLE datasets (
    id       INTEGER PRIMARY KEY,
    name     TEXT NOT NULL,
    num_rows INTEGER
);

INSERT INTO datasets VALUES (1, 'mnist', 60000);
INSERT INTO datasets VALUES (2, 'cifar10', 50000);
INSERT INTO datasets VALUES (3, 'imdb_reviews', 25000);

Now modify experiments so they reference dataset used:

ALTER TABLE experiments ADD COLUMN dataset_id INTEGER;
UPDATE experiments SET dataset_id = 1 WHERE id IN (1, 4);
UPDATE experiments SET dataset_id = 2 WHERE id IN (2, 5);
UPDATE experiments SET dataset_id = 3 WHERE id IN (3, 6);

dataset_id in experiments points to id in datasets. This is foreign key — reference from one table to another. Now we can combine them:

SELECT e.model, e.accuracy, d.name AS dataset, d.num_rows
FROM experiments e
JOIN datasets d ON e.dataset_id = d.id
WHERE e.accuracy > 0.90;

model          | accuracy | dataset      | num_rows
---------------+----------+--------------+---------
random_forest  | 0.91     | cifar10      | 50000
xgboost        | 0.94     | imdb_reviews | 25000
xgboost        | 0.95     | cifar10      | 50000

JOIN ... ON says: "for each experiment row, find dataset row where IDs match, glue them side by side." e and d are aliases — shorthand so you do not type full table name every time.

JOIN has several flavors. One above is INNER JOIN — it returns only rows with match in both tables. If experiment had no dataset_id, it gets excluded. LEFT JOIN keeps all rows from left table and fills NULLs where no match exists on right. Use LEFT JOIN when you want "give me all experiments, and if they have dataset, show it — otherwise leave blank."

Subqueries and CTEs

Sometimes one query must feed another. Two tools: subqueries (queries nested inside queries) and CTEs (Common Table Expressions — named intermediate results).

Subquery example — find experiments beating overall average:

SELECT model, accuracy
FROM experiments
WHERE accuracy > (SELECT AVG(accuracy) FROM experiments);

Inner query runs first, computes average (0.89-ish), outer query uses that number as filter. Works, but after three or four nests, query becomes unreadable. CTEs save you there.

WITH avg_baseline AS (
    SELECT AVG(accuracy) AS val FROM experiments
),
top_models AS (
    SELECT model, accuracy
    FROM experiments
    WHERE accuracy > (SELECT val FROM avg_baseline)
)
SELECT * FROM top_models ORDER BY accuracy DESC;

WITH introduces CTE. Each CTE is named temporary result that exists only for query duration. Read top to bottom: compute baseline, find top models, select from them. Same logic as nested subqueries, but linear so human can follow.

I resisted CTEs for while because subqueries felt more "compact." Then I revisited my own queries three weeks later and could not tell what they did. CTEs are not for database. CTEs are for future-you reading own work.

Window Functions

GROUP BY has limitation: it collapses rows. If you want aggregate and still keep individual rows visible, you get stuck. Window functions solve this.

SELECT model, accuracy, train_date,
       RANK() OVER (ORDER BY accuracy DESC) AS rank,
       AVG(accuracy) OVER () AS global_avg
FROM experiments;

model          | accuracy | train_date | rank | global_avg
---------------+----------+------------+------+-----------
xgboost        | 0.95     | 2024-02-02 | 1    | 0.891
xgboost        | 0.94     | 2024-01-17 | 2    | 0.891
random_forest  | 0.91     | 2024-01-16 | 3    | 0.891
random_forest  | 0.89     | 2024-02-03 | 4    | 0.891
logistic_reg   | 0.84     | 2024-02-01 | 5    | 0.891
logistic_reg   | 0.82     | 2024-01-15 | 6    | 0.891

Every row stays. Now each row also carries rank and global average. OVER() defines window — set of rows function looks at. Empty OVER() means "entire table." Add ORDER BY inside it, function computes running calculation.

You can partition windows too:

SELECT model, accuracy, train_date,
       ROW_NUMBER() OVER (PARTITION BY model ORDER BY accuracy DESC) AS model_rank
FROM experiments;

This ranks experiments inside each model separately. PARTITION BY splits window into independent groups — like GROUP BY, but without collapsing. Common window functions: ROW_NUMBER(), RANK(), DENSE_RANK(), LAG(), LEAD(), and aggregates like SUM() OVER or AVG() OVER.

In ML feature engineering, window functions are essential. "Rolling 7-day average of user activity," "rank of this product within its category," "previous purchase amount" — all window functions.

Writing Data: INSERT, UPDATE, DELETE

We have been reading data. Now modify it.

INSERT adds new rows (already seen). UPDATE modifies existing rows. DELETE removes them.

-- Fix a wrong accuracy
UPDATE experiments SET accuracy = 0.83 WHERE id = 1;

-- Remove experiments older than a date
DELETE FROM experiments WHERE train_date < '2024-01-16';

-- Insert with explicit columns (safer than relying on order)
INSERT INTO experiments (id, model, accuracy, train_date, dataset_id)
VALUES (7, 'neural_net', 0.96, '2024-02-10', 1);

Critical habit: always include WHERE with UPDATE and DELETE. Running DELETE FROM experiments without WHERE wipes every row in table. No confirmation dialog. No undo. Gone. I have seen production data vanish this way. Database trusts that you meant what you said.

Schema Design and Normalization

How you structure tables matters more than how you query them. Well-designed schema makes queries easy and data consistent. Bad schema makes every query nightmare of self-joins and case statements.

Core principle of normalization: every fact is stored exactly once. If dataset name appears in 10,000 experiment rows and someone renames it, you must update 10,000 rows. With normalization, name lives in one place (datasets table), and experiments reference it by ID.

Levels — first normal form, second, third, Boyce-Codd — are academic categories. Practice rule: if you keep duplicating data across rows, move it into its own table and reference it with foreign key. If column makes sense for only some rows, it probably belongs in related table.

For ML, tension exists. Normalized schemas help data integrity but need JOINs to rebuild full picture. Analytical queries often benefit from denormalized tables — wide tables with redundant data scanned without joins. Data warehouses (BigQuery, Snowflake, Redshift) are built for this pattern. Normalize for writes. Denormalize for reads. Most ML feature tables are denormalized.

Indexes and Query Performance

When you run WHERE accuracy > 0.90 on three rows, database checks each one. On 400 million rows, that hurts. Index is data structure (usually B-tree) that lets database jump straight to matching rows instead of scanning everything.

CREATE INDEX idx_experiments_accuracy ON experiments(accuracy);

Now queries filtering on accuracy can use index to skip irrelevant rows. Like book index: instead of reading every page to find "backpropagation," you look in index and jump to page 247.

Tradeoff: indexes speed reads but slow writes, because every INSERT/UPDATE must also update index. They also use storage. Practice rule: index columns you filter on (WHERE), join on (ON), or sort by (ORDER BY). Do not index everything.

To see how database plans to execute query:

EXPLAIN QUERY PLAN
SELECT * FROM experiments WHERE accuracy > 0.90;

This shows whether engine uses index or does full table scan. When query is slow, EXPLAIN is first diagnostic tool. I am still building intuition for complex query plans, but even basics — "is it scanning whole table or using index?" — save huge time.

SQL in the ML Workflow

Here all parts come together. In production ML system, SQL appears at nearly every stage:

Data extraction — pulling training data from warehouse. Often most complex SQL you write. Feature engineering queries join 10 tables, compute rolling windows, handle NULLs, output clean feature matrix.

WITH user_features AS (
    SELECT user_id,
           COUNT(*) AS total_orders,
           AVG(amount) AS avg_order_value,
           MAX(order_date) AS last_order_date,
           DATEDIFF('day', MAX(order_date), CURRENT_DATE) AS days_since_last
    FROM orders
    WHERE order_date >= DATEADD('month', -6, CURRENT_DATE)
    GROUP BY user_id
),
user_labels AS (
    SELECT user_id,
           CASE WHEN DATEDIFF('day', last_activity, CURRENT_DATE) > 30
                THEN 1 ELSE 0 END AS churned
    FROM user_activity
)
SELECT f.*, l.churned
FROM user_features f
JOIN user_labels l ON f.user_id = l.user_id;

This CTE computes features (order count, average value, recency) and joins them with binary churn label. Output goes straight into training pipeline. This is work SQL — not SELECT * FROM table, but multi-stage transformations producing ML-ready datasets.

Feature stores — systems like Feast or Tecton store precomputed features in tables that are essentially SQL-queryable. Knowing SQL lets you debug feature pipelines, trace data lineage, validate that features look correct.

Model monitoring — track prediction distributions, detect drift, compare model versions. All SQL against logging tables.

Experiment tracking — querying experiment metadata. "Which hyperparameter configurations achieved top-5 accuracy on this dataset?" That is SQL query against experiment tracker database.
