SQL

Chapter 1: Python & Programming Foundations 14 code blocks

SQL — Structured Query Language — is the language you use to talk to relational databases. It was born in the 1970s at IBM, standardized in 1986, and today it powers nearly every system that stores structured data at scale. Every ML pipeline that pulls training data from a warehouse, every feature store that serves real-time predictions, every analytics dashboard your stakeholders stare at — there's SQL underneath.

Why SQL Matters for ML

I'll be honest — I spent the first year of my data journey avoiding SQL. I had Pandas. I could load CSVs. Why would I need some ancient language from the 1970s that looked like someone tried to make English into a programming language and got stuck halfway?

Then I joined a team where the training data lived in a data warehouse with 400 million rows across 80 tables. My notebook that loaded everything into a DataFrame crashed. The senior engineer looked at me and said, "Write a query." That was the end of my avoidance. Here is that dive.

Before we start: we're going to build up from a tiny three-row table to production-grade patterns used in real ML feature pipelines. You don't need prior database experience. We'll add the concepts we need one at a time, with explanation.

This isn't a short journey, but I hope you'll be glad you came.

Tables: The Atomic Unit

A relational database stores data in tables. A table is a grid — rows and columns, like a spreadsheet, but with rules. Each column has a name and a type. Each row is one record. That's it.

Let's build a tiny one. Imagine we're tracking ML experiments:

CREATE TABLE experiments (
    id        INTEGER PRIMARY KEY,
    model     TEXT NOT NULL,
    accuracy  REAL,
    train_date DATE
);

INSERT INTO experiments VALUES (1, 'logistic_reg', 0.82, '2024-01-15');
INSERT INTO experiments VALUES (2, 'random_forest', 0.91, '2024-01-16');
INSERT INTO experiments VALUES (3, 'xgboost', 0.94, '2024-01-17');

We now have a table called experiments with four columns and three rows. The INTEGER PRIMARY KEY means each row gets a unique identifier — no two experiments share an id. The NOT NULL on model means you can't insert a row without specifying which model you ran. These are constraints — the rules that keep your data sane.

If a DataFrame is a table in memory, a database table is a table on disk — with guarantees about correctness, concurrency, and durability that a DataFrame will never give you.

Your First Query: SELECT

Reading data from a table is called a query. The command is SELECT.

SELECT model, accuracy FROM experiments;

model          | accuracy
---------------+---------
logistic_reg   | 0.82
random_forest  | 0.91
xgboost        | 0.94

We asked for two columns, and we got all three rows. The asterisk * means "everything":

SELECT * FROM experiments;

That gives you every column. In production, you almost never want * — you're pulling data across a network, and dragging 50 columns when you need 3 is wasteful. But for exploration, it's fine.

The mental model: SELECT doesn't change anything. It's a read operation. You can run it a thousand times and your data stays exactly the same. Think of it like shining a flashlight on a filing cabinet — you're looking at what's there, not moving it.

Filtering with WHERE

The WHERE clause is your filter. It lets you ask for only the rows that satisfy a condition.

SELECT model, accuracy
FROM experiments
WHERE accuracy > 0.90;

model          | accuracy
---------------+---------
random_forest  | 0.91
xgboost        | 0.94

The database engine scans the rows, checks the condition against each one, and only returns the ones that pass. You can combine conditions with AND and OR:

SELECT * FROM experiments
WHERE accuracy > 0.85 AND train_date >= '2024-01-16';

This gives us models that are both accurate and recent. The pattern is always the same: SELECT what you want, FROM where it lives, WHERE the conditions hold. That three-part structure is the spine of every SQL query you'll ever write.

Sorting and Limiting

Tables have no inherent order. The rows come back in whatever order the engine feels like unless you tell it otherwise.

SELECT model, accuracy
FROM experiments
ORDER BY accuracy DESC;

DESC means descending — best accuracy first. ASC (the default) goes the other way.

When you have millions of rows and want the top 10:

SELECT model, accuracy
FROM experiments
ORDER BY accuracy DESC
LIMIT 10;

This is the pattern you'll use constantly in ML work — "give me the best-performing experiments" or "show me the most recent 100 training runs." Without LIMIT, a query on a production table can return millions of rows and crash your notebook. I learned this the hard way.

Aggregations: GROUP BY

So far we've been looking at individual rows. But often you want summaries — averages, counts, maximums across groups. This is where GROUP BY enters.

Let's expand our table with more experiments:

INSERT INTO experiments VALUES (4, 'logistic_reg', 0.84, '2024-02-01');
INSERT INTO experiments VALUES (5, 'xgboost', 0.95, '2024-02-02');
INSERT INTO experiments VALUES (6, 'random_forest', 0.89, '2024-02-03');

Now we have multiple runs per model. What's the average accuracy for each?

SELECT model, AVG(accuracy) AS avg_acc, COUNT(*) AS num_runs
FROM experiments
GROUP BY model;

model          | avg_acc | num_runs
---------------+---------+---------
logistic_reg   | 0.83    | 2
random_forest  | 0.90    | 2
xgboost        | 0.945   | 2

GROUP BY model collapses all rows with the same model name into one group. Then AVG(accuracy) computes the mean within each group. COUNT(*) tells you how many rows fell into each bucket. The AS keyword gives the computed column a name — without it, you'd get something ugly like AVG(accuracy) as the column header.

The aggregate functions — COUNT, SUM, AVG, MIN, MAX — are the five you'll use 95% of the time. They take a column of many values and reduce it to one value per group.

Want to filter after grouping? That's HAVING:

SELECT model, AVG(accuracy) AS avg_acc
FROM experiments
GROUP BY model
HAVING AVG(accuracy) > 0.85;

WHERE filters individual rows before grouping. HAVING filters groups after aggregation. I confused these two for months. The key: if your condition involves an aggregate function, it goes in HAVING.

Rest Stop

Congratulations on making it this far. You can stop if you want.

You now have a mental model that handles maybe 70% of the SQL you'll encounter in the wild: select columns, filter rows, sort, limit, and aggregate into groups. That's enough to pull training data, compute basic feature statistics, and explore a new dataset you've been handed.

It doesn't tell the complete story. Real data lives across multiple tables — users in one, orders in another, products in a third. To combine them, you need JOINs. To write complex logic, you need subqueries and CTEs. To compute things like "running averages" or "rank within a group," you need window functions. These are what separate someone who can use SQL from someone who thinks in SQL.

The short version: JOINs let you combine tables by matching rows on a shared key. CTEs let you name intermediate results. Window functions let you compute aggregates without collapsing rows. There. You're 70% of the way there.

But if the discomfort of not knowing what's underneath is nagging at you, read on.

JOINs: Connecting Tables

Real databases don't store everything in one giant table. They split data across multiple tables and connect them with shared keys. This is normalization — and the reason for it is avoiding redundancy. If a user's name changes, you update it in one place, not in every row of every table that mentions them.

Let's add a second table:

CREATE TABLE datasets (
    id       INTEGER PRIMARY KEY,
    name     TEXT NOT NULL,
    num_rows INTEGER
);

INSERT INTO datasets VALUES (1, 'mnist', 60000);
INSERT INTO datasets VALUES (2, 'cifar10', 50000);
INSERT INTO datasets VALUES (3, 'imdb_reviews', 25000);

Now let's modify our experiments to reference which dataset they used:

ALTER TABLE experiments ADD COLUMN dataset_id INTEGER;
UPDATE experiments SET dataset_id = 1 WHERE id IN (1, 4);
UPDATE experiments SET dataset_id = 2 WHERE id IN (2, 5);
UPDATE experiments SET dataset_id = 3 WHERE id IN (3, 6);

The dataset_id column in experiments points to the id column in datasets. This is a foreign key — a reference from one table to another. Now we can combine them:

SELECT e.model, e.accuracy, d.name AS dataset, d.num_rows
FROM experiments e
JOIN datasets d ON e.dataset_id = d.id
WHERE e.accuracy > 0.90;

model          | accuracy | dataset      | num_rows
---------------+----------+--------------+---------
random_forest  | 0.91     | cifar10      | 50000
xgboost        | 0.94     | imdb_reviews | 25000
xgboost        | 0.95     | cifar10      | 50000

The JOIN ... ON clause says: "for each experiment row, find the dataset row where the IDs match, and glue them together side by side." The e and d are aliases — shorthand so you don't have to type the full table name every time.

There are several flavors of JOIN. The one above is an INNER JOIN — it only returns rows where a match exists in both tables. If an experiment had no dataset_id, it would be excluded. A LEFT JOIN keeps all rows from the left table and fills in NULLs where no match exists on the right. You'll reach for LEFT JOIN when you want to say "give me all experiments, and if they have a dataset, show it — otherwise leave it blank."

Subqueries and CTEs

Sometimes you need the result of one query to feed into another. You have two tools for this: subqueries (queries nested inside queries) and CTEs (Common Table Expressions — named intermediate results).

Subquery example — find experiments that beat the overall average:

SELECT model, accuracy
FROM experiments
WHERE accuracy > (SELECT AVG(accuracy) FROM experiments);

The inner query runs first, computes the average (0.89-ish), and the outer query uses that number as a filter. It works, but once you nest three or four of these, your query becomes unreadable. That's where CTEs save you.

WITH avg_baseline AS (
    SELECT AVG(accuracy) AS val FROM experiments
),
top_models AS (
    SELECT model, accuracy
    FROM experiments
    WHERE accuracy > (SELECT val FROM avg_baseline)
)
SELECT * FROM top_models ORDER BY accuracy DESC;

The WITH keyword introduces a CTE. Each CTE is a named temporary result that exists only for the duration of the query. You read it top to bottom: first compute the baseline, then find top models, then select from them. It's the same logic as nested subqueries, but laid out linearly so a human can follow it.

I'll be honest — I resisted CTEs for a while because subqueries felt more "compact." Then I came back to my own queries three weeks later and couldn't figure out what they did. CTEs are not about the database. They're about future-you being able to read your own work.

Window Functions

GROUP BY has a limitation: it collapses rows. If you want to compute an aggregate but keep the individual rows visible, you're stuck. Window functions solve this.

SELECT model, accuracy, train_date,
       RANK() OVER (ORDER BY accuracy DESC) AS rank,
       AVG(accuracy) OVER () AS global_avg
FROM experiments;

model          | accuracy | train_date | rank | global_avg
---------------+----------+------------+------+-----------
xgboost        | 0.95     | 2024-02-02 | 1    | 0.891
xgboost        | 0.94     | 2024-01-17 | 2    | 0.891
random_forest  | 0.91     | 2024-01-16 | 3    | 0.891
random_forest  | 0.89     | 2024-02-03 | 4    | 0.891
logistic_reg   | 0.84     | 2024-02-01 | 5    | 0.891
logistic_reg   | 0.82     | 2024-01-15 | 6    | 0.891

Every row is still there. But now each row also carries its rank and the global average. The OVER() clause defines the window — the set of rows the function looks at. An empty OVER() means "the entire table." Adding ORDER BY inside it makes the function compute a running calculation.

You can also partition windows:

SELECT model, accuracy, train_date,
       ROW_NUMBER() OVER (PARTITION BY model ORDER BY accuracy DESC) AS model_rank
FROM experiments;

This ranks experiments within each model separately. PARTITION BY splits the window into independent groups — like GROUP BY, but without collapsing. The common window functions you'll reach for: ROW_NUMBER(), RANK(), DENSE_RANK(), LAG(), LEAD(), and aggregates like SUM() OVER or AVG() OVER.

In ML feature engineering, window functions are indispensable. "Rolling 7-day average of user activity," "rank of this product within its category," "previous purchase amount" — all window functions.

Writing Data: INSERT, UPDATE, DELETE

We've been reading data. Now let's modify it.

INSERT adds new rows (you've already seen this). UPDATE modifies existing rows. DELETE removes them.

-- Fix a wrong accuracy
UPDATE experiments SET accuracy = 0.83 WHERE id = 1;

-- Remove experiments older than a date
DELETE FROM experiments WHERE train_date < '2024-01-16';

-- Insert with explicit columns (safer than relying on order)
INSERT INTO experiments (id, model, accuracy, train_date, dataset_id)
VALUES (7, 'neural_net', 0.96, '2024-02-10', 1);

A critical habit: always include a WHERE clause with UPDATE and DELETE. Running DELETE FROM experiments without a WHERE wipes every row in the table. No confirmation dialog. No undo. Gone. I've seen production data vanish this way. The database trusts that you meant what you said.

Schema Design and Normalization

How you structure your tables matters more than how you query them. A well-designed schema makes queries easy and data consistent. A bad schema makes every query a nightmare of self-joins and case statements.

The core principle of normalization is: every fact is stored exactly once. If a dataset's name appears in 10,000 experiment rows, and someone renames it, you'd have to update 10,000 rows. With normalization, the name lives in one place (the datasets table), and experiments reference it by ID.

The levels — first normal form, second, third, Boyce-Codd — are academic categories. What matters in practice: if you find yourself duplicating data across rows, pull it into its own table and reference it with a foreign key. If a column only makes sense for some rows but not others, it probably belongs in a related table.

For ML specifically, there's a tension. Normalized schemas are great for data integrity but require JOINs to reconstruct the full picture. Analytical queries often benefit from denormalized tables — wide tables with redundant data that can be scanned without joins. Data warehouses (BigQuery, Snowflake, Redshift) are designed for this pattern. You normalize for writes, denormalize for reads. Most ML feature tables are denormalized.

Indexes and Query Performance

When you run WHERE accuracy > 0.90 on three rows, the database checks each one. When you run it on 400 million rows, that's a problem. An index is a data structure (usually a B-tree) that lets the database jump directly to matching rows instead of scanning everything.

CREATE INDEX idx_experiments_accuracy ON experiments(accuracy);

Now queries filtering on accuracy can use this index to skip irrelevant rows. Think of it like a book index: instead of reading every page to find "backpropagation," you look it up in the index and jump to page 247.

The tradeoff: indexes speed up reads but slow down writes (because every INSERT/UPDATE must also update the index). They also consume storage. The rule in practice: index columns you frequently filter on (WHERE), join on (ON), or sort by (ORDER BY). Don't index everything.

To see how the database plans to execute your query:

EXPLAIN QUERY PLAN
SELECT * FROM experiments WHERE accuracy > 0.90;

This shows whether the engine is using an index or doing a full table scan. When a query is slow, EXPLAIN is your first diagnostic tool. I'm still developing my intuition for reading complex query plans, but even the basics — "is it scanning the whole table or using an index?" — save enormous amounts of time.

SQL in the ML Workflow

Here's where all of this comes together. In a production ML system, SQL appears at nearly every stage:

Data extraction — pulling training data from a warehouse. This is often the most complex SQL you'll write. Feature engineering queries that join 10 tables, compute rolling windows, handle NULLs, and output a clean feature matrix.

WITH user_features AS (
    SELECT user_id,
           COUNT(*) AS total_orders,
           AVG(amount) AS avg_order_value,
           MAX(order_date) AS last_order_date,
           DATEDIFF('day', MAX(order_date), CURRENT_DATE) AS days_since_last
    FROM orders
    WHERE order_date >= DATEADD('month', -6, CURRENT_DATE)
    GROUP BY user_id
),
user_labels AS (
    SELECT user_id,
           CASE WHEN DATEDIFF('day', last_activity, CURRENT_DATE) > 30
                THEN 1 ELSE 0 END AS churned
    FROM user_activity
)
SELECT f.*, l.churned
FROM user_features f
JOIN user_labels l ON f.user_id = l.user_id;

This CTE computes features (order count, average value, recency) and joins them with a binary churn label. The output goes straight into your training pipeline. This is the kind of SQL you write at work — not SELECT * FROM table, but multi-stage transformations that produce ML-ready datasets.

Feature stores — systems like Feast or Tecton store precomputed features in tables that are essentially SQL-queryable. Knowing SQL means you can debug feature pipelines, trace data lineage, and validate that features look correct.

Model monitoring — tracking prediction distributions, detecting drift, comparing model versions. All SQL against logging tables.

Experiment tracking — querying your experiment metadata. "Which hyperparameter configurations achieved top-5 accuracy on this dataset?" That's a SQL query against your experiment tracker's database.
