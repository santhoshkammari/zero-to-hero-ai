File I/O

Chapter 1: Python & Programming Foundations 5 concept areas

ML is 80% data plumbing. If you can't efficiently read configs, stream large files, and handle paths cleanly, you'll lose time every single day. This section isn't about open() — you know open() — it's about the patterns that keep your I/O fast, portable, and production-safe.

The Basics in 30 Seconds

open(), with statement, modes (r/w/a/rb/wb) — you know these. Two rules that matter:

  • Always use with (context manager) to auto-close. No exceptions. Leaked file descriptors pile up silently until your long-running training script dies with OSError: Too many open files.
  • Binary mode rb/wb for anything that isn't text. Model weights, images, pickle files, .npy arrays — text mode will corrupt them because it tries to decode bytes as UTF-8.
# The only pattern you should ever use
with open("data.csv") as f:
    content = f.read()
# f is closed here — guaranteed, even if an exception fired

# Binary for non-text
with open("model.pkl", "rb") as f:
    model = pickle.load(f)

pathlib — The Modern Way

Here's the thing about pathlib — once you start chaining paths with the / operator, os.path.join starts to look like a relic. It's object-oriented, chainable, cross-platform, and the code reads like English.

from pathlib import Path

# Building dataset paths — clean and readable
project = Path(__file__).parent
data_dir = project / "data" / "processed"
output_dir = project / "outputs" / "experiment_03"

# Create directories — one line, no errors if they exist
output_dir.mkdir(parents=True, exist_ok=True)

# Find all CSVs in nested directories
csv_files = list(data_dir.glob("**/*.csv"))
print(f"Found {len(csv_files)} CSV files")

# Process each file — .stem, .suffix, .parent all just work
for csv_path in csv_files:
    print(f"  {csv_path.stem} ({csv_path.suffix}) in {csv_path.parent.name}/")
    df = pd.read_csv(csv_path)
    out = output_dir / f"{csv_path.stem}_cleaned.parquet"
    df.to_parquet(out)

The key methods you'll use daily:

  • p / "subdir" / "file.txt" — the / operator joins paths. No more os.path.join(a, b, c).
  • p.glob("**/*.csv") — recursive file search. Returns a generator, so it's lazy on huge trees.
  • p.stem → "train", p.suffix → ".csv", p.name → "train.csv" — anatomy of a path, as attributes.
  • p.parent — go up one level. Chain it: p.parent.parent for two levels.
  • p.mkdir(parents=True, exist_ok=True) — creates the full directory tree. The two flags you always want.
  • p.read_text() / p.write_text(s) — one-liner for small files, no with needed.
  • Path(__file__).parent — resolve paths relative to the script, not the current working directory. This is the single biggest portability win.
Never hardcode paths. "/home/user/data" breaks on every other machine. "data/train.csv" breaks when run from a different directory. Path(__file__).parent / "data" / "train.csv" works everywhere — it resolves relative to where the script lives, not where you launched it from.

Config Files — JSON, YAML, TOML

Every ML project needs config files. The question is which format — and this is one of those genuine decision points where a table is the right tool:

FormatStrengthsWeaknessesUse When
JSON Built-in (json module), universal, strict syntax No comments, verbose, no trailing commas Data exchange, API responses, simple configs
YAML Human-friendly, comments, anchors & aliases for reuse Indentation-sensitive (bugs from tabs vs spaces), yaml.load() is a security hole Experiment configs, Kubernetes manifests, anything humans edit often
TOML Simple, readable, built-in since Python 3.11 (tomllib) Less expressive than YAML, no built-in write support yet pyproject.toml, simple project configs

The pattern you'll use most — loading experiment configs with YAML:

import yaml
from pathlib import Path

config_path = Path(__file__).parent / "configs" / "experiment.yaml"

# ALWAYS safe_load — never yaml.load() which executes arbitrary Python
with open(config_path) as f:
    config = yaml.safe_load(f)

lr = config["training"]["learning_rate"]      # 0.001
batch_size = config["training"]["batch_size"]  # 32
model_name = config["model"]["architecture"]   # "resnet50"
Never use yaml.load() — it can execute arbitrary Python code embedded in the YAML file. Always yaml.safe_load(). This isn't a style preference, it's a security vulnerability on the level of eval().

Handling Large Files

When your dataset doesn't fit in memory — and it will happen sooner than you think — you need to stop reading entire files at once. The good news: Python already has the right primitives.

Line-by-line reading is already lazy. This is the thing people miss — for line in file doesn't load the whole file. It reads one line at a time from the OS buffer. For text processing, this is your default pattern:

# Streams a 50GB log file without loading it all into memory
with open("training.log") as f:
    errors = [line for line in f if "ERROR" in line]

Streaming CSV with generators — when you need to process rows without materializing the whole dataset:

import csv

def stream_csv(path):
    """Yields one row at a time — constant memory regardless of file size."""
    with open(path) as f:
        reader = csv.DictReader(f)
        for row in reader:
            yield row

# Process 10M rows without loading them all
for row in stream_csv("huge_dataset.csv"):
    if float(row["score"]) > 0.95:
        process(row)

Memory-mapped files — when you need random access into a huge file without reading it all. The OS maps the file into virtual memory and pages in only what you access. This is how you'd read into a large binary file of pre-computed embeddings or model weights:

import mmap

with open("embeddings.bin", "rb") as f:
    mm = mmap.mmap(f.fileno(), 0, access=mmap.ACCESS_READ)
    # Read bytes 1000-2000 without loading the whole file
    chunk = mm[1000:2000]
    mm.close()

Binary serialization formats — for saving and loading Python objects:

  • pickle — fast, handles any Python object, but Python-only and a security risk (unpickling executes arbitrary code). Fine for internal checkpoints you control.
  • joblib — better than pickle for large numpy arrays (compressed, memory-mapped loading). The scikit-learn standard for saving models.
  • Never load untrusted pickle/joblib files — treat them like executables, because they are.
import joblib

# Save — compressed, efficient for numpy arrays
joblib.dump(trained_model, "model.joblib", compress=3)

# Load — can memory-map large arrays automatically
model = joblib.load("model.joblib", mmap_mode="r")

What You Should Now Be Able To Do

  • Use pathlib for all path operations — the / operator, .glob(), .stem, .parent, .mkdir() — and never go back to os.path.join
  • Build paths relative to Path(__file__).parent, never hardcoded, never dependent on cwd
  • Choose the right config format — JSON for data exchange, YAML for human-edited configs (safe_load only), TOML for project metadata
  • Stream large files without loading them into memory — for line in file, CSV generators, mmap for random access
  • Pick the right serialization — pickle for quick checkpoints, joblib for numpy-heavy models, and never load untrusted files from either
  • Explain why with is mandatory (resource leaks) and why binary mode matters (data corruption)
