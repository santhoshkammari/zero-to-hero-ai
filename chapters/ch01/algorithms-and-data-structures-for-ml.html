Nearest-neighbor search takes 40 minutes on 2 million embeddings. Feature store times out in production. Reason unclear. Then data structures and algorithms stop feeling academic. Start dive.

Algorithms and data structures hold up every ML system. Not whiteboard versions you code from scratch — versions already inside NumPy, PyTorch, FAISS, and feature stores. Understanding them separates "works on laptop" from "works for 100 million users."

Arrays and the Gospel of Contiguous Memory

Start with boring foundation. Tiny movie recommendation engine. Three movies. Five users. Each user rates each movie 1 to 5. Total: 15 numbers — matrix. Store as Python list of lists or NumPy array.

Thing that tripped me for years: math stays same. Algorithm stays same. Change is where those 15 numbers live in memory.

Python list of lists stores each number as separate Python object, scattered across memory like books on floor. Add two lists element-by-element, Python chases 15 pointers, unwraps 15 objects, does arithmetic, wraps 15 new objects, scatters them back. NumPy ndarray stores all 15 numbers in one contiguous block, like books on shelf. CPU sweeps through block with SIMD (Single Instruction, Multiple Data) instructions — multiple numbers per clock cycle.

import numpy as np

# Python list: each element is a pointer to an object scattered in memory
ratings_list = [[4, 5, 2], [3, 1, 4], [5, 5, 3], [2, 3, 5], [4, 2, 1]]

# NumPy array: one contiguous block of memory, 15 floats side by side
ratings = np.array(ratings_list, dtype=np.float32)

# This triggers optimized C code operating on that contiguous block
avg_per_movie = ratings.mean(axis=0)  # [3.6, 3.2, 3.0]

Last line runs roughly 100x faster than equivalent Python loop. Reason: math not smarter; contiguous memory lets hardware do hardware work. PyTorch tensors, TensorFlow tensors, JAX arrays — same principle. Contiguous array is ML bedrock. Everything else sits on top.

Contiguous arrays have limit. Great when data shape is known ahead of time and work is bulk numerical ops. Terrible for lookup like "what are user_42's features?" across 10 million users. Need different structure.

Hash Maps — The Backbone of Serving

Back to recommendation engine. Now 10 million users. Need any user's features in under millisecond for real-time serving. If features live in list, finding user_42 means scanning maybe all 10 million entries — O(n) time, n = user count. At 10 million users, latency dies.

Hash map (Python dict) fixes this. Compute hash of key — mathematical fingerprint — jump straight to right memory location. No scan. No search. O(1) average time, with 100 users or 100 million.

# Our recommendation engine's feature store
# O(1) lookup regardless of how many users exist
feature_store = {
    "user_42": {"avg_rating": 3.8, "genre_prefs": [0.9, 0.2, 0.7], "active_days": 145},
    "user_99": {"avg_rating": 2.1, "genre_prefs": [0.3, 0.8, 0.1], "active_days": 12},
    # ... imagine 10 million more entries
}

features = feature_store["user_42"]  # instant, regardless of store size

Pattern shows up everywhere in ML systems. NLP vocabulary mappings (token_to_index) are hash maps. Prediction caches are hash maps. Config lookups, deduplication checks, recommendation embedding tables mapping user ID to dense vector — hash maps. Production feature stores like Feast, Tecton, Uber's Michelangelo are distributed hash maps with persistence bolted on.

I took O(1) dictionary lookups for granted for years. Production made lesson real. Dict-backed feature store served 50,000 requests per second. Naive list-backed version barely hit 200.

Hash maps shine for exact lookups. Different question: not "give me user_42's data" but "give me 10 users most similar to user_42." Need trees.

Trees — Three Different Lives in ML

Trees show up in three separate ML contexts. Mix them up, get confused. Untangle them.

Decision trees as models. In Random Forest, XGBoost, and LightGBM, tree is model. Each internal node asks feature question ("is user's average rating above 3.5?"), each branch is answer, each leaf holds prediction. These models still dominate tabular ML in production. In recommendation engine, if task is predicting clicks from structured features like age, watch history, time of day, tree-based models are often first thing to try and often last thing you need.

B-trees in databases. When training pipeline queries PostgreSQL for 50 million user-movie interaction rows and result returns in seconds, B-tree index did work. B-trees keep data sorted in balanced structure, giving O(log n) lookups. Without index, database scans every row — O(n). With B-tree index on user_id, finding all interactions for user_42 is nearly instant.

KD-trees for nearest-neighbor search. Story gets interesting here — and painful. KD-tree (k-dimensional tree) partitions space recursively, splitting along one dimension at time. sklearn.neighbors.KNeighborsClassifier uses this under hood. In recommendation engine, if each movie is point in 3-dimensional space (genre scores), KD-tree finds 10 nearest movies to query movie in O(log n) time. Beautiful.

Catch exists. Took me embarrassingly long to learn.

The Curse of Dimensionality and the Death of KD-Trees

I once used KD-tree for nearest neighbors on 768-dimensional sentence embeddings. Worked. Still slower than brute-force linear scan. I thought code was wrong. Code was fine. KD-tree was collapsing in high dimensions.

Problem: KD-tree recursively splits space one dimension at time. In 3 dimensions, each split removes roughly half remaining points. As dimensions rise, each split says less. By 100+ dimensions, splits are almost random — tree searches nearly every branch, query time degrades from O(log n) toward O(n). That is curse of dimensionality: in high-dimensional spaces, points become roughly equidistant, spatial partitioning stops working.

Practical threshold is low. KD-trees work well up to roughly 20-30 dimensions. Modern ML embeddings — 300-dimensional word embeddings, 768-dimensional sentence embeddings, 2048-dimensional image embeddings — exceed that by order of magnitude.

So tool every intro ML course teaches for nearest-neighbor search is mostly useless for nearest-neighbor search you need in practice. Frustrating. Also reason production ML built major new algorithm families.

Approximate Nearest Neighbors and HNSW

Key insight for high-dimensional similarity search: exact nearest neighbor not required. You need very close neighbor, fast. Tiny accuracy loss for huge speedup is usually right trade.

Several algorithms use this tradeoff. Locality-Sensitive Hashing (LSH) uses hash functions designed so similar items land in same bucket — opposite of cryptographic hashes. Spotify's Annoy builds forests of random projection trees. 2024 industry standard is HNSW — Hierarchical Navigable Small World graphs.

Analogy that made HNSW click: navigating city. Need get from one side of city to one address. Start on highway — long-range links, few exits, fast travel. Reach right neighborhood, drop to main road. Then side street. Then exact address. Speed comes from moving across scales.

HNSW builds exactly that. It makes multi-layer graph where each node is data point (embedding vector). Top layers are sparse — few nodes, long-range links, like highways. Lower layers get denser, with more nodes and shorter-range links. Bottom layer contains every point.

To search, enter top layer and greedily walk toward nodes closer to query vector. When no closer node exists in current layer, drop to next layer. Reach bottom already in right neighborhood, so final search is fast.

I am still building full intuition for why layered structure works so well — theory involves random graph properties and "small world" network behavior. Practical result is huge: HNSW finds approximate nearest neighbors in 100 million 768-dimensional vectors in single-digit milliseconds. This powers Pinecone, Milvus, Redis vector search, Elasticsearch vector search. FAISS uses it as go-to production index.

Back to recommendation engine: once each movie becomes 256-dimensional embedding, HNSW finds "movies similar to this one" at serving time across millions of items in milliseconds. Without it, modern embedding-based recommendation and retrieval would not exist at production scale.

HNSW is powerful, but narrow. It solves similar-item search. ML systems have other structural needs. Need different connection model.

Graphs — Autograd Is a Graph Algorithm

Once this clicked, it stayed: calling loss.backward() in PyTorch runs graph algorithm. Not metaphor. Literal fact.

Every PyTorch tensor operation — addition, multiplication, activation — builds directed acyclic graph (DAG) called computation graph. Tensors are nodes. Operations are edges. Call .backward(), PyTorch does reverse topological sort on graph and walks backward, applying chain rule at each node to compute gradients.

Topological sort is ordering of directed-graph nodes where every node comes after all nodes it depends on. Picture recipe: cannot frost cake before baking, cannot bake before mixing batter. Order "mix → bake → frost" is topological sort of dependency graph. Reverse it, get backprop order: start at loss (frosted cake), trace backward through each operation, measure how each ingredient (parameter) contributed.

import torch

# Every operation here builds a computation graph
x = torch.tensor([2.0], requires_grad=True)
y = x ** 2 + 3 * x + 1  # graph: x → x² → +3x → +1 → y

# .backward() does a reverse topological sort of this graph
# then computes gradients via chain rule
y.backward()
print(x.grad)  # tensor([7.]) — dy/dx = 2x + 3 = 7

Graphs show up elsewhere too. Airflow and Kubeflow run ML pipelines as DAGs — each node is task (data ingestion, feature engineering, training, evaluation), edges encode dependencies. Knowledge graphs store entity relationships for recommendation and search. Graph Neural Networks (GNNs) work directly on graph-structured data — social networks, molecules, citation networks — where nodes pass messages to neighbors to learn representations. GNN message passing is graph algorithm: each node aggregates neighbor info, updates state, repeats for several rounds.

Graphs give language for relationships and dependencies. Sometimes question is not relationship but selection: "out of million candidates, which 10 are best?" Heaps handle that.

Heaps, Priority Queues, and Beam Search

Back to recommendation engine. We scored all 5 million movies for user_42. Now need top 10. Naive move: sort all 5 million scores, take first 10. Cost: O(n log n) — about 112 million operations for n = 5 million.

Heap (priority queue) does this in O(n log k), where k is item count we want. Here, about 17 million operations — roughly 6x faster. Difference: heap keeps small sorted structure of size k, scans data, works only when new item beats current worst heap item.

import heapq

# 5 million movie scores (simulated)
import random
scores = [random.random() for _ in range(5_000_000)]

# Top 10 without sorting all 5 million
top_10 = heapq.nlargest(10, scores)  # O(n log k), not O(n log n)

Heaps become essential in ML during beam search. When LLM generates text, each step produces probability distribution over whole vocabulary often 50,000+ tokens. Beam search keeps top-k most promising partial sequences at each step, expands them, keeps best k again. That repeated "keep best k" step is heap work. Without heaps, beam search must sort whole vocabulary at every step — much more expensive.

NumPy has even more aggressive top-K optimization: np.argpartition. It finds top-k indices in O(n) average time with selection algorithm related to quickselect. It does not return sorted items — it partitions array so top-k items sit on one side — but that is often enough.

Heaps solve "find best few." Related problem is subtler: "have I seen this before?" When "this" means one of billion training documents, even hash maps cost too much memory. Need something stranger.

Tries — What's Inside Your Tokenizer

Every time you run sentence through language model, first step is tokenization — break text into subword tokens. If model uses BPE (Byte Pair Encoding), tokenizer must find longest matching token from vocabulary at every input position. With 50,000-token vocabulary, scanning whole vocabulary at every position is ruinously slow.

Trie (pronounced "try," short for retrieval) is tree where each node represents character, and root-to-marked-node paths spell valid entries. Tokens with shared prefix share initial path. To find longest matching token at any position, walk trie character by character until path ends, then take last valid token seen.

Imagine recommendation engine now includes natural-language search: users type "movies like Inception but darker." Tokenizer breaks text into tokens. With trie-backed vocabulary, each token lookup takes O(L) time where L is token length in characters — independent of vocabulary size. Naive list scan takes O(V), V = vocabulary size. At 50,000 tokens, difference hurts.

HuggingFace tokenizers library implements BPE tokenizer with trie written in Rust. Not structure you usually build yourself. Still worth knowing it is there. It explains why tokenization is fast for some models, slow for others, and why vocabulary size matters beyond model quality.

Dynamic Programming — Viterbi to Beam Search

Dynamic programming (DP) is not data structure — it is technique. Break problem into overlapping subproblems. Cache results. It belongs here because it powers two important ML decoding algorithms.

Viterbi algorithm is classic DP applied to sequence models. In Hidden Markov Model, common in speech recognition before deep learning took over, goal is most likely hidden-state sequence given observed sequence. Trying every path is exponential. Viterbi uses fact that best path to any state at time t depends only on best paths to all states at time t-1. It builds table of partial solutions moving forward through time, then traces back to recover optimal path. Complexity drops from exponential to O(T × S²), where T is sequence length and S is hidden-state count.

Beam search replaced Viterbi when neural networks made state space too large for exact DP. Instead of tracking best path to every possible state, beam search tracks top-k partial sequences at each step and expands from there. It is approximate — no guarantee of globally optimal sequence — but practical. Every LLM you use generates text with some variant of beam search or close relatives like greedy decoding and nucleus sampling.

Connection: Viterbi is exact DP. Beam search is approximate DP with fixed-width frontier. That lineage explains why beam width is quality-versus-speed tradeoff, and why increasing beam width eventually stops helping — beam is still approximating exponentially large space.

Bloom Filters — Deduplicating a Billion Documents

Idea behind Bloom filter felt wrong first time I saw it. Data structure that says "item definitely NOT in set" but only "item PROBABLY in set"? Why want that?

Reason: large language model training means billions of web-crawled documents. Duplicates make model memorize passages, hurt generalization, waste compute. For each incoming document, you need seen-it-before check. Storing hashes for billion documents in normal hash set costs tens of gigabytes. Bloom filter stores same membership signal in fraction of space — price is occasional false positive, saying "yes, duplicate" when it is not. But it never says "no, new" when document is actual duplicate — no false negatives.

Mechanically, Bloom filter is bit array. Add item: hash it with several hash functions, set corresponding bit positions to 1. Check membership: hash item again, test whether all corresponding positions are 1. If any position is 0, item is definitely new. If all are 1, item is probably duplicate — bits may have been set by other items.

This is how OpenAI, Google, and others deduplicate training corpora at scale. False positive rate is tunable: more bits and more hash functions reduce false positives. Bloom filter with 10 bits per element and 7 hash functions stays under 1% false positives. For deduplication, that trade is fine — lose few unique documents, catch nearly all duplicates, use tiny fraction of hash-set memory.

Algorithmic Complexity — The One Skill That Prevents Production Fires

Direct confession: I once shipped code with O(n²) pairwise distance computation hidden inside feature-engineering step. Test dataset had 5,000 items, so run finished in 2 seconds. Production dataset had 200,000 items, so run took over 22 hours. Complexity explained disaster.

Idea behind Big-O notation looks simple. It describes how time or memory grows with input size, ignoring constant factors. O(n) means doubling input roughly doubles time. O(n²) means doubling input roughly quadruples time. O(log n) means doubling input adds one extra step.

Practical question Big-O answers: "If data grows 10x, how much longer will this take?"

ComplexityNameML Example10x Data →
O(1) Constant Dict lookup in a feature store Same time
O(log n) Logarithmic B-tree index lookup, bisect on sorted data One more step
O(n) Linear Single pass over a dataset, vectorized NumPy op 10x time
O(n log n) Linearithmic Sorting predictions for ranking, AUC-ROC ~13x time
O(n²) Quadratic Pairwise distances, full self-attention 100x time
O(n³) Cubic Naive matrix multiplication, matrix inversion 1000x time

Last row — O(n³) — explains why matrix multiplication is hot topic in ML infrastructure. Standard multiplication of two n×n matrices needs n³ multiplications. Strassen's algorithm cuts that to about O(n^2.81), and optimized BLAS libraries use it for large matrices. Difference matters: for n = 10,000, normal deep-learning size, gap between n³ and n^2.81 saves billions of operations.

Practical complexity stance differs for offline and online work. Offline (training, batch jobs): O(n log n) is almost always fine. O(n²) works until roughly 50,000 items. Beyond that, rethink approach. Online (inference, serving): need O(1) or O(log n) per request. O(n) per request is tolerable only when n is small — hundreds, not millions. If serving path contains O(n²), scaling fire is waiting.

The Quadratic Wall of Self-Attention

Most consequential O(n²) operation in modern ML is transformer self-attention. Standard attention computes relevance score between every pair of tokens in input sequence. If sequence has n tokens, attention matrix is n × n — O(n²) time and memory.

For 512 tokens, that is about 260,000 pairwise computations per attention head — manageable. For 4,096 tokens, nearly 17 million. For 32,000 tokens, context length for GPT-4-level models, over billion. Quadratic growth is core reason long context windows cost so much, and why extending context length is active research.

Several "efficient attention" families try to break quadratic barrier. Longformer uses sparse attention — each token attends to local window plus few global tokens — reaching O(n). Performer uses kernel approximations to decompose attention matrix without computing it fully, also O(n). Reformer uses locality-sensitive hashing from ANN section to group similar tokens and compute attention only within groups, reaching O(n log n).

None fully replace standard attention in practice — quality tradeoffs remain — but reason they exist is quadratic wall. Reason quadratic wall matters is Big-O. That is why path to this section mattered.

Top-K Selection — Why You Should Stop Sorting

One more algorithmic pattern shows up constantly. In recommendation engine, after scoring 5 million movies, we need top 10. In beam search, each step needs top-k sequences from expanded candidate set. In evaluation, Recall@K, Precision@K, NDCG@K all need top-K items.

Instinct is sort. Sorting gives top-K in sorted order, and it works. But sorting is O(n log n). If you only need top-K, you can do better.

NumPy np.argpartition uses selection algorithm related to quickselect, running in O(n) average time. It does not sort array — it rearranges array so K largest items sit at end in arbitrary order. If you also need sorted order, sort only those K items afterward — O(K log K), negligible when K is small.

import numpy as np

scores = np.random.rand(5_000_000)  # 5 million movie scores

# Full sort: O(n log n) ≈ 112 million operations
top_10_sorted = np.argsort(scores)[-10:][::-1]

# Partial selection: O(n) ≈ 5 million operations, then sort 10 items
top_10_idx = np.argpartition(scores, -10)[-10:]
top_10_idx = top_10_idx[np.argsort(scores[top_10_idx])[::-1]]

PyTorch torch.topk and TensorFlow tf.math.top_k do same operation on GPU. Critical for beam search and ranking work inside training loop. Pattern is universal: if you need only K items, do not sort n.

I still catch myself writing sorted(scores)[-10:] from habit. Every time, I think about 5 million unnecessary comparisons.
