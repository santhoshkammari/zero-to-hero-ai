Data Visualization

Chapter 1: Python & Programming Foundations Moderate Depth

You visualize data for two reasons: to understand it yourself, and to convince someone else. The choice of plot matters more than making it pretty. This section is about choosing the right visualization, not memorizing matplotlib kwargs.

Which Plot When

This is the core decision table. Given what you want to see, pick the right plot — not the one you happen to remember.

What You Want to SeePlot TypeLibrary / Function
Distribution of a featureHistogram / KDEplt.hist() / sns.kdeplot()
Feature vs targetScatterplt.scatter() / sns.scatterplot()
Category comparisonBar chartsns.barplot()
Correlation between featuresHeatmapsns.heatmap(df.corr())
Feature over time / epochsLine plotplt.plot()
Model performance breakdownConfusion matrixsns.heatmap() / ConfusionMatrixDisplay
Distribution across categoriesBox / Violinsns.boxplot() / sns.violinplot()
All feature relationships at oncePair plotsns.pairplot(df)

That's it. Eight plot types cover 95% of what you'll ever need. The skill isn't knowing more plot types — it's picking the right one on the first try.

Matplotlib — The Foundation

Here's the honest truth about matplotlib — it's the assembly language of Python visualization. Verbose, explicit, powerful. You need to know it because everything else is built on top of it, but you'll use seaborn for most of your actual work.

The one pattern that matters: always use the OO interface.

# ✅ This — explicit axes, composable, works everywhere
fig, ax = plt.subplots()
ax.plot(epochs, losses)
ax.set_title("Training Loss")

# ❌ Not this — hidden global state, breaks in loops and functions
plt.plot(epochs, losses)
plt.title("Training Loss")

Why? Because fig, ax = plt.subplots() gives you an object you can pass around, customize, and save. The plt.plot() style maintains hidden "current axes" state that silently breaks the moment you have subplots, loops, or functions that create figures.

The pattern you'll use constantly — a 2×2 diagnostic grid:

import matplotlib.pyplot as plt
import seaborn as sns

fig, axes = plt.subplots(2, 2, figsize=(12, 10))

# Each axes[row, col] is an independent plot
sns.histplot(df, x="feature_1", kde=True, ax=axes[0, 0])
sns.heatmap(df.corr(), annot=True, fmt=".2f", ax=axes[0, 1])
sns.boxplot(data=df, x="target", y="feature_2", ax=axes[1, 0])
sns.scatterplot(data=df, x="feature_1", y="feature_2",
                hue="target", ax=axes[1, 1])

plt.tight_layout()
fig.savefig("eda_overview.png", dpi=150, bbox_inches="tight")
plt.close(fig)  # prevent memory leaks in loops/scripts

Seaborn — The 80% Solution

Seaborn handles 80% of your visualization needs in one line. It's built on matplotlib, understands DataFrames natively, and makes statistical plots that would take 20 lines of matplotlib into single function calls.

The three seaborn calls that cover most of your EDA:

import seaborn as sns

# 1. Instant overview of all feature relationships
sns.pairplot(df, hue="species")  # colored by target class

# 2. Correlation analysis in one line
sns.heatmap(df.corr(), annot=True, cmap="coolwarm", center=0)

# 3. Make everything look decent immediately
sns.set_theme()  # call once, every plot looks better

That's it. You don't need to memorize 50 seaborn functions. pairplot for exploration, heatmap for correlations, and set_theme() for not looking like it's 2003. When you need something specific — a violin plot, a KDE, a count plot — look it up. The API is consistent enough that if you've used one, you've used them all.

Plotly — When You Need Interactive

When your Jupyter notebooks need to be shared with non-technical stakeholders, Plotly makes charts interactive — zoom, hover, filter — with almost the same API as matplotlib. It's also the backbone of Dash for building data apps. You don't need it for your own exploration, but it's the right tool when someone else needs to explore your results.

Production Patterns

Visualization in notebooks is exploration. Visualization in production is different.

# Saving figures properly — the one line everyone forgets
fig.savefig("plot.png", dpi=150, bbox_inches="tight")
# bbox_inches="tight" prevents labels from being clipped
# Use PDF/SVG for papers, PNG for dashboards and Slack

Reproducibility: set random seeds before any plot that involves sampling or shuffling, and save the exact code that produced each figure. Six months from now, your manager will ask "can you regenerate that chart?" and you'll be glad you did.

If you're using MLflow or Weights & Biases, log figures automatically during training — mlflow.log_figure(fig, "confusion_matrix.png") or wandb.log({"chart": fig}). Manual screenshot-and-paste doesn't scale past your first project.

⚠️ Memory leak in loops

Each unclosed figure leaks memory. In notebooks, plt.show() auto-closes. In production scripts and loops, always call plt.close(fig) after saving. Generating 1000 figures without closing them will crash your process.

What You Should Now Be Able To Do

  • Given a data question, pick the right plot type from the decision table on the first try
  • Use the OO matplotlib interface (fig, ax = plt.subplots()) and explain why you'd never use plt.plot() directly
  • Build a 2×2 EDA grid — distribution, correlation heatmap, boxplot, scatter — for any new dataset
  • Use seaborn's pairplot and heatmap for instant exploratory analysis
  • Save figures properly with bbox_inches="tight" and plt.close() to avoid memory leaks
