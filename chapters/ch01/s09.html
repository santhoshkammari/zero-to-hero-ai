Error Handling

Chapter 1: Python & Programming Foundations 3 subtopics

TL;DR

Errors are inevitable — network calls fail, users provide garbage input, files go missing. Python's exception system gives you structured tools to detect, handle, and recover from errors without your program crashing. Master try/except/else/finally, learn to raise meaningful custom exceptions, and adopt the EAFP philosophy. Good error handling is the difference between a script that works on your laptop and a system that survives production.

Why Error Handling Matters

Your ML training pipeline runs for six hours. At hour five, it tries to save a checkpoint and the disk is full. Without error handling, the program crashes. All five hours of work are gone. No log tells you what happened.

With proper error handling, you catch the disk-full error, log it, save the checkpoint to a backup location, and the pipeline continues. Or at minimum, you get a clear message: "Disk full while saving checkpoint at epoch 47. Last successful save: epoch 46."

Error handling is not about preventing errors. It is about controlling what happens when things go wrong.

Exceptions

What Are Exceptions?

An exception is an event that disrupts the normal flow of a program. When Python encounters an error, it creates an exception object and "raises" it. If nothing catches the exception, the program terminates with a traceback.

# These all raise exceptions:
# int("hello")      → ValueError
# 1 / 0             → ZeroDivisionError
# [1, 2, 3][10]     → IndexError
# {"a": 1}["b"]     → KeyError
# open("nope.txt")  → FileNotFoundError

# The traceback tells you what went wrong and where:
def divide(a, b):
    return a / b

def calculate():
    return divide(10, 0)

# calculate()
# Traceback (most recent call last):
#   File "...", line X, in calculate
#     return divide(10, 0)
#   File "...", line X, in divide
#     return a / b
# ZeroDivisionError: division by zero

Common Exception Types

Python has a rich hierarchy of built-in exceptions. Here are the ones you will encounter most often:

# ValueError — right type, wrong value
int("hello")           # ValueError: invalid literal for int()
float("not_a_number")  # ValueError

# TypeError — wrong type entirely
"hello" + 5            # TypeError: can only concatenate str to str
len(42)                # TypeError: object of type 'int' has no len()

# IndexError — sequence index out of range
[1, 2, 3][10]          # IndexError: list index out of range

# KeyError — dictionary key not found
{"a": 1}["b"]          # KeyError: 'b'

# FileNotFoundError — file doesn't exist
open("ghost.txt")      # FileNotFoundError

# AttributeError — object doesn't have that attribute
"hello".append("!")    # AttributeError: 'str' object has no attribute 'append'

# ImportError — module not found
# import nonexistent_module  # ModuleNotFoundError (subclass of ImportError)

# StopIteration — iterator is exhausted
it = iter([1])
next(it)  # 1
# next(it)  # StopIteration

The try/except Block

The try/except statement lets you catch and handle exceptions:

# Basic try/except
def safe_divide(a, b):
    try:
        result = a / b
    except ZeroDivisionError:
        print("Cannot divide by zero!")
        result = float('inf')
    return result

print(safe_divide(10, 3))   # 3.333...
print(safe_divide(10, 0))   # Cannot divide by zero! → inf

Catching Specific Exceptions

Always catch specific exceptions. Catching everything hides bugs:

def parse_config(raw):
    """Parse a config string like 'key=value'."""
    try:
        key, value = raw.split("=")
        return {key.strip(): value.strip()}
    except ValueError:
        print(f"Bad format: {raw!r} (expected 'key=value')")
        return {}

print(parse_config("name=Alice"))       # {'name': 'Alice'}
print(parse_config("broken"))           # Bad format: 'broken'
print(parse_config("a=b=c"))            # Bad format: 'a=b=c'

Multiple except Clauses

You can handle different exceptions differently:

def load_setting(config, key, default=None):
    """Safely load a setting from a config dict."""
    try:
        value = config[key]
        return int(value)
    except KeyError:
        print(f"Key '{key}' not found, using default={default}")
        return default
    except (ValueError, TypeError) as e:
        print(f"Cannot convert '{config[key]}' to int: {e}")
        return default

config = {"port": "8080", "workers": "abc", "debug": "1"}
print(load_setting(config, "port"))       # 8080
print(load_setting(config, "workers"))    # Cannot convert... → None
print(load_setting(config, "missing", 0)) # Key not found → 0

The else and finally Clauses

The full try statement has four parts:

def read_first_line(filepath):
    """Demonstrate try/except/else/finally."""
    try:
        f = open(filepath, "r")
    except FileNotFoundError:
        print(f"File not found: {filepath}")
        return None
    else:
        # Runs ONLY if no exception was raised in try
        content = f.readline().strip()
        print(f"Read: {content!r}")
        return content
    finally:
        # Runs ALWAYS — whether exception or not
        print("Cleanup: closing resources")
        # In real code you'd close the file here,
        # but 'with' statements handle this better

# Example
read_first_line("nonexistent.txt")
# File not found: nonexistent.txt
# Cleanup: closing resources
Key Insight: The else block runs only when the try block succeeds. Put your "happy path" code there instead of inside the try. This way, the try block is minimal — it covers only the line that might fail. The finally block is guaranteed to run, even if there is a return statement earlier. It is your last chance to clean up resources. 
# Real-world pattern: database transaction
def save_record(db, record):
    """Save with proper error handling."""
    connection = db.connect()
    try:
        connection.begin_transaction()
        connection.insert(record)
    except db.IntegrityError as e:
        connection.rollback()
        print(f"Duplicate record: {e}")
        return False
    except db.ConnectionError as e:
        connection.rollback()
        print(f"Database unreachable: {e}")
        return False
    else:
        connection.commit()
        print("Record saved successfully")
        return True
    finally:
        connection.close()  # Always close, no matter what

Raising and Custom Exceptions

The raise Statement

You raise exceptions when your code detects an error condition. This is how you tell callers "something is wrong, deal with it":

def set_learning_rate(lr):
    """Validate and set learning rate."""
    if not isinstance(lr, (int, float)):
        raise TypeError(f"Learning rate must be numeric, got {type(lr).__name__}")
    if lr <= 0:
        raise ValueError(f"Learning rate must be positive, got {lr}")
    if lr > 1:
        raise ValueError(f"Learning rate too large: {lr} (max 1.0)")
    return lr

# These work:
print(set_learning_rate(0.001))  # 0.001

# These raise clear errors:
# set_learning_rate("fast")  # TypeError: ... must be numeric, got str
# set_learning_rate(-0.01)   # ValueError: ... must be positive, got -0.01
# set_learning_rate(5.0)     # ValueError: ... too large: 5.0 (max 1.0)

Creating Custom Exception Classes

For larger projects, custom exceptions make error handling precise and self-documenting:

class ModelError(Exception):
    """Base exception for our ML framework."""
    pass

class TrainingError(ModelError):
    """Something went wrong during training."""
    def __init__(self, message, epoch=None, metric=None):
        self.epoch = epoch
        self.metric = metric
        super().__init__(message)

class DataError(ModelError):
    """Data validation failed."""
    def __init__(self, message, column=None, row=None):
        self.column = column
        self.row = row
        super().__init__(message)

class ConfigError(ModelError):
    """Invalid configuration."""
    pass

# Usage
def validate_data(X, y):
    if len(X) != len(y):
        raise DataError(
            f"Feature/label mismatch: {len(X)} samples vs {len(y)} labels",
            row=len(X)
        )
    if len(X) == 0:
        raise DataError("Empty dataset", row=0)

def train(model, X, y, epochs=10):
    validate_data(X, y)
    for epoch in range(epochs):
        loss = model.step(X, y)
        if loss != loss:  # NaN check
            raise TrainingError(
                "Loss became NaN",
                epoch=epoch,
                metric=loss
            )

# Catching custom exceptions
try:
    validate_data([1, 2], [1])
except DataError as e:
    print(f"Data problem at row {e.row}: {e}")
except ModelError as e:
    print(f"General model error: {e}")
Key Insight: Custom exceptions form a hierarchy. Catching ModelError catches all three subclasses. Catching TrainingError catches only training problems. This lets callers choose their level of granularity. Every serious library — requests, SQLAlchemy, PyTorch — defines custom exception hierarchies.

Exception Chaining

When you catch one exception and raise another, use from to preserve the original cause:

import json

def load_config(filepath):
    """Load JSON config with clear error chain."""
    try:
        with open(filepath) as f:
            return json.load(f)
    except FileNotFoundError as e:
        raise ConfigError(f"Config file missing: {filepath}") from e
    except json.JSONDecodeError as e:
        raise ConfigError(f"Invalid JSON in {filepath}") from e

# The traceback shows both exceptions:
# ConfigError: Config file missing: settings.json
#   caused by:
# FileNotFoundError: [Errno 2] No such file or directory: 'settings.json'

Best Practices

EAFP vs LBYL

Two philosophies for handling potential errors:

LBYL (Look Before You Leap) — check conditions before acting:

# LBYL style
def get_value_lbyl(data, key):
    if key in data:
        return data[key]
    else:
        return None

EAFP (Easier to Ask Forgiveness than Permission) — try it, catch the error:

# EAFP style — Pythonic
def get_value_eafp(data, key):
    try:
        return data[key]
    except KeyError:
        return None

Python strongly favors EAFP. It is faster when the key usually exists (no double lookup), it avoids race conditions, and it is more readable in complex scenarios:

# LBYL becomes verbose quickly
def lbyl_approach(config):
    if isinstance(config, dict):
        if "database" in config:
            if isinstance(config["database"], dict):
                if "host" in config["database"]:
                    return config["database"]["host"]
    return "localhost"

# EAFP is cleaner
def eafp_approach(config):
    try:
        return config["database"]["host"]
    except (KeyError, TypeError):
        return "localhost"
Watch Out: EAFP is not an excuse to use exceptions for flow control. If you expect an error on most calls (not the exceptional case), LBYL might be faster and clearer. The rule: use EAFP when the error is truly exceptional.

Avoiding Bare except

# NEVER do this:
try:
    result = risky_operation()
except:  # Bare except — catches EVERYTHING
    pass  # Silently swallows all errors, including KeyboardInterrupt

# This hides bugs, makes debugging impossible, and can catch
# things like SystemExit and KeyboardInterrupt.

# DO this instead:
try:
    result = risky_operation()
except Exception as e:
    print(f"Operation failed: {e}")
    # Log it, re-raise it, or handle it — but NEVER ignore it

# If you must catch and re-raise:
try:
    result = risky_operation()
except Exception:
    logging.exception("Unexpected error")  # Logs full traceback
    raise  # Re-raise the original exception
Critical: A bare except: with pass is the most dangerous pattern in Python. It turns errors into silent, invisible bugs that surface days or weeks later in completely unrelated places. Always catch specific exceptions. Always do something with the error.

Logging Exceptions

import logging

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)

def process_batch(items):
    results = []
    errors = []
    for i, item in enumerate(items):
        try:
            result = transform(item)
            results.append(result)
        except Exception as e:
            # Log with full traceback but continue processing
            logger.error(f"Failed on item {i}: {item!r}", exc_info=True)
            errors.append((i, item, str(e)))

    logger.info(f"Processed {len(results)}/{len(items)} items "
                f"({len(errors)} errors)")
    return results, errors

def transform(item):
    """Simulate transformation that sometimes fails."""
    if not isinstance(item, (int, float)):
        raise TypeError(f"Expected numeric, got {type(item).__name__}")
    return item ** 2

results, errors = process_batch([1, 2, "bad", 4, None, 6])
print(f"Results: {results}")   # [1, 4, 16, 36]
print(f"Errors: {errors}")     # [(2, 'bad', ...), (4, None, ...)]

Context Managers for Resource Cleanup

from contextlib import contextmanager

@contextmanager
def managed_resource(name):
    """Generic resource manager with error handling."""
    print(f"Acquiring {name}")
    resource = {"name": name, "active": True}
    try:
        yield resource
    except Exception as e:
        print(f"Error in {name}: {e}")
        raise  # Re-raise after logging
    finally:
        resource["active"] = False
        print(f"Released {name}")

# Usage
with managed_resource("database") as db:
    print(f"Using {db['name']}")
    # If an exception occurs here, the resource is still released

# Acquiring database
# Using database
# Released database

Putting It All Together: Robust Data Pipeline

import logging
from contextlib import contextmanager

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger("pipeline")

class PipelineError(Exception):
    """Base pipeline exception."""
    pass

class ValidationError(PipelineError):
    def __init__(self, message, record=None):
        self.record = record
        super().__init__(message)

def validate_record(record):
    if not isinstance(record, dict):
        raise ValidationError(f"Expected dict, got {type(record).__name__}", record)
    required = ["id", "value"]
    missing = [f for f in required if f not in record]
    if missing:
        raise ValidationError(f"Missing fields: {missing}", record)
    if not isinstance(record["value"], (int, float)):
        raise ValidationError(
            f"'value' must be numeric, got {type(record['value']).__name__}",
            record
        )

def process_records(records):
    """Process records with comprehensive error handling."""
    good, bad = [], []

    for i, record in enumerate(records):
        try:
            validate_record(record)
        except ValidationError as e:
            logger.warning(f"Record {i} invalid: {e}")
            bad.append({"index": i, "record": e.record, "error": str(e)})
            continue

        try:
            result = {"id": record["id"], "result": record["value"] ** 2}
            good.append(result)
        except Exception as e:
            logger.error(f"Processing failed for record {i}", exc_info=True)
            bad.append({"index": i, "record": record, "error": str(e)})

    return good, bad

# Test data with various problems
test_data = [
    {"id": 1, "value": 10},
    {"id": 2, "value": "bad"},     # Wrong type
    {"id": 3},                      # Missing field
    "not a dict",                   # Wrong structure
    {"id": 4, "value": 20},
]

good, bad = process_records(test_data)
print(f"\nSuccessful: {len(good)}, Failed: {len(bad)}")
for r in good:
    print(f"  ✓ {r}")
for r in bad:
    print(f"  ✗ Record {r['index']}: {r['error']}")
Key Insight: Production pipelines process millions of records. A single bad record should not crash the whole pipeline. The pattern above — validate, catch per-record errors, continue processing, collect failures for later review — is how real data engineering works. Every ETL framework follows this model.
