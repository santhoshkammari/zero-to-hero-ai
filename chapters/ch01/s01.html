Python Essentials

Python is the glue of ML — data loading, feature engineering, training loops, deployment scripts, all of it flows through Python. This section is your cheat-sheet for the parts that actually matter: the power tools that save you hours, the gotchas that bite in production, and the decisions you'll make every day.

Types & Basics — The 30-Second Pass

int, float, str, bool, None — you know these. Here's what's actually worth remembering:

Float comparison will burn you. 0.1 + 0.2 != 0.3 — that's IEEE 754, not a Python bug. Use math.isclose(a, b) or set an explicit tolerance. Every loss comparison, every threshold check in your pipeline should use this.

Identity vs equality for None. Always x is None, never x == None. There's exactly one None object in memory — identity check (is) is both correct and faster. Custom __eq__ methods can make == return weird things.

String concat in a loop is O(n²). Every += creates a new string object, copies everything, then discards the old one. Collect into a list and "".join(parts) at the end — it's the difference between 2 seconds and 20 minutes on large text corpora.

bool is secretly int. True + True == 2. This isn't trivia — it's why sum(flags) counts truthy values and why boolean masks work the way they do in NumPy.

Data Structures — Which One When

The decision table. This is where a table actually earns its keep:

StructureUse WhenGotchaLookup
list Ordered, mutable, need index access b = a copies the reference. in is O(n). [[0]*3]*3 creates 3 refs to same inner list. O(n)
tuple Fixed records, dict keys, function returns Immutable ≠ hashable — ([1,2], 3) has a mutable inside, can't be a dict key. O(n)
dict Key→value mapping, configs, lookups Keys must be hashable. Insertion order preserved (3.7+). Use defaultdict for accumulation. O(1) avg
set Membership tests, dedup, set math Unordered — no indexing. If checking in on a list repeatedly, convert to set first. O(1) avg
frozenset Need a set as a dict key or inside another set Immutable set. Same O(1) lookup, but you can't add/remove. O(1) avg

The rule of thumb: need order? list. Need immutability? tuple. Need fast lookup? dict or set. If you're doing if x in my_list inside a loop, you've got an O(n²) bug hiding in plain sight.

The Real Stuff: collections + itertools + functools

This is where you go from "I know Python" to "I'm dangerous with Python." These three modules are the power tools that replace dozens of lines of manual code with one clean, battle-tested call. Once you internalize these, you'll spot opportunities to use them everywhere.

collections

Counter — the thing about Counter is that it replaces every manual counting loop you've ever written. Token frequencies in NLP, class distributions in datasets, error type tallies in logs — all one line:

from collections import Counter

# word frequency from a corpus — replaces 5 lines of dict bookkeeping
word_freq = Counter(corpus.split())
word_freq.most_common(10)          # top-10 tokens, sorted
word_freq["the"]                   # direct access, returns 0 for missing (not KeyError)

# class balance check in a dataset
label_dist = Counter(y_train)      # {0: 4500, 1: 500} — instantly see imbalance

Counter also supports arithmetic: counter_a - counter_b gives you the diff, counter_a & counter_b gives the min overlap. Handy for comparing vocabulary between two corpora.

defaultdict — eliminates the "check if key exists, create if not" pattern. You hand it a factory function, and missing keys auto-initialize. The mental shift: stop thinking "does this key exist?" and start thinking "just append/add/increment":

from collections import defaultdict

# group samples by label — no if/else, no setdefault
by_label = defaultdict(list)
for features, label in dataset:
    by_label[label].append(features)

# aggregate metrics per experiment
metrics = defaultdict(lambda: {"total": 0, "correct": 0})
for exp_id, is_correct in results:
    metrics[exp_id]["total"] += 1
    metrics[exp_id]["correct"] += int(is_correct)

deque — a double-ended queue with O(1) append/pop on both ends. Lists are O(n) for insert(0, x) or pop(0) because they shift everything. Deque is purpose-built for sliding windows and recent-N tracking:

from collections import deque

# sliding window of last 100 losses for smoothing
recent_losses = deque(maxlen=100)
for batch_loss in training_loop:
    recent_losses.append(batch_loss)
    smoothed = sum(recent_losses) / len(recent_losses)

The maxlen argument is the magic — it auto-evicts the oldest element when full. No manual slicing, no off-by-one bugs.

OrderedDict — "but regular dicts preserve order since 3.7?" True. OrderedDict still matters for two reasons: move_to_end(key) for LRU-style reordering, and equality that cares about order (OrderedDict(a=1, b=2) != OrderedDict(b=2, a=1), while regular dicts would be equal). You'll see it in PyTorch's state_dict for model layer ordering.

namedtuple — lightweight data records when you want named fields but don't need a full class. Zero runtime overhead over a regular tuple, but you get readable attribute access:

from collections import namedtuple

Sample = namedtuple("Sample", ["features", "label", "weight"])
s = Sample([0.2, 0.8], label=1, weight=0.5)
s.label          # 1 — readable
s[1]             # 1 — still works as a tuple
# immutable, hashable, works as dict keys — free

For anything with default values or mutability, reach for dataclasses instead. Namedtuple is for the cases where immutability is a feature, not a limitation.

itertools

itertools is lazy by default — nothing materializes until you consume it. That matters when your data doesn't fit in memory.

chain — flatten nested iterables without copying. When you have train/val/test splits and need to iterate all of them:

from itertools import chain

# iterate all splits as one stream — zero memory overhead
for sample in chain(train_data, val_data, test_data):
    process(sample)

# flatten a list of lists (one level)
flat = list(chain.from_iterable(nested_lists))

product / combinations — hyperparameter grid search without nested loops. This is the one that cleans up the ugliest code in your experiment scripts:

from itertools import product, combinations

# grid search — replaces 3 nested for loops
lrs = [0.001, 0.01, 0.1]
batch_sizes = [32, 64, 128]
dropouts = [0.1, 0.3, 0.5]

for lr, bs, drop in product(lrs, batch_sizes, dropouts):
    run_experiment(lr=lr, batch_size=bs, dropout=drop)

# all pairs of features for interaction terms
for feat_a, feat_b in combinations(feature_names, 2):
    create_interaction(feat_a, feat_b)

groupby — group sorted data on the fly. Critical caveat: the data must be sorted by the grouping key first, or you get fragmented groups:

from itertools import groupby
from operator import itemgetter

# group predictions by class — data MUST be sorted by key first
sorted_preds = sorted(predictions, key=itemgetter("class"))
for cls, group in groupby(sorted_preds, key=itemgetter("class")):
    group_list = list(group)  # consume the iterator
    print(f"{cls}: {len(group_list)} predictions")

islice — lazy slicing for huge iterables. When you want the first 1000 lines of a 50GB file without loading it all:

from itertools import islice

# preview first 5 items from any iterator — no list conversion
first_five = list(islice(huge_generator, 5))

# skip header, take next 1000
batch = list(islice(data_stream, 1, 1001))

accumulate — running totals and cumulative operations. Useful for cumulative distribution plots, running loss averages, or prefix sums:

from itertools import accumulate
import operator

losses = [0.9, 0.7, 0.5, 0.3, 0.2]
cumulative = list(accumulate(losses))           # running sum
running_max = list(accumulate(losses, max))     # running max
running_prod = list(accumulate(losses, operator.mul))  # running product

functools

lru_cache / cache — memoization with one decorator. If a function is pure (same input → same output) and expensive, cache it. @cache (3.9+) is unbounded; @lru_cache(maxsize=256) evicts least-recently-used entries:

from functools import lru_cache

@lru_cache(maxsize=1024)
def extract_features(file_path):
    """Expensive I/O + computation — cache it."""
    data = load_and_parse(file_path)
    return compute_features(data)

# second call with same path is instant
features = extract_features("data/sample_001.wav")
features = extract_features("data/sample_001.wav")  # cache hit
Gotcha: arguments must be hashable (no lists, no dicts). If you need to cache something that takes a list, convert to tuple first: tuple(my_list).

partial — pre-fill arguments to create specialized versions of general functions. This shows up constantly in ML pipelines where you want to configure a transform once and pass it around:

from functools import partial

# create configured versions of a general function
normalize = partial(sklearn_normalize, norm="l2", axis=1)
tokenize_en = partial(tokenizer, lang="en", lowercase=True)

# use in map/pipeline — clean, no lambdas
processed = list(map(tokenize_en, documents))

The difference from lambda: partial is picklable (works with multiprocessing), inspectable, and doesn't have the late-binding gotcha that closures do.

reduce — fold an iterable into a single value. You rarely need it (most cases are covered by sum, max, min, or accumulate), but when you do, nothing else fits:

from functools import reduce

# merge a list of dicts into one (Python 3.9+ has dict | operator, but reduce works everywhere)
configs = [base_config, model_config, experiment_overrides]
merged = reduce(lambda a, b: {**a, **b}, configs)

Control Flow — Only What's Non-Obvious

You know if, for, while. Here are the four patterns people miss:

for/else — the else block runs only if the loop completes without hitting break. It eliminates the "found" flag variable:

for sample in dataset:
    if is_corrupted(sample):
        log_error(sample)
        break
else:
    print("All samples clean")  # only runs if no break

Walrus operator := (3.8+) — assign and use in one expression. Most useful in while loops and filtering:

# read until empty — no awkward "prime the loop" pattern
while (line := file.readline()):
    process(line)

# filter with expensive computation — compute once, not twice
valid = [result for x in data if (result := expensive(x)) is not None]

Ternary — x = a if cond else b. One level only. The moment you nest ternaries, you've lost the readability war. Use a plain if/else.

match/case (3.10+) — not just a switch statement. It destructures. case [x, y]: unpacks a 2-element list, case {"type": "error", "msg": m}: extracts from dicts. Worth knowing it exists; reach for it when parsing structured messages or command patterns.

Functions — The Patterns That Matter

*args/**kwargs — pass-through for wrappers and decorators. args is a tuple, kwargs is a dict. Common in ML: Model(**config) unpacks a config dict into keyword arguments.

Closures — a function that captures variables from its enclosing scope. This is the factory pattern, and it's everywhere in ML — creating configured transforms, building callback functions, generating loss functions with specific parameters:

def make_augmenter(noise_level, crop_size):
    """Factory that returns a configured augmentation function."""
    def augment(image):
        noisy = add_noise(image, sigma=noise_level)
        return random_crop(noisy, size=crop_size)
    return augment

# create specialized versions
light_aug = make_augmenter(noise_level=0.01, crop_size=224)
heavy_aug = make_augmenter(noise_level=0.1, crop_size=196)
The closure loop trap: [lambda: i for i in range(3)] — all three lambdas return 2, not 0, 1, 2. Closures capture the variable, not the value. By the time you call them, i is already 2. Fix: lambda i=i: i binds the current value as a default argument.

Lambda — use for throwaway one-liners: sort keys (sorted(items, key=lambda x: x[1])), quick transforms in map. The moment it needs a name, a docstring, or more than one expression, switch to def. Don't assign lambdas to variables — that's just a worse function definition.

Type hints — they don't affect runtime, but they're how you communicate intent. IDEs give you autocomplete, mypy catches bugs before they happen, and your future self can actually read the function signatures:

def train(model: nn.Module, data: DataLoader, epochs: int = 10) -> dict[str, float]:
    ...

In production ML code, type hints are the cheapest documentation you'll ever write.

Pythonic Patterns

Writing Python is one thing. Writing Pythonic Python is where readability meets performance — it's the gap between code that works and code that a team can maintain, extend, and ship. This section covers the patterns that separate amateur scripts from production code.

Comprehensions

You already know [x**2 for x in range(10)]. The non-obvious ones: dict comprehensions for building lookup tables ({name: idx for idx, name in enumerate(labels)}), set comprehensions for dedup ({row["category"] for row in data}), and nested for flattening ([x for row in matrix for x in row]). Once nesting goes past two levels, stop — write a loop. Readability always wins.

Memory matters: (x**2 for x in range(10_000_000)) with round brackets is a generator expression — it produces values lazily, using almost zero memory. Square brackets [] builds the entire list in RAM first. When you're feeding something into sum(), max(), or any(), always use the generator form.

Generators & Lazy Evaluation

Here's the scenario: you have 10GB of training data on disk. You need to read it, clean it, tokenize it, and feed it to a model. If you load all of it into a list, your machine runs out of RAM and the OOM killer ends your process. Generators solve this by producing values one at a time — memory stays O(1) regardless of data size.

The keyword is yield. When a function contains yield, calling it doesn't execute the body — it returns a generator object. Each time you pull the next value (via next() or a for loop), the function runs until it hits yield, hands you the value, and freezes. Next call, it resumes exactly where it left off. All local variables are preserved.

The mental model: a generator is a recipe, not a meal. It describes how to produce values without producing them all at once. A list is a meal — every dish is on the table, taking up space, whether you eat it or not.

def read_lines(path):
    """Yields one line at a time — handles files larger than RAM."""
    with open(path) as f:
        for line in f:
            stripped = line.strip()
            if stripped:  # skip blanks
                yield stripped

def tokenize(lines):
    """Takes any iterable of strings, yields token lists."""
    for line in lines:
        yield line.lower().split()

def filter_short(token_lists, min_len=3):
    """Drop sequences shorter than min_len."""
    for tokens in token_lists:
        if len(tokens) >= min_len:
            yield tokens

# Compose the pipeline — nothing executes yet
lines   = read_lines("corpus.txt")       # generator (lazy)
tokens  = tokenize(lines)                # generator (lazy)
cleaned = filter_short(tokens)           # generator (lazy)

# NOW it runs — one item flows through the entire chain at a time
for batch in cleaned:
    model.train_on(batch)

Each stage is its own generator. Data flows through the pipeline one item at a time — at no point does the full dataset live in memory. This is exactly how PyTorch's DataLoader and TensorFlow's tf.data work under the hood.

yield from lets you compose generators cleanly. Instead of looping through a sub-generator and re-yielding each item, delegate:

def read_all_files(paths):
    for path in paths:
        yield from read_lines(path)  # flattens automatically
The exhaustion gotcha: A generator can only be iterated once. After one pass, it's empty — silently. No error, no warning, just... no data on the second loop. If you need multiple passes, either convert to a list (data = list(gen)) or recreate the generator. This bites people in debugging all the time: the first loop works, the second produces nothing, and the code looks perfectly fine.
Decision rule: Need len(), indexing, or multiple passes? → List. Processing sequentially and data might be large? → Generator. When in doubt, start with a generator — you can always list() it later. You can't un-materialize a list.

Decorators

You don't need to build decorator factories. You don't need to understand triple-nested closures. What you do need is to write the one decorator everyone actually uses — and recognize the built-in ones you'll see in every codebase.

The one decorator worth writing yourself — a timing wrapper:

import time
import functools

def timer(func):
    @functools.wraps(func)  # preserves func.__name__, __doc__
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        result = func(*args, **kwargs)
        elapsed = time.perf_counter() - start
        print(f"{func.__name__} took {elapsed:.4f}s")
        return result
    return wrapper

@timer
def train_epoch(model, data):
    ...  # train_epoch took 42.1337s

The decorators you'll read in framework code (don't memorize — just know what they mean when you see them):

  • @property — turns a method into an attribute access. model.num_params instead of model.get_num_params().
  • @staticmethod — a function that lives in a class namespace but doesn't touch self or cls.
  • @classmethod — receives the class as first arg. Used for factory constructors: Model.from_pretrained("bert").
  • @functools.lru_cache — memoizes return values. Great for expensive pure functions. Gotcha: it holds references to all cached arguments — on large objects like DataFrames, this silently leaks memory.

Error Handling

Production ML code fails constantly — bad data, network timeouts, GPU OOM. The one rule that matters: catch specific exceptions, never bare except:. A bare except catches KeyboardInterrupt, SystemExit, memory errors — bugs you absolutely want to see. except Exception is almost as bad. Name exactly what you expect: except ValueError, except FileNotFoundError.

LBYL vs EAFP

Two philosophies. LBYL (Look Before You Leap): check first, then act. EAFP (Easier to Ask Forgiveness than Permission): just do it, handle failure. Python's culture is EAFP — and there's a real reason beyond style:

# LBYL — two lookups, and in concurrent code, the key
# could disappear between the check and the access
if key in config:
    value = config[key]

# EAFP — one lookup, atomic, Pythonic
try:
    value = config[key]
except KeyError:
    value = default

EAFP is faster on the happy path (no redundant check) and safer in concurrent code (no race condition between check and access). Use LBYL only when the check itself is the point — like validating user input before expensive processing.

Custom Exceptions

When your library raises ValueError for everything, callers can't distinguish "bad input shape" from "model not fitted" without parsing error strings. That's fragile. One custom exception fixes it:

class ModelNotFittedError(RuntimeError):
    """Raised when predict() is called before fit()."""

class MyModel:
    def predict(self, X):
        if self.weights is None:
            raise ModelNotFittedError(
                f"{self.__class__.__name__}.fit() must be called before predict()"
            )
        ...

Now callers can except ModelNotFittedError and handle it cleanly — retrain, load from cache, whatever. Always inherit from a built-in exception (RuntimeError, ValueError) so generic except Exception still catches it.

Context Managers

The with statement guarantees cleanup — even if an exception fires midway. You've seen with open(f) as fh:. But you can make your own without writing a class:

from contextlib import contextmanager

@contextmanager
def gpu_memory_scope(device):
    """Allocate GPU memory, guarantee release."""
    torch.cuda.set_device(device)
    try:
        yield device
    finally:
        torch.cuda.empty_cache()

with gpu_memory_scope(0) as dev:
    model.to(dev)
    # ... train ...
# GPU cache cleared even if training crashes

Any resource that needs cleanup — files, database connections, locks, GPU memory — should be a context manager. If you find yourself writing try: ... finally: cleanup(), you want with instead.

String Formatting & f-strings

f-strings won. Use them. f"{accuracy:.4f}" for ML metrics, f"{name!r}" for debugging (adds quotes). The trick most people don't know: f"{variable=}" prints both the name and the value — variable=42. Invaluable for quick debugging without writing print(f"variable={variable}") every time.

Unpacking & Assignment Patterns

These look small but they save code every single day in data processing.

# Star unpacking — grab first, collect the rest
first, *rest = [1, 2, 3, 4, 5]       # first=1, rest=[2,3,4,5]
*init, last = [1, 2, 3, 4, 5]        # init=[1,2,3,4], last=5
head, *_, tail = range(10)            # grab endpoints, discard middle

# Swap without a temp variable
a, b = b, a

# Underscore for throwaway values
_, accuracy = model.evaluate(X, y)    # don't need the loss
for _, row in df.iterrows():          # don't need the index

# Multiple return → tuple unpacking
def train_step(model, batch):
    ...
    return loss, accuracy, gradients

loss, acc, grads = train_step(model, batch)

The star unpacking pattern (first, *rest) is especially useful for parsing CSVs, splitting command arguments, and any situation where you know the structure of the first few items but the rest varies. The _ convention tells other readers "I'm intentionally ignoring this value" — it's not magic, just convention.

File I/O

ML is 80% data plumbing. If you can't efficiently read configs, stream large files, and handle paths cleanly, you'll lose time every single day. This section isn't about open() — you know open() — it's about the patterns that keep your I/O fast, portable, and production-safe.

The Basics in 30 Seconds

open(), with statement, modes (r/w/a/rb/wb) — you know these. Two rules that matter:

  • Always use with (context manager) to auto-close. No exceptions. Leaked file descriptors pile up silently until your long-running training script dies with OSError: Too many open files.
  • Binary mode rb/wb for anything that isn't text. Model weights, images, pickle files, .npy arrays — text mode will corrupt them because it tries to decode bytes as UTF-8.
# The only pattern you should ever use
with open("data.csv") as f:
    content = f.read()
# f is closed here — guaranteed, even if an exception fired

# Binary for non-text
with open("model.pkl", "rb") as f:
    model = pickle.load(f)

pathlib — The Modern Way

Here's the thing about pathlib — once you start chaining paths with the / operator, os.path.join starts to look like a relic. It's object-oriented, chainable, cross-platform, and the code reads like English.

from pathlib import Path

# Building dataset paths — clean and readable
project = Path(__file__).parent
data_dir = project / "data" / "processed"
output_dir = project / "outputs" / "experiment_03"

# Create directories — one line, no errors if they exist
output_dir.mkdir(parents=True, exist_ok=True)

# Find all CSVs in nested directories
csv_files = list(data_dir.glob("**/*.csv"))
print(f"Found {len(csv_files)} CSV files")

# Process each file — .stem, .suffix, .parent all just work
for csv_path in csv_files:
    print(f"  {csv_path.stem} ({csv_path.suffix}) in {csv_path.parent.name}/")
    df = pd.read_csv(csv_path)
    out = output_dir / f"{csv_path.stem}_cleaned.parquet"
    df.to_parquet(out)

The key methods you'll use daily:

  • p / "subdir" / "file.txt" — the / operator joins paths. No more os.path.join(a, b, c).
  • p.glob("**/*.csv") — recursive file search. Returns a generator, so it's lazy on huge trees.
  • p.stem → "train", p.suffix → ".csv", p.name → "train.csv" — anatomy of a path, as attributes.
  • p.parent — go up one level. Chain it: p.parent.parent for two levels.
  • p.mkdir(parents=True, exist_ok=True) — creates the full directory tree. The two flags you always want.
  • p.read_text() / p.write_text(s) — one-liner for small files, no with needed.
  • Path(__file__).parent — resolve paths relative to the script, not the current working directory. This is the single biggest portability win.
Never hardcode paths. "/home/user/data" breaks on every other machine. "data/train.csv" breaks when run from a different directory. Path(__file__).parent / "data" / "train.csv" works everywhere — it resolves relative to where the script lives, not where you launched it from.

Config Files — JSON, YAML, TOML

Every ML project needs config files. The question is which format — and this is one of those genuine decision points where a table is the right tool:

FormatStrengthsWeaknessesUse When
JSON Built-in (json module), universal, strict syntax No comments, verbose, no trailing commas Data exchange, API responses, simple configs
YAML Human-friendly, comments, anchors & aliases for reuse Indentation-sensitive (bugs from tabs vs spaces), yaml.load() is a security hole Experiment configs, Kubernetes manifests, anything humans edit often
TOML Simple, readable, built-in since Python 3.11 (tomllib) Less expressive than YAML, no built-in write support yet pyproject.toml, simple project configs

The pattern you'll use most — loading experiment configs with YAML:

import yaml
from pathlib import Path

config_path = Path(__file__).parent / "configs" / "experiment.yaml"

# ALWAYS safe_load — never yaml.load() which executes arbitrary Python
with open(config_path) as f:
    config = yaml.safe_load(f)

lr = config["training"]["learning_rate"]      # 0.001
batch_size = config["training"]["batch_size"]  # 32
model_name = config["model"]["architecture"]   # "resnet50"
Never use yaml.load() — it can execute arbitrary Python code embedded in the YAML file. Always yaml.safe_load(). This isn't a style preference, it's a security vulnerability on the level of eval().

Handling Large Files

When your dataset doesn't fit in memory — and it will happen sooner than you think — you need to stop reading entire files at once. The good news: Python already has the right primitives.

Line-by-line reading is already lazy. This is the thing people miss — for line in file doesn't load the whole file. It reads one line at a time from the OS buffer. For text processing, this is your default pattern:

# Streams a 50GB log file without loading it all into memory
with open("training.log") as f:
    errors = [line for line in f if "ERROR" in line]

Streaming CSV with generators — when you need to process rows without materializing the whole dataset:

import csv

def stream_csv(path):
    """Yields one row at a time — constant memory regardless of file size."""
    with open(path) as f:
        reader = csv.DictReader(f)
        for row in reader:
            yield row

# Process 10M rows without loading them all
for row in stream_csv("huge_dataset.csv"):
    if float(row["score"]) > 0.95:
        process(row)

Memory-mapped files — when you need random access into a huge file without reading it all. The OS maps the file into virtual memory and pages in only what you access. This is how you'd read into a large binary file of pre-computed embeddings or model weights:

import mmap

with open("embeddings.bin", "rb") as f:
    mm = mmap.mmap(f.fileno(), 0, access=mmap.ACCESS_READ)
    # Read bytes 1000-2000 without loading the whole file
    chunk = mm[1000:2000]
    mm.close()

Binary serialization formats — for saving and loading Python objects:

  • pickle — fast, handles any Python object, but Python-only and a security risk (unpickling executes arbitrary code). Fine for internal checkpoints you control.
  • joblib — better than pickle for large numpy arrays (compressed, memory-mapped loading). The scikit-learn standard for saving models.
  • Never load untrusted pickle/joblib files — treat them like executables, because they are.
import joblib

# Save — compressed, efficient for numpy arrays
joblib.dump(trained_model, "model.joblib", compress=3)

# Load — can memory-map large arrays automatically
model = joblib.load("model.joblib", mmap_mode="r")

What You Should Now Be Able To Do

  • Use pathlib for all path operations — the / operator, .glob(), .stem, .parent, .mkdir() — and never go back to os.path.join
  • Build paths relative to Path(__file__).parent, never hardcoded, never dependent on cwd
  • Choose the right config format — JSON for data exchange, YAML for human-edited configs (safe_load only), TOML for project metadata
  • Stream large files without loading them into memory — for line in file, CSV generators, mmap for random access
  • Pick the right serialization — pickle for quick checkpoints, joblib for numpy-heavy models, and never load untrusted files from either
  • Explain why with is mandatory (resource leaks) and why binary mode matters (data corruption)

Modules and Packages

Clean project structure and proper packaging separate scripts from maintainable ML systems. This section is about the structure, not the syntax — how to organize code so your teammates can clone, install, and run in under a minute.

Imports in 10 Seconds

import x, from x import y, from x import y as z — you know these. One gotcha worth internalizing: circular imports. They happen when module A imports B and B imports A — both try to load each other before either finishes. Fix: move the shared code to a third module, or move the import inside the function that needs it (lazy import). If you're hitting circular imports often, your project structure needs rethinking.

Project Structure — The Real Value

This is what actually matters. Not import syntax — everyone can Google that. The difference between a script that only works on your laptop and a project anyone can contribute to comes down to structure.

my_project/
├── src/
│   └── my_project/
│       ├── __init__.py
│       ├── data/
│       │   ├── __init__.py
│       │   └── loader.py
│       ├── models/
│       │   ├── __init__.py
│       │   └── resnet.py
│       ├── training/
│       │   ├── __init__.py
│       │   └── trainer.py
│       └── utils/
│           ├── __init__.py
│           └── metrics.py
├── tests/
│   ├── test_loader.py
│   └── test_trainer.py
├── configs/
│   └── experiment.yaml
├── pyproject.toml
└── README.md

Why the src/ layout? Without it, Python can accidentally import your local package directory instead of the installed version. When your project root contains a folder named my_project/ and you're sitting in that root, import my_project grabs the local folder — not the version you pip install-ed. The src/ layer prevents this. You run pip install -e . from the root, and imports always resolve to the installed package. It's a small structural choice that eliminates an entire class of "works on my machine" bugs.

__init__.py — what it does and when to care:

  • It marks a directory as a Python package. Without it, Python won't recognize the directory for imports.
  • Keep it empty most of the time. Seriously. An empty __init__.py is the right default.
  • Put stuff in it when you want a clean public API — re-exporting the things users of your package should access directly:
# src/my_project/__init__.py
from .models.resnet import ResNet50
from .training.trainer import Trainer
from .data.loader import DataLoader

# Now users can do: from my_project import ResNet50, Trainer
# Instead of: from my_project.models.resnet import ResNet50

__all__ controls what from package import * exports. It does not prevent direct imports — it only restricts star imports. Define it when your module has internal helpers that shouldn't leak into *:

# models/resnet.py
__all__ = ["ResNet50", "ResNet101"]

def _build_block(channels):   # internal — not in __all__
    ...

class ResNet50:  # public
    ...

Virtual Environments

The core idea: isolation. Your project's dependencies don't pollute system Python, and your TensorFlow project doesn't break your PyTorch one. If you're not using virtual environments, every pip install is a ticking time bomb.

# Create, activate, install — three commands, done
python -m venv .venv
source .venv/bin/activate      # Windows: .venv\Scripts\activate
pip install -r requirements.txt

# Verify you're in the right env
which python   # should show .venv/bin/python
pip list       # only this project's dependencies

When to use conda instead: When you need CUDA libraries, compiled C/Fortran packages (like some scipy builds), or non-Python dependencies. conda manages binary packages across languages — it can install CUDA toolkit, cuDNN, and PyTorch with GPU support in one command. venv + pip can only install Python packages.

Requirements management — the ecosystem has converged, but there are still choices:

  • requirements.txt — a flat list of pinned packages. Simple, universal. Generate with pip freeze > requirements.txt. Problem: includes transitive dependencies, gets bloated.
  • pyproject.toml — the modern standard. Replaces setup.py, setup.cfg, and can specify dependencies. What you should use for new projects.
  • poetry.lock / uv.lock — lock files that pin exact versions of every transitive dependency. True reproducibility. Worth the tooling cost for production systems.
Pin your versions. pip install numpy installs the latest — which might break your code next month. Always use numpy==1.26.4 in requirements files. Unpinned dependencies are the #1 cause of "it worked yesterday" bugs.

Packaging Tools — Which One

ToolWhat It DoesUse When
pip + requirements.txt Simple dependency list, manual pinning Quick scripts, small projects, tutorials. Everyone has it.
pip + pyproject.toml Modern standard, replaces setup.py, declarative metadata Any new project. This is the direction Python packaging is going.
Poetry Dependency resolution, lock files, virtual env management, publishing Libraries you'll publish, teams that need strict reproducibility.
uv Fast Rust-based pip/venv replacement, lock files, project management When speed matters (10-100× faster installs). Rapidly gaining adoption.
conda Cross-language package manager, manages binaries and CUDA GPU workloads, scientific computing, non-Python dependencies.

For most ML projects: start with pip + pyproject.toml. If you need GPU libraries that are painful to compile, add conda. If you need reproducible lock files and plan to publish, consider Poetry or uv.

if __name__ == "__main__":

Guards code that should only run when the file is executed directly, not when imported. Without it, importing a module that trains a model would start training.
