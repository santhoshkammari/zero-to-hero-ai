To ground everything, we'll build a tiny running example throughout this section: a small experiment tracker. Every ML engineer needs one — something that records which hyperparameters you tried, what score you got, and when. results = []
results.append({"lr": 0.01, "epochs": 10, "accuracy": 0.82})
results.append({"lr": 0.001, "epochs": 20, "accuracy": 0.91})
results.append({"lr": 0.01, "epochs": 50, "accuracy": 0.89})

Three experiments. A list of dictionaries. That's our tracker, version one. It will get better.

Variables Are Not Boxes

Here's the mental model shift that took me an embarrassingly long time. In most programming introductions, variables are described as boxes that hold values. You put the number 5 into a box labeled x. That mental model is actively wrong in Python, and it's the source of some of the most persistent bugs in production code.

In Python, a variable is a name binding — a label, a post-it note stuck onto an object that lives somewhere in memory. The object exists independently of any names attached to it. Multiple names can point to the same object. No name "contains" anything.

Let's see what this means concretely. Consider our experiment tracker:

best_run = results[1]       # {"lr": 0.001, "epochs": 20, "accuracy": 0.91}
alias = best_run

How many dictionaries exist in memory right now? One. The names best_run, alias, and results[1] are all post-it notes stuck on the same dictionary object. If we modify through any of those names, every other name sees the change.

alias["accuracy"] = 0.95
print(best_run["accuracy"])  # 0.95 — same object, remember

This behavior has a name: aliasing. When two or more names refer to the same object in memory, they are aliases. You can verify this with Python's built-in id() function, which returns the memory address of an object in CPython (the standard Python implementation, written in C, that you're almost certainly using).

print(id(best_run) == id(alias))  # True — same object
print(best_run is alias)          # True — 'is' checks identity

That is keyword checks whether two names point to the same object in memory — identity. The == operator checks whether two objects have the same value — equality. These are different questions. Two separate dictionaries can have identical contents (== returns True) while being distinct objects in memory (is returns False).

This distinction is why you always write x is None, never x == None. There's exactly one None object in all of Python. Identity check is both semantically correct and faster. A custom __eq__ method on some object could make == None return something unexpected.

I'll be honest — when this post-it-note model finally clicked for me, half the "weird" Python behaviors I'd been memorizing as special cases suddenly became obvious consequences of a single rule. But that model alone doesn't tell us everything. We need one more distinction.

Mutable vs Immutable — Why the Distinction Exists

If variables are post-it notes on objects, the next question is: can we change the object itself, or do we have to create a new one? This is the mutable versus immutable distinction.

Mutable objects — lists, dictionaries, sets — can be changed in place. You can append to a list, update a dictionary key, add to a set. The object at that memory address gets modified. All names pointing to it see the change.

Immutable objects — integers, floats, strings, tuples — cannot be changed once created. When you do x = x + 1, Python creates a new integer object with the value x + 1 and rebinds the name x to point at it. The old integer object is left behind, unnamed, eventually collected by the garbage collector.

name = "experiment"
print(id(name))        # some address, say 140234567890
name = name + "_v2"
print(id(name))        # different address — new string object was created

Compare that with a mutable list:

scores = [0.82, 0.91]
print(id(scores))      # some address
scores.append(0.89)
print(id(scores))      # same address — the list was modified in place

Why does this matter? Because it determines whether passing something to a function or assigning it to a new variable creates a shared reference that can lead to surprising mutations — or an independent value that's safe to change.

Here's a bug that has bitten me in production. Suppose we write a function to add a default metric to an experiment result:

def add_default_metric(result, defaults=[]):
    defaults.append("loss")
    result["defaults"] = defaults
    return result

The default argument [] is evaluated once, when the function is defined — not on every call. That empty list is a single mutable object shared across all calls. Every invocation appends "loss" to the same list.

r1 = add_default_metric({"lr": 0.01})
r2 = add_default_metric({"lr": 0.001})
print(r1["defaults"])  # ["loss", "loss"] — surprise!

The fix is the Python idiom you'll see everywhere:

def add_default_metric(result, defaults=None):
    if defaults is None:
        defaults = []    # fresh list on every call
    defaults.append("loss")
    result["defaults"] = defaults
    return result

This is one of those patterns where the why matters more than the what. The fix is mechanical. Understanding that default arguments are evaluated at definition time — because they're object bindings, not box values — is the insight that prevents you from re-discovering this bug in twelve different forms.

One more consequence worth knowing: CPython caches small integers from -5 to 256 as singleton objects (a performance optimization called interning). So a = 256; b = 256; a is b returns True — same cached object. But a = 257; b = 257; a is b may return False — two separate objects. This is why is should never be used to compare values. Use == for values, is for identity. The interning behavior is an implementation detail, not a language guarantee.

We now have a solid foundation: names bind to objects, objects are either mutable or immutable, and this determines everything about copying, aliasing, and function argument behavior. But our experiment tracker is still a plain list of dictionaries. We need to think about which containers to reach for.

Data Structures — Choosing the Right Container

Python gives us four core container types, and the choice between them isn't stylistic — it has real performance and correctness consequences. The decisive question is always: what operations do I need to be fast?

Lists — Ordered, Mutable Sequences

A list in CPython is a contiguous array of pointers. Not the elements themselves — pointers to the elements, which live elsewhere on the heap. This is why a Python list can hold mixed types (each pointer can point to any object) and why index access is O(1) (pointer arithmetic).

Appending to a list is amortized O(1) because Python over-allocates. When the internal array fills up, CPython allocates roughly 12.5% more space than currently needed. So most appends are a simple pointer write, with occasional resizing.

The trap: checking membership with in is O(n). Python has to scan the entire list. This is fine for 10 items. For 10 million items — say, checking if a token exists in a vocabulary — it's a disaster. That's where sets come in.

Dictionaries — The Hash Table

A dict is Python's implementation of a hash table. When you write d["lr"] = 0.01, Python computes a hash of the key "lr", uses it to find a slot in an internal array, and stores the key-value pair there. Lookup, insertion, and deletion are all O(1) on average.

Our experiment tracker is already using dicts, and for good reason. When you need to associate keys with values — hyperparameters with their settings, metric names with their scores — dict is the answer.

Since Python 3.7, dictionaries preserve insertion order. This isn't an accident — it fell out of a clever reimplementation by Raymond Hettinger that separated the hash indices from the actual entries, making dicts both more memory-efficient and ordered as a side effect. Before 3.7, if you needed ordered keys, you had to use collections.OrderedDict.

The constraint: dict keys must be hashable, which in practice means immutable. Strings, numbers, tuples (of hashable elements) — all fine. Lists and dicts — not hashable, can't be keys. The reason is that if a key mutated after being inserted, its hash would change, and the hash table would lose track of it.

Sets — Fast Membership

A set is essentially a dict without values — same hash table underneath. This means membership testing (x in my_set) is O(1).

Imagine our tracker grows, and we want to quickly check if we've already tried a specific learning rate:

tried_lrs = {r["lr"] for r in results}   # set comprehension
if 0.01 in tried_lrs:
    print("Already tried this learning rate")

With a list, this check would scan every element. With a set, it computes one hash and checks one slot.

Tuples — Immutable Records

A tuple is like a list that can't be changed after creation. Same contiguous array of pointers internally, but with the immutability guarantee. This makes tuples slightly lighter on memory (no over-allocation needed) and — critically — hashable. You can use a tuple as a dict key or add it to a set.

# using tuples as composite keys
best_by_config = {}
for r in results:
    key = (r["lr"], r["epochs"])    # tuple is hashable, list would fail
    if key not in best_by_config or r["accuracy"] > best_by_config[key]["accuracy"]:
        best_by_config[key] = r

The limitation of all these containers is that they hold data passively. To process that data — iterate through experiments, filter results, transform values — we need to understand how Python's iteration actually works. And it's more interesting than most people think.

The Iteration Protocol

When you write a for loop in Python, something specific happens under the hood. It's not pattern matching, it's not indexing — it's a protocol, meaning a set of methods that objects agree to implement.

Here's what for item in results: actually does:

# Python translates this:
for item in results:
    print(item)

# Into roughly this:
_iter = iter(results)          # calls results.__iter__()
while True:
    try:
        item = next(_iter)     # calls _iter.__next__()
        print(item)
    except StopIteration:
        break

Two methods make this work. __iter__() returns an iterator object — something that knows how to produce the next element. __next__() on that iterator returns the next element, or raises StopIteration when there are no more elements. That's it. That's the entire iteration protocol.

An iterable is any object that has an __iter__ method — lists, dicts, sets, strings, files, range objects. An iterator is the object returned by __iter__, which has a __next__ method. Think of the iterable as the book and the iterator as the bookmark. You can have multiple bookmarks in the same book, each tracking a different position.

Here's why this matters: you can make anything iterable by implementing these two methods. Want to iterate over experiment results sorted by accuracy?

class SortedResults:
    def __init__(self, results):
        self.data = sorted(results, key=lambda r: r["accuracy"], reverse=True)
        self.index = 0

    def __iter__(self):
        return self

    def __next__(self):
        if self.index >= len(self.data):
            raise StopIteration
        item = self.data[self.index]
        self.index += 1
        return item

for run in SortedResults(results):
    print(f"accuracy: {run['accuracy']}")

This works, but writing a class with explicit state management for every custom iteration pattern is verbose. There's a far more elegant tool for this, and it's one of Python's best ideas.

Generators — Lazy Evaluation

A generator is a function that produces a sequence of values lazily — one at a time, on demand, instead of computing them all upfront and storing them in a list. You create one by using the yield keyword instead of return.

def top_results(results, threshold=0.85):
    for r in results:
        if r["accuracy"] >= threshold:
            yield r

When you call top_results(results), Python does not execute the function body. It returns a generator object — which implements the iterator protocol automatically. The function body only runs when you call next() on the generator, and it runs only until it hits the next yield, at which point it suspends, remembering its entire state (local variables, where it paused), and hands back the yielded value.

gen = top_results(results, threshold=0.85)
print(next(gen))  # runs until first yield: {"lr": 0.001, "epochs": 20, "accuracy": 0.91}
print(next(gen))  # resumes, runs to next yield: {"lr": 0.01, "epochs": 50, "accuracy": 0.89}
print(next(gen))  # no more matches — raises StopIteration

The magic is that the function's state — every local variable, the position in the loop — is preserved between calls to next(). The function is frozen mid-execution, like pausing a movie.

Why does this matter? Memory. When you're processing a dataset that's too large to fit in memory — millions of training examples, gigabytes of text — a generator lets you process one item (or one batch) at a time without ever loading everything. This is exactly how PyTorch's DataLoader works under the hood: it uses an iterator that yields batches lazily.

def batch_generator(data, batch_size=32):
    batch = []
    for item in data:
        batch.append(item)
        if len(batch) == batch_size:
            yield batch
            batch = []
    if batch:
        yield batch    # don't forget the last incomplete batch

for batch in batch_generator(huge_dataset, batch_size=64):
    model.train_on_batch(batch)

I still occasionally get tripped up by one generator behavior: they can only be iterated once. After a generator is exhausted, it's done. Calling next() again raises StopIteration. Converting to a list with list(gen) exhausts it. Passing an exhausted generator to another for loop produces nothing. No error, no warning — an empty loop. This has caused me more silent bugs than I'd like to admit.

gen = top_results(results)
first_pass = list(gen)       # [... results ...]
second_pass = list(gen)      # [] — empty! generator is spent

You now have a mental model of how Python actually works: variables are names bound to objects, objects are mutable or immutable, the four core containers each serve a specific purpose based on the operations you need to be fast, and iteration is a protocol that generators implement elegantly for lazy, memory-efficient processing.

With these tools you can write real, working Python for ML — load data, process it in batches, choose the right containers, and avoid the aliasing bugs that haunt production systems. That's genuinely useful.

But it doesn't tell the complete story. We haven't talked about the patterns that let you write Python that's maintainable at scale — decorators that add behavior without modifying functions, context managers that guarantee resource cleanup, the EAFP philosophy that makes Python's error handling different from every other language. These are the things that separate scripts from production code, and they're what senior engineers probe for in interviews.

The short version: functions are objects you can pass around, decorators wrap functions to add behavior, context managers ensure cleanup, and Python prefers trying things and handling failures over checking conditions upfront. There. You're about 60% of the way there.

Functions Are Objects

This is one of those ideas that sounds academic until you see what it makes possible. In Python, a function is an object — an instance of the function type, living on the heap like any other object. It has attributes (__name__, __doc__, __defaults__). You can assign it to a variable, put it in a list, pass it as an argument to another function, return it from a function.

This property is called being a first-class citizen, and it's the foundation of everything that follows — closures, decorators, higher-order functions, callbacks.

def square(x):
    return x * x

def apply(func, value):
    return func(value)

print(apply(square, 5))   # 25 — we passed a function as an argument

We already used this when we wrote sorted(results, key=lambda r: r["accuracy"]). The key parameter accepts a function. We passed a lambda — an anonymous function expression — as the sorting criterion. The sorted function calls our lambda on each element to decide the ordering.

A lambda is nothing special. It's a shorthand for defining a function that consists of a single expression. lambda r: r["accuracy"] is exactly equivalent to:

def get_accuracy(r):
    return r["accuracy"]

The only difference is that lambda doesn't bind a name. It creates the function object and hands it over, unnamed. Use lambda for short, throwaway functions. Use def for anything more than one expression — readability matters more than cleverness.

Functions being objects is interesting on its own, but the real power emerges when a function remembers variables from the scope where it was created. That's a closure.

Closures — Captured State

A closure is a function that remembers variables from the enclosing scope even after that scope has finished executing. The function "closes over" those variables, carrying them along wherever it goes.

def make_threshold_filter(min_accuracy):
    def passes(result):
        return result["accuracy"] >= min_accuracy
    return passes

is_good = make_threshold_filter(0.85)
print(is_good(results[0]))  # False (0.82 < 0.85)
print(is_good(results[1]))  # True (0.91 >= 0.85)

When make_threshold_filter(0.85) is called, it creates the inner function passes and returns it. At this point, make_threshold_filter has finished executing — its local scope is gone. But passes still has access to min_accuracy because Python stores it in a cell object attached to the function. The inner function carries its environment with it.

There's a gotcha here that catches even experienced developers. Closures capture the variable, not the value. This is called late binding:

filters = []
for threshold in [0.7, 0.8, 0.9]:
    filters.append(lambda r: r["accuracy"] >= threshold)

# You'd expect three different filters. But:
print(filters[0]({"accuracy": 0.75}))  # False — threshold is 0.9, not 0.7!

All three lambdas closed over the same variable threshold, not the value it held at each iteration. By the time you call any of them, threshold has its final loop value of 0.9. The fix is to capture the current value as a default argument:

filters = []
for threshold in [0.7, 0.8, 0.9]:
    filters.append(lambda r, t=threshold: r["accuracy"] >= t)
    # t=threshold evaluates at definition time, freezing the value

Remember our earlier discussion about default arguments being evaluated at definition time? That behavior, which caused the mutable default argument bug, is the same mechanism that fixes the late binding problem here. Same rule, opposite consequence. Python is consistent — it's our mental models that are inconsistent.

Closures are the engine that makes decorators work.

Decorators — Wrapping Behavior

A decorator is a function that takes another function as input and returns a new function that usually extends or modifies the original's behavior. Think of it as gift wrapping — the gift (original function) stays the same, but the wrapper adds something around it.

Suppose we want to time every experiment in our tracker:

import time

def timed(func):
    def wrapper(*args, **kwargs):
        start = time.time()
        result = func(*args, **kwargs)
        elapsed = time.time() - start
        print(f"{func.__name__} took {elapsed:.2f}s")
        return result
    return wrapper

The timed function receives our original function, creates a wrapper that records the start time, calls the original, measures the elapsed time, prints it, and returns the original's result. The wrapper is a closure — it closes over func.

We can apply it with the @ syntax:

@timed
def train_model(lr, epochs):
    # ... training code ...
    return {"lr": lr, "epochs": epochs, "accuracy": 0.91}

result = train_model(0.001, 20)
# prints: train_model took 3.42s

The @timed line is pure syntax sugar. It's exactly equivalent to writing:

train_model = timed(train_model)

After decoration, the name train_model no longer points to the original function. It points to wrapper. This means train_model.__name__ would return "wrapper", not "train_model" — confusing for debugging. The fix is functools.wraps, which copies the original function's metadata onto the wrapper:

from functools import wraps

def timed(func):
    @wraps(func)
    def wrapper(*args, **kwargs):
        start = time.time()
        result = func(*args, **kwargs)
        elapsed = time.time() - start
        print(f"{func.__name__} took {elapsed:.2f}s")
        return result
    return wrapper

Always use @wraps. It preserves __name__, __doc__, and __module__, which tools like debuggers, logging frameworks, and documentation generators rely on.

In production ML code, decorators show up everywhere. @torch.no_grad() disables gradient computation for inference. @functools.lru_cache memoizes expensive computations. Custom @retry decorators handle flaky API calls to data services. The pattern is always the same: take a function, wrap it, add behavior around the edges.

Our gift-wrapping analogy carries forward: you can stack decorators. Each one wraps the result of the previous one. @decorator_a above @decorator_b means func = decorator_a(decorator_b(func)) — the outermost wrapper runs first.

Decorators let us add behavior without modifying the function itself. But there are other patterns of "add behavior around something" in Python, and the most important one involves resource management.

Context Managers — Resource Safety

Every production system manages resources — files, database connections, GPU memory, network sockets, locks. The pattern is always the same: acquire the resource, do something with it, release it — even if something goes wrong in the middle. Especially if something goes wrong.

Python's with statement and the context manager protocol handle this. You've seen it with files:

with open("experiments.json", "w") as f:
    json.dump(results, f)

When execution enters the with block, Python calls the context manager's __enter__ method (which opens the file and returns the file object). When execution exits the block — whether normally, via an exception, or via a return statement — Python calls __exit__ (which closes the file). The cleanup is guaranteed.

Without with, you'd need a try/finally block:

f = open("experiments.json", "w")
try:
    json.dump(results, f)
finally:
    f.close()

Same result, more typing, easier to forget. The with statement is not syntactic sugar for convenience — it's a safety mechanism. In long-running ML training scripts, a forgotten f.close() can mean file handles leaking over hours, eventually crashing your process when the OS runs out of file descriptors.

You can write your own context managers. For our experiment tracker, maybe we want to log the start and end of each experiment:

from contextlib import contextmanager

@contextmanager
def experiment_context(name):
    print(f"Starting experiment: {name}")
    start = time.time()
    try:
        yield    # everything inside the 'with' block runs here
    finally:
        elapsed = time.time() - start
        print(f"Experiment {name} finished in {elapsed:.2f}s")

with experiment_context("baseline"):
    result = train_model(lr=0.01, epochs=10)

The @contextmanager decorator from contextlib is itself a beautiful example of generators meeting decorators. The code before yield runs on entry, the code after yield (in the finally block) runs on exit. Every concept we've built so far — generators, decorators, closures — is working together here.

In production ML, context managers appear as torch.no_grad() for disabling gradients during inference, torch.cuda.amp.autocast() for mixed precision training, database connection pools, and distributed training synchronization primitives. The pattern is always: set up, yield control, tear down safely.

Exception Handling — The EAFP Philosophy

Most languages follow LBYL — "Look Before You Leap." Check if the file exists, then open it. Check if the key is in the dictionary, then access it. Python takes the opposite approach: EAFP — "Easier to Ask Forgiveness than Permission." Try the operation, and handle the exception if it fails.

# LBYL style (common in other languages)
if "accuracy" in result:
    acc = result["accuracy"]
else:
    acc = None

# EAFP style (Pythonic)
try:
    acc = result["accuracy"]
except KeyError:
    acc = None

Why does Python prefer EAFP? Three reasons. First, it's often faster — in the common case where the key exists, there's no check overhead. Second, it handles race conditions — between your check and your access, another thread could modify the data. Third, it maps naturally to Python's dynamic nature, where you don't always know the type or structure of an object ahead of time.

For production code, the critical rules are:

Never use bare except: without specifying the exception type. A bare except catches everything, including KeyboardInterrupt and SystemExit, making your program impossible to kill cleanly. Always catch the specific exception you expect.

# Dangerous — catches everything, hides real bugs
try:
    result = train_model()
except:
    pass

# Correct — catches only what you expect
try:
    result = train_model()
except ValueError as e:
    logger.error(f"Invalid hyperparameter: {e}")
    raise    # re-raise after logging

Never silently swallow exceptions. If you catch an exception, either handle it (log, retry, fallback) or re-raise it. A silent except: pass is how bugs become mysteries that take days to track down.

Custom exceptions are worth the effort for any non-trivial project. They make your error handling specific and your error messages actionable:

class ExperimentFailedError(Exception):
    def __init__(self, experiment_name, reason):
        self.experiment_name = experiment_name
        self.reason = reason
        super().__init__(f"Experiment '{experiment_name}' failed: {reason}")

I'll be honest — I spent years writing generic except Exception blocks in ML scripts, thinking that detailed exception handling was overkill for research code. Then I spent a weekend debugging a training run that had silently caught and ignored a CUDA out-of-memory error, continued "training" on meaningless tensors, and produced results that looked plausible but were garbage. Custom exceptions would have caught it in seconds.

Comprehensions and Pythonic Patterns

Python offers a compact syntax for creating new collections from existing ones. A list comprehension is a single expression that produces a list:

# instead of:
good_results = []
for r in results:
    if r["accuracy"] > 0.85:
        good_results.append(r["accuracy"])

# write:
good_results = [r["accuracy"] for r in results if r["accuracy"] > 0.85]

The same pattern works for sets and dicts:

# set comprehension — unique learning rates
unique_lrs = {r["lr"] for r in results}

# dict comprehension — map config to score
scores_by_lr = {r["lr"]: r["accuracy"] for r in results}

Replace the brackets with parentheses and you get a generator expression — lazy, memory-efficient, produces values on demand instead of building the full collection:

# generator expression — doesn't build a list in memory
total = sum(r["accuracy"] for r in results)

Use list comprehensions when you need to index into the result or iterate multiple times. Use generator expressions when you're feeding the result into a function that consumes it once (like sum, max, any, all). The difference is invisible for small data, but for millions of items, a generator uses kilobytes where a list comprehension would use gigabytes.

A few more patterns worth having in your toolkit:

Unpacking lets you assign multiple values at once. Extended unpacking with * captures the rest:

best, *rest = sorted(results, key=lambda r: r["accuracy"], reverse=True)
# best is the top result, rest is everything else

The walrus operator (:=), introduced in Python 3.8, lets you assign and use a value within an expression. It's useful in comprehensions where you'd otherwise compute the same thing twice:

# without walrus — computes expensive_metric twice
good = [expensive_metric(r) for r in results if expensive_metric(r) > 0.9]

# with walrus — computes once, names it 'm'
good = [m for r in results if (m := expensive_metric(r)) > 0.9]

f-strings, available since Python 3.6, are the cleanest way to build strings with embedded expressions:

for r in results:
    print(f"lr={r['lr']}, epochs={r['epochs']}, acc={r['accuracy']:.2%}")

The :.2% inside the f-string is a format specifier — it formats the number as a percentage with two decimal places. f-strings can contain any valid Python expression, including function calls and method invocations.

Type Hints — Why They Matter at Scale

Python is dynamically typed. You can pass anything to any function, and Python won't complain until it actually tries to do something that doesn't work. This flexibility is liberating for small scripts and exploratory code. For a 500,000-line ML codebase with 40 contributors, it's terrifying.

Type hints (also called type annotations) are optional annotations that document what types a function expects and returns. They don't change runtime behavior — Python ignores them. But tools like mypy, pyright, and pyre can analyze them statically, before the code ever runs, catching bugs that would otherwise surface only in production.

from typing import Optional

def best_result(results: list[dict], metric: str = "accuracy") -> Optional[dict]:
    if not results:
        return None
    return max(results, key=lambda r: r[metric])

The type annotations say: this function takes a list of dicts and an optional string, and returns either a dict or None. If someone passes an integer where results should be, mypy catches it before anyone runs the code.

For ML codebases, type hints are especially valuable because the data flowing through your pipeline is complex — tensors of specific shapes, nested dicts of configs, optional fields that may or may not be present. The Protocol class from typing lets you define structural types — "any object with a predict method" — without requiring inheritance. This is Python's way of formalizing duck typing: if it has a predict method, it's a predictor, regardless of its class hierarchy.

from typing import Protocol

class Predictor(Protocol):
    def predict(self, X) -> list: ...

def evaluate(model: Predictor, test_data) -> float:
    predictions = model.predict(test_data)
    # ... compute metrics ...
    return accuracy

Any object with a predict method that matches this signature — a scikit-learn model, a PyTorch wrapper, a custom class — will satisfy the type checker. No base class needed. I'm still developing my intuition for when Protocol is the right choice versus an abstract base class, but the general guideline is: use Protocol when you don't control the classes you're type-checking against.
