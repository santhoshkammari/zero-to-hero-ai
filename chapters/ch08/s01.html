Optimization Algorithms

Chapter 8: Training Deep Networks Section 1 of 16 ~18 min read
TL;DR

Training a neural network means finding weights that minimize a loss function. Gradient descent is the workhorse, but vanilla gradient descent is painfully slow. Momentum helps the optimizer barrel through flat regions. Adaptive methods like Adam give each parameter its own learning rate. AdamW fixes a subtle bug in how Adam handles weight decay. No single optimizer wins every task — SGD with momentum often generalizes better for vision, while AdamW dominates for transformers. Knowing why matters more than memorizing formulas.

Why Optimization Matters

Your model is only as good as the minimum it finds. You could design the most expressive architecture in the world, but if your optimizer gets stuck in a bad region of the loss landscape, performance will be mediocre. Optimization is the engine that turns a randomly initialized network into something useful. Everything else — architecture, data, regularization — is fuel. Without a good engine, the fuel doesn't matter.

The core idea is straightforward. You have a loss function L(θ) that measures how wrong your model is. You want to find parameters θ that make L as small as possible. You can't solve this analytically (the function is too complex), so you use an iterative approach: compute the gradient, take a step in the direction that reduces the loss, repeat. The details of that step — how big, in what direction, with what memory of past steps — are what distinguish one optimizer from another.

Gradient Descent Variants

Batch, Stochastic, and Mini-Batch

Batch gradient descent computes the gradient using the entire training set before making one update. The gradient estimate is accurate, but the cost is enormous. If you have a million images, you process all million before adjusting a single weight. On modern datasets, this is impractical.

Stochastic gradient descent (SGD) goes to the other extreme: it updates weights after every single example. The gradient is noisy — one image might push weights in a misleading direction — but you make progress thousands of times faster. That noise turns out to be a feature, not a bug. It helps the optimizer escape shallow local minima and find flatter regions of the loss landscape that generalize better.

Mini-batch gradient descent splits the difference. You sample a batch of, say, 32 or 64 examples, compute the average gradient over that batch, and update. This is what everyone actually uses. The batch size matters: larger batches give more stable gradients but can converge to sharper minima. Smaller batches introduce more noise but often find solutions that generalize better. A typical starting point is 32–256, constrained by GPU memory.

import torch
import torch.nn as nn

model = nn.Linear(784, 10)
criterion = nn.CrossEntropyLoss()

# Vanilla SGD (mini-batch in practice)
optimizer = torch.optim.SGD(model.parameters(), lr=0.01)

for epoch in range(10):
    for x_batch, y_batch in train_loader:       # mini-batches
        optimizer.zero_grad()                     # clear old gradients
        loss = criterion(model(x_batch), y_batch)
        loss.backward()                           # compute gradients
        optimizer.step()                          # update weights

Momentum-Based Methods

Classical Momentum

Imagine pushing a ball down a hilly landscape. Without momentum, the ball stops the instant the ground flattens out, even if there's a steep valley just ahead. With momentum, the ball accumulates speed from past slopes and rolls through flat patches or small bumps.

Momentum adds exactly this to SGD. Instead of using only the current gradient, you maintain a running average of past gradients called the velocity. The update rule becomes:

# Momentum update (conceptual)
# v = momentum * v_prev + lr * gradient
# θ = θ - v

# In PyTorch: just add momentum parameter
optimizer = torch.optim.SGD(model.parameters(), lr=0.01, momentum=0.9)

The hyperparameter momentum (typically 0.9) controls how much past gradients influence the current step. A value of 0.9 means the velocity is 90% memory and 10% new gradient. This dramatically accelerates convergence in narrow valleys — situations where the gradient oscillates back and forth along one dimension while making slow progress along another. Momentum dampens the oscillations and amplifies the consistent direction.

Nesterov Accelerated Gradient

Nesterov accelerated gradient (NAG) is a smarter version of momentum. The insight: if you already know you're going to move in the direction of the accumulated velocity, why not compute the gradient at the position you'll end up at, rather than where you currently are? It's like looking ahead before deciding how to steer.

In practice, NAG reduces overshooting. When the optimizer is hurtling toward a minimum, standard momentum might overshoot because it computes the gradient at the old position. Nesterov peeks ahead, sees the gradient changing, and corrects course earlier. The improvement is modest but consistent, especially near convergence.

# Nesterov momentum in PyTorch
optimizer = torch.optim.SGD(
    model.parameters(), lr=0.01, momentum=0.9, nesterov=True
)

Adaptive Learning Rate Methods

The methods above use a single learning rate for every parameter. But think about this: some parameters see frequent gradients (common features), while others see sparse gradients (rare features). Should they really learn at the same pace? Adaptive methods say no — each parameter gets its own effective learning rate based on its gradient history.

AdaGrad

AdaGrad (Adaptive Gradient) keeps a running sum of squared gradients for each parameter. Parameters with large accumulated gradients get smaller learning rates; parameters with small accumulated gradients get larger ones. This is excellent for sparse data — rare features get bigger updates when they finally appear.

The fatal flaw: the accumulated sum only grows. It never shrinks. Over time, every parameter's effective learning rate decays to near zero, and training stalls. AdaGrad works well for convex problems and sparse features (like NLP with bag-of-words), but it chokes on deep networks that need to train for many iterations.

RMSprop

RMSprop (Root Mean Square Propagation) fixes AdaGrad's death spiral with one change: instead of accumulating all past squared gradients, it uses an exponential moving average. Recent gradients matter more; ancient ones fade away. The decay factor (commonly 0.99) controls how quickly old information is forgotten.

This means the effective learning rate can increase again if a parameter's recent gradients are small. RMSprop was proposed by Geoffrey Hinton in a Coursera lecture — it was never formally published in a paper, yet it became one of the most widely used optimizers.

Adam

Adam (Adaptive Moment Estimation) combines the best of momentum and RMSprop. It maintains two exponential moving averages: one for the gradient (first moment, like momentum) and one for the squared gradient (second moment, like RMSprop). The update uses the first moment for direction and the second moment for per-parameter scaling.

There's a subtlety. Both moving averages are initialized to zero, which biases them toward zero at the start of training. Adam includes a bias correction step that compensates for this. Without the correction, early updates would be too small.

# Adam in PyTorch
optimizer = torch.optim.Adam(
    model.parameters(),
    lr=1e-3,           # default starting point
    betas=(0.9, 0.999), # (β1 for momentum, β2 for squared gradients)
    eps=1e-8            # prevents division by zero
)

# What Adam does under the hood (simplified):
# m = β1 * m + (1 - β1) * grad          ← first moment (direction)
# v = β2 * v + (1 - β2) * grad²         ← second moment (scaling)
# m_hat = m / (1 - β1^t)                 ← bias correction
# v_hat = v / (1 - β2^t)                 ← bias correction
# θ = θ - lr * m_hat / (√v_hat + ε)      ← update

Adam's defaults work remarkably well. For most problems, lr=1e-3, β1=0.9, β2=0.999 is a solid starting point. This "set it and forget it" quality made Adam the default optimizer for most deep learning work from 2015 onward.

AdamW: Fixing Weight Decay

Here's where things get subtle. L2 regularization adds a penalty λ||θ||² to the loss function. In standard SGD, this is equivalent to weight decay — subtracting a small fraction of the weight at each step. But in Adam, these two things are not equivalent. The adaptive scaling in Adam distorts the L2 penalty differently for each parameter, so what you think is weight decay is actually something weaker and inconsistent.

AdamW (proposed by Loshchilov and Hutter, 2019) fixes this by decoupling the weight decay from the gradient update. Instead of adding the regularization term to the loss, AdamW applies weight decay directly to the parameters after the Adam step. The result: proper, consistent weight decay regardless of adaptive scaling.

# AdamW — the standard for modern training
optimizer = torch.optim.AdamW(
    model.parameters(),
    lr=1e-3,
    weight_decay=0.01   # decoupled weight decay (not L2!)
)

AdamW has become the default optimizer for training transformers (BERT, GPT, ViT). If you're starting a new project today, AdamW is the safe choice.

Other Notable Optimizers

LARS and LAMB

LARS (Layer-wise Adaptive Rate Scaling) adjusts the learning rate for each layer based on a trust ratio: the ratio of the weight norm to the gradient norm. Layers where gradients are disproportionately large relative to weights get a smaller learning rate. This was designed for very large batch training (batch sizes in the thousands), where standard SGD diverges.

LAMB applies the same layer-wise scaling idea on top of Adam instead of SGD. Google used LAMB to train BERT with a batch size of 65,536 in 76 minutes — something standard Adam couldn't handle.

Sharpness-Aware Minimization (SAM)

SAM doesn't just try to find a low point in the loss landscape — it looks for a low point in a flat region. The idea: if you perturb the weights slightly and the loss shoots up, you're in a sharp minimum that probably won't generalize. SAM computes the gradient at a perturbed point (the point of worst loss within a small neighborhood) and then updates from there. This doubles the computational cost (two forward-backward passes per step) but consistently improves generalization, especially for vision models.

Optimizer Selection Guide

There is no universally best optimizer. Here's a practical guide:

The SGD vs Adam Debate

You'll hear experienced practitioners say "SGD generalizes better than Adam." There is truth to this — for CNNs trained from scratch on tasks like ImageNet, SGD with momentum often reaches slightly higher test accuracy. The theory is that SGD's noisier updates bias it toward flatter minima.

But this comes with a cost: SGD requires much more careful tuning of the learning rate schedule. Adam converges faster and is far less sensitive to hyperparameters. For transformers, Adam (or AdamW) is almost always better. The generalization gap that researchers observed with CNNs hasn't materialized in the same way for attention-based models.

The honest advice: start with AdamW. If you're training a CNN and have time for hyperparameter tuning, try SGD with momentum and cosine annealing. Measure on your validation set. Let the numbers decide.

Second-Order Methods: Practical Aspects

All the methods above are first-order — they use only the gradient (first derivative). Second-order methods also use the Hessian matrix (second derivatives), which captures the curvature of the loss landscape. With curvature information, you can take bigger steps in flat directions and smaller steps in steep directions.

The Fisher Information Matrix approximates the Hessian and leads to natural gradient descent, which updates parameters in a way that respects the geometry of the probability distributions the model represents. This sounds elegant — and it is — but the Hessian for a model with millions of parameters is a matrix with trillions of entries. Storing and inverting it is completely impractical.

Approximations exist. L-BFGS approximates the inverse Hessian using gradient history. K-FAC approximates the Fisher as a Kronecker product of smaller matrices. These work well for smaller models or specific architectures but haven't displaced first-order methods for large-scale training. The extra memory and computation usually aren't worth the faster convergence per step.

In practice, most teams never touch second-order methods. AdamW is already partially adaptive (the second moment gives a diagonal approximation of curvature), and that's "good enough" for the vast majority of workloads. Second-order methods remain an active research area, but they live more in papers than in production.

Putting It All Together

import torch
import torch.nn as nn
from torch.optim import SGD, Adam, AdamW
from torch.optim.lr_scheduler import CosineAnnealingLR

# A small CNN for demonstration
model = nn.Sequential(
    nn.Conv2d(3, 32, 3, padding=1), nn.ReLU(), nn.MaxPool2d(2),
    nn.Conv2d(32, 64, 3, padding=1), nn.ReLU(), nn.MaxPool2d(2),
    nn.Flatten(),
    nn.Linear(64 * 8 * 8, 256), nn.ReLU(),
    nn.Linear(256, 10)
)

# Option A: SGD + Momentum + Nesterov (good for CNNs)
opt_sgd = SGD(model.parameters(), lr=0.1, momentum=0.9, nesterov=True)
sched_sgd = CosineAnnealingLR(opt_sgd, T_max=200)

# Option B: AdamW (good for transformers, safe default)
opt_adamw = AdamW(model.parameters(), lr=1e-3, weight_decay=0.01)

# Training loop (using AdamW here)
optimizer = opt_adamw
criterion = nn.CrossEntropyLoss()

for epoch in range(200):
    model.train()
    for images, labels in train_loader:
        optimizer.zero_grad()
        loss = criterion(model(images), labels)
        loss.backward()
        torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
        optimizer.step()

    # Validate and log
    model.eval()
    with torch.no_grad():
        val_loss = sum(
            criterion(model(xb), yb).item() for xb, yb in val_loader
        ) / len(val_loader)
    print(f"Epoch {epoch}: val_loss={val_loss:.4f}")

The optimizer you choose shapes how your model learns. Momentum gives it inertia to push through noise. Adaptive methods give each parameter its own pace. Weight decay keeps the model honest. There's no magic formula — but understanding what each optimizer does and why it exists gives you the judgment to pick the right one for your problem.
