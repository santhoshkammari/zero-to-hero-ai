TL;DR

Train neural network = navigate rugged, high-dimensional loss landscape, find weights that generalize. SGD starts here: compute gradient, step. Momentum gives SGD memory; stops ravine bounce. Adam adapts separate learning rate for every parameter from running gradient stats; model converges without heroic LR tuning. AdamW fixes subtle but important weight decay bug in Adam; use this. For learning rate schedule, modern default: cosine annealing with linear warmup. Optimizer picks direction. Schedule controls stride length. AdamW + cosine + warmup handles 90% of cases.

Most people treat optimizer like dropdown menu: pick Adam, set lr=1e-3, move on. But gap between “model sort of converges” and “model converges to something that generalizes” lives in gradient steps and how you shrink them over time. Same architecture can produce wildly different test accuracy from optimizer and schedule choice.

Plan: start with simplest picture — loss function as terrain — then walk from vanilla SGD to modern AdamW + cosine + warmup recipe behind transformers and LLMs. Running example: tiny image classifier sorting cats, dogs, and birds. Start with toy 2-parameter model so landscape becomes visible. Scale up when intuition is solid.

Before we start: section assumes you already know what gradient is and how backpropagation computes it. If shaky, Chapter 2 (Calculus) and backpropagation section earlier in this book cover it. We won’t re-derive chain rule here.

The Loss Landscape

Tiny model with two learnable weights, w1 and w2, classifies images into cat, dog, or bird. Loss function — number telling you how wrong predictions are — forms surface above (w1, w2) plane. Every point on plane is different weight pair. Height is loss. Training means finding low point.

For two-parameter toy, you can plot surface directly. Looks like terrain — hills, valleys, saddle points, plateaus. Saddle point looks like minimum in one direction, maximum if you turn 90 degrees, like middle of horse saddle. Real networks have millions of parameters and absurdly high-dimensional terrain, but intuition transfers: optimizer is hiker navigating landscape, using different strategy to find low ground.

Key insight: flat minima generalize better than sharp ones. Flat minimum is wide valley — nudge weights slightly, like when evaluating on new data, and loss barely changes. Sharp minimum is narrow spike — perturb weights and loss shoots up. Much of optimizer and schedule design steers model toward wide, flat valleys. Momentum, mini-batch noise, and learning rate schedules help make this happen.

Landscape is non-convex. No clean march toward one global minimum. Many local minima. Many saddle points. Vast plateaus where gradient is nearly zero. Good news: in high dimensions, most local minima are roughly as good as global one. Bad news: saddle points and plateaus can trap naïve optimizer for thousands of steps. Motivation set. Start hiking.

Vanilla SGD: The Simplest Step

Stochastic Gradient Descent is starting point for deep learning optimizers. Idea is simple: compute loss gradient for each weight on mini-batch, then step in opposite direction.

# Vanilla SGD update for one parameter
theta = theta - lr * gradient

That’s whole algorithm. For cat/dog/bird classifier, each training step grabs mini-batch of 32 images, computes current loss, backpropagates for gradient on every weight, then nudges each weight downhill by lr * gradient.

It works. Models train this way. But watch vanilla SGD run and three problems appear fast.

First, every parameter gets same learning rate. Rarely activated early feature detector and heavily used final classifier weight get identical step size. Bad fit. Second, ravine oscillation: if loss surface is steep in one direction and shallow in another, SGD bounces across steep walls and crawls along valley floor. Third, noise. Mini-batch gradients are noisy estimates of true gradient. Vanilla SGD amplifies noise because it has no memory — each step starts from scratch, blind to previous steps.

Back to terrain analogy: vanilla SGD is hiker who checks slope under feet, takes one downhill step, forgets everything, repeats. No memory of travel direction. No sense of scale. Enough motivation for first big upgrade.

SGD + Momentum: Giving the Ball a Memory

Replace forgetful hiker with heavy ball rolling downhill. Ball does not stop and re-evaluate every point — it builds speed in direction it keeps rolling. If terrain oscillates side-to-side, ball’s momentum cancels those jolts. If terrain slopes one way, ball accelerates.

That’s momentum. Add velocity vector v that stores exponential moving average of past gradients. Each step, velocity keeps fraction of old direction and adds new gradient.

# SGD with momentum
v = momentum * v + gradient        # accumulate velocity
theta = theta - lr * v             # step using velocity, not raw gradient

momentum coefficient is usually 0.9, so 90% of previous velocity survives. Effect on cat/dog/bird classifier is dramatic: where gradient keeps flipping sign, like ravine walls, velocity averages toward zero and damps oscillation. Where gradient points same way, like valley floor, velocity builds and speeds progress. On toy surface, momentum turns zigzag path into near-straight line toward minimum.

Nesterov Accelerated Gradient

Yurii Nesterov added clever twist: don’t compute gradient at current position. First look ahead to where momentum would carry you. Compute gradient there.

# Nesterov momentum (conceptual)
lookahead = theta + momentum * v
gradient = compute_gradient(lookahead)
v = momentum * v + gradient
theta = theta - lr * v

Why help? Standard momentum can overshoot — ball rolls past minimum because velocity piled up. Nesterov says: peek ahead. If uphill coming, pull back. Result: corrective, anticipatory step. Improvement over standard momentum is usually small but consistent. PyTorch makes it one flag:

optimizer = torch.optim.SGD(
    model.parameters(), lr=0.1, momentum=0.9, nesterov=True
)

SGD with momentum, Nesterov or not, remains workhorse for vision models. Many ResNet and EfficientNet papers use SGD+momentum with carefully tuned learning rate schedule. It often finds flatter minima than Adam-family optimizers, which often means better generalization. Catch: LR schedule tuning takes effort. High reward. High effort. That trade-off pushed next optimizer wave.

Limitation that motivates what’s next: momentum fixes oscillation, but every parameter still gets same learning rate. Parameter hit by large, frequent gradients and parameter seeing tiny, rare gradients both step by lr. We need per-parameter adaptation.

The Adaptive Revolution: From Adagrad to RMSprop

In 2011, John Duchi and collaborators published Adagrad (Adaptive Gradient). Big idea: no single global learning rate. Use per-parameter learning rate that adapts from that parameter’s gradient history.

Adagrad accumulates sum of squared gradients for each parameter in variable G. Update divides learning rate by √G:

# Adagrad (conceptual, one parameter)
G = G + gradient ** 2
theta = theta - (lr / (sqrt(G) + epsilon)) * gradient

For cat/dog/bird classifier, imagine two parameters: one in final dense layer seeing strong gradients every batch, one in early convolutional filter activating only on rare edge patterns. Adagrad gives first parameter small effective LR because G is huge. Second parameter gets large effective LR because G is small. Optimizer automatically gives more learning to parameters that need it. Big win for sparse problems — NLP embeddings and recommender systems.

But Adagrad has fatal flaw: G only grows. Sum never resets. Over training, G gets bigger, effective learning rate shrinks toward zero, optimizer almost stops learning. For dense networks trained many epochs, this breaks things.

Geoffrey Hinton proposed fix in 2012, in unpublished Coursera lecture. Canonical reference for RMSprop is lecture slides, not peer-reviewed paper. Fix: don’t sum squared gradients forever. Use exponential moving average of squared gradients. Recent history matters more. Old history fades.

# RMSprop (conceptual, one parameter)
G = decay * G + (1 - decay) * gradient ** 2
theta = theta - (lr / (sqrt(G) + epsilon)) * gradient

With decay = 0.99, gradients from 100 steps ago barely matter in G. Effective learning rate can rise again if recent gradients get smaller. Dying-LR problem solved. RMSprop became default adaptive optimizer for a while and still appears in reinforcement learning codebases and RNN training. Next optimizer combined both ideas seen so far — momentum and per-parameter adaptivity — into one algorithm.

Adam: Momentum Meets Adaptivity

Adam (Adaptive Moment Estimation, Kingma & Ba 2014) is probably most used optimizer in deep learning history. Important part is why it works, not memorizing four math lines. Build it from pieces we already have.

Adam tracks two quantities for every parameter. First moment m is exponential moving average of gradient itself — momentum piece. It tells you recent gradient direction. Second moment v is exponential moving average of squared gradient — RMSprop piece. It tells you recent gradient magnitude for that parameter.

# Adam update (simplified, one parameter)
m = beta1 * m + (1 - beta1) * gradient        # first moment (momentum)
v = beta2 * v + (1 - beta2) * gradient ** 2   # second moment (RMSprop)

# Bias correction
m_hat = m / (1 - beta1 ** t)
v_hat = v / (1 - beta2 ** t)

theta = theta - lr * m_hat / (sqrt(v_hat) + epsilon)

Division by √v creates per-parameter magic. Parameters receiving large, frequent gradients build large v, which shrinks effective step size. Optimizer moves cautiously on already well-constrained weights. Parameters with rare or small gradients keep small v, so effective step size gets larger. Optimizer explores harder where signal is weak. That automatic per-parameter LR scaling is why Adam often converges faster than SGD with less tuning.

Bias Correction: The Startup Fix

Both m and v start at zero. In first few steps, both are biased toward zero because history is tiny. Without correction, effective learning rate early in training would be too small and model would barely move.

Bias correction term m / (1 - β₁⊃t) fixes this. At step t=1 with β₁=0.9, denominator is 1 - 0.9 = 0.1, so m gets multiplied by 10×. By step t=20, correction is tiny. Startup fix. Matters early. Fades fast.

The Default Hyperparameters

Original paper proposed defaults that survived a decade of scrutiny:

  • β₁ = 0.9 — momentum decay for the first moment
  • β₂ = 0.999 — decay for the second moment (slower, so the magnitude estimate is more stable)
  • ε = 1e-8 — prevents division by zero
  • lr = 0.001 — starting point; almost always needs a schedule on top
optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)
The fine-tuning trap: Adam’s default LR of 1e-3 is for training from scratch. If you fine-tune pretrained model, like BERT for text classification or pretrained ResNet for cat/dog/bird classifier, 1e-3 can blast pretrained weights out of good loss-landscape region. Use 1e-5 to 5e-5 for fine-tuning. Wrong LR can make fine-tuned model worse than frozen pretrained one.

Limitation: Adam combines momentum and adaptivity well, but subtle problem remains in how it interacts with regularization. Next stop: optimizer you should use.

AdamW: What You Should Actually Use

AdamW is not “Adam with weight decay added on.” It fixes real bug in how original Adam handles regularization. Difference matters.

The Bug in Adam + L2

Weight decay is standard regularization: penalize large weights by adding term proportional to weight magnitude to loss, pushing weights smaller. In vanilla SGD, weight decay and L2 regularization are mathematically equivalent — adding λ · θ² to loss produces gradient term 2λθ, which matches directly shrinking each weight by fraction λ each step.

But in Adam, equivalence breaks. If you implement weight decay as L2 regularization, weight decay gradient 2λθ flows through Adam’s adaptive mechanism — it gets divided by √v like everything else. Result: parameters with large accumulated squared gradients, large v, get weaker weight decay. Regularization stops being uniform. Large weights in high-gradient regions escape penalty. Optimizer fights regularizer.

The Fix: Decoupled Weight Decay

Loshchilov and Hutter (2019) proposed clean fix: apply weight decay directly to weights, completely outside gradient-based update. Keep it out of moments. Keep adaptive scaling away from it.

# Adam + L2 (the broken way)
gradient = gradient + weight_decay * theta     # decay inside gradient
m = beta1 * m + (1 - beta1) * gradient        # decay pollutes moments
v = beta2 * v + (1 - beta2) * gradient ** 2
theta = theta - lr * m_hat / (sqrt(v_hat) + eps)

# AdamW (the correct way)
m = beta1 * m + (1 - beta1) * gradient        # clean gradient only
v = beta2 * v + (1 - beta2) * gradient ** 2
theta = theta - lr * m_hat / (sqrt(v_hat) + eps)   # adaptive step
theta = theta - lr * weight_decay * theta           # decay applied separately

Code difference looks tiny — one moved line. Training effect is not tiny. With AdamW, regularization does what you intend: it pushes all weights toward zero uniformly, regardless of gradient history. Model generalizes better because regularization stays consistent.

# AdamW in PyTorch — this should be your default
optimizer = torch.optim.AdamW(
    model.parameters(),
    lr=1e-3,
    weight_decay=0.01    # typical range: 0.01 to 0.1
)

If someone says “Adam” in modern context, they usually mean AdamW. It is default optimizer for transformers, large language models, and most modern architectures. Virtually every GPT, BERT, and Vision Transformer paper uses AdamW. Use torch.optim.AdamW, not torch.optim.Adam.

Learning Rate Schedules: Why They Matter

Optimizer gives direction of step. Learning rate schedule controls how far you step, and how distance changes during training. Wrong schedule is common reason model trains but fails to generalize.

Back to hiking analogy. Early in training, you are on high plateau — random weights, terrible loss, need big strides to get somewhere useful. Late in training, you are near good valley — big strides overshoot and bounce off walls. You need tiny steps into flat bottom.

Fixed learning rate is compromise that is almost never optimal. Set it high for early progress and you overshoot later. Set it low for precise convergence and first 80% of training is wasted. Learning rate schedule automates transition: large steps early, small steps later.

Schedule is not nice-to-have. Schedule is load-bearing infrastructure. Break it into pieces.

Warmup: Surviving the First Few Steps

First training steps are ugly. Weights are randomly initialized, so gradients are mostly noise in arbitrary directions. If you use Adam, moment estimates m and v start at zero and have not converged yet — bias correction helps, but estimates are still unreliable. If you use batch normalization, running statistics are not stable yet.

Large steps on garbage gradients cause early training instability: loss spikes, NaNs appear, model sometimes never recovers. Linear warmup fixes this: start with very small learning rate and linearly ramp to target LR over set number of steps.

warmup_steps = 1000
target_lr = 1e-3

for step in range(total_steps):
    if step < warmup_steps:
        lr = target_lr * (step / warmup_steps)
    else:
        lr = decay_schedule(step)   # cosine, step decay, etc.

    for param_group in optimizer.param_groups:
        param_group['lr'] = lr

At step 0, LR is almost zero. At step 500, it is half target. At step 1000, it reaches full target LR, then decay schedule takes over. Ramp gives Adam’s moment estimates time to become useful, gives batch norm statistics time to stabilize, and stops one garbage gradient from causing irreversible damage.

Typical warmup length is 5–10% of total training steps. For 100-epoch ImageNet run, that is roughly 5 epochs. For transformer fine-tuning, maybe 500–2000 steps. Warmup is especially critical for transformer architectures — attention mechanism is sensitive to initialization, and large early learning rates can collapse attention weights. Original “Attention Is All You Need” paper used warmup for reason. Every major LLM training run since followed.

Cosine Annealing: The Modern Default

After warmup, you need decay strategy for learning rate. Most popular modern choice is cosine annealing: LR follows cosine curve from peak value down to minimum, often near zero.

Formula is straightforward. If t is current step and T is total number of steps:

# Cosine annealing formula
lr_t = lr_min + 0.5 * (lr_max - lr_min) * (1 + cos(pi * t / T))

At t=0, cosine is 1, so lr_t = lr_max. At t=T, cosine is −1, so lr_t = lr_min. Between them, decay stays smooth and spends more time at moderate learning rates than at either extreme.

Why does this beat older step decay, dropping LR by 10× at fixed epoch milestones? Step decay creates discontinuities — model cruises at one LR, then rug gets pulled. Training dynamics change abruptly, and first few epochs after drop often get wasted while optimizer re-adjusts. Cosine annealing avoids these shocks. Curve stays smooth, model spends more time at intermediate learning rates where updates still matter, and gentle tail gives many steps to settle into flat region of landscape.

# Cosine annealing in PyTorch
scheduler = torch.optim.lr_scheduler.CosineAnnealingLR(
    optimizer,
    T_max=total_epochs,   # one full cosine half-cycle
    eta_min=1e-6          # floor LR
)

for epoch in range(total_epochs):
    train_one_epoch(model, optimizer)
    scheduler.step()

Cosine with Warm Restarts

Variant worth knowing: cosine annealing with warm restarts (SGDR, Loshchilov & Hutter 2017). LR decays near zero on cosine curve, then snaps back to initial LR and decays again. Each restart is deliberate shock that can help optimizer escape local minimum. Later cycles are often longer, like doubling period each time, giving model more time to converge.

scheduler = torch.optim.lr_scheduler.CosineAnnealingWarmRestarts(
    optimizer,
    T_0=10,       # steps in the first cycle
    T_mult=2      # each subsequent cycle is 2x longer
)

Warm restarts are less common than plain cosine in transformer training, but they appear in vision tasks and ensemble methods where you want diverse models from different restart points.

One-Cycle Policy: Super-Convergence

Leslie Smith’s one-cycle policy is more aggressive schedule built on counterintuitive insight: high learning rate in middle of training acts like regularizer. High LR stops model from settling into sharp minima too early. It forces more exploration of loss landscape and favors wide, flat valleys that generalize better.

Schedule has three phases: ramp LR from small value to high peak, hold near peak briefly, then anneal it down to value much smaller than start. Momentum cycles inversely — high momentum when LR is low, low momentum when LR is high — because high LR already gives lots of exploration, and extra momentum can turn that into instability.

scheduler = torch.optim.lr_scheduler.OneCycleLR(
    optimizer,
    max_lr=0.01,
    total_steps=total_steps,
    pct_start=0.3,         # 30% of training is the warmup ramp
    anneal_strategy='cos'  # cosine decay after the peak
)

# CRITICAL: step per batch, not per epoch
for batch in train_loader:
    loss = train_step(model, batch, optimizer)
    scheduler.step()
Gotcha that bites everyone once: OneCycleLR counts steps per batch. If you call scheduler.step() per epoch instead, schedule stretches across whole run, LR peak arrives at wrong time, and performance quietly drops. Bug does not crash. Bug gives mediocre results. You blame model.

Smith found that one-cycle policy can reach same final accuracy in far fewer epochs, a phenomenon he called super-convergence. It pairs especially well with SGD+momentum. High-LR phase does something similar to large-batch training — it smooths loss landscape from optimizer’s view and biases trajectory toward flat minima.

Beyond the Defaults: LAMB, LARS, and SWA

Three more optimizers worth knowing, even if you do not use them daily.

LARS (Layer-wise Adaptive Rate Scaling) and LAMB (Layer-wise Adaptive Moments optimizer for Batch training) solve specific problem: with very large batch sizes in distributed training, effective learning rate becomes too aggressive for some layers and too timid for others. LARS, used with SGD, and LAMB, used with Adam, compute separate trust ratio for each layer, scaling learning rate by ratio of layer weight norm to gradient norm. This keeps relative update magnitude consistent across layers regardless of batch size. Google’s BERT famously used LAMB on batches of 65,536 sequences. If you are not doing large-batch distributed training, you usually do not need these. But they appear in large-scale ML system design interviews.

Stochastic Weight Averaging (SWA) takes different path. Instead of optimizing harder, it averages weights from multiple late-training points. Idea comes from flat minima principle: average of several points near minimum likely lands closer to center of flat region than any single point. In practice, train normally until LR has decayed a lot, then keep training with constant or cyclical LR while maintaining running average of weights. At end, use averaged weights for inference.

# SWA in PyTorch (built-in support)
from torch.optim.swa_utils import AveragedModel, SWALR

swa_model = AveragedModel(model)
swa_scheduler = SWALR(optimizer, swa_lr=0.05)

for epoch in range(swa_start, total_epochs):
    train_one_epoch(model, optimizer)
    swa_model.update_parameters(model)
    swa_scheduler.step()

# After training: update batch norm stats for swa_model
torch.optim.swa_utils.update_bn(train_loader, swa_model)

SWA is cheap: no extra optimizer state, minimal compute overhead. It can improve generalization by 0.5–1.5% on vision tasks. Good interview topic. Good last-mile performance trick.

The Practical Recipe

After theory, here is practical default. Recipe works for vast majority of tasks. You will see it in most modern codebases.

import torch
from torch.optim import AdamW
from torch.optim.lr_scheduler import CosineAnnealingLR

model = YourModel()
optimizer = AdamW(model.parameters(), lr=1e-3, weight_decay=0.01)
scheduler = CosineAnnealingLR(optimizer, T_max=num_epochs, eta_min=1e-6)

# With warmup
warmup_steps = int(0.05 * total_steps)   # 5% warmup

for step in range(total_steps):
    if step < warmup_steps:
        lr_scale = step / warmup_steps
        for pg in optimizer.param_groups:
            pg['lr'] = 1e-3 * lr_scale

    loss = train_step(model, batch, optimizer)

    if step >= warmup_steps:
        scheduler.step()

That is AdamW + cosine annealing + linear warmup. It handles 90% of cases. For remaining 10%, use domain-specific adjustments:

Computer vision from scratch (ResNets, ConvNets): SGD + momentum (0.9) + Nesterov, LR=0.1, cosine annealing. Vision has decades of SGD tuning recipes, and SGD still often wins on generalization for these architectures. Weight decay around 1e-4.

Computer vision fine-tuning (pretrained backbone + new head): AdamW, LR=1e-4 to 5e-5, linear decay or short cosine. Pretrained weights are already good, so use gentle updates. Consider differential learning rates — lower LR for early layers, higher for new head.

NLP and Transformers: AdamW, LR=1e-4 to 5e-4, cosine decay with linear warmup (5–10% of total steps). This matches original BERT recipe. It barely changed since.

LLM pre-training: AdamW, LR around 3e-4, cosine decay to 10% of peak, 2000-step warmup. β₂ is sometimes lowered to 0.95 for stability with very long sequences. Weight decay 0.1.

Quick prototyping: Adam, even plain Adam is fine, LR=1e-3, no schedule. Get something running. Validate data pipeline and architecture. Optimize training recipe later.

The checkpoint gotcha: When resuming training from checkpoint, you must save and restore optimizer state (optimizer.state_dict()) and scheduler state (scheduler.state_dict()) along with model weights. If you load only model weights, Adam’s moment estimates reset to zero, scheduler restarts from step 0, warmup runs again, and training quality silently degrades. Model still trains. Model is worse. You may not notice until evaluation.

The LR Finder

Before committing to learning rate, find good starting point empirically. Leslie Smith’s LR finder (2017) works like this: start with tiny LR, train for one epoch while exponentially increasing LR each batch, and record loss at each step. Plot loss versus LR on log scale. Ideal LR is where loss is decreasing fastest — steepest downward slope — not where loss reaches minimum. Minimum usually means LR is already too high for stable long-term training. Steepest descent marks most efficient learning zone.

def lr_finder(model, train_loader, optimizer, criterion,
              lr_start=1e-7, lr_end=10, num_steps=100):
    lrs, losses = [], []
    lr_mult = (lr_end / lr_start) ** (1 / num_steps)
    lr = lr_start

    model.train()
    data_iter = iter(train_loader)

    for step in range(num_steps):
        try:
            batch = next(data_iter)
        except StopIteration:
            data_iter = iter(train_loader)
            batch = next(data_iter)

        x, y = batch
        for pg in optimizer.param_groups:
            pg['lr'] = lr

        optimizer.zero_grad()
        loss = criterion(model(x), y)
        loss.backward()
        optimizer.step()

        lrs.append(lr)
        losses.append(loss.item())
        lr *= lr_mult

        if step > 0 and loss.item() > 4 * min(losses):
            break

    return lrs, losses

Run this. Plot with plt.plot(lrs, losses); plt.xscale('log'). Look for steepest downhill segment. That is your learning rate neighborhood.
