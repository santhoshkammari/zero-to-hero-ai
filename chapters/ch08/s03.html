Normalization Techniques

Chapter 8: Training Deep Networks Section 3 of 16 ~18 min read
TL;DR

Normalization layers stabilize training by keeping activations in a well-behaved range. Batch normalization normalizes across the batch dimension and dominated CNN training for years — but it fails with small batches and sequential models. Layer normalization normalizes across features within a single example, making it batch-size independent and the standard for transformers. Group normalization splits channels into groups and normalizes within each, providing a solid alternative when batch sizes are tiny. Each technique has tradeoffs, and using the wrong one for your architecture can silently hurt performance.

Why Normalization Exists

Deep networks are chains of transformations. Each layer takes the previous layer's output and feeds it through weights, biases, and an activation function. The problem: as weights update during training, the distribution of each layer's inputs shifts. Layer 5 learns to handle inputs from layer 4, but layer 4 is also changing. By the time layer 5 sees the next batch, the inputs look different. It's like trying to hit a target that keeps moving.

Without normalization, this cascading instability forces you to use tiny learning rates and careful initialization. Training is slow and fragile. Normalization layers act as stabilizers — they re-center and re-scale activations at each layer so that subsequent layers always see inputs with consistent statistics. This lets you use higher learning rates, which means faster training and often better final performance.

Batch Normalization

Batch normalization (BN), proposed by Ioffe and Szegedy in 2015, was the first normalization technique to achieve widespread adoption. The core idea: for each feature (or channel in a CNN), compute the mean and variance across all examples in the current mini-batch, then normalize.

The Mechanics

Given a mini-batch of activations, BN does four things:

  1. Compute batch statistics: For each feature, calculate the mean μ_B and variance σ²_B across the batch dimension.
  2. Normalize: Subtract the mean and divide by the standard deviation: x̂ = (x - μ_B) / √(σ²_B + ε). The small constant ε (typically 1e-5) prevents division by zero.
  3. Scale and shift: Apply learnable parameters γ (scale) and β (shift): y = γx̂ + β. These let the network undo the normalization if that's optimal. Without them, you'd be forcing every feature to have zero mean and unit variance, which might destroy useful information.
  4. Track running statistics: Maintain exponential moving averages of the mean and variance for use at inference time.
import torch
import torch.nn as nn

# BN in a CNN: normalizes across (N, H, W) for each channel
class ConvBlock(nn.Module):
    def __init__(self, in_ch, out_ch):
        super().__init__()
        self.conv = nn.Conv2d(in_ch, out_ch, 3, padding=1)
        self.bn = nn.BatchNorm2d(out_ch)   # one γ, β per channel
        self.relu = nn.ReLU()

    def forward(self, x):
        x = self.conv(x)
        x = self.bn(x)       # normalize before activation (original placement)
        x = self.relu(x)
        return x

# BN in a fully connected network
class FCBlock(nn.Module):
    def __init__(self, in_features, out_features):
        super().__init__()
        self.fc = nn.Linear(in_features, out_features)
        self.bn = nn.BatchNorm1d(out_features)
        self.relu = nn.ReLU()

    def forward(self, x):
        return self.relu(self.bn(self.fc(x)))

Training vs. Inference Mode

This is where BN gets tricky. During training, BN uses the current mini-batch's mean and variance — each batch produces different statistics. During inference, you might process a single example, so there's no "batch" to compute statistics from. BN solves this by maintaining running averages of the mean and variance during training (updated with an exponential moving average, typically with momentum 0.1). At inference, these fixed running statistics replace the batch statistics.

# Critical: switch between training and inference modes
model.train()   # BN uses batch statistics, updates running stats
for x, y in train_loader:
    loss = criterion(model(x), y)
    loss.backward()
    optimizer.step()

model.eval()    # BN uses running (fixed) statistics
with torch.no_grad():
    predictions = model(test_data)

Forgetting to call model.eval() before inference is one of the most common bugs in deep learning. Your model will use batch statistics from whatever single example (or small group) you pass in, producing noisy and inconsistent predictions. If your model works well during training but poorly during evaluation, always check this first.

Internal Covariate Shift: The Original Story and the Reality

Ioffe and Szegedy's original paper motivated BN with the concept of internal covariate shift — the idea that the distribution of layer inputs changes as preceding layers update, making optimization harder. BN was supposed to fix this by normalizing these shifting distributions.

In 2018, Santurkar et al. published a provocative paper ("How Does Batch Normalization Help Optimization?") that challenged this explanation. They showed that BN doesn't actually reduce internal covariate shift (by their measurements). Instead, they argued BN works because it smooths the loss landscape — making the loss function more Lipschitz continuous, with more predictable gradients. A smoother landscape means the optimizer can take larger steps without overshooting.

The practical takeaway: BN works, but not necessarily for the reason its creators originally proposed. The smoothing effect is real and measurable. Whether you call it "reducing covariate shift" or "smoothing the optimization landscape," the end result is the same — faster, more stable training.

BN Failure Modes

BN has a critical dependency: it needs a reasonably large batch size to compute meaningful statistics. When the batch size is small (say, 1–4), the batch mean and variance are terrible estimates of the true statistics. Training becomes noisy and unstable.

This is a real problem in several important settings:

  • Object detection and segmentation: High-resolution images eat up GPU memory, forcing batch sizes of 1–2 per GPU.
  • 3D medical imaging: Volumetric data is huge. Batch size 1 is common.
  • Reinforcement learning: Trajectories come in one at a time.
  • Recurrent networks: Sequence lengths vary, making batching awkward. BN statistics become meaningless across sequences of different lengths.

For distributed training, BN adds another complication. Each GPU computes batch statistics from its local mini-batch. If you have 8 GPUs each processing 4 examples, BN sees batches of 4, not 32. Synchronized BN aggregates statistics across GPUs but adds communication overhead.

# Synchronized BN for multi-GPU training
model = nn.SyncBatchNorm.convert_sync_batchnorm(model)
# Now BN statistics are computed across all GPUs

BN Placement: Before or After Activation?

The original paper placed BN before the activation function: Conv → BN → ReLU. Some practitioners place it after: Conv → ReLU → BN. Both work. Empirical comparisons show marginal differences. The before-activation placement is more common and is what PyTorch's built-in layers assume, so stick with that unless you have a specific reason to deviate.

BN as Implicit Regularizer

BN has an underappreciated side effect: it acts as a regularizer. Because each example's normalization depends on the other examples in the batch, there's randomness in the normalized values (different batches produce different statistics). This noise has a regularizing effect similar to dropout. In fact, some practitioners find that using BN with dropout simultaneously can be counterproductive — the two sources of noise interfere with each other, particularly because dropout changes the variance of activations, confusing BN's variance estimates.

Layer Normalization

Layer normalization (LN), proposed by Ba, Kiros, and Hinton in 2016, takes a fundamentally different approach. Instead of normalizing across the batch dimension, LN normalizes across the feature dimension for each individual example. Each example is normalized independently — the batch size doesn't matter at all.

How It Works

For a single input with features [x₁, x₂, ..., x_d], LN computes the mean and variance across all d features and normalizes. Like BN, it has learnable γ and β parameters.

# Layer Normalization — the standard for transformers
class TransformerBlock(nn.Module):
    def __init__(self, d_model=512, n_heads=8, d_ff=2048):
        super().__init__()
        self.attn = nn.MultiheadAttention(d_model, n_heads, batch_first=True)
        self.ff = nn.Sequential(
            nn.Linear(d_model, d_ff),
            nn.GELU(),
            nn.Linear(d_ff, d_model)
        )
        self.ln1 = nn.LayerNorm(d_model)   # normalizes across d_model features
        self.ln2 = nn.LayerNorm(d_model)

    def forward(self, x):
        # Pre-norm architecture (GPT-2, modern standard)
        normed = self.ln1(x)
        x = x + self.attn(normed, normed, normed)[0]   # residual connection
        normed = self.ln2(x)
        x = x + self.ff(normed)                          # residual connection
        return x

Why Transformers Use Layer Norm

Transformers process sequences of tokens. Each token is a vector of features, and the sequence length varies across examples. BN would compute statistics across the batch for each feature position — but what does position 47 in sentence A have in common with position 47 in sentence B? Nothing semantically meaningful. LN instead normalizes across the features of each token independently, which makes much more sense for sequential data.

There's also a practical reason. Transformers are often trained with sequences of variable length. BN would need careful handling of padding and masking to compute correct statistics. LN sidesteps all of this because it operates within each example independently.

Training vs. Inference with LN

Unlike BN, LN behaves identically during training and inference. There are no running statistics to track, no mode switching. The computation is the same whether you're processing a batch of 64 or a single example. This eliminates an entire class of bugs (forgetting model.eval() doesn't affect LN) and makes LN friendlier for deployment.

Pre-Norm vs. Post-Norm

The original transformer ("Attention Is All You Need") placed layer norm after the residual connection: x = LN(x + sublayer(x)). This is called post-norm. Modern transformers (GPT-2, GPT-3, LLaMA) place it before: x = x + sublayer(LN(x)). This is pre-norm.

Pre-norm makes training more stable — gradients flow through the residual connection unobstructed, without passing through the normalization layer. Post-norm can achieve slightly better final performance but requires more careful initialization and warmup. Most practitioners use pre-norm because the training stability is worth more than marginal accuracy gains.

Group Normalization

Group normalization (GN), proposed by Wu and He in 2018, is a compromise between BN and LN designed for convolutional networks. It divides channels into groups and normalizes within each group, independently per example.

The Mechanics

If you have 64 channels and use 8 groups, each group contains 8 channels. For each example, GN computes mean and variance across the spatial dimensions (H, W) and the 8 channels within each group. It doesn't touch the batch dimension.

# Group Normalization — great for small batch sizes in CNNs
class GNConvBlock(nn.Module):
    def __init__(self, in_ch, out_ch, num_groups=8):
        super().__init__()
        self.conv = nn.Conv2d(in_ch, out_ch, 3, padding=1)
        # num_groups must divide out_ch evenly
        self.gn = nn.GroupNorm(num_groups=num_groups, num_channels=out_ch)
        self.relu = nn.ReLU()

    def forward(self, x):
        return self.relu(self.gn(self.conv(x)))

# Example: 32 channels, 8 groups → 4 channels per group
block = GNConvBlock(3, 32, num_groups=8)
x = torch.randn(2, 3, 64, 64)  # batch=2, works fine (even batch=1 works)
out = block(x)
print(out.shape)  # torch.Size([2, 32, 64, 64])

When to Use Group Normalization

GN was specifically designed for situations where BN fails — small batch sizes. The original paper showed that GN matches BN's performance on ImageNet when the batch size is 32 or larger, and dramatically outperforms BN when the batch size drops to 2 or 4. For object detection (Mask R-CNN, Faster R-CNN), where per-GPU batch sizes are often 1–2, GN is the standard choice.

The number of groups is a hyperparameter. Common choices are 32 groups (the default in many detection frameworks) or groups sized so each group has 16–32 channels. The performance is relatively insensitive to this choice as long as the number isn't extreme.

Other Normalization Techniques

Instance Normalization

Instance normalization (IN) is the extreme case of group normalization where each channel is its own group. It normalizes across spatial dimensions (H, W) for each channel of each example independently. IN is primarily used in style transfer and image generation, where you want to remove instance-specific contrast information.

# Instance Norm — each channel normalized independently per example
norm = nn.InstanceNorm2d(64)  # for 64-channel feature maps

RMS Normalization

RMS normalization (RMSNorm) is a simplified version of layer normalization that skips the mean-centering step. It only divides by the root mean square of the activations. This saves computation (no mean subtraction) and has been shown to work comparably to full LN. LLaMA and several modern large language models use RMSNorm instead of LayerNorm.

# RMSNorm (not built into PyTorch, but simple to implement)
class RMSNorm(nn.Module):
    def __init__(self, d_model, eps=1e-6):
        super().__init__()
        self.weight = nn.Parameter(torch.ones(d_model))
        self.eps = eps

    def forward(self, x):
        rms = torch.sqrt(torch.mean(x ** 2, dim=-1, keepdim=True) + self.eps)
        return x / rms * self.weight

Weight Normalization

Weight normalization reparameterizes the weight vectors instead of normalizing activations. Each weight vector is decomposed into a magnitude and a direction: w = g × v/||v||. This decouples the length of the weight vector from its direction, making optimization easier. Weight normalization is simpler and faster than BN but generally less effective. It's occasionally used in generative models (WaveNet) and RL.

The choice of normalization layer is not just a technical detail — it's an architectural decision that interacts with your batch size, model type, and training setup. BN is powerful but fragile. LN is robust but slightly less effective for convolutions. GN is the reliable middle ground for vision tasks with constrained memory. Know which tool fits your situation, and don't copy-paste a normalization choice from a different architecture without thinking about whether the assumptions still hold.
