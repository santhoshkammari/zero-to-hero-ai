PyTorch Training

Chapter 8: Training Deep Networks Section 3 of 16 ~30 min read
TL;DR

PyTorch is the dominant deep learning framework — it won the research world years ago and is now taking over production too. Its core ideas: tensors with GPU support and automatic differentiation, a dynamic computation graph built on-the-fly as you write Python, and nn.Module as the building block for everything. The training loop is explicit and yours to control — no magic hidden behind .fit(). This section gives you the patterns you'll use in every PyTorch project: the training loop, custom datasets, model definition, saving/loading, device management, and inference. If you learn one framework deeply, make it PyTorch.

Why PyTorch

PyTorch won. It's the default in ML research, dominant on arXiv, the backbone of Hugging Face, and increasingly the choice for production systems (via TorchServe, ONNX export, and torch.compile). Three things make it work:

  • Dynamic computation graph (define-by-run): The graph is built as your Python code executes. Every if statement, every for loop, every conditional branch — they're all part of the graph. No separate "compile the graph, then run it" step. This makes debugging trivial: you can set breakpoints, print intermediate tensors, and use standard Python tooling.
  • Pythonic: PyTorch feels like NumPy with superpowers. If you know Python, you'll read PyTorch code naturally. There's no domain-specific language, no graph placeholders, no sessions.
  • The ecosystem: Hugging Face Transformers, PyTorch Lightning, torchvision, torchaudio, torchtext, TIMM, Detectron2 — virtually every major ML library is PyTorch-first.

If you learn one framework deeply, make it PyTorch. Everything else becomes easier once you have this foundation.

Tensors & Autograd

The entire framework rests on two ideas: tensors and automatic differentiation.

A tensor is a multi-dimensional array — conceptually identical to a NumPy ndarray, but with two critical additions: it can live on a GPU, and it can track the operations performed on it so that gradients can be computed automatically.

import torch

# Create tensors — just like NumPy
x = torch.tensor([1.0, 2.0, 3.0])
W = torch.randn(3, 3)                    # random normal
zeros = torch.zeros(2, 4)                # explicit shape
eye = torch.eye(3)                       # identity matrix

# NumPy interop (shared memory — no copy!)
import numpy as np
np_array = np.array([1.0, 2.0, 3.0])
t = torch.from_numpy(np_array)           # numpy → tensor
back = t.numpy()                         # tensor → numpy

# GPU tensors
if torch.cuda.is_available():
    x_gpu = x.to('cuda')                 # move to GPU
    x_gpu = x.cuda()                     # equivalent shorthand

Autograd: Automatic Differentiation

Setting requires_grad=True on a tensor tells PyTorch to track every operation performed on it. This builds a computational graph on the fly — and when you call .backward() on a scalar output, PyTorch walks that graph in reverse (backpropagation) to compute gradients for every tracked tensor.

# Autograd in action
x = torch.tensor(3.0, requires_grad=True)
y = x ** 2 + 2 * x + 1          # y = x² + 2x + 1

y.backward()                     # dy/dx = 2x + 2 = 8.0
print(x.grad)                    # tensor(8.)

# The graph is dynamic — built fresh each forward pass
# This means you can use Python control flow freely:
def weird_function(x):
    if x.item() > 0:
        return x ** 2
    else:
        return x ** 3

x = torch.tensor(2.0, requires_grad=True)
y = weird_function(x)
y.backward()
print(x.grad)                    # tensor(4.) — gradient of x², as expected

The key mental model: every operation you write IS the graph. There's no separate graph definition step. The graph exists only during the forward pass and is consumed during .backward(). Next forward pass? A fresh graph is built. This is why it's called "define-by-run."

The Training Loop — The Pattern You Should Know Cold

Unlike Keras' .fit(), PyTorch gives you the raw loop. This is a feature, not a bug — you have total control over every step. Here's the canonical pattern:

model.train()                    # Enable training mode (dropout, BN)

for epoch in range(num_epochs):
    running_loss = 0.0
    for batch_idx, (inputs, targets) in enumerate(dataloader):
        # 1. Move data to the right device
        inputs, targets = inputs.to(device), targets.to(device)

        # 2. Zero out gradients from the previous step
        optimizer.zero_grad()

        # 3. Forward pass — compute predictions
        outputs = model(inputs)

        # 4. Compute loss
        loss = criterion(outputs, targets)

        # 5. Backward pass — compute gradients
        loss.backward()

        # 6. (Optional) Clip gradients to prevent explosion
        torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)

        # 7. Update parameters
        optimizer.step()

        running_loss += loss.item()

    # 8. Step the learning rate scheduler (once per epoch)
    scheduler.step()

    avg_loss = running_loss / len(dataloader)
    print(f"Epoch {epoch+1}/{num_epochs}, Loss: {avg_loss:.4f}")

Let's dissect every line, because each one matters:

model.train()

Sets the model to training mode. This affects layers that behave differently during training vs. inference: Dropout layers randomly zero out neurons (active during training, disabled during eval), and BatchNorm layers use per-batch statistics (training) vs. running statistics (eval). If you forget this after calling model.eval(), your model will train with dropout disabled and frozen BN stats. Your loss will look weird and you'll spend hours debugging.

inputs.to(device), targets.to(device)

Moves data to the same device as your model (CPU or GPU). If your model is on cuda and your data is on cpu, you get a RuntimeError. Every tensor involved in computation must be on the same device. The .to() call on tensors returns a new tensor (it's not in-place), so you must reassign: inputs = inputs.to(device).

optimizer.zero_grad()

This is the line people forget and then spend an hour debugging. By default, PyTorch accumulates gradients — calling .backward() adds to the existing .grad attributes rather than replacing them. This is actually useful for gradient accumulation (we'll see later), but during normal training you want a clean slate each step. zero_grad() sets all parameter gradients to zero. Call it before the forward pass.

💡 Performance Tip

Use optimizer.zero_grad(set_to_none=True) instead of the default. Instead of filling gradient tensors with zeros, it sets them to None, which is slightly faster and uses less memory. This is the default in PyTorch 2.0+.

outputs = model(inputs)

The forward pass. Calling a model as a function invokes its forward() method (plus some internal hooks). This is where the computational graph is built — every matrix multiply, activation, and normalization creates graph nodes that .backward() will traverse later.

loss = criterion(outputs, targets)

Computes a scalar loss. Common choices: nn.CrossEntropyLoss() for classification (expects raw logits, not softmax), nn.MSELoss() for regression, nn.BCEWithLogitsLoss() for binary classification. The loss tensor is the root node that backward will differentiate from.

loss.backward()

The magic line. Walks the computational graph backward from the loss, computing ∂loss/∂param for every parameter with requires_grad=True. After this call, every param.grad contains the gradient. The graph is then destroyed (freed from memory) — this is why each forward pass builds a fresh graph.

clip_grad_norm_

Optional but increasingly standard. Clips the total norm of all gradients to max_norm. Without this, a single bad batch can produce enormous gradients that blow up your weights. Essential for RNNs and transformers, good practice everywhere.

optimizer.step()

Uses the gradients stored in .grad to update the parameters. For SGD: param = param - lr * param.grad. For Adam: something more complex involving momentum and adaptive learning rates. The optimizer holds references to your model's parameters — that's why you passed model.parameters() when creating it.

scheduler.step()

Updates the learning rate according to the schedule. Typically called once per epoch (some schedulers, like OneCycleLR, are called once per batch). Always call this after optimizer.step().

Dataset & DataLoader

PyTorch separates what your data is (Dataset) from how it's batched, shuffled, and loaded (DataLoader).

Custom Dataset

Inherit from torch.utils.data.Dataset and implement two methods: __len__ and __getitem__.

from torch.utils.data import Dataset, DataLoader
import pandas as pd

class TabularDataset(Dataset):
    """Custom dataset for tabular data from a CSV file."""

    def __init__(self, csv_path, target_col='label'):
        df = pd.read_csv(csv_path)
        self.features = torch.tensor(
            df.drop(columns=[target_col]).values, dtype=torch.float32
        )
        self.targets = torch.tensor(
            df[target_col].values, dtype=torch.long
        )

    def __len__(self):
        return len(self.features)

    def __getitem__(self, idx):
        return self.features[idx], self.targets[idx]


class ImageDataset(Dataset):
    """Custom dataset for image classification."""

    def __init__(self, image_paths, labels, transform=None):
        self.image_paths = image_paths
        self.labels = labels
        self.transform = transform

    def __len__(self):
        return len(self.image_paths)

    def __getitem__(self, idx):
        from PIL import Image
        image = Image.open(self.image_paths[idx]).convert('RGB')
        label = self.labels[idx]

        if self.transform:
            image = self.transform(image)

        return image, label

DataLoader

The DataLoader wraps a Dataset and handles batching, shuffling, and parallel data loading. These are the parameters you'll use constantly:

train_loader = DataLoader(
    dataset=train_dataset,
    batch_size=64,             # samples per batch
    shuffle=True,              # randomize order each epoch (ALWAYS for training)
    num_workers=4,             # parallel data loading processes
    pin_memory=True,           # faster CPU→GPU transfer (use when training on GPU)
    drop_last=True,            # drop incomplete final batch (avoids BN issues)
    persistent_workers=True,   # keep workers alive between epochs (saves startup time)
)

val_loader = DataLoader(
    dataset=val_dataset,
    batch_size=128,            # can be larger — no gradients stored
    shuffle=False,             # no need to shuffle validation data
    num_workers=4,
    pin_memory=True,
)

num_workers is the one that bites people. Set it to the number of CPU cores available (or start with 4 and tune). Setting it too high wastes memory; setting it to 0 means the main process does all loading (slow). On Windows, num_workers > 0 requires wrapping your training code inside an if __name__ == '__main__': guard due to how Python multiprocessing works.

pin_memory=True allocates data in page-locked (pinned) CPU memory, which enables faster async transfers to the GPU. Always use this when training on CUDA devices.

nn.Module — The Building Block

Every model in PyTorch is an nn.Module. Every layer is an nn.Module. Models composed of layers? Also nn.Modules. It's modules all the way down.

The contract is simple: define your layers in __init__, define the computation in forward. PyTorch automatically registers every nn.Module or nn.Parameter you assign as an attribute.

import torch.nn as nn
import torch.nn.functional as F

class ConvClassifier(nn.Module):
    """A clean CNN for image classification."""

    def __init__(self, num_classes=10):
        super().__init__()

        # Feature extraction
        self.features = nn.Sequential(
            nn.Conv2d(3, 32, kernel_size=3, padding=1),
            nn.BatchNorm2d(32),
            nn.ReLU(inplace=True),
            nn.MaxPool2d(2),

            nn.Conv2d(32, 64, kernel_size=3, padding=1),
            nn.BatchNorm2d(64),
            nn.ReLU(inplace=True),
            nn.MaxPool2d(2),

            nn.Conv2d(64, 128, kernel_size=3, padding=1),
            nn.BatchNorm2d(128),
            nn.ReLU(inplace=True),
            nn.AdaptiveAvgPool2d(1),       # global average pooling
        )

        # Classification head
        self.classifier = nn.Sequential(
            nn.Flatten(),
            nn.Dropout(0.5),
            nn.Linear(128, num_classes),
        )

    def forward(self, x):
        x = self.features(x)
        x = self.classifier(x)
        return x                           # return raw logits (not softmax!)


# Instantiate and inspect
model = ConvClassifier(num_classes=10)
print(model)                               # prints full architecture

# model.parameters() gives all learnable params — pass this to the optimizer
optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)

# Count parameters
total_params = sum(p.numel() for p in model.parameters())
trainable_params = sum(p.numel() for p in model.parameters() if p.requires_grad)
print(f"Total: {total_params:,}  Trainable: {trainable_params:,}")

A few important patterns with nn.Module:

  • Always return raw logits from forward() — don't apply softmax. nn.CrossEntropyLoss applies log-softmax internally and is more numerically stable than doing it yourself.
  • Use nn.Sequential for straightforward stacks of layers. Use explicit forward() logic when you need skip connections, multiple inputs, or conditional paths.
  • Assign layers as attributes — if you store them in a plain Python list, PyTorch won't find them. Use nn.ModuleList or nn.ModuleDict for dynamic collections.
# WRONG — parameters won't be registered
class BadModel(nn.Module):
    def __init__(self):
        super().__init__()
        self.layers = [nn.Linear(10, 10) for _ in range(3)]  # plain list!

# RIGHT — use nn.ModuleList
class GoodModel(nn.Module):
    def __init__(self):
        super().__init__()
        self.layers = nn.ModuleList([nn.Linear(10, 10) for _ in range(3)])

    def forward(self, x):
        for layer in self.layers:
            x = F.relu(layer(x))
        return x

Device Management

GPU training requires that your model and data live on the same device. Here's the standard pattern:

# Pick the best available device
device = torch.device('cuda' if torch.cuda.is_available() else 'cpu')
print(f"Using device: {device}")

# For Apple Silicon Macs:
# device = torch.device('mps' if torch.backends.mps.is_available() else 'cpu')

# Move model to device — this IS in-place for modules
model = ConvClassifier(num_classes=10)
model.to(device)                          # modifies model in-place, returns self

# Move data to device — this is NOT in-place for tensors
inputs = torch.randn(16, 3, 32, 32)
inputs = inputs.to(device)               # must reassign! .to() returns a new tensor

# In the training loop, always move batch data:
for inputs, targets in dataloader:
    inputs = inputs.to(device)            # tensor: must reassign
    targets = targets.to(device)
    outputs = model(inputs)               # both on same device — works

Critical subtlety: .to(device) behaves differently for modules and tensors. For nn.Module, it moves all parameters and buffers in-place and returns self. For tensors, it returns a new tensor on the target device — the original tensor is unchanged. This asymmetry trips up everyone at least once.

Saving & Loading

Always save the state_dict, not the model object. Saving the entire model uses Python's pickle, which breaks when you move files between projects, rename classes, or refactor code.

# === Save just the model weights ===
torch.save(model.state_dict(), 'model_weights.pt')

# === Load weights ===
model = ConvClassifier(num_classes=10)     # create model with same architecture
model.load_state_dict(torch.load('model_weights.pt', map_location=device))
model.to(device)
model.eval()

Full Checkpoint (For Resuming Training)

If you want to resume training after a crash or interruption, you need more than just model weights. Save everything:

# === Save a full training checkpoint ===
checkpoint = {
    'epoch': epoch,
    'model_state_dict': model.state_dict(),
    'optimizer_state_dict': optimizer.state_dict(),
    'scheduler_state_dict': scheduler.state_dict(),
    'train_loss': avg_train_loss,
    'val_loss': avg_val_loss,
    'best_val_loss': best_val_loss,
}
torch.save(checkpoint, f'checkpoint_epoch_{epoch}.pt')

# === Resume from checkpoint ===
checkpoint = torch.load('checkpoint_epoch_42.pt', map_location=device)

model = ConvClassifier(num_classes=10)
model.load_state_dict(checkpoint['model_state_dict'])
model.to(device)

optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)
optimizer.load_state_dict(checkpoint['optimizer_state_dict'])

scheduler = torch.optim.lr_scheduler.CosineAnnealingLR(optimizer, T_max=100)
scheduler.load_state_dict(checkpoint['scheduler_state_dict'])

start_epoch = checkpoint['epoch'] + 1
best_val_loss = checkpoint['best_val_loss']

print(f"Resumed from epoch {start_epoch}, best val loss: {best_val_loss:.4f}")

The map_location argument in torch.load() is essential. If you saved a checkpoint on GPU and load on CPU (or a different GPU), map_location=device handles the remapping automatically.

Inference Mode

Running inference correctly requires two things — and forgetting either one is a common source of subtle bugs:

# Both are needed for correct, efficient inference
model.eval()                              # 1. Switch to evaluation mode

with torch.inference_mode():              # 2. Disable gradient computation
    inputs = test_batch.to(device)
    outputs = model(inputs)
    predictions = outputs.argmax(dim=1)

model.eval() changes layer behavior: Dropout is disabled (all neurons active), and BatchNorm uses its stored running mean/variance instead of computing batch statistics. Without this, dropout randomly zeros out neurons during inference — your predictions become nondeterministic and worse.

torch.inference_mode() (or the older torch.no_grad()) disables gradient tracking. Since you don't need gradients for inference, this saves significant memory (no computational graph stored) and speeds up computation. Prefer inference_mode() over no_grad() — it's stricter (tensors created inside can't accidentally flow into gradient computations later) and faster.

💡 eval() vs no_grad() — They Do Different Things

model.eval() changes layer behavior (dropout, batchnorm). torch.no_grad() / torch.inference_mode() disables gradient computation. They are independent and you need both. Using only one and not the other is a bug.

Complete Evaluation Loop

def evaluate(model, dataloader, criterion, device):
    """Standard evaluation loop with metrics."""
    model.eval()
    total_loss = 0.0
    correct = 0
    total = 0

    with torch.inference_mode():
        for inputs, targets in dataloader:
            inputs, targets = inputs.to(device), targets.to(device)

            outputs = model(inputs)
            loss = criterion(outputs, targets)

            total_loss += loss.item() * inputs.size(0)
            _, predicted = outputs.max(dim=1)
            correct += predicted.eq(targets).sum().item()
            total += targets.size(0)

    avg_loss = total_loss / total
    accuracy = correct / total
    return avg_loss, accuracy


# Usage
val_loss, val_acc = evaluate(model, val_loader, criterion, device)
print(f"Val Loss: {val_loss:.4f}, Val Acc: {val_acc:.2%}")

# IMPORTANT: switch back to training mode after evaluation
model.train()

Common Gotchas

⚠️ PyTorch Pitfalls That Will Waste Your Time
  • Forgetting optimizer.zero_grad() — Gradients accumulate by default. Without zeroing, each batch adds to the previous batch's gradients. Your loss will look like it's converging, but the effective gradient is a messy average of many batches. Insidious because it doesn't crash — it just silently degrades performance.
  • Forgetting model.eval() during inference — Dropout is still randomly zeroing out neurons. BatchNorm uses noisy per-batch statistics instead of smooth running averages. Your eval metrics look worse than they should, and they vary between runs.
  • Device mismatch — Model on GPU, data on CPU (or vice versa). You'll get RuntimeError: Expected all tensors to be on the same device. Always check: is everything on the same device?
  • Using torch.no_grad() instead of torch.inference_mode() — no_grad() still creates tensors that could be used in gradient computation later (it just disables tracking). inference_mode() is stricter and faster. Use inference_mode() unless you have a specific reason not to.
  • DataLoader(num_workers > 0) on Windows — Without if __name__ == '__main__': guard, Python's multiprocessing will fail with cryptic errors or infinite spawning. Always wrap your training entry point.
  • .to(device) not reassigned for tensors — Writing tensor.to(device) without tensor = tensor.to(device) does nothing. The original tensor stays on CPU. Modules are in-place; tensors are not.
  • Calling loss.item() inside autograd — loss.item() extracts a Python number, breaking the graph. Fine for logging, but don't use it in computations that need gradients. Use loss.detach() if you need the value as a tensor without graph.
  • Storing batch tensors in a list for logging — If you append loss (not loss.item()) to a list, you're keeping the entire computational graph alive. Memory usage grows linearly with training steps. Always call .item() or .detach() for logging values.

Practical Patterns

Gradient Accumulation

When your GPU can't fit the batch size you want, simulate a larger batch by accumulating gradients across multiple smaller batches:

accumulation_steps = 4                    # effective batch = batch_size × 4
optimizer.zero_grad()

for step, (inputs, targets) in enumerate(dataloader):
    inputs, targets = inputs.to(device), targets.to(device)

    outputs = model(inputs)
    loss = criterion(outputs, targets)
    loss = loss / accumulation_steps      # normalize loss
    loss.backward()                       # gradients accumulate

    if (step + 1) % accumulation_steps == 0:
        torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
        optimizer.step()
        optimizer.zero_grad()

The key insight: because .backward() accumulates gradients by default, you can call it multiple times before calling optimizer.step(). Divide the loss by accumulation_steps to keep the gradient magnitude equivalent to a single large batch.

Mixed Precision Training

Modern GPUs have specialized hardware (Tensor Cores) for float16 operations. Mixed precision uses float16 for forward/backward passes and float32 for weight updates — 2× faster training with minimal accuracy impact:

from torch.cuda.amp import GradScaler, autocast

scaler = GradScaler()

for inputs, targets in dataloader:
    inputs, targets = inputs.to(device), targets.to(device)
    optimizer.zero_grad()

    # Forward pass in float16 (where safe)
    with autocast(device_type='cuda'):
        outputs = model(inputs)
        loss = criterion(outputs, targets)

    # Backward pass — scaler handles float16 gradient underflow
    scaler.scale(loss).backward()
    scaler.unscale_(optimizer)
    torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
    scaler.step(optimizer)
    scaler.update()

We cover mixed precision in detail in the dedicated section — this is just the pattern to recognize.

Reproducibility

If you need deterministic results (for debugging or paper submissions), set all the seeds:

import random
import numpy as np
import torch

def set_seed(seed=42):
    random.seed(seed)
    np.random.seed(seed)
    torch.manual_seed(seed)
    torch.cuda.manual_seed_all(seed)

    # Force deterministic algorithms (may be slower)
    torch.backends.cudnn.deterministic = True
    torch.backends.cudnn.benchmark = False
    # PyTorch 2.0+:
    torch.use_deterministic_algorithms(True)

set_seed(42)

Be aware: torch.backends.cudnn.deterministic = True and torch.use_deterministic_algorithms(True) can slow down training significantly. Use them for debugging and reproducibility checks, but consider disabling them for final training runs. Setting torch.backends.cudnn.benchmark = True (the opposite) can actually speed up training when input sizes are constant, by letting cuDNN auto-tune convolution algorithms.

Putting It All Together

Here's a complete, runnable training script that combines everything from this section. This is the template you'll start from for nearly every project:

import torch
import torch.nn as nn
from torch.utils.data import DataLoader, TensorDataset

# ── Setup ────────────────────────────────────────────────────────────
device = torch.device('cuda' if torch.cuda.is_available() else 'cpu')

# Synthetic data for demonstration
X_train = torch.randn(1000, 3, 32, 32)
y_train = torch.randint(0, 10, (1000,))
X_val = torch.randn(200, 3, 32, 32)
y_val = torch.randint(0, 10, (200,))

train_dataset = TensorDataset(X_train, y_train)
val_dataset = TensorDataset(X_val, y_val)

train_loader = DataLoader(train_dataset, batch_size=64, shuffle=True,
                          num_workers=2, pin_memory=True)
val_loader = DataLoader(val_dataset, batch_size=128, shuffle=False,
                        num_workers=2, pin_memory=True)

# ── Model ────────────────────────────────────────────────────────────
model = ConvClassifier(num_classes=10).to(device)

criterion = nn.CrossEntropyLoss()
optimizer = torch.optim.AdamW(model.parameters(), lr=1e-3, weight_decay=1e-2)
scheduler = torch.optim.lr_scheduler.CosineAnnealingLR(
    optimizer, T_max=20, eta_min=1e-6
)

# ── Training ─────────────────────────────────────────────────────────
num_epochs = 20
best_val_loss = float('inf')

for epoch in range(num_epochs):
    # --- Training phase ---
    model.train()
    train_loss = 0.0

    for inputs, targets in train_loader:
        inputs, targets = inputs.to(device), targets.to(device)

        optimizer.zero_grad(set_to_none=True)
        outputs = model(inputs)
        loss = criterion(outputs, targets)
        loss.backward()
        torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
        optimizer.step()

        train_loss += loss.item() * inputs.size(0)

    scheduler.step()
    avg_train_loss = train_loss / len(train_dataset)

    # --- Validation phase ---
    model.eval()
    val_loss = 0.0
    correct = 0

    with torch.inference_mode():
        for inputs, targets in val_loader:
            inputs, targets = inputs.to(device), targets.to(device)
            outputs = model(inputs)
            loss = criterion(outputs, targets)
            val_loss += loss.item() * inputs.size(0)
            correct += (outputs.argmax(1) == targets).sum().item()

    avg_val_loss = val_loss / len(val_dataset)
    val_acc = correct / len(val_dataset)

    print(f"Epoch {epoch+1:>2}/{num_epochs}  "
          f"Train Loss: {avg_train_loss:.4f}  "
          f"Val Loss: {avg_val_loss:.4f}  "
          f"Val Acc: {val_acc:.2%}")

    # --- Checkpointing ---
    if avg_val_loss < best_val_loss:
        best_val_loss = avg_val_loss
        torch.save({
            'epoch': epoch,
            'model_state_dict': model.state_dict(),
            'optimizer_state_dict': optimizer.state_dict(),
            'scheduler_state_dict': scheduler.state_dict(),
            'val_loss': avg_val_loss,
        }, 'best_model.pt')

print(f"\nTraining complete. Best val loss: {best_val_loss:.4f}")
