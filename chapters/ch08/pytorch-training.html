The Confession

PyTorch won because honest. No .fit() hiding machinery. You write loop. Need understand what zero_grad() clears, why graph vanishes after .backward(), and what pin_memory pins. Here dive.

PyTorch shipped from Facebook AI Research in 2016, built on Lua-based Torch. Big idea: build computation graph as Python code runs, not from pre-declared static graph. That idea — define-by-run — matched what researchers and engineers wanted. PyTorch now dominates arXiv, powers Hugging Face, and keeps growing in production through TorchServe, ONNX export, and torch.compile.

Tensors: The Thing That Remembers

Everything in PyTorch starts with tensor. Multi-dimensional array, conceptually same as NumPy ndarray, but with two powers: can live on GPU, can remember how created.

Build running example. Tiny sentiment classifier: three movie reviews, each small feature vector. Use it through section. Start tiny. Add complexity until production-grade training pipeline.

import torch

# Our three movie reviews as feature vectors (3 samples, 4 features each)
reviews = torch.tensor([
    [0.8, 0.1, 0.9, 0.2],   # positive review
    [0.1, 0.9, 0.2, 0.8],   # negative review
    [0.5, 0.5, 0.5, 0.5],   # ambiguous review
], dtype=torch.float32)

labels = torch.tensor([1, 0, 1], dtype=torch.long)  # 1=positive, 0=negative

dtype=torch.float32 matters more than it looks. Tensors track data type, device, and shape as first-class properties. When two tensors meet in one operation, all three must match or PyTorch throws RuntimeError. Strict at first. Saves you from silent numerical bugs later.

NumPy interop is nearly free. torch.from_numpy() shares underlying memory — no copy — and .numpy() goes other way. Catch: shared memory means changing one changes other. Easy debug surprise.

import numpy as np
arr = np.array([1.0, 2.0, 3.0])
t = torch.from_numpy(arr)     # shared memory — no copy
arr[0] = 99.0
print(t[0])                    # tensor(99.) — surprise

GPU tensors work differently. .to('cuda') copies data into GPU memory. CPU tensor and GPU tensor become independent.

Autograd: The Tape Recorder

If tensors are nouns of PyTorch, autograd is verb. System makes training possible. Understanding it marks difference between writing training loop and knowing what training loop does.

Analogy: cooking show recording. Chop vegetable, add spice, camera records it. Recording is computation graph. Later someone asks "how did salt affect final taste?" Rewind tape, trace salt to finished dish. That rewind is .backward().

Setting requires_grad=True on tensor turns camera on. From that point, every operation involving that tensor gets recorded. Each operation creates Function node in graph, and each tensor carries grad_fn pointing to Function that created it.

x = torch.tensor(3.0, requires_grad=True)
y = x ** 2 + 2 * x + 1      # y = x² + 2x + 1

print(y.grad_fn)              # <AddBackward0> — the last operation that created y

y.backward()                  # rewind the tape: dy/dx = 2x + 2 = 8.0
print(x.grad)                 # tensor(8.)

Under hood: x ** 2 creates PowBackward and saves x for derivative of x². + 2 * x creates MulBackward and AddBackward. + 1 creates another AddBackward. Nodes form directed acyclic graph from y back to x.

Calling y.backward() walks graph in reverse — that is "back" in backpropagation — computing ∂y/∂x with chain rule at each step. Result lands in x.grad. Then graph is destroyed. Freed from memory. Next forward pass builds new one.

This is define-by-run: graph IS your Python code. Want if statement? Fine. Want for loop whose length depends on input? Fine. Graph captures whatever actually executed.

def weird_function(x):
    if x.item() > 0:
        return x ** 2
    else:
        return x ** 3

x = torch.tensor(2.0, requires_grad=True)
y = weird_function(x)
y.backward()
print(x.grad)                 # tensor(4.) — gradient of x², because x > 0

Dynamic graph can handle arbitrary Python control flow and still compute correct gradients. Trick: autograd does not know about if statement. It sees only operations that actually ran. If x > 0, it records x ** 2 and never sees x ** 3. Graph is record of what happened, not description of what could happen.

Internalize few autograd rules or get bitten. Gradients accumulate by default. Calling .backward() adds to existing .grad, not replaces it. Design choice we use later for gradient accumulation, but it means you must zero gradients before each training step. Tape is gone after .backward(); call it twice on same graph and you get RuntimeError unless you pass retain_graph=True. .detach() severs tensor from graph, producing new tensor with same values but no history. Use it for logging without leaking memory.

The Training Loop — Seven Lines That Run the World

Keras gives .fit(). PyTorch gives seven lines. Feature, not limitation. Those seven lines are heartbeat of every PyTorch project. Knowing why each line exists — not only what it does, but why — separates use from understanding.

Let's train our sentiment classifier. One weight matrix, one bias, one loss function, one optimizer:

import torch
import torch.nn as nn

# Our toy data
reviews = torch.tensor([[0.8,0.1,0.9,0.2],[0.1,0.9,0.2,0.8],[0.5,0.5,0.5,0.5]])
labels = torch.tensor([1, 0, 1])

# A single linear layer: 4 features → 2 classes
model = nn.Linear(4, 2)
criterion = nn.CrossEntropyLoss()
optimizer = torch.optim.SGD(model.parameters(), lr=0.1)

for epoch in range(5):
    optimizer.zero_grad()            # 1. clear old gradients
    outputs = model(reviews)         # 2. forward pass — builds the graph
    loss = criterion(outputs, labels)# 3. compute scalar loss
    loss.backward()                  # 4. walk graph backward — compute gradients
    optimizer.step()                 # 5. update parameters using gradients
    print(f"Epoch {epoch}: loss={loss.item():.4f}")

Five lines inside toy loop. Real training loop has seven core operations. Here full canonical pattern with batched data, GPU handling, and rest real project needs:

model.train()                             # set training mode

for epoch in range(num_epochs):
    for inputs, targets in dataloader:
        inputs = inputs.to(device)        # move data to GPU
        targets = targets.to(device)

        optimizer.zero_grad()             # clear accumulated gradients
        outputs = model(inputs)           # forward pass (graph is built here)
        loss = criterion(outputs, targets)# compute loss
        loss.backward()                   # backward pass (graph is consumed here)
        optimizer.step()                  # update weights

    scheduler.step()                      # adjust learning rate

Walk through each line. Every one can bite.

model.train() toggles two layer behaviors. Dropout starts randomly zeroing neurons. BatchNorm switches to current-batch statistics instead of stored running averages. That is all; it does nothing to gradient computation. Forget it after eval(), and model trains with dropout disabled and BatchNorm frozen. Loss curve can look fine. Model underperforms. You blame learning rate.

inputs.to(device) moves tensors to device where model lives. Model on GPU and data on CPU? Immediate RuntimeError. Every tensor in one operation must share device — same-room rule. Subtle asymmetry between .to() on modules and tensors comes later.

optimizer.zero_grad() catches almost everyone once. Gradients accumulate by default, so skipping it makes each step pile onto previous ones. Loss may look like it converges, but model behaves erratically because effective gradient is running sum of earlier steps. Fix is easy once known. Diagnosis is not, because nothing crashes.

outputs = model(inputs) runs forward pass. Calling model like function invokes forward() plus any registered hooks. Here computation graph gets built — every matmul, activation, and normalization becomes node that .backward() later traverses.

loss = criterion(outputs, targets) computes scalar loss. Key detail: nn.CrossEntropyLoss expects raw logits, not probabilities. It applies LogSoftmax internally with LogSumExp trick, which is more numerically stable than doing softmax() then log() yourself. Feed it softmax'd outputs, and loss effectively computes softmax twice. Gradients go wrong.

loss.backward() walks graph backward from loss scalar, applies chain rule at each node, and writes ∂loss/∂param into every parameter's .grad. After this call, graph is destroyed and memory freed. That is why you cannot call .backward() twice without retain_graph=True.

optimizer.step() reads .grad on each parameter and updates parameter. For SGD, literal rule is param -= lr * param.grad. For Adam, update is more complex: running averages of gradients and squared gradients produce adaptive learning rates. Optimizer holds references to model parameters from model.parameters().

One more line matters: scheduler.step() adjusts learning rate. Some schedulers (CosineAnnealing, StepLR) run per epoch. Others (OneCycleLR) run per batch. Wrong frequency creates subtle bug — LR schedule becomes too aggressive or barely noticeable. Check scheduler docs.

Dataset and DataLoader: The Data Pipeline

Training loop above fed all three reviews at once. Fine for toy data. Real datasets have millions of samples, live on disk, and need transforms. PyTorch splits job in two: Dataset defines what each sample is, DataLoader handles how samples become batches.

Grow sentiment classifier. Replace three hardcoded vectors with CSV of movie reviews and pre-extracted features.

from torch.utils.data import Dataset, DataLoader

class ReviewDataset(Dataset):
    def __init__(self, csv_path):
        import pandas as pd
        df = pd.read_csv(csv_path)
        self.features = torch.tensor(df.drop('label', axis=1).values, dtype=torch.float32)
        self.labels = torch.tensor(df['label'].values, dtype=torch.long)

    def __len__(self):
        return len(self.features)

    def __getitem__(self, idx):
        return self.features[idx], self.labels[idx]

Two methods. That is contract. __len__ returns sample count. __getitem__ returns one sample for one index. PyTorch does not call Dataset with whole batch of indices; it calls __getitem__ one by one, then DataLoader assembles batch.

DataLoader is where pipeline gets interesting — and where silent performance bottlenecks live.

train_loader = DataLoader(
    dataset=train_dataset,
    batch_size=64,
    shuffle=True,              # randomize order each epoch
    num_workers=4,             # parallel loading processes
    pin_memory=True,           # page-locked memory for faster GPU transfer
    prefetch_factor=2,         # batches prefetched per worker
    persistent_workers=True,   # keep workers alive between epochs
    drop_last=True,            # drop incomplete final batch
)

With num_workers=0, data loading happens in main process, sequentially. GPU finishes training step, waits for next batch, processes it, waits again. Big chunk of training time goes to loading.

With num_workers=4, PyTorch spawns four Python processes. Each one calls __getitem__, loads data, applies transforms, and puts result into shared-memory queue. Main process pulls from queue. While GPU crunches one batch, workers prepare next. With prefetch_factor=2, each worker stays two batches ahead, so up to num_workers × prefetch_factor batches wait in queue.

Think kitchen: four prep cooks, one chef. Chef (GPU) should never wait for ingredients. If chef finishes dish and nothing is prepped, restaurant slows down. You want enough prep cooks to keep chef busy, not so many that they fight for counter space (RAM).

pin_memory=True ties pipeline to GPU. Page-locked CPU memory enables asynchronous, non-blocking transfers to GPU via cuda.Stream. Without it, each CPU→GPU transfer waits for copy to finish. With it, transfer happens in background while other work continues. Use this when training on CUDA.

persistent_workers=True keeps worker processes alive between epochs. Without it, PyTorch kills and respawns workers at every epoch boundary — 5-15 seconds of pure overhead per epoch for large datasets. With it, workers stay warm.

worker_init_fn matters when Dataset applies random augmentations. Each worker is forked from same parent and inherits same random seed. Without per-worker seed, all workers can produce identical augmentations. Fix:

def worker_init(worker_id):
    np.random.seed(42 + worker_id)

loader = DataLoader(dataset, num_workers=4, worker_init_fn=worker_init)

nn.Module: The Building Block That Registers Itself

Every model in PyTorch is nn.Module. Every layer is nn.Module. Model made of layers is Module of Modules. Turtles all way down.

Contract is two methods: define layers in __init__, define computation in forward. Machinery under that contract makes nn.Module powerful — and sometimes surprising.

Build proper classifier for reviews:

import torch.nn as nn
import torch.nn.functional as F

class SentimentClassifier(nn.Module):
    def __init__(self, input_dim=4, hidden_dim=32, num_classes=2):
        super().__init__()
        self.layer1 = nn.Linear(input_dim, hidden_dim)
        self.bn = nn.BatchNorm1d(hidden_dim)
        self.dropout = nn.Dropout(0.3)
        self.layer2 = nn.Linear(hidden_dim, num_classes)

    def forward(self, x):
        x = self.layer1(x)
        x = self.bn(x)
        x = F.relu(x)
        x = self.dropout(x)
        x = self.layer2(x)
        return x    # raw logits — never apply softmax here

When you write self.layer1 = nn.Linear(input_dim, hidden_dim), something quiet happens. nn.Module overrides Python's __setattr__ and inspects assigned value. If value is nn.Parameter, it goes into internal _parameters dictionary. If value is another nn.Module, it goes into _modules. That is how model.parameters() recursively finds every learnable weight in nested model: it walks _modules and _parameters all way down.

Auto-registration makes plain Python list a trap:

# BROKEN — parameters invisible to optimizer
class BadModel(nn.Module):
    def __init__(self):
        super().__init__()
        self.layers = [nn.Linear(10, 10) for _ in range(3)]  # plain list!

# CORRECT — use nn.ModuleList
class GoodModel(nn.Module):
    def __init__(self):
        super().__init__()
        self.layers = nn.ModuleList([nn.Linear(10, 10) for _ in range(3)])

Plain list bypasses __setattr__. PyTorch never knows those layers exist. model.parameters() returns nothing. Optimizer has nothing to update. Training runs, loss never improves, nothing crashes.

Another registration mechanism matters: register_buffer. It stores tensors that belong to model state but are not parameters — things like BatchNorm running mean and variance. Buffers go into state_dict(), move with .to(device), but optimizer never updates them.

class ModelWithState(nn.Module):
    def __init__(self):
        super().__init__()
        self.linear = nn.Linear(10, 10)
        self.register_buffer('running_count', torch.zeros(1))

    def forward(self, x):
        self.running_count += 1    # tracked in state_dict, moves with .to()
        return self.linear(x)

Hooks are third mechanism worth knowing. register_forward_hook attaches function that runs after every forward pass — useful for extracting intermediate features, debugging, or monitoring activations. register_full_backward_hook does same for backward pass — useful for gradient analysis or custom gradient modification. Old register_backward_hook was deprecated because it could return wrong results for modules with multiple outputs.

Device Management: The Same-Room Rule

Every tensor in one operation must be on same device. Model on GPU, data on CPU? RuntimeError. Two tensors on different GPUs? RuntimeError. Same-room rule. Common beginner error.

device = torch.device('cuda' if torch.cuda.is_available() else 'cpu')

model = SentimentClassifier().to(device)   # in-place for modules
inputs = torch.randn(16, 4)
inputs = inputs.to(device)                 # NOT in-place for tensors — must reassign

Critical asymmetry trips almost everyone once. For nn.Module, .to(device) moves parameters and buffers in-place and returns self. For tensors, it returns new tensor on target device; original stays put. Writing tensor.to(device) without reassignment silently does nothing. Next line crashes with device mismatch.

Apple Silicon uses torch.device('mps'), not 'cuda'. Not all operations work on MPS yet, but coverage improves each PyTorch release.

Saving and Loading: Checkpoints Done Right

Always save state_dict(), never model object. Saving whole model uses Python pickle, which embeds class structure, import paths, and module layout into saved file. Rename class? Move file? Change import? Checkpoint breaks.

# Save weights only
torch.save(model.state_dict(), 'model_weights.pt')

# Load weights — create architecture first, then load state
model = SentimentClassifier()
model.load_state_dict(torch.load('model_weights.pt', map_location=device))
model.to(device)
model.eval()

To resume training after crash, you need full training state — not only model weights, but optimizer state (momentum buffers for Adam), scheduler state, current epoch, and best validation loss. Without optimizer state, Adam adaptive learning rates reset and training takes visible hit.

# Save full checkpoint
checkpoint = {
    'epoch': epoch,
    'model_state_dict': model.state_dict(),
    'optimizer_state_dict': optimizer.state_dict(),
    'scheduler_state_dict': scheduler.state_dict(),
    'best_val_loss': best_val_loss,
}
torch.save(checkpoint, f'checkpoint_epoch_{epoch}.pt')

# Resume
ckpt = torch.load('checkpoint_epoch_42.pt', map_location=device)
model.load_state_dict(ckpt['model_state_dict'])
optimizer.load_state_dict(ckpt['optimizer_state_dict'])
scheduler.load_state_dict(ckpt['scheduler_state_dict'])
start_epoch = ckpt['epoch'] + 1

map_location=device is essential. Save on GPU 0 and load on CPU or different GPU, and map_location remaps tensors. Without it, PyTorch tries original device, which may not exist on current machine.

For partial loading — new layers added, old weights still wanted — use strict=False:

model.load_state_dict(ckpt['model_state_dict'], strict=False)
# Missing keys: new layers initialized randomly
# Unexpected keys: old layers that no longer exist are ignored

Inference: Two Switches, Not One

Correct inference needs two independent switches. Forget either one and bug appears. Different bug each time.

model.eval()                              # switch 1: change layer behavior

with torch.inference_mode():              # switch 2: disable gradient tracking
    outputs = model(inputs.to(device))
    predictions = outputs.argmax(dim=1)

model.eval() changes specific layer behavior. Dropout stops zeroing neurons, so all neurons stay active. BatchNorm switches from per-batch statistics to stored running mean and variance. Without this, dropout makes inference nondeterministic and BatchNorm uses noisy single-batch statistics instead of stable running averages from training.

torch.inference_mode() disables tape recorder entirely. No computation graph gets built, no gradient metadata gets stored. This saves significant memory and speeds computation. It is modern replacement for torch.no_grad() — stricter and faster.

These two switches are independent. eval() changes layer behavior but not gradients. inference_mode() disables gradients but not layer behavior. You need both. Using only one is real production bug.

After evaluation, switch back: model.train(). Otherwise next training epoch runs with dropout disabled.

The Patterns That Matter: Gradient Accumulation

You want batch size 256, but GPU fits only 64 samples. Standard advice: use gradient accumulation. Mechanically, what does that mean?

Autograd accumulates gradients by default. Earlier footgun. Here, feature. Call .backward() four times before optimizer.step(), and each parameter's .grad holds sum of four backward passes. Divide loss by accumulation count to keep gradient magnitude equal to one large batch.

accum_steps = 4                            # effective batch = 64 × 4 = 256
optimizer.zero_grad()

for step, (inputs, targets) in enumerate(train_loader):
    inputs, targets = inputs.to(device), targets.to(device)
    outputs = model(inputs)
    loss = criterion(outputs, targets) / accum_steps   # normalize
    loss.backward()                                     # gradients accumulate

    if (step + 1) % accum_steps == 0:
        torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
        optimizer.step()
        optimizer.zero_grad()

Math works: if each mini-batch contributes gradients g₁, g₂, g₃, g₄, accumulated gradient is (g₁ + g₂ + g₃ + g₄) / 4. That is mean gradient over 256 samples — same as one batch of 256. Kitchen analogy: chef tastes four dishes, averages feedback, then adjusts recipe.

One subtlety: BatchNorm still sees batches of 64, not 256. Running statistics and per-batch normalization use actual batch size. For very small batches, consider GroupNorm or LayerNorm instead.

Mixed Precision: Two Number Systems

Modern GPUs have specialized hardware — Tensor Cores on NVIDIA, Matrix Engines on AMD — that run float16 or bfloat16 at roughly 2× float32 throughput. Mixed precision uses that: expensive operations (matrix multiplications, convolutions) run in lower precision for speed, sensitive operations (loss computation, softmax, weight updates) stay in float32 for accuracy.

PyTorch's Automatic Mixed Precision (AMP) handles precision routing automatically:

from torch.amp import GradScaler, autocast

scaler = GradScaler('cuda')

for inputs, targets in train_loader:
    inputs, targets = inputs.to(device), targets.to(device)
    optimizer.zero_grad()

    with autocast('cuda'):                # forward pass in mixed precision
        outputs = model(inputs)
        loss = criterion(outputs, targets)

    scaler.scale(loss).backward()         # backward in scaled float16
    scaler.unscale_(optimizer)            # unscale gradients for clipping
    torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
    scaler.step(optimizer)                # step only if no inf/nan
    scaler.update()                       # adjust scale factor

autocast hooks into PyTorch dispatcher. For each operation, it checks policy: safe operations (matmul, conv, linear) get cast to float16; risky operations (softmax, log, loss functions) stay in float32. You do not choose casts. PyTorch does, based on core-team numerical analysis.

GradScaler solves float16 range problem: dynamic range is tiny (roughly 6×10⁻⁵ to 6.5×10⁴). Small gradients — especially in early layers of deep networks — can underflow to zero. GradScaler multiplies loss by large factor before backward, pushing gradients into representable range. Before optimizer step, it scales them back down. If it sees inf or nan, it skips optimizer step and lowers scale factor. Over time, scale factor finds workable range automatically.

Important development: bfloat16 has same exponent range as float32 (8 exponent bits vs. float16's 5), so gradient underflow is much less likely. On Ampere and newer NVIDIA GPUs (A100, H100), bfloat16 is often preferred because you may not need GradScaler at all:

with autocast('cuda', dtype=torch.bfloat16):
    outputs = model(inputs)
    loss = criterion(outputs, targets)
loss.backward()            # no scaler needed with bfloat16
optimizer.step()

Gradient Checkpointing: Trading Compute for Memory

During forward pass, PyTorch stores intermediate activations at every layer because backward needs them for gradients. For transformer with 24 layers and batch size 32, activations can consume 10+ GB. Gradient checkpointing says: do not store them. Recompute them during backward.

from torch.utils.checkpoint import checkpoint

class CheckpointedModel(nn.Module):
    def __init__(self):
        super().__init__()
        self.block1 = nn.Sequential(nn.Linear(256, 256), nn.ReLU())
        self.block2 = nn.Sequential(nn.Linear(256, 256), nn.ReLU())
        self.head = nn.Linear(256, 10)

    def forward(self, x):
        x = checkpoint(self.block1, x, use_reentrant=False)
        x = checkpoint(self.block2, x, use_reentrant=False)
        return self.head(x)

With checkpointing, forward saves only inputs to each block. During backward, when autograd needs intermediate activations from block1, it re-runs block1 forward pass to regenerate them. This roughly halves memory use for models with many sequential blocks, at cost of one extra forward pass. For N layers, optimal checkpointing drops memory from O(N) to O(√N).

Checkpointing payoff depends on ratio of memory savings to compute overhead. For transformers with many identical blocks, usually worth it. For models with lightweight layers, recomputation overhead may not justify memory savings.

One critical requirement: checkpointed function must be deterministic. If it contains dropout with different random seed during recomputation, gradients go wrong. PyTorch handles this by saving and restoring RNG state, but custom stochastic operations need explicit care.

torch.compile: The JIT That Actually Works

PyTorch 2.0 introduced torch.compile. After years of abandoned JIT attempts, skepticism made sense. But torch.compile is different. It works on real code, and speedups are meaningful.

model = SentimentClassifier().to(device)
model = torch.compile(model)               # one line — that's it

Under hood, three systems work together. TorchDynamo hooks into Python interpreter at bytecode level, watches model forward pass, and captures PyTorch operations into FX graph — clean intermediate representation. TorchInductor, compiler backend, takes FX graph and generates optimized machine code: fusing operations, cutting memory round-trips, and generating custom Triton kernels for GPU (or C++ for CPU). Compiled code gets cached, so future calls skip compilation step.

When Dynamo hits Python code it cannot trace — dynamic control flow, calls to non-PyTorch libraries — it creates graph break. Code on one side runs compiled, break itself runs in eager mode, and code after break gets compiled separately. This fallback is why torch.compile works on real models where TorchScript choked.

Three compilation modes matter:

torch.compile(model, mode='default')          # safe, moderate speedup
torch.compile(model, mode='reduce-overhead')  # uses CUDA graphs — lower launch overhead
torch.compile(model, mode='max-autotune')     # benchmarks multiple kernel variants — slowest compile, fastest run

Typical speedups range from 15-40%, depending on model architecture and how memory-bound operations are. First forward pass is slow because compilation happens. Later calls run at compiled speed. In training, first batch of each epoch may be slow, but rest fly.

Distributed Training: DDP and FSDP

When one GPU is not enough — dataset is massive or model takes days to converge — you need distributed training. PyTorch offers two main strategies: DistributedDataParallel (DDP) for data parallelism, and FullyShardedDataParallel (FSDP) when model itself does not fit on one GPU.

DDP is workhorse. Each GPU gets full copy of model and different slice of data. After each backward pass, DDP synchronizes gradients across GPUs with all-reduce — usually ring all-reduce via NCCL, where each GPU talks only to two neighbors and passes gradient chunks in N-1 steps for N GPUs. Clever part: DDP overlaps communication with backward pass. It groups gradients into buckets, and as soon as bucket gradients are ready, all-reduce starts while rest of backward continues. GPU computes and communicates at same time.

import os
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP

dist.init_process_group(backend='nccl')
local_rank = int(os.environ['LOCAL_RANK'])
model = SentimentClassifier().to(local_rank)
model = DDP(model, device_ids=[local_rank])

# Training loop is almost identical — DDP handles gradient sync
# Launch with: torchrun --nproc_per_node=4 train.py

FSDP goes further. Instead of replicating full model on each GPU, it shards parameters, gradients, and optimizer states across GPUs. Each GPU holds only fraction of model. During forward, it gathers needed parameters from other GPUs, computes, then releases them. This is PyTorch equivalent of DeepSpeed's ZeRO-3. It enables training models far larger than one GPU's memory.

Tradeoff: DDP is simpler, more mature, and faster for models that fit in GPU memory. FSDP has more communication overhead but unlocks models that otherwise cannot train. If model fits on one GPU, use DDP. If not, use FSDP.

The Gotchas That Will Eat Your Afternoon

These come from repeated debugging. Each one is real bug that does not crash. It silently degrades results.

Forgetting optimizer.zero_grad() — Gradients pile up across batches. Loss appears to converge. Model performance plateaus far below where it should. Nothing crashes. You blame learning rate, architecture, or data.

Forgetting model.eval() during inference — Dropout still randomly kills neurons. BatchNorm uses noisy per-batch statistics instead of smooth running averages. Eval metrics look worse than they should and vary between runs. You think model is bad. Maybe not.

The .to(device) asymmetry — Writing tensor.to('cuda') without tensor = tensor.to('cuda') does nothing. Tensor stays on CPU. Next line throws device mismatch. Modules are in-place; tensors are not.

Storing loss instead of loss.item() for logging — Append loss tensor to list and you keep whole computation graph alive. Memory usage grows linearly with training steps. Eventually process gets OOM-killed and you blame model size. Fix: loss.item() extracts Python float and severs graph.

In-place operations breaking autograd — Operations like relu_() or tensor[0] = value modify tensors in-place. If autograd saved that tensor for backward and you modify it later, PyTorch throws "one of the variables needed for gradient computation has been modified by an inplace operation." Error message is confusing because in-place op may be far from where autograd saved tensor. Avoid in-place ops during training unless you know they are safe.

.detach() vs .data — Both sever tensor from graph, but .data is dangerous: it returns view that shares storage with original and bypasses autograd safety checks. Modify result and you can silently corrupt gradients. Use .detach(). Safer choice.

CrossEntropyLoss with softmax'd inputs — nn.CrossEntropyLoss expects raw logits. It applies log-softmax internally. If model already applies softmax in forward(), you compute softmax twice — once explicitly and once inside loss. Gradients go wrong and model converges to worse solution. Return raw logits from forward(). Always.

Putting It All Together

Here is complete, production-grade training script — tiny sentiment classifier grown up. It combines patterns from this section: training loop, DataLoader, device management, mixed precision, gradient clipping, checkpointing, and evaluation.

import torch
import torch.nn as nn
from torch.utils.data import DataLoader, TensorDataset
from torch.amp import GradScaler, autocast

# ── Setup ──────────────────────────────────────────────────
device = torch.device('cuda' if torch.cuda.is_available() else 'cpu')

X_train = torch.randn(1000, 4)
y_train = torch.randint(0, 2, (1000,))
X_val = torch.randn(200, 4)
y_val = torch.randint(0, 2, (200,))

train_loader = DataLoader(TensorDataset(X_train, y_train),
    batch_size=64, shuffle=True, num_workers=2, pin_memory=True)
val_loader = DataLoader(TensorDataset(X_val, y_val),
    batch_size=128, shuffle=False, num_workers=2, pin_memory=True)

# ── Model ──────────────────────────────────────────────────
model = SentimentClassifier(input_dim=4, hidden_dim=64, num_classes=2)
model = model.to(device)
# model = torch.compile(model)  # uncomment for PyTorch 2.0+ speedup

criterion = nn.CrossEntropyLoss()
optimizer = torch.optim.AdamW(model.parameters(), lr=1e-3, weight_decay=1e-2)
scheduler = torch.optim.lr_scheduler.CosineAnnealingLR(optimizer, T_max=20)
scaler = GradScaler('cuda', enabled=(device.type == 'cuda'))

# ── Training ───────────────────────────────────────────────
best_val_loss = float('inf')

for epoch in range(20):
    model.train()
    train_loss = 0.0

    for inputs, targets in train_loader:
        inputs, targets = inputs.to(device), targets.to(device)
        optimizer.zero_grad(set_to_none=True)

        with autocast('cuda', enabled=(device.type == 'cuda')):
            outputs = model(inputs)
            loss = criterion(outputs, targets)

        scaler.scale(loss).backward()
        scaler.unscale_(optimizer)
        torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
        scaler.step(optimizer)
        scaler.update()

        train_loss += loss.item() * inputs.size(0)

    scheduler.step()

    # ── Validation ─────────────────────────────────────────
    model.eval()
    val_loss, correct = 0.0, 0
    with torch.inference_mode():
        for inputs, targets in val_loader:
            inputs, targets = inputs.to(device), targets.to(device)
            outputs = model(inputs)
            val_loss += criterion(outputs, targets).item() * inputs.size(0)
            correct += (outputs.argmax(1) == targets).sum().item()

    avg_train = train_loss / len(train_loader.dataset)
    avg_val = val_loss / len(val_loader.dataset)
    val_acc = correct / len(val_loader.dataset)

    print(f"Epoch {epoch+1:>2}/20  Train: {avg_train:.4f}  "
          f"Val: {avg_val:.4f}  Acc: {val_acc:.1%}")

    if avg_val < best_val_loss:
        best_val_loss = avg_val
        torch.save({
            'epoch': epoch,
            'model_state_dict': model.state_dict(),
            'optimizer_state_dict': optimizer.state_dict(),
            'scheduler_state_dict': scheduler.state_dict(),
            'val_loss': avg_val,
        }, 'best_model.pt')

We started with three hardcoded tensors and five-line loop. We learned how autograd records computation graph like tape recorder, then erases tape during backward. We built canonical training loop and saw why each of its seven lines exists. We saw how DataLoader orchestrates parallel workers, pinned memory, and prefetching to keep GPU fed. We opened nn.Module and found hidden __setattr__ that auto-registers parameters. Then patterns that make real training fast: mixed precision, gradient accumulation, checkpointing, compilation, and distribution.

Next time you stare at PyTorch training script — hundred lines or ten thousand — goal is simple: read each line and know what it does, what it costs, and what breaks if you remove it. That is difference between using framework and understanding it.
