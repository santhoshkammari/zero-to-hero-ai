The Landscape You're Navigating

Before debugging, understand terrain. Training neural network means searching good parameters — weights and biases — that minimize loss function. Loss function defines surface over parameter space. For network with n parameters, surface lives in n+1 dimensions: n for parameters, one for loss. Our sentiment classifier might have 5 million parameters. So we navigate 5,000,001-dimensional space. Spatial intuition dies fast.

Big trap for me: imagining loss landscapes as bumpy 2D mountains. That picture breaks in high-dimensional space. In two dimensions, saddle point is rare mountain pass. In millions of dimensions, saddle points are everywhere. They vastly outnumber local minima.

Saddle Points Dominate (And That's Good News)

Dauphin et al. (2014) formalized suspicion many practitioners had: in high-dimensional non-convex optimization, critical points are mostly saddle points, not local minima. Critical point means gradient is zero. Local minimum curves upward in every direction. Saddle point curves up in some directions, down in others. Probabilistic view helps. At random critical point in n-dimensional space, each direction has roughly 50% chance of curving up or down. True local minimum needs all n directions curving up. Chance is roughly 2-n. For 5-million-parameter network, chance is astronomically small. So critical points almost always have escape routes where loss drops.

Good news. SGD rarely gets "trapped" in local minima like early neural-net fear said. Real problem at saddle points: gradients get small, training slows. Momentum optimizers like Adam help because velocity carries through low-gradient saddle region.

Sharp vs Flat Minima: Not All Solutions Are Equal

Low-loss regions do not generalize equally. Keskar et al. (2017) showed sharp minima — regions where loss changes fast when weights move — usually generalize poorly. Flat minima — regions where weights can wiggle and loss barely moves — usually generalize well. Intuition: sharp minimum is fragile. Training data lines up in one specific way, then small distribution shift throws solution out of valley. Flat minimum is robust. Small perturbations do little, so solution captures something more fundamental in data.

Batch size connects here. Small-batch SGD adds noise to gradient estimates. Noise acts like implicit regularizer: pushes optimizer away from sharp minima, because noisy updates overshoot and escape, and toward flat minima, because noise cannot kick model out. Large-batch training gives more precise gradients, so optimizer settles into narrow valley it finds. Reason naive large-batch scaling often hurts generalization: gradient-noise regularization disappears.

Skip Connections Reshape the Landscape

Li et al. (2018) visualized loss landscapes and got striking result. Plain deep network like VGG has chaotic surface — sharp peaks, narrow valleys, big flat regions with useless gradients. Add skip connections, as in ResNet, and landscape gets much smoother. Almost convex-looking. Broad smooth valleys lead to good minima.

Reason ResNets train easier than similar-depth VGGs. Skip connections not merely "better" architecture in abstract sense. They reshape optimization problem and make landscape easier. If deep network will not train, adding skip connections is not hack. It is fundamental improvement to loss-surface geometry.

Mode Connectivity

Wild result. Garipov et al. (2018) showed independently trained neural networks — same architecture, same data, different random seeds — reach solutions connected by simple low-loss paths in parameter space. Train model A and model B from different initializations, then find smooth curve connecting them while loss stays low. Good solutions are not isolated pockets. More like connected manifold. Debugging takeaway: many good solutions exist, and optimizer can find them from many starts.

SAM: Explicitly Seeking Flat Minima

If flat minima generalize better, optimize flatness directly. Sharpness-Aware Minimization (SAM), from Foret et al. (2021), does that. Instead of minimizing loss at current point, SAM minimizes worst-case loss in small neighborhood around current point. Two steps: compute gradient, step in direction that maximizes loss to find worst neighbor, then compute gradient at worst neighbor and step to minimize it. Effect: SAM avoids parameters with low loss at exact point but high loss nearby — definition of sharp minimum.

# SAM optimizer — simplified core logic
# Standard optimizers minimize L(w). SAM minimizes max_{||e||<=rho} L(w + e)

class SAM(torch.optim.Optimizer):
    def __init__(self, params, base_optimizer, rho=0.05):
        defaults = dict(rho=rho)
        super().__init__(params, defaults)
        self.base_optimizer = base_optimizer(self.param_groups)

    @torch.no_grad()
    def first_step(self):
        grad_norm = self._grad_norm()
        for group in self.param_groups:
            scale = group["rho"] / (grad_norm + 1e-12)
            for p in group["params"]:
                if p.grad is None:
                    continue
                # Save current weights, then perturb toward steepest ascent
                self.state[p]["old_p"] = p.data.clone()
                e_w = p.grad * scale
                p.add_(e_w)  # climb to worst neighbor

    @torch.no_grad()
    def second_step(self):
        for group in self.param_groups:
            for p in group["params"]:
                if p.grad is None:
                    continue
                p.data = self.state[p]["old_p"]  # restore original weights
        self.base_optimizer.step()  # step using gradient at perturbed point

    def _grad_norm(self):
        norm = torch.norm(
            torch.stack([
                p.grad.norm(p=2)
                for group in self.param_groups
                for p in group["params"] if p.grad is not None
            ]),
            p=2
        )
        return norm

Cost: roughly 2x standard training, because each step needs two forward-backward passes. Benefit: better generalization, especially near overfitting edge. For our sentiment classifier, SAM might add 0.5-1% test accuracy. Not revolution. Still useful in competitive settings.

Learning Rate Warmup: Why Starting High Is Dangerous

At initialization, weights are random. Loss landscape at random point is chaotic — gradients large, directions mostly arbitrary. Start with high learning rate, and those noisy gradients fling model into worse regions. Loss spikes. Gradients grow more. Death spiral. Often ends in NaN. Learning rate warmup starts tiny and increases over first few hundred or thousand steps. Network gets time to reach more structured region where gradients carry real signal before steps get large. Especially critical for transformers and architectures without skip connections.

💡 Key Insight

Loss landscape not fixed fate. Architecture choices like skip connections, optimizer choices like SAM or momentum, and hyperparameters like small batch or warmup all reshape effective landscape. Debugging stubborn network often means making landscape easier, not searching bad landscape harder.

The Single-Batch Overfit Test: Your First Debugging Move

Andrej Karpathy called this "most important debugging test in deep learning." Fair. It is like segfault check in systems programming. Before tuning hyperparameters, regularization, or data augmentation, verify model can memorize one batch. If it cannot, problem is fundamental bug — not hyperparameters, not capacity.

Test is simple. Grab one batch from training set. Feed same batch to model for 500-1000 steps with reasonable learning rate; 1e-3 for Adam usually works. Loss should drop steadily toward zero or near-zero, depending on task. For 2-class sentiment classifier, initial loss should be about -log(1/2) = 0.693. After 500 steps on one batch of 32 samples, loss should be below 0.01. If not, stop. Do not touch learning rate. Do not add layers. Fix bug.

import torch
import torch.nn as nn

# Our baby sentiment classifier
class SentimentModel(nn.Module):
    def __init__(self, vocab_size, embed_dim=128, hidden_dim=256, num_classes=2):
        super().__init__()
        self.embedding = nn.Embedding(vocab_size, embed_dim)
        self.lstm = nn.LSTM(embed_dim, hidden_dim, batch_first=True)
        self.classifier = nn.Linear(hidden_dim, num_classes)

    def forward(self, x):
        embeds = self.embedding(x)
        _, (hidden, _) = self.lstm(embeds)
        return self.classifier(hidden.squeeze(0))

# The overfit test
model = SentimentModel(vocab_size=10000)
optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)
criterion = nn.CrossEntropyLoss()

# Grab ONE batch and freeze it
train_loader_iter = iter(train_loader)
fixed_batch = next(train_loader_iter)
inputs, labels = fixed_batch

model.train()
for step in range(500):
    optimizer.zero_grad()
    logits = model(inputs)
    loss = criterion(logits, labels)
    loss.backward()
    optimizer.step()

    if step % 50 == 0:
        acc = (logits.argmax(dim=1) == labels).float().mean()
        print(f"Step {step}: loss={loss.item():.4f}, acc={acc.item():.3f}")

# After 500 steps: loss should be < 0.01, accuracy should be 1.0
# If not: YOU HAVE A BUG

What Failure Tells You

Failure mode tells much:

Loss stuck at initial value (e.g., 0.693 for binary classification). Model learns nothing. Common causes: gradients not flowing, maybe detached tensors, wrong dtype, or frozen parameters; optimizer not connected to model parameters; or loss function mismatched with model output format.

Loss decreases but plateaus well above zero. Model gets some gradient signal but cannot fit batch. Maybe capacity is too low for 32 examples, though that is uncommon unless architecture is tiny. More often reason is data preprocessing bug — labels shuffled against inputs, or tokenization destroys information.

Loss immediately goes to NaN or infinity. Numerical instability from first step. Check division by zero in custom layers, log(0) in losses, or absurdly high learning rate. NaN debugging comes soon.

Loss oscillates wildly without converging. Learning rate is too high for this architecture and data. Try 10x lower. If oscillation stays, likely data-label mismatch — wrong labels or shuffled labels.

🔥 Hard-Won Lesson

I once wasted two days tuning hyperparameters for non-converging model. Grid search on learning rates, architectures, batch sizes. Nothing worked. Then single-batch overfit test showed loss stuck at 2.303 — exactly -log(1/10) for 10-class classification. Model guessed randomly. Real bug: DataLoader had shuffle=True but labels tensor was not shuffled with inputs. Two days lost. Overfit test would have caught it in 30 seconds.

A Systematic Debugging Flow

Once overfit test passes — or while figuring out why it fails — use system, not guessing. Random hyperparameter changes are gambling. Flow below follows rough order of most likely bugs.

Step 1: Inspect the Data Pipeline

Most "model bugs" are data bugs. Literal truth. Print what model sees. Not what you think it sees. Not what docs say it should see. Print tensor values entering forward pass.

# ALWAYS do this before anything else
batch = next(iter(train_loader))
inputs, labels = batch

print(f"Input shape: {inputs.shape}")        # Expected: (batch_size, seq_len)
print(f"Input dtype: {inputs.dtype}")         # Should be torch.long for embeddings
print(f"Label shape: {labels.shape}")         # Expected: (batch_size,)
print(f"Label dtype: {labels.dtype}")         # Should be torch.long for CrossEntropyLoss
print(f"Label values: {labels[:10]}")         # Are these in [0, num_classes-1]?
print(f"Input range: [{inputs.min()}, {inputs.max()}]")  # Within vocab size?

# For text: decode back to words and visually inspect
# (assumes you have a tokenizer from your preprocessing pipeline)
for i in range(3):
    tokens = inputs[i].tolist()
    text = tokenizer.decode(tokens)
    label = labels[i].item()
    print(f"Label={label}: {text[:200]}")
    # Does the sentiment match the label? You'd be surprised how often it doesn't.

For our sentiment classifier, this check caught: tokenizer using wrong vocabulary so every word became UNK; padding token not ignored in LSTM, polluting hidden state; labels starting at 1 instead of 0, which makes CrossEntropyLoss throw index error with 2 classes or silently go off by one with 3+ classes; and sequences truncated to 10 tokens when average review had 200.

Step 2: Verify the Expected Initial Loss

Before training, weights are random, so predictions should be roughly uniform over classes. For classification with C classes, expected initial loss under CrossEntropyLoss is:

expected_initial_loss = -log(1/C) = log(C)

Number of Classes (C) Expected Initial Loss
2 (binary)0.693
10 (CIFAR-10)2.303
100 (CIFAR-100)4.605
1000 (ImageNet)6.908
30,000 (language model vocab)10.31

If initial loss is far from this, something is wrong. Much lower? Model has class bias before training — maybe information leak, maybe bad last-layer bias initialization. Much higher? Loss function computes something unexpected. I once saw 47.0 initial loss on 10-class problem. Cause: model output went through softmax before CrossEntropyLoss, which already applies log-softmax internally. Double softmax compressed probabilities and made log-probability far more negative than expected.

# Sanity check: initial loss
model.eval()
with torch.no_grad():
    logits = model(inputs)
    loss = criterion(logits, labels)
    print(f"Initial loss: {loss.item():.4f}")
    print(f"Expected for {num_classes} classes: {torch.log(torch.tensor(num_classes, dtype=torch.float)).item():.4f}")
    print(f"Logit stats: mean={logits.mean():.4f}, std={logits.std():.4f}")
    # Logits should have mean near 0 and moderate std if initialization is reasonable

Step 3: The Learning Rate Finder

Leslie Smith (2017) proposed strong technique: stop guessing learning rate. Sweep wide range and let data answer. Start tiny, like 1e-7. Increase exponentially each batch until large value, like 1.0 or 10.0. Record loss each step. Plot loss vs learning rate on log scale. Good learning rate is roughly 10x below point where loss is lowest before divergence starts.

import math

def lr_finder(model, train_loader, criterion, start_lr=1e-7, end_lr=1.0, num_steps=200):
    """Sweep learning rate from start_lr to end_lr, record loss at each step."""
    optimizer = torch.optim.SGD(model.parameters(), lr=start_lr)
    lr_mult = (end_lr / start_lr) ** (1 / num_steps)

    lrs, losses = [], []
    best_loss = float('inf')
    avg_loss = 0.0
    beta = 0.98  # smoothing factor

    model.train()
    data_iter = iter(train_loader)

    for step in range(num_steps):
        try:
            inputs, labels = next(data_iter)
        except StopIteration:
            data_iter = iter(train_loader)
            inputs, labels = next(data_iter)

        optimizer.zero_grad()
        logits = model(inputs)
        loss = criterion(logits, labels)
        loss.backward()
        optimizer.step()

        # Exponential smoothing
        avg_loss = beta * avg_loss + (1 - beta) * loss.item()
        smoothed_loss = avg_loss / (1 - beta ** (step + 1))

        # Stop if loss is diverging
        if step > 10 and smoothed_loss > 4 * best_loss:
            break

        if smoothed_loss < best_loss:
            best_loss = smoothed_loss

        current_lr = start_lr * (lr_mult ** step)
        lrs.append(current_lr)
        losses.append(smoothed_loss)

        # Update LR for next step
        for param_group in optimizer.param_groups:
            param_group['lr'] *= lr_mult

    return lrs, losses
    # Plot lrs (log scale) vs losses
    # Pick LR about 10x below the minimum loss point

For our sentiment classifier with Adam, learning rate finder usually shows loss falling around 1e-5, bottoming near 1e-3 to 3e-3, then blowing up above 1e-2. So pick around 1e-3 to 3e-4. Not final word — optimal LR still depends on batch size, model size, and warmup — but this gets you near right range fast instead of wasting runs on LR=0.1 or LR=1e-6.

Step 4: Gradient Health Checks

If model trains but something feels wrong — stagnant loss, painfully slow convergence, or missing feature learning — inspect gradients directly.

def check_gradient_health(model):
    """Print gradient statistics per layer — the training vital signs."""
    for name, param in model.named_parameters():
        if param.grad is not None:
            grad = param.grad
            print(f"{name:40s} | "
                  f"grad mean={grad.mean().item():+.2e}, "
                  f"grad std={grad.std().item():.2e}, "
                  f"grad norm={grad.norm().item():.2e}, "
                  f"param norm={param.norm().item():.2e}")
        else:
            print(f"{name:40s} | NO GRADIENT")

# Call after loss.backward() but before optimizer.step()
loss.backward()
check_gradient_health(model)
optimizer.step()

Watch for this: NO GRADIENT on any parameter means that parameter is disconnected from loss. Maybe frozen with requires_grad=False, or graph broke because you called .detach(), .item(), or .numpy() on intermediate tensor. Gradient norms near zero in early layers but healthy later means vanishing gradients — signal dies on backprop path. Wildly different gradient norms across layers, like 1e-8 early and 1e+3 late, suggest need for better initialization, normalization, or gradient clipping.

Step 5: Architecture Bugs — The Double Softmax Trap

Own subsection needed. Bug catches many people, including experienced practitioners. Me too.

PyTorch's nn.CrossEntropyLoss expects raw logits — unnormalized scores. It applies log_softmax and then nll_loss internally. If model ends with nn.Softmax() or F.softmax(), softmax happens twice. First softmax squashes logits into [0, 1]. Second softmax inside CrossEntropyLoss applies log-softmax to already squashed values. Loss is mathematically valid, gradients still flow, but optimization landscape gets distorted and training becomes badly crippled.

# BUG: Double softmax — the model applies softmax, then CrossEntropyLoss applies it again
class BrokenModel(nn.Module):
    def __init__(self, input_dim, num_classes):
        super().__init__()
        self.fc = nn.Linear(input_dim, num_classes)
        self.softmax = nn.Softmax(dim=1)  # <-- THIS IS THE BUG

    def forward(self, x):
        return self.softmax(self.fc(x))  # Returns probabilities, not logits

# CrossEntropyLoss internally does log_softmax(input), so this computes:
# loss = nll_loss(log_softmax(softmax(logits)))  <-- double softmax!

# FIX: Return raw logits from the model
class FixedModel(nn.Module):
    def __init__(self, input_dim, num_classes):
        super().__init__()
        self.fc = nn.Linear(input_dim, num_classes)

    def forward(self, x):
        return self.fc(x)  # Raw logits — let the loss function handle normalization

Sneaky bug. No crash. No NaN. Model still learns, sort of. Maybe stops around 70% accuracy on task that should reach 90%. Easy mistake: blame model size or hyperparameters and waste days. Real problem is wrong loss setup.

Step 6: torch.autograd.detect_anomaly()

When manual checks fail and NaN or strange values remain, PyTorch has nuclear option: torch.autograd.detect_anomaly(). It instruments every autograd op, checks forward and backward for NaN and Inf, and gives traceback to exact bad op.

# ONLY use for debugging — this makes training 5-10x slower
with torch.autograd.detect_anomaly():
    logits = model(inputs)
    loss = criterion(logits, labels)
    loss.backward()
    # If any operation produces NaN/Inf, you'll get a RuntimeError
    # with the exact traceback showing which operation is the culprit

Never leave this on in production training. Overhead is huge. Find bug. Fix bug. Remove it.

💡 Key Insight

Debugging flow has hierarchy: data first, loss second, gradients third, architecture fourth. Most bugs live in first two. Jumping to architecture or hyperparameters before data integrity checks means debugging blind.

NaN Loss: Causes, Mechanisms, and Fixes

NaN loss is training version of kernel panic. Everything looks fine, then every number in model becomes NaN and run is dead. I have seen this at hour 47 of 48-hour training. Brutal. Need understand how NaN spreads and how to stop it.

The Chain Reaction

NaN rarely starts everywhere. It starts in one op, one layer, one parameter. Then it spreads through computation graph like infection. One NaN gradient makes one weight NaN. That weight makes NaN activations next forward pass. Those activations make NaN gradients for other weights. In one or two steps, whole model is NaN. Recovery needs checkpoint.

Typical chain: bad batch — outlier inputs or mislabeled samples — makes unusually large loss. Large loss makes large gradients. Large gradients make huge weight update. Updated weight becomes extreme. Next forward pass turns that extreme weight into activation overflow, then infinity. Infinity times anything stays infinity. Infinity minus infinity becomes NaN. NaN times anything stays NaN. Game over.

Float16 Overflow

If you train in mixed precision, float16 max value is about 65,504. Sounds large. Not large. Square 256 and you already hit 65,536 — overflow. Softmax computes exp(x), which overflows float16 for x > 11.09. Large hidden-dimension matrix multiplies can easily make pre-activations bigger than that.

# Float16 range demonstration
import torch
print(torch.finfo(torch.float16).max)   # 65504.0
print(torch.finfo(torch.float16).min)   # -65504.0
print(torch.finfo(torch.float16).tiny)  # 6.1035e-05 (smallest positive normal)

# This overflows in float16:
x = torch.tensor([256.0], dtype=torch.float16)
print(x ** 2)  # tensor([inf], dtype=torch.float16)

# This is why softmax needs to be computed in float32 even in mixed precision
logits_fp16 = torch.tensor([12.0, 13.0, 14.0], dtype=torch.float16)
print(torch.exp(logits_fp16))  # tensor([inf, inf, inf]) in float16!

logits_fp32 = logits_fp16.float()
print(torch.exp(logits_fp32))  # tensor([162755., 442413., 1202604.]) — fine in fp32

Common Numerical Instability Sources

log(0): If model outputs exact 0.0 and you take log, result is -inf. Arithmetic on -inf can become NaN. Fix: add epsilon guard. Use torch.log(x + 1e-8) instead of torch.log(x) in custom losses. Built-in CrossEntropyLoss already handles this through log_softmax.

sqrt(0) with gradients: Derivative of sqrt(x) is 1/(2*sqrt(x)). At x=0, that is infinity. Forward pass may survive. Backward pass explodes. Fix: torch.sqrt(x + 1e-8).

exp(large): exp(x) overflows above about 88 in float32, or 11 in float16. Common places: softmax, Gaussian mixture models, probability calculations. Standard fix is log-sum-exp trick: instead of exp(x_i) / sum(exp(x_j)), compute exp(x_i - max(x)) / sum(exp(x_j - max(x))). PyTorch's built-in softmax already does this.

Division by zero: Common in custom normalization, attention with zero denominator, or any division by computed quantity that might hit zero. Add epsilon to denominators. Always.

# Common epsilon guards for numerical stability

# BAD: log(0) = -inf, then -inf * weight = NaN
log_prob = torch.log(predicted_prob)

# GOOD: epsilon guard prevents log(0)
log_prob = torch.log(predicted_prob + 1e-8)

# BAD: sqrt gradient is inf at 0
norm = torch.sqrt(sum_of_squares)

# GOOD: epsilon inside the sqrt
norm = torch.sqrt(sum_of_squares + 1e-8)

# BAD: division by potentially zero norm
normalized = x / x.norm(dim=-1, keepdim=True)

# GOOD: clamp the denominator
normalized = x / x.norm(dim=-1, keepdim=True).clamp(min=1e-8)

# BAD: custom attention without numerical stability
attn_weights = torch.exp(scores) / torch.exp(scores).sum(dim=-1, keepdim=True)

# GOOD: use the numerically stable built-in softmax
attn_weights = torch.softmax(scores, dim=-1)

GradScaler for Mixed Precision

PyTorch's GradScaler is front-line defense against NaN in mixed-precision training. Core idea: float16 has limited dynamic range, so small gradients underflow to zero and large gradients overflow to inf. GradScaler scales loss up before backward pass, which scales gradients into float16's representable range. Before optimizer step, it scales gradients back down. If any gradient is inf or NaN, meaning overflow still happened, GradScaler skips optimizer step and lowers scale factor for next iteration.

from torch.amp import autocast, GradScaler

scaler = GradScaler()
model = SentimentModel(vocab_size=10000).cuda()
optimizer = torch.optim.Adam(model.parameters(), lr=3e-4)
criterion = nn.CrossEntropyLoss()

for epoch in range(num_epochs):
    for inputs, labels in train_loader:
        inputs, labels = inputs.cuda(), labels.cuda()
        optimizer.zero_grad()

        # Forward pass in mixed precision (float16 where safe, float32 where needed)
        with autocast(device_type='cuda'):
            logits = model(inputs)
            loss = criterion(logits, labels)

        # Backward pass: scale loss up, compute gradients in float16
        scaler.scale(loss).backward()

        # Unscale gradients back to float32 range
        # If any gradients are inf/NaN, skip this update
        scaler.unscale_(optimizer)

        # Optional: clip gradients AFTER unscaling, BEFORE step
        torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)

        # Step: skips if inf/NaN detected, adjusts scale factor
        scaler.step(optimizer)
        scaler.update()

        # To monitor: check if steps are being skipped
        # scaler.get_scale() will decrease when inf is detected

scaler.update() adjusts scale factor dynamically. Successful step with no inf/NaN may increase scale to use more of float16 range. Skipped step decreases scale. Self-tuning behavior means you rarely need to touch initial scale. Default 216 = 65536 works for most runs.

Using detect_anomaly() to Find the Source

When NaN appears and source is unknown, detect_anomaly becomes debugger. Use it surgically. Wrap a few suspicious batches, not whole training loop.

# Strategy: run normally, switch to anomaly detection only when NaN detected
for step, (inputs, labels) in enumerate(train_loader):
    optimizer.zero_grad()
    logits = model(inputs)
    loss = criterion(logits, labels)

    if torch.isnan(loss) or torch.isinf(loss):
        print(f"NaN/Inf detected at step {step}!")
        print(f"Input stats: min={inputs.min()}, max={inputs.max()}")
        print(f"Logit stats: min={logits.min()}, max={logits.max()}")

        # Reload last good checkpoint and re-run this batch with anomaly detection
        model.load_state_dict(last_good_checkpoint)
        with torch.autograd.detect_anomaly():
            logits = model(inputs)
            loss = criterion(logits, labels)
            loss.backward()  # This will raise with traceback to the exact bad op
    else:
        loss.backward()
        optimizer.step()
        # Save checkpoint periodically
        if step % 100 == 0:
            last_good_checkpoint = {k: v.clone() for k, v in model.state_dict().items()}
⚠️ Warning

Common mixed-precision mistake: calling .item() on scaled loss for logging instead of real unscaled loss. Scaled loss may be 65536x larger than truth. Always log loss.item() from loss computed inside autocast context, not scaled version sent to backward().

Reading Loss Curves Like a Cardiologist Reads EKGs

Loss curve is model's EKG. Learn to read it, not glance and shrug. Each pathology leaves signature. Catalog time.

The Healthy Training Curve

Healthy curve for our sentiment classifier looks like this: training loss drops steeply in early epochs, then levels off. Validation loss follows same path, a bit higher. Gap stays small and fairly constant. Both curves eventually flatten. That gap is generalization gap — how much model memorized training-specific patterns instead of learning reusable features. Small stable gap means regularization is working.

Overfitting: The Crossover

Classic overfitting pattern is clear. Training loss keeps falling smoothly. At some epoch — crossover point — validation loss stops falling and starts rising. Model used up generalizable patterns and now memorizes noise in training set. Each extra epoch after crossover hurts unseen data.

Fix depends on severity. Mild overfitting, where validation loss rises slowly: add dropout, more weight decay, or early stopping. Severe overfitting, where validation loss climbs fast while train loss nears zero: get more data, add data augmentation, or shrink model. For our sentiment classifier, changing 3-layer LSTM to 1-layer LSTM, or adding dropout=0.5 between layers, often helps without killing too much capacity.

Underfitting: Both Curves High and Flat

If both training and validation loss stay high and barely move, model is underfitting. Maybe capacity is too low. Maybe training time is too short. Maybe learning rate is too low. Check learning rate first. Tiny LR can look exactly like low-capacity underfitting, and one fix takes 5 seconds while other needs architecture changes.

Double Descent: When More Capacity Gets Worse, Then Better

This phenomenon is deeply counterintuitive and breaks classical bias-variance story many ML students learned. Double descent works like this: as model capacity grows, test error first follows classical U-curve — down in underfitting region, then up in overfitting region. But keep increasing capacity past interpolation threshold, where model can fit training data perfectly, and test error starts falling again. Classical view says overparameterized models should be terrible. Practice says they are often best.

Interpolation threshold is point where model has enough parameters for zero training loss. There, model gets forced into one specific and often brittle solution — effectively only weight configuration that fits perfectly. Add more parameters and many zero-training-loss solutions appear. SGD, with implicit regularization from noise, tends to find smooth and generalizable ones. Polynomial analogy helps: with exactly as many coefficients as data points, fit is forced and wild. With many more coefficients, infinitely many perfect fits exist, and regularization can pick smooth one.

Double descent also appears across epochs. For fixed-size model, test error may rise during training, then fall again if you keep training long enough. Unsettling if you learned "stop early to avoid overfitting," but repeatedly observed across architectures and datasets.

Grokking: The Surprise Generalization

Power et al. (2022) described stranger thing: grokking, or delayed generalization. Model memorizes training set fast — training loss hits zero, training accuracy hits 100%. Validation stays near random. Then nothing happens for thousands of extra steps. Then validation accuracy suddenly jumps from chance to near-perfect. Model generalizes long after it memorizes.

Grokking shows strongest on small algorithmic tasks like modular arithmetic, but similar behavior appears elsewhere. Mechanism seems tied to weight decay. During memorization phase, model uses large weights to encode specific input-output mappings. As training continues, weight decay slowly shrinks weights until model "discovers" simpler, more generalizable solution with same training loss and smaller weights. Then phase transition happens: memorization solution becomes generalization solution.

Practical lesson is uncomfortable. Early stopping is not always trustworthy. Sometimes model needs much longer training to reach generalizable solution. If curves look like grokking — perfect training, flat validation — try more weight decay and more patience.

Loss Spikes

Watch loss curve and sudden sharp spike appears — from 0.3 to 2.5 in one step — then maybe recovers. Common causes: bad batch with outliers, corrupted samples, or wrong labels causing unusually large loss; or sudden effective learning-rate change, maybe from scheduler with warmup restarts.

With gradient clipping, spike often recovers because clipping limits damage from anomalous gradient. Without clipping, one spike can poison optimizer momentum and recovery takes many steps — or never comes. Reason gradient clipping is near-mandatory for long training runs.

💡 Key Insight

Log loss per batch, not per epoch. Epoch averages hide spikes and anomalies that matter for diagnosis. One batch that spikes loss 100x disappears inside epoch average but can still destabilize training. Per-batch logs expose it.

Reproducibility: Fighting Nondeterminism

Reproducibility in deep learning is harder than it should be. Set one random seed and expect same results? Nope. Typical PyTorch training pipeline has at least six sources of nondeterminism. Miss one and runs differ even with same seed.

The Sources of Chaos

Random number generators: Python random, NumPy numpy.random, and PyTorch torch.random are separate RNGs. Seeding one does nothing to others. Data augmentation might use NumPy, dropout uses PyTorch, shuffling uses Python. Seed all three.

CUDA atomics: Ops like scatter_add, index_add, and embedding lookups with duplicate indices use atomic GPU ops. Atomic ordering on CUDA depends on thread scheduling, so order changes between runs. Exact math says result is same. Floating-point math says different summation order gives different bits, because addition is not associative.

cuDNN benchmark mode: By default, PyTorch benchmarks several convolution algorithms on first call and keeps fastest. Chosen algorithm can change between runs because hardware state changes, and different algorithms produce slightly different floating-point results. Some cuDNN algorithms are also intrinsically non-deterministic.

DataLoader workers: With num_workers > 0, each worker is separate process with own dataset copy and own random state. Default seed depends on base seed plus worker ID, but worker initialization still needs care or worker randomness becomes unpredictable.

Python hash randomization: Python 3 randomizes hash seed at startup for security. That changes dictionary iteration and set order. If data pipeline depends on dict or set order, results change.

The Seeding Ceremony

import random
import numpy as np
import torch
import os

def seed_everything(seed=42):
    """Set all random seeds for reproducibility."""
    random.seed(seed)
    np.random.seed(seed)
    torch.manual_seed(seed)
    torch.cuda.manual_seed(seed)
    torch.cuda.manual_seed_all(seed)  # for multi-GPU
    os.environ['PYTHONHASHSEED'] = str(seed)

    # Force cuDNN to use deterministic algorithms (slower but reproducible)
    torch.backends.cudnn.deterministic = True
    torch.backends.cudnn.benchmark = False

seed_everything(42)

torch.backends.cudnn.deterministic = True forces deterministic cuDNN convolution algorithms. Cost: performance hit, sometimes 10-20%, because fastest algorithms may be non-deterministic. benchmark = False disables algorithm search, saving first-iteration time and removing another nondeterminism source.

Worker Initialization

DataLoader workers need extra care. Each worker has separate Python, NumPy, and PyTorch random states. If you do not seed workers explicitly, they get unpredictable seeds.

def worker_init_fn(worker_id):
    """Ensure each DataLoader worker has a unique but deterministic seed."""
    worker_seed = torch.initial_seed() % 2**32
    np.random.seed(worker_seed + worker_id)
    random.seed(worker_seed + worker_id)

train_loader = torch.utils.data.DataLoader(
    train_dataset,
    batch_size=32,
    shuffle=True,
    num_workers=4,
    worker_init_fn=worker_init_fn,
    generator=torch.Generator().manual_seed(42)  # Seed the shuffling RNG too
)

generator seeds RNG used for shuffling. Without it, shuffle order depends on global PyTorch RNG state, which model initialization or other ops may have changed between global seeding and DataLoader shuffle start.

The Strictest Mode

PyTorch has nuclear option: torch.use_deterministic_algorithms(True). Stricter than cudnn.deterministic. It forces every PyTorch op to use deterministic implementation. If op has none, PyTorch raises RuntimeError instead of silently accepting nondeterminism. Good for finding nondeterministic ops in pipeline. Can also break training if you use GPU ops without deterministic versions, like some scatter ops.

# Strictest reproducibility — will error on any non-deterministic op
torch.use_deterministic_algorithms(True)

# Some ops (like scatter_add on CUDA) will raise RuntimeError.
# You can set a fallback for those:
os.environ['CUBLAS_WORKSPACE_CONFIG'] = ':4096:8'
# This controls the workspace size for cuBLAS, which affects determinism of matmuls

The Uncomfortable Truth

Even after seeding everything, enabling deterministic flags, and initializing workers carefully, results can still differ across PyTorch versions, CUDA versions, GPU architectures — V100 vs A100 — and even different physical GPUs of same model because floating-point units vary slightly. If conclusions depend on exact 4th decimal place across machines, experiment is fragile. Document exact software and hardware stack. Report confidence intervals across multiple seeds. Accept that bitwise reproducibility across environments is losing battle.

🔥 Hard-Won Lesson

I once spent a week trying to reproduce paper results. My numbers stayed 0.3% lower. Cause was simple: paper used PyTorch 1.9, I used 1.13. Default initialization for nn.MultiheadAttention changed between versions, and that alone shifted results. Code matched. Seeds matched. Library versions did not.

Training Monitoring: Your Early Warning System

Training deep network without monitoring is like flying plane without instruments. Maybe you land. When something breaks, you learn only after crash. Set up instruments first.

What to Log and Why

Loss per batch: Not per epoch. Per batch. Epoch averages hide critical signal — catastrophic batch, slow drift, oscillations tied to data order. Batch-level losses show real training dynamics.

Gradient norms: Global gradient norm across all parameters is best early warning signal. If it creeps upward over time, instability is coming. Sudden spike means bad batch or learning-rate change. Near-zero norm means model stopped learning — maybe vanishing gradients, maybe convergence.

Parameter norms: Track L2 norm of each layer's weights over time. Unbounded growth usually means trouble — missing weight decay or architectural feedback loop. Collapse toward zero means learning signal is absent or regularization is too aggressive.

Learning rate: Always log actual LR, especially with schedulers. Easy bug: thinking scheduler is doing cosine annealing while it is stuck at initial LR because scheduler.step() never runs.

import torch
from torch.utils.tensorboard import SummaryWriter

writer = SummaryWriter('runs/sentiment_v1')

global_step = 0
for epoch in range(num_epochs):
    model.train()
    for batch_idx, (inputs, labels) in enumerate(train_loader):
        optimizer.zero_grad()
        logits = model(inputs)
        loss = criterion(logits, labels)
        loss.backward()

        # ---- MONITORING BLOCK ----
        # 1. Log the loss (use .item() to detach from graph!)
        writer.add_scalar('train/loss_batch', loss.item(), global_step)

        # 2. Gradient norm — the most important health metric
        total_norm = 0.0
        for p in model.parameters():
            if p.grad is not None:
                total_norm += p.grad.data.norm(2).item() ** 2
        total_norm = total_norm ** 0.5
        writer.add_scalar('train/grad_norm', total_norm, global_step)

        # 3. Per-layer gradient norms (find where gradients vanish/explode)
        for name, p in model.named_parameters():
            if p.grad is not None:
                writer.add_scalar(f'grad_norm/{name}', p.grad.norm().item(), global_step)

        # 4. Learning rate
        current_lr = optimizer.param_groups[0]['lr']
        writer.add_scalar('train/lr', current_lr, global_step)

        # 5. Alert conditions
        if total_norm > 100:
            print(f"WARNING: grad norm {total_norm:.1f} at step {global_step}")
        if torch.isnan(loss):
            print(f"FATAL: NaN loss at step {global_step}")
            break
        # ---- END MONITORING BLOCK ----

        # Clip gradients before stepping
        torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
        optimizer.step()
        scheduler.step()
        global_step += 1

writer.close()

The Memory Leak Trap: loss vs loss.item()

Common and nasty bug. Compute loss = criterion(logits, labels), and loss tensor carries entire computation graph — every intermediate tensor, weight reference, everything needed for .backward(). Store loss in list like losses.append(loss), and you keep whole graph alive. After 10,000 batches, you keep 10,000 graphs. GPU runs out of memory. Error says CUDA out of memory, but not why.

# BUG: This stores the entire computation graph for every batch
all_losses = []
for inputs, labels in train_loader:
    loss = criterion(model(inputs), labels)
    loss.backward()
    all_losses.append(loss)  # MEMORY LEAK! Keeps the graph alive
    optimizer.step()
    optimizer.zero_grad()

# FIX: Use .item() to extract the Python float, releasing the graph
all_losses = []
for inputs, labels in train_loader:
    loss = criterion(model(inputs), labels)
    loss.backward()
    all_losses.append(loss.item())  # Just a float — graph is freed
    optimizer.step()
    optimizer.zero_grad()

Same trap hits logging. writer.add_scalar('loss', loss, step) is fine because TensorBoard extracts value, but wandb.log({"loss": loss}) may keep reference depending on library version. Use loss.item() whenever extracting scalar for logging or accumulation.

Per-Layer Gradient Monitoring

Global gradient norm shows overall health. Per-layer norms show where trouble lives. In our sentiment classifier, healthy profile looks like: classifier head has largest gradients, LSTM layers have medium gradients, embedding layer has smallest. If embedding gradients are zero while rest look healthy, embeddings are frozen — maybe on purpose, maybe bug. If LSTM gradients are orders of magnitude larger than classifier gradients, recurrent computation likely has instability.

# Periodic detailed gradient check — run every N steps
def log_gradient_histogram(model, writer, step):
    """Log gradient histograms to TensorBoard for visual inspection."""
    for name, param in model.named_parameters():
        if param.grad is not None:
            writer.add_histogram(f'gradients/{name}', param.grad, step)
            writer.add_histogram(f'weights/{name}', param.data, step)

            # Also check for concerning patterns
            grad = param.grad
            if torch.any(torch.isnan(grad)):
                print(f"NaN gradient in {name} at step {step}")
            if torch.any(torch.isinf(grad)):
                print(f"Inf gradient in {name} at step {step}")
            if grad.abs().max().item() < 1e-10:
                print(f"Vanishing gradient in {name} at step {step}: max={grad.abs().max().item():.2e}")
💡 Key Insight

Ratio of gradient norm to parameter norm tells more than either alone. If ||grad|| / ||param|| > 1, update is larger than parameters themselves — instability warning. Healthy ratio is usually 1e-4 to 1e-1. Track ratio per layer and catch trouble early.

The Silent Bugs: No Error, No Warning, Quietly Broken

These bugs are worst. No crash. No NaN. No warning. They sit quietly and make model 5-15% worse than it should be. Then you compensate with hyperparameter tuning, bigger models, or more data, and none fully fixes training-loop bug. List below comes from personal pain and many colleagues hitting same traps.

Forgetting model.eval() During Validation

Probably most common silent bug in PyTorch training code. During training, model.train() enables dropout and makes BatchNorm use batch statistics. During validation, model.eval() disables dropout and makes BatchNorm use running statistics from training.

Forget model.eval() during validation and two things break. First, dropout still zeros neurons during inference, so validation predictions get noisier and worse. Second, BatchNorm uses current validation batch mean and variance instead of stable running stats. Small validation batches make those stats noisy and performance worse. Result: validation loss looks artificially high and noisy, model appears to generalize worse than it does, and you may add more regularization and make model even worse.

# THE BUG: No model.eval() during validation
for epoch in range(num_epochs):
    # Training
    model.train()
    for inputs, labels in train_loader:
        # ... training loop ...
        pass

    # Validation — BUG: model is still in train mode!
    val_loss = 0
    for inputs, labels in val_loader:
        with torch.no_grad():
            logits = model(inputs)  # Dropout is active! BatchNorm uses batch stats!
            val_loss += criterion(logits, labels).item()

# THE FIX: Explicitly set eval mode and restore train mode after
for epoch in range(num_epochs):
    model.train()
    for inputs, labels in train_loader:
        # ... training loop ...
        pass

    model.eval()  # <-- Critical: disables dropout, uses running stats for BN
    val_loss = 0
    with torch.no_grad():
        for inputs, labels in val_loader:
            logits = model(inputs)
            val_loss += criterion(logits, labels).item()
    model.train()  # <-- Restore train mode for next epoch

The Inverse: Forgetting model.train() After Evaluation

Mirror bug: you call model.eval() for validation, but forget model.train() before next training epoch. Now dropout stays off during training. Model trains without dropout regularization and often overfits. Validation loss, computed correctly in eval mode, shows overfitting, and you wonder why regularization fails. Reason: you accidentally turned it off.

The Double Softmax Trap (Revisited)

We covered this earlier, but repeat it because it is common silent bug. If model ends with nn.Softmax(dim=1) and you use nn.CrossEntropyLoss, softmax runs twice. Model still trains. Model still learns. Predictions are not random. But accuracy ceiling drops — maybe 82% instead of 91%. Double softmax compresses probability distribution, dampens gradients, and flattens loss surface in wrong way.

Gradient Accumulation + Scheduler Misalignment

Gradient accumulation simulates larger batch sizes when full batch does not fit in GPU memory. You accumulate gradients across N mini-batches, then do one optimizer step. Effective batch size is N × mini_batch_size. Bug appears when learning-rate scheduler steps every mini-batch instead of every optimizer step.

# BUG: Scheduler steps every mini-batch, not every optimizer step
accumulation_steps = 4
for step, (inputs, labels) in enumerate(train_loader):
    logits = model(inputs)
    loss = criterion(logits, labels) / accumulation_steps
    loss.backward()

    if (step + 1) % accumulation_steps == 0:
        optimizer.step()
        optimizer.zero_grad()

    scheduler.step()  # BUG: steps 4x too often!

# FIX: Step the scheduler only on optimizer steps
accumulation_steps = 4
for step, (inputs, labels) in enumerate(train_loader):
    logits = model(inputs)
    loss = criterion(logits, labels) / accumulation_steps
    loss.backward()

    if (step + 1) % accumulation_steps == 0:
        optimizer.step()
        optimizer.zero_grad()
        scheduler.step()  # Only step when the optimizer steps

If cosine annealing scheduler expects 1000 optimizer steps but steps 4000 times, it finishes full cycle in first quarter of training and spends remaining 75% at minimum LR. Effective learning-rate schedule is completely wrong.

Wrong Dtype: Integer Inputs to Floating-Point Operations

If you pass torch.int64 where torch.float32 is expected, PyTorch often handles it silently — but gradients do not flow through integer ops. For our sentiment classifier, token IDs should be torch.long for embedding lookup, but if you cast embedding output or another intermediate value to int, gradients die silently.

# Subtle dtype bug example
x = torch.tensor([1.5, 2.7, 3.2], requires_grad=True)
y = x.int()  # Casts to integer — NO GRADIENTS through this!
z = y.float() * 2.0  # Back to float, but grad chain is broken
z.sum().backward()
print(x.grad)  # None! The int() cast severed the gradient chain.

Augmentation on Validation Data

Random data augmentation — flips, crops, color jitter, and similar — belongs only on training data. Apply random augmentation to validation data and validation metrics gain useless variance. Some samples get easy augmentations and pass. Some get hard augmentations and fail. Metric jumps around between epochs, so real model improvement is hard to see. Validation should use deterministic preprocessing like center crop and fixed resize.

# Correct: separate transforms for train and val
train_transform = transforms.Compose([
    transforms.RandomHorizontalFlip(),
    transforms.RandomCrop(224),
    transforms.ColorJitter(brightness=0.2),
    transforms.ToTensor(),
    transforms.Normalize(mean, std),
])

val_transform = transforms.Compose([
    transforms.CenterCrop(224),       # Deterministic — no randomness
    transforms.ToTensor(),
    transforms.Normalize(mean, std),  # Same normalization as training
])

Preprocessing Leakage

Computing normalization statistics — mean and standard deviation — on whole dataset, including validation and test, is subtle data leakage. Preprocessing has now "seen" validation distribution. Effect is often small because mean and std change little when you add 10-20% more data. But on small datasets or competitive settings, metrics can inflate measurably. Correct move: compute mean and std on training set only, then use same values for both training and validation.

Frozen Pretrained Layers You Forgot About

When fine-tuning pretrained model, like BERT for our sentiment classifier, common pattern is freezing pretrained layers first and training only classifier head. Bug comes later: forgetting to unfreeze pretrained layers when full fine-tuning should start.

# Phase 1: Freeze BERT, train only the classifier head
for param in model.bert.parameters():
    param.requires_grad = False

# ... train for a few epochs on just the head ...

# Phase 2: Full fine-tuning — BUG if you forget this!
for param in model.bert.parameters():
    param.requires_grad = True  # Unfreeze BERT layers

# Also: you probably want a smaller learning rate for fine-tuning
optimizer = torch.optim.AdamW([
    {'params': model.bert.parameters(), 'lr': 2e-5},       # Small LR for pretrained
    {'params': model.classifier.parameters(), 'lr': 1e-3},  # Larger LR for head
])

Forget to unfreeze and model gets only whatever accuracy randomly initialized head can squeeze from fixed BERT features — maybe 85% instead of 93%. Easy wrong conclusion: BERT is not helping. Real problem: fine-tuning stayed frozen.

🔥 Hard-Won Lesson

Run parameter audit before training starts. Print every parameter name, shape, whether requires_grad is True, and optimizer group. Takes five lines. Catches frozen parameters, missing parameters, and optimizer misconfiguration in one pass.

# Parameter audit — run this before every training job
print("=" * 80)
print(f"{'Parameter':<50} {'Shape':<20} {'Grad':>5} {'#Params':>10}")
print("=" * 80)
total_params = 0
trainable_params = 0
for name, param in model.named_parameters():
    n = param.numel()
    total_params += n
    if param.requires_grad:
        trainable_params += n
    print(f"{name:<50} {str(list(param.shape)):<20} {str(param.requires_grad):>5} {n:>10,}")
print("=" * 80)
print(f"Total parameters: {total_params:,}")
print(f"Trainable parameters: {trainable_params:,}")
print(f"Frozen parameters: {total_params - trainable_params:,}")

Gradient Clipping: The Safety Net

Gradient clipping is technique you almost always want. Real question is not "should I clip?" but "what threshold?" Idea is simple: if one batch produces unusually large gradient — outlier input, numerical edge case, or unlucky sample mix — cap gradient magnitude before applying update. Direction stays same. Step size gets limited.

How clip_grad_norm_ Works

PyTorch's torch.nn.utils.clip_grad_norm_ computes global gradient norm — L2 norm of vector made by concatenating all parameter gradients. If norm exceeds max_norm, every gradient gets scaled by same factor until total norm becomes max_norm. Formula:

Given total gradient norm ||g|| and clipping threshold max_norm:

If ||g|| > max_norm: scale every gradient by max_norm / ||g||

If ||g|| <= max_norm: do nothing

This is global operation — all parameters together, not each parameter alone. Important reason: relative gradient magnitudes across layers stay preserved. If layer A has norm 5 and layer B has norm 3, total norm is 5.83, and you clip to 1.0, both get scaled by 1.0/5.83 ≈ 0.17. Layer A becomes 0.86, layer B 0.51 — same ratio as before. Per-parameter clipping would distort those ratios.

# The standard gradient clipping pattern
optimizer.zero_grad()
logits = model(inputs)
loss = criterion(logits, labels)
loss.backward()

# AFTER backward, BEFORE step — order matters!
grad_norm = torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
# grad_norm is the ORIGINAL norm before clipping — useful for logging

optimizer.step()

# Log the pre-clip gradient norm to monitor training health
writer.add_scalar('train/grad_norm_preclip', grad_norm.item(), global_step)

Choosing the Clipping Threshold

No universal best value, but good defaults and tuning method exist. HuggingFace Transformers uses max_norm=1.0 for BERT-style models, and that works for most transformers. RNNs and LSTMs often use 0.5 to 5.0. CNNs with BatchNorm may not need clipping because BatchNorm already stabilizes gradient magnitudes.

Practical method: start without clipping, log gradient norms for first hundred steps, inspect distribution, then set threshold around 90th or 95th percentile. That clips truly anomalous batches while leaving normal training alone.

Architecture Typical max_norm Notes
BERT / Transformers 1.0 HuggingFace default
GPT / Large LMs 1.0 Critical for training stability
LSTM / GRU 1.0 – 5.0 RNNs are prone to gradient spikes
ResNet / CNN 5.0 or none BatchNorm provides natural stability
GAN discriminator 0.5 – 1.0 Very unstable training dynamics

Monitoring the Clip Rate

If gradients get clipped every step, red flag. Gradient magnitudes are consistently too large, so clipping keeps interfering with optimizer updates. Common causes: learning rate too high, batch size too small so gradients are noisy, or architecture structurally amplifies gradients. If gradients are never clipped, threshold is either well chosen or clipping is unnecessary. Healthy clip rate sits between those extremes: most steps normal, occasional outlier batch tamed.

# Track clip statistics over training
clip_count = 0
total_steps = 0

for inputs, labels in train_loader:
    optimizer.zero_grad()
    loss = criterion(model(inputs), labels)
    loss.backward()

    grad_norm = torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
    total_steps += 1
    if grad_norm.item() > 1.0:
        clip_count += 1

    optimizer.step()

    if total_steps % 100 == 0:
        clip_rate = clip_count / total_steps
        print(f"Clip rate: {clip_rate:.1%} over {total_steps} steps")
        # Healthy: 1-10%. Concerning: >50%. Investigate: 100%.
⚠️ Warning

clip_grad_value_ also exists. It clips each gradient element independently into [-value, value]. That is different from norm clipping and usually less useful because it distorts gradient direction. Norm clipping preserves direction and only scales magnitude, which is usually what you want. Use clip_grad_norm_ unless you have specific reason not to.

Weight Initialization: Where It All Begins

Initial weight values choose optimizer's starting point. Get them wrong and model may not learn at all — activations saturate, gradients vanish, loss stays at initial value. Get them right and gradient flow stays healthy from first step. Initialization theory is one of deep learning's cleaner mathematical pieces.

The Core Problem

Consider layer y = Wx where x has n inputs and W is weight matrix. Each element of y is sum of n terms: y_j = sum(W_ji * x_i). If each weight W_ji has variance Var(w) and each input x_i has variance Var(x), with independence, then variance of y_j is:

Var(y) = n × Var(w) × Var(x)

If Var(w) is too large, Var(y) greatly exceeds Var(x). Each layer amplifies signal. After 10 layers, activations scale by (n × Var(w))^10. If n × Var(w) = 2, that is 210 = 1024× amplification. Activations explode. If Var(w) is too small, each layer attenuates signal, and after 10 layers activations and gradients become tiny.

Goal: choose Var(w) so n × Var(w) ≈ 1, meaning Var(w) = 1/n. Then activation variance stays roughly constant across layers. Core idea. Exact details depend on activation function.

Xavier / Glorot Initialization (2010)

Xavier Glorot and Yoshua Bengio proposed initializing weights with variance 2 / (fan_in + fan_out), where fan_in is number of input neurons and fan_out is number of output neurons. Averaging fan_in and fan_out keeps both forward-pass activations and backward-pass gradients at stable variance. Derivation assumes linear activations or tanh, which is near-linear around zero.

# Xavier/Glorot initialization — designed for tanh or sigmoid
import torch.nn.init as init

linear = nn.Linear(256, 128)
init.xavier_uniform_(linear.weight)   # Uniform distribution variant
# or
init.xavier_normal_(linear.weight)    # Normal distribution variant

# The uniform variant samples from:
# U(-sqrt(6 / (fan_in + fan_out)), sqrt(6 / (fan_in + fan_out)))
# For fan_in=256, fan_out=128: U(-0.0625, 0.0625)

# The normal variant samples from:
# N(0, 2 / (fan_in + fan_out))
# For fan_in=256, fan_out=128: N(0, 0.0052) i.e., std ≈ 0.072

Kaiming / He Initialization (2015)

Xavier initialization assumes activation is roughly linear near zero. ReLU breaks that. ReLU zeros all negative activations, effectively killing half signal. Use Xavier with ReLU and activation variance halves each layer because half neurons output zero. After 10 layers, activations shrink by 2-10 ≈ 0.001×. Gradients suffer same fate.

Kaiming He et al. (2015) fixed this by doubling variance to offset ReLU's 50% kill rate: Var(w) = 2 / fan_in. Factor 2 compensates for half the neurons going dead. For Leaky ReLU with negative slope a, formula generalizes to Var(w) = 2 / ((1 + a²) × fan_in).

# Kaiming/He initialization — designed for ReLU
linear = nn.Linear(256, 128)
init.kaiming_uniform_(linear.weight, nonlinearity='relu')
# or
init.kaiming_normal_(linear.weight, nonlinearity='relu')

# For Leaky ReLU with negative slope 0.01:
init.kaiming_normal_(linear.weight, a=0.01, nonlinearity='leaky_relu')

# PyTorch's default for nn.Linear is actually Kaiming Uniform
# So if you're using ReLU, you rarely need to change initialization manually

PyTorch Defaults

PyTorch defaults: nn.Linear uses Kaiming Uniform, He variant. Good default for ReLU networks. nn.Conv2d also uses Kaiming Uniform. nn.Embedding initializes from N(0, 1). nn.LSTM initializes weights from U(-1/sqrt(hidden_size), 1/sqrt(hidden_size)). In practice these defaults work for most architectures. Manual initialization is usually needed only when symptoms appear, like vanishing activations in deep nets or instability in first few steps.

BatchNorm Makes Initialization Matter Less

One underappreciated BatchNorm benefit: it makes weight initialization less critical. BatchNorm normalizes each layer's activations to zero mean and unit variance during training. So even if weights start badly and activations have wild scales, BatchNorm resets scale each layer. Initialization still matters for very first forward pass before BatchNorm has good statistics, but overall impact drops a lot.

Reason modern BatchNorm architectures like ResNets train so easily. They survive initialization choices that would kill unnormalized networks.

Transformer Initialization

Transformers use different initialization convention. Standard approach from original GPT and BERT initializes weights from N(0, 0.02). Small standard deviation keeps pre-LayerNorm activations from growing too large early in training. Some implementations also scale residual-path weights by 1/sqrt(N), where N is number of residual layers, following GPT-2. Goal: keep residual stream output from growing with depth at initialization.

# Transformer-style initialization
def init_transformer_weights(module):
    """Initialize weights following GPT-2 / BERT convention."""
    if isinstance(module, nn.Linear):
        torch.nn.init.normal_(module.weight, mean=0.0, std=0.02)
        if module.bias is not None:
            torch.nn.init.zeros_(module.bias)
    elif isinstance(module, nn.Embedding):
        torch.nn.init.normal_(module.weight, mean=0.0, std=0.02)
    elif isinstance(module, nn.LayerNorm):
        torch.nn.init.ones_(module.weight)
        torch.nn.init.zeros_(module.bias)

model.apply(init_transformer_weights)
💡 Key Insight

Rule of thumb: ReLU without BatchNorm needs Kaiming initialization. BatchNorm makes initialization less critical. Transformers usually want small-std convention, 0.02, and sometimes residual scaling. If unsure, PyTorch defaults are reasonable. But verify by printing first-batch activation stats. If some layer output has mean far from zero or variance far from one, initialization is likely culprit.

Putting It All Together: A Debugging Checklist

Turn whole discussion into checklist. If sentiment classifier — or any model — trains badly, work list in order. Each step either fixes problem or rules out one bug class.

Before Training Starts

1. Data sanity check. Print a few samples. Decode token IDs back to text. Verify labels match content. Check shapes, dtypes, and value ranges. Two minutes. Catches most bugs.

2. Parameter audit. Print parameter names, shapes, requires_grad status, and total counts. Verify nothing froze by accident and optimizer sees all trainable parameters.

3. Initial loss check. Run one batch forward, compute loss, compare with random-guess baseline -log(1/C). If value is far off, investigate before training.

4. Single-batch overfit test. Train one fixed batch for 500 steps. Loss must reach near-zero. If not, fundamental bug exists. Fix it first.

During Training

5. Monitor loss per batch, not per epoch. Look for spikes, plateaus, and divergence.

6. Monitor gradient norms. Track global norm each step and per-layer norms periodically. Alert when norm crosses threshold.

7. Gradient clipping enabled. max_norm=1.0 is safe default. Log clip rate.

8. model.eval() for validation, model.train() for training. Every time. No exceptions.

9. loss.item() for logging. Never store loss tensor itself in list.

10. Separate transforms for train and val. Random augmentation only on training data.

When Things Go Wrong

11. NaN loss? Check numerical instability: log(0), sqrt(0), exp(large). Add epsilon guards. If using mixed precision, verify GradScaler setup. Use detect_anomaly() to find exact op.

12. Loss plateaus? Run learning-rate finder. Check optimizer is stepping; scheduler may have driven LR to zero. Verify all layers have non-zero gradients.

13. Val loss diverges while train loss drops? Overfitting. Add dropout, weight decay, data augmentation, or more data.

14. Both losses high? Underfitting. Increase model capacity, raise learning rate, or train longer.

15. Results differ between runs? Seed all RNGs, set deterministic flags, use worker_init_fn. Accept that cross-machine reproducibility is approximate.

# The complete, battle-tested training loop for our sentiment classifier
import torch
import torch.nn as nn
from torch.amp import autocast, GradScaler
import random
import numpy as np

# ---- REPRODUCIBILITY ----
def seed_everything(seed=42):
    random.seed(seed)
    np.random.seed(seed)
    torch.manual_seed(seed)
    torch.cuda.manual_seed_all(seed)
    torch.backends.cudnn.deterministic = True
    torch.backends.cudnn.benchmark = False

seed_everything(42)

# ---- MODEL ----
model = SentimentModel(vocab_size=10000).cuda()

# ---- PARAMETER AUDIT ----
total = sum(p.numel() for p in model.parameters())
trainable = sum(p.numel() for p in model.parameters() if p.requires_grad)
print(f"Parameters: {total:,} total, {trainable:,} trainable")

# ---- OPTIMIZER AND SCHEDULER ----
optimizer = torch.optim.AdamW(model.parameters(), lr=3e-4, weight_decay=0.01)
scheduler = torch.optim.lr_scheduler.CosineAnnealingLR(optimizer, T_max=num_epochs)
criterion = nn.CrossEntropyLoss()
scaler = GradScaler()

# ---- INITIAL LOSS CHECK ----
model.eval()
with torch.no_grad():
    sample_batch = next(iter(train_loader))
    init_loss = criterion(model(sample_batch[0].cuda()), sample_batch[1].cuda())
    expected = torch.log(torch.tensor(2.0))  # 2-class: 0.693
    print(f"Initial loss: {init_loss.item():.4f} (expected: {expected.item():.4f})")
model.train()

# ---- TRAINING LOOP ----
best_val_loss = float('inf')
patience_counter = 0

for epoch in range(num_epochs):
    model.train()
    epoch_loss = 0.0
    clip_count = 0

    for step, (inputs, labels) in enumerate(train_loader):
        inputs, labels = inputs.cuda(), labels.cuda()
        optimizer.zero_grad()

        with autocast(device_type='cuda'):
            logits = model(inputs)
            loss = criterion(logits, labels)

        scaler.scale(loss).backward()
        scaler.unscale_(optimizer)

        # Gradient clipping — log the pre-clip norm
        grad_norm = torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
        if grad_norm.item() > 1.0:
            clip_count += 1

        scaler.step(optimizer)
        scaler.update()

        # Safe logging — .item() detaches from graph
        batch_loss = loss.item()
        epoch_loss += batch_loss

        if torch.isnan(loss):
            print(f"NaN at epoch {epoch}, step {step}. Stopping.")
            break

    scheduler.step()

    # ---- VALIDATION (eval mode!) ----
    model.eval()
    val_loss = 0.0
    correct = 0
    total_samples = 0
    with torch.no_grad():
        for inputs, labels in val_loader:
            inputs, labels = inputs.cuda(), labels.cuda()
            logits = model(inputs)
            val_loss += criterion(logits, labels).item() * labels.size(0)
            correct += (logits.argmax(1) == labels).sum().item()
            total_samples += labels.size(0)

    val_loss /= total_samples
    val_acc = correct / total_samples
    avg_train_loss = epoch_loss / len(train_loader)

    print(f"Epoch {epoch}: train_loss={avg_train_loss:.4f}, "
          f"val_loss={val_loss:.4f}, val_acc={val_acc:.3f}, "
          f"clip_rate={clip_count/len(train_loader):.1%}, "
          f"lr={optimizer.param_groups[0]['lr']:.2e}")

    # Early stopping
    if val_loss < best_val_loss:
        best_val_loss = val_loss
        patience_counter = 0
        torch.save(model.state_dict(), 'best_model.pt')
    else:
        patience_counter += 1
        if patience_counter >= 5:
            print("Early stopping triggered.")
            break

That is training loop I use for every new project, sentiment classifier or not. Not shortest loop. It includes monitoring, clipping, mixed precision, reproducibility, and early stopping. Each feature has saved me debugging hours at least once. Extra 20 lines pay for themselves when first training run goes bad.

💡 Key Insight

Debugging deep learning training is not about memorizing every failure mode. It is about following system — data, loss, gradients, architecture — and working methodically. Bugs here cover about 95% of failures I see in practice. Remaining 5% are novel, and by then you have enough evidence to diagnose them.
