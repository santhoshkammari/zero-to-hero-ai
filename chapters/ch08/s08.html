TensorFlow & Keras

Chapter 8: Training Deep Networks 7 subtopics
TL;DR

TensorFlow is Google's end-to-end machine learning platform, and Keras is its high-level API for building and training neural networks. Together they give you a spectrum of control: Keras Sequential and Functional APIs for fast prototyping, tf.data for efficient input pipelines, GradientTape for custom training loops when you need full control, and a deployment story that stretches from mobile phones (TF Lite) to production servers (TF Serving). This section walks through each layer of the stack with working code you can run today.

tf.Tensor Basics

Before you build any model, you need to understand what TensorFlow actually computes on. A tf.Tensor is an immutable, multi-dimensional array — the fundamental data structure in TensorFlow. If you have used NumPy arrays, tensors will feel familiar. The key difference is that tensors can live on GPUs and TPUs, and TensorFlow can automatically differentiate operations performed on them.

Every tensor has a shape (the dimensions), a dtype (the data type, like float32 or int64), and a device placement (CPU, GPU, or TPU). You create tensors from Python lists, NumPy arrays, or with factory functions. Once created, tensors are immutable — operations return new tensors rather than modifying existing ones.

import tensorflow as tf
import numpy as np

# Creating tensors
a = tf.constant([[1.0, 2.0], [3.0, 4.0]])         # From a Python list
b = tf.zeros((3, 4))                                # 3x4 matrix of zeros
c = tf.random.normal((2, 3), mean=0.0, stddev=1.0)  # Random normal

# From NumPy (zero-copy when possible)
np_array = np.array([1.0, 2.0, 3.0])
d = tf.constant(np_array)

# Inspecting tensors
print(a.shape)    # TensorShape([2, 2])
print(a.dtype)    # <dtype: 'float32'>
print(a.device)   # /job:localhost/replica:0/task:0/device:CPU:0

# Basic operations — all return new tensors
e = tf.matmul(a, tf.transpose(a))   # Matrix multiply
f = tf.reduce_mean(a, axis=1)       # Mean along axis 1
g = a + 10                           # Broadcasting scalar
h = tf.reshape(a, (4,))             # Reshape to 1D

# Moving to GPU (if available)
if tf.config.list_physical_devices('GPU'):
    with tf.device('/GPU:0'):
        gpu_tensor = tf.constant([1.0, 2.0, 3.0])

One thing that catches newcomers: TensorFlow 2.x runs in eager mode by default. That means operations execute immediately and return concrete values, just like normal Python code. In TensorFlow 1.x, you had to build a computation graph first and then run it inside a session. Eager mode is far more intuitive for debugging. When you need the performance of graph execution, you annotate functions with @tf.function, and TensorFlow traces them into an optimized graph automatically.

💡 Key Insight

TensorFlow's @tf.function decorator is your bridge between eager debugging and graph performance. Write and debug your code eagerly, then add @tf.function to the hot path. The first call traces the function into a graph; subsequent calls reuse the cached graph. Be careful with Python side effects inside traced functions — they only execute during tracing, not on subsequent calls.

Keras Sequential API

The fastest way to build a neural network in TensorFlow is the Keras Sequential API. It is exactly what the name suggests: a linear stack of layers where each layer has one input and one output. You hand it a list of layers, and it wires them together in order. For the majority of standard architectures — image classifiers, simple text models, basic regression networks — Sequential is all you need.

Let us build a complete image classifier from scratch. This example trains on MNIST (handwritten digits), but the pattern is identical for any classification task. The Sequential API handles weight initialization, forward pass construction, and shape inference automatically.

import tensorflow as tf
from tensorflow import keras
from tensorflow.keras import layers

# Load and preprocess data
(x_train, y_train), (x_test, y_test) = keras.datasets.mnist.load_data()
x_train = x_train.astype("float32") / 255.0   # Normalize to [0, 1]
x_test = x_test.astype("float32") / 255.0
x_train = x_train.reshape(-1, 28, 28, 1)       # Add channel dimension
x_test = x_test.reshape(-1, 28, 28, 1)

# Build model with Sequential API
model = keras.Sequential([
    layers.Input(shape=(28, 28, 1)),
    layers.Conv2D(32, kernel_size=3, activation="relu"),
    layers.BatchNormalization(),
    layers.MaxPooling2D(pool_size=2),
    layers.Conv2D(64, kernel_size=3, activation="relu"),
    layers.BatchNormalization(),
    layers.MaxPooling2D(pool_size=2),
    layers.Flatten(),
    layers.Dropout(0.3),
    layers.Dense(128, activation="relu"),
    layers.Dense(10, activation="softmax"),
])

# Compile: specify optimizer, loss, and metrics
model.compile(
    optimizer=keras.optimizers.Adam(learning_rate=1e-3),
    loss="sparse_categorical_crossentropy",
    metrics=["accuracy"],
)

# Print architecture summary
model.summary()

# Train
history = model.fit(
    x_train, y_train,
    batch_size=64,
    epochs=10,
    validation_split=0.1,
)

# Evaluate on held-out test set
test_loss, test_acc = model.evaluate(x_test, y_test)
print(f"Test accuracy: {test_acc:.4f}")

A few things to notice. The compile step does not actually run any computation — it configures the training loop by selecting the optimizer, loss function, and metrics to track. The fit call is where all the real work happens: forward pass, loss calculation, backward pass, and parameter updates, repeated for every batch across all epochs. The validation_split=0.1 argument holds out 10% of the training data for validation, giving you an honest check on overfitting after each epoch.

⚠️ Warning

When using sparse_categorical_crossentropy, your labels should be integers (like [0, 3, 7]). If your labels are one-hot encoded (like [[1,0,0,...], [0,0,0,...,1,0,...]]), use categorical_crossentropy instead. Mixing these up is one of the most common Keras bugs, and the error message is not always helpful.

Keras Functional API

Sequential models break down the moment your architecture is not a straight line. Need skip connections like ResNet? Multiple inputs? Multiple outputs? Shared layers? That is where the Keras Functional API comes in. Instead of stacking layers linearly, you treat each layer as a callable function and explicitly wire inputs to outputs. This gives you full control over the computation graph while still getting all the Keras conveniences like compile, fit, and evaluate.

Here is a concrete example: a small residual network with skip connections. The skip connection adds the input of a block directly to its output, which helps gradients flow through deep networks without vanishing.

from tensorflow.keras import layers, Model, Input

# Define the input tensor
inputs = Input(shape=(28, 28, 1), name="image_input")

# First conv block
x = layers.Conv2D(32, 3, padding="same", activation="relu")(inputs)
x = layers.BatchNormalization()(x)

# Residual block — this is why we need Functional API
shortcut = x                                         # Save for skip connection
x = layers.Conv2D(32, 3, padding="same", activation="relu")(x)
x = layers.BatchNormalization()(x)
x = layers.Conv2D(32, 3, padding="same")(x)
x = layers.BatchNormalization()(x)
x = layers.Add()([x, shortcut])                      # Skip connection
x = layers.Activation("relu")(x)

# Head
x = layers.GlobalAveragePooling2D()(x)
x = layers.Dense(64, activation="relu")(x)
x = layers.Dropout(0.3)(x)
outputs = layers.Dense(10, activation="softmax", name="class_output")(x)

# Create and compile model
model = Model(inputs=inputs, outputs=outputs, name="mini_resnet")
model.compile(
    optimizer="adam",
    loss="sparse_categorical_crossentropy",
    metrics=["accuracy"],
)
model.summary()

The Functional API also shines when you need multiple inputs or multiple outputs. For example, imagine a model that takes both an image and metadata (like the camera settings) and predicts both the digit class and whether the image is high quality. You would define two Input layers, merge them with a Concatenate layer, and have two output heads with separate loss functions. The Functional API makes this trivial; the Sequential API cannot express it at all.

# Multi-input, multi-output example
image_input = Input(shape=(28, 28, 1), name="image")
meta_input = Input(shape=(4,), name="metadata")

# Image branch
x = layers.Conv2D(32, 3, activation="relu")(image_input)
x = layers.GlobalAveragePooling2D()(x)

# Merge branches
merged = layers.Concatenate()([x, meta_input])
merged = layers.Dense(64, activation="relu")(merged)

# Two output heads
class_out = layers.Dense(10, activation="softmax", name="class")(merged)
quality_out = layers.Dense(1, activation="sigmoid", name="quality")(merged)

multi_model = Model(
    inputs=[image_input, meta_input],
    outputs=[class_out, quality_out],
)
multi_model.compile(
    optimizer="adam",
    loss={"class": "sparse_categorical_crossentropy", "quality": "binary_crossentropy"},
    loss_weights={"class": 1.0, "quality": 0.5},
    metrics={"class": "accuracy", "quality": "AUC"},
)

tf.data Pipeline

Your model is only as fast as its data pipeline. If the GPU is waiting for the CPU to load and preprocess the next batch, you are wasting expensive compute. The tf.data API solves this by building efficient, parallelized input pipelines that can overlap data loading, preprocessing, and training. Think of it as an assembly line: while the GPU trains on batch N, the CPU is already preparing batch N+1.

The core abstraction is a tf.data.Dataset, which represents a sequence of elements. You chain transformations like map, batch, shuffle, prefetch, and cache to build your pipeline. Each transformation returns a new Dataset, so you can compose them freely.

import tensorflow as tf

# Create dataset from tensors
(x_train, y_train), _ = tf.keras.datasets.mnist.load_data()
x_train = x_train.astype("float32") / 255.0

dataset = tf.data.Dataset.from_tensor_slices((x_train, y_train))

# Preprocessing function
def preprocess(image, label):
    image = tf.expand_dims(image, axis=-1)         # Add channel dim
    image = tf.image.random_flip_left_right(image)  # Data augmentation
    image = tf.image.random_brightness(image, 0.1)
    return image, label

# Build the pipeline
BATCH_SIZE = 64
SHUFFLE_BUFFER = 10000

train_ds = (
    dataset
    .shuffle(SHUFFLE_BUFFER)              # Randomize order
    .map(preprocess,
         num_parallel_calls=tf.data.AUTOTUNE)  # Parallelize preprocessing
    .batch(BATCH_SIZE)                    # Group into batches
    .cache()                              # Cache after first epoch
    .prefetch(tf.data.AUTOTUNE)           # Overlap with training
)

# Use directly in model.fit
model.fit(train_ds, epochs=10)

The order of operations matters. You generally want to shuffle before batching (so each batch has a random mix), map before or after batching depending on whether your preprocessing benefits from vectorization, and always prefetch last. The cache call stores the dataset in memory (or on disk if you pass a filename) after the first full pass, so subsequent epochs skip the expensive loading and preprocessing steps.

💡 Key Insight

tf.data.AUTOTUNE tells TensorFlow to dynamically tune the number of parallel calls and the prefetch buffer size at runtime. Use it instead of hard-coding values. It monitors your pipeline's throughput and adjusts automatically, which is especially helpful when running on different hardware where the optimal parallelism varies.

Reading from Files

For datasets that do not fit in memory, tf.data can read directly from TFRecord files, CSV files, or text files without loading everything at once. TFRecord is TensorFlow's own binary format and is the fastest option for large-scale training. You serialize your data into protocol buffers, write them to disk, and then stream them during training.

# Reading from TFRecord files
filenames = ["data/train_001.tfrecord", "data/train_002.tfrecord"]

raw_dataset = tf.data.TFRecordDataset(filenames)

# Parse function for your specific schema
def parse_example(serialized):
    features = tf.io.parse_single_example(serialized, {
        "image": tf.io.FixedLenFeature([], tf.string),
        "label": tf.io.FixedLenFeature([], tf.int64),
    })
    image = tf.io.decode_raw(features["image"], tf.float32)
    image = tf.reshape(image, (28, 28, 1))
    return image, features["label"]

parsed_ds = (
    raw_dataset
    .map(parse_example, num_parallel_calls=tf.data.AUTOTUNE)
    .shuffle(10000)
    .batch(64)
    .prefetch(tf.data.AUTOTUNE)
)

Custom Training with GradientTape

Keras's model.fit handles the training loop for you, and that is great — until it is not. Sometimes you need full control: custom loss calculations that depend on multiple model outputs, gradient clipping with specific logic, adversarial training where two models train against each other, or research experiments where you modify gradients before applying them. That is where GradientTape comes in.

tf.GradientTape records operations as they execute so it can compute gradients via automatic differentiation. You wrap your forward pass inside a with tf.GradientTape() as tape: block, compute your loss, then call tape.gradient(loss, model.trainable_variables) to get the gradients of the loss with respect to every trainable parameter. After that, you hand the gradients to an optimizer.

import tensorflow as tf
from tensorflow import keras

# Build a simple model
model = keras.Sequential([
    keras.layers.Flatten(input_shape=(28, 28, 1)),
    keras.layers.Dense(128, activation="relu"),
    keras.layers.Dense(10),
])

optimizer = keras.optimizers.Adam(learning_rate=1e-3)
loss_fn = keras.losses.SparseCategoricalCrossentropy(from_logits=True)
train_acc_metric = keras.metrics.SparseCategoricalAccuracy()

@tf.function  # Compile to graph for performance
def train_step(images, labels):
    with tf.GradientTape() as tape:
        logits = model(images, training=True)   # Forward pass
        loss = loss_fn(labels, logits)

    # Compute gradients
    gradients = tape.gradient(loss, model.trainable_variables)

    # Optional: clip gradients to prevent explosion
    gradients = [tf.clip_by_norm(g, 1.0) for g in gradients]

    # Apply gradients
    optimizer.apply_gradients(zip(gradients, model.trainable_variables))

    # Update metrics
    train_acc_metric.update_state(labels, logits)
    return loss

# Full training loop
EPOCHS = 10

for epoch in range(EPOCHS):
    train_acc_metric.reset_state()
    epoch_loss = 0.0
    num_batches = 0

    for images, labels in train_ds:
        loss = train_step(images, labels)
        epoch_loss += loss.numpy()
        num_batches += 1

    avg_loss = epoch_loss / num_batches
    train_acc = train_acc_metric.result().numpy()
    print(f"Epoch {epoch+1}/{EPOCHS} — "
          f"Loss: {avg_loss:.4f}, Accuracy: {train_acc:.4f}")

Notice the training=True argument in model(images, training=True). This is critical. Layers like Dropout and BatchNormalization behave differently during training and inference. Dropout randomly zeros out neurons during training but passes everything through during inference. BatchNormalization updates its running statistics during training but uses the stored statistics during inference. Forgetting training=True is a subtle bug that leads to mysteriously bad training performance.

⚠️ Warning

By default, GradientTape releases its resources after you call tape.gradient() once. If you need to compute multiple gradients from the same tape (for example, gradients of different losses), create it with tf.GradientTape(persistent=True). But remember to delete the tape manually with del tape when you are done to free memory.

Callbacks

Callbacks are hooks that execute at specific points during training — at the start or end of an epoch, before or after a batch, and so on. They let you bolt on behavior without modifying the training loop. Keras ships with several built-in callbacks, and you can write your own by subclassing keras.callbacks.Callback.

The most commonly used callbacks are ModelCheckpoint (save the model when validation loss improves), EarlyStopping (stop training when the model stops improving), ReduceLROnPlateau (reduce the learning rate when progress stalls), and TensorBoard (log metrics for visualization). You pass them as a list to model.fit.

from tensorflow.keras.callbacks import (
    ModelCheckpoint,
    EarlyStopping,
    ReduceLROnPlateau,
    TensorBoard,
)

callbacks = [
    # Save best model based on validation loss
    ModelCheckpoint(
        filepath="best_model.keras",
        monitor="val_loss",
        save_best_only=True,
        verbose=1,
    ),
    # Stop if val_loss doesn't improve for 5 epochs
    EarlyStopping(
        monitor="val_loss",
        patience=5,
        restore_best_weights=True,
    ),
    # Halve the learning rate if val_loss plateaus for 3 epochs
    ReduceLROnPlateau(
        monitor="val_loss",
        factor=0.5,
        patience=3,
        min_lr=1e-6,
    ),
    # Log to TensorBoard
    TensorBoard(log_dir="./logs", histogram_freq=1),
]

history = model.fit(
    train_ds,
    validation_data=val_ds,
    epochs=100,           # Set high — EarlyStopping will stop you
    callbacks=callbacks,
)

A practical workflow: set your epoch count high and let EarlyStopping decide when to stop. The restore_best_weights=True parameter ensures your model reverts to the best checkpoint after stopping, not the last (possibly overfit) epoch. Combined with ReduceLROnPlateau, you get a training run that automatically adapts its learning rate schedule and stops at the right time.

Custom Callbacks

Writing your own callback is straightforward. You subclass keras.callbacks.Callback and override the methods you care about. For instance, here is a callback that logs the learning rate at the end of each epoch and sends a notification when training finishes.

class TrainingMonitor(keras.callbacks.Callback):
    def on_epoch_end(self, epoch, logs=None):
        lr = float(self.model.optimizer.learning_rate)
        print(f"\n  → Epoch {epoch+1} LR: {lr:.2e} | "
              f"val_loss: {logs.get('val_loss', 'N/A'):.4f}")

    def on_train_end(self, logs=None):
        print("\n✅ Training complete!")
        # You could send a Slack notification, save a report, etc.
💡 Key Insight

Callbacks are the Keras answer to "I want to do X during training." Before writing a custom training loop with GradientTape, check if a callback can solve your problem. Callbacks keep your training loop clean and your custom logic isolated and reusable across projects.

SavedModel Format

Training a model is only half the job. You need to save it in a way that is portable, versionable, and usable in production. TensorFlow's SavedModel format is the standard answer. It serializes the entire model — architecture, weights, optimizer state, and even the computation graph — into a directory on disk. Unlike simple weight checkpoints, a SavedModel is self-contained: you can load and run it without having the original Python code that defined the model.

# Save the entire model in SavedModel format
model.save("my_model")   # Creates a directory called my_model/

# What's inside:
# my_model/
#   ├── saved_model.pb      # Serialized computation graph
#   ├── fingerprint.pb      # Model fingerprint
#   ├── variables/
#   │   ├── variables.data-00000-of-00001
#   │   └── variables.index
#   └── assets/             # Extra files (e.g., vocabulary)

# Load it back — no need for the original model code
loaded_model = tf.keras.models.load_model("my_model")

# Use it for inference
predictions = loaded_model.predict(x_test[:5])
print(predictions.argmax(axis=1))

# You can also save in the newer .keras format
model.save("my_model.keras")
loaded = tf.keras.models.load_model("my_model.keras")

The SavedModel format also supports signatures, which define the expected input and output shapes and types. This is crucial for production serving because it lets the serving infrastructure validate requests before running inference. When you save a Keras model, it automatically creates a default serving signature based on the model's input and output specifications.

# Saving with explicit signatures for serving
@tf.function(input_signature=[
    tf.TensorSpec(shape=(None, 28, 28, 1), dtype=tf.float32)
])
def serve(image):
    return {"predictions": model(image, training=False)}

tf.saved_model.save(
    model,
    "serving_model",
    signatures={"serving_default": serve}
)

# Load and inspect signatures
loaded = tf.saved_model.load("serving_model")
infer = loaded.signatures["serving_default"]
output = infer(tf.constant(x_test[:3]))
print(output["predictions"])
⚠️ Warning

The older model.save("model.h5") HDF5 format is still supported but is being phased out. It cannot serialize custom objects, @tf.function traces, or complex model topologies as reliably as the SavedModel or .keras formats. For new projects, always use model.save("name.keras") or the SavedModel directory format.

TensorBoard

TensorBoard is TensorFlow's built-in visualization toolkit. It reads log files written during training and gives you interactive dashboards for loss curves, accuracy metrics, weight histograms, computation graphs, image samples, and more. It runs as a local web server and opens in your browser.

The simplest way to use TensorBoard is through the Keras TensorBoard callback shown earlier. But you can also write custom scalars, images, text, and histograms using tf.summary. This is especially useful when you are writing custom training loops with GradientTape.

import datetime

# Set up log directory with timestamp
log_dir = "logs/" + datetime.datetime.now().strftime("%Y%m%d-%H%M%S")
summary_writer = tf.summary.create_file_writer(log_dir)

# Inside your custom training loop:
for epoch in range(EPOCHS):
    for step, (images, labels) in enumerate(train_ds):
        loss = train_step(images, labels)

        # Log custom scalars
        with summary_writer.as_default():
            tf.summary.scalar("train_loss", loss, step=epoch * steps_per_epoch + step)
            tf.summary.scalar("learning_rate",
                              optimizer.learning_rate, step=epoch)

    # Log weight histograms at end of each epoch
    with summary_writer.as_default():
        for var in model.trainable_variables:
            tf.summary.histogram(var.name, var, step=epoch)

# Launch TensorBoard from terminal:
# tensorboard --logdir=logs/

TensorBoard also supports comparing multiple runs. If you save logs to different subdirectories (one per experiment), TensorBoard overlays them on the same chart. This makes hyperparameter comparison effortless — run five experiments with different learning rates, and you can see all five loss curves side by side.

TF Lite

Training happens on big machines with GPUs, but inference often happens on phones, embedded devices, and microcontrollers. TensorFlow Lite (TF Lite) is a lightweight runtime designed for on-device inference. It takes a trained TensorFlow model, converts it to a compact .tflite format with optimizations like quantization, and runs it with a small, fast interpreter.

The conversion process uses the tf.lite.TFLiteConverter. You point it at a SavedModel or a Keras model, optionally apply optimizations, and get a flatbuffer file that is typically 2-10x smaller than the original.

# Convert a Keras model to TF Lite
converter = tf.lite.TFLiteConverter.from_keras_model(model)

# Optional: apply quantization for smaller size and faster inference
converter.optimizations = [tf.lite.Optimize.DEFAULT]

# Optional: full integer quantization (requires representative dataset)
def representative_dataset():
    for i in range(100):
        yield [x_train[i:i+1].astype("float32")]

converter.representative_dataset = representative_dataset
converter.target_spec.supported_types = [tf.float16]

# Convert and save
tflite_model = converter.convert()
with open("model.tflite", "wb") as f:
    f.write(tflite_model)

print(f"TF Lite model size: {len(tflite_model) / 1024:.1f} KB")

# Run inference with the TF Lite interpreter
interpreter = tf.lite.Interpreter(model_path="model.tflite")
interpreter.allocate_tensors()

input_details = interpreter.get_input_details()
output_details = interpreter.get_output_details()

# Feed a single image
test_image = x_test[0:1].astype("float32")
interpreter.set_tensor(input_details[0]["index"], test_image)
interpreter.invoke()

prediction = interpreter.get_tensor(output_details[0]["index"])
print(f"Predicted digit: {prediction.argmax()}")

Quantization is the key optimization in TF Lite. The default optimization converts 32-bit floating point weights to 8-bit integers, which roughly quarters the model size and speeds up inference on CPUs that have optimized integer arithmetic paths. Full integer quantization goes further by also quantizing the activations, but it requires a representative dataset so the converter can calibrate the quantization ranges. There is always a small accuracy tradeoff, but for most models it is negligible.

💡 Key Insight

TF Lite is not just for phones. It runs on Raspberry Pi, NVIDIA Jetson, Coral Edge TPU, and even bare-metal microcontrollers (via TF Lite Micro). If your model needs to run anywhere without a full TensorFlow installation, TF Lite is your answer. The tradeoff is that not all TensorFlow operations are supported — check the compatibility guide before designing your architecture.

TF Serving

On the other end of the deployment spectrum is TensorFlow Serving, a production-grade serving system designed for high-throughput, low-latency inference on servers. While TF Lite handles mobile and edge, TF Serving handles cloud and data center deployments. It loads SavedModel directories, exposes them over gRPC and REST APIs, handles model versioning, and supports batching requests for GPU efficiency.

The typical workflow is: train your model, export it as a SavedModel with a version number, and point TF Serving at the model directory. TF Serving automatically picks up new versions when you drop them into the directory, enabling zero-downtime model updates.

# Export model with version directory structure
import os

MODEL_DIR = "serving_models/mnist_classifier"
VERSION = 1
export_path = os.path.join(MODEL_DIR, str(VERSION))

model.save(export_path)
# Creates: serving_models/mnist_classifier/1/saved_model.pb + variables/

# Start TF Serving with Docker (run in terminal):
# docker pull tensorflow/serving
# docker run -p 8501:8501 \
#   --mount type=bind,source=$(pwd)/serving_models/mnist_classifier,target=/models/mnist \
#   -e MODEL_NAME=mnist \
#   tensorflow/serving

# Query the model via REST API
import json
import requests

data = json.dumps({
    "signature_name": "serving_default",
    "instances": x_test[:3].tolist()
})

response = requests.post(
    "http://localhost:8501/v1/models/mnist:predict",
    data=data,
    headers={"content-type": "application/json"}
)

predictions = response.json()["predictions"]
for i, pred in enumerate(predictions):
    print(f"Sample {i}: predicted digit {max(range(10), key=lambda j: pred[j])}")

TF Serving supports request batching out of the box. When multiple inference requests arrive within a short time window, it groups them into a single batch and runs them through the model together. This dramatically improves GPU utilization because GPUs are most efficient when processing large batches. You configure the batching window and maximum batch size in a batching configuration file.

For model versioning, TF Serving has a concept called model version policy. By default, it serves the latest version. You can also configure it to serve multiple versions simultaneously (useful for A/B testing) or to serve a specific version. When you deploy a new version, TF Serving loads it into memory while continuing to serve the old one, then atomically swaps them — no requests are dropped.

💡 Key Insight

TF Serving's gRPC interface is significantly faster than REST for production workloads — roughly 2-3x lower latency because it uses protocol buffers for serialization instead of JSON. Use REST for debugging and quick tests; switch to gRPC for production traffic. The Python tensorflow-serving-api package provides a convenient gRPC client.

Putting It All Together

TensorFlow and Keras give you a complete stack. Start prototyping with the Sequential API. When your architecture outgrows a linear pipeline, move to the Functional API. Feed your data through tf.data pipelines so your GPU never starves. Use callbacks to automate checkpointing, early stopping, and learning rate schedules. When you need full control for research, drop down to GradientTape. Visualize everything in TensorBoard. Save your model in SavedModel format. Deploy to phones with TF Lite and to servers with TF Serving.

The honest tradeoff compared to PyTorch: TensorFlow's deployment story is unmatched. The path from training to production is smoother and more mature, especially for mobile and embedded. But PyTorch often feels more Pythonic and is dominant in research. Many teams use both — PyTorch for research and prototyping, TensorFlow for production deployment. The good news is that the concepts transfer directly. Once you understand backpropagation, computation graphs, and data pipelines in one framework, the other is just syntax.

⚠️ Warning

TensorFlow's API has changed significantly across versions. Code written for TF 1.x (with tf.Session, tf.placeholder) will not work in TF 2.x without modification. Always check that tutorials and Stack Overflow answers target your version. If you see tf.compat.v1 in code, that is TF 1.x compatibility mode — avoid it in new projects.
