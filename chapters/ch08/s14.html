Hardware for Deep Learning

Chapter 8: Training Deep Networks Section 14 of 16 7 subtopics
TL;DR

Deep learning is fundamentally a hardware story. Neural networks are massive matrix multiplications, and the hardware that does matrix math fastest wins. GPUs dominate because they have thousands of small cores that multiply matrices in parallel. Tensor Cores on modern NVIDIA GPUs do mixed-precision matrix multiply in hardware, delivering 10–30x more throughput than standard CUDA cores. TPUs take this further with systolic arrays — fixed-function silicon designed exclusively for matrix operations. This section explains GPU and TPU architectures from first principles, compares every major chip generation, and helps you pick the right hardware for your budget and workload.

Why Hardware Matters for Deep Learning

Training a neural network means doing one thing billions of times: multiplying matrices. A single forward pass through a transformer with 1 billion parameters involves roughly 2 billion floating-point multiply-add operations per token. Multiply that by thousands of tokens per batch, thousands of batches per epoch, and hundreds of epochs. The total is staggering — training GPT-3 required approximately 3.14 × 10²³ floating-point operations.

A modern CPU can do maybe 1 teraFLOP (10¹² operations per second) of floating-point math. At that rate, GPT-3 training would take about 10,000 years. An NVIDIA H100 GPU does 989 teraFLOPs in half-precision. Suddenly that same computation takes weeks instead of millennia. Hardware isn't just a detail — it's the difference between "possible" and "impossible."

Understanding hardware also explains otherwise mysterious advice. Why does mixed-precision training help? Because Tensor Cores are physically faster at float16 math. Why does batch size affect training speed non-linearly? Because of the GPU's memory hierarchy. Why is data loading a bottleneck? Because the CPU-to-GPU data transfer has limited bandwidth. Once you understand the hardware, the software advice makes sense.

GPU Architecture Fundamentals

The CPU vs. GPU Design Philosophy

A CPU is designed to do one thing really fast. It has a few powerful cores (8–64 on modern chips), each with sophisticated branch prediction, out-of-order execution, and large caches. It's optimized for complex, branching control flow — running operating systems, databases, web servers.

A GPU is designed to do one thing massively in parallel. It has thousands of simpler cores organized into groups. Each core is slower than a CPU core, but there are so many of them that the total throughput for parallel work is orders of magnitude higher. Deep learning is embarrassingly parallel — matrix multiplication decomposes into independent multiply-add operations — so GPUs dominate.

NVIDIA GPU Architecture: SMs and CUDA Cores

An NVIDIA GPU is organized into Streaming Multiprocessors (SMs). Think of an SM as a mini-processor containing a group of CUDA cores that share control logic and memory. The H100 GPU has 132 SMs. Each SM contains 128 CUDA cores — simple arithmetic units that each perform one floating-point multiply-add per clock cycle. That's 132 × 128 = 16,896 CUDA cores total.

CUDA cores handle general-purpose floating-point math: addition, multiplication, fused multiply-add (FMA). For float32 operations, the H100 delivers about 67 teraFLOPs from its CUDA cores alone. That's already 50x faster than a CPU for matrix math. But the real acceleration comes from a different unit.

Tensor Cores: The Deep Learning Accelerator

Tensor Cores are specialized hardware units designed for one specific operation: small matrix multiplications. While a CUDA core computes one multiply-add per cycle, a Tensor Core computes an entire 4×4 matrix multiply-accumulate in a single cycle. That's 64 multiply-add operations at once.

Here's the key insight: Tensor Cores operate on lower-precision data types. Instead of float32, they consume float16 (or bfloat16) inputs and accumulate in float32. This is the hardware foundation of mixed-precision training. Your weights and activations are stored in float16 (half the memory), Tensor Cores process them at extreme speed, and the accumulation in float32 preserves numerical accuracy.

The throughput difference is dramatic. The H100 delivers 67 teraFLOPs in float32 on CUDA cores, but 989 teraFLOPs in float16 on Tensor Cores — nearly 15x more throughput. Every modern training run uses mixed-precision specifically to exploit this hardware advantage.

GPU Memory Hierarchy

Understanding the memory hierarchy explains why some operations are fast and others are slow.

Registers are the fastest memory — directly inside each SM, accessed in a single cycle. Each SM has 256 KB of register file. But there's very little of it, and it's private to each thread.

Shared Memory / L1 Cache is the next level. Each SM has a configurable shared memory pool (up to 228 KB on H100) that all threads within an SM can access. This is about 10x slower than registers but critical for reducing redundant memory reads. When you see "kernel fusion" in deep learning — combining multiple operations into one GPU kernel — the main benefit is keeping intermediate results in shared memory instead of writing them back to slow global memory.

L2 Cache sits between the SMs and main memory. The H100 has 50 MB of L2 cache shared across all SMs. It catches frequent memory accesses and reduces traffic to the much slower main memory.

HBM (High Bandwidth Memory) is the main GPU memory — what you see as "VRAM" in nvidia-smi. The H100 has 80 GB of HBM3 with 3.35 TB/s bandwidth. This is where your model weights, activations, gradients, and optimizer states live. Despite the impressive bandwidth number, HBM is 10–100x slower than shared memory, and most deep learning workloads are bottlenecked by how fast they can move data from HBM to the compute units.

This hierarchy is why the roofline model matters. Every operation is either compute-bound (limited by FLOP throughput) or memory-bound (limited by memory bandwidth). Matrix multiplications with large dimensions are compute-bound — the compute units are the bottleneck. Element-wise operations (activation functions, normalization) are memory-bound — reading and writing data takes longer than the actual arithmetic. Understanding which regime your operation falls in tells you where to optimize.

NVIDIA GPU Generations for Deep Learning

V100 (Volta, 2017)

The V100 was the first GPU with Tensor Cores, and it defined the era. It has 32 GB (later 32/16 GB variants) of HBM2 with 900 GB/s bandwidth, 5,120 CUDA cores, and 640 first-generation Tensor Cores delivering 125 TFLOPS in FP16. The V100 dominated deep learning from 2017 to 2020. Many landmark models — BERT, GPT-2, early ResNet training — were trained on V100 clusters. You still find V100s in cloud instances at steep discounts.

A100 (Ampere, 2020)

The A100 brought three key improvements. First, 80 GB of HBM2e with 2 TB/s bandwidth — enough to fit much larger models in a single GPU. Second, third-generation Tensor Cores supporting TF32 (a format with float32 range but reduced mantissa) and BF16, giving 312 TFLOPS in FP16. Third, Multi-Instance GPU (MIG) partitioning — you can split one A100 into up to seven independent GPU instances, each with isolated memory and compute. This was a game-changer for inference serving and multi-tenant clusters.

The A100 also introduced sparsity acceleration. If a matrix has structured 2:4 sparsity (at least 2 of every 4 values are zero), the Tensor Cores can skip the zero multiplications, effectively doubling throughput for sparse models.

H100 (Hopper, 2022)

The H100 is the current workhorse for large-scale training. Key advances: fourth-generation Tensor Cores with FP8 support (989 TFLOPS in FP16, roughly 2,000 TFLOPS in FP8), 80 GB HBM3 with 3.35 TB/s bandwidth, and a Transformer Engine that dynamically switches between FP8 and FP16 precision per layer, per iteration, to maximize throughput while preserving accuracy.

The H100 also introduced NVLink 4.0 with 900 GB/s bidirectional bandwidth between GPUs — 7x more than PCIe 5.0. For multi-GPU training, this matters enormously. Gradient synchronization, tensor parallelism, and pipeline parallelism all require moving large amounts of data between GPUs. The faster that interconnect, the better multi-GPU training scales.

B200 (Blackwell, 2024)

The B200 is NVIDIA's latest architecture. The headliner is fifth-generation Tensor Cores with FP4 support, delivering roughly 2,250 TFLOPS in FP16 and up to 4,500 TFLOPS in FP8 — more than double the H100. It comes with 192 GB of HBM3e at 8 TB/s bandwidth, and a second-generation Transformer Engine with improved dynamic precision scaling.

The Blackwell architecture also doubles the NVLink bandwidth to 1.8 TB/s per GPU, and introduces NVLink Switch for scaling to thousands of GPUs with full bandwidth. At the system level, NVIDIA's GB200 NVL72 rack contains 72 Blackwell GPUs fully connected via NVLink, delivering over 1.4 exaFLOPS of FP8 compute in a single rack.

TPU Architecture

What Is a TPU?

A Tensor Processing Unit (TPU) is Google's custom-designed ASIC (application-specific integrated circuit) built exclusively for neural network computation. Unlike GPUs, which evolved from graphics rendering and remain general-purpose parallel processors, TPUs are designed from the ground up for matrix multiplication and nothing else.

Systolic Arrays: The Core of a TPU

The fundamental compute unit in a TPU is a systolic array — a grid of multiply-accumulate (MAC) units arranged in a 2D mesh. In a 128×128 systolic array, data flows through the grid in a wave-like pattern. One matrix enters from the left, the other from the top, and partial products ripple through the array. By the time data exits the bottom-right corner, the full matrix multiplication is complete.

The beauty of systolic arrays is simplicity and efficiency. Each MAC unit does one multiply-add and passes the result to its neighbor. There's no complex scheduling, no cache hierarchy to manage — data just flows. This means nearly 100% of the silicon is doing useful math, compared to GPUs where a significant fraction of die area goes to caches, schedulers, and control logic.

The tradeoff is flexibility. A GPU can run arbitrary parallel algorithms. A systolic array is hardwired for matrix multiplication. If your workload is anything other than dense linear algebra, a TPU doesn't help. But since neural networks are ~90% matrix multiplication, this tradeoff works well in practice.

TPU Pods and XLA Compilation

A TPU Pod is a cluster of TPU chips connected via Google's custom high-bandwidth interconnect (ICI). A v4 Pod contains up to 4,096 chips connected in a 3D torus topology, delivering over 1.1 exaFLOPS of aggregate BF16 compute. The 3D torus means every chip has direct, low-latency connections to its six nearest neighbors — this topology is excellent for the all-reduce communication patterns used in data-parallel training.

TPUs run code compiled through XLA (Accelerated Linear Algebra), a domain-specific compiler that takes your JAX or TensorFlow computation graph, optimizes it (operation fusion, memory layout, communication scheduling), and generates TPU machine code. XLA can aggressively fuse operations — combining a matmul, bias add, and activation into a single kernel — because it sees the entire computation graph at compile time. The downside is compilation overhead: the first run is slow while XLA compiles, and dynamic shapes require recompilation.

AMD and Intel Alternatives

AMD Instinct GPUs

AMD's Instinct MI300X is their flagship competitor. It's a chiplet-based design with 192 GB of HBM3 (matching the B200 in capacity), 5.3 TB/s memory bandwidth, and 1,307 TFLOPS of FP16 compute. The software stack is ROCm (the AMD equivalent of CUDA), which has matured significantly but still has gaps in library support and debugging tools.

The MI300X's main selling point is the massive memory — 192 GB in a single GPU is enough to hold a 70B parameter model in FP16 without any model parallelism. For inference workloads where memory capacity is the bottleneck (serving large language models), the MI300X is genuinely competitive. For training, the CUDA ecosystem advantage still gives NVIDIA the edge in most setups, though ROCm support in PyTorch continues to improve.

Intel Gaudi and Xe GPUs

Intel has entered the AI accelerator market with Gaudi (through its Habana Labs acquisition) and the Xe GPU series. Gaudi 2 offers competitive inference performance and was adopted by some cloud providers. The Gaudi 3, released in 2024, promises significant training capability. However, Intel's AI accelerators have limited ecosystem support compared to NVIDIA and even AMD. The developer tools and library coverage are still catching up.

CPU vs. GPU vs. TPU: When to Use What

The choice isn't always "use the most expensive GPU." Each hardware type has a sweet spot:

CPU is the right choice for data preprocessing, feature engineering, small models (random forests, linear models, small MLPs), and prototyping. If your model trains in minutes on a CPU, adding a GPU just complicates the setup without meaningful speedup. CPUs also handle irregular, branching logic well — something GPUs and TPUs struggle with.

GPU is the default choice for deep learning. It handles everything from small CNNs to large language models. The CUDA ecosystem is mature, with optimized libraries for every common operation (cuBLAS for linear algebra, cuDNN for convolutions, NCCL for multi-GPU communication). If you're training any neural network that takes more than a few minutes on a CPU, a GPU will typically be 10–100x faster.

TPU excels at large-scale training with regular computation patterns — transformers, large CNNs, anything that's predominantly matrix multiplication. TPUs are particularly cost-effective on Google Cloud, where they're often cheaper per FLOP than GPUs. The JAX/XLA ecosystem is the natural fit. The limitations: TPUs don't support arbitrary CUDA code, debugging is harder, and custom operations may not be available.

Memory Bandwidth: The Real Bottleneck

Most people focus on FLOPS when comparing hardware. But for many deep learning operations, memory bandwidth is the actual bottleneck.

Consider the arithmetic intensity of different operations. A large matrix multiplication (say, 4096×4096 × 4096×4096) does O(n³) arithmetic on O(n²) data — it's very compute-dense, and the compute units are the bottleneck. But an element-wise operation like ReLU(x) does O(n) arithmetic on O(n) data — one operation per data element. The time is dominated by reading and writing the data, not the computation itself.

In a transformer model, the attention computation is memory-bound for small batch sizes (reading keys and values dominates), while the feed-forward layers are compute-bound (large matrix multiplications). This is why FlashAttention — which reorganizes the attention computation to minimize HBM reads/writes — delivers such dramatic speedups. It doesn't reduce the total arithmetic; it reduces the memory traffic.

The roofline model formalizes this. Plot your operation's arithmetic intensity (FLOPs per byte of memory accessed) against the hardware's compute ceiling (peak TFLOPS) and memory ceiling (peak bandwidth × arithmetic intensity). If your operation falls below the memory ceiling line, you're memory-bound — buying a GPU with more TFLOPS won't help. You need more memory bandwidth, or you need to restructure the computation to do more work per byte loaded.

Training Memory Analysis

A common surprise: training a 1 billion parameter model doesn't take 4 GB of memory (1B × 4 bytes per float32). It takes roughly 20 GB. Here's the breakdown:

Parameters: 4 bytes per parameter in float32, or 2 bytes in float16/bfloat16. For 1B parameters: 4 GB (FP32) or 2 GB (FP16).

Gradients: Same size as parameters. During backpropagation, you store one gradient per parameter. Another 4 GB (FP32) or 2 GB (FP16).

Optimizer states: This is the big one. Adam stores two additional values per parameter — the first moment (mean of gradients) and second moment (mean of squared gradients). Both are typically kept in float32 for stability. That's 2 × 4 bytes × 1B = 8 GB. With mixed-precision training using the Adam optimizer, you also keep a float32 master copy of the weights. Total optimizer overhead: 12 GB for 1B parameters.

Activations: During the forward pass, intermediate outputs are saved for the backward pass. The size depends on batch size, sequence length, and model architecture. For a transformer, activations scale as O(batch × seq_len × hidden_dim × num_layers). With a batch size of 32 and sequence length of 2,048 on a 1B parameter transformer, activations can easily consume 10–30 GB.

# Memory estimation for a transformer model
def estimate_training_memory(
    num_params_billions: float,
    batch_size: int,
    seq_len: int,
    hidden_dim: int,
    num_layers: int,
    mixed_precision: bool = True
):
    """Estimate GPU memory for training (in GB)."""
    P = num_params_billions * 1e9
    
    if mixed_precision:
        params_gb = P * 2 / 1e9        # FP16 weights
        grads_gb = P * 2 / 1e9         # FP16 gradients
        optimizer_gb = P * 12 / 1e9    # FP32 master + 2x FP32 moments
    else:
        params_gb = P * 4 / 1e9
        grads_gb = P * 4 / 1e9
        optimizer_gb = P * 8 / 1e9     # 2x FP32 moments
    
    # Rough activation estimate for transformer
    bytes_per_elem = 2 if mixed_precision else 4
    act_per_layer = batch_size * seq_len * hidden_dim * bytes_per_elem
    # Each layer saves ~10-12 tensors for backward
    activation_gb = num_layers * 12 * act_per_layer / 1e9
    
    total = params_gb + grads_gb + optimizer_gb + activation_gb
    
    print(f"Parameters:       {params_gb:.1f} GB")
    print(f"Gradients:        {grads_gb:.1f} GB")
    print(f"Optimizer states: {optimizer_gb:.1f} GB")
    print(f"Activations:      {activation_gb:.1f} GB")
    print(f"Total:            {total:.1f} GB")
    return total

# Example: 7B parameter model
estimate_training_memory(
    num_params_billions=7,
    batch_size=4, seq_len=2048,
    hidden_dim=4096, num_layers=32,
    mixed_precision=True
)

For models above 10 billion, you're in cluster territory — hundreds to thousands of GPUs, and hardware interconnect bandwidth becomes as important as raw compute.

GPU Profiling and Optimization

Identifying Bottlenecks

Before optimizing, measure. The three common bottlenecks in a training pipeline are:

Data loading. If your GPU utilization is low (under 80%) and you see periodic gaps in the GPU timeline, data loading is the bottleneck. The GPU is idle waiting for the next batch. Fixes: increase num_workers in your DataLoader, use pin_memory=True for faster CPU-to-GPU transfers, pre-process data into a fast-access format like WebDataset or LMDB.

Compute. If GPU utilization is consistently high (95%+) and you've already enabled mixed precision, your training is compute-bound. This means you're using the hardware efficiently — the only way to go faster is a faster GPU or more GPUs.

Communication. In multi-GPU training, gradient synchronization can become the bottleneck. If you see gaps in the compute timeline where all GPUs are waiting for an all-reduce to complete, communication is the limiting factor. Fixes: use NVLink instead of PCIe, overlap communication with computation (gradient bucketing in DDP), or switch to model parallelism to reduce the gradient sync volume.

Practical Optimization Checklist

  1. Enable mixed precision. Use torch.amp.autocast and GradScaler. This alone often gives a 2–3x speedup on Tensor Core GPUs.
  2. Use torch.compile. PyTorch 2.0's compiler fuses operations and generates optimized CUDA kernels automatically. One line of code: model = torch.compile(model).
  3. Pin memory and increase workers. Set pin_memory=True and num_workers=4+ in DataLoader.
  4. Use CUDA graphs for fixed-shape workloads. CUDA graphs record a sequence of GPU operations and replay them with near-zero CPU overhead. Great for inference; trickier for training due to dynamic shapes.
  5. Profile before optimizing. Never guess where the bottleneck is. Use the PyTorch profiler or NVIDIA Nsight Systems to measure.
