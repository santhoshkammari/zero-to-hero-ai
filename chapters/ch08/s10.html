Distributed Training

Chapter 8: Training Deep Networks 5 subtopics
TL;DR

Modern models are too large and datasets too massive to train on a single GPU. Distributed training splits the work across multiple devices. Data parallelism replicates the model on every GPU and splits the data — gradients are averaged via AllReduce after each step. Model parallelism splits the model itself when it won't fit in one GPU's memory. Pipeline parallelism chains model stages across GPUs like a factory assembly line, using micro-batches to keep all devices busy. The real challenge isn't the math — it's the communication overhead. Ring-AllReduce, gradient compression, and computation-communication overlap are the engineering tricks that make distributed training practical at scale.

Why One GPU Isn't Enough

Let's do some arithmetic. GPT-3 has 175 billion parameters. Each parameter is a 32-bit float — that's 4 bytes. Just storing the parameters takes 700 GB. Add optimizer states (Adam stores two additional values per parameter): another 1.4 TB. Add gradients: another 700 GB. You need roughly 2.8 TB of memory just to train this model, before counting activations. The largest single GPU today has 80–192 GB of memory. You need dozens to hundreds of GPUs working together.

Even for models that fit on one GPU, training time is the bottleneck. Training BERT on one V100 takes roughly 12 days. On 64 V100s? About 4.5 hours. If your research cycle requires experimenting with 50 hyperparameter configurations, the difference between 12 days and 4.5 hours per run is the difference between publishing this year and publishing never.

Distributed training is not optional for modern AI. It's the engineering foundation that makes large-scale learning possible.

Data Parallelism

The Core Idea: Replicate the Model, Split the Data

Data parallelism is the simplest and most widely used distribution strategy. Every GPU holds a complete copy of the model. A large batch of training data is split into mini-batches, one per GPU. Each GPU runs the forward and backward pass on its own mini-batch independently. Then all GPUs share their gradients, average them, and apply the same update. After the update, all model copies are identical again.

# Conceptual data-parallel training step
# N GPUs, each with a full model copy

for each GPU i in parallel:
    local_batch = global_batch[i * chunk : (i+1) * chunk]
    local_loss = forward(model, local_batch)
    local_grads = backward(local_loss)

# Synchronize: average gradients across all GPUs
avg_grads = AllReduce(local_grads) / N

for each GPU i in parallel:
    model[i].params -= lr * avg_grads

The key insight: if every GPU starts with the same weights and applies the same averaged gradient, they stay synchronized without ever needing to share weights — only gradients. This is why data parallelism scales well. The forward and backward passes are embarrassingly parallel. The only communication happens during the gradient averaging step.

Synchronous vs Asynchronous

In synchronous data parallelism, all GPUs finish their backward pass, exchange gradients, and update together. No GPU moves to the next batch until everyone is done. This gives mathematically equivalent results to training with a larger batch on a single GPU. The downside: the slowest GPU (the straggler) holds everyone back.

In asynchronous data parallelism, each GPU updates a shared parameter server as soon as it finishes its own gradient, without waiting for others. This eliminates the straggler problem but introduces stale gradients — by the time GPU 3's gradient arrives, the model has already been updated by GPUs 1 and 2. The gradient GPU 3 computed no longer matches the current parameters. Staleness causes instability and often requires lower learning rates, partially negating the speedup.

In practice, synchronous training dominates. The straggler problem is managed by using homogeneous hardware and careful batch sizing. Asynchronous approaches saw interest around 2015–2017 (Google's DistBelief, early TensorFlow) but have largely fallen out of favor for large-scale training.

Communication Primitives

The gradient averaging step — "everyone shares their gradients and everyone gets the average" — is implemented using collective communication primitives. Understanding these is essential because communication is the bottleneck that determines whether adding more GPUs actually speeds up training.

AllReduce

AllReduce is the workhorse operation. It takes an array from each of N devices, applies a reduction (usually sum or average), and distributes the result back to every device. After AllReduce, every GPU has the identical averaged gradient.

The naive implementation: every GPU sends its gradient to one central GPU, that GPU averages everything, and broadcasts the result back. This is terrible. The central GPU becomes a bandwidth bottleneck, and you're sending 2 × (N−1) × gradient_size bytes through a single link.

Ring-AllReduce

The standard solution is Ring-AllReduce. Arrange the N GPUs in a logical ring. Split each gradient into N chunks. In the first phase (reduce-scatter), each GPU sends one chunk to its right neighbor and receives one from its left, accumulating partial sums. After N−1 steps, each GPU holds the fully reduced version of one chunk. In the second phase (all-gather), each GPU sends its complete chunk around the ring until everyone has all chunks.

# Ring-AllReduce in two phases
# N GPUs in a ring, gradient split into N chunks

# Phase 1: Reduce-Scatter (N-1 steps)
# After this, GPU i holds the fully summed chunk i
for step in range(N - 1):
    send chunk[(my_id - step) % N] to right neighbor
    recv chunk from left neighbor, accumulate (sum)

# Phase 2: All-Gather (N-1 steps)
# After this, every GPU has all N fully-reduced chunks
for step in range(N - 1):
    send my completed chunk to right neighbor
    recv completed chunk from left neighbor

The critical property: each GPU sends and receives exactly 2 × (N−1)/N × gradient_size bytes total. As N grows, this approaches 2 × gradient_size per GPU — the communication per GPU is nearly constant regardless of how many GPUs you add. Compare this to the naive approach where the central GPU must handle N times the data. Ring-AllReduce is what makes data parallelism scalable to hundreds of GPUs.

In practice, NVIDIA's NCCL (NVIDIA Collective Communications Library) implements Ring-AllReduce and other collectives optimized for specific GPU interconnect topologies. On systems with NVLink (high-bandwidth GPU-to-GPU connections), NCCL may use tree-based or hierarchical reduction that exploits the topology better than a simple ring.

Other Collectives

Beyond AllReduce, several other operations appear in distributed training:

All-Gather: every GPU has a piece; after the operation, every GPU has all pieces concatenated. Used when you need to reconstruct a full tensor that was sharded across devices.

Reduce-Scatter: every GPU has a full copy; after the operation, each GPU has the reduced (summed) version of one shard. This is the first half of AllReduce.

Broadcast: one GPU sends data to all others. Used during initialization to distribute the initial model weights.

All-to-All: each GPU sends a different piece to each other GPU. Used in expert parallelism (Mixture of Experts models) where different tokens get routed to different expert GPUs.

Communication Optimization

Three key techniques reduce communication overhead:

Gradient compression: instead of sending full 32-bit gradients, compress them. Simple approaches: send only the top-k largest gradient values, or quantize to 16-bit or even 1-bit. Research (like 1-bit Adam from DeepSpeed) shows that aggressive compression can work with error feedback — track the compression error and add it to the next round's gradient.

Computation-communication overlap: don't wait until the entire backward pass finishes to start communicating. As soon as a layer's gradient is computed, start the AllReduce for that layer while later layers are still computing their gradients. This hides communication latency behind computation. PyTorch DDP does this automatically by registering backward hooks that trigger AllReduce as gradients become ready.

Bucket fusion: instead of launching a separate AllReduce for every parameter tensor (which adds per-operation overhead), group small tensors into large buckets (typically 25 MB) and AllReduce the entire bucket at once. Fewer, larger messages use the network bandwidth more efficiently.

# PyTorch DDP — bucket fusion and overlap happen automatically
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP

# Initialize process group
dist.init_process_group(backend='nccl')

# Wrap your model — DDP handles gradient sync
model = DDP(model.to(device), device_ids=[local_rank])

# Training loop looks almost identical to single-GPU
for batch in dataloader:
    loss = model(batch)
    loss.backward()      # gradients are AllReduced here
    optimizer.step()
    optimizer.zero_grad()

Scaling Challenges

Doubling GPUs doesn't halve training time. The linear scaling rule says: if you multiply the batch size by k (because you have k GPUs), multiply the learning rate by k too. This keeps the expected gradient update magnitude approximately constant. It works well up to a point — typically batch sizes of 8K–32K for language models, 8K–64K for vision models — after which training becomes unstable regardless of learning rate. The square root scaling rule (multiply lr by √k) is more conservative and sometimes works better at extreme scales.

The straggler problem means your effective throughput is limited by the slowest GPU in each step. Causes include: thermal throttling on individual GPUs, network congestion on shared switches, OS jitter from background processes, and imbalanced data loading. Mitigation strategies include homogeneous hardware, proper cooling, dedicated networks, and sometimes dropping the slowest worker's gradient entirely (bounded staleness).

Model Parallelism

When a model doesn't fit in one GPU's memory, you split the model itself across devices. This is model parallelism, and it comes in two major flavors.

Tensor Parallelism

Tensor parallelism splits individual layers across GPUs. A matrix multiplication Y = XW can be split by dividing W column-wise across GPUs: GPU 0 computes XW₁, GPU 1 computes XW₂, etc. The results are concatenated (or reduced) to form the full output. Megatron-LM popularized this approach for transformer attention and feed-forward layers.

The advantage: it works within a single transformer layer, so latency per layer stays low. The disadvantage: it requires high-bandwidth interconnects (like NVLink) because GPUs must communicate at every layer boundary. You typically use tensor parallelism within a node (where NVLink provides 600+ GB/s) and other parallelism strategies between nodes.

Pipeline Parallelism

Pipeline parallelism assigns different layers to different GPUs. GPU 0 runs layers 1–10, GPU 1 runs layers 11–20, and so on. When GPU 0 finishes processing a batch through its layers, it sends the activations to GPU 1 and starts working on the next batch.

The naive approach has a fatal flaw: the pipeline bubble. While GPU 0 processes the first batch, GPUs 1–3 are idle. While GPU 3 processes the first batch's backward pass, GPUs 0–2 are idle. Most of the time, most of the GPUs are waiting.

# Pipeline bubble illustration (4 GPUs, 1 micro-batch)
# F = forward, B = backward, _ = idle
# Time →
# GPU 0: F _ _ _ B _ _ _
# GPU 1: _ F _ _ _ B _ _
# GPU 2: _ _ F _ _ _ B _
# GPU 3: _ _ _ F _ _ _ B
# Massive idle time!

# GPipe fix: split batch into 4 micro-batches (m1-m4)
# GPU 0: F1 F2 F3 F4 B4 B3 B2 B1
# GPU 1: __ F1 F2 F3 F4 B4 B3 B2
# GPU 2: __ __ F1 F2 F3 F4 B4 B3
# GPU 3: __ __ __ F1 F2 F3 F4 B4
# Much less idle time — the bubble shrinks as micro-batches increase

The solution, introduced by GPipe, is micro-batching. Split each batch into many micro-batches and pipeline them through the stages. While GPU 0 processes micro-batch 2, GPU 1 processes micro-batch 1. The pipeline bubble shrinks as you use more micro-batches. With M micro-batches and P pipeline stages, the bubble fraction is approximately (P−1)/(M+P−1). With M=32 and P=4, only about 9% of time is wasted in the bubble.

PipeDream (Microsoft) and 1F1B scheduling (one forward, one backward) further optimize pipeline schedules by interleaving forward and backward passes to reduce peak memory usage. Instead of doing all forward passes first and then all backward passes, 1F1B alternates them, keeping fewer activations in memory at any time.

Expert Parallelism

Expert parallelism is specific to Mixture of Experts (MoE) models. Different expert sub-networks live on different GPUs. A routing function decides which tokens go to which experts. This requires All-to-All communication — each GPU sends its tokens to the appropriate expert GPU and receives back the results. MoE models like Mixtral and Switch Transformer use this to scale parameter count without proportionally scaling computation.

Combining Parallelism Strategies

Large-scale training doesn't use just one strategy. A system training a 175B-parameter model might use all three simultaneously:

Tensor parallelism within a node: split each layer across 8 GPUs connected by NVLink (high bandwidth, low latency).

Pipeline parallelism across nodes: assign different layer groups to different 8-GPU nodes, connected by InfiniBand (lower bandwidth, higher latency — but pipeline parallelism sends less data per step).

Data parallelism across pipeline replicas: replicate the entire pipeline multiple times, each processing a different data shard, with gradient AllReduce across replicas.

# 3D parallelism: TP × PP × DP
# Example: 64 GPUs = 8 TP × 4 PP × 2 DP
#
# Node 0 (8 GPUs): Layers 1-10, tensor-parallel, Data shard 0
# Node 1 (8 GPUs): Layers 11-20, tensor-parallel, Data shard 0
# Node 2 (8 GPUs): Layers 21-30, tensor-parallel, Data shard 0
# Node 3 (8 GPUs): Layers 31-40, tensor-parallel, Data shard 0
# Node 4-7: Same pipeline, Data shard 1 (data-parallel replica)

This "3D parallelism" approach (tensor × pipeline × data) is how Megatron-LM, DeepSpeed, and other frameworks train the largest models. The art is choosing the degree of each parallelism dimension to minimize total communication overhead given your specific hardware topology — how many GPUs per node, what interconnects exist, and what your model's compute-to-communication ratio looks like.

Fault Tolerance and Elasticity

When you're running on 512 GPUs for two weeks, hardware failures are not hypothetical — they're expected. A GPU might overheat. A network switch might flap. A node might crash. Without fault tolerance, any failure restarts training from the last checkpoint, potentially losing hours of work.

Checkpoint-based recovery is the baseline: periodically save model weights, optimizer states, and the data loader position to persistent storage. After a failure, restart from the last checkpoint. The tradeoff: checkpoints are expensive (saving 700 GB takes time and storage), so you don't checkpoint every step. Typical intervals are every 15–60 minutes. You accept losing up to one interval of work.

Elastic training (PyTorch Elastic / TorchElastic) goes further: training can continue with fewer workers after a failure and scale back up when the worker recovers. The process group reconfigures itself dynamically, redistributing data shards across remaining workers. This is especially valuable in cloud environments where spot/preemptible instances can be reclaimed at any time.

The fundamental law of distributed training is that communication overhead grows with GPU count while computation per GPU stays constant (or shrinks). Every technique in this section — Ring-AllReduce, gradient compression, computation-communication overlap, careful parallelism dimension selection — is an attempt to keep communication from dominating. The single most important metric is scaling efficiency: what fraction of linear speedup do you actually get? Good systems achieve 85–95% scaling efficiency on well-connected clusters. Drop to 50% and you're wasting half your GPU budget on waiting.
