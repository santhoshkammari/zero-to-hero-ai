Modern Training Infrastructure

Chapter 8: Training Deep Networks 16 subtopics
TL;DR

Training a frontier model is as much an infrastructure challenge as a research one. DeepSpeed ZeRO partitions optimizer states, gradients, and parameters across GPUs — each stage sheds more memory at the cost of more communication. FSDP brings similar sharding natively into PyTorch. Megatron-LM pioneered efficient tensor, pipeline, and sequence parallelism for transformers. Beyond frameworks, modern training requires mixed-precision arithmetic, gradient checkpointing, fault-tolerant checkpointing, high-bandwidth cluster networking, throughput monitoring, multi-node debugging, reproducibility discipline, and even awareness of power, cooling, and carbon costs. This section covers the full stack — from optimizer math to data-center thermals.

DeepSpeed and the ZeRO Optimizer

Here's the memory problem, in numbers. Training a 7B-parameter model in FP32 requires: 28 GB for parameters, 28 GB for gradients, and 56 GB for Adam optimizer states (two 4-byte values per parameter — first and second moment estimates). That's 112 GB just for "model state" — before counting activations, temporary buffers, or the framework overhead. A single 80 GB A100 can't even hold the model state, let alone train it.

DeepSpeed, developed by Microsoft Research, attacks this with ZeRO (Zero Redundancy Optimizer). The key insight: in standard data parallelism, every GPU holds a complete copy of parameters, gradients, and optimizer states. With N GPUs, you're storing N redundant copies of everything. ZeRO eliminates this redundancy progressively through three stages.

ZeRO Stage 1: Optimizer State Partitioning

Instead of every GPU storing all optimizer states, partition them: GPU 0 stores the optimizer states for parameters 0–1B, GPU 1 for parameters 1B–2B, and so on. Each GPU still holds all parameters and gradients (for the forward and backward pass), but optimizer states are sharded. For Adam with a 7B model on 8 GPUs, optimizer memory drops from 56 GB per GPU to 7 GB per GPU. Total memory savings: 8×.

After the backward pass, each GPU reduces the gradients for its partition, updates its shard of the parameters using its shard of the optimizer state, and broadcasts the updated parameters back. Communication is slightly higher than standard data parallelism, but memory savings are dramatic.

ZeRO Stage 2: Gradient Partitioning

Stage 2 adds gradient partitioning on top of Stage 1. Each GPU only stores the gradients for the parameters whose optimizer states it owns. During the backward pass, gradients are reduce-scattered immediately — each GPU ends up with the reduced gradient for its shard only. Memory savings now cover both optimizer states and gradients: for a 7B model on 8 GPUs, per-GPU gradient memory drops from 28 GB to 3.5 GB.

# ZeRO Stage comparison — 7B model, 8 GPUs
# Per-GPU memory for model state (params + grads + optim):
#
# Standard DDP:   28 + 28 + 56 = 112 GB  (impossible on 80GB GPU)
# ZeRO Stage 1:   28 + 28 +  7 =  63 GB  (optimizer sharded)
# ZeRO Stage 2:   28 + 3.5 + 7 = 38.5 GB (+ gradients sharded)
# ZeRO Stage 3:  3.5 + 3.5 + 7 =  14 GB  (everything sharded)

ZeRO Stage 3: Parameter Partitioning

Stage 3 shards the parameters themselves. No GPU holds the full model. During forward and backward passes, parameters are gathered on-the-fly from other GPUs when needed for a given layer, and released immediately after. This means each layer's parameters exist on all GPUs only briefly during their computation, then only the owning shard is retained.

The communication cost increases significantly — each forward and backward pass requires all-gather operations to reconstruct parameters layer by layer. But memory per GPU drops to 1/N of the total model state. With 8 GPUs and a 7B model, each GPU needs only about 14 GB of model state memory. This is what makes it possible to train 13B and 70B models on hardware that seemed too small.

ZeRO-Infinity and Offloading

ZeRO-Offload extends ZeRO by moving optimizer states and optionally gradients to CPU memory, using the much larger (but slower) system RAM. The GPU computes forward and backward passes; optimizer updates happen on the CPU. The overlap between GPU computation and CPU-GPU data transfers hides much of the latency.

ZeRO-Infinity pushes further, offloading to NVMe SSDs. A single machine with 8 GPUs, 1 TB of RAM, and a few NVMe drives can theoretically train a trillion-parameter model. The practical throughput is lower than pure GPU training, but for researchers with limited GPU budgets, it unlocks model sizes that would otherwise be inaccessible.

# DeepSpeed ZeRO Stage 3 config (ds_config.json)
{
    "zero_optimization": {
        "stage": 3,
        "offload_optimizer": {
            "device": "cpu",
            "pin_memory": true
        },
        "offload_param": {
            "device": "cpu",
            "pin_memory": true
        },
        "overlap_comm": true,
        "contiguous_gradients": true,

}

ZeRO Communication Analysis

Each ZeRO stage trades memory for communication. Stage 1 adds no extra communication over standard DDP — the AllReduce volume is the same. Stage 2 replaces AllReduce with reduce-scatter plus point-to-point broadcasts, keeping total communication roughly equal. Stage 3 adds two all-gather operations per layer (one in forward, one in backward) for parameter reconstruction, increasing communication by approximately 1.5× compared to standard DDP. In practice, the overlap between communication and computation (prefetching the next layer's parameters while computing the current layer) hides most of this cost.

Fully Sharded Data Parallel (FSDP)

FSDP is PyTorch's native answer to ZeRO Stage 3. It shards parameters, gradients, and optimizer states across data-parallel workers. Conceptually, it does the same thing as ZeRO-3: during forward and backward passes, parameters are all-gathered just in time, and released after use.

from torch.distributed.fsdp import FullyShardedDataParallel as FSDP
from torch.distributed.fsdp import MixedPrecision

# Wrap model with FSDP
mp_policy = MixedPrecision(
    param_dtype=torch.bfloat16,
    reduce_dtype=torch.bfloat16,
    buffer_dtype=torch.bfloat16,
)
model = FSDP(
    model,
    mixed_precision=mp_policy,
    use_orig_params=True,       # needed for torch.compile
    limit_all_gathers=True,     # prevents OOM from prefetching
)

# Training loop is standard — FSDP handles sharding transparently
for batch in dataloader:
    loss = model(batch)
    loss.backward()
    optimizer.step()
    optimizer.zero_grad()

FSDP vs DeepSpeed: When to Choose Each

Choose FSDP when: you want native PyTorch integration without a third-party dependency; you plan to use torch.compile for additional optimization; your team prefers staying within the PyTorch ecosystem; you need tight integration with PyTorch's other features (DDP, tensor parallelism, DTensor).

Choose DeepSpeed when: you need ZeRO-Infinity (NVMe offloading); you want the built-in profiling, logging, and elastic training features; you need pipeline parallelism out of the box; your model is extremely large and you need every memory optimization DeepSpeed offers; you're already using the Hugging Face Trainer (which has deep DeepSpeed integration).

Performance is comparable for most workloads. The ecosystem integration is the main differentiator. FSDP is converging toward feature parity with DeepSpeed as PyTorch invests more in distributed training primitives.

Megatron-LM

Megatron-LM, developed by NVIDIA, is the blueprint for efficient large transformer training. It introduced optimized implementations of tensor parallelism, pipeline parallelism, and sequence parallelism — specifically designed for the transformer architecture.

Tensor Parallelism in Megatron

Megatron splits the attention and MLP layers of a transformer across GPUs within a node. For the self-attention layer, the Q, K, V projection matrices are column-split across GPUs. Each GPU computes attention for its head shard independently. The output projection is row-split, and an AllReduce synchronizes the result. For the MLP, the first linear layer is column-split and the second is row-split. This pattern requires only two AllReduce operations per transformer block — minimal communication for maximum parallelism.

Sequence Parallelism

Sequence parallelism splits the input sequence across GPUs for operations that don't depend on tensor parallelism (like LayerNorm and dropout). In standard tensor parallelism, these operations are redundantly computed on every GPU. Sequence parallelism replaces this redundancy with a more memory-efficient split. The activation memory for LayerNorm and dropout — which scales linearly with sequence length — is divided among GPUs. This is especially valuable for long-context models where activation memory dominates.

Pipeline Parallelism Schedules

Megatron implements interleaved 1F1B (one-forward-one-backward) scheduling, which assigns multiple non-consecutive stages to each GPU. Instead of GPU 0 owning layers 1–10 and GPU 1 owning layers 11–20, GPU 0 might own layers 1–5 and 21–25. This reduces the pipeline bubble by allowing more micro-batches to be in flight simultaneously, at the cost of slightly more communication.

Mixed Precision Training

Mixed precision training uses lower-precision floating-point formats (FP16 or BF16) for most computations while keeping a master copy of weights in FP32. The benefits are twofold: lower-precision operations are 2–8× faster on modern GPUs (which have dedicated tensor cores for FP16/BF16), and they use half the memory.

The challenge: FP16 has a limited range (max ~65,504) and limited precision (about 3 decimal digits). Gradients can underflow to zero or overflow to infinity. Loss scaling addresses this by multiplying the loss by a large factor (say 1024) before the backward pass, which scales up all gradients. After computing the gradients, you divide by the same factor. If gradients overflow, the scale is reduced; if they don't, the scale is increased. This dynamic loss scaling is what Automatic Mixed Precision (AMP) does automatically.

# PyTorch AMP — automatic mixed precision
from torch.cuda.amp import autocast, GradScaler

scaler = GradScaler()

for batch in dataloader:
    optimizer.zero_grad()
    with autocast(dtype=torch.bfloat16):
        loss = model(batch)
    scaler.scale(loss).backward()
    scaler.step(optimizer)
    scaler.update()

BF16 (bfloat16) has the same exponent range as FP32 but reduced mantissa precision. This means it doesn't suffer from the overflow/underflow problems of FP16, making loss scaling unnecessary. BF16 is the preferred format on A100, H100, and TPU hardware. If your hardware supports BF16, use it — it's strictly simpler than FP16 with loss scaling.

FP8 training is emerging on H100 GPUs and beyond. It uses 8-bit floats for matrix multiplications, doubling throughput over FP16. FP8 requires careful per-tensor scaling and is typically used only for the compute-heavy matmuls while keeping accumulation in higher precision.

Gradient Checkpointing (Activation Recomputation)

Gradient checkpointing (also called activation recomputation) trades compute for memory. During the forward pass, instead of storing all intermediate activations for the backward pass, you store only a few checkpoints. During backprop, when you need an activation that wasn't stored, you recompute it from the nearest checkpoint. This typically reduces activation memory by 5–10× at the cost of one additional forward pass (about 33% more computation).

from torch.utils.checkpoint import checkpoint

class TransformerBlock(nn.Module):
    def forward(self, x):
        # Checkpoint this block — activations are recomputed during backward
        return checkpoint(self._forward, x, use_reentrant=False)

    def _forward(self, x):
        x = x + self.attention(self.norm1(x))
        x = x + self.mlp(self.norm2(x))
        return x

The sweet spot: checkpoint at transformer block boundaries. You store one activation per block instead of all intermediate activations within the block. For a model with 32 blocks, this reduces activation memory by roughly the depth of each block (typically 10–20×) while adding only ~33% compute overhead. Almost every large model training run uses gradient checkpointing.

Checkpoint Strategies for Fault Tolerance

Training for days or weeks means failures are inevitable. Checkpoint strategy is about balancing three costs: the time spent saving checkpoints, the storage space they consume, and the amount of work lost when a failure occurs.

Synchronous checkpointing: pause training, save all state to persistent storage (usually a shared filesystem like Lustre or a cloud object store like S3), resume. The downside: for a model with hundreds of GB of state, saving takes minutes, during which expensive GPUs sit idle.

Asynchronous checkpointing: snapshot state to CPU memory or local SSD in seconds, then flush to persistent storage in the background while training continues. This reduces idle time from minutes to seconds. PyTorch's distributed checkpoint API supports this, as does DeepSpeed.

Incremental checkpointing: save only the parameters that changed since the last checkpoint. Most parameters change by a tiny amount each step, so delta-encoding can compress checkpoints dramatically.

Checkpoint frequency is a risk calculation. If a failure loses 1 hour of training on 512 GPUs, the cost is 512 GPU-hours — potentially thousands of dollars in cloud compute. Most teams checkpoint every 15–30 minutes for large runs. Some save every 100–500 steps.

Cluster Networking and Collective Communications

The network connecting GPUs is often the bottleneck in distributed training. Understanding the hardware matters.

NVLink and NVSwitch: within a single server, NVIDIA's NVLink provides 600–900 GB/s bidirectional bandwidth between GPUs. An NVSwitch connects all 8 GPUs in a DGX node at full NVLink bandwidth to any other GPU — no ring topology needed. Tensor parallelism exploits this high intra-node bandwidth.

InfiniBand (IB): between servers, InfiniBand is the standard. HDR InfiniBand provides 200 Gb/s per port; NDR provides 400 Gb/s. A typical large cluster uses multiple IB rails (ports) per node — 8 rails for 8 GPUs, each GPU paired with its own network adapter, providing 1.6–3.2 Tb/s aggregate bandwidth per node. NVIDIA's Quantum-2 switches support SHARP (Scalable Hierarchical Aggregation and Reduction Protocol), which performs AllReduce in the network switch hardware itself — GPUs send partial data and the switch returns the reduced result, halving the communication volume.

RoCE (RDMA over Converged Ethernet): a cheaper alternative to InfiniBand, running RDMA over standard Ethernet switches. Performance is lower (especially latency) and requires careful network configuration (PFC, ECN), but the cost savings can be significant for large clusters. Some cloud providers (like AWS with EFA) use proprietary RDMA solutions.

Memory Optimization Beyond ZeRO

Several additional techniques complement ZeRO and gradient checkpointing:

Gradient accumulation: simulate a large batch by accumulating gradients over multiple forward-backward passes before updating. With 8 GPUs and 4 accumulation steps, an effective batch size of 32 per GPU becomes 8 × 32 × 4 = 1,024. No extra memory needed — you just don't zero the gradients between accumulation steps.

Selective activation checkpointing: instead of checkpointing every block uniformly, checkpoint only the memory-heavy layers (like attention with long sequences) and keep the memory-cheap layers' activations. This achieves most of the memory savings with less recomputation overhead.

CPU offloading for activations: move activations to CPU RAM during the forward pass and fetch them back during the backward pass. Overlapping transfers with computation hides latency. This is especially useful for very long sequences where activation memory is the dominant cost.

Flash Attention (technically a kernel optimization, not a memory strategy) reduces attention memory from O(N²) to O(N) by computing attention in tiles without materializing the full attention matrix. This is now standard — nearly every large model uses it.

Throughput Metrics and Monitoring

You can't optimize what you don't measure. Key metrics for distributed training:

Tokens per second (or samples per second): the top-level throughput metric. For language models, this is total tokens processed per second across all GPUs. Compare against hardware-theoretical peak to gauge efficiency.

Model FLOPS Utilization (MFU): the fraction of theoretical peak FLOPS that your training achieves. An H100 delivers ~990 TFLOPS of BF16. If your training achieves 500 TFLOPS per GPU, your MFU is ~50%. Good large-model training achieves 40–60% MFU. Values below 30% indicate significant inefficiency — usually communication bottlenecks or pipeline bubbles.

Hardware FLOPS Utilization (HFU): similar to MFU but counts all FLOPS, including recomputation from gradient checkpointing. HFU is always ≥ MFU.

GPU utilization: percentage of time the GPU is actively computing. High utilization doesn't guarantee high throughput — the GPU might be running inefficient kernels. MFU is more informative.

Communication-to-computation ratio: time spent on collective communications versus time spent on forward/backward. If communication exceeds 20–30% of total step time, you have a networking bottleneck.

Training Orchestration

Launching training on hundreds of GPUs across dozens of nodes requires orchestration beyond "python train.py". Key tools:

SLURM: the standard job scheduler for on-premise GPU clusters. Allocates nodes, manages job queues, handles multi-node job launch via srun. Training scripts use torchrun (or DeepSpeed's launcher) within SLURM jobs.

Kubernetes + GPU operators: for cloud-native training. NVIDIA's GPU Operator and Kubeflow Training Operator manage GPU allocation, multi-node job scheduling, and fault recovery. More complex to set up than SLURM but integrates with cloud autoscaling.

torchrun (PyTorch's elastic launcher): handles process spawning, environment variable setup (RANK, WORLD_SIZE, LOCAL_RANK), and fault tolerance. It replaces the older torch.distributed.launch.

# Launch distributed training on 4 nodes, 8 GPUs each
# On SLURM:
srun --nodes=4 --ntasks-per-node=8 --gpus-per-node=8 \
    torchrun --nnodes=4 --nproc-per-node=8 \
    --rdzv-backend=c10d --rdzv-endpoint=$MASTER_ADDR:$MASTER_PORT \
    train.py --config config.yaml

# Using DeepSpeed launcher:
deepspeed --num_nodes=4 --num_gpus=8 \
    --hostfile=hostfile.txt \
    train.py --deepspeed ds_config.json

Multi-Node Debugging

Debugging distributed training is notoriously painful. A hang on rank 37 out of 256 could be a deadlocked collective, a slow NIC, or a GPU that silently produces NaNs.

NCCL debugging: set NCCL_DEBUG=INFO to see which collectives are in flight. Set NCCL_DEBUG_SUBSYS=ALL for detailed topology and ring information. For hangs, NCCL_DEBUG=WARN plus TORCH_DISTRIBUTED_DEBUG=DETAIL will report which rank is stuck in which collective.

Gradient NaN detection: check for NaN/Inf after each backward pass. When detected, log the batch, the offending rank, and the layer. DeepSpeed has built-in NaN detection. In PyTorch, wrap your training loop with checks: torch.isnan(loss).any() and periodically check gradient norms.

Reproducibility aids: when a bug is intermittent, fix the seed, use deterministic algorithms (torch.use_deterministic_algorithms(True)), and checkpoint right before the failing step. This lets you replay the exact failure on a single node for easier debugging.

Reproducibility

Exact reproducibility across runs is harder than it sounds. Sources of non-determinism:

CUDA non-determinism: atomic operations in CUDA kernels (used in backward passes of scatter/gather operations) are non-deterministic. torch.use_deterministic_algorithms(True) forces deterministic alternatives, but some operations become significantly slower or throw errors.

Floating-point summation order: AllReduce across GPUs may sum gradients in a different order each time. Since floating-point addition isn't associative (a + b) + c ≠ a + (b + c) in float arithmetic, the result varies by tiny amounts. Over thousands of steps, these differences compound. This is inherent to distributed training — you can't fix it without fixing the reduction order.

Data shuffling and loading: ensure the random seed for your data sampler is set and that data loading order is deterministic given the seed. Save and restore the data loader state alongside the model state in checkpoints.

Practical stance: aim for near-reproducibility — same hyperparameters and data should produce statistically similar (not bit-identical) results. Log everything: config, git commit, library versions, hardware. If two runs produce different final accuracy, you should be able to determine whether the difference is meaningful or just floating-point noise.

Elastic Training

Elastic training allows the number of workers to change during a training run. Workers can be added (scale-up) or removed (scale-down, or failure recovery) without restarting from scratch.

TorchElastic implements this through a rendezvous mechanism: when a worker fails, remaining workers detect the failure, re-establish a smaller process group, reshard the data, and continue training. When a replacement worker comes online, the group expands again. The learning rate and batch size may need adjustment when worker count changes — if you go from 32 to 28 GPUs, the effective batch size drops by 12.5%, which can affect convergence.

Elastic training is especially valuable on preemptible cloud instances (AWS Spot, GCP Preemptible). These instances cost 60–90% less but can be reclaimed with little notice. Elastic training converts these unreliable instances into a usable (if variable) training cluster.

Power, Cooling, and Carbon-Aware Training

Training a large model consumes enormous energy. Training GPT-3 reportedly used ~1,300 MWh — enough to power about 120 US homes for a year. This energy has both financial and environmental costs.

Power management: a single DGX H100 node (8 GPUs) draws about 10 kW. A 512-GPU cluster draws ~640 kW. At $0.10/kWh, a two-week training run costs about $21,500 in electricity alone — on top of hardware and cooling costs. GPU power limits can be capped (e.g., 300W vs 400W per H100) to reduce power draw at a modest throughput reduction (typically 10–15% less throughput for 25% less power).

Cooling: GPUs throttle when they overheat. Air cooling works up to about 500W per GPU; beyond that, liquid cooling (direct-to-chip or immersion) is necessary. Hot spots in a data center — where exhaust air from one rack enters the intake of another — can cause intermittent throttling that looks like a straggler problem. Thermal monitoring is a legitimate debugging tool for distributed training.

Carbon-aware training: electricity carbon intensity varies by time of day and grid region. Training during off-peak hours when grids rely more on renewables (midday solar, nighttime wind) can reduce carbon footprint by 30–50% without changing anything else. Some organizations schedule training runs to coincide with periods of low-carbon electricity, or preferentially use data centers in regions with cleaner grids (e.g., Quebec hydropower vs. coal-heavy regions).

Reporting is becoming standard. Papers increasingly include compute budgets (GPU-hours), energy consumption (kWh), and estimated CO₂ emissions. The ML community is slowly building norms around responsible compute usage — not just for environmental reasons, but because compute-efficiency is a competitive advantage. A team that trains in half the GPU-hours at the same quality is twice as productive.

Putting It All Together: A Training Configuration

Here's what a state-of-the-art training setup for a 70B model might look like:

# Infrastructure for training a 70B parameter model
# Hardware: 512 × H100 GPUs across 64 nodes (8 GPUs/node)
# Interconnect: NVLink within node, 8× NDR400 InfiniBand between nodes

# Parallelism strategy:
#   Tensor Parallelism (TP) = 8  (within node, on NVLink)
#   Pipeline Parallelism (PP) = 8  (across 8 nodes per replica)
#   Data Parallelism (DP) = 8  (8 pipeline replicas)
#   Total: 8 × 8 × 8 = 512 GPUs

# Memory optimization:
#   - BF16 mixed precision (no loss scaling needed)
#   - ZeRO Stage 1 (optimizer state sharding across DP ranks)
#   - Gradient checkpointing (every transformer block)
#   - Flash Attention 2 (O(N) memory for attention)
#   - Sequence parallelism (activation memory split across TP ranks)

# Batch configuration:
#   - Micro-batch size: 2 per GPU
#   - Gradient accumulation steps: 4
#   - Pipeline micro-batches: 8
#   - Global batch size: 2 × 4 × 8 (DP) = 64 sequences
#   - Sequence length: 8192 tokens
#   - Effective tokens per step: 64 × 8192 = 524,288

# Fault tolerance:
#   - Async checkpointing every 200 steps (~15 min)
#   - Checkpoint to parallel filesystem (Lustre / GPFS)
#   - NCCL timeout = 30 min (allows recovery from transient network issues)
#   - Automatic restart on failure with torchrun elastic

# Monitoring:
#   - Weights & Biases for loss curves and learning rate
#   - GPU utilization, memory, temperature via DCGM
#   - Network bandwidth via IB counters
#   - MFU computed and logged every 100 steps
#   - Alert on gradient norm spikes, NaN loss, or MFU drops > 10%
The deepest lesson of modern training infrastructure is that training a frontier model is a systems engineering problem as much as a machine learning one. The researcher who understands ZeRO stages, pipeline bubbles, network topologies, and failure modes can train models that a pure ML researcher cannot — not because of better ideas, but because of better execution. As models continue to grow, infrastructure expertise becomes not a nice-to-have but a prerequisite for doing cutting-edge work.
