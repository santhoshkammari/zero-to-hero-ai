JAX

Chapter 8: Training Deep Networks 5 subtopics
TL;DR

JAX is Google's numerical computing library built on a radical idea: take NumPy, make every function transformable. You can automatically differentiate (grad), compile to hardware (jit), vectorize over batches (vmap), and parallelize across devices (pmap) — all by composing function transformations. It demands a functional programming style with pure functions and immutable state, which feels unfamiliar at first but unlocks compiler optimizations that PyTorch and TensorFlow cannot match. Neural network libraries like Flax and Haiku provide the familiar layers-and-models abstraction on top. JAX is the engine behind AlphaFold, Gemini, and much of Google DeepMind's research.

Why Another Framework?

You already know PyTorch. Maybe you've used TensorFlow. So why does JAX exist? The short answer: PyTorch and TensorFlow both started as "define a neural network and train it" frameworks. They grew their automatic differentiation around that goal. JAX started from a completely different place — it asked, "what if we took NumPy and made it fully differentiable and compilable?" The result is a library that's lower-level than PyTorch, more composable, and dramatically more flexible for research that pushes beyond standard training loops.

If you're implementing a paper that involves second-order derivatives, per-example gradients, custom physics simulations, or anything that doesn't fit neatly into a forward-backward-update cycle, JAX often lets you express it in ten lines where PyTorch would need a hundred. The cost? You have to think differently about how you write code.

The Functional Paradigm

Pure Functions: The Core Contract

JAX's power comes from one strict rule: your functions must be pure. A pure function takes inputs, produces outputs, and does nothing else. No side effects. No mutating global variables. No printing inside compiled code. No reading from files mid-computation. Given the same inputs, a pure function always returns the same outputs.

Why does this matter? Because when a function is pure, a compiler can reason about it completely. It can reorder operations, fuse them together, run them on different hardware, compute their derivatives symbolically — all because it knows the function has no hidden dependencies. The moment you sneak in a side effect (say, appending to a Python list inside your loss function), the compiler's assumptions break and you get silent wrong answers or cryptic errors.

# Pure function — JAX loves this
def pure_loss(params, x, y):
    predictions = model_apply(params, x)
    return jnp.mean((predictions - y) ** 2)

# Impure function — JAX will break or silently miscompute
counter = 0
def impure_loss(params, x, y):
    global counter
    counter += 1  # side effect!
    predictions = model_apply(params, x)
    return jnp.mean((predictions - y) ** 2)

In PyTorch, your model is an object. It holds its own weights in self.parameters(). You call model(x) and it mutates internal state during the forward pass (storing activations for backprop). In JAX, there is no model object in that sense. Your parameters are just a data structure — typically a nested dictionary of arrays — and your forward pass is a pure function that takes those parameters as an explicit argument. This feels weird at first. It's also what makes everything else possible.

Immutable Arrays

JAX arrays cannot be modified in place. There is no x[0] = 5. Instead you write x = x.at[0].set(5), which returns a new array. This isn't just a stylistic preference — it's enforced because in-place mutation breaks the compiler's ability to trace and transform your function. Under JIT compilation, JAX can often optimize the "copy" away entirely, so the performance cost is zero. But the programming model forces you to think in terms of "data in, data out" at every level.

import jax.numpy as jnp

# This raises an error in JAX
# x = jnp.array([1, 2, 3])
# x[0] = 99  # TypeError!

# Instead, use functional update syntax
x = jnp.array([1, 2, 3])
x = x.at[0].set(99)  # returns a new array: [99, 2, 3]

The Four Transformations

JAX's killer feature is four composable function transformations. Each one takes a function and returns a new function with added capabilities. Because they compose, you can stack them in any order.

jit — Just-In-Time Compilation

jit (just-in-time compilation) traces your Python function, converts it to an intermediate representation called HLO (High-Level Optimizer), and compiles it through XLA (Accelerated Linear Algebra) into optimized machine code for your specific hardware — CPU, GPU, or TPU. The first call is slow (compilation), but subsequent calls with the same input shapes skip Python entirely and run native code.

import jax
import jax.numpy as jnp

def slow_function(x):
    for _ in range(10):
        x = jnp.sin(x) + jnp.cos(x)
    return x

fast_function = jax.jit(slow_function)

x = jnp.ones(1000)
# First call: traces + compiles (slow)
result = fast_function(x)
# Second call: runs cached compiled code (fast)
result = fast_function(x)

XLA doesn't just translate your operations one-to-one. It fuses them. If you compute jnp.sin(x) + jnp.cos(x), Python without JIT would: (1) allocate memory for sin result, (2) compute sin, (3) allocate memory for cos result, (4) compute cos, (5) allocate memory for sum, (6) compute sum. XLA fuses this into a single kernel that reads x once, computes sin + cos in a single pass, and writes the result once. Fewer memory round-trips means dramatically faster execution, especially on GPUs where memory bandwidth is the bottleneck.

The tradeoff: jit requires that your function's control flow doesn't depend on the values of the arrays. Python if statements that branch on array values will fail inside jit. You need to use jax.lax.cond for conditional logic and jax.lax.scan or jax.lax.fori_loop for loops. This is the price of compilation — the compiler needs to see all possible execution paths at trace time.

grad — Automatic Differentiation

grad takes a scalar-valued function and returns a new function that computes its gradient. That's it. No loss.backward(), no zero_grad(), no optimizer.step() — just a function that eats parameters and returns derivatives.

def mse_loss(params, x, y):
    pred = jnp.dot(x, params)
    return jnp.mean((pred - y) ** 2)

# grad returns a *function* that computes gradients
grad_fn = jax.grad(mse_loss)

# Call it like any other function
params = jnp.array([1.0, 2.0, 3.0])
x = jnp.ones((10, 3))
y = jnp.ones(10)
grads = grad_fn(params, x, y)
# grads has the same shape as params

By default, grad differentiates with respect to the first argument. Use argnums to differentiate with respect to other arguments. Need both the value and the gradient? Use jax.value_and_grad — it's more efficient than calling the function and its gradient separately because it shares the forward computation.

Here's where things get powerful. Because grad returns a regular function, you can differentiate again. jax.grad(jax.grad(f)) gives you second derivatives. Need the full Hessian matrix? jax.hessian(f). Need the Hessian-vector product (critical for second-order optimization) without forming the full Hessian? Compose jax.jvp (forward-mode) with jax.grad (reverse-mode). In PyTorch, computing higher-order derivatives requires create_graph=True and careful bookkeeping. In JAX, it's just function composition.

# Higher-order derivatives — just compose
second_deriv = jax.grad(jax.grad(lambda x: x ** 3))
print(second_deriv(2.0))  # 12.0, because d²/dx²(x³) = 6x

# Efficient Hessian-vector product (no full Hessian in memory)
def hvp(f, x, v):
    return jax.grad(lambda x: jnp.dot(jax.grad(f)(x), v))(x)

vmap — Automatic Vectorization

vmap (vectorized map) takes a function that operates on a single example and returns a function that operates on a batch. You write the logic for one data point, and vmap handles the batching.

# Function that handles a single example
def single_predict(params, x):
    return jnp.dot(x, params['w']) + params['b']

# Automatically batched version — maps over first axis of x
batch_predict = jax.vmap(single_predict, in_axes=(None, 0))
# None means don't map over params; 0 means map over axis 0 of x

params = {'w': jnp.ones(5), 'b': 0.0}
batch_x = jnp.ones((32, 5))  # 32 examples
predictions = batch_predict(params, batch_x)  # shape: (32,)

Why is this better than just writing your function to handle batches from the start? Two reasons. First, code clarity — the single-example version is simpler to write, test, and debug. Second, composability. Combine vmap with grad to get per-example gradients: the gradient of the loss for each individual example in the batch, not summed or averaged. In PyTorch, per-example gradients require libraries like BackPACK or functorch. In JAX, it's one line: jax.vmap(jax.grad(loss_fn))(params, batch_x, batch_y). Per-example gradients are critical for differential privacy and certain meta-learning algorithms.

pmap — Parallel Map Across Devices

pmap (parallel map) replicates a function across multiple devices (GPUs or TPU cores) and runs them simultaneously. Each device gets a shard of the data. You write the single-device logic; pmap handles distribution and communication.

# Suppose you have 4 GPUs
@jax.pmap
def parallel_step(params, batch):
    grads = jax.grad(loss_fn)(params, batch)
    # Average gradients across all devices
    grads = jax.lax.pmean(grads, axis_name='devices')
    return update_params(params, grads)

# Data is split across 4 devices along first axis
# params are replicated on each device
new_params = parallel_step(replicated_params, sharded_batches)

The jax.lax.pmean call is a collective communication operation — it computes the mean of gradients across all devices. Other collectives include psum (sum), pmax (max), and all_gather (concatenate). These map directly to hardware-level communication primitives like NCCL AllReduce on GPUs.

XLA Compilation Under the Hood

When you call jax.jit, JAX traces your function by feeding it abstract "tracer" values that record what operations are performed without actually executing them. The trace produces an HLO graph — a hardware-independent computation graph. XLA then takes this graph and optimizes it for your target hardware.

Key XLA optimizations include: operator fusion (combining multiple elementwise operations into one kernel to reduce memory traffic), layout optimization (rearranging how data is stored in memory for better cache behavior), buffer reuse (allocating and deallocating memory to minimize peak usage), and automatic tiling (splitting large operations into chunks that fit in cache or shared memory). On TPUs, XLA also handles the mapping of operations to the TPU's systolic array architecture.

This is why JAX code can be faster than equivalent PyTorch code even when the operations are mathematically identical. PyTorch dispatches each operation individually to CUDA kernels. JAX compiles the entire computation graph and optimizes globally. The gap is most noticeable for computations with many small operations (where per-kernel launch overhead dominates in PyTorch) and on TPUs (which were designed specifically for XLA).

Flax and Haiku: Neural Network Libraries

Raw JAX gives you arrays and transformations. You could build a neural network from scratch using jnp.dot and jax.grad, but managing parameter initialization, layer composition, and training state by hand gets tedious fast. That's what Flax and Haiku solve.

Flax (Linen API)

Flax is Google's recommended neural network library for JAX. Its Linen API uses a module system where you define layers as classes, but — crucially — the parameters live outside the module in a separate data structure.

import flax.linen as nn
import jax
import jax.numpy as jnp

class MLP(nn.Module):
    hidden_dim: int
    out_dim: int

    @nn.compact
    def __call__(self, x):
        x = nn.Dense(self.hidden_dim)(x)
        x = nn.relu(x)
        x = nn.Dense(self.out_dim)(x)
        return x

# Create the model (no weights yet — it's just a structure)
model = MLP(hidden_dim=128, out_dim=10)

# Initialize parameters with a dummy input
key = jax.random.PRNGKey(0)
params = model.init(key, jnp.ones((1, 784)))

# Forward pass is a pure function: model.apply(params, x)
predictions = model.apply(params, jnp.ones((32, 784)))

Notice the pattern: model.init produces a parameter tree, and model.apply is a pure function of (params, input). There's no hidden state inside the model object. This means you can jit, grad, vmap, and pmap over model.apply with zero friction.

Haiku

Haiku, from DeepMind, takes a different approach. It lets you write code that looks more like PyTorch — you create layers and call them in a forward function — but wraps everything in a hk.transform that extracts the pure function and parameter initialization.

import haiku as hk
import jax

def forward(x):
    mlp = hk.nets.MLP([128, 10])
    return mlp(x)

# Transform converts the impure-looking function into
# a pair of pure functions: init and apply
model = hk.transform(forward)
key = jax.random.PRNGKey(0)
params = model.init(key, jnp.ones((1, 784)))
output = model.apply(params, key, jnp.ones((32, 784)))

Haiku was the default in many DeepMind papers (AlphaFold used Haiku). However, as of 2024, Flax has become the more actively maintained and community-recommended choice. For new projects, Flax Linen is the safer bet.

Optax: Composable Optimizers

Optax is the standard optimizer library for JAX. Instead of monolithic optimizer classes, Optax provides composable gradient transformations that you chain together.

import optax

# Compose: clip gradients, then apply Adam
optimizer = optax.chain(
    optax.clip_by_global_norm(1.0),
    optax.adam(learning_rate=1e-3)
)

opt_state = optimizer.init(params)

# Training step — again, a pure function
def train_step(params, opt_state, batch):
    loss, grads = jax.value_and_grad(loss_fn)(params, batch)
    updates, opt_state = optimizer.update(grads, opt_state, params)
    params = optax.apply_updates(params, updates)
    return params, opt_state, loss

# JIT the entire training step
train_step = jax.jit(train_step)

This composability is powerful. Want gradient clipping, then weight decay, then Adam, then an exponential moving average of parameters? Chain four transformations. In PyTorch, you'd be subclassing optimizers or adding hooks. In Optax, it's four lines.

JAX vs PyTorch: When to Use Which

This isn't a "which is better" debate. They optimize for different things.

Choose PyTorch when: you want the largest ecosystem of pretrained models and libraries (Hugging Face, torchvision, torchaudio); you need eager execution for debugging complex logic; your team already knows it; you're doing standard supervised learning on GPUs; you want the path of least resistance.

Choose JAX when: you need higher-order derivatives (meta-learning, physics-informed neural nets); you need per-example gradients (differential privacy); you're working with TPUs (JAX is the best-supported framework for TPUs); you want to compile entire training steps into a single fused kernel; your research involves non-standard training loops or custom mathematical transformations; you value composability over convenience.

Honest tradeoffs: JAX's ecosystem is smaller. Debugging jit-compiled code is harder — errors appear at trace time with abstract shapes, not at runtime with real values. The functional style has a learning curve. Random number generation requires explicit PRNG key management (no global random state), which is verbose but reproducible. Model serialization and deployment tooling is less mature than PyTorch's TorchScript/ONNX pipeline.

The practical reality in 2024–2025: PyTorch dominates industry and most academic research. JAX dominates Google/DeepMind research and TPU-heavy workloads. If you're starting a new research project that involves anything beyond standard training, learning JAX will pay dividends. If you're building a product, PyTorch's ecosystem advantage is hard to ignore. Many researchers are fluent in both — the concepts transfer, only the syntax and mental model differ.

Random Number Generation in JAX

One JAX quirk that trips up every beginner: there's no global random state. In NumPy, you call np.random.randn(5) and the global seed advances. In JAX, every random call requires an explicit PRNG key, and you manually split keys to get independent streams.

key = jax.random.PRNGKey(42)
key1, key2 = jax.random.split(key)  # two independent keys

# Each random call consumes a key
noise1 = jax.random.normal(key1, shape=(3,))
noise2 = jax.random.normal(key2, shape=(3,))

# WRONG: reusing the same key gives the same "random" values
bad1 = jax.random.normal(key, shape=(3,))
bad2 = jax.random.normal(key, shape=(3,))
# bad1 == bad2! Not what you want.

This feels verbose, but it's intentional. Explicit keys make code perfectly reproducible — no hidden global state means the same key always produces the same result, even across different hardware and parallelization strategies. This is a major advantage for scientific reproducibility, and it's essential for correctness inside jit and pmap where operations may be reordered.

Putting It All Together

import jax
import jax.numpy as jnp
import flax.linen as nn
import optax

# 1. Define model (Flax)
class Classifier(nn.Module):
    @nn.compact
    def __call__(self, x):
        x = nn.Dense(256)(x)
        x = nn.gelu(x)
        x = nn.Dense(10)(x)
        return x

# 2. Initialize
model = Classifier()
key = jax.random.PRNGKey(0)
params = model.init(key, jnp.ones((1, 784)))
tx = optax.adam(1e-3)
opt_state = tx.init(params)

# 3. Training step — pure function, JIT compiled
@jax.jit
def step(params, opt_state, x, y):
    def loss_fn(p):
        logits = model.apply(p, x)
        return optax.softmax_cross_entropy_with_integer_labels(
            logits, y
        ).mean()
    loss, grads = jax.value_and_grad(loss_fn)(params)
    updates, opt_state = tx.update(grads, opt_state, params)
    params = optax.apply_updates(params, updates)
    return params, opt_state, loss

# 4. Training loop
for epoch in range(10):
    for x_batch, y_batch in dataloader:
        params, opt_state, loss = step(
            params, opt_state, x_batch, y_batch
        )
    print(f"Epoch {epoch}, Loss: {loss:.4f}")

The entire training step — forward pass, loss computation, backward pass, gradient clipping, optimizer update — compiles into a single XLA program. On a GPU, this runs as a fused sequence of kernels with minimal Python overhead. On a TPU, it maps to the hardware's matrix units. The same code runs on both without changes.

That's JAX in a nutshell. Functional programming for numerical computing, with four transformations that compose like LEGO. The learning curve is real, but the payoff is code that's fast, composable, and correct by construction.
