Frameworks: TensorFlow, Keras & JAX

Chapter 8: Training Deep Networks Section 4 of 16 ~8 min read
TL;DR

PyTorch is your primary tool. But the deep learning world isn't a monoculture. TensorFlow/Keras dominates production and mobile deployment. JAX offers functional purity with compiler-level transforms for large-scale research at Google. You don't need to master all three — but you should know they exist, what they're good at, and when you'd reach for them.

The Framework Landscape in 2024

You've just learned to train models in PyTorch. Good — that's the right first choice. PyTorch dominates research papers, most open-source models ship in PyTorch, and its ecosystem is huge. But you'll encounter other frameworks in the wild, and knowing the map saves you from confusion when you do.

Three frameworks matter:

  • PyTorch — the research standard, increasingly used in production too (via TorchServe, ONNX export, etc.)
  • TensorFlow / Keras — the production workhorse, especially for mobile (TFLite) and serving (TF Serving). Still enormous in industry.
  • JAX — the functional, compiler-first framework. Used at Google/DeepMind for large-scale research. Think "NumPy that runs on GPUs and auto-differentiates."

TensorFlow & Keras

Keras is TensorFlow's high-level API. Where PyTorch gives you an explicit training loop — forward pass, loss, backward, step — Keras wraps all of that into three calls:

import tensorflow as tf
from tensorflow import keras

# Define model
model = keras.Sequential([
    keras.layers.Dense(128, activation='relu', input_shape=(784,)),
    keras.layers.Dropout(0.2),
    keras.layers.Dense(10, activation='softmax')
])

# Compile: specify optimizer, loss, metrics
model.compile(optimizer='adam',
              loss='sparse_categorical_crossentropy',
              metrics=['accuracy'])

# Train: one line
model.fit(x_train, y_train, epochs=5, batch_size=32, validation_split=0.1)

# Predict: one line
predictions = model.predict(x_test)

That's it. compile → fit → predict. If you know PyTorch, Keras feels like training wheels — less control, but dramatically less boilerplate. For quick prototyping or standard architectures, this conciseness is genuinely useful.

Where You'll See TensorFlow

  • TFLite — optimized runtime for mobile and edge devices. If your model needs to run on a phone, TFLite is a mature path.
  • TF Serving — production model serving with versioning, batching, and gRPC/REST endpoints out of the box.
  • Legacy production systems — many companies built their ML infrastructure on TF 1.x. These systems still exist and still need maintenance.
  • Keras ecosystem — pretrained models, callbacks, built-in data pipelines (tf.data) that handle large datasets efficiently.
TF2 vs TF1: TensorFlow 2 added eager execution (like PyTorch), making it much friendlier for debugging. But TF's core identity is still graph-mode compilation — you define a computation graph, then TF optimizes and compiles it for fast execution. This is what makes TF Serving and TFLite fast in production.

JAX

JAX takes a fundamentally different approach. It's built around pure functions — no hidden state, no object mutation. You write NumPy-like code, and JAX provides function transforms that do powerful things:

  • jax.grad — automatic differentiation. Pass a function, get its gradient function.
  • jax.jit — just-in-time compilation via XLA. Your Python function gets compiled to optimized machine code.
  • jax.vmap — automatic vectorization. Write code for a single example, vmap makes it work on batches.
  • jax.pmap — automatic parallelization across multiple devices.
import jax
import jax.numpy as jnp

# A simple loss function (pure function — no side effects)
def loss_fn(params, x, y):
    preds = jnp.dot(x, params['w']) + params['b']
    return jnp.mean((preds - y) ** 2)

# Get the gradient function — jax.grad returns a NEW function
grad_fn = jax.grad(loss_fn)

# JIT-compile it for speed
fast_grad_fn = jax.jit(grad_fn)

# Use it
params = {'w': jnp.ones((3,)), 'b': 0.0}
grads = fast_grad_fn(params, x_batch, y_batch)
# grads is a dict with same structure as params: {'w': ..., 'b': ...}

Notice the difference from PyTorch: there's no loss.backward(). You don't attach gradients to tensors. Instead, jax.grad takes a function and returns a new function that computes gradients. It's functional programming applied to deep learning.

The JAX Ecosystem

JAX itself is low-level — it gives you arrays and transforms, not neural network layers. Libraries build on top:

  • Flax — Google's neural network library for JAX. Provides nn.Module equivalents.
  • Optax — gradient processing and optimizers (Adam, SGD, etc.) composable as chains of transforms.
  • Equinox — a more PyTorch-like nn library for JAX (modules as pytrees).
When you'd use JAX: Research requiring custom differentiation (e.g., differentiating through an ODE solver), TPU-heavy workloads (JAX has first-class TPU support via XLA), projects in the Google/DeepMind ecosystem, or when you need composable function transforms that don't exist elsewhere.

Framework Comparison

PyTorch TensorFlow / Keras JAX
Paradigm Imperative, object-oriented Keras: declarative. TF core: graph-based Functional, transform-based
Strengths Pythonic, great debugging, huge research community Production tooling, mobile/edge (TFLite), enterprise ecosystem Composable transforms, JIT compilation, TPU support, mathematical elegance
Weaknesses Production deployment less mature (improving) Complex API surface, TF1→TF2 migration baggage Steep learning curve, smaller community, requires functional mindset
Best for Research, prototyping, most new projects Production serving, mobile deployment, quick prototyping (Keras) Google-scale research, custom math, TPU workloads
Community Largest in research, growing in industry Largest in production/industry Smaller but growing, concentrated at Google/DeepMind

The Honest Take

Learn PyTorch deeply. It's what you'll use for research, for most open-source code you'll read, and increasingly for production. The explicit training loop teaches you what's actually happening.

Know Keras exists for quick prototyping and when you need to deploy to mobile. If someone hands you a TensorFlow codebase, you won't be lost — the concepts (layers, optimizers, losses) are the same, just with different syntax.

Know JAX exists for functional, compiler-optimized research at Google scale. If you're doing heavy math research or working with TPUs, JAX's transforms are genuinely powerful. For most people, it's "I'll learn it if I need it."

Framework fluency ≠ framework mastery. The underlying concepts — gradient descent, backpropagation, loss functions, regularization — are the same everywhere. Once you deeply understand one framework, switching to another is syntax, not fundamentals.

What You Should Now Be Able To Do

  • Explain the high-level differences between PyTorch, TensorFlow/Keras, and JAX
  • Read a basic Keras compile → fit → predict workflow and understand what it's doing
  • Describe JAX's core transforms (grad, jit, vmap) and why the functional paradigm matters
  • Identify which framework fits a given scenario (research vs. mobile deployment vs. TPU-scale training)
  • Confidently stick with PyTorch while knowing when to reach for an alternative
