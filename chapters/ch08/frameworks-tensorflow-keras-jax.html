TL;DR

PyTorch owns research. Fast in production too. TensorFlow still runs huge share of deployed models — graph compilation and mobile path (TFLite) still unmatched. JAX = wildcard: functional, compiler-first library. Write NumPy, get auto-diff, JIT compilation, vectorization, multi-device parallelism through composable function transforms. Keras 3 runs on all three backends. No need master every framework. Need know why each exists, trade-off each makes, when each fits. This comes up in senior ML infrastructure talks.

Why This Section Exists

Last section trained models with explicit forward passes, losses, .backward(), optimizer step. Big-company production ML system? Usually TensorFlow. DeepMind paper pushing scale? Usually JAX. Fast afternoon prototype? Usually Keras. These frameworks give different answers to one question: what should contract between programmer and GPU look like?

Three Philosophies, One Goal

Framework differences live in how they organize contract between code and hardware. Use house analogy:

Think house build. PyTorch = construction crew following orders in real time — "put brick here, now there." You see each step. Stop mid-wall, inspect mortar, change mind. That's eager execution. TensorFlow = hand architect blueprint; architect optimizes whole plan before first brick — rearrange plumbing, fuse redundant walls, pre-cut lumber. That's graph compilation. JAX goes further: describe mathematical function for "what house should look like," JAX returns new function that builds it — optimized, parallelized, differentiable. That's function transformation.

Same house. Three very different relationships with process.

Our Running Example: A Tiny Classifier

Keep things concrete. Build same thing in every framework: small neural network for handwritten digits (MNIST — 28×28 grayscale images, 10 classes). Two hidden layers, ReLU activations, softmax output. Simple setup. Architecture stays same everywhere — code you write and machinery underneath change.

Watch same four operations in each framework: define model, compute loss, get gradients, update parameters. End result: gut-level sense for why these frameworks feel different while doing same math.

TensorFlow: The Graph Compiler

TensorFlow came from Google in 2015. First few years, dominant deep learning framework. Full stop. Doing deep learning from 2015 to 2019? Probably writing TensorFlow.

Need know early days: TensorFlow 1.x hurt. You defined whole computation as static graph with placeholder variables, then "ran" graph inside Session. Could not print tensor value during execution. Could not set breakpoint and inspect intermediate results. Debugging felt like fixing car engine from blueprint, not open hood.

TensorFlow 2 (2019) fixed much of this by making eager execution default — ops run immediately, like PyTorch. But TF identity stayed same. Under hood, TensorFlow still wants compiled computation graph. Still wants architect with blueprint.

Eager Mode vs Graph Mode — The Central Tension

Most important TensorFlow idea. Shows up in interviews all time. TF lives in two modes. Tension between them explains most of framework design.

In eager mode, every op runs immediately and returns concrete value. Print it, inspect with Python debugger, mix with regular Python code. Feels natural. Also slower, because each op goes through Python interpreter and per-op GPU dispatch overhead.

In graph mode, wrap function with @tf.function. TensorFlow traces it — records every op without executing — and builds computation graph. Then TF hands graph to compiler (XLA, same compiler JAX uses), which fuses ops, removes redundancies, optimizes memory layout, produces one compiled kernel. Result runs fast because Python stays out of execution loop.

import tensorflow as tf

# Eager mode: this runs immediately, returns a concrete value
x = tf.constant([1.0, 2.0, 3.0])
y = x * 2 + 1
print(y)  # tf.Tensor([3. 5. 7.], shape=(3,), dtype=float32)

# Graph mode: wrap with @tf.function to compile
@tf.function
def compute(x):
    return x * 2 + 1

# First call: TF traces the function, builds a graph, compiles it
# Subsequent calls with same shape/dtype: runs compiled version directly
result = compute(x)

First call to @tf.function-decorated function makes TF trace it — special tracer objects flow through function and record every op. Future calls with same input shapes and dtypes skip Python and run compiled graph. Different shapes? TF retraces and compiles again.

Tricky part: TF uses AutoGraph to turn Python control flow (if-statements, for-loops) into graph ops. Python if becomes tf.cond. Python for loop becomes tf.while_loop. Usually works. Sometimes breaks, error messages brutal. Common trap: Python conditional works in eager mode but does wrong thing inside @tf.function.

The tracing trap: Side effects inside @tf.function happen during tracing, not execution. print() inside tf.function fires once (during tracing), then never again. Use tf.print(). Huge TensorFlow gotcha.

Our MNIST Classifier in Keras

Keras is TensorFlow's high-level API. PyTorch gives explicit training loop — forward, loss, backward, step. Keras collapses all that into three method calls. Here digit classifier:

import tensorflow as tf
from tensorflow import keras

model = keras.Sequential([
    keras.layers.Flatten(input_shape=(28, 28)),
    keras.layers.Dense(128, activation='relu'),
    keras.layers.Dense(64, activation='relu'),
    keras.layers.Dense(10, activation='softmax')
])

model.compile(
    optimizer='adam',
    loss='sparse_categorical_crossentropy',
    metrics=['accuracy']
)

model.fit(x_train, y_train, epochs=5, batch_size=32, validation_split=0.1)
predictions = model.predict(x_test)

compile configures optimizer and loss. fit runs whole training loop — batching, forward pass, loss computation, backprop, parameter update, metric tracking, validation. predict runs inference. Three calls. Whole training pipeline.

Coming from PyTorch, where you wrote 30-line training loop, this feels like cheating. In some sense, yes — trade control for development speed. For standard architectures and standard training, Keras is hard to beat for iteration speed. Need custom training step, non-standard loss, or loop that branches on intermediate results? Then you fight Keras abstractions instead of using them.

Keras has middle ground: subclass keras.Model, override train_step. You get PyTorch-level control plus Keras metric tracking and callbacks. Most production Keras code lives here — not three-liner, not raw TensorFlow, middle ground.

Where TensorFlow Still Wins

TensorFlow stays relevant despite PyTorch research dominance for one big reason: deployment infrastructure. Took years to build. PyTorch has no equally mature equivalent.

TF Serving is production model server with automatic batching, model versioning, gRPC/REST endpoints, built-in canary deployments. Export SavedModel, TF Serving handles rest. Companies serving millions of predictions per second — Google, Airbnb, Twitter — often run TF Serving or derivatives.

TFLite compiles models for mobile and edge devices. Need model on phone with 2GB RAM and no internet? TFLite is most battle-tested path. It handles quantization (float32 to int8), operator fusion, hardware-specific acceleration (GPU delegates, NNAPI on Android, CoreML on iOS).

TensorFlow.js runs models in browser. tf.data pipelines handle datasets too large for memory with prefetching, shuffling, parallel loading. Not flashy. Core plumbing for production ML at scale.

The TF1 → TF2 baggage: You will still see TF1-style code at companies that built ML infrastructure in 2016-2019. Session-based graphs, tf.placeholder, tf.variable_scope — deprecated, but still in production at many companies. Someone hands you TF1 codebase? Do not panic. Math stays same. API changes.

Keras 3: The Plot Twist

Surprise move. In late 2023, François Chollet (Keras creator) released Keras 3. It is no longer tied to TensorFlow. Keras 3 is multi-backend. Same Keras code runs on TensorFlow, PyTorch, or JAX — pick backend with environment variable.

# Set before importing keras
import os
os.environ["KERAS_BACKEND"] = "jax"  # or "torch" or "tensorflow"

import keras
from keras import layers

model = keras.Sequential([
    layers.Dense(128, activation='relu'),
    layers.Dense(10, activation='softmax')
])

model.compile(optimizer='adam', loss='categorical_crossentropy')
model.fit(x_train, y_train, epochs=5)

Same model definition. Same compile → fit. Under hood, running on JAX's XLA compiler, PyTorch's autograd, or TensorFlow's graph executor. Idea: "write once, run anywhere" for deep learning.

Big deal. Keras is no longer "TensorFlow's high-level API" — it wants universal-interface role. Prototype in Keras, later need JAX function transforms for specific research direction? Change one line, not whole model rewrite. Company production stack uses TensorFlow but research team uses PyTorch? Keras 3 can bridge gap.

Caveat: not everything works equally well on every backend yet. Custom ops, some advanced layers, some preprocessing utilities have backend-specific limits. But core model building and training loop work, and support improves fast. Note: Keras 3 is standalone keras package — not tf.keras, which stays TensorFlow-only and mostly frozen.

JAX: Functions All the Way Down

JAX came from Google in 2018. It starts with premise that sounds almost academic: what if everything were a pure function?

Pure function gives same output for same input and has no side effects — no global state changes, no argument mutation, no printing. In regular Python, nice-to-have. In JAX, hard requirement for strongest features.

Why take that constraint? Pure functions are safe to transform. If function has no hidden dependencies and no side effects, you can safely compile it, differentiate it, vectorize it, or run it across eight TPUs — result stays correct. Deal JAX offers: give up mutation, get transformations.

Back to house analogy. In JAX, you do not build house directly. You write mathematical description of house — function from (lot, materials, blueprint) → house. Then JAX gives new functions: one computes how house changes when blueprint changes (differentiation), one builds whole neighborhood from single-house function (vectorization), one assigns different workers to different houses in parallel (device parallelism). All from same original function description.

The Four Transforms

JAX gives four function transforms. Understand these, understand why people use JAX. Build MNIST classifier piece by piece.

jax.grad takes function and returns new function that computes gradient. Not gradient value — gradient function. Biggest break from PyTorch. In PyTorch, you call loss.backward() and gradients accumulate on tensor .grad attributes. In JAX, no .backward(), no gradient attributes, no side effects. Put function in, get function out.

import jax
import jax.numpy as jnp

def mse_loss(params, x, y):
    preds = jnp.dot(x, params['w']) + params['b']
    return jnp.mean((preds - y) ** 2)

# grad returns a FUNCTION, not a value
grad_fn = jax.grad(mse_loss)

# Call the gradient function with actual data
params = {'w': jnp.ones((3,)), 'b': 0.0}
grads = grad_fn(params, x_batch, y_batch)  # x_batch, y_batch are your data arrays
# grads has the same structure as params: {'w': array(...), 'b': float}

Subtle point: grads is dictionary with exact same structure as params. JAX can differentiate through nested data structures — dicts, lists, tuples, mixes of all. These nested structures are pytrees, and JAX uses them for model parameters. No special Parameter class, no nn.Module with .parameters(). Parameters are data. Gradients are data with same shape.

jax.jit takes function and compiles it with XLA (Accelerated Linear Algebra), same compiler behind TensorFlow graph mode. First call to jit-compiled function makes JAX trace it — special tracer objects flow through function, record every op — then compile resulting computation graph. Future calls with same input shapes run compiled version directly, no Python in path.

@jax.jit
def training_step(params, x, y, lr=0.01):
    grads = jax.grad(mse_loss)(params, x, y)
    # Update params (functional style — return new params, don't mutate)
    new_params = jax.tree.map(lambda p, g: p - lr * g, params, grads)
    return new_params

# First call: traces + compiles. Subsequent calls: runs compiled code.
params = training_step(params, x_batch, y_batch)

Speed jump can be dramatic. jit-compiled JAX function can run 10-100x faster than same function without JIT, because XLA fuses ops, optimizes memory access, generates hardware-specific kernels.

jax.vmap is automatic vectorization. Write function for one example, vmap lifts it to whole batch. No loops. No manual batch-dimension handling. Very useful for per-example gradient computation — painful in PyTorch, trivial in JAX.

def predict_single(params, x):
    """Process a SINGLE input vector — no batch dimension."""
    return jnp.dot(x, params['w']) + params['b']

# vmap it: now it handles batches automatically
predict_batch = jax.vmap(predict_single, in_axes=(None, 0))
# None = don't map over params, 0 = map over first axis of x

predictions = predict_batch(params, x_batch)  # works on entire batch

jax.pmap is like vmap across devices. Write function for one GPU, pmap spreads it across all GPUs or TPUs. Each device handles slice of data. This is how Google trains models on TPU pods with hundreds of accelerators — write single-device logic, let pmap handle distribution.

Real power is composability. These transforms stack. jax.jit(jax.vmap(jax.grad(loss_fn))) gives compiled, batched gradient function in one line. You can differentiate vectorized function, compile result, distribute it across devices. Each transform stays independent and composes cleanly because everything is pure function with no hidden state to corrupt.

The Gotchas — Why JAX Isn't for Everyone

JAX constraints are not free. Here are real costs of functional purity:

No in-place mutation. JAX arrays are immutable. x[0] = 5 fails. Use x.at[0].set(5), which returns new array. Coming from NumPy or PyTorch, this feels like fighting language. It is deliberate — makes JIT compilation safe — but takes time to internalize.

Manual random state. NumPy has global random number generator. Call np.random.normal() anywhere, get random numbers. JAX has no global state (global state = side effect). Instead, pass explicit random key and split it every time you need more randomness:

key = jax.random.PRNGKey(42)
key, subkey = jax.random.split(key)
noise = jax.random.normal(subkey, shape=(3,))

# Need more random numbers? Split again.
key, subkey = jax.random.split(key)
more_noise = jax.random.normal(subkey, shape=(5,))

Reuse key without splitting, get same random numbers. By design — reproducibility is built in — but easy footgun.

Python control flow inside JIT. Because jax.jit traces function, Python if statements and for loops that depend on array values do not work as expected. They are evaluated once during tracing and baked into compiled graph. Need data-dependent branching? Use jax.lax.cond. Need data-dependent loops? Use jax.lax.scan or jax.lax.while_loop. Same tracing problem TensorFlow has with @tf.function. Similar confusing errors.

The debugging problem: print() inside jit-compiled function runs during tracing (once), not during actual computation. Debugging loss that returns NaN? print(loss) shows tracer object, not number. Use jax.debug.print() or drop @jit while debugging.

The JAX Ecosystem: Flax, Equinox, and Optax

JAX itself does not provide neural network layers. It gives arrays and transforms. Building neural network from raw JAX is like building car from raw steel — possible, educational, bad way to ship product. Libraries fill gap.

Flax is Google's official neural network library for JAX. It provides nn.Module base class with explicit parameter management. Parameters live in separate dictionary, not on module object. This feels odd if you're used to PyTorch, but fits functional framework — model definition stays separate from model parameters.

import flax.linen as nn

class MNISTClassifier(nn.Module):
    @nn.compact
    def __call__(self, x):
        x = x.reshape((x.shape[0], -1))        # flatten
        x = nn.Dense(128)(x)
        x = nn.relu(x)
        x = nn.Dense(64)(x)
        x = nn.relu(x)
        x = nn.Dense(10)(x)
        return nn.log_softmax(x)

model = MNISTClassifier()
# Initialize: pass a random key and a dummy input to discover parameter shapes
params = model.init(jax.random.PRNGKey(0), jnp.ones((1, 28, 28)))
# Forward pass: model.apply takes params explicitly
logits = model.apply(params, x_batch)

See it? model.init returns parameters as pytree, and model.apply takes them as argument. Model object itself holds no state. You can jit-compile model.apply, take gradient, vmap it — because it is pure function of (params, input).

Equinox takes different approach. Instead of separating parameters from modules, Equinox makes modules themselves pytrees. Model is dataclass whose fields are parameters and sub-modules. Feels more like PyTorch — model "has" parameters — but still works cleanly with JAX transforms because pytrees are JAX-native.

Optax is optimizer library. Instead of monolithic optimizer classes, Optax builds optimizers by chaining small transforms — scale by learning rate, apply momentum, clip gradients. Compose chain you want:

import optax

optimizer = optax.chain(
    optax.clip_by_global_norm(1.0),  # gradient clipping
    optax.adam(learning_rate=1e-3)    # Adam update
)
opt_state = optimizer.init(params)

# Training step: get grads, apply optimizer, return new params and state
grads = jax.grad(loss_fn)(params, x, y)
updates, opt_state = optimizer.update(grads, opt_state, params)
params = optax.apply_updates(params, updates)

This composability is Optax superpower. Want gradient clipping + Adam + weight decay + learning rate warmup? Chain them. Each transform stays independent and testable. Same philosophy as JAX itself — small, composable, functional pieces.

The Convergence: torch.compile and the Blurring Lines

Interesting shift happened in 2023. Lines between frameworks blurred. PyTorch 2.0 introduced torch.compile, bringing compilation into eager-execution world.

Under hood, torch.compile uses two new pieces. TorchDynamo is frontend that intercepts Python bytecode to capture computation graph — no code changes needed. TorchInductor is backend compiler that takes graph and generates optimized GPU kernels with Triton (GPU programming language) or C++/CUDA.

import torch

model = MyModel()
# One line to compile — no code changes needed
compiled_model = torch.compile(model)

# Use exactly as before
output = compiled_model(input_batch)

Key difference from JAX JIT: when Dynamo hits Python code it cannot trace (dynamic condition, unsupported operation), it falls back to eager mode for that section and keeps going. JAX would throw error. So torch.compile is more forgiving — optimize what it can, run rest normally. Trade-off: may not reach JAX-level optimization on fully traceable code.

Convergence is real. PyTorch now has compilation (like TF and JAX). TensorFlow has eager execution (like PyTorch). JAX has high-level neural network libraries (like PyTorch and Keras). Frameworks learn from each other. Gaps are smaller. But core philosophies — imperative vs. graph-compiled vs. functional — still shape how you think and write code in each one.

The Full Picture

PyTorch TensorFlow / Keras JAX
Core paradigm Imperative, eager-first, OOP. torch.compile adds optional graph compilation. Graph compilation at its core. Eager mode as default UX since TF2. Keras provides declarative high-level API. Functional, transform-based. Pure functions + composable transforms (grad, jit, vmap, pmap).
Gradient computation loss.backward() — gradients accumulate on tensor .grad attributes (imperative, stateful) tf.GradientTape context manager records ops, then tape.gradient() (imperative with explicit recording) jax.grad(fn) returns a new function (purely functional, no state)
Compilation torch.compile (Dynamo + Inductor). Graceful fallback on untraceable code. @tf.function + XLA. AutoGraph converts Python control flow. @jax.jit + XLA. Strict — requires pure functions, no side effects.
Production deployment TorchServe, ONNX export, TorchScript. Improving rapidly. TF Serving, TFLite (mobile), TF.js (browser), SavedModel. Most mature. Limited native deployment. Often export to TF SavedModel or ONNX.
Multi-device DistributedDataParallel (DDP), FSDP. Manual setup. tf.distribute.Strategy. Relatively simple API. jax.pmap / jax.sharding. First-class, especially on TPUs.
Ecosystem HuggingFace, torchvision, torchaudio. Largest open-source model ecosystem. TF Hub, tf.data, TFX pipeline. Enterprise and mobile ecosystem. Flax, Equinox, Optax, Orbax. Smaller but concentrated at Google/DeepMind.
Community (2024) ~70% of research papers. Growing in production. Declining in research, still massive in deployed production. Small but influential. Used for AlphaFold, Gemini, PaLM.

When to Reach for What

Skip diplomacy. Be direct. Every "framework comparison" article ends with "it depends" plus pros and cons. True. Not helpful. Here is better heuristic.

Default to PyTorch. Use it for research, learning, most new projects. Ecosystem is biggest. Open-source models you want are usually in PyTorch. Debugging experience (eager execution, real Python stack traces) is best. PyTorch 2.0 torch.compile narrows performance gap with compiled frameworks. Meta, Microsoft, Tesla, most AI startups use PyTorch. If you learn one framework deeply, pick this one.

Reach for TensorFlow/Keras when deploying to mobile or edge devices (TFLite has no real PyTorch equivalent at same maturity), inheriting existing TF production system (rewrite rarely worth it), or prototyping something standard and wanting fastest path from idea to trained model (Keras compile → fit is hard to beat for standard architectures). With Keras 3, you also get multi-backend flexibility — prototype in Keras, deploy on backend your infrastructure supports.

Reach for JAX when you need composable function transforms not found elsewhere (differentiate through physics simulation, compute per-example gradients efficiently, custom higher-order derivatives), you work with TPU pods (JAX has first-class TPU support through XLA), or you live in Google/DeepMind ecosystem where JAX is primary framework. AlphaFold, Gemini, PaLM all used JAX. If large-scale Google research is your world, JAX is not optional.

Best part about frameworks: once you understand one deeply, others take days, not months. Concepts — layers, losses, optimizers, gradient computation, batching, data loading — are same. What changes is expression. nn.Linear in PyTorch = keras.layers.Dense in Keras = nn.Dense in Flax. Weights are same matrix. Forward pass is same multiplication. Gradient is same derivative.

The interview question: When senior engineer asks "Why choose TensorFlow over PyTorch?" they do not want feature list. They want trade-off reasoning: graph compilation vs. eager debugging, production deployment maturity, existing infrastructure, team expertise, specific problem constraints. Best answer starts with deployment target and team stack, then gets specific.
