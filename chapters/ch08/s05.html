Gradient Management

Chapter 8: Training Deep Networks 2 subtopics
TL;DR

Gradients are the steering wheel of training, but they can jerk wildly or demand more memory than you have. Gradient clipping tames explosive updates by capping how large gradients can grow—either per-value or by scaling the entire gradient vector's norm. Gradient accumulation lets you simulate enormous batch sizes on modest hardware by summing gradients across multiple forward passes before updating weights. Together, these two techniques keep training stable and accessible, but each comes with gotchas around learning rate scaling, batch normalization, and when to actually zero out your optimizer.

Gradient Clipping

Here is the problem. You are training a recurrent network or a deep transformer, loss is dropping nicely, and then—boom—the loss spikes to infinity and your weights are filled with NaN. What happened? A single unlucky batch produced a gradient so large that one optimizer step catapulted your parameters into nonsense territory. This is the exploding gradient problem, and it is the primary reason gradient clipping exists.

The fix is surprisingly blunt: before you hand the gradients to your optimizer, you check how big they are. If they exceed some threshold, you shrink them. The model still moves in the same direction—you are not changing where the gradient points—you are just limiting how far it can push in a single step. Think of it like a speed limiter on a car. The steering wheel still works, but the car cannot suddenly accelerate to 300 km/h because of one steep hill.

There are three common flavors, and each makes a different tradeoff between simplicity and preserving the relative magnitudes across parameters.

Clip by Value

Clip by value is the most naive approach. You pick a threshold—say 1.0—and every individual gradient element that exceeds that threshold gets clamped to it. If a gradient element is 47.3, it becomes 1.0. If it is -200.0, it becomes -1.0. Elements already within [-1.0, 1.0] are left alone.

import torch

# Manual clip by value
for param in model.parameters():
    if param.grad is not None:
        param.grad.data.clamp_(-1.0, 1.0)

# Or use the built-in utility
torch.nn.utils.clip_grad_value_(model.parameters(), clip_value=1.0)

The problem is obvious once you think about it. Suppose one layer has gradients around 0.01 and another has gradients around 100.0. After clipping at 1.0, the small layer is untouched but the large layer has every single element squashed to exactly 1.0 or -1.0. You have destroyed the relative information within that layer's gradient—the direction is mangled into a grid of ±1 values. In practice, clip by value is rarely used for serious training. It is fine for quick experiments or toy models, but it distorts gradient geometry too aggressively.

Clip by Norm

Clip by norm takes a smarter approach. Instead of clamping individual elements, you look at the L2 norm (the length) of each parameter's gradient tensor. If that norm exceeds your threshold, you scale the entire tensor down so its norm equals the threshold. Critically, this preserves the direction of the gradient within each parameter—every element gets multiplied by the same scalar.

# Clip each parameter's gradient independently by its own norm
max_norm = 1.0
for param in model.parameters():
    if param.grad is not None:
        norm = param.grad.data.norm(2)
        if norm > max_norm:
            param.grad.data.mul_(max_norm / norm)

This is better than clip by value because the relative magnitudes within each tensor are preserved. But it still treats each parameter independently. One layer might get clipped while another does not, which can distort the relative update sizes across layers. For many architectures, this is good enough. For the most careful training runs, though, people reach for global norm clipping.

Global Norm Clipping

Global norm clipping is what you will see in virtually every modern large-model training codebase. The idea: compute a single norm across all parameters in the entire model, then scale all gradients by the same factor if that global norm exceeds the threshold.

# PyTorch's built-in global norm clipping
max_norm = 1.0
total_norm = torch.nn.utils.clip_grad_norm_(
    model.parameters(),
    max_norm=max_norm,
    norm_type=2  # L2 norm (default)
)
print(f"Global gradient norm: {total_norm:.4f}")

Under the hood, PyTorch computes the square root of the sum of squared norms across every parameter tensor. If that total exceeds max_norm, every gradient in the model is scaled by max_norm / total_norm. If the total is already below the threshold, nothing happens.

Why is this superior? Because it preserves the relative gradient magnitudes both within each tensor and across all tensors. The entire gradient vector gets uniformly shortened, so the optimizer step still points in the exact same direction in parameter space—it just takes a shorter step. This is why papers like the original GPT-2 and GPT-3 training reports all specify a global norm clip value (typically between 0.5 and 1.0).

💡 Key Insight

The return value of clip_grad_norm_ is the total norm before clipping. Logging this value over training is one of the best diagnostic tools you have. A steadily increasing gradient norm often signals instability before the loss itself shows any sign of trouble. If you see the norm spiking, you can lower the learning rate or tighten the clip before your run explodes.

A practical workflow looks like this: set your clip threshold to 1.0, log the gradient norm every step, and inspect the distribution after a few hundred iterations. If the norm is almost always below 1.0, your threshold is not doing anything—which is fine. If it is clipping on 80% of steps, you might have a learning rate that is too high or an architecture that is struggling to converge. Clipping should be a safety net, not a crutch.

optimizer.zero_grad()
loss = compute_loss(model, batch)
loss.backward()

# Clip and log
grad_norm = torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
if grad_norm > 1.0:
    print(f"Clipped! Original norm: {grad_norm:.2f}")

optimizer.step()

Gradient Accumulation

Now for a completely different problem. You want to train with a batch size of 256 because larger batches give you smoother gradient estimates and better utilization of your GPU. But your model is so large that a batch size of 8 already fills your GPU memory. You cannot fit 256 samples through the model at once without running out of VRAM. What do you do?

Gradient accumulation is the answer. The core insight is that gradients are additive. If you run two forward-backward passes on two different mini-batches and add their gradients together, the result is mathematically identical to running a single forward-backward pass on both mini-batches combined (assuming you average properly). So instead of calling optimizer.step() after every backward pass, you call it after every N backward passes. During those N passes, the gradients just pile up in the .grad attributes.

Simulating Large Batches

Here is the simplest version. Suppose your GPU can handle a micro-batch of 8 samples, and you want an effective batch size of 64. You set your accumulation steps to 64 / 8 = 8. Every 8 forward-backward passes, you take one optimizer step.

accumulation_steps = 8
optimizer.zero_grad()

for i, batch in enumerate(dataloader):
    loss = compute_loss(model, batch)
    # Scale loss so accumulated gradients average correctly
    loss = loss / accumulation_steps
    loss.backward()

    if (i + 1) % accumulation_steps == 0:
        optimizer.step()
        optimizer.zero_grad()

The division by accumulation_steps is essential. Most loss functions return the mean loss over the micro-batch. If you just sum up 8 of those means, your effective gradient is 8× larger than what you would get from a true batch of 64. Dividing the loss by the number of accumulation steps ensures the final accumulated gradient matches the true large-batch gradient.

Memory Efficiency

Why does this work without using more memory? Because loss.backward() computes gradients and then immediately discards the computation graph and all intermediate activations. Only the .grad tensors remain, and those are the same size regardless of batch size—they have the shape of the parameters, not the shape of the data. So each micro-batch forward-backward pass uses the same peak memory as training with a batch of 8. You just do it more times before calling optimizer.step().

The tradeoff is wall-clock time. With true large batches, those 8 micro-batches would run in parallel on the GPU. With accumulation, they run sequentially. You get the same gradient quality, but it takes 8× longer per optimizer step. This is the fundamental deal: gradient accumulation trades time for memory. On a single GPU, there is no way around this. With multiple GPUs, you get the best of both worlds—accumulation across time and parallelism across devices.

Effective Batch Size Calculation

In distributed training, the effective batch size is the total number of samples that contribute to one optimizer step. The formula is straightforward:

effective_batch_size = accumulation_steps × micro_batch_size × num_gpus

# Example: 4 accumulation steps, 16 samples per GPU, 8 GPUs
# effective_batch_size = 4 × 16 × 8 = 512

This matters because learning rate and batch size are deeply intertwined. The most widely used heuristic is the linear scaling rule: if you multiply the batch size by k, multiply the learning rate by k. The intuition is that a larger batch gives a more accurate gradient estimate, so you can afford to take a bigger step. If your base recipe uses a batch size of 64 with a learning rate of 0.001, and you scale up to an effective batch size of 512 (8× bigger), you should try a learning rate of 0.008.

This is not a law of physics—it is a heuristic that works well in practice for moderate scaling factors (up to about 8–16×). Beyond that, you often need learning rate warmup, where you start with a small learning rate and ramp up to the target over the first few hundred steps. The LARS and LAMB optimizers were specifically designed to handle very large batch training by adapting the learning rate per layer, but that is a topic for another section.

# Complete training loop with gradient accumulation and clipping
accumulation_steps = 4
micro_batch_size = 16
num_gpus = 1  # single GPU example
effective_batch = accumulation_steps * micro_batch_size * num_gpus

base_lr = 1e-3
base_batch = 64
scaled_lr = base_lr * (effective_batch / base_batch)

optimizer = torch.optim.AdamW(model.parameters(), lr=scaled_lr)
optimizer.zero_grad()

for step, batch in enumerate(dataloader):
    loss = compute_loss(model, batch)
    loss = loss / accumulation_steps
    loss.backward()

    if (step + 1) % accumulation_steps == 0:
        torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
        optimizer.step()
        optimizer.zero_grad()
💡 Key Insight

Always apply gradient clipping after accumulation is complete but before the optimizer step. If you clip after each micro-batch, you are clipping partial gradients—the accumulated result will have a different magnitude than if you had clipped the full accumulated gradient. The correct order is: accumulate all micro-batches → clip → step → zero grad.

Gradient Accumulation Pitfalls

Gradient accumulation seems deceptively simple. Divide the loss, accumulate, step. But there are two common pitfalls that can silently degrade your results.

The first is batch normalization statistics mismatch. Batch normalization computes running mean and variance from each micro-batch independently during the forward pass. If your micro-batch size is 8, BatchNorm sees batches of 8—not your effective batch of 64. This means the normalization statistics are noisier than they would be with a true batch of 64, and the running statistics tracked for inference are computed from small batches. For convolutional networks that rely heavily on BatchNorm, this can meaningfully hurt performance. The workarounds are to use Group Normalization or Layer Normalization instead (both are independent of batch size), or to use synchronized BatchNorm across accumulation steps (which is complex and rarely done). This is one reason why modern transformer architectures almost universally use LayerNorm—it sidesteps this entire problem.

⚠️ Warning

If your model uses BatchNorm layers and you rely on gradient accumulation, the normalization statistics will be computed on your micro-batch size, not your effective batch size. This can cause a noticeable gap between training and inference behavior. Prefer LayerNorm or GroupNorm when using gradient accumulation.

The second pitfall is forgetting to adjust the learning rate. If you switch from a batch size of 64 to accumulating 8 micro-batches of 8 (same effective batch size of 64), no learning rate change is needed—the gradients are mathematically identical. But if you are increasing the effective batch size beyond your original recipe, you need to scale the learning rate accordingly. A common mistake is to add accumulation steps to "improve training stability" without realizing that you have also changed the effective batch size, which demands a learning rate adjustment. The symptoms are subtle: training still converges, but slower than expected, and final accuracy is a fraction of a percent worse. Always compute your effective batch size explicitly and compare it to the reference recipe.

There is one more subtlety worth mentioning. Some training frameworks—particularly those with learning rate schedulers tied to step counts—need you to decide whether a "step" means a micro-batch forward pass or an optimizer update. If your scheduler expects 1000 steps and you have 4 accumulation steps, you need 4000 micro-batch iterations to complete training. Getting this wrong means your learning rate schedule runs 4× too fast or too slow, which can be the difference between a good model and a mediocre one. Always verify that your scheduler counts optimizer steps, not data batches.

# Pitfall demo: scheduler must count optimizer steps, not micro-batches
accumulation_steps = 4
total_optimizer_steps = 1000

scheduler = torch.optim.lr_scheduler.CosineAnnealingLR(
    optimizer, T_max=total_optimizer_steps
)

optimizer_step_count = 0
for i, batch in enumerate(dataloader):
    loss = compute_loss(model, batch)
    loss = loss / accumulation_steps
    loss.backward()

    if (i + 1) % accumulation_steps == 0:
        torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
        optimizer.step()
        scheduler.step()  # only advance scheduler on optimizer steps
        optimizer.zero_grad()
        optimizer_step_count += 1

    if optimizer_step_count >= total_optimizer_steps:
        break

Gradient clipping and gradient accumulation are not glamorous. They do not make headlines like new architectures or clever attention mechanisms. But they are the kind of infrastructure that makes or breaks real training runs. Clip your gradients to keep training stable. Accumulate your gradients to train big models on small hardware. Know the pitfalls—BatchNorm, learning rate scaling, scheduler step counting—and you will avoid days of debugging mysterious convergence failures. These are the tools that separate someone who has read about training from someone who has actually shipped a model.
