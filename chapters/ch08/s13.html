Training Debugging Methodology

Chapter 8: Training Deep Networks Section 13 of 16 5 subtopics
TL;DR

When your model won't train, resist the urge to randomly tweak hyperparameters. Instead, follow a systematic debugging checklist: verify your data pipeline, overfit one batch to confirm the model can learn at all, check gradient flow for dead or exploding layers, sweep the learning rate to find a sane range, and validate your loss function matches your labels. Most training failures trace back to simple bugs — wrong data types, misaligned labels, or a forgotten normalization step. This section gives you a concrete, repeatable methodology to diagnose and fix every common training failure.

Why You Need a Debugging System

Here's what happens when a model won't train: you stare at a flat loss curve, feel a rising panic, and start randomly changing things. You drop the learning rate. You add layers. You switch optimizers. Nothing helps, because you're treating symptoms instead of finding the root cause.

Training failures have causes. Those causes are discoverable. But you need a system — a checklist you follow every single time, in order, from the most likely culprits to the rarest edge cases. The debugging methodology in this section has saved me hundreds of hours. It catches about 95% of training failures in the first three steps.

The core principle is simple: start from the data, move toward the model. Most bugs live in the data pipeline, not in the architecture. By the time you're debugging gradient flow through residual connections, you've already eliminated the boring-but-deadly mistakes that cause the majority of training failures.

The Systematic Debugging Checklist

Step 1: Data Pipeline Verification

Before you even look at the model, verify that the data reaching it is correct. This step catches more bugs than all other steps combined.

First, visually inspect your batches. For image tasks, render the actual tensors your data loader produces — after all transforms, augmentations, and normalization. Do the images look right? Are labels aligned with the correct images? For text tasks, decode the tokenized inputs back to strings. Do they make sense?

# Visual sanity check for image data
import matplotlib.pyplot as plt

batch = next(iter(train_loader))
images, labels = batch

# Undo normalization for visualization
mean = torch.tensor([0.485, 0.456, 0.406]).view(3, 1, 1)
std = torch.tensor([0.229, 0.224, 0.225]).view(3, 1, 1)

fig, axes = plt.subplots(2, 4, figsize=(12, 6))
for i, ax in enumerate(axes.flat):
    img = images[i] * std + mean  # Denormalize
    ax.imshow(img.permute(1, 2, 0).clip(0, 1))
    ax.set_title(f"Label: {labels[i].item()}")
    ax.axis('off')
plt.tight_layout()
plt.savefig("batch_sanity_check.png")

Second, check for label correctness. Count the distribution of labels in your training set. Does it match what you expect? Are there classes with zero samples? Are labels 0-indexed or 1-indexed? A surprisingly common bug is labels that start at 1 when your model expects labels starting at 0 — this silently shifts every prediction by one class.

# Label distribution check
from collections import Counter

all_labels = []
for _, labels in train_loader:
    all_labels.extend(labels.numpy().tolist())

dist = Counter(all_labels)
print(f"Label range: [{min(dist.keys())}, {max(dist.keys())}]")
print(f"Num classes found: {len(dist)}")
print(f"Distribution: {sorted(dist.items())}")

# Sanity: does num_classes match model output dim?
assert max(dist.keys()) < model.fc.out_features, \
    f"Label {max(dist.keys())} >= output dim {model.fc.out_features}!"

Third, verify data types and shapes. Is your input float32, not float64 or int? Are image dimensions in the right order (channels-first for PyTorch, channels-last for TensorFlow)? Is your target the right dtype — long for classification, float for regression? These silent type mismatches can cause the loss to return nonsense without any error message.

Step 2: The Overfit-One-Batch Test

This is the single most valuable debugging test in all of deep learning. Take one small batch — say 8 or 16 samples — and train on it for hundreds of iterations without any regularization. Your model should drive the training loss to near zero.

Why? Because if a model can't memorize 8 examples, something is fundamentally broken. The architecture, the loss function, the optimizer — something isn't working. This test isolates the learning machinery from all the noise of real training (data augmentation, regularization, large datasets).

# The overfit-one-batch test
model = MyModel().to(device)
optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)

# Grab one batch and reuse it
fixed_batch = next(iter(train_loader))
inputs, targets = fixed_batch[0].to(device), fixed_batch[1].to(device)

# Disable all regularization
model.train()
for step in range(500):
    optimizer.zero_grad()
    outputs = model(inputs)
    loss = criterion(outputs, targets)
    loss.backward()
    optimizer.step()
    
    if step % 50 == 0:
        # For classification, check accuracy too
        preds = outputs.argmax(dim=-1)
        acc = (preds == targets).float().mean()
        print(f"Step {step}: loss={loss.item():.4f}, acc={acc.item():.2%}")

# After 500 steps, loss should be near 0 and accuracy near 100%

If the overfit-one-batch test fails, the problem is in your model, loss function, or optimizer — not in your data. Common causes: the model has too few parameters to fit even 8 examples, the loss function doesn't match the task (using MSE for classification), or the learning rate is so wrong that no learning occurs at all.

If the test passes, you know the core machinery works. Any further training failures must come from the data pipeline, regularization settings, or scaling issues.

Step 3: Gradient Flow Analysis

Gradient flow analysis means checking whether gradients are actually reaching every layer of your network during backpropagation. Dead gradients mean dead layers — parameters that never update, wasting capacity.

The classic technique is to plot the mean absolute gradient for every layer after a backward pass. Healthy networks show gradients that are roughly the same order of magnitude across layers. Sick networks show gradients that collapse to zero in early layers (vanishing gradients) or explode to huge values in certain layers (exploding gradients).

# Gradient flow visualization
import matplotlib.pyplot as plt
import numpy as np

def plot_gradient_flow(named_parameters):
    """Plot gradient magnitudes across layers."""
    ave_grads, max_grads, layers = [], [], []
    
    for name, param in named_parameters:
        if param.requires_grad and param.grad is not None:
            layers.append(name)
            ave_grads.append(param.grad.abs().mean().item())
            max_grads.append(param.grad.abs().max().item())
    
    plt.figure(figsize=(14, 5))
    plt.bar(range(len(max_grads)), max_grads, alpha=0.3, color='c', label='max')
    plt.bar(range(len(ave_grads)), ave_grads, alpha=0.7, color='b', label='mean')
    plt.xticks(range(len(layers)), layers, rotation=90, fontsize=7)
    plt.xlabel("Layers")
    plt.ylabel("Gradient magnitude")
    plt.title("Gradient Flow")
    plt.legend()
    plt.yscale('log')
    plt.tight_layout()
    plt.savefig("gradient_flow.png")

# Usage: run one forward-backward pass, then call
loss.backward()
plot_gradient_flow(model.named_parameters())

What to look for: if early layers show gradients many orders of magnitude smaller than later layers, you have a vanishing gradient problem. Solutions include residual connections, better initialization (Kaiming/Xavier), or switching from sigmoid/tanh to ReLU activations. If any layer shows exactly zero gradients, check whether you accidentally froze that layer, used an in-place operation that broke the computation graph, or hit a dead ReLU problem.

Step 4: Learning Rate Sanity Check

The learning rate is the single most important hyperparameter. Too high, and the loss diverges or oscillates wildly. Too low, and training appears stuck — the loss decreases so slowly it looks flat.

The learning rate finder (popularized by Leslie Smith) is a quick diagnostic. You start with a tiny learning rate (say 1e-7) and exponentially increase it over one epoch. You plot loss vs. learning rate on a log scale. The ideal learning rate is roughly one order of magnitude before the loss starts exploding.

# Learning rate finder
def lr_finder(model, train_loader, criterion, device,
              start_lr=1e-7, end_lr=10, num_steps=200):
    """Sweep learning rate and record loss."""
    model_state = {k: v.clone() for k, v in model.state_dict().items()}
    optimizer = torch.optim.SGD(model.parameters(), lr=start_lr)
    
    lr_mult = (end_lr / start_lr) ** (1 / num_steps)
    lrs, losses = [], []
    smooth_loss = None
    
    data_iter = iter(train_loader)
    for step in range(num_steps):
        try:
            inputs, targets = next(data_iter)
        except StopIteration:
            data_iter = iter(train_loader)
            inputs, targets = next(data_iter)
        
        inputs, targets = inputs.to(device), targets.to(device)
        optimizer.zero_grad()
        output = model(inputs)
        loss = criterion(output, targets)
        
        # Smooth the loss for plotting
        if smooth_loss is None:
            smooth_loss = loss.item()
        else:
            smooth_loss = 0.9 * smooth_loss + 0.1 * loss.item()
        
        lrs.append(optimizer.param_groups[0]['lr'])
        losses.append(smooth_loss)
        
        # Stop if loss explodes
        if smooth_loss > 4 * losses[0]:
            break
        
        loss.backward()
        optimizer.step()
        
        # Increase learning rate
        for pg in optimizer.param_groups:
            pg['lr'] *= lr_mult
    
    # Restore model
    model.load_state_dict(model_state)
    
    plt.figure(figsize=(10, 4))
    plt.plot(lrs, losses)
    plt.xscale('log')
    plt.xlabel('Learning Rate')
    plt.ylabel('Smoothed Loss')
    plt.title('LR Finder — pick LR ~10x before the minimum')
    plt.savefig("lr_finder.png")
    return lrs, losses

A typical curve drops steadily, hits a minimum, then shoots upward. Pick a learning rate about 10x smaller than the one at the minimum. If the curve never drops at all, your model has a deeper problem — go back to the overfit-one-batch test.

Step 5: Loss Function Correctness

Verify that your loss function matches your task and your label format. This sounds trivial, but it's a common source of confusion:

For multi-class classification, PyTorch's CrossEntropyLoss expects raw logits (not softmax outputs) and integer class labels. If you apply softmax before passing to CrossEntropyLoss, you're applying softmax twice — the loss contains a log-softmax internally. The model will still train, but much worse than it should.

For binary classification, make sure you're using BCEWithLogitsLoss (which includes the sigmoid) rather than BCELoss with a manual sigmoid. The fused version is numerically more stable.

For regression, check that your targets are the same scale as your model's outputs. If targets are in the millions and your model outputs values near zero, the loss will be enormous and gradients will explode. Normalize your targets.

# Common loss function mistakes and fixes

# WRONG: double softmax
probs = torch.softmax(logits, dim=-1)
loss = F.cross_entropy(probs, targets)  # Bug! CE already does log_softmax

# CORRECT: raw logits to cross entropy
loss = F.cross_entropy(logits, targets)

# WRONG: BCELoss with manual sigmoid is numerically unstable
probs = torch.sigmoid(logits)
loss = F.binary_cross_entropy(probs, targets.float())

# CORRECT: fused version
loss = F.binary_cross_entropy_with_logits(logits, targets.float())

# WRONG: regression targets not normalized
# targets are house prices in [100_000, 5_000_000]
loss = F.mse_loss(model(x), targets)  # Huge loss, exploding grads

# CORRECT: normalize targets
target_mean, target_std = targets.mean(), targets.std()
normalized_targets = (targets - target_mean) / target_std
loss = F.mse_loss(model(x), normalized_targets)

Common Training Failures and Fixes

Training Loss Not Decreasing

This is the most common complaint. The loss stays flat or bounces around a constant value. Work through these causes in order:

Learning rate too low. The model is learning, but so slowly it appears stuck. Run the LR finder. Try increasing the learning rate by 10x.

Learning rate too high. The loss oscillates around a high value without trending downward. The optimizer overshoots the minima every step. Try decreasing by 10x.

Bug in the training loop. This is more common than you'd think. Did you forget optimizer.zero_grad()? Gradients accumulate across batches, and the effective gradient becomes nonsense. Did you forget loss.backward()? Did you accidentally call model.eval() before training starts (disabling dropout and changing batch norm behavior)?

Frozen parameters. If you loaded a pretrained model and forgot to unfreeze the layers you want to train, those parameters have requires_grad=False and will never update. Check with:

# Check which parameters are actually being trained
trainable = sum(p.numel() for p in model.parameters() if p.requires_grad)
total = sum(p.numel() for p in model.parameters())
print(f"Trainable: {trainable:,} / {total:,} ({trainable/total:.1%})")

# If trainable is 0, nothing will learn!
for name, param in model.named_parameters():
    if not param.requires_grad:
        print(f"  FROZEN: {name}")

Model Stuck at Random Baseline

For a 10-class classification problem, random guessing gives 10% accuracy. If your model stays at exactly 10%, the logits are probably nearly uniform — the model isn't learning to discriminate. This is different from "loss not decreasing" because the loss might decrease slightly (the model learns the class distribution) but accuracy stays at chance level.

Common causes: the learning rate is way too small for the model size, there's a major data pipeline bug (labels are random or disconnected from inputs), or the model architecture has a bottleneck so severe that information can't flow through it — for example, a single hidden unit between large layers.

Validation Loss Diverges from Training Loss

Training loss goes down, but validation loss goes up or stays flat. This is classic overfitting. But the question is: how severe is it, and what's the right fix?

If the gap appears after many epochs, you're training too long — add early stopping. If the gap appears almost immediately, your model has too much capacity for the data — add dropout, reduce model size, or add data augmentation. If validation loss is noisy but trending upward while training loss is smooth and trending downward, check that your validation set is representative and large enough.

Loss Returns NaN or Inf

NaN in the loss means a numerical operation produced an undefined value. The most common sources:

Taking the log of zero or a negative number. If your model outputs probabilities and one prediction is exactly 0, then log(0) = -inf, and any arithmetic with inf produces NaN. Fix: add a small epsilon: torch.log(probs + 1e-8).

Division by zero. If you're normalizing by the count of non-padding tokens or non-zero elements, an empty batch (all padding) divides by zero. Fix: add a clamp: count.clamp(min=1).

Exploding gradients causing overflow in float16. The loss itself might be finite, but after multiplication with very large gradients, the parameter update overflows. Fix: gradient clipping, or switch to bfloat16 which has the same exponent range as float32.

# NaN debugging toolkit
def check_for_nans(model, loss, step):
    """Call after loss.backward() to detect NaN early."""
    if torch.isnan(loss) or torch.isinf(loss):
        print(f"Step {step}: Loss is {loss.item()}")
        return True
    
    for name, param in model.named_parameters():
        if param.grad is not None:
            if torch.isnan(param.grad).any():
                print(f"Step {step}: NaN gradient in {name}")
                return True
            if torch.isinf(param.grad).any():
                print(f"Step {step}: Inf gradient in {name}")
                return True
    return False

# Use PyTorch's anomaly detection for detailed traceback
torch.autograd.set_detect_anomaly(True)  # Slow! Only for debugging

Common Bugs: The Boring Stuff That Wastes Hours

Data Loading Bugs

The most insidious bugs live in data loading because they don't cause errors — they cause silently wrong behavior.

Wrong normalization stats. You're using ImageNet mean/std but your data is medical images with completely different pixel distributions. The model can still learn, but it has to waste capacity undoing wrong normalization. Fix: compute the actual mean and std of your dataset.

Augmentation applied to validation set. Random cropping, flipping, and color jittering should only happen during training. If you apply them to validation, your validation metrics will be noisy and unreliable. In PyTorch, make separate transforms for train and val.

Data loader not shuffling. If your data is sorted by class (all cats, then all dogs, then all birds), and you don't shuffle, each batch contains only one class. Batch normalization statistics will be wildly wrong, and the model sees a non-representative gradient at each step. Always set shuffle=True for training loaders.

Num workers and prefetching. Setting num_workers=0 means data loading is synchronous — the GPU idles while waiting for the next batch. But setting num_workers too high can cause memory issues or race conditions. Start with num_workers=4 and increase until GPU utilization is high.

Label Mismatch Bugs

Off-by-one label indexing. Your label file uses 1-indexed classes (1, 2, 3, ..., N) but PyTorch's CrossEntropyLoss expects 0-indexed (0, 1, 2, ..., N-1). The model still trains — it just predicts the wrong class for every input. This bug is invisible unless you manually verify predictions on a few known examples.

Swapped labels. If you reshuffle your data after separating features from labels — for example, by loading features and labels from different files that got reordered — every sample now has the wrong label. The model appears to train (loss decreases a bit) but accuracy stays near random.

Multi-label vs multi-class confusion. In multi-class problems, each sample has exactly one label. In multi-label problems, each sample can have multiple labels. If you use CrossEntropyLoss (multi-class) for a multi-label problem, you're training the model to pick one label when it should output multiple. Use BCEWithLogitsLoss for multi-label.

Monitoring and Visualization

What to Log During Training

At minimum, log these metrics at every step or every few steps:

Training loss (per step, smoothed with exponential moving average). Validation loss (every epoch or every N steps). Learning rate (especially with schedules — verify the schedule is doing what you think). Gradient norm (the global L2 norm of all gradients — a sudden spike means something went wrong). GPU utilization and memory (if utilization is low, your data pipeline is the bottleneck, not the model).

# Compact training loop with logging
import torch
from torch.utils.tensorboard import SummaryWriter

writer = SummaryWriter("runs/experiment_01")

for epoch in range(num_epochs):
    model.train()
    for step, (inputs, targets) in enumerate(train_loader):
        inputs, targets = inputs.to(device), targets.to(device)
        
        optimizer.zero_grad()
        outputs = model(inputs)
        loss = criterion(outputs, targets)
        loss.backward()
        
        # Log gradient norm before clipping
        grad_norm = torch.nn.utils.clip_grad_norm_(
            model.parameters(), max_norm=1.0
        )
        optimizer.step()
        scheduler.step()
        
        global_step = epoch * len(train_loader) + step
        writer.add_scalar("train/loss", loss.item(), global_step)
        writer.add_scalar("train/grad_norm", grad_norm.item(), global_step)
        writer.add_scalar("train/lr", optimizer.param_groups[0]['lr'], global_step)
    
    # Validation
    model.eval()
    val_loss = 0
    with torch.no_grad():
        for inputs, targets in val_loader:
            inputs, targets = inputs.to(device), targets.to(device)
            outputs = model(inputs)
            val_loss += criterion(outputs, targets).item()
    val_loss /= len(val_loader)
    writer.add_scalar("val/loss", val_loss, epoch)

Experiment Tracking with Weights & Biases

Weights & Biases (W&B) is the most popular experiment tracker in practice. It logs metrics, hyperparameters, code, system metrics, and model artifacts to a cloud dashboard. The integration is minimal — a few lines of code.

import wandb

wandb.init(project="my-project", config={
    "learning_rate": 1e-3,
    "batch_size": 32,
    "architecture": "ResNet50",
    "epochs": 100,
})

for epoch in range(num_epochs):
    train_loss = train_one_epoch(model, train_loader)
    val_loss, val_acc = evaluate(model, val_loader)
    
    wandb.log({
        "train/loss": train_loss,
        "val/loss": val_loss,
        "val/accuracy": val_acc,
        "epoch": epoch,
    })

wandb.finish()

The killer feature is comparison. Once you've logged 50 experiments with different hyperparameters, you can sort, filter, and plot them side by side. You can create parallel coordinate plots to see which hyperparameter ranges work best. And because every run logs the exact code and config, you can always reproduce a good result.

Activation and Weight Distribution Monitoring

Beyond loss curves, monitoring the distribution of activations and weights across layers can reveal problems before they show up in the loss. If activations in a layer are collapsing to zero, that layer is dying. If weights are growing unbounded, the model is heading toward instability.

# Log activation and weight histograms to TensorBoard
def log_histograms(model, writer, step):
    for name, param in model.named_parameters():
        writer.add_histogram(f"weights/{name}", param.data, step)
        if param.grad is not None:
            writer.add_histogram(f"gradients/{name}", param.grad, step)

# Hook to capture activations
activations = {}
def get_activation(name):
    def hook(module, input, output):
        activations[name] = output.detach()
    return hook

# Register hooks on key layers
model.layer1.register_forward_hook(get_activation('layer1'))
model.layer2.register_forward_hook(get_activation('layer2'))

# After a forward pass:
for name, act in activations.items():
    writer.add_histogram(f"activations/{name}", act, step)
    # Check for dead neurons
    dead_frac = (act == 0).float().mean()
    writer.add_scalar(f"dead_neurons/{name}", dead_frac, step)

The Complete Debugging Flowchart

When training goes wrong, follow this exact order:

  1. Visualize your data. Render actual training batches. Check labels. Verify shapes and dtypes. This catches 40% of bugs.
  2. Overfit one batch. If it fails: the bug is in model, loss, or optimizer. If it passes: the bug is in data, regularization, or scale.
  3. Check gradient flow. Are gradients reaching all layers? Are any layers dead? Are gradients exploding?
  4. Run the LR finder. Is your learning rate in a sane range? Does the loss ever decrease at any learning rate?
  5. Verify the loss function. Does it match your task? Are you accidentally applying softmax twice? Are targets the right format?
  6. Simplify everything. Remove augmentation, regularization, fancy scheduling. Train a minimal setup. Add complexity back one piece at a time.
  7. Compare with a known baseline. Run a reference implementation on the same data. If the reference works and yours doesn't, the bug is in your code, not the problem.

Most training bugs are boring. They're off-by-one errors, forgotten function calls, and wrong data types. The methodology above is boring too — and that's exactly why it works. It replaces panic with a checklist, and checklists catch bugs that intuition misses.
