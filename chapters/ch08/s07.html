PyTorch

Chapter 8: Training Deep Networks 7 subtopics
TL;DR

PyTorch is a Python-first deep learning framework built around tensors that track their own gradients. You define models as plain Python classes, write training loops explicitly, and debug with ordinary print statements. This section walks through every layer of the stack — tensors, automatic differentiation, the module system, data loading, the training loop, model compilation and export, and reproducibility — with runnable code for each. By the end you will understand not just the API calls, but why each piece exists and what tradeoffs it makes.

Tensors

Everything in PyTorch starts with the tensor. A tensor is an n-dimensional array, similar to a NumPy ndarray, but with two superpowers: it can live on a GPU and it can remember every operation you perform on it so that gradients flow backward automatically. If you understand NumPy arrays, you already understand 80% of tensors. The remaining 20% is what makes deep learning work.

Creation and Operations

You can create tensors from Python lists, NumPy arrays, or with factory functions. The most common factory functions give you zeros, ones, random values, or ranges. Operations look exactly like NumPy — addition, matrix multiplication, slicing, reshaping — because the PyTorch team deliberately kept the API familiar.

import torch
import numpy as np

# From a Python list
a = torch.tensor([1.0, 2.0, 3.0])

# From NumPy (shares memory — changes to one affect the other)
np_arr = np.array([4.0, 5.0, 6.0])
b = torch.from_numpy(np_arr)

# Factory functions
zeros = torch.zeros(3, 4)          # 3x4 matrix of zeros
ones = torch.ones(2, 3)            # 2x3 matrix of ones
rand = torch.randn(5, 5)           # 5x5 matrix, standard normal
ids = torch.eye(3)                 # 3x3 identity matrix
rng = torch.arange(0, 10, 2)      # tensor([0, 2, 4, 6, 8])

# Arithmetic — elementwise by default
c = a + b                          # tensor([5., 7., 9.])
d = a * b                          # tensor([4., 10., 18.])

# Matrix multiply
W = torch.randn(3, 4)
x = torch.randn(4)
y = W @ x                          # shape (3,) — standard matmul

One subtle but important detail: torch.from_numpy shares memory with the original array. If you modify the NumPy array, the tensor changes too. Use torch.tensor(np_arr) instead if you want an independent copy.

GPU Transfer

Tensors live on a device — either CPU or a specific GPU. Moving data to the GPU is how you unlock the massive parallelism of modern hardware. The API is a single method call: .to(device).

# Pick GPU if available, otherwise CPU
device = torch.device("cuda" if torch.cuda.is_available() else "cpu")

x = torch.randn(1000, 1000)
x_gpu = x.to(device)              # now lives on the GPU

# All operations on GPU tensors happen on the GPU
y_gpu = x_gpu @ x_gpu.T           # matrix multiply on GPU

# Move back to CPU (e.g., for NumPy interop)
y_cpu = y_gpu.cpu()
y_np = y_cpu.numpy()
⚠️ Warning

Every .to(device) call copies data across the PCIe bus, which is slow. Do not move tensors back and forth inside a training loop. Move your model and data to the GPU once, do all computation there, and only move results back when you need them for logging or saving.

Tensor Shapes and Broadcasting

Shape bugs are the most common source of errors in deep learning code. A tensor's shape is a tuple of integers describing its dimensions. A vector has shape (n,), a matrix has shape (rows, cols), and a batch of images might have shape (batch, channels, height, width). You reshape tensors with .view() or .reshape(), add dimensions with .unsqueeze(), and remove size-one dimensions with .squeeze().

Broadcasting is the rule that lets PyTorch operate on tensors of different shapes without explicit copying. When you add a vector of shape (3,) to a matrix of shape (4, 3), PyTorch silently expands the vector across rows. The rules match NumPy: dimensions are compared right-to-left, and each pair must either be equal or one of them must be 1.

# Reshape
x = torch.arange(12)              # shape (12,)
x = x.view(3, 4)                  # shape (3, 4)
x = x.reshape(4, 3)               # shape (4, 3) — same data, new view

# Add/remove dimensions
x = torch.randn(3, 4)             # shape (3, 4)
x = x.unsqueeze(0)                # shape (1, 3, 4) — added batch dim
x = x.squeeze(0)                  # shape (3, 4) — removed it

# Broadcasting example
bias = torch.tensor([1.0, 2.0, 3.0, 4.0])   # shape (4,)
mat = torch.randn(3, 4)                       # shape (3, 4)
result = mat + bias                            # bias broadcast to (3, 4)

Tensor Datatypes

Every tensor has a dtype. The default for floating-point tensors is torch.float32, which gives you about 7 decimal digits of precision and is the workhorse dtype for training. You will also encounter torch.float16 (half precision, used in mixed-precision training), torch.bfloat16 (same exponent range as float32 but less mantissa precision — popular on TPUs and modern GPUs), torch.float64 (double precision, rarely needed), and torch.int64 (for indices and labels). Mismatched dtypes cause runtime errors, so cast explicitly when needed.

x = torch.tensor([1.0, 2.0])      # float32 by default
x_half = x.half()                  # float16
x_bf16 = x.to(torch.bfloat16)     # bfloat16
x_int = torch.tensor([0, 1, 2])   # int64 by default

# Explicit cast
labels = torch.tensor([0, 1, 1], dtype=torch.long)    # int64 for cross-entropy
probs = torch.tensor([0.1, 0.9], dtype=torch.float32) # explicit float32

Autograd

Training a neural network means computing how much each parameter contributed to the loss, then nudging each parameter in the direction that reduces the loss. The math is backpropagation — the chain rule applied across a long graph of operations. PyTorch handles this automatically through its autograd engine. You do the forward computation in plain Python, call .backward() on the loss, and every parameter's gradient appears in its .grad attribute. No manual calculus required.

Computation Graph and requires_grad

When you create a tensor with requires_grad=True, PyTorch starts recording every operation you perform on it. This builds a computation graph — a directed acyclic graph where each node is an operation and each edge is a tensor. The graph is built dynamically, on every forward pass, which means you can use Python if statements, for loops, and any control flow you want. This is PyTorch's key design choice: the graph is defined by the code you run, not by a static blueprint you declare upfront.

x = torch.tensor(3.0, requires_grad=True)
y = x ** 2 + 2 * x + 1    # y = x^2 + 2x + 1

# The graph now knows: y depends on x via power, multiply, and add ops
print(y)                    # tensor(16., grad_fn=<AddBackward0>)
print(y.grad_fn)            # AddBackward0 — the last operation

backward() and the .grad Attribute

Calling .backward() on a scalar tensor traverses the computation graph in reverse and computes the gradient of that scalar with respect to every leaf tensor that has requires_grad=True. The gradient accumulates into the .grad attribute of each leaf.

x = torch.tensor(3.0, requires_grad=True)
y = x ** 2 + 2 * x + 1
y.backward()                # dy/dx = 2x + 2 = 8 at x=3

print(x.grad)              # tensor(8.)

# Verify: the derivative of x^2 + 2x + 1 is 2x + 2
# At x=3: 2*3 + 2 = 8  ✓

torch.no_grad() and Gradient Accumulation

During inference, you don't need gradients. Wrapping code in torch.no_grad() tells PyTorch to skip building the computation graph, saving memory and speeding up execution. This is essential during evaluation and any time you update parameters manually.

Gradient accumulation is a critical gotcha. By default, calling .backward() adds the new gradients to whatever is already in .grad. It does not replace them. This is intentional — it lets you accumulate gradients over multiple mini-batches to simulate a larger batch size. But it means you must zero out gradients before each optimization step, or they'll grow without bound.

# No-grad context for inference
x = torch.randn(3, requires_grad=True)
with torch.no_grad():
    y = x * 2       # no graph built, no memory overhead
    print(y.requires_grad)   # False

# Gradient accumulation demo — the trap
w = torch.tensor(2.0, requires_grad=True)

loss1 = (w * 3) ** 2
loss1.backward()
print(w.grad)               # tensor(36.)

loss2 = (w * 3) ** 2
loss2.backward()
print(w.grad)               # tensor(72.) — doubled! Gradients accumulated

# Fix: zero gradients before each backward pass
w.grad.zero_()
loss3 = (w * 3) ** 2
loss3.backward()
print(w.grad)               # tensor(36.) — correct
💡 Key Insight

Gradient accumulation is not a bug — it's a feature. If your GPU can only fit a batch of 8 but you want an effective batch size of 32, you can call .backward() four times without zeroing, then call optimizer.step() once. This gives you the same gradient as a single batch of 32, without needing 4x the memory.

nn.Module

Raw tensors and autograd are enough to train a model, but writing everything from scratch gets tedious fast. The nn.Module class is PyTorch's answer: a base class for all neural network components. Every layer, every model, every building block inherits from it. A Module knows how to hold parameters, move them to the GPU, save them to disk, and compose with other Modules. You define two things: the layers in __init__ and the forward pass in forward().

Custom Module Definition

import torch.nn as nn

class LinearClassifier(nn.Module):
    def __init__(self, input_dim, num_classes):
        super().__init__()
        self.fc1 = nn.Linear(input_dim, 128)
        self.relu = nn.ReLU()
        self.fc2 = nn.Linear(128, num_classes)

    def forward(self, x):
        x = self.fc1(x)
        x = self.relu(x)
        x = self.fc2(x)
        return x

model = LinearClassifier(input_dim=784, num_classes=10)
print(model)

The forward() method defines what happens when you pass data through the model. You never call forward() directly — instead you call the model like a function, model(x), which invokes forward() plus any registered hooks.

Parameter Registration

When you assign an nn.Linear, nn.Conv2d, or any other nn.Module as an attribute in __init__, PyTorch automatically registers its parameters. You can iterate over all parameters with model.parameters(), which is what the optimizer uses to know what to update. If you need a bare tensor as a learnable parameter, wrap it in nn.Parameter.

class ScaledModel(nn.Module):
    def __init__(self):
        super().__init__()
        self.linear = nn.Linear(10, 5)
        # Custom learnable parameter
        self.scale = nn.Parameter(torch.ones(5))

    def forward(self, x):
        return self.linear(x) * self.scale

model = ScaledModel()
for name, param in model.named_parameters():
    print(f"{name}: shape={param.shape}")
# Output:
# linear.weight: shape=torch.Size([5, 10])
# linear.bias: shape=torch.Size([5])
# scale: shape=torch.Size([5])

Module Composition and nn.Sequential

Real models are trees of Modules. A ResNet block contains convolutions, batch norms, and a skip connection. A Transformer contains attention layers, feed-forward layers, and normalization. You compose them by assigning sub-modules in __init__. For simple linear stacks of layers, nn.Sequential is a shortcut that saves you from writing a forward() method.

# nn.Sequential for simple stacks
simple_model = nn.Sequential(
    nn.Linear(784, 256),
    nn.ReLU(),
    nn.Dropout(0.2),
    nn.Linear(256, 128),
    nn.ReLU(),
    nn.Dropout(0.2),
    nn.Linear(128, 10)
)

# Equivalent to writing a custom Module with the same layers
# Sequential is convenient but can't handle skip connections or branching

# ModuleList for dynamic architectures
class FlexibleMLP(nn.Module):
    def __init__(self, layer_sizes):
        super().__init__()
        self.layers = nn.ModuleList([
            nn.Linear(layer_sizes[i], layer_sizes[i + 1])
            for i in range(len(layer_sizes) - 1)
        ])

    def forward(self, x):
        for layer in self.layers[:-1]:
            x = torch.relu(layer(x))
        return self.layers[-1](x)

flex = FlexibleMLP([784, 256, 128, 10])
⚠️ Warning

If you store sub-modules in a plain Python list instead of nn.ModuleList, PyTorch won't know about them. Their parameters won't appear in model.parameters(), won't move when you call model.to(device), and won't be saved by torch.save(model.state_dict()). Always use nn.ModuleList or nn.ModuleDict for collections of layers.

Data Pipeline

A model is only as good as its data pipeline. PyTorch provides a clean abstraction: you define what your data looks like with a Dataset, and how to serve it with a DataLoader. The DataLoader handles batching, shuffling, parallel loading, and memory pinning. Getting this right can easily double your training throughput.

Dataset Class

The Dataset class is an abstract interface with two required methods: __len__ (how many samples) and __getitem__ (return the i-th sample). You subclass it for your specific data.

from torch.utils.data import Dataset, DataLoader

class CSVDataset(Dataset):
    def __init__(self, csv_path, transform=None):
        import pandas as pd
        self.data = pd.read_csv(csv_path)
        self.transform = transform

    def __len__(self):
        return len(self.data)

    def __getitem__(self, idx):
        row = self.data.iloc[idx]
        features = torch.tensor(row[:-1].values, dtype=torch.float32)
        label = torch.tensor(row.iloc[-1], dtype=torch.long)
        if self.transform:
            features = self.transform(features)
        return features, label

DataLoader, Transforms, and Samplers

The DataLoader wraps a Dataset and provides an iterator that yields batches. You control the batch size, whether to shuffle, how many background workers load data in parallel, and whether to pin memory for faster GPU transfers.

Transforms are functions that preprocess each sample — normalization, data augmentation, conversion to tensors. For images, torchvision.transforms provides a rich library. For custom data, you write your own callables.

A Sampler controls the order in which indices are drawn. The default for training is RandomSampler (shuffled), and for evaluation it is SequentialSampler. For imbalanced datasets, WeightedRandomSampler oversamples minority classes.

from torch.utils.data import DataLoader, random_split, WeightedRandomSampler

# Create dataset
dataset = CSVDataset("data.csv")

# Split into train/val
train_size = int(0.8 * len(dataset))
val_size = len(dataset) - train_size
train_set, val_set = random_split(dataset, [train_size, val_size])

# DataLoader with parallel loading
train_loader = DataLoader(
    train_set,
    batch_size=64,
    shuffle=True,
    num_workers=4,        # background data-loading processes
    pin_memory=True,      # faster CPU-to-GPU transfer
    drop_last=True        # drop incomplete last batch
)

val_loader = DataLoader(
    val_set,
    batch_size=128,
    shuffle=False,
    num_workers=4,
    pin_memory=True
)

# Iterate
for batch_features, batch_labels in train_loader:
    batch_features = batch_features.to(device)
    batch_labels = batch_labels.to(device)
    # ... training step
    break  # just showing the pattern

The num_workers parameter spawns separate processes to load and preprocess data while the GPU is busy with the previous batch. A good starting point is 4 workers per GPU. Too many workers waste memory and CPU; too few starve the GPU. The pin_memory flag allocates data in page-locked (pinned) memory, which makes the .to(device) transfer asynchronous and faster.

💡 Key Insight

If your GPU utilization is below 90%, the bottleneck is almost always the data pipeline, not the model. Profile with torch.utils.bottleneck or just time your data loading loop separately. Increasing num_workers, enabling pin_memory, or moving preprocessing offline into a cached format can dramatically improve throughput.

Large-Scale Data Loading

When your dataset is too large to fit on disk as individual files — think billions of text tokens or millions of images — the standard Dataset pattern breaks down. Opening millions of small files is slow, and random access across a network filesystem is even slower. The solution is sharded data: you pack your data into large sequential files (tar archives, TFRecord-like formats) and stream through them.

WebDataset is a popular library that stores samples as tar archives. Each sample is a set of files (image.jpg, label.txt) grouped together. The DataLoader streams through the tar sequentially, which is ideal for both local SSDs and cloud storage. Mosaic StreamingDataset takes a similar approach but adds deterministic shuffling, resumption after crashes, and multi-node coordination out of the box. For Hugging Face datasets, the datasets library uses memory-mapped Arrow files that provide random access without loading everything into RAM.

# WebDataset example — streaming from sharded tar files
import webdataset as wds

dataset = (
    wds.WebDataset("train-{000000..000099}.tar")
    .shuffle(1000)                # shuffle buffer
    .decode("pil")                # decode images
    .to_tuple("input.jpg", "target.cls")
    .map_tuple(transform, lambda x: int(x))
    .batched(64)
)

loader = wds.WebLoader(dataset, num_workers=4, batch_size=None)

# Prefetching — overlap data loading with GPU compute
loader = loader.unbatched().shuffle(5000).batched(64)

# The key insight: tar files stream sequentially,
# which is 10-100x faster than random small-file reads

Regardless of which format you choose, the principle is the same: large sequential reads beat many small random reads. Pack your data once, stream it during training, and let the prefetch buffer hide the I/O latency.

Training Loop

Unlike some frameworks that hide the training loop behind a .fit() call, PyTorch makes you write it yourself. This is a deliberate design choice. It means more boilerplate, but it also means you understand and control every step. There is no magic — just a clear sequence of operations that repeats every batch.

The Complete Loop

Here is an end-to-end example: a small convolutional network trained on synthetic data. Every step is annotated.

import torch
import torch.nn as nn
import torch.optim as optim
from torch.utils.data import DataLoader, TensorDataset

# --- 1. Define the model ---
class ConvNet(nn.Module):
    def __init__(self, num_classes=10):
        super().__init__()
        self.features = nn.Sequential(
            nn.Conv2d(1, 32, kernel_size=3, padding=1),
            nn.ReLU(),
            nn.MaxPool2d(2),
            nn.Conv2d(32, 64, kernel_size=3, padding=1),
            nn.ReLU(),
            nn.MaxPool2d(2),
        )
        self.classifier = nn.Sequential(
            nn.Flatten(),
            nn.Linear(64 * 7 * 7, 128),
            nn.ReLU(),
            nn.Dropout(0.5),
            nn.Linear(128, num_classes),
        )

    def forward(self, x):
        x = self.features(x)
        x = self.classifier(x)
        return x

# --- 2. Create synthetic data ---
X_train = torch.randn(1000, 1, 28, 28)
y_train = torch.randint(0, 10, (1000,))
X_val = torch.randn(200, 1, 28, 28)
y_val = torch.randint(0, 10, (200,))

train_loader = DataLoader(
    TensorDataset(X_train, y_train), batch_size=64, shuffle=True
)
val_loader = DataLoader(
    TensorDataset(X_val, y_val), batch_size=128
)

# --- 3. Setup ---
device = torch.device("cuda" if torch.cuda.is_available() else "cpu")
model = ConvNet(num_classes=10).to(device)
criterion = nn.CrossEntropyLoss()
optimizer = optim.Adam(model.parameters(), lr=1e-3)

# --- 4. Training loop ---
num_epochs = 5
for epoch in range(num_epochs):
    model.train()                          # enable dropout, batchnorm training mode
    running_loss = 0.0
    correct = 0
    total = 0

    for batch_x, batch_y in train_loader:
        batch_x = batch_x.to(device)
        batch_y = batch_y.to(device)

        optimizer.zero_grad()              # reset gradients from previous step
        outputs = model(batch_x)           # forward pass
        loss = criterion(outputs, batch_y) # compute loss
        loss.backward()                    # backward pass — compute gradients
        optimizer.step()                   # update parameters

        running_loss += loss.item() * batch_x.size(0)
        _, predicted = outputs.max(1)
        correct += predicted.eq(batch_y).sum().item()
        total += batch_y.size(0)

    train_loss = running_loss / total
    train_acc = correct / total

    # --- 5. Evaluation ---
    model.eval()                           # disable dropout, use running stats for batchnorm
    val_loss = 0.0
    val_correct = 0
    val_total = 0

    with torch.no_grad():                  # no gradients needed for evaluation
        for batch_x, batch_y in val_loader:
            batch_x = batch_x.to(device)
            batch_y = batch_y.to(device)
            outputs = model(batch_x)
            loss = criterion(outputs, batch_y)
            val_loss += loss.item() * batch_x.size(0)
            _, predicted = outputs.max(1)
            val_correct += predicted.eq(batch_y).sum().item()
            val_total += batch_y.size(0)

    val_loss /= val_total
    val_acc = val_correct / val_total

    print(f"Epoch {epoch+1}/{num_epochs} | "
          f"Train Loss: {train_loss:.4f} Acc: {train_acc:.4f} | "
          f"Val Loss: {val_loss:.4f} Acc: {val_acc:.4f}")

Let's break down why each step matters. optimizer.zero_grad() clears the gradient buffers because PyTorch accumulates gradients by default. The forward pass runs your input through the model and produces predictions. The loss computation compares predictions to ground truth using a differentiable loss function. loss.backward() walks the computation graph backward, computing the gradient of the loss with respect to every parameter. optimizer.step() uses those gradients to update the parameters according to whatever optimization algorithm you chose — SGD, Adam, AdamW, and so on.

Evaluation Mode

Calling model.eval() switches layers like Dropout and BatchNorm to evaluation behavior. Dropout stops dropping neurons. BatchNorm uses its running statistics instead of batch statistics. This is critical — if you forget it, your validation metrics will be noisy and unreliable. Always pair model.eval() with torch.no_grad() to also save memory during evaluation. Remember to call model.train() before the next training epoch.

Model Saving and Loading

PyTorch gives you two levels of saving. The recommended approach saves only the state dict — a Python dictionary mapping parameter names to tensors. This is portable and doesn't depend on the exact class definition being importable. The alternative saves the entire model object with pickle, which is fragile and breaks if you move code around.

# Save state dict (recommended)
torch.save(model.state_dict(), "model_weights.pth")

# Load state dict
model = ConvNet(num_classes=10)
model.load_state_dict(torch.load("model_weights.pth", weights_only=True))
model.to(device)
model.eval()

# Save full checkpoint (for resuming training)
checkpoint = {
    "epoch": epoch,
    "model_state_dict": model.state_dict(),
    "optimizer_state_dict": optimizer.state_dict(),
    "train_loss": train_loss,
    "val_loss": val_loss,
}
torch.save(checkpoint, "checkpoint.pth")

# Resume from checkpoint
checkpoint = torch.load("checkpoint.pth", weights_only=False)
model.load_state_dict(checkpoint["model_state_dict"])
optimizer.load_state_dict(checkpoint["optimizer_state_dict"])
start_epoch = checkpoint["epoch"] + 1
💡 Key Insight

Always save checkpoints that include both the model and optimizer state dicts, plus the current epoch and loss. This lets you resume training after a crash or preemption. For long training runs, save checkpoints periodically — losing 12 hours of training to a power outage is not fun.

Model Compilation and Export

PyTorch's default mode is eager execution: each operation runs immediately, just like regular Python. This is great for debugging, but it leaves performance on the table because the framework can't see ahead to optimize the full computation. Model compilation and export address this by analyzing the computation graph and generating optimized code.

torch.compile

torch.compile is PyTorch's modern answer to performance optimization, introduced in PyTorch 2.0. Under the hood, it uses TorchDynamo to capture the computation graph from your Python code and TorchInductor to compile it into optimized triton kernels for GPU or C++ code for CPU. The beautiful part: it works on your existing nn.Module with a single line change.

import torch

# Your existing model — no changes needed
model = ConvNet(num_classes=10).to(device)

# Compile it — that's it
compiled_model = torch.compile(model)

# Use it exactly the same way
x = torch.randn(64, 1, 28, 28, device=device)
output = compiled_model(x)

# The first call is slow (compilation happens), subsequent calls are fast
# Common mode options:
#   "default"       — balanced optimization
#   "reduce-overhead" — minimize GPU kernel launch overhead (best for small models)
#   "max-autotune"  — try many kernel variants, pick fastest (slow compile, fast run)

compiled_fast = torch.compile(model, mode="reduce-overhead")
compiled_tuned = torch.compile(model, mode="max-autotune")

In practice, torch.compile can give you 20–60% speedups on standard models with zero code changes. It handles most Python control flow, dynamic shapes, and even data-dependent branching. When it encounters something it can't compile (like a call to a Python library that isn't torch-aware), it falls back to eager execution for that portion — a mechanism called graph breaks. Fewer graph breaks means better optimization.

💡 Key Insight

torch.compile is not an all-or-nothing deal. It compiles what it can and falls back to eager mode for the rest. You can run torch._dynamo.explain(model)(sample_input) to see exactly where graph breaks occur and whether they're hurting your performance. Fix the easy ones — often it's just replacing a NumPy call with its torch equivalent.

TorchScript: torch.jit.script vs torch.jit.trace

TorchScript is an older approach that serializes your model into a format that can run without Python. This is useful for deploying models in C++ applications or environments where you can't install Python. There are two ways to create TorchScript models.

torch.jit.trace runs your model once with example input and records what happened. It captures the exact sequence of operations but misses any data-dependent control flow — if your forward() has an if statement that depends on the input, tracing only records the branch that ran during tracing.

torch.jit.script analyzes your Python source code directly and compiles it into TorchScript IR. It handles control flow, loops, and conditionals correctly. The tradeoff: it only supports a subset of Python, so complex code may need refactoring.

# Tracing — works for models without data-dependent control flow
model.eval()
example_input = torch.randn(1, 1, 28, 28, device=device)
traced_model = torch.jit.trace(model, example_input)
traced_model.save("model_traced.pt")

# Scripting — handles control flow but requires compatible Python
class ConditionalModel(nn.Module):
    def __init__(self):
        super().__init__()
        self.linear = nn.Linear(10, 5)

    def forward(self, x):
        if x.sum() > 0:           # data-dependent branch
            return self.linear(x)
        else:
            return self.linear(x) * 0.5

scripted_model = torch.jit.script(ConditionalModel())
scripted_model.save("model_scripted.pt")

# Load in Python (or C++) without the original class definition
loaded = torch.jit.load("model_traced.pt")
output = loaded(example_input)

ONNX Export

ONNX (Open Neural Network Exchange) is an open format for representing machine learning models. Exporting to ONNX lets you run your model with ONNX Runtime (which is highly optimized), TensorRT (NVIDIA's inference engine), or in browsers with ONNX.js. It's the lingua franca of model deployment.

# Export to ONNX
model.eval()
dummy_input = torch.randn(1, 1, 28, 28, device="cpu")
model_cpu = model.cpu()

torch.onnx.export(
    model_cpu,
    dummy_input,
    "model.onnx",
    input_names=["input"],
    output_names=["output"],
    dynamic_axes={
        "input": {0: "batch_size"},    # allow variable batch size
        "output": {0: "batch_size"},
    },
    opset_version=17,
)

# Verify with ONNX Runtime
import onnxruntime as ort

session = ort.InferenceSession("model.onnx")
input_name = session.get_inputs()[0].name
result = session.run(None, {input_name: dummy_input.numpy()})
print(f"ONNX output shape: {result[0].shape}")
⚠️ Warning

ONNX export uses tracing under the hood, so the same limitations apply: data-dependent control flow won't be captured correctly. If your model has conditional logic that depends on input values, either refactor to remove it or use TorchScript scripting as an intermediate step. Also check that all your custom operators have ONNX equivalents — exotic operations may need custom ONNX op registrations.

Graph-Mode Optimizations

Both torch.compile and TorchScript perform graph-level optimizations that eager mode cannot. These include operator fusion (combining a convolution, bias add, and ReLU into a single kernel launch), constant folding (pre-computing values that don't change), dead code elimination (removing computations whose results aren't used), and memory planning (reusing memory buffers to reduce allocation overhead). The Inductor backend in torch.compile goes further by generating custom Triton GPU kernels that are tailored to your exact model architecture, often outperforming hand-written CUDA kernels from cuDNN for unusual layer combinations.

Training Reproducibility

Reproducibility in deep learning is harder than it looks. Your results depend on random weight initialization, random data shuffling, random dropout masks, and nondeterministic GPU algorithms. Even with the same code and data, you can get different results across runs — unless you take deliberate steps to control every source of randomness.

Random Seed Management

The first step is seeding every random number generator your code touches. PyTorch has its own RNG, Python has one, NumPy has one, and CUDA has one. You need to seed all of them.

import torch
import numpy as np
import random

def set_seed(seed=42):
    random.seed(seed)                          # Python built-in
    np.random.seed(seed)                       # NumPy
    torch.manual_seed(seed)                    # PyTorch CPU
    torch.cuda.manual_seed(seed)               # current GPU
    torch.cuda.manual_seed_all(seed)           # all GPUs

set_seed(42)

Deterministic Algorithms

Seeding the RNGs is necessary but not sufficient. Many GPU operations have nondeterministic implementations — they use atomic operations whose order depends on thread scheduling. PyTorch lets you force deterministic algorithms at the cost of some speed.

# Force deterministic behavior
torch.backends.cudnn.deterministic = True
torch.backends.cudnn.benchmark = False    # disable auto-tuner (it's nondeterministic)
torch.use_deterministic_algorithms(True)  # strictest mode — errors on nondeterministic ops

torch.use_deterministic_algorithms(True) is the strictest option. It throws a runtime error if any operation would use a nondeterministic algorithm. This is useful for debugging: run it once to discover which operations in your pipeline are nondeterministic, then decide which ones you can tolerate and which you need to fix.

⚠️ Warning

The cudnn.benchmark flag is a common performance trap. When set to True (the default for some setups), cuDNN runs multiple convolution algorithms on the first batch and picks the fastest one. This selection is nondeterministic across runs. Set it to False for reproducibility, but be aware this may slow down convolution-heavy models by 10–30%.

CUDA Non-Determinism

Some operations are inherently nondeterministic on the GPU. Scatter-add operations (like index_add_), certain backward passes for convolutions, and atomicAdd-based operations can produce slightly different results across runs because floating-point addition is not associative — adding the same numbers in different orders gives different rounding errors. In most cases, these differences are tiny and don't affect final model quality. But if you need bit-for-bit reproducibility (for regulatory or audit reasons), you have to accept the performance penalty of deterministic algorithms and test thoroughly.

Reproducible Data Loading

The DataLoader's shuffling behavior depends on its worker processes. Each worker has its own RNG state, and the order of data delivery depends on how fast each worker finishes. To make this reproducible, set the worker_init_fn to seed each worker deterministically and use a Generator for the sampler.

def seed_worker(worker_id):
    worker_seed = torch.initial_seed() % (2**32)
    np.random.seed(worker_seed)
    random.seed(worker_seed)

g = torch.Generator()
g.manual_seed(42)

train_loader = DataLoader(
    train_set,
    batch_size=64,
    shuffle=True,
    num_workers=4,
    worker_init_fn=seed_worker,
    generator=g,
)

# Now the order of batches is deterministic across runs
# (assuming same number of workers and same seed)
💡 Key Insight

Perfect reproducibility and maximum performance are at odds. Deterministic algorithms are often slower, disabling cuDNN benchmark removes auto-tuning, and single-threaded data loading eliminates timing-dependent ordering. For research, set everything to deterministic during final experiments. For production training, accept small numerical differences and focus on statistical reproducibility — running the same experiment three times should give results within a tight confidence interval, even if the exact numbers differ.

Putting It All Together

Here is a condensed template that combines every concept from this section into a single production-ready training script. You can use this as a starting point for any PyTorch project.

import torch
import torch.nn as nn
import torch.optim as optim
from torch.utils.data import DataLoader, TensorDataset
import random
import numpy as np

# --- Reproducibility ---
def set_seed(seed=42):
    random.seed(seed)
    np.random.seed(seed)
    torch.manual_seed(seed)
    torch.cuda.manual_seed_all(seed)
    torch.backends.cudnn.deterministic = True
    torch.backends.cudnn.benchmark = False

set_seed(42)

# --- Device ---
device = torch.device("cuda" if torch.cuda.is_available() else "cpu")

# --- Model ---
class MLP(nn.Module):
    def __init__(self, input_dim, hidden_dim, output_dim):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(input_dim, hidden_dim),
            nn.ReLU(),
            nn.Dropout(0.2),
            nn.Linear(hidden_dim, output_dim),
        )

    def forward(self, x):
        return self.net(x)

model = MLP(784, 256, 10).to(device)

# --- Compile (PyTorch 2.0+) ---
compiled_model = torch.compile(model)

# --- Data ---
X = torch.randn(5000, 784)
y = torch.randint(0, 10, (5000,))
train_ds = TensorDataset(X[:4000], y[:4000])
val_ds = TensorDataset(X[4000:], y[4000:])

def seed_worker(worker_id):
    np.random.seed(torch.initial_seed() % (2**32))
    random.seed(torch.initial_seed() % (2**32))

g = torch.Generator().manual_seed(42)
train_loader = DataLoader(train_ds, batch_size=64, shuffle=True,
                          num_workers=2, pin_memory=True,
                          worker_init_fn=seed_worker, generator=g)
val_loader = DataLoader(val_ds, batch_size=128, num_workers=2, pin_memory=True)

# --- Training ---
criterion = nn.CrossEntropyLoss()
optimizer = optim.AdamW(compiled_model.parameters(), lr=1e-3, weight_decay=1e-2)
scheduler = optim.lr_scheduler.CosineAnnealingLR(optimizer, T_max=20)

best_val_loss = float("inf")
for epoch in range(20):
    compiled_model.train()
    total_loss, correct, total = 0.0, 0, 0
    for xb, yb in train_loader:
        xb, yb = xb.to(device), yb.to(device)
        optimizer.zero_grad()
        out = compiled_model(xb)
        loss = criterion(out, yb)
        loss.backward()
        torch.nn.utils.clip_grad_norm_(compiled_model.parameters(), max_norm=1.0)
        optimizer.step()
        total_loss += loss.item() * xb.size(0)
        correct += (out.argmax(1) == yb).sum().item()
        total += yb.size(0)
    scheduler.step()

    compiled_model.eval()
    val_loss, val_correct, val_total = 0.0, 0, 0
    with torch.no_grad():
        for xb, yb in val_loader:
            xb, yb = xb.to(device), yb.to(device)
            out = compiled_model(xb)
            loss = criterion(out, yb)
            val_loss += loss.item() * xb.size(0)
            val_correct += (out.argmax(1) == yb).sum().item()
            val_total += yb.size(0)

    vl = val_loss / val_total
    print(f"Epoch {epoch+1:02d} | Train Loss {total_loss/total:.4f} "
          f"Acc {correct/total:.4f} | Val Loss {vl:.4f} "
          f"Acc {val_correct/val_total:.4f}")

    if vl < best_val_loss:
        best_val_loss = vl
        torch.save(model.state_dict(), "best_model.pth")

# --- Export ---
model.load_state_dict(torch.load("best_model.pth", weights_only=True))
model.eval()
dummy = torch.randn(1, 784)
torch.onnx.export(model, dummy, "model.onnx",
                   input_names=["input"], output_names=["output"],
                   dynamic_axes={"input": {0: "batch"}, "output": {0: "batch"}})

This script covers reproducibility, device management, model definition, compilation, data loading with deterministic workers, a training loop with gradient clipping and learning rate scheduling, evaluation, checkpointing the best model, and ONNX export. Every piece you've learned in this section shows up here. Copy it, adapt it to your data, and you have a solid foundation for any PyTorch project.
