Mixed Precision & Hyperparameter Optimization

Chapter 8: Training Deep Networks Section 7
TL;DR

Mixed precision training is a free lunch: add 4 lines to your PyTorch loop, get ~2× speedup and ~40% less memory with negligible accuracy loss. Hyperparameter optimization (HPO) is about spending your compute budget wisely — random search beats grid search almost always, and Bayesian methods (Optuna) beat both. The punchline: most models are undertrained, not undertuned. Fix your training recipe before doing extensive HPO.

Mixed Precision Training

What It Is and Why You Want It

By default, neural network training uses float32 (FP32) — 32 bits per number. It's precise, it's safe, and it's wasteful. Most of the computations in a neural network — matrix multiplies, convolutions, element-wise operations — don't need 7 decimal digits of precision. They do just fine with 3.

Float16 (FP16, half precision) uses 16 bits per number. It's 2× smaller in memory and 2× faster on modern GPU Tensor Cores. The problem: FP16 has a much narrower dynamic range (max ~65,504 vs FP32's ~3.4 × 10³⁸). Small gradients can underflow to zero and vanish silently.

Mixed precision is the practical solution. Keep a master copy of weights in FP32 (for the optimizer), run the forward and backward passes in FP16 (for speed), and use loss scaling to prevent gradient underflow. The result:

  • ~2× speedup on Tensor Core GPUs (V100, A100, H100)
  • ~40% less GPU memory (activations are halved)
  • Negligible accuracy loss — SGD noise dwarfs FP16 rounding errors

BF16 vs FP16

Not all 16-bit formats are created equal.

BFloat16 (BF16) was designed by Google to keep the same exponent range as FP32 (8 exponent bits) while sacrificing mantissa precision (7 bits instead of FP16's 10). This means BF16 can represent the same range of magnitudes as FP32 — no more gradient underflow, no more loss scaling headaches. The trade-off is slightly less numerical precision, which rarely matters for training.

FormatBitsExponentMantissaMax ValueLoss Scaling Needed?
FP3232823~3.4 × 10³⁸No
FP1616510~65,504Yes
BF161687~3.4 × 10³⁸No

Rule of thumb: if your hardware supports BF16 (A100+, TPUs), use BF16 — it's simpler and just as fast. If you're on a V100 or older, use FP16 with loss scaling.

PyTorch AMP — The 4-Line Upgrade

PyTorch's Automatic Mixed Precision (AMP) handles everything: which ops run in FP16 vs FP32, dynamic loss scaling, gradient checking. Here's the pattern:

from torch.amp import autocast, GradScaler

scaler = GradScaler("cuda")

for inputs, targets in dataloader:
    optimizer.zero_grad()

    with autocast("cuda"):                       # forward in FP16
        outputs = model(inputs)
        loss = criterion(outputs, targets)

    scaler.scale(loss).backward()                # scaled backward
    scaler.step(optimizer)                       # unscale → check → step
    scaler.update()                              # adjust scale factor

That's it. Four extra lines — GradScaler, autocast, scaler.scale, scaler.update — for a ~2× speedup. The autocast context manager routes matmuls and convolutions to FP16 Tensor Cores while keeping numerically sensitive ops (softmax, layer norm, loss functions) in FP32. The GradScaler dynamically adjusts the loss scaling factor: if gradients overflow, it halves the scale and skips the step; if things are stable, it doubles the scale to capture smaller gradients.

For BF16 on supported hardware, it's even simpler — no scaler needed:

for inputs, targets in dataloader:
    optimizer.zero_grad()

    with autocast("cuda", dtype=torch.bfloat16):
        outputs = model(inputs)
        loss = criterion(outputs, targets)

    loss.backward()
    optimizer.step()

When NOT to Use Mixed Precision

  • Loss functions that need high precision — some custom losses with tiny epsilon terms break in FP16. AMP handles standard losses, but custom ones may need manual FP32 casts.
  • Tiny models — if your model fits easily in memory and trains in minutes, the overhead of AMP (casting, scaling logic) isn't worth the complexity.
  • Debugging numerical issues — always debug in FP32 first. If training diverges, rule out numerical precision before investigating other causes.
  • CPU-only training — mixed precision benefits come from GPU Tensor Cores. On CPU, there's no speedup.
💡 Key Insight

Mixed precision is not an approximation — it's an engineering optimization. With FP32 master weights and proper loss scaling, the mathematical updates are identical to full FP32 training. The rounding noise from FP16 is completely dominated by the inherent noise of mini-batch SGD. If your mixed precision run diverges but your FP32 run doesn't, the bug is in your scaling setup, not in the concept.

Hyperparameter Optimization

What Are Hyperparameters?

Neural networks learn their parameters (weights, biases) through gradient descent. But there's a whole other set of choices you make before training starts — choices that gradient descent can't help with:

  • Learning rate — the single most important hyperparameter
  • Batch size — affects generalization, training speed, and memory
  • Weight decay — L2 regularization strength
  • Dropout rate — how aggressively you regularize
  • Architecture choices — number of layers, hidden size, number of attention heads
  • Optimizer choice — Adam vs SGD vs AdamW
  • Learning rate schedule — cosine, linear warmup, step decay

These are hyperparameters. You don't learn them — you search for them. The question is how to search efficiently.

Grid Search

The naive approach. Pick a set of values for each hyperparameter, try every combination.

# Grid search: try every combination
learning_rates = [1e-4, 1e-3, 1e-2]
batch_sizes    = [32, 64, 128]
dropouts       = [0.1, 0.3, 0.5]

# 3 × 3 × 3 = 27 experiments — manageable
# But add two more hyperparameters with 3 values each:
# 3 × 3 × 3 × 3 × 3 = 243 experiments — getting expensive
# With 5 hyperparameters and 5 values each:
# 5⁵ = 3,125 experiments — don't.

Grid search is exhaustive and exponential in the number of hyperparameters. It works fine for 1–2 hyperparameters. Beyond that, it's a waste of compute. The fundamental problem: grid search allocates equal budget to every hyperparameter, even the ones that barely matter.

Random Search

Instead of a grid, sample hyperparameters randomly from distributions. This sounds careless, but it's surprisingly effective — and the reason is subtle.

Most hyperparameters don't matter equally. In a typical setup, the learning rate might explain 80% of the performance variation, while dropout explains 5%. Grid search wastes most of its budget exploring the dropout dimension at the expense of the learning rate dimension. Random search, by contrast, gives you a unique learning rate value for every experiment, exploring that important dimension much more thoroughly.

This is the core insight from Bergstra & Bengio (2012): when only a few hyperparameters matter (which is almost always the case), random search finds good configurations faster than grid search, even with the same total budget. With 60 random trials, you have a 95% chance of finding a configuration within the top 5% of the search space for the important dimensions.

💡 Why Random Beats Grid

Imagine learning rate matters a lot but dropout barely matters. A 5×5 grid gives you only 5 unique learning rates across 25 experiments. Random search with 25 experiments gives you 25 unique learning rates. You're sampling the important dimension 5× more densely for the same budget.

Bayesian Optimization

Random search is good, but it's memoryless — each trial ignores the results of all previous trials. Bayesian optimization remembers.

The idea: build a surrogate model (a probabilistic model) of the objective function — how validation loss varies as a function of hyperparameters. After each trial, update the surrogate with the new data point. Then use an acquisition function to decide where to sample next, balancing:

  • Exploitation — sample near the best configurations found so far
  • Exploration — sample in regions of high uncertainty (where the surrogate has little data)

Common surrogate models include Gaussian Processes (for small search spaces) and Tree-structured Parzen Estimators (TPE, which scales better to high dimensions). The acquisition function (Expected Improvement, Upper Confidence Bound, etc.) turns the surrogate's predictions into a score for "how promising is this configuration?"

In practice, Bayesian optimization consistently finds better hyperparameters in fewer trials than random search — typically 2–5× more efficient for 10+ dimensional search spaces.

Optuna — The Practical Tool

You don't need to implement Bayesian optimization from scratch. Optuna is the gold standard for HPO in Python. It uses TPE by default, supports pruning (killing bad trials early), and integrates cleanly with any training loop.

import optuna

def objective(trial):
    # Suggest hyperparameters from distributions
    lr = trial.suggest_float('lr', 1e-5, 1e-2, log=True)
    weight_decay = trial.suggest_float('weight_decay', 1e-6, 1e-2, log=True)
    dropout = trial.suggest_float('dropout', 0.0, 0.5)
    n_layers = trial.suggest_int('n_layers', 2, 6)
    hidden_size = trial.suggest_categorical('hidden_size', [128, 256, 512])

    # Build model with these hyperparameters
    model = build_model(n_layers, hidden_size, dropout)
    optimizer = torch.optim.AdamW(model.parameters(), lr=lr,
                                   weight_decay=weight_decay)

    # Train and evaluate
    for epoch in range(num_epochs):
        train_one_epoch(model, optimizer, train_loader)
        val_loss = evaluate(model, val_loader)

        # Report intermediate result for pruning
        trial.report(val_loss, epoch)
        if trial.should_prune():
            raise optuna.TrialPruned()

    return val_loss

# Create study and optimize
study = optuna.create_study(direction='minimize')
study.optimize(objective, n_trials=100)

# Best hyperparameters
print(study.best_params)
# {'lr': 0.00032, 'weight_decay': 0.00014, 'dropout': 0.18, ...}

Key Optuna features worth knowing:

  • Pruning — stop bad trials early using a MedianPruner or HyperbandPruner. If a trial is performing worse than the median of completed trials at the same epoch, kill it. This alone can save 50–80% of your HPO compute budget.
  • Visualization — built-in plots for parameter importance, optimization history, and parallel coordinate charts. optuna.visualization.plot_param_importances(study) tells you which hyperparameters actually matter.
  • Parallel execution — point multiple workers at the same study (backed by a shared database) and they'll coordinate automatically. Scales linearly with the number of GPUs.
  • Log-scale sampling — use log=True for any hyperparameter that spans orders of magnitude (learning rate, weight decay). This ensures uniform coverage across scales.

HPO Methods Compared

Method Efficiency Best For Scales with Dimensions Ease of Use
Grid Search Low 1–2 hyperparameters; quick sanity checks Exponential blowup Trivial
Random Search Medium Early exploration; <10 hyperparameters Linear — unaffected by dimensionality Easy
Bayesian (Optuna) High Serious HPO; expensive training runs Sublinear — learns which dims matter Moderate (requires objective function)

For most practitioners: start with random search to understand the landscape, then switch to Optuna once you've narrowed the ranges and want to squeeze out the last bit of performance.

Practical HPO Advice

Lessons from thousands of collective GPU-hours spent tuning models:

1. Tune the most impactful hyperparameters first. The priority order, from experience across many domains:

  1. Learning rate — almost always the most important. A 10× wrong learning rate is worse than getting every other hyperparameter wrong combined.
  2. Batch size — affects both optimization dynamics and generalization.
  3. Weight decay — critical for transformers, less so for CNNs with dropout.
  4. Architecture (depth, width) — matters, but less than most people think within a reasonable range.
  5. Dropout / regularization — usually has a relatively narrow optimal range.

2. Always use log-uniform for learning rate. The difference between 1e-4 and 2e-4 matters about as much as the difference between 1e-3 and 2e-3. Linear sampling wastes budget on the high end. trial.suggest_float('lr', 1e-5, 1e-2, log=True) — always.

3. Use early stopping and pruning. Don't run every trial to completion. If a trial is clearly bad after 20% of training, kill it. Optuna's MedianPruner does this automatically. Combined with Bayesian search, you can evaluate 100 configurations in the wall-clock time of 30 full runs.

4. Start coarse, then refine. First run: wide ranges, 30–50 random trials. Identify the promising region. Second run: narrow ranges around the best region, 50–100 Bayesian trials. This two-phase approach is far more efficient than one massive search.

5. Don't tune what doesn't matter. If Optuna's importance plot shows that a hyperparameter explains <2% of variance, fix it at a reasonable default and remove it from the search. Fewer dimensions = faster convergence for everything else.

⚠️ The Most Common HPO Mistake

Most models are undertrained, not undertuned. Before launching a 200-trial HPO study, ask: Am I training long enough? Is my data pipeline a bottleneck? Is my learning rate schedule reasonable? Fixing your training recipe — longer warmup, proper weight decay, cosine schedule — often delivers more improvement than any amount of hyperparameter search. HPO is the last step, not the first.
