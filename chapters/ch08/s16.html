Hyperparameter Optimization

Chapter 8: Training Deep Networks Section 16 of 16 6 subtopics
TL;DR

Hyperparameters are the knobs you set before training starts — learning rate, batch size, dropout rate, number of layers. Picking them well can mean the difference between a mediocre model and a state-of-the-art one. Grid search tries every combination (exhaustive but expensive). Random search samples randomly (surprisingly effective). Bayesian optimization builds a surrogate model of the search space and intelligently picks the next configuration to try. Multi-fidelity methods like Hyperband save compute by quickly killing bad configurations. Frameworks like Optuna and Ray Tune automate the whole process. This section takes you from manual tuning to automated, budget-aware hyperparameter optimization.

What Are Hyperparameters and Why Do They Matter?

There are two kinds of numbers in a machine learning system. Parameters are learned during training — the weights and biases of your neural network. Hyperparameters are set before training begins — they control the learning process itself. The learning rate, batch size, number of layers, hidden dimension, dropout rate, weight decay, optimizer choice, learning rate schedule — all hyperparameters.

Why not just pick reasonable defaults and move on? Because the interaction effects are enormous. A learning rate that works great with batch size 32 might diverge with batch size 512. A model with 4 layers might benefit from dropout 0.3, while the same architecture with 12 layers needs dropout 0.1. The right hyperparameters are not universal — they depend on your specific data, model, and task.

The difference matters in practice. A well-tuned ResNet-50 can outperform a poorly-tuned ResNet-152. Hyperparameter optimization is often a better use of compute than making the model bigger.

Search Spaces: Defining What to Tune

Before searching, you need to define the search space — the set of hyperparameters and their ranges. Good search space design is half the battle.

Continuous parameters like learning rate (1e-5 to 1e-1) and weight decay (1e-6 to 1e-2) should usually be searched on a log scale. The difference between 1e-4 and 2e-4 matters about as much as the difference between 1e-2 and 2e-2 — log-uniform sampling reflects this.

Integer parameters like number of layers (2 to 12), hidden dimension (64 to 1024), and batch size (16 to 512). Some are best searched on a log scale (batch size), others on a linear scale (number of layers).

Categorical parameters like optimizer choice (Adam, SGD, AdamW), activation function (ReLU, GELU, SiLU), or scheduler type (cosine, linear, step).

Conditional parameters are hyperparameters that only matter when another hyperparameter takes a certain value. For example, momentum is only relevant when the optimizer is SGD. A good search framework handles these dependencies — Optuna does this natively.

# Example search space definition (conceptual)
search_space = {
    'learning_rate': LogUniform(1e-5, 1e-1),
    'weight_decay': LogUniform(1e-6, 1e-2),
    'batch_size': CategoricalInt([32, 64, 128, 256]),
    'num_layers': IntRange(2, 8),
    'hidden_dim': CategoricalInt([128, 256, 512, 1024]),
    'dropout': Uniform(0.0, 0.5),
    'optimizer': Categorical(['adam', 'sgd', 'adamw']),
    # Conditional: momentum only for SGD
    'momentum': Uniform(0.8, 0.99),  # only if optimizer == 'sgd'
}

Grid Search and Random Search

Grid Search

Grid search evaluates every combination of a predefined grid of values. If you search over 5 learning rates, 4 batch sizes, and 3 dropout values, that's 5 × 4 × 3 = 60 experiments. Every combination is tried exactly once.

The advantage is simplicity and completeness — you won't miss a good configuration within the grid. The disadvantage is the curse of dimensionality. Add two more hyperparameters with 5 values each, and you're at 60 × 5 × 5 = 1,500 experiments. With 10 hyperparameters, grid search is computationally impossible.

Grid search also wastes compute on unimportant hyperparameters. If learning rate matters a lot but dropout barely matters, grid search still tries every dropout value for every learning rate. It spends equal effort on important and unimportant dimensions.

Random Search

Random search (Bergstra and Bengio, 2012) independently samples each hyperparameter from its range. Instead of a grid, you run N random configurations.

The key insight from the original paper: random search is more efficient than grid search because it explores more unique values of each hyperparameter. In a 5×5 grid with two parameters, you only see 5 unique values of each parameter. In 25 random samples, you see 25 unique values of each parameter. If one parameter matters much more than the other (which is almost always the case), random search explores it 5x more thoroughly.

In practice, random search with the same budget as grid search almost always finds a better configuration. It's embarrassingly parallel (each trial is independent), simple to implement, and scales to any number of hyperparameters.

import random
import math

def random_search(train_and_evaluate, search_space, n_trials=50):
    """Simple random search over hyperparameter space."""
    best_score = float('-inf')
    best_config = None
    
    for trial in range(n_trials):
        # Sample a random configuration
        config = {
            'lr': 10 ** random.uniform(-5, -1),      # Log-uniform
            'weight_decay': 10 ** random.uniform(-6, -2),
            'batch_size': random.choice([32, 64, 128, 256]),
            'dropout': random.uniform(0.0, 0.5),
            'num_layers': random.randint(2, 8),
        }
        
        score = train_and_evaluate(config)
        print(f"Trial {trial}: score={score:.4f}, config={config}")
        
        if score > best_score:
            best_score = score
            best_config = config
    
    print(f"\nBest: score={best_score:.4f}, config={best_config}")
    return best_config

Bayesian Optimization

The Core Idea

Random search treats each trial independently — it doesn't learn from previous results. Bayesian optimization does the opposite: after each trial, it updates a probabilistic model (a "surrogate") of how hyperparameters map to performance. It then uses this surrogate to intelligently pick the next configuration to try — focusing on regions likely to contain the best configuration.

This is the key advantage: Bayesian optimization is sample-efficient. It typically finds better configurations in fewer trials than random search because it avoids wasting trials on regions it already knows are bad.

Gaussian Process (GP) Based Optimization

The classical approach uses a Gaussian Process (GP) as the surrogate model. A GP models the objective function as a distribution over functions — given observed (hyperparameter, score) pairs, it provides a posterior distribution over the score for any new hyperparameter configuration. This posterior gives both a mean prediction (expected score) and uncertainty (how unsure the model is).

The acquisition function uses this posterior to decide where to sample next. The most common acquisition function is Expected Improvement (EI): it balances exploitation (sampling where the predicted score is high) and exploration (sampling where the uncertainty is high). Regions with high expected score and high uncertainty get the most attention.

GP-based Bayesian optimization works beautifully for low-dimensional search spaces (up to ~20 hyperparameters) but struggles with more. The GP scales cubically with the number of observations, and high-dimensional spaces require exponentially more observations to build an accurate surrogate.

Tree-Structured Parzen Estimator (TPE)

The Tree-Structured Parzen Estimator (TPE) is the default algorithm in Optuna, and it works differently from GPs. Instead of modeling p(score | hyperparameters) directly, TPE models two distributions: p(hyperparameters | score is good) and p(hyperparameters | score is bad). It splits the observed trials into "good" (top 25%) and "bad" (bottom 75%) based on their scores, fits separate kernel density estimators to each group, and samples new configurations that have high probability under the "good" model relative to the "bad" model.

TPE handles conditional hyperparameters naturally (hence "tree-structured"), scales better to high-dimensional spaces than GPs, and is computationally cheap. It's the reason Optuna works well out of the box for most use cases.

Multi-Fidelity Methods: Saving Compute

The Problem

Training a model to completion to evaluate one hyperparameter configuration is expensive. If each trial takes 4 hours on a GPU, 100 trials take 400 GPU-hours. Most of those 100 configurations are clearly bad — you could tell after just 10% of training that they won't work. Multi-fidelity methods exploit this by evaluating many configurations cheaply (short training runs) and only investing full training budget in the promising ones.

Successive Halving

Successive Halving (Jamieson and Talwalkar, 2016) works like a tournament. Start N configurations, each with a small budget (e.g., 1 epoch). Evaluate all of them. Keep the top half and double their budget (2 epochs). Evaluate again. Keep the top half again and double (4 epochs). Continue until one configuration remains, trained with the full budget.

With 64 initial configurations and a total budget equivalent to 64 full training runs, successive halving evaluates all 64 at 1 epoch, 32 at 2 epochs, 16 at 4 epochs, 8 at 8 epochs, 4 at 16 epochs, 2 at 32 epochs, and 1 at 64 epochs. You've tested 64 configurations for the cost of ~128 epochs total, instead of 64 × 64 = 4,096 epochs for exhaustive evaluation.

Hyperband

Hyperband (Li et al., 2018) solves a problem with successive halving: it assumes you know the right tradeoff between number of configurations and budget per configuration. Hyperband runs successive halving multiple times with different starting budgets — some brackets start with many cheap configurations (aggressive early stopping) and others start with fewer configurations but more budget per configuration (conservative). It automatically hedges across these tradeoffs.

# Hyperband with Optuna (built-in)
import optuna

def objective(trial):
    lr = trial.suggest_float('lr', 1e-5, 1e-1, log=True)
    dropout = trial.suggest_float('dropout', 0.0, 0.5)
    num_layers = trial.suggest_int('num_layers', 2, 8)
    hidden_dim = trial.suggest_categorical('hidden_dim', [128, 256, 512])
    
    model = create_model(num_layers, hidden_dim, dropout)
    optimizer = torch.optim.Adam(model.parameters(), lr=lr)
    
    # Hyperband uses pruning — report intermediate results
    for epoch in range(50):
        train_loss = train_one_epoch(model, train_loader, optimizer)
        val_acc = evaluate(model, val_loader)
        
        # Report to Optuna for pruning decisions
        trial.report(val_acc, epoch)
        
        # Optuna may prune this trial if it's clearly worse than others
        if trial.should_prune():
            raise optuna.TrialPruned()
    
    return val_acc

# Create study with Hyperband pruner
study = optuna.create_study(
    direction='maximize',
    pruner=optuna.pruners.HyperbandPruner(
        min_resource=1,      # Min epochs before pruning
        max_resource=50,     # Max epochs
        reduction_factor=3,  # Keep top 1/3 at each round
    )
)
study.optimize(objective, n_trials=100)
print(f"Best: {study.best_params}, score: {study.best_value:.4f}")

Population-Based Training (PBT)

Population-Based Training (Jaderberg et al., 2017) combines parallel random search with online adaptation. Instead of fixing hyperparameters for an entire training run, PBT adjusts them during training based on what's working for other members of the population.

Here's how it works. Start a population of, say, 20 models, each with random hyperparameters, all training in parallel. At regular intervals (every N steps), evaluate all models on a validation set. Each model in the bottom 20% "exploits" a better model by copying its weights and hyperparameters. Then it "explores" by randomly perturbing the copied hyperparameters (e.g., multiplying the learning rate by 0.8 or 1.2).

The result is a population that naturally gravitates toward good hyperparameters, and — crucially — hyperparameters that change over training. PBT might discover that a high learning rate works well early in training but a low learning rate works better later. This is something static HPO methods can't discover (unless you explicitly search over schedules).

PBT is most useful for expensive training runs where you want to simultaneously tune hyperparameters and train the final model. It's been used to great effect in reinforcement learning (where training is extremely expensive) and large-scale language model training.

# Simplified PBT concept
import copy
import random

class PBTMember:
    def __init__(self, model, config):
        self.model = model
        self.config = config
        self.score = 0.0

def pbt_step(population, exploit_fraction=0.2, perturb_factors=[0.8, 1.2]):
    """One PBT exploit-and-explore step."""
    scores = [m.score for m in population]
    sorted_pop = sorted(population, key=lambda m: m.score)
    cutoff = int(len(population) * exploit_fraction)
    
    bottom = sorted_pop[:cutoff]
    top = sorted_pop[-cutoff:]
    
    for weak in bottom:
        # Exploit: copy weights and config from a random top performer
        strong = random.choice(top)
        weak.model.load_state_dict(
            copy.deepcopy(strong.model.state_dict())
        )
        weak.config = copy.deepcopy(strong.config)
        
        # Explore: perturb hyperparameters
        for key in ['lr', 'weight_decay']:
            factor = random.choice(perturb_factors)
            weak.config[key] *= factor
        
        # Update optimizer with new config
        for pg in weak.optimizer.param_groups:
            pg['lr'] = weak.config['lr']
            pg['weight_decay'] = weak.config['weight_decay']

Optuna: Practical Hyperparameter Optimization

Optuna is the most popular Python framework for hyperparameter optimization. It uses TPE by default, supports pruning (Successive Halving, Hyperband), handles conditional search spaces, and has excellent visualization tools. Here's a complete, practical example:

import optuna
import torch
import torch.nn as nn
import torch.optim as optim
from torch.utils.data import DataLoader

def create_model(trial):
    """Define model architecture with Optuna-suggested hyperparams."""
    num_layers = trial.suggest_int('num_layers', 2, 6)
    hidden_dim = trial.suggest_categorical('hidden_dim', [64, 128, 256, 512])
    dropout = trial.suggest_float('dropout', 0.0, 0.5)
    activation = trial.suggest_categorical('activation', ['relu', 'gelu', 'silu'])
    
    act_fn = {'relu': nn.ReLU, 'gelu': nn.GELU, 'silu': nn.SiLU}[activation]
    
    layers = []
    in_dim = INPUT_DIM
    for i in range(num_layers):
        layers.extend([
            nn.Linear(in_dim, hidden_dim),
            act_fn(),
            nn.Dropout(dropout),
        ])
        in_dim = hidden_dim
    layers.append(nn.Linear(hidden_dim, NUM_CLASSES))
    
    return nn.Sequential(*layers)

def objective(trial):
    """Full training loop as Optuna objective."""
    # Hyperparameters
    lr = trial.suggest_float('lr', 1e-5, 1e-2, log=True)
    weight_decay = trial.suggest_float('weight_decay', 1e-6, 1e-2, log=True)
    batch_size = trial.suggest_categorical('batch_size', [32, 64, 128])
    optimizer_name = trial.suggest_categorical('optimizer', ['adam', 'adamw', 'sgd'])
    
    # Create model and data
    model = create_model(trial).to(device)
    train_loader = DataLoader(train_dataset, batch_size=batch_size, shuffle=True)
    
    # Conditional hyperparameter: momentum only for SGD
    if optimizer_name == 'sgd':
        momentum = trial.suggest_float('momentum', 0.8, 0.99)
        optimizer = optim.SGD(model.parameters(), lr=lr,
                              weight_decay=weight_decay, momentum=momentum)
    elif optimizer_name == 'adamw':
        optimizer = optim.AdamW(model.parameters(), lr=lr,
                                weight_decay=weight_decay)
    else:
        optimizer = optim.Adam(model.parameters(), lr=lr,
                               weight_decay=weight_decay)
    
    criterion = nn.CrossEntropyLoss()
    
    # Training with intermediate reporting for pruning
    for epoch in range(30):
        model.train()
        for inputs, targets in train_loader:
            inputs, targets = inputs.to(device), targets.to(device)
            optimizer.zero_grad()
            loss = criterion(model(inputs), targets)
            loss.backward()
            optimizer.step()
        
        # Evaluate
        model.eval()
        correct, total = 0, 0
        with torch.no_grad():
            for inputs, targets in val_loader:
                inputs, targets = inputs.to(device), targets.to(device)
                preds = model(inputs).argmax(dim=-1)
                correct += (preds == targets).sum().item()
                total += targets.size(0)
        val_acc = correct / total
        
        # Report for pruning
        trial.report(val_acc, epoch)
        if trial.should_prune():
            raise optuna.TrialPruned()
    
    return val_acc

# Run the optimization
study = optuna.create_study(
    study_name='my-dl-experiment',
    direction='maximize',
    pruner=optuna.pruners.MedianPruner(n_warmup_steps=5),
    storage='sqlite:///optuna_study.db',  # Persistent storage
)

study.optimize(objective, n_trials=100, timeout=3600 * 4)  # 4 hour budget

# Results
print(f"\nBest trial:")
print(f"  Value (val acc): {study.best_trial.value:.4f}")
print(f"  Params: {study.best_trial.params}")

# Visualization
fig1 = optuna.visualization.plot_optimization_history(study)
fig2 = optuna.visualization.plot_param_importances(study)
fig3 = optuna.visualization.plot_parallel_coordinate(study)
fig1.show()
fig2.show()
fig3.show()

Key Optuna Features

Define-by-run API. Unlike frameworks that require you to declare the full search space upfront, Optuna defines hyperparameters inside the objective function using trial.suggest_*. This means conditional search spaces are natural — you only call trial.suggest_float('momentum', ...) inside an if optimizer == 'sgd' block.

Pruning. By calling trial.report(metric, step) during training, Optuna can terminate unpromising trials early. The MedianPruner kills a trial if its intermediate metric is worse than the median of all completed trials at the same step. HyperbandPruner uses the Hyperband algorithm for more sophisticated early stopping.

Persistent storage. By specifying storage='sqlite:///...', the study is saved to a database. You can resume optimization, run distributed workers, and analyze results after the fact. For large-scale distributed HPO, use a PostgreSQL backend instead of SQLite.

Parameter importance. After running many trials, plot_param_importances shows which hyperparameters had the most impact on performance. This guides future tuning — focus on the parameters that matter, widen their ranges, and fix the unimportant ones to reasonable defaults.

Ray Tune: Distributed Hyperparameter Optimization

Ray Tune is the go-to framework when you need to scale HPO across multiple GPUs or machines. It integrates with Optuna's search algorithms but adds distributed scheduling, fault tolerance, and resource management.

from ray import tune
from ray.tune.schedulers import ASHAScheduler
from ray.tune.search.optuna import OptunaSearch

search_space = {
    "lr": tune.loguniform(1e-5, 1e-2),
    "batch_size": tune.choice([32, 64, 128]),
    "num_layers": tune.randint(2, 8),
    "hidden_dim": tune.choice([128, 256, 512]),
    "dropout": tune.uniform(0.0, 0.5),
}

scheduler = ASHAScheduler(
    max_t=50,           # Max epochs per trial
    grace_period=5,     # Min epochs before pruning
    reduction_factor=2, # Halve survivors each round
)

tuner = tune.Tuner(
    train_function,  # Your training function
    param_space=search_space,
    tune_config=tune.TuneConfig(
        metric="val_acc",
        mode="max",
        num_samples=100,
        scheduler=scheduler,
        search_alg=OptunaSearch(),
    ),
    run_config=tune.RunConfig(
        name="my_experiment",
        storage_path="./ray_results",
    ),
)

results = tuner.fit()
best = results.get_best_result()
print(f"Best config: {best.config}")
print(f"Best val_acc: {best.metrics['val_acc']:.4f}")

Ray Tune's main advantage is automatic resource management. If you have 8 GPUs and each trial needs 1 GPU, it runs 8 trials in parallel. If a trial is pruned, its GPU is immediately reassigned. For teams with access to GPU clusters, this can reduce wall-clock HPO time from days to hours.

Practical HPO Strategy

Given finite compute, here's a sensible approach:

  1. Start with manual tuning. Train a few runs by hand, varying learning rate by orders of magnitude. Get a rough sense of what works. This builds intuition and catches bugs before wasting compute on automated search.
  2. Define a focused search space. Based on manual tuning, narrow the ranges. Don't search learning rate from 1e-7 to 10 if you know 1e-4 to 1e-2 is the right neighborhood.
  3. Run Optuna with pruning. Use TPE + MedianPruner or Hyperband. 50–100 trials is usually enough for 5–10 hyperparameters. Report intermediate metrics so unpromising trials die fast.
  4. Analyze importances. Use Optuna's visualization to identify which hyperparameters actually mattered. Fix unimportant ones and re-run with a tighter space for the important ones.
  5. Final training. Take the best configuration and train to completion with full budget, possibly with a longer schedule. Repeat 3–5 times with different seeds to estimate variance.

The honest truth: for most projects, careful random search with 30–50 trials and good pruning gets you 95% of the way to the optimal configuration. Bayesian optimization squeezes out the last few percent, which matters when you're competing on leaderboards or deploying at scale — but not for prototyping or research exploration.
