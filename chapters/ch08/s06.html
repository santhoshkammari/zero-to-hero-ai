Distributed Training & Hardware

TL;DR

Deep learning runs on GPUs because matrix multiplications parallelize across thousands of cores. VRAM is your primary constraint—model weights, optimizer states, and activations all compete for it. When one GPU isn't enough, DistributedDataParallel (DDP) gives near-linear scaling across multiple GPUs with minimal code changes. For models too large to fit on a single GPU, Fully Sharded Data Parallel (FSDP) or model parallelism splits the model itself. Knowing how to estimate VRAM, launch DDP, and pick the right parallelism strategy is the difference between training successfully and staring at OOM errors.

GPU Fundamentals: Why GPUs for Deep Learning

A CPU has a handful of powerful cores optimized for complex, sequential logic. A GPU has thousands of simpler cores designed to do the same operation on many data points simultaneously. Deep learning is almost entirely matrix multiplication, element-wise operations, and reductions—operations that decompose perfectly into thousands of independent threads. A single matrix multiply that takes a CPU several seconds completes on a GPU in milliseconds.

That's the good news. The constraint is VRAM (Video RAM)—the GPU's dedicated memory. Unlike system RAM, VRAM is limited and not easily expandable. Your model's parameters, the optimizer's internal states, intermediate activations from the forward pass, and the gradients from the backward pass all live in VRAM simultaneously during training.

⚠️ The Error You'll See Most

RuntimeError: CUDA out of memory. Tried to allocate X MiB — this is the single most common error in deep learning. It means your VRAM is full. Before Googling exotic solutions, try the simple ones first: reduce batch size, enable mixed precision (torch.cuda.amp), use gradient accumulation, or enable gradient checkpointing. These four fixes solve 90% of OOM errors.

VRAM Budget: Back-of-the-Envelope Math

Before you launch a training run, you should be able to estimate whether your model will even fit. Here's the arithmetic:

Parameter Memory

Each parameter is a floating-point number. In float32, that's 4 bytes. In float16 or bfloat16, it's 2 bytes.

# Quick parameter count
total_params = sum(p.numel() for p in model.parameters())
print(f"Parameters: {total_params / 1e9:.2f}B")
print(f"  float32: {total_params * 4 / 1e9:.2f} GB")
print(f"  float16: {total_params * 2 / 1e9:.2f} GB")

Optimizer State Memory

Adam (and AdamW) stores two extra tensors per parameter: the first moment (mean of gradients) and second moment (mean of squared gradients). Each is the same size as the parameter itself. So Adam adds 2× the parameter memory on top of the parameters and gradients.

Total training memory for parameters + gradients + optimizer:

PrecisionParamsGradientsAdam StatesTotal
float324 bytes4 bytes8 bytes (2×4)16 bytes per param
Mixed (float16 weights, float32 optimizer)2 bytes2 bytes8 bytes + 4 bytes master copy~16 bytes per param

Activation Memory

Activations are the intermediate outputs saved during the forward pass so they're available for the backward pass. Their size scales with batch size, sequence length (or spatial dimensions), hidden dimensions, and number of layers. There's no simple formula—it depends on architecture—but activations often dominate VRAM for large batch sizes.

Worked Example: 1B Parameter Model

Componentfloat16 Training
Parameters (1B × 2 bytes)~2 GB
Gradients (1B × 2 bytes)~2 GB
Adam states (1B × 8 bytes, kept in float32)~8 GB
Master weights (1B × 4 bytes, float32 copy)~4 GB
Activations (varies with batch size)~2-8 GB
Total~18-24 GB

A 1B parameter model in mixed precision barely fits on a 24GB GPU (RTX 3090/4090) with a small batch size. For comfortable training, you'd want 40GB+ (A100-40GB) or use gradient checkpointing to trade compute for memory.

💡 Quick Rule of Thumb

For mixed-precision training with Adam: multiply your parameter count by ~18-20 bytes for a rough VRAM estimate (before activations). A 7B model needs ~130-140 GB just for parameters/gradients/optimizer—that's two A100-80GB GPUs minimum, before activations.

Multi-GPU Training: DataParallel vs DistributedDataParallel

Once your model or batch size won't fit on a single GPU—or training is simply too slow—you need multiple GPUs. PyTorch offers two main approaches for data parallelism: DataParallel (DP) and DistributedDataParallel (DDP). One of them you should never use.

DataParallel (DP) — Don't Use This

nn.DataParallel is the tempting one-liner:

# DON'T do this in any serious project
model = nn.DataParallel(model)

It's simple, but fatally flawed:

  • Single-process: One Python process manages all GPUs, hitting Python's GIL (Global Interpreter Lock)
  • Asymmetric memory: GPU 0 gathers all outputs and computes the loss, using far more memory than other GPUs
  • Communication bottleneck: Gradients are gathered to GPU 0, then broadcast back—sequential, slow
  • Poor scaling: Often slower with 4 GPUs than 2 because the GPU-0 bottleneck dominates
⚠️ Production Rule

nn.DataParallel exists for historical reasons. Never use it. Always use DistributedDataParallel. The extra setup is minimal, and the performance difference is enormous. Even on a single machine with multiple GPUs, DDP is the correct choice.

DistributedDataParallel (DDP) — The Standard

DDP spawns one process per GPU. Each process has its own copy of the model, receives a different slice of data, and computes gradients independently. Gradients are then synchronized across all processes using an all-reduce operation (typically ring all-reduce via NCCL), which is highly optimized and overlaps with backward computation.

The result: near-linear scaling. 4 GPUs ≈ 4× throughput.

Here's the minimal DDP setup pattern you'll reuse in every distributed training script:

import os
import torch
import torch.distributed as dist
import torch.nn as nn
from torch.nn.parallel import DistributedDataParallel as DDP
from torch.utils.data import DataLoader, DistributedSampler

def setup(rank, world_size):
    """Initialize the distributed process group."""
    os.environ["MASTER_ADDR"] = "localhost"
    os.environ["MASTER_PORT"] = "12355"
    dist.init_process_group("nccl", rank=rank, world_size=world_size)
    torch.cuda.set_device(rank)

def cleanup():
    dist.destroy_process_group()

def train(rank, world_size):
    setup(rank, world_size)

    # Create model on the correct GPU
    model = MyModel().to(rank)
    model = DDP(model, device_ids=[rank])

    # DistributedSampler ensures each GPU gets different data
    dataset = MyDataset()
    sampler = DistributedSampler(dataset, num_replicas=world_size, rank=rank)
    dataloader = DataLoader(dataset, batch_size=32, sampler=sampler)

    optimizer = torch.optim.AdamW(model.parameters(), lr=1e-4)

    for epoch in range(num_epochs):
        sampler.set_epoch(epoch)  # IMPORTANT: shuffle differently each epoch
        for batch in dataloader:
            batch = batch.to(rank)
            loss = model(batch)
            loss.backward()
            optimizer.step()
            optimizer.zero_grad()

    cleanup()

# Launch: DON'T call train() directly — use torchrun
# $ torchrun --nproc_per_node=4 train_script.py

Key things to notice:

  • init_process_group("nccl") — NCCL is NVIDIA's multi-GPU communication library. Always use it for GPU training.
  • DistributedSampler — splits the dataset so each GPU processes a unique subset. Without this, all GPUs would train on identical data (wasting compute).
  • sampler.set_epoch(epoch) — ensures different shuffling each epoch. Forgetting this means the same data split every epoch.
  • torchrun — the launcher that spawns one process per GPU. Don't use mp.spawn() in new code; torchrun handles environment variables, fault tolerance, and elastic scaling.

FSDP: Fully Sharded Data Parallel

DDP replicates the entire model on every GPU. That works fine when the model fits in a single GPU's memory. But what about a 7B or 70B parameter model that doesn't?

FSDP (Fully Sharded Data Parallel) is PyTorch's answer. Instead of replicating the full model, FSDP shards the model parameters, gradients, and optimizer states across all GPUs. Each GPU only holds a fraction of the model at any time. When a layer needs its full parameters for a forward or backward pass, FSDP gathers them temporarily from other GPUs, computes, then discards the non-local shards.

If you've heard of DeepSpeed ZeRO (Zero Redundancy Optimizer)—FSDP is essentially the same idea, built natively into PyTorch. ZeRO has three stages:

  • ZeRO Stage 1: Shard optimizer states only
  • ZeRO Stage 2: Shard optimizer states + gradients
  • ZeRO Stage 3: Shard optimizer states + gradients + parameters (= FSDP)
from torch.distributed.fsdp import FullyShardedDataParallel as FSDP

# Wrap your model with FSDP — replaces the DDP wrapper
model = FSDP(
    model,
    # Shard parameters across GPUs
    use_orig_params=True,
    # Mixed precision for efficiency
    mixed_precision=MixedPrecision(
        param_dtype=torch.bfloat16,
        reduce_dtype=torch.bfloat16,
        buffer_dtype=torch.bfloat16,
    ),
)
# Rest of training loop is similar to DDP

The tradeoff: FSDP adds communication overhead (gathering/scattering parameters for each layer) in exchange for dramatically reduced per-GPU memory. For models that fit on a single GPU, DDP is simpler and faster. For models that don't, FSDP makes training possible.

Data Parallelism vs Model Parallelism

These are the two fundamental strategies for distributing training. They solve different problems and are often combined.

Data Parallelism

Same model on every GPU, different data on each. This is what DDP and FSDP do. It scales training throughput—you process more batches per second—but each GPU must still hold at least a working portion of the model.

Model Parallelism

The model itself is split across GPUs. There are two main flavors:

Tensor Parallelism — a single layer's computation is split across GPUs. For example, a large matrix multiply is split so each GPU computes a portion of the output, then the results are combined. This is used within transformer layers (splitting attention heads or the MLP across GPUs). Very fast on high-bandwidth interconnects (NVLink), but requires tight coupling between GPUs.

Pipeline Parallelism — different layers are assigned to different GPUs. GPU 0 runs layers 1-10, GPU 1 runs layers 11-20, etc. Data flows through the pipeline. The challenge is pipeline bubbles: while GPU 1 waits for GPU 0's output, it sits idle. Micro-batching (splitting a batch into smaller chunks that flow through the pipeline) reduces this waste but doesn't eliminate it.

StrategyWhat's DistributedWhen to Use
Data Parallel (DDP)Data (different batches per GPU)Model fits on one GPU; need more throughput
FSDP / ZeROData + sharded model stateModel's training state doesn't fit on one GPU
Tensor ParallelIndividual layer computationSingle layers are too large; high-bandwidth interconnect available
Pipeline ParallelGroups of layersVery deep models; willing to accept some bubble overhead
💡 Modern Large-Model Training

State-of-the-art systems like Megatron-LM and the infrastructure behind GPT-4, LLaMA, and Gemini use 3D parallelism: data parallelism × tensor parallelism × pipeline parallelism, combined with FSDP/ZeRO. You're unlikely to need this unless you're training models with tens of billions of parameters, but knowing the vocabulary helps you read papers and engineering blogs.

Practical Multi-GPU Tips

Distributed training is full of subtle gotchas. Here are the ones that will save you debugging time:

1. Always Use DDP, Never DP

This bears repeating. Even for a quick experiment on a 2-GPU machine, the extra 15 lines of DDP setup are worth it. nn.DataParallel will waste your GPUs.

2. Scale Learning Rate with Effective Batch Size

If you go from 1 GPU with batch size 32 to 4 GPUs with batch size 32 each, your effective batch size is now 128. The linear scaling rule says: multiply your learning rate by the same factor you multiplied the batch size. Going from 32 → 128 means 4× the learning rate. But large learning rates can be unstable at the start, so pair this with a warmup period (gradually ramp up the LR over the first few hundred steps).

# Linear scaling rule
base_lr = 1e-4
base_batch_size = 32
effective_batch_size = batch_size_per_gpu * world_size  # e.g., 32 * 4 = 128
scaled_lr = base_lr * (effective_batch_size / base_batch_size)
# scaled_lr = 4e-4

# Use a warmup scheduler to avoid instability
from torch.optim.lr_scheduler import LinearLR, CosineAnnealingLR, SequentialLR

warmup = LinearLR(optimizer, start_factor=0.1, total_iters=500)
cosine = CosineAnnealingLR(optimizer, T_max=total_steps - 500)
scheduler = SequentialLR(optimizer, [warmup, cosine], milestones=[500])

3. Use DistributedSampler Correctly

Without DistributedSampler, every GPU sees the entire dataset and processes the same batches—you get no speedup, just wasted electricity. And don't forget sampler.set_epoch(epoch) to reshuffle each epoch.

4. Sync BatchNorm Across GPUs

Standard BatchNorm computes statistics per-GPU. With small per-GPU batch sizes, these statistics become noisy and hurt performance. SyncBatchNorm computes statistics across all GPUs:

# Convert all BatchNorm layers to SyncBatchNorm BEFORE wrapping with DDP
model = nn.SyncBatchNorm.convert_sync_batchnorm(model)
model = model.to(rank)
model = DDP(model, device_ids=[rank])

5. Log and Save Only on Rank 0

Every process runs the same training loop. If you log metrics or save checkpoints from every process, you'll get duplicate logs and corrupted checkpoints. Guard these operations:

if dist.get_rank() == 0:
    print(f"Epoch {epoch}, Loss: {loss.item():.4f}")
    wandb.log({"loss": loss.item()})
    torch.save(model.module.state_dict(), "checkpoint.pt")
# Note: model.module to access the unwrapped model inside DDP

6. Mind the Interconnect

GPU-to-GPU communication speed matters enormously. NVLink (600 GB/s on H100) is dramatically faster than PCIe (64 GB/s). On cloud instances, check that your GPUs are connected via NVLink/NVSwitch—not just PCIe. For multi-node training, InfiniBand or high-speed Ethernet is essential.

Training Infrastructure

Hardware is only part of the story. The tooling around your training runs determines whether you can reproduce results, debug failures, and iterate quickly.

Experiment Tracking

This is arguably the most important infrastructure decision you'll make. Track every experiment from day 1. You will never remember what hyperparameters produced that good result from three weeks ago.

  • Weights & Biases (W&B) — the most popular choice. Logs metrics, hyperparameters, system stats, model checkpoints, and artifacts. Beautiful dashboard. Free tier is generous.
  • MLflow — open-source alternative. Self-hosted or managed. Good if you need to keep data on-prem.
  • TensorBoard — built into PyTorch/TensorFlow. Basic but functional. Good for quick local experiments.
import wandb

Mixed Precision & Hyperparameter Optimization

TL;DR

Mixed precision training is a free lunch: add 4 lines to your PyTorch loop, get ~2× speedup and ~40% less memory with negligible accuracy loss. Hyperparameter optimization (HPO) is about spending your compute budget wisely — random search beats grid search almost always, and Bayesian methods (Optuna) beat both. The punchline: most models are undertrained, not undertuned. Fix your training recipe before doing extensive HPO.

Mixed Precision Training

What It Is and Why You Want It

By default, neural network training uses float32 (FP32) — 32 bits per number. It's precise, it's safe, and it's wasteful. Most of the computations in a neural network — matrix multiplies, convolutions, element-wise operations — don't need 7 decimal digits of precision. They do just fine with 3.

Float16 (FP16, half precision) uses 16 bits per number. It's 2× smaller in memory and 2× faster on modern GPU Tensor Cores. The problem: FP16 has a much narrower dynamic range (max ~65,504 vs FP32's ~3.4 × 10³⁸). Small gradients can underflow to zero and vanish silently.

Mixed precision is the practical solution. Keep a master copy of weights in FP32 (for the optimizer), run the forward and backward passes in FP16 (for speed), and use loss scaling to prevent gradient underflow. The result:

  • ~2× speedup on Tensor Core GPUs (V100, A100, H100)
  • ~40% less GPU memory (activations are halved)
  • Negligible accuracy loss — SGD noise dwarfs FP16 rounding errors

BF16 vs FP16

Not all 16-bit formats are created equal.

BFloat16 (BF16) was designed by Google to keep the same exponent range as FP32 (8 exponent bits) while sacrificing mantissa precision (7 bits instead of FP16's 10). This means BF16 can represent the same range of magnitudes as FP32 — no more gradient underflow, no more loss scaling headaches. The trade-off is slightly less numerical precision, which rarely matters for training.

FormatBitsExponentMantissaMax ValueLoss Scaling Needed?
FP3232823~3.4 × 10³⁸No
FP1616510~65,504Yes
BF161687~3.4 × 10³⁸No

Rule of thumb: if your hardware supports BF16 (A100+, TPUs), use BF16 — it's simpler and just as fast. If you're on a V100 or older, use FP16 with loss scaling.

PyTorch AMP — The 4-Line Upgrade

PyTorch's Automatic Mixed Precision (AMP) handles everything: which ops run in FP16 vs FP32, dynamic loss scaling, gradient checking. Here's the pattern:

from torch.amp import autocast, GradScaler

scaler = GradScaler("cuda")

for inputs, targets in dataloader:
    optimizer.zero_grad()

    with autocast("cuda"):                       # forward in FP16
        outputs = model(inputs)
        loss = criterion(outputs, targets)

    scaler.scale(loss).backward()                # scaled backward
    scaler.step(optimizer)                       # unscale → check → step
    scaler.update()                              # adjust scale factor

That's it. Four extra lines — GradScaler, autocast, scaler.scale, scaler.update — for a ~2× speedup. The autocast context manager routes matmuls and convolutions to FP16 Tensor Cores while keeping numerically sensitive ops (softmax, layer norm, loss functions) in FP32. The GradScaler dynamically adjusts the loss scaling factor: if gradients overflow, it halves the scale and skips the step; if things are stable, it doubles the scale to capture smaller gradients.

For BF16 on supported hardware, it's even simpler — no scaler needed:

for inputs, targets in dataloader:
    optimizer.zero_grad()

    with autocast("cuda", dtype=torch.bfloat16):
        outputs = model(inputs)
        loss = criterion(outputs, targets)

    loss.backward()
    optimizer.step()

When NOT to Use Mixed Precision

  • Loss functions that need high precision — some custom losses with tiny epsilon terms break in FP16. AMP handles standard losses, but custom ones may need manual FP32 casts.
  • Tiny models — if your model fits easily in memory and trains in minutes, the overhead of AMP (casting, scaling logic) isn't worth the complexity.
  • Debugging numerical issues — always debug in FP32 first. If training diverges, rule out numerical precision before investigating other causes.
  • CPU-only training — mixed precision benefits come from GPU Tensor Cores. On CPU, there's no speedup.
💡 Key Insight

Mixed precision is not an approximation — it's an engineering optimization. With FP32 master weights and proper loss scaling, the mathematical updates are identical to full FP32 training. The rounding noise from FP16 is completely dominated by the inherent noise of mini-batch SGD. If your mixed precision run diverges but your FP32 run doesn't, the bug is in your scaling setup, not in the concept.

Hyperparameter Optimization

What Are Hyperparameters?

Neural networks learn their parameters (weights, biases) through gradient descent. But there's a whole other set of choices you make before training starts — choices that gradient descent can't help with:

  • Learning rate — the single most important hyperparameter
  • Batch size — affects generalization, training speed, and memory
  • Weight decay — L2 regularization strength
  • Dropout rate — how aggressively you regularize
  • Architecture choices — number of layers, hidden size, number of attention heads
  • Optimizer choice — Adam vs SGD vs AdamW
  • Learning rate schedule — cosine, linear warmup, step decay

These are hyperparameters. You don't learn them — you search for them. The question is how to search efficiently.

Grid Search

The naive approach. Pick a set of values for each hyperparameter, try every combination.

# Grid search: try every combination
learning_rates = [1e-4, 1e-3, 1e-2]
batch_sizes    = [32, 64, 128]
dropouts       = [0.1, 0.3, 0.5]

# 3 × 3 × 3 = 27 experiments — manageable
# But add two more hyperparameters with 3 values each:
# 3 × 3 × 3 × 3 × 3 = 243 experiments — getting expensive
# With 5 hyperparameters and 5 values each:
# 5⁵ = 3,125 experiments — don't.

Grid search is exhaustive and exponential in the number of hyperparameters. It works fine for 1–2 hyperparameters. Beyond that, it's a waste of compute. The fundamental problem: grid search allocates equal budget to every hyperparameter, even the ones that barely matter.

Random Search

Instead of a grid, sample hyperparameters randomly from distributions. This sounds careless, but it's surprisingly effective — and the reason is subtle.

Most hyperparameters don't matter equally. In a typical setup, the learning rate might explain 80% of the performance variation, while dropout explains 5%. Grid search wastes most of its budget exploring the dropout dimension at the expense of the learning rate dimension. Random search, by contrast, gives you a unique learning rate value for every experiment, exploring that important dimension much more thoroughly.

This is the core insight from Bergstra & Bengio (2012): when only a few hyperparameters matter (which is almost always the case), random search finds good configurations faster than grid search, even with the same total budget. With 60 random trials, you have a 95% chance of finding a configuration within the top 5% of the search space for the important dimensions.

💡 Why Random Beats Grid

Imagine learning rate matters a lot but dropout barely matters. A 5×5 grid gives you only 5 unique learning rates across 25 experiments. Random search with 25 experiments gives you 25 unique learning rates. You're sampling the important dimension 5× more densely for the same budget.

Bayesian Optimization

Random search is good, but it's memoryless — each trial ignores the results of all previous trials. Bayesian optimization remembers.

The idea: build a surrogate model (a probabilistic model) of the objective function — how validation loss varies as a function of hyperparameters. After each trial, update the surrogate with the new data point. Then use an acquisition function to decide where to sample next, balancing:

  • Exploitation — sample near the best configurations found so far
  • Exploration — sample in regions of high uncertainty (where the surrogate has little data)

Common surrogate models include Gaussian Processes (for small search spaces) and Tree-structured Parzen Estimators (TPE, which scales better to high dimensions). The acquisition function (Expected Improvement, Upper Confidence Bound, etc.) turns the surrogate's predictions into a score for "how promising is this configuration?"

In practice, Bayesian optimization consistently finds better hyperparameters in fewer trials than random search — typically 2–5× more efficient for 10+ dimensional search spaces.

Optuna — The Practical Tool

You don't need to implement Bayesian optimization from scratch. Optuna is the gold standard for HPO in Python. It uses TPE by default, supports pruning (killing bad trials early), and integrates cleanly with any training loop.

import optuna

def objective(trial):
    # Suggest hyperparameters from distributions
    lr = trial.suggest_float('lr', 1e-5, 1e-2, log=True)
    weight_decay = trial.suggest_float('weight_decay', 1e-6, 1e-2, log=True)
    dropout = trial.suggest_float('dropout', 0.0, 0.5)
    n_layers = trial.suggest_int('n_layers', 2, 6)
    hidden_size = trial.suggest_categorical('hidden_size', [128, 256, 512])

    # Build model with these hyperparameters
    model = build_model(n_layers, hidden_size, dropout)
    optimizer = torch.optim.AdamW(model.parameters(), lr=lr,
                                   weight_decay=weight_decay)

    # Train and evaluate
    for epoch in range(num_epochs):
        train_one_epoch(model, optimizer, train_loader)
        val_loss = evaluate(model, val_loader)

        # Report intermediate result for pruning
        trial.report(val_loss, epoch)
        if trial.should_prune():
            raise optuna.TrialPruned()

    return val_loss

# Create study and optimize
study = optuna.create_study(direction='minimize')
study.optimize(objective, n_trials=100)

# Best hyperparameters
print(study.best_params)
# {'lr': 0.00032, 'weight_decay': 0.00014, 'dropout': 0.18, ...}

Key Optuna features worth knowing:

  • Pruning — stop bad trials early using a MedianPruner or HyperbandPruner. If a trial is performing worse than the median of completed trials at the same epoch, kill it. This alone can save 50–80% of your HPO compute budget.
  • Visualization — built-in plots for parameter importance, optimization history, and parallel coordinate charts. optuna.visualization.plot_param_importances(study) tells you which hyperparameters actually matter.
  • Parallel execution — point multiple workers at the same study (backed by a shared database) and they'll coordinate automatically. Scales linearly with the number of GPUs.
  • Log-scale sampling — use log=True for any hyperparameter that spans orders of magnitude (learning rate, weight decay). This ensures uniform coverage across scales.

HPO Methods Compared

Method Efficiency Best For Scales with Dimensions Ease of Use
Grid Search Low 1–2 hyperparameters; quick sanity checks Exponential blowup Trivial
Random Search Medium Early exploration; <10 hyperparameters Linear — unaffected by dimensionality Easy
Bayesian (Optuna) High Serious HPO; expensive training runs Sublinear — learns which dims matter Moderate (requires objective function)

For most practitioners: start with random search to understand the landscape, then switch to Optuna once you've narrowed the ranges and want to squeeze out the last bit of performance.

Practical HPO Advice

Lessons from thousands of collective GPU-hours spent tuning models:

1. Tune the most impactful hyperparameters first. The priority order, from experience across many domains:

  1. Learning rate — almost always the most important. A 10× wrong learning rate is worse than getting every other hyperparameter wrong combined.
  2. Batch size — affects both optimization dynamics and generalization.
  3. Weight decay — critical for transformers, less so for CNNs with dropout.
  4. Architecture (depth, width) — matters, but less than most people think within a reasonable range.
  5. Dropout / regularization — usually has a relatively narrow optimal range.

2. Always use log-uniform for learning rate. The difference between 1e-4 and 2e-4 matters about as much as the difference between 1e-3 and 2e-3. Linear sampling wastes budget on the high end. trial.suggest_float('lr', 1e-5, 1e-2, log=True) — always.

3. Use early stopping and pruning. Don't run every trial to completion. If a trial is clearly bad after 20% of training, kill it. Optuna's MedianPruner does this automatically. Combined with Bayesian search, you can evaluate 100 configurations in the wall-clock time of 30 full runs.

4. Start coarse, then refine. First run: wide ranges, 30–50 random trials. Identify the promising region. Second run: narrow ranges around the best region, 50–100 Bayesian trials. This two-phase approach is far more efficient than one massive search.

5. Don't tune what doesn't matter. If Optuna's importance plot shows that a hyperparameter explains <2% of variance, fix it at a reasonable default and remove it from the search. Fewer dimensions = faster convergence for everything else.

⚠️ The Most Common HPO Mistake

Most models are undertrained, not undertuned. Before launching a 200-trial HPO study, ask: Am I training long enough? Is my data pipeline a bottleneck? Is my learning rate schedule reasonable? Fixing your training recipe — longer warmup, proper weight decay, cosine schedule — often delivers more improvement than any amount of hyperparameter search. HPO is the last step, not the first.
