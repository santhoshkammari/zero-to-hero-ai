Scaling & Efficiency

Scaling deep learning training means two things: make one model train faster across multiple devices, and stop wasting compute on things that do not matter. First half: distributed training — split work across GPUs. Second: efficiency — mixed precision, memory tricks, and scaling laws linking model size, data, and compute. Both are now table stakes for non-trivial training.

The VRAM Problem: Where All Your Memory Goes

Start with reason everything else exists. Your GPU has fixed VRAM — Video RAM, GPU's dedicated memory. Everything during training must live there at same time: model parameters, gradients from backpropagation, optimizer bookkeeping, and intermediate activations saved during forward pass so backward pass can use them.

Concrete picture. Train modest 1-billion parameter model. Each parameter is floating-point number. In standard float32, that is 4 bytes each. Parameters alone use 4 GB. But parameters are small part.

Gradients from backward pass match parameter size — another 4 GB. Optimizer state hurts most. Adam (and AdamW, what almost everyone uses) stores two extra tensors per parameter: running mean of gradients (first moment) and running mean of squared gradients (second moment). That is another 8 GB. Before one data point, your 1B model already uses 16 GB for parameters, gradients, and optimizer state alone.

Then activations — intermediate outputs of every layer, saved during forward pass because backpropagation needs them. Size scales with batch size, sequence length, hidden dimension, and layer count. For transformer with reasonable batch size, activations can easily double or triple total memory use.

Make it tangible. Arithmetic for 1B parameter model using mixed precision (float16 weights, float32 optimizer):

ComponentSize
Parameters (1B × 2 bytes in FP16)~2 GB
Gradients (1B × 2 bytes)~2 GB
Adam states (1B × 8 bytes, kept in FP32)~8 GB
Master weights (1B × 4 bytes, FP32 copy)~4 GB
Activations (varies with batch size)~2–8 GB
Total~18–24 GB

That 1B model barely fits on 24 GB consumer GPU (RTX 3090/4090) with small batch size. Now think about 7B model: 7× everything, roughly 130–140 GB for parameters, gradients, and optimizer state alone — before activations. Need at least two A100-80GB GPUs, and throughput still not addressed.

This drives everything in this section. Every technique here — distributed training, mixed precision, gradient checkpointing, sharding — exists because VRAM runs out. Need split work or shrink memory footprint.

⚠️ The Error You'll See Most

RuntimeError: CUDA out of memory. Tried to allocate X MiB — most common deep learning error. Before exotic fixes, try four simple ones in order: reduce batch size, enable mixed precision, use gradient accumulation, enable gradient checkpointing. These solve 90% of OOM errors.

Mixed Precision: The Free Lunch

First tool when memory tight — and even when not — is mixed precision training. Idea simple. Most neural network arithmetic does not need full float32 precision. Matrix multiplications, convolutions, element-wise operations work fine with half bits. Run expensive parts in float16 (or bfloat16), keep master weight copy in float32 for optimizer, get both benefits.

Payoff real: about 2× speedup on Tensor Core GPUs (V100 onward) and about 40% less memory because activations halve. Accuracy cost is negligible — rounding noise from 16-bit arithmetic gets drowned out by mini-batch stochastic gradient descent noise.

One wrinkle. Float16 has narrow dynamic range — max value is about 65,504, versus float32's 3.4 × 10³⁸. Small gradients can underflow to zero and vanish. Fix: loss scaling. Multiply loss by large number before backward pass, push gradients into float16's representable range, then divide gradients back down before optimizer step.

BFloat16 avoids this entirely. Google designed it with same exponent range as float32 (8 exponent bits) but fewer mantissa bits (7 instead of float16's 10). Same dynamic range. Slightly less precision. No loss scaling needed. If your hardware supports BF16 (A100 and newer, or TPUs), use it. Life gets easier.

FormatBitsExponentMantissaMax ValueLoss Scaling?
FP3232823~3.4 × 10³⁸No
FP1616510~65,504Yes
BF161687~3.4 × 10³⁸No

In PyTorch, implementation is almost trivial. Four extra lines enable Automatic Mixed Precision (AMP):

from torch.amp import autocast, GradScaler

scaler = GradScaler("cuda")

for inputs, targets in dataloader:
    optimizer.zero_grad()

    with autocast("cuda"):
        outputs = model(inputs)
        loss = criterion(outputs, targets)

    scaler.scale(loss).backward()
    scaler.step(optimizer)
    scaler.update()

autocast sends matrix multiplications and convolutions to FP16 Tensor Cores, while keeping numerically sensitive operations (softmax, layer norm, loss functions) in FP32. GradScaler adjusts loss scaling dynamically — if gradients overflow, it halves scale and skips step; if stable, it gradually increases scale to capture smaller gradients.

For BF16 on A100 or newer, no scaler needed:

with autocast("cuda", dtype=torch.bfloat16):
    outputs = model(inputs)
    loss = criterion(outputs, targets)
loss.backward()
optimizer.step()

At first, "half precision, same results" sounds wrong. But math holds. Optimizer updates happen in FP32, so accumulated weight changes stay identical. Only new noise appears in forward and backward passes, and mini-batch SGD stochasticity dwarfs it. If your mixed precision run diverges but your FP32 run does not, bug is in scaling setup, not concept.

Memory Optimization: Trading Compute for Bytes

Mixed precision halves activation memory. Sometimes not enough. Two more techniques push farther by trading compute time for memory.

Gradient Accumulation

Idea almost too simple. You want effective batch size 128, but only 32 samples fit in memory. Run 4 forward-backward passes with batch size 32, accumulate gradients without stepping optimizer, then step once after all 4. Gradients match processing 128 samples at once.

accumulation_steps = 4

for i, (inputs, targets) in enumerate(dataloader):
    with autocast("cuda"):
        outputs = model(inputs)
        loss = criterion(outputs, targets) / accumulation_steps

    scaler.scale(loss).backward()

    if (i + 1) % accumulation_steps == 0:
        scaler.step(optimizer)
        scaler.update()
        optimizer.zero_grad()

Division by accumulation_steps matters — without it, effective learning rate becomes 4× too large, because gradients from 4 micro-batches get summed, not averaged. Training time per step stays same — total compute is same. Peak memory stays at single micro-batch level.

Gradient Checkpointing

This trades compute for memory more aggressively. In normal forward pass, every layer saves its output (activations) because backpropagation needs them. For a 100-layer network, memory holds 100 layers of activations.

With gradient checkpointing (also called activation checkpointing or activation recomputation), save activations only at chosen "checkpoint" layers. When backward pass reaches layer whose activations were not saved, it recomputes them on-the-fly from nearest checkpoint. This cuts activation memory from O(N) to about O(√N) for N layers, at cost of about 30% extra compute time — parts of forward pass run twice.

from torch.utils.checkpoint import checkpoint

# Instead of: output = transformer_block(x)
# Use: output = checkpoint(transformer_block, x, use_reentrant=False)

In practice, people checkpoint every transformer block or every few residual blocks. Go-to technique when model fits in memory for inference but not for training, because training must store activations and gradients on top of parameters.

Still building intuition for optimal checkpoint placement. Real tradeoff: memory reduction versus recomputation wall-clock cost. Depends heavily on architecture and hardware. Safe default: checkpoint every transformer layer and accept ~30% slowdown.

DistributedDataParallel: The Workhorse

Picture scenario. Model fits on one GPU, but training takes four days. You have 4 GPUs. Split data four ways, give each GPU same model, let each compute gradients on its shard, then combine gradients so every GPU gets same update. That is data parallelism.

PyTorch offers two implementations of this idea. One you should never touch.

DataParallel: The One You Should Never Use

nn.DataParallel looks like tempting one-liner — model = nn.DataParallel(model) — and it is badly flawed. It runs in single Python process, so it hits the GIL (Global Interpreter Lock). GPU 0 gathers all outputs and computes loss, using far more memory than other GPUs. Gradients collect on GPU 0, then broadcast back sequentially. Result: often slower on 4 GPUs than 2 because GPU-0 bottleneck dominates. Historical artifact. Ignore it.

DistributedDataParallel: How It Actually Works

DDP does this right. It spawns one process per GPU. Each process holds its own model copy, gets unique slice of training data via DistributedSampler, and computes gradients independently. Magic is gradient synchronization.

Core operation is all-reduce. Every GPU computes gradients for local batch. Every GPU must end with average of all gradients. Naive plan — send everything to one GPU, average there, send back — creates bottleneck. Ring all-reduce avoids that.

Imagine GPUs in ring. Each GPU splits gradient tensor into N chunks (N = GPU count). In first phase (reduce-scatter), each GPU sends one chunk to neighbor and receives one from other side, adding as chunks move around ring. After N-1 steps, each GPU holds fully reduced version of one chunk. In second phase (all-gather), complete chunks circulate until every GPU has full result. Key point: each GPU sends and receives simultaneously at every step, saturating network bandwidth. Total data each GPU sends is 2 × (N-1)/N times gradient size — approaching 2× gradient size no matter how many GPUs you add. That is bandwidth-optimal.

But DDP does not wait for whole backward pass before communication starts. It uses gradient bucketing: as gradients are computed layer by layer during backward pass, they are grouped into buckets (25 MB each by default). Once bucket fills, its all-reduce starts in background on separate CUDA stream, overlapping communication with computation of earlier layers' gradients. By backward-pass end, most gradient synchronization is already finished.

Result: near-linear scaling. 4 GPUs usually give 3.6–3.9× throughput of one.

Pattern you will reuse in every DDP training script:

import os, torch, torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP
from torch.utils.data import DataLoader, DistributedSampler

def setup(rank, world_size):
    os.environ["MASTER_ADDR"] = "localhost"
    os.environ["MASTER_PORT"] = "12355"
    dist.init_process_group("nccl", rank=rank, world_size=world_size)
    torch.cuda.set_device(rank)

def train(rank, world_size):
    setup(rank, world_size)

    model = MyModel().to(rank)
    model = DDP(model, device_ids=[rank])

    sampler = DistributedSampler(dataset, num_replicas=world_size, rank=rank)
    dataloader = DataLoader(dataset, batch_size=32, sampler=sampler)
    optimizer = torch.optim.AdamW(model.parameters(), lr=1e-4)

    for epoch in range(num_epochs):
        sampler.set_epoch(epoch)  # different shuffle each epoch
        for batch in dataloader:
            loss = model(batch.to(rank))
            loss.backward()
            optimizer.step()
            optimizer.zero_grad()

    dist.destroy_process_group()

# Launch with: torchrun --nproc_per_node=4 train_script.py

Few things to notice. init_process_group("nccl") uses NVIDIA's NCCL library — standard GPU-to-GPU communication backend, and implementation behind ring all-reduce. DistributedSampler splits dataset so each GPU processes unique subset; without it, all GPUs train on identical data and you gain nothing. sampler.set_epoch(epoch) ensures different shuffling each epoch — forget it, and you get same data split every epoch. torchrun is launcher that spawns one process per GPU, handles environment variables, and supports fault tolerance.

Two more gotchas. First, log metrics and save checkpoints only on rank 0 — every process runs same training loop, and saving from all ranks gives duplicate logs and corrupted checkpoint files. Access unwrapped model via model.module when saving. Second, if using BatchNorm, convert to SyncBatchNorm before wrapping with DDP, because standard BatchNorm computes statistics per GPU, and small per-GPU batch sizes make those statistics noisy.

When the Model Doesn't Fit: FSDP and DeepSpeed ZeRO

DDP replicates entire model on every GPU. For 7B parameter model with Adam, that is ~130 GB of state per GPU — more than any single GPU has. This is where sharding enters.

Core insight: most memory sitting on each GPU during DDP training is redundant. Every GPU holds full copy of optimizer state, even though each GPU needs those states only during optimizer step. With 8 GPUs, you store same 8 GB of Adam moments 8 times. That is 56 GB wasted.

DeepSpeed ZeRO (Zero Redundancy Optimizer), built by Microsoft, attacks this redundancy in three stages. Think: increasingly aggressive sharding.

ZeRO Stage 1 partitions only optimizer states across GPUs. Each GPU still holds all parameters and gradients, but stores only 1/N of optimizer state (N = GPU count). Memory per GPU drops from P + G + O to P + G + O/N.

ZeRO Stage 2 adds gradient partitioning. Each GPU holds all parameters but only 1/N of gradients and optimizer states. Memory: P + G/N + O/N. Communication pattern changes — instead of all-reduce on gradients, each GPU reduce-scatters its gradient shard directly, which is slightly more efficient.

ZeRO Stage 3 goes all the way: parameters, gradients, and optimizer states all sharded. Each GPU holds only 1/N of everything. Memory per GPU: (P + G + O) / N. When layer needs full parameters for forward or backward pass, participating GPUs gather them temporarily, compute, then discard non-local shards. Maximum memory savings. Maximum communication overhead.

PyTorch's FSDP (Fully Sharded Data Parallel) is basically ZeRO Stage 3 built into PyTorch. Same idea. Different implementation. Practical difference: FSDP integrates tightly with PyTorch's API and ecosystem, while DeepSpeed offers more advanced features (NVMe offloading, custom optimizers, more knobs) and tends to edge ahead at massive scale (64+ GPUs).

from torch.distributed.fsdp import FullyShardedDataParallel as FSDP
from torch.distributed.fsdp import MixedPrecision

model = FSDP(
    model,
    use_orig_params=True,
    mixed_precision=MixedPrecision(
        param_dtype=torch.bfloat16,
        reduce_dtype=torch.bfloat16,
        buffer_dtype=torch.bfloat16,
    ),
)

Tradeoff is always same: more sharding means less memory per GPU but more communication to gather and scatter parameters for every layer. Practical rule: use lowest ZeRO stage — or DDP if possible — that fits model. DDP first. If that OOMs, try ZeRO Stage 1. Then Stage 2. Then FSDP/ZeRO Stage 3. Each step buys memory headroom by paying communication overhead.

DeepSpeed also offers ZeRO-Infinity, extending offloading to CPU RAM and even NVMe storage. This makes model size technically bounded only by disk space, but IO overhead is large. Last-resort tool — useful for experimenting with enormous models on limited hardware, not something you want for production training.

Model Parallelism: Splitting the Model Itself

Everything so far is data parallelism — same model (or shards of it), different data on each GPU. But there is second axis: split model computation itself across GPUs. This is model parallelism. Two flavors.

Tensor Parallelism

Tensor parallelism splits individual layers across GPUs. Take large linear layer multiplying hidden state by weight matrix. If weight matrix is too wide for one GPU — or you want faster computation — split matrix column-wise across 4 GPUs, have each GPU compute its quarter of output, then combine results. In transformer architectures, this usually applies to attention heads and MLP layers: each GPU computes subset of attention heads or part of feed-forward network.

Requirement: fast interconnect between GPUs. Tensor parallelism communicates at every layer — partial results must be combined before next layer can proceed. On NVLink (600 GB/s on H100), this is practical. Over PCIe (64 GB/s), communication overhead kills throughput. That is why tensor parallelism usually stays within one node with NVLink, not across nodes.

Pipeline Parallelism

Pipeline parallelism partitions model by layers. GPU 0 gets layers 1–10, GPU 1 gets layers 11–20, and so on. Data flows through pipeline: GPU 0 processes micro-batch, sends activations to GPU 1, then immediately starts next micro-batch. In principle, all GPUs stay busy once pipeline fills.

In practice, problem appears: pipeline bubbles. At start and end of batch, some GPUs sit idle because pipeline is not full yet. With 4 stages and 4 micro-batches, first GPU finishes its forward passes before last GPU has started, and same pattern reverses during backward pass. Micro-batching helps — more micro-batches means fuller pipeline — but bubble never fully disappears. Typical pipeline efficiency is 80–90%, so 10–20% of GPU time gets wasted waiting.

Plain truth: pipeline parallelism feels least elegant of the three approaches. Bubble overhead is real, scheduling gets complicated (GPT-style interleaved schedules, 1F1B scheduling), and debugging pipeline-parallel code is painful. But for very deep models that need splits across many nodes, sometimes it is only option.

3D Parallelism: Combining Everything

State-of-the-art systems like Megatron-LM (NVIDIA's framework for training very large language models) use all three dimensions at once. A 64-GPU cluster might use 4-way data parallelism × 4-way pipeline parallelism × 4-way tensor parallelism: 4 × 4 × 4 = 64 GPUs. Tensor parallelism works within node (fast NVLink), pipeline parallelism spans nodes, and data parallelism replicates entire pipeline for higher throughput.

This is architecture behind GPT-4, LLaMA, Gemini, and every other model with tens of billions of parameters. You probably will not implement 3D parallelism yourself — frameworks like Megatron-LM, DeepSpeed, and Hugging Face Accelerate handle orchestration — but understanding vocabulary is essential for reading engineering blogs and papers about these systems.

Flash Attention: Rethinking the Memory Hierarchy

So far, memory fight has targeted parameters, gradients, and optimizer states. Another huge memory hog hides inside transformers: attention computation itself.

Standard self-attention computes N × N matrix (N = sequence length) of attention scores — queries times keys. For sequence length 4,096, that is 16 million elements per attention head. For 32 heads, half a billion numbers. Matrix gets materialized in GPU HBM (main GPU memory), then softmax runs, then values multiply it. Three reads and writes through slow HBM. For long sequences, this becomes bottleneck.

Flash Attention (Tri Dao et al., 2022) is IO-aware algorithm that restructures computation so full attention matrix never materializes. Key insight: GPUs have small, ultra-fast on-chip memory called SRAM (about 100 KB per streaming multiprocessor, versus 80 GB of HBM on an A100). Standard attention bounces data between fast SRAM and slow HBM. Flash Attention tiles computation — processes small blocks of queries, keys, and values that fit entirely in SRAM, computes partial softmax results, and combines them. Full N × N matrix never exists in memory.

Result: memory drops from O(N²) to O(N) in sequence length, and wall-clock time falls by 2–4× because GPU spends less time waiting on memory transfers and more time doing math. And this part surprises people: it is exact. Flash Attention gives mathematically identical result to standard attention. No approximation. No accuracy loss. Pure scheduling optimization that respects GPU memory hierarchy.

In practice, you do not implement Flash Attention yourself. PyTorch 2.0+ includes it automatically through torch.nn.functional.scaled_dot_product_attention, and Hugging Face transformers enable it with one flag. Standard now. No reason not to use it.

Scaling Laws: How to Spend Your Compute Budget

All techniques above answer how to train efficiently. Scaling laws answer different question: what to train. With fixed compute budget, how big should model be, and how much data should it see?

In 2020, Kaplan et al. (OpenAI) showed that language model loss decreases as power law with model size N, dataset size D, and compute budget C:

L(N, D) ≈ L∞ + kN · N-α + kD · D-β

Plain English: bigger model, lower loss. More data, lower loss. Both follow smooth, predictable curves. This makes neural network scaling remarkable — unlike most engineering problems, returns stay predictable across orders of magnitude.

But catch exists. Kaplan's analysis suggested that, for fixed compute budget, you should make model as big as possible and train on relatively little data. That pushed field into era of enormous models trained on "only" a few hundred billion tokens.

In 2022, Hoffmann et al. (DeepMind) challenged this with Chinchilla, showing that most large models were undertrained. Their refined analysis found that for fixed compute budget C, optimal allocation is roughly:

D* ≈ 20 × N

That is, train on about 20 tokens per parameter. Compute budget relates to both as C ≈ 6 × N × D (where C is in FLOPs). A 7B parameter model should ideally see about 140 billion tokens. A 70B model should see about 1.4 trillion tokens.

This single insight reshaped field. It explains why LLaMA-1 (7B) trained on 1 trillion tokens — far more than earlier models of similar size. It shifted conversation from "how big can model get" to "do we have enough data."

Best part of scaling laws: aside from high-level empirical result, no one fully knows why they hold so cleanly. We know loss decreases as power law with compute. We can fit exponents. But theoretical reason neural networks show such predictable scaling is still active research. It works. We can plan around it. We still do not fully know why.

💡 Practical Implications

Before spending $100K training large model, do Chinchilla math. C = 6ND. For compute budget C, optimal model has N = √(C/120) parameters and trains on D = 20N tokens. If you cannot get 20 tokens per parameter, make model smaller. Optimally trained smaller model beats undertrained larger one at same compute cost.

Hyperparameter Optimization: Searching Wisely

Model architecture and distributed training setup get most attention. Quieter question decides whether run succeeds: did you choose reasonable hyperparameters?

Hyperparameters are choices made before training starts — learning rate, batch size, weight decay, dropout rate, number of layers, optimizer choice, learning rate schedule. Gradient descent cannot choose these. You must search.

Grid search tries every combination from predefined grid. Exhaustive and exponential: 3 learning rates × 3 batch sizes × 3 dropout rates = 27 experiments. Add two more hyperparameters and you reach 243. Problem: grid search gives equal budget to every dimension, even ones that barely matter.

Random search samples from distributions at random. Sounds careless. Works surprisingly well. Bergstra & Bengio (2012) showed why: most hyperparameters do not matter equally. Learning rate might explain 80% of performance variation, while dropout explains 5%. A 5×5 grid gives only 5 unique learning rates across 25 experiments. Random search gives 25 unique learning rates — sampling important dimension 5× more densely. With 60 random trials, you have 95% chance of finding configuration within top 5% of search space for dimensions that matter.

Bayesian optimization goes further. It builds surrogate model — probabilistic model of how validation loss changes with hyperparameters — and uses acquisition function to decide where to sample next, balancing exploitation (near best results so far) and exploration (high-uncertainty regions). Optuna is go-to tool, using Tree-structured Parzen Estimators (TPE) by default. Combined with trial pruning (kill bad runs early), Bayesian optimization can find better hyperparameters in 2–5× fewer trials than random search.

import optuna

def objective(trial):
    lr = trial.suggest_float('lr', 1e-5, 1e-2, log=True)
    weight_decay = trial.suggest_float('weight_decay', 1e-6, 1e-2, log=True)
    dropout = trial.suggest_float('dropout', 0.0, 0.5)

    model = build_model(dropout=dropout)
    optimizer = torch.optim.AdamW(model.parameters(), lr=lr,
                                   weight_decay=weight_decay)

    for epoch in range(num_epochs):
        train_one_epoch(model, optimizer, train_loader)
        val_loss = evaluate(model, val_loader)
        trial.report(val_loss, epoch)
        if trial.should_prune():
            raise optuna.TrialPruned()

    return val_loss

study = optuna.create_study(direction='minimize')
study.optimize(objective, n_trials=100)
print(study.best_params)

Practical priority order, from hundreds of experiments: learning rate is almost always most important hyperparameter. A 10× wrong learning rate is worse than getting everything else wrong combined. Batch size comes next (it affects optimization dynamics and generalization). Weight decay is critical for transformers, less so for CNNs with dropout. Architecture choices (depth, width) matter less than most people think within reasonable range.

⚠️ The Most Common HPO Mistake

Most models are undertrained, not undertuned. Before launching 200-trial HPO study, ask: am I training long enough? Is data pipeline bottleneck? Is learning rate schedule reasonable? Fix training recipe — longer warmup, proper weight decay, cosine schedule — often gives more improvement than more hyperparameter search. HPO is last step, not first.

Putting It Together: The Decision Tree

If you are staring at training job and wondering which technique to apply, use this decision process:

Start with single GPU. Enable mixed precision (BF16 if GPU supports it, FP16 with GradScaler otherwise). Enable Flash Attention if training transformer. Free wins — no meaningful tradeoffs, no extra complexity worth worrying about.

If you hit OOM, reduce batch size first. If effective batch size gets too small for good convergence, add gradient accumulation. If activations are bottleneck (check with torch.cuda.memory_summary()), enable gradient checkpointing and accept ~30% slowdown.

If one GPU is too slow, move to DDP. Scale learning rate linearly with effective batch size and add warmup period. Use torchrun as launcher.

If model does not fit on one GPU even with mixed precision and checkpointing, move to FSDP (PyTorch native) or DeepSpeed ZeRO (if you need more advanced features). Start with lowest ZeRO stage that fits.

If you are training at true scale — tens of billions of parameters — you are in 3D parallelism territory. Use Megatron-LM or DeepSpeed's pipeline parallelism, combined with tensor parallelism within nodes and data parallelism across pipeline replicas. At this point, you probably have infrastructure engineers. This section gives enough vocabulary for productive conversations with them.

Before committing significant compute to any large training run, do Chinchilla math. Make sure model size and data budget are balanced. Optimally trained 7B model will outperform undertrained 13B model at same compute cost.
