TL;DR

Computers don't do math perfectly — they approximate. Floating-point numbers have finite precision, and naive implementations of common ML operations (softmax, log-probabilities, loss functions) will silently produce garbage. This section covers the practical numerical tricks every ML engineer needs: stability patterns like log-sum-exp, efficient solvers for linear systems, sparse matrix formats, GPU computing fundamentals, and mixed-precision training. These aren't theoretical niceties — they're the difference between a model that trains and one that returns NaN.

Floating-Point Arithmetic

Before we talk about fancy algorithms, we need to talk about how computers store numbers. Because they don't store them the way you think.

When you write 0.1 in Python, the computer doesn't store exactly one-tenth. It can't. Computers think in binary, and just like 1/3 has no finite decimal representation (0.333...), 1/10 has no finite binary representation. So the computer stores the closest binary fraction it can, and that introduces a tiny error. This is not a bug. It's a fundamental constraint of representing real numbers in finite memory.

The IEEE 754 standard defines how floating-point numbers are stored. Every number is broken into three parts: a sign bit (positive or negative), an exponent (which determines the magnitude), and a mantissa (also called the significand, which carries the precision digits). Think of it like scientific notation: -2.998 × 10⁸ has a sign (-), a mantissa (2.998), and an exponent (8). IEEE 754 does the same thing in base 2.

The two formats you'll encounter daily are float32 (single precision) and float64 (double precision). Float32 uses 1 sign bit, 8 exponent bits, and 23 mantissa bits, giving you roughly 7 decimal digits of precision. Float64 uses 1 sign bit, 11 exponent bits, and 52 mantissa bits, giving you about 15-16 decimal digits. NumPy defaults to float64. PyTorch defaults to float32. This matters more than you think.

Machine epsilon is the smallest number ε such that 1.0 + ε ≠ 1.0 in floating-point. It tells you the fundamental resolution limit of your number format. For float32, it's about 1.19 × 10⁻⁷. For float64, it's about 2.22 × 10⁻¹⁶. Any difference smaller than machine epsilon, relative to the numbers involved, simply vanishes.

import numpy as np

# Machine epsilon for different types
print(np.finfo(np.float32).eps)   # 1.1920929e-07
print(np.finfo(np.float64).eps)   # 2.220446049250313e-16

# The classic demonstration
a = np.float64(0.1) + np.float64(0.2)
print(a == 0.3)                    # False
print(f"{a:.20f}")                 # 0.30000000000000004441

# This is why you never compare floats with ==
print(np.isclose(a, 0.3))         # True (uses a tolerance)

# Precision limits in float32
big = np.float32(1e7)
small = np.float32(1.0)
print(big + small == big)          # True — the 1.0 is lost entirely

Why does this matter for ML? Gradient accumulation is essentially adding many small numbers to a large one, thousands of times. If your loss is 1000.0 and a gradient update is 0.00001, that update might get swallowed entirely in float32. Loss comparisons using == will randomly fail. Running sums drift. These aren't hypothetical problems — they're Tuesday afternoon bugs that cost you hours of debugging.

Watch Out

Never compare floating-point numbers with ==. Use np.isclose(a, b) or np.allclose(a, b) with appropriate tolerances. This applies to loss values, metrics, convergence checks — everywhere.

Numerical Stability

Floating-point errors are usually tiny. The real danger is when algorithms amplify those tiny errors into catastrophic ones. That's what numerical instability means, and ML is riddled with opportunities for it.

Catastrophic cancellation happens when you subtract two nearly equal numbers. If a = 1.000000001 and b = 1.000000000, the true difference is 10⁻⁹, but in float32 both numbers round to the same value and the difference becomes exactly zero. You just lost all significant digits. This shows up whenever you compute things like variance using the formula E[X²] - E[X]² — if the variance is small relative to the mean, this formula self-destructs.

The single most important stability trick in all of ML is the log-sum-exp trick. You'll encounter it in softmax, in log-likelihood computations, in attention mechanisms, everywhere. The problem: you need to compute log(e^a₁ + e^a₂ + ... + e^aₙ). If any aᵢ is large (say 1000), e^1000 overflows to infinity. If all aᵢ are very negative (say -1000), every e^aᵢ underflows to zero, and you're taking log(0) = -infinity. The trick is simple: factor out the maximum value. log(Σ e^aᵢ) = max(a) + log(Σ e^(aᵢ - max(a))). Now the largest exponent is e⁰ = 1, so nothing overflows, and at least one term is non-zero, so the log is safe.

import numpy as np

def naive_logsumexp(x):
    """This WILL overflow or underflow."""
    return np.log(np.sum(np.exp(x)))

def stable_logsumexp(x):
    """The log-sum-exp trick — subtract the max first."""
    c = np.max(x)
    return c + np.log(np.sum(np.exp(x - c)))

# Overflow example
big_values = np.array([1000.0, 1001.0, 1002.0])
print(naive_logsumexp(big_values))   # inf (overflow!)
print(stable_logsumexp(big_values))  # 1002.4076... (correct)

# Underflow example
small_values = np.array([-1000.0, -1001.0, -1002.0])
print(naive_logsumexp(small_values)) # -inf (underflow!)
print(stable_logsumexp(small_values))# -999.5924... (correct)

# NumPy has this built in
print(np.logaddexp.reduce(big_values))  # same correct answer

This is exactly why we work with log-probabilities instead of raw probabilities throughout ML. A probability of 10⁻³⁰⁰ underflows to zero in any floating-point format. But its log, -690.7, is a perfectly ordinary number. Multiplying probabilities becomes adding log-probabilities. Everything stays in a numerically comfortable range.

The softmax function uses the same trick. Naively computing e^xᵢ / Σe^xⱼ overflows when any xᵢ is large. The fix: subtract max(x) from every element before exponentiating. The result is mathematically identical (the constant cancels in the ratio) but numerically safe.

import numpy as np

def naive_softmax(x):
    """Overflows for large inputs."""
    exp_x = np.exp(x)
    return exp_x / np.sum(exp_x)

def stable_softmax(x):
    """Subtract max for numerical stability."""
    shifted = x - np.max(x)
    exp_x = np.exp(shifted)
    return exp_x / np.sum(exp_x)

logits = np.array([1000.0, 1001.0, 1002.0])
print(naive_softmax(logits))   # [nan, nan, nan]  — useless
print(stable_softmax(logits))  # [0.0900, 0.2447, 0.6652] — correct

# The sigmoid has a similar issue
def stable_sigmoid(x):
    """Stable for both large positive and negative x."""
    pos_mask = x >= 0
    z = np.zeros_like(x)
    z[pos_mask] = 1.0 / (1.0 + np.exp(-x[pos_mask]))
    exp_x = np.exp(x[~pos_mask])
    z[~pos_mask] = exp_x / (1.0 + exp_x)
    return z
Key Insight

Every time you see np.exp() in your code, pause and ask: can the input be large? Can it be very negative? If yes, you probably need the log-sum-exp trick or the subtract-the-max pattern. This one habit will save you from 90% of numerical bugs.

Linear System Solvers

Here's a mistake nearly every new engineer makes: given the system Ax = b, they compute x = A⁻¹b by first inverting the matrix. Never do this. Computing a matrix inverse is roughly three times more expensive than solving the system directly, and it introduces significantly more numerical error. Use np.linalg.solve(A, b) instead — it uses LU decomposition internally, factoring A into a lower-triangular and upper-triangular matrix, then solving two easy triangular systems by back-substitution.

If your matrix A is symmetric and positive definite (common in ML — think covariance matrices, kernel matrices, the Gram matrix XᵀX), you can do even better. Cholesky decomposition factors A = LLᵀ, where L is lower-triangular. It's about twice as fast as general LU and numerically more stable. Use np.linalg.cholesky(A) followed by scipy.linalg.cho_solve().

But how do you know if your system is "hard" to solve? The condition number of a matrix, κ(A), tells you how much input errors get amplified in the solution. If κ(A) = 10ᵏ, you lose roughly k digits of precision. A condition number of 10¹⁶ means your float64 solution has essentially zero reliable digits. You can check with np.linalg.cond(A).

This connects directly to ridge regression. The normal equations give (XᵀX)β = Xᵀy. If features are correlated, XᵀX is ill-conditioned. Adding a regularization term λI gives (XᵀX + λI)β = Xᵀy. That λ on the diagonal improves the condition number dramatically. Regularization isn't just about overfitting — it's also about making the math actually solvable on a computer.

import numpy as np

# Never invert — solve instead
A = np.array([[4.0, 1.0], [1.0, 3.0]])
b = np.array([1.0, 2.0])

# Bad: x = A_inv @ b (slow, less accurate)
x_bad = np.linalg.inv(A) @ b

# Good: direct solve via LU decomposition
x_good = np.linalg.solve(A, b)

print(np.allclose(x_bad, x_good))     # True, but solve is faster and stabler
print(f"Condition number: {np.linalg.cond(A):.2f}")  # 2.62 — well-conditioned

# Cholesky for symmetric positive definite matrices
L = np.linalg.cholesky(A)             # A = L @ L.T
x_chol = np.linalg.solve(L.T, np.linalg.solve(L, b))
print(np.allclose(x_good, x_chol))    # True

# Ill-conditioned example — regularization helps
X = np.random.randn(100, 50)
XtX = X.T @ X
print(f"Unregularized cond: {np.linalg.cond(XtX):.0f}")
lam = 1.0
print(f"Regularized cond:   {np.linalg.cond(XtX + lam * np.eye(50)):.0f}")
Watch Out

If np.linalg.cond(A) returns a number larger than 10¹⁰, treat any solution with suspicion. Consider regularization, better feature scaling, or a different algorithmic approach entirely.

Iterative Methods

LU and Cholesky are direct solvers — they give you an exact answer (up to floating-point) in a fixed number of steps. But they scale as O(n³), and when n is a million (say, a large graph Laplacian or a kernel matrix for a big dataset), that's simply not feasible. You can't even store the full matrix, let alone factor it.

Iterative methods approach the solution gradually, giving you a better approximation each step. The workhorse for symmetric positive definite systems is the conjugate gradient method. It only needs to compute matrix-vector products Av, never the full matrix, so it works beautifully with sparse matrices. In exact arithmetic, it converges in at most n steps, but in practice a good approximation arrives much sooner — often in O(√κ) iterations, where κ is the condition number.

Conjugate gradient belongs to the family of Krylov subspace methods, which build solutions from the sequence {b, Ab, A²b, ...}. For non-symmetric systems, GMRES and BiCGSTAB are common alternatives. The convergence of all these methods depends heavily on the condition number, which is where preconditioning comes in: instead of solving Ax = b, you solve M⁻¹Ax = M⁻¹b, choosing M so that M⁻¹A is better conditioned. Finding a good preconditioner is often more art than science, but incomplete Cholesky or algebraic multigrid are reliable starting points. In ML, iterative solvers show up in Gaussian processes, graph-based semi-supervised learning, and large-scale kernel methods.

Numerical Integration

Sometimes you need to compute an integral that has no closed-form solution. In one or two dimensions, quadrature rules work well. The trapezoidal rule approximates the area under a curve using trapezoids — simple but only second-order accurate. Simpson's rule uses parabolic segments and is fourth-order. Gaussian quadrature chooses evaluation points and weights optimally to be exact for polynomials up to a certain degree, giving you remarkable accuracy with very few function evaluations.

But quadrature rules fall apart in high dimensions. A grid with 100 points per dimension becomes 100¹⁰ = 10²⁰ points in 10 dimensions. This is the curse of dimensionality. The solution is Monte Carlo integration: estimate ∫f(x)dx by averaging f over random samples. The error decreases as 1/√N regardless of dimension — no curse. The catch is that 1/√N convergence is slow; you need 100x more samples for one extra digit of accuracy.

This connects directly to Bayesian ML. Computing posterior expectations like E[f(θ)|data] requires integrating over the parameter space, which is almost never tractable analytically. Markov Chain Monte Carlo (MCMC) methods generate samples from the posterior distribution, and averaging f(θ) over those samples gives you Monte Carlo estimates of any expectation you need. The entire field of Bayesian deep learning rests on clever integration approximations.

Eigensolvers

Computing all eigenvalues of a large matrix is O(n³) and usually unnecessary. In ML, you typically need only the top few eigenvectors — the first k principal components, or the smallest eigenvectors of a graph Laplacian for spectral clustering.

Power iteration is the simplest eigensolver: start with a random vector, repeatedly multiply by A and normalize. This converges to the eigenvector with the largest eigenvalue. It's slow (convergence depends on the ratio of the top two eigenvalues), but it's dead simple and works on enormous matrices. Google's original PageRank algorithm is essentially power iteration on the web's link matrix.

For sparse matrices, the Lanczos algorithm is the standard tool. It's a Krylov method that builds a tridiagonal matrix whose eigenvalues approximate those of A, converging to extremal eigenvalues first. SciPy's scipy.sparse.linalg.eigsh() uses Lanczos under the hood. It needs only matrix-vector products, making it ideal for matrices too large to store densely.

For truly massive matrices (millions of rows), randomized SVD is the modern approach. The idea is counterintuitive: project the matrix into a much lower-dimensional random subspace, then compute an exact SVD of the small projection. The mathematical guarantee (due to random matrix theory) is that this approximation is surprisingly good. Scikit-learn's PCA uses randomized SVD by default when you set svd_solver='randomized'. It turns a problem that would take hours into one that takes seconds.

Sparse Computations

Most matrices in real-world ML are sparse — they're mostly zeros. A document-term matrix with 100,000 documents and 50,000 words has 5 billion entries, but any given document uses maybe 500 unique words. That's 99.999% zeros. Storing and computing with the full dense matrix is absurd.

Sparse matrices are stored using compact formats that only record non-zero entries. The two most common are CSR (Compressed Sparse Row) and CSC (Compressed Sparse Column). CSR stores three arrays: the non-zero values, their column indices, and pointers to where each row starts. This makes row slicing and matrix-vector products fast. CSC is the column-oriented analog, better for column slicing and for solvers that work column by column.

In Python, scipy.sparse is the standard library. You construct sparse matrices from coordinate lists (coo_matrix) then convert to CSR or CSC for computation. The rule of thumb: if your matrix is more than ~90% zeros, sparse formats save both memory and time. Below 90%, the indexing overhead makes dense operations faster.

Sparse structures are everywhere in ML. Bag-of-words features are sparse. One-hot encoded categorical variables are sparse. Graph adjacency matrices are sparse. Attention masks in transformers are sparse (or at least structured). Even neural network weights, after pruning, are sparse. Understanding sparse computation is not optional — it's a practical necessity for working with real data at real scale.

GPU Computing Fundamentals

A modern CPU has maybe 16 powerful cores, each capable of complex control flow and branch prediction. A modern GPU has thousands of simpler cores, each designed to do the same operation on different data simultaneously. This is SIMD parallelism — Single Instruction, Multiple Data — and it's exactly what matrix multiplication needs. Multiplying two 4096×4096 matrices requires about 137 billion multiply-add operations, all independent. A CPU grinds through them with a few cores; a GPU blasts through them with thousands.

The GPU memory hierarchy is critical to understand. Global memory (the GPU's main VRAM — 24GB on a 4090, 80GB on an A100) is large but slow. Shared memory is a small, fast scratchpad shared within a thread block. Registers are the fastest, private to each thread. The key insight is that memory bandwidth, not compute, is usually the bottleneck. Moving data from global memory to the cores takes far longer than the actual arithmetic. This is why memory-efficient algorithms (FlashAttention, kernel fusion) matter so much — they reduce data movement, not FLOPs.

Modern GPUs also include Tensor Cores, specialized hardware for small matrix multiplies (typically 4×4 or 8×8) in reduced precision. Tensor Cores can perform mixed-precision operations — multiplying in float16 while accumulating in float32 — at roughly 2-8x the throughput of standard CUDA cores. This hardware is specifically what makes mixed-precision training practical.

Mixed-Precision Arithmetic

Training in full float32 is safe but wasteful. The insight behind mixed-precision training is that most neural network operations don't need 32 bits of precision. Forward and backward passes run in half precision (float16 or bfloat16), which uses half the memory and runs faster on Tensor Cores. Only the master copy of weights and the gradient accumulation step use float32.

The main danger with float16 is its limited range: the smallest positive normal float16 is about 6×10⁻⁵. Gradient values smaller than this underflow to zero, and your model stops learning. Loss scaling solves this elegantly: before the backward pass, multiply the loss by a large constant (say 1024). This scales all gradients up into float16's representable range. After the backward pass, divide by the same constant before updating weights. PyTorch's GradScaler does this automatically, dynamically adjusting the scale factor.

The choice between bfloat16 and float16 is a tradeoff between range and precision. Float16 has 5 exponent bits and 10 mantissa bits — good precision but limited range. Bfloat16 has 8 exponent bits (same as float32!) but only 7 mantissa bits — less precision but the same range as float32. In practice, bfloat16 is more robust because it almost never overflows or underflows, making loss scaling unnecessary. This is why Google's TPUs and NVIDIA's A100+ GPUs all support bfloat16 natively.

The practical impact is substantial: mixed-precision training halves your memory usage (which means you can double your batch size or train a larger model), and runs 1.5-3x faster thanks to Tensor Core acceleration. In 2024, there is essentially no reason to train in full float32 unless you're debugging a numerical issue. Mixed precision isn't an optimization — it's the default.

Danger

Never reduce precision for the weight update step itself. Accumulating small gradient updates into float16 weights will destroy your training through rounding errors. The pattern is: forward in fp16, backward in fp16, accumulate and update in fp32. Always keep a float32 master copy of your weights.
