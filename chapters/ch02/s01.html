Mathematical Notation

Chapter 2: Mathematical Foundations 4 subtopics
TL;DR

Math notation is just shorthand — every symbol maps to something you already understand in code. Sets are collections (like your dataset), functions are mappings (like your model), sigma/pi are loops (like sum() and prod()), and logarithms are the trick that makes multiplying tiny probabilities tractable. Master these four building blocks and ML papers stop looking like ancient scrolls.

ML papers look like alien hieroglyphics — let's decode them. The truth is, most of the math you'll encounter in machine learning boils down to a surprisingly small toolkit. You don't need a PhD in pure mathematics. You need fluency in about a dozen notational conventions that researchers use as shorthand. Think of this section as your Rosetta Stone: we'll map every Greek letter and funny symbol to something concrete you can touch in Python. By the end, you'll read a loss function formula the way you read a for-loop — mechanically, without fear.

Sets and Set Operations

Set Definition and Membership

A set is an unordered collection of distinct elements. That's it. No duplicates, no ordering. When a paper writes x ∈ S, it means "x is a member of set S." You already use sets constantly: your training dataset is literally a set of examples. When someone says "sample a mini-batch from the training set," they're performing a set operation — picking a subset.

Sets can be defined by listing elements, like S = {1, 2, 3}, or by a rule, like S = {x | x > 0} which reads "the set of all x such that x is greater than zero." In ML, your feature space is typically ℝⁿ — the set of all n-dimensional real-valued vectors. Your label space might be a finite set like {cat, dog, bird} for classification.

Union, Intersection, Complement

The union of two sets, written A ∪ B, gives you everything in either set. Combine your training set and validation set? That's a union. The intersection, A ∩ B, gives you only elements in both. Want to find which features appear in both dataset A and dataset B? Intersection. The complement of A, written Aᶜ, is everything NOT in A — like the set of all samples you didn't select for your test split.

These operations show up in data preprocessing all the time. Finding overlapping users between two datasets, removing banned tokens from a vocabulary, computing the difference between predicted labels and ground truth labels — all set operations under the hood.

Subsets and Supersets

If every element of A is also in B, then A is a subset of B, written A ⊆ B. Every mini-batch is a subset of your full dataset. Your validation set is a subset of your total collected data. The full dataset is the superset. This notation shows up when papers define constraints — "for all x in some subset of the input space" is a common pattern in optimization proofs.

Cartesian Product

The Cartesian product A × B is the set of all ordered pairs (a, b) where a ∈ A and b ∈ B. If A has 3 elements and B has 4, A × B has 12 pairs. Recognize this? It's exactly what happens in grid search. If your learning rates are {0.001, 0.01, 0.1} and your batch sizes are {16, 32, 64, 128}, grid search evaluates every element of their Cartesian product — all 12 combinations. This is also why the curse of dimensionality is brutal: the Cartesian product of many sets grows exponentially.

Why This Matters in Practice

When a paper says a model maps from X × Y → Z, it means the model takes a pair of inputs — one from space X, one from space Y — and produces an output in space Z. Multi-modal models (text + image → caption) are literally functions on Cartesian products.

Functions and Mappings

Domain, Range, Codomain

A function f: A → B maps every element of set A to exactly one element of set B. The set A is the domain, B is the codomain, and the actual outputs that f produces form the range (a subset of B). Here's the punchline: every ML model is a function. A classifier f: ℝⁿ → {0, 1, ..., k} maps feature vectors to class labels. A regression model f: ℝⁿ → ℝ maps features to a continuous value. Once you see models as functions, everything clicks — training is searching for the best function, and the hypothesis space is the set of all functions your architecture can represent.

Injective, Surjective, Bijective

An injective (one-to-one) function never maps two different inputs to the same output — information is preserved. A surjective (onto) function hits every element in the codomain — nothing in the output space is unreachable. A function that is both is bijective — a perfect, reversible pairing.

Why care? Autoencoders compress data from a high-dimensional space to a low-dimensional latent space. That encoder is NOT injective — multiple inputs map to the same code, so you lose information. That's the whole point of compression. But normalizing flows, a type of generative model, use bijective transformations specifically because bijectivity lets you compute exact likelihoods by applying the change-of-variables formula. The math only works when the mapping is invertible.

Composition of Functions

Composition means chaining functions: (g ∘ f)(x) = g(f(x)). Apply f first, then feed the result into g. A neural network is literally a composition of functions. Each layer is a function: linear transform, then activation. Stack three layers and you get f₃(f₂(f₁(x))). The "deep" in deep learning just means "lots of compositions." Backpropagation uses the chain rule to differentiate through these compositions — which is why understanding composition matters before you touch calculus.

Inverse Functions

If f: A → B is bijective, its inverse f⁻¹: B → A undoes what f did: f⁻¹(f(x)) = x. In an autoencoder, the decoder is attempting to approximate the inverse of the encoder. In generative models like normalizing flows, the inverse is exact and used to generate new samples by mapping from a simple distribution (like a Gaussian) back to the data space. Invertible neural networks are a hot research area precisely because having an exact inverse unlocks powerful capabilities.

Summation and Product Notation

Sigma Notation

The sigma notation Σ is just a for-loop that adds things up. When you see Σᵢ₌₁ⁿ xᵢ, read it as: "start with i=1, go up to i=n, and add up all the xᵢ values." That's literally sum(x[1:n+1]) in Python. Loss functions are almost always sums. Mean Squared Error? It's (1/n) Σᵢ₌₁ⁿ (yᵢ - ŷᵢ)². Let's decode that piece by piece: for each of your n samples, compute the difference between the true value yᵢ and the predicted value ŷᵢ, square it, add them all up, and divide by n. Nothing magical.

# MSE decoded: sigma notation → NumPy
import numpy as np

y_true = np.array([3.0, 5.0, 2.5, 7.0])
y_pred = np.array([2.8, 5.2, 2.3, 6.8])

# The "Σ (yᵢ - ŷᵢ)²" part, done element-wise
squared_errors = (y_true - y_pred) ** 2   # [0.04, 0.04, 0.04, 0.04]

# The "1/n Σ" part
mse = np.mean(squared_errors)  # 0.04
print(f"MSE: {mse:.4f}")

Product Notation Pi

The product notation Π is the same idea but with multiplication instead of addition. Πᵢ₌₁ⁿ xᵢ means multiply all the xᵢ together. Where does this appear? Likelihood functions. If you assume your data points are independent, the joint probability is the product of individual probabilities: P(data) = Πᵢ₌₁ⁿ P(xᵢ). Naive Bayes is the classic example — the "naive" part is the assumption that features are independent, which lets you write the class-conditional probability as a product.

The problem with products of probabilities? They get tiny fast. Multiply 1000 numbers that are each less than 1 and you get something so close to zero that floating-point arithmetic chokes. This is exactly why we use logarithms — but we'll get there in a moment.

Common Summation Identities

A few identities show up repeatedly in derivations. The sum of a constant: Σᵢ₌₁ⁿ c = nc. Linearity: Σᵢ (axᵢ + b) = a·Σᵢ xᵢ + nb. You can pull constants out of sums, split sums apart, and change the order of double sums. These aren't deep truths — they're bookkeeping rules that let you simplify ugly expressions. When you see a derivation rearranging a loss function, they're almost always using one of these identities. Recognizing them saves you from thinking each step is a clever trick when it's actually routine algebra.

Logarithms and Exponentials

Exponential Functions

The exponential function eˣ (where e ≈ 2.718) grows faster than any polynomial. It shows up everywhere in ML. The softmax function uses it to turn a vector of raw scores (logits) into probabilities: softmax(zᵢ) = eᶻⁱ / Σⱼ eᶻʲ. Why exponentiate? Because eˣ is always positive (probabilities must be positive) and it amplifies differences (the largest logit dominates). Exponential decay appears in learning rate schedules — shrink the step size by a factor each epoch so you take big steps early and fine-tune later.

Natural Logarithm ln

The natural logarithm ln(x) is the inverse of eˣ: if eʸ = x, then ln(x) = y. The killer feature of logarithms for ML is this: ln(a · b) = ln(a) + ln(b). Products become sums. Remember that likelihood function that was a product of 1000 tiny probabilities? Take the log and it becomes a sum of 1000 manageable negative numbers. This is the log-likelihood, and maximizing it is equivalent to maximizing the likelihood (because log is monotonically increasing). We literally do this to avoid numerical underflow.

Log Rules: Product, Quotient, Power

Three rules handle almost everything. Product rule: ln(ab) = ln(a) + ln(b) — turns products into sums. Quotient rule: ln(a/b) = ln(a) - ln(b) — turns division into subtraction. Power rule: ln(aⁿ) = n·ln(a) — pulls exponents down as multipliers. These three rules are why log-likelihood derivations simplify so beautifully. When you see a paper go from a messy product of exponentials to a clean sum of linear terms, they used these rules. Every single time.

Change of Base Formula

You can convert between logarithm bases: log_b(x) = ln(x) / ln(b). In ML, you'll see ln (base e, natural log), log₂ (base 2, common in information theory), and sometimes log₁₀. Information-theoretic quantities like entropy are traditionally defined with log₂ (measured in bits), but implementations often use ln (measured in nats). The choice of base is just a scaling constant — it doesn't change which distribution has higher entropy. Most ML frameworks use natural log internally.

Logarithms in ML: Log-Likelihood and Cross-Entropy

This is arguably the single most important formula in applied ML. Cross-entropy loss for classification is: L = -Σᵢ₌₁ⁿ Σⱼ₌₁ᶜ yᵢⱼ · ln(pᵢⱼ). Here, n is the number of samples, c is the number of classes, yᵢⱼ is 1 if sample i belongs to class j (and 0 otherwise), and pᵢⱼ is the model's predicted probability that sample i belongs to class j. For binary classification, this simplifies to: L = -(1/n) Σᵢ [yᵢ ln(pᵢ) + (1-yᵢ) ln(1-pᵢ)].

Where does this come from? It's the negative log-likelihood under a categorical distribution. Maximizing likelihood equals minimizing negative log-likelihood equals minimizing cross-entropy. Three different names for the same optimization. Let's see it in code:

# Cross-entropy loss from scratch
import numpy as np

# Binary classification: 4 samples
y_true = np.array([1, 0, 1, 1])          # actual labels
y_pred = np.array([0.9, 0.1, 0.8, 0.7])  # predicted probabilities

# Binary cross-entropy: -[y·ln(p) + (1-y)·ln(1-p)]
epsilon = 1e-15  # avoid ln(0) which is -infinity
y_pred = np.clip(y_pred, epsilon, 1 - epsilon)

bce = -(y_true * np.log(y_pred) + (1 - y_true) * np.log(1 - y_pred))
loss = np.mean(bce)
print(f"BCE Loss: {loss:.4f}")  # ~0.2014
The ln(0) Trap

Never compute ln(0) — it's negative infinity and will destroy your gradients. Every practical implementation clips predicted probabilities away from exactly 0 and 1. That tiny epsilon = 1e-15 isn't a hack — it's required numerical hygiene. If your loss suddenly shows NaN or inf, check for zeros being passed to a log first. It's the #1 cause of numerical instability in classification training loops.

Let's tie it all together with one more example. Naive Bayes computes the log-probability of a class given features by using every rule we covered — products become sums via log, and each term is a simple log-probability lookup:

# Naive Bayes: using log to avoid underflow
import numpy as np

# Suppose 3 binary features, 2 classes
# P(feature_j=1 | class_k) — learned from training data
log_probs = {
    "spam":    np.log([0.8, 0.1, 0.9]),  # P(feat|spam)
    "not_spam": np.log([0.2, 0.7, 0.3])  # P(feat|not_spam)
}
log_prior = {"spam": np.log(0.4), "not_spam": np.log(0.6)}

# New email with features [1, 0, 1]
features = np.array([1, 0, 1])

for cls in ["spam", "not_spam"]:
    # Π P(xⱼ|class) becomes Σ ln P(xⱼ|class) via log
    log_likelihood = np.sum(
        features * log_probs[cls] + (1 - features) * np.log(1 - np.exp(log_probs[cls]))
    )
    score = log_prior[cls] + log_likelihood
    print(f"Log P({cls} | features) ∝ {score:.3f}")
The Pattern You'll See Everywhere

Almost every ML loss function follows the same template: take a probabilistic model, write its likelihood (a product), take the log (now a sum), negate it (because optimizers minimize), and average over your dataset. Cross-entropy, KL divergence, variational lower bounds — they all emerge from this one recipe. Once you see the pattern, new loss functions stop being scary.

That's your mathematical phrasebook. Sets give you the vocabulary to talk about data collections and spaces. Functions give you the framework to describe what models actually do. Summation and product notation are the shorthand for the loops that compute losses and likelihoods. And logarithms are the numerical trick that makes it all computationally feasible. You don't need to memorize theorems — you need to read notation fluently, the way a musician reads sheet music. And like music, the only way to get fluent is to practice: every time you see a formula, pause and translate it to code in your head. It'll click faster than you think.
