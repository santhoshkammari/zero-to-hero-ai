Optimization

Chapter 2: Mathematical Foundations
TL;DR

Optimization is the engine under every ML algorithm. Convex problems are the easy ones — one valley, one answer. Add constraints and you get Lagrangians and KKT conditions. Neural networks live in non-convex landscapes full of saddle points, yet SGD navigates them surprisingly well. Some problems demand discrete answers, pushing us into combinatorial optimization where even finding the optimum is fundamentally hard. This section gives you the theory and the code to work with all four flavors.

Convex Optimization

Before we talk about optimization being hard, let's talk about when it's easy. Convex optimization is the happy path — the case where the math actually guarantees you'll find the best answer. Understanding it well helps you recognize when a problem is tractable and when you're wandering into rougher terrain.

Convex Sets and Convex Functions

Start with a simple picture. A convex set is any region where, if you pick two points inside it, every point on the straight line between them is also inside. A filled circle? Convex. A crescent moon? Not convex — draw a line between the two tips and part of it leaves the shape. This sounds abstract, but it matters: the feasible region of many optimization problems needs to be convex for the nice guarantees to kick in.

Now layer on functions. A convex function is one that curves upward — or more precisely, the line segment between any two points on its graph never dips below the graph itself. Think of a bowl sitting on a table. No matter where you place a marble, it rolls to the same bottom. Mathematically, a function f is convex if for any two points x and y and any t between 0 and 1:

f(t·x + (1-t)·y) <= t·f(x) + (1-t)·f(y)

That inequality says the function evaluated at a weighted average of inputs is at most the weighted average of the function values. If you can verify this, you know your function has a single valley.

Why Convexity Matters

Here's the punchline: for convex functions, every local minimum is a global minimum. There are no deceptive valleys. If gradient descent finds a flat spot, that flat spot is the answer. This is a massive deal because it means simple iterative algorithms are guaranteed to converge to the best solution, not just a good-enough one.

A related result is Jensen's inequality, which states that for a convex function f and a random variable X: f(E[X]) ≤ E[f(X)]. The function of the average is at most the average of the function. This shows up all over ML. It's the backbone of the Evidence Lower Bound (ELBO) in variational inference — when you can't compute an expectation directly, Jensen's gives you a tractable bound to optimize instead. It also explains why the log-likelihood in expectation-maximization algorithms has a nice lower bound you can iteratively tighten.

Convex Problems in ML

Many classical ML models produce convex optimization problems. Linear regression with squared error is convex — the loss surface is a paraboloid with exactly one bottom. Logistic regression with the log-loss is also convex, which is why it converges reliably even on large datasets. Support Vector Machines minimize a convex quadratic objective subject to linear constraints. Ridge regression adds an L2 penalty that preserves convexity. Lasso adds an L1 penalty that's convex but not differentiable everywhere — still convex, just trickier to optimize.

The moment you move to neural networks, convexity vanishes. But knowing which problems are convex helps you choose the right tool. If your problem is convex, don't throw a deep net at it — use a convex solver and get the exact answer.

import numpy as np
from scipy.optimize import minimize

# Convex problem: minimize f(x) = (x[0] - 3)^2 + (x[1] + 1)^2
# This is a simple paraboloid — one global minimum at (3, -1)

def objective(x):
    return (x[0] - 3)**2 + (x[1] + 1)**2

def gradient(x):
    return np.array([2*(x[0] - 3), 2*(x[1] + 1)])

# Start from a random point — convexity guarantees we find the global min
x0 = np.array([10.0, 10.0])
result = minimize(objective, x0, jac=gradient, method='BFGS')

print(f"Minimum at: {result.x}")       # [3.0, -1.0]
print(f"Min value:  {result.fun}")      # 0.0
print(f"Converged:  {result.success}")  # True
Key Insight

If you can prove your loss function is convex, you've essentially solved the hard part. The optimizer becomes a detail — almost any reasonable algorithm will find the answer. The real difficulty in ML is that most interesting models produce non-convex landscapes.

Constrained Optimization

Real problems rarely say "minimize this function, no strings attached." There are budgets to respect, probabilities that must sum to one, physical quantities that can't go negative, and resource limits that can't be exceeded. Constrained optimization is where you minimize (or maximize) an objective function while satisfying a set of constraints.

The Lagrangian Method

The core trick is elegant: turn a constrained problem into an unconstrained one. Suppose you want to minimize f(x) subject to the equality constraint g(x) = 0. The Lagrangian combines them into a single function:

L(x, λ) = f(x) + λ · g(x)

Here λ is a Lagrange multiplier — a new variable the optimizer must find alongside x. At the optimum, the gradient of f is proportional to the gradient of g, which means you can't improve the objective without violating the constraint. That proportionality constant is exactly λ, and it tells you the sensitivity of your optimal value to the constraint: how much would your best objective improve if you relaxed the constraint by a tiny amount?

KKT Conditions

Equality constraints are tidy, but inequality constraints are messier. You might require g(x) ≤ 0. The Karush-Kuhn-Tucker (KKT) conditions generalize Lagrange multipliers to handle both types. The conditions are:

1. Stationarity: the gradient of the Lagrangian is zero. 2. Primal feasibility: all constraints are satisfied. 3. Dual feasibility: multipliers for inequality constraints are non-negative. 4. Complementary slackness: for each inequality constraint, either the constraint is tight (active) or its multiplier is zero — never both loose and penalized.

Complementary slackness is the most subtle. It says the optimizer only "pays attention" to constraints that are actually binding. If a constraint has slack (isn't tight), its multiplier is zero and it's irrelevant. This is deeply connected to how SVMs work — only the support vectors (data points where the margin constraint is tight) actually influence the decision boundary.

Duality

Every constrained optimization problem has a dual problem. The original is called the primal. Instead of minimizing over x with constraints, the dual maximizes over the Lagrange multipliers. Weak duality always holds: the dual optimum is a lower bound on the primal optimum. Strong duality says the two are equal — and it holds whenever the primal problem is convex and satisfies mild regularity conditions (Slater's condition). Strong duality is why SVMs can be solved efficiently in the dual form, and it's the theoretical foundation for much of convex programming.

LP, QP, and SDP

Constrained optimization comes in standard forms, each with specialized solvers. Linear Programming (LP) has a linear objective and linear constraints — think supply chain routing or diet problems. Quadratic Programming (QP) adds a quadratic objective but keeps linear constraints — this is exactly the SVM formulation. Semidefinite Programming (SDP) generalizes further by constraining a matrix to be positive semidefinite, which appears in kernel learning and relaxations of combinatorial problems. Each class has solvers tuned to exploit its specific structure.

Interior point methods are the modern workhorse for all three. Unlike the simplex method (which walks along edges of the feasible polytope), interior point methods walk through the interior, using barrier functions to stay away from constraint boundaries. They scale better for large problems and are what scipy uses under the hood for many constrained problems.

SVM as Constrained Optimization

The SVM is a textbook example. You want the widest possible margin between classes, subject to every training point being on the correct side. The primal is a QP. The dual reveals that the solution depends only on the support vectors — the few points closest to the boundary. This is not just theoretically clean; it's computationally practical. And the kernel trick works naturally in the dual, letting you operate in high-dimensional feature spaces without ever computing coordinates there.

import numpy as np
from scipy.optimize import linprog

# Linear Programming: minimize cost of a diet
# Minimize: 2*bread + 3.5*milk + 8*cheese  (cost per unit)
# Subject to:
#   6*bread + 2*milk + 1*cheese >= 10  (min protein)
#   1*bread + 8*milk + 7*cheese >= 15  (min calcium)
#   bread, milk, cheese >= 0

# linprog does minimization with <= constraints, so flip signs
c = [2, 3.5, 8]                           # cost coefficients
A_ub = [[-6, -2, -1], [-1, -8, -7]]       # negated for >= to <=
b_ub = [-10, -15]                          # negated RHS
bounds = [(0, None), (0, None), (0, None)] # non-negativity

result = linprog(c, A_ub=A_ub, b_ub=b_ub, bounds=bounds, method='highs')
print(f"Optimal amounts: bread={result.x[0]:.2f}, milk={result.x[1]:.2f}, "
      f"cheese={result.x[2]:.2f}")
print(f"Minimum cost: ${result.fun:.2f}")
Watch Out

When converting "greater than or equal to" constraints for solvers that expect "less than or equal to," you negate both sides. It's the most common source of sign errors in constrained optimization. Always double-check by plugging your solution back into the original constraints.

Non-Convex Optimization

Now for the uncomfortable truth: almost every interesting deep learning problem is non-convex. The loss surface of a neural network has an astronomically large number of parameters, and the interactions between layers create a landscape riddled with hills, valleys, and — more importantly — saddle points. Yet somehow, training works. Understanding why is one of the most active areas in ML theory.

Local Minima vs. Saddle Points

For years, people worried about local minima — valleys that aren't the deepest but trap gradient descent. It turns out the bigger problem in high dimensions is saddle points: places where the gradient is zero but the surface curves up in some directions and down in others. In a function with millions of parameters, for a critical point to be a true local minimum, the curvature must be positive in every single direction. That's extremely unlikely by random chance. Most critical points in high-dimensional landscapes are saddle points, and gradient descent can escape them — it just needs a nudge in the right direction.

Research by Dauphin et al. showed that the loss values at different local minima in large networks tend to be remarkably similar. So even if you get stuck in a local minimum, it's probably almost as good as the global one. The loss landscape of overparameterized networks is surprisingly benign — not because it's convex, but because the bad local minima seem to be rare.

Heuristics That Work

When you can't guarantee finding the global optimum, you lean on heuristics. Random restarts run the optimizer multiple times from different starting points and keep the best result — simple but effective. Simulated annealing borrows from metallurgy: accept worse solutions with some probability early on (high "temperature"), then gradually become pickier. This lets you escape shallow local minima. Genetic algorithms maintain a population of candidate solutions, combining and mutating them over generations. These are slow but flexible, especially when gradients aren't available.

But for neural networks, the star is SGD with momentum. The stochasticity in mini-batch gradient descent acts as natural noise that helps escape saddle points. Momentum accumulates velocity, allowing the optimizer to barrel through small bumps. Together, they navigate non-convex landscapes surprisingly well — not with any theoretical guarantee, but with overwhelming empirical evidence across decades of practice.

Loss Landscape Intuition

Visualizing loss landscapes helps build intuition. Imagine a vast mountainous terrain, but in millions of dimensions. Research by Li et al. introduced "filter-normalized" visualizations that project this high-dimensional surface onto 2D. What they found is striking: well-designed architectures (like ResNets with skip connections) produce smoother, more navigable landscapes than architectures without them. The architecture doesn't just determine what the network can represent — it shapes the optimization terrain itself.

import numpy as np
from scipy.optimize import minimize

# Non-convex function: Rastrigin — many local minima
# Global minimum at origin, but the landscape is full of traps
def rastrigin(x):
    A = 10
    n = len(x)
    return A * n + np.sum(x**2 - A * np.cos(2 * np.pi * x))

# Try multiple random restarts to escape local minima
best_result = None
np.random.seed(42)

for _ in range(20):
    x0 = np.random.uniform(-5.12, 5.12, size=5)
    result = minimize(rastrigin, x0, method='L-BFGS-B',
                      bounds=[(-5.12, 5.12)] * 5)
    if best_result is None or result.fun < best_result.fun:
        best_result = result

print(f"Best minimum found: {best_result.fun:.4f}")
print(f"At position: {np.round(best_result.x, 3)}")
# Global min is 0.0 at the origin — how close did we get?
Key Insight

In high-dimensional non-convex optimization, saddle points are far more common than local minima. SGD's noise is a feature, not a bug — it provides the perturbation needed to escape saddle points. This is one reason SGD often outperforms full-batch gradient descent despite being "noisier."

Combinatorial Optimization

Everything so far assumed continuous variables — you can nudge x by a tiny amount and see a tiny change in f(x). But many real problems demand discrete answers. Which features should you select? Which nodes should you include in a network? How should you route deliveries? Welcome to combinatorial optimization, where the search space is finite but astronomically large.

Integer Programming and Relaxation

Integer programming (IP) is linear programming where some or all variables must be integers. This tiny change — from continuous to integer — makes the problem fundamentally harder. A linear program with a million variables can be solved in seconds. Make those variables integers and you might wait until the heat death of the universe.

The practical workaround is LP relaxation: drop the integer requirement, solve the continuous version (which is easy), then round the solution to integers. This doesn't always give the optimal integer solution, but it provides a bound — you know the true optimum can't be better than the relaxed one. More sophisticated approaches like branch and bound systematically explore the integer solution space, using LP relaxations at each node to prune branches that can't possibly beat the best solution found so far. It's a tree search, but with math doing the pruning.

NP-Hardness

Some combinatorial problems are NP-hard — a complexity class meaning that no known algorithm solves them in polynomial time, and most researchers believe no such algorithm exists. The traveling salesman problem, graph coloring, and many scheduling problems fall here. NP-hardness doesn't mean "impossible." It means "don't expect an exact solution for large instances." Instead, you use approximation algorithms with provable guarantees ("this solution is at most 1.5 times the optimum") or heuristics that work well in practice without formal guarantees.

ML Connections

Feature selection is inherently combinatorial — given d features, which subset of k features gives the best model? There are "d choose k" subsets, and that grows explosively. Greedy methods (forward selection, backward elimination) are the standard heuristics. Neural architecture search (NAS) faces the same combinatorial explosion: which layers, which connections, which hyperparameters? Early NAS methods used reinforcement learning or evolutionary algorithms; modern approaches use weight-sharing and differentiable relaxations to make the discrete search tractable.

Discrete latent variables in generative models (like the categorical choices in a mixture model) create combinatorial structure. The Gumbel-Softmax trick provides a differentiable approximation to discrete sampling, letting you backpropagate through what would otherwise be a non-differentiable, combinatorial operation. It's LP relaxation's spiritual cousin — replace the hard discrete choice with a soft continuous approximation, optimize, then sharpen.

import numpy as np
from scipy.optimize import linprog, milp, Bounds, LinearConstraint

# Feature selection as integer programming (simplified)
# Select exactly 3 out of 6 features to maximize total "importance"
# (In practice, importance scores come from a model)

importance = np.array([0.9, 0.3, 0.8, 0.2, 0.7, 0.5])

# MILP: maximize importance = minimize negative importance
c = -importance  # negate because milp minimizes

# Constraint: select exactly 3 features (sum of binary vars = 3)
A_eq = np.ones((1, 6))
b_eq = np.array([3.0])
constraints = LinearConstraint(A_eq, b_eq, b_eq)

# Each variable is binary: 0 or 1
integrality = np.ones(6)  # 1 = integer variable
bounds = Bounds(lb=0, ub=1)

result = milp(c, constraints=constraints, integrality=integrality,
              bounds=bounds)
selected = np.where(result.x > 0.5)[0]
print(f"Selected features: {selected}")          # [0, 2, 4]
print(f"Total importance:  {-result.fun:.1f}")    # 2.4
Watch Out

Combinatorial optimization problems can look deceptively similar to their continuous counterparts, but adding integrality constraints can change a problem from "solvable in milliseconds" to "intractable." Always check whether your problem truly requires discrete solutions. Sometimes a continuous relaxation followed by rounding is good enough — and orders of magnitude faster.

Optimization theory gives you a map of the terrain. Convex problems are the well-paved roads. Constrained problems add guardrails. Non-convex landscapes are the wilderness that deep learning has learned to navigate by feel. And combinatorial problems are the cliff faces where you sometimes need to accept "good enough." Knowing which terrain you're on is half the battle — it tells you which tools to reach for and what guarantees you can expect.
