TL;DR

Computers lie about numbers. Store approximations, not exact values. Naive math on approximations makes silent garbage — NaN losses, zero gradients, training runs drifting into nonsense. This section builds instinct for traps: IEEE 754 floating point, catastrophic cancellation, log-sum-exp trick, why you never invert matrix, how GPUs move data, mixed-precision regime now default for serious training. Not academic footnotes. Difference between model that converges and one that silently fails at 3 AM.

Kept getting NaN in training runs. I kept lowering learning rate until two-day debug found unprotected log(0). Enough to make numbers deserve real look.

Numerical computing studies math on machines that cannot do math perfectly. Field goes back to 1950s, when scientists saw computers add error every time they store number. In ML, errors compound across millions of operations per training step. Result: subtle accuracy drift to spectacular NaN explosions.

Computers Lie About Numbers

Start with most famous lie in computing.

a = 0.1 + 0.2
print(a == 0.3)        # False
print(f"{a:.20f}")     # 0.30000000000000004441

First time I saw this, thought Python was broken. Not broken. Problem goes deeper than any programming language. Computers use binary — base 2 — and in base 2, 0.1 has no exact representation. Same reason 1/3 has no exact decimal representation: 0.33333... goes on forever. In binary, 0.1 also goes forever. Computer stores closest binary fraction it can fit. Tiny error appears. Not bug. Law of universe, like friction.

IEEE 754 standard, set in 1985 and used by almost every computer, defines these approximations. Every floating-point number stores three pieces: sign bit (positive or negative), exponent (sets magnitude), and mantissa (also called significand, carries precision digits). Think scientific notation. Number -2.998 × 10⁸ has sign (negative), significand (2.998), and exponent (8). IEEE 754 does same thing in base 2: value is (-1)^sign × 1.mantissa × 2^(exponent - bias).

Good mental model: ruler with uneven tick marks. Near zero, ticks pack tight — very small numbers stay distinguishable. At larger magnitudes, ticks spread apart. Between 1.0 and 2.0, float32 has millions of representable values. Between 10,000,000 and 10,000,001, maybe none. Ruler covers huge range, but pays by getting coarser as numbers grow.

Formats you'll see in practice: float64 (double precision: 1 sign + 11 exponent + 52 mantissa bits, about 15-16 decimal digits of precision), float32 (single precision: 1 sign + 8 exponent + 23 mantissa bits, about 7 decimal digits), float16 (half precision: 1 sign + 5 exponent + 10 mantissa bits, about 3.3 decimal digits), and bfloat16 (brain float: 1 sign + 8 exponent + 7 mantissa bits, about 2.4 decimal digits but same range as float32). NumPy defaults to float64. PyTorch defaults to float32. Gap matters a lot. We come back to it.

The Number Line Has Holes

Number called machine epsilon, often written ε, tells you fundamental resolution of your floating-point format. Smallest number you can add to 1.0 and get something different from 1.0. For float32, machine epsilon is about 1.19 × 10⁻⁷. For float64, about 2.22 × 10⁻¹⁶. Anything smaller than that, relative to number you add it to, gets swallowed. Vanishes. Like it never existed.

import numpy as np

print(np.finfo(np.float32).eps)   # 1.1920929e-07
print(np.finfo(np.float64).eps)   # 2.220446049250313e-16

# When adding small to large, the small disappears
big = np.float32(1e7)
small = np.float32(1.0)
print(big + small == big)          # True — the 1.0 is gone entirely

Feel weight of last line. Add 1.0 to ten million, get ten million. 1.0 got absorbed. In float32, ten million and ten million plus one are same number. Tick marks on ruler sit too far apart to tell them apart.

Now think about gradient accumulation. It adds many small gradient updates to large running sum, thousands of times. If accumulated loss is in thousands and gradient update is 0.00001, update can be smaller than machine epsilon relative to sum. It disappears. Model looks like it trains, loss prints every epoch, weights have not budged. Seen this in production. Quietest bug you will meet.

There are special values too. When exponent bits are all ones and mantissa is zero, you get infinity — overflow result, like computing e^1000. When exponent bits are all ones but mantissa is non-zero, you get NaN (Not a Number), from 0/0 or infinity minus infinity. NaN is contagious: any operation involving NaN produces NaN. Strangest part — NaN is only floating-point value not equal to itself. That is how you detect it: x != x is true only when x is NaN.

At other extreme, when all exponent bits are zero, you get denormalized numbers (also called subnormals). These are tiny numbers between zero and smallest normal float, filling what would otherwise be abrupt gap. They matter because gradients near zero sometimes live in this range. Catch: some hardware processes denormals 10-100x slower than normal floats, which can cause mysterious training slowdowns.

Watch Out

Never compare floating-point numbers with ==. Use np.isclose(a, b) or np.allclose(a, b) with appropriate tolerances. Applies to loss values, metrics, convergence checks — everywhere. Also: np.isnan(x) is friend. Sprinkle it when debugging training.

When Small Errors Become Catastrophes

Floating-point errors are usually tiny — 10⁻⁷ relative error in float32, 10⁻¹⁶ in float64. Real danger comes when algorithm amplifies tiny errors into disasters. That is numerical instability. ML is full of chances for it.

Most common villain is catastrophic cancellation. It happens when you subtract two numbers that are nearly equal. Suppose a = 1.00000005 and b = 1.00000004 in float32. True difference is 10⁻⁸, but float32 can represent only about 7 decimal digits. Both numbers store same way. Difference becomes exactly zero. All significant digits, gone.

Where does this ambush us in practice? Variance. Textbook one-pass formula says Var(X) = E[X²] - E[X]². Both terms can be huge when mean is large, but their difference — actual variance — might be tiny. In float32, this formula can return zero for data with small spread around large mean. Worse, it can return negative number, which makes no sense for variance. Batch normalization computes variance of activations on every forward pass. If activations have large mean and small variance, textbook formula silently poisons normalization layer.

Fix is Welford's algorithm, online method that accumulates variance without subtracting two large numbers. It keeps running mean and running sum of squared deviations from that mean, updating both as each data point arrives. Single pass through data. Numerically stable. Every serious library uses this under hood.

import numpy as np

# The textbook formula self-destructs for large means with small variance
data = np.float32([10000000.0, 10000001.0, 10000002.0])
naive_var = np.mean(data**2) - np.mean(data)**2
print(f"Naive variance:   {naive_var}")    # 0.0 — completely wrong
print(f"Correct variance: {np.var(data)}") # 0.6667 — np.var uses stable algorithm

# Welford's algorithm — stable single-pass variance
def welford_var(data):
    n = 0
    mean = 0.0
    M2 = 0.0
    for x in data:
        n += 1
        delta = x - mean
        mean += delta / n
        delta2 = x - mean
        M2 += delta * delta2
    return M2 / n

print(f"Welford variance: {welford_var(data.astype(np.float64)):.4f}")  # 0.6667

Catastrophic cancellation also lurks in log(1 + x) when x is tiny. If x = 10⁻¹⁵, then 1 + x rounds to 1.0 in float64, and log(1.0) is exactly 0. True answer, about 10⁻¹⁵, is gone. Python and NumPy provide np.log1p(x), which computes log(1 + x) directly without adding 1 first, keeping full precision. Similarly, np.expm1(x) computes e^x - 1 for small x without cancellation. These show up constantly in log-likelihood computations and loss functions.

Another subtle trap: Kahan summation. When you add millions of small numbers (say, per-sample losses across dataset), running sum grows large while each new term stays small. Small terms start getting swallowed by machine epsilon. Kahan's algorithm keeps separate "compensation" variable that tracks accumulated rounding error and feeds it back into next addition. It turns O(n·ε) drift into O(ε) drift. Some frameworks use it internally for gradient accumulation, but naive Python sum() does not.

The Log-Sum-Exp Trick

If you learn one numerical trick for ML, make it this one. It shows up in softmax, log-likelihood computations, attention scores, variational inference, everywhere. Single idea does huge amount of work.

Problem starts innocently. You have vector of numbers — say logits from neural network — and need to compute log(e^a₁ + e^a₂ + ... + e^aₙ). Maybe you compute softmax, log-partition function, or log-marginal likelihood. In exact math, straightforward. On computer, minefield.

Walk through concrete example. Suppose logits are [1000, 1001, 1002]. We need e^1000. How big? Number with 434 digits. Float64 maxes out around 10³⁰⁸. So e^1000 overflows to infinity, and whole computation returns inf. Now suppose logits are [-1000, -1001, -1002]. Every e^aᵢ underflows to 0.0, and we take log(0), which gives negative infinity. Both are garbage.

Trick looks simple. Factor out maximum value:

log(Σ e^aᵢ) = max(a) + log(Σ e^(aᵢ - max(a)))

Why does this work? Because Σ e^aᵢ = e^max(a) · Σ e^(aᵢ - max(a)), and taking log separates it into max(a) + log(Σ e^(aᵢ - max(a))). Now largest exponent in sum is e^(max(a) - max(a)) = e⁰ = 1. Nothing overflows. And at least one term in sum is 1, so argument to log is always at least 1, so we never take log of zero. Answer is mathematically identical to original expression. Numerics are completely different.

import numpy as np

def naive_logsumexp(x):
    return np.log(np.sum(np.exp(x)))

def stable_logsumexp(x):
    c = np.max(x)
    return c + np.log(np.sum(np.exp(x - c)))

big = np.array([1000.0, 1001.0, 1002.0])
print(naive_logsumexp(big))    # inf — overflow
print(stable_logsumexp(big))   # 1002.408 — correct

small = np.array([-1000.0, -1001.0, -1002.0])
print(naive_logsumexp(small))  # -inf — underflow
print(stable_logsumexp(small)) # -999.592 — correct

# scipy has it built in, and so does PyTorch
from scipy.special import logsumexp
print(logsumexp(big))          # 1002.408

Softmax uses same insight. Naively computing e^xᵢ / Σe^xⱼ overflows when any xᵢ is large. Fix: subtract max(x) from every element before exponentiating. Because constant appears in both numerator and denominator, it cancels perfectly — output is mathematically identical, but every exponent is now ≤ 0, so nothing overflows.

def stable_softmax(x):
    shifted = x - np.max(x)
    exp_x = np.exp(shifted)
    return exp_x / np.sum(exp_x)

logits = np.array([1000.0, 1001.0, 1002.0])
print(stable_softmax(logits))  # [0.0900, 0.2447, 0.6652] — correct

This is why we work with log-probabilities throughout ML instead of raw probabilities. Probability 10⁻³⁰⁰ underflows to zero in any floating-point format. But its log, -690.7, is ordinary number that fits comfortably in float32. Multiplying probabilities becomes adding log-probabilities. Division becomes subtraction. Everything stays in numerically comfortable range.

This is exactly why PyTorch's CrossEntropyLoss expects raw logits, not probabilities. Internally, it computes log_softmax — combining log and softmax into one numerically stable operation — and then applies negative log-likelihood. If you apply softmax yourself first and then pass result to NLLLoss, you break stability guarantee. Seen this mistake in production code. Model trains, sort of, but loss landscape gets noisy because of floating-point gremlins in separate softmax-then-log pipeline.

Key Insight

Every time you see np.exp() in your code, pause. Can input be large positive? Can it be very negative? If so, you probably need subtract-the-max pattern. Every time you see np.log(), ask: can argument be zero or negative? If so, you need log1p or log-sum-exp formulation. Build this reflex. It saves you from 90% of numerical bugs.

Solving Systems Without Inverting Matrices

I still reach for matrix inversion. Notation x = A⁻¹b looks clean on paper, so it feels natural to compute. Wrong move. Computing matrix inverse is roughly three times more expensive than solving system directly, and numerical errors are worse. Mathematically elegant approach and computationally smart approach are different here.

Given Ax = b, use np.linalg.solve(A, b). Under hood, this performs LU decomposition — it factors A into lower-triangular matrix L and upper-triangular matrix U, then solves two easy triangular systems by back-substitution. Same answer. Less work. Fewer rounding errors.

If your matrix A is symmetric and positive definite — common in ML, think covariance matrices, kernel matrices, Gram matrix XᵀX — you can do better. Cholesky decomposition factors A = LLᵀ where L is lower-triangular. It is about twice as fast as general LU, and more numerically stable because it exploits matrix structure. Use np.linalg.cholesky() followed by scipy.linalg.cho_solve().

But how do you know if system is trustworthy? Condition number of matrix, written κ(A), measures how much input errors get amplified in output. If κ(A) = 10ᵏ, you lose roughly k digits of precision in solution. Condition number of 10¹⁶ means your float64 answer has essentially zero reliable digits — you have been rearranging noise. Practical rule is simple: check it with np.linalg.cond(A), and if number is bigger than about 10¹⁰, treat solution with deep suspicion.

import numpy as np

A = np.array([[4.0, 1.0], [1.0, 3.0]])
b = np.array([1.0, 2.0])

# Don't do this:
x_bad = np.linalg.inv(A) @ b

# Do this:
x_good = np.linalg.solve(A, b)

print(f"Condition number: {np.linalg.cond(A):.2f}")  # 2.62 — healthy

# Ill-conditioned example
X = np.random.randn(100, 50)
XtX = X.T @ X
lam = 1.0
print(f"Unregularized cond: {np.linalg.cond(XtX):.0f}")
print(f"Regularized cond:   {np.linalg.cond(XtX + lam * np.eye(50)):.0f}")

That last example connects to something deep. In ridge regression, normal equations give (XᵀX)β = Xᵀy. If features are correlated, XᵀX becomes ill-conditioned — smallest eigenvalues sit near zero, condition number explodes, and solution oscillates wildly with tiny changes in data. Adding regularization term λI to get (XᵀX + λI)β = Xᵀy inflates those tiny eigenvalues and pulls condition number back to something sane. Regularization is not only about preventing overfitting. It also makes linear algebra solvable on machine with finite precision. Two problems, one fix.

When Matrices Get Too Big

LU and Cholesky are direct solvers — they give answer (up to floating-point rounding) in fixed number of steps. But they scale as O(n³). When n is a million — large graph Laplacian, kernel matrix for big dataset — you cannot even store full matrix in memory, let alone factor it.

Iterative methods approach solution gradually, producing better approximation with each step. Workhorse for symmetric positive definite systems is conjugate gradient method. It only needs matrix-vector products Av, never full matrix itself. That means it works beautifully with sparse matrices — matrices mostly full of zeros, stored in compact formats that record only non-zero entries.

Sparse matrices are everywhere in ML. Document-term matrix with 100,000 documents and 50,000 words has 5 billion entries, but single document uses maybe 500 unique words. That is 99.999% zeros. Storing full dense matrix is absurd. Two standard sparse formats are CSR (Compressed Sparse Row) and CSC (Compressed Sparse Column). CSR stores three arrays — non-zero values, their column indices, and pointers to where each row starts — making row slicing and matrix-vector products fast. CSC is column-oriented analog. In Python, scipy.sparse handles both. Rule of thumb: above about 90% zeros, sparse wins on memory and speed.

For finding eigenvalues of large sparse matrices — the top principal components, the smallest eigenvectors of a graph Laplacian for spectral clustering — the tools are different again. Power iteration is the simplest: multiply by A and normalize, repeat. It converges to the largest eigenvector. Google's original PageRank was essentially power iteration on the web's link matrix. For more eigenvectors simultaneously, the Lanczos algorithm (SciPy's scipy.sparse.linalg.eigsh()) is the standard. And for truly massive matrices where even Lanczos is slow, randomized SVD — project into a random low-dimensional subspace, then compute exact SVD there — is surprisingly effective. Scikit-learn's PCA uses it by default.

The GPU: A Factory Floor, Not an Office

A CPU is like a small team of brilliant engineers. Maybe 16 of them, each capable of complex reasoning, branching logic, and creative problem-solving. A GPU is like a factory floor with thousands of workers, each doing one simple repetitive task in lockstep. The CPU excels when the work is varied and sequential. The GPU excels when the work is uniform and parallel. Matrix multiplication — the heartbeat of neural networks — is exactly the kind of uniform, embarrassingly parallel work that a factory floor devours.

The GPU memory hierarchy is where most performance stories are actually told. Global memory — the VRAM you see on spec sheets, 24GB on an RTX 4090, 80GB on an A100 — is large but slow. Shared memory is a small, fast scratchpad (tens of KB) shared within a thread block, sitting on the chip itself. Registers are the fastest, private to each thread. The counterintuitive reality is that memory bandwidth, not compute, is usually the bottleneck. Moving data from global memory to the compute cores takes far longer than the arithmetic itself. A modern GPU can perform trillions of operations per second, but if every operation needs to fetch from global memory, it's sitting idle waiting for data most of the time.

This is why algorithms like FlashAttention matter. Standard attention computes and stores the full N×N attention matrix in global memory. FlashAttention tiles the computation so that each block fits in shared memory, processes it entirely on-chip, and writes only the final output back to global memory. The FLOPs are actually the same or slightly higher. But the memory traffic drops dramatically, so it's much faster. The lesson generalizes: in GPU computing, the question is rarely "how many operations?" and almost always "how many bytes moved?"

Modern GPUs also include Tensor Cores, specialized hardware units that perform small matrix multiplies (4×4 or 8×8 blocks) in reduced precision at extraordinary speed. A Tensor Core can multiply in float16 while accumulating the result in float32, getting roughly 2-8x the throughput of standard CUDA cores. This hardware is what makes the next section practical.

Mixed Precision — Training with Half the Bits

Training in full float32 is safe but wasteful. That's the insight behind mixed-precision training: most neural network operations — forward pass matrix multiplies, convolutions, attention — don't need 32 bits of precision. They work fine in 16-bit. So run the forward and backward passes in half precision (float16 or bfloat16), which uses half the memory and runs faster on Tensor Cores. Keep a master copy of the weights in float32, and do the weight update step in float32. That's the whole scheme.

The danger with float16 is its limited range. The smallest positive normal float16 is about 6×10⁻⁵. Gradient values smaller than this underflow to zero, and your model silently stops learning. Loss scaling handles this: before the backward pass, multiply the loss by a large constant (say 1024). This scales all gradients up into float16's representable range. After the backward pass, divide by the same constant before updating weights. PyTorch's GradScaler does this automatically, dynamically increasing the scale when training is stable and decreasing it when it detects overflow.

import torch
from torch.cuda.amp import autocast, GradScaler

model = MyModel().cuda()
optimizer = torch.optim.Adam(model.parameters())
scaler = GradScaler()

for inputs, targets in dataloader:
    optimizer.zero_grad()
    with autocast():                          # forward in fp16
        outputs = model(inputs.cuda())
        loss = loss_fn(outputs, targets.cuda())
    scaler.scale(loss).backward()             # backward in fp16 with scaled loss
    scaler.step(optimizer)                    # unscale gradients, update in fp32
    scaler.update()                           # adjust scale factor

The choice between float16 and bfloat16 is a tradeoff between precision and range. Float16 gives you 5 exponent bits and 10 mantissa bits — decent precision, but a narrow range that requires loss scaling. Bfloat16 gives you 8 exponent bits (the same as float32!) but only 7 mantissa bits — coarser precision, but the same range as float32. In practice, bfloat16 almost never overflows or underflows, which means you can often skip loss scaling entirely. This is why Google designed it for their TPUs, and why NVIDIA's A100 and H100 both support it natively. If your hardware supports bfloat16, use it. It's the less fussy option.

There's also FP8, which emerged in 2023-2024 on NVIDIA's Hopper architecture (H100). It comes in two flavors: E4M3 (4 exponent, 3 mantissa bits) for forward passes and E5M2 (5 exponent, 2 mantissa bits) for backward passes. The H100's Tensor Cores operate natively on FP8, offering up to 4x throughput over float16. My intuition for when FP8 is safe is still developing, but the trajectory is clear — each hardware generation pushes viable precision lower.

The practical impact of mixed precision is enormous: it roughly halves memory usage (which means you can double your batch size or fit a larger model), and runs 1.5-3x faster on Tensor Core-enabled hardware. In 2024, there is no good reason to train in full float32 unless you're actively debugging a numerical issue. Mixed precision isn't an optimization. It's the default.

Danger

Never reduce precision for the weight update itself. Accumulating tiny gradient updates into float16 weights destroys training through rounding errors — the updates are smaller than float16's epsilon relative to the weights, so they vanish. The pattern is sacred: forward in fp16, backward in fp16, accumulate and update in fp32. Always keep a float32 master copy of your weights.
