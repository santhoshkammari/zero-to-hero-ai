Optimization asks for best answer — lowest loss, tightest fit, widest margin. Every ML model does optimization under hood. Linear regression minimizes squared error. SVMs maximize margin. Neural networks minimize loss over millions of parameters. Math changes; question stays same: where bottom of landscape?

This section covers mathematical foundations: optimization landscape shape (convex vs. non-convex), constrained optimization with Lagrangians and KKT conditions, duality theory, second-order methods. These tools support everything from SVMs to variational inference.

What Optimization Is

Imagine blindfolded hiker on huge landscape. Goal: find lowest point — deepest valley. Hiker cannot see terrain, only feel slope under feet. Ground tilts left, step left. Ground drops hard ahead, take bigger step. That is optimization. Move through unseen landscape using local downhill signal.

In math language, we have function f(x) and want input x* that makes f as small as possible. Function f is objective function (in ML, usually loss function). Input x is parameter vector — in neural network, millions of weights and biases. Landscape is surface from plotting loss against all parameter values. Point lowest among nearby points is local minimum. Point lowest everywhere is global minimum. Blindfolded hiker wants global minimum, but often ends at good local one.

Fundamental tool is gradient, written ∇f(x). It is vector pointing in direction of steepest ascent, and its magnitude tells steepness. To go downhill, walk opposite gradient. That is core intuition behind gradient-based optimization. Most of what follows makes that move smarter.

Convex vs. Non-Convex

Now terrain matters. Same gradient descent algorithm behaves very differently depending on landscape shape.

Convex function curves upward everywhere — think bowl. More precisely, draw straight line between any two points on graph; line stays above or on graph. No dips. No bumps. One smooth valley. Canonical example: f(x) = x², parabola. Mathematically, f is convex if for any two points x, y and any t ∈ [0, 1]:

f(t·x + (1-t)·y) ≤ t·f(x) + (1-t)·f(y)

Inequality says: function value at blend of two inputs is at most blend of two function values. If this holds for your function, hard optimization part mostly done.

Why? For convex functions, every local minimum is global minimum. No deceptive valleys. If blindfolded hiker finds flat spot at bottom of bowl, that spot is bottom. Gradient descent converges to this unique optimum (with small enough learning rate). No tricks. No restarts. No prayers.

Convex set is geometric cousin: region where line segment between any two interior points stays inside region. Filled circle is convex. Crescent moon is not. This matters because many optimization problems restrict x to feasible region. If region is convex and objective is convex, problem stays tractable.

Classical ML lives mostly in convex territory. Linear regression with squared error gives paraboloid — one bottom, guaranteed. Logistic regression with log-loss is convex. SVMs minimize convex quadratic under linear constraints. Ridge and Lasso add convex penalties. For these models, gradient descent (or specialized solver) finds global optimum, not approximation. Strong guarantee. Big reason these models stay trustworthy.

Bad news: non-convex function has many valleys, ridges, plateaus, saddle points. Hiker landscape is no longer one bowl — it is mountain range. Deep learning loss functions are strongly non-convex. Layer interactions, nonlinear activations, huge dimensionality — all create landscape with enormous number of critical points. Gradient descent may find local minimum, but no guarantee of global one.

Remarkable empirical fact: this usually does not matter much. Reason involves saddle points, strange high-dimensional landscape structure, noise in SGD. But theory split matters: convex means relax; non-convex means use tools that handle uncertainty.

Related result: Jensen's inequality. For convex function f and random variable X, f(E[X]) ≤ E[f(X)]. Function of average is at most average of function. This supports Evidence Lower Bound (ELBO) in variational inference and explains why many ML bounds stay tight for convex objectives and loose for non-convex ones.

Key Insight

If you can prove loss function is convex, hard part is mostly solved. Optimizer becomes detail — almost any reasonable algorithm finds answer. Real ML difficulty: most interesting models make non-convex landscapes. Know terrain first. Then pick tools.

Constrained Optimization

Everything so far was unconstrained: minimize f(x) with x free to roam anywhere. Real problems are stricter. Probabilities must sum to one. Budgets cannot be exceeded. Physical quantities cannot go negative. Resource allocations have upper bounds. Constrained optimization means minimize objective while keeping solution inside feasible region defined by constraints.

Constraints come in two flavors: equality constraints (g(x) = 0, solution lies exactly on surface) and inequality constraints (h(x) ≤ 0, solution stays on one side of boundary). Standard form:

minimize f(x)
subject to  g_i(x) = 0  for i = 1, ..., m      (equality constraints)
            h_j(x) ≤ 0  for j = 1, ..., p      (inequality constraints)

The Lagrangian Method

Core idea: turn constrained problem into unconstrained one by folding constraints into objective. For problem with equality constraints, Lagrangian is:

L(x, λ) = f(x) + Σ λ_i · g_i(x)

Each constraint gets its own Lagrange multiplier λ_i — new variable optimizer finds alongside x. At optimum, gradient of f must be proportional to gradient of each active constraint. Multipliers capture how strongly each constraint pulls solution.

Concrete example. Minimize f(x, y) = x² + y² (point closest to origin) subject to constraint x + y = 4 (point must lie on line x + y = 4). Without constraint, answer is (0, 0). With constraint, find point on line x + y = 4 closest to origin.

Lagrangian is L(x, y, λ) = x² + y² + λ(x + y − 4). Take partial derivatives. Set them to zero:

∂L/∂x = 2x + λ = 0  →  x = -λ/2
∂L/∂y = 2y + λ = 0  →  y = -λ/2
∂L/∂λ = x + y - 4 = 0

From first two equations, x = y. Plug into constraint: 2x = 4, so x = y = 2 and λ = −4. Optimal point is (2, 2), closest point on line to origin. Multiplier λ = −4 gives sensitivity: if we relax constraint from x + y = 4 to x + y = 4 + ε, optimal objective decreases by about 4ε. Multiplier is constraint "price."

KKT Conditions

Equality constraints are tidy. Inequality constraints add complexity. Karush-Kuhn-Tucker (KKT) conditions extend Lagrange multipliers to handle both types. For constrained optimization problem to have optimal solution, these four conditions must hold:

1. Stationarity: Gradient of Lagrangian with respect to x is zero. Same "gradient equals zero" rule, but now Lagrangian includes objective and constraint terms.

2. Primal feasibility: All constraints are satisfied. Solution must actually lie in feasible region — cannot minimize cost by spending money you do not have.

3. Dual feasibility: Multipliers for inequality constraints are non-negative (μ_j ≥ 0). This makes inequality constraints act as penalties, not rewards. Negative multiplier would make constraint "help" objective instead of constrain it.

4. Complementary slackness: For each inequality constraint, either constraint is tight (h_j(x) = 0, constraint is active) or its multiplier is zero (μ_j = 0). Cannot have loose constraint and nonzero multiplier together. Optimizer only "pays attention" to constraints that are actually binding.

Complementary slackness is deepest rule. At optimum, inactive constraints do not matter — their multipliers are zero, so they add nothing to Lagrangian. Only active constraints shape solution. Same idea drives Support Vector Machines: only data points where margin constraint is tight (support vectors) determine decision boundary. Remove other training points. Solution stays same.

Duality

Every constrained optimization problem has twin: dual problem. Original is primal. Primal minimizes over x with constraints; dual maximizes over Lagrange multipliers. Dual comes from Lagrangian:

Primal: minimize_x  f(x)  subject to constraints
Dual:   maximize_λ  min_x L(x, λ)  subject to λ ≥ 0

Weak duality always holds: dual optimum is lower bound on primal optimum. Dual cannot beat primal. Strong duality says they are equal — gap is zero — and holds when primal problem is convex and satisfies mild regularity conditions (Slater's condition, roughly meaning feasible region has interior point). When strong duality holds, solving dual equals solving primal, and dual is sometimes easier.

SVMs are textbook example. Primal SVM is quadratic program over weight vector w. Dual is quadratic program over Lagrange multipliers α_i, one per training point. Strong duality holds, so both give same solution. Dual has crucial advantage: it depends on training data only through dot products x_i · x_j. That makes kernel trick possible — replace dot product with kernel function K(x_i, x_j) that implicitly computes dot products in high-dimensional feature space, without building coordinates. Dual formulation does not only make SVMs solvable — it makes them powerful.

LP and QP

Constrained optimization problems come in standard forms, each with specialized solvers. Linear Programming (LP) uses linear objective and linear constraints — supply chain routing, resource allocation, diet problems. Quadratic Programming (QP) adds quadratic objective with linear constraints — exact SVM formulation. Semidefinite Programming (SDP) constrains matrix to be positive semidefinite; this appears in kernel learning and relaxations of combinatorial problems.

Interior point methods are modern workhorse. Unlike simplex method (which walks along edges of feasible polytope), interior point methods move through interior, using barrier functions to stay away from constraint boundaries. They have polynomial-time complexity and scale well to large problems.

import numpy as np
from scipy.optimize import linprog, minimize

# Linear Programming: diet problem
# Minimize cost: 2*bread + 3.5*milk + 8*cheese
# Subject to:
#   6*bread + 2*milk + 1*cheese >= 10  (protein)
#   1*bread + 8*milk + 7*cheese >= 15  (calcium)
#   all quantities >= 0

c = [2, 3.5, 8]
A_ub = [[-6, -2, -1], [-1, -8, -7]]  # negate for >= to <=
b_ub = [-10, -15]
bounds = [(0, None), (0, None), (0, None)]

result = linprog(c, A_ub=A_ub, b_ub=b_ub, bounds=bounds, method='highs')
print(f"Bread={result.x[0]:.2f}, Milk={result.x[1]:.2f}, "
      f"Cheese={result.x[2]:.2f}")
print(f"Minimum cost: ${result.fun:.2f}")

# Constrained nonlinear: minimize x^2 + y^2 subject to x + y = 4
# Using Lagrangian approach via scipy
def objective(xy):
    return xy[0]**2 + xy[1]**2

constraint = {'type': 'eq', 'fun': lambda xy: xy[0] + xy[1] - 4}
result = minimize(objective, [0, 0], constraints=constraint)
print(f"\nClosest point on x+y=4 to origin: ({result.x[0]:.2f}, {result.x[1]:.2f})")
print(f"Distance squared: {result.fun:.2f}")  # 8.0
Watch Out

When converting "≥" constraints for solvers that expect "≤", negate both sides. Most common source of sign errors in constrained optimization. Always plug solution back into original constraints. Correct solution to wrong problem is worse than useless.

Second-Order Methods

Hessian intuition is hard. Theory is clear, equations are clear, but "curvature matrix" for network with 100 million parameters is hard to picture. So here is practical view: what these methods do, why they exist, why people rarely use them.

Gradient descent uses gradient — first-order information. It knows downhill direction, but not how slope changes. Second-order methods use Hessian — matrix of second partial derivatives — which tells curvature of landscape. Hessian shows whether slope gets steeper or flatter, and in which directions.

Canonical second-order method is Newton's method. Instead of step proportional to gradient, Newton's method takes step proportional to inverse Hessian times gradient:

x_new = x_old - H⁻¹ · ∇f(x_old)

Why? Hessian tells shape of local bowl. If bowl is wide (gentle curvature), Hessian says "take bigger step." If narrow (steep curvature), it says "take smaller step." Newton's method adjusts step size and direction to match local geometry, so it can converge in far fewer steps — often one step for quadratic functions, because quadratic approximation is exact.

For pure quadratic like f(x) = (1/2)x^T A x − b^T x, Newton's method finds exact minimum in one step, regardless of how badly conditioned A is. Gradient descent on same problem may need thousands of steps if A has wide spread of eigenvalues. That is appeal of second-order methods: they resist ill-conditioning that hurts first-order methods.

Problem is cost. Hessian for model with n parameters is n × n matrix. For network with 10 million parameters, that is 10^14 entries — roughly 100 terabytes in single precision. Cannot compute it. Cannot store it. Cannot invert it. Even approximate methods struggle at this scale.

L-BFGS (Limited-memory Broyden-Fletcher-Goldfarb-Shanno) is the most practical second-order-ish method. It doesn't compute the Hessian at all. Instead, it approximates the inverse Hessian using the last m gradient evaluations (typically m = 5-20). The memory cost is O(mn) instead of O(n²), which is manageable. L-BFGS works well for problems where the number of parameters is moderate (thousands to low millions) and the function is smooth. It's the default optimizer in scipy's minimize function and works beautifully for logistic regression, CRFs, and other classical models. But for deep learning with hundreds of millions of parameters and noisy mini-batch gradients, L-BFGS doesn't shine.

K-FAC (Kronecker-Factored Approximate Curvature, Martens and Grosse, 2015) is designed specifically for neural networks. It approximates the Fisher information matrix (closely related to the Hessian for neural networks) using Kronecker products of smaller matrices, one per layer. This reduces the n² cost to something tractable. K-FAC has shown speedups over SGD on some benchmarks, but it adds significant implementation complexity and memory overhead. It's an active research area, not a mainstream tool.

The bottom line is this: first-order methods dominate deep learning because the Hessian is too expensive to compute, and the noise in mini-batch gradients undermines the accuracy of second-order approximations anyway. Adam is, in a sense, a cheap second-order approximation — its second-moment estimate v tracks the diagonal of the Hessian (roughly: the curvature in each parameter direction independently). It captures the most important curvature information at negligible cost. The dream of practical second-order deep learning optimization remains open.

We started with the simplest possible picture: a blindfolded hiker feeling the slope, stepping downhill. We saw that convex landscapes are the friendly ones — one valley, guaranteed convergence — while non-convex landscapes are full of deceptive structure. Jensen's inequality connected convexity to probabilistic bounds used across ML.

Constrained optimization gave us Lagrangians, KKT conditions, duality, and the kernel trick that makes SVMs work. Second-order methods showed us what we're missing (curvature information) and why we can't afford it (the Hessian is too big).
