ML Taxonomy

Chapter 4: ML Fundamentals & Core Concepts Section 1 of 8
TL;DR

ML comes in a handful of flavors — supervised, unsupervised, reinforcement, and a couple of hybrids. Picking the right one is mostly about what kind of data you have and what you're trying to get out of it. This section is the map. The rest of the chapter is the terrain.

The Taxonomy

Every ML problem falls into one of these buckets. The deciding factor isn't which algorithm sounds coolest — it's what your data looks like and what you need back.

Type You Have You Get Reach for It When
Supervised — Classification Inputs + discrete labels Category predictions Spam/not-spam, disease diagnosis, image tagging
Supervised — Regression Inputs + continuous values Number predictions House prices, demand forecasting, ETA estimation
Unsupervised — Clustering Inputs, no labels Natural groupings Customer segmentation, topic discovery, gene expression grouping
Unsupervised — Dimensionality Reduction High-dimensional inputs Compressed representation Visualization, noise removal, preprocessing before another model
Unsupervised — Anomaly Detection Mostly "normal" data Outlier flags Fraud detection, manufacturing defects, intrusion detection
Reinforcement Learning An environment + reward signal A policy (strategy) Game playing, robotics, resource allocation where you can simulate
Semi-Supervised Few labels + lots of unlabeled data Better predictions than labels alone Medical imaging, any domain where labeling is expensive
Self-Supervised Raw data (no labels at all) Rich learned representations Language models (GPT, BERT), vision pre-training (DINO, MAE)

Supervised learning is the workhorse — probably 80% of production ML. If you have labels, start there. Unsupervised is what you reach for when labels don't exist or when you're exploring data you don't yet understand. Reinforcement learning is powerful but niche; unless you have a simulator or a game, you probably don't need it yet. Semi-supervised and self-supervised are how modern large models get built — they're the reason GPT can exist without someone hand-labeling trillions of tokens.

Key Terminology — Quick-Fire

These terms show up everywhere. Internalize them now so you're never tripped up later.

Features (X) — the input columns. Square footage, pixel values, word counts. Also called predictors or input variables. If your data is a spreadsheet, features are the columns you feed the model.

Target (y) — the thing you're predicting. Sale price, spam/not-spam, next word. Only exists in supervised settings.

Loss function — how the model measures its own wrongness. MSE for regression, cross-entropy for classification. Training = minimize the loss. Pick the wrong loss and the model will cheerfully optimize for something you don't care about.

Parameters — values the model learns during training. Weights in a neural net, coefficients in linear regression. You never set these by hand.

Hyperparameters — values you set before training. Learning rate, number of trees, regularization strength. The model can't learn these — you choose them via experimentation or cross-validation.

# Parameters vs hyperparameters in one glance
from sklearn.ensemble import RandomForestClassifier

clf = RandomForestClassifier(
    n_estimators=200,   # ← hyperparameter (you pick)
    max_depth=10,       # ← hyperparameter (you pick)
)
clf.fit(X_train, y_train)
# clf.estimators_[0].tree_.threshold  ← parameter (learned)
# clf.estimators_[0].tree_.value      ← parameter (learned)

Generalization — the model's ability to work on data it wasn't trained on. This is the whole game. If your model memorizes training data but chokes on new data, you've built an expensive lookup table. The gap between training performance and test performance tells you how well you're generalizing.

When Not to Use ML

The best ML engineers know when to close the laptop. If a handful of if/else statements or a SQL query solves the problem, use that — it's faster to build, easier to debug, and won't silently degrade when the data shifts. ML earns its keep when patterns are too complex to hand-code, when the rules change over time, or when you have far more data than you have intuition about what drives the outcome. If you can't articulate what "correct" looks like — or you have fewer than a few hundred examples — you're not ready for ML yet.

✅ What You Should Now Be Able To Do
  1. Look at a new problem and classify it as supervised (classification or regression), unsupervised, reinforcement, semi-supervised, or self-supervised.
  2. Explain the difference between features and target — and spot them in a real dataset.
  3. Distinguish parameters (learned) from hyperparameters (set by you).
  4. Describe what a loss function does and why choosing the right one matters.
  5. Define generalization and explain why it's the central goal of ML.
  6. Articulate when ML is the wrong tool and a simpler approach would be better.
  7. Given a scenario ("I have 50k unlabeled images and 200 labeled ones"), pick the right learning type.
