ML Optimization

Chapter 4: ML Fundamentals & Core Concepts Section 7 of 8 9 subtopics
TL;DR

Training a model means finding the parameters that minimize a loss function — and gradient descent is how you do it. You compute the slope, take a step downhill, repeat. The learning rate controls step size and is the single most important hyperparameter you'll ever tune. SGD is noisy; momentum smooths it out; Adam adapts the learning rate per-parameter and is the default for deep learning. Learning rate schedules, gradient clipping, and the distinction between sharp and flat minima round out the practical toolkit. Backpropagation is the calculus trick that makes computing gradients feasible for neural networks — we plant the seed here and go deep later.

The Intuition: Hiking in Fog

Picture this. You're standing in thick fog on a hilly landscape. You can't see where the lowest valley is. You can't even see twenty feet ahead. But you can feel the slope under your boots — the ground tilts left, and downward, at about a 15-degree angle.

So you take a step in the downhill direction. Then you feel the slope again. Take another step. Keep going. Eventually, you end up in a valley. Maybe not the deepest valley on the whole landscape, but a valley.

That's gradient descent. The entire idea.

  • The landscape is your loss function plotted over all possible parameter values.
  • The slope under your feet is the gradient — a vector of partial derivatives telling you which direction is steepest uphill.
  • The step you take goes in the negative gradient direction (downhill).
  • The step size is the learning rate.

In math, the update rule is beautifully simple:

# The entire algorithm in one line
θ = θ - α * ∇L(θ)

# θ = parameters (the thing we're learning)
# α = learning rate (step size)
# ∇L(θ) = gradient of the loss with respect to θ (the slope)

That's it. Every optimizer you'll ever use — SGD, Adam, AdaGrad, all of them — is a variation on this theme. They all compute some version of "which direction is downhill" and take some version of "a step in that direction." The differences are in how they compute the direction and how they decide the step size.

Gradient Descent Variants

The gradient ∇L(θ) is defined as the derivative of the loss over the entire dataset. But computing it over the entire dataset for every single step is expensive. This tension — accuracy of the gradient estimate vs. computational cost per step — gives rise to three variants.

Batch Gradient Descent

Compute the gradient using all N training examples, then take one step.

# Batch gradient descent — one epoch
gradient = compute_gradient(X_all, y_all, θ)  # uses ALL data
θ = θ - α * gradient

The gradient estimate is exact (well, exact for this dataset). The path toward the minimum is smooth. But if you have 10 million training examples, you're computing a gradient over 10 million examples just to take one step. Then you do it again. For large datasets, this is impractical.

Stochastic Gradient Descent (SGD)

Compute the gradient using one randomly sampled example, then take one step. Repeat.

# SGD — one epoch
for i in random_permutation(range(N)):
    gradient = compute_gradient(X[i], y[i], θ)  # ONE sample
    θ = θ - α * gradient

The gradient estimate is noisy — a single sample is a terrible estimate of the true gradient. The path toward the minimum looks drunk. But you're taking N steps per pass through the data instead of one. And here's the surprising part: the noise is actually helpful. It helps the optimizer escape shallow local minima and saddle points. A smooth, exact gradient might settle into a mediocre valley and stay there. A noisy one bounces around enough to find better valleys.

Mini-batch Gradient Descent

The practical middle ground. Compute the gradient on a random batch of B examples (typically 32–512), then take a step.

# Mini-batch GD — one epoch
for batch_X, batch_y in random_batches(X, y, batch_size=64):
    gradient = compute_gradient(batch_X, batch_y, θ)  # B samples
    θ = θ - α * gradient

This is what everyone actually uses. When someone says "SGD" in a paper or codebase, they almost always mean mini-batch SGD. The batch size controls the noise-accuracy tradeoff:

In practice, pick the largest batch size that fits in GPU memory and gives you good generalization. For most deep learning tasks, 32–256 is the sweet spot. Very large batch sizes (thousands) can work but often require learning rate warmup and careful tuning.

Learning Rate: The Most Important Hyperparameter

If you only tune one thing, tune the learning rate. Nothing else comes close.

The learning rate α controls how big each step is. Get it wrong and nothing else matters:

  • Too high: You overshoot the minimum. The loss oscillates wildly or explodes to infinity. Your model diverges and you get NaN losses.
  • Too low: You creep toward the minimum painfully slowly. Training takes forever, and you might get stuck in a bad local minimum because you lack the momentum to escape.
  • Just right: You converge quickly to a good minimum. The loss drops steadily, maybe with some healthy oscillation.
import numpy as np

# Simple demo: minimize f(x) = x² (minimum at x=0)
def gradient(x):
    return 2 * x

# Too high — diverges
x = 5.0
for step in range(10):
    x = x - 1.1 * gradient(x)
    print(f"Step {step}: x={x:.2f}, f(x)={x**2:.2f}")
# x explodes: 5.0 → -6.0 → 7.2 → -8.64 → ...

# Too low — barely moves
x = 5.0
for step in range(10):
    x = x - 0.001 * gradient(x)
    print(f"Step {step}: x={x:.4f}, f(x)={x**2:.6f}")
# x crawls: 5.0 → 4.99 → 4.98 → ...

# Just right
x = 5.0
for step in range(10):
    x = x - 0.3 * gradient(x)
    print(f"Step {step}: x={x:.4f}, f(x)={x**2:.6f}")
# x converges: 5.0 → 2.0 → 0.8 → 0.32 → ...
💡 Practical Tip

A good starting learning rate for Adam is 3e-4 (the "Karpathy constant"). For SGD with momentum, start around 0.1 and decay it. If your loss goes to NaN, your learning rate is too high. If your loss barely drops after thousands of steps, it's too low. The learning rate finder (train for one epoch with exponentially increasing LR, plot loss vs. LR, pick the steepest descent point) is a fast way to find a good starting value.

Momentum: Smoothing Out the Noise

Vanilla SGD has a problem. The gradient at any single mini-batch is noisy, and the loss landscape is often shaped like an elongated valley — steep in one direction, shallow in another. SGD oscillates back and forth across the steep walls while making slow progress along the valley floor. It's like trying to roll a ball down a half-pipe — it wastes energy bouncing side to side.

Momentum fixes this by keeping a running average of past gradients. Instead of stepping based only on the current gradient, you accumulate velocity:

# SGD with Momentum
v = 0  # velocity (same shape as θ)
β = 0.9  # momentum coefficient (typically 0.9 or 0.99)

for batch_X, batch_y in batches:
    g = compute_gradient(batch_X, batch_y, θ)
    v = β * v + g           # accumulate: 90% old velocity + new gradient
    θ = θ - α * v           # step using velocity, not raw gradient

Think of it as a heavy ball rolling downhill. In directions where the gradient keeps pointing the same way (the valley floor), velocity builds up — the ball accelerates. In directions where the gradient keeps flipping sign (the valley walls), the positive and negative contributions cancel out — the oscillations dampen. Same physics as a low-pass filter.

The momentum coefficient β is typically 0.9 (which means the velocity is an exponential moving average of roughly the last 10 gradients). Some people use 0.99 for smoother paths.

Nesterov Momentum

Standard momentum computes the gradient at your current position, then applies velocity. Nesterov accelerated gradient (NAG) does it in reverse: first take a step in the direction of accumulated velocity (a "lookahead"), then compute the gradient at that lookahead position.

# Nesterov Momentum
v = 0
β = 0.9

for batch_X, batch_y in batches:
    θ_lookahead = θ - α * β * v         # peek ahead
    g = compute_gradient(batch_X, batch_y, θ_lookahead)  # gradient at lookahead
    v = β * v + g
    θ = θ - α * v

Why does this matter? If momentum is carrying you past the minimum, Nesterov catches the mistake sooner because it evaluates the gradient at the future position rather than the current one. It's a small tweak with meaningful improvements in convergence, especially for convex problems. In practice, the difference over standard momentum is modest but free — use it when available.

Adaptive Learning Rate Methods

Here's the key insight behind all adaptive methods: different parameters need different learning rates.

Consider word embeddings. The embedding for "the" gets updated on nearly every batch. The embedding for "defenestrate" might get updated once in a thousand batches. If you use the same learning rate for both, you're either updating "the" too aggressively or "defenestrate" too conservatively. Adaptive methods solve this by maintaining a per-parameter learning rate that adjusts automatically based on the history of gradients for that parameter.

AdaGrad

AdaGrad tracks the sum of squared gradients for each parameter. Parameters that have received large gradients get smaller learning rates; parameters with small gradients get larger learning rates.

# AdaGrad
cache = 0  # sum of squared gradients (same shape as θ)
ε = 1e-8   # small constant to avoid division by zero

for batch_X, batch_y in batches:
    g = compute_gradient(batch_X, batch_y, θ)
    cache = cache + g ** 2
    θ = θ - α * g / (np.sqrt(cache) + ε)

For sparse problems (NLP, recommender systems), AdaGrad is great — rare features get amplified. But there's a fatal flaw: cache only grows. It never shrinks. Over time, the effective learning rate drops to near zero and learning stops entirely. For long training runs, AdaGrad dies.

RMSprop

Geoffrey Hinton's fix for AdaGrad's dying learning rate problem. Instead of accumulating all past squared gradients, use an exponential moving average:

# RMSprop
cache = 0
β = 0.999  # decay rate (sometimes 0.99)
ε = 1e-8

for batch_X, batch_y in batches:
    g = compute_gradient(batch_X, batch_y, θ)
    cache = β * cache + (1 - β) * g ** 2   # EMA of squared gradients
    θ = θ - α * g / (np.sqrt(cache) + ε)

The exponential moving average "forgets" old gradients, so the learning rate doesn't decay to zero. RMSprop was never published in a paper — Hinton introduced it in a Coursera lecture. It works well in practice and is the direct ancestor of Adam.

Adam: The Default Optimizer

Adam (Adaptive Moment Estimation) combines the best of momentum and RMSprop. It maintains two running averages:

  • First moment (m) — exponential moving average of gradients (like momentum)
  • Second moment (v) — exponential moving average of squared gradients (like RMSprop)
# Adam — the full algorithm
m = 0       # first moment estimate
v = 0       # second moment estimate
β1 = 0.9    # decay rate for first moment
β2 = 0.999  # decay rate for second moment
ε = 1e-8
t = 0       # timestep

for batch_X, batch_y in batches:
    t += 1
    g = compute_gradient(batch_X, batch_y, θ)

    m = β1 * m + (1 - β1) * g          # update first moment (momentum)
    v = β2 * v + (1 - β2) * g ** 2     # update second moment (RMSprop)

    m_hat = m / (1 - β1 ** t)          # bias correction
    v_hat = v / (1 - β2 ** t)          # bias correction

    θ = θ - α * m_hat / (np.sqrt(v_hat) + ε)

The bias correction terms (m_hat and v_hat) are important. Since m and v are initialized to zero, they're biased toward zero in the early steps. Dividing by (1 - β^t) corrects this — it's a bigger correction early on and vanishes as t grows.

Adam is the most widely used optimizer in deep learning. The default hyperparameters (β1=0.9, β2=0.999, ε=1e-8) work well across a huge range of problems. When in doubt, use Adam.

AdamW: What You Should Actually Use

There's a subtle bug in Adam's interaction with weight decay (L2 regularization). The original Adam paper applied weight decay as part of the gradient, but Adam's adaptive scaling distorts the regularization effect. AdamW fixes this by decoupling the weight decay from the gradient update:

# AdamW — decoupled weight decay
# ... (same m, v, bias correction as Adam) ...

θ = θ - α * m_hat / (np.sqrt(v_hat) + ε) - α * λ * θ
#                                            ^^^^^^^^
#                      weight decay applied DIRECTLY to params, not through gradient

The difference is significant enough to matter in practice. AdamW consistently outperforms Adam with L2 regularization, particularly on larger models. Use AdamW over Adam unless you have a specific reason not to. PyTorch's torch.optim.AdamW and Hugging Face's defaults both use AdamW.

⚠️ SGD vs. Adam: The Ongoing Debate

Adam converges faster and is easier to tune. SGD with a well-tuned learning rate schedule often generalizes slightly better on vision tasks. In NLP and transformers, AdamW dominates. The practical answer: start with AdamW. If you're squeezing out the last 0.3% accuracy on an image classification benchmark and you have the compute budget to tune LR schedules, try SGD with momentum + cosine annealing.

Learning Rate Schedules

A fixed learning rate is almost never optimal. The intuition is simple: early in training, you're far from the minimum and need big steps to explore broadly. Late in training, you're close to the minimum and need small steps to refine without overshooting. Starting high and annealing down captures this perfectly.

Step decay: Multiply the learning rate by a factor (e.g., 0.1) at fixed epochs. Simple, effective, requires you to pick the schedule manually. Common in older vision papers: "divide LR by 10 at epochs 30, 60, 90."

Cosine annealing: Smoothly decays the LR following a cosine curve from the initial value to near zero. No discrete jumps. Works well across many tasks and is the default schedule for many modern training recipes.

Warmup: Start with a very small learning rate and linearly increase it over the first few hundred or thousand steps. This stabilizes training early on when the model's parameters are random and gradients are unreliable. Almost always used with transformers. A typical recipe: linear warmup for 5–10% of training steps, then cosine decay for the rest.

import torch.optim as optim
from torch.optim.lr_scheduler import CosineAnnealingLR, LinearLR, SequentialLR

optimizer = optim.AdamW(model.parameters(), lr=3e-4, weight_decay=0.01)

# Warmup for 1000 steps, then cosine decay for 9000 steps
warmup = LinearLR(optimizer, start_factor=0.01, total_iters=1000)
cosine = CosineAnnealingLR(optimizer, T_max=9000, eta_min=1e-6)
scheduler = SequentialLR(optimizer, [warmup, cosine], milestones=[1000])

One-cycle policy: Ramp the LR up to a high value, then anneal it down below the starting value, all within one training run. Leslie Smith showed this can match or beat carefully tuned step schedules with zero manual scheduling decisions. PyTorch has it built in: OneCycleLR.

Convergence Challenges

Saddle Points, Not Local Minima

For years, people worried that gradient descent would get trapped in bad local minima. Turns out, in high dimensions, local minima aren't the problem — saddle points are.

A saddle point is where the gradient is zero, but it's a minimum in some directions and a maximum in others (like the center of a horse saddle or a Pringles chip). In a network with millions of parameters, for a critical point to be a true local minimum, the loss must curve upward in every single one of those millions of directions. That's astronomically unlikely. Almost every critical point in high-dimensional loss landscapes is a saddle point.

The good news: SGD with momentum usually escapes saddle points because the noise in stochastic gradients provides enough perturbation to push past them. This is another reason why the "noise" in SGD is a feature, not a bug.

Sharp vs. Flat Minima

Not all minima are equally good. A sharp minimum sits at the bottom of a narrow, steep-walled valley. A flat minimum sits in a broad, gently-sloped basin. Both have low training loss, but flat minima generalize better.

Why? A sharp minimum means tiny changes in parameters cause large changes in loss. When your training and test distributions differ even slightly (they always do), a sharp minimum's performance collapses. A flat minimum is robust — small perturbations barely change the loss.

This connects to practical choices: large batch training tends to find sharp minima (bad generalization). Smaller batches, thanks to gradient noise, tend to find flat minima (better generalization). It's one reason why just cranking up the batch size doesn't always help.

Gradient Clipping

Sometimes gradients explode. This is especially common in recurrent neural networks (RNNs) and early-stage transformer training. One bad batch produces a gradient with a norm of 10,000 instead of 1, and a single update destroys your model's learned parameters.

The fix is embarrassingly simple: if the gradient norm exceeds a threshold, scale it down.

# Gradient clipping in PyTorch — one line
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)

# What it does under the hood:
# total_norm = sqrt(sum(p.grad.norm()**2 for all params))
# if total_norm > max_norm:
#     scale = max_norm / total_norm
#     for p in params: p.grad *= scale

Clip value of 1.0 is a safe default. Most major training codebases use gradient clipping — it's essentially free insurance against exploding gradients.

Connection to Backpropagation

Gradient descent and backpropagation are often confused with each other. They're not the same thing. They solve two different problems:

  • Gradient descent says: "Given the gradient of the loss with respect to every parameter, update each parameter by stepping in the downhill direction." It's the optimization algorithm.
  • Backpropagation says: "Here's how to compute that gradient efficiently for a neural network." It's the calculus trick — an application of the chain rule that propagates error signals backward through the network's layers.

Without backpropagation, you could still do gradient descent — you'd just have to compute gradients by finite differences (perturb each parameter by ε, measure the change in loss). For a model with 100 million parameters, that means 100 million forward passes per gradient computation. Backpropagation computes the exact same gradient in one backward pass. That's the magic — it turns an O(n) problem into O(1) in terms of forward passes.

We'll go deep on the mechanics of backpropagation in the deep learning chapters. For now, just remember the division of labor: gradient descent decides what to do with gradients. Backprop decides how to compute them.

Second-Order Methods (Brief)

Everything we've discussed so far is first-order — we only use the gradient (first derivative). Second-order methods also use the curvature (second derivative, encoded in the Hessian matrix). The Hessian tells you not just which direction is downhill, but how the slope is changing — whether you're approaching a cliff, a gentle valley, or a flat plateau.

Newton's method uses the full Hessian to take "perfect" steps that account for curvature. Near the minimum, it converges quadratically (extremely fast). The problem: the Hessian for a model with n parameters is an n × n matrix. For a network with 10 million parameters, that's 100 trillion entries. Not happening.

L-BFGS (Limited-memory Broyden-Fletcher-Goldfarb-Shanno) approximates the Hessian using the last k gradient updates, without ever storing the full matrix. It works brilliantly for small-to-medium models. Scikit-learn's logistic regression uses L-BFGS as its default solver. For deep learning, L-BFGS is impractical because it doesn't play well with stochastic mini-batches.

Bottom line: if you're training a model with fewer than ~100K parameters and using full-batch gradients, L-BFGS will probably converge faster than Adam. For anything larger or mini-batch-based, stick with first-order methods.

Putting It All Together: A Practical Recipe

import torch
import torch.nn as nn
import torch.optim as optim
from torch.optim.lr_scheduler import CosineAnnealingLR, LinearLR, SequentialLR

# 1. Define your model
model = YourModel()

# 2. Pick AdamW (the default for almost everything)
optimizer = optim.AdamW(
    model.parameters(),
    lr=3e-4,           # the Karpathy constant
    weight_decay=0.01, # mild regularization
    betas=(0.9, 0.999) # default momentum terms
)

# 3. Learning rate schedule: warmup + cosine decay
total_steps = len(train_loader) * num_epochs
warmup_steps = int(0.05 * total_steps)
warmup = LinearLR(optimizer, start_factor=0.01, total_iters=warmup_steps)
cosine = CosineAnnealingLR(optimizer, T_max=total_steps - warmup_steps, eta_min=1e-6)
scheduler = SequentialLR(optimizer, [warmup, cosine], milestones=[warmup_steps])

# 4. Training loop with gradient clipping
for epoch in range(num_epochs):
    for batch_X, batch_y in train_loader:
        optimizer.zero_grad()
        loss = criterion(model(batch_X), batch_y)
        loss.backward()                                      # backprop computes gradients
        torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)  # clip
        optimizer.step()                                     # gradient descent takes a step
        scheduler.step()                                     # update learning rate

That's the entire optimization stack for modern deep learning: AdamW + warmup + cosine decay + gradient clipping. It's what GPT, BERT, Vision Transformers, and most serious training runs use. Understand each piece, and you can read any training codebase.
