Evaluation Metrics

Chapter 4: ML Fundamentals & Core Concepts Section 6 of 8 8 subtopics
TL;DR

A model that predicts "not fraud" every single time scores 99% accuracy when only 1% of transactions are fraudulent — and catches zero fraud. Accuracy lies in imbalanced settings. Precision tells you how many of your alarms are real. Recall tells you how many real events you caught. F1 balances the two. ROC-AUC measures ranking quality across all thresholds. For regression, RMSE penalizes big misses; MAE treats all errors equally. Pick the metric that hurts when it's wrong — that's the right one for your problem.

The Problem: Why "Accuracy" Can Lie to You

Imagine you work at a bank. Your job is to build a model that flags fraudulent credit card transactions. You pull a dataset: 100,000 transactions, and exactly 1,000 of them (1%) are fraud. The rest are legitimate.

You train a model. It achieves 99% accuracy. Your manager is thrilled. Ship it.

But here's the thing. A model that prints return "not fraud" for every single transaction — no machine learning at all — also gets 99% accuracy. It correctly classifies all 99,000 legitimate transactions. It misses every single fraud case. Zero frauds caught. 99% accurate. Completely useless.

This is why evaluation metrics matter. The number you optimize tells your model what "good" means. Pick the wrong number, and your model optimizes for the wrong thing. In fraud detection, missing a real fraud (a false negative) costs the bank thousands of dollars. Flagging a legitimate purchase (a false positive) annoys a customer but costs much less. These asymmetries are everywhere in real systems, and a single accuracy number hides them.

Classification Metrics

The Confusion Matrix: Where Everything Starts

A confusion matrix is a 2×2 table (for binary classification) that counts four things: how many times your model was right about each class, and how many times it was wrong about each class.

Predicted: Not FraudPredicted: Fraud
Actual: Not FraudTrue Negative (TN)False Positive (FP)
Actual: FraudFalse Negative (FN)True Positive (TP)

True Positive (TP) — the model said fraud, and it was fraud. Correct alarm. True Negative (TN) — the model said not fraud, and it was not fraud. Correct silence. False Positive (FP) — the model said fraud, but it was actually legitimate. False alarm. False Negative (FN) — the model said not fraud, but it was actually fraud. Missed catch.

Every classification metric you'll ever see is some combination of these four numbers. Here's how to compute it with sklearn and a concrete fraud scenario:

import numpy as np
from sklearn.metrics import confusion_matrix, classification_report

# Simulate: 10,000 transactions, 1% fraud
np.random.seed(42)
y_true = np.array([1]*100 + [0]*9900)  # 100 frauds, 9900 legit

# A decent model's predictions (not perfect)
y_pred = np.zeros(10000, dtype=int)
y_pred[:85] = 1    # catches 85 of 100 frauds
y_pred[100:150] = 1  # but also flags 50 legit ones as fraud

cm = confusion_matrix(y_true, y_pred)
print("Confusion Matrix:")
print(cm)
# [[9850,  50],
#  [  15,  85]]

print("
Classification Report:")
print(classification_report(y_true, y_pred, target_names=["Legit", "Fraud"]))

Accuracy — And When to Ignore It

Accuracy is the fraction of predictions that are correct: (TP + TN) / (TP + TN + FP + FN). In our example, that's (85 + 9850) / 10000 = 99.35%. Sounds great. But remember: predicting all-legit gives 99.0%. Our model improved from 99.0% to 99.35% — a 0.35 percentage point gain that hides the fact that it catches 85% of fraud. Accuracy is fine when classes are balanced (roughly 50/50). When they're imbalanced, it's nearly meaningless.

Precision — "When You Say Fraud, Are You Right?"

Precision answers: of all the transactions my model flagged as fraud, how many were actually fraud? The formula: TP / (TP + FP). In our example: 85 / (85 + 50) = 63.0%. That means about 4 out of every 10 alarms are false. If a human investigator reviews every flagged transaction, they'll waste time on false alarms when precision is low.

Recall — "Of All Real Frauds, How Many Did You Catch?"

Recall (also called sensitivity or true positive rate) answers: of all the actual fraud cases, how many did my model find? The formula: TP / (TP + FN). In our example: 85 / (85 + 15) = 85.0%. We're missing 15% of real fraud. Each missed fraud is potentially thousands of dollars lost.

Specificity — The Other Side of Recall

Specificity (also called true negative rate) asks: of all legitimate transactions, how many did the model correctly leave alone? The formula: TN / (TN + FP). In our example: 9850 / (9850 + 50) = 99.5%. High specificity means few false alarms on the negative class.

The Precision-Recall Tradeoff

Here's the fundamental tension. You can always increase recall by lowering your threshold — flag more transactions as suspicious, and you'll catch more fraud. But you'll also flag more legitimate ones, which drops precision. Push the other way — only flag transactions you're very confident about — and precision goes up, but you miss more fraud.

Think of it like a smoke detector. You can make it extremely sensitive — it'll catch every fire, but it'll also go off when you make toast. Or you can make it less sensitive — fewer false alarms, but it might miss a real fire. There's no free lunch here. The right tradeoff depends on what's more expensive: missing a real event, or raising a false alarm.

In fraud detection, missing fraud is usually far more costly than investigating a false alarm. So you'd bias toward higher recall, accepting lower precision. In email spam filtering, flagging a legitimate email as spam (false positive) might cause someone to miss an important message. There you'd bias toward higher precision.

F1 Score — Balancing Precision and Recall

The F1 score is the harmonic mean of precision and recall: 2 × (precision × recall) / (precision + recall). Why harmonic mean instead of a regular average? Because the harmonic mean punishes extreme imbalance. If precision is 100% and recall is 1%, the arithmetic mean is 50.5% — sounds okay. The harmonic mean is 1.98% — which correctly signals that something is very wrong.

In our example: F1 = 2 × (0.63 × 0.85) / (0.63 + 0.85) = 0.723. A single number that captures both sides of the tradeoff.

F-beta Score — Weighting the Tradeoff

The F-beta score generalizes F1 by letting you control how much more you care about recall versus precision. The formula: F_β = (1 + β²) × (precision × recall) / (β² × precision + recall). When β = 1, you get F1. When β = 2 (F2 score), recall is weighted twice as much as precision — use this for fraud detection where missing fraud is worse. When β = 0.5 (F0.5 score), precision is weighted more — use this for spam filtering where false positives hurt more.

ROC Curve and AUC

Most classifiers don't output a hard yes/no. They output a probability score — "this transaction has a 73% chance of being fraud." You then pick a threshold (a cutoff value) to convert that probability into a decision. If the score is above 0.5, call it fraud. But why 0.5? Why not 0.3 or 0.7?

The ROC curve (Receiver Operating Characteristic) shows you what happens at every possible threshold. It plots the true positive rate (recall) on the y-axis against the false positive rate (1 − specificity) on the x-axis. A perfect model hugs the top-left corner. A random coin-flip model traces the diagonal.

AUC-ROC (Area Under the ROC Curve) collapses that whole curve into a single number between 0 and 1. AUC = 1.0 means perfect separation. AUC = 0.5 means random guessing. AUC = 0.85 means that if you pick a random fraud case and a random legit case, there's an 85% chance the model assigns a higher score to the fraud case.

from sklearn.metrics import roc_curve, roc_auc_score
import matplotlib.pyplot as plt

# Simulated probability scores (not hard labels)
np.random.seed(42)
y_true = np.array([1]*100 + [0]*9900)
# Fraud cases get higher scores on average
scores_fraud = np.random.beta(5, 2, size=100)      # skewed high
scores_legit = np.random.beta(1.5, 5, size=9900)   # skewed low
y_scores = np.concatenate([scores_fraud, scores_legit])

fpr, tpr, thresholds = roc_curve(y_true, y_scores)
auc = roc_auc_score(y_true, y_scores)

plt.figure(figsize=(7, 5))
plt.plot(fpr, tpr, color='#6c3fba', lw=2, label=f'ROC curve (AUC = {auc:.3f})')
plt.plot([0, 1], [0, 1], 'k--', lw=1, label='Random (AUC = 0.5)')
plt.xlabel('False Positive Rate')
plt.ylabel('True Positive Rate (Recall)')
plt.title('ROC Curve — Fraud Detection Model')
plt.legend(loc='lower right')
plt.grid(True, alpha=0.3)
plt.tight_layout()
plt.savefig('roc_curve.png', dpi=150)
plt.show()
⚠️ AUC-ROC Can Be Misleading on Imbalanced Data

When 99% of your data is one class, the false positive rate stays low even with many false positives in absolute terms. A model with 500 false positives out of 99,000 negatives has an FPR of only 0.5% — looks great on the ROC curve. But 500 false alarms might overwhelm your investigation team. For imbalanced problems, the Precision-Recall curve is often more informative.

Precision-Recall Curve and Average Precision

The Precision-Recall (PR) curve plots precision (y-axis) against recall (x-axis) at every threshold. Unlike the ROC curve, it focuses entirely on the positive class (fraud), which is exactly what matters in imbalanced settings. A perfect model hits the top-right corner (precision = 1, recall = 1). A random model's baseline is the fraction of positives in the data — in our case, 1%.

Average Precision (AP) summarizes the PR curve as a single number — the weighted mean of precision at each threshold, with the increase in recall as the weight. It's closely related to the area under the PR curve. AP is the go-to summary metric for imbalanced classification.

Log Loss — Punishing Confident Mistakes

Log loss (also called cross-entropy loss) doesn't care about your hard yes/no predictions. It evaluates the probability scores directly. The formula: -1/N × Σ[y_i × log(p_i) + (1 - y_i) × log(1 - p_i)]. If a fraud case gets a predicted probability of 0.99, log loss barely penalizes you. If that same fraud case gets 0.01 — meaning you were 99% confident it was legit, and you were dead wrong — log loss penalizes you severely. Log loss rewards well-calibrated confidence, not bold predictions.

Cohen's Kappa

Cohen's Kappa measures how much better your model is than random chance, adjusted for the class distribution. It ranges from −1 to 1: 0 means your model is doing no better than random, 1 means perfect agreement, and negative means worse than random. It's especially useful when class sizes are very unequal, because it accounts for the agreement you'd expect by chance. Kappa above 0.8 is generally considered strong agreement.

Matthews Correlation Coefficient (MCC)

The Matthews Correlation Coefficient is often considered the most balanced single-number metric for binary classification. It uses all four cells of the confusion matrix: MCC = (TP×TN − FP×FN) / √((TP+FP)(TP+FN)(TN+FP)(TN+FN)). It ranges from −1 (perfect inverse prediction) to +1 (perfect prediction), with 0 meaning no better than random. Unlike F1, MCC treats both classes symmetrically. A high MCC means the model does well on both positives and negatives, which makes it harder to game.

Multi-Class Metrics

Everything above was binary — fraud or not fraud. Real-world classification often has more than two classes: is this image a cat, dog, or bird? Is this news article about sports, politics, tech, or entertainment?

The Multi-Class Confusion Matrix

With K classes, the confusion matrix becomes a K×K grid. Each row represents the actual class, each column represents the predicted class. The diagonal shows correct predictions. Off-diagonal cells show specific types of mistakes — for example, you might discover your model confuses cats with dogs far more than cats with birds.

Macro, Micro, and Weighted Averaging

You have precision, recall, and F1 for each class individually. But you want one number to summarize performance across all classes. Three ways to do it:

Macro averaging computes the metric for each class, then takes the unweighted average. Every class counts equally, regardless of how many samples it has. If you have 10,000 cats and 50 birds, the bird class matters as much as the cat class. Use this when every class is equally important to you.

Micro averaging pools all true positives, false positives, and false negatives across classes, then computes the metric from the totals. Classes with more samples dominate. For multi-class problems (where every sample gets exactly one label), micro F1 equals accuracy.

Weighted averaging computes the metric per class, then averages weighted by the number of samples in each class. It's a compromise: large classes matter more, but small classes still contribute.

One-vs-All and One-vs-One

One-vs-All (OvA) turns a K-class problem into K binary problems. For each class, you ask: "is this sample in class k, or not?" You train K separate classifiers and pick the most confident one. One-vs-One (OvO) trains a binary classifier for every pair of classes — that's K(K−1)/2 classifiers. Each pair votes, and the class with the most votes wins. OvO is more expensive but can work better when decision boundaries between specific pairs are important.

Top-K Accuracy

Top-K accuracy asks: was the correct class in the model's top K predictions? This makes sense when there are many classes — ImageNet has 1,000. Top-1 accuracy might be 75%, but top-5 accuracy could be 93%, meaning the right answer was almost always in the model's top 5 guesses. Useful for systems where a human makes the final pick from a shortlist.

Regression Metrics

Classification asks "which category?" Regression asks "how much?" — predicting house prices, temperatures, stock returns. The metrics here measure the distance between predicted values and actual values.

Mean Absolute Error (MAE)

MAE is the average of the absolute differences between predictions and actual values: MAE = (1/N) × Σ|y_i − ŷ_i|. If you predict a house is worth $300K and it's actually $320K, the error is $20K. MAE treats all errors linearly — a $10K error is twice as bad as a $5K error. It's easy to interpret: "on average, our predictions are off by $X."

Mean Squared Error (MSE) and Root MSE

MSE squares each error before averaging: MSE = (1/N) × Σ(y_i − ŷ_i)². Squaring does two things: it makes all errors positive, and it punishes large errors much more than small ones. A $100K error contributes 100× more to MSE than a $10K error. RMSE is the square root of MSE, bringing it back to the same units as the target: RMSE = √MSE. RMSE is easier to interpret than MSE because it's in dollars (or whatever your unit is), not dollars-squared.

When should you care about large errors more? When a $100K pricing mistake is not "ten times worse" than a $10K mistake, but catastrophically worse — think insurance risk or structural engineering. Use MSE/RMSE. When all errors should be penalized equally, use MAE.

R-squared (Coefficient of Determination)

R² measures how much of the variance in your target variable the model explains. The formula: R² = 1 − (SS_res / SS_tot), where SS_res is the sum of squared errors and SS_tot is the total variance. R² = 1.0 means your model explains all the variance (perfect predictions). R² = 0.0 means your model is no better than predicting the mean every time. R² can even be negative — that means your model is worse than predicting the mean.

Adjusted R² penalizes you for adding more features. Plain R² always goes up (or stays the same) when you add a feature, even a useless one. Adjusted R² goes down if the added feature doesn't improve the model enough to justify its complexity. Use adjusted R² when comparing models with different numbers of features.

MAPE and Median Absolute Error

Mean Absolute Percentage Error (MAPE) expresses errors as percentages: MAPE = (1/N) × Σ|y_i − ŷ_i| / |y_i| × 100. "Our predictions are off by 5% on average" is easier for business stakeholders to grasp than "our MAE is $15,000." The catch: MAPE explodes when actual values are near zero, and it penalizes over-predictions and under-predictions asymmetrically.

Median Absolute Error is the median of all absolute errors instead of the mean. It's robust to outliers — a single wildly wrong prediction doesn't move it much. Use it when your data has outliers you can't remove.

Choosing the Right Metric

Start with the Business Problem

Don't pick a metric because it's popular. Pick it because it captures what actually hurts when your model is wrong. Ask these questions:

What's the cost of a false positive versus a false negative? In fraud detection, false negatives (missed fraud) cost thousands. In criminal justice, false positives (wrongly convicted) destroy lives. In medical screening, false negatives (missed cancer) can be fatal. The answer to this question alone often determines your metric.

Are your classes imbalanced? If yes, throw accuracy away. Use precision, recall, F1, or AUC. If the imbalance is extreme (1:1000 or worse), PR-AUC is more informative than ROC-AUC.

Cost-Sensitive Evaluation

Sometimes you can put actual dollar amounts on each type of error. Say a missed fraud costs $5,000 on average, and investigating a false alarm costs $50. Your total cost is: 5000 × FN + 50 × FP. Now you can directly optimize for dollars lost instead of abstract statistical quantities. This is cost-sensitive evaluation, and it often changes which model you'd pick.

Threshold Selection

Most classifiers produce a probability. You choose a threshold to turn that probability into a decision. The default 0.5 is arbitrary. In fraud detection, you might lower it to 0.3 — flag anything with a 30%+ fraud probability. You catch more fraud (higher recall) at the cost of more false alarms (lower precision). Plot precision and recall against threshold, find the sweet spot that minimizes your business cost, and use that.

from sklearn.metrics import precision_recall_curve

# Using the same y_true and y_scores from the ROC example
precisions, recalls, thresholds = precision_recall_curve(y_true, y_scores)

# Find threshold where recall >= 0.90
idx = np.where(recalls >= 0.90)[0][-1]
chosen_threshold = thresholds[idx]
print(f"To get 90% recall, set threshold = {chosen_threshold:.3f}")
print(f"At this threshold, precision = {precisions[idx]:.3f}")

# Apply threshold to make predictions
y_pred_custom = (y_scores >= chosen_threshold).astype(int)
print(classification_report(y_true, y_pred_custom, target_names=["Legit", "Fraud"]))

Custom Metrics

Standard metrics don't always fit. Maybe you care about precision only above a recall of 80%. Maybe errors in the $1M+ range should be penalized 10× more. In sklearn, you can write custom scorers with make_scorer and use them in cross-validation. In production, define your metric in code and track it alongside standard ones. The best metric is the one closest to the thing your business actually cares about.

Ranking Metrics

Some tasks aren't about classifying or predicting a number — they're about ordering. Search engines rank web pages. Recommendation systems rank movies. Ad systems rank ads. The question isn't "is this result relevant?" but "did the best results appear near the top?"

Mean Reciprocal Rank (MRR)

MRR looks at where the first relevant result appears. If the first correct result is at position 1, the reciprocal rank is 1/1 = 1. If it's at position 3, it's 1/3. MRR averages this across all queries. It only cares about the first correct hit — good for navigational queries like "take me to the Facebook login page" where there's one right answer.

Mean Average Precision (MAP)

MAP considers all relevant results, not the first one. For each query, compute precision at every position where a relevant result appears, average those precisions (that's Average Precision for one query), then average across all queries. MAP rewards putting relevant results early and penalizes relevant results buried deep in the ranking.

Normalized Discounted Cumulative Gain (NDCG)

NDCG handles graded relevance — not "relevant or not," but "how relevant on a scale." A result with relevance 3 is better than relevance 1. NDCG computes a gain from each result, discounts it by position (logarithmically — position 1 counts most, position 10 counts much less), sums the discounted gains, and normalizes by the ideal ordering. NDCG@10 looks at the top 10 results. It ranges from 0 to 1, where 1 means the results are in perfect order.

Offline vs Online Metrics (and When They Disagree)

Offline metrics are everything we've covered so far — precision, recall, AUC, RMSE — measured on a held-out test set before deploying the model. Online metrics are what happens in the real world after deployment — click-through rate, conversion rate, revenue, user retention, time on site.

They disagree more often than you'd expect. A recommendation model might have great offline NDCG but lower click-through rates because it keeps recommending things the user already knows about. An ad model might have excellent offline AUC but lower revenue because it's great at predicting clicks on cheap ads, not expensive ones.

The gap happens because offline datasets are snapshots of the past. They don't capture how users behave when the model changes what they see. This feedback loop — model changes environment, environment changes data — is invisible in offline evaluation. The fix: use offline metrics to shortlist candidates, then run A/B tests (online experiments) to make the final decision. Trust online metrics when they conflict with offline ones.

Balanced Accuracy

Balanced accuracy is the average of recall computed for each class: (recall_class_0 + recall_class_1) / 2. In our fraud example, recall for class 0 (legit) is specificity = 99.5%, recall for class 1 (fraud) is sensitivity = 85%. Balanced accuracy = (99.5 + 85) / 2 = 92.25%. Compare that to regular accuracy of 99.35%. Balanced accuracy gives a more honest picture when classes are unequal because it weights each class equally, regardless of size. For multi-class problems, it extends to the average of per-class recalls.

Calibration

What Is Calibration?

Here's a question that sounds strange but matters enormously: when your model says "80% probability of fraud," is the transaction actually fraudulent about 80% of the time? If yes, your model is well-calibrated. If your model says 80% but the true frequency is only 40%, your model is overconfident. If it says 80% but the true frequency is 95%, it's underconfident.

Why does this matter? Because downstream systems use those probabilities. If an insurance company uses your model's probability to price a policy, overconfident probabilities lead to underpricing risk. If a medical system decides whether to order a biopsy based on cancer probability, bad calibration means bad clinical decisions.

Reliability Diagrams

A reliability diagram (also called a calibration curve) is the visual test. You bin predictions by their predicted probability (0-10%, 10-20%, …, 90-100%), then for each bin, plot the average predicted probability (x-axis) against the actual fraction of positives (y-axis). A perfectly calibrated model traces the diagonal — predicted 30% means 30% true, predicted 70% means 70% true. Points above the diagonal mean the model is underconfident. Points below mean overconfident.

from sklearn.calibration import calibration_curve
import matplotlib.pyplot as plt

# Using y_true and y_scores from earlier
prob_true, prob_pred = calibration_curve(y_true, y_scores, n_bins=10)

plt.figure(figsize=(7, 5))
plt.plot(prob_pred, prob_true, 's-', color='#6c3fba', label='Model')
plt.plot([0, 1], [0, 1], 'k--', label='Perfectly calibrated')
plt.xlabel('Mean predicted probability')
plt.ylabel('Fraction of positives')
plt.title('Reliability Diagram — Fraud Model')
plt.legend()
plt.grid(True, alpha=0.3)
plt.tight_layout()
plt.savefig('calibration_curve.png', dpi=150)
plt.show()

Brier Score

The Brier score quantifies calibration as a single number: BS = (1/N) × Σ(p_i − y_i)², where p_i is the predicted probability and y_i is the actual outcome (0 or 1). It ranges from 0 (perfect) to 1 (worst). It's the mean squared error of probability estimates. Lower is better. The Brier score captures both calibration (are probabilities honest?) and discrimination (can the model tell classes apart?). A model can have great AUC (good discrimination) but terrible Brier score (bad calibration).

Fixing Calibration: Platt Scaling and Isotonic Regression

What if your model is accurate but poorly calibrated? You can fix the calibration after training using post-hoc calibration methods.

Platt scaling fits a logistic regression on top of your model's raw scores, using a held-out validation set. It learns two parameters (slope and intercept) that transform raw scores into calibrated probabilities. It works well when the calibration curve is roughly sigmoid-shaped (S-curved).

Isotonic regression is more flexible. It fits a non-decreasing step function to the data — no assumption about the shape of the miscalibration. It can fix any monotonic distortion. The downside: it needs more data to fit reliably, and it can overfit on small datasets.

from sklearn.calibration import CalibratedClassifierCV
from sklearn.ensemble import RandomForestClassifier
from sklearn.model_selection import train_test_split

# Train a model and calibrate it
X = np.random.randn(10000, 10)
y = (X[:, 0] + X[:, 1] + np.random.randn(10000)*0.5 > 1).astype(int)

X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.3)

# Uncalibrated model
rf = RandomForestClassifier(n_estimators=100, random_state=42)
rf.fit(X_train, y_train)

# Calibrated with Platt scaling (sigmoid)
rf_platt = CalibratedClassifierCV(rf, method='sigmoid', cv=5)
rf_platt.fit(X_train, y_train)

# Calibrated with Isotonic regression
rf_iso = CalibratedClassifierCV(rf, method='isotonic', cv=5)
rf_iso.fit(X_train, y_train)

# Compare Brier scores
from sklearn.metrics import brier_score_loss
for name, model in [("Uncalibrated", rf), ("Platt", rf_platt), ("Isotonic", rf_iso)]:
    probs = model.predict_proba(X_test)[:, 1]
    bs = brier_score_loss(y_test, probs)
    print(f"{name:15s} Brier Score: {bs:.4f}")
💡 Practical Tip

Random forests tend to push probabilities toward 0.5 (underconfident). Logistic regression is often naturally well-calibrated. Gradient boosting and neural networks are frequently overconfident. Always plot a reliability diagram before trusting your model's probability outputs in production.

Putting It All Together

Here's a decision framework. When you get a new problem, ask yourself these questions in order:

Is it classification or regression? For classification, start with precision, recall, F1. For regression, start with RMSE or MAE. For ranking, start with NDCG or MAP.

Are classes imbalanced? If yes, don't use accuracy. Use F1, PR-AUC, or MCC. Consider balanced accuracy as a quick sanity check.

What's the cost asymmetry? If false negatives are more costly (fraud, disease), optimize for recall or F2. If false positives are more costly (spam filter, criminal conviction), optimize for precision or F0.5. If costs are symmetric, use F1 or MCC.

Do you need calibrated probabilities? If downstream decisions depend on the actual probability value (not the ranking), check calibration with a reliability diagram and Brier score. Apply Platt scaling or isotonic regression if needed.

Will this go to production? Measure offline metrics first. Then run A/B tests to measure online metrics. Trust the online numbers when they disagree.
