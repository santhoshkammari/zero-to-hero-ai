TL;DR

Optimization makes model learn — loss defines what "wrong" means, gradient descent finds parameter values that minimize it. Learning rate matters most. SGD noisy, but noise helps. Momentum smooths chaos. Adam adapts learning rates per-parameter; practical default for almost everything. AdamW fixes Adam weight decay bug. Learning rate schedules (warmup + cosine decay) and gradient clipping complete modern recipe. Backpropagation powers all of it by making gradient computation feasible.

Every call to optimizer.step() runs full optimization algorithm. No intuition, no chance when training fails to converge. This section builds intuition from ground up.

Optimization in machine learning finds best parameter values for model. Formalized before ML in operations research and numerical analysis. Modern deep learning mix — stochastic gradient descent, momentum, adaptive learning rates — came together around 2012–2015. Today, engine behind neural networks, language models, recommendation systems learning from data.

What Does "Wrong" Mean? The Loss Function

Imagine model predicting house prices. One input — square footage. One output — price. Model simple: multiply square footage by w (weight), add b (bias), get prediction.

predicted_price = w * square_footage + b

w and b random now. Model says 1,500 sqft house costs $47. Wrong. Question: how wrong? Need number capturing wrongness. That number is loss — single value measuring distance between predictions and reality. Common choice: mean squared error. Take difference between each prediction and actual price, square it (negative and positive errors no longer cancel), average across houses.

# Our tiny dataset: 3 houses
actual_prices  = [300_000, 450_000, 200_000]
predicted      = [47, 47, 47]       # random initial predictions

# Mean squared error
errors_squared = [(300000-47)**2, (450000-47)**2, (200000-47)**2]
loss = sum(errors_squared) / 3      # a very large number

Loss enormous. Good. Model knows nothing yet. Optimization goal: find w and b values making this number as small as possible. That's whole game. Every optimizer, training loop, convergence curve chases smaller loss.

The Foggy Landscape

Mental image that makes this click: loss as physical terrain. Every w, b combination is point on vast landscape. Height at that point is loss. High ground means bad predictions. Valleys mean good ones. Deepest valley is best model can do.

Now imagine standing on this landscape in thick fog. Can't see valleys. Can't see horizon. Can feel slope under boots — ground tilts left and slightly down. That's all you get.

So take step downhill. Feel slope again. Take another step. Keep going. Eventually reach valley. Not deepest valley on whole landscape, but valley. Loss drops. Model improves.

That's gradient descent. Whole idea. Keep this foggy landscape in mind, because terrain gets weird — saddle points, narrow ravines, flat plateaus. Core rule stays same: feel slope, step downhill.

Feeling the Slope: The Gradient

Slope under your feet has technical name: gradient. Vector — one number per parameter — telling you steepest uphill direction. For house-price model with two parameters (w and b), gradient has two components: how loss changes when you nudge w, and how loss changes when you nudge b.

Those individual numbers are partial derivatives. Partial derivative of loss with respect to w answers one question: "If I increase w a tiny amount, does loss go up or down, and by how much?" If answer is "up by lot," decrease w. If "down by little," increase it gently.

Gradient bundles partial derivatives into one uphill arrow. We want downhill, so we move opposite way — negative gradient.

# For our 2-parameter model:
# gradient = [∂L/∂w, ∂L/∂b]
#
# ∂L/∂w answers: "how does the loss change when I wiggle w?"
# ∂L/∂b answers: "how does the loss change when I wiggle b?"
#
# We step in the OPPOSITE direction:
# w_new = w - step_size * ∂L/∂w
# b_new = b - step_size * ∂L/∂b

Partial derivative notation looks scarier than concept. Concept simple: wiggle one knob, see what happens to loss. That's partial derivative.

Taking a Step: The Update Rule

With gradient in hand, update is almost boring:

θ = θ - α * ∇L(θ)

# θ = all parameters (w and b in our case)
# α = learning rate (how big a step)
# ∇L(θ) = gradient of loss with respect to θ (the slope)

That single line is gradient descent. Every optimizer — SGD, Adam, AdaGrad, all of them — is variation on this theme. They compute some version of "which direction is downhill" and take some version of "step that way." Difference is how they estimate direction and choose step size.

Watch it work on house-price model. Simplify to one parameter — weight w — and use toy loss where best value is w = 150.

import numpy as np

# Toy: loss = (w - 150)^2, minimum at w = 150
def loss(w):
    return (w - 150) ** 2

def gradient(w):
    return 2 * (w - 150)

w = 0.0          # start with a terrible guess
alpha = 0.1      # learning rate

for step in range(8):
    grad = gradient(w)
    w = w - alpha * grad
    print(f"Step {step}: w = {w:.1f}, loss = {loss(w):.0f}")

# Step 0: w = 30.0,  loss = 14400
# Step 1: w = 54.0,  loss = 9216
# Step 2: w = 73.2,  loss = 5898
# Step 3: w = 88.6,  loss = 3775
# Step 4: w = 100.8, loss = 2416
# ...converging toward 150

Each step, loss drops. Gradient says "you're too low, go up," and we go up. Closer to 150, gradient gets smaller (slope flattens), so steps shrink naturally. Rolling into valley. Slowing down.

How Much Data Per Step? Batch, Stochastic, Mini-batch

Toy example had 3 houses. Gradient was cheap. Real datasets have millions of examples. Computing exact gradient over all of them for every step is like surveying whole mountainside before one step. Accurate. Painfully slow.

This tension — gradient accuracy vs compute cost per step — creates three gradient descent flavors.

Batch gradient descent uses every training example to compute one gradient, then takes one step. Gradient is as accurate as dataset allows. Path downhill is smooth and predictable. But with 10 million examples, one step costs 10 million calculations. Then repeat. Impractical for large datasets.

Stochastic gradient descent (SGD) goes opposite direction. Pick one random training example, compute gradient from that example, take step. Gradient estimate is terrible — one house barely represents whole market. Path downhill looks drunk. But you take N steps per pass through data instead of one.

Surprising part: noise helps. Single-sample gradient is inaccurate, yes, but inaccuracy acts like random jostling. It kicks optimizer out of shallow valleys and past saddle points where clean exact gradient might settle. Noise is feature, not bug. We'll revisit this with sharp and flat minima.

Mini-batch gradient descent is practical middle ground. Grab random batch of B examples (usually 32 to 512), compute gradient from that batch, take step. Estimate is much better than single sample, cost stays manageable, GPUs process batches efficiently. This is what people actually use. When paper or codebase says "SGD," it usually means mini-batch SGD with batch size 32 to 256.

# What "training" actually looks like
for epoch in range(num_epochs):
    for batch_X, batch_y in random_batches(X, y, batch_size=64):
        gradient = compute_gradient(batch_X, batch_y, θ)
        θ = θ - α * gradient

Batch size controls tradeoff. Larger batches give cleaner gradient estimates and better GPU efficiency. Smaller batches add more noise, which often improves generalization. Right answer depends, but 32 to 256 is sweet spot for most tasks.

The Learning Rate: One Number to Rule Them All

If you tune one hyperparameter, tune learning rate. Nothing else close.

Learning rate α controls step size for each update. Most consequential number in training config. Get it wrong, nothing else matters.

Set it too high, overshoot valley. Loss stops decreasing — oscillates wildly or explodes to infinity. Terminal fills with NaN. Set it too low, crawl forward so slowly training takes days instead of hours, and never escape mediocre valley because you lack energy to cross ridge to better one.

# Too high — the optimizer overshoots and diverges
w = 5.0
for step in range(5):
    w = w - 1.1 * (2 * w)  # α = 1.1
    print(f"Step {step}: w = {w:.1f}")
# w: 5.0 → -6.0 → 7.2 → -8.6 → 10.4   (exploding!)

# Too low — barely moves
w = 5.0
for step in range(5):
    w = w - 0.001 * (2 * w)  # α = 0.001
    print(f"Step {step}: w = {w:.4f}")
# w: 5.0 → 4.990 → 4.980 → 4.970 → 4.960  (glacial)

# Sweet spot
w = 5.0
for step in range(5):
    w = w - 0.3 * (2 * w)  # α = 0.3
    print(f"Step {step}: w = {w:.3f}")
# w: 5.0 → 2.0 → 0.8 → 0.32 → 0.128       (converging!)

Useful deep learning folklore: Andrej Karpathy used 3e-4 (0.0003) as default Adam learning rate so often that community started calling it "Karpathy constant." Not magic — empirically strong starting point across surprising range of architectures. For SGD with momentum, common starting point is around 0.1, then decay during training.

💡 The Learning Rate Finder

Practical trick: train one epoch while exponentially increasing learning rate from tiny (1e-7) to huge (10). Plot loss vs learning rate. Loss drops, hits minimum, then explodes. Pick learning rate at steepest descent — about one order of magnitude below explosion point. Takes minutes. Saves hours of guessing.

Momentum: The Heavy Ball

Vanilla SGD has annoying habit. Loss landscape often looks like long valley — steep walls, gentle floor slope. SGD oscillates across steep walls while crawling along valley floor. Energy wasted bouncing sideways instead of rolling forward.

Picture heavy ball rolling downhill. In directions where gravity keeps pulling same way (valley floor), ball builds speed — it accelerates. In directions with back-and-forth pushes (valley walls), pushes cancel and oscillations dampen.

Momentum captures this idea. Instead of stepping only from today's gradient, keep running average — velocity — accumulating past gradients:

v = 0            # velocity, same shape as parameters
β = 0.9          # momentum coefficient

for batch_X, batch_y in batches:
    g = compute_gradient(batch_X, batch_y, θ)
    v = β * v + g           # 90% old velocity + new gradient
    θ = θ - α * v           # step using velocity, not raw gradient

Coefficient β is usually 0.9, so velocity is roughly exponential moving average of last 10 gradients. Consistent gradients build speed. Oscillating gradients cancel. Same physics as low-pass filter — smooth noise, pass signal.

Heavy-ball analogy comes back. It connects momentum, Nesterov, eventually Adam — all variants of "what if ball were smarter?"

Nesterov: Looking Before You Leap

Standard momentum computes gradient at current position, then applies velocity. Nesterov accelerated gradient flips order: first step in direction of accumulated velocity ("lookahead"), then compute gradient at future position.

v = 0
β = 0.9

for batch_X, batch_y in batches:
    θ_lookahead = θ - α * β * v              # peek ahead
    g = compute_gradient(batch_X, batch_y, θ_lookahead)
    v = β * v + g
    θ = θ - α * v

Why this matters: if momentum carries heavy ball past valley floor and up other side, standard momentum notices only after ball is already climbing. Nesterov evaluates gradient where ball is about to be, so it sees uphill sooner and brakes earlier. Small tweak — one line — but convergence tightens, especially for well-behaved (convex) problems. Use it when framework offers it. Free.

Adaptive Learning Rates: AdaGrad, RMSprop, Adam

Key insight: different parameters need different learning rates.

Think about word embeddings. Vector for "the" gets updated on nearly every batch — it appears in almost every sentence. Vector for "defenestrate" might get updated once in thousand batches. Same learning rate for both means either "the" updates too aggressively or "defenestrate" too conservatively. Better plan: turn learning rate down for frequently updated parameters and up for rare ones. Adaptive methods do this automatically.

AdaGrad (2011) tried first. It tracks sum of squared gradients for each parameter. Parameters receiving large gradients accumulate large sum, which divides learning rate down. Parameters with small, infrequent gradients keep higher effective learning rate.

# AdaGrad
cache = 0       # sum of squared gradients, per parameter
ε = 1e-8        # tiny constant to avoid division by zero

for batch_X, batch_y in batches:
    g = compute_gradient(batch_X, batch_y, θ)
    cache = cache + g ** 2
    θ = θ - α * g / (sqrt(cache) + ε)

For sparse problems — NLP, recommender systems — AdaGrad is beautiful. Rare features get amplified. Fatal flaw: cache only grows. Never shrinks. After thousands of steps, accumulated sum gets so large that effective learning rate falls near zero and learning stops. AdaGrad dies on long runs.

RMSprop fixes this with one change: instead of accumulating all past squared gradients, use exponential moving average that gradually forgets old gradients. Learning rate adapts but never decays to zero.

# RMSprop
cache = 0
β = 0.999       # decay rate

for batch_X, batch_y in batches:
    g = compute_gradient(batch_X, batch_y, θ)
    cache = β * cache + (1 - β) * g ** 2    # EMA, forgets old gradients
    θ = θ - α * g / (sqrt(cache) + ε)

RMSprop never got paper publication. Geoffrey Hinton introduced it in 2012 Coursera lecture. Most consequential slide deck in deep learning history. It worked. It became direct Adam ancestor.

Adam (Adaptive Moment Estimation, 2014) combines momentum and RMSprop. It keeps two running averages: first moment m (EMA of gradients, like momentum) and second moment v (EMA of squared gradients, like RMSprop). Together, they give smoothed gradient direction and per-parameter learning rate.

# Adam — the full algorithm
m = 0          # first moment (momentum)
v = 0          # second moment (RMSprop)
β1 = 0.9       # decay rate for first moment
β2 = 0.999     # decay rate for second moment
ε = 1e-8
t = 0          # timestep

for batch_X, batch_y in batches:
    t += 1
    g = compute_gradient(batch_X, batch_y, θ)

    m = β1 * m + (1 - β1) * g          # momentum-style update
    v = β2 * v + (1 - β2) * g ** 2     # RMSprop-style update

    m_hat = m / (1 - β1 ** t)          # bias correction
    v_hat = v / (1 - β2 ** t)          # bias correction

    θ = θ - α * m_hat / (sqrt(v_hat) + ε)

Those bias correction terms (m_hat, v_hat) need one sentence. m and v start at zero, so early steps bias them toward zero. Dividing by (1 - β^t) scales them back to true size. Big correction early. Vanishes as t grows.

Favorite thing about Adam: beyond high-level explanations like this, nobody is completely sure why it works so well across such wide range of problems. Default hyperparameters (β1=0.9, β2=0.999, ε=1e-8, lr=3e-4) work from tiny classifiers to huge language models. Unusual in ML. Most things need tuning. Adam barely does.

AdamW: The One You Should Actually Use

Subtle but important bug appears when Adam meets weight decay, regularization penalizing large parameter values to prevent overfitting. Standard Adam applies weight decay through gradient — add term pushing parameters toward zero. Problem: Adam's adaptive scaling distorts that push. Parameters with large historical gradients get less regularization. Parameters with small historical gradients get more. Regularization becomes inconsistent and unpredictable.

In 2019, Loshchilov and Hutter published "Decoupled Weight Decay Regularization." Fix is elegant: apply weight decay directly to parameters, separate from gradient update, so adaptive scaling never touches it.

# AdamW — decoupled weight decay
# ... (same m, v, bias correction as Adam) ...

# The key difference: weight decay is DECOUPLED
θ = θ - α * m_hat / (sqrt(v_hat) + ε) - α * λ * θ
#                                        ^^^^^^^^^^^
#           applied directly to params, NOT through the gradient

Result: consistent, predictable regularization matching plain SGD weight decay. Empirically, AdamW generalizes better than Adam with L2 regularization across many tasks. Default optimizer in Hugging Face, default in most transformer recipes, and choice you should use unless you have specific reason not to. PyTorch ships it as torch.optim.AdamW.

⚠️ SGD vs. Adam: The Debate That Won't Die

Adam converges faster and needs less tuning. SGD with carefully crafted learning rate schedule sometimes generalizes slightly better on vision tasks (ResNets, for example). In NLP and transformer land, AdamW dominates. Field still lacks clean unified answer. Practical advice: start with AdamW. If you are chasing last 0.3% accuracy on image classification benchmark and have compute budget for schedule tuning, try SGD + momentum + cosine annealing.

Learning Rate Schedules

Fixed learning rate is almost never optimal. Foggy-landscape intuition maps directly: early in training, far from any valley, so you need big steps to explore. Late in training, near valley floor, so you need small careful steps to settle without overshooting. Schedule that starts high and decays matches this.

Step decay is oldest approach: multiply learning rate by factor (usually 0.1) at fixed epochs. "Divide LR by 10 at epochs 30, 60, 90" was standard recipe in older vision papers. Works. Requires manual drop points.

Cosine annealing smoothly decays learning rate along cosine curve from initial value to near zero. No abrupt jumps. No hand-designed schedule. Default in many modern training recipes because it works well and asks little.

Warmup fixes start-of-training problem. Freshly initialized parameters are basically random, so gradients are unreliable. Large steps on unreliable gradients are reckless. Warmup starts with tiny learning rate and linearly increases it over first few hundred or thousand steps, letting model stabilize before full throttle. Mechanism still debated. Empirically essential for transformer training.

Modern recipe: linear warmup for 5–10% of total steps, then cosine decay for rest.

import torch.optim as optim
from torch.optim.lr_scheduler import CosineAnnealingLR, LinearLR, SequentialLR

optimizer = optim.AdamW(model.parameters(), lr=3e-4, weight_decay=0.01)

# Warmup for 1000 steps, then cosine decay for 9000 steps
warmup = LinearLR(optimizer, start_factor=0.01, total_iters=1000)
cosine = CosineAnnealingLR(optimizer, T_max=9000, eta_min=1e-6)
scheduler = SequentialLR(optimizer, [warmup, cosine], milestones=[1000])

One-cycle policy, from Leslie Smith, takes different route: ramp learning rate up to high value, then anneal it below starting value, all inside one training run. Sounds backwards — why increase learning rate mid-training? Empirically, it matches or beats carefully tuned step schedules and removes manual scheduling choices. PyTorch ships it as OneCycleLR.

The Terrain Gets Weird: Saddle Points and Flat Valleys

Back to foggy landscape. Terrain is stranger than it first looked.

For years, people worried gradient descent would get trapped in bad local minima — valleys that are not deepest, but have zero gradient so optimizer has no reason to leave. In high dimensions, this fear is mostly misplaced.

Reason: for critical point (gradient zero) to be true local minimum, loss must curve upward in every direction. In model with million parameters, million directions must all curve up at once. Chance of that happening randomly is tiny. Almost every critical point in high-dimensional loss landscape is saddle point — minimum in some directions, maximum in others, like horse saddle or Pringles chip.

Saddle points do slow training, because gradient near saddle is close to zero, so steps get tiny. But SGD noise gives enough random perturbation to nudge optimizer off saddle and back onto descending path. Another reason to value stochastic-gradient noise — it rescues you from saddle points.

Sharp vs. Flat Minima

Not all valleys are equal. Sharp minimum sits at bottom of narrow, steep-walled crevice. Flat minimum sits in broad, gently sloped basin. Both can achieve low training loss. They behave very differently on new data.

Sharp minimum is fragile. Tiny parameter changes cause large loss changes. When training data and real-world data differ even slightly — and they always do — model in sharp minimum collapses. Flat minimum is robust. Small perturbations barely change loss, so predictions stay reliable on unseen data.

This explains practical observation: larger batch sizes tend to find sharp minima (worse generalization), smaller batches tend to find flat ones (better generalization). Noise from small batches acts like constant jostling that keeps optimizer out of narrow crevices — it can rest only in wide basins where noise does not bounce it out. Cranking batch size for speed can hurt final model, even with more accurate gradient estimate.

This area is still active research. Sharpness-Aware Minimization (SAM), introduced in 2021, explicitly seeks flat minima by optimizing not lowest loss at one point, but lowest loss in worst-case neighborhood around that point. It shows consistent gains, but roughly doubles compute per step. Theory connecting sharpness to generalization is compelling, not settled — some researchers argue connection is more nuanced than clean story above.

Gradient Clipping: Insurance Against Catastrophe

Sometimes, despite best effort, gradient explodes. One bad batch produces gradient norm of 10,000 when usual norm is around 1. One update with that gradient destroys what model learned. Especially common in recurrent neural networks and early transformer training, where long sequences create multiplicative chains amplifying gradients.

Fix is simple: if gradient's total magnitude exceeds threshold, scale whole gradient down proportionally. Direction stays same. Magnitude gets capped.

# Gradient clipping in PyTorch — one line
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)

# Under the hood:
# 1. Compute total_norm = sqrt(sum(p.grad.norm()**2 for all params))
# 2. If total_norm > max_norm:
#        scale = max_norm / total_norm
#        multiply every gradient by scale

Clip value of 1.0 is safe default. Used in almost every serious training codebase — GPT, BERT, LLaMA, all of them. Think of it as insurance: most steps have fine gradients and clipping does nothing, but rogue gradient triggers it and catastrophe stops. Cost is one norm computation per step. Basically free.

Backpropagation Is Not Gradient Descent

These two get confused constantly. Draw bright line.

Gradient descent answers: "Given gradient of loss with respect to every parameter, how do I update parameters?" It is optimization algorithm — stepping-downhill part.

Backpropagation answers: "How do I compute that gradient efficiently for neural network?" It is calculus trick — chain-rule application propagating error signals backward through layers.

Without backpropagation, you could still do gradient descent. You would compute gradients by finite differences: nudge each parameter by tiny amount ε, measure loss change, divide. For model with 100 million parameters, that means 100 million forward passes per gradient step. Backpropagation computes exact same gradient in one backward pass. That is transformation — O(n) forward passes to O(1). It makes neural network training feasible.

We'll go deep on backpropagation mechanics in deep learning chapters. For now, remember division of labor: gradient descent decides what to do with gradients, backpropagation decides how to compute them.

Second-Order Methods (Brief Detour)

Everything so far was first-order — we used only gradient, first derivative. Second-order methods also use curvature, encoded in matrix called Hessian. Hessian tells you not only downhill direction, but how slope itself changes — cliff, gentle valley, or flat plateau.

Back on foggy landscape: first-order methods feel tilt under feet. Second-order methods also feel ground curvature — whether next step will steepen or flatten. More information. Better steps.

Newton's method uses full Hessian to take "perfect" curvature-aware steps. Near minimum, it converges quadratically — much faster than first-order methods. Problem is Hessian. For model with n parameters, it is n × n matrix. Model with 10 million parameters gives Hessian with 100 trillion entries. Not happening.

L-BFGS (Limited-memory BFGS) approximates Hessian using last k gradient updates without materializing full matrix. It works brilliantly for small-to-medium models and is default solver for scikit-learn logistic regression. For deep learning, L-BFGS is impractical because it handles mini-batch gradient noise poorly.

Practical bottom line: for models under ~100K parameters with full-batch gradients, L-BFGS probably converges faster than Adam. For anything larger or mini-batch-based, stick with first-order methods. Most people never need second-order optimization, but knowing it exists and why it exists closes conceptual gap.

The Full Recipe

Everything built so far — gradient, learning rate, momentum, adaptive rates, schedules, clipping — comes together in training loop that fits on one screen. This is what modern deep learning training looks like:

import torch
import torch.nn as nn
import torch.optim as optim
from torch.optim.lr_scheduler import CosineAnnealingLR, LinearLR, SequentialLR

model = YourModel()

# AdamW: the default for almost everything
optimizer = optim.AdamW(
    model.parameters(),
    lr=3e-4,             # the "Karpathy constant"
    weight_decay=0.01,   # mild, decoupled regularization
    betas=(0.9, 0.999)   # momentum + RMSprop decay rates
)

# Schedule: warmup + cosine decay
total_steps = len(train_loader) * num_epochs
warmup_steps = int(0.05 * total_steps)  # 5% warmup
warmup = LinearLR(optimizer, start_factor=0.01, total_iters=warmup_steps)
cosine = CosineAnnealingLR(optimizer, T_max=total_steps - warmup_steps, eta_min=1e-6)
scheduler = SequentialLR(optimizer, [warmup, cosine], milestones=[warmup_steps])

# Training loop
for epoch in range(num_epochs):
    for batch_X, batch_y in train_loader:
        optimizer.zero_grad()
        loss = criterion(model(batch_X), batch_y)
        loss.backward()                                        # backprop: compute gradients
        torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)  # clip: insurance
        optimizer.step()                                       # gradient descent: update params
        scheduler.step()                                       # schedule: adjust learning rate

That's whole optimization stack. AdamW + warmup + cosine decay + gradient clipping. GPT uses it. BERT uses it. Vision Transformers use it. Understand each piece — and now you do — and you can read any training codebase and know exactly what is happening.
