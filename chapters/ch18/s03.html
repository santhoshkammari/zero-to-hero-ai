Privacy-Preserving ML

Chapter 18: Responsible AI and Ethics 6 subtopics
TL;DR

Mathematical privacy guarantees (differential privacy) beat heuristic anonymization every time. Federated learning keeps data on-device but isn't automatically private — you still need secure aggregation and DP on top. GDPR's "right to be forgotten" creates the machine unlearning problem, arguably the hardest open challenge in privacy engineering today.

In 2006, Netflix released 10 million movie ratings — "anonymized" by stripping names. Researchers cross-referenced it with public IMDb reviews and re-identified users within weeks. If that sounds like ancient history, it isn't — the same class of attack still works on most "anonymized" datasets today. The fix isn't better anonymization. It's mathematical guarantees.

Differential Privacy

The Epsilon-Delta Definition

A mechanism is differentially private if its output barely changes whether your data is in the dataset or not. An adversary looking at the output can't confidently determine if any single individual participated. Here's the formal statement:

P[M(D) ∈ S] ≤ eε · P[M(D') ∈ S] + δ

Where D and D' differ by exactly one record. That's the entire definition. Everything else is engineering around it.

Epsilon (ε) controls how much the output can shift when one person's data changes. ε = 0 means perfect privacy (output is pure noise — useless). ε = ∞ means no privacy at all. Practical deployments live in the range 0.1 to 10. Apple uses ε ≈ 2-8 for emoji stats. The 2020 US Census used ε ≈ 19.6 (controversial — many statisticians argued it was too loose).

Delta (δ) is the probability of catastrophic privacy failure — a "whoops, we leaked everything" event. Set it to something negligible, typically δ << 1/n where n is the dataset size. A common choice: δ = 1/n².

Privacy Degrades with Use — The Budget Concept

Every query against a private dataset eats into your privacy budget. Ask the same database 1000 questions and you've degraded privacy 1000 times. This is the composition theorem, and it's the reason DP is hard in practice.

Basic composition: k queries at ε each → kε total privacy loss. Simple but pessimistic.

Advanced composition: k queries at ε each → roughly ε√(2k · ln(1/δ)) total. Much tighter for large k. The moments accountant (used in DP-SGD) gives the tightest tracking of all — it's why Opacus can train for hundreds of epochs without blowing the budget.

Laplace and Gaussian Mechanisms

Two workhorses. Both add calibrated noise to a query's true answer. The Laplace mechanism satisfies pure (ε, 0)-DP. The Gaussian mechanism gives (ε, δ)-DP but composes tighter across multiple queries — use Gaussian when you're running many operations.

import numpy as np

def laplace_mechanism(true_value, sensitivity, epsilon):
    """Pure (epsilon, 0)-DP. Use for single queries or small composition."""
    scale = sensitivity / epsilon
    noise = np.random.laplace(0, scale)
    return true_value + noise

def gaussian_mechanism(true_value, sensitivity, epsilon, delta):
    """(epsilon, delta)-DP. Composes tighter — prefer for repeated queries."""
    sigma = sensitivity * np.sqrt(2 * np.log(1.25 / delta)) / epsilon
    noise = np.random.normal(0, sigma)
    return true_value + noise

# Example: average income with DP
incomes = np.array([45000, 72000, 51000, 89000, 63000, 55000, 120000, 47000])
true_avg = np.mean(incomes)

# Sensitivity = (max_income - min_income) / n for average query
sensitivity = (150000 - 0) / len(incomes)  # assuming income range [0, 150k]

dp_avg_laplace = laplace_mechanism(true_avg, sensitivity, epsilon=1.0)
dp_avg_gaussian = gaussian_mechanism(true_avg, sensitivity, epsilon=1.0, delta=1e-5)

print(f"True average:     ${true_avg:,.0f}")
print(f"Laplace DP avg:   ${dp_avg_laplace:,.0f}")
print(f"Gaussian DP avg:  ${dp_avg_gaussian:,.0f}")

DP-SGD — Differentially Private Training

Standard SGD computes gradients, averages them, updates the model. DP-SGD adds two steps: clip each per-example gradient to a fixed norm (bounds sensitivity), then add calibrated Gaussian noise to the average gradient. The clipping prevents any single example from dominating the update. The noise makes the update statistically indistinguishable whether that example was included or not.

import torch
import torch.nn as nn

def dp_sgd_step(model, loss_fn, batch_x, batch_y, 
                max_grad_norm=1.0, noise_multiplier=1.1, lr=0.01):
    """Simplified DP-SGD. For production, use Opacus or TF Privacy."""
    per_example_grads = []
    
    # Compute per-example gradients
    for x_i, y_i in zip(batch_x, batch_y):
        model.zero_grad()
        loss = loss_fn(model(x_i.unsqueeze(0)), y_i.unsqueeze(0))
        loss.backward()
        
        # Collect gradients for this example
        grads = [p.grad.clone() for p in model.parameters()]
        per_example_grads.append(grads)
    
    # Step 1: Clip each example's gradient to max_grad_norm
    clipped_grads = []
    for grads in per_example_grads:
        total_norm = torch.sqrt(sum(g.norm() ** 2 for g in grads))
        clip_factor = min(1.0, max_grad_norm / (total_norm + 1e-8))
        clipped_grads.append([g * clip_factor for g in grads])
    
    # Step 2: Average clipped gradients + add Gaussian noise
    batch_size = len(batch_x)
    noise_scale = noise_multiplier * max_grad_norm / batch_size
    
    with torch.no_grad():
        for i, param in enumerate(model.parameters()):
            avg_grad = torch.stack([g[i] for g in clipped_grads]).mean(dim=0)
            noise = torch.normal(0, noise_scale, size=avg_grad.shape)
            param -= lr * (avg_grad + noise)
The privacy-utility tradeoff is real. DP-SGD typically costs 2–10% accuracy compared to standard training. Larger datasets tolerate it better because noise gets averaged over more examples. On CIFAR-10, state-of-the-art DP models at ε=8 reach ~96% accuracy vs ~99% without DP. On small medical datasets, the gap can be devastating. Always benchmark your task before committing to a privacy budget.

For production, don't roll your own — use Opacus (PyTorch) or TF Privacy (TensorFlow). They handle the moments accountant, per-example gradient computation via hooks, and privacy budget tracking automatically.

Local vs Global DP

Global DP: a trusted curator holds the raw data and adds noise to query results. This is what the US Census does. Better utility because noise is added once to the aggregate.

Local DP: each user randomizes their own data before sending it. No trusted curator needed. Apple uses this for emoji and word frequency stats — each iPhone flips bits locally before reporting. The cost? You need vastly more data for the same accuracy. Local DP essentially squares the sample size requirement.

Google's RAPPOR protocol is the canonical local DP system — it uses randomized response with Bloom filters to collect browser statistics without trusting Google's servers.

Federated Learning

FedAvg Algorithm

The core idea: instead of sending data to the model, send the model to the data. Each client trains locally, sends back model updates, and a central server averages them. That's FedAvg — still the most widely deployed algorithm.

import torch
import torch.nn as nn
from copy import deepcopy
from typing import List, Dict

class FederatedServer:
    def __init__(self, global_model: nn.Module):
        self.global_model = global_model
    
    def distribute_model(self) -> Dict[str, torch.Tensor]:
        return deepcopy(self.global_model.state_dict())
    
    def aggregate(self, client_updates: List[Dict[str, torch.Tensor]], 
                  client_sizes: List[int]):
        """Weighted average of client models (FedAvg)."""
        total_size = sum(client_sizes)
        new_state = {}
        
        for key in self.global_model.state_dict():
            new_state[key] = sum(
                client_updates[i][key] * (client_sizes[i] / total_size)
                for i in range(len(client_updates))
            )
        
        self.global_model.load_state_dict(new_state)


class FederatedClient:
    def __init__(self, client_id: str, local_data, local_labels, 
                 local_epochs: int = 5, lr: float = 0.01):
        self.client_id = client_id
        self.data = local_data
        self.labels = local_labels
        self.local_epochs = local_epochs
        self.lr = lr
    
    def train(self, global_state: Dict[str, torch.Tensor], 
              model_fn) -> Dict[str, torch.Tensor]:
        """Train locally and return updated model state."""
        model = model_fn()
        model.load_state_dict(global_state)
        optimizer = torch.optim.SGD(model.parameters(), lr=self.lr)
        loss_fn = nn.CrossEntropyLoss()
        
        model.train()
        for _ in range(self.local_epochs):
            optimizer.zero_grad()
            output = model(self.data)
            loss = loss_fn(output, self.labels)
            loss.backward()
            optimizer.step()
        
        return deepcopy(model.state_dict())


# Training loop
def federated_training(server, clients, model_fn, rounds=50):
    for r in range(rounds):
        global_state = server.distribute_model()
        
        # Each client trains locally (in practice, this happens on-device)
        updates = []
        sizes = []
        for client in clients:
            local_state = client.train(global_state, model_fn)
            updates.append(local_state)
            sizes.append(len(client.data))
        
        server.aggregate(updates, sizes)
        
        if (r + 1) % 10 == 0:
            print(f"Round {r+1}/{rounds} complete — "
                  f"{len(clients)} clients participated")

The Non-IID Problem

Federated data is almost never IID. A Spanish-speaking user's keyboard data looks nothing like a Japanese user's. A rural hospital sees different diseases than an urban one. FedAvg was designed assuming roughly uniform distributions, and it diverges when data is highly heterogeneous.

FedProx is the standard fix: add a proximal term μ/2 · ||w - wglobal||² to each client's loss function, penalizing local models that drift too far from the global model. Think of it as an elastic band keeping clients from wandering off. μ = 0.01 to 1.0 works for most cases.

For extreme non-IID scenarios, personalization approaches (per-client fine-tuning layers, mixture of global + local models) often outperform trying to force a single global model.

Secure Aggregation

Federated learning is NOT automatically private. Model updates (gradients) can leak training data. The gradient inversion attack can reconstruct individual training images from a single gradient update with startling fidelity. Never assume "data stays on device" equals "data is private."

Secure aggregation ensures the server only sees the sum of client updates, not individual ones. Clients use cryptographic masking — each pair of clients generates shared random masks that cancel out in the aggregate. If client A adds mask +r and client B adds mask -r, the server sees the true sum without seeing either individual update. Google deployed this for GBoard in 2017.

For real privacy guarantees, combine FL + secure aggregation + DP noise on the client updates. All three layers.

Data Anonymization

K-Anonymity, L-Diversity, T-Closeness

Technique Guarantee Weakness
K-Anonymity Each record indistinguishable from k-1 others on quasi-identifiers Homogeneity attack — if all k records share the same sensitive value, privacy is gone
L-Diversity At least l distinct sensitive values per equivalence class Skewness attack — if 1 of l values dominates, attacker still learns a lot
T-Closeness Sensitive attribute distribution within each group is within distance t of overall distribution Computationally expensive, hard to tune t 
import pandas as pd

def check_k_anonymity(df: pd.DataFrame, quasi_identifiers: list, k: int) -> dict:
    """Check if a DataFrame satisfies k-anonymity.
    
    Returns dict with status and any violating groups.
    """
    grouped = df.groupby(quasi_identifiers).size().reset_index(name='count')
    violations = grouped[grouped['count'] < k]
    
    return {
        'satisfies_k_anonymity': len(violations) == 0,
        'k': k,
        'total_groups': len(grouped),
        'violating_groups': len(violations),
        'min_group_size': grouped['count'].min(),
        'violations': violations.to_dict('records') if len(violations) > 0 else []
    }

# Example usage
data = pd.DataFrame({
    'age_range': ['20-30', '20-30', '20-30', '30-40', '30-40'],
    'zipcode':   ['100**', '100**', '100**', '200**', '200**'],
    'disease':   ['flu',   'flu',   'cold',  'HIV',   'HIV']
})

result = check_k_anonymity(data, ['age_range', 'zipcode'], k=2)
print(f"Satisfies 2-anonymity: {result['satisfies_k_anonymity']}")
print(f"Min group size: {result['min_group_size']}")
# Note: group (30-40, 200**) has k=2 but BOTH have 'HIV'
# This passes k-anonymity but fails l-diversity — the homogeneity attack.

Synthetic Data

GANs, VAEs, and diffusion models can generate synthetic datasets that preserve statistical properties without containing real records. Combine with DP training (DP-GAN) for formal guarantees. Without DP, synthetic data offers no provable privacy — membership inference and linkage attacks can still succeed.

Think of synthetic data as a convenience, not a guarantee. It's useful for development and testing, but for regulatory compliance you need either DP or a formal privacy analysis.

Privacy Regulations — What ML Engineers Actually Need to Know

GDPR Principles for ML

  • Data minimization: Only collect features you actually need. That 200-column user profile table? Justify every column.
  • Purpose limitation: Data collected for recommendation can't be repurposed for insurance scoring without new consent.
  • Right to erasure (Article 17): Users can demand deletion. If their data influenced a model, you may need to retrain or prove the model doesn't "remember" them.
  • Data protection by design: Privacy isn't an afterthought — it must be built into the system architecture from day one.
  • Right to explanation (Article 22): Automated decisions with significant effects require meaningful explanation. This intersects directly with interpretability (previous section).

Machine Unlearning — The Engineering Challenge

A user requests deletion. You have a model trained on 10 million records including theirs. Retraining from scratch costs $50,000 in compute. Do you really retrain every time someone exercises GDPR Article 17?

SISA training (Sharded, Isolated, Sliced, and Aggregated) is the clever solution: partition data into shards, train a separate sub-model on each shard, ensemble them for predictions. When a user requests deletion, you only retrain the one shard containing their data — not the entire model.

import torch
import torch.nn as nn
from typing import List, Optional
from copy import deepcopy

class SISATrainer:
    """SISA: Sharded, Isolated, Sliced, and Aggregated training.
    
    Enables efficient unlearning by retraining only affected shards.
    """
    def __init__(self, model_fn, num_shards: int = 10, lr: float = 0.01):
        self.model_fn = model_fn
        self.num_shards = num_shards
        self.lr = lr
        self.shard_models: List[nn.Module] = []
        self.shard_data: List[dict] = []  # tracks which data is in which shard
        self.record_to_shard: dict = {}   # record_id -> shard_index
    
    def _assign_shards(self, record_ids: list):
        """Deterministically assign records to shards."""
        for i, rid in enumerate(record_ids):
            shard_idx = i % self.num_shards
            self.record_to_shard[rid] = shard_idx
    
    def train_all_shards(self, data_by_shard: dict, labels_by_shard: dict, 
                         epochs: int = 10):
        """Train each shard independently."""
        self.shard_models = []
        for shard_idx in range(self.num_shards):
            model = self.model_fn()
            optimizer = torch.optim.SGD(model.parameters(), lr=self.lr)
            loss_fn = nn.CrossEntropyLoss()
            
            X = data_by_shard[shard_idx]
            y = labels_by_shard[shard_idx]
            
            model.train()
            for _ in range(epochs):
                optimizer.zero_grad()
                loss = loss_fn(model(X), y)
                loss.backward()
                optimizer.step()
            
            self.shard_models.append(model)
            print(f"Shard {shard_idx}: trained on {len(X)} records")
    
    def predict(self, x: torch.Tensor) -> torch.Tensor:
        """Ensemble prediction: average logits across all shards."""
        logits = []
        for model in self.shard_models:
            model.eval()
            with torch.no_grad():
                logits.append(model(x))
        return torch.stack(logits).mean(dim=0)
    
    def unlearn(self, record_id: str, remaining_data: torch.Tensor, 
                remaining_labels: torch.Tensor, epochs: int = 10):
        """Retrain only the affected shard — O(1/num_shards) cost."""
        shard_idx = self.record_to_shard.pop(record_id)
        
        model = self.model_fn()
        optimizer = torch.optim.SGD(model.parameters(), lr=self.lr)
        loss_fn = nn.CrossEntropyLoss()
        
        model.train()
        for _ in range(epochs):
            optimizer.zero_grad()
            loss = loss_fn(model(remaining_data), remaining_labels)
            loss.backward()
            optimizer.step()
        
        self.shard_models[shard_idx] = model
        print(f"Unlearned record '{record_id}' — retrained shard {shard_idx} only "
              f"(1/{self.num_shards} of total compute)")

Exact unlearning means the model is identical to one that never saw the deleted data. SISA achieves this. Approximate unlearning uses gradient-based methods to "nudge" the model away from the deleted point — faster but harder to verify. Regulators haven't settled on which standard they'll require.

Advanced Privacy-Preserving Techniques

Homomorphic Encryption (HE)

Compute directly on encrypted data — additions and multiplications on ciphertexts produce the correct encrypted result. Fully homomorphic encryption (FHE) supports arbitrary computation but is 1,000–1,000,000× slower than plaintext. Practical today only for simple inference on sensitive data (e.g., encrypted medical records) where latency is acceptable. Libraries: Microsoft SEAL, Zama Concrete-ML.

Secure Multi-Party Computation (SMPC)

Multiple parties jointly compute a function without any party revealing its input. Uses secret sharing — split each value into random shares distributed across parties. Computation happens on shares. The bottleneck is communication overhead — every multiplication requires a round of communication. Works well for 2–10 parties on moderate-sized computations, not for training a 7B parameter model.

Trusted Execution Environments (TEEs)

Hardware enclaves (Intel SGX, ARM TrustZone, AWS Nitro) provide isolated execution — even the OS and hypervisor can't access the data inside. Fast (near-native speed) but you're trusting the hardware vendor. Intel SGX has had multiple side-channel attacks (Foreshadow, Plundervolt, SGAxe). Use TEEs as defense-in-depth, not as your only privacy layer.
