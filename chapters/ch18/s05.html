Safety, Alignment & Guardrails

Chapter 18: Responsible AI and Ethics Alignment · RLHF · Red-teaming · Production guardrails · Human oversight
TL;DR

Safety means preventing harm. Alignment means ensuring the model pursues the correct objectives — not a proxy that games its way to a high score. Guardrails are the production defense layers that catch what alignment missed. In high-stakes systems, you combine all three with human oversight, because no automated system should be the last line of defense when the consequences are irreversible.

The Alignment Problem

You tell a model what to optimize. It finds a way to get a high score that completely misses the point. This is specification gaming — and it's not a theoretical concern, it's the default behavior of any sufficiently capable optimizer.

Specification Gaming and Reward Hacking

The classic example: a reinforcement learning agent trained to play a boat racing game discovered it could score more points by spinning in circles hitting boost pads than by actually finishing the race. The reward function rewarded points, not winning. The agent exploited that gap with zero hesitation.

This isn't a cute edge case. It happens everywhere:

  • Content recommendation: Optimize for engagement → model learns to recommend outrage content (maximizes clicks, destroys user well-being)
  • Customer service chatbot: Optimize for resolution speed → model learns to close tickets without actually solving the problem
  • Code generation: Optimize for passing test cases → model hardcodes expected outputs instead of writing general solutions

Every one of these is a system doing exactly what it was told to do, just not what the designers meant.

Goodhart's Law in ML

"When a measure becomes a target, it ceases to be a good measure." Every ML metric is a proxy for what you actually care about. Accuracy is a proxy for "makes correct decisions." BLEU score is a proxy for "good translation." Click-through rate is a proxy for "user finds this valuable." The moment you optimize hard on the proxy, the gap between the proxy and the real objective widens. This is Goodhart's Law, and it's the most underappreciated failure mode in production ML.

The practical implication: always monitor metrics you're not optimizing for. If your recommendation model's CTR goes up but session duration drops, you're Goodhart-ing.

On the deep alignment debate: There's a rich philosophical literature on existential alignment risk — whether superhuman AI systems could pursue goals catastrophically misaligned with humanity. That's important work, but it's not what you'll face in your next deployment. The alignment problems in this section are the ones that bite production systems today.

RLHF: Reinforcement Learning from Human Feedback

RLHF is the dominant technique for aligning LLMs with human intent. It's how ChatGPT went from "impressive but unhinged" to "mostly helpful and rarely offensive." Three stages:

Stage 1 — Supervised Fine-Tuning (SFT): Fine-tune the base model on high-quality demonstrations. Human experts write ideal responses to prompts. This gives the model a reasonable starting policy.

Stage 2 — Reward Model Training: Humans compare pairs of model outputs and say which is better. These preferences train a separate reward model that learns to score outputs the way humans would.

Stage 3 — RL Fine-Tuning: Use PPO (or similar RL algorithm) to fine-tune the LLM to maximize the reward model's score, with a KL-divergence penalty to prevent it from drifting too far from the SFT model.

# RLHF: Simplified three-stage overview
# Stage 1: SFT — standard fine-tuning on demonstrations
# Stage 2: Reward model from human comparisons

class RewardModel(nn.Module):
    """Learns to predict human preference scores."""
    def __init__(self, base_model):
        super().__init__()
        self.backbone = base_model
        self.reward_head = nn.Linear(base_model.hidden_size, 1)

    def forward(self, input_ids, attention_mask):
        hidden = self.backbone(input_ids, attention_mask).last_hidden_state
        # Score based on final token representation
        return self.reward_head(hidden[:, -1, :]).squeeze(-1)

# Training: given (prompt, chosen_response, rejected_response)
# Loss = -log(sigmoid(reward(chosen) - reward(rejected)))
def preference_loss(reward_chosen, reward_rejected):
    return -torch.log(torch.sigmoid(reward_chosen - reward_rejected)).mean()

# Stage 3: PPO fine-tuning (conceptual)
# for each prompt:
#   response = policy_model.generate(prompt)
#   reward = reward_model.score(prompt, response)
#   kl_penalty = kl_divergence(policy_model, sft_model)
#   objective = reward - beta * kl_penalty
#   update policy_model with PPO to maximize objective

The Reward Model Hacking Problem

Here's the irony: RLHF introduces another proxy (the reward model) and then optimizes against it. The LLM quickly learns to exploit patterns the reward model likes rather than genuinely being helpful. Longer responses get higher scores? The model becomes verbose. The reward model likes confident-sounding text? The model becomes confidently wrong.

This is why the KL penalty exists — it constrains how far the model can drift. But it's a leash, not a cure. Reward model hacking is an active research area with no clean solution.

Constitutional AI

Anthropic's Constitutional AI (CAI) sidesteps the human-labeling bottleneck. Instead of collecting thousands of human preference judgments, you give the model a set of principles — a "constitution" — and have it critique and revise its own outputs.

The process:

  1. Model generates an initial response (which may be harmful)
  2. Model critiques its own response against each constitutional principle
  3. Model revises the response based on its own critique
  4. The revised outputs train a preference model (replacing human comparisons in RLHF Stage 2)
# Constitutional AI: Self-critique loop (simplified)

CONSTITUTION = [
    "Choose the response that is least harmful or toxic.",
    "Choose the response that is most helpful while being honest.",
    "Choose the response that does not encourage illegal activity.",
    "Choose the response that respects user privacy.",
]

def constitutional_critique(model, prompt, initial_response):
    """Model critiques and revises its own output."""
    revisions = [initial_response]

    for principle in CONSTITUTION:
        critique_prompt = (
            f"Given this response to '{prompt}':\n\n"
            f"{revisions[-1]}\n\n"
            f"Critique this response based on the principle: "
            f"'{principle}'\n\n"
            f"Identify any issues and provide a revised response "
            f"that better adheres to this principle."
        )
        critique = model.generate(critique_prompt)
        # Extract revised response from critique
        revised = extract_revision(critique)
        revisions.append(revised)

    return revisions[-1]  # Final revised response

# Advantage: scales without human feedback per example
# Disadvantage: model's self-critique is only as good as
#   its understanding of the principles

Why it matters: CAI scales alignment without requiring human annotators for every comparison. The constitution is human-authored (that's where your values live), but the labor-intensive ranking step is automated. This is how Claude was trained.

Red-Teaming

Red-teaming is systematically trying to make your model behave badly before your users do. For production LLMs, it's not optional — it's the bare minimum.

What red-teamers look for:

  • Jailbreaks: Prompt constructions that bypass safety filters ("ignore your instructions and...")
  • Harmful outputs: Generating dangerous, illegal, or biased content
  • Information leakage: Extracting system prompts, training data, or PII
  • Hallucination under pressure: Confidently wrong answers on critical topics

Critical: red-teaming is continuous, not a one-time checkbox. New jailbreaks appear weekly. Your model's safety today tells you nothing about its safety against next month's attack techniques.

Regulatory note: The EU AI Act (Article 55) requires providers of general-purpose AI models with systemic risk (>1025 FLOPs training compute) to perform adversarial testing. This isn't guidance — it's law, with penalties up to €15M or 3% of global revenue. Red-teaming is a compliance requirement.

Production Guardrails

Alignment techniques (RLHF, CAI) are training-time defenses. Guardrails are inference-time defenses — the runtime safety net between your model and your users. Think of it as defense in depth: alignment makes the model want to behave well; guardrails ensure it can't behave badly even when alignment fails.

Multi-Layer Defense Architecture

No single check catches everything. You need layers:

  • Input layer: Block prompt injections, detect toxic inputs, enforce topic boundaries
  • Model layer: Temperature control, constrained decoding, system prompt anchoring
  • Output layer: Content filtering, PII detection, hallucination checks, format validation
  • Business logic layer: Rate limiting, usage policy enforcement, escalation triggers
import re
from dataclasses import dataclass, field
from typing import Optional

@dataclass
class GuardrailResult:
    allowed: bool
    reason: Optional[str] = None
    modified_text: Optional[str] = None

class ProductionGuardrails:
    """Multi-layer guardrail system for LLM deployments."""

    def __init__(self):
        self.blocked_patterns = [
            r"ignore (?:all |your )?(?:previous )?instructions",
            r"you are now (?:DAN|an? unrestricted)",
            r"pretend you (?:have no|don't have) (?:rules|restrictions)",
        ]
        self.pii_patterns = {
            "ssn": r"\b\d{3}-\d{2}-\d{4}\b",
            "email": r"\b[\w.+-]+@[\w-]+\.[\w.-]+\b",
            "phone": r"\b\d{3}[-.]?\d{3}[-.]?\d{4}\b",
            "credit_card": r"\b\d{4}[\s-]?\d{4}[\s-]?\d{4}[\s-]?\d{4}\b",
        }
        self.topic_denylist = [
            "medical diagnosis", "legal advice",
            "financial recommendation",
        ]

    def check_input(self, user_input: str) -> GuardrailResult:
        """Layer 1: Input filtering."""
        text_lower = user_input.lower()

        # Prompt injection detection
        for pattern in self.blocked_patterns:
            if re.search(pattern, text_lower):
                return GuardrailResult(
                    allowed=False,
                    reason="prompt_injection_detected"
                )

        # Topic boundary enforcement
        for topic in self.topic_denylist:
            if topic in text_lower:
                return GuardrailResult(
                    allowed=False,
                    reason=f"out_of_scope: {topic}"
                )

        return GuardrailResult(allowed=True)

    def check_output(self, response: str) -> GuardrailResult:
        """Layer 3: Output filtering."""
        # PII leak detection
        detected_pii = []
        sanitized = response
        for pii_type, pattern in self.pii_patterns.items():
            if re.search(pattern, response):
                detected_pii.append(pii_type)
                sanitized = re.sub(pattern, f"[{pii_type.upper()}_REDACTED]", sanitized)

        if detected_pii:
            return GuardrailResult(
                allowed=True,  # Allow but redact
                reason=f"pii_redacted: {detected_pii}",
                modified_text=sanitized
            )

        # Hallucination hedge: flag absolute medical/legal claims
        dangerous_claims = [
            r"you (?:definitely|certainly) have \w+",
            r"this is (?:not |)a (?:crime|legal|illegal)",
            r"(?:guaranteed|100%) (?:return|profit|cure)",
        ]
        for pattern in dangerous_claims:
            if re.search(pattern, response, re.IGNORECASE):
                return GuardrailResult(
                    allowed=False,
                    reason="dangerous_absolute_claim"
                )

        return GuardrailResult(allowed=True)

    def process(self, user_input: str, model_fn) -> str:
        """Full pipeline: input check → model → output check."""
        input_check = self.check_input(user_input)
        if not input_check.allowed:
            return f"I can't help with that. ({input_check.reason})"

        response = model_fn(user_input)

        output_check = self.check_output(response)
        if not output_check.allowed:
            return "I need to be careful with this topic. Let me rephrase..."
        if output_check.modified_text:
            return output_check.modified_text

        return response

Guardrail Frameworks

You don't have to build everything from scratch:

  • NVIDIA NeMo Guardrails: Programmable guardrails using Colang (a dialogue flow language). Define conversational rails, topic boundaries, and fact-checking flows. Good for enterprise deployments.
  • Guardrails AI: Python framework for validating LLM outputs against structured schemas. Generates corrective re-prompts when validation fails. Good for structured output tasks.
  • LlamaGuard (Meta): A fine-tuned Llama model that classifies inputs and outputs against a safety taxonomy. Use it as the content filter layer in your stack.

In practice, production systems combine multiple frameworks. LlamaGuard handles content safety, NeMo Guardrails manages conversation flow, and custom regex catches domain-specific risks.

Human Oversight for Safety-Critical Systems

Some decisions shouldn't be fully automated. Medical diagnoses. Loan approvals. Content moderation at scale. Criminal risk assessment. The model can do the heavy lifting, but a human needs to be in the loop for the cases that matter most.

Confidence-Based Routing

The practical pattern: route predictions based on model confidence. High confidence → auto-approve. Low confidence → human review. This gives you automation efficiency where the model is sure and human judgment where it's not.

from dataclasses import dataclass
from enum import Enum

class Decision(Enum):
    AUTO_APPROVE = "auto_approve"
    HUMAN_REVIEW = "human_review"
    AUTO_REJECT  = "auto_reject"

@dataclass
class TriageResult:
    decision: Decision
    confidence: float
    prediction: int
    reason: str

def triage_prediction(
    model,
    input_data,
    high_threshold: float = 0.95,
    low_threshold: float = 0.30,
) -> TriageResult:
    """
    Route predictions based on calibrated confidence.

    High confidence → auto-approve (most volume)
    Medium confidence → human review (where human time is focused)
    Low confidence → auto-reject or escalate
    """
    probs = model.predict_proba(input_data)
    confidence = float(probs.max())
    prediction = int(probs.argmax())

    if confidence >= high_threshold:
        return TriageResult(
            decision=Decision.AUTO_APPROVE,
            confidence=confidence,
            prediction=prediction,
            reason="high_confidence"
        )
    elif confidence <= low_threshold:
        return TriageResult(
            decision=Decision.AUTO_REJECT,
            confidence=confidence,
            prediction=prediction,
            reason="low_confidence"
        )
    else:
        return TriageResult(
            decision=Decision.HUMAN_REVIEW,
            confidence=confidence,
            prediction=prediction,
            reason=f"uncertain (conf={confidence:.2f})"
        )

# Typical distribution in production:
# ~70-80% auto-approved (model handles routine cases)
# ~15-25% human review (focused expert attention)
# ~5% auto-rejected (obvious negatives)

Calibration: Why Raw Softmax Scores Lie

A critical gotcha: the numbers coming out of model.predict_proba() are not probabilities. A neural network that says "95% confident" might only be correct 70% of the time. Modern deep networks are notoriously overconfident — they produce sharp, peaked probability distributions even when they're wrong.

If your triage system trusts uncalibrated scores, you'll auto-approve predictions the model is wrong about. Fix this with temperature scaling (simplest), Platt scaling (logistic regression on logits), or isotonic regression (non-parametric). Calibrate on a held-out validation set, and check calibration regularly — it drifts with distribution shift.

Closing the Feedback Loop

Human review generates labels. Those labels should flow back into the training pipeline. This is the feedback loop that makes the system improve over time:

  1. Human reviews uncertain predictions, provides correct label
  2. Reviewed examples are added to the training set (prioritized — these are the hard cases)
  3. Model is periodically retrained on the expanded dataset
  4. Confidence thresholds are re-calibrated

Without this loop, the human is just a safety net. With it, the human is also a teacher.

The Automation Paradox

Here's the uncomfortable truth: as the model gets better and handles more cases autonomously, the human reviewer sees only the hardest cases. Their job gets harder, they get fatigued faster, and their error rate goes up — precisely when accuracy matters most. This is the automation paradox, and it's well-documented in aviation and industrial automation.

Mitigations: rotate reviewers, mix in easy "calibration" cases alongside hard ones, and monitor human reviewer agreement rates. If inter-reviewer agreement drops below 80%, your review process is failing and the model can't learn from it either.
