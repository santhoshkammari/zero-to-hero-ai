Privacy

Chapter 18: Responsible AI and Ethics 6 subtopics
TL;DR

ML models are hungry for data, but that data belongs to people. Differential privacy provides a mathematical framework for bounding how much any individual's data can influence a model's outputs. Federated learning keeps data on-device while still training global models. Anonymization techniques (k-anonymity, l-diversity) protect datasets but can be defeated by linkage attacks. Regulations like GDPR and CCPA give individuals rights over their data, including the "right to be forgotten" — which creates the technical challenge of machine unlearning. Privacy isn't a constraint on ML — it's a design requirement.

In 2006, Netflix released a dataset of 100 million movie ratings, "anonymized" by removing names. Researchers at the University of Texas showed they could re-identify users by cross-referencing the Netflix data with public IMDB reviews. Two data points — a moderately unusual movie and an approximate date — were enough to uniquely identify someone in most cases. Netflix settled a privacy lawsuit and cancelled a planned sequel competition.

The lesson: removing names doesn't make data anonymous. ML models can memorize individual training examples. And the data you collect today might be linked to other datasets in ways you can't predict. Privacy in ML requires mathematical guarantees, not good intentions.

Differential Privacy

Differential privacy (DP) is the gold standard for privacy guarantees because it provides a mathematical bound, not a heuristic one. The core idea: a computation is differentially private if its output is approximately the same whether or not any single individual's data is included in the input.

The Epsilon-Delta Definition

A randomized mechanism M satisfies (ε, δ)-differential privacy if for any two datasets D and D' that differ in exactly one record, and for any set of outputs S:

P[M(D) ∈ S] ≤ eε · P[M(D') ∈ S] + δ

Epsilon (ε) is the privacy budget — smaller ε means stronger privacy. At ε = 0, the output reveals nothing about any individual (but is also useless). At ε = ∞, there's no privacy guarantee. Practical values range from 0.1 (very private) to 10 (weak privacy). Delta (δ) is the probability of a catastrophic privacy failure — typically set to something much smaller than 1/n (the dataset size). Think of δ as the probability that the guarantee breaks down entirely.

# Differential Privacy: Core mechanisms
import numpy as np

def laplace_mechanism(true_value, sensitivity, epsilon):
    """Add Laplace noise calibrated to sensitivity/epsilon.
    
    Args:
        true_value: The actual computation result
        sensitivity: Max change in output from changing one record
        epsilon: Privacy budget (smaller = more private)
    
    Returns:
        Noisy result satisfying epsilon-differential privacy
    """
    scale = sensitivity / epsilon
    noise = np.random.laplace(0, scale)
    return true_value + noise

def gaussian_mechanism(true_value, sensitivity, epsilon, delta):
    """Add Gaussian noise for (epsilon, delta)-DP.
    
    Gaussian mechanism is preferred when composing many queries
    because Gaussian noise composes more tightly.
    """
    sigma = sensitivity * np.sqrt(2 * np.log(1.25 / delta)) / epsilon
    noise = np.random.normal(0, sigma)
    return true_value + noise

# Example: computing average income with differential privacy
true_average = 52000  # True average salary
sensitivity = 200000 / 1000  # Max salary / number of records
epsilon = 1.0

noisy_average = laplace_mechanism(true_average, sensitivity, epsilon)
print(f"True average: ${true_average:,.0f}")
print(f"DP average (ε={epsilon}): ${noisy_average:,.0f}")
print(f"Noise magnitude: ±${abs(noisy_average - true_average):,.0f}")

Privacy Budget and Composition

Every query against a private dataset consumes some of the privacy budget. If you make k queries, each with privacy parameter ε, the total privacy loss is at most kε under basic composition. The advanced composition theorem gives a tighter bound: roughly ε√(k·ln(1/δ)). The moments accountant (used in DP-SGD) provides even tighter tracking. The key insight: privacy degrades with use. You can't query a dataset infinitely many times and maintain privacy.

DP-SGD: Differentially Private Training

DP-SGD (Differentially Private Stochastic Gradient Descent) makes the model training process itself private. It works by clipping per-example gradients (bounding each example's influence) and adding calibrated Gaussian noise to the aggregated gradient before updating model parameters.

# DP-SGD: The core algorithm (simplified)
import torch
import numpy as np

def dp_sgd_step(model, loss_fn, batch_x, batch_y,
                max_grad_norm, noise_multiplier, lr):
    """One step of DP-SGD training.
    
    Key differences from standard SGD:
    1. Compute per-example gradients (not batch gradient)
    2. Clip each example's gradient to max_grad_norm
    3. Average clipped gradients
    4. Add calibrated Gaussian noise
    5. Update parameters
    """
    per_example_grads = []
    
    # Step 1: Per-example gradients
    for i in range(len(batch_x)):
        model.zero_grad()
        output = model(batch_x[i:i+1])
        loss = loss_fn(output, batch_y[i:i+1])
        loss.backward()
        
        # Collect gradients for this example
        example_grad = []
        for param in model.parameters():
            if param.grad is not None:
                example_grad.append(param.grad.clone())
        per_example_grads.append(example_grad)
    
    # Step 2: Clip each example's gradient
    for grad_list in per_example_grads:
        total_norm = torch.sqrt(
            sum(g.norm()**2 for g in grad_list)
        )
        clip_factor = min(1.0, max_grad_norm / (total_norm + 1e-6))
        for g in grad_list:
            g.mul_(clip_factor)
    
    # Step 3: Average clipped gradients
    n = len(per_example_grads)
    avg_grads = []
    for param_idx in range(len(per_example_grads[0])):
        stacked = torch.stack(
            [per_example_grads[i][param_idx] for i in range(n)]
        )
        avg_grads.append(stacked.mean(dim=0))
    
    # Step 4: Add noise
    for avg_g in avg_grads:
        noise = torch.normal(
            0, noise_multiplier * max_grad_norm / n,
            size=avg_g.shape
        )
        avg_g.add_(noise)
    
    # Step 5: Update parameters
    with torch.no_grad():
        for param, grad in zip(model.parameters(), avg_grads):
            param.sub_(lr * grad)

The privacy-utility tradeoff is real. DP-SGD typically reduces model accuracy by 2-10% compared to non-private training, depending on the privacy budget and dataset size. Larger datasets tolerate DP better because each individual's contribution is smaller. For small datasets, the noise required for meaningful privacy can destroy model utility. This is a genuine engineering tradeoff, not a problem with a clever solution.

Local vs Global Differential Privacy

Global DP assumes a trusted curator who holds the raw data and adds noise to query results. Local DP assumes no trusted party — each individual adds noise to their own data before sharing it. Apple uses local DP for collecting emoji usage statistics; Google uses RAPPOR (Randomized Aggregatable Privacy-Preserving Ordinal Response) for Chrome telemetry. Local DP provides stronger privacy guarantees but requires much more data to achieve the same utility as global DP.

Federated Learning

Federated learning (FL) trains a shared model without centralizing data. Instead of sending data to a server, the server sends the model to the data. Each device (phone, hospital, bank) trains the model locally and sends only model updates (gradients or weights) back to the server. The data never leaves the device.

Federated Averaging (FedAvg)

The standard FL algorithm is FedAvg: the server sends the current global model to selected clients. Each client runs several epochs of local SGD on their data. Clients send updated model weights to the server. The server averages the updates (weighted by dataset size) to produce a new global model. Repeat.

# Federated Averaging: simplified implementation
import numpy as np
from copy import deepcopy

class FederatedServer:
    """Central server for federated learning."""
    
    def __init__(self, global_model):
        self.global_model = global_model
    
    def aggregate(self, client_models, client_sizes):
        """Weighted average of client models (FedAvg)."""
        total_size = sum(client_sizes)
        
        # Weighted average of parameters
        global_params = self.global_model.state_dict()
        for key in global_params:
            global_params[key] = sum(
                client_models[i].state_dict()[key] * (client_sizes[i] / total_size)
                for i in range(len(client_models))
            )
        
        self.global_model.load_state_dict(global_params)
        return self.global_model

class FederatedClient:
    """A participant in federated learning."""
    
    def __init__(self, local_data, local_labels):
        self.data = local_data
        self.labels = local_labels
    
    def local_train(self, model, epochs=5, lr=0.01):
        """Train model on local data for several epochs."""
        local_model = deepcopy(model)
        optimizer = torch.optim.SGD(local_model.parameters(), lr=lr)
        
        for epoch in range(epochs):
            optimizer.zero_grad()
            output = local_model(self.data)
            loss = torch.nn.functional.cross_entropy(output, self.labels)
            loss.backward()
            optimizer.step()
        
        return local_model

Cross-Device vs Cross-Silo

Cross-device FL involves millions of mobile devices with tiny datasets each (Google Keyboard, Apple Siri). Challenges: devices are unreliable, communication is expensive, data is highly non-IID. Cross-silo FL involves a smaller number of institutions (hospitals, banks) with larger datasets. Communication is more reliable, but legal and competitive concerns about data sharing are the primary motivation. The technical challenges differ significantly.

Non-IID Data Challenges

The biggest technical challenge in federated learning is non-IID data — the data on each device is not identically distributed. Your phone's text patterns are very different from mine. A hospital in rural India sees different diseases than one in Tokyo. When data distributions vary significantly across clients, FedAvg converges slowly or to a worse solution than centralized training. Techniques like FedProx (adding a proximal term to keep local models close to the global model) and personalization approaches partially address this.

Secure Aggregation

Even model updates can leak information. Gradient attacks can reconstruct training images from gradients. Secure aggregation protocols ensure the server only sees the aggregate update, not individual client updates. This combines cryptographic techniques with federated learning to close the gradient leakage attack vector.

Federated learning is not automatically private. FL reduces the attack surface by keeping raw data on-device, but model updates still contain information about the training data. For genuine privacy guarantees, combine FL with differential privacy (add noise to local updates before aggregation) and secure aggregation (hide individual updates from the server). Each layer of defense addresses a different threat.

Data Anonymization

K-Anonymity, L-Diversity, T-Closeness

K-anonymity requires that every record in a released dataset is indistinguishable from at least k-1 other records with respect to quasi-identifiers (attributes that could be used for re-identification, like zip code + age + gender). The Netflix attack worked because combinations of quasi-identifiers were unique. K-anonymity with k=5 means every combination appears at least 5 times.

L-diversity extends k-anonymity by requiring that each group of indistinguishable records contains at least l distinct values of the sensitive attribute. Without this, a k-anonymous group where everyone has the same disease still leaks the disease status.

T-closeness further requires that the distribution of sensitive attributes within each group is close (within threshold t) to the overall distribution. This prevents attacks based on the distribution of sensitive values within a group.

# K-Anonymity check
import pandas as pd

def check_k_anonymity(df, quasi_identifiers, k):
    """Check if a dataset satisfies k-anonymity.
    
    Returns groups that violate k-anonymity.
    """
    group_sizes = df.groupby(quasi_identifiers).size()
    violations = group_sizes[group_sizes < k]
    
    if len(violations) == 0:
        print(f"✓ Dataset satisfies {k}-anonymity")
        print(f"  Smallest group size: {group_sizes.min()}")
    else:
        print(f"✗ Dataset VIOLATES {k}-anonymity")
        print(f"  {len(violations)} groups have fewer than {k} records")
        print(f"  Smallest group size: {group_sizes.min()}")
        print(f"  Records in violating groups: {violations.sum()}")
    
    return violations

def generalize_for_k_anonymity(df, quasi_identifiers, k):
    """Simple generalization: bin numeric values to achieve k-anonymity.
    
    In practice, you'd use more sophisticated algorithms
    (Mondrian, Incognito, etc.)
    """
    df_gen = df.copy()
    
    for col in quasi_identifiers:
        if df[col].dtype in ['int64', 'float64']:
            # Bin numeric values into ranges
            n_bins = max(2, len(df) // (k * 2))
            df_gen[col] = pd.cut(df[col], bins=n_bins).astype(str)
    
    # Check if we achieved k-anonymity
    check_k_anonymity(df_gen, quasi_identifiers, k)
    return df_gen

Synthetic Data for Privacy

An increasingly popular approach: generate synthetic data that preserves the statistical properties of the real data without containing any actual records. Techniques range from simple methods (fitting a multivariate distribution and sampling) to sophisticated ones (training a GAN or diffusion model on the real data, then generating synthetic records). The privacy guarantee depends on how the synthetic data generator is trained — if it memorizes training examples, synthetic data can still leak private information. Combining synthetic data generation with differential privacy provides formal guarantees.

Privacy Regulations and Compliance

GDPR

The EU's General Data Protection Regulation establishes several principles directly relevant to ML: data minimization (collect only what you need), purpose limitation (use data only for the stated purpose), right to erasure (individuals can request deletion of their data), and data protection by design (build privacy into systems from the start, not as an afterthought). For ML practitioners, the right to erasure is particularly challenging — how do you "delete" someone's data from a trained model?

Machine Unlearning

Machine unlearning is the problem of removing the influence of specific training examples from a trained model without retraining from scratch. Exact unlearning is provably equivalent to retraining without the data — achievable for simple models or with clever data partitioning (SISA: Sharded, Isolated, Sliced, Aggregated training). Approximate unlearning modifies the model to approximately remove the influence — faster but without formal guarantees. This is an active research area because retraining large models from scratch every time someone exercises their right to erasure is computationally infeasible.

# SISA Training: enabling efficient exact unlearning
# The idea: partition training data into shards, train separate
# models on each shard, aggregate predictions via ensemble.
# To unlearn: retrain only the shard containing the deleted data.

class SISATrainer:
    """Sharded, Isolated, Sliced, Aggregated training.
    
    Enables efficient exact unlearning by partitioning data.
    """
    
    def __init__(self, model_factory, n_shards=5):
        self.n_shards = n_shards
        self.model_factory = model_factory
        self.shard_models = []
        self.shard_data = []
    
    def train(self, X, y):
        """Train an ensemble of models on disjoint data shards."""
        indices = np.random.permutation(len(X))
        shard_size = len(X) // self.n_shards
        
        self.shard_models = []
        self.shard_data = []
        
        for i in range(self.n_shards):
            start = i * shard_size
            end = start + shard_size if i < self.n_shards - 1 else len(X)
            shard_idx = indices[start:end]
            
            model = self.model_factory()
            model.fit(X[shard_idx], y[shard_idx])
            
            self.shard_models.append(model)
            self.shard_data.append(set(shard_idx))
    
    def predict(self, X):
        """Ensemble prediction: majority vote or average."""
        predictions = np.array([m.predict(X) for m in self.shard_models])
        # Majority vote for classification
        from scipy import stats
        return stats.mode(predictions, axis=0).mode[0]
    
    def unlearn(self, X, y, remove_indices):
        """Remove specific examples by retraining affected shards only."""
        remove_set = set(remove_indices)
        
        for i, shard_idx in enumerate(self.shard_data):
            if shard_idx & remove_set:  # Intersection: shard is affected
                # Retrain this shard without the removed examples
                remaining = list(shard_idx - remove_set)
                if remaining:
                    self.shard_models[i] = self.model_factory()
                    self.shard_models[i].fit(X[remaining], y[remaining])
                    self.shard_data[i] = set(remaining)
                    print(f"  Retrained shard {i} "
                          f"({len(remaining)} examples remain)")

Privacy-Preserving ML Techniques

Homomorphic Encryption

Homomorphic encryption (HE) allows computation on encrypted data without decrypting it. You encrypt your medical records, send them to a cloud ML service, the service runs inference on the encrypted data, and sends back an encrypted result that only you can decrypt. The service never sees your data. Fully homomorphic encryption (FHE) supports arbitrary computations but is currently 1,000-1,000,000x slower than plaintext computation. Partially homomorphic schemes (supporting either addition or multiplication, but not both) are more practical for specific use cases.

Secure Multi-Party Computation

Secure multi-party computation (SMPC) allows multiple parties to jointly compute a function over their inputs without revealing those inputs to each other. Two hospitals can train a joint model on their combined patient data without either hospital seeing the other's patients. SMPC protocols use secret sharing, garbled circuits, or oblivious transfer. The communication overhead is significant but decreasing as protocols improve.

Trusted Execution Environments

Trusted Execution Environments (TEEs) like Intel SGX and ARM TrustZone provide hardware-isolated enclaves where code and data are protected even from the operating system and hypervisor. Data is encrypted in memory, and only code running inside the enclave can access it. TEEs provide strong privacy guarantees with minimal performance overhead, but they rely on hardware trust — if the hardware is compromised (as has happened with SGX side-channel attacks), the guarantees evaporate.

Informed Consent and Data Rights

Consent Frameworks

Meaningful consent requires that individuals understand what data is being collected, how it will be used, and what the consequences are. In practice, consent in ML is often a legal fiction — nobody reads the terms of service, and even if they did, the downstream uses of data in ML training are too complex and speculative to be meaningfully communicated. Some researchers argue for moving beyond consent to structural protections (differential privacy, data minimization) that protect people regardless of whether they read the fine print.

Right to be Forgotten

GDPR Article 17 establishes the right to erasure. For traditional databases, this is straightforward: delete the record. For ML models, it's a research problem. The model has processed the data during training. How do you certify that a model no longer "remembers" a particular training example? Membership inference attacks (covered in Section 5) provide one test: if an attacker can't determine whether a specific example was in the training data, the model has effectively "forgotten" it. But proving a negative is hard, and this remains an open challenge.

Privacy is not anonymization. Removing names, hashing IDs, and dropping sensitive columns provides a false sense of security. Linkage attacks, model memorization, and inference attacks can reconstruct private information from seemingly anonymous data. Real privacy requires mathematical guarantees (differential privacy), architectural choices (federated learning), or cryptographic protection (HE, SMPC). If your privacy strategy is "we removed the names," it's not a privacy strategy.
