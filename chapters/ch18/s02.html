Explainability & Interpretability

Chapter 18: Responsible AI and Ethics Section 2
TL;DR

Interpretable models (logistic regression, EBMs, decision trees) let you read the decision logic directly. Post-hoc explanations (SHAP, LIME, counterfactuals) approximate what a black box is doing after the fact. Start with interpretable — reach for a black box + SHAP only when you've verified the accuracy gap is real and worth the explainability cost.

Interpretability vs Explainability

Interpretability: you can look at the model and understand its mechanics. A linear regression coefficient is the explanation. Explainability: you can produce a human-understandable reason for a specific prediction, even if the model is opaque. SHAP values for an XGBoost prediction are explanations — approximations of a process you can't inspect directly.

The distinction matters because interpretable models give you faithful explanations by definition. Post-hoc explanations are always approximations — and approximations can lie.

The Explainability-Accuracy Tradeoff

The conventional wisdom: more accurate models are less interpretable. This is partially true for deep learning on images/text, but the gap is often much smaller than assumed on tabular data. Tuned EBMs routinely match gradient-boosted ensembles. Always benchmark an interpretable model before assuming you need a black box. If the accuracy difference is 0.2%, you just saved yourself an entire explainability pipeline.

Inherently Interpretable Models

Logistic Regression

Each coefficient directly tells you the effect. For a loan default model: coefficient of 2.3 on debt-to-income ratio means each unit increase multiplies the odds of default by e2.3 ≈ 10×. No approximations, no sampling.

import numpy as np
from sklearn.linear_model import LogisticRegression
from sklearn.preprocessing import StandardScaler

def explain_logistic_model(model, feature_names, top_k=10):
    """Extract and rank feature contributions from a logistic regression.
    Standardize features first so coefficients are comparable."""

    coefs = model.coef_[0]
    odds_ratios = np.exp(coefs)

    explanations = sorted(
        zip(feature_names, coefs, odds_ratios),
        key=lambda x: abs(x[1]),
        reverse=True
    )[:top_k]

    print("Feature                  | Coefficient | Odds Ratio")
    print("-" * 55)
    for name, coef, odds in explanations:
        direction = "↑" if coef > 0 else "↓"
        print(f"{name:25s}| {coef:+10.4f}  | {odds:.3f} {direction}")

    return explanations

Explainable Boosting Machines (EBMs) — The Power Tool

EBMs are the best-kept secret in ML fairness. They're Generalized Additive Models that use boosted trees to learn each feature's contribution independently, then (optionally) pairwise interactions. The result: a model that's as accurate as gradient boosting on most tabular tasks but where you can plot exactly what each feature does.

from interpret.glassbox import ExplainableBoostingClassifier
from interpret import show

# Train — API is sklearn-compatible
ebm = ExplainableBoostingClassifier(
    max_bins=256,
    interactions=10,       # top-10 pairwise interactions
    outer_bags=8,
    inner_bags=0,
    learning_rate=0.01,
    max_rounds=5000,
    early_stopping_rounds=50
)
ebm.fit(X_train, y_train)

# Global explanation: what does each feature do overall?
global_exp = ebm.explain_global()
show(global_exp)  # Interactive plots in Jupyter

# Local explanation: why this specific prediction?
local_exp = ebm.explain_local(X_test[:5], y_test[:5])
show(local_exp)

# Extract the shape function for a single feature
# This IS the model — not an approximation
feature_idx = 0  # e.g., "age"
bins = ebm.bins_[feature_idx][0]
scores = ebm.term_scores_[feature_idx]
print(f"Feature: {ebm.feature_names_in_[feature_idx]}")
print(f"Bins: {len(bins)}, Score range: [{scores.min():.3f}, {scores.max():.3f}]")
When to reach for EBMs: Any tabular classification or regression where you need to explain individual predictions (credit, healthcare, insurance, HR). They train slower than XGBoost but the explanations are exact — not approximations. If you're currently using XGBoost + SHAP, benchmark an EBM first. You may not need SHAP at all.

Decision trees and rule lists are interpretable by inspection but too weak for most production tasks. Use them for quick baselines or when regulations require literal rule-based explanations.

Post-Hoc Explanation Methods — The Deep Part

SHAP (SHapley Additive exPlanations)

SHAP is built on Shapley values from cooperative game theory. The intuition: each feature is a "player" in a game where the "payout" is the prediction. A Shapley value tells you how much each player contributed, computed by averaging the marginal contribution over all possible coalitions (orderings) of features.

The uniqueness theorem (Shapley, 1953) proves this is the only attribution method satisfying four axioms simultaneously: efficiency (contributions sum to the prediction), symmetry, linearity, and a null player property. That's why SHAP is the standard — it's not just popular, it's mathematically principled.

TreeSHAP vs KernelSHAP:

TreeSHAPKernelSHAP
Works withTree-based models only (XGBoost, LightGBM, RF, CatBoost)Any model (treats it as a black box)
SpeedO(TLD²) — polynomial, fastO(2ᴹ) approximated by sampling — slow
Exact?Yes (exact Shapley for trees)Approximate (Monte Carlo sampling)

import shap
import xgboost as xgb

# Train a model
model = xgb.XGBClassifier(n_estimators=200, max_depth=5, random_state=42)
model.fit(X_train, y_train)

# TreeSHAP — exact and fast for tree models
explainer = shap.TreeExplainer(model)
shap_values = explainer.shap_values(X_test)

# Global feature importance — mean |SHAP| per feature
shap.summary_plot(shap_values, X_test, plot_type="bar")

# Beeswarm — shows direction and magnitude per feature per sample
shap.summary_plot(shap_values, X_test)

# Single prediction explanation — waterfall
shap.waterfall_plot(explainer.expected_value, shap_values[0], X_test.iloc[0])

# Dependence plot — how one feature's SHAP interacts with another
shap.dependence_plot("age", shap_values, X_test, interaction_index="income")

LIME (Local Interpretable Model-agnostic Explanations)

LIME's approach: perturb the input around a specific prediction, observe how the model's output changes, and fit a simple interpretable model (typically linear) to those perturbations. The local linear model is the explanation.

import lime
import lime.lime_tabular

explainer = lime.lime_tabular.LimeTabularExplainer(
    training_data=X_train.values,
    feature_names=X_train.columns.tolist(),
    class_names=["Rejected", "Approved"],
    mode="classification",
    discretize_continuous=True
)

# Explain a single prediction
exp = explainer.explain_instance(
    X_test.iloc[0].values,
    model.predict_proba,
    num_features=10,
    num_samples=5000    # more samples = more stable, slower
)

# Display in notebook
exp.show_in_notebook()

# Get as a list of (feature_rule, weight) pairs
print(exp.as_list())
# e.g., [('debt_ratio > 0.42', -0.31), ('credit_score <= 650', -0.28), ...]

Neural Network Methods (Brief)

Integrated Gradients computes attribution by integrating the gradient along a straight-line path from a baseline input (typically zeros) to your actual input. It's the go-to for deep learning because it satisfies completeness (attributions sum to the difference between baseline and actual output) and is implementation-agnostic.

DeepLIFT decomposes the output difference from a reference into feature contributions using modified backpropagation. Faster than Integrated Gradients but limited to specific architectures. Both are available in Captum (PyTorch) and tf-explain (TensorFlow).

Counterfactual Explanations with DiCE

Instead of "these features mattered most," counterfactuals answer: "what's the smallest change that would flip the decision?" That's often what end users actually want to know. "If your income were $5K higher or your debt-to-income ratio were under 0.35, you would have been approved."

import dice_ml

# Wrap your data and model

    query,
    total_CFs=4,                    # generate 4 alternatives
    desired_class="opposite",
    features_to_vary=["income", "debt_ratio", "credit_score"],  # actionable only
    permitted_range={"income": [20000, 200000]}                  # realistic bounds
)
counterfactuals.visualize_as_dataframe()
Actionability matters. A counterfactual that says "if you were 10 years younger" is useless. Always constrain counterfactuals to actionable features the user can actually change. DiCE supports this natively with features_to_vary and permitted_range.

TCAV (Testing with Concept Activation Vectors)

TCAV lets you test whether a neural network uses human-defined concepts. Instead of "pixel 4,312 matters," you ask "does the concept 'striped texture' influence this classification?" Useful for auditing image models in domains where you care about concept-level reasoning (medical imaging, content moderation). Requires curated concept example sets.

Attention as Explanation — A Warning

Attention weights in transformers are not reliable explanations. Jain & Wallace (2019) showed that very different attention distributions can produce identical predictions, and that attention often doesn't correlate with gradient-based feature importance. Use attention for debugging intuition, not for formal explanations or compliance.

Faithfulness Testing

An explanation is useless if it doesn't reflect what the model actually does. Faithfulness testing answers: "if I remove the features this method says are most important, does the prediction actually change?"

import numpy as np

def faithfulness_test(model, X_sample, shap_values, feature_names, top_k=5):
    """Remove top-attributed features and check if predictions change.
    If they don't, your explanations are unfaithful."""

    results = []
    for i in range(len(X_sample)):
        x = X_sample.iloc[[i]].copy()
        original_pred = model.predict_proba(x)[0, 1]

        # Get top-k features by |SHAP value|
        top_features = np.argsort(np.abs(shap_values[i]))[-top_k:]
        top_names = [feature_names[j] for j in top_features]

        # Zero out (or replace with mean) top features
        x_modified = x.copy()
        for feat_idx in top_features:
            x_modified.iloc[0, feat_idx] = X_sample.iloc[:, feat_idx].mean()

        modified_pred = model.predict_proba(x_modified)[0, 1]
        delta = abs(original_pred - modified_pred)

        results.append({
            "sample": i,
            "original": round(original_pred, 4),
            "modified": round(modified_pred, 4),
            "delta": round(delta, 4),
            "top_features": top_names
        })

    avg_delta = np.mean([r["delta"] for r in results])
    print(f"Average prediction change after removing top-{top_k} features: {avg_delta:.4f}")

    if avg_delta < 0.05:
        print("⚠️  Low faithfulness — explanations may not reflect actual model behavior")
    else:
        print("✓ Explanations appear faithful")

    return results

# Usage:
# faithfulness_test(model, X_test[:50], shap_values[:50], feature_names)
Run this before you trust any explanation method. If removing the top-5 SHAP features doesn't meaningfully change predictions, your SHAP explanations are decorative, not informative. This happens more often than you'd expect, especially with KernelSHAP on high-dimensional data.

Production Explainability

Real-Time Serving Constraints

Computing full SHAP values per prediction is too slow for many real-time systems. Practical patterns:

  • TreeSHAP is fast enough for real-time on tree models with moderate features (< 200)
  • Pre-compute explanations for common input patterns and cache them
  • Linear proxy model: train a logistic regression to approximate your black box, serve the proxy's coefficients as "explanations" in real-time, log full SHAP for async audit
  • Batch explanations: compute SHAP nightly, attach to each prediction in the data warehouse for retrospective audit

Explaining to Non-Technical Users

A SHAP waterfall chart is great for a data scientist. For the loan applicant, you need natural language: "Your application was declined primarily because your debt-to-income ratio (42%) exceeds our typical threshold. Your strong credit history was a positive factor." Translating feature attributions into actionable language is a UX problem, not a modeling problem — and it's where most explainability efforts fail.

The pragmatist's playbook: (1) Benchmark an EBM or logistic regression. (2) If you genuinely need a black box, use TreeSHAP for tree models or Integrated Gradients for deep learning. (3) Run faithfulness tests before trusting any explanation method. (4) Translate technical explanations into language your stakeholders can act on.
