RAG for NLP Applications

Chapter 11: Natural Language Processing 6 subtopics
TL;DR

Large language models know a lot, but they hallucinate, can't cite sources, and their knowledge has a cutoff date. Retrieval-Augmented Generation (RAG) fixes this by fetching relevant documents at query time and stuffing them into the prompt. The retrieval side is its own engineering challenge: dense embeddings vs. sparse keyword matching, chunking documents so they fit in context windows, and reranking results so the best chunks rise to the top. RAG is how you build an AI system that answers questions about your data — not just the internet's data.

Retrieval Methods

The RAG Concept

Here's the problem. You ask a language model "What was our Q3 revenue?" and it confidently makes up a number. The model has never seen your company's financial data — it's not in the training set. Even if it were, the model might confuse your numbers with another company's. LLMs are pattern-completion engines, not databases.

Retrieval-Augmented Generation solves this by adding a lookup step before generation. When a user asks a question: (1) search a knowledge base for relevant documents, (2) paste those documents into the prompt as context, (3) let the model generate an answer grounded in that context. The model isn't recalling from memory — it's reading and summarizing documents you gave it. This dramatically reduces hallucination and lets the system answer questions about private, up-to-date information.

The original RAG paper (Lewis et al., 2020) trained the retriever and generator end-to-end. In practice today, most teams use a simpler pipeline: an off-the-shelf embedding model for retrieval and a separate LLM for generation. The engineering is in the plumbing — how you chunk, embed, store, retrieve, and present documents to the model.

Dense Retrieval

Dense retrieval uses neural networks to encode both queries and documents into dense vectors (embeddings) in the same vector space. At query time, you compute the query embedding and find the nearest document embeddings using cosine similarity or dot product. Documents that are semantically similar to the query — even if they share no keywords — get retrieved.

DPR (Dense Passage Retrieval) was one of the first successful dense retrievers. It uses two separate BERT encoders — one for queries, one for passages — and trains them with contrastive learning. Positive pairs are questions matched with their gold passages; negative pairs are random or "hard" passages that look similar but don't answer the question. After training, the passage encoder embeds your entire knowledge base once, and the query encoder embeds each incoming question at search time.

Contriever improved on DPR by using unsupervised contrastive pretraining — no labeled question-passage pairs needed. It generates training pairs by taking two random spans from the same document as positives. This makes it much easier to deploy on new domains where you don't have QA training data.

from sentence_transformers import SentenceTransformer
import numpy as np

# Load a dense retrieval model
model = SentenceTransformer("BAAI/bge-small-en-v1.5")

# Your knowledge base
documents = [
    "RAG combines retrieval with generation to ground LLM responses.",
    "BM25 is a sparse retrieval algorithm based on term frequency.",
    "Vector databases store embeddings for fast similarity search.",
    "Fine-tuning adapts a pretrained model to a specific task.",
    "Chunking splits documents into smaller pieces for retrieval.",
]

# Encode documents once (offline)
doc_embeddings = model.encode(documents, normalize_embeddings=True)

# Encode query at search time
query = "How does RAG reduce hallucination?"
query_embedding = model.encode([query], normalize_embeddings=True)

# Cosine similarity (dot product since normalized)
scores = query_embedding @ doc_embeddings.T
top_indices = np.argsort(scores[0])[::-1][:3]

print("Top 3 results:")
for i in top_indices:
    print(f"  [{scores[0][i]:.3f}] {documents[i]}")

Sparse Retrieval (BM25)

BM25 is the classic keyword-matching algorithm that search engines have used for decades. It scores documents based on term frequency (how often query terms appear in the document), inverse document frequency (rare terms get higher weight), and document length normalization. No neural networks, no embeddings, no GPU — just counting words, cleverly.

BM25 is fast, interpretable, and surprisingly strong. For queries that use specific technical terms, product names, or error codes, BM25 often beats dense retrieval because it matches exact tokens. Dense models can struggle with rare terms they haven't seen enough during training. The tradeoff: BM25 misses semantic matches. "How to fix a broken pipe" won't match a document about "repairing damaged plumbing" because the words don't overlap.

# BM25 with rank_bm25
from rank_bm25 import BM25Okapi

documents = [
    "RAG combines retrieval with generation to ground LLM responses",
    "BM25 is a sparse retrieval algorithm based on term frequency",
    "Vector databases store embeddings for fast similarity search",
    "Fine-tuning adapts a pretrained model to a specific task",
    "Chunking splits documents into smaller pieces for retrieval",
]

# Tokenize (simple whitespace for demo)
tokenized_docs = [doc.lower().split() for doc in documents]
bm25 = BM25Okapi(tokenized_docs)

query = "sparse retrieval term frequency"
tokenized_query = query.lower().split()
scores = bm25.get_scores(tokenized_query)

top_indices = scores.argsort()[::-1][:3]
for i in top_indices:
    print(f"  [{scores[i]:.3f}] {documents[i]}")

Hybrid Search

Hybrid search combines dense and sparse retrieval to get the best of both worlds. The idea is simple: run both BM25 and a dense retriever, then merge their results. The standard merging technique is Reciprocal Rank Fusion (RRF): for each document, compute 1 / (k + rank) for both retriever rankings, sum them, and re-sort. The parameter k (typically 60) controls how much weight later ranks get.

Why does this work so well? Dense retrieval catches semantic matches that BM25 misses, and BM25 catches exact keyword matches that dense models miss. A query like "CUDA error 11 out of memory" needs BM25 to match the exact error code, while "how to handle GPU memory issues in PyTorch" needs dense retrieval to understand the intent. Hybrid search handles both.

def reciprocal_rank_fusion(rankings, k=60):
    """Merge multiple ranked lists using RRF."""
    fused_scores = {}
    for ranking in rankings:
        for rank, doc_id in enumerate(ranking):
            if doc_id not in fused_scores:
                fused_scores[doc_id] = 0
            fused_scores[doc_id] += 1.0 / (k + rank + 1)
    
    return sorted(fused_scores.items(), key=lambda x: x[1], reverse=True)

# Example: merge BM25 and dense retrieval rankings
bm25_ranking = ["doc_2", "doc_5", "doc_1", "doc_3", "doc_4"]
dense_ranking = ["doc_1", "doc_2", "doc_3", "doc_5", "doc_4"]

merged = reciprocal_rank_fusion([bm25_ranking, dense_ranking])
print("Hybrid ranking:", [doc_id for doc_id, score in merged])

Embedding Models

The quality of your dense retrieval depends entirely on your embedding model. Popular choices include OpenAI's text-embedding-3-small, Cohere's embed-v3, and open-source models like BAAI/bge-large-en-v1.5 or intfloat/e5-large-v2. These models are trained specifically for retrieval — they learn to place questions near their answers in the embedding space, which is fundamentally different from what a general-purpose LLM does.

When choosing an embedding model, check the MTEB leaderboard (Massive Text Embedding Benchmark). It evaluates models across classification, clustering, retrieval, and semantic similarity tasks. But also run your own evaluation: domain-specific retrieval accuracy often doesn't correlate well with generic benchmark scores. A medical QA system might need a model fine-tuned on biomedical text, not the MTEB #1.

HyDE (Hypothetical Document Embeddings)

HyDE is a clever trick for when the user's query is short or vague. Instead of embedding the raw query, you first ask an LLM to generate a hypothetical answer — what a perfect document might look like. Then you embed that hypothetical document and use it as the search query. Because the hypothetical answer is closer in style and vocabulary to actual documents, it often retrieves better results than the raw question. The cost: one extra LLM call per query, plus the risk that the hypothetical answer might steer retrieval in the wrong direction if it's off-base.

Chunking & Indexing

Chunking Strategies

Documents are too long to embed as a single vector — the meaning gets diluted. So you split them into chunks. The chunking strategy directly affects retrieval quality. Too small and chunks lack context. Too large and they contain irrelevant information that dilutes the signal.

Common approaches, from simplest to most sophisticated:

Fixed-size chunking splits text every N tokens (typically 256–512) with some overlap (50–100 tokens). Simple, predictable, but it breaks mid-sentence and mid-paragraph, creating chunks that start and end awkwardly.

Recursive character splitting tries to split on paragraph boundaries first, then sentences, then words, falling back through a hierarchy of separators. This is what LangChain's RecursiveCharacterTextSplitter does. It produces more natural chunks that respect document structure.

Semantic chunking uses an embedding model to detect topic shifts. You compute embeddings for each sentence, then split where the cosine similarity between consecutive sentences drops below a threshold. This produces chunks that are topically coherent, but it's slower and more complex to implement.

# Recursive character splitting with LangChain
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50,
    separators=["\n\n", "\n", ". ", " ", ""]
)

text = """RAG systems combine retrieval with generation.

The retrieval component searches a knowledge base for relevant documents.
These documents are then passed to a language model as context.
The model generates an answer grounded in the retrieved information.

This approach has several advantages. It reduces hallucination by
providing factual context. It enables the model to answer questions
about private or recent data not in its training set."""

chunks = splitter.split_text(text)
for i, chunk in enumerate(chunks):
    print(f"Chunk {i} ({len(chunk)} chars): {chunk[:80]}...")

Vector Databases

Vector databases store embeddings and provide fast approximate nearest-neighbor (ANN) search. Instead of computing cosine similarity against every document (brute force), they use index structures like HNSW (Hierarchical Navigable Small World graphs) or IVF (Inverted File Index) to find the top-k nearest vectors in milliseconds, even with millions of documents.

Popular options: FAISS (Meta's library, great for local/single-machine use), Pinecone (managed cloud service), Weaviate (open-source, supports hybrid search natively), Chroma (lightweight, popular for prototyping), and Qdrant (open-source, Rust-based, fast). For most projects, start with Chroma for prototyping, then move to FAISS or a managed service for production. The choice of ANN algorithm matters less than getting your chunking and embedding right.

Re-ranking & Filtering

Cross-Encoder Re-rankers

Dense retrieval uses bi-encoders: encode query and document separately, compare with dot product. This is fast because document embeddings are precomputed. But it's also limited — the query and document never "see" each other until the final comparison.

Cross-encoder re-rankers fix this by feeding the query and document together into a transformer, separated by a [SEP] token. The model can attend to both simultaneously, capturing fine-grained interactions between query terms and document content. This is dramatically more accurate than bi-encoder scoring — but also dramatically slower, because you can't precompute anything. You must run the full model for every (query, document) pair.

The solution: use a two-stage pipeline. First, retrieve 50–100 candidates quickly with a bi-encoder or BM25. Then, re-rank just those candidates with a cross-encoder to get the best 3–5. This gives you cross-encoder accuracy at near-bi-encoder speed.

from sentence_transformers import CrossEncoder

# Load a cross-encoder reranker
reranker = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

query = "How does RAG reduce hallucination?"
candidates = [
    "RAG grounds LLM responses in retrieved documents, reducing made-up content.",
    "Fine-tuning changes model weights to improve performance on specific tasks.",
    "Retrieval-augmented generation provides factual context to language models.",
    "BM25 is a sparse retrieval algorithm used in search engines.",
]

# Score each (query, candidate) pair
pairs = [[query, doc] for doc in candidates]
scores = reranker.predict(pairs)

# Sort by score
ranked = sorted(zip(scores, candidates), reverse=True)
for score, doc in ranked:
    print(f"  [{score:.3f}] {doc}")

Answer Relevance

Even after re-ranking, not every retrieved chunk is useful. Answer relevance filtering adds a final check: is this chunk actually relevant to the question? You can use a lightweight classifier or even a simple LLM call ("Is this passage relevant to the question? Yes/No") to filter out noise. Some teams use a threshold on the cross-encoder score — anything below 0.5 gets dropped, even if it was in the top-k. This prevents the model from being distracted by weakly-related context that could hurt generation quality.

Advanced RAG Patterns

GraphRAG

GraphRAG adds a knowledge graph layer on top of standard vector retrieval. Instead of treating documents as independent chunks, you extract entities and relationships to build a graph. When a query arrives, you search both the vector store and the knowledge graph, retrieving not just relevant chunks but also structured relationships between entities. This is particularly powerful for multi-hop questions like "Which investors funded companies that compete with our main product?" where the answer requires traversing relationships across multiple documents.

Self-RAG and Corrective RAG

Self-RAG teaches the model to decide when it needs retrieval. Instead of always retrieving, the model generates special tokens indicating "I need to look something up" or "I'm confident enough to answer directly." This reduces unnecessary retrieval calls and improves latency for simple questions.

Corrective RAG (CRAG) adds a quality check after retrieval. A lightweight evaluator scores the retrieved documents for relevance. If the documents are relevant, proceed normally. If they're ambiguous, the system triggers a web search for additional information. If they're irrelevant, the system falls back to the LLM's internal knowledge. This adaptive approach handles the common failure mode where the knowledge base simply doesn't contain the answer.

Agentic RAG

Agentic RAG wraps the retrieval pipeline in an agent loop. The agent can decide to reformulate the query, retrieve from multiple sources, verify answers against retrieved evidence, and iterate until it's confident. This is more expensive per query but dramatically more robust for complex questions. Think of it as giving the RAG system the ability to "think" about its retrieval strategy rather than blindly executing a fixed pipeline.

RAG Evaluation

RAGAS Framework

Evaluating RAG systems is notoriously tricky because there are multiple components that can fail independently. The RAGAS framework provides four key metrics: Faithfulness — does the answer only contain information from the retrieved context (no hallucination)? Answer Relevance — does the answer actually address the question? Context Precision — are the retrieved documents relevant to the question? Context Recall — did the retriever find all the relevant documents?

These metrics decompose the end-to-end problem into diagnosable components. If faithfulness is high but answer relevance is low, your model is correctly reading the context but the retrieved documents don't contain the answer — a retrieval problem. If context precision is high but faithfulness is low, you retrieved the right documents but the model is ignoring them — a generation problem.

RAG vs. Fine-Tuning

RAG adds knowledge at inference time (dynamic, no retraining needed, easy to update). Fine-tuning bakes knowledge into model weights (faster inference, no retrieval latency, but expensive to update). Use RAG when your knowledge changes frequently or is too large to memorize. Use fine-tuning when you need the model to learn a specific style, format, or reasoning pattern. Many production systems use both: a fine-tuned model augmented with RAG for up-to-date facts.

The 5 RAG Failure Modes

(1) Missing content — the answer isn't in your knowledge base at all. (2) Poor chunking — the answer spans two chunks and neither contains enough context. (3) Embedding mismatch — the query and answer use different vocabulary, so dense search misses. (4) Lost in the middle — the relevant chunk is retrieved but buried among irrelevant ones, and the model ignores it. (5) Extraction failure — the model has the right context but generates the wrong answer anyway. Diagnose which failure mode you're hitting before adding complexity.
