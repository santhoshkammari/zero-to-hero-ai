Raw text must become numbers before neural network can work. Embeddings and tokenization are front door of every language model. Nothing else starts before this.

Word embeddings are dense numerical word representations — compact vectors, usually 100 to 1,024 dimensions, where geometric proximity encodes semantic similarity. Idea goes back to distributional semantics in 1950s; modern jump started with Word2Vec in 2013. Tokenization decides where to cut text into pieces before embedding layer sees them. Together, embeddings and tokenization are front door of every language model. Nothing else starts before this.

One-Hot Encoding and Why It Fails

Imagine movie review analyzer. It reads film comments, decides liked or not. First problem: model speaks numbers, users speak English. Need translation layer.

Start with tiny vocabulary of five words: "love," "hate," "great," "movie," and "boring." Simplest numeric mapping is one-hot encoding — give each word vector as long as vocabulary, put single 1 at assigned position, 0s everywhere else.

# Our tiny vocabulary of 5 words
#           love  hate  great  movie  boring
# love    = [1,    0,    0,     0,     0   ]
# hate    = [0,    1,    0,     0,     0   ]
# great   = [0,    0,    1,     0,     0   ]
# movie   = [0,    0,    0,     1,     0   ]
# boring  = [0,    0,    0,     0,     1   ]

Five words, five dimensions. Each word sits at one corner of five-dimensional hypercube. Check distance between "love" and "great" — similar in review context. Dot product is 0. Check "love" and "boring." Also 0. Every distinct pair sits equally far apart. Representation tells model "love" is as unrelated to "great" as to "boring."

First problem: one-hot vectors carry zero information about meaning. Glorified ID numbers with many zeros.

Second problem: scale. Real English vocabulary has 50,000 to 100,000 words. Each word becomes 100,000-dimensional vector with 99,999 zeros. Multiply by dataset size, store ocean of nothing. Address system only. No clue which words live near each other.

Need representation where "love" and "great" sit close, "hate" and "boring" sit close, "love" and "boring" sit far. Geometry should mirror meaning. Word embeddings do this: dense vectors, usually 100 to 300 dimensions, where proximity encodes similarity.

Question: how do we learn them?

The Distributional Hypothesis

Answer comes from strong linguistics idea. In 1957, J.R. Firth wrote: "You shall know a word by the company it keeps." Distributional hypothesis: words in similar contexts tend to have similar meanings.

Back to movie reviews. Suppose dataset has these sentences:

"I love this great movie"
"I love this amazing movie"
"I hate this boring movie"
"I hate this terrible movie"

Look at slots around "great" and "amazing." Both appear between "this" and "movie," after "love." Look at "boring" and "terrible." Same pattern, after "hate." No one told model meanings. Context alone shows "great" and "amazing" play similar roles, "boring" and "terrible" play similar roles, first pair links to "love," second pair to "hate."

Distributional hypothesis is not proof that context equals meaning. Empirical rule. Works very well. If two words keep showing up in same neighborhoods, they are probably related. Not identical — "big" and "small" share contexts ("the ___ house") but mean opposites. Wrinkle later. Starting point: context carries lots of signal.

Next step: turn linguistic idea into training procedure. Tomas Mikolov and team at Google did this in 2013 with Word2Vec.

Word2Vec: Learning Embeddings from Prediction

Skip-gram — Predict the Neighbors

Skip-gram asks odd question. Given one word, can model predict nearby words? Task looks strange. Task forces model to learn rich representations.

Setup: take sentence like "I love this great movie" and slide window across it. If window size is 2, look two words each direction. When window centers on "great," context words are "love," "this," "great," and "movie." Task: given "great," predict each neighbor.

Inside model sit two weight matrices. First is embedding matrix: maps each one-hot vector to dense embedding. This matrix matters; keep it after training. Second is context matrix: maps embedding space back to vocabulary space for predictions. Usually discard this one.

Feed "great" into model. Model looks up embedding for "great" — one row in embedding matrix — then uses it to predict "love," "this," and "movie." Loss measures error. Gradients flow backward, pushing embedding for "great" toward nearby words and away from unrelated words.

Repeat for every word in every sentence across corpus. Billions of small nudges. After enough passes, words from similar neighborhoods get similar vectors. "Great" and "amazing" converge because they predict similar context. "Great" and "refrigerator" diverge because neighborhoods differ.

That is Skip-gram. Simple task. Strong result. Vectors capture semantic structure no one hand-coded.

CBOW — Predict the Center

Continuous Bag of Words (CBOW) flips Skip-gram. Use neighbors to predict center word. Given "love," "this," and "movie," predict "great."

Model averages context word embeddings into one vector — "bag of words" part. Word order inside context window gets dropped. Averaged vector predicts center word.

CBOW trains faster because each window position makes one prediction, not several. Skip-gram often works better for rare words. Reason: rare center word in Skip-gram gets many gradient updates, one per context word. In CBOW, rare context word gets averaged with common words, so signal weakens.

In practice, Skip-gram shows up more, especially on smaller datasets. Gap shrinks as data grows. Both make good embeddings. Next problem: computation bottleneck.

The Softmax Bottleneck and Negative Sampling

Problem almost killed Word2Vec early. To predict context word, model must produce probability distribution over entire vocabulary. For each training example, it scores every word, exponentiates all scores, divides by sum. That is softmax over 100,000+ words. Do this for every window position in every sentence across corpus of billions of words.

Cost is O(V) per training step, where V is vocabulary size. Brutal. Training would take years.

Negative sampling makes Word2Vec practical. Instead of asking "which word among 100,000 is most likely?" ask smaller question: "is this word-context pair real or fake?"

For each real pair — like ("great", "movie") — sample k random words that almost surely were not in context. Maybe "elephant," "Tuesday," and "carburetor." Now you have one positive and k negatives. Train binary classifier: push dot product of "great" and "movie" up, push dot products of "great" with noise words down.

# Negative sampling objective for one (center, context) pair:
#
# Maximize:  log σ(v_movie · v_great)          ← real pair, push together
#          + log σ(-v_elephant · v_great)       ← fake pair, push apart
#          + log σ(-v_tuesday · v_great)        ← fake pair, push apart
#          + log σ(-v_carburetor · v_great)     ← fake pair, push apart
#
# σ is the sigmoid function
# k = 3 negative samples in this example (typically 5-15)

Now each training step touches k + 1 words, not 100,000. Cost drops from O(V) to O(k), where k is usually 5 to 15. Larger datasets need fewer negatives; smaller datasets often need more.

Subtle detail: negative words are not sampled uniformly. They are sampled proportional to frequency^3/4. Mikolov team tried several exponents; 3/4 won empirically. Effect: frequency distribution gets flatter. Common words like "the" still appear often as negatives, but rare words appear more than raw frequency would give, so their embeddings train faster. Exponent 3/4 is empirical sweet spot, not theorem.

Another trick: subsampling frequent words. Words like "the" and "is" appear in almost every context window. They add little information but dominate training data. Word2Vec randomly drops high-frequency words during training, with drop probability rising with frequency. Training gets faster. Embedding quality often improves because remaining pairs carry more signal.

The King–Queen Magic (and Its Limits)

Most famous Word2Vec result is analogy test. Take vector for "king," subtract vector for "man," add vector for "woman," then find nearest neighbor. Result: "queen."

Why this happens: model never got rules about gender or royalty. Co-occurrence patterns across huge text were enough to carve out something like gender direction and royalty direction in vector space. Arithmetic along those directions can land on sensible answers.

# vec("king") - vec("man") + vec("woman") ≈ vec("queen")
#
# The vector difference (king - man) encodes something like
# "royalty without gender." Adding it to "woman" gives
# "royalty + female" → queen.
#
# Similarly:
# vec("paris") - vec("france") + vec("germany") ≈ vec("berlin")
# vec("bigger") - vec("big") + vec("cold") ≈ vec("colder")

Need caution. King–queen works because gender signal is strong and consistent in training data. Many analogies fail. Some succeed for bad reasons: "doctor" minus "man" plus "woman" often returns "nurse" because corpus contains stereotypes. Embeddings mirror training data, biases included. Bolukbasi et al. showed in 2016 that Word2Vec links "programmer" with male names and "homemaker" with female names. Real system needs bias auditing. Not optional.

GloVe: The Global View

Word2Vec learns by sliding small window across text, one example at a time. No full-picture view. In 2014, Pennington, Socher, and Manning asked different question: what if we start with global picture?

GloVe (Global Vectors for Word Representation) starts with co-occurrence matrix. Giant table: one row per word, one column per word, entry X_ij counts how often word i appears near word j across corpus. Movie review vocabulary might look like this:

#           love  hate  great  movie  boring
# love      -     0     12     45     1
# hate      0     -     1      38     15
# great     12    1     -      30     0
# movie     45    38    30     -      20
# boring    1     15    0      20     -
#
# (numbers are illustrative)

"Love" co-occurs heavily with "movie" and "great," rarely with "boring." "Hate" co-occurs heavily with "movie" and "boring," rarely with "great." Meaning structure already shows up in table. GloVe compresses that structure into dense vectors.

Model learns vectors w_i and w_j plus bias terms so dot product approximates log co-occurrence count:

# GloVe objective:
# w_i · w_j + b_i + b_j  ≈  log(X_ij)
#
# The loss weights each pair by f(X_ij), a function that:
# - Caps the influence of very frequent pairs (so "the"+"the" doesn't dominate)
# - Ignores very rare pairs (noise)
# f(x) = (x / x_max)^0.75  if x < x_max, else 1

Why logarithm? Raw co-occurrence counts span huge range. "The" might appear with "movie" 50,000 times while "exquisite" appears with "movie" 3 times. Log-space compresses range and makes optimization behave better.

Key GloVe idea: ratios of co-occurrence probabilities. For "ice," "steam," and "solid," ratio P(solid | ice) / P(solid | steam) is large. For "gas," ratio is small. For "water," ratio is near 1. These ratios separate "ice" from "steam" more cleanly than raw probabilities. GloVe trains dot products to reflect those ratios.

Surprising link: Levy and Goldberg showed in 2014 that Word2Vec with negative sampling implicitly factorizes a shifted pointwise mutual information matrix, close to log co-occurrence matrix. So Word2Vec and GloVe are more alike than they look. One uses local prediction, other uses global factorization, but both reach similar solutions. In practice, good Word2Vec and good GloVe embeddings perform similarly on many benchmarks. Choice often depends on pretrained vectors, not inherent superiority.

FastText: Breaking Words into Pieces

Word2Vec and GloVe share hard flaw: each word is atomic. Model sees "loving" as unrelated to "love." If word never appeared in training, no vector. Game over. Model emits useless [UNK] token.

Feel pain in movie review analyzer. User writes: "unbelievably loveable cinematography." "Unbelievably" was not in training corpus. Neither was "loveable" if corpus only had "lovable." Word2Vec fails on both. Two of three words lost.

FastText, from Bojanowski et al. at Facebook AI Research in 2017, fixes this by cracking words open. Instead of treating "loving" as one symbol, FastText represents it as bag of character n-grams — overlapping slices of 3 to 6 characters, plus whole word:

# FastText decomposes "loving" into character n-grams
# (using angle brackets as word boundary markers)
#
# n=3:  <lo, lov, ovi, vin, ing, ng>
# n=4:  <lov, lovi, ovin, ving, ing>
# n=5:  <lovi, lovin, oving, ving>
# n=6:  <lovin, loving, oving>
# whole word: <loving>
#
# The word's embedding = sum of all these n-gram vectors

Each character n-gram gets its own vector. Word embedding is sum of all n-gram vectors. Training works like Word2Vec Skip-gram with negative sampling; only center-word vector construction changes.

This gives two big gains. First, out-of-vocabulary words stop being fatal. Even if "unbelievably" never appeared in training, FastText can build reasonable vector from character n-grams seen in words like "unbelievable," "incredibly," and "believable." Model never saw exact word, but saw most parts.

Second, morphological relationships emerge automatically. "Love," "loved," "loving," and "loveable" share n-grams like "lov" and "love." Their vectors end up close — not because anyone encoded English conjugation rules, but because shared character sequences share n-gram vectors. This helps especially in morphologically rich languages like Turkish, Finnish, and German. In Turkish, one root verb can create hundreds of inflected forms. Word2Vec treats forms as separate; FastText captures structural kinship.

Back to movie review analyzer. With FastText, "unbelievably" gets reasonable vector, close to "unbelievable" and "incredibly." "Loveable" gets vector close to "lovable" and "love." Model can still work. FastText can place unseen words from their parts.

Cosine Similarity and Embedding Geometry

Need way to compare word vectors. Standard tool is cosine similarity: cosine of angle between two vectors. Range: −1 for opposite directions, 0 for perpendicular or unrelated, 1 for same direction.

# cos(a, b) = (a · b) / (||a|| × ||b||)
#
# Think of it as: how much do these vectors point
# in the same direction, ignoring how long they are?

Why cosine, not Euclidean distance? Cosine ignores magnitude and focuses on direction. That matters because embedding magnitude often correlates with word frequency. You do not want "the" to look similar to everything. Direction should carry semantic signal. Cosine factors out length.

Common surprise: "good" and "bad" often have cosine similarity around 0.5 to 0.7. Feels wrong; they are antonyms. But distributional similarity means similar usage, not similar sentiment. Antonyms fill same syntactic slots: "the food was ___," "that was a ___ idea." Raw embeddings cannot separate meaning similarity from usage similarity. Sentiment tasks need extra task-specific layers.

The Polysemy Problem

Back to movie review analyzer. User writes: "The film's bank scenes were shot on location near the river bank." Same word appears twice with different meanings. With Word2Vec, GloVe, or FastText, both instances get exact same vector — weighted average of financial and geographic senses, dominated by whichever sense was more common in training. Model cannot tell them apart.

This is polysemy: one word, multiple meanings, one representation. "Play" as theatrical performance and "play" as game verb share one vector. "Light" as illumination, not heavy, or ignite share one vector. Language is full of ambiguity. Static embeddings cannot resolve it.

For some tasks, this limit is tolerable. If 95% of "bank" in corpus means financial bank, static embedding may work for document classification. But tasks needing precise understanding — question answering, named entity recognition, machine translation — hit hard ceiling with one vector per word.

ELMo: Context Enters the Picture

ELMo (Embeddings from Language Models), from Peters et al. in 2018, broke this ceiling. Core idea: do not look up fixed vector. Run whole sentence through deep neural network and use internal states as word representations. Sentence context shapes each word vector.

ELMo uses two-layer bidirectional LSTM trained as language model. Forward LSTM reads left-to-right and predicts next word. Backward LSTM reads right-to-left and predicts previous word. At inference time, each word position gets three vectors: one from character-level CNN at bottom for morphology and spelling patterns, one from first LSTM layer for syntax like part-of-speech and grammatical role, and one from second LSTM layer for semantics like word sense and topic.

Final embedding is learned weighted mix of all three layers. Key trick: downstream tasks learn different weights. Part-of-speech tagger leans on first LSTM layer. Sentiment classifier leans more on second. Each task finds its own mix automatically.

Now "bank" in "I deposited money at the bank" gets vector shaped by "deposited" and "money." "Bank" in "I sat on the river bank" gets different vector shaped by "river." Same word. Different representations. That is contextual embedding.

ELMo hit state of art on six NLP benchmarks when published. Bigger contribution was conceptual. Word vectors do not need fixed lookup table. They can come from function conditioned on context.

From ELMo to BERT: The Transformer Revolution

ELMo used bidirectional LSTMs. They work, but they process text sequentially, word by word. That limits speed and usable context range. In late 2018, Devlin et al. published BERT (Bidirectional Encoder Representations from Transformers), replacing LSTMs with self-attention. Self-attention lets every word attend to every other word at once, regardless of distance. Relevant point here: BERT embeddings are richer, faster to compute, and better at long-range dependencies than ELMo embeddings.

BERT takes token sequence, passes it through 12 or 24 self-attention layers, and outputs contextual embedding for each token position. Output at each position is function of all input tokens. Whole sentence shapes every word representation.

After BERT, contextual embeddings became default. GPT, RoBERTa, XLNet, ALBERT — all use contextual embeddings. Architectures and training differ, core idea stays: words do not have fixed meanings; context determines meaning.

Jump from one vector per word to different vector for same word in every sentence can feel unsettling. Harder to inspect, visualize, and debug. Performance gain is real. Idea is also more honest. Words do change meaning with context.

The Vocabulary Problem

We keep talking about turning words into vectors. Harder question hides underneath: what is a "word"?

Back to movie review analyzer. Vocabulary has 50,000 English words. User writes: "The cinematographer's visualizations were unbelievably breathtaking." Which words are in vocabulary? "The" and "were," yes. "Cinematographer's," maybe not. "Visualizations," maybe not. "Unbelievably," unlikely. "Breathtaking," maybe.

Every out-of-vocabulary word collapses into one [UNK] token. Unknown. Model learns nothing from it. In domains like medical records, legal contracts, or user social media, 10 to 15 percent of words may become unknown. Like reading book with every seventh word blacked out.

Could go other way: character-level tokenization. Make every letter, space, and punctuation mark its own token. Then nothing is unknown; vocabulary is about 256 symbols or fewer for English. Problem: sequences get absurdly long. "The cat sat" jumps from 3 tokens to 11. Transformer context window is fixed. You spend it spelling words instead of processing meaning. Model also must learn that "c-a-t," "C-A-T," and "c-a-t-s" relate, even though letters arrive separately.

Subword tokenization is sweet spot. Keep common words whole — "the," "movie," "great" stay single tokens. Split rare words into reusable pieces — "unbelievably" becomes "un" + "believ" + "ably." Pieces carry meaning and can be reused. "Unbreakable" shares "un" and "able" with other words. Like Lego bricks: many words, small reusable parts.

Vocabulary often lands around 30,000 to 50,000 tokens. Sequences stay manageable. Nothing is truly unknown. Next question: which pieces should tokenizer keep?

Byte Pair Encoding (BPE)

Byte Pair Encoding started as 1994 compression algorithm, then Sennrich, Haddow, and Birch repurposed it for NLP in 2016. Recipe: start with characters. Find most frequent adjacent pair. Merge pair into new token. Repeat until target vocabulary size.

Walk through movie words. Suppose training corpus contains four words with these frequencies:

# Word frequencies in our corpus:
# "loved"  (appears 50 times)  → l o v e d _
# "love"   (appears 40 times)  → l o v e _
# "movie"  (appears 60 times)  → m o v i e _
# "movies" (appears 30 times)  → m o v i e s _
#
# (_ marks end-of-word)

Count every adjacent pair across words, weighted by frequency. Pair ("o", "v") appears in "loved" 50 times, "love" 40, "movie" 60, "movies" 30 = 180 total. Most frequent pair. Merge it.

# Merge 1: ("o", "v") → "ov"
#
# "loved"  → l ov e d _    (×50)
# "love"   → l ov e _      (×40)
# "movie"  → m ov i e _    (×60)
# "movies" → m ov i e s _  (×30)

Recount. Now ("l", "ov"), ("ov", "i"), and ("ov", "e") each appear 90 times. Three-way tie. Break tie alphabetically and merge ("l", "ov").

# Merge 2: ("l", "ov") → "lov"
#
# "loved"  → lov e d _     (×50)
# "love"   → lov e _       (×40)
# "movie"  → m ov i e _    (×60)
# "movies" → m ov i e s _  (×30)

Now ("lov", "e") appears 90 times. Merge it.

# Merge 3: ("lov", "e") → "love"
#
# "loved"  → love d _      (×50)
# "love"   → love _        (×40)
# "movie"  → m ov i e _    (×60)
# "movies" → m ov i e s _  (×30)

After three merges, "love" is single token. "Loved" becomes "love" + "d" + "_." Algorithm discovered that "loved" contains "love" from character frequency alone. No linguistics rules. No hand-coded verb tense logic.

Number of merges is hyperparameter that controls vocabulary size. Start with ~256 base characters, add one token per merge. GPT-2 uses about 50,257 tokens. GPT-4 uses around 100,000. LLaMA 3 uses 128,000. More tokens mean shorter sequences but larger embedding table.

Critical inference detail: BPE replays merge rules in exact order learned during training. It does not simply grab longest match. Merge priority list is tokenizer memory. Model must use exact tokenizer it trained with. Different merge table gives different token IDs, and embedding layer stops meaning same thing.

WordPiece

WordPiece came from Schuster and Nakajima in 2012 for Japanese and Korean speech processing, then became famous as BERT tokenizer. At first glance it looks like BPE — start with characters, merge pairs. Key difference: merge criterion.

BPE merges most frequent pair. WordPiece merges pair that maximizes training-data likelihood. If tokens "a" and "b" are common alone but rarely together, merging helps little. If they appear together often relative to their separate frequencies, merge captures real pattern. This is closely tied to pointwise mutual information (PMI).

Result: vocabulary is a bit more linguistically aware than BPE. WordPiece more often merges real morphological units, not mere high-frequency accidents.

WordPiece uses marker ## for continuation tokens. First piece of word has no prefix. Later pieces get ##. Reconstruction is easy: strip prefix, join pieces.

# BERT (WordPiece) tokenization examples:
#
# "unbelievable"  →  ["un", "##bel", "##ie", "##va", "##ble"]
# "tokenization"  →  ["token", "##ization"]
# "the cat sat"   →  ["the", "cat", "sat"]
#
# Common words stay whole. Rare words split into reusable pieces.
# "un" appears in un-happy, un-usual, un-likely — it learns negation.
# "##ble" appears in read-able, cap-able, comfort-able — it learns suffixes.

Model learns morphology without hand-written rules. Prefix "un" carries negation across many words. Suffix "##tion" appears in "action," "nation," "tokenization" and other nominalized forms. Patterns emerge from training statistics.

Unigram and SentencePiece

BPE and WordPiece build bottom-up: start with characters, merge upward. Unigram, introduced by Taku Kudo in 2018, goes opposite way. Start with very large initial vocabulary — substrings up to some length plus all characters — then iteratively remove tokens that help corpus likelihood least.

Each token has probability, learned through expectation-maximization. A word can have several valid segmentations. "Unbreakable" could be ["un", "break", "able"], ["unbreak", "able"], or characters. Unigram scores each segmentation by product of token probabilities and picks highest-scoring one.

Multiple segmentations are feature, not bug. During training, you can sample different segmentations of same word. This is subword regularization. Sometimes "unbreakable" becomes ["un", "break", "able"], sometimes ["unbreak", "able"]. That acts like data augmentation and makes model robust to segmentation changes at test time. BPE and WordPiece are deterministic. Unigram gives controlled randomness.

SentencePiece, also by Kudo, is tokenization library that implements Unigram and BPE. Key idea: treat input as raw character stream, not pre-segmented words. Most tokenizers assume whitespace split already happened. SentencePiece does not. It treats spaces as normal characters, shown as ▁. So same system works for English, Chinese, Japanese, and more, without language-specific pre-processing rules.

SentencePiece powers T5, ALBERT, XLNet, and many multilingual models. No pre-tokenization design made it default choice for multilingual settings.

Byte-Level BPE

GPT-2 added clever twist to BPE. Instead of starting with Unicode characters — about 150,000 possibilities — it starts with 256 raw UTF-8 byte values. Any text — English, Mandarin, Arabic, emoji, even binary data — first becomes byte sequence, then BPE merges run on bytes.

Result: no unknown token. Ever. Any byte sequence is representable because base vocabulary covers every possible byte. Unseen scripts, weird Unicode, creative emoji combos — all still tokenize. [UNK] disappears.

Tradeoff is cross-script efficiency. Common English word like "the" is 3 bytes in UTF-8, and merge table usually learns it as one token. Chinese character is also 3 bytes, rare emoji can be 4. If merge table has not learned that character well, one character may cost 3 to 4 tokens. In practice, byte-level BPE often uses about 1.3 to 1.5 times more tokens for non-Latin scripts than dedicated tokenizers for those languages.

Consequence is real. If tokenizer needs 1.5 times more tokens for Hindi than English, Hindi users get 1.5 times less context window and pay 1.5 times more per API call. Real equity issue in LLM deployment. Newer models like LLaMA 3 and Gemma use larger vocabularies — 128,000 to 256,000 tokens — partly to improve efficiency across languages.

Vocabulary Size and Embedding Dimension

Two hyperparameters shape many downstream choices: vocabulary size and embedding dimension.

Vocabulary Size

If vocabulary is too small — say 4,000 tokens — common words get shredded. "Understanding" might become five tokens. Sequences grow, context window gets wasted on fragments, training slows. If vocabulary is too large — say 500,000 tokens — embedding table explodes. Each token needs dense vector, often 768 to 4,096 dimensions, so embedding layer alone can hold hundreds of millions of parameters. Rare tokens also get few training examples, so embeddings stay noisy.

English-centric models often land around 30,000 to 50,000 tokens. Multilingual models go higher, around 100,000 to 256,000, because they must cover multiple scripts efficiently. Trend in 2023–2024 moved toward larger vocabularies as models grew. When rest of model has 70 billion parameters, bigger embedding table hurts less.

Embedding Dimension

For classic static embeddings like Word2Vec and GloVe, common dimensions are 50, 100, 200, or 300. Smaller dimensions train faster and use less memory but may miss fine-grained distinctions. Larger dimensions can capture richer relations but may overfit on small datasets and usually show diminishing returns past 300.

For Transformer models, embedding dimension usually matches model hidden dimension. BERT-base uses 768, BERT-large 1,024, GPT-3 12,288, LLaMA-70B 8,192. Embedding width and model width should scale together. Little point in 4,096-dimensional embeddings feeding 768-dimensional Transformer.

Rough heuristic: make embedding dimension large enough that extra dimensions still help downstream performance, no larger. In practice, match pretrained model. If training from scratch, start with community defaults for your model scale, then adjust from validation results.

Positional Embeddings: Giving Order to Orderless Models

Words become vectors. Order problem remains. Transformers process tokens in parallel, so they have no built-in sense of sequence order. "dog bites man" and "man bites dog" would look identical if you used only word embeddings, because same words appear. Self-attention compares all pairs but does not know which token came first.

Positional embeddings fix this by injecting position information into each token representation. Original Transformer paper used sinusoidal positional encodings — fixed mathematical functions that give each position unique pattern:

# Sinusoidal positional encoding:
# PE(pos, 2i)   = sin(pos / 10000^(2i/d))
# PE(pos, 2i+1) = cos(pos / 10000^(2i/d))
#
# pos = position in the sequence (0, 1, 2, ...)
# i = dimension index
# d = embedding dimension
#
# Each position gets a unique vector. Different dimensions
# oscillate at different frequencies, creating a unique
# "fingerprint" for each position.

Sinusoidal encodings are fixed, not learned. Nice property: relative positions can be computed from them. PE(pos + k) can be written as linear function of PE(pos), which in theory lets model learn relative position behavior.

Most modern models use learned positional embeddings instead: separate embedding table where each position from 0 to max_length − 1 gets trainable vector. BERT and GPT-2 use this approach. Model learns whatever positional patterns help, without sinusoid constraint.

More recent models often use Rotary Position Embeddings (RoPE), including LLaMA. RoPE does not add positional vector. It rotates query and key vectors in attention by angle proportional to position. That naturally encodes relative position because angle between tokens depends on distance, not absolute index. RoPE handles long sequences better than fixed position tables and became dominant in modern LLMs.

Regardless of method, final model input combines token embedding and positional embedding. Token embedding says what token is. Positional embedding says where token sits. Need both.

The Full Picture

Start: one-hot vectors — huge, wasteful, meaning-blind. Then distributional context showed how to learn meaning, leading to Skip-gram and CBOW. Negative sampling fixed softmax bottleneck. GloVe used global co-occurrence matrices. FastText broke words into character n-grams so unknown words became less fatal. Static embeddings hit ceiling because one word always meant one vector. ELMo and BERT broke that ceiling with context-shaped representations. Then tokenization took over: BPE merges bottom-up, WordPiece merges by likelihood, Unigram prunes top-down, byte-level BPE removes unknown tokens. Final knobs: vocabulary size, embedding dimension, positional encoding.

Next time you see tokenizer config or embedding layer in model architecture, goal is simple: know what each piece does and why it exists. That mental model explains how raw text turns into tensors neural network can use.
