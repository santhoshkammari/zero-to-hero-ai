Understanding what happens between "model produces logits" and "text on screen" — what temperature, top-p, beam search do to probability distribution — essential for anyone working with language models.

Text generation = producing token sequences — words, subwords, characters — one at a time. Machine translation is oldest, most demanding application. Together they form backbone of modern NLP: chatbots, code completion, summarization, translation, creative writing. Decoding strategies here are same ones powering every major language model in production.

The Autoregressive Loop

Running example for this section. Building tiny story-writing assistant. Vocabulary of exactly five tokens: the, cat, sat, on, mat. Whole language. Five words.

Assistant's job: continue prompt by predicting one token at a time. Give it "the", it produces probability distribution over all five tokens — five numbers summing to one. Each number = model's confidence that token comes next:

Prompt: "the"

Model output (probabilities):
  the  → 0.02
  cat  → 0.70
  sat  → 0.15
  on   → 0.08
  mat  → 0.05

Model thinks cat most likely after the. Pick cat, append to sequence, prompt becomes "the cat". Feed back in, get new distribution. Pick again. Append again. Repeat.

This loop — predict, pick, append, repeat — is autoregressive generation. "Autoregressive" = self-feeding: each generated token becomes input for predicting next one. Token 5 depends on tokens 1–4. Token 50 depends on tokens 1–49. Inherently sequential, so generation much slower than training. Training processes all positions in parallel (target sequence known). Generation builds sequence one piece at a time, can't peek ahead.

Sequential bottleneck is why KV-caching matters. Without it, model recomputes attention over entire sequence for every new token — quadratic cost. With KV-cache, model stores key/value tensors from previous positions, only computes attention for newest token. Cost drops from O(n²) per token to O(n). For 2,000-token generation: difference between minutes and seconds.

Key question glossed over: "we pick cat." How? Model gave five probabilities. Strategy to turn probabilities into concrete token choice = decoding strategy. Changes everything about output quality.

Greedy Decoding and Its Discontents

Most obvious strategy: pick token with highest probability. Every time. No randomness. In toy example, cat has probability 0.70, pick cat. Next step, whatever has highest probability, pick that too.

This is greedy decoding. Fast, deterministic (same input → same output), first thing everyone tries. Also, in many situations, terrible.

Problem: greedy decoding can't backtrack. If globally best five-word story requires picking sat (0.15) at step one instead of cat (0.70), greedy never finds it. Locked onto locally optimal path at every step. Like hiker always taking steepest trail at every fork — sometimes steepest trail leads to dead end, gentler path led to summit. Greedy hiker never finds out.

In practice, greedy decoding produces repetitive, bland text. Classic example: "I think that this is a great idea. I think that this is a great idea. I think that this is a great idea." Model keeps picking same high-probability continuation because at every step, that continuation looks best. Never has reason to deviate.

Need controlled randomness. Enter temperature.

Temperature: Reshaping Confidence

Temperature seems like magical creativity dial — turn up, get wackier text. Not wrong, but hides what's happening. Once you see mechanics, knob makes sense.

What's happening: before model produces probabilities, it produces logits — raw scores for each token, not normalized. To turn logits into probabilities, apply softmax. Temperature inserts right before softmax: divide every logit by temperature value T.

Trace through five-token vocabulary with concrete numbers. Model outputs these logits:

Raw logits:
  the  → -1.0
  cat  →  3.0
  sat  →  1.5
  on   →  0.5
  mat  → -0.2

At temperature T = 1.0, nothing changes. Divide by 1, apply softmax, get original distribution. Cat dominates.

At T = 0.5, divide each logit by 0.5, doubling them: cat's logit → 6.0, sat's → 3.0. Softmax on amplified logits: gap between cat and everything else becomes enormous. Cat's probability jumps from ~0.65 to ~0.95. Distribution gets sharper — peakier. Model more confident, more predictable. As T→0, converges to greedy decoding.

At T = 2.0, divide by 2, halving logits: cat's → 1.5, sat's → 0.75. Differences between logits shrink. Softmax produces flatter distribution — cat drops to ~0.35, other tokens get more mass. More randomness, more surprise. As T→∞, every token equally likely.

The formula is: P(token_i) = exp(logit_i / T) / Σ exp(logit_j / T)

Analogy: temperature = contrast knob on photograph. Low T makes bright spots blinding, shadows pitch black — dominant tokens absorb all probability. High T brings out detail in shadows — rare tokens get fighting chance.

In practice: T=0.7–1.0 for creative text. T=0.1–0.3 for factual tasks where model should stick to most confident predictions. Problem: even at good temperature, still sampling from entire vocabulary. Includes garbage tokens — vanishingly small probability, no reasonable continuation would ever include them.

Top-K: Drawing a Hard Line

Top-k sampling addresses garbage problem with hard line. Before sampling, find k tokens with highest probabilities, set everything else to zero, renormalize remaining to sum to one. Sample from k survivors.

Toy example with k = 3:

Original probabilities:
  the  → 0.02    cat  → 0.70    sat  → 0.15
  on   → 0.08    mat  → 0.05

Top-3 survivors: cat (0.70), sat (0.15), on (0.08)
Total surviving mass: 0.93

After renormalization:
  cat  → 0.70 / 0.93 ≈ 0.753
  sat  → 0.15 / 0.93 ≈ 0.161
  on   → 0.08 / 0.93 ≈ 0.086
  the  → 0    mat  → 0

Now the and mat can't be picked. Only sample from top three. Eliminates risk of drawing nonsensical token from distribution tail.

But k is fixed — that's the limitation. Sometimes model extremely confident: cat has 0.95, everything else is noise. With k=50, still considering 49 garbage tokens. Other times model genuinely uncertain — 200 tokens all have reasonable probability, k=50 cuts off valid continuations. Right k varies at every generation step. No perfect fixed number.

Top-P (Nucleus Sampling): A Smarter Line

Top-p sampling, aka nucleus sampling (Holtzman et al., 2020), fixes top-k's rigidity: instead of fixing number of tokens, fix total probability mass you're willing to consider.

Sort tokens by probability highest to lowest. Walk down list, accumulating probability. Stop when cumulative probability hits threshold p. Everything included = nucleus — smallest set of tokens whose probabilities sum to at least p.

Trace with p = 0.90:

Sorted probabilities:
  cat  → 0.70  (cumulative: 0.70)  ← included
  sat  → 0.15  (cumulative: 0.85)  ← included
  on   → 0.08  (cumulative: 0.93)  ← included (this pushes us past 0.90)
  mat  → 0.05  (cumulative: 0.98)  ← excluded
  the  → 0.02  (cumulative: 1.00)  ← excluded

Nucleus contains three tokens: cat, sat, on. Renormalize and sample.

Why smarter than top-k. Different step where model very confident:

  cat  → 0.92  (cumulative: 0.92)  ← included, and we've hit 0.90
  sat  → 0.04  ← excluded
  on   → 0.02  ← excluded
  mat  → 0.01  ← excluded
  the  → 0.01  ← excluded

With p=0.90, nucleus is single token: cat. Almost certainly pick cat — model was confident. But top-k=50 would still consider four other tokens unnecessarily. Opposite scenario: model uncertain, 200 tokens each at 0.005 probability — top-p includes all 200, top-k=50 chops off 150 valid options.

Top-p=0.95 is common default, works well across many tasks. Can combine top-p with temperature — apply temperature first to reshape distribution, then top-p to trim tails.

Neither temperature nor top-p addresses specific maddening failure mode: repetition.

Repetition Penalties: Breaking the Loop

Even with good sampling settings, models get stuck in loops. "The cat sat on the mat. The cat sat on the mat. The cat sat on the mat." Model keeps generating same phrase because at every step, those tokens genuinely have high probability given context — context that now contains phrase it keeps repeating.

Repetition penalty fights this by checking which tokens already appeared in generated sequence and discouraging them. For each already-generated token: if logit positive, divide by penalty factor; if negative, multiply by it. Either way, token becomes less attractive.

Penalty 1.0 = no change. Values 1.1–1.3 work well. Careful: too high and model starts avoiding common words it legitimately needs. Get unnatural thesaurus-stuffed prose: "the feline rested upon the rug. said creature positioned itself atop the floor covering." Worse than repetition.

no_repeat_ngram_size is blunter: flat-out forbids repeating any n-gram of specified length. Set to 2 = never produce same two-word phrase twice. Effective for summarization, too aggressive for dialogue where repeating "I don't know" is natural.

OpenAI's API offers two related knobs: frequency_penalty (penalizes tokens proportional to appearance count) and presence_penalty (penalizes based on whether appeared at all, regardless of count). Distinction matters. Frequency penalty punishes "the" more each time it appears. Presence penalty punishes "the" same amount whether appeared once or fifty times. For creative generation, small presence penalty (0.1–0.6) encourages exploring new topics without over-penalizing common function words.

All sampling-based approaches share fundamental trait: stochastic. Same prompt twice → different outputs. For creative writing, brainstorming — feature. For translation, code generation, structured data extraction — want single best output, not random good one. Enter beam search.

Beam Search: Scouting Multiple Paths

Beam search = sending scouts down a trail.

Remember greedy hiker picking steepest trail at every fork? Beam search sends team of hikers — say five. At each fork, all five explore their best option. Group compares notes, keeps only five with best overall trails (measured by total path quality from start). Others eliminated. Next fork: surviving five explore again, compare again, prune again.

Number of hikers (candidates) = beam width b. With b=1, beam search is greedy decoding. With b = vocabulary size, exhaustive search (computationally impossible for real vocabularies). Values 4–10 typical in practice.

Trace toy example with beam width b = 2:

Step 1. Prompt: "the"
  Model output: cat (0.70), sat (0.15), on (0.08), mat (0.05), the (0.02)
  Keep top-2: "the cat" (score: log 0.70 = -0.36)
               "the sat" (score: log 0.15 = -1.90)

Step 2. Expand each beam:
  "the cat" → cat (0.05), sat (0.60), on (0.25), mat (0.08), the (0.02)
  "the sat" → cat (0.10), sat (0.05), on (0.70), mat (0.10), the (0.05)

  All candidates with cumulative scores:
    "the cat sat"  → -0.36 + log(0.60) = -0.87
    "the cat on"   → -0.36 + log(0.25) = -1.75
    "the sat on"   → -1.90 + log(0.70) = -2.26
    "the cat mat"  → -0.36 + log(0.08) = -2.86
    ... (rest are worse)

  Keep top-2: "the cat sat" (-0.87), "the cat on" (-1.75)

"the sat on" eliminated even though "on" had high probability in that context — cumulative score from bad first step dragged it down. Power of beam search: evaluates entire sequences, not individual steps.

Subtlety: without adjustment, beam search favors shorter sequences. Adding log probabilities means longer sequences always have lower scores. Length penalty corrects this — divide score by sequence length raised to power α. With α=0.7 (common), penalty softly encourages longer outputs without forcing them.

Beam search excels at tasks with "correct" answer — translation, summarization, code completion. For open-ended creative writing, produces safe, generic prose. Gravitates toward high-probability sequences, which are by definition least surprising. Most interesting writing involves lower-probability choices that pay off later — beam search with small width never finds those paths.

In production, systems combine strategies: beam search with small width for translation, top-p with temperature for chatbots. Some use beam search to generate candidates then rescore with separate quality metric. No universal answer — right decoding strategy depends on what you're generating.

Machine Translation: The Original Sequence Problem

Translation is where seq2seq models cut their teeth. Attention mechanism invented to solve translation problem. Transformer introduced in translation paper. "T" in GPT = "Transformer," which exists because of translation research. Understanding history explains why modern NLP looks the way it does.

Earliest attempts (1950s–1980s): rule-based. Linguists hand-wrote thousands of grammar rules, morphological transformations, bilingual dictionaries. Translate English→Russian: parse English into syntax tree, apply transformation rules for Russian syntax, look up words in dictionary. Worked for narrow domains like weather reports. Fell apart everywhere else. Idioms don't translate literally. Word order varies wildly across languages. "Bank" = riverbank or financial institution depending on context — rules can't resolve ambiguity reliably. Every new language pair needed completely new rule set.

Statistical MT (1990s–2015) replaced handcrafted rules with learned probabilities. Key insight: noisy channel model — assume foreign sentence is "corrupted" English sentence, find English sentence most likely to produce it. Math: argmax_e P(e|f) = argmax_e P(f|e) · P(e). P(f|e) (translation model) learned from parallel corpora — aligned sentence pairs in both languages. P(e) (language model) learned from English text alone, ensuring fluency. Phrase-based SMT extended word-level to phrase-level alignment — "in spite of" translates to "malgré" as unit. Google Translate used phrase-based SMT 2006–2016.

Neural MT (2014–present) replaced entire statistical pipeline with single neural network. First version (Sutskever et al., 2014): encoder-decoder architecture — RNN reads source sentence, compresses into single fixed-length vector, second RNN generates target from that vector. Fatal flaw: all information about 40-word sentence must pass through one vector. Short sentences fine. Long sentences fell apart — decoder couldn't remember beginning by time it reached end.

Attention (Bahdanau et al., 2015) solved this — decoder peeks back at every encoder state at every generation step. Instead of one compressed vector, decoder gets weighted view of entire source sentence, weights shifting based on current target word. Breakthrough that changed everything. Transformer (Vaswani et al., 2017) took further, replacing RNNs entirely with self-attention and cross-attention. Every modern language model descends from this architecture.

Parallel Corpora and the Training Data

Parallel corpus = dataset of sentence pairs: same meaning in two languages. European Parliament proceedings (Europarl), United Nations parallel corpus, web-crawled datasets like ParaCrawl — millions of pairs. Fuel for training translation models.

Catch: high-quality parallel data scarce for most language pairs. Abundant English↔French data (EU translating parliamentary proceedings for decades). English↔Yoruba? English↔Quechua? Almost nothing. Data scarcity for low-resource languages = central challenge in translation, drives architectural decisions in modern multilingual models.

Back-translation: train target→source model, translate monolingual target-language text back to source, use synthetic pairs as additional training data. Translations imperfect, but model still learns useful patterns. Introduced by Sennrich et al. (2016), remains most effective way to overcome data scarcity.

Measuring Translation: BLEU, METEOR, chrF, COMET

BLEU score of 0.32 feels unanchored without context. Here's how to interpret.

BLEU (Bilingual Evaluation Understudy) — oldest, most widely reported automatic metric for translation quality. Measures n-gram precision: what fraction of n-grams in machine translation also appear in human reference?

Compute by hand. Model translates French sentence, produces: "The cat is sitting on the mat." Reference: "The cat sat on the mat."

Machine output:  The cat is sitting on the mat .
Reference:       The cat sat on the mat .

1-gram matches: The ✓, cat ✓, is ✗, sitting ✗, on ✓, the ✓, mat ✓, . ✓
  → 6 out of 8 = 0.75

2-gram matches: "The cat" ✓, "cat is" ✗, "is sitting" ✗, "sitting on" ✗,
                "on the" ✓, "the mat" ✓, "mat ." ✓
  → 4 out of 7 = 0.57

3-gram matches: "The cat is" ✗, "cat is sitting" ✗, "is sitting on" ✗,
                "sitting on the" ✗, "on the mat" ✓, "the mat ." ✓
  → 2 out of 6 = 0.33

4-gram matches: "The cat is sitting" ✗, "cat is sitting on" ✗,
                "is sitting on the" ✗, "sitting on the mat" ✗,
                "on the mat ." ✓
  → 1 out of 5 = 0.20

BLEU takes geometric mean of 1-gram through 4-gram precisions, multiplies by brevity penalty (punishes too-short translations — could cheat by outputting single high-confidence word). Final score: ~0.41 here. Decent for single sentence.

BLEU ranges 0–1 (often reported 0–100). BLEU 30+ = generally understandable. 50+ = high quality. Human translators score 60–80 against other humans — not 100, because many valid ways to translate any sentence, no two translators make same choices.

BLEU's virtue: fast, cheap, reproducible. Flaw: only captures surface-level word overlap. Two equally good translations using different vocabulary score poorly against each other. "The automobile halted" and "The car stopped" share almost no n-grams but mean same thing.

METEOR (Metric for Evaluation of Translation with Explicit ORdering) addresses this — incorporates synonyms, stems, paraphrases. Aligns words using exact matches first, then stems ("running" matches "ran"), then synonyms (WordNet). Computes both precision and recall (BLEU only uses precision), applies penalty for word-order differences. Correlates better with human judgments than BLEU, especially sentence-level.

chrF (character n-gram F-score) — different approach: works at character level instead of word level. Computes F-score over character n-grams. Handles morphologically rich languages far better than BLEU. In German, Finnish, Turkish — single word takes dozens of inflected forms. "Translated" and "translating" share zero word-level n-grams but many character n-grams ("translat", "ranslati", "anslatin"...). chrF captures that overlap. Also works for languages without clear word boundaries (Chinese) where tokenization itself is contentious.

COMET (Crosslingual Optimized Metric for Evaluation of Translation) — newest generation: learned metric. Feeds source sentence, machine translation, reference into multilingual neural encoder, predicts quality score. Trained on human quality judgments, captures meaning — paraphrases score well without word overlap. Correlates significantly better with human evaluation than BLEU, chrF, METEOR. Cost: needs GPU, black box.

Recommendation: report BLEU for comparability with literature (lingua franca of MT evaluation). Add chrF for morphologically rich target languages. Use COMET for most accurate automatic quality assessment. For high-stakes: include human evaluation — no automatic metric perfectly proxies whether translation communicates right meaning.

Multilingual Models: One Model, Many Languages

Early neural MT required separate model per language pair. English→French: one model. English→German: another. French→German: third. With 100 languages = 9,900 separate models. Unsustainable.

Breakthrough: languages share deep structure. Agent-action-object exists across almost every human language, even when surface word order differs. Model learning English→French picks up patterns — mapping verb tenses, handling articles, adjective placement — that transfer to English→Spanish because French and Spanish share Latin roots and similar grammar.

mBART (multilingual BART) extended this — pretrained single encoder-decoder transformer with denoising objective across 25 languages: corrupt text in any language, reconstruct it. Forced model to learn internal structure of each language. Fine-tuning for translation then needed far fewer parallel pairs — model already understood each language. Target-language tag (special token like <2fr>) told decoder which language to generate.

NLLB (No Language Left Behind), Meta 2022, scaled to 200+ languages — including low-resource languages previous systems ignored. Uses sparse mixture-of-experts: different expert sub-networks activate for different language families. Adding Zulu doesn't degrade French — they use different experts. For many low-resource pairs, NLLB improved BLEU by 44%+ vs previous SOTA. Not incremental — paradigm shift for communities whose languages were invisible to translation technology.

Surprising capability: zero-shot translation — translating between language pair model never explicitly trained on. Model learned English↔French and English↔German, can it do French→German directly? Often yes. Shared internal representation acts as universal meaning space — interlingua — bridging languages without explicit bridge data. Quality typically lower than trained directions. Model sometimes produces off-target translation: output in wrong language, usually defaulting to English (dominated training data).

Why zero-shot translation works remains somewhat mysterious. Conventional explanation — "model learns shared meaning space" — feels right but vague. Working at all without single French-German pair is remarkable. Failing by slipping into English reminds us: pattern matchers, not understanding machines.

Constrained Generation and Structured Output

So far, generation treated as open-ended: give model prompt, let it produce whatever tokens, shape output with temperature and sampling. But many real applications need output following specific structure — valid JSON, particular schema, SQL queries, function calls. Model needs guardrails.

Constrained decoding enforces structure at token level. Each generation step, before sampling, check which tokens valid according to grammar — formal specification of allowed sequences. Generating JSON and produced {"name": ": only valid next tokens are string-value characters. Closing brace syntactically invalid here, probability set to zero. Model forced to follow grammar.

Libraries like Outlines and Microsoft Guidance implement this by compiling grammar (CFG, regex, or JSON schema) into finite state machine. Each step, FSM says which tokens valid, only those get nonzero probability. Output guaranteed valid — not "usually valid" or "valid if you ask nicely in prompt," but mathematically guaranteed.

Different philosophy from prompt engineering ("Please return valid JSON" and hope for best). Constrained decoding doesn't hope — enforces. OpenAI's function calling and structured outputs use similar ideas under hood — model output constrained to match specified schema.

Tradeoff: speed. Maintaining grammar state and masking invalid tokens each step adds overhead. For complex grammars, significant. Subtler cost: constraining model can force awkward phrasings or suboptimal token sequences it wouldn't choose freely. Model might "know" better way to express something, but grammar won't allow it.

Watermarking Generated Text

As language models produce more text we read, natural question: can we tell if text was written by model or human? Watermarking embeds detectable but invisible signal in generated text, enabling identification of machine-generated content after the fact.

Most influential scheme (Kirchenbauer et al., 2023) works at generation time. Core idea: each step, PRNG (seeded by previous token + secret key) splits vocabulary into two sets: green list and red list (~50/50). Before sampling, add small bonus to logits of all green-list tokens, making them slightly more likely to be chosen.

Any single token looks natural — bias small enough that no individual word seems suspicious. But over hundreds of tokens, statistical signal accumulates. Detector knowing secret key reconstructs green/red split for each position, counts how many tokens landed in green lists, compares to expected rate. Watermarked text shows statistically significant overrepresentation of green-list tokens. Unwatermarked text won't.

Elegant idea. Robustness: minor edits (typos, few word changes) don't destroy watermark — signal distributed across hundreds of tokens. Heavy paraphrasing or running through separate model can break it. Length requirement (~200+ tokens for reliable detection) means doesn't work for short outputs. Only works if generator embeds watermark — can't retroactively watermark text generated without it.

The Unsolved Problem: Evaluating Open-Ended Generation

For translation: BLEU, chrF, COMET — imperfect but useful. For open-ended generation — stories, dialogue, essays, brainstorming — almost nothing works well automatically. Genuinely hard unsolved problem in NLP. Anyone claiming good solution is overselling.

Fundamental challenge: no reference to compare against. Ask model "write short story about lonely astronaut" — infinitely many good stories possible. What counts as "good"? Fluency? Coherence? Creativity? Emotional resonance? Factual accuracy? Different dimensions, often conflicting — most creative response may sacrifice coherence, most coherent may be boring.

Perplexity measures how surprised model is by text — exponentiated average negative log-likelihood per token. Lower = model found text more predictable. Useful for comparing language models on same test set, but tells nothing about whether generated text is good. Low perplexity model can still produce repetitive, dull output. Creative, surprising text might have high perplexity precisely because it's interesting.

MAUVE (Pillutla et al., 2021) — distributional approach: measures how similar distribution of generated text is to human-written text in neural embedding space. Model outputs in same region as human writing → high MAUVE. Captures both quality and diversity — model producing one great sentence repeatedly scores low on diversity; diverse garbage scores low on quality. Correlates better with human judgments than perplexity for open-ended generation.

Distinct-n measures diversity: unique n-grams as fraction of total n-grams. Repetitive model → low distinct-1, distinct-2 scores. Self-BLEU measures diversity by computing BLEU between pairs of generated outputs — similar outputs → high self-BLEU (bad).

All proxies. Honest assessment: for open-ended generation, human evaluation remains gold standard. Multiple evaluators rate text on fluency, coherence, informativeness, engagement. Use Likert scales or pairwise comparisons (A vs B). Expensive, slow, noisy — evaluators disagree, same evaluator rates differently on different days. But closest to ground truth for whether generated text is good.

Evaluation problem forces confronting what "good text" means. No formula for it. Might never have one. Might be okay — something irreducibly human about judging language quality that no metric has captured yet.
