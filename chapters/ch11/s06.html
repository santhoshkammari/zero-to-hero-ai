Text Generation

Chapter 11: Natural Language Processing 4 subtopics
TL;DR

Text generation is next-token prediction in a loop. The model outputs probabilities over the vocabulary, and your decoding strategy decides which token to actually pick. Greedy takes the top-1 every time (fast, boring, repetitive). Temperature reshapes the probability curve. Top-k and top-p prune unlikely tokens before sampling. Beam search keeps multiple hypotheses alive. Mastering these knobs is the difference between a chatbot that sounds robotic and one that sounds human.

Language Modeling

Next Token Prediction

Every modern text generator — GPT, LLaMA, Mistral — does exactly one thing at inference time: predict the next token. You feed in a sequence of tokens, and the model returns a probability distribution over its entire vocabulary (often 32,000–128,000 tokens). The token with the highest probability is the model's "best guess" for what comes next. Then you append that token to your input and repeat. This loop is autoregressive generation: each new token depends on all the tokens generated before it.

Why does this work so well? During training, the model saw billions of sentences and learned statistical patterns — that "the cat sat on the" is very likely followed by "mat" or "floor" or "couch." But it also learned higher-level patterns: narrative structure, argument flow, code syntax. The magic is that a single objective — predict the next token — forces the model to learn an enormous amount about language.

Here's the simplest possible generation loop with HuggingFace:

from transformers import AutoTokenizer, AutoModelForCausalLM
import torch

model_name = "gpt2"
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForCausalLM.from_pretrained(model_name)

prompt = "The best way to learn machine learning is"
input_ids = tokenizer.encode(prompt, return_tensors="pt")

# Manual autoregressive loop
for _ in range(50):
    with torch.no_grad():
        outputs = model(input_ids)
    # outputs.logits shape: (batch, seq_len, vocab_size)
    next_token_logits = outputs.logits[:, -1, :]  # last position
    next_token_id = torch.argmax(next_token_logits, dim=-1, keepdim=True)
    input_ids = torch.cat([input_ids, next_token_id], dim=-1)

print(tokenizer.decode(input_ids[0]))

That torch.argmax on the last line is greedy decoding — always picking the single most likely token. It's the simplest strategy, and it's often the worst.

Perplexity Metric

Perplexity measures how "surprised" a language model is by a text. Formally, it's the exponentiated average negative log-likelihood per token. If a model assigns high probability to each actual next token, perplexity is low. If it's frequently surprised, perplexity is high. A perplexity of 1 means the model predicted every token perfectly. GPT-2 achieved a perplexity of about 30 on standard benchmarks; GPT-3 brought it down to around 20.

In practice, you use perplexity to compare language models on the same test set. Lower is better. But beware: perplexity depends on the tokenizer. A model with a larger vocabulary might get lower perplexity simply because it has more granular tokens — not because it understands language better. Always compare models that share the same tokenization scheme.

import torch
from transformers import AutoModelForCausalLM, AutoTokenizer

model = AutoModelForCausalLM.from_pretrained("gpt2")
tokenizer = AutoTokenizer.from_pretrained("gpt2")

text = "Machine learning is the study of algorithms that improve through experience."
inputs = tokenizer(text, return_tensors="pt")

with torch.no_grad():
    outputs = model(**inputs, labels=inputs["input_ids"])
    # outputs.loss is the average cross-entropy per token
    perplexity = torch.exp(outputs.loss)

print(f"Perplexity: {perplexity.item():.2f}")

Autoregressive Generation

The term autoregressive means "self-feeding." Each generated token becomes part of the input for generating the next token. This creates a sequential dependency: token 50 depends on tokens 1–49. That's why generation is inherently slow — you can't parallelize across time steps the way you can during training (where you use teacher forcing and process all positions simultaneously).

This sequential bottleneck is why techniques like KV-caching exist. Without caching, you'd recompute attention over the entire sequence for every new token. With KV-cache, the model stores the key and value tensors from previous positions and only computes attention for the new token. This turns generation from O(n²) per token to O(n), which matters enormously for long outputs.

Decoding Strategies

The model gives you a probability distribution. The decoding strategy decides what you do with it. This choice dramatically affects output quality. Let's walk through each strategy, building from simplest to most sophisticated.

Greedy Decoding

Greedy decoding always picks the single highest-probability token at each step. It's fast and deterministic — same input always produces the same output. But it has a fatal flaw: it can't backtrack. If the globally best sequence requires picking a lower-probability token at step 5, greedy will never find it. It's stuck following the locally optimal path.

In practice, greedy decoding produces repetitive, bland text. You'll see loops like "I think that this is a very good idea. I think that this is a very good idea." The model keeps picking the same high-probability continuation because it never has a reason to deviate.

Temperature Sampling

Temperature is a scalar that reshapes the probability distribution before sampling. The logits (raw model outputs) are divided by the temperature value before the softmax. Temperature = 1.0 leaves the distribution unchanged. Temperature < 1.0 makes the distribution sharper (peaks get peakier, the model becomes more confident). Temperature > 1.0 flattens the distribution (more uniform, more random).

Mathematically: P(token_i) = exp(logit_i / T) / Σ exp(logit_j / T). When T approaches 0, this converges to greedy decoding. When T approaches infinity, every token becomes equally likely. In practice, temperatures between 0.7 and 1.0 work well for creative text, while 0.1–0.3 works for factual tasks where you want the model to stick to its most confident predictions.

import torch
import torch.nn.functional as F

def sample_with_temperature(logits, temperature=1.0):
    """Apply temperature and sample from the distribution."""
    scaled_logits = logits / temperature
    probs = F.softmax(scaled_logits, dim=-1)
    return torch.multinomial(probs, num_samples=1)

# Example: same logits, different temperatures
logits = torch.tensor([[2.0, 1.0, 0.5, 0.1, -1.0]])

for temp in [0.1, 0.5, 1.0, 2.0]:
    probs = F.softmax(logits / temp, dim=-1)
    print(f"T={temp}: {probs.numpy().round(3)}")
# T=0.1: almost all mass on token 0
# T=2.0: much more spread out

Top-K Sampling

Top-k sampling restricts the candidate pool to the k most likely tokens, then renormalizes and samples from just those k. If k=50, the model first finds the 50 highest-probability tokens, throws away the rest, redistributes the probability mass among the survivors, and then randomly samples.

The problem with top-k is that k is fixed. Sometimes the model is very confident and only 3 tokens make sense — but you're still sampling from 50. Other times, 200 tokens are all reasonable continuations but you're cutting off at 50. The "right" k varies per step, and there's no perfect fixed value.

Top-P (Nucleus) Sampling

Top-p sampling (also called nucleus sampling) fixes top-k's rigidity. Instead of a fixed count, you set a cumulative probability threshold p. Sort tokens by probability, then include tokens from the top until their cumulative probability reaches p. If p=0.9, you keep the smallest set of tokens whose probabilities sum to at least 0.9.

This adapts automatically. When the model is confident, the nucleus might contain just 2–3 tokens. When the model is uncertain, it might contain hundreds. Top-p = 0.95 is a common default that works well across many tasks.

from transformers import AutoTokenizer, AutoModelForCausalLM

tokenizer = AutoTokenizer.from_pretrained("gpt2")
model = AutoModelForCausalLM.from_pretrained("gpt2")

prompt = "In a world where AI can"
input_ids = tokenizer.encode(prompt, return_tensors="pt")

# Compare different strategies using HuggingFace's generate()
strategies = {
    "greedy": dict(max_new_tokens=60, do_sample=False),
    "temperature_0.7": dict(max_new_tokens=60, do_sample=True, temperature=0.7),
    "top_k_50": dict(max_new_tokens=60, do_sample=True, top_k=50),
    "top_p_0.95": dict(max_new_tokens=60, do_sample=True, top_p=0.95),
    "combined": dict(max_new_tokens=60, do_sample=True, 
                     temperature=0.8, top_k=50, top_p=0.95),
}

for name, kwargs in strategies.items():
    output = model.generate(input_ids, **kwargs, pad_token_id=tokenizer.eos_token_id)
    text = tokenizer.decode(output[0], skip_special_tokens=True)
    print(f"\n[{name}]\n{text}")

Repetition Penalty

Repetition penalty discourages the model from repeating tokens it has already generated. For each token that appears in the existing sequence, its logit is divided by the penalty factor (if the logit is positive) or multiplied by it (if negative). A penalty of 1.0 means no change. Values between 1.1 and 1.3 work well in practice. Too high and the model starts avoiding common words it legitimately needs to reuse — producing unnatural, synonym-stuffed prose.

Beam Search

Beam search is a compromise between greedy (keep 1 hypothesis) and exhaustive search (keep all hypotheses). You maintain the top-b candidates at each step, where b is the beam width. At each step, you expand all b candidates, score the results, and keep the top-b again. With a beam width of 5, you're exploring 5 paths simultaneously.

Beam search tends to produce more coherent and higher-likelihood sequences than greedy decoding. It's particularly good for tasks with a "right answer" — machine translation, summarization, code generation. But for open-ended creative writing, beam search can be too conservative. It gravitates toward high-probability, generic responses because the scoring rewards likelihood over surprise.

# Beam search with HuggingFace
output = model.generate(
    input_ids,
    max_new_tokens=60,
    num_beams=5,           # beam width
    no_repeat_ngram_size=2, # block repeated 2-grams
    early_stopping=True,
    pad_token_id=tokenizer.eos_token_id
)
print(tokenizer.decode(output[0], skip_special_tokens=True))
Beam Search vs. Sampling

Use beam search when correctness matters (translation, summarization, structured output). Use sampling (top-p + temperature) when creativity and diversity matter (story writing, brainstorming, dialogue). Many production systems combine both: beam search with a small beam width plus a mild temperature.

Summarization

Extractive Summarization

Extractive summarization selects and concatenates the most important sentences from the original document — no new words are generated. Think of it as highlighting with a marker. Algorithms like TextRank build a graph of sentences (edges weighted by similarity), then use PageRank-style scoring to find the most "central" sentences. It's simple, faithful to the source, and can't hallucinate — but the results often read choppily because the selected sentences weren't written to flow together.

Abstractive Summarization

Abstractive summarization generates new text that captures the meaning of the source. This is what models like BART, T5, and Pegasus do. They read the full document and produce a condensed version in their own words — paraphrasing, merging ideas, dropping details. The results read more naturally, but the model can hallucinate facts that weren't in the source. This is the fundamental tradeoff: extractive is safe but clunky; abstractive is fluent but risky.

from transformers import pipeline

summarizer = pipeline("summarization", model="facebook/bart-large-cnn")

article = """
Machine learning engineers spend most of their time on data preparation,
not model building. A recent survey found that data cleaning, feature 
engineering, and data collection account for roughly 80% of the effort 
in a typical ML project. Model selection and training, the part that 
gets all the attention, takes up less than 20%. This suggests that 
investing in better data tools and processes may yield higher returns 
than chasing marginal improvements in model architecture.
"""

summary = summarizer(article, max_length=50, min_length=20, do_sample=False)
print(summary[0]["summary_text"])

ROUGE Metrics

ROUGE (Recall-Oriented Understudy for Gisting Evaluation) measures overlap between a generated summary and reference summaries. ROUGE-1 counts unigram overlap, ROUGE-2 counts bigram overlap, and ROUGE-L measures the longest common subsequence. Higher is better. ROUGE is cheap to compute and widely used, but it only measures surface-level word overlap — a summary that perfectly paraphrases with synonyms would score poorly. That's why newer metrics like BERTScore exist (we'll cover those in Section 8).

Question Answering

Extractive QA

Extractive QA finds the answer span within a given context passage. The model reads a question and a passage, then predicts start and end positions marking where the answer lives in the text. BERT was the breakthrough here — fine-tuned on SQuAD (Stanford Question Answering Dataset), it learns to highlight the exact substring that answers the question. No generation, no hallucination risk — but the answer must literally appear in the context.

from transformers import pipeline

qa = pipeline("question-answering", model="deepset/roberta-base-squad2")

context = """
Transformers were introduced in 2017 by Vaswani et al. in the paper
'Attention Is All You Need'. The architecture replaced recurrent layers 
entirely with self-attention mechanisms, enabling much better parallelization 
during training. BERT, released by Google in 2018, applied the transformer 
encoder to produce bidirectional contextualized embeddings.
"""

result = qa(question="When were transformers introduced?", context=context)
print(f"Answer: {result['answer']} (score: {result['score']:.3f})")
# Answer: 2017 (score: 0.952)

Generative QA

Generative QA produces the answer as free-form text rather than extracting a span. Models like T5 or GPT are fine-tuned to read a question (and optionally a context) and generate an answer from scratch. This is more flexible — the answer doesn't need to appear verbatim in any passage — but it introduces the risk of hallucination. The model might confidently generate a plausible-sounding but factually wrong answer.

Open-Domain QA

Open-domain QA answers questions without being given a specific context passage. The system must first retrieve relevant documents from a large corpus (like Wikipedia), then extract or generate an answer from the retrieved text. This is exactly where RAG (Retrieval-Augmented Generation) shines — and we'll cover that in depth in the next section. The retrieve-then-read pipeline is the backbone of most modern QA systems.

Reading Comprehension and SQuAD

The SQuAD (Stanford Question Answering Dataset) benchmark has two versions. SQuAD 1.1 contains 100,000+ questions where the answer always exists in the passage. SQuAD 2.0 adds 50,000 unanswerable questions — the model must also learn to say "I don't know." This was a crucial addition because real-world questions often can't be answered from the given context, and a system that always guesses is worse than useless. Human performance on SQuAD 2.0 is an F1 of about 86.8 — modern transformers exceed 90.

The Generation Quality Checklist

When deploying text generation in production, always check: (1) Does the output contain repetition loops? Add repetition penalty. (2) Is it too generic? Increase temperature or top-p. (3) Is it too random or incoherent? Decrease temperature or lower top-p. (4) Does it hallucinate facts? Consider constrained decoding or RAG. (5) Is it too slow? Use KV-caching and consider smaller models or speculative decoding.

Constrained Decoding

Constrained decoding restricts what the model can generate at each step. Instead of sampling freely from the full vocabulary, you enforce rules — the output must be valid JSON, must contain certain keywords, must follow a grammar. Libraries like outlines and guidance implement this by masking logits: at each step, tokens that would violate the constraint get their logits set to negative infinity, so they can never be selected.

This is critical for production systems. If your API needs to return structured JSON, you can't afford a model that occasionally outputs malformed responses. Constrained decoding guarantees structural validity while letting the model fill in the content freely within those constraints. It's the best of both worlds: creative content, rigid structure.

# Conceptual example of constrained generation
# Using HuggingFace's generate with a logits processor
from transformers import LogitsProcessor

class ForceJSONLogitsProcessor(LogitsProcessor):
    """Simple example: force output to start with '{'"""
    def __call__(self, input_ids, scores):
        # On first generated token, only allow '{'
        if input_ids.shape[1] == 1:  # first token
            brace_id = tokenizer.encode("{", add_special_tokens=False)[0]
            mask = torch.full_like(scores, float("-inf"))
            mask[:, brace_id] = scores[:, brace_id]
            return mask
        return scores

In practice, tools like the outlines library handle this far more elegantly — you define a JSON schema or regular expression, and the library automatically computes the valid token set at each step. This has become a standard technique for building reliable AI APIs.
