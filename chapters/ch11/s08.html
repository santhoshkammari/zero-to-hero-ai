NLP Practical Methods

Chapter 11: Natural Language Processing 5 subtopics
TL;DR

The HuggingFace ecosystem is the standard toolkit for practical NLP. The transformers library gives you pretrained models with a one-line pipeline API. The datasets library handles loading, streaming, and preprocessing data. Model cards document what a model can and can't do. Evaluation metrics like BLEU, ROUGE, and BERTScore tell you how good your outputs are. This section is a hands-on tutorial — by the end, you'll know how to load a model, run inference, fine-tune on your data, and evaluate the results.

HuggingFace Transformers Library

Why HuggingFace

Before HuggingFace, using a pretrained NLP model meant hunting down the author's GitHub repo, installing their custom dependencies, converting checkpoints between frameworks, and praying the code still ran. HuggingFace standardized all of this. Every model — BERT, GPT-2, T5, LLaMA, Mistral, thousands more — uses the same API: AutoModel.from_pretrained("model-name"). Same interface for PyTorch and TensorFlow. Same tokenizer API. Same configuration format. This consistency is why HuggingFace became the de facto standard.

The library has three layers, and understanding them will save you hours of confusion:

Pipelines are the highest level. One line of code, zero configuration. You say what task you want (sentiment analysis, translation, summarization) and it handles tokenization, model inference, and post-processing automatically. Use pipelines for prototyping and demos.

Auto classes (AutoTokenizer, AutoModel, AutoModelForSequenceClassification, etc.) are the middle level. You control the tokenization and inference steps separately, which lets you customize the pipeline. Use these when you need to modify how inputs are processed or outputs are decoded.

Specific model classes (BertModel, GPT2LMHeadModel, etc.) are the lowest level. You use the exact architecture class. This gives you full control but locks you to a specific model type. Use these only when you need architecture-specific features.

from transformers import (
    pipeline,
    AutoTokenizer,
    AutoModelForSequenceClassification,
)
import torch

# === LEVEL 1: Pipeline (one line) ===
classifier = pipeline("sentiment-analysis")
result = classifier("I love learning about NLP!")
print(result)
# [{'label': 'POSITIVE', 'score': 0.9998}]

# === LEVEL 2: Auto classes (full control) ===
model_name = "distilbert-base-uncased-finetuned-sst-2-english"
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForSequenceClassification.from_pretrained(model_name)

text = "I love learning about NLP!"
inputs = tokenizer(text, return_tensors="pt", padding=True, truncation=True)
print(f"Token IDs: {inputs['input_ids']}")
print(f"Tokens: {tokenizer.convert_ids_to_tokens(inputs['input_ids'][0])}")

with torch.no_grad():
    outputs = model(**inputs)
    logits = outputs.logits
    probs = torch.softmax(logits, dim=-1)
    predicted_class = torch.argmax(probs, dim=-1).item()

labels = model.config.id2label
print(f"Prediction: {labels[predicted_class]} ({probs[0][predicted_class]:.4f})")

Text Pipelines

HuggingFace provides pipelines for every major NLP task. Here's a quick tour of the most useful ones:

from transformers import pipeline

# --- Sentiment Analysis ---
sentiment = pipeline("sentiment-analysis")
print(sentiment("This movie was absolutely terrible."))
# [{'label': 'NEGATIVE', 'score': 0.9998}]

# --- Named Entity Recognition ---
ner = pipeline("ner", grouped_entities=True)
entities = ner("Elon Musk founded SpaceX in Hawthorne, California.")
for ent in entities:
    print(f"  {ent['word']:20s} → {ent['entity_group']} ({ent['score']:.3f})")

# --- Text Generation ---
generator = pipeline("text-generation", model="gpt2")
output = generator("The future of AI is", max_new_tokens=50, do_sample=True, 
                    temperature=0.7, top_p=0.9)
print(output[0]["generated_text"])

# --- Summarization ---
summarizer = pipeline("summarization", model="facebook/bart-large-cnn")
article = """
Artificial intelligence has transformed many industries over the past decade.
Healthcare uses AI for diagnosis and drug discovery. Finance uses it for 
fraud detection and algorithmic trading. Manufacturing uses it for quality 
control and predictive maintenance. The common thread is that AI excels at 
finding patterns in large datasets that humans would miss.
"""
summary = summarizer(article, max_length=50, min_length=20)
print(summary[0]["summary_text"])

# --- Translation ---
translator = pipeline("translation_en_to_fr", model="Helsinki-NLP/opus-mt-en-fr")
print(translator("Machine learning is fascinating."))

# --- Question Answering ---
qa = pipeline("question-answering")
result = qa(
    question="What does AI excel at?",
    context="AI excels at finding patterns in large datasets that humans would miss."
)
print(f"Answer: {result['answer']} (score: {result['score']:.3f})")

# --- Zero-Shot Classification ---
zero_shot = pipeline("zero-shot-classification")
result = zero_shot(
    "I need to book a flight to New York next Tuesday",
    candidate_labels=["travel", "finance", "food", "technology"]
)
print(f"Best label: {result['labels'][0]} ({result['scores'][0]:.3f})")

# --- Fill Mask ---
fill = pipeline("fill-mask", model="bert-base-uncased")
results = fill("Machine learning is a branch of [MASK] intelligence.")
for r in results[:3]:
    print(f"  {r['token_str']:15s} ({r['score']:.3f})")
Choosing the Right Model

Start with the default pipeline model for prototyping. When you need better accuracy, browse the HuggingFace Model Hub and filter by task. Sort by downloads or likes to find community-proven models. For production, consider model size — distilbert is 40% smaller and 60% faster than bert-base with 97% of the accuracy. Always benchmark on YOUR data, not just published benchmarks.

Model Cards

A model card is a standardized document that accompanies every model on HuggingFace Hub. It describes what the model does, what data it was trained on, its intended use cases, known limitations, and evaluation results. Think of it as the nutrition label for ML models.

Before using any model in production, read its model card. Key things to check: (1) What data was it trained on? If it was trained on English Wikipedia, don't expect it to work well on medical jargon or slang. (2) What are the intended use cases? A sentiment model trained on movie reviews may fail on financial text. (3) What are the known biases? Models trained on internet text inherit societal biases. (4) What's the license? Some models are research-only, not cleared for commercial use.

from huggingface_hub import ModelCard

# Load and inspect a model card programmatically
card = ModelCard.load("distilbert-base-uncased-finetuned-sst-2-english")
print(card.text[:500])

# You can also check model metadata
from transformers import AutoConfig
config = AutoConfig.from_pretrained("distilbert-base-uncased-finetuned-sst-2-english")
print(f"Architecture: {config.architectures}")
print(f"Vocab size: {config.vocab_size}")
print(f"Hidden size: {config.hidden_size}")
print(f"Num layers: {config.num_hidden_layers}")
print(f"Num heads: {config.num_attention_heads}")

Datasets Library

Loading and Exploring Datasets

The HuggingFace datasets library is the companion to transformers. It provides access to thousands of datasets in a unified format, with built-in streaming for datasets too large to fit in memory. It uses Apache Arrow under the hood, which means operations like filtering, mapping, and shuffling are memory-efficient — the data lives on disk and is memory-mapped, not loaded into RAM.

from datasets import load_dataset, list_datasets

# Load a dataset
dataset = load_dataset("imdb")
print(dataset)
# DatasetDict({
#     train: Dataset({features: ['text', 'label'], num_rows: 25000})
#     test: Dataset({features: ['text', 'label'], num_rows: 25000})
#     unsupervised: Dataset({features: ['text', 'label'], num_rows: 50000})
# })

# Inspect a sample
print(dataset["train"][0])
# {'text': 'I rented I AM CURIOUS...', 'label': 0}

# Check label distribution
import collections
labels = dataset["train"]["label"]
print(collections.Counter(labels))
# Counter({0: 12500, 1: 12500})  -- perfectly balanced

# Filter and transform
positive_reviews = dataset["train"].filter(lambda x: x["label"] == 1)
print(f"Positive reviews: {len(positive_reviews)}")

# Stream large datasets without downloading everything
large_dataset = load_dataset("c4", "en", split="train", streaming=True)
for i, example in enumerate(large_dataset):
    if i >= 3:
        break
    print(f"Sample {i}: {example['text'][:100]}...")

Preprocessing for Transformers

The most common preprocessing pattern: tokenize the text column and format the dataset for PyTorch or TensorFlow. The .map() method applies a function to every example (or batch), and it's smart about caching — if you run it twice with the same function, it loads from cache instead of recomputing.

from transformers import AutoTokenizer
from datasets import load_dataset

tokenizer = AutoTokenizer.from_pretrained("distilbert-base-uncased")
dataset = load_dataset("imdb", split="train[:1000]")  # small subset

def tokenize_function(examples):
    return tokenizer(
        examples["text"],
        padding="max_length",
        truncation=True,
        max_length=512,
    )

# Apply tokenization to all examples (batched for speed)
tokenized = dataset.map(tokenize_function, batched=True)
print(tokenized.column_names)
# ['text', 'label', 'input_ids', 'attention_mask']

# Set format for PyTorch
tokenized.set_format("torch", columns=["input_ids", "attention_mask", "label"])

# Now you can use it directly with a DataLoader
from torch.utils.data import DataLoader

loader = DataLoader(tokenized, batch_size=16, shuffle=True)
batch = next(iter(loader))
print(f"input_ids shape: {batch['input_ids'].shape}")
print(f"labels shape: {batch['label'].shape}")

Explainability for NLP

LIME for Text

LIME (Local Interpretable Model-agnostic Explanations) explains individual predictions by perturbing the input and observing how the output changes. For text, this means randomly removing words and checking whether the prediction changes. Words whose removal most changes the output are the most "important" for that prediction.

LIME is model-agnostic — it treats the model as a black box and only needs the predict function. This makes it work with any NLP model, from logistic regression to massive transformers. The downside: it's slow (many forward passes per explanation) and the explanations can be unstable — run it twice and you might get slightly different word importances.

Attention-Based Explanations

Transformers compute attention weights at every layer. It's tempting to use these as explanations: "the model attended to word X, so X must be important." Be careful. Research has shown that attention weights don't always correlate with feature importance. A head might attend to a word for syntactic reasons (it's a period, it's the [CLS] token) rather than semantic reasons. Attention is a description of information flow, not a causal explanation of the prediction. Use attention visualizations as one signal among many, not as ground truth.

Data Augmentation for NLP

Back-Translation

Back-translation generates paraphrases by translating text to another language and back. "The cat sat on the mat" → (translate to French) "Le chat s'est assis sur le tapis" → (translate back) "The cat sat on the carpet." The round-trip introduces natural variation while preserving meaning. This is one of the most effective augmentation techniques for NLP, especially for low-resource languages where labeled data is scarce.

Easy Data Augmentation (EDA)

EDA applies four simple operations: synonym replacement (swap words with WordNet synonyms), random insertion (add a random synonym of a random word), random swap (swap two words' positions), and random deletion (drop words with some probability). Each operation is applied with low probability (10–20%) to keep the text readable. EDA is embarrassingly simple but surprisingly effective — it improved classification accuracy by 1–3% in the original paper, especially on small datasets under 500 examples.

# Simple EDA implementation
import random
from nltk.corpus import wordnet

def synonym_replacement(words, n=1):
    """Replace n words with synonyms."""
    new_words = words.copy()
    replaceable = [w for w in words if wordnet.synsets(w)]
    random.shuffle(replaceable)
    
    replaced = 0
    for word in replaceable:
        synonyms = wordnet.synsets(word)
        if synonyms:
            synonym = synonyms[0].lemmas()[0].name()
            if synonym != word:
                new_words = [synonym if w == word else w for w in new_words]
                replaced += 1
        if replaced >= n:
            break
    return new_words

def random_deletion(words, p=0.1):
    """Randomly delete words with probability p."""
    if len(words) == 1:
        return words
    return [w for w in words if random.random() > p]

# Example
text = "The movie was absolutely wonderful and touching"
words = text.split()
print("Original:", text)
print("Synonym:", " ".join(synonym_replacement(words, n=2)))
print("Deletion:", " ".join(random_deletion(words, p=0.2)))

Evaluation Metrics

BLEU Score

BLEU (Bilingual Evaluation Understudy) measures how much the generated text overlaps with reference text in terms of n-grams. It computes precision for unigrams, bigrams, trigrams, and 4-grams, then takes a geometric mean. A brevity penalty discourages outputs that are too short. BLEU ranges from 0 to 1 (often reported as 0–100). It was designed for machine translation but is used broadly for any text generation task.

BLEU's big limitation: it's purely surface-level. "The cat sat on the mat" and "A feline rested upon the rug" have low BLEU despite meaning the same thing. It also doesn't handle synonyms, paraphrasing, or different-but-valid sentence structures. Use BLEU as one signal, never as the only metric.

import evaluate

# Load metrics
bleu_metric = evaluate.load("bleu")
rouge_metric = evaluate.load("rouge")

# Example: evaluate generated summaries
predictions = [
    "Machine learning models improve through experience with data.",
    "RAG reduces hallucination by grounding responses in retrieved documents.",
]
references = [
    ["ML models learn and improve from data and experience."],
    ["Retrieval-augmented generation grounds LLM answers in factual documents."],
]

# BLEU score
bleu_result = bleu_metric.compute(predictions=predictions, references=references)
print(f"BLEU: {bleu_result['bleu']:.4f}")

# ROUGE scores
rouge_result = rouge_metric.compute(predictions=predictions, 
                                      references=[r[0] for r in references])
print(f"ROUGE-1: {rouge_result['rouge1']:.4f}")
print(f"ROUGE-2: {rouge_result['rouge2']:.4f}")
print(f"ROUGE-L: {rouge_result['rougeL']:.4f}")

ROUGE Score

ROUGE focuses on recall rather than precision. While BLEU asks "what fraction of generated n-grams appear in the reference?", ROUGE asks "what fraction of reference n-grams appear in the generated text?" This makes ROUGE more suitable for summarization, where you want to ensure the summary covers the key points of the source, even if it includes additional text. ROUGE-1 (unigram), ROUGE-2 (bigram), and ROUGE-L (longest common subsequence) are the standard variants. Most summarization papers report all three.

BERTScore

BERTScore addresses the fundamental problem with BLEU and ROUGE: they only match exact words. BERTScore instead computes contextual embeddings for each token in the generated and reference texts, then finds the maximum cosine similarity between each generated token and any reference token (and vice versa). This captures semantic similarity — "automobile" matches "car," "happy" matches "joyful." BERTScore correlates much better with human judgments than BLEU or ROUGE, especially for creative and paraphrasing-heavy tasks.

import evaluate

bertscore = evaluate.load("bertscore")

predictions = ["The cat rested on the carpet."]
references = ["The cat sat on the mat."]

# BERTScore captures semantic similarity even with different words
results = bertscore.compute(
    predictions=predictions,
    references=references,
    lang="en"
)
print(f"BERTScore Precision: {results['precision'][0]:.4f}")
print(f"BERTScore Recall:    {results['recall'][0]:.4f}")
print(f"BERTScore F1:        {results['f1'][0]:.4f}")
Metric Selection Guide

Use BLEU for machine translation (it's the standard, even if imperfect). Use ROUGE for summarization (recall matters more than precision). Use BERTScore when paraphrasing is common and you need semantic matching. Use exact match + F1 for extractive QA (compare predicted span to gold span). For open-ended generation, combine automatic metrics with human evaluation — no automatic metric fully captures quality, coherence, and helpfulness.

End-to-End Evaluation Pipeline

Here's a complete pattern for evaluating a summarization model on a standard benchmark:

from transformers import pipeline
from datasets import load_dataset
import evaluate

# Load model, data, and metrics
summarizer = pipeline("summarization", model="facebook/bart-large-cnn")
dataset = load_dataset("cnn_dailymail", "3.0.0", split="test[:50]")
rouge = evaluate.load("rouge")

# Generate predictions
predictions = []
references = []

for example in dataset:
    article = example["article"][:1024]  # truncate for speed
    summary = summarizer(article, max_length=128, min_length=30, do_sample=False)
    predictions.append(summary[0]["summary_text"])
    references.append(example["highlights"])

# Compute metrics
results = rouge.compute(predictions=predictions, references=references)
print(f"ROUGE-1: {results['rouge1']:.4f}")
print(f"ROUGE-2: {results['rouge2']:.4f}")
print(f"ROUGE-L: {results['rougeL']:.4f}")
The Practical NLP Workflow

1. Explore: Use pipelines to test a few models quickly on your data. 2. Evaluate: Pick the best candidate and run proper metrics on a held-out test set. 3. Fine-tune: If off-the-shelf models aren't good enough, fine-tune on your domain data using the Trainer API. 4. Augment: If you have limited labeled data, apply back-translation or EDA before fine-tuning. 5. Explain: Use LIME or attention visualization to debug failures and build trust with stakeholders. 6. Monitor: Track metric drift in production — model performance degrades as language evolves.
