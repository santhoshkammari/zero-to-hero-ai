The NLP Task Landscape

Understanding what connects NLP tasks under hood — difference between token classification and sequence classification, why it matters — skeleton key to field.

NLP = collection of tasks — not one task with many names. Each task asks different question about text, each question demands different output shape from model. Some tasks label entire document. Some label every token. Some generate new text. Understanding which shape goes with which question = skeleton key to field. Lets you look at new business problem and immediately know which architecture to reach for.

Covers: classification, entity recognition, parsing, entailment, question answering, summarization, coreference, relation extraction, and benchmarks tying them together. No prior knowledge needed. Each built from concrete toy example, same running scenario threads through all. Concepts added one piece at time.

The Three Shapes of NLP

Nearly every NLP task falls into one of three shapes. Shape determines architecture. Think: sorting, coloring, rewriting.

Sequence classification takes entire input — sentence, paragraph, whole email — assigns single label. Spam or not spam. Positive or negative. Topic A, B, or C. Model reads whole thing, produces one vector (typically from special [CLS] token in transformer), maps vector to label through classification head. One input, one output. That's sorting.

Token classification assigns label to every token in input. Sentence goes in, tag comes out for each word: person name, location, verb. Model produces hidden state for each token, each gets own classification head. One input, many outputs — one per word. That's coloring.

Sequence-to-sequence (seq2seq) takes input sequence, generates completely new output sequence. Output can be shorter, longer, or same length. Translation, summarization, abstractive QA all live here. Encoder reads input; decoder writes output, one token at time. That's rewriting.

Once three-way split clicks, half NLP confusion evaporates. Sentiment analysis? Sorting. NER? Coloring. Summarization? Rewriting. Every task in this chapter maps to one of three shapes. That's compass for whole journey.

Compass alone doesn't show terrain. Let's build map.

Our Running Example: ShopBot

Building ShopBot — customer support system for small online store selling headphones, laptops, phone cases. Customers send messages like:

"My laptop screen is cracked and I want a refund."
"These headphones are amazing! Best purchase ever."
"I ordered a phone case from Austin, TX and it hasn't arrived."
"Does the warranty cover water damage?"
"The laptop was great but the delivery took forever."

Five messages. Five different things ShopBot needs: route to right team, detect sentiment, find product and location names, answer questions, summarize complaint threads. Each need = different NLP task. Each task demands different model output shape. We'll keep coming back to these five messages throughout.

Text Classification — Sorting Mail into Bins

First thing ShopBot needs: decide where message goes. Complaint? Product question? Shipping inquiry? Compliment? That's text classification — most fundamental NLP task. One document in, one label out.

Mailroom analogy. Letters arrive. Person reads each, stamps category, drops into corresponding bin. Letter doesn't change. Gets label.

Example: ShopBot has three bins: billing, shipping, product. Message: "My laptop screen is cracked and I want a refund." Classifier reads this, outputs single label. Classical way: count words. Build bag-of-words vector — list of word counts, one entry per vocabulary word — feed to logistic regression or Naive Bayes. Words like "refund" and "cracked" push toward billing. Words like "arrived" and "shipping" push toward shipping. Counts become votes.

Fancier version: TF-IDF (term frequency–inverse document frequency). Downweights common words like "and" and "I" that appear everywhere, upweights rare informative words like "refund" and "cracked." Word appearing in every document tells nothing about which bin document belongs to. Word appearing in few documents = strong signal.

Limitation: bag-of-words throws away word order. "The laptop was great but the delivery was terrible" and "The delivery was great but the laptop was terrible" produce identical vectors despite opposite meanings. People tolerated this for years. Then CNNs and RNNs entered — encoding word sequences as vectors preserving order. TextCNN (Yoon Kim, 2014) slides small filters over word embedding sequences — width 3, 4, 5 words — capturing local phrase patterns like "want a refund" or "screen is cracked."

Modern approach: fine-tune pretrained transformer — BERT, RoBERTa, or similar — on labeled data. Transformer reads entire message, [CLS] token (special token prepended to input) accumulates representation of whole sequence. Single linear layer on top maps to label set. Few hundred labeled examples beats classical approaches by wide margin.

Zero labeled data? Zero-shot classification with LLM. Write prompt: "Classify this customer message as billing, shipping, or product: 'My laptop screen is cracked and I want a refund.'" Model gives answer. Won't match fine-tuned specialist, but requires no training data. Tradeoff — accuracy vs. setup cost — constant decision in production NLP.

Text classification = mailroom metaphor. But some mail requires reading emotion between lines, not topic on label. Enter sentiment analysis.

Sentiment Analysis: The Trickiest Bin

Sentiment analysis = text classification with more nuanced costume. Instead of routing message to department, gauging emotional temperature: customer happy, angry, or neutral?

ShopBot messages: "These headphones are amazing! Best purchase ever." — positive. "My laptop screen is cracked and I want a refund." — negative. Easy. Now: "The laptop was great but the delivery took forever." Single message carries positive sentiment toward product, negative toward shipping. This is aspect-based sentiment analysis — detecting sentiment toward specific aspects within same text. Same document, two emotions.

Classical approach: sentiment lexicon — dictionary assigning polarity scores to words. VADER (Valence Aware Dictionary and sEntiment Reasoner) = most widely used English lexicon. Knows "amazing" is positive, "cracked" is negative, capitalization intensifies ("GREAT" hits harder than "great"). But lexicons break on sarcasm. "Oh wonderful, my package is lost again" — lexicon sees "wonderful" and smiles. Humans don't.

Sarcasm trips up models and humans alike. Negation another trap — "not bad" is positive despite containing "bad." Domain shift real: model trained on movie reviews learns "unpredictable" = praise (thriller), but restaurant review model learns it's complaint (chef). Same words; meaning flips based on unseen context.

Modern sentiment analysis: fine-tune transformer on labeled data for specific domain. Domain-specificity = key insight. General-purpose sentiment model = starting point, not finished product. Always evaluate on your domain data before shipping.

Sorting and sentiment tell us about whole message. But ShopBot also needs to pick out specific names and places within message. Different shape entirely.

Named Entity Recognition — Highlighting the Nouns That Matter

Third ShopBot message: "I ordered a phone case from Austin, TX and it hasn't arrived." ShopBot needs to know "phone case" = product, "Austin, TX" = location. Not enough to know message topic — need to reach inside sentence and tag specific words.

Named Entity Recognition (NER) = token classification task. Every token gets own label: person, organization, location, product, date, or — for majority of tokens — nothing.

Key question: if every token gets label, how mark multi-word entities? "Austin" and "TX" = two tokens, one location. "Phone" and "case" = two tokens, one product mention. Need scheme showing where entity starts and continues.

BIO Tagging: A Coloring Scheme for Tokens

Answer: BIO tagging (also called IOB tagging). Every token gets one of three tag types:

B-XXX = token is beginning of entity type XXX. I-XXX = token is inside already-started entity. O = token is outside any entity — not part of named entity.

ShopBot message trace:

Token        Tag
─────────    ──────
I            O
ordered      O
a            O
phone        B-PROD
case         I-PROD
from         O
Austin       B-LOC
,            I-LOC
TX           I-LOC
and          O
it           O
hasn't       O
arrived      O

"Phone" gets B-PROD — begins product entity. "Case" gets I-PROD — continues same entity. "Austin" starts new location (B-LOC), comma and "TX" continue it (I-LOC). Everything else = O.

Why need B tag? Why not tag everything I-LOC? Because consecutive same-type entities would merge. "I visited Austin TX and Dallas TX" — without B tags, all four location tokens look like one continuous location. B tag on "Dallas" signals: new entity, separate from previous.

Classical NER: Conditional Random Fields (CRFs) with hand-crafted features — word capitalized? In gazetteer of city names? Surrounding words? CRFs model transition probabilities between tags. Learn constraints like "I-PER cannot follow B-LOC" — person tag can't continue location tag. At inference, Viterbi algorithm finds globally optimal tag sequence respecting all constraints.

Neural breakthrough: BiLSTM-CRF — bidirectional LSTM reads sentence both directions, builds context-aware representations per token, CRF layer enforces valid tag transitions. Dominant NER architecture ~2015–2018. Modern NER: fine-tune BERT, take each token's hidden state, classify through linear layer. Some add CRF on top of BERT; others find BERT's contextual representations rich enough to enforce valid transitions alone.

But BIO tagging has blind spot.

Span-Based NER and SpanBERT

"Bank of America" = organization. But "America" inside it = also location. BIO tagging assigns each token only one tag, can't represent both simultaneously. This is nested NER problem — entities inside other entities.

Span-based approaches change question. Instead of "what tag does each token get?" → "for every possible span, is this span an entity?" Model enumerates all candidate spans up to max length (say 10 tokens), classifies each independently. Span can be entity even if overlapping with another entity span.

SpanBERT (Joshi et al., 2019) pretrained with span-level objective: instead of masking individual tokens like BERT, masks contiguous spans and trains model to predict masked content using span boundary tokens. Gives SpanBERT strong representations for contiguous text spans — natural fit for span-based NER (and extractive QA and coreference resolution).

Tradeoff: computational cost. Sentence with n tokens → O(n²) possible spans. Short sentences: fine. Long documents: need pruning — only consider spans scoring above threshold, or limit max span length. But handling nested entities makes span-based approaches essential for biomedicine, where nested mentions are norm (gene inside protein complex inside pathway).

NER tells us what word is. But language has more structure. Words play grammatical roles, those roles form skeleton holding sentence together.

POS Tagging and Dependency Parsing — The Grammar Skeleton

Part-of-speech (POS) tagging = another token classification task: every word gets grammatical label. Noun, verb, adjective, preposition, determiner. "My laptop screen is cracked" tags:

Token      POS
────────   ────
My         PRON
laptop     NOUN
screen     NOUN
is         AUX
cracked    ADJ

Universal POS Tagset defines 17 coarse-grained tags working across languages: NOUN, VERB, ADJ, ADV, PRON, DET, ADP, NUM, and others. Finer-grained tagsets like Penn Treebank distinguish past-tense verbs (VBD), gerunds (VBG), third-person singular present (VBZ) — over 36 tags. Universal tagset trades granularity for cross-lingual consistency.

POS tagging sounds solved — modern taggers achieve 97%+ accuracy — but remaining 3% hides interesting cases. "Book" = noun in "I read a book," verb in "Book a flight." "Running" = verb in "She is running," noun in "Running is fun." Ambiguous cases = exactly what trips up downstream tasks.

Dependency parsing goes deeper. Not "what part of speech?" but "which word does this word depend on, what's the relationship?" Every word (except root, usually main verb) has exactly one head — word it modifies or is governed by. Relationship between word and its head = dependency relation.

Parse "My laptop screen is cracked":

cracked  ← root (the main verb, anchor of the sentence)
  ├── screen ← nsubj (the subject of "cracked")
  │     ├── My ← nmod:poss (possessive modifier of "screen")
  │     └── laptop ← compound (compound modifier of "screen")
  └── is ← cop (copula — the linking verb)

Tree shows "screen" = subject of "cracked," "My" possessively modifies "screen," "laptop" = compound modifier of "screen." Universal Dependencies framework standardizes relation labels across 100+ languages — parsers work across languages with same annotation scheme.

Why matters for ShopBot? Dependency parsing extracts structured meaning. Customer writes "The headphones from your store have terrible sound quality" — parsing reveals "terrible" modifies "quality" (not "headphones" or "store"), "quality" = subject complement. Without structure, might associate "terrible" with wrong noun — route complaint to wrong team.

POS tagging and parsing seem like relics from pre-transformer era. For many production tasks, true. But in low-resource languages, domain-specific applications lacking training data, and systems needing explainable structure (legal tech, medical NLP) — these remain essential. spaCy ships fast, accurate parsers bolting onto any pipeline.

Semantic Similarity — Measuring Meaning Distance

ShopBot gets two tickets:

Ticket A: "My headphones stopped working after one week."
Ticket B: "The earbuds I bought died within seven days."

Share almost no words, but describe same problem. Keyword-matching says unrelated. Human says near-duplicates. Semantic textual similarity (STS) gives machines human answer — continuous score from 0 (completely unrelated) to 5 (identical meaning).

Naive approach: compute word embedding (Word2Vec, GloVe) per word, average into single vector per sentence, compute cosine similarity. Works better than expected, but averaging destroys word order. "Dog bites man" and "Man bites dog" produce identical vectors. Unsatisfying.

Sentence-BERT (SBERT) (Reimers and Gurevych, 2019) = breakthrough for text similarity. Fine-tunes BERT with Siamese architecture: feed both sentences through same BERT encoder, pool each output into single vector (typically averaging all token embeddings), train with contrastive or regression objective. Produces dense sentence embeddings in single forward pass. Similarity = cosine between two vectors — takes microseconds, even for millions of pairs.

Think: SBERT gives every sentence address in high-dimensional space. Similar meanings = nearby addresses. Different meanings = far apart. Finding similar documents = nearest-neighbor lookup — same operation powering image similarity search, but for text.

STS Benchmark = standard evaluation: sentence pairs from captions, news headlines, forum posts, scored by human annotators. Models evaluated by correlation between predicted and human scores (Spearman or Pearson). SOTA achieves correlations above 0.90 — sounds good, but real-world performance often falls behind benchmarks, especially on domain-specific text unlike training data. Always benchmark on own data.

Measuring sentence similarity useful. But sometimes question isn't "how similar?" but "does one follow from other?" Fundamentally different question with own task.

Textual Entailment and Natural Language Inference

Customer writes ShopBot: "I bought this laptop last Tuesday and the screen is cracked." Warranty policy says: "Products purchased within 30 days are eligible for free replacement." Does policy cover this customer?

Natural language inference (NLI), also called textual entailment. Two pieces of text — premise and hypothesis — decide: does premise entail hypothesis, contradict it, or neither?

Three concrete examples:

Premise:    "The package was delivered on Monday."
Hypothesis: "The customer received a delivery."
Label:      ENTAILMENT (if the package was delivered, a delivery happened)

Premise:    "The package was delivered on Monday."
Hypothesis: "The package has not been shipped yet."
Label:      CONTRADICTION (delivered and not shipped can't both be true)

Premise:    "The package was delivered on Monday."
Hypothesis: "The customer was satisfied with the product."
Label:      NEUTRAL (delivery says nothing about satisfaction)

NLI = sequence classification task with twist. Input = sentence pair, not single sentence. Concatenate premise and hypothesis with separator token, feed through transformer, classify [CLS] token into three labels: entailment, contradiction, neutral.

NLI powerful because it's building block for other tasks. Need zero-shot text classification? Frame as entailment. For each candidate label, create hypothesis: "This message is about shipping." Feed with message as premise. Highest entailment score wins. Many zero-shot classification systems work this way — running NLI in disguise.

Major NLI datasets: SNLI (Stanford Natural Language Inference, 570K sentence pairs) and MultiNLI (Multi-Genre NLI, 433K pairs across ten genres). MultiNLI = one of GLUE benchmark tasks.

Hard NLI cases mirror hard human reasoning. Numerical reasoning ("The train arrives at 3:15" → entails "arrives in afternoon"?), world knowledge ("painting is by Picasso" → entails "cubist"?), multi-step inference — all remain challenging. Where models fail at entailment still active research area.

Question Answering — Pointing vs. Generating

Customer asks ShopBot: "Does the warranty cover water damage?" ShopBot has warranty text paragraph to search. How answer?

Two fundamentally different approaches. Confusing them = common production bug source.

Extractive QA points to answer. Model reads question and passage, identifies contiguous span answering question. Answer always verbatim quote — model never generates new words. Like highlighter: model highlights relevant span, says "here's your answer."

Mechanics: model receives question and passage concatenated:

[CLS] Does the warranty cover water damage? [SEP] Our standard warranty
covers manufacturing defects for 12 months. Water damage and physical
damage are not covered under any warranty plan. [SEP]

Two linear heads on top of transformer. One predicts start position of answer span. Other predicts end position. Each head scores every passage token. Softmax converts to probabilities. Model picks span with highest combined start-end probability — here: "Water damage and physical damage are not covered under any warranty plan."

SQuAD (Stanford Question Answering Dataset) popularized this approach. SQuAD 1.1: 100K+ question-answer pairs, every question answerable from passage. SQuAD 2.0 twist: some questions unanswerable, model must learn to say "I don't know" — harder than finding answer when it exists.

Abstractive QA generates answer in own words. Instead of pointing to span, model reads question and context, produces new text via encoder-decoder (T5, BART, or LLM). Answer might paraphrase, combine info from multiple sentences, or express same fact differently.

Tradeoff: trust vs. fluency. Extractive QA gives provenance — always point to where answer came from. Abstractive QA gives natural, conversational answers — but can hallucinate, generating fluent text unsupported by source. Legal/medical where traceability matters → extractive safer. Chatbots where naturalness matters → abstractive default.

Open-domain QA goes further: no passage provided. Model must find answer from large corpus — all of Wikipedia, or company's entire knowledge base. Standard architecture: retrieve-then-read pipeline — retriever fetches most relevant passages, reader extracts or generates answer. This = pattern behind retrieval-augmented generation (RAG), covered later.

Extractive QA = "pointing" — span classification predicting start/end positions. Abstractive QA = "rewriting" — seq2seq generating new text. Same question, different architecture shapes.

Summarization — Pruning vs. Rewriting

ShopBot has long complaint thread — fifteen back-and-forth messages between customer and two agents. Manager needs three-sentence summary. Same extractive-vs.-abstractive split from QA appears here.

Extractive summarization selects most important sentences from original, concatenates them. Text never rewritten — only pruned. Classic algorithm: TextRank, analogous to Google's PageRank. Build graph where each sentence = node. Edges weighted by similarity (cosine similarity of TF-IDF vectors). Run PageRank-like algorithm scoring sentence importance — sentences similar to many important sentences get high scores. Pick top-N. That's your summary.

PageRank: web pages important if many important pages link to them. TextRank: sentences important if many important sentences similar to them. Same logic, different domain.

Abstractive summarization generates new text capturing document meaning. Seq2seq task: encoder reads document, decoder generates summary token by token. BART pretrained by corrupting text (deleting, shuffling, masking sentences) and learning to reconstruct — pretraining objective directly mirrors summarization skill. Pegasus went further: masked entire sentences during pretraining, trained model to generate them — even closer to downstream task.

Evaluation: ROUGE scores (Recall-Oriented Understudy for Gisting Evaluation). ROUGE-1 = unigram overlap between generated and reference summary. ROUGE-2 = bigram overlap. ROUGE-L = longest common subsequence. ROUGE limitation: two summaries expressing same meaning with different words score poorly. Newer metrics like BERTScore compare via embedding similarity rather than word overlap — better captures semantic equivalence.

LLM-based summarization now default for many applications. But processing millions of documents — summarizing every support thread in company history — fine-tuned BART at 400M params orders of magnitude cheaper than 100B+ model API calls. Architecture choice always returns to tradeoff: quality, cost, scale.

Coreference Resolution — The Glue of Document Understanding

ShopBot support thread illustrating subtle problem:

Customer: "I bought a laptop last week. It arrived with a cracked screen."
Agent: "I'm sorry about that. We'll send you a replacement."
Customer: "When will it arrive?"

"It" appears three times. First "it" = laptop. Second "it" (agent's implicit context) = replacement. Third "it" = replacement, not laptop. Human instantly connects dots. Machine needs same to answer "When will it arrive?" correctly.

Coreference resolution = figuring out which mentions refer to same real-world entity. "The laptop," "it" (first), "a laptop" = one cluster. "A replacement" and "it" (third) = another cluster. Two stages: mention detection (find all noun phrases, pronouns, named entities that could refer to something) and mention clustering (group ones referring to same thing).

Classical systems: hand-crafted features — pronoun gender match candidate antecedent? Numbers agree (singular vs. plural)? Distance in text? Feature-based systems brittle. "The CEO" and "she" co-refer only if you know CEO is female — info might not appear in local context.

Landmark neural approach: end-to-end model by Lee et al. (2017). Instead of pipeline (detect mentions then cluster), does both jointly. Considers every possible span as potential mention, scores each span's "mentionness," ranks all possible antecedents. Key innovation: mention detection and clustering in one pass — errors don't propagate. Uses span representations — why SpanBERT became preferred backbone for coreference.

Why coreference matters for ShopBot? Almost every other task depends on it. Summarization can't produce coherent summary without knowing "it" and "the laptop" = same thing. QA can't connect info across sentences. Relation extraction can't build knowledge graph when entities referred to by pronouns. Coreference = glue holding document-level understanding together.

Coreference scoring metrics (MUC, B³, CEAF, LEA) each measure different cluster quality aspects, can disagree with each other. Fine-grained clustering decisions remain genuinely hard, even for SOTA models.

Relation Extraction — Building Knowledge from Text

NER tells us "Austin, TX" = location, "phone case" = product. But doesn't tell relationship between them. Relation extraction fills gap: given entities in sentence, identify semantic relationship connecting them.

From ShopBot message "I ordered a phone case from Austin, TX and it hasn't arrived," relation extraction produces triple: ("phone case", ordered_from, "Austin, TX"). How knowledge graphs get built — every fact in Wikidata or Google's Knowledge Graph either hand-curated or extracted from text.

Simplest: supervised relation extraction — label training set with entity pairs and relationships, train classifier. Features: entities themselves, words between them, dependency path in parse tree, entity types. Person + location more likely linked by "born_in" than "manufactured_by." Downside: can only extract relationship types labeled in training. Labeled data expensive.

Distant supervision = clever workaround. Take existing knowledge base (Wikidata), find all entity pairs with known relationship, search large corpus for sentences mentioning both entities. Assume — often incorrectly but usefully on average — those sentences express known relationship. Gives noisy but abundant training data without manual labeling. Noise = price of scale.

Modern neural approaches: encode sentence with transformer, insert special marker tokens around entities helping model attend to boundaries:

Input: [CLS] I ordered a [E1] phone case [/E1] from [E2] Austin, TX [/E2] [SEP]

The model sees exactly where each entity starts and ends,
attends to the tokens between and around them, and classifies
the relationship between E1 and E2.

Relation extraction = sequence classification with twist: classify relationship between specific entity pair, not whole document. Shape still "one input, one label" — but input is entity-pair-aware.

The Benchmarks: GLUE, SuperGLUE, and SQuAD

Tasks aren't measured in isolation. Field converged on standardized benchmarks comparing models across tasks. Understanding benchmarks essential — nearly every NLP paper references them.

GLUE (General Language Understanding Evaluation), introduced 2018, collection of nine tasks:

Task     What it tests                              Shape
────     ─────────────────────────────────────────   ──────────────────────
CoLA     Linguistic acceptability (is this grammar   Sequence classification
         okay?)
SST-2    Sentiment (positive/negative)               Sequence classification
MRPC     Paraphrase detection (same meaning?)        Sentence-pair classification
QQP      Question equivalence (same question?)       Sentence-pair classification
STS-B    Semantic similarity (how similar, 0-5?)     Sentence-pair regression
MNLI     Natural language inference (3-way)          Sentence-pair classification
QNLI     QA as entailment (does the sentence         Sentence-pair classification
         contain the answer?)
RTE      Textual entailment (2-way)                  Sentence-pair classification
WNLI     Winograd schema (coreference)               Sentence-pair classification

Pattern: most GLUE tasks = sequence classification or sentence-pair classification — sorting tasks. GLUE designed to test language understanding. Classification = cleanest way to measure understanding without text generation confounds.

BERT demolished GLUE leaderboard in 2018. Within year, models surpassed estimated human performance. Benchmark became too easy.

SuperGLUE (2019) raised bar with harder tasks: BoolQ (yes/no reading comprehension), CB (commitment bank — nuanced 3-way entailment, few training examples), COPA (causal reasoning), WiC (word-in-context — same word same meaning in two sentences?), WSC (Winograd Schema Challenge — coreference requiring world knowledge), MultiRC (multi-sentence reading comprehension), ReCoRD (reading comprehension with commonsense). SuperGLUE requires deeper reasoning: commonsense, causal thinking, world knowledge — surface-level pattern matching can't solve.

SQuAD (Stanford Question Answering Dataset) = benchmark for extractive QA. Version 1.1: 100K+ question-answer pairs, every question answerable from passage. Version 2.0: mixed in 50K+ unanswerable questions — forcing models to learn "I don't know." Scores: exact match (EM, percentage matching ground truth exactly) and F1 (average word-level overlap between prediction and ground truth).

Benchmarks have real limitations. High GLUE scores ≠ model understands language deeply — models exploit dataset artifacts, spurious correlations, annotation biases. SuperGLUE harder, but still falls short of human reasoning. Benchmarks measure specific capability slice, not general understanding. Useful as thermometers, not definitions of intelligence.
