Classical Text Representations

Chapter 11: Natural Language Processing 2 subtopics
TL;DR

Machines can't read words — they need numbers. Bag of Words counts how often each word appears in a document and builds a sparse vector. TF-IDF refines that by down-weighting common words and boosting rare, informative ones. Both methods ignore word order entirely, which limits them, but they're fast, interpretable, and still surprisingly effective for many classification and retrieval tasks. Understanding these foundations makes the jump to word embeddings and transformers far less mysterious.

The Problem: Text Is Not Numbers

A machine learning model is, at its core, a function that takes numbers in and pushes numbers out. Matrix multiplications, dot products, gradient updates — all of it requires numerical input. But text is a sequence of characters grouped into words grouped into sentences. There's no obvious way to multiply "the cat sat on the mat" by a weight matrix.

So we need a translation step. We need to convert a document — whether it's a tweet, a legal contract, or a product review — into a fixed-length vector of numbers that a model can chew on. The simplest family of techniques for doing this is called classical text representations, and the two workhorses are Bag of Words and TF-IDF. They've been around for decades, they're dead simple, and they still power a surprising number of production systems.

Let's build the intuition from scratch.

Bag of Words

Vocabulary Construction

The first step is deciding which words we care about. You collect every unique word across all your documents. This set is your vocabulary. If your dataset contains 50,000 unique words, your vocabulary has 50,000 entries. Each entry gets a fixed position — an index. "apple" might be index 0, "banana" index 1, "cat" index 2, all the way up to "zebra" at index 49,999.

This vocabulary becomes the backbone of your representation. Every document, regardless of its actual length, will be turned into a vector of length 50,000 — one slot per vocabulary word.

Count Vectorization

The Bag of Words model is almost comically simple. For each document, you count how many times each vocabulary word appears. That's it. You literally ignore word order, grammar, sentence structure, punctuation — everything except raw word counts.

Let's walk through a concrete example. Suppose we have three tiny documents:

Doc 1: "the cat sat on the mat"
Doc 2: "the dog sat on the log"
Doc 3: "the cat chased the dog"

First, we build the vocabulary from all three documents: {the, cat, sat, on, mat, dog, log, chased}. That's 8 unique words, so each document becomes a vector of length 8.

Now we count. Doc 1 has "the" twice, "cat" once, "sat" once, "on" once, "mat" once, and zero occurrences of "dog", "log", and "chased". So Doc 1's vector is [2, 1, 1, 1, 1, 0, 0, 0]. Doc 2 becomes [2, 0, 1, 1, 0, 1, 1, 0]. Doc 3 becomes [2, 1, 0, 0, 0, 1, 0, 1].

Notice what just happened. We turned variable-length text into fixed-length numerical vectors. A model can now compute distances between documents, feed them into logistic regression, run k-means clustering — whatever it wants. The representation is crude, but it works.

Binary Bag of Words

Sometimes you don't care about counts — just presence. The binary BoW variant replaces every nonzero count with 1. Doc 1 becomes [1, 1, 1, 1, 1, 0, 0, 0]. This can actually work better for short documents where word frequency isn't very informative. If a word appears once or three times in a tweet, the fact that it appears at all is the signal.

N-grams: Capturing Word Pairs

Plain BoW has a glaring weakness: it loses all word order. The sentences "the dog bit the man" and "the man bit the dog" produce identical BoW vectors. That's a problem.

N-grams offer a partial fix. Instead of treating individual words (unigrams) as your vocabulary units, you also include consecutive word pairs (bigrams) or triples (trigrams). With bigrams, your vocabulary grows to include entries like "not good", "very happy", "machine learning". Now "not good" is a distinct vocabulary item from just "good", so the model can learn that they carry opposite sentiment.

The tradeoff is vocabulary size. If you have 50,000 unigrams, adding bigrams might push you to 500,000 entries. Trigrams push you into the millions. Each document vector gets correspondingly longer and sparser.

The Sparsity Problem

Here's the uncomfortable truth about BoW. If your vocabulary has 50,000 words and a typical document contains 200 words, then roughly 49,800 entries in that document's vector are zero. That's 99.6% zeros. This is what we call a sparse representation.

Sparsity isn't automatically bad — sparse matrix libraries handle it efficiently — but it creates real issues. Models need more data to learn meaningful patterns because the signal is spread thinly across a huge space. Distance metrics start misbehaving in very high dimensions (the curse of dimensionality). And storage scales with vocabulary size, not document length. These limitations motivate everything that comes later: TF-IDF, dimensionality reduction, and eventually dense embeddings.

Code: Bag of Words with sklearn

from sklearn.feature_extraction.text import CountVectorizer

docs = [
    "the cat sat on the mat",
    "the dog sat on the log",
    "the cat chased the dog",
]

# Unigram BoW
vectorizer = CountVectorizer()
X = vectorizer.fit_transform(docs)

print("Vocabulary:", vectorizer.get_feature_names_out())
# ['cat', 'chased', 'dog', 'log', 'mat', 'on', 'sat', 'the']

print("BoW matrix:\n", X.toarray())
# [[1 0 0 0 1 1 1 2]
#  [0 0 1 1 0 1 1 2]
#  [1 1 1 0 0 0 0 2]]

# Bigram extension — capture word pairs
bigram_vec = CountVectorizer(ngram_range=(1, 2))
X_bigram = bigram_vec.fit_transform(docs)
print(f"\nUnigrams: {X.shape[1]} features")
print(f"Uni+Bigrams: {X_bigram.shape[1]} features")

# Binary BoW
binary_vec = CountVectorizer(binary=True)
X_binary = binary_vec.fit_transform(docs)
print("\nBinary BoW:\n", X_binary.toarray())

Notice that CountVectorizer returns a sparse matrix by default. You call .toarray() only for display. In production with large vocabularies, you keep it sparse to save memory.

TF-IDF

The Intuition

BoW treats every word equally. "The" and "quantum" get the same treatment — both are just counts. But obviously "quantum" carries far more information than "the." If a word appears in nearly every document, it's not useful for distinguishing one document from another. If a word appears in only a handful of documents, it's a strong signal.

TF-IDF (Term Frequency–Inverse Document Frequency) formalizes this intuition. It takes the raw count from BoW and multiplies it by a factor that penalizes common words and rewards rare ones.

Term Frequency

The term frequency TF(t, d) measures how often term t appears in document d. In its simplest form, it's just the raw count — exactly what BoW gives us. Some variants normalize by document length (divide by total words in d) to prevent long documents from dominating, but the raw count is the most common starting point.

Inverse Document Frequency

The inverse document frequency IDF(t) measures how rare a term is across the entire corpus. The formula is:

IDF(t) = log(N / df(t))

where N is the total number of documents and df(t) is the number of documents that contain term t. If a word appears in every document, df(t) = N, so IDF = log(1) = 0 — the word gets completely zeroed out. If a word appears in only 1 out of 1,000 documents, IDF = log(1000) ≈ 6.9 — it gets a big boost.

In practice, sklearn uses a slightly smoothed version — log((1 + N) / (1 + df(t))) + 1 — to avoid division by zero and to prevent terms from being completely eliminated. But the core idea is identical: rare words get higher weights.

The Combined Formula

The final TF-IDF score for term t in document d is simply:

TF-IDF(t, d) = TF(t, d) × IDF(t)

A high TF-IDF score means the word is frequent in this particular document but rare across the corpus. That's exactly the kind of word that characterizes what this document is about.

Worked Example

Let's reuse our three documents. We have N = 3 documents.

Consider the word "cat". It appears in Doc 1 and Doc 3, so df("cat") = 2. Its IDF is log(3/2) ≈ 0.405. In Doc 1, TF("cat", Doc 1) = 1. So TF-IDF = 1 × 0.405 = 0.405.

Now consider "the". It appears in all 3 documents, so df("the") = 3. Its IDF is log(3/3) = log(1) = 0. No matter how many times "the" appears in a document, its TF-IDF is always zero. The word is completely uninformative for distinguishing documents — which is exactly right.

Now consider "mat". It appears only in Doc 1, so df("mat") = 1. Its IDF is log(3/1) ≈ 1.099. In Doc 1, TF("mat", Doc 1) = 1, so TF-IDF = 1 × 1.099 = 1.099. This is the highest score so far, because "mat" is the most discriminative word — it uniquely identifies Doc 1.

This is the power of IDF. Without it, "the" and "mat" look equally important. With it, the model automatically learns to focus on the words that actually matter.

Document Similarity with Cosine Similarity

Once you have TF-IDF vectors, you can measure how similar two documents are using cosine similarity. This computes the cosine of the angle between two vectors — 1.0 means identical direction (same topic), 0.0 means completely unrelated, and -1.0 means opposite (rare in practice since TF-IDF values are non-negative).

Why cosine and not Euclidean distance? Because cosine is length-invariant. A 5,000-word article and a 50-word abstract about the same topic will have very different vector magnitudes but similar directions. Cosine captures that shared direction. Euclidean distance would say they're far apart just because one vector is longer.

Code: TF-IDF with sklearn

from sklearn.feature_extraction.text import TfidfVectorizer
from sklearn.metrics.pairwise import cosine_similarity

docs = [
    "the cat sat on the mat",
    "the dog sat on the log",
    "the cat chased the dog",
]

# Fit TF-IDF
tfidf = TfidfVectorizer()
X = tfidf.fit_transform(docs)

print("Vocabulary:", tfidf.get_feature_names_out())
print("TF-IDF matrix (rounded):")
print(X.toarray().round(3))

# Document similarity
sim_matrix = cosine_similarity(X)
print("\nCosine similarity matrix:")
print(sim_matrix.round(3))
# Doc 1 vs Doc 2 will be moderately similar (shared "sat", "on")
# Doc 1 vs Doc 3 will be somewhat similar (shared "cat")
# Doc 2 vs Doc 3 will be somewhat similar (shared "dog")
Why sklearn's TF-IDF Numbers Look Different

If you compute TF-IDF by hand and compare with sklearn's output, the numbers won't match exactly. That's because TfidfVectorizer applies smoothing to the IDF formula, adds 1 to prevent zero divisions, and then L2-normalizes each document vector so they all have unit length. This normalization bakes cosine similarity into the representation itself — the dot product of two L2-normalized vectors is their cosine similarity. The conceptual idea is identical; sklearn just adds engineering guardrails.

Limitations of TF-IDF

TF-IDF is a massive improvement over raw counts, but it has clear limitations you should know about.

First, there is no word order. "Dog bites man" and "man bites dog" produce identical TF-IDF vectors. N-grams help somewhat, but they blow up the vocabulary and only capture local ordering.

Second, there is no semantics. "Happy" and "joyful" are as unrelated as "happy" and "refrigerator" in TF-IDF space. The method has no notion of meaning, synonymy, or even that words can be conceptually related. Each word is an independent dimension.

Third, the curse of dimensionality bites hard. Real-world vocabularies easily reach 100,000+ terms. Every document lives in a 100,000-dimensional space where most coordinates are zero. Models struggle to learn in such high-dimensional, sparse spaces without enormous amounts of training data.

These limitations are precisely what motivated the development of dense word embeddings (Word2Vec, GloVe) and eventually contextual representations (ELMo, BERT, GPT) — all of which you'll encounter in the next sections.

When BoW and TF-IDF Still Work (and When They Don't)

Despite being decades old, these methods haven't retired. TF-IDF remains the backbone of many information retrieval systems, including search engines. If you're building a document search feature, a spam filter, or a topic classifier with limited labeled data, TF-IDF plus logistic regression is a shockingly strong baseline. It trains in seconds, requires no GPU, and is fully interpretable — you can inspect exactly which words drove a prediction.

TF-IDF works best when the task depends on which words appear, not on subtle relationships between them. Spam detection ("buy", "discount", "click here"), topic classification ("election", "candidate", "vote"), and document deduplication are all good fits. When you have small datasets — hundreds or a few thousand labeled examples — TF-IDF often beats neural approaches that are starved for data.

Where do they fail? Anything that requires understanding meaning. Sentiment analysis stumbles because "not great" looks similar to "great" in BoW space. Paraphrase detection fails because "the movie was fantastic" and "the film was excellent" share zero words. Question answering, summarization, and machine translation are all beyond reach. For these tasks, you need representations that capture semantics — and that's where embeddings and transformers come in.

The Practical Rule of Thumb

Always start with TF-IDF. Seriously. Before reaching for BERT or a large language model, build a TF-IDF baseline. It takes five minutes, costs nothing to run, and gives you a performance floor. If TF-IDF gets you 90% of the way there, you might not need a transformer at all. If it only gets you 60%, now you know exactly how much the fancy model is buying you — and you can justify the added complexity and cost.
