NLP in practice goes far beyond text-in-text-out. System handling customer messages in multiple languages, receipt photos, voice recordings, and policy-based reasoning at once needs more than basic text classifier.

This section covers edges of core NLP: language structure beyond word sequences, models transferring across languages, text meeting images and audio, and ways to push language models to reason, stay safe, and call external tools. Not footnotes. Pieces that turn basic text classifier into system that works in messy real world.

Linguistic Structure: Parsing Sentences

Building customer support bot. Customer writes: "The defective battery in my new laptop died." Bot must know "defective" describes "battery," "my new laptop" gives location, and "died" names event. Bag-of-words model sees seven unrelated tokens. Parser sees structure.

Two main ways diagram sentence. Each reveals different thing.

Constituency parsing — also called phrase structure parsing — asks: "How do words group into phrases, and how do phrases nest inside larger phrases?" Output: tree. Bottom level holds words. Higher levels hold groups: "The defective battery" becomes Noun Phrase (NP). "in my new laptop" becomes Prepositional Phrase (PP). "died" is Verb Phrase (VP). Phrases nest upward to single root node labeled S for "sentence." Grammar behind this is context-free grammar (CFG), defining rules like S → NP VP and NP → Det Adj N.

Like Russian nesting dolls. Smallest doll = word. Next doll = phrase holding words. Biggest doll = sentence holding phrases. Each doll fits inside one bigger doll.

Dependency parsing takes different angle. Instead of asking "what groups together," it asks "what depends on what?" Every word has one parent — word it modifies or depends on. Result: directed graph with labeled grammatical relationships. "battery" is subject of "died," "defective" is adjectival modifier of "battery," "laptop" is object of "in." Root is main verb — "died."

Like organizational chart. CEO (main verb) sits on top. Every employee (word) reports to one manager. Edge label shows role.

For support bot, dependency parsing is usually more practical. To extract that battery died, we need subject-verb link — "battery → died" — one edge in dependency tree. Constituency tree makes us climb through nested phrases for same fact. That is why tools like spaCy default to dependency parsing. It gives who-did-what-to-whom structure needed for downstream tasks like information extraction and relation identification.

Limitation: both parsers assume single, well-formed language. They assume grammar rules hold. Customer message like "battery dead laptop won't turn on pls help" — no verb agreement, no articles, telegram-style abbreviations — makes parsers struggle. Real text is messy. Parsing helps when you need precise structure from reasonably well-formed sentences. For rest, need other approaches.

Cross-Lingual Transfer: One Model, Many Languages

Support bot gets new requirement: handle Spanish and Japanese with English. Naive plan: train three separate models. That means three labeled datasets, three training pipelines, three bug sets. Need better way.

Better way exists. Multilingual BERT (mBERT) trained on Wikipedia dumps from 104 languages at once, with no explicit alignment. No parallel sentences. No translation dictionaries. Training stayed same as regular BERT — masked language modeling — except data came from many languages. Result: model learned similar vectors for similar meanings across languages. Train sentiment classifier in English, run it on Spanish text, and it works. Not perfect. Still striking for zero Spanish examples.

Reason this works without parallel data: shared subword vocabulary. Byte-pair encoding finds common character sequences across all 104 languages. Many languages share roots, borrowed words, numbers, and punctuation. Word "internet" appears in many languages with near-identical spelling. Overlapping tokens act as anchor points, pulling representations toward shared space. Not perfect space. Close enough for transfer.

XLM-RoBERTa (XLM-R) pushed further. It trained on CommonCrawl data across 100+ languages, using RoBERTa improvements like dynamic masking, larger batches, and longer training. XLM-R is current cross-lingual workhorse. Pattern is simple: fine-tune on English labeled data, deploy on any supported language. Name: zero-shot cross-lingual transfer.

Nesting-doll analogy still fits, at bigger scale. Outside looks different — different languages. Inside structure stays similar — shared semantic representations. That is what multilingual models build.

Catch: transfer quality drops for languages far from training-heavy ones. English-to-German works well: same script, similar structure. English-to-Japanese is weaker: different script, word order, morphology. Out of roughly 7,000 world languages, only few hundred get meaningful training representation. Cross-lingual transfer is powerful, not magic. It reflects data bias.

Code-Switching: When Languages Collide Mid-Sentence

In many regions, people do not use one language at time. Customer in Miami might write: "My order no ha llegado, can you check the tracking?" Sentence is half English, half Spanish. This is code-switching — alternating between two or more languages inside one utterance. Not mistake. Natural, systematic behavior used by hundreds of millions of bilingual speakers.

And it breaks most NLP pipeline.

First problem: tokenization. Subword tokenizers train on monolingual corpora. When they see "no ha llegado, can you," they may split Spanish words into strange subtokens because those patterns were rare in training data. Second problem: almost no labeled code-switched training data exists. Multilingual models like mBERT trained on separate documents per language, not sentences switching mid-stream. Third problem: grammar. Code-switched sentences follow their own syntactic patterns, not clean match for either language.

Current approaches: create synthetic code-switched data, add language identification as token-level auxiliary task, and build tokenizers aware of script boundaries. Benchmarks like LinCE and GLUECoS are starting to standardize evaluation. Problem is still genuinely unsolved in NLP. Good reminder: language is living, evolving thing. Single model will not pin it down cleanly.

Multimodal NLP: Where Text Meets Vision

Support bot gets another upgrade: customers can send photos of damaged products with complaints. We need model that understands both text like "this arrived cracked" and image showing broken screen. This is domain of multimodal models — systems processing more than one input type.

Model that made this mainstream is CLIP (Contrastive Language-Image Pretraining), released by OpenAI in 2021. Architecture is straightforward. CLIP has two encoders: image encoder, usually Vision Transformer, and text encoder, a 12-layer Transformer much like BERT using byte-pair encoding. Each encoder maps input to vector in same shared embedding space — 512-dimensional space where images and text coexist.

Training is key move. Take batch of 32,768 image-text pairs scraped from internet. Compute embedding for each image and each caption. Training objective is contrastive: push image embedding toward matching caption embedding, away from 32,767 non-matching captions. Loss function is InfoNCE, basically massive softmax over pairings in batch. After training on 400 million image-text pairs, CLIP can match images to text descriptions it never saw before — zero-shot image classification by comparing image embedding with candidate labels like "a photo of a dog" and "a photo of a cat."

For support bot, CLIP text encoder is key piece. It embeds text into space where visual concepts matter. "Cracked screen" and photo of cracked screen land near each other. Foundation for multimodal search, content moderation, and automated damage assessment.

Flamingo, from DeepMind, took different route. Instead of two separate encoders, Flamingo injects visual information into frozen language model through cross-attention layers called Perceiver Resampler blocks. Feed sequence of images mixed with text — like conversation with screenshots — and model reasons over both together. Flamingo showed strong few-shot multimodal learning: give two "image → description" examples, then generalize to new images without fine-tuning.

Limitation of current multimodal models: compositionality. CLIP understands "a red car" and "a blue house" but struggles with "a blue car in front of a red house." Model learns concept-image associations, but spatial relationships and composed scenes stay hard.

Document Understanding: Reading the Layout

Now customer uploads receipt photo for refund. Standard NLP model would OCR text and see something like: "Apple $1 3 Banana $2 1 Total $5." Spatial information — columns, price alignment, "Total" label at bottom — disappears. Model cannot tell that "$1" belongs to "Apple" instead of "Banana."

LayoutLM, developed by Microsoft, fixes this by adding spatial position to input. Standard BERT takes three inputs per token: word embedding, segment embedding, and position embedding for sequence position. LayoutLM adds fourth input: 2D spatial embedding encoding bounding box coordinates [x₀, y₀, x₁, y₁] for where word appears on page. All four embeddings are summed before Transformer encoder.

So model knows more than "Apple" and "$1" appearing in same document. It knows they sit on same horizontal line. It can learn that vertically aligned tokens in one column share relationship. It can learn that "Total" at bottom of receipt plays different role from "Total" inside prose paragraph.

Like adding office locations to org chart. Knowing two employees sit next to each other tells you something about working relationship, even when formal reporting lines do not connect them. Spatial signal is that strong.

LayoutLMv2 extended this with visual features from document image like font size, bolding, and color. LayoutLMv3 unified text, layout, and image in one pretraining framework. These models dominate benchmarks for form understanding (FUNSD), receipt extraction (SROIE), and document classification — boring-looking but highly valuable paperwork automation.

Speech-Text Integration: Hearing Words

Support bot's final sensory upgrade: voice input. Customer calls and describes problem. We need spoken words converted to text. This is automatic speech recognition (ASR). Model that made it feel accessible is OpenAI's Whisper.

Whisper is encoder-decoder Transformer. Encoder side converts raw audio into log-mel spectrogram — 2D representation with time on one axis, frequency on other, and intensity showing loudness per frequency over time. Think heat map of sound. Transformer encoder layers compress this spectrogram into sequence of hidden representations — distilled summary of what was said.

Decoder is autoregressive, generating text one token at time. At each step, it attends to encoded audio through cross-attention and to already generated tokens through masked self-attention. Output is plain-text transcript.

What made Whisper remarkable was not architecture; encoder-decoder Transformers for ASR already existed. Real difference was training data: 680,000 hours of labeled audio scraped from internet across 99 languages. Scale of weakly supervised data gave Whisper robustness earlier models lacked. It handles accents, background noise, multiple speakers, and even translates speech directly into English text without intermediate transcription.

For support bot, Whisper turns phone call into text transcript that can enter same NLP pipeline used for text messages. CLIP text encoder, document layout model, and dependency parser all work on text. Whisper is bridge that brings spoken language into that world.

Limitation: latency. Whisper processes audio in 30-second chunks, and autoregressive decoder is sequential by nature. Real-time applications like live captioning and simultaneous interpretation need streaming architectures that emit partial transcripts as audio arrives. Whisper was not built for that. Adapting it is still active work.

Prompt Engineering Taxonomy

Rewording question can drastically change AI answer. Model scoring 40% on math benchmark can reach 75% by changing only prompt — no retraining, no new data, same weights, same architecture. That makes prompt engineering real engineering discipline.

Here is landscape, organized by how much structure you impose on model behavior.

Zero-shot prompting is simplest form: describe task and provide input, with no examples. "Translate this to French: 'The battery is dead.'" Model relies only on what pretraining taught it. Works well for common tasks seen many times in training data. Works poorly for unusual formats or niche domains.

Few-shot prompting adds handful of input-output examples before real query. "English: The cat sleeps → French: Le chat dort. English: The dog runs → French: Le chien court. English: The battery is dead → French: ___." Examples set pattern. Usually two to five is enough. More often does not help, and noisy examples can hurt.

Instruction-based prompting adds explicit constraints: "Summarize in three sentences," "Respond as a JSON object," "Do not include personal opinions." These prompts shape output format and scope. Models fine-tuned on instruction-following data — InstructGPT, FLAN-T5, ChatGPT — follow these much better than base models.

Role-based prompting assigns persona: "You are a senior electronics repair technician. A customer describes the following problem..." This primes model toward specific knowledge register and communication style. Sounds gimmicky. Still shifts output quality reliably on domain tasks.

Self-consistency prompting generates multiple answers to same question, usually with temperature > 0, then takes majority vote. Like asking five doctors instead of one. Compute cost rises linearly, but reasoning accuracy often rises a lot — especially with chain-of-thought, next topic.

Org-chart analogy fits here too. Zero-shot is hiring employee with no onboarding. Few-shot is short training manual. Instruction-based is detailed job description. Role-based is telling employee which department. Self-consistency is team vote. Each layer adds structure. Structure adds reliability.

Limitation of prompting strategies: they do not change model capabilities. They change how model accesses what it already knows. If knowledge is not in weights, no prompt will pull it out. Prompting is navigation, not creation.

Chain-of-Thought Reasoning

Chain-of-thought (CoT) prompting, introduced by Wei et al. in 2022, tells model to show work — generate intermediate reasoning steps before final answer.

Example from support bot. Customer asks: "I bought 3 items at $12 each but used a $10 coupon. Then I returned 1 item. What's my refund?" Without chain-of-thought, model might guess "$12" or "$26." With chain-of-thought, prompt includes worked examples showing step-by-step reasoning:

Q: I bought 2 shirts at $20 each and used a $5 coupon. I returned 1 shirt. What's my refund?
A: Let me work through this. 2 shirts × $20 = $40 total.
   After coupon: $40 - $5 = $35 paid.
   Per-item cost after coupon: $35 / 2 = $17.50.
   Refund for 1 shirt: $17.50.

Q: I bought 3 items at $12 each but used a $10 coupon.
   Then I returned 1 item. What's my refund?
A:

Given this structure, model generates step-by-step reasoning: 3 × $12 = $36, after coupon $36 - $10 = $26, per-item $26 / 3 ≈ $8.67, refund $8.67. Explicit decomposition forces computation onto each sub-problem instead of one jump to answer.

Even stronger surprise: zero-shot CoT. Append "Let's think step by step" to prompt. That is all. No worked examples. This one phrase raised GSM8K math performance by more than 30 percentage points. Best current theory: phrase gives model more "thinking tokens." Each generated token becomes computation step, and autoregressive transformers turn more output tokens into more serial computation.

Tree-of-thought (ToT) extends this. Instead of one chain, model explores multiple reasoning branches, scores promising paths, and prunes dead ends. Like mapping several forest trails instead of walking one. ToT costs more because it needs multiple generation passes, but it solves problems linear chains cannot — dead-end puzzles, creative tasks with multiple valid approaches, and planning problems where step order matters.

Limitation of CoT and related methods: they amplify existing reasoning flaws. If model has bad probability intuition, chain-of-thought can produce detailed, convincing, wrong derivation. Showing work makes errors easier to spot. It does not prevent them.

Constitutional AI: Teaching Models to Self-Correct

Now we hit problem prompting alone cannot solve. Support bot sometimes gives responses that are helpful but harmful — suggesting safety bypasses or promising refund policies that do not exist. Model needs internal rules for what it should and should not say.

Constitutional AI (CAI), developed by Anthropic, solves this with simple idea: instead of using thousands of human labelers to rate every response, write principles — a "constitution" — and train model to critique and revise its own outputs against those principles.

Process has two phases. First phase: model generates response, critiques that response against specific principle like "Is this response harmful? Does it encourage dangerous behavior?", then writes revised response. This self-critique loop can repeat multiple times, improving answer each pass. Output is dataset of (original response, revised response) pairs generated by AI itself.

Second phase: Anthropic trains reward model on this AI-generated preference data and uses it for reinforcement learning. This is RLAIF — Reinforcement Learning from AI Feedback — instead of RLHF, Reinforcement Learning from Human Feedback. Constitutional principles replace human labelers as source of truth for good behavior.

For support bot, constitution might say: "Do not guarantee specific refund amounts," "Do not recommend actions that void product warranty," and "If you're uncertain about a policy, say so rather than guessing." CAI is useful because principles are explicit, auditable, and editable. Policy changes? Update constitution instead of recollecting thousands of preference labels.

Subtle point matters: CAI does not make model deeply "understand" ethics. It makes model good at pattern-matching against written rules. Constitution quality controls behavior quality. Vague principles produce vague compliance.

Tool Use and Function Calling

Final support-bot piece: customer asks "Where is my order?" Bot must look up order #4521 in database. Pretraining data does not contain this customer's order status. Model must reach outside itself and call API.

Tool use — also called function calling — is mechanism where language model decides to invoke external function, gets result, and uses it in response. Architecture works like this:

First, developer registers available tools with names, descriptions, and parameter schemas in JSON format. Each tool definition looks like function signature: name, purpose, arguments, and types. Registration happens before conversation starts.

{
  "name": "lookup_order",
  "description": "Get the current status and tracking info for a customer order",
  "parameters": {
    "type": "object",
    "properties": {
      "order_id": {"type": "string", "description": "The order ID to look up"}
    },
    "required": ["order_id"]
  }
}

When model processes user message, it decides between plain text and structured function call. If it chooses function call, it emits JSON specifying function and arguments: {"name": "lookup_order", "arguments": {"order_id": "4521"}}. Host application executes call against real API, gets result, and feeds result back as new message. Model then writes natural-language answer using real data: "Your order #4521 shipped yesterday and is expected to arrive by Thursday."

Model does not execute anything itself. It only generates structured request. Execution happens in application layer, which can enforce permissions, rate limits, and validation. Like dispatcher at call center: dispatcher (model) decides which department gets call and what information goes with it, but department (API) does actual work.

Key insight: modern LLMs, after fine-tuning on function-call conversations, learn to emit well-formed JSON with correct argument types and values. They learn meta-skill: "when knowledge runs out, call backup." Model can chain tool calls — look up order, query shipping carrier API, calculate delivery date — then combine results into coherent response.

Limitation: reliability. Models sometimes hallucinate function calls, invoking nonexistent tools or sending malformed arguments. Sometimes they call tools when they should not, because answer was already in prompt. Sometimes they fail to call tools when they should, guessing order status instead of looking it up. Robust tool use needs validation layers, retry logic, and careful prompts that define when to call versus when to answer directly.
