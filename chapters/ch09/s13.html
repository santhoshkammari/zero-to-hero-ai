Pose Estimation

Chapter 9: CNNs & Computer Vision 5 subtopics
TL;DR

Pose estimation locates body joints (keypoints) in images and connects them into a skeleton. Top-down methods detect people first, then estimate each person's pose individually — accurate but slow for crowds. Bottom-up methods detect all keypoints first, then group them into people — faster for multi-person scenes. OpenPose pioneered real-time bottom-up estimation using Part Affinity Fields. HRNet maintains high-resolution features throughout, producing sharper heatmaps. MediaPipe runs on phones. 3D pose estimation lifts 2D skeletons into three dimensions, enabling biomechanics, AR, and motion capture without expensive marker suits.

Every image of a person contains an implicit skeleton. Your brain reconstructs it effortlessly — you know where someone's elbows are even when they're wearing a puffy jacket. Pose estimation teaches machines to do the same thing: given an image (or video frame), predict the 2D or 3D coordinates of anatomical landmarks like the nose, shoulders, elbows, wrists, hips, knees, and ankles.

Why does this matter? Fitness apps count push-up reps by tracking joint angles. Sports analytics systems measure a pitcher's elbow extension at release. Sign language recognition starts with hand and body keypoints. Physical therapy platforms monitor rehabilitation exercises. Autonomous vehicles detect pedestrian pose to predict whether someone is about to step into the road. Animation studios use pose estimation instead of (or alongside) expensive motion capture suits. It's one of the most commercially deployed computer vision capabilities.

Keypoint Detection Fundamentals

The core task is keypoint detection: given an image, predict the (x, y) coordinates of predefined body landmarks. The standard representation uses 17 keypoints (COCO format): nose, left/right eye, left/right ear, left/right shoulder, left/right elbow, left/right wrist, left/right hip, left/right knee, left/right ankle. Some datasets add more — MPII uses 16, the body25 format uses 25, and hand-specific models can track 21 keypoints per hand.

Almost all modern methods predict keypoints using heatmaps. Instead of directly regressing (x, y) coordinates (which is numerically hard — small errors in coordinates mean large pixel displacements), the model outputs one heatmap per keypoint. Each heatmap is a 2D probability map where the brightest point indicates the most likely location of that joint. The ground truth heatmap is a Gaussian blob centered on the annotated joint location. The model learns to produce similar Gaussian-like blobs. At inference time, you find the peak of each heatmap to get the predicted coordinates.

import numpy as np

def generate_heatmap(center_x, center_y, heatmap_size=64, sigma=2):
    """Generate a Gaussian heatmap for one keypoint."""
    heatmap = np.zeros((heatmap_size, heatmap_size))
    for y in range(heatmap_size):
        for x in range(heatmap_size):
            dist_sq = (x - center_x) ** 2 + (y - center_y) ** 2
            heatmap[y, x] = np.exp(-dist_sq / (2 * sigma ** 2))
    return heatmap

# Left shoulder at position (30, 20) on a 64×64 heatmap
heatmap = generate_heatmap(30, 20, heatmap_size=64, sigma=2)
# Peak at (30, 20) with value ≈ 1.0, fading with distance

# At inference: find the peak
pred_y, pred_x = np.unravel_index(heatmap.argmax(), heatmap.shape)
# pred_x=30, pred_y=20

Why heatmaps instead of direct coordinate regression? Three reasons. First, heatmaps are spatially structured — the model can use convolutional features that preserve spatial relationships. Regression to coordinates discards this structure. Second, heatmaps naturally handle uncertainty — a wide, flat blob means "the joint is somewhere around here," while a sharp peak means high confidence. Third, heatmaps allow the loss function (typically MSE between predicted and target heatmaps) to provide dense, per-pixel gradients, making optimization easier than regressing two numbers per joint.

The Resolution-Accuracy Tradeoff

Heatmaps are typically generated at 1/4 the input resolution (e.g., 64×64 for a 256×256 input). This means you can only localize keypoints to a 4-pixel grid. Sub-pixel refinement techniques — like fitting a quadratic to the heatmap peak's neighbors — can recover some precision. This refinement is critical: on the COCO benchmark, it can improve AP by 1–2 points for free.

Top-Down vs. Bottom-Up Approaches

When there are multiple people in the image, you face a grouping problem: which keypoints belong to which person? Two fundamentally different strategies exist.

Top-Down: Detect First, Then Estimate

Top-down methods first run a person detector (like Faster R-CNN or YOLO) to get bounding boxes around each person. Then, a single-person pose estimator runs independently on each cropped bounding box. The output is one skeleton per detected person.

The advantage: the pose estimator always sees a neatly cropped image of one person, making the task much easier. You can use powerful but expensive architectures because each crop is processed independently. State-of-the-art accuracy on benchmarks is almost always achieved by top-down methods.

The disadvantage: runtime scales linearly with the number of people. One person? One forward pass through the pose network. Fifty people in a crowd? Fifty forward passes. For crowd scenes, this becomes prohibitively slow. Also, if the person detector misses someone (occluded or too small), their pose is never estimated at all.

Bottom-Up: Detect All Joints, Then Group

Bottom-up methods detect all keypoints in the image simultaneously — every nose, every elbow, every ankle — regardless of which person they belong to. Then, a grouping step assembles them into individual skeletons. The key challenge is: "this left elbow and this left wrist — do they belong to the same person?"

The advantage: runtime is roughly constant regardless of the number of people. One forward pass detects everything. This makes bottom-up methods much faster for multi-person scenes. The disadvantage: grouping is hard, especially when people overlap heavily. Accuracy is typically a few points below top-down methods on benchmarks.

OpenPose

OpenPose (2017, CMU) was the first real-time multi-person pose estimation system and remains one of the most recognizable names in the field. It's a bottom-up method that introduced a clever solution to the grouping problem: Part Affinity Fields (PAFs).

Here's the idea. The network has two output branches. The first branch predicts heatmaps for all keypoints (one heatmap per joint type). The second branch predicts Part Affinity Fields — 2D vector fields that encode the location and orientation of limbs. For example, the PAF for the "right shoulder → right elbow" connection is a field of 2D vectors that, along the line from shoulder to elbow, point from shoulder toward elbow. Away from any limbs, the vectors are zero.

To group keypoints into people, OpenPose uses PAFs as a scoring function. For a candidate shoulder-elbow pair, it integrates the PAF vectors along the line connecting them. If the PAF vectors consistently point along that line, the integral is high — these two keypoints probably belong to the same limb. If the vectors point elsewhere or are zero, the integral is low — these keypoints belong to different people. The assignment problem (match shoulders to elbows, elbows to wrists, etc.) is solved greedily using the Hungarian algorithm on these PAF scores.

# Conceptual PAF scoring
def compute_paf_score(paf_field, point_a, point_b, num_samples=10):
    """Score how well a PAF connects two candidate keypoints."""
    # Direction vector from A to B
    direction = point_b - point_a
    norm = np.linalg.norm(direction)
    if norm < 1e-6:
        return 0.0
    direction = direction / norm

    # Sample points along the line from A to B
    score = 0.0
    for i in range(num_samples):
        t = i / (num_samples - 1)
        sample_point = point_a + t * (point_b - point_a)
        x, y = int(sample_point[0]), int(sample_point[1])

        # PAF value at this point (2D vector)
        paf_vector = paf_field[:, y, x]  # shape: [2]

        # Dot product: does the PAF point along our line?
        score += np.dot(paf_vector, direction)

    return score / num_samples

OpenPose processes both branches through multiple stages, where each stage refines the predictions from the previous one. This iterative refinement is crucial — early stages make coarse predictions, and later stages fix mistakes by looking at both the image features and the previous stage's predictions. The typical configuration uses 6 stages, with each stage consuming the concatenation of image features and the previous stage's heatmaps and PAFs.

For practical deployment, OpenPose provides pre-trained models for body (25 keypoints), hands (21 keypoints per hand), and face (70 keypoints). The combined body+hand+face model was one of the first systems to provide full-body pose estimation in real time. However, by modern standards, OpenPose is showing its age — newer methods are both more accurate and faster.

HRNet: High-Resolution Representations

Most CNNs follow a pattern: high resolution early, progressively downsampled through pooling or strided convolutions, ending with low-resolution, high-semantic features. This is great for classification (you just need the global category) but destructive for localization (you need to know exactly where joints are, down to the pixel).

HRNet (High-Resolution Network, 2019) challenges this convention. Instead of a sequential high-to-low pathway, HRNet maintains parallel multi-resolution branches throughout the entire network. It starts with a high-resolution branch. At regular intervals, it spawns new lower-resolution branches (1/2, 1/4, 1/8 scale). Critically, all branches run in parallel and repeatedly exchange information through multi-scale fusion: high-resolution branches send fine spatial details down to low-resolution branches, and low-resolution branches send rich semantic information up to high-resolution branches.

# Conceptual HRNet structure
class HRNetBlock:
    """
    Multi-resolution parallel processing with cross-branch fusion.
    
    Branch 1: 64×64 (high-res)  ←→  fusion  ←→  Branch 2: 32×32
                                 ←→  fusion  ←→  Branch 3: 16×16
                                 ←→  fusion  ←→  Branch 4: 8×8
    
    All branches process simultaneously.
    Fusion: upsample low→high, downsample high→low, combine.
    """
    pass

# The key difference from other architectures:
# ResNet:   256×256 → 128×128 → 64×64 → 32×32 → 16×16 (sequential)
# HRNet:    64×64  →  64×64   →  64×64 →  64×64 →  64×64  (maintained!)
#                     32×32   →  32×32 →  32×32 →  32×32
#                               16×16 →  16×16 →  16×16
#                                         8×8  →   8×8

The result: HRNet's final high-resolution features are both spatially precise (because the high-res branch was never downsampled) and semantically rich (because it continuously receives information from the low-res branches that have larger receptive fields). For heatmap-based pose estimation, this is exactly what you want — precise spatial localization informed by global context.

HRNet became the backbone of choice for pose estimation and achieved state-of-the-art results on COCO and MPII benchmarks. Its design principle — maintain high resolution and fuse scales — also proved effective for semantic segmentation, object detection, and facial landmark detection. Variants like HRNet-W32 and HRNet-W48 (where the number indicates the base channel width) offer different accuracy-speed tradeoffs.

MediaPipe: Real-Time On-Device Pose

MediaPipe (Google) is an open-source framework for building on-device ML pipelines, and its pose estimation model is perhaps the most widely deployed in the world. It runs on smartphones, web browsers (via WebAssembly), and edge devices — anywhere that a full-blown HRNet would be too expensive.

MediaPipe Pose uses a two-stage pipeline designed for speed. First, a lightweight detector finds the person (or uses tracking from the previous frame — detecting every frame is wasteful when the person moves smoothly). Second, a small regression network predicts 33 keypoints directly from the cropped person image. Notice: MediaPipe uses direct coordinate regression rather than heatmaps, which is unusual. The model is tiny — a few MB — and runs at 30+ fps on a mid-range phone.

The accuracy tradeoff is real. On standard benchmarks, MediaPipe is significantly less accurate than HRNet or other heavyweight models. But for many applications — fitness tracking, casual AR effects, gesture control — the accuracy is sufficient, and the ability to run in real-time on a phone is the enabling factor. No one is going to run HRNet-W48 on a smartphone at 30 fps.

# MediaPipe Pose in Python — remarkably simple
import mediapipe as mp
import cv2

mp_pose = mp.solutions.pose
mp_drawing = mp.solutions.drawing_utils

cap = cv2.VideoCapture(0)  # webcam
with mp_pose.Pose(min_detection_confidence=0.5,
                  min_tracking_confidence=0.5) as pose:
    while cap.isOpened():
        ret, frame = cap.read()
        if not ret:
            break

        # Convert BGR to RGB
        rgb = cv2.cvtColor(frame, cv2.COLOR_BGR2RGB)
        results = pose.process(rgb)

        # Draw skeleton overlay
        if results.pose_landmarks:
            mp_drawing.draw_landmarks(
                frame, results.pose_landmarks,
                mp_pose.POSE_CONNECTIONS)

            # Access individual keypoints
            left_elbow = results.pose_landmarks.landmark[mp_pose.PoseLandmark.LEFT_ELBOW]
            print(f"Left elbow: ({left_elbow.x:.2f}, {left_elbow.y:.2f}, {left_elbow.z:.2f})")

        cv2.imshow('Pose', frame)
        if cv2.waitKey(5) & 0xFF == ord('q'):
            break

cap.release()
BlazePose: The Engine Inside MediaPipe

MediaPipe's pose model is called BlazePose internally. It uses a lightweight encoder-decoder architecture inspired by MobileNet, with skip connections for spatial precision. A key trick: instead of detecting the person from scratch each frame, it uses the previous frame's pose to predict a bounding box for the current frame (tracking). The detector only runs when tracking fails (the person moves too fast or leaves the frame). This "detect-then-track" strategy is why MediaPipe feels so fast.

3D Pose Estimation

Everything so far has been 2D: predict (x, y) pixel coordinates on the image plane. But the real world is 3D, and many applications need depth. A physical therapist needs to know the actual angle of a patient's knee, not the apparent angle in a camera image (which changes with viewing direction). A VR avatar needs 3D joint positions to move naturally. Biomechanics researchers need 3D kinematics to analyze gait.

3D pose estimation predicts (x, y, z) coordinates for each joint, where z represents depth (distance from the camera). This is fundamentally harder than 2D because a single 2D image is an ambiguous projection of the 3D world — the same 2D joint positions could correspond to many different 3D poses (a phenomenon called depth ambiguity).

Lifting 2D to 3D

The most practical approach, and the one that works best in the wild: first estimate 2D poses using a strong 2D model (like HRNet), then use a second network to "lift" the 2D coordinates into 3D. The lifting network takes a 2D skeleton (just the joint coordinates, no image) and predicts the depth of each joint.

This two-stage approach has a huge advantage: the 2D model can be trained on massive datasets (COCO has 200K+ annotated people), while the 3D model can be trained on smaller but 3D-annotated datasets (Human3.6M, MPI-INF-3DHP). The 2D model handles the hard part (finding joints in complex images), and the 3D model handles the relatively easier part (resolving depth ambiguity given known 2D positions).

import torch
import torch.nn as nn

class Simple2Dto3DLifter(nn.Module):
    """Lift 2D keypoints to 3D using a simple MLP."""
    def __init__(self, num_joints=17):
        super().__init__()
        self.net = nn.Sequential(
            # Input: 2D coords for 17 joints = 34 values
            nn.Linear(num_joints * 2, 1024),
            nn.BatchNorm1d(1024),
            nn.ReLU(),
            nn.Dropout(0.25),

            nn.Linear(1024, 1024),
            nn.BatchNorm1d(1024),
            nn.ReLU(),
            nn.Dropout(0.25),

            # Output: 3D coords for 17 joints = 51 values
            nn.Linear(1024, num_joints * 3)
        )

    def forward(self, pose_2d):
        # pose_2d: [batch, 34]  →  pose_3d: [batch, 51]
        return self.net(pose_2d)

Direct 3D from Images

End-to-end methods predict 3D pose directly from the image, skipping the explicit 2D stage. These models extend heatmap-based methods into the third dimension, either by predicting 2.5D heatmaps (2D heatmaps plus a depth value per joint) or volumetric heatmaps (3D grids where each voxel represents a probability of a joint being at that spatial location). Volumetric approaches are more accurate but memory-intensive — a 64×64×64 grid per joint is a lot of voxels.

Multi-view approaches use multiple calibrated cameras to triangulate 3D positions from 2D detections. This sidesteps depth ambiguity entirely (two camera angles uniquely determine a 3D point) but requires a controlled multi-camera setup — not practical for in-the-wild applications.

Temporal 3D Pose

For video, temporal consistency provides a powerful signal. A person's joints don't teleport between frames. Models like VideoPose3D take a sequence of 2D poses (from, say, 243 consecutive frames) and predict the 3D pose of the center frame. The temporal context helps resolve depth ambiguity: if a hand appears to be at the same 2D position in two frames but the body is turning, the network can infer that the hand must be at different depths. Temporal models also produce smoother, more physically plausible 3D pose sequences — no jittering or impossible joint angles.

The Training Data Problem

3D pose ground truth is hard to obtain. The standard dataset, Human3.6M, was recorded in a single lab with 11 subjects performing 17 actions. Models trained exclusively on it struggle in the wild — different body proportions, unusual poses, complex backgrounds. Weakly supervised and self-supervised methods that leverage abundant 2D annotations plus physics constraints (bones have fixed lengths, joints have limited rotation ranges) are an active research frontier for closing this domain gap.
