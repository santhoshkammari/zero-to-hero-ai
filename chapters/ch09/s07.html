Object Detection

Chapter 9: CNNs & Computer Vision 3 major subtopics
TL;DR

Image classification answers "what." Object detection answers "what and where" — it draws bounding boxes around every object and labels each one. Two-stage detectors (R-CNN family) propose regions first, then classify — accurate but slow. One-stage detectors (YOLO, SSD, RetinaNet) predict boxes and classes in a single pass — fast but historically less accurate, until Focal Loss closed the gap. Modern detectors have gone anchor-free (FCOS, CenterNet, YOLOv8+) and even transformer-based (DETR), eliminating hand-crafted components one by one. Evaluation uses mAP — mean Average Precision across classes and IoU thresholds.

You've trained a CNN that says "cat" when it sees a cat photo. Great. Now someone hands you a street scene with three cars, two pedestrians, a stop sign, and a dog. Classification gives you one label for the entire image — useless. You need to find every object, draw a box around it, and label it. That's object detection, and it's one of the hardest problems in computer vision. It's the backbone of self-driving cars, security cameras, medical imaging, retail checkout systems, and robotics. Let's trace how we got from painfully slow brute-force approaches to real-time detectors that run on a phone.

The Core Toolkit

Before we dive into architectures, you need four fundamental concepts. Every detector — from R-CNN to YOLOv11 — uses some version of these building blocks.

Bounding Boxes

A bounding box is a rectangle that tightly encloses an object. Sounds simple, but there are three common formats, and mixing them up is one of the most frequent bugs in detection code:

# Three bounding box formats for the same box:

# 1. xyxy: top-left and bottom-right corners
#    (x_min, y_min, x_max, y_max)
box_xyxy = [100, 50, 300, 200]

# 2. xywh: top-left corner + width and height
#    (x_min, y_min, width, height)  — COCO dataset uses this
box_xywh = [100, 50, 200, 150]

# 3. cxcywh: center + width and height
#    (center_x, center_y, width, height)  — YOLO uses this
box_cxcywh = [200, 125, 200, 150]

# All three describe the same rectangle!

Every dataset and every framework picks a different format. COCO uses xywh. Pascal VOC uses xyxy. YOLO uses cxcywh normalized to [0, 1]. If your model is predicting garbage boxes, check your format conversions first. It's always the format conversions.

Intersection over Union (IoU)

Intersection over Union (IoU) measures how much two boxes overlap. It's the area of their overlap divided by the area of their union. An IoU of 1.0 means perfect alignment. An IoU of 0.0 means no overlap at all. Typically, a prediction with IoU ≥ 0.5 against a ground-truth box counts as a correct detection.

  Predicted (blue)         Ground Truth (green)
  ┌──────────────┐
  │              │
  │    ┌─────────┼────────┐
  │    │░░░░░░░░░│        │
  │    │░OVERLAP░│        │
  └────┼─────────┘        │
       │                  │
       └──────────────────┘

  IoU = Area of Overlap (░) / Area of Union (both boxes combined)
      = Intersection / Union

def compute_iou(box1, box2):
    """Compute IoU between two boxes in xyxy format.
    box = [x_min, y_min, x_max, y_max]
    """
    # Intersection coordinates
    x1 = max(box1[0], box2[0])
    y1 = max(box1[1], box2[1])
    x2 = min(box1[2], box2[2])
    y2 = min(box1[3], box2[3])

    # Intersection area (0 if no overlap)
    intersection = max(0, x2 - x1) * max(0, y2 - y1)

    # Union area
    area1 = (box1[2] - box1[0]) * (box1[3] - box1[1])
    area2 = (box2[2] - box2[0]) * (box2[3] - box2[1])
    union = area1 + area2 - intersection

    return intersection / union if union > 0 else 0.0

# Example
pred = [100, 100, 300, 300]    # predicted box
gt   = [150, 120, 320, 310]    # ground truth box
print(f"IoU: {compute_iou(pred, gt):.3f}")  # IoU: 0.520

Anchor Boxes

Anchor boxes (also called prior boxes or default boxes) are pre-defined reference rectangles of various sizes and aspect ratios, placed at regular grid positions across the image. Instead of predicting box coordinates from scratch, the model predicts small adjustments (offsets) from the nearest anchor. This makes learning easier — the network only needs to learn "shift this anchor left by 5 pixels and make it 10% wider" rather than "the box is at (147, 83, 263, 195)." Most classic detectors (Faster R-CNN, SSD, YOLOv2–v5) rely heavily on anchors, though the trend is moving away from them.

  Image divided into grid cells, each with multiple anchors:

  ┌─────┬─────┬─────┬─────┐
  │ ┌┐  │ ┌┐  │ ┌┐  │ ┌┐  │  ← tall, thin anchors
  │ └┘  │ └┘  │ └┘  │ └┘  │
  │┌──┐ │┌──┐ │┌──┐ │┌──┐ │  ← wide, short anchors
  │└──┘ │└──┘ │└──┘ │└──┘ │
  │ ┌─┐ │ ┌─┐ │ ┌─┐ │ ┌─┐ │  ← square anchors
  │ └─┘ │ └─┘ │ └─┘ │ └─┘ │
  ├─────┼─────┼─────┼─────┤
  │ ... │ ... │ ... │ ... │  (repeated for every cell)
  └─────┴─────┴─────┴─────┘

  Each anchor predicts: (Δx, Δy, Δw, Δh, objectness, class_scores)
  Model learns offsets FROM the anchor, not absolute coordinates.

Non-Maximum Suppression (NMS)

A detector typically produces hundreds or thousands of overlapping predictions for the same object. Non-Maximum Suppression (NMS) filters these duplicates: keep the highest-confidence box, remove all other boxes that overlap it beyond an IoU threshold, then repeat for the next-highest box. What remains are clean, non-overlapping detections.

def nms(boxes, scores, iou_threshold=0.5):
    """Non-Maximum Suppression.
    boxes: list of [x1, y1, x2, y2]
    scores: list of confidence scores
    Returns: indices of boxes to keep.
    """
    # Sort by confidence (highest first)
    order = sorted(range(len(scores)), key=lambda i: scores[i], reverse=True)
    keep = []

    while order:
        # Keep the highest-scoring box
        best = order.pop(0)
        keep.append(best)

        # Remove boxes that overlap too much with the best
        remaining = []
        for idx in order:
            if compute_iou(boxes[best], boxes[idx]) < iou_threshold:
                remaining.append(idx)  # not overlapping — keep candidate
        order = remaining

    return keep

# Example: 5 overlapping boxes for the same cat
boxes  = [[100,100,250,250], [110,105,260,255], [105,98,248,245],
          [400,300,550,450], [410,310,560,460]]
scores = [0.9, 0.85, 0.78, 0.95, 0.70]

kept = nms(boxes, scores, iou_threshold=0.5)
print(f"Kept box indices: {kept}")  # [3, 0] — one per object cluster
NMS Pitfall

NMS struggles when objects genuinely overlap — like a crowd of people standing close together. The algorithm suppresses valid detections because it thinks they're duplicates. Soft-NMS decays scores instead of hard-removing them, and DETR avoids NMS entirely by using set-based predictions with Hungarian matching.

Two-Stage Detectors: The Accuracy Path

The earliest deep learning detectors split the problem into two steps: first propose candidate regions, then classify each one. This two-stage approach dominated accuracy benchmarks for years.

  Evolution of Two-Stage Detectors:

  R-CNN (2014)          Fast R-CNN (2015)       Faster R-CNN (2015)
  ─────────────         ────────────────        ──────────────────
  Selective Search      Selective Search        Region Proposal
  → ~2000 regions       → ~2000 regions         Network (RPN)
       │                      │                      │
       ▼                      ▼                      ▼
  Crop & resize each    Share ONE CNN pass      Share ONE CNN pass
  → CNN per region      → RoI Pooling           → RoI Align
       │                      │                      │
       ▼                      ▼                      ▼
  SVM classifier        FC layers per region    FC layers per region
                                                + FPN multi-scale
  ─────────────         ────────────────        ──────────────────
  47 sec/image          ~2 sec/image            ~0.2 sec/image
  NOT end-to-end        Partially trainable     Fully end-to-end

R-CNN: The Pioneer

R-CNN (Regions with CNN features, 2014) was the first to show that CNNs could crush hand-crafted features for detection. The recipe: use a traditional algorithm called Selective Search to propose ~2000 candidate regions in the image. Crop and resize each region to 227×227. Run each crop through AlexNet to get a feature vector. Feed those features to an SVM classifier and a bounding box regressor. It worked — mAP jumped from ~30% to ~58% on Pascal VOC — but it was absurdly slow. You're running a CNN forward pass 2000 times per image. At 47 seconds per image, real-time detection was a fantasy.

Fast R-CNN: Share the Computation

The obvious waste in R-CNN is processing every region independently. Fast R-CNN (2015) fixed this by running the entire image through the CNN once to produce a shared feature map. Then, for each proposed region, it uses RoI Pooling (Region of Interest Pooling) to extract a fixed-size feature chunk from the shared feature map. This made the per-region cost almost free. Training became end-to-end (no separate SVM step), and speed improved ~10×. But the bottleneck shifted: Selective Search still took 2 seconds per image on CPU, dominating the runtime.

Faster R-CNN: Learn the Proposals

Faster R-CNN (2015) asked: why not learn to propose regions too? It replaced Selective Search with a Region Proposal Network (RPN) — a small CNN that slides over the shared feature map and predicts, for each position, whether an object exists and roughly where it is. The RPN shares convolutional features with the detection network, so proposals are nearly free. This made Faster R-CNN fully end-to-end and pushed speed to ~5 FPS on a GPU.

Later improvements added the Feature Pyramid Network (FPN), which builds a top-down pathway with lateral connections to create rich feature maps at multiple scales. This dramatically improved detection of small objects — a persistent weakness of early detectors that only used the final feature map.

RoI Pooling vs. RoI Align

Original RoI Pooling quantizes (rounds) region boundaries to integer coordinates, which introduces spatial misalignment. For classification, this small error doesn't matter much. But for tasks requiring pixel-level precision (like instance segmentation), it's fatal. RoI Align (introduced in Mask R-CNN, 2017) fixes this by using bilinear interpolation instead of rounding, preserving exact spatial correspondence. Today, RoI Align is the standard — there's no reason to use the older version.

One-Stage Detectors: The Speed Path

Two-stage detectors are accurate, but the "propose then classify" pipeline has inherent overhead. What if you could skip proposals entirely and predict boxes and classes in a single shot?

YOLO v1: Detection as Regression

YOLO (You Only Look Once, 2016) reframed object detection as a single regression problem. Divide the image into an S×S grid. Each grid cell predicts B bounding boxes, confidence scores, and class probabilities — all in one forward pass. At 45 FPS, YOLO was the first real-time deep detector. The tradeoff: it struggled with small objects (each cell predicted at most B objects, so dense clusters were missed) and localization accuracy lagged behind Faster R-CNN. But the core insight — detection as regression — changed the field permanently.

SSD: Multi-Scale Predictions

SSD (Single Shot MultiBox Detector, 2016) improved on YOLO by predicting from multiple feature maps at different resolutions. Early layers (high resolution) handle small objects; deeper layers (low resolution) handle large objects. Each location at each scale has a set of default boxes (anchors) of different aspect ratios. This multi-scale approach significantly improved small-object detection without sacrificing speed.

RetinaNet & Focal Loss: Closing the Accuracy Gap

Why were one-stage detectors historically less accurate? Not because of architecture — because of class imbalance. In a typical image, a one-stage detector evaluates ~100,000 anchor locations. Maybe 10 contain objects. The other 99,990 are easy negatives (background). Standard cross-entropy loss is dominated by these easy examples — the model spends all its gradient budget on "yep, that's background" instead of learning to distinguish a cat from a dog.

RetinaNet (2017) solved this with Focal Loss: a modified cross-entropy that down-weights easy examples and focuses on hard ones. The formula adds a modulating factor (1 − p_t)^γ, where p_t is the predicted probability for the correct class. When the model is already confident (p_t close to 1), the loss is near zero. When the model is uncertain (p_t close to 0.5), the loss is full-strength. With γ = 2 (the sweet spot), RetinaNet matched two-stage accuracy while maintaining one-stage speed. This was a landmark result.

Why Focal Loss Matters Beyond Detection

Class imbalance isn't unique to object detection — it plagues fraud detection, medical diagnosis, anomaly detection, and any task where positives are rare. Focal Loss (and its variants) has become a standard tool in the ML engineer's toolkit for any imbalanced problem.

DETR: Transformers Enter Detection

DETR (DEtection TRansformer, 2020) threw out anchors, NMS, and hand-crafted components entirely. It feeds CNN features into a Transformer encoder-decoder. The decoder takes a fixed set of learned "object queries" and, through attention, maps each query to an object or "no object." Training uses Hungarian matching — a combinatorial algorithm that finds the optimal one-to-one assignment between predictions and ground-truth objects, treating detection as a set prediction problem.

DETR is elegant and powerful, but the original version converged slowly (500 epochs vs. ~36 for Faster R-CNN) and struggled with small objects. Follow-up work like Deformable DETR, DINO, and RT-DETR addressed these issues. RT-DETR in particular achieved real-time speed, making transformer-based detection practical for deployment.

Anchor-Free Detectors

Anchors were a useful crutch, but they introduced painful hyperparameters: how many anchors? What sizes and aspect ratios? How do you match anchors to ground-truth boxes? Anchor-free detectors sidestep all of this by directly predicting object properties from feature map locations.

CornerNet (2018) detects objects as pairs of top-left and bottom-right corner keypoints. CenterNet (2019) simplifies further: detect the center point of each object, then regress the width and height directly from that point. FCOS (Fully Convolutional One-Stage, 2019) predicts, for each pixel in the feature map, the distances to the four sides of the bounding box (left, top, right, bottom). If a pixel falls inside a ground-truth box, it's a positive sample; otherwise, negative. No anchors, no proposals, no IoU-based matching at training time.

Anchor-free designs have become the dominant paradigm. YOLOv8 and later YOLO versions are all anchor-free. They're simpler to implement, easier to tune, and perform as well as or better than anchor-based approaches.

The YOLO Evolution

No discussion of object detection is complete without tracing YOLO's evolution. It's the most deployed detector family in the real world, and each version introduced ideas that shaped the field.

Version Year Key Innovation Anchors?
YOLOv1 2016 Detection as regression. Single-pass, 45 FPS. No
YOLOv2 2017 Batch norm, anchor boxes (k-means clustered), multi-scale training. Yes
YOLOv3 2018 FPN-like multi-scale predictions, Darknet-53 backbone. Yes
YOLOv4 2020 Bag of freebies (Mosaic augmentation, CIoU loss), CSPDarknet backbone. Yes
YOLOv5 2020 PyTorch-native, auto-anchor, export-friendly. Community-driven (Ultralytics). Yes
YOLOv7 2022 E-ELAN architecture, model reparameterization, auxiliary heads. Yes
YOLOv8 2023 Anchor-free, decoupled head (separate classification & regression branches). No
YOLOv9 2024 Programmable Gradient Information (PGI), GELAN architecture. No
YOLOv10 2024 NMS-free training (consistent dual assignments), efficiency-accuracy redesign. No
YOLOv11 2024 C3k2 blocks, attention mechanisms, improved small-object detection. No

Notice the trend: YOLOv1 started anchor-free, v2–v7 added anchors for better accuracy, and v8 onward went anchor-free again — but this time with the architecture maturity to make it work. The other big shift is the decoupled head in YOLOv8: instead of one branch predicting both class and box, two separate branches handle classification and regression independently. This improves both tasks because they have conflicting optimization needs — classification wants translation-invariant features, while regression needs precise spatial information.

How to Choose

Evaluation: Understanding mAP

You'll see detectors compared by mAP (mean Average Precision), but few people truly understand what it measures. Let's unpack it step by step.

First, for a single class (say "car"), rank all detections across the dataset by confidence score. Go through the list from highest to lowest confidence. At each detection, if it matches a ground-truth box with IoU ≥ threshold, it's a True Positive (TP); otherwise, it's a False Positive (FP). At each point in the ranked list, compute Precision (TP / all detections so far) and Recall (TP / all ground-truth cars). Plot these as a Precision-Recall (PR) curve. The Average Precision (AP) for "car" is the area under this PR curve.

Now repeat for every class and take the mean — that's mAP. But the IoU threshold matters enormously:

  • mAP@0.5 (Pascal VOC metric): A detection counts if IoU ≥ 0.5. Lenient — your box can be fairly sloppy.
  • mAP@0.75: Stricter — requires tighter boxes.
  • mAP@0.5:0.95 (COCO primary metric): Average mAP across IoU thresholds from 0.5 to 0.95 in steps of 0.05. This is the gold standard because it rewards both detection ability and localization precision.

COCO also reports mAPS (small objects, area < 32²), mAPM (medium, 32²–96²), and mAPL (large, > 96²). Small-object mAP is where most detectors fall apart — it's the number that separates good architectures from great ones.

Bounding Box Regression & Loss Functions

Every detector needs to predict bounding box coordinates. The model outputs raw numbers; a loss function tells it how wrong those numbers are. The evolution of box regression losses mirrors the evolution of detectors themselves — each generation fixed a problem with the previous one.

L1 and Smooth L1 Loss

The simplest approach: compute the absolute difference (L1) or a smoothed version (Smooth L1 / Huber loss) between predicted and ground-truth coordinates. Smooth L1 is less sensitive to outliers than L2 (squared error) because it behaves like L1 for large errors and like L2 for small ones. Faster R-CNN uses Smooth L1. The problem: L1 treats all coordinates independently. A box that's 10 pixels off on width is penalized the same whether the box is 20 pixels wide or 2000 pixels wide.

IoU-Based Losses

IoU Loss directly optimizes what we actually care about — the overlap between predicted and ground-truth boxes. Loss = 1 − IoU. But standard IoU loss has a gradient problem: when two boxes don't overlap at all (IoU = 0), the gradient is zero. The model gets no signal about how to move the predicted box toward the target.

GIoU (Generalized IoU) fixes this by adding a penalty based on the smallest enclosing box. Even when boxes don't overlap, GIoU provides a meaningful gradient pointing in the right direction.

DIoU (Distance IoU) goes further: it explicitly minimizes the distance between the centers of the predicted and target boxes. This speeds up convergence because the model learns to move the box center first, then refine the size.

CIoU (Complete IoU) adds an aspect ratio consistency term on top of DIoU. It penalizes the model when the predicted box's aspect ratio differs from the ground truth's, even if the IoU is already decent. CIoU is the default in YOLOv4 and later — it converges faster and produces tighter boxes.

  Loss Function Evolution:

  L1 / Smooth L1
    │  Problem: treats coordinates independently, scale-unaware
    ▼
  IoU Loss  (1 − IoU)
    │  Problem: zero gradient when boxes don't overlap
    ▼
  GIoU Loss  (adds enclosing box penalty)
    │  Problem: slow convergence when boxes are far apart
    ▼
  DIoU Loss  (adds center distance penalty)
    │  Problem: doesn't consider aspect ratio
    ▼
  CIoU Loss  (adds aspect ratio consistency)
    │  Current default in modern YOLO versions
    ▼
  SIoU, WIoU, EIoU ...  (further refinements)
Common Mistake

Don't use plain L1/L2 loss for bounding box regression in modern detectors. IoU-based losses (CIoU or GIoU at minimum) are strictly better — they're scale-invariant, directly optimize the evaluation metric, and converge faster. If you're getting poor localization despite reasonable classification, switching to CIoU loss is often the fix.

Putting It All Together

Object detection has come remarkably far in a decade. Here's the trajectory:

  2014          2015            2016          2017         2020         2023+
  ────          ────            ────          ────         ────         ─────
  R-CNN     → Faster R-CNN  → YOLO/SSD   → RetinaNet → DETR       → RT-DETR
  (47s/img)   (0.2s/img)      (real-time)   (focal)     (no NMS)     (real-time
  (SVMs)      (end-to-end)    (regression)  (imbalance   (Hungarian    + no NMS
  (2000 CNNs) (RPN)           (single-pass)  solved)     matching)    + transformer)

  Trend: fewer hand-crafted components, more learned components.
  Anchors → Anchor-free.  NMS → NMS-free.  CNN → Transformer.

The field is converging on detectors that are fast, accurate, and simple — with fewer hyperparameters to tune and fewer post-processing hacks. If you're starting a new detection project today, YOLOv8 or RT-DETR is your starting point. If you need to understand the fundamentals to debug and improve, now you know the full story of how each piece was invented to solve the previous generation's problems.
