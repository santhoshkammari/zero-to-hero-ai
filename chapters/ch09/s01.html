Convolution Operation

Chapter 9: CNNs & Computer Vision 12 subtopics
TL;DR

A convolution slides a small learnable filter across an input, producing a feature map that detects local patterns. Unlike fully-connected layers, convolutions exploit spatial structure through weight sharing and local connectivity — slashing parameter counts from billions to thousands. Understanding kernel size, stride, padding, dilation, and grouped/depthwise variants gives you the full toolkit to reason about any CNN architecture you'll encounter.

Why convolution? Why not just flatten an image into a vector and throw a fully-connected layer at it? Let's do the math. A modest 224×224 RGB image has 150,528 input values. A single fully-connected layer mapping to just 1,000 hidden units would need 150,528 × 1,000 = 150 million parameters. For one layer. That's insane. It would overfit immediately, train slowly, and ignore the most obvious fact about images: nearby pixels are related. A cat's ear is a local pattern — it doesn't care what's happening in the opposite corner of the image.

Convolution exploits this spatial structure. Instead of connecting every input to every output, you slide a small filter — say 3×3 — across the image, computing a dot product at each position. The same filter is reused everywhere. This is called weight sharing, and it means you go from 150 million parameters to just 27 (3×3×3 for RGB). That one idea — local connectivity plus weight sharing — is what makes CNNs practical.

1D, 2D, and 3D Convolutions

The dimensionality of a convolution refers to the number of spatial dimensions the filter slides across. A 1D convolution slides along a single axis — think of an audio waveform or a time series. Your filter might be length 5, and it slides across a sequence of 1,000 timesteps, producing a 1D output. This is the workhorse behind temporal CNNs used in speech processing and text classification.

A 2D convolution slides across height and width — the bread and butter of image processing. Your 3×3 filter moves both horizontally and vertically across a 2D feature map. This is what people usually mean when they say "convolution" in a deep learning context.

A 3D convolution slides across three spatial dimensions. The classic use case is video, where you have height, width, and time. Medical imaging uses 3D convolutions for volumetric data like CT scans and MRI. The filter is a small cube — maybe 3×3×3 — and it captures spatiotemporal patterns like "an object moving left over three frames."

# PyTorch convolution layers for each dimensionality
import torch
import torch.nn as nn

# 1D: (batch, channels, length) — e.g., audio signal
conv1d = nn.Conv1d(in_channels=1, out_channels=16, kernel_size=5, padding=2)
x_1d = torch.randn(1, 1, 1000)   # 1 sample, 1 channel, 1000 timesteps
out_1d = conv1d(x_1d)             # shape: (1, 16, 1000)

# 2D: (batch, channels, height, width) — e.g., image
conv2d = nn.Conv2d(in_channels=3, out_channels=64, kernel_size=3, padding=1)
x_2d = torch.randn(1, 3, 224, 224)  # 1 RGB image
out_2d = conv2d(x_2d)                # shape: (1, 64, 224, 224)

# 3D: (batch, channels, depth, height, width) — e.g., video
conv3d = nn.Conv3d(in_channels=3, out_channels=32, kernel_size=3, padding=1)
x_3d = torch.randn(1, 3, 16, 112, 112)  # 16-frame video clip
out_3d = conv3d(x_3d)                     # shape: (1, 32, 16, 112, 112)

Filters and Kernels

A kernel (or filter) is a small tensor of learnable weights. For a 2D convolution on an RGB image, a single kernel has shape (3, k, k) — it spans all input channels but covers only a small spatial region of k×k pixels. The network learns these weights during training, just like it learns weights in a fully-connected layer.

Each kernel detects one kind of pattern. Early layers typically learn kernels that respond to edges at various orientations — horizontal, vertical, diagonal. Deeper layers compose these into more complex patterns. The number of kernels you use determines the number of output channels. If you have 64 kernels, you get 64 feature maps out, each one "looking for" a different pattern.

Here's an important subtlety: the word "filter" and "kernel" are used interchangeably in practice, but technically a filter refers to the full 3D block (spanning all input channels), while a kernel refers to one 2D slice. A filter for processing an RGB input with a 3×3 spatial size is a (3, 3, 3) tensor — three 3×3 kernels stacked, one per input channel. The outputs from each input channel are summed element-wise to produce a single 2D feature map.

Classic Hand-Designed Kernels

Before deep learning, people hand-designed kernels. A Sobel filter detects horizontal or vertical edges. A Gaussian kernel blurs an image. A Gabor filter detects texture at a specific frequency and orientation — neuroscientists found that cells in the primary visual cortex respond to patterns remarkably similar to Gabor filters. The beauty of learned convolutions is that the network discovers whatever kernels are useful for the task at hand, often rediscovering these classic patterns in the first layer and building far more sophisticated detectors in deeper layers.

Stride

Stride controls how many positions the filter moves at each step. With stride 1, the filter moves one pixel at a time — you get a dense output. With stride 2, the filter jumps two pixels, cutting the output spatial dimensions in half. Stride is one of two common ways to downsample feature maps (the other being pooling, covered next section).

Modern architectures increasingly prefer strided convolutions over pooling for downsampling. The argument is simple: if you're going to reduce spatial resolution, why not learn how to do it? A strided convolution combines feature extraction and downsampling in one operation, while max pooling throws away information in a fixed, non-learnable way. ResNet, for example, uses stride-2 convolutions at the start of each stage.

Padding

Without padding, every convolution shrinks the spatial dimensions. A 3×3 filter on a 5×5 input produces a 3×3 output — you lose one pixel on each side. Over many layers, your feature maps would shrink to nothing. Padding adds extra values (usually zeros) around the border of the input to control the output size.

Valid padding (or "no padding") means no padding at all — the output shrinks. Same padding adds enough zeros so the output has the same spatial size as the input (assuming stride 1). For a 3×3 kernel, same padding adds 1 pixel on each side. For a 5×5 kernel, it adds 2 pixels.

There's also causal padding, which matters for 1D temporal data. It pads only the left side, ensuring the output at timestep t depends only on inputs at time ≤ t. This preserves the causal structure — the model can't peek into the future. WaveNet used causal padding for audio generation.

# Padding examples in PyTorch
# Valid (no padding) — output shrinks
conv_valid = nn.Conv2d(3, 16, kernel_size=3, padding=0)  # 224→222

# Same padding — output same size (stride=1)
conv_same = nn.Conv2d(3, 16, kernel_size=3, padding=1)   # 224→224

# Same padding for 5×5 kernel
conv_same5 = nn.Conv2d(3, 16, kernel_size=5, padding=2)  # 224→224

# Asymmetric padding via nn.ZeroPad2d
padder = nn.ZeroPad2d((1, 1, 2, 2))  # left, right, top, bottom

Output Size Formula

This is the one formula you absolutely must memorize. For any spatial dimension:

output_size = floor((input_size + 2 * padding - dilation * (kernel_size - 1) - 1) / stride + 1)

For the common case of no dilation (dilation=1), this simplifies to:

output_size = floor((input_size + 2 * padding - kernel_size) / stride + 1)

Let's verify: input 224, kernel 3, padding 1, stride 1. Output = (224 + 2 − 3) / 1 + 1 = 224. Same padding works. Now with stride 2: output = (224 + 2 − 3) / 2 + 1 = 112. Halved, as expected. Sanity-check everything with this formula — mismatched tensor shapes are the #1 source of CNN bugs.

# Quick helper to verify output sizes
def conv_output_size(input_size, kernel_size, stride=1, padding=0, dilation=1):
    return (input_size + 2 * padding - dilation * (kernel_size - 1) - 1) // stride + 1

print(conv_output_size(224, 3, stride=1, padding=1))   # 224
print(conv_output_size(224, 3, stride=2, padding=1))   # 112
print(conv_output_size(224, 7, stride=2, padding=3))   # 112  (ResNet stem)

Dilation (Atrous Convolution)

Dilated convolution (also called atrous convolution) inserts gaps between kernel elements. A 3×3 kernel with dilation 2 looks at a 5×5 region but only samples 9 positions within it — every other pixel. This dramatically increases the receptive field without adding parameters or reducing spatial resolution.

Why is this useful? In semantic segmentation, you need to classify every pixel, so you can't afford to downsample too aggressively. But you also need large receptive fields to understand context — is this green pixel grass or a tree leaf? Dilated convolutions let you see a wide area without shrinking the feature map. DeepLab used dilated convolutions to great effect, replacing pooling with progressively larger dilation rates (1, 2, 4, 8) to capture multi-scale context.

# Dilated convolution — same parameters, larger receptive field
conv_dilated = nn.Conv2d(64, 64, kernel_size=3, padding=2, dilation=2)
# Effective receptive field: 5×5, but only 9 learned parameters per input channel
# padding=2 maintains spatial size: (224 + 4 - 2*(3-1) - 1)/1 + 1 = 224

# Multi-scale dilation (Atrous Spatial Pyramid Pooling idea)
rates = [1, 6, 12, 18]
aspp_convs = nn.ModuleList([
    nn.Conv2d(256, 256, 3, padding=r, dilation=r) for r in rates
])

Depthwise Separable Convolution

Standard convolution is expensive. A 3×3 conv from 256 channels to 256 channels requires 256 × 256 × 3 × 3 = 589,824 parameters. Depthwise separable convolution breaks this into two cheaper steps. First, a depthwise convolution applies one filter per input channel independently — 256 separate 3×3 filters, one per channel. Then, a pointwise convolution (1×1 conv) mixes channels — mapping 256 channels to 256 channels with a 1×1 kernel.

The parameter count: depthwise has 256 × 3 × 3 = 2,304 parameters. Pointwise has 256 × 256 × 1 × 1 = 65,536. Total: 67,840 — roughly 8.7× fewer parameters than standard convolution. The computational savings are similar. This factorization was the key insight behind MobileNet, making it feasible to run CNNs on phones and edge devices.

# Depthwise separable convolution in PyTorch
class DepthwiseSeparableConv(nn.Module):
    def __init__(self, in_ch, out_ch, kernel_size=3, stride=1, padding=1):
        super().__init__()
        # Depthwise: groups=in_ch means each channel is convolved independently
        self.depthwise = nn.Conv2d(in_ch, in_ch, kernel_size,
                                    stride=stride, padding=padding, groups=in_ch)
        # Pointwise: 1×1 conv to mix channels
        self.pointwise = nn.Conv2d(in_ch, out_ch, kernel_size=1)

    def forward(self, x):
        x = self.depthwise(x)
        x = self.pointwise(x)
        return x

dw_sep = DepthwiseSeparableConv(256, 256)
print(sum(p.numel() for p in dw_sep.parameters()))  # ~67,840 vs 589,824

Group Convolution

Group convolution splits input channels into groups and applies separate convolutions to each group. If you have 256 input channels and 4 groups, each group processes 64 channels independently. AlexNet originally used group convolutions to split computation across two GPUs — it was a hardware hack that turned out to have a nice regularization effect.

Depthwise convolution is the extreme case where groups equals the number of input channels. Setting groups to an intermediate value gives you a tradeoff between standard convolution (groups=1, full channel mixing) and depthwise convolution (groups=C, no channel mixing). ResNeXt used 32 groups with great success, showing that "cardinality" (number of groups) is an effective dimension to scale, often better than just making the network wider or deeper.

# Group convolution examples
standard = nn.Conv2d(256, 256, 3, padding=1, groups=1)   # standard: 589,824 params
grouped  = nn.Conv2d(256, 256, 3, padding=1, groups=32)  # ResNeXt-style: 18,432 params
depthwise = nn.Conv2d(256, 256, 3, padding=1, groups=256) # depthwise: 2,304 params

for name, conv in [("standard", standard), ("grouped", grouped), ("depthwise", depthwise)]:
    params = sum(p.numel() for p in conv.parameters())
    print(f"{name}: {params:,} parameters")

Transposed Convolution

Transposed convolution (sometimes misleadingly called "deconvolution") goes in the opposite direction — it upsamples. If a stride-2 convolution halves your feature map, a stride-2 transposed convolution doubles it. This is essential in architectures that need to go from low-resolution to high-resolution, like autoencoders, generators in GANs, and segmentation models.

How does it work? Conceptually, you insert zeros between input elements (to undo the stride), then apply a regular convolution. The output is larger than the input. The name "transposed" comes from the fact that the operation can be expressed as multiplication by the transpose of the convolution matrix.

One notorious issue with transposed convolutions is checkerboard artifacts — grid-like patterns in the output caused by uneven overlap when stride doesn't evenly divide the kernel size. A common fix is to use bilinear upsampling followed by a regular convolution instead. But transposed convolutions remain widely used because they're learnable and fast.

# Transposed convolution — doubles spatial size
up = nn.ConvTranspose2d(256, 128, kernel_size=4, stride=2, padding=1)
x = torch.randn(1, 256, 14, 14)
print(up(x).shape)  # (1, 128, 28, 28) — spatial size doubled

# Alternative: upsample + conv (avoids checkerboard artifacts)
class UpsampleConv(nn.Module):
    def __init__(self, in_ch, out_ch):
        super().__init__()
        self.up = nn.Upsample(scale_factor=2, mode='bilinear', align_corners=False)
        self.conv = nn.Conv2d(in_ch, out_ch, 3, padding=1)

    def forward(self, x):
        return self.conv(self.up(x))

Deformable Convolution

Standard convolutions sample from a fixed grid. A 3×3 kernel always looks at the same 9 positions relative to the center. Deformable convolution adds learnable offsets to each sampling position. The kernel might shift its attention slightly up-left for one pixel and down-right for another, adapting to the geometry of the object.

This is incredibly powerful for tasks where objects have irregular shapes or undergo geometric transformations. Think detecting a person in various poses — their limbs bend, their body rotates. A standard convolution can only learn one rigid 3×3 pattern at each position. A deformable convolution adapts its sampling grid to follow the shape of whatever it's looking at.

In practice, a small auxiliary convolution predicts 2D offsets for each position in the filter. Deformable Convolution v2 added a learned modulation weight per sampling point, allowing the network to also control how much each position contributes. Both COCO object detection winners and video understanding models have used deformable convolutions extensively.

Parameter Count Calculation

Knowing how to calculate parameter counts is essential for debugging, for comparing architectures, and for interview questions. Here's the formula for a standard 2D convolution:

parameters = out_channels × (in_channels / groups × kernel_h × kernel_w + bias)
           = out_channels × (in_channels / groups × kernel_h × kernel_w + 1)  # with bias
           = out_channels × (in_channels / groups × kernel_h × kernel_w)      # without bias

Let's work through real examples. ResNet's first conv layer: 3 input channels, 64 output channels, 7×7 kernel. Parameters = 64 × (3 × 7 × 7 + 1) = 64 × 148 = 9,472. A mid-network 3×3 conv from 256 to 256: 256 × (256 × 3 × 3 + 1) = 256 × 2,305 = 590,080. Notice how the deep layers dominate — the first layer has almost no parameters despite its large 7×7 kernel, because it only has 3 input channels.

# Verify parameter counts programmatically
conv = nn.Conv2d(3, 64, kernel_size=7, bias=True)
total = sum(p.numel() for p in conv.parameters())
print(f"ResNet stem: {total:,} params")  # 9,472

conv2 = nn.Conv2d(256, 256, kernel_size=3, bias=True)
total2 = sum(p.numel() for p in conv2.parameters())
print(f"Mid-network 3×3: {total2:,} params")  # 590,080

# FLOPs for a convolution (multiply-accumulate operations)
# FLOPs = output_h × output_w × out_channels × (in_channels × kernel_h × kernel_w)
# For 256→256, 3×3 on 14×14 feature map:
flops = 14 * 14 * 256 * (256 * 3 * 3)
print(f"FLOPs: {flops:,}")  # 115,605,504 ≈ 116M MACs

Convolution as Matrix Multiplication

Under the hood, most deep learning frameworks don't actually slide a kernel pixel by pixel. That would be slow. Instead, they use a trick called im2col (image to column). The idea: rearrange the input so that each receptive field becomes a column in a matrix. Then the entire convolution becomes a single matrix multiplication, which GPUs are absurdly fast at.

For a 3×3 conv on a 4×4 input (1 channel, stride 1, no padding), im2col extracts all 4 overlapping 3×3 patches, flattens each into a 9-element column, and stacks them into a 9×4 matrix. The kernel is flattened into a row vector of length 9. A single matrix multiply produces the 4-element output. Scale this up to real sizes with multiple channels and filters, and you get a large matrix multiply that saturates GPU cores.

This is why convolution performance depends heavily on matrix shapes. Odd channel counts, prime number spatial sizes, and non-standard kernel sizes can all hurt performance by reducing the efficiency of the underlying GEMM (General Matrix Multiply). That's the practical reason so many architectures use channel counts that are powers of 2 or multiples of 8 — it's not mathematical necessity, it's hardware optimization.

The 5×5 vs Two Stacked 3×3 Trick

Two consecutive 3×3 convolutions see the same 5×5 receptive field as a single 5×5 convolution, but with fewer parameters (2 × 9 = 18 vs 25) and two non-linearities instead of one — strictly more expressive. This insight, introduced by VGGNet, is why almost no modern architecture uses kernels larger than 3×3 (with the notable exception of ConvNeXt's 7×7 depthwise convolutions). Three stacked 3×3 layers cover a 7×7 receptive field with 27 weights instead of 49, plus three ReLUs instead of one.

Convolution vs Cross-Correlation

Technically, what deep learning calls "convolution" is actually cross-correlation. True mathematical convolution flips the kernel both horizontally and vertically before sliding it. Cross-correlation doesn't flip. The distinction doesn't matter in practice because the kernel weights are learned — whether the kernel starts flipped or not, gradient descent arrives at the same solution. But if you ever read a signal processing textbook and wonder why formulas look different from your PyTorch code, this is why. Every major framework uses cross-correlation and calls it convolution. Nobody loses sleep over it.

Upsampling Methods

Beyond transposed convolutions, there are several ways to increase spatial resolution. Nearest-neighbor upsampling simply repeats each pixel — fast and simple, but blocky. Bilinear interpolation computes a weighted average of the four nearest pixels — smoother but still parameter-free. Pixel shuffle (sub-pixel convolution) rearranges channels into spatial dimensions. If you have a feature map with shape (batch, C×r², H, W), pixel shuffle reshapes it to (batch, C, H×r, W×r), effectively learning an upsampling in channel space. This was introduced for super-resolution and is widely used because it's learnable and artifact-free.

# Upsampling methods comparison
x = torch.randn(1, 64, 16, 16)

# Nearest-neighbor
up_nn = nn.Upsample(scale_factor=2, mode='nearest')
print(up_nn(x).shape)  # (1, 64, 32, 32)

# Bilinear
up_bi = nn.Upsample(scale_factor=2, mode='bilinear', align_corners=False)
print(up_bi(x).shape)  # (1, 64, 32, 32)

# Pixel shuffle (sub-pixel convolution)
# First expand channels by r², then rearrange into spatial dims
conv_expand = nn.Conv2d(64, 64 * 4, kernel_size=3, padding=1)  # 4 = 2²
shuffle = nn.PixelShuffle(upscale_factor=2)
print(shuffle(conv_expand(x)).shape)  # (1, 64, 32, 32)
Common Gotcha

When you see shape mismatches in a CNN, the output size formula is your best friend. Check every layer: floor((H + 2p - d(k-1) - 1) / s + 1). Off-by-one errors with even-sized kernels and stride 2 are particularly sneaky — a 4×4 kernel with stride 2 and padding 1 produces different output than a 3×3 kernel with the same stride and padding. Always compute manually or use torchinfo.summary() to verify.

Computational Analysis Summary

For a single conv layer with output spatial size H_out × W_out, the FLOPs are H_out × W_out × C_out × (C_in × K × K). Memory for activations is batch × C_out × H_out × W_out × 4 bytes (float32). Memory for weights is C_out × C_in × K × K × 4 bytes. In real networks, the early high-resolution layers dominate memory (large spatial dimensions), while the late low-resolution layers dominate parameter counts (many channels). This asymmetry is why tricks like depthwise convolutions and stride-2 downsampling in early layers matter so much — they attack the expensive parts of the computation.
