CNN Building Blocks

Chapter 9: CNNs & Computer Vision Convolution · Pooling · Feature hierarchies · Receptive field

A convolution is a pattern detector that slides across an image. That's really the whole idea. A small matrix of learned weights — the kernel or filter — moves position by position across the input, computing a dot product at each location. The output is a feature map: a spatial grid showing where that pattern was detected and how strongly.

Why not just flatten the image into a vector and use fully-connected layers? Three devastating reasons. First, parameter explosion: a modest 224×224×3 input has 150,528 pixels. A single hidden layer of 1,000 neurons would need 150 million weights — for one layer. Second, spatial blindness: an FC layer treats pixel (0,0) and pixel (223,223) as equally connected. It has no notion of "nearby." Third, no translation equivariance: a cat in the top-left would activate completely different weights than the same cat in the bottom-right. The network would need to learn separate detectors for every possible position. Convolutions solve all three problems at once through weight sharing and local connectivity.

The Convolution Operation — Going Deep

What's Actually Happening

Think of a 3×3 kernel as a tiny "template." At each position, it looks at a 3×3 patch of the input, multiplies each pixel value by the corresponding kernel weight, sums everything up, and writes the result to the output feature map. Then it slides one pixel over and does it again.

Here's the key insight: the same kernel is used at every position. A horizontal edge detector doesn't care whether the edge is at the top or bottom of the image — it uses the same 9 weights everywhere. This is parameter sharing, and it's why a conv layer with a 3×3 kernel on 64 input channels only needs 576 weights per filter, not millions.

import torch
import torch.nn as nn

# A single convolution layer
conv = nn.Conv2d(
    in_channels=3,     # RGB input
    out_channels=16,   # learn 16 different filters
    kernel_size=3,     # 3×3 filter
    stride=1,          # move 1 pixel at a time
    padding=1           # pad input so output size = input size
)

x = torch.randn(1, 3, 224, 224)  # batch=1, 3 channels, 224×224
out = conv(x)
print(out.shape)  # torch.Size([1, 16, 224, 224])

# Each of the 16 filters produced a 224×224 feature map
# Total parameters: (3×3×3 + 1) × 16 = 448
print(sum(p.numel() for p in conv.parameters()))  # 448

Let's unpack what those 16 filters are doing. Each one learns a different pattern. In early layers of a trained network, you'd typically see filters that respond to horizontal edges, vertical edges, diagonal edges, color blobs, and simple textures. The network wasn't told to learn these — gradient descent discovered that these are the useful low-level features for whatever task it's solving.

The Math, Concisely

For a 2D convolution with kernel K of size k×k applied to input X:

# Output at position (i, j):
# out[i, j] = Σ_m Σ_n  X[i+m, j+n] · K[m, n]  +  bias
#
# For multi-channel: sum over input channels too
# out[i, j] = Σ_c Σ_m Σ_n  X[c, i+m, j+n] · K[c, m, n]  +  bias

Technically, deep learning "convolution" is actually cross-correlation — we don't flip the kernel. Nobody cares about this distinction in practice because the kernel weights are learned, so the flipped version of any optimal kernel is also optimal. But it's worth knowing when you're reading signal-processing literature where the distinction matters.

Stride: How Far the Kernel Jumps

With stride 1, the kernel moves one pixel at a time. With stride 2, it skips every other position, halving the output dimensions. Stride is the simplest way to downsample.

When to use stride > 1: When you want to reduce spatial dimensions and extract features in one operation. ResNet uses stride-2 convolutions instead of separate pooling layers at each downsampling point — it's more parameter-efficient than pooling followed by convolution, and the network can learn how to downsample.

# Stride-2 conv: 224×224 → 112×112 in one shot
downsample = nn.Conv2d(64, 128, kernel_size=3, stride=2, padding=1)
x = torch.randn(1, 64, 224, 224)
print(downsample(x).shape)  # torch.Size([1, 128, 112, 112])

Padding: Controlling the Output Size

"Valid" padding (no padding): the kernel only visits positions where it fully fits inside the input. A 3×3 kernel on a 5×5 input gives a 3×3 output — you lose border pixels. "Same" padding: you pad the input with zeros so the output has the same spatial dimensions as the input. For a 3×3 kernel, that's 1 pixel of padding on each side.

Why does this matter? Without padding, your feature maps shrink at every layer. In a 20-layer network, you'd lose 40 pixels of width — that's a lot. Same padding lets you stack many conv layers without shrinking, then downsample explicitly with stride or pooling when you choose to.

# "Valid" — no padding
conv_valid = nn.Conv2d(1, 1, kernel_size=3, padding=0)
x = torch.randn(1, 1, 32, 32)
print(conv_valid(x).shape)  # torch.Size([1, 1, 30, 30]) — lost 2 pixels

# "Same" — preserve spatial dims
conv_same = nn.Conv2d(1, 1, kernel_size=3, padding=1)
print(conv_same(x).shape)   # torch.Size([1, 1, 32, 32]) — same size

Dilation: Expanding the Receptive Field Without More Parameters

A dilated (or "atrous") convolution spaces out the kernel elements with gaps. A 3×3 kernel with dilation=2 covers a 5×5 area but only uses 9 weights — the same parameter count as a standard 3×3 kernel but with a much wider field of view.

When to use dilation: When you need a large receptive field but can't afford the parameters or computation of a large kernel. This is critical in semantic segmentation (DeepLab uses it extensively) and audio processing (WaveNet uses exponentially increasing dilation to capture long-range dependencies). The trade-off is that you skip intermediate positions, which can create "gridding artifacts" if you're not careful.

# Standard 3×3 covers 3×3 = 9 positions
conv_standard = nn.Conv2d(64, 64, kernel_size=3, padding=1, dilation=1)

# Dilated 3×3 covers 5×5 area but still only 9 weights
conv_dilated = nn.Conv2d(64, 64, kernel_size=3, padding=2, dilation=2)

# Heavily dilated — covers 9×9 area with 9 weights
conv_heavy = nn.Conv2d(64, 64, kernel_size=3, padding=4, dilation=4)

x = torch.randn(1, 64, 32, 32)
# All produce same output size (with appropriate padding), but
# each "sees" a different area of the input
print(conv_standard(x).shape)  # [1, 64, 32, 32]
print(conv_dilated(x).shape)   # [1, 64, 32, 32]
print(conv_heavy(x).shape)     # [1, 64, 32, 32]

Output Size Formula

One formula, commit it to memory:

output_size = (W - K + 2P) / S + 1

# W = input size, K = kernel size, P = padding, S = stride
# Example: W=224, K=3, P=1, S=1 → (224 - 3 + 2) / 1 + 1 = 224
# Example: W=224, K=7, P=3, S=2 → (224 - 7 + 6) / 2 + 1 = 112

That second example is exactly what ResNet's first layer does: a 7×7 conv with stride 2, taking 224×224 down to 112×112 in one shot.

Convolutions in Different Dimensions

The same sliding-window idea works in any dimension. Conv1d slides a 1D kernel along a sequence — used for audio, time series, and text (before transformers took over). Conv2d slides a 2D kernel across height and width — the standard for images. Conv3d slides a 3D kernel through height, width, and depth (or time) — used for video and medical volumes like CT scans.

# 1D: sequence data (batch, channels, length)
conv1d = nn.Conv1d(in_channels=16, out_channels=32, kernel_size=5, padding=2)
seq = torch.randn(4, 16, 100)   # batch=4, 16 features, length=100
print(conv1d(seq).shape)         # [4, 32, 100]

# 3D: volumetric data (batch, channels, depth, height, width)
conv3d = nn.Conv3d(in_channels=1, out_channels=16, kernel_size=3, padding=1)
volume = torch.randn(2, 1, 64, 64, 64)  # e.g., CT scan
print(conv3d(volume).shape)               # [2, 16, 64, 64, 64]

Depthwise Separable Convolution — The Mobile Efficiency Trick

Standard convolution is expensive because every output channel looks at all input channels simultaneously. For a 3×3 conv with 256 input and 256 output channels, that's 3×3×256×256 = 589,824 parameters. Depthwise separable convolution splits this into two cheaper steps:

Step 1 — Depthwise convolution: Apply a separate 3×3 filter to each input channel independently. 256 channels → 256 filters → 256 output maps. Parameters: 3×3×256 = 2,304.

Step 2 — Pointwise convolution: A 1×1 convolution that mixes information across channels. 256 input → 256 output. Parameters: 1×1×256×256 = 65,536.

Total: 2,304 + 65,536 = 67,840 parameters instead of 589,824. That's roughly 8-9× fewer parameters and a proportional reduction in computation. The general ratio is:

# Standard conv: K² × M × N  parameters
# Depthwise separable: K² × M + M × N  parameters
# Ratio: 1/N + 1/K² ≈ 1/K² for large N
# For K=3: roughly 9× reduction

# PyTorch implementation
class DepthwiseSeparableConv(nn.Module):
    def __init__(self, in_ch, out_ch, kernel_size=3, stride=1, padding=1):
        super().__init__()
        # Depthwise: groups=in_ch means each channel gets its own filter
        self.depthwise = nn.Conv2d(
            in_ch, in_ch, kernel_size,
            stride=stride, padding=padding, groups=in_ch
        )
        # Pointwise: 1×1 conv to mix channels
        self.pointwise = nn.Conv2d(in_ch, out_ch, kernel_size=1)

    def forward(self, x):
        return self.pointwise(self.depthwise(x))

# Compare parameter counts
dw_sep = DepthwiseSeparableConv(256, 256)
standard = nn.Conv2d(256, 256, 3, padding=1)

dw_params = sum(p.numel() for p in dw_sep.parameters())
std_params = sum(p.numel() for p in standard.parameters())
print(f"Depthwise separable: {dw_params:,}")   # 68,096
print(f"Standard conv:       {std_params:,}")   # 590,080
print(f"Ratio: {std_params / dw_params:.1f}×")  # ~8.7×

MobileNet V1 (2017) was built entirely on this idea and achieved AlexNet-level accuracy with 1/30th the parameters. MobileNet V2 added inverted residuals with expansion — first expand the channels with a 1×1, apply the depthwise conv in the expanded space, then project back down. This gave better feature extraction at minimal extra cost. EfficientNet later used depthwise separable convs as its fundamental building block, scaling width, depth, and resolution together.

When to use depthwise separable convolutions

Use them when deployment matters — mobile, edge, real-time. For server-side training where memory and compute are abundant, standard convolutions train faster and can achieve slightly higher accuracy because the depthwise step limits cross-channel interaction. The accuracy gap is often negligible for practical applications.

Transposed Convolution — Learnable Upsampling

Regular convolutions shrink spatial dimensions (or keep them the same). But what if you need to grow them — go from a 7×7 feature map back to 224×224? That's what transposed convolutions (sometimes misleadingly called "deconvolutions") do.

The idea: insert zeros between input elements to "spread them out," then apply a regular convolution. The result is a larger output. It's used in every architecture that needs upsampling: U-Net (segmentation), autoencoders, GANs (image generation), super-resolution networks.

# Transposed conv: upsamples spatial dimensions
upsample = nn.ConvTranspose2d(
    in_channels=256, out_channels=128,
    kernel_size=4, stride=2, padding=1
)

x = torch.randn(1, 256, 14, 14)
out = upsample(x)
print(out.shape)  # torch.Size([1, 128, 28, 28]) — doubled spatial size

# Output size formula for transposed conv:
# out = (in - 1) × stride - 2 × padding + kernel_size
# (14 - 1) × 2 - 2 × 1 + 4 = 28
Checkerboard Artifacts

Transposed convolutions are notorious for producing checkerboard patterns in the output — visible grid-like artifacts caused by uneven overlap when stride doesn't divide evenly into the kernel size. The fix that most modern architectures use: bilinear upsampling followed by a regular convolution. This separates the "make it bigger" step from the "learn features" step, producing much cleaner outputs.

# Better alternative: upsample + conv (avoids checkerboard)
class SmoothUpsample(nn.Module):
    def __init__(self, in_ch, out_ch):
        super().__init__()
        self.up = nn.Upsample(scale_factor=2, mode='bilinear', align_corners=False)
        self.conv = nn.Conv2d(in_ch, out_ch, kernel_size=3, padding=1)

    def forward(self, x):
        return self.conv(self.up(x))

smooth = SmoothUpsample(256, 128)
x = torch.randn(1, 256, 14, 14)
print(smooth(x).shape)  # torch.Size([1, 128, 28, 28]) — clean upsampling

Pooling Layers

Max Pooling

Max pooling slides a window across the feature map and keeps only the maximum value in each region. With a 2×2 window and stride 2, the spatial dimensions are halved.

Why the max? Because conv feature maps represent "how strongly is this pattern present here?" If a horizontal edge detector fires with values [0.1, 0.3, 2.8, 0.5] in a 2×2 region, the 2.8 tells you "there's an edge somewhere in this area." The exact sub-pixel location doesn't matter — you care that the edge exists. Max pooling provides a degree of translation invariance: if the edge shifts by one pixel, the max within the window is likely still 2.8.

# Max pooling: 2×2 window, stride 2
pool = nn.MaxPool2d(kernel_size=2, stride=2)
x = torch.randn(1, 64, 224, 224)
print(pool(x).shape)  # [1, 64, 112, 112] — halved spatial, channels untouched

# Zero learnable parameters — this is a fixed operation
print(sum(p.numel() for p in pool.parameters()))  # 0

Average Pooling

Average pooling takes the mean of all values in the window. Where max pooling asks "is this feature present anywhere in the region?", average pooling asks "how much of this feature is present on average?" It's smoother, less aggressive, and doesn't amplify single strong responses. In practice, max pooling dominates in mid-network layers — it empirically works better for feature detection. Average pooling shows up mainly as Global Average Pooling at the network's end.

Global Average Pooling — The FC Layer Killer

GAP collapses an entire H×W feature map into a single number by averaging all spatial positions. If your final conv layer outputs (batch, 512, 7, 7), GAP produces (batch, 512) — one scalar per channel. Clean.

Before GAP, networks ended with massive fully-connected layers. VGG's first FC layer alone: 4,096 × 25,088 = 102 million parameters. That's over 70% of the entire network's parameters in one layer. GAP replaces this with a simple average, then a single FC layer for classification: 512 × 1,000 = 512,000 parameters. A 200× reduction.

GoogLeNet (2014) popularized GAP, and it's been nearly universal since. It also acts as a structural regularizer — fewer parameters means less overfitting — and enables input-size flexibility because there are no FC layers expecting a fixed number of inputs.

# Global Average Pooling in practice
gap = nn.AdaptiveAvgPool2d(1)  # output 1×1 regardless of input size

x = torch.randn(8, 512, 7, 7)
out = gap(x)              # (8, 512, 1, 1)
out = out.flatten(1)      # (8, 512) — ready for classifier

# Typical end-of-network pattern
classifier = nn.Sequential(
    nn.AdaptiveAvgPool2d(1),
    nn.Flatten(),
    nn.Dropout(0.2),
    nn.Linear(512, 1000)   # 512K params vs VGG's 102M
)

Stride-2 Conv as an Alternative to Pooling

"Let the network learn its own downsampling." A stride-2 convolution reduces spatial dimensions just like 2×2 max pooling, but it has learnable parameters — it can learn how to downsample optimally rather than being forced to take the max or average. The paper "Striving for Simplicity" (Springenberg et al., 2014) showed this works as well or better, and ResNet adopted the approach. The trade-off: pooling is parameter-free and fast; strided convs add weights and compute.

Feature Hierarchies — Why CNNs Work

This is the deep reason CNNs are powerful, and it's worth sitting with for a moment. As you stack convolutional layers, the network builds a hierarchy of increasingly abstract features:

Early layers (1-2): Edges, color gradients, simple textures. Individual 3×3 filters respond to horizontal lines, vertical lines, color transitions. These are nearly universal — you'll find nearly identical filters in every trained CNN, regardless of dataset.

Middle layers (3-5): Combinations of edges become parts. A "corner" is where a horizontal edge meets a vertical edge. A "circle" is many short edge segments arranged in a curve. Texture patterns like stripes, dots, and grids emerge.

Deep layers (6+): Parts combine into objects. Wheels + body + windows = car. Eyes + nose + mouth = face. At the deepest layers, you get neurons that respond to high-level concepts: "dog," "text," "sky." Remarkably, this hierarchy emerges purely from backpropagation — nobody programs it.

This is not just interesting trivia — it's the foundation of transfer learning. Early layers learn universal features applicable to almost any visual task. That's why you can take a ResNet trained on ImageNet and fine-tune it for medical imaging: the edge detectors and texture recognizers transfer perfectly. Only the deep, task-specific layers need retraining.

Receptive Field — What Each Neuron Can "See"

Theoretical Receptive Field

The receptive field of a neuron is the region of the input image that can influence its value. A single 3×3 conv layer: receptive field is 3×3. Stack two 3×3 layers: the second layer's neuron combines 3×3 outputs from the first layer, each of which looked at a 3×3 input region. Effective receptive field: 5×5. Three layers: 7×7. The pattern: each 3×3 layer adds 2 to the receptive field.

# Receptive field grows with depth
# After L layers of K×K conv with stride 1:
# RF = L × (K - 1) + 1
#
# K=3, L=1: RF = 3
# K=3, L=2: RF = 5
# K=3, L=3: RF = 7
# K=3, L=5: RF = 11
# K=3, L=10: RF = 21

# This is why VGG stacked 3×3 layers instead of using larger kernels:
# Two 3×3 layers = 5×5 receptive field with 18 params (per channel pair)
# One 5×5 layer  = 5×5 receptive field with 25 params
# Same coverage, fewer parameters, plus an extra non-linearity between them

Stride and pooling multiply the receptive field growth. A stride-2 layer means subsequent layers "see" twice as much input per pixel. This is why architectures progressively increase stride/pooling — to ensure deep layers have a receptive field large enough to capture entire objects.

Effective Receptive Field

Here's the catch: the theoretical receptive field is much larger than the effective one. Research by Luo et al. (2016) showed that the actual influence of input pixels on a deep neuron follows a Gaussian distribution — center pixels contribute massively more than border pixels. The effective receptive field often covers only a fraction of the theoretical one, roughly proportional to its square root.

This has practical consequences. If you're building an object detection network and your objects span 200×200 pixels, your network's effective receptive field at the detection layer needs to be at least 200×200. The theoretical field might need to be 400×400+ to get there. Dilated convolutions, aggressive downsampling, and feature pyramid networks are all strategies to ensure the receptive field is adequate.

Receptive field reality check

If your network can't detect large objects, the first thing to check is the receptive field. If it doesn't cover the object, the network literally cannot see it — no amount of training data or longer training will help. This is a structural limitation, not a data problem.

Parameter Count Calculation

Being able to compute parameter counts in your head is a genuinely useful skill — it tells you where the memory goes, which layers dominate training time, and whether your model will fit on a given device.

# For one conv layer:
# params = (K × K × C_in + 1) × C_out
#                              ↑ bias term (one per output channel)

# Example: Conv2d(3, 64, 7)
# (7 × 7 × 3 + 1) × 64 = 148 × 64 = 9,472

# Example: Conv2d(64, 128, 3)
# (3 × 3 × 64 + 1) × 128 = 577 × 128 = 73,856

# Example: Conv2d(512, 512, 3)
# (3 × 3 × 512 + 1) × 512 = 4,609 × 512 = 2,359,808  ← 2.3M params in one layer!

Notice how parameters explode with channel count. The 512→512 layer has 250× more parameters than the 3→64 layer. This is why deeper (more channels) architectures like VGG-16 have 138M parameters while shallower ones can get by with far fewer. Depthwise separable convolutions attack exactly this scaling problem.

# Verify with PyTorch
import torch.nn as nn

layers = [
    ("Conv2d(3, 64, 7)",   nn.Conv2d(3, 64, 7)),
    ("Conv2d(64, 128, 3)", nn.Conv2d(64, 128, 3)),
    ("Conv2d(512, 512, 3)", nn.Conv2d(512, 512, 3)),
]

for name, layer in layers:
    params = sum(p.numel() for p in layer.parameters())
    print(f"{name:25s} → {params:>10,} parameters")

Putting It All Together — A Typical Conv Block

In practice, conv layers rarely appear alone. The standard pattern is Conv → BatchNorm → ReLU, repeated many times. Here's what a typical building block looks like and how the numbers flow through:

import torch
import torch.nn as nn

class ConvBlock(nn.Module):
    """Standard Conv-BN-ReLU block used in most modern architectures."""
    def __init__(self, in_ch, out_ch, kernel_size=3, stride=1, padding=1):
        super().__init__()
        self.conv = nn.Conv2d(in_ch, out_ch, kernel_size,
                              stride=stride, padding=padding, bias=False)
        self.bn = nn.BatchNorm2d(out_ch)
        self.relu = nn.ReLU(inplace=True)

    def forward(self, x):
        return self.relu(self.bn(self.conv(x)))

# Build a simple feature extractor
model = nn.Sequential(
    ConvBlock(3, 64),                           # 224×224×3  → 224×224×64
    ConvBlock(64, 64),                           # 224×224×64 → 224×224×64
    nn.MaxPool2d(2, 2),                          # 224×224×64 → 112×112×64

    ConvBlock(64, 128),                          # 112×112×64  → 112×112×128
    ConvBlock(128, 128),                         # 112×112×128 → 112×112×128
    nn.MaxPool2d(2, 2),                          # 112×112×128 → 56×56×128

    ConvBlock(128, 256, stride=2),               # 56×56×128 → 28×28×256 (stride downsample)
    ConvBlock(256, 256),                         # 28×28×256 → 28×28×256

    nn.AdaptiveAvgPool2d(1),                     # 28×28×256 → 1×1×256
    nn.Flatten(),                                # → 256
    nn.Linear(256, 10)                           # → 10 classes
)

x = torch.randn(2, 3, 224, 224)
print(model(x).shape)  # torch.Size([2, 10])

total = sum(p.numel() for p in model.parameters())
print(f"Total parameters: {total:,}")  # ~957K — compact and effective
Design intuition

Notice the pattern: spatial dimensions go down, channel count goes up. 224→112→56→28→1 spatially, 3→64→128→256 in channels. This is nearly universal in CNNs. You're trading spatial resolution for feature richness — early layers know where things are at high resolution; deep layers know what things are with many channels.
