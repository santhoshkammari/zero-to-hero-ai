Object detection finds what objects image contains and where they are. Core computer vision problem since early 2010s. Segmentation goes further: classify every pixel, not rough rectangles. Together, they power autonomous driving, medical imaging, robotics, and systems that must understand visual scenes, not only label them.

Object Detection

Compare two questions. "Is there cat in this photo?" Classification — one image, one label. "Where are all cats, dogs, and people in this photo, and how big is each one?" Detection. Hard part is not "what." Hard part is "how many."

Classifier gives exactly one output. Detector may give zero predictions or fifty. Variable-length output makes detection hard. Fixed-size output layer fails when object count is unknown. Every major detector answers one question: how handle variable, unknown number of outputs?

Concrete example: security camera for small warehouse. Camera sees loading dock — forklifts move, workers walk, pallets stack. Job: detect and locate every person, forklift, and pallet in each frame. Some frames have two objects. Some have twenty. System must handle both.

The Shared Vocabulary

Every detection model, from R-CNN to newest DETR variant, shares four core concepts. These are tools every builder carries.

Bounding Boxes

Bounding box is axis-aligned rectangle saying "object is somewhere inside." Rough approximation — angled person gets rectangle with lots of empty background — but useful and cheap to predict.

Two main formats matter. Mixing them up causes common detection bugs:

# Center format: where is the center, how big is it?
(cx, cy, w, h)    # YOLO uses this

# Corner format: where are the top-left and bottom-right corners?
(x1, y1, x2, y2)  # Pascal VOC uses this

# COCO uses a hybrid: top-left corner + size
(x, y, w, h)      # x,y is top-left, NOT center

Conversion is easy arithmetic. Mixing formats silently shifts or rescales boxes wrong. Real production bugs come from this. Always verify format expected by data and model.

IoU — Intersection over Union

IoU measures predicted-box match to ground-truth box. Compute overlap area. Divide by union area. Done.

IoU = Area of Overlap / Area of Union

Ranges from 0 (no overlap) to 1 (perfect match)

Warehouse example: model predicts forklift box, but right side is too wide. If predicted box and true box share 70% of combined area, IoU = 0.7. Standard "correct detection" threshold is IoU > 0.5. COCO is stricter: average thresholds from 0.5 to 0.95 in steps of 0.05. IoU is scale-invariant — tiny boxes and huge boxes get same IoU when relative overlap matches.

Subtle limit: if boxes do not overlap, IoU is zero. Close miss and totally wrong look same. That led to GIoU (Generalized IoU), DIoU (Distance IoU), and CIoU (Complete IoU), which also use distance and aspect ratio. These often serve as regression losses during training, helping box prediction converge faster.

Anchor Boxes

Warehouse detector faces ugly search space. Boxes can be any size and any aspect ratio. Standing worker is tall and narrow. Flat pallet is wide and short. Forklift is roughly square. Predicting raw coordinates from scratch is hard for neural networks because search space is enormous.

Anchor boxes shrink search space. They are predefined box templates — shapes and sizes — placed at every spatial location in feature map. Instead of predicting box from nothing, network predicts offsets: "box at this grid location is 1.2× wider and 0.8× taller than anchor template #4, and shifted 5 pixels right."

Small offsets from good starting point are much easier than absolute coordinates. Common setup uses 3 aspect ratios (1:1, 1:2, 2:1) at 3 scales, giving 9 anchors per location. Many teams run k-means clustering on ground-truth boxes to choose anchor shapes that fit dataset.

Anchors dominated detection from 2015 to about 2022. Newer anchor-free methods can match or beat anchor-based ones with simpler designs. More on that shift later.

Non-Maximum Suppression (NMS)

Warehouse detector evaluates thousands of candidate locations per image. Many fire on same forklift, creating cluster of overlapping boxes. Need one final detection. NMS — Non-Maximum Suppression — does this.

Idea is simple. Sort predicted boxes by confidence. Keep most confident box. Remove any other box overlapping it above IoU threshold (usually 0.5). Take next surviving box. Repeat until no boxes remain.

NMS usually works. Weakness: real overlap. Two workers standing close may have high-IoU correct boxes, so NMS may suppress valid detection. Soft-NMS helps by decaying scores instead of deleting boxes. DETR removes NMS entirely with different approach.

The Detection Recipe

Generate candidate locations → predict box offsets (from anchors or directly) → classify boxes → apply NMS to remove duplicates. Every classical detector follows some form of this pipeline. Big differences lie in how candidates are generated and when classification happens.

Two-Stage Detectors — The R-CNN Family

Modern detection story starts with family of models using two explicit steps: first propose regions that might contain objects, then classify each proposal and refine box. This two-step approach dominated from 2014 to 2016. Descendants remain competitive today.

Trace evolution because each step fixes specific painful bottleneck. Understanding bottlenecks makes architecture choices feel inevitable, not arbitrary.

R-CNN (2014) — The Proof of Concept

R-CNN (Regions with CNN features) proved deep learning could dominate detection. Idea was brute force but effective: use Selective Search to generate about 2,000 region proposals per image — rectangular crops that "might contain something." Warp each crop to fixed size. Feed each through CNN independently. Classify with SVMs. Refine boxes with regression.

Accurate? Yes — it crushed Pascal VOC. Practical? No. Each image needed 2,000 separate CNN forward passes. Inference took 47 seconds per image. Training took 84 GPU hours. Warehouse system needs 30 FPS; this gives about 23 minutes per frame. Bad.

But insight was powerful: CNNs extract features so rich that even simple classifier on top beats hand-engineered detectors. Next question: how keep that power while removing absurd redundancy?

Fast R-CNN (2015) — Share the Computation

Waste in R-CNN was obvious. Those 2,000 proposals overlap heavily. Many share same image pixels. No reason to recompute CNN features from scratch for each one.

Fast R-CNN fixed this with one key change: run CNN once on full image to get one shared feature map. Then extract each proposal's features from that map with RoI Pooling (Region of Interest Pooling). RoI Pooling turns variable-sized region into fixed-size output by splitting region into grid and max-pooling within each cell.

Massive speedup. Expensive CNN computation happens once, not 2,000 times. Classification and box regression now happen in one end-to-end network. But one bottleneck remained: Selective Search still ran on CPU and took about 2 seconds per image. Neural network was fast. Proposal generation was not.

Faster R-CNN (2015) — Proposals Go Neural

Faster R-CNN removed last hand-crafted piece. Instead of Selective Search, it introduced Region Proposal Network (RPN) — small convolutional network sliding over shared feature map. At each position and for each anchor, it predicts two things: objectness and box offsets.

# Faster R-CNN pipeline
Image → Backbone CNN → Shared Feature Map
                              ↓
                   Region Proposal Network (RPN)
                     "Is there something here? Adjust this anchor."
                              ↓
                   ~300 highest-scoring proposals
                              ↓
                   RoI Align → Fixed-size features per proposal
                              ↓
                   Classification head: "What is it?"
                   Box regression head: "Where exactly?"

RPN is light — only few convolutional layers — and shares backbone features, so proposals cost almost nothing. What took 2 seconds with Selective Search takes about 10 milliseconds with RPN. Full pipeline — features, proposals, classification, box regression — becomes one end-to-end network with multi-task loss.

In 18 months, jump was huge. R-CNN: 47 seconds per image, four separate stages, SVMs, external proposals. Faster R-CNN: 200 milliseconds per image, fully differentiable, one network. Same core idea — propose then classify. Engineering changed from "research prototype" to production foundation.

RoI Align — A Small Fix That Mattered Enormously

Need mention RoI Align. This detail separates real understanding from memorized names. Original RoI Pooling snapped proposal coordinates to nearest integer grid cell. That quantization causes small spatial misalignments. Classification tolerates it. Pixel-level tasks do not.

RoI Align (Mask R-CNN, 2017) replaces rounding with bilinear interpolation. Instead of snapping to cells, it samples exact floating-point coordinates. Small code change. Big mask-quality gain. Modern two-stage detectors use RoI Align by default.

Why Two Stages Still Make Sense

RPN acts like coarse attention — it removes ~99% background, so second stage classifies only few hundred promising regions. Divide-and-conquer gives two-stage detectors top accuracy on benchmarks. Cost is speed.

One-Stage Detectors — The Speed Path

Two-stage detectors are accurate. Obvious question: need two steps at all? What if class labels and bounding boxes come directly from feature map in one pass?

That is one-stage detection. It trades careful region-by-region attention for speed. With right training tricks, accuracy gap can shrink too.

YOLO — You Only Look Once

YOLO first looked almost too simple. YOLO (2016) divides input image into S×S grid. Each grid cell handles objects whose centers fall inside it. Each cell predicts B bounding boxes with confidence and C class probabilities. One forward pass does everything.

# YOLO's output tensor
Input image → CNN backbone → S × S × (B × 5 + C) tensor

# Each grid cell predicts:
#   B boxes: (cx, cy, w, h, confidence) for each
#   C class probabilities shared across all boxes in that cell

# Example: S=7, B=2, C=20 (Pascal VOC)
# Output: 7 × 7 × (2 × 5 + 20) = 7 × 7 × 30

Back to warehouse. With 7×7 grid, image splits into 49 cells. Forklift whose center lands in cell (3, 4) gets detected by that cell. Cell predicts two boxes and confidence scores, and class distribution says "98% forklift." Full detection happens in one shot — no proposals, no second stage.

YOLOv1 ran at 45 FPS — real time on one GPU. But limits were clear. Each cell could detect only B objects, so nearby objects in same cell caused misses. Small objects suffered because they occupied tiny parts of cells. Still, core insight — treat detection as one regression problem — opened new design space.

The YOLO Evolution

YOLO family changed dramatically across versions. Evolution shows what matters in practical detection.

YOLOv2 (2017) added anchor boxes, so network predicted offsets from templates instead of raw coordinates. It added batch normalization everywhere and trained at multiple input resolutions. Accuracy jumped.

YOLOv3 (2018) added multi-scale predictions — three feature-map resolutions via Feature Pyramid Network. This fixed small-object weakness. Backbone moved to much deeper Darknet-53.

YOLOv4 (2020) was engineering masterclass. It added "bag of freebies" — mosaic augmentation, CIoU loss, self-adversarial training — which improve accuracy without slowing inference, plus "bag of specials" — CSPDarknet and PANet — which add small inference cost for larger accuracy gains.

YOLOv5 (2020) had no academic paper and naming was controversial, but Ultralytics' PyTorch implementation became most-deployed YOLO variant because engineering was excellent — easy training, easy export, strong docs.

YOLOv8 (2023) made major shift: anchor-free detection. Instead of template offsets, it predicts object centers and dimensions directly. Decoupled head separates classification and box regression, letting each task use features tuned for its own goal. YOLOv8 supports detection, instance segmentation, and pose estimation in one architecture.

YOLO11 (2024) is latest version, with better backbone efficiency and better accuracy across tasks.

Trend across versions is clear: stronger backbones, better feature fusion, fewer hand-crafted parts, more general tasks. YOLO started as speed hack. Now it is production-grade family that rivals two-stage detectors on accuracy while keeping real-time performance.

SSD — Multi-Scale Was the Key

SSD (Single Shot MultiBox Detector, 2016) attacked YOLO's small-object weakness by predicting from multiple feature-map layers at once. Early high-resolution layers handle small objects. Later large-receptive-field layers handle big ones. SSD attaches prediction heads to each layer, each with anchors scaled to that resolution.

This multi-scale idea proved fundamental. Feature Pyramid Networks (FPN), YOLOv3's three-scale predictions, and nearly every modern detector adopted some version.

RetinaNet and Focal Loss — The Training Problem

For years, one-stage detectors were faster but less accurate than two-stage ones. Many thought reason was architecture — two stages give richer features. RetinaNet (2017) showed real problem was training.

Problem: one-stage detector evaluates about 100,000 candidate locations per image. Maybe 10 contain objects. Rest are background. Standard cross-entropy weights all examples equally, so 99.99% of training compute goes into trivially easy background. Hard examples — true objects, ambiguous boundaries — get drowned out.

Two-stage detectors avoid this because RPN removes most background before classification. One-stage detectors see everything at once. Imbalance is brutal.

Focal loss fixed this with simple modification:

# Standard cross-entropy
CE(p) = -log(p)

# Focal loss
FL(p) = -(1 - p)^γ × log(p)

# When p is high (easy example, model already confident):
#   (1 - 0.95)^2 = 0.0025 → loss multiplied by 0.0025 → nearly zero
#
# When p is low (hard example, model struggling):
#   (1 - 0.3)^2 = 0.49 → loss barely reduced → full learning signal
#
# γ = 2 is typical. Easy examples get ~100× less weight.

(1 - p)^γ acts as dynamic difficulty weight. Easier example means higher model confidence, so loss gets crushed harder. Training compute shifts naturally toward hard, informative examples.

With focal loss, RetinaNet became first one-stage detector to match Faster R-CNN accuracy at one-stage speed. It proved gap was training problem — specifically loss-function problem — not fundamental architectural limit. That insight changed field direction.

Focal Loss Beyond Detection

Focal loss matters anywhere dense predictions face extreme class imbalance — segmentation, dense retrieval, multi-label classification with many rare labels. Principle is general: stop spending training capacity on examples model already gets right.

DETR — When Transformers Replaced the Pipeline

Everything so far shares pattern: anchors, proposals, NMS, hand-tuned thresholds. These parts work, but they are engineering artifacts. DETR (DEtection TRansformer, 2020) asked: what if all of it disappears?

DETR treats detection as direct set prediction. Image goes through CNN backbone. Feature map becomes sequence. Transformer encoder processes sequence. Transformer decoder uses N learned object queries — embeddings that learn to ask about object locations. Decoder outputs N predictions. No anchors. No NMS. No proposal stage.

# DETR pipeline
Image → CNN backbone → Feature map → Flatten to sequence
    → Transformer Encoder (global reasoning across image)
    → Transformer Decoder (N learned object queries attend to image)
    → N predictions: (class, bounding box) pairs

# N is fixed (e.g., 100). Most slots predict "no object."
# Each query specializes during training to handle different
# spatial regions or object types.

Key idea is bipartite matching. During training, DETR uses Hungarian algorithm to find optimal one-to-one assignment between N predictions and ground-truth objects. Each object matches exactly one prediction slot. Unmatched slots learn "no object." Duplicate detections disappear by design, so NMS is unnecessary.

DETR paper felt elegant. No hand-crafted components. End-to-end differentiable. Conceptually clean. Price was high: DETR needed 500 training epochs (vs. ~36 for Faster R-CNN) and struggled with small objects. Global attention treats all spatial locations equally, so DETR lacked multi-scale bias that FPN detectors get free.

Follow-up work fixed practical issues. Deformable DETR replaced global attention with deformable attention — sparse learned sampling points instead of all locations — cutting convergence to ~50 epochs and greatly improving small objects. DINO added denoising training and better query initialization, reaching or beating best traditional detectors. RT-DETR optimized real-time inference, making transformer detection deployable.

DETR family shows important lesson: transformers can work for detection, but they needed targeted engineering — deformable attention, multi-scale features, better training recipes — before they became practical. Initial elegance was necessary. Not sufficient.

Anchor-Free Detectors — Simplifying the Pipeline

Alongside DETR, another movement simplified detection from different direction: remove anchors.

Anchor design is tedious. Need aspect ratios, scales, and counts. Wrong choices miss objects. Teams often run k-means on ground-truth boxes to pick anchors. Works, but remains hand-tuned step in learned pipeline.

CornerNet (2018) predicted top-left and bottom-right corners as heatmaps, then grouped them into boxes. CenterNet (2019) simplified further: predict object centers as heatmaps and regress width/height from each center. FCOS (2019) went fully convolutional — each feature-map pixel predicts distances to four box edges of nearest object, plus centerness score that down-weights poor predictions far from object center.

These methods are simpler, use fewer hyperparameters, and avoid anchor-object matching complexity. Modern YOLOv8+ adopted anchor-free designs. Field has mostly moved past explicit anchors — useful intermediate step, but direct regression from points or centers proved enough.

Detection Evaluation — Metrics That Matter

Detection evaluation is harder than classification because metric must answer two questions at once: did model find right objects, and did it place boxes correctly?

For each class, sort detections by confidence. Sweep down list and compute precision and recall at each step. Area under precision-recall curve is Average Precision (AP) for that class — measure of how well model ranks true detections above false ones. Average AP across classes gives mAP (mean Average Precision), standard detector comparison number.

But mAP depends on IoU threshold: how tight must box be to count as correct?

mAP@0.5 (Pascal VOC) counts boxes with IoU above 0.5 as correct. Lenient. mAP@0.75 demands tighter boxes. mAP@[0.5:0.95] (COCO) averages ten thresholds — gold standard because it rewards precise localization, not rough hits.

COCO also reports size breakdown: APS (small, area < 32²), APM (medium), APL (large). This matters because detectors behave very differently by size. Model with strong overall mAP may still fail on small objects. In warehouse, pallets are large and easy. Loose bolts on floor are tiny and critical. Always inspect size breakdown.

Faster R-CNN Two-stage ~5–15 FPS High Offline processing, accuracy-critical tasks
YOLOv8 / YOLO11 One-stage, anchor-free ~80–160 FPS High Real-time video, edge deployment

These speeds are rough guides. Real performance depends heavily on backbone, input resolution, hardware, and batch size. Core tradeoff remains: more computation buys accuracy; simplicity buys speed. Gap narrows every year.

Segmentation

Detection draws rectangles around things. Useful, but rectangle is blunt. It includes background, ignores shape, and cannot say which pixel belongs to which object. In warehouse, one box around stacked pallets includes empty gaps. Robotic arm needs exact boundary, not "somewhere inside rectangle."

Segmentation is pixel-level classification. Instead of one label per image or one label per box, it assigns label to every pixel. Output is map same size as input. Each pixel carries prediction. Three flavors exist, each answering different question.

Three Flavors of Segmentation

Semantic Segmentation — "What class is each pixel?"

Every pixel gets class label. All forklift pixels are "forklift." All floor pixels are "floor." All worker pixels are "worker." Limitation: semantic segmentation cannot separate individual objects of same class. Two nearby workers become one blob of "worker" pixels. You know class in region. You do not know count or exact instance boundaries.

Fine when instance identity does not matter — autonomous driving, satellite land-use mapping, scene parsing. Not fine when you must count items or track individuals.

Instance Segmentation — "Which object does each pixel belong to?"

Each object gets its own pixel-perfect mask. Worker 1, Worker 2, Worker 3 — separate boundaries. Harder task: model must classify pixels and group them into distinct instances. It combines detection with segmentation.

In warehouse, instance segmentation lets system count exact pallet stacks and trace precise outline of each one. Critical for robotic manipulation and inventory management.

Panoptic Segmentation — "Everything, all at once"

Panoptic segmentation unifies both flavors. Scene splits into stuff (amorphous regions like floor, walls, sky) and things (countable objects like forklifts, workers, pallets). Stuff gets semantic labels. Things get individual instance masks. Every pixel is covered — no gaps, no overlaps.

TaskWhat It Tells YouWarehouse ExampleLimitation

Recent Mask2Former deserves mention. It treats all three segmentation tasks as one problem — predict set of binary masks with class labels — using Transformer decoder with learned mask embeddings. One architecture. Three tasks. State-of-the-art results on COCO, ADE20K, and Cityscapes. Strong sign field is converging toward unified models.

FCN — The Starting Point

Before 2015, pixel-level segmentation meant sliding patch classifier across image, one pixel at a time. Thousands of overlapping forward passes per image. Very slow.

Fully Convolutional Network (Long et al., 2015) made one key observation: replace fully connected layers in classification CNN with 1×1 convolutions, and whole network becomes fully convolutional. Then it accepts any input size and outputs spatial map in one forward pass. Instead of "this image is a cat," it outputs spatial probability map of cat-ness.

Catch: repeated pooling makes output tiny — maybe 7×7 for 224×224 input. FCN restores resolution with transposed convolutions (learned upsampling). Paper showed that combining upsampled predictions with earlier-layer features (FCN-32s → FCN-16s → FCN-8s) — skip connections — sharpened boundaries.

FCN proved end-to-end pixel prediction was possible. But outputs stayed coarse, and architecture lacked principled way to mix low-level spatial detail with high-level semantics. Then U-Net arrived.

U-Net — The Architecture That Conquered Medical Imaging

If one segmentation architecture deserves deep understanding, it is U-Net (Ronneberger et al., 2015). Built first for biomedical image segmentation, where labeled data is scarce. Now default segmentation architecture across domains — cells, satellites, industrial inspection, even backbone inside diffusion models like Stable Diffusion.

The Shape That Gives It a Name

U-Net has symmetric structure that looks like letter U. Left side is encoder (contracting path): convolution stacks plus max-pooling, steadily shrinking spatial resolution while increasing channels. Each pooling step halves resolution and doubles channels. This path captures what image contains, but loses where details.

Right side is decoder (expanding path): transposed convolutions that raise spatial resolution while reducing channels. This path tries to recover spatial detail destroyed by pooling.

Skip Connections — Why U-Net Actually Works

Encoder-decoder alone is not special — FCN had that. U-Net magic is skip connections. At each level, encoder feature maps are concatenated — not added, concatenated — with decoder upsampled maps before more convolution.

What does this do? Bottleneck features (1024 channels, tiny spatial map) hold strong semantics — "this region is tumor" — but almost no spatial precision. Encoder features at same scale hold exact spatial structure — "edge exists at these coordinates" — but weaker semantics. Concatenation gives decoder both: high-level "what" from deep features and fine-grained "where" from encoder skip. Later convolutions learn fusion.

Without skip connections, decoder reconstructs detail from heavily compressed bottleneck — like painting detailed picture from blurry sketch. With skip connections, fine details shortcut directly from encoder. Spatial information does not need to survive bottleneck. That is why U-Net gives sharp boundaries even with limited data.

Why Medical Imaging Fell in Love

Medical imaging has two properties that fit U-Net almost perfectly:

Data scarcity. Expert pixel-level annotation from radiologists is expensive and slow. U-Net's skip connections provide strong inductive bias — architecture already expects encoder spatial detail to guide decoder — so model learns effective segmentation from hundreds of images, not millions.

Boundary precision matters clinically. Few pixels of error around tumor can change "operable" to "invaded surrounding tissue." U-Net's direct spatial transfer preserves sub-pixel precision needed for clinical decisions.

Modern variants extend idea. U-Net++ uses nested dense skip connections to capture features at multiple semantic levels. Attention U-Net adds learnable gates to skips, so model can decide which spatial details pass through. nnU-Net (no-new-Net) auto-configures U-Net hyperparameters for any medical segmentation task — preprocessing, depth, patch size, augmentation — and often wins medical segmentation challenges without manual tuning. Core idea stays same: symmetric encoder-decoder with skip concatenation.

DeepLab — Seeing More Without Losing Detail

U-Net strategy: lose spatial detail in encoder, then recover it in decoder with skip connections. DeepLab asks different question: what if spatial detail never disappears?

Atrous (Dilated) Convolutions

Standard convolutions with pooling or stride reduce spatial resolution to grow receptive field. Large receptive field gives context — is pixel part of car or building? — but pooling destroys spatial precision. Fundamental tension.

Atrous convolutions (dilated convolutions) solve this by inserting gaps between kernel elements. Name comes from French "à trous" — "with holes." Kernel samples spaced-out positions instead of contiguous ones:

Standard 3×3 conv (receptive field = 3×3):
  X X X        9 parameters, sees 3×3 area
  X X X
  X X X

Dilated 3×3 conv, rate=2 (receptive field = 5×5):
  X . X . X    still 9 parameters, but sees 5×5 area
  . . . . .
  X . X . X
  . . . . .
  X . X . X

Dilated 3×3 conv, rate=4 (receptive field = 9×9):
  X . . . X . . . X    9 parameters, sees 9×9 area

ASPP — Multi-Scale Context in Parallel

In warehouse, nearby forklift may cover thousands of pixels. Same forklift across room may cover hundred. Model must recognize both. ASPP (Atrous Spatial Pyramid Pooling) solves this by applying atrous convolutions with multiple dilation rates in parallel and concatenating results:

ASPP Module (DeepLabv3):

                  ┌── Atrous Conv, rate=6  ──┐
                  ├── Atrous Conv, rate=12 ──┤
Feature map ──────┼── Atrous Conv, rate=18 ──┼──▶ Concat ──▶ 1×1 Conv ──▶ Output
                  ├── 1×1 Conv ──────────────┤
                  └── Global Avg Pool ───────┘

Each branch captures context at a different spatial scale.
The 1×1 conv captures point-level features.
Global avg pool captures image-level context.
Concatenation fuses everything.

DeepLabv3+ combines ASPP with lightweight decoder and modified backbone (Xception or ResNet with atrous convolutions in later stages). Result: multi-scale context awareness plus sharp boundary recovery. It remains top choice for semantic segmentation in driving scenes and outdoor environments.

Mask R-CNN — Detection Meets Pixel Masks

Here detection and segmentation meet. Mask R-CNN (He et al., 2017) takes Faster R-CNN — two-stage detector from earlier — and adds one elegant branch: for each detected region, predict binary segmentation mask along with class label and bounding box.

Mask R-CNN:

Image → Backbone (ResNet + FPN) → Region Proposal Network
                                         ↓
                                   RoI Proposals
                                         ↓
                                    RoI Align
                                  ╱      │      ╲
                            Class Head  Box Head  Mask Head
                            (softmax)  (regress)  (28×28 binary
                                                   mask per class)

Two design choices make it work. First, RoI Align replaces RoI Pooling. Quantization errors that barely hurt classification destroy pixel-level masks. Bilinear interpolation preserves sub-pixel precision. Second, mask head predicts K separate binary masks — one per class — and uses mask for predicted class. This decouples classification from segmentation. Mask branch focuses only on pixel location. Classification head handles "what." Mask head handles "where, exactly." Each task gets features tuned for its own job.

Mask R-CNN ruled instance segmentation benchmarks for years. It is two-stage, so not fastest, but it remains go-to starting point for instance segmentation projects and foundation for many production systems.

SAM — Segment Anything, Zero-Shot

Every architecture so far is trained for fixed class set. Cityscapes model segments roads and cars. Medical model segments tumors. Train on one domain. Deploy on same domain. SAM — Segment Anything Model (Meta AI, 2023) — broke that pattern.

SAM is segmentation foundation model. Trained on more than 1 billion masks across 11 million diverse images, it can segment objects never explicitly trained as classes. No fine-tuning. No domain-specific labels. Give image plus prompt — click, bounding box, point set — and it returns mask.

SAM Architecture:

┌─────────────────────────────────────────────────────┐
│ Image Encoder (ViT-H)                               │
│ Input image → Patch embeddings → Transformer layers  │
│ → Dense image embeddings                             │  ← Heavy. Runs once.
└────────────────────────┬────────────────────────────┘
                         │
┌────────────────────────▼────────────────────────────┐
│ Prompt Encoder                                       │
│ Points/boxes → positional encodings                  │
│ Masks → lightweight CNN                              │  ← Lightweight.
│                        ↓                             │     Runs per prompt.
│ Mask Decoder (lightweight transformer)               │
│ Cross-attention between prompt and image embeddings  │
│ → Predicted masks + confidence scores                │
└─────────────────────────────────────────────────────┘

Architecture is intentionally split. Image encoder — huge Vision Transformer (ViT-Huge) — is expensive but runs once per image and produces reusable embeddings. Prompt encoder and mask decoder are light, so you can try many prompts interactively in real time. Click worker helmet, get helmet mask. Click torso, get full-body mask. Draw box around pallet stack, get pixel-accurate stack boundary.

This design makes SAM transformative for data annotation. Hand-drawing masks takes minutes per object. SAM can produce usable mask from one click in milliseconds. Teams can annotate segmentation datasets 10× to 50× faster. SAM masks also feed downstream pipelines — classifiers, trackers, 3D reconstruction — where SAM handles "where" and other models handle "what."

SAM 2 (2024) extends idea to video, keeping segmentation masks consistent across frames with memory mechanism. Prompt object in one frame, and SAM 2 tracks mask through full video. Move from "segment anything in image" to "segment anything in video" is natural and opens video annotation, editing, and temporal analysis at scale.

Zero-shot ability still has limits. Highly specialized domains — some medical imaging modalities, unusual industrial inspection cases — may need fine-tuning. But broad pattern is clear: segmentation foundation models work remarkably well out of box, and they are changing how field thinks about task-specific training.

Loss Functions for Segmentation

Segmentation losses face problem classification losses do not: extreme pixel-level class imbalance. In driving scene, maybe 60% of pixels are "road," 20% "sky," and last 20% split across many classes. Model that predicts "road" everywhere gets 60% pixel accuracy and is still useless.

Cross-entropy, applied per pixel, is baseline. It works, but majority class dominates. You can add class weights — larger loss for rare classes — but then you must tune another hyperparameter.

# Weighted cross-entropy
weights = torch.tensor([0.5, 2.0, 3.0, 5.0, ...])  # rare classes get higher weight
loss = nn.CrossEntropyLoss(weight=weights)

Dice loss directly optimizes overlap between predicted and ground-truth masks. It measures fraction of predicted and actual regions that coincide, and naturally handles class imbalance because it depends on overlap ratio, not raw pixel counts:

Dice coefficient = 2 × |Prediction ∩ Ground Truth|
                   ─────────────────────────────────
                     |Prediction| + |Ground Truth|

Dice loss = 1 − Dice coefficient

Perfect overlap → Dice = 1.0 → Loss = 0.0
No overlap     → Dice = 0.0 → Loss = 1.0

Dice loss matters especially in medical imaging, where lesion or tumor may cover only 1–2% of image. Cross-entropy can let model ignore those pixels. Dice loss forces attention onto small region because denominator normalizes by region size.

In practice, most production segmentation pipelines combine both: Loss = CE + λ × Dice. Cross-entropy gives stable per-pixel gradients for optimization. Dice loss prevents small regions from being ignored. Together they beat either one alone reliably.

Segmentation Evaluation

MetricFormulaWhat It MeasuresWhen to Use
IoU (Jaccard) |P ∩ G| / |P ∪ G| Overlap per class Per-class analysis
mIoU Mean of per-class IoU Average quality across classes Standard for semantic segmentation (Cityscapes, ADE20K)
Dice Score 2|P ∩ G| / (|P| + |G|) Overlap (equivalent to pixel-level F1) Standard in medical imaging
Pixel Accuracy Correct / Total pixels Raw accuracy Avoid as primary metric — misleading with imbalanced classes

IoU and Dice are related: Dice = 2 × IoU / (1 + IoU). Dice is always ≥ IoU for same prediction. For benchmarking, mIoU is standard on PASCAL VOC, Cityscapes, and ADE20K. In medical imaging, Dice score is standard. Know both. Know when each is expected.
