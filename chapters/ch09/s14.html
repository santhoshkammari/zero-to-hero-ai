Visual Search & Retrieval

Chapter 9: CNNs & Computer Vision 4 subtopics
TL;DR

Visual search turns "find images like this one" into a math problem. Extract a feature vector (embedding) from each image using a CNN or Vision Transformer. Train the embedding space with metric learning losses — contrastive loss pulls matching pairs together, triplet loss pushes non-matches apart. Store millions of embeddings in a vector database like FAISS, which uses approximate nearest neighbor algorithms to find the closest matches in milliseconds. Re-ranking refines the initial results using more expensive comparisons. Together, these components power reverse image search, product discovery, and content deduplication at scale.

You photograph a chair you like at a friend's house. You want to buy one just like it. You open a shopping app, upload the photo, and instantly see visually similar chairs from thousands of online stores, ranked by similarity. That's visual search in action. Behind the scenes, it's a retrieval pipeline with four stages: extract features from the query image, find similar images from a database of millions, rank the results, and return the top matches. Each stage involves distinct engineering decisions, and getting any one wrong collapses the entire user experience.

Visual search is also one of the clearest examples of embeddings at work. If you understood word embeddings in NLP (king − man + woman ≈ queen), the same principle applies here: images get mapped to vectors in a space where geometric distance corresponds to visual similarity. The entire system is built around this representation.

Feature Extraction for Retrieval

The first step is turning an image into a fixed-size vector — a feature embedding. This vector must capture the visual essence of the image: two photos of the same red sneaker from different angles should produce similar vectors; a red sneaker and a blue boot should produce distant vectors.

The simplest approach: take a CNN pretrained on ImageNet (ResNet, EfficientNet, etc.), remove the final classification layer, and use the output of the last pooling layer as the embedding. For a ResNet-50, that's a 2048-dimensional vector. This works surprisingly well for general-purpose similarity, because ImageNet pretraining forces the network to learn a rich hierarchy of visual features — textures, shapes, parts, objects — that transfer to many domains.

import torch
from torchvision import models, transforms
from PIL import Image

# Load pretrained ResNet and remove classification head
model = models.resnet50(pretrained=True)
model = torch.nn.Sequential(*list(model.children())[:-1])  # remove FC layer
model.eval()

preprocess = transforms.Compose([
    transforms.Resize(256),
    transforms.CenterCrop(224),
    transforms.ToTensor(),
    transforms.Normalize(mean=[0.485, 0.456, 0.406],
                         std=[0.229, 0.224, 0.225])
])

def extract_embedding(image_path):
    img = Image.open(image_path).convert('RGB')
    tensor = preprocess(img).unsqueeze(0)
    with torch.no_grad():
        embedding = model(tensor).squeeze()  # [2048]
    # L2 normalize for cosine similarity
    embedding = embedding / embedding.norm()
    return embedding.numpy()

# Two images → two embeddings → cosine similarity
emb_a = extract_embedding('red_sneaker_1.jpg')
emb_b = extract_embedding('red_sneaker_2.jpg')
similarity = emb_a @ emb_b  # dot product of L2-normalized = cosine sim
print(f"Similarity: {similarity:.3f}")  # e.g., 0.87

Off-the-shelf embeddings from ImageNet models have limitations. They encode generic visual features, not domain-specific ones. For fashion search, you need the model to distinguish between "crew neck" and "V-neck" — subtleties that ImageNet classification doesn't require. For product search, you need the model to match the same product across different backgrounds, lighting, and photography styles. This is where metric learning comes in: fine-tuning the embedding space so that distances correspond to your specific notion of similarity.

Modern practice increasingly uses Vision Transformers (ViT) or CLIP-style models as feature extractors. CLIP embeddings are particularly powerful for retrieval because they're trained on 400 million image-text pairs, giving them a broad understanding of visual concepts. A CLIP embedding encodes not just visual patterns but semantic understanding — it "knows" that a photo of a sunset and a painting of a sunset are conceptually similar, even though their pixel-level features differ dramatically.

Embedding Dimensionality

Higher dimensions preserve more information but cost more to store and search. A database of 100 million images at 2048-d float32 requires ~800 GB just for embeddings. Reducing to 256-d cuts that to ~100 GB with only a modest accuracy drop. PCA, learned dimensionality reduction, or training with a smaller embedding head are all valid strategies. In production, 128–512 dimensions is the sweet spot for most visual search systems.

Metric Learning: Contrastive and Triplet Losses

Off-the-shelf embeddings get you started. Metric learning gets you to production quality. The goal: train the embedding function so that the distance between embeddings directly reflects your task's notion of similarity. Images of the same product should have near-zero distance; images of different products should be far apart.

Contrastive Loss

Contrastive loss operates on pairs of images. For a matching pair (same class), it minimizes the distance between their embeddings. For a non-matching pair (different classes), it pushes them apart — but only if they're closer than a margin m. If non-matching pairs are already far apart, no gradient is applied (we don't waste effort pushing already-separated examples even further).

import torch
import torch.nn.functional as F

def contrastive_loss(emb_a, emb_b, label, margin=1.0):
    """
    label=1: matching pair (same class)  → pull together
    label=0: non-matching pair           → push apart (up to margin)
    """
    distance = F.pairwise_distance(emb_a, emb_b)

    # Matching: minimize distance²
    loss_match = label * distance.pow(2)

    # Non-matching: minimize max(0, margin - distance)²
    loss_diff = (1 - label) * F.relu(margin - distance).pow(2)

    return (loss_match + loss_diff).mean()

Triplet Loss

Triplet loss is the workhorse of visual retrieval systems. It operates on triplets: an anchor image, a positive (same class as anchor), and a negative (different class). The loss says: the anchor-positive distance should be smaller than the anchor-negative distance by at least a margin m. Formally: L = max(0, d(a,p) - d(a,n) + m).

def triplet_loss(anchor, positive, negative, margin=0.3):
    """
    Push d(anchor, positive) to be at least `margin` less
    than d(anchor, negative).
    """
    d_pos = F.pairwise_distance(anchor, positive)
    d_neg = F.pairwise_distance(anchor, negative)
    loss = F.relu(d_pos - d_neg + margin)
    return loss.mean()

The critical engineering challenge with triplet loss is mining: selecting which triplets to train on. Most triplets are "easy" — the negative is already far from the anchor, so the loss is zero and no learning happens. Only "hard" or "semi-hard" triplets provide useful gradients. Hard negatives are the closest non-matching images to the anchor — the ones the model is most confused about. Semi-hard negatives are farther than the positive but within the margin — challenging but not impossibly confusing.

In practice, you mine triplets within a mini-batch: compute all pairwise distances, then select the hardest or semi-hardest negatives for each anchor. Batch-hard mining (the most aggressive strategy) picks the hardest positive and hardest negative for each anchor in the batch. This requires large batch sizes (hundreds or thousands of images) to ensure enough diversity for good mining.

Beyond Triplets: ArcFace and Beyond

Triplet loss has a fundamental limitation: it only considers relative distances (positive closer than negative) without enforcing any absolute structure on the embedding space. ArcFace and similar methods (CosFace, SphereFace) take a different approach: they treat metric learning as a modified classification problem. The key idea is to add an angular margin penalty to the softmax classification loss, forcing embeddings of the same class to cluster tightly and different classes to be angularly separated.

ArcFace projects embeddings onto a hypersphere (L2-normalize them) and adds a fixed angular margin m to the angle between the embedding and its class center before computing softmax. This margin forces the model to produce more discriminative embeddings than standard softmax would require. In practice, ArcFace-trained embeddings consistently outperform triplet-loss-trained ones on retrieval benchmarks, and the training is simpler (no triplet mining, just standard classification batches).

The Domain Gap Problem

Metric learning models trained on one domain (e.g., online product photos with white backgrounds) often fail on another domain (e.g., user-uploaded photos with cluttered backgrounds). The model learns domain-specific shortcuts rather than true visual similarity. Always evaluate on realistic query images, not just the same distribution as training. Domain adaptation, data augmentation with realistic backgrounds, and training on diverse data sources all help.

Vector Databases and FAISS

You've extracted embeddings and trained a great metric space. Now you have 100 million images, each represented by a 256-dimensional vector. A user uploads a query image. You need to find the 10 closest vectors. Brute force — computing the distance from the query to all 100 million vectors — takes about 100ms on a modern GPU. That might sound fast, but at 1000 queries per second, you'd need 100 GPUs just for retrieval. You need approximate nearest neighbor (ANN) search.

FAISS (Facebook AI Similarity Search) is the industry-standard library for this. It implements several ANN algorithms that trade a small amount of accuracy (returning the approximately nearest neighbors instead of the exact ones) for orders-of-magnitude speedups.

Key FAISS Index Types

Flat (brute force): IndexFlatL2 or IndexFlatIP. Computes exact distances to every vector. Perfect accuracy, slowest. Use as a baseline or for small datasets (under 1 million vectors).

IVF (Inverted File Index): IndexIVFFlat. Clusters the vectors into nlist Voronoi cells using k-means. At query time, only searches the nprobe nearest cells instead of all vectors. With nlist=4096 and nprobe=64, you search roughly 1.5% of the database — 60× faster than brute force, with recall above 95%.

PQ (Product Quantization): IndexIVFPQ. Compresses each vector from full precision (256 × 4 bytes = 1KB) to a compact code (e.g., 64 bytes). This reduces memory by 16× and makes distance computation faster because it uses lookup tables instead of floating-point math. Some accuracy is lost due to compression, but it's remarkably small — typically less than 5% recall drop.

HNSW (Hierarchical Navigable Small World): IndexHNSWFlat. Builds a graph where vectors are nodes and edges connect nearby vectors. Search follows the graph from a random entry point, greedily moving to closer neighbors. HNSW offers the best query speed (often under 1ms for 100 million vectors) but uses more memory than IVF+PQ because it stores the full vectors plus the graph structure.

import numpy as np
import faiss

# Generate example data: 1 million vectors of dimension 256
dimension = 256
num_vectors = 1_000_000
database = np.random.randn(num_vectors, dimension).astype('float32')
faiss.normalize_L2(database)  # L2 normalize for cosine similarity

# --- Brute force (exact) ---
index_flat = faiss.IndexFlatIP(dimension)  # inner product = cosine sim
index_flat.add(database)

query = np.random.randn(1, dimension).astype('float32')
faiss.normalize_L2(query)

distances, indices = index_flat.search(query, k=10)
print(f"Exact top-10 indices: {indices[0]}")

# --- IVF+PQ (approximate, memory-efficient) ---
nlist = 1024     # number of Voronoi cells
m = 32           # number of PQ sub-quantizers
quantizer = faiss.IndexFlatIP(dimension)
index_ivfpq = faiss.IndexIVFPQ(quantizer, dimension, nlist, m, 8)

index_ivfpq.train(database)     # learn the clustering + codebook
index_ivfpq.add(database)
index_ivfpq.nprobe = 64         # search 64 of 1024 cells

distances_approx, indices_approx = index_ivfpq.search(query, k=10)
print(f"Approx top-10 indices: {indices_approx[0]}")

# Compare: how many of the approximate top-10 are in the exact top-10?
recall = len(set(indices[0]) & set(indices_approx[0])) / 10
print(f"Recall@10: {recall:.0%}")  # typically 90-98%

Choosing the right index is an engineering decision based on your constraints. If you have enough RAM to hold all vectors in memory, HNSW gives the best latency. If memory is tight (billions of vectors), IVF+PQ compresses aggressively. If you need exact results and the dataset is small, brute force is simplest. In production systems, you often combine strategies: use PQ compression for the initial candidate retrieval, then re-score the top candidates with exact distances from full-precision vectors.

GPU-Accelerated Search

FAISS supports GPU indexing, and for brute-force and IVF indices, GPU acceleration provides 5–10× speedups over CPU. A single V100 GPU can brute-force search a 100-million-vector database in about 10ms. For latency-sensitive applications with moderate database sizes, GPU search can eliminate the need for approximate methods entirely.

Re-Ranking

ANN search gives you a candidate set — say, the top 100 most similar images. But the initial ranking is based on a single global embedding, which captures the overall "gist" of an image but can miss details. Re-ranking refines this initial list using more expensive but more accurate comparisons. It's the same idea as a search engine: the first pass uses a cheap index to find candidates, the second pass uses a powerful model to sort them.

Geometric Verification

For tasks like landmark recognition or product matching, geometric verification checks whether two images share spatially consistent feature correspondences. Extract local features (SIFT, SuperPoint, or CNN-based patches), find tentative matches, then use RANSAC to estimate a geometric transformation (affine or homography). If enough inliers survive — meaning the feature matches are geometrically consistent — the images likely show the same physical object. This is expensive (hundreds of milliseconds per pair) but extremely precise, and it catches cases where global embeddings are fooled by similar-looking but different objects.

Query Expansion

Query expansion improves recall by augmenting the original query. After the initial retrieval, take the top-k results that you're confident about, average their embeddings with the query embedding, and search again with this refined "expanded" query. The intuition: the initial results contain additional visual information about what the user is looking for. If the query is a side view of a car, and the top results include front and rear views of similar cars, the expanded query captures a more complete representation of the target.

def query_expansion(query_emb, initial_results, database, index, 
                    top_k_expand=5, alpha=0.5):
    """
    Average the query embedding with top-k result embeddings,
    then re-search with the expanded query.
    """
    # Get embeddings of top-k initial results
    top_embs = database[initial_results[:top_k_expand]]

    # Weighted average: original query + confident results
    expanded = alpha * query_emb + (1 - alpha) * top_embs.mean(axis=0)

    # Re-normalize
    expanded = expanded / np.linalg.norm(expanded)
    expanded = expanded.reshape(1, -1).astype('float32')

    # Search again with expanded query
    distances, indices = index.search(expanded, k=100)
    return distances[0], indices[0]

Learned Re-Ranking

More sophisticated re-ranking uses a dedicated model to score each (query, candidate) pair. This model sees both images and can attend to fine-grained details that the global embedding missed. Cross-attention transformer re-rankers are the current state of the art: they take the query and candidate image features, compute cross-attention between them, and output a refined similarity score. The cost is high — a forward pass per candidate pair — which is why re-ranking only applies to the top 100–1000 candidates from ANN search, not the full database.

Putting the Full Pipeline Together

A production visual search system typically stacks these components:

Stage 1 — Embedding: Extract a 256-d vector from the query image using a metric-learned model. Latency: 5–20ms.

Stage 2 — ANN retrieval: Search a FAISS index to find the top 1000 candidates from a database of 100M+ images. Latency: 1–10ms.

Stage 3 — Re-ranking: Score the top 1000 candidates using a more expensive model (cross-attention, geometric verification, or both). Prune to top 100. Latency: 50–200ms.

Stage 4 — Business logic: Filter by availability, price range, geography, user preferences. Return the top 10–50 to the user. Latency: 1–5ms.

Total end-to-end latency: under 300ms, which feels instantaneous to the user. Each stage acts as a progressively finer sieve — cheap and coarse first, expensive and precise last. This cascade design is fundamental to building retrieval systems that are both accurate and fast at scale.

The Index Update Problem

Real databases aren't static — new products are added daily, old ones are removed. Updating a FAISS IVF index requires either rebuilding (expensive, involves re-training the clustering) or adding vectors to existing cells (fast but degrades quality as cells become unbalanced). Many production systems rebuild the index nightly and serve updates through a small "delta" index that handles the day's additions. Managing this lifecycle is often more engineering effort than building the initial search system.
