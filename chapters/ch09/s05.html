Object Detection

Chapter 9: CNNs & Computer Vision YOLO · Faster R-CNN · DETR · Anchors · NMS
TL;DR

Object detection finds what objects are in an image and where they are, outputting class labels plus bounding boxes. Two-stage detectors (Faster R-CNN) propose regions first, then classify — accurate but slower. One-stage detectors (YOLO, SSD, RetinaNet) predict everything in a single pass — fast enough for real-time. Modern approaches like DETR use Transformers to eliminate hand-crafted components like anchors and NMS entirely. Choosing a detector means choosing a point on the speed–accuracy tradeoff curve.

Classification tells you what's in the image. Detection tells you what and where — with bounding boxes. It's the foundation for autonomous driving, surveillance, medical imaging, and basically any system that needs to find things. If classification is "there's a cat in this photo," detection is "there's a cat at these coordinates, a dog over here, and a person in the background."

Detection is harder than classification in a fundamental way: you don't know how many objects are in the image ahead of time. A classifier always outputs one label. A detector might output zero boxes, or fifty. This variable-length output is what makes detection architecturally interesting — and why the field has produced such a diverse set of approaches.

The Core Toolkit

Before diving into architectures, you need four concepts that every detector uses. Think of these as the shared vocabulary of the detection world.

Bounding Boxes

A bounding box is just an axis-aligned rectangle that says "the object is somewhere inside here." You'll encounter two common formats:

# Format 1: center + size
(cx, cy, w, h)   # center-x, center-y, width, height

# Format 2: corner coordinates
(x1, y1, x2, y2) # top-left corner, bottom-right corner

YOLO uses center format. COCO dataset annotations use (x, y, w, h) where (x, y) is the top-left corner. Pascal VOC uses corner format. Converting between them is trivial — but getting it wrong is one of the most common detection bugs. Always check which format your data and model expect.

IoU (Intersection over Union)

IoU is the metric for detection. It measures how well a predicted box matches a ground-truth box:

IoU = Area of Overlap / Area of Union

IoU ranges from 0 (no overlap at all) to 1 (perfect match). The standard threshold is IoU > 0.5 — if your predicted box overlaps the ground truth by more than 50%, that counts as a correct detection. COCO evaluation uses multiple thresholds (0.5 to 0.95 in steps of 0.05) for a more nuanced picture. IoU is beautifully simple and scale-invariant: a tiny predicted box on a tiny object gets the same IoU as a huge box on a huge object, as long as the relative overlap is the same.

Anchor Boxes

Here's the problem: the network needs to predict boxes, but boxes can be any size and any aspect ratio. Predicting raw coordinates from scratch is hard. Anchor boxes solve this by giving the network a set of predefined box shapes (templates) at each spatial location. Instead of predicting "there's a box at (142, 87) with size (200, 150)," the network predicts "the box at grid cell (3, 2) is 1.2× wider and 0.8× taller than anchor template #4, and shifted 5 pixels right."

Predicting offsets from a good starting point is much easier than predicting absolute coordinates. Common anchor configurations might include 3 aspect ratios (1:1, 1:2, 2:1) at 3 scales, giving 9 anchors per location. Faster R-CNN, SSD, and YOLO (v2+) all use anchors, though the details vary.

Non-Maximum Suppression (NMS)

You get 50 overlapping boxes for one cat. NMS keeps the best one. Here's the algorithm:

  1. Sort all predicted boxes by confidence score (highest first).
  2. Take the top box — it's a keeper.
  3. Remove all remaining boxes that have IoU > threshold (typically 0.5) with the keeper.
  4. Repeat from step 2 with the remaining boxes until none are left.

NMS is simple, effective, and a bit hacky. It works well most of the time, but it can suppress valid detections when objects genuinely overlap (think a crowd of people). Soft-NMS decays scores instead of hard-removing boxes, which helps in crowded scenes. DETR, as we'll see, eliminates the need for NMS entirely.

The Detection Pipeline in One Sentence

Generate candidate regions → predict box offsets from anchors → classify each box → apply NMS to remove duplicates. Every classical detector follows some variation of this pattern. The differences lie in how they generate candidates and when they do the classification.

Two-Stage Detectors: The Accuracy Path

Two-stage detectors split detection into two steps: first propose regions that might contain objects, then classify each proposal and refine its bounding box. This two-step process is more computationally expensive but historically achieves higher accuracy because the classifier focuses only on promising regions.

The R-CNN Family Evolution

R-CNN (2014) was the breakthrough. It used Selective Search (a classical algorithm) to generate ~2000 region proposals per image, warped each one to a fixed size, and fed each through a CNN independently. Accurate, but absurdly slow — each image required 2000 forward passes. Training took 84 hours on a GPU, and inference took 47 seconds per image.

Fast R-CNN (2015) fixed the redundant computation. Instead of running the CNN on each proposal separately, it runs the CNN once on the entire image to produce a feature map, then extracts features for each proposal from that shared map using RoI Pooling. This is dramatically faster — the expensive CNN computation happens once, not 2000 times. But Selective Search was still the bottleneck: it ran on CPU and took about 2 seconds per image.

Faster R-CNN (2015) completed the picture by replacing Selective Search with a Region Proposal Network (RPN) — a small CNN that runs on the shared feature map and proposes regions. The RPN slides a small window over the feature map and, at each position, predicts whether each anchor box contains an object and how to refine it. Now the entire pipeline — feature extraction, proposal generation, classification, and box regression — is a single neural network trained end-to-end. Proposals went from 2 seconds to 10 milliseconds.

# Faster R-CNN pipeline (conceptual)
image → Backbone CNN → Feature Map
                           ↓
                    Region Proposal Network (RPN)
                           ↓
                    ~300 region proposals
                           ↓
                    RoI Align → Fixed-size features
                           ↓
                    Classification head + Box regression head

RoI Align vs. RoI Pooling

RoI Pooling has a subtle problem: when a proposed region doesn't align perfectly with the feature map grid, it gets quantized (snapped to the nearest grid cell). This introduces small spatial misalignments that hurt precision, especially for small objects. RoI Align (from Mask R-CNN, 2017) fixes this by using bilinear interpolation instead of rounding. No quantization, no artifacts. It's a small change that mattered a lot for pixel-level tasks like instance segmentation. If you're using any modern two-stage detector, RoI Align is the default.

Why Two Stages?

The first stage (RPN) acts as an attention mechanism — it eliminates 99% of the image that's just background. The second stage then has a much easier job: classify and refine a few hundred promising regions instead of millions of possible locations. This divide-and-conquer approach is why two-stage detectors remain competitive on accuracy benchmarks.

One-Stage Detectors: The Speed Path

One-stage detectors skip the proposal step entirely. They predict class labels and bounding boxes directly from the feature map in a single forward pass. Less accurate historically, but much faster — fast enough for real-time video.

YOLO: You Only Look Once

YOLO (2016) was a paradigm shift. Instead of "propose then classify," YOLO divides the image into an S×S grid. Each grid cell predicts B bounding boxes, confidence scores for those boxes, and class probabilities — all at once. The entire detection happens in a single forward pass through the network.

# YOLO grid prediction (conceptual)
Input image → CNN backbone → S × S × (B × 5 + C) output tensor

# For each grid cell:
#   B boxes, each with (cx, cy, w, h, confidence)
#   C class probabilities
# Total predictions per cell: B × 5 + C

YOLOv1 was fast (45 FPS) but struggled with small objects and objects near grid boundaries. Each grid cell could only predict B objects, so densely packed scenes were problematic.

The YOLO Evolution

The YOLO family has evolved dramatically:

  • YOLOv2 (2017) — Added anchor boxes, batch normalization, multi-scale training. Much more accurate.
  • YOLOv3 (2018) — Predictions at three scales using a Feature Pyramid Network (FPN). Better for small objects. Darknet-53 backbone.
  • YOLOv4 (2020) — Bag of freebies (training tricks that cost nothing at inference) and bag of specials (architecture tweaks). CSPDarknet backbone, PANet neck, Mish activation.
  • YOLOv5 (2020) — PyTorch implementation by Ultralytics. Debated naming (no paper), but became the most widely-used version due to excellent engineering and ease of use.
  • YOLOv8 (2023) — Anchor-free design, decoupled head for classification and regression, strong performance across detection, segmentation, and pose estimation.
  • YOLO11 (2024) — Latest iteration with improved efficiency and accuracy across multiple vision tasks.

The trend is clear: each version adds better backbones, better feature fusion, better training strategies, and more task generality. Modern YOLO models are anchor-free, use decoupled heads, and rival two-stage detectors in accuracy while maintaining real-time speed.

SSD: Multi-Scale Feature Maps

SSD (Single Shot MultiBox Detector, 2016) tackled YOLO's weakness with small objects. Its key insight: detect objects at multiple feature map resolutions. Early layers (high resolution) detect small objects. Later layers (low resolution, large receptive field) detect large objects. SSD attaches prediction heads to multiple layers of the backbone, each with its own set of anchors scaled to that resolution.

This multi-scale approach became standard — it shows up in Feature Pyramid Networks (FPN), YOLOv3+, and virtually every modern detector.

RetinaNet and Focal Loss

For years, one-stage detectors lagged behind two-stage ones in accuracy. The culprit wasn't architecture — it was class imbalance. In a typical image, the detector evaluates ~100,000 candidate locations, but only ~10 contain actual objects. The overwhelming majority are easy background examples. Standard cross-entropy loss assigns equal weight to all examples, so the model spends most of its training time on trivially easy "this is background" predictions. The sheer volume of easy negatives drowns out the learning signal from the few hard, useful examples.

Focal Loss (2017) fixed this with an elegant modification:

# Standard cross-entropy
CE(p) = -log(p)

# Focal loss — down-weight easy examples
FL(p) = -(1 - p)^γ × log(p)

# γ = 0: standard CE
# γ = 2: (typical) easy examples with p > 0.9 get ~100× less weight

When the model is already confident (p close to 1 for the correct class), the (1 - p)^γ factor crushes the loss to near zero. The model focuses its learning on the hard examples it's getting wrong. With focal loss, RetinaNet became the first one-stage detector to match two-stage accuracy while maintaining one-stage speed. It proved the one-stage gap was a training problem, not an architecture problem.

The Class Imbalance Lesson

Focal loss is important beyond detection. Whenever you have extreme class imbalance and a dense prediction task (many predictions per input), standard cross-entropy will waste training capacity on trivial examples. Focal loss, or its variants, should be your go-to solution. It's one of those ideas that seems obvious in hindsight but took years to figure out.

DETR: Transformers Enter Detection

DETR (DEtection TRansformer, 2020) reimagined detection from the ground up. Instead of anchors, NMS, and hand-designed post-processing, DETR treats detection as a direct set prediction problem. It uses a Transformer encoder-decoder architecture and a bipartite matching loss to produce a fixed-size set of predictions — one per object, no duplicates.

# DETR pipeline
image → CNN backbone → flatten to sequence → Transformer Encoder
    → Transformer Decoder (with N learned object queries)
    → N predictions: (class, box) pairs

# N is fixed (e.g., 100). Unused slots predict "no object."

The magic is in the bipartite matching: during training, DETR uses the Hungarian algorithm to find the optimal one-to-one assignment between predictions and ground-truth objects. Each ground-truth object matches exactly one prediction. This eliminates duplicate detections by design — no NMS needed.

DETR is elegant. No anchors, no NMS, no hand-tuned components. The entire detection pipeline is end-to-end differentiable. But it comes with costs: it's slow to train (500 epochs vs. ~36 for Faster R-CNN) and initially struggled with small objects because the Transformer's global attention doesn't naturally handle multi-scale features.

Follow-up work addressed these issues: Deformable DETR uses deformable attention (attending to a small set of sampling points rather than all locations) and converges 10× faster. DINO further improved training efficiency and accuracy, becoming competitive with the best traditional detectors. The DETR lineage shows that Transformers can work for detection — they just needed engineering refinements to become practical.

Anchor-Free Detectors

Anchors work, but they're a headache. You need to choose aspect ratios, scales, and counts. Get it wrong and you miss objects. A parallel line of research asked: can we skip anchors entirely?

  • CornerNet (2018) — Predicts object corners (top-left and bottom-right) as heatmaps, then groups them into boxes using associative embeddings.
  • CenterNet (2019) — Predicts object centers as heatmaps, then regresses width and height from each center point. Beautifully simple.
  • FCOS (2019) — Fully Convolutional One-Stage detector. Each pixel in the feature map predicts distances to the four edges of the nearest object's bounding box, plus a "centerness" score to suppress low-quality predictions far from object centers.

Anchor-free methods are simpler to implement, have fewer hyperparameters, and avoid the anchor-object matching complexity. Modern YOLOv8+ and many state-of-the-art detectors have adopted anchor-free designs. The field has largely moved away from explicit anchors.

Detector Comparison

Detector Type Speed Accuracy When to Use
Faster R-CNN Two-stage Medium (~5-15 FPS) High Accuracy-critical applications; offline processing
YOLOv8/YOLO11 One-stage, anchor-free Very fast (~80-160 FPS) High Real-time detection; video; edge deployment
SSD One-stage Fast (~30-60 FPS) Medium Embedded systems; legacy codebases
RetinaNet One-stage Medium (~10-20 FPS) High When you need one-stage speed with two-stage accuracy
DETR / DINO Transformer Medium (~10-20 FPS) Very high Research; when you want NMS-free pipelines
FCOS / CenterNet One-stage, anchor-free Fast (~30-50 FPS) High Simple pipeline; fewer hyperparameters

The speed numbers are rough guides — actual performance depends on backbone, input resolution, hardware, and batch size. The key takeaway is the tradeoff: more stages and more computation buy you accuracy; simplicity and single-pass designs buy you speed. Modern one-stage detectors have closed much of the accuracy gap, making them the default choice for most practical applications.

Evaluation Metrics

Detection evaluation is more nuanced than classification. You can't just compute accuracy — you need to handle the fact that detections can be correct at different IoU thresholds and that there are different objects to detect.

Precision, Recall, and AP

For each class, you rank all detections by confidence, then walk down the list computing precision and recall at each step. Plot precision vs. recall — the area under that curve is Average Precision (AP) for that class. Intuitively, AP measures how well the model ranks true detections above false ones.

mAP (Mean Average Precision)

Average the AP across all object classes. This is mAP — the single number most commonly used to compare detectors. But which IoU threshold? Conventions vary:

  • mAP@0.5 (Pascal VOC style) — A detection is correct if IoU > 0.5. Lenient.
  • mAP@0.75 — Stricter. Requires tighter boxes.
  • mAP@[0.5:0.95] (COCO style) — Average mAP across IoU thresholds from 0.5 to 0.95 in steps of 0.05. This is the gold standard. Rewards detectors that produce tight, precise boxes.

COCO Metrics

COCO evaluation also breaks down performance by object size:

  • APS — Small objects (area < 32²)
  • APM — Medium objects (32² < area < 96²)
  • APL — Large objects (area > 96²)

This is invaluable because most detectors have very different performance on small vs. large objects. A detector with great overall mAP might be terrible on small objects. Always check the size breakdown.

Quick Metric Reference

IoU — how well one box matches another (0 to 1).
AP — area under precision-recall curve, per class, at a given IoU threshold.
mAP — mean AP across classes.
mAP@[0.5:0.95] — the standard COCO metric. Use this when comparing detectors.
APS/APM/APL — tells you where your detector succeeds and fails by object size.
