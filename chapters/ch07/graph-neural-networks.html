Deep learning often assumes ordered data — images as grids, text as sequence. Much world not like that: social connections, molecular structures, recommendation systems. When relationships carry as much signal as entities, need architecture that treats structure as first-class input. Graph Neural Networks do that.

Graph Neural Networks form family of neural architectures for graph-structured data — nodes linked by edges, with features on both. Core ideas trace to mid-2000s, modern era starts around 2017 with Kipf & Welling's GCN paper. Since then, GNNs became go-to tool for molecular property prediction, social network analysis, recommendation engines, knowledge graph reasoning.

What is a graph, really

Start concrete. Imagine three friends — Alice, Bob, Carol. Alice and Bob know each other. Bob and Carol know each other. Alice and Carol don't. Graph: three nodes (people), two edges (friendships).

Graph = collection of things plus connections between them. Things are nodes (vertices). Connections are edges (links). Unlike image grid or left-to-right sentence, graph has no inherent ordering. No "first" node. No spatial layout. Only connections.

Graphs feel powerful and awkward. Powerful because much world is relational — atoms in molecule, users following each other, entities linked in knowledge base. Awkward because most deep learning machinery expects neat, ordered arrays.

The adjacency matrix

Need way to hand friendship graph to computer. Standard approach: adjacency matrix — square grid where rows and columns represent nodes, and 1 at position (i, j) means node i connects to node j.

For three friends, label Alice = 0, Bob = 1, Carol = 2:

        Alice  Bob  Carol
Alice  [  0     1     0  ]
Bob    [  1     0     1  ]
Carol  [  0     1     0  ]

This is undirected graph — if Alice knows Bob, Bob knows Alice — so matrix is symmetric. 1 at row 0, column 1 (Alice→Bob) mirrors row 1, column 0 (Bob→Alice). If friendships were one-way — like Twitter follows — graph would be directed and matrix might not be symmetric.

First time I saw adjacency matrices, they felt wasteful. 3×3 grid to store two friendships? But format is perfect for linear algebra behind GNNs. Multiply matrix by vector of node features, and each node gathers neighbor information. That observation drives rest of chapter.

Node features and edge features

Graph with only connectivity is like spreadsheet with row labels and no data. In practice, each node carries feature vector — list of numbers describing it. For friends, maybe movie ratings:

# Each person rates three movies: Action, Comedy, Drama
Alice = [5, 1, 3]   # loves action
Bob   = [4, 4, 2]   # balanced tastes
Carol = [1, 5, 4]   # comedy and drama fan

Stack these into node feature matrix X, where each row holds one node's features. Here, X is 3×3 — three nodes, three features each.

Edges can carry features too. In molecule, bond between atoms might encode single, double, or aromatic bond. In social network, edge might carry number of messages exchanged. We won't use edge features much here, but they matter — many real-world GNN applications use them.

Neighborhood gossip — why structure matters

Now structure matters. Say we want predict what Carol thinks of new action movie. If we only use Carol's features — ratings [1, 5, 4] — guess is no. Low action score.

But Carol connects to Bob, and Bob likes action movies. If friends influence each other's taste — real world says yes — Carol's link to Bob is signal. Maybe she is more open to action than her own ratings suggest.

Core GNN insight: node identity comes not only from own features, but from connections. Like neighborhood gossip — learn about person by listening to neighbors. Then neighbors' neighbors. Deeper layers gather more context.

Think of it this way. Movie preferences alone tell little. Add three closest friends' preferences, learn more. Add friends' friends, get richer social context shaping taste. GNN does exactly that, layer by layer.

Question: how do we do this mathematically?

The message passing framework

Every modern GNN — every one — is variation of same three-step loop. One loop per layer.

First, each node prepares message. Usually node's current feature vector, maybe transformed by weight matrix. In movie example, Alice prepares ratings [5, 1, 3] as message.

Second, each node aggregates messages from neighbors. Aggregate means combine into one summary. Function might be sum, mean, or max — but it must be permutation-invariant. Crucial detail. Graphs have no inherent node ordering, so result cannot change if neighbor order changes. Sum ignores order. Mean ignores order. Function that reads neighbors left-to-right would care, and that breaks everything.

Third, each node updates own representation by combining current features with aggregated neighbor message, usually through learnable weight matrix and nonlinearity like ReLU.

Trace it concretely. Bob connects to Alice and Carol. In message step, Alice sends [5, 1, 3] and Carol sends [1, 5, 4]. Bob aggregates by averaging: [(5+1)/2, (1+5)/2, (3+4)/2] = [3, 3, 3.5]. Then Bob combines that with own features [4, 4, 2] through weight matrix and nonlinearity. Exact combination depends on architecture, but dance stays same: message, aggregate, update.

That is whole framework. Understand this loop, understand skeleton of every GNN architecture published. Differences between GCN, GAT, and GraphSAGE come from how they run these three steps — how they weight messages, aggregate, update. Skeleton stays identical.

Matrix multiplication connects directly here. Take adjacency matrix A and multiply by node feature matrix X. Result is new matrix where each row contains sum of that node's neighbors' features. Single multiply — A × X — is message passing step.

# A is the adjacency matrix, X is the node feature matrix
# A @ X gathers neighbor features for every node simultaneously

A = [[0, 1, 0],    X = [[5, 1, 3],    A @ X = [[4, 4, 2],
     [1, 0, 1],         [4, 4, 2],              [6, 6, 7],
     [0, 1, 0]]         [1, 5, 4]]              [4, 4, 2]]

# Row 0 (Alice): got Bob's features [4,4,2]
# Row 1 (Bob):   got Alice's [5,1,3] + Carol's [1,5,4] = [6,6,7]
# Row 2 (Carol): got Bob's features [4,4,2]

One matrix multiplication, and every node has gathered neighbor information. Stack learnable weight matrix W after it, add nonlinearity, and you have GNN layer.

GCN — the weighted average that started it all

In 2017, Thomas Kipf and Max Welling published paper that turned GNNs from theoretical curiosity into practical tool. Their insight: take complex spectral theory of graph convolutions — eigendecompositions of graph Laplacian and Chebyshev polynomial approximations — and simplify it into one elegant layer rule.

Backstory matters. Mathematicians had defined "convolution" on graphs in frequency domain — transform node signals with graph eigenvectors, apply filter, transform back. Expensive because it is expensive. It required eigendecomposition of graph Laplacian, impractical for reasonably sized graph. Defferrard et al. (2016) made it cheaper by approximating spectral filter with Chebyshev polynomials up to order K. Kipf and Welling asked: what if K = 1? What if we use crudest possible approximation?

Resulting layer rule: take each node's features, average with neighbors' features (including itself), multiply by learnable weight matrix, apply nonlinearity. That is entire Graph Convolutional Network.

Formula: H(l+1) = σ( D̃−½ Ã D̃−½ H(l) W(l) )

Unpack symbols. H(l) is matrix of node features at layer l — each row one node representation. W(l) is learnable weight matrix for layer. σ is nonlinearity, usually ReLU. Ã is A + I — adjacency matrix with self-loops, so each node also includes own features during aggregation. D̃ is degree matrix of Ã — diagonal matrix where entry D̃ii counts connections for node i, including self-loop. D̃−½ on both sides performs symmetric normalization.

Why normalization matters. Without it, node with 500 neighbors would aggregate features 500 times larger than node with 2 neighbors. Numerical scale blows up. Training gets unstable. Symmetric normalization — dividing by square root of both source and target degrees — keeps everything comparable. Why symmetric version beats simpler choices like dividing by degree alone traces back to spectral theory. In practice, it gives more stable gradients and better performance.

Walk through GCN on three-friend graph. Add self-loops, so Ã becomes:

Ã = [[1, 1, 0],    # Alice: connected to self + Bob (degree 2)
     [1, 1, 1],    # Bob: connected to self + Alice + Carol (degree 3)
     [0, 1, 1]]    # Carol: connected to self + Bob (degree 2)

After symmetric normalization, each entry is divided by √(deg_i × deg_j). Bob's features get slightly less weight when aggregated into Alice's representation — Bob is more "popular" (higher degree), so signal gets diluted. By design. Without this, hub nodes would dominate representations of everyone they touch.

Power of GCN: simplicity. One matrix multiply to gather neighbors, one matrix multiply to transform features, one nonlinearity. Weakness: every neighbor gets same weight after normalization. Fine for many problems. Bad when some connections matter much more than others.

GAT — paying attention to the right neighbors

GCN averages all neighbors equally. But in friend analogy, if you want predict what Carol likes, does friend being movie critic matter more than friend who watches one movie per year? Yes. Some neighbors carry more information, and network should learn who to listen to.

This motivates Graph Attention Networks, introduced by Veličković et al. in 2018. Idea comes from attention mechanism in Transformers — with one critical difference. In Transformer, every token attends to every other token (global attention). In GAT, each node attends only to immediate neighbors (local attention, set by graph structure). That makes GAT sparse and efficient.

Step by step for one node. Say we want Bob's new representation. Bob has two neighbors: Alice and Carol.

First, project everyone's features through shared weight matrix W. Bob's features become WhBob, Alice's become WhAlice, Carol's become WhCarol.

Next, compute attention score for each neighbor pair. For (Bob, Alice), concatenate WhBob and WhAlice into one long vector, take dot product with learnable vector a, apply LeakyReLU. Result, eBob,Alice, is raw score for how much Alice's features matter to Bob.

Do same for (Bob, Carol) to get eBob,Carol. Then normalize both scores with softmax across Bob's neighbors so they sum to one. These normalized scores — αBob,Alice and αBob,Carol — are attention weights. Bob's updated representation is weighted sum: αBob,Alice · WhAlice + αBob,Carol · WhCarol.

If you've seen Transformer attention, this feels familiar but different. No separate query/key/value projection. GAT uses one shared weight matrix plus separate attention vector. Leaner mechanism. Works well because graph neighborhoods are usually small — tens of neighbors, not thousands of tokens.

One practical note: GAT uses multi-head attention, same trick that stabilizes Transformers. Multiple independent heads compute their own weights, and results are concatenated (or averaged in final layer). This lets model capture different relationship types at same time.

Original GAT has subtle limitation discovered later. Attention scores come from concatenation followed by dot product with fixed vector — and ranking of scores can stay same regardless of query node. Brody et al. (2021) fixed this in GATv2 by applying nonlinearity after concatenation, making attention truly dynamic. For new project, use GATv2.

GraphSAGE — sampling your way to scale

GCN and GAT share one fundamental assumption: whole graph is available during training. Every node participates in message passing, even unlabeled ones. Fine for citation network with few thousand nodes. Impossible for Pinterest graph with billions of pins and users.

Hamilton, Ying, and Leskovec (2017) tackled this with GraphSAGE — SAmple and aggreGatE. Name captures two core ideas.

First idea: inductive learning. Instead of learning fixed embedding for each node — which requires seeing every node during training and fails on new nodes — GraphSAGE learns aggregation function. When new user signs up, feed neighborhood through same learned function and get embedding immediately. No retraining.

Second idea: neighborhood sampling. Instead of aggregating all neighbors — which for hub node could mean millions of connections — GraphSAGE randomly samples fixed number at each layer. Say 25 neighbors at first hop and 10 at second hop. This caps computation per node and makes mini-batch training possible. Sampling adds variance — you do not always see same neighbors — but in practice that acts like regularization and works well.

Back to gossip analogy. GCN and GAT are like asking every person in neighborhood what they think. GraphSAGE is like asking random handful at coffee shop. You do not hear from everyone, but if you ask enough times (enough training iterations), you get gist.

GraphSAGE offers several aggregation functions: mean (default, hard to beat), LSTM (they shuffle neighbor order each time because LSTMs care about order — clever hack), and max-pool (apply MLP to each neighbor, then take element-wise maximum). Mean aggregator is what most practitioners use.

Practical impact is enormous. GraphSAGE-style sampling made GNNs deployable at companies like Pinterest, Uber, Twitter. If you need to train GNN on graph that does not fit in GPU memory, start here.

Three kinds of questions you can ask a graph

We have focused on how GNNs compute node representations, but what do we do with them? Three task types. Difference is which entity gets prediction.

Node-level tasks. Most common setup. Given partially labeled graph, predict labels for unlabeled nodes. This is semi-supervised learning — message passing runs across entire graph, including unlabeled nodes, but loss function only looks at labeled nodes. Think: classifying academic papers by topic, detecting fraudulent accounts in transaction network, predicting protein functions in biological interaction graph.

Edge-level tasks. Question: which edges are missing? Or which edges form next? Run GNN to get node embeddings, then score candidate node pairs — simplest decoder is dot product between two node embeddings. High score means "these two nodes should probably be connected." This powers "people you may know" and knowledge graph completion (predict missing facts like "who directed this movie?"). Beginner trap: if target edge stays in graph during message passing, model can trivially detect it. Always remove edge you are trying to predict.

Graph-level tasks. One prediction for entire graph. Is molecule toxic? Is program's control flow graph malicious? This needs one extra step beyond message passing: pooling.

Graph pooling — squeezing a graph into a vector

Node-level tasks are straightforward — each node already has embedding after message passing. But graph-level tasks need one vector representing whole graph. Pooling gives that.

Simplest approaches are global mean pooling (average all node embeddings) and global sum pooling (add them up). No learnable parameters. Fast. Surprisingly effective for many tasks. Sum pooling has theoretical advantage: it can distinguish graphs with different node counts but same average features. Mean pooling cannot.

When you need more expressiveness, learned pooling methods exist. DiffPool (Ying et al., 2018) learns soft cluster assignment matrix — it groups nodes into supernodes at each pooling step, building hierarchy. Coarsened graph goes through more GNN layers. Powerful but memory-hungry. SAGPool uses attention scores to select top-k most important nodes and drops rest, creating sparser graph for next layer.

For most practical work, global mean or sum pooling works well. Learned methods shine on benchmarks with highly structured graphs where hierarchy matters — certain molecular and biological datasets — but they add complexity and compute cost.

Over-smoothing — the blur problem

Here is trap I promised earlier. Each GNN layer lets every node absorb information from immediate neighbors. Two layers: friends-of-friends. Three layers: three hops out. Natural instinct is stacking more layers, like deep CNNs or Transformers. Deeper network, richer representations, right?

Wrong. In GNNs, more depth makes representations worse after certain point. This is over-smoothing. Helpful analogy: blurring photograph.

Each GNN layer is like blur filter on image. One pass gently smooths noise while keeping important features. Two passes and detail starts disappearing. Five passes and everything becomes washed-out smudge. Ten passes and every pixel converges to same average gray.

Mathematics says same thing. Repeated multiplication by normalized adjacency matrix drives node representations toward dominant eigenvector — space where every node looks same. In small-world graph, like most social networks, five hops can reach nearly every node. That means five layers of message passing gives every node access to almost entire graph. Then representations collapse.

In practice, 2–3 layers is sweet spot for most GNNs. Standard explanation: many real-world graphs have small diameters, so two hops already reaches most relevant context.

Mitigations exist. Residual connections (add input of each layer to output, like ResNet) help preserve node identity. DropEdge randomly removes edges during training, slowing mixing. JumpingKnowledge networks concatenate representations from all layers, letting final classifier choose most useful depth for each node. PairNorm explicitly pushes node representations apart during normalization.

None fully solve problem. If you genuinely need long-range dependencies — information traveling ten or twenty hops — emerging approach is Graph Transformers, which allow global attention between all nodes and sidestep layer-by-layer propagation bottleneck. But that is different chapter.

What GNNs can't see — the Weisfeiler-Lehman wall

There is deeper limitation. Not about depth — about what message-passing GNNs can fundamentally distinguish.

In 1968, Boris Weisfeiler and Andrei Lehman developed test for whether two graphs are structurally identical (isomorphic). Test, now called 1-WL test or color refinement, works like this: give every node initial "color" based on features. Then iteratively update each node's color from multiset of neighbors' colors. If, after some rounds, two graphs have different color distributions, they are definitely not isomorphic. If distributions match, they might be isomorphic — test cannot always tell.

Punchline: Xu et al. (2019) proved standard message-passing GNNs are at most as powerful as 1-WL test. Iterative "gather neighbors, update" loop in GNN is same computation as color refinement. Any graph pair that fools 1-WL also fools message-passing GNN.

What graphs fool it? Highly symmetric ones. Two different six-node graphs where every node has same number of neighbors can be indistinguishable to 1-WL, and therefore to GNNs, even though human can see different structure. For most practical applications — molecules, social networks, recommendations — ceiling rarely matters. But for some molecular structures where subtle symmetry changes chemical properties, it matters.

Same paper introduced GIN (Graph Isomorphism Network), GNN architecture designed to be maximally powerful — matching 1-WL exactly. Key design choice: sum aggregation, not mean or max, plus injective MLP update function. GCN and GraphSAGE with mean aggregation are strictly less powerful because mean and max can map different multisets to same output.

Going beyond 1-WL barrier requires going beyond standard message passing — higher-order GNNs that consider groups of nodes together, or architectures that incorporate subgraph structures. These exist but cost heavy compute. For most practitioners, knowing ceiling exists and using GIN when expressiveness matters is enough.

Where GNNs live in the real world

Theory is nice, but where do GNNs earn keep?

Molecules. This is where GNNs fit most naturally. Atom is node, bond is edge, task is predicting molecular properties — toxicity, solubility, binding affinity, reaction outcomes. Graph structure is not convenient abstraction here; it is molecule. Drug discovery pipelines at pharmaceutical companies and startups use GNNs for virtual screening — narrowing millions of candidate compounds to handful worth synthesizing and testing in lab. GNNs also get used for retrosynthesis — given target molecule, figuring out what reactions produce it.

Social networks. Fraud detection is major application. Fraudulent accounts tend to cluster together — they follow each other, interact with same targets — and GNNs are excellent at detecting these community patterns. Pinterest's PinSage system (built on GraphSAGE) generates embeddings for billions of pins by treating user-pin interaction graph as data structure. Influence prediction — figuring out which users are most likely to spread information — is another natural fit.

Recommendation systems. At scale, recommendations are graph problem. Users and items are nodes. Interactions (purchases, clicks, ratings) are edges. GNNs propagate preference signals through this bipartite graph. LightGCN, simplified GCN variant for recommendations, showed that stripping away feature transformations and nonlinearities and keeping only neighborhood aggregation can actually improve recommendation quality. Sometimes simpler wins.

Knowledge graphs. Knowledge graph stores facts as (entity, relation, entity) triples — (Berlin, capital_of, Germany). Graph is massive and incomplete. GNNs help with link prediction: given existing facts, predict missing ones. R-GCN (Relational GCN) was one of first architectures to handle multiple relation types. Knowledge graph completion powers question answering systems, search engines, personal assistants.

Beyond these four, GNNs show up in traffic prediction (road networks), combinatorial optimization (graph-structured problems like TSP), program analysis (code as graph of functions and calls), and physics simulation (particles as nodes, interactions as edges). Common thread: when relationships carry as much signal as entities, GNNs are worth considering.

Tools of the trade — PyG and DGL

Two frameworks dominate GNN development: PyTorch Geometric (PyG) and Deep Graph Library (DGL).

PyG feels like home if you already use PyTorch. API mirrors PyTorch conventions — define layers, write forward method, run training loop. Graphs are stored as edge_index tensor (2×num_edges array listing source-target pairs) plus node feature matrix x. Everything is sparse by default. Swapping GCN for GAT or GraphSAGE is often one-line change.

import torch
import torch.nn.functional as F
from torch_geometric.nn import GCNConv
from torch_geometric.datasets import Planetoid

# Cora: 2708 papers, 10556 citation edges, 7 subject classes
dataset = Planetoid(root='./data', name='Cora')
data = dataset[0]

class TwoLayerGCN(torch.nn.Module):
    def __init__(self, in_dim, hidden_dim, out_dim):
        super().__init__()
        self.conv1 = GCNConv(in_dim, hidden_dim)
        self.conv2 = GCNConv(hidden_dim, out_dim)

    def forward(self, x, edge_index):
        # First message passing layer + nonlinearity
        x = F.relu(self.conv1(x, edge_index))
        x = F.dropout(x, p=0.5, training=self.training)
        # Second message passing layer (raw logits for classification)
        return self.conv2(x, edge_index)

model = TwoLayerGCN(dataset.num_features, 64, dataset.num_classes)
optimizer = torch.optim.Adam(model.parameters(), lr=0.01, weight_decay=5e-4)

for epoch in range(200):
    model.train()
    optimizer.zero_grad()
    out = model(data.x, data.edge_index)
    # Loss only on labeled nodes — semi-supervised setup
    loss = F.cross_entropy(out[data.train_mask], data.y[data.train_mask])
    loss.backward()
    optimizer.step()

Training loop looks exactly like any other PyTorch classification task. Only GNN-specific detail: forward takes node features plus edge connectivity, and loss is computed only on nodes with labels (train_mask). Unlabeled nodes still participate in message passing — their features flow through graph and influence labeled nodes' representations — but they do not contribute to loss.

Large graphs need NeighborLoader. It implements GraphSAGE-style sampling:

from torch_geometric.loader import NeighborLoader

# Sample 15 first-hop, 10 second-hop neighbors per target node
loader = NeighborLoader(
    data, num_neighbors=[15, 10], batch_size=128, shuffle=True
)

for batch in loader:
    out = model(batch.x, batch.edge_index)
    loss = F.cross_entropy(
        out[:batch.batch_size], batch.y[:batch.batch_size]
    )

For batching multiple small graphs, common in molecular tasks, PyG stacks them into one large disconnected graph and tracks graph membership for each node via batch vector. That lets you use global pooling — global_mean_pool(x, batch) — to get one vector per graph.

DGL (backed by AWS) takes different approach. Its message-passing API is more explicit — you define message functions and reduce functions operating on edges and nodes. More verbose but finer control. That matters for heterogeneous graphs (multiple node and edge types) and distributed training across multiple machines. If you are building production system at scale — billions of edges, multiple GPUs, real-time updates — DGL is often better choice. For research, prototyping, standard benchmarks, PyG is where most practitioners start.
