Loss Functions

Chapter 7: Deep Learning Foundations 5 subtopics
TL;DR

A loss function is how a neural network knows it's wrong — and by how much. It takes a prediction and the true answer, then spits out a single number: the "badness" score. Training is just the process of making that number smaller. Different tasks need different loss functions: squared error for regression, cross-entropy for classification, margin-based losses for learning similarity. Choosing the wrong loss function is like grading a math test with a spelling rubric — the model optimizes for the wrong thing. This section breaks down every major loss function, its math, its intuition, and when to reach for each one.

Why Loss Functions Exist

Imagine you're teaching someone to throw darts. They throw, and you say "you missed." That's useless. They throw again, and you say "you missed by 3 inches to the left." Now they can adjust. A loss function is that second kind of feedback — it doesn't just tell the model it's wrong, it tells the model how wrong and in what direction.

Without a loss function, a neural network is just a pile of numbers with no purpose. The loss function defines the purpose. It takes the model's prediction ŷ and the true value y, and computes a scalar that represents how far apart they are. During training, the optimizer's entire job is to adjust the model's weights so this scalar gets smaller. Everything — backpropagation, gradient descent, learning rate schedules — exists in service of minimizing a loss function.

The choice of loss function matters more than most beginners realize. Two models with identical architectures trained on identical data can produce wildly different behavior simply because they optimize different loss functions. The loss function encodes your priorities: Do you care more about large errors or small ones? Should confident-but-wrong predictions be punished harshly? Are false positives worse than false negatives? The loss function is where these decisions live in the math.

Regression Losses

Regression means predicting a continuous number — house prices, temperatures, stock returns. The loss measures the distance between the predicted number and the actual number. The question is: how do you define "distance"?

Mean Squared Error (MSE)

Mean Squared Error is the default loss for regression, and probably the first loss function anyone learns. The formula is:

MSE = (1/n) × Σ(yᵢ - ŷᵢ)²

Take each prediction, subtract the true value, square the difference, and average across all samples. Squaring does two things: it makes all errors positive (a prediction that's 5 too high and 5 too low both contribute 25), and it penalizes large errors disproportionately. An error of 10 doesn't contribute twice as much as an error of 5 — it contributes four times as much (100 vs 25).

This quadratic penalty is both MSE's superpower and its weakness. It's a superpower because the gradient is smooth and well-behaved — the derivative is simply 2(ŷ - y), which means the bigger the error, the bigger the gradient, and the faster the model corrects. The optimization landscape is a nice convex bowl with a clear bottom. It's a weakness because outliers dominate the loss. One sample that's off by 100 contributes 10,000 to the total, drowning out a thousand samples that are each off by 1 (contributing 1,000 total). If your data has outliers, MSE will warp your model trying to accommodate them.

Mean Absolute Error (MAE)

Mean Absolute Error takes a different approach — just use the absolute value instead of squaring:

MAE = (1/n) × Σ|yᵢ - ŷᵢ|

Now an error of 10 contributes exactly twice as much as an error of 5 (10 vs 5). No quadratic blowup. This makes MAE far more robust to outliers. The model won't contort itself to fit a few wild data points.

But there's a tradeoff. The gradient of the absolute value is either +1 or -1 regardless of how close you are to the target. When you're way off, the gradient is 1. When you're almost exactly right, the gradient is still 1. This means the model takes the same size steps near the solution as it does far away, making convergence slower and less stable. At exactly zero error, the gradient is undefined (the absolute value has a kink at zero), which can cause oscillation around the minimum.

MSE vs MAE — Concrete Example

Suppose you have 5 predictions with errors: [1, 1, 1, 1, 20]. That last one is an outlier.
MAE = (1+1+1+1+20)/5 = 4.8
MSE = (1+1+1+1+400)/5 = 80.8
The outlier contributes 20/24 = 83% of the MAE, but 400/404 = 99% of the MSE. With MSE, the model will move heaven and earth to fix that one outlier. With MAE, the outlier matters, but it doesn't dominate. If the outlier is noise, MAE gives you a better model. If the outlier is real and important, MSE is more responsive to it.

Huber Loss

What if you want the best of both worlds? Huber Loss is a piecewise function that behaves like MSE when errors are small and like MAE when errors are large:

Huber(y, ŷ) =
  0.5 × (y - ŷ)²           if |y - ŷ| ≤ δ
  δ × |y - ŷ| - 0.5 × δ²   if |y - ŷ| > δ

The hyperparameter δ (delta) controls the transition point. Below δ, you get the smooth quadratic gradient of MSE, so the model converges precisely when it's close to the answer. Above δ, you get the linear penalty of MAE, so outliers don't blow up the loss. You can think of Huber as a "shock absorber" — smooth ride on normal road, dampened response on potholes.

The catch: you have to pick δ, and there's no universal right answer. Too small and you're basically using MAE. Too large and you're basically using MSE. In practice, people set δ based on the expected error scale of their problem — often trying a few values and seeing what works.

Log-Cosh Loss

Log-Cosh Loss is a smooth alternative to Huber that doesn't require choosing a threshold:

Log-Cosh(y, ŷ) = Σ log(cosh(ŷᵢ - yᵢ))

The function log(cosh(x)) is approximately x²/2 for small x (behaves like MSE) and approximately |x| - log(2) for large x (behaves like MAE). Unlike Huber, it's twice differentiable everywhere, which means second-order optimization methods (Newton's method, natural gradient) work cleanly with it. It's less common in practice than Huber but appears in gradient boosting libraries like XGBoost where smooth second derivatives are computationally useful.

Classification Losses

Classification means predicting which category something belongs to. The model outputs probabilities, and the loss measures how far those probabilities are from reality. This is where information theory enters the picture.

Binary Cross-Entropy (BCE)

Start with the simplest case: two classes, labeled 0 and 1. The model outputs a single probability p (via sigmoid), representing its confidence that the sample is class 1. Binary Cross-Entropy is:

BCE = -(1/n) × Σ[yᵢ × log(ŷᵢ) + (1 - yᵢ) × log(1 - ŷᵢ)]

This looks intimidating, but break it apart. When the true label y = 1, only the first term survives: -log(ŷ). If the model predicts ŷ = 0.99 (highly confident and correct), the loss is -log(0.99) ≈ 0.01 — tiny. If the model predicts ŷ = 0.01 (highly confident and wrong), the loss is -log(0.01) ≈ 4.6 — huge. When y = 0, the second term does the same thing in reverse. The key insight: BCE penalizes confident wrong answers astronomically. Being 99% sure of the wrong class is punished far more than being 51% sure of the wrong class.

The Information Theory Intuition

Cross-entropy comes from information theory, where it measures the average number of bits needed to encode data from one distribution using a code optimized for a different distribution. If you have the true distribution (labels) and the predicted distribution (model output), cross-entropy tells you how many "extra bits" you waste by using the model's distribution instead of the truth. A perfect model wastes zero extra bits — its cross-entropy equals the entropy of the true distribution. A terrible model wastes many bits. Training minimizes this waste.

Categorical Cross-Entropy

When you have more than two classes, the model outputs a probability distribution over C classes using softmax. Categorical Cross-Entropy generalizes BCE:

CCE = -(1/n) × Σᵢ Σⱼ yᵢⱼ × log(ŷᵢⱼ)

Here, yᵢⱼ is 1 if sample i belongs to class j, and 0 otherwise (one-hot encoding). Since only one yᵢⱼ is nonzero per sample, this simplifies to: for each sample, look at the predicted probability for the correct class and take -log of it. Predicted 90% on the correct class? Loss is 0.105. Predicted 1% on the correct class? Loss is 4.605. The model is heavily punished for assigning low probability to the correct answer.

Sparse Categorical Cross-Entropy

Sparse Categorical Cross-Entropy is mathematically identical to categorical cross-entropy. The only difference is the input format. Instead of one-hot encoded labels [0, 0, 1, 0, 0], you pass the integer class index: 2. This is a practical convenience when you have thousands of classes (imagine one-hot encoding ImageNet's 1,000 classes for every sample — that's a lot of wasted zeros). The loss computation is the same: look up the predicted probability at index y, take -log of it.

Focal Loss

Focal Loss was invented to solve a very specific problem: the extreme class imbalance in object detection. In a typical image, a detector might evaluate 100,000 candidate regions. Maybe 10 contain actual objects. The other 99,990 are background — and they're easy to classify correctly. Standard cross-entropy gives each sample equal weight, so the enormous number of easy negatives dominates the total loss. The model spends its gradient budget getting slightly better at easy cases instead of learning to detect rare objects.

Focal Loss adds a modulating factor to cross-entropy:

FL(pₜ) = -αₜ × (1 - pₜ)ᵞ × log(pₜ)

Here, pₜ is the predicted probability for the correct class. The key is (1 - pₜ)ᵞ. When the model is confident and correct (pₜ → 1), this factor approaches 0, so easy examples contribute almost nothing to the loss. When the model is wrong or uncertain (pₜ → 0), the factor approaches 1, so hard examples contribute fully. The parameter γ (gamma), typically set to 2, controls how aggressively easy examples are downweighted. The αₜ factor handles class frequency weighting.

The original RetinaNet paper (Lin et al., 2017) showed that focal loss allowed a simple one-stage detector to beat all previous two-stage detectors. The insight was that the loss function, not the architecture, was the bottleneck — class imbalance was drowning the useful signal in a sea of easy negatives.

Margin-Based Losses

Margin-based losses don't measure probability accuracy. They measure whether the model's score for the correct answer exceeds its score for the wrong answer by a sufficient margin. They're used in SVMs, ranking problems, and representation learning.

Hinge Loss

Hinge Loss is the loss function behind Support Vector Machines. For binary classification with labels y ∈ {-1, +1}:

Hinge(y, ŷ) = max(0, 1 - y × ŷ)

If the model predicts the right sign and the score exceeds the margin (y × ŷ ≥ 1), the loss is zero — the model isn't penalized for being "more right than necessary." If the prediction is on the wrong side or too close to the boundary (y × ŷ < 1), the loss grows linearly. This creates a flat loss surface away from the decision boundary, so only the hard examples near the boundary (the "support vectors") drive learning.

The non-differentiability at the hinge point (where y × ŷ = 1) is handled in practice by subgradient methods or smooth approximations.

Squared Hinge Loss

Squared Hinge Loss simply squares the hinge loss:

Squared Hinge(y, ŷ) = max(0, 1 - y × ŷ)²

This penalizes margin violations quadratically instead of linearly, making the loss more sensitive to large violations. The gradient is continuous at the hinge point, which often makes optimization smoother. In practice, squared hinge loss tends to produce slightly different decision boundaries than standard hinge — it's more aggressive about pushing misclassified points away from the boundary.

Triplet Loss

Triplet Loss is the workhorse of metric learning — learning embeddings where similar things are close and dissimilar things are far apart. Think face recognition: you want two photos of the same person to produce nearby vectors, and photos of different people to produce distant vectors.

The setup requires three samples per training step: an anchor (a), a positive (p, same class as anchor), and a negative (n, different class). The loss is:

Triplet(a, p, n) = max(0, ‖f(a) - f(p)‖² - ‖f(a) - f(n)‖² + margin)

In words: the distance from anchor to negative should exceed the distance from anchor to positive by at least the margin. If it does, the loss is zero. If it doesn't, the loss equals the shortfall. The margin prevents the trivial solution where all embeddings collapse to the same point.

The hard part of triplet loss is mining — selecting informative triplets. Easy triplets (where the negative is already far away) produce zero loss and zero gradient, wasting computation. Hard triplets (where the negative is closer than the positive) drive the most learning but can cause training instability. Semi-hard triplets (negative is farther than positive but still within the margin) tend to be the sweet spot.

Contrastive Loss

Contrastive Loss works with pairs instead of triplets. Given two samples and a label indicating whether they're from the same class (y = 0) or different classes (y = 1):

Contrastive(x₁, x₂, y) = (1 - y) × ½ × D² + y × ½ × max(0, margin - D)²

where D = ‖f(x₁) - f(x₂)‖ is the distance between embeddings. For same-class pairs, the loss penalizes large distances. For different-class pairs, the loss penalizes distances smaller than the margin. The effect: same-class items cluster together, different-class items are pushed at least margin distance apart.

PyTorch Loss Functions in Practice

Enough math. Let's see how these losses look in code. PyTorch provides all the major loss functions in torch.nn, and you can compose custom ones easily.

import torch
import torch.nn as nn

# ── Regression Losses ─────────────────────────────────────
y_true = torch.tensor([3.0, 5.0, 2.5, 8.0])
y_pred = torch.tensor([2.8, 5.5, 2.0, 15.0])  # last one is an outlier

mse = nn.MSELoss()(y_pred, y_true)
mae = nn.L1Loss()(y_pred, y_true)
huber = nn.HuberLoss(delta=1.0)(y_pred, y_true)

print(f"MSE:   {mse.item():.4f}")    # 12.5650 — outlier dominates
print(f"MAE:   {mae.item():.4f}")    #  1.9250 — outlier tamed
print(f"Huber: {huber.item():.4f}")  #  3.0825 — in between

# ── Binary Cross-Entropy ─────────────────────────────────
labels = torch.tensor([1.0, 0.0, 1.0, 0.0])
# Model outputs (before sigmoid)
logits = torch.tensor([2.0, -1.5, 0.5, -3.0])

bce_logits = nn.BCEWithLogitsLoss()(logits, labels)
print(f"
BCE (with logits): {bce_logits.item():.4f}")

# ── Categorical Cross-Entropy ────────────────────────────
# 3 samples, 5 classes
class_logits = torch.randn(3, 5)
class_targets = torch.tensor([2, 0, 4])  # sparse integer labels

ce = nn.CrossEntropyLoss()(class_logits, class_targets)
print(f"Cross-Entropy: {ce.item():.4f}")

# ── Focal Loss (custom implementation) ───────────────────
def focal_loss(logits, targets, gamma=2.0, alpha=0.25):
    """Focal loss for binary classification."""
    bce = nn.functional.binary_cross_entropy_with_logits(
        logits, targets, reduction='none'
    )
    pt = torch.exp(-bce)  # probability of correct class
    focal_weight = alpha * (1 - pt) ** gamma
    return (focal_weight * bce).mean()

fl = focal_loss(logits, labels)
print(f"Focal Loss:    {fl.item():.4f}")

# ── Triplet Loss ─────────────────────────────────────────
triplet_loss = nn.TripletMarginLoss(margin=1.0)

anchor   = torch.randn(4, 128)  # 4 samples, 128-dim embeddings
positive = anchor + 0.1 * torch.randn(4, 128)  # slightly perturbed
negative = torch.randn(4, 128)  # random different samples

tl = triplet_loss(anchor, positive, negative)
print(f"
Triplet Loss:  {tl.item():.4f}")

# ── Hinge Loss (for SVM-style classification) ────────────
hinge = nn.HingeEmbeddingLoss(margin=1.0)
distances = torch.tensor([0.2, 0.8, 1.5, 0.1])
# y = 1 means same class (want small distance), y = -1 means different
pair_labels = torch.tensor([1, -1, -1, 1])

hl = hinge(distances, pair_labels)
print(f"Hinge Loss:    {hl.item():.4f}")
Practical Tip — Use the Logits Variant

Always prefer BCEWithLogitsLoss over BCELoss, and CrossEntropyLoss (which includes softmax internally) over manually applying softmax then computing the loss. The "with logits" variants combine the sigmoid/softmax and log operations into a single numerically stable step using the log-sum-exp trick. Applying softmax and then log separately can produce NaN values due to floating-point precision issues.

Loss Function Properties

Not all loss functions are created equal. Three mathematical properties determine how well a loss function behaves during optimization.

Convexity

A loss function is convex if any line segment between two points on its surface lies above or on the surface. Intuitively, a convex loss has a single valley with no deceptive local minima. MSE is convex (it's a parabola). Cross-entropy is convex with respect to the model's output probabilities. Convex losses guarantee that gradient descent will find the global minimum — no matter where you start, you'll reach the bottom.

However, the total loss of a neural network is almost never convex, because the model itself (the function from weights to predictions) is non-linear and non-convex. A convex loss function composed with a non-convex model yields a non-convex optimization problem. So convexity of the loss is necessary but not sufficient for easy optimization. It matters most for linear models (logistic regression, SVMs) where the composition preserves convexity.

Differentiability

Differentiability means the loss has a well-defined gradient everywhere. This is essential for gradient-based optimization — backpropagation needs to compute ∂L/∂ŷ at every point. MSE is everywhere differentiable. MAE is not differentiable at zero (the V-shaped kink). Hinge loss is not differentiable at the hinge point. In practice, non-differentiable points are handled with subgradients (pick any value in the range of valid slopes at the kink) or smooth approximations. Huber loss was specifically designed to smooth the kink in MAE. Log-Cosh goes further and is twice differentiable everywhere.

Sensitivity to Outliers

This is the practical property that most affects model quality. Losses that grow superlinearly with error (MSE, cross-entropy with extreme predictions) are sensitive to outliers — a few bad data points can dominate the entire loss and warp the model. Losses that grow linearly (MAE, Huber for large errors) or sublinearly are robust to outliers.

There's always a tradeoff. Outlier-sensitive losses converge faster on clean data because they provide strong gradients for large errors. Outlier-robust losses are slower to converge but give better results on messy, real-world data. If you control your data quality (clean labels, no measurement errors), MSE is fine. If you're working with noisy real-world data (user-generated labels, sensor measurements), consider Huber or MAE.

Loss vs Cost Function Distinction

People use "loss function" and "cost function" interchangeably. They're related but technically different things, and the distinction matters when you're reading papers or debugging training.

Per-Sample Loss

The loss function operates on a single sample. It takes one prediction and one true value, and returns one number:

ℓ(yᵢ, ŷᵢ) = (yᵢ - ŷᵢ)²     # loss for sample i

This is the fundamental building block. Every loss function we've discussed above is defined at the per-sample level.

Batch-Aggregated Cost

The cost function (also called the objective function) aggregates the loss across your training set or mini-batch:

J(θ) = (1/n) × Σᵢ ℓ(yᵢ, f(xᵢ; θ))

This is what the optimizer actually minimizes. The aggregation is typically a mean (makes the cost independent of batch size) but can also be a sum. The choice between mean and sum affects the effective learning rate — sum-reduction with a batch size of 32 produces gradients 32× larger than mean-reduction, so you'd need to scale the learning rate accordingly. PyTorch's loss functions default to reduction='mean', and you should usually leave it that way.

Regularization Terms in Cost

The cost function often includes additional terms beyond the aggregated loss. Regularization adds a penalty on the model's weights to prevent overfitting:

J(θ) = (1/n) × Σᵢ ℓ(yᵢ, f(xᵢ; θ)) + λ × R(θ)

Common regularizers include L2 (weight decay, penalizes ||θ||²) and L1 (sparsity, penalizes ||θ||₁). These aren't part of the loss function per se — they don't depend on the predictions or the data at all, only on the model's parameters. But they're part of the cost function because the optimizer minimizes the total cost, including these terms.

In PyTorch, regularization is often implemented in the optimizer (e.g., weight_decay parameter in Adam) rather than added to the loss explicitly. This is a practical detail but it means the loss printed during training may not include the regularization term. If you're comparing training loss to validation loss for overfitting diagnosis, make sure you know what's included in each.

Choosing a Loss Function — Decision Guide

Regression, clean data: MSE — fast convergence, well-behaved gradients.
Regression, noisy data: Huber or MAE — outlier robustness.
Binary classification: BCEWithLogitsLoss — numerically stable sigmoid + cross-entropy.
Multi-class classification: CrossEntropyLoss — handles softmax + log internally.
Extreme class imbalance: Focal Loss — downweights easy negatives.
Learning embeddings/similarity: Triplet or Contrastive Loss — push and pull in embedding space.
SVM-style max-margin: Hinge Loss — penalizes only violations of the margin.
