Backpropagation

Chapter 7: Deep Learning Foundations 7 subtopics ~3500 words
TL;DR

Backpropagation is the algorithm that makes neural networks learnable. It runs the input forward to get a prediction, measures how wrong that prediction is, then works backward through the network asking each parameter: "how much did you contribute to the error?" It uses the chain rule from calculus to efficiently compute gradients for every weight in the network in a single backward sweep. Without backprop, we'd have to wiggle each weight individually to estimate its effect — impossibly slow for networks with millions of parameters. Every modern deep learning framework (PyTorch, TensorFlow, JAX) implements this as automatic differentiation on a computational graph.

A Concrete Example: Forward and Backward Through a Tiny Network

Let's learn backpropagation by doing it. We'll build the smallest possible network that still teaches the real thing: one input, one hidden neuron, one output neuron. No batches, no bias terms, no activation functions yet — just raw arithmetic so you can see every moving part.

Here's our setup. We have a single training example: input x = 2.0, target y = 1.0. Our network has two weights: w₁ = 0.5 (connecting input to hidden) and w₂ = 0.8 (connecting hidden to output). We'll use mean squared error as our loss.

The Forward Pass

The forward pass is the simple part. Data flows left to right, and each layer computes its output from its input. That's it. No mystery.

Step 1 — Input propagation: The input value x = 2.0 enters the network.

Step 2 — Layer-wise computation: The hidden neuron multiplies the input by its weight. h = x × w₁ = 2.0 × 0.5 = 1.0. Then the output neuron multiplies the hidden value by its weight. ŷ = h × w₂ = 1.0 × 0.8 = 0.8.

Step 3 — Loss computation: We compare our prediction to the target. L = (ŷ − y)² = (0.8 − 1.0)² = (−0.2)² = 0.04. Our network predicted 0.8, but the answer was 1.0. The loss is 0.04 — not zero, so the network is wrong and needs to adjust.

# Forward pass — spelled out
x  = 2.0     # input
y  = 1.0     # target
w1 = 0.5     # weight: input → hidden
w2 = 0.8     # weight: hidden → output

h    = x * w1          # hidden = 2.0 * 0.5 = 1.0
y_hat = h * w2          # output = 1.0 * 0.8 = 0.8
loss = (y_hat - y) ** 2 # loss   = (-0.2)^2  = 0.04

The forward pass gave us the loss. Now comes the hard question: which weight should change, and by how much? That's what the backward pass answers.

The Backward Pass

The backward pass works right to left, computing how much the loss changes when each parameter changes. These quantities are called gradients — they tell you the direction and rate of steepest increase of the loss.

We need two gradients: ∂L/∂w₂ (how does the loss change when we wiggle w₂?) and ∂L/∂w₁ (same for w₁). Let's compute them by hand.

Start at the loss. L = (ŷ − y)². The derivative with respect to ŷ is: ∂L/∂ŷ = 2(ŷ − y) = 2(0.8 − 1.0) = −0.4. This is the output layer gradient — it tells us the loss decreases if ŷ increases (because our prediction was too low).

Now move one step left. ŷ = h × w₂. Using basic calculus: ∂ŷ/∂w₂ = h = 1.0. So the parameter gradient for w₂ is: ∂L/∂w₂ = ∂L/∂ŷ × ∂ŷ/∂w₂ = (−0.4) × 1.0 = −0.4. The negative sign means increasing w₂ will decrease the loss. Good — that's the direction we want to go.

Keep going left to find the hidden layer gradient. ∂ŷ/∂h = w₂ = 0.8. So ∂L/∂h = ∂L/∂ŷ × ∂ŷ/∂h = (−0.4) × 0.8 = −0.32.

One more step. h = x × w₁, so ∂h/∂w₁ = x = 2.0. Therefore ∂L/∂w₁ = ∂L/∂h × ∂h/∂w₁ = (−0.32) × 2.0 = −0.64.

# Backward pass — by hand
dL_dy_hat = 2 * (y_hat - y)           # = -0.4
dL_dw2    = dL_dy_hat * h             # = -0.4 * 1.0 = -0.4
dL_dh     = dL_dy_hat * w2            # = -0.4 * 0.8 = -0.32
dL_dw1    = dL_dh * x                 # = -0.32 * 2.0 = -0.64

# Update weights (learning rate = 0.1)
lr = 0.1
w1 = w1 - lr * dL_dw1   # 0.5 - 0.1*(-0.64) = 0.564
w2 = w2 - lr * dL_dw2   # 0.8 - 0.1*(-0.4)  = 0.84

After one update: w₁ went from 0.5 to 0.564, and w₂ went from 0.8 to 0.84. Both increased, pushing the prediction closer to 1.0. If you run the forward pass again with the new weights: h = 2.0 × 0.564 = 1.128, ŷ = 1.128 × 0.84 = 0.948. The prediction jumped from 0.8 to 0.948. The loss dropped from 0.04 to 0.0027. One step of backprop made the network six times more accurate.

That's the entire algorithm. Forward, compute loss, backward, update. Repeat thousands of times until the loss is small enough.

The Chain Rule: Why Backprop Works

You just saw backprop in action. But why did we keep multiplying gradients together as we moved backward? The answer is the chain rule from calculus — the single mathematical idea that makes neural network training possible.

The Assembly Line Analogy

Imagine an assembly line in a factory. Station A cuts metal sheets to a certain size. Station B bends those sheets. Station C welds the bent sheets into a box. If the final box is 2% too wide, you need to figure out which station to adjust and by how much.

You can't just look at Station C in isolation. The width of the final box depends on Station C's welding, which depends on Station B's bending angle, which depends on Station A's cut size. A small change at Station A ripples through every downstream station. To figure out how a change at Station A affects the final product, you multiply the effects at each stage together.

That's exactly the chain rule. If y = f(g(h(x))), then dy/dx = f'(g(h(x))) × g'(h(x)) × h'(x). You multiply the local rates of change at each stage to get the total effect end to end.

Local and Upstream Gradients

In backpropagation, each node in the network computes two things during the backward pass. The local gradient is the derivative of that node's output with respect to its own inputs — how its little piece of the computation responds to small changes. The upstream gradient is the gradient flowing in from the layers above it — how much the final loss cares about this node's output.

The node multiplies these two together and passes the result backward. That's the whole job of each node during backprop: receive the upstream gradient, multiply by the local gradient, pass it downstream. It's strikingly simple and local. No node needs to know about the overall network architecture. Each one just handles its own local multiplication.

Look back at our example. When we computed ∂L/∂w₁, we multiplied ∂L/∂ŷ (upstream: how the loss responds to the output) × ∂ŷ/∂h (local to the output node) × ∂h/∂w₁ (local to the hidden node). Three local multiplications, chained together, gave us the gradient for a weight two layers deep.

Composite Functions and Deep Networks

A deep network is just a deeply nested composite function: L(f_n(f_{n-1}(... f_1(x) ...))). The chain rule scales naturally. For a network with 100 layers, you just chain 100 local gradient multiplications. The computation cost is proportional to the number of layers — linear, not exponential. This is why backprop is efficient enough to train networks with billions of parameters.

Backward Pass: The Details

Gradient Accumulation

Our tiny example had a clean chain — each node fed into exactly one other node. Real networks are messier. A single hidden neuron's output might feed into multiple neurons in the next layer. When that happens, gradients from all downstream paths add up. This is called gradient accumulation.

If neuron h feeds into both output neurons o₁ and o₂, then ∂L/∂h = ∂L/∂o₁ × ∂o₁/∂h + ∂L/∂o₂ × ∂o₂/∂h. You sum the gradients from every path. This follows from the multivariate chain rule, and frameworks handle it automatically. But it's important to know it's happening — it explains why you must zero your gradients before each training step in PyTorch (otherwise gradients from previous batches accumulate incorrectly).

Gradient Verification: Am I Doing This Right?

Gradient checking is a debugging technique that verifies your analytical gradients are correct. The idea is simple: if your calculus-derived gradient says ∂L/∂w = −0.4, you can verify it numerically by actually wiggling w by a tiny amount ε and measuring what happens to the loss.

The finite difference method computes the numerical gradient as: (L(w + ε) − L(w − ε)) / (2ε). This centered difference gives a more accurate estimate than a one-sided difference. You compare the numerical gradient to the analytical gradient. If they agree to about 7 decimal places (relative error below 10⁻⁷), your implementation is correct.

def gradient_check(f, w, eps=1e-5):
    """Verify analytical gradient with finite differences."""
    numerical_grads = []
    for i in range(len(w)):
        w_plus = w.copy(); w_plus[i] += eps
        w_minus = w.copy(); w_minus[i] -= eps
        numerical_grad = (f(w_plus) - f(w_minus)) / (2 * eps)
        numerical_grads.append(numerical_grad)
    return numerical_grads

# Example: verify our manual gradients
import numpy as np
def network_loss(weights):
    w1, w2 = weights
    h = 2.0 * w1
    y_hat = h * w2
    return (y_hat - 1.0) ** 2

numerical = gradient_check(network_loss, np.array([0.5, 0.8]))
analytical = [-0.64, -0.4]
for n, a in zip(numerical, analytical):
    rel_error = abs(n - a) / (abs(n) + abs(a) + 1e-8)
    print(f"Numerical: {n:.6f}, Analytical: {a:.6f}, "
          f"Relative error: {rel_error:.2e}")
# Numerical: -0.640000, Analytical: -0.640000, Relative error: 0.00e+00
# Numerical: -0.400000, Analytical: -0.400000, Relative error: 0.00e+00
⚠️ Practical Note

Gradient checking is slow — it requires two forward passes per parameter. Use it only during debugging on small networks, never in production training. A relative error threshold of 10⁻⁷ is standard; anything above 10⁻⁵ suggests a bug, and above 10⁻³ means something is definitely wrong.

Computational Graphs

So far we've been computing gradients by hand. That's fine for two weights. It's completely unworkable for GPT-4's 1.8 trillion parameters. We need a systematic way to track computations and automate differentiation. Enter the computational graph.

Nodes and Edges

A computational graph breaks down a computation into elementary operations. Each node represents either a variable (input, parameter, intermediate result) or an operation (addition, multiplication, applying an activation function). Each edge represents data flow — the output of one node feeding into another as input.

Our tiny network's computational graph looks like this:

x ──→ [× w₁] ──→ h ──→ [× w₂] ──→ ŷ ──→ [loss] ──→ L
       ↑                  ↑                  ↑
       w₁                 w₂                 y (target)

During the forward pass, data flows left to right. During the backward pass, gradients flow right to left. Each node, when asked for a gradient, only needs to know its local derivative and the gradient coming from downstream. This is why backprop is so elegant — it turns a global optimization problem into a series of local multiplications.

Forward Mode vs Reverse Mode Differentiation

There are actually two ways to propagate derivatives through a computational graph. Forward mode differentiation starts at the inputs and pushes derivatives forward through the graph. Reverse mode differentiation starts at the output and pulls gradients backward. Backpropagation is reverse mode.

Why does the direction matter? It's about efficiency. Forward mode computes the derivative of all outputs with respect to one input in a single pass. Reverse mode computes the derivative of one output with respect to all inputs in a single pass.

In a neural network, you have millions of inputs (parameters) but a single scalar output (the loss). Reverse mode gives you all the gradients you need in one backward sweep. Forward mode would require a separate sweep for each parameter — millions of passes instead of one. That's why neural networks use reverse mode exclusively. It's not a minor optimization. It's the difference between "trainable in hours" and "trainable in millennia."

Automatic Differentiation

Automatic differentiation (autodiff) is the technique that computational graphs enable. It's neither symbolic differentiation (which manipulates mathematical expressions algebraically) nor numerical differentiation (which approximates derivatives with finite differences). Autodiff computes exact derivatives by applying the chain rule to elementary operations recorded in the graph.

Here's autodiff in action with PyTorch:

import torch

# Define inputs and parameters with gradient tracking
x = torch.tensor(2.0)
y = torch.tensor(1.0)
w1 = torch.tensor(0.5, requires_grad=True)
w2 = torch.tensor(0.8, requires_grad=True)

# Forward pass — PyTorch builds the graph as you compute
h = x * w1
y_hat = h * w2
loss = (y_hat - y) ** 2

# Backward pass — one call computes ALL gradients
loss.backward()

print(f"dL/dw1 = {w1.grad:.4f}")  # -0.6400 ✓
print(f"dL/dw2 = {w2.grad:.4f}")  # -0.4000 ✓

# Gradient checking — PyTorch has it built in
from torch.autograd import gradcheck
def func(w1, w2):
    return ((x * w1 * w2 - y) ** 2).unsqueeze(0)

w1_check = torch.tensor(0.5, requires_grad=True, dtype=torch.float64)
w2_check = torch.tensor(0.8, requires_grad=True, dtype=torch.float64)
x_64 = torch.tensor(2.0, dtype=torch.float64)
y_64 = torch.tensor(1.0, dtype=torch.float64)

test = gradcheck(
    lambda w1, w2: ((x_64 * w1 * w2 - y_64) ** 2).unsqueeze(0),
    (w1_check, w2_check), eps=1e-6
)
print(f"Gradient check passed: {test}")  # True

Notice what happened. We wrote the forward pass as normal Python arithmetic. PyTorch silently built the computational graph behind the scenes. One call to .backward() traversed that graph in reverse and filled in the .grad attribute for every tensor that had requires_grad=True. We never wrote a single derivative by hand. That's the power of autodiff.

Static vs Dynamic Computation Graphs

This is where the deep learning frameworks diverge — and where a design decision shaped the entire research landscape.

A static graph (define-and-run) requires you to declare the entire computation graph upfront, before any data flows through it. TensorFlow 1.x worked this way. You'd write a graph definition in Python, then "compile" it, then feed data through. The graph was fixed — if you wanted an if-statement that depended on the data, you had to use TensorFlow's special control flow operations, not Python's.

# TensorFlow 1.x style (static graph) — simplified illustration
# import tensorflow as tf
# 
# # Step 1: DEFINE the graph (no computation happens yet)
# x = tf.placeholder(tf.float32)
# w = tf.Variable(0.5)
# y_hat = x * w
# loss = (y_hat - 1.0) ** 2
# grad = tf.gradients(loss, w)
#
# # Step 2: RUN the graph in a session
# with tf.Session() as sess:
#     sess.run(tf.global_variables_initializer())
#     result = sess.run(grad, feed_dict={x: 2.0})
# 
# The graph is fixed. If you want different logic for different
# inputs, you need tf.cond, tf.while_loop — not Python if/while.

A dynamic graph (define-by-run) builds the graph on the fly as your Python code executes. PyTorch pioneered this approach. Every operation you write is immediately executed, and the graph is recorded as a side effect. Want a Python if-statement? Just write one. Want a for-loop whose length depends on the input? Go ahead.

# PyTorch style (dynamic graph) — this is real, runnable code
import torch

def flexible_network(x, w1, w2, use_skip=False):
    """Normal Python control flow — the graph adapts each call."""
    h = x * w1
    if use_skip:           # Plain Python if — works perfectly
        out = h + x * w2   # Skip connection path
    else:
        out = h * w2       # Normal path
    return out

x = torch.tensor(2.0)
w1 = torch.tensor(0.5, requires_grad=True)
w2 = torch.tensor(0.8, requires_grad=True)

# Different graph each call — no problem
out1 = flexible_network(x, w1, w2, use_skip=False)
out2 = flexible_network(x, w1, w2, use_skip=True)

Why did dynamic graphs win for research? Three reasons. First, debugging: you can set a breakpoint on any line and inspect tensors with print(). With static graphs, the Python code doesn't actually compute anything — it just builds a graph description, so print statements show symbolic objects, not values. Second, flexibility: many architectures (recursive neural networks, attention mechanisms with variable-length inputs, reinforcement learning agents) need control flow that depends on the data. Dynamic graphs handle this naturally. Third, iteration speed: researchers could prototype faster when the language felt like normal Python instead of a graph-definition DSL.

ℹ️ The Convergence

The story doesn't end with "dynamic won." TensorFlow 2.0 adopted eager execution by default (dynamic graphs), while PyTorch added torch.compile() and torch.jit.trace() for static graph optimizations. JAX takes a middle path with function transformations. The industry converged: dynamic for development, optional static compilation for production speed. But PyTorch remains the default for research precisely because it got this right first.

Backpropagation Through Time (BPTT)

Everything so far assumed feedforward networks — data flows in one direction. But what about recurrent neural networks (RNNs), where the output at time step t feeds back as input to time step t+1? How do you backpropagate through a loop?

The Unrolled Graph

The trick is unrolling. You take the recurrent loop and lay it out as a sequence of identical layers, one per time step. An RNN processing a 50-word sentence becomes a feedforward network with 50 copies of the same layer, each sharing the same weights. Now standard backpropagation works — you just apply the chain rule across all 50 steps. This is backpropagation through time (BPTT).

# Conceptual unrolling of an RNN
# Original: h_t = tanh(W_hh @ h_{t-1} + W_xh @ x_t)
# 
# Unrolled for 4 time steps:
#
# x₁ → [W_xh] → (+) → [tanh] → h₁
#                 ↑                ↓ [W_hh]
# x₂ → [W_xh] → (+) → [tanh] → h₂
#                 ↑                ↓ [W_hh]
# x₃ → [W_xh] → (+) → [tanh] → h₃
#                 ↑                ↓ [W_hh]
# x₄ → [W_xh] → (+) → [tanh] → h₄ → loss
#
# Backward pass flows from loss through ALL time steps

The Long-Range Dependency Problem

Here's the catch. When you backpropagate through 50 time steps, you're multiplying 50 gradients together (one per step). If each local gradient is slightly less than 1.0, say 0.9, then after 50 multiplications the gradient is 0.9⁵⁰ ≈ 0.005. It has virtually vanished. If each is slightly above 1.0, say 1.1, then 1.1⁵⁰ ≈ 117 — the gradient has exploded.

This is the vanishing/exploding gradient problem for sequences. It means standard RNNs struggle to learn dependencies between events that are far apart in a sequence. The word at position 3 can't easily influence learning at position 50. This problem motivated architectures like LSTMs and GRUs, which use gating mechanisms to control gradient flow — effectively providing a "highway" for gradients to travel long distances without being multiplied away.

Truncated BPTT

For very long sequences, full BPTT is impractical — you'd need to store the hidden state at every time step, consuming enormous memory. Truncated BPTT is the pragmatic compromise. Instead of unrolling the entire sequence, you unroll only k steps at a time, run backprop through those k steps, update the weights, then continue from where you left off. This trades gradient accuracy for memory efficiency. You lose the ability to learn dependencies longer than k steps, but the model remains trainable on sequences of any length. The tradeoff — sequence length vs memory — is one you'll tune on every recurrent model.

Jacobian and Hessian in Backprop

Our examples used scalar values — one loss, one gradient per weight. Real networks have vector-valued layers. When a layer maps a vector of inputs to a vector of outputs, the complete picture of "how each output changes with respect to each input" is a matrix, not a single number.

The Jacobian Matrix

The Jacobian matrix generalizes the derivative to vector-valued functions. If a layer maps an n-dimensional input to an m-dimensional output, the Jacobian is an m × n matrix where entry (i, j) is ∂output_i/∂input_j. During backpropagation, the gradient flowing through a layer is computed by multiplying the upstream gradient vector by the Jacobian. In our scalar example, the "Jacobian" was just a single number — the local gradient. In real networks, it's a matrix multiply at each layer.

The Hessian Matrix

The Hessian matrix contains the second derivatives — it describes the curvature of the loss landscape. Entry (i, j) is ∂²L/∂wᵢ∂wⱼ. While the gradient tells you the slope (which direction is downhill), the Hessian tells you how the slope itself is changing. Is the loss surface a gentle bowl or a steep ravine? The Hessian answers this.

Second-Order Optimization

Newton's method uses the Hessian to take smarter optimization steps. Instead of just following the gradient downhill (first-order), it considers the curvature to jump directly toward the minimum: w_new = w − H⁻¹ ∇L. In a perfectly quadratic loss surface, Newton's method converges in a single step.

The catch? For a model with n parameters, the Hessian is an n × n matrix. GPT-3 has 175 billion parameters. Storing a 175B × 175B matrix is physically impossible — it would require more atoms than exist in the observable universe to store in bits. That's why we almost never compute the full Hessian in deep learning. Instead, we use approximations.

Hessian-Vector Products and the Fisher Information Matrix

Hessian-vector products (HVPs) compute H·v — the Hessian times a vector — without ever forming the full matrix. Autodiff can compute this efficiently using a trick: differentiate the gradient (which is already a function of w) in the direction of v. This takes about the same time as one backward pass.

import torch

# Efficient Hessian-vector product via autodiff
def hvp(loss_fn, params, v):
    """Compute Hessian-vector product without forming the Hessian."""
    grads = torch.autograd.grad(loss_fn, params, create_graph=True)
    flat_grad = torch.cat([g.flatten() for g in grads])
    flat_v = torch.cat([vi.flatten() for vi in v])
    grad_dot_v = flat_grad @ flat_v
    hvp_result = torch.autograd.grad(grad_dot_v, params)
    return hvp_result

The Fisher Information Matrix is a close cousin of the Hessian that appears in natural gradient methods. It measures how sensitive the model's output distribution is to parameter changes. For many loss functions, the Fisher and the Hessian are equivalent (or approximately so). Natural gradient descent uses the Fisher to precondition updates, effectively accounting for the geometry of the parameter space. In practice, even the Fisher is approximated — methods like K-FAC compute block-diagonal approximations that are tractable.

Differentiation Methods Compared

We've been using automatic differentiation, but it's worth understanding the alternatives and why autodiff won.

Symbolic Differentiation

Symbolic differentiation applies the rules of calculus algebraically. Give it f(x) = x² sin(x) and it returns f'(x) = 2x sin(x) + x² cos(x). Tools like SymPy and Mathematica do this. The result is exact and interpretable. The problem? Expressions explode in size. Differentiating a deep network symbolically produces expressions with an exponentially growing number of terms — most of which contain common subexpressions that could be shared but aren't. For a network with millions of operations, symbolic differentiation is hopelessly slow and memory-hungry.

Numerical Differentiation

Numerical differentiation approximates the gradient using the finite difference method we saw earlier: (f(x+ε) − f(x−ε)) / 2ε. It's simple and works for any function — you don't need to know the function's formula, just be able to evaluate it. But it has two fatal flaws for neural networks. First, it's slow: you need two function evaluations per parameter, so a model with n parameters requires 2n forward passes. Second, it's approximate: choosing ε is a balancing act between truncation error (ε too large) and floating-point rounding error (ε too small). For gradient checking it's fine. For training, it's unusable.

Automatic Differentiation

Automatic differentiation is the sweet spot. It computes exact derivatives (up to floating-point precision). It's efficient — reverse-mode autodiff computes the gradient of a scalar function with respect to all inputs in time proportional to just one forward pass (roughly 2-3x the cost). And it works on arbitrary code, including loops and branches.

Method Exact? Cost (n params) Handles Code? Used For
Symbolic Yes Expression swell Math formulas only Small analytic models
Numerical Approximate O(n) forward passes Any function Gradient checking
Autodiff (forward) Yes O(n) passes Any code Few-input functions
Autodiff (reverse) Yes O(1) backward pass Any code Neural network training

The reason every deep learning framework is built around reverse-mode autodiff is now clear. It gives you exact gradients, for arbitrary code, at the cost of a single backward pass regardless of how many parameters you have. That's as good as it gets.

Putting It All Together in PyTorch

Let's end with a complete example that ties together everything we've discussed — forward pass, backward pass, gradient checking, and the computational graph — in a realistic (but small) training loop:

import torch
import torch.nn as nn

# A real 2-layer network
model = nn.Sequential(
    nn.Linear(3, 4),   # layer 1: 3 inputs → 4 hidden
    nn.ReLU(),
    nn.Linear(4, 1)    # layer 2: 4 hidden → 1 output
)

# Synthetic data
torch.manual_seed(42)
X = torch.randn(16, 3)      # 16 samples, 3 features
y = torch.randn(16, 1)      # 16 targets

optimizer = torch.optim.SGD(model.parameters(), lr=0.01)
loss_fn = nn.MSELoss()

# Training loop — the forward-backward cycle in practice
for epoch in range(5):
    # ---- Forward pass ----
    y_hat = model(X)              # data flows through all layers
    loss = loss_fn(y_hat, y)      # scalar loss computed
    
    # ---- Backward pass ----
    optimizer.zero_grad()          # clear old gradients!
    loss.backward()                # compute all gradients via autodiff
    
    # Inspect gradients (they exist on every parameter)
    for name, p in model.named_parameters():
        print(f"  {name:20s} grad norm: {p.grad.norm():.4f}")
    
    # ---- Update ----
    optimizer.step()               # w = w - lr * grad
    
    print(f"Epoch {epoch}: loss = {loss.item():.4f}
")

# Verify: gradients should be zero after zero_grad
optimizer.zero_grad()
for name, p in model.named_parameters():
    assert p.grad is None or p.grad.abs().max() == 0
ℹ️ The Three-Line Core

Every PyTorch training loop has the same three-line heart: optimizer.zero_grad() to reset, loss.backward() to compute gradients, optimizer.step() to update weights. Everything else — data loading, model architecture, logging — is scaffolding around these three lines. Master them and you understand the entire training process.

Backpropagation isn't magic. It's the chain rule applied systematically to a computational graph, with gradients flowing backward from the loss to every parameter. The math is first-year calculus. The engineering — building frameworks that do this automatically, efficiently, for models with billions of parameters — is what took decades. But the core idea fits in a single sentence: to learn, figure out what went wrong, then trace back through each step to find who's responsible and by how much.
