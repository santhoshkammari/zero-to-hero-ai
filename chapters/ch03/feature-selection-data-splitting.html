Feature Selection & Data Splitting

Shipped fraud detection model. Validation looked brilliant, production no better than coin flip. Culprit: careless split leaked answer; features smuggled target into input.

Feature selection chooses input variables model sees. Data splitting partitions dataset for training, tuning, final evaluation. Together, they decide whether reported performance means anything. Get one wrong, build on sand.

The Kitchen Pantry Problem

Imagine cooking dish, someone dumps whole grocery store on counter. Every spice, every grain, every obscure sauce. More ingredients do not make better cook. They make you slower, confused, more likely to ruin dinner with ingredient that does not belong.

Features work same way. You engineered 200 — transaction amount, time since last purchase, merchant category, IP geolocation, user age, 195 more. Many are noise. Some are near-duplicates. Few truly predict. Feature selection finds helpers, throws rest away.

This matters more than most realize. Curse of dimensionality: more features make feature space volume explode exponentially. Data points that felt dense in five dimensions become isolated specks in 500. Distance metrics break down — "nearest" and "farthest" neighbors end up almost equally far. This is Hughes phenomenon. Past some point, more features hurt model.

First time I read that more features could make things worse, I did not believe it. Felt like saying more information is bad. Math says otherwise. With 20 data points and 200 features, model lacks examples for general pattern. It memorizes. It overfits. Then it fails on new data.

Kitchen pantry analogy stays with us. First throw out stale ingredients. Then remove duplicates. Then taste-test rest to find ones that matter.

Variance Threshold — Throwing Out the Dead Weight

First question for any feature: does it vary? Feature with same value in 99.99% of rows carries almost no information. Like spice jar full of air — takes counter space, adds nothing.

Consider tiny dataset of six customers in fraud system. One column is "country" and every row says "US." Feature has zero variance. It cannot distinguish fraud from legitimate transactions because value is same for everyone.

from sklearn.feature_selection import VarianceThreshold

selector = VarianceThreshold(threshold=0.01)
X_filtered = selector.fit_transform(X)
kept = selector.get_support().sum()
print(f"Kept {kept} of {X.shape[1]} features")

Threshold 0.01 removes any feature with variance below 1%. For binary features, variance 0.01 means one value appears 99.5% of time, other 0.5%. Effectively dead.

This is fast — milliseconds, even on large datasets. Pantry version: open jars, throw out empty or expired ones. Great selection? No. Good counter cleanup before real work? Yes.

Limitation obvious: variance says nothing about predictive value. Feature with high variance can still be pure noise. Need something smarter.

Correlation — Catching the Copycats

Two features moving in lockstep carry same information. If "transaction_amount_usd" and "transaction_amount_cents" both exist — one is 100× other — keeping both is redundant. Model gains nothing from second copy. Some models, like linear regression, get real numerical problems from multicollinearity.

Pearson correlation measures linear relationship between two variables on scale from −1 to +1. Correlation 0.95 means two features are nearly identical in linear signal.

import numpy as np
import pandas as pd

corr_matrix = pd.DataFrame(X_filtered).corr().abs()
upper_triangle = corr_matrix.where(
    np.triu(np.ones(corr_matrix.shape), k=1).astype(bool)
)
redundant = [col for col in upper_triangle.columns
             if any(upper_triangle[col] > 0.90)]

We scan upper triangle of correlation matrix to avoid self-comparisons and double-counting, then flag anything above 0.90. From each highly correlated pair, drop one — usually feature with lower individual correlation to target.

Pantry version: notice three bottles of olive oil — regular, extra virgin, "premium" that tastes same as extra virgin. Keep one. Toss duplicates.

But Pearson fails here. Pearson measures only linear relationships. If feature X and target Y relate in curved, U-shaped, or other nonlinear way, Pearson can return near zero — as if no relationship exists. For "distance from city center" predicting house prices, prices may be high in city core and wealthy suburbs but low between. Pearson shrugs: "no correlation." Omission lie.

Mutual Information — The Nonlinear Detective

Mutual information does not care about relationship shape. It asks more basic question: how much does knowing one variable reduce uncertainty about other? If knowing customer's "average transaction amount" strongly helps predict fraud — linear, quadratic, weird staircase, anything — mutual information catches it.

Value ranges from 0 (complete independence — knowing one tells you nothing about other) upward toward infinity (knowing one perfectly determines other). Unlike Pearson's tidy −1 to +1, MI values are not directly comparable across datasets. Good for ranking features inside one dataset.

from sklearn.feature_selection import mutual_info_classif

mi_scores = mutual_info_classif(X, y, random_state=42)
ranking = np.argsort(mi_scores)[::-1]

for i in ranking[:10]:
    print(f"Feature {i}: MI = {mi_scores[i]:.3f}")

I am still building intuition for MI estimation under hood — it partitions feature space and estimates probability densities, which gets computationally hairy with many features. But as ranking tool, it is remarkably reliable. Think metal detector that finds only iron (Pearson) versus one that finds any metal (MI).

These three techniques — variance threshold, correlation filtering, mutual information ranking — are filter methods. Name fits: they filter with statistics alone, without training model. Fast. Model-agnostic. Good first pass. But they judge each feature alone, or in pairs for correlation. They can miss features weak alone but strong together. For that, need different approach.

Wrapper Methods — Search with Feedback

What if, instead of scoring features with statistics, we train model, check performance, then ask: "What happens if I remove this feature?" Core wrapper idea. Wrap model inside search loop — try subsets, measure performance, converge on best set.

Recursive Feature Elimination (RFE) is most common wrapper. It trains model on all features, ranks them by importance — coefficients for linear models, impurity reduction for trees — removes least important one, repeats. Cooking competition version: each round, weakest ingredient goes home.

from sklearn.feature_selection import RFECV
from sklearn.ensemble import RandomForestClassifier

rfe = RFECV(
    estimator=RandomForestClassifier(n_estimators=100, random_state=42),
    step=1, cv=5, scoring="f1", min_features_to_select=5
)
rfe.fit(X_train, y_train)
print(f"Optimal feature count: {rfe.n_features_}")

RFECV adds cross-validation, so you get reliable performance estimate at each feature count. Optimal number is where CV score peaks.

Cost is real. With 200 features and 5-fold CV, you train about 1,000 models. If each takes minute, compute bill is 16 hours. Bad for daily production pipeline. Fine for one-time feature audit on new dataset.

Subtle instability too. RFE can produce different feature sets from slightly different data samples, especially when features are correlated. If features A and B carry similar information, one run may keep A and drop B, another may do opposite. Stability selection helps: run selection on many random subsamples, keep only features that survive consistently. Compute cost grows again.

Pantry version: instead of eyeballing ingredients, cook 1,000 versions of dish, each time leaving one ingredient out, then see which removals change nothing. Thorough. Exhausting.

Embedded Methods — Selection Baked Into Training

What if model itself decides which features matter while learning? That is embedded methods. Feature selection is not separate step — it is built into training.

Tree-based models like Random Forest and XGBoost naturally produce feature importance scores. Every time tree splits on feature, it records how much split reduced prediction error. Features used often, high in tree, get high importance. Features trees rarely use get low importance.

from sklearn.ensemble import RandomForestClassifier

rf = RandomForestClassifier(n_estimators=200, random_state=42)
rf.fit(X_train, y_train)

importances = rf.feature_importances_
top_10 = np.argsort(importances)[-10:][::-1]
for idx in top_10:
    print(f"Feature {idx}: importance = {importances[idx]:.4f}")

This is computationally free — you already planned to train model, and importance scores come as byproduct. Limitation: scores are specific to model that made them. Feature Random Forest ignores may matter a lot to neural network with different inductive biases.

The L1 Geometry That Makes Lasso Special

L1 regularization, also called Lasso, does what L2 (Ridge) cannot: it drives feature coefficients to exactly zero. Not close. Exactly zero. Feature is gone. That makes Lasso real feature selector.

Reason: geometry. Hard to visualize beyond two dimensions, but 2D picture is enough. You want best coefficients for linear model. Without regularization, solution lands at unconstrained optimum. Add constraint: sum of absolute coefficient values must stay below budget.

In 2D, L1 constraint region is diamond — square rotated 45 degrees. L2 constraint region is circle. Loss function makes elliptical contours around unconstrained optimum. Expand contours until they touch constraint region. With L2 circle, touch point can be anywhere on smooth boundary — almost never axis, so both coefficients stay nonzero. With L1 diamond, contours usually hit corner first, and corners sit on axes where one coefficient is exactly zero.

Whole trick. Sharp corners of L1 diamond act like feature guillotine.

from sklearn.linear_model import LassoCV
import numpy as np

lasso = LassoCV(cv=5, random_state=42).fit(X_train, y_train)
surviving = np.where(lasso.coef_ != 0)[0]
eliminated = np.where(lasso.coef_ == 0)[0]
print(f"Lasso kept {len(surviving)} features, eliminated {len(eliminated)}")

Regularization strength — controlled by lambda, which LassoCV tunes via cross-validation — sets how tight diamond is. Tighter diamond means fewer surviving features, higher bias, lower variance. Turn dial. Get tradeoff.

The Sealed Envelope — Why Splitting Matters

Think of test set as sealed envelope with final exam. Study all you want (training). Quiz yourself with practice tests (validation). Envelope stays sealed until real exam. Peek during studying, score means nothing — memorization, not understanding.

Analogy not perfect, but lesson is. Test set gives honest, unbiased estimate of performance on unseen data. Each time you inspect test results and then tweak model, you tear envelope open more. Eventually it stops being test and becomes second practice exam. Reported "test accuracy" becomes self-lie.

Early career, I treated data splitting like formality. Split, train, evaluate, done. Then one deployment failed spectacularly — model's 97% validation accuracy became 52% in production. Lesson: how you split matters as much as what you build.

Train, Validation, Test — Three Distinct Jobs

Make this concrete with fraud detection system. We have 100,000 transactions. We split them into three sets. Each has different job.

Training set is where model learns. It sees examples, adjusts parameters, builds internal picture of fraud. Validation set is tuning workbench. Use it to compare hyperparameters, try feature sets, decide on regularization. You will inspect validation scores many times. Fine — that is job. Test set is final one-time scorecard. Touch once. Report number. Live with result.

Common split ratios are 70/15/15 or 80/10/10. Conventions, not laws. With millions of rows, 98/1/1 can work — 1% of million is still 10,000 examples. With few hundred rows, fixed validation set is too noisy. Cross-validation is better.

from sklearn.model_selection import train_test_split

# Always carve out the test set first — then split the remainder
X_temp, X_test, y_temp, y_test = train_test_split(
    X, y, test_size=0.15, random_state=42, stratify=y
)
X_train, X_val, y_train, y_val = train_test_split(
    X_temp, y_temp, test_size=0.176, random_state=42, stratify=y_temp
)

Notice two-step process. First extract test set — seal envelope. Then split remainder into train and validation. 0.176 looks strange because 17.6% of remaining 85% is about 15% of total. We pass stratify=y to preserve fraud/legitimate ratio in each set. More soon.

Data Leakage — Contaminating the Crime Scene

If splitting keeps evaluation honest, leakage is how that discipline breaks. Data leakage happens when information unavailable at prediction time sneaks into training.

Analogy: crime scene. Detectives seal scene, wear gloves, log evidence. If investigator walks through in street shoes, tracks mud everywhere, touches things bare-handed, evidence is contaminated. Case falls apart in court. Leakage contaminates model evidence same way. Metrics look strong. Production truth says otherwise.

Several common sources cause this. Sneaky ones.

Preprocessing leakage is most common one in practice. You compute feature mean and standard deviation across entire dataset, then split. Now scaler carries test-set information into training. Fix is rigid: split first, then fit all preprocessors — scalers, imputers, encoders — on training set only, and apply fitted transformers to validation and test. Scikit-learn Pipeline enforces this automatically. Good practitioners do not scale outside pipeline.

Target leakage is subtler. Feature derived from target — or available only after target is known — hands model answer directly. In loan default model, "total_payments_made" is answer in disguise: if customer defaulted, payments stopped. At prediction time, when loan is issued, that number does not exist yet. Hospital mortality model using "total blood transfused" has same problem — total is finalized only after patient outcome is known.

Temporal leakage happens when time-ordered data gets shuffled randomly. Stock price model that sees tomorrow's data during training will look spectacular in validation and useless in practice. Fix comes below: time-based splitting.

Group leakage happens when same entity appears in both training and test. If patient has 20 medical records and 15 go to training while 5 go to test, model can learn patient-specific patterns — it recognizes patient, not general medicine.

Telltale sign of leakage is always same: performance looks too good. If model hits 99.8% accuracy on problem where domain experts expect 85%, do not celebrate. Investigate. Suspiciously good results come from leakage far more often than algorithmic brilliance.

Stratified Splits — Preserving the Balance

Back to fraud detection dataset: suppose 2% of transactions are fraud. In random 80/20 split of 1,000 examples, test set gets 200 examples. Average fraud count is only 4. Random variation can easily give test set 0 or 1 fraud cases, making fraud evaluation meaningless.

Stratified splitting ensures every partition preserves original class distribution. If full dataset is 2% fraud, training is 2% fraud, validation is 2% fraud, test is 2% fraud. No accidents.

from sklearn.model_selection import StratifiedKFold

skf = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)
for fold, (train_idx, val_idx) in enumerate(skf.split(X, y)):
    train_fraud_pct = y[train_idx].mean() * 100
    val_fraud_pct = y[val_idx].mean() * 100
    print(f"Fold {fold}: train {train_fraud_pct:.1f}% fraud, val {val_fraud_pct:.1f}% fraud")

For classification problems, always use stratify=y in splits. Almost no downside. Big protection against bad luck.

For regression, target is continuous, so you cannot stratify directly. Workaround: bin target into quantile buckets and stratify on bins. Not perfect. Prevents all extreme values from landing in one split.

import pandas as pd

y_binned = pd.qcut(y, q=5, labels=False, duplicates="drop")
X_train, X_test, y_train, y_test = train_test_split(
    X, y, test_size=0.2, stratify=y_binned, random_state=42
)

Time-Based Splits — Never Peek at the Future

Everything so far assumes data points are independent — order does not matter. For time series, that assumption is catastrophically wrong.

If you predict tomorrow's stock price and training set contains data from yesterday and next week, model peeked at future. Random shuffling destroys temporal structure that defines problem. Only valid approach: split on time. Train on past, validate on near future, test on farther future.

df = df.sort_values("timestamp")

train = df[df["timestamp"] < "2023-01-01"]
val   = df[df["timestamp"].between("2023-01-01", "2023-06-30")]
test  = df[df["timestamp"] >= "2023-07-01"]

I still dislike this emotionally. You put recent data — most relevant data — into test set instead of training. Every instinct says use freshest data for training. Alternative is self-deception. Time series model evaluated with shuffled splits gives beautiful metrics that mean nothing.

For cross-validation on temporal data, sklearn provides TimeSeriesSplit. It uses expanding window. Each later fold trains on all data up to cutoff and validates on data immediately after. No fold trains on future data.

from sklearn.model_selection import TimeSeriesSplit

tscv = TimeSeriesSplit(n_splits=5)
for fold, (train_idx, val_idx) in enumerate(tscv.split(X)):
    print(f"Fold {fold}: train {len(train_idx)} samples, val {len(val_idx)} samples")

In finance, practitioners go further with purging and embargo. Purging removes training samples that overlap validation window when labels are computed over periods crossing boundary. Embargo adds gap between training and validation to block autocorrelation leakage. Not academic niceties. Difference between trustworthy backtest and lying one.

Group-Based Splits — Keeping Entities Together

In medical studies, one patient can contribute dozens of data points — lab results across visits, multiple scans, repeated measurements. If some records land in training and others in test, model can learn patient identity instead of disease. It memorizes faces, not medicine.

Group-based splitting keeps all records from same entity in same partition. If patient 42 is in training set, all records for patient 42 stay in training set.

from sklearn.model_selection import GroupKFold

gkf = GroupKFold(n_splits=5)
for train_idx, val_idx in gkf.split(X, y, groups=df["patient_id"]):
    train_patients = df["patient_id"].iloc[train_idx].nunique()
    val_patients = df["patient_id"].iloc[val_idx].nunique()
    print(f"Train: {train_patients} patients, Val: {val_patients} patients")

This goes far beyond medicine. Multiple reviews from same user. Multiple images from same camera. Multiple transactions from same account. Whenever data has grouping structure, random splitting breaks independence and inflates metrics. Group splits preserve evaluation integrity.

Cross-Validation — When One Split Isn't Enough

With small dataset, any single train/validation split is noisy. Move few examples and accuracy swings 5%. That does not tell you about model. It tells you about luck of draw.

K-fold cross-validation fixes this by making K different splits, each time using different chunk as validation set and averaging results. Every example serves as validation exactly once. Estimate variance drops. Performance picture gets much more stable.

from sklearn.model_selection import cross_val_score

scores = cross_val_score(model, X, y, cv=5, scoring="f1")
print(f"F1: {scores.mean():.3f} ± {scores.std():.3f}")

5-fold is standard default. 10-fold is slightly more stable but costs twice as much. Leave-one-out, where K equals sample count, is elegant in theory but painfully slow beyond few hundred rows and still gives high-variance estimates.

Real power of cross-validation: it lets you use all data for both training and evaluation, which matters when data is scarce. But it does not replace test set. Cross-validation is for model development and hyperparameter tuning. Test set stays sealed envelope for final evaluation.

Handling Class Imbalance

When 99% of transactions are legitimate and 1% are fraud, model that always predicts "legitimate" gets 99% accuracy and catches zero fraud. Accuracy is meaningless. Model is useless. This is class imbalance. It touches feature selection — some features matter only for minority class — and splitting — you need stratification so minority class stays represented.

Three approaches handle this. In practice, you often combine them.

Resampling changes what model sees during training. SMOTE (Synthetic Minority Over-sampling Technique) creates synthetic minority examples by interpolating between real ones — pick two fraud cases and generate new point between them in feature space. Better than duplicating existing minority examples, which causes overfitting. Critical rule: resample training set only. Never validation or test. Resampled evaluation sets inflate metrics.

from imblearn.over_sampling import SMOTE

smote = SMOTE(random_state=42)
X_train_resampled, y_train_resampled = smote.fit_resample(X_train, y_train)

Class weights tell model to care more about minority-class errors. Instead of treating every misclassification equally, weighted model penalizes missed fraud case 99× more than false flag on legitimate one. Most sklearn estimators support this via class_weight="balanced", which sets weights inversely proportional to class frequencies.

from sklearn.ensemble import RandomForestClassifier

rf = RandomForestClassifier(class_weight="balanced", random_state=42)
rf.fit(X_train, y_train)

Better metrics avoid accuracy trap. Precision, recall, F1, AUC-ROC each show different part of story. With imbalanced data, these numbers matter. Chapter 4 covers them in depth.
