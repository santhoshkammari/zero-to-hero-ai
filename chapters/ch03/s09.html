Data Preprocessing Pipelines

Chapter 3: Data Fundamentals 5 subtopics
TL;DR

A preprocessing pipeline chains every transformation — scaling, encoding, imputation — into a single object that you fit once on training data and replay identically on test data (or in production). Scikit-learn's Pipeline and ColumnTransformer are the core tools: Pipeline enforces sequential order, ColumnTransformer routes different columns to different transformers in parallel. You can write custom transformers for domain-specific logic, merge parallel feature branches with FeatureUnion, and serialize the entire pipeline with joblib so the exact same preprocessing runs months later without a single line of transformation code in your serving path. Pipelines eliminate an entire class of bugs — the kind where training and inference disagree on how data was prepared.

Why Pipelines Instead of Loose Code

Here's a scenario you'll hit on your first real project. You scale the features, encode the categoricals, train a model, ship it. Three months later someone feeds new data into the model and gets garbage predictions. Why? Because the scaler was fit on the training set but someone forgot to apply it — or applied it with different parameters. The encoding mapped "red" to 0 and "blue" to 1 during training, but the serving code mapped them differently. You didn't have a bug in any single step. You had a bug in the gaps between steps.

A preprocessing pipeline is an object that bundles every transformation into a single, ordered sequence. You call .fit() once on training data, and every step learns what it needs (means for scaling, category mappings for encoding). Then .transform() replays those exact learned parameters on new data. No gaps. No mismatches. One object to save, one object to load.

Scikit-learn has had pipelines since its early days, and they remain one of the most underused features in the library. Let's build them up from scratch.

Sklearn Pipeline and ColumnTransformer

The simplest pipeline is a chain of steps where each step is a transformer (or an estimator at the end). Think of it like a conveyor belt: data enters, gets modified by step 1, the output feeds into step 2, and so on.

from sklearn.pipeline import Pipeline
from sklearn.preprocessing import StandardScaler
from sklearn.decomposition import PCA
from sklearn.linear_model import LogisticRegression

pipe = Pipeline([
    ("scaler", StandardScaler()),
    ("pca", PCA(n_components=5)),
    ("clf", LogisticRegression())
])

# fit does: scaler.fit_transform -> pca.fit_transform -> clf.fit
pipe.fit(X_train, y_train)

# predict does: scaler.transform -> pca.transform -> clf.predict
pipe.predict(X_test)

Each tuple is a (name, transformer) pair. The name is just a string label — you pick it. Every step except the last must implement .transform(). The last step can be a classifier or regressor (it only needs .fit() and .predict()). When you call pipe.fit(X, y), scikit-learn calls fit_transform on every intermediate step and fit on the final one. When you call pipe.predict(X), it calls transform on every intermediate step and predict on the last.

This is already useful, but real datasets have mixed types. You might have numeric columns that need scaling and categorical columns that need one-hot encoding. You can't shove both through the same StandardScaler. That's where ColumnTransformer comes in.

from sklearn.compose import ColumnTransformer
from sklearn.preprocessing import StandardScaler, OneHotEncoder
from sklearn.impute import SimpleImputer
from sklearn.pipeline import Pipeline
from sklearn.ensemble import RandomForestClassifier

# Define which columns are which
numeric_features = ["age", "income", "credit_score"]
categorical_features = ["occupation", "city", "education"]

# Build a mini-pipeline for each column group
numeric_pipeline = Pipeline([
    ("imputer", SimpleImputer(strategy="median")),
    ("scaler", StandardScaler())
])

categorical_pipeline = Pipeline([
    ("imputer", SimpleImputer(strategy="most_frequent")),
    ("encoder", OneHotEncoder(handle_unknown="ignore"))
])

# ColumnTransformer routes columns to the right pipeline
preprocessor = ColumnTransformer([
    ("num", numeric_pipeline, numeric_features),
    ("cat", categorical_pipeline, categorical_features)
])

# Full pipeline: preprocess then classify
full_pipe = Pipeline([
    ("preprocessor", preprocessor),
    ("classifier", RandomForestClassifier(n_estimators=100))
])

full_pipe.fit(X_train, y_train)
accuracy = full_pipe.score(X_test, y_test)

Each entry in the ColumnTransformer is a (name, transformer, columns) tuple. The columns argument can be a list of column names (if your data is a DataFrame), a list of integer indices, or even a callable that selects columns. Behind the scenes, ColumnTransformer splits the data, sends each slice to its transformer, and horizontally concatenates the results.

The remainder Parameter

By default, any column not mentioned in the ColumnTransformer is dropped. That's a quiet data loss bug. If you want unmentioned columns passed through unchanged, set remainder="passthrough". If you want them explicitly dropped (the default), leave it or set remainder="drop". You can even pass a transformer — say, remainder=StandardScaler() — to apply something to the leftover columns.

preprocessor = ColumnTransformer(
    transformers=[
        ("num", numeric_pipeline, numeric_features),
        ("cat", categorical_pipeline, categorical_features)
    ],
    remainder="passthrough"  # don't silently discard columns
)

One thing that trips people up: after transformation, the output column order is the concatenation of each transformer's output in the order you listed them, followed by the remainder columns. Your feature names shift. Use preprocessor.get_feature_names_out() in newer scikit-learn versions to see what came out where.

Preprocessing Order — Why It Matters

The order of steps in a pipeline is not arbitrary. It encodes assumptions about your data. Get it wrong and you'll introduce subtle bugs that don't throw errors — they just produce worse models.

Consider imputation and scaling. If you scale first, your missing values (often represented as NaN) will cause the scaler to fail or produce nonsense statistics. So imputation must come before scaling. That's obvious. But here's a less obvious one: if you do feature selection before scaling, the selection algorithm sees unscaled features. Some selection methods (like those based on variance) are sensitive to scale. A feature with values in the thousands will look more "important" than one with values between 0 and 1, even if the small-scale feature is more predictive.

A reasonable default order for most tabular problems looks like this: handle missing values first (imputation), then encode categoricals, then scale numerics, then do any dimensionality reduction, and finally the model. But "reasonable default" doesn't mean "always correct." Domain knowledge should override convention. If you know that outliers are present and you want to clip them, that clipping should happen before the scaler computes means and standard deviations — otherwise the outliers distort the scaling parameters.

from sklearn.pipeline import Pipeline
from sklearn.impute import SimpleImputer
from sklearn.preprocessing import StandardScaler, PowerTransformer
from sklearn.feature_selection import SelectKBest, f_classif

# Order matters: impute -> transform distribution -> scale -> select -> model
numeric_pipeline = Pipeline([
    ("imputer", SimpleImputer(strategy="median")),
    ("power", PowerTransformer(method="yeo-johnson")),
    ("scaler", StandardScaler()),
    ("selector", SelectKBest(f_classif, k=10))
])

The Data Leakage Trap

The most dangerous ordering mistake isn't about which transformer comes first. It's about when you fit. If you scale your entire dataset (train + test) and then split, the scaler has seen test data. That's data leakage — your test metrics will be optimistically biased because the preprocessing "knows" about the test distribution.

Pipelines prevent this automatically. When you call pipe.fit(X_train, y_train), every transformer fits only on training data. When you call pipe.transform(X_test), it uses the parameters learned from training. You don't have to think about it. That's arguably the single biggest reason to use pipelines: they make leakage structurally impossible, not just something you "try to avoid."

This also matters during cross-validation. If you use cross_val_score(pipe, X, y, cv=5), scikit-learn refits the entire pipeline on each fold's training set. Every scaler, every encoder, every imputer — refitted fresh. Without a pipeline, you'd have to manually refit each step inside each fold, and in practice, people forget.

Custom Transformers

Scikit-learn's built-in transformers handle the common cases. But real projects always have domain-specific transformations. Maybe you need to extract the hour from a datetime column, or compute the ratio of two features, or apply a business rule that clips values at a regulatory threshold. You need a custom transformer.

The simplest approach is FunctionTransformer, which wraps a plain function as a transformer:

from sklearn.preprocessing import FunctionTransformer
import numpy as np

log_transformer = FunctionTransformer(np.log1p, validate=True)

# Works inside a pipeline like any other transformer
pipe = Pipeline([
    ("log", log_transformer),
    ("scaler", StandardScaler())
])

That works when your transformation is stateless — it doesn't need to learn anything from the training data. But sometimes your custom step does need to learn. Maybe you want to bin a feature into quantiles based on the training distribution. For that, you write a class that inherits from BaseEstimator and TransformerMixin.

from sklearn.base import BaseEstimator, TransformerMixin
import numpy as np

class OutlierClipper(BaseEstimator, TransformerMixin):
    """Clips values beyond k standard deviations from the mean.
    Learns mean and std from training data only."""

    def __init__(self, k=3.0):
        self.k = k

    def fit(self, X, y=None):
        self.mean_ = np.mean(X, axis=0)
        self.std_ = np.std(X, axis=0)
        self.lower_ = self.mean_ - self.k * self.std_
        self.upper_ = self.mean_ + self.k * self.std_
        return self

    def transform(self, X):
        return np.clip(X, self.lower_, self.upper_)

A few rules to follow. The __init__ method should only store hyperparameters — no computation. Attributes learned from data should be set in fit() and by convention end with an underscore (mean_, std_). The fit() method must return self. The transform() method must return the transformed data. If you inherit from TransformerMixin, you get fit_transform() for free — it just calls fit then transform.

Inheriting from BaseEstimator gives you get_params() and set_params(), which means your custom transformer works with scikit-learn's grid search. You can tune the k parameter of OutlierClipper just like you'd tune C in an SVM:

from sklearn.model_selection import GridSearchCV

pipe = Pipeline([
    ("clipper", OutlierClipper()),
    ("scaler", StandardScaler()),
    ("clf", LogisticRegression())
])

param_grid = {
    "clipper__k": [2.0, 3.0, 4.0],
    "clf__C": [0.1, 1.0, 10.0]
}

search = GridSearchCV(pipe, param_grid, cv=5, scoring="accuracy")
search.fit(X_train, y_train)
print(search.best_params_)

Notice the double-underscore syntax: clipper__k means "the parameter k of the step named clipper." This works at any nesting depth. If your pipeline has a ColumnTransformer named preprocessor containing a sub-pipeline named num with a step named imputer, you'd write preprocessor__num__imputer__strategy to tune the imputation strategy. It's verbose, but unambiguous.

Feature Union — Parallel Pipelines

A Pipeline is sequential: step 1 feeds into step 2. But sometimes you want to compute two different feature sets from the same input and then concatenate them. For example, you might want both PCA features (which capture global variance) and hand-engineered statistical features (which capture domain knowledge). These two computations are independent — there's no reason one should wait for the other.

FeatureUnion runs multiple transformers in parallel on the same input and horizontally concatenates their outputs. Think of it as "run these side by side, then glue the columns together."

from sklearn.pipeline import FeatureUnion, Pipeline
from sklearn.decomposition import PCA
from sklearn.preprocessing import StandardScaler, FunctionTransformer
import numpy as np

def statistical_features(X):
    """Compute row-wise statistics as features."""
    return np.column_stack([
        np.mean(X, axis=1),
        np.std(X, axis=1),
        np.max(X, axis=1) - np.min(X, axis=1)  # range
    ])

feature_union = FeatureUnion([
    ("pca", PCA(n_components=5)),
    ("stats", FunctionTransformer(statistical_features))
])

pipe = Pipeline([
    ("scaler", StandardScaler()),
    ("features", feature_union),
    ("clf", LogisticRegression())
])

pipe.fit(X_train, y_train)

The output of this pipeline's feature step has 5 + 3 = 8 columns: five from PCA and three from the statistical function. Both branches received the same scaled input (because scaler runs before features in the outer pipeline), but they computed independently.

FeatureUnion vs. ColumnTransformer

These two look similar but solve different problems. ColumnTransformer routes different columns to different transformers — it splits the input vertically. FeatureUnion sends the same input to multiple transformers and concatenates the results — it duplicates and recombines. You'll often use both in the same pipeline: a ColumnTransformer at the top to handle mixed types, and a FeatureUnion further down to combine engineered features.

In modern scikit-learn (1.0+), you can also use set_output(transform="pandas") on your pipeline to keep DataFrame outputs flowing through, which makes debugging much easier. But be aware that FeatureUnion historically outputs NumPy arrays, so you may lose column names at that stage unless you're on a recent version and configure output explicitly.

Serialization — Saving and Loading Pipelines

You've built a pipeline. It preprocesses, engineers features, and classifies. It scores well on your test set. Now what? You need to save it so that the serving system — which might be a Flask app, a batch job, or a Lambda function — can load it and call .predict() without reimplementing any transformation logic.

Serialization means converting a Python object into a byte stream you can write to disk and reconstruct later. For scikit-learn pipelines, two tools dominate: joblib and pickle.

import joblib

# Save the fitted pipeline
joblib.dump(full_pipe, "preprocessing_pipeline.joblib")

# Later, in your serving code:
loaded_pipe = joblib.load("preprocessing_pipeline.joblib")
predictions = loaded_pipe.predict(new_data)

joblib is generally preferred over raw pickle for scikit-learn objects because it handles large NumPy arrays more efficiently — it memory-maps them instead of copying. For small models the difference is negligible, but for pipelines that contain a fitted TF-IDF vectorizer with a massive vocabulary matrix, joblib can be noticeably faster.

That said, pickle works fine too and is part of the standard library:

import pickle

with open("pipeline.pkl", "wb") as f:
    pickle.dump(full_pipe, f)

with open("pipeline.pkl", "rb") as f:
    loaded_pipe = pickle.load(f)

Versioning and Compatibility Gotchas

Here's the honest tradeoff. Pickle-based serialization (including joblib) is tied to the exact versions of your libraries. A pipeline saved with scikit-learn 1.2 may fail to load under scikit-learn 1.4 if internal class structures changed. This isn't a theoretical concern — it happens regularly. Always save the library version alongside the model file. A simple approach is to store metadata:

import sklearn
import joblib

artifact = {
    "pipeline": full_pipe,
    "sklearn_version": sklearn.__version__,
    "feature_names": list(X_train.columns),
    "target_name": "churn"
}
joblib.dump(artifact, "model_artifact.joblib")

# On load, verify versions match
artifact = joblib.load("model_artifact.joblib")
assert artifact["sklearn_version"] == sklearn.__version__, (
    f"Version mismatch: saved with {artifact['sklearn_version']}, "
    f"running {sklearn.__version__}"
)

For production systems with long-lived models, consider ONNX export or similar framework-agnostic formats. But for experimentation, rapid iteration, and internal tooling, joblib is simple and reliable — just pin your dependency versions.

Security Note

Never load a pickle file from an untrusted source. Pickle can execute arbitrary code during deserialization. If someone hands you a .pkl file, it could contain malicious payloads. This is not a hypothetical risk — it's a well-known attack vector. Only load serialized pipelines that you (or your team) created, stored in a controlled environment.

That's the full lifecycle: build a pipeline with Pipeline and ColumnTransformer, order the steps deliberately to avoid leakage and statistical distortion, plug in custom transformers for domain logic, combine parallel feature branches with FeatureUnion, and serialize the whole thing for deployment. One object, fit once, replayed everywhere. It's not glamorous, but it eliminates an entire category of bugs that would otherwise haunt you in production.
