Nice to Know

Chapter 3: Data Fundamentals 8 topics

These are things you'll encounter in the wild but don't need to master right now. Recognize them on sight, know when they matter, and look them up when you need them.

The Topics

DVC (Data Version Control)

Git tracks code, DVC tracks data. It stores content hashes in small .dvc pointer files (committed to Git) while actual data lives in S3/GCS/Azure. Switch dataset versions like switching Git branches: git checkout v2 && dvc pull.

$ dvc init
$ dvc add data/training_images/     # creates .dvc pointer file
$ dvc remote add -d store s3://my-bucket/dvc-store
$ dvc push                           # uploads data to S3

Use it when: you need reproducible experiments and your datasets are too large for Git. Most teams skip it until they've been burned by "which data did model v2.3 train on?"

MLflow Data Tracking

MLflow logs datasets alongside experiment runs — which data version, which hyperparameters, which metrics. Not as rigorous as DVC for version control, but integrates naturally if you're already using MLflow for experiment tracking.

import mlflow
mlflow.log_input(mlflow.data.from_pandas(df, name="training_v3"))

Feature Stores

A feature store (Feast, Tecton, Hopsworks) serves precomputed features consistently across training and inference. Without one, training computes features in a batch job while inference recomputes them in real-time — and the two implementations inevitably diverge. Feature stores solve the train/serve skew problem at the feature level.

When you need it: when your company has multiple models sharing the same features, or when training/serving feature parity is a real pain point. Most teams don't need one until they have 5+ models in production.

Delta Lake and Lakehouse

Delta Lake adds ACID transactions and time travel to data lakes (Parquet files on S3). You can query data as it existed at any point in time: df = spark.read.format("delta").option("timestampAsOf", "2024-01-15").load(path). The "lakehouse" pattern combines the flexibility of a data lake with warehouse-like reliability.

GDPR / CCPA / HIPAA — What ML Engineers Need to Know

GDPR (EU) — right to erasure means users can request deletion of their data. If that data trained your model, the model's weights may still encode it. This spawned "machine unlearning" research. CCPA (California) — similar rights, broader definition of "sale" that can include sharing with model providers. HIPAA (US healthcare) — de-identification requires removing 18 specific identifiers (Safe Harbor) or statistical certification.

Practical impact: data minimization (collect only what you need), purpose limitation (fraud model can't be repurposed for marketing without new consent), and data lineage (track which records went into which training runs).

Differential Privacy

Adds calibrated noise to queries or gradients to provide mathematical guarantees that individual records can't be extracted. The tradeoff: more privacy = more noise = less model accuracy. Used in production at Apple and Google for aggregate statistics. For ML training, DP-SGD clips per-sample gradients and adds noise — works but typically costs 5-15% accuracy.

Synthetic Data Generation

Generate fake but statistically plausible data when real data is scarce, expensive, or privacy-sensitive. Tools: Faker for realistic names/addresses, SDV (Synthetic Data Vault) for tabular data, GANs/diffusion models for images. The risk: synthetic data only captures patterns you already know about — it can't surprise you with real-world edge cases.

Data Catalogs and Contracts

A data catalog (DataHub, Amundsen) makes datasets discoverable — searchable metadata so you don't reinvent features someone already computed. A data contract is a formal agreement between data producer and consumer about schema, freshness, and quality guarantees. Both become essential once your organization has 50+ datasets and multiple ML teams.

The Point of This Section

You don't need to master any of these right now. When you encounter DVC in a colleague's workflow, or your company starts discussing feature stores, or a compliance officer mentions GDPR — you'll know what they're talking about. That recognition is the goal.
