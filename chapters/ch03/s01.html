Data Types, Sources & Formats

TL;DR

You know what CSVs and databases are. This section focuses on what actually matters: which format to use for ML workloads (Parquet, almost always), how to load data without footguns, and the data type distinctions that affect modeling decisions.

Data Types That Affect Modeling

The structured/unstructured/semi-structured distinction is real but you already know it. What actually bites you in practice are the subtler type decisions.

Continuous vs. discrete — linear regression expects continuous targets, Poisson models expect counts. Treating one as the other silently degrades performance.

Nominal vs. ordinal categorical — ordinal data has a ranking (education level, satisfaction score). Encoding ordinal as plain integers tells the model the gaps are uniform. They usually aren't. Nominal data (colors, country codes) has no order — one-hot encode it, don't label-encode it into a number line that doesn't exist.

Time series — order matters. Shuffling rows destroys the signal. Never random-split time series (see Section 5).

Text and images — unstructured, but featurizable. The important thing: don't reach for a Transformer when a TF-IDF + gradient boosting pipeline solves the problem in 5 minutes. Match model complexity to data complexity.

Data Sources — What Matters in Practice

SQL databases, APIs, CSVs from a shared drive, Parquet files in S3, streaming from Kafka — you'll encounter all of them. Two things to internalize:

Push filtering to the source. Don't SELECT * FROM orders and filter in Python. Let the database do what it's good at. This is the single biggest performance mistake in data-backed ML pipelines.

Know where your data actually lives in production. Data warehouses (Snowflake, BigQuery) store clean analytical data. Data lakes (S3, HDFS) store raw everything. Streaming systems (Kafka, Kinesis) deliver real-time events. Most ML training reads from warehouses or lakes. Real-time inference needs streaming.

File Formats — The Only Decision That Matters

FormatUse WhenAvoid When
CSVSharing with non-technical people, quick explorationAnything in a pipeline. Slow, no types, no compression.
ParquetDefault for ML data. Columnar, compressed, type-safe, reads only the columns you need.You need human-readable files.
FeatherIntermediate pipeline files where speed matters more than compression.Long-term storage or cross-system interchange.
HDF5Large numerical arrays, scientific computing, model weights.Tabular data (Parquet is better).
JSONAPI responses, config files, semi-structured data.Large tabular datasets. Verbose, slow columnar access.

The honest truth: if your data is going into an ML pipeline, use Parquet. A 10 GB CSV becomes 1-2 GB in Parquet, reads 3-5× faster, and preserves type information so "02134" doesn't lose its leading zero.

# The three lines that replace CSV in your workflow
df.to_parquet("data.parquet", engine="pyarrow")
df = pd.read_parquet("data.parquet", columns=["feature_1", "target"])  # reads ONLY those columns

# Quick format comparison on your own data
import time, os
for write_fn, read_fn, path in [
    (lambda: df.to_csv("d.csv", index=False), lambda: pd.read_csv("d.csv"), "d.csv"),
    (lambda: df.to_parquet("d.pq"), lambda: pd.read_parquet("d.pq"), "d.pq"),
]:
    write_fn()
    t = time.time(); read_fn(); elapsed = time.time() - t
    print(f"{path:10s} {os.path.getsize(path)/1e6:.1f} MB  {elapsed:.3f}s")

Loading Data Without Footguns

A few patterns that save hours:

# Force types on load — prevents silent type coercion
df = pd.read_csv("data.csv", dtype={"zip_code": str, "quantity": "Int32"})

# Category dtype for low-cardinality strings — 90% memory savings
df["country"] = df["country"].astype("category")

# Chunked reading for files that don't fit in memory
chunks = pd.read_csv("huge.csv", chunksize=50_000)
df = pd.concat([c[c["status"] == "active"] for c in chunks], ignore_index=True)

# SQL — push the WHERE clause down
from sqlalchemy import create_engine
engine = create_engine("sqlite:///db.sqlite")
df = pd.read_sql("SELECT user_id, revenue FROM orders WHERE revenue > 0", engine)

For truly large datasets (tens of gigabytes), switch to Polars or Dask. Pandas is single-threaded and in-memory — it's the right default until it isn't.

What You Should Now Be Able To Do

  • Choose Parquet over CSV for any ML pipeline without hesitation
  • Explain why ordinal encoding integers can mislead a model
  • Load a CSV with explicit dtypes to prevent silent type coercion
  • Push filtering to SQL rather than loading everything into Python
  • Know when to reach for Polars/Dask vs. Pandas

Exploratory Data Analysis

TL;DR

EDA is a conversation with your data. You ask questions with code, it answers with numbers and plots, and you decide what to do next. This section is a workflow — the exact sequence of moves you run on any new dataset. Skip it, and you'll spend hours debugging a model only to discover a column was stored as strings.

The EDA Workflow

Every EDA follows the same arc: shape → types → missing → distributions → relationships → quality. Think of it as peeling layers. Each layer reveals problems the next layer needs you to have seen.

Step 1: Shape, Types, and First Look

import pandas as pd
import numpy as np

df = pd.read_csv("housing.csv")

print(df.shape)          # (rows, columns) — how big is this?
print(df.dtypes)         # a column showing 'object' that should be numeric = dirty data
df.info()                # non-null counts + dtypes + memory — your one-stop summary
print(df.sample(5))      # random rows > head() — head can hide patterns

A column that should be numeric but shows object almost always contains stray commas, "N/A" strings, or encoding garbage. Catching this here saves you hours downstream.

Step 2: Summary Statistics

print(df.describe())                 # count, mean, std, min, 25%, 50%, 75%, max
print(df.describe(include='all'))    # includes categorical columns too

What to look for in describe():

  • min/max sanity — negative ages? Prices of zero? Future dates in historical data?
  • mean vs. median gap — large gap = skewed distribution = outliers pulling the mean
  • std relative to mean — near-zero std = near-constant feature = useless for prediction
  • count < total rows — you have missing data in that column

Step 3: Missing Data Patterns

# Quick missing overview
print(df.isnull().sum().sort_values(ascending=False))
print(f"\n% missing:\n{(df.isnull().mean() * 100).sort_values(ascending=False)}")

# Visual pattern — look for correlated missingness
import missingno as msno
msno.matrix(df, figsize=(10, 4), fontsize=8)
# Columns that go missing together → MAR or MNAR (see Section 3)

If two columns always have missing values in the same rows, that's a signal — the missingness is correlated. Understanding why data is missing matters more than how you fill it.

Step 4: Distributions

Numerical Features

import matplotlib.pyplot as plt
import seaborn as sns

# Histograms for all numeric columns — fast first pass
df.select_dtypes(include=[np.number]).hist(bins=50, figsize=(14, 10))
plt.tight_layout()
plt.show()

What you're looking for: skewness (long tails — consider log transform), multimodality (multiple peaks — possibly mixed populations), hard boundaries (values capped at 500,000 — data was clipped), spikes (everyone has exactly 0 or exactly 999 — sentinel values masquerading as real data).

Categorical Features

print(df.nunique())                                    # cardinality per column
print(df["ocean_proximity"].value_counts())             # raw counts
print(df["ocean_proximity"].value_counts(normalize=True))  # proportions

A category with 95% of the data gives almost no signal. A column with 50,000 unique values out of 50,001 rows is an ID column, not a feature. Categories appearing <1-2% of the time should likely be bucketed into "Other" to prevent overfitting.

Step 5: Relationships

Correlation Matrix

# Pearson (linear) correlation
corr = df.select_dtypes(include=[np.number]).corr()

plt.figure(figsize=(10, 8))
sns.heatmap(corr, annot=True, fmt=".2f", cmap="coolwarm", center=0, square=True)
plt.title("Pearson Correlation")
plt.tight_layout()
plt.show()

Pearson captures linear relationships. Spearman captures monotonic (including non-linear). Use Spearman when you suspect curved but still monotonic patterns. Zero correlation ≠ independence — two variables can be perfectly related non-linearly and still show zero Pearson.

Scatter Plots and Box Plots

fig, axes = plt.subplots(1, 2, figsize=(14, 5))

# Numeric vs numeric — looking for trends, clusters, outliers
axes[0].scatter(df["median_income"], df["median_house_value"], alpha=0.1, s=5)
axes[0].set(xlabel="Median Income", ylabel="House Value", title="Income vs Value")

# Numeric vs categorical — do groups differ?
sns.boxplot(x="ocean_proximity", y="median_house_value", data=df, ax=axes[1])
axes[1].tick_params(axis='x', rotation=45)
plt.tight_layout()
plt.show()

If box plots look identical across groups, that categorical feature probably doesn't help predict the numeric one. If they differ dramatically, you've found a useful feature.

Grouped Statistics

print(df.groupby("ocean_proximity")["median_house_value"].agg(
    ["mean", "median", "std", "count"]
))

This is often more informative than a plot. The numbers tell you exactly how much separation exists between groups.

Step 6: Duplicates and Data Quality

# Exact duplicates
print(f"Duplicate rows: {df.duplicated().sum()}")

# Integrity checks — domain-specific
assert (df["age"] >= 0).all(), "Negative ages found"
assert df["price"].between(0, 1e8).all(), "Prices outside reasonable range"

# Inconsistencies
# "New York" vs "new york" vs "NY" — standardize before modeling
print(df["city"].str.lower().nunique(), "vs", df["city"].nunique())

Step 7: Target Variable

Always examine the target separately. For classification: check class balance. For regression: check distribution shape (skewed targets often benefit from log transform).

# Classification — check balance
print(df["label"].value_counts(normalize=True))
# If 98% is one class, you need stratified splitting and weighted metrics

# Regression — check skew
print(f"Target skewness: {df['price'].skew():.2f}")
# Skew > 1: consider log1p transform

Automated EDA Tools

For a first pass, these generate everything above in one call:

# ydata-profiling — most comprehensive
from ydata_profiling import ProfileReport
ProfileReport(df, title="EDA Report", explorative=True).to_file("report.html")

# sweetviz — great for train/test comparison
import sweetviz as sv
sv.compare([train, "Train"], [test, "Test"]).show_html("comparison.html")

The honest tradeoff: automated tools show you everything but can't tell you what matters for your problem. They also struggle with 1M+ rows. Use them to get oriented, then dig into the specifics manually.

The Real Point of EDA

EDA isn't a checkbox. It's iterative — inspect, find something odd, investigate, clean, inspect again. The goal isn't a perfect dataset. It's a well-understood dataset. A model built on data you understand will always outperform one built on data you threw at an algorithm and hoped for the best.

What You Should Now Be Able To Do

  • Run the full EDA workflow (shape → types → missing → distributions → relationships → quality) on any dataset
  • Interpret df.describe() to spot skew, outliers, near-constant features, and missing data
  • Choose between Pearson and Spearman correlation based on relationship type
  • Identify class imbalance, sentinel values, and data integrity violations
  • Use automated EDA tools for a first pass and know their limitations
  • Explain why EDA is iterative, not a one-shot step
