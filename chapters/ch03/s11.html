Data Versioning and Lineage

Chapter 3: Data Fundamentals 7 subtopics
TL;DR

Your model is only as trustworthy as the data that built it, and you cannot trust what you cannot trace. Data versioning gives you Git-like snapshots of datasets so every experiment is reproducible. Tools like DVC manage large files alongside code, MLflow links data to training runs, and Delta Lake brings time travel to data lakes. Feature stores serve precomputed features consistently across training and inference. Data catalogs make datasets discoverable, data contracts enforce schema agreements between teams, and schema evolution strategies keep your pipelines from breaking when the world changes. Together these practices form the lineage layer — the audit trail that tells you exactly which bytes produced which model.

Why Data Versioning Matters

Imagine you trained a fraud-detection model last Tuesday. It worked great. On Thursday someone re-ran the pipeline and accuracy dropped eight points. What changed? If you versioned your code with Git but not your data, you have no idea. The code is identical. The data is not, and you have no way to prove it.

Data versioning means recording an immutable snapshot of your dataset at a point in time, the same way Git records a snapshot of your source code. It sounds like an obvious thing to do, but most teams skip it because datasets are large, they change often, and Git was never designed to handle multi-gigabyte binary files.

There are three reasons you should care. First, reproducibility. If you cannot recreate the exact training set that produced model v2.3, you cannot debug it, retrain it, or compare it fairly against model v2.4. Second, debugging. When a model regresses, the first question is always "did the data change?" Versioning gives you the diff. Third, compliance. Regulations like GDPR and the EU AI Act increasingly require that you prove which data was used to train a model, and that you can delete a specific person's records retroactively. Without versioned lineage, that proof does not exist.

Data lineage is the closely related idea of tracking where data came from, what transformations were applied, and which downstream artifacts consumed it. Versioning tells you what the data looked like. Lineage tells you how it got that way. You need both.

DVC — Data Version Control

DVC (Data Version Control) is an open-source tool that bolts large-file versioning and pipeline management onto Git. The idea is simple: Git tracks small pointer files, and DVC tracks the actual data in a separate storage backend — S3, GCS, Azure Blob, or even a local directory. You get the full Git workflow (branches, tags, merges) while your data lives somewhere cost-effective.

How DVC Works

When you run dvc add on a file, DVC computes a content hash (MD5 by default), moves the file into a local cache directory, and creates a tiny .dvc pointer file that records the hash, the file size, and the path. You commit the .dvc file to Git. When a colleague clones the repo, they run dvc pull and DVC fetches the matching data from the configured remote. It is essentially Git LFS done right for ML workflows.

# Initialize DVC in an existing Git repo
$ git init && dvc init

# Track a large dataset
$ dvc add data/training_images/
# This creates data/training_images.dvc and updates .gitignore

# Commit the pointer file to Git
$ git add data/training_images.dvc data/.gitignore
$ git commit -m "Track training images v1"

# Configure remote storage
$ dvc remote add -d myremote s3://my-bucket/dvc-store

# Push data to the remote
$ dvc push

Now anyone with access to the repo and the S3 bucket can reproduce your dataset with git checkout <tag> && dvc pull. Switching between dataset versions is as easy as switching Git branches.

Pipeline DAGs

DVC also lets you define reproducible pipelines. A dvc.yaml file describes stages — each with a command, dependencies, and outputs. DVC builds a directed acyclic graph (a DAG) from these stages and only re-runs the parts whose inputs have changed.

# dvc.yaml
stages:
  preprocess:
    cmd: python src/preprocess.py --input data/raw.csv --output data/clean.csv
    deps:
      - src/preprocess.py
      - data/raw.csv
    outs:
      - data/clean.csv

  train:
    cmd: python src/train.py --data data/clean.csv --model models/rf.pkl
    deps:
      - src/train.py
      - data/clean.csv
    outs:
      - models/rf.pkl
    metrics:
      - metrics.json:
          cache: false
# Run the full pipeline (only changed stages execute)
$ dvc repro

# Compare metrics across Git branches
$ dvc metrics diff main

This is powerful because it ties data, code, and model artifacts together in a single reproducible graph — all tracked by Git commits.

MLflow Data Tracking

MLflow is better known for experiment tracking, but it also lets you log datasets alongside runs. The idea is that every training run should record not just hyperparameters and metrics, but also a fingerprint of the data it consumed.

import mlflow
import mlflow.data
from mlflow.data.pandas_dataset import from_pandas
import pandas as pd

df = pd.read_parquet("data/clean.parquet")
dataset = from_pandas(df, source="data/clean.parquet", name="training_set")

with mlflow.start_run():
    mlflow.log_input(dataset, context="training")
    mlflow.log_param("n_estimators", 200)
    # ... train model ...
    mlflow.log_metric("accuracy", 0.94)

Now the MLflow UI shows which dataset was used for each run. If accuracy drops, you can compare the dataset digests between the good run and the bad run. MLflow computes a hash of the dataframe's contents, so even if filenames stay the same, content changes are caught.

This is lighter-weight than DVC. You are not storing the full dataset in a versioned cache — you are recording a reference and a hash. For teams already using MLflow, it is a quick win. For full reproducibility, you would pair it with DVC or a data lake that supports time travel.

Delta Lake and the Lakehouse Pattern

What if your data is not a static file but a constantly updating table in a data lake? This is where Delta Lake comes in. Delta Lake is an open-source storage layer that adds ACID transactions, schema enforcement, and time travel to Apache Spark-compatible data lakes.

Under the hood, Delta Lake stores data as Parquet files plus a transaction log (the _delta_log directory). Every write operation — insert, update, delete, or schema change — is recorded as a JSON commit in this log. That means you can query the table as it existed at any previous version or timestamp.

# PySpark: read a Delta table as it existed at version 5
df = spark.read.format("delta").option("versionAsOf", 5).load("/data/events")

# Or by timestamp
df = spark.read.format("delta") \
    .option("timestampAsOf", "2024-01-15T08:00:00Z") \
    .load("/data/events")

This time travel capability is a form of data versioning built into the storage layer itself. You do not need a separate tool to snapshot your tables. You just query the version you need. Delta Lake retains history based on a configurable retention period (default 30 days), after which old files are vacuumed to save storage.

The lakehouse pattern takes this further by combining the scalability of a data lake with the reliability of a data warehouse. You store everything in open formats (Parquet plus Delta log), run both SQL analytics and ML training on the same copy, and trust the transaction log to keep things consistent. Apache Iceberg and Apache Hudi offer similar capabilities with slightly different tradeoffs — Iceberg excels at schema evolution and partition evolution, while Hudi is optimized for upsert-heavy workloads.

Feature Stores

A feature store is a centralized system that manages the computation, storage, and serving of features — the engineered input variables your models consume. The core problem it solves is this: in training, you compute features from historical data using batch jobs. In production, you need the same features computed in real time for inference. If those two code paths diverge, your model sees different inputs in production than it saw during training. This is called training-serving skew, and it is one of the most common causes of silent model failures.

Online and Offline Serving

A feature store typically has two layers. The offline store holds historical feature values, usually in a data warehouse or Parquet files, and is used for generating training datasets. The online store holds the latest feature values in a low-latency database (Redis, DynamoDB) and is used for real-time inference. The feature store keeps both in sync.

Point-in-Time Correctness

One subtle but critical detail is point-in-time correctness. When you build a training set, you must not use feature values that were computed after the event you are trying to predict. That is data leakage. A feature store handles this by maintaining timestamped feature values and performing point-in-time joins automatically.

# Feast example: define a feature view
from feast import Entity, FeatureView, Field, FileSource
from feast.types import Float32, Int64

driver = Entity(name="driver_id", join_keys=["driver_id"])

driver_stats_source = FileSource(
    path="data/driver_stats.parquet",
    timestamp_field="event_timestamp",
)

driver_stats_fv = FeatureView(
    name="driver_hourly_stats",
    entities=[driver],
    schema=[
        Field(name="trips_today", dtype=Int64),
        Field(name="avg_rating", dtype=Float32),
    ],
    source=driver_stats_source,
    ttl=timedelta(hours=2),
)
# Retrieve training data with point-in-time join
from feast import FeatureStore

store = FeatureStore(repo_path="feature_repo/")
training_df = store.get_historical_features(
    entity_df=entity_df,  # DataFrame with driver_id + event_timestamp
    features=["driver_hourly_stats:trips_today", "driver_hourly_stats:avg_rating"],
).to_df()

Feast is open-source and infrastructure-agnostic. Tecton is a managed alternative that adds real-time feature pipelines, monitoring, and tighter integration with cloud data platforms. Both solve the same core problem — one set of feature definitions, consistent across training and serving.

Data Catalogs

As your organization accumulates dozens (or hundreds) of datasets, a new problem emerges: discovery. A data catalog is a searchable inventory of all datasets, tables, feature stores, and data assets in your organization. Think of it as a library card catalog for data.

A good data catalog stores technical metadata (schema, column types, row counts, freshness), business metadata (descriptions, owners, tags, data classification), and lineage information (which pipelines produce and consume each dataset). Tools like DataHub, Apache Atlas, Amundsen, and OpenMetadata all serve this purpose. Cloud providers offer their own: Google Data Catalog, AWS Glue Data Catalog, Azure Purview.

The practical value is straightforward. Instead of Slacking a colleague to ask "where is the user-events table and what do the columns mean?", you search the catalog. Instead of guessing whether a table is still actively maintained, you check the freshness metadata. For governance, the catalog tracks who has access to what and whether datasets contain PII.

A catalog is not something you build from scratch. You integrate it with your existing infrastructure — it crawls your databases, S3 buckets, and pipelines, extracting metadata automatically. The hard part is keeping descriptions and ownership up to date. The tooling helps, but it requires cultural buy-in: someone has to own the documentation.

Data Contracts

A data contract is a formal agreement between a data producer and a data consumer about the schema, semantics, and quality guarantees of a dataset. It is the API specification of the data world.

Here is the problem it solves. Team A owns a pipeline that writes user-event data to a table. Team B's ML pipeline reads that table to generate features. One day, Team A renames a column from user_id to uid. Team B's pipeline breaks at 3 AM. Nobody is happy.

A data contract prevents this by making the schema explicit and enforced. The producer publishes a contract (often a YAML or JSON schema file) that says: this table will always have these columns with these types, updated at this frequency, with this SLA for freshness. Any breaking change requires a version bump and a migration path.

# Example data contract (YAML)
name: user_events
version: 2
owner: team-a@company.com
description: Clickstream events from the web app
schema:
  - name: user_id
    type: string
    required: true
    pii: true
  - name: event_type
    type: string
    required: true
    enum: [click, view, purchase, signup]
  - name: timestamp
    type: timestamp
    required: true
  - name: page_url
    type: string
    required: false
quality:
  freshness: 1h
  completeness:
    user_id: 100%
    event_type: 100%

Enforcement can happen at multiple levels. You can validate the contract in CI before deploying schema changes. You can run runtime checks that alert when freshness or completeness drops below the agreed threshold. Tools like Soda, Great Expectations, and dbt tests can all serve as the enforcement engine. The key insight is that the contract shifts data quality from "the consumer's problem" to "a shared responsibility with explicit expectations."

Schema Evolution

Schema evolution is the discipline of changing a dataset's structure — adding columns, renaming fields, changing types — without breaking downstream consumers. It is the long game of data versioning.

There are two directions to worry about. Backward compatibility means new data can be read by old code. If you add a new column with a default value, old readers simply ignore it — that is backward compatible. If you remove a column that old code expects, it breaks. Forward compatibility means old data can be read by new code. If new code expects a column that only exists in recent records, it needs to handle the missing values gracefully.

Avro and Protobuf Approaches

Serialization formats like Avro and Protobuf have built-in rules for safe schema evolution. In Avro, every field can have a default value. When the reader encounters a field that exists in the writer's schema but not in the reader's, it skips it. When the reader expects a field that does not exist in the writer's schema, it fills in the default. This makes adding and removing optional fields safe.

Protobuf takes a numbered-field approach. Each field has a unique integer tag. You can add new fields with new tag numbers, and old readers ignore unknown tags. You should never reuse a tag number for a different field. This makes additive changes safe by default, but renaming or changing a field's semantics requires care.

// Protobuf: safe evolution
// v1
message UserEvent {
  string user_id = 1;
  string event_type = 2;
  int64 timestamp = 3;
}

// v2 — added a field (backward and forward compatible)
message UserEvent {
  string user_id = 1;
  string event_type = 2;
  int64 timestamp = 3;
  string page_url = 4;  // new field, old readers skip tag 4
}

For tabular data, Delta Lake and Iceberg handle schema evolution at the storage layer. Delta Lake supports adding columns, widening types (e.g., int to long), and reordering columns — all without rewriting existing Parquet files. You enable it with .option("mergeSchema", "true") on write.

The practical advice is this: design for additive changes. Adding a new column is almost always safe. Removing or renaming a column is a breaking change that requires a deprecation period, a new version of the contract, and coordination with consumers. If you treat every schema change like an API change — with versioning, migration guides, and a sunset timeline — you will save yourself a lot of 3 AM pages.
