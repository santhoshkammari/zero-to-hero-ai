Data Cleaning & Feature Engineering

Chapter 3: Data Fundamentals Section 3 of 10
TL;DR

This is where 80% of ML improvement comes from. Cleaning fixes the mess — missing values, outliers, inconsistent types. Feature engineering transforms what's left into representations models can actually learn from. A logistic regression with brilliantly engineered features will crush a neural network eating raw garbage. Every cleaning and encoding decision is a modeling decision.

Missing Data — When to Drop, Impute, or Flag

Before you impute anything, understand why the data is missing. The reason determines which fix is safe.

MCAR (Missing Completely at Random) — pure bad luck. A sensor glitched, a page failed to load. Dropping is safe; you just lose power. MAR (Missing at Random) — missingness depends on other observed variables. Younger people skip the income question. Condition your imputation on those variables. MNAR (Missing Not at Random) — the dangerous one. High earners refuse to report income. No statistical trick fully fixes this. Add a missing indicator flag and be honest about residual bias.

Decision Framework

SituationActionWhy
<5% rows affected, looks MCARDrop rowsMinimal information loss
Column >50% emptyDrop columnImputation at this point is mostly invention
Skewed numericsMedian imputationRobust to outliers (mean is not)
Column relationships matterKNN or MICE imputationRespects inter-column correlations
Low-cardinality categoricalMode imputationSimple, usually adequate
Suspect MNARImpute + add missing indicatorLet the model learn from the missingness pattern
Target variableNever imputeA model trained on fabricated labels learns fiction
 
from sklearn.impute import SimpleImputer, KNNImputer
from sklearn.experimental import enable_iterative_imputer
from sklearn.impute import IterativeImputer

# Simple imputation
median_imp = SimpleImputer(strategy="median")
X[["age", "income"]] = median_imp.fit_transform(X[["age", "income"]])

# KNN imputation — respects feature relationships, but scale first
from sklearn.preprocessing import StandardScaler
scaler = StandardScaler()
X_scaled = scaler.fit_transform(X)
X_imputed = KNNImputer(n_neighbors=5).fit_transform(X_scaled)
X_imputed = scaler.inverse_transform(X_imputed)

# MICE — gold standard for MAR, computationally heavier
mice = IterativeImputer(max_iter=10, random_state=42)
X_imputed = mice.fit_transform(X)

# Missing indicator — when missingness itself is predictive
X["income_missing"] = X["income"].isnull().astype(int)

Outlier Handling

An outlier is either a data error or a real but rare observation. Your response should differ.

Detection

# IQR method — no distribution assumptions
Q1, Q3 = df["salary"].quantile([0.25, 0.75])
IQR = Q3 - Q1
mask = df["salary"].between(Q1 - 1.5*IQR, Q3 + 1.5*IQR)

# Z-score — assumes roughly normal
from scipy import stats
z = np.abs(stats.zscore(df["salary"].dropna()))
mask = z < 3  # flag anything beyond 3 std devs

What to Do

SituationAction
Clearly an error (negative age, $5M salary typo)Remove or correct
Real but extreme (a genuine billionaire)Cap/clip (Winsorize) to the 1st/99th percentile
Using tree-based modelsOften leave alone — trees handle outliers naturally
Using linear/distance-based modelsTransform (log) or clip — outliers dominate these models
 
# Winsorization — clip extremes to percentile bounds
lower, upper = df["salary"].quantile([0.01, 0.99])
df["salary_clipped"] = df["salary"].clip(lower, upper)

Feature Scaling

Distance-based models (k-NN, SVM) and gradient-based models (linear regression, neural nets) are sensitive to feature scale. A feature ranging 0-1,000,000 dominates one ranging 0-1. Tree-based models don't care.

ScalerWhat It DoesUse When
StandardScalerMean=0, std=1Default for most models. Assumes roughly Gaussian.
MinMaxScalerSquish to [0, 1]Neural nets, or when you need bounded values.
RobustScalerUses median + IQRData with outliers. Ignores extreme values.
 
from sklearn.preprocessing import StandardScaler, RobustScaler

# Always fit on training data only, transform both
scaler = StandardScaler()
X_train_scaled = scaler.fit_transform(X_train)
X_test_scaled = scaler.transform(X_test)  # uses train's mean/std

Log and Power Transforms

Real-world data loves long right tails — income, website traffic, word frequencies. Log transforms compress the tail and make distributions more symmetric, which helps linear models.

# Log transform — use log1p for data with zeros
df["income_log"] = np.log1p(df["income"])

# Box-Cox — finds optimal power parameter (positive data only)
# Yeo-Johnson — handles zeros and negatives
from sklearn.preprocessing import PowerTransformer
pt = PowerTransformer(method="yeo-johnson")
X_transformed = pt.fit_transform(X[["income", "age"]])

Categorical Encoding

Models eat numbers. Encoding is how you translate categories into numbers. The encoding you choose is a modeling decision — not just a preprocessing step.

EncodingUse WhenWatch Out
One-HotLow cardinality (<15 categories), nominal dataExplodes with high cardinality. Use drop='first' for linear models to avoid multicollinearity.
OrdinalTrue ordered categories (education level, size S/M/L/XL)Don't use for nominal data — implies a distance between categories that doesn't exist.
Target encodingHigh cardinality categoricals. Replaces each category with the mean target value.Leaks target info if not done with CV folds. Always regularize.
Frequency encodingWhen the category's frequency itself is informative.Two categories with the same frequency become indistinguishable.
HashingVery high cardinality (100k+ categories), streaming data.Collisions are inevitable. Irreversible — can't decode back.
 
from sklearn.preprocessing import OneHotEncoder, OrdinalEncoder
import category_encoders as ce

# One-Hot — drop='first' avoids the dummy variable trap
ohe = OneHotEncoder(sparse_output=False, drop='first', handle_unknown='ignore')
X_encoded = ohe.fit_transform(df[["color", "city"]])

# Target encoding — use with cross-validation to prevent leakage
te = ce.TargetEncoder(cols=["city"], smoothing=1.0)
X_train["city_enc"] = te.fit_transform(X_train["city"], y_train)
X_test["city_enc"] = te.transform(X_test["city"])

Creating Features — Where the Real Gains Live

Feature engineering is using domain knowledge to create new columns that make patterns obvious to models. A logistic regression with the right features will beat a neural network eating raw data.

Interaction Features

# Manual — when you know the domain
df["price_per_sqft"] = df["price"] / df["sqft"]
df["lot_coverage"] = df["house_sqft"] / df["lot_sqft"]

# Automated — polynomial features (use sparingly)
from sklearn.preprocessing import PolynomialFeatures
poly = PolynomialFeatures(degree=2, interaction_only=True, include_bias=False)
X_interactions = poly.fit_transform(X[["age", "income"]])
# Creates: [age, income, age*income]

Polynomial features explode combinatorially. 50 features at degree 3 = thousands of columns. Use interaction_only=True or apply with strong regularization.

Datetime Features

df["date"] = pd.to_datetime(df["date"])
df["hour"] = df["date"].dt.hour
df["day_of_week"] = df["date"].dt.dayofweek
df["month"] = df["date"].dt.month
df["is_weekend"] = df["day_of_week"].isin([5, 6]).astype(int)

# Cyclical encoding — so the model knows hour 23 is close to hour 0
df["hour_sin"] = np.sin(2 * np.pi * df["hour"] / 24)
df["hour_cos"] = np.cos(2 * np.pi * df["hour"] / 24)

Binning Continuous Features

# When the exact value matters less than the range
from sklearn.preprocessing import KBinsDiscretizer

binner = KBinsDiscretizer(n_bins=5, encode='ordinal', strategy='quantile')
df["age_bin"] = binner.fit_transform(df[["age"]])

Binning throws away information. Use it deliberately — when you know the relationship is step-wise, not smooth, or when domain experts think in ranges ("young / middle-aged / senior").

Text Features (Quick Version)

from sklearn.feature_extraction.text import TfidfVectorizer

tfidf = TfidfVectorizer(max_features=5000, ngram_range=(1, 2))
X_text = tfidf.fit_transform(df["description"])
# Produces a sparse matrix — feed directly to linear models or tree models

Aggregation Features

# Per-group statistics as features
user_stats = df.groupby("user_id").agg(
    avg_purchase=("amount", "mean"),
    total_purchases=("amount", "sum"),
    purchase_count=("amount", "count"),
    days_since_last=("date", lambda x: (pd.Timestamp.now() - x.max()).days)
)
df = df.merge(user_stats, on="user_id", how="left")

Aggregation features are some of the most powerful in tabular ML. They inject group-level context into individual-level predictions. Kaggle winners use them relentlessly.

The Feature Engineering Mindset

The question isn't "what transformations exist?" — it's "what would make the pattern obvious if I were a model seeing only numbers?" Every ratio, difference, aggregation, and encoding is you translating domain knowledge into a language the algorithm speaks.

What You Should Now Be Able To Do

  • Diagnose missing data patterns (MCAR/MAR/MNAR) and choose the right imputation strategy
  • Decide when to drop, clip, or transform outliers based on model type
  • Pick the right scaler (Standard/MinMax/Robust) based on your data and model
  • Choose the right categorical encoding (one-hot/ordinal/target/hashing) based on cardinality and data type
  • Engineer interaction, datetime, cyclical, and aggregation features
  • Apply log/power transforms to handle skewed distributions
  • Explain why feature engineering often matters more than model choice
