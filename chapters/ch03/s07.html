Chapter 3: Data Fundamentals Labeling · Augmentation · Validation
Section Overview

This section combines Data Labeling, Data Augmentation and Data Quality & Validation into one comprehensive reference.

Data Labeling

TL;DR

Supervised learning needs labels, and labels are expensive. The quality of your labels sets the ceiling for your model — no architecture trick fixes sloppy annotations. This section covers labeling strategies, quality measurement (inter-annotator agreement), and two production-relevant shortcuts: weak supervision (programmatic labeling) and active learning (labeling the examples that matter most).

Labeling Well

The most underrated step in any labeling project is writing clear annotation guidelines. Without them, ten annotators make ten different decisions on the same ambiguous example. Good guidelines include: positive and negative examples for every class, explicit rules for boundary cases, and a process for escalating disagreements. Spend a day on guidelines before you spend a month on labeling.

Tools

Label Studio — open-source, supports images/text/audio/video. Prodigy — by the spaCy team, NLP-optimized, built-in active learning. CVAT — best for bounding boxes and video tracking. SageMaker Ground Truth — managed workforce with quality controls. Pick based on your data type and whether you need a built-in workforce.

Measuring Agreement

If only one person labels your data, you can't tell whether labels reflect the task or that person's quirks. Have at least two annotators label the same examples and measure consistency.

Cohen's Kappa measures agreement between two annotators, correcting for chance: κ = (p_observed − p_expected) / (1 − p_expected). A κ below 0.60 means your task definition or guidelines need work. Above 0.80 is near-perfect.

from sklearn.metrics import cohen_kappa_score

a1 = ['pos','pos','neg','pos','neg','neg','pos','pos','neg','pos']
a2 = ['pos','pos','neg','neg','neg','neg','pos','pos','neg','pos']
print(f"Cohen's Kappa: {cohen_kappa_score(a1, a2):.3f}")  # 0.800

For 3+ annotators, use Fleiss' Kappa — same intuition, generalized to N raters.

Quality Control

Gold standard examples (honeypots) — items with known answers slipped into the queue. If an annotator gets these wrong, their batch needs review. Adjudication — two annotators label everything; auto-accept agreements, route disagreements to a senior reviewer. This keeps costs down while catching errors.

Weak Supervision — Programs That Label

Manual annotation is accurate but slow. Weak supervision writes labeling functions — small Python functions using heuristics to guess labels. Each function is noisy and incomplete, but combining many noisy functions produces surprisingly good labels.

from snorkel.labeling import labeling_function

SPAM, HAM, ABSTAIN = 1, 0, -1

@labeling_function()
def lf_contains_free(x):
    return SPAM if "free money" in x.text.lower() else ABSTAIN

@labeling_function()
def lf_short_email(x):
    return HAM if len(x.text.split()) < 5 else ABSTAIN

@labeling_function()
def lf_known_sender(x):
    return HAM if x.sender in known_contacts else ABSTAIN

Snorkel learns which functions to trust and produces probabilistic labels for your entire dataset. The trade: you swap label quality for label quantity. For many practical problems — spam, information extraction, medical coding — that tradeoff is worth it. For highly subjective tasks (sarcasm detection), it produces noisy garbage.

Active Learning — Label Smarter, Not Harder

You have a million unlabeled images and budget for 10,000 labels. Which 10,000? Random sampling wastes budget on easy examples the model already handles. Active learning asks the model which examples it's most confused about, and you label those first.

def uncertainty_sample(model, X_unlabeled, n=100):
    probs = model.predict_proba(X_unlabeled)
    uncertainty = 1 - np.max(probs, axis=1)  # highest near 0.5 for binary
    return np.argsort(uncertainty)[-n:]       # most confused examples

The loop: train → score unlabeled pool → label the most uncertain → retrain → repeat. Teams using active learning often reach the same performance with 30-50% fewer labels compared to random sampling. The tradeoff: you need infrastructure for the train/score/label/retrain loop.

Data Augmentation

TL;DR

Data augmentation creates new training examples by applying label-preserving transformations. It acts as a regularizer — forcing models to learn invariances rather than memorize surface patterns. In low-data regimes, it's often the single biggest lever before reaching for a larger model. The constraint: the transformation must preserve the label, which requires domain knowledge.

Why It Works

A model trained on 1,000 cat photos memorizes "tabby on blue couch." Flip the image — it hesitates. It learned the background, not the cat. Augmentation forces the model to find features that survive transformations: ears, whiskers, body shape. Mathematically, it smooths the loss landscape like dropout or weight decay, but from the data side.

Image Augmentation

import torchvision.transforms as T

train_transform = T.Compose([
    T.RandomResizedCrop(224, scale=(0.08, 1.0)),
    T.RandomHorizontalFlip(p=0.5),
    T.RandomRotation(degrees=15),
    T.ColorJitter(brightness=0.4, contrast=0.4, saturation=0.4, hue=0.1),
    T.ToTensor(),
    T.Normalize(mean=[0.485, 0.456, 0.406], std=[0.229, 0.224, 0.225]),
])

# NEVER augment validation/test — it must reflect real distribution
val_transform = T.Compose([
    T.Resize(256), T.CenterCrop(224), T.ToTensor(),
    T.Normalize(mean=[0.485, 0.456, 0.406], std=[0.229, 0.224, 0.225]),
])

Albumentations is faster than torchvision for complex pipelines and supports bounding box / mask transforms for detection and segmentation tasks.

Advanced Mixing Strategies

CutMix — cuts a patch from one image and pastes it onto another, mixing labels proportionally. MixUp — blends two images and their labels with a weighted average. Both are strong regularizers that reduce overconfidence. Cutout / Random Erasing — masks random patches, forcing the model not to depend on any single region.

Text Augmentation

TechniqueHowWhen
Back-translationTranslate to another language and backBest quality. Slow. Use for critical datasets.
Synonym replacementSwap words with WordNet synonymsFast. Good for simple classification.
Random insertion / deletion / swapRandomly modify a few wordsEasy baseline. Can break meaning if overdone.
LLM paraphrasingAsk GPT/Claude to rephraseHighest quality but most expensive.

Tabular Augmentation

Less mature than image/text, but still useful. SMOTE (covered in Section 5) synthesizes minority-class examples by interpolating between neighbors. Gaussian noise injection adds small random perturbations to numeric features. Feature-wise dropout randomly zeros out features during training. All of these act as regularizers for tabular models.

Test-Time Augmentation (TTA)

# Cheap accuracy boost at inference: average predictions over multiple augmented copies
predictions = []
for _ in range(5):
    augmented = train_transform(image)  # different random augmentation each time
    pred = model(augmented.unsqueeze(0))
    predictions.append(pred)
final_pred = torch.stack(predictions).mean(dim=0)

TTA typically gives a 1-2% accuracy improvement for free. Use it when inference latency isn't critical.

The One Rule

Every augmentation choice requires domain knowledge. Flipping a car horizontally? Fine. Rotating a handwritten "6" by 180°? You just turned it into a "9." The augmentation must preserve the label. If you're not sure, don't apply it.

Data Quality & Validation

TL;DR

Bad data doesn't throw errors — your pipeline runs, your model trains, it just quietly gets worse. Data validation catches problems before they silently ruin predictions. This section covers validation frameworks (Great Expectations, Pandera), schema enforcement, drift detection, and how to treat data tests as first-class citizens alongside unit tests.

Why Data Quality Beats Model Complexity

A team spends three months on an elaborate model. Deploys it. Accuracy craters within two weeks. Root cause: a column silently changed format upstream. The model was fine. The data wasn't. And nobody had a check in place to catch it.

Fixing a data issue at ingestion costs almost nothing — a schema check, a range assertion. Fixing it after a model has been retrained on corrupted data and deployed? That costs incident response, rollbacks, and lost trust. Catch it early.

Great Expectations

Write declarative "expectations" about what your data should look like. Think unit tests for DataFrames.

import great_expectations as gx

context = gx.get_context()

# Write expectations
validator.expect_column_values_to_not_be_null("user_id")
validator.expect_column_values_to_be_between("age", min_value=0, max_value=150)
validator.expect_column_values_to_be_in_set("status", ["active", "inactive", "pending"])
validator.expect_table_row_count_to_be_between(min_value=1000, max_value=500000)

# Run validation — returns pass/fail with diagnostics
results = validator.validate()

Expectations are stored as JSON (version-controllable). Checkpoints tie data sources to expectation suites and run in production — trigger a Slack alert or fail a pipeline step when checks fail.

Pandera — Lightweight Schema Validation

import pandera as pa

schema = pa.DataFrameSchema({
    "age": pa.Column(int, pa.Check.in_range(0, 150)),
    "income": pa.Column(float, pa.Check.greater_than(0)),
    "status": pa.Column(str, pa.Check.isin(["active", "inactive"])),
})

validated_df = schema.validate(df)  # raises SchemaError on failure

Pandera is lighter than Great Expectations — perfect for data validation inside a pipeline without the full GX infrastructure. Use GX when you need HTML reports, checkpoints, and team-wide data documentation. Use Pandera when you just need a schema guard.

Drift Detection

Data drift is when the statistical properties of your data change over time. Customer behavior shifts, a sensor degrades, a data provider changes their schema. Your model was trained on the old distribution and quietly gets worse on the new one.

Detecting Drift

MethodWhat It MeasuresWhen to Use
KS TestMax difference between two CDFsNumeric features, sensitive to distribution shape changes
PSI (Population Stability Index)Divergence between binned distributionsStandard in finance/credit scoring. PSI > 0.25 = significant drift.
Chi-square testDifference in categorical distributionsCategorical features
 
from scipy.stats import ks_2samp

# Compare training distribution to production distribution
stat, p_value = ks_2samp(train_df["age"], prod_df["age"])
if p_value < 0.05:
    print(f"Drift detected in 'age' (KS stat={stat:.3f}, p={p_value:.4f})")

Data Testing in CI/CD

Treat data tests like code tests. Run them in your pipeline before training starts.

# In your training pipeline (pseudocode)
def train_pipeline():
    df = load_data()
    
    # Data tests — fail fast if data is broken
    assert len(df) > 1000, f"Too few rows: {len(df)}"
    assert df["target"].isnull().sum() == 0, "Null targets"
    assert df["age"].between(0, 150).all(), "Age out of range"
    
    # Schema validation
    schema.validate(df)
    
    # Drift check against reference distribution
    check_drift(reference_df, df)
    
    # Only then: preprocess and train
    model = train(df)
