Data Splitting Strategies

Chapter 3: Data Fundamentals Section 7
TL;DR

How you split your data determines whether your performance estimates are trustworthy. A careless split can make a mediocre model look brilliant and a brilliant model look mediocre. This section covers train/validation/test splits, stratified splitting for imbalanced classes, time-based splitting for temporal data, group-based splitting for clustered observations, and the leakage traps that make all of this harder than it sounds.

Why Splitting Matters

The entire point of machine learning is to make predictions on data the model hasn't seen. If you train on all your data and then evaluate on the same data, you're testing memory, not learning. The model could have memorized every example and still look perfect. This is why we split: to simulate "unseen data" before the model encounters the real world.

But splitting isn't just about randomly chopping your dataset. The how of splitting encodes assumptions about what "unseen" means. A random split assumes that future data looks statistically identical to past data. A time-based split assumes the future is different from the past. A group-based split assumes you'll encounter entirely new groups. Choose the wrong split strategy, and your evaluation is a comforting fiction.

Train / Validation / Test Split

The standard three-way split serves three distinct purposes. The training set is what the model learns from — it sees these examples during fitting. The validation set is for tuning hyperparameters and making modeling decisions (which algorithm, how many layers, what regularization strength). The test set is your final, unbiased performance estimate. You touch it once, at the very end, and you report that number honestly.

Common ratios are 70/15/15 or 80/10/10 (train/validation/test). With very large datasets (millions of rows), you can shrink the validation and test sets to 1-5% each and still have enough samples for reliable estimates. With small datasets (thousands of rows), you might skip the validation set entirely and use cross-validation instead.

from sklearn.model_selection import train_test_split

# Two-step split: first separate test set, then split remainder into train/val
X_temp, X_test, y_temp, y_test = train_test_split(
    X, y, test_size=0.15, random_state=42
)
X_train, X_val, y_train, y_val = train_test_split(
    X_temp, y_temp, test_size=0.176, random_state=42  # 0.176 of 0.85 ≈ 0.15 of total
)

print(f"Train: {len(X_train)}, Val: {len(X_val)}, Test: {len(X_test)}")

A critical rule: once you look at test set results and then go back to change your model, the test set is no longer unbiased. It's become a second validation set. In Kaggle competitions, this is why the public leaderboard (which you can probe repeatedly) and the private leaderboard (revealed only at the end) often tell different stories. Guard your test set like a sealed envelope.

Stratified Splitting

Random splits work fine when your classes are balanced. But when 95% of your data belongs to class A and 5% to class B, a random split can give you a validation set that's 98% class A and 2% class B, or even a fold with zero class B examples. Your performance estimate becomes wildly unstable.

Stratified splitting preserves the class distribution in every split. If the original data is 95/5, each resulting set will also be approximately 95/5. This gives you more reliable performance estimates, especially for the minority class.

from sklearn.model_selection import train_test_split, StratifiedKFold

# Stratified train/test split
X_train, X_test, y_train, y_test = train_test_split(
    X, y, test_size=0.2, stratify=y, random_state=42
)

# Verify class proportions are preserved
import numpy as np
print(f"Original class distribution: {np.bincount(y) / len(y)}")
print(f"Train class distribution:    {np.bincount(y_train) / len(y_train)}")
print(f"Test class distribution:     {np.bincount(y_test) / len(y_test)}")

# Stratified K-Fold cross-validation
skf = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)
for fold, (train_idx, val_idx) in enumerate(skf.split(X, y)):
    print(f"Fold {fold}: train={len(train_idx)}, val={len(val_idx)}, "
          f"val_class_1_ratio={y[val_idx].mean():.3f}")

Always use stratified splitting for classification problems. There's virtually no downside and it prevents the silent failure of having under-represented classes in a fold. For regression, you can stratify on binned target values to ensure each fold has a similar distribution of target values.

Time-Based Splitting

Imagine you're predicting stock prices. You train on data from 2020-2023 and test on a random 20% sample. Some of your test points are from 2021, sandwiched between training points from 2020 and 2022. Your model can effectively interpolate — peeking at the future to predict the past. The accuracy looks amazing. It's also completely fake.

Time-based splitting respects the arrow of time. The training set contains only data from before a cutoff date. The validation set contains data after the training period but before the test period. The test set is the most recent data. This mimics how the model will actually be used: predicting the future from the past.

import pandas as pd

# Sort by time
df = df.sort_values('date')

# Fixed cutoff split
train_end = '2023-01-01'
val_end = '2023-07-01'

train = df[df['date'] < train_end]
val = df[(df['date'] >= train_end) & (df['date'] < val_end)]
test = df[df['date'] >= val_end]

print(f"Train: {train['date'].min()} to {train['date'].max()} ({len(train)} rows)")
print(f"Val:   {val['date'].min()} to {val['date'].max()} ({len(val)} rows)")
print(f"Test:  {test['date'].min()} to {test['date'].max()} ({len(test)} rows)")

For cross-validation with time series, use an expanding or sliding window approach. Sklearn's TimeSeriesSplit implements this: each fold uses all data up to a point for training and the next chunk for validation. The window moves forward but never backward.

from sklearn.model_selection import TimeSeriesSplit

tscv = TimeSeriesSplit(n_splits=5)
for fold, (train_idx, val_idx) in enumerate(tscv.split(X)):
    print(f"Fold {fold}: train indices {train_idx[0]}-{train_idx[-1]}, "
          f"val indices {val_idx[0]}-{val_idx[-1]}")
# Fold 0: train 0-199, val 200-399
# Fold 1: train 0-399, val 400-599
# ... and so on — training set grows, validation window slides forward

Use time-based splitting whenever your data has a temporal dimension and you're predicting future events. This includes financial forecasting, demand prediction, user churn, and any scenario where the deployment setting is "predict what happens next."

Group-Based Splitting

Sometimes rows in your dataset aren't independent. Medical records from the same patient, transactions from the same user, images from the same camera — these are correlated. If patient 42's records appear in both training and test sets, the model can learn patient-specific patterns and appear to generalize when it's really just recognizing familiar individuals.

Group-based splitting ensures that all records from the same group end up in the same split. If patient 42 is in the training set, all of patient 42's records are in the training set. None leak into validation or test.

from sklearn.model_selection import GroupShuffleSplit, GroupKFold

# Group-aware train/test split
groups = df['patient_id']  # the grouping variable
gss = GroupShuffleSplit(n_splits=1, test_size=0.2, random_state=42)
train_idx, test_idx = next(gss.split(X, y, groups))

# Verify no group overlap
train_groups = set(groups[train_idx])
test_groups = set(groups[test_idx])
assert train_groups.isdisjoint(test_groups), "Group leakage detected!"
print(f"Train groups: {len(train_groups)}, Test groups: {len(test_groups)}")

# Group K-Fold cross-validation
gkf = GroupKFold(n_splits=5)
for fold, (train_idx, val_idx) in enumerate(gkf.split(X, y, groups)):
    train_g = set(groups[train_idx])
    val_g = set(groups[val_idx])
    print(f"Fold {fold}: {len(train_g)} train groups, {len(val_g)} val groups, "
          f"overlap: {len(train_g & val_g)}")  # should always be 0

The key question to ask yourself: "In production, will I see data from groups that were in my training set, or entirely new groups?" If you'll see new groups (new patients, new users, new stores), use group-based splitting. If the same groups will recur, standard splitting might be fine — but group-based splitting gives you a more conservative and often more honest estimate.

Data Leakage Prevention in Splitting

Data leakage occurs when information from outside the training set contaminates the model during training. It's the single most common source of unrealistically good results in machine learning. Your model reports 99% accuracy in development. You deploy it. It gets 70%. The gap is leakage.

The most insidious form of leakage comes from preprocessing. Consider this workflow: you standardize all your features (compute mean and standard deviation), then split into train and test. The problem? The mean and standard deviation were computed using test data. Your training features now contain information about the test set — specifically, its statistical properties. This is train-test contamination.

from sklearn.pipeline import Pipeline
from sklearn.preprocessing import StandardScaler
from sklearn.linear_model import LogisticRegression
from sklearn.model_selection import cross_val_score

# WRONG: fitting scaler on all data before splitting
scaler = StandardScaler()
X_scaled = scaler.fit_transform(X)  # leaks test info into training
scores = cross_val_score(LogisticRegression(), X_scaled, y, cv=5)

# RIGHT: use a Pipeline so scaling happens inside each CV fold
pipe = Pipeline([
    ('scaler', StandardScaler()),
    ('model', LogisticRegression())
])
scores = cross_val_score(pipe, X, y, cv=5)
# Now each fold fits the scaler only on that fold's training data
⚠️ Common Leakage Sources

Feature computation leakage: Computing statistics (means, frequencies) using the entire dataset including test rows. Target leakage: Using a feature that is computed from or is a proxy for the target variable — e.g., using "was_treated" to predict "has_disease" when treatment was given because of the disease. Temporal leakage: Using future information to predict past events. Duplicate leakage: Near-duplicate rows landing in both train and test. Always ask: "Could my model know this at prediction time?"

The pipeline pattern is your primary defense. Every transformation — scaling, encoding, imputation, feature selection — should be fitted only on training data and then applied to validation and test data. Sklearn's Pipeline enforces this automatically when used with cross-validation or grid search. Make it a habit to put every preprocessing step inside a pipeline. The day this habit saves you from a leakage bug will be worth all the extra code.

from sklearn.pipeline import Pipeline
from sklearn.compose import ColumnTransformer
from sklearn.preprocessing import StandardScaler, OneHotEncoder
from sklearn.impute import SimpleImputer
from sklearn.ensemble import RandomForestClassifier
from sklearn.model_selection import cross_val_score

# Full leak-proof pipeline
numeric_features = ['age', 'income', 'score']
categorical_features = ['city', 'education']

preprocessor = ColumnTransformer([
    ('num', Pipeline([
        ('imputer', SimpleImputer(strategy='median')),
        ('scaler', StandardScaler())
    ]), numeric_features),
    ('cat', Pipeline([
        ('imputer', SimpleImputer(strategy='most_frequent')),
        ('encoder', OneHotEncoder(handle_unknown='ignore'))
    ]), categorical_features)
])

full_pipeline = Pipeline([
    ('preprocess', preprocessor),
    ('model', RandomForestClassifier(n_estimators=100, random_state=42))
])

# Cross-validation with zero leakage
scores = cross_val_score(full_pipeline, X, y, cv=5, scoring='f1')
print(f"F1 scores: {scores}")
print(f"Mean F1: {scores.mean():.4f} ± {scores.std():.4f}")

A final thought: if your model seems too good to be true, it probably is. Suspiciously high accuracy is the leading symptom of data leakage. Before celebrating, check your splitting strategy, your preprocessing pipeline, and your features for any path that lets future information flow backward into the training process.
