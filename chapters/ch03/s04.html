Data Cleaning

Chapter 3: Data Fundamentals 6 subtopics
TL;DR

Real-world data is messy — missing values, outliers, wrong types, duplicates, and inconsistent text. Data cleaning is the unglamorous work that determines whether your model learns signal or noise. This section walks through each category of mess, shows you how to detect it, and gives you concrete pandas and scikit-learn code to fix it. There is no single right answer; every cleaning decision is a tradeoff between losing information and introducing bias.

Handling Missing Values

Before you impute anything, you need to understand why the data is missing. The reason matters because it determines which fix is safe and which one silently poisons your model.

Types of Missingness

Missing Completely at Random (MCAR) means the missingness has nothing to do with any variable in your dataset. A sensor glitched, a survey page failed to load — pure bad luck. If your data is MCAR, dropping those rows won't bias your results. You just lose some statistical power.

Missing at Random (MAR) means the missingness depends on other observed variables, but not on the missing value itself. For example, younger respondents skip the income question more often. The missingness relates to age (which you observe), not to income itself. You can correct for this with imputation techniques that condition on the observed variables.

Missing Not at Random (MNAR) is the dangerous one. Here the missingness depends on the unobserved value itself. High earners refuse to report income. Sick patients drop out of clinical trials. No statistical trick fully fixes MNAR — you need domain knowledge, and you should be honest in your reporting that bias may remain.

In practice, you rarely know for certain which type you're dealing with. Start by visualizing your missing data patterns.

import pandas as pd
import missingno as msno

df = pd.read_csv("patient_records.csv")

# Quick count of missing values per column
print(df.isnull().sum())
print(f"\n% missing per column:\n{df.isnull().mean() * 100}")

# Visual pattern — look for correlated missingness
msno.matrix(df, figsize=(10, 4), fontsize=8)
# If you see columns that go missing together, that's a hint it's MAR or MNAR

Dropping Rows or Columns

The simplest fix is deletion. Drop a column if more than 40–60% of its values are missing — at that point imputation is mostly invention. Drop a row if only a handful are incomplete and you have plenty of data. But be careful: if the rows you drop are systematically different from the rows you keep, you've just introduced selection bias.

Imputation with Mean, Median, and Mode

Imputation means filling in missing values with plausible substitutes. The simplest approach: replace missing numeric values with the column's mean or median, and missing categorical values with the mode (most frequent value). Mean imputation is fast but shrinks variance and distorts correlations. Median is more robust to skewed distributions. Mode works for categories but can overrepresent the dominant class.

import pandas as pd
import numpy as np

df = pd.DataFrame({
    "age": [25, np.nan, 34, 45, np.nan, 29],
    "salary": [50000, 60000, np.nan, 80000, 55000, np.nan],
    "department": ["eng", "sales", np.nan, "eng", "sales", "eng"]
})

# Mean imputation for salary (sensitive to outliers)
df["salary_mean"] = df["salary"].fillna(df["salary"].mean())

# Median imputation for age (robust to skew)
df["age_median"] = df["age"].fillna(df["age"].median())

# Mode imputation for categorical column
df["department_mode"] = df["department"].fillna(df["department"].mode()[0])

print(df[["age", "age_median", "salary", "salary_mean", "department", "department_mode"]])

Forward Fill, Backward Fill, and Interpolation

Forward fill carries the last observed value forward; backward fill copies the next observed value backward. These make sense for time-series data where values change slowly — yesterday's stock price is a reasonable stand-in for a missing today. They make no sense for cross-sectional data where row order is arbitrary.

Interpolation fits a curve (linear by default) between known points and fills in the gaps. It's a natural choice for sensor data or any series with a smooth underlying trend. Pandas gives you df["col"].interpolate(method="linear") — you can also pass method="spline" or method="polynomial" for non-linear patterns.

KNN Imputation

KNN imputation fills each missing value by averaging the values from its k nearest neighbors in feature space. Unlike mean imputation, it respects the relationships between columns. A 30-year-old engineer's missing salary gets imputed from other 30-ish engineers, not from the global average that includes executive salaries. The tradeoff: it's slower and sensitive to the scale of your features, so always standardize first.

from sklearn.impute import KNNImputer
from sklearn.preprocessing import StandardScaler
import pandas as pd
import numpy as np

df = pd.DataFrame({
    "age": [25, 30, np.nan, 45, 50, 28, np.nan, 38],
    "experience": [2, 5, 7, 20, 25, 3, 10, np.nan],
    "salary": [50000, 60000, 65000, np.nan, 95000, 52000, 70000, 75000]
})

# Scale first — KNN is distance-based
scaler = StandardScaler()
df_scaled = pd.DataFrame(scaler.fit_transform(df), columns=df.columns)

# Impute with 3 nearest neighbors
imputer = KNNImputer(n_neighbors=3)
df_imputed = pd.DataFrame(imputer.fit_transform(df_scaled), columns=df.columns)

# Inverse transform to original scale
df_result = pd.DataFrame(scaler.inverse_transform(df_imputed), columns=df.columns)
print(df_result.round(1))

Iterative Imputation (MICE)

Iterative imputation, also called MICE (Multiple Imputation by Chained Equations), treats each column with missing values as a prediction target. It builds a regression model using all other columns, imputes the missing values, then moves to the next column, cycling through multiple rounds until the values stabilize. This is the gold standard for MAR data — it captures complex inter-column relationships. The cost: it's computationally heavier and you need enough observed data to train those internal models.

Adding Missing Indicator Features

Sometimes the fact that a value is missing carries information. If patients with severe conditions skip follow-up appointments, the missingness itself is predictive. Adding a binary column — salary_was_missing = df["salary"].isnull().astype(int) — lets your model learn from the missingness pattern alongside the imputed value.

When to Drop vs. Impute

Here's a practical decision framework. If fewer than 5% of rows are affected and the data looks MCAR, just drop. If a column is more than half empty, drop the column. If you're in between, impute — use median for skewed numerics, KNN or MICE when column relationships matter, and mode for low-cardinality categories. Always add a missing indicator if you suspect MNAR. And never impute the target variable — a model trained on fabricated labels learns fiction.

Handling Outliers

An outlier is a data point that sits far from the rest. Some outliers are errors — a typo turned $50,000 into $5,000,000. Others are real but rare — a genuine billionaire in a salary dataset. Your job is to figure out which is which, and your response should differ accordingly.

Detecting Outliers: IQR Method

The Interquartile Range (IQR) method is the workhorse. Compute Q1 (25th percentile) and Q3 (75th percentile). The IQR is Q3 − Q1. Anything below Q1 − 1.5 × IQR or above Q3 + 1.5 × IQR is flagged. This method makes no assumptions about distribution shape, so it works well for skewed data.

import pandas as pd
import numpy as np

np.random.seed(42)
df = pd.DataFrame({
    "salary": np.concatenate([
        np.random.normal(60000, 15000, 200),
        [500000, -10000, 950000]  # obvious outliers
    ])
})

Q1 = df["salary"].quantile(0.25)
Q3 = df["salary"].quantile(0.75)
IQR = Q3 - Q1

lower = Q1 - 1.5 * IQR
upper = Q3 + 1.5 * IQR

outliers = df[(df["salary"] < lower) | (df["salary"] > upper)]
print(f"IQR: {IQR:,.0f}, Bounds: [{lower:,.0f}, {upper:,.0f}]")
print(f"Outliers found: {len(outliers)}")
print(outliers.head())

Z-Score Method

The Z-score measures how many standard deviations a point is from the mean. Flag anything with |Z| > 3. This works cleanly for roughly normal data but breaks down for skewed distributions — the mean and standard deviation themselves get pulled by the outliers you're trying to find.

Visual Detection

Numbers alone can mislead you. Box plots show you the IQR, whiskers, and individual outlier points in one glance. Scatter plots reveal outliers that only appear in combination — a 22-year-old with 30 years of experience is fine on each axis alone, but absurd together. Always plot before you delete.

Domain Knowledge for Outlier Detection

No statistical method replaces domain knowledge. If you know human ages can't exceed ~120, a value of 999 is clearly a placeholder, not an outlier. If you know a stock can gap up 50% on earnings day, that point is real and important. Talk to the people who generated the data before you throw anything away.

Removing, Capping, and Transforming

You have three main responses to outliers. Removal is appropriate for clear errors. Capping (also called winsorizing) clips extreme values to a boundary — say the 1st and 99th percentiles. This preserves the row while limiting the outlier's influence. Log transformation compresses the right tail of skewed distributions, making extreme values less extreme without removing them. Use np.log1p() (log of 1+x) to handle zeros safely.

import numpy as np
import pandas as pd

# Capping / Winsorizing at 1st and 99th percentile
lower_cap = df["salary"].quantile(0.01)
upper_cap = df["salary"].quantile(0.99)
df["salary_capped"] = df["salary"].clip(lower=lower_cap, upper=upper_cap)

# Log transformation for right-skewed data
df["salary_log"] = np.log1p(df["salary"].clip(lower=0))  # clip negatives first

print(f"Before: std={df['salary'].std():,.0f}")
print(f"Capped: std={df['salary_capped'].std():,.0f}")
print(f"Log:    std={df['salary_log'].std():.2f}")

When to Keep Outliers

Fraud detection, anomaly detection, rare-event prediction — in these domains the outliers are the signal. Removing them means removing the very thing you're trying to model. Even in standard regression, blindly trimming outliers to improve R² is dishonest. If the outlier is a real observation, your model should be robust enough to handle it, or you should use a model that is — like a tree-based method that's naturally resistant to extreme values.

Data Type Corrections

Converting Strings to Numeric

CSV files don't carry type information. A column that looks numeric might be stored as strings because one row contains "N/A" or "$50,000". Use pd.to_numeric(df["col"], errors="coerce") to convert, turning unparseable values into NaN so you can handle them with your imputation pipeline.

Parsing Dates

Dates arrive in dozens of formats: "2024-01-15", "01/15/2024", "Jan 15, 2024", "15-Jan-24". Use pd.to_datetime(df["date_col"], format="mixed") to let pandas guess, or pass an explicit format string for consistency. Once parsed, you can extract useful features — day of week, month, quarter, days since some reference date.

Fixing Inconsistent Categories

The same category entered three ways — "New York", "new york", "NY" — creates three separate classes where there should be one. Lowercase everything, strip whitespace, and build a mapping dictionary for known variants. For messier cases, fuzzy matching libraries like fuzzywuzzy can catch "Callifornia" as a misspelling of "California".

Handling Mixed Types in Columns

A single column sometimes contains both numbers and strings — "42", "unknown", "37", "—". The fix is to coerce to one type, pushing the rest to NaN: pd.to_numeric(df["col"], errors="coerce"). But first, check if those strings carry meaning. If "unknown" is a deliberate entry, you may want to preserve it in a separate indicator column before coercing.

Handling Duplicates

Identifying Exact Duplicates

An exact duplicate is a row that matches another row on every single column. These sneak in through double-clicks on submit buttons, overlapping data exports, or careless joins. They inflate your dataset and bias your model toward the duplicated examples.

import pandas as pd

df = pd.DataFrame({
    "name": ["Alice", "Bob", "Alice", "Carol", "Bob"],
    "email": ["a@x.com", "b@x.com", "a@x.com", "c@x.com", "b@x.com"],
    "signup": ["2024-01-01", "2024-01-02", "2024-01-01", "2024-01-03", "2024-01-02"]
})

# Find duplicates
print(f"Duplicate rows: {df.duplicated().sum()}")
print(df[df.duplicated(keep=False)])  # show all copies, not just the "extra" ones

# Drop duplicates, keep first occurrence
df_clean = df.drop_duplicates(keep="first")
print(f"\nAfter dedup: {len(df_clean)} rows")

Near-Duplicate Detection

Near-duplicates are rows that aren't identical but clearly refer to the same entity. "Robert Smith" and "Rob Smith" at the same address. "iPhone 14 Pro" and "Apple iPhone 14 Pro". Catching these requires fuzzy matching — compare strings with edit distance or token overlap, then set a similarity threshold. For large datasets, blocking strategies (only compare records that share a zip code, for example) keep the computation tractable.

Deciding What to Drop

Not all duplicates are mistakes. A customer buying the same product twice is a real repeat purchase, not a duplicate. Before deduplication, define what makes a record "the same" — which columns form a natural key. Then decide which copy to keep: the most recent, the most complete, or the first one recorded.

Text Data Cleaning

Trimming, Casing, and Special Characters

Text data needs normalization before it's useful. Leading and trailing whitespace breaks exact matches — "Alice " ≠ "Alice". Mixed case creates false distinctions. Special characters (em-dashes, smart quotes, zero-width spaces) lurk invisibly and cause havoc. A basic cleaning pass covers all three: df["name"] = df["name"].str.strip().str.lower().str.replace(r"[^\w\s]", "", regex=True).

Standardizing Formats

Phone numbers, postal codes, and addresses are especially messy. "(555) 123-4567", "555-123-4567", and "5551234567" are the same phone number. Strip to digits first, then reformat consistently. For addresses, consider a dedicated library — standardizing "St." vs "Street" vs "ST" by hand is a losing battle. The key principle: normalize early so that every downstream step sees clean, consistent inputs.

Model-Based Outlier Detection

When your data has many dimensions, simple statistical methods like IQR or Z-scores struggle. A point can look normal on every individual feature yet be wildly anomalous in combination. Model-based methods catch these multivariate outliers.

Isolation Forest

Isolation Forest works on a clever insight: outliers are easier to isolate. The algorithm builds random decision trees that split features at random values. Normal points, surrounded by similar neighbors, require many splits to isolate. Outliers, sitting alone in sparse regions, get isolated in just a few splits. The average number of splits needed becomes the anomaly score.

from sklearn.ensemble import IsolationForest
import pandas as pd
import numpy as np

np.random.seed(42)
normal = np.random.normal(0, 1, (200, 2))
outliers = np.array([[8, 8], [-7, -7], [9, -6]])
X = pd.DataFrame(np.vstack([normal, outliers]), columns=["f1", "f2"])

iso = IsolationForest(contamination=0.02, random_state=42)
X["anomaly"] = iso.fit_predict(X[["f1", "f2"]])  # -1 = outlier, 1 = normal

print(f"Outliers detected: {(X['anomaly'] == -1).sum()}")
print(X[X["anomaly"] == -1][["f1", "f2"]])

DBSCAN-Based Outlier Detection

DBSCAN (Density-Based Spatial Clustering of Applications with Noise) is a clustering algorithm, but it naturally identifies outliers as a side effect. Points that don't belong to any cluster get labeled as noise (cluster label −1). This is useful when your data has natural clusters and the outliers are the points that fall between them. The downside: you need to tune two parameters — eps (neighborhood radius) and min_samples — and the results are sensitive to both.

Local Outlier Factor (LOF)

Local Outlier Factor compares the local density around a point to the local density around its neighbors. If a point's neighborhood is much sparser than its neighbors' neighborhoods, it's a local outlier. LOF's strength is that it adapts to varying densities — a point that's normal in a dense cluster can be flagged as anomalous if it sits at the edge of a sparse one. This makes it more nuanced than Isolation Forest for non-uniform data distributions.

Putting It All Together

Cleaning steps aren't independent — they form a pipeline. Fix types first so numeric operations work. Then handle duplicates so you don't impute from copies. Then address missing values, then outliers, then text normalization. Here's a skeleton that ties the key steps together.

import pandas as pd
import numpy as np
from sklearn.impute import KNNImputer
from sklearn.ensemble import IsolationForest
from sklearn.preprocessing import StandardScaler
from sklearn.pipeline import Pipeline

def clean_dataframe(df):
    df = df.copy()

    # 1. Fix types
    for col in ["age", "salary"]:
        df[col] = pd.to_numeric(df[col], errors="coerce")
    if "signup_date" in df.columns:
        df["signup_date"] = pd.to_datetime(df["signup_date"], errors="coerce")

    # 2. Drop exact duplicates
    df = df.drop_duplicates()

    # 3. Normalize text columns
    text_cols = df.select_dtypes(include="object").columns
    for col in text_cols:
        df[col] = df[col].str.strip().str.lower()

    # 4. Add missing indicators before imputation
    num_cols = df.select_dtypes(include=np.number).columns
    for col in num_cols:
        if df[col].isnull().any():
            df[f"{col}_was_missing"] = df[col].isnull().astype(int)

    # 5. Impute numeric columns with KNN
    if df[num_cols].isnull().any().any():
        scaler = StandardScaler()
        scaled = scaler.fit_transform(df[num_cols])
        imputer = KNNImputer(n_neighbors=5)
        imputed = imputer.fit_transform(scaled)
        df[num_cols] = scaler.inverse_transform(imputed)

    # 6. Cap outliers at 1st/99th percentile
    for col in num_cols:
        lo, hi = df[col].quantile(0.01), df[col].quantile(0.99)
        df[col] = df[col].clip(lower=lo, upper=hi)

    return df

# Usage
df_raw = pd.read_csv("messy_data.csv")
df_clean = clean_dataframe(df_raw)
print(f"Rows: {len(df_raw)} -> {len(df_clean)}")
print(f"Remaining nulls: {df_clean.isnull().sum().sum()}")

Data cleaning is never finished — it's an iterative process. Every time you explore your data further, you'll find new problems. Every time you train a model, error analysis will reveal cleaning steps you missed. The key discipline is to keep your cleaning code in a reproducible pipeline, never in ad-hoc notebook cells you can't trace. Future you, debugging a production model at 2 AM, will be grateful.
