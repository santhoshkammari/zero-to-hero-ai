Feature Engineering

Chapter 3: Data Fundamentals Section 5
TL;DR

Raw data rarely arrives in a form that algorithms can learn from efficiently. Feature engineering is the art of transforming your raw columns into representations that make patterns obvious to models. This covers numerical transforms (scaling, log, polynomial, binning), categorical encoding (one-hot, label, target, ordinal, binary, frequency, hashing), text features (BoW, TF-IDF, n-grams, embeddings), datetime cyclical encoding, interaction features, domain-specific tricks, and automated tools like Featuretools. Good features often matter more than fancy models.

Why Feature Engineering Matters

Here's an uncomfortable truth about machine learning: the algorithm you pick matters less than the features you feed it. A logistic regression with brilliantly engineered features will crush a deep neural network eating raw garbage. Andrew Ng once said that applied machine learning is "basically feature engineering." He wasn't exaggerating.

Feature engineering is the process of using domain knowledge and data transformations to create input variables that make machine learning algorithms work better. It's the bridge between "I have data" and "my model actually learns something useful." You're not changing the data — you're changing how you present it so that the statistical patterns become easier for a model to detect.

Consider predicting house prices. You have square footage of the lot and square footage of the house. Individually, each is useful. But the ratio — how much of the lot the house covers — might be far more predictive of price in dense urban neighborhoods. That ratio didn't exist in your raw data. You engineered it.

Numerical Transformations

Feature Scaling

Many algorithms — particularly those based on distances (k-NN, SVM) or gradient descent (linear regression, neural networks) — are sensitive to the scale of input features. If one feature ranges from 0 to 1 and another ranges from 0 to 1,000,000, the large-range feature will dominate. The model will barely notice the small one. Feature scaling fixes this by putting all features on a comparable range.

There are three main flavors. Standardization (z-score scaling) subtracts the mean and divides by the standard deviation, centering features around zero with unit variance. Min-max scaling squishes everything into [0, 1]. Robust scaling uses the median and interquartile range instead of mean and standard deviation, making it resilient to outliers.

import numpy as np
import pandas as pd
from sklearn.preprocessing import StandardScaler, MinMaxScaler, RobustScaler

X = pd.DataFrame({
    'age': [25, 30, 35, 60, 45],
    'income': [30000, 50000, 70000, 200000, 90000]
})

# Standardization: mean=0, std=1
standard = StandardScaler()
X_standard = standard.fit_transform(X)
# age and income now both centered at 0

# Min-Max: squeeze to [0, 1]
minmax = MinMaxScaler()
X_minmax = minmax.fit_transform(X)

# Robust: resistant to outliers (uses median + IQR)
robust = RobustScaler()
X_robust = robust.fit_transform(X)
# That 200k income outlier won't distort the scaling

When do you scale? Always before distance-based or gradient-based models. When do you not scale? Tree-based models (random forest, XGBoost) split on thresholds and don't care about magnitude — scaling won't help and can actually obscure interpretability.

Log and Power Transforms

Real-world data loves to be skewed. Income distributions, website traffic, word frequencies — they all have long right tails. Most values cluster low, with a few extreme outliers. This skewness confuses linear models that expect roughly symmetric, bell-shaped distributions.

A log transformation compresses the tail and stretches the lower range. Income of $30k, $50k, and $1M becomes 10.3, 10.8, and 13.8 in log space — much more manageable. For data with zeros, use np.log1p(x) which computes log(1 + x).

Power transforms generalize this idea. The Box-Cox transform finds the optimal power parameter λ that makes your data as close to normal as possible, but it only works on strictly positive values. The Yeo-Johnson transform handles negative values too.

from sklearn.preprocessing import PowerTransformer

# Box-Cox (requires strictly positive data)
pt_bc = PowerTransformer(method='box-cox')
X_bc = pt_bc.fit_transform(X[['income']])

# Yeo-Johnson (handles zeros and negatives)
pt_yj = PowerTransformer(method='yeo-johnson')
X_yj = pt_yj.fit_transform(X[['income']])

Polynomial Features

Linear models can only learn linear relationships. If the true relationship between a feature and your target is quadratic or involves interactions, a straight line won't fit. Polynomial features solve this by creating new columns that are powers and products of existing features.

With two features [a, b] and degree=2, you get: [a, b, a², ab, b²]. Now a linear model can learn curved boundaries.

from sklearn.preprocessing import PolynomialFeatures

X_small = pd.DataFrame({'x1': [1, 2, 3], 'x2': [4, 5, 6]})

poly = PolynomialFeatures(degree=2, include_bias=False)
X_poly = poly.fit_transform(X_small)
print(poly.get_feature_names_out())
# ['x1', 'x2', 'x1^2', 'x1 x2', 'x2^2']

The tradeoff: polynomial features explode combinatorially. With 50 features and degree 3, you'd create thousands of new columns. This invites overfitting. Use interaction_only=True to limit to cross-products without powers, or apply regularization afterward.

Binning and Discretization

Sometimes continuous values matter less than the range they fall in. A customer's exact age (34.7 years) is noise. Whether they're in their 20s, 30s, or 40s might be what actually drives behavior. Binning (also called discretization) converts a continuous feature into categorical buckets.

from sklearn.preprocessing import KBinsDiscretizer

# Equal-width bins
binner = KBinsDiscretizer(n_bins=4, encode='ordinal', strategy='uniform')
X['age_binned'] = binner.fit_transform(X[['age']])

# Quantile-based bins (equal number of samples per bin)
binner_q = KBinsDiscretizer(n_bins=4, encode='ordinal', strategy='quantile')
X['age_quantile'] = binner_q.fit_transform(X[['age']])

Binning throws away information. That's the cost. The benefit: it can capture non-linear effects in linear models, reduces the impact of outliers, and sometimes matches how domain experts think about the data. Use it deliberately, not by default.

Categorical Encoding

Machine learning models eat numbers. They can't digest strings like "red", "blue", or "Berlin" directly. Categorical encoding translates category labels into numerical representations that algorithms can process. The encoding you choose dramatically affects model performance — this isn't a boring preprocessing step, it's a modeling decision.

One-Hot Encoding

One-hot encoding creates a new binary column for each category. A "color" column with values [red, blue, green] becomes three columns: is_red, is_blue, is_green. Each row has exactly one 1 and the rest are 0s.

from sklearn.preprocessing import OneHotEncoder
import pandas as pd

df = pd.DataFrame({'color': ['red', 'blue', 'green', 'red']})

# sklearn approach
ohe = OneHotEncoder(sparse_output=False, drop='first')  # drop='first' avoids multicollinearity
encoded = ohe.fit_transform(df[['color']])

# pandas shortcut
pd.get_dummies(df, columns=['color'], drop_first=True)

One-hot encoding is safe and universal. It introduces no artificial ordering. The downside: high-cardinality features (a city column with 10,000 unique cities) produce 10,000 new columns, making your feature matrix enormous and sparse. That's where other encodings step in.

Label and Ordinal Encoding

Label encoding assigns each category an integer: red=0, blue=1, green=2. It's compact — one column instead of many. But it introduces a fake ordering. The model might interpret green (2) as "more" than red (0), which is nonsensical for nominal categories.

Ordinal encoding is label encoding with intent. When categories have a natural order (low < medium < high, or elementary < bachelor < master < PhD), the integer mapping preserves real meaning.

from sklearn.preprocessing import LabelEncoder, OrdinalEncoder

# Label encoding (for nominal categories — use cautiously)
le = LabelEncoder()
df['color_label'] = le.fit_transform(df['color'])

# Ordinal encoding (for ordered categories)
edu = pd.DataFrame({'education': ['high_school', 'bachelor', 'master', 'phd']})
oe = OrdinalEncoder(categories=[['high_school', 'bachelor', 'master', 'phd']])
edu['edu_ordinal'] = oe.fit_transform(edu[['education']])

Rule of thumb: use ordinal encoding only when the ordering is real and meaningful. For everything else, prefer one-hot or target encoding.

Target Encoding

Target encoding (also called mean encoding) replaces each category with the average target value for that category. If you're predicting house prices and the average price in neighborhood "Downtown" is $500k, then "Downtown" becomes 500000.

This is powerful because it captures the relationship between the category and the target directly. It handles high cardinality naturally — 10,000 cities become 10,000 numbers in one column, not 10,000 columns.

The danger: target leakage. You're using the target variable to construct a feature. If done naively on the training set, the encoding memorizes noise. The fix: use leave-one-out encoding or fit the encoder only within cross-validation folds.

# Target encoding with sklearn (uses internal cross-fitting to prevent leakage)
from sklearn.preprocessing import TargetEncoder

te = TargetEncoder(smooth='auto')
# smooth parameter regularizes toward global mean for rare categories
X_encoded = te.fit_transform(df[['color']], y=target)

Binary, Frequency, and Hash Encoding

Binary encoding converts category integers to binary representation, then splits each bit into its own column. A feature with 8 categories needs only 3 columns (2³ = 8) instead of 8 for one-hot. It's a compromise between label and one-hot encoding.

Frequency encoding replaces each category with how often it appears in the dataset. If "red" appears in 40% of rows, it becomes 0.4. Simple, no leakage risk, and it captures the intuition that rare categories might behave differently than common ones.

Hash encoding (feature hashing) maps categories to a fixed number of columns using a hash function. You choose the number of output columns (say, 8), and the hash function determines which column each category activates. Collisions happen — two categories might hash to the same column — but in practice, this works surprisingly well for very high cardinality features. It's memory-efficient and doesn't require maintaining a vocabulary.

# Frequency encoding (manual, works great)
freq = df['color'].value_counts(normalize=True)
df['color_freq'] = df['color'].map(freq)

# Hash encoding (using category_encoders library)
# pip install category_encoders
from category_encoders import HashingEncoder
he = HashingEncoder(n_components=4)
X_hashed = he.fit_transform(df[['color']])

Text Features

Bag of Words and TF-IDF

Text is inherently unstructured. To feed it to a model, you need to convert it to numbers. The simplest approach is Bag of Words (BoW): build a vocabulary of all unique words in your corpus, then represent each document as a vector counting how many times each word appears. Word order is lost — "the dog bit the man" and "the man bit the dog" produce similar vectors — but it works remarkably well for many tasks.

TF-IDF (Term Frequency–Inverse Document Frequency) improves on raw counts. It upweights words that are distinctive to a document and downweights words that appear everywhere. The word "the" appears in every document and gets a low score. The word "mitochondria" appears in few documents and gets a high score where it appears.

from sklearn.feature_extraction.text import CountVectorizer, TfidfVectorizer

docs = [
    "the cat sat on the mat",
    "the dog sat on the log",
    "cats and dogs are friends"
]

# Bag of Words
bow = CountVectorizer()
X_bow = bow.fit_transform(docs)
print(bow.get_feature_names_out())
# ['and', 'are', 'cat', 'cats', 'dog', 'dogs', 'friends', 'log', 'mat', 'on', 'sat', 'the']

# TF-IDF
tfidf = TfidfVectorizer()
X_tfidf = tfidf.fit_transform(docs)
# "the" gets low weight (appears in 2/3 docs), "friends" gets high weight (unique to doc 3)

N-grams

N-grams capture word sequences instead of individual words. A bigram (n=2) for "the cat sat" produces: ["the cat", "cat sat"]. This preserves some word order and captures phrases like "not good" (which means something very different from "good" alone).

# Capture unigrams and bigrams
tfidf_ng = TfidfVectorizer(ngram_range=(1, 2), max_features=1000)
X_ngrams = tfidf_ng.fit_transform(docs)
# Now includes features like "the_cat", "cat_sat", etc.

The tradeoff is vocabulary explosion. Bigrams already create far more features than unigrams. Trigrams are worse. Use max_features to cap the vocabulary size and min_df / max_df to filter out extremely rare or extremely common n-grams.

Word Embeddings

Word embeddings represent words as dense vectors (typically 50–300 dimensions) where similar words are close together in the vector space. Unlike BoW (which creates sparse vectors with thousands of dimensions), embeddings capture semantic meaning. "King" and "queen" are close. "King" and "potato" are far apart.

Pre-trained embeddings like Word2Vec, GloVe, and FastText give you these vectors without training on your own data. To represent a document, you can average the embeddings of all its words — crude but effective for many classification tasks. For more sophisticated approaches, models like BERT generate context-dependent embeddings where the same word gets different vectors depending on its sentence.

# Using pre-trained embeddings (conceptual example)
# 1. Load pre-trained vectors (e.g., GloVe 100d)
# 2. For each document, look up each word's vector
# 3. Average the vectors to get a document embedding

import numpy as np

def document_embedding(text, word_vectors, dim=100):
    words = text.lower().split()
    vecs = [word_vectors[w] for w in words if w in word_vectors]
    if not vecs:
        return np.zeros(dim)
    return np.mean(vecs, axis=0)

Date and Time Features

Cyclical Encoding with Sine and Cosine

Dates and times are tricky. You can extract the hour, day of week, month, and so on as plain integers. But this creates a discontinuity problem: hour 23 and hour 0 are one hour apart in reality, but 23 units apart as integers. The model sees them as maximally different when they should be neighbors.

Cyclical encoding solves this by mapping periodic features onto a circle using sine and cosine. Points that are close on the clock or calendar end up close in the transformed space.

import numpy as np

def cyclical_encode(values, period):
    """Encode a periodic feature as sin/cos pair."""
    sin_vals = np.sin(2 * np.pi * values / period)
    cos_vals = np.cos(2 * np.pi * values / period)
    return sin_vals, cos_vals

# Hour of day (period = 24)
hours = pd.Series([0, 6, 12, 18, 23])
df_time = pd.DataFrame()
df_time['hour_sin'], df_time['hour_cos'] = cyclical_encode(hours, 24)

# Day of week (period = 7)
days = pd.Series([0, 1, 2, 3, 4, 5, 6])  # Mon=0 .. Sun=6
df_time['dow_sin'], df_time['dow_cos'] = cyclical_encode(days, 7)

# Month of year (period = 12)
months = pd.Series([1, 6, 12])
df_time['month_sin'], df_time['month_cos'] = cyclical_encode(months, 12)

Now hour 23 and hour 0 have nearly identical sin/cos values — exactly what we want. Always extract plain integer features (hour, day, month) alongside cyclical ones. Some patterns are cyclical (traffic peaks at rush hour every day) and some are linear (sales trend upward over years). Give the model both representations.

Interaction Features

Sometimes the relationship between a feature and the target depends on the value of another feature. Temperature might not predict ice cream sales on its own, but temperature × weekend (a binary flag) captures the idea that hot weekends drive sales more than hot weekdays.

Interaction features are products, ratios, or differences of two or more existing features. They let linear models capture non-additive effects without switching to a more complex algorithm.

# Manual interaction features
df['price_per_sqft'] = df['price'] / df['sqft']           # ratio
df['rooms_x_floor'] = df['rooms'] * df['floor']            # product
df['age_minus_experience'] = df['age'] - df['experience']   # difference

# Automated with PolynomialFeatures (interaction_only=True)
from sklearn.preprocessing import PolynomialFeatures
inter = PolynomialFeatures(degree=2, interaction_only=True, include_bias=False)
X_inter = inter.fit_transform(X[['age', 'income']])
# Creates: [age, income, age*income]

Be selective. Creating all pairwise interactions for 100 features gives you 5,000 new columns. Most will be noise. Use domain knowledge to choose which interactions make sense, or rely on tree-based models that find interactions automatically through their splitting mechanism.

Domain-Specific Engineering

The most powerful features come from understanding your problem domain, not from generic transformations. This is where data science becomes a craft rather than a recipe.

In geospatial problems, you might compute the Haversine distance between a property and the nearest subway station, the count of restaurants within a 500-meter radius, or cluster coordinates into neighborhoods using k-means. In recommendation systems, features like "number of items this user rated in the last 30 days" or "average rating this item received from similar users" capture behavioral patterns that raw user-item IDs can't.

In finance, rolling window statistics (7-day moving average, 30-day volatility) transform time-series data into features that capture trends and seasonality. In NLP, features like sentence length, punctuation density, and readability scores complement the bag-of-words representation.

There's no sklearn function for "insight." Domain features require talking to subject matter experts, reading about the problem space, and looking at your data until you understand what drives the outcome. This is the part of ML that's hardest to automate and most valuable to master.

Automated Feature Engineering

For relational datasets with multiple tables, manually engineering aggregation features (average transaction amount per customer, count of logins per user per week) is tedious and error-prone. Automated feature engineering tools like Featuretools systematize this process.

Featuretools works by defining entities (tables) and relationships between them, then applying transformation and aggregation primitives automatically. It generates hundreds of candidate features like "MEAN(transactions.amount)", "COUNT(logins WHERE device='mobile')", and "MAX(orders.total) - MIN(orders.total)".

# Conceptual Featuretools example
import featuretools as ft

# Define entity set
es = ft.EntitySet(id='ecommerce')
es = es.add_dataframe(dataframe=customers_df, dataframe_name='customers',
                      index='customer_id')
es = es.add_dataframe(dataframe=orders_df, dataframe_name='orders',
                      index='order_id', time_index='order_date')
es = es.add_relationship('customers', 'customer_id', 'orders', 'customer_id')

# Deep Feature Synthesis — auto-generates features
feature_matrix, feature_defs = ft.dfs(
    entityset=es,
    target_dataframe_name='customers',
    max_depth=2    # controls complexity of generated features
)
print(f"Generated {len(feature_defs)} features automatically")

Automated tools generate a lot of candidates. Most won't be useful. You still need feature selection (covered in the next section) to prune the noise. Think of automated feature engineering as brainstorming — it generates ideas quickly, but you need judgment to decide which ones to keep.

⚠️ Key Principle

Feature engineering is iterative. Start simple — scaling and basic encodings. Check model performance. Then add complexity: interactions, domain features, polynomial terms. At each step, validate that the new features actually improve your model. More features isn't always better. Sometimes the best feature engineering is removing a feature that adds noise.
