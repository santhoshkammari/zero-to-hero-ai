Neural Architecture Search (NAS)

Chapter 16: Advanced Deep Learning 5 subtopics
TL;DR

Designing neural network architectures by hand is slow, biased by human intuition, and leaves performance on the table. Neural Architecture Search automates this: an algorithm explores a space of possible architectures, evaluates candidates, and converges on designs that often outperform hand-crafted ones. The field has evolved from brute-force RL-based methods (thousands of GPU-days) to differentiable and one-shot approaches that finish in hours. Hardware-aware NAS adds real-world constraints — latency, memory, FLOPs — so the result actually runs fast on your target device, not on some theoretical ideal.

Why Automate Architecture Design?

Picture this: you're building an image classifier. You pick ResNet because everyone uses ResNet. You choose 50 layers because the paper said so. You use 3×3 convolutions because — well, that's what you've seen. Every one of those decisions is a guess informed by convention, not evidence. Maybe 47 layers would be better for your specific dataset. Maybe a mix of 3×3 and 5×5 convolutions in a particular pattern would squeeze out another percent of accuracy. You'll never know because the search space is enormous and your patience is finite.

This is the problem Neural Architecture Search (NAS) solves. Instead of a human choosing the number of layers, the type of operations, the connectivity pattern, and the channel widths, an algorithm searches through these choices systematically. The human defines the search space — "consider architectures with between 8 and 20 cells, each cell containing some combination of convolutions, pooling, and skip connections" — and the algorithm finds the best design within that space.

The payoff is real. Google's NASNet beat every hand-designed architecture on ImageNet when it was published. EfficientNet, also discovered through NAS, became the standard backbone for years. These aren't marginal improvements — NAS-discovered architectures often find non-obvious patterns that humans wouldn't try.

Search Strategies

RL-Based NAS (Zoph & Le)

The original NAS paper from 2017 framed architecture search as a reinforcement learning problem. The idea is elegant: a controller — implemented as an RNN — generates descriptions of neural network architectures token by token. Each "token" is a design decision: number of filters, filter size, stride, skip connection. The generated architecture gets trained on your dataset, and its validation accuracy becomes the reward signal that updates the controller through REINFORCE.

Think of it like this: the controller is a student proposing architecture designs. Each time it proposes one, a teacher (the training process) grades it. Over many rounds, the controller learns which design choices lead to higher grades.

import torch
import torch.nn as nn
import torch.nn.functional as F

class NASController(nn.Module):
    """Simplified RL-based NAS controller that samples architecture decisions."""
    def __init__(self, num_layers=12, num_ops=5, hidden_size=64):
        super().__init__()
        self.num_layers = num_layers
        self.num_ops = num_ops
        # The controller is an LSTM that outputs architecture tokens
        self.lstm = nn.LSTMCell(hidden_size, hidden_size)
        self.op_classifier = nn.Linear(hidden_size, num_ops)
        self.embed = nn.Embedding(num_ops, hidden_size)
        self.hidden_size = hidden_size

    def forward(self):
        """Sample an architecture: one operation per layer."""
        actions, log_probs = [], []
        # Start with a learned initial input
        inp = torch.zeros(1, self.hidden_size)
        h = torch.zeros(1, self.hidden_size)
        c = torch.zeros(1, self.hidden_size)

        for layer_idx in range(self.num_layers):
            h, c = self.lstm(inp, (h, c))
            logits = self.op_classifier(h)
            probs = F.softmax(logits, dim=-1)
            # Sample an operation for this layer
            action = torch.multinomial(probs, 1).squeeze()
            log_prob = torch.log(probs.squeeze()[action])

            actions.append(action.item())
            log_probs.append(log_prob)
            inp = self.embed(action.unsqueeze(0))

        return actions, log_probs


def train_controller(controller, num_episodes=100):
    """REINFORCE loop: sample architectures, evaluate, update controller."""
    optimizer = torch.optim.Adam(controller.parameters(), lr=0.001)
    baseline = 0.0  # exponential moving average of rewards

    for episode in range(num_episodes):
        # Step 1: Controller samples an architecture
        actions, log_probs = controller()

        # Step 2: Build and train the child network (simplified here)
        # In practice, this means training a full network for N epochs
        reward = evaluate_architecture(actions)  # returns validation accuracy

        # Step 3: Update baseline (variance reduction)
        baseline = 0.95 * baseline + 0.05 * reward

        # Step 4: REINFORCE update
        loss = sum([-(reward - baseline) * lp for lp in log_probs])
        optimizer.zero_grad()
        loss.backward()
        optimizer.step()

        if episode % 10 == 0:
            print(f"Episode {episode}: reward={reward:.4f}, arch={actions}")

The catch? The original paper used 800 GPUs for 28 days. Each architecture candidate needed to be trained from scratch to get an accuracy signal. That's around 22,400 GPU-days. This brute-force cost is what drove the entire field toward more efficient methods.

DARTS (Differentiable Architecture Search)

Here's the key insight behind DARTS: what if we didn't treat architecture search as a discrete decision problem? Instead of sampling "use a 3×3 conv here or a 5×5 conv," what if we used all operations simultaneously and learned a continuous weight for each one?

In DARTS, every edge in the computation graph carries all candidate operations in parallel. A set of architecture parameters (α) controls how much each operation contributes. During search, you jointly optimize the network weights (w) on the training set and the architecture parameters (α) on the validation set through standard gradient descent. No reinforcement learning, no sampling — pure backpropagation.

The continuous relaxation works through a softmax over operations. For edge (i, j) in the cell, the output is:

output(i,j) = Σ_o [ exp(α_o^(i,j)) / Σ_o' exp(α_o'^(i,j)) ] * op_o(x_i)

Where:
  α_o^(i,j) = architecture weight for operation o on edge (i,j)
  op_o       = candidate operation (3x3 conv, 5x5 conv, max pool, skip, zero)
  x_i        = input from node i

After search, you discretize: pick the operation with the highest α on each edge. That gives you the final architecture.

import torch
import torch.nn as nn
import torch.nn.functional as F

# Candidate operations for each edge in a DARTS cell
OPS = {
    'sep_conv_3x3': lambda C: SepConv(C, C, 3, 1),
    'sep_conv_5x5': lambda C: SepConv(C, C, 5, 2),
    'dil_conv_3x3': lambda C: DilConv(C, C, 3, 2, 2),
    'max_pool_3x3': lambda C: nn.MaxPool2d(3, stride=1, padding=1),
    'skip_connect': lambda C: nn.Identity(),
    'zero':         lambda C: Zero(),
}

class SepConv(nn.Module):
    """Depthwise separable convolution."""
    def __init__(self, C_in, C_out, kernel, padding):
        super().__init__()
        self.op = nn.Sequential(
            nn.Conv2d(C_in, C_in, kernel, padding=padding, groups=C_in, bias=False),
            nn.Conv2d(C_in, C_out, 1, bias=False),
            nn.BatchNorm2d(C_out),
            nn.ReLU(inplace=True),
        )
    def forward(self, x):
        return self.op(x)

class DilConv(nn.Module):
    """Dilated separable convolution."""
    def __init__(self, C_in, C_out, kernel, padding, dilation):
        super().__init__()
        self.op = nn.Sequential(
            nn.Conv2d(C_in, C_in, kernel, padding=padding,
                      dilation=dilation, groups=C_in, bias=False),
            nn.Conv2d(C_in, C_out, 1, bias=False),
            nn.BatchNorm2d(C_out),
            nn.ReLU(inplace=True),
        )
    def forward(self, x):
        return self.op(x)

class Zero(nn.Module):
    """Zero operation — effectively removes the edge."""
    def forward(self, x):
        return x * 0


class MixedOp(nn.Module):
    """A single edge that mixes all candidate operations via softmax."""
    def __init__(self, C):
        super().__init__()
        self.ops = nn.ModuleList([fn(C) for fn in OPS.values()])

    def forward(self, x, weights):
        # weights: softmax over architecture parameters for this edge
        return sum(w * op(x) for w, op in zip(weights, self.ops))


class DARTSCell(nn.Module):
    """One cell in DARTS with N intermediate nodes."""
    def __init__(self, C, num_nodes=4):
        super().__init__()
        self.num_nodes = num_nodes
        self.num_ops = len(OPS)
        self.edges = nn.ModuleDict()

        # Create mixed ops for every possible edge
        # Nodes 0,1 are inputs; nodes 2..num_nodes+1 are intermediate
        for j in range(2, num_nodes + 2):
            for i in range(j):
                self.edges[f"{i}_{j}"] = MixedOp(C)

        # Architecture parameters: one set per edge
        num_edges = sum(range(2, num_nodes + 2))
        self.arch_params = nn.Parameter(
            1e-3 * torch.randn(num_edges, self.num_ops)
        )

    def forward(self, s0, s1):
        states = [s0, s1]
        edge_idx = 0
        for j in range(2, self.num_nodes + 2):
            # Each intermediate node sums contributions from all prior nodes
            node_input = []
            for i in range(j):
                weights = F.softmax(self.arch_params[edge_idx], dim=-1)
                node_input.append(
                    self.edges[f"{i}_{j}"](states[i], weights)
                )
                edge_idx += 1
            states.append(sum(node_input))

        # Concat all intermediate node outputs
        return torch.cat(states[2:], dim=1)

    def derive_architecture(self):
        """After search, pick the best operation per edge."""
        arch = {}
        edge_idx = 0
        op_names = list(OPS.keys())
        for j in range(2, self.num_nodes + 2):
            for i in range(j):
                best_op = op_names[self.arch_params[edge_idx].argmax().item()]
                arch[f"{i}->{j}"] = best_op
                edge_idx += 1
        return arch

The bilevel optimization alternates: one step of gradient descent on network weights w using training data, then one step on architecture parameters α using validation data. This completes in about 1 GPU-day — roughly 20,000× cheaper than the original RL approach.

DARTS Pitfall: Collapse

A known failure mode of DARTS is performance collapse: the architecture parameters converge to skip connections everywhere because skip connections have the smoothest gradient flow. The searched architecture looks like a glorified residual network and underperforms. Fixes include adding regularization to α (progressive DARTS), restricting the number of skip connections, or using early stopping based on validation performance during search.

Evolutionary NAS

Evolutionary NAS treats architecture search like biological evolution. You maintain a population of architectures. Each generation, you evaluate their fitness (accuracy), select the best ones, mutate them (swap an operation, add a layer, change a connection), and repeat. No gradients, no RL — nature's oldest optimization algorithm.

AmoebaNet from Google used regularized evolution: instead of always killing the worst individual, it removes the oldest one. This prevents the population from getting stuck in local optima too early.

import random
import copy
from collections import namedtuple

Architecture = namedtuple('Architecture', ['ops', 'accuracy', 'age'])

OP_CHOICES = ['conv3x3', 'conv5x5', 'sep_conv3x3', 'max_pool', 'avg_pool', 'skip', 'zero']

def random_architecture(num_layers=8):
    """Create a random architecture."""
    return [random.choice(OP_CHOICES) for _ in range(num_layers)]

def mutate(arch_ops):
    """Mutate one random position in the architecture."""
    child = copy.deepcopy(arch_ops)
    pos = random.randint(0, len(child) - 1)
    child[pos] = random.choice(OP_CHOICES)
    return child

def evaluate_architecture(ops):
    """Train and evaluate an architecture. Returns validation accuracy.
    In practice this trains a real model — here we simulate it."""
    # Placeholder: real NAS trains the network for N epochs
    score = random.uniform(0.85, 0.97)
    # Penalize heavy architectures (more convolutions = slower)
    heavy_ops = sum(1 for o in ops if 'conv5x5' in o)
    score -= heavy_ops * 0.005
    return score

def evolutionary_nas(pop_size=20, num_generations=50, tournament_size=5):
    """Regularized evolution for architecture search."""
    # Initialize population with random architectures
    population = []
    for _ in range(pop_size):
        ops = random_architecture()
        acc = evaluate_architecture(ops)
        population.append(Architecture(ops=ops, accuracy=acc, age=0))

    best_ever = max(population, key=lambda a: a.accuracy)

    for gen in range(num_generations):
        # Tournament selection: pick the best from a random sample
        candidates = random.sample(population, tournament_size)
        parent = max(candidates, key=lambda a: a.accuracy)

        # Mutate the parent to create a child
        child_ops = mutate(parent.ops)
        child_acc = evaluate_architecture(child_ops)
        child = Architecture(ops=child_ops, accuracy=child_acc, age=gen)

        # Add child and remove oldest (regularized evolution)
        population.append(child)
        population.pop(0)  # remove oldest member

        if child_acc > best_ever.accuracy:
            best_ever = child

        if gen % 10 == 0:
            avg_acc = sum(a.accuracy for a in population) / len(population)
            print(f"Gen {gen}: best={best_ever.accuracy:.4f}, avg={avg_acc:.4f}")

    print(f"\nBest architecture: {best_ever.ops}")
    print(f"Best accuracy: {best_ever.accuracy:.4f}")
    return best_ever

# Run the search
best = evolutionary_nas()

Evolutionary methods are surprisingly competitive. They're embarrassingly parallel — you can evaluate hundreds of candidates across a GPU cluster simultaneously. And they make no differentiability assumptions, so they work with any search space, including ones with discrete, non-relaxable decisions like "use attention or don't."

Bayesian Optimization for NAS

Bayesian Optimization (BO) brings sample efficiency to NAS. Instead of blindly sampling architectures, BO builds a probabilistic model (a surrogate) of the relationship between architecture choices and performance. It then uses an acquisition function — typically Expected Improvement — to decide which architecture to try next, balancing exploration (trying something new) and exploitation (refining what looks good).

BANANAS (Bayesian Optimization with Neural Architectures for NAS) encodes architectures as path-based features and uses an ensemble of neural networks as the surrogate model. This is effective when each evaluation is expensive: BO tries to find the best architecture in the fewest trials.

Efficiency Methods

One-Shot and Weight-Sharing NAS

The biggest cost in NAS is training each candidate architecture from scratch. Weight-sharing NAS eliminates this by training one massive supernet that contains every possible architecture as a subgraph. Want to evaluate a specific architecture? Activate only its corresponding subgraph and measure its performance — no retraining needed.

Think of the supernet as a building with every possible room layout already constructed. Evaluating a floor plan means walking through its specific rooms, not constructing a new building.

class SuperNet(nn.Module):
    """A supernet where each layer has all possible operations.
    During training, we randomly sample a path (one op per layer)."""
    def __init__(self, num_layers=8, channels=64):
        super().__init__()
        self.layers = nn.ModuleList()
        for _ in range(num_layers):
            self.layers.append(nn.ModuleDict({
                'conv3x3': nn.Conv2d(channels, channels, 3, padding=1),
                'conv5x5': nn.Conv2d(channels, channels, 5, padding=2),
                'skip':    nn.Identity(),
            }))
        self.classifier = nn.Linear(channels, 10)

    def forward(self, x, path=None):
        """If path is given, use that specific subnetwork.
        Otherwise, sample a random path (for training)."""
        op_names = ['conv3x3', 'conv5x5', 'skip']
        if path is None:
            path = [random.choice(op_names) for _ in self.layers]

        for layer, op_name in zip(self.layers, path):
            x = layer[op_name](x)
            if op_name != 'skip':
                x = F.relu(x)

        x = x.mean(dim=[2, 3])  # global average pooling
        return self.classifier(x)

# Training loop: randomly sample paths each step
supernet = SuperNet()
optimizer = torch.optim.SGD(supernet.parameters(), lr=0.01)

for batch_x, batch_y in dataloader:
    # Each step trains a random sub-architecture
    output = supernet(batch_x)  # path=None triggers random sampling
    loss = F.cross_entropy(output, batch_y)
    optimizer.zero_grad()
    loss.backward()
    optimizer.step()

Performance Predictors and Early Stopping

Performance predictors go one step further: instead of training an architecture at all, predict its final accuracy from cheaper signals. Train a regression model on (architecture encoding → final accuracy) pairs from previously evaluated architectures. Then score new candidates instantly.

Common cheap signals include accuracy after just 5 epochs (instead of 200), accuracy on a small proxy dataset, or even zero-cost proxies like the number of linear regions in the network at initialization. These proxies aren't perfect, but they're good enough to filter out the bottom 90% of candidates, focusing expensive full training on the most promising architectures.

Early stopping strategies complement predictors. Learning curve extrapolation fits a parametric model to partial training curves and predicts whether an architecture will eventually beat the current best. If it won't, kill it early and move on. Multi-fidelity methods like Hyperband formalize this: allocate small budgets to many candidates, then progressively increase the budget for the top performers.

Hardware-Aware NAS

MnasNet and Multi-Objective Search

Accuracy alone doesn't make a useful model. A model that's 0.5% more accurate but 10× slower on a phone is worse in practice. MnasNet from Google introduced multi-objective NAS: the search optimizes both accuracy and real-device latency simultaneously.

The reward function combines both objectives:

Reward = Accuracy(arch) × [Latency(arch) / TargetLatency]^w

Where w is a weight factor (typically w = -0.07 to -0.15):
  - If latency < target: small bonus (the ratio is < 1, raised to negative power > 1)
  - If latency > target: large penalty

The key innovation: MnasNet measures latency on the actual target device (a Pixel phone), not with FLOPs estimates. FLOPs are a poor proxy because they ignore memory bandwidth, cache effects, and operator-level hardware optimizations. A depthwise convolution has fewer FLOPs than a standard convolution but might not be faster if the hardware's memory system is the bottleneck.

ProxylessNAS

ProxylessNAS makes hardware-aware search differentiable. It adds a latency loss term directly into the training objective. A lookup table maps each operation to its measured latency on the target hardware. Since the architecture is parameterized with continuous weights (like DARTS), the expected latency is differentiable:

def compute_expected_latency(arch_params, latency_table):
    """Differentiable latency estimation using architecture parameters."""
    total_latency = 0.0
    for layer_idx, alpha in enumerate(arch_params):
        probs = F.softmax(alpha, dim=-1)
        # Weighted sum of latencies for all operations at this layer
        layer_latency = sum(
            p * latency_table[layer_idx][op_idx]
            for op_idx, p in enumerate(probs)
        )
        total_latency += layer_latency
    return total_latency

EfficientNet and Compound Scaling

EfficientNet is the flagship result of NAS applied at scale. But its real contribution isn't the searched architecture — it's the compound scaling insight. When you want to make a model bigger (for more accuracy), you have three knobs: depth (more layers), width (more channels per layer), and resolution (larger input images). Most people scale only one dimension. EfficientNet showed that scaling all three together, in a balanced way, is far more efficient.

The compound scaling formula:

depth:      d = α^φ
width:      w = β^φ
resolution: r = γ^φ

Constraints:
  α · β² · γ² ≈ 2    (so doubling φ roughly doubles the computation)
  α ≥ 1, β ≥ 1, γ ≥ 1

For EfficientNet-B0 (found via NAS):
  α = 1.2, β = 1.1, γ = 1.15

Then scale with compound coefficient φ:
  B0: φ = 0  → baseline
  B1: φ = 1  → 1.2 depth, 1.1 width, 1.15 resolution
  ...
  B7: φ = 7  → much larger across all dimensions

The baseline architecture (EfficientNet-B0) was found using MnasNet-style NAS with a mobile-sized search space. Then compound scaling produced B1 through B7 — each one a larger, more accurate variant. EfficientNet-B7 achieved state-of-the-art ImageNet accuracy while being 8.4× smaller and 6.1× faster than the best existing models at the time.

Why β² and γ²?

Width and resolution are squared in the constraint because FLOPs scale quadratically with both. Doubling the width doubles the number of channels, which roughly quadruples the computation (each filter convolves with twice as many input channels AND produces twice as many output channels). Doubling the resolution quadruples the number of pixels each convolution processes. Depth scales linearly — twice as many layers means twice the computation.

Evaluation and Benchmarks

NAS-Bench-101, 201, and 301

A fundamental problem in NAS research: how do you compare search algorithms fairly if each paper uses a different search space, different training pipeline, and different evaluation protocol? NAS-Bench solves this by precomputing the performance of every architecture in a defined search space.

NAS-Bench-101 tabulates 423,624 unique architectures in a cell-based search space, each trained three times on CIFAR-10. Want to know how your search algorithm performs? Query the table instead of training. This makes NAS research accessible to anyone with a laptop.

# Using NAS-Bench-201 (simpler, covers CIFAR-10, CIFAR-100, ImageNet-16-120)

NAS-Bench-301 takes a different approach: instead of exhaustively evaluating all architectures (impossible for larger spaces like DARTS with 10^18 candidates), it trains a surrogate model to predict performance. You query the surrogate instead of a lookup table, getting approximate but fast estimates for any architecture in the space.

Search Space Design and Bias

Here's an uncomfortable truth: the search space matters more than the search algorithm. If your search space only contains good architectures, even random search will find something decent. Multiple studies have shown that random search within a well-designed search space matches or beats sophisticated NAS algorithms in poorly designed spaces.

This means that the human is still deeply involved in NAS. You choose which operations to include, how cells connect, whether to use a cell-based or macro search space, and the ranges for hyperparameters. Each choice embeds your assumptions. Including skip connections guarantees ResNet-like architectures will dominate. Limiting to depthwise separable convolutions biases toward MobileNet-like designs. NAS finds the best architecture within your biases — it doesn't escape them.

Reproducibility Challenges

NAS papers have a reproducibility problem. Many early papers showed inflated results because the evaluation protocol was entangled with the search. Common issues: training the searched architecture with a much larger budget than baselines, using different data augmentation for NAS-found models, or reporting the best of multiple search runs without accounting for the search cost. NAS-Bench addresses the search-side reproducibility, but evaluation-side practices still vary widely across papers.

Related Concepts

The Lottery Ticket Hypothesis

The Lottery Ticket Hypothesis, proposed by Frankle and Carlin in 2019, makes a striking claim: dense neural networks contain sparse subnetworks (the "winning tickets") that, when trained in isolation from the same initialization, match the full network's accuracy. The large network isn't needed for its capacity — it's needed to find the right subnetwork through training.

Here's the concrete procedure:

import torch
import torch.nn as nn
import copy

def find_lottery_ticket(model_fn, train_fn, eval_fn, prune_pct=0.2, rounds=5):
    """Iterative Magnitude Pruning to find a lottery ticket.

    1. Initialize network with weights w_0
    2. Train to get weights w_T
    3. Prune smallest-magnitude weights, creating a mask
    4. Reset remaining weights back to w_0 (rewind!)
    5. Repeat from step 2 with the pruned network
    """
    # Step 1: Initialize and save the original weights
    model = model_fn()
    initial_weights = copy.deepcopy(model.state_dict())

    # Start with everything unmasked
    masks = {name: torch.ones_like(p) for name, p in model.named_parameters()
             if 'weight' in name and p.dim() > 1}

    for round_num in range(rounds):
        # Step 2: Train the (masked) network
        train_fn(model, masks)
        accuracy = eval_fn(model)
        total_params = sum(m.sum().item() for m in masks.values())
        total_possible = sum(m.numel() for m in masks.values())
        sparsity = 1 - (total_params / total_possible)
        print(f"Round {round_num}: accuracy={accuracy:.4f}, "
              f"sparsity={sparsity:.1%}")

        # Step 3: Prune the smallest weights globally
        all_weights = []
        for name, param in model.named_parameters():
            if name in masks:
                # Only consider currently alive weights
                alive = param.data.abs() * masks[name]
                all_weights.append(alive.flatten())

        all_weights = torch.cat(all_weights)
        alive_weights = all_weights[all_weights > 0]
        threshold = torch.quantile(alive_weights, prune_pct)

        for name, param in model.named_parameters():
            if name in masks:
                masks[name] *= (param.data.abs() > threshold).float()

        # Step 4: Rewind — reset weights to original initialization
        model.load_state_dict(initial_weights)

        # Apply masks so pruned weights stay dead
        with torch.no_grad():
            for name, param in model.named_parameters():
                if name in masks:
                    param.mul_(masks[name])

    return model, masks

# Result: a network at 90%+ sparsity that matches the original's accuracy
# when trained from the SAME initialization with pruning mask applied.

Why does this matter for NAS? Because it suggests that architecture search might partly be about finding the right sparsity pattern. A 10-layer network with certain connections pruned might be equivalent to a specific 6-layer architecture that NAS would discover. Both approaches — NAS and lottery ticket pruning — converge on the same idea: most of the network's parameters aren't needed, and the art is finding which ones matter.

Relationship to Pruning

NAS and pruning attack the same problem from opposite directions. NAS builds up: start from nothing and construct the right architecture. Pruning tears down: start with a large network and remove what's unnecessary. In practice, many production pipelines combine both — use NAS to find a good base architecture, then prune it further for deployment.

Structured pruning (removing entire filters or layers) is especially close to NAS. When you prune an entire convolutional filter, you're making an architectural decision — this filter shouldn't exist. A NAS algorithm that chooses between "16 filters" and "32 filters" at each layer is making the same kind of decision. The difference is mostly in the search procedure, not the outcome.

Network Morphisms

Network morphisms are function-preserving transformations that modify a network's architecture without changing its output. You can widen a layer (add filters initialized to zero plus identity scaling), deepen a network (insert an identity layer), or change kernel sizes — all while preserving the network's current predictions exactly.

This is powerful for NAS because it enables warm-starting: instead of training each new candidate from scratch, morph a previously trained architecture into the new one and continue training. The morphed network starts where the old one left off, dramatically reducing training time. Net2Net and EAS (Efficient Architecture Search) use this idea to cut NAS costs by an order of magnitude.

The Big Picture

NAS has matured from an academic curiosity (expensive, unreproducible) into a practical engineering tool. Modern NAS runs in hours on a single GPU, targets real hardware constraints, and produces architectures you can deploy directly. But the field's biggest lesson might be this: good search spaces matter more than clever search algorithms. The human still defines the game — NAS finds the best move within it. As search spaces grow more expressive and benchmarks more rigorous, the gap between human-designed and machine-discovered architectures will keep widening.
