Domain-Specific ML Applications

Chapter 16: Advanced Deep Learning 10 subtopics
TL;DR

General-purpose ML knowledge gets you maybe 40% of the way in any real domain. The remaining 60% is domain-specific: regulatory constraints in healthcare, lookahead bias in finance, sensor fusion in autonomous vehicles, cold-start problems in recommender systems. This section tours ten major domains where ML has found serious traction — not to make you an expert in each, but to show you what makes each domain uniquely challenging and what architectures have emerged to handle those challenges. Think of it as a map before you pick which territory to explore.

1. ML for Healthcare

A radiologist reads hundreds of scans per day. Fatigue sets in. Subtle nodules get missed. That is the problem medical imaging ML tries to solve — not replacing the doctor, but giving them a tireless second pair of eyes. The stakes here are literally life and death, which is why this domain has regulatory machinery that no other ML field comes close to matching.

U-Net is the workhorse architecture for medical image segmentation. It was invented specifically for biomedical images where you have very few labeled samples. The key insight is a symmetric encoder-decoder with skip connections — the encoder compresses the image to capture "what," and the decoder expands it back to capture "where," while skip connections preserve fine spatial details that would otherwise be lost during compression.

# U-Net architecture concept (simplified)
import torch
import torch.nn as nn

class UNetBlock(nn.Module):
    def __init__(self, in_ch, out_ch):
        super().__init__()
        self.conv = nn.Sequential(
            nn.Conv2d(in_ch, out_ch, 3, padding=1),
            nn.BatchNorm2d(out_ch),
            nn.ReLU(inplace=True),
            nn.Conv2d(out_ch, out_ch, 3, padding=1),
            nn.BatchNorm2d(out_ch),
            nn.ReLU(inplace=True),
        )
    def forward(self, x):
        return self.conv(x)

class MiniUNet(nn.Module):
    def __init__(self, n_classes=1):
        super().__init__()
        # Encoder (downsampling)
        self.enc1 = UNetBlock(1, 64)
        self.enc2 = UNetBlock(64, 128)
        self.pool = nn.MaxPool2d(2)

        # Bottleneck
        self.bottleneck = UNetBlock(128, 256)

        # Decoder (upsampling) with skip connections
        self.up2 = nn.ConvTranspose2d(256, 128, 2, stride=2)
        self.dec2 = UNetBlock(256, 128)  # 256 because of skip concat
        self.up1 = nn.ConvTranspose2d(128, 64, 2, stride=2)
        self.dec1 = UNetBlock(128, 64)

        self.out_conv = nn.Conv2d(64, n_classes, 1)

    def forward(self, x):
        # Encode
        e1 = self.enc1(x)           # keep for skip connection
        e2 = self.enc2(self.pool(e1))

        # Bottleneck
        b = self.bottleneck(self.pool(e2))

        # Decode with skip connections
        d2 = self.dec2(torch.cat([self.up2(b), e2], dim=1))
        d1 = self.dec1(torch.cat([self.up1(d2), e1], dim=1))
        return self.out_conv(d1)

Regulatory Reality: FDA and SaMD

Software as a Medical Device (SaMD) is the FDA's classification for ML models used in clinical decisions. If your model recommends a diagnosis, it is a medical device — full stop. That means you need clinical trials, a predicate device for 510(k) clearance, or the De Novo pathway for novel AI. Your model versioning, training data provenance, and performance monitoring all become regulated artifacts. This is why healthcare ML teams spend as much time on documentation and validation as on model development.

On the data side, MIMIC-III/IV is the most widely used public electronic health record (EHR) dataset — ICU records from Beth Israel Deaconess. Clinical NLP extracts structured information from doctor notes (think: "patient presents with SOB" → shortness of breath, not what you might think). And because hospitals cannot share patient data freely, federated learning lets multiple hospitals train a shared model without raw data ever leaving each site.

Drug Discovery Shortcut

ML in drug discovery works on molecular graphs — atoms are nodes, bonds are edges. Graph neural networks predict binding affinity between a candidate molecule and a protein target. This can compress early-stage screening from years to weeks, but wet-lab validation remains the bottleneck. The model does not replace the chemistry; it narrows the search space.

2. ML for Finance

Finance is seductive for ML practitioners because the data is abundant, numerical, and timestamped. It is also a graveyard of overconfident models. The core reason: markets are adversarial. Your alpha signal decays the moment others discover it. And the most dangerous bug in financial ML is not a coding error — it is lookahead bias, where your model accidentally peeks at future information during training.

Lookahead Bias: The Silent Killer

Imagine you are predicting whether a stock will go up tomorrow. You compute a 30-day moving average as a feature. If you compute that average using the full dataset (including future data points), your model has seen the future. It will look brilliant in backtesting and fail catastrophically in production. The fix is a strict temporal split — never let information from time t+1 leak into features or labels at time t.

# Preventing lookahead bias in feature engineering
import pandas as pd

def safe_features(df, price_col='close', window=30):
    """Compute features using ONLY past data at each point."""
    df = df.sort_values('date').copy()

    # CORRECT: expanding/rolling uses only past data
    df['ma_30'] = df[price_col].rolling(window=window, min_periods=window).mean()
    df['volatility_30'] = df[price_col].rolling(window=window).std()
    df['momentum'] = df[price_col].pct_change(periods=5)

    # CORRECT: target is FUTURE return (shift negative = future)
    df['target'] = df[price_col].shift(-1) / df[price_col] - 1

    # WRONG (lookahead): using future data in features
    # df['future_vol'] = df[price_col].shift(-30).rolling(30).std()  # NO!
    return df.dropna()

def temporal_split(df, train_end, val_end):
    """Strict temporal split — no shuffling allowed."""
    train = df[df['date'] <= train_end]
    val = df[(df['date'] > train_end) & (df['date'] <= val_end)]
    test = df[df['date'] > val_end]
    return train, val, test

# Walk-forward validation: retrain periodically on expanding window
def walk_forward_backtest(df, retrain_every='3M'):
    """Simulate real deployment: train on past, predict next period."""
    results = []
    for period_start in pd.date_range(df['date'].min(), df['date'].max(), freq=retrain_every):
        train = df[df['date'] < period_start]
        test_end = period_start + pd.DateOffset(months=3)
        test = df[(df['date'] >= period_start) & (df['date'] < test_end)]
        if len(train) < 100 or len(test) == 0:
            continue
        # model.fit(train[features], train['target'])
        # preds = model.predict(test[features])
        results.append({'period': period_start, 'n_train': len(train), 'n_test': len(test)})
    return pd.DataFrame(results)

Fraud detection is the other big finance ML use case. It is an extreme class-imbalance problem — maybe 0.1% of transactions are fraudulent. You need precision-recall tradeoffs, not accuracy. Real-time inference latency matters too: you have milliseconds to flag a transaction before it clears. Credit scoring models face fairness constraints — regulations like the Equal Credit Opportunity Act require that you can explain why someone was denied credit.

Backtesting ≠ Reality

A backtest assumes you can execute trades at historical prices. In reality, your large order moves the market (slippage), some trades fail (fill rate), and transaction costs eat into returns. A strategy that returns 20% in backtest might return 2% live. Always model transaction costs and slippage before celebrating.

3. ML for Autonomous Vehicles

An autonomous vehicle needs to answer three questions every 100 milliseconds: Where am I? What is around me? What should I do next? These correspond to localization, perception, and planning — and each is a hard ML problem on its own. The perception pipeline alone combines cameras, LiDAR, radar, and sometimes ultrasonics into a coherent 3D understanding of the world.

Sensor Fusion and BEV

Sensor fusion combines data from multiple sensor types. Early fusion concatenates raw sensor data before any processing. Late fusion runs separate models on each sensor and combines their predictions. Mid fusion — the most common in practice — extracts features from each sensor independently, then merges them in a shared representation. The dominant representation today is Bird's Eye View (BEV): project everything onto a top-down 2D grid. BEV makes it easy to reason about distances, lanes, and object relationships because you are looking straight down instead of through distorted camera perspectives.

For 3D object detection, models like PointPillars process raw LiDAR point clouds by grouping points into vertical columns (pillars), encoding each pillar as a feature vector, and scattering them onto a 2D pseudo-image that you can process with standard 2D convolutions. Camera-based approaches like BEVFormer use transformer cross-attention to lift 2D image features into 3D BEV space without LiDAR at all.

Motion planning decides the actual trajectory. Classical approaches use optimization with safety constraints. Imitation learning trains a neural network to mimic human driving data. The hybrid approach — predict multiple trajectory candidates with a neural network, then score them with a rule-based safety checker — is what most production systems actually deploy.

The Long Tail Problem

Autonomous driving works well 99% of the time. The challenge is the remaining 1%: construction zones, emergency vehicles, a mattress on the highway. These rare scenarios are hard to collect data for and hard to test. This is why simulation (like CARLA) and scenario generation are critical — you need to manufacture the weird edge cases your model has never seen.

4. ML for Recommender Systems

Every time Netflix suggests a show or Amazon suggests a product, a recommender system is at work. The fundamental idea: if users A and B liked the same items in the past, they will probably like similar items in the future. This is collaborative filtering — you do not need to understand the content at all, only the interaction patterns.

Matrix Factorization

Matrix factorization decomposes the giant user-item interaction matrix into two smaller matrices — user embeddings and item embeddings. The predicted rating for user u and item i is the dot product of their embeddings. This was the winning approach in the Netflix Prize and remains a strong baseline.

# Matrix Factorization for recommendations
import torch
import torch.nn as nn

class MatrixFactorization(nn.Module):
    def __init__(self, n_users, n_items, n_factors=64):
        super().__init__()
        self.user_emb = nn.Embedding(n_users, n_factors)
        self.item_emb = nn.Embedding(n_items, n_factors)
        self.user_bias = nn.Embedding(n_users, 1)
        self.item_bias = nn.Embedding(n_items, 1)
        self.global_bias = nn.Parameter(torch.zeros(1))

        # Initialize small weights
        nn.init.normal_(self.user_emb.weight, std=0.01)
        nn.init.normal_(self.item_emb.weight, std=0.01)

    def forward(self, user_ids, item_ids):
        u = self.user_emb(user_ids)   # (batch, n_factors)
        i = self.item_emb(item_ids)   # (batch, n_factors)
        # Dot product + biases
        dot = (u * i).sum(dim=1, keepdim=True)
        pred = dot + self.user_bias(user_ids) + self.item_bias(item_ids) + self.global_bias
        return pred.squeeze()

# Two-Tower model: separate encoders for user and item
class TwoTower(nn.Module):
    def __init__(self, user_feat_dim, item_feat_dim, embed_dim=128):
        super().__init__()
        self.user_tower = nn.Sequential(
            nn.Linear(user_feat_dim, 256), nn.ReLU(),
            nn.Linear(256, embed_dim), nn.LayerNorm(embed_dim),
        )
        self.item_tower = nn.Sequential(
            nn.Linear(item_feat_dim, 256), nn.ReLU(),
            nn.Linear(256, embed_dim), nn.LayerNorm(embed_dim),
        )

    def forward(self, user_features, item_features):
        u_emb = self.user_tower(user_features)  # encode user context
        i_emb = self.item_tower(item_features)  # encode item context
        # Cosine similarity for scoring
        return torch.cosine_similarity(u_emb, i_emb, dim=1)

The cold-start problem is when a new user or new item has zero interactions. Matrix factorization cannot help — there is no history to factorize. This is where content-based features come in: use item metadata (genre, description) and user demographics to bootstrap recommendations. The two-tower architecture shown above handles this naturally because it encodes raw features, not just IDs. At serving time, you pre-compute all item embeddings, index them with approximate nearest neighbors (ANN), and retrieve the top-K candidates in milliseconds. Neural Collaborative Filtering (NCF) replaces the dot product with a small MLP, letting the model learn non-linear user-item interactions.

5. ML for Speech

You talk to Siri, Alexa, or Google Assistant every day. Under the hood, your voice waveform goes through an Automatic Speech Recognition (ASR) pipeline: raw audio → spectrogram features → acoustic model → text. The hard part is alignment — the audio and text have different lengths, and you do not know which audio frame corresponds to which character.

CTC and Modern ASR

Connectionist Temporal Classification (CTC) solves the alignment problem by introducing a blank token. The model outputs a probability distribution over characters (plus blank) at every audio frame. CTC marginalizes over all possible alignments — you do not need to know the alignment, only the final text. This was a breakthrough because it eliminated the need for frame-level labels.

# CTC loss in practice — the key idea
import torch
import torch.nn as nn

# Model outputs: (time_steps, batch, n_characters + 1_blank)
# Suppose 200 audio frames, batch of 8, 29 chars (26 letters + space + apostrophe + blank)
log_probs = torch.randn(200, 8, 29).log_softmax(dim=2)

# Targets: variable-length text sequences (no alignment needed!)
targets = torch.randint(0, 28, (8, 40))     # 8 sentences, max 40 chars
input_lengths = torch.full((8,), 200)         # all 200 frames
target_lengths = torch.randint(10, 40, (8,))  # actual lengths vary

ctc_loss = nn.CTCLoss(blank=28, zero_infinity=True)
loss = ctc_loss(log_probs, targets, input_lengths, target_lengths)
# CTC sums over ALL valid alignments internally — you train on the text alone

OpenAI's Whisper changed the game by training a single encoder-decoder transformer on 680,000 hours of multilingual audio. It uses an encoder (audio spectrogram → features) and a decoder (features → text tokens, autoregressively). Whisper handles transcription, translation, and language identification in one model. On the self-supervised side, wav2vec 2.0 learns speech representations by masking portions of the audio and predicting them — the BERT approach, but for sound waves. For Text-to-Speech (TTS), models like VITS generate natural-sounding speech by combining variational autoencoders with adversarial training. And speaker verification creates voice embeddings — if two audio clips produce similar embeddings, they are the same person.

6. ML for Tabular Data

Here is an uncomfortable truth for deep learning enthusiasts: on tabular data (spreadsheets, SQL tables, CSV files), gradient-boosted decision trees — XGBoost, LightGBM, CatBoost — still beat neural networks in most benchmarks. A 2022 paper by Grinsztajn et al. confirmed this rigorously. Why? Tabular data has irregular feature distributions, meaningful feature interactions that trees capture naturally through splits, and no spatial or sequential structure for neural networks to exploit.

When Deep Learning Fights Back

TabNet uses sequential attention to select which features to focus on at each decision step — it is a learned, soft version of a decision tree's feature selection. FT-Transformer treats each feature as a token (after a per-feature linear embedding), then applies a standard transformer. FT-Transformer actually matches or beats trees on some benchmarks, especially when the dataset is large and features interact in complex non-linear ways.

# Feature Tokenizer concept (core idea of FT-Transformer)
import torch
import torch.nn as nn

class FeatureTokenizer(nn.Module):
    """Each tabular feature gets its own embedding — like words in NLP."""
    def __init__(self, n_numerical, n_categories, cat_cardinalities, d_token=192):
        super().__init__()
        # Numerical features: linear projection per feature
        self.num_projections = nn.ModuleList([
            nn.Linear(1, d_token) for _ in range(n_numerical)
        ])
        # Categorical features: embedding per feature
        self.cat_embeddings = nn.ModuleList([
            nn.Embedding(card, d_token) for card in cat_cardinalities
        ])
        # [CLS] token for final prediction
        self.cls_token = nn.Parameter(torch.randn(1, 1, d_token))

    def forward(self, x_num, x_cat):
        # Each numerical feature → a token
        num_tokens = [proj(x_num[:, i:i+1]) for i, proj in enumerate(self.num_projections)]
        # Each categorical feature → a token
        cat_tokens = [emb(x_cat[:, i]) for i, emb in enumerate(self.cat_embeddings)]

        tokens = num_tokens + cat_tokens  # list of (batch, d_token)
        tokens = torch.stack(tokens, dim=1)  # (batch, n_features, d_token)

        # Prepend CLS token
        cls = self.cls_token.expand(tokens.size(0), -1, -1)
        return torch.cat([cls, tokens], dim=1)
        # Now feed this into a standard Transformer encoder

Target encoding replaces a categorical feature with the mean of the target variable for that category — powerful but leak-prone if you do not use out-of-fold encoding. AutoML tools like AutoGluon, FLAML, and H2O automate the boring parts: feature engineering, model selection, hyperparameter tuning. For most tabular problems, running AutoGluon for two hours will beat a hand-tuned neural network. Accept this and move on to harder problems.

Practical Tabular Advice

Start with LightGBM. If it works well enough, ship it. Try FT-Transformer only if you have 50K+ rows, complex feature interactions, and a reason to believe trees are leaving performance on the table. Ensemble both if you need every last fraction of a percent.

7. ML for Time Series

Forecasting demand, predicting server load, estimating energy consumption — time series is everywhere in industry. The unique challenge: temporal dependencies span multiple scales. Sales spike on Fridays (weekly pattern), drop in January (yearly pattern), and trend upward over years (trend). A good model captures all three simultaneously.

Modern Architectures

Temporal Fusion Transformer (TFT) combines LSTMs for local temporal processing with multi-head attention for long-range dependencies. It also handles static covariates (store location), known future inputs (day of week), and unknown future inputs (weather) through separate processing pathways. The variable selection network automatically learns which features matter — you get built-in interpretability.

N-BEATS takes a different approach: pure MLPs with no recurrence, organized into stacks that explicitly decompose the signal into trend and seasonality. PatchTST borrows the patch idea from Vision Transformers — split the time series into fixed-length patches, treat each patch as a token, and run a transformer. This drastically reduces the sequence length the transformer sees, making it more efficient and often more accurate.

DeepAR produces probabilistic forecasts — not point predictions, but full probability distributions. This is critical for business decisions: knowing that demand will be "between 80 and 120 units with 90% confidence" is far more useful than "100 units." Foundation models for time series are emerging too: Google's TimesFM and Amazon's Chronos are pretrained on massive time series corpora. Chronos tokenizes time series values into bins and uses a language model architecture — treating forecasting as a token prediction problem. These models show promising zero-shot performance on unseen datasets.

The Dirty Secret of Time Series

In many real-world forecasting competitions, a well-tuned exponential smoothing or ARIMA model with proper feature engineering beats deep learning. Always benchmark against statistical baselines. Deep learning wins when you have many related time series (cross-learning) or very long histories.

8. AI for Science

ML is not replacing scientists. It is giving them superpowers. Protein structure prediction used to take months of lab work and computation. AlphaFold 2 predicts a protein's 3D structure from its amino acid sequence in minutes, with accuracy rivaling experimental methods. This is arguably ML's single biggest scientific contribution so far.

AlphaFold Pipeline

AlphaFold works in stages: search for evolutionary related sequences (MSA — multiple sequence alignment), extract pairwise residue relationships, then iteratively refine a 3D structure using an architecture called Evoformer (a transformer variant that jointly attends over MSA rows and residue pairs). The final structure module converts abstract representations into actual 3D coordinates using equivariant transformations that respect physical symmetries. RoseTTAFold achieves similar results with a three-track architecture that processes 1D sequence, 2D distance maps, and 3D coordinates simultaneously.

Weather prediction is another domain being disrupted. Pangu-Weather (Huawei) uses a 3D Swin Transformer trained on 39 years of ERA5 reanalysis data. It produces 7-day global forecasts in seconds that match or beat the European Centre for Medium-Range Weather Forecasts (ECMWF) operational model — which runs on a supercomputer for hours. GraphCast from DeepMind achieves similar results with a graph neural network on a mesh grid.

Fourier Neural Operators for PDEs

Fourier Neural Operators (FNO) solve partial differential equations by learning the mapping between function spaces. Instead of learning point-to-point mappings (like a standard neural network), FNO operates in Fourier space — it learns to manipulate frequency components of the solution. This makes it resolution-invariant: train on a coarse grid, evaluate on a fine grid. FNO is orders of magnitude faster than traditional PDE solvers for problems like fluid dynamics, heat transfer, and climate modeling.

The AlphaFold Effect

When AlphaFold's predictions were released for 200+ million proteins, it did not end structural biology — it accelerated it. Researchers now use AlphaFold predictions as starting points, focusing experimental effort on cases where the model is uncertain. This is the ideal human-AI collaboration: the model handles the routine, the scientist handles the frontier.

9. ML for Code

You are probably already using ML for code — GitHub Copilot, Cursor, Codeium. But how do these models learn to write code? Two training innovations made this work: Fill-in-the-Middle (FIM) training and execution feedback.

FIM Training

Standard language models are trained left-to-right: predict the next token given all previous tokens. But code completion often requires context from both before and after the cursor. FIM rearranges the training data: take a code document, split it into prefix, middle, and suffix, then train the model to predict the middle given the prefix and suffix.

# Fill-in-the-Middle (FIM) training format
# Original code:
# def fibonacci(n):
#     if n <= 1:
#         return n
#     return fibonacci(n-1) + fibonacci(n-2)

# FIM transformation for training:
def fim_transform(code, split_point_start, split_point_end):
    """Transform code into FIM format for training."""
    prefix = code[:split_point_start]
    middle = code[split_point_start:split_point_end]  # model learns to predict this
    suffix = code[split_point_end:]

    # PSM format (Prefix-Suffix-Middle):
    # Special tokens tell the model the structure
    fim_input = f"<|fim_prefix|>{prefix}<|fim_suffix|>{suffix}<|fim_middle|>"
    fim_target = middle
    return fim_input, fim_target

# At inference (code completion):
# User has typed prefix, cursor is in the middle, suffix exists below
# Model sees: <|fim_prefix|>def fib(n):\n    <|fim_suffix|>\n    return fib(n-1) + fib(n-2)<|fim_middle|>
# Model generates: "if n <= 1:\n        return n"

Code models also benefit from Abstract Syntax Tree (AST) awareness — understanding that code has structure beyond raw text. Some models are trained with AST-based masking or use tree-structured encoders. Retrieval-Augmented Generation (RAG) pulls relevant code snippets from the repository into the model's context — this is how Copilot knows about your codebase without being fine-tuned on it.

The execution feedback loop is a newer paradigm: generate code, run it, check if it passes tests, and feed the result back to the model for refinement. This is the core idea behind systems like AlphaCode and CodeContests. The model generates many candidate solutions, executes them against test cases, and filters or re-ranks based on actual correctness. This closes the gap between "code that looks right" and "code that actually works."

10. Cross-Domain Concepts

After touring nine domains, patterns emerge. Certain challenges — and solutions — repeat everywhere. Before you reach for ML in any domain, you should ask a more fundamental question: do I even need ML here?

ML vs Rules: A Decision Framework

Use rules when the logic is well-defined, changes rarely, and must be auditable. Use ML when the logic is too complex to specify by hand, the data is available to learn from, and the cost of occasional errors is manageable. Many production systems use both: rules for hard constraints (a patient's age cannot be negative), ML for soft predictions (this patient is at risk of readmission).

# Decision framework: when to use ML vs rules
decision_framework = {
    "use_rules_when": [
        "Logic can be written in < 50 if/else statements",

Model Risk Management

Model Risk Management (MRM) is a discipline born in finance but applicable everywhere. The idea: every model in production carries risk. That risk needs to be identified, measured, monitored, and mitigated. An MRM framework typically includes model validation (independent team checks the model), ongoing monitoring (does performance degrade over time?), and a model inventory (what models are running where, trained on what data, owned by whom?). In healthcare this is FDA compliance. In finance it is SR 11-7 guidance from the Federal Reserve. In autonomous vehicles it is safety cases and SOTIF analysis. Different vocabulary, same underlying discipline.

The Common Thread

Every domain-specific ML application shares a structure: unique data modalities, specialized preprocessing, domain-aware evaluation metrics, regulatory or safety constraints, and human-in-the-loop requirements. The ML algorithms themselves are often the same transformers, CNNs, and tree ensembles everywhere. What makes a healthcare ML engineer different from a finance ML engineer is not the models — it is the domain knowledge wrapped around those models.
