Multimodal Models

Chapter 16: Advanced Deep Learning Section 3 Flagship Deep Dive
TL;DR

CLIP taught us that contrastive pre-training on image-text pairs creates a universal embedding space good enough for zero-shot classification, image search, and as the visual backbone for everything that followed. Vision-Language Models (Flamingo, LLaVA, GPT-4V, Gemini) wire a visual encoder into an LLM through various fusion strategies. Text-to-image models (Stable Diffusion, DALL-E) use text embeddings to condition diffusion processes. The field is converging on unified architectures that treat every modality — text, image, audio, video — as tokens in a single transformer. If you build ML systems, multimodal capability is no longer optional; it's table stakes.

CLIP: The Foundation of Modern Multimodal AI

Every serious multimodal system built since 2021 traces its lineage back to CLIP (Contrastive Language-Image Pre-training). Understanding CLIP deeply isn't optional — it's the Rosetta Stone of this entire field.

The core idea is almost embarrassingly simple. Take 400 million image-text pairs scraped from the internet. Train two encoders — a Vision Transformer (ViT) for images, a text transformer for captions — and push matching pairs close together in a shared embedding space while pushing non-matching pairs apart. No ImageNet labels. No bounding boxes. Just natural language supervision at scale.

CLIP's Contrastive Pre-training

Given a batch of N image-text pairs, CLIP computes an N×N similarity matrix between all images and all texts. The diagonal entries (correct pairs) should have high similarity; everything off-diagonal should be low. The training objective is symmetric cross-entropy: image→text matching + text→image matching. The learned temperature parameter τ controls how peaked the softmax distribution is — lower τ means the model must be more confident about its matches.

Here's the contrastive loss that makes it all work:

import torch
import torch.nn as nn
import torch.nn.functional as F

class CLIPLoss(nn.Module):
    def __init__(self, temperature=0.07):
        super().__init__()
        # Temperature is *learned* during training
        self.log_temp = nn.Parameter(torch.log(torch.tensor(1.0 / temperature)))

    def forward(self, image_emb, text_emb):
        # L2 normalize — cosine similarity on the unit hypersphere
        image_emb = F.normalize(image_emb, dim=-1)
        text_emb = F.normalize(text_emb, dim=-1)

        temperature = self.log_temp.exp().clamp(max=100)

        # N×N similarity matrix
        logits = (image_emb @ text_emb.T) * temperature

        # Ground truth: diagonal entries are the correct matches
        labels = torch.arange(logits.shape[0], device=logits.device)

        # Symmetric loss: both directions matter equally
        loss_i2t = F.cross_entropy(logits, labels)
        loss_t2i = F.cross_entropy(logits.T, labels)
        return (loss_i2t + loss_t2i) / 2

# In practice: batch sizes of 32,768 are critical for CLIP
# More negatives = harder contrastive task = better representations
loss_fn = CLIPLoss()
img_emb = torch.randn(256, 512)   # 256 images, 512-d embeddings
txt_emb = torch.randn(256, 512)
loss = loss_fn(img_emb, txt_emb)
print(f"Loss: {loss.item():.4f}")
Batch Size Is the Secret Ingredient

CLIP's original paper used a batch size of 32,768. That's not a typo. With contrastive learning, each sample in the batch serves as a negative for every other sample. A batch of 256 gives you 255 negatives; a batch of 32K gives you 32,767. The quality of learned representations scales directly with the number of negatives. If you're training your own contrastive model and it's not working, insufficient batch size is suspect #1. Gradient accumulation or distributed training across many GPUs is how you get there.

Why CLIP Enables Zero-Shot Everything

The magic of CLIP isn't the training — it's what you get after training. You have two encoders that map images and text into the same space. That means:

Zero-shot classification: Encode the image. Encode text prompts like "a photo of a dog" and "a photo of a cat." Pick the prompt with highest cosine similarity. No fine-tuning. CLIP matched or beat fully-supervised ImageNet models on dozens of benchmarks it never trained on.

Image search: Encode all your images once, store the vectors. At query time, encode the text query and find the nearest image embeddings. This powers most modern visual search systems.

Content moderation: Encode suspect images and a set of policy-violation descriptions. Flag anything with high similarity to the violation prompts.

Here's a practical zero-shot classifier:

import torch
import torch.nn.functional as F
from transformers import CLIPModel, CLIPProcessor

def zero_shot_classify(image, candidate_labels, model_name="openai/clip-vit-base-patch32"):
    """Classify an image into one of the candidate labels — no training needed."""
    model = CLIPModel.from_pretrained(model_name)
    processor = CLIPProcessor.from_pretrained(model_name)

    # Prompt engineering matters! "a photo of a {label}" beats just "{label}"
    prompts = [f"a photo of a {label}" for label in candidate_labels]

    inputs = processor(text=prompts, images=image, return_tensors="pt", padding=True)
    outputs = model(**inputs)

    # Cosine similarity between image and each text prompt
    logits = outputs.logits_per_image.softmax(dim=-1)

    results = {label: score.item() for label, score in zip(candidate_labels, logits[0])}
    return dict(sorted(results.items(), key=lambda x: -x[1]))

# Example usage:
# from PIL import Image
# img = Image.open("golden_retriever.jpg")
# print(zero_shot_classify(img, ["dog", "cat", "bird", "fish"]))
# → {'dog': 0.94, 'bird': 0.03, 'cat': 0.02, 'fish': 0.01}
Prompt Engineering for CLIP

CLIP's zero-shot accuracy improves significantly with prompt engineering. "a photo of a dog" beats "dog" because the training data had natural captions, not single words. The original paper found that ensembling 80 different prompt templates ("a photo of a {label}", "a blurry photo of a {label}", "a sculpture of a {label}", etc.) improved accuracy by 3-5%. In production, you should always use templated prompts — and test several templates on your specific domain.

CLIP's Limitations (Be Honest About These)

CLIP is powerful, but let's not oversell it:

  • Counting: "Three cats on a sofa" and "one cat on a sofa" get similar embeddings. CLIP doesn't count.
  • Spatial reasoning: "A is left of B" vs "B is left of A" — often indistinguishable.
  • Fine-grained attributes: Struggles with color binding ("red car and blue truck" vs "blue car and red truck").
  • Domain-specific tasks: Medical images, satellite imagery, or specialized industrial inspection need fine-tuned variants (BiomedCLIP, RemoteCLIP, etc.).

These limitations stem from contrastive learning operating at the whole-image, whole-caption level. CLIP learns global alignment, not fine-grained grounding. Models like GLIP and Grounding DINO address this by adding region-level contrastive objectives.

Vision-Language Models: The Evolution

CLIP gives you a shared space for matching images and text, but it can't generate text about images. It can't answer "What's happening in this photo?" — it can only rank pre-written answers. The next leap was wiring visual encoders into language models so they could actually reason about images.

Flamingo: Pioneering Visual Chat

DeepMind's Flamingo (2022) was the first model that could take interleaved sequences of images and text — like "[image1] Describe this. [image2] How is this different?" — and generate coherent responses. The architecture is clever:

  1. A frozen vision encoder (NFNet) extracts image features
  2. A Perceiver Resampler compresses variable-length visual features into a fixed set of visual tokens (typically 64)
  3. Gated cross-attention layers are inserted into a frozen LLM, letting text generation attend to visual tokens

The Perceiver Resampler is worth understanding. Raw ViT output gives you 196+ tokens per image — too many to feed directly into an LLM at every layer. The resampler uses a small set of learned query tokens that cross-attend to the visual features, distilling them down to a compact representation. Think of it as an information bottleneck that asks: "What visual information does the language model actually need?"

The gated cross-attention is also elegant — each cross-attention layer has a learned gate initialized near zero, so the model starts by behaving like the original LLM and gradually learns to incorporate visual information. This preserves the LLM's language abilities while adding vision.

LLaVA: The Simplicity That Worked

LLaVA (Large Language and Vision Assistant) looked at Flamingo's careful architecture and asked: "What if we just... project CLIP features through a linear layer and concatenate them with text tokens?" Turns out that works remarkably well.

The LLaVA recipe:

  1. Take a pre-trained CLIP ViT-L/14 as the visual encoder
  2. Add a simple MLP projection layer (two linear layers with GELU) to map visual features into the LLM's embedding dimension
  3. Concatenate projected visual tokens with text tokens as input to the LLM
  4. Two-stage training: first align the projection layer on image-caption pairs (LLM frozen), then fine-tune everything on visual instruction data

LLaVA demonstrated something important: the hard part isn't the architecture — it's the training data. The authors used GPT-4 to generate high-quality visual instruction-following data from existing image-caption datasets. 150K instruction-following examples were enough to make a competitive visual chatbot. The open-source community ran with this, and LLaVA variants now rival commercial systems.

GPT-4V and Gemini: Commercial Scale

GPT-4V (GPT-4 with vision) was the first commercial multimodal model to achieve genuinely useful visual reasoning. It can read handwriting, interpret charts, debug code from screenshots, and understand memes. The exact architecture isn't published, but the capabilities suggest deep integration of visual features throughout the transformer stack — not just an adapter bolted on.

Gemini from Google took a different philosophy: natively multimodal from scratch. Rather than training a language model and then adding vision, Gemini was designed from the ground up to process text, images, audio, and video as first-class citizens. The model interleaves different modality tokens in a single sequence and applies attention across all of them.

The practical difference? Natively multimodal models tend to be better at tasks that require tight cross-modal reasoning (e.g., understanding a diagram with labels, or following along with a video while reading subtitles). Adapter-based approaches are easier to build and iterate on, but they have a ceiling on cross-modal integration depth.

How Multimodal Models Actually Work: Fusion Strategies

Every multimodal model needs to answer one question: how do you combine information from different modalities? This design choice has enormous practical implications for what the model can do, how fast it runs, and how easy it is to build.

The Practical Decision Framework

Start with late fusion if you're prototyping or combining pre-trained models. It's the safest bet: use CLIP for images, a text encoder for text, combine the pooled representations, done. Move to cross-attention when your task requires fine-grained reasoning across modalities ("what color is the object left of the cat?"). Use early fusion only if you're training from scratch at large scale and need maximum cross-modal integration — this is what Google and Anthropic do, not what most teams should attempt.

Here's cross-attention implemented so you can see exactly what's happening when "text reads from image":

import torch
import torch.nn as nn
import torch.nn.functional as F

class VisionLanguageCrossAttention(nn.Module):
    """Cross-attention: text tokens query image patches.
    This is the core mechanism in Flamingo, BLIP-2, etc."""
    def __init__(self, text_dim=768, vision_dim=1024, num_heads=8):
        super().__init__()
        self.num_heads = num_heads
        self.head_dim = text_dim // num_heads

        self.q_proj = nn.Linear(text_dim, text_dim)      # queries from text
        self.k_proj = nn.Linear(vision_dim, text_dim)     # keys from image
        self.v_proj = nn.Linear(vision_dim, text_dim)     # values from image
        self.out_proj = nn.Linear(text_dim, text_dim)
        self.gate = nn.Parameter(torch.zeros(1))          # Flamingo-style gate
        self.norm = nn.LayerNorm(text_dim)

    def forward(self, text_tokens, image_patches):
        """text_tokens: (B, T, text_dim), image_patches: (B, P, vision_dim)"""
        residual = text_tokens
        B, T, _ = text_tokens.shape
        P = image_patches.shape[1]

        Q = self.q_proj(text_tokens).view(B, T, self.num_heads, self.head_dim).transpose(1, 2)
        K = self.k_proj(image_patches).view(B, P, self.num_heads, self.head_dim).transpose(1, 2)
        V = self.v_proj(image_patches).view(B, P, self.num_heads, self.head_dim).transpose(1, 2)

        attn = (Q @ K.transpose(-2, -1)) * (self.head_dim ** -0.5)
        attn = F.softmax(attn, dim=-1)

        out = (attn @ V).transpose(1, 2).reshape(B, T, -1)
        out = self.out_proj(out)

        # Gated residual: starts at 0, model learns how much vision to mix in
        return self.norm(residual + torch.tanh(self.gate) * out)

# 20 text tokens attend to 196 image patches (14×14 ViT output)
xattn = VisionLanguageCrossAttention()
text = torch.randn(2, 20, 768)
image = torch.randn(2, 196, 1024)
fused = xattn(text, image)
print(f"Output: {fused.shape}")  # (2, 20, 768) — text enriched with visual info

Text-to-Image: How Text Conditioning Works

Text-to-image generation flips the script. Instead of understanding images through text, we're creating images from text. The core mechanism in modern systems: a diffusion model guided by text embeddings.

Stable Diffusion's Architecture

Stable Diffusion is the most widely-deployed text-to-image system and understanding its pipeline clarifies the whole field. There are three components:

  1. Text encoder (CLIP's text transformer): converts your prompt into a sequence of embedding vectors
  2. U-Net denoiser: operates in a compressed latent space (not on raw pixels — that's the "latent" in Latent Diffusion), progressively removing noise from a random starting point
  3. VAE decoder: converts the denoised latent back to pixel space

The text conditioning happens inside the U-Net through — you guessed it — cross-attention. At each denoising step, the U-Net's intermediate features generate queries, and the text embeddings provide keys and values. The model literally "reads" the prompt at every step of the generation process, deciding how to denoise each spatial region based on what the text says should be there.

Classifier-Free Guidance (CFG)

The quality trick behind every modern text-to-image model. During training, randomly drop the text conditioning (replace it with an empty prompt) some percentage of the time. At inference, compute two predictions: one with the prompt (ε_cond) and one without (ε_uncond). Then extrapolate: ε = ε_uncond + w × (ε_cond - ε_uncond). The guidance weight w controls the fidelity-diversity tradeoff. w=1 means no guidance (diverse but may ignore the prompt); w=7-12 is typical for good prompt adherence. Higher values give sharper, more prompt-faithful images but can become oversaturated.

DALL-E's Evolution

The DALL-E family shows how the field's thinking evolved:

  • DALL-E 1 (2021): Treated image generation as sequence prediction. Tokenize images into discrete codes with a dVAE, concatenate with text tokens, train a transformer to predict the sequence. Clever, but limited resolution.
  • DALL-E 2 (2022): Pivoted to diffusion. A "prior" model maps text CLIP embeddings → image CLIP embeddings. Then a diffusion decoder generates pixels from the image embedding. Operating in CLIP space gave it inherited text understanding.
  • DALL-E 3 (2023): The breakthrough was data, not architecture. Trained on detailed, LLM-rewritten captions instead of noisy alt-text. Integrated with ChatGPT for prompt rewriting. The lesson: better descriptions of training images matters more than a better model.
The Compositionality Problem

Current text-to-image models struggle with compositional prompts. "A red cube on top of a blue sphere" might give you a blue cube on a red sphere. The model understands all the individual concepts but fails to bind attributes to objects and maintain spatial relationships. This isn't a minor issue — it's a fundamental limitation of how text embeddings encode structure. Layout-conditioned generation and region-specific prompting are active research areas addressing this.

Audio, Video, and Beyond

Vision-language gets most of the attention, but the multimodal frontier extends to every sensory input humans process.

Audio-Language Models

Models like Whisper (speech→text) and AudioLDM (text→audio) operate on spectrograms or learned audio tokens. The interesting multimodal angle is systems that combine audio and vision — like audio-visual speech recognition that reads lips when the audio is noisy, or models that watch a video and predict what sounds should accompany it.

ImageBind from Meta demonstrated that you can create a shared embedding space across six modalities (images, text, audio, depth, thermal, IMU) by anchoring everything to images during training. You don't need paired data for every modality combination — just pair each modality with images, and the shared image anchor creates emergent cross-modal connections. Audio-to-text retrieval works even though the model never saw audio-text pairs.

Video Understanding

Video adds temporal reasoning to the mix. A Video Vision Transformer (ViViT) divides video into spatiotemporal patches and applies attention across space and time. The computational challenge is brutal: a 10-second video at 30fps has 300 frames × 196 patches = 58,800 tokens. Full attention over that is quadratic and impractical.

Solutions include factorized attention (spatial within frames, temporal across frames), token sampling, and — increasingly — just feeding keyframes to a powerful image-language model. GPT-4V and Gemini handle video by sampling frames at regular intervals and treating them as a sequence of images with temporal context. It's crude but surprisingly effective for many use cases.

Production Patterns: Building With Multimodal Models

Theory is nice. Let's talk about what you'd actually build.

CLIP Embeddings as Infrastructure

The most immediately useful multimodal capability for most engineers: CLIP embeddings as a universal representation layer. Pre-compute CLIP embeddings for your image corpus, store them in a vector database, and you have instant multimodal search.

import torch
import numpy as np
from transformers import CLIPModel, CLIPProcessor
from PIL import Image

class MultimodalSearchIndex:
    """Production-ready pattern: CLIP embeddings + vector similarity search."""
    def __init__(self, model_name="openai/clip-vit-base-patch32"):
        self.model = CLIPModel.from_pretrained(model_name)
        self.processor = CLIPProcessor.from_pretrained(model_name)
        self.image_embeddings = []
        self.metadata = []

    @torch.no_grad()
    def add_image(self, image, metadata=None):
        inputs = self.processor(images=image, return_tensors="pt")
        emb = self.model.get_image_features(**inputs)
        emb = emb / emb.norm(dim=-1, keepdim=True)  # L2 normalize
        self.image_embeddings.append(emb.squeeze().numpy())
        self.metadata.append(metadata)

    @torch.no_grad()
    def search(self, text_query, top_k=5):
        inputs = self.processor(text=[text_query], return_tensors="pt", padding=True)
        text_emb = self.model.get_text_features(**inputs)
        text_emb = text_emb / text_emb.norm(dim=-1, keepdim=True)

        if not self.image_embeddings:
            return []

        corpus = np.stack(self.image_embeddings)
        scores = (text_emb.numpy() @ corpus.T).squeeze()
        top_indices = np.argsort(scores)[::-1][:top_k]
        return [(self.metadata[i], float(scores[i])) for i in top_indices]

Multimodal RAG

Traditional RAG retrieves text chunks to ground an LLM's responses. Multimodal RAG extends this to images, diagrams, and mixed media. The pattern:

  1. Index your documents with both text embeddings (for text chunks) and CLIP embeddings (for images, figures, diagrams)
  2. At query time, retrieve relevant chunks and images based on the query
  3. Feed both retrieved text and images to a VLM (GPT-4V, Gemini, LLaVA) for answer generation

This is particularly powerful for technical documentation (architecture diagrams), e-commerce (product images), medical records (scans + notes), and any domain where the visual content carries information that text alone can't capture.

The Practical Multimodal RAG Stack

A realistic multimodal RAG pipeline: Indexing — use ColPali or a similar document-aware embedder that handles both text and visual layout; store in a vector DB (Weaviate, Qdrant, or Pinecone with multimodal support). Retrieval — hybrid search combining text BM25 with CLIP similarity. Generation — pass retrieved images + text to GPT-4V or an open-source VLM. The biggest gotcha: chunking strategy for documents with embedded figures. You need to keep images co-located with their referring text.

Handling Missing Modalities in Production

Real systems have messy inputs. The camera drops a frame. The user uploads an image with no description. The audio stream cuts out. Your model needs to degrade gracefully.

Two battle-tested approaches:

  • Modality dropout during training: randomly zero out entire modality inputs (20-30% of the time). The model learns to make predictions from any subset of available modalities.
  • Learned default tokens: when a modality is missing, substitute a learned "null" embedding (similar to BERT's [MASK] token). The model learns that this token means "no information available here" rather than "this modality contains a zero vector."

Evaluating Multimodal Models

Evaluation is where multimodal AI gets uncomfortable. For text, you have perplexity and BLEU. For images, FID and IS. For multimodal? You need to evaluate alignment between modalities, and that's fundamentally harder to automate.

Key challenges:

  • A generated image can be beautiful and completely wrong. High FID score (good image quality) says nothing about whether it matches the prompt.
  • VQA models can get right answers for wrong reasons — answering "what sport?" correctly because of uniform colors, not because it understood the action.
  • Human evaluation doesn't scale but remains the gold standard for alignment quality. Expect to include it for any serious deployment.

CLIPScore (cosine similarity between CLIP embeddings of generated image and prompt) has become the standard automated metric for text-image alignment. It's not perfect — it inherits all of CLIP's limitations — but it correlates well enough with human judgment for most practical purposes.

Where This Is All Going

The trajectory is unmistakable: separate unimodal models are being replaced by unified architectures that treat all modalities as tokens. GPT-4o processes text, images, and audio in a single model with real-time audio I/O. Gemini 1.5 handles million-token contexts mixing text, images, video, and code.

The "any-to-any" model — one system that takes any combination of inputs and generates any combination of outputs — is no longer theoretical. The remaining challenges are mostly engineering: efficient tokenization for each modality, managing the enormous context lengths that video and audio require, and training at sufficient scale. The research frontier has moved to grounding (spatial reasoning, counting, attribute binding) and reasoning (multimodal chain-of-thought that references visual elements in its reasoning steps).

For practitioners: if your system operates on a single modality, ask whether additional modalities would improve it. The answer is increasingly "yes," and the tools to build multimodal systems are now accessible enough to act on that answer.
