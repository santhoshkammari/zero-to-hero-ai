Equivariant Neural Networks and Geometric Deep Learning

Chapter 16: Advanced Deep Learning 4 subtopics
TL;DR

A molecule doesn't change when you rotate it. A physics simulation shouldn't either. Yet standard neural networks are blind to these symmetries — they waste capacity re-learning what a rotated input looks like from scratch. Equivariant neural networks bake symmetry directly into the architecture so the model respects geometric structure by construction. Geometric deep learning unifies CNNs, GNNs, and Transformers under a single mathematical framework based on symmetry groups acting on domains. The payoff is enormous: AlphaFold, molecular dynamics, particle physics — anywhere the data lives in a space with structure.

Foundations: Why Symmetry Matters

The Problem That Started It All

Imagine you're predicting whether a molecule is toxic. You feed a 3D point cloud of atom positions into your neural network and get a prediction. Now rotate that molecule 45 degrees. Same molecule. Same toxicity. But your network might give a completely different answer. That's a problem. You could augment your training data with every possible rotation — but there are infinitely many rotations in 3D. Good luck with that.

This is where the core idea lands: instead of teaching your model about symmetry through brute-force data augmentation, you build the symmetry into the architecture itself. The model becomes physically incapable of giving different answers for rotated versions of the same input. That's not a nice-to-have. For scientific applications, it's table stakes.

Invariance vs Equivariance

Invariance means the output doesn't change when you transform the input. Rotate a cat photo — it's still a cat. The classification label stays the same. Mathematically: if T is a transformation (like rotation), then f(T(x)) = f(x). The function ignores the transformation entirely.

Equivariance means the output transforms in the same way as the input. Rotate an image, and the segmentation mask should rotate with it. The pixel labels move with the pixels. Mathematically: f(T(x)) = T(f(x)). Apply the transformation, then compute — or compute, then apply the transformation. You get the same result either way.

Here's the key insight: invariance is a special case of equivariance where the output transformation is the identity (do nothing). Every invariant function is equivariant, but not every equivariant function is invariant. When you're predicting a single number (energy of a molecule), you want invariance. When you're predicting vectors (forces on atoms), you want equivariance — rotate the molecule and the force vectors should rotate too.

import torch
import torch.nn as nn

# Invariance example: output unchanged by transformation
def demonstrate_invariance():
    """Global average pooling is translation-invariant."""
    pool = nn.AdaptiveAvgPool2d(1)
    x = torch.randn(1, 3, 8, 8)

    # Shift the image by rolling pixels
    x_shifted = torch.roll(x, shifts=2, dims=-1)

    out_original = pool(x)
    out_shifted = pool(x_shifted)
    # These are approximately equal (exactly equal for circular shifts)
    print(f"Difference: {(out_original - out_shifted).abs().max():.6f}")

# Equivariance example: output transforms with input
def demonstrate_equivariance():
    """Standard convolution is translation-equivariant."""
    conv = nn.Conv2d(3, 16, kernel_size=3, padding=1)
    x = torch.randn(1, 3, 8, 8)

    x_shifted = torch.roll(x, shifts=2, dims=-1)

    out_then_shift = torch.roll(conv(x), shifts=2, dims=-1)
    shift_then_out = conv(x_shifted)
    # These should be nearly identical
    print(f"Equivariance error: {(out_then_shift - shift_then_out).abs().max():.6f}")

demonstrate_invariance()
demonstrate_equivariance()

Group Theory Basics — Only What You Need

A group in mathematics is a set of transformations with four properties: there's an identity transformation (do nothing), every transformation has an inverse (undo it), transformations can be composed (do one then another), and composition is associative. That's it. You don't need a PhD in abstract algebra. You need to know that symmetries form groups, and groups tell you what your network should be invariant or equivariant to.

The groups that matter most in geometric deep learning:

Translation group: shift things left, right, up, down. This is what CNNs already handle. A convolution filter that detects an edge at position (10, 10) also detects it at (50, 50). That's translation equivariance baked in.

SO(2): rotations in 2D. Spin a 2D image around its center. Useful for satellite imagery, microscopy — anything where "up" is arbitrary.

SO(3): rotations in 3D. Spin a molecule, a protein, a point cloud. This is the big one for scientific ML. The "SO" stands for Special Orthogonal — rotations that preserve distances and handedness.

SE(3): rotations AND translations in 3D. The full rigid-body motion group. If you pick up a molecule, rotate it, and set it down somewhere else, the physics doesn't change. SE(3) captures that full symmetry.

E(3): SE(3) plus reflections. Rotations, translations, and mirror flips. Some applications need reflection invariance (molecular energy), others explicitly don't (chirality in drug design — a molecule and its mirror image can have very different biological effects).

CNN Translation Equivariance — What You Already Know

You've been using equivariant networks all along. A standard convolutional neural network is equivariant to translations. Slide the input image 5 pixels to the right, and the feature maps slide 5 pixels to the right too. This isn't an accident — it's a consequence of weight sharing. Every spatial position uses the same learned filter. The architecture makes it structurally impossible to treat different positions differently.

But CNNs are NOT equivariant to rotations. Rotate an image 30 degrees, and the feature maps don't rotate in any predictable way. The network has to learn rotation invariance from augmented data. That's wasteful. It uses model capacity to rediscover a symmetry you already knew about.

Data Augmentation vs Built-In Equivariance

Data augmentation is the brute-force approach: train on every rotated, flipped, and translated version of your data. It works. It's also expensive, approximate, and sample-inefficient. You're essentially asking the model to learn the symmetry empirically from examples rather than encoding it mathematically.

Built-in equivariance is the principled approach. The model is equivariant by construction — no amount of adversarial input will break the symmetry. The practical benefits are real: you need less training data, you get better generalization, and the model is guaranteed to respect physical laws. The tradeoff? The architectures are more complex to implement and sometimes slower to run.

When to Use Which

Use data augmentation when the symmetry group is small and discrete (horizontal flips, 90° rotations). Use built-in equivariance when the symmetry group is large or continuous (full SO(3) rotations), when you have limited data, or when the application demands exact symmetry guarantees — as in physical simulations where conservation laws must hold precisely.

Architectures That Respect Geometry

Group-Equivariant Convolutions

The first major extension beyond translation equivariance came from Taco Cohen and Max Welling in 2016. The idea: instead of just sliding a filter across spatial positions, also rotate (and flip) the filter, and keep track of all the responses. A standard convolution produces one feature map per filter. A group convolution produces one feature map per filter per group element.

For the group p4 (90° rotations), each filter produces 4 feature maps — one for each rotation. The next layer convolves over both space and rotation, so it can detect patterns like "edge at angle θ followed by edge at angle θ+90°" without learning separate filters for each orientation.

import torch
import torch.nn as nn
import torch.nn.functional as F

class GroupConv2d(nn.Module):
    """
    Group-equivariant convolution for the p4 group (4 rotations: 0°, 90°, 180°, 270°).
    Input: (batch, channels, rotations, H, W)  -- or (batch, channels, H, W) for first layer
    Output: (batch, out_channels, 4, H, W)
    """
    def __init__(self, in_channels, out_channels, kernel_size, num_rotations=4,
                 first_layer=False):
        super().__init__()
        self.num_rotations = num_rotations
        self.first_layer = first_layer
        if first_layer:
            self.weight = nn.Parameter(
                torch.randn(out_channels, in_channels, kernel_size, kernel_size)
            )
        else:
            self.weight = nn.Parameter(
                torch.randn(out_channels, in_channels, num_rotations,
                            kernel_size, kernel_size)
            )
        nn.init.kaiming_uniform_(self.weight.view(out_channels, -1,
                                  kernel_size, kernel_size))

    def rotate_filter(self, w, k):
        """Rotate filter by k * 90 degrees."""
        return torch.rot90(w, k=k, dims=[-2, -1])

    def forward(self, x):
        outputs = []
        for r in range(self.num_rotations):
            if self.first_layer:
                # Rotate the filter and convolve
                rotated_w = self.rotate_filter(self.weight, r)
                out = F.conv2d(x, rotated_w, padding=1)
            else:
                # For each output rotation r, cycle the input rotation axis
                # and rotate the spatial filter
                rolled_weight = torch.roll(self.weight, shifts=r, dims=2)
                # Merge input channels and rotations for conv
                B, C, R, H, W = x.shape
                x_flat = x.reshape(B, C * R, H, W)
                w_flat = rolled_weight.reshape(
                    rolled_weight.shape[0], -1,
                    rolled_weight.shape[-2], rolled_weight.shape[-1]
                )
                w_rotated = self.rotate_filter(w_flat, r)
                out = F.conv2d(x_flat, w_rotated, padding=1)
            outputs.append(out)
        return torch.stack(outputs, dim=2)  # (B, C_out, num_rotations, H, W)

# Usage
conv1 = GroupConv2d(3, 16, kernel_size=3, first_layer=True)
conv2 = GroupConv2d(16, 32, kernel_size=3)
x = torch.randn(2, 3, 28, 28)
h = conv1(x)           # (2, 16, 4, 28, 28)
h = conv2(h)           # (2, 32, 4, 28, 28)
# Pool over rotations for invariant prediction
h_inv = h.mean(dim=2)  # (2, 32, 28, 28) — rotation invariant
print(f"After group conv1: {conv1(x).shape}")
print(f"After group conv2: {h.shape}")
print(f"Rotation-pooled:   {h_inv.shape}")

E(n)-Equivariant Neural Networks (EGNN)

EGNN, introduced by Satorras et al. in 2021, handles the full Euclidean group — rotations, translations, and reflections — on point clouds. The beauty is in its constraint: it only uses relative distances between points (which are invariant to E(n)) and updates positions using direction vectors (which are equivariant). No spherical harmonics, no Wigner-D matrices. Elegant and practical.

The update rules are clean. For each pair of nodes, compute a message based on their distance, features, and edge attributes. Update features by aggregating messages (invariant operation). Update positions by moving along the direction to neighbors, weighted by learned coefficients (equivariant operation). Positions transform correctly because the direction vector x_i - x_j rotates with the coordinate system.

import torch
import torch.nn as nn

class EGNNLayer(nn.Module):
    """
    E(n)-Equivariant Graph Neural Network layer.
    Operates on node features h and coordinates x.
    Equivariant to rotations, translations, and reflections.
    """
    def __init__(self, feature_dim, hidden_dim=128):
        super().__init__()
        # Edge model: takes feature pair + squared distance
        self.edge_mlp = nn.Sequential(
            nn.Linear(2 * feature_dim + 1, hidden_dim),
            nn.SiLU(),
            nn.Linear(hidden_dim, hidden_dim),
            nn.SiLU()
        )
        # Node model: updates node features from aggregated messages
        self.node_mlp = nn.Sequential(
            nn.Linear(feature_dim + hidden_dim, hidden_dim),
            nn.SiLU(),
            nn.Linear(hidden_dim, feature_dim)
        )
        # Coordinate model: scalar weight for position updates
        self.coord_mlp = nn.Sequential(
            nn.Linear(hidden_dim, hidden_dim),
            nn.SiLU(),
            nn.Linear(hidden_dim, 1, bias=False)
        )

    def forward(self, h, x, edge_index):
        """
        h: (N, feature_dim)  — node features (scalars, invariant)
        x: (N, 3)            — node coordinates (equivariant)
        edge_index: (2, E)   — edges [senders, receivers]
        """
        src, dst = edge_index
        # Relative positions and squared distances
        rel_pos = x[src] - x[dst]                         # (E, 3) equivariant
        dist_sq = (rel_pos ** 2).sum(dim=-1, keepdim=True) # (E, 1) invariant!

        # Edge messages from features + distance (all invariant inputs)
        edge_input = torch.cat([h[src], h[dst], dist_sq], dim=-1)
        m_ij = self.edge_mlp(edge_input)                   # (E, hidden)

        # Aggregate messages per node
        agg = torch.zeros_like(h[:, :m_ij.shape[-1]])
        # Use scatter_add for proper message aggregation
        agg = agg.new_zeros(h.shape[0], m_ij.shape[-1])
        agg.scatter_add_(0, dst.unsqueeze(-1).expand_as(m_ij), m_ij)

        # Update node features (invariant update)
        h_new = h + self.node_mlp(torch.cat([h, agg], dim=-1))

        # Update coordinates (equivariant update)
        # Scalar weight * direction vector = equivariant displacement
        coord_weight = self.coord_mlp(m_ij)                # (E, 1) invariant
        weighted_rel = rel_pos * coord_weight               # (E, 3) equivariant
        coord_agg = torch.zeros_like(x)
        coord_agg.scatter_add_(0, dst.unsqueeze(-1).expand_as(weighted_rel),
                               weighted_rel)
        x_new = x + coord_agg

        return h_new, x_new

# Demo: process a small molecule (5 atoms, fully connected)
layer = EGNNLayer(feature_dim=64)
N = 5
h = torch.randn(N, 64)
x = torch.randn(N, 3)
# Fully connected edges (no self-loops)
src = torch.arange(N).repeat_interleave(N - 1)
dst = torch.cat([torch.cat([torch.arange(i), torch.arange(i+1, N)])
                 for i in range(N)])
edge_index = torch.stack([src, dst])

h_out, x_out = layer(h, x, edge_index)
print(f"Input:  features {h.shape}, coords {x.shape}")
print(f"Output: features {h_out.shape}, coords {x_out.shape}")

# Verify equivariance: rotate input, check output rotates too
R = torch.tensor([[0., -1., 0.], [1., 0., 0.], [0., 0., 1.]])  # 90° around z
h_rot, x_rot = layer(h, x @ R.T, edge_index)
_, x_then_rot = layer(h, x, edge_index)
x_then_rot = x_then_rot @ R.T
print(f"Equivariance error: {(x_rot - x_then_rot).abs().max():.8f}")
Why Squared Distance?

EGNN uses squared distances ||x_i - x_j||² rather than raw distances ||x_i - x_j||. This avoids the gradient singularity at zero distance that comes from the square root. It's a small implementation detail that matters a lot during training — gradients stay well-behaved even when atoms are close together.

SE(3)-Transformers and Steerable CNNs

SE(3)-Transformers (Fuchs et al., 2020) combine the attention mechanism of Transformers with SE(3) equivariance. The key ingredient is type-l features: scalar features (type-0) are invariant, vector features (type-1) rotate like 3D vectors, and higher-order features (type-2 and above) transform according to Wigner-D matrices. Attention weights are computed from invariant quantities (distances, scalar features), while value updates respect the type structure.

Steerable CNNs generalize group convolutions using representation theory. Instead of physically rotating a filter, you express it in a basis of steerable functions — functions whose rotated versions are linear combinations of each other. This is mathematically cleaner and scales to continuous groups where you can't enumerate all transformations. The e3nn library implements this with irreducible representations (irreps) of SO(3) as the building blocks.

Spherical CNNs and Tensor Field Networks

Spherical CNNs define convolutions on the sphere S². Regular 2D convolutions break equivariance on the sphere because the grid distorts near the poles (think Mercator projection — Greenland isn't that big). Spherical CNNs work in the spectral domain using spherical harmonics — the natural Fourier basis for the sphere. Convolution becomes pointwise multiplication in the spectral domain, and equivariance under SO(3) rotations comes for free because spherical harmonics transform predictably under rotation.

Tensor Field Networks (Thomas et al., 2018) operate on 3D point clouds and use spherical harmonics to build equivariant features. Each point carries features decomposed into irreducible representations of SO(3). Messages between points are filtered through learned radial functions multiplied by spherical harmonics of the direction vector. The tensor product of irreps at the sender with the spherical harmonic filter produces irreps at the receiver — all while maintaining exact SE(3) equivariance.

# Spherical harmonics — the basis functions for SO(3)-equivariant features
# Using e3nn, the standard library for equivariant neural networks
# pip install e3nn

import torch
from e3nn import o3

# Spherical harmonics up to degree 2
# Y_0^0: 1 scalar (invariant)
# Y_1^{-1,0,1}: 3 components (transform like a vector)
# Y_2^{-2,-1,0,1,2}: 5 components (transform like a symmetric traceless tensor)
directions = torch.randn(10, 3)
directions = directions / directions.norm(dim=-1, keepdim=True)

# Compute spherical harmonics for these directions
Y = o3.spherical_harmonics(
    l=[0, 1, 2],        # degrees
    x=directions,
    normalize=True
)
print(f"Spherical harmonics shape: {Y.shape}")  # (10, 1+3+5) = (10, 9)

# Irreducible representations: the language of equivariant features
# "1x0e + 1x1o + 1x2e" = 1 scalar (even) + 1 vector (odd) + 1 rank-2 tensor (even)
irreps = o3.Irreps("2x0e + 1x1o + 1x2e")
print(f"Irreps: {irreps}")
print(f"Dimension: {irreps.dim}")  # 2*1 + 1*3 + 1*5 = 10

# Tensor product: the equivariant "multiplication" operation
tp = o3.FullyConnectedTensorProduct(
    irreps_in1="2x0e + 1x1o",   # 2 scalars + 1 vector = 5 dims
    irreps_in2="1x0e + 1x1o",   # 1 scalar + 1 vector = 4 dims
    irreps_out="2x0e + 1x1o",   # 2 scalars + 1 vector = 5 dims
)
x1 = torch.randn(10, 5)
x2 = torch.randn(10, 4)
result = tp(x1, x2)
print(f"Tensor product output: {result.shape}")  # (10, 5)

The Geometric Deep Learning Blueprint

Unifying Everything Under One Roof

In 2021, Bronstein, Bruna, Cohen, and Veličković published "Geometric Deep Learning: Grids, Groups, Graphs, Geodesics, and Gauges" — a paper that did something remarkable. It showed that CNNs, GNNs, Transformers, and equivariant networks are all special cases of a single principle: learning functions on domains with symmetry structure.

The unifying equation is deceptively compact. A message-passing layer on a domain Ω with symmetry group G looks like:

f(x) = σ( Σ_{y∈N(x)} ψ(x, y) ⊙ φ(features(y)) )

The differences between architectures are just different choices of domain, neighborhood, and symmetry group. A CNN has Ω = grid, G = translations, N(x) = local patch. A GNN has Ω = graph, G = permutations, N(x) = adjacent nodes. A Transformer has Ω = set, G = permutations, N(x) = all other elements. An equivariant network adds a continuous symmetry group like SO(3) on top.

The 5G Framework

The 5G framework organizes geometric deep learning along five domains, each adding structure:

Grids: the domain of classical CNNs. Regular lattice structure. Translation symmetry. Images, audio spectrograms, time series on uniform grids. The convolution operation exploits the grid's regularity — weight sharing across positions is the direct consequence of translation equivariance.

Groups: the domain of group-equivariant CNNs. Features live not only on spatial positions but on group elements too. Steerable CNNs, spherical CNNs. The convolution generalizes from translations to arbitrary symmetry groups — if your data has rotational symmetry, your features should live on the rotation group.

Graphs: the domain of GNNs. Irregular, non-Euclidean structure. Permutation symmetry — the node ordering is arbitrary. Social networks, molecular graphs, knowledge graphs. Message passing replaces convolution because there's no regular grid to slide a filter over.

Geodesics: the domain of meshes and manifolds. Surfaces like the cortex of the brain, the surface of a protein, or the Earth's terrain. The geometry isn't flat — distances are measured along the surface (geodesics), not through the ambient space. Intrinsic mesh convolutions generalize CNNs to curved surfaces.

Gauges: the most general domain. Manifolds without a canonical coordinate system. Every point has a local reference frame (a gauge), and the network must be equivariant to changes of gauge — choosing a different local coordinate system shouldn't change the answer. This is the deepest level of the hierarchy. Gauge equivariance on general manifolds subsumes all the other cases.

The Hierarchy

Gauges ⊃ Geodesics ⊃ Graphs ⊃ Groups ⊃ Grids. Each level adds structure and constraints. More structure means more symmetry to exploit, which means more efficient learning. A gauge-equivariant network on a flat manifold with a regular grid reduces to a standard CNN. The framework doesn't just classify existing architectures — it points toward new ones by asking: what symmetries does my domain have, and how do I build them in?

Gauge Equivariance on Manifolds

Gauge equivariance is the trickiest concept in the 5G hierarchy. On a flat grid, there's a global coordinate system — "right" means the same thing everywhere. On a curved manifold, there's no such luxury. At every point, you pick a local reference frame, and there's no canonical choice. A gauge transformation changes these local frames.

A gauge-equivariant network produces the same result regardless of how you choose local frames. This is borrowed directly from physics — gauge theories in particle physics have exactly the same structure. The practical implication: if your data lives on a surface (protein surfaces, brain cortex meshes, climate data on the sphere), gauge equivariance ensures your network doesn't accidentally depend on arbitrary choices of local coordinates.

Applications: Where Geometry Wins

Molecular Property Prediction

Molecules are 3D objects with atoms at specific coordinates. The energy of a molecule doesn't change if you rotate or translate it — that's E(3) invariance. The forces on each atom rotate with the molecule — that's E(3) equivariance. Networks like SchNet, DimeNet, PaiNN, and EGNN exploit these symmetries to predict molecular properties with remarkable accuracy. They're replacing expensive quantum chemistry calculations (DFT) for many practical applications, running orders of magnitude faster.

import torch
import torch.nn as nn

class SimpleEquivariantMoleculeNet(nn.Module):
    """
    Minimal E(3)-invariant network for molecular property prediction.
    Demonstrates the pattern: equivariant backbone → invariant pooling → prediction.
    """
    def __init__(self, num_atom_types=10, feature_dim=64, num_layers=3):
        super().__init__()
        self.atom_embedding = nn.Embedding(num_atom_types, feature_dim)
        self.layers = nn.ModuleList([
            EGNNLayer(feature_dim) for _ in range(num_layers)
        ])
        self.output_head = nn.Sequential(
            nn.Linear(feature_dim, feature_dim),
            nn.SiLU(),
            nn.Linear(feature_dim, 1)
        )

    def forward(self, atom_types, positions, edge_index, batch_index):
        """
        atom_types: (N,) int — atom type per node
        positions: (N, 3) — 3D coordinates
        edge_index: (2, E) — molecular bonds
        batch_index: (N,) — which molecule each atom belongs to
        """
        h = self.atom_embedding(atom_types)

        for layer in self.layers:
            h, positions = layer(h, positions, edge_index)

        # Per-atom predictions
        atom_energies = self.output_head(h)  # (N, 1)

        # Sum over atoms in each molecule (invariant aggregation)
        num_molecules = batch_index.max() + 1
        mol_energies = torch.zeros(num_molecules, 1, device=h.device)
        mol_energies.scatter_add_(0, batch_index.unsqueeze(-1), atom_energies)

        return mol_energies  # (num_molecules, 1) — E(3)-invariant!

Protein Structure Prediction — The AlphaFold Story

AlphaFold2 is the poster child for geometric deep learning. Predicting protein structure from amino acid sequence was an unsolved problem for 50 years. AlphaFold2 cracked it by building SE(3) equivariance into its structure module — the part that converts abstract representations into 3D coordinates.

The architecture is a masterclass in using the right equivariance at the right level. The Evoformer (the main trunk) operates on pairwise representations that encode distances and orientations — invariant quantities. The Structure Module takes these invariant representations and produces equivariant outputs: backbone frames (rotation + translation for each residue) that update iteratively. The Invariant Point Attention (IPA) mechanism computes attention scores from invariant quantities (distances in local frames) while producing equivariant coordinate updates. It's equivariance by design, not by accident.

The result: median GDT score of 92.4 at CASP14, where the next-best method scored 67. Not an incremental improvement. A paradigm shift. And the key architectural ingredient was geometric deep learning — respecting the SE(3) symmetry of 3D space.

Physics Simulations

Equivariant networks are replacing classical numerical solvers for many physics simulations. The argument is simple: physical laws are equivariant. Newton's laws don't change when you rotate your coordinate system. Conservation of momentum holds regardless of your reference frame. An equivariant neural network respects these laws by construction. A non-equivariant network might learn to violate them in ways that produce plausible-looking but physically wrong results.

Graph Network Simulators (GNS) model particle-based physics. Each particle is a node, interactions are edges. Learned message passing replaces hand-coded force calculations. When you add E(3) equivariance, the simulator automatically conserves angular momentum — because E(3)-equivariant functions can't produce torque from nothing. This isn't a soft constraint learned from data. It's a hard guarantee from the architecture.

Particle Physics

At the Large Hadron Collider, detectors record millions of particle collisions per second. Each collision produces a spray of particles — a point cloud with energy, momentum, and charge at each point. The physics is Lorentz-invariant: the laws don't change under rotations or boosts (changes of reference frame). Lorentz-equivariant networks (like LorentzNet) build this symmetry in, dramatically improving jet classification and particle identification tasks compared to non-equivariant baselines. When your training data is expensive (each collision takes real beam time), sample efficiency from built-in equivariance matters enormously.

The Bigger Picture

Geometric deep learning isn't a niche subfield anymore. It's a lens for thinking about all of deep learning. Whenever you design a neural network, ask: what are the symmetries of my problem? What should be invariant? What should be equivariant? If you can answer those questions and build the answers into your architecture, you'll get a model that's more data-efficient, more generalizable, and more scientifically trustworthy. That's not ideology — it's engineering.
