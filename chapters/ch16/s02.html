Interpretability and Explainability

Chapter 16: Responsible AI & MLOps Section 2 5 major topics
TL;DR

Your model says "reject the loan application" — but why? Interpretability is about cracking open the black box so humans can understand, trust, and debug model decisions. Model-agnostic methods (LIME, SHAP) work on any model by probing inputs and outputs. Gradient-based methods (saliency maps, Grad-CAM, integrated gradients) exploit the chain rule to trace which input features a neural network actually relied on. Attention weights look tempting as explanations but are surprisingly unreliable. Concept-based methods (TCAV, network dissection) let you ask higher-level questions like "did the model use the concept of stripes?" And none of this matters unless you evaluate your explanations — faithfulness, stability, and sanity checks separate real insight from comforting illusions.

Why Interpretability Matters

Imagine a deep learning model tells a doctor that a patient has a 94% chance of developing sepsis within 12 hours. The doctor has two choices: trust the number blindly, or demand to know what the model saw that led to that prediction. In high-stakes settings — medical diagnosis, criminal sentencing, autonomous driving, loan approvals — "the model said so" is not a valid answer. Regulators require explanations. Patients deserve them. Engineers need them to find bugs.

Interpretability is the degree to which a human can understand the cause of a decision. Explainability is the degree to which the internal mechanics of a model can be described in human terms. Some people use these interchangeably. Others draw a subtle distinction: interpretability means the model is inherently transparent (like a decision tree), while explainability means you apply a post-hoc technique to explain an opaque model (like a deep neural network). For our purposes, think of them as two sides of the same coin — making model behavior understandable to humans.

There's also a practical reason beyond ethics: debugging. If your image classifier achieves 99% accuracy on the test set but you discover through Grad-CAM that it learned to recognize the hospital logo watermark instead of actual pathology, you've caught a catastrophic shortcut. Interpretability tools are your microscope for inspecting what the model actually learned versus what you hoped it learned.

Model-Agnostic Methods

Model-agnostic methods treat the model as a black box. They don't care whether it's a random forest, a neural network, or an ensemble of seventeen gradient-boosted trees stacked in a trench coat. They only need the ability to feed inputs and observe outputs. This makes them incredibly versatile — you can swap the model underneath without changing your explanation pipeline.

LIME (Local Interpretable Model-Agnostic Explanations)

LIME answers a focused question: why did the model make this particular prediction for this particular input? The core idea is beautifully pragmatic. Even if the global decision boundary is insanely complex, the local neighborhood around any single prediction is often approximately linear. So LIME fits a simple, interpretable model (usually a linear regression or sparse linear model) to that local neighborhood.

Here's the recipe. Take the input you want to explain. Generate a bunch of perturbed versions — for tabular data, randomly change some feature values; for images, randomly mask superpixels; for text, randomly drop words. Feed all these perturbed inputs through the original model to get predictions. Now weight these samples by how close they are to the original input (closer neighbors matter more). Fit a linear model on the weighted samples. The coefficients of that linear model tell you which features pushed the prediction up or down, locally.

import lime
import lime.lime_tabular
import numpy as np
from sklearn.ensemble import GradientBoostingClassifier

# Train any black-box model
model = GradientBoostingClassifier(n_estimators=200)
model.fit(X_train, y_train)

# Create a LIME explainer for tabular data
explainer = lime.lime_tabular.LimeTabularExplainer(
    training_data=X_train,
    feature_names=feature_names,
    class_names=['Rejected', 'Approved'],
    mode='classification'
)

# Explain a single prediction
instance = X_test[42]
explanation = explainer.explain_instance(
    instance,
    model.predict_proba,
    num_features=10,       # top 10 contributing features
    num_samples=5000       # how many perturbations to generate
)

# Show which features drove this prediction
for feature, weight in explanation.as_list():
    print(f"{feature}: {weight:+.4f}")
# Example output:
# income > 75000: +0.3421
# debt_to_income < 0.3: +0.2187
# credit_score < 620: -0.4102

The catch with LIME is that the explanation depends on the perturbation strategy and the neighborhood size. Change how you generate neighbors, and you might get a different explanation. This is a real stability concern we'll revisit in the evaluation section.

SHAP (SHapley Additive exPlanations)

SHAP brings game theory into machine learning. It's rooted in Shapley values, a concept from cooperative game theory that answers: "how much did each player contribute to the total payout?" Replace "players" with "features" and "payout" with "model prediction," and you've got SHAP.

The Shapley value for a feature is computed by looking at every possible coalition (subset) of features, adding the feature to each coalition, and measuring the marginal contribution. Then you average across all possible orderings. This gives you a mathematically principled attribution that satisfies some very nice properties: the contributions sum to the difference between the prediction and the average prediction (efficiency), identical features get identical attributions (symmetry), and a feature that never changes any prediction gets zero attribution (dummy).

The problem? Computing exact Shapley values requires evaluating the model for every possible subset of features. With n features, that's 2n evaluations. For any real-world model, that's computationally intractable. Enter the approximations.

Kernel SHAP combines the ideas of LIME and Shapley values. It uses a specially designed weighting kernel in a linear regression framework to approximate Shapley values. Think of it as LIME but with a theoretically grounded weighting scheme instead of an ad-hoc distance kernel. It works on any model but can be slow for high-dimensional inputs.

Tree SHAP is a specialized algorithm for tree-based models (random forests, XGBoost, LightGBM). It exploits the tree structure to compute exact Shapley values in polynomial time — O(TLD2) where T is the number of trees, L the maximum number of leaves, and D the maximum depth. This makes it blazingly fast and exact, which is why SHAP became the go-to explanation tool for tabular ML.

import shap
import xgboost as xgb

# Train an XGBoost model
model = xgb.XGBClassifier(n_estimators=300, max_depth=6)
model.fit(X_train, y_train)

# Tree SHAP — fast and exact for tree models
explainer = shap.TreeExplainer(model)
shap_values = explainer.shap_values(X_test)

# Visualize global feature importance
shap.summary_plot(shap_values, X_test, feature_names=feature_names)

# Explain a single prediction with a force plot
shap.force_plot(
    explainer.expected_value,
    shap_values[42],         # SHAP values for instance 42
    X_test[42],
    feature_names=feature_names
)

# For deep learning models, use DeepExplainer or GradientExplainer
deep_model = trained_pytorch_model
deep_explainer = shap.DeepExplainer(deep_model, background_data)
deep_shap_values = deep_explainer.shap_values(test_images)
SHAP vs LIME: When to Use Which

Use Tree SHAP whenever you have a tree-based model — it's fast, exact, and theoretically grounded. Use Kernel SHAP or LIME for neural networks or other non-tree models. LIME is faster but less stable; Kernel SHAP is slower but offers stronger theoretical guarantees. For deep learning specifically, consider gradient-based methods instead — they're much faster and leverage the model's internals.

Permutation Feature Importance

Permutation feature importance measures how much the model's performance drops when you shuffle a single feature's values across the dataset. If scrambling a feature tanks the accuracy, that feature was important. If nothing changes, the model didn't rely on it.

The algorithm is dead simple: measure baseline performance, then for each feature, randomly permute its column, re-evaluate performance, and record the drop. The beauty is that it captures the full effect of a feature, including interactions, because destroying the feature's relationship with the target also destroys its interactions with other features. The danger is that it can be misleading when features are correlated — permuting one correlated feature creates unrealistic data points that the model has never seen, potentially underestimating or overestimating importance.

Partial Dependence Plots and ICE Curves

A Partial Dependence Plot (PDP) shows the marginal effect of one or two features on the predicted outcome, averaged over all other features. You pick a feature, grid out its range, and for each grid value you replace that feature's value across all samples and average the predictions. The resulting curve shows how the prediction changes as the feature moves from low to high, holding everything else constant (on average).

Individual Conditional Expectation (ICE) curves are the disaggregated version. Instead of averaging, you plot one curve per instance. This reveals heterogeneous effects. Maybe increasing income raises predicted loan approval for most applicants, but for a subgroup with high debt, it barely moves the needle. PDPs would hide that — ICE curves expose it.

PDPs and ICE curves share the same limitation as permutation importance: they assume features are independent. When features are correlated, marginalizing over one creates impossible combinations. A 25-year-old with 30 years of work experience doesn't exist, but the PDP blissfully averages over that ghost.

Gradient-Based Methods

If your model is a differentiable neural network, you have a superpower that model-agnostic methods can't match: gradients. The chain rule gives you, for free, the derivative of the output with respect to every input. That derivative tells you which input dimensions the model is most sensitive to. Gradient-based methods exploit this to produce explanations that are fast (a single backward pass) and model-specific (they actually look inside the network).

Vanilla Gradients (Saliency Maps)

The simplest gradient-based explanation: compute the gradient of the output class score with respect to the input pixels. Large gradient magnitudes indicate pixels that, if changed slightly, would significantly change the prediction. This produces a saliency map — a heatmap of "what the model is sensitive to."

import torch
import torch.nn.functional as F

def compute_saliency_map(model, input_tensor, target_class):
    """
    Compute vanilla gradient saliency map.
    
    Args:
        model: trained PyTorch model
        input_tensor: (1, C, H, W) input image tensor
        target_class: integer class index to explain
    Returns:
        saliency: (H, W) numpy array of gradient magnitudes
    """
    model.eval()
    input_tensor = input_tensor.clone().detach().requires_grad_(True)
    
    # Forward pass
    output = model(input_tensor)
    score = output[0, target_class]
    
    # Backward pass — gradient of class score w.r.t. input
    score.backward()
    
    # Take absolute value, max across color channels
    saliency = input_tensor.grad.data.abs()
    saliency, _ = saliency.max(dim=1)  # (1, H, W)
    
    return saliency.squeeze().cpu().numpy()

# Usage
saliency = compute_saliency_map(resnet, image_tensor, target_class=281)

Vanilla gradients are fast but noisy. The gradient at a single point captures infinitesimal sensitivity, which can be dominated by high-frequency fluctuations rather than semantically meaningful regions. The resulting saliency maps often look like TV static sprinkled vaguely around the object.

Grad-CAM

Grad-CAM (Gradient-weighted Class Activation Mapping) is one of the most popular visual explanation techniques for CNNs. Instead of looking at gradients of the input, it looks at gradients flowing into the last convolutional layer. The insight: the last conv layer still retains spatial information (it's a feature map with height and width), but it's also high-level enough to capture semantic concepts. By weighting each feature map channel by the average gradient of the target class score with respect to that channel, you get a coarse localization map showing which spatial regions the model "looked at" for a given class.

import torch
import torch.nn.functional as F
import numpy as np

class GradCAM:
    def __init__(self, model, target_layer):
        self.model = model
        self.model.eval()
        self.gradients = None
        self.activations = None
        
        # Register hooks on the target convolutional layer
        target_layer.register_forward_hook(self._save_activation)
        target_layer.register_full_backward_hook(self._save_gradient)
    
    def _save_activation(self, module, input, output):
        self.activations = output.detach()
    
    def _save_gradient(self, module, grad_input, grad_output):
        self.gradients = grad_output[0].detach()
    
    def generate(self, input_tensor, target_class):
        # Forward pass
        output = self.model(input_tensor)
        
        # Zero all gradients
        self.model.zero_grad()
        
        # Backward pass for target class
        target_score = output[0, target_class]
        target_score.backward()
        
        # Global average pooling of gradients -> channel weights
        # gradients shape: (1, C, H, W)
        weights = self.gradients.mean(dim=(2, 3), keepdim=True)  # (1, C, 1, 1)
        
        # Weighted combination of activation maps
        cam = (weights * self.activations).sum(dim=1, keepdim=True)  # (1, 1, H, W)
        
        # ReLU — we only care about features that positively
        # influence the target class
        cam = F.relu(cam)
        
        # Normalize to [0, 1]
        cam = cam - cam.min()
        cam = cam / (cam.max() + 1e-8)
        
        # Upsample to input resolution
        cam = F.interpolate(cam, size=input_tensor.shape[2:],
                           mode='bilinear', align_corners=False)
        
        return cam.squeeze().cpu().numpy()

# Usage with a ResNet
import torchvision.models as models
resnet = models.resnet50(pretrained=True)
# Target the last conv layer in ResNet's layer4
grad_cam = GradCAM(resnet, resnet.layer4[-1].conv3)

cam = grad_cam.generate(image_tensor, target_class=281)
# cam is now a (H, W) heatmap you can overlay on the original image

Grad-CAM++ extends this idea by using a weighted average of pixel-wise gradients (instead of global average pooling) to better handle cases where multiple instances of a class appear in the image or when the object only partially fills the frame.

Grad-CAM Shows "Where," Not "Why"

Grad-CAM tells you which spatial region was important for the prediction, but it doesn't tell you what specific features within that region mattered. A Grad-CAM heatmap highlighting a dog's face doesn't tell you whether the model used the nose shape, fur texture, or ear position. For finer-grained explanations, you need methods like integrated gradients or concept-based approaches.

Integrated Gradients

Integrated Gradients addresses a fundamental flaw in vanilla gradients: they only capture sensitivity at a single point. A feature could be critically important to the prediction but happen to have a near-zero gradient at the current input (imagine a saturated sigmoid). Integrated gradients fix this by accumulating gradients along a straight-line path from a baseline (typically a black image or zero vector) to the actual input.

Mathematically, for each input dimension, you integrate the gradient from baseline to input. In practice, you approximate this integral by taking small steps along the path and summing the gradients. The method satisfies two key axioms: completeness (attributions sum to the difference between the output at the input and the output at the baseline) and sensitivity (if a feature causes any change in the output, it receives non-zero attribution).

import torch
import numpy as np

def integrated_gradients(model, input_tensor, target_class,
                         baseline=None, steps=200):
    """
    Compute integrated gradients attribution.
    
    Args:
        model: trained PyTorch model
        input_tensor: (1, C, H, W) input image
        target_class: class index to explain
        baseline: reference input (default: zeros)
        steps: number of interpolation steps
    Returns:
        attributions: (C, H, W) numpy array
    """
    model.eval()
    
    if baseline is None:
        baseline = torch.zeros_like(input_tensor)
    
    # Generate interpolated inputs along the path
    # alpha goes from 0 (baseline) to 1 (input)
    alphas = torch.linspace(0, 1, steps + 1, device=input_tensor.device)
    
    # Compute gradients at each interpolation point
    gradient_sum = torch.zeros_like(input_tensor)
    
    for alpha in alphas:
        interpolated = baseline + alpha * (input_tensor - baseline)
        interpolated = interpolated.clone().detach().requires_grad_(True)
        
        output = model(interpolated)
        score = output[0, target_class]
        score.backward()
        
        gradient_sum += interpolated.grad.data
    
    # Average the gradients (trapezoidal approximation)
    avg_gradients = gradient_sum / (steps + 1)
    
    # Scale by the input - baseline difference
    attributions = (input_tensor - baseline) * avg_gradients
    
    return attributions.squeeze().cpu().numpy()

# Usage
attrs = integrated_gradients(resnet, image_tensor, target_class=281)
# attrs shape: (3, H, W) — attribution per channel per pixel

SmoothGrad

SmoothGrad takes a different angle on the noise problem. Instead of integrating along a path, it adds Gaussian noise to the input multiple times, computes the gradient each time, and averages all the gradient maps. The idea is that the noise jitters the input around the local neighborhood, and averaging washes out the high-frequency noise while preserving the consistent signal. Typical settings use 50–200 noisy samples with noise standard deviation of 10–20% of the input range.

Guided Backpropagation and Layer-wise Relevance Propagation

Guided Backpropagation modifies the backward pass through ReLU layers: during backprop, it zeros out gradients that are negative in the top-down signal (not only those corresponding to negative activations in the forward pass). This produces visually sharper attributions than vanilla gradients. The resulting maps tend to look like edge detectors — crisp but not necessarily faithful to the model's actual reasoning.

Layer-wise Relevance Propagation (LRP) works top-down, distributing the model's output "relevance" backward through the layers using conservation rules. At each layer, the total relevance flowing in equals the total relevance flowing out. Different propagation rules (LRP-0, LRP-ε, LRP-γ) control how relevance is split among neurons. LRP has the advantage of producing signed attributions (positive = evidence for the class, negative = evidence against) and can be applied layer by layer to understand which neurons at each depth contributed most.

Attention-Based Interpretability

Transformers have attention weights baked right into their architecture. Every attention head produces a matrix showing how much each token attends to every other token. It's tempting to treat these weights as explanations: "The model predicted 'positive' because the word 'excellent' attended heavily to 'movie'." But the story is more complicated than it appears.

Attention Visualization

The most basic approach: extract the attention weight matrices and visualize them as heatmaps. For a given layer and head, you get a matrix where entry (i, j) shows how much token i attended to token j. Tools like BertViz make this easy. You can see patterns emerge — some heads specialize in attending to the next token, others attend to separators, others capture syntactic relationships like subject-verb agreement.

Attention Rollout

Attention Rollout addresses a key limitation of looking at single-layer attention: tokens don't flow through one layer in isolation. Information mixes across layers. Rollout computes a combined attention map by recursively multiplying attention matrices across layers, accounting for residual connections (which act like identity mappings, so you average each layer's attention with the identity matrix before multiplying). The result is a more holistic view of how information flows from input tokens to the final representation.

The Attention-as-Explanation Debate

Here's where things get contentious. In 2019, Jain and Wallace published "Attention is not Explanation," showing that (a) attention weights often don't correlate with gradient-based feature importance measures, and (b) you can find alternative attention distributions that produce the same predictions. If multiple explanations are consistent with the same output, how can any single one be "the" explanation?

Wiegreffe and Pinter countered with "Attention is not not Explanation," arguing that the existence of alternative attention patterns doesn't invalidate the original ones, and that under certain conditions attention weights do provide meaningful signal about which inputs matter.

The current consensus leans toward caution: attention weights are a useful diagnostic tool, but they are not a reliable standalone explanation method. They show where the model could be looking but not necessarily where it is looking for purposes of making the decision. Use them as one signal among many, not as ground truth.

Don't Trust Attention Alone

Presenting raw attention weights to end users as "this is what the model focused on" is misleading. Attention weights are intermediate computational quantities, not causal explanations. A token can have high attention weight but contribute nothing to the final output (because the value vector is near-zero or gets overwritten by subsequent layers). Always cross-validate attention-based explanations with gradient or perturbation methods.

BERTology: Probing Transformers

BERTology is the subfield devoted to understanding what BERT-like models learn internally. A common technique is probing: freeze the pretrained model, extract hidden representations from a specific layer, and train a simple linear classifier (the probe) to predict some linguistic property — part-of-speech tags, dependency relations, named entity types, coreference chains.

If the probe achieves high accuracy, the representation at that layer encodes that linguistic property. Researchers have found that lower layers tend to capture surface-level features (POS tags), middle layers capture syntax (parse trees), and upper layers capture semantics (entity types, relations). This gives us a rough dissection map of what each layer is doing.

The caveat: a linear probe might succeed not because the representation explicitly encodes the property, but because any high-dimensional representation contains enough information for a linear classifier to latch onto incidental correlations. More rigorous probing approaches use "control tasks" — training the same probe on randomized labels to establish a baseline — and measure selectivity (how much better the probe does on real labels versus random ones).

Concept-Based Explanations

Feature-level explanations tell you "pixel (127, 42) mattered" or "the word 'denied' had high attribution." That's useful for debugging, but it's not how humans think. A doctor doesn't reason about individual pixels — they reason about concepts like "enlarged heart" or "fluid in lungs." Concept-based methods bridge this gap by connecting model behavior to human-understandable concepts.

TCAV (Testing with Concept Activation Vectors)

TCAV lets you ask: "How important is the concept of 'stripes' to the model's prediction of 'zebra'?" You define a concept by providing a set of positive examples (images with stripes) and negative examples (random images without stripes). Then you train a linear classifier in the model's internal activation space to separate the concept from the non-concept. The normal vector of that decision boundary is the Concept Activation Vector (CAV).

To measure the concept's importance, you compute directional derivatives: how much does the model's prediction of "zebra" change when you move in the direction of the "stripes" CAV? If the directional derivative is positive for most zebra images, the model is positively influenced by the presence of stripes — exactly what you'd hope. TCAV reports the fraction of inputs for which the concept positively influences the prediction, giving you a global, human-interpretable importance score.

Network Dissection

Network Dissection takes a bottom-up approach: instead of starting with a concept and asking how important it is, you start with individual neurons and ask what concept each neuron has learned. For each neuron in a convolutional layer, you feed a large dataset through the network, record the neuron's activation map, threshold it to create a binary mask, and compare that mask against ground-truth segmentation masks for known concepts (objects, parts, textures, colors, materials, scenes).

If a neuron's activation map aligns strongly with "dog face" segmentation masks, that neuron is a "dog face detector." This approach has revealed that neural networks learn surprisingly interpretable internal representations — individual neurons that detect specific textures, object parts, or even entire objects — without any explicit supervision for those concepts.

Concept Bottleneck Models

Concept Bottleneck Models take interpretability further by building it into the architecture. Instead of a direct mapping from inputs to outputs, the model first predicts a set of human-defined concepts (e.g., "has wings," "has yellow breast," "has pointed beak"), then predicts the final class from those concept predictions. The bottleneck is the concept layer — a human-interpretable intermediate representation.

The payoff is powerful: you can inspect the concept predictions to understand why the model made its decision ("It predicted 'goldfinch' because it detected: wings=yes, yellow breast=yes, small size=yes"). Even better, if a concept prediction is wrong, a human expert can intervene and correct it, and the correction propagates to the final output. The cost is that you need concept annotations for your training data, and constraining the model to reason through a fixed set of concepts can reduce accuracy compared to an unconstrained end-to-end model.

Prototype Networks for Interpretability

Prototype networks explain predictions by pointing to concrete examples: "This image is classified as a robin because this patch looks like prototypical robin-breast #3 from the training data, and this patch looks like prototypical robin-wing #7." The model learns a set of prototype vectors during training. At inference time, it compares the input's learned representation to each prototype, and the prediction is based on the similarity scores.

This is enormously intuitive. Instead of abstract feature attributions, you can show a user: "Here are the training examples the model thinks your input looks most like." The challenge is ensuring the prototypes remain truly interpretable and don't degenerate into abstract high-dimensional points that don't correspond to recognizable examples.

The Interpretability–Accuracy Tradeoff

There's a persistent belief that interpretable models are less accurate. It's not always true. For tabular data, well-tuned interpretable models (GAMs, rule lists) often match or approach the accuracy of black-box models. The tradeoff is more real for vision and NLP, where the raw capacity of deep neural networks is harder to replicate with constrained architectures. Concept bottleneck models typically lose 1–5% accuracy compared to their unconstrained counterparts — a price many practitioners are willing to pay for auditability.

Evaluating Explanations

Here's the uncomfortable truth: anyone can generate a heatmap and overlay it on an image. The question is whether the explanation actually reflects the model's reasoning, or whether it's a compelling-looking fiction. Evaluating explanations is hard, and it's an active area of research. But we do have some tools.

Faithfulness Metrics

Faithfulness measures whether the explanation accurately reflects the model's true decision process. A faithful explanation should identify features that, when removed, actually change the prediction. The most common test: rank features by their attributed importance, remove them from most-important to least-important, and measure how quickly the model's prediction changes. A faithful explanation should cause rapid degradation — if you remove the supposedly most important features and nothing happens, the explanation was lying to you.

The complementary test is insertion: start from a blank baseline and add features in order of attributed importance. A good explanation should cause the prediction confidence to rise quickly. Plotting both deletion and insertion curves gives you the "area under the deletion/insertion curve" metric — lower deletion AUC and higher insertion AUC indicate better faithfulness.

Stability and Robustness

If you slightly perturb the input (add imperceptible noise to an image), does the explanation change dramatically? If yes, the explanation is unstable. Humans expect continuity: similar inputs should have similar explanations. Stability is measured by computing explanations for an input and its perturbed version, then measuring the distance between the two explanation maps (e.g., Spearman rank correlation of feature importance scores).

Some methods are notoriously unstable. Vanilla gradients can flip dramatically with tiny input changes. LIME's stochastic perturbation sampling means you might get different explanations on different runs for the same input. Integrated gradients and SHAP tend to be more stable, which is one reason they're preferred in practice.

Human Evaluation

Ultimately, explanations exist for humans. Automated metrics are proxies, but the real test is whether a human can use the explanation to make better decisions, catch model errors, or build appropriate trust. Human evaluation studies typically measure:

Simulatability — Can a human, given the explanation, predict what the model will do on a new input? If explanations help humans simulate the model more accurately, they're useful. Trust calibration — Do explanations help humans know when to trust the model and when to override it? Good explanations should increase trust on correct predictions and decrease trust on incorrect ones. Debugging utility — Can an ML engineer use the explanation to identify and fix a model bug?

Sanity Checks for Saliency Maps

In a landmark 2018 paper, Adebayo et al. proposed simple but devastating sanity checks. The idea: if your explanation method is truly reflecting learned model behavior, then randomizing the model's weights should produce a different explanation. They tested popular methods by progressively randomizing network layers from the top down and checking whether the saliency maps changed.

The results were sobering. Some methods — particularly guided backpropagation and guided Grad-CAM — produced visually similar saliency maps regardless of whether the model was trained or randomly initialized. These methods were essentially acting as edge detectors on the input image, not reflecting anything the model actually learned. Methods that passed the sanity check (vanilla gradients, Grad-CAM, integrated gradients) did produce different explanations for trained versus random models.

Always Run Sanity Checks

Before deploying any explanation method in production, run the model randomization test. Compare explanations from your trained model against explanations from a randomly initialized version of the same architecture. If they look similar, your explanation method is not reflecting learned behavior — it's reflecting input structure. Also compare explanations when training labels are randomized; a faithful method should produce different attributions when the model learns different decision rules.

The field of interpretability is evolving fast. New methods appear monthly, each claiming to be more faithful, more stable, or more human-friendly than the last. The key takeaway is this: no single explanation method is universally best. Use multiple methods, cross-validate them against each other, run sanity checks, and always remember that an explanation is a model of a model — it's an approximation of an approximation. Treat explanations as debugging tools and trust-building aids, not as ground truth about what the model "thinks." The model doesn't think. It computes. Your job is to make sure those computations align with human values and expectations.
