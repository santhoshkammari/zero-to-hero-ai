Interpretability and Explainability

Chapter 16: Advanced Deep Learning Section 2 Moderate Depth
TL;DR

Your model says "deny the claim" — but the regulator asks why. SHAP and LIME are the two workhorses for post-hoc explanations: SHAP gives you theoretically grounded, globally consistent attributions; LIME is faster and good enough for quick local checks. Feature importance has three flavors (built-in, permutation, SHAP-based) and they disagree more often than you'd expect. Attention weights are not explanations — the research is clear on this. Gradient-based methods (Integrated Gradients, Grad-CAM) are useful for neural nets when you need per-input attributions fast. In production, the real challenge isn't computing explanations — it's making them useful to stakeholders who don't speak ML.

The Practical Case for Explainability

This isn't a "nice to have" anymore. The EU AI Act, FDA guidance on clinical decision support, US banking regulations (SR 11-7), and insurance solvency directives all increasingly demand that automated decisions come with explanations. If you're deploying models in healthcare, finance, or insurance, interpretability is a shipping requirement, not a research curiosity.

Beyond compliance, there are two purely engineering reasons to care:

Debugging > Trust

Bug detection: Your image classifier hits 99% accuracy but Grad-CAM reveals it learned the hospital watermark, not the pathology. Feature leakage: SHAP shows your fraud model relies heavily on a feature that's only available after fraud is confirmed. Shortcut learning: Permutation importance reveals your NLP model keys on document length, not content. Explainability tools catch these before production does.

SHAP vs LIME — The Two Workhorses

These are the methods you'll actually use in production. Both are model-agnostic (they work on any model), both explain individual predictions, but they come at the problem from different angles.

SHAP: The Theoretically Grounded One

SHAP borrows from cooperative game theory. The intuition: imagine each feature is a "player" in a game, and the "payout" is the prediction. Shapley values tell you the fair contribution of each player, averaged over all possible coalitions. You don't need the math — just know that SHAP values have three properties that LIME doesn't guarantee:

  • Consistency — if a feature's true contribution increases, its SHAP value won't decrease
  • Local accuracy — SHAP values for all features sum exactly to the difference between the prediction and the baseline
  • Missingness — features that are absent get zero attribution

The catch? Computing exact Shapley values is exponential in the number of features. In practice, you use one of several fast approximations:

VariantWorks OnSpeedNotes
TreeSHAPTree models (XGBoost, LightGBM, RF)Very fastExact Shapley values. This is why SHAP dominates tabular ML.
KernelSHAPAny modelSlow for high-dimLIME-style sampling with Shapley weighting kernel.
DeepSHAPNeural networksModerateCombines DeepLIFT with Shapley. Good for image/text models.
GradientSHAPDifferentiable modelsFastExpected gradients approach. Noisier but scales well.

Here's the typical workflow with TreeSHAP — what you'll actually write in production:

import shap
import xgboost as xgb

# Train a model (any tree-based model works)
model = xgb.XGBClassifier(n_estimators=300, max_depth=6)
model.fit(X_train, y_train)

# TreeSHAP — fast and exact for tree models
explainer = shap.TreeExplainer(model)
shap_values = explainer.shap_values(X_test)

# Global view: which features matter most across all predictions?
shap.summary_plot(shap_values, X_test, feature_names=feature_names)

# Local view: why did the model make THIS prediction?
shap.force_plot(
    explainer.expected_value,
    shap_values[42],
    X_test[42],
    feature_names=feature_names
)

# Waterfall plot — cleaner for stakeholder presentations
shap.waterfall_plot(shap.Explanation(
    values=shap_values[42],
    base_values=explainer.expected_value,
    data=X_test[42],
    feature_names=feature_names
))

LIME: The Fast-and-Loose One

LIME takes a different approach: perturb the input, see what happens, fit a local linear model. It doesn't care about game theory — it just asks "what's approximately linear in this neighborhood?"

The recipe: take your input, generate thousands of perturbed copies (shuffle features for tabular, mask superpixels for images, drop words for text), feed them all through the model, weight by proximity to the original, and fit a sparse linear model. The coefficients are your explanation.

LIME is faster than KernelSHAP, easier to understand, and often "good enough." But it has a known stability problem — run it twice on the same input and you may get different explanations, because the perturbations are random. This matters a lot in regulated settings where you need reproducible explanations.

The Honest Recommendation

If you're using tree-based models (and for tabular data, you probably should be), just use TreeSHAP. It's fast, exact, and well-supported. LIME makes more sense when you need quick visual explanations for images or text, or when SHAP is too slow for your use case. For anything going into a regulatory filing, use SHAP.

Feature Importance — Three Methods That Disagree

Feature importance sounds simple. It isn't. There are three common approaches and they often give different rankings. Understanding why is the difference between "my model uses income" and actually knowing what's going on.

Built-in (Impurity-Based) Importance

Tree models like Random Forest and XGBoost report feature importance based on how much each feature reduces impurity (Gini or entropy) across all splits. It's free — you get it from model.feature_importances_. The problem: it's biased toward high-cardinality features. A random ID column with 10,000 unique values will look "important" because it creates many pure splits, even though it's memorizing noise.

Permutation Importance

Shuffle one feature at a time, measure how much the model's performance drops. Bigger drop = more important. This is model-agnostic and harder to game, but it has its own trap: correlated features share importance. If income and credit_score are highly correlated, permuting one barely hurts because the other compensates. Both look unimportant when they're actually critical.

SHAP-Based Importance

Take the mean absolute SHAP value for each feature across your dataset. This handles interactions and correlations better than permutation importance, and it gives you directionality (not just "important" but "pushes prediction up or down"). The downside is compute cost for non-tree models.

When They Disagree

If built-in importance says feature X matters but permutation importance says it doesn't, X is probably correlated with other features (redundant information). If permutation importance says X matters but SHAP says it doesn't have high mean |SHAP|, there might be interaction effects at play. When in doubt, trust SHAP — it has the strongest theoretical backing. But always look at the SHAP dependence plots, not just the bar chart.

Attention ≠ Explanation

This one burns people. You have a transformer model, you visualize the attention weights, they seem to highlight the "right" tokens, and you conclude the model is "looking at" the important parts. Feels intuitive. It's also mostly wrong.

Jain and Wallace (2019) showed that:

  • Attention weights often don't correlate with gradient-based feature importance
  • You can find alternative attention distributions that produce the same predictions
  • Adversarial attention patterns can maintain model output while attending to completely different tokens

Wiegreffe and Pinter (2019) pushed back — "attention is not not explanation" — arguing that attention can be plausibly explanatory in some settings, especially when there's a clear alignment between attention and human-annotated rationales.

The Practical Reality

Attention visualization is useful for debugging and intuition — "huh, the model is attending to punctuation a lot, that seems wrong." It's not suitable for formal explanations to regulators or patients. If you need to explain a transformer's prediction rigorously, use Integrated Gradients or SHAP, not attention weights. Think of attention as a conversation starter, not a conclusion.

Gradient-Based Methods for Neural Networks

When your model is differentiable, you can compute ∂output/∂input in a single backward pass. That gradient tells you which input dimensions the model is locally most sensitive to. The family of gradient-based methods builds on this idea with increasing sophistication.

Integrated Gradients

The most principled gradient-based method. Vanilla gradients are noisy because they capture local sensitivity at a single point. Integrated Gradients (Sundararajan et al., 2017) fixes this by integrating gradients along a straight-line path from a baseline (e.g., a black image) to the actual input.

import torch

def integrated_gradients(model, input_tensor, baseline, target_class, steps=50):
    """Compute Integrated Gradients attribution."""
    # Generate interpolated inputs along the path
    alphas = torch.linspace(0, 1, steps).view(-1, 1, 1, 1).to(input_tensor.device)
    interpolated = baseline + alphas * (input_tensor - baseline)
    interpolated.requires_grad_(True)

    # Forward pass on all interpolated inputs
    outputs = model(interpolated)
    scores = outputs[:, target_class]

    # Backward pass
    grads = torch.autograd.grad(scores.sum(), interpolated)[0]

    # Average gradients and scale by (input - baseline)
    avg_grads = grads.mean(dim=0)
    attribution = (input_tensor - baseline).squeeze() * avg_grads

    return attribution

Why this works: it satisfies completeness (attributions sum to the difference between the output at input and at baseline) and sensitivity (if a feature changes the output, it gets non-zero attribution). These are the same axioms that motivate Shapley values, which is why IG and SHAP often agree.

Grad-CAM

For CNNs specifically, Grad-CAM gives you a coarse spatial heatmap by weighting the last convolutional layer's feature maps by the gradients flowing into them. It's lower resolution than pixel-level methods but much more interpretable visually — you get a blob over the region the model "cared about," not a noisy pixel soup.

Grad-CAM is great for quick visual debugging. It's less useful for formal attributions because it only gives spatial localization, not per-pixel importance. If you need to tell a regulator which specific pixels drove the prediction, use Integrated Gradients.

SmoothGrad

A simple trick that improves any gradient-based method: add Gaussian noise to the input N times, compute gradients each time, and average. This smooths out the high-frequency noise that makes vanilla saliency maps look like static. It's not a method per se — it's an enhancement you layer on top.

Production Patterns

Computing SHAP values in a Jupyter notebook is the easy part. The hard part is building an explanation system that works reliably at scale and actually helps people make decisions.

Explaining Predictions to Stakeholders

Your audience determines your explanation format. A data scientist wants SHAP waterfall plots. A loan officer wants "top 3 reasons for denial" in plain English. A regulator wants a documented methodology with stability guarantees.

The Translation Layer

In practice, you build a mapping from feature names + SHAP directions to human-readable reason codes: {"credit_score": {"negative": "Credit score below threshold", "positive": "Strong credit history"}}. The model produces SHAP values; a post-processing layer converts them into the top-K reasons using pre-written templates. This is boring engineering but it's what actually ships.

Regulatory Compliance Patterns

For regulated industries, you typically need:

  • Deterministic explanations — same input must produce same explanation (rules out vanilla LIME without fixed seeds)
  • Adverse action reasons — in lending, you must provide specific reasons for denial (SHAP top-K features)
  • Model documentation — which explanation method, why it was chosen, validation results
  • Stability testing — show that small input perturbations don't wildly change the explanation
def generate_adverse_action_reasons(shap_values, feature_names, reason_map, top_k=4):
    """Convert SHAP values into regulatory-compliant denial reasons."""
    # Sort features by negative SHAP contribution (hurting the applicant)
    feature_impacts = list(zip(feature_names, shap_values))
    negative_impacts = [(f, v) for f, v in feature_impacts if v < 0]
    negative_impacts.sort(key=lambda x: x[1])  # most negative first

    reasons = []
    for feat, val in negative_impacts[:top_k]:
        direction = "negative" if val < 0 else "positive"
        reason_text = reason_map.get(feat, {}).get(direction, f"Factor: {feat}")
        reasons.append({"feature": feat, "shap_value": round(val, 4), "reason": reason_text})

    return reasons

# Example output:
# [{"feature": "credit_score", "shap_value": -0.4102,
#   "reason": "Credit score below minimum threshold"},
#  {"feature": "debt_to_income", "shap_value": -0.2891,
#   "reason": "Debt-to-income ratio exceeds guidelines"}]

Performance Considerations

TreeSHAP is fast enough for real-time serving (sub-millisecond per prediction). KernelSHAP is not — if you need model-agnostic explanations at scale, precompute them in batch and cache. For neural nets, Integrated Gradients with 50 steps means 50x the forward/backward cost. That's fine for batch explanation jobs, not for real-time APIs unless you reduce the step count.

Caching Strategy

For high-throughput systems: compute explanations asynchronously. Return the prediction immediately, queue the explanation job, and store the result. If a customer or regulator requests the explanation later, serve it from cache. Most predictions are never questioned — don't pay the explanation tax on every single one.

Evaluating Your Explanations

Anyone can produce a pretty heatmap. The question is whether it actually reflects what the model is doing. Two quick sanity checks you should always run:

Faithfulness (deletion test): Rank features by attributed importance, remove them one by one from most to least important, and measure prediction change. A faithful explanation should cause rapid prediction degradation. If removing the "most important" features barely changes anything, your explanation is lying.

Model randomization test (Adebayo et al., 2018): Compare explanations from your trained model against a randomly initialized version. If they look similar, your explanation method is detecting input structure, not learned behavior. Guided Backpropagation famously fails this test — it's basically an edge detector regardless of what the model learned.

Methods That Failed Sanity Checks

Guided Backpropagation and Guided Grad-CAM produce visually appealing saliency maps that barely change when you randomize the model weights. They're effectively edge detectors. Vanilla gradients, Grad-CAM, Integrated Gradients, and SHAP all pass the sanity checks. If you're choosing a method for production, stick to the ones that passed.
