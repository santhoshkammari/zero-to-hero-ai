Neural ODEs and Continuous-Depth Models

Chapter 16: Advanced Deep Learning 11 subtopics
TL;DR

A ResNet adds a small residual to its hidden state at every layer. If you shrink those steps to be infinitely small, you get a Neural ODE — a network whose depth is continuous, not discrete. This lets you trade compute for memory, handle irregular time series natively, and build normalizing flows without painful determinant computations. Flow Matching has since emerged as a faster, more stable way to train these continuous dynamics. Neural SDEs add stochasticity on top, opening the door to uncertainty quantification. This section builds from the ResNet→ODE connection all the way to modern generative modeling with continuous flows.

From ResNets to Differential Equations

Let's start with something you already know well: a residual block. In a ResNet, each layer computes:

h_{t+1} = h_t + f(h_t, θ_t)

You take the current hidden state, run it through some learned function f, and add the result back. That's the "residual" in residual network. Now stare at that equation for a second. If you've ever taken a numerical methods class, it should look suspiciously familiar. That's Euler's method — the simplest way to numerically solve a differential equation.

Euler's method approximates the solution of dh/dt = f(h, t) by taking discrete steps of size Δt = 1. Each ResNet layer is one Euler step. A 50-layer ResNet is 50 Euler steps. The question that launched the Neural ODE paper (Chen et al., 2018) was: what if we stopped discretizing? What if we let the network define a continuous transformation and used a proper ODE solver to integrate it?

That's the core idea. Instead of stacking 50 discrete layers with 50 sets of parameters, you parameterize a single vector field f(h(t), t, θ) — one neural network that tells you how the hidden state should change at every point in continuous time. Then you hand it to an ODE solver and say: "Start at h(0) = x (my input), and give me h(T) (my output)."

Neural ODE Formulation

The Neural ODE formulation is surprisingly clean. You define a neural network that represents the derivative of the hidden state with respect to time:

dh(t)/dt = f(h(t), t, θ)

Given an input h(0) = x, you integrate from time t₀ to t₁ to get the output. The solver handles the rest — choosing step sizes, controlling error, doing whatever it needs to trace that trajectory accurately.

Here's what this looks like in code using torchdiffeq, the library that accompanied the original paper:

import torch
import torch.nn as nn
from torchdiffeq import odeint

class ODEFunc(nn.Module):
    """The vector field f(h, t, θ) — this IS the 'network'."""
    def __init__(self, hidden_dim):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(hidden_dim, 128),
            nn.Tanh(),
            nn.Linear(128, 128),
            nn.Tanh(),
            nn.Linear(128, hidden_dim),
        )

    def forward(self, t, h):
        # t is a scalar (current time), h is the state
        return self.net(h)


class NeuralODE(nn.Module):
    def __init__(self, input_dim, hidden_dim, output_dim):
        super().__init__()
        self.encoder = nn.Linear(input_dim, hidden_dim)
        self.ode_func = ODEFunc(hidden_dim)
        self.decoder = nn.Linear(hidden_dim, output_dim)

    def forward(self, x):
        # Encode input into initial hidden state
        h0 = self.encoder(x)

        # Integration times: start at 0, end at 1
        t = torch.tensor([0.0, 1.0])

        # Solve the ODE — the solver picks its own internal steps
        # h_trajectory shape: (2, batch_size, hidden_dim)
        h_trajectory = odeint(self.ode_func, h0, t, method='dopri5')

        # Take the final state h(1)
        h_final = h_trajectory[-1]
        return self.decoder(h_final)


# Usage
model = NeuralODE(input_dim=2, hidden_dim=64, output_dim=2)
x = torch.randn(32, 2)  # batch of 32 samples, 2 features
out = model(x)
print(f"Input shape:  {x.shape}")
print(f"Output shape: {out.shape}")

Notice what's different from a standard network. There are no "layers" in the traditional sense. The ODEFunc is evaluated as many times as the solver needs. On easy inputs, the solver might take 10 internal steps. On hard inputs, it might take 50. The network adapts its effective depth to the problem. That's a profound shift from fixed-depth architectures.

The Adjoint Sensitivity Method

Here's the catch. If you naively backpropagate through the ODE solver, you need to store every intermediate state from the forward pass — just like storing activations in a normal deep network. For a solver that took 200 internal steps, that's 200 sets of activations eating your GPU memory.

The adjoint sensitivity method is the trick that makes Neural ODEs practical. Instead of backpropagating through the solver's steps, you solve another ODE backwards in time. The key insight: you can reconstruct any intermediate state by running the ODE backward from the final state. So you never need to store intermediate states at all.

The adjoint state a(t) = dL/dh(t) evolves according to its own ODE:

da(t)/dt = -a(t)ᵀ · (∂f/∂h)

And the gradient with respect to parameters accumulates as:

dL/dθ = -∫ a(t)ᵀ · (∂f/∂θ) dt

Memory complexity drops from O(L) to O(1), where L is the number of solver steps. You only store the final state and the adjoint, then integrate backward. In practice you trade off a bit of compute for a lot of memory savings, which is often a great deal on modern GPUs where memory is the bottleneck.

Memory Trade-Off in Practice

The adjoint method's O(1) memory is the theoretical ideal. In practice, most implementations use checkpointing — storing states at a few time points and reconstructing in between. This avoids the numerical drift that can happen when integrating backward over long time horizons. The torchdiffeq library handles this transparently. You get most of the memory savings without worrying about accumulating numerical error.

ResNet as Euler Discretization

Let's make the ResNet–ODE connection airtight. Take the ODE dh/dt = f(h, t, θ) and discretize with step size Δt:

h(t + Δt) = h(t) + Δt · f(h(t), t, θ)

Set Δt = 1 and you recover exactly the ResNet update: h_{t+1} = h_t + f(h_t). A ResNet is literally a first-order Euler discretization of a Neural ODE with unit step size. This isn't just a cute analogy — it has practical consequences. Euler's method is the least accurate ODE solver. By moving to proper solvers (Runge-Kutta, Adams, etc.), you get better numerical accuracy for the same compute. Or you can match the accuracy of a 100-layer ResNet with far fewer function evaluations.

This also explains why deeper ResNets (with more layers and smaller effective step sizes) tend to learn smoother transformations. They're approximating a continuous flow more faithfully.

Augmented Neural ODEs

Vanilla Neural ODEs have a limitation that's easy to miss. Because the ODE defines a continuous, non-intersecting flow in state space, it can only represent homeomorphisms — bijective continuous maps. Translation: trajectories can't cross. This means a Neural ODE operating on ℝ² literally cannot learn to separate two concentric circles. The trajectories would have to cross each other to pull the circles apart.

The Augmented Neural ODE (Dupont et al., 2019) fixes this by padding the state with extra dimensions. If your data lives in ℝ², you lift it into ℝ⁵ or ℝ¹⁰, run the ODE in the higher-dimensional space, and project back down. In the augmented space, trajectories have room to maneuver around each other.

class AugmentedODEFunc(nn.Module):
    """ODE function operating in augmented state space."""
    def __init__(self, data_dim, aug_dim):
        super().__init__()
        total_dim = data_dim + aug_dim
        self.net = nn.Sequential(
            nn.Linear(total_dim, 128),
            nn.Tanh(),
            nn.Linear(128, 128),
            nn.Tanh(),
            nn.Linear(128, total_dim),
        )

    def forward(self, t, h):
        return self.net(h)


class AugmentedNeuralODE(nn.Module):
    def __init__(self, data_dim, aug_dim=4, output_dim=2):
        super().__init__()
        self.aug_dim = aug_dim
        self.ode_func = AugmentedODEFunc(data_dim, aug_dim)
        self.decoder = nn.Linear(data_dim + aug_dim, output_dim)

    def forward(self, x):
        # Augment: pad input with zeros
        aug = torch.zeros(x.shape[0], self.aug_dim, device=x.device)
        h0 = torch.cat([x, aug], dim=-1)

        t = torch.tensor([0.0, 1.0])
        h_final = odeint(self.ode_func, h0, t, method='dopri5')[-1]
        return self.decoder(h_final)

The augmented dimensions start at zero and evolve according to the ODE. Think of it as giving the network scratch space to work with during its continuous transformation. Empirically, this dramatically improves expressiveness with minimal overhead.

Continuous Normalizing Flows

Normalizing flows are generative models that transform a simple base distribution (say, a Gaussian) into a complex data distribution through a sequence of invertible transformations. The catch with discrete flows: at each step, you compute the log-determinant of the Jacobian to track how probability density changes. For a D-dimensional state, that Jacobian is D×D, and its determinant costs O(D³). People designed clever architectures (coupling layers, autoregressive flows) specifically to make this determinant cheap, but those architectures are restrictive.

Continuous Normalizing Flows (CNFs) solve this elegantly. When your transformation is defined by an ODE, the change in log-probability follows the instantaneous change-of-variables formula:

d log p(h(t)) / dt = -Tr(∂f/∂h)

Instead of computing the full D×D determinant, you only need the trace of the Jacobian — a single number. And here's the kicker: you can estimate the trace stochastically using Hutchinson's estimator. Pick a random vector ε, and:

Tr(∂f/∂h) ≈ εᵀ · (∂f/∂h) · ε

That right-hand side is a Jacobian-vector product, which automatic differentiation computes efficiently. No need to ever form the full Jacobian. Cost drops from O(D³) to O(D). This lets you use any architecture for f — no coupling layers, no autoregressive structure, no restrictions.

import torch
import torch.nn as nn
from torchdiffeq import odeint

class CNFFunc(nn.Module):
    """Dynamics for Continuous Normalizing Flow.
    Computes both dh/dt and d(log p)/dt jointly."""

    def __init__(self, dim):
        super().__init__()
        self.dim = dim
        self.net = nn.Sequential(
            nn.Linear(dim, 128),
            nn.Tanh(),
            nn.Linear(128, 128),
            nn.Tanh(),
            nn.Linear(128, dim),
        )

    def forward(self, t, state):
        # state is [h, log_p] concatenated
        h = state[..., :self.dim]
        # We only need f(h) for dh/dt
        with torch.enable_grad():
            h = h.detach().requires_grad_(True)
            f = self.net(h)

            # Hutchinson's trace estimator
            epsilon = torch.randn_like(h)
            # Compute εᵀ · (∂f/∂h) · ε via two autodiff calls
            f_eps = torch.sum(f * epsilon)
            grad_f_eps = torch.autograd.grad(f_eps, h, create_graph=True)[0]
            trace_estimate = torch.sum(grad_f_eps * epsilon, dim=-1, keepdim=True)

        # dh/dt = f(h), d(log p)/dt = -Tr(df/dh)
        d_log_p = -trace_estimate
        return torch.cat([f, d_log_p], dim=-1)


class ContinuousNormalizingFlow(nn.Module):
    def __init__(self, dim):
        super().__init__()
        self.dim = dim
        self.dynamics = CNFFunc(dim)

    def forward(self, x):
        """Transform data x to base distribution, return z and log-likelihood."""
        # Initial log-probability contribution is zero
        log_p0 = torch.zeros(x.shape[0], 1, device=x.device)
        state0 = torch.cat([x, log_p0], dim=-1)

        t = torch.tensor([0.0, 1.0])
        state_final = odeint(self.dynamics, state0, t, method='dopri5')[-1]

        z = state_final[..., :self.dim]
        delta_log_p = state_final[..., self.dim:]

        # Log-likelihood under base distribution (standard normal)
        log_pz = -0.5 * (z ** 2 + torch.log(torch.tensor(2 * 3.14159)))
        log_pz = log_pz.sum(dim=-1, keepdim=True)

        # Total log-likelihood = base + change from flow
        log_px = log_pz + delta_log_p
        return z, log_px

    @torch.no_grad()
    def sample(self, num_samples):
        """Generate samples by integrating backward from noise."""
        z = torch.randn(num_samples, self.dim)
        log_p0 = torch.zeros(num_samples, 1)
        state0 = torch.cat([z, log_p0], dim=-1)

        # Reverse time: go from t=1 to t=0
        t = torch.tensor([1.0, 0.0])
        state_final = odeint(self.dynamics, state0, t, method='dopri5')[-1]
        return state_final[..., :self.dim]

The beauty here is that you train with exact log-likelihood, sample by integrating backward, and never worry about designing invertible architectures. The ODE solver guarantees invertibility for free — run it forward for encoding, backward for sampling.

Flow Matching: The Modern Alternative

Neural ODEs trained with maximum likelihood (as in CNFs) have a practical problem: the ODE solver is in the training loop. Every gradient step requires solving an ODE, which means dozens of network evaluations per training step. Training is slow.

Flow Matching (Lipman et al., 2022) sidesteps this entirely. Instead of learning the dynamics by maximizing likelihood through an ODE solve, you directly regress the vector field. The idea: if you know what the velocity field should look like at each point, you can train with a simple MSE loss — no ODE solver needed during training.

The trick is Conditional Flow Matching. For each data point x₁, define a simple conditional probability path from noise x₀ to data x₁. The easiest choice is a straight line:

x_t = (1 - t) · x₀ + t · x₁     where x₀ ~ N(0, I)

The conditional velocity field for this path is just:

u_t(x_t | x₁) = x₁ - x₀

Train the network to predict this velocity:

import torch
import torch.nn as nn


class FlowMatchingVelocity(nn.Module):
    """Velocity field v(x, t, θ) for flow matching."""
    def __init__(self, dim):
        super().__init__()
        # +1 for the time input
        self.net = nn.Sequential(
            nn.Linear(dim + 1, 256),
            nn.SiLU(),
            nn.Linear(256, 256),
            nn.SiLU(),
            nn.Linear(256, dim),
        )

    def forward(self, x, t):
        # t is (batch,) -> expand to (batch, 1) for concatenation
        t_expanded = t.unsqueeze(-1) if t.dim() == 1 else t
        return self.net(torch.cat([x, t_expanded], dim=-1))


def flow_matching_loss(model, x1):
    """Conditional Flow Matching loss — no ODE solver needed!"""
    batch_size = x1.shape[0]

    # Sample noise and random time
    x0 = torch.randn_like(x1)
    t = torch.rand(batch_size, device=x1.device)

    # Interpolate: x_t = (1-t)*x0 + t*x1
    t_expanded = t.unsqueeze(-1)
    xt = (1 - t_expanded) * x0 + t_expanded * x1

    # Target velocity: dx/dt = x1 - x0
    target_velocity = x1 - x0

    # Predict velocity
    predicted_velocity = model(xt, t)

    # Simple MSE loss
    loss = ((predicted_velocity - target_velocity) ** 2).mean()
    return loss


# Training loop sketch
dim = 2
model = FlowMatchingVelocity(dim)
optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)

for step in range(5000):
    # x1 = your real data batch
    x1 = torch.randn(256, dim) * 0.5 + torch.tensor([[2.0, 2.0]])
    loss = flow_matching_loss(model, x1)
    optimizer.zero_grad()
    loss.backward()
    optimizer.step()

# To sample: solve the ODE at inference only
from torchdiffeq import odeint

with torch.no_grad():
    z = torch.randn(500, dim)
    t_eval = torch.linspace(0, 1, 50)
    def velocity_wrapper(t_scalar, x):
        t_batch = torch.full((x.shape[0],), t_scalar.item(), device=x.device)
        return model(x, t_batch)
    trajectory = odeint(velocity_wrapper, z, t_eval, method='dopri5')
    samples = trajectory[-1]

This is the approach that powers modern diffusion-like generative models (like Meta's approach in their image generation work). Training is fast because there's no ODE in the loop. You only solve the ODE at inference to generate samples. The velocity field is trained with a regression loss on interpolated points. Compared to score-based diffusion models, flow matching often converges faster and produces straighter trajectories, meaning fewer ODE solver steps at inference.

Flow Matching vs. Diffusion Models

Diffusion models and flow matching are closely related — both define continuous-time processes from noise to data. Diffusion uses a forward noising process and learns the reverse (score function). Flow matching directly learns a velocity field without committing to a specific noising schedule. In practice, flow matching often produces straighter paths from noise to data, which means the ODE solver can take fewer steps during sampling. This is why many recent large-scale generative models have shifted from DDPM-style diffusion to flow matching formulations.

Neural SDEs: Adding Stochasticity

A Neural ODE traces a single deterministic trajectory through state space. Sometimes you want randomness — for uncertainty quantification, for modeling noisy systems, or because the underlying process is genuinely stochastic.

A Neural Stochastic Differential Equation (Neural SDE) adds a noise term:

dh(t) = f(h(t), t, θ) dt + g(h(t), t, θ) dW(t)

Here f is the drift (same as before), g is the diffusion coefficient controlling noise magnitude, and dW is a Wiener process (Brownian motion increments). The same input will produce different trajectories on different runs — the model becomes a distribution over paths, not a single path.

This is a natural fit for financial modeling (stock prices are noisy), physics simulations (thermal fluctuations), and any setting where you need calibrated uncertainty. If you run 100 forward passes and get a tight bundle of trajectories, the model is confident. If the trajectories fan out wildly, the model is uncertain. That fan-out emerges from the learned diffusion coefficient g, not from dropout hacks or ensemble tricks.

The torchsde library provides the same clean interface as torchdiffeq but for SDEs. Training uses either the adjoint method adapted for SDEs or pathwise gradients depending on the noise type.

Adaptive Step-Size ODE Solvers

One of the underappreciated benefits of Neural ODEs is adaptive computation. The default solver in torchdiffeq is dopri5 — a Dormand-Prince method that automatically adjusts its step size based on a local error estimate. When the vector field is smooth and slowly changing, it takes large steps. When the dynamics are sharp or rapidly changing, it takes small steps.

This means the network automatically spends more compute on "hard" parts of the transformation and less on "easy" parts. You don't have to decide the depth ahead of time. The solver figures it out. You control accuracy through the atol and rtol parameters:

# Loose tolerance — fast but approximate
h_fast = odeint(func, h0, t, method='dopri5', atol=1e-3, rtol=1e-3)

# Tight tolerance — slow but precise
h_precise = odeint(func, h0, t, method='dopri5', atol=1e-7, rtol=1e-7)

# Fixed-step solver — predictable cost, like a ResNet
h_fixed = odeint(func, h0, t, method='euler', options={'step_size': 0.1})

During training, you might use loose tolerances for speed. At inference, you can tighten them for accuracy. This is a knob that traditional architectures don't give you. You can also use fixed-step solvers like Euler or RK4 if you need deterministic compute budgets — sacrificing adaptivity for predictability.

Applications to Irregular Time Series

This is the killer application. Medical records, sensor data, financial ticks — real-world time series almost never arrive at regular intervals. A patient gets blood tests on Monday, then nothing for a week, then three tests in one day. Traditional RNNs expect evenly-spaced inputs. You can hack around it with imputation or time deltas, but you're fighting the architecture.

Neural ODEs handle irregular timestamps naturally. The hidden state evolves continuously, and you can observe it (or update it) at any time point. Here's a complete example:

import torch
import torch.nn as nn
from torchdiffeq import odeint


class LatentODEFunc(nn.Module):
    """Continuous dynamics of the latent state."""
    def __init__(self, latent_dim):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(latent_dim, 64),
            nn.Tanh(),
            nn.Linear(64, 64),
            nn.Tanh(),
            nn.Linear(64, latent_dim),
        )

    def forward(self, t, h):
        return self.net(h)


class GRUUpdate(nn.Module):
    """GRU-style update when a new observation arrives."""
    def __init__(self, obs_dim, latent_dim):
        super().__init__()
        self.gru_cell = nn.GRUCell(obs_dim, latent_dim)

    def forward(self, h, obs):
        return self.gru_cell(obs, h)


class LatentODEIrregular(nn.Module):
    """Latent ODE for irregularly-sampled time series.

    The idea: between observations, the hidden state evolves
    according to the ODE. When an observation arrives, we update
    the hidden state using a GRU cell.
    """
    def __init__(self, obs_dim, latent_dim):
        super().__init__()
        self.latent_dim = latent_dim
        self.ode_func = LatentODEFunc(latent_dim)
        self.updater = GRUUpdate(obs_dim, latent_dim)
        self.decoder = nn.Linear(latent_dim, obs_dim)
        self.h0 = nn.Parameter(torch.randn(latent_dim) * 0.1)

    def forward(self, times, observations, mask=None):
        """
        Args:
            times: (seq_len,) observation timestamps (irregular!)
            observations: (batch, seq_len, obs_dim)
            mask: (batch, seq_len) — 1 if observed, 0 if missing

        Returns:
            predictions at each time point
        """
        batch_size = observations.shape[0]

        # Initialize hidden state
        h = self.h0.unsqueeze(0).expand(batch_size, -1)
        predictions = []

        for i in range(len(times)):
            if i > 0:
                # Evolve hidden state from t_{i-1} to t_i via ODE
                t_span = torch.tensor([times[i - 1], times[i]])
                h = odeint(self.ode_func, h, t_span, method='dopri5')[-1]

            # Decode current state to make prediction
            pred = self.decoder(h)
            predictions.append(pred)

            # If we have an observation, update the hidden state
            if mask is not None:
                obs_i = observations[:, i, :]
                mask_i = mask[:, i].unsqueeze(-1)  # (batch, 1)
                h_updated = self.updater(h, obs_i)
                h = h * (1 - mask_i) + h_updated * mask_i

        return torch.stack(predictions, dim=1)


# Simulating irregular medical data
obs_dim = 3    # e.g., heart rate, blood pressure, temperature
latent_dim = 16

model = LatentODEIrregular(obs_dim, latent_dim)

# Irregular timestamps — not evenly spaced!
times = torch.tensor([0.0, 0.5, 0.7, 2.1, 2.3, 5.0, 8.0, 8.1])
batch_size = 4
seq_len = len(times)

observations = torch.randn(batch_size, seq_len, obs_dim)
# Some observations are missing (mask = 0)
mask = torch.ones(batch_size, seq_len)
mask[:, 3] = 0  # No observation at t=2.1 for any patient
mask[1, 5] = 0  # Patient 1 missing at t=5.0

preds = model(times, observations, mask)
print(f"Timestamps:  {times.tolist()}")
print(f"Predictions: {preds.shape}")  # (4, 8, 3)

Notice how the time gaps between observations vary wildly — 0.2, 1.4, 0.2, 2.7, 3.0, 0.1. A standard RNN would treat each step identically. The Latent ODE lets the hidden state evolve for exactly the right amount of time between observations. The ODE solver handles all of it. No discretization artifacts, no need for imputation, no time-delta features.

This approach (originally called ODE-RNN or Latent ODE) has shown strong results on medical datasets like PhysioNet (ICU patient data) and MIMIC-III, where observation patterns are inherently irregular and clinically meaningful — a flurry of measurements often indicates the patient's condition is deteriorating.

Applications to Generative Modeling

CNFs and flow matching have become serious contenders in generative modeling. The appeal: exact likelihood computation (via the trace formula), flexible architectures (no coupling layer constraints), and smooth, continuous transformations that are easy to control and interpolate.

In image generation, flow matching has been adopted by several high-profile projects. The training is stable (MSE regression is about as tame as losses get), and the quality scales well with model size. At inference, you solve an ODE from noise to data, and the number of solver steps directly trades off quality vs. speed. Want fast, approximate samples? Use 10 steps. Want publication-quality? Use 100 steps.

For point clouds, molecular generation, and protein structure prediction, Neural ODEs offer another advantage: they naturally preserve geometric properties. If you design the vector field to be equivariant to rotations and translations, the entire flow inherits those symmetries. Discrete architectures can also be equivariant, but the continuous formulation makes it more natural to reason about and implement.

When to Use Neural ODEs

Good fit: Irregularly sampled time series, generative modeling with exact likelihood, problems where adaptive computation depth is valuable, physics-informed models where the dynamics are naturally continuous.
Think twice: Standard supervised learning on tabular or image data (regular networks are simpler and faster), real-time applications where inference latency must be deterministic (adaptive solvers have variable cost), very large-scale training where the ODE solver overhead is a bottleneck (consider flow matching instead).

Putting It All Together

The arc of this section traces one big idea: going from discrete to continuous. ResNets stack discrete layers. Neural ODEs parameterize continuous dynamics. CNFs use that continuity to build density estimators without painful Jacobian determinants. Flow matching removes the ODE from training altogether, keeping it only at inference. Neural SDEs bring stochasticity into the picture for uncertainty-aware models.

Each step solved a real bottleneck. ResNets needed hundreds of layers of parameters — Neural ODEs reuse a single network. Discrete normalizing flows needed restrictive architectures — CNFs freed us from that. CNFs were slow to train — flow matching made training a simple regression problem. Deterministic models couldn't express uncertainty — Neural SDEs added principled noise.

The underlying math is differential equations, but the motivation is thoroughly pragmatic: more expressive models, less memory, faster training, and natural handling of continuous-time data. If your data lives in continuous time or your model needs to output distributions, the tools in this section are worth having in your toolkit.
