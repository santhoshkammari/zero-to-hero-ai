Meta-Learning and Few-Shot Learning

Chapter 16: Advanced Topics in Deep Learning Section 5
TL;DR

Traditional deep learning needs thousands of examples per class. Meta-learning flips the script: instead of training a model to classify cats vs dogs, you train a model that learns how to learn new tasks from just a handful of examples. The field breaks into three families — metric-based methods (compare embeddings), optimization-based methods (learn a good weight initialization), and memory-based methods (store and retrieve past experiences). Today, large language models have emerged as surprisingly powerful few-shot learners through in-context learning, where you show a few examples in the prompt and the model generalizes — no gradient updates required.

Why Meta-Learning?

Imagine you're building a wildlife monitoring system. A new species shows up on a camera trap — one you've never seen before. You have exactly three photos of it. Training a standard deep learning classifier on three images is hopeless. It would overfit within seconds and generalize to nothing. Yet a human child could look at those three photos, notice the striped tail and long ears, and recognize the animal in a fourth photo without breaking a sweat.

This is the gap meta-learning tries to close. Standard machine learning optimizes for performance on a single task with lots of data. Meta-learning optimizes for the ability to adapt quickly to new tasks with very little data. The distinction matters. A normal model gets better at one thing. A meta-learner gets better at getting better.

The practical stakes are high. Medical imaging rarely has thousands of labeled examples per rare disease. Robotics needs to adapt to new objects on the fly. Fraud detection faces new attack patterns weekly. Anywhere labeled data is scarce or expensive, few-shot learning offers an escape hatch.

Foundations

Learning to Learn

The phrase "learning to learn" sounds circular, but it has a precise meaning. In standard training, you have one learning loop: you feed data to a model, compute a loss, and update weights with gradient descent. In meta-learning, you have two nested loops. The inner loop learns a specific task (like classifying birds). The outer loop learns how to learn tasks — it adjusts the learning process itself so the inner loop works better on new tasks it has never seen.

Think of it like this. The inner loop is a student taking a quiz. The outer loop is the teacher designing the curriculum. After watching many students struggle with many quizzes, the teacher figures out the best way to prepare students for any quiz, even one that hasn't been written yet.

Task Distribution

In regular supervised learning, you sample data points from a dataset. In meta-learning, you sample entire tasks from a task distribution. Each task is a small learning problem — maybe "classify these 5 bird species" or "classify these 5 dog breeds." The meta-learner sees hundreds or thousands of such tasks during training. At test time, it faces a brand new task drawn from the same distribution — species it has never encountered before — and must adapt using only a few examples.

This is the key shift in thinking. Your training set is not a pile of labeled images. It is a pile of mini classification problems, each one a self-contained learning episode.

Support Set and Query Set

Each task gets split into two parts. The support set contains the few labeled examples the model gets to learn from — think of it as a tiny training set. The query set contains the examples the model must classify after seeing the support set — think of it as a tiny test set. During meta-training, the loss on the query set is what drives learning. This forces the model to actually generalize from the support set, not memorize it.

A concrete example: you give the model 3 photos of flamingos and 3 photos of pelicans (the support set), then ask it to classify 5 new photos of flamingos and pelicans (the query set). The model's accuracy on the query set tells you how well it learned from those 6 examples.

N-way K-shot Formulation

The standard benchmark format is called N-way K-shot classification. N is the number of classes per task, K is the number of labeled examples per class in the support set. So "5-way 1-shot" means: given 1 example each of 5 new classes, classify new instances. "5-way 5-shot" means you get 5 examples per class instead.

Common benchmarks like Omniglot (handwritten characters from 50 alphabets) and mini-ImageNet (100 classes drawn from ImageNet) are almost always evaluated in this format. The hardest setting is 1-shot: one example per class. If your method can handle that, it can almost certainly handle 5-shot.

Episodic Training

Episodic training is the training procedure that mirrors the test-time setup. Instead of shuffling all images into one big batch, you construct episodes. Each episode simulates a few-shot task: you randomly pick N classes, sample K examples per class for the support set, and sample a few more for the query set. The model learns from the support set and is evaluated on the query set, and the loss from the query set flows backward through the entire process.

The beauty of episodic training is that training conditions match test conditions. The model never gets comfortable with thousands of examples — it always practices the "learn from a handful" skill. Here is a minimal episode sampler:

import torch
import random

def sample_episode(dataset, n_way=5, k_shot=1, q_queries=15):
    """Sample one N-way K-shot episode from a dataset.
    
    Args:
        dataset: dict mapping class_label -> list of tensors
        n_way: number of classes per episode
        k_shot: support examples per class
        q_queries: query examples per class
    """
    classes = random.sample(list(dataset.keys()), n_way)
    
    support_x, support_y = [], []
    query_x, query_y = [], []
    
    for i, cls in enumerate(classes):
        samples = random.sample(dataset[cls], k_shot + q_queries)
        support_x.extend(samples[:k_shot])
        support_y.extend([i] * k_shot)
        query_x.extend(samples[k_shot:])
        query_y.extend([i] * q_queries)
    
    return (torch.stack(support_x), torch.tensor(support_y),
            torch.stack(query_x), torch.tensor(query_y))

Every call to sample_episode gives you a different mini-task. Over thousands of episodes, the meta-learner sees a wide variety of class combinations and learns transferable knowledge about how to distinguish new classes from few examples.

Metric-Based Methods

The simplest family of few-shot methods is built on a clean idea: learn an embedding space where similar things are close and different things are far apart. At test time, embed the support examples, embed the query, and classify by proximity. No gradient updates at test time — the embedding network does all the heavy lifting.

Siamese Networks

Siamese networks are the grandfather of metric-based few-shot learning. The architecture passes two inputs through the same neural network (shared weights), producing two embeddings. Then a distance function — usually Euclidean or cosine — tells you how similar they are. If the distance is small, the inputs are the same class. If it is large, they are different.

The network is trained on pairs: same-class pairs should have small distances, different-class pairs should have large distances. The contrastive loss penalizes the model when same-class pairs are far apart or different-class pairs are too close:

import torch
import torch.nn as nn
import torch.nn.functional as F

class SiameseNetwork(nn.Module):
    def __init__(self, backbone):
        super().__init__()
        self.backbone = backbone  # e.g., a CNN feature extractor
    
    def forward_one(self, x):
        return F.normalize(self.backbone(x), dim=-1)
    
    def forward(self, x1, x2):
        z1 = self.forward_one(x1)
        z2 = self.forward_one(x2)
        return z1, z2

def contrastive_loss(z1, z2, labels, margin=1.0):
    """labels: 1 if same class, 0 if different."""
    dist = torch.sum((z1 - z2) ** 2, dim=-1)
    loss_same = labels * dist
    loss_diff = (1 - labels) * F.relu(margin - dist.sqrt()) ** 2
    return (loss_same + loss_diff).mean()

At inference, you compare a query image against each support image and pick the class with the smallest distance. The limitation is that Siamese nets compare exactly two inputs at a time, which doesn't scale elegantly to N-way classification. That's where the next methods improve.

Matching Networks

Matching Networks (Vinyals et al., 2016) introduced episodic training to few-shot learning and formalized the N-way K-shot setup. The core idea: embed all support examples and the query using a learned embedding function, then classify the query using a soft nearest-neighbor approach with attention. The probability of query x belonging to class c is a weighted sum over support examples, where the weights come from cosine similarity in embedding space.

Matching Networks also introduced the idea of conditioning the embedding on the entire support set using a bidirectional LSTM, so the representation of each support example is context-aware. This was clever, though later methods found that simpler approaches worked almost as well.

Prototypical Networks

Prototypical Networks (Snell et al., 2017) simplified matching networks with an elegant observation: instead of comparing the query to every support example individually, compute a single prototype (mean embedding) for each class, then classify the query by distance to the nearest prototype. That's it. No attention, no LSTM conditioning. Take the average and measure distance.

Why does this work? In a well-learned embedding space, examples of the same class cluster together. The mean of a cluster is a robust representative — more robust than any individual example, especially when K is small and individual examples may be atypical.

import torch
import torch.nn as nn
import torch.nn.functional as F

class PrototypicalNetwork(nn.Module):
    def __init__(self, encoder, embed_dim=64):
        super().__init__()
        self.encoder = encoder
    
    def compute_prototypes(self, support_x, support_y, n_way):
        """Compute class prototypes as mean embeddings."""
        z_support = self.encoder(support_x)  # (n_way * k_shot, embed_dim)
        prototypes = []
        for c in range(n_way):
            mask = (support_y == c)
            prototypes.append(z_support[mask].mean(dim=0))
        return torch.stack(prototypes)  # (n_way, embed_dim)
    
    def forward(self, support_x, support_y, query_x, n_way):
        prototypes = self.compute_prototypes(support_x, support_y, n_way)
        z_query = self.encoder(query_x)  # (n_queries, embed_dim)
        
        # Negative squared Euclidean distance to each prototype
        dists = torch.cdist(z_query, prototypes)  # (n_queries, n_way)
        log_probs = F.log_softmax(-dists, dim=-1)
        return log_probs  # classify by argmax

# Training loop for one episode
def train_episode(model, support_x, support_y, query_x, query_y,
                  n_way, optimizer):
    optimizer.zero_grad()
    log_probs = model(support_x, support_y, query_x, n_way)
    loss = F.nll_loss(log_probs, query_y)
    loss.backward()
    optimizer.step()
    
    acc = (log_probs.argmax(dim=-1) == query_y).float().mean()
    return loss.item(), acc.item()

Prototypical networks remain a strong baseline. They are fast (no inner-loop optimization), easy to implement, and surprisingly competitive with more complex methods. The distance metric matters: the original paper found Euclidean distance outperformed cosine similarity, though this can depend on the dataset.

Why Euclidean Over Cosine?

Prototypical networks use negative squared Euclidean distance because it is equivalent to a linear model in embedding space (when combined with softmax). This means the decision boundaries between prototypes are hyperplanes — clean, simple, and effective. Cosine similarity loses this equivalence and can produce more complex (and sometimes worse) boundaries.

Relation Networks

Relation Networks (Sung et al., 2018) take the next logical step: instead of using a fixed distance metric like Euclidean or cosine, learn the distance function itself. The architecture has two modules: an embedding module that produces representations, and a relation module — a small neural network that takes two embeddings as input and outputs a similarity score between 0 and 1.

The relation module is trained end-to-end with the embedding module. This means the model can learn non-linear, task-appropriate notions of similarity that a fixed metric like Euclidean distance might miss. The cost is more parameters and slightly more complex training, but the flexibility can pay off on harder tasks.

Embedding Space Learning

All metric-based methods share a common backbone: learning a good embedding space. The quality of the embedding determines everything. If the encoder maps different classes to overlapping regions, no distance metric will save you. Training objectives like contrastive loss, triplet loss, and the episodic classification losses above all push toward the same goal — an embedding space with tight, well-separated clusters for each class.

A useful mental model: the encoder is doing most of the "learning to learn." It transforms raw pixels into a representation where class identity is the dominant axis of variation. Once you have that, few-shot classification reduces to a nearest-neighbor lookup — which is exactly why these methods need no gradient updates at test time.

Optimization-Based Methods

Metric-based methods are elegant, but they bake in an assumption: a good embedding plus a distance function is enough. What if you need something more flexible — a full neural network that can be fine-tuned on each new task? The problem is that fine-tuning on 5 examples normally leads to catastrophic overfitting. Optimization-based meta-learning solves this by learning an initialization that is designed to be fine-tuned from few examples.

MAML (Model-Agnostic Meta-Learning)

MAML (Finn et al., 2017) is the most influential optimization-based method. The idea is beautifully simple. Find an initialization of model weights such that a few steps of gradient descent on a new task's support set produces a model that performs well on that task's query set. You are not learning a model that is good at any particular task. You are learning a starting point that is good for fast adaptation to any task.

Here is how it works. You have a model with parameters θ. For each task in a meta-batch:

1. Inner loop: Take a few gradient steps on the task's support set, producing task-specific parameters θ'ᵢ. This is regular gradient descent — nothing fancy.

2. Outer loop: Evaluate the adapted parameters θ'ᵢ on the task's query set. The loss on the query set is what drives the meta-update to θ.

The key insight is that the outer loop computes gradients through the inner loop. You are differentiating through the optimization process itself. This means θ learns to be a starting point from which a few gradient steps lead to good task-specific performance.

import torch
import torch.nn as nn
import torch.nn.functional as F
from collections import OrderedDict

def maml_inner_loop(model, support_x, support_y, inner_lr=0.01,
                    inner_steps=5):
    """Run MAML inner loop: adapt model to one task."""
    # Clone parameters so we don't modify the original
    params = OrderedDict(model.named_parameters())
    
    for step in range(inner_steps):
        logits = functional_forward(model, support_x, params)
        loss = F.cross_entropy(logits, support_y)
        
        grads = torch.autograd.grad(loss, params.values(),
                                     create_graph=True)
        
        # Gradient descent on cloned params
        params = OrderedDict(
            (name, param - inner_lr * grad)
            for (name, param), grad in zip(params.items(), grads)
        )
    
    return params

def maml_outer_step(model, tasks, outer_optimizer, inner_lr=0.01,
                    inner_steps=5):
    """One outer loop step across a batch of tasks."""
    outer_optimizer.zero_grad()
    meta_loss = 0.0
    
    for support_x, support_y, query_x, query_y in tasks:
        # Inner loop: adapt to this task
        adapted_params = maml_inner_loop(
            model, support_x, support_y, inner_lr, inner_steps
        )
        
        # Evaluate adapted model on query set
        query_logits = functional_forward(model, query_x, adapted_params)
        task_loss = F.cross_entropy(query_logits, query_y)
        meta_loss += task_loss
    
    meta_loss /= len(tasks)
    meta_loss.backward()  # gradients flow through inner loop!
    outer_optimizer.step()
    
    return meta_loss.item()

The create_graph=True in the inner loop is the critical detail. It tells PyTorch to build a computation graph of the gradient computation itself, so the outer loop can backpropagate through the inner loop's gradient steps. Without it, you are doing first-order MAML (which we cover next).

MAML's Compute Cost

Second-order MAML requires computing gradients of gradients (Hessian-vector products). For a model with millions of parameters, this is expensive in both memory and compute. Each inner step adds another layer to the computation graph. In practice, people often use 1-5 inner steps and relatively small models. For large-scale settings, first-order approximations are almost always preferred.

First-Order MAML and Reptile

The second-order gradients in MAML are expensive. First-Order MAML (FOMAML) drops them. Instead of differentiating through the inner loop, it treats the adapted parameters as if they were obtained by a process that didn't involve θ. Mathematically, it ignores the second-order terms in the meta-gradient. Surprisingly, this works almost as well as full MAML in practice.

Reptile (Nichol et al., 2018) goes even further in simplicity. For each task, take several gradient steps on the support set (standard training), then move the meta-parameters toward the adapted parameters. No query set needed. No differentiation through the inner loop. The update rule is:

# Reptile meta-update (simplified)
# theta: current meta-parameters
# theta_prime: parameters after k steps of SGD on task i

theta = theta + epsilon * (theta_prime - theta)

Reptile is doing something subtle: by averaging the direction from the initial point to the adapted point across many tasks, it finds an initialization that is close (in parameter space) to the optimal parameters for all tasks. The math shows that Reptile approximates the MAML gradient in expectation, plus an additional term that pushes toward initializations with good generalization within tasks.

Inner Loop / Outer Loop

The two-loop structure deserves emphasis because it is the defining feature of optimization-based meta-learning. The inner loop is a standard learning algorithm — it optimizes for a specific task. The outer loop is the meta-learning algorithm — it optimizes the starting conditions (and sometimes the learning rate, architecture, or loss function) of the inner loop.

You can think of the inner loop as a function: it takes an initialization and a dataset, and produces adapted parameters. The outer loop optimizes over the space of possible initializations. This decomposition is what makes MAML "model-agnostic" — the inner loop can use any differentiable model, and the outer loop still works.

Task-Specific Adaptation

One limitation of vanilla MAML is that every parameter gets the same inner learning rate. Some parameters (like the final classification layer) might need to adapt quickly, while others (like early convolutional filters) should barely change. Task-specific adaptation strategies address this by learning per-parameter or per-layer learning rates, or by restricting which parameters get updated in the inner loop.

A common trick: freeze the backbone (feature extractor) and only adapt the classification head in the inner loop. This is much cheaper than adapting the entire network and often works better, because early features are general enough to transfer across tasks while the classifier needs to specialize.

Meta-SGD

Meta-SGD (Li et al., 2017) extends MAML by making the inner-loop learning rate a learnable parameter — not a single scalar, but a vector with one learning rate per model parameter. The meta-learner simultaneously finds a good initialization and a good per-parameter learning rate for fast adaptation.

The update rule for the inner loop becomes:

# Standard MAML inner step:
#   theta' = theta - alpha * grad(L)

# Meta-SGD inner step:
#   theta' = theta - alpha_vec * grad(L)
# where alpha_vec has the same shape as theta and is learned

This is a small change that can make a meaningful difference. Parameters that need big updates get big learning rates. Parameters that should stay stable get small ones. The outer loop learns both the initialization and the learning rate vector jointly.

Memory-Based Methods

The third family of meta-learning methods uses external memory to store and retrieve information about past experiences. Instead of encoding everything into network weights or embeddings, these models have an explicit memory bank they can read from and write to. The intuition: when you see a new example, store it in memory. When you need to classify a query, read from memory and compare.

Neural Turing Machines

Neural Turing Machines (NTMs), introduced by Graves et al. in 2014, were one of the first architectures with differentiable external memory. An NTM has a controller (a neural network) that interacts with a memory matrix through read and write heads. The heads use soft attention to address memory locations, making the entire process differentiable and trainable with backpropagation.

The controller decides what to write to memory (encoding new information) and what to read from memory (retrieving past information). The addressing mechanism supports both content-based lookup ("find the memory slot that looks most like this vector") and location-based addressing ("read the next slot after the one I just read"). For few-shot learning, the content-based lookup is what matters: store support examples in memory, then retrieve the most relevant ones when classifying a query.

Memory-Augmented Neural Networks (MANN)

Memory-Augmented Neural Networks (Santoro et al., 2016) adapted NTMs specifically for meta-learning. The key insight was to use the external memory as a persistent store across a sequence of learning episodes. The network sees labeled examples one at a time (with a deliberate one-step delay between seeing an example and seeing its label), and it must learn to bind each example to its label in memory so it can classify future examples of the same class.

The training procedure is clever. Labels are presented offset by one step — the network sees image xₜ alongside label yₜ₋₁ (the label for the previous image). This forces the model to write the current image into memory and wait for the label, then bind them together when the label arrives. At test time, this means the model can store new class examples and retrieve them — genuine one-shot learning through memory.

Memory vs. Metric Methods

Memory-based methods are more flexible than metric-based ones — they can store arbitrary information, not just class prototypes. But they are also harder to train, more computationally expensive, and haven't scaled as well to large-scale few-shot benchmarks. In practice, prototypical networks and MAML remain more popular for standard few-shot classification, while memory methods shine in sequential and continual learning settings.

External Memory Modules

The idea of external memory extends beyond NTMs. Any architecture that maintains a key-value store, a memory buffer, or a differentiable database can benefit from the same principle. Modern transformers use attention over long contexts in a way that resembles content-based memory addressing — each position in the sequence is a "memory slot," and the attention mechanism performs soft lookups.

Some recent approaches attach explicit memory modules to transformer backbones: a retrieval-augmented network might store support set embeddings in a memory bank and attend over them when processing a query. This blurs the line between memory-based meta-learning and modern retrieval-augmented generation, which we will revisit shortly.

Modern Few-Shot Learning

The landscape of few-shot learning shifted dramatically with the rise of large pretrained models. Methods that were state-of-the-art in 2018 — sophisticated meta-learning algorithms trained from scratch on mini-ImageNet — are now routinely outperformed by fine-tuning a pretrained vision transformer or prompting a large language model. The core insight remained, but the implementation changed.

In-Context Learning: LLMs as Few-Shot Learners

GPT-3 (Brown et al., 2020) demonstrated something remarkable: a large language model, with no parameter updates at all, can perform few-shot learning by conditioning on examples provided in the input prompt. You show the model a few input-output pairs, then give it a new input, and it produces the correct output. This is called in-context learning.

# In-context learning: no training, no gradient updates
# The model "learns" from the examples in the prompt

prompt = """
Classify the sentiment of movie reviews.

Review: "This film was a masterpiece of storytelling."
Sentiment: Positive

Review: "Terrible acting and a nonsensical plot."
Sentiment: Negative

Review: "The visuals were stunning but the script was weak."
Sentiment: Mixed

Review: "I couldn't stop laughing — best comedy this year."
Sentiment:"""

# The model completes with "Positive"
# Zero gradient updates. Zero fine-tuning. 
# The task was "learned" from 3 examples in context.

This is few-shot learning in a profound sense, but it works through a completely different mechanism than classical meta-learning. There is no inner loop or outer loop. There are no support-set embeddings. The model's parameters do not change. Instead, the attention mechanism over the prompt context serves as an implicit learning algorithm — the model figures out the pattern from the examples and applies it to the new input.

Research has shown that transformer attention heads can implement gradient-descent-like algorithms internally. In a sense, the forward pass of a large transformer on an in-context learning prompt is doing something analogous to the inner loop of MAML — but it is doing it through activation patterns rather than weight updates.

In-Context Learning Is Fragile

In-context learning is sensitive to the format, order, and selection of examples in the prompt. Shuffling the order of few-shot examples can swing accuracy by 10-20 percentage points. Adversarial prompt design can make models ignore the examples entirely and fall back on pretraining biases. Always validate in-context learning with diverse prompt orderings before trusting the results.

Prompt-Based Few-Shot Learning

Prompt-based few-shot learning extends in-context learning with more structured approaches to prompt construction. Instead of hand-writing prompts, you can learn them. Prompt tuning prepends a set of learnable continuous vectors (soft prompts) to the input and optimizes them on a small labeled set. The base model's parameters stay frozen — only the prompt vectors are updated.

This is remarkably parameter-efficient. A typical soft prompt might be 20 tokens × 4096 dimensions = about 80K parameters, compared to billions of parameters in the base model. Yet it can adapt the model to a new task nearly as well as full fine-tuning, especially when the base model is large enough.

Other prompt-based strategies include chain-of-thought prompting (showing the model how to reason step-by-step through examples), retrieval-augmented prompting (dynamically selecting the most relevant examples from a database), and instruction tuning (training the model to follow natural language task descriptions so it needs fewer examples at test time).

Foundation Models for Few-Shot

The biggest change in few-shot learning has been the emergence of foundation models — large models pretrained on diverse data at scale. Vision transformers pretrained with self-supervised methods (DINO, MAE, CLIP) produce embeddings that are so general-purpose that a simple nearest-neighbor classifier on top achieves state-of-the-art few-shot accuracy. No meta-learning algorithm needed.

CLIP, for example, was trained on 400 million image-text pairs. Its image encoder maps photos into a space where semantic concepts are well-organized. Want to do 5-way 1-shot classification on a new dataset? Encode the 5 support images with CLIP, encode the query, and pick the nearest prototype. This straightforward approach beats many meta-learning methods that were carefully designed for few-shot learning.

Why does this work? The pretraining data was so large and diverse that the model has already seen something similar to almost any new class. The representation has already learned to separate concepts that matter. Meta-learning's job — learning a representation good for fast adaptation — was already done during pretraining, at massive scale.

Cross-Domain Few-Shot Transfer

Standard few-shot benchmarks assume the training tasks and test tasks come from the same domain (e.g., both are natural images). Cross-domain few-shot transfer is the harder, more realistic setting where the test tasks come from a different domain entirely — train on natural images, test on medical X-rays or satellite imagery.

This is where meta-learning methods often struggle. An embedding space optimized for distinguishing dog breeds may not transfer to distinguishing cell types. Foundation models pretrained on diverse data have a natural advantage here because their representations are more general. But even they can fail when the target domain is far from anything in the pretraining data.

Current best practices for cross-domain few-shot transfer combine three ingredients: (1) a strong pretrained backbone, (2) lightweight task-specific adaptation (like prompt tuning or adapter layers), and (3) domain-specific data augmentation to bridge the distribution gap. This combination acknowledges that no single approach — meta-learning, transfer learning, or prompting — is sufficient on its own, but together they cover each other's weaknesses.

The Meta-Learning Landscape Today

Classical meta-learning (MAML, prototypical networks) remains important for understanding how fast adaptation works and for settings where you cannot rely on a large pretrained model. But in practice, the field has converged on "pretrain big, adapt small" as the dominant recipe. The principles of meta-learning — episodic evaluation, support/query splits, learning to adapt — now live inside the training procedures and evaluation protocols of foundation models rather than as standalone algorithms.
