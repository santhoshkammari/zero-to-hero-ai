Meta-Learning and Few-Shot Learning

TL;DR

Meta-learning trains models that learn how to learn — adapting to new tasks from just a handful of examples. Prototypical Networks (embed & compare to class means) and MAML (learn a weight initialization designed for fast fine-tuning) are the two methods you actually need to know. In 2024+, few-shot prompting and fine-tuning large foundation models often beat classical meta-learning outright. But meta-learning still wins in on-device, low-resource, and fast-adaptation scenarios where you can't phone home to a 70B-parameter model.

The Setup: Episodes, Support Sets, and N-way K-shot

Normal supervised learning: you have a big dataset, you train a model on it, done. Meta-learning flips this. Instead of one big task, you train on hundreds of tiny tasks, each designed to simulate the "I only have a few examples" situation you'll face at deployment.

The Meta-Learning Loop

Inner loop: Learn a specific task from a few examples (like a student taking a quiz).
Outer loop: Improve the learning process itself across many tasks (like a teacher refining the curriculum after watching many students).

Each tiny task is called an episode. An episode has two parts:

  • Support set — the few labeled examples the model gets to "train" on (think: tiny training set)
  • Query set — examples the model must classify after seeing the support set (think: tiny test set)

The standard benchmark format is N-way K-shot: N classes, K labeled examples per class in the support set. So "5-way 1-shot" means: here are 5 classes you've never seen, with 1 example each — now classify these new images. Brutal, but that's the game.

Why Episodic Training Works

Training conditions match test conditions. The model never gets comfortable with thousands of examples — it always practices the "learn from a handful" skill. Over thousands of episodes with different random class combinations, the meta-learner builds transferable knowledge about how to distinguish new classes quickly.

Metric-Based: Prototypical Networks

The simplest and most practical family of few-shot methods. The core insight: learn an embedding space where similar things cluster together, then classify by proximity. No gradient updates at test time — the encoder does all the heavy lifting.

How Prototypical Networks Work

Snell et al. (2017) asked: what's the simplest thing that could possibly work? Their answer:

  1. Embed all support examples through a shared encoder
  2. Compute the prototype (mean embedding) for each class
  3. Embed the query, measure distance to each prototype, classify by nearest one

That's the whole algorithm. Take the average and measure distance. Why does the mean work so well? In a well-learned embedding space, class examples cluster tightly. The mean is more robust than any individual example — especially when K is small and individual examples may be atypical.

import torch
import torch.nn.functional as F

def prototypical_loss(encoder, support_x, support_y, query_x, query_y, n_way):
    """One episode of Prototypical Network training."""
    z_support = encoder(support_x)
    z_query = encoder(query_x)

    # Compute class prototypes (mean of support embeddings per class)
    prototypes = torch.stack([
        z_support[support_y == c].mean(dim=0) for c in range(n_way)
    ])

    # Classify queries by nearest prototype (negative distance = logits)
    dists = torch.cdist(z_query, prototypes)
    log_probs = F.log_softmax(-dists, dim=-1)

    loss = F.nll_loss(log_probs, query_y)
    acc = (log_probs.argmax(dim=-1) == query_y).float().mean()
    return loss, acc

The distance metric matters. The original paper found that Euclidean distance outperformed cosine similarity — because negative squared Euclidean distance + softmax is equivalent to a linear classifier, producing clean hyperplane decision boundaries between prototypes.

Why Prototypical Networks Are Still Relevant

They're fast (single forward pass at inference), dead simple to implement, and serve as a surprisingly tough baseline. Many fancier methods from 2018–2020 only beat ProtoNets by 1–2% while being 10× more complex. If you need a few-shot classifier today and can't use a foundation model, start here.

Optimization-Based: MAML

Metric-based methods assume that a good embedding + distance is enough. MAML (Finn et al., 2017) takes a different bet: what if you could find a weight initialization so good that a few gradient steps on any new task produce a well-adapted model?

MAML in One Sentence

Find initial weights θ such that a few gradient descent steps on a new task's support set yields parameters that perform well on that task's query set. You're not learning a model — you're learning a starting point for fast adaptation.

The mechanics:

  1. For each task in a meta-batch, take a few gradient steps on the support set → adapted parameters θ'
  2. Evaluate θ' on the query set → compute loss
  3. Backpropagate that query loss through the inner gradient steps to update θ
# MAML pseudocode (the idea, not production code)
for episode in episodes:
    support_x, support_y, query_x, query_y = episode

    theta_prime = theta.clone()
    for step in range(inner_steps):                      # inner loop
        loss = cross_entropy(model(support_x, theta_prime), support_y)
        theta_prime = theta_prime - inner_lr * grad(loss, theta_prime)

    query_loss = cross_entropy(model(query_x, theta_prime), query_y)
    theta = theta - outer_lr * grad(query_loss, theta)   # outer loop

The key detail: grad(query_loss, theta) differentiates through the inner loop. That's second-order MAML — you're computing gradients of gradients. This is expensive (Hessian-vector products, big computation graphs), which is why people usually use only 1–5 inner steps.

First-Order Shortcuts

FOMAML drops the second-order terms entirely — just pretend the inner loop didn't involve θ. Surprisingly, it works nearly as well. Reptile (Nichol et al., 2018) goes further: just take several SGD steps on the support set, then nudge θ toward the result. No query set, no second derivatives. Both are much cheaper and often good enough in practice.

Comparison: Which Method When?

Method Core Idea Test-Time Cost Flexibility Best For
Prototypical Nets Embed → mean → nearest prototype Very low (forward pass only) Limited to metric space Fast inference, simple deployment
Matching Nets Embed → attention-weighted nearest neighbor Low Slightly more than ProtoNet When individual examples matter more than means
MAML Learn initialization → fine-tune on task Higher (gradient steps at test time) High (any differentiable model) Heterogeneous tasks, regression, RL
Reptile / FOMAML Cheap approximation of MAML Moderate High When MAML is too expensive

Rule of thumb: If your tasks are all classification with a shared embedding, ProtoNets win on simplicity. If your tasks are structurally different (some classification, some regression, different output spaces), MAML's flexibility justifies the cost.

The Honest Truth: Meta-Learning vs. Foundation Models

Here's the uncomfortable reality for the meta-learning community. In 2024+, for most practical few-shot problems, you should try these first:

  1. Few-shot prompting an LLM (for text tasks) — zero training, often good enough
  2. Fine-tuning a foundation model (CLIP, DINOv2, etc.) with a linear probe on your few examples
  3. LoRA / adapter fine-tuning a pretrained model on your support set

These approaches leverage billions of parameters trained on internet-scale data. A CLIP embedding + nearest-neighbor on 5-way 1-shot ImageNet subsets beats many purpose-built meta-learning algorithms. The representation has already "learned to learn" during pretraining, at a scale no meta-learning dataset can match.

So Is Meta-Learning Dead?

No. But its role has shifted. The principles of meta-learning (episodic evaluation, learning to adapt, support/query protocol) are baked into how we train and evaluate foundation models. The standalone algorithms matter less when you have access to a strong pretrained backbone. The concepts are alive — the specific 2017-era methods are increasingly niche.

Where Meta-Learning Still Wins

Foundation models aren't always available. Meta-learning earns its keep in specific scenarios:

  • On-device / edge deployment — You can't run a 7B model on a microcontroller. A lightweight ProtoNet encoder (2M params) can adapt to new classes in milliseconds.
  • Truly novel domains — Medical imaging modalities, satellite hyperspectral data, or industrial defect detection where pretrained models have zero relevant training data.
  • Rapid adaptation loops — Robotics tasks where the robot must adapt to a new object or environment in seconds, not minutes of fine-tuning.
  • Privacy-constrained settings — When data can't leave the device and you can't query an API. Meta-learning gives you a local adaptation mechanism.
  • Task distribution shifts — When new task types appear frequently and you need a systematic way to handle them, not manual prompt engineering each time.
Memory-Augmented Methods: A Brief Note

A third family — memory-augmented neural networks (MANNs) — uses external key-value memory to store and retrieve support examples. These were exciting circa 2016 but haven't scaled well compared to metric and optimization methods. The concept lives on in retrieval-augmented approaches, but the original NTM-style architectures are mostly of historical interest.

Practical Decision Flowchart

When you're facing a few-shot problem in practice, think through it like this:

  1. Do you have API access to a large model? → Try few-shot prompting or fine-tuning first. Seriously, just try it.
  2. Is the domain well-covered by existing pretrained models? → Linear probe or LoRA on a foundation model's embeddings.
  3. Are you constrained by latency, model size, or privacy? → Prototypical Networks with a small backbone.
  4. Do your tasks vary structurally (different output types, different loss functions)? → MAML or Reptile.
  5. Is this a research benchmark? → Meta-learning methods are still the right comparison point for algorithmic advances.

Emerging Architectures and Continual Learning

TL;DR

Two awareness topics in one section. First: the 2024–2025 shift from "train bigger" to "think harder at inference time," plus the post-transformer architectures worth watching. Second: why neural networks catastrophically forget old tasks when you train on new ones, and the three families of defenses — regularization, replay, and architecture-based methods. Neither topic needs deep mastery here; you need to know they exist, roughly how they work, and when they'd show up in your production life.

Test-Time Compute Scaling — The Big Idea

For years, the recipe was simple: want a better model? Train a bigger one on more data. GPT-2 → GPT-3 → GPT-4 — each jump meant orders of magnitude more parameters. But around 2024 we hit diminishing returns. Training costs are astronomical, high-quality data is running out, and the gains per additional parameter are flattening.

So researchers asked a different question: what if we spend more compute when the model is already answering, not during training?

Test-Time Compute Scaling

Instead of a single forward pass, you let the model think longer. Generate multiple candidate answers. Score them. Search over reasoning chains. Verify intermediate steps. The model stays the same size, but output quality goes up because you invest more FLOPS at inference.

The simplest version: Best-of-N sampling. Generate N completions, score each with a reward model, pick the winner. If your model is correct 30% of the time in a single shot, Best-of-16 can push that to ~95% — provided the reward model can actually tell good from bad.

The more sophisticated version: tree search over reasoning steps. Treat each reasoning step as a node in a tree, use a process reward model (PRM) to score individual steps, and prune bad branches early. This is the core idea behind OpenAI's o1/o3 models — they don't generate one chain of thought, they explore a search tree.

# The concept in pseudocode
def test_time_scaling(model, reward_model, prompt, n=16):
    """Generate N candidates, return the best-scored one."""
    candidates = [
        model.generate(prompt, temperature=0.8)
        for _ in range(n)
    ]
    scored = [(c, reward_model.score(prompt, c)) for c in candidates]
    return max(scored, key=lambda x: x[1])[0]
The New Scaling Law

Pre-2024: performance ∝ log(training_compute). Post-2024 we have a second axis: performance also scales with log(inference_compute). A well-trained 7B model with enough inference budget can match a 70B model on reasoning tasks. The economic implication is huge — training is a one-time cost, but inference is per-query tunable.

When this matters in practice: any task with verifiable answers (math, code, formal logic) benefits enormously from test-time scaling. Tasks requiring broad world knowledge or nuanced judgment? The bigger model still wins. Plan your serving budget accordingly.

Transformer Alternatives — A Quick Map

The transformer's self-attention is O(n²) in sequence length. For 2K tokens that's manageable. For 128K tokens it's over 16 billion comparisons. Several architectures aim to fix this.

We covered State Space Models (Mamba) in depth in Ch 10 — go read that section if you haven't. The short version: Mamba gets competitive quality with O(n) compute by replacing attention with a selective state-space mechanism. It processes sequences like an RNN at inference (constant memory per token) but trains in parallel like a transformer.

Similarly, Mixture of Experts (MoE) got a thorough treatment in Ch 12. The key idea: activate only a subset of parameters per token so you get the capacity of a huge model at the cost of a small one.

Beyond those, here's what else is worth knowing about:

Which One Wins?

As of early 2025, no single alternative has dethroned the transformer. The pragmatic money is on hybrids — Jamba-style architectures that use cheap linear layers for the bulk of processing and sprinkle in a few attention layers where the "random-access lookup" capability actually matters. Pure transformers still dominate at the frontier, but the cost gap is closing fast.

What Else Is Worth Watching

BitNet / 1-bit LLMs. Ternary weights (-1, 0, +1) — no floating-point multiply at all, just add and subtract. BitNet b1.58 matches full-precision transformers of the same size on language modeling. The promise: models that run on cheap integer hardware with 10× memory savings. The catch: you have to train from scratch with a special recipe, can't just quantize an existing model.

Mixture-of-Depths. Not all tokens need all layers. A lightweight router decides which tokens get processed by each layer and which skip via the residual connection. Cuts FLOPS by 30–50% with minimal quality loss. Think MoE but routing between "compute" and "skip" instead of between experts.

Byte-level models. No tokenizer — process raw UTF-8 bytes. Eliminates BPE artifacts, OOV tokens, and "the model can't spell" problems. The challenge is that byte sequences are 3–4× longer, so you need efficient architectures (Mamba, local-global attention) to handle the length. Meta's MambaByte shows it's viable.

Matryoshka embeddings. Train embeddings so every prefix is a valid embedding at lower dimension. Your 1024-dim vector? The first 128 dims are a solid 128-dim embedding. One model, multiple operating points, no retraining. OpenAI's text-embedding-3 uses this. Great for retrieval systems — cheap coarse search, then full-precision reranking.

Catastrophic Forgetting — The Core Problem

Here's a scenario that haunts production ML. You train a model to classify animals — cats, dogs, birds. Works great. Business says "add cars and trucks." You fine-tune on vehicle images. Deploy. And suddenly the model can't tell a cat from a dog anymore.

This isn't a bug. It's a fundamental property of neural networks. Every gradient update that improves vehicle recognition overwrites the weight configurations that encoded animal knowledge. The name: catastrophic forgetting.

The Stability-Plasticity Dilemma

Plasticity = ability to learn new information (reshape weights). Stability = ability to retain old information (resist changes). Too much plasticity → forget everything. Too much stability → can't learn anything new. Every continual learning method is a different answer to this tradeoff.

The naive fix — retrain from scratch on all data combined — works if you have all the old data and unlimited compute. In the real world, the original data might be deleted, governed by privacy laws, or too expensive to keep. Medical records get purged. User data expires. And retraining a large model from scratch every time new classes arrive is absurdly wasteful.

Three Families of Defenses

1. Regularization: Protect Important Weights

The philosophy: don't prevent weight changes — make the important changes expensive. If a weight was critical for Task A, penalize the loss when it drifts too far from its Task A value.

The landmark method is Elastic Weight Consolidation (EWC). After training on Task A, estimate each weight's importance using the Fisher Information Matrix. Then add a penalty during Task B training:

L_total = L_taskB(θ) + (λ/2) Σᵢ Fᵢ(θᵢ - θ*_taskA,i)²

# Fᵢ = Fisher Information (importance) for weight i
# High Fᵢ = stiff spring, weight stays near Task A value
# Low Fᵢ  = loose spring, weight is free to change

Think of it as per-weight spring constants — the "elastic" in the name. Variants like Synaptic Intelligence (SI) accumulate importance during training instead of computing it after, and Memory Aware Synapses (MAS) measure output sensitivity without needing labels. In practice all three perform similarly. The limit: with many tasks, the springs pile up and eventually there's no room to move at all.

2. Replay: Keep Some Old Data Around

Conceptually the simplest defense. Maintain a small buffer of examples from previous tasks (maybe 200–500 per task). Each training batch mixes new data with replay samples. It's crude and it works surprisingly well.

# Simplified replay training loop
for new_x, new_y in new_task_loader:
    replay_x, replay_y = buffer.sample(batch_size=len(new_x) // 2)
    combined_x = torch.cat([new_x, replay_x])
    combined_y = torch.cat([new_y, replay_y])
    loss = F.cross_entropy(model(combined_x), combined_y)
    loss.backward()
    optimizer.step()

A smarter variant: A-GEM (Averaged Gradient Episodic Memory). Instead of mixing data, it uses the buffer to constrain gradients — if the new-task gradient would increase the loss on old tasks, project it to a safe direction. More principled, roughly the same cost.

When you can't store real data (privacy, legal), generative replay trains a generator alongside the classifier and synthesizes pseudo-examples of old tasks. Elegant, but if the generator forgets too… cascading failure.

3. Architecture-Based: Separate Parameters Per Task

Progressive Networks add a new column for each task, freeze old columns, and connect them with lateral links. Zero forgetting by construction, but the network grows linearly with tasks — impractical after a dozen.

PackNet is cleverer: train on Task 1, prune aggressively, freeze surviving weights, train the pruned-away weights on Task 2. Each task gets its own weight subset within a fixed network. Works for 10–20 tasks before you run out of capacity.

Dynamic Expandable Networks split the difference — try existing capacity first, grow only if needed. A trivial task adds zero neurons; a radically different task adds more.

When Continual Learning Matters in Production

You probably don't need formal continual learning methods if you can retrain from scratch cheaply and have all your data. Most teams can. But here's when you can't avoid it:

Streaming data with concept drift. Fraud detection, ad click prediction, recommendation — the underlying distribution shifts weekly. You need to adapt without forgetting rare-but-important patterns from months ago.

Privacy-constrained updates. Medical imaging models need new disease classes, but old patient data must be deleted after a retention period. You literally can't retrain from scratch.

On-device personalization. A phone keyboard model adapts to your typing patterns. It can't ship your data to the cloud, can't store unlimited history, and needs to keep working on English even after you start texting in Spanish.

Foundation model fine-tuning. Every time you fine-tune an LLM on a new domain, you risk degrading its general capabilities. This is continual learning whether you call it that or not. Methods like EWC-style regularization and replay of general-domain data are increasingly standard in fine-tuning pipelines.

Honest Assessment: Where the Field Is

No continual learning method fully solves catastrophic forgetting at scale. The best results come from combining approaches — EWC-style regularization plus a small replay buffer plus task-aware output heads. In practice, most production systems still just retrain periodically on all available data, using continual learning techniques mainly to bridge the gap between full retrains. The field is important and active, but don't expect a drop-in solution that "just works" today.

Putting It Together: The 2025 Landscape

Both topics in this section share a theme: making models smarter about how they use their capacity.

Emerging architectures ask: can we spend compute more wisely — at inference time instead of training time, on important tokens instead of all tokens, with cheaper operations instead of full attention? The answer is increasingly yes, and the practical impact is that smaller, cheaper models can punch above their weight when paired with the right inference strategy.

Continual learning asks: can we update models incrementally instead of retraining from scratch? The answer is "sort of, with caveats." The methods work, but they're not magic — you're always trading off plasticity against stability, and the more tasks you stack, the harder the tradeoff gets.

Both are awareness topics. You don't need to implement RWKV from scratch or build an EWC training loop for your next project. But you do need to know these concepts exist so you can recognize when they're relevant — and reach for them when the standard "train big, retrain often" playbook stops working.
