Graph Neural Networks

Chapter 16 Graphs GCN GAT GraphSAGE PyG

Most deep learning assumes grids (images) or sequences (text). Graphs — social networks, molecules, knowledge bases — don't fit either mold. GNNs learn node representations by iteratively passing messages between neighbors. The three architectures you actually need to know: GCN (simple weighted average of neighbors), GAT (learned attention over neighbors), and GraphSAGE (sampled neighborhoods for scale). Most ML engineers won't use GNNs daily, but when your data is a graph, nothing else comes close.

When Do You Actually Need a GNN?

Before you reach for PyTorch Geometric, ask yourself: is my data fundamentally a graph, or am I just forcing it into one?

Good fits for GNNs:

  • Molecules — atoms are nodes, bonds are edges. Predicting toxicity, solubility, binding affinity.
  • Social networks — fraud detection, community detection, influence prediction.
  • Knowledge graphs — entity/relation completion (e.g., "who directed X?").
  • Recommendation — user-item interaction graphs (Pinterest, Twitter use GNNs in production).
  • Traffic/logistics — road networks, supply chains with spatial structure.

When you probably don't need one: if you can flatten your graph into a feature vector per entity without losing critical relational info, a good old XGBoost or MLP will likely beat a GNN with less hassle. GNNs shine when the topology itself carries signal — when who-is-connected-to-whom matters as much as the features on each node.

Rule of thumb: If removing all edges and training on node features alone gives you 95% of the performance, you don't need a GNN. Test this before committing.

The Message Passing Paradigm

This is the one mental model you need. Every modern GNN — GCN, GAT, GraphSAGE, GIN, you name it — is a variation of the same three-step loop, repeated once per layer:

1. Message: Each node creates a message for each neighbor (often just its own feature vector, sometimes transformed).

2. Aggregate: Each node collects all incoming messages and reduces them — sum, mean, max, or something learned. This must be order-invariant because graphs don't have a canonical node ordering.

3. Update: Each node combines its own features with the aggregated message via a learnable function (linear layer + nonlinearity, MLP, etc.).

# The entire message passing idea in ~5 lines of math-flavored pseudocode
def mp_layer(node_feats, adjacency, W):
    messages = adjacency @ node_feats   # aggregate neighbor features
    updated  = relu(messages @ W)       # transform + nonlinearity
    return updated

Stack k layers and each node's representation captures its k-hop neighborhood. Two layers = friends-of-friends. Three layers = three hops out. This is both the power and the trap of GNNs (more on over-smoothing shortly).

Key insight: Message passing is basically a learnable, iterative version of "average your neighbors' features." The architecture differences (GCN vs GAT vs GraphSAGE) boil down to how you weight, select, and combine those neighbor features.

The Big Three Architectures

GCN — Graph Convolutional Network

Kipf & Welling (2017) simplified spectral graph convolutions down to one elegant rule: average neighbor features (including yourself), multiply by a weight matrix, apply ReLU. That's it.

The layer equation: H(l+1) = σ(D̃−½ Ã D̃−½ H(l) W(l))

Where Ã = A + I (adjacency with self-loops) and D̃ is the corresponding degree matrix. The symmetric normalization prevents high-degree nodes from dominating. In practice you precompute D̃−½ Ã D̃−½ once and reuse it.

GCN's strength is simplicity. Its weakness: all neighbors get equal weight (after degree normalization). Sometimes that's fine. Sometimes it's not.

GAT — Graph Attention Network

In a citation network, not all references are equally relevant. GAT learns which neighbors matter by computing attention scores between node pairs — essentially the same idea as Transformer attention, but restricted to graph edges.

For each neighbor j of node i:

  • Project both features: Whi, Whj
  • Score: eij = LeakyReLU(aT[Whi ‖ Whj])
  • Normalize: αij = softmaxj(eij)
  • Aggregate: h'i = σ(Σ αij · Whj)

Multi-head attention (same as Transformers) stabilizes training. Pro tip: use GATv2 (Brody et al., 2021) for new projects — it fixes a subtle expressiveness limitation in the original where attention rankings are static regardless of the query node.

GraphSAGE — SAmple and aggreGatE

GCN and GAT need the full graph at training time — every node, even unlabeled ones. That's fine for a static citation dataset. It's useless for Pinterest where new pins appear every second.

GraphSAGE fixes this with two ideas:

  1. Inductive learning: Instead of learning fixed per-node embeddings, learn an aggregation function. New node shows up? Feed its neighborhood through the same function.
  2. Neighborhood sampling: At each layer, randomly sample a fixed number of neighbors (e.g., 25 first-hop, 10 second-hop). This caps computation and enables mini-batch training on billion-edge graphs.

Aggregation options: mean (default, works great), LSTM (technically not order-invariant — they shuffle neighbors), or max-pool (MLP per neighbor, then element-wise max).

Start with GCN. If error analysis suggests "the model doesn't know which neighbors matter," try GAT. If you need to scale past what fits in GPU memory, go GraphSAGE.

The Three Task Types

Node Classification

Given a partially-labeled graph, predict labels for the rest. Classic semi-supervised learning — message passing touches all nodes (even unlabeled ones), but loss is computed only on labeled nodes. Think: classifying papers by topic, detecting fraudulent accounts.

Link Prediction

Which edges are missing? Run GNN to get node embeddings, then score candidate pairs — dot product is the simplest decoder. This powers "people you may know" features and knowledge graph completion. Watch your negative sampling: accidentally using real edges as negatives will silently corrupt training.

Graph Classification

One label per graph. Is this molecule toxic? Is this code snippet malicious? Run message passing, then pool all node embeddings into a single graph vector (mean, sum, or learned attention pooling), and classify. This is the molecular property prediction setup.

PyTorch Geometric in Practice

PyG is the de facto library. Here's a realistic node classification example on Cora (the "MNIST of GNNs"):

import torch
import torch.nn.functional as F
from torch_geometric.nn import GCNConv, GATConv, SAGEConv
from torch_geometric.datasets import Planetoid

dataset = Planetoid(root='./data', name='Cora')
data = dataset[0]  # single graph: 2708 nodes, 10556 edges, 7 classes

class FlexGNN(torch.nn.Module):
    def __init__(self, in_ch, hidden, out_ch, conv_type='gcn'):
        super().__init__()
        Conv = {'gcn': GCNConv, 'gat': GATConv, 'sage': SAGEConv}[conv_type]
        self.conv1 = Conv(in_ch, hidden)
        self.conv2 = Conv(hidden, out_ch)

    def forward(self, x, edge_index):
        x = F.relu(self.conv1(x, edge_index))
        x = F.dropout(x, p=0.5, training=self.training)
        return self.conv2(x, edge_index)

model = FlexGNN(dataset.num_features, 64, dataset.num_classes, conv_type='gcn')
optimizer = torch.optim.Adam(model.parameters(), lr=0.01, weight_decay=5e-4)

# Training loop
for epoch in range(200):
    model.train()
    optimizer.zero_grad()
    out = model(data.x, data.edge_index)
    loss = F.cross_entropy(out[data.train_mask], data.y[data.train_mask])
    loss.backward()
    optimizer.step()

# Eval
model.eval()
pred = model(data.x, data.edge_index).argmax(dim=1)
acc = (pred[data.test_mask] == data.y[data.test_mask]).float().mean()
print(f"Test accuracy: {acc:.3f}")  # ~81% for GCN, ~83% for GAT

Notice how swapping architectures is a one-line change — conv_type='gat'. PyG abstracts the message passing, so you focus on which Conv layer and what hyperparameters.

PyG data format: Graphs are stored as edge_index (2 × num_edges tensor of source/target pairs) plus x (node feature matrix). No dense adjacency matrices — everything is sparse by default. For batching multiple graphs, PyG stacks them into one big disconnected graph with a batch vector to track which node belongs to which graph.

Production Gotchas

Over-Smoothing

The #1 GNN pitfall. Each layer averages a node with its neighbors. Two layers: friends-of-friends. Five layers in a small-world graph: every node sees almost every other node. All representations converge to the same blob. It's like Gaussian-blurring an image ten times.

In practice, 2–3 layers is the sweet spot for most GNNs. Mitigations exist — residual connections, DropEdge (randomly drop edges during training), JumpingKnowledge (concatenate representations from all layers) — but none fully solve it. If you need long-range dependencies, look at Graph Transformers rather than stacking more GNN layers.

Scalability

Full-batch GCN/GAT on a billion-node graph? Not happening. Your options:

  • Neighbor sampling (GraphSAGE-style) — cap the neighborhood per layer
  • Cluster-GCN — partition the graph into clusters, train on subgraphs
  • Mini-batch via PyG's NeighborLoader — the practical default
from torch_geometric.loader import NeighborLoader

# Sample 15 first-hop and 10 second-hop neighbors per target node
loader = NeighborLoader(data, num_neighbors=[15, 10], batch_size=128, shuffle=True)

for batch in loader:
    out = model(batch.x, batch.edge_index)
    # batch.batch_size tells you how many target nodes are in this batch
    loss = F.cross_entropy(out[:batch.batch_size], batch.y[:batch.batch_size])
    # ... backward, step, etc.

Other Things That Bite

  • Feature leakage in link prediction: If you include the target edge during message passing, the model trivially learns to "see" it. Always remove target edges from the message-passing graph.
  • Heterogeneous graphs: Real-world graphs have multiple node/edge types. Use R-GCN or HGT — standard GCN assumes homogeneity.
  • Dynamic graphs: If edges change over time (transactions, interactions), look at Temporal Graph Networks (TGN) rather than running static GNNs on snapshots.
  • Expressiveness ceiling: Standard message-passing GNNs cannot distinguish certain non-isomorphic graphs (bounded by the 1-WL test). For molecular tasks where this matters, consider GIN or higher-order methods.
