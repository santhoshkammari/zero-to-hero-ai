Continual and Lifelong Learning

Chapter 16: Advanced Deep Learning 6 subtopics
TL;DR

Neural networks are catastrophically forgetful — train on task A, then task B, and performance on task A collapses. Continual learning is the field that fights this. The three main strategies are: regularization (penalize changes to important weights, like EWC using the Fisher Information Matrix), replay (keep a small buffer of old examples and mix them in), and architecture-based methods (grow the network or freeze parts of it per task). None fully solves the problem yet, but combining them gets you surprisingly far. The Avalanche framework makes it practical to benchmark these methods on standard splits like Split CIFAR and Split MNIST.

Problem Setting

Here is a scenario that will haunt you if you build production ML systems. You spend weeks training a model to classify animals — cats, dogs, birds. It works great. Then the business says "add cars and trucks." You fine-tune on vehicle images. You deploy. And suddenly the model cannot tell a cat from a dog anymore. It has forgotten everything about animals.

This is not a bug in your code. It is a fundamental property of how neural networks learn. Every gradient update that makes the network better at vehicles overwrites the weight configurations that encoded animal knowledge. This problem has a name: catastrophic forgetting.

The naive fix — retrain from scratch on all data combined — works if you have all the old data and unlimited compute. In the real world, you often don't. The original training data might be deleted, governed by privacy laws, or too large to store. Medical records get purged. User behavior data expires. And even if you have everything, retraining a large model from scratch every time new categories arrive is absurdly expensive.

Continual learning (also called lifelong learning or incremental learning) is the study of how to train a model on a sequence of tasks without forgetting previous ones — and ideally, getting better at new tasks because of what was learned before.

The Stability-Plasticity Dilemma

At the heart of continual learning lies a tension that neuroscientists identified decades ago. A learning system needs two opposing properties. Plasticity is the ability to learn new information — to reshape weights in response to new data. Stability is the ability to retain old information — to resist changes that would destroy previous knowledge. Too much plasticity and you forget everything. Too much stability and you cannot learn anything new. Every continual learning method is, at its core, a different answer to this tradeoff.

Three Flavors of Continual Learning

Not all continual learning problems are equal. The field distinguishes three scenarios, each progressively harder:

Task-Incremental Learning is the easiest setting. At test time, the model is told which task the input belongs to. "This is a Task 2 input — classify it among Task 2 classes." The model can maintain separate output heads per task. Think of it like having different exams for different subjects — you know which exam you're taking.

Domain-Incremental Learning is trickier. The task structure stays the same (e.g., sentiment analysis), but the data distribution shifts — maybe you start with movie reviews, then get restaurant reviews, then product reviews. The model is not told which domain the input comes from. Same output format, different input statistics.

Class-Incremental Learning is the hardest. New classes arrive over time, and at test time the model must distinguish among all classes ever seen — without being told which "task" the input belongs to. That animal-then-vehicle scenario? That's class-incremental. The model needs a single unified classifier that grows.

Transfer Metrics: BWT and FWT

To evaluate continual learning methods, we need metrics beyond plain accuracy. Two are especially important.

Backward Transfer (BWT) measures how learning a new task affects performance on old tasks. Negative BWT means forgetting. Positive BWT means the new task actually improved performance on old ones (rare, but it happens when tasks share useful structure).

Forward Transfer (FWT) measures how previous tasks help with new ones. High FWT means the model has learned representations that generalize — knowledge from Task 1 gives a head start on Task 3.

# Computing BWT and FWT from an accuracy matrix R
# R[i][j] = accuracy on task j after training on task i
import numpy as np

def compute_transfer_metrics(R):
    """
    R: (T x T) matrix where R[i][j] = acc on task j after training up to task i
    """
    T = R.shape[0]

    # Backward Transfer: average of R[T-1][j] - R[j][j] for j < T
    bwt = np.mean([R[T-1, j] - R[j, j] for j in range(T - 1)])

    # Forward Transfer: average of R[i-1][i] - baseline[i] for i > 0
    # baseline[i] is accuracy on task i before any training (random init)
    # Often approximated as R[i-1][i] if baseline isn't recorded
    fwt = np.mean([R[i-1, i] for i in range(1, T)])

    avg_accuracy = np.mean([R[T-1, j] for j in range(T)])

    return {
        'avg_accuracy': avg_accuracy,
        'backward_transfer': bwt,
        'forward_transfer': fwt
    }

# Example: 3-task scenario
R = np.array([
    [0.95, 0.00, 0.00],   # After task 0: great on task 0
    [0.72, 0.91, 0.00],   # After task 1: task 0 dropped (forgetting!)
    [0.65, 0.80, 0.93],   # After task 2: more forgetting on tasks 0 and 1
])

metrics = compute_transfer_metrics(R)
print(f"Average Accuracy: {metrics['avg_accuracy']:.3f}")   # 0.793
print(f"Backward Transfer: {metrics['backward_transfer']:.3f}")  # -0.245 (forgetting)
print(f"Forward Transfer: {metrics['forward_transfer']:.3f}")    # 0.000

A BWT of -0.245 tells you the model lost about 24.5 percentage points on old tasks on average. That is catastrophic forgetting in action. The goal of every method below is to push BWT closer to zero (or even positive) without sacrificing accuracy on the current task.

Regularization-Based Methods

The first family of defenses against forgetting takes a philosophical stance: don't prevent weight changes — just make the important weight changes expensive. If a weight was critical for Task A, penalize the loss function when that weight moves too far from its Task A value. Unimportant weights can change freely to learn Task B.

The question is: how do you know which weights are important?

Elastic Weight Consolidation (EWC)

Elastic Weight Consolidation is the landmark method from Kirkpatrick et al. (2017, DeepMind). The core idea is beautiful. After training on Task A, compute how important each weight is for Task A's performance. Then, when training on Task B, add a penalty that discourages moving important weights away from their Task A values.

The importance of each weight is estimated using the Fisher Information Matrix. Intuitively, the Fisher tells you: "If I wiggle this weight a tiny bit, how much does the output distribution change?" If the output changes a lot, the weight is important — the model's predictions depend heavily on it. If wiggling the weight barely changes anything, the weight is unimportant and can be repurposed for the new task.

Mathematically, the EWC loss when training on task B is:

L_total = L_B(θ) + (λ / 2) Σᵢ Fᵢ (θᵢ - θ*_A,i)²

Where:
  L_B(θ)      = standard loss on task B data
  θ*_A        = optimal weights after training on task A
  Fᵢ          = diagonal Fisher Information for weight i (importance)
  λ            = hyperparameter controlling regularization strength
  θᵢ           = current weight value

The Fisher acts like a per-weight spring constant. High Fisher value = stiff spring = weight stays close to its Task A value. Low Fisher value = loose spring = weight is free to change. This is where the "elastic" in the name comes from.

Here is a complete PyTorch implementation:

import torch
import torch.nn as nn
import torch.nn.functional as F
from copy import deepcopy

class EWC:
    def __init__(self, model, dataloader, device, importance=1000):
        self.model = model
        self.importance = importance  # λ in the formula
        self.device = device

        # Store the parameters after training on previous task
        self.prev_params = {
            name: param.clone().detach()
            for name, param in model.named_parameters()
            if param.requires_grad
        }

        # Compute Fisher Information Matrix (diagonal approximation)
        self.fisher = self._compute_fisher(dataloader)

    def _compute_fisher(self, dataloader):
        """Estimate diagonal Fisher Information from data."""
        fisher = {
            name: torch.zeros_like(param)
            for name, param in self.model.named_parameters()
            if param.requires_grad
        }

        self.model.eval()
        n_samples = 0

        for inputs, targets in dataloader:
            inputs, targets = inputs.to(self.device), targets.to(self.device)
            self.model.zero_grad()

            # Forward pass — use log-softmax for proper Fisher computation
            outputs = self.model(inputs)
            log_probs = F.log_softmax(outputs, dim=1)

            # Sample from the model's own distribution (empirical Fisher)
            # Alternatively, use true labels — both are common approximations
            labels = torch.argmax(outputs, dim=1)
            loss = F.nll_loss(log_probs, labels)
            loss.backward()

            for name, param in self.model.named_parameters():
                if param.requires_grad and param.grad is not None:
                    # Fisher = E[grad^2] (diagonal approximation)
                    fisher[name] += param.grad.detach() ** 2
                    n_samples += 1

        # Average over samples
        for name in fisher:
            fisher[name] /= len(dataloader)

        return fisher

    def penalty(self):
        """Compute EWC penalty: (λ/2) Σ F_i (θ_i - θ*_i)²"""
        loss = 0.0
        for name, param in self.model.named_parameters():
            if param.requires_grad and name in self.fisher:
                diff = param - self.prev_params[name]
                loss += (self.fisher[name] * diff ** 2).sum()
        return (self.importance / 2) * loss


# Training loop with EWC
def train_with_ewc(model, optimizer, dataloader, ewc=None, epochs=5, device='cpu'):
    model.train()
    for epoch in range(epochs):
        total_loss = 0
        for inputs, targets in dataloader:
            inputs, targets = inputs.to(device), targets.to(device)
            optimizer.zero_grad()

            outputs = model(inputs)
            ce_loss = F.cross_entropy(outputs, targets)

            # Add EWC penalty if we have a previous task
            if ewc is not None:
                ewc_loss = ewc.penalty()
                loss = ce_loss + ewc_loss
            else:
                loss = ce_loss

            loss.backward()
            optimizer.step()
            total_loss += loss.item()

        print(f"Epoch {epoch+1}: loss={total_loss/len(dataloader):.4f}")
Why Diagonal Fisher?

The full Fisher Information Matrix is of size N×N, where N is the number of parameters. For a ResNet-18 with 11 million parameters, that is a 11M × 11M matrix — impossible to store, let alone invert. The diagonal approximation keeps only the diagonal entries, assuming parameter importances are independent. It is a crude approximation, but it works surprisingly well in practice. Some variants (like K-FAC) use block-diagonal approximations for a middle ground.

Synaptic Intelligence (SI) and Memory Aware Synapses (MAS)

Synaptic Intelligence (Zenke et al., 2017) takes a different approach to computing importance. Instead of estimating it after training via the Fisher, SI accumulates importance during training. Every time a weight changes and contributes to a loss reduction, that contribution is recorded. Weights that contributed a lot to learning Task A are deemed important.

The importance ωᵢ for weight θᵢ is essentially the path integral of the gradient times the weight change over the training trajectory. This is cheaper to compute than the Fisher and captures the actual learning dynamics rather than a snapshot at the end.

Memory Aware Synapses (Aljundi et al., 2018) computes importance as the sensitivity of the learned function's output to weight perturbations — measured on unlabeled data. This is attractive because it does not need task labels, making it applicable to unsupervised continual learning scenarios. The importance for each weight is the average gradient magnitude of the model output with respect to that weight, evaluated over the data.

# Simplified MAS importance computation
def compute_mas_importance(model, dataloader, device):
    """Compute importance as sensitivity of output to weight changes."""
    importance = {
        name: torch.zeros_like(param)
        for name, param in model.named_parameters()
        if param.requires_grad
    }

    model.eval()
    for inputs, _ in dataloader:
        inputs = inputs.to(device)
        model.zero_grad()

        # Compute L2 norm of the output (no labels needed!)
        outputs = model(inputs)
        loss = torch.norm(outputs, p=2, dim=1).mean()
        loss.backward()

        for name, param in model.named_parameters():
            if param.requires_grad and param.grad is not None:
                importance[name] += param.grad.abs().detach()

    # Normalize
    for name in importance:
        importance[name] /= len(dataloader)

    return importance

All three methods — EWC, SI, and MAS — share the same skeleton: compute per-weight importance, then add a quadratic penalty when weights drift from their previous-task values. They differ in how they estimate importance. EWC uses the Fisher (a statistical measure). SI uses the actual training trajectory (a dynamic measure). MAS uses output sensitivity (a functional measure). In practice, their performance is often surprisingly similar, though MAS has the advantage of not needing labels.

Replay-Based Methods

Regularization fights forgetting by constraining weights. Replay fights forgetting the old-fashioned way: by keeping some old data around and training on it alongside new data. It is conceptually the closest thing to how humans avoid forgetting — we periodically revisit old material.

Experience Replay and Episodic Memory

Experience replay maintains a small buffer (episodic memory) of examples from previous tasks. When training on a new task, each mini-batch mixes new data with randomly sampled buffer data. The buffer is typically much smaller than the original dataset — maybe 200-500 examples per task — so the storage overhead is modest.

The key design decisions are: how big is the buffer, how do you select which examples to store (random? diverse? hard examples?), and how do you balance old vs new data in each batch?

import random
import torch
from torch.utils.data import DataLoader, TensorDataset

class ExperienceReplay:
    def __init__(self, buffer_size=500):
        self.buffer_size = buffer_size
        self.buffer_x = []
        self.buffer_y = []

    def add_task_data(self, dataloader, samples_per_task=None):
        """Store a subset of the current task's data in the buffer."""
        all_x, all_y = [], []
        for inputs, targets in dataloader:
            all_x.append(inputs)
            all_y.append(targets)
        all_x = torch.cat(all_x, dim=0)
        all_y = torch.cat(all_y, dim=0)

        # Reservoir sampling: keep buffer_size total across all tasks
        n = samples_per_task or (self.buffer_size // max(1, len(self.buffer_x) // 100 + 1))
        indices = random.sample(range(len(all_x)), min(n, len(all_x)))
        self.buffer_x.append(all_x[indices])
        self.buffer_y.append(all_y[indices])

        # If buffer exceeds limit, trim older entries proportionally
        self._trim_buffer()

    def _trim_buffer(self):
        """Ensure total buffer size doesn't exceed limit."""
        total = sum(len(x) for x in self.buffer_x)
        if total > self.buffer_size:
            ratio = self.buffer_size / total
            for i in range(len(self.buffer_x)):
                keep = max(1, int(len(self.buffer_x[i]) * ratio))
                self.buffer_x[i] = self.buffer_x[i][:keep]
                self.buffer_y[i] = self.buffer_y[i][:keep]

    def sample(self, batch_size):
        """Sample a batch from the replay buffer."""
        if not self.buffer_x:
            return None, None
        all_x = torch.cat(self.buffer_x, dim=0)
        all_y = torch.cat(self.buffer_y, dim=0)
        indices = random.sample(range(len(all_x)), min(batch_size, len(all_x)))
        return all_x[indices], all_y[indices]


def train_with_replay(model, optimizer, dataloader, replay_buffer, device, epochs=5):
    """Train on new task data mixed with replay buffer samples."""
    model.train()
    for epoch in range(epochs):
        for inputs, targets in dataloader:
            inputs, targets = inputs.to(device), targets.to(device)

            # Get replay samples
            replay_x, replay_y = replay_buffer.sample(batch_size=inputs.size(0) // 2)

            if replay_x is not None:
                replay_x = replay_x.to(device)
                replay_y = replay_y.to(device)
                # Mix current and replay data
                combined_x = torch.cat([inputs, replay_x], dim=0)
                combined_y = torch.cat([targets, replay_y], dim=0)
            else:
                combined_x, combined_y = inputs, targets

            optimizer.zero_grad()
            outputs = model(combined_x)
            loss = torch.nn.functional.cross_entropy(outputs, combined_y)
            loss.backward()
            optimizer.step()

    # After training, store some data from this task
    replay_buffer.add_task_data(dataloader)

Gradient Episodic Memory (GEM) and A-GEM

Gradient Episodic Memory (Lopez-Paz & Ranzato, 2017) is a more principled version of replay. Instead of mixing old and new data in the loss, GEM uses the buffer to constrain gradients. The rule is: the gradient for the new task must not increase the loss on any previous task. If it would, GEM projects the gradient onto the closest vector that satisfies all the constraints.

This is elegant but expensive — you need to compute gradients on the buffer for every previous task at every training step, then solve a quadratic program. With many tasks, this becomes a bottleneck.

Averaged GEM (A-GEM) simplifies this by using a single average gradient over the entire buffer instead of per-task gradients. If the current gradient conflicts with the average buffer gradient (their dot product is negative), A-GEM projects it. This reduces the per-step cost from O(T) gradient computations to O(1), with only a modest drop in performance.

def agem_gradient_update(model, new_loss, buffer_x, buffer_y, device):
    """A-GEM: project gradient if it conflicts with buffer gradient."""
    # Compute gradient on new task
    new_loss.backward(retain_graph=True)
    new_grad = torch.cat([
        p.grad.view(-1) for p in model.parameters() if p.grad is not None
    ])

    # Compute gradient on buffer (reference gradient)
    model.zero_grad()
    buf_out = model(buffer_x.to(device))
    buf_loss = torch.nn.functional.cross_entropy(buf_out, buffer_y.to(device))
    buf_loss.backward()
    ref_grad = torch.cat([
        p.grad.view(-1) for p in model.parameters() if p.grad is not None
    ])

    # Check for conflict: dot product < 0 means gradients oppose
    dot = torch.dot(new_grad, ref_grad)

    if dot < 0:
        # Project new gradient: g' = g - (g·g_ref / g_ref·g_ref) * g_ref
        proj = new_grad - (dot / torch.dot(ref_grad, ref_grad)) * ref_grad
    else:
        proj = new_grad  # No conflict, keep original gradient

    # Write projected gradient back to model parameters
    offset = 0
    for param in model.parameters():
        if param.grad is not None:
            numel = param.numel()
            param.grad.copy_(proj[offset:offset + numel].view_as(param))
            offset += numel

Generative Replay

Generative replay (Shin et al., 2017) is a clever twist: instead of storing real examples, train a generative model (like a GAN or VAE) alongside the classifier. When a new task arrives, the generator produces pseudo-examples from previous tasks. The classifier trains on a mix of real new data and generated old data.

The beauty is that you never store raw data from previous tasks — only the generator captures the old distribution. This is useful when storing real data is restricted (privacy, legal, storage). The downside: if the generator itself forgets (and it suffers from the same problem!), you get a cascade of degradation. The generator forgets the old distribution, produces bad pseudo-examples, and the classifier forgets too. Modern approaches tackle this with techniques like dual-memory architectures or by applying continual learning to the generator itself.

Architecture-Based Methods

Regularization and replay try to share the same weights across tasks. Architecture-based methods take a different philosophy: give each task its own parameters. The question is how to do this without the network growing out of control.

Progressive Neural Networks

Progressive Neural Networks (Rusu et al., 2016) are the most literal interpretation. For each new task, you add an entirely new column (a new network) alongside the previous ones. The previous columns are frozen — their weights never change again, so forgetting is impossible by construction. The new column receives lateral connections from all previous columns, allowing it to leverage previously learned features.

# Conceptual architecture of Progressive Neural Networks
#
#   Task 1 Column    Task 2 Column    Task 3 Column
#   ┌──────────┐    ┌──────────┐    ┌──────────┐
#   │  Layer 3  │    │  Layer 3  │◄───│  Layer 3  │
#   │ (frozen)  │    │ (frozen)  │    │           │
#   ├──────────┤    ├──────────┤    ├──────────┤
#   │  Layer 2  │───►│  Layer 2  │───►│  Layer 2  │
#   │ (frozen)  │    │ (frozen)  │    │           │
#   ├──────────┤    ├──────────┤    ├──────────┤
#   │  Layer 1  │───►│  Layer 1  │───►│  Layer 1  │
#   │ (frozen)  │    │ (frozen)  │    │           │
#   └──────────┘    └──────────┘    └──────────┘
#
# Arrows = lateral connections (knowledge transfer)
# Only the latest column is trainable

import torch
import torch.nn as nn

class ProgressiveColumn(nn.Module):
    def __init__(self, input_dim, hidden_dim, output_dim, prev_columns=None):
        super().__init__()
        self.layers = nn.ModuleList()
        self.lateral = nn.ModuleList()

        n_prev = len(prev_columns) if prev_columns else 0

        # Layer 1: takes input + lateral from prev columns' layer 1
        self.layers.append(nn.Linear(input_dim, hidden_dim))
        if n_prev > 0:
            self.lateral.append(nn.Linear(n_prev * hidden_dim, hidden_dim))
        else:
            self.lateral.append(None)

        # Layer 2: takes hidden + lateral from prev columns' layer 2
        self.layers.append(nn.Linear(hidden_dim, hidden_dim))
        if n_prev > 0:
            self.lateral.append(nn.Linear(n_prev * hidden_dim, hidden_dim))
        else:
            self.lateral.append(None)

        self.output = nn.Linear(hidden_dim, output_dim)
        self.prev_columns = prev_columns or []

    def forward(self, x, prev_activations=None):
        """
        prev_activations: list of lists — one list per previous column,
                          each containing that column's activations per layer
        """
        h = torch.relu(self.layers[0](x))
        if self.lateral[0] is not None and prev_activations:
            lat_input = torch.cat([pa[0] for pa in prev_activations], dim=1)
            h = h + torch.relu(self.lateral[0](lat_input))

        h = torch.relu(self.layers[1](h))
        if self.lateral[1] is not None and prev_activations:
            lat_input = torch.cat([pa[1] for pa in prev_activations], dim=1)
            h = h + torch.relu(self.lateral[1](lat_input))

        return self.output(h)

The guaranteed no-forgetting property is the biggest advantage. The biggest disadvantage: the network grows linearly with the number of tasks. After 100 tasks, you have 100 columns. This is impractical for long task sequences.

PackNet: Pruning for Continual Learning

PackNet (Mallya & Lazebnik, 2018) is a much more parameter-efficient approach. The idea: train a network on Task 1, then prune it aggressively (remove weights that are close to zero). The surviving weights are frozen for Task 1. Now train the pruned-away weights (the ones that were zeroed out) on Task 2, prune again, freeze, and repeat.

Each task gets its own subset of the network's weights. Since modern networks are massively over-parameterized, you can often pack 10-20 tasks into a single network. The network size stays fixed — no growth at all. At inference time, you apply a binary mask that selects the correct subset of weights for each task. The catch: you need to know which task you're evaluating (task-incremental setting), and eventually you run out of free weights.

Dynamic Expandable Networks

Dynamic Expandable Networks (DEN) (Yoon et al., 2018) strike a balance. The network starts at a normal size. For each new task, DEN first tries to learn it using the existing capacity (with selective retraining and group sparsity regularization). If the existing capacity is insufficient — measured by whether the loss plateaus above a threshold — DEN dynamically adds new neurons. But it adds only as many as needed, not an entire column. This gives you controlled growth that adapts to task complexity. A trivial new task adds zero neurons. A radically different task adds more.

Knowledge Distillation for Continual Learning

A different angle entirely: instead of protecting weights or replaying data, protect the model's behavior. If the old model mapped input X to output distribution Y, make sure the new model still maps X to something close to Y — even as it learns new things.

Learning without Forgetting (LwF)

Learning without Forgetting (Li & Hoiem, 2017) is the seminal method here. Before training on the new task, run the new task's data through the old model to get "soft targets" — probability distributions over old classes. Then train on the new task with a combined loss: cross-entropy on the new task labels plus a distillation loss that keeps the model's predictions on old classes close to the soft targets.

import torch
import torch.nn as nn
import torch.nn.functional as F
from copy import deepcopy

class LwF:
    def __init__(self, model, temperature=2.0, alpha=0.5):
        """
        model: the model BEFORE training on the new task
        temperature: softmax temperature for distillation (higher = softer)
        alpha: balance between old task distillation and new task CE
        """
        self.old_model = deepcopy(model)
        self.old_model.eval()
        for p in self.old_model.parameters():
            p.requires_grad = False
        self.temperature = temperature
        self.alpha = alpha

    def distillation_loss(self, new_outputs, inputs):
        """KL divergence between old and new model's predictions."""
        with torch.no_grad():
            old_outputs = self.old_model(inputs)

        # Soft targets from old model
        old_probs = F.softmax(old_outputs / self.temperature, dim=1)
        new_log_probs = F.log_softmax(new_outputs / self.temperature, dim=1)

        # KL divergence, scaled by T² (standard distillation scaling)
        loss = F.kl_div(new_log_probs, old_probs, reduction='batchmean')
        return loss * (self.temperature ** 2)

    def total_loss(self, new_outputs, targets, inputs):
        """Combined loss: new task CE + old task distillation."""
        ce_loss = F.cross_entropy(new_outputs, targets)
        dist_loss = self.distillation_loss(new_outputs, inputs)
        return (1 - self.alpha) * ce_loss + self.alpha * dist_loss

The clever part: LwF does not store any old data. It only needs the old model (frozen) and the new task's data. The old model acts as a "teacher" that keeps the new model honest about old tasks. The temperature parameter controls how much information flows through the soft targets — higher temperatures produce softer probability distributions that reveal more about the model's uncertainty structure.

Distillation-Based Regularization

LwF opened a whole family of methods. Some use distillation at intermediate feature layers (not just the output), forcing the new model's internal representations to stay close to the old model's. Others combine distillation with replay — store a small buffer of old data and apply distillation loss on it for stronger regularization. The combination of replay + distillation consistently ranks among the top-performing strategies in benchmarks.

Combining Strategies

In practice, the best continual learning systems combine approaches. For example: use a small replay buffer (500 examples), apply EWC regularization, AND add a distillation loss from the previous model. The three losses — task loss, EWC penalty, distillation loss — are weighted and summed. This defense-in-depth approach gives much better results than any single strategy alone. The Avalanche framework makes it straightforward to compose these strategies.

Benchmarks and Evaluation

You cannot improve what you cannot measure. The continual learning community has standardized on a set of benchmarks and a shared framework.

Standard Benchmarks

Split MNIST is the "hello world" of continual learning. Take MNIST (handwritten digits 0-9) and split it into 5 tasks: Task 1 classifies 0 vs 1, Task 2 classifies 2 vs 3, and so on. It is easy enough that most methods work well — useful for debugging, but not for serious evaluation.

Split CIFAR-10 and Split CIFAR-100 are harder. CIFAR-100 split into 10 tasks of 10 classes each is a standard benchmark. The images are more complex, the classes are more confusable, and 10 tasks is enough to stress-test forgetting.

Split Mini-ImageNet pushes further with higher-resolution images and more classes. And for the truly ambitious, there is CORe50 — a benchmark specifically designed for continual object recognition with domain shifts, occlusion, and real-world variability.

The Avalanche Framework

Avalanche is the standard open-source framework for continual learning research. Built on PyTorch, it provides ready-made benchmarks, strategies (EWC, replay, LwF, GEM, and many more), evaluation plugins, and logging — all composable.

import torch
from avalanche.benchmarks.classic import SplitCIFAR100
from avalanche.models import SimpleMLP
from avalanche.training.supervised import EWC, Naive, Replay
from avalanche.evaluation.metrics import (
    accuracy_metrics, forgetting_metrics, forward_transfer_metrics
)
from avalanche.training.plugins import EvaluationPlugin
from avalanche.logging import InteractiveLogger

# 1. Create benchmark: CIFAR-100 split into 10 tasks
benchmark = SplitCIFAR100(
    n_experiences=10,
    seed=42,
    return_task_id=True
)

# 2. Create model
model = SimpleMLP(num_classes=100, input_size=3 * 32 * 32, hidden_size=512)

# 3. Set up evaluation — track accuracy, forgetting, and forward transfer
eval_plugin = EvaluationPlugin(
    accuracy_metrics(epoch=True, experience=True, stream=True),
    forgetting_metrics(experience=True, stream=True),
    forward_transfer_metrics(experience=True, stream=True),
    loggers=[InteractiveLogger()]
)

# 4. Pick a strategy — let's compare EWC vs Naive (no continual learning)
optimizer = torch.optim.Adam(model.parameters(), lr=0.001)
strategy = EWC(
    model=model,
    optimizer=optimizer,
    criterion=torch.nn.CrossEntropyLoss(),
    ewc_lambda=1000,      # Regularization strength
    train_epochs=5,
    train_mb_size=128,
    evaluator=eval_plugin,
    device=torch.device('cuda' if torch.cuda.is_available() else 'cpu')
)

# 5. Train sequentially on each experience (task)
results = []
for experience in benchmark.train_stream:
    print(f"\n--- Training on Task {experience.current_experience} ---")
    strategy.train(experience)

    # Evaluate on ALL tasks seen so far
    print(f"--- Evaluating after Task {experience.current_experience} ---")
    res = strategy.eval(benchmark.test_stream[:experience.current_experience + 1])
    results.append(res)

With about 30 lines of code, you get a complete continual learning experiment with proper evaluation. Swapping EWC for Replay or Naive requires changing one line. Avalanche also supports custom strategies — you can subclass BaseStrategy and override the training loop while keeping the evaluation infrastructure.

Metrics Summary

Here is a quick reference for the four standard metrics:

# Metric definitions for a T-task continual learning experiment
# R[i][j] = accuracy on task j after training on all tasks up to i

# 1. Average Accuracy (higher is better)
#    How well does the final model perform across all tasks?
AA = (1/T) * Σⱼ R[T-1][j]

# 2. Forgetting Measure (lower is better, negative means forgetting)
#    How much did performance drop on old tasks?
FM = (1/(T-1)) * Σⱼ0 (R[i-1][i] - baseline[i])

A well-tuned EWC on Split CIFAR-100 (10 tasks) typically achieves an Average Accuracy around 55-65%, compared to 30-40% for naive fine-tuning. Replay with a buffer of 500 examples often pushes this to 60-70%. The joint training upper bound (training on all data simultaneously) sits around 75-80%. The gap between the best continual learning methods and joint training is shrinking, but it is not closed.

The Practical Takeaway

If you are building a production system that needs to learn incrementally, start with experience replay — it is the most robust and easiest to implement. Add EWC regularization on top if forgetting is still too high. Use distillation (LwF) if you cannot store old data. And if your system has very distinct tasks with clear boundaries, consider architecture-based methods. The field is moving fast — foundation models with prompt-tuning are emerging as a powerful continual learning paradigm where you freeze the backbone and only learn small task-specific prompts, avoiding weight interference entirely.
