The Game of Twenty Questions

Decision trees look simple—flowchart as machine learning model. But backbone of tabular winners: Random Forests, XGBoost, LightGBM. Need inside view. Here it is.

Decision trees became formal learning algorithm in 1984 with CART by Leo Breiman and team. Idea came earlier; Ross Quinlan later published ID3 in 1986 and C4.5 in 1993. Today trees power credit scoring, medical diagnosis, fraud detection, recommendation systems. Not strongest alone. Core unit of ensemble methods.

A Tiny Fruit Stand

Run fruit stand. Customers dump fruit into bin. You sort: apple, orange, lemon. Each fruit has three measurements—color on redness scale 0 to 10, diameter in centimeters, surface texture (smooth = 0, bumpy = 1). Here are six fruits:

Fruit   | Color | Diameter | Texture | Label
--------|-------|----------|---------|-------
  1     |   8   |   7.0    |    0    | Apple
  2     |   3   |   6.5    |    1    | Orange
  3     |   7   |   7.2    |    0    | Apple
  4     |   2   |   5.0    |    0    | Lemon
  5     |   4   |   6.8    |    1    | Orange
  6     |   1   |   4.5    |    0    | Lemon

Decision tree plays Twenty Questions. Ask one yes-or-no question on one feature. Split fruit into two piles. Ask next question on each pile. Each question narrows options. Each answer sends you down one branch.

For fruit, tree might ask: "Is color > 5?" High-redness fruits (1 and 3, both apples) go left. Rest go right. Then right pile might ask: "Is texture == 1?" Bumpy fruits (2 and 5, both oranges) go one way. Smooth low-color fruits (4 and 6, both lemons) go other way.

Is color > 5?
├── Yes → Apple  (fruits 1, 3)
└── No  → Is texture == 1?
          ├── Yes → Orange  (fruits 2, 5)
          └── No  → Lemon   (fruits 4, 6)

That is decision tree. Three questions. Fruit sorted perfectly. Each internal node asks one feature question. Each branch is yes-or-no answer. Each leaf is prediction. Napkin logic. Anyone can follow.

Key question: why ask color first, not diameter? Why threshold 5, not 4 or 6? Tree did not guess. Tried every possible question. Picked cleanest split. Need way to measure mess.

Measuring Mess: Gini Impurity

Before any question, all six fruits sit in one pile: 2 apples, 2 oranges, 2 lemons. Pile fully mixed. Random grab, random label guess from proportions, many misses. Need number for that mess.

Gini impurity measures exactly that. Probability random sample gets misclassified if labeled randomly by class proportions at that node. Formula:

Gini = 1 - Σ(pᵢ²)

where pᵢ is class i proportion. For starting pile, three classes, each 2/6 = 1/3.

Gini = 1 - ((1/3)² + (1/3)² + (1/3)²)
     = 1 - (1/9 + 1/9 + 1/9)
     = 1 - 3/9
     = 1 - 0.333
     = 0.667

Near maximum mess. Now split on "color > 5." Left child gets fruits 1 and 3—both apples. Its Gini:

Gini_left = 1 - (1.0²) = 0.0

Zero. Pure node. One class only. Right child gets fruits 2, 4, 5, 6—two oranges, two lemons.

Gini_right = 1 - ((2/4)² + (2/4)²)
           = 1 - (0.25 + 0.25)
           = 0.5

To score split overall, take weighted average of child impurity by sample counts. Left gets 2 of 6. Right gets 4 of 6.

Gini_split = (2/6) × 0.0 + (4/6) × 0.5
           = 0 + 0.333
           = 0.333

Start at 0.667. Drop to 0.333. Gini reduction = 0.334. Tree tries every split on every feature, computes this drop, picks biggest one. Goal: cleanest child nodes. Fruit piles close to single class.

Gini 0 means pure node. Gini 0.5 for binary, or higher for multi-class, means maximum confusion. Tree hunts zeros.

The Other Yardstick: Entropy and Information Gain

Key point: Gini and entropy usually behave same. Entropy still matters because it comes from information theory and shows same idea from another angle.

Entropy measures uncertainty. If you might pick any fruit from pile, entropy high. If pile is all apples, outcome known—entropy zero. Formula:

Entropy = -Σ(pᵢ × log₂(pᵢ))

For our starting pile of 2 apples, 2 oranges, 2 lemons:

Entropy = -(1/3 × log₂(1/3)) × 3
        = -(1/3 × (-1.585)) × 3
        = 1.585

Pure node has entropy 0. Binary 50/50 split has entropy 1.0. Main difference from Gini: scale. Entropy can exceed 1.0 in multi-class problems. Shape stays similar. Both peak when classes are evenly distributed and hit zero when node is pure.

Information gain = parent entropy minus weighted average of child entropies. It measures uncertainty removed by split. Biggest reduction wins.

Key point: in practice, Gini and entropy produce almost same trees. Research shows chosen split differs in under 2% of cases. Gini is slightly faster because no logarithm. Scikit-learn defaults to Gini. Good default. In interviews, honest answer: choice rarely matters.

Historical wrinkle: Quinlan's ID3 (1986) used information gain, but information gain favors features with many distinct values. Feature like "customer_id" can get huge information gain by creating tiny pure nodes. Quinlan fixed this in C4.5 (1993) with gain ratio—information gain divided by feature's own entropy (split information), penalizing features that scatter data into many small groups. Breiman's CART used Gini impurity instead and avoided much of problem by always making binary splits.

The CART Algorithm — Greedy and Proud of It

With mess metric ready, algorithm is straightforward. CART—Classification and Regression Trees—came from Breiman, Friedman, Olshen, and Stone in 1984. It still underlies scikit-learn's DecisionTreeClassifier.

At each node, CART checks each feature. For continuous feature, it sorts values and tests every midpoint between adjacent values as candidate threshold. For fruit color values [1, 2, 3, 4, 7, 8], thresholds are 1.5, 2.5, 3.5, 5.5, 7.5. For each candidate, it splits data left and right, computes weighted Gini impurity of children, records reduction from parent. After trying every threshold on every feature, it picks feature-threshold pair with biggest impurity reduction. Then it repeats on each child until node is pure or some limit stops growth.

Trace one step. Root has all six fruits. CART tests every possible split:

Feature: Color
  threshold 1.5 → left: {Lemon}  right: {Apple,Orange,Apple,Lemon,Orange}
  threshold 2.5 → left: {Lemon,Lemon}  right: {Apple,Orange,Apple,Orange}
  threshold 3.5 → left: {Orange,Lemon,Lemon}  right: {Apple,Apple,Orange}
  threshold 5.5 → left: {Orange,Lemon,Orange,Lemon}  right: {Apple,Apple}
  ...
Feature: Diameter
  (similarly, tries every midpoint)
Feature: Texture
  threshold 0.5 → left: {Apple,Apple,Lemon,Lemon}  right: {Orange,Orange}

For each candidate, CART computes Gini reduction. Split "color > 5.5" makes one pure apple child (Gini = 0) and one child with two oranges and two lemons (Gini = 0.5). Weighted reduction is 0.334, best among candidates. CART picks it.

Process is greedy. At each node, CART picks locally best split. Future splits do not matter. Split that looks weak now might enable great split two levels later, but CART never sees that. Finding globally optimal tree is NP-hard (Hyafil and Rivest, 1976). So greedy wins.

Optimal tree is NP-hard. So every trained decision tree is approximation. Good approximation. Still approximation.

Greedy search has big consequence: first split casts long shadow. If one extra sample changes root from "color > 5.5" to "diameter > 6.0," every later split changes too. Whole structure cascades from that first choice. That instability matters later. It becomes core tree property.

Pruning: The Art of Knowing When to Stop

Without checks, tree keeps splitting until every leaf holds one class—or one sample. On tiny fruit set, that gives perfect accuracy. On real data with ten thousand samples and hundred features, that gives nightmare tree with thousands of leaves, memorizing quirks and noise. Training accuracy: 100%. Test accuracy: garbage.

Think gardener pruning rose bush. Wild branches waste energy, produce no flowers. Pruning cuts excess, keeps signal. Tree pruning does same. Cut branches that learn noise, not pattern.

Two strategies. Two philosophies.

Pre-pruning: stop wild growth early. Set limits before training. Main controls: max_depth, min_samples_split, min_samples_leaf. Most important is max_depth. Depth 5 means at most 2⁵ = 32 leaves. Depth 20 means over million. Difference: interpretable model vs lookup table.

Risk of pre-pruning: stop too early. Split that looks useless at depth 3 might unlock strong split at depth 4. Hard cutoff hides it.

Post-pruning does opposite. Grow full tree first, then cut back. Common version is cost-complexity pruning, exposed in scikit-learn by ccp_alpha. Idea:

R_α(T) = R(T) + α × |T|

where R(T) is tree misclassification rate, |T| is number of leaves, and α is complexity penalty. Larger α means stronger penalty, more pruning, simpler tree. α = 0 means no penalty and full unpruned tree. As α grows, subtrees collapse into leaves when extra accuracy is not worth complexity.

Elegant part: α is not guessed. Scikit-learn's cost_complexity_pruning_path computes α values where tree structure changes. Each α gives different tree size. Cross-validation picks best validation performer.

from sklearn.tree import DecisionTreeClassifier
from sklearn.model_selection import cross_val_score
from sklearn.datasets import load_breast_cancer
import numpy as np

X, y = load_breast_cancer(return_X_y=True)

# Grow full tree to compute the pruning path
full_tree = DecisionTreeClassifier(random_state=42)
path = full_tree.cost_complexity_pruning_path(X, y)
alphas = path.ccp_alphas[:-1]  # last alpha gives a single-node tree

# Cross-validate each candidate alpha
cv_scores = [
    cross_val_score(
        DecisionTreeClassifier(ccp_alpha=a, random_state=42),
        X, y, cv=5, scoring='accuracy'
    ).mean()
    for a in alphas
]

best_alpha = alphas[np.argmax(cv_scores)]
print(f"Best alpha: {best_alpha:.4f}, CV accuracy: {max(cv_scores):.4f}")

# Train the final pruned tree
tree = DecisionTreeClassifier(ccp_alpha=best_alpha, random_state=42)
tree.fit(X, y)
print(f"Depth: {tree.get_depth()}, Leaves: {tree.get_n_leaves()}")

This usually gives tree 4–8 levels deep instead of 20+. Pruning path gives menu. Cross-validation picks item. Post-pruning sees whole tree before cutting. Often better than pre-pruning alone.

Common practice: set reasonable max_depth (say 10–15) to keep training fast, then use ccp_alpha to tune final size.

Feature Importance — and Its Trap

Each split records impurity reduction. After training, sum total reduction from each feature, normalize to 1.0, rank features by contribution. This is impurity-based feature importance. Comes free with trained tree.

from sklearn.datasets import load_breast_cancer

data = load_breast_cancer()
feature_names = data.feature_names
X_train, y_train = data.data, data.target

tree = DecisionTreeClassifier(max_depth=5, random_state=42)
tree.fit(X_train, y_train)

for name, importance in sorted(
    zip(feature_names, tree.feature_importances_),
    key=lambda x: -x[1]
)[:5]:
    print(f"{name:30s} {importance:.4f}")

Useful for quick sense of model drivers. But trap waiting.

Impurity-based importance is biased toward features with many unique values. Reason: more unique values mean more candidate split points. More candidates mean more chances for one lucky large impurity drop. Random column with unique values per row—like customer ID—can look "important" because it creates pure single-sample nodes.

Fix: permutation importance. After training, shuffle one feature across samples, breaking its link to target. Measure accuracy drop. Big drop means feature truly mattered. Tiny drop means it did not, no matter how many split points it had.

from sklearn.inspection import permutation_importance
from sklearn.model_selection import train_test_split

X_train, X_test, y_train, y_test = train_test_split(
    data.data, data.target, test_size=0.2, random_state=42)

tree = DecisionTreeClassifier(max_depth=5, random_state=42)
tree.fit(X_train, y_train)

result = permutation_importance(tree, X_test, y_test,
                                n_repeats=10, random_state=42)
for name, importance in sorted(
    zip(feature_names, result.importances_mean),
    key=lambda x: -x[1]
)[:5]:
    print(f"{name:30s} {importance:.4f}")

Permutation importance measures real predictive power, not accident of many candidate splits. Best answer to "which features matter?" especially in production, where rankings guide decisions.

Deeper option: SHAP values (SHapley Additive exPlanations). They come from game theory and give per-sample, per-feature attribution. For tree models, TreeSHAP computes them in polynomial time, so large datasets stay practical. But permutation importance is strong default for quick, reliable rankings.

The Variance Problem: Why One Tree Is Never Enough

Most important tree fact. Ensembles, boosting, Kaggle pipelines—all trace back here.

Decision tree is high-variance, low-bias learner. It can carve very complex decision boundaries (low bias), but reacts strongly to exact training data (high variance). Remove five samples, add five new ones, and you may get completely different tree—different root feature, thresholds, structure, predictions.

Small data change can produce tree that looks nothing like original. Not slightly different. Unrecognizably different.

Reason goes back to greedy algorithm. Root split casts long shadow. If two features are close at root—say Gini reduction 0.334 vs 0.331—a few samples can flip winner. Once root changes, child nodes hold different subsets. Next splits use different data. Cascade continues. By leaves, trees may share almost nothing.

Twenty Questions again: change first question from "Is it a mammal?" to "Does it live in water?" Every follow-up changes. Whole game takes different path.

Why this matters: model that changes sharply under small data perturbations is not reliable alone. Two trees trained on different random samples from same population may agree on broad trends but disagree hard on specific predictions. That disagreement is variance. Variance means unreliable predictions.

Insight that changed tree ML: if single trees have low bias and high variance, average many trees. Averaging independent estimates reduces variance without raising bias. That became Random Forests (Breiman, 2001): train many trees on bootstrap samples, use random feature subsets at each split, average predictions. Individual trees are wild. Average is stable and strong.

Instead of averaging independent trees, build each new tree to fix previous mistakes. That is Gradient Boosting (Friedman, 1999). It powers XGBoost, LightGBM, and CatBoost—the models that dominate structured data in industry and competitions.

Single-tree instability is not defect. Raw material for ensembles. Without variance, nothing to average out. Without diversity, nothing to boost. Weak single tree helps make strong forest.

What Trees Can't Do

Decision trees have real limits. Know these better than strength lists.

Trees cannot extrapolate. Fundamental limit. If training house prices range from $100k to $500k, regression tree will never predict $600k. Each leaf predicts mean of training samples in that leaf, and mean of values between 100k and 500k cannot exceed 500k. Linear model can extend beyond training range. So standalone tree fails on time series or any domain where future values exceed history. Same limit hits Random Forests too.

Trees draw staircase boundaries. Each split is perpendicular to one feature axis—"Is color > 5?" gives vertical line in feature space. Tree cannot draw diagonal boundary directly. If true separation runs at 45 degrees through color-diameter space, tree approximates it with staircase: horizontal and vertical cuts zigzagging along real boundary. More depth gives finer staircase, but more overfitting. This is axis-aligned split limitation. Oblique decision trees split on linear combinations of features (like "0.3 × color + 0.7 × diameter > 4") and can draw diagonals with fewer nodes, but they are harder to train, harder to interpret, and rare in practice. Ensembles of axis-aligned trees usually work better.

Categorical features require encoding in scikit-learn. DecisionTreeClassifier in sklearn cannot handle string categories natively. You must one-hot encode or ordinal encode first. One-hot encoding with 50 categories makes 50 binary columns, fragmenting split decisions and diluting importance. Ordinal encoding adds arbitrary order that tree treats as meaningful. LightGBM and CatBoost handle categorical features natively by finding optimal category groupings at each split. Real advantage.

Missing values need workarounds. Sklearn trees cannot handle NaN values. You must impute before training. Original CART had clever fix: surrogate splits. For each split, CART finds other feature splits that mimic primary split closely. If sample is missing primary split feature, tree uses best available surrogate to route it. XGBoost and LightGBM do something else: learn default direction for missing values at each split, trying left and right during training and keeping lower-loss option. Elegant. Often better than manual imputation.

No feature scaling needed. Practical advantage. Trees care about ordering, not magnitude. Income in dollars or millions gives same optimal threshold. No StandardScaler. No MinMaxScaler. Raw features fine.

Regression Trees: Same Idea, Different Score

So far, examples used classification. CART does regression too. Logic stays same—recursive greedy splitting. Score changes. Instead of Gini impurity or entropy, regression tree minimizes mean squared error (MSE) at each split. It chooses split that maximizes variance reduction in target.

Each leaf predicts mean of training samples that land there. That is why regression trees cannot extrapolate. Prediction is always mean of training values, never beyond max or min being averaged.

Regression trees are building blocks of gradient boosted regression, arguably strongest method for tabular regression. Trees inside boosting are usually shallow—depth 3 to 6—and each fits residual errors of current ensemble. More on that in ensemble section.

The History We Inherited

Worth knowing where these algorithms came from. Design choices still echo in every tree trained today.

In 1984, Leo Breiman and three co-authors published CART. CART introduced binary splits, Gini impurity, cost-complexity pruning, and surrogate splits for missing values. It handled classification and regression. First comprehensive, principled tree-building algorithm.

Two years later, Ross Quinlan published ID3. Simpler algorithm. Used information gain (entropy) and multi-way splits for categorical features. Easier to implement, but weaker: no continuous features, no missing values, and split bias toward high-cardinality features.

Quinlan fixed these in C4.5 (1993). It added gain ratio to debias splitting, support for continuous features via thresholds, missing-value handling through probability-based routing, and post-pruning. C4.5 became one of most cited ML algorithms.

Today, scikit-learn follows CART approach: binary splits, Gini or entropy, cost-complexity pruning. Gradient boosting libraries—XGBoost, LightGBM, CatBoost—still build on CART foundation with their own innovations. Tree trained in 2024 is direct descendant of Breiman's 1984 book.
