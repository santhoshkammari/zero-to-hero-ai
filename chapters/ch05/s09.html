Advanced Supervised Learning Topics

Chapter 5: Supervised Learning 13 subtopics ~20 min read
TL;DR

This section surveys the landscape beyond basic classification and regression. Real-world supervised learning involves messy labels, shifting distributions, imbalanced classes, and models that need to know when they don't know. You'll learn when to use multi-label classification, how online learning differs from batch, why active learning saves labeling budgets, and how to handle class imbalance with techniques like SMOTE and cost-sensitive learning. Each topic here is a door — we'll give you enough to walk through it and know when you need to.

Multi-Label Classification

In standard classification, each example belongs to exactly one class. But many real problems aren't like that. A movie can be both "comedy" and "drama." A medical patient can have multiple diagnoses simultaneously. Multi-label classification assigns a set of labels to each example.

Three main approaches exist. Binary relevance trains one independent binary classifier per label — simple but ignores label correlations (if a movie is "action," it's more likely also "thriller"). Classifier chains order the labels and feed each classifier the predictions of previous ones, capturing dependencies at the cost of being sensitive to label ordering. Label powerset treats each unique label combination as a distinct class — captures all correlations but explodes combinatorially when you have many labels.

Metrics change too. Accuracy per label is insufficient. You need Hamming loss (fraction of label-example pairs that are wrong), subset accuracy (fraction of examples where the entire label set is correct — very strict), and micro/macro-averaged F1 scores across all labels. In practice, binary relevance with a strong base classifier is a solid starting point. If label correlations matter and you have moderate numbers of labels, classifier chains are worth trying.

Online Learning vs Batch Learning

Batch learning is what you usually do: collect all training data, train the model, deploy it. If new data arrives, retrain from scratch. Online learning updates the model incrementally as each new example (or mini-batch) arrives. The model is always adapting.

Why does this matter? If your data distribution changes over time — a recommendation system where user preferences evolve, a fraud detector where attack patterns shift — batch retraining introduces latency. By the time you retrain, the world has moved. Online learning algorithms like stochastic gradient descent with a decaying learning rate, Passive-Aggressive classifiers, and online variants of SVMs and logistic regression update continuously.

The tradeoff: online learning is more responsive but can be unstable. A burst of anomalous data can push the model off course. You often want a hybrid: online updates with periodic batch retraining as a safety net. Scikit-learn's partial_fit() method supports incremental learning for several estimators including SGDClassifier, MultinomialNB, and MiniBatchKMeans.

Active Learning

Labeling data is expensive. A radiologist's time is limited. Annotating legal documents requires domain expertise. Active learning asks: instead of labeling random examples, what if the model could choose which examples to label next? The idea is to label the examples that would be most informative for improving the model, reducing the total labeling cost.

The most common strategy is uncertainty sampling: pick the examples where the current model is least confident. For a binary classifier, that means examples with predicted probabilities near 0.5. Query-by-committee trains multiple models and picks examples where they disagree most. Expected model change selects examples that would cause the largest gradient update.

Active learning works best when the labeling budget is small relative to the pool of unlabeled data. In practice, it can achieve the same performance as random sampling with 2–10× fewer labels. The risk: uncertainty sampling can get stuck in regions of the feature space near the decision boundary, ignoring large regions it's already confident about (possibly wrongly). Diversity-based sampling — selecting examples that are both uncertain and spread across the feature space — helps mitigate this.

Label Noise Handling

Real datasets have labeling errors. Crowdsourced annotations might have 5–15% noise. Even expert labels have disagreements. Training on noisy labels naively causes the model to memorize the noise, hurting generalization.

Noise-robust loss functions are one defense. The standard cross-entropy loss heavily penalizes confident wrong predictions, which amplifies the effect of mislabeled examples. Alternatives like the symmetric cross-entropy, generalized cross-entropy, and mean absolute error loss are more tolerant of label noise — they don't explode when the model predicts correctly but the label is wrong.

Confident Learning (implemented in the cleanlab library) takes a different approach: identify and remove (or relabel) noisy examples. The idea: train a model, collect its predicted probabilities for each example via cross-validation, then flag examples where the model's prediction strongly disagrees with the given label. These are likely mislabeled. The approach is model-agnostic and works surprisingly well — often removing 1–5% of training examples as noise improves test accuracy.

Discriminative vs Generative Models

This is a fundamental distinction in machine learning. Discriminative models learn the boundary between classes directly — they model P(Y|X), the probability of the label given the features. Logistic regression, SVMs, neural networks, random forests — all discriminative. Generative models learn the full joint distribution P(X, Y) = P(X|Y) · P(Y) — they model how the data was generated for each class, then use Bayes' rule for classification. Naive Bayes, Gaussian Discriminant Analysis, and Hidden Markov Models are generative.

Discriminative models generally achieve better classification accuracy because they focus directly on the decision boundary rather than modeling the entire data distribution. Generative models have advantages when you have very little labeled data (they can incorporate prior knowledge about data structure), when you want to generate synthetic examples, or when you need to handle missing features gracefully. Andrew Ng and Michael Jordan showed empirically that discriminative models asymptotically outperform generative ones, but generative models can win with small sample sizes.

Adversarial Robustness

Neural networks are brittle in a specific and alarming way. Adding tiny, carefully computed perturbations to an input — imperceptible to humans — can completely flip a model's prediction. A stop sign with a few modified pixels gets classified as a speed limit sign. A loan application with negligibly different income figures flips from "approved" to "denied." These are adversarial examples.

Adversarial training is the most effective defense. During training, you generate adversarial examples on the fly and include them in each mini-batch. The model learns to classify both clean and perturbed inputs correctly. Projected Gradient Descent (PGD) is the standard attack used during adversarial training — it iteratively perturbs the input in the direction that maximizes the loss, projecting back onto an ε-ball around the original input at each step.

The honest tradeoff: adversarial robustness almost always costs you some accuracy on clean (non-perturbed) data. There's an inherent tension between standard accuracy and robust accuracy. How much you care depends on your deployment context. Safety-critical systems (autonomous driving, medical diagnosis) should pay the accuracy cost. A movie recommender probably shouldn't bother.

Covariate Shift Detection and Correction

Your model was trained on data from one distribution, but at deployment the input distribution has shifted. This is covariate shift: P(X) changes but P(Y|X) stays the same. Example: you train a medical model on data from one hospital, deploy it at another hospital with a different patient demographic mix. The relationship between symptoms and diagnosis hasn't changed, but the mix of patients has.

Detection is straightforward: train a classifier to distinguish training data from deployment data. If it can, the distributions differ. The standard correction is importance weighting: weight each training example by the ratio P_test(x) / P_train(x). Examples that are rare in training but common in deployment get upweighted. In practice, you estimate these density ratios using methods like Kernel Mean Matching (KMM) or by training the train-vs-deploy classifier and converting its predicted probabilities to weights. The risk: if the shift is too extreme (some deployment inputs are completely unlike anything in training), no reweighting can save you — you need more training data from the new distribution.

Transfer Learning for Tabular Data

Transfer learning revolutionized computer vision and NLP — pretrain on a huge dataset, fine-tune on your specific task. For tabular data, it's much harder and less established. Tables don't have the universal spatial or sequential structure that makes ImageNet features or language model embeddings transferable.

That said, there are emerging approaches. Pretraining on related tabular datasets and fine-tuning can help when your target dataset is small and a larger related dataset exists (e.g., pretrain on all hospital records, fine-tune on a specific rare disease). Self-supervised pretraining methods like VIME (Value Imputation and Mask Estimation) and SCARF corrupt tabular features and train the model to reconstruct them, learning useful representations. Foundation models for tabular data — like TabPFN, which uses in-context learning to do classification without gradient-based training — are a fascinating recent development. The honest assessment: for most tabular tasks, a well-tuned gradient-boosted tree still wins. Transfer learning for tables is promising but not yet a default choice.

Classifier Calibration

When your model says "80% probability of class A," is it actually correct 80% of the time? If so, the model is calibrated. Many models aren't. Random forests tend to push probabilities toward 0 and 1. SVMs with Platt scaling can be overconfident. Neural networks trained with cross-entropy are often well-calibrated initially but become overconfident with modern training tricks like data augmentation and label smoothing.

Calibration matters when you use predicted probabilities for decisions — medical risk scores, insurance pricing, or any application where the magnitude of the probability, not just the ranking, matters. Platt scaling fits a logistic regression on the model's output scores using a held-out calibration set. Isotonic regression fits a non-parametric monotonic function — more flexible but needs more calibration data. Temperature scaling (common for neural networks) divides the logits by a learned temperature T before softmax — T > 1 softens the distribution, reducing overconfidence.

Diagnose calibration with a reliability diagram: bin predictions by their predicted probability, plot the actual fraction of positives in each bin. A perfectly calibrated model follows the diagonal. The Expected Calibration Error (ECE) summarizes this into a single number — the weighted average gap between predicted and actual probability across bins.

Bayesian vs Frequentist Approach

This isn't just philosophy — it changes what your model gives you. In the frequentist approach, parameters are fixed unknown quantities. You get point estimates (the single "best" set of weights), confidence intervals, and p-values. This is what scikit-learn, statsmodels, and most deep learning frameworks do. In the Bayesian approach, parameters are random variables with probability distributions. You start with a prior (what you believe before seeing data), update with data using Bayes' rule, and get a posterior distribution over parameters.

The Bayesian approach naturally gives you uncertainty estimates: instead of a single prediction, you get a distribution of predictions. This is valuable when you need to know not just "what does the model predict?" but "how confident is the model?" — critical in medical diagnosis, autonomous driving, and any high-stakes application. The cost: Bayesian inference is computationally expensive. Exact posterior computation is intractable for most models. You use approximations — Markov Chain Monte Carlo (MCMC) for accurate but slow inference, or Variational Inference for faster but approximate posteriors. Libraries like PyMC, Stan, and NumPyro make Bayesian modeling accessible. For deep learning, MC Dropout (running inference multiple times with dropout on) provides a cheap approximation to Bayesian uncertainty.

Multi-Modal Data Fusion

Real data often comes in multiple forms simultaneously. A medical record might include tabular data (lab values, demographics), text (doctor's notes), and images (X-rays). A product listing has tabular features (price, category), text (description), and images (product photos). Multi-modal fusion combines these different data types into a single predictive model.

Three fusion strategies exist. Early fusion concatenates all features into a single vector and feeds them to one model. Simple, but tabular features and image embeddings live on very different scales and may need different processing. Late fusion trains separate models for each modality and combines their predictions (by averaging, voting, or stacking). This respects each modality's structure but can't capture cross-modal interactions. Intermediate fusion processes each modality through separate encoders, merges the learned representations at some intermediate layer, and continues with shared layers. This is the most powerful but hardest to train — you need enough data for the cross-modal layers to learn useful interactions. In practice, late fusion is the safest starting point; intermediate fusion with attention mechanisms gives the best results when you have sufficient data.

Few-Shot and Semi-Supervised Learning

What if you have very few labeled examples? Few-shot learning (typically 1–10 examples per class) uses meta-learning: the model is trained on many different small tasks so it learns to learn quickly from limited data. Prototypical networks compute a class prototype (mean embedding of support examples) and classify new examples by distance to prototypes. MAML (Model-Agnostic Meta-Learning) finds an initialization point from which a few gradient steps on a new task produce a good model.

Semi-supervised learning occupies the middle ground: you have a small labeled set and a large unlabeled set. Self-training (pseudo-labeling) uses the current model to label unlabeled examples, adds confident predictions to the training set, and retrains. MixMatch and FixMatch combine consistency regularization (the model should give similar predictions for augmented versions of the same input) with pseudo-labeling. These methods can dramatically reduce the labeled data requirement — sometimes achieving full-data performance with only 10% of labels. The risk: if early pseudo-labels are wrong, errors can compound. Confidence thresholding and strong augmentation help, but semi-supervised learning works best when the unlabeled data is from the same distribution as the labeled data.

Class Imbalance Handling

This is one of the most common real-world problems. Fraud detection: 99.9% of transactions are legitimate. Disease screening: 1% prevalence. Defect detection in manufacturing: defective items are rare. When one class vastly outnumbers the other, standard classifiers learn to predict the majority class and still achieve high accuracy — 99.9% accuracy by predicting "not fraud" every time. Useless.

SMOTE and Variants

SMOTE (Synthetic Minority Over-sampling Technique) generates new synthetic examples of the minority class. For each minority example, it finds its k nearest minority neighbors, picks one randomly, and creates a new example somewhere on the line between the two in feature space. This is more informative than simple duplication because each synthetic example is slightly different.

ADASYN (Adaptive Synthetic Sampling) is a smarter version: it generates more synthetic examples in regions where the minority class is harder to learn (near the decision boundary) and fewer in regions where it's already easy. Borderline-SMOTE only generates synthetic examples from minority instances that are near the class boundary, focusing effort where it matters most.

Undersampling and Cleaning

Instead of creating more minority examples, you can remove majority examples. Random undersampling is the simplest approach but throws away potentially useful data. Tomek Links are more surgical: a Tomek link is a pair of nearest-neighbor examples from different classes. Removing the majority-class member of each Tomek link cleans the decision boundary without discarding examples deep in majority territory. Edited Nearest Neighbors (ENN) removes any example whose class label disagrees with the majority of its k nearest neighbors — effectively removing noisy examples and tightening class boundaries.

Cost-Sensitive Learning

Instead of resampling data, you can modify the learning algorithm to penalize mistakes on the minority class more heavily. In scikit-learn, most classifiers accept a class_weight parameter. Setting class_weight='balanced' automatically weights classes inversely proportional to their frequency. This is often the simplest and most effective approach — no data manipulation needed, and it works with any classifier that supports it.

Threshold Tuning

By default, a classifier predicts the positive class when its probability exceeds 0.5. But with imbalanced data, 0.5 is rarely the optimal threshold. Threshold tuning adjusts this decision boundary. You can optimize for a specific metric — F1 score, precision at a fixed recall, or a custom business metric. Plot the precision-recall curve, pick the threshold that gives you the tradeoff your application needs. A fraud detector might use a low threshold (catch most fraud, accept some false alarms). A spam filter might use a high threshold (only flag messages you're very sure about, because false positives are annoying).

Putting It Together: Code

Here's a practical example using imbalanced-learn to handle severe class imbalance: 

import numpy as np
from sklearn.datasets import make_classification
from sklearn.model_selection import cross_val_score, StratifiedKFold
from sklearn.ensemble import RandomForestClassifier
from sklearn.metrics import f1_score, make_scorer
from imblearn.over_sampling import SMOTE, ADASYN
from imblearn.under_sampling import TomekLinks
from imblearn.combine import SMOTETomek
from imblearn.pipeline import Pipeline as ImbPipeline

# Create imbalanced dataset: 5% minority class
X, y = make_classification(
    n_samples=2000, n_features=20,
    n_informative=10, n_redundant=5,
    weights=[0.95, 0.05], random_state=42
)
print(f"Class distribution: {np.bincount(y)}")
# → Class distribution: [1900  100]

cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)
f1 = make_scorer(f1_score)

# Baseline: no resampling
baseline = RandomForestClassifier(n_estimators=100, random_state=42)
scores = cross_val_score(baseline, X, y, cv=cv, scoring=f1)
print(f"Baseline F1:      {scores.mean():.3f} ± {scores.std():.3f}")

# Strategy 1: SMOTE oversampling
smote_pipe = ImbPipeline([
    ('smote', SMOTE(random_state=42)),
    ('clf', RandomForestClassifier(n_estimators=100, random_state=42))
])
scores = cross_val_score(smote_pipe, X, y, cv=cv, scoring=f1)
print(f"SMOTE F1:         {scores.mean():.3f} ± {scores.std():.3f}")

# Strategy 2: ADASYN (adaptive oversampling)
adasyn_pipe = ImbPipeline([
    ('adasyn', ADASYN(random_state=42)),
    ('clf', RandomForestClassifier(n_estimators=100, random_state=42))
])
scores = cross_val_score(adasyn_pipe, X, y, cv=cv, scoring=f1)
print(f"ADASYN F1:        {scores.mean():.3f} ± {scores.std():.3f}")

# Strategy 3: SMOTE + Tomek Links (oversample then clean)
smote_tomek_pipe = ImbPipeline([
    ('smote_tomek', SMOTETomek(random_state=42)),
    ('clf', RandomForestClassifier(n_estimators=100, random_state=42))
])
scores = cross_val_score(smote_tomek_pipe, X, y, cv=cv, scoring=f1)
print(f"SMOTE+Tomek F1:   {scores.mean():.3f} ± {scores.std():.3f}")

# Strategy 4: Cost-sensitive learning (no resampling needed)
weighted_clf = RandomForestClassifier(
    n_estimators=100, class_weight='balanced', random_state=42
)
scores = cross_val_score(weighted_clf, X, y, cv=cv, scoring=f1)
print(f"Cost-sensitive F1: {scores.mean():.3f} ± {scores.std():.3f}")
💡 Practical Advice

Always apply resampling inside the cross-validation loop (using imblearn.pipeline.Pipeline), never before splitting. Otherwise you leak information from validation folds into training through the synthetic examples, and your scores will be overly optimistic. Also: start with class_weight='balanced' — it's often competitive with SMOTE and involves zero data manipulation.

Combined Strategies

The best real-world approach often combines several techniques. A typical effective recipe: (1) Use SMOTE or ADASYN to bring the minority class up to 20–40% of the majority (not necessarily 50/50). (2) Apply Tomek Links to clean the boundary. (3) Use class_weight='balanced' in the classifier anyway — belt and suspenders. (4) Tune the decision threshold on a validation set using the precision-recall curve. (5) Evaluate with the right metrics: F1, precision-recall AUC, or a custom business metric — never accuracy alone. There's no single best approach for all problems, but this workflow gives you reliable starting points and the knobs to tune.
