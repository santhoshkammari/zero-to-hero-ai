Support Vector Machines

Support Vector Machines look intimidating — "Lagrangian dual" and "Mercer's theorem" do that. Core idea: geometric, intuitive. Concepts like margin maximization, kernel trick, and structural risk minimization show up across modern ML. Dive starts here.

Support Vector Machines came from Vladimir Vapnik and Corinna Cortes in 1995 paper "Support-Vector Networks." Late 1990s through early 2010s, they dominated machine learning — text classification, handwriting recognition, bioinformatics. Deep learning took crown. SVMs stayed. Right tool in some settings, and ideas behind them — margin maximization, kernel trick, structural risk minimization — show up across modern ML.

The Line-Drawing Problem

Imagine email spam classifier. Two features per email — frequency of word "free" and frequency of exclamation marks. Plot six emails on flat surface: three spam (red dots), three legitimate (blue dots). They separate cleanly — red dots in one region, blue dots in another.

Now draw line between them. One side: spam. Other side: not spam.

Problem: infinitely many lines work. Tilt left. Shift up. Make angle steeper. All classify these six emails perfectly. Which line do you pick?

Naive approach says "any line that works is fine." Bad idea. New email arrives. If line hugs red dots, slightly unusual email can land on wrong side. Need breathing room. Need line that gives best chance on unseen emails.

That instinct — wanting breathing room — is SVM foundation.

Margin and Support Vectors

Picture separating line as road between two neighborhoods. One side: spam emails. Other side: legitimate emails. Margin is road width — distance between decision boundary and nearest data points from each class.

SVMs pick widest road possible. Whole idea.

Data points on road edge — closest to boundary from each side — are support vectors. Key insight: only those edge points determine boundary. Other data points do not matter. Delete them, move them, add thousands more — if support vectors stay same, decision boundary stays same.

First hearing this feels wrong. How can handful of points determine boundary that generalizes? Geometry explains it. Widest road between two groups is determined by points that constrain road width. Interior points sit too far away to matter.

In two dimensions, boundary is line. In three dimensions, flat plane. In higher dimensions, hyperplane — an (n−1)-dimensional flat surface cutting through n-dimensional space. Word sounds scary. Intuition stays same: find widest road between classes.

The Math Behind the Widest Road

Now see what optimizer does. Hyperplane is defined by weight vector w and bias term b. For any point x, decision function is w·x + b. One side gives positive value, other side negative value, points on hyperplane give zero.

Distance from any point x_i to hyperplane is |w·x_i + b| / ‖w‖. Denominator — ‖w‖, length of weight vector — matters. Same hyperplane can use many different (w, b) pairs. Scale both w and b by same constant, hyperplane stays same but raw scores change. That is difference between functional margin (raw score y_i(w·x_i + b), scale-dependent) and geometric margin (actual distance y_i(w·x_i + b) / ‖w‖, scale-independent).

To remove scale ambiguity, use convention: scale w and b so support vectors — closest points — have functional margin exactly 1. That is, y_i(w·x_i + b) = 1 for support vectors. With this convention, geometric margin — actual road width — becomes 2/‖w‖.

Maximizing 2/‖w‖ is same as minimizing ‖w‖, which is same as minimizing ½‖w‖². So SVM optimization problem in primal form is: minimize ½‖w‖² subject to y_i(w·x_i + b) ≥ 1 for every training point. Simple rule. Minimize weight-vector length while keeping every point on correct side of margin.

The ½ makes derivative cleaner. Nothing deeper.

When the Road Gets Messy — Soft Margin and C

Widest-road story assumed clean separation. Real email data looks messier. Some legitimate emails contain word "free." Some spam emails use professional restraint. Red and blue dots overlap. No line separates them perfectly.

Hard margin SVM allows zero violations — every point must stay on correct side of margin. If data is not linearly separable, optimization has no solution. Even when data is separable, one noisy point — one spam email that looks suspiciously professional — can squeeze road almost to nothing and make model fragile.

Fix: soft margin SVM. Introduce slack variables (ξ_i) so some points can violate margin or even land on wrong side. Optimization becomes: minimize ½‖w‖² + C × Σξ_i, subject to y_i(w·x_i + b) ≥ 1 − ξ_i and ξ_i ≥ 0.

C parameter controls this trade-off. Think of C as strictness dial.

Crank C high, optimizer pays heavily for every violation. It will chase correct training classifications even if that means narrow road bending around outliers. High C means low bias, high variance — model memorizes training data. Push C toward zero, optimizer cares more about road width than correctness. It tolerates misclassifications to get wider, smoother road. Low C means high bias, low variance — model generalizes but can miss patterns.

C is inverted relative to regularization strength λ in ridge or lasso regression. In ridge, high λ means more regularization (simpler model). In SVMs, high C means less regularization (more complex model). Same concept — balancing fit versus simplicity — but knob turns opposite way.

Hinge Loss — The Loss Function Hiding Inside SVMs

Another view of soft margin SVM connects it to rest of machine learning. Rewrite optimization as: minimize ½‖w‖² + C × Σ max(0, 1 − y_i(w·x_i + b)).

That max(0, 1 − y_i f(x_i)) term is hinge loss. It is zero when point is classified correctly with margin ≥ 1. It grows linearly as point drifts into margin or crosses wrong side. Name "hinge" comes from shape — flat at zero, then straight ramp, with sharp bend where they meet.

Viewed this way, soft margin SVM does familiar thing: minimize loss function (hinge loss) plus regularization term (½‖w‖², L2 regularization). Same structure as logistic regression with L2 regularization — only loss function changes. Logistic regression uses log-loss, which decays smoothly. Hinge loss cuts off at zero: once point is far enough on correct side, SVM stops caring. That is why SVMs produce sparse solutions — most points contribute zero loss, and only support vectors matter.

Internalize this connection. SVMs are not alien formulation. They are linear model with particular loss function and L2 regularization. Magic is not in loss — magic is in kernel trick, coming next.

When Lines Aren't Enough — The Kernel Trick

Back to email classifier. Suppose spam pattern is not high frequency of "free" or high count of exclamation marks alone. Suppose it is specific combination — emails where both features fall in particular range. On 2D plot, spam emails form cluster in middle, surrounded by legitimate emails. No line separates them. No hyperplane in original feature space works.

Classic response: project data into higher-dimensional space where it does become linearly separable. Add third feature — say, z = x² + y² — and concentric clusters in 2D lift apart in 3D, where flat plane can separate them. Math guarantee helps: given enough dimensions, any finite dataset becomes linearly separable. This is Cover's theorem.

Problem with this approach: cost. Mapping function φ can send 2-dimensional data into space with hundreds, thousands, or millions of dimensions. Computing φ(x) for every data point, storing those vectors, and computing dot products between them becomes brutally expensive.

Here comes trick. Look closely at SVM optimization problem, and one fact jumps out: data points appear only as dot products with each other. Optimizer never needs actual coordinates in high-dimensional space. It only needs dot product φ(x_i) · φ(x_j) for pairs of points.

Kernel function K(x_i, x_j) computes that high-dimensional dot product directly from original low-dimensional coordinates, without performing mapping φ. You get result of high-dimensional work without paying full cost of going there.

Think of it like this. Want distance between two cities. Expensive approach: drive to each city, record GPS coordinates in global reference frame, then compute Euclidean distance. Kernel trick approach: look up driving distance directly, without computing absolute coordinates. Answer comes from relationship between two points, not their positions in absolute space.

This analogy comes back in dual form.

The RBF Kernel and Infinite Dimensions

Most common kernel is RBF (Radial Basis Function), also called Gaussian kernel: K(x, y) = exp(−γ‖x − y‖²). It computes similarity score between two points using only distance between them — close points score near 1, distant points score near 0.

RBF gets wild because of feature space it implicitly maps to. Expand exponential as Taylor series, and each term in infinite series matches one dimension in feature space. RBF kernel's implied feature space is literally infinite-dimensional. You do linear classification in space with infinitely many features, while computation still takes O(d) time, where d is original number of features.

Favorite thing about RBF kernel: even with high-level explanations, no one has fully satisfying intuition for why it works so well across so many problems. It is universal kernel — it can approximate any continuous function — and in practice it is strong first choice when you do not know data structure.

Math guarantee behind this is Mercer's theorem. It says any continuous, symmetric, positive semi-definite function can be decomposed as inner product in some feature space. If kernel satisfies Mercer's condition, it corresponds to legitimate feature space, even if you cannot write that space explicitly.

Other common kernels are linear kernel (K(x, y) = x·y, no mapping at all — useful for sparse, high-dimensional data like TF-IDF text features) and polynomial kernel (K(x, y) = (γ·x·y + r)^d, which maps to polynomial interaction terms up to degree d). In practice, use RBF most times, linear for text, polynomial rarely.

Gamma — The Spotlight Knob

γ parameter in RBF kernel controls how far each training point's influence reaches. Think of every support vector as lamp on decision landscape. Gamma decides whether lamp is narrow spotlight or broad floodlight.

High gamma means narrow spotlight. Each support vector affects only nearby points. Decision boundary can twist around individual points and create complex, fragmented regions. Path to overfitting.

Low gamma means broad floodlight. Each support vector influences large area. Boundary becomes smooth and sweeping — it cannot capture fine details. Path to underfitting.

Mathematically, γ = 1/(2σ²), where σ is Gaussian bandwidth. High gamma means tight Gaussian — influence drops fast. Low gamma means wide Gaussian — influence reaches large radius. Default in scikit-learn (gamma='scale') sets γ = 1/(n_features × Var(X)), which is reasonable starting point that adapts to data spread.

C and Gamma Together

C and gamma are two knobs for RBF kernel. They interact in ways that trip people up.

High C + high gamma: model insists on classifying every training point correctly and uses boundary that can twist around individual points. Overfitting territory — model memorizes training set, including noise.

Low C + low gamma: model tolerates errors and uses smooth, sweeping boundary. This can underfit — model is too gentle to capture real pattern.

Sweet spot sits between extremes. Reliable way to find it: grid search over both parameters with cross-validation. Use logarithmic scales — optimal values can differ by orders of magnitude.

C_range = [0.01, 0.1, 1, 10, 100, 1000]
gamma_range = [0.0001, 0.001, 0.01, 0.1, 1, 10]
# 36 combinations. With 5-fold CV, that's 180 fits.
# Scikit-learn handles this in seconds for moderate datasets.

Do not tune these by hand. Interaction is non-linear and non-intuitive. Trust grid search.

Why the Dual Form Matters

Dual formulation matters because it makes kernel trick possible. To understand why data appears only as dot products, you need dual.

Primal form says: minimize ½‖w‖² subject to constraints involving w, b, and data. To solve this constrained optimization problem, introduce Lagrange multipliers — one α_i for each training point — and construct Lagrangian. Take derivatives with respect to w and b, set them to zero, and get two key relationships: w = Σ α_i y_i x_i (weight vector is linear combination of training points) and Σ α_i y_i = 0 (multipliers balance across classes).

Substitute these back and you get dual form: maximize Σα_i − ½ΣΣ α_i α_j y_i y_j (x_i · x_j), subject to α_i ≥ 0 and Σ α_i y_i = 0.

Look at where data appears: only in dot product x_i · x_j. That is why kernel trick works. Replace every x_i · x_j with K(x_i, x_j), and optimizer keeps going as if nothing changed. It still solves same kind of quadratic program, but dot products now correspond to implicit mapping that can be infinite-dimensional.

Full Lagrangian derivation is easy to forget. Punchline matters: dual form makes data appear only as dot products, and that lets us swap in kernels. That is insight worth remembering.

One more thing the dual tells us. The KKT (Karush-Kuhn-Tucker) complementary slackness condition says: α_i × [y_i(w·x_i + b) − 1] = 0. This means either α_i = 0 (the point is beyond the margin and doesn't influence the boundary) or y_i(w·x_i + b) = 1 (the point sits on the margin and is a support vector). That's the mathematical confirmation of what we said earlier: only the support vectors matter.

Practical Matters — Scaling, Multi-Class, Probabilities

Feature scaling is mandatory. SVMs compute distances and dot products. If one feature ranges from 0 to 1 and another from 0 to 1,000,000, the second feature will dominate the dot product entirely. The model effectively ignores the first feature. Always standardize (zero mean, unit variance) or min-max scale before fitting an SVM. And do it inside a pipeline so you don't leak test set statistics into training.

Multi-class classification. SVMs are inherently binary classifiers. To handle multiple classes, scikit-learn's SVC uses one-vs-one (OvO) by default: for k classes, it trains k(k−1)/2 binary classifiers, one for each pair of classes, and predicts by majority vote. The alternative is one-vs-rest (OvR): k classifiers, each separating one class from all the others. OvO trains more models but each on less data; OvR trains fewer models but each faces class imbalance. For most problems, OvO works well — it's what scikit-learn gives you out of the box.

Probability estimates. SVMs don't natively output probabilities. When you set probability=True in scikit-learn, it uses Platt scaling — a post-hoc calibration that fits a sigmoid to the SVM's decision scores using internal cross-validation. The output is P(y=1|f) = 1/(1 + exp(Af + B)), where A and B are fit on held-out data. It works reasonably well, but it's slower (extra CV loop), and the probabilities are less reliable than what you'd get from a natively probabilistic model like logistic regression. Use it when you need probabilities, but know what you're getting.

SVMs for Regression

SVMs can do regression too. Support Vector Regression (SVR) uses an epsilon-insensitive loss: errors smaller than ε contribute zero loss. The model fits a tube of width 2ε around the data, and only points outside the tube — the support vectors — shape the regression function. Points inside the tube are "close enough" and ignored.

The same kernel trick applies, so SVR can fit non-linear functions. But in practice, gradient boosted trees handle most regression tasks faster and with less tuning. You'll encounter SVR more in textbooks and interview questions than in production code.

When SVMs Still Win Today

SVMs were the king of machine learning for over a decade. That reign ended when deep learning demonstrated that learning features end-to-end from raw data beats hand-crafting features and running an SVM on top. But SVMs didn't disappear. They retreated to the niches where their strengths are hardest to replicate.

Text classification. A linear SVM on TF-IDF features remains one of the best approaches for document classification when you don't have the data or GPU budget for transformers. The data is naturally sparse and high-dimensional — exactly the regime where SVMs excel. In many benchmark comparisons, linear SVM matches or beats more complex approaches on small to medium text corpora.

High-dimensional, low-sample data. Genomics, spectroscopy, medical imaging features — domains where you have thousands of features but only hundreds of samples. The margin-maximization objective provides strong implicit regularization that prevents overfitting even when features vastly outnumber samples.

Small datasets with clear structure. When you have hundreds to low thousands of labeled examples, SVMs can outperform tree ensembles because the geometric margin gives them a better inductive bias for clean, structured data. No bootstrap aggregation needed — the margin does the regularization work.

Embedded systems and latency-sensitive applications. A trained SVM stores only the support vectors. Inference is a handful of kernel evaluations — fast, deterministic, and lightweight. No GPU required.

For anything with more than about 50,000 samples, the training time becomes painful (SVM training is O(n²) to O(n³), using the SMO algorithm under the hood). For tabular data at scale, reach for LightGBM or XGBoost. For images, text, or audio, deep learning dominates. But in those niches — small data, high dimensions, no GPU — SVMs are still a serious contender.

SVM in Code

The pattern that matters: scaler inside the pipeline, grid search over C and gamma on log scales, cross-validation to select the best combination.

from sklearn.svm import SVC
from sklearn.pipeline import make_pipeline
from sklearn.preprocessing import StandardScaler
from sklearn.model_selection import GridSearchCV

# The scaler MUST be inside the pipeline.
# Scaling outside leaks test set statistics into training.
svm_pipe = make_pipeline(
    StandardScaler(),
    SVC(kernel="rbf")
)

param_grid = {
    "svc__C": [0.1, 1, 10, 100],
    "svc__gamma": ["scale", 0.01, 0.1, 1],
}

search = GridSearchCV(svm_pipe, param_grid, cv=5, scoring="accuracy", n_jobs=-1)
search.fit(X_train, y_train)

print(f"Best params: {search.best_params_}")
print(f"Test accuracy: {search.score(X_test, y_test):.4f}")

That "scale" in the gamma grid is scikit-learn's default: γ = 1/(n_features × Var(X)). It adapts to the data's spread and is a good anchor point. The numeric values in the grid explore outward from there.

For text classification, swap RBF for linear and use LinearSVC (faster for high-dimensional sparse data). For probability outputs, add probability=True to SVC, but know it slows training due to internal cross-validation for Platt scaling.
