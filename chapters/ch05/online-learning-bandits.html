Batch ML loop feels safe: collect data, train, evaluate, deploy. Online learning kills loop — learn from stream, one example each time, no do-overs. Powers ad systems, recommendation engines, adaptive A/B tests in production. Dive starts here.

Online learning: model gets examples one at time, updates after each one. Formalized in 1950s with Perceptron, matured through work of Littlestone, Vovk, others in 1990s. Multi-armed bandits — special case where you only see outcome of chosen action — go back to Thompson in 1933 and got modern theory from Robbins in 1952. Today these ideas drive production systems at Google, Netflix, Spotify, every major ad platform.

The Prediction Game

Start with scenario. You run small coffee shop. Every morning, choose single daily special for chalkboard before customers arrive. Pick one, day happens, end of day you see how many people ordered it. Tomorrow, pick again.

Five candidates: lavender latte, cold brew float, matcha espresso, caramel cortado, chai affogato. No historical data. No focus groups. Chalkboard. One shot per day.

Essence of online learning. Not "collect data, then train." Instead: predict, observe, update, repeat. One round each time. No second passes. No reshuffling. World gives situation, you choose, you see outcome, you adjust.

Formal game version. Each morning (round t), you see day info — maybe weather forecast, day of week. You make prediction or choice. World reveals outcome. You update strategy. Then round t + 1 starts.

Striking part: no assumption about data source. Data need not be "independent and identically distributed." Weather could be adversarial. Customer preferences could shift week to week. Framework still works. Power comes from that. Difference from batch ML comes from that.

Regret: Keeping Score Against Hindsight

Batch ML scores model with accuracy or test loss. Online learning asks harder question: how much worse did I do, over all T rounds, compared to best fixed strategy I could have chosen if I'd had crystal ball?

This quantity has name. Regret.

Back to coffee shop. After 100 days, total daily-special sales = 620 cups. If you had featured cold brew float every day — best choice in hindsight — you would have sold 780 cups. Regret = 780 − 620 = 160 cups. Price of not knowing best special early. Price of experimentation days.

More precise: regret = your cumulative loss minus cumulative loss of single best fixed predictor from your hypothesis class, computed after seeing all T rounds. You do not compete against magical oracle that changes each round. You compete against single best unchanging strategy — one that wins overall in hindsight.

Good online algorithm gets sublinear regret — regret grows slower than round count. If regret is O(√T), then after 10,000 rounds, average per-round regret scales like 1/√10000 = 1/100. It shrinks. You approach best fixed strategy performance, even without seeing all data at once.

First time regret metric hit me, felt defeatist. Measure how much you lost? But metric is sharper than accuracy. Accuracy says how often you were right. Regret says how fast you learned to be right. In streaming world, learning speed matters.

The Perceptron: Where Online Learning Began

Perceptron, published by Rosenblatt in 1958, is oldest online learning algorithm most people know. It predicts one data point at time. It updates only on mistakes.

Setup. Each morning, using weather forecast (vector of numbers — temperature, humidity, rain chance), you predict whether it will be "hot drinks" day or "cold drinks" day. You keep weight vector w. It encodes current belief about which weather features matter.

When new forecast x arrives, compute w · x (dot product). Positive: predict "hot drinks." Negative: "cold drinks." If right, do nothing. If wrong, nudge weights: add x if true label should be positive, subtract x if true label should be negative.

import numpy as np

def perceptron_online(stream):
    w = None
    mistakes = 0

    for x, y in stream:  # y is +1 or -1
        if w is None:
            w = np.zeros(len(x))

        prediction = 1 if w @ x >= 0 else -1

        if prediction != y:
            w += y * x    # nudge toward correct side
            mistakes += 1

    return w, mistakes

Whole algorithm. No learning rate tuning. No loss differentiation. Wrong? Adjust.

Now remarkable result. If data is linearly separable — some perfect weight vector w* classifies everything correctly with margin γ (every point at least γ from decision boundary) — Perceptron makes at most (R/γ)² mistakes, where R is maximum length of any data point.

This is Perceptron Convergence Theorem. Bound: (R/γ)² mistakes. Think finish line. Each mistake pushes weights closer to correct solution. Large γ means classes well separated, so each mistake makes big stride. Small R means data points cannot overshoot. Big strides. Short track. Fewer steps.

Why this matters: Perceptron was first algorithm with guarantee on online performance. Data order does not matter. Adversary choosing hardest next example does not matter. Bound still holds. Guarantees without distributional assumptions — core philosophy of online learning.

But Perceptron only handles linearly separable data. What if no perfect separator exists? What if task is regression, not classification? Need something more general.

Online Gradient Descent

Workhorse of online learning is Online Gradient Descent (OGD). Looks like stochastic gradient descent you know, but framing changes. In SGD, you minimize fixed loss on fixed dataset, using random samples to approximate gradient. In OGD, loss functions can change each round. No fixed dataset. Prediction game keeps going. Each round brings new loss.

Apply this to coffee shop. Instead of choosing one special, you set prices for all five drinks. Each morning you choose price vector w. Customers arrive, end of day you see revenue lost versus optimal pricing. That loss is ft(w). It can change every day because customer behavior shifts.

OGD rule: after each day, take one gradient step against today's loss.

import numpy as np

def online_gradient_descent(stream, d, lr_fn=None):
    """OGD for online convex optimization.
    stream yields (loss_fn, gradient_fn) pairs each round."""
    w = np.zeros(d)
    total_loss = 0.0

    for t, (loss_fn, grad_fn) in enumerate(stream, 1):
        # suffer today's loss
        total_loss += loss_fn(w)

        # step size shrinks over time
        lr = lr_fn(t) if lr_fn else 1.0 / np.sqrt(t)

        # update using today's gradient
        w = w - lr * grad_fn(w)

    return w, total_loss

With learning rate ηt = 1/√t, OGD gets O(√T) regret for convex losses. Best possible without stronger assumptions. If losses are strongly convex — sharp bowl, not flat plate — regret becomes O(log T), much better. After 10,000 rounds, √T = 100 while log T ≈ 9. Difference: "pretty close to optimal" versus "almost indistinguishable from optimal."

Surprising connection: Perceptron is OGD applied to hinge loss. "Only update on mistakes" behavior comes from hinge-loss gradient being zero on correctly classified points. Perceptron is not separate invention. It is special case of same framework. Online learning existed since 1950s, even when people did not call it that.

Prediction with Expert Advice

Another online learning flavor worth knowing before moving on. Instead of one model, imagine N "experts" — different models, heuristics, weather forecasters. Each round, every expert gives prediction. You combine them into one, then see truth.

Back to coffee shop. Suppose three advisors: barista (cold drinks always win), regular customer (seasonal specials win), random internet algorithm. Each morning all choose special to feature. You want to learn which advisor to trust. You want fast learning.

Weighted Majority Algorithm (Littlestone and Warmuth, 1994) does exactly this. Start with equal weight for every expert. Each round, predict using weighted vote. After outcome, multiply weight of each wrong expert by (1 − η), where η is small learning rate. Correct experts keep weight. Over time, bad experts fade.

Guarantee: if best expert makes M mistakes, Weighted Majority makes at most about 2(1 + η)M + (2 ln N)/η mistakes. With right η, this is about M + O(√(M ln N)). You stay close to best expert, without knowing early which one it is.

Randomized version — Hedge algorithm (also called Exponential Weights) — is beautiful learning theory result. Instead of hard vote, sample expert proportional to weight. Regret sharpens to O(√(T ln N)). Logarithmic dependence on N matters: with million experts, cost of not knowing best one grows only like ln(1,000,000) ≈ 14. Fourteen extra rounds of regret, no matter how many experts you juggle.

Intuition, rough version: exponential weighting works because multiplicative updates amplify small expert differences faster than additive updates. Slightly worse expert loses weight exponentially fast. Slightly better expert keeps gaining influence. Exponential function does heavy lifting. Same trick shows up later in EXP3 for adversarial bandits.

Limit exists. In prediction-with-expert-advice and OGD, you see full outcome after each round. You learn what would have happened for every possible action, not only chosen one. This is full information feedback. Many real problems do not give this luxury.

But real missing piece: what if you can't see full outcome? You choose lavender latte as today's special and never learn how cold brew float would have done. That is bandit problem. That is where exploration-exploitation tradeoff lives.

If hidden outcomes still bother you, keep going.

The Bandit Problem: When You Can't See the Road Not Taken

Shift happens here. In online learning, you choose action and environment reveals loss for all actions — full picture, even for unchosen options. In multi-armed bandit problem, you only see reward for action you actually took. Everything else stays hidden.

Name comes from gambler facing row of slot machines — each machine has one arm, so "one-armed bandits." Each machine has different unknown payout probability. You have finite coins. Which machines do you play?

Coffee shop version: choose one daily special each morning. End of day, you know cups sold. But you do not know what different special would have done. Lavender latte sold 12 cups today — good? bad? Cannot know. You did not try alternatives.

This is partial information feedback, or bandit feedback. Problem gets fundamentally harder. Full-information online learning lets you compute gradients. Lets you evaluate every strategy each round. Bandit feedback leaves you partly blind.

Central tension shows immediately: keep featuring special that sells well (exploit), or try different one to gather info that might lead to better sales (explore)? Over-exploit and you miss true winner. Over-explore and you waste days on specials you already suspect are mediocre.

This is exploration-exploitation tradeoff. Core idea in decision theory. Shows up everywhere — restaurants, venture capital allocation, clinical trials, ads.

Formal version: you have K arms. At each round t, you pull arm at and receive random reward from that arm's distribution. Bandit regret compares your total reward with reward from pulling best arm every round: Regret(T) = T · μ* − Σ μat, where μ* is mean of best arm. Lai-Robbins lower bound (1985) proves even best algorithm must incur at least Ω(log T) regret. Exploration is unavoidable. Cleverness cannot erase it.

Question is not whether to explore. Question is how.

Epsilon-Greedy: The Coin-Flip Approach

Most obvious strategy. Each round, flip biased coin. With probability (1 − ε), choose special with highest average sales so far. With probability ε, choose random special.

Coffee shop example. Say ε = 0.1. Most days — 90% — you feature current best seller. One day in ten, you try random alternative. Maybe that random day reveals chai affogato is sleeper hit. Maybe it confirms matcha espresso is mediocre. Either way, you gather information.

import numpy as np

class EpsilonGreedy:
    def __init__(self, n_arms, epsilon=0.1):
        self.epsilon = epsilon
        self.counts = np.zeros(n_arms)
        self.values = np.zeros(n_arms)

    def select_arm(self):
        if np.random.random() < self.epsilon:
            return np.random.randint(len(self.counts))
        return np.argmax(self.values)

    def update(self, arm, reward):
        self.counts[arm] += 1
        # incremental mean: new_mean = old_mean + (reward - old_mean) / n
        self.values[arm] += (reward - self.values[arm]) / self.counts[arm]

Incremental mean update matters. Instead of storing all past rewards and recomputing average, fold each new observation into running average with one line. Algebra works because (reward − old_mean) / n is exact correction. Same trick appears in Welford's algorithm for online variance.

Epsilon-greedy works. Works surprisingly well for something so crude. But obvious problem: ε is fixed. After 5,000 days — years of coffee shop data — when you are very sure cold brew float wins, you still waste 10% of days on random experiments. That is 500 lost-sales days, not because info is needed, but because algorithm never learns when exploration can slow down.

You can decay ε over time — εt = 1/t, or ε₀/(1 + t/τ) for constant τ. With right decay schedule, regret improves to O(log T). Catch: "right schedule" depends on arm gaps, and you do not know them. You tune hyperparameter in setting built to avoid missing info. That circularity pushed people toward better methods.

UCB: Optimism as a Strategy

Instead of random exploration, what if exploration were strategic? What if you always picked option that could be best, given what you know and what you do not know?

That is Upper Confidence Bound (UCB). For each arm, compute two pieces: best estimate of average reward, and bonus for uncertainty. Pick arm with largest sum.

Coffee shop again. Cold brew float averages 15 cups/day over 200 days — high confidence. Chai affogato averages 14 cups/day over only 5 days — low confidence. UCB says give chai affogato big uncertainty bonus because data is scarce. Maybe true average is 20 cups/day and noisy 5-day sample hid it.

UCB1 formula makes this precise:

Pick arm a that maximizes: μ̂a + √(2 ln t / Na)

First term, μ̂a, is empirical mean — exploitation. Second term, √(2 ln t / Na), is confidence bonus — exploration. Small Na means little data, so big bonus. Large Na means lots of data, so bonus shrinks toward zero. ln t in numerator ensures even well-explored arms get revisited eventually, but less and less often.

class UCB1:
    def __init__(self, n_arms):
        self.counts = np.zeros(n_arms)
        self.values = np.zeros(n_arms)

    def select_arm(self):
        n_arms = len(self.counts)
        for a in range(n_arms):
            if self.counts[a] == 0:
                return a  # try each arm at least once

        t = np.sum(self.counts)
        bonuses = np.sqrt(2 * np.log(t) / self.counts)
        return np.argmax(self.values + bonuses)

    def update(self, arm, reward):
        self.counts[arm] += 1
        self.values[arm] += (reward - self.values[arm]) / self.counts[arm]

UCB1 gets O(log T) regret, matching Lai-Robbins lower bound up to constants. No hyperparameters. No coin flips. Algorithm decides on its own how much to explore and when to stop. Phrase to remember: "optimism in face of uncertainty". When you do not know, assume maybe wonderful, then test it. If it is wonderful, good. If mediocre, uncertainty bonus shrinks and wasted pulls stop.

Same principle appears all over ML. UCT in AlphaGo's Monte Carlo Tree Search is UCB on game trees. Bayesian optimization uses similar acquisition function (Upper Confidence Bound variant). Optimistic initialization in reinforcement learning follows same philosophy. Idea travels well.

But UCB has quirk. It explores all uncertain arms with roughly equal enthusiasm. If you have 1,000 arms and 990 look terrible after few pulls, UCB still gives each one uncertainty bonus that forces occasional revisits. Principled. Not always efficient. Next algorithm often beats it in practice.

Thompson Sampling: Let Uncertainty Do the Thinking

Thompson Sampling appeared in 1933 — yes, 1933 — from William R. Thompson, statistician working on clinical trials. Then mostly forgotten for about 80 years, rediscovered by tech industry in 2010s, now default bandit algorithm at most major tech companies. Gap between invention and adoption is wild applied statistics story.

Idea is Bayesian. For each arm, keep probability distribution — posterior — over possible true reward rate. Each round, sample one value from each arm's posterior. Pull arm with highest sample.

Make it concrete with coffee shop. Suppose reward for each special is binary: customer orders it (1) or does not (0). After observing successes and failures for each special, keep Beta distribution for each one. Beta distribution is natural posterior for binary data — parameterized by α (successes + 1) and β (failures + 1).

Day one: every special has Beta(1, 1) — flat distribution. You know nothing. So posterior samples are wild and spread out. Any special can win. Pure exploration.

After 50 days, say cold brew float has 40 orders and 10 non-orders: Beta(41, 11). Posterior is tight, centered near 0.78. Chai affogato has 3 orders and 2 non-orders: Beta(4, 3). Posterior is wide, centered near 0.57, but could plausibly be 0.85 or 0.25. On some rounds, chai samples 0.82 and beats cold brew sample 0.76. That triggers exploration. You try chai again. Most rounds, cold brew's tight high posterior wins, and you exploit.

class ThompsonSampling:
    def __init__(self, n_arms):
        self.alpha = np.ones(n_arms)  # successes + 1
        self.beta = np.ones(n_arms)   # failures + 1

    def select_arm(self):
        samples = np.random.beta(self.alpha, self.beta)
        return np.argmax(samples)

    def update(self, arm, reward):
        self.alpha[arm] += reward
        self.beta[arm] += 1 - reward

Whole algorithm. Five logic lines. Posterior does main work — naturally balances exploration and exploitation. Arm with few observations has wide posterior, so samples sometimes jump high and trigger exploration. Arm with many observations and low reward has tight posterior near zero — it almost never wins sample contest. No confidence bounds. No epsilon. No tuning.

Thompson Sampling gets O(log T) regret, matching Lai-Robbins lower bound. But practitioners love it for more than theory. UCB explores all uncertain arms with similar intensity. Thompson focuses exploration on arms that are plausibly near-optimal. If two arms look similar, Thompson naturally samples between them. If one arm looks clearly bad, its posterior quickly becomes tight and low, and Thompson stops wasting pulls. With thousands of arms — common in ad serving — this efficiency gap is huge.

Favorite thing about Thompson Sampling: beyond high-level stories like this one, nobody was fully sure why it worked so well in practice. Theory proving Lai-Robbins-level performance arrived only in 2012 (Agrawal and Goyal), almost 80 years after proposal. For most of that time, users relied on faith and experiments. Theory caught up later. Practitioners had already picked right tool.

For continuous rewards (say, average revenue per customer instead of click/no-click), swap Beta posterior for Gaussian. Track sample mean and variance for each arm, then sample from Normal distribution with appropriate posterior uncertainty:

class ThompsonGaussian:
    def __init__(self, n_arms):
        self.counts = np.zeros(n_arms)
        self.means = np.zeros(n_arms)
        self.sum_sq = np.zeros(n_arms)  # Welford's running variance

    def select_arm(self):
        samples = np.zeros(len(self.counts))
        for a in range(len(self.counts)):
            if self.counts[a] < 2:
                samples[a] = np.inf  # explore arms with too few observations
            else:
                var = self.sum_sq[a] / (self.counts[a] - 1)
                # posterior uncertainty shrinks as 1/sqrt(n)
                samples[a] = np.random.normal(
                    self.means[a], np.sqrt(var / self.counts[a]))
        return np.argmax(samples)

    def update(self, arm, reward):
        self.counts[arm] += 1
        old = self.means[arm]
        self.means[arm] += (reward - old) / self.counts[arm]
        self.sum_sq[arm] += (reward - old) * (reward - self.means[arm])

Adversarial Bandits and EXP3

Everything so far assumed stochastic rewards — each arm has fixed unknown distribution, rewards drawn independently from it. But what if environment is adversarial? What if someone actively tries to make your algorithm look bad, changing rewards each round to maximize regret?

Coffee shop analogy: competitor across street watches chalkboard each morning, then runs promotion on whatever special you chose and steals customers. Less dramatic version: rewards become non-stationary in unpredictable ways — seasons, viral trends, supply chain disruptions.

For adversarial bandits, need EXP3 — Exponential-weight algorithm for Exploration and Exploitation (Auer et al., 2002). Key challenge: with bandit feedback, you only see reward for pulled arm. You need estimate rewards for arms you didn't pull, without observing them. EXP3 solves this with importance-weighted estimation.

Idea works like this. Keep weight for each arm. Convert weights to probabilities (with some mixing for guaranteed exploration). Sample arm from those probabilities. Observe reward. Then trick: estimate chosen arm's reward by dividing observed reward by probability of choosing it. If you chose arm a with probability pa and saw reward r, estimate is r / pa. Reason: you only see this arm with probability pa, so estimate must be inflated to compensate. In expectation, estimate is unbiased — average equals true reward.

class EXP3:
    def __init__(self, n_arms, gamma=0.1):
        self.n_arms = n_arms
        self.gamma = gamma
        self.weights = np.ones(n_arms)

    def _probabilities(self):
        total = np.sum(self.weights)
        # mix exponential weights with uniform exploration
        return (1 - self.gamma) * (self.weights / total) \
               + self.gamma / self.n_arms

    def select_arm(self):
        probs = self._probabilities()
        return np.random.choice(self.n_arms, p=probs)

    def update(self, arm, reward):
        probs = self._probabilities()
        # importance-weighted reward estimate
        estimated_reward = reward / probs[arm]
        # exponential weight update
        self.weights[arm] *= np.exp(
            self.gamma * estimated_reward / self.n_arms)

EXP3 gets O(√(TK ln K)) regret in adversarial setting, where K is number of arms. Worse than O(log T) from UCB or Thompson in stochastic setting — but stochastic algorithms can fail badly when rewards are adversarial. EXP3 gives guarantee no matter what environment does.

Importance weighting took longest to make intuitive for me. Division by pa looks scary — what if pa is tiny? It can blow up, and that is why estimate variance is high. γ/K floor on each probability keeps things from completely breaking, but price is noisier updates. Fundamental cost of adversarial robustness: trade tight estimates for worst-case guarantees.

Contextual Bandits: Because People Are Different

Everything so far treated each round as identical. Pull arm, get reward from arm's distribution. Real world cares about context. Lavender latte may sell on rainy Mondays to morning commuters, and fail on sunny Friday afternoons to college students. Optimal action depends on who walks in and when.

Contextual bandits formalize this. Each round, you observe feature vector xt — context — before choosing action. Expected reward depends on both action and context. You still only see reward for chosen action.

This is exact setup for recommendation systems. Arms are items to show (news articles, ads, products). Context is everything known about user (demographics, session features, time of day, device). Reward is engagement — click, purchase, time spent. Goal: learn policy — function from context to action — that maximizes expected reward.

LinUCB and the Yahoo Story

Most influential contextual bandit algorithm is LinUCB, published by Li et al. in 2010 and built at Yahoo for personalized news recommendation. Model is clean: assume expected reward for arm a given context x is linear. That is, E[reward | x, a] = θaTx, where θa is unknown weight vector for each arm.

For each arm, LinUCB keeps ridge regression estimate of θa plus confidence ellipsoid — multi-dimensional confidence interval. Given context x, predicted reward plus exploration bonus gives UCB:

UCBa(x) = θ̂aTx + α √(xT Aa−1 x)

First term is predicted reward — exploitation. Second term, √(xT Aa−1 x), measures how unfamiliar this context is to arm a. If current user has features arm a rarely saw, bonus is large and LinUCB explores. If past data covers context well, bonus is small and LinUCB exploits.

class LinUCB:
    def __init__(self, n_arms, d, alpha=1.0):
        self.alpha = alpha
        # per-arm: A is d×d, b is d-vector
        self.A = [np.eye(d) for _ in range(n_arms)]
        self.b = [np.zeros(d) for _ in range(n_arms)]

    def select_arm(self, x):
        ucbs = np.zeros(len(self.A))
        for a in range(len(self.A)):
            A_inv = np.linalg.inv(self.A[a])
            theta = A_inv @ self.b[a]
            ucbs[a] = theta @ x + self.alpha * np.sqrt(x @ A_inv @ x)
        return np.argmax(ucbs)

    def update(self, arm, x, reward):
        self.A[arm] += np.outer(x, x)   # tighten confidence ellipsoid
        self.b[arm] += reward * x         # update reward signal

Aa matrix accumulates outer products of contexts seen by arm a. Its inverse represents uncertainty — directions in feature space with little data. Update np.outer(x, x) tightens ellipsoid along observed context direction. Over time, more data accumulates, exploration bonuses shrink, LinUCB converges to best linear policy.

Yahoo deployment made industry take contextual bandits seriously. System served personalized news articles to millions of Yahoo front-page users, and bandit approach beat both random selection and fixed supervised model. Key insight: supervised model trained on historical click data suffers feedback loops. If old system never showed sports articles to users over 60, supervised model learns "users over 60 do not click sports articles" — but only because system never tested it. LinUCB explores and uncovers hidden preferences.

Difference from full reinforcement learning matters. In contextual bandits, each round is independent — action today does not change tomorrow's context. In RL, actions change state. Recommend video today, change what user wants next. Contextual bandits ignore that sequential dependency. Real limitation. Also reason they stay tractable enough for web-scale production.

Online Convex Optimization: The Unifying Framework

There's a theoretical framework that ties together nearly everything we've discussed. Online Convex Optimization (OCO) is the general game: each round, you choose a point from a convex set, an adversary reveals a convex loss function, you incur that loss. The goal: minimize regret against the best fixed point in hindsight.

OGD is one algorithm for this game. But there's a more elegant, unifying approach: Follow the Regularized Leader (FTRL). At each round, choose the point that minimizes the sum of all past losses plus a regularization term:

wt = argminw [ Σs=1..t-1 fs(w) + R(w) ]

Without the regularizer, this would be Follow the Leader (FTL) — always play whatever would have been best so far. FTL sounds reasonable but can oscillate wildly. If today's best answer is the opposite of yesterday's, FTL whiplashes back and forth. The regularizer penalizes large deviations, smoothing the trajectory.

Here's what makes FTRL beautiful: different regularizers give you different algorithms. L2 regularization (R(w) = ||w||²/2η) recovers Online Gradient Descent. Negative entropy regularization (R(w) = Σ wi ln wi) gives you the Exponential Weights / Hedge algorithm. It's the same template, with the regularizer acting as a knob that controls the geometry of the update.

Google's ad serving system uses a variant called FTRL-Proximal, which adds L1 regularization for sparsity. When you have billions of features (most of which are irrelevant for any given ad), L1 regularization drives irrelevant weights to exactly zero, keeping the model tractable. The paper describing this (McMahan et al., 2013, "Ad Click Prediction: a View from the Trenches") is one of the most cited papers in applied ML, and it's fundamentally an online learning algorithm.

The OCO framework also gives a clean way to state the regret bounds we've been collecting. For convex losses: O(√T). For strongly convex losses: O(log T). For exp-concave losses: O(d log T) where d is the dimension. These bounds hold against adversarial loss sequences — no distributional assumptions needed. That's the power of the framework.

Where All of This Lives in the Real World

Let's bring it all back from theory to practice.

A/B testing. Traditional A/B testing splits traffic 50/50, waits two weeks, then computes a p-value. During those two weeks, half your users see the worse variant. If variant B is clearly superior after day 3, you're still forcing 50% of traffic to the loser for 11 more days. Bandit-based testing — typically Thompson Sampling — gradually shifts traffic toward the winning variant while the experiment is running. Early on, it's roughly 50/50. As evidence accumulates, it becomes 90/10. Less wasted exposure, faster convergence. Google, Netflix, and Microsoft all use variants of this. One critical caveat: because the sampling is adaptive, standard frequentist confidence intervals become invalid. You need "always-valid" confidence sequences or careful sequential testing on top of the bandit allocation.

Ad placement. Every ad impression is a bandit round. The context is the user profile plus page content. The arms are candidate ads. The reward is a click or a purchase. The system needs to maximize click-through rate while still discovering which creatives work for which audiences. This is the canonical contextual bandit application, and Thompson Sampling is the industry standard. Netflix uses a similar approach for choosing which thumbnail to show for each title — the same movie gets different artwork depending on who's looking at the catalog.

Recommendation systems. When you open Spotify or YouTube, a contextual bandit (or something close) decides what to surface. The reason bandits beat pure supervised models here is the feedback loop problem. A supervised model trained on historical data only learns about items the old system chose to show. If the old system never showed jazz to rock fans, the supervised model concludes rock fans don't like jazz — not because it's true, but because it never ran the experiment. Bandits, by construction, explore past the edges of what's already known.

Clinical trials. Adaptive clinical trials use bandit algorithms to shift more patients toward the treatment arm that's performing better. The response-adaptive randomization used during COVID-19 vaccine trials is an example. Compared to fixed equal randomization, this approach can reduce the number of patients receiving inferior treatment while maintaining statistical validity. The ethical argument is hard to ignore: if you have evidence mid-trial that treatment A works better than treatment B, is it ethical to keep assigning half the patients to B?

Hyperparameter tuning. Bayesian optimization for hyperparameter search is, at its core, a bandit problem with continuous arms. Each "arm" is a hyperparameter configuration, the reward is validation performance, and Gaussian process UCB or Thompson Sampling guides which configuration to try next. Tools like Optuna and BoTorch are bandit algorithms wearing a different hat.
