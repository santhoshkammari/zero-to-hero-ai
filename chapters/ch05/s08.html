Time Series Forecasting

Chapter 5: Supervised Learning Walk-forward · Lag features · ML approach

What Makes Time Series Special

Most ML assumes your data points are independent and identically distributed (i.i.d.). Time series violates this assumption in the most fundamental way possible: each observation depends on the ones before it. Today's stock price depends on yesterday's. This week's sales depend on last week's. This month's temperature is correlated with last month's.

This single fact — temporal dependence — changes everything about how you build, validate, and deploy models:

  • No random shuffling. If you randomly split time series data into train and test, you'll have future data leaking into training. Your model will look brilliant in evaluation and fail catastrophically in production.
  • Autocorrelation exists. Consecutive observations are correlated. Today's value tells you a lot about tomorrow's. This is a feature you can exploit, not a nuisance to remove.
  • Stationarity matters. Many models assume the statistical properties of the series (mean, variance) don't change over time. Real-world series almost always violate this with trends and seasonality.
  • Leakage is everywhere. Computing a rolling average that includes future values, scaling with statistics from the full dataset, using a feature that wouldn't be available at prediction time — time series leakage is subtle and devastatingly easy to introduce.

Treat time series as a different discipline from cross-sectional ML. The modeling tools overlap, but the validation strategy, feature engineering, and production concerns are fundamentally different.

Walk-Forward Validation — The Only Correct Way

Random K-fold cross-validation on time series data is wrong. Full stop. If fold 3 contains data from March and fold 1 contains data from June, your model in fold 1 is training on the future to predict the past. The resulting metrics are meaningless.

The correct approach is walk-forward validation (also called time series cross-validation or expanding/sliding window validation):

  1. Train on months 1–6. Test on month 7.
  2. Train on months 1–7. Test on month 8.
  3. Train on months 1–8. Test on month 9.
  4. … continue until you run out of data.

At every step, you only train on the past and evaluate on the immediate future — exactly what happens in production. The average of all test-fold scores gives you a realistic estimate of how your model will perform on unseen future data.

Expanding window (shown above) uses all available history. Sliding window uses only the most recent N months, dropping old data. Use sliding window when you believe old data is no longer relevant (e.g., consumer behavior shifts over time).

from sklearn.model_selection import TimeSeriesSplit

tscv = TimeSeriesSplit(n_splits=5)
for train_idx, test_idx in tscv.split(X):
    X_train, X_test = X.iloc[train_idx], X.iloc[test_idx]
    y_train, y_test = y.iloc[train_idx], y.iloc[test_idx]
    # Train on past, evaluate on next period — always forward in time

Classical Approaches (Know They Exist)

Before reaching for ML, you should know the classical baselines. They're often sufficient for simple univariate forecasting, and they make excellent sanity checks — if your fancy model can't beat ARIMA, something is wrong.

ARIMA

AutoRegressive Integrated Moving Average — three components packed into one model:

  • AR (AutoRegressive): The prediction is a linear combination of past values. "Tomorrow's temperature is 0.8 × today's + 0.1 × yesterday's."
  • I (Integrated): Differencing to achieve stationarity. Instead of modeling the raw values, model the changes. First difference: y(t) - y(t-1). If that's not stationary, difference again.
  • MA (Moving Average): The prediction incorporates past forecast errors. "My model overshot by 3 yesterday, so adjust down today."

ARIMA(p, d, q) has three parameters: p lags for AR, d differencing orders, q lags for MA. The seasonal variant SARIMA adds seasonal components. pmdarima's auto_arima will find good parameters automatically — use it instead of guessing.

Exponential Smoothing

A weighted average of all past observations where recent observations get exponentially more weight. Simple exponential smoothing handles level, Holt's method adds trend, and Holt-Winters adds seasonality. These are reliable baselines for business forecasting with clear seasonal patterns.

The practical takeaway: always compare your ML model against these baselines. If gradient boosted trees can't beat exponential smoothing on your daily sales forecast, the ML complexity isn't earning its keep.

The ML Approach — Feature Engineering Is the Key

Here's the core insight that unlocks ML for time series: reframe the forecasting problem as a supervised learning problem by engineering features that capture temporal patterns. Once you've done that, you can use any tabular model — and LightGBM works brilliantly here.

Lag Features

The most fundamental time series features. If you're predicting y(t), create features y(t-1), y(t-2), … y(t-k). How many lags? Depends on the autocorrelation structure. Plot the autocorrelation function (ACF) and include lags that show significant correlation.

# Create lag features
for lag in [1, 2, 3, 7, 14, 28]:
    df[f"lag_{lag}"] = df["target"].shift(lag)

Rolling Statistics

Capture recent trends and volatility. Rolling mean smooths noise; rolling standard deviation captures volatility spikes; rolling min/max captures recent extremes.

# Rolling window features
for window in [7, 14, 30]:
    df[f"rolling_mean_{window}"] = df["target"].rolling(window).mean()
    df[f"rolling_std_{window}"] = df["target"].rolling(window).std()

Calendar Features

Time itself carries signal. Day of week captures weekly patterns (restaurant sales spike on weekends). Month captures seasonality (retail spikes in December). Is_holiday is often the single strongest feature for business data.

# Calendar features
df["day_of_week"] = df["date"].dt.dayofweek
df["month"] = df["date"].dt.month
df["quarter"] = df["date"].dt.quarter
df["is_weekend"] = df["day_of_week"].isin([5, 6]).astype(int)

Trend and Seasonality Decomposition

Use statsmodels.tsa.seasonal_decompose to break the series into trend, seasonal, and residual components. You can feed these as features, or detrend the series first and let your model handle the residuals. Tree-based models handle trend poorly (they can't extrapolate beyond training data range), so detrending often helps.

Critical warning: Every feature must be computed using only past data at prediction time. A rolling mean centered on the current point uses future data — that's leakage. Always use .shift(1) or backward-looking windows.

Stationarity

A stationary series has statistical properties — mean, variance, autocorrelation — that don't change over time. Most classical models (ARIMA, exponential smoothing) require or assume stationarity. ML models are more flexible, but even they benefit from stationary inputs.

Check stationarity with the Augmented Dickey-Fuller (ADF) test:

from statsmodels.tsa.stattools import adfuller

result = adfuller(series)
print(f"ADF statistic: {result[0]:.4f}")
print(f"p-value: {result[1]:.4f}")
# p < 0.05 → reject null hypothesis → series IS stationary

If the series isn't stationary, make it so via differencing (subtract previous value: y(t) - y(t-1)) or detrending (fit and remove a trend line). Log-transforming first can stabilize variance if it grows with the level.

Seasonality and Trend

Most real-world time series have both a trend (long-term direction: sales growing 5% per year) and seasonality (repeating patterns: sales spike every December). Understanding these components is crucial because different models handle them differently.

Decompose the series to see what you're working with:

from statsmodels.tsa.seasonal import seasonal_decompose

result = seasonal_decompose(series, model="additive", period=12)
# result.trend, result.seasonal, result.resid

Additive decomposition: series = trend + seasonal + residual. Use when seasonal variation is roughly constant over time. Multiplicative: series = trend × seasonal × residual. Use when seasonal variation grows proportional to the trend (e.g., sales seasonality gets bigger as the business grows).

Tree-based models can learn seasonality from calendar features, but they fundamentally cannot extrapolate trends — they can only predict values within the range they've seen during training. If your data has an upward trend, a tree model will plateau at the maximum training value. Handle this by detrending before modeling, or by predicting differences instead of levels.

Prophet — Quick Business Forecasts

Facebook's Prophet decomposes a time series into trend + seasonality + holidays and fits each component. It's designed for business time series with daily observations and strong seasonal effects.

The good: excellent defaults, handles missing data and outliers gracefully, built-in holiday effects, uncertainty intervals out of the box, and analysts with minimal ML background can use it productively. The bad: it's not a general-purpose forecasting tool. It struggles with high-frequency data (minute-level), doesn't handle multivariate series well, and can be outperformed by gradient boosted trees on complex datasets.

Use Prophet when you need a quick, defensible forecast for business stakeholders and don't have time for feature engineering. Use ML when you have multiple input signals and need maximum accuracy.

Deep Learning for Time Series

LSTMs had their moment as the go-to deep learning model for sequences. Here's the uncomfortable truth: on most tabular time series datasets (the kind with rows and columns of features), gradient boosted trees beat LSTMs. The deep learning overhead — architecture design, GPU requirements, longer training, more hyperparameters — rarely pays off when your data is structured.

Where deep learning does shine for time series:

  • Complex multivariate series with many interacting channels (hundreds of sensors, correlated financial instruments)
  • Very long sequences where capturing distant dependencies matters
  • Raw signal data (audio, sensor waveforms) where automatic feature extraction is valuable

Temporal Fusion Transformers (TFT) are the current state-of-the-art for complex multivariate forecasting. They combine attention mechanisms with explicit handling of static covariates, known future inputs (like holidays), and varying-length histories. They also provide interpretable attention weights showing which past time steps and features mattered for each prediction.

The practical advice: start with LightGBM + engineered features. Only reach for deep learning if you have a genuinely complex multivariate problem, sufficient data (tens of thousands of time steps), and the engineering resources to support it.

Production Gotchas

Time series models in production face challenges that cross-sectional models don't. Get these wrong and your model will silently degrade.

Concept Drift

The relationship between features and target changes over time. A model trained on pre-COVID data is useless for predicting post-COVID behavior. Consumer preferences shift. Market dynamics evolve. Your model trained on the past is predicting a future that increasingly doesn't resemble its training data.

The fix: retrain regularly. Monitor prediction error over time. When error starts climbing, retrain on recent data. Some teams retrain daily or weekly as a matter of course.

Forecasting Horizon Degradation

Predicting 1 day ahead is much easier than predicting 30 days ahead. Model accuracy degrades as the forecast horizon extends — and it degrades non-linearly. A model that's 95% accurate at 1-day ahead might be only 60% accurate at 30-day ahead. Always report performance per horizon, not just the average.

Data Leakage (The Time Series Special)

Leakage in time series is insidious because it's so easy to accidentally use future information:

  • Computing features with a centered (not backward-looking) rolling window
  • Scaling the entire dataset before splitting (test set statistics leak into training)
  • Using features that wouldn't be available at prediction time (e.g., "end-of-month total" when predicting mid-month)
  • Random train/test split instead of temporal split

Rule: At every point in time, ask "would I have this information when making this prediction in production?" If the answer is no, it's leakage.

Scaling

Scale train and test sets separately. Fit the scaler on training data only, then transform both train and test. If you fit the scaler on the full dataset, the test set's mean and variance leak into training. This applies to any stateful transformation — scaling, encoding, imputation.

LightGBM on Time Series — The Full Pattern

This is the workhorse approach: engineer temporal features, then throw LightGBM at it with walk-forward validation. It wins a surprising number of forecasting competitions.

import pandas as pd
import numpy as np
import lightgbm as lgb
from sklearn.model_selection import TimeSeriesSplit
from sklearn.metrics import mean_absolute_error

def create_features(df):
    df = df.copy()
    for lag in [1, 7, 14, 28]:
        df[f"lag_{lag}"] = df["y"].shift(lag)
    for w in [7, 14, 30]:
        df[f"roll_mean_{w}"] = df["y"].shift(1).rolling(w).mean()
    df["dow"] = df["date"].dt.dayofweek
    df["month"] = df["date"].dt.month
    return df.dropna()

df = create_features(df)
features = [c for c in df.columns if c not in ["date", "y"]]

tscv = TimeSeriesSplit(n_splits=5)
scores = []
for train_idx, test_idx in tscv.split(df):
    tr, te = df.iloc[train_idx], df.iloc[test_idx]
    model = lgb.LGBMRegressor(n_estimators=500, learning_rate=0.05, num_leaves=31)
    model.fit(tr[features], tr["y"],
              eval_set=[(te[features], te["y"])],
              callbacks=[lgb.early_stopping(50, verbose=False)])
    preds = model.predict(te[features])
    scores.append(mean_absolute_error(te["y"], preds))

print(f"Walk-forward MAE: {np.mean(scores):.4f} ± {np.std(scores):.4f}")

Note the .shift(1) inside roll_mean — that ensures rolling statistics don't include the current day's value. Without it, you'd be leaking the target into the features.
