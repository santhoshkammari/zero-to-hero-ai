Support Vector Machines

Chapter 5: Supervised Learning Margins · Kernels · Interview classic

Finding the Widest Street Between Classes

Picture two groups of points on a flat surface — red and blue — and you need to draw a line separating them. There are infinitely many lines that could work. Which one should you pick?

SVM's answer: the line that maximizes the margin — the distance between the decision boundary and the nearest data points from each class. Think of it as drawing the widest possible street between the two neighborhoods. The wider the street, the more confident you are that new points falling on either side are correctly classified.

Those nearest data points — the ones sitting right on the edge of the street — are called support vectors. Here's the critical insight: only the support vectors determine the boundary. You could throw away every other training example and the decision boundary wouldn't change. This makes SVMs surprisingly memory-efficient — the model only needs to remember a handful of critical points, not the entire dataset.

In higher dimensions, the "line" becomes a hyperplane — a flat (n-1)-dimensional surface cutting through n-dimensional space. The math generalizes perfectly; the intuition stays the same.

Hard Margin vs. Soft Margin

The clean "widest street" story assumes the data is perfectly linearly separable. Real data almost never is. You've got noise, outliers, and genuine overlap between classes.

Hard margin SVM insists on zero misclassifications. If the data isn't linearly separable, the optimization simply fails — no solution exists. Even when data is separable, a single outlier can dramatically narrow the margin, making the model fragile.

Soft margin SVM (what you actually use in practice) introduces slack variables that allow some points to violate the margin or even land on the wrong side. The trade-off is controlled by the C parameter:

  • High C — "I really don't want misclassifications." The model pays a heavy penalty for violations, resulting in a narrow margin that hugs the training data tightly. More complex boundary, lower bias, higher variance.
  • Low C — "Some violations are fine if I get a wider margin." The model tolerates errors in exchange for a simpler, more generalizable boundary. Higher bias, lower variance.

C is your bias-variance knob. It plays the same role as the regularization strength λ in ridge regression — just inverted. High C is like low regularization (complex model); low C is like high regularization (simple model). This is a guaranteed interview question, so burn it into memory.

The Kernel Trick — The Key Insight

Here's where SVMs get genuinely clever. What if your data isn't linearly separable in its original space? A circle of blue points surrounded by a ring of red points — no line will ever work.

The classic solution: map the data to a higher-dimensional space where it becomes linearly separable. If you add a feature z = x² + y², those concentric circles in 2D suddenly separate with a flat plane in 3D. The math always works — given enough dimensions, any finite dataset becomes linearly separable.

The problem? Computing coordinates in that high-dimensional space can be astronomically expensive. If your kernel maps to a million dimensions, you'd need to compute and store million-dimensional vectors for every data point.

The kernel trick sidesteps this entirely. SVM's optimization only ever needs dot products between data points — it never needs the actual coordinates. A kernel function K(x₁, x₂) computes the dot product in the high-dimensional space directly from the original coordinates, without ever performing the mapping. You get all the power of the high-dimensional space at a fraction of the cost.

This isn't just an optimization hack — it's mathematically beautiful. The kernel function implicitly defines the feature space, and for some kernels, that implicit space is infinite-dimensional. You're performing linear classification in a space with infinitely many features, and it's computationally tractable.

Common Kernels

KernelImplicit spaceWhen to use
Linear
K(x,y) = x·y		 Original space (no mapping) Linearly separable data. High-dimensional sparse data like text (TF-IDF vectors). When n_features > n_samples.
RBF (Gaussian)
K(x,y) = exp(-γ‖x-y‖²)		 Infinite dimensions The default choice. Most versatile. Works for most non-linear problems when you don't know what else to try.
Polynomial
K(x,y) = (γ·x·y + r)^d		 All polynomial terms up to degree d When you suspect interaction features matter. Degree 2 or 3 is typical — higher degrees overfit and are numerically unstable.

If you're unsure, start with RBF. If your features are sparse and high-dimensional (like text), use linear. You'll rarely need polynomial in practice.

RBF Kernel Parameters: C and Gamma Together

With the RBF kernel, you have two knobs that jointly control the model's complexity:

gamma (γ) controls the "reach" of each training example's influence:

  • High gamma — each point only influences its immediate neighborhood. The decision boundary can twist and curve around individual points. Think of it as each support vector casting a narrow spotlight. Complex boundary → overfitting risk.
  • Low gamma — each point influences a wide area. The boundary becomes smooth and gentle. Each support vector casts a flood light. Simple boundary → underfitting risk.

C and gamma interact — and this is where people get tripped up. High C + high gamma = a model that desperately tries to classify every training point correctly with an extremely flexible boundary. This is overfitting city. Low C + low gamma = a very smooth, very tolerant model that might underfit.

The practical approach: grid search over both simultaneously. A typical grid:

# The classic SVM grid search
C_range = [0.01, 0.1, 1, 10, 100, 1000]
gamma_range = [0.0001, 0.001, 0.01, 0.1, 1, 10]
# That's 36 combinations — completely manageable with cross-validation

Use logarithmic scales. The optimal values often span several orders of magnitude, so linear grids waste your time testing values in a narrow range.

SVMs for Regression (SVR)

SVMs can do regression too, though it's less common. Support Vector Regression (SVR) uses an epsilon-insensitive loss: errors smaller than ε are completely ignored. The model only cares about points that fall outside the ε-tube around the prediction.

Think of it as fitting a tube of width 2ε around the data. Points inside the tube contribute zero loss — the model treats them as "close enough." Only the points outside the tube (the support vectors, again) shape the regression function.

SVR shares the same kernel trick advantages as SVC, but in practice, gradient boosted trees and neural networks handle most regression tasks better. You'll encounter SVR more in textbooks and interviews than in production code.

Strengths and Limitations

StrengthsLimitations
Effective in high-dimensional spaces, even when dimensions > samples Slow on large datasets — training is O(n²) to O(n³). 100k samples is painful; 1M is impractical
Memory-efficient: only stores support vectors, not entire dataset Feature scaling is mandatory — SVMs compute distances, so unscaled features with large ranges will dominate. Always standardize first.
The kernel trick is mathematically elegant and genuinely powerful Doesn't output probabilities natively. probability=True in sklearn uses Platt scaling — it's a post-hoc calibration, not native confidence
Works well with clear margins and medium-sized datasets Choosing the right kernel and tuning C/gamma requires careful cross-validation
Robust to overfitting in high-dimensional space (with proper regularization) Not interpretable — the decision boundary in kernel space is a black box

When to Use SVMs Today

Let's be honest: SVMs have largely been overtaken by gradient boosted trees (for tabular data) and neural networks (for everything else). But they still earn their keep in specific scenarios:

  • Text classification — A linear SVM on TF-IDF features is still one of the best approaches for document classification when you don't have GPU resources or enough data for transformers. It's fast, effective, and embarrassingly simple.
  • Small datasets with clear margins — When you have hundreds to low thousands of samples, SVMs can outperform tree ensembles because the margin-maximizing objective provides strong regularization.
  • High-dimensional, low-sample data — Genomics, spectroscopy, and other domains where you have more features than samples. SVMs handle this gracefully.
  • Interviews — SVMs test your understanding of margins, kernels, the bias-variance tradeoff, and the mathematics of optimization. Know them cold.

For most other problems — especially anything with more than ~50k samples — reach for LightGBM or XGBoost instead. They're faster, easier to tune, handle mixed feature types, and generally more accurate.

SVM in Code

The standard pattern: always scale features, always use a pipeline so the scaler is fitted on training data only.

from sklearn.svm import SVC
from sklearn.pipeline import make_pipeline
from sklearn.preprocessing import StandardScaler
from sklearn.model_selection import GridSearchCV

# Pipeline: scale first, then SVM — never scale outside the pipeline
svm_pipe = make_pipeline(
    StandardScaler(),
    SVC(kernel="rbf")
)

param_grid = {
    "svc__C": [0.1, 1, 10, 100],
    "svc__gamma": ["scale", 0.01, 0.1, 1],
}

search = GridSearchCV(svm_pipe, param_grid, cv=5, scoring="accuracy", n_jobs=-1)
search.fit(X_train, y_train)
print(f"Best params: {search.best_params_}")
print(f"Test accuracy: {search.score(X_test, y_test):.4f}")

Note the scaler inside the pipeline — this prevents data leakage. If you scale before splitting, your test set statistics leak into the training process. It's a subtle bug that inflates your scores.
