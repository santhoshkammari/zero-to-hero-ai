Section Overview

This section combines k-Nearest Neighbors and Naive Bayes into one comprehensive reference.

k-Nearest Neighbors

The Whole Algorithm in One Sentence

To classify a new point, find the k training points closest to it and let them vote. Majority class wins. For regression, take the average of those k neighbors' values. That's it — there's no training phase, no learned weights, no optimization. The model is the data.

This makes KNN what's called a lazy learner: it memorizes the entire training set and does all the work at prediction time. Compare that to logistic regression, which grinds through the data once to learn coefficients and then makes predictions in microseconds. KNN has zero training cost and pays for it later on every single prediction.

Distance Matters More Than You Think

"Nearest" implies distance, and the choice of distance metric shapes everything. The three you'll encounter:

  • Euclidean (L2): straight-line distance. The default, and usually fine. sqrt(Σ(xᵢ − yᵢ)²)
  • Manhattan (L1): sum of absolute differences along each axis. More robust to outliers in individual features. Σ|xᵢ − yᵢ|
  • Minkowski (Lp): generalizes both — Euclidean is p=2, Manhattan is p=1. Higher p weighs large single-feature differences more heavily.

Here's the critical point: feature scaling is mandatory. If one feature is "annual income" (range 20,000–200,000) and another is "age" (range 18–80), income will utterly dominate every distance calculation. Age becomes invisible. You must standardize or min-max scale all features before computing distances. No exceptions. Forget this and your KNN model is really a "nearest income" model.

Choosing k

The value of k is your bias-variance dial:

  • k = 1: the decision boundary follows every training point exactly. Extreme variance, zero bias on training data, overfits badly to noise. One mislabeled point poisons all predictions near it.
  • k = n (all training data): every point gets the same prediction — the majority class in the entire dataset. Maximum bias, zero variance. Completely useless.
  • Sweet spot: somewhere in between. Typically k = 5 to 15 for most problems, but always choose via cross-validation.

Practical tip: for binary classification, use odd k to avoid ties. With k = 4 and a 2-vs-2 split, the algorithm has to break the tie arbitrarily. Odd k guarantees a clear majority.

The Curse of Dimensionality

KNN works by assuming nearby points are similar. That assumption breaks in high dimensions.

Here's the intuition: in 2D, the nearest neighbor to a point is actually nearby. In 1,000 dimensions, all points are roughly the same distance apart. The "nearest" neighbor is barely closer than the farthest point. When distances become meaningless, voting on neighbors becomes meaningless too.

The math is stark: in d dimensions, the volume of a hypersphere grows so slowly relative to the enclosing hypercube that to capture just 1% of the data within a neighborhood, you need to extend that neighborhood to cover most of each feature's range. By ~20 features, KNN is struggling. Past 50, it's usually hopeless without dimensionality reduction (PCA, feature selection) first.

Computational Cost

Every prediction requires comparing the new point against every training point. That's O(n · d) per prediction — n points, d features each. Train on a million samples? Every single prediction scans a million distances.

MethodComplexityWhen to use
Brute forceO(n · d) per querySmall datasets, high d
KD-treeO(d · log n) averageLow dimensions (d < ~20)
Ball treeO(d · log n) averageModerate dimensions, non-Euclidean metrics

KD-trees partition space with axis-aligned splits, so they degrade to brute force in high dimensions. Ball trees use hyperspheres, handling moderate d better. sklearn picks the algorithm automatically (algorithm='auto'), but knowing this helps you understand why KNN prediction slows down as your dataset grows.

When to Use KNN (and When Not To)

Good fits: small datasets (< 10k samples), quick baselines, prototyping, low-dimensional data, problems where you want locally interpretable decisions ("this patient is similar to these five other patients who all recovered").

Bad fits: large datasets (prediction is painfully slow), high-dimensional feature spaces (curse of dimensionality), when you need a compact deployable model (KNN has to carry the entire training set), or when features have very different types/scales that are hard to combine into a single distance metric.

Code: KNN Pipeline with Scaling

from sklearn.neighbors import KNeighborsClassifier
from sklearn.preprocessing import StandardScaler
from sklearn.pipeline import Pipeline
from sklearn.datasets import load_iris
from sklearn.model_selection import cross_val_score

X, y = load_iris(return_X_y=True)

pipe = Pipeline([
    ('scaler', StandardScaler()),            # mandatory for KNN
    ('knn', KNeighborsClassifier(n_neighbors=5))
])

scores = cross_val_score(pipe, X, y, cv=5)
print(f"Accuracy: {scores.mean():.3f} ± {scores.std():.3f}")
# Accuracy: 0.967 ± 0.022

The pipeline ensures scaling happens inside each CV fold — no data leakage. Swap n_neighbors and re-run to see the bias-variance tradeoff in action.

Naive Bayes

Bayes' Theorem as a Classifier

You've seen Bayes' theorem in a probability class. Here's the version that matters for classification:

P(class | features) ∝ P(features | class) × P(class)

Read it as: the probability that a sample belongs to a class, given its features, is proportional to how likely those features are within that class, times how common the class is overall. We don't need the denominator P(features) because it's the same for every class — it's just a normalizing constant. We compare the numerators and pick the class with the biggest one.

Here's the problem. Computing P(features | class) for a realistic feature vector is impossible without making assumptions. If you have 1,000 binary features, there are 2¹⁰⁰⁰ possible feature combinations. You'll never see enough data to estimate those probabilities directly.

The "naive" fix: assume all features are conditionally independent given the class. That lets you factorize the joint probability into a product of individual feature probabilities:

P(x₁, x₂, ..., x_d | class) = P(x₁ | class) × P(x₂ | class) × ... × P(x_d | class)

Now you only need to estimate d individual probabilities per class instead of one enormous joint distribution. That's tractable even with small data.

Why "Naive" Works Anyway

The independence assumption is almost always wrong. In a spam filter, "free" and "money" aren't independent — they co-occur in the same scammy emails far more often than chance would predict. In medical data, symptoms cluster. In image data, adjacent pixels are correlated. So why does Naive Bayes still work?

Because classification only requires getting the ranking right, not the exact probabilities. As long as the true class ends up with a higher score than the false classes, you win — even if that score is wildly miscalibrated. And it turns out that when you multiply a bunch of slightly wrong individual probabilities together, the errors often partially cancel or at least preserve the correct ordering.

Think of it this way: if every feature independently "leans" toward the right class, multiplying those leans together still points in the right direction — even if the magnitude is off. The decision boundary is what matters, not whether you predict 0.92 or 0.73 for the true class.

That said, the probability values themselves are usually garbage. Don't trust a Naive Bayes probability of 0.99 as genuinely meaning 99% confidence. If you need calibrated probabilities, use logistic regression or apply post-hoc calibration (Platt scaling, isotonic regression).

Three Variants: Match the Distribution to Your Data

VariantAssumes features are…Best for
GaussianNBContinuous, normally distributedGeneral tabular data with real-valued features
MultinomialNBNon-negative countsText classification (word counts, TF-IDF scores)
BernoulliNBBinary (0/1)Binary feature vectors, short text (word presence/absence)

GaussianNB estimates a mean and variance per feature per class, then plugs new values into the Gaussian PDF. Simple, but if your features aren't remotely normal (heavy tails, multimodal), performance degrades. No hyperparameters to tune — which is either a strength or a limitation depending on your perspective.

MultinomialNB models each class as a multinomial distribution over feature counts. This is the workhorse for text: each feature is a word, each value is how many times that word appears. The model learns which words are common in which classes. It's fast, simple, and remarkably effective.

BernoulliNB cares only about whether a feature is present or absent, ignoring how many times it appears. For text, this means "does the word occur at all?" rather than "how often?" It penalizes the absence of features explicitly — unlike MultinomialNB, which simply ignores zero-count features. This makes BernoulliNB better for short documents where word presence is more informative than frequency.

Text Classification: The Killer App

Naive Bayes + TF-IDF is one of those combinations that has no right to work as well as it does. It's the classic spam filter. It's a strong baseline for sentiment analysis, topic classification, language detection, and document categorization. On small text datasets (hundreds to low thousands of documents), it often beats neural networks and gradient-boosted trees.

Why? Text data is naturally high-dimensional (thousands of unique words) and sparse (each document uses a tiny fraction of the vocabulary). Most models choke on this combination. Naive Bayes thrives — it only needs per-word statistics, so sparse data is fine. The independence assumption hurts less than you'd think because TF-IDF already down-weights common words, reducing the most obvious correlations.

The practical rule: always start a text classification project with MultinomialNB + TF-IDF. If it gives you 88% accuracy in five minutes, you now have a baseline that tells you exactly how much improvement your fancy BERT model needs to justify its training cost.

Laplace Smoothing

Here's a catastrophic failure mode. During training, the word "cryptocurrency" never appeared in any legitimate (ham) email. So P("cryptocurrency" | ham) = 0. Now a ham email with the word "cryptocurrency" arrives. The entire product goes to zero — P(ham | features) = 0 — regardless of every other feature. One unseen word kills the prediction.

Laplace smoothing (controlled by the alpha parameter) fixes this by adding a small count to every feature:

P(word | class) = (count(word, class) + α) / (total_words_in_class + α × vocabulary_size)

With α = 1 (the standard "add-one" smoothing), no probability is ever zero. A word never seen in a class gets a tiny but non-zero probability instead of zero. Larger α values smooth more aggressively, shrinking all probabilities toward uniform. Smaller α values trust the data more. In practice, α = 1 is the default and works well; occasionally tuning it via cross-validation helps.

Strengths

  • Fast — both training and prediction. Training is a single pass through the data computing counts and means. Prediction is a dot product plus a log. Scales to millions of samples and features effortlessly.
  • Works with tiny datasets. Because it estimates so few parameters per class, it generalizes reasonably even with dozens of training examples per class.
  • Handles high-dimensional data. While KNN breaks above ~20 features, Naive Bayes handles 50,000 text features without blinking.
  • Naturally multiclass. No OVR or OVO wrappers needed — the Bayesian formulation handles any number of classes natively.
  • Great baseline. If Naive Bayes gets 85% on your problem, you know there's real signal. If it gets 51%, your features might be the problem, not your model.

Weaknesses

  • Poor probability calibration. The probabilities are often pushed toward 0 and 1 due to the independence assumption multiplying many small numbers together. Use the predictions for ranking, not for interpreting confidence.
  • Correlated features get double-counted. If two features carry the same information (like "height in cm" and "height in inches"), Naive Bayes counts that evidence twice. This doesn't break the ranking for mildly correlated features, but heavy redundancy degrades performance.
  • Can't learn feature interactions. "Feature A is only predictive when Feature B is high" — Naive Bayes can never capture this. Each feature contributes independently. Tree-based models and neural networks handle interactions naturally.
  • Gaussian assumption can mislead. GaussianNB assumes features are normally distributed within each class. Real data often isn't. MultinomialNB and BernoulliNB are more robust because their distributional assumptions better match their target domains (text counts, binary indicators).

Code: Text Classification with MultinomialNB

from sklearn.naive_bayes import MultinomialNB
from sklearn.feature_extraction.text import TfidfVectorizer
from sklearn.pipeline import Pipeline
from sklearn.datasets import fetch_20newsgroups
from sklearn.metrics import classification_report

train = fetch_20newsgroups(subset='train')
test  = fetch_20newsgroups(subset='test')

pipe = Pipeline([
    ('tfidf', TfidfVectorizer(stop_words='english', max_df=0.5)),
    ('nb',    MultinomialNB(alpha=1.0))
])

pipe.fit(train.data, train.target)
preds = pipe.predict(test.data)
print(classification_report(test.target, preds,
                            target_names=train.target_names,
                            zero_division=0))

That's 20 newsgroups — 20 classes, ~18,000 documents, tens of thousands of features — classified in under a second. Try beating that speed with a transformer. The accuracy won't be state-of-the-art, but as a baseline you can build in five minutes, it's hard to argue with.
