k-Nearest Neighbors (k-NN)

Chapter 5: Supervised Learning 6 subtopics
TL;DR

k-NN is the simplest idea in machine learning: to predict something about a data point, find the k most similar points in your training set and let them vote. No training phase, no learned parameters — just a lookup at prediction time. It works beautifully on small, low-dimensional datasets with non-linear boundaries, but collapses under high dimensions and large data. Feature scaling is non-negotiable, and choosing k is the main knob you turn.

Algorithm Fundamentals

The Core Intuition

Imagine you're trying to predict the price of a house. You don't have a fancy model. But you know the prices of every other house in the neighborhood. So you do the obvious thing: you find the five houses most similar to yours — similar square footage, similar number of bedrooms, similar lot size — and you average their prices. That's k-NN. If a house looks like other expensive houses nearby, it's probably expensive too.

The same logic works for classification. You want to know if a tumor is malignant or benign. You measure its features — size, texture, symmetry — and find the five most similar tumors in your database. If four of them are benign and one is malignant, you predict benign. Majority vote wins.

That's the entire algorithm. No equations to solve, no parameters to optimize during training. You just store the data, and when a new point arrives, you look up its neighbors.

Instance-Based (Lazy) Learning

Most machine learning algorithms have a clear training phase. Linear regression learns coefficients. Decision trees learn split rules. Neural networks learn weights. They compress the training data into a compact model, then throw the data away.

Instance-based learning (also called lazy learning) does the opposite. It skips the training phase entirely. It stores every training example in memory and defers all computation to prediction time. k-NN is the canonical example. The "model" is the data.

This has a profound tradeoff. Training is instantaneous — O(1), literally just copying data into memory. But prediction is expensive — for every new point, you must scan the entire dataset to find the nearest neighbors. With n training points and d features, a brute-force search costs O(n · d) per prediction. That's fine for a thousand points. It's a nightmare for a million.

Distance Metrics

The entire algorithm hinges on one question: how do you define "similar"? The answer is a distance metric — a function that takes two points and returns a number representing how far apart they are.

Euclidean distance is the default. It's the straight-line distance between two points, the one you learned in geometry class. For two points with features (x₁, x₂, ..., x_d) and (y₁, y₂, ..., y_d):

distance = sqrt( (x₁ - y₁)² + (x₂ - y₂)² + ... + (x_d - y_d)² )

It works well when features are continuous and roughly on the same scale. It's sensitive to large values — one feature with big numbers can dominate the distance calculation.

Manhattan distance sums the absolute differences along each axis instead of squaring them. Think of walking along city blocks — you can't cut diagonally. It's more robust to outliers because it doesn't square the differences.

Minkowski distance is the generalization. With parameter p, it includes Euclidean (p=2) and Manhattan (p=1) as special cases. In practice, p=2 is the default, and you rarely need to change it.

For categorical or mixed data, you might use Hamming distance (counts how many features differ) or Gower distance (handles mixed types). The right metric depends on your data, but Euclidean handles most continuous-feature problems well.

Choosing k

k is the one hyperparameter that matters most. It controls the bias-variance tradeoff directly.

With k=1, the prediction is determined by the single nearest neighbor. This is maximally flexible — the decision boundary follows every twist and bump in the data. But it's also maximally noisy. One mislabeled point or one outlier will cause a wrong prediction. High variance, low bias.

With k=100 (or some large value), you're averaging over many neighbors. The decision boundary becomes smooth, almost like a linear model. You're immune to noise, but you've lost the ability to capture local patterns. Low variance, high bias.

The sweet spot is somewhere in between. Rules of thumb: start with k = √n (where n is the number of training points), use odd values for binary classification to avoid ties, and — most importantly — use cross-validation to pick the best k. Try k = 1, 3, 5, 7, ..., 21, and pick the one with the lowest validation error. It's a cheap search because k-NN has no training phase.

Weighted k-NN

Standard k-NN gives every neighbor an equal vote. But intuitively, a neighbor that's very close should have more influence than one that's barely inside the k-nearest circle.

Weighted k-NN solves this by weighting each neighbor's vote by the inverse of its distance. A neighbor at distance 0.1 gets 10× the weight of a neighbor at distance 1.0. This makes predictions more robust, especially when k is large and some neighbors are much farther away than others.

In scikit-learn, this is a single parameter: weights='distance' instead of the default weights='uniform'. It almost always helps and rarely hurts. Use it by default.

Computational Considerations

Feature Scaling Is Non-Negotiable

This is the single most common mistake people make with k-NN. Let me show you why with a concrete example.

Suppose you're predicting whether someone will default on a loan. You have two features: annual income (ranging from $20,000 to $200,000) and age (ranging from 18 to 70). Now compute the Euclidean distance between two people:

Person A: income = $50,000,  age = 30
Person B: income = $150,000, age = 31

distance = sqrt( (50000 - 150000)² + (30 - 31)² )
         = sqrt( 10,000,000,000 + 1 )
         ≈ 100,000

That distance is completely dominated by income. The age difference — which might be totally irrelevant — contributes exactly 1 to the distance. Income contributes 10 billion. The algorithm is effectively ignoring age entirely.

The fix is feature scaling. Two common approaches: standardization (subtract mean, divide by standard deviation — each feature gets mean 0 and std 1) or min-max normalization (scale everything to [0, 1]). After standardization, income and age contribute proportionally to the distance.

from sklearn.preprocessing import StandardScaler

scaler = StandardScaler()
X_train_scaled = scaler.fit_transform(X_train)
X_test_scaled = scaler.transform(X_test)  # use training stats!

Critical detail: fit the scaler on training data only, then use that same scaler to transform test data. If you fit on test data, you're leaking information from the future.

The Curse of Dimensionality

k-NN works by finding nearby points. But "nearby" stops meaning anything useful in high dimensions. This is the curse of dimensionality, and it's the fundamental limitation of k-NN.

Here's the intuition. In 1 dimension, if you have 100 points evenly spread between 0 and 1, your nearest neighbor is about 0.01 away. In 2 dimensions, those same 100 points are spread across a unit square, and the nearest neighbor is about 0.1 away. In 10 dimensions, a unit hypercube with 100 points is almost entirely empty space. To keep the same density of neighbors, you'd need 100^10 = 10^20 points.

In practice, this means that in high dimensions, all points are roughly the same distance from each other. The nearest neighbor isn't meaningfully nearer than the farthest one. The concept of "neighborhood" breaks down. k-NN becomes no better than random guessing.

When does this matter? If you have more than about 20 features and a moderate dataset, k-NN will struggle. Solutions include dimensionality reduction (PCA, feature selection) before running k-NN, or switching to an algorithm that handles high dimensions natively (tree-based methods, neural networks).

KD-Trees, Ball Trees, and LSH

Brute-force neighbor search is O(n · d) per query. For large n, this is too slow. Three data structures speed things up.

A KD-tree (k-dimensional tree) recursively partitions the feature space along one axis at a time, creating a binary tree. At prediction time, you traverse the tree, pruning branches that can't contain closer neighbors. Average case: O(d · log n) per query. But in high dimensions (d > 20), the pruning becomes ineffective and KD-trees degrade to brute force.

A Ball tree partitions data into nested hyperspheres ("balls") instead of axis-aligned cuts. This works better in moderate dimensions where data has intrinsic structure — roughly 20–50 features. The construction is slower, but queries can be faster than KD-trees in these cases.

Locality-Sensitive Hashing (LSH) takes a completely different approach. Instead of exact nearest neighbors, it finds approximate nearest neighbors using hash functions that map similar points to the same bucket with high probability. It's the go-to method for very large datasets (millions of points) where you can tolerate a small error in neighbor selection. Think recommendation systems and image search at web scale.

In scikit-learn, you pick the strategy with the algorithm parameter:

KNeighborsClassifier(n_neighbors=5, algorithm='kd_tree')   # low dimensions
KNeighborsClassifier(n_neighbors=5, algorithm='ball_tree')  # moderate dimensions
KNeighborsClassifier(n_neighbors=5, algorithm='auto')       # let sklearn decide

Variants

k-NN for Classification

The standard use case. Find k nearest neighbors, each neighbor votes for its class, and the majority class wins. Ties can be broken randomly, by distance, or by picking the class of the closest neighbor. Use KNeighborsClassifier in scikit-learn.

k-NN naturally handles multi-class problems with zero extra effort. With 10 classes, each neighbor votes for one of the 10, and the class with the most votes wins. No one-vs-rest gymnastics, no softmax, no special loss function. This is a genuine advantage over algorithms like logistic regression or SVMs that require special handling for more than two classes.

k-NN for Regression

Instead of voting, average the neighbors' target values. Want to predict house prices? Find the 5 nearest houses and average their prices. The weighted variant (inverse distance weighting) is especially useful here — nearby houses should matter more.

from sklearn.neighbors import KNeighborsRegressor

reg = KNeighborsRegressor(n_neighbors=5, weights='distance')
reg.fit(X_train_scaled, y_train)
predictions = reg.predict(X_test_scaled)

Radius-Based Neighbors

Radius-based neighbors flips the approach: instead of fixing k and finding the k nearest points, you fix a radius r and use all points within distance r. In a dense neighborhood, you might use 50 neighbors. In a sparse one, maybe 3. This adapts naturally to varying data density.

The downside: if no points fall within the radius, you have no neighbors and no prediction. You need to handle this edge case (scikit-learn raises an error by default). Use RadiusNeighborsClassifier or RadiusNeighborsRegressor.

Condensed and Edited Nearest Neighbors

Storing the entire training set is wasteful. Most points in the interior of a class don't affect predictions — only the points near the decision boundary matter.

Condensed Nearest Neighbors (CNN) shrinks the training set by keeping only the points needed to correctly classify the rest. Start with one point per class, then iteratively add any training point that the current condensed set misclassifies. You end up with a much smaller reference set that produces the same predictions as the full dataset. Faster predictions, same accuracy.

Edited Nearest Neighbors (ENN) goes the other direction: it removes noisy points. For each training point, check if its k nearest neighbors agree with its label. If not, remove it — it's probably noise or an outlier. The result is a cleaner dataset with smoother decision boundaries. ENN is also widely used as an undersampling technique for imbalanced datasets.

When to Use k-NN

k-NN earns its place in specific situations. Here's the honest assessment.

Small to medium datasets (up to ~100K points). Prediction speed is O(n), so it scales linearly with data size. Beyond 100K, consider approximate methods or a different algorithm entirely.

Non-linear decision boundaries. k-NN can capture arbitrary decision shapes without you specifying the form. It doesn't assume linear, quadratic, or any particular boundary shape. The data draws the boundary.

Multi-class problems. As mentioned, k-NN handles any number of classes natively. If you have 50 classes and a small dataset, k-NN is a strong baseline.

Low-dimensional data (under ~20 features). The curse of dimensionality is real. If you have hundreds of features, k-NN is almost certainly the wrong choice.

Don't use k-NN when: you have very large datasets (slow predictions), high-dimensional data (distances lose meaning), you need a compact model (the "model" is the entire training set), or you need feature importance or interpretability (k-NN tells you nothing about which features matter).

Handling Missing Data in k-NN

k-NN has no built-in mechanism for missing data. You can't compute a distance when a feature is missing. Three practical approaches:

First, impute before training. Replace missing values with the column mean, median, or mode. This is simple but introduces bias — you're making up data.

Second, ignore missing dimensions. When computing distance between two points, skip any feature that's missing in either point, and rescale the distance to account for the reduced number of features. This is principled but not supported natively in scikit-learn.

Third — and ironically — use k-NN itself for imputation. For each sample with a missing value, find its k nearest neighbors (using the non-missing features), and impute the missing value as the average of the neighbors' values for that feature. This is circular in a satisfying way: the algorithm that needs complete data can be used to create complete data. Scikit-learn provides KNNImputer for exactly this.

from sklearn.impute import KNNImputer

imputer = KNNImputer(n_neighbors=5)
X_complete = imputer.fit_transform(X_with_missing)

k-NN and Kernel Density Estimation

k-NN and Kernel Density Estimation (KDE) are two sides of the same coin. Both estimate local properties of data by looking at nearby points. The difference is what they fix and what they let vary.

k-NN fixes the number of neighbors (k) and lets the radius vary — in dense regions, the k neighbors are close; in sparse regions, they're far apart. KDE fixes the radius (the kernel bandwidth) and lets the number of points vary — dense regions capture many points, sparse regions capture few.

In classification, k-NN implicitly estimates class-conditional densities. The fraction of neighbors belonging to class A is an estimate of P(class=A | x) — the probability of class A at point x. This connects k-NN directly to Bayes' theorem and probabilistic classification. As n → ∞ and k/n → 0 (but k → ∞), the k-NN classifier converges to the Bayes-optimal classifier — the theoretical best. Few algorithms come with that kind of guarantee.

Putting It All Together: Complete Example

import numpy as np
from sklearn.datasets import load_iris
from sklearn.model_selection import cross_val_score, GridSearchCV
from sklearn.preprocessing import StandardScaler
from sklearn.neighbors import KNeighborsClassifier
from sklearn.pipeline import Pipeline

# Load data
X, y = load_iris(return_X_y=True)

# Always build a pipeline so scaling and model stay together
pipe = Pipeline([
    ('scaler', StandardScaler()),
    ('knn', KNeighborsClassifier())
])

# Search for the best k and weighting scheme
param_grid = {
    'knn__n_neighbors': [1, 3, 5, 7, 9, 11, 15, 21],
    'knn__weights': ['uniform', 'distance'],
    'knn__metric': ['euclidean', 'manhattan']
}

grid = GridSearchCV(pipe, param_grid, cv=5, scoring='accuracy')
grid.fit(X, y)

print(f"Best params: {grid.best_params_}")
print(f"Best CV accuracy: {grid.best_score_:.3f}")

# Typical output:
# Best params: {'knn__metric': 'euclidean', 'knn__n_neighbors': 7,
#               'knn__weights': 'distance'}
# Best CV accuracy: 0.980

Key points in this example: the scaler is inside the pipeline so it doesn't leak test information during cross-validation. We search over k, the weighting scheme, and the distance metric simultaneously. Five-fold cross-validation gives a reliable estimate of generalization performance. On Iris, k-NN achieves ~98% accuracy — competitive with far more complex algorithms.

k-NN won't win Kaggle competitions or power production recommendation engines at scale. But it's a powerful baseline, a useful tool for small data problems, and — most importantly — it builds the right intuition for how all of machine learning works: use similar examples to make predictions about new ones. Every algorithm you'll learn from here forward is, in some deep sense, a smarter version of the same idea.
