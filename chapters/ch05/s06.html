Ensemble Methods

Chapter 5: Supervised Learning Random Forest · XGBoost · LightGBM · The tabular ML workhorses

One Tree Is a Gamble. Many Trees Are a Strategy.

If you read the Decision Trees section, you left with a nagging problem: a single decision tree is unstable. Change a few training examples and the entire tree structure can flip. That's high variance, and it means your model's predictions are noisy.

The fix is almost embarrassingly simple: train many trees and average their predictions. Each individual tree might be wrong in its own unique way, but if their errors are uncorrelated, averaging cancels the noise. The ensemble's variance drops proportional to 1/N while the bias stays the same.

That's the entire core idea of ensemble methods. Everything else — bagging, boosting, stacking, Random Forests, XGBoost — is a variation on how you generate diverse weak learners and how you combine them. But the principle is always: diversity + aggregation = better predictions.

These methods dominate tabular data. If you're doing ML on structured data — customer churn, fraud detection, pricing, click-through rate, medical diagnosis — an ensemble of trees is your default weapon. Kaggle leaderboards have been screaming this for a decade, and production systems agree.

Bagging: Bootstrap Aggregating

Bagging was Leo Breiman's 1996 insight: if high variance is the problem, and averaging reduces variance, then just train a bunch of models on different subsets of the data and average them.

Here's the recipe:

  1. Bootstrap: From your N training examples, sample N examples with replacement. Some examples appear multiple times, some don't appear at all. On average, each bootstrap sample contains about 63.2% of the unique training examples (the rest are duplicates).
  2. Train: Fit an independent model on each bootstrap sample. Usually a deep, unpruned decision tree.
  3. Aggregate: For regression, average the predictions. For classification, take a majority vote.

Why does this work? Each tree overfits in a different direction because it saw a different subset of the data. When you average their predictions, the overfitting cancels out. You get the low bias of a deep tree with much lower variance. Bias stays the same (each individual tree is still a high-capacity model); variance drops because you're averaging.

The key insight: bagging only helps high-variance, low-bias models. Bagging a linear regression is mostly pointless — linear models are already low-variance. Bagging a deep, unpruned decision tree? That's where the magic happens.

Random Forest — Your First Call

Random Forest is bagging with one extra trick that makes it dramatically better. When each tree chooses a split, it only considers a random subset of features, not all of them. This sounds counterintuitive — why would ignoring features help?

Because it decorrelates the trees. Without feature subsampling, if one feature is very strong, every bagged tree will split on it first. All your trees end up looking similar, and averaging similar predictions doesn't reduce variance much. By forcing each tree to work with a random handful of features at each split, you get trees that are individually worse but collectively much more diverse. And diversity is what makes ensembles work.

The Mechanics

At each split in each tree:

  • Classification: randomly sample √p features (where p is total features), find the best split among those
  • Regression: randomly sample p/3 features
  • The tree is grown deep — typically no max depth limit, though min_samples_leaf=1 is the default

Each tree is fully independent of every other tree. This is embarrassingly parallel — you can train 500 trees on 500 cores and it scales linearly. Set n_jobs=-1 in scikit-learn and it'll use all your cores.

Key Hyperparameters

Out-of-Bag Score: Free Validation

Here's something beautiful about bagging. Each bootstrap sample leaves out about 36.8% of the training data (the examples that weren't sampled). For any given training example, roughly one-third of the trees never saw it during training. So you can predict that example using only the trees that didn't train on it — and you get a validation score without needing a separate holdout set.

from sklearn.ensemble import RandomForestClassifier

rf = RandomForestClassifier(n_estimators=500, oob_score=True, n_jobs=-1)
rf.fit(X_train, y_train)
print(f"OOB accuracy: {rf.oob_score_:.4f}")  # Free! No train/val split needed

The OOB score is essentially equivalent to leave-one-out cross-validation. It's a reliable estimate of generalization performance, and it comes for free. Use it during development; use proper cross-validation for final reporting.

Feature Importance: The Trap

Random Forest gives you .feature_importances_ — the mean decrease in impurity (MDI) averaged across all trees. It's convenient, but it's biased toward high-cardinality features. A feature with many unique values (like a user ID or zip code) gets more opportunities to split, so it accumulates more importance even if it's just memorizing noise.

The fix: permutation importance. Shuffle one feature's values, measure how much the model's score drops. If the feature matters, performance tanks. If it doesn't matter, performance is unchanged. This is model-agnostic and doesn't have the cardinality bias.

from sklearn.inspection import permutation_importance

result = permutation_importance(rf, X_val, y_val, n_repeats=10, n_jobs=-1)
for i in result.importances_mean.argsort()[::-1][:10]:
    print(f"{feature_names[i]:30s} {result.importances_mean[i]:.4f} ± {result.importances_std[i]:.4f}")

Rule of thumb: Use MDI for quick exploration. Use permutation importance for anything you'd show a stakeholder or use for feature selection.

When Random Forest Wins

When you have no idea what model to try, Random Forest is your first call. It handles mixed feature types, doesn't need feature scaling, is robust to outliers, tolerates some missing values (with imputation), rarely overfits with default hyperparameters, and gives you feature importance and OOB scores for free. It's not the highest-accuracy model on most problems — gradient boosting usually wins by 1-3% — but it's the safest starting point.

Boosting: Fix What You Got Wrong

Bagging trains trees independently and averages them. Boosting trains trees sequentially, where each new tree focuses on correcting the mistakes of the current ensemble. This is a fundamentally different strategy.

Bagging reduces variance (averaging uncorrelated predictors). Boosting reduces bias (iteratively reducing the residual error). That's why boosted models can outperform bagged models — they can learn patterns that no single tree, however deep, can capture alone.

The trade-off: boosting is more prone to overfitting and more sensitive to hyperparameters. You need to tune carefully. But when tuned well, gradient boosted trees are the most powerful models for structured data, period.

AdaBoost: The Original Booster

AdaBoost (Adaptive Boosting, Freund & Schapire 1997) was the first practical boosting algorithm. The idea is elegant:

  1. Train a weak learner (typically a shallow "decision stump" — one split) on the data.
  2. Look at which examples it got wrong. Increase the weight of those misclassified examples.
  3. Train the next weak learner on the reweighted data. It naturally focuses on the hard cases.
  4. Repeat. The final prediction is a weighted vote of all learners, where better learners get more vote weight.

AdaBoost is historically important and theoretically elegant, but it has practical weaknesses: it's very sensitive to outliers and noisy labels (because it keeps upweighting hard examples, which might just be mislabeled), and it's been largely superseded by gradient boosting. You should know what it is; you probably won't use it in production.

Gradient Boosting: Fit the Residuals

Gradient boosting (Friedman 2001) takes a different approach than AdaBoost. Instead of reweighting examples, each new tree literally predicts the residuals — the errors — of the current ensemble.

Think of it this way: your ensemble currently predicts ŷ. The true value is y. The residual is (y - ŷ). Train a new tree to predict that residual. Add it to the ensemble. Now your ensemble's prediction is closer to y. Repeat.

# Pseudocode for gradient boosting
F₀(x) = mean(y)                      # Start with a constant prediction
for m in 1, 2, ..., M:
    residuals = y - Fₘ₋₁(x)          # What the ensemble gets wrong
    tree_m = fit_tree(X, residuals)    # New tree predicts the errors
    Fₘ(x) = Fₘ₋₁(x) + η · tree_m(x) # Add it (with learning rate η)

Why "gradient"? Because the residuals (y - ŷ) are the negative gradient of the squared error loss. For other loss functions (log-loss, absolute error, quantile loss), you compute the actual gradient of the loss with respect to the predictions and fit trees to that. The framework generalizes to any differentiable loss function.

Learning Rate (Shrinkage)

The learning rate η (typically 0.01 to 0.1) controls how much each new tree contributes. Smaller values mean each tree makes a smaller correction, which requires more trees but generalizes better. Think of it as taking smaller, more careful steps toward the optimum instead of lunging.

The practical rule: Set the learning rate small (0.01–0.05), then increase n_estimators until validation performance plateaus. Smaller learning rates almost always generalize better — you just need more trees (and more compute) to compensate.

The Overfitting Danger

Gradient boosting is extremely powerful but easy to overfit. Unlike Random Forest (where more trees never hurts), adding too many boosting rounds will eventually memorize the training data. This is why early stopping is non-negotiable: monitor validation loss, and stop training when it stops improving. Every serious implementation supports this. Use it. Always.

XGBoost: Production-Grade Gradient Boosting

XGBoost (Extreme Gradient Boosting, Tianqi Chen 2014) took gradient boosting and made it fast, scalable, and regularized. It's not a different algorithm — it's gradient boosting done right, with engineering that matters in practice.

What Makes XGBoost Special

  • Regularization in the objective: XGBoost adds L1 (reg_alpha) and L2 (reg_lambda) penalties on the leaf weights directly into the loss function. This is built into the tree-building process, not bolted on afterwards. It means the tree grows aware of complexity cost.
  • Histogram-based splitting: Instead of considering every possible threshold, XGBoost buckets continuous features into histograms. Faster splits, lower memory, and often acts as regularization too.
  • Native missing value handling: During training, XGBoost learns which direction (left or right child) to send missing values at each split. No imputation needed — the model figures it out.
  • Column subsampling: Like Random Forest, XGBoost can randomly sample features per tree or per split (colsample_bytree, colsample_bylevel). Decorrelates trees and acts as regularization.
  • GPU training: tree_method="gpu_hist" and you're on the GPU. 10-50× speedup for large datasets.
  • Distributed training: Scales across clusters via Spark, Dask, or Ray.

Early Stopping in Practice

import xgboost as xgb

model = xgb.XGBClassifier(
    n_estimators=10000,        # Set high — early stopping will find the right number
    learning_rate=0.05,
    max_depth=6,
    subsample=0.8,
    colsample_bytree=0.8,
    early_stopping_rounds=50,  # Stop if no improvement for 50 rounds
    eval_metric="logloss",
)

model.fit(
    X_train, y_train,
    eval_set=[(X_val, y_val)],
    verbose=100,
)

print(f"Best iteration: {model.best_iteration}")
print(f"Best score: {model.best_score:.4f}")

Set n_estimators absurdly high and let early stopping find the right number. This is the pattern. Memorize it.

LightGBM: Faster, Often Better

LightGBM (Microsoft, 2017) is the speed king. On large datasets, it's often 5-10× faster than XGBoost while matching or beating its accuracy. It's become the default gradient boosting library for many practitioners.

What Makes LightGBM Different

Leaf-wise tree growth. XGBoost grows trees level-by-level (all nodes at depth d before depth d+1). LightGBM grows leaf-wise: it always splits the leaf with the highest loss reduction, regardless of depth. This produces deeper, more asymmetric trees that converge faster.

# Level-wise (XGBoost):          Leaf-wise (LightGBM):
#       [root]                          [root]
#      /      \                        /      \
#    [A]      [B]                    [A]      [B]
#   /  \     /  \                   /  \        
# [C]  [D] [E]  [F]              [C]  [D]      
#                                 /  \           
#                               [G]  [H]

The downside: leaf-wise growth is more prone to overfitting on small datasets because it creates deeper, more complex trees. Use num_leaves to control this (a leaf-wise tree with num_leaves=31 is roughly equivalent to a level-wise tree with max_depth=5).

GOSS (Gradient-based One-Side Sampling). Not all training examples are equally informative. Examples with large gradients (the model is getting them very wrong) carry more information than examples with small gradients (already well-predicted). GOSS keeps all large-gradient samples and randomly subsamples small-gradient ones. You train on fewer examples per round without losing much information.

Native categorical feature support. Instead of one-hot encoding (which creates sparse, high-dimensional splits), LightGBM can split on categorical features directly using an optimal partition algorithm. Pass your categorical columns and skip the encoding.

import lightgbm as lgb

# Tell LightGBM which features are categorical
cat_features = ["city", "device_type", "browser"]
train_data = lgb.Dataset(X_train, label=y_train, categorical_feature=cat_features)

CatBoost: The Categorical Specialist

CatBoost (Yandex, 2017) was designed from the ground up for datasets heavy on categorical features — think recommendation systems, ad click prediction, anything with city names, device types, user segments.

Ordered target encoding. CatBoost encodes categoricals using target statistics, but with a clever twist: for each example, it only uses target values from preceding examples (in a random permutation). This avoids target leakage — the encoding for example i doesn't peek at example i's label. Other libraries that do target encoding naively get data leakage; CatBoost handles it correctly by construction.

Ordered boosting. Regular gradient boosting has a subtle bias: the residuals used to train tree m were computed on the same data used to train trees 1 through m−1. CatBoost uses a technique called ordered boosting that computes residuals on different subsets to reduce this "prediction shift."

Great defaults. CatBoost often works well out of the box with minimal hyperparameter tuning. If you don't have time to tune extensively, CatBoost's defaults are the best of the three.

The downside: CatBoost is the slowest to train of the three libraries, especially on large datasets. It also has a steeper API learning curve.

Stacking: Ensembles of Ensembles

Stacking takes predictions from multiple different models — say, a Random Forest, a LightGBM, and a logistic regression — and uses them as input features to a meta-model (often a simple linear model or another LightGBM). The meta-model learns which base models to trust in which situations.

The catch: you must generate the base model predictions using out-of-fold predictions (cross-validation), or the meta-model will just learn to trust the base model that overfits the most.

from sklearn.ensemble import StackingClassifier
from sklearn.linear_model import LogisticRegression

stack = StackingClassifier(
    estimators=[
        ("rf", RandomForestClassifier(n_estimators=300)),
        ("lgb", LGBMClassifier(n_estimators=500, learning_rate=0.05)),
        ("xgb", XGBClassifier(n_estimators=500, learning_rate=0.05)),
    ],
    final_estimator=LogisticRegression(),
    cv=5,  # Out-of-fold predictions for meta-features
)
stack.fit(X_train, y_train)

Stacking can squeeze out another 0.5-1% accuracy on top of the best single model. In competitions, that's worth it. In production, it's usually not — you're tripling model complexity, inference time, and maintenance burden for marginal gains. Know it exists, save it for Kaggle.

Production Considerations

Ensembles in notebooks and ensembles in production are different beasts. Here's what bites people:

Model size. A gradient boosted model is n_estimators × average_tree_size. A LightGBM model with 5,000 trees and 255 leaves each can be 100+ MB serialized. If you're deploying to edge devices or need fast cold starts, this matters. Reduce n_estimators and increase learning_rate to trade a tiny bit of accuracy for a much smaller model.

Prediction latency. An ensemble prediction means running the input through every tree and aggregating. That's N sequential memory lookups (trees aren't SIMD-friendly). For real-time serving at thousands of QPS, LightGBM's prediction is fastest. Also consider ONNX export for optimized inference runtimes.

Feature engineering still matters. GBMs can learn feature interactions (tree splits naturally capture them), but explicit feature engineering still helps. A feature like price_per_sqft = price / sqft is one split for the model to learn automatically but zero effort for you to create. Hand-crafted features reduce the number of trees needed and improve generalization.

Always use early stopping. Always. This isn't optional. Without early stopping, you're guessing how many trees to train, and you'll either underfit or overfit. Early stopping makes n_estimators adaptive — it trains exactly as many trees as needed and stops. There is zero reason not to use it.

The Pattern Everyone Uses: LightGBM + Early Stopping + CV

This is the template that wins. Memorize this shape — you'll use it on every tabular problem.

import lightgbm as lgb
import numpy as np
from sklearn.model_selection import StratifiedKFold
from sklearn.metrics import roc_auc_score

params = {
    "objective": "binary",
    "metric": "auc",
    "verbosity": -1,
    "learning_rate": 0.05,
    "num_leaves": 31,
    "feature_fraction": 0.8,
    "bagging_fraction": 0.8,
    "bagging_freq": 1,
    "min_data_in_leaf": 20,
    "lambda_l1": 0.1,
    "lambda_l2": 1.0,
}

kf = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)
oof_preds = np.zeros(len(X_train))

for fold, (train_idx, val_idx) in enumerate(kf.split(X_train, y_train)):
    X_tr, X_va = X_train.iloc[train_idx], X_train.iloc[val_idx]
    y_tr, y_va = y_train.iloc[train_idx], y_train.iloc[val_idx]

    train_set = lgb.Dataset(X_tr, label=y_tr)
    val_set = lgb.Dataset(X_va, label=y_va)

    model = lgb.train(
        params,
        train_set,
        num_boost_round=10000,          # Set high
        valid_sets=[val_set],
        callbacks=[lgb.early_stopping(50), lgb.log_evaluation(200)],
    )

    oof_preds[val_idx] = model.predict(X_va)
    print(f"Fold {fold}: AUC = {roc_auc_score(y_va, oof_preds[val_idx]):.4f}"
          f" | Best iter: {model.best_iteration}")

print(f"\nOverall OOF AUC: {roc_auc_score(y_train, oof_preds):.4f}")

This gives you: per-fold scores (to check stability), out-of-fold predictions (for stacking or calibration), automatic early stopping (no overfitting), and a reliable estimate of test performance. Every competitive ML pipeline starts here.
