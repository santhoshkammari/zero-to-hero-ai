Linear Regression

Chapter 5: Supervised Learning Foundation · Loss · Optimization · Regularization

Why a Straight Line?

You're predicting house prices. You've got square footage in one column, sale price in another. You plot them and see an obvious upward trend — but it's noisy. Some 1,500 sqft houses sold for $200k, others for $260k.

You need a formula. The simplest possible guess is a line:

price = w₀ + w₁ × sqft

Two reasons this is a great starting point. First, interpretability. If w₁ = 150, you tell your stakeholder: "each extra square foot adds ~$150." Try explaining a neural network's reasoning that clearly. Second, it works surprisingly often. Most real-world relationships are approximately linear within the range of data you actually observe — even when the true global relationship is curved.

A line is the default. You only reach for something fancier when it demonstrably fails.

The Loss Function: Least Squares

You have n points. You want the line ŷ = w₀ + w₁x that's "closest" to all of them. But what does "close" mean? You need a single number that measures total wrongness. The standard choice:

Loss = Σᵢ (yᵢ - ŷᵢ)² = Σᵢ (yᵢ - w₀ - w₁xᵢ)²

Why squared errors specifically?

  • Makes all errors positive — being $10 too high is equally bad as $10 too low.
  • Punishes large errors disproportionately — off by $100 contributes 10,000; ten errors of $10 contribute only 1,000 total.
  • The math works out beautifully — you get a smooth, convex (bowl-shaped) surface with a single minimum and a closed-form solution.

Take the derivatives, set to zero, solve. Done:

w₁ = Σ(xᵢ - x̄)(yᵢ - ȳ) / Σ(xᵢ - x̄)²
w₀ = ȳ - w₁x̄

No iteration, no hyperparameters. Plug in data, get answer. Legendre published this in 1805. It's still your first tool 220 years later.

Why Not Absolute Errors?

You could minimize |yᵢ - ŷᵢ| instead (L1 loss, a.k.a. Least Absolute Deviations). It's more robust to outliers — one wild data point can't drag the line as far. But it has no closed-form solution and isn't differentiable at zero, making optimization trickier. Use it when you suspect heavy-tailed noise. Otherwise, squared errors win on convenience.

The Normal Equation

One feature is a toy problem. Real models use dozens or hundreds. With multiple features, you write the model in matrix form:

ŷ = Xw    where X is n×(p+1), w is (p+1)×1

The loss becomes (y - Xw)ᵀ(y - Xw). Differentiate, set to zero, and you get the normal equation:

XᵀX w = Xᵀy   →   w = (XᵀX)⁻¹ Xᵀy

Elegant. Exact. One shot. But there's a catch.

When to Use It vs. Gradient Descent

FactorNormal EquationGradient Descent
Features (p)Fast up to ~10kScales to millions
Samples (n)Any (but need n > p)Any — mini-batch on huge data
ComplexityO(p³) for matrix inversionO(npk) for k iterations
HyperparametersNoneLearning rate, batch size, epochs
Exact solution?YesApproximation (converges)
Singular XᵀX?Fails (use pseudoinverse)Still works

Rule of thumb: if your feature matrix fits comfortably in memory and p < 10,000, use the normal equation. Otherwise, gradient descent.

Gradient Descent

When XᵀX is too big to invert, you iterate instead. Start with random weights. Compute how wrong you are. Nudge in the direction that reduces the error. Repeat.

w ← w - α · ∇Loss(w)

∇Loss(w) = -2Xᵀ(y - Xw)     # the gradient
α = learning rate              # step size

The learning rate α is your first real hyperparameter. Too large → the loss oscillates or explodes. Too small → you waste hours crawling toward the answer.

Batch vs. Mini-Batch vs. Stochastic

VariantGradient computed onTradeoff
BatchAll n samplesAccurate gradient, expensive per step
Mini-batchRandom subset (32–512)Good balance — the standard choice
Stochastic (SGD)Single sampleNoisy but fast; good for online learning

Mini-batch is what everyone uses in practice. It's noisy enough to escape shallow local minima (less relevant for linear regression's convex loss, but critical habit for deep learning) and cheap enough to scale to billions of rows.

Critical: gradient descent requires feature scaling. If sqft ranges from 500–3000 and bedrooms ranges from 1–5, the loss surface becomes an elongated ellipse and convergence slows dramatically. Always standardize first: (x - mean) / std.

Multiple Regression

Going from 1 feature to many is conceptually straightforward — just more columns in X. But it introduces a new problem.

Multicollinearity

Include both "square footage" and "number of rooms." They're heavily correlated — big houses have more rooms. The model can't decide how to split credit. Coefficient for sqft might flip sign if you add one more data point. The individual coefficients become meaningless even though predictions remain stable.

The Variance Inflation Factor (VIF) quantifies this:

VIF(j) = 1 / (1 - R²_j)
# R²_j = R² from regressing feature j on all other features
# VIF > 10 = serious problem.  VIF > 5 = investigate.

Fixes: drop one of the correlated features, combine them (sqft per room), or just use regularization — it handles collinearity gracefully.

Regularization

This is the single most important concept in this section. Everything you learn here — L1 penalties, L2 penalties, the bias-variance tradeoff they embody — you'll use in every model going forward. Neural networks, SVMs, even tree ensembles use analogous ideas.

The problem: more features → lower training error → but higher test error. Coefficients blow up, canceling each other in elaborate ways that work on training data and nowhere else. Regularization adds a penalty for large weights:

Total Loss = Data Loss + λ × Complexity Penalty

Ridge Regression (L2)

Loss_ridge = Σ(yᵢ - ŷᵢ)² + λ Σ wⱼ²

Shrinks all coefficients toward zero, but never to zero. Every feature stays in the model with a dampened voice. Closed-form solution still exists:

w_ridge = (XᵀX + λI)⁻¹ Xᵀy

That λI term is the magic — it makes XᵀX always invertible, even with perfect multicollinearity. Ridge is literally adding stability to the matrix inversion.

Bayesian interpretation: ridge = Gaussian prior on coefficients centered at zero. "I believe weights are probably small; convince me otherwise with data."

Lasso Regression (L1)

Loss_lasso = Σ(yᵢ - ŷᵢ)² + λ Σ |wⱼ|

The dramatic difference: lasso drives coefficients to exactly zero. Not approximately. Exactly. It performs automatic feature selection. If 8 of your 100 features matter, lasso zeros the other 92.

Why? Geometrically, the L1 constraint region is a diamond with sharp corners on the axes. The loss function's contours tend to hit those corners — where one or more coordinates are exactly zero. L2's circle has no corners, so the contours touch at smooth points where everything is nonzero.

No closed-form solution. Solved by coordinate descent (optimize one weight at a time, cycle through all).

ElasticNet (L1 + L2)

Loss = Σ(yᵢ - ŷᵢ)² + λ[ρ Σ|wⱼ| + (1-ρ) Σwⱼ²]
# ρ = l1_ratio in sklearn. ρ=1 → Lasso. ρ=0 → Ridge.

Gets you sparsity (some zeros) plus stability (correlated features share coefficients instead of one randomly winning). Use when you have groups of correlated features.

Which Regularizer? Decision Table

SituationUseWhy
All features probably matterRidgeShrinks everything, drops nothing
Many features are noiseLassoZeros out the irrelevant ones
Groups of correlated featuresElasticNetLasso picks one randomly; ElasticNet keeps the group
Don't know / first tryElasticNet (ρ=0.5)Hedge your bets, tune ρ with CV
Interpretability criticalLassoSparse model = fewer coefficients to explain
p >> n (more features than samples)Lasso or ElasticNetMust reduce dimensionality; Ridge can't

Assumptions (LINE)

OLS rests on four assumptions. You need to know them not because they're exam material, but because when they break, your model lies to you in specific, diagnosable ways.

AssumptionWhat It MeansWhat Breaks When ViolatedHow to Check
Linearity True relationship is linear in features Systematic prediction errors; residuals show curves Residuals vs. fitted plot
Independence Errors don't influence each other Underestimated standard errors → overconfident p-values Durbin-Watson test; residual autocorrelation plot
Normality Residuals are normally distributed Confidence intervals and hypothesis tests become unreliable Q-Q plot of residuals
Equal variance Residual spread is constant (homoscedasticity) Standard errors are wrong → confidence intervals lie Residuals vs. fitted: look for funnel shape

Practical note: normality is the least important for prediction. With enough data, the Central Limit Theorem covers you. Linearity and equal variance are the ones that silently wreck your model. Always plot residuals.

When assumptions hold, OLS is the Best Linear Unbiased Estimator (BLUE) — that's the Gauss-Markov theorem. But "unbiased" isn't always what you want. A little bias (via regularization) often buys you a lot of variance reduction, and your test error drops.

Weighted Least Squares fixes heteroscedasticity when you know the variance structure: Loss = Σ wᵢ(yᵢ - ŷᵢ)².

R² and Adjusted R²

R² answers: "what fraction of variance in y does the model explain?"

R² = 1 - (SS_res / SS_tot)

SS_res = Σ(yᵢ - ŷᵢ)²    # residual sum of squares
SS_tot = Σ(yᵢ - ȳ)²     # total sum of squares

R² = 0.85 means 85% of the variation is captured. R² = 0 means you're no better than predicting the mean. R² = 1.0 is suspicious — you're probably overfitting or leaking the target.

The Trap

R² never decreases when you add features. Throw in a column of random noise and R² ticks up by chance. This is why Adjusted R² exists — it penalizes for the number of features:

Adjusted R² = 1 - ((1 - R²)(n - 1)) / (n - p - 1)
# n = samples, p = features

If adding a feature doesn't improve adjusted R², it's not earning its keep. In practice, cross-validated metrics (RMSE, MAE) are more reliable than either R² variant for model selection.

Production Gotchas

Feature Scaling Is Not Optional (for Regularization)

Ridge and Lasso penalize large coefficients. If sqft is in thousands and bedrooms is 1–5, the regularizer hammers the sqft coefficient unfairly — it's bigger only because the units are bigger. Always standardize features before regularized regression. sklearn's Ridge/Lasso don't do this for you.

Polynomial Features for Non-Linearity

If residuals show a curve, add polynomial terms: x², x³, or interaction terms x₁×x₂. This is still "linear" regression — "linear" means linear in the parameters, not the features. Pair with regularization or the coefficients will explode.

Always Check Residual Plots

R² can look great while your model is systematically wrong. Plot residuals vs. predicted values. You're looking for random scatter. Patterns mean the model is missing something structural.

Influence Diagnostics

Cook's Distance measures how much all predictions change when you remove one point. Threshold: Cook's D > 4/n deserves a look. Don't delete influential points reflexively — investigate whether they're errors or genuinely informative outliers.

Putting It All Together

import numpy as np
from sklearn.linear_model import LinearRegression, Ridge, Lasso, ElasticNet
from sklearn.preprocessing import StandardScaler
from sklearn.pipeline import make_pipeline
from sklearn.model_selection import cross_val_score
from sklearn.datasets import make_regression

# 200 samples, 20 features, only 5 actually informative
X, y = make_regression(n_samples=200, n_features=20,
                       n_informative=5, noise=10, random_state=42)

models = {
    "OLS":        make_pipeline(StandardScaler(), LinearRegression()),
    "Ridge":      make_pipeline(StandardScaler(), Ridge(alpha=1.0)),
    "Lasso":      make_pipeline(StandardScaler(), Lasso(alpha=0.5)),
    "ElasticNet": make_pipeline(StandardScaler(), ElasticNet(alpha=0.5, l1_ratio=0.5)),
}

print(f"{'Model':<13} {'CV R² (mean ± std)'}")
print("-" * 38)
for name, pipe in models.items():
    scores = cross_val_score(pipe, X, y, cv=5, scoring="r2")
    print(f"{name:<13} {scores.mean():.3f} ± {scores.std():.3f}")

# Inspect Lasso's sparsity
lasso_pipe = models["Lasso"].fit(X, y)
coefs = lasso_pipe.named_steps["lasso"].coef_
n_zeros = (np.abs(coefs) < 1e-6).sum()
print(f"\nLasso zeroed {n_zeros}/{len(coefs)} coefficients")

This is what real projects look like. Load data. Try multiple models. Compare with cross-validation. Inspect sparsity. Pick the one that generalizes best.
