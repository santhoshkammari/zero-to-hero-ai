Decision Trees

Chapter 5: Supervised Learning 7 subtopics
TL;DR

Decision trees split data with a series of yes/no questions, building a flowchart that any human can read. They choose each split by measuring impurity — entropy or Gini — picking the question that best separates the classes. Without pruning they overfit badly, so you trim the tree with max-depth limits or cost-complexity pruning. Trees need no feature scaling, handle missing values naturally, and give you feature importance for free. Their main weakness is instability: small data changes can produce wildly different trees — which is exactly why ensembles (Random Forests, Gradient Boosting) exist.

You Already Use Decision Trees

You already use decision trees every day — if it's raining, take an umbrella; else, check the temperature; if it's above 30 °C, bring water; otherwise, just go. That chain of if/else questions is a decision tree. Every node asks one question about one feature. Every branch represents one answer. Every leaf gives a final decision. The algorithm just automates the process of figuring out which questions to ask and in what order.

Why should you care when fancier models exist? Three reasons. First, interpretability: you can print a decision tree, hand it to a doctor or a regulator, and they can follow every step. Try that with a neural network. Second, decision trees are the building block inside Random Forests and Gradient Boosting — the algorithms that win most tabular-data competitions on Kaggle. Understanding a single tree is prerequisite to understanding those ensembles. Third, trees expose every important concept in ML — overfitting, bias-variance tradeoff, feature selection — in a format you can literally draw on a napkin.

CART: Classification and Regression Trees

Binary Splits and Recursive Partitioning

The most widely used tree algorithm is CART (Classification and Regression Trees), published by Breiman, Friedman, Olshen, and Stone in 1984. CART builds trees using binary splits — every internal node asks exactly one yes/no question. "Is income ≤ $50k?" Left child gets the yes cases, right child gets the no cases. This is different from older algorithms like ID3, which could create a branch for every unique value of a feature.

Recursive partitioning is the strategy. Start with the full dataset at the root. Find the single feature and threshold that best separates the data. Split. Now you have two subsets. Repeat the process on each subset independently. Keep going until a stopping condition fires — maybe you've hit a maximum depth, or each leaf has only one sample. The result is a binary tree where each leaf contains a group of training samples that are (hopefully) similar to each other.

Classification Trees vs Regression Trees

When the target is categorical — spam or not spam, species A/B/C — you're building a classification tree. Each leaf predicts the majority class of the training samples that landed there. If 80 of the 100 samples in a leaf are "spam," that leaf predicts spam.

When the target is continuous — house prices, temperature — you're building a regression tree. Each leaf predicts the mean of the target values of the training samples in that leaf. If 50 houses in a leaf have a mean price of $320,000, that leaf predicts $320,000 for any new data point that reaches it. The splitting criterion changes (we use variance reduction instead of impurity), but the tree-building process is identical.

Splitting Criteria: How the Tree Chooses Questions

This is the heart of the algorithm. At every node, the tree considers every feature and every possible threshold for that feature. For each candidate split, it measures how "mixed" the resulting child nodes are. It picks the split that reduces the mix the most. But how do we measure "mixed"? There are two main options for classification, and they agree far more often than they disagree.

Entropy and Information Gain

Shannon entropy measures the uncertainty in a set of labels. If every sample in a node belongs to the same class, entropy is 0 — no uncertainty, the node is pure. If the samples are evenly split across classes, entropy is at its maximum — maximal uncertainty.

For a node with two classes, entropy is defined as:

Entropy(S) = -p₁ · log₂(p₁) - p₂ · log₂(p₂)

where p₁ = fraction of class 1, p₂ = fraction of class 2

Let's make this concrete. Suppose a node has 10 samples: 7 cats and 3 dogs.

# Entropy calculation — 7 cats, 3 dogs
import math

p_cat = 7 / 10   # 0.7
p_dog = 3 / 10   # 0.3

entropy = -(p_cat * math.log2(p_cat)) - (p_dog * math.log2(p_dog))
# entropy = -(0.7 × (-0.5146)) - (0.3 × (-1.7370))
# entropy = 0.3602 + 0.5211
# entropy ≈ 0.8813

print(f"Entropy: {entropy:.4f}")   # 0.8813

An entropy of 0.88 out of a maximum of 1.0 (for two classes) tells us this node is fairly mixed. A pure node (all cats) would have entropy 0. A perfectly split node (5 cats, 5 dogs) would have entropy 1.0.

Information gain is the reduction in entropy after a split. You compute the entropy of the parent node, then subtract the weighted average entropy of the two child nodes. The split with the highest information gain wins.

# Information Gain example
# Parent: 7 cats, 3 dogs → Entropy = 0.8813

# Candidate split: "Is weight ≤ 8 kg?"
# Left child:  6 cats, 1 dog  (7 samples)
# Right child: 1 cat,  2 dogs (3 samples)

p_left = 7 / 10
p_right = 3 / 10

entropy_left  = -(6/7)*math.log2(6/7) - (1/7)*math.log2(1/7)   # ≈ 0.5917
entropy_right = -(1/3)*math.log2(1/3) - (2/3)*math.log2(2/3)   # ≈ 0.9183

weighted_entropy = p_left * entropy_left + p_right * entropy_right
# = 0.7 × 0.5917 + 0.3 × 0.9183
# ≈ 0.4142 + 0.2755 = 0.6897

info_gain = 0.8813 - 0.6897   # ≈ 0.1916
print(f"Information Gain: {info_gain:.4f}")   # 0.1916

One subtle problem with information gain: it has a bias toward features with many unique values. A feature like "customer ID" would always give perfect splits (each child is pure!) but would be useless for generalization. The gain ratio, introduced in C4.5 (Quinlan, 1993), addresses this by normalizing information gain by the "split information" — the entropy of the split itself. This penalizes features that create many small branches. The ID3 algorithm uses raw information gain; its successor C4.5 uses gain ratio and also handles continuous features and missing values.

Gini Impurity

The Gini impurity (also called Gini index) measures the probability that a randomly chosen sample from the node would be misclassified if you assigned it a label randomly according to the class distribution in the node. Like entropy, Gini is 0 for a pure node and reaches its maximum (0.5 for two classes) when classes are equally distributed.

Gini(S) = 1 - p₁² - p₂²

For K classes: Gini(S) = 1 - Σ pₖ²

Let's compute Gini for the same node — 7 cats, 3 dogs:

# Gini impurity calculation — 7 cats, 3 dogs
p_cat = 7 / 10
p_dog = 3 / 10

gini = 1 - p_cat**2 - p_dog**2
# gini = 1 - 0.49 - 0.09
# gini = 0.42

print(f"Gini impurity: {gini:.2f}")   # 0.42

For the same candidate split (left: 6 cats/1 dog, right: 1 cat/2 dogs):

# Weighted Gini after split
gini_left  = 1 - (6/7)**2 - (1/7)**2   # ≈ 0.2449
gini_right = 1 - (1/3)**2 - (2/3)**2   # ≈ 0.4444

weighted_gini = (7/10) * gini_left + (3/10) * gini_right
# = 0.7 × 0.2449 + 0.3 × 0.4444
# ≈ 0.1714 + 0.1333 = 0.3048

gini_reduction = 0.42 - 0.3048   # ≈ 0.1152
print(f"Gini reduction: {gini_reduction:.4f}")   # 0.1152
Entropy vs Gini — does it matter?

In practice, entropy and Gini produce nearly identical trees over 95% of the time. Gini is slightly faster to compute (no logarithm). Entropy tends to produce slightly more balanced trees. Scikit-learn's DecisionTreeClassifier defaults to Gini, and that default is fine for almost every use case. CART uses Gini. ID3/C4.5 use entropy. Don't overthink the choice.

Variance Reduction for Regression

For regression trees, there is no class to be "impure" about. Instead, we measure impurity as the variance (or MSE) of the target values in a node. A node where all house prices are around $300k has low variance. A node with prices ranging from $100k to $900k has high variance. The algorithm picks the split that reduces variance the most — MSE reduction. Some implementations also support MAE (mean absolute error) as the criterion, which is more robust to outliers.

# Variance reduction example for regression trees
import numpy as np

# Parent node: house prices (in $k)
prices = np.array([200, 220, 250, 280, 500, 520, 550, 600])
parent_var = np.var(prices)   # ≈ 22,425

# Split: "sqft ≤ 1500?"
left  = np.array([200, 220, 250, 280])   # small homes
right = np.array([500, 520, 550, 600])   # large homes

weighted_var = (len(left)/len(prices)) * np.var(left) + \
               (len(right)/len(prices)) * np.var(right)
# left var ≈ 875, right var ≈ 1,875
# weighted = 0.5 × 875 + 0.5 × 1,875 = 1,375

variance_reduction = parent_var - weighted_var
print(f"Variance reduction: {variance_reduction:,.0f}")   # ≈ 21,050

Building and Visualizing a Decision Tree in Scikit-learn

Enough theory. Let's build a real tree, look at it, and understand every piece of the output.

import numpy as np
import matplotlib.pyplot as plt
from sklearn.datasets import load_iris
from sklearn.tree import DecisionTreeClassifier, plot_tree
from sklearn.model_selection import train_test_split
from sklearn.metrics import accuracy_score

# Load the iris dataset: 150 flowers, 4 features, 3 classes
X, y = load_iris(return_X_y=True)
feature_names = load_iris().feature_names
class_names = load_iris().target_names

# Split into train/test
X_train, X_test, y_train, y_test = train_test_split(
    X, y, test_size=0.3, random_state=42
)

# Build a decision tree (intentionally no depth limit to see overfitting)
tree_clf = DecisionTreeClassifier(
    criterion="gini",       # default; try "entropy" to compare
    max_depth=None,         # grow full tree
    min_samples_split=2,    # default
    random_state=42
)
tree_clf.fit(X_train, y_train)

# Evaluate
train_acc = accuracy_score(y_train, tree_clf.predict(X_train))
test_acc  = accuracy_score(y_test,  tree_clf.predict(X_test))
print(f"Train accuracy: {train_acc:.3f}")   # likely 1.000
print(f"Test accuracy:  {test_acc:.3f}")    # likely lower

# Visualize the tree
fig, ax = plt.subplots(figsize=(20, 10))
plot_tree(
    tree_clf,
    feature_names=feature_names,
    class_names=class_names,
    filled=True,          # color nodes by majority class
    rounded=True,
    fontsize=9,
    ax=ax
)
plt.title("Decision Tree — Iris Dataset (No Pruning)")
plt.tight_layout()
plt.savefig("decision_tree_full.png", dpi=150)
plt.show()

# Feature importance — how much each feature reduced impurity overall
for name, imp in zip(feature_names, tree_clf.feature_importances_):
    print(f"  {name:<25s} {imp:.4f}")

Notice that train accuracy is 100% — the unpruned tree memorized every training sample. Test accuracy is lower because the tree also memorized the noise. This is overfitting in action, and it's exactly why we need pruning.

Pruning: Taming the Overfit

An unpruned decision tree will keep splitting until every leaf is pure or contains a single sample. That tree has zero training error and terrible generalization. Pruning fixes this, and there are two families of approaches.

Pre-Pruning (Early Stopping)

Pre-pruning stops the tree from growing in the first place. You set constraints before training:

  • max_depth — limit how many levels deep the tree can go. A depth of 3 means at most 3 questions before reaching a leaf. This is the single most important hyperparameter.
  • min_samples_split — a node must have at least this many samples to be split further. Setting it to 20 means the tree won't try to split a node with only 15 samples.
  • min_samples_leaf — every leaf must contain at least this many samples. Prevents the tree from creating leaves with just 1 or 2 samples.
  • max_features — limit how many features the tree considers at each split. Primarily used in Random Forests to decorrelate trees, but can also reduce overfitting in single trees.
  • min_impurity_decrease — only split if the impurity decrease exceeds this threshold. A small value like 0.01 prevents splits that barely improve anything.

Pre-pruning is simple and fast. The downside: it might stop too early. A split that looks useless now might enable a very useful split two levels down. Pre-pruning can't see that future benefit.

Post-Pruning: Cost-Complexity Pruning (CCP)

Post-pruning takes the opposite approach: grow the full tree first, then cut it back. The most important post-pruning method is cost-complexity pruning (CCP), also called minimal cost-complexity pruning or weakest-link pruning.

The idea: assign a penalty to every leaf. The more leaves (more complex tree), the higher the penalty. The total cost of a tree is its training error plus α × (number of leaves), where α (alpha, the complexity parameter) controls the tradeoff. A small α keeps more leaves (complex tree). A large α prunes aggressively (simple tree). You select the best α by cross-validation.

Reduced error pruning is a simpler alternative: hold out a validation set, then try removing each subtree one at a time. If removing a subtree doesn't increase validation error, prune it. This is intuitive but requires a separate validation set, which wastes data.

Here's how to do CCP with cross-validation in scikit-learn:

import numpy as np
import matplotlib.pyplot as plt
from sklearn.datasets import load_iris
from sklearn.tree import DecisionTreeClassifier
from sklearn.model_selection import cross_val_score

X, y = load_iris(return_X_y=True)

# Step 1: Get the effective alphas by building the full tree
#         and computing the cost-complexity pruning path
full_tree = DecisionTreeClassifier(random_state=42)
full_tree.fit(X, y)
path = full_tree.cost_complexity_pruning_path(X, y)
ccp_alphas = path.ccp_alphas    # array of alpha values
impurities = path.impurities    # total impurity at each alpha

# Step 2: For each alpha, build a tree and evaluate with CV
cv_scores = []
for alpha in ccp_alphas:
    clf = DecisionTreeClassifier(ccp_alpha=alpha, random_state=42)
    scores = cross_val_score(clf, X, y, cv=5, scoring="accuracy")
    cv_scores.append(scores.mean())

cv_scores = np.array(cv_scores)

# Step 3: Pick the alpha with the best CV score
best_idx = np.argmax(cv_scores)
best_alpha = ccp_alphas[best_idx]
print(f"Best alpha: {best_alpha:.4f}")
print(f"Best CV accuracy: {cv_scores[best_idx]:.4f}")

# Step 4: Train the final pruned tree with the best alpha
pruned_tree = DecisionTreeClassifier(
    ccp_alpha=best_alpha, random_state=42
)
pruned_tree.fit(X, y)
print(f"Pruned tree depth: {pruned_tree.get_depth()}")
print(f"Pruned tree leaves: {pruned_tree.get_n_leaves()}")

# Step 5: Plot alpha vs CV accuracy
fig, ax = plt.subplots(figsize=(10, 5))
ax.plot(ccp_alphas, cv_scores, marker="o", markersize=3, drawstyle="steps-post")
ax.axvline(best_alpha, color="red", linestyle="--", label=f"Best α = {best_alpha:.4f}")
ax.set_xlabel("Cost-Complexity Parameter (α)")
ax.set_ylabel("5-Fold CV Accuracy")
ax.set_title("Cost-Complexity Pruning: Selecting Alpha")
ax.legend()
plt.tight_layout()
plt.savefig("ccp_alpha_selection.png", dpi=150)
plt.show()

This is the gold-standard workflow: grow the full tree, compute the pruning path, evaluate each candidate α with cross-validation, and pick the winner. The resulting pruned tree is usually much shallower — often 3 to 5 levels — and generalizes far better.

Tree Properties: The Good, the Bad, and the Unstable

Interpretability

Decision trees are the most interpretable non-trivial model in machine learning. You can export the tree as a set of if/else rules and hand it to a non-technical stakeholder. Each prediction has a clear explanation path: "This patient was classified as high-risk because their blood pressure was above 140, their age was above 60, and their cholesterol was above 240." No other algorithm gives you that for free.

Feature Importance

Every feature gets an importance score based on how much it reduced impurity across all the splits where it was used. In scikit-learn, access it via tree.feature_importances_. This is a quick and dirty feature selection method. Caveat: importance is biased toward high-cardinality features (features with many unique values get more chances to split), and it's computed on training data, so it can be misleading for overfit trees.

Handling Missing Values

Some tree implementations handle missing values natively. CART uses surrogate splits — if the primary split feature is missing for a sample, the tree falls back to a different feature that would have produced a similar split. XGBoost handles missingness by learning a default direction at each node. Scikit-learn's DecisionTreeClassifier gained native missing value support in version 1.4. Before that, you had to impute missing values yourself.

No Feature Scaling Required

Trees only care about the order of feature values, not their magnitude. Whether a feature ranges from 0 to 1 or from 0 to 1,000,000, the tree asks "Is feature ≤ threshold?" and the ranking stays the same. This means no standardization, no normalization, no Min-Max scaling. One less preprocessing step to worry about — and one less thing to get wrong.

Instability (High Variance)

Here's the big weakness. Decision trees are unstable. Remove a single training sample, or change one value slightly, and the tree can look completely different. Why? Because if the root split changes, every subsequent split changes too. Small perturbations cascade through the entire structure. This is high variance in the bias-variance sense.

This instability is not a bug to work around — it's actually a feature that makes ensemble methods work. Random Forests train hundreds of unstable trees on random subsets of the data and average their predictions. The individual trees disagree wildly, but their average is remarkably stable. The instability of a single tree becomes the diversity that makes the ensemble strong.

Advanced Variants

Conditional Inference Trees

Conditional inference trees fix a subtle bias in standard CART: the tendency to select features with many unique values. Instead of picking the split that maximizes impurity reduction, conditional inference trees use statistical hypothesis tests. At each node, they first test whether any feature is significantly associated with the target (using a permutation test). Only if a significant association exists do they split — and they choose the feature with the smallest p-value. This produces unbiased feature selection and built-in stopping (if no feature is significant, the node becomes a leaf). They're available in R's partykit package.

Minimum Description Length (MDL) Principle

The MDL principle views tree construction through the lens of compression. The best tree is the one that gives the shortest description of the data — meaning the tree structure itself (the "model code") plus the corrections needed where the tree's predictions are wrong (the "data code"). A tree that is too complex has a long model code. A tree that is too simple has a long data code (many errors). MDL naturally balances model complexity against fit, providing an alternative to cross-validation for model selection.

Oblique (Multivariate) Decision Trees

Standard CART only makes axis-aligned splits: "Is feature X ≤ threshold?" Oblique trees allow splits on linear combinations of features: "Is 0.3 × feature₁ + 0.7 × feature₂ ≤ threshold?" This lets the tree carve diagonal decision boundaries instead of only horizontal and vertical ones. The tradeoff is clear: oblique trees can fit some data patterns with far fewer nodes, but finding the optimal oblique split is much more expensive computationally, and the resulting tree is harder for humans to interpret. Implementations include OC1 and the oblique-tree package.

⚠ When to use a single decision tree

In practice, you'll almost never deploy a single decision tree in production for prediction accuracy. Use one when interpretability is the primary goal — regulatory settings, medical explainability, initial data exploration. For prediction performance, use an ensemble: Random Forest (Chapter 5, Section 6) or Gradient Boosting (Chapter 5, Section 7). But understand the single tree first. You can't debug a forest if you don't understand a tree.
