Decision Trees

Chapter 5: Supervised Learning Splits · Pruning · Feature importance · The variance problem

It's Just a Flowchart

A decision tree asks a series of yes/no questions about your features, splitting the data into increasingly pure groups. That's it. There's no gradient, no loss surface, no matrix algebra. It's literally a flowchart.

Is income > $50k?
├── Yes → Is age > 30?
│         ├── Yes → Approve loan
│         └── No  → Review manually
└── No  → Is credit score > 700?
          ├── Yes → Approve loan
          └── No  → Deny loan

Every internal node is a question about one feature. Every branch is a yes/no answer. Every leaf is a prediction. You can print a decision tree on a whiteboard and a non-technical stakeholder can follow the logic — try doing that with a neural network.

This makes decision trees THE most interpretable model in the standard ML toolkit. That's not just aesthetically nice — it's legally required in domains like credit scoring and healthcare where you need to explain why a prediction was made.

But interpretability isn't why you're learning trees in depth. You're learning them because they're the building block for ensemble methods — Random Forests, Gradient Boosted Trees, XGBoost, LightGBM. Those are the models that dominate tabular data in practice. You can't understand ensembles without understanding the single tree they're built from.

How Splits Are Chosen: Gini vs Entropy

At every node, the tree needs to pick a feature and a threshold to split on. It wants splits that produce purer child nodes — groups where most samples belong to the same class. But how do you measure "purity"?

Gini Impurity

Gini measures the probability of misclassifying a randomly chosen sample if you labeled it according to the class distribution at that node:

Gini = 1 - Σ(pᵢ²)

Where pᵢ is the proportion of class i at the node. A pure node (all one class) has Gini = 0. Maximum impurity for a binary problem is Gini = 0.5 (50/50 split). The tree picks the split that produces the largest weighted decrease in Gini across the two child nodes.

Entropy & Information Gain

Entropy comes from information theory — it measures uncertainty:

Entropy = -Σ(pᵢ log₂ pᵢ)

A pure node has entropy 0. Maximum entropy for binary classification is 1.0. Information gain is the parent's entropy minus the weighted average of children's entropy. The split that maximizes information gain wins.

Which One Should You Use?

Honestly? Don't overthink it. In practice, Gini and entropy produce nearly identical trees. Research has shown they disagree on the chosen split in less than 2% of cases. Gini is marginally faster because it avoids computing logarithms. Sklearn defaults to Gini, and that default is fine.

Regression Trees: Variance Reduction

For regression, there's no "class purity." Instead, the tree minimizes MSE (mean squared error) at each split. It picks the split that maximizes the reduction in variance — the child nodes should have less variance in the target than the parent. Each leaf predicts the mean of the training samples that ended up there.

The Greedy Algorithm: CART

The algorithm behind most decision tree implementations is CART (Classification and Regression Trees). Here's what it does at every node:

  1. For every feature, sort the values and consider every midpoint between adjacent values as a candidate threshold.
  2. For every candidate threshold, compute the impurity reduction if you split there.
  3. Pick the (feature, threshold) pair with the highest impurity reduction.
  4. Split the data and recurse on each child.
  5. Stop when a stopping criterion is met (max depth, min samples, pure node, etc.).

This is greedy: it picks the locally best split at each step without considering what splits come later. A split that looks mediocre now might enable amazing splits downstream — but CART will never find it. Finding the globally optimal tree is NP-hard, so greedy is what we've got.

The greedy nature has a practical consequence: the order of early splits matters enormously. A slightly different first split can cascade into a completely different tree structure. This instability is a feature, not a bug — it's exactly what ensemble methods will exploit.

Pruning: Because Unchecked Trees Memorize Everything

An unpruned decision tree will keep splitting until every leaf contains exactly one class (or even one sample). This gives you 100% training accuracy and terrible test accuracy. The tree has memorized the training data, noise and all.

Pruning is how you control this. There are two strategies:

Pre-pruning (Restriction)

Stop growing the tree early by constraining its structure. These are your real hyperparameters:

ParameterWhat it doesTypical range
max_depthMaximum tree depth3–20 (start with 5)
min_samples_splitMinimum samples to split a node2–50
min_samples_leafMinimum samples in a leaf1–20
max_featuresMax features considered per split"sqrt", "log2", or int

max_depth is the single most important hyperparameter. A depth-5 tree has at most 2⁵ = 32 leaves. A depth-20 tree has over a million. Start shallow and increase until validation accuracy plateaus.

Post-pruning (Cost-Complexity Pruning)

Grow the full tree first, then prune it back. Sklearn implements this with the ccp_alpha parameter (cost-complexity pruning). Larger alpha values prune more aggressively. Think of it like regularization strength for trees.

The idea: for each subtree, is the improvement in accuracy worth the added complexity? If not, collapse it back to a leaf. This often produces better trees than pre-pruning alone because it can evaluate the full tree structure before deciding what to cut.

Feature Importance: Built-In Feature Selection

Every time a tree makes a split, it records how much that split reduced impurity. Sum these reductions across all splits for each feature, normalize, and you get feature importance scores. It's automatic — no extra computation needed.

tree = DecisionTreeClassifier(max_depth=5).fit(X_train, y_train)
for name, imp in sorted(zip(feature_names, tree.feature_importances_),
                         key=lambda x: -x[1]):
    print(f"{name:25s} {imp:.4f}")

This is genuinely useful for understanding your data. But there's a trap:

High-cardinality features get inflated importance. A feature with 1,000 unique values has many more candidate split points than a binary feature. More opportunities to split → more chances to reduce impurity → artificially higher importance. A random ID column can appear "important" simply because it has a unique value for every sample.

The fix: use permutation importance instead. Shuffle a feature's values, measure how much accuracy drops. This measures actual predictive power regardless of cardinality. Sklearn provides permutation_importance() for exactly this.

Handling Different Data Types

Categorical Features

Base sklearn trees can't handle categorical features natively — you need to encode them first (one-hot or ordinal). This is annoying and can introduce problems: one-hot encoding a 50-category feature creates 50 binary columns, which fragments the split decisions. Ordinal encoding imposes an arbitrary ordering.

LightGBM and CatBoost handle categoricals natively — they find optimal category groupings without encoding. This is one of their real advantages over raw sklearn trees.

Missing Values

Sklearn's DecisionTreeClassifier can't handle NaN values — you must impute first. But XGBoost and LightGBM learn which direction to send missing values at each split. During training, they try both directions and pick the one that reduces loss more. This is elegant and often better than any imputation strategy.

Feature Scaling: Not Needed

This is a big practical advantage. Trees only care about the ordering of feature values, not their magnitude. Whether income is measured in dollars or millions of dollars, the same threshold gets found. No StandardScaler, no MinMaxScaler, no worrying about different units across features. Just feed raw features in.

Strengths and Weaknesses

StrengthsWeaknesses
Highly interpretable — print and explainHigh variance — small data changes → completely different tree
No feature scaling requiredProne to overfitting without pruning
Handles non-linear relationships naturallyGreedy splits aren't globally optimal
Built-in feature importanceCan't extrapolate beyond training data range
Fast training and predictionUnstable — high sensitivity to training data
Works with both classification and regressionAxis-aligned splits only (staircase decision boundaries)

The axis-aligned split limitation deserves a note. Trees can only split perpendicular to feature axes — they can't draw diagonal boundaries. If the true decision boundary is a 45° line in feature space, a tree approximates it with a staircase of horizontal and vertical cuts. More depth → finer staircase → but also more overfitting.

The Variance Problem: Why Single Trees Aren't Enough

This is the most important thing to understand about decision trees, because it's the entire motivation for ensemble methods.

A single decision tree is a high-variance, low-bias model. It can fit arbitrarily complex patterns (low bias) but is extremely sensitive to the specific training data it sees (high variance). Remove a handful of samples, add a few new ones, and you can get a completely different tree with different features, different thresholds, different structure.

Why? Because the greedy algorithm compounds small changes. If a slightly different sample causes the first split to use a different feature, every subsequent split changes too. The instability cascades.

Think of it as the tree overfitting to the particular training set — not just the noise in the data, but the specific random sample of the population it happened to see. Two trees trained on different random subsets of the same population might agree on the overall trend but disagree wildly on specifics.

Here's the key insight: if individual trees have low bias but high variance, what if we averaged many of them? Averaging reduces variance without increasing bias. That's Random Forests. And what if instead of averaging independent trees, we built each new tree to specifically correct the errors of the previous ones? That's Gradient Boosting.

The instability of a single tree isn't a bug — it's the raw material that ensemble methods need. Without variance, there would be nothing to average out.

Production Gotchas

Trees can't extrapolate. If your training data has house prices ranging from $100k to $500k, the tree will never predict $600k. Every leaf prediction is the mean (regression) or majority class (classification) of training samples in that leaf. This is a fundamental limitation for time series or any domain where the target can exceed the training range.

Imbalanced classes need explicit handling. Trees optimize purity, and a node that's 95% majority class already looks pretty "pure." Use class_weight='balanced' to make the tree pay more attention to minority classes, or resample your data before training.

Large trees are slow in production. A depth-30 tree requires up to 30 sequential comparisons per prediction. For latency-sensitive serving, keep trees shallow or use model distillation.

Don't trust single-tree feature importance. As discussed, it's biased toward high-cardinality features and unstable across runs. Use permutation_importance() or, better yet, get feature importance from an ensemble (where instability averages out).

Code: Decision Tree with Cross-Validated Pruning

from sklearn.datasets import load_breast_cancer
from sklearn.tree import DecisionTreeClassifier
from sklearn.model_selection import cross_val_score
import numpy as np

X, y = load_breast_cancer(return_X_y=True)

# Step 1: find optimal ccp_alpha via built-in cost-complexity path
base_tree = DecisionTreeClassifier(random_state=42)
path = base_tree.cost_complexity_pruning_path(X, y)
alphas = path.ccp_alphas[:-1]  # last is trivial (single-node tree)

# Step 2: cross-validate each alpha
scores = [cross_val_score(
    DecisionTreeClassifier(ccp_alpha=a, random_state=42),
    X, y, cv=5, scoring='accuracy'
).mean() for a in alphas]

best_alpha = alphas[np.argmax(scores)]
print(f"Best alpha: {best_alpha:.4f}  CV accuracy: {max(scores):.4f}")

# Step 3: train final pruned tree
tree = DecisionTreeClassifier(ccp_alpha=best_alpha, random_state=42)
tree.fit(X, y)
print(f"Tree depth: {tree.get_depth()}, leaves: {tree.get_n_leaves()}")

This is the right way to prune: let the data tell you how much complexity is justified. The cost-complexity path gives you a menu of tree sizes, and cross-validation picks the best one. You'll typically end up with a tree that's 5–10 nodes deep instead of 20+.
