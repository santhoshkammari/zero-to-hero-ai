Clustering

Chapter 6: Unsupervised Learning Finding structure without labels

The Setup: Groups Nobody Told You About

You've got 50,000 customers. Purchase history, browsing patterns, session durations, average order value — the works. Marketing wants "segments." Nobody has labeled a single customer. There's no target column. You're staring at a cloud of points in feature-space and asking: does this thing have structure?

That's clustering. Every algorithm in this section answers the same question differently, because they each define "group" differently. K-Means says a group is "stuff near a center." DBSCAN says it's "a dense blob." GMMs say it's "a Gaussian distribution." None of them are wrong — they're just looking at different aspects of the geometry. Your job is to pick the definition that matches your data.

K-Means: The Workhorse

Lloyd's Algorithm in 30 Seconds

Pick k random centers. Assign each point to its closest center. Move each center to the mean of its assigned points. Repeat. Done.

That's Lloyd's algorithm, and it's the engine under every K-Means call. Two steps, alternating:

# The actual loop inside KMeans, conceptually:
for _ in range(max_iter):
    # ASSIGN: each point → nearest centroid
    labels = np.argmin(cdist(X, centroids), axis=1)
    # UPDATE: each centroid → mean of its points
    for j in range(k):
        centroids[j] = X[labels == j].mean(axis=0)
    # Stop when assignments don't change

The objective being minimized is within-cluster sum of squares (WCSS), also called inertia:

WCSS = Σⱼ Σᵢ∈Cⱼ ‖xᵢ − μⱼ‖²

Each assign step can only decrease (or maintain) WCSS. Each update step can only decrease (or maintain) WCSS — because the mean is the unique minimizer of squared distances. Bounded below by zero, monotonically decreasing: convergence is guaranteed. Usually in 10–30 iterations.

But convergence means local minimum. Not global. Where you start matters.

K-Means++ Init: The One Trick That Actually Matters

Random init is a coin flip. You might get centroids clumped in one corner, producing one bloated cluster and two tiny ones. Running K-Means 10 times with different seeds and keeping the best (the default n_init=10) helps, but it's brute force.

K-Means++ is smarter. Pick the first centroid uniformly at random. For each subsequent centroid, choose a point with probability proportional to its squared distance from the nearest existing centroid. Points far from all current centroids get picked next. This is D² weighting.

The guarantee: the initial solution is O(log k) competitive with the optimal. In practice, it's often nearly as good as the final answer before a single Lloyd iteration runs. Sklearn uses it by default — you don't need to do anything.

Choosing k

The eternal question. Three approaches, ranked by how much I trust them:

Silhouette score (most reliable): for each point, compare how close it is to its own cluster (a) versus the next-nearest cluster (b). Score = (b − a) / max(a, b). Ranges from −1 (wrong cluster) to +1 (perfect). Sweep k, pick the peak.

Elbow method (decent, often ambiguous): plot WCSS vs. k. Look for the "elbow" where returns diminish. The problem? Real data rarely gives a clean elbow. You end up squinting.

Gap statistic (principled but slow): compare your WCSS to what you'd expect from uniform noise. The k where the "gap" is largest wins. Requires generating reference datasets, so it's expensive.

import numpy as np
from sklearn.cluster import KMeans
from sklearn.metrics import silhouette_score
from sklearn.datasets import make_blobs

X, _ = make_blobs(n_samples=1500, centers=4,
                   cluster_std=1.0, random_state=42)

# Sweep k, track both metrics
results = {}
for k in range(2, 10):
    km = KMeans(n_clusters=k, n_init=10, random_state=42).fit(X)
    results[k] = {
        'inertia': km.inertia_,
        'silhouette': silhouette_score(X, km.labels_)
    }
    print(f"k={k}  inertia={km.inertia_:,.0f}  "
          f"silhouette={results[k]['silhouette']:.3f}")

# Best silhouette → best k
best_k = max(results, key=lambda k: results[k]['silhouette'])
print(f"\nBest k by silhouette: {best_k}")

Where K-Means Breaks

K-Means minimizes squared Euclidean distance, which carves the feature space into Voronoi cells — convex polygons with straight boundaries. This bakes in three assumptions:

  • Clusters are roughly spherical (or at least convex)
  • Clusters are roughly equal-sized
  • You know k upfront

Two crescent moons? K-Means cuts them with a straight line through the middle. One cluster with 10,000 points and one with 50? The big one steals from the small one. Outliers? A single extreme point drags the centroid sideways.

Mini-Batch K-Means. Standard K-Means reads every point every iteration. With millions of rows, that's painful. MiniBatchKMeans samples a random batch per step — 10x faster, typically within 1–2% of the full solution. Use it when n > 100k.

DBSCAN: Let Density Define the Clusters

A Different Definition of "Group"

K-Means thinks in terms of centers. DBSCAN thinks in terms of crowds. A cluster is a dense region. The gap between clusters is sparse. You don't need to know how many clusters exist — you just need to define what "dense" means.

Two parameters do that: eps (ε, the neighborhood radius) and min_samples (minimum neighbors within that radius to count as dense).

Core, Border, Noise

Every point in the dataset gets one of three labels:

  • Core point — has ≥ min_samples neighbors within eps. The heart of a cluster.
  • Border point — within eps of a core point, but doesn't have enough neighbors itself. Cluster fringe.
  • Noise point — neither. Outlier. Gets label −1.

The algorithm: pick an unvisited core point, recursively expand its neighborhood (every core neighbor's neighborhood is reachable too), mark all reachable points as one cluster. Repeat with the next unvisited core point. Border points attach to whatever cluster their nearest core point belongs to. Noise stays alone.

That recursive expansion is why DBSCAN finds arbitrary shapes. Two moons? No problem — the dense arcs connect through their core points, not through any centroid.

Tuning eps and min_samples

The k-distance plot: for each point, compute the distance to its k-th nearest neighbor (k = min_samples). Sort descending and plot. The elbow tells you where density drops off — set eps there. For min_samples, start with 2 × n_features as a rule of thumb, minimum 5.

from sklearn.cluster import DBSCAN
from sklearn.datasets import make_moons
from sklearn.preprocessing import StandardScaler
from sklearn.neighbors import NearestNeighbors
import numpy as np

X, _ = make_moons(n_samples=1000, noise=0.07, random_state=42)
X = StandardScaler().fit_transform(X)

# k-distance plot to find eps
k = 5
nn = NearestNeighbors(n_neighbors=k).fit(X)
distances, _ = nn.kneighbors(X)
k_dist = np.sort(distances[:, -1])[::-1]
# Plot k_dist — look for the elbow. Here it's around 0.25–0.30

db = DBSCAN(eps=0.28, min_samples=5).fit(X)
labels = db.labels_

n_clusters = len(set(labels) - {-1})
n_noise = np.sum(labels == -1)
print(f"Clusters: {n_clusters}, Noise points: {n_noise}")

# Compare to K-Means on moons — it fails spectacularly
from sklearn.cluster import KMeans
km = KMeans(n_clusters=2, random_state=42).fit(X)
print(f"K-Means just drew a vertical line through the moons.")

The Density Trap

DBSCAN has one fundamental weakness: a single eps for the whole dataset. If one cluster is tight (urban) and another is diffuse (rural), no single radius captures both. You'll either merge the tight clusters or shatter the loose ones.

Also: high-dimensional data is hostile territory. Distances converge (curse of dimensionality), and the concept of "neighborhood" breaks down. Always reduce dimensions before running DBSCAN in high-d.

HDBSCAN: The Modern Default

HDBSCAN fixes the single-eps problem by running DBSCAN at every possible eps, building a hierarchy, then using a cluster stability metric to extract the most persistent clusters. Clusters that survive across many density levels are real. Ones that flicker in and out are noise.

The result: no eps to tune, automatic cluster count detection, graceful handling of varying densities, and a built-in notion of "I don't know" for ambiguous points. If you're reaching for DBSCAN in 2025, reach for HDBSCAN instead.

from hdbscan import HDBSCAN

hdb = HDBSCAN(min_cluster_size=15, min_samples=5)
hdb_labels = hdb.fit_predict(X)

# Points HDBSCAN isn't sure about get label -1
n_clusters = len(set(hdb_labels) - {-1})
n_noise = np.sum(hdb_labels == -1)
print(f"HDBSCAN clusters: {n_clusters}, noise: {n_noise}")

# Bonus: confidence scores per point
print(f"Mean confidence: {hdb.probabilities_.mean():.3f}")
HDBSCAN vs DBSCAN, honestly. HDBSCAN has one fewer critical hyperparameter (no eps), handles mixed-density data, and gives you confidence scores. The only reason to use DBSCAN is if you need a specific, interpretable density threshold for domain reasons — like "all points within 500 meters." Otherwise, HDBSCAN.

Hierarchical Clustering: The Merge Tree

Bottom-Up Thinking

Agglomerative clustering starts with every point as its own cluster. At each step, merge the two closest clusters. Repeat until everything is one big cluster. The result is a tree — a dendrogram — that records every merge and the distance at which it happened.

You don't have to commit to a specific k upfront. Cut the tree at any height and you get a different number of clusters. High cut = few big clusters. Low cut = many small clusters. It's clustering at every resolution, stored in one data structure.

Linkage: How "Close" Are Two Clusters?

This is the make-or-break decision. "Distance between two clusters" isn't obvious when clusters have multiple points. Four definitions, each producing dramatically different results:

LinkageDistance measureCluster shapeWatch out for
SingleMin distance between any pairLong, chain-likeChaining effect — one bridge point merges two clusters
CompleteMax distance between any pairCompact, sphericalForces equal sizes, breaks elongated clusters
AverageMean of all pairwise distancesBalancedGood default, but no strong guarantees
WardIncrease in total WCSSCompact, spherical (like K-Means)Requires Euclidean distance

Ward is the default for a reason — it produces the same kind of clusters K-Means does, but without needing to pick k first. Use single linkage only when you expect chain-shaped clusters (rare, but it happens in geographic data).

Reading a Dendrogram

The y-axis is merge distance. Tall bars mean "these clusters were far apart when they merged" — those are your real cluster boundaries. Short bars are nearby points collapsing together early. Draw a horizontal line; the number of vertical lines it crosses is your cluster count.

Pro tip: look for the biggest vertical gap in the dendrogram. That's where the merge distance jumps — the natural cluster boundary.

from sklearn.cluster import AgglomerativeClustering
from scipy.cluster.hierarchy import dendrogram, linkage, fcluster
from sklearn.datasets import make_blobs
from sklearn.metrics import silhouette_score

X, _ = make_blobs(n_samples=300, centers=4,
                   cluster_std=0.7, random_state=42)

# Compare linkage methods
for link in ['ward', 'complete', 'average', 'single']:
    agg = AgglomerativeClustering(n_clusters=4, linkage=link)
    labels = agg.fit_predict(X)
    sil = silhouette_score(X, labels)
    print(f"{link:10s} → silhouette = {sil:.3f}")

# Dendrogram with scipy (for visualization)
Z = linkage(X, method='ward')
# dendrogram(Z, truncate_mode='level', p=5)  # plot this

# Cut the tree at a specific number of clusters
labels_3 = fcluster(Z, t=3, criterion='maxclust')
labels_6 = fcluster(Z, t=6, criterion='maxclust')
print(f"\n3 clusters: silhouette = {silhouette_score(X, labels_3):.3f}")
print(f"6 clusters: silhouette = {silhouette_score(X, labels_6):.3f}")

When Hierarchical Wins

Use it when you genuinely need multi-resolution analysis — you want to see the data at 3 clusters, 5 clusters, and 10 clusters, and understand which groups split into which. Taxonomies, gene expression data, document hierarchies. Also good when you need to explain the clustering to non-technical stakeholders — dendrograms are surprisingly intuitive.

The cost: O(n² log n) time and O(n²) memory for standard agglomerative clustering. Beyond ~20k points, consider BIRCH (builds a compact summary tree in a single pass, then clusters the summaries) as a scalable alternative.

Spectral Clustering: When Geometry Gets Weird

Concentric rings. Spirals. Two interlocking crescents. All the shapes where K-Means face-plants because there's no sensible "center." Spectral clustering handles these by changing the representation before clustering.

The Graph Laplacian Idea

Step 1: Build a similarity graph. Each point is a node. Edge weights reflect similarity — typically a Gaussian kernel: W_ij = exp(−‖xᵢ − xⱼ‖² / 2σ²).

Step 2: Compute the graph Laplacian L = D − W, where D is the diagonal degree matrix. This matrix encodes the connectivity structure of your data.

Step 3: Take the k smallest eigenvectors of L. Stack them as columns. Each row is now a new representation of a data point — a spectral embedding.

Step 4: Run K-Means on the embedded points.

Why does this work? The eigenvectors of the Laplacian encode the "smoothest" signals on the graph. Points in the same cluster get similar eigenvector values. In the new space, even wildly non-convex clusters become neat, separable blobs.

from sklearn.cluster import SpectralClustering, KMeans
from sklearn.datasets import make_circles

# Concentric circles — K-Means can't handle this
X, y = make_circles(n_samples=500, factor=0.5,
                     noise=0.05, random_state=42)

# K-Means: two half-circles (wrong)
km = KMeans(n_clusters=2, random_state=42).fit(X)

# Spectral: inner ring vs outer ring (correct)
sc = SpectralClustering(n_clusters=2, affinity='rbf',
                         gamma=10, random_state=42)
sc_labels = sc.fit_predict(X)

from sklearn.metrics import adjusted_rand_score
print(f"K-Means ARI: {adjusted_rand_score(y, km.labels_):.3f}")
print(f"Spectral ARI: {adjusted_rand_score(y, sc_labels):.3f}")
The catch: O(n²) space and O(n³) time. You're building a full affinity matrix and computing eigendecompositions. Beyond ~10k points, it gets painful. Nyström approximation and sparse affinity matrices help, but spectral clustering is fundamentally a small-to-medium data tool. If your dataset is large and non-convex, HDBSCAN is usually the better bet.

Gaussian Mixture Models: Soft K-Means

Probabilities, Not Labels

K-Means gives you hard labels: point 47 is in cluster 2, period. But what if point 47 sits exactly between two clusters? K-Means assigns it to one and throws away the uncertainty. Gaussian Mixture Models keep it: point 47 is 62% cluster A, 35% cluster B, 3% cluster C.

A GMM models the data as a mixture of k Gaussians, each with its own mean μₖ, covariance Σₖ, and mixing weight πₖ:

p(x) = Σₖ πₖ · 𝒩(x | μₖ, Σₖ)     # mixing weights sum to 1

EM: The Engine

The Expectation-Maximization algorithm fits GMMs by alternating two steps. It's the same assign-then-update rhythm as K-Means, but with probabilities instead of hard assignments:

E-step: compute responsibilities — the probability each component generated each point. This is Bayes' theorem:

r_ik = πₖ · ��(xᵢ | μₖ, Σₖ) / Σⱼ πⱼ · 𝒩(xᵢ | μⱼ, Σⱼ)

M-step: re-estimate μₖ, Σₖ, πₖ using responsibilities as soft counts. The new mean is the responsibility-weighted average of all points.

Here's the punchline: if you force spherical covariances and hard (0/1) responsibilities, EM reduces exactly to Lloyd's algorithm. K-Means is a degenerate special case of GMMs.

Covariance Types — Match the Shape

TypeShape per clusterParameters per componentWhen to use
sphericalSphere1 (single σ²)Like K-Means but with varying sizes
diagonalAxis-aligned ellipsoiddFeatures are independent within clusters
tiedSame ellipsoid everywhered(d+1)/2 (shared)Same cluster shape, different centers
fullArbitrary ellipsoidd(d+1)/2 eachMaximum flexibility, needs more data

Start with full. If you're overfitting (BIC increases), step down to diagonal or tied.

Choosing k: BIC Over Elbows

GMMs have something K-Means doesn't: a likelihood. That lets you use information criteria:

from sklearn.mixture import GaussianMixture, BayesianGaussianMixture
from sklearn.datasets import make_blobs
import numpy as np

X, _ = make_blobs(n_samples=800, centers=3,
                   cluster_std=[1.0, 1.5, 0.5], random_state=42)

# Model selection via BIC — lower is better
for k in range(1, 8):
    gm = GaussianMixture(n_components=k,
                          covariance_type='full',
                          random_state=42).fit(X)
    print(f"k={k}  BIC={gm.bic(X):>10.1f}  AIC={gm.aic(X):>10.1f}")

# Fit the best model
gmm = GaussianMixture(n_components=3, covariance_type='full',
                       random_state=42).fit(X)

labels = gmm.predict(X)
probs = gmm.predict_proba(X)  # The whole point: soft assignments

# Show uncertainty for a few points
for i in [0, 100, 400]:
    p = ', '.join(f'{v:.2f}' for v in probs[i])
    print(f"Point {i}: [{p}] → assigned to {labels[i]}")

Bayesian GMM: Let the Model Choose k

Don't want to sweep k at all? BayesianGaussianMixture puts a Dirichlet process prior on the weights. Set a generous upper bound (say 15 components), and it prunes unused components to near-zero weight automatically.

bgmm = BayesianGaussianMixture(
    n_components=15, covariance_type='full',
    weight_concentration_prior_type='dirichlet_process',
    random_state=42
).fit(X)

active = np.sum(bgmm.weights_ > 0.01)
print(f"Active components: {active} / 15")
print(f"Weights: {np.round(bgmm.weights_[:6], 3)}")
When GMMs beat K-Means. Whenever you need soft assignments (anomaly detection: "how unlikely is this point under the model?"), clusters of different shapes/sizes (full covariance handles ellipses), or a generative model you can sample new data from. The cost: EM is slower, can hit local optima (run multiple inits), and full covariance needs ~10× more data per cluster than K-Means to be stable.

Evaluation: How Do You Know It Worked?

Clustering has no accuracy score. There's no y_test to compare against. You're evaluating structure that may or may not exist. This is genuinely hard, and anyone who tells you otherwise is selling something.

Internal Metrics (No Labels)

Silhouette score — the go-to. For each point, compare intra-cluster tightness (a) to nearest-cluster separation (b). Score = (b − a) / max(a, b).

s(i) = (b(i) - a(i)) / max(a(i), b(i))

  +1 → perfectly placed
   0 → on the boundary
  -1 → probably in the wrong cluster

  > 0.7 strong structure
  > 0.5 reasonable
  > 0.25 weak, might be noise

Davies-Bouldin Index — average worst-case overlap between each cluster and its most similar neighbor. Lower is better. Fast, but biased toward convex shapes.

Calinski-Harabasz — ratio of between-cluster to within-cluster variance. Higher is better. Think of it as a clustering F-statistic.

from sklearn.metrics import (silhouette_score,
                              davies_bouldin_score,
                              calinski_harabasz_score)

# Assume X is your data, labels from any clustering algorithm
print(f"Silhouette:        {silhouette_score(X, labels):.3f}")
print(f"Davies-Bouldin:    {davies_bouldin_score(X, labels):.3f}")
print(f"Calinski-Harabasz: {calinski_harabasz_score(X, labels):.0f}")

External Metrics (When You Have Labels)

For benchmarks, validation sets, or sanity-checking against known categories:

Adjusted Rand Index (ARI): counts pairs of points. Same-cluster-same-class (good) vs. same-cluster-different-class (bad). Adjusted for chance. Range: −1 to 1. Zero = random.

Normalized Mutual Information (NMI): how much does knowing the cluster assignment reduce uncertainty about the true label? Range: 0 to 1. Zero = independent.

from sklearn.metrics import (adjusted_rand_score,
                              normalized_mutual_info_score)

# y_true = ground truth, labels = your clustering
ari = adjusted_rand_score(y_true, labels)
nmi = normalized_mutual_info_score(y_true, labels)
print(f"ARI: {ari:.3f}  NMI: {nmi:.3f}")
The metric bias trap. Silhouette favors compact, spherical clusters — exactly what K-Means produces. So it'll happily rate K-Means higher than DBSCAN on data where DBSCAN found the correct non-convex clusters. Internal metrics reflect the algorithm's assumptions, not ground truth. Always visualize. Run multiple algorithms. Compare with external metrics when possible.

Stability: The Underrated Check

Cluster your data multiple times with different random subsets or bootstrap samples. If the same clusters keep appearing, they're real. If the result reshuffles every time, your "clusters" are artifacts. This is especially valuable when metrics are ambiguous and no one approach is clearly winning.

Decision Framework: Which Algorithm When

Don't memorize this table — build intuition for it. The key insight is that each algorithm defines "cluster" differently.

SituationFirst choiceWhy
Know k, expect blobs, need speedK-MeansFast, simple, interpretable
n > 100k, need speedMiniBatchKMeans10× faster, ~same quality
Don't know k, expect noise/outliersHDBSCANAuto k, handles noise, varying density
Non-convex shapes, moderate nSpectralHandles any topology via graph Laplacian
Need probabilities, not hard labelsGMMSoft assignments, full covariance flexibility
Need multi-resolution explorationAgglomerativeDendrogram gives every possible k at once
Need interpretable density thresholdDBSCANeps has a physical meaning you can explain
Very high-d dataK-Means (after PCA/UMAP)Reduce dims first, then cluster
Anomaly scoring from clustersGMMLow-likelihood points = anomalies

The real answer: run 2–3 methods, compare visually and via metrics, pick the one that tells a story domain experts believe. No algorithm is universally best because "cluster" has no universal definition.

# The practical workflow: compare multiple algorithms
from sklearn.cluster import KMeans, DBSCAN, SpectralClustering
from sklearn.mixture import GaussianMixture
from sklearn.metrics import silhouette_score
from sklearn.preprocessing import StandardScaler

# Always scale first
X_scaled = StandardScaler().fit_transform(X)

algos = {
    'KMeans(4)':   KMeans(n_clusters=4, n_init=10, random_state=42),
    'DBSCAN':      DBSCAN(eps=0.5, min_samples=5),
    'GMM(4)':      GaussianMixture(n_components=4, random_state=42),
}

for name, algo in algos.items():
    if hasattr(algo, 'fit_predict'):
        labels = algo.fit_predict(X_scaled)
    else:
        labels = algo.fit(X_scaled).predict(X_scaled)

    # Skip if only 1 cluster or all noise
    n_clusters = len(set(labels) - {-1})
    if n_clusters < 2:
        print(f"{name:20s} → only {n_clusters} cluster(s), skip")
        continue

    mask = labels != -1
    sil = silhouette_score(X_scaled[mask], labels[mask])
    print(f"{name:20s} → {n_clusters} clusters, "
          f"silhouette = {sil:.3f}")

What You Should Now Be Able To Do

  • Use K-Means++ initialization and explain why D² weighting produces better starting centroids
  • Choose k using silhouette scores, elbow plots, and BIC — and know which to trust most
  • Describe DBSCAN's core/border/noise classification and tune eps via k-distance plots
  • Explain why HDBSCAN is strictly better than DBSCAN for most problems and use it in code
  • Pick the right linkage method for hierarchical clustering and read a dendrogram to choose k
  • Explain the spectral clustering pipeline: affinity → Laplacian → eigenvectors → K-Means
  • Fit a GMM with EM, extract soft assignments, and select k via BIC
  • Compute silhouette, ARI, and NMI scores and know when each is appropriate
  • Look at a new clustering problem and pick the right algorithm based on data shape, size, and what you need from the output
