A Confession and an Invitation

Every explanation starts reasonable — "PCA finds directions of maximum variance" — then drops into eigenvectors and covariance matrices with no motivation. Problem was not math. Nobody explained why these tools exist before dumping machinery, or showed tiny example where you watch thing work.

This chapter gives explanation I wanted back then. Start with real problem — too many features, not enough signal — then build each tool from scratch with examples small enough for head. Start with PCA, workhorse for 80% of cases, then move to t-SNE, UMAP, and specialized methods PCA cannot handle.

The Curse of Dimensionality

Before learning reduction, feel why high dimensions hurt. Not abstract "it's computationally expensive" hurt. Real hurt. Hurt that changes how you think about data.

Start with two data points. In one dimension: 0 and 1. Distance = 1. Add second dimension — (0, 0) and (1, 1). Distance = √2 ≈ 1.41. Three dimensions: (0, 0, 0) and (1, 1, 1). Distance = √3 ≈ 1.73. In 100 dimensions, distance between corners of unit hypercube is √100 = 10.

Sounds harmless. Then intuition breaks. Generate thousand random points uniformly inside 100-dimensional unit hypercube and measure all pairwise distances. Average distance is large. More important: spread of distances is tiny relative to mean. Every point sits roughly same distance from every other point.

import numpy as np

np.random.seed(42)
for d in [2, 10, 100, 1000]:
    points = np.random.rand(500, d)
    # Compute all pairwise distances
    diffs = points[:, np.newaxis, :] - points[np.newaxis, :, :]
    dists = np.sqrt((diffs ** 2).sum(axis=2))
    # Get upper triangle (unique pairs)
    idx = np.triu_indices(500, k=1)
    pair_dists = dists[idx]
    ratio = pair_dists.std() / pair_dists.mean()
    print(f"d={d:4d}  mean={pair_dists.mean():.2f}  "
          f"std={pair_dists.std():.3f}  std/mean={ratio:.4f}")

Run code. Watch std/mean shrink toward zero. In 1000 dimensions, farthest pair and nearest pair are almost same distance. This is distance concentration. Pattern clean: relative spread of distances scales roughly as 1/√d. More dimensions. Less discrimination.

Now think about k-nearest neighbors. Algorithm finds k closest points, then votes. If every point is roughly equidistant, "nearest" neighbors are barely closer than far strangers. Vote turns random. K-NN does not slowly degrade in high dimensions. It effectively stops working.

Three things break as dimensions grow. Speed — distance calculations scale linearly with features, and tree-based indexes like KD-trees collapse toward brute-force search past about 20 dimensions. Overfitting — 200 samples with 500 features gives model huge room to memorize noise. Some spurious direction in 500-D space will separate classes by accident. Visualization — you cannot plot 500 axes. Barely fake 3.

Reason for dimensionality reduction: without it, large parts of machine learning stop working.

The Manifold Hypothesis — Why Reduction Even Works

Question bugged me long time: if we throw away dimensions, do we throw away information? How compress 500 features to 50 and still keep useful model?

Answer is manifold hypothesis. Name sounds fancy. Idea simple. Real high-dimensional data usually does not fill whole space. It lies on or near lower-dimensional surface — manifold — inside high-dimensional space.

Picture Swiss roll. Take flat 2D sheet and curl it in 3D space. Data points live in 3D, but real structure — relationships between points — is 2D. Two points far apart in 3D — one outside roll, one inside — may be close when you unroll sheet. 2D surface is manifold.

This happens all time in real data. Face images live in millions of pixel dimensions, but real variation — pose, lighting, expression, identity — has maybe few hundred degrees of freedom. User behavior dataset may have 300 features, but users often vary along handful of axes: price-sensitive vs. brand-loyal, frequent-browser vs. targeted-buyer, impulse vs. deliberate.

Dimensionality reduction works because data was never truly high-dimensional. We are not crushing information. We are finding lower-dimensional surface where data lives and throwing away empty space around it. Think map projection: Earth's surface is 2D even though embedded in 3D. Good map preserves relationships that matter and drops dimension — altitude, for most uses — that does not.

Keep map projection analogy. Different dimensionality reduction methods are different map projections. Each sacrifices something. Right choice depends on what you need to preserve.

PCA: Finding the Best Camera Angle

The Photography Analogy

Imagine 3D sculpture. Need one catalog photo — one 2D image that keeps as much detail as possible. Walk around searching for angle that shows most. Front shows face and arms. Side hides arms behind torso. Directly above flattens everything into blob.

PCA finds best camera angle. It rotates coordinate system so first axis captures most variation, second axis captures most remaining variation perpendicular to first, and so on. Then it drops axes that show least — axes where sculpture looks same from every angle.

Key insight: PCA does not warp, stretch, or distort data. It rotates data. New axes — principal components — are ordered by how much spread they capture. First few show interesting directions. Rest is noise you can ignore.

A Tiny Example: 4 Products, 3 Features

Make it concrete. Suppose you build product recommendation system for small online store. You have 4 products. For each, you measure 3 user-behavior features: average time on product page in seconds, wishlist adds, and purchases.

import numpy as np

# 4 products × 3 features: [page_time, wishlists, purchases]
X = np.array([
    [120,  15,  45],   # Product A: popular, high engagement
    [100,  12,  40],   # Product B: somewhat popular
    [ 20,   2,   5],   # Product C: niche, low engagement
    [ 30,   4,   8],   # Product D: niche, slightly more visible
])

# Step 1: Center the data (subtract the mean of each feature)
X_centered = X - X.mean(axis=0)
print("Centered data:")
print(X_centered)
# [[ 52.5  6.75 20.5 ]
#  [ 32.5  3.75 15.5 ]
#  [-47.5 -6.25 -19.5 ]
#  [-37.5 -4.25 -16.5 ]]

Look at centered values. Products A and B are positive across all three features. Products C and D are negative across all three. Features move together — page time up, wishlists and purchases up too. PCA exploits this correlated movement.

If all three features tell roughly same story — "product popular" — you do not need three numbers per product. One number — "popularity score" — may capture most of it. First principal component becomes that score.

The Covariance Matrix

To find direction of maximum spread, quantify how features vary together. covariance matrix does that.

# Step 2: Compute the covariance matrix
n = X_centered.shape[0]
C = (1 / (n - 1)) * X_centered.T @ X_centered
print("Covariance matrix (3×3):")
print(np.round(C, 1))
# [[2091.7  266.6  634.2]
#  [ 266.6   35.6   83.2]
#  [ 634.2   83.2  203.0]]

C = (1/(n−1)) Xᵀ X has shape (features × features) — here, 3×3. Diagonal entries are feature variances. C[0,0] = 2091.7 is variance of page_time; large because page_time uses big numbers. Off-diagonal C[0,1] = 266.6 is covariance between page_time and wishlists — positive, so they move together.

Each entry says how two features relate. Covariance matrix summarizes linear relationships in data. PCA looks for directions where variance concentrates.

Eigendecomposition

First time I saw PCA described as "eigendecomposition of covariance matrix," I almost closed tab. Phrase sounded built to scare beginners.

Actual eigendecomposition: give it square matrix — our covariance matrix C. It returns eigenvectors — directions — and eigenvalues — one number per direction showing variance along that direction. Eigenvectors are perpendicular. Sort by eigenvalue. First direction captures most variance, second captures next, and so on.

# Step 3: Eigendecompose the covariance matrix
eigenvalues, eigenvectors = np.linalg.eigh(C)

# Sort by eigenvalue descending (eigh returns ascending)
idx = np.argsort(eigenvalues)[::-1]
eigenvalues = eigenvalues[idx]
eigenvectors = eigenvectors[:, idx]

print("Eigenvalues:", np.round(eigenvalues, 1))
# [2322.6    7.2    0.5]

print("Explained variance ratios:")
ratios = eigenvalues / eigenvalues.sum()
print(np.round(ratios, 4))
# [0.9967  0.0031  0.0002]

print("\nFirst eigenvector (PC1 direction):")
print(np.round(eigenvectors[:, 0], 4))
# [-0.9539 -0.1220 -0.2740]

Look at eigenvalues. First one (2322.6) crushes other two (7.2 and 0.5). So 99.67% of variance in this 3-feature dataset lives along one direction. Data cloud is not sphere. Data cloud is cigar. We started with 3 features. Data screams: one dimension.

First eigenvector [-0.95, -0.12, -0.27] gives cigar direction. It loads heavily on page_time, with smaller contributions from wishlists and purchases. This is our "popularity score" direction.

Formal decomposition: C = VΛVᵀ. V holds eigenvectors as columns. Λ holds eigenvalues on diagonal. Equation done. Columns of V are principal component directions. Eigenvalues in Λ say how important each direction is.

# Step 4: Project the data onto the first principal component
pc1 = X_centered @ eigenvectors[:, 0]  # 4 products → 4 scores
print("PC1 scores:", np.round(pc1, 1))
# [-56.7  -35.5   50.8   41.5]

# Products A and B get large negative scores (popular)
# Products C and D get large positive scores (niche)
# One number now captures most of the 3-feature story

That is PCA. Center data. Compute covariance matrix. Find eigenvectors. Project onto top k. Everything else is optimization, numerics, and practical advice.

SVD: What Actually Runs Under the Hood

In toy example with 3 features, we built Xᵀ X and eigendecomposed it. Fine for small problems. Now imagine gene expression data with 20,000 features. Xᵀ X becomes 20,000 × 20,000 — 400 million entries. Building it costs a lot. Decomposing it costs more. Worse, multiplying Xᵀ X can amplify floating-point error and hurt numerical stability.

In practice, nobody forms Xᵀ X. We apply Singular Value Decomposition directly to data matrix X:

# X = U Σ Vᵀ
# U: (n × n) — left singular vectors (row space)
# Σ: (n × p) diagonal — singular values
# V: (p × p) — right singular vectors (= principal component directions!)

U, sigma, Vt = np.linalg.svd(X_centered, full_matrices=False)

print("Singular values:", np.round(sigma, 2))
# [83.47   4.65   1.24]

# The relationship: eigenvalue_i = sigma_i^2 / (n - 1)
print("Eigenvalues from SVD:", np.round(sigma**2 / (n - 1), 1))
# [2322.6    7.2    0.5]  — matches eigendecomposition exactly!

print("PC directions from SVD (rows of Vt):")
print(np.round(Vt[0], 4))
# Same direction as eigenvectors[:, 0], possibly sign-flipped

Right singular vectors — rows of Vᵀ — are principal component directions. Singular values σᵢ relate to eigenvalues by λᵢ = σᵢ² / (n − 1). Same answer. Better numerics. No covariance matrix build.

Randomized SVD goes further. If you need top 50 components out of 20,000 features, do not compute all 20,000. Randomized algorithms use random projections to approximate top-k singular vectors much faster. Scikit-learn does this automatically when you set svd_solver='randomized' or 'auto' and n_components is small relative to features.

Big limitation: eigendecomposition and SVD find linear directions. They can rotate data. They cannot bend data. Remember Swiss roll — 2D sheet curled in 3D. PCA would flatten by maximum variance, project spiral onto line, and mangle structure. PCA finds best camera angle for sculpture. PCA cannot unroll sculpture into flat photograph. Nonlinear methods handle that. Next section.

How Many Components? The Scree Plot

Knowing PCA mechanics is one thing. Choosing how many components to keep is real question.

explained variance ratio for component k is its eigenvalue divided by sum of all eigenvalues. In product example, PC1 explained 99.67%. Lucky case. Data was almost one-dimensional. Real datasets are messier.

scree plot shows explained variance ratios from largest to smallest. Look for elbow — point where curve drops and each extra component gives diminishing returns. Past elbow: mostly noise.

import numpy as np
from sklearn.decomposition import PCA
from sklearn.preprocessing import StandardScaler
from sklearn.datasets import load_digits

X, y = load_digits(return_X_y=True)
print(f"Original shape: {X.shape}")  # (1797, 64) — 64 pixel features

# ALWAYS standardize before PCA
X_scaled = StandardScaler().fit_transform(X)

# Fit with all components to inspect the variance breakdown
pca_full = PCA().fit(X_scaled)
cumvar = np.cumsum(pca_full.explained_variance_ratio_)

# How many components for 95% variance?
n_95 = np.argmax(cumvar >= 0.95) + 1
print(f"Components for 95% variance: {n_95}")  # ~28

# Now reduce for real
pca = PCA(n_components=n_95)
X_reduced = pca.fit_transform(X_scaled)
print(f"Reduced shape: {X_reduced.shape}")  # (1797, 28)

# Sanity check: reconstruction error
X_back = pca.inverse_transform(X_reduced)
mse = np.mean((X_scaled - X_back) ** 2)
print(f"Reconstruction MSE: {mse:.4f}")

Pragmatic rule: keep enough components to reach 90–95% cumulative explained variance. Then validate on real downstream task. If classification accuracy holds, compression worked. If not, raise threshold.

Confession: 90–95% rule is starting point, not gospel. Some datasets are fine at 80%. Others fail even at 99% because signal lives in low-variance components. Scree plot guides. Downstream metric decides.

PCA in Production

Common PCA mistake: fit PCA on whole dataset, including test set, before cross-validation. That leaks information. Principal components use data statistics — means, variances, covariances. If those include test data, model already peeked.

Standardize first! PCA maximizes variance. If feature A ranges from 0 to 1,000,000 and feature B from 0 to 1, PCA will think A dominates — not because A is more informative, but because A has bigger numbers. Use StandardScaler before PCA. Every time. My first PCA reduced 50 features to one component that was 99.9% "revenue in dollars."

Fix: put PCA inside scikit-learn Pipeline, so it sees only training data in each fold:

from sklearn.pipeline import Pipeline
from sklearn.ensemble import RandomForestClassifier
from sklearn.model_selection import cross_val_score

# The right way: PCA inside the pipeline, after scaling
pipe = Pipeline([
    ('scaler', StandardScaler()),
    ('pca', PCA(n_components=0.95)),  # float = auto-select for 95% variance
    ('clf', RandomForestClassifier(n_estimators=100, random_state=42))
])

scores = cross_val_score(pipe, X, y, cv=5, scoring='accuracy')
print(f"PCA pipeline accuracy: {scores.mean():.3f} ± {scores.std():.3f}")

# Compare: no PCA
pipe_full = Pipeline([
    ('scaler', StandardScaler()),
    ('clf', RandomForestClassifier(n_estimators=100, random_state=42))
])
scores_full = cross_val_score(pipe_full, X, y, cv=5, scoring='accuracy')
print(f"Full features accuracy: {scores_full.mean():.3f} ± {scores_full.std():.3f}")

Passing n_components=0.95 as float tells PCA: keep however many components reach 95% explained variance. No manual tuning. Component count adapts to data in each training fold.

PCA ≈ Linear Autoencoder

This connection clicked once, then changed how I think about learned representations.

Build neural network with one hidden layer, linear activations, and MSE loss. Input and output are same data — autoencoder reconstructs its own input. Hidden layer has k neurons, fewer than input dimension — bottleneck. Train it.

Network learns projection down to k dimensions and back up, minimizing reconstruction error. Subspace it finds is exactly PCA subspace. Same math. Encoder learns top-k eigenvectors. Decoder learns their transpose.

Add nonlinear activations and you get something stronger — nonlinear autoencoder that captures curves and folds PCA misses. But linear case is PCA. So PCA is baseline for learned compressions. Fancy autoencoder claim? Check whether it beats linear projection first.

When PCA Fails: The Nonlinear World

Remember Swiss roll — 2D sheet curled into spiral in 3D? Watch PCA touch it.

from sklearn.datasets import make_swiss_roll
from sklearn.decomposition import PCA

X_swiss, color = make_swiss_roll(n_samples=1500, noise=0.5, random_state=42)
print(f"Swiss roll shape: {X_swiss.shape}")  # (1500, 3)

# PCA: project from 3D to 2D
pca = PCA(n_components=2)
X_pca = pca.fit_transform(X_swiss)

# If you plot X_pca colored by 'color', you'll see the spiral
# has been flattened — points that were far apart on the sheet
# (but close in 3D because the sheet is rolled up) are now
# overlapping. PCA preserved the 3D distances, not the
# 2D-on-the-manifold distances.
print(f"Variance explained: {pca.explained_variance_ratio_.sum():.2%}")
# High! PCA thinks it did great. But the structure is mangled.

PCA finds two directions of maximum variance in 3D and projects onto them. Those are width and height of roll. Spiral structure — actual geometry of data — gets flattened and folded onto itself. Points far apart along sheet surface land on top of each other in PCA projection.

Back to map analogy: this is like making flat Earth map from one photo in space. You capture one hemisphere and lose other. Need real map projection — algorithm that knows Earth is curved and unfolds it well. Nonlinear methods do that.

Kernel PCA is one option — apply kernel trick to PCA, map data into higher-dimensional space where linear PCA captures nonlinear structure. It works, but you must choose kernel — RBF usually — it scales poorly on large datasets, and newer methods mostly replaced it in practice.

t-SNE: The Visualization Microscope

Say this upfront: t-SNE is visualization tool. Not feature engineering method. Not preprocessing step. Not input generator for downstream model. It makes pretty 2D pictures of high-dimensional data. That job it does extremely well. Anything else is mistake.

I still misread t-SNE plots sometimes. Years later, I still stare at gap between clusters and think "those groups are very different." t-SNE does not support that conclusion. Old habits stick.

How It Works

Core idea has three steps. Each one matters.

Step 1 — Build probability distribution in high-D. For each data point, t-SNE computes probability that it would pick every other point as neighbor. Gaussian kernel sits on each point: nearby points get high probability, distant points get near-zero. Kernel width adapts per point so each point has roughly same number of effective neighbors, controlled by perplexity. Result is probability distribution over point pairs that encodes local neighborhood structure.

Step 2 — Build similar distribution in 2D, but with different kernel. Place points randomly in 2D. Compute pairwise similarities again, but use Student's t-distribution with one degree of freedom — same as Cauchy — instead of Gaussian. This is key innovation. This gives name t-SNE.

Step 3 — Make distributions match. Use gradient descent to minimize KL divergence between high-D distribution and low-D distribution. Points shuffle in 2D until neighborhood structure matches as closely as possible. Things close in high-D end up close in 2D.

The Crowding Problem and Why Heavy Tails Fix It

This insight made t-SNE work where earlier methods like original SNE failed. Took me while to internalize.

In 100 dimensions, one point can have dozens of roughly equidistant neighbors. Hypersphere surface area is enormous — room for everyone. In 2D, maybe 6 equidistant neighbors fit around point before collisions start. Trying to preserve all high-D neighborhoods with Gaussian in 2D creates traffic jam — everything smushes into center. This is crowding problem.

Student's t-distribution has heavier tails than Gaussian. Points moderately far apart in high-D get pushed much farther apart in 2D embedding. This creates breathing room — clusters separate, local neighborhoods survive, blob collapse stops. Heavy tails add extra lanes to highway.

Perplexity: The One Knob That Matters

Perplexity (typical range: 5–50) roughly controls how many effective neighbors each point considers. Think soft version of k in k-nearest neighbors. Low perplexity (5–10) means tight local focus — fine-grained structure shows up, but clusters may split into fake subclusters. High perplexity (30–50) means broader context — clusters get chunkier and relative positions stabilize, but internal structure can disappear.

Golden rule for t-SNE interpretation. Run perplexity 5, 30, and 50. If cluster appears in all three, it is probably real. If it appears only at low perplexity, it may be artifact from algorithm splitting continuous group.

The Gotchas — Read This Before Interpreting Any t-SNE Plot

Cluster sizes are meaningless. Algorithm can inflate or squash clusters freely. Big blob on screen may be tighter in original space than tiny blob beside it.

Distances between clusters are meaningless. Two well-separated groups may be nearly identical in high-D. Gap between them tells you nothing reliable.

Only cluster existence and membership are trustworthy. "These points group together" — yes. Believe that. "This cluster is far from that one" — no. Do not believe that.

Never use t-SNE output as features for model. Embedding is optimized for visual aesthetics, not predictive power. Axes have no stable meaning. Different random seeds give different layouts. Training classifier on t-SNE coordinates is like training on random art.

Run it multiple times. Even with PCA initialization, which helps a lot, results can vary. If structure changes a lot across runs, be skeptical.

from sklearn.manifold import TSNE
from sklearn.datasets import load_digits

X, y = load_digits(return_X_y=True)

tsne = TSNE(
    n_components=2,
    perplexity=30,
    learning_rate='auto',   # sklearn auto-tunes since v1.2
    init='pca',             # much more stable than random init
    random_state=42,
    n_iter=1000
)
X_2d = tsne.fit_transform(X)

# Visualization code (uncomment with matplotlib):
# import matplotlib.pyplot as plt

# plt.title('t-SNE of Digits Dataset')

Barnes-Hut approximation reduces t-SNE complexity from O(n²) to O(n log n), so it works up to about 100,000 points. Beyond that, runtime hurts. For larger datasets, use UMAP.

UMAP: The Modern Default

UMAP (Uniform Manifold Approximation and Projection) arrived in 2018 and quickly became default first choice. Faster than t-SNE. Preserves more global structure. Big one: you can use output as features for downstream models.

How It Works

UMAP theory involves fuzzy simplicial sets and Riemannian geometry. Paper gets heavy. Practical algorithm is much cleaner:

Step 1 — Build weighted k-nearest-neighbor graph in high-D. For each point, find n_neighbors closest points. Assign edge weights from distance: closer means stronger. Normalize each point's distances by its local scale, so dense and sparse regions get fair treatment.

Step 2 — Build similar graph in low-D. Initialize points in 2D or target dimension with spectral embedding. Compute edge weights in low-D space.

Step 3 — Optimize low-D layout so graphs match. Minimize cross-entropy between high-D and low-D edge-weight distributions with stochastic gradient descent.

That cross-entropy objective is key difference from t-SNE. KL divergence, t-SNE's loss, penalizes things that should be close but are not — attractive forces. Cross-entropy also penalizes things that should be far apart but are not — repulsive forces. That is why UMAP preserves global structure better: it pushes apart points that do not belong together, preserving large-scale cluster relationships, not only local neighborhoods.

The Two Knobs

n_neighbors (default 15): number of neighbors that define local structure. Small values (5–15) emphasize fine local detail — more cluster detail, sometimes more noise. Large values (50–200) capture broader topology — big picture of cluster relationships, less internal detail. Most important parameter.

min_dist (default 0.1): how tightly points can pack in embedding. Set 0.0 for dense, well-separated clusters with clear boundaries. Set 0.5 or higher if you want continuous structure inside clusters instead of hard edges.

# pip install umap-learn
import umap
from sklearn.datasets import load_digits

X, y = load_digits(return_X_y=True)

# 2D visualization
reducer = umap.UMAP(
    n_neighbors=15,
    min_dist=0.1,
    n_components=2,
    metric='euclidean',
    random_state=42
)
X_2d = reducer.fit_transform(X)
print(f"UMAP 2D shape: {X_2d.shape}")

# Feature engineering (higher-D) — this is what t-SNE CAN'T do
reducer_feat = umap.UMAP(n_components=10, random_state=42)
X_features = reducer_feat.fit_transform(X)
print(f"UMAP 10D feature shape: {X_features.shape}")
# Feed X_features into any classifier — these are usable features

# Supervised mode — uses labels to guide the embedding
reducer_sup = umap.UMAP(n_components=2, random_state=42)
X_sup = reducer_sup.fit_transform(X, y=y)
print(f"Supervised UMAP shape: {X_sup.shape}")
# Class boundaries become much sharper — great for exploration

Supervised mode matters. Pass labels to fit_transform, and UMAP adjusts neighbor graph to favor same-class connections. Result: cleaner class boundaries, more interpretable structure. Not cheating. Using available information to draw better map.

Honest answer: use UMAP by default. Use t-SNE for second opinion on visualization, or when reviewer asks. Run both with multiple parameter settings before making biological, business, or scientific claims from plots.

LDA: Supervised Dimensionality Reduction

Everything so far is unsupervised — PCA, t-SNE, UMAP ignore class labels. Linear Discriminant Analysis flips goal: use labels to find directions that best separate classes. Different goal. Different math. Sometimes much better results.

Fisher's Criterion

Imagine two point clouds in 2D — class A and class B. You must project them onto one line in 1D so classes separate cleanly. Which line?

PCA chooses direction of maximum overall variance. That direction may have total class overlap — big spread, but all within-class spread. LDA does better: find direction where class means are far apart — high between-class scatter, S_B — and points inside each class stay tight — low within-class scatter, S_W.

Formally, LDA finds vector w that maximizes wᵀS_Bw / wᵀS_Ww. This is generalized eigenvalue problem. It yields at most C − 1 components, where C is number of classes. Binary classification? One component. Ten digit classes? At most nine. Hard ceiling. Also LDA's strength: extreme compression. Also limitation.

from sklearn.discriminant_analysis import LinearDiscriminantAnalysis
from sklearn.datasets import load_digits

X, y = load_digits(return_X_y=True)

# 10 digit classes → maximum 9 LDA components
lda = LinearDiscriminantAnalysis(n_components=9)
X_lda = lda.fit_transform(X, y)
print(f"LDA shape: {X_lda.shape}")  # (1797, 9)

# These 9 dimensions are OPTIMIZED for separating the 10 digit classes.
# Compare: PCA needed ~28 components for 95% variance.
# LDA often outperforms PCA for classification despite having far fewer dimensions.

LDA assumes each class follows Gaussian distribution with equal covariance matrices. When that is roughly true — more often than you expect — LDA works very well. Nine dimensions versus 28 from PCA, and those nine target class boundaries instead of general variance. When assumptions fail badly — curved decision boundaries, wildly different class shapes — LDA underperforms.

Name collision warning. LDA in dimensionality reduction means Linear Discriminant Analysis (Fisher, 1936). LDA in NLP means Latent Dirichlet Allocation (Blei, 2003). Different algorithms. Same acronym. In this chapter, LDA always means Fisher's method. In topic modeling talk, LDA means Blei's.

ICA: The Cocktail Party Solver

PCA finds uncorrelated directions. ICA finds statistically independent sources. Difference matters. Uncorrelated means zero covariance — no linear relationship. Independent means knowing one signal tells you nothing about others — not even nonlinear relationships. Independence is much stronger.

The Cocktail Party Problem

Three people talk at party. Three microphones sit around room. Each records different mix of three voices: mic1 = 0.7·Alice + 0.2·Bob + 0.1·Carol. You have mixed recordings. You do not know mixing weights. Can you recover original voices?

ICA says yes, under one key assumption: original sources must be non-Gaussian. Why? Central Limit Theorem. Mix independent signals, mixture moves toward Gaussian. So most non-Gaussian direction in mixed data corresponds to recovered source. ICA finds directions that maximize non-Gaussianity, measured by kurtosis or negentropy, and those directions recover original independent signals. FastICA does this efficiently with fixed-point iteration.

from sklearn.decomposition import FastICA
import numpy as np

# Simulate the cocktail party: 3 sources, 3 mixtures
np.random.seed(42)
t = np.linspace(0, 1, 1000)
s1 = np.sin(2 * np.pi * 5 * t)            # sine wave
s2 = np.sign(np.sin(2 * np.pi * 3 * t))   # square wave
s3 = np.random.laplace(size=1000)          # noise burst (non-Gaussian!)
S = np.c_[s1, s2, s3]

A = np.array([[1, 0.5, 0.3],
              [0.5, 1, 0.7],
              [0.3, 0.7, 1]])  # unknown mixing matrix
X = S @ A.T  # observed mixtures

# ICA recovers the sources (up to sign and permutation)
ica = FastICA(n_components=3, random_state=42)
S_recovered = ica.fit_transform(X)
print(f"Recovered sources shape: {S_recovered.shape}")  # (1000, 3)
# Columns of S_recovered ≈ original sources, possibly reordered and sign-flipped

ICA is specialist tool. Use it for EEG/MEG signal processing — separating brain signals from eye blinks and muscle artifacts — audio source separation, or financial factor discovery. Do not use it for general dimensionality reduction. It needs number of sources to equal number of observations, and sources must be non-Gaussian. If not, choose something else.

NMF: Parts-Based Decomposition

Problem with PCA: components can have negative values. Decompose face image with PCA and first eigenvector may load positive on nose and negative on eyes. What is "negative eye"? Nothing intuitive. PCA components are holistic — global patterns of variation. Individual entries are hard to interpret.

Non-negative Matrix Factorization (NMF) adds one constraint: everything stays non-negative. Original data, basis vectors, coefficients — all ≥ 0. Small change. Big effect on what algorithm learns.

Instead of holistic eigenfaces mixing positive and negative regions, NMF discovers parts. On face images, components become eyes, noses, mouths, jawlines — recognizable pieces that add together to build face. On text data, word-frequency matrices, components become topics — word groups that co-occur because they describe same subject.

from sklearn.decomposition import NMF
from sklearn.datasets import load_digits

X, y = load_digits(return_X_y=True)
# Digits pixel values are already non-negative (0-16 grayscale)

nmf = NMF(n_components=15, random_state=42, max_iter=500)
W = nmf.fit_transform(X)     # (1797, 15) — coefficients per sample
H = nmf.components_           # (15, 64) — the 15 basis "parts"

print(f"Coefficients shape: {W.shape}")
print(f"Components shape: {H.shape}")
print(f"Reconstruction error: {nmf.reconstruction_err_:.2f}")

# Each row of H is a 8×8 "part" image
# Unlike PCA eigenvectors, these parts are all non-negative —
# you can visualize them as actual image patches

NMF is right tool when data is naturally non-negative — pixel intensities, word counts, chemical concentrations, user-item ratings — and you want components that are interpretable by construction. Show NMF component to domain expert and they usually recognize it. Show PCA eigenvector and get blank stare.

Topic modeling connection is direct. Apply NMF to term-document matrix, TF-IDF, and components become topics. Each topic is word distribution, non-negative. Each document is topic mixture, also non-negative. Simple. Effective. Good way to find themes in text corpus.

Random Projections: The Brute Force Shortcut

Every method so far learns from data — PCA learns covariance structure, t-SNE learns neighborhood probabilities, UMAP learns nearest-neighbor graph. Learning costs time and compute. What if you have neither?

Johnson-Lindenstrauss lemma says something wild: project n points from high-D to O(log n / ε²) dimensions with random matrix, and all pairwise distances stay within factor (1 ± ε). No learning. No data dependence. Generate random matrix. Multiply. Done.

from sklearn.random_projection import GaussianRandomProjection
from sklearn.datasets import load_digits
import numpy as np

X, y = load_digits(return_X_y=True)

# Project from 64D to ~25D using a random matrix
rp = GaussianRandomProjection(n_components=25, random_state=42)
X_rp = rp.fit_transform(X)
print(f"Random projection shape: {X_rp.shape}")  # (1797, 25)

# Check distance preservation
from sklearn.metrics import pairwise_distances
D_orig = pairwise_distances(X[:100])
D_proj = pairwise_distances(X_rp[:100])
ratios = D_proj / (D_orig + 1e-10)
print(f"Distance ratio — mean: {ratios[ratios > 0].mean():.3f}, "
      f"std: {ratios[ratios > 0].std():.3f}")
# Ratios should be close to 1.0 — distances are approximately preserved

Random projection is fastest dimensionality reduction possible. O(n·d·k) — one matrix multiply. No iteration. No convergence. No hyperparameters beyond target dimension. It does not find best projection — PCA does better for same number of components — but with million samples, ten thousand features, and seconds to spare, random projection gets you most of way at fraction of cost.

I think of it as "good enough" option. When speed matters more than optimality, or you need quick baseline before fancier method, random projection helps.

Choosing a Method — The Decision Framework

Want formula? None. Better: decision table that matches what works in practice, then blunt meta-advice.

Meta-advice matters more than table: evaluate by downstream performance. Dimensionality reduction is preprocessing step, not final answer. Run PCA → train model → measure metric. Run UMAP → train model → measure metric. Method that wins on metric you care about wins. Mathematical elegance does not matter if task does not improve.

I usually try 2–3 methods and compare. No formula tells you in advance which one will win on your data. Table gives starting point. Downstream metric gives answer.

One more thing: sometimes dimensionality reduction makes things worse. If features are already clean, fairly low-dimensional, and informative, compression only loses signal. Always compare against "no reduction" baseline. If baseline wins, let it win. No prize for fancier pipeline.

Putting It All Together

"""
Full workflow: compare PCA, LDA, UMAP, and no-reduction head-to-head.
"""
import numpy as np
from sklearn.datasets import load_digits
from sklearn.preprocessing import StandardScaler
from sklearn.decomposition import PCA
from sklearn.discriminant_analysis import LinearDiscriminantAnalysis
from sklearn.ensemble import GradientBoostingClassifier
from sklearn.model_selection import cross_val_score
from sklearn.pipeline import Pipeline

X, y = load_digits(return_X_y=True)

# --- Baseline: all 64 features ---
pipe_full = Pipeline([
    ('scaler', StandardScaler()),
    ('clf', GradientBoostingClassifier(n_estimators=100, random_state=42))
])
score_full = cross_val_score(pipe_full, X, y, cv=5).mean()

# --- PCA: keep 95% variance ---
pipe_pca = Pipeline([
    ('scaler', StandardScaler()),
    ('pca', PCA(n_components=0.95)),
    ('clf', GradientBoostingClassifier(n_estimators=100, random_state=42))
])
score_pca = cross_val_score(pipe_pca, X, y, cv=5).mean()

# --- LDA: max 9 components for 10 classes ---
pipe_lda = Pipeline([
    ('scaler', StandardScaler()),
    ('lda', LinearDiscriminantAnalysis(n_components=9)),
    ('clf', GradientBoostingClassifier(n_estimators=100, random_state=42))
])
score_lda = cross_val_score(pipe_lda, X, y, cv=5).mean()

print(f"Full features (64D): {score_full:.3f}")
print(f"PCA (~28D):          {score_pca:.3f}")
print(f"LDA (9D):            {score_lda:.3f}")
# LDA often wins — 9 dimensions PURPOSE-BUILT to separate digits
# vs. 28 PCA dimensions that maximize variance (not separation)
