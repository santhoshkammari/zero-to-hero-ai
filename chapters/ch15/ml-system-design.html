Key takeaways and essential system design references

Third beautifully trained model died when production traffic hit. I stopped ignoring everything around model. Could discuss attention mechanisms for hours, could not explain how prediction reaches user at two in morning. Here dive.

ML system design = discipline of architecting full lifecycle of machine learning product — from moment business person says "we want to predict X" to moment prediction reaches real user, and moment after that, when you learn whether prediction was any good. Term took formal shape after Google published landmark 2015 paper on technical debt in ML systems. Since then, core interview topic at major tech companies.

The Restaurant That Only Hired a Chef

Open restaurant. Pour every dollar into hiring world's greatest chef. Michelin stars. Flawless technique. Food coming out of kitchen: extraordinary.

But no host to seat guests. No waiter to take orders. No refrigerator — ingredients sit on counter. No dishwasher — plates pile up. No menu, so customers do not know what is available. Nobody checks whether customers liked food.

Chef brilliant. Restaurant fails in week.

Same in most ML projects. Model — chef — gets 95% of attention. Everything around it — data collection, feature engineering, serving infrastructure, monitoring, feedback — becomes afterthought. Google published diagram in 2015 paper every ML engineer should know. ML model code? Tiny rectangle in center. Everything around it — data verification, feature extraction, configuration, serving, monitoring — dwarfs it.

Took me two failed production deployments to learn this. I kept thinking "if model good enough, rest sorts itself out." Wrong. Mediocre model inside well-designed system beats brilliant model inside broken one. Every time. Brilliant model that cannot retrain when data shifts, cannot serve within latency budget, and cannot tell you when wrong — liability, not asset.

We keep coming back to restaurant. Chef matters. Restaurant around chef decides whether anyone eats.

From Business Problem to ML Problem

First step in ML system design — part most people rush — is to not talk about models. Talk about business.

Make it concrete. We work at small streaming service called StreamFlix. Product manager says: "People aren't finding movies they like. Can ML help?" Business problem. Job: translate it into ML problem.

Start tiny. StreamFlix has three users — Alice, Bob, Carol — and five movies. Alice watched and liked "The Matrix" and "Inception." Bob watched "The Matrix" and "Toy Story." Carol watched "Inception" and "Frozen." Full picture:

              Matrix  Inception  Toy Story  Frozen  Interstellar
Alice           ✓        ✓
Bob             ✓                    ✓
Carol                    ✓                     ✓

Business question: what should StreamFlix show each user next? Translation step. Turn "people aren't finding movies" into ranking problem — for each user, score every unseen movie and show highest-scoring ones first.

Single sentence — "score every unseen movie for each user and rank by score" — is ML problem formulation. It defines input (user-movie pair), output (relevance score), consumer (homepage), and task type (ranking). Without this clarity, teams wander for months. I once watched team spend six months building binary classifier — "will this user like this movie, yes or no?" — when product needed ranker. Different problems. Different loss functions, evaluation, and serving patterns.

Framework I use every time: What is input? What is output? Who consumes it? How fast do they need it? And question most people forget: what happens when model is wrong?

For StreamFlix, wrong prediction means user sees bad recommendation. Annoying, not catastrophic. For fraud detection system, wrong prediction means stolen money or blocked legitimate customer. Severity of being wrong shapes everything downstream — model complexity, threshold, human review. More on this when we walk through real system designs.

But first, need way to know whether system is good.

Three Layers of Truth: Metrics That Matter

Early-career trap got me. I optimized recommendation model until NDCG — Normalized Discounted Cumulative Gain, metric measuring ranking quality by giving more credit to relevant items higher in list — looked fantastic on held-out data. Shipped it. Homepage click-through rate dropped. Model predicted what users watched in past very well, but surfaced things they would actually click going forward very badly.

Problem: three metric layers. They do not always agree.

First layer: offline metrics — measured on historical data before deployment. For StreamFlix: precision@k (of top k movies we recommend, how many did user actually watch?) or NDCG (how good is ranking order?). ML engineers care about these. Cheap to compute. Fast to iterate on.

Second layer: online metrics — measured on live traffic after deployment. Click-through rate. Watch-through rate. Session length. Product managers care about these. Need real deployment, so expensive to measure.

Third layer: business metrics — things keeping company alive. Revenue. Subscriber retention. Churn reduction. Executives care about these. Often lag by weeks or months.

Make it concrete with StreamFlix. Suppose model gets perfect offline NDCG of 1.0 — it perfectly predicts order of movies users watched in past. Great. But after deployment, it keeps recommending "The Matrix" to everyone because most-watched movie in dataset. Bob clicks again because he liked it. Alice ignores it because she already watched it. CTR goes up slightly (Bob clicked), but subscriber retention tanks (Alice gets bored and cancels). Offline metric was perfect. Business metric went wrong way.

Critical question separating good ML teams from great ones: does improving offline metric actually improve online metric, which actually improves business metric? You must validate each link in this chain with A/B tests. If NDCG goes up but CTR does not move, offline metric is lying. Proxy drifted from reality.

Which offline metrics best predict which online outcomes? Depends on domain, product, and user population. Nobody has universal formula. Test it.

Unsatisfying. Next hard question: do we even have data to build system?

Where Signal Lives: Data Strategy

Before spending months of engineering, be brutally honest about three things. First: do we have enough data? If StreamFlix has three users and five movies, signal is too small for any model to learn from. Need critical mass. For recommendation systems, common rule: at least 10 interactions per user and 10 interactions per item before data becomes useful.

Second: is signal actually in data? Suppose StreamFlix only tracks which movies users clicked, not which ones they watched to completion. Clicks are noisy — people click things they end up hating. Train on click data, model learns clickbait, not quality. Signal we need (did user enjoy movie?) might not be in data we have.

Third: can organization act on predictions? If StreamFlix homepage is hardcoded and frontend team cannot add dynamic recommendations for six months, best model in world sits in warehouse.

Expand StreamFlix example. Suppose we now have 10,000 users and 2,000 movies. Enough data. But what data? We have explicit signals — ratings users gave (1 to 5 stars). High quality, but sparse. Most users rate fewer than 1% of movies. We also have implicit signals — what users watched, how much they watched, what they searched for, what they added to list. Noisier, but abundant.

User who watched 95% of movie probably liked it. User who quit after 8 minutes probably did not. But user who watched 50%? Ambiguous. Maybe movie was too long. Maybe they fell asleep. Maybe they loved it but had to catch flight. Implicit signals are ambiguous, and interpreting them is more art than science.

Data strategy for StreamFlix: data warehouse (BigQuery or Snowflake-like) stores all historical user interactions — every watch, click, search, and rating. Source of truth for training. streaming layer (Kafka or Kinesis) captures real-time events — what user is doing now. data validation system checks incoming data for anomalies — did watch-time column start emitting nulls because upstream service changed schema? If corruption reaches model, model learns garbage.

Cardinal rule: data should be immutable and versioned. If we cannot reproduce exact dataset model was trained on, we cannot debug model. Sounds obvious until 3 AM, model retrained overnight starts producing bizarre predictions, and nobody can tell you whether training data changed.

Data in place. Next question: what should model actually look at?

Features: Compute Once, Serve Everywhere

Back to StreamFlix. Raw data: Alice watched "The Matrix" on Tuesday at 9 PM on her phone and finished 92% of it. Raw event log. Model does not see raw events. It sees features — engineered signals derived from raw data.

For Alice, we might compute: total movies watched (2), average watch completion (88%), favorite genre (sci-fi, from her history), days since last watch (3), typical watch time (evenings), device (mobile). For "The Matrix" we might compute: average rating (4.2), total watches (847), genre (sci-fi/action), release year (1999), trending score (watched a lot this week?). For pair Alice + "The Matrix," we might compute: genre match (high — Alice likes sci-fi), collaborative signal (users similar to Alice watched this).

Problem that kept biting me: during training, I compute these features in batch pipeline — PySpark or SQL query over historical data. During serving, I need same features in real time — fast lookup when Alice opens app. If I write feature logic twice — once in PySpark for training, once in Java or Go for serving — they will diverge. Different null handling. Different float precision. Different timezone assumptions. Model sees different feature values in production than during training. This is training-serving skew. Silent killer of ML systems.

The fix is a feature store — a system that computes features in one place and serves them to both training and inference. Think of it as the restaurant's prep station. The chef doesn't chop onions during service — the prep cook chops them once, stores them properly, and both the lunch service and the dinner service use the same onions. Same computation, no divergence.

# The feature store pattern: one computation, two access patterns

# TRAINING: fetch historical features with point-in-time correctness
# "What did Alice's features look like at 9 PM on Tuesday?"
training_df = feature_store.get_historical_features(
    entity_df=users_with_timestamps,
    features=[
        "user:movies_watched_30d",
        "user:avg_completion_rate",
        "user:favorite_genre",
        "item:avg_rating",
        "item:trending_score",
    ],
)

# SERVING: fetch latest feature values, low-latency lookup
# "What do Alice's features look like right now?"
online_features = feature_store.get_online_features(
    features=["user:movies_watched_30d",
              "user:avg_completion_rate"],
    entity_rows=[{"user_id": "alice_42"}],
)
# Same logic produced both. That's the whole point.

The "point-in-time correctness" in the training path is a subtlety worth pausing on. When we build training data, we need Alice's features as they were at the time she watched the movie, not her features today. If we accidentally use today's features to predict a decision she made three months ago, we've introduced data leakage — the model looks unrealistically good offline because it's peeking into the future. Then it falls apart in production where no future data is available. I've been bitten by this one, and the debugging took days because the offline metrics looked amazing.

With features in place, we need to pick a model.

Choosing Your Weapon: Model Selection

This is the part most people want to start with. I understand the temptation — models are the exciting part. But notice how much groundwork we laid before we got here. The model is the chef. We've been building the restaurant.

For StreamFlix, here's how I'd think about model selection. Start with the dumbest thing that could work. For recommendations, that's "show the most popular movies." No ML required. This is our baseline, and it's load-bearing infrastructure — not a stepping stone we'll discard, but a fallback that keeps the product running when the fancy model crashes at 2 AM.

Next step up: collaborative filtering. Users who watched similar movies to Alice probably share her taste. We can find them by computing similarity between user watch histories, then recommend what those similar users watched that Alice hasn't seen yet. This is still relatively cheap to run and surprisingly effective. For our toy StreamFlix, Alice watched "The Matrix" and "Inception." Bob also watched "The Matrix." So Bob is somewhat similar to Alice. Bob watched "Toy Story." Maybe Alice would like "Toy Story" too.

When collaborative filtering hits its ceiling — it struggles with new users who have no history, and with new movies nobody has watched yet — we move to a learned ranking model. This takes all those features we engineered (user features, item features, pair features) and learns a function that predicts relevance. A gradient boosted tree like XGBoost is a strong default for tabular features. A two-tower neural network — one tower encoding the user, one encoding the item, trained so that the dot product of their embeddings predicts relevance — is the current industry workhorse for large-scale retrieval.

Key principle: match model complexity to available data and serving constraints. 500-million-parameter neural ranker is absurd for StreamFlix with 10,000 users. Logistic regression is probably too weak for Netflix with 200 million users. Right model is one where extra complexity stops improving metric that matters — online metric, not offline one.

Temptation for more complex model still trips me. Antidote: always have baseline deployed and running. If fancy model cannot clearly beat it on business metric, fancy model does not ship.

The Training Pipeline

In Kaggle, training happens once. Download dataset. Run notebook. Submit predictions. Done. In production, training is recurring pipeline — infrastructure that runs on schedule, autonomously, while you sleep.

For StreamFlix, training pipeline might look like this: every night at midnight, scheduled job pulls last 90 days of user interactions from data warehouse, joins them with features from feature store (using point-in-time correctness), trains ranking model, evaluates it against held-out week, and — if evaluation passes quality gate — registers new model artifact in model registry.

Model registry matters. It is versioned catalog of every model artifact team has produced. Think of it as Git, but for trained models. Each entry records model version, training data version, hyperparameters, evaluation metrics, and who approved deployment. When production model starts behaving strangely, you can roll back to previous version in minutes because it sits in registry.

# A training pipeline in pseudocode

def nightly_training_pipeline():
    # 1. Pull fresh training data
    train_data = warehouse.query(
        "SELECT * FROM interactions WHERE date > today - 90"
    )

    # 2. Join with features (point-in-time)
    features = feature_store.get_historical_features(train_data)

    # 3. Train
    model = RankingModel()
    model.fit(features, labels=train_data["watched"])

    # 4. Evaluate against held-out data
    metrics = evaluate(model, holdout_data)

    # 5. Quality gate — only register if it beats the current model
    if metrics["ndcg"] > current_production_model_ndcg * 0.98:
        registry.register(model, metrics=metrics, data_version="v42")
    else:
        alert_team("New model failed quality gate", metrics)

That 0.98 threshold in quality gate matters. We allow new model to be slightly worse on offline metric because offline metrics have noise. What we won't tolerate is significant regression. If new model is dramatically worse, something broke in data or training, and we want alert immediately — not after it serves bad recommendations to million users.

One more training trap: experiment tracking. When you run 50 experiments varying hyperparameters, feature sets, and model architectures, you need record of what you tried and what happened. MLflow and Weights & Biases are standard tools. Without them, you end up with folder of model files named model_v2_final_FINAL_really_final.pkl, and nobody can tell you which one is in production.

Batch, Real-Time, or Both

Back to StreamFlix restaurant. Two ways to run kitchen.

First approach: chef cooks every possible dish in advance. At midnight, kitchen prepares personalized meals for every registered customer and stores them in walk-in refrigerator. When customer arrives next day, waiter opens fridge and pulls out pre-made plate. Fast service. Cheap to run. But if customer came from gym and wants something high-protein instead of pasta you pre-made twelve hours ago, bad luck. That is batch serving.

Second approach: chef cooks to order. Customer arrives, places order, and kitchen fires up. Food is fresh and personalized to moment. But customer waits, kitchen must be staffed at all times, and bills are higher. That is real-time serving.

For StreamFlix recommendation system, batch serving means precomputing "top 20 recommendations" for every user overnight and storing them in fast key-value store like Redis. When Alice opens app, we look up her precomputed list. Response time: under millisecond. Infrastructure cost: low — batch job runs during off-peak hours.

But batch breaks here. Alice watched "Interstellar" an hour ago. Her tastes shifted toward cerebral sci-fi. Batch recommendations were computed twelve hours ago, before she watched it. We still show same list. Recommendations are stale.

Real-time serving computes recommendations when Alice opens app. Server fetches latest features — including fact that she watched "Interstellar" an hour ago — runs ranking model, and returns fresh recommendations. Response time: 50–200ms. Infrastructure: always-on model servers, feature store with low-latency online access, and careful model optimization to stay within latency budget.

For some problems, freshness is non-negotiable. Fraud detection at moment of credit card transaction. Content moderation before post goes live. Search ranking as user types. These need real-time inference. For others — weekly email of "movies you might like" — batch is more than sufficient.

Production reality: almost every system at scale uses hybrid. Computationally expensive first pass — candidate generation, where we narrow 10,000 movies to 500 possibilities — runs in batch. Lighter second pass — ranking those 500 by personalized relevance — runs in real time. Batch efficiency where freshness does not matter. Real-time precision where it does.

Netflix, YouTube, and LinkedIn all run some version of this hybrid pattern. When I first learned this, it felt like cheating — are you not supposed to pick one architecture? But pragmatism makes sense. Expensive computation (scanning millions of items) does not need to happen every request. Cheap computation (scoring 500 items) does. Split work where economics make sense.

Where draw batch-vs-real-time line? Ask: "If user's behavior changed five minutes ago, does prediction need to reflect that?" If yes, that component needs real-time. If not, batch is fine.

The System That Eats Its Own Tail

This debt hides well. This one scared me most when I understood it.

StreamFlix deploys recommendation model. Model learned that "The Matrix" is popular, so it recommends "The Matrix" to many users. Users click it — partly because interested, partly because it sits at top of page and they trust system. Click gets logged as positive interaction. Next night, model retrains on this data and concludes that "The Matrix" is even more popular than before. So it recommends it more aggressively. More clicks. More training signal. Loop tightens.

Meanwhile, "Interstellar" — movie many users would love — never gets recommended because it was not popular enough to surface in first round. Nobody clicks it because nobody sees it. Model never learns that users like it. It lives in recommendation graveyard.

This is feedback loop. Model predictions influence user behavior, which generates training data for next model version. System eats own tail. Result: rich-get-richer dynamic where popular items get more popular and niche items get buried — regardless of quality.

Direct feedback loops are simple: model predicts → user acts on prediction → model trains on that action. Indirect feedback loops are sneakier. Suppose StreamFlix has separate model predicting churn risk. Recommendation model shows popular content to keep engagement up. Churn model sees that engaged users who watch popular content tend not to churn. So churn model tells product team: "everything is fine." But under surface, users who prefer niche content are quietly leaving because they never see anything they like. Recommendation model behavior infected churn model training data, and neither model flags problem.

Fixes are not glamorous, but critical. First: exploration traffic. Reserve 5–10% of recommendations for random or diverse picks model would not have chosen. This ensures unpopular items get exposure and model gets training signal on them. Second: diversity injection. After model ranks items, apply post-processing step so final list is not dominated by single genre or single item. Third: feedback loop metrics. Track how concentrated recommendations are. If top 1% of items get 90% of impressions, loop is tightening. Sound alarm.

Detecting indirect feedback loops is hard. I have not found reliable way beyond careful logging and skeptical analysis. Best defense I know: assume they exist and build monitoring that surfaces their effects.

Patterns That Survived Production

Work on enough ML systems, same architectural patterns recur. These are not theoretical — they survived real traffic, real failure modes, and real cloud bills. Four matter most.

Retrieve → Rank

Single most important pattern in production ML. We have been building toward it with StreamFlix all along.

Problem: StreamFlix has 2,000 movies. For each user, we want score for every movie. Ranking model uses 200 features per user-movie pair and takes about 0.1ms per prediction. Scoring all 2,000 movies takes 200ms. Right at edge of latency budget for real-time request. But if StreamFlix grows to 100,000 movies, scoring takes 10 seconds. Page never loads.

Solution: don't score everything. Split problem into two phases.

Phase 1 — Retrieval: Use cheap, fast model to narrow 100,000 movies to 500 candidates. Model does not need precision first — it needs speed and high recall (it should include most good movies, even if it also includes some bad ones). Two-tower model works well here: encode user as embedding, encode each item as embedding (pre-computed in batch), and find 500 items whose embeddings are closest to user's. This is approximate nearest neighbor (ANN) search, and libraries like FAISS and ScaNN can do it in under 10ms over millions of items.

Phase 2 — Ranking: Apply expensive, feature-rich model to those 500 candidates. This model uses full 200 features — user history, item metadata, context (time of day, device), cross-features (genre match, collaborative signals). Scoring 500 items at 0.1ms each takes 50ms. Well within budget.

Often there is Phase 3 people forget: business rules. After ranker scores items, we apply constraints — diversity (do not show five sci-fi movies in row), freshness (include new releases), legal restrictions (age-gating), and deduplication (do not show same movie twice). Host seating guests — chef picked dishes, host arranges dining experience.

Why split it? Running 200-feature neural ranker on 100,000 candidates would cost fortune per request. Retrieval stage makes ranking stage economically feasible. Whole trick.

Cascade: Cheap First, Expensive Last

Cascade pattern extends retrieve→rank idea to any system with series of increasingly expensive models, each filtering traffic for next. Think series of increasingly fine sieves. Coarse sieve is cheap and catches easy cases. Fine sieve is expensive and handles hard ones.

Content moderation is canonical example. Social media platform might process billions of posts per day. Running state-of-the-art transformer model on every post would require staggering compute. Instead:

# Content moderation as a cascade of sieves

def moderate(content):
    # Sieve 1: keyword + hash matching (~0.1ms)
    # Catches known-bad content: blocklisted words, known
    # CSAM hashes, exact-match spam. Handles ~60% of violations.
    if keyword_and_hash_filter(content):
        return "BLOCKED"

    # Sieve 2: lightweight classifier (~5ms)
    # A small model (distilled BERT or logistic regression on
    # TF-IDF features). Handles another ~30% of violations.
    score = fast_classifier.predict(content)
    if score > 0.95:
        return "BLOCKED"
    if score < 0.05:
        return "APPROVED"

    # Sieve 3: heavy model for the ambiguous middle (~100ms)
    # Full transformer, possibly multimodal (text + image).
    # Only sees the ~10% of traffic that made it past the
    # first two sieves.
    score = deep_model.predict(content)
    if score > 0.7:
        return "HUMAN_REVIEW"
    return "APPROVED"

Result: 90% of traffic is handled by cheap models. Expensive deep model processes hardest 10%. GPU bill drops by order of magnitude while quality stays constant on cases that matter. Most ambiguous content — 2% where even deep model is not confident — goes to human reviewers. They make final call, and decisions feed next round of training data.

Sieve analogy again: first sieve has big holes (catches obvious violations, lets everything else through). Second sieve has medium holes. Third has tiny holes. Each sieve is more expensive to operate, but sees less traffic. Economics work well.

Shadow Mode: Test Before You Trust

Before promoting new model to production, run it in shadow mode. New model receives real traffic and makes real predictions, but predictions are logged, not served. Existing model still handles all users. Compare shadow model outputs against live model and ground truth.

Shadow mode is safer than A/B testing for initial deployment because zero users see bad predictions. You discover problems — latency spikes, edge case failures, unexpected output distributions — before any user is affected. Only when shadow model proves itself across days of real traffic do you promote it to small percentage of users for proper A/B test.

Back to restaurant: like having new chef prepare dishes alongside existing chef for week. New chef's food goes to tasting panel, not customers. If tasting panel approves, new chef starts serving few tables. Then more. Then all.

Human-in-the-Loop: Know When to Defer

For high-stakes decisions — medical diagnoses, loan approvals, content moderation edge cases — model should not have final word. Route low-confidence predictions to human reviewers. Model handles 85% of cases where confidence is high. Humans handle rest. Not crutch — design choice that acknowledges limits of model training data.

The hard part is calibrating the confidence threshold. Set it too high and humans drown in reviews. Set it too low and the model makes confident-but-wrong calls on cases it shouldn't have touched. The threshold is a knob you tune continuously based on the cost of human review versus the cost of a wrong automated decision.

Walking the Full Design: Four Real Systems

Everything we've built so far — problem formulation, metrics, data strategy, features, models, serving, feedback loops, patterns — comes together when you design a complete system. Let's walk through four systems that come up over and over in interviews and in practice. For each one, I'll trace the path from business problem to architecture, using the framework we've developed.

Recommendation System (StreamFlix)

We've been building this throughout. Let me tie it all together.

Business problem: Users can't find movies they like. Subscriber retention is dropping.

ML formulation: For each user, rank all unseen movies by predicted relevance. Show the top 20 on the homepage.

Metrics: Offline — NDCG@20 and Recall@50. Online — CTR on recommendations and watch completion rate. Business — subscriber retention and revenue per user.

Architecture: Hybrid batch + real-time. Candidate generation (two-tower model producing embeddings, ANN index for fast retrieval) runs in batch, refreshed every few hours. Ranking (feature-rich gradient boosted tree or neural ranker using 200+ features) runs in real-time when the user opens the app. A business rules layer handles diversity and freshness. The feature store handles both training-time and serving-time feature access, eliminating training-serving skew.

Feedback loop defense: 5% exploration traffic. Diversity injection in the business rules layer. Weekly audit of impression concentration — if the top 1% of movies receive more than 50% of impressions, the exploration rate increases automatically.

Fallback: If the ranking model is down, serve the batch-computed candidate list (stale but functional). If the batch pipeline also failed, fall back to "most popular this week." The restaurant always has food — even if it's from the backup menu.

Search Ranking

Business problem: Users search for movies but the results are poor. "Action movies with robots" returns irrelevant results.

ML formulation: Given a query and a corpus of movies, rank movies by relevance to the query.

Search ranking introduces a new stage we haven't discussed: query understanding. Before we retrieve anything, we need to understand what the user means. "Action movies with robots" needs to be parsed into intent: genre=action, theme=robots. Misspelled queries need correction. Ambiguous queries ("The Ring" — the horror movie or the jewelry?) need disambiguation. This is often a separate NLP model or a set of heuristics that enriches the raw query before it hits the retrieval stage.

Architecture: Query understanding → retrieval (BM25 keyword matching combined with dense retrieval using query and movie embeddings) → ranking (a learned ranker using query features, movie features, and click-through signals) → re-ranking (a personalization layer that adjusts for the specific user's taste). Each stage is a finer sieve — the same cascade principle at work.

Key metric subtlety: In search, the user tells you what they want (the query), unlike in recommendations where you have to guess. This means the feedback signal is richer — a click on result #5 implies that results #1 through #4 were probably less relevant. This pairwise signal is gold for training learning-to-rank models, which are trained on pairs of documents rather than individual relevance scores.

Fraud Detection

Business problem: Fraudulent transactions cost the platform $150 on average (chargebacks plus investigation). False positives cost $5 (support tickets plus lost goodwill).

ML formulation: For each transaction, predict the probability of fraud. Block or flag high-probability transactions before payment authorization.

Fraud has a fundamental challenge that StreamFlix doesn't: delayed labels. When a user watches a movie, we know almost immediately whether they liked it (did they finish it?). When a transaction happens, we don't know it was fraudulent until the chargeback arrives — days to weeks later. The model makes a decision in 100ms, but the ground truth takes 14 days to materialize. Training on incomplete labels is tricky. Some transactions labeled "not fraud" today will turn out to be fraudulent next week.

Architecture: Real-time serving is non-negotiable — we need to block fraud before the payment goes through. We use a cascade: a rules engine catches the obvious cases (blocked IP addresses, impossible shipping addresses, velocity limits like "5 purchases in 2 minutes from the same card"), a lightweight gradient boosted tree handles 85% of the remaining traffic, and a deep model handles the hardest 5%. Fallback: if the model server is unresponsive, the rules engine takes over alone — higher false positives, but no missed fraud.

Feature strategy: A mix of precomputed user-level features in the feature store (purchase history, device fingerprints, number of transactions in the last hour) and transaction-level features computed at request time (amount, merchant category, time of day). The velocity features — "how many purchases has this card made in the last 10 minutes?" — are the most predictive and the hardest to compute in real-time. They require a streaming aggregation system that maintains running counts and sums over sliding time windows.

Training: Daily retraining on a rolling 90-day window, with a staggered labeling pipeline. Transactions with no chargeback after 7 days are provisionally labeled "not fraud." As chargebacks arrive over the following weeks, labels are corrected and the model retrains with the updated ground truth.

Content Moderation

Business problem: Harmful content (hate speech, violence, spam) damages user trust and creates legal liability. The platform processes millions of posts per day.

ML formulation: For each piece of content (text, image, or video), predict the probability that it violates platform policy. Block, flag for review, or approve.

Content moderation is uniquely hard for three reasons. First, it's multimodal — a benign image with hateful overlaid text, or a video that's fine in the first 30 seconds and violent in the last 10. The model needs to understand multiple modalities and their interactions. Second, it's adversarial — bad actors actively try to evade detection by misspelling slurs, using code words, modifying images to defeat classifiers. The model is playing a game against opponents who adapt. Third, context matters enormously — the word "kill" means very different things in "I'd kill for a pizza" versus a direct threat.

Architecture: The cascade pattern — keyword and hash filter → lightweight text/image classifier → heavy multimodal transformer model → human review. The human reviewers serve a dual purpose: they handle edge cases and they generate the labeled training data that improves the model over time. This is the feedback loop working in our favor — every human decision makes the automated system better.

Key design choice: Content moderation has an asymmetric cost structure. Missing harmful content (false negative) can cause real-world harm and legal liability. Over-blocking (false positive) frustrates users but is less dangerous. This asymmetry pushes the thresholds toward higher recall (catch more bad content) at the expense of precision (some good content gets flagged too). The human review layer absorbs the resulting false positives.

The Flywheel

ML systems aren't shipped and forgotten. The best ones create a flywheel: deploy a model → collect user interactions → use those interactions to improve features and labels → retrain a better model → deploy it → collect more interactions. Each cycle makes the system smarter. The data moat deepens.

At StreamFlix, the flywheel looks like this. V1 launches with collaborative filtering. Users click on recommendations, generating engagement data. That data reveals patterns the V1 model couldn't capture — maybe users who watch documentaries on weekday evenings prefer different genres on weekends. V2 adds time-of-day features and a more expressive model. The improved recommendations drive higher engagement, which generates more data, which enables V3.

The flywheel is why the first version doesn't need to be perfect. It needs to be good enough to generate useful data. The data makes the second version better. The second version generates better data. And so on.

This is also why monitoring matters so much. If you don't measure how V1 is performing in production — which recommendations get clicked, which get ignored, where users bounce — you have no data to build V2 from. The monitoring layer isn't overhead. It's the engine of improvement.
