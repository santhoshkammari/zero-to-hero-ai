Evaluating Generative Models

Chapter 15: Generative Models 8 subtopics
TL;DR

Evaluating generative models is fundamentally harder than evaluating classifiers — there's no single "accuracy" number. You need to measure quality (are individual samples good?), diversity (does the model cover the full distribution?), and faithfulness (does it follow the prompt?). FID is the standard automatic metric for image generators. CLIP Score measures text-image alignment. Human evaluation remains the gold standard but is expensive. Every production generative system needs a multi-metric evaluation pipeline combining automated scores with targeted human review.

Why Evaluation Is Hard

With a classifier, you have labels. You predict "cat" or "dog," compare against ground truth, and compute accuracy. Done. With a generative model, what's "correct"? If you prompt "a cat sitting on a moon" and get back an image — who says what the right answer looks like?

This is the core problem: there is no ground truth for generated content. A good image generator should produce outputs that are:

  • High quality — each individual sample looks realistic, sharp, well-composed
  • Diverse — the model doesn't just memorize and replay training data; it covers the full distribution of possible outputs
  • Faithful — if you gave it a text prompt, the output actually matches what you asked for

These three goals are in tension. You can get perfect quality by memorizing 10 training images and replaying them — but diversity is zero. You can get perfect diversity by generating random noise — but quality is zero. And you can get a photorealistic image that completely ignores your prompt — high quality, zero faithfulness.

Human evaluation is the gold standard. Show people generated images, have them rate quality, compare against real images, assess prompt adherence. But it's expensive, slow, and doesn't scale. You can't run human eval after every training checkpoint. So we need automatic metrics that correlate with human judgment. None of them are perfect. You need several, and you need to understand what each one actually measures.

Fréchet Inception Distance (FID)

FID is the metric you'll see in every generative modeling paper since 2018. If someone tells you their model's FID is 2.0 on ImageNet 256×256, that's state-of-the-art territory. If it's 50.0, the samples are visibly bad. Here's how it works.

The Idea

Take a pretrained InceptionV3 network (trained on ImageNet classification). Strip off the final classification layer and use the 2048-dimensional feature vector from the penultimate layer as a "perceptual representation" of each image. Now you have two sets of feature vectors:

  1. Features from real images (your validation set)
  2. Features from generated images

Model each set as a multivariate Gaussian: compute the mean vector μ and covariance matrix Σ. FID is then the Fréchet distance (also called Wasserstein-2 distance) between these two Gaussians:

FID = ||μ_real - μ_gen||² + Tr(Σ_real + Σ_gen - 2(Σ_real · Σ_gen)^(1/2))

Lower is better. FID = 0 means the distributions are identical (in Inception feature space). In practice, even comparing two different random samples from the same dataset gives FID > 0 due to finite sample effects.

Why It Works

The Inception features capture semantic and perceptual information — they encode things like "this region has a face" or "this texture looks like fur." By comparing the statistics of these features across the whole set, FID captures both quality and diversity in a single number:

  • If generated images are low quality (blurry, artifacts), their features will cluster differently from real images → high FID
  • If generated images lack diversity (mode collapse), their covariance will be smaller than the real distribution → high FID
⚠️ FID Pitfalls

Sample size matters. You need ~50,000 generated images for stable FID estimates. With 1,000 images, your FID can fluctuate by ±10 between runs. Always report the number of samples used. Preprocessing matters. Resizing images differently (bilinear vs. bicubic) can shift FID by several points. Use the same pipeline as the reference. The clean-fid library exists specifically to standardize this.

Computing FID in Practice

# Using clean-fid (recommended — handles preprocessing correctly)
# pip install clean-fid

from cleanfid import fid

# Compare a folder of generated images against a folder of real images
score = fid.compute_fid(
    fdir1="path/to/real_images/",
    fdir2="path/to/generated_images/",
    mode="clean",        # standardized preprocessing
    num_workers=4,
    batch_size=64
)
print(f"FID: {score:.2f}")

# Or against a precomputed reference (faster for repeated evaluation)
# First, precompute stats on your real dataset once:
fid.make_custom_stats(
    name="my_dataset",
    fdir="path/to/real_images/",
    mode="clean"
)
# Then compare against saved stats:
score = fid.compute_fid(
    fdir1="path/to/generated_images/",
    dataset_name="my_dataset",

)

Inception Score (IS)

Inception Score came before FID (Salimans et al., 2016) and you'll still see it reported, though it's less reliable. The idea has two parts:

  1. Quality: Feed each generated image through InceptionV3. If the image is clear and recognizable, the classifier should be confident — the output distribution p(y|x) should have low entropy (peaked at one class).
  2. Diversity: Across all generated images, the marginal distribution p(y) = E[p(y|x)] should have high entropy — you want images of many different classes, not just dogs.

IS combines both by computing the KL divergence between these distributions:

IS = exp(E_x[KL(p(y|x) || p(y))])

Higher is better. Real ImageNet images score around 250. A good GAN in 2018 scored around 50–100.

Why IS Is Problematic

IS has serious blind spots that make it unreliable as a standalone metric:

  • No comparison to real data. IS only looks at generated images. A model that generates perfect ImageNet images but completely ignores your specific dataset distribution still gets a high IS.
  • ImageNet bias. The Inception network was trained on ImageNet's 1,000 classes. If you're generating faces, medical images, or anything not in those classes, IS is basically meaningless — the classifier wasn't trained on your domain.
  • Insensitive to intra-class diversity. If your model generates 50,000 different-looking dogs but nothing else, IS might still give a decent score because each dog is classifiable and the images look "good" to Inception.

Use IS as a sanity check alongside FID, never as your primary metric.

CLIP Score

Once text-to-image models became mainstream (DALL·E 2, Stable Diffusion, Midjourney), people needed a way to measure: does the generated image actually match the text prompt? FID doesn't help here — it measures distributional similarity, not per-sample faithfulness.

CLIP Score is simple: take the CLIP model (which was trained to align text and image embeddings), encode both the text prompt and the generated image, and compute their cosine similarity:

CLIP Score = cos(CLIP_image(generated_image), CLIP_text(prompt))

Higher is better. A score of 0.30+ typically indicates strong text-image alignment. Below 0.20 means the image is ignoring the prompt.

# Computing CLIP Score
# pip install transformers pillow

import torch
from transformers import CLIPProcessor, CLIPModel
from PIL import Image

model = CLIPModel.from_pretrained("openai/clip-vit-large-patch14")
processor = CLIPProcessor.from_pretrained("openai/clip-vit-large-patch14")

def clip_score(image_path, text_prompt):
    image = Image.open(image_path)
    inputs = processor(text=[text_prompt], images=image, return_tensors="pt")

    with torch.no_grad():
        outputs = model(**inputs)
        # Normalized embeddings
        img_emb = outputs.image_embeds / outputs.image_embeds.norm(dim=-1, keepdim=True)
        txt_emb = outputs.text_embeds / outputs.text_embeds.norm(dim=-1, keepdim=True)

    similarity = (img_emb @ txt_emb.T).item()
    return similarity

# Example usage
score = clip_score("generated_cat_on_moon.png", "a cat sitting on the moon")
print(f"CLIP Score: {score:.4f}")
💡 CLIP Score Limitations

CLIP Score measures semantic alignment but not image quality. A blurry, low-resolution image that vaguely depicts the right concept can score higher than a photorealistic image with slightly different composition. Always pair CLIP Score with a quality metric like FID. Also, CLIP inherits biases from its training data — it may score certain concepts or demographics differently.

LPIPS Perceptual distance between two images Correlates well with human perception of similarity; per-pair metric Requires paired images (e.g., input → output); not a distribution metric Image-to-image tasks: super-resolution, inpainting, style transfer
KID Distribution similarity (like FID but kernel-based) Unbiased estimator; works with fewer samples (~1K); has confidence intervals Higher variance than FID; less widely reported When you can't generate 50K samples; small-scale experiments

Practical Evaluation Pipeline

Here's what a real evaluation pipeline looks like when you're developing a generative model. You don't run all of this every checkpoint — you tier your evaluations by cost.

Tier 1: Every Checkpoint (Cheap, Automated)

Generate a small batch (e.g., 1,000 images with fixed seeds) and eyeball a grid. Compute training loss and any lightweight metrics. This catches catastrophic failures — mode collapse, NaN outputs, sudden quality drops.

Tier 2: Periodic (Daily or Weekly)

Generate 50K images, compute FID and KID against your validation set. If text-conditioned, compute CLIP Score against a fixed set of prompts. Track trends over time.

Tier 3: Milestones (Before Releases)

Full human evaluation. A/B testing against your previous best model. Targeted failure mode analysis.

"""
Complete evaluation pipeline for a text-to-image model.
Run this after each training milestone.
"""
import os
import json
import torch
import numpy as np
from pathlib import Path
from cleanfid import fid
from PIL import Image
from transformers import CLIPProcessor, CLIPModel

class GenerativeEvalPipeline:
    def __init__(self, real_images_dir, output_dir="eval_results"):
        self.real_dir = real_images_dir
        self.output_dir = Path(output_dir)
        self.output_dir.mkdir(exist_ok=True)

        # Load CLIP for text-image alignment scoring
        self.clip_model = CLIPModel.from_pretrained("openai/clip-vit-large-patch14")
        self.clip_processor = CLIPProcessor.from_pretrained("openai/clip-vit-large-patch14")
        self.clip_model.eval()

    def compute_fid_score(self, generated_dir):
        """FID: distribution-level quality + diversity."""
        return fid.compute_fid(
            fdir1=self.real_dir,
            fdir2=generated_dir,
            mode="clean",
            num_workers=4,
            batch_size=64
        )

    def compute_clip_scores(self, image_paths, prompts):
        """CLIP Score: per-sample text-image alignment."""
        scores = []
        for img_path, prompt in zip(image_paths, prompts):
            image = Image.open(img_path)
            inputs = self.clip_processor(
                text=[prompt], images=image, return_tensors="pt"
            )
            with torch.no_grad():
                outputs = self.clip_model(**inputs)
                img_emb = outputs.image_embeds / outputs.image_embeds.norm(dim=-1, keepdim=True)
                txt_emb = outputs.text_embeds / outputs.text_embeds.norm(dim=-1, keepdim=True)
            scores.append((img_emb @ txt_emb.T).item())
        return {
            "mean": float(np.mean(scores)),
            "std": float(np.std(scores)),
            "min": float(np.min(scores)),
            "max": float(np.max(scores)),
            "scores": scores
        }

    def run_full_eval(self, generated_dir, prompts_file=None):
        """Run the complete evaluation suite."""
        results = {}

        # FID
        print("Computing FID...")
        results["fid"] = self.compute_fid_score(generated_dir)
        print(f"  FID: {results['fid']:.2f}")

        # CLIP Score (if prompts provided)
        if prompts_file:
            print("Computing CLIP Scores...")
            with open(prompts_file) as f:
                prompt_data = json.load(f)  # [{"image": "001.png", "prompt": "..."}, ...]

            image_paths = [os.path.join(generated_dir, p["image"]) for p in prompt_data]
            prompts = [p["prompt"] for p in prompt_data]
            results["clip"] = self.compute_clip_scores(image_paths, prompts)
            print(f"  CLIP Score: {results['clip']['mean']:.4f} ± {results['clip']['std']:.4f}")

        # Save results
        results_path = self.output_dir / "eval_results.json"
        with open(results_path, "w") as f:
            json.dump(results, f, indent=2)
        print(f"Results saved to {results_path}")

        return results

Human Evaluation

Automatic metrics correlate with human judgment, but they don't replace it. FID can't tell you that a model generates hands with six fingers. CLIP Score can't tell you that the generated image is aesthetically ugly despite matching the prompt. For anything going into production, you need humans in the loop.

Common Protocols

Mean Opinion Score (MOS): Show evaluators a single image, ask them to rate quality on a 1–5 scale. Simple but noisy — calibration varies across raters. Best for absolute quality assessment.

Two-Alternative Forced Choice (2AFC): Show two images side by side (from model A and model B), ask "which is better?" Removes the calibration problem — people are much better at relative judgments than absolute ones. This is what most serious evaluations use.

Elo Ratings: Run many 2AFC comparisons across multiple models and compute Elo ratings (like chess rankings). Each comparison updates the ratings. After enough comparisons, you get a robust ranking. This is what Chatbot Arena uses for LLMs and what image generation leaderboards increasingly adopt.

"""
Minimal Elo rating system for generative model comparison.
"""
import random
import math

class EloRating:
    def __init__(self, models, k=32, initial_rating=1500):
        self.ratings = {m: initial_rating for m in models}
        self.k = k
        self.history = []

    def expected_score(self, rating_a, rating_b):
        return 1.0 / (1.0 + math.pow(10, (rating_b - rating_a) / 400))

    def update(self, winner, loser):
        """Update ratings after a pairwise comparison."""
        exp_w = self.expected_score(self.ratings[winner], self.ratings[loser])
        exp_l = self.expected_score(self.ratings[loser], self.ratings[winner])

        self.ratings[winner] += self.k * (1 - exp_w)
        self.ratings[loser] += self.k * (0 - exp_l)

        self.history.append({"winner": winner, "loser": loser})

    def get_rankings(self):
        return sorted(self.ratings.items(), key=lambda x: -x[1])

    def generate_matchup(self):
        """Pick two models for the next comparison."""
        models = list(self.ratings.keys())
        return random.sample(models, 2)

# Example: comparing 4 diffusion model checkpoints
elo = EloRating(["sd-v1.5", "sd-v2.1", "sdxl", "sd3"])

# After collecting human judgments:
elo.update(winner="sdxl", loser="sd-v1.5")
elo.update(winner="sd3", loser="sdxl")
elo.update(winner="sd3", loser="sd-v2.1")
# ... hundreds more comparisons ...

for model, rating in elo.get_rankings():
    print(f"  {model}: {rating:.0f}")
📊 Scaling Human Eval

For production evaluation, you don't need to evaluate every generated image. Sample strategically: pick prompts that cover your known failure modes (complex scenes, text rendering, hands, unusual objects). 200–500 human judgments with 3 raters each is usually enough to get statistically significant differences between two models. Use inter-rater agreement (Krippendorff's alpha or Fleiss' kappa) to filter out unreliable annotators.

The Diversity–Quality Tradeoff

There's a fundamental tension in generative models that every evaluation must account for: you can trade diversity for quality, and most deployed systems do.

The Truncation Trick (GANs)

In StyleGAN, you sample latent codes from a normal distribution. The truncation trick scales these codes toward the mean: z_truncated = ψ * z where ψ < 1. At ψ = 1.0 you get full diversity but occasional bad samples. At ψ = 0.5 you get consistently good-looking images but less variety. At ψ = 0.0 every image is identical — the "average" face.

Classifier-Free Guidance (Diffusion Models)

This is the modern version of the same idea. During sampling, you interpolate between the conditional prediction (what matches the prompt) and the unconditional prediction:

noise_pred = unconditional + guidance_scale * (conditional - unconditional)

At guidance_scale = 1.0, you get the raw model output — diverse but sometimes ignoring the prompt. At guidance_scale = 7.5 (Stable Diffusion's default), you get sharp, prompt-faithful images but reduced variety. At guidance_scale = 20+, images become oversaturated and repetitive.

What This Means for Evaluation

FID captures this tradeoff directly. If you crank up guidance scale, per-image quality improves but diversity drops — and FID has a sweet spot. Plot FID vs. guidance scale and you'll see a U-shaped curve: too low and images are bad, too high and mode collapse hurts you.

This is why you should always report the guidance scale (or truncation ψ) alongside FID. A model claiming FID = 2.0 at guidance_scale = 30 is not really better than one with FID = 5.0 at guidance_scale = 3.0 — the first is just hiding diversity loss behind cherry-picked settings.

# Sweep guidance scale and plot FID to find the sweet spot
import matplotlib.pyplot as plt
from cleanfid import fid

guidance_scales = [1.0, 2.0, 3.0, 5.0, 7.5, 10.0, 15.0, 20.0]
fid_scores = []

for scale in guidance_scales:
    # generate_images() is your model's sampling function
    # generate_images(n=50000, guidance_scale=scale, output_dir=f"gen_{scale}")
    score = fid.compute_fid(
        fdir1="path/to/real_images/",
        fdir2=f"gen_{scale}/",
        mode="clean"
    )
    fid_scores.append(score)
    print(f"guidance_scale={scale:.1f}  FID={score:.2f}")

plt.plot(guidance_scales, fid_scores, "o-")
plt.xlabel("Guidance Scale")
plt.ylabel("FID (lower is better)")
plt.title("Diversity-Quality Tradeoff")
plt.grid(True)
plt.savefig("fid_vs_guidance.png", dpi=150, bbox_inches="tight")
⚠️ Don't Optimize for FID Alone

It's tempting to tune hyperparameters until FID is minimized. But FID is a proxy, not the goal. Models can "hack" FID by generating images that match Inception statistics without actually looking good to humans. Always validate FID improvements with visual inspection and, for important decisions, human evaluation. The best practice is to report FID, CLIP Score, and human preference together.
