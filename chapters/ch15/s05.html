Energy-Based Models

Chapter 15: Generative Models 4 subtopics
TL;DR

Energy-Based Models assign a scalar "energy" to every possible configuration of data — low energy means the model thinks the data is real, high energy means it thinks the data is fake. Unlike GANs or VAEs, EBMs don't need to compute exact probabilities. The catch: you can't easily sample from them or compute their normalizing constant. Training tricks like Contrastive Divergence, Score Matching, and Noise Contrastive Estimation sidestep these problems. Sampling uses MCMC methods like Langevin Dynamics. Classic architectures include Restricted Boltzmann Machines and the modern Joint Energy Model that does classification and generation in a single framework.

Why Energy? The Intuition

Every generative model answers one question: how likely is this data point? GANs answer indirectly (through a discriminator), VAEs answer with a lower bound, and autoregressive models factor the probability into a chain. Energy-Based Models take a different route — they assign a single number, the energy, to every possible input and say "lower energy means more likely."

Think of it like a landscape. Valleys represent real data — images of faces, coherent sentences, realistic audio. Hills represent garbage — random noise, distorted images, nonsense text. The model learns to sculpt this landscape so that real data sits in deep valleys and everything else sits on hilltops. You don't need to know the exact altitude of every point. You only need the shape of the terrain to be right.

This is powerful because many interesting distributions are hard to write down exactly but easy to compare. You might not know the probability of a specific face image, but you can tell that a clear photo has lower energy than a blurry one. EBMs exploit this asymmetry.

Foundations

The Energy Function

An Energy-Based Model defines a function E_θ(x) that maps any input x to a real-valued scalar. The subscript θ means the function is parameterized — typically by a neural network. The only requirement is that it outputs a single number. No sigmoid at the end, no softmax, no probability constraints. It can be any neural network that ends with a single output neuron.

import torch
import torch.nn as nn

class EnergyNetwork(nn.Module):
    """A neural network that maps inputs to scalar energy values."""
    def __init__(self, input_dim=784, hidden_dim=256):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(input_dim, hidden_dim),
            nn.LeakyReLU(0.2),
            nn.Linear(hidden_dim, hidden_dim),
            nn.LeakyReLU(0.2),
            nn.Linear(hidden_dim, hidden_dim),
            nn.LeakyReLU(0.2),
            nn.Linear(hidden_dim, 1)  # scalar energy output
        )

    def forward(self, x):
        # x: (batch, input_dim) -> energy: (batch, 1)
        return self.net(x)

The energy value itself has no inherent meaning — only the relative ordering matters. If E_θ(cat_image) = -5 and E_θ(noise) = 12, the model considers the cat more likely. The absolute numbers are arbitrary.

The Boltzmann Distribution

To convert energies into probabilities, we use the Boltzmann distribution (also called the Gibbs distribution):

p_θ(x) = exp(-E_θ(x)) / Z(θ)

The numerator exp(-E_θ(x)) converts energy to an unnormalized probability. Lower energy → higher exponential → higher probability. The denominator Z(θ) is the partition function — it sums (or integrates) exp(-E_θ(x)) over all possible inputs x to make the probabilities sum to 1.

# Conceptually (not tractable in practice):
# Z(θ) = ∫ exp(-E_θ(x)) dx   (continuous)
# Z(θ) = Σ exp(-E_θ(x))       (discrete)

Here's the problem: for images of size 28×28 with 256 intensity levels per pixel, the sum has 256^784 terms. That number has more digits than atoms in the observable universe. Computing Z(θ) exactly is impossible for any interesting problem. This intractable partition function is the central challenge of EBMs — and the reason we need clever training methods.

Unnormalized Density

Because Z(θ) is intractable, EBMs work with the unnormalized density exp(-E_θ(x)) whenever possible. Many useful quantities depend only on energy differences, not absolute values. For instance, comparing how likely two points are:

# Ratio of probabilities — Z cancels out!
# p(x₁) / p(x₂) = exp(-E(x₁)) / exp(-E(x₂))
#                 = exp(E(x₂) - E(x₁))

This cancellation is what makes EBMs practical. We don't need Z to compare data points, rank them, or take gradient steps that push energy down on real data and up on fake data.

Training Methods

Training an EBM means adjusting θ so that real data gets low energy and non-data gets high energy. The maximum likelihood gradient has two terms: push energy down on training examples (the "positive phase") and push energy up on samples from the model (the "negative phase"). The negative phase requires sampling from the current model — which brings us back to the intractable Z problem. Each training method below tackles this differently.

Contrastive Divergence

Contrastive Divergence (CD), invented by Geoffrey Hinton in 2002, is the workhorse training algorithm for EBMs. The insight: instead of running a Markov chain to equilibrium (which could take forever), run it for a few steps starting from real data.

import torch

def contrastive_divergence(model, x_real, lr=0.01, k=1, step_size=0.01):
    """
    CD-k training step for an energy-based model.
    k: number of MCMC steps (CD-1 uses k=1)
    """
    # Positive phase: compute energy on real data
    x_real.requires_grad_(True)
    E_pos = model(x_real)

    # Negative phase: run k steps of Langevin dynamics from real data
    x_neg = x_real.clone().detach()
    for _ in range(k):
        x_neg.requires_grad_(True)
        E_neg_sample = model(x_neg).sum()
        grad = torch.autograd.grad(E_neg_sample, x_neg)[0]
        # Langevin step: move toward lower energy + noise
        x_neg = x_neg.detach() - step_size * grad + \
                (2 * step_size) ** 0.5 * torch.randn_like(x_neg)

    E_neg = model(x_neg.detach())

    # Loss: push real energy down, fake energy up
    loss = E_pos.mean() - E_neg.mean()
    return loss

CD-1 (k=1) is biased but fast. CD-k with larger k reduces bias but costs more compute. In practice, CD-1 works surprisingly well for Restricted Boltzmann Machines. The bias comes from the fact that a single Markov chain step from real data doesn't give you a true sample from the model — it gives you a slightly corrupted version of the real data. But the gradient direction is close enough to be useful.

Score Matching

Score Matching avoids MCMC entirely. The score function is the gradient of the log-density with respect to the input: ∇_x log p(x). A beautiful property: the score doesn't depend on Z, because ∇_x log p(x) = -∇_x E_θ(x) (the Z term vanishes when you take the gradient with respect to x, not θ).

def score_matching_loss(model, x, noise_std=0.01):
    """
    Denoising score matching loss.
    Instead of matching the true data score (intractable),
    we match the score of a noisy version of the data.
    """
    # Add noise to data
    noise = torch.randn_like(x) * noise_std
    x_noisy = x + noise

    x_noisy.requires_grad_(True)
    energy = model(x_noisy).sum()
    # Score = -∇_x E(x)  (gradient of log unnormalized density)
    score = -torch.autograd.grad(energy, x_noisy, create_graph=True)[0]

    # Target score for Gaussian noise: (x_clean - x_noisy) / σ²
    target_score = -noise / (noise_std ** 2)

    # MSE between predicted and target score
    loss = ((score - target_score) ** 2).sum(dim=-1).mean()
    return loss

Score matching trains the model to predict the direction toward clean data from noisy data. It connects directly to diffusion models — which are essentially score matching at multiple noise levels. This is one of those ideas that seemed niche in 2005 and turned out to underpin the biggest advances in generative modeling two decades later.

Noise Contrastive Estimation

Noise Contrastive Estimation (NCE) turns the density estimation problem into a classification problem. Instead of computing Z, train a classifier to distinguish between real data and samples from a known noise distribution (like a Gaussian).

def nce_loss(model, x_real, noise_dist, num_noise=1):
    """
    Noise Contrastive Estimation loss.
    noise_dist: a distribution we can sample from and evaluate log-prob
    """
    batch_size = x_real.size(0)

    # Generate noise samples
    x_noise = noise_dist.sample((batch_size * num_noise,))

    # Log unnormalized density for real and noise
    log_p_real = -model(x_real).squeeze()  # -E(x) ≈ log p_unnorm(x)
    log_p_noise = -model(x_noise).squeeze()

    # Log noise density
    log_q_real = noise_dist.log_prob(x_real).sum(dim=-1)
    log_q_noise = noise_dist.log_prob(x_noise).sum(dim=-1)

    # Binary classification: real vs noise
    logit_real = log_p_real - log_q_real
    logit_noise = log_p_noise - log_q_noise

    loss_real = -torch.nn.functional.logsigmoid(logit_real).mean()
    loss_noise = -torch.nn.functional.logsigmoid(-logit_noise).mean()

    return loss_real + loss_noise

NCE effectively learns Z as a byproduct of training. The proof is elegant: at the optimal solution, the learned model equals the true data distribution, and the implicit normalizing constant equals the true Z. Word2Vec's negative sampling is actually a special case of NCE — that's how fundamental this technique is.

Stein Discrepancy

The Stein Discrepancy measures how well an EBM approximates the true data distribution using a kernel-based comparison. The key property of Stein's identity: for any smooth function f and distribution p, a certain expected value involving scores equals zero. When it doesn't equal zero, the discrepancy tells you how far apart two distributions are — all without needing Z.

The Kernelized Stein Discrepancy (KSD) uses a reproducing kernel Hilbert space to make this computable:

def kernelized_stein_discrepancy(model, x_samples, bandwidth=1.0):
    """Compute KSD between model and empirical distribution."""
    x = x_samples.requires_grad_(True)
    energy = model(x).sum()
    score = -torch.autograd.grad(energy, x, create_graph=True)[0]

    n = x.size(0)
    # RBF kernel
    diff = x.unsqueeze(0) - x.unsqueeze(1)  # (n, n, d)
    sq_dist = (diff ** 2).sum(-1)            # (n, n)
    K = torch.exp(-sq_dist / (2 * bandwidth ** 2))

    # KSD involves kernel, scores, and kernel gradients
    # Simplified: trace of Stein operator applied to kernel
    score_K = (score.unsqueeze(0) * K.unsqueeze(-1)).sum(1)
    ksd = (score * score_K).sum(-1).mean()
    return ksd

In practice, Stein Discrepancy is more commonly used for model evaluation and goodness-of-fit testing than for training. It answers the question: "Is my learned energy function a good fit for the data?" without ever computing Z.

Sampling from EBMs

Once you've trained an EBM, you need to generate new data by sampling from p_θ(x) ∝ exp(-E_θ(x)). Since you can't compute Z, you can't do inverse CDF sampling or rejection sampling efficiently. Instead, you use Markov Chain Monte Carlo (MCMC) methods that only need energy differences.

Langevin Dynamics

Langevin Dynamics is the most popular MCMC method for EBMs. It's essentially gradient descent on the energy function plus random noise. Start from a random point and iteratively take small steps toward lower energy regions, with noise to ensure you explore the full distribution (not the single lowest-energy point).

@torch.no_grad()
def langevin_sample(model, dim=784, n_steps=100, step_size=0.01,
                    n_samples=64, device='cpu'):
    """
    Generate samples using Langevin dynamics.
    """
    # Initialize from random noise
    x = torch.randn(n_samples, dim, device=device)

    for step in range(n_steps):
        x.requires_grad_(True)
        energy = model(x).sum()
        grad = torch.autograd.grad(energy, x)[0]

        # Langevin update: gradient step + Gaussian noise
        noise = torch.randn_like(x)
        x = x.detach() - step_size * grad + \
            (2 * step_size) ** 0.5 * noise

    return x.detach()

The noise term is critical. Without it, you'd do gradient descent and collapse to the single lowest-energy point. The noise ensures that the stationary distribution of the chain is exactly p_θ(x) — as long as you run enough steps and use a small enough step size. In practice, 100–1000 steps with a step size around 0.01 works for MNIST-scale problems.

Hamiltonian Monte Carlo

Hamiltonian Monte Carlo (HMC) introduces auxiliary "momentum" variables to make MCMC moves more efficient. Instead of a random walk, HMC simulates a physical system where a particle rolls around the energy landscape with momentum. This produces proposals that are far from the current point but still in high-probability regions.

def hmc_sample(model, dim=784, n_steps=50, leapfrog_steps=10,
               step_size=0.01, n_samples=64, device='cpu'):
    """Hamiltonian Monte Carlo sampling."""
    x = torch.randn(n_samples, dim, device=device)

    for _ in range(n_steps):
        # Sample momentum
        p = torch.randn_like(x)
        x_new, p_new = x.clone(), p.clone()

        # Leapfrog integration
        for lf in range(leapfrog_steps):
            x_new.requires_grad_(True)
            energy = model(x_new).sum()
            grad = torch.autograd.grad(energy, x_new)[0]

            p_new = p_new - step_size / 2 * grad   # half-step momentum
            x_new = (x_new.detach() + step_size * p_new).detach()

            x_new.requires_grad_(True)
            energy = model(x_new).sum()
            grad = torch.autograd.grad(energy, x_new)[0]
            p_new = p_new - step_size / 2 * grad   # half-step momentum
            x_new = x_new.detach()

        # Metropolis-Hastings acceptance
        E_old = model(x).squeeze() + 0.5 * (p ** 2).sum(-1)
        E_new = model(x_new).squeeze() + 0.5 * (p_new ** 2).sum(-1)
        accept_prob = torch.exp(E_old - E_new).clamp(max=1.0)
        accept_mask = (torch.rand(n_samples, device=device) < accept_prob)
        x = torch.where(accept_mask.unsqueeze(-1), x_new, x)

    return x

HMC's leapfrog integrator preserves volume in phase space, which means proposals are accepted at a high rate — often 60–90%. Compare this to naive random-walk MCMC where acceptance rates in high dimensions can drop below 1%. The cost is more gradient evaluations per step, but the payoff in mixing speed is enormous.

Persistent Contrastive Divergence

Persistent Contrastive Divergence (PCD) keeps the MCMC chains alive between training steps instead of restarting from data every time. You maintain a "replay buffer" of negative samples that get updated by a few MCMC steps each iteration. Over many training steps, these chains mix and explore the model's distribution more thoroughly than CD-k.

class PersistentCDTrainer:
    def __init__(self, model, buffer_size=10000, dim=784):
        self.model = model
        # Persistent replay buffer — initialized once
        self.buffer = torch.randn(buffer_size, dim) * 0.01
        self.buffer_size = buffer_size

    def get_negative_samples(self, batch_size, n_steps=20, step_size=0.01):
        """Get negative samples from persistent buffer."""
        # Random indices from buffer
        idx = torch.randint(0, self.buffer_size, (batch_size,))
        x_neg = self.buffer[idx].clone()

        # Run a few Langevin steps to update
        for _ in range(n_steps):
            x_neg.requires_grad_(True)
            energy = self.model(x_neg).sum()
            grad = torch.autograd.grad(energy, x_neg)[0]
            x_neg = x_neg.detach() - step_size * grad + \
                    (2 * step_size) ** 0.5 * torch.randn_like(x_neg)

        # Write updated samples back to buffer
        self.buffer[idx] = x_neg.detach()
        return x_neg.detach()
💡 Buffer Reinitialization

A common trick: with 5% probability, reinitialize buffer entries from random noise instead of continuing the chain. This prevents the buffer from getting stuck in local modes and ensures diverse negative samples. Du and Mordatch (2019) showed this trick is essential for stable EBM training on image data.

Applications

Restricted Boltzmann Machines

The Restricted Boltzmann Machine (RBM) is the classic EBM architecture. It has visible units (your data) and hidden units (latent features), with connections only between layers — no connections within a layer. This "restricted" connectivity makes inference tractable: given the visible units, all hidden units are conditionally independent (and vice versa).

class RBM(nn.Module):
    """Restricted Boltzmann Machine with binary units."""
    def __init__(self, n_visible=784, n_hidden=256):
        super().__init__()
        self.W = nn.Parameter(torch.randn(n_visible, n_hidden) * 0.01)
        self.v_bias = nn.Parameter(torch.zeros(n_visible))
        self.h_bias = nn.Parameter(torch.zeros(n_hidden))

    def energy(self, v, h):
        """E(v,h) = -v^T W h - b_v^T v - b_h^T h"""
        return -(v @ self.W * h).sum(-1) - \
               (v * self.v_bias).sum(-1) - \
               (h * self.h_bias).sum(-1)

    def sample_h_given_v(self, v):
        """P(h_j=1|v) = sigmoid(W^T v + b_h)"""
        prob_h = torch.sigmoid(v @ self.W + self.h_bias)
        return prob_h, torch.bernoulli(prob_h)

    def sample_v_given_h(self, h):
        """P(v_i=1|h) = sigmoid(W h + b_v)"""
        prob_v = torch.sigmoid(h @ self.W.t() + self.v_bias)
        return prob_v, torch.bernoulli(prob_v)

    def cd_k(self, v, k=1):
        """Contrastive Divergence training step."""
        v_pos = v
        prob_h_pos, h_pos = self.sample_h_given_v(v_pos)

        h_neg = h_pos
        for _ in range(k):
            prob_v_neg, v_neg = self.sample_v_given_h(h_neg)
            prob_h_neg, h_neg = self.sample_h_given_v(v_neg)

        # CD gradient approximation
        pos_associations = v_pos.t() @ prob_h_pos
        neg_associations = v_neg.t() @ prob_h_neg

        loss = -(pos_associations - neg_associations).mean()
        return loss, v_neg

RBMs were the building block of the "deep learning revolution." Hinton's 2006 paper showed that greedily stacking RBMs could pretrain deep networks, which was the first time deep architectures trained successfully. While superseded by modern techniques, understanding RBMs gives you the conceptual foundation for all energy-based thinking.

Deep Boltzmann Machines

A Deep Boltzmann Machine (DBM) stacks multiple layers of hidden units, with all connections being undirected. Unlike a Deep Belief Network (which mixes directed and undirected layers), a DBM is fully undirected. This means approximate inference must propagate both bottom-up and top-down, typically using mean-field variational inference.

The advantage: DBMs learn richer internal representations because information flows in both directions during inference. The disadvantage: training is significantly harder than RBMs. You need variational inference for the positive phase and MCMC for the negative phase. Modern practitioners rarely use DBMs directly, but the ideas about bidirectional inference influenced variational autoencoders and other latent variable models.

EBMs for Compositional Generation

One of the most exciting modern uses of EBMs is compositional generation. Because energies are numbers, you can combine them with arithmetic. Want an image that's both "red" AND "a car"? Add the energies: E_combined(x) = E_red(x) + E_car(x). Want "red" OR "blue"? Use LogSumExp. Want "a dog" but NOT "in a park"? Subtract: E(x) = E_dog(x) - E_park(x).

class CompositionalEBM:
    """Combine multiple energy functions via arithmetic."""
    def __init__(self, energy_fns):
        self.energy_fns = energy_fns  # list of EBM models

    def and_compose(self, x):
        """AND: sum energies (product of distributions)."""
        return sum(fn(x) for fn in self.energy_fns)

    def or_compose(self, x, temperature=1.0):
        """OR: LogSumExp of negative energies."""
        neg_energies = torch.stack([-fn(x) for fn in self.energy_fns])
        return -temperature * torch.logsumexp(
            neg_energies / temperature, dim=0)

    def negate(self, x, fn_idx=0):
        """NOT: negate one energy function."""
        return -self.energy_fns[fn_idx](x)

No other generative model family supports this kind of compositionality as naturally. With diffusion models, you need classifier guidance or other hacks. With GANs, you can't combine generators at all. EBMs make Boolean logic over concepts a first-class operation.

Joint Energy Models

The Joint Energy Model (JEM) by Grathwohl et al. (2020) showed that any classifier already defines an EBM. Take a standard softmax classifier with logits f_θ(x). The energy is E(x) = -LogSumExp(f_θ(x)). This means every classifier you've ever trained is secretly an energy-based model — it was not trained as one.

class JointEnergyModel(nn.Module):
    """Classifier + generative model in a single network."""
    def __init__(self, input_dim=784, hidden_dim=256, n_classes=10):
        super().__init__()
        self.backbone = nn.Sequential(
            nn.Linear(input_dim, hidden_dim),
            nn.LeakyReLU(0.2),
            nn.Linear(hidden_dim, hidden_dim),
            nn.LeakyReLU(0.2),
            nn.Linear(hidden_dim, n_classes)  # class logits
        )

    def classify(self, x):
        """Standard classification."""
        return self.backbone(x)

    def energy(self, x):
        """E(x) = -LogSumExp(logits) — treats classifier as EBM."""
        logits = self.backbone(x)
        return -torch.logsumexp(logits, dim=-1)

    def training_step(self, x_labeled, y, x_unlabeled):
        """Combined discriminative + generative training."""
        # Discriminative loss: standard cross-entropy
        logits = self.classify(x_labeled)
        cls_loss = nn.functional.cross_entropy(logits, y)

        # Generative loss: push energy down on real data
        E_real = self.energy(x_labeled).mean()
        E_unlabeled = self.energy(x_unlabeled).mean()

        # Total: classification + energy shaping
        return cls_loss + 0.1 * (E_real + E_unlabeled)
⚠️ Training Stability

JEM training is notoriously fragile. The generative loss and discriminative loss can fight each other, causing training to diverge. Practical tips: use spectral normalization on all layers, warm up with classification-only training for the first few epochs, and keep the MCMC step size small (0.001 or lower). Even with these tricks, expect to tune hyperparameters carefully.

The promise of JEM is tantalizing: a single model that classifies, detects out-of-distribution inputs (high energy = OOD), generates data, and provides calibrated uncertainty — all from one set of weights. The practice is still catching up to the theory, but the framework connects classification and generation in a way that no other approach does as elegantly.
