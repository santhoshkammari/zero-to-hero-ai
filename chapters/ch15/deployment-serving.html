Avoided deployment too long. Could train models, tune hyperparameters, hit strong hold-out metrics. But when someone asked "so how do we get this in front of users?" I mumbled something about Flask and changed subject. Models lived in notebooks, died in notebooks. Watching other people ship same models faster and cheaper hurt. This chapter is that dive.

Deployment takes trained model out of development environment and puts it somewhere that serves predictions to real users, systems, or devices. It covers portable format conversion, server wrapping, phone-size compression, and safe replacement with better version without 3 AM wake-up. Field matured fast in last few years: dedicated serving frameworks, hardware-aware compilers, and container orchestration systems for ML workloads.

The Restaurant That Explains Deployment

Imagine small restaurant. One chef (model), kitchen (server), menu with three dishes: pasta, salad, soup. Customers walk in, order dish, chef prepares it. Basic model serving: request comes in, prediction goes out.

We return to this restaurant throughout chapter. Every deployment concept maps to restaurant problem: prep food ahead or cook to order? handle dinner rush? replace chef without closing restaurant? open food truck serving same menu in parking lot?

Starting state: three dishes, one chef, no customers. First problem: write chef's recipes in format any kitchen can use.

Getting the Model Out of Python — Serialization

Problem: trained model lives inside Python process. It has weights, computation graph, framework runtime (PyTorch, TensorFlow). But production server might not be Python. Phone definitely isn't. Browser isn't. To serve model anywhere, capture architecture, weights, computation steps in portable file. This process is called serialization. Chosen format decides where model runs, how fast, and which serving tools fit.

Chef knows how to make pasta. Second restaurant in another city cannot ship chef. Need recipe written precisely enough that different chef — one who has never seen your kitchen — reproduces dish exactly. Recipe-book format matters. Some formats work only in Italian kitchens. Others work everywhere but lose nuance.

ONNX — The Universal Recipe Book

ONNX stands for Open Neural Network Exchange. Framework-agnostic format: export from PyTorch or TensorFlow, then run model with ONNX Runtime, TensorRT, OpenVINO, or nearly any production serving framework. ONNX represents model as graph of operators — mathematical operations like matrix multiply, convolution, or ReLU.

Next step: export restaurant's simple dish-classifier model to ONNX. Model takes food image and classifies it as pasta, salad, or soup.

import torch

model = load_trained_dish_classifier()  # Our 3-class model
model.eval()

# ONNX export needs a sample input to trace the computation graph
sample_image = torch.randn(1, 3, 224, 224)

torch.onnx.export(
    model,
    sample_image,
    "dish_classifier.onnx",
    opset_version=17,
    input_names=["food_image"],
    output_names=["dish_scores"],
    dynamic_axes={"food_image": {0: "batch_size"}}
)

Key pieces here. opset_version is ONNX operator spec version — recipe-book edition. Version 17 has broad 2024 support. Older opset limits exportable operations. Too-new opset may fail in target runtime. dynamic_axes tells ONNX batch dimension can change. Without it, model is locked to one image at a time — restaurant that serves one customer regardless of demand.

Now serve model with ONNX Runtime. It applies graph optimizations — fusing operations, eliminating redundant computations, planning memory layouts — to get 2-3× speed of raw PyTorch.

import onnxruntime as ort
import numpy as np

session = ort.InferenceSession(
    "dish_classifier.onnx",
    providers=["CUDAExecutionProvider", "CPUExecutionProvider"]
)

food_photo = preprocess_image("customer_photo.jpg")
scores = session.run(None, {"food_image": food_photo})
# scores[0] → array([[0.85, 0.10, 0.05]])  → probably pasta

ONNX export does not always go smoothly. Some custom PyTorch operations have no ONNX equivalent. Dynamic control flow (if-statements that depend on tensor values) can break tracer. When export fails, error messages are cryptic. Run onnx.checker.check_model, then onnx-simplifier on every export. Good sanity check.

TorchScript — Staying in the PyTorch Family

If whole serving stack is PyTorch, TorchScript is PyTorch's own serialization format. Two flavors. torch.jit.trace records operations from one forward pass with sample input — fast, good for models with no input-dependent if-statements or loops. torch.jit.script parses Python code into statically typed representation — it handles control flow, but not every Python construct survives translation.

Newer PyTorch tools: torch.export and torch.compile. They are gradually replacing TorchScript for many use cases. torch.compile stands out because one extra line often gives 1.5-3× speedup, with no format conversion.

TensorFlow SavedModel — The TF Ecosystem's Format

If model is built in TensorFlow or Keras, SavedModel is format. It bundles computation graph, weights, and signatures into directory structure that TF Serving, TFLite, and TensorFlow.js can consume. Little reason to use it outside TensorFlow ecosystem. Inside it, integration is seamless.

Limit of all these formats: they freeze model at one point in time. Recipe is written down, but cannot adapt. If you change architecture or add output, you need to re-export. Obvious, but teams still debug production for hours before noticing they serve stale export.

When Does the Cooking Happen? — Batch vs. Real-Time vs. Streaming

Back to restaurant. Food can be served in different ways. Same for model predictions. Key question: when does model run relative to when someone needs answer?

Batch — The Buffet

Buffet prepares all food before customers arrive. Customer takes what is already cooked. ML version: run model over all known inputs ahead of time, store predictions in database, look them up when user makes request. No model server at serving time — serving layer is database read.

import redis

def run_nightly_scoring():
    """Score every known user, cache results for tomorrow's traffic."""
    model = load_dish_recommender()
    users = load_all_user_profiles()

    cache = redis.Redis(host="redis-server", port=6379)
    pipe = cache.pipeline()

    for user in users:
        recommendation = model.predict(user.features)
        pipe.set(f"recs:{user.id}", recommendation, ex=86400)

    pipe.execute()  # Write all predictions at once

Beauty of batch: low operational burden. No model server running. No GPU kept warm. No tail latency worries. Nightly cron job, database, done. This stretches far. Precompute recommendations for top 80% of active users; fall back to popular items for rest.

Buffet shows limit too. Customer wants something not prepped? Brand-new user has no precomputed recommendation. Fraud detection cannot wait for tonight's batch — fraudster swipes card now. When freshness matters, buffet fails.

Real-Time — The Made-to-Order Kitchen

Customer orders. Chef cooks. Dish goes out. ML version: request hits endpoint, model runs inference on that specific input, and response comes back inside latency budget, typically under 100 milliseconds. Hardest paradigm to operate. It demands live model server, GPU management, health monitoring, and auto-scaling. But when prediction must reflect most recent context — "is this particular credit card transaction, right now, fraudulent?" — it is only option.

Streaming — The Conveyor Belt Sushi Bar

Conveyor belt sushi bar: kitchen continuously prepares dishes and sends them down belt. Nobody stands at counter waiting for one specific plate. Food flows continuously. Diners grab what matters.

Streaming inference works same way. Events flow in continuously — from Kafka topics, IoT sensors, clickstreams — and model processes each one on arrival, emitting predictions downstream. No client waits for response. System is event-driven.

from kafka import KafkaConsumer, KafkaProducer
import json

consumer = KafkaConsumer(
    "order-events",
    bootstrap_servers=["kafka:9092"],
    value_deserializer=lambda m: json.loads(m)
)
producer = KafkaProducer(
    bootstrap_servers=["kafka:9092"],
    value_serializer=lambda v: json.dumps(v).encode()
)
model = load_fraud_model()

for message in consumer:
    event = message.value
    features = extract_features(event)
    score = model.predict(features)
    producer.send("fraud-scores", value={
        "order_id": event["id"],
        "score": float(score)
    })

Critical distinction: real-time is request-response. Customer waits at counter. Streaming is fire-and-forget. Sushi goes on belt whether anyone grabs it or not. Streaming fits continuous monitoring, anomaly detection, and anything that maintains running state over time windows.

One more paradigm: serverless inference (AWS Lambda, Google Cloud Functions). Upload model and code, cloud handles scaling. Pay per invocation. Good for bursty, low-traffic workloads. Terrible for latency-sensitive work because idle function causes cold start — model loads from scratch — which can add 2 to 30 seconds of delay. Eternity in most serving contexts.

The Kitchen Equipment — Serving Frameworks

Need real-time kitchen. First instinct: wrap model in familiar web framework. FastAPI, Flask, something known.

from fastapi import FastAPI
from pydantic import BaseModel
import numpy as np

app = FastAPI()
model = load_dish_classifier()

class DishRequest(BaseModel):
    image_url: str

@app.post("/predict")
async def predict(req: DishRequest):
    image = download_and_preprocess(req.image_url)
    scores = model.predict(image)
    dish = ["pasta", "salad", "soup"][np.argmax(scores)]
    return {"dish": dish, "confidence": float(np.max(scores))}

@app.get("/health")
async def health():
    return {"status": "ok"}

For demo, internal tool, or model getting ten requests per minute, this works. As restaurant grows, problems appear that web framework was never built to solve.

First problem: GPU utilization. GPU is massive griddle — it can cook 32 burgers at once, but one burger uses almost same energy and time. When requests arrive one at a time and we process them individually, we heat whole griddle for one patty. Need dynamic batching: collect requests arriving within short window (say 5 milliseconds), bundle them, and run inference on whole batch. Throughput goes up. Per-request cost goes down.

Second problem: multi-model management. Restaurant now serves breakfast, lunch, and dinner — three menus, three models. Need to load and unload them, version them, route traffic to right one, and monitor each independently. Web framework gives none of this.

Third problem: GPU memory management. Models are big. One transformer can eat 4 GB of GPU memory. Two models on one GPU need careful coordination. Loading new model version while old one still serves needs even more care.

Reason dedicated model-serving frameworks exist. They handle batching, versioning, GPU scheduling, health checks, and protocol support (REST and gRPC) out of box. Next: main ones.

Triton Inference Server — The Industrial Kitchen

NVIDIA's Triton is Swiss Army knife. It serves models from any framework — PyTorch, TensorFlow, ONNX, TensorRT, even plain Python — on same GPU, simultaneously. Killer feature is concurrent model execution. If NLP model uses 40% of GPU, Triton schedules vision model on remaining 60%. In multi-model deployments, this can cut GPU bill in half.

Triton configuration is verbose. Each model lives in directory with config.pbtxt file specifying input/output shapes, batching parameters, and scheduling policy. First setup is not afternoon task. Once configured, it is most capable serving infrastructure I've worked with.

TorchServe — The PyTorch Kitchen

If stack is all PyTorch, TorchServe provides deep integration. You write handler class that defines three steps — preprocess, inference, postprocess — and package everything into .mar archive.

from ts.torch_handler.base_handler import BaseHandler
import torch

class DishClassifierHandler(BaseHandler):
    def preprocess(self, requests):
        images = []
        for req in requests:
            img = decode_image(req["body"])
            images.append(self.transform(img))
        return torch.stack(images)

    def inference(self, batch):
        with torch.no_grad():
            return self.model(batch)

    def postprocess(self, outputs):
        probs = torch.softmax(outputs, dim=1)
        dishes = ["pasta", "salad", "soup"]
        return [
            {dishes[i]: float(p[i]) for i in range(3)}
            for p in probs
        ]

TorchServe handles batching, model versioning, and health monitoring. Tradeoff versus Triton: it only serves PyTorch models, and it does not do concurrent multi-model GPU sharing as effectively.

TF Serving, BentoML, and vLLM

TF Serving is TensorFlow's production server. Mature, battle-tested, tight integration with TF ecosystem. If models are TensorFlow, it is reliable. It only serves TensorFlow models.

BentoML takes different approach: framework-agnostic. You define serving logic in Python class, and BentoML builds production container from it — complete with batching, API routes, and dependency management. Appeal: you write Python, it handles infrastructure plumbing.

vLLM is purpose-built for large language models. Core innovation is PagedAttention, which manages GPU memory used by key-value cache during text generation like operating system virtual memory — in pages that can be allocated, freed, and shared. Combined with continuous batching (starting new requests as soon as slot opens, rather than waiting for whole batch to finish), vLLM achieves 2-4× throughput of naive LLM serving. If serving LLMs, reach for this.

A Quick Note on Protocols

Most serving frameworks expose two interfaces: REST (JSON over HTTP) and gRPC (Protocol Buffers over HTTP/2). REST is simple and debuggable — you can test it with curl. gRPC serializes data in compact binary format and is 2-5× faster for high-throughput internal services. Many production systems use both: gRPC between microservices, REST for external APIs, gateway translating between them.

Shrinking the Recipe — Optimization for Production

Restaurant is running, but head chef (model) is expensive. Transformer at 200ms per prediction on $2/hour GPU, serving 100 requests per second, costs about $50,000 per month. VP of engineering wants $5,000. Next step: make recipe faster, cheaper, smaller without ruining dish.

Every optimization technique trades some accuracy for speed or size. Goal: hit latency and cost targets while staying inside acceptable accuracy budget. Next: three major techniques.

Quantization — Using Less Precise Measurements

Chef measures ingredients with scale showing four decimal places. Does 2.3847 grams of salt versus 2.4 grams matter? Dish tastes same. Quantization does same for model weights: reduce numerical precision from 32-bit floating point (FP32, 4 bytes per weight) to 16-bit (FP16, 2 bytes), 8-bit integers (INT8, 1 byte), or even 4-bit (INT4, half a byte).

Why this works: weight of 0.23847291 and 0.23828125 produce same prediction. Network learned patterns and relationships, not decimal places.

Two approaches. Post-Training Quantization (PTQ) takes trained model, converts weights to lower precision, done. It needs small calibration dataset — a few hundred samples — to determine best mapping from floating point to integer ranges. No retraining.

import torch
from torch.quantization import quantize_dynamic

model_fp32 = load_dish_classifier()

# Dynamic quantization: convert Linear layers to INT8
model_int8 = quantize_dynamic(
    model_fp32,
    {torch.nn.Linear},
    dtype=torch.qint8
)

Result: model is roughly 4× smaller. On CPU, inference is 2-4× faster. On GPU, FP16 is effectively free (every modern GPU has FP16 hardware), and INT8 support is broad across both CPUs and recent GPUs.

Other approach is Quantization-Aware Training (QAT). Instead of quantizing after training, QAT simulates quantization during training by inserting fake quantization nodes into forward pass. Model learns to tolerate precision loss. More work, but it recovers 50-100% of accuracy PTQ drops. For models where last fraction of percent matters — medical imaging, autonomous driving — QAT is worth effort.

PTQ versus QAT intuition: for non-LLM models, INT8 PTQ is almost free accuracy-wise and worthwhile first step on any deployment. For large language models, specialized quantization methods like GPTQ (which uses Hessian information to protect most important weights) and AWQ (which protects 1% of weights that handle salient activations) enable 4-bit quantization with minimal quality loss, cutting serving costs 4-8×.

Pruning — Removing Unnecessary Ingredients

Research on lottery ticket hypothesis revealed something surprising: large neural networks are massively over-parameterized. Inside big network, there is much smaller network that would have performed nearly as well if trained from scratch. Pruning finds and removes weights that do not contribute meaningfully to output.

Two flavors, and distinction matters. Unstructured pruning zeroes out individual weights wherever they appear. You might prune 90% of weights this way. Problem: resulting weight matrices are sparse (mostly zeros), and standard GPU hardware does not know how to exploit that sparsity. Matrix stays same shape; zeros still get multiplied. You save storage, not compute time — unless you have specialized sparse-compute hardware.

Structured pruning removes entire neurons, channels, or attention heads. Compression ratio is lower — maybe 30-50% rather than 90% — but result is regular, dense model that runs faster on any hardware. No special sparse libraries needed. Usually what you want for deployment.

import torch.nn.utils.prune as prune

# Remove 30% of channels from every Conv2d layer
for name, module in model.named_modules():
    if isinstance(module, torch.nn.Conv2d):
        prune.ln_structured(
            module, name="weight",
            amount=0.3, n=2, dim=0
        )

# Make the pruning permanent (remove the mask, bake zeros into weights)
for name, module in model.named_modules():
    if hasattr(module, "weight_orig"):
        prune.remove(module, "weight")

Most effective recipe is iterative: prune 10-20%, retrain for a few epochs to let model recover, prune again. Each cycle, model adapts. This reaches much higher compression than one-shot pruning.

Common trap: getting excited about "90% sparse!" numbers without benchmarking actual inference time. NVIDIA's Ampere and Hopper GPUs support 2:4 structured sparsity (every group of 4 weights has at most 2 non-zeros) with dedicated hardware. Other sparsity patterns need software support that may or may not exist for your setup. Always measure wall-clock time, not parameter count.

Distillation — Teaching a Smaller Chef

Different approach: instead of shrinking existing chef, hire new cheaper chef and train them by watching expensive one cook.

Knowledge distillation trains small "student" model to mimic large "teacher" model. Key insight is what student learns from. Normally, model trains on hard labels: this image is pasta (1), not salad (0), not soup (0). But teacher's output is richer. Teacher might say: this is probably pasta (0.85), but it has some salad-like qualities (0.12), and a tiny bit of soup resemblance (0.03). Those soft probabilities carry inter-class relationships that hard labels completely miss.

import torch.nn.functional as F

def distillation_loss(student_logits, teacher_logits, true_labels,
                      temperature=4.0, alpha=0.7):
    # Soften both distributions with temperature
    soft_teacher = F.softmax(teacher_logits / temperature, dim=-1)
    soft_student = F.log_softmax(student_logits / temperature, dim=-1)

    # KL divergence between soft distributions
    distill_loss = F.kl_div(soft_student, soft_teacher,
                            reduction="batchmean")
    distill_loss *= temperature ** 2  # Scale gradient magnitude

    # Standard cross-entropy with hard labels
    hard_loss = F.cross_entropy(student_logits, true_labels)

    # Blend: mostly learn from teacher, some from ground truth
    return alpha * distill_loss + (1 - alpha) * hard_loss

temperature parameter controls how soft probabilities become. At temperature 1, they are teacher's raw output. At temperature 4, differences between classes are amplified, so soft probabilities become more informative. alpha parameter balances learning from teacher versus ground truth labels.

Results are remarkable. DistilBERT is 40% smaller and 60% faster than BERT while retaining 97% of its language understanding. TinyBERT achieves 7.5× compression through more aggressive two-stage distillation that also matches teacher's intermediate representations. These are not research curiosities — they run in production at companies processing millions of requests per day.

Stacking It All Together — Compilation

One more lever: graph compilation. Tools like TensorRT, ONNX Runtime, and torch.compile analyze model's computation graph and fuse operations. Sequence of MatMul → Add → ReLU becomes single fused kernel that reads memory once and writes once, instead of three separate operations that each read and write independently. Since memory bandwidth is typically inference bottleneck, this matters enormously.

Real power comes from stacking techniques. Distill model → quantize to INT8 → compile with TensorRT. Model that started at 2 GB and 200ms per inference can end up at 50 MB and 5ms. That is 40× improvement in both size and speed. Each technique is roughly multiplicative with others, which is why production ML teams routinely apply all three.

Packing the Kitchen Into a Box — Containerization

Restaurant metaphor gets literal here. Imagine shipping entire kitchen — stove, ingredients, recipe book, specific brand of olive oil chef insists on — inside standardized shipping container. Drop it anywhere in world, open doors, kitchen works exactly as it did at home.

That is Docker for ML. Problem it solves is dependency nightmare. Model needs specific version of PyTorch (say 2.2.1), specific CUDA version (12.1), cuDNN (8.9), Python 3.11, and many other packages in exact versions. If any one does not match between development machine and production, things break in mysterious ways. "It works on my machine" is not deployment strategy.

FROM python:3.11-slim AS base

RUN apt-get update && apt-get install -y --no-install-recommends \
    libgomp1 && rm -rf /var/lib/apt/lists/*

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY models/ /app/models/
COPY src/ /app/src/
WORKDIR /app
EXPOSE 8080

HEALTHCHECK --interval=30s --timeout=10s --retries=3 \
    CMD curl -f http://localhost:8080/health || exit 1

CMD ["uvicorn", "src.server:app", "--host", "0.0.0.0", "--port", "8080"]

Ordering in this Dockerfile matters. requirements.txt is copied and installed before application code. Docker caches each layer. If you change Python code but not dependencies, package-install layer is cached and skipped on rebuild. This turns 10-minute rebuild into 30-second one. Putting application code first invalidates cache every time and reinstalls everything from scratch.

For GPU models, start from nvidia/cuda base image instead of python:3.11-slim. These images are big — often 8 GB or more — because they include CUDA libraries, cuDNN, and GPU drivers. Pin exact versions everywhere. torch==2.2.1, not torch>=2.0. When Tuesday morning rebuild grabs new minor version that is subtly incompatible with your CUDA driver, you will be grateful for pin.

Once kitchen is in container, Kubernetes decides how many copies of that container to run, restarts ones that crash, and routes traffic to healthy ones. We come back to this in scaling section.

Taking the Kitchen to the Customer — Edge Deployment

Sometimes you cannot send customer's order to central kitchen. Customer may be in car with spotty cell service. Or customer is security camera that must detect intruders in 5 milliseconds, and 200-millisecond cloud round trip means intruder is already inside. Or customer is hospital, and sending patient images to external server violates privacy regulations.

Edge deployment means running model on device itself — phone, camera, car, industrial sensor. Three forces drive it: latency (5ms on-device versus 50-200ms round-trip to cloud), privacy (data never leaves device), and offline capability (cars and drones cannot depend on WiFi).

Challenge: edge devices are constrained. Phone's neural processing unit has fraction of cloud GPU's power. RAM is measured in gigabytes, not hundreds of gigabytes. Battery matters. So models need to be small and fast, which is where optimization techniques we covered — quantization, pruning, distillation — become essential rather than optional.

Each platform has preferred format. TFLite (TensorFlow Lite) is go-to for Android, with deep integration into NNAPI hardware acceleration layer. Core ML dominates iOS, where Apple's Neural Engine provides remarkably low latency for models that fit its constraints. ONNX Runtime Mobile is cross-platform option — it can use NNAPI on Android and Core ML on iOS as backend delegates.

import tensorflow as tf

# Convert our dish classifier to TFLite with INT8 quantization
converter = tf.lite.TFLiteConverter.from_saved_model("dish_classifier/")
converter.optimizations = [tf.lite.Optimize.DEFAULT]

# Provide representative data so the converter can calibrate quantization
def representative_dataset():
    for image in calibration_images[:100]:
        yield [image.numpy()]

converter.representative_dataset = representative_dataset
converter.target_spec.supported_ops = [
    tf.lite.OpsSet.TFLITE_BUILTINS_INT8
]
tflite_model = converter.convert()

with open("dish_classifier.tflite", "wb") as f:
    f.write(tflite_model)

In practice, most edge deployments are hybrid. Small, fast model on device handles common cases — keyboard's next-word prediction, camera's face detection. Hard or unusual cases get routed to more powerful cloud model. Device model acts as first pass: fast for 95% of inputs it can handle confidently, with fallback for rest.

On-device debugging feels very different from cloud debugging. In cloud, you can SSH in, attach debugger, check GPU utilization. On phone, model is black box inside app. Logging is limited. Profiling is awkward. Test on real devices early and often — simulator lies about performance characteristics.

Opening More Locations — Scaling and Auto-Scaling

Restaurant is popular. At noon, line runs out door. At 2 AM, dining room is empty. Need to scale up during rush hour and scale back down to avoid paying staff to stand around.

Vertical scaling means hiring faster chef (bigger GPU). Simplest approach, but ceiling exists — GPUs only get so big. Horizontal scaling means opening more kitchens — running more replicas of model server behind load balancer. This is how production systems scale, and Kubernetes makes it possible with Horizontal Pod Autoscaler (HPA).

Standard HPA scales on CPU or memory utilization. For ML serving, that is usually wrong metric. What you care about is request latency or queue depth. If latency creeps above target — say, p95 latency exceeds 100ms — that is signal to add replicas. Kubernetes supports custom metrics for this, often piped in from Prometheus.

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: dish-classifier-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: dish-classifier
  minReplicas: 2
  maxReplicas: 20
  metrics:
  - type: Pods
    pods:
      metric:
        name: inference_latency_p95_ms
      target:
        type: AverageValue
        averageValue: "80"

GPU workloads add one more nuance: GPU sharing. GPUs are expensive, and single model often does not saturate one. Triton's concurrent model execution helps here — running multiple models on one GPU, as discussed. Another approach is MIG (Multi-Instance GPU) on NVIDIA A100/H100 hardware, which physically partitions single GPU into isolated slices, each with its own memory and compute. One physical GPU becomes, say, seven independent mini-GPUs.

Load Balancing for ML

Standard round-robin load balancing does not work well for ML. Not all requests are equally expensive. Classifying 10-word text takes 10ms; processing 10-page document takes 200ms. Round-robin might send three expensive requests to same server while others sit idle.

Least-connections routing is better: send each new request to server with fewest active requests. It naturally adapts to varying request cost. For multi-model GPU deployments, you may also need model-aware routing — sending requests to server that actually has right model loaded in GPU memory, because loading model on-the-fly adds seconds of latency.

Speed of Service — Latency Optimization

Customer who waits too long leaves. In production serving, every millisecond costs either users (higher latency → fewer conversions) or money (bigger hardware to meet latency targets). Next: main levers, roughly order I'd try them.

Dynamic Batching

We touched this already, but it deserves closer look. GPU processing 1 request takes, say, 15ms. Processing 32 requests takes 18ms. Not typo — GPU's massive parallelism makes batch processing nearly free. Problem: requests do not arrive in neat batches. They trickle in one at a time.

Dynamic batcher solves this by holding incoming requests in queue for short window — 5ms is typical — and bundling whatever has accumulated into single batch. Tension is simple: wait longer, get bigger batches and better throughput, but each individual request waits longer. Wait less, get smaller batches and lower throughput, but faster per-request response. max_wait_ms parameter is where you dial this tradeoff.

import asyncio
from collections import deque

class DynamicBatcher:
    def __init__(self, model, max_batch=32, max_wait_ms=5):
        self.model = model
        self.max_batch = max_batch
        self.max_wait_ms = max_wait_ms
        self.queue = deque()

    async def predict(self, input_data):
        """Each caller awaits their individual result."""
        future = asyncio.Future()
        self.queue.append((input_data, future))

        if len(self.queue) >= self.max_batch:
            self._flush()
        else:
            # Schedule a flush after max_wait_ms
            asyncio.get_event_loop().call_later(
                self.max_wait_ms / 1000, self._flush
            )
        return await future

    def _flush(self):
        if not self.queue:
            return
        inputs, futures = [], []
        while self.queue and len(inputs) < self.max_batch:
            inp, fut = self.queue.popleft()
            inputs.append(inp)
            futures.append(fut)
        results = self.model.batch_predict(inputs)
        for fut, result in zip(futures, results):
            fut.set_result(result)

Warm-Up and Caching

Two quick wins that are easy to overlook. Model warm-up: first inference after loading model is always slow because GPU needs to allocate memory, compile kernels, and populate caches. Run few dummy predictions at startup. Small defensive step. It prevents first real user from seeing 10× normal latency.

Response caching: if same input appears frequently, cache prediction. Redis layer in front of model server can absorb 30-60% of traffic for many workloads. Hash input features, check cache, call model only on cache misses. For dish classifier, if people keep photographing same restaurant's menu items, caching pays quickly.

Circuit Breakers — When the Kitchen Catches Fire

In real restaurant, if stove malfunctions, you stop sending orders to that station. Circuit breaker does same for failing model server. After N consecutive failures (say, 5), circuit "opens" — no more traffic goes to that server. After cooling-off period (say, 60 seconds), it sends one test request. If it succeeds, circuit "closes" and normal traffic resumes. If it fails again, circuit stays open.

This prevents single sick server from cascading failures across whole system. Without circuit breaker, load balancer keeps sending requests to failing server, those requests time out, timeouts pile up, back-pressure builds, and suddenly all servers are slow.

In production, service meshes like Istio and Linkerd provide circuit breakers, retries, timeouts, and load balancing out of box. Building your own is good learning exercise; relying on it in production invites heartache.

Changing the Menu Without Closing the Restaurant

Chef developed better pasta recipe. How do we start serving it without risking dinner-rush disaster if new recipe is terrible? Core challenge of deployment strategies: get new model in front of users without breaking things.

Canary Deployment — One Table Gets the New Dish

Name one table in restaurant "canary table." Serve them new pasta. Everyone else gets old pasta. Watch closely. If canary table loves it, gradually expand: two tables, then half restaurant, then everyone. If they get food poisoning, pull it back immediately.

In ML terms: route 5% of production traffic to new model, 95% to old one. Monitor for errors, latency spikes, and metric degradation. If everything looks clean, ramp: 10%, 25%, 50%, 100%. If something goes wrong, instantly route everything back. Blast radius stays contained to 5% of users.

# Istio VirtualService: 5% canary traffic split
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: dish-classifier
spec:
  hosts:
    - dish-classifier
  http:
    - route:
        - destination:
            host: dish-classifier
            subset: stable
          weight: 95
        - destination:
            host: dish-classifier
            subset: canary
          weight: 5

Blue-Green — Two Complete Kitchens

Run two identical environments. Blue serves current model. Green has new model loaded and ready. When confident, flip switch — all traffic moves from blue to green at once. If green fails, flip back. Advantage over canary: switch is instant in both directions. Disadvantage: all-or-nothing. No gradual ramp-up, so you go from 0% to 100% in one step.

Shadow (Dark Launch) — The Secret Kitchen

Route production traffic to both old and new models. Serve old model's response to users. Log new model's response for comparison. Zero user impact — you test new model with real traffic without exposing anyone to its predictions. Tradeoff: you pay double compute cost for duration of shadow test.

Shadow deployments are particularly valuable for high-stakes models — fraud detection, medical diagnosis — where you want extensive evidence that new model behaves correctly before any user sees its output.

A/B Testing — Measuring if the New Dish is Actually Better

Important distinction teams still get wrong. Canary deployment tests whether new model is safe — it does not crash, latency is reasonable, no error spikes. A/B testing measures whether new model is better — it improves business metric like conversion rate, click-through rate, or revenue.

For A/B testing, split users (not requests) into control and treatment groups. Same user must always see same version, or experiment gets contaminated.

import hashlib

def assign_group(user_id: str, experiment: str) -> str:
    """Deterministic assignment: same user always gets same group."""
    key = f"{user_id}:{experiment}"
    hash_val = int(hashlib.sha256(key.encode()).hexdigest(), 16)
    return "treatment" if hash_val % 100 < 10 else "control"

A/B tests require statistical rigor. Define hypothesis and minimum detectable effect before experiment starts. Calculate required sample size. Run full planned duration — do not peek at intermediate results and stop early when they look good, because that inflates false positive rate. Teams still declare winner after two days of planned two-week test because numbers "looked really good." Often random fluctuation. Long-run model performance turns out worse.
