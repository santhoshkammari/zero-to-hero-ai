Recommendation model shipped. Returned wrong predictions three straight weeks. No alert fired. No error log. No Slack message. API said 200 OK, JSON well-formed, uptime dashboard green. Meanwhile, model confidently suggested winter coats to July users — upstream feature pipeline silently sending stale data. Three weeks. Deploying a model and monitoring a model are different disciplines.

Monitoring: watching ML system post-deployment to detect unexpected behavior. Observability: broader ability to investigate why it misbehaved, using signals you recorded. Together: statistical drift detection to 2 AM investigation playbook. Roots in DevOps, but ML-specific layers — data drift, silent degradation, delayed ground truth — are challenges classical software monitoring never faced.

The Silent Failure Problem

ShopFlow sells three products: headphones, notebook, coffee mug. Recommendation model picks which product to show first. Training: younger users → headphones, older → notebooks, mugs as fallback. Day-one accuracy: 78%.

Traditional software monitor checks: server up? Response under 200ms? Error rate below 1%? All three stay “yes” even if model recommends coffee mugs to every person on earth.

Fundamental asymmetry: broken model produces valid outputs. No stack trace for “wrong prediction.” No HTTP 500 for “world changed, model didn’t notice.” Model keeps returning valid JSON while business metric — actual purchases — quietly tanks.

Weather station analogy: traditional monitoring checks instruments are powered and transmitting. ML monitoring must also check if thermometer reads 35°C in January. Instruments work perfectly while producing nonsense. Gap between “system running?” and “system right?” — that’s where this section lives.

The Three Faces of Drift

“Drift” sounds like one thing. Actually three distinct phenomena. Confusing them leads to wrong response.

Covariate Shift — The Inputs Moved

Training data came from campaign targeting users aged 25–35. New campaign targets retirees: users now 55–70, region model barely saw. Relationship between age and preference unchanged (older users still prefer notebooks), but model never trained on this population. Extrapolation instead of interpolation.

Formal: input distribution P(X) changed; mapping P(Y|X) unchanged. That’s covariate shift. Model’s learned function still correct — trained on region it’s no longer seeing.

Concrete numbers: training avg user age 30, std 5. Production avg 62, std 8. Two distributions side by side:

Training distribution:     [    ████████████    ]
                          20   25   30   35   40

Production distribution:                    [  ██████████████  ]
                                           50  55   62   70  75

Almost no overlap. The model is flying blind.

Fix: retrain with data covering new input distribution, or collect from underrepresented region. Old training data still useful — underlying relationship unchanged, just expanding coverage.

Concept Drift — The Rules Changed

Same age distribution as training. Cultural shift happened — viral TikTok made mugs “cool” for twenty-somethings. Users aged 25 who always bought headphones now buy mugs. Same inputs, different correct answer.

P(X) untouched. P(Y|X) changed — mapping from features to correct label. That’s concept drift, scarier of the two. Old training data becomes actively misleading — encodes relationship that no longer exists.

Fix: retrain on recent data with fresh labels. Old data harmful, not helpful. May need shorter training window or time-based weighting.

Prior Probability Shift — The Outcomes Moved

Neither demographics nor preferences changed, but BOGO promotion on notebooks pushed return rate from 5% to 25%. Distribution of outcomes P(Y) shifted. Model’s predictions as accurate as before, but business impact of each prediction changed. Model calibrated for 5% returns makes poor decisions in 25% return world.

That’s prior probability shift (also called label drift). Fix: recalibrate thresholds and decision boundaries. May not need to retrain at all — adjusting operating point can be enough.

Why the distinction matters: Teams waste a week retraining for what was prior probability shift — threshold adjustment fixes it in minutes. Others adjust thresholds when problem is concept drift — adjusting thermostat while house is on fire. Different diagnoses demand different treatments.

Our ShopFlow weather station analogy helps here. Covariate shift is like measuring temperature in a city you’ve never calibrated for — your thermometer works, but it’s in an unfamiliar environment. Concept drift is like the laws of physics changing — the temperature reading is correct, but what that temperature means for people has shifted. Prior probability shift is like the overall climate warming — the instrument and the physics are fine, but the baseline expectation needs updating.

Each type is real, each type happens in production, and the statistical tests we’ll build next can help distinguish them. But the tests themselves don’t tell you which kind of drift is happening. They tell you that something moved, and then you need judgment to figure out what.

Measuring Drift from Scratch

Drift is dangerous. Need numbers, not feelings. Three statistical tools, starting with the most intuitive.

Population Stability Index (PSI) — The Bin-and-Compare Approach

PSI formula looks like gibberish first time. Actually doing something straightforward.

1,000 training users, 500 production users. Question: did age distribution change?

Chop training ages into bins. Three bins:

Bin         Training (1000 users)    Production (500 users)
18-25       300  (30%)               50   (10%)
26-35       500  (50%)               200  (40%)
36+         200  (20%)               250  (50%)

Bins defined from training data, applied to production. Proportions per bin: 30% vs 10%, 50% vs 40%, 20% vs 50%.

PSI: for each bin, how much did proportion change, weighted by surprise? Formula for single bin:

PSI_bin = (actual% - expected%) × ln(actual% / expected%)

First bin (ages 18–25). Expected (training): 0.30. Actual (production): 0.10.

PSI_bin1 = (0.10 - 0.30) × ln(0.10 / 0.30)
         = (-0.20) × ln(0.333)
         = (-0.20) × (-1.099)
         = 0.220

For the second bin (26–35): expected 0.50, actual 0.40.

PSI_bin2 = (0.40 - 0.50) × ln(0.40 / 0.50)
         = (-0.10) × (-0.223)
         = 0.022

For the third bin (36+): expected 0.20, actual 0.50.

PSI_bin3 = (0.50 - 0.20) × ln(0.50 / 0.20)
         = (0.30) × (0.916)
         = 0.275

Total PSI = 0.220 + 0.022 + 0.275 = 0.517

Well above 0.25 threshold (from credit risk industry). Distribution drifted significantly.

When actual = expected: difference zero, log ratio zero, PSI contributes nothing. When they diverge: both grow, amplifying signal. ln(actual/expected) borrowed from information theory — PSI is symmetrized version of Kullback-Leibler divergence.

In code:

import numpy as np

def compute_psi(expected, actual, bins=10):
    # Define bin edges from the expected (training) data
    breakpoints = np.percentile(expected, np.linspace(0, 100, bins + 1))
    breakpoints[0], breakpoints[-1] = -np.inf, np.inf

    # Count proportions in each bin (add tiny epsilon to avoid log(0))
    exp_pct = np.histogram(expected, bins=breakpoints)[0] / len(expected) + 1e-6
    act_pct = np.histogram(actual, bins=breakpoints)[0] / len(actual) + 1e-6

    # The PSI formula: sum of (actual - expected) * ln(actual / expected)
    return np.sum((act_pct - exp_pct) * np.log(act_pct / exp_pct))

np.percentile creates bin edges from training data using quantiles — each bin has roughly equal expected counts. 1e-6 epsilon prevents division by zero. Final line: formula vectorized across all bins.

Standard thresholds:

PSI ValueWhat It MeansWhat To Do
< 0.1Distributions are similarNo action needed
0.1 – 0.25Moderate shift detectedInvestigate, consider retraining
> 0.25Significant shiftModel is out of distribution — retrain

PSI: simple to explain, works for continuous and categorical data, single interpretable number. Limitation: bin count matters, no p-value — no statistical confidence, no “significant at 5% level.” For that, need proper statistical test.

The Kolmogorov-Smirnov Test — Maximum Distance Between Distributions

PSI gives score. Kolmogorov-Smirnov test (KS test) gives verdict. Nonparametric — no assumptions about distribution shape.

Geometric idea: take both samples, sort, compute cumulative distribution functions (CDFs). CDF answers: “what fraction of data falls below this value?” Find point where two CDFs farthest apart. That maximum gap is the KS statistic.

Training CDF climbs steeply 25–35 (most training users). Production CDF climbs steeply 36–65. Around age 35: training CDF at 0.80, production CDF at 0.45. Gap of 0.35 is the KS statistic.

from scipy import stats
import numpy as np

# Simulate our ShopFlow age distributions
training_ages = np.random.normal(30, 5, 1000)
production_ages = np.random.normal(62, 8, 500)

ks_statistic, p_value = stats.ks_2samp(training_ages, production_ages)
# ks_statistic ≈ 0.97, p_value ≈ 0.0 (extremely significant)

stats.ks_2samp returns two things: ks_statistic (maximum CDF gap, 0–1) and p_value (“if samples came from same distribution, how likely is gap this large?”). Tiny p-value: drift is real.

KS test: best for continuous features, no shape assumptions. Blind spot: huge sample sizes flag tiny, meaningless shifts as “statistically significant.” Always pair statistical significance with practical significance — check if magnitude large enough to hurt model.

KL Divergence — How Much Information Did You Lose?

Information theory approach. Kullback-Leibler divergence asks: if I built encoding scheme optimized for distribution P, how many extra bits wasted encoding data from distribution Q?

Model learned to be efficient recognizing users from training distribution — carved decision boundaries assuming 50% in 26–35 range. When production shifts those proportions, model’s internal “encoding” is wasteful. KL divergence quantifies that waste.

import numpy as np

def kl_divergence(p, q):
    """KL(P || Q): extra bits needed when using Q to encode P."""
    # Both must be proper probability distributions (sum to 1)
    # Add epsilon to avoid log(0)
    p = np.array(p) + 1e-10
    q = np.array(q) + 1e-10
    return np.sum(p * np.log(p / q))

# ShopFlow age bins: training vs production proportions
training_props = [0.30, 0.50, 0.20]
production_props = [0.10, 0.40, 0.50]

kl_train_to_prod = kl_divergence(training_props, production_props)
kl_prod_to_train = kl_divergence(production_props, training_props)
# These two numbers are DIFFERENT — KL is asymmetric

Asymmetry is key: KL(P||Q) ≠ KL(Q||P). “How surprised is training by production” ≠ “how surprised is production by training.” PSI sums KL in both directions — symmetric, doesn’t depend on which distribution is “reference.”

Catch: KL divergence undefined when Q has zero probability where P has nonzero probability (log of infinity). Fix: smooth both distributions with small epsilon, or use Jensen-Shannon divergence (average of KL both directions, always finite).

When to Use Which

Mental model:

MethodBest ForGives YouWatch Out For
PSIBusiness reporting, stakeholder dashboardsSingle interpretable numberDepends on bin count; no p-value
KS testContinuous features, statistical rigorStatistic + p-valueOverly sensitive at large sample sizes
KL divergenceInformation-theoretic analysis, model internalsDivergence score in nats/bitsAsymmetric; undefined when bins are empty

Many teams run all three. PSI on stakeholder dashboard. KS tests feed alerting. KL divergence in deep-dive notebooks. Different lenses, same question: did data move?

Statistical tests don’t catch schema drift — structural, not statistical: column renamed, new enum value appeared, field now has nulls. Usually caused by upstream engineer refactoring event logging without telling you. Defense: validation checks that run before model ever sees data.

def validate_schema(df, expected):
    """Structural data checks — run BEFORE inference."""
    issues = []
    for col, rules in expected.items():
        if col not in df.columns:
            issues.append(f"Missing column: {col}")
            continue
        null_rate = df[col].isnull().mean()
        if null_rate > rules.get("max_null_rate", 0.01):
            issues.append(f"{col}: null rate {null_rate:.1%} exceeds limit")
        if "allowed_values" in rules:
            unexpected = set(df[col].dropna().unique()) - set(rules["allowed_values"])
            if unexpected:
                issues.append(f"{col}: unexpected values {unexpected}")
    return issues

Walks each expected column: exists? null rate within bounds? categorical values in allowed set? Catches class of problems statisticians call “bugs not drift” — causes ~40% of production ML incidents.

Short version: monitor input distributions with PSI or KS tests, validate schema before every inference batch, know that drift type determines response (retrain, recalibrate, or collect new data).

Detection is half the story. Drift type determines what to do, how to test replacement safely, how to investigate 2 AM incident.

Performance Monitoring Without Ground Truth

Uncomfortable truth: can’t compute accuracy in real time without ground truth. ShopFlow recommendation: won’t know if correct until user buys or doesn’t — minutes to days later. Loan default prediction: months. Long-term health outcomes: years.

Two-layer strategy. Like doctor’s checkup: layer one is vital signs — temperature, heart rate. Don’t tell you the disease, but abnormal means something wrong. Layer two is lab results taking days — definitive diagnosis.

Layer one — proxy signals (real-time): Track output distribution. Model’s avg confidence changing? Predicting one class more often? Features drifting? None confirm model is wrong, but changes warrant investigation — vital signs.

Layer two — true metrics (delayed): When ground truth arrives, compute accuracy, precision, recall, AUC. Backfill dashboards. Trigger retroactive alerts if numbers bad.

Clever bridging technique from NannyML: Confidence-Based Performance Estimation (CBPE). During validation (have labels), learn relationship between model’s confidence and actual accuracy. “90% confidence → correct 87% of time.” In production, use calibration curve to estimate performance from confidence scores alone.

import numpy as np

def estimate_accuracy_cbpe(confidences, calibration_map):
    """
    Estimate accuracy without ground truth using CBPE.
    calibration_map: dict mapping confidence bins to historical accuracy.
    Example: {0.5: 0.52, 0.6: 0.61, 0.7: 0.73, 0.8: 0.82, 0.9: 0.91}
    """
    bins = sorted(calibration_map.keys())
    estimated_correct = 0
    for conf in confidences:
        # Find the closest calibration bin
        closest_bin = min(bins, key=lambda b: abs(b - conf))
        estimated_correct += calibration_map[closest_bin]
    return estimated_correct / len(confidences)

Takes batch of production confidence scores, maps each to historically observed accuracy. Result: estimate of current model performance without labels.

Critical assumption: calibration relationship must hold in production. Concept drift breaks CBPE — model stays confident but is wrong. CBPE works for covariate shift (model less confident in unfamiliar regions), breaks for concept drift.

Slice metrics. Aggregate numbers hide disasters. ShopFlow’s 78% overall could be 95% for returning users and 40% for new users. Or 90% desktop, 55% mobile. Aggregate looks fine. One segment on fire. Monitor every dimension: geography, device, user cohort, time of day, traffic source.

Alerting That Doesn’t Destroy Your Sanity

50 alerts per day, 48 false positives. Within two weeks everyone ignores all 50. Real incident — feature pipeline silently dropping values — came in at alert 49. Nobody looked for three days.

Alert fatigue = boy who cried wolf. Solution: not fewer checks — severity architecture that respects team attention.

Two threshold types work together. Static thresholds: absolute floor — “if accuracy drops below 0.70, wake someone up.” Catches catastrophic failures fast. Misses gradual decline. ShopFlow accuracy could decay 0.78 → 0.72 over six weeks and never trigger.

Statistical change detection catches slow bleed. Asks: “is this week’s performance statistically different from trailing four-week average?” CUSUM (Cumulative Sum Control Chart) and Page-Hinkley test accumulate small deviations, fire when cumulative evidence exceeds threshold.

Use both. Static for “building on fire.” Statistical for “something slowly getting worse.” Different urgencies, different channels.

class CUSUMDetector:
    """Cumulative sum change-point detector for streaming metrics."""
    def __init__(self, threshold=5.0, drift_rate=0.5):
        self.threshold = threshold
        self.drift_rate = drift_rate
        self.pos_cusum = 0
        self.neg_cusum = 0

    def update(self, value, expected_mean):
        self.pos_cusum = max(0, self.pos_cusum + value - expected_mean - self.drift_rate)
        self.neg_cusum = max(0, self.neg_cusum - value + expected_mean - self.drift_rate)

        if self.pos_cusum > self.threshold:
            self.pos_cusum = 0
            return "upward_shift"
        if self.neg_cusum > self.threshold:
            self.neg_cusum = 0
            return "downward_shift"
        return None

drift_rate controls sensitivity — how much deviation tolerated before accumulating evidence. threshold controls how much accumulated evidence triggers alert. Start with defaults, calibrate against historical data. More than two false alerts per week: increase threshold. Real incidents slipping through: decrease it.

Structure alerts into tiers. Critical: page on-call (accuracy catastrophically below floor, serving errors above 5%, all-nulls in required feature). Warning: Slack channel for investigation within hours (significant PSI, confidence distribution shift). Info: weekly dashboard review (minor drift, new categorical values). Hierarchy keeps pager sacred — when it fires, people respond.

Shadow Mode and the Champion-Challenger Pattern

Drift detected. Model retrained. Now need to deploy. But how to know it’s actually better in production, not just better on validation data?

Weather station: before replacing thermometer, run old and new side by side comparing against reference. Shadow mode does same for ML models.

Deployed model = champion. New candidate = challenger. Shadow mode: every request processed by both. Champion’s predictions go to users, affect outcomes. Challenger’s predictions logged, thrown away — affect nothing. Both see identical inputs, compare after the fact.

def serve_with_shadow(request, champion_model, challenger_model):
    # Champion serves the real response
    champion_pred = champion_model.predict(request.features)

    # Challenger runs in shadow — its output is logged, not used
    challenger_pred = challenger_model.predict(request.features)

    log_shadow_comparison(
        request_id=request.id,
        champion_prediction=champion_pred,
        challenger_prediction=challenger_pred,
        timestamp=time.time(),
    )

    # Only the champion's prediction reaches the user
    return champion_pred

After enough data and labels arrive, compare both. Challenger consistently wins on primary metric and doesn’t degrade guardrails (latency, error rate, fairness) → promote to champion.

Shadow mode: zero-risk. Users never see challenger output. Cost: running two models per request. Cheap for ShopFlow recommendation; for LLM inference, shadow on sampled subset instead.

Blind spot: can’t measure behavioral differences — effects only manifest when users see prediction. Challenger recommends different products → different browsing, different purchases. Shadow tells you “challenger would have predicted X,” not what would have happened if user saw X. For that, need real experiment.

A/B Testing ML Models — The Traps Nobody Warns You About

A/B testing button color: split users, measure click-through, run t-test. A/B testing ML models has traps.

ShopFlow tests v2 vs v1: split users 50/50. Standard so far. Here’s where it gets uncomfortable.

Trap 1 — Network effects. v2 aggressively recommending headphones might deplete stock, affecting v1 users who would have bought headphones. Treatments aren’t isolated. Marketplaces, ad auctions, social platforms — interference real, makes results meaningless. Defense: split by isolated units (geographic regions, time periods) not individual users.

Trap 2 — Delayed outcomes. ShopFlow’s real metric: “purchase within 7 days.” Experiment must run at least 7 days after last user assignment before computing results. Peeking early and stopping when results look good inflates false positive rate. Peeking problem: endemic in ML A/B tests with long feedback loops.

Trap 3 — Multiple metrics. Testing purchase rate, revenue, latency, satisfaction simultaneously at 5% significance → ~19% chance of at least one false positive. Bonferroni correction: divide significance threshold by number of tests, α = 0.05/4 = 0.0125 per metric.

Non-negotiable: deterministic assignment. Same user must always see same model variant, regardless of server. Hash-based assignment:

import hashlib

def assign_variant(experiment_id, user_id, control_pct=0.50):
    """Deterministic, reproducible variant assignment."""
    raw = f"{experiment_id}:{user_id}".encode()
    hash_val = int(hashlib.sha256(raw).hexdigest(), 16)
    bucket = (hash_val % 10000) / 10000
    return "control" if bucket < control_pct else "treatment"

Same assignment every time for same user and experiment. SHA-256 distributes users uniformly. No database lookups, no state, works across distributed servers.

Define guardrail metrics before experiment starts. Metrics you refuse to degrade even if primary metric improves. ShopFlow guardrails: p99 latency below 200ms, error rate below 1%, no demographic group’s purchase rate drops more than 5%. Any guardrail fails → experiment fails, regardless of primary metric gain.

Retraining Triggers — When to Pull the Lever

Drift detected. Performance degrading. When to retrain? Too early: waste compute on normal fluctuation. Too late: users suffer bad predictions. Calibration problem — no universal answer, but patterns exist.

Time-based triggers: retrain every week/month/quarter. Easy (cron job), guarantees freshness. Retrain when nothing changed; fail to retrain when something changes between runs.

Performance-based triggers: fire when metric drops below threshold — “retrain if weekly accuracy < 0.85.” More responsive, but requires ground truth labels — often delayed.

Drift-based triggers: fire when statistical tests detect significant distribution change — “retrain if PSI > 0.25 or KS test p < 0.001.” No ground truth needed, fires immediately. Risk: false alarms from natural variation.

Volume-based triggers: fire after accumulating enough new data — “retrain after 100,000 new labeled examples.” Ensures sufficient training data, ignores whether data actually changed.

Robust approach: combine multiple triggers. ShopFlow: monthly scheduled baseline + immediate retrain when PSI > 0.25 on any top-5 feature + retrain within a week when weekly accuracy < 0.75 (once delayed labels arrive).

class RetrainingDecision:
    """Combine multiple signals to decide when to retrain."""
    def __init__(self, max_age_days=30, psi_threshold=0.25,
                 accuracy_floor=0.75, min_new_samples=10000):
        self.max_age_days = max_age_days
        self.psi_threshold = psi_threshold
        self.accuracy_floor = accuracy_floor
        self.min_new_samples = min_new_samples

    def should_retrain(self, model_age_days, max_feature_psi,
                       current_accuracy, new_sample_count):
        reasons = []
        if model_age_days > self.max_age_days:
            reasons.append(f"Model is {model_age_days} days old")
        if max_feature_psi > self.psi_threshold:
            reasons.append(f"Feature PSI {max_feature_psi:.3f} exceeds threshold")
        if current_accuracy is not None and current_accuracy < self.accuracy_floor:
            reasons.append(f"Accuracy {current_accuracy:.3f} below floor")

        should = len(reasons) > 0 and new_sample_count >= self.min_new_samples
        return should, reasons

min_new_samples guard: don’t retrain when insufficient fresh data. Function returns decision and reasons — log why you retrained. Audit these decisions later.

The Observability Stack — Logs, Metrics, Traces for ML

Monitoring: predefined checks against known questions. Observability: broader capability — answer questions you didn’t anticipate. “Why did predictions for Brazilian users flip Tuesday?” No alert for that specific scenario, but solid observability stack lets you investigate after the fact.

Three pillars, settled on years ago: logs, metrics, traces. Apply to ML with adaptations. Weather station: logs are detailed journal entries, metrics are time-series charts, traces are end-to-end timeline for single measurement.

Logs — What to Capture for Every Prediction

What to never skip logging:

import json, time, uuid

def log_prediction(model_version, features, prediction, confidence, metadata=None):
    """Structured prediction log — your future debugging self will thank you."""
    return {
        "prediction_id": str(uuid.uuid4()),
        "timestamp": time.time(),
        "model_version": model_version,

        # Input snapshot — you WILL need this for debugging
        "features": features,
        "feature_hash": hashlib.md5(
            json.dumps(features, sort_keys=True).encode()
        ).hexdigest(),

        # Output details
        "prediction": prediction,
        "confidence": confidence,

        # Context for investigation
        "latency_ms": None,  # filled by serving layer
        "user_id": metadata.get("user_id") if metadata else None,
        "request_source": metadata.get("source") if metadata else None,
        "experiment_variant": metadata.get("variant") if metadata else None,
    }

prediction_id: correlation key — ties prediction to downstream outcomes and debug sessions. model_version: which model made call (critical during A/B tests or shadow deployments). features: input snapshot — without it, can’t reproduce bad prediction three weeks later. feature_hash: compact fingerprint for deduplication.

Storage cost: logging every feature vector for every prediction gets expensive at scale. If constrained, log hash and store full feature vectors in time-limited queryable store (90-day TTL on columnar database). Hash lets you find matching records when needed.

Metrics — The Time-Series Dashboard Layer

Metrics: aggregated numbers tracked over time. Essential ones for ML serving:

from prometheus_client import Histogram, Counter, Gauge

# How long each prediction takes
PRED_LATENCY = Histogram(
    "ml_prediction_seconds", "Inference latency",
    buckets=[0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1.0]
)

# How many predictions we serve, broken down by class
PRED_COUNTER = Counter(
    "ml_predictions_total", "Predictions served",
    ["model_version", "predicted_class"]
)

# Current PSI score for each feature
DRIFT_PSI = Gauge(
    "ml_feature_psi", "PSI score per feature",
    ["feature_name"]
)

Histogram tracks full latency distribution (not average — averages hide tail behavior). Counter tracks prediction volume and class distribution — if model suddenly predicts one class 90% of time, visible here. Gauge for PSI makes drift scores visible on Grafana and alertable via Prometheus rules.

Traces — Following a Prediction Through the System

Single prediction at ShopFlow touches: API gateway, feature store lookup, preprocessing, model inference, post-processing with business rules, response serializer. Latency spikes — which stage?

from opentelemetry import trace

tracer = trace.get_tracer("shopflow-ml")

def predict_with_tracing(request):
    with tracer.start_as_current_span("ml-prediction") as parent:

        with tracer.start_as_current_span("feature-retrieval"):
            features = fetch_features(request.user_id)

        with tracer.start_as_current_span("preprocessing"):
            tensor = preprocess(features)

        with tracer.start_as_current_span("model-inference"):
            raw_output = model(tensor)

        with tracer.start_as_current_span("postprocessing"):
            result = apply_business_rules(raw_output)

        parent.set_attribute("model.version", MODEL_VERSION)
        parent.set_attribute("model.confidence", float(result.confidence))
        return result

Each start_as_current_span creates timed segment. Spans nest — “feature-retrieval” appears as child of “ml-prediction.” Ship to Jaeger or Tempo: visual timeline per request. “Predictions slow today” → pull trace view, immediately see bottleneck: feature store, model, or business rules.

Three pillars connect via shared identifiers. prediction_id from logs matches trace ID from OpenTelemetry, labels Prometheus metrics. Dashboard shows latency spike (metrics) → click through to traces for that window, identify slow span → search logs for prediction IDs to see features involved. Cross-pillar navigation turns monitoring data into debugging power.

Debugging Production Models — The Investigation Playbook

2 AM. Alert fires. ShopFlow recommendation accuracy dropped 78% → 52% over last 6 hours. Playbook in order — each step resolves issue or narrows search space.

First: infrastructure or model? Check latency, error rates, CPU/memory, pod health. Serving container out of memory and crashing — ops problem, not ML. Weather station: check if thermometer is plugged in before questioning readings.

Second: data problem? Run schema validation. Check null rates, feature distributions, PSI scores. ~40% of “model problems” are data pipeline problems: upstream column renamed, third-party API changed format, feature store cache expired. Highest hit rate, least glamorous check.

Third: specific segment? Slice accuracy by every dimension — country, device, user cohort, time, traffic source. 52% overall but 78% desktop and 15% mobile → blast radius dramatically narrowed. Segment tells you where to look next.

Fourth: when did it start? Plot metric over time, find inflection point. Overlay deployments, pipeline runs, upstream changes, external events. Accuracy dropped at 8 PM, pipeline ran at 7:45 PM — not coincidence.

Fifth: can you reproduce it? Pull production examples where model was wrong. Run through model locally with exact same feature values. If features were logged: straightforward. If not — this is when you promise to start logging them.

def investigate(predictions_log, bad_period, good_period):
    """Automated first-pass: compare bad period against a known-good baseline."""
    bad = predictions_log.query(f"timestamp.between('{bad_period[0]}', '{bad_period[1]}')")
    good = predictions_log.query(f"timestamp.between('{good_period[0]}', '{good_period[1]}')")

    report = {}

    # Has the prediction distribution shifted?
    from scipy import stats
    report["prediction_shift"] = stats.ks_2samp(
        good["prediction"], bad["prediction"]
    )

    # Which features drifted?
    feature_cols = [c for c in bad.columns if c.startswith("feat_")]
    report["drifted_features"] = {
        col: stats.ks_2samp(good[col].dropna(), bad[col].dropna())
        for col in feature_cols
        if stats.ks_2samp(good[col].dropna(), bad[col].dropna()).pvalue < 0.01
    }

    # Which segments are worst?
    for seg in ["country", "device", "source"]:
        if seg in bad.columns:
            report[f"segment_{seg}"] = bad.groupby(seg)["correct"].mean().to_dict()

    return report

Automates first-pass investigation: compares prediction distribution, identifies drifted features, breaks down accuracy by segment. Won’t solve problem — still requires judgment — but saves 20 minutes of manual slicing at 2 AM when brain runs at half capacity.

Production ML incident breakdown: ~40% data pipeline problems, ~25% upstream schema/feature changes, ~20% genuine drift, ~15% infrastructure. Start with data — not a guess, playing the odds.

Choosing Your Tools

Tool landscape crowded. Honest assessment after working with several:

Evidently AI: open-source, Python-native, interactive reports. Fastest path from zero monitoring to useful drift dashboard. Best for batch monitoring, notebook exploration, full control without SaaS pricing. Limitation: not optimized for high-volume real-time streaming.

NannyML: open-source, specializes in delayed-label problem. CBPE implementation estimates performance without ground truth — no other tool matches this. Labels take days or weeks (insurance, loans, medical) → NannyML fills gap.

WhyLabs: SaaS with open-source agents. Enterprise scale: privacy-preserving analytics, real-time monitoring, cross-team collaboration. Best for larger orgs with compliance requirements (healthcare, banking) willing to pay for managed infrastructure.

Prometheus + Grafana: DIY path. Ops team already runs this stack? Expose ML-specific metrics from serving endpoint, integrate into existing dashboards. Build ML-specific parts yourself. Full control, zero vendor lock-in.

Alibi Detect: drift-detection toolkit from Seldon, strong statistical tests and outlier detection. Toolkit, not platform — bring own infrastructure for storing results and alerting.

For ShopFlow (small, budget-conscious): Evidently for drift reports, Prometheus + Grafana for operational metrics. Labels delayed → add NannyML. Company grows → evaluate WhyLabs. No one tool covers everything.
