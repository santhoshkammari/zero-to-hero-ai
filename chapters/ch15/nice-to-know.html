I avoided these topics for while. Each felt like someone else's job: technical debt for platform team, regulations for legal, carbon footprint for policy. Then I spent year shipping models to production, and every one hit me personally. Here dive.

Unglamorous production ML realities. These do not show up in model architecture papers, but they decide whether system survives real world. Silent technical debt. Data errors compounding like interest. Feedback loops going haywire. Regulations reshaping what you can build. Team dynamics that can sink project faster than any bug.

ML Technical Debt

In 2015, Google team published "Hidden Technical Debt in Machine Learning Systems." It became one of most cited papers in ML engineering. Not because it introduced new algorithm. Because it named thing everyone felt. Central diagram blunt: "ML Code" is tiny black rectangle inside massive system of configuration, data collection, feature extraction, serving infrastructure, and monitoring. Model code — thing everyone obsesses over — is smallest piece.

Make this concrete. Imagine building fraud detection system at mid-size fintech company. You start with notebook, clean dataset, and XGBoost model at 0.96 AUC. Beautiful. Ship it. Six months later, system is different beast.

First debt type: glue code. Duct tape for system. Fraud model does not live alone. It pulls features from transaction database, calls external IP geolocation API, reads from feature store another team maintains, and pushes predictions into downstream rules engine. These systems do not speak same language. So you write glue: converters, adapters, serializers, retry logic. In fraud system, you end up with roughly 2,000 lines of model code and 15,000 lines of glue. Sculley paper found this ratio typical. Often worse.

Second type: pipeline jungles. Over months, fraud system grows data pipelines: one for real-time transaction features, one for nightly historical aggregates, one for external enrichment data, one colleague wrote for holiday edge case and never removed. Each pipeline has own schedule, own failure modes, own hidden assumptions about data format. Nobody holds complete mental model. One Tuesday, nightly aggregate job runs 20 minutes late. Three downstream pipelines consume stale data. Nobody notices for week.

Then configuration debt. Fraud model has dozens of parameters outside code: feature thresholds, sampling rates, model version pointers, serving timeouts, fallback logic. These rarely get code-level rigor. No code review. No version control. Sometimes no documentation. Paper points out single misconfigured threshold can silently degrade model performance by 20% and stay hidden for months, because nobody monitored config values themselves.

First time I read that paper, I thought it exaggerated. Then I inherited production ML system and found configuration files untouched for two years, still pointing to renamed feature store column. Model kept running, pulled zeros for that feature, compensated with other signals. Technical debt is not dramatic. It is quiet.

Uncomfortable takeaway: ML technical debt does not come from carelessness. It comes from doing what feels rational now — shipping fast, reusing what exists, deferring cleanup. Debt compounds not with time alone, but with number of interacting systems. In production ML, everything interacts with everything.

Data Cascades

If technical debt is slow rot of code and configuration, data cascades are slow rot of thing code operates on. In 2021, Sambasivan and colleagues at Google interviewed 53 ML practitioners across high-stakes domains: healthcare, agriculture, conservation. They coined data cascades: compounding events where data quality issues in one stage propagate and amplify through later stages, creating problems hard to trace back to origin.

Paper title says whole thing: "Everyone wants to do the model work, not the data work."

Back to fraud detection system. Suppose ingested transaction data has subtle issue: in certain European countries, currency field sometimes arrives as empty string instead of ISO currency code. Maybe 0.3% of transactions. Data pipeline fills default — USD. Now feature computing "transaction amount relative to average for this currency" is slightly wrong for those transactions. Not catastrophic. Wrong enough for model to learn subtle bias: European transactions look slightly more fraudulent than they are, because amounts look anomalous against wrong currency baseline.

That 0.3% raw-data error cascades through feature engineering, training, and prediction. Six months later, European fraud analysts notice model flags their region too often. They file ticket. Engineer investigates. Three weeks later, someone traces it to that currency default. Original data-quality issue stayed invisible — no test, no monitor, no dashboard caught it — because value was technically valid.

Sambasivan study found 92% of interviewed practitioners had seen data cascades. Not once. Repeatedly. Cascades were usually discovered late, often by downstream users noticing model behavior "felt off." Median time from cascade start to discovery: weeks to months.

Deeper problem is cultural. Study found persistent hierarchy in ML teams: data work — collection, cleaning, labeling, validation — gets treated as lower-status work. "Real" work is architecture and training. So people closest to data-quality issues often lack power to delay launch over data concern. I have watched this happen: data engineer flags inconsistent labels, gets told "we'll clean it up in v2," and v2 never comes because model is already in production and making money.

This is not technology problem. It is values problem. It shows up in hiring too: teams interview for months to find right senior researcher, then hand data labeling to cheapest vendor they can find.

Feedback Loops Gone Wrong

Feedback loop in ML happens when model predictions shape data future model versions train on. Best case: virtuous cycle — data flywheel from earlier in chapter. Worst case: model reinforces own mistakes until they become self-fulfilling prophecies.

Most chilling example: predictive policing. Model trains on historical arrest data to predict where crime will occur. Police go to predicted locations. More arrests happen there — not necessarily because more crime happens there, but because more officers are looking. New arrest data confirms model predictions, so next version doubles down on those neighborhoods. Cycle accelerates. Whole communities get locked into over-policing loop created not by true crime patterns, but by historical enforcement bias baked into training data.

This is not hypothetical. Oakland, California deployed predictive policing and found system repeatedly sent officers to predominantly Black and low-income neighborhoods, not because crime rates were objectively higher, but because those neighborhoods had historically heavier police presence — and therefore more documented incidents.

Recommendation systems have milder but pervasive version of same problem. Consider music streaming platform. User listens to jazz track once, maybe by accident. Recommender notices and suggests more jazz. User sees jazz everywhere and plays another track — maybe from curiosity, maybe because little else shows up. Recommender reads this as strong signal. Within weeks, whole feed is jazz. Actual taste did not change, but recommendation profile got trapped in filter bubble — self-reinforcing loop where system suggestions shape behavior system later measures.

YouTube's recommendation algorithm faced scrutiny for more dangerous version. Users who watched one mildly political video got recommended increasingly extreme content, because extreme content drove higher engagement, and higher engagement confirmed recommendation. Algorithm was not designed to radicalize anyone. It was designed to maximize watch time. Radicalization turned out to work for that objective.

Fraud system is not immune. Suppose model aggressively flags certain transaction patterns, and human reviewers — trusting model — confirm those flags at high rate. Model learns those patterns really are fraudulent, because predictions keep getting confirmed, and flags them even harder next cycle. Meanwhile, patterns model misses never get reviewed, never get labeled as fraud, and never enter training data. Model grows blind spots it cannot correct, because it never sees own mistakes.

I am still building intuition for how to detect feedback loops before damage starts. Honest answer: most organizations find them only after visible break — news article, customer complaint, or internal audit showing bias compounded for months.

The AI Incident Database

ML systems fail in real world with depressing reliability. Strange part: for most of field's history, there was no systematic way to learn from those failures. Aviation has NTSB. Medicine has adverse event reporting. Software engineering has post-mortems. ML had blog posts and Twitter threads.

AI Incident Database (AIID), maintained by Partnership on AI, was built to fill that gap. It is public, searchable collection of incidents where AI systems caused or contributed to real-world harm. Anyone can submit incident. Editors review and categorize each entry with metadata: type of harm, domain, technology involved, affected population.

Some entries are famous. COMPAS recidivism prediction system — risk scoring tool used in American criminal justice system — systematically rated Black defendants as higher risk than they were, and white defendants as lower risk. Amazon built resume screening tool that learned to penalize applications containing word "women's" — as in "women's chess club" — because training data came from decade of hiring skewed heavily male. Apple's credit card algorithm offered women lower credit limits than their husbands, even when women had higher credit scores.

AIID matters not because individual stories are new — press covered many of them. Value is aggregate. Browse hundreds of incidents, and patterns appear. Common failure mode: model trains on historical data, inherits historical bias, gets deployed at scale, and amplifies bias in ways developers did not expect. Another pattern: system looks good in testing, then hits edge cases in real world missing from training or evaluation data.

Database also shows how long incidents stay hidden — often months or years. It shows who lacks power to report them. People most affected are often least empowered. Criminal defendants do not file bug reports against sentencing algorithms.

Spending hour in AIID is sobering. Not because failures are exotic. Because hindsight makes them look predictable. Many could have been caught with more diverse evaluation, more domain expertise in room, or more attention to who system was deployed on instead of who system was built for.

Chaos Engineering for ML

Netflix broke things on purpose and called it engineering. In early 2010s, they built Chaos Monkey — tool that randomly terminates production servers to verify system can handle loss. Philosophy: if system cannot tolerate expected failures, better learn on Tuesday morning than during holiday traffic spike.

ML systems have own failure modes, and most are subtler than server outage. What happens to fraud detection system when feature store returns stale data? When model serving endpoint is 500ms slower than usual? When new deployment gets input data with slightly different schema? When third-party API providing enrichment features goes offline?

Chaos engineering for ML means injecting these failures in controlled conditions and watching result. Netflix extended its tools — ChAP (Chaos Automation Platform) and FIT (Failure Injection Testing) — to systems supporting recommendation models. They do not only test "does server stay up?" They test "when primary recommendation model is unavailable, does system gracefully fall back to simpler baseline model, and what is user impact?"

Practice follows structured loop. First, define steady state — maybe "99.9% of recommendation requests complete in under 100ms." Then hypothesize failure: "If primary model server goes down, fallback model will serve within 200ms." Inject failure. Measure result. If reality does not match hypothesis, you found vulnerability before users did.

For fraud system, this might look like: deliberately feeding model batch of transactions with missing features to confirm fallback logic activates correctly. Or throttling model serving endpoint to simulate network degradation and verifying system queues transactions for review instead of silently dropping them. Or deploying model version with corrupted weights to verify health check catches it before any predictions are served.

Most ML teams I talk to do none of this. They test that model works. They do not test what happens when it fails. Gap between those two is where production incidents live.

The Carbon Cost of ML

In 2019, Strubell and colleagues at UMass Amherst put hard numbers on something field already felt uneasy about. Training one large NLP model — Transformer with architecture search — produced roughly 284 tonnes of CO₂ equivalent. About five times lifetime emissions of average American car, including manufacturing.

Those numbers came from 2019. Models got larger after that.

Carbon cost of ML has two sources: training and inference. Training gets headlines. Inference is quieter and often larger. Model trains once — or retrains weekly — but serves predictions millions of times per day, every day, for years. Aggregate inference energy across fleet of models often exceeds training cost by order of magnitude.

For fraud system, this is not abstract. Suppose model processes 50 million transactions per day. Each inference uses fraction of GPU-second, but multiplied by 50 million, that becomes real energy. Add nightly retraining, data pipelines, feature computation, and monitoring infrastructure. Total energy footprint becomes much larger than model training alone.

Tools like CodeCarbon — Python library estimating CO₂ emissions of compute — and ML CO2 Impact — web calculator — let you measure this. Carbon-aware scheduling goes further: run training when electrical grid near data center draws more from renewable sources. Training at 2 AM in wind-powered region produces meaningfully less carbon than training at 6 PM in region relying on natural-gas peakers.

My thinking changed here. Few years ago, I filed "ML carbon footprint" under "not my problem." Then I saw numbers for production system I owned, and total annual compute bill — rough proxy for energy use — was four times higher than I expected. Most of that came from inference, not training. Nobody had added it up before, because nobody asked.

Field is slowly moving toward papers reporting not only accuracy, but also compute cost and estimated emissions. Similar to drug trials reporting not only efficacy, but also side effects. Question is not only whether model is accurate. Question is whether accuracy is worth cost.

The Regulatory Landscape

For most of ML history, regulatory environment was basically "ship whatever you want and hope for best." That era is ending. Two frameworks are reshaping production ML now. Understanding them is not optional. It is prerequisite for building systems that do not drag you or company into legal trouble.

EU AI Act, formally adopted in 2024, classifies AI systems into four risk tiers. Top tier: unacceptable risk — systems banned outright. These include government social scoring, real-time biometric surveillance in public spaces with narrow exceptions, manipulation of vulnerable populations, and some forms of predictive policing. If system lands here, there is no compliance path. Do not build it.

Next tier is high risk. Most production ML teams need to care here. High-risk systems include AI used in hiring, credit scoring, education, law enforcement, critical infrastructure, and medical devices. If fraud detection system makes automated decisions about access to financial services, it likely qualifies. High-risk systems face mandatory requirements: formal risk assessments, high-quality training data with documented provenance, human oversight, technical documentation sufficient for third-party audit, and ongoing post-deployment monitoring. Noncompliance can cost up to €35 million or 7% of global revenue, whichever is larger.

Remaining tiers are limited risk — transparency duties, like telling users they interact with AI — and minimal risk — mostly unregulated, where most research and open-source tools sit.

Across Atlantic, FDA has been building regulatory framework for ML in healthcare through Software as a Medical Device (SaMD). If ML model diagnoses, monitors, or treats medical condition, it is medical device and needs FDA clearance. Classification uses standard Class I / II / III structure, with most ML-based SaMDs in Class II. Hard part is self-updating models. Traditional devices do not change after approval. ML models retrain on new data. FDA response is Algorithm Change Protocols: pre-approved plans describing how model will update, what performance thresholds must hold, and how changes will be validated. Model can evolve, but only inside protocol bounds.

Both frameworks share one thread: they do not regulate technology itself. They regulate impact on people. Model can use any architecture it wants. But if it makes decisions about health, freedom, employment, or financial access, you must show it works responsibly, transparently, and with accountability.

Regulatory landscape is still genuinely uncertain. EU AI Act passed, but many provisions will not be enforced until 2025-2027, and interpretive guidance is still being written. FDA approach to continuously learning models is evolving in real time. If you build production ML in regulated domain, part of job is tracking regulations themselves, because target keeps moving.

ML Team Anti-Patterns

Every production ML failure I have seen was people problem wearing technology costume. Model did not fail because gradient descent broke. It failed because team was organized in way that made failure inevitable.

Most common anti-pattern: "all researchers, no plumbers." Team is all data scientists and ML researchers with PhDs. They build beautiful notebook models. Nobody can deploy them. Nobody wants to deploy them, because deployment is "engineering work" and beneath team self-image. Models rot in notebooks while company waits. In fraud system, this looks like state-of-the-art AUC on benchmark dataset and no path to production without three-month engineering rescue from another team.

Inverse is just as destructive: "all plumbers, no researchers." Team of software engineers can build excellent infrastructure but lacks statistical grounding to know when model is subtly broken. They ship model with 95% test accuracy without noticing test set shifted away from production, or that wrong 5% is concentrated in one demographic group.

Then comes premature specialization. Startup with 8 people hires dedicated computer vision specialist, dedicated NLP specialist, and dedicated recommendation systems specialist before company even knows which ML problem matters most. Six months later, product pivots, and two of three specialists work outside hiring target. Early-stage teams need generalists who can handle whatever ML problem appears. Specialization comes later, after problem space stabilizes.

No MLOps representation is silent killer. Team of 10 has 8 data scientists, 1 engineering manager, and 1 backend engineer. Nobody owns deployment pipeline, monitoring infrastructure, data validation, or model versioning. Work gets done ad hoc by whoever has spare capacity, which means it gets done badly or not at all. By time team realizes it needs dedicated MLOps expertise, technical debt is already large.

Final anti-pattern is organizational isolation. ML team sits in own corner, away from product, engineering, design, and domain experts. They build what they think is needed, not what is needed. In fraud system, ML team optimizes for AUC while fraud analysts care about precision at top of ranked list. Team measures success differently from people using model, and nobody notices because they do not talk often enough.

Fix for all of this is unsexy: cross-functional teams, clear ownership, healthy balance of research and engineering talent, and enough proximity to end users that team cannot accidentally build wrong thing. I have never seen this done perfectly. I have seen it done well enough.

Build vs. Buy

At some point, every ML team hits this question: build own ML infrastructure, or buy existing platform? Answer is always "it depends." And dependency is almost never what people first think.

Instinct — especially on strong engineering teams — is to build. Custom solutions feel flexible. Tailored to exact needs. Owning whole stack feels good. But custom infrastructure carries maintenance tax that compounds over time. Every component you build becomes component you must maintain, debug, upgrade, and staff. In-house feature store looks great until engineer who designed it leaves and nobody else understands custom serialization format.

Vendor route — AWS SageMaker, Google Vertex AI, Databricks, MLflow as managed service — trades flexibility for speed and lower maintenance burden. You get working system in weeks instead of months. But you also inherit someone else's opinions about how ML should work, baked into platform abstractions. When use case does not fit those abstractions, you fight platform instead of using it.

For fraud system, decision matrix looks like this. Data pipeline is mostly standard ETL plus some real-time streaming — buying this, like Kafka managed service or cloud data warehouse, is usually right. Feature engineering is domain-specific and tightly tied to fraud patterns team learned — building custom feature computation logic is usually right. Model training is mostly standard supervised learning — managed training service is reasonable. Model serving with low-latency requirements and custom fallback logic often requires building, because vendor serving solutions rarely support exact failure modes you need to handle.

Practical answer for most teams is hybrid: buy undifferentiated heavy lifting — compute, storage, orchestration, experiment tracking — and build components tied to competitive advantage or domain-specific customization. Biggest mistake is not choosing build or buy. Biggest mistake is failing to revisit choice as team and product evolve. Custom feature store that made sense with 3 features and 1 model becomes liability with 200 features and 15 models. Vendor platform that felt constraining at 2 engineers becomes essential at 20.

Heuristic I keep coming back to: if component would be boring to build — no proprietary logic, no domain-specific innovation, no competitive advantage — buy it. Save engineering hours for parts that make system uniquely good at job.
