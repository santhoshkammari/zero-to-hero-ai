Key takeaways and recommended data infrastructure references

First two ML years, I treated model as product. Spent weeks squeezing extra 0.3% AUC, deployed, watched performance decay over months. Post-mortems said same thing: not model. Data. Missed schema change. Feature computed differently in training and serving. Labeling batch used old guidelines. I treated data infrastructure like boring plumbing for later. Plumbing kept biting. Time for overdue dive.

Data infrastructure = systems that ingest, store, transform, validate, label, version, serve, and protect data ML models consume. Covers where bytes live (warehouses, lakes, lakehouses), how features reach model at serving time (feature stores), how pipelines run (Airflow, Prefect, Dagster), how labels scale (Labelbox, active learning), how rot gets caught (Great Expectations, TFDV). Field matured fast around 2019, when companies started publishing hard production lessons.

The Running Example: BiteBridge

This section builds data infrastructure for tiny food delivery startup called BiteBridge. Right now: three restaurants, five regular customers, one overloaded engineer (you). App tracks three events: order_placed, delivery_completed, and rating_submitted.

Goal: build recommendation model that predicts which restaurant user orders from next. Need place to store events, compute features, label history, validate everything, and serve features at prediction time. Each section adds one piece.

Start small on purpose. If you can trace every byte through every system with three restaurants, architecture scales to three thousand. Problems stay same. Scale makes them louder.

Where Data Lives: Lakes, Warehouses, and Lakehouses

First data-team decision: where data lives. Sounds small. Choice drives downstream speed, cost, and pain when five customers become five million.

Think of BiteBridge’s data like kitchen ingredients. Data warehouse = organized pantry: jars labeled, shelves assigned, everything sorted before entry. You define schema before writing data — schema-on-write. BigQuery, Snowflake, and Redshift work like this. Query engine is built in, so analytics are fast. Tradeoff: data must fit predefined layout. Images, audio, free-form JSON logs — these fit poorly in warehouse’s rigid tables.

Data lake = walk-in freezer. Dump data in whatever container arrives. Raw JSON, Parquet files, images, log files — all in. Structure comes later on read, called schema-on-read. Amazon S3 or Azure Data Lake Storage often host lakes. Cost per gigabyte stays very low because storage is commodity object storage. Problem: without structure, lake degrades into data swamp — terabytes nobody understands, trusts, or queries well.

First time I saw these patterns, I simplified: “warehouse = structured, lake = unstructured.” Too simple. ML teams need both: structured features for model training (warehouse strength) and raw data for reprocessing when new feature appears six months later (lake strength). Running both in parallel means syncing data between them. New headaches.

Data lakehouse tries to end split. It layers open table formats like Delta Lake or Apache Iceberg on cheap object storage, adding ACID transactions, schema enforcement, and warehouse-style query performance at lake storage prices. For BiteBridge, raw order_placed events can land in S3, then be queried with SQL like warehouse tables. We can even “time travel” and inspect exact data from any date.

# BiteBridge: time-travel query with Delta Lake
# "What did user features look like the day before that bad model deploy?"
from pyspark.sql import SparkSession

spark = SparkSession.builder.appName("bitebrige_features").getOrCreate()

# Current version of the features table
current = spark.read.format("delta").load("s3://bitebrige-lake/user_features")

# Exact data from January 8th — before the bad deploy on January 9th
historical = (
    spark.read.format("delta")
    .option("timestampAsOf", "2024-01-08")
    .load("s3://bitebrige-lake/user_features")
)

Time-travel alone saves huge debugging time. Model performance drops? First question: “did data change?” Lakehouse answers in minutes, not days.

Lakehouse solves storage and query. Not delivery. Kitchen organized. No trucks. Need pipelines next.

How Data Moves: Batch Pipelines and Streaming

BiteBridge has three event streams: orders, deliveries, ratings. With five customers, hand-processing works. But infrastructure must survive five thousand. Real question: not whether to automate movement. How often data must arrive.

Batch pipeline runs on schedule — hourly, daily, weekly. It wakes up, processes accumulated chunk, writes results, shuts down. For BiteBridge’s recommendation model, yesterday’s orders are fresh enough. No need to react to order placed 3 seconds ago to recommend restaurant for tomorrow’s dinner. Nightly batch job computing “orders in last 30 days per user per restaurant” is enough.

# BiteBridge: nightly batch pipeline with PySpark
from pyspark.sql import SparkSession, functions as F

spark = SparkSession.builder.appName("nightly_features").getOrCreate()

orders = spark.read.format("delta").load("s3://bitebrige-lake/raw/orders/")

user_restaurant_features = (
    orders
    .filter(F.col("order_date") >= F.date_sub(F.current_date(), 30))
    .groupBy("user_id", "restaurant_id")
    .agg(
        F.count("*").alias("orders_30d"),
        F.avg("order_total").alias("avg_order_value"),
        F.max("order_date").alias("last_order_date"),
    )
)
user_restaurant_features.write.mode("overwrite") \
    .format("delta") \
    .save("s3://bitebrige-lake/features/user_restaurant_stats/")

Batch is ML infrastructure workhorse. Predictable. Debuggable. You pay for compute only when job runs. If something breaks, rerun it. Simplicity matters more than many architects admit.

Now imagine BiteBridge adds fraud detection model. User places order with stolen credit card. You need score before delivery driver leaves restaurant — seconds, not hours. Batch fails here. Need streaming pipeline that processes events continuously as they arrive.

Apache Kafka is backbone of most streaming architectures. Think conveyor belt. Producers (app, payment processor) drop events onto belt, organized by topic. Consumers (fraud model, analytics dashboard) read at their own pace. Events stay on belt for days or weeks, so slow consumer does not lose data — it catches up later. This decoupling is Kafka’s superpower.

Apache Flink sits downstream of Kafka and does heavy computation — windowed aggregations, joins across streams, complex event processing. If Kafka is conveyor belt, Flink is worker watching belt and computing “total spend by this user in last 5 minutes” in real time. Flink handles ugly streaming realities: out-of-order events, late-arriving data, exactly-once guarantees, and backpressure when data arrives faster than you can process it.

# Conceptual Flink SQL: real-time fraud features over Kafka stream
-- Count orders per user in a sliding 5-minute window
SELECT
    user_id,
    COUNT(*) AS orders_5min,
    SUM(order_total) AS spend_5min,
    COUNT(DISTINCT restaurant_id) AS unique_restaurants_5min
FROM orders_stream
GROUP BY
    user_id,
    HOP(event_time, INTERVAL '1' MINUTE, INTERVAL '5' MINUTES)

Rule I use: if stale data can cause costly wrong decision — like approving fraudulent transaction — and that cost exceeds price of always-on streaming infrastructure, stream. Otherwise batch. Most teams overestimate how many features need real time. BiteBridge’s recommendation model works fine on nightly batch. Fraud model needs streaming. Two pipelines. Fine.

Batch and streaming solve data movement. Not data orchestration. When nightly batch has five dependent steps and step three fails at 3 AM, who restarts it? Who keeps step four from running on half-written data? Need orchestrator.

Who Tells the Pipes When to Run: Orchestration

First orchestrator most teams use is cron. Cron job at 2 AM runs Python script. Works until it doesn’t — script fails silently, nobody notices for week, model trains on stale data. Cron does not know dependencies. It does not retry. It does not alert. Timer, not orchestrator.

Apache Airflow replaced cron for most data teams around 2017. You define pipeline as DAG (Directed Acyclic Graph) — set of tasks where some must finish before others start. For BiteBridge, DAG might be: ingest raw orders → compute features → validate features → train model. If ingestion fails, everything downstream halts automatically and you get paged. Airflow is battle-tested, has massive connector ecosystem, and is de facto standard at large companies.

Catch: Airflow was built for data engineering, not ML. Dynamic workflows, where task count depends on data, are awkward. Testing DAG locally means spinning up scheduler. XCom for passing data between tasks often feels like fighting tool.

Prefect and Dagster are modern challengers. Prefect wraps plain Python functions with decorators, so pipelines feel like normal code. Dagster goes further with software-defined assets: instead of defining tasks that run, you define datasets that should exist, and Dagster figures out what must run to produce them. For ML teams, Dagster’s asset-centric model maps naturally to features, training sets, and models.

# BiteBridge: daily pipeline in Prefect
from prefect import flow, task

@task(retries=2, retry_delay_seconds=300)
def ingest_orders(date: str):
    raw = fetch_orders_from_api(date)
    write_to_lake(raw, f"s3://bitebrige-lake/raw/orders/{date}/")
    return raw

@task
def compute_features(raw_orders):
    features = build_user_restaurant_features(raw_orders)
    validate_features(features)  # halt if validation fails
    return features

@task
def retrain_if_needed(features):
    model = train_model(features)
    metrics = evaluate(model)
    if metrics["ndcg"] < 0.65:
        raise ValueError(f"NDCG {metrics['ndcg']:.3f} below threshold")
    return model

@flow(name="bitebrige-daily")
def daily_pipeline(date: str):
    raw = ingest_orders(date)
    features = compute_features(raw)
    retrain_if_needed(features)

Specific orchestrator matters less than having any orchestrator. Starting fresh? Prefect or Dagster usually hurts less than Airflow. Company already runs Airflow? Don’t rewrite. It works. Important part: pipelines need dependency management, retry logic, alerting, and dashboard showing what ran and what failed.

Now BiteBridge can move data from event to feature table reliably. Good. Next trap is subtle. It appears when model starts serving.

The Training-Serving Skew Trap, and Feature Stores

War story. BiteBridge’s recommendation model trains offline with feature avg_order_value_30d. Training pipeline computes it with Spark over lakehouse. Serving system wants low latency, so separate Python microservice computes same feature from Redis cache. Two codebases. Two definitions of “30 days.” Training uses calendar days. Microservice uses rolling 720 hours. For most users, difference is small. For weekend-heavy users, numbers diverge enough to skew predictions. Production performs worse than offline — not enough to trigger alert, so nobody notices for three months.

This is training-serving skew. I lost more sleep to it than to model architecture problems. Nasty part: pipeline runs, model trains, offline metrics look fine, production performance slowly degrades.

Feature store kills this by giving one place to define, compute, store, and serve features. It has two storage layers: offline store (usually data warehouse or lakehouse) holding full feature history for training, and online store (usually Redis or DynamoDB) holding latest values for low-latency serving. Same feature definition populates both stores. One codebase. No skew.

Hardest piece for me was point-in-time correctness. When training model, you need features as they looked when event happened, not as they look today. If user churned on January 15th, training example should use features from January 14th. Using today’s features on historical event is data leakage — model peeks into future. Feature stores handle this with point-in-time joins: given entities and timestamps, they fetch feature values valid at each timestamp.

BiteBridge example with Feast, most popular open-source feature store.

# bitebrige/feature_repo/features.py
from feast import Entity, FeatureView, Field, FileSource
from feast.types import Float32, Int64
from datetime import timedelta

# An entity is the "thing" features describe
user = Entity(name="user_id", join_keys=["user_id"])

# Source: where Feast reads the feature data
user_stats_source = FileSource(
    path="s3://bitebrige-lake/features/user_stats.parquet",
    timestamp_field="event_timestamp",
)

# Feature view: a named, versioned group of features
user_stats_fv = FeatureView(
    name="user_stats",
    entities=[user],
    ttl=timedelta(days=1),  # features older than 1 day are stale
    schema=[
        Field(name="orders_30d", dtype=Int64),
        Field(name="avg_order_value_30d", dtype=Float32),
        Field(name="unique_restaurants_30d", dtype=Int64),
    ],
    source=user_stats_source,
)

# Training: point-in-time correct feature retrieval
entity_df = pd.DataFrame({
    "user_id": ["u001", "u002", "u003"],
    "event_timestamp": [
        datetime(2024, 1, 14),  # the day before u001 churned
        datetime(2024, 1, 20),
        datetime(2024, 2, 1),
    ]
})

training_df = store.get_historical_features(
    entity_df=entity_df,
    features=[
        "user_stats:orders_30d",
        "user_stats:avg_order_value_30d",
    ],
).to_df()

# Serving: latest features, same definition, no skew
online_features = store.get_online_features(
    features=["user_stats:orders_30d", "user_stats:avg_order_value_30d"],
    entity_rows=[{"user_id": "u001"}],
).to_dict()

Same feature definitions. Same code path for training and serving. Whole point.

Tecton is Feast’s managed commercial cousin, built by Feast’s original creators. It adds streaming feature transformations, built-in monitoring, and orchestration. If Feast is self-assembled kitchen, Tecton is restaurant-grade kitchen that comes pre-installed. Tradeoff: vendor lock-in and cost.

Earlier I oversimplified lakehouse story. Lakehouse is great for analytical queries but terrible for point lookups. Looking up one user’s features in Delta Lake table can take hundreds of milliseconds. Production serving endpoint needs single-digit milliseconds. That is why feature stores keep separate online store backed by Redis or DynamoDB. Access pattern is fundamentally different.

Feature stores solve consistency. They assume data inside is correct and reproducible. If someone overwrites last month’s feature table, need versioning next.

Versioning Data Like Code

Every engineer versions code with Git. But data driving 80% of model performance often lives unversioned in some bucket with filename like training_data_final_v3_FINAL.parquet. I’ve been that engineer. Embarrassing in retrospect.

Core problem: model performance depends on exact training dataset. Six months later, when accuracy drops, you need answer to “did model change, or did data change?” Without versioned data, you cannot reproduce previous result. You fly blind.

DVC (Data Version Control) solves this by wrapping Git. It stores lightweight pointer files — few bytes containing hash — in Git repository, while actual data, which may be gigabytes, lives in remote storage like S3 or GCS. When you git checkout previous commit, pointer file changes, and dvc checkout downloads exact dataset from that commit. You get Git semantics — branches, tags, diffs — for datasets.

# BiteBridge: version a new training dataset
dvc init
dvc remote add -d storage s3://bitebrige-lake/dvc-store

# Track the dataset (DVC computes a hash, stores a pointer file)
dvc add data/training_set_2024q1.parquet
git add data/training_set_2024q1.parquet.dvc data/.gitignore
git commit -m "Training data: Q1 2024, 12k orders, 3 restaurants"
git tag dataset-v1

# Three months later, after adding two new restaurants...
dvc add data/training_set_2024q2.parquet
git add data/training_set_2024q2.parquet.dvc
git commit -m "Training data: Q2 2024, 48k orders, 5 restaurants"
git tag dataset-v2

# Reproduce the old experiment exactly
git checkout dataset-v1
dvc checkout      # pulls the exact Q1 dataset from S3
python train.py   # guaranteed identical to the original run

Pointer file contains hash of data — fingerprint. Change even one byte, hash changes, and DVC knows it is different version. Actual data never enters Git, which would choke on gigabyte files. But data identity is tracked with full Git history.

DVC works well for individual files and experiments. Entire data lake with multiple pipelines writing concurrently is different. lakeFS takes another approach: object storage gets Git-like branching. Create branch, write experimental data, validate, merge to main. Production pipeline never sees half-written data. For BiteBridge, data engineer can test new feature computation on lakeFS branch without risking production feature table.

Versioned data gives reproducibility. Still missing resource that version control cannot conjure: labels.

Getting Labels: From Spreadsheets to Active Learning

Supervised learning needs labels. For BiteBridge’s recommendation model, label is restaurant user actually ordered from. That comes free — it lives in order event. But imagine BiteBridge adds food photo quality detector: is photo appetizing enough for app? Now someone must look at each photo and rate it. Five restaurants with ten photos each? One person finishes in afternoon. Five thousand restaurants with twenty photos each? Need system.

At scale, labeling means distributing tasks to many annotators, measuring agreement (inter-annotator agreement, often Cohen’s kappa), embedding known-answer gold questions to catch random clicking, and versioning labeling guidelines so you know which instructions produced which labels. Labelbox and Scale AI dominate this workflow — Labelbox for teams managing annotators in-house, Scale AI for teams outsourcing operation.

Deeper problem is cost. Labeling is expensive, and not all labels matter equally. If model already classifies 95% of food photos with high confidence, paying annotators to label easy cases is waste. Valuable photos are ambiguous ones — cases where model is uncertain.

Active learning exploits this. Train model on labels you have, then ask: “which unlabeled examples help most?” Simplest strategy is uncertainty sampling — find examples where model’s predicted probability is closest to 50/50 for binary classification, and send those to annotators.

# BiteBridge: active learning for food photo quality
import numpy as np
from sklearn.linear_model import LogisticRegression

def select_photos_for_labeling(labeled_X, labeled_y, unlabeled_X, budget=50):
    """Pick the `budget` most uncertain photos for human review."""
    model = LogisticRegression().fit(labeled_X, labeled_y)
    probs = model.predict_proba(unlabeled_X)

    # Uncertainty: how close to 50/50 is the model's prediction?
    uncertainty = 1 - np.max(probs, axis=1)

    # Return indices of the most uncertain photos
    most_uncertain = np.argsort(uncertainty)[-budget:]
    return most_uncertain

Important trick: active learning has cold-start problem. If initial model is garbage because labels are scarce, it selects garbage examples — “uncertain” for wrong reasons. Fix: bootstrap with about 200 random labels before switching to active selection. Also keep sprinkling random samples so model does not learn only decision boundary and ignore rest of distribution.

Other strategies exist beyond uncertainty sampling. query-by-committee trains multiple models and selects examples where they disagree most. diversity sampling spreads selections across feature space so you do not label hundred near-identical photos. In practice, uncertainty sampling gets about 80% of benefit for 20% of complexity.

Another path exists: let domain experts encode knowledge as rules instead of labeling one example at time. Rule like “if photo is too dark, it’s low quality” is noisy — some predictions will be wrong. But combine enough noisy rules with Snorkel, and labels become surprisingly usable. Snorkel’s label model learns which rules are accurate and which are correlated, producing probabilistic labels without ground-truth examples. You still need small manually labeled validation set to check output, but effort can drop by order of magnitude.

Labeled, versioned data flowing through orchestrated pipelines into feature store — real infrastructure now. Still one hole. How do we know data is correct?

Trusting Your Data: Validation and Quality Gates

Data bugs are silent. Pipeline runs. Model trains. No errors. Metrics look fine. Three months later, someone notices recommendations have been terrible since February, and root cause is upstream schema change that renamed order_total to total_amount. Feature pipeline kept running — it filled nulls. Model trained happily on dataset where every order looked like zero value. Nothing technically failed. Nobody got alerted.

I’m still not over this one. It happened to me. Fix took thirty minutes. Three months of degraded predictions were gone forever.

Data validation is not optional. It is quality gate — checkpoint in pipeline that halts everything if data misses explicit expectations.

Great Expectations is most popular validation framework. You write “expectations” in code — statements about what data should look like — and tool checks them on every pipeline run. Think unit tests, but for data.

# BiteBridge: data quality gate before feature computation
import great_expectations as gx

context = gx.get_context()

# Define what "correct" looks like for our order data
suite = context.add_expectation_suite("order_data_quality")
suite.add_expectation(
    gx.expectations.ExpectColumnValuesToNotBeNull(column="user_id")
)
suite.add_expectation(
    gx.expectations.ExpectColumnValuesToNotBeNull(column="order_total")
)
suite.add_expectation(
    gx.expectations.ExpectColumnValuesToBeBetween(
        column="order_total", min_value=0.01, max_value=500.0
    )
)
suite.add_expectation(
    gx.expectations.ExpectTableRowCountToBeGreaterThan(value=100)
)
# If any expectation fails, the pipeline halts. No silent corruption.

Google’s TensorFlow Data Validation (TFDV) takes complementary approach. Instead of writing rules manually, it auto-generates schema from training data, then flags anomalies when new data deviates. Column that used to have 3 categories now has 47? Flagged. Mean of feature shifted by three standard deviations? Flagged. TFDV is especially strong at catching distribution drift — slow silent changes that gradually poison model.

# TFDV: auto-detect schema anomalies
import tensorflow_data_validation as tfdv

# Generate baseline from known-good training data
train_stats = tfdv.generate_statistics_from_csv("data/orders_train.csv")
schema = tfdv.infer_schema(train_stats)

# Validate new batch against baseline
new_stats = tfdv.generate_statistics_from_csv("data/orders_2024_02.csv")
anomalies = tfdv.validate_statistics(new_stats, schema)
tfdv.display_anomalies(anomalies)
# Output: "restaurant_id: Unexpected string values ['NEW_REST_006', 'NEW_REST_007']"

Great Expectations and TFDV are not either/or. Great Expectations is better for explicit business rules (“order_total must be positive”). TFDV is better for detecting statistical drift (“distribution of this feature changed”). Mature pipeline runs both.

Minimum validation for any ML pipeline: schema integrity (column names and types unchanged), completeness (null rates within bounds), volume (row counts in expected range — catches silent upstream outages), distribution stability (feature statistics have not drifted hard), freshness (most recent timestamp is actually recent), and uniqueness (primary keys are unique). If pipeline skips these, you do not have data quality. You have hope.

Now data can be stored, moved, validated, versioned. Features stay consistent between training and serving. Labels arrive efficiently. Operational side is covered. But BiteBridge now has recommendation team, fraud team, search team — and all ask same thing: “does data I need already exist somewhere?”

Finding Your Data: Data Catalogs

At five-person company, you find data by asking engineer who built pipeline. At fifty, you ask around on Slack. At five hundred, nobody knows where anything is, teams build redundant pipelines because they cannot find existing data, and half undocumented warehouse tables are abandoned experiments nobody dares delete.

Data catalog is searchable inventory of every dataset, table, feature, and model in organization. It stores metadata: columns, owner, last update time, producing pipelines, consuming models. Think library catalog, except books are datasets and catalog also knows which books were derived from which others. That is data lineage.

For BiteBridge, catalog means fraud team can search “user order count” and discover recommendation team already computes exact feature. No duplicate pipeline. No diverging definitions. Major open-source catalogs are DataHub (from LinkedIn, strong on lineage and ML metadata), OpenMetadata (newer, strong governance), and Amundsen (from Lyft, focused on search and discovery). All track datasets, tables, features, and models as first-class entities, with lineage graphs showing flow from source to consumption.

Data catalogs feel like least exciting part of data infrastructure. They do not make models faster or more accurate. But once organization passes about 50 data practitioners, cost of not knowing what data exists exceeds cost of maintaining catalog. Infrastructure you regret not having when you finally need it.

Catalog helps teams find and share data. But some data cannot be shared freely. BiteBridge knows where users live, what they eat, how often they order. Powerful for recommendations. Deeply personal. Final infrastructure piece is protecting it.

Protecting Your Data: Privacy

BiteBridge’s recommendation model needs to learn from user behavior. But user behavior is private. “This person orders from vegan restaurant every Friday” reveals dietary preferences. “This person ordered at 3 AM from restaurant near hospital” reveals more than anyone consented to share. Training model on private data risks leaking that information, either through model outputs or through training data itself being accessed by someone who should not see it.

Two complementary techniques help: differential privacy protects data during computation. federated learning protects data during collection.

Differential privacy adds carefully calibrated random noise to computations so output looks almost same whether any one individual’s data is included or not. Intuition: survey people about embarrassing question. Instead of asking directly, each person flips private coin. Heads: answer truthfully. Tails: answer randomly. Any individual response is meaningless — maybe truth, maybe tails. In aggregate, with enough responses, true statistics still emerge from noise.

Formal guarantee is controlled by epsilon (ε), privacy budget. Smaller epsilon means more noise and stronger privacy. Larger epsilon means less noise and more accurate results. This is fundamental tradeoff. Practical epsilon choices are still hard. Apple uses ε values around 1-4 for on-device collection. U.S. Census Bureau used ε around 19 for 2020 census, and many privacy researchers argued that was too high.

# Differential privacy: adding noise to a simple aggregate
import numpy as np

def private_mean(values, epsilon=1.0):
    """Compute mean with differential privacy guarantee."""
    true_mean = np.mean(values)
    sensitivity = (max(values) - min(values)) / len(values)
    # Laplace noise scaled by sensitivity / epsilon
    noise = np.random.laplace(0, sensitivity / epsilon)
    return true_mean + noise

# BiteBridge: private average order value
order_values = [12.50, 18.00, 9.75, 22.00, 15.50]
private_avg = private_mean(order_values, epsilon=1.0)
# True mean: 15.55, Private mean: ~15.55 ± some noise
# Any single user's order value is protected

Each query against data spends part of privacy budget. Once budget is exhausted, you cannot safely release more information. Differential privacy forces you to think about total set of analyses, not isolated queries.

Federated learning takes different route: data never leaves user’s device. Instead of sending BiteBridge users’ order histories to central server, each user’s phone trains local model on its own data and sends only model updates (gradients or weight deltas) to server. Server aggregates updates from many users to improve global model, but never sees any individual’s raw data. Google pioneered this for Gboard — model learns from millions of users’ typing patterns without Google seeing what anyone typed.

Federated learning plus differential privacy is powerful: data stays on-device (federated), and even model updates are noised before leaving device (differential privacy). Double protection. Gold standard for privacy-sensitive ML, used in production by Apple, Google, and Meta.

My favorite thing about differential privacy: aside from formal epsilon definition, nobody is fully certain how to choose “right” epsilon for specific application. Research is still active. Practical guidance is still evolving. That uncertainty is honest — balancing utility and privacy is genuinely hard.
