Ignored ML infrastructure too long. GPU types, cloud pricing tiers, cluster orchestration came up, head nodded, follow-up feared. Model felt like job. Metal felt like somebody else's problem.

Then four A100 GPUs ran through long weekend. Bill hit $2,300 for fine-tuning job finished Friday afternoon. Pain got sharp. Time to learn what sits underneath.

ML infrastructure = hardware, cloud services, orchestration tools that turn model.fit() into real run. Goal: fast enough, cheap enough, reliable enough. GPUs took over deep learning around 2012. Cloud ML platforms showed up around 2017–2018. LLM boom in 2022–2023 sped everything up. Good setup vs bad setup can mean 10× cost gap for same work.

Why Hardware Matters at All

Section uses one running example. Build PetSnap — mobile app that identifies dog breeds from photos. ResNet-50 works on laptop. Accuracy good. Latency fine. Team gets excited.

Then question lands: "Can we serve this to 10,000 users?" Everything changes.

Laptop GPU has 8 GB memory. Model needs 100 milliseconds per image. At 10,000 requests per hour, need faster GPU or many GPUs in parallel. If team wants larger vision-language model, model may not fit in 8 GB at all.

Core tension: model is science. Hardware is physics. Elegant architecture still must live on silicon with fixed memory, fixed arithmetic units, fixed data flow rate. Ignore physical limits, notebook success turns into production failure.

Think kitchen. Training model = cooking meal. GPU = stove. Stove size limits how many pots run at once. Burner heat sets how fast pots boil. Counter space — VRAM — limits how many ingredients stay out. Best chef still fails if kitchen too small for recipe.

The GPU as a Kitchen: VRAM, Compute, and Bandwidth

GPU has three numbers that matter for ML. Learn what each number does and which one bottlenecks first. That intuition pays off fast.

VRAM (Video RAM, also called GPU memory) = counter space. Model weights, input data, intermediate activations, gradients all must fit at same time. If model plus working data exceeds VRAM, job crashes. No graceful fallback. Out-of-memory error. Training run dies.

For PetSnap ResNet-50, model weights in FP32 (32-bit floating point, 4 bytes per parameter) take about 100 million parameters × 4 bytes = 400 MB. Tiny. An 8 GB GPU handles that easily. But 7-billion parameter LLM in FP16 (2 bytes per parameter) needs 14 GB for weights alone. Add gradients, optimizer states, activations during training, and total reaches 80–100 GB. No single consumer GPU fits that.

Compute (measured in TFLOPS — trillions of floating-point operations per second) = burner heat. More TFLOPS means more matrix multiplications per second. Result: faster training, faster inference. Modern GPUs hit top TFLOPS with Tensor Cores, circuits built for matrix multiply-accumulate operations that dominate deep learning.

Memory bandwidth = speed from fridge to counter. Measured in GB/s. It controls how fast data moves between VRAM and compute units. Easy mistake: assume TFLOPS is everything. For LLM inference, model spends most time reading weights from memory, not doing arithmetic. Bottleneck becomes bandwidth, not compute. Name: memory-bandwidth-bound. Result: two GPUs with similar TFLOPS can have very different inference speeds if bandwidth differs.

Punchline: training is usually compute-bound. Need maximum TFLOPS to chew through billions of tokens. Inference, especially for large language models, is usually bandwidth-bound. Need fast memory reads to stream weights into compute units. Pick hardware without knowing bottleneck, team pays H100 prices for work cheaper GPU could do at same speed.

The GPU Landscape: T4 Through H100 and Beyond

Make this concrete. These are GPUs you actually see in cloud ML today, ordered from cheapest to most expensive. PetSnap example stays with us.

NVIDIA T4 = budget workhorse. Specs: 16 GB VRAM, about 65 TFLOPS at FP16, about $0.50–1.00 per hour in cloud. Built on Turing architecture from 2018. For PetSnap ResNet-50 inference, T4 is enough. Model fits easily. Compute handles hundreds of predictions per second. Weak spot: models above about 3 billion parameters and serious training jobs.

NVIDIA L4 = T4 successor, built on Ada Lovelace architecture (2023). Specs: 24 GB VRAM, about 121 FP16 TFLOPS, fourth-generation Tensor Cores, about $0.80–1.20 per hour. For PetSnap with 3–7B vision-language model using INT8 quantization, L4 is sweet spot. Good inference GPU for most production workloads without massive models.

NVIDIA A100 changed ML infrastructure in 2020. Specs: 80 GB HBM2e memory, 312 TF32 TFLOPS, 2,039 GB/s memory bandwidth, about $3–5 per hour in cloud. It became training workhorse. For PetSnap, A100 is overkill for inference. For fine-tuning 7B language model on pet breed descriptions, one A100 can handle QLoRA. Two A100s can handle full fine-tuning up to about 13B parameters.

NVIDIA H100 (Hopper architecture, 2022) = current frontier. Same 80 GB VRAM as A100, but 3× memory bandwidth (3,350 GB/s via HBM3), nearly 3× compute (989 TF32 TFLOPS), native FP8 support. Cloud pricing runs about $8–12 per hour. Benchmarks show H100 training same task 3–4× faster than A100. Finish sooner, and cost per training run can end up lower despite higher hourly rate. Math works.

Beyond NVIDIA, know two more players. Google TPUs (Tensor Processing Units) are custom silicon for matrix multiplications. TPU v5e offers 16 GB HBM at about $1.20–2.00 per hour. Good pricing, but locked into Google's JAX/TensorFlow ecosystem and XLA compilation. TPU v5p, with 95 GB HBM and 459 TFLOPS, competes with A100 for large GCP training runs. AWS custom chips — Trainium2 for training and Inferentia2 for inference — can offer good performance at lower prices, but require Amazon's NeuronSDK, which means rewriting parts of training code.

Rule of thumb: if team is already deep in GCP with JAX, TPUs make sense. If team is already on AWS and running standard PyTorch, stay with NVIDIA GPUs unless cost pressure forces exploration of alternatives. Lock-in cost matters.

The VRAM Calculation You'll Do a Hundred Times

Every infrastructure decision starts with one question: how much GPU memory do I need? Learn arithmetic once. GPU choices get much easier.

Back to PetSnap. Upgrade from ResNet-50 (100M parameters) to 7-billion parameter vision-language model so users can ask pet questions in natural language. How much VRAM needed?

For inference, main cost is model weights. Each parameter uses bytes based on numerical precision. FP32 uses 4 bytes per parameter. FP16 and BF16 use 2 bytes. INT8 uses 1 byte. INT4 uses half byte. So 7B model in FP16 takes 14 GB. In INT4, 3.5 GB. Then add KV cache overhead — memory for previous tokens during text generation — usually 20–30% more. Result: FP16 inference needs about 18 GB. INT4 inference needs about 4.5 GB.

Meaning: INT4 inference fits on L4 (24 GB). FP16 inference needs A100 (80 GB) or at least GPU with 24+ GB if setup is careful. Hardware choice gets concrete: L4 for quantized serving. A100 for full precision.

For training, story gets harsher. Need memory for model weights, gradients, optimizer states. Adam — most common optimizer — stores two extra FP32 copies of every parameter (first and second moment estimates). So 7B model training in BF16 needs about 14 GB (weights in BF16) + 14 GB (gradients in BF16) + 56 GB (optimizer states in FP32, 7B × 4 bytes × 2 copies). That is about 84 GB before activations. Add activation memory — highly variable with batch size and sequence length, often 20–50 GB — and total goes well above 100 GB. No single GPU handles that. Need multiple GPUs together.

def estimate_vram_gb(params_billions, precision="fp16", mode="inference"):
    """The back-of-envelope VRAM calculation every ML engineer needs."""
    bytes_per_param = {"fp32": 4, "fp16": 2, "bf16": 2, "int8": 1, "int4": 0.5}
    bp = bytes_per_param[precision]
    model_mem = params_billions * 1e9 * bp / (1024**3)

    if mode == "inference":
        # Model weights + KV cache overhead (~20-30%)
        return model_mem * 1.3
    else:
        # Weights + gradients + Adam optimizer (2 × FP32 copies) + activations
        optimizer_mem = params_billions * 1e9 * 4 * 2 / (1024**3)
        grad_mem = model_mem
        activation_mem = model_mem  # rough lower bound
        return model_mem + optimizer_mem + grad_mem + activation_mem

# PetSnap's 7B vision-language model:
# INT4 inference: ~4.5 GB  → fits on an L4 (24 GB) with room to spare
# BF16 training:  ~105 GB  → needs multiple GPUs (2× A100 80GB, or 1 H100 with FSDP)

Important asymmetry: inference memory scales linearly with model size and precision. Predictable. Easy to plan. Training memory grows faster because optimizer-states term gets added. That is why teams often serve model on cheaper hardware than hardware used to train model.

When One GPU Isn't Enough: Scaling Up and Scaling Out

PetSnap 7B model needs ~105 GB for training. A100 has 80 GB. Model does not fit. Now what?

Two directions exist. Trade-offs differ a lot. Think bigger kitchen versus second kitchen across town.

Scaling up (also called vertical scaling) means adding more GPUs inside same machine. Server with 8× A100 GPUs connected by NVLink — high-speed interconnect moving data between GPUs at 600 GB/s on A100s and 900 GB/s on H100s — acts like one giant kitchen with many stoves sharing one counter. Data moves between GPUs almost as fast as inside one GPU. Usually first move for most teams. PetSnap training job needing 105 GB across two A100s on one node works well. NVLink keeps overhead low.

Scaling out (also called horizontal scaling) means adding more machines. Go from one 8×A100 node to four nodes. Now kitchens sit in different buildings. Data travels between machines over InfiniBand (50–100 GB/s) or, in cheaper setups, plain Ethernet (~12.5 GB/s for 100 Gigabit). That is 10–70× slower than NVLink. Every time GPUs on different machines exchange information — especially gradient updates after every batch — they pay communication tax.

Hierarchy matters: NVLink within node at 900 GB/s, InfiniBand between nodes at 50–100 GB/s, Ethernet at 12.5 GB/s. If NVLink transfer takes one second for some data size, same transfer takes about 10 seconds over InfiniBand and 70 seconds over Ethernet. Not minor detail. This decides which parallelism strategies are feasible.

For inference scaling, life is easier. Each model replica serves requests independently, so machines can be added without communication between replicas. Put load balancer in front. Add replicas as traffic grows. Hard part is auto-scaling: GPUs cost money when idle and cold-start slowly — loading large model takes 30–60 seconds. For PetSnap, pre-warm some replicas for expected spikes instead of scaling from zero.

Multi-GPU Strategies: Data, Tensor, and Pipeline Parallelism

When training needs multiple GPUs, next question is simple: how split work? Three strategies exist. Trade-offs differ. Concrete example makes them stick.

Assume PetSnap's 7B model has 32 transformer layers, and we have 4 GPUs on one machine.

Data parallelism is most intuitive. Every GPU gets full copy of model. Training data splits into 4 chunks — GPU 0 handles batch chunk 0, GPU 1 handles batch chunk 1, and so on. After each forward and backward pass, GPUs exchange gradient updates with AllReduce so every copy stays synchronized. This is what PyTorch's DistributedDataParallel (DDP) does. Easy to set up. Works when model fits on one GPU. But PetSnap's 7B model needs 105 GB for training. It does not fit on one A100. Data parallelism alone cannot help.

Then Fully Sharded Data Parallelism (FSDP) enters. Instead of replicating whole model on every GPU, FSDP shards parameters, gradients, optimizer states across GPUs. Each GPU holds only 1/N of model memory footprint, where N is number of GPUs. When layer needs full parameters for forward or backward pass, GPUs gather relevant shard, compute, then release it. Our 105 GB training job across 4 GPUs becomes about 26 GB per GPU. Each A100 handles that comfortably.

from torch.distributed.fsdp import FullyShardedDataParallel as FSDP
from torch.distributed.fsdp import MixedPrecision, ShardingStrategy

# Wrap an existing model with FSDP — shards across all GPUs
model = FSDP(
    model,
    sharding_strategy=ShardingStrategy.FULL_SHARD,
    mixed_precision=MixedPrecision(param_dtype=torch.bfloat16),
)
# 105 GB model → ~26 GB per GPU across 4 A100s
# The trade-off: more communication between GPUs, but the model fits

Tensor parallelism slices differently. It does not split data or full memory footprint. It splits individual layers. Imagine PetSnap transformer layer with weight matrix shape [4096, 16384]. Tensor parallelism splits matrix into 4 columns — GPU 0 handles columns 0–4095, GPU 1 handles 4096–8191, and so on. Each GPU computes part of layer output, then results get combined. NVIDIA's Megatron-LM does this. Catch: GPUs must exchange partial results inside every layer. Communication must be extremely fast. NVLink works. InfiniBand does not. So tensor parallelism is viable only inside one node.

Pipeline parallelism splits model by layers. GPU 0 gets layers 0–7, GPU 1 gets layers 8–15, GPU 2 gets layers 16–23, GPU 3 gets layers 24–31. Data moves through GPU 0, then GPU 1, then GPU 2, then GPU 3 — assembly line style. Communication happens only between adjacent stages, and transfer size is smaller than in tensor parallelism. That tolerates higher-latency interconnects, so it works for multi-node training. Downside: pipeline bubbles. While GPU 0 processes micro-batch 2, GPU 3 may wait for micro-batch 1 to finish earlier stages. Idle time appears unless micro-batches are interleaved carefully.

In practice, large-scale training combines all three. 3D parallelism means data parallelism across nodes, tensor parallelism inside node, pipeline parallelism across groups of nodes. Frontier labs use this for models with hundreds of billions of parameters. DeepSpeed and Megatron-LM provide machinery. For PetSnap's 7B model, FSDP across 4 GPUs on one node is enough. 3D approach becomes necessary around 70B+ scale.

Rule of thumb: doubling GPUs rarely halves training time. Communication overhead eats gains. 75% scaling efficiency — 8 GPUs delivering 6× throughput of 1 GPU — counts as good. Below 60%, money burns on communication. Benchmark before committing to larger clusters.

The Three Cloud Kingdoms: SageMaker, Vertex AI, and Azure ML

Say PetSnap has grown. We train models regularly, serve millions of users, and choose cloud instead of buying hardware. Three platforms dominate: AWS SageMaker, GCP Vertex AI, Azure ML. Key question is not "best platform." Key question is: where does data already live?

AWS SageMaker is kitchen-sink option. It has service for almost everything — training jobs, real-time endpoints, batch inference, feature stores, pipelines, experiment tracking, AutoML (Autopilot), bias detection (Clarify). For PetSnap on AWS, training data goes to S3, training job gets defined with instance type and count, and SageMaker provisions GPU cluster, runs job, then tears cluster down. Billing covers training time only. No idle GPUs.

from sagemaker.pytorch import PyTorch

estimator = PyTorch(
    entry_point="train.py",
    instance_count=2,
    instance_type="ml.g5.2xlarge",  # 1× A10G per node, 2 nodes total
    distribution={"torch_distributed": {"enabled": True}},
    hyperparameters={"epochs": 10, "lr": 1e-4},
)
estimator.fit({"train": "s3://petsnap-data/images/"})
# Cluster spins up → trains → saves model to S3 → tears down

Trade-off: SageMaker instances cost 20–40% more than raw EC2 GPU instances. Premium buys cluster management, auto-teardown, integration with rest of AWS ML stack. Teams that want focus on models often accept premium. Teams with strong DevOps skills may save money with raw EC2 plus custom orchestration.

GCP Vertex AI has two real differentiators. First: TPU access — Google's custom AI chips exist nowhere else. For teams using JAX, TPUs can be 30–50% cheaper per training-FLOP than equivalent NVIDIA GPUs. Second: Model Garden offers one-click fine-tuning and deployment for foundation models like Gemini, Llama, Mistral. Vertex AI Pipelines run on Kubeflow under hood, which is open-source and somewhat portable across clouds. Feature Store is backed by BigQuery, natural fit when data warehouse already lives there.

Azure ML wins on enterprise governance. Responsible AI Dashboard bundles fairness assessment, interpretability, error analysis, counterfactual explanations in one interface. It is most comprehensive of three. Azure OpenAI Service gives enterprise-grade GPT-4o access with data residency guarantees, VNet isolation, content filtering that regulated industries like healthcare and finance require. If PetSnap were veterinary AI company under data regulations, Azure would be strong choice for that reason alone.

Honest assessment: all three platforms can do almost everything. Feature checklists are converging. Real decider is data gravity — ML infrastructure orbits data. If PetSnap images already sit in S3, use SageMaker. Moving terabytes to another cloud costs money ($0.09 per GB of egress) and time. If analytics pipeline runs on BigQuery, use Vertex AI. If company is Microsoft shop with Active Directory and Power BI, use Azure ML. That is decision.

Spot Instances: 70% Savings with a Catch

Cloud GPU instances at on-demand prices are expensive. PetSnap training on two A10G instances for 20 hours costs about $60. Fine. But training 70B model on 8× H100 nodes for one week costs more than $13,000. At that scale, cost optimization stops being optional.

Spot instances (AWS term) or preemptible VMs (GCP term) are cloud GPUs sold at 60–90% discount because provider uses spare capacity. Catch: cloud can reclaim them with as little as 30 seconds notice. Training job dies mid-step. If progress is not saved, everything since last checkpoint is gone.

Lesson is simple. Run training without checkpointing once, get burned once, never repeat. Spot requires checkpoint-and-resume pattern.

Pattern: save checkpoints to durable storage (S3, GCS) every N steps. When job restarts — because spot reclaimed instance or any other interruption happened — load latest checkpoint and continue. Loss is capped at N steps of work.

import torch, os

CHECKPOINT_DIR = "s3://petsnap-checkpoints/"  # Durable storage, survives instance death

def save_checkpoint(model, optimizer, epoch, step, path):
    torch.save({
        "model": model.state_dict(),
        "optimizer": optimizer.state_dict(),
        "epoch": epoch,
        "step": step,
    }, path)

def resume_from_checkpoint(model, optimizer, checkpoint_dir):
    checkpoints = sorted(list_s3_objects(checkpoint_dir))
    if not checkpoints:
        return 0, 0  # Fresh start
    latest = torch.load(download(checkpoints[-1]))
    model.load_state_dict(latest["model"])
    optimizer.load_state_dict(latest["optimizer"])
    return latest["epoch"], latest["step"]

# Every 500 steps, save. If spot reclaims, we lose at most 500 steps.
# At 60-90% discount, the savings easily justify the occasional re-work.

For PetSnap training, spot instances with checkpointing every 500 steps cut compute bill by 60–70%. Job may get interrupted 2–3 times, losing maybe 20 minutes total across 20-hour run. Math strongly favors spot.

For inference, spot is riskier. Reclaimed instance means dropped requests — users stare at loading screen. Use spot for batch inference, like processing image queue overnight, where latency does not matter. Keep on-demand or reserved instances for real-time serving. Reserved instances (AWS) and committed use discounts (GCP) lock you into 1–3 year contracts at 30–60% savings over on-demand. Rule: commit only for baseline load — minimum compute always needed. Handle spikes with on-demand. Handle experiments with spot.

The On-Prem vs. Cloud Decision

Eventually every serious ML team asks one question: buy our own GPUs or not?

Arithmetic is more nuanced than either side admits. Cloud side says flexibility wins. On-prem side says hardware purchase saved 60%. Both can be right. Numbers decide.

Suppose PetSnap now runs GPU workloads 18 hours a day, 7 days a week. That is about 6,500 GPU-hours per year per GPU. In cloud, one A100 at $4/hour costs about $26,000/year. Buying A100-based server — around $150,000 for 8-GPU system, about $19,000 per GPU slot after amortization — plus $2,000/year for electricity and cooling gives about $21,000/year for first three years, then about $2,000/year for remaining life. Over three years, cloud costs about $78,000 per GPU. On-prem costs about $25,000 per GPU. Savings are real.

But hidden costs matter. On-prem means owning capacity planning — if PetSnap traffic doubles overnight, new GPUs do not appear in minutes. It means owning maintenance — when GPU fails, and at scale they do, spare hardware and skilled people are needed. It means owning cooling — room full of H100s throws huge heat. It means owning depreciation risk — $150,000 server bought today may be beaten by hardware costing half as much in 18 months.

Break-even point from most analyses: if GPU capacity stays at 40–50% utilization or higher for 3+ years, on-prem becomes cheaper. Below that, or with unpredictable workloads, cloud wins on flexibility.

Pattern that works for many teams: on-prem for steady-state — inference fleet running 24/7 — and cloud for burst — large training runs, experiments, traffic spikes. PetSnap might own 4 GPUs for always-on inference and rent cloud GPUs for monthly retraining. This captures benefits of both without full commitment to either.

Serverless Inference: When You Don't Want GPUs at All

Question keeps coming up: what if team does not want to manage any of this?

Serverless ML inference means team deploys model and platform handles rest — provisioning GPUs when requests arrive, scaling to zero when traffic disappears, billing by millisecond of actual compute. No GPU instance to manage. No auto-scaler to configure. Model wakes when called and sleeps otherwise.

For early PetSnap — sporadic traffic, unpredictable usage, three-person team that wants model work instead of infrastructure work — serverless is compelling. AWS SageMaker Serverless Inference, Google Cloud Run with GPU support, and platforms like Modal and Replicate all offer this model.

Trade-off is cold start latency. If requests stop for a while, platform deallocates GPU. Next request triggers reallocation and model loading — from about 5 seconds for small model to 60+ seconds for large one. For PetSnap, user uploading photo and waiting 30 seconds for breed prediction gets bad experience. Serverless works when cold starts are acceptable — batch processing, internal tools, low-traffic APIs. It falls apart when consistent low latency matters.

Other limit: serverless platforms give less control over GPU type, quantization settings, batching strategy, memory configuration. If serving setup must squeeze every ounce of performance, serverless gets outgrown and dedicated instances with custom configs become necessary.

Good rule: use serverless for prototypes and internal tools. For production-facing inference with latency requirements, dedicated GPU instances plus proper auto-scaler are more reliable.

Cluster Management: Kubernetes, Ray, or Managed Platforms

As PetSnap grows — multiple models, multiple teams, mix of training and inference workloads — somebody must decide how GPU resources get shared and orchestrated. This is cluster management question. Three main answers exist.

Managed platforms (SageMaker, Vertex AI, Azure ML) are lowest-resistance path. Team does not manage cluster at all. Jobs get submitted. Endpoints get deployed. Platform handles scheduling, scaling, GPU allocation. For PetSnap with team of 5–15 ML engineers, this is right starting point. Running own cluster is not worth overhead until pain shows up: auto-scaling mismatches traffic, container constraints block model architecture, or team needs fine-grained GPU scheduling priorities.

Kubernetes (K8s) is industry standard for running own cluster. It lets team request GPUs like CPU or memory, schedule jobs across machine pool, enforce resource quotas per team, and auto-scale serving deployments from metrics like queue depth or latency. Kubeflow adds ML-specific layers: pipeline orchestration, hyperparameter tuning via Katib, model serving via KServe. Investment is real. Installing, configuring, maintaining Kubeflow requires dedicated platform engineers. Best fit: organizations that already know Kubernetes and need multi-cloud portability.

Ray takes different path. It does not build on Kubernetes abstractions first. Ray provides its own distributed runtime where team writes Python and Ray handles cluster distribution. Ray Train orchestrates distributed training. Ray Serve handles serving with request batching and multi-model composition. Ray Data manages distributed preprocessing. Ray can run on Kubernetes via KubeRay operator or on bare VMs. Strength is composability. If PetSnap needs pipeline where breed classifier feeds health assessment model which feeds recommendation engine, Ray Serve wires pieces together in Python instead of YAML.

from ray import serve

@serve.deployment(num_replicas=2, ray_actor_options={"num_gpus": 1})
class BreedClassifier:
    def __init__(self):
        self.model = load_model("petsnap-breed-v3")

    async def __call__(self, request):
        image = (await request.json())["image"]
        return self.model.predict(image)

# Each replica gets its own GPU. Ray handles load balancing and scaling.
app = BreedClassifier.bind()
serve.run(app, host="0.0.0.0", port=8000)

Honest recommendation: start with managed platforms. Move to Ray when team needs multi-model composition or has Python-heavy engineers who do not want Kubernetes. Move to Kubeflow/KServe when dedicated platform engineers exist and Kubernetes-native infrastructure must span multiple clouds or on-prem clusters.
