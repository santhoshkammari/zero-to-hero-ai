Seq2Seq & the Birth of Attention

Attention = single most important idea between basic RNNs and Transformer. Started as fix for one specific bottleneck. Understand bottleneck + fix → Transformer design feels inevitable, not magical.

Sequence-to-sequence (seq2seq) introduced 2014 by Sutskever, Vinyals, Le + parallel work by Cho et al. Workhorse for machine translation, summarization, chatbots — any task where input and output are variable-length sequences. Attention mechanism (Bahdanau, Cho, Bengio, 2015) fixed most crippling flaw of original design, planted seed that grew into Transformer.

A Tiny Translation Problem

Building model that translates short Spanish cooking instructions into English. Starting absurdly small — vocabulary of seven Spanish words and seven English words — so we can trace every moving part by hand.

Training data — three sentences:

"corta las cebollas"   →  "chop the onions"
"mezcla las especias"  →  "mix the spices"
"cocina las cebollas"  →  "cook the onions"

Both sides are sequences, don't have to be same length. Spanish sentence might be three words; English translation might be four. Can't do this with single RNN mapping one input token to one output token per step. Need two stages: one that reads Spanish, one that writes English.

Two-stage setup = heart of everything in this section. Let's build it.

Two RNNs Stitched Together

Idea (published independently by Sutskever et al. and Cho et al., 2014): take two RNNs, connect through single vector.

First RNN = encoder. Reads Spanish input one token at a time — "corta," "las," "cebollas" — updating hidden state after each. Hidden state = encoder's running summary of everything seen so far. After last token, final hidden state captures meaning of entire input sentence. This final vector = context vector.

Second RNN = decoder. Receives context vector as starting state, generates English output one token at a time. Predicts "chop," feeds "chop" back as input, predicts "the," feeds "the" back, predicts "onions," emits end-of-sequence token.

That's it. Two RNNs, one vector between them. Architecture called encoder-decoder or seq2seq (sequence-to-sequence).

import torch
import torch.nn as nn

class Encoder(nn.Module):
    def __init__(self, vocab_size, embed_dim, hidden_dim):
        super().__init__()
        self.embedding = nn.Embedding(vocab_size, embed_dim)
        self.rnn = nn.LSTM(embed_dim, hidden_dim, batch_first=True)

    def forward(self, src):
        embedded = self.embedding(src)
        # all_states: hidden state at every position (we'll need these later)
        # (hidden, cell): the final states — our "context vector"
        all_states, (hidden, cell) = self.rnn(embedded)
        return all_states, hidden, cell

class Decoder(nn.Module):
    def __init__(self, vocab_size, embed_dim, hidden_dim):
        super().__init__()
        self.embedding = nn.Embedding(vocab_size, embed_dim)
        self.rnn = nn.LSTM(embed_dim, hidden_dim, batch_first=True)
        self.fc_out = nn.Linear(hidden_dim, vocab_size)

    def forward(self, token, hidden, cell):
        embedded = self.embedding(token)
        output, (hidden, cell) = self.rnn(embedded, (hidden, cell))
        prediction = self.fc_out(output.squeeze(1))
        return prediction, hidden, cell

Encoder reads "corta las cebollas," compresses into vector. Decoder takes vector, produces "chop the onions" one word at a time.

Trace through: encoder processes "corta" → hidden state holds representation of "corta." After "las" → state blends in "las." After "cebollas" → final hidden state (vector of 256 numbers) supposed to represent "chop the onions." Decoder starts from that vector, predicts next English word at each step.

Elegant. Deeply flawed.

The Sticky-Note Bottleneck

Flaw: context vector is fixed-size. Maybe 256 dimensions. Maybe 1024. Doesn't matter — it's fixed.

For three-word cooking instructions, fine. 256 dims enough to represent "chop the onions." But 50-word recipe paragraph? All 50 words, relationships, which ingredients match which verbs, step order — crammed into same 256-dim vector. 200-word recipe? Same vector size.

When BLEU scores tanked on long sentences, easy to assume data problem — not enough long training examples. Problem is structural: one vector fundamentally not enough to represent variable-length sequence.

Analogy: someone reads full page of cooking instructions in Spanish, you recite English version. Only thing you carry = one sticky note of fixed size. Short instructions? Scribble key words, fine. Full page? Quantities, timing, specific technique for julienning vs. dicing — gone.

What you want: keep original Spanish text on table, glance back at relevant line while translating each part. Not all at once — just the line that matters for whatever word you're currently producing.

That's attention. But first — how we train decoder, because training has its own plot twist.

Training the Decoder: Teacher Forcing

During inference, decoder feeds own previous prediction as input to next step. Predicts "chop" at step 1, "chop" goes into step 2. Sounds reasonable until you think about training.

Early in training, predictions are garbage. Model predicts "onions" instead of "chop." Step 2 receives "onions" as context, has to predict "the" from that — can't. Step 2 prediction garbage too. Step 3 receives that garbage. Errors cascade like dominoes. Training agonizingly slow.

Fix = teacher forcing. During training, ignore model's own predictions. Feed correct previous token (ground truth) at every step. Model always sees "chop" before predicting "the," always sees "the" before predicting "onions," regardless of own predictions. Like teacher correcting each step before student moves on. Convergence dramatically faster.

But teacher forcing creates subtle problem. Training = decoder always sees perfect previous tokens. Inference = sees own imperfect predictions. Model practiced translating in world where it never makes mistakes, then dropped into world where it makes mistakes constantly. This train/inference mismatch = exposure bias. Especially damaging for long outputs where single early error derails entire sequence.

Scheduled sampling (Bengio et al., 2015) starts with 100% teacher forcing — all ground truth. As training progresses, gradually increases probability of feeding model's own predictions back in. By end of training, model practiced recovering from own errors. In code — coin flip at each step:

import random

def train_decoder_step(decoder, encoder_hidden, target_seq,
                       teacher_forcing_ratio=0.5):
    hidden, cell = encoder_hidden
    input_token = target_seq[:, 0:1]   # start with <SOS> token
    outputs = []

    for t in range(1, target_seq.size(1)):
        prediction, hidden, cell = decoder(input_token, hidden, cell)
        outputs.append(prediction)

        # coin flip: ground truth or own prediction?
        if random.random() < teacher_forcing_ratio:
            input_token = target_seq[:, t:t+1]
        else:
            input_token = prediction.argmax(dim=-1, keepdim=True)

    return torch.stack(outputs, dim=1)

Ratio follows schedule — 1.0 at start (pure teacher forcing), decaying toward 0.5 or lower as model improves. Exact schedule tuned by experiment.

Choosing Words at Inference: Beam Search and Friends

At inference, decoder produces probability distribution over entire vocabulary each step. Needs strategy for choosing which token to emit. Same trained model produces wildly different outputs depending on decoding strategy.

Most obvious: greedy decoding — each step pick highest-probability token. Fast but short-sighted. Word that looks best now might lead to dead end two steps later. Model 60% sure next word is "cook," 40% sure it's "prepare." Greedy picks "cook." But "prepare the spiced mixture" might be much better translation — model never explores that path.

Beam search keeps multiple candidates alive. Key parameter = beam width (k). Trace with k=2 on cooking example, starting from <SOS> token.

Step 1: model predicts first word probabilities. "chop" = 0.6, "cook" = 0.3. Keep both — two beams. Step 2: expand each beam to all possible next words. "chop the" = 0.6 × 0.8 = 0.48, "chop a" = 0.6 × 0.1 = 0.06, "cook the" = 0.3 × 0.7 = 0.21. Four candidates total, keep top 2: "chop the" (0.48), "cook the" (0.21). Continue expanding + pruning until both beams produce end token. Highest cumulative score wins.

Beam search dominates when single "right" answer exists (translation). Beam width 4–10 typical. Wider beams = better translations but proportionally more compute, tend to produce repetitive "safe" text.

For diversity (story generation, dialogue, creative writing) — sampling wins. Top-k sampling: pick randomly from k most likely tokens. Nucleus sampling (top-p): pick from smallest set whose cumulative probability exceeds threshold p (typically 0.9–0.95). Adapts candidate set dynamically — confident predictions → narrow set, uncertain → wider pool. Temperature: sharpens/flattens distribution before sampling. Low (0.2–0.5) = more confident. High (1.0+) = more adventurous.

All decoding strategies work with any seq2seq model — with or without attention. They handle what happens after model produces probabilities. But probability quality depends on architecture, and vanilla seq2seq has that sticky-note bottleneck.

Letting the Decoder Peek: Bahdanau Attention

2015: Dzmitry Bahdanau, Kyunghyun Cho, Yoshua Bengio published "Neural Machine Translation by Jointly Learning to Align and Translate." Modest title. Seismic impact.

Idea: instead of forcing decoder to work from one sticky note, let it look at full stack of encoder notes — one per input position — and decide which notes matter right now.

Trace with cooking example. Encoder reads "corta las cebollas," produces hidden state at each position: h₁ for "corta," h₂ for "las," h₃ for "cebollas." Original paper uses bidirectional encoder — reads left-to-right and right-to-left, concatenates two hidden states per position — so each hᵢ captures context from both directions. Three vectors sitting there, waiting to be consulted.

Decoder about to predict first English word. Current hidden state = s₀ (initialized from encoder). Question: which of three encoder states should it attend to?

Scoring formula looks like three random operations stitched together. But logic is clear. Want single number (score) saying "how relevant is encoder state hᵢ to what decoder is doing right now?" Bahdanau's solution — run both vectors through small neural network:

eₜᵢ = vT tanh(W₁ sₜ₋₁ + W₂ hᵢ)

Unpacking: W₁ projects decoder state into shared space. W₂ projects encoder state into same space. Add together — hence additive attention. tanh squashes sum into [-1, 1]. v (learned vector) collapses result into single scalar score. Three operations, each with clear job: project, combine, score.

Compute score for every encoder position → list of scores: e₁, e₂, e₃ for three-word input. Normalize into probability distribution via softmax:

αₜᵢ = exp(eₜᵢ) / (exp(eₜ₁) + exp(eₜ₂) + exp(eₜ₃))

These α values = attention weights. Sum to 1. Tell us: "when generating current output token, pay this fraction of attention to each input position."

When generating "chop," weights might be α₁=0.85 ("corta"), α₂=0.05 ("las"), α₃=0.10 ("cebollas"). Model mostly looking at "corta" (verb) to produce English verb "chop." Makes intuitive sense.

Final step: weighted sum of encoder hidden states.

cₜ = α₁ h₁ + α₂ h₂ + α₃ h₃

This cₜ = context vector for this specific decoder step. Key insight: it's different at every step. When decoder predicts "the," recomputes new scores, gets new weights (maybe more spread — "las" maps to "the"), produces new context vector. When predicting "onions," attention shifts to "cebollas." Each output token gets own custom view of input.

Sticky note gone. Decoder has full Spanish text on table, glances at relevant line for each word.

import torch
import torch.nn as nn
import torch.nn.functional as F

class BahdanauAttention(nn.Module):
    def __init__(self, encoder_dim, decoder_dim, attention_dim):
        super().__init__()
        self.W1 = nn.Linear(decoder_dim, attention_dim, bias=False)
        self.W2 = nn.Linear(encoder_dim, attention_dim, bias=False)
        self.v = nn.Linear(attention_dim, 1, bias=False)

    def forward(self, decoder_state, encoder_outputs):
        # decoder_state:   (batch, decoder_dim)
        # encoder_outputs: (batch, src_len, encoder_dim)

        query = self.W1(decoder_state).unsqueeze(1)   # (batch, 1, attn_dim)
        keys = self.W2(encoder_outputs)               # (batch, src_len, attn_dim)

        # additive scoring: project, combine, score
        scores = self.v(torch.tanh(query + keys))     # (batch, src_len, 1)
        scores = scores.squeeze(-1)                   # (batch, src_len)

        weights = F.softmax(scores, dim=-1)           # (batch, src_len)

        # weighted sum of encoder states
        context = torch.bmm(
            weights.unsqueeze(1), encoder_outputs
        ).squeeze(1)                                  # (batch, enc_dim)

        return context, weights

Important detail: Bahdanau computes attention before decoder RNN step. Decoder state used for scoring = sₜ₋₁ (previous step). Resulting context vector concatenated with current input embedding, fed into decoder RNN to produce sₜ. This ordering means attention influences decoder's hidden state, not other way around.

Limitation: small neural network inside scoring function has learnable parameters (W₁, W₂, v) — flexible but slow. Every decoder step requires forward pass through mini-network for every encoder position. 100-token input + 50-token output = 5,000 forward passes per sentence. People tolerated this a year, but suspected leaner way existed.

A Faster Way to Score: Luong Attention

Later 2015: Minh-Thang Luong, Hieu Pham, Christopher Manning published "Effective Approaches to Attention-based Neural Machine Translation." Question: do we need neural network for relevance scores, or can something cheaper work?

Most popular variant replaces Bahdanau's mini-network with plain dot product:

eₜᵢ = sₜT hᵢ

That's it. Multiply decoder state and encoder state element-wise, sum products. No learned parameters — dot product is free. Called multiplicative attention (scores from multiplying vectors, vs. Bahdanau's additive combination).

Concrete example. Decoder step 2: sₜ = [0.3, -0.5, 0.8, 0.1], encoder state for "cebollas" h₃ = [0.2, -0.4, 0.9, 0.0]. Dot product = (0.3×0.2) + (-0.5×-0.4) + (0.8×0.9) + (0.1×0.0) = 0.06 + 0.20 + 0.72 + 0.0 = 0.98. High score — decoder considers "cebollas" highly relevant. Compare h₂ for "las" scoring 0.15. After softmax, "cebollas" gets most attention weight. Same pattern: score, normalize, weighted sum.

Luong proposed three scoring variants:

Dot variant (sₜᵀ hᵢ): zero extra parameters, fast + memory-efficient. Catch: requires encoder and decoder hidden dims to match — can't dot-product different-sized vectors.

General variant (sₜᵀ W hᵢ): learnable weight matrix W between vectors. Allows different dimensions, adds one knob — expressiveness without full Bahdanau overhead.

Concat variant (vᵀ tanh(W[sₜ; hᵢ])): concatenates states, runs through same architecture as Bahdanau. Equivalent expressiveness, included for completeness.

Second structural difference (easy to miss): Bahdanau computes attention before RNN step using previous state sₜ₋₁. Luong computes after RNN step using current state sₜ. Luong decoder: run RNN → get sₜ → attend over encoder → combine context with sₜ → predict. Both orderings work. Luong approach slightly more common in modern implementations because dot product scoring integrates cleanly with RNN output.

class LuongDotAttention(nn.Module):
    """Simplest Luong variant: dot-product scoring."""
    def forward(self, decoder_output, encoder_outputs):
        # decoder_output:  (batch, hidden_dim)
        # encoder_outputs: (batch, src_len, hidden_dim)

        scores = torch.bmm(
            encoder_outputs,
            decoder_output.unsqueeze(2)
        ).squeeze(2)                        # (batch, src_len)

        weights = F.softmax(scores, dim=-1)
        context = torch.bmm(
            weights.unsqueeze(1),
            encoder_outputs
        ).squeeze(1)                        # (batch, hidden_dim)

        return context, weights

Dot product scoring = direct ancestor of Transformer. Vaswani et al. (2017) took Luong's dot product, added one tweak: dividing by √d_k (square root of key dimension) to prevent scores growing too large in high dimensions. Lineage is direct.

Getting ahead. Step back — name the pattern both Bahdanau and Luong share.

The Three-Step Pattern You'll See Everywhere

Every attention mechanism — Bahdanau, Luong, self-attention, cross-attention, multi-head — does exactly three things:

First: compute score between query and each key. Query = what you're matching. Keys = candidates you're searching. Score = "how relevant is this key to this query?" Bahdanau uses neural network. Luong uses dot product. Transformer uses scaled dot product. Different mechanisms, same purpose.

Second: normalize scores into weights summing to 1 via softmax. Raw scores → probability distribution. "85% interested in position 1, 5% in position 2, 10% in position 3."

Third: compute weighted sum of values using those weights. In Bahdanau/Luong, values = encoder hidden states (same as keys). In Transformer, values get own separate learned projection — more flexibility.

Score. Normalize. Aggregate. Query, key, value. That's entire machinery. Interviewer asks "what is attention?" — that's the answer. Everything else = details about where Q, K, V come from and how score is computed.

In attention built so far: query comes from decoder ("what should I focus on?"), keys and values from encoder ("here's what I have"). Query and key/value from different sequences = cross-attention. In Transformer, queries, keys, values can all come from same sequence.

Mathematical structure = soft hash table lookup — query finds most relevant keys, returns blend of associated values. No one fully certain why this pattern generalizes so powerfully across domains — language, vision, audio, protein structure. It works. Beautifully.

Reading the Alignment Map

Beautiful side effect of attention: weights form matrix you can visualize.

Every decoder output token produces attention weights — one per input position. Stack all weight vectors into rows → 2D grid. Columns = input (Spanish) positions. Rows = output (English) positions. Cell intensity = how much decoder focused on that input word when generating that output word.

For "corta las cebollas" → "chop the onions," alignment matrix looks like:

import numpy as np

# Rows = English output tokens, Columns = Spanish input tokens
#                 corta   las    cebollas
alignment = np.array([
    [0.88,  0.04,  0.08],    # "chop"   — focuses on "corta" (the verb)
    [0.06,  0.82,  0.12],    # "the"    — focuses on "las" (the article)
    [0.03,  0.07,  0.90],    # "onions" — focuses on "cebollas" (the noun)
])

Pattern nearly diagonal — "chop" aligns with "corta," "the" with "las," "onions" with "cebollas." Spanish/English share similar word order here, so alignment is tidy. For different word orders (English → Japanese, verb moves to end) — alignment shifts and crosses. Adjective-noun flips. SOV rearrangements. All visible in matrix.

Not decorative — genuine debugging tool. Model failing on long sentences? Plot attention. Weights collapsing toward uniform (equal attention everywhere) = model gave up on aligning. Decoder fixating on one position for many output steps = stuck in repetition loop, generating same word because it can't look away from one source token.

Quick heuristic: sharp attention (most weight on one or two source positions per output step) = model learned meaningful alignment. Fuzzy, spread-out attention = it hasn't.

The Full Picture: Seq2Seq with Attention

Wiring everything together. Complete architecture: encoder storing hidden state at every input position, Bahdanau attention computing fresh context vector each decoder step, decoder using both context and own state to predict each output token.

class Seq2SeqWithAttention(nn.Module):
    def __init__(self, src_vocab, tgt_vocab, embed_dim,
                 hidden_dim, attn_dim):
        super().__init__()
        self.encoder = Encoder(src_vocab, embed_dim, hidden_dim)
        self.attention = BahdanauAttention(
            hidden_dim, hidden_dim, attn_dim
        )
        self.dec_embedding = nn.Embedding(tgt_vocab, embed_dim)
        self.dec_rnn = nn.LSTMCell(embed_dim + hidden_dim, hidden_dim)
        self.output_proj = nn.Linear(hidden_dim * 2, tgt_vocab)

    def forward(self, src, tgt, teacher_forcing_ratio=0.5):
        batch_size, tgt_len = tgt.size()
        enc_outputs, hidden, cell = self.encoder(src)
        h = hidden.squeeze(0)
        c = cell.squeeze(0)

        outputs = []
        input_token = tgt[:, 0]

        for t in range(1, tgt_len):
            embedded = self.dec_embedding(input_token)

            # attention: decoder asks "what should I look at?"
            context, _ = self.attention(h, enc_outputs)

            # feed both the embedded token and context into decoder
            rnn_input = torch.cat([embedded, context], dim=1)
            h, c = self.dec_rnn(rnn_input, (h, c))

            # predict from decoder state + context
            prediction = self.output_proj(
                torch.cat([h, context], dim=1)
            )
            outputs.append(prediction)

            # teacher forcing decision
            if random.random() < teacher_forcing_ratio:
                input_token = tgt[:, t]
            else:
                input_token = prediction.argmax(dim=-1)

        return torch.stack(outputs, dim=1)

Difference from vanilla seq2seq concentrated in one line: attention call inside loop. Instead of starting from one compressed vector and hoping, decoder consults full encoder states every step. Difference between memorizing recipe and cooking with recipe open on counter.

On WMT translation benchmarks, this improved BLEU scores dramatically — especially for sentences longer than 20 tokens, exactly where vanilla seq2seq collapsed. Bottleneck gone.

From Cross-Attention to Self-Attention

What attention gave us: decoder no longer relies on compressed sticky note. Has direct access to encoder's full memory through dynamic, learned lookup. Recipe stays open on counter.

Question that unlocks next revolution: what if we applied this same lookup not between two separate sequences, but within single sequence?

So far: query from decoder, keys/values from encoder. Two different sequences. But nothing about score-normalize-aggregate requires that. Query, keys, values can all come from same sequence.

Consider: "The chef seasoned the steak before it was served." What does "it" refer to? Human knows "it" = "steak," not "chef." Let every word attend to every other word in same sentence → representation of "it" incorporates info from "steak" regardless of distance. No recurrence needed. No hidden state threading info step by step. Direct connection.

That's self-attention. Each position attends to all positions, including itself. Query, key, value all from same sequence. Scoring — dot product, softmax, weighted sum — exactly same machinery built in this section.

2017: Vaswani et al. published "Attention Is All You Need," built Transformer entirely on self-attention, discarding recurrence. Timeline: Sutskever (2014) → encoder-decoder framework. Bahdanau (2015) → eliminated bottleneck with cross-attention. Luong (2015) → simplified scoring to dot product. Vaswani (2017) → "what if attention is all we need?" Each step consequence of one before it.
