State Space Models & Mamba

Chapter 10: Sequence Models & Attention Beyond O(n²) · Selective state spaces · The emerging challenger
TL;DR

Transformers pay O(n²) for attention — painful at long sequence lengths. State space models (SSMs) borrow from control theory to process sequences with linear complexity. The trick: the same math can run as a convolution during training (parallel, GPU-friendly) and as a recurrence during inference (O(1) per step, constant memory). Mamba adds input-dependent selectivity — the model learns what to remember and what to forget based on content, not just position. The result is transformer-quality modeling at a fraction of the cost for long sequences. The field is young, moving fast, and increasingly converging toward hybrid architectures.

The Transformer's Achilles' Heel

Self-attention is powerful because every token can look at every other token. But that power has a cost: for a sequence of length n, attention computes an n × n matrix of scores. Double the sequence length, quadruple the compute. That's O(n²) in both time and memory.

For a 512-token input, this is fine. For a 100K-token context window — the kind you need for an entire codebase or a full novel — it's brutal. Modern LLMs use clever tricks to cope: FlashAttention for memory efficiency, KV caches, sliding windows, sparse attention patterns. But these are band-aids on a fundamentally quadratic algorithm.

Which raises a question: is there a model that processes sequences without the quadratic bottleneck, while still capturing long-range dependencies?

State Space Models — The Core Idea

The answer comes from an unexpected place: control theory. Engineers have modeled dynamic systems for decades using state space equations. A satellite tracking system, a thermostat, an aircraft autopilot — all described by the same math:

# Continuous-time state space model (the math, not code)
# x'(t) = A x(t) + B u(t)    ← state update: how the hidden state evolves
# y(t)  = C x(t) + D u(t)    ← output: what we observe

# x(t) is the hidden state (what the system "remembers")
# u(t) is the input signal
# y(t) is the output signal
# A governs state dynamics, B maps input to state,
# C maps state to output, D is a skip connection

The key insight: this is a continuous-time recurrence. The hidden state x(t) summarizes everything the system has seen so far, and it evolves smoothly over time based on the current input. Sound familiar? It's conceptually similar to an RNN's hidden state — but grounded in a mathematical framework with 70+ years of theory behind it.

To use this for discrete sequences (like tokens), you discretize the equations. Pick a step size Δ, and convert the continuous A, B into discrete versions Ā, B̄ using standard techniques (zero-order hold, bilinear transform). Now you have:

# Discretized state space model
# x_k = Ā x_{k-1} + B̄ u_k    ← recurrence: process one token at a time
# y_k = C x_k                  ← (D term often dropped for simplicity)

# This looks exactly like an RNN! But there's a crucial difference...

The Dual Nature: Convolution + Recurrence

Here's where SSMs get clever. That recurrence above? If you unroll it across the entire sequence, something beautiful happens. The output at each position is a weighted combination of all previous inputs — and the weights form a fixed kernel determined by A, B, C. In other words, the recurrence can be rewritten as a convolution.

# Unrolling the recurrence reveals a convolution kernel:
# K = (C B̄, C Ā B̄, C Ā² B̄, ..., C Ā^{L-1} B̄)
#
# y = K * u   (where * is convolution)
#
# During TRAINING:  use convolution → parallel, O(n log n) via FFT
# During INFERENCE: use recurrence  → sequential, O(1) per step

# Best of both worlds!
import torch

def ssm_convolution(u, A_bar, B_bar, C, L):
    """Training mode: build kernel, convolve (parallel)."""
    kernel = []
    A_power = torch.eye(A_bar.shape[0])
    for i in range(L):
        kernel.append(C @ A_power @ B_bar)
        A_power = A_power @ A_bar
    K = torch.stack(kernel)  # shape: (L,)
    return torch.nn.functional.conv1d(
        u.unsqueeze(0), K.flip(0).unsqueeze(0).unsqueeze(0), padding=L-1
    )[..., :L]

def ssm_recurrence(u, A_bar, B_bar, C):
    """Inference mode: step-by-step (O(1) memory per step)."""
    x = torch.zeros(A_bar.shape[0])
    outputs = []
    for u_k in u:
        x = A_bar @ x + B_bar * u_k
        outputs.append(C @ x)
    return torch.stack(outputs)
Why This Matters

RNNs can do O(1) per step at inference but are sequential during training — painfully slow. Transformers are parallel during training but O(n²). SSMs get O(n log n) parallel training and O(1)-per-step inference. This isn't a compromise — it's genuinely the best of both worlds.

HiPPO: Teaching the State to Remember

There's a catch. A random matrix A will produce a state that forgets quickly — the recurrence either explodes or decays to zero, just like vanilla RNNs. The model needs to remember long-range dependencies, and that means initializing A very carefully.

This is the HiPPO framework (High-order Polynomial Projection Operator). The intuition: at every time step, the hidden state should optimally compress the entire history of inputs it has seen. HiPPO does this by projecting the input history onto a basis of orthogonal polynomials — think of it as fitting a polynomial approximation to everything you've seen so far and updating that approximation efficiently as new inputs arrive.

You don't need to understand the math of Legendre polynomials to get the key takeaway: HiPPO gives you a specific matrix A that is provably optimal for remembering history. And it works — models initialized with HiPPO can capture dependencies thousands of steps in the past, where vanilla RNNs and even LSTMs fail completely.

Don't Confuse the A Matrix with Learned Weights

In early SSMs, A is initialized with the HiPPO matrix and may be partially learned during training. It's more like a carefully designed inductive bias than a random initialization. This structural prior is what makes SSMs fundamentally different from "just another RNN."

S4: Making SSMs Practical

The HiPPO matrix has a specific structure — it's a particular kind of low-rank-plus-diagonal matrix. S4 (Structured State Spaces for Sequence Modeling) exploits this structure to make the computation efficient. Computing the convolution kernel naively for large state dimensions is expensive, but S4 decomposes the problem using techniques from numerical linear algebra — specifically, the connection between the HiPPO matrix and Cauchy kernels — to bring the cost down dramatically.

S4 was the proof-of-concept that SSMs could actually compete. It achieved state-of-the-art results on the Long Range Arena benchmark — a suite of tasks specifically designed to test long-range dependency modeling, where transformers struggled. Sequences of 1K–16K tokens, tasks like classifying documents by their full content or determining if two points in a path are connected. S4 crushed these benchmarks while using a fraction of the compute.

But S4 had limitations. The A, B, C matrices were fixed — they didn't change based on what the model was looking at. The model processed "the cat sat" and "important: remember this" with exactly the same dynamics. It couldn't selectively pay attention to content.

Mamba: The Breakthrough

This is the big one. Mamba (Gu & Dao, 2023) makes a deceptively simple change that transforms SSMs from a niche research curiosity into a genuine transformer competitor: the matrices B, C, and the step size Δ become functions of the input.

# Standard SSM (S4-style): parameters are fixed
# x_k = Ā x_{k-1} + B̄ u_k       ← same Ā, B̄ for every token

# Mamba (selective SSM): parameters depend on the input
# B_k = Linear(u_k)               ← B changes per token
# C_k = Linear(u_k)               ← C changes per token
# Δ_k = softplus(Linear(u_k))     ← step size changes per token
# Ā_k, B̄_k = discretize(A, B_k, Δ_k)
# x_k = Ā_k x_{k-1} + B̄_k u_k   ← dynamics change based on content!

# Pseudocode for a Mamba block (simplified)
import torch
import torch.nn as nn

class SimplifiedMambaBlock(nn.Module):
    def __init__(self, d_model, d_state, d_conv=4):
        super().__init__()
        self.d_state = d_state
        # Input projections
        self.in_proj = nn.Linear(d_model, 2 * d_model, bias=False)
        # Local convolution for short-range patterns
        self.conv1d = nn.Conv1d(d_model, d_model, d_conv,
                                padding=d_conv - 1, groups=d_model)
        # Selective parameters — these depend on the input!
        self.x_proj = nn.Linear(d_model, d_state * 2 + 1, bias=False)
        # A is still initialized with structure (log-spaced)
        self.A_log = nn.Parameter(
            torch.log(torch.arange(1, d_state + 1).float()))
        self.out_proj = nn.Linear(d_model, d_model, bias=False)

    def forward(self, x):
        # x: (batch, seq_len, d_model)
        z, x = self.in_proj(x).chunk(2, dim=-1)

        # Local conv captures short-range patterns
        x = self.conv1d(x.transpose(1, 2))[..., :x.shape[1]]
        x = x.transpose(1, 2)
        x = torch.nn.functional.silu(x)

        # Input-dependent B, C, Δ — the "selective" part
        x_dbl = self.x_proj(x)
        B = x_dbl[..., :self.d_state]
        C = x_dbl[..., self.d_state:2*self.d_state]
        delta = torch.nn.functional.softplus(x_dbl[..., -1:])

        # Selective scan (the actual SSM computation)
        A = -torch.exp(self.A_log)
        y = self.selective_scan(x, A, B, C, delta)

        # Gate and project
        return self.out_proj(y * torch.nn.functional.silu(z))

Why does this matter so much? Think about reading a document. Some tokens are important ("The answer is 42"), others are filler ("As previously mentioned in the above paragraph"). A fixed SSM treats them all the same. Mamba's selective mechanism lets the model open the gate wide for important tokens (large Δ → write strongly to state) and close it for irrelevant ones (small Δ → state barely changes). It's content-based filtering — conceptually similar to what attention does, but with linear complexity.

The Selectivity Trade-off

Making B, C, Δ input-dependent breaks the convolution trick — the kernel is no longer fixed, so you can't use FFT. Mamba compensates with a hardware-aware parallel scan algorithm that exploits GPU memory hierarchy (registers → SRAM → HBM). This is why Mamba isn't just a math paper — it's also a systems paper. The algorithm is carefully designed to be fast on actual hardware, not just in Big-O notation.

An Honest Assessment

Let's be real: transformers still dominate. Every frontier model — GPT-4, Claude, Gemini, LLaMA 3 — is a transformer. The ecosystem, the tooling, the training recipes, the years of accumulated engineering wisdom — all built around attention. Mamba isn't replacing transformers tomorrow.

But the results are compelling enough that you can't ignore them. Mamba matches transformer quality at medium scale while being significantly faster for long sequences. For specific use cases — long-document processing, genomics, audio, time series — SSMs already make more sense than transformers.

Don't Bet the Farm

If you're starting a production project today, use a transformer. The tooling, community support, and proven scaling properties are unmatched. Follow SSMs/Mamba closely, but don't architect your system around a paradigm that's still proving itself at scale. The time to adopt is when — not if — the ecosystem matures.

The research trajectory is moving fast:

  • Mamba-2 (2024) — reformulates the selective SSM as structured masked attention, unifying SSM and attention theory. Faster and simpler to implement.
  • Jamba (AI21 Labs) — a production hybrid mixing Mamba and transformer layers. 256K context window, deployed at scale.
  • RWKV — a related linear-attention approach with similar O(n) properties, trained at up to 14B parameters.
  • Griffin / Hawk (Google DeepMind) — recurrence-based models that compete with transformers on standard benchmarks.

Hybrid Architectures: The Pragmatic Future

Here's the thing — it probably won't be "transformers vs. Mamba." It'll be both.

Jamba is the clearest example. It interleaves transformer layers (for precise, long-range retrieval via attention) with Mamba layers (for efficient sequence compression). You get attention where you need it — tasks like in-context retrieval where exact token lookup matters — and SSM layers where you don't, saving compute on the bulk of sequence processing.

# Conceptual hybrid architecture (Jamba-style)
class HybridBlock(nn.Module):
    """Alternates between Mamba and Attention layers."""
    def __init__(self, d_model, n_layers, attention_every=4):
        super().__init__()
        self.layers = nn.ModuleList()
        for i in range(n_layers):
            if (i + 1) % attention_every == 0:
                # Every 4th layer: full attention (precise retrieval)
                self.layers.append(TransformerBlock(d_model))
            else:
                # Most layers: Mamba (efficient sequence processing)
                self.layers.append(MambaBlock(d_model))

    def forward(self, x):
        for layer in self.layers:
            x = layer(x)
        return x

# Ratio matters: Jamba uses ~1 attention layer per 7 Mamba layers
# This gives most of attention's retrieval ability
# at a fraction of the compute cost

The intuition: attention is a precise lookup — "find me exactly the token I need." An SSM is a compressed summary — "here's the gist of everything so far." Most of the time, the gist is enough. When it's not, a few attention layers handle the hard cases.

The Analogy

Think of it like human memory. You don't have perfect recall of every conversation you've ever had (that would be attention over your entire life — O(n²) on your lifespan). Instead, you compress most experiences into general knowledge (SSM-like state), while keeping a few recent or critical memories in vivid detail (attention-like retrieval). Hybrid models work the same way.
