GPT Internals

Chapter 10: Large Language Models Section 9 ~22 min read
TL;DR

GPT is a decoder-only Transformer trained to predict the next token. That's it. A triangular causal mask prevents the model from peeking at future tokens during training, and cross-entropy loss punishes wrong predictions. This absurdly simple objective — when scaled to billions of parameters and trillions of tokens — produces models that can write code, pass bar exams, and reason about novel problems. GPT-1 proved unsupervised pre-training works. GPT-2 showed zero-shot abilities. GPT-3 unlocked few-shot prompting. GPT-4 added multimodality and RLHF alignment. Along the way, scaling laws revealed a clean power-law relationship between compute, data, and loss — and the Chinchilla result showed that most models were starved of data. The philosophy underneath all of this: predicting the next token well enough may require understanding the world itself.

The Autoregressive Language Model Objective

Why did autoregressive language models win? Because their training objective is almost embarrassingly simple: given all the tokens so far, predict the next one. Formally, you're modeling P(x_t | x_1, x_2, ..., x_{t-1}). The probability of a full sentence is just the product of these conditional probabilities, one per token, left to right. No labels. No annotation pipeline. No task-specific heads. Just a giant corpus of text and a model that learns to continue it.

The loss function is cross-entropy loss. At each position t, the model outputs a probability distribution over the entire vocabulary — say, 50,257 tokens for GPT-2. Cross-entropy measures how far that predicted distribution is from the "true" distribution, which is just a one-hot vector placing all probability mass on the actual next token. In code, this boils down to -log(p(correct_token)) averaged over all positions. If the model assigns 90% probability to the right token, the loss is small. If it assigns 0.1%, the loss is huge. Summed over trillions of training tokens, this signal is extraordinarily rich.

Why is this simple objective so powerful? Because predicting what comes next in natural language requires an enormous amount of implicit knowledge. To predict the next word after "The capital of France is," you need geography. After "def fibonacci(n):", you need programming. After "She felt a pang of guilt because," you need theory of mind. The next-token objective doesn't teach any of these things directly. It teaches all of them indirectly, because you can't consistently predict well without building internal representations of how the world works.

Training uses a technique called teacher forcing. During training, you don't feed the model its own predictions — you feed it the ground truth. At position t, the model gets the actual tokens x_1 through x_{t-1} as input and must predict x_t. This is critical for efficiency: because you always feed ground truth, you can compute the loss at every position in parallel. The model processes the full sequence in one forward pass and gets a gradient signal from every single token. Without teacher forcing, you'd need to generate one token at a time during training, which would be thousands of times slower.

Causal Masking

Here's the problem. A standard Transformer's self-attention lets every token attend to every other token. That's fine for BERT-style tasks where you want bidirectional context. But for generation, it's cheating. If token 5 can attend to token 10, the model can trivially "predict" token 6 by just reading ahead. The predictions would be perfect during training and useless during inference, because at inference time, future tokens don't exist yet.

The solution is a causal mask — also called a look-ahead mask. It's a triangular matrix that blocks attention from any position to any future position. Token 1 can only attend to itself. Token 2 can attend to tokens 1 and 2. Token t can attend to tokens 1 through t. This enforces the autoregressive property at the attention level: each position's representation depends only on past and present, never on the future.

The implementation detail that trips people up: the mask is additive, not multiplicative. You don't multiply attention scores by 0 to block them. Instead, you add -inf (or a very large negative number like -1e9) to the attention scores at blocked positions before applying softmax. Since softmax(-inf) = 0, those positions get zero attention weight. This is numerically more stable and plays nicely with the softmax computation. Here's how you build one:

import torch

def create_causal_mask(seq_len):
    """Create a causal (look-ahead) mask for autoregressive attention.
    
    Returns a matrix where masked positions are -inf and allowed positions are 0.
    Shape: (seq_len, seq_len)
    """
    # torch.triu returns upper triangular part; diagonal=1 means above the main diagonal
    mask = torch.triu(torch.ones(seq_len, seq_len), diagonal=1)
    # Replace 1s with -inf (blocked) and 0s stay as 0 (allowed)
    mask = mask.masked_fill(mask == 1, float('-inf'))
    return mask

mask = create_causal_mask(5)
print(mask)
# tensor([[  0., -inf, -inf, -inf, -inf],
#         [  0.,   0., -inf, -inf, -inf],
#         [  0.,   0.,   0., -inf, -inf],
#         [  0.,   0.,   0.,   0., -inf],
#         [  0.,   0.,   0.,   0.,   0.]])

Look at that matrix. Row 0 (the first token) sees only column 0 — itself. Row 3 sees columns 0 through 3. Every entry above the diagonal is -inf, which means those attention connections are dead. You add this mask to the raw attention scores (before softmax), and the model can't cheat. During training, the model processes the full sequence at once but each position only "sees" what it's allowed to see.

Decoder-Only Architecture

The original Transformer from the "Attention Is All You Need" paper had both an encoder and a decoder, connected by cross-attention. BERT used only the encoder. GPT stripped it down to just the decoder — and not even the full decoder. There's no cross-attention because there's no encoder to attend to. What remains is remarkably simple: token embeddings, positional embeddings, then a stack of identical layers, each containing masked multi-head self-attention followed by a feed-forward network (FFN), with layer normalization and residual connections.

Why is simpler better here? Two reasons. First, the decoder-only architecture treats everything as a single sequence. There's no notion of "source" and "target" — just a stream of tokens. This makes it trivially general. You can do translation ("Translate to French: The cat sat on the mat. →"), summarization ("Summarize: [long text] →"), question answering ("Q: What is DNA? A:"), and conversation ("User: Hi! Assistant:") all with the same architecture and the same training objective. No architectural changes per task. Second, it simplifies engineering. One kind of attention, one data pipeline, one inference path. At the scale GPT operates — hundreds of billions of parameters, thousands of GPUs — every simplification compounds.

GPT models use Byte Pair Encoding (BPE) for tokenization, which differs from BERT's WordPiece in an important way. Both are subword tokenizers — they split rare words into pieces while keeping common words intact. BPE works bottom-up: start with individual characters, then repeatedly merge the most frequent adjacent pair. "learning" might become ["learn", "ing"]. WordPiece uses a similar idea but selects merges based on likelihood improvement rather than raw frequency. In practice, BPE tends to produce slightly different tokenizations. GPT-2's tokenizer has 50,257 tokens and operates on bytes rather than Unicode characters, which means it can handle any text without "unknown token" errors — a pragmatic advantage.

GPT-1 → GPT-2 → GPT-3 → GPT-4 Evolution

GPT-1 appeared in 2018 with 117 million parameters, 12 Transformer layers, and a 768-dimensional embedding. The core contribution wasn't architectural novelty — it was the demonstration that unsupervised pre-training on a large text corpus (BooksCorpus, about 7,000 books) followed by supervised fine-tuning produced state-of-the-art results on a range of NLP benchmarks. Before GPT-1, the dominant paradigm was training task-specific models from scratch, often with carefully engineered features. GPT-1 said: just pre-train a general language model, then add a thin task-specific layer on top and fine-tune. The pre-training captures so much about language that the fine-tuning can be minimal.

GPT-2 arrived in 2019 with 1.5 billion parameters — roughly 13× larger. It was trained on WebText, a dataset of about 8 million web pages curated by following links from Reddit posts with high karma. Two things made GPT-2 historic. First, it showed that a large enough language model develops zero-shot abilities: it could perform tasks it was never explicitly trained on, just from the prompt. You could write "TL;DR:" after a paragraph and GPT-2 would produce a summary — without any fine-tuning for summarization. Second, OpenAI initially refused to release the full model, claiming it was "too dangerous" due to its ability to generate convincing fake text. This kicked off a massive debate about AI safety and open release that continues today. Architecturally, GPT-2 introduced Pre-Norm — moving layer normalization before the attention and FFN sublayers rather than after them (Post-Norm). This seemingly small change significantly stabilized training for deeper models.

GPT-3 arrived in 2020 with 175 billion parameters, 96 layers, and a context window of 2,048 tokens. The training data was a filtered version of Common Crawl plus books and Wikipedia — about 300 billion tokens. GPT-3's breakthrough was few-shot learning via prompting. You didn't need to fine-tune it at all. Just write a few examples of the task in the prompt, and GPT-3 would pick up the pattern. "Translate English to French: sea otter → loutre de mer, cheese → fromage, hello →" and it would output "bonjour." This was called in-context learning, and it was qualitatively new. The model was learning from examples at inference time, without any gradient updates. For many tasks, few-shot GPT-3 matched or beat fine-tuned BERT models — a result that startled the field.

GPT-4 appeared in 2023 and marked another leap. OpenAI released almost no architectural details, but the model is widely rumored to be a Mixture of Experts (MoE) architecture — multiple expert sub-networks, with a routing mechanism that activates only a subset for each token. This would explain how it achieves massive effective capacity while keeping inference cost manageable. GPT-4 is multimodal: it can process images alongside text. It was trained with Reinforcement Learning from Human Feedback (RLHF) to align its outputs with human preferences — making it more helpful, less harmful, and more honest. The reasoning abilities improved dramatically: GPT-4 passed the bar exam in the 90th percentile, solved competition math problems, and wrote working programs for complex specifications.

What Changed at Each Scale

The pattern across GPT generations is consistent: more parameters, more data, longer training, new capabilities. GPT-1 proved the paradigm (pre-train then fine-tune). GPT-2 showed zero-shot generalization (no fine-tuning needed for simple tasks). GPT-3 showed few-shot learning (learning from examples in the prompt). GPT-4 showed near-expert-level reasoning and multimodal understanding. Each generation didn't just do the same things better — it unlocked qualitatively different abilities that the previous generation simply could not do.

Emergent Abilities

Some abilities don't appear gradually as you scale up a language model. They appear suddenly. A 1B-parameter model can't do three-digit addition. A 10B-parameter model still can't. A 100B-parameter model suddenly can. This phenomenon is called emergent abilities — capabilities that are absent in smaller models and present in larger ones, with a sharp transition rather than a smooth improvement.

The examples are striking. Arithmetic: small models fail at multi-digit addition; large models get it right. Chain-of-thought reasoning: small models produce incoherent reasoning chains; large models can work through multi-step logic problems if you prompt them with "Let's think step by step." Code generation: GPT-2 produces syntactically plausible but logically broken code; GPT-3 and GPT-4 write correct solutions to LeetCode problems. These aren't different tasks the models were trained on differently — they all trained on the same next-token objective. The capability emerged from scale alone.

But there's a heated debate about whether emergent abilities are real or an artifact of measurement. A 2023 paper by Schaeffer et al. argued that many "emergent" abilities disappear if you change the evaluation metric. When you use exact-match accuracy (did the model get the answer perfectly right?), you see a sharp jump at some scale. But if you switch to a continuous metric like token-level log-likelihood, the improvement is smooth and gradual. The argument: the model was getting better all along, but the binary metric made it look like a phase transition. This doesn't fully resolve the debate — some abilities do seem genuinely discontinuous — but it's an important reminder that how you measure matters as much as what you measure.

In-Context Learning

In-context learning is one of the most surprising properties of large language models. You write examples of a task directly in the prompt, and the model performs the task on a new input — without any weight updates. No fine-tuning. No gradient descent. The model's weights are frozen. It "learns" entirely from the context window.

Zero-shot means you describe the task but give no examples: "Translate the following English text to French: 'The weather is nice today.'" The model must infer the task from the instruction alone. Few-shot means you provide a handful of examples: "English: sea otter → French: loutre de mer / English: cheese → French: fromage / English: hello → French:" and the model completes the pattern. The more examples, the better the performance — up to a point.

Why does this work? Nobody fully knows, but there are two leading hypotheses. The first is that in-context learning is a form of implicit gradient descent. Some researchers have shown that Transformer attention layers can implement a single step of gradient descent in their forward pass — the examples in the prompt are like a training set, and the model internally "fits" a small model to them. The second hypothesis is simpler: the model has seen so many task patterns during pre-training that few-shot examples are just task recognition. The examples don't teach the model anything new; they tell it which of its already-learned behaviors to activate.

In-Context Learning Is Fragile

The order of examples matters — sometimes dramatically. The format of the prompt matters: slight changes in wording can swing accuracy by 20 points. The choice of examples matters: cherry-picked examples outperform random ones. For production systems, this means you need to invest in prompt engineering and example selection, not just assume that any few-shot prompt will work. Techniques like retrieving examples similar to the test input (retrieval-augmented few-shot) can help stabilize performance.

from transformers import pipeline

generator = pipeline("text-generation", model="gpt2")

# Zero-shot: describe the task
zero_shot_prompt = """Classify the sentiment of the following review as Positive or Negative.
Review: "This movie was a complete waste of time."
Sentiment:"""

# Few-shot: provide examples
few_shot_prompt = """Classify the sentiment of each review.

Review: "Absolutely loved it, best film of the year!"
Sentiment: Positive

Review: "Terrible acting and a predictable plot."
Sentiment: Negative

Review: "A masterpiece of modern cinema."
Sentiment: Positive

Review: "I fell asleep halfway through."
Sentiment:"""

output = generator(few_shot_prompt, max_new_tokens=5, do_sample=False)
print(output[0]["generated_text"])

Scaling Laws

In 2020, Kaplan et al. at OpenAI published a landmark result: the test loss of a language model follows a power law relationship with the number of parameters, the size of the training dataset, and the amount of compute used. Double the parameters, and loss drops by a predictable amount. Double the data, and loss drops by a different but equally predictable amount. These relationships hold over many orders of magnitude — from models with millions of parameters to ones with hundreds of billions.

The power law takes the form L = a * N^(-α) where L is the loss, N is the number of parameters (or tokens, or compute), and α is a scaling exponent. The exponents differ: scaling parameters alone has diminishing returns compared to scaling data or compute. The critical insight was that these laws are smooth and predictable, which means you can extrapolate. You can train small models cheaply, measure the loss, and predict what loss a 100× larger model would achieve — before spending the millions of dollars to train it.

Then came Chinchilla. In 2022, Hoffmann et al. at DeepMind trained a suite of models with different ratios of parameters to training tokens, all with the same total compute budget. They found that the Kaplan scaling laws had a flaw: they assumed you should scale parameters faster than data. The Chinchilla result showed the optimal strategy is to scale both proportionally. A 70B-parameter model trained on 1.4 trillion tokens (Chinchilla) outperformed a 280B-parameter model trained on 300 billion tokens (Gopher) — despite using the same total compute. The implication was stark: most existing large models were significantly undertrained. They had too many parameters and not enough data.

For practitioners, the takeaway is pragmatic. If you have a fixed compute budget, don't just make the model bigger. Make it the right size and train it on enough data. The Chinchilla-optimal ratio suggests roughly 20 training tokens per parameter. A 7B model should see about 140B tokens. Many open-source models now follow this recipe — LLaMA, for instance, was explicitly trained as a Chinchilla-optimal model and punched far above its weight class because of it.

Scaling Laws in Practice

Scaling laws don't tell you everything. They predict loss on a held-out test set, not performance on specific benchmarks or tasks. A model with lower loss generally performs better, but the relationship between loss and useful capability isn't linear. Small improvements in loss near the frontier can unlock qualitatively new abilities (see: emergent abilities above). And scaling laws say nothing about alignment — a model that predicts text perfectly could still be unhelpful, deceptive, or harmful. Loss is necessary but not sufficient.

The Philosophy of Next-Token Prediction

Ilya Sutskever, co-founder of OpenAI, has made a provocative argument that goes something like this: if you're predicting the next token of a text that describes a complex situation — say, a detective solving a murder — then to predict well, you need to understand the detective's reasoning, the motive of the killer, the physical layout of the crime scene, and the reliability of each witness. To predict the next token perfectly, you'd need to understand everything the text is about. In the limit, perfect next-token prediction requires a world model.

There's a deep connection here to information theory. Compression and intelligence are two sides of the same coin. A language model is literally a compression algorithm — it assigns shorter codes (higher probabilities) to likely continuations and longer codes (lower probabilities) to unlikely ones. A model that compresses text well must have learned the patterns in that text: grammar, facts, reasoning, style. The better the compression, the deeper the understanding. This isn't just metaphor; it's the mathematical foundation of minimum description length and Kolmogorov complexity.

But there are real limitations. Next-token prediction captures correlations, not causation. A model trained on medical text can predict that "Smoking causes" is likely followed by "cancer," but it doesn't know what "causes" means in a scientific sense. It's learned that these words co-occur in certain patterns, not that there's a biological mechanism linking them. This is why language models hallucinate — they generate text that sounds plausible because it matches statistical patterns, even when the content is factually wrong. The model can't distinguish "this is true" from "this sounds like the kind of thing that would be true."

There are also structural limitations. Next-token prediction is inherently sequential and left-to-right. Humans don't think this way. We plan, we revise, we reason backwards from goals. We write a paragraph, realize the logic doesn't work, and restructure. A language model commits to each token irrevocably (during a single generation pass) and can't go back. Techniques like chain-of-thought prompting and tree-of-thought search are patches for this limitation — they let the model "think out loud" before committing to an answer — but they don't change the fundamental architecture.

What can't next-token prediction capture? Genuinely novel reasoning that doesn't resemble anything in the training data. Reliable factual recall (the model is a fuzzy lookup table, not a database). Consistent long-term planning over thousands of tokens. Formal verification — the model can write a proof that looks right without checking that every step is logically valid. These aren't temporary limitations that will disappear with scale. They're structural consequences of the objective. Scale helps enormously — GPT-4 is far more reliable than GPT-2 — but the failure modes remain, just at a lower frequency.

from transformers import AutoModelForCausalLM, AutoTokenizer
import torch

# Load GPT-2 and see next-token prediction in action
tokenizer = AutoTokenizer.from_pretrained("gpt2")
model = AutoModelForCausalLM.from_pretrained("gpt2")

prompt = "The Eiffel Tower is located in"
inputs = tokenizer(prompt, return_tensors="pt")

with torch.no_grad():
    logits = model(**inputs).logits

# Get probabilities for the next token after "in"
next_token_probs = torch.softmax(logits[0, -1, :], dim=0)
top5 = torch.topk(next_token_probs, 5)

print("Top 5 next-token predictions:")
for prob, idx in zip(top5.values, top5.indices):
    token = tokenizer.decode(idx)
    print(f"  '{token}' — {prob.item():.3f}")
# Likely output: ' Paris' with high probability,
# followed by ' the', ' France', etc.

This is the miracle and the limitation of GPT in a single code snippet. The model assigns high probability to "Paris" because it has seen "Eiffel Tower" and "located in" together thousands of times during training. It's statistically right. But it doesn't "know" that the Eiffel Tower is in Paris the way you know it — by having seen pictures, read history, understood geography. It knows it the way a compression algorithm knows it: as a pattern that reduces prediction error. Whether that's "real" understanding or a sophisticated mimic is the central philosophical question of modern AI. The honest answer is: we don't know yet. And that uncertainty, more than any benchmark score, is what makes this field so fascinating.
