Next: tour of important ideas beside standard Transformer. Some became stepping stones to modern LLMs. Others dead ends. Few still fight for throne. None required for core chapter, but spotting them fast saves hours and helps in architecture discussion.

The Context Window Problem — Transformer-XL

Imagine system summarizing long documents — legal contracts. Transformer has context window of 512 tokens. Contract has 4,000 tokens. Standard move: chop contract into eight 512-token segments, process each alone. Segment one gets summary. Segment two gets summary. But segment two knows nothing from segment one. If contract says "the party defined in Section 1" in segment five, segment five never saw Section 1. Name: context fragmentation.

Like reading novel while someone tears out each chapter after you finish. You understand paragraphs, but cannot track plot across chapters. Model stuck there.

Transformer-XL (Dai et al., 2019) fixed this with segment-level recurrence. Model finishes segment one, keeps hidden states instead of discarding them. Caches them. When segment two arrives, attention reaches back into cache and attends to segment one's representations. Effective context window grows — not by enlarging attention matrix, but by carrying memory forward.

One wrinkle. Standard transformers use absolute positional encodings — token at position 47 always gets same positional signal. Concatenate segments, and position 0 of segment two is really position 512 of document. Model gets distance confused. Transformer-XL added relative positional encoding to fix this. Instead of encoding "I am at position 47," each attention computation encodes "this key is 3 positions to my left." Model reasons about distances, not absolute coordinates.

Think this way: absolute encoding is GPS coordinate stamped on every word. Relative encoding is knowing "that word was two sentences ago." Second one still works when you move to new document section.

Cached hidden states plus relative positions make effective context grow linearly with number of layers. 16-layer Transformer-XL can attend about 16 segments back, giving context hundreds of times longer than raw segment length. No increase in quadratic attention cost per segment. Each segment still does standard attention inside itself; it attends to cache for free because states already exist.

Limitation: model still processes segments sequentially during training — cannot cache what does not exist yet — so training is slower than model that parallelizes full sequence. Cache size is fixed. You choose how many previous segments to remember; everything older disappears. Sliding window, not infinite memory.

Still, Transformer-XL was key precursor to XLNet and shaped how modern LLMs handle long contexts. Hear "extending context window" for GPT-style models; intellectual DNA leads back here.

The Masking Dilemma — XLNet

Transformer-XL handled long contexts. Different problem came next. By 2019, field split into two camps for pretraining language models, and both had blind spots.

GPT camp trains left-to-right: predict each word from earlier words. This is autoregressive training. Clean and natural — model generates text same way it trained. But it only sees left context. If word "bank" needs word on right for meaning, model loses.

BERT camp masks random words and predicts them from remaining context. This is masked language modeling. Model sees left and right context. Catch: those [MASK] tokens appear during training but never in real use. Model trains on slightly alien input. When multiple tokens are masked, BERT also treats them as independent, often false. If "New" and "York" are both masked, BERT predicts each separately and ignores tight correlation.

First time I read XLNet paper, random prediction order looked useless. Sounded like chaos for no reason. Took three reads to see trick.

XLNet (Yang et al., 2019) introduced permutation language modeling. Idea: take sentence like "The cat sat on the mat." Normally you predict left-to-right. Instead, randomly shuffle prediction order — maybe position 4 first, then 1, then 5, then 3. For each position, predict it using all positions that came before it in that random order.

Trace tiny example. Sentence has five tokens: [A, B, C, D, E]. One random prediction order might be [3, 1, 5, 2, 4]. First predict C from no context, then A from C, then E from C and A, then B from C, A, and E, then D from everything. Another training step might use [2, 4, 1, 5, 3], where B comes first and contexts shift completely.

Over many training steps, every token sees many subsets of other tokens as context. Position 3 sometimes has left and right context. Sometimes almost everything. Sometimes almost nothing. Model learns prediction from any partial sentence view — effectively bidirectional, like BERT, but without [MASK] hack.

Key insight: actual input tokens never change. Words stay in original input order. What changes is factorization order — which token gets predicted when. Model needs way to know "I'm predicting position 4, and already saw positions 3 and 1 in this permutation." XLNet uses two-stream self-attention: content stream carries actual token information, query stream knows which position is being predicted but cannot peek at token there. Trickiest part of architecture. Two streams stop model from cheating — seeing answer to its own question.

XLNet sits on Transformer-XL, reusing segment-level recurrence and relative positional encoding, so it also handles long contexts. At release, it beat BERT on 20 benchmarks. Early proof that bidirectional context did not require masking.

Limitation: permutation training costs a lot. You must sample many permutations per sentence to cover space well, so training is slower than GPT or BERT. In practice, only subset of positions are predicted per permutation — usually last few in factorization order. Helps, but adds complexity. Field later moved toward scaling simple autoregressive models instead of elaborate pretraining objectives. But insight remains: prediction factorization order is design choice, not fixed constraint.

The Quadratic Cost — Reformer and LSH Attention

So far we tackled "how much can model see." Transformer-XL extended context with recurrence. XLNet enriched context with permutations. Underneath sits rougher problem: attention itself scales quadratically with sequence length.

For 512 tokens, attention matrix has 512 × 512 = about 262,000 entries. Manageable. For 4,096 tokens, about 16 million entries. For 65,000 tokens, more than 4 billion entries. Per layer. Per attention head. Math turns ugly fast.

Reformer (Kitaev et al., 2020) asked sharp question: does every token need attention to every other token? In trained attention matrices, pattern is often sparse — each token strongly attends to few others and ignores rest. What if we could guess which tokens would attend to each other before computing full attention matrix?

Answer: locality-sensitive hashing (LSH) attention. Core idea: LSH comes from nearest-neighbor search literature. You build hash function with special property: similar vectors land in same bucket with high probability, dissimilar vectors land in different buckets. Like sorting library by topic instead of alphabet — books on same subject share shelf even if titles start with different letters.

In Reformer, queries and keys go through LSH into buckets. Attention is computed only between tokens in same bucket — tokens that likely would have high attention weights anyway. Tokens in different buckets are assumed to have negligible attention and get skipped. Complexity drops from O(n²) to about O(n log n).

To lower risk of missing important connections — two related tokens landing in different buckets — Reformer runs multiple hashing rounds with different random hash functions and combines results. Probabilistic bet: if two tokens are truly similar, they collide in at least one round.

Reformer had second innovation: reversible layers. Normal transformers store every layer's activations for backpropagation — big memory cost in deep models. Reversible layers, borrowed from RevNets, let you recompute any layer's activations from layer above it. Trade compute for memory. Instead of storing activations for all L layers (O(n·L) memory), you store one layer and recompute on fly (O(n) memory).

Outcome time. Reformer was brilliant engineering, but constant factors in LSH attention were large. Hashing, sorting, and bucket management added overhead that ate theoretical gains. Then Flash Attention arrived and attacked same problem from different angle — reorganizing GPU memory access instead of approximating attention itself — and won. Flash Attention gives exact attention, faster, with no approximation error.

Reformer matters historically. It pushed community to ask right questions about attention efficiency. But production use today is rare. When cited, it usually appears in lineage section of papers that carried its ideas further.

Giving Networks Scratch Paper — Neural Turing Machines

Everything so far stores information in same place: network hidden states. Transformer-XL caches hidden states. XLNet changes prediction order but still computes hidden states. Even attention mostly routes information between hidden states. Network memory and computation live in same substrate.

But think how you solve hard problem. You do not hold everything in head. You write notes. Make lists. Cross out items. Flip back to page one. Working memory is tiny; notebook is huge. Notebook is external storage — you read from it, write to it, and thinking stays separate from storage medium.

In 2014, Graves, Wayne, and Danihelka at DeepMind asked: what if neural network had notebook?

Neural Turing Machine (NTM) is neural network plus external memory matrix — think table with N rows and M columns, where each row is memory slot. Controller network, usually LSTM or feedforward net, reads input, decides what to read from memory, what to write, and produces output. Every time step, controller issues read and write instructions.

Critical trick is how controller addresses memory. Regular computer uses hard indices: "read slot 47." Hard indexing is not differentiable — you cannot compute gradient through "go to slot 47." Without gradients, end-to-end backpropagation fails.

NTMs fix this with soft addressing. Instead of pointing to one slot, controller outputs weight vector over all N slots — distribution saying "pay 60% attention to slot 12, 30% to slot 13, sprinkle rest." Reading from memory becomes weighted sum across rows: r = Σ wᵢ · Mᵢ. Writing works similarly — erase little from every row, weighted by attention, then add new content, also weighted. Every operation stays continuous and differentiable. Gradients flow through memory, addressing, controller, all way back to input.

Addressing mechanism is hard to visualize. NTM combines two addressing types: content-based — find memory row that looks most like query vector; sound familiar? It is attention — and location-based — shift focus left or right from last read, like moving read head along tape. Combination lets network do things like "find row containing X, then read next three rows in order." Very close to how traditional computer traverses data structure.

Results were striking. NTMs learned sequence copying, list sorting, and associative recall — all from input-output examples, with no explicit programming for those tasks. Network discovered algorithm itself and used memory as scratch space.

In 2016, same team released Differentiable Neural Computer (DNC), improved version. DNC added usage tracking — which memory slots are used or free — temporal links — order of writes, so sequences can be replayed — and dynamic memory allocation. It answered questions about family trees and navigated London Underground map by storing facts in memory and reasoning over them.

Limitation: NTMs and DNCs are hard to train. Soft addressing over large memories creates noisy optimization landscapes full of plateaus. They never scaled to sizes where transformers thrive. But intellectual contribution is deep: memory and computation can be separated in neural network, and boundary between "learned program" and "neural network" is blurrier than people thought.

The Differentiable Everything Insight

NTM trick — making memory operations differentiable so gradient descent can train them — is one case of bigger idea: differentiable programming.

Core insight: traditional programming writes explicit instructions — if-else, loops, array indexing. Traditional deep learning stacks layers and trains weights. Differentiable programming asks: what if whole program — control flow, memory access, data structures — were built from differentiable operations? Then you could train whole program end-to-end with gradient descent.

NTM external memory is differentiable data structure. Attention is differentiable dictionary lookup. Softmax router in Mixture of Experts, which comes next, is differentiable if-else. Positional encodings are differentiable representations of sequence position. Much of modern transformer is differentiable replacement for hard-coded algorithmic steps.

Yann LeCun called deep learning "a subset of differentiable programming" in 2018. Good call. Field trajectory is not only more layers or more data, though both help. Trajectory is making more of computational pipeline differentiable so optimizer can tune all of it.

Neural ODEs (Chen et al., 2018) made network depth differentiable — instead of discrete layers, hidden state evolves under learned differential equation, and you can choose how many "layers" to use at inference time. Differentiable rendering lets you train 3D scene representations from 2D images by making rendering pipeline differentiable. Differentiable sorting lets you include sorting operations in neural network and backpropagate through them.

Favorite part of differentiable programming: outside high-level explanations like this, nobody fully knows limits. Can whole database engine be differentiable? Compiler? Unknown. Each year, boundary moves.

Not Every Expert Needs to Show Up — Mixture of Experts

Back to transformers and practical problem. You want bigger model because bigger models learn better representations. But bigger also means slower — more parameters evaluated on every forward pass. Can huge model exist without paying full compute cost on every input?

Answer: conditional computation. Activate only model parts relevant to each input. Do not make every parameter work on every token. Let model choose, per token, which parameters to use.

Oldest version of this idea is Mixture of Experts (MoE), from Jacobs et al. in 1991 — long before transformers. Concept: instead of one big neural network, use several smaller "expert" networks plus "gating" network that chooses which experts to consult for each input. Gating network reads input and produces weights: "For this input, Expert 2 gets 70% weight and Expert 5 gets 30%."

In transformer context, MoE layer usually replaces feed-forward network (FFN) in each transformer block. Instead of one FFN for every token, you have, say, 64 expert FFNs plus router. Router examines each token's hidden state and sends it to top-k experts, often k = 1 or k = 2. Experts process token, and outputs are combined using router weights.

Now interesting part. Model can have 400 billion total parameters spread across experts, but each token activates maybe 12 billion — only parameters inside assigned experts. Training touches all parameters because different tokens go to different experts. Inference stays fast because each token evaluates only tiny fraction of full model.

GShard (Lepikhin et al., 2020) was Google's framework for training huge MoE models across thousands of TPUs. Each expert lives on different device, and router decides which device receives each token. Switch Transformer (Fedus et al., 2021) simplified routing to top-1: each token goes to exactly one expert. Sounds restrictive, but made system much easier to scale and worked very well. Switch Transformers reached trillion-parameter scale with training costs similar to much smaller dense models.

Mixtral (Mistral, 2023) brought MoE to open-source world with 8 experts and 2 active per token, showing approach works at practical scales on accessible hardware. GPT-4 is widely believed, though not officially confirmed, to use MoE architecture. Same for Google's Gemini.

Hardest MoE problem is not architecture — it is load balancing. If router sends all tokens to Expert 1 and none to Expert 7, parameters sit idle on some experts and hot expert gets overloaded. Models use auxiliary losses that penalize uneven expert utilization — imbalance tax. Setting tax well is art. Too weak, experts collapse and one expert handles everything. Too strong, router makes random useless assignments to stay balanced.

No one fully knows why sparse routing works this well. Intuition says different experts specialize in different token types — some become "syntax experts," others handle "rare words," others specialize in "numbers." But when researchers inspect trained routers, specialization is often messier and less interpretable than story suggests. It works. Exact reason still under investigation.

The Post-Attention Generation — RetNet and RWKV

Everything so far accepts one premise: attention — quadratic, every-token-looks-at-every-token mechanism — is right foundation for sequence modeling. Reformer tried to make attention cheaper. Flash Attention tried to make it faster on hardware. MoE tried to cut FFN cost. But attention stayed core.

What if attention is not answer?

RNN era had one beautiful property transformers threw away: constant memory during inference. LSTM processes sequence token by token and keeps fixed-size hidden state. Thousandth token costs same as first. Transformers must store and attend to whole KV cache, which grows with each new token. For long sequences and real-time applications, this hurts.

Two architectures — RetNet and RWKV — asked whether transformer-level quality can coexist with RNN-level efficiency. They attack from slightly different angles but share one insight: replace attention with mechanism that can be computed either in parallel, like attention for fast training, or recurrently, like RNN for efficient inference.

Retentive Networks (RetNet) (Sun et al., 2023) replace attention with retention mechanism. Key idea: exponentially decaying weighted sum over past tokens. At each position, model computes something like: output_t = Σ (γ^(t-i)) · (query_t · key_i) · value_i, where γ is decay factor between 0 and 1. Recent tokens contribute more; distant tokens fade exponentially.

Beautiful part of this formulation: it has dual form. You can write it as matrix multiplication — "parallel" form, like attention with triangular decay mask, great for GPU training — or as recurrence — "recurrent" form, great for inference, where fixed-size state updates token by token. Same math. Two implementations. Each fits its use case.

Decay factor γ acts like knob for memory horizon. Different attention heads can use different decay rates — some heads with high γ remember far back, others with low γ focus on recent context. Reminiscent of Clockwork RNNs, where groups updated at different rates, but derived from different mathematical starting point.

RWKV (Peng et al., 2023) takes more radical path. Name comes from key parameter matrices: R (receptance), W (weight), K (key), V (value). RWKV replaces attention with mechanism based on token-shift mixing and channel-wise weighted sums. Each token representation is learned mix of its own embedding and previous token's embedding — "token shift" — followed by recurrent update that looks like weighted moving average.

RWKV recurrence is per-channel: each hidden-state dimension updates independently. No matrix multiply in recurrent step — each channel decays and accumulates on its own. Per-step cost becomes O(d) instead of O(d²), where d is hidden dimension. Stacking multiple RWKV layers provides cross-channel mixing that recurrence alone lacks.

I still build intuition for when these alternatives truly beat transformers and when they trade expressiveness for speed. Empirical results look promising: RWKV models up to 14 billion parameters have been trained and show competitive perplexity with similar-size transformers, at fraction of inference cost for long sequences. RetNet claims "impossible triangle" of parallel training, efficient inference, and strong performance. But "competitive with transformers" keeps moving, and transformers keep improving too.

Most exciting part of RetNet and RWKV is not whether they beat transformers. They forced field to state clearly what attention actually gives — and what it costs. If exponential decay and channel-wise recurrences can approach attention-level performance, maybe quadratic all-pairs comparison does less work than assumed. Or maybe value of attention sits in few important interactions, and well-designed decay can approximate those at fraction of price. Either way, understanding of why transformers work is sharper because of these alternatives.
