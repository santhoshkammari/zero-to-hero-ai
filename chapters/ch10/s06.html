Interpretability & Efficient Architectures

Chapter 10: Sequence Models & Attention What models learn · Flash Attention · KV cache · Making it scale
TL;DR

Two things you need to know exist but don't need to master yet. Interpretability: we can peek inside transformers — visualize attention weights, probe what layers encode, even reverse-engineer circuits — but the tools are young and the conclusions are fragile. Efficient architectures: standard attention is O(n²) in sequence length, which breaks at long contexts. Flash Attention (exact, memory-efficient) is the dominant fix in practice. Sparse attention, linear attention, and KV cache tricks handle the rest. Most modern LLMs stack several of these together.

Part 1: What Is the Model Actually Doing?

You've trained a transformer. It works. But why does it work? What did it learn? Interpretability research tries to answer that, and it ranges from "easy to try, easy to misread" to "cutting-edge research that might eventually let us audit AI systems." Here's the landscape.

Attention Visualization

The most obvious thing to try: plot the attention weights. Each attention head produces an n×n matrix showing how much each token "looks at" every other token. You can visualize this as a heatmap or a set of arcs connecting tokens. When translating "the cat sat on the mat," you might see the decoder's attention lighting up on "cat" when generating "chat." Intuitive. Satisfying. And dangerously easy to over-interpret.

Attention ≠ Explanation

There's a famous debate here. Jain & Wallace (2019) showed that attention weights often don't correlate with gradient-based importance measures, and you can find completely different attention distributions that produce the same predictions. Wiegreffe & Pinter (2019) pushed back with nuance. The consensus: attention weights tell you where the model routed information, not what information mattered for the output. They're a useful debugging signal, not a faithful explanation. Treat them like X-rays, not like proofs.

Probing Classifiers

Probing answers a more precise question: what information is encoded in the model's internal representations? The recipe is simple:

  • Freeze the model — don't change any weights
  • Extract hidden states from a specific layer for your dataset
  • Train a small, simple classifier (usually linear) on those hidden states to predict some linguistic property
  • If the probe succeeds, that information is accessible in that layer's representations

Researchers have used this to map out what different layers encode. Typical findings: early layers capture surface features (word identity, POS tags), middle layers capture syntax (dependency relations, constituent structure), and later layers capture semantics (coreference, sentiment, entity types). It's like an MRI for neural networks — you can see which brain regions light up for different tasks.

The Probing Caveat

If your probe is too powerful (say, a deep MLP instead of a linear classifier), it might learn the property rather than detect it. A sufficiently complex probe can extract patterns from random noise. Keep probes simple — the whole point is that if a linear classifier can find the information, it must be represented in a fairly accessible way.

Mechanistic Interpretability

This is the ambitious frontier — trying to reverse-engineer what individual neurons, attention heads, and circuits (combinations of components) actually compute. The Anthropic team and researchers like Neel Nanda have made striking discoveries:

  • Induction heads: pairs of attention heads that implement a copy-paste pattern. If "Harry Potter" appeared earlier in the context, an induction head helps predict "Potter" after seeing "Harry" again. These form reliably during training and are a key mechanism for in-context learning.
  • Superposition: individual neurons don't encode single clean features. Instead, models pack more features than they have neurons by encoding them in overlapping, almost-orthogonal directions. One neuron might respond to both "academic writing" and "the color blue" — not a bug, but a compression trick.

Mechanistic interpretability is intellectually exciting and potentially critical for AI safety. It's also painstaking, manual, and doesn't yet scale to the largest models. Think of it as the particle physics of deep learning — important findings, but don't expect to use it in your next sprint.

Part 2: Making Attention Scale

Now for the engineering side. Standard self-attention computes a score between every pair of tokens. That's O(n²) in sequence length. Let's make the pain concrete:

Sequence Length Attention Entries Vibe Check
1,000 1 million Fine
8,000 64 million Manageable
32,000 1 billion Getting uncomfortable
100,000 10 billion GPU says no

Context length is capability — a model reading 4K tokens handles a conversation, one reading 128K can digest an entire codebase. Every technique below attacks that quadratic wall from a different angle.

Sparse Attention

The simplest idea: don't attend to everything. Most attention weights are near-zero anyway, so skip the pairs that don't matter. The patterns people use:

  • Local window: each token attends only to its w nearest neighbors. O(n·w) — linear in n. Information propagates across layers, so 32 layers × 512 window = reach across 16K tokens.
  • Strided: attend to every k-th token, giving a coarse global view.
  • Global tokens: designate a few tokens (like [CLS] or start-of-sequence) that attend to everything and that everything attends to. Cheap global information routing.
  • Random: randomly sample some attention connections. Surprisingly effective as a complement to local patterns.

Longformer combines local + global. BigBird adds random connections on top. Both achieve near-full-attention quality at a fraction of the cost for long documents.

Linear Attention

The math trick: standard attention computes softmax(QKT)V, which forces you to build the n×n matrix first. But if you replace softmax with a decomposable kernel — writing softmax(q·k) ≈ φ(q)·φ(k) — you can rearrange the multiplication order:

# Standard: (Q @ K.T) @ V  → build n×n matrix first, then multiply by V
# Linear:   Q @ (K.T @ V)  → build d×d matrix first (much smaller!), then multiply by Q

The associativity of matrix multiplication lets you compute KTV first — that's d×d instead of n×n, where d is the head dimension (64 or 128). Complexity drops to O(n·d²). The Performer and Random Feature Attention use this approach with different kernel approximations. The catch: the approximation isn't free — you lose some quality, especially on tasks where precise token-to-token attention patterns matter.

Flash Attention

This Is What Actually Won

Flash Attention is not an approximation. It computes exact standard attention — same math, same outputs, bit-for-bit identical. The trick is entirely about how the computation is organized on the GPU.

The problem it solves: to compute softmax(QKT)V naively, you materialize the full n×n attention matrix in GPU high-bandwidth memory (HBM). HBM is large but slow. GPU SRAM is tiny but fast. Flash Attention tiles the computation — it processes the attention matrix in small blocks that fit in SRAM, never writing the full n×n matrix to HBM. It uses an online softmax trick to compute exact softmax without needing all the values at once.

Results: 2–4× wall-clock speedup, dramatically less memory, and you can train with much longer sequences on the same hardware. Flash Attention (and its successor FlashAttention-2) is what actually runs under the hood of essentially every modern LLM. If you only remember one efficiency technique, make it this one.

Multi-Query & Grouped-Query Attention

Standard multi-head attention gives each head its own Q, K, and V projections. That means during autoregressive generation, you have to cache separate K and V tensors for every head. With 32 heads across 80 layers, the KV cache gets enormous.

  • Multi-Query Attention (MQA): all query heads share a single set of K and V heads. Slashes KV cache by the number of heads (e.g., 32×). Used in PaLM, Falcon.
  • Grouped-Query Attention (GQA): a middle ground — group query heads and share K/V within each group. 8 KV heads serving 32 query heads gives a 4× reduction. Used in Llama 2, Mistral, Gemma. This is the current sweet spot — most of MQA's memory savings with almost no quality loss.

KV Cache

During autoregressive generation (producing one token at a time), the model runs attention over the entire sequence at each step. But all the previous tokens' keys and values haven't changed — only the new token is new. So you cache the K and V vectors from all previous steps and only compute Q, K, V for the new token.

Without KV cache: generating token 1000 recomputes all 1000 keys and values. With KV cache: you compute 1 new key and 1 new value, then attend over the 1000 cached ones. The speedup is massive — it's why LLM inference is feasible at all. The tradeoff: memory scales linearly with sequence length × number of layers × number of KV heads × head dimension. For a 70B model at 128K context, the KV cache alone can consume tens of gigabytes. That's why MQA/GQA matter so much — they shrink the thing that actually dominates inference memory.

The Real Bottleneck

For serving LLMs in production, KV cache memory is often the binding constraint — not compute, not model weights, but the per-request cache for long conversations. Techniques like PagedAttention (used in vLLM) treat KV cache like virtual memory pages to reduce fragmentation. If you ever deploy LLMs at scale, you'll spend more time thinking about KV cache management than about model architecture.

Comparison: Efficient Attention Approaches

Approach Complexity Exact? Practical Status
Standard attention O(n²) Yes Baseline — still used for short sequences
Sparse (Longformer, BigBird) O(n·w) No (subset) Good for long docs; less common in LLMs
Linear (Performer) O(n·d²) No (approx) Research interest; quality gap remains
Flash Attention O(n²) math, O(n) memory Yes Dominant in practice — used everywhere
MQA / GQA Same compute, less memory Yes Standard in Llama 2, Mistral, Gemma
KV Cache O(1) per new token Yes Universal — every autoregressive model
