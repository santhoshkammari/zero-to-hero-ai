State Space Models & Mamba

Chapter 10: Sequence Models Beyond Attention Section 10 of 14 ~20 min read
TL;DR

Transformers are powerful but their attention mechanism scales quadratically with sequence length — doubling the context costs four times the compute. State Space Models (SSMs) offer a mathematically principled alternative rooted in control theory: they maintain a fixed-size hidden state that compresses the entire history, giving linear-time sequence processing. S4 showed SSMs could rival transformers on long-range benchmarks by cleverly switching between a convolution view (fast training) and a recurrence view (fast inference). Mamba took it further by making the state dynamics input-dependent — letting the model selectively remember or forget based on content — while using hardware-aware algorithms to stay fast on GPUs. The result: transformer-quality language modeling with linear scaling. The field is converging toward hybrids that mix SSM layers with sparse attention.

Here's the problem. Transformers work by letting every token attend to every other token. That's a beautiful idea — it means the model has direct access to all context at all times. But beauty has a price. If your sequence has n tokens, computing attention requires an n × n matrix. Double the context window from 4K to 8K tokens and you quadruple the memory and compute. Push to 100K tokens and you need roughly 625 times the resources of a 4K context. This quadratic cost is the fundamental bottleneck that prevents transformers from processing book-length documents, hour-long audio, or video frames at high resolution.

State Space Models attack this problem from a completely different angle. Instead of letting every token look at every other token, they maintain a compact hidden state — a fixed-size vector that summarizes everything the model has seen so far. Each new token updates this state and produces an output. The cost per token is constant, regardless of how long the sequence is. The total cost is linear in sequence length. The idea comes from control theory and signal processing, fields that have been modeling dynamical systems for decades. What changed recently is that researchers figured out how to make these models actually competitive with transformers on language tasks.

What Is a State Space Model?

A State Space Model (SSM) describes how a system evolves over time using two equations. The first is the state equation: dx/dt = Ax + Bu. The second is the output equation: y = Cx + Du. Here, u is the input signal (a token embedding at a given time), x is the hidden state (a vector of size N), and y is the output. The matrices A, B, C, and D are learnable parameters that define the system's dynamics.

If you've taken a controls or signals course, this looks familiar. If you haven't, here's the intuition. The state vector x is like compressed memory. It's a fixed-size summary of everything that's happened so far. The matrix A governs how the state evolves on its own — it controls what gets remembered and what fades. The matrix B controls how new input gets written into the state. The matrix C reads from the state to produce output. And D is a skip connection, letting the input flow directly to the output.

Think of it like a leaky bucket. Water (input) pours in through a faucet controlled by B. The bucket leaks at a rate controlled by A — old water slowly drains away. At any moment, the water level (the state) reflects a weighted mix of recent and past inputs, with newer inputs contributing more. The output is a reading of the water level, scaled by C. The bucket has finite capacity, so it can't store every detail — it has to compress. That compression is exactly what makes SSMs efficient: the state is always the same size, no matter how long the sequence.

The critical design question is how big the state vector should be and how the A matrix should be structured. A bigger state can remember more, but costs more to compute. The structure of A determines the nature of the memory: how quickly it decays, what frequency patterns it captures, whether it favors recent information or retains old information. Getting A right turns out to be the key breakthrough that separates SSMs that actually work from SSMs that are just elegant math.

From Continuous to Discrete

There's an immediate practical problem with the continuous-time formulation: we don't have continuous signals. We have sequences of discrete tokens — words, audio samples, pixels. We need to convert those continuous differential equations into discrete update rules that process one token at a time.

This process is called discretization, and the standard approach uses the zero-order hold (ZOH) method. The idea is simple: assume the input is constant between time steps. Under this assumption, you can solve the differential equation exactly and get a discrete recurrence:

x_k = Ā x_{k-1} + B̄ u_k

y_k = C x_k + D u_k

The bar notation means these are the discretized versions: Ā = exp(ΔA) and B̄ = (ΔA)⁻¹(exp(ΔA) − I) · ΔB, where Δ is the discretization step size. Don't worry about memorizing those formulas. The key point is that Δ controls the resolution of the discretization. A small Δ means fine-grained time steps, like sampling audio at 44kHz. A large Δ means coarse steps, like looking at one frame per second. The model can learn Δ, effectively learning how fast to "read" the input.

This discretization isn't just a technicality — it's what gives SSMs their dual nature, which we'll see in S4. The continuous formulation provides mathematical structure. The discrete formulation provides computational recipes. Both describe the same system.

Why Not Just Start Discrete?

You might wonder: why bother with continuous equations at all if we're going to discretize anyway? The continuous formulation gives us access to powerful tools from control theory and differential equations. In particular, it lets us reason about the system's frequency response, stability, and memory properties in a principled way. The discretization step Δ also becomes a learnable hyperparameter that controls the model's temporal resolution — something you don't get if you just write down a discrete recurrence from scratch.

The HiPPO Framework

Here's the problem that stumped early SSM research: if you initialize the A matrix randomly, the model forgets almost everything within a few dozen time steps. Long-range dependencies — the whole reason we're building these models — vanish before the model can learn to use them. The gradients either explode or vanish (the same problem that plagued vanilla RNNs).

The HiPPO (High-order Polynomial Projection Operators) framework solved this by answering a precise mathematical question: what A matrix gives optimal compression of input history? The key insight is to project the input history onto a polynomial basis. At every time step, the state vector holds the coefficients of a polynomial that best approximates the entire input history up to that point.

The most important variant is HiPPO-LegS (Legendre Scale), which uses Legendre polynomials. It has a remarkable property: it weights recent inputs and distant inputs roughly equally when measured on a logarithmic time scale. This means it gives the model a "telescoping" memory — high resolution for recent events, lower resolution for older events, but nothing is completely forgotten. Contrast this with an exponentially decaying memory (like a vanilla RNN), where anything older than a few dozen steps is effectively gone.

The HiPPO A matrix has a specific closed-form structure. For an N-dimensional state using Legendre polynomials, the entries are: A_{nk} = -(2n+1)^{1/2} (2k+1)^{1/2} if n > k, A_{nk} = -(n+1) if n = k, and 0 otherwise. You don't need to memorize this. What matters is that this isn't random — it's derived from the mathematics of optimal function approximation, and it gives SSMs the ability to remember over thousands of time steps.

S4: Structured State Spaces

Having a good A matrix from HiPPO is necessary but not sufficient. The naive discrete SSM recurrence — computing x_k = Ā x_{k-1} + B̄ u_k one step at a time — is inherently sequential. You can't parallelize it across time steps because each step depends on the previous one. This makes training painfully slow on GPUs, which thrive on parallel computation. Transformers, despite their quadratic cost, train fast because attention is a giant matrix multiplication that GPUs love.

S4 (Structured State Spaces for Sequence Modeling) solved this with a clever observation: if you unroll the recurrence across all time steps and simplify, the entire SSM computation is equivalent to a convolution. The output sequence y is the input sequence u convolved with a fixed kernel K̄ that can be precomputed from Ā, B̄, and C. Convolutions can be computed in O(n log n) time using the Fast Fourier Transform. That's the training trick: treat the SSM as a convolution and compute it in parallel.

But here's the beautiful part — at inference time, when you're generating tokens one at a time, you don't need the convolution view. You switch back to the recurrence view: x_k = Ā x_{k-1} + B̄ u_k. This is O(1) per step with O(N) memory (the state vector). No KV-cache growing linearly with context length. No recomputation. Just a fixed-size state that gets updated at each step. This dual nature — train as convolution, infer as recurrence — is what makes S4 special.

The remaining challenge was making the HiPPO A matrix computationally tractable. S4 introduced the DPLR (Diagonal Plus Low-Rank) parameterization: decompose A into a diagonal matrix plus a low-rank correction. This enables efficient computation of the convolution kernel via the Cauchy kernel trick. The details are deep linear algebra, but the payoff is that S4 matches or beats transformers on the Long Range Arena benchmark — tasks requiring models to reason over sequences of 1K-16K tokens.

# Discrete SSM step (single time step)
import torch

def ssm_step(A_bar, B_bar, C, D, x_prev, u):
    """
    A_bar: (N, N) discretized state matrix
    B_bar: (N, 1) discretized input matrix
    C:     (1, N) output matrix
    D:     (1, 1) feedthrough matrix
    x_prev: (N, 1) previous hidden state
    u:     scalar input at this time step
    """
    x_new = A_bar @ x_prev + B_bar * u    # state update
    y = C @ x_new + D * u                 # output
    return x_new, y

# Example: N=4 state, single step
N = 4
A_bar = torch.randn(N, N) * 0.1
B_bar = torch.randn(N, 1)
C = torch.randn(1, N)
D = torch.zeros(1, 1)

x = torch.zeros(N, 1)  # initial state
u = torch.tensor(1.0)  # one input token embedding (simplified)

x, y = ssm_step(A_bar, B_bar, C, D, x, u)
print(f"New state shape: {x.shape}, Output: {y.item():.4f}")

Mamba: Selective State Spaces

S4 proved that SSMs could work, but it had a fundamental limitation: the A, B, and C matrices were the same at every time step, regardless of the input. The system was time-invariant (or equivalently, Linear Time-Invariant, LTI). This means the model applied the same compression dynamics to every token. It couldn't decide to remember a keyword more carefully or forget a filler word faster. The convolution trick that made S4 parallelizable required this time-invariance — convolution only works with a fixed kernel.

Mamba broke through this limitation with a radical idea: make the parameters input-dependent. Specifically, Mamba makes B, C, and Δ (the discretization step size) all functions of the current input token. The matrix B becomes B(x_t), computed by a linear projection of the input. Same for C(x_t) and Δ(x_t). This is called selectivity: the model can select what to write into memory and what to read out, conditioned on the content it's currently seeing.

Why does selectivity matter so much? Consider a language model processing the sentence "The capital of France, which is known for its beautiful architecture and cuisine, is Paris." A time-invariant SSM treats every word with equal importance — "which", "is", "known", "for" all get written into the state with the same dynamics. A selective SSM can learn to crank up Δ (expand the time step) for filler words, effectively fast-forwarding through them, while shrinking Δ for important content words, sampling them at high resolution. It can modulate B to write "France" deeply into the state and barely write "which" at all. This content-dependent gating is analogous to what attention does — but with a fixed-size state instead of an ever-growing KV-cache.

But making parameters input-dependent destroys the convolution trick. You can't precompute a fixed kernel when the kernel changes at every time step. How do you train efficiently? Mamba's answer is a hardware-aware parallel scan algorithm. The parallel scan (also called parallel prefix sum) is a classic algorithm that computes all partial cumulative operations of a sequence in O(n) work and O(log n) depth on a parallel processor. Mamba implements this as a custom CUDA kernel that fuses the discretization, state update, and output computation into a single GPU kernel. It keeps the state in fast SRAM (on-chip memory) rather than slow HBM (high-bandwidth memory), avoiding the memory bandwidth bottleneck that plagues naive implementations.

# Selective SSM pseudocode (simplified Mamba forward pass)
import torch
import torch.nn as nn

class SelectiveSSM(nn.Module):
    def __init__(self, d_model, d_state):
        super().__init__()
        self.d_state = d_state
        # Input-dependent projections
        self.proj_B = nn.Linear(d_model, d_state, bias=False)
        self.proj_C = nn.Linear(d_model, d_state, bias=False)
        self.proj_delta = nn.Linear(d_model, d_model, bias=True)
        # A is NOT input-dependent (stays diagonal, learned as log)
        self.log_A = nn.Parameter(torch.randn(d_model, d_state))
        self.D = nn.Parameter(torch.ones(d_model))

    def forward(self, u):
        """
        u: (batch, seq_len, d_model)
        Returns: (batch, seq_len, d_model)
        """
        batch, seq_len, d = u.shape
        A = -torch.exp(self.log_A)           # (d, N) always negative for stability
        B = self.proj_B(u)                    # (batch, seq_len, N) -- input-dependent!
        C = self.proj_C(u)                    # (batch, seq_len, N) -- input-dependent!
        delta = torch.softplus(               # (batch, seq_len, d) -- input-dependent!
            self.proj_delta(u)
        )

        # Discretize: Ā = exp(Δ · A), B̄ = Δ · B (simplified Euler discretization)
        # In practice, Mamba uses a fused selective_scan CUDA kernel here
        # that avoids materializing the full (batch, seq_len, d, N) tensor

        outputs = []
        x = torch.zeros(batch, d, self.d_state, device=u.device)
        for k in range(seq_len):
            delta_k = delta[:, k, :].unsqueeze(-1)           # (batch, d, 1)
            A_bar = torch.exp(delta_k * A.unsqueeze(0))      # (batch, d, N)
            B_bar = delta_k * B[:, k, :].unsqueeze(1)        # (batch, d, N)

            x = A_bar * x + B_bar * u[:, k, :].unsqueeze(-1) # state update
            y = (x * C[:, k, :].unsqueeze(1)).sum(dim=-1)    # read output
            y = y + self.D * u[:, k, :]                      # skip connection
            outputs.append(y)

        return torch.stack(outputs, dim=1)  # (batch, seq_len, d)

The full Mamba architecture wraps this selective SSM into a block that also includes a 1D depthwise convolution (to capture local patterns), a SiLU gating mechanism (for controlling information flow), and a linear projection. Crucially, there is no attention mechanism anywhere. The architecture stacks these Mamba blocks like transformer layers, with layer normalization and residual connections. The result is a model that processes sequences in linear time and constant memory per step — a stark contrast to the quadratic attention matrix.

The Parallel Scan is Not Optional

You might look at the Python loop above and think Mamba is slow. The loop is for illustration only. In practice, Mamba uses a fused CUDA kernel that implements the parallel scan in O(n) work with O(log n) parallel depth, keeping the state in GPU SRAM. This hardware-aware implementation is critical — a naive Python loop would be 10-100× slower than the CUDA kernel. The algorithm is correct in both cases, but the constant factors matter enormously on real hardware.

Linear Attention Connection

It's worth stepping back and seeing where SSMs fit in the bigger picture. Standard attention computes softmax(QKT/√d)V. That QKT product creates an n × n matrix — the source of the quadratic cost. Linear attention replaces the softmax with a feature map φ and rearranges the computation: instead of φ(Q)(φ(K)TV), you compute φ(K)TV first, which is a d × d matrix (independent of sequence length), then multiply by φ(Q). This avoids ever materializing the n × n matrix.

Here's the deep connection: that d × d matrix in linear attention is essentially a state — a fixed-size summary of the sequence, updated incrementally as each new key-value pair arrives. This is exactly what an SSM's state vector does. Recent theoretical work on State Space Duality (SSD) has made this connection precise: structured SSMs and structured linear attention are mathematically equivalent under certain parameterizations. Mamba-2 exploits this duality directly, connecting the selective SSM framework with a structured form of multi-head attention.

The practical takeaway is that SSMs and linear attention are two sides of the same coin. Both maintain a fixed-size state that summarizes the context. Both achieve linear-time sequence processing. The difference is the lineage: SSMs come from control theory and use discretized differential equations, while linear attention comes from approximating the softmax kernel. They arrived at the same destination from different starting points.

Mamba vs Transformer: Honest Tradeoffs

Let's be clear-eyed about what Mamba does well and where it struggles. On the plus side: Mamba processes sequences in O(n) time and memory, compared to O(n²) for standard attention. This means Mamba can handle very long sequences — 100K tokens or more — without the memory explosion that transformers suffer. Inference is fast because each token only requires a constant-time state update, with no KV-cache to manage or grow. Throughput at generation time is roughly 5× higher than a comparable transformer at long context lengths.

But Mamba has weaknesses. The fixed-size state means it can't do "random access" to arbitrary past tokens. A transformer can attend to the 500th token from a 10,000 token context just as easily as the most recent token. Mamba can only access the past through whatever was compressed into the state. For tasks that require precise recall of specific details from the distant past — like "what was the third item on the list mentioned in paragraph two?" — Mamba can struggle. The state might not have retained that specific detail. This is a fundamental tradeoff: fixed-size memory gives you efficiency but loses exactness.

In-context learning presents another challenge. Transformers excel at few-shot prompting partly because attention can directly copy patterns from the examples to the current query. SSMs, with their compressed state, have a harder time with exact pattern matching and copying. Empirically, Mamba matches transformers on perplexity for language modeling but can lag behind on tasks requiring precise in-context retrieval.

The Hybrid Solution

The emerging consensus is that you don't have to choose one or the other. Models like Jamba (AI21 Labs) and Zamba (Zyphra) interleave Mamba layers with a small number of attention layers. The Mamba layers handle the bulk of sequence processing efficiently, while the attention layers provide the random-access recall that pure SSMs lack. Jamba uses a ratio of roughly 7:1 (Mamba to attention layers) and achieves strong performance with much lower memory than a pure transformer of the same size. This hybrid approach is likely where production architectures are heading.

Where does this leave us? Transformers still dominate in production, but the gap is closing fast. Mamba and its successors have proven that attention is not the only path to strong sequence modeling. For applications with very long contexts — document understanding, genomics, audio, video — SSMs already have a compelling advantage. For standard language modeling at moderate context lengths, transformers remain the safe default with the most mature tooling. The smart money is on hybrid architectures that take the best of both worlds: the efficiency of SSMs for most of the sequence processing, punctuated by attention where precise recall matters.

The deeper lesson here isn't about any specific architecture. It's that the attention mechanism — as revolutionary as it was — is a particular solution to the problem of modeling dependencies in sequences. State space models are a different solution, rooted in a different mathematical tradition, with different strengths and weaknesses. The field is at its healthiest when it has multiple competing paradigms. Competition drives innovation, and we're all better off because researchers asked: "What if we didn't need attention at all?"
