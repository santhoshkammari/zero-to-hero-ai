Sequence-to-Sequence

Chapter 10: Sequence Models & Attention 4 subtopics
TL;DR

When input and output sequences have different lengths — translation, summarization, dialogue — you need an encoder to read the source and a decoder to write the target. The encoder compresses the input into a context vector; the decoder unrolls from that vector one token at a time. Training uses teacher forcing (feed ground-truth tokens) for speed, but beam search and sampling strategies handle the messiness of generation at inference time.

Think about translating "The cat sat on the mat" into French: "Le chat s'est assis sur le tapis." Six English words become seven French words. The word order shifts. Some concepts that take one English word need multiple French words. You can't just map position 1 → position 1, position 2 → position 2. The input and output are different lengths, and the alignment between them is non-trivial. This is the core problem that sequence-to-sequence (seq2seq) models solve.

Translation isn't the only task with this shape. Summarization takes a long document and produces a short one. Dialogue systems take a user's message and generate a response of unpredictable length. Question answering takes a paragraph and a question and produces an answer span. Any time your output sequence length doesn't match your input sequence length, you're in seq2seq territory.

The Encoder-Decoder Architecture

The original seq2seq idea, published by Sutskever et al. in 2014, is elegant. Split the model into two halves: an encoder and a decoder. The encoder's job is to read the entire input sequence and compress it into a single fixed-length vector. The decoder's job is to take that vector and generate the output sequence, one token at a time.

Here's how it works concretely with our translation example. The encoder — typically an LSTM or GRU — processes "The", "cat", "sat", "on", "the", "mat" one token at a time. At each step it updates its hidden state. After processing the last token "mat", its final hidden state is a vector that (in theory) captures the meaning of the entire English sentence. This final hidden state is the context vector.

The decoder is a separate RNN. It receives the context vector as its initial hidden state. Then it starts generating: it takes a special start-of-sequence token as its first input, produces "Le" as its first output, then feeds "Le" back in as input to produce "chat", feeds "chat" back in to produce "s'est", and so on until it produces an end-of-sequence token. Each step, the decoder picks one token and feeds it forward.

import torch
import torch.nn as nn

class Encoder(nn.Module):
    def __init__(self, vocab_size, embed_dim, hidden_dim, num_layers=1):
        super().__init__()
        self.embedding = nn.Embedding(vocab_size, embed_dim)
        self.rnn = nn.LSTM(embed_dim, hidden_dim,
                           num_layers=num_layers, batch_first=True)

    def forward(self, src):
        # src: (batch, src_len)
        embedded = self.embedding(src)          # (batch, src_len, embed_dim)
        outputs, (hidden, cell) = self.rnn(embedded)
        # hidden: (num_layers, batch, hidden_dim) — the context vector
        return hidden, cell


class Decoder(nn.Module):
    def __init__(self, vocab_size, embed_dim, hidden_dim, num_layers=1):
        super().__init__()
        self.embedding = nn.Embedding(vocab_size, embed_dim)
        self.rnn = nn.LSTM(embed_dim, hidden_dim,
                           num_layers=num_layers, batch_first=True)
        self.fc_out = nn.Linear(hidden_dim, vocab_size)

    def forward(self, token, hidden, cell):
        # token: (batch,) — single token at current time step
        token = token.unsqueeze(1)              # (batch, 1)
        embedded = self.embedding(token)        # (batch, 1, embed_dim)
        output, (hidden, cell) = self.rnn(embedded, (hidden, cell))
        prediction = self.fc_out(output.squeeze(1))  # (batch, vocab_size)
        return prediction, hidden, cell


class Seq2Seq(nn.Module):
    def __init__(self, encoder, decoder, device):
        super().__init__()
        self.encoder = encoder
        self.decoder = decoder
        self.device = device

    def forward(self, src, trg, teacher_forcing_ratio=0.5):
        # src: (batch, src_len), trg: (batch, trg_len)
        batch_size = src.shape[0]
        trg_len = trg.shape[1]
        trg_vocab_size = self.decoder.fc_out.out_features

        outputs = torch.zeros(batch_size, trg_len, trg_vocab_size).to(self.device)
        hidden, cell = self.encoder(src)

        # First decoder input is the start-of-sequence token (index 1)
        input_token = trg[:, 0]

        for t in range(1, trg_len):
            prediction, hidden, cell = self.decoder(input_token, hidden, cell)
            outputs[:, t] = prediction
            # Teacher forcing: use ground-truth or model's own prediction
            if torch.rand(1).item() < teacher_forcing_ratio:
                input_token = trg[:, t]       # ground-truth token
            else:
                input_token = prediction.argmax(dim=1)  # model's best guess
        return outputs

Notice how the Seq2Seq model's forward pass loops through the target length. At each step it decides whether to feed the decoder the real next token or its own prediction. That decision is controlled by teacher_forcing_ratio, which we'll dig into shortly.

The Bottleneck Problem

There's a fundamental flaw in this design. The entire meaning of "The cat sat on the mat" gets squeezed into one fixed-size vector — say, 256 or 512 floats. For a short sentence, that's fine. But what about a paragraph? A legal document? A chapter of a novel? You're asking one vector to remember everything.

Imagine you had to read an entire book, close it, and then recite it from memory using only a single sticky note of reminders. That's the bottleneck problem. Information gets lost, especially details from the beginning of long sequences. The encoder's hidden state gets overwritten as it processes later tokens, and early information fades.

Empirically, researchers found that seq2seq performance degrades sharply on sentences longer than about 20-30 tokens. The model just can't pack enough signal into that one vector. This limitation directly motivated the invention of attention mechanisms, which we'll cover in the next section. Attention lets the decoder peek back at all of the encoder's hidden states — not just the final one — so it can focus on the relevant parts of the input at each decoding step.

Why Not Just Make the Vector Bigger?

You could increase the hidden dimension from 512 to 4096. But bigger vectors mean more parameters, slower training, and diminishing returns. The problem isn't the vector's size — it's the compression itself. A 1000-word paragraph has too much structure to capture in any single point, regardless of dimension. You need a fundamentally different approach, and that's attention.

Teacher Forcing

Training a seq2seq model without any tricks is painfully slow. Here's why: the decoder at step t uses its own prediction from step t-1 as input. Early in training, those predictions are garbage — essentially random tokens. So the decoder is trying to learn from inputs that are nonsensical, which makes the gradients noisy and convergence sluggish.

Teacher forcing is the fix. During training, instead of feeding the decoder its own (bad) prediction, you feed it the ground-truth token from the target sequence. Translating to "Le chat s'est assis sur le tapis"? At step 2, you feed the decoder "Le" regardless of what it actually predicted. At step 3, you feed it "chat". And so on. The decoder always gets perfect context, so it can focus on learning the mapping from correct context to the next correct token.

The speedup is dramatic — often 2-5× faster convergence. But there's a catch. During training the decoder never sees its own mistakes. It always gets perfect input. At inference time, there's no teacher. The decoder must eat its own cooking, and if it makes one wrong prediction, the error cascades. It's never practiced recovering from errors. This gap between training and inference is called exposure bias.

Exposure bias causes real problems. A translation model might produce fluent-sounding output for the first few tokens, then suddenly derail into incoherent text because one early mistake pushed it into a region of input space it never encountered during training.

Scheduled Sampling

Scheduled sampling is the pragmatic middle ground. Instead of always using teacher forcing or never using it, you start training with heavy teacher forcing and gradually reduce it. Early on, when the model's predictions are useless, you use ground-truth tokens most of the time. As training progresses and predictions improve, you increasingly let the model use its own output.

A common schedule is linear decay: start with a teacher forcing probability of 1.0 and decrease it by a small amount each epoch until you reach something like 0.25. Others use exponential decay or inverse sigmoid schedules. The key idea is the same — train with a curriculum that gently transitions from "always correct input" to "mostly your own predictions."

Scheduled sampling doesn't fully eliminate exposure bias, but it substantially reduces it. The model gets practice handling its own imperfect outputs while still getting enough ground-truth signal to learn efficiently.

Decoding Strategies

Once the model is trained, you need to actually generate output sequences. How you do this matters enormously. The decoder produces a probability distribution over the vocabulary at each time step. The question is: how do you pick from that distribution?

Greedy Decoding

The simplest approach: at each step, pick the token with the highest probability. Greedy decoding is fast — one forward pass per token, just take the argmax. But it's myopic. It doesn't consider that choosing the second-best token now might lead to a much better overall sequence later. Greedy decoding can get stuck in locally optimal but globally poor translations.

For our example: greedy decoding might pick "Le" (correct), then "chat" (correct), but then maybe "était" instead of "s'est" because "était" has slightly higher probability in isolation, even though "s'est assis" is the correct phrase as a whole.

Beam Search

Beam search is the standard workaround. Instead of keeping only the single best candidate, you keep the top k candidates at every step. This parameter k is called the beam width. At each time step, you expand all k candidates by one token, score the resulting k × V possibilities (where V is the vocabulary size), and keep only the top k. At the end, you pick the highest-scoring complete sequence.

Beam width is a knob. Beam width 1 is greedy decoding. Beam width 5 is typical for machine translation. Beam width 10-20 can improve quality but slows decoding proportionally. Beyond 20, improvements are usually negligible. The tradeoff is clear: wider beams explore more of the search space but cost more computation.

import torch
import torch.nn.functional as F

def beam_search(decoder, hidden, cell, start_token, end_token,
                max_len=50, beam_width=5):
    """
    Beam search decoding for a trained LSTM decoder.
    Returns the highest-scoring complete sequence.
    """
    # Each beam: (log_prob, token_sequence, hidden_state, cell_state)
    beams = [(0.0, [start_token], hidden, cell)]
    completed = []

    for _ in range(max_len):
        all_candidates = []

        for log_prob, seq, h, c in beams:
            last_token = torch.tensor([seq[-1]])
            prediction, new_h, new_c = decoder(last_token, h, c)
            log_probs = F.log_softmax(prediction, dim=-1).squeeze(0)

            # Expand this beam with top-k tokens
            topk_log_probs, topk_indices = log_probs.topk(beam_width)

            for i in range(beam_width):
                token = topk_indices[i].item()
                new_log_prob = log_prob + topk_log_probs[i].item()
                new_seq = seq + [token]

                if token == end_token:
                    # Normalize by length to avoid short-sequence bias
                    length_penalty = len(new_seq) ** 0.6
                    normalized_score = new_log_prob / length_penalty
                    completed.append((normalized_score, new_seq))
                else:
                    all_candidates.append(
                        (new_log_prob, new_seq, new_h, new_c)
                    )

        if not all_candidates:
            break

        # Keep top-k beams by log probability
        all_candidates.sort(key=lambda x: x[0], reverse=True)
        beams = all_candidates[:beam_width]

    # If no sequence completed, take the best partial beam
    if not completed:
        best = max(beams, key=lambda x: x[0])
        return best[1]

    # Return the highest-scoring completed sequence
    best = max(completed, key=lambda x: x[0])
    return best[1]

Length Normalization

Notice the length_penalty in the beam search code above. Without it, beam search has a systematic bias toward shorter sequences. Why? Sequence scores are products of probabilities (or sums of log-probabilities). Every additional token multiplies in another probability less than 1, which makes the total score smaller. A three-token translation will almost always outscore a seven-token translation, even if the longer one is correct.

Length normalization fixes this by dividing the log-probability score by a function of the sequence length. The simplest version divides by len(sequence). A softer version uses len(sequence) ^ α, where α is typically between 0.6 and 1.0. Google's Neural Machine Translation system popularized using α = 0.6. If α = 0, there's no normalization. If α = 1, you're computing the average log-probability per token. Values in between provide a gentler correction.

Top-K Sampling

Beam search works well for translation, where there's usually one correct output. But for open-ended generation — writing stories, dialogue, creative text — beam search produces text that's fluent but boring. It gravitates toward generic, high-probability phrases. "I don't know" is always more probable than a creative response.

Top-K sampling takes a different approach. At each step, instead of picking the best token, you sample randomly from the top k most probable tokens. You zero out everything outside the top k and renormalize the remaining probabilities. With k = 50, the model can choose from 50 reasonable options, introducing variety while avoiding the tail of truly bizarre tokens.

Nucleus (Top-P) Sampling

The problem with top-K is that k is fixed. Sometimes the model is very confident and puts 95% of its probability mass on two tokens — sampling from 50 tokens adds unnecessary noise. Other times the distribution is flat and 50 tokens isn't enough diversity. Nucleus sampling (also called top-p sampling) adapts dynamically. Instead of picking a fixed number of tokens, you pick the smallest set of tokens whose cumulative probability exceeds a threshold p (typically 0.9 or 0.95). When the model is confident, the nucleus is tiny. When it's uncertain, the nucleus expands.

In practice, top-p sampling produces more natural, varied text than beam search or top-K for open-ended generation. Most modern language model APIs expose a top_p parameter for exactly this reason.

Which Decoding Strategy When?

Machine translation and summarization: beam search with length normalization (beam width 4-6). You want the single most accurate output. Open-ended dialogue and creative writing: top-p sampling (p = 0.9-0.95) with a temperature parameter to control randomness. You want diverse, interesting output. Greedy decoding: only when speed matters more than quality, such as real-time autocomplete.

Applications

Machine Translation

The original killer app. Seq2seq with attention (and later, Transformers) powers Google Translate, DeepL, and every modern translation system. The encoder reads the source language; the decoder writes the target language. Billions of parallel sentence pairs scraped from the web serve as training data. Even today, the encoder-decoder structure persists — the Transformer is still an encoder-decoder model at its core.

Text Summarization

Feed in an article, get out a summary. Abstractive summarization uses a seq2seq model to generate new sentences rather than just extracting existing ones. The encoder reads the full document; the decoder produces a condensed version. The challenge is that summaries must be factually consistent with the source — a problem called hallucination that remains actively researched.

Dialogue Systems

Chatbots and virtual assistants are fundamentally seq2seq. The user's message is the input sequence; the response is the output sequence. Early dialogue systems used vanilla seq2seq and produced generic responses ("I don't know", "That's interesting"). Modern systems layer in retrieval, persona conditioning, and reinforcement learning from human feedback, but the underlying generation mechanism is still a decoder producing tokens one at a time.

The seq2seq framework is one of those ideas that seems almost too simple — split reading from writing, compress, then expand. Its limitations (the bottleneck problem, exposure bias) are real, but each limitation spawned a targeted solution. Attention mechanisms, which we cover next, address the bottleneck. Scheduled sampling addresses exposure bias. And the encoder-decoder structure itself proved so useful that it survived the transition from RNNs to Transformers. The architecture changed; the fundamental idea didn't.
