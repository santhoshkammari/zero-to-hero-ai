Transformer Architecture

Chapter 10: Sequence Models & Attention Section 7 of 12 ~20 min read
TL;DR

The Transformer replaces recurrence with self-attention, letting every token look at every other token in parallel. It stacks encoder blocks (self-attention + feed-forward) and decoder blocks (masked self-attention + cross-attention + feed-forward), connected by residual connections and layer normalization. Positional encodings inject word-order information since attention itself is order-blind. The architecture is embarrassingly parallel during training, scales to billions of parameters, and is the backbone of GPT, BERT, LLaMA, and virtually every frontier model today. Understand it deeply — everything else in modern AI is a footnote to this design.

In 2017, a team at Google published "Attention Is All You Need," and quietly detonated the old order. Before that paper, sequence modeling meant recurrent neural networks — LSTMs, GRUs — where tokens were processed one at a time, left to right. Training was slow because you couldn't parallelize across time steps. Long-range dependencies were fragile because information had to survive hundreds of sequential hops. The Transformer blew that away by using attention as the only mechanism for relating tokens to each other. No recurrence. No convolutions. Just attention, feed-forward layers, and some clever wiring. It was simpler than what came before, yet dramatically more powerful. Within two years it had conquered machine translation, language modeling, speech recognition, and computer vision. If you learn one architecture in deep learning, make it this one.

Let's build it from the ground up, starting with a problem the Transformer creates for itself: it has no idea what order your words are in.

Positional Encoding

Why do we need positional encoding at all? Consider what self-attention actually does: for each token, it computes a weighted sum of all other tokens based on similarity scores. Nothing in that computation depends on where a token sits in the sequence. If you shuffled the input tokens, attention would produce exactly the same outputs (just in a different order). Attention is permutation-equivariant. That's a problem because "the dog bit the man" and "the man bit the dog" have the same set of tokens but very different meanings. We need to inject position information somehow.

Sinusoidal Encoding

The original Transformer used sinusoidal positional encoding. The idea is elegant: for each position in the sequence, generate a vector of the same dimension as the token embedding, using sine and cosine functions at different frequencies. Then add this position vector to the token embedding before feeding it into the Transformer.

The formula is: for position pos and dimension i, use sin(pos / 10000^(2i/d_model)) for even dimensions and cos(pos / 10000^(2i/d_model)) for odd dimensions. Each dimension oscillates at a different frequency — the first dimensions change rapidly (high frequency), the last dimensions change slowly (low frequency). The intuition is like binary counting: the lowest bit flips every step, the next bit flips every two steps, the next every four, and so on. Each position gets a unique pattern, and the model can learn to decode relative positions from these patterns because PE(pos + k) can be expressed as a linear function of PE(pos) for any fixed offset k.

import torch
import math

def sinusoidal_positional_encoding(max_len, d_model):
    """Generate sinusoidal positional encodings.

    Returns a (max_len, d_model) tensor where each row is the
    position vector for that sequence position.
    """
    pe = torch.zeros(max_len, d_model)
    position = torch.arange(0, max_len, dtype=torch.float).unsqueeze(1)

    # Compute the division term: 10000^(2i / d_model)
    div_term = torch.exp(
        torch.arange(0, d_model, 2).float() * (-math.log(10000.0) / d_model)
    )

    pe[:, 0::2] = torch.sin(position * div_term)  # even dims
    pe[:, 1::2] = torch.cos(position * div_term)  # odd dims
    return pe

# Example: 100 positions, 512-dimensional model
pe = sinusoidal_positional_encoding(100, 512)
print(pe.shape)  # torch.Size([100, 512])
print(pe[0, :8])  # first position, first 8 dims

Sinusoidal encodings have a nice property: they're deterministic and require no training. You can also handle sequences longer than anything seen during training, since the formula works for any position. The downside is that they're fixed — the model can't learn a better positional representation if one exists.

Learned Positional Embeddings

Learned positional embeddings take a simpler approach: just make the position vectors trainable parameters. You create a lookup table of shape (max_seq_len, d_model), initialize it randomly, and let backpropagation figure out the best position representations. This is what BERT and GPT-2 use. It's dead simple — just an nn.Embedding layer indexed by position.

The tradeoff is clear. Learned embeddings can capture position patterns that sinusoids might miss, and they consistently perform as well or better when the training data is plentiful. But they can't extrapolate: if you trained with a max length of 512, position 513 has no embedding. You'd need to retrain or fine-tune to support longer sequences. For many practical applications this is fine — you know your max context length ahead of time.

RoPE (Rotary Position Embeddings)

Rotary Position Embeddings (RoPE), introduced by Su et al. in 2021, are the position encoding of choice for most modern LLMs including LLaMA, Mistral, and Qwen. The insight is beautiful: instead of adding position information to the embeddings, you rotate the query and key vectors in attention by an angle proportional to their position. Pairs of dimensions are treated as 2D coordinates and rotated by pos × θ_i, where θ_i varies across dimension pairs (using the same geometric frequency schedule as sinusoidal encodings).

Why is rotation clever? When you compute the dot product between a rotated query at position m and a rotated key at position n, the rotation angles partially cancel, and the result depends only on the relative distance m - n. This means RoPE naturally encodes relative position without explicitly computing it. It also extrapolates better to unseen sequence lengths than learned embeddings because the rotation operation is well-defined for any position. In practice, people extend RoPE to longer contexts by scaling the frequency bases — techniques like NTK-aware scaling and YaRN are all modifications to RoPE's frequency schedule.

ALiBi (Attention with Linear Biases)

ALiBi, from Press et al. (2022), takes the most radical approach: no positional embedding at all. Instead, you add a linear bias to the attention scores before the softmax. The bias is simply -m × |i - j|, where m is a head-specific slope and |i - j| is the distance between token positions. Closer tokens get a small penalty (or none), distant tokens get a large penalty. Each attention head uses a different slope m, set by a fixed geometric sequence — no learning required.

ALiBi is the simplest method conceptually and has outstanding length generalization. A model trained on sequences of 1024 tokens can often handle 2048 or even 4096 at inference time with minimal degradation. The downside is that the linear penalty is a strong inductive bias — it strictly favors local attention. For tasks requiring truly global attention patterns (like some code-understanding tasks), RoPE may be more flexible. In practice, ALiBi was popular for a while (used in BLOOM and some early LLaMA variants) but RoPE has become the more common choice in frontier models.

Position Encoding Cheat Sheet

Sinusoidal: deterministic, no training, decent extrapolation. Learned: simple, slightly better in-distribution, can't extrapolate. RoPE: relative position via rotation, good extrapolation, used in LLaMA/Mistral. ALiBi: no embedding at all, just bias in attention scores, best length generalization. When in doubt for a new project, start with RoPE — it's the most battle-tested option as of 2024.

Encoder Block

The encoder's job is to read the entire input sequence and build a rich representation of each token that accounts for context from every other token. An encoder block has three sub-components, stacked in sequence: multi-head self-attention, a position-wise feed-forward network, and residual connections with layer normalization wrapping each of the first two. Let's walk through each.

Multi-Head Self-Attention

Self-attention lets every token in a sequence attend to every other token (including itself). For each token, we compute three vectors: a query (what am I looking for?), a key (what do I contain?), and a value (what information do I provide if matched?). These come from three separate linear projections of the input embedding. You compute attention scores by taking the dot product of each query with all keys, scale by 1 / √d_k to prevent the dot products from growing too large, apply softmax to get weights, and multiply by the values.

Why scale by √d_k? When the dimension d_k is large, the dot products get large in magnitude, pushing the softmax into regions where its gradients are tiny. Dividing by √d_k keeps the variance of the dot products around 1, which keeps the softmax in a well-behaved regime. It's a small detail that makes a big difference in training stability.

Multi-head attention runs several attention computations in parallel, each with its own Q/K/V projection matrices. If the model dimension is 512 and you use 8 heads, each head operates in a 64-dimensional subspace. The outputs of all heads are concatenated and projected back to the full model dimension. Why multiple heads? Each head can learn to attend to different types of relationships — one head might focus on syntactic dependencies, another on semantic similarity, another on positional proximity. The diversity of attention patterns gives the model much more expressive power than a single attention pass.

import torch
import torch.nn as nn
import torch.nn.functional as F
import math

class MultiHeadSelfAttention(nn.Module):
    def __init__(self, d_model=512, n_heads=8):
        super().__init__()
        assert d_model % n_heads == 0
        self.d_model = d_model
        self.n_heads = n_heads
        self.d_k = d_model // n_heads

        self.W_q = nn.Linear(d_model, d_model)
        self.W_k = nn.Linear(d_model, d_model)
        self.W_v = nn.Linear(d_model, d_model)
        self.W_o = nn.Linear(d_model, d_model)

    def forward(self, x, mask=None):
        batch, seq_len, _ = x.shape

        # Project to Q, K, V and reshape for multi-head
        Q = self.W_q(x).view(batch, seq_len, self.n_heads, self.d_k).transpose(1, 2)
        K = self.W_k(x).view(batch, seq_len, self.n_heads, self.d_k).transpose(1, 2)
        V = self.W_v(x).view(batch, seq_len, self.n_heads, self.d_k).transpose(1, 2)
        # Shape: (batch, n_heads, seq_len, d_k)

        # Scaled dot-product attention
        scores = torch.matmul(Q, K.transpose(-2, -1)) / math.sqrt(self.d_k)
        if mask is not None:
            scores = scores.masked_fill(mask == 0, float('-inf'))
        attn_weights = F.softmax(scores, dim=-1)

        # Weighted sum of values
        context = torch.matmul(attn_weights, V)

        # Concatenate heads and project
        context = context.transpose(1, 2).contiguous().view(batch, seq_len, self.d_model)
        return self.W_o(context)

Position-Wise Feed-Forward Network

After attention, each token's representation passes through a position-wise feed-forward network (FFN). "Position-wise" means the same two-layer network is applied independently to each token — there's no interaction between positions in this sub-layer (that already happened in attention). The FFN typically expands the dimension by a factor of 4, applies a nonlinearity, then projects back down: FFN(x) = W_2 · ReLU(W_1 · x + b_1) + b_2, where W_1 has shape (d_model, 4 × d_model) and W_2 has shape (4 × d_model, d_model).

Why expand and then contract? The expansion creates a high-dimensional space where the model can represent complex feature combinations, and the contraction squeezes it back to the model dimension. Recent research suggests that the FFN layers are where factual knowledge is primarily stored — "Rome is the capital of Italy" is encoded in specific FFN weight patterns more than in attention weights. Modern variants replace ReLU with GELU or SwiGLU (used in LLaMA), which tend to give slightly better performance.

Add & Norm (Residual Connections + LayerNorm)

Each sub-layer (attention and FFN) is wrapped with a residual connection followed by layer normalization. The pattern is: output = LayerNorm(x + SubLayer(x)). The residual connection (x + SubLayer(x)) means the sub-layer only needs to learn the delta — what to add to the existing representation — rather than reconstructing the entire representation from scratch. This makes optimization dramatically easier and prevents vanishing gradients in deep stacks.

Layer normalization normalizes across the feature dimension for each token independently. Unlike batch normalization (which normalizes across the batch dimension), layer norm doesn't depend on batch statistics, making it stable for variable-length sequences and small batch sizes. It rescales the activations so that each token's representation has zero mean and unit variance, then applies learned scale and shift parameters.

Stacking N Encoder Blocks

The original Transformer uses N = 6 encoder blocks stacked on top of each other. Each block takes the full sequence of token representations, refines them through attention and feed-forward processing, and passes the result to the next block. Early blocks tend to capture local syntactic patterns. Later blocks capture more abstract, long-range semantic relationships. BERT-base uses 12 encoder blocks; BERT-large uses 24. Each additional block adds expressive power but also computational cost. The output of the final encoder block is the contextualized representation of the input — every token's vector now encodes information about the entire sequence.

Decoder Block

The decoder generates output one token at a time, and it needs two kinds of context: what it has already generated, and what the input says. This gives the decoder block three sub-layers instead of the encoder's two.

Masked Self-Attention

The first sub-layer is self-attention over the decoder's own previous outputs, but with a critical constraint: masked self-attention prevents each position from attending to subsequent positions. When generating token 5, the model can only look at tokens 1 through 5, not tokens 6 onward (which haven't been generated yet). This is enforced with a causal mask — a lower-triangular matrix of ones. Positions that correspond to zeros (upper triangle) get their attention scores set to negative infinity before softmax, which drives their weights to zero.

During training, you process all positions simultaneously using teacher forcing (feeding the ground-truth sequence), but the mask ensures each position can only "see" the past. This preserves the autoregressive property while still allowing parallel computation across all positions — a huge speed advantage over RNNs, which must literally run sequentially.

Cross-Attention

Cross-attention is the bridge between encoder and decoder. The queries come from the decoder (what does the decoder need to know?), but the keys and values come from the encoder's output (what information is available from the input?). This is how a translation model works: the encoder processes "Le chat est sur le tapis," the decoder is generating "The cat is on the" and asks cross-attention "what should the next word be?" The attention mechanism routes the decoder's query to the relevant encoder representations — in this case, the representation of "tapis" — and retrieves the information needed to produce "mat."

Cross-attention uses the exact same scaled dot-product mechanism as self-attention. The only difference is where Q, K, and V come from. In self-attention, all three come from the same sequence. In cross-attention, Q comes from the decoder and K/V come from the encoder. This asymmetry is the whole trick.

Auto-Regressive Generation

At inference time, generation is a loop. Start with a special start-of-sequence token. Feed it through the decoder. The output is a probability distribution over the vocabulary. Sample or pick the most likely token. Append it to the decoder input. Repeat until you generate an end-of-sequence token or hit a maximum length. Each step requires a full forward pass through the decoder (though KV-caching avoids redundant computation for previous tokens).

This sequential generation is the biggest bottleneck in Transformer inference. Training is parallelizable because you have the ground truth and can process all positions at once. But generation is inherently sequential — you can't generate token 10 until you've committed to tokens 1 through 9. Techniques like speculative decoding, where a small draft model proposes multiple tokens and a large model verifies them in one pass, help mitigate this cost.

Encoder-Only vs. Decoder-Only vs. Encoder-Decoder

The original Transformer has both encoder and decoder. But most modern models use only one half. BERT is encoder-only (great for understanding tasks like classification and NER). GPT is decoder-only (great for generation). T5 and BART use the full encoder-decoder. Decoder-only models have dominated since GPT-3 because they unify understanding and generation under a single paradigm. If someone says "transformer" in 2024, they probably mean a decoder-only model with causal attention.

Residual Connections Deep Dive

Residual connections are so simple that it's easy to underestimate how important they are. Without them, deep transformers simply don't train. Let's look at why.

A residual connection (or skip connection) means the output of a layer is x + f(x) rather than just f(x). During backpropagation, the gradient flows through two paths: through the layer's computations (which may shrink the gradient) and directly through the skip connection (which preserves the gradient perfectly). The skip path creates a gradient highway — even if the layer's gradients vanish or explode, the identity path keeps information flowing. This is why ResNets trained with 152 layers when previous architectures collapsed at 30.

There's a deeper way to think about this. Imagine the token representations as a residual stream — a running sum that flows through the entire Transformer. Each attention layer and each FFN layer reads from this stream, computes something, and writes an additive update back to the stream. The stream carries the original token embedding all the way through, and each layer simply contributes its piece. This "residual stream" view, popularized by the mechanistic interpretability community, makes it much easier to reason about what each layer does: it's adding information, not transforming the entire representation.

This view also explains why you can sometimes delete entire layers from a trained Transformer and still get reasonable outputs. If each layer is adding a small delta to a running sum, removing one delta degrades performance but doesn't destroy it. The stream retains the contributions of all other layers. Try removing a layer from a network without residual connections and the outputs will be nonsense.

Pre-Norm vs Post-Norm

The original Transformer paper places layer normalization after the residual addition: output = LayerNorm(x + SubLayer(x)). This is called Post-Norm. It's what Vaswani et al. used, and it's mathematically clean. But it's hard to train for deep models. The gradients through the normalization layer can become unstable, and you often need learning rate warmup — a hack where you start with a tiny learning rate and gradually increase it — to prevent early training from diverging.

Pre-Norm flips the order: output = x + SubLayer(LayerNorm(x)). The normalization happens before the sub-layer, so the residual connection is a clean addition of the raw input and the sub-layer output. This seemingly minor change has a dramatic effect on training stability. Gradients flow more cleanly through the skip connection because they don't have to pass through the normalization. Pre-Norm models train reliably without warmup, even at large depth.

import torch.nn as nn

class PostNormBlock(nn.Module):
    """Original Transformer: normalize AFTER residual add."""
    def __init__(self, d_model, sublayer):
        super().__init__()
        self.sublayer = sublayer
        self.norm = nn.LayerNorm(d_model)

    def forward(self, x):
        return self.norm(x + self.sublayer(x))   # norm wraps the sum


class PreNormBlock(nn.Module):
    """GPT-2 style: normalize BEFORE the sublayer."""
    def __init__(self, d_model, sublayer):
        super().__init__()
        self.sublayer = sublayer
        self.norm = nn.LayerNorm(d_model)

    def forward(self, x):
        return x + self.sublayer(self.norm(x))    # clean skip connection

The tradeoff is subtle. Post-Norm can sometimes achieve slightly higher peak performance when you can get it to converge (the normalization after addition acts as a gentle regularizer). Pre-Norm is more forgiving and almost universally preferred in practice. GPT-2, GPT-3, LLaMA, and most modern LLMs use Pre-Norm. Some recent work (like DeepNorm from Microsoft) tries to get the best of both worlds by using Post-Norm with a carefully tuned residual scaling factor, but Pre-Norm remains the safe default for new projects.

RMSNorm: The Modern Default

Many recent models (LLaMA, Mistral, Gemma) have replaced LayerNorm with RMSNorm (Root Mean Square Normalization). RMSNorm drops the mean-centering step and only divides by the root mean square of the activations. It's computationally cheaper (one fewer reduction operation), and empirically it works just as well. If you're starting a new Transformer project today, Pre-Norm with RMSNorm is the standard recipe.

Key Properties

Let's step back and look at the Transformer as a system. Several properties explain its dominance.

Parallelizable Training

Unlike RNNs, where you must process token 1 before token 2 before token 3, the Transformer processes all tokens simultaneously during training. Self-attention computes all pairwise interactions in one matrix multiplication. The FFN applies the same network to all positions in parallel. This maps beautifully onto GPU hardware, which excels at large matrix operations. A Transformer training step on a 1024-token sequence takes roughly the same wall-clock time regardless of sequence length (up to memory limits), while an RNN takes time proportional to the length. This parallelism is why transformers can be trained on trillion-token datasets in weeks, while RNNs of equivalent capacity would take years.

O(n²) Attention Complexity

The attention matrix is n × n where n is the sequence length. Computing and storing it costs O(n²) in both time and memory. For a 2048-token sequence, that's about 4 million entries per head per layer — manageable. For a 128,000-token sequence (GPT-4's context window), that's 16 billion entries. This quadratic scaling is the Transformer's Achilles' heel. A mountain of research aims to fix it: sparse attention (attend only to nearby tokens and a few distant ones), linear attention (approximate softmax with kernels), FlashAttention (use GPU memory hierarchies to reduce the memory cost without approximation), and ring attention (distribute the sequence across multiple GPUs). FlashAttention, in particular, has become standard — it computes exact attention but tiles the computation to minimize slow GPU memory reads, giving a 2-4x speedup in practice.

Long-Range Dependencies

In an RNN, information from token 1 must survive a chain of N sequential operations to influence token N. Each operation is a potential bottleneck where information can be lost. In a Transformer, token 1 can attend directly to token N in a single attention step — it's one hop, not N hops. This direct connection is why Transformers handle long documents, long code files, and multi-turn conversations far better than RNNs ever could. The bottleneck shifts from information propagation to the model's capacity to meaningfully attend to long contexts, which is a much more tractable problem.

Architecture Overview

Here is what the full Transformer looks like, end to end. The input tokens pass through a token embedding layer that maps each token ID to a dense vector. Positional encodings are added to these embeddings. The result feeds into a stack of N encoder blocks (each containing multi-head self-attention → add & norm → feed-forward → add & norm). The encoder output — a sequence of contextualized vectors — is then handed to the decoder. On the decoder side, output tokens (shifted right by one position) pass through their own embedding and positional encoding, then through N decoder blocks (each containing masked self-attention → add & norm → cross-attention over encoder output → add & norm → feed-forward → add & norm). The final decoder output is projected through a linear layer to vocabulary size and passed through softmax to produce next-token probabilities. In decoder-only models like GPT, the entire encoder side and cross-attention are removed — you just have the decoder stack with causal masking and no cross-attention.

Putting It Together: A Minimal Transformer Block

Let's wire up a single Transformer encoder block in PyTorch to make everything concrete. This combines multi-head attention, the feed-forward network, residual connections, and Pre-Norm into one unit.

import torch
import torch.nn as nn

class TransformerEncoderBlock(nn.Module):
    """A single Pre-Norm Transformer encoder block."""
    def __init__(self, d_model=512, n_heads=8, d_ff=2048, dropout=0.1):
        super().__init__()
        self.norm1 = nn.LayerNorm(d_model)
        self.attn = nn.MultiheadAttention(
            d_model, n_heads, dropout=dropout, batch_first=True
        )
        self.norm2 = nn.LayerNorm(d_model)
        self.ffn = nn.Sequential(
            nn.Linear(d_model, d_ff),
            nn.GELU(),
            nn.Dropout(dropout),
            nn.Linear(d_ff, d_model),
            nn.Dropout(dropout),
        )

    def forward(self, x, mask=None):
        # Pre-Norm self-attention with residual
        normed = self.norm1(x)
        attn_out, _ = self.attn(normed, normed, normed, attn_mask=mask)
        x = x + attn_out

        # Pre-Norm feed-forward with residual
        x = x + self.ffn(self.norm2(x))
        return x


# Stack 6 encoder blocks
class TransformerEncoder(nn.Module):
    def __init__(self, vocab_size, d_model=512, n_heads=8, n_layers=6,
                 d_ff=2048, max_len=512, dropout=0.1):
        super().__init__()
        self.tok_emb = nn.Embedding(vocab_size, d_model)
        self.pos_emb = nn.Embedding(max_len, d_model)  # learned positions
        self.layers = nn.ModuleList([
            TransformerEncoderBlock(d_model, n_heads, d_ff, dropout)
            for _ in range(n_layers)
        ])
        self.final_norm = nn.LayerNorm(d_model)
        self.dropout = nn.Dropout(dropout)

    def forward(self, token_ids):
        seq_len = token_ids.size(1)
        positions = torch.arange(seq_len, device=token_ids.device)

        x = self.tok_emb(token_ids) + self.pos_emb(positions)
        x = self.dropout(x)

        for layer in self.layers:
            x = layer(x)

        return self.final_norm(x)

# Quick test
encoder = TransformerEncoder(vocab_size=30000)
dummy_input = torch.randint(0, 30000, (2, 128))  # batch=2, seq_len=128
output = encoder(dummy_input)
print(output.shape)  # torch.Size([2, 128, 512])

That's a complete, runnable Transformer encoder. The decoder would look similar but with an additional masked self-attention layer and a cross-attention layer between the self-attention and FFN. In practice, most people use nn.TransformerEncoder or the HuggingFace transformers library rather than writing this from scratch. But understanding every line here means you'll never be confused when reading model code or debugging training issues.

Common Pitfalls

Three mistakes that bite beginners: (1) Forgetting to scale attention by √d_k — training will appear to work but converge to a much worse solution because softmax saturates. (2) Using Post-Norm without learning rate warmup — the first few training steps will produce NaNs. (3) Not applying the causal mask correctly in the decoder — your model will "cheat" by looking at future tokens during training and produce garbage at inference time. Always verify your mask by checking that attention weights form a lower-triangular pattern for causal models.

The Transformer is one of those rare designs that is simultaneously simple enough to explain over coffee and deep enough to build an entire industry on. Master the pieces — attention, FFN, residuals, normalization, position encoding — and you'll have the vocabulary to understand every major AI model published in the last seven years. The details change (SwiGLU instead of ReLU, RoPE instead of sinusoidal, grouped-query attention instead of full multi-head), but the skeleton is always the same. Learn it once, and you'll recognize it everywhere.
