BERT & GPT: The Two Paradigms

Same Transformer base. One choice — which tokens attend where — splits models. Split sets training objective, downstream use, scaling behavior, generation-vs-understanding divide. This section traces both.

BERT and GPT both arrived in 2018, built on Transformer from 2017's "Attention Is All You Need" paper. Same engine, different machine. BERT became workhorse for understanding tasks — classification, named entity recognition, question answering. GPT became engine for text generation, chatbots, later large language model revolution. Together, they define two modern NLP paradigms.

The Fork in the Road

BERT and GPT start with same Transformer self-attention mechanism. Query, key, value — machinery from previous section. Split comes from one small design choice with huge effect: which tokens can attend to which?

Let every token see every other token — past and future, left and right — get BERT. Restrict each token to tokens before it — get GPT. That fork. Training objectives, downstream uses, scaling behavior, generation-vs-understanding divide all flow from attention mask choice.

Small structural change, huge model split. Feels like detective vs novelist split by reading case files forward or backward. Analogy fits better than expected. Now see why.

A Tiny Language for a Big Idea

Need concrete example for whole section. Build one.

Imagine movie review classifier. Whole training corpus has five reviews:

Review 1: "The movie was dark but the ending was bright"     → mixed
Review 2: "The acting felt flat and lifeless"                 → negative
Review 3: "A bright and moving performance"                   → positive
Review 4: "The dark theater made the movie feel immersive"    → positive
Review 5: "Flat acting ruined an otherwise bright script"     → negative

Notice "dark" in reviews 1 and 4. In review 1, "dark" describes movie tone — probably negative. In review 4, "dark" describes theater — neutral, maybe positive in context. "bright" does same trick: literal brightness in review 4, quality in reviews 1, 3, and 5. "flat" too — about acting (bad) or something else?

Model reading these reviews must figure out what each word means in context. Key question: how much context can it see?

That question marks BERT/GPT split. Start with model that sees everything.

BERT: The Fill-in-the-Blank Reader

Reading-comprehension exam analogy. Read whole passage first — beginning, middle, end — then answer questions. Hiding second half and answering from first half alone cripples you.

BERT works same way. It is encoder-only Transformer: keep encoder, drop decoder from original architecture. No causal mask. No left-to-right limit. Every input token attends to every other token — left and right context, same time, every layer.

Back to movie reviews. When BERT processes "dark" in "The movie was dark but the ending was bright," it sees "movie" on left and "ending" and "bright" on right. By layer 2 or 3, representation of "dark" absorbs whole-sentence information and lands in embedding-space region meaning "negative tone." When it processes "dark" in "The dark theater made the movie feel immersive," it sees "theater" on right, and representation shifts to "physically dark — not value judgment." Same word, different internal representations, because BERT reads whole sentence before decisions.

Bidirectional context is defining advantage. It also creates training problem. Need clever trick.

Why Masking Needs the 80/10/10 Trick

Paradox: if BERT sees every token and you ask it to predict one token, it cheats. It looks at answer directly. No challenge. Model learns nothing.

Fix: original paper's Masked Language Modeling (MLM). Take sentence, hide some words, ask model what sits behind curtain. Fill-in-the-blank test at billion-sentence scale.

Trace one review. Take "The movie was dark but the ending was bright." BERT randomly selects about 15% of tokens — say "dark" and "ending." Subtle part: if BERT always replaces selected tokens with special [MASK], mismatch appears. During pre-training, model sees [MASK] everywhere. During fine-tuning and real inference, [MASK] tokens do not exist. Model may learn to trigger prediction machinery only when special symbol appears.

Fix: 80/10/10 strategy. Of 15% selected tokens, 80% become [MASK]. 10% become random vocabulary word — maybe "dark" becomes "purple" or "seventeen." Remaining 10% stay unchanged — "dark" stays "dark," but model still must predict it.

Why each piece matters. Original sentence:

Original: "The movie was dark but the ending was bright"

Selected tokens: "dark" (position 4), "ending" (position 7)

80% case → "The movie was [MASK] but the [MASK] was bright"
10% case → "The movie was purple but the ending was bright"
10% case → "The movie was dark but the ending was bright"  (unchanged!)

Random replacement ("purple") forces BERT to keep accurate representations at every position, not only where [MASK] appears. Model never knows which tokens were tampered with, so it must keep honest internal model of whole sentence. Unchanged case blocks another shortcut — without it, model could learn "input looks normal, skip prediction." With unchanged case, normal-looking sentence may still contain tokens to predict. BERT must stay vigilant everywhere.

Loss counts only selected tokens — BERT is not penalized for 85% not selected. But 80/10/10 mix blocks cheap signals.

import torch

def apply_mlm_masking(token_ids, vocab_size, mask_token_id, mask_prob=0.15):
    """The 80/10/10 strategy that makes BERT's training work."""
    labels = token_ids.clone()
    masked_input = token_ids.clone()

    # Select ~15% of positions for prediction
    selected = torch.bernoulli(
        torch.full(token_ids.shape, mask_prob)
    ).bool()

    # Only compute loss on selected positions
    labels[~selected] = -100

    # Within selected positions: 80% become [MASK]
    mask_indices = (
        torch.bernoulli(torch.full(token_ids.shape, 0.8)).bool()
        & selected
    )
    masked_input[mask_indices] = mask_token_id

    # 10% become a random token (the remaining selected, split 50/50)
    random_indices = (
        torch.bernoulli(torch.full(token_ids.shape, 0.5)).bool()
        & selected & ~mask_indices
    )
    masked_input[random_indices] = torch.randint(
        0, vocab_size, token_ids.shape
    )[random_indices]

    # The last 10% stay unchanged — already correct in masked_input
    return masked_input, labels

That code is whole masking pipeline. -100 label tells PyTorch cross-entropy loss to ignore those positions, so training hits only tokens chosen for prediction.

80/10/10 split looks like small implementation detail. Not small. Without it, whole pre-training scheme breaks. Tiny details can carry heavy load.

Next Sentence Prediction — and Why It Failed

BERT's second pre-training objective was Next Sentence Prediction (NSP). Take text segments A and B. Train model to predict whether B truly follows A in original document or is random sentence from somewhere else. 50% real pair, 50% random pair. [CLS] representation goes to binary classifier: IsNext or NotNext.

Motivation made sense. Sentence relationships matter for question answering ("Does this paragraph answer question?") and natural language inference ("Does sentence B follow from sentence A?").

Result: mostly useless.

RoBERTa, published in 2019, ran ablation study: train BERT with NSP and without NSP. Version without NSP did better on downstream tasks. Problem: real-next vs random-sentence task is too easy. Random sentence usually comes from totally different topic — maybe cooking vs astrophysics. Model learns shallow topic matching ("both about food, so IsNext"), not real discourse reasoning.

Lesson hurt, but sticks. Not every idea in landmark paper is essential. ALBERT replaced NSP with Sentence Order Prediction (SOP) — predict whether sentences A and B are in correct order, not whether B is random. SOP is harder and more useful because both sentences come from same document. DeBERTa dropped sentence-level objectives entirely. Field moved on.

Dressing Up the Input: [CLS], [SEP], and Three Embeddings

Before BERT reads movie review, input needs specific package. Several pieces work together. Each has job.

WordPiece tokenization breaks text into subword units. BERT does not operate on whole words. "lifeless" from review might become ["life", "##less"], where ## means piece continues previous token. This gives BERT fixed vocabulary of about 30,000 tokens that can represent any English word — even unseen words — by composing known pieces. Like Lego bricks: enough small pieces, no need unique brick for every shape.

Special tokens frame input. Every sequence starts with [CLS], classification token whose final hidden state becomes aggregate representation for whole sequence. Sentences are separated by [SEP]. Single-review classification: [CLS] The movie was dark [SEP]. Two-sentence comparison: [CLS] Sentence A [SEP] Sentence B [SEP].

Why [CLS] works as summary: self-attention lets it attend to every other token. Through 12 attention layers, whole-sequence information flows into one position. Training (via NSP, later fine-tuning) pushes [CLS] representation to be useful for classification. Learned aggregation point — not magic token, designated place where model gathers what matters.

Final BERT input is sum of three separate embeddings:

# Concrete example: classifying "The acting felt flat"
# after tokenization and special token insertion

tokens    = ["[CLS]", "The", "acting", "felt", "flat", "[SEP]"]
token_ids = [  101,   1996,   3772,   2371,  4257,   102 ]
segment   = [    0,      0,      0,      0,     0,     0 ]   # all segment A
position  = [    0,      1,      2,      3,     4,     5 ]

# What BERT actually receives:
# input = token_embedding[101]  + segment_embedding[0] + position_embedding[0]
#       + token_embedding[1996] + segment_embedding[0] + position_embedding[1]
#       + ... and so on for each position

Token embeddings are learned vectors for each WordPiece token — content at each position. Segment embeddings tell model which sentence token belongs to — sentence A gets embedding 0, sentence B gets embedding 1. Useful for two-text comparison tasks. Position embeddings encode token location in sequence. BERT uses learned, not sinusoidal, position embeddings for positions 0 through 511, so max sequence length is 512 tokens.

All three are added element-wise. They live in same 768-dimensional space, and model learns to disentangle signals during training.

Fine-Tuning: Swapping the Head

Pre-training teaches general language understanding. Fine-tuning adapts model to specific task. Recipe: take pre-trained BERT, attach small task-specific layer, train on labeled data for few epochs.

Fine-tune BERT on movie review classification. Need map [CLS] token's 768-dimensional hidden state to one of three labels: positive, negative, or mixed. That is one linear layer — 768 inputs, 3 outputs — then softmax.

from transformers import BertForSequenceClassification, BertTokenizer
from transformers import Trainer, TrainingArguments
from datasets import load_dataset

tokenizer = BertTokenizer.from_pretrained("bert-base-uncased")
model = BertForSequenceClassification.from_pretrained(
    "bert-base-uncased",
    num_labels=3   # positive, negative, mixed
)

dataset = load_dataset("imdb")  # or your own labeled dataset
train_dataset = dataset["train"]
val_dataset = dataset["test"]

training_args = TrainingArguments(
    output_dir="./review_classifier",
    num_train_epochs=3,            # more than 3 and you risk overfitting
    per_device_train_batch_size=16,
    learning_rate=2e-5,            # the critical number
    weight_decay=0.01,
    warmup_steps=100
)

trainer = Trainer(
    model=model,
    args=training_args,
    train_dataset=train_dataset,
    eval_dataset=val_dataset
)
trainer.train()

Learning rate 2e-5 matters. Not arbitrary. Thousands of practitioners converged on narrow range. Go higher — say 1e-4 — and you overwrite pre-trained weights. Model forgets language learned during pre-training. Go lower — say 1e-6 — and model barely adapts. Sweet spot 2e-5 to 5e-5: big enough to learn task, small enough to preserve pre-trained knowledge. Three epochs is usually enough. Beyond that, model starts memorizing training data instead of generalizing.

Setup is versatile. Different task — say named entity recognition — swap head. Instead of one classification for whole sequence, add linear layer that classifies each token: person, organization, location, or none. Pre-trained body stays same. For extractive question answering, head predicts start and end positions of answer span. Same pre-trained weights, different head, different task.

Pattern matters: one expensive pre-training, many cheap fine-tunings. Train base model once at huge cost, then fine-tune for specific task with few hundred labeled examples and one GPU for an hour.

GPT: The Storyteller Who Can't Peek Ahead

Imagine writing novel. Start with first word and build sentence forward, one word at time. You can look back at everything already written — characters introduced, tone established, plot threads set up. You cannot look ahead. Next word depends entirely on what came before.

GPT works that way. It is decoder-only Transformer with causal attention mask. When processing 5th token, it can attend to tokens 1 through 4. Token 6 does not exist yet. Triangular mask sets all future-position attention scores to negative infinity before softmax, so they get zero attention weight.

Trace movie review example. Consider review "The acting felt flat and lifeless." When GPT processes "flat" at position 4, it can attend to "The" (position 1), "acting" (position 2), and "felt" (position 3). It does not see "and" or "lifeless" — those come later. BERT sees whole sentence at once. When BERT processes "flat," it already knows about "lifeless" on right, giving richer context for what "flat" means.

Why choose restricted view? Generation is sequential. When producing text, words after cursor do not exist yet. Nothing to peek at. GPT training matches inference — at training time and generation time, each token is predicted only from preceding tokens. That alignment is one of GPT's deepest strengths.

import torch
import torch.nn.functional as F

def causal_attention(Q, K, V):
    """The core of GPT: self-attention that can't see the future."""
    d_k = Q.size(-1)
    scores = Q @ K.transpose(-2, -1) / (d_k ** 0.5)

    # The causal mask: a lower-triangular matrix of ones
    seq_len = scores.size(-1)
    mask = torch.triu(torch.ones(seq_len, seq_len), diagonal=1).bool()
    scores = scores.masked_fill(mask, float('-inf'))

    weights = F.softmax(scores, dim=-1)
    return weights @ V

# For a 4-token sequence, the attention pattern looks like:
# Token 1 sees: [1, -, -, -]        (only itself)
# Token 2 sees: [1, 2, -, -]        (itself and token 1)
# Token 3 sees: [1, 2, 3, -]        (first three tokens)
# Token 4 sees: [1, 2, 3, 4]        (everything so far)

torch.triu creates upper-triangular matrix of ones, and masked_fill replaces those positions with negative infinity. After softmax, negative infinity becomes zero. Future tokens become invisible. That is whole mechanism separating GPT from BERT — one line of code.

Next-Token Prediction: The Deceptively Powerful Objective

GPT's training objective is bare. Given sequence of tokens, predict next one. That is all. No masking strategy, no auxiliary objectives, no sentence pairs. For every position in every training sequence, compute cross-entropy loss between predicted distribution and actual next token.

Formally: given tokens [t₁, t₂, ..., tₙ], minimize negative log-likelihood:

L = −Σ log P(tᵢ | t₁, t₂, ..., tᵢ₋₁)

Trace one review. Take "The acting felt flat and lifeless." During training, GPT tries to predict each token from its predecessors:

Given ["The"]                          → predict "acting"
Given ["The", "acting"]                → predict "felt"
Given ["The", "acting", "felt"]        → predict "flat"
Given ["The", "acting", "felt", "flat"] → predict "and"
...and so on

At each step, model produces probability distribution over whole vocabulary (say, 50,000 tokens). Loss penalizes low probability on correct next token. Over billions of training examples, model learns grammar, facts, reasoning patterns, writing styles — anything that helps predict what comes next.

Key insight: predicting next token in truly diverse corpus requires understanding almost everything about language. To predict next word in legal brief, need legal reasoning. To predict next token in Python code, need programming logic. To predict next word in story, need narrative structure. Objective is simple. Capability needed to do it well is not.

Strange part: no one is fully sure why next-token prediction works so well. Gap remains between "predicting next word should help with language understanding" (plausible) and "predicting next word produces system that can pass bar exam" (astonishing). We are still closing that gap.

The GPT Scaling Story: 117M to ??? Parameters

GPT history shows what happens when straightforward idea gets scaled relentlessly. Each generation did not only improve old tasks — it unlocked capabilities smaller models could not do at all.

GPT-1 arrived in June 2018 with 117 million parameters, trained on BookCorpus — about 7,000 unpublished books. Contribution was not model architecture (12-layer Transformer decoder, nothing exotic). Contribution was paradigm: pre-train on large text corpus with next-token prediction, then fine-tune on downstream task. Same recipe BERT would use few months later. At 117M parameters, GPT-1 was proof of concept. It needed fine-tuning for every task, same as BERT.

GPT-2 appeared in February 2019, scaled to 1.5 billion parameters — about 13× larger — and trained on 40GB of internet text. OpenAI initially withheld full model, claiming it could generate misleading text too convincingly. Real technical bombshell: GPT-2 could do tasks without any fine-tuning. Phrase input as question, it answers. Give few translation examples, it translates. No weight updates, no gradient descent — prompt alone is enough. First concrete evidence that scale unlocks qualitatively new behaviors. 117M-parameter model cannot do this no matter how you prompt it. 1.5B-parameter model can, even though nobody trained it to.

GPT-3 landed in June 2020 with 175 billion parameters — 100× larger than GPT-2 — trained on 300 billion tokens. This model changed industry. GPT-3 demonstrated few-shot learning: put handful of input-output examples in prompt, and model generalizes to new inputs. No weight updates. Give three sentiment-labeled reviews, it correctly classifies fourth. Give five English-to-French translations, it translates sixth. Estimated training cost: $4.6 million.

GPT-4 arrived in March 2023. OpenAI stopped publishing architecture details, but capabilities were clear: it processes images alongside text, passes bar exam in 90th percentile, writes sophisticated code, follows complex multi-step instructions. Credible reports say it uses mixture-of-experts architecture — multiple specialized sub-networks that activate selectively — though OpenAI has not confirmed this.

Pattern across four generations matters most. GPT-1 proved pre-training works. GPT-2 proved zero-shot works. GPT-3 proved few-shot works. GPT-4 proved multimodal reasoning works. Each generation crossed threshold smaller models could not reach, regardless of clever prompting or engineering.

In-Context Learning: The Prompt Is the Program

This capability turned GPT from interesting research result into platform. You do not fine-tune model. You do not update weights. You write instructions and examples directly in prompt, and model follows them.

Back to movie reviews. Here are three prompting modes, each giving model different amount of guidance:

Zero-shot — describe task, provide no examples:

Classify this movie review as positive, negative, or mixed.

Review: "The movie was dark but the ending was bright"
Sentiment:

Model must infer task format and answer from description alone. With GPT-3 and larger models, this often works.

Few-shot — provide handful of labeled examples first:

Classify each movie review.

"The acting felt flat and lifeless" → negative
"A bright and moving performance" → positive
"Flat acting ruined an otherwise bright script" → negative

"The movie was dark but the ending was bright" →

Model sees pattern — reviews followed by labels — and continues it. This works well, even for tasks model was never explicitly trained on.

Many-shot — with modern context windows stretching to 100K+ tokens, you can include dozens or hundreds of examples. Performance keeps climbing with more examples, sometimes approaching accuracy of fine-tuned BERT model.

Why in-context learning works at all? Two competing theories. Both compelling. First: implicit gradient descent. Research shows Transformer attention layers can implement something mathematically similar to one gradient descent step in forward pass. Examples in prompt act like tiny training set, and model internally "fits" to them — not by changing weights, but by adjusting activations. Second: task recognition. Model saw thousands of task formats during pre-training (Q&A pairs, classification examples, translation pairs), and few-shot examples teach nothing new. They tell model which existing behavior to activate.

Both theories have experimental support. Exact explanation still unclear. In practice, result is same: one model, infinite tasks, no retraining.

Emergent Abilities — Real or Mirage?

Some abilities seem to appear from nowhere as models get bigger. 1-billion-parameter model cannot do three-digit addition. 10-billion-parameter model still cannot. 100-billion-parameter model suddenly can. Not gradual improvement — looks like phase transition, like water turning to ice.

Chain-of-thought reasoning is vivid example. Append "Let's think step by step" to math problem, and large models start decomposing problem into intermediate steps and reaching correct answer. Small models given same prompt produce plausible-looking but wrong reasoning chains. Ability to decompose problems into steps was not explicit training objective. It existed in training data — humans write step-by-step explanations — and model learned to use it, but only at sufficient scale.

Real debate exists about whether this is real. Schaeffer, Miranda, and Koyejo published provocative 2023 paper titled "Are Emergent Abilities of Large Language Models a Mirage?" Their argument: apparent phase transition is measurement artifact. If you use binary exact-match accuracy (right or wrong, no partial credit), you see sharp jump. Switch to continuous metric like token-level log-likelihood, and improvement is smooth and gradual all along. Model was improving continuously — all-or-nothing metric failed to show it until threshold was crossed.

Debate is not fully settled. Some capabilities — like in-context learning itself — do seem absent at small scales. 117M-parameter model does not do few-shot learning badly; it does not do it at all. But Schaeffer's work is critical reminder: how you measure determines what you see. Keep that in mind whenever reading claims about "breakthrough" capabilities at new scale.

Chinchilla: The Data Diet Breakthrough

For years, scaling strategy was simple: make model bigger. More parameters, better performance. GPT-3 had 175 billion parameters trained on 300 billion tokens. Most compute budget went into parameters.

In 2022, DeepMind's Chinchilla paper overturned that. Key finding: scale data and parameters together. Optimal ratio is roughly 20 training tokens per parameter. By that rule, GPT-3's 175B parameters needed about 3.5 trillion tokens, not 300 billion. GPT-3 was massively undertrained.

DeepMind proved point by training Chinchilla — 70 billion parameters on 1.4 trillion tokens. It beat Gopher, DeepMind's own 280-billion-parameter model trained on 300 billion tokens. Model 4× smaller won because it saw 4.7× more data. Total training compute was same — allocation changed.

Exercise-and-nutrition analogy. For years, field kept going to gym more (bigger models) while eating same small diet (same data amount). Chinchilla showed better results come from increasing both together. You cannot out-train bad diet, and you cannot out-data tiny model. Balance matters.

Insight echoes BERT story. Remember RoBERTa? Same architecture, 10× more data, much better results. Lesson repeats: many "architectural breakthroughs" are really training-discipline breakthroughs. Data volume and training duration matter as much as parameter count.

The BERT Family Tree: RoBERTa, ALBERT, DeBERTa

Original BERT paper left performance on table. Not because architecture was wrong, but because training was incomplete. Several teams found this and published improvements. Three matter most. Each teaches different lesson.

RoBERTa (Robustly Optimized BERT Approach, 2019) changed nothing in architecture. Zero architectural modifications. It trained same model with better discipline: 10× more training data, removed NSP, used dynamic masking (re-randomize masked tokens each epoch instead of fixing mask pattern during preprocessing), larger batch sizes, and longer training. Result: massive improvements across benchmarks.

RoBERTa lesson is humbling. Original BERT was not trained hard enough. Architecture was fine — training recipe was incomplete. Pattern repeats across deep learning: before redesigning architecture, try training existing one properly.

ALBERT (A Lite BERT, 2019) went opposite direction — smaller, more efficient BERT. Two key ideas. First, cross-layer parameter sharing: instead of 12 Transformer layers each having independent weights, all layers share same weights. Model applies same transformation 12 times. This cuts parameter count sharply (from 110M to as low as 12M) with surprisingly small performance loss. Second, factorized embedding parameterization: instead of one large embedding matrix mapping vocabulary tokens directly to hidden dimension (30,000 × 768 = 23M parameters), ALBERT first maps to smaller intermediate dimension (30,000 × 128) and then projects up (128 × 768). This decouples vocabulary size from hidden dimension, saving millions of parameters. ALBERT also replaced NSP with Sentence Order Prediction, which is harder and more useful.

DeBERTa (Decoding-enhanced BERT with Disentangled Attention, 2020) made biggest architectural change and produced strongest results. Core innovation is disentangled attention: instead of computing attention scores from one combined embedding that mixes content and position information, DeBERTa keeps content and position separate.

In standard BERT, attention score between tokens i and j comes from embeddings that already combine "what token is" with "where token is." DeBERTa untangles those signals. Attention score becomes sum of three components: content-to-content (what is token i, and what is token j?), content-to-position (what is token i, and where is token j?), and position-to-content (where is token i, and what is token j?). Each component uses separate learned projections.

Crude visualization: in movie review "The acting felt flat," standard attention computes "how relevant is 'flat' to 'acting'?" as one score mixing word identity and position. DeBERTa asks three separate questions: "Is concept 'flat' relevant to concept 'acting'?" (content-to-content), "Is concept 'flat' relevant to whatever is at position 2?" (content-to-position), and "Is whatever is at position 4 relevant to concept 'acting'?" (position-to-content). Then it sums three scores. Model can learn, for example, that adjectives at certain relative positions matter more than adjectives far away — separately from learning that "flat" is semantically relevant to "acting."

DeBERTa surpassed human baseline on SuperGLUE, premier language understanding benchmark, and remains strongest model in BERT family. Success suggests separating "what" from "where" in attention is important architectural principle.

Encoder vs. Decoder: The Paradigm Comparison

We have seen both sides. Now put them face to face.

Back to movie review classifier. If we want to classify "The movie was dark but the ending was bright" as positive, negative, or mixed, model must understand entire review before decision. Word "but" does critical work — it signals contrast between "dark" and "bright." Model reading left-to-right and reaching "dark" might guess "negative" too early, and correction comes later. BERT reads whole thing at once and catches contrast from start. That is BERT advantage for understanding tasks.

Now imagine model must write review. "Generate a mixed sentiment review about a sci-fi movie." Model must produce text word by word, and it cannot look ahead because words do not exist yet. That is GPT advantage for generation tasks.

Now practical part — differences that matter when choosing model for production system:

Dimension BERT (Encoder) GPT (Decoder)
Attention pattern Bidirectional — every token sees every other token Causal — each token sees only past tokens
Training objective Masked language modeling (fill in blanks) Next-token prediction (continue the sequence)
Natural strength Understanding: classification, NER, similarity, extractive QA Generation: text, dialogue, code, translation
How you adapt it Fine-tune on labeled data with a task-specific head Prompt engineering, few-shot, or fine-tune
Inference speed Fast — one forward pass for the entire input Slower — one forward pass per generated token

Hard truth for BERT paradigm: GPT-style models are becoming competitive on understanding tasks too. GPT-4 does sentiment classification, named entity recognition, and question answering through prompting — no fine-tuning, no labeled data. For many tasks, accuracy gap between prompted GPT-4 and fine-tuned DeBERTa has narrowed.

"Narrowed" is not "gone." BERT-style models still win in some cases. Need millisecond latency? BERT processes whole input in one forward pass, while GPT generates one token at a time. Need cheap inference at scale? DistilBERT is 66 million parameters; GPT-4 is orders of magnitude larger. Need guaranteed output format? Classification head outputs exactly one of N labels. GPT can hallucinate anything. Need deterministic, reproducible results? Fine-tuned BERT with fixed weights gives same output every time. GPT with temperature sampling does not.

For production classification pipelines handling millions of items per hour, fine-tuned DistilBERT running at 2ms per input and costing fractions of a cent per thousand calls is still pragmatic choice.

The Hybrid Workflow

In practice, strongest teams do not choose one paradigm over other. They use both, in sequence.

Workflow goes like this: start with GPT for prototyping. Use prompting to test whether task is solvable — can language model distinguish positive, negative, and mixed reviews from prompt alone? If yes, use GPT to generate labeled data by classifying large batch of unlabeled examples. Then take GPT-generated labels, fine-tune small BERT-style model (DistilBERT, RoBERTa) on them, and deploy small model to production.

# Phase 1: GPT generates labels (prototyping)
from openai import OpenAI
client = OpenAI()

def label_with_gpt(review_text):
    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{
            "role": "user",
            "content": (
                "Classify this movie review as positive, negative, "
                "or mixed. Reply with one word only.\n\n"
                f"Review: {review_text}"
            )
        }],
        temperature=0
    )
    return response.choices[0].message.content.strip().lower()

# Phase 2: Fine-tune a small model on GPT's labels (production)
from transformers import AutoModelForSequenceClassification, AutoTokenizer

model = AutoModelForSequenceClassification.from_pretrained(
    "distilbert-base-uncased", num_labels=3
)
tokenizer = AutoTokenizer.from_pretrained("distilbert-base-uncased")
# Train on the GPT-labeled data → deploy a 66M-param model
# that runs in 2ms and costs 100× less per call

Result: GPT gives flexibility for exploration and data generation, while BERT gives speed and cost efficiency for deployment. Paradigms are not competitors — they are complements. GPT is research assistant. BERT is production engineer.

Running example comes full circle. We started with five movie reviews and question about ambiguous words like "dark" and "bright." BERT reads full review and resolves ambiguity with left and right context. GPT reads left-to-right and resolves it with only previous context — but compensates with scale, in-context learning, and huge volume of language patterns absorbed during training. For classifying those reviews in production, small fine-tuned BERT is efficient choice. For exploring new classification schemes or generating training data, GPT is flexible choice.
