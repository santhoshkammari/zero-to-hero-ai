Two questions become urgent once transformers hit production. First: what did model learn, and can we peek inside without breaking it? That's interpretability. Second: standard self-attention scales quadratically with sequence length, so long documents break it. That's efficiency problem. This section covers both.

Ground this in concrete scenario. We build customer support chatbot for small online bookstore. Chatbot reads customer message — "I ordered The Great Gatsby last Tuesday and it still hasn't arrived" — and must generate helpful reply. Model is small transformer with 4 layers, 4 attention heads per layer, and vocabulary of 10,000 tokens. We keep returning to this bookstore bot while we unpack interpretability and efficiency.

Attention Heatmaps and Why They Lie

First move — mine too — is plotting attention weights. Each attention head produces matrix where value in row i, column j tells us how much token i "looked at" token j when computing output. Draw it as heatmap. Results look convincing.

For bookstore bot, feed sentence "The book arrived damaged." We pull out attention matrix from head 2 of layer 3. It's 5×5 grid (one row and column per token). Visualize it, and token "damaged" strongly attends to "book." Feels satisfying. Feels like understanding.

Problem: we haven't. In 2019, Jain and Wallace showed you could find completely different attention distributions — ones nothing like original — that produced exact same model predictions. If two different "explanations" give same answer, neither truly explains. Wiegreffe and Pinter added important nuance later that year, but consensus landed here: attention weights show where information was routed, not what information mattered for final decision.

This tripped me up. I wanted attention heatmaps to be X-ray showing broken bone. They're more like traffic map showing which roads had cars on them. Useful for debugging routing, but not why someone drove to that destination.

If heatmaps can't tell us what model learned, what can?

Probing Classifiers — An MRI for Neural Networks

Sharper question: does specific layer of our bookstore bot "know" part of speech of each token? Does it know which tokens refer to people versus products? Does it understand sentence structure? Probing classifiers answer these questions.

Idea is elegant. Freeze model — not one weight changes. Extract hidden state vectors from particular layer for many labeled examples. Train tiny classifier — usually linear, so it can only draw straight-line decision boundaries — on those hidden states to predict some property. If linear classifier predicts part-of-speech tags from layer 2's hidden states with 95% accuracy, that information sits there in accessible form.

Walk through this with bookstore bot. We have sentence "I ordered The Great Gatsby last Tuesday." After running it through model, layer 1 produces hidden state vector for each token — seven vectors, each maybe 256 dimensions. We collect thousands of such sentences with part-of-speech labels. Then we train linear classifier: given 256-dimensional vector, predict whether token is noun, verb, adjective, or something else. If this trivial classifier hits high accuracy, model's first layer has already separated tokens by grammatical role.

Researchers did this systematically, layer by layer, for BERT and similar models. Pattern is remarkably consistent. Early layers — layers 1 and 2 — capture surface features: word identity, part-of-speech tags, morphology. Middle layers — around layers 4 through 8 in 12-layer BERT — capture syntax: dependency relations, phrase structure, whether two tokens are in same clause. Later layers capture semantics: coreference (knowing "it" refers to "the book"), sentiment, and entity types.

Think of it like medical imaging pipeline. Early layers are raw pixel detector — they see edges and shapes. Middle layers assemble those into anatomical structures. Later layers are radiologist reading image and saying "this is abnormal."

Catch: if you make probe too powerful — say, deep neural network instead of linear classifier — it might learn property from scratch rather than detecting it in hidden states. Sufficiently complex probe can extract patterns from random noise. That defeats whole purpose. Keeping probes simple makes this technique meaningful. If linear classifier can find it, representation must be doing real work.

Probing tells us what information lives in each layer. It does not tell us how model uses that information — which heads and neurons work together to produce prediction. For that, we need something more invasive.

BERTology — The Layer-by-Layer Map

Those probing results came not from one paper but hundreds. Field got so prolific that Rogers, Kovaleva, and Rumshisky published 2020 survey, "A Primer in BERTology," synthesizing findings. Name stuck: BERTology studies what BERT-like models learn and how they organize that knowledge.

Few findings stand out. First, not all attention heads matter equally. You can prune large fraction of heads with minimal performance hit. Some heads are specialists — one focuses on direct objects, another on punctuation, another on coreference. Second, knowledge is distributed: no single neuron or head owns any linguistic feature. It's collective property. Third, representations evolve across layers. Word's embedding at layer 1 looks nothing like its embedding at layer 12. Model progressively refines from surface to syntax to semantics.

For bookstore bot, this means something concrete. If bot fails to understand that "it" in "I returned it yesterday" refers to book, problem likely lives in later layers. If it's confused about word boundaries in misspelled query, issue is probably in early layers. BERTology gives rough map of where to look.

But BERTology is still observational. We see map, not terrain. Can we go deeper — can we reverse-engineer actual computations?

Mechanistic Interpretability — Reverse-Engineering the Circuits

Now things get ambitious. Mechanistic interpretability tries to identify specific computations performed by individual neurons, attention heads, and component combinations. Not "this layer knows syntax" but "these two attention heads, working together, implement a copy-paste operation."

Most famous discovery is induction head. Go back to bookstore bot. Earlier in conversation, customer wrote "I'm looking for Harry Potter." Later, customer writes "I need another Harry —". Induction head is pair of attention heads that complete this pattern: one head scans backward for previous occurrence of current token ("Harry"), and second head copies what came after it last time ("Potter"). Pattern-matching circuit baked into weights.

Researchers at Anthropic and elsewhere showed induction heads emerge at specific point during training — almost phase transition where model's ability to learn from context suddenly improves, and that improvement coincides with formation of these heads. One of cleanest examples of neural network developing identifiable algorithm through training.

Second major discovery is superposition. Intuition: model with 256 neurons per layer does not store 256 concepts. It stores far more than 256 concepts by encoding them as overlapping, nearly-orthogonal directions in that 256-dimensional space. One neuron might respond to both "academic writing" and "the color blue" — not because concepts are related, but because model learned to pack them into same space efficiently. Compression trick. Reason individual neurons in isolation give confusing results.

Anthropic's more recent work uses sparse autoencoders and dictionary learning to disentangle these superimposed features. They isolated specific concepts — like "Golden Gate Bridge" concept — as individual features in Claude's activations and then manipulated them. Extraordinary. Little unsettling.

Intuition for why superposition works so well is still developing. High-level explanation — "there are more concepts than neurons, so model compresses" — is satisfying like one-sentence novel summary. Full picture involves geometry about how many nearly-orthogonal vectors fit into space, and field is still working through details.

Mechanistic interpretability is particle physics of deep learning. Findings are striking, potentially critical for AI safety and model auditing. But work is painstaking, often manual, and has not scaled to largest production models yet. If you're practitioner, know these tools exist and are improving fast. If interview asks, know two key concepts — induction heads and superposition — and why they matter for understanding in-context learning.

The Quadratic Wall

Go back to bookstore bot. So far, customer messages were short — 20 or 30 tokens. At those lengths, attention is fast. Now we want bot to read customer's entire order history before responding. That's page of text. Maybe ten pages. Standard self-attention computes score between every pair of tokens. For sequence of length n, that's n × n scores.

Make pain concrete with bookstore scenario. Short customer message is about 50 tokens: 50 × 50 = 2,500 attention scores. Comfortable. Full conversation thread is 2,000 tokens: 2,000 × 2,000 = 4 million scores. Fine. Entire order history with all correspondence is 32,000 tokens: 32,000 × 32,000 = about 1 billion scores. Now problem starts. Full product catalog bot might search hits 100,000 tokens: 100,000 × 100,000 = 10 billion scores. GPU gives up.

This is quadratic wall. Every technique in rest of this section attacks it from different angle. Think traffic problem: small town does not need highways, but growing city needs multiple strategies — wider roads, bypasses, express lanes, smarter traffic lights. No single solution handles everything.

Sparse Attention — Don't Look at Everything

First observation: most entries in attention matrix are near zero anyway. When bookstore bot processes 2,000-token conversation, token "Gatsby" at position 47 probably has no meaningful relationship with token "thank" at position 1,983. We compute score between them and answer is effectively zero. Wasted work.

Sparse attention says: skip pairs that don't matter. Instead of computing all n² scores, define pattern that picks important ones.

Simplest pattern is local window. Each token attends to its w nearest neighbors — tokens directly before and after it. If w is 512, each token sees 512 neighbors instead of all n tokens. Cost becomes n × w instead of n × n. For our 32,000-token order history with window of 512, that's about 16 million scores instead of 1 billion. 60× reduction.

But wait — does this mean distant tokens can never interact? Not quite. Information propagates through layers. If layer 1 lets token 1 see tokens 1–512, and layer 2 lets it see those same neighbors (who themselves saw their 512 neighbors in layer 1), then after 2 layers, token 1 has indirect access to tokens 1–1024. Stack 12 layers and information can travel across 6,000 tokens. Like game of telephone, except information is encoded in high-dimensional vectors and surprisingly little gets lost.

Local window handles nearby context well, but what about genuine long-range dependencies? What if customer's complaint at end of conversation references order number from beginning? Three additional sparse patterns help.

Global tokens are handful of designated positions — often [CLS] token or start-of-sequence marker — that attend to every token and that every token attends to. They act as information hubs: global token can pick up order number from position 5 and make it available to complaint at position 1,900. Cost is small because there are only few global tokens.

Strided attention makes each token attend to every k-th token, giving coarse bird's-eye view of whole sequence. If k is 64, token at position 1,000 sees positions 0, 64, 128, 192, and so on — sparse but wide perspective.

Random connections are what they sound like: for each token, randomly select small number of other tokens to attend to. Sounds reckless, but random connections create shortcuts in information graph, similar to how random edges in social network shrink "six degrees of separation" distance.

Longformer (2020) combines local window attention with global tokens. Every token sees its neighbors, and few special tokens see everything. This handles tasks like document classification and question answering over long texts. BigBird (2020) adds random connections on top, and proved something theoretically striking: local + global + random connections give same theoretical expressiveness as full attention, measured by ability to simulate Turing machines. Both achieve near-full-attention quality at fraction of cost.

Limitation? Sparse patterns are hand-designed. Someone must decide window size, number of global tokens, stride. Different tasks might want different patterns. And for tasks where important attention connections are neither local nor random — where model needs to discover which pairs matter — fixed patterns can miss things.

Linear Attention — Rearranging the Math

Instead of deciding which pairs to skip, what if we change math so quadratic cost disappears entirely? That's idea behind linear attention.

Standard attention computes softmax(QKT)V. Bottleneck is QKT, which produces n × n attention matrix. Key matrix algebra observation: matrix multiplication is associative: (AB)C = A(BC). If we skip softmax — or approximate it — we can compute KTV first. That product has shape d × d (where d is head dimension, usually 64 or 128). Then we multiply Q by that d × d result. Cost goes from O(n²d) to O(nd²). Since d is small and fixed, that's linear in sequence length.

Catch is softmax. It's applied element-wise to n × n matrix, which means you can't factor it out of multiplication. Trick used by Performer (2020) and Linformer (2020): replace softmax with kernel approximation. Performer uses FAVOR+ (Fast Attention Via positive Orthogonal Random features), which approximates softmax kernel with random feature maps. Linformer takes different approach: it projects sequence length dimension of K and V down to smaller fixed size k using learned projection matrices, reducing attention matrix from n × n to n × k.

# Standard attention: O(n²d) — build n×n first
scores = Q @ K.T           # shape: n × n  ← this is the problem
weights = softmax(scores)   # shape: n × n
output = weights @ V        # shape: n × d

# Linear attention (Performer-style): O(nd²) — build d×d first
Q_prime = phi(Q)            # apply random feature map
K_prime = phi(K)            # apply random feature map
kv = K_prime.T @ V          # shape: d × d  ← much smaller!
output = Q_prime @ kv       # shape: n × d

Promise is beautiful: linear scaling means 100,000-token sequence costs 100× more than 1,000 tokens, not 10,000× more. Reality is messier. Kernel approximation introduces error. On tasks where exact token-to-token attention pattern matters — where model must attend very sharply to one or two specific tokens — approximation smooths things out and quality drops. Performer and Linformer showed impressive results on some benchmarks but never achieved dominance of next technique.

Think of sparse and linear attention like two strategies for crowded library. Sparse attention says "Look only at nearby shelves and few random ones across room." Linear attention says "Build compressed summary of every shelf so no need to visit each one." Both work. Both have blind spots. Neither is what most production systems ended up using.

FlashAttention — The Trick That Actually Won

Need correct misconception. When I first heard "efficient attention," I assumed solution was mathematical — find clever approximation. FlashAttention is not that. It computes exact standard attention. Same math, same outputs, bit-for-bit identical results. Whole trick is how computation is organized on GPU hardware.

To understand FlashAttention, understand one GPU fact. Modern GPU has two kinds of memory. High-bandwidth memory (HBM) is large — 40 to 80 gigabytes on an A100 — but relatively slow to access. Think warehouse: lots of space, but trip to shelf takes time. SRAM (on-chip memory, sometimes called shared memory) is tiny — around 20 megabytes total across all streaming multiprocessors — but extremely fast. Think workbench: everything is within arm's reach.

Naive attention computation goes like this. Compute Q × KT, getting full n × n attention matrix. Write that matrix to HBM. Read it back from HBM to compute softmax. Write softmax result to HBM. Read it back to multiply by V. Write final output to HBM. Each read and write is trip to warehouse. For long sequences, we spend more time moving data between HBM and SRAM than doing arithmetic. Computation is memory-bound, not compute-bound.

FlashAttention, introduced by Tri Dao in 2022, eliminates those round trips with tiling. Instead of computing entire n × n attention matrix at once, it processes matrix in small blocks — tiles — that fit in SRAM. Algorithm loads tile of Q and tile of K into SRAM, computes attention scores, applies softmax (partially), multiplies by corresponding tile of V, and accumulates result. Then it moves to next tile. Full n × n matrix is never materialized in HBM.

Clever part is online softmax trick. Softmax needs denominator — sum of all exponentials — which normally requires seeing all values first. How do you compute softmax in tiles without seeing every value? Answer: maintain running statistics. For each row, track maximum value seen so far and running sum of exponentials. As each new tile arrives, update these statistics incrementally. When all tiles are processed, running statistics give exact softmax. No approximation. No error.

Trace tiny example. Suppose bookstore bot has 4-token sentence and we tile it into blocks of 2. We have Q with 4 rows and K with 4 columns. Instead of computing full 4 × 4 attention matrix, we process two tiles. Tile 1: Q rows [1,2] × K columns [1,2] — 2 × 2 block. We compute these 4 scores, find max per row, compute partial exponentials, and start accumulating weighted V. Tile 2: Q rows [1,2] × K columns [3,4] — another 2 × 2 block. We compute these scores, update running max (if we found larger value, we rescale previous partial sums), update running sum, and add more weighted V. After tile 2, we have exact answer for rows 1 and 2. Then we repeat for Q rows [3,4]. Full 4 × 4 matrix was never stored anywhere.

Results are dramatic. FlashAttention delivers 2–4× wall-clock speedup over naive attention and uses far less memory, because it stores only O(n) values in HBM instead of O(n²). FlashAttention-2 (2023) improved parallelism further. This runs under hood of almost every modern large language model. If you remember one technique from this section, make it FlashAttention.

Pause on this, because insight is profound. Math didn't change. Algorithm didn't change. What changed was respecting hardware. Much of high-performance computing is understanding that data movement costs more than arithmetic, and designing algorithms that keep data close to computation. FlashAttention is masterclass in this principle.

Library analogy still applies. Sparse attention says "visit fewer shelves." Linear attention says "make summary map." FlashAttention says "bring shelves to my workbench in small batches." Same library, same books, same reading — but faster, because we stopped walking back and forth.

The KV Cache — Why Inference Memory Blows Up

We've been talking about making attention efficient during training, where we process whole sequence at once. Inference is different beast. When bookstore bot generates response — one token at time, autoregressively — it runs full attention computation at every step. Generate token 1, run attention. Generate token 2, run attention over tokens 1–2. Generate token 100, run attention over all 100 tokens so far.

Wasteful part: when generating token 100, model computes new key and value vectors for every previous token — but tokens 1 through 99 haven't changed. Their keys and values are exactly same as when we generated token 99. We recompute same thing over and over.

KV cache is fix. After computing key and value vectors for each token, we store them. When generating next token, we only compute Q, K, and V for single new token, then look up cached keys and values for all previous tokens. Attention computation goes from processing n tokens' worth of K and V at step n to processing 1 new token plus cache lookup.

Trace this for bookstore bot generating "Your order has shipped." At step 1, we compute K₁ and V₁ for "Your" and cache them. At step 2, we compute K₂ and V₂ for "order", cache them, and attend over [K₁, K₂] and [V₁, V₂] using only new query Q₂. At step 3, we compute K₃ and V₃ for "has", cache them, and attend over [K₁, K₂, K₃] using Q₃. Each step adds one entry to cache instead of recomputing everything.

KV cache is what makes LLM inference feasible at all. Without it, generating each new token would be as expensive as processing entire sequence from scratch. With it, per-token cost is roughly constant. Difference between chatbot that responds in milliseconds and one that takes minutes.

Tradeoff: cache consumes memory. For each layer, for each attention head, we store key vector and value vector for every token generated so far. Rough calculation for 70-billion-parameter model with 80 layers, 64 attention heads, and head dimension of 128: KV cache for single sequence of 4,096 tokens requires about 80 × 64 × 4,096 × 128 × 2 (K and V) × 2 bytes (FP16) ≈ 10.7 gigabytes. For single user's conversation. Now imagine serving hundreds of users simultaneously — KV cache alone can dwarf model weights in memory consumption.

This is why KV cache became dominant bottleneck in LLM serving. Not compute, not model weights — per-request cache for ongoing conversations.

Multi-Query and Grouped-Query Attention — Shrinking the Cache

KV cache problem has structural cause. In standard multi-head attention, every attention head has its own Q, K, and V projection matrices. If we have 64 heads, we store 64 separate K tensors and 64 separate V tensors per layer. That's what makes cache enormous.

In 2019, Noam Shazeer proposed multi-query attention (MQA): keep 64 separate query heads, but share single K head and single V head across all of them. Every query head still gets its own perspective (its own Q projection), but all attend over same keys and values. KV cache shrinks by factor of 64 — from 64 K/V pairs to 1.

Trace this for bookstore bot. In standard multi-head attention with 4 heads, generating one token produces 4 key vectors and 4 value vectors to cache. After 100 tokens, cache holds 800 vectors per layer. With MQA, generating one token produces 1 key vector and 1 value vector, shared across all 4 heads. After 100 tokens, cache holds 200 vectors per layer. 4× reduction.

Concern with MQA is quality loss. Different heads are forced to attend over identical keys and values, which might limit model's ability to attend to different kinds of information simultaneously. In practice, quality hit is small for many tasks but noticeable for others.

Grouped-query attention (GQA), introduced in Llama 2 paper (2023), is compromise that stuck. Instead of all heads sharing one K/V pair, you group heads. With 64 query heads and 8 groups, you have 8 K/V pairs — each serving 8 query heads. KV cache shrinks by 8× (instead of 64×), but each query group gets its own, somewhat specialized K and V. GQA is used in Llama 2, Llama 3, Mistral, and Gemma. Current sweet spot — most of MQA's memory savings with almost no measurable quality loss.

Think of it like desks in office. Standard multi-head attention gives every employee own filing cabinet. MQA puts everyone around one shared cabinet. GQA creates team filing cabinets — small groups share, but not whole company. Practical trade-off between personalization and space efficiency.

Complementary technique: PagedAttention, used in vLLM serving framework. Even with GQA shrinking per-head cache, managing memory for thousands of concurrent users is messy. Traditional implementations allocate large contiguous memory block for each user's cache, even if conversation is short. PagedAttention borrows from operating systems: divide KV cache into fixed-size pages and map them dynamically, like virtual memory. Short conversations use few pages; long ones grow as needed. No wasted memory from over-allocation. If you deploy LLMs at scale, you'll spend more time thinking about KV cache management than model architecture.

Speculative Decoding — Two Models, One Answer

Even with FlashAttention for efficient computation and GQA for lean cache, autoregressive generation has fundamental limitation: it's sequential. We generate one token, wait for it, use it to generate next, wait, and so on. GPU — machine built for massive parallelism — spends most of its time idle, waiting for each token to finish. Like factory with thousand workers where only one can work at time.

Speculative decoding breaks this sequential bottleneck with clever trick. We use two models: small, fast draft model and large, accurate target model. Draft model guesses next several tokens — say 8 at time. Then target model verifies all 8 guesses in single forward pass (which can be done in parallel, since we have all inputs). If first 5 guesses match what target model would have generated, we accept them all and move on. If guess 6 diverges, we reject it and everything after it, keeping only first 5.

Trace this for bookstore bot. Customer asks "When will my order arrive?" Draft model — small, cheap model — quickly generates 8 candidate tokens: "Your", "order", "is", "expected", "to", "arrive", "by", "Friday". Target model (our big, accurate model) takes these 8 tokens and, in single parallel forward pass, checks: would I have generated each of these? It agrees with "Your", "order", "is", "expected", "to", "arrive" but at position 7, it would have said "on" instead of "by". We accept first 6 tokens, discard rest, and start new draft from "arrive."

Math works because verification is cheaper than generation. Target model can process 8 tokens in parallel about as fast as it processes 1 token — attention computation scales well with batch size. So instead of 8 sequential forward passes through big model (slow), we do 8 cheap passes through draft model (fast) plus 1 forward pass through big model (same cost as generating 1 token). If draft model is right most of time — and well-chosen draft model can be right 70–80% of time — we get 2–3× speedup with zero quality loss. Outputs are mathematically identical to what target model would have produced on its own.

I still find it surprising that this works so well. Key insight: most tokens in well-trained LLM are highly predictable — articles, prepositions, common phrases. Small model gets those right. Big model is truly needed only for harder decisions, and speculative decoding lets us skip ahead past easy ones.
