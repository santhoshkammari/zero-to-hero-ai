Transformer Interpretability & Analysis

Chapter 10: Transformers 6 subtopics
TL;DR

Transformers are powerful but opaque. Interpretability research cracks them open. Attention visualization shows where the model looks, but attention weights alone don't explain decisions — that's a common trap. Probing classifiers reveal that lower layers encode syntax and upper layers encode semantics, mirroring the classical NLP pipeline. Mechanistic interpretability reverse-engineers actual circuits inside the network: QK circuits for matching, OV circuits for moving information, and induction heads for in-context copying. The superposition hypothesis says models cram more concepts than they have dimensions; sparse autoencoders can disentangle them into interpretable features. Tools like the Logit Lens let you peek at what the model is "thinking" at each layer, and techniques like ROME let you surgically edit individual facts without retraining.

You've trained a transformer that answers questions, writes code, and summarizes documents. Then it confidently tells a user that the Eiffel Tower is in London. Your first instinct is to look inside the model and figure out why. Your second instinct is despair — the thing has 7 billion parameters spread across 32 layers. Where do you even start? That gap between "something went wrong" and "here's exactly which computation went wrong" is the interpretability problem. It matters for safety (will the model behave in deployment?), for debugging (why did it hallucinate?), and for trust (can a doctor rely on this diagnosis?). This section is your toolkit for closing that gap.

We'll work our way from the simplest tools — literally plotting attention weights — up through probing classifiers, mechanistic interpretability, superposition, and finally surgical knowledge editing. Each technique trades off between ease of use and depth of insight. None is a silver bullet. Use them together, and you start to understand what these models actually compute.

Attention Pattern Analysis

The most natural place to start is attention. Every transformer layer has attention heads, and each head produces a matrix of weights that says "how much does token i attend to token j." You can visualize this as a heatmap. Tools like BertViz make it a few lines of code. When you plot these heatmaps for a well-trained model, distinct patterns jump out immediately.

Some heads are positional heads — they attend strongly to the previous token or the next token, regardless of content. Others are syntactic heads: given the sentence "The cat that I adopted yesterday sleeps," a syntactic head attending from "sleeps" will point straight back at "cat," its grammatical subject, skipping over the entire relative clause. You'll also find separator-attending heads that focus on punctuation and special tokens like [SEP] or [CLS]. These patterns are remarkably consistent across different inputs, suggesting each head has learned a specific "role."

It's incredibly tempting to treat these attention weights as explanations. "The model predicted 'cat' because head 7 attended to 'feline.'" Stop right there. Attention weights are not feature importance scores. This was the central finding of the "Attention is not Explanation" paper (Jain & Wallace, 2019). The problem is that attention weights tell you which tokens were mixed together, but not what the model did with that mixture. Two different attention distributions can produce the same output, and a head can attend heavily to a token while the downstream computation completely ignores that information. Attention is a routing mechanism, not an explanation.

A partial fix is attention rollout. Raw attention from one layer only shows direct connections. But information flows through multiple layers: token A attends to token B in layer 3, and token B attended to token C in layer 1, so token A indirectly "saw" token C. Attention rollout multiplies the attention matrices across layers (accounting for residual connections) to produce a single token-level attribution map that captures these multi-hop paths. It's more honest than single-layer attention, but it still only tracks the routing — not the computation performed on the routed information.

import torch
from transformers import AutoTokenizer, AutoModel

tokenizer = AutoTokenizer.from_pretrained("bert-base-uncased")
model = AutoModel.from_pretrained("bert-base-uncased", output_attentions=True)

text = "The cat that I adopted yesterday sleeps peacefully"
inputs = tokenizer(text, return_tensors="pt")

with torch.no_grad():
    outputs = model(**inputs)

# outputs.attentions is a tuple: one (batch, heads, seq, seq) per layer
# Attention rollout: multiply attention matrices across layers
attentions = outputs.attentions  # 12 layers for bert-base
rollout = torch.eye(attentions[0].size(-1))

for att in attentions:
    # Average across heads, then add residual connection and renormalize
    att_heads_avg = att.mean(dim=1).squeeze(0)  # (seq, seq)
    att_with_residual = 0.5 * att_heads_avg + 0.5 * torch.eye(att_heads_avg.size(0))
    att_with_residual /= att_with_residual.sum(dim=-1, keepdim=True)
    rollout = att_with_residual @ rollout

tokens = tokenizer.convert_ids_to_tokens(inputs["input_ids"][0])
# rollout[i, j] = how much token i's final representation was influenced by token j
print("Rollout from [CLS] to each token:")
for tok, weight in zip(tokens, rollout[0]):
    print(f"  {tok:15s} {weight:.4f}")

The takeaway: attention visualizations are a useful first pass. They reveal structural patterns in how information is routed. But if someone asks "why did the model say X?", attention alone can't faithfully answer that question. You need deeper tools.

Probing Classifiers

If attention shows you the wiring, probing shows you what's stored in the wires. The idea behind a probing classifier is beautifully simple. Take a trained transformer, freeze its weights, and extract the hidden representations at some layer. Now train a small, simple classifier — typically a linear layer — on top of those frozen representations to perform some linguistic task. If a linear probe can accurately extract part-of-speech tags from layer 4's hidden states, then layer 4's representations must encode syntactic category information in a linearly accessible way.

This technique lets you ask precise questions. Does layer 2 know about word boundaries? Does layer 6 encode dependency parse trees? Does layer 10 encode coreference (that "she" refers to "Alice")? You build a separate probe for each question and measure accuracy layer by layer.

The results, replicated across many studies, paint a consistent picture. Lower layers (1–4 in BERT) encode surface features: character-level patterns, word identity, positional information. Middle layers (5–8) encode syntactic information: POS tags, dependency relations, constituent structure. Upper layers (9–12) encode semantic information: word sense, semantic roles, coreference. This progression is strikingly similar to the traditional NLP pipeline — tokenization → POS tagging → parsing → semantic analysis — except the transformer learned it from raw text without any explicit supervision on these tasks.

This body of work is sometimes called BERTology, and it established several key findings. BERT's representations improve on every probing task from layer 1 through some peak layer, then degrade in the final layers (which specialize for the pretraining objective). Different heads within the same layer often specialize for different linguistic relations. And the information is remarkably robust — even randomly initialized BERT has some structural priors baked in from the architecture itself.

The Probe's Own Intelligence

There's a fundamental tension in probing: if your probe is too powerful, it might learn the task from scratch rather than revealing what the model already knows. A nonlinear probe with 3 hidden layers can learn POS tagging from nearly random representations. That's why most researchers use linear probes — they can only extract information that's already linearly encoded. But even linear probes aren't immune to this criticism. The field uses control tasks (probing representations from an untrained model) and selectivity measures (how much better the probe does on real representations vs. random baselines) to keep results honest.

Mechanistic Interpretability

Probing tells you what information is present. Mechanistic interpretability asks the harder question: how does the model actually compute with that information? The goal is to reverse-engineer the algorithms inside the network, the same way you'd reverse-engineer a compiled binary. You can't read the source code (it was never written), so you trace the computation paths.

The foundational framework comes from Transformer Circuits work (Elhage et al., Anthropic). The key insight is that an attention head can be decomposed into two independent circuits. The QK circuit determines which tokens attend to which — it computes the "matching" pattern. The OV circuit determines what information gets moved when attention fires — it computes the "moving" operation. These two circuits are mathematically independent (QK uses W_Q and W_K; OV uses W_O and W_V), which means you can analyze them separately.

The poster child for mechanistic interpretability is the induction head. This is a specific two-head circuit that implements in-context copying. Suppose the model sees the sequence ... A B ... A. An induction head will predict B as the next token. The mechanism works in two steps: a "previous-token head" in an earlier layer shifts information one position back, so each token's representation includes information about the preceding token. Then the induction head in a later layer uses its QK circuit to find "where did this token A appear before?" and its OV circuit to copy what came after it (B). This circuit emerges reliably in models of various sizes and explains a huge fraction of in-context learning ability.

Circuit discovery generalizes this approach: given a specific model behavior (e.g., correctly completing "The Eiffel Tower is in" → "Paris"), find the minimal subnetwork — the handful of attention heads and MLP layers — responsible for that behavior. Techniques include activation patching (replace one component's output with a corrupted version and measure how much the behavior degrades) and path patching (trace the information flow through specific edges in the computational graph). Think of it like finding which specific transistors on a chip implement a particular logic gate. Hard, painstaking, but it yields genuinely mechanistic understanding.

The Reverse-Engineering Analogy

Mechanistic interpretability is to neural networks what reverse-engineering is to compiled software. You don't have the source code. You can't ask the original developer. You have the binary (weights) and you can run it on any input and observe the output. Techniques like activation patching are the neural network equivalent of setting breakpoints and watching memory. It's slow, but it produces ground-truth understanding — not correlational evidence, but actual causal mechanisms.

Superposition and Sparse Autoencoders

Here's a puzzle. A transformer layer might have a hidden dimension of 4,096. But the world has far more than 4,096 concepts. How does the model represent them all? The answer is the superposition hypothesis: the model stores more features than it has dimensions by representing them as overlapping directions in activation space. Each individual neuron doesn't correspond to a single clean concept. Instead, concepts are encoded as directions in the high-dimensional space, and because most concepts are rarely active at the same time, the model can pack them in without too much interference.

This is why individual neurons are so hard to interpret. Neuron 2,847 in layer 15 might fire for French text, code comments, and recipes involving cheese — not because those things are related, but because they rarely co-occur and can share the same neuron without colliding. The model is doing a form of compressed sensing, trading a small amount of interference for a massive increase in representational capacity.

The breakthrough for dealing with superposition came from Anthropic's Scaling Monosemanticity work, which uses sparse autoencoders (SAEs) to disentangle the superimposed features. The idea: take the dense activation vector from a transformer layer and learn to decompose it into a much larger (say 100×) set of features, with the constraint that only a small number of features are active for any given input. The autoencoder has a wide hidden layer with a sparsity penalty, forcing it to learn a dictionary of independent, interpretable features.

The results were remarkable. Applied to Claude (Anthropic's model), sparse autoencoders found features corresponding to the Golden Gate Bridge, code bugs, deception, sycophantic behavior, safety-relevant content, and thousands of other interpretable concepts. Each feature has a consistent activation pattern across diverse inputs and a clear semantic meaning that humans can label. Even more importantly, you can intervene on these features: clamping the "Golden Gate Bridge" feature to a high value causes the model to insert Golden Gate Bridge references into unrelated conversations.

Why does this matter for AI safety? If you can identify a feature corresponding to deceptive reasoning, you can monitor it during deployment or suppress it. If you can find a "refuse harmful requests" feature, you can ensure it stays active. Sparse autoencoders transform the alignment problem from "we can't see what the model is doing" to "here are specific knobs we can monitor and turn." That's a massive step forward, though we're still far from fully enumerating all features in a large model.

Representation Analysis Tools

While mechanistic interpretability digs deep into circuits, a family of lighter-weight tools lets you inspect what a transformer is "thinking" at each layer without the full reverse-engineering effort.

The Logit Lens is the simplest and most elegant. At each intermediate layer, take the hidden state and project it through the model's final unembedding matrix (the one that normally converts the last layer's output into vocabulary probabilities). This gives you a probability distribution over tokens at every layer, not just the final one. You can literally watch the model's "belief" evolve as information flows through the network. For a factual query like "The capital of France is," early layers might spread probability across many tokens, middle layers might converge on geography-related tokens, and the final layers sharply peak on "Paris." It's like watching a thought form in slow motion.

The Tuned Lens improves on this by training a small affine probe at each layer instead of using the raw unembedding matrix. The raw logit lens is biased because intermediate representations weren't trained to be compatible with the final unembedding — they're in a "different coordinate system." The tuned lens corrects for this, giving more accurate per-layer predictions.

Both tools rest on the residual stream hypothesis. In a transformer, the hidden state is a running sum: each attention layer and each MLP layer adds its contribution to the residual stream. The final output is just the accumulated sum of all these contributions. This means you can decompose the final prediction into per-layer contributions and ask: which layer most changed the model's mind? Which attention head was responsible for shifting probability from "London" to "Paris"?

import torch
from transformers import AutoTokenizer, AutoModelForCausalLM

tokenizer = AutoTokenizer.from_pretrained("gpt2")
model = AutoModelForCausalLM.from_pretrained("gpt2", output_hidden_states=True)

text = "The capital of France is"
inputs = tokenizer(text, return_tensors="pt")

with torch.no_grad():
    outputs = model(**inputs)

# Logit Lens: project each layer's hidden state through the unembedding matrix
unembed = model.lm_head.weight  # (vocab_size, hidden_dim)
hidden_states = outputs.hidden_states  # tuple of (batch, seq, hidden_dim), one per layer

print(f"{'Layer':>6}  {'Top prediction':>20}  {'Prob':>8}")
print("-" * 40)

for layer_idx, hidden in enumerate(hidden_states):
    last_token_hidden = hidden[0, -1, :]  # hidden state at last position
    logits = last_token_hidden @ unembed.T  # project to vocab
    probs = torch.softmax(logits, dim=-1)
    top_token = tokenizer.decode(probs.argmax().item())
    top_prob = probs.max().item()
    print(f"{layer_idx:>6}  {top_token:>20}  {top_prob:>8.4f}")

What about knowledge in FFN layers? The MLP (feed-forward) sublayers in a transformer are increasingly understood as key-value memories. Each row of the first linear layer acts as a "key" that matches certain input patterns, and the corresponding row of the second linear layer acts as a "value" that gets written into the residual stream when that key matches. Dai et al. (2022) showed that individual FFN neurons can store specific factual associations. A single neuron might activate for "Eiffel Tower" and push the residual stream toward representations associated with "Paris" and "France." This finding that knowledge is localized — not diffusely spread across all parameters — is what makes the next technique possible.

Knowledge Editing

If factual knowledge lives in specific MLP layers, can you surgically change it? ROME (Rank-One Model Editing) says yes. Suppose the model believes "The president of the United States is Biden" and you want to update it to a new president without retraining the entire model. ROME identifies which MLP layer stores that association (typically in the middle layers), computes a rank-one update to that layer's weight matrix, and inserts the new fact. One matrix tweak. No gradient descent. No training data.

The technique works by identifying a specific layer where the factual association is most strongly encoded (using causal tracing — a form of activation patching). It then computes the key vector (what input pattern triggers the association) and the desired new value vector (what output the MLP should produce). The weight update is a rank-one outer product of these vectors, applied to the second weight matrix of the MLP. It's elegant and surprisingly effective: the model correctly answers questions about the edited fact and generalizes to paraphrases ("Who leads the US?" also returns the new answer).

MEMIT (Mass-Editing Memory In a Transformer) extends ROME to edit multiple facts simultaneously by distributing updates across several layers. This is important because single-layer edits can interfere with each other when you try to stack them — MEMIT spreads the load to avoid degrading general model performance.

Editing Has Limits

Knowledge editing is powerful but brittle. Edits can have unintended side effects — changing "Eiffel Tower is in Paris" to "Eiffel Tower is in London" might not update related facts like "What country is the Eiffel Tower in?" consistently. Edits also degrade model performance if you make too many. Think of it as patching a binary: a few patches work fine, but hundreds of patches and the program starts crashing in unexpected places. For large-scale corrections, fine-tuning or retrieval-augmented generation remain more robust approaches.

The broader implication of knowledge editing is philosophical as much as practical. The fact that you can locate and surgically modify individual facts inside a neural network challenges the old narrative that these models are impenetrable black boxes. They're not. They're complex — absurdly complex — but they have internal structure, and that structure is becoming legible. Every technique in this section, from attention visualization to sparse autoencoders to ROME, chips away at the black-box problem. We're not done. We probably won't be done for a long time. But the direction is clear: transformers are becoming less mysterious, one circuit, one probe, one feature at a time.
