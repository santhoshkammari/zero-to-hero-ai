GRU

Chapter 10: Sequence Models & Attention 3 subtopics
TL;DR

The Gated Recurrent Unit (GRU) delivers nearly the same performance as LSTM with fewer parameters and simpler architecture. It merges the forget and input gates into a single update gate, drops the separate cell state entirely, and uses a reset gate to control how much history feeds into the candidate. Two gates instead of three, one state vector instead of two — roughly 25% fewer parameters, often comparable results, and faster training. Reach for GRU first when data is limited or speed matters; switch to LSTM if you need extra capacity for very long, complex sequences.

Why GRU? The Case for Simplicity

By now you've seen how LSTM solves the vanishing gradient problem with its cell state highway and three gates. It works beautifully. But take a step back and count the moving parts: a forget gate, an input gate, an output gate, a cell state, and a hidden state. Each gate has its own weight matrix for the input and its own weight matrix for the previous hidden state, plus biases. That's four sets of weight computations per timestep (three gates plus the candidate cell). For a model with hidden size 256 and input size 100, that's over 900,000 parameters in a single LSTM layer.

In 2014, Kyunghyun Cho and colleagues asked a simple question: do we really need all of that? Their answer was the Gated Recurrent Unit (GRU) — a streamlined alternative that collapses the gating machinery down to two gates and eliminates the separate cell state entirely. The result is a model that trains faster, uses less memory, and in many benchmarks performs just as well as LSTM.

The design philosophy is worth appreciating. LSTM was engineered gate by gate over years of research. GRU was designed by asking "what's the minimal gating mechanism that still solves vanishing gradients?" Sometimes less really is more.

The Two Gates

Update Gate — The Blending Knob

The update gate is the heart of the GRU. It controls how much of the previous hidden state to keep versus how much of the new candidate to accept. Think of it as a blending knob on a mixing console. Turn it all the way to one side and you keep the old state untouched (perfect memory). Turn it the other way and you completely overwrite with new information.

Mathematically, at timestep t:

zₜ = σ(Wz · [hₜ₋₁, xₜ])

Here σ is the sigmoid function, so zₜ is a vector of values between 0 and 1. hₜ₋₁ is the previous hidden state, xₜ is the current input, and Wz is the update gate's weight matrix. The brackets [hₜ₋₁, xₜ] mean we concatenate the two vectors before multiplying.

Here's the clever bit: this single gate does the job of both the LSTM's forget gate and input gate. When zₜ is close to 1, the model says "trust the new candidate." When it's close to 0, the model says "keep what I already know." One knob, two jobs.

Reset Gate — Selective Amnesia

The reset gate controls how much of the previous hidden state should be visible when computing the new candidate. It decides: "when I'm formulating what I want to remember next, how much of my past should I even look at?"

rₜ = σ(Wr · [hₜ₋₁, xₜ])

Same structure as the update gate — sigmoid over a linear combination of past state and current input — but with its own weight matrix Wr. When rₜ is close to 0, the candidate is computed almost entirely from the current input, as if the model has no memory. When rₜ is close to 1, the full previous state flows into the candidate computation. This is useful when the model encounters a topic shift — say, a new sentence in a different language — and needs to start fresh.

Computing the New State

Candidate Hidden State

With the reset gate in hand, we compute the candidate hidden state — a proposal for what the new hidden state could be:

h̃ₜ = tanh(W · [rₜ * hₜ₋₁, xₜ])

Notice rₜ * hₜ₋₁ — that's element-wise multiplication. The reset gate modulates the previous hidden state before it gets concatenated with the input. If some dimension of rₜ is near zero, that dimension of the past is effectively erased for the purpose of this computation. The tanh squashes the result to the range [-1, 1], giving us a properly scaled candidate.

Final Hidden State — The Elegant Blend

Now comes the payoff. The final hidden state is a simple linear interpolation controlled by the update gate:

hₜ = (1 - zₜ) * hₜ₋₁ + zₜ * h̃ₜ

Read that again. There is no separate cell state. No output gate deciding what fraction of memory to expose. Just one clean blend between "what I had" and "what I propose." When zₜ is 0, the hidden state passes through unchanged — gradients flow backward without degradation, solving the vanishing gradient problem. When zₜ is 1, the state is fully replaced. In between, you get a smooth mix.

This is the elegant core of GRU. One equation replaces the LSTM's separate cell state update, forget gate application, input gate application, and output gate filtering. Fewer operations, fewer parameters, same expressive power for most tasks.

Why the Interpolation Trick Matters

The formula hₜ = (1 - zₜ) * hₜ₋₁ + zₜ * h̃ₜ guarantees that the coefficients sum to 1. This means the hidden state is always a convex combination of old and new — it can't explode or vanish just from the blending step. This structural constraint provides gradient stability by design, not just by hope.

GRU from Scratch

Let's implement a single GRU cell forward pass so you can see exactly where each gate operates:

import numpy as np

def sigmoid(x):
    return 1 / (1 + np.exp(-x))

def gru_cell_forward(x_t, h_prev, params):
    """Single GRU cell forward pass.

    Args:
        x_t:    current input, shape (input_size,)
        h_prev: previous hidden state, shape (hidden_size,)
        params: dict with weight matrices and biases
    Returns:
        h_next: new hidden state, shape (hidden_size,)
    """
    # Unpack parameters
    Wz, Wr, Wh = params['Wz'], params['Wr'], params['Wh']
    bz, br, bh = params['bz'], params['br'], params['bh']

    # Concatenate previous hidden state and current input
    combined = np.concatenate([h_prev, x_t])

    # Update gate: how much of the new candidate to blend in
    z_t = sigmoid(Wz @ combined + bz)

    # Reset gate: how much past to expose when computing candidate
    r_t = sigmoid(Wr @ combined + br)

    # Candidate hidden state: proposal for new memory
    combined_reset = np.concatenate([r_t * h_prev, x_t])
    h_candidate = np.tanh(Wh @ combined_reset + bh)

    # Final hidden state: blend old and new
    h_next = (1 - z_t) * h_prev + z_t * h_candidate

    return h_next

# --- Demo ---
np.random.seed(42)
input_size, hidden_size = 4, 3

# Initialize small random weights
def rand(rows, cols):
    return np.random.randn(rows, cols) * 0.1

params = {
    'Wz': rand(hidden_size, hidden_size + input_size),
    'Wr': rand(hidden_size, hidden_size + input_size),
    'Wh': rand(hidden_size, hidden_size + input_size),
    'bz': np.zeros(hidden_size),
    'br': np.zeros(hidden_size),
    'bh': np.zeros(hidden_size),
}

h = np.zeros(hidden_size)
sequence = [np.random.randn(input_size) for _ in range(5)]

for t, x_t in enumerate(sequence):
    h = gru_cell_forward(x_t, h, params)
    print(f"t={t}  h={np.round(h, 4)}")

Count the weight matrices: three (Wz, Wr, Wh). An LSTM would have four (Wf, Wi, Wc, Wo). That's where the parameter savings come from.

GRU vs LSTM — Side by Side

Let's put the two architectures next to each other and count what's different:

LSTM: 3 gates (forget, input, output) + cell state + hidden state. Four weight matrices. Two state vectors carried across timesteps. The cell state acts as a protected memory highway; the output gate filters what the next layer sees.

GRU: 2 gates (update, reset) + hidden state only. Three weight matrices. One state vector carried across timesteps. The hidden state itself is the memory — no separate highway, no output filtering.

The parameter difference is straightforward. For hidden size h and input size x, each gate requires a weight matrix of size h × (h + x) plus a bias of size h. LSTM has 4 such sets, GRU has 3. That's a 25% reduction in parameters. For large models, that translates directly into less GPU memory and faster training.

Let's verify this with PyTorch:

import torch.nn as nn

input_size  = 64
hidden_size = 128
num_layers  = 1

lstm = nn.LSTM(input_size, hidden_size, num_layers, batch_first=True)
gru  = nn.GRU(input_size, hidden_size, num_layers, batch_first=True)

lstm_params = sum(p.numel() for p in lstm.parameters())
gru_params  = sum(p.numel() for p in gru.parameters())

print(f"LSTM parameters: {lstm_params:,}")
print(f"GRU  parameters: {gru_params:,}")
print(f"GRU saves:       {lstm_params - gru_params:,} "
      f"({100*(lstm_params - gru_params)/lstm_params:.1f}%)")

# Output:
# LSTM parameters: 99,328
# GRU  parameters: 74,496
# GRU saves:       24,832 (25.0%)

Exactly 25% fewer parameters. The API is nearly identical — swap nn.LSTM for nn.GRU and your training loop barely changes. The only difference: GRU's forward method returns (output, h_n) while LSTM returns (output, (h_n, c_n)) because there is no cell state to return.

When to Choose GRU vs LSTM

This is the question everyone asks, and the honest answer is: it depends, but less than you'd think. Here's a practical decision framework:

Choose GRU when: your dataset is small or moderate (fewer parameters means less overfitting risk), training speed matters (GRU is typically 15-20% faster per epoch), you're prototyping and want quick iteration, or your sequences are relatively short (under a few hundred timesteps). GRU is also a good default when you're not sure — start simple, add complexity only if needed.

Choose LSTM when: you're dealing with very long sequences where the separate cell state highway provides measurably better gradient flow, your task involves complex nested dependencies (like long-range grammatical agreement in language), you have plenty of data to support the extra parameters, or your benchmarks show LSTM outperforming GRU on your specific problem.

In practice: dozens of papers have compared the two across tasks like language modeling, machine translation, speech recognition, and music generation. The consistent finding is that neither dominates universally. Performance differences are usually within a small margin, and the winner flips depending on the task and hyperparameter tuning. When in doubt, try both — a GRU run is cheap enough to serve as a quick baseline.

The Real-World Shortcut

In modern practice, transformers have largely replaced both LSTM and GRU for large-scale tasks. But recurrent models remain valuable for edge deployment (small model size), streaming data (process one token at a time without caching the full sequence), and time-series forecasting. In these niches, GRU's efficiency advantage over LSTM is especially meaningful because every parameter and every millisecond counts.

GRU in Practice

A few practical notes from real-world usage. For hidden size, 64-256 is common for small tasks (sentiment analysis, simple time series), 256-512 for medium tasks (language modeling on modest corpora), and 512+ when you're stacking multiple layers. Speaking of stacking, 1-3 GRU layers is typical. Beyond that, you're likely better off with a transformer.

Dropout between GRU layers (not within a single layer's recurrence) is standard practice — PyTorch's nn.GRU has a dropout parameter that handles this for multi-layer configurations. Bidirectional GRUs, where you run one GRU forward and another backward over the sequence, are popular for tasks where you have access to the full sequence upfront (like text classification) rather than streaming prediction.

Gradient clipping (typically to a max norm of 1-5) is still recommended with GRU, just as with LSTM. The gating mechanism helps with vanishing gradients, but exploding gradients are a separate issue that clipping addresses.

One last thing: GRU's simplicity makes it a great teaching tool. If you understand how the update and reset gates interact, you understand the fundamental principle behind all gated recurrent architectures — that learned, differentiable gates controlling information flow are the key to training deep sequence models. LSTM is the same idea with more knobs. Transformers are the same idea with attention replacing recurrence. The gating intuition carries forward everywhere.
