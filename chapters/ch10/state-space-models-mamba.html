State space models (SSMs) challenge Transformer's quadratic attention cost. Borrowed from control theory (modeling dynamic systems since 1960s) — SSMs process sequences with linear complexity while capturing dependencies thousands of steps back. S4 (Gu et al., 2022) proved concept; Mamba (Gu & Dao, 2023) made it competitive. This section builds both from first principles.

A Thermostat That Learns

Imagine building smart thermostat. Not fixed-schedule kind — one that understands thermal dynamics of house. Monitors temperature every minute, predicts what happens next.

House at 68°F. Furnace blasting at some intensity. Outside 30°F and dropping. What will temperature be one minute from now?

Following earlier RNN sections, you might reach for recurrent network. Feed current temperature, maintain hidden state, predict next value. Works. But engineers designing actual thermostats solved this since 1960s with different tool: state space equations. Describe how system's internal state evolves over time, how inputs push it around, how we observe behavior.

Thermostat = running example through this section. Starts tiny — one room, one sensor, one input. Grows into something much more interesting by end.

The State Space Equations

Core math. Continuous-time state space model described by two equations:

# The state space equations (math, not code):
#
# x'(t) = A · x(t) + B · u(t)     ← how the hidden state evolves
# y(t)  = C · x(t) + D · u(t)     ← what we observe

Unpacking each symbol via thermostat. x(t) = hidden state — everything system "remembers" at time t. For thermostat: small vector [wall temperature, air temperature, residual heat in furnace]. Can't directly measure all, but they influence what happens next.

u(t) = input signal — furnace setting, maybe outside temperature reading. y(t) = output — temperature reading on thermostat display.

Matrices: A governs how state evolves on its own. In thermostat, captures physics: heat dissipates through walls at some rate, furnace core cools when off, air temperature drifts toward wall temperature. B maps input to state — how much cranking furnace changes internal heat. C maps state to observation — sensor reads air temperature, not wall temperature. D = direct skip connection from input to output, often dropped in practice.

First equation: rate of change of state (x'(t) = derivative) depends on current state position and arriving input. Second equation: output = linear readout of state.

First reaction: "fancy linear system, how could it model language?" Key realization: state x(t) = compressed summary of entire input history. Every past input left fingerprint on x(t), decayed and mixed by A matrix. That's what makes SSMs relevant for sequence modeling.

From Continuous to Discrete

Those equations describe system evolving in continuous time — temperature changing smoothly. But thermostat samples once per minute. Language model processes one token at a time. Need to convert continuous equations into discrete steps.

Process called discretization. Most common technique: zero-order hold. Idea: assume input u(t) stays constant between sample points. Furnace set to "high" at time k? Stays "high" until next sample at k+1.

Under that assumption, solve differential equation exactly → discrete versions of A and B:

# Zero-order hold discretization:
#
# Ā = exp(A · Δ)                   ← matrix exponential
# B̄ = (Ā - I) · A⁻¹ · B           ← integrated input effect
#
# Now the recurrence is:
# x_k = Ā · x_{k-1} + B̄ · u_k     ← new state from old state + input
# y_k = C · x_k                    ← output readout

That recurrence: x_k = Ā · x_{k-1} + B̄ · u_k. New state = linear combination of previous state + current input. Squint and it looks exactly like vanilla RNN — hidden state × weight matrix + input × another weight matrix.

Not coincidence. SSM is a linear RNN. Two crucial differences make it much more powerful than earlier RNNs. First: A matrix isn't random — comes from carefully designed continuous-time system. Second difference even more important — explored next.

About step size Δ: controls how much continuous dynamics happen between each discrete step. Large Δ = big jumps, system evolves lot between samples. Small Δ = tiny steps, system barely changes. Think of it as "playback speed" of underlying continuous process. This knob becomes very important in Mamba.

The Convolution–Recurrence Trick

This is the part that made me sit up.

We have recurrence: x_k = Ā · x_{k-1} + B̄ · u_k. Unroll from beginning, x_0 = 0 (empty state). Step 1: state = B̄ · u_1. Step 2: Ā · B̄ · u_1 + B̄ · u_2. Step 3: Ā² · B̄ · u_1 + Ā · B̄ · u_2 + B̄ · u_3.

Output at step k, after applying C:

# y_k = C · Ā^(k-1) · B̄ · u_1
#     + C · Ā^(k-2) · B̄ · u_2
#     + ...
#     + C · B̄ · u_k
#
# This is a weighted sum of all past inputs!
# The weights form a fixed kernel:
#
# K = [C·B̄,  C·Ā·B̄,  C·Ā²·B̄,  ...,  C·Ā^(L-1)·B̄]
#
# And y = K ∗ u    (where ∗ means convolution)

Key insight: output at every position = weighted combination of all previous inputs. Weights determined entirely by matrices A, B, C and step size Δ. Weights form fixed pattern — a convolution kernel. Convolutions computed in parallel via Fast Fourier Transform (FFT), O(n log n) time.

This is the magic: same model has two faces.

During training: entire sequence available → compute kernel K once, convolve with full input. FFT makes it parallel and fast — no sequential bottleneck, no vanishing gradients from unrolling through time.

During inference: tokens arrive one at a time → run recurrence. Previous state × Ā, add B̄ × new input, read out with C. Each step O(1) in time and memory. State = fixed size regardless of tokens processed.

Compare to earlier sequence models. RNNs = recurrences — O(1) per step at inference, but painfully sequential during training, vanishing gradients. Transformers = parallel during training (great!), but attention costs O(n²) for length n, KV cache grows linearly at inference.

SSMs get O(n log n) parallel training and O(1)-per-step inference. "There has to be a catch." There is. Not where expected.

HiPPO — Teaching the State to Remember

Catch = A matrix. Random A → recurrence either explodes (eigenvalues > 1 amplify states exponentially) or decays (eigenvalues < 1 shrink old inputs to zero). Either way, model forgets distant past. Same vanishing/exploding gradient problem that plagued vanilla RNNs.

Back to thermostat. System needs to remember window opened 3 hours ago — room still hasn't recovered. Random A matrix loses that info within few time steps. Need A matrix specifically designed to preserve long-range history.

Enter HiPPO — High-order Polynomial Projection Operator. Name sounds intimidating, intuition isn't.

Think of it: watching stock price over course of day. At any moment, want to recall general shape of price curve over entire day so far — not every tick, but trend, bumps, overall trajectory. One way: fit polynomial to price history. Degree-0 = average. Degree-1 = average + slope. Degree-2 adds curvature. Higher degrees capture finer details.

HiPPO says: at every time step, maintain coefficients representing best polynomial fit to entire input history. Uses Legendre polynomials — family of orthogonal polynomials studied for centuries. Orthogonal = each coefficient captures independent information about history, no redundancy.

Beautiful part: specific matrix A (HiPPO matrix) exists such that running recurrence x_k = Ā · x_{k-1} + B̄ · u_k with this particular A → state x_k automatically encodes optimal polynomial approximation of everything seen so far. State is, provably, best possible fixed-size compression of input history.

Still developing intuition for why this particular matrix has this property — derivation involves measure theory and function approximation. Practical takeaway: models initialized with HiPPO capture dependencies thousands of steps back, where vanilla RNNs and even LSTMs fail completely.

HiPPO's A matrix isn't learned weight in usual sense. More like architectural choice — carefully designed inductive bias giving model head start on remembering. Difference between memorizing phone number by staring (random A) vs. using mnemonic system (HiPPO). Same brain, radically different retention.

S4 — Making It Practical

HiPPO gave right A matrix. Convolution–recurrence duality gave two compute modes. Engineering problem remained: computing kernel K = [C·B̄, C·Ā·B̄, C·Ā²·B̄, ...] requires repeated matrix powers of Ā — expensive for large state dimensions and long sequences.

S4 — Structured State Spaces for Sequence Modeling (Gu et al., 2022) — solved this by exploiting HiPPO matrix structure. HiPPO A has low-rank-plus-diagonal structure. S4 decomposes kernel computation using Cauchy kernels from numerical linear algebra, reducing cost dramatically. Details dense, but effect practical: compute SSM kernel for thousands of tokens in reasonable time on GPU.

S4 = proof SSMs could compete with transformers on real benchmarks. Long Range Arena — suite of tasks testing long-range dependency modeling — became proving ground. Tasks: classifying document by full content (1K–4K length), determining if two points connected by path in image (1K–16K), matching patterns in long byte-level sequences. Transformers struggled on several. S4 crushed them, often by wide margins, fraction of compute.

But S4 had fundamental limitation. Time to confront it.

Matrices A, B, C in S4 are fixed. Don't change based on what model sees. Every token processed with exact same dynamics. "the" and "emergency" pass through identical state transitions. Model has no ability to selectively pay more attention to some inputs and less to others.

For thermostat: processing "window opened" with same weight as "furnace humming at steady state." One demands immediate state changes. Other = background noise. Fixed-parameter SSM can't tell difference.

Mamba — The Selective Revolution

Mamba (Gu & Dao, December 2023) makes one change to SSM framework. One change. Transforms entire field.

In S4: B, C, Δ learned once during training, applied identically to every token. In Mamba: B, C, and Δ become functions of current input.

# S4 (fixed parameters):
# x_k = Ā · x_{k-1} + B̄ · u_k        ← same Ā, B̄ for every token

# Mamba (selective parameters):
# B_k = Linear(u_k)                    ← B depends on this token
# C_k = Linear(u_k)                    ← C depends on this token
# Δ_k = softplus(Linear(u_k))          ← step size depends on this token
# Ā_k, B̄_k = discretize(A, B_k, Δ_k)  ← discretize with token-specific params
# x_k = Ā_k · x_{k-1} + B̄_k · u_k    ← dynamics change per token!

Connecting back to thermostat. S4: every temperature reading processed with same dynamics — 68°F during normal operation updates state same way as sudden spike to 95°F indicating fire. Mamba: system looks at incoming reading and adjusts behavior on fly. Boring expected reading? Small Δ — barely update state. Alarming anomaly? Large Δ — write strongly into memory.

Step size Δ = crucial lever. Large Δ → discretized Ā_k shrinks toward zero (exp(A · Δ) decays faster) → model forgets more old state, writes new input forcefully. Small Δ → model preserves existing state, barely notices new input.

Confusing at first. Expected "selective" to mean attention scores — model explicitly comparing token pairs. Not that. More subtle. Selection happens at write step: model decides, per token, how strongly to imprint onto running state. Important tokens written with large Δ. Irrelevant tokens pass through with almost no effect. Content-based filtering — O(n) complexity instead of O(n²).

Mamba block also includes short convolution (kernel size 3–4) before selective scan. Handles very local patterns — adjacent-token interactions — cheaply. Tiny preprocessing step: convolution captures "the cat" as local bigram, SSM captures "the cat mentioned three paragraphs ago."

Full Mamba block has gated architecture. Input split into two paths: one through convolution + selective SSM, other held as gate. Multiplied together at end, similar to gated linear units. Gives model way to modulate SSM output with direct signal from input.

The Hardware Story

Where things get interesting — Mamba becomes systems paper, not just math paper.

Remember convolution–recurrence duality from S4? Relied on kernel being fixed — same for every token. Once B, C, Δ depend on input, kernel changes at every position. Can't precompute single kernel and use FFT. Convolution mode gone.

Without convolution shortcut, stuck with recurrence: x_k = Ā_k · x_{k-1} + B̄_k · u_k. Recurrences are sequential — step k depends on step k-1. Same problem that made RNN training slow.

Mamba's solution: hardware-aware parallel scan. Parallel scan (parallel prefix sum) = well-known algorithm from 1980s — computes cumulative operations like running sums in O(log n) parallel steps instead of O(n) sequential. Mamba adapts this to SSM recurrence.

But knowing algorithm isn't enough. Modern GPUs have memory hierarchy — registers (fastest, tiniest), SRAM on each streaming multiprocessor (fast, small, ~20MB total on A100), and HBM/high-bandwidth memory (large, 40–80GB, much slower). Naive parallel scan bounces data between HBM and SRAM repeatedly, wasting time on memory transfers instead of computation.

Tri Dao (co-author of both FlashAttention and Mamba — not coincidence) designed selective scan kernel to keep state in SRAM. Sequence split into chunks fitting in SRAM. Within each chunk, scan runs entirely in fast on-chip memory. Between chunks, only boundary states move to HBM. Computation fused — discretization, scan, gating, output projection all happen in one kernel call, minimizing memory round-trips.

Mental model for memory hierarchy: doing long math problem. Scratch paper fits on desk (SRAM) → work fast. Keep walking to filing cabinet across room (HBM) for intermediate results → crawl. Mamba keeps scratch paper on desk.

Result: Mamba's selective scan runs ~3× throughput of equivalent transformer at sequence length 2K. Gap widens with longer sequences. At 64K tokens, advantage closer to 5×.

Mamba vs. Transformer

Side-by-side comparison. Not declaring winner — honest answer is "depends" — but understanding trade-offs.

Training cost. Transformer self-attention computes n × n score matrix. O(n²) in time and memory. 512 tokens: manageable. 100K tokens (full codebase or novel): brutal. Mamba's selective scan = O(n). Double sequence, double cost.

Inference cost. Generating one token at a time, transformer must consult KV cache — every previously generated key and value. Cache grows linearly with context. Generating 50,000th token requires reading 50,000 key-value pairs. Mamba maintains fixed-size state vector regardless of tokens processed. 50,000th token costs same as 5th.

Long-range retrieval. Where transformers shine. Attention gives every token direct path to every other token. Answer buried in token 47 of 10,000-token document? Attention looks right at it. Mamba must have compressed info from token 47 into fixed-size state — whether compression preserved right details depends on learned dynamics. Practice: Mamba does well on most long-range benchmarks, but "needle in a haystack" retrieval gives attention an edge.

In-context learning. Transformers remarkably good at learning patterns from prompt examples. Ability seems connected to attention's explicit retrieval and comparison. Mamba can do in-context learning too, but results less studied, sometimes less robust. Active research area.

Ecosystem. Transformers have years of accumulated engineering wisdom. HuggingFace, vLLM, TensorRT, quantization tools, RLHF pipelines, distributed training recipes — all built around attention. Mamba's ecosystem growing but young. Shipping to production tomorrow? Transformer ecosystem = significant practical advantage.

Proven scale. GPT-4, Claude, Gemini, LLaMA 3 — every frontier model = transformer. Mamba validated convincingly up to ~3B parameters, some results at larger scales. Scaling laws encouraging but not proven at 100B+ where transformers show emergent capabilities.

Neither architecture dominates every axis. Transformers better at precise retrieval; Mamba better at long-sequence efficiency. Transformers have battle-tested ecosystem; Mamba has more favorable scaling trajectory. Exactly the situation that leads to hybrids — which is what happened.

Mamba-2 and the Duality Theorem

Six months after Mamba, Gu and Dao dropped another paper changing how we think about SSM–attention relationship. Mamba-2 introduces Structured State Space Duality (SSD): mathematical proof that selective SSMs and certain attention form are, under specific parameterizations, computing same thing.

Intuition: attention computes weighted sum of values, weights from comparing queries and keys. SSM recurrence computes weighted sum of past inputs, weights from state transition matrices. Mamba-2 shows both computations fit common framework — dual views of same operation.

Why matters: lets you pick best implementation per setting. Short sequences → attention-like view (materialized matrix product) faster on GPUs. Long sequences → SSM recurrence more efficient. Mamba-2 dynamically chooses based on hardware and sequence length.

Practical result: Mamba-2 is 2–8× faster than Mamba-1 on typical workloads, same or better quality. Implementation cleaner — connection to attention makes it easier to use existing transformer infrastructure (optimizers, parallelism, quantization) with SSM models.

Full SSD proof involves structured matrices and semi-separable forms — still working through it. High-level takeaway is profound: attention and state spaces are not rival paradigms. They're different views of same underlying computation. Debate isn't "which is better?" but "which implementation faster for this setting?"

Hybrid Architectures

If SSMs and attention are two views of same thing, obvious question: why not use both?

Jamba (AI21 Labs, 2024) = clearest answer. 52B parameter model interleaving Mamba layers with transformer layers, wrapped in Mixture of Experts (MoE). ~12B parameters active per token. 256K-token context window.

Architecture: ~1 attention layer for every 7 Mamba layers. Mamba layers handle bulk of sequence processing — compressing information, maintaining running state, heavy lifting of "reading" input. Occasional attention layer = precision instrument — when model needs specific fact buried in context, attention's direct token-to-token path handles retrieval.

Back to thermostat one last time. Monitoring building with 1,000 sensors over full year. Most readings follow predictable patterns — SSM layers handle this efficiently, compressing months of routine data. But occasionally need specific answer: "Temperature in room 307 at 2:47 PM on March 15th?" Attention layer reaches back and pulls exact data point.

Jamba isn't only hybrid. Google DeepMind's Griffin and Hawk use recurrence-based layers competing with transformers on standard benchmarks. RWKV takes related approach with linear attention, scaling to 14B+. Trend clear: future isn't "transformers vs. SSMs." Future = architectures using each mechanism where strongest.

# Conceptual hybrid: Jamba-style interleaving
class HybridModel:
    def __init__(self, d_model, n_layers, attn_every=8):
        self.layers = []
        for i in range(n_layers):
            if (i + 1) % attn_every == 0:
                # Sparse attention layers for precise retrieval
                self.layers.append(AttentionLayer(d_model))
            else:
                # Mamba layers for efficient sequence compression
                self.layers.append(MambaLayer(d_model))

    def forward(self, tokens):
        x = embed(tokens)
        for layer in self.layers:
            x = layer(x)     # Each layer: either fast SSM or precise attention
        return x

# The ratio matters — Jamba found ~1:7 (attention:mamba)
# captures most of attention's retrieval ability
# at a fraction of the compute

Why This Matters

You might be wondering: if transformers work well enough, why should I care?

Because the sequences are getting longer. Codebases are hundreds of thousands of tokens. Genomic sequences — DNA strands where a model needs to understand regulatory elements — can be millions of base pairs. Audio waveforms at 16kHz produce 960,000 samples per minute. A full day of sensor data from our imaginary building thermostat, sampled every second, is 86,400 time steps.

At O(n²), attention on a million-token sequence would require roughly a trillion operations for a single layer. At O(n), the SSM handles it with a million. That's not an optimization — it's the difference between possible and impossible.

There are domains where SSMs already make more practical sense than transformers: long-document processing, time-series forecasting at high frequency, genomics and protein modeling, continuous audio processing, and any application where you need inference with constant memory — like running a language model on an edge device where the KV cache can't grow without bound.

We started with a thermostat and the control theory equations that describe how systems evolve over time. We discretized those equations and discovered they look like a linear RNN — but one with a mathematical backbone that gives it two faces: a convolution for fast parallel training and a recurrence for efficient streaming inference. We learned that the HiPPO framework provides a principled initialization that lets the state remember the distant past, and that S4 made this practically computable. Then Mamba came along and added the ability to select — to decide, based on content, what's worth remembering and what's noise. And we saw the field converging toward hybrid architectures that use the best tool for each layer.

My hope is that the next time you see a paper comparing "linear complexity" sequence models to transformers, or someone mentions Mamba in a discussion about LLM efficiency, instead of that vague sense of "I should learn about this eventually," you'll have a clear mental model of what's happening under the hood — from the state equations all the way up to the hardware-aware scan.
