Convergence Rates of Optimization Algorithms

TL;DR

Convergence rates tell you how fast an optimizer shrinks the gap between your current loss and the best possible loss. Gradient descent on smooth convex problems converges at O(1/t). Strong convexity speeds that to linear (exponential) convergence. SGD is slower — O(1/√t) for convex — because of gradient noise. Nesterov acceleration achieves the provably optimal O(1/t²) rate for first-order methods. Adaptive methods like Adam are fast in practice but have known convergence failures that AMSGrad patches. For non-convex neural network landscapes, we settle for finding approximate stationary points and rely on structure like the PL condition to recover faster rates. Lower bounds prove no algorithm can beat Nesterov on convex problems using only gradient queries.

Why Convergence Rates Matter

You're training a language model. After 10,000 steps the loss is still wobbling. Should you wait another 10,000 steps? Another 100,000? Without understanding convergence rates, you're guessing. And guessing costs GPU hours — which cost real money.

A convergence rate is the mathematical guarantee for how quickly an optimization algorithm closes the gap between your current solution and the optimal one. It's the difference between "this will finish overnight" and "this will finish next week." Every time you pick an optimizer, a learning rate, or a batch size, you're implicitly making a choice that affects convergence. Let's make those choices deliberate.

Gradient Descent Convergence

We start with the simplest setting: full-batch gradient descent on a deterministic objective. You compute the exact gradient over all your data, take a step, repeat. No randomness. No mini-batches. This is the baseline everything else is compared against.

Smooth Convex Functions: The O(1/t) Rate

Suppose your loss function f is convex and has a smooth gradient. Smooth here has a precise meaning: the gradient doesn't change too abruptly. Formally, the Lipschitz gradient condition says that ‖∇f(x) − ∇f(y)‖ ≤ L · ‖x − y‖ for all x and y, where L is the Lipschitz constant. Think of L as a speed limit on how fast the gradient can twist and turn. A small L means the landscape is gently curved. A large L means sharp valleys and ridges.

Under this condition, gradient descent with a fixed step size of η = 1/L gives you: f(x_t) − f(x*) ≤ O(1/t). After t steps, your loss is within roughly 1/t of optimal. Double your compute, halve your error. That's the sublinear rate for smooth convex problems.

Step Size Selection: The 1/L Rule

Why η = 1/L? If you step too aggressively (η > 2/L), the iterates oscillate and diverge. If you step too conservatively (η ≪ 1/L), you converge but agonizingly slowly. The sweet spot is 1/L. In practice, you rarely know L exactly, so you use line search or just tune the learning rate — but the 1/L rule is the theory behind the tuning.

Strongly Convex Functions: Linear Convergence

Now add another assumption: the function curves upward everywhere, not just being convex but strongly convex with parameter μ > 0. This means f(x) ≥ f(y) + ∇f(y)ᵀ(x − y) + (μ/2)‖x − y‖² for all x, y. The function bowls upward at least as fast as a quadratic with curvature μ.

With strong convexity, gradient descent converges exponentially fast: f(x_t) − f(x*) ≤ (1 − μ/L)^t · (f(x₀) − f(x*)). This is called linear convergence (confusing name — it means linear on a log scale, which is exponential on a normal scale). Each step multiplies the error by a fixed factor less than one. This is dramatically faster than O(1/t).

Condition Number and Convergence Speed

The ratio κ = L/μ is the condition number. It measures how "elongated" the level sets of your objective are. κ = 1 means perfectly round contours — gradient descent goes straight to the optimum. κ = 1,000,000 means extremely elongated ellipses — gradient descent zigzags painfully. The convergence factor is (1 − 1/κ), so large κ means each step barely makes progress. Most real problems have enormous condition numbers, which is why plain gradient descent struggles.

ℹ️ Key Insight

The condition number κ = L/μ is the single most important quantity for understanding optimization speed. Preconditioning, normalization, and adaptive methods all work by effectively reducing κ. When someone says "this problem is ill-conditioned," they mean κ is huge and plain gradient descent will be painfully slow. 

# Comparing GD convergence: smooth convex vs. strongly convex
import numpy as np

np.random.seed(42)

# Smooth convex (but not strongly convex): f(x) = (1/2)||Ax - b||^2 with rank-deficient A
n, d = 200, 50
A = np.random.randn(n, d)
b = np.random.randn(n)
# Make it strongly convex by adding regularization
lam = 0.01  # small regularization = large condition number

def f_convex(x):
    r = A @ x - b
    return 0.5 * np.dot(r, r)

def grad_convex(x):
    return A.T @ (A @ x - b)

def f_strong(x):
    r = A @ x - b
    return 0.5 * np.dot(r, r) + 0.5 * lam * np.dot(x, x)

def grad_strong(x):
    return A.T @ (A @ x - b) + lam * x

# Compute Lipschitz constant L (largest eigenvalue of A^T A)
eigvals = np.linalg.eigvalsh(A.T @ A)
L_convex = eigvals[-1]
L_strong = L_convex + lam
mu = eigvals[0] + lam  # smallest eigenvalue + regularization

print(f"Condition number (strong convex): {L_strong / mu:.0f}")

# Run GD on both
T = 500
x_c, x_s = np.zeros(d), np.zeros(d)
f_star_c = f_convex(np.linalg.lstsq(A, b, rcond=None)[0])
x_star_s = np.linalg.solve(A.T @ A + lam * np.eye(d), A.T @ b)
f_star_s = f_strong(x_star_s)

gaps_convex, gaps_strong = [], []

for t in range(T):
    gaps_convex.append(f_convex(x_c) - f_star_c)
    gaps_strong.append(f_strong(x_s) - f_star_s)
    x_c = x_c - (1.0 / L_convex) * grad_convex(x_c)
    x_s = x_s - (1.0 / L_strong) * grad_strong(x_s)

# After 500 steps:
print(f"Convex gap at t=500:  {gaps_convex[-1]:.6f}")    # Still > 0, O(1/t) decay
print(f"Strong gap at t=500:  {gaps_strong[-1]:.2e}")     # Near machine precision

Run that code and watch the difference. The strongly convex version (with regularization) reaches near-zero gap in hundreds of steps. The plain convex version is still grinding away. That's the power of linear versus sublinear convergence.

Stochastic Gradient Descent Convergence

Full-batch gradient descent is a luxury. When you have millions of training examples, computing the full gradient each step is absurd. Stochastic gradient descent (SGD) estimates the gradient from a random subset, making each step cheap. But the price of that cheapness is noise — and noise changes everything about convergence.

SGD for Convex: O(1/√t) Rate

For smooth convex functions, SGD with a diminishing learning rate achieves f(x̄_t) − f(x*) ≤ O(1/√t), where x̄_t is typically the average of all iterates. That's much worse than the O(1/t) of full GD. The noise in the gradient estimate prevents SGD from getting close to the optimum quickly. You need four times as many steps to halve your error.

SGD for Strongly Convex: O(1/t) Rate

Strong convexity helps SGD too. With learning rate η_t = O(1/(μt)), SGD achieves E[f(x_t)] − f(x*) ≤ O(1/(μt)). Better than O(1/√t), but still worse than the exponential convergence of full GD on strongly convex functions. The gradient noise creates a "noise floor" that prevents exponential convergence no matter what.

Learning Rate Schedules

The learning rate schedule is not a minor detail — it's fundamental to SGD convergence. A diminishing schedule like η_t = η₀/t satisfies the Robbins-Monro conditions: the sum of learning rates diverges (so you can reach any point) but the sum of squared learning rates converges (so the noise dies out). In practice, people use step decay (drop η by 10x at fixed epochs), cosine annealing (η_t = η₀ · (1 + cos(πt/T))/2), or warm-up followed by decay. Cosine schedules have become especially popular in deep learning because they naturally provide a smooth transition from exploration to fine-grained convergence.

Mini-Batch Size and Critical Batch Size

Making the mini-batch larger reduces gradient noise variance by a factor of 1/B (where B is batch size). This helps — up to a point. The critical batch size is where further increases in B stop giving proportional speedups. Below the critical batch size, doubling B roughly halves the number of steps needed. Above it, you're wasting compute on noise reduction that isn't the bottleneck anymore. For modern language models, the critical batch size can be in the millions of tokens.

Variance Reduction: SVRG and SAGA

What if you want SGD's cheap iterations and GD's fast convergence? Variance reduction methods do exactly this. SVRG (Stochastic Variance Reduced Gradient) periodically computes the full gradient and uses it to correct the stochastic gradient, reducing variance without increasing per-step cost much. SAGA maintains a table of past gradients and uses them for correction. Both achieve linear convergence on strongly convex problems — matching full GD — while keeping the per-step cost similar to SGD. In theory these are amazing. In practice, they're less popular for deep learning because the non-convex landscape makes the theoretical guarantees weaker, and Adam-style methods often work well enough. 

# SGD convergence: O(1/sqrt(t)) vs O(1/t) on strongly convex
import numpy as np

np.random.seed(0)
n, d = 1000, 20
A = np.random.randn(n, d)
x_true = np.random.randn(d)
b = A @ x_true + 0.1 * np.random.randn(n)
lam = 0.1

# Optimal solution
x_star = np.linalg.solve(A.T @ A + lam * n * np.eye(d), A.T @ b)
def f(x):
    return 0.5 * np.sum((A @ x - b)**2) / n + 0.5 * lam * np.dot(x, x)

f_star = f(x_star)

# SGD with diminishing LR: eta_t = eta0 / (1 + t * decay)
T = 5000
x_sgd = np.zeros(d)
eta0 = 0.01
gaps = []

for t in range(T):
    gaps.append(f(x_sgd) - f_star)
    i = np.random.randint(n)  # pick one sample
    grad_i = (A[i] @ x_sgd - b[i]) * A[i] + lam * x_sgd
    eta_t = eta0 / (1 + 0.001 * t)
    x_sgd = x_sgd - eta_t * grad_i

# Check convergence behavior
for checkpoint in [100, 500, 1000, 5000]:
    print(f"t={checkpoint:5d}  gap={gaps[checkpoint-1]:.6f}")

# You'll see the gap shrink, but much slower than full GD
# and with fluctuations from the stochastic noise.
⚠️ Warning

A constant learning rate in SGD does not converge to the optimum — it converges to a neighborhood whose radius depends on η and the gradient variance. If you see your training loss plateau and oscillate, this is often why. You must decay the learning rate to converge, or accept the noise floor and use the iterate "near" the optimum.

Accelerated Methods

Plain gradient descent on smooth convex functions gives O(1/t). Can we do better? Yurii Nesterov showed that yes, we can — and he showed exactly how much better.

Nesterov's Accelerated Gradient: O(1/t²) Rate

Nesterov's accelerated gradient (NAG) achieves f(x_t) − f(x*) ≤ O(L‖x₀ − x*‖² / t²). That's a quadratic improvement over standard gradient descent. The idea is deceptively minimal: instead of computing the gradient at the current point, compute it at a "lookahead" point that extrapolates from the current momentum direction.

The update looks like this: first compute y_t = x_t + β_t(x_t − x_{t-1}), a momentum step that overshoots the current position. Then update x_{t+1} = y_t − η∇f(y_t). The momentum coefficient β_t is carefully chosen — roughly (t−1)/(t+2) for the convex case. The intuition? Imagine a ball rolling downhill. It doesn't stop at each point to recompute direction — it uses its momentum to look ahead, which lets it navigate long narrow valleys much faster.

Optimality of Nesterov: Lower Bounds

Here's the remarkable part: O(1/t²) is optimal for first-order methods on smooth convex functions. Nesterov himself proved the matching lower bound. No algorithm that only uses gradient evaluations can do better in the worst case. When you use NAG, you're running against the theoretical speed limit.

Momentum and Condition Number

For strongly convex functions, Nesterov acceleration improves the convergence from (1 − 1/κ)^t to (1 − 1/√κ)^t. When κ is 10,000, plain GD's per-step factor is 0.9999, while Nesterov's is 0.99. That's a 100x improvement in the number of steps to reach a given accuracy. The heavy ball method (Polyak, 1964) adds a fixed momentum term β(x_t − x_{t-1}) to the gradient step. It achieves the same rate on quadratics but is less robust than Nesterov on general convex functions — it can diverge if the momentum parameter is badly tuned.

GD vs. Nesterov: A Rule of Thumb

If gradient descent needs κ steps to converge on a strongly convex problem, Nesterov needs √κ steps. For a problem with condition number 1,000,000, that's a reduction from 1,000,000 iterations to 1,000. This is why momentum-based methods are standard in deep learning — even though the theory is for convex functions, the speedup on poorly conditioned landscapes is dramatic in practice.

Adaptive Methods Convergence

Adaptive gradient methods — Adam, AdaGrad, RMSProp — adjust the learning rate per parameter based on historical gradients. They're wildly popular in practice. But their convergence theory is more complicated, and sometimes broken.

Adam Convergence (and Failure Cases)

Adam maintains running averages of the gradient (first moment) and squared gradient (second moment), using them to normalize updates. The original paper claimed O(√T) regret in the online convex optimization setting, which implies O(1/√T) convergence. But in 2018, Reddi et al. showed this proof was wrong. They constructed a simple convex problem where Adam provably diverges. The issue: Adam's exponential moving average of the squared gradient can "forget" large past gradients, causing the effective learning rate to spike at bad times.

AMSGrad Fix

AMSGrad fixes Adam's convergence failure by maintaining the running maximum of the second moment estimate rather than just the moving average: v̂_t = max(v̂_{t-1}, v_t). This ensures the effective learning rate never increases, which is the key property needed for the convergence proof. In practice, AMSGrad often performs similarly to Adam — the pathological cases are rare in real networks. But if you need theoretical guarantees, AMSGrad is the safe choice.

AdaGrad for Sparse Gradients

AdaGrad accumulates the sum of all past squared gradients per coordinate. For sparse gradient problems — where most coordinates are zero most of the time, like in NLP with large vocabularies — AdaGrad is excellent. Rarely-updated parameters get larger learning rates, frequently-updated parameters get smaller ones. Its convergence rate for online convex optimization is O(1/√T), and it adapts automatically to the geometry of the problem without tuning. The downside: the accumulated sum grows without bound, eventually shrinking the learning rate to zero. That's why RMSProp and Adam use exponential moving averages instead.

Coordinate Descent Convergence

Coordinate descent updates one variable at a time while holding others fixed. For separable or block-structured problems (think Lasso regression, SVMs), it can be extremely efficient. With random coordinate selection, it converges at rate O(d/(μt)) on strongly convex functions where d is dimension — each step is d times cheaper than a full gradient step, so the total work is comparable, but the method naturally exploits problem structure. 

# Comparing Adam vs. AMSGrad on a simple problem
import numpy as np

def adam_vs_amsgrad(T=3000, seed=42):
    """Compare Adam and AMSGrad on a constructed convex problem."""
    np.random.seed(seed)

    # Simple 1D problem where Adam can oscillate:
    # alternate between gradient +C and -1, chosen to exploit Adam's forgetting
    beta1, beta2, eps = 0.9, 0.99, 1e-8
    lr = 0.01

    # Adam
    x_adam, m_adam, v_adam = 0.0, 0.0, 0.0
    # AMSGrad
    x_ams, m_ams, v_ams, v_hat_ams = 0.0, 0.0, 0.0, 0.0

    trace_adam, trace_ams = [], []

    for t in range(1, T + 1):
        # Synthetic gradient: mostly small, occasionally large
        if t % 10 == 0:
            g = 10.0 * np.sign(x_adam + 0.5)  # large gradient
        else:
            g_adam = -1.0 * np.sign(x_adam + 0.5)
            g_ams = -1.0 * np.sign(x_ams + 0.5)

            # Adam update
            m_adam = beta1 * m_adam + (1 - beta1) * g_adam
            v_adam = beta2 * v_adam + (1 - beta2) * g_adam**2
            m_hat = m_adam / (1 - beta1**t)
            v_hat = v_adam / (1 - beta2**t)
            x_adam -= lr * m_hat / (np.sqrt(v_hat) + eps)

            # AMSGrad update
            m_ams = beta1 * m_ams + (1 - beta1) * g_ams
            v_ams = beta2 * v_ams + (1 - beta2) * g_ams**2
            v_hat_ams = max(v_hat_ams, v_ams)
            m_hat_a = m_ams / (1 - beta1**t)
            x_ams -= lr * m_hat_a / (np.sqrt(v_hat_ams) + eps)

            continue

        # Large gradient step for both
        g_adam = g
        g_ams = 10.0 * np.sign(x_ams + 0.5)

        m_adam = beta1 * m_adam + (1 - beta1) * g_adam
        v_adam = beta2 * v_adam + (1 - beta2) * g_adam**2
        m_hat = m_adam / (1 - beta1**t)
        v_hat = v_adam / (1 - beta2**t)
        x_adam -= lr * m_hat / (np.sqrt(v_hat) + eps)

        m_ams = beta1 * m_ams + (1 - beta1) * g_ams
        v_ams = beta2 * v_ams + (1 - beta2) * g_ams**2
        v_hat_ams = max(v_hat_ams, v_ams)
        m_hat_a = m_ams / (1 - beta1**t)
        x_ams -= lr * m_hat_a / (np.sqrt(v_hat_ams) + eps)

        trace_adam.append(x_adam)
        trace_ams.append(x_ams)

    print(f"Adam final x:    {x_adam:.4f}  (optimal: -0.5)")
    print(f"AMSGrad final x: {x_ams:.4f}  (optimal: -0.5)")

adam_vs_amsgrad()

Non-Convex Optimization

Everything above assumed convexity or strong convexity. Neural networks are neither. Their loss landscapes are wildly non-convex — full of saddle points, flat regions, and local minima. So what can we actually guarantee?

Convergence to Stationary Points

For general smooth non-convex functions, the best we can guarantee with gradient descent is convergence to an approximate stationary point — a point where ‖∇f(x)‖ ≤ ε. Gradient descent achieves this in O(1/ε²) steps. SGD needs O(1/ε⁴) steps. These are much weaker guarantees than finding a global minimum, but they're the best available without structural assumptions.

The Polyak-Łojasiewicz (PL) Condition

The Polyak-Łojasiewicz (PL) condition says that ‖∇f(x)‖² ≥ 2μ(f(x) − f(x*)) for all x. This is weaker than strong convexity — it doesn't require convexity at all. But it's strong enough to guarantee that gradient descent converges linearly to a global minimum. Remarkably, some overparameterized neural networks satisfy the PL condition in a neighborhood of the initialization, which partially explains why gradient descent finds good solutions despite non-convexity.

Escaping Saddle Points

Saddle points are a bigger threat than local minima in high dimensions. At a saddle point, the gradient is zero but the Hessian has negative eigenvalues — there are directions of descent the gradient doesn't reveal. Plain gradient descent can get stuck at saddle points for exponentially long times. Adding small noise (perturbed gradient descent) provably escapes strict saddle points in polynomial time. SGD naturally provides this noise, which is one reason it works well for neural networks — the stochastic noise is a feature, not a bug.

Landscape Analysis for Neural Networks

For certain neural network architectures, the loss landscape has favorable structure. Overparameterized networks (more parameters than training examples) often have the property that every local minimum is a global minimum — or at least achieves near-zero training loss. The Neural Tangent Kernel (NTK) theory shows that sufficiently wide networks behave like linear models in a lifted feature space, making the optimization landscape effectively convex near initialization. This is theoretically clean but requires impractically wide networks. For realistic architectures, the practical observation is that SGD with momentum almost always finds solutions with very low training loss, even though the worst-case theory gives no such guarantee.

ℹ️ Key Insight

The gap between optimization theory and deep learning practice is enormous. Theory says non-convex optimization is hard (NP-hard in general). Practice shows SGD works great on neural networks. The resolution seems to be that neural network landscapes have special structure — overparameterization, smooth basins, favorable saddle-point geometry — that makes them much easier than worst-case non-convex problems. The theory is catching up, but slowly.

Lower Bounds: The Fundamental Speed Limits

We've seen upper bounds — how fast specific algorithms converge. Lower bounds tell us the flip side: no algorithm in a given class can converge faster than a certain rate. They're the speed limits of optimization.

Nesterov's Lower Bound for First-Order Methods

Nesterov proved that for any first-order method (one that only accesses the function through gradient evaluations), there exists a smooth convex function where the method needs at least Ω(1/t²) to find an ε-optimal point. More precisely, after t gradient evaluations: f(x_t) − f(x*) ≥ Ω(L‖x₀ − x*‖² / t²). Since Nesterov's accelerated gradient matches this, the O(1/t²) upper bound is tight. For strongly convex functions, the lower bound is (1 − 2/(√κ + 1))^t, and Nesterov's method matches this too. The algorithm is optimal.

Information-Theoretic Lower Bounds

Information-theoretic lower bounds go beyond a specific algorithm class. They ask: given a specific "budget" of information about the function (gradient queries, function evaluations, Hessian-vector products), what is the fastest any algorithm can converge? For stochastic optimization, the O(1/√T) lower bound for convex functions means SGD's rate is optimal in a minimax sense. You cannot overcome gradient noise with clever algorithms alone — you need more samples or stronger assumptions.

Oracle Complexity

Oracle complexity formalizes this by modeling the optimizer as querying an "oracle" that returns gradients (or other information) at requested points. The complexity is the number of oracle calls needed to reach ε-accuracy. This framework enables clean comparisons: gradient descent uses O(κ log(1/ε)) oracle calls on strongly convex problems, Nesterov uses O(√κ log(1/ε)), and the lower bound matches Nesterov. Second-order methods (Newton's method) have different oracle complexity because each call returns richer information (the Hessian), so the comparison is within the first-order oracle model.

Summary: The Convergence Rate Landscape

Here's how all the rates compare. Every row is a setting, every column is a method class. The entries are the number of iterations to reach ε-accuracy.

Setting GD Nesterov SGD Lower Bound
Smooth Convex O(1/ε) O(1/√ε) O(1/ε²) Ω(1/√ε) [first-order]
Strongly Convex (κ) O(κ log(1/ε)) O(√κ log(1/ε)) O(1/(με)) Ω(√κ log(1/ε))
Non-Convex (stationary) O(1/ε²) O(1/ε²) O(1/ε⁴) Ω(1/ε²)
PL Condition O(κ log(1/ε)) — O(κ/ε) —

The pattern is clear: stronger assumptions (strong convexity, PL condition) buy you faster rates. Stochasticity always costs you. And acceleration closes the gap between what gradient descent achieves and what is theoretically possible. When you pick an optimizer for your next project, you're navigating this table — trading off assumptions, per-step cost, and convergence speed.

⚠️ Warning

These rates are worst-case guarantees. Your specific problem might converge faster or slower. A method with a worse theoretical rate (like Adam) often outperforms a theoretically optimal method (like Nesterov) on specific deep learning tasks because the theory's worst-case assumptions don't match your actual landscape. Use the theory as a guide, not a gospel. Always measure empirically.

Consider whether your problem is ill-conditioned and needs momentum. Ask whether you're above the critical batch size. The theory isn't abstract — it's the toolkit for diagnosing and fixing slow training.
