Bayesian Optimization

Chapter 14: Probabilistic & Bayesian ML 6 subtopics
TL;DR

Bayesian Optimization (BO) is how you tune expensive black-box functions — like hyperparameters of a neural network that takes hours to train — without exhaustive search. It builds a cheap probabilistic model (surrogate) of the expensive function, uses that model to decide which point to evaluate next (acquisition function), evaluates it, updates the model, and repeats. This section walks through the full framework, the major acquisition functions, surrogate model choices, scaling to high dimensions, and the practical tools you will actually use.

Framework

You have trained a deep learning model. It has a learning rate, batch size, weight decay, number of layers, dropout rate — maybe 10 hyperparameters. Each combination takes 4 hours to train and evaluate. Grid search over even a coarse grid of 5 values per parameter means 5^10 ≈ 10 million combinations. At 4 hours each, that is 4,500 years. You need a smarter strategy.

Bayesian Optimization is that smarter strategy. The core idea: after evaluating a few hyperparameter combinations, you have enough information to predict which regions of hyperparameter space are promising and which are dead ends. You use those predictions to choose the next point intelligently instead of blindly.

Surrogate Model (GP, Random Forest, TPE)

The surrogate model is a cheap-to-evaluate probabilistic model that approximates the expensive objective function. It takes a hyperparameter configuration as input and outputs a predicted performance along with an uncertainty estimate. The most common choice is a Gaussian Process (GP), which gives you a mean prediction and a variance at every point in the search space.

Other surrogates work too. Random forests (used in SMAC) predict performance by averaging trees and estimate uncertainty from the variance across trees. Tree-structured Parzen Estimators (TPE, used in Hyperopt) model p(hyperparameters | good results) and p(hyperparameters | bad results) separately, then pick configurations that maximize their ratio.

Acquisition Function

The acquisition function uses the surrogate's predictions and uncertainties to score candidate points. It answers: "Given what I know so far, which point should I evaluate next?" The acquisition function encodes the exploration-exploitation tradeoff — should you try a point where the predicted performance is high (exploit) or a point where the uncertainty is high (explore)?

You maximize the acquisition function — a cheap optimization problem because the surrogate is fast to evaluate — to find the next candidate. Then you evaluate that candidate on the real expensive function, add the result to your dataset, update the surrogate, and repeat.

Sequential Optimization Loop

# The Bayesian Optimization loop:
#
# 1. Initialize: evaluate f(x) at a few random points
# 2. Fit surrogate model to observed (x, f(x)) pairs
# 3. Find x_next = argmax acquisition_function(x)
# 4. Evaluate f(x_next) — the expensive step
# 5. Add (x_next, f(x_next)) to observations
# 6. Go to step 2
#
# Typical budget: 20-200 evaluations total
# Compare to grid search needing thousands-to-millions

import numpy as np

def bayesian_optimization_loop(objective_fn, bounds, n_init=5, n_iter=25):
    """Minimal BO loop — conceptual illustration."""
    # Step 1: Random initialization
    X_observed = np.random.uniform(bounds[:, 0], bounds[:, 1],
                                    size=(n_init, bounds.shape[0]))
    y_observed = np.array([objective_fn(x) for x in X_observed])

    for i in range(n_iter):
        # Step 2: Fit surrogate (GP in this case)
        gp = fit_gaussian_process(X_observed, y_observed)

        # Step 3: Optimize acquisition function
        x_next = maximize_acquisition(gp, bounds, y_best=y_observed.min())

        # Step 4: Evaluate expensive function
        y_next = objective_fn(x_next)

        # Step 5: Update dataset
        X_observed = np.vstack([X_observed, x_next])
        y_observed = np.append(y_observed, y_next)

        print(f"Iter {i+1}: f(x) = {y_next:.4f}, best so far = {y_observed.min():.4f}")

    best_idx = y_observed.argmin()
    return X_observed[best_idx], y_observed[best_idx]

Exploration-Exploitation Tradeoff in BO

This tradeoff is the heart of BO. Pure exploitation (always pick the point with the best predicted value) gets stuck in local optima — you keep searching near your current best and miss better regions. Pure exploration (always pick the most uncertain point) wastes your evaluation budget on points far from anything useful. Good acquisition functions balance both.

The tradeoff shifts over the course of optimization. Early on, you want more exploration to map the landscape. Later, when the budget is running out, you want to exploit the best region you have found. Some acquisition functions have explicit knobs to control this (UCB's β parameter). Others handle it implicitly (Expected Improvement naturally explores when the surrogate is uncertain).

Acquisition Functions

The acquisition function is where the art of BO lives. Each function makes different assumptions about what "most informative next evaluation" means. Let us walk through the major options.

Expected Improvement (EI)

Expected Improvement is the most popular acquisition function and the default in most BO libraries. It computes the expected amount by which a candidate point will improve over the current best observed value. "Expected" means averaged over the surrogate's uncertainty — if the surrogate thinks a point might be great or might be terrible, EI accounts for both possibilities.

import numpy as np
from scipy.stats import norm

def expected_improvement(X_candidates, gp_model, y_best, xi=0.01):
    """
    Expected Improvement acquisition function.

    X_candidates: points to evaluate the acquisition at
    gp_model: fitted GP surrogate
    y_best: best observed value so far (for minimization)
    xi: exploration-exploitation parameter (small = exploit, large = explore)
    """
    mu, sigma = gp_model.predict(X_candidates, return_std=True)
    sigma = np.maximum(sigma, 1e-9)  # Avoid division by zero

    # Improvement over current best
    improvement = y_best - mu - xi    # For minimization
    Z = improvement / sigma

    ei = improvement * norm.cdf(Z) + sigma * norm.pdf(Z)
    ei[sigma < 1e-9] = 0.0  # No improvement where we already know the value

    return ei

# EI is 0 at observed points (no uncertainty → no expected improvement)
# EI is high where: (a) predicted value is good, OR (b) uncertainty is high
# This automatically balances exploration and exploitation

The parameter ξ (xi) tilts the balance. With ξ = 0, EI is purely greedy. With larger ξ, EI values exploration more. In practice, ξ = 0.01 works well for most problems.

Upper Confidence Bound (UCB / GP-UCB)

Upper Confidence Bound takes the direct approach: score each point as the predicted mean minus κ times the predicted standard deviation (for minimization). The parameter κ explicitly controls the exploration-exploitation tradeoff.

def upper_confidence_bound(X_candidates, gp_model, kappa=2.0):
    """
    UCB acquisition function (for minimization: use Lower Confidence Bound).
    x_next = argmin mu(x) - kappa * sigma(x)
    """
    mu, sigma = gp_model.predict(X_candidates, return_std=True)
    return mu - kappa * sigma   # minimize this → pick low predicted + uncertain

# kappa = 0: pure exploitation (just minimize predicted value)
# kappa → ∞: pure exploration (just maximize uncertainty)
# kappa = 2.0: typical default — theoretically motivated by GP-UCB bounds

The GP-UCB paper by Srinivas et al. (2010) proved regret bounds: with κ growing slowly as log(t), the algorithm converges to the global optimum. This theoretical grounding is why UCB is popular in the Bayesian optimization literature despite EI being more common in practice.

Probability of Improvement (PI)

Probability of Improvement is the oldest acquisition function. It asks: "What is the probability that this point will beat the current best?" It is the CDF of the improvement distribution — simple to compute, easy to understand, but it has a flaw: it does not account for how much improvement is expected. A point with 51% chance of a tiny improvement beats a point with 49% chance of a massive improvement. This makes PI too greedy in practice. EI fixes this by weighting probability by magnitude.

Knowledge Gradient

The Knowledge Gradient asks a more sophisticated question: "After evaluating this point and updating my surrogate, how much better will my best predicted point be?" It looks one step ahead, accounting for the information gain from the new observation. This makes it myopically optimal — the best single-step acquisition function — but it is more expensive to compute because you need to simulate the posterior update for each candidate.

Thompson Sampling for BO

Thompson Sampling takes yet another approach: draw a random function from the surrogate's posterior, then find the minimum of that function. Each posterior sample is a complete function f(x), so minimizing it gives you a single candidate point. Because the posterior concentrates near observed data, sampled functions tend to be "interesting" — they explore uncertain regions while respecting what you have already learned.

Thompson Sampling is naturally parallelizable. Want to evaluate 10 points in parallel? Draw 10 posterior samples, minimize each, done. No coordination needed between the candidates. This makes it popular in batch BO settings.

Entropy Search and Predictive Entropy Search

Entropy Search (Hennig & Schuler, 2012) reframes the acquisition problem as information gain: "Which evaluation will most reduce my uncertainty about where the global optimum is?" It maximizes the expected reduction in entropy of the distribution over the optimum's location. Predictive Entropy Search (Hernández-Lobato et al., 2014) is a computationally cheaper approximation that reverses the conditioning: instead of computing the entropy of the optimum given a new observation, it computes the mutual information between the new observation and the optimum.

These methods are theoretically elegant but computationally expensive. They shine when evaluation budget is extremely tight (fewer than 20 evaluations) and each evaluation is very expensive (hours or days).

Surrogate Models

Gaussian Process Surrogates

GPs are the default surrogate for BO. They provide exact uncertainty quantification, handle small datasets gracefully, and have a well-understood kernel design language. The kernel choice matters: a Matérn 5/2 kernel is the standard default because it assumes the objective function is twice differentiable but not infinitely smooth — a good match for most hyperparameter response surfaces.

The limitation of GP surrogates is scalability. Fitting a GP costs O(n³) in the number of observations and O(n²) in memory. This is fine for typical BO budgets (20-200 evaluations) but becomes a bottleneck for larger-scale problems.

Tree-structured Parzen Estimator (TPE)

The TPE (used in Hyperopt and Optuna) models the search differently from GP-based methods. Instead of modeling f(x) directly, it models two densities: l(x) = p(x | f(x) < y*) (the distribution of good configurations) and g(x) = p(x | f(x) ≥ y*) (bad configurations), where y* is a quantile threshold. The acquisition score is l(x)/g(x) — pick configurations that look like winners, not losers.

TPE handles categorical and conditional parameters natively because it models distributions over the search space, not over the objective. This is a huge practical advantage: neural architecture hyperparameters like "optimizer type" (Adam, SGD, RMSProp) or "use batch norm? (yes/no → if yes, what momentum?)" are natural for TPE but awkward for GPs.

Random Forest Surrogates (SMAC)

SMAC (Sequential Model-based Algorithm Configuration) uses random forests as surrogates. The mean prediction is the forest's average, and uncertainty is estimated from the variance across individual trees. Random forests handle mixed continuous/categorical spaces and scale better than GPs. The tradeoff: uncertainty estimates are less principled than GP posteriors.

Bayesian Neural Network Surrogates

For high-dimensional problems where GPs struggle, you can use a small BNN as the surrogate. The BNN provides uncertainty through the methods we covered in the previous section (MC Dropout, ensembles, or variational weights). This scales better than GPs but is harder to tune and less sample-efficient — the BNN itself needs enough data to produce good uncertainty estimates.

Scalability and Extensions

Vanilla BO works brilliantly for 5-20 hyperparameters. But real-world problems push beyond that envelope. Here is how the field has responded.

High-Dimensional BO (Random Embeddings, Trust Regions)

GPs degrade in high dimensions because the search space grows exponentially while your observation count stays fixed. Random embedding methods (REMBO) project the high-dimensional space into a random low-dimensional subspace and optimize there, betting that the objective has low effective dimensionality. Trust region methods (TuRBO) maintain local GP models within restricted regions, moving and resizing the region based on success or failure.

TuRBO (Eriksson et al., 2019) is particularly practical: it runs multiple independent BO loops, each in its own trust region, and allocates more evaluations to regions that are performing well. It has set state-of-the-art results on high-dimensional benchmarks up to 200 dimensions.

Multi-Fidelity BO (Fabolas, Freeze-Thaw)

Multi-fidelity BO exploits cheap approximations of the expensive objective. Training a model for 5 epochs is much cheaper than training for 100 epochs, and the 5-epoch result is correlated with the 100-epoch result. Methods like Fabolas treat dataset size as an extra input to the surrogate — evaluate cheap low-data configurations first to narrow the search, then spend budget on full evaluations of the most promising candidates.

Freeze-Thaw BO starts training many configurations for a few epochs, models their learning curves with the surrogate, and decides which to continue training and which to abandon. This is far more efficient than running each configuration to completion before starting the next.

Multi-Objective BO (Pareto Front)

Often you care about multiple objectives: accuracy and latency, or F1-score and model size. Multi-objective BO maintains a surrogate for each objective and uses acquisition functions that target the Pareto front — the set of configurations where no objective can be improved without worsening another. Expected Hypervolume Improvement (EHVI) is the standard acquisition function: it measures the expected increase in the volume dominated by the Pareto front.

BOHB (Bayesian Optimization + HyperBand)

BOHB (Falkner et al., 2018) combines the best of two worlds. HyperBand is a bandit-based method that efficiently allocates budget by early-stopping poor configurations. BOHB replaces HyperBand's random sampling with TPE-guided sampling, so the configurations that enter each HyperBand round are intelligently chosen. This consistently outperforms both vanilla BO and vanilla HyperBand across a wide range of problems.

# BOHB conceptually:
#
# HyperBand alone:
#   1. Sample many random configs
#   2. Train each for a small budget
#   3. Keep top fraction, increase budget, repeat (successive halving)
#
# BOHB:
#   1. Sample configs from a TPE model (not random!)
#   2. Train each for a small budget
#   3. Keep top fraction, increase budget, repeat
#   4. Update the TPE model with results
#
# Result: early rounds inform the TPE, which proposes better configs
#         for later rounds. Converges faster than either method alone.

Categorical and Conditional Hyperparameters

Real search spaces have structure. The "optimizer" hyperparameter is categorical (Adam, SGD, AdaGrad). The "momentum" hyperparameter only matters when optimizer = SGD. These conditional dependencies make GP-based BO awkward because GPs operate over continuous spaces. TPE and SMAC handle this natively. For GP-based methods, you can use one-hot encoding for categoricals and set conditional parameters to a fixed value when their parent condition is not met.

Noisy Observations Handling

Stochastic training means the same hyperparameter configuration gives different results each run. BO handles this naturally: the GP's observation noise parameter (the nugget) absorbs this variability. You can also evaluate the same point multiple times and the GP will learn the noise level from the repeated observations. EI and UCB both account for observation noise in their calculations.

Transfer Learning and Warm-Starting in BO

If you have optimized hyperparameters for a similar task before, you can warm-start the surrogate with those results. The surrogate learns "learning rate around 1e-3 is usually good for image classification" from previous experiments and starts the new search from a better position. Methods include multi-task GPs (model the old and new task jointly) and initializing the GP with scaled observations from the previous task.

BO for Neural Architecture Search (NAS)

Architecture search is hyperparameter optimization over a structured, high-dimensional, largely categorical space. BO has been applied successfully with custom kernels that measure similarity between architectures (e.g., based on edit distance of architecture graphs). BANANAS (White et al., 2021) uses a neural network to encode architectures into a continuous space, then runs GP-based BO in that space.

Tools and Frameworks

You almost never implement BO from scratch. Here are the tools ranked by practical use.

Optuna

Optuna is the most popular hyperparameter optimization framework in the Python ecosystem. It uses TPE by default, supports pruning (early stopping of bad trials), and has a clean define-by-run API where you specify the search space inside the objective function.

import optuna

def objective(trial):
    # Define search space inline
    lr = trial.suggest_float("lr", 1e-5, 1e-1, log=True)
    n_layers = trial.suggest_int("n_layers", 1, 5)
    dropout = trial.suggest_float("dropout", 0.0, 0.5)
    optimizer_name = trial.suggest_categorical("optimizer", ["Adam", "SGD"])

    model = build_model(n_layers=n_layers, dropout=dropout)
    optimizer = getattr(torch.optim, optimizer_name)(model.parameters(), lr=lr)

    # Train and evaluate
    for epoch in range(100):
        train_loss = train_one_epoch(model, optimizer, train_loader)
        val_acc = evaluate(model, val_loader)

        # Report intermediate value for pruning
        trial.report(val_acc, epoch)
        if trial.should_prune():
            raise optuna.TrialPruned()

    return val_acc

study = optuna.create_study(direction="maximize",
                             pruner=optuna.pruners.MedianPruner())
study.optimize(objective, n_trials=100, n_jobs=4)  # 4 parallel workers

print(f"Best trial: {study.best_trial.params}")
print(f"Best accuracy: {study.best_value:.4f}")

Hyperopt

Hyperopt was one of the first TPE-based frameworks. Its API is less convenient than Optuna's, but it is still widely used, especially in older codebases. It supports TPE and random search, with MongoDB-backed distributed optimization for cluster environments.

BoTorch (PyTorch)

BoTorch is Meta's research-grade BO library built on GPyTorch. It gives you full control: custom GP models, any acquisition function, batch optimization, multi-objective BO, and multi-fidelity methods. It is what you use when Optuna's defaults are not enough and you need to implement a custom acquisition function or surrogate.

import torch
from botorch.models import SingleTaskGP
from botorch.fit import fit_gpytorch_mll
from botorch.acquisition import ExpectedImprovement
from botorch.optim import optimize_acqf
from gpytorch.mlls import ExactMarginalLogLikelihood

# Observed data (normalized to [0, 1] domain)
train_X = torch.tensor([[0.1, 0.3], [0.5, 0.7], [0.9, 0.2]], dtype=torch.float64)
train_Y = torch.tensor([[0.8], [0.3], [0.6]], dtype=torch.float64)

# Fit GP surrogate
gp = SingleTaskGP(train_X, train_Y)
mll = ExactMarginalLogLikelihood(gp.likelihood, gp)
fit_gpytorch_mll(mll)

# Expected Improvement acquisition
best_f = train_Y.min()
EI = ExpectedImprovement(model=gp, best_f=best_f)

# Find next point to evaluate
bounds = torch.tensor([[0.0, 0.0], [1.0, 1.0]], dtype=torch.float64)
candidate, acq_value = optimize_acqf(EI, bounds=bounds, q=1,
                                       num_restarts=5, raw_samples=20)
print(f"Next point to evaluate: {candidate}")

Ax (Adaptive Experimentation Platform)

Ax is Meta's higher-level platform built on BoTorch. It handles the full experiment management lifecycle: defining search spaces, running trials, tracking results, and choosing the next trial. It includes multi-objective optimization, bandit optimization for A/B testing, and integration with external job schedulers.

SMAC

SMAC (Sequential Model-based Algorithm Configuration) uses random forests as surrogates and handles mixed categorical/continuous spaces exceptionally well. It is the go-to tool when your search space has many conditional parameters — common when configuring entire ML pipelines (which preprocessor, which model, which model-specific hyperparameters).

Spearmint

Spearmint was the original GP-based BO tool from Snoek et al. (2012) that helped popularize BO for hyperparameter tuning. It uses a GP with Matérn kernel and EI acquisition. Historically important but largely superseded by BoTorch and Optuna for new projects.

When to Use BO vs Population-Based Methods

Population-based methods (evolutionary strategies, PBT) maintain a population of configurations and evolve them. They are embarrassingly parallel and handle complex search spaces but need large evaluation budgets (hundreds to thousands). BO is the opposite: it is sequential by nature but extremely sample-efficient. Use BO when each evaluation is expensive (hours/days). Use population-based methods when evaluations are cheap (minutes) and you have a cluster.

Limitations of Bayesian Optimization

⚠️ Limitations

BO struggles with more than ~20 continuous dimensions (the GP becomes too uncertain). It is inherently sequential — the next point depends on all previous results — limiting parallelism. The surrogate can mislead optimization if the objective has sharp discontinuities or if the kernel is misspecified. And the overhead of fitting the GP and optimizing the acquisition function is wasted if f(x) is cheap to evaluate — you should use random search or grid search instead.
