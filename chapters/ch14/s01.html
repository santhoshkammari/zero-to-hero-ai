Section Overview

This section combines Bayesian Inference and Bayesian Linear Regression into one comprehensive reference.

Bayesian Inference

TL;DR

Bayesian inference gives you a distribution over possible answers instead of a single best guess. Uncertainty is a feature, not a bug. Bayes’ theorem is the engine: prior belief × data likelihood → updated posterior belief. Everything else in this chapter — conjugate priors, MAP estimation, credible intervals — is tooling to make that engine practical.

Classical (frequentist) statistics asks: “What single parameter value makes the observed data most likely?” Bayesian statistics asks a richer question: “Given the data I’ve seen, what is the full distribution of plausible parameter values?” That distinction matters. The frequentist gives you a point estimate. The Bayesian gives you a distribution — which means you get calibrated uncertainty for free.

If you’ve trained a neural network, you’ve done frequentist ML: find the weights that minimize loss. If you’ve ever wanted your model to say “I’m not sure about this prediction,” you want Bayesian ML.

Bayes’ Theorem

The formula is short. The implications are enormous:

P(θ|D) = P(D|θ) × P(θ) / P(D)

posterior  = likelihood × prior / evidence

Each piece has a specific job:

  • Prior P(θ) — what you believed about parameters before seeing any data. A Gaussian centered at zero says “I expect small values.” A uniform says “I have no idea.”
  • Likelihood P(D|θ) — how well each parameter value explains the observed data. This is the same likelihood you maximize in MLE.
  • Evidence P(D) — the normalizing constant. It integrates the numerator over all possible θ values. Often intractable — which is why Section 4 exists.
  • Posterior P(θ|D) — your updated belief. This is the answer.

The key insight: the posterior is proportional to likelihood × prior. You don’t always need the evidence. For many tasks, knowing the shape of the posterior is enough.

import numpy as np

# Disease testing: the classic "base rate neglect" example
# A disease affects 1 in 10,000 people. Test is 99% accurate.
p_disease = 0.0001
p_pos_given_disease = 0.99     # sensitivity
p_pos_given_healthy = 0.01     # false positive rate

# Evidence: P(positive test)
p_pos = p_pos_given_disease * p_disease + p_pos_given_healthy * (1 - p_disease)

# Posterior: P(disease | positive test)
p_disease_given_pos = (p_pos_given_disease * p_disease) / p_pos
print(f"P(disease | positive) = {p_disease_given_pos:.4f}")
# Output: ~0.0098 — less than 1%!
# The test is accurate, but the disease is so rare that most positives are false.

This is Bayes’ theorem doing real work. The prior (disease is rare) overwhelms the likelihood (test is positive). If you ignore the prior, you panic. If you use it, you order a second test.

Prior Distributions — Encoding Beliefs

The prior is the part that makes frequentists uncomfortable and makes Bayesians powerful. You’re explicitly stating what you believe before seeing data. That honesty has consequences: with enough data, the prior washes out. With little data, it matters enormously.

Practical rule: If you genuinely don’t know, use a weakly informative prior. It constrains without dominating. Andrew Gelman’s recommendation: “Use priors that rule out absurd values but don’t try to be too clever.”

Informative priors encode real domain knowledge. If you know a drug effect should be between 0 and 5, say so. Weakly informative priors (e.g., Normal(0, 10) for regression coefficients) gently regularize without committing to specific values. Non-informative priors (flat, Jeffreys’) try to “let the data speak,” but truly non-informative priors often don’t exist or have pathological behavior.

Jeffreys’ prior is invariant under reparameterization — if you switch from measuring in meters to feet, the prior adjusts. For a Bernoulli parameter, Jeffreys’ prior is Beta(0.5, 0.5), which puts slightly more mass near 0 and 1 than a uniform Beta(1, 1).

Conjugate Priors

A prior is conjugate to a likelihood if the posterior has the same functional form as the prior. Why care? Because it gives you closed-form posterior updates — no sampling, no optimization, just algebra.

LikelihoodConjugate PriorPosteriorUse Case

Normal (known variance)Normal(μ0, σ0²)Normal(weighted mean, smaller variance)Sensor fusion, estimation
PoissonGamma(α, β)Gamma(α + ∑xi, β + n)Event counts, rate estimation

The Beta-Binomial is the workhorse. Let’s see it in action:

import numpy as np
from scipy import stats
import matplotlib.pyplot as plt

# Coin flipping: we observe 7 heads in 10 flips
# Prior: Beta(2, 2) — mildly believing the coin is fair
alpha_prior, beta_prior = 2, 2
heads, tails = 7, 3

# Posterior: Beta(alpha + heads, beta + tails)
alpha_post = alpha_prior + heads    # 9
beta_post = beta_prior + tails      # 5

theta = np.linspace(0, 1, 200)
fig, ax = plt.subplots(figsize=(8, 4))
ax.plot(theta, stats.beta.pdf(theta, alpha_prior, beta_prior),
        label=f'Prior Beta({alpha_prior},{beta_prior})', ls='--')
ax.plot(theta, stats.beta.pdf(theta, alpha_post, beta_post),
        label=f'Posterior Beta({alpha_post},{beta_post})', lw=2)
ax.axvline(heads / (heads + tails), color='red', ls=':', label='MLE = 0.7')
ax.set_xlabel('θ (probability of heads)')
ax.legend()
ax.set_title('Prior → Posterior with 10 coin flips')
plt.tight_layout()
plt.show()
# The posterior sits between the prior (0.5) and the MLE (0.7)
# With more data, the posterior will converge to the MLE
Key insight: The posterior mean for a Beta(α, β) is α/(α+β). As data accumulates, the prior’s α and β become negligible relative to the counts. More data → prior matters less. This is why Bayesian and frequentist results converge with enough data.

Sequential Updating

One of the most elegant properties of Bayesian inference: today’s posterior becomes tomorrow’s prior. You don’t need to reprocess all historical data when new data arrives — just update.

# Sequential Bayesian updating: process data in batches
alpha, beta = 2, 2  # start with mild prior

batches = [
    (3, 7),    # batch 1: 3 heads, 7 tails
    (8, 2),    # batch 2: 8 heads, 2 tails
    (5, 5),    # batch 3: 5 heads, 5 tails
]

print(f"Start: Beta({alpha}, {beta}), mean = {alpha/(alpha+beta):.3f}")
for i, (h, t) in enumerate(batches, 1):
    alpha += h
    beta += t
    mean = alpha / (alpha + beta)
    print(f"After batch {i}: Beta({alpha}, {beta}), "
          f"mean = {mean:.3f}, 95% CI = "
          f"[{stats.beta.ppf(0.025, alpha, beta):.3f}, "
          f"{stats.beta.ppf(0.975, alpha, beta):.3f}]")
# Same result whether you process all 30 flips at once or in 3 batches

This is why Bayesian methods are natural for online learning and A/B testing. You can check results at any time without worrying about multiple-testing corrections. The posterior is always valid.

MAP vs. Full Bayesian Inference

Two ways to use the posterior:

MAP (Maximum A Posteriori) takes the mode of the posterior. It finds the single most probable parameter value, accounting for the prior. MAP is a point estimate — you lose uncertainty information but gain simplicity.

The key connection: MAP with a Gaussian prior is exactly L2-regularized MLE (Ridge regression). MAP with a Laplace prior is exactly L1-regularized MLE (Lasso). Every time you add a regularization penalty, you’re secretly being Bayesian.

# MAP is just regularized MLE
# log P(theta|D) = log P(D|theta) + log P(theta) - log P(D)
#                = log-likelihood   + log-prior   - constant
#
# Maximizing log P(theta|D) is equivalent to:
# argmax_theta [log-likelihood + log-prior]
#
# If prior is N(0, sigma^2):
#   log P(theta) = -theta^2 / (2*sigma^2) + const
# This is exactly L2 regularization with lambda = 1/sigma^2

Full Bayesian inference integrates over the entire posterior. Instead of predicting with a single θ, you average predictions across all plausible θ values, weighted by their posterior probability. This gives you:

  • Predictions that account for parameter uncertainty
  • Predictive distributions (not just point predictions)
  • Natural protection against overfitting

The downside? That integral is usually intractable. That’s why we need approximate inference (Section 4).

Credible Intervals vs. Confidence Intervals

This is the thing everyone confuses. They sound similar, but they answer fundamentally different questions.

A 95% credible interval says: “Given the data, there is a 95% probability that θ lies in this interval.” That’s the intuitive thing you want to say.

A 95% confidence interval says: “If I repeated this experiment infinitely many times, 95% of the intervals constructed this way would contain the true θ.” It says nothing about this particular interval. The true θ is either in it or it isn’t — there’s no probability involved (because frequentists treat θ as fixed).

In practice: For large samples, credible intervals and confidence intervals are nearly identical. The philosophical difference matters most with small data or strong priors. If a colleague says “there’s a 95% chance the true value is in this range,” they’re thinking Bayesian — whether they know it or not.

When Does Bayesian Actually Matter in Production?

Bayesian methods aren’t free. They’re computationally heavier, harder to scale, and require more thought about model specification. Use them when the benefits justify the cost:

  • Small data — you have 50 samples, not 50 million. Priors provide regularization that data alone can’t.
  • Uncertainty is critical — medical decisions, autonomous driving, financial risk. “I predict 0.7” is less useful than “I predict 0.7 ± 0.3.”
  • Sequential decision-making — A/B testing where you want to stop early. Bandits. Active learning. The posterior updates naturally.
  • Hierarchical data — data from many groups (hospitals, users, regions) where you want to share strength across groups.
  • Model comparison — the marginal likelihood provides a principled way to compare models of different complexity.

When to not bother: large datasets where a point estimate is good enough, latency-critical serving where you can’t afford MCMC, problems where uncertainty doesn’t change the decision.

import numpy as np
from scipy import stats

# Practical example: A/B test with Bayesian updating
# Control: 120 conversions out of 1000 visitors
# Treatment: 145 conversions out of 1000 visitors

# Priors: Beta(1, 1) = Uniform — no prior knowledge
alpha_A, beta_A = 1 + 120, 1 + 880   # Beta(121, 881)
alpha_B, beta_B = 1 + 145, 1 + 855   # Beta(146, 856)

# Sample from posteriors to estimate P(B > A)
samples_A = np.random.beta(alpha_A, beta_A, 100_000)
samples_B = np.random.beta(alpha_B, beta_B, 100_000)

p_b_better = np.mean(samples_B > samples_A)
lift = np.mean((samples_B - samples_A) / samples_A)

print(f"P(B > A) = {p_b_better:.3f}")
print(f"Expected lift = {lift:.1%}")
print(f"95% credible interval for lift: "
      f"[{np.percentile((samples_B-samples_A)/samples_A, 2.5):.1%}, "
      f"{np.percentile((samples_B-samples_A)/samples_A, 97.5):.1%}]")
# No p-values, no multiple testing corrections, just direct probability statements

The Marginal Likelihood — Model Selection’s Secret Weapon

The evidence P(D) that we’ve been treating as “just a normalizer” is actually powerful for model comparison. It’s the probability of the data under the entire model, averaging over all possible parameter values:

P(D|M) = ∫ P(D|θ, M) P(θ|M) dθ

This automatically implements Occam’s razor. A complex model spreads its prior probability over many possible parameter settings. If the data can be explained by a simpler model, the simpler model’s marginal likelihood will be higher because its prior mass is concentrated on the parameter values that matter.

The Bayes factor compares two models: BF = P(D|M1) / P(D|M2). A Bayes factor of 10 means the data is 10 times more likely under model 1. No arbitrary significance thresholds needed.

from scipy.special import betaln

# Model comparison: is this coin fair or biased?
# Data: 7 heads, 3 tails

# M1: fair coin, theta = 0.5 (no free parameters)
log_lik_fair = 7 * np.log(0.5) + 3 * np.log(0.5)

# M2: biased coin with Beta(1,1) prior (uniform on theta)
# Marginal likelihood = Beta(1+7, 1+3) / Beta(1, 1)
log_ml_biased = betaln(1 + 7, 1 + 3) - betaln(1, 1)

bayes_factor = np.exp(log_lik_fair - log_ml_biased)
print(f"Bayes factor (fair vs biased): {bayes_factor:.2f}")
# BF around 0.7 — slight evidence for biased, but not conclusive
# With only 10 flips, we can't distinguish. That's the RIGHT answer.

Bayesian Linear Regression

TL;DR

Bayesian linear regression puts distributions on weights instead of point estimates. The payoff: predictions come with uncertainty bands that widen where you have less data. Ridge regression is secretly MAP Bayesian regression with a Gaussian prior. When you need the full picture — calibrated uncertainty, not just a number — go Bayesian.

In ordinary least squares, you find a single set of weights w that minimizes squared error. The answer is crisp: w = (XTX)-1XTy. But that crispness is a lie. Those weights have uncertainty — OLS just doesn’t tell you about it.

Bayesian linear regression makes that uncertainty explicit. Every weight has a distribution, not just a value. And when you make predictions, those weight distributions propagate into predictive uncertainty that varies across the input space. Predictions near your training data are confident; predictions far away are uncertain. That’s exactly what you want.

The Setup

The model is the same linear regression you know: y = Xw + ε, where ε ~ N(0, σ²). The Bayesian twist: put a prior on w.

With a Gaussian prior w ~ N(0, σw²I) and Gaussian likelihood, the posterior is also Gaussian (conjugacy!). The closed-form posterior is:

Posterior mean:       m = (X^T X + (σ²/σ_w²) I)^{-1} X^T y
Posterior covariance: S = σ² (X^T X + (σ²/σ_w²) I)^{-1}

Look at that posterior mean. It’s exactly the Ridge regression solution with λ = σ²/σw². Ridge regression has been Bayesian all along — it just didn’t know it.

The connection: Ridge = MAP with Gaussian prior. Lasso = MAP with Laplace prior. Elastic Net = MAP with a mixture prior. Every regularization penalty corresponds to a prior distribution. The Bayesian framework tells you why regularization works: you’re encoding the belief that extreme parameter values are unlikely.

The Posterior Predictive Distribution

This is the killer feature. For a new input x*, the predictive distribution isn’t a single number. It’s a Gaussian:

p(y* | x*, D) = N(x*^T m,  x*^T S x* + σ²)
                            ├───────┘   └─── noise uncertainty
                            └─── parameter uncertainty

Two sources of uncertainty combine: epistemic uncertainty (we’re not sure about the weights) and aleatoric uncertainty (the data is inherently noisy). The epistemic part shrinks with more training data. The aleatoric part is irreducible.

The practical consequence: uncertainty bands widen as you move away from training data. A model that says “I’m confident here but uncertain there” is far more useful than one that’s equally confident everywhere.

BLR in Practice

import numpy as np
import matplotlib.pyplot as plt
from sklearn.linear_model import BayesianRidge

# Generate data with a gap to show uncertainty behavior
np.random.seed(42)
X_train = np.sort(np.concatenate([
    np.random.uniform(0, 3, 20),
    np.random.uniform(7, 10, 20)
]))
y_train = np.sin(X_train) + 0.3 * np.random.randn(len(X_train))

# Fit Bayesian Ridge (uses conjugate priors internally)
X_feat = np.column_stack([X_train**i for i in range(6)])  # polynomial features
model = BayesianRidge(compute_score=True)
model.fit(X_feat, y_train)

# Predict with uncertainty
X_test = np.linspace(-1, 11, 300)
X_test_feat = np.column_stack([X_test**i for i in range(6)])
y_pred, y_std = model.predict(X_test_feat, return_std=True)

# Plot
fig, ax = plt.subplots(figsize=(10, 5))
ax.scatter(X_train, y_train, c='black', s=20, label='Training data')
ax.plot(X_test, y_pred, 'b-', label='Predictive mean')
ax.fill_between(X_test, y_pred - 2*y_std, y_pred + 2*y_std,
                alpha=0.2, color='blue', label='95% predictive interval')
ax.set_xlabel('x')
ax.set_ylabel('y')
ax.legend()
ax.set_title('Bayesian Ridge: uncertainty widens in the gap (3-7) and at edges')
plt.tight_layout()
plt.show()

Notice the uncertainty bands in the gap between x=3 and x=7: they balloon out. The model knows it hasn’t seen data there. An OLS model would blissfully extrapolate a polynomial through the gap with no warning. The Bayesian model screams “I’m guessing here!”

When to Use BLR

BLR shines when:

  • You have small data (dozens to hundreds of samples) and need reliable predictions
  • You need calibrated uncertainty — know where the model is confident and where it’s not
  • Features >> samples (the prior prevents overfitting naturally)
  • You want automatic relevance determination (ARD) — BLR variants can learn per-feature regularization, effectively doing feature selection

Skip BLR when: you have millions of samples (the prior is irrelevant), you don’t need uncertainty, or the relationship is highly nonlinear (use a GP or neural network instead).

from sklearn.linear_model import BayesianRidge, Ridge
from sklearn.model_selection import cross_val_score
from sklearn.datasets import make_regression

# Small-data regime: BLR vs Ridge
X, y = make_regression(n_samples=30, n_features=20, noise=10, random_state=42)

blr = BayesianRidge()
ridge = Ridge(alpha=1.0)

blr_scores = cross_val_score(blr, X, y, cv=5, scoring='r2')
ridge_scores = cross_val_score(ridge, X, y, cv=5, scoring='r2')

print(f"BLR  R²: {blr_scores.mean():.3f} ± {blr_scores.std():.3f}")
print(f"Ridge R²: {ridge_scores.mean():.3f} ± {ridge_scores.std():.3f}")
# BLR often matches or beats Ridge in small-data regimes
# because it marginalizes over hyperparameters instead of fixing them
