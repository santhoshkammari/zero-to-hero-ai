Approximate Inference: Sampling Methods (MCMC)

Chapter 14: Probabilistic & Bayesian ML 5 subtopics
TL;DR

When you cannot compute a posterior distribution in closed form — which is most of the time — you draw samples from it instead and use those samples as a stand-in for the true distribution. Monte Carlo methods give you the basic idea: estimate an integral by averaging random samples. Markov Chain Monte Carlo (MCMC) extends this to distributions you can only evaluate up to a normalizing constant. Metropolis-Hastings and Gibbs sampling are the classic workhorses. Hamiltonian Monte Carlo and NUTS are the modern upgrades that make Bayesian inference practical on hundreds of parameters. This section builds the full ladder from rejection sampling to NUTS, with runnable PyMC and NumPyro code at every step.

Monte Carlo Fundamentals

Imagine you need to compute the expected value of some function under a complicated probability distribution. The integral has no closed-form solution. What do you do? You fake it with randomness. Draw a bunch of samples from the distribution, evaluate the function at each sample, and average the results. That average converges to the true expectation as you draw more samples. This is the Monte Carlo method — one of the most powerful ideas in computational science.

Law of Large Numbers for Estimation

The theoretical foundation is the law of large numbers. If you draw N independent samples x₁, x₂, …, xₙ from a distribution p(x), then the sample average of any function f converges to the true expected value:

(1/N) Σ f(xᵢ) → E_p[f(x)] as N → ∞

The error shrinks at a rate of 1/√N regardless of the dimensionality of x. That is both a blessing and a curse — you always get convergence, but cutting the error in half requires quadrupling your samples.

import numpy as np

# Estimate E[X^2] where X ~ Normal(0, 1)
# Analytical answer: Var(X) + E[X]^2 = 1 + 0 = 1
np.random.seed(42)
samples = np.random.normal(0, 1, size=100_000)
estimate = np.mean(samples ** 2)
print(f"Monte Carlo estimate of E[X²]: {estimate:.4f}")  # ≈ 1.0000

# Track convergence
running_avg = np.cumsum(samples ** 2) / np.arange(1, len(samples) + 1)
print(f"After 100 samples: {running_avg[99]:.4f}")
print(f"After 10,000 samples: {running_avg[9999]:.4f}")
print(f"After 100,000 samples: {running_avg[-1]:.4f}")

Importance Sampling

What if you cannot sample from p(x) directly? Importance sampling lets you sample from an easier proposal distribution q(x) and reweight each sample by the ratio p(x)/q(x). The idea is that samples from q that land in high-probability regions of p get upweighted, and samples in low-probability regions of p get downweighted.

import numpy as np
from scipy import stats

# Estimate P(X > 4) where X ~ Normal(0,1)
# This is a rare event — naive MC would need millions of samples
np.random.seed(42)

# Naive Monte Carlo
naive_samples = np.random.normal(0, 1, size=100_000)
naive_estimate = np.mean(naive_samples > 4)
print(f"Naive MC: P(X>4) ≈ {naive_estimate:.6f}")

# Importance sampling with proposal q = Normal(4, 1)
n_samples = 10_000
q_samples = np.random.normal(4, 1, size=n_samples)
weights = stats.norm.pdf(q_samples, 0, 1) / stats.norm.pdf(q_samples, 4, 1)
is_estimate = np.mean(weights * (q_samples > 4))
print(f"Importance sampling: P(X>4) ≈ {is_estimate:.6f}")
print(f"True value: {1 - stats.norm.cdf(4):.6f}")  # 3.167e-5

The catch: if q is a poor match for p, some weights become enormous while most are tiny. Your estimate is then dominated by a few lucky samples, and the variance explodes. This is called weight degeneracy, and it gets worse in high dimensions.

Rejection Sampling

Rejection sampling takes a different approach. You find a proposal distribution q(x) and a constant M such that M·q(x) ≥ p(x) everywhere. You sample from q, then accept the sample with probability p(x)/(M·q(x)). Accepted samples are exactly distributed according to p.

The problem is finding a tight M. In high dimensions, the acceptance rate plummets because the proposal envelope wastes volume in regions where p is small. For a 100-dimensional distribution, you might accept one sample in every 10^50 tries. This makes rejection sampling useless for the posterior distributions we care about in Bayesian ML.

import numpy as np
from scipy import stats

# Rejection sampling: sample from Beta(2.7, 6.3) using Uniform(0,1)
np.random.seed(42)
target = stats.beta(2.7, 6.3)
M = target.pdf(target.mean())  # upper bound on the density

accepted = []
n_proposed = 0
while len(accepted) < 10_000:
    x = np.random.uniform(0, 1)
    u = np.random.uniform(0, M)
    n_proposed += 1
    if u <= target.pdf(x):
        accepted.append(x)

acceptance_rate = len(accepted) / n_proposed
print(f"Acceptance rate: {acceptance_rate:.2%}")
print(f"Sample mean: {np.mean(accepted):.4f} (true: {target.mean():.4f})")

Ancestral Sampling

Ancestral sampling works when your model is a directed graphical model (a Bayesian network). You sample each variable in topological order, conditioning on the values you have already drawn for its parents. Each step samples from a simple conditional distribution. This is efficient when you can evaluate and sample from each conditional, but it does not help when you need samples from the posterior given observed data — for that, you need MCMC.

Markov Chain Monte Carlo (MCMC)

Here is the core problem: you have a posterior distribution p(θ|data) that you can evaluate up to a normalizing constant. You can compute p(data|θ)·p(θ) for any θ, but the integral in the denominator — the evidence p(data) — is intractable. None of the basic Monte Carlo methods work well in high dimensions for this situation.

Markov Chain Monte Carlo solves this by constructing a Markov chain whose stationary distribution is exactly the posterior you want. You run the chain long enough, and the samples it produces become draws from the posterior. The brilliance is that you never need to know the normalizing constant — the chain cancels it out automatically.

Markov Chain Construction

A Markov chain is a sequence of random variables where each value depends only on the previous one — not on the entire history. You define it by specifying a transition kernel T(θ′|θ): given the current state θ, what is the probability of moving to θ′?

Stationary Distribution and Detailed Balance

A distribution π is the stationary distribution of the chain if, when the chain is in state θ distributed according to π and you apply one transition step, the resulting state θ′ is also distributed according to π. The standard way to guarantee this is detailed balance: π(θ)T(θ′|θ) = π(θ′)T(θ|θ′). This says the flow of probability from θ to θ′ equals the flow from θ′ to θ. If your transition kernel satisfies this condition with respect to the posterior, then the posterior is the chain's stationary distribution.

Ergodicity, Burn-In, and Thinning

Ergodicity means the chain can reach any state from any other state — it does not get trapped in a subset of the space. Combined with detailed balance, ergodicity guarantees that the chain converges to the stationary distribution regardless of where it starts.

But the chain's starting point is usually far from the bulk of the posterior. The early samples are biased toward the initialization. We discard these early samples in a burn-in period. How long should burn-in be? There is no universal answer, but convergence diagnostics help you decide.

Thinning means keeping only every k-th sample to reduce autocorrelation between consecutive draws. If your chain moves slowly, consecutive samples are nearly identical and give you less information than independent draws would. Thinning trades sample size for lower autocorrelation. Modern practice favors running longer chains over aggressive thinning, since you are throwing away computation.

Convergence Diagnostics (R̂ and Trace Plots)

How do you know the chain has converged? You run multiple chains from different starting points and compare them. The R̂ statistic (Gelman-Rubin diagnostic) compares the variance within each chain to the variance between chains. If R̂ is close to 1.0 (typically < 1.01), the chains have converged to the same distribution. Values above 1.1 are a red flag.

Trace plots show the sampled values over time. A well-mixed chain looks like a fuzzy caterpillar — no trends, no long excursions, rapid oscillation around the mean. A poorly mixed chain shows sticky patches or slow drifts.

import pymc as pm
import arviz as az
import numpy as np

# Generate data from a known model
np.random.seed(42)
true_mu, true_sigma = 5.0, 2.0
data = np.random.normal(true_mu, true_sigma, size=100)

# Build Bayesian model and run MCMC
with pm.Model() as normal_model:
    mu = pm.Normal("mu", mu=0, sigma=10)
    sigma = pm.HalfNormal("sigma", sigma=10)
    likelihood = pm.Normal("obs", mu=mu, sigma=sigma, observed=data)

    # Draw 4 chains × 2000 samples each
    trace = pm.sample(2000, chains=4, random_seed=42)

# Convergence diagnostics
summary = az.summary(trace, var_names=["mu", "sigma"])
print(summary[["mean", "sd", "r_hat", "ess_bulk"]])
# r_hat should be ≈ 1.00 for both parameters

# Trace plots reveal mixing behavior
az.plot_trace(trace, var_names=["mu", "sigma"])

Metropolis-Hastings

The Metropolis-Hastings algorithm is the granddaddy of MCMC. It gives you a recipe for constructing a transition kernel that satisfies detailed balance with respect to any target distribution — even one you can only evaluate up to a constant.

Proposal Distribution and Acceptance Ratio

The algorithm works in two steps. First, propose a new state θ′ from a proposal distribution q(θ′|θ). Second, accept the proposal with probability α = min(1, [p(θ′)q(θ|θ′)] / [p(θ)q(θ′|θ)]). If you accept, move to θ′. If you reject, stay at θ. The normalizing constant of p cancels in the ratio, which is the whole trick.

import numpy as np
from scipy import stats

def metropolis_hastings(log_target, proposal_std, n_samples, init=0.0):
    """Generic Metropolis-Hastings with Gaussian random walk proposal."""
    samples = np.zeros(n_samples)
    current = init
    current_log_p = log_target(current)
    accepted = 0

    for i in range(n_samples):
        # Propose: symmetric Gaussian ⟹ q(θ'|θ) = q(θ|θ'), ratio cancels
        proposed = current + np.random.normal(0, proposal_std)
        proposed_log_p = log_target(proposed)

        # Accept/reject
        log_alpha = proposed_log_p - current_log_p
        if np.log(np.random.uniform()) < log_alpha:
            current = proposed
            current_log_p = proposed_log_p
            accepted += 1

        samples[i] = current

    print(f"Acceptance rate: {accepted / n_samples:.2%}")
    return samples

# Target: mixture of two Gaussians
def log_target(x):
    return np.log(0.3 * stats.norm.pdf(x, -2, 0.5) +
                  0.7 * stats.norm.pdf(x, 3, 1.0))

np.random.seed(42)
samples = metropolis_hastings(log_target, proposal_std=1.5, n_samples=50_000)
burn_in = 5000
posterior_samples = samples[burn_in:]
print(f"Posterior mean: {np.mean(posterior_samples):.3f}")

Random Walk Metropolis

When the proposal is a symmetric distribution centered on the current state — like q(θ′|θ) = Normal(θ, σ²) — the proposal ratio q(θ|θ′)/q(θ′|θ) equals 1, and the acceptance ratio simplifies to min(1, p(θ′)/p(θ)). This is the random walk Metropolis algorithm. It is the simplest MCMC method to implement and understand.

Tuning the Proposal

The proposal standard deviation σ controls the tradeoff between exploration and acceptance rate. Too small: nearly every proposal is accepted, but the chain takes tiny steps and explores slowly. Too large: the chain proposes big jumps into low-probability regions and rejects most of them. The optimal acceptance rate for a random walk Metropolis in high dimensions is approximately 23.4% — a theoretical result by Roberts, Gelman, and Gilks (1997). Most libraries auto-tune σ during burn-in to hit this target.

💡 Tuning Heuristic

Start with a proposal standard deviation roughly equal to the posterior standard deviation. During warm-up, adjust it until your acceptance rate is between 20% and 50%. PyMC and Stan do this automatically, but knowing the rule helps you diagnose problems when auto-tuning fails.

Gibbs Sampling

Gibbs sampling avoids the accept/reject step entirely. Instead of proposing a move in all dimensions at once, it updates one variable at a time by sampling from its full conditional distribution — the distribution of that variable given the current values of all other variables.

Full Conditional Distributions

For each variable θᵢ, the full conditional is p(θᵢ | θ₁, …, θᵢ₋₁, θᵢ₊₁, …, θ_d, data). In conjugate models, these conditionals have closed-form distributions. For example, in a normal-normal model, the conditional for the mean given the variance is another normal distribution. You sample from it directly — no proposal, no rejection.

import numpy as np

def gibbs_normal_model(data, n_samples=10_000, burn_in=1_000):
    """Gibbs sampler for Normal(mu, 1/tau) with conjugate priors.

    Priors: mu ~ Normal(0, 1/tau_0), tau ~ Gamma(a, b)
    """
    n = len(data)
    x_bar = np.mean(data)

    # Hyperparameters
    mu_0, tau_0 = 0.0, 0.01   # prior on mu
    a_0, b_0 = 1.0, 1.0       # prior on tau (precision)

    # Initialize
    mu = 0.0
    tau = 1.0

    mu_samples = np.zeros(n_samples)
    tau_samples = np.zeros(n_samples)

    for i in range(n_samples + burn_in):
        # Sample mu | tau, data ~ Normal(posterior_mean, 1/posterior_precision)
        prec_mu = tau_0 + n * tau
        mean_mu = (tau_0 * mu_0 + tau * n * x_bar) / prec_mu
        mu = np.random.normal(mean_mu, 1.0 / np.sqrt(prec_mu))

        # Sample tau | mu, data ~ Gamma(posterior_a, posterior_b)
        a_post = a_0 + n / 2
        b_post = b_0 + 0.5 * np.sum((data - mu) ** 2)
        tau = np.random.gamma(a_post, 1.0 / b_post)

        if i >= burn_in:
            mu_samples[i - burn_in] = mu
            tau_samples[i - burn_in] = tau

    return mu_samples, tau_samples

np.random.seed(42)
data = np.random.normal(5.0, 2.0, size=200)
mu_post, tau_post = gibbs_normal_model(data)
print(f"Posterior mu: {np.mean(mu_post):.3f} ± {np.std(mu_post):.3f}")
print(f"Posterior sigma: {np.mean(1/np.sqrt(tau_post)):.3f}")

Systematic vs. Random Scan

In systematic scan Gibbs, you cycle through variables in a fixed order: update θ₁, then θ₂, …, then θ_d, then repeat. In random scan, you pick a variable uniformly at random at each step. Both produce valid MCMC chains. Systematic scan is more common in practice because it guarantees every variable gets updated in each sweep.

Block and Collapsed Gibbs Sampling

Block Gibbs sampling updates groups of variables jointly instead of one at a time. If two variables are highly correlated, updating them individually creates slow, correlated chains. Updating them together breaks the correlation. Collapsed Gibbs sampling integrates out some variables analytically and samples only the remaining ones. Fewer variables means faster mixing. The classic example is collapsed Gibbs for Latent Dirichlet Allocation, where the topic-word and document-topic distributions are integrated out analytically.

Advanced MCMC Methods

Hamiltonian Monte Carlo (HMC)

Random walk Metropolis and Gibbs both struggle in high dimensions because they make small, correlated steps. Hamiltonian Monte Carlo (HMC) borrows a trick from physics. It treats the negative log-posterior as a potential energy surface and introduces fictitious momentum variables. The sampler then simulates a particle rolling across this surface using Hamilton's equations of motion. Instead of a random walk, the particle follows a smooth trajectory that can travel far from the current state while staying in high-probability regions.

The physics simulation is done with the leapfrog integrator, a numerical method that is time-reversible and volume-preserving — properties needed to maintain detailed balance. At the end of the trajectory, you do a Metropolis accept/reject step to correct for numerical error in the integration.

HMC has two tuning parameters: the step size ε and the number of leapfrog steps L. Too large a step size causes the integrator to diverge. Too few steps and you are back to a random walk. Too many steps and you waste computation retracing your path.

import numpy as np

def hmc(log_prob, grad_log_prob, init, n_samples, step_size, n_leapfrog):
    """Hamiltonian Monte Carlo sampler."""
    dim = len(init)
    samples = np.zeros((n_samples, dim))
    current_q = init.copy()
    accepted = 0

    for i in range(n_samples):
        q = current_q.copy()
        p = np.random.normal(0, 1, size=dim)  # sample momentum
        current_p = p.copy()

        # Leapfrog integration
        p += 0.5 * step_size * grad_log_prob(q)  # half step momentum
        for _ in range(n_leapfrog - 1):
            q += step_size * p                      # full step position
            p += step_size * grad_log_prob(q)       # full step momentum
        q += step_size * p                          # final position step
        p += 0.5 * step_size * grad_log_prob(q)     # final half step momentum
        p = -p  # negate momentum for reversibility

        # Metropolis correction
        current_H = -log_prob(current_q) + 0.5 * np.sum(current_p ** 2)
        proposed_H = -log_prob(q) + 0.5 * np.sum(p ** 2)

        if np.log(np.random.uniform()) < current_H - proposed_H:
            current_q = q
            accepted += 1

        samples[i] = current_q

    print(f"HMC acceptance rate: {accepted / n_samples:.2%}")
    return samples

# 2D correlated Gaussian target
rho = 0.95
cov = np.array([[1, rho], [rho, 1]])
cov_inv = np.linalg.inv(cov)

log_prob = lambda q: -0.5 * q @ cov_inv @ q
grad_log_prob = lambda q: -cov_inv @ q

np.random.seed(42)
samples = hmc(log_prob, grad_log_prob, init=np.zeros(2),
              n_samples=5000, step_size=0.15, n_leapfrog=20)
print(f"Sample mean: {samples[500:].mean(axis=0)}")
print(f"Sample corr: {np.corrcoef(samples[500:].T)[0,1]:.3f} (true: {rho})")

No-U-Turn Sampler (NUTS)

The No-U-Turn Sampler eliminates the need to choose the number of leapfrog steps L. It builds the trajectory in both directions until the trajectory starts to double back on itself — a "U-turn." This adaptive criterion automatically picks a good trajectory length for each sample. NUTS, combined with step-size adaptation during warm-up, is the default sampler in PyMC and Stan. It is the reason modern probabilistic programming works as well as it does.

import pymc as pm
import numpy as np

# NUTS in action via PyMC — you rarely write NUTS by hand
np.random.seed(42)
N = 500
true_beta = np.array([1.5, -2.0, 0.8])
X = np.random.randn(N, 3)
y = X @ true_beta + np.random.normal(0, 0.5, N)

with pm.Model() as regression_model:
    beta = pm.Normal("beta", mu=0, sigma=5, shape=3)
    sigma = pm.HalfNormal("sigma", sigma=5)
    mu = pm.math.dot(X, beta)
    likelihood = pm.Normal("y", mu=mu, sigma=sigma, observed=y)

    # PyMC auto-selects NUTS for continuous parameters
    trace = pm.sample(2000, chains=4, target_accept=0.9, random_seed=42)

import arviz as az
print(az.summary(trace, var_names=["beta", "sigma"])[["mean", "sd", "r_hat"]])
⚠️ Divergences

NUTS reports divergent transitions — leapfrog steps where the numerical integrator went off the rails. Divergences indicate the sampler is struggling with the geometry of the posterior (often a funnel shape or sharp curvature). If you see divergences, increase target_accept (e.g., to 0.95 or 0.99) or reparameterize your model. Never ignore divergences — they mean your samples may be biased.

Slice Sampling, Parallel Tempering, and Reversible Jump

Slice sampling introduces an auxiliary variable to convert sampling from a distribution into uniform sampling within a region under the density curve. It requires no tuning parameters and adapts automatically to the local scale of the distribution.

Parallel tempering runs multiple chains at different "temperatures." Hot chains explore freely (flattened distribution), cold chains stay near the posterior modes. Chains periodically swap states, allowing the cold chain to jump between modes. This is invaluable for multimodal posteriors where a single chain would get stuck in one mode forever.

Reversible jump MCMC extends standard MCMC to models where the number of parameters itself is unknown — for example, choosing the number of components in a mixture model. The chain can propose to add or remove a dimension, making it possible to do Bayesian model selection within a single MCMC run.

import numpyro
import numpyro.distributions as dist
from numpyro.infer import MCMC, NUTS
import jax.numpy as jnp
import jax.random as random

# NumPyro: fast NUTS on JAX
def bayesian_regression(X, y=None):
    n_features = X.shape[1]
    beta = numpyro.sample("beta", dist.Normal(0, 5).expand([n_features]))
    sigma = numpyro.sample("sigma", dist.HalfNormal(5))
    mu = jnp.dot(X, beta)
    numpyro.sample("y", dist.Normal(mu, sigma), obs=y)

kernel = NUTS(bayesian_regression, target_accept_prob=0.9)
mcmc = MCMC(kernel, num_warmup=1000, num_samples=2000, num_chains=4)
mcmc.run(random.PRNGKey(42), X=jnp.array(X), y=jnp.array(y))
mcmc.print_summary()
