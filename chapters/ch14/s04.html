Exact Inference

Chapter 14: Probabilistic & Bayesian ML 3 subtopics
TL;DR

Exact inference computes precise marginal or MAP probabilities in graphical models. Variable elimination systematically sums out variables one at a time, with computation cost depending heavily on elimination ordering. Belief propagation passes messages between nodes — exact on trees, approximate on loopy graphs. The junction tree algorithm transforms any graph into a tree of cliques where belief propagation becomes exact. These algorithms are the foundation upon which approximate methods are motivated.

You have built a beautiful graphical model. Nodes represent diseases, symptoms, test results, and risk factors. The edges encode which variables influence which. Now the patient walks in with specific symptoms, and you need to answer: "What is the probability of each disease, given the observed symptoms?" That is inference — computing posterior probabilities given evidence.

For small models with a handful of variables, you could enumerate all configurations and sum. But with 50 variables, each taking 10 values, you would need 10⁵⁰ operations. Exact inference algorithms exploit the graph structure to do this in polynomial time (for tree-structured graphs) or manageable exponential time (for general graphs with small treewidth). Understanding these algorithms is essential — even if you end up using approximate methods in practice, you need to know what the exact answer looks like to evaluate approximations.

Variable Elimination

Elimination Ordering

Variable elimination is the most intuitive exact inference algorithm. The idea: to compute a marginal like P(X₁), sum out all other variables one at a time. At each step, you combine all factors involving the variable being eliminated, sum over that variable, and produce a new (smaller) factor. The order in which you eliminate variables dramatically affects the computational cost.

Consider a chain: X₁ → X₂ → X₃ → X₄ → X₅. To compute P(X₁), eliminate X₅ first (involves only X₄, X₅), then X₄ (involves X₃ and the new factor), and so on. Each step involves small factors. But if you eliminate X₃ first in a more connected graph, you might create a giant intermediate factor. Choosing the optimal elimination ordering is NP-hard in general, but good heuristics (min-degree, min-fill) work well.

import numpy as np
from itertools import product

# Variable elimination on a simple chain: A -> B -> C -> D
# Question: compute P(A) after observing D=1

# Define conditional probability tables
# P(A): 2 states
P_A = np.array([0.6, 0.4])

# P(B|A): shape (A, B)
P_B_given_A = np.array([
    [0.7, 0.3],  # A=0
    [0.2, 0.8],  # A=1
])

# P(C|B): shape (B, C)
P_C_given_B = np.array([
    [0.9, 0.1],  # B=0
    [0.3, 0.7],  # B=1
])

# P(D|C): shape (C, D)
P_D_given_C = np.array([
    [0.8, 0.2],  # C=0
    [0.1, 0.9],  # C=1
])

# Step 1: Incorporate evidence D=1
# Reduce P(D|C) to P(D=1|C)
factor_D = P_D_given_C[:, 1]  # [0.2, 0.9]

# Step 2: Eliminate C (sum over C)
# Combine P(C|B) and factor_D, then sum over C
# factor_BC[b] = sum_c P(C=c|B=b) * factor_D[c]
factor_B = np.einsum('bc,c->b', P_C_given_B, factor_D)

# Step 3: Eliminate B (sum over B)
# Combine P(B|A) and factor_B, then sum over B
factor_A = np.einsum('ab,b->a', P_B_given_A, factor_B)

# Step 4: Combine with P(A) and normalize
unnormalized = P_A * factor_A
P_A_given_D1 = unnormalized / unnormalized.sum()

print(f"P(A | D=1) = {P_A_given_D1}")
# Observing D=1 shifts our belief about A through the chain

Intermediate Factor Computation

Each elimination step creates an intermediate factor. When eliminating variable X from a set of factors, you multiply all factors containing X, sum over X, and get a new factor over the remaining variables. The size of intermediate factors determines the computational cost. If the largest intermediate factor involves k variables each with d states, that step costs O(dᵏ). The maximum intermediate factor size over the whole elimination is related to the treewidth of the graph.

Complexity vs Treewidth

The treewidth of a graph measures how "tree-like" it is. Trees have treewidth 1. A complete graph on n nodes has treewidth n-1. Variable elimination runs in O(n × dᵗʷ⁺¹) time, where tw is the treewidth. For tree-structured graphs, inference is linear in the number of nodes. For graphs with bounded treewidth, it is polynomial. For general graphs, exact inference is NP-hard — which is why approximate methods exist.

Treewidth in Practice

Hidden Markov models have treewidth 1 (they are chains). Grid-structured MRFs (like images) have treewidth proportional to the grid width — manageable for narrow grids, intractable for large images. Most real-world Bayesian networks have moderate treewidth, making variable elimination practical. When treewidth is too high, switch to approximate inference.

Belief Propagation (Sum-Product)

Message Passing on Trees

Belief propagation computes marginals by passing local messages between neighboring nodes. Each node sends a message to each neighbor summarizing what it "knows" based on messages received from other neighbors. On tree-structured graphs, two passes (leaves to root, then root to leaves) compute all marginals exactly. This is the sum-product algorithm.

The message from node i to node j is: m_{i→j}(x_j) = Σ_{x_i} ψ(x_i, x_j) × ∏_{k∈N(i)\j} m_{k→i}(x_i). In words: consider all states of x_i, weight each by the pairwise potential and all incoming messages except from j, and sum. The marginal at node i is proportional to the product of all incoming messages: P(x_i) ∝ ∏_{k∈N(i)} m_{k→i}(x_i).

import numpy as np

class BeliefPropagation:
    """Sum-product belief propagation on a tree (chain)."""

    def __init__(self, unary, pairwise):
        """
        unary: list of arrays, unary[i] = local evidence for node i
        pairwise: list of arrays, pairwise[i] = potential between node i and i+1
        """
        self.unary = unary
        self.pairwise = pairwise
        self.n = len(unary)

    def run(self):
        n = self.n
        # Forward messages (left to right)
        fwd = [None] * n
        fwd[0] = self.unary[0]
        for i in range(1, n):
            # Message from i-1 to i
            msg = self.pairwise[i-1].T @ fwd[i-1]
            fwd[i] = self.unary[i] * msg

        # Backward messages (right to left)
        bwd = [None] * n
        bwd[n-1] = np.ones_like(self.unary[n-1])
        for i in range(n-2, -1, -1):
            msg = self.pairwise[i] @ (self.unary[i+1] * bwd[i+1])
            bwd[i] = msg

        # Marginals
        marginals = []
        for i in range(n):
            belief = fwd[i] * bwd[i]
            belief /= belief.sum()
            marginals.append(belief)

        return marginals

# Example: chain of 5 binary variables
np.random.seed(42)

# Unary potentials (local evidence)
unary = [np.array([0.8, 0.2]),  # node 0 likely state 0
         np.array([0.5, 0.5]),  # node 1 uninformative
         np.array([0.3, 0.7]),  # node 2 likely state 1
         np.array([0.5, 0.5]),  # node 3 uninformative
         np.array([0.9, 0.1])]  # node 4 likely state 0

# Pairwise potentials (encourage agreement)
pairwise = [np.array([[2, 1], [1, 2]]) for _ in range(4)]

bp = BeliefPropagation(unary, pairwise)
marginals = bp.run()

for i, m in enumerate(marginals):
    print(f"Node {i}: P(state=0)={m[0]:.3f}, P(state=1)={m[1]:.3f}")
# Messages propagate evidence along the chain

Forward-Backward Algorithm (Chains)

The forward-backward algorithm is belief propagation specialized to chains — exactly what you use for hidden Markov models (HMMs). The forward pass computes α_t(j) = P(x_1:t, z_t=j), accumulating evidence from left to right. The backward pass computes β_t(j) = P(x_{t+1:T} | z_t=j), accumulating from right to left. The product gives the smoothed marginals: P(z_t=j | x_1:T) ∝ α_t(j) × β_t(j). This is the exact same algorithm as belief propagation on a chain, wearing a different hat.

Marginal Computation

At each node, the marginal is the element-wise product of all incoming messages, normalized. For pairwise marginals (the joint distribution of two connected nodes), multiply the messages and the pairwise potential. These pairwise marginals are useful for learning — when you need expected sufficient statistics under the posterior for parameter updates.

Max-Product (MAP Inference)

Replace sums with max operations, and sum-product becomes max-product — also known as the Viterbi algorithm on chains. Instead of computing marginal probabilities, max-product finds the most probable joint configuration: argmax P(X). For HMMs, this gives you the most likely hidden state sequence. The messages propagate the best partial solution forward, and a traceback step reconstructs the optimal sequence.

import numpy as np

def viterbi(unary, pairwise):
    """Max-product (Viterbi) on a chain."""
    n = len(unary)
    n_states = len(unary[0])

    # Forward pass with max
    delta = [None] * n
    backptr = [None] * n
    delta[0] = np.log(unary[0] + 1e-10)

    for t in range(1, n):
        delta[t] = np.zeros(n_states)
        backptr[t] = np.zeros(n_states, dtype=int)
        for j in range(n_states):
            scores = delta[t-1] + np.log(pairwise[t-1][:, j] + 1e-10)
            backptr[t][j] = np.argmax(scores)
            delta[t][j] = scores[backptr[t][j]] + np.log(unary[t][j] + 1e-10)

    # Traceback
    path = np.zeros(n, dtype=int)
    path[-1] = np.argmax(delta[-1])
    for t in range(n-2, -1, -1):
        path[t] = backptr[t+1][path[t+1]]

    return path

# Reuse the same unary/pairwise from BP example
path = viterbi(unary, pairwise)
print(f"MAP sequence: {path}")
# The most likely joint assignment, accounting for both local
# evidence and pairwise consistency
Marginals ≠ MAP

A common mistake: taking the most likely state at each node independently (from sum-product marginals) and treating that as the MAP solution. This is wrong. The independently most probable states may be jointly impossible or highly unlikely. The Viterbi/max-product algorithm finds the joint optimum, which can differ from the marginal modes. Always use the right algorithm for your question.

Junction Tree Algorithm

Triangulation (Chordal Graph)

Belief propagation is exact on trees. What about graphs with loops? The junction tree algorithm converts any graph into a tree of cliques where belief propagation becomes exact. Step one: triangulate the graph — add edges until every cycle of length 4 or more has a chord (a shortcut edge). The resulting graph is called chordal. The added edges increase clique sizes but make the tree decomposition possible.

Clique Tree Construction

From the chordal graph, identify all maximal cliques. Build a tree over these cliques such that the running intersection property holds: for any variable that appears in two cliques, it must appear in every clique on the path between them. This tree is the junction tree (also called a clique tree). The edges between cliques are labeled with separators — the variables shared between adjacent cliques.

Calibration

Calibration is the process of running belief propagation on the junction tree. Messages between cliques are functions of the separator variables. After two passes (upward and downward), every clique has a calibrated potential — a table proportional to the marginal over the clique's variables. You can then read off marginals by summing out variables from the appropriate clique potential.

import numpy as np

# Simplified junction tree example
# Original graph: A - B - C - D with extra edge A - C (a triangle)
# Cliques: {A, B, C} and {C, D}
# Separator: {C}

# Clique potential for {A, B, C}: 2x2x2 tensor
np.random.seed(42)
phi_ABC = np.random.rand(2, 2, 2) + 0.1  # all positive

# Clique potential for {C, D}: 2x2 matrix
phi_CD = np.random.rand(2, 2) + 0.1

# Message from {C,D} to {A,B,C} via separator {C}
# Sum out D from phi_CD
msg_CD_to_ABC = phi_CD.sum(axis=1)  # shape (2,) — over C

# Update clique {A,B,C} with the message
# Multiply along the C axis
calibrated_ABC = phi_ABC * msg_CD_to_ABC[np.newaxis, np.newaxis, :]

# Message from {A,B,C} to {C,D} via separator {C}
# Sum out A and B
msg_ABC_to_CD = calibrated_ABC.sum(axis=(0, 1))  # shape (2,) — over C

# Update clique {C,D}
calibrated_CD = phi_CD * msg_ABC_to_CD[:, np.newaxis]

# Extract marginals
P_C = calibrated_ABC.sum(axis=(0, 1))
P_C /= P_C.sum()
print(f"P(C): {P_C}")

P_A = calibrated_ABC.sum(axis=(1, 2))
P_A /= P_A.sum()
print(f"P(A): {P_A}")

Running Intersection Property

The running intersection property is what makes the junction tree correct. It ensures that information about a shared variable flows consistently through the tree. Without this property, messages about the same variable could take different paths and give inconsistent results. Verifying this property is a standard step when constructing junction trees — and it is what distinguishes a valid junction tree from an arbitrary tree over cliques.

When Exact Inference Is Impractical

The junction tree's complexity is exponential in the size of the largest clique. For densely connected graphs, triangulation creates enormous cliques, and the algorithm becomes intractable. A 100-node graph with treewidth 20 means cliques of size 21, each requiring tables of size d²¹. For binary variables, that is about 2 million entries per clique — feasible. For variables with 100 states, each table has 100²¹ entries — absurd. This is exactly where approximate inference (Chapter 14, Sections 5 and beyond) becomes necessary.
