Gaussian Processes & Bayesian Optimization

Chapter 14: Probabilistic & Bayesian ML 7 subtopics
TL;DR

Gaussian Processes define a distribution over functions — instead of picking a specific model shape, you let the data decide. GPs give beautiful uncertainty estimates but scale as O(n³). Bayesian Optimization uses GPs (or similar surrogates) to intelligently search for the optimum of expensive black-box functions. It’s the smart way to tune hyperparameters.

Gaussian Processes: A Distribution Over Functions

Most regression models fix a functional form: linear, polynomial, neural network with a specific architecture. A Gaussian Process says: “I won’t commit to a form. Instead, I’ll define a distribution over all possible functions and let the data narrow it down.”

Formally: a GP is a collection of random variables, any finite subset of which has a joint Gaussian distribution. A GP is fully specified by a mean function m(x) and a kernel (covariance) function k(x, x’):

f(x) ~ GP(m(x), k(x, x'))

For any finite set of points x1, ..., xn:
[f(x1), ..., f(xn)]^T ~ N(μ, K)

where μ_i = m(x_i) and K_ij = k(x_i, x_j)

The kernel controls everything: how smooth the functions are, how quickly they vary, whether they’re periodic. You don’t pick an architecture — you pick a kernel.

Kernel Functions

The kernel k(x, x’) measures how correlated function values at x and x’ should be. Nearby points → high correlation → smooth functions. Distant points → low correlation → more flexibility.

KernelFormulaPropertiesUse When
RBF (Squared Exponential)exp(-||x-x’||² / 2l²)Infinitely smoothDefault choice, smooth functions
Matérn-3/2(1+√3r/l) exp(-√3r/l)Once differentiablePhysical processes, less unrealistically smooth than RBF
Matérn-5/2(1+√5r/l+5r²/3l²) exp(-√5r/l)Twice differentiableGood default, most practical applications
Periodicexp(-2sin²(π|x-x’|/p) / l²)Repeating patternsSeasonal data, periodic signals
Linearσ² + x·x’Encodes linearityWhen you believe the function is linear

The lengthscale l controls how far apart points can be and still be correlated. Short lengthscale → wiggly functions. Long lengthscale → smooth functions. The lengthscale is typically learned from data by maximizing the marginal likelihood.

GP Regression

Condition the GP on observed data and the posterior is also a GP — with updated mean and covariance. The math is just Gaussian conditioning (block matrix algebra), giving you:

Predictive mean:     μ* = K*^T (K + σ²I)^{-1} y
Predictive variance: σ*² = K** - K*^T (K + σ²I)^{-1} K*

The mean is a weighted combination of training outputs. The variance shrinks near training points and grows in unexplored regions. Exactly the behavior you want.

import numpy as np
from sklearn.gaussian_process import GaussianProcessRegressor
from sklearn.gaussian_process.kernels import Matern, WhiteKernel
import matplotlib.pyplot as plt

# Generate sparse, noisy data
np.random.seed(42)
X_train = np.sort(np.random.uniform(0, 10, 15)).reshape(-1, 1)
y_train = np.sin(X_train).ravel() + 0.2 * np.random.randn(15)

# Fit GP with Matern kernel + noise
kernel = Matern(nu=2.5, length_scale=1.0) + WhiteKernel(noise_level=0.1)
gp = GaussianProcessRegressor(kernel=kernel, n_restarts_optimizer=10)
gp.fit(X_train, y_train)

# Predict with uncertainty
X_test = np.linspace(-1, 11, 300).reshape(-1, 1)
y_pred, y_std = gp.predict(X_test, return_std=True)

fig, ax = plt.subplots(figsize=(10, 5))
ax.scatter(X_train, y_train, c='black', s=30, zorder=5, label='Observations')
ax.plot(X_test, y_pred, 'b-', label='GP mean')
ax.fill_between(X_test.ravel(), y_pred - 2*y_std, y_pred + 2*y_std,
                alpha=0.2, color='blue', label='95% CI')
ax.plot(X_test, np.sin(X_test), 'r--', alpha=0.5, label='True function')
ax.set_title(f'GP Regression (learned kernel: {gp.kernel_})')
ax.legend()
plt.tight_layout()
plt.show()

GP Limitations and Sparse Approximations

The elephant in the room: GP regression requires inverting an n×n kernel matrix, which is O(n³) in time and O(n²) in memory. For n > ~10,000, this is impractical.

Solutions:

  • Sparse GPs (inducing points): Select m << n “pseudo-inputs” and approximate the full GP using only these points. Reduces complexity to O(nm²). Libraries: GPyTorch, GPflow.
  • Random Fourier Features: Approximate the kernel with random features, reducing the GP to a Bayesian linear regression problem.
  • GPyTorch: GPU-accelerated GP inference with modern approximations. Can handle 100K+ points.

Bayesian Optimization

The problem: you have a function that’s expensive to evaluate (training a neural network, running a simulation, a wet-lab experiment). You want to find the optimum in as few evaluations as possible. Grid search is wasteful. Random search is better but still uninformed.

Bayesian optimization uses a surrogate model (typically a GP) to model the objective function, and an acquisition function to decide where to evaluate next. The loop:

  1. Fit the GP surrogate to all evaluations so far
  2. Maximize the acquisition function to find the most promising next point
  3. Evaluate the true (expensive) function at that point
  4. Add the new observation and repeat

Acquisition Functions

The acquisition function balances exploration (try uncertain regions) vs. exploitation (try regions near the current best). Three classics:

Expected Improvement (EI): How much improvement over the current best do we expect at this point? EI is zero at the current best and high where the GP predicts high mean or high variance. The default choice.

Upper Confidence Bound (UCB): μ(x) + κσ(x). Simple, interpretable, κ controls exploration. Higher κ → more exploration.

Probability of Improvement (PI): What is the probability that this point beats the current best? Simple but too greedy in practice — prefers small certain improvements over risky large ones.

In practice: Use EI as your default. If you suspect the objective has multiple good regions, increase exploration with UCB. PI is rarely the best choice.

BayesOpt in Practice: Tuning XGBoost

import optuna
from sklearn.datasets import make_classification
from sklearn.model_selection import cross_val_score
from xgboost import XGBClassifier

# Create a dataset
X, y = make_classification(n_samples=1000, n_features=20, random_state=42)

def objective(trial):
    params = {
        'n_estimators': trial.suggest_int('n_estimators', 50, 500),
        'max_depth': trial.suggest_int('max_depth', 3, 10),
        'learning_rate': trial.suggest_float('learning_rate', 0.01, 0.3, log=True),
        'subsample': trial.suggest_float('subsample', 0.5, 1.0),
        'colsample_bytree': trial.suggest_float('colsample_bytree', 0.5, 1.0),
        'reg_alpha': trial.suggest_float('reg_alpha', 1e-8, 10.0, log=True),
        'reg_lambda': trial.suggest_float('reg_lambda', 1e-8, 10.0, log=True),
    }
    clf = XGBClassifier(**params, eval_metric='logloss', random_state=42)
    scores = cross_val_score(clf, X, y, cv=5, scoring='accuracy')
    return scores.mean()

# Optuna uses Tree-structured Parzen Estimators (TPE) by default
# For GP-based BayesOpt, use the GP sampler:
# sampler = optuna.samplers.GPSampler()
study = optuna.create_study(direction='maximize')
study.optimize(objective, n_trials=50, show_progress_bar=True)

print(f"Best accuracy: {study.best_value:.4f}")
print(f"Best params: {study.best_params}")
# 50 intelligent trials often beats 500 random trials

For most hyperparameter tuning tasks, Optuna with TPE is the practical default. It’s not technically a GP-based BayesOpt, but TPE is a competent surrogate that handles mixed (continuous + categorical) search spaces natively. Use BoTorch when you need custom acquisition functions or when the search space is purely continuous.

When to Use GPs

GPs shine when: small data (n < 1000), you need principled uncertainty, the input is low-dimensional (< ~20D), you don’t know the functional form.

Skip GPs when: large data (use a neural network), high-dimensional inputs (use deep learning or random forests), you need fast inference at serving time (GP prediction is O(n) per point).

BayesOpt shines when: each evaluation is expensive (minutes to hours), you have a moderate number of hyperparameters (< 20), you want sample efficiency over parallelism.
