Evaluating Generative Models

Generative model papers love "FID = 3.5." Number alone settles nothing. What does number mean? If two models make different images from same prompt, which is better? What does "better" mean with no single right answer? Evaluating generative models has been hard ML problem since image generation took off around 2014.

The Problem No One Warned You About

Classifier evaluation easy. Model predicts "cat." Label says "dog." Model wrong. Count right and wrong. Get accuracy. Straight path.

Generative models break this. Ask for "a watercolor painting of a lighthouse at sunset." Model makes image. Is it correct? Infinitely many valid watercolor lighthouses at sunset exist. No ground truth for comparison. Question not "is this right?" Question is "is this good?"

"Good" has at least three fighting dimensions. Quality: does each sample look crisp, coherent, realistic? Diversity: does model make wide variety, or same lighthouse again and again? Faithfulness: does image show what you asked for?

Tension lives here. Memorize ten beautiful lighthouse paintings, replay them: perfect quality, zero diversity. Output random noise: perfect diversity, zero quality. Make stunning photorealistic horse at dawn: high quality, strong diversity, zero faithfulness to lighthouse prompt. Any metric for "how good is this generator?" must balance all three forces. Hard problem.

A Tiny Art Gallery

Make this concrete. Build running example, return to it often. You run tiny art gallery commissioning AI-generated paintings. You have five real paintings — your "ground truth" for good art. You test three AI artists (generative models). Each makes five paintings.

Artist A makes five paintings. Stunning detail. Technical perfection. But all nearly identical — same landscape, same color palette, same composition. Visitors say "wow, beautiful" and "didn't I already see this one?"

Artist B makes five wildly different paintings — different styles, subjects, moods. But each messy: blurry edges, inconsistent lighting, visible artifacts. Visitors say "interesting variety" but buy none.

Artist C makes five paintings. Reasonably good-looking. Reasonably diverse. None beat Artist A's best. None match Artist B's range. Visitors find them... satisfying.

Which artist wins? Depends what you value. That is whole generative model evaluation problem in miniature. We return to these three artists for each metric. Different metrics crown different winners.

Comparing Distributions, Not Images

Key insight makes automatic evaluation possible: stop comparing individual images. Compare collections of images instead. More precisely, compare statistical distribution of generated images against statistical distribution of real images.

Raw pixel distributions useless. Two photos of same cat, shifted one pixel right, look statistically different in pixel space but perceptually same. Need feature space matching human perception — "this is face," "this has fur texture," not "pixel (127, 42) has value 0.73."

Pretrained neural network helps here. Take InceptionV3, trained to classify ImageNet into 1,000 categories. Along way, network learns rich hierarchical features — edges in early layers, textures in middle layers, objects and scenes in later layers. Remove final classification layer. What remains: 2,048-dimensional feature vector for each image — perceptual fingerprint capturing what image is about, not what pixels happen to be.

Now feed real images through Inception. Get cloud of 2,048-dimensional points. Feed generated images through. Get another cloud. Ask: how similar are these two clouds? Clear statistical question. FID answers it.

FID: The Math, the Intuition, the Flaws

First time I saw FID formula, I thought: too complicated. Then I saw what it does. Opposite feeling. Elegant compression of hard problem into something computable in few lines of code. Let me show shift.

We have two clouds of feature vectors — real and generated. FID makes one big simplifying assumption: each cloud roughly looks like multivariate Gaussian — bell curve in 2,048 dimensions. Gaussian is fully described by mean (cloud center) and covariance matrix (cloud shape and spread). So for each image set, compute:

μreal and Σreal — mean vector and covariance matrix for real image features.

μgen and Σgen — same for generated image features.

Fréchet distance — also called Wasserstein-2 distance for Gaussians — between these two distributions is:

FID = ||μ_real - μ_gen||² + Tr(Σ_real + Σ_gen - 2·(Σ_real · Σ_gen)^(1/2))

Two parts. Each tells different story. First part, ||μ_real - μ_gen||², is squared distance between cloud centers. If generated images are "about different things" than real images — faces training set, landscapes output — centers sit far apart. Term explodes. Think: "are you even in right ballpark?"

Second part uses covariance matrices. Harder to picture, but intuition helps. Imagine both clouds as 2D ellipses, not 2,048-dimensional blobs. Covariance matrix describes size, shape, orientation of each ellipse. If generated ellipse matches real one, term goes to zero. If generated ellipse too small (mode collapse — generator explores narrow region), too big (generator sprays features everywhere), or tilted wrong (generator learned wrong feature correlations), term grows.

Lower FID is better. FID = 0 means two Gaussians are identical — generated distribution is indistinguishable from real one in Inception feature space. State-of-the-art ImageNet 256×256 models now reach FID around 1.5–2.0. FID of 50 means visibly poor samples.

Back to art gallery. Artist A — beautiful but identical paintings — would have low mean distance because paintings look "real" in feature space, but large covariance term because generated cloud is tiny versus real one. All five feature vectors cluster tight. Artist B — diverse but messy — might have large mean distance because blurry artifact-ridden images land in different feature regions than clean ones. Artist C — balanced — likely gets best FID. Not perfect on either axis. Best compromise.

That is FID: compromise metric. It captures quality and diversity in one number. Big strength. Fundamental limit. It cannot tell which problem you have — "images are ugly" and "images not varied enough" collapse into same big number.

⚠️ FID's Practical Gotchas

Gaussian assumption matters. Inception features are not truly Gaussian, so FID is approximation of approximation. You need roughly 50,000 generated images for stable estimate — with 1,000 images, score can swing by ±10 between runs. Preprocessing pipeline matters more than expected: bilinear vs. bicubic resize before Inception can shift FID by several points. clean-fid exists to standardize all this. Use it.

Inception Score — The Metric That Came First and Aged Worst

Before FID came Inception Score (IS), introduced by Salimans et al. in 2016. Idea sounds clever first time. Then you think harder. It unravels.

Idea has two halves. First, feed each generated image through InceptionV3. It outputs probability distribution over 1,000 ImageNet classes. If image is clear and recognizable — sharp dog photo — classifier should be confident: p(y|x) should be peaked, with most mass on one class. Low entropy means image "looks like something." Quality signal.

Second, look at marginal distribution p(y) across all generated images — average of per-image predictions. If model generates diverse images, this marginal should spread across many classes. High entropy means variety. Diversity signal.

IS combines both with KL divergence:

IS = exp(E_x[KL(p(y|x) || p(y))])

Higher is better. Real ImageNet images score around 250. Good GAN circa 2018 scored 50–100.

Here system breaks. IS never compares against real images. It looks only at generated images in isolation. Model producing perfect, diverse ImageNet images with nothing to do with your training data would still score beautifully. Like restaurant critic rating food without checking what was ordered.

Worse: Inception classifier was trained on ImageNet's 1,000 specific classes. If you generate faces, medical scans, satellite imagery, or anything outside those categories, classifier has no idea what it sees. Asking it to judge face generator is like asking sommelier to judge cocktail competition.

Subtler problem too. If model generates 50,000 different-looking dogs — golden retrievers, poodles, huskies — and nothing else, IS may still look decent. Each dog image is confidently classified (quality ✓), and marginal may still spread across several dog-related classes (partial diversity ✓). Complete absence of cats, cars, everything else goes unpunished.

I still see IS in new papers sometimes. Historical convention keeps it alive. Fast to compute. Treat it as quick sanity check, never primary metric.

CLIP Score — When Faithfulness Matters

FID and IS both miss something critical for modern text-to-image models like Stable Diffusion and DALL·E: they do not measure whether generated image matches text prompt. You can make world's most photorealistic image and score well on FID. But if prompt says "a cat sitting on the moon" and output is horse on beach, model failed.

CLIP Score attacks this directly. CLIP (Contrastive Language-Image Pre-training, from OpenAI) was trained on hundreds of millions of text-image pairs. It maps text and images into shared embedding space where related concepts land near each other. To compute CLIP Score, encode prompt and generated image separately, then measure cosine similarity between vectors:

CLIP Score = cosine_similarity(CLIP_image(generated_image), CLIP_text(prompt))

Higher means better alignment. Above 0.30 usually means strong text-image correspondence. Below 0.20 means image probably ignores prompt.

Back to art gallery. You asked all three artists for "a stormy sea crashing against a red lighthouse." Artist A may make five identical but perfectly rendered lighthouses — high CLIP Score each time because images clearly match description. Artist B may make wildly varied scenes, but one is desert and another abstract splatter — low CLIP Scores for misses. Artist C may hit prompt reasonably well each time, with moderate CLIP Scores across board.

Catch: CLIP Score measures semantic alignment but is blind to image quality. Blurry low-resolution blob vaguely resembling lighthouse in storm can outscore photorealistic painting with lighthouse slightly pink instead of red. CLIP was trained on internet data with all its biases — some demographics, cultures, visual styles may score differently. Worst part: if generation model used CLIP-guided optimization during sampling, as many early diffusion models did, CLIP Score becomes circular. Model was trained to maximize same metric now used for evaluation. Like grading student with practice test.

LPIPS — Measuring What Your Eyes Measure

Sometimes distribution comparison is not what you need. Simpler question: how different do these two specific images look?

This shows up constantly in image-to-image tasks: super-resolution (does upscaled image look like original?), inpainting (does filled region match surrounding context?), style transfer (is content preserved?). Traditional pixel metrics like PSNR and SSIM are bad at perceptual similarity. Two images can share identical PSNR but look wildly different to human because PSNR treats every pixel error equally — critical facial feature and irrelevant background texture count same.

LPIPS (Learned Perceptual Image Patch Similarity), introduced by Zhang et al. in 2018, uses same core insight as FID: deep network features capture what humans perceive. Feed both images through pretrained network (VGG, AlexNet), extract feature maps at several layers, compute channel-wise L2 distance at each layer, weight distances with learned coefficients, sum everything. Weights were calibrated against human perceptual judgments — real people saying "these two images look same" or "these look different."

Lower LPIPS means more perceptually similar. Paper title — "The Unreasonable Effectiveness of Deep Features as a Perceptual Metric" — sounds bold, but result holds up. LPIPS matches human similarity judgments far better than PSNR or SSIM. Default perceptual distance metric for paired image comparisons.

Limit is "paired." LPIPS compares two specific images — not distributions. You cannot use it for unconditional image generator producing images from scratch with no reference. For that, you still need FID or cousins.

Precision and Recall for Generators

FID gives one number. Convenient. Frustrating. When FID rises, did quality drop or diversity collapse? You cannot tell. In 2019, Kynkäänniemi et al. proposed fix: split evaluation into two numbers, borrowing precision and recall from classification but giving them new meanings for generative models.

Ideas map cleanly onto art gallery. Precision asks: of generated paintings, what fraction look like plausible real paintings? Quality measurement. If Artist A makes five paintings and all five could fool gallery visitor, precision is 1.0. If Artist B makes five and only two look real, precision is 0.4.

Recall asks opposite: of real paintings in gallery collection, what fraction are "covered" by generated set? Diversity measurement. If gallery has landscapes, portraits, abstracts, and generator makes only landscapes, recall is low — whole categories missing. Artist A, with five identical landscapes, has low recall. Artist B, despite messiness, may cover more ground.

In practice, both are computed in Inception feature space with nearest-neighbor distances. Technical details build manifold estimate around each point set and measure overlap, but intuition matters most: precision catches quality failures, recall catches diversity failures. FID mixes both into one number; precision and recall separate them.

Naeem et al. refined this in 2020 with density and coverage as sturdier alternatives. Density measures how many generated samples cluster near each real sample — smoother precision. Coverage measures fraction of real samples with at least one generated neighbor — more stable recall. Less sensitive to outliers. More reliable with small sample sizes.

Precision/recall framing is useful because it forces you to name which failure mode worries you. When stakeholder says "model got worse," precision/recall lets you answer "quality is fine, diversity dropped" — much more actionable than "FID went up by 3 points."

The Likelihood Trap

Tempting alternative exists. If generative model assigns probabilities to data — as VAEs, normalizing flows, and autoregressive models do — why not evaluate by likelihood on real test data? Model assigning higher probability to real images should be better. Right?

Sounds logical. Mostly wrong. Theis, van den Oord, and Bethge showed this in influential 2016 paper, "A Note on the Evaluation of Generative Models." Result still gives newcomers headaches.

Core problem: high likelihood does not imply good samples, and good samples do not imply high likelihood. These properties are independent. Model can assign high likelihood to real test images yet generate terrible samples. How? By smearing probability mass broadly across image space. It covers everything, gives real images reasonable probability, but samples come out blurry nonsense because mass is spread too thin.

Reverse case also happens. Model memorizing 100 training images and placing all probability on exact copies of those images will generate stunning samples — literal photographs — but assign near-zero likelihood to any test image outside memorized set.

Standard image likelihood metric is bits per dimension: negative log-likelihood in base 2, divided by pixel count. Lower is better. 3.5 bits per dimension on CIFAR-10 is decent; 3.0 is strong. But model with 3.0 can make worse-looking samples than model with 3.5. Feels wrong every time.

Information-theoretic explanation involves typical set — narrow band of probability where real data overwhelmingly lives in high-dimensional spaces. Practical lesson simple: never use likelihood alone for evaluation. It measures something real. Not thing you care about when sample quality matters.

Mode Collapse — How to Catch a Generator Cheating

Mode collapse happens when generator decides variety is for other models. Instead of learning full training-data diversity, it grabs few outputs that satisfy training objective and repeats them again and again.

In GAN training, this is famous failure mode: generator finds few outputs that fool discriminator and replays them. In diffusion models, rarer but possible — can appear when model is undertrained, dataset has severe class imbalance, or guidance scale is too high.

Symptoms are easy if you know where to look. Generate grid of 64 images from different random seeds. If all look suspiciously similar — same pose, composition, color palette — you have mode collapse. Crude method. Works often. Visual inspection at fixed training intervals catches most mode collapse events in practice.

For quantitative diagnosis, recall and coverage from precision/recall framework help most. Mode-collapsed generator can have decent precision — few generated things look real — but terrible recall because it covers tiny slice of real data variety. FID will rise too, but as discussed, it will not tell you why. Recall will.

Another method: compute pairwise distances between generated samples in feature space. If distance distribution peaks tightly near zero, samples are suspiciously similar. You can also run pretrained classifier on generated images and inspect class distribution — if 90% of outputs are "golden retriever," something broke.

For GANs specifically, latent interpolation reveals much. Take two random latent codes z₁ and z₂. Generate images at evenly spaced points between them. If model learned smooth diverse manifold, interpolated images should change gradually. If model collapsed, interpolation gets jerky — abrupt jumps between few known modes, nothing between.

The Diversity–Quality Tradeoff

Here is surprising part: in every deployed generative system I know, quality knob and diversity knob are same knob. Turn one way: quality rises, diversity falls. Turn other way: opposite. Every shipped model picks some compromise setting.

In GANs, this knob is truncation trick. StyleGAN samples latent codes from normal distribution. Truncation parameter ψ scales codes toward mean: z_truncated = ψ · z. At ψ = 1.0, you get full untruncated distribution — maximum diversity but occasional ugly tail samples. At ψ = 0.5, images stay attractive but variety drops hard. At ψ = 0.0, every image is identical — average face. Picture normal distribution as landscape. Truncation builds fence closer and closer to center, keeping only safe territory.

In diffusion models, same knob is classifier-free guidance (CFG). During sampling, compute conditional prediction — what matches prompt — and unconditional prediction, then amplify difference:

noise_pred = unconditional + guidance_scale × (conditional - unconditional)

At guidance_scale = 1.0, you get raw model — diverse but sometimes ignores prompt. At 7.5 (Stable Diffusion default), images are sharp and prompt-faithful but less varied. At 20+, outputs become oversaturated and repetitive — model pushes every image toward most stereotypical interpretation of prompt.

This matters directly for evaluation. Plot FID against guidance scale and you get U-shaped curve: too low, images are poor quality (high FID); too high, diversity collapses (also high FID). Sweet spot sits in middle where FID is minimized. So always report guidance scale — or truncation ψ — alongside FID. Model bragging about FID = 2.0 at guidance_scale = 30 is hiding diversity loss behind cherry-picked settings. Not better model. More aggressively truncated model.

Precision and recall make this tradeoff visible in way FID alone cannot. As guidance scale rises, precision goes up — each image looks better — and recall goes down — generator covers less of real distribution. Watching both numbers move while you turn knob says far more than watching one FID number bounce around.

Asking Humans (And Doing It Right)

Every automatic metric here is proxy for real question: would human look at output and think it is good? FID cannot tell you model makes hands with six fingers. CLIP Score cannot tell you colors feel muddy. Precision/recall cannot tell you image is technically correct but aesthetically lifeless. Anything headed to production needs humans in loop.

Three main protocols exist. Each has different strength.

Mean Opinion Score (MOS) is simplest: show evaluator one image, ask for quality rating from 1 to 5. Problem is calibration — people use wildly different standards. Your 4 may be my 2. MOS works for rough absolute quality measurement but needs many ratings to average out noise.

Two-Alternative Forced Choice (2AFC) is what most serious evaluations use. Show two images side by side — one from model A, one from model B — and ask: "which is better?" Humans are much better at relative comparisons than absolute ratings. No need to agree on what "4" means; only which image is preferred. Randomize left/right positions, blind evaluators to model identity, get surprisingly consistent results.

Elo ratings extend 2AFC to many models. Run many pairwise comparisons, then use Elo algorithm — same one chess uses — to compute rating for each model. Every comparison updates ratings: beat higher-rated opponent, gain more points. After few hundred comparisons per model, rankings stabilize. Chatbot Arena uses this for LLM evaluation. Image generation leaderboards increasingly do same.

Practical guideline: 200 to 500 comparisons with 3 raters each is usually enough to detect statistically significant differences between two models. Always measure inter-rater agreement — Fleiss' kappa or Krippendorff's alpha — to find and remove unreliable annotators. Design prompt set carefully. Do not evaluate on random prompts. Target known failure modes: complex scenes, text rendering, human hands, unusual objects. Model doing well on "a photo of a dog" but failing on "a left hand holding a clock that reads 3:15" has not been evaluated hard enough.

Goodhart's Law, or: When the Metric Becomes the Enemy

There's a principle from economics that haunts evaluation in every field of machine learning, but nowhere more acutely than in generative modeling. Goodhart's Law states: "When a measure becomes a target, it ceases to be a good measure."

This is not abstract philosophy. It's a concrete, practical problem. The moment the field collectively decided that FID was the number to minimize, models started getting optimized — directly or indirectly — to produce images that match Inception feature statistics, whether or not those images actually look good to humans.

Here are some of the ways FID can be gamed, and I've seen each of these in practice. You can tune your guidance scale or truncation to hit exactly the FID sweet spot, hiding diversity problems behind a good headline number. You can adjust your image preprocessing pipeline to happen to match the reference set's pipeline, getting several free FID points from an implementation detail rather than a model improvement. You can generate images at a resolution that Inception handles well, rather than the resolution your model actually targets. You can report FID on a convenient subset of the test set rather than the full distribution.

CLIP Score is even more vulnerable. If your generation pipeline includes CLIP-guided optimization — which many early text-to-image systems used — then optimizing for CLIP Score is circular. The model was trained to maximize the very metric you're now reporting as evidence of quality. It's like measuring a student's learning with the exact practice problems they've been drilling.

The defense against Goodhart's Law is not a better metric. It's a collection of metrics, measured by different underlying models, supplemented by human evaluation, and reported with full transparency about methodology. Report FID and CLIP Score and precision/recall and the guidance scale and the sample size and the preprocessing pipeline. A model that improves on all fronts is genuinely better. A model that improves on one while the others regress is probably gaming the system.

I find this the most important lesson in the entire section: the metrics are tools, not goals. The goal is to build a model that produces outputs humans actually want. Any single number that claims to capture that is either lying to you or lying to itself.
