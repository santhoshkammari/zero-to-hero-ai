Federated Learning

Chapter 13: ML Systems & Production 6 subtopics

TL;DR

Federated learning trains ML models across multiple devices or organizations without sharing raw data. Each participant trains locally on their own data, then shares only model updates (gradients or weights) with a central server that aggregates them into a global model. This preserves data privacy, satisfies regulatory constraints (GDPR, HIPAA), and enables collaboration between entities that can't or won't share data. The tradeoffs: communication overhead, heterogeneous data challenges, and new attack surfaces that require careful security design.

Why Federated Learning Matters

Imagine you're building a medical AI to detect tumors from MRI scans. Hospital A has 10,000 scans, Hospital B has 8,000, Hospital C has 15,000. Combining all 33,000 scans would give you the best model. But hospitals can't share patient data — HIPAA forbids it, patients didn't consent to it, and the legal liability is enormous. What do you do?

Federated learning answers: don't move the data. Move the model. Send a copy of the model to each hospital, let each hospital train on their local data, send back only the model updates, and combine those updates into a better global model. The raw MRI scans never leave each hospital. The model gets the benefit of all 33,000 training examples without any single party seeing data they shouldn't.

This isn't just a healthcare story. Google uses federated learning to improve Gboard predictions without uploading what you type. Apple uses it for Siri improvements. Banks use it for fraud detection across institutions. Any scenario where data is siloed — by regulation, competition, or logistics — is a candidate for federated learning.

Core Algorithms

Federated Averaging (FedAvg)

Federated Averaging (FedAvg) is the foundational algorithm, introduced by McMahan et al. at Google in 2017. The protocol is elegant in its simplicity:

import numpy as np
from copy import deepcopy

def federated_averaging(global_model, clients, rounds=100,
                        local_epochs=5, client_fraction=0.1):
    """Standard FedAvg implementation."""
    for round_num in range(rounds):
        # 1. Server selects a random subset of clients
        selected = np.random.choice(
            clients, size=max(1, int(len(clients) * client_fraction)),
            replace=False
        )

        # 2. Send global model to selected clients
        client_updates = []
        client_sizes = []
        for client in selected:
            local_model = deepcopy(global_model)
            # 3. Each client trains locally on their data
            local_model = client.train(local_model, epochs=local_epochs)
            client_updates.append(local_model.state_dict())
            client_sizes.append(len(client.dataset))

        # 4. Server aggregates updates (weighted by dataset size)
        total_size = sum(client_sizes)
        global_state = global_model.state_dict()
        for key in global_state:
            global_state[key] = sum(
                client_updates[i][key] * (client_sizes[i] / total_size)
                for i in range(len(selected))
            )
        global_model.load_state_dict(global_state)

    return global_model

FedAvg works well when data is relatively uniform across clients. But real-world data is messy, which brings us to the harder problem.

FedProx: Handling Heterogeneity

FedProx adds a proximal term to each client's local objective — a penalty for deviating too far from the global model. This prevents any single client's quirky data from pulling the global model off course. The proximal term is controlled by a parameter μ: higher μ means clients stay closer to the global model, at the cost of not fully exploiting their local data.

def fedprox_local_loss(model, global_model, batch, mu=0.01):
    """FedProx loss: task loss + proximal penalty."""
    predictions = model(batch['features'])
    task_loss = F.cross_entropy(predictions, batch['labels'])

    # Proximal term: penalize divergence from global model
    proximal_penalty = 0.0
    for local_param, global_param in zip(
        model.parameters(), global_model.parameters()
    ):
        proximal_penalty += ((local_param - global_param) ** 2).sum()

    return task_loss + (mu / 2) * proximal_penalty

SCAFFOLD: Variance Reduction

SCAFFOLD tackles client drift more directly. Each client maintains control variates that track the difference between its local gradient direction and the global gradient direction. During local training, these control variates correct for the bias introduced by non-identical data. SCAFFOLD converges faster than FedAvg on heterogeneous data, especially when clients can only do few local steps per round.

FedMA: Matched Averaging

FedMA (Federated Matched Averaging) addresses a subtle problem: when clients have different model architectures (or the same architecture but with neurons that have evolved to serve different roles), naive averaging destroys information. FedMA aligns neurons across client models by their learned function before averaging, using techniques from optimal transport. This is particularly relevant when clients start from different initializations.

Data Heterogeneity

The Non-IID Challenge

The biggest practical challenge in federated learning is non-IID (non-independently and identically distributed) data. In the real world, data across clients is rarely uniform. Hospital A specializes in brain tumors, Hospital B in lung cancer. User A types in English, User B in Japanese. Each client's local data represents a biased slice of the true distribution.

Non-IID data causes two problems. First, local models diverge — each client's model specializes on its local data and moves away from the others. Second, averaging divergent models can produce a global model that's worse than any individual local model. This is called client drift.

Personalized Federated Learning

Sometimes you don't want a single global model — you want personalized models that perform well on each client's specific data. Personalized federated learning trains a shared global model but allows each client to adapt it locally. Per-FedAvg uses meta-learning (MAML-style) to learn an initialization that's easy to personalize. Local fine-tuning takes the global model and fine-tunes it on each client's data for a few steps. FedBN keeps batch normalization layers local (since they capture data statistics) while sharing all other layers.

Multi-Task Federated Learning

When clients have fundamentally different tasks (not just different data distributions), multi-task federated learning learns related but distinct models. The shared layers capture common knowledge, while task-specific heads capture each client's unique objective. Think of hospitals that all want medical imaging models but for different diagnostic tasks — they share low-level feature extractors while maintaining separate classifiers.

Communication Efficiency

The Bandwidth Problem

Sending full model updates over the network every round is expensive. A 100M parameter model requires ~400 MB per update in FP32. With 1,000 clients and 100 rounds, that's 40 TB of network traffic. On mobile devices, bandwidth is limited and metered. Communication efficiency isn't a nice-to-have — it's a hard requirement.

Gradient Compression and Sparsification

Gradient compression reduces the size of updates. Techniques include: Top-K sparsification (send only the K largest gradient values), random sparsification (randomly sample a subset of gradients), and quantized gradients (send 1-bit or 2-bit gradients instead of 32-bit). Error feedback mechanisms accumulate the unsent gradients and add them to the next round, ensuring nothing is permanently lost.

Federated Distillation

Federated distillation replaces model weight sharing with knowledge sharing. Instead of sending model parameters, clients send predictions on a shared public dataset (or class-level statistics). Other clients learn from these predictions via knowledge distillation. This dramatically reduces communication — predictions are much smaller than model weights — and enables heterogeneous model architectures across clients.

Asynchronous Aggregation and Straggler Mitigation

Standard FedAvg waits for all selected clients to finish before aggregating — the speed of the slowest client determines the round time. Asynchronous aggregation updates the global model as each client finishes, without waiting for others. This improves throughput but introduces staleness — fast clients contribute to a newer global model than slow clients. Careful weighting of stale updates is needed to maintain convergence.

Privacy and Security

Differential Privacy in Federated Learning

Federated learning doesn't automatically guarantee privacy. Model updates can leak information about training data. For example, gradient inversion attacks can reconstruct training images from gradients. Differential privacy (DP) adds calibrated noise to model updates before sharing them, providing a mathematical guarantee about information leakage.

def apply_dp_to_update(model_update, clip_norm=1.0, noise_scale=0.1):
    """Apply differential privacy to a model update."""
    # 1. Clip the update norm (bound sensitivity)
    total_norm = torch.sqrt(
        sum(p.norm() ** 2 for p in model_update.values())
    )
    clip_factor = min(1.0, clip_norm / (total_norm + 1e-6))
    clipped = {k: v * clip_factor for k, v in model_update.items()}

    # 2. Add Gaussian noise (proportional to sensitivity)
    noised = {}
    for k, v in clipped.items():
        noise = torch.randn_like(v) * noise_scale * clip_norm
        noised[k] = v + noise

    return noised
Privacy budget is finite: Each round of differentially private federated learning consumes some privacy budget (ε). More rounds = more leakage. Eventually, you've leaked as much information as if you'd shared the raw data. Track your cumulative ε and set a hard budget. When it's spent, stop training.

Secure Aggregation

Secure aggregation ensures the server can compute the aggregate of client updates without seeing any individual update. Clients encrypt their updates using secret sharing — each client splits their update into random shares sent to other clients, such that only the sum is recoverable. The server sees only the aggregate. Even a malicious server can't reconstruct individual client contributions.

Model Poisoning and Data Poisoning

Model poisoning attacks send malicious updates designed to corrupt the global model. A compromised client might send gradients that cause the global model to misclassify specific inputs (a backdoor attack). Data poisoning corrupts the client's local training data to indirectly poison the updates. Defenses include Byzantine-robust aggregation methods like Krum, trimmed mean, and coordinate-wise median, which are designed to be resilient even when a fraction of clients are adversarial.

FL Variants

Cross-Device vs. Cross-Silo

Cross-device FL involves millions of edge devices (phones, IoT sensors). Challenges: devices are unreliable, have limited compute, and are available only intermittently. Google's Gboard is the canonical example. Cross-silo FL involves a small number of organizations (hospitals, banks). Challenges are different: data heterogeneity is more extreme, but communication is reliable and compute is abundant.

Horizontal, Vertical, and Transfer FL

Horizontal FL is the standard case: clients have the same features but different samples (different hospitals, same scan types). Vertical FL is when clients have different features for the same samples — a bank and an e-commerce company have different data about the same customers. This requires split learning architectures where each party processes their features and shares intermediate activations. Federated transfer learning handles the case where clients differ in both features and samples.

Deployment and Evaluation

FL Frameworks

Flower (flwr) is the most popular open-source FL framework. It's framework-agnostic (works with PyTorch, TensorFlow, JAX) and emphasizes simplicity — you can federate an existing training script with minimal code changes.

import flwr as fl
import torch

class MyClient(fl.client.NumPyClient):
    def __init__(self, model, trainloader, valloader):
        self.model = model
        self.trainloader = trainloader
        self.valloader = valloader

    def get_parameters(self, config):
        return [val.cpu().numpy() for val in self.model.parameters()]

    def set_parameters(self, parameters):
        for param, new_val in zip(self.model.parameters(), parameters):
            param.data = torch.tensor(new_val)

    def fit(self, parameters, config):
        self.set_parameters(parameters)
        train_one_epoch(self.model, self.trainloader)
        return self.get_parameters(config), len(self.trainloader.dataset), {}

    def evaluate(self, parameters, config):
        self.set_parameters(parameters)
        loss, accuracy = evaluate(self.model, self.valloader)
        return float(loss), len(self.valloader.dataset), {"accuracy": accuracy}

# Start federated training
fl.client.start_numpy_client(server_address="[::]:8080", client=MyClient(...))

PySyft (by OpenMined) focuses on privacy-preserving computation, integrating differential privacy and secure multi-party computation. TensorFlow Federated (TFF) provides a TensorFlow-native FL experience with strong simulation tools for research.

Evaluation Beyond Accuracy

Evaluating federated models requires thinking beyond global accuracy. Measure: per-client accuracy (is the model fair across participants?), communication cost (total bytes transferred), convergence speed (rounds to target accuracy), privacy budget consumed (cumulative ε), and robustness to client dropout (what happens when 30% of clients disappear?). A model that's 1% less accurate but uses 10x less communication might be the better choice in practice.

When FL is overkill: If you can anonymize data and centralize it, do that instead. Federated learning adds complexity — debugging is harder, convergence is slower, and the engineering overhead is significant. FL is the right choice when data truly cannot move: legal restrictions (GDPR, HIPAA), competitive concerns (banks sharing models but not data), or physical constraints (edge devices with limited connectivity). Don't use FL for the aesthetics — use it when you have no alternative.
