Benchmarking

Chapter 13: ML Systems & Production 2 subtopics
TL;DR

Benchmarks are the yardsticks of ML — they tell us whether models are improving and by how much. But they are also deeply flawed. This section covers how to design honest benchmarks, the standard benchmarks every practitioner should know (ImageNet, GLUE, MMLU, HumanEval), the corrosive effects of leaderboard culture, and why benchmark contamination is the elephant in the room for modern LLMs.

Benchmark Design

What Makes a Good Benchmark

A benchmark is a standardized dataset and evaluation protocol that lets you compare different models on the same task under the same conditions. It sounds simple, but designing a good benchmark is surprisingly hard.

A good benchmark has several properties. Validity — it actually measures what you care about. A reading comprehension benchmark where most questions can be answered by pattern matching (without understanding the passage) is not measuring comprehension. Difficulty — it should not be trivially solvable. If every model scores above 95%, the benchmark has stopped being useful. Diversity — it should cover the range of difficulty and variety you expect in practice. A math benchmark that only contains arithmetic problems tells you nothing about a model's ability to do algebra. Clean labels — if the ground-truth labels contain 10% errors, you cannot distinguish a 90% model from a 92% model.

System Benchmarks

Beyond model quality, you also need to benchmark the system — how fast and efficiently models run. Latency profiling measures how long a single prediction takes (p50, p95, p99 — the median, 95th percentile, and 99th percentile response times). Throughput measurement counts how many predictions per second the system handles under load. Memory footprint matters for edge deployment — can the model fit on a phone?

MLPerf is the industry standard for system benchmarks. It defines standardized workloads (image classification, object detection, language modeling, recommendation) and measures training time and inference throughput on different hardware. MLPerf results let you compare whether NVIDIA's latest GPU is actually faster than the previous generation for your workload — not just in theory, but on a real training run.

# Latency profiling for a model server
import time
import numpy as np
import requests

def benchmark_latency(url, payload, num_requests=1000):
    """Measure serving latency distribution."""
    latencies = []
    for _ in range(num_requests):
        start = time.perf_counter()
        response = requests.post(url, json=payload)
        elapsed = (time.perf_counter() - start) * 1000  # ms
        latencies.append(elapsed)

    latencies = np.array(latencies)
    print(f"Latency (ms):")
    print(f"  p50: {np.percentile(latencies, 50):.1f}")
    print(f"  p95: {np.percentile(latencies, 95):.1f}")
    print(f"  p99: {np.percentile(latencies, 99):.1f}")
    print(f"  max: {np.max(latencies):.1f}")
    print(f"  throughput: {num_requests / (sum(latencies) / 1000):.0f} req/s")

benchmark_latency("http://model-server:8080/predict", {"text": "sample input"})

Standard Benchmarks You Should Know

Vision: ImageNet and Beyond

ImageNet is the benchmark that launched the deep learning revolution. It contains 1.2 million training images in 1,000 categories. When AlexNet dramatically beat traditional methods on ImageNet in 2012, it convinced the world that deep neural networks were real. Today, top-1 accuracy on ImageNet has climbed above 90%, and the benchmark is often considered "saturated" — improvements are incremental and may not reflect real-world gains.

Newer vision benchmarks include COCO (object detection and segmentation), ADE20K (scene parsing), and robustness benchmarks like ImageNet-C (corrupted images) and ImageNet-R (renditions and sketches that test out-of-distribution generalization).

NLP: GLUE, SuperGLUE, and the LLM Era

GLUE (General Language Understanding Evaluation) was introduced in 2018 as a collection of nine NLP tasks — sentiment analysis, natural language inference, paraphrase detection, and more. Models submit predictions and get a single aggregate score. GLUE was so quickly saturated (models surpassed human performance within a year) that SuperGLUE was created with harder tasks. It too was quickly saturated.

For large language models, the benchmarks are different. MMLU (Massive Multitask Language Understanding) tests knowledge across 57 subjects — everything from abstract algebra to world religions. HumanEval tests code generation — can the model write correct Python functions from docstrings? HellaSwag tests commonsense reasoning through sentence completion. GSM8K tests grade-school math word problems. TruthfulQA tests whether models generate truthful answers to questions where common misconceptions exist.

Leaderboard Culture and Its Problems

Goodhart's Law in ML

Goodhart's Law states: "When a measure becomes a target, it ceases to be a good measure." This is exactly what happens with ML benchmarks. Teams optimize specifically for benchmark performance — tuning hyperparameters on the test set (even implicitly, through repeated evaluation), engineering features that exploit benchmark quirks, and selecting model architectures that perform well on the benchmark even if they are impractical for real-world use.

The result: benchmark scores go up, but real-world usefulness does not follow proportionally. A model that scores 2 points higher on MMLU might not be noticeably better at answering your users' questions.

Benchmark Contamination

Benchmark contamination is the most serious problem in modern ML evaluation. It happens when a model's training data contains examples from the benchmark's test set. Since LLMs are trained on massive internet crawls, and benchmark datasets are often publicly available online, contamination is nearly inevitable. If the model has memorized the answers, its benchmark score is inflated — it is not demonstrating understanding, just recall.

Detecting contamination is hard. Simple n-gram overlap checks miss paraphrased versions of test examples. And even partial contamination (seeing similar but not identical examples) can inflate scores. The community has responded with dynamic benchmarks that generate new questions regularly (like LMSYS Chatbot Arena, which uses human preference votes in real-time), private test sets, and contamination detection tools. But the fundamental tension remains: public benchmarks are useful because anyone can reproduce results, but being public makes them vulnerable to contamination.

Beyond Leaderboards

Benchmarks are useful starting points, not endpoints. When evaluating a model for your use case, standard benchmarks tell you whether the model is in the right ballpark. But to know if it will work for your users with your data, you need custom evaluation. Build an evaluation set from your actual data. Annotate it carefully. Test the specific capabilities that matter for your application — and test the failure modes that would be most damaging. A model that is mediocre on MMLU but excellent at answering customer support questions in your domain is the right model for your customer support chatbot. Do not let leaderboard rankings override domain-specific evaluation.
