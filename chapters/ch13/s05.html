MLOps

Chapter 13: ML Systems & Production Section 5 of 9
TL;DR

MLOps is not "DevOps for ML" — it's DevOps plus data versioning, experiment tracking, continuous training, model registries, and pipeline orchestration, all stitched together because ML systems have three moving axes (code, data, model) instead of one. This section covers the full stack: CI/CD that actually works for ML, experiment tracking that saves your sanity, model registries for governance, orchestrators for pipeline automation, reproducibility practices, feature stores in the MLOps context, metadata management, and Google's maturity framework so you know where you stand.

What MLOps Actually Is

Not Just "DevOps for ML"

Every conference talk starts with "MLOps is DevOps for ML." That's like saying surgery is "carpentry for humans" — technically has some overlap, completely misses the point.

DevOps solves one problem: getting code from a developer's laptop to production reliably. MLOps has to handle three entangled artifacts — code, data, and models — each with its own versioning, testing, and deployment lifecycle. Change the code, you need to retest. Change the data, you might need to retrain. Change the model, you need to re-validate. And they interact: a code fix might change model behavior, a data shift might invalidate your code's assumptions.

Here's the core tension that makes MLOps its own discipline:

DimensionSoftware EngineeringML Engineering
What changesCodeCode + Data + Model
TestingDeterministic (pass/fail)Statistical (thresholds, distributions)
Build outputBinary / containerModel artifact + serving config
DegradationCrashes or wrong outputSilently gets worse over time
RollbackDeploy previous versionWhich version? Code, data, or model?
Root causeUsually in the code diffOften in data you never committed

The silent degradation row is the killer. A web app either works or it doesn't. A model can quietly go from 92% accuracy to 78% accuracy and nobody notices for weeks — because it still returns valid JSON, still responds in 50ms, still passes your health checks. The requests that used to get classified correctly are now getting classified wrong, but nothing broke.

The Three Axes Problem

In software, you version one thing: code. In ML, you version three:

# The three axes of ML versioning — all three must be tracked together
experiment:
  code_version: "git:abc123f"         # Which training code
  data_version: "dvc:v2.3.1"         # Which dataset snapshot
  model_version: "registry:fraud-v7"  # Which trained artifact
  config:                              # Which hyperparameters
    learning_rate: 0.001
    batch_size: 256
    epochs: 50

# Change ANY of these → different model behavior
# Miss tracking ANY of these → irreproducible results

If you can't reconstruct exactly what produced the model currently in production — the code commit, the data snapshot, the config, the random seed — you don't have MLOps. You have a model and a prayer.

CI/CD for ML

Three Pipelines, Not One

Software CI/CD has one pipeline: test code → build → deploy. ML needs three distinct pipelines running in coordination.

Code CI — the familiar one. Linting, unit tests, integration tests. Runs on every commit. Fast, deterministic. Nothing special here.

Data CI — the one most teams skip. Validates incoming data against expected schemas, distributions, and freshness constraints. A schema change in an upstream table has silently broken more production models than any code bug.

Model CI — the unique one. Trains a candidate model, evaluates it against quality gates, compares it to the current production model. This pipeline can take hours (GPU training) and its tests are statistical, not deterministic.

# .github/workflows/ml_cicd.yml — Three pipelines in one workflow
name: ML CI/CD

on:
  push:
    branches: [main]
  schedule:
    - cron: '0 6 * * 1'  # Weekly continuous training

jobs:
  # === PIPELINE 1: Code CI (minutes) ===
  code-ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: |
          pip install -r requirements.txt
          ruff check src/
          pytest tests/unit/ -x --tb=short
          pytest tests/integration/ -x --tb=short

  # === PIPELINE 2: Data CI (minutes) ===
  data-ci:
    runs-on: ubuntu-latest
    needs: code-ci
    steps:
      - name: Validate data schema and distributions
        run: |
          python -m data_checks.validate \
            --source s3://data-lake/features/latest/ \
            --schema schemas/features_v3.json \
            --check-nulls --check-ranges --check-distributions \
            --reference s3://data-lake/features/baseline/
          # Fails if: new columns, type changes, null rate spikes,
          # distribution drift beyond threshold

  # === PIPELINE 3: Model CI (hours) ===
  model-ci:
    runs-on: [self-hosted, gpu]
    needs: data-ci
    steps:
      - name: Train candidate model
        run: |
          python train.py \
            --config configs/prod.yaml \
            --data-version $(dvc version --short)

      - name: Quality gates
        run: |
          python evaluate.py \
            --candidate outputs/model_latest/ \
            --champion models:/fraud-detector/Production \
            --min-f1 0.82 \
            --max-latency-p99-ms 100 \
            --max-model-size-mb 500 \
            --check-slices region,device_type,user_tenure \
            --no-regression  # Must beat current production model

      - name: Register if all gates pass
        if: success()
        run: |
          python register.py \
            --model outputs/model_latest/ \
            --stage staging

Continuous Training

Continuous Training (CT) is the ML-specific addition to CI/CD. When new data arrives — or when monitoring detects drift — the pipeline automatically retrains, evaluates, and promotes. Humans only get paged when something fails.

The trigger matters. Don't just retrain on a schedule because someone told you to. Retrain when:

  • Data drift is detected (feature distributions shifted beyond thresholds)
  • Performance degrades (monitoring catches accuracy drops on live traffic)
  • New labeled data reaches a meaningful batch size
  • Upstream schema changes require feature pipeline updates

Calendar-based retraining ("every Monday") is fine as a starting point. But it's wasteful when data is stable and dangerous when data shifts mid-week.

Model Quality Gates

Quality gates are automated checks a model must pass before promotion. The non-obvious ones matter more than the obvious ones:

GateWhat It CatchesWhy It's Not Obvious
Overall accuracy ≥ thresholdBroken trainingEveryone does this. Necessary but insufficient.
No slice regressionSubgroup performance dropsOverall accuracy can improve while a critical segment tanks.
Champion vs. challengerRegression vs. production modelA model can pass all absolute thresholds but still be worse than what's live.
Latency p99 ≤ SLAModels too slow to serveAccuracy-latency tradeoff — the best model is useless if it misses SLA.
Model size ≤ limitDeployment constraintsEdge devices, memory budgets, cold start times.
Prediction distribution checkCalibration issuesA model that predicts 0.99 for everything passes accuracy if the class is imbalanced.

Experiment Tracking

What to Actually Log

You've run 200 experiments. You need to find the one from three weeks ago that had surprisingly good recall on international transactions. If you didn't log the right things, you're scrolling through notebooks.

Log everything in this hierarchy:

# The experiment tracking contract — log ALL of these
run_metadata = {
    # Identity
    "run_id": "auto-generated",
    "experiment_name": "fraud-detection-v3",
    "run_name": "xgb-depth8-lr001",  # Human-readable

    # Inputs (what went in)
    "params": {
        "model_type": "xgboost",
        "max_depth": 8,
        "learning_rate": 0.01,
        "n_estimators": 500,
        "feature_set": "v3_with_velocity",
        "data_version": "dvc:abc123",
        "train_size": 1_200_000,
    },

    # Outputs (what came out)
    "metrics": {
        "f1": 0.847,
        "precision": 0.812,
        "recall": 0.884,
        "auc_roc": 0.923,
        "latency_p50_ms": 12,
        "latency_p99_ms": 45,
    },

    # Slice metrics (the ones that actually matter)
    "slice_metrics": {
        "domestic":      {"f1": 0.871, "n": 1_000_000},
        "international": {"f1": 0.764, "n": 200_000},
        "high_value":    {"f1": 0.832, "n": 50_000},
    },

    # Environment (for reproducibility)
    "env": {
        "git_commit": "abc123f",
        "python_version": "3.11.5",
        "gpu": "A100-40GB",
        "seed": 42,
    },

    # Artifacts (the heavy stuff)
    "artifacts": ["model.pkl", "feature_importance.png",
                  "confusion_matrix.png", "calibration_curve.png"],
}

The slice metrics are not optional. A model that looks great on aggregate but fails on 15% of your users is a model you'll be debugging in production.

Model Registry

Versioning, Staging, Promotion, Rollback

A model registry is Git for models — but with lifecycle stages that code doesn't need. Every model version moves through a defined promotion path:

  ┌──────────┐     ┌──────────┐     ┌────────────┐     ┌──────────┐
  │   None    │ ──▶ │ Staging  │ ──▶ │ Production │ ──▶ │ Archived │
  │ (logged)  │     │ (testing)│     │  (serving) │     │ (retired)│
  └──────────┘     └──────────┘     └────────────┘     └──────────┘
        │                │                  │
        │           Integration         Canary/shadow
        │            tests               evaluation
     Auto-logged     + load test       + business metrics
     from training   + A/B readiness     validation

The critical thing people miss: rollback means knowing what to roll back to. When your v7 model starts misbehaving in production, can you instantly promote v6 back? That requires v6 to still be in the registry with all its metadata — the data it was trained on, the config, the evaluation metrics. Not just the model weights in a pickle file somewhere.

import mlflow

client = mlflow.tracking.MlflowClient()

# Promote model through stages
client.transition_model_version_stage(
    name="fraud-detector", version=7, stage="Staging"
)

# After staging validation passes:
client.transition_model_version_stage(
    name="fraud-detector", version=7, stage="Production"
)

# Something goes wrong — instant rollback:
client.transition_model_version_stage(
    name="fraud-detector", version=7, stage="Archived"
)
client.transition_model_version_stage(
    name="fraud-detector", version=6, stage="Production"
)

# Load whatever is currently in production — no hardcoded versions
model = mlflow.sklearn.load_model("models:/fraud-detector/Production")

Model Lineage and Model Cards

Lineage answers: "this model in production — what exactly produced it?" In regulated industries (finance, healthcare), auditors require this. But even without regulation, you'll want it the first time a model misbehaves and nobody knows what data it was trained on.

Model cards are nutrition labels for models — a structured doc (introduced by Google) covering what the model does, what it was trained on, how it performs across subgroups, its limitations, and its ethical considerations. Takes 30 minutes to write. Saves hours of "wait, what does this model actually do?"

model_card = {
    "model_name": "fraud-detector-v7",
    "description": "Binary classifier for real-time payment fraud detection",
    "owner": "ml-platform-team",
    "intended_use": "Score e-commerce transactions at checkout",
    "out_of_scope": "Cryptocurrency, peer-to-peer transfers",
    "training_data": {
        "source": "transaction_logs_2024",
        "size": "15M transactions",
        "fraud_rate": "0.28%",
        "date_range": "2024-01-01 to 2024-06-30",
    },
    "performance": {
        "overall_f1": 0.847,
        "slices": {
            "domestic":      {"f1": 0.871, "n": 12_000_000},
            "international": {"f1": 0.764, "n": 3_000_000},  # Known weakness
        },
    },
    "limitations": [
        "Lower recall on international transactions (training data skew)",
        "Not validated on transactions > $50,000",
        "Cold-start: poor on first-time users with no history",
    ],
    "ethical_notes": [
        "False positives disproportionately affect international customers",
        "Must not be sole decision-maker for account freezes",
    ],
}

Pipeline Orchestration

Why You Need an Orchestrator

A training script is not a pipeline. A pipeline has dependencies between steps, handles failures and retries, manages resources (this step needs a GPU, that one doesn't), tracks lineage, and can be triggered automatically. When your "pipeline" is a bash script that calls Python scripts in sequence, the first intermittent S3 timeout at 3 AM means you're debugging at 3 AM.

The Orchestrator Decision Matrix

OrchestratorPhilosophyBest ForWatch Out
AirflowGeneral-purpose DAG schedulerData engineering teams already using it; mixed data + ML workloadsNot ML-aware; no native experiment tracking; DAGs are config, not code
KubeflowML-native on KubernetesTeams with Kubernetes expertise; need strong isolation per stepHeavy Kubernetes dependency; steep learning curve; debugging is painful
PrefectPythonic workflow engineData + ML teams wanting modern DX; dynamic workflowsSmaller ecosystem than Airflow; SaaS model for orchestration layer
DagsterSoftware-defined assetsTeams that think in data assets, not tasks; great testing storyAsset-centric model requires mindset shift; newer community
Vertex AI / SageMakerManaged ML platformTeams that want zero infrastructure; cloud-native shopsVendor lock-in; less flexible; costs can surprise you
MetaflowHuman-centric ML frameworkData scientists who want to write Python, not YAMLOpinionated about execution; AWS-centric (expanding)
ZenMLInfrastructure-agnostic MLTeams wanting to switch orchestrators without rewritingAbstraction layer adds complexity; newer project

The honest decision framework:

  • Already have Airflow and it works? Add ML DAGs to Airflow. Don't migrate.
  • Greenfield ML platform on Kubernetes? Kubeflow.
  • Want the best developer experience? Prefect or Dagster.
  • Want zero infrastructure management? Vertex AI (GCP) or SageMaker (AWS).
  • Data scientists who hate YAML? Metaflow.
# Kubeflow pipeline — each step is a container
from kfp import dsl, compiler

@dsl.component(base_image="python:3.11",
               packages_to_install=["pandas", "scikit-learn", "pyarrow"])
def preprocess(input_path: str, output_path: dsl.Output[dsl.Dataset]):
    import pandas as pd
    df = pd.read_parquet(input_path)
    df = df.dropna(subset=["target"])
    df.to_parquet(output_path.path)

@dsl.component(base_image="pytorch/pytorch:2.1.0-cuda12.1-cudnn8-runtime",
               packages_to_install=["mlflow"])
def train(dataset: dsl.Input[dsl.Dataset],
          model_out: dsl.Output[dsl.Model],
          lr: float = 0.001, epochs: int = 50):
    import torch, pandas as pd
    df = pd.read_parquet(dataset.path)
    model = build_and_train(df, lr=lr, epochs=epochs)
    torch.save(model.state_dict(), model_out.path)

@dsl.component(packages_to_install=["scikit-learn", "mlflow"])
def evaluate_and_gate(model: dsl.Input[dsl.Model],
                      test_data: str, min_f1: float = 0.82):
    metrics = run_evaluation(model.path, test_data)
    if metrics["f1"] < min_f1:
        raise ValueError(f"F1={metrics['f1']:.3f} below gate {min_f1}")
    register_model(model.path, metrics, stage="staging")

@dsl.pipeline(name="fraud-detection-pipeline")
def fraud_pipeline(data_path: str = "s3://data/transactions/"):
    prep = preprocess(input_path=data_path)
    trained = train(dataset=prep.outputs["output_path"], lr=0.001)
    evaluate_and_gate(model=trained.outputs["model_out"],
                      test_data="s3://data/test/")

compiler.Compiler().compile(fraud_pipeline, "pipeline.yaml")

Reproducibility

The Four Pillars

Reproducibility in ML is hard because there are four independent sources of "it works on my machine":

1. Environment. Python 3.11 vs 3.12. NumPy 1.26 vs 2.0. CUDA 11.8 vs 12.1. Any of these can silently change model behavior. Pin everything.

# requirements.txt — pin EXACT versions, not ranges
torch==2.2.1
numpy==1.26.4
scikit-learn==1.4.0
pandas==2.2.0
xgboost==2.0.3

# Better: use a lockfile
# pip-compile requirements.in → requirements.txt
# or: poetry lock → poetry.lock
# or: conda-lock → conda-lock.yml

2. Data snapshots. Code lives in Git. Data is too large for Git. Use DVC or lakehouse versioning to snapshot the exact data used for each experiment.

# DVC tracks data alongside code
$ dvc add data/training_set.parquet
$ git add data/training_set.parquet.dvc data/.gitignore
$ git commit -m "Data v2.3: added Q2 2024 transactions"
$ dvc push  # Uploads actual data to remote storage (S3, GCS, etc.)

# Reproduce any past experiment:
$ git checkout abc123f     # Restore code version
$ dvc checkout             # Restore corresponding data version
$ python train.py          # Same code + same data = same model

3. Random seeds. Training involves randomness in weight initialization, data shuffling, dropout, and augmentation. Control all of them.

import torch, numpy as np, random

def set_all_seeds(seed: int = 42):
    random.seed(seed)
    np.random.seed(seed)
    torch.manual_seed(seed)
    torch.cuda.manual_seed_all(seed)
    torch.backends.cudnn.deterministic = True
    torch.backends.cudnn.benchmark = False

4. Config management. Hyperparameters, feature flags, threshold values — all of it in version-controlled config files, never hardcoded in training scripts.

# configs/fraud_detection_prod.yaml — single source of truth
model:
  type: xgboost
  max_depth: 8
  learning_rate: 0.01
  n_estimators: 500
  early_stopping_rounds: 20

data:
  train_path: s3://data-lake/features/v2.3/train.parquet
  test_path: s3://data-lake/features/v2.3/test.parquet
  target_column: is_fraud

evaluation:
  min_f1: 0.82
  min_slice_f1: 0.70
  max_latency_p99_ms: 100
  slices: [region, device_type, user_tenure]

training:
  seed: 42
  gpu: true

Feature Stores in the MLOps Context

Feature stores were covered in Data Infrastructure, but their MLOps role is distinct: they're the bridge between training and serving that prevents the most common production bug — training/serving skew.

The problem: you compute features one way in your training notebook (Pandas on a CSV) and a different way in your serving endpoint (SQL query on live data). Subtle differences — different null handling, different aggregation windows, different join logic — mean your model sees different feature distributions at inference than it saw during training. Performance degrades and nobody can figure out why.

# Feature store solves training/serving skew
# SAME feature definition used everywhere

# feast_feature_repo/features.py
from feast import Entity, Feature, FeatureView, FileSource
from feast.types import Float32, Int64
from datetime import timedelta

user = Entity(name="user_id", join_keys=["user_id"])

user_features = FeatureView(
    name="user_transaction_features",
    entities=[user],
    schema=[
        Feature(name="txn_count_7d", dtype=Int64),
        Feature(name="avg_txn_amount_30d", dtype=Float32),
        Feature(name="distinct_merchants_7d", dtype=Int64),
        Feature(name="max_txn_amount_24h", dtype=Float32),
    ],
    source=FileSource(path="s3://features/user_txn_features.parquet"),
    ttl=timedelta(hours=24),
)

# Training: get historical features (point-in-time correct)
training_df = store.get_historical_features(
    entity_df=labeled_transactions,  # Has user_id + event_timestamp
    features=["user_transaction_features:txn_count_7d",
              "user_transaction_features:avg_txn_amount_30d"],
).to_df()

# Serving: get latest features (same computation, same code path)
serving_features = store.get_online_features(
    features=["user_transaction_features:txn_count_7d",
              "user_transaction_features:avg_txn_amount_30d"],
    entity_rows=[{"user_id": "u_12345"}],
).to_dict()

The MLOps value: one definition, two execution paths, zero skew. The feature store also handles point-in-time correctness for training — it knows that for a transaction on January 15th, you should only use features computed from data before January 15th, not after. Get this wrong and you have data leakage that looks like a great model but performs terribly in production.

ML Metadata Management

The Metadata Graph

Every ML artifact — dataset, feature, model, prediction — is connected to other artifacts by lineage relationships. Metadata management tracks this graph so you can answer questions like:

  • "Which models were trained on this dataset?" (impact analysis)
  • "What data produced this model that's misbehaving?" (root cause)
  • "Which features does this model use, and who owns them?" (dependency mapping)
  • "Has this model been approved for production use on PII data?" (governance)
# ML Metadata (MLMD) — the lineage backbone
# Used internally by TFX and Kubeflow Pipelines

# Conceptual metadata graph:
#
#   raw_data_v3 ──▶ feature_pipeline_run_42 ──▶ feature_set_v7
#                                                      │
#                                                      ▼
#   config_prod.yaml ──▶ training_run_891 ──▶ model_fraud_v7
#                                                      │
#                                                      ▼
#                              evaluation_run_891 ──▶ metrics
#                                                      │
#                                                      ▼
#                              promotion_event ──▶ serving_endpoint

# This graph answers:
# "model_fraud_v7 was trained by training_run_891,
#  using feature_set_v7 from feature_pipeline_run_42,
#  on raw_data_v3, with config_prod.yaml,
#  achieved F1=0.847, promoted to production on 2024-07-15"

You don't need a dedicated metadata store on day one. MLflow tracks most of this. But as your model count grows past ~20 production models, the questions get harder ("which of our 47 models uses the user_age feature that's about to be deprecated?"), and a proper metadata layer becomes worth the investment. Tools: ML Metadata (MLMD), OpenLineage, DataHub, Amundsen.

MLOps Maturity Levels

Google's Levels 0–2

Google's MLOps maturity framework gives you a clear ladder. Be honest about where you are — most teams are at Level 0 and pretending they're at Level 1.

Level 0 — Manual process. A data scientist trains a model in a notebook, exports it, hands it to an engineer who wraps it in an API and deploys it. Retraining is manual. There's no pipeline. There's no CI/CD. Deployments happen quarterly and involve a heroic effort.

This is where you start. It's fine for one or two models. It collapses at ten.

Level 1 — ML pipeline automation. Training is automated. A pipeline ingests data, validates it, trains a model, evaluates it, and deploys it — triggered by a schedule or data event. Continuous training exists. Feature stores provide consistency. But the pipeline code itself is still deployed manually.

# Level 0: "Hey DevOps, can you deploy this pickle file?"
# Level 1: Pipeline runs automatically, but changing the pipeline is manual
# Level 2: Pipeline code is CI/CD'd too

# Level 1 pipeline trigger (automated training, manual pipeline deployment)
# Cloud Scheduler → triggers Cloud Function → starts Vertex AI Pipeline
#
# Level 2 adds:
# git push → CI tests pipeline code → CD deploys pipeline definition
# → pipeline runs automatically on schedule/trigger

Level 2 — CI/CD pipeline automation. The ML pipeline itself is CI/CD'd. Changes to pipeline code, model code, or configurations go through automated testing and deployment. You have CI for your pipelines (not just for your models). The pipeline is treated as a software artifact that's versioned, tested, and deployed like any other service.

Where to aim: If you have <5 models in production, Level 1 is your target. Get automated training and a model registry working. Level 2 is for teams managing many models where the pipeline itself changes frequently. Don't over-engineer — Level 2 for a team with two models is waste.

The Decision Framework (Not a Logo Wall)

There are 300+ MLOps tools. Listing them all is useless. Instead, here's how to think about assembling your stack. Each MLOps function needs exactly one tool — pick by asking three questions:

  1. What do we already have? (Don't replace working infrastructure)

  3. Cloud-native vs cloud-agnostic? (Single cloud = use their tools. Multi-cloud = use agnostic tools)

The Pragmatic Starting Stack

If you're building from scratch and want to ship in weeks, not months:

# The "good enough to start" MLOps stack
#
# 1. Experiment tracking:   MLflow (free, self-hosted, industry standard)
# 2. Model registry:        MLflow Model Registry (comes with MLflow)
# 3. Pipeline:              GitHub Actions or your existing CI/CD
#                           (Graduate to Airflow/Prefect when you outgrow it)
# 4. Data versioning:       DVC (works with any cloud storage)
# 5. Config management:     Hydra or plain YAML + git
# 6. Monitoring:            Evidently (open source, easy to start)
# 7. Feature store:         Skip until you hit training/serving skew
#
# Total cost: $0 in tooling. Your time is the expensive part.
# Total time to set up: 1-2 weeks for a competent ML engineer.
#
# When to upgrade:
# - MLflow UI too basic?           → Add W&B for development, keep MLflow for registry
# - GitHub Actions too limited?    → Move to Prefect or Dagster
# - Manual feature engineering?    → Add Feast
# - 10+ models in production?     → Consider a managed platform

The biggest mistake teams make is buying a $200K/year MLOps platform when they have two models and three data scientists. Start with open source, hit real pain points, then upgrade the specific component that hurts. Don't let a vendor sell you "Level 2 maturity" when you haven't finished Level 0.

What You Should Now Be Able To Do

  • Explain why MLOps is not just "DevOps for ML" — articulate the three-axis problem (code, data, model) and why silent degradation makes ML systems fundamentally different
  • Design an ML CI/CD pipeline with separate code CI, data CI, and model CI stages — including quality gates that go beyond simple accuracy thresholds
  • Set up experiment tracking (MLflow or W&B), knowing exactly what to log: params, metrics, slice metrics, artifacts, environment, and data version
  • Implement a model registry workflow with versioning, staging → production promotion, and instant rollback capability
  • Choose an orchestrator (Airflow, Kubeflow, Prefect, Dagster, or managed) based on your team's existing infrastructure and constraints — not hype
  • Achieve reproducibility across the four pillars: environment pinning, data snapshots (DVC), random seeds, and version-controlled configs
  • Explain how feature stores prevent training/serving skew and when you actually need one
  • Assess your team's MLOps maturity level (Google 0–2) honestly and identify the next concrete step to level up
  • Assemble a pragmatic MLOps tool stack starting from open source, upgrading components only when you hit real pain points
