Deployment Paradigms

Chapter 13: ML Systems & Production 5 subtopics
TL;DR

You have trained a great model. Now how does it reach users? The answer depends on latency requirements, traffic patterns, cost constraints, and where your users are. This section covers the five deployment paradigms you will encounter in production: batch inference for scale, real-time serving for freshness, streaming for continuous processing, edge deployment for offline and low-latency scenarios, and serverless for bursty workloads.

Batch Inference

The Concept

Batch inference means running your model on a large set of inputs at once, usually on a schedule. Instead of predicting one user's recommendations when they open the app, you predict recommendations for all users overnight and store the results. When a user opens the app, you look up their precomputed results — no model computation at serving time.

This is the simplest deployment pattern. It requires no real-time model server, no feature store, and no GPU at serving time. Just a scheduled job (cron, Airflow, or a cloud scheduler) that runs your model, writes results to a database or key-value store, and shuts down.

# Batch inference pipeline
import pandas as pd
from joblib import load
import redis

def run_batch_inference(date: str):
    """Score all users and cache results."""
    model = load(f"models/recommender_latest.joblib")
    users = pd.read_parquet(f"features/user_features_{date}.parquet")

    # Score in chunks to manage memory
    results = []
    chunk_size = 10_000
    for start in range(0, len(users), chunk_size):
        chunk = users.iloc[start:start + chunk_size]
        predictions = model.predict(chunk)
        results.append(pd.DataFrame({
            "user_id": chunk["user_id"].values,
            "recommendations": predictions
        }))

    all_results = pd.concat(results)

    # Write to Redis for fast serving
    r = redis.Redis(host="redis-server", port=6379)
    pipe = r.pipeline()
    for _, row in all_results.iterrows():
        pipe.set(f"recs:{row['user_id']}", row["recommendations"])
    pipe.execute()

    print(f"Scored {len(all_results)} users, cached to Redis")

When Batch Works — and When It Does Not

Batch inference excels when: the set of inputs is known in advance (all users, all products), predictions do not need to reflect the last few minutes of activity, and cost matters more than freshness. It is the right choice for daily email personalization, weekly report generation, or recommendation systems where "good enough" is acceptable.

It fails when: you need predictions for new or unseen inputs (a new user who just signed up has no precomputed results), you need real-time freshness (the user just watched a movie — update recommendations now), or latency between action and prediction matters (fraud detection cannot wait until tonight's batch run).

Precomputation Strategies

Smart precomputation can stretch batch inference further than you might expect. Precompute results for the most active users (who account for 80% of traffic) and fall back to a simpler model or popular items for the rest. Cache results with TTLs (time-to-live) and recompute incrementally — only re-score users whose features have changed. Use result versioning so you can roll back to previous batch results if the new model produces bad recommendations.

Real-Time (Online) Inference

The Concept

Real-time inference computes predictions on demand, at the moment a request arrives. A user types a search query, the model ranks results and returns them within 100 milliseconds. A payment is submitted, the fraud model scores it before authorization, within 200 milliseconds. This is the most demanding deployment paradigm.

Latency Optimization

In real-time serving, every millisecond matters. Here are the main levers for reducing latency:

Model optimization: Quantize the model (reduce weights from 32-bit to 8-bit or lower), prune unnecessary connections, use knowledge distillation to create a smaller model that mimics the large one, or export to an optimized runtime like ONNX Runtime or TensorRT.

Request batching: Instead of running inference on one request at a time, collect requests over a short window (say, 5 milliseconds) and run them as a batch. GPUs are massively parallel — processing 32 requests takes barely longer than processing one. This improves throughput at the cost of a small latency increase.

# Dynamic batching with a simple queue
import asyncio
import time
from collections import deque

class BatchingServer:
    def __init__(self, model, max_batch_size=32, max_wait_ms=5):
        self.model = model
        self.max_batch_size = max_batch_size
        self.max_wait_ms = max_wait_ms
        self.queue = deque()

    async def predict(self, input_data):
        """Add request to queue and wait for batch result."""
        future = asyncio.Future()
        self.queue.append((input_data, future))

        # If batch is full, process immediately
        if len(self.queue) >= self.max_batch_size:
            self._process_batch()

        return await future

    def _process_batch(self):
        batch_inputs, futures = [], []
        while self.queue and len(batch_inputs) < self.max_batch_size:
            inp, fut = self.queue.popleft()
            batch_inputs.append(inp)
            futures.append(fut)

        # Run model on entire batch (GPU-efficient)
        results = self.model.predict_batch(batch_inputs)

        for fut, result in zip(futures, results):
            fut.set_result(result)

Model warm-up: The first request to a model server is always slow — the model needs to be loaded into memory and the GPU needs to initialize. Run a few dummy predictions when the server starts ("warm-up") to avoid this cold-start penalty hitting real users.

Auto-scaling: Traffic is not constant. E-commerce traffic spikes at lunchtime and during sales events. Auto-scaling adjusts the number of model server replicas based on traffic. Scale up before predicted peaks, scale down during quiet hours to save cost. Kubernetes Horizontal Pod Autoscaler (HPA) is the standard tool for this.

Streaming Inference

The Concept

Streaming inference processes a continuous flow of events. Unlike batch (which processes a fixed dataset) or real-time (which responds to individual requests), streaming operates on an unbounded stream — events arrive continuously, and the system processes them as they come.

Think of IoT sensor data, social media feeds, clickstreams, or financial market data. Events arrive at varying rates, and the system needs to maintain running computations (sliding window aggregates, session detection, anomaly detection) while applying models.

# Streaming inference with Apache Kafka + model
from kafka import KafkaConsumer, KafkaProducer
import json

consumer = KafkaConsumer(
    "user-events",
    bootstrap_servers=["kafka:9092"],
    value_deserializer=lambda m: json.loads(m),
    group_id="fraud-detection",
)

producer = KafkaProducer(
    bootstrap_servers=["kafka:9092"],
    value_serializer=lambda v: json.dumps(v).encode(),
)

model = load_fraud_model()
feature_aggregator = SlidingWindowAggregator(window_size="5min")

for message in consumer:
    event = message.value

    # Update streaming features (e.g., transaction count in last 5 min)
    feature_aggregator.update(event)
    features = feature_aggregator.get_features(event["user_id"])

    # Run model
    score = model.predict(features)

    # Emit prediction to downstream topic
    result = {"transaction_id": event["id"], "fraud_score": score}
    producer.send("fraud-scores", value=result)

    # Alert if high risk
    if score > 0.8:
        producer.send("fraud-alerts", value=result)

Streaming vs Real-Time

The distinction matters. Real-time inference is request-response — a client sends a request and waits for a prediction. Streaming inference is event-driven — events flow through, predictions are emitted downstream, and there is no client waiting. Streaming is the right choice for continuous monitoring (anomaly detection, real-time dashboards), event-driven architectures, and scenarios where you need to maintain running state (like aggregated features over time windows).

Edge Deployment

Why Deploy on the Edge

Edge deployment means running the model on the user's device — a phone, a camera, a car, a factory sensor — instead of in the cloud. Three reasons to do this: latency (a round trip to the cloud takes 50-200 milliseconds; on-device inference takes 5-20 milliseconds), privacy (the data never leaves the device — critical for health data, keyboard predictions, and face recognition), and offline capability (the model works without an internet connection — essential for cars, drones, and remote industrial equipment).

Model Compression for Edge

Edge devices have limited compute, memory, and battery. A 500MB ResNet is not going to run on a smartwatch. Compression techniques make models small and fast enough for constrained hardware:

Quantization reduces the precision of weights from 32-bit floating point to 8-bit integers (or even 4-bit). This cuts model size by 4× and speeds up inference on hardware that supports integer arithmetic. Post-training quantization is the easiest approach — quantize after training with minimal accuracy loss. Quantization-aware training is more complex but preserves more accuracy.

Pruning removes weights that are close to zero — they contribute little to the output. Structured pruning removes entire filters or attention heads, which is more hardware-friendly than removing individual weights. Unstructured pruning achieves higher compression but requires specialized sparse-computation hardware to actually speed things up.

Knowledge distillation trains a small "student" model to mimic a large "teacher" model. The student learns from the teacher's soft probability outputs (which contain more information than hard labels). MobileNet and DistilBERT are famous examples — models designed to be small and fast while retaining most of the teacher's quality.

Edge-Cloud Hybrid

In practice, most edge deployments are hybrid. A small model runs on-device for fast initial predictions, while complex cases are sent to the cloud. For example, a phone's keyboard predicts the next word locally for low latency, but sends longer compositions to a cloud model for grammar checking. The on-device model handles 95% of cases; the cloud handles the rest. This gives you the best of both worlds — low latency for common cases and high quality for hard ones.

Serverless ML

The Concept

Serverless inference uses cloud functions (AWS Lambda, Google Cloud Functions, Azure Functions) to run models without managing servers. You upload your model and inference code, and the cloud provider handles scaling, load balancing, and infrastructure. You pay only for the compute time you actually use — no idle servers burning money overnight.

# AWS Lambda function for ML inference
import json
import boto3
import numpy as np
from joblib import load

# Load model once (outside the handler, reused across invocations)
model = load("/opt/ml/model/classifier.joblib")

def lambda_handler(event, context):
    """Handle a prediction request."""
    body = json.loads(event["body"])
    features = np.array(body["features"]).reshape(1, -1)

    prediction = model.predict(features)[0]
    probability = model.predict_proba(features)[0].max()

    return {
        "statusCode": 200,
        "body": json.dumps({
            "prediction": int(prediction),
            "confidence": float(probability),
        })
    }

The Cold Start Problem

The biggest challenge with serverless ML is cold starts. When a function has not been invoked recently, the cloud provider deallocates its resources. The next invocation must start from scratch — loading the container, loading the model into memory, and initializing the runtime. For a scikit-learn model, this might add 2-5 seconds. For a PyTorch model, it could be 10-30 seconds. This is unacceptable for user-facing latency.

Mitigation strategies: keep functions warm with periodic ping requests, use provisioned concurrency (AWS Lambda's feature to pre-warm a set number of instances), minimize model and package size (smaller = faster load), and use optimized runtimes (ONNX Runtime loads faster than PyTorch).

When Serverless Is Right for ML

Serverless is ideal for: bursty, unpredictable traffic (your model is called 10 times per hour most days but 10,000 times during a flash sale), low-traffic models where provisioning a dedicated server is wasteful, and internal tools where latency requirements are relaxed (a few seconds is fine). It is not ideal for: high-throughput, low-latency serving (the cold start penalty is too high), large models that take long to load (most deep learning models), or GPU-dependent models (serverless GPU support is limited and expensive).

A practical rule of thumb: if your model fits in memory on a standard Lambda (up to 10 GB), loads in under 3 seconds, and your p99 latency target allows for occasional cold starts, serverless can work beautifully. For anything else, use a dedicated model server.
