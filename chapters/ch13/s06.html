Monitoring & Observability

Chapter 13: ML Systems & Production Section 6 of 9

TL;DR

Deploying a model is shipping v0. Keeping it honest in production is the actual job. ML systems fail silently — no stack trace, no 500 error, just confidently wrong predictions while your metrics dashboard stays green. You need monitoring (predefined questions: “Is accuracy above 90%?”) and observability (ad-hoc investigation: “Why did predictions for Brazilian users flip on Tuesday?”). This section covers drift detection, performance monitoring, alerting that doesn’t destroy your team’s sanity, A/B testing that actually proves something, and the investigation playbook for when things go sideways at 2 AM.

Why ML Monitoring Is a Different Animal

Software bugs crash. ML bugs whisper. A null pointer gives you a stack trace, a log line, a pager alert. A model returning garbage predictions looks exactly the same as one returning correct predictions at the API layer — HTTP 200, valid JSON, happy load balancer. Your uptime dashboard stays green while your model silently destroys business value.

That’s the fundamental asymmetry. Traditional software monitoring answers “is the service running?” ML monitoring must also answer “is the service right?”

DimensionSoftware MonitoringML Monitoring
Failure modeCrashes, errors, timeoutsSilent degradation, wrong-but-valid outputs
Root causeCode bugs, infra issuesData shift, concept drift, upstream schema changes
DetectionError codes, exceptionsStatistical tests, distribution comparisons
Ground truthImmediately available (pass/fail)Often delayed by days, weeks, months
DebuggingRead the stack trace“Something changed somewhere” — good luck

You still need latency, error rate, and throughput monitoring. That’s table stakes. But on top of that, you need a whole statistical monitoring layer that traditional software teams have never had to think about.

Data Drift — The Three Flavors

People throw “drift” around as one concept. It’s actually three distinct things, and confusing them leads to wrong responses.

Covariate shift — your input distribution changed. The model sees data that looks different from training. Credit scoring model trained on pre-pandemic spending patterns suddenly sees everyone buying toilet paper and nothing else. The mapping from features → label is still correct; you’re just in a region of feature space the model never learned.

Concept drift — the relationship between inputs and outputs changed. Same features, different correct answer. “Spammy email” in 2020 ≠ “spammy email” in 2025. The model’s learned function is stale.

Label drift — the distribution of outcomes shifted. Your e-commerce return rate went from 8% → 15%. Even if your model’s inputs and predictions are stable, the world moved.

Why the distinction matters: Covariate shift → retrain or expand training data. Concept drift → retrain with fresh labels (old data is actively harmful). Label drift → recalibrate thresholds. Different diagnoses, different treatments.

Detection Methods That Actually Work

The Kolmogorov-Smirnov test is the workhorse for continuous features — it measures the maximum distance between two CDFs. Fast, nonparametric, no distributional assumptions.

from scipy import stats
import numpy as np

def check_drift(reference, production, feature_name, alpha=0.05):
    """KS test for continuous feature drift."""
    ks_stat, p_value = stats.ks_2samp(reference, production)
    drifted = p_value < alpha
    if drifted:
        print(f"⚠️  {feature_name}: KS={ks_stat:.4f}, p={p_value:.2e} — DRIFT")
    return {"feature": feature_name, "ks": ks_stat, "p": p_value, "drifted": drifted}

# Run across all features
ref_data = {"age": np.random.normal(35, 10, 10000),
            "income": np.random.lognormal(10.5, 0.8, 10000)}
prod_data = {"age": np.random.normal(38, 12, 5000),   # shifted!
             "income": np.random.lognormal(10.5, 0.8, 5000)}  # stable

for feat in ref_data:
    check_drift(ref_data[feat], prod_data[feat], feat)

For categorical features: Chi-Square test. For a single-number “how much did this distribution move?” score: Population Stability Index (PSI).

import numpy as np

def psi(reference, current, bins=10):
    """Population Stability Index — the credit risk industry's gift to ML."""
    breakpoints = np.percentile(reference, np.linspace(0, 100, bins + 1))
    breakpoints[0], breakpoints[-1] = -np.inf, np.inf

    ref_pct = np.histogram(reference, bins=breakpoints)[0] / len(reference) + 1e-6
    cur_pct = np.histogram(current, bins=breakpoints)[0] / len(current) + 1e-6

    return np.sum((cur_pct - ref_pct) * np.log(cur_pct / ref_pct))

< 0.1No significant driftSleep well
0.1 – 0.25Moderate driftInvestigate, maybe retrain
> 0.25Significant driftModel is out of distribution — retrain

Schema drift deserves a mention. It’s not statistical — it’s structural. A column was renamed, a new enum value appeared, a field that was never null suddenly has nulls. Usually caused by someone upstream “refactoring” event logging without telling you. Catch it with validation checks before your model ever sees the data.

def validate_schema(df, expected_schema):
    """Catch structural data issues before they become model issues."""
    issues = []
    for col, specs in expected_schema.items():
        if col not in df.columns:
            issues.append(f"Missing column: {col}")
            continue
        null_rate = df[col].isnull().mean()
        if null_rate > specs.get("max_null_rate", 0.01):
            issues.append(f"{col}: null rate {null_rate:.1%} exceeds {specs['max_null_rate']:.1%}")
        if "allowed_values" in specs:
            unexpected = set(df[col].dropna().unique()) - set(specs["allowed_values"])
            if unexpected:
                issues.append(f"{col}: unexpected values {unexpected}")
    return issues

schema = {
    "country": {"max_null_rate": 0.0, "allowed_values": ["US", "UK", "DE", "FR", "JP"]},
    "age": {"max_null_rate": 0.02},
    "purchase_amount": {"max_null_rate": 0.01},
}

Model Performance Monitoring — When Metrics Degrade Silently

Here’s the uncomfortable truth: you often can’t compute accuracy in real time because you don’t have ground truth yet. A loan default takes months to materialize. A content recommendation’s real value unfolds over weeks. So you need a two-layer strategy.

Layer 1 — Proxy signals (real-time): prediction distribution shifts, confidence score trends, feature drift scores. These don’t tell you the model is wrong, but they tell you something changed.

Layer 2 — True metrics (delayed): accuracy, precision, recall, AUC computed when labels finally arrive. Backfill your dashboards and trigger alerts retroactively.

from collections import deque
import numpy as np

class PerformanceMonitor:
    """Track metric degradation with sliding windows."""
    def __init__(self, baseline_metric, warn_drop=0.03, critical_drop=0.07):
        self.baseline = baseline_metric
        self.warn_drop = warn_drop
        self.critical_drop = critical_drop
        self.window = deque(maxlen=1000)

    def record(self, y_true, y_pred):
        self.window.append(int(y_true == y_pred))
        if len(self.window) < 100:
            return None  # not enough data yet

        current = np.mean(self.window)
        drop = self.baseline - current

        if drop > self.critical_drop:
            return f"🔴 CRITICAL: accuracy dropped {drop:.1%} from baseline"
        elif drop > self.warn_drop:
            return f"🟡 WARNING: accuracy dropped {drop:.1%} from baseline"
        return None

monitor = PerformanceMonitor(baseline_metric=0.92)

Segment-Level Monitoring — Where the Bodies Are Buried

Aggregate metrics are liars. Your overall accuracy might be 92%, but that’s 98% for your majority segment and 60% for a minority one. The aggregate looks fine. The minority segment is on fire.

Monitor across slices: geography, user demographic, device type, time of day, traffic source. This is where fairness monitoring meets production monitoring — they’re the same thing.

def segment_metrics(predictions_df, segments, metric_fn):
    """Compute metrics per segment — catches what aggregates hide."""
    results = {}
    for segment_col in segments:
        for segment_val, group in predictions_df.groupby(segment_col):
            key = f"{segment_col}={segment_val}"
            results[key] = metric_fn(group["y_true"], group["y_pred"])
    return results

Alerting Strategies — Don’t Cry Wolf

Alert fatigue kills monitoring programs. If your team gets 50 alerts a day and 48 are false positives, they stop reading all 50. Then the real one comes in at alert #49 and nobody notices.

Thresholds vs. Statistical Tests

Static thresholds are the simplest approach: “alert if accuracy drops below 0.85.” Easy to set, easy to understand, and terrible at handling normal variation. If your accuracy naturally fluctuates between 0.89 and 0.93, a threshold at 0.85 catches catastrophic failures but misses gradual decline.

Statistical tests are better for detecting change: “alert if this week’s metric is statistically significantly different from the trailing 4-week average.” This adapts to seasonal variation and catches subtler shifts.

Use both. Static thresholds as guardrails (absolute floor). Statistical tests for detecting trends.

Alert TypeGood ForWatch Out For
Static thresholdCatching catastrophic failures fastMisses slow degradation; requires manual tuning
Relative change“5% drop from last week”Fails if baseline was already degraded
Statistical test (z/t-test)Detecting real shifts vs. noiseNeeds enough samples; can be slow to fire
CUSUM / Page-HinkleySequential change-point detectionTuning the sensitivity parameter

A/B Testing for ML — Harder Than You Think

A/B testing a button color is straightforward. A/B testing ML models has traps that will burn you if you treat it the same way.

The ML-Specific Pitfalls

Network effects: In a marketplace, treating user A with model-v2 affects the experience of user B who’s on model-v1 (e.g., search ranking changes affect inventory availability). Random user-level splitting doesn’t fully isolate treatments.

Delayed outcomes: If your metric is “30-day retention,” you need to run the test for at least 30 days after the last user enters. Peeking at results early and stopping inflates your false positive rate.

Multiple metrics: You care about click-through rate AND revenue AND user satisfaction AND latency. Testing all of them multiplies your false positive risk.

Proper Setup

import hashlib
import numpy as np
from scipy import stats

class MLExperiment:
    """Deterministic, reproducible A/B split for ML models."""
    def __init__(self, experiment_id, control_pct=0.50):
        self.experiment_id = experiment_id
        self.control_pct = control_pct

    def assign(self, user_id):
        """Deterministic assignment — same user always gets same variant."""
        hash_input = f"{self.experiment_id}:{user_id}"
        hash_val = int(hashlib.sha256(hash_input.encode()).hexdigest(), 16)
        return "control" if (hash_val % 10000) / 10000 < self.control_pct else "treatment"

    @staticmethod
    def analyze(control_metric, treatment_metric, alpha=0.05):
        """Two-sample t-test with practical significance check."""
        t_stat, p_value = stats.ttest_ind(control_metric, treatment_metric)
        control_mean = np.mean(control_metric)
        treatment_mean = np.mean(treatment_metric)
        lift = (treatment_mean - control_mean) / control_mean

        return {
            "control_mean": control_mean,
            "treatment_mean": treatment_mean,
            "lift": lift,
            "p_value": p_value,
            "significant": p_value < alpha,
            "sample_size": len(control_metric) + len(treatment_metric),
        }

exp = MLExperiment("rec-model-v3-vs-v2")
# For each request: variant = exp.assign(user_id)

Guardrail Metrics

Guardrails are metrics you don’t want to degrade, even if the primary metric improves. Your new recommendation model might boost click-through rate by 5%, but if it also increases page load time by 300ms, that’s not a win.

Common guardrails: latency (p50, p99), error rate, revenue per user, unsubscribe rate, customer support ticket volume. Define them before the experiment starts. If any guardrail degrades beyond its threshold, the experiment fails regardless of primary metric lift.

Logging & Observability — What to Log for ML Inference

Monitoring answers questions you anticipated. Observability lets you answer questions you didn’t. The difference is in what you capture.

For every ML prediction, log this:

import json, time, uuid, hashlib

def log_prediction(model_version, features, prediction, confidence, metadata=None):
    """Structured prediction log — your future debugging self will thank you."""
    log_entry = {
        "prediction_id": str(uuid.uuid4()),
        "timestamp": time.time(),
        "model_version": model_version,

        # Input snapshot — you WILL need this for debugging
        "features": features,
        "feature_hash": hashlib.md5(
            json.dumps(features, sort_keys=True).encode()
        ).hexdigest(),

        # Output details
        "prediction": prediction,
        "confidence": confidence,
        "all_class_probabilities": None,  # fill for classifiers

        # Context for investigation
        "latency_ms": None,  # filled by serving layer
        "user_id": metadata.get("user_id") if metadata else None,
        "request_source": metadata.get("source") if metadata else None,
        "experiment_variant": metadata.get("variant") if metadata else None,
    }
    # Ship to your log pipeline (Kafka, Kinesis, stdout via Fluentd, etc.)
    return log_entry
Log features, not just predictions. When you’re debugging a production issue three weeks from now, you’ll need to know what the model saw, not just what it said. Yes, this increases storage costs. It’s worth it. If storage is truly a constraint, log a hash of the features and keep the raw features in a queryable store with a 90-day TTL.

Distributed Tracing for ML Pipelines

A prediction request touches multiple services: feature store lookup, preprocessing, model inference, postprocessing, business rules. When latency spikes, you need to know which stage caused it.

from opentelemetry import trace

tracer = trace.get_tracer("ml-serving")

def predict_with_tracing(request):
    with tracer.start_as_current_span("ml-prediction") as parent:

        with tracer.start_as_current_span("feature-retrieval"):
            features = fetch_features(request.user_id)

        with tracer.start_as_current_span("preprocessing"):
            tensor = preprocess(features)

        with tracer.start_as_current_span("model-inference"):
            raw_output = model(tensor)

        with tracer.start_as_current_span("postprocessing"):
            result = postprocess(raw_output)

        parent.set_attribute("model.version", "v2.3")
        parent.set_attribute("model.confidence", float(result.confidence))
        return result

Pair with Jaeger or Zipkin for visualization. Your ops team already understands traces — this just extends the pattern to ML-specific stages.

Debugging Production ML — The Investigation Playbook

It’s 2 AM. An alert fires. Model performance dropped. Here’s the playbook, in order:

Step 1: Is it the model or the infrastructure? Check latency, error rates, resource utilization. If the serving pod is OOMing, that’s an infra problem, not an ML problem.

Step 2: Is it the data? Run your schema validation. Check null rates, feature distributions, PSI scores. 80% of “model problems” are actually data pipeline problems. Someone upstream changed a column name, a third-party API started returning different formats, a feature went stale.

Step 3: Is it a specific segment? Slice your metrics by every dimension you have. Country, device, time, traffic source. If it’s isolated to one segment, that narrows the blast radius and points to the cause.

Step 4: When did it start? Correlate with deployments, data pipeline changes, upstream service changes, external events. The timestamp of the degradation is your biggest clue.

Step 5: Can you reproduce it? Pull production examples that the model got wrong. Run them through the model locally with the same feature values. If you logged features (see above), this is easy. If you didn’t — this is why you should.

from scipy import stats

def investigate_degradation(predictions_log, time_range, baseline_period):
    """Automated first-pass investigation."""
    recent = predictions_log.query(
        f"timestamp >= '{time_range[0]}' and timestamp <= '{time_range[1]}'"
    )
    baseline = predictions_log.query(
        f"timestamp >= '{baseline_period[0]}' and timestamp <= '{baseline_period[1]}'"
    )

    report = {}

    # Check prediction distribution shift
    report["prediction_distribution"] = {
        "recent_mean": recent["prediction"].mean(),
        "baseline_mean": baseline["prediction"].mean(),
        "ks_pvalue": stats.ks_2samp(
            baseline["prediction"], recent["prediction"]
        ).pvalue,
    }

    # Check confidence distribution shift
    report["confidence_shift"] = {
        "recent_mean_conf": recent["confidence"].mean(),
        "baseline_mean_conf": baseline["confidence"].mean(),
    }

    # Check per-segment performance
    for segment in ["country", "device_type", "source"]:
        if segment in recent.columns:
            report[f"segment_{segment}"] = (
                recent.groupby(segment)["correct"].mean().to_dict()
            )

    # Check feature-level drift
    feature_cols = [c for c in recent.columns if c.startswith("feat_")]
    report["feature_drift"] = {}
    for col in feature_cols:
        ks = stats.ks_2samp(baseline[col].dropna(), recent[col].dropna())
        if ks.pvalue < 0.01:
            report["feature_drift"][col] = {"ks": ks.statistic, "p": ks.pvalue}

    return report
The 80/20 of ML debugging: In practice, ~40% of production issues trace back to data pipeline problems, ~25% to upstream schema or feature changes, ~20% to genuine drift, and ~15% to infrastructure. Start with data.

Evidently AI — The Quick Win

If you’re starting from zero monitoring, Evidently gets you the most insight with the least effort:

from evidently.report import Report
from evidently.metric_preset import DataDriftPreset, DataQualityPreset
from evidently.metrics import ColumnDriftMetric

# Comprehensive drift report in 4 lines
report = Report(metrics=[
    DataDriftPreset(),          # drift across all features
    DataQualityPreset(),        # nulls, duplicates, out-of-range
    ColumnDriftMetric(column_name="prediction"),  # output drift
])
report.run(reference_data=train_df, current_data=prod_df)
report.save_html("weekly_drift_report.html")

# For CI/CD integration — get results as dict
results = report.as_dict()
drift_detected = results["metrics"][0]["result"]["dataset_drift"]
if drift_detected:
    print("🚨 Dataset-level drift detected — block deployment")

Prometheus + Grafana — The DIY Path

If your org already runs Prometheus, just expose ML-specific metrics from your serving endpoint:

from prometheus_client import Histogram, Counter, Gauge, start_http_server

PRED_LATENCY = Histogram(
    "ml_prediction_seconds", "Inference latency",
    buckets=[0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1.0]
)
PRED_COUNTER = Counter(
    "ml_predictions_total", "Predictions served",
    ["model_version", "predicted_class"]
)
CONFIDENCE_GAUGE = Gauge(
    "ml_confidence_score", "Rolling average confidence"
)
DRIFT_PSI = Gauge(
    "ml_feature_psi", "PSI score per feature",
    ["feature_name"]
)

@PRED_LATENCY.time()
def predict(features):
    result = model.predict(features)
    PRED_COUNTER.labels(
        model_version="v2.3", predicted_class=str(result.label)
    ).inc()
    CONFIDENCE_GAUGE.set(result.confidence)
    return result

start_http_server(8000)  # /metrics endpoint for Prometheus scraping

Build Grafana dashboards that layer: business KPIs at the top, model metrics in the middle, infrastructure at the bottom. When a business metric dips, drill down through the layers.

Putting It Together — A Production Monitoring Stack

Here’s what a mature ML monitoring setup looks like, from the most urgent to least urgent layer:

┌──────────────────────────────────────────────────────┐
│  Business KPIs           (revenue, engagement)       │  ← Stakeholders watch
├──────────────────────────────────────────────────────┤
│  Model Performance       (accuracy, AUC, lift)       │  ← ML team watches
├──────────────────────────────────────────────────────┤
│  Prediction Behavior     (distribution, confidence)  │  ← Proxy when no labels
├──────────────────────────────────────────────────────┤
│  Data Quality & Drift    (PSI, KS, schema checks)   │  ← Catches 80% of issues
├──────────────────────────────────────────────────────┤
│  Infrastructure          (latency, throughput, RAM)  │  ← SRE team watches
└──────────────────────────────────────────────────────┘

Alerting flow:
  Business KPI dips → check model performance → check data quality → check infra
  (Top-down investigation, bottom-up root cause)
Start with data quality monitoring. It’s the least glamorous layer and catches the most problems. A simple “are all expected columns present, are null rates stable, are value ranges sane” check running before every inference batch will prevent more incidents than any fancy drift detection algorithm.

What You Should Now Be Able To Do

  • Explain why ML monitoring requires a statistical layer on top of standard software monitoring
  • Distinguish covariate shift, concept drift, and label drift — and know which response each requires
  • Implement KS tests and PSI for drift detection on continuous features; Chi-Square for categoricals
  • Design a two-layer monitoring system: real-time proxy signals + delayed ground-truth metrics
  • Set up alerting with both static thresholds and statistical change detection to avoid alert fatigue
  • Run a proper A/B test for ML models with deterministic assignment, guardrail metrics, and correct statistical analysis
  • Structure prediction logs to support future debugging (features, predictions, confidence, context)
  • Walk through the 5-step investigation playbook when production metrics degrade
  • Choose between Evidently, WhyLabs, Arize, and Prometheus+Grafana based on your team’s needs
  • Build segment-level monitoring that catches problems hidden by aggregate metrics
