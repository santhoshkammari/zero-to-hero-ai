Model Optimization

Chapter 13: ML Systems & Production 4 subtopics

TL;DR

A model that's accurate but too slow or too large for production is useless. Model optimization is the art of making models smaller, faster, and cheaper without destroying their quality. Quantization reduces numerical precision. Pruning removes unnecessary weights. Knowledge distillation trains a small student model to mimic a large teacher. Runtime optimizers like TensorRT and ONNX Runtime squeeze out the last drops of hardware performance. These techniques are what transform a research prototype into a production-ready system.

The Production Reality Check

You've trained a beautiful model. 95% accuracy on your test set. Your team celebrates. Then someone asks: "How much does it cost to serve?" And the room goes quiet. Your transformer model takes 200ms per inference on a GPU, and GPUs cost $2/hour. At 100 requests per second, you're looking at $50k/month in infrastructure. Your VP asks if you can get that down to $5k. Now you need model optimization.

The goal isn't smaller models for their own sake. The goal is hitting your latency, throughput, and cost targets while staying within an acceptable accuracy budget. Every optimization technique trades some accuracy for speed or size. Your job is finding the sweet spot.

Quantization

The Core Idea

Quantization reduces the numerical precision of model weights and activations. A standard model uses 32-bit floating point (FP32) numbers. Each weight takes 4 bytes. Quantization converts these to lower precision — FP16 (2 bytes), INT8 (1 byte), or even INT4 (0.5 bytes). This reduces model size proportionally and speeds up computation because lower-precision arithmetic is faster on modern hardware.

Why does this work? Neural network weights cluster in narrow ranges. A weight value of 0.23847291 and 0.23828125 (the nearest FP16 representation) produce essentially identical predictions. The network learned rough patterns, not precise decimal places.

Post-Training Quantization (PTQ)

Post-Training Quantization is the simplest approach: take a trained model and convert its weights to lower precision. No retraining needed. You typically need a small calibration dataset (a few hundred samples) to determine the optimal scale and zero-point for each layer's quantization mapping.

import torch
from torch.quantization import quantize_dynamic

# Dynamic quantization: weights quantized statically, activations at runtime
model_fp32 = load_trained_model()
model_int8 = quantize_dynamic(
    model_fp32,
    {torch.nn.Linear},  # Quantize linear layers
    dtype=torch.qint8
)

# Check size reduction
import os
torch.save(model_fp32.state_dict(), "model_fp32.pt")
torch.save(model_int8.state_dict(), "model_int8.pt")
fp32_size = os.path.getsize("model_fp32.pt") / 1e6
int8_size = os.path.getsize("model_int8.pt") / 1e6
print(f"FP32: {fp32_size:.1f} MB → INT8: {int8_size:.1f} MB")
print(f"Compression: {fp32_size/int8_size:.1f}x")

Quantization-Aware Training (QAT)

Quantization-Aware Training simulates quantization during training. The model learns to be robust to the precision loss. During forward passes, weights and activations are "fake quantized" — converted to low precision then back to high precision, introducing the same rounding errors that real quantization would cause. This typically recovers 50-100% of the accuracy lost by PTQ.

import torch.quantization as quant

# Prepare model for QAT
model.train()
model.qconfig = quant.get_default_qat_qconfig('fbgemm')
model_prepared = quant.prepare_qat(model)

# Train as normal — fake quantization happens automatically
for epoch in range(fine_tune_epochs):
    for batch in dataloader:
        loss = model_prepared(batch).loss
        loss.backward()
        optimizer.step()
        optimizer.zero_grad()

# Convert to actual quantized model for inference
model_quantized = quant.convert(model_prepared)

INT8, INT4, and Binary Quantization

INT8 quantization (8-bit integers) is the safe default. It's well-supported by hardware, gives ~4x compression over FP32, and typically costs less than 1% accuracy. INT4 (4-bit) pushes further — 8x compression — but accuracy loss is more significant and hardware support is less universal. Binary quantization (1-bit) reduces each weight to +1 or -1. It's extreme but enables specialized hardware that replaces multiplications with simple additions.

LLM-Specific Quantization: GPTQ, AWQ, GGUF

Large language models have driven a wave of quantization innovation. GPTQ (GPT Quantization) uses second-order information (the Hessian) to determine which weights are most important and should lose the least precision. It can quantize a 70B model to 4-bit with minimal quality loss. AWQ (Activation-Aware Weight Quantization) protects the 1% of weights that handle salient activations, achieving better quality than naive 4-bit approaches. GGUF is llama.cpp's format that enables quantized models to run on CPUs — it's why you can run a 7B model on a laptop without a GPU.

Practical guidance: For serving LLMs, start with AWQ or GPTQ at 4-bit. Run your evaluation benchmarks. If quality is acceptable, you've just cut your serving costs by 4-8x. If not, try 8-bit. For non-LLM models, INT8 PTQ is almost always sufficient and is a no-brainer first step.

Dynamic vs. Static Quantization

Static quantization pre-computes the scale factors using a calibration dataset — the quantization parameters are fixed. Dynamic quantization computes scale factors on-the-fly for each input. Static is faster at inference (no runtime computation) but requires calibration data. Dynamic is simpler to apply but slightly slower. For most production deployments, static quantization wins.

Pruning

The Intuition

Pruning removes weights from a neural network that contribute little to the output. Research consistently shows that large networks are heavily over-parameterized — you can remove 90% of the weights in some models with barely any accuracy loss. The analogy: a sculptor removes marble to reveal the statue that was always inside.

Unstructured vs. Structured Pruning

Unstructured pruning removes individual weights (sets them to zero) anywhere in the network. It achieves the highest compression ratios but creates sparse matrices that don't map efficiently to standard hardware. You need specialized sparse compute libraries to get actual speedups.

Structured pruning removes entire neurons, channels, or attention heads. The resulting model is a regular, smaller dense model that runs faster on any hardware without special support. The compression ratio is lower, but the practical speedup is typically better.

import torch
import torch.nn.utils.prune as prune

model = load_trained_model()

# Unstructured: remove 50% of smallest-magnitude weights
for name, module in model.named_modules():
    if isinstance(module, torch.nn.Linear):
        prune.l1_unstructured(module, name='weight', amount=0.5)

# Structured: remove 30% of output channels with smallest L2 norm
for name, module in model.named_modules():
    if isinstance(module, torch.nn.Conv2d):
        prune.ln_structured(module, name='weight', amount=0.3, n=2, dim=0)

# Make pruning permanent (remove the mask, zero out the weights)
for name, module in model.named_modules():
    if hasattr(module, 'weight_orig'):
        prune.remove(module, 'weight')

Magnitude-Based Pruning

The simplest pruning strategy: remove weights with the smallest absolute values. The assumption is that small weights contribute less to the output. It's crude but works surprisingly well. Sort all weights by magnitude, zero out the bottom X%, fine-tune for a few epochs to recover accuracy. Repeat for iterative pruning.

The Lottery Ticket Hypothesis

The Lottery Ticket Hypothesis (Frankle & Carlin, 2019) made a provocative claim: within a large, randomly initialized network, there exists a small subnetwork (a "winning ticket") that, when trained in isolation from the same initialization, reaches the same accuracy as the full network. This means large networks aren't needed for their capacity — they're needed because they make it easier to find the good subnetwork during training. The practical implication: train big, prune small, and the pruned model retains the "winning ticket."

Iterative Pruning with Retraining

The most effective pruning recipe is iterative: prune a little (10-20%), retrain to recover accuracy, prune again, retrain again. Each cycle, the model adapts to the reduced capacity. This reaches much higher sparsity levels than one-shot pruning. The downside: it takes multiple training cycles, which multiplies your training cost.

The sparsity vs. speed trap: A model that's 90% sparse sounds impressive, but if your hardware can't exploit sparsity, you get zero speedup. NVIDIA's Ampere and Hopper GPUs support 2:4 structured sparsity (2 out of every 4 weights are zero) with dedicated hardware. Any other sparsity pattern needs software-level sparse matrix support. Always benchmark actual inference speed, not just parameter count.

Knowledge Distillation

Teacher-Student Framework

Knowledge distillation trains a small "student" model to mimic a large "teacher" model. Instead of training the student on hard labels (cat = 1, dog = 0), you train it on the teacher's soft probability distribution (cat = 0.85, dog = 0.12, fox = 0.03). These soft labels carry richer information — the teacher is saying "this looks like a cat, but there's a bit of dog-ness to it." The student learns these inter-class relationships, which it could never learn from hard labels alone.

import torch
import torch.nn.functional as F

def distillation_loss(student_logits, teacher_logits, labels,
                      temperature=4.0, alpha=0.7):
    """Combined distillation + classification loss."""
    # Soft targets from teacher (high temperature → softer distribution)
    soft_teacher = F.softmax(teacher_logits / temperature, dim=-1)
    soft_student = F.log_softmax(student_logits / temperature, dim=-1)
    distill_loss = F.kl_div(soft_student, soft_teacher, reduction='batchmean')
    distill_loss *= temperature ** 2  # Scale by T² to match gradient magnitude

    # Hard label loss (standard cross-entropy)
    hard_loss = F.cross_entropy(student_logits, labels)

    return alpha * distill_loss + (1 - alpha) * hard_loss

# Training loop
teacher.eval()
for batch in dataloader:
    with torch.no_grad():
        teacher_logits = teacher(batch['input'])
    student_logits = student(batch['input'])
    loss = distillation_loss(student_logits, teacher_logits, batch['label'])
    loss.backward()
    optimizer.step()
    optimizer.zero_grad()

Temperature and the Alpha Parameter

The temperature parameter controls how soft the teacher's outputs are. Temperature=1 gives the original distribution. Higher temperatures (3-10) flatten the distribution, revealing more of the inter-class structure. The alpha parameter balances distillation loss vs. hard-label loss. Typical values are 0.5-0.9, favoring distillation.

Feature-Based Distillation

Beyond output logits, you can distill intermediate representations. Force the student's hidden layers to match the teacher's hidden activations (after a projection to match dimensions). This gives the student more learning signal and often produces better results than output-only distillation, especially when teacher and student architectures differ significantly.

Self-Distillation

Self-distillation uses the model as its own teacher. Train a model, use its predictions as soft targets, and train the same architecture again. Surprisingly, this often improves performance — the soft labels act as a form of regularization, smoothing the loss landscape.

Real-World Examples: DistilBERT and TinyBERT

DistilBERT distilled BERT into a model 40% smaller and 60% faster while retaining 97% of BERT's language understanding. It used a combination of output distillation, hidden state alignment, and embedding layer matching. TinyBERT went further with a two-stage distillation process — first distilling during pre-training, then during fine-tuning — achieving 7.5x compression. These models proved that distillation scales to real applications.

Runtime Optimization

ONNX Runtime

ONNX (Open Neural Network Exchange) provides a standard format for ML models. Export your PyTorch or TensorFlow model to ONNX, then run it with ONNX Runtime — Microsoft's high-performance inference engine. ONNX Runtime applies graph optimizations (operator fusion, constant folding, memory planning) that the training framework doesn't bother with.

# Export PyTorch model to ONNX
import torch
dummy_input = torch.randn(1, 3, 224, 224)
torch.onnx.export(model, dummy_input, "model.onnx",
                   opset_version=17,
                   input_names=['image'],
                   output_names=['prediction'],
                   dynamic_axes={'image': {0: 'batch_size'}})

# Run inference with ONNX Runtime
import onnxruntime as ort
session = ort.InferenceSession("model.onnx",
    providers=['CUDAExecutionProvider', 'CPUExecutionProvider'])

result = session.run(None, {"image": input_array})
# Typically 2-3x faster than PyTorch eager mode

TensorRT

NVIDIA's TensorRT is the gold standard for GPU inference optimization. It takes a model, analyzes the target GPU architecture, and generates optimized GPU kernels. It applies layer fusion (combining Conv+BatchNorm+ReLU into a single kernel), precision calibration (INT8/FP16), dynamic tensor memory management, and kernel auto-tuning. Speedups of 2-6x over PyTorch are common.

OpenVINO

Intel's OpenVINO optimizes models for Intel hardware — CPUs, integrated GPUs, and VPUs. If you're deploying on Intel-based edge devices or servers without NVIDIA GPUs, OpenVINO is your best option. It provides model compression tools, runtime optimization, and efficient CPU inference that often beats ONNX Runtime on Intel hardware.

Graph Optimization and Operator Fusion

All these runtimes share a core optimization: graph-level analysis. They look at your model as a directed graph of operations and find patterns that can be fused into single, optimized kernels. For example, a sequence of MatMul → Add → ReLU becomes one fused kernel with one memory read and one memory write, instead of three separate kernels with six memory operations. Memory bandwidth is often the bottleneck for inference, so reducing memory operations is the biggest win.

Compiler-Based Optimization: TVM and XLA

Apache TVM is an open-source ML compiler that generates optimized code for any hardware target. It searches the space of possible implementations (tile sizes, loop orders, vectorization strategies) to find the fastest one for your specific model on your specific hardware. XLA does similar optimizations for TensorFlow and JAX, with particularly strong support for TPUs. The trend is clear: ML inference is becoming a compiler problem, and these tools automate optimizations that used to require manual CUDA kernel writing.

Combining techniques: The most aggressive optimizations stack. Start with knowledge distillation (small student model), then quantize to INT8, then run through TensorRT. A model that started as 2 GB / 200ms can end up at 50 MB / 5ms. That's a 40x size reduction and 40x speedup. Each technique is multiplicative with the others.
