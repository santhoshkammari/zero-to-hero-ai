Deployment & Serving

Chapter 13: ML Systems & Production Section 4 of 9
TL;DR

Your model is worthless sitting in a notebook. Deployment is where it starts earning its keep. This section covers how models reach users — batch vs. real-time vs. streaming paradigms, serialization formats (ONNX, TorchScript, SavedModel), serving frameworks (Triton, TorchServe, vLLM, BentoML), optimization for production (quantization, pruning, distillation, compilation), containerization, scaling, latency tricks, and deployment strategies that don't blow up your production system at 3 AM.

Deployment Paradigms

Every deployment decision boils down to one question: when does inference happen relative to the request? Before the user asks (batch), when they ask (real-time), or continuously whether they ask or not (streaming). Everything else — infrastructure, cost, complexity — follows from that answer.

Batch Inference

Score everything ahead of time. Store the results. Look them up when a user shows up. No model server needed at serving time — just a database read.

# Batch inference: score all users nightly, cache to Redis
import pandas as pd
from joblib import load
import redis

def run_batch_inference(date: str):
    model = load("models/recommender_latest.joblib")
    users = pd.read_parquet(f"features/user_features_{date}.parquet")

    r = redis.Redis(host="redis-server", port=6379)
    pipe = r.pipeline()

    for start in range(0, len(users), 10_000):
        chunk = users.iloc[start:start + 10_000]
        preds = model.predict(chunk)
        for uid, rec in zip(chunk["user_id"], preds):
            pipe.set(f"recs:{uid}", rec)

    pipe.execute()
    print(f"Scored {len(users)} users → Redis")

Batch is the simplest pattern, and you'd be surprised how far it stretches. Precompute for the top 80% of active users, fall back to popular items for the rest. Use TTLs and incremental re-scoring — only re-score users whose features actually changed. It fails when you need freshness (fraud detection can't wait for tonight's batch) or when inputs aren't known in advance (a brand-new user has no precomputed results).

Real-Time (Online) Inference

A user hits an endpoint. The model scores their request. The response comes back in <100ms. This is the hardest paradigm — it demands a live model server, GPU management, latency optimization, and auto-scaling. But it's the only option when predictions must reflect the most recent data.

Streaming Inference

Events flow in continuously — Kafka topics, IoT sensors, clickstreams. The model processes each event as it arrives and emits predictions downstream. No client is waiting for a response; the system is event-driven.

# Streaming inference: Kafka consumer with sliding-window features
from kafka import KafkaConsumer, KafkaProducer
import json

consumer = KafkaConsumer("user-events", bootstrap_servers=["kafka:9092"],
                         value_deserializer=lambda m: json.loads(m),
                         group_id="fraud-detection")
producer = KafkaProducer(bootstrap_servers=["kafka:9092"],
                         value_serializer=lambda v: json.dumps(v).encode())

model = load_fraud_model()
aggregator = SlidingWindowAggregator(window_size="5min")

for message in consumer:
    event = message.value
    aggregator.update(event)
    features = aggregator.get_features(event["user_id"])
    score = model.predict(features)
    producer.send("fraud-scores", value={"tx_id": event["id"], "score": score})
    if score > 0.8:
        producer.send("fraud-alerts", value={"tx_id": event["id"], "score": score})

The key distinction: real-time is request-response (client waits). Streaming is fire-and-forget (events flow through, predictions emitted downstream). Streaming is the right choice for continuous monitoring, anomaly detection, and anything where you maintain running state over time windows.

Edge Deployment

Run the model on the device — phone, camera, car, factory sensor. Three reasons: latency (5ms on-device vs. 50-200ms round-trip to cloud), privacy (data never leaves the device), offline capability (cars and drones can't depend on WiFi).

In practice, most edge deployments are hybrid. A small model on-device handles 95% of cases fast; hard cases get shipped to a cloud model. Your keyboard's next-word prediction runs locally; grammar checking goes to the cloud.

Serverless (AWS Lambda, Cloud Functions) is a deployment paradigm worth knowing: you upload model + code, the cloud handles scaling. You pay per invocation. Great for bursty traffic and internal tools. Terrible for anything latency-sensitive because cold starts add 2-30 seconds when the function hasn't been called recently.

Model Serialization

Before you can serve a model, you need to get it out of Python and into a format a serving runtime can load. This sounds trivial. It isn't — the format you choose determines which serving frameworks you can use, how fast inference runs, and whether your model works cross-platform.

ONNX

ONNX (Open Neural Network Exchange) is the universal interchange format. Export from PyTorch or TensorFlow, run with ONNX Runtime, TensorRT, OpenVINO, or almost any serving framework. ONNX Runtime alone gets you 2-3× speedup over PyTorch eager mode through graph optimizations — operator fusion, constant folding, memory planning.

# Export PyTorch → ONNX
import torch
dummy = torch.randn(1, 3, 224, 224)
torch.onnx.export(model, dummy, "model.onnx",
                   opset_version=17,
                   input_names=["image"],
                   output_names=["prediction"],
                   dynamic_axes={"image": {0: "batch_size"}})

# Serve with ONNX Runtime
import onnxruntime as ort
session = ort.InferenceSession("model.onnx",
    providers=["CUDAExecutionProvider", "CPUExecutionProvider"])
result = session.run(None, {"image": input_array})

TorchScript

PyTorch's own serialization. Two flavors: torch.jit.trace (records operations from a sample input — fast but can't handle dynamic control flow) and torch.jit.script (parses your Python code into a static graph — handles control flow but not all Python). Use TorchScript when you're staying in the PyTorch ecosystem and need torch.compile or TorchServe.

TensorFlow SavedModel

If you're TensorFlow, this is your format. It's tightly integrated with TF Serving, TFLite (mobile), and TF.js (browser). No real reason to use it outside the TF ecosystem.

Serving Frameworks

Why Not Just FastAPI?

You can serve a model from a FastAPI endpoint. For a demo or low-traffic internal tool, it works. At scale, you need things a web framework doesn't give you: dynamic batching (collecting requests into GPU-efficient batches), multi-model management, model versioning, GPU memory management, and health monitoring. That's what dedicated model servers provide.

# FastAPI is fine for small-scale. Production needs more.
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI(title="Fraud Detection API", version="2.1.0")
model = load("models/fraud_detector_v2.joblib")

class Request(BaseModel):
    transaction_amount: float
    merchant_category: str
    hour_of_day: int

@app.post("/predict")
async def predict(req: Request):
    features = extract_features(req)
    prob = model.predict_proba(features)[0][1]
    return {"fraud_probability": round(prob, 4), "model_version": "2.1.0"}

@app.get("/health")
async def health():
    return {"status": "healthy", "model_loaded": model is not None}

Design rules for any serving API: include /health for load balancers, return model version with every prediction, validate inputs with Pydantic (return 422, not a crash), set timeouts.

REST vs. gRPC

REST (JSON over HTTP) is simple and debuggable — you can curl it. gRPC (Protocol Buffers, binary) is 2-5× faster for high-throughput internal services. Many production systems use both: gRPC between internal services, REST for external APIs, with an API gateway translating between them.

Triton's killer feature deserves expansion. If your NLP model uses 40% of a GPU, Triton schedules another model on the remaining 60%. In multi-model deployments, this can cut your GPU bill in half. No other serving framework does this as well.

# TorchServe handler — the interface between your model and the server
from ts.torch_handler.base_handler import BaseHandler
import torch

class ImageClassifier(BaseHandler):
    def preprocess(self, requests):
        images = []
        for req in requests:
            img = Image.open(io.BytesIO(req["body"]))
            images.append(self.transform(img))
        return torch.stack(images)

    def inference(self, inputs):
        with torch.no_grad():
            return self.model(inputs)

    def postprocess(self, outputs):
        probs = torch.softmax(outputs, dim=1)
        top_probs, top_classes = probs.topk(5)
        return [{self.mapping[str(c.item())]: p.item()
                 for p, c in zip(prob, cls)}
                for prob, cls in zip(top_probs, top_classes)]
# Package and deploy with TorchServe
torch-model-archiver --model-name resnet50 --version 1.0 \
  --model-file model.py --serialized-file resnet50.pth \
  --handler handler.py --export-path model_store/

torchserve --start --model-store model_store/ --models resnet50=resnet50.mar

Model Optimization for Production

Your model is accurate. Then someone asks "how much does it cost to serve?" and the room goes quiet. A transformer at 200ms per inference on a $2/hour GPU, at 100 req/s, is $50k/month. Your VP wants $5k. Now you need optimization.

Every optimization technique trades some accuracy for speed or size. Your job is finding the sweet spot where you hit your latency, throughput, and cost targets while staying within an acceptable accuracy budget.

Quantization

Reduce numerical precision. FP32 (4 bytes per weight) → FP16 (2 bytes) → INT8 (1 byte) → INT4 (0.5 bytes). Why does this work? Because a weight of 0.23847291 and 0.23828125 produce essentially the same prediction. The network learned rough patterns, not decimal places.

Post-Training Quantization (PTQ) — take a trained model, convert weights to lower precision, done. Needs a small calibration dataset (a few hundred samples). No retraining.

import torch
from torch.quantization import quantize_dynamic

model_fp32 = load_trained_model()
model_int8 = quantize_dynamic(model_fp32, {torch.nn.Linear}, dtype=torch.qint8)

# Check the damage
import os
torch.save(model_fp32.state_dict(), "model_fp32.pt")
torch.save(model_int8.state_dict(), "model_int8.pt")
fp32_size = os.path.getsize("model_fp32.pt") / 1e6
int8_size = os.path.getsize("model_int8.pt") / 1e6
print(f"FP32: {fp32_size:.1f} MB → INT8: {int8_size:.1f} MB ({fp32_size/int8_size:.1f}x)")

Quantization-Aware Training (QAT) — simulate quantization during training so the model learns to tolerate the precision loss. Recovers 50-100% of accuracy lost by PTQ. More effort, better results.

import torch.quantization as quant

model.train()
model.qconfig = quant.get_default_qat_qconfig("fbgemm")
model_prepared = quant.prepare_qat(model)

# Fine-tune with fake quantization baked in
for epoch in range(fine_tune_epochs):
    for batch in dataloader:
        loss = model_prepared(batch).loss
        loss.backward()
        optimizer.step()
        optimizer.zero_grad()

model_quantized = quant.convert(model_prepared)  # Now actually quantized

For LLMs specifically: GPTQ uses Hessian information to protect important weights during 4-bit quantization. AWQ protects the 1% of weights handling salient activations. GGUF is llama.cpp's format that lets you run a 7B model on a laptop CPU. Start with AWQ or GPTQ at 4-bit, benchmark, and you've likely cut serving costs 4-8×.

Practical default: For non-LLM models, INT8 PTQ is a no-brainer first step — almost free accuracy-wise, 4× size reduction. For LLMs, start with AWQ/GPTQ at 4-bit. Run your eval benchmarks before committing.

Pruning

Remove weights that don't matter. Research shows large networks are massively over-parameterized — you can cut 90% of weights in some models with barely any accuracy loss.

Unstructured pruning zeroes out individual weights anywhere. High compression, but creates sparse matrices that standard hardware can't exploit — you need sparse compute libraries for actual speedups.

Structured pruning removes entire neurons, channels, or attention heads. Lower compression ratio, but the result is a regular dense model that runs faster on any hardware. This is usually what you want.

import torch.nn.utils.prune as prune

# Structured: remove 30% of conv channels with smallest L2 norm
for name, module in model.named_modules():
    if isinstance(module, torch.nn.Conv2d):
        prune.ln_structured(module, name="weight", amount=0.3, n=2, dim=0)

# Make permanent
for name, module in model.named_modules():
    if hasattr(module, "weight_orig"):
        prune.remove(module, "weight")

The most effective recipe is iterative: prune 10-20%, retrain to recover accuracy, prune again. Each cycle the model adapts. This reaches much higher sparsity than one-shot pruning.

The sparsity trap: A 90% sparse model sounds impressive, but if your hardware can't exploit sparsity, you get zero speedup. NVIDIA Ampere/Hopper support 2:4 structured sparsity with dedicated hardware. Any other pattern needs software sparse-compute support. Always benchmark actual inference time, not parameter count.

Knowledge Distillation

Train a small "student" to mimic a large "teacher." Instead of hard labels (cat=1, dog=0), the student learns from the teacher's soft probability distribution (cat=0.85, dog=0.12, fox=0.03). Those soft labels carry rich inter-class structure the student could never learn from hard labels alone.

import torch.nn.functional as F

def distillation_loss(student_logits, teacher_logits, labels,
                      temperature=4.0, alpha=0.7):
    soft_teacher = F.softmax(teacher_logits / temperature, dim=-1)
    soft_student = F.log_softmax(student_logits / temperature, dim=-1)
    distill = F.kl_div(soft_student, soft_teacher, reduction="batchmean")
    distill *= temperature ** 2  # Scale to match gradient magnitude
    hard = F.cross_entropy(student_logits, labels)
    return alpha * distill + (1 - alpha) * hard

DistilBERT: 40% smaller, 60% faster, 97% of BERT's quality. TinyBERT: 7.5× compression via two-stage distillation. These aren't toys — they're running in production at scale.

Compilation & Runtime Optimization

These tools analyze your model as a graph and fuse operations. A sequence of MatMul→Add→ReLU becomes one kernel with one memory read/write instead of three. Memory bandwidth is usually the inference bottleneck, so this matters enormously.

Stack them. Distill → quantize to INT8 → run through TensorRT. A model that started at 2 GB / 200ms can end up at 50 MB / 5ms. That's 40× on both size and speed. Each technique is multiplicative with the others.

Containerization

"It works on my machine" is not a deployment strategy. ML models have wicked dependency chains — specific versions of PyTorch, CUDA, cuDNN, Python, and dozens of libraries. Docker packages your model, code, and all dependencies into a single artifact that runs identically everywhere.

# Production ML Dockerfile
FROM python:3.11-slim AS base

RUN apt-get update && apt-get install -y --no-install-recommends \
    libgomp1 && rm -rf /var/lib/apt/lists/*

# Dependencies first (cached layer — code changes don't reinstall packages)
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY models/ /app/models/
COPY src/ /app/src/
WORKDIR /app
EXPOSE 8080

HEALTHCHECK --interval=30s --timeout=10s --retries=3 \
    CMD curl -f http://localhost:8080/health || exit 1

CMD ["uvicorn", "src.server:app", "--host", "0.0.0.0", "--port", "8080"]

ML Docker rules:

  • Multi-stage builds — separate build env from runtime. Your final image doesn't need gcc.
  • Pin exact versions — torch==2.2.1, not torch>=2.0. Reproducibility matters.
  • Dependency layer first — COPY requirements.txt before code. Rebuilds after code changes skip package installation.
  • GPU images — start from nvidia/cuda base images for CUDA compatibility. Yes, they'll be 8GB+.
  • Slim base images — python:3.11-slim, not python:3.11. Save 600MB+ instantly.

Scaling Serving

Horizontal vs. Vertical

Vertical scaling: bigger GPU. Simple, limited ceiling. Horizontal scaling: more replicas behind a load balancer. This is how production systems scale — Kubernetes HPA (Horizontal Pod Autoscaler) adds replicas when traffic spikes and removes them during quiet hours.

Load Balancing for ML

Standard round-robin doesn't work well for ML. Not all requests are equally expensive — a 10-word text classifies in 10ms, a 10-page document takes 200ms. Round-robin sends three expensive requests to the same server while others sit idle.

Least-connections is better: route to the server with fewest active requests. It naturally adapts to varying request cost. For GPU serving, you may also need model-aware routing — sending requests to the GPU that actually has the right model loaded.

Health Checks and Circuit Breakers

Your health check should verify the model is loaded and responsive — not just that the process is alive. A circuit breaker stops sending traffic to a failing server after N consecutive failures, waits a cooling-off period, then sends a test request. If it passes, traffic resumes. This prevents one sick server from cascading failures across the whole system.

class CircuitBreaker:
    def __init__(self, failure_threshold=5, reset_timeout=60):
        self.failures = 0
        self.state = "closed"  # closed=normal, open=blocking
        self.last_failure = 0

    def call(self, func, *args, **kwargs):
        if self.state == "open":
            if time.time() - self.last_failure > self.reset_timeout:
                self.state = "half-open"
            else:
                raise Exception("Circuit open")

        try:
            result = func(*args, **kwargs)
            if self.state == "half-open":
                self.state = "closed"
                self.failures = 0
            return result
        except Exception:
            self.failures += 1
            self.last_failure = time.time()
            if self.failures >= self.failure_threshold:
                self.state = "open"
            raise

In production, use a service mesh (Istio, Linkerd) that gives you circuit breakers, retries, timeouts, and load balancing out of the box. Building these yourself is educational but error-prone.

Latency Optimization

In real-time serving, every millisecond counts. Here are the levers, roughly in order of impact:

Dynamic Batching

GPUs are massively parallel — processing 32 requests takes barely longer than processing 1. Collect requests over a short window (5ms), batch them, run inference once. Throughput goes way up at the cost of a tiny latency increase per request.

import asyncio
from collections import deque

class BatchingServer:
    def __init__(self, model, max_batch=32, max_wait_ms=5):
        self.model = model
        self.max_batch = max_batch
        self.max_wait_ms = max_wait_ms
        self.queue = deque()

    async def predict(self, input_data):
        future = asyncio.Future()
        self.queue.append((input_data, future))
        if len(self.queue) >= self.max_batch:
            self._process_batch()
        return await future

    def _process_batch(self):
        batch_inputs, futures = [], []
        while self.queue and len(batch_inputs) < self.max_batch:
            inp, fut = self.queue.popleft()
            batch_inputs.append(inp)
            futures.append(fut)
        results = self.model.predict_batch(batch_inputs)
        for fut, result in zip(futures, results):
            fut.set_result(result)

Model Warm-Up

The first request is always slow — model loads into memory, GPU initializes. Run a few dummy predictions at startup. This is one line of defensive code that prevents your first real user from seeing 10× normal latency.

Response Caching

If the same input appears frequently, cache the prediction. A Redis cache in front of your model server can absorb 30-60% of traffic for many workloads. Hash the input features, check cache, only hit the model on cache misses.

Async Inference

If the caller doesn't need the result immediately, accept the request, return a job ID, and process asynchronously. The client polls or gets a webhook callback. This decouples your API latency from model inference time — critical when models are slow or traffic is spiky.

Deployment Strategies

You've built the model, optimized it, containerized it. Now you need to get it in front of users without breaking things. The strategy you choose determines your blast radius when something goes wrong.

Canary

Route 5% of traffic to the new model, 95% to the old one. Monitor for errors, latency spikes, metric degradation. If OK, gradually ramp: 10%, 25%, 50%, 100%. If bad, instantly route everything back. Safest strategy — blast radius is 5% of users.

# Istio canary: 5% traffic to new model
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: model-serving
spec:
  hosts:
    - model-serving
  http:
    - route:
        - destination:
            host: model-serving
            subset: stable
          weight: 95
        - destination:
            host: model-serving
            subset: canary
          weight: 5

Blue-Green

Two identical environments. Blue runs the current model, green runs the new one. Flip the switch — all traffic moves at once. Simpler than canary (no traffic splitting), but the flip is all-or-nothing. Good when you need instant rollback capability.

Shadow (Dark Launch)

Route production traffic to both old and new models. Serve the old model's response. Log the new model's response for comparison. Zero user impact — you're testing the new model with real traffic without exposing anyone to its results. The tradeoff: you need to run both models simultaneously, doubling compute cost.

A/B Testing

Split users (not requests — same user must always see the same version) into control and treatment groups. Measure the impact on business metrics with statistical rigor.

import hashlib

def get_experiment_group(user_id: str, experiment: str):
    """Deterministic: same user always gets same group."""
    h = int(hashlib.sha256(f"{user_id}:{experiment}".encode()).hexdigest(), 16)
    return "treatment" if h % 100 < 10 else "control"  # 10% treatment

Critical distinction: canary tests whether the new model is safe (no errors, no latency spike). A/B tests whether it's better (improves business metrics). Canary is a deployment safeguard. A/B testing is a product experiment. Don't conflate them.

A/B tests need statistical rigor. Define your hypothesis and minimum detectable effect before the experiment. Calculate required sample size. Run for the full planned duration — don't peek and stop early when results look good (this inflates false positives). Use proper statistical tests. Account for multiple comparisons.
