Model Development

Chapter 13: ML Systems & Production Section 3 of 9
TL;DR

Model development is a loop: experiment, debug, measure, repeat — dozens to hundreds of times before you ship anything. The hard part isn't training. It's knowing what went wrong, whether your fix actually helped, and when to stop. This section covers experiment tracking (MLflow, W&B), systematic debugging (data bugs vs model bugs vs infra bugs), reproducibility, benchmarking that actually means something, error analysis at scale, and the discipline of knowing when good enough is good enough.

The Development Loop

Every model ships through the same cycle:

Experiment → Validate → Debug → Repeat → Ship
     ↑                              |
     └──────────────────────────────┘

The loop looks simple. It isn't. Most teams spend 80% of their time in "Debug → Repeat" and 20% actually training models. The bottleneck is never GPU hours — it's figuring out why the model is wrong and whether your hypothesis about fixing it is correct.

Here's the real workflow on a production team:

  1. Baseline first. Train the dumbest thing that could work. Logistic regression. A single decision tree. A pre-trained model with a linear head. This is your anchor — everything else gets measured against it.
  2. One change at a time. Change the features OR the model OR the hyperparameters. Never all three. You need to know what helped.
  3. Evaluate on what matters. Offline metrics first, then online if it passes the bar. More on this below.
  4. Ship or kill. If it beats the baseline by a meaningful margin, ship it. If not, document what you learned and move on.

The "document what you learned" step is where most teams fail. Without experiment tracking, you're just running on vibes.

Experiment Tracking

Monday: you train a model. Friday: you've run 47 experiments with different hyperparameters, feature sets, and preprocessing. One of them had great results. Which one? What learning rate? What data version?

If you can't answer that, you can't reproduce your best result. And if you can't reproduce it, you can't ship it.

What to Log (and What People Forget)

The obvious stuff: hyperparameters, metrics, model artifacts. Everyone logs those. Here's what separates production-grade tracking from notebook experiments:

  • Data version — which exact slice of data trained this model (hash or DVC version)
  • Code version — git commit SHA, not "I think I used the feature_v3 branch"
  • Environment — library versions, GPU model, CUDA version (results differ across GPU architectures)
  • Training wall time — because a 2% accuracy gain that costs 10x more GPU hours may not be worth it
  • Sample predictions — a few inputs + outputs at checkpoints, so you can spot-check qualitatively

MLflow — The Open-Source Default

Self-hosted, you own your data, integrates with the broader MLflow ecosystem (model registry, serving). The right choice for teams that care about data sovereignty or run in air-gapped environments.

import mlflow
import mlflow.sklearn
from sklearn.ensemble import RandomForestClassifier
from sklearn.metrics import f1_score, precision_score, recall_score

mlflow.set_experiment("churn-prediction")

with mlflow.start_run(run_name="rf_baseline_v2"):
    params = {
        "n_estimators": 200,
        "max_depth": 10,
        "min_samples_leaf": 5,
        "class_weight": "balanced",
        "data_version": "2024-01-15",
        "feature_set": "v3_with_engagement",
    }
    mlflow.log_params(params)

    model = RandomForestClassifier(**{k: v for k, v in params.items()
                                      if k in RandomForestClassifier().get_params()})
    model.fit(X_train, y_train)
    y_pred = model.predict(X_val)

    metrics = {
        "f1": f1_score(y_val, y_pred),
        "precision": precision_score(y_val, y_pred),
        "recall": recall_score(y_val, y_pred),

Debugging ML Models: The Systematic Approach

Debugging ML is different from debugging software. Software bugs crash or produce wrong output for a specific input. Model bugs produce slightly worse output — the loss goes down but not fast enough, accuracy is 85% when it should be 92%, or the model works on average but systematically fails on one subgroup. You can stare at code for hours and find nothing because the code is correct — the problem is in the data, the objective, or the gap between what you're measuring and what you actually care about.

First step: classify the bug.

The Three Bug Families

CategorySymptomsWhere to Look
Data bugsSuspiciously high or low performance, model fixates on one feature, performance varies wildly across subgroupsLabel noise, leakage, distribution shift, missing values encoded as 0
Model bugsLoss plateaus early, training loss doesn't decrease, val performance far below trainingWrong loss function, capacity mismatch, learning rate, architecture choice
Infrastructure bugsNon-reproducible results, NaN loss, OOM, different results on different machinesSeed management, GPU non-determinism, data loader ordering, mixed precision issues

Data bugs are the most common and the hardest to find. If your model achieves 99.5% accuracy on a problem where state-of-the-art is 92% — your first hypothesis should be leakage, not genius.

Data Leakage: The Silent Killer

Leakage means information from the test set (or from the future) bleeds into training. The model's evaluation looks fantastic, then production performance collapses. Classic patterns:

  • Target variable accidentally included as a feature (yes, this happens more than you'd think)
  • Future information leaking backward (using next month's revenue to predict this month's churn)
  • Same data point in both train and test sets (duplicate rows, same user appearing in both)

The detection heuristic: check feature importance. If a single feature dominates and it shouldn't — follow that thread.

Overfitting vs Underfitting — The Quick Read

Overfitting: training loss drops, val loss rises. The gap widens over time. Fix with regularization (dropout, weight decay, data augmentation), more data, smaller model, or early stopping.

Underfitting: both training and val performance are poor. Fix with more capacity, longer training, less regularization, better features, or a more expressive architecture.

Sometimes underfitting is the data's fault — there's not enough signal in your features. No architecture change fixes that.

Reading Training Curves Like an EKG

Plot training and val loss over time. They tell you everything:

  • Both plateau early → underfitting
  • Train drops, val doesn't follow → overfitting
  • Loss is noisy (jumping up and down) → learning rate too high or batch size too small
  • Loss drops to zero suspiciously fast → data leakage
  • NaN loss → numerical instability (check learning rate, gradient clipping, mixed precision)

Behavioral Testing

Standard evaluation measures average performance. Behavioral testing (from the CheckList framework) probes specific capabilities. This catches bugs that aggregate metrics hide.

# Behavioral tests for a sentiment classifier
invariance_tests = [
    # Changing names should NOT change sentiment
    ("I love this restaurant, John recommended it.", "positive"),
    ("I love this restaurant, Maria recommended it.", "positive"),
]

directional_tests = [
    # Adding "not" SHOULD flip sentiment
    ("The food was great.", "positive"),
    ("The food was not great.", "negative"),
]

for text, expected in invariance_tests + directional_tests:
    pred = model.predict(text)
    status = "PASS" if pred == expected else "FAIL"
    print(f"[{status}] '{text}' -> predicted={pred}, expected={expected}")

Write these tests before you think you have a good model. They become your regression suite.

Reproducibility

Same script, same data, same hyperparameters. Different results. This happens constantly in deep learning — random initialization, random shuffling, random dropout masks, non-deterministic GPU ops, different library versions. Full reproducibility requires controlling all of them.

import torch
import numpy as np
import random

def set_seed(seed=42):
    """Pin every source of randomness."""
    random.seed(seed)
    np.random.seed(seed)
    torch.manual_seed(seed)
    torch.cuda.manual_seed_all(seed)
    torch.backends.cudnn.deterministic = True
    torch.backends.cudnn.benchmark = False

set_seed(42)
# For full determinism (slower, some ops will error if no deterministic impl exists):
# torch.use_deterministic_algorithms(True)

Even with seeds pinned, some CUDA operations (like atomicAdd) are inherently non-deterministic because floating-point addition order depends on thread scheduling. torch.use_deterministic_algorithms(True) forces deterministic paths but some operations throw errors because no deterministic implementation exists. Trade-off: slower training for exact reproducibility.

The Reproducibility Checklist

For every experiment, record:

  1. Exact code version (git commit SHA)
  2. Exact data version (hash or DVC version)
  3. All hyperparameters
  4. Random seed
  5. Library versions (pinned in requirements.txt)
  6. Hardware (GPU model matters — results differ between architectures)

Experiment tracking tools automate most of this. DVC handles data versioning — it tracks data files alongside code, storing actual data in cloud storage while keeping lightweight metadata in Git. Check out a code version, DVC fetches the matching data version. Time travel for experiments.

Benchmarking Properly

A benchmark is only as useful as the question it answers. And most people are asking the wrong question.

Offline Metrics vs Online Metrics

Offline metrics (accuracy, F1, AUC on a held-out set) tell you whether the model learned something. Online metrics (click-through rate, revenue, user engagement in a live system) tell you whether it matters. These are different things.

A model that improves F1 by 3% might move revenue by 0.01%. Or it might move revenue by 5% because the improvement is concentrated on high-value predictions. You don't know until you measure online.

Metric TypeMeasuresWhen to UsePitfall
OfflineModel quality on held-out dataDevelopment iteration, quick feedbackDoesn't capture user behavior or system effects
OnlineBusiness impact in productionFinal validation before full rolloutExpensive, slow, needs traffic

Workflow: iterate fast on offline metrics. When offline metrics pass the bar, run an A/B test to measure online impact. Never skip the online step before a major launch.

Statistical Significance: Don't Lie to Yourself

Model A scores 0.847 F1. Model B scores 0.852 F1. Is B better?

Maybe. Maybe not. With a small test set, that difference could be noise. You need statistical testing.

import numpy as np
from scipy import stats

def paired_bootstrap_test(y_true, scores_a, scores_b, n_bootstrap=10000):
    """Bootstrap test: is the difference between two models significant?"""
    n = len(y_true)
    diff_observed = np.mean(scores_b) - np.mean(scores_a)

    count_greater = 0
    for _ in range(n_bootstrap):
        idx = np.random.choice(n, size=n, replace=True)
        diff_boot = np.mean(scores_b[idx]) - np.mean(scores_a[idx])
        if diff_boot <= 0:
            count_greater += 1

    p_value = count_greater / n_bootstrap
    return {"diff": diff_observed, "p_value": p_value,
            "significant": p_value < 0.05}

Rule of thumb: if you can't show p < 0.05, you can't claim the new model is better. This is especially important when reporting results to stakeholders — "the new model is 0.5% better" sounds great until you realize the confidence interval includes zero.

A/B Testing Basics

A/B testing is the gold standard for online evaluation. Split live traffic: some users get model A (control), others get model B (treatment). Measure the business metric you actually care about. Run long enough to reach statistical significance.

Common mistakes:

  • Peeking — checking results daily and stopping when p < 0.05. This inflates false positive rates massively. Decide your sample size upfront and stick to it.
  • Wrong randomization unit — randomizing by request instead of by user. The same user sees both models on different visits, contaminating the comparison.
  • Too short — some effects take days or weeks to manifest (e.g., recommendation quality affects retention, not just clicks).

System Benchmarks: Latency Matters Too

Model quality isn't the only thing to benchmark. A model that's 1% more accurate but 10x slower is usually the wrong trade-off.

import time
import numpy as np
import requests

def benchmark_latency(url, payload, num_requests=1000):
    """Measure serving latency distribution."""
    latencies = []
    for _ in range(num_requests):
        start = time.perf_counter()
        response = requests.post(url, json=payload)
        elapsed = (time.perf_counter() - start) * 1000  # ms
        latencies.append(elapsed)

    latencies = np.array(latencies)
    print(f"p50: {np.percentile(latencies, 50):.1f}ms")
    print(f"p95: {np.percentile(latencies, 95):.1f}ms")
    print(f"p99: {np.percentile(latencies, 99):.1f}ms")
    print(f"throughput: {num_requests / (sum(latencies) / 1000):.0f} req/s")

Always report p95 and p99, not just p50. The tail is where user experience goes to die.

Error Analysis at Scale

The single most valuable debugging technique. Most practitioners underinvest in it by an order of magnitude.

Confusion Matrices at Scale

For a 3-class problem, a confusion matrix is straightforward. For 1,000 classes — or an open-ended generation task — you need to think differently. Group errors into failure modes rather than individual class pairs.

import pandas as pd

errors = val_df[val_df["predicted"] != val_df["actual"]].copy()
print(f"Error rate: {len(errors)/len(val_df)*100:.1f}%")

# Which classes are hardest?
print("\nErrors by actual class (top 10):")
print(errors["actual"].value_counts().head(10))

# Slice-based evaluation: performance across subgroups
correct = val_df[val_df["predicted"] == val_df["actual"]]
for subgroup in ["mobile", "desktop", "tablet"]:
    subset = val_df[val_df["device"] == subgroup]
    acc = (subset["predicted"] == subset["actual"]).mean()
    print(f"  {subgroup}: {acc:.3f} ({len(subset)} examples)")

Slice-Based Evaluation

Aggregate metrics lie. A model with 95% overall accuracy can be 99% accurate on the majority class and 60% on a critical minority. Slice-based evaluation breaks performance down by meaningful subgroups — device type, user geography, input length, content category, time of day.

The slices that matter depend on your domain. For a content moderation model, slice by language. For a recommendation system, slice by user activity level. For a medical model, slice by demographic group. If you don't choose your slices, your failures choose them for you — usually in the form of an angry customer escalation.

Failure Mode Taxonomy

Instead of a flat list of errors, categorize failures into why the model got it wrong:

  • Ambiguous label — reasonable people would disagree on the ground truth
  • Insufficient context — the input simply doesn't contain enough information
  • Distribution gap — the input looks nothing like training data
  • Systematic bias — the model consistently fails on a pattern (e.g., negation, rare entities)
  • Noisy label — the ground truth is wrong

Each category has a different fix. Ambiguous labels need better annotation guidelines. Distribution gaps need more diverse training data. Systematic biases need targeted augmentation or architectural changes. Noisy labels need data cleaning. If you don't classify your errors, you apply the wrong fix — more data won't help a labeling problem.

When to Stop Iterating

This is the hardest skill in model development. Not the math, not the engineering — knowing when to stop.

Recognizing Diminishing Returns

Plot your improvement over time. The first week, you go from 70% to 88%. The second week, from 88% to 91%. The third week, you spend 60 GPU-hours to get from 91% to 91.3%. That's the signal.

Ask yourself: would my users notice this improvement? If the answer is "probably not," ship what you have and invest your time elsewhere — better features, faster inference, or an entirely different approach.

Good Enough for Production

"Good enough" isn't lazy — it's strategic. A model in production generating real feedback loops is worth more than a model on your laptop that's 2% better on a held-out set. Production models learn from user behavior, get retrained on fresh data, and compound improvements over time. The model sitting on your laptop does none of that.

Practical heuristics:

  • Beats the current system (even if the current system is just rules or heuristics)
  • Meets the latency/cost budget
  • No catastrophic failure modes on critical slices
  • Error analysis shows no easy wins left — remaining errors are genuinely hard cases

If all four are true: ship it. You can always iterate in production with A/B tests and retraining cycles. Perfect is the enemy of deployed.

What You Should Now Be Able To Do

  • Set up MLflow or W&B experiment tracking with proper logging of params, metrics, data versions, and code versions
  • Classify model bugs into data / model / infrastructure categories and know where to look for each
  • Detect data leakage by examining feature importance and suspiciously high performance
  • Read training curves and diagnose overfitting, underfitting, learning rate issues, and leakage from the shape of the loss
  • Write behavioral tests (invariance, directional) that catch bugs aggregate metrics miss
  • Pin all sources of randomness for reproducibility and explain why GPU non-determinism still leaks through
  • Distinguish offline metrics from online metrics and explain when each matters
  • Run a paired bootstrap test to determine if a model improvement is statistically significant
  • Perform slice-based error analysis and categorize failures into a taxonomy that maps to actionable fixes
  • Recognize diminishing returns and make the call to ship a "good enough" model to production
