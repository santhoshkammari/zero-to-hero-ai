Infrastructure & Cloud

Chapter 13: ML Systems & Production Section 7 of 9
TL;DR

Your model is only as good as the metal running it. GPUs vs TPUs, horizontal vs vertical scaling, managed vs DIY, AWS vs GCP vs Azure — these are the decisions that determine whether your ML system costs $500/month or $50,000/month. This section is the decision guide: which hardware for which job, which cloud platform for which team, and how to avoid lighting money on fire with idle GPU instances.

GPU / TPU Selection

The Hardware Landscape

Training and inference are fundamentally different workloads. Training is throughput-bound — you want maximum TFLOPS to chew through billions of tokens. Inference is latency-bound — you want fast time-to-first-token and high tokens-per-second per dollar. Picking hardware without knowing which problem you're solving is how teams end up paying H100 prices for a job an L4 could handle.

The Decision Logic

Training a model >7B parameters? You need A100 or H100. No shortcuts. The VRAM alone demands it, and the Tensor Core throughput makes training finish in days instead of weeks.

Fine-tuning a 7B model with LoRA? A single A100 or even an A10G with QLoRA. Don't overspend on H100s for adapter training.

Serving an LLM in production? L4s and Inferentia2 give you the best cost per token. T4s work for models under 7B with quantization.

Running on GCP and using JAX? TPUs are genuinely cheaper for large training runs. Their systolic array architecture is tailor-made for matrix multiplies. But you're locked into XLA compilation and Google's ecosystem.

# Quick VRAM estimation — the calculation you'll do a hundred times
def estimate_vram_gb(params_billions, precision="fp16", mode="inference"):
    bytes_per_param = {"fp32": 4, "fp16": 2, "bf16": 2, "int8": 1, "int4": 0.5}
    bp = bytes_per_param[precision]
    model_mem = params_billions * 1e9 * bp / (1024**3)

    if mode == "inference":
        # Model weights + KV cache overhead (~20-30% for typical batch)
        return model_mem * 1.3
    else:
        # Training: weights + gradients + optimizer states (Adam: 2x fp32)
        optimizer_mem = params_billions * 1e9 * 4 * 2 / (1024**3)
        grad_mem = model_mem
        activation_mem = model_mem  # rough estimate
        return model_mem + optimizer_mem + grad_mem + activation_mem

print(f"Llama 3 8B inference (INT4): {estimate_vram_gb(8, 'int4', 'inference'):.0f} GB")   # ~7 GB → fits on L4
print(f"Llama 3 8B training (BF16): {estimate_vram_gb(8, 'bf16', 'training'):.0f} GB")     # ~105 GB → multi-GPU
print(f"Llama 3 70B inference (INT4): {estimate_vram_gb(70, 'int4', 'inference'):.0f} GB")  # ~46 GB → A100 80GB
The VRAM wall is the real bottleneck. You'll almost always run out of GPU memory before you run out of compute. Every hardware decision starts with: "How much VRAM do I need?" Compute follows from there.

Scaling Strategies

Vertical vs. Horizontal

Vertical scaling means getting a bigger machine — more VRAM, more cores, faster interconnects. Going from an A10G to an A100, or from 1 GPU to 8 GPUs on the same node. It's simple, it avoids network overhead, and it's the right first move for most teams.

Horizontal scaling means adding more machines. Going from one 8×A100 node to four of them. This is where distributed training and serving get complex. You pay a communication tax — data has to flow between nodes over InfiniBand or (worse) Ethernet.

The rule of thumb:

ScaleWhenComplexityCommunication Cost
Vertical (bigger GPU)Model fits, need faster training/inferenceLowNone
Vertical (more GPUs, same node)Model doesn't fit on 1 GPU, or need more throughputMediumNVLink: 900 GB/s (H100)
Horizontal (multi-node)Model needs >8 GPUs, or serving requires >1 node throughputHighInfiniBand: 50–100 GB/s

Scaling Inference vs. Scaling Training

Inference scaling is usually straightforward horizontal scaling. Each replica is independent — stick a load balancer in front, add replicas as traffic grows. The challenge is auto-scaling: GPUs are expensive to keep idle but slow to cold-start (30–60 seconds to load a large model). Pre-warming, model caching, and smart scaling policies matter here.

Training scaling is fundamentally harder because GPUs have to communicate. You can't just add more machines independently — they all need to synchronize gradients. This is where the parallelism strategies from Ch 8 come in (data, tensor, and pipeline parallelism). The short version:

  • Data parallelism — replicate the model, split the data. Easy. Use DDP or FSDP.
  • Tensor parallelism — split individual layers across GPUs. Within a node only (needs NVLink bandwidth).
  • Pipeline parallelism — split layers across GPUs/nodes. Tolerates higher latency between nodes.

See Ch 8, Section 10: Distributed Training for the full treatment.

Distributed Training Recap

We covered distributed training in depth in Chapter 8. Here's what matters for infrastructure decisions:

# The most common pattern: FSDP for training large models
# Each GPU shards parameters, gradients, and optimizer states
from torch.distributed.fsdp import FullyShardedDataParallel as FSDP

model = FSDP(
    my_large_model,
    sharding_strategy=ShardingStrategy.FULL_SHARD,  # Shard everything
    mixed_precision=MixedPrecision(param_dtype=torch.bfloat16),
)
# Memory per GPU drops from N to N/num_gpus
# Trade-off: more communication, less memory

FSDP vs. DeepSpeed ZeRO — FSDP is native PyTorch, ZeRO is Microsoft's solution with more knobs (ZeRO-1/2/3, CPU offloading). Both solve the same problem: "model too big for one GPU." FSDP is simpler to start with; DeepSpeed gives more control for extreme-scale training.

Scaling efficiency rule: Doubling GPUs rarely halves training time. 75% efficiency (8 GPUs → 6× speedup) is good. Below 60%, you're burning money on communication overhead. Always benchmark before committing to larger clusters.

Interconnects determine your parallelism strategy. NVLink within a node (900 GB/s on H100) handles tensor parallelism. InfiniBand between nodes (50–100 GB/s) handles data and pipeline parallelism. If you're on Ethernet (~12.5 GB/s for 100GbE), stick to data parallelism only — anything else will bottleneck on communication.

AWS SageMaker — The Kitchen Sink

SageMaker has a service for everything. Training jobs abstract away cluster management — you specify an instance type and count, it handles provisioning and teardown. You pay only for training time, no idle GPUs.

# SageMaker training job — you define what, it handles where
from sagemaker.pytorch import PyTorch

estimator = PyTorch(
    entry_point="train.py",
    instance_count=4,
    instance_type="ml.p4d.24xlarge",       # 8×A100 per node → 32 GPUs total
    distribution={"torch_distributed": {"enabled": True}},
    hyperparameters={"epochs": 10, "lr": 1e-4},
)
estimator.fit({"train": "s3://my-bucket/data/"})
# Cluster spins up → trains → saves to S3 → tears down. Done.

The downside: SageMaker instances cost 20–40% more than raw EC2. For teams with infrastructure skills, raw EC2 + custom orchestration can be cheaper. For teams that want to focus on models, the premium buys you sanity.

GCP Vertex AI — The Clean API

Vertex AI's superpower is TPU access and the best generative AI integration. Model Garden gives you one-click fine-tuning and deployment of Gemini, Llama, Mistral, and other foundation models. If you're building GenAI applications, Vertex's integration here is hard to beat.

# Vertex AI training — TPU support is the differentiator
from google.cloud import aiplatform

job = aiplatform.CustomTrainingJob(
    display_name="train-on-tpus",
    script_path="train.py",
    container_uri="us-docker.pkg.dev/vertex-ai/training/pytorch-xla-tpu:latest",
)
model = job.run(
    machine_type="ct5lp-hightpu-4t",  # TPU v5e pod slice
    accelerator_type="TPU_V5_LITEPOD",
    accelerator_count=4,
)

Vertex AI Pipelines use Kubeflow Pipelines under the hood — open-source, somewhat portable, but heavier than SageMaker's SDK. The offline Feature Store runs on BigQuery, which is natural if your data already lives there.

Azure ML — The Enterprise Play

Azure ML's differentiator is governance. The Responsible AI Dashboard bundles fairness, interpretability, error analysis, and counterfactual explanations in one interface. Azure OpenAI Service gives you enterprise-grade GPT-4o access with data residency guarantees, VNet integration, and content filtering. For regulated industries, this matters.

# Azure ML training — workspace-centric model
from azure.ai.ml import MLClient, command

training_job = command(
    code="./src",
    command="python train.py --data ${{inputs.data}} --lr 1e-4",
    environment="AzureML-pytorch-2.1-cuda12@latest",
    compute="gpu-cluster",
    instance_count=2,
    distribution={"type": "PyTorch", "process_count_per_instance": 4},
)
ml_client.jobs.create_or_update(training_job)
The real answer: Pick based on where you already are. Data in S3 → SageMaker. GCP shop or TPU needs → Vertex AI. Enterprise on Azure → Azure ML. Cross-cloud ML is possible but adds complexity that's almost never worth the cost.

Managed vs. Self-Managed

This is the question behind every infrastructure decision: do you use the managed service, or build it yourself?

Managed (SageMaker, Vertex, etc.)Self-Managed (raw VMs + your code)
Setup timeMinutes to hoursDays to weeks
Operational burdenLow — they handle scaling, patching, monitoringHigh — you own everything
Cost per unitHigher (managed service premium)Lower (raw compute only)
FlexibilityConstrained by platform's opinionsComplete control
DebuggingBlack box when things go wrongFull visibility
Team neededML engineersML engineers + infra engineers

The honest answer for most teams: start managed, migrate components to self-managed as you hit pain points. SageMaker training jobs save you from building cluster orchestration. SageMaker endpoints save you from building serving infrastructure. But the moment their auto-scaling doesn't fit your traffic pattern or their container constraints block your model architecture — that's when you spin up your own stack.

The hybrid pattern that works: managed compute (let the cloud handle GPUs and scaling), self-managed orchestration (Airflow or Prefect for pipelines), and open-source tracking (MLflow, W&B) so you're not locked into any platform's experiment tracking.

Cost Optimization

GPU compute is the largest line item in any ML team's budget. Getting this wrong by 2× is common. Getting it wrong by 10× happens more than anyone admits.

Spot / Preemptible Instances

Cloud GPU instances at on-demand prices are expensive. An 8×H100 node runs $25+/hour — over $18,000/month if left running. Spot instances (AWS) and preemptible VMs (GCP) offer 60–90% discounts but can be reclaimed with 30 seconds to 2 minutes notice.

For training, spot instances are a no-brainer if you build for them:

# Checkpoint-and-resume pattern for spot training
import signal, torch, os

CHECKPOINT_DIR = "checkpoints/"

def save_checkpoint(model, optimizer, step):
    torch.save({
        "model": model.state_dict(),
        "optimizer": optimizer.state_dict(),
        "step": step,
    }, os.path.join(CHECKPOINT_DIR, f"ckpt_{step}.pt"))

def load_latest_checkpoint(model, optimizer):
    ckpts = sorted(os.listdir(CHECKPOINT_DIR))
    if not ckpts:
        return 0
    state = torch.load(os.path.join(CHECKPOINT_DIR, ckpts[-1]))
    model.load_state_dict(state["model"])
    optimizer.load_state_dict(state["optimizer"])
    return state["step"]

# Save every N steps — when spot reclaims, you lose at most N steps
for step, batch in enumerate(dataloader, start=resume_step):
    loss = train_step(model, batch)
    if step % 500 == 0:
        save_checkpoint(model, optimizer, step)

For inference, spot is riskier — a reclamation means dropped requests. Use spot for non-latency-sensitive batch inference; keep on-demand or reserved for real-time endpoints.

Reserved Capacity & Committed Use

If you know you'll use GPUs for 1–3 years, reserved instances (AWS) or committed use discounts (GCP) save 30–60% over on-demand. The catch: you pay whether you use them or not. Only commit for your baseline workload — the minimum compute you know you'll always need. Handle spikes with on-demand or spot.

Right-Sizing

The most common cost mistake isn't overpaying per GPU — it's using the wrong GPU entirely.

# Check GPU utilization — if it's consistently below 50%, you're overpaying
$ nvidia-smi --query-gpu=utilization.gpu,utilization.memory,memory.used,memory.total \
    --format=csv -l 5

# Typical findings:
# - GPU compute utilization: 30% → data loading bottleneck, not a GPU problem
# - Memory used: 12 GB / 80 GB → you're on an A100 when an L4 would do
# - Memory used: 78 GB / 80 GB → you're right-sized (or need more VRAM)

Profile first, then right-size. An L4 at $1/hr doing inference is better than an A100 at $4/hr sitting 70% idle.

The GPU-hours fallacy: Teams track cost per GPU-hour but ignore utilization. Running 8 A100s at 30% utilization costs the same as running them at 95% utilization — but delivers 3× less work. Track cost per useful TFLOP, not cost per GPU-hour.

Kubernetes for ML

Why K8s

Kubernetes isn't required for ML. If you're running a single training job on SageMaker and serving one model on a managed endpoint, K8s is overkill. But the moment you have multiple teams, multiple models, heterogeneous hardware, and complex scheduling requirements — K8s becomes the lingua franca.

What K8s gives you for ML:

  • GPU scheduling — request GPUs like any other resource; K8s handles placement
  • Multi-tenancy — teams share a GPU cluster with resource quotas and priorities
  • Reproducible environments — containers ensure training runs identically everywhere
  • Auto-scaling — scale serving pods based on queue depth or latency, not just CPU

Kubeflow

Kubeflow is the ML-specific layer on top of Kubernetes. It bundles pipeline orchestration (Kubeflow Pipelines), hyperparameter tuning (Katib), model serving (KServe), and notebook management. The value proposition: a complete ML platform on any K8s cluster — cloud or on-prem.

The catch: Kubeflow is complex. Installing, configuring, and maintaining it is a significant investment. It's best suited for teams that already have K8s expertise and need multi-cloud portability.

# KServe InferenceService — deploy a model on K8s
apiVersion: serving.kserve.io/v1beta1
kind: InferenceService
metadata:
  name: llama-8b-serving
spec:
  predictor:
    model:
      modelFormat:
        name: pytorch
      storageUri: "s3://models/llama-8b/"
      resources:
        limits:
          nvidia.com/gpu: "1"    # Request 1 GPU
        requests:
          memory: "32Gi"
  minReplicas: 1
  maxReplicas: 10                # Auto-scale up to 10 replicas

Ray

Ray is a different philosophy. Where Kubeflow builds on top of K8s abstractions, Ray provides its own distributed runtime. You write Python, Ray handles distribution across a cluster. Ray Train orchestrates distributed training. Ray Serve handles model serving with request batching and multi-model composition. Ray Data does distributed data preprocessing.

Ray can run on K8s (via KubeRay) or standalone. Its sweet spot: teams that want distributed computing without learning K8s primitives, or teams that need to compose multiple models into a single serving pipeline.

# Ray Serve — compose models in Python, not YAML
from ray import serve

@serve.deployment(num_replicas=2, ray_actor_options={"num_gpus": 1})
class LLMDeployment:
    def __init__(self):
        self.model = load_model("llama-8b")

    async def __call__(self, request):
        prompt = (await request.json())["prompt"]
        return self.model.generate(prompt, max_tokens=256)

app = LLMDeployment.bind()
serve.run(app, host="0.0.0.0", port=8000)

Multi-Cloud & Hybrid

When It Makes Sense

Most teams should not go multi-cloud for ML. The complexity tax is enormous: different APIs, different IAM models, different networking, data gravity pulling you toward one provider. But there are legitimate reasons:

  • Regulatory requirements — data residency laws may require specific regions on specific clouds
  • Hardware access — TPUs only exist on GCP; certain GPU types may have better availability on one provider
  • Vendor leverage — large organizations negotiate better pricing with multi-cloud commitments
  • Acquisition — you acquired a company on AWS and your stack is on GCP

The Hybrid Pattern That Works

Train where it's cheapest (spot GPUs on whatever provider has availability), store data where your other systems are (don't move petabytes of data between clouds), and serve where your users are (latency matters for inference).

The portability stack that makes this feasible:

# Cloud-agnostic ML stack
# Models:      ONNX format (runs anywhere)
# Tracking:    MLflow (self-hosted, cloud-agnostic)
# Orchestration: Airflow / Prefect (cloud-agnostic)
# Serving:     KServe on K8s (runs on any cloud)
# Containers:  Docker (universal packaging)
# Config:      Hydra / OmegaConf (code-level portability)

# The anti-pattern: using SageMaker Pipelines + SageMaker Feature Store +
# SageMaker Model Monitor + SageMaker Endpoints. You're locked in.
# Good when intentional. Painful when you need to move.
Data gravity wins. Your ML infrastructure will orbit your data. If you have 50 TB in S3, your training will be on AWS — transferring data between clouds costs $0.09/GB and takes forever. Make the cloud decision based on where your data lives. Everything else follows.

On-Prem GPU Clusters

On-prem makes economic sense at scale. If you're consistently using 100+ GPUs, buying hardware (even amortized over 3 years with maintenance) is cheaper than cloud. NVIDIA DGX systems, Dell PowerEdge, and Supermicro GPU servers are the usual suspects. The trade-off: you own capacity planning, maintenance, cooling, and the risk that the hardware you bought today is obsolete in 18 months.

The hybrid sweet spot: on-prem for steady-state workloads (your always-running inference fleet), cloud for burst capacity (large training runs, experiments, spike handling).

What You Should Now Be Able To Do

  • Estimate VRAM requirements for training and inference given a model size and precision, and pick the right GPU
  • Explain when to use vertical scaling (bigger/more GPUs on one node) vs. horizontal scaling (multi-node) and the communication trade-offs
  • Decide between FSDP and DeepSpeed ZeRO for distributed training based on your framework and scale
  • Compare AWS SageMaker, GCP Vertex AI, and Azure ML on their respective strengths and choose based on your existing stack
  • Make the managed vs. self-managed decision for training, serving, and orchestration independently — not as a blanket choice
  • Implement spot/preemptible training with robust checkpointing to cut training costs by 60–90%
  • Right-size GPU instances by profiling utilization, not guessing
  • Evaluate whether your team needs Kubernetes (Kubeflow/KServe) or Ray for ML orchestration, or whether managed services suffice
  • Design a cloud-agnostic ML stack using ONNX, MLflow, and containerized serving when portability is a requirement
  • Reason about data gravity and explain why multi-cloud ML is usually a bad idea unless you have a specific forcing function
