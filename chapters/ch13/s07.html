Model Serving

Chapter 13: ML Systems & Production 6 subtopics
TL;DR

Model serving is the infrastructure that puts your model in front of users. This section covers the full serving stack: designing REST and gRPC APIs, choosing model servers (TorchServe, Triton, TF Serving), containerizing models with Docker, deploying safely with canary and blue-green strategies, running A/B tests to prove your model actually helps, and load balancing to handle traffic at scale.

REST and gRPC APIs for ML

REST APIs

The most common way to serve an ML model is behind a REST API. A client sends an HTTP POST request with input data (as JSON), the server runs the model, and returns the prediction as JSON. REST is simple, widely understood, and works with every programming language and framework.

# Simple REST API for model serving with FastAPI
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
import numpy as np
from joblib import load

app = FastAPI(title="Fraud Detection API", version="2.1.0")
model = load("models/fraud_detector_v2.joblib")

class PredictionRequest(BaseModel):
    transaction_amount: float
    merchant_category: str
    hour_of_day: int
    num_transactions_1h: int
    is_international: bool

class PredictionResponse(BaseModel):
    fraud_probability: float
    is_fraud: bool
    model_version: str

@app.post("/predict", response_model=PredictionResponse)
async def predict(request: PredictionRequest):
    features = np.array([[
        request.transaction_amount,
        hash(request.merchant_category) % 100,  # simplified encoding
        request.hour_of_day,
        request.num_transactions_1h,
        int(request.is_international),
    ]])

    probability = model.predict_proba(features)[0][1]

    return PredictionResponse(
        fraud_probability=round(probability, 4),
        is_fraud=probability > 0.5,
        model_version="2.1.0",
    )

@app.get("/health")
async def health():
    return {"status": "healthy", "model_loaded": model is not None}

Design your API for production. Include a /health endpoint for load balancers. Return the model version with every prediction (so you can track which model produced each result). Use Pydantic or similar for input validation — a missing field should return a clear 422 error, not a cryptic model crash. Set timeouts — if the model hangs, the request should fail gracefully after a deadline.

gRPC for High-Throughput Serving

gRPC is a high-performance RPC framework that uses Protocol Buffers for serialization instead of JSON. It is binary (smaller payloads), supports streaming, and is significantly faster than REST for high-throughput, low-latency serving. The tradeoff: it is harder to debug (you cannot just curl the endpoint) and requires generated client code.

# Proto definition for model serving
# model_service.proto
syntax = "proto3";

service ModelService {
    rpc Predict (PredictRequest) returns (PredictResponse);
    rpc PredictBatch (PredictBatchRequest) returns (PredictBatchResponse);
    rpc StreamPredict (stream PredictRequest) returns (stream PredictResponse);
}

message PredictRequest {
    repeated float features = 1;
    string request_id = 2;
}

message PredictResponse {
    float score = 1;
    string label = 2;
    string model_version = 3;
    float latency_ms = 4;
}

Use REST when: you need simplicity, external clients will call the API, or debugging ease matters. Use gRPC when: you need maximum throughput, the client and server are internal services, or you need streaming predictions. Many production systems use both — gRPC between internal services and REST for external-facing APIs, with an API gateway that translates between the two.

Model Servers

Why Not Just Flask?

You can serve a model from a Flask or FastAPI endpoint, and for small-scale deployments, that works fine. But at scale, you need features that a basic web framework does not provide: dynamic batching (collecting multiple requests into a single GPU batch), multi-model management (loading and unloading models without downtime), model versioning (serving multiple versions simultaneously), GPU memory management, and health monitoring.

This is what model servers provide. They are specialized HTTP/gRPC servers designed for ML inference.

TorchServe

TorchServe is the official serving solution for PyTorch models. You package your model with a handler (a Python class that defines preprocessing, inference, and postprocessing) into a .mar (Model Archive) file, then deploy it to TorchServe. It supports dynamic batching, multi-model serving, and A/B testing out of the box.

# TorchServe handler example
# handler.py
import torch
from torchvision import transforms
from ts.torch_handler.base_handler import BaseHandler

class ImageClassifier(BaseHandler):
    def __init__(self):
        super().__init__()
        self.transform = transforms.Compose([
            transforms.Resize(256),
            transforms.CenterCrop(224),
            transforms.ToTensor(),
            transforms.Normalize([0.485, 0.456, 0.406], [0.229, 0.224, 0.225]),
        ])

    def preprocess(self, requests):
        """Convert raw images to tensors."""
        images = []
        for request in requests:
            image = Image.open(io.BytesIO(request["body"]))
            images.append(self.transform(image))
        return torch.stack(images)

    def inference(self, inputs):
        """Run model forward pass."""
        with torch.no_grad():
            outputs = self.model(inputs)
        return outputs

    def postprocess(self, outputs):
        """Convert logits to predictions."""
        probs = torch.softmax(outputs, dim=1)
        top_probs, top_classes = probs.topk(5)
        results = []
        for prob, cls in zip(top_probs, top_classes):
            results.append({
                self.mapping[str(c.item())]: p.item()
                for p, c in zip(prob, cls)
            })
        return results

#   --handler handler.py --export-path model_store/

NVIDIA Triton Inference Server

Triton is NVIDIA's model server, and it is the most feature-rich option. It supports virtually every ML framework (PyTorch, TensorFlow, ONNX, TensorRT, scikit-learn via custom backends), dynamic batching, model ensembles (chaining multiple models in a pipeline), concurrent model execution on the same GPU, and model versioning with automatic rollback.

Triton's killer feature is concurrent model execution. A single GPU can run multiple models simultaneously, filling GPU utilization gaps. If your NLP model uses 40% of the GPU's compute capacity, Triton can schedule another model on the remaining 60%. This dramatically improves hardware utilization in multi-model deployments.

TensorFlow Serving

TensorFlow Serving is the original production model server, designed specifically for TensorFlow SavedModel format. It is mature, well-tested, and tightly integrated with the TensorFlow ecosystem. If you are a TensorFlow shop, it is the natural choice. It supports batching, model versioning, and gRPC and REST APIs. The downside: it only serves TensorFlow models (with limited ONNX support).

BentoML

BentoML takes a different approach — it is a framework for packaging models from any framework into production-ready serving containers. You write a service definition in Python, and BentoML handles containerization, API generation, and deployment to Kubernetes, AWS Lambda, or other platforms. It is particularly good for teams that use multiple frameworks and want a unified serving pipeline.

Containerization with Docker

Why Containers Matter for ML

ML models have complex dependencies — specific versions of PyTorch, CUDA, cuDNN, Python, and dozens of libraries. "It works on my machine" is not a valid deployment strategy. Docker containers package your model, code, and all dependencies into a single, reproducible artifact that runs identically everywhere — on your laptop, in CI/CD, and in production.

# Dockerfile for ML model serving
FROM python:3.11-slim as base

# Install system dependencies
RUN apt-get update && apt-get install -y --no-install-recommends \
    libgomp1 \
    && rm -rf /var/lib/apt/lists/*

# Install Python dependencies (separate layer for caching)
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Copy model and serving code
COPY models/ /app/models/
COPY src/ /app/src/

WORKDIR /app
EXPOSE 8080

# Health check
HEALTHCHECK --interval=30s --timeout=10s --retries=3 \
    CMD curl -f http://localhost:8080/health || exit 1

# Run the server
CMD ["uvicorn", "src.server:app", "--host", "0.0.0.0", "--port", "8080"]

Best Practices for ML Docker Images

ML images tend to be huge — a PyTorch + CUDA image can easily be 8GB+. Use multi-stage builds to separate the build environment from the runtime. Pin exact library versions. Use slim base images. Separate dependency installation from code copying so Docker caches the dependency layer — rebuilding after a code change should not reinstall all packages. For GPU models, start from NVIDIA's base images (nvidia/cuda) to ensure CUDA compatibility.

Canary, Blue-Green, and A/B Testing

Canary Deployment

A canary deployment routes a small percentage of traffic (say, 5%) to the new model while the remaining 95% continues using the current model. You monitor the canary for errors, latency spikes, and metric degradation. If everything looks good, you gradually increase the canary's traffic share (10%, 25%, 50%, 100%). If something goes wrong, you instantly route all traffic back to the old model.

Canary is the safest deployment strategy. The blast radius of a bad model is limited to 5% of users. The tradeoff: it takes longer to fully deploy, and you need infrastructure to split traffic and monitor both versions.

# Kubernetes canary deployment with Istio
# VirtualService that splits traffic between versions
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: model-serving
spec:
  hosts:
    - model-serving
  http:
    - route:
        - destination:
            host: model-serving
            subset: stable    # current model
          weight: 95
        - destination:
            host: model-serving
            subset: canary    # new model
          weight: 5

Blue-Green Deployment

In blue-green deployment, you maintain two identical environments. "Blue" runs the current model. "Green" runs the new model. All traffic goes to blue. When green is ready and tested, you flip the switch — all traffic moves to green instantly. If green fails, flip back to blue. The advantage over canary: no partial traffic splitting, simpler monitoring. The disadvantage: when you flip, all users are affected at once.

A/B Testing

A/B testing is an experiment, not just a deployment strategy. You split users into groups (not requests — the same user should always see the same model version to avoid confusion). Group A gets the current model, group B gets the new model. You measure the impact on business metrics (conversion rate, engagement, revenue) with statistical rigor.

The critical difference from canary: canary tests whether the new model is safe (no errors, no latency spike). A/B testing tests whether the new model is better (improves business metrics). Canary is a deployment safeguard. A/B testing is a product experiment.

# A/B test assignment and logging
import hashlib

def get_experiment_group(user_id: str, experiment: str, groups=("control", "treatment")):
    """Deterministic assignment: same user always gets same group."""
    hash_input = f"{user_id}:{experiment}"
    hash_val = int(hashlib.sha256(hash_input.encode()).hexdigest(), 16)
    group_index = hash_val % len(groups)
    return groups[group_index]

def serve_prediction(user_id: str, input_data):
    group = get_experiment_group(user_id, "model_v3_experiment")

    if group == "control":
        prediction = model_v2.predict(input_data)
    else:
        prediction = model_v3.predict(input_data)

    # Log for analysis
    log_event({
        "user_id": user_id,
        "experiment": "model_v3_experiment",
        "group": group,
        "prediction": prediction,
        "timestamp": now(),
    })

    return prediction

A/B tests need statistical rigor. Define your hypothesis and minimum detectable effect before the experiment starts. Calculate the required sample size. Run the experiment for the full duration — do not peek at results and stop early when they look good (this inflates false positive rates). Use proper statistical tests (t-test, chi-squared, or Bayesian methods). Account for multiple comparisons if you test multiple metrics.

Load Balancing

Why ML Serving Needs Special Load Balancing

Standard load balancing distributes requests evenly across servers. ML serving has a complication: not all requests are equally expensive. A short text takes 10ms to classify. A long document takes 200ms. If a round-robin load balancer sends three long documents to the same server, that server gets overloaded while others are idle.

Least-connections load balancing routes each request to the server with the fewest active requests. This naturally adapts to varying request complexity — a server processing a slow request gets fewer new requests until it finishes. For ML serving, least-connections generally outperforms round-robin.

GPU-Aware Load Balancing

When serving on GPUs, load balancing becomes even more nuanced. GPU memory is limited — you might only fit one model on a GPU. If you serve multiple models, you need to route requests to the GPU that has the right model loaded. Some advanced serving setups use model multiplexing — swapping models in and out of GPU memory based on request patterns — but this adds latency during swaps.

Health Checks and Circuit Breakers

Load balancers should continuously health-check model servers. A health check verifies that the server is up and the model is loaded (not just that the process is running). If a server becomes unhealthy (model crash, GPU memory error, hang), the load balancer stops routing traffic to it.

A circuit breaker goes further: if a server fails repeatedly, the circuit breaker "opens" and stops sending requests for a cooling-off period. After the period, it sends a test request. If it succeeds, the circuit breaker "closes" and normal traffic resumes. This prevents cascading failures where a failing server drags down the entire system.

# Circuit breaker implementation
import time

class CircuitBreaker:
    def __init__(self, failure_threshold=5, reset_timeout=60):
        self.failure_threshold = failure_threshold
        self.reset_timeout = reset_timeout
        self.failures = 0
        self.state = "closed"  # closed = normal, open = blocking
        self.last_failure_time = 0

    def call(self, func, *args, **kwargs):
        if self.state == "open":
            if time.time() - self.last_failure_time > self.reset_timeout:
                self.state = "half-open"  # try one request
            else:
                raise Exception("Circuit breaker is open")

        try:
            result = func(*args, **kwargs)
            if self.state == "half-open":
                self.state = "closed"
                self.failures = 0
            return result
        except Exception as e:
            self.failures += 1
            self.last_failure_time = time.time()
            if self.failures >= self.failure_threshold:
                self.state = "open"
            raise

In production, use a service mesh like Istio or Linkerd that provides circuit breakers, retries, timeouts, and load balancing out of the box. Building these patterns yourself is educational but error-prone. Let proven infrastructure handle the networking complexity so your team can focus on the ML.
