Data Infrastructure

Chapter 13: ML Systems & Production Section 2 of 9
TL;DR

Your model is a function of your data. Every production ML failure I have seen traces back to data — stale features, silent schema changes, training-serving skew, label noise. This section covers the full data stack: pipelines (batch and streaming), feature stores, data versioning, labeling and augmentation at scale, data quality gates, and storage architecture. If you nail data infrastructure, the modeling part is almost boring.

Data Pipelines for ML

ETL vs ELT — The One Distinction That Matters

ETL (Extract, Transform, Load) cleans data before storing it. ELT (Extract, Load, Transform) dumps raw data first, transforms in place. The difference sounds academic until you realize ELT means you never have to re-extract from production databases when you discover a new feature you need six months from now.

Storage is cheap. Re-extraction from production systems is not. Most modern ML teams use ELT.

ETLELT
Transform happensBefore loadingAfter loading (in warehouse)
Raw data retained?No — only cleaned versionYes — always available
Best whenPII must be stripped before storageYou need iterative feature engineering
ToolsCustom scripts, Informaticadbt + BigQuery/Snowflake, Spark
ML team preferenceRareAlmost always
 
# ELT in practice: land raw, transform with dbt or Spark
# Step 1: Land raw data (the "EL" part)
raw_events = spark.read.json("s3://raw-events/2024/01/15/")
raw_events.write.parquet("s3://data-lake/raw/events/2024-01-15/")

# Step 2: Transform in place (the "T" part) — weeks later, as needed
events = spark.read.parquet("s3://data-lake/raw/events/2024-01-15/")
user_features = (
    events
    .filter(F.col("event_type") == "purchase")
    .groupBy("user_id")
    .agg(
        F.count("*").alias("purchases_30d"),
        F.avg("amount_usd").alias("avg_order_value"),
    )
)
user_features.write.parquet("s3://features/user_purchase_stats/")

Batch vs Streaming

Batch pipelines run on a schedule — hourly, daily, weekly. They process a chunk of data, write the results, and shut down. Streaming pipelines run continuously, processing events as they arrive. The question is not which is “better.” It is: how fresh do your features need to be?

A fraud detection model needs features computed in the last 5 seconds. A churn prediction model is fine with yesterday’s data. Match the pipeline type to the latency requirement, not to hype.

BatchStreaming
LatencyMinutes to hoursSeconds to milliseconds
ComplexityLow — runs, finishes, doneHigh — backpressure, out-of-order events, checkpointing
CostPay for compute only when runningAlways-on infrastructure
DebuggingRerun the jobGood luck replaying a Kafka topic
ToolsSpark, dbt, SQLFlink, Beam on Dataflow, Kafka Streams

The pragmatic play: start batch. Add streaming only for features where latency genuinely matters. Most teams overestimate how many features need to be real-time.

Spark and Beam — The Workhorses

Apache Spark is the default for batch. You write DataFrame transforms that feel like Pandas, and Spark parallelizes across your cluster. The catch: Spark is lazy — errors don’t surface until you trigger an action like .write() or .count(). Always test on a sample first.

from pyspark.sql import SparkSession, functions as F

spark = SparkSession.builder.appName("feature_pipeline").getOrCreate()

events = spark.read.parquet("s3://data-lake/events/2024/01/")
user_features = (
    events
    .filter(F.col("event_type") == "page_view")
    .groupBy("user_id")
    .agg(
        F.count("*").alias("page_views_30d"),
        F.countDistinct("session_id").alias("sessions_30d"),
        F.avg("session_duration_sec").alias("avg_session_duration"),
        F.max("timestamp").alias("last_active"),
    )
)
user_features.write.mode("overwrite").parquet("s3://features/user_engagement/")

Apache Beam gives you one API for both batch and streaming. Write a pipeline once, run it on Dataflow, Flink, or Spark. Beam’s real power is windowing — grouping events into time windows for streaming aggregations.

import apache_beam as beam

with beam.Pipeline() as p:
    (
        p
        | "Read" >> beam.io.ReadFromPubSub(topic="projects/myproj/topics/events")
        | "Parse" >> beam.Map(parse_event_json)
        | "Window" >> beam.WindowInto(beam.window.FixedWindows(300))  # 5-min windows
        | "Key" >> beam.Map(lambda e: (e["user_id"], e))
        | "Aggregate" >> beam.CombinePerKey(compute_features)
        | "Write" >> beam.io.WriteToBigQuery("project:dataset.features_5min")
    )

Use Spark for batch. Use Beam/Flink when you genuinely need streaming. Don’t use streaming to look cool on your architecture diagram.

Orchestration: Airflow vs Prefect

Production pipelines have dependencies. Feature computation waits for data ingestion. Training waits for features. If ingestion fails at 3 AM, everything downstream should halt and someone should get paged.

Airflow is the industry standard — DAG-based, battle-tested, massive ecosystem. Also: painful scheduler, clunky dynamic workflows, everything serializes through XCom or cloud storage.

Prefect (and Dagster) are the modern alternatives — pure Python decorators, easier testing, better dynamic workflows. For greenfield projects, these are usually the better choice.

# Prefect: feels like normal Python
from prefect import flow, task

@task(retries=2, retry_delay_seconds=600)
def ingest(date: str):
    return fetch_and_land(date)

@task
def compute_features(raw):
    features = build_features(raw)
    validate(features)
    return features

@task
def train_and_gate(features):
    model = train(features)
    metrics = evaluate(model, features)
    if metrics["auc"] < 0.75:
        raise ValueError(f"AUC {metrics['auc']} below gate")
    return model

@flow(name="daily-ml-pipeline")
def daily_pipeline(date: str):
    raw = ingest(date)
    features = compute_features(raw)
    model = train_and_gate(features)

Pick one. Be consistent. The specific orchestrator matters less than having any orchestrator. Cron jobs with bash scripts are not a pipeline — they’re a time bomb.

Feature Stores

The Problem They Solve

Two teams compute “user_avg_session_duration” independently. One uses a 30-day window, the other uses 7 days. Neither documents it. The model trains on one definition and serves with the other. Performance is silently terrible, and nobody knows why for three months.

That’s training-serving skew, and it’s the most insidious bug in production ML. Feature stores kill it by providing:

  • A single source of truth for feature definitions
  • An offline store (data warehouse) for historical features during training
  • An online store (Redis/DynamoDB) for low-latency features during serving
  • Point-in-time correctness — features as they were when the event happened, not as they are now

That last one is critical. If a user churned on January 15, you need their features from January 14. Using today’s features to train on historical events is data leakage.

Feast in Practice

Feast is the most popular open-source feature store. Define features in code, materialize to online/offline stores, fetch with the same API for training and serving.

# feature_repo/features.py
from feast import Entity, FeatureView, Field, FileSource
from feast.types import Float32, Int64
from datetime import timedelta

user = Entity(name="user_id", join_keys=["user_id"])

user_stats_source = FileSource(
    path="data/user_stats.parquet",
    timestamp_field="event_timestamp",
)

user_stats_fv = FeatureView(
    name="user_stats",
    entities=[user],
    ttl=timedelta(days=1),
    schema=[
        Field(name="login_count_7d", dtype=Int64),
        Field(name="avg_session_duration", dtype=Float32),
        Field(name="total_spend_30d", dtype=Float32),
        Field(name="support_tickets_30d", dtype=Int64),
    ],
    source=user_stats_source,
)

# Training: get historical features (point-in-time correct)
training_df = store.get_historical_features(
    entity_df=entity_df_with_timestamps,
    features=["user_stats:login_count_7d", "user_stats:total_spend_30d"],
).to_df()

# Serving: get latest features (low latency)
online_features = store.get_online_features(
    features=["user_stats:login_count_7d", "user_stats:total_spend_30d"],
    entity_rows=[{"user_id": "u123"}],
).to_dict()

Same feature definitions. Same code. No skew. That’s the whole point.

Feature freshness — how stale your online features are — depends on materialization frequency. Batch features might be hours old. Streaming features can be seconds old. Choose based on your actual latency requirements, not aspirational ones.

Data Versioning

You version your code with Git. You should version your data too.

Here’s why: your model’s performance depends on the exact dataset it trained on. Six months from now, when accuracy drops, you need to answer: “Did the model change, or did the data change?” Without versioned data, you can’t reproduce any previous result. You’re flying blind.

DVC (Data Version Control)

DVC wraps Git. It tracks large files (datasets, models) by storing lightweight pointer files in Git and the actual data in remote storage (S3, GCS, Azure Blob). You get Git-like semantics — branches, tags, diffs — for datasets.

# Initialize DVC in your Git repo
dvc init
dvc remote add -d myremote s3://my-bucket/dvc-store

# Track a dataset
dvc add data/training_v2.parquet
git add data/training_v2.parquet.dvc data/.gitignore
git commit -m "Training data v2: added Q1 labels"
git tag data-v2

# Reproduce a previous experiment
git checkout data-v1
dvc checkout     # pulls the exact dataset from that commit
python train.py  # guaranteed same data as last time

lakeFS — Git for Data Lakes

lakeFS takes a different approach: it gives your data lake Git-like branching. Create a branch, write experimental data to it, validate, merge to main. Your production pipeline never sees half-written data.

# lakeFS: branch your data lake like code
import lakefs_client

# Create a branch for a new data pipeline
client.branches.create_branch(
    repository="ml-data",
    branch_creation={"name": "feature/new-labels", "source": "main"}
)

# Write new data to the branch (production is untouched)
upload_new_labels("lakefs://ml-data/feature/new-labels/labels/v3/")

# Validate, then merge
run_validation("lakefs://ml-data/feature/new-labels/labels/v3/")
client.refs.merge_into_branch(
    repository="ml-data",
    source_ref="feature/new-labels",
    destination_branch="main"
)

Training Data Management

Labeling Pipelines

Supervised learning needs labels. At scale, this means a system, not a spreadsheet. You need task distribution, quality control, inter-annotator agreement measurement, and versioned guidelines that evolve as you discover edge cases.

Here’s the workflow that actually works:

  1. Write labeling guidelines with examples — especially edge cases (“Is a bratwurst in a bun a hot dog?”)
  2. Pilot with 3+ annotators on 200 examples. Measure Cohen’s kappa. Below 0.8? Guidelines are ambiguous. Fix them.
  3. Embed gold questions — known-answer items sprinkled into the task stream to catch random clicking
  4. Adjudicate disagreements — a senior reviewer breaks ties
  5. Version everything — which guidelines produced which labels, so you can retrain when guidelines change
# Label Studio: programmatic project setup
from label_studio_sdk import Client

ls = Client(url="http://localhost:8080", api_key="your-api-key")

project = ls.start_project(
    title="Product Image Classification v3",
    label_config='''
    <View>
      <Image name="image" value="$image"/>
      <Choices name="category" toName="image" choice="single-required">
        <Choice value="electronics"/>
        <Choice value="clothing"/>
        <Choice value="food"/>
        <Choice value="other"/>
      </Choices>
    </View>
    '''
)

project.import_tasks([
    {"image": "s3://data/images/img_001.jpg"},
    {"image": "s3://data/images/img_002.jpg"},
])

Tools: Label Studio (open source, self-hosted), Labelbox (managed, enterprise), Scale AI (fully outsourced), Prodigy (active-learning-first, by the spaCy team).

Active Learning: Label Smarter, Not More

Not all unlabeled examples are equally useful. The model already knows that an obvious cat is a cat. It’s the blurry, ambiguous images that teach it something. Active learning asks the model: “Which examples would help you most?” — then sends those to annotators.

import numpy as np
from sklearn.linear_model import LogisticRegression

def select_for_labeling(labeled_X, labeled_y, unlabeled_X, budget=100):
    model = LogisticRegression().fit(labeled_X, labeled_y)
    probs = model.predict_proba(unlabeled_X)

    # Uncertainty = 1 - max(predicted class probability)
    uncertainty = 1 - np.max(probs, axis=1)

    # Send the most uncertain examples to annotators
    return np.argsort(uncertainty)[-budget:]

Common strategies: uncertainty sampling (simplest, usually good enough), query-by-committee (train multiple models, pick where they disagree), diversity sampling (spread across feature space to avoid labeling 100 near-identical items).

Pitfall: active learning has a cold-start problem. If your initial model is garbage, it selects garbage examples. Bootstrap with ~200 random labels before switching to active selection. And periodically add random samples to counteract the bias toward decision boundaries.

Weak Supervision: Labels from Heuristics

What if domain experts could write rules instead of labeling thousands of examples? Each rule is noisy — some get it wrong — but combine enough noisy rules cleverly and you get surprisingly good labels. That’s Snorkel.

from snorkel.labeling import labeling_function, PandasLFApplier, LFAnalysis
from snorkel.labeling.model import LabelModel

SPAM, NOT_SPAM, ABSTAIN = 1, 0, -1

@labeling_function()
def lf_contains_free(x):
    return SPAM if "free" in x.text.lower() else ABSTAIN

@labeling_function()
def lf_short_email(x):
    return NOT_SPAM if len(x.text.split()) < 10 else ABSTAIN

@labeling_function()
def lf_known_sender(x):
    return NOT_SPAM if x.sender in known_contacts else ABSTAIN

@labeling_function()
def lf_all_caps_subject(x):
    return SPAM if x.subject == x.subject.upper() and len(x.subject) > 5 else ABSTAIN

# Apply, analyze, combine
lfs = [lf_contains_free, lf_short_email, lf_known_sender, lf_all_caps_subject]
L_train = PandasLFApplier(lfs=lfs).apply(df=train_df)

label_model = LabelModel(cardinality=2, verbose=True)
label_model.fit(L_train=L_train, n_epochs=500, lr=0.001)
prob_labels = label_model.predict_proba(L=L_train)  # soft labels

Snorkel’s label model learns which labeling functions are accurate and which are correlated — without any ground-truth labels. The output is probabilistic labels you can train on directly. You still need a small manually-labeled validation set to check your work.

Data Augmentation at Scale

Augmentation creates “new” training examples by transforming existing ones. Flip a cat image → still a cat. Essentially free labeled data.

Vision has the richest toolkit:

import albumentations as A
from albumentations.pytorch import ToTensorV2

train_transform = A.Compose([
    A.RandomResizedCrop(height=224, width=224, scale=(0.8, 1.0)),
    A.HorizontalFlip(p=0.5),
    A.ColorJitter(brightness=0.2, contrast=0.2, saturation=0.2, hue=0.1, p=0.8),
    A.CoarseDropout(max_holes=8, max_height=32, max_width=32, p=0.5),
    A.Normalize(mean=[0.485, 0.456, 0.406], std=[0.229, 0.224, 0.225]),
    ToTensorV2(),
])

# RandAugment: two knobs instead of a dozen
# N = number of transforms, M = magnitude. That's it.
simple_transform = A.Compose([
    A.RandAugment(num_ops=2, magnitude=9),
    A.Normalize(mean=[0.485, 0.456, 0.406], std=[0.229, 0.224, 0.225]),
    ToTensorV2(),
])

Text is harder — small changes can flip meaning. Safe options: back-translation, synonym replacement, LLM paraphrasing (prompt: “Rephrase while preserving meaning”). Always verify labels are preserved.

Audio: SpecAugment (mask frequency bands and time steps in spectrograms), time stretching, pitch shifting, background noise injection.

The cardinal rule: never augment validation or test sets. And validate that your augmentation actually improves metrics — rotating a “6” by 180° makes it a “9”. Context matters.

Data Quality

Here’s the thing about data bugs: they’re silent. The pipeline runs. The model trains. No errors. Three months later, someone notices conversion rates tanked because an upstream schema change turned a numeric column into strings. Your model was happily training on garbage the whole time.

Data validation is not optional. It’s a quality gate in every pipeline.

Great Expectations

The most popular data validation framework. You define expectations in code, and the tool checks them on every pipeline run. Failed expectations halt the pipeline before bad data reaches your model.

import great_expectations as gx

context = gx.get_context()
datasource = context.sources.add_pandas("my_data")
asset = datasource.add_dataframe_asset("daily_features")

suite = context.add_expectation_suite("feature_quality")
suite.add_expectation(
    gx.expectations.ExpectColumnValuesToNotBeNull(column="user_id")
)
suite.add_expectation(
    gx.expectations.ExpectColumnMeanToBeBetween(
        column="avg_session_duration", min_value=0, max_value=3600
    )
)
suite.add_expectation(
    gx.expectations.ExpectTableRowCountToBeGreaterThan(value=1000)
)
# Pipeline halts if any expectation fails. No silent corruption.

TFX Data Validation

Google’s TensorFlow Data Validation (TFDV) takes a different approach: it auto-generates a schema from your training data, then flags anomalies when new data deviates. Useful for catching distribution drift — “this column used to have 5 categories, now it has 47.”

import tensorflow_data_validation as tfdv

# Generate baseline schema from training data
train_stats = tfdv.generate_statistics_from_csv("train.csv")
schema = tfdv.infer_schema(train_stats)

# Validate new data against the baseline
new_stats = tfdv.generate_statistics_from_csv("new_batch.csv")
anomalies = tfdv.validate_statistics(new_stats, schema)
tfdv.display_anomalies(anomalies)
# Flags: new categories, missing features, type changes, distribution shifts

The Minimum Viable Validation Checklist

  • Schema: column names, types, and allowed values haven’t changed
  • Completeness: null rates are within expected bounds
  • Volume: row count is in the expected range (catches silent upstream failures)
  • Distribution: feature means, medians, and ranges haven’t drifted dramatically
  • Freshness: the most recent timestamp is actually recent (catches stale data)
  • Uniqueness: primary keys are actually unique

If your pipeline doesn’t check at least these six things, you don’t have data quality — you have hope.

Storage Patterns for ML

Where you put your data matters more than most people think. The storage architecture determines what queries are fast, what’s cheap, and how much pain you’ll endure as data grows.

Data WarehouseData LakeLakehouse

SchemaSchema-on-writeSchema-on-readSchema-on-read with enforcement
Query engineBuilt-in (BigQuery, Redshift)External (Spark, Presto)Built-in (Databricks, Snowflake)

The Lakehouse: Why It’s Winning

The lakehouse pattern — open table formats like Delta Lake or Apache Iceberg on top of cheap object storage — is eating both warehouses and lakes. You get ACID transactions, schema enforcement, and time-travel queries on data that lives in S3 at pennies per GB.

For ML teams, the killer feature is time travel:

# Delta Lake: query data as it existed at a specific point in time
from delta.tables import DeltaTable

# Current version
current = spark.read.format("delta").load("s3://lake/user_features")

# Data as it was 7 days ago — perfect for training set reproducibility
historical = (
    spark.read.format("delta")
    .option("timestampAsOf", "2024-01-08")
    .load("s3://lake/user_features")
)

# Or by version number
v42 = (
    spark.read.format("delta")
    .option("versionAsOf", 42)
    .load("s3://lake/user_features")
)

This means you can always reproduce the exact dataset any model trained on. Combined with DVC for model artifacts and Git for code, you get full experiment reproducibility.

File Formats: The Short Version

Parquet for tabular data (columnar, compressed, fast). TFRecord or WebDataset for training data that needs to be streamed efficiently to GPUs. JSON/JSONL for schema flexibility during early exploration. Avro for data that needs schema evolution (adding/removing fields over time).

If you’re not sure, use Parquet. You’ll be right 80% of the time.
