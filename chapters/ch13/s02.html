Data Engineering

Chapter 13: ML Systems & Production 4 subtopics
TL;DR

Models are only as good as the data that feeds them. Data engineering is the discipline of building reliable pipelines that collect, clean, transform, and deliver data at scale. This section covers ETL vs ELT, batch and stream processing with Spark and Beam, feature stores with Feast, and orchestration with Airflow and Prefect. If your data pipeline is broken, your model is broken — you just do not know it yet.

ETL and ELT Pipelines

What Is ETL?

ETL stands for Extract, Transform, Load. It is the classic pattern for moving data from source systems into a data warehouse. You extract raw data from databases, APIs, or log files. You transform it — clean, join, aggregate, and reshape it. Then you load the result into a destination where analysts and models can use it.

The key detail: the transformation happens before the data lands in its final home. This was the standard approach for decades because storage was expensive — you did not want to store raw, messy data. You cleaned it first and only stored the refined version.

# Classic ETL example: extract from API, transform, load to warehouse
import requests
import pandas as pd
from sqlalchemy import create_engine

# Extract
response = requests.get("https://api.example.com/transactions?date=2024-01-15")
raw_data = pd.DataFrame(response.json())

# Transform
clean = raw_data.dropna(subset=["amount", "user_id"])
clean["amount_usd"] = clean.apply(convert_currency, axis=1)
clean["hour_of_day"] = pd.to_datetime(clean["timestamp"]).dt.hour
daily_agg = clean.groupby("user_id").agg(
    total_spend=("amount_usd", "sum"),
    num_transactions=("amount_usd", "count"),
    avg_transaction=("amount_usd", "mean"),
).reset_index()

# Load
engine = create_engine("postgresql://warehouse:5432/analytics")
daily_agg.to_sql("user_daily_spend", engine, if_exists="append", index=False)

What Is ELT?

ELT flips the order: Extract, Load, Transform. You dump the raw data into a data lake or modern warehouse first, then transform it in place. This became practical when cloud storage got cheap (pennies per gigabyte per month) and compute engines like BigQuery and Snowflake could run transformations on massive datasets efficiently.

The advantage of ELT is flexibility. You keep the raw data, so if you later realize you need a different transformation (a new feature, a different aggregation window), you do not need to re-extract from the source. The raw data is already there. For ML, this is valuable because feature engineering is iterative — you rarely know in advance which transformations you will need.

When to Use Which

Use ETL when data privacy requires you to filter or anonymize data before storage (you cannot store raw PII in your warehouse). Use ELT when you need flexibility and your destination supports in-place transformations. In practice, most modern ML teams use ELT — land the raw data, then transform it with dbt, Spark, or SQL.

Batch and Stream Processing: Spark and Beam

Apache Spark

Apache Spark is the workhorse of big data processing. It distributes computation across a cluster of machines, letting you process terabytes of data in minutes. For ML, Spark is everywhere — feature computation, data cleaning, ETL, and even model training (via MLlib, though most teams prefer PyTorch or TensorFlow for training).

Spark's key abstraction is the DataFrame — a distributed table. You write transformations using an API that feels like Pandas, and Spark figures out how to parallelize the work across your cluster. Under the hood, Spark's query optimizer (Catalyst) rearranges your operations for efficiency.

from pyspark.sql import SparkSession
from pyspark.sql import functions as F

spark = SparkSession.builder.appName("feature_pipeline").getOrCreate()

# Read raw event data (could be billions of rows)
events = spark.read.parquet("s3://data-lake/events/2024/01/")

# Compute user-level features
user_features = (
    events
    .filter(F.col("event_type") == "page_view")
    .groupBy("user_id")
    .agg(
        F.count("*").alias("page_views_30d"),
        F.countDistinct("session_id").alias("sessions_30d"),
        F.avg("session_duration_sec").alias("avg_session_duration"),
        F.max("timestamp").alias("last_active"),
    )
)

# Write to feature store / warehouse
user_features.write.mode("overwrite").parquet("s3://features/user_features/")

A common trap with Spark: it is lazy. Transformations are not executed until you call an action (like write, count, or collect). This means errors in your transformation logic do not surface until the job actually runs, which can be frustrating during development. Test on small samples first.

Apache Beam

Apache Beam is a unified programming model for both batch and stream processing. You write your pipeline once, and it can run on multiple execution engines — Google Cloud Dataflow, Apache Flink, or Spark. The appeal is portability and the ability to use the same code for both historical backfills (batch) and real-time processing (streaming).

import apache_beam as beam

# Beam pipeline that works for both batch and streaming
with beam.Pipeline() as pipeline:
    (
        pipeline
        | "Read" >> beam.io.ReadFromPubSub(topic="projects/myproject/topics/events")
        | "Parse" >> beam.Map(parse_event_json)
        | "Window" >> beam.WindowInto(beam.window.FixedWindows(300))  # 5-minute windows
        | "Key by user" >> beam.Map(lambda e: (e["user_id"], e))
        | "Aggregate" >> beam.CombinePerKey(compute_user_features)
        | "Write" >> beam.io.WriteToBigQuery("project:dataset.user_features_5min")
    )

Beam's streaming model uses windowing — grouping events by time windows (fixed, sliding, or session-based) so you can compute aggregates over time. This is essential for real-time feature computation. For example, "number of transactions in the last 5 minutes" is a sliding window aggregate that updates continuously as new events arrive.

Spark vs Beam

Spark is more mature, has a larger ecosystem, and is the default choice for batch processing. Beam shines when you need true streaming semantics or want to run the same pipeline on different execution engines. Many teams use Spark for batch feature pipelines and a streaming framework (Flink or Beam on Dataflow) for real-time features.

Feature Stores

Why Feature Stores Exist

Feature engineering is the most time-consuming part of building ML systems. Teams compute the same features over and over for different models. Worse, the features used during training (computed offline in Spark) often differ subtly from the features used during serving (computed online in Java or Go). This training-serving skew silently degrades model performance.

A feature store solves both problems. It provides a central repository of feature definitions that can be shared across teams and models, and it ensures the same features are served for both training and inference.

Feast: An Open-Source Feature Store

Feast (Feature Store) is the most popular open-source feature store. It has two key components: an offline store (backed by a data warehouse like BigQuery or Redshift) for serving historical features during training, and an online store (backed by Redis, DynamoDB, or similar) for serving the latest feature values at low latency during inference.

# feature_repo/features.py — Define features in Feast
from feast import Entity, FeatureView, Field, FileSource
from feast.types import Float32, Int64
from datetime import timedelta

# Define the entity (the "who" or "what" the features describe)
user = Entity(name="user_id", join_keys=["user_id"])

# Define the data source
user_stats_source = FileSource(
    path="data/user_stats.parquet",
    timestamp_field="event_timestamp",
)

# Define the feature view
user_stats_fv = FeatureView(
    name="user_stats",
    entities=[user],
    ttl=timedelta(days=1),  # features expire after 1 day
    schema=[
        Field(name="login_count_7d", dtype=Int64),
        Field(name="avg_session_duration", dtype=Float32),
        Field(name="total_spend_30d", dtype=Float32),
        Field(name="support_tickets_30d", dtype=Int64),
    ],
    source=user_stats_source,
)

The workflow is: define your features in code, materialize them to the online store (feast materialize), then fetch them at training time via get_historical_features and at serving time via get_online_features. The same feature definitions power both paths, eliminating skew.

Key Concepts

Point-in-time correctness is the most important concept in feature stores. When training a model, you need features as they were at the time of each training example — not as they are today. If a user churned on January 15th, you need their features from January 14th, not from today. Without point-in-time correctness, you get data leakage — the model trains on features that include information from the future, and it learns patterns that will not exist at inference time.

Feature freshness is how up-to-date the features in the online store are. For batch-computed features, freshness depends on how often you run the materialization pipeline. For streaming features, freshness can be near-real-time (seconds). Choose based on your latency requirements.

Pipeline Orchestration: Airflow and Prefect

Why You Need Orchestration

A production ML system has dozens of interdependent pipelines. The feature pipeline depends on the data ingestion pipeline. The training pipeline depends on the feature pipeline. The model deployment depends on the training pipeline passing validation. If any step fails, downstream steps should not run. If a step fails at 3 AM, someone should be alerted.

Pipeline orchestration manages these dependencies, schedules runs, handles retries, and provides visibility into what is running, what failed, and why.

Apache Airflow

Apache Airflow is the most widely used orchestrator for data and ML pipelines. You define your pipeline as a DAG (Directed Acyclic Graph) — a set of tasks with dependencies between them. Airflow schedules the DAG, runs tasks in the right order, retries failed tasks, and provides a web UI for monitoring.

from airflow import DAG
from airflow.operators.python import PythonOperator
from airflow.operators.bash import BashOperator
from datetime import datetime, timedelta

default_args = {
    "owner": "ml-team",
    "retries": 2,
    "retry_delay": timedelta(minutes=10),
    "email_on_failure": True,
}

with DAG(
    "daily_ml_pipeline",
    default_args=default_args,
    schedule_interval="0 6 * * *",  # Run daily at 6 AM
    start_date=datetime(2024, 1, 1),
    catchup=False,
) as dag:

    ingest = BashOperator(
        task_id="ingest_raw_data",
        bash_command="python pipelines/ingest.py --date {{ ds }}",
    )

    validate = PythonOperator(
        task_id="validate_data",
        python_callable=run_data_validation,
        op_kwargs={"date": "{{ ds }}"},
    )

    features = BashOperator(
        task_id="compute_features",
        bash_command="spark-submit pipelines/features.py --date {{ ds }}",
    )

    train = BashOperator(
        task_id="train_model",
        bash_command="python pipelines/train.py --features-date {{ ds }}",
    )

    evaluate = PythonOperator(
        task_id="evaluate_model",
        python_callable=run_evaluation,
    )

    # Define execution order
    ingest >> validate >> features >> train >> evaluate

Airflow is powerful but has rough edges. The scheduler can be slow for large DAGs. Dynamic task generation is awkward. And the execution model (each task runs as a separate process) makes it hard to pass large data between tasks — you end up writing intermediate results to cloud storage.

Prefect

Prefect is a modern alternative that addresses many of Airflow's pain points. Pipelines are defined as Python functions decorated with @flow and @task. The code looks like normal Python, which makes it easier to test and debug locally. Prefect also handles dynamic workflows (tasks that spawn other tasks based on runtime data) more gracefully.

from prefect import flow, task
from prefect.tasks import task_input_hash
from datetime import timedelta

@task(retries=2, retry_delay_seconds=600,
      cache_key_fn=task_input_hash, cache_expiration=timedelta(hours=12))
def ingest_data(date: str):
    """Pull raw data from source systems."""
    raw = fetch_from_api(date)
    save_to_lake(raw, date)
    return raw

@task
def compute_features(raw_data):
    """Transform raw data into ML features."""
    features = build_features(raw_data)
    validate_features(features)
    return features

@task
def train_and_evaluate(features):
    """Train model and check quality gates."""
    model = train_model(features)
    metrics = evaluate_model(model, features)
    if metrics["auc"] < 0.75:
        raise ValueError(f"Model AUC {metrics['auc']} below threshold")
    register_model(model, metrics)
    return metrics

@flow(name="daily-ml-pipeline")
def daily_pipeline(date: str):
    raw = ingest_data(date)
    features = compute_features(raw)
    metrics = train_and_evaluate(features)
    return metrics

Choosing Between Them

Airflow is the safe choice for large organizations with existing Airflow infrastructure and teams that know it well. It has the largest community and the most integrations. Prefect is the better developer experience — easier to write, test, and debug. For new projects without legacy constraints, Prefect (or the similar Dagster) is often the better choice. Both work. Pick one and be consistent.

Data Validation in Pipelines

Every pipeline should validate its data before passing it downstream. Great Expectations is the most popular tool for this. You define expectations — "column X should never be null," "the mean of column Y should be between 10 and 100," "there should be at least 1000 rows" — and the tool checks them on every pipeline run. Failed expectations halt the pipeline before bad data poisons your model.

import great_expectations as gx

context = gx.get_context()
datasource = context.sources.add_pandas("my_data")
asset = datasource.add_dataframe_asset("daily_features")

# Define expectations
expectation_suite = context.add_expectation_suite("feature_quality")
expectation_suite.add_expectation(
    gx.expectations.ExpectColumnValuesToNotBeNull(column="user_id")
)
expectation_suite.add_expectation(
    gx.expectations.ExpectColumnMeanToBeBetween(
        column="avg_session_duration", min_value=0, max_value=3600
    )
)
expectation_suite.add_expectation(
    gx.expectations.ExpectTableRowCountToBeGreaterThan(value=1000)
)

Treat data validation as a first-class citizen. Your model cannot be better than its data. And unlike code bugs, data bugs are silent — the pipeline runs successfully, the model trains without errors, and months later someone notices that conversion rates dropped because an upstream schema change turned a numeric column into strings.
