Cloud ML Platforms

Chapter 13: ML Systems & Production 3 subtopics

TL;DR

The three major cloud providers — AWS, GCP, and Azure — each offer comprehensive ML platforms that handle the heavy lifting of training, deploying, and managing models at scale. AWS SageMaker is the most mature and feature-rich. GCP Vertex AI is tightly integrated with Google's AI ecosystem and offers strong AutoML. Azure ML pairs enterprise features with deep Microsoft/OpenAI integration. Understanding each platform's strengths helps you pick the right one — or mix and match — for your production ML workloads.

Why Cloud ML Platforms Exist

Building ML infrastructure from scratch is brutally hard. You need compute orchestration, data pipelines, model registries, serving infrastructure, monitoring, and a dozen other components. Each one is a full-time engineering job. Cloud ML platforms package all of this into managed services. You trade some flexibility and significant cost for speed-to-production and reduced operational burden.

The real question isn't "should I use a cloud platform?" — it's "which parts should I use?" Most mature teams adopt a hybrid approach: use the cloud platform for compute and deployment, but keep custom components where the platform's opinions don't fit your workflow.

AWS SageMaker

The Big Picture

AWS SageMaker is the most comprehensive cloud ML platform, with services spanning the entire ML lifecycle. It launched in 2017 and has since grown into a sprawling ecosystem. Its greatest strength is depth — there's a SageMaker service for almost everything. Its greatest weakness is also depth — the number of services and configuration options can be overwhelming.

SageMaker Training Jobs

SageMaker Training Jobs abstract away cluster management for model training. You provide a training script, specify instance types and count, and SageMaker handles provisioning, data loading, distributed training setup, and teardown. It natively supports frameworks like PyTorch, TensorFlow, and HuggingFace, with pre-built containers for each.

import sagemaker
from sagemaker.pytorch import PyTorch

estimator = PyTorch(
    entry_point="train.py",
    source_dir="src/",
    role=sagemaker.get_execution_role(),
    instance_count=4,
    instance_type="ml.p4d.24xlarge",  # 8x A100 GPUs per node
    framework_version="2.1",
    py_version="py310",
    distribution={
        "torch_distributed": {"enabled": True}  # Auto-configures DDP
    },
    hyperparameters={
        "epochs": 10,
        "learning_rate": 1e-4,
        "batch_size": 32,
    },
)

# Launches training cluster, runs training, saves model, tears down
estimator.fit({
    "train": "s3://my-bucket/data/train/",
    "validation": "s3://my-bucket/data/val/",
})

The key benefit is that you pay only for the training time. Instances spin up, train, and spin down. No idle GPU costs. For teams doing infrequent training, this is significantly cheaper than maintaining a persistent GPU cluster.

SageMaker Endpoints

SageMaker Endpoints handle model serving. You deploy a model to a real-time endpoint that auto-scales based on traffic. It supports A/B testing (route percentages of traffic to different model versions), shadow deployments (new model runs alongside old, results compared but not served), and multi-model endpoints (multiple models on one instance to reduce costs).

from sagemaker.pytorch import PyTorchModel

model = PyTorchModel(
    model_data="s3://my-bucket/models/model.tar.gz",
    role=sagemaker.get_execution_role(),
    framework_version="2.1",
    entry_point="inference.py",
)

# Deploy with auto-scaling
predictor = model.deploy(
    initial_instance_count=2,
    instance_type="ml.g5.xlarge",
    endpoint_name="my-model-v2",
)

# Configure auto-scaling
client = boto3.client('application-autoscaling')
client.register_scalable_target(
    ServiceNamespace='sagemaker',
    ResourceId=f'endpoint/my-model-v2/variant/AllTraffic',
    ScalableDimension='sagemaker:variant:DesiredInstanceCount',
    MinCapacity=1,
    MaxCapacity=10,
)

SageMaker Pipelines

SageMaker Pipelines is AWS's ML workflow orchestrator. It lets you define DAGs (directed acyclic graphs) of processing, training, evaluation, and deployment steps. Each step runs on managed infrastructure, with automatic data lineage tracking. It integrates with SageMaker's model registry for versioning and approval workflows — a model goes from "Pending" to "Approved" before it can be deployed to production.

SageMaker Feature Store

The Feature Store provides both an online store (low-latency lookups for real-time inference) and an offline store (S3-backed for training). The key value proposition: compute a feature once, use it everywhere. No more inconsistencies between training features and serving features — the infamous training-serving skew that silently kills model performance.

SageMaker Model Monitor

Model Monitor automatically detects data drift and model quality degradation on deployed endpoints. It creates a baseline from your training data, then continuously compares production traffic against that baseline. Violations trigger CloudWatch alarms. It's not as sophisticated as dedicated monitoring tools like Evidently or Arize, but it's zero-effort if you're already on SageMaker.

SageMaker pricing gotcha: SageMaker instances cost more than equivalent EC2 instances (roughly 20-40% premium). You're paying for the managed service layer. For teams with strong infrastructure skills, running training on raw EC2 with custom orchestration can be cheaper. For teams that want to focus on ML rather than infrastructure, the premium is worth it.

GCP Vertex AI

The Big Picture

Vertex AI is Google's unified ML platform, consolidating what used to be scattered across AI Platform, AutoML, and various APIs. Its strengths: tight integration with Google's AI tools (TensorFlow, JAX, TPUs), strong AutoML capabilities, and first-class support for generative AI through Model Garden and Gemini integration.

Vertex AI Training

Vertex AI Training supports custom training jobs with GPU and TPU support. TPU access is a genuine differentiator — if your workload benefits from TPUs (transformer training, large batch sizes), Vertex AI is the only major platform that offers them. It supports pre-built containers for all major frameworks and custom containers for specialized setups.

from google.cloud import aiplatform

aiplatform.init(project="my-project", location="us-central1")

job = aiplatform.CustomTrainingJob(
    display_name="my-training-job",
    script_path="train.py",
    container_uri="us-docker.pkg.dev/vertex-ai/training/pytorch-gpu.2-1:latest",
    requirements=["transformers", "datasets"],
)

model = job.run(
    replica_count=1,
    machine_type="a2-highgpu-4g",  # 4x A100 GPUs
    accelerator_type="NVIDIA_TESLA_A100",
    accelerator_count=4,
    args=["--epochs=10", "--lr=1e-4"],
)

Vertex AI Prediction

Vertex AI Prediction handles model serving with auto-scaling endpoints. It supports online predictions (real-time, low-latency) and batch predictions (high-throughput, cost-efficient). A feature unique to Vertex: Model Evaluation is built in — when you train with Vertex or import a model, it automatically computes and tracks evaluation metrics across model versions.

Vertex AI Pipelines

Vertex AI Pipelines uses Kubeflow Pipelines (KFP) as its SDK, which is both a strength and a weakness. Strength: KFP is an open-source standard, so your pipelines are somewhat portable. Weakness: KFP's abstractions can feel heavyweight compared to simpler workflow tools. Pipelines integrate with Vertex's metadata store for automatic experiment tracking and lineage.

Vertex AI Feature Store

Similar to SageMaker's offering, Vertex Feature Store provides online and offline serving. Google's implementation leverages BigQuery as the offline store, which is a natural fit if your data already lives in GCP's ecosystem. The Feature Store supports point-in-time lookups — essential for generating historically accurate training sets without data leakage.

AutoML on Vertex

Google's AutoML is arguably the most polished automated machine learning offering. It handles tabular, image, text, and video data, performing architecture search, hyperparameter tuning, and feature engineering automatically. For teams without deep ML expertise, AutoML can produce surprisingly competitive models. For experienced teams, it provides a strong baseline to beat.

Vertex AI's generative AI advantage: Vertex Model Garden gives you one-click access to foundation models including Gemini, PaLM, Imagen, and open-source models like Llama and Mistral. You can fine-tune these models directly within Vertex using your own data, then deploy to endpoints — all without managing infrastructure. If your primary use case is generative AI applications, Vertex AI's integration here is hard to beat.

Azure Machine Learning

The Big Picture

Azure Machine Learning is Microsoft's platform, with deep integration into the Azure and Microsoft ecosystem. Its distinctive strengths: the Responsible AI Dashboard (fairness, interpretability, and error analysis in one place), Azure OpenAI Service for GPT model access, and natural integration with enterprise tools like Active Directory and DevOps pipelines.

Azure ML Workspaces

Everything in Azure ML lives in a Workspace — a central place that contains your experiments, models, compute targets, datastores, and environments. Workspaces provide collaboration features, RBAC (role-based access control), and audit logging out of the box. For enterprise teams that need governance and compliance, this workspace model is valuable.

from azure.ai.ml import MLClient, command, Input
from azure.identity import DefaultAzureCredential

ml_client = MLClient(
    DefaultAzureCredential(),
    subscription_id="...",
    resource_group_name="my-rg",
    workspace_name="my-workspace"
)

# Define a training job
training_job = command(
    code="./src",
    command="python train.py --data ${{inputs.training_data}} --lr 1e-4",
    inputs={
        "training_data": Input(
            type="uri_folder",
            path="azureml://datastores/training/paths/data/"
        )
    },
    environment="AzureML-pytorch-2.1-cuda12@latest",
    compute="gpu-cluster",  # Pre-created compute cluster
    instance_count=2,
    distribution={"type": "PyTorch", "process_count_per_instance": 4},
)

returned_job = ml_client.jobs.create_or_update(training_job)

Azure ML Pipelines

Azure ML Pipelines can be defined in Python, YAML, or through the visual designer in Azure ML Studio. The visual designer is genuinely useful for prototyping workflows before codifying them. Pipelines support step caching, parallel execution, and conditional logic. They integrate with Azure DevOps and GitHub Actions for CI/CD triggers.

Azure ML Endpoints

Azure ML offers Managed Online Endpoints (real-time serving with auto-scaling) and Batch Endpoints (high-throughput offline inference). Blue/green deployments come built-in — deploy a new model version, gradually shift traffic, and roll back if metrics degrade. Managed endpoints handle infrastructure, scaling, and certificate management.

Azure ML Responsible AI Dashboard

The Responsible AI Dashboard is Azure ML's standout feature. It integrates fairness assessment (Fairlearn), interpretability (InterpretML), error analysis (identifying cohorts where the model fails), causal analysis (what-if scenarios), and counterfactual explanations into a single, interactive interface. For regulated industries, this dashboard helps satisfy audit requirements and document model behavior.

Azure OpenAI Service

Azure provides exclusive enterprise access to OpenAI models (GPT-4, GPT-4o, DALL-E, Whisper) through the Azure OpenAI Service. The key differentiators from using OpenAI's API directly: your data doesn't leave the Azure boundary, you get enterprise SLAs, content filtering is built in, and you can use Azure's networking features (VNets, Private Endpoints) for security compliance. For enterprises that want GPT-class models with enterprise governance, this is the path.

Choosing a platform: Don't pick based on features — pick based on where you already are. If your data is in S3 and your infra is on AWS, use SageMaker. If you're a Google Cloud shop or want TPU access, choose Vertex AI. If your organization runs on Azure and needs enterprise governance, Azure ML is the natural fit. Cross-cloud ML is possible but adds significant complexity that's rarely worth the cost.

Comparing the Big Three

All three platforms are converging on similar capabilities. Each has training orchestration, model serving, pipelines, feature stores, and monitoring. The real differences are in integration depth (with the respective cloud's ecosystem), pricing models, and opinionated defaults. SageMaker has the broadest feature set but the steepest learning curve. Vertex AI has the cleanest developer experience and best generative AI integration. Azure ML has the strongest enterprise governance and responsible AI tooling.

One pattern that works well: use cloud-agnostic formats (ONNX for models, Kubeflow/Airflow for orchestration, MLflow for tracking) for your core ML workflow, and only use cloud-specific features where they provide clear advantages. This gives you portability without sacrificing the benefits of managed services.
