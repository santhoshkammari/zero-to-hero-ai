On-Device ML

Chapter 13: ML Systems & Production 3 subtopics

TL;DR

On-device ML runs models directly on phones, edge devices, and microcontrollers — no cloud round-trip needed. This gives you real-time inference, works offline, preserves user privacy, and eliminates serving costs. But the constraints are brutal: limited memory (kilobytes to a few gigabytes), limited compute, strict power budgets, and no room for error. This section covers mobile deployment (TensorFlow Lite, CoreML, PyTorch Mobile), edge inference on specialized hardware (Jetson, Coral, Raspberry Pi), and the emerging world of TinyML on microcontrollers.

Why Run ML on the Device?

Every time your phone autocorrects a word, recognizes your face, or blurs the background in a photo, an ML model is running locally on the device. No API call, no cloud server, no internet connection required. There are four compelling reasons to do this instead of calling a server.

Latency. A cloud round-trip takes 50-200ms minimum. On-device inference takes 5-50ms. For real-time applications — augmented reality, autonomous driving, voice assistants — that difference is everything. You can't wait 200ms to detect a pedestrian in front of your car.

Privacy. Data never leaves the device. Your face scan, your voice recording, your health data — it all stays on hardware you physically control. For privacy-sensitive applications, this isn't a feature, it's a requirement.

Connectivity. Cloud ML fails when the network is slow, unreliable, or nonexistent. Industrial sensors in remote locations, phones on airplanes, field devices in disaster zones — these all need models that work offline.

Cost. No server means no serving cost. If you have a million users each making 100 predictions per day, on-device inference eliminates $100K+/month in cloud GPU costs. The inference compute is "free" — paid for by the user's device.

Mobile Deployment

TensorFlow Lite

TensorFlow Lite (TFLite) is Google's framework for on-device inference. It converts a TensorFlow or Keras model into an optimized flat buffer format that's designed for mobile and embedded environments. TFLite models are typically 2-5x smaller than their TensorFlow counterparts and run with a lightweight interpreter that has minimal dependencies.

import tensorflow as tf

# Convert a trained Keras model to TFLite
model = tf.keras.models.load_model("my_model.h5")

converter = tf.lite.TFLiteConverter.from_keras_model(model)
# Enable optimizations
converter.optimizations = [tf.lite.Optimize.DEFAULT]
# Quantize to int8 for maximum compression and speed
converter.target_spec.supported_types = [tf.int8]

# Provide representative dataset for calibration
def representative_dataset():
    for sample in calibration_data:
        yield [sample.numpy().astype(np.float32)]
converter.representative_dataset = representative_dataset

tflite_model = converter.convert()

# Save — this is what ships in your mobile app
with open("model.tflite", "wb") as f:
    f.write(tflite_model)
print(f"Model size: {len(tflite_model) / 1e6:.1f} MB")

TFLite supports multiple execution backends: CPU (XNNPACK delegate for optimized SIMD operations), GPU (OpenGL/OpenCL/Metal delegates), NNAPI (Android's neural network API for hardware-specific acceleration), and custom delegates for specialized hardware. The same model file can run on Android, iOS, Linux, and even microcontrollers via TFLite Micro.

CoreML (Apple)

CoreML is Apple's ML framework, deeply integrated with the Apple ecosystem. It automatically selects the best compute unit for each layer of your model — CPU, GPU, or the Apple Neural Engine (ANE). The ANE is dedicated ML hardware that's incredibly power-efficient — it can run 15.8 TOPS (trillion operations per second) on the iPhone 15 while sipping milliwatts of power.

import coremltools as ct
import torch

# Convert PyTorch model to CoreML
model = load_trained_pytorch_model()
model.eval()

# Trace the model with example input
example_input = torch.randn(1, 3, 224, 224)
traced_model = torch.jit.trace(model, example_input)

# Convert to CoreML format
mlmodel = ct.convert(
    traced_model,
    inputs=[ct.ImageType(name="image", shape=(1, 3, 224, 224),
                         scale=1/255.0)],
    compute_precision=ct.precision.FLOAT16,  # Use FP16 for ANE
    minimum_deployment_target=ct.target.iOS17,
)

# Configure compute units
mlmodel = ct.models.MLModel(
    mlmodel.get_spec(),
    compute_units=ct.ComputeUnit.ALL  # Let CoreML choose CPU/GPU/ANE
)
mlmodel.save("MyModel.mlpackage")

CoreML's strength is the integration. You drop a .mlpackage into Xcode, and it auto-generates Swift interfaces for your model. The Vision framework handles image preprocessing. The Natural Language framework handles text. For Apple-only deployments, CoreML provides the best performance per watt.

NNAPI (Android Neural Networks API)

NNAPI is Android's hardware abstraction layer for ML inference. It provides a unified API that routes computation to the best available hardware — Qualcomm's Hexagon DSP, Samsung's NPU, MediaTek's APU, or the GPU. You don't write code for NNAPI directly — frameworks like TFLite and PyTorch Mobile use it as a backend delegate. The benefit: your model automatically takes advantage of whatever AI accelerator the user's phone has.

PyTorch Mobile

PyTorch Mobile brings the PyTorch ecosystem to mobile devices. It uses TorchScript (JIT compilation) or the newer ExecuTorch framework to run optimized models on iOS and Android. PyTorch Mobile's advantage over TFLite is staying within the PyTorch ecosystem — no format conversion gymnastics if your training pipeline is already PyTorch.

import torch
from torch.utils.mobile_optimizer import optimize_for_mobile

model = load_trained_model()
model.eval()

# Script the model (captures control flow, unlike tracing)
scripted = torch.jit.script(model)

# Apply mobile-specific optimizations
optimized = optimize_for_mobile(scripted)

# Save for mobile deployment
optimized._save_for_lite_interpreter("model.ptl")
# This .ptl file ships in your mobile app

On-Device Training

On-device training updates the model directly on the user's device. This enables personalization without sending data to a server. Apple uses this for personalized Siri predictions. Google uses it for Gboard next-word prediction. The constraints are severe: training must be power-efficient (only run while charging), use minimal memory (no large optimizer states), and converge in very few steps (small batches of user data). Techniques like fine-tuning only the last few layers and using SGD instead of Adam help meet these constraints.

Framework choice decision tree: Apple-only app? Use CoreML. Android-only? Use TFLite with NNAPI. Cross-platform? Use TFLite (widest hardware support) or PyTorch Mobile (if already in PyTorch ecosystem). Need maximum portability including microcontrollers? TFLite Micro. Need maximum iOS performance? CoreML with Neural Engine targeting.

Edge Inference

Edge Hardware Landscape

Edge devices sit between phones and the cloud — more capable than microcontrollers, less powerful than data center GPUs. The key players:

NVIDIA Jetson is the edge GPU platform. The Jetson Orin Nano delivers 40 TOPS of AI performance for around $200. It runs full Linux, supports CUDA, and can execute TensorRT-optimized models. It's the go-to for robotics, industrial inspection, and smart cameras that need real-time video inference.

Google Coral uses Google's Edge TPU, a tiny ASIC that runs TFLite models at 4 TOPS with only 2 watts of power. Available as a USB accelerator, PCIe card, or system-on-module. Coral excels at classification and detection tasks where power efficiency matters more than raw throughput.

Raspberry Pi with AI HAT+ brings dedicated ML inference to the hobbyist and prototyping world. The Hailo-8L accelerator on the AI HAT+ delivers 13 TOPS. It's not the fastest, but the ecosystem — documentation, community, software support — makes it unbeatable for getting started with edge ML.

Model Optimization for Edge

Edge models need aggressive optimization. The techniques from the Model Optimization section (quantization, pruning, distillation) are essential here. But edge deployment adds additional constraints:

Model architecture matters. MobileNet, EfficientNet, and ShuffleNet were designed from the ground up for mobile and edge — they use depthwise separable convolutions and inverted residuals that trade a small accuracy drop for dramatic compute savings. Don't try to squeeze ResNet-152 onto an edge device. Use an architecture designed for the constraints.

# MobileNetV3: designed for edge inference
import torch
from torchvision.models import mobilenet_v3_small

model = mobilenet_v3_small(pretrained=True)

# Profile to understand the compute budget
from torchinfo import summary
summary(model, input_size=(1, 3, 224, 224))
# MobileNetV3-Small: 2.54M params, 56.5M MACs
# Compare: ResNet-50: 25.6M params, 4.1B MACs (73x more compute!)

Real-Time Constraints

Edge applications often have hard real-time requirements. A factory inspection camera must process each frame within 33ms (30 FPS). An autonomous vehicle must run object detection within 50ms. Missing the deadline isn't a latency issue — it's a safety issue. This means: no garbage collection pauses, no dynamic memory allocation during inference, deterministic execution times. Profile your model's worst-case latency, not just the average.

Power-Efficient Inference

Battery-powered edge devices have a power budget measured in milliwatts. Running a GPU-class model would drain the battery in minutes. Power-efficient inference requires: quantized models (INT8 uses less energy per operation than FP32), hardware-specific optimizations (using the NPU instead of the CPU), batching when possible (one large computation is more efficient than many small ones), and duty cycling (run inference only when needed, sleep the rest of the time).

Model Splitting: Edge-Cloud Collaboration

Model splitting runs the first few layers of a model on the edge device and the remaining layers in the cloud. The edge device extracts initial features (which are much smaller than the raw input), sends those features to the cloud, and the cloud completes the inference. This balances latency (early layers run locally), privacy (raw data stays on device), and accuracy (full model computation happens in the cloud). The split point is chosen to minimize the size of the intermediate representation sent over the network.

The edge deployment pipeline: Train in the cloud (full precision, full resources) → Optimize (quantize, prune, distill) → Convert to edge format (TFLite, CoreML, TensorRT) → Validate on target hardware (test accuracy, latency, memory) → Deploy OTA (over-the-air updates). Each step can reveal surprises. A model that's 95% accurate on your workstation might be 90% accurate after INT8 quantization on the actual edge hardware due to numerical differences. Always validate end-to-end on the target device.

Embedded ML and TinyML

What is TinyML?

TinyML is machine learning on microcontrollers — chips with kilobytes (not gigabytes) of memory, megahertz (not gigahertz) clock speeds, and milliwatt power budgets. An Arduino Nano 33 BLE has 256 KB of RAM. A typical TinyML model must fit in under 100 KB. This is a completely different engineering challenge from running models on phones or edge devices.

Why would you do this? Because microcontrollers are everywhere. Every sensor, every appliance, every industrial device has one. TinyML lets you add intelligence to a $2 chip that runs on a coin cell battery for years. Keyword detection ("Hey Siri"), vibration-based anomaly detection in machinery, gesture recognition on a smartwatch — these are all TinyML applications.

Microcontroller Deployment

TensorFlow Lite Micro (TFLM) is the leading framework for microcontroller ML. It's a stripped-down C++ interpreter with no dynamic memory allocation, no operating system dependencies, and a footprint under 20 KB. Models are stored as C arrays compiled directly into the firmware.

// TensorFlow Lite Micro inference on a microcontroller
#include "tensorflow/lite/micro/micro_interpreter.h"
#include "tensorflow/lite/micro/micro_mutable_op_resolver.h"
#include "model_data.h"  // Model stored as a C byte array

// Allocate memory arena (all memory the model will use)
constexpr int kTensorArenaSize = 10 * 1024;  // 10 KB arena
uint8_t tensor_arena[kTensorArenaSize];

void setup() {
    // Load model from compiled-in byte array
    const tflite::Model* model = tflite::GetModel(model_data);

    // Register only the ops this model needs (saves memory)
    static tflite::MicroMutableOpResolver<5> resolver;
    resolver.AddConv2D();
    resolver.AddMaxPool2D();
    resolver.AddFullyConnected();
    resolver.AddReshape();
    resolver.AddSoftmax();

    // Create interpreter
    static tflite::MicroInterpreter interpreter(
        model, resolver, tensor_arena, kTensorArenaSize);
    interpreter.AllocateTensors();

    // Run inference
    float* input = interpreter.input(0)->data.f;
    // ... fill input with sensor data ...
    interpreter.Invoke();
    float* output = interpreter.output(0)->data.f;
    // ... use prediction ...
}

Memory-Constrained Model Design

Designing models for microcontrollers requires thinking about every byte. Standard architectures are too large. Instead, use: Depthwise separable convolutions (MobileNet-style) that cut parameters by 8-9x. Extreme quantization — INT8 or even binary weights. Architecture search constrained by memory and latency budgets — tools like MCUNet co-design the neural architecture and the memory scheduling for a specific microcontroller. The resulting models are tiny (50-200 KB) but surprisingly capable for tasks like keyword spotting and sensor classification.

Sensor Fusion on Edge

Sensor fusion combines data from multiple sensors — accelerometer, gyroscope, microphone, temperature, light — into a single ML model. On edge devices, this enables context-aware applications: is the user walking, running, or cycling? Is this machine vibrating normally or abnormally? Is someone at the door? The challenge is synchronizing sensor streams with different sampling rates and handling the multi-modal input within the device's memory constraints.

# Example: Activity recognition from sensor fusion
# Model designed for microcontroller deployment
import tensorflow as tf

def create_tiny_fusion_model():
    """Multi-sensor activity recognition model for MCU."""
    # Accelerometer input: 50 samples × 3 axes
    accel_input = tf.keras.Input(shape=(50, 3), name='accelerometer')
    # Gyroscope input: 50 samples × 3 axes
    gyro_input = tf.keras.Input(shape=(50, 3), name='gyroscope')

    # Lightweight 1D convolution per sensor (depthwise separable)
    accel_features = tf.keras.layers.SeparableConv1D(8, 5, activation='relu')(accel_input)
    accel_features = tf.keras.layers.GlobalAveragePooling1D()(accel_features)

    gyro_features = tf.keras.layers.SeparableConv1D(8, 5, activation='relu')(gyro_input)
    gyro_features = tf.keras.layers.GlobalAveragePooling1D()(gyro_features)

    # Fuse sensor features
    fused = tf.keras.layers.Concatenate()([accel_features, gyro_features])
    fused = tf.keras.layers.Dense(16, activation='relu')(fused)
    output = tf.keras.layers.Dense(5, activation='softmax')(fused)  # 5 activities

    model = tf.keras.Model(inputs=[accel_input, gyro_input], outputs=output)
    return model  # ~2,000 parameters, ~8 KB quantized

model = create_tiny_fusion_model()
model.summary()
The TinyML frontier: We're entering an era where every sensor can have a brain. A $0.50 microcontroller with a 50 KB model can detect anomalies, classify sounds, recognize gestures, and make local decisions without ever connecting to the internet. The implications for industrial IoT, healthcare wearables, environmental monitoring, and smart infrastructure are enormous. The constraint — extreme resource limits — forces creative engineering that pushes the boundaries of what's possible with ML.
