ML System Design

Chapter 13: ML Systems & Production Section 1 of 9
TL;DR

The model is the smallest part of any ML system that actually works. This section walks the full architecture — requirements → data → features → model → serving → monitoring → feedback — and the design decisions that separate systems that survive production from systems that survive a demo. You'll learn the patterns that work (retrieve→rank, cascade, shadow mode), the anti-patterns that kill projects, and how to do back-of-envelope cost estimation before you burn a quarter of runway on GPU bills.

Why System Design > Model Quality

Google published a diagram in 2015 that should be tattooed on every ML engineer's forearm. The ML model code? A tiny rectangle. Everything around it — data collection, feature extraction, data verification, serving infrastructure, monitoring, configuration — absolutely dwarfs it.

And yet, most ML courses spend 95% of the time on that tiny rectangle.

Here's the uncomfortable truth: a mediocre model inside a well-designed system will outperform a brilliant model inside a broken one, every single time. The brilliant model that can't retrain when data shifts, can't serve under latency constraints, and can't tell you when it's wrong? That model is a liability, not an asset.

# What actually ships an ML product:
model_quality      = 0.10   # important, but not the bottleneck
data_pipelines     = 0.25   # this is where bugs live
feature_eng        = 0.20   # this is where signal lives
serving_infra      = 0.15   # this is where latency lives
monitoring         = 0.15   # this is where trust lives
organizational     = 0.15   # this is where projects die
# ───────────────────────────
# total engineering effort = 1.0
# model_quality is 10% of the work

System design is the discipline of thinking through the entire lifecycle before writing a single line of training code. Skip it, and you build something that works on your laptop and collapses in production. I've watched it happen enough times that I stopped counting.

Requirements Gathering

Problem Formulation

The first ten minutes of any ML system design should be spent not talking about models. Start with the business need in plain language:

"We want to predict which products a user is most likely to buy in the next 7 days so we can personalize their homepage."

One sentence. Now you know the task (ranking), the time horizon (7 days), and the surface (homepage). Without this clarity, teams wander for months building the wrong thing. I've seen a team spend six months building a classifier when the product needed a ranker.

The framework: What is the input? What is the output? Who consumes the output? How fast do they need it? What happens when the model is wrong?

Business Metrics vs ML Metrics

This is where good teams separate from great ones. The data scientist optimizes AUC-ROC. The business cares about revenue. These are not the same thing. A model with 0.95 AUC that doesn't change user behavior is a vanity project.

LayerMetric TypeExamplesWho Cares
OfflineML metricAUC, F1, NDCG, RMSEML engineers
OnlineProduct metricCTR, conversion rate, latencyProduct managers
BusinessBusiness metricRevenue, retention, cost savedExecutives

The critical question: does improving the offline metric actually improve the online metric? You have to validate this empirically. If your NDCG goes up but CTR doesn't move, your offline metric is lying to you.

Feasibility — Kill Bad Ideas Early

Before committing engineering months, be brutally honest about three things:

  1. Do you have enough labeled data? If the event you're predicting happens once per million requests, you might not have enough positives to learn from.
  2. Is the signal in the data? Predicting stock prices from public data alone? The efficient market hypothesis is laughing at you.
  3. Can the org actually act on predictions? A churn model is worthless if no team exists to contact at-risk customers.

Run a feasibility spike. One week. Logistic regression on hand-crafted features. If it can't beat "predict the majority class," the problem likely isn't solvable with available data. Better to learn this in week one than month six.

Baselines — Your Floor and Your Fallback

Every system needs a baseline — not because baselines are exciting, but because they serve two critical functions:

  1. Floor to beat. If your deep learning model can't outperform "show the most popular items," maybe you don't need deep learning.
  2. Fallback at 2 AM. When your fancy model crashes, the baseline keeps the product running.
# Common baselines by task
BASELINES = {
    "recommendations":  "most popular items",
    "search_ranking":   "BM25 (keyword match)",
    "classification":   "logistic regression / majority class",
    "fraud_detection":  "rule-based (velocity + blocklist)",
    "forecasting":      "last value / seasonal naive",
    "content_mod":      "keyword blocklist",
}

Never design a system without one. The baseline isn't a stepping stone — it's load-bearing infrastructure.

End-to-End ML Architecture

A production ML system has five layers. Not four, not three — five. Teams that skip one always regret it.

┌─────────────────────────────────────────────────┐
│                  MONITORING LAYER                │  ← watches everything
│    prediction drift · feature drift · alerts     │
├──────────────────┬──────────────────────────────┤
│   SERVING LAYER  │       TRAINING LAYER         │
│  model server    │  experiment tracking          │
│  A/B routing     │  hyperparameter search        │
│  fallback logic  │  model registry               │
│  latency budget  │  distributed training         │
├──────────────────┴──────────────────────────────┤
│                  FEATURE LAYER                   │
│  feature store · feature computation · validation│
├─────────────────────────────────────────────────┤
│                   DATA LAYER                     │
│  ingestion · cleaning · storage · versioning     │
└─────────────────────────────────────────────────┘

Data Layer

Everything starts here. Raw data flows in from user interactions, transactions, sensors — whatever your domain produces. This layer ingests, cleans, validates, and stores.

Key components: a data warehouse for historical analysis (BigQuery, Snowflake), a streaming layer for real-time events (Kafka, Kinesis), and a data validation system that catches schema changes and distribution shifts before they poison your model. (Great Expectations and TensorFlow Data Validation are the go-to tools.)

The cardinal rule: data should be immutable and versioned. If you can't reproduce the exact dataset a model was trained on, you can't debug it.

Feature Layer

This is the most underrated layer. Features are where signal lives, and a feature store ensures the same feature computation is shared between training (offline) and serving (online). Without it, you inevitably write the feature logic twice — once in PySpark for training, once in Java for serving — and they will diverge.

# Feature store: one computation, two access patterns
from feast import FeatureStore

store = FeatureStore(repo_path="feature_repo/")

# TRAINING: historical features with point-in-time correctness
training_df = store.get_historical_features(
    entity_df=entities_with_timestamps,
    features=[
        "user_features:purchase_count_7d",
        "user_features:avg_session_seconds",
        "item_features:price",
        "item_features:category_embedding",
    ],
).to_df()

# SERVING: latest feature values, low-latency lookup
online_features = store.get_online_features(
    features=["user_features:purchase_count_7d",
              "user_features:avg_session_seconds"],
    entity_rows=[{"user_id": 12345}],
).to_dict()
# Same logic, no skew. That's the whole point.

Training Layer

The part everyone thinks is "ML." Takes features + labels, trains a model, evaluates it, stores the artifact. But in production, training isn't a one-time event — it's a recurring pipeline.

The training layer includes: experiment tracking (MLflow, W&B), hyperparameter tuning, distributed training when data or models exceed one machine, and a model registry where artifacts are versioned, tested, and approved for deployment. Think of it as CI/CD but for models.

Serving Layer

Where the model meets reality. A request arrives, features are fetched, the model runs inference, and a prediction returns — all within a latency budget (typically 50–200ms for user-facing systems).

Serving also manages: multiple model versions (for A/B tests), input validation, output postprocessing (thresholds, business rules), and fallback logic. What happens when the model server is down? If the answer is "the product breaks," your design is incomplete.

Monitoring Layer

The most neglected layer. Also the one that saves you at 3 AM.

Models degrade silently. The data distribution shifts. User behavior changes. An upstream pipeline starts emitting nulls. Your model makes increasingly bad predictions without raising a single error. This is not hypothetical — this is the default outcome for every deployed model.

Monitor three things:

  1. Prediction distribution. Model used to predict 5% fraud → now predicts 40%. Something broke.
  2. Feature drift. Mean of feature X shifted 3σ from training distribution.
  3. Business metric degradation. CTR dropped 15% since last deployment.

We cover monitoring in depth in Section 6, but know this now: if you don't monitor it, you don't own it.

Batch vs Real-Time Architecture

This single decision shapes your infrastructure cost, debugging complexity, latency, and team size. Choose wrong and you'll either over-engineer a batch problem or under-serve a real-time one.

DimensionBatchReal-TimeHybrid
LatencyHours to a day50–200msSeconds to minutes
FreshnessStale (last nightly run)Current (request-time)Mixed per stage
ComplexityLowHighMedium-High
CostLow (off-peak compute)High (always-on GPUs)Medium
DebuggingEasy (deterministic)Hard (stateful, async)Medium
Good forRecs, reports, risk scoresFraud, search, content modMost production systems

Batch: Simple, Cheap, Stale

Precompute predictions for all (or many) inputs on a schedule. Users get fast lookups because predictions are already in a key-value store.

# Nightly batch pipeline — simple and debuggable
import pandas as pd
from joblib import load

model = load("models/recommender_v3.joblib")
users = pd.read_parquet("features/user_features_latest.parquet")

predictions = model.predict(users)

results = pd.DataFrame({
    "user_id": users["user_id"],
    "top_items": predictions,
})
# Push to Redis for sub-ms lookups at serving time
push_to_redis(results)

Batch works when: inputs are known in advance (all users), freshness isn't critical (daily is fine), and you want simple infrastructure. It breaks when the user just watched a horror movie and you're still showing them romcoms from last night's run.

Real-Time: Fresh, Expensive, Complex

Compute predictions on demand. User loads page → server fetches features → model runs → prediction returns. All within the latency budget.

You need: a model server with high throughput (TorchServe, Triton, TF Serving), a feature store with low-latency online access, careful model optimization (quantization, batching, caching), and an SLA-aware architecture. For fraud detection, content moderation, and search ranking, real-time is usually non-negotiable.

Hybrid — What Production Actually Looks Like

Most systems at scale use a hybrid. The computationally expensive first pass (candidate generation) runs in batch. The lighter second pass (ranking) runs in real-time. This gives you batch efficiency where freshness doesn't matter and real-time precision where it does.

Netflix, YouTube, LinkedIn, and basically every recommendation system you've used runs some version of this pattern. If someone tells you their system is "fully real-time end to end," they're either at massive scale or they're wrong.

Technical Debt in ML Systems

Google's 2015 paper "Hidden Technical Debt in Machine Learning Systems" is one of the most-cited ML engineering papers for a reason. ML systems accrue technical debt faster than traditional software because they have an extra axis of complexity: data. Your code can be flawless, but if the data shifts, the system breaks.

Here are the debts that will crush you if you don't manage them.

Hidden Data Dependencies

In normal software, you trace dependencies through imports and function calls. In ML, your model silently depends on every feature in its input vector. When an upstream team tweaks how they compute one feature — even slightly — your model's behavior changes. No import statement flagged it. No type checker caught it.

The fix: version features, write data contracts with upstream teams, monitor feature distributions continuously. If a feature changes, you want to know in minutes, not weeks.

Pipeline Jungles

One pipeline feeds features. Another cleans labels. A third joins data from a new source. A fourth does something nobody remembers but everyone is afraid to touch. This is a pipeline jungle, and it grows organically in every ML team that doesn't actively fight it.

# Signs you have a pipeline jungle:
# 1. Nobody can draw the full data flow on a whiteboard
# 2. "Which pipeline produces this feature?" has no quick answer
# 3. You have Airflow DAGs that trigger other Airflow DAGs
#    that trigger Spark jobs that write to tables
#    that other DAGs read from
# 4. Deleting any single pipeline feels risky

# The fix: consolidate pipelines, use orchestration
# (Airflow/Prefect/Dagster), treat pipeline code
# with the same rigor as application code.

Dead Features and Zombie Experiments

ML teams experiment constantly. A feature gets added to test a hypothesis, the experiment is inconclusive, the feature stays forever. After a year, nobody knows why it exists, but nobody dares remove it. These dead features increase complexity, slow training, and occasionally introduce subtle bugs.

Rule of thumb: if removing a feature doesn't degrade the model by more than noise, delete it. Run this audit quarterly.

Feedback Loops

The debt nobody sees coming. Your model's predictions influence user behavior, which generates the training data for the next model version. If your recommendation model only shows popular items, users can only click popular items, and the next model thinks popular items are even more important. The system eats its own tail.

Direct feedback loops are obvious (model predicts → user acts → model trains on that action). Indirect loops are sneaky (model A influences data that model B trains on). Both can amplify biases and create echo chambers. The fix: hold out exploration traffic, track feedback loop metrics, and inject diversity.

Design Patterns That Actually Work

Retrieve → Rank (The Workhorse)

The single most important pattern in production ML. Used by every major recommendation and search system.

Phase 1 — Retrieval: Narrow millions of candidates to hundreds using a cheap model (embedding similarity, collaborative filtering, or even heuristics). Speed matters more than precision here.

Phase 2 — Ranking: Apply an expensive, feature-rich model to those hundreds. Precision matters more than speed here.

def recommend(user_id: str, k: int = 20) -> list:
    # Phase 1: Retrieve ~500 candidates (cheap, ~10ms)
    # Uses ANN index (FAISS/ScaNN) over item embeddings
    candidates = retrieval_model.get_candidates(user_id, k=500)

    # Phase 2: Rank with a feature-rich model (~50ms)
    # Uses 200+ features: user history, item metadata,
    # context (time, device), cross-features
    features = feature_store.get_features(user_id, candidates)
    scores = ranking_model.predict(features)

    # Phase 3: Business rules (often forgotten, always needed)
    # Diversity, freshness, dedup, legal, age-gating
    final = apply_business_rules(candidates, scores)
    return final[:k]

Why split it? Because running a 200-feature neural ranker on 10 million candidates would cost you a small fortune per request. The retrieval stage makes the ranking stage economically feasible.

Cascade: Cheap First, Expensive Last

A series of increasingly expensive models, each filtering traffic for the next.

# Content moderation cascade
def moderate(content: str) -> str:
    # Stage 1: Keyword blocklist (~0.1ms, catches 60%)
    if keyword_filter.is_blocked(content):
        return "BLOCKED"

    # Stage 2: Lightweight classifier (~5ms, catches 30%)
    score = fast_classifier.predict(content)
    if score > 0.95:
        return "BLOCKED"
    if score < 0.05:
        return "APPROVED"

    # Stage 3: Heavy model for ambiguous cases (~100ms, 10%)
    score = deep_model.predict(content)
    if score > 0.7:
        return "REVIEW"
    return "APPROVED"

Result: 90% of traffic is handled by cheap models. The expensive deep model only processes the hardest 10%. Your GPU bill drops by an order of magnitude while quality stays constant on the cases that matter.

Feature Store: Compute Once, Serve Everywhere

The core idea: compute features in one place, serve them to both training and inference. This eliminates training-serving skew — the silent killer where your model sees different feature values in production than it saw during training.

Without a feature store, you'll write feature logic in PySpark for training and Java for serving. They'll produce subtly different results (null handling, float precision, timezone bugs), and your model's performance will degrade in ways that are nearly impossible to debug.

Shadow Mode: Test Before You Trust

Before promoting a new model, run it in shadow mode: it receives real traffic and makes predictions, but those predictions are logged, not served. The old model still handles users. You compare the shadow model's outputs against the live model and against ground truth.

Only when the shadow model wins on all the metrics that matter — accuracy, latency, error rates, edge case behavior — do you promote it. This is safer than A/B testing for the initial deployment because zero users see bad predictions.

Human-in-the-Loop: Know When to Defer

For high-stakes decisions (medical diagnoses, loan approvals, content moderation edge cases), route low-confidence predictions to human reviewers. The model handles the 85% of cases where it's confident. Humans handle the rest.

def predict_with_human_fallback(input_data, confidence_threshold=0.85):
    score = model.predict_proba(input_data)
    confidence = max(score)

    if confidence >= confidence_threshold:
        return {"decision": model.predict(input_data),
                "source": "model", "confidence": confidence}
    else:
        queue_for_human_review(input_data, score)
        return {"decision": "PENDING_REVIEW",
                "source": "human_queue", "confidence": confidence}

This combines ML throughput with human judgment. The trick is calibrating the threshold: too high and humans drown in reviews, too low and the model makes confident-but-wrong calls.

Anti-Patterns That Kill ML Projects

Anti-PatternWhat HappensThe Fix
Offline-only eval Metrics look great on held-out data. Model tanks in production because offline metrics don't capture feedback loops, display bias, or user behavior changes. Always A/B test before declaring victory. Offline eval is necessary but not sufficient.
Premature complexity You build a feature store, model registry, K8s cluster, and custom serving framework before you have a working model. Start with a cron job + Flask endpoint + PostgreSQL. Add infra when you hit real scaling walls, not imagined ones.
Deploy and forget Model works on launch day. Six months later, it's silently making terrible predictions. Nobody noticed because there's no monitoring. Treat model health like server health. Dashboards. Alerts. Oncall rotation.
Training-serving skew Feature code is PySpark for training, Java for serving. They handle nulls, floats, and timestamps differently. Model sees different data in prod. Use a feature store. Or at minimum, share a single feature computation codebase.
Metric mismatch Team optimizes AUC while business cares about revenue. Model improves on ML metrics but product metrics don't budge. Define the causal chain: offline metric → online metric → business metric. Validate each link.
Data leakage Feature accidentally encodes the label (e.g., using future data to predict the past). Model gets 0.99 AUC offline, random performance in prod. Strict temporal splits. Point-in-time feature computation. Paranoid feature auditing.

The meta-anti-pattern: treating ML like a research project instead of software engineering. Research optimizes for novelty. Production optimizes for reliability. The skills are different.

Cost Estimation

ML systems have cost structures that traditional software doesn't: GPU time for training, storage for massive datasets, compute for feature pipelines, and always-on serving infrastructure. If you don't estimate costs before you build, your cloud bill will provide an unpleasant surprise.

Back-of-Envelope Math

# ── Training costs ──────────────────────────────
gpu_price_per_hour = 3.50   # A100 on-demand (AWS/GCP)
num_gpus = 4
hours_per_run = 8
experiments_per_week = 5
retrains_per_week = 1

weekly_train = (experiments_per_week + retrains_per_week) \
               * num_gpus * hours_per_run * gpu_price_per_hour
monthly_train = weekly_train * 4.3
# → ~$2,900/month

# ── Serving costs ───────────────────────────────
requests_per_day = 1_000_000
ms_per_request = 50
gpu_seconds_per_day = requests_per_day * (ms_per_request / 1000)
gpu_hours_per_day = gpu_seconds_per_day / 3600
# Need to provision for ~10x peak vs average
monthly_serve = gpu_hours_per_day * 10 * 30 * gpu_price_per_hour
# → varies wildly, but $5K–$50K/month is typical

# ── Storage ─────────────────────────────────────
# Training data, features, model artifacts, logs
# Surprise: prediction logs often cost more than serving
monthly_storage = 2000  # ~$2K/month

# ── The hidden cost ─────────────────────────────
engineer_salary_monthly = 18000  # ~$216K/year fully loaded
# A system needing 1 FTE to maintain: $216K/year
# Simpler systems are *literally* cheaper systems

Design Walkthrough: Fraud Detection

Let's tie everything together with a real system design — fraud detection for an e-commerce platform. This is the kind of exercise you'd do in a system design interview or a design review at work.

Step 1: Requirements

  • Goal: Flag fraudulent transactions in real-time, before payment authorization.
  • Latency budget: 200ms end-to-end.
  • Traffic: ~10K transactions/minute at peak.
  • Fraud rate: 0.3% (heavily imbalanced).
  • Cost of a miss: $150 average (chargeback + investigation).
  • Cost of a false positive: $5 (support ticket + lost goodwill).

Step 2: Metrics

LayerMetricTarget
OfflineRecall @ 50%+ precision≥ 80% recall
OnlineFraud loss rate (undetected $ / total $)< 0.05%
OnlineFalse positive rate< 1%
Business$ fraud prevented - system costPositive ROI

Step 3: Architecture

Real-time serving is non-negotiable here. We use a cascade:

Transaction → [Rules Engine] → [GBT Model] → [Deep Model] → Decision
                  ↓ block          ↓ score        ↓ score
              known bad IPs     handles 85%    handles hardest 5%
              impossible addr   of traffic     of traffic
              velocity limits

Fallback: if model server is unresponsive, fall back to
rules-only (higher false positives, but no missed fraud).

Feature strategy: Precompute user-level features in a feature store (purchase history, device fingerprints, velocity features like "purchases in the last hour"). Transaction-level features (amount, merchant category, time of day) are computed at request time — they're cheap.

Training: Daily retraining on a rolling 90-day window. Fraud labels arrive delayed (chargebacks take days to weeks), so we use a staggered labeling pipeline with provisional labels.

Step 4: Monitoring

  • Prediction distribution: If the model flags 20% of transactions instead of the usual 1%, something is very wrong.
  • Feature drift: Average transaction amount shifted dramatically? Investigate.
  • Ground truth tracking: When fraud reports come in weeks later, measure actual recall. This is your source of truth.
  • Latency: p50, p95, p99. If p99 breaches 200ms, the circuit breaker triggers.

Step 5: Cost

# Monthly cost estimate
serving_gpus     = 10_000   # 4 GPUs, always-on
feature_pipeline = 2_000    # Spark jobs + feature store
storage_logging  = 1_000    # Data warehouse + logs
daily_retraining = 3_000    # GPU hours for daily retrain
total_system     = 16_000   # ~$16K/month

fraud_prevented  = 200_000  # Estimated from fraud rate × volume
roi = fraud_prevented - total_system  # $184K/month net benefit

The ROI is clear. But notice — we didn't justify the ML system by comparing it to "no system." We justified it against the rules-only baseline, which catches maybe 40% of fraud. The ML cascade catches 80%+. That 40 percentage point improvement is worth $100K+/month in prevented fraud.

This is what ML system design looks like in practice. No magic. No glamour. Just careful requirement analysis, honest tradeoff assessment, and relentless attention to the "boring" parts — data quality, monitoring, fallback logic, cost — that separate systems that work from systems that work in a demo.
