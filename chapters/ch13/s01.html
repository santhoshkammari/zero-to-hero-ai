ML System Design

Chapter 13: ML Systems & Production 10 subtopics
TL;DR

Building a real ML system is 90% engineering and 10% modeling. This section covers the full arc — from gathering requirements and choosing between batch and real-time architectures, to identifying technical debt before it buries you. We walk through system components, design patterns, anti-patterns, and honest cost estimation so you can ship ML products that actually survive contact with users.

Why System Design Matters More Than Your Model

Here is a truth that surprises most people entering ML: the model is the easy part. A research paper can demonstrate a brilliant architecture on a clean benchmark in a few pages. Deploying that same model to serve millions of users, keeping it accurate as data shifts, making it fast enough that people do not tap away — that is the hard part. Google's famous 2015 paper called ML code "the small box in the middle" of a massive system. Everything around it — data pipelines, feature stores, serving infrastructure, monitoring — dwarfs the model itself.

This is why ML system design exists as a discipline. It is the practice of thinking through the entire lifecycle of an ML product before writing a single line of training code. If you skip this step, you will build something that works on your laptop and collapses in production. I have seen it happen dozens of times.

Requirements Gathering

Problem Formulation

Every ML system starts with a question: what are we actually trying to do? Not "let's use deep learning" — that is a solution looking for a problem. Start with the business need. Are we trying to reduce customer support tickets? Increase click-through rates? Detect fraudulent transactions before they settle?

Write the problem statement in plain English. "We want to predict which products a user is most likely to buy in the next 7 days so we can personalize their homepage." Now you know the task (ranking), the time horizon (7 days), and the surface (homepage). Without this clarity, teams wander for months building the wrong thing.

Business Metrics vs ML Metrics

This is where many teams trip. The data scientist optimizes for AUC-ROC or F1 score because those are clean mathematical quantities. The business cares about revenue, retention, or customer satisfaction. These are not the same thing. A model with 0.95 AUC might not move revenue at all if the predictions are not actionable.

Bridge the gap explicitly. Define an offline metric (what you measure during training — accuracy, precision, recall, NDCG) and an online metric (what you measure in production — click-through rate, conversion rate, revenue per user). Then validate the relationship: does improving the offline metric actually improve the online metric? If not, your offline metric is a vanity number.

Feasibility Assessment

Before committing engineering months, answer three questions honestly. First, do you have enough labeled data? If the event you are predicting happens once per million requests, you might not have enough positive examples to learn from. Second, is the signal in the data? If you are predicting stock prices from public data alone, the efficient market hypothesis is working against you. Third, can the organization actually act on predictions? A churn model is useless if no team exists to contact at-risk customers.

Run a quick feasibility spike. Spend a week. Build the simplest possible model — logistic regression on hand-crafted features. If it cannot beat a reasonable baseline (like "predict the majority class"), the problem might not be solvable with available data. Better to learn this in week one than month six.

Baseline Selection

Every system needs a baseline — the simplest approach that could work. For recommendations, the baseline might be "show the most popular items." For search ranking, it might be BM25 (a classic keyword-matching algorithm). For classification, it might be a hand-written rule or a logistic regression model.

The baseline serves two purposes. It gives you a floor to beat. And it gives you a fallback — when your fancy model crashes at 2 AM, the baseline keeps the product running. Never design a system without one.

End-to-End ML Architecture

An ML system has four major components, and they need to talk to each other cleanly. Think of them as four boxes with arrows between them.

The Data Layer

This is where everything starts. Raw data flows in from user interactions, transactions, sensor readings, or whatever your domain produces. The data layer is responsible for ingesting this raw data, cleaning it, transforming it into features, and storing it where both training and serving can access it.

Key components include a data warehouse for historical analysis (like BigQuery or Snowflake), a feature store for serving precomputed features at low latency (like Feast or Tecton), and a data validation layer that catches schema changes and distribution shifts before they poison your model.

# Simplified feature store interaction
from feast import FeatureStore

store = FeatureStore(repo_path="feature_repo/")

# At training time: get historical features
training_df = store.get_historical_features(
    entity_df=entity_df,  # DataFrame with entity keys and timestamps
    features=[
        "user_features:login_count_7d",
        "user_features:avg_session_duration",
        "item_features:price",
        "item_features:category_embedding",
    ],
).to_df()

# At serving time: get latest features for a user
online_features = store.get_online_features(
    features=["user_features:login_count_7d", "user_features:avg_session_duration"],
    entity_rows=[{"user_id": 12345}],
).to_dict()

The Training Layer

This is the part most people think of as "ML." It takes prepared features and labels, trains a model, evaluates it, and stores the trained artifact. But in a production system, training is not a one-time event — it is a recurring pipeline. You retrain on fresh data weekly, daily, or even continuously.

The training layer includes experiment tracking (MLflow, Weights & Biases), hyperparameter tuning, distributed training infrastructure (when your data or model is too big for one machine), and a model registry where trained models are versioned and approved for deployment.

The Serving Layer

This is where the model meets users. A request comes in (a user loads a page, a transaction is submitted, a photo is uploaded), the serving layer runs the model on that input, and returns a prediction. Speed matters here — if your fraud detection model takes 5 seconds, the transaction has already settled.

The serving layer handles model loading, input preprocessing, inference, postprocessing, and response formatting. It also manages multiple model versions (for A/B testing) and fallback logic (what happens when the model server is down).

The Monitoring Layer

This is the most neglected component, and the one that will save you at 3 AM. Models degrade silently. The data distribution shifts, user behavior changes, a upstream data pipeline starts producing nulls — and your model starts making bad predictions without raising a single error.

Model monitoring watches for prediction distribution shifts ("the model used to predict 5% fraud, now it predicts 40% — something is wrong"), feature drift ("the average value of feature X changed by 3 standard deviations"), and performance degradation ("our click-through rate dropped 15% since the last model deployment"). We will cover monitoring in depth in a later section.

Batch vs Real-Time Architecture

This is one of the most consequential design decisions you will make. It affects your infrastructure costs, latency, complexity, and debugging difficulty.

Batch Systems

A batch system precomputes predictions for all (or many) inputs on a schedule. Every night, you run your recommendation model on all users and store their top-10 recommendations in a database. When a user opens the app, you just look up their precomputed results. Fast serving, simple architecture.

Batch works well when: predictions do not need to reflect the last few minutes of activity, the set of inputs is known in advance (all users, all products), and latency requirements are relaxed. It fails when you need real-time personalization ("the user just watched a horror movie — update recommendations immediately") or when inputs are unpredictable (like incoming text that needs moderation).

# Batch prediction pipeline (simplified)
import pandas as pd
from joblib import load

model = load("models/recommender_v3.joblib")

# Score all users
all_users = pd.read_parquet("features/user_features_latest.parquet")
predictions = model.predict(all_users)

# Store for serving
results = pd.DataFrame({
    "user_id": all_users["user_id"],
    "top_items": predictions
})
results.to_parquet("serving/recommendations_2024_01_15.parquet")
# Also push to a key-value store for fast lookups
push_to_redis(results)

Real-Time Systems

A real-time system computes predictions on demand, at request time. The user loads a page, the server fetches their latest features, runs the model, and returns results — all within a latency budget (often 50-200 milliseconds).

Real-time is harder. You need a model server that can handle high throughput, a feature store that serves features at low latency, and careful optimization to keep inference fast. But it gives you freshness — predictions reflect the very latest data. For fraud detection, content moderation, and search ranking, real-time is usually non-negotiable.

Hybrid Approaches

Most production systems use a hybrid. Candidate generation (the computationally expensive first pass) is done in batch, while ranking (the lighter second pass) is done in real-time. This gives you the efficiency of batch with the freshness of real-time where it matters most. Netflix, YouTube, and LinkedIn all use variations of this pattern.

Technical Debt in ML Systems

Technical debt is code or architecture that is quick to build but expensive to maintain. ML systems accumulate technical debt faster than traditional software because they have an additional axis of complexity: data. Your code can be perfect, but if the data changes, the system breaks.

Data Dependencies

In traditional software, you can trace dependencies through function calls and imports. In ML, your model depends on every feature in its input. If an upstream team changes how they compute a feature — even slightly — your model's behavior changes. These hidden data dependencies are the number one source of ML bugs in production.

Protect yourself. Version your features. Write data contracts with upstream teams. Monitor feature distributions continuously. If a feature changes, you want to know immediately, not three weeks later when a VP asks why conversion rates dropped.

Pipeline Jungles

As a system evolves, data pipelines grow organically. One pipeline feeds features, another cleans labels, a third joins data from a new source. Before you know it, you have a tangled web of scripts that nobody fully understands. This is a pipeline jungle. The fix is aggressive refactoring — consolidate pipelines, use a proper orchestration framework (Airflow, Prefect, Dagster), and treat pipeline code with the same rigor as application code.

Dead Experimental Code and Features

ML teams experiment constantly. A feature gets added to test a hypothesis, the experiment is inconclusive, and the feature stays in the pipeline forever. After a year, nobody remembers why it is there, but nobody dares remove it. These dead features increase complexity, slow training, and occasionally introduce bugs. Audit your features regularly. If a feature does not improve the model, remove it.

Design Patterns for ML Systems

Two-Phase Prediction (Retrieve → Rank)

This is the workhorse pattern for recommendations and search. Phase one (retrieval) quickly narrows millions of candidates to hundreds using a cheap model or heuristic. Phase two (ranking) applies an expensive model to those hundreds to produce the final ordered list. Splitting the work this way makes real-time serving feasible at scale.

# Two-phase recommendation
def recommend(user_id):
    # Phase 1: Retrieve top-500 candidates (cheap, ~10ms)
    candidates = retrieval_model.get_candidates(user_id, k=500)

    # Phase 2: Rank candidates with a detailed model (~50ms)
    features = feature_store.get_features(user_id, candidates)
    scores = ranking_model.predict(features)

    # Apply business rules (diversity, freshness, deduplication)
    final = apply_business_rules(candidates, scores)
    return final[:20]

Cascade Pattern

Use a series of increasingly expensive models. A fast rule-based filter catches the obvious cases (90% of traffic). A lightweight model handles the medium cases. An expensive deep model handles only the hardest 5%. This dramatically reduces compute costs while maintaining quality where it matters.

Feature Store Pattern

Compute features once and share them across training and serving. Without a feature store, you end up computing features differently in the training pipeline (Python + Spark) and the serving path (Java + Redis), leading to training-serving skew — the model sees different feature values in production than it did during training, and performance degrades silently.

Shadow Mode

Before deploying a new model to users, run it in shadow mode — it receives real traffic and makes predictions, but those predictions are logged, not served. The old model still serves users. You compare the shadow model's predictions against the live model and against ground truth. Only when you are confident the new model is better do you promote it.

Human-in-the-Loop

For high-stakes decisions (medical diagnoses, loan approvals, content moderation of edge cases), route low-confidence predictions to human reviewers. The model handles the easy cases (90%+ confidence), and humans handle the rest. This combines the throughput of ML with the judgment of humans.

Anti-Patterns to Avoid

Offline-Only Evaluation

You test the model on a held-out dataset, the metrics look great, and you ship it. But offline metrics do not capture everything — feedback loops, user behavior changes, display biases. Always validate with online experiments (A/B tests) before declaring victory.

Premature Complexity

You do not need a feature store, a model registry, Kubernetes, and a custom serving framework on day one. Start simple. Use a cron job for batch predictions. Serve from a Flask endpoint. Add infrastructure as you grow. Many successful ML systems started as a scikit-learn model behind a REST API.

No Monitoring After Deployment

The model works on launch day. Six months later, nobody has checked it. The world has changed, and the model is silently making terrible predictions. This is not hypothetical — it happens at most companies. Monitor your models. Set up alerts. Treat model health like server health.

Training-Serving Skew

The feature engineering code in your training pipeline is written in PySpark. The feature engineering code in your serving path is rewritten in Java for speed. They are supposed to produce identical results, but they do not — edge cases around null handling, floating point precision, or timestamp parsing differ. The model was trained on PySpark features but serves with Java features. Performance drops, and the bug is nearly invisible. The fix: use a feature store, or at minimum, share the feature computation code.

Documentation That Actually Helps

Most ML system documentation is either nonexistent or useless. Here is what actually helps.

Model cards document what the model does, what data it was trained on, how it performs across different subgroups, and its known limitations. Google introduced model cards in 2019, and they have become a best practice. Think of them as nutrition labels for models.

A system design document describes the architecture: components, data flows, latency requirements, failure modes, and rollback procedures. Write it before you build, and update it as the system evolves. Future you (and the engineer who inherits the system) will be grateful.

Runbooks describe how to respond to incidents. "If prediction latency exceeds 500ms, check the feature store first, then the model server, then the network." Concrete, step-by-step procedures. Write them after every incident you encounter.

Cost Estimation

ML systems have costs that traditional software does not: GPU time for training, storage for large datasets, compute for feature pipelines, and serving infrastructure that scales with traffic. If you do not estimate costs upfront, you will get an unpleasant surprise on your cloud bill.

Training Costs

Estimate how much each training run costs (GPU hours × price per hour), how often you retrain (daily, weekly), and how many experiments you run per week. A single training run of a large model on 8 GPUs for 24 hours costs $200-$800 on cloud infrastructure. If you retrain daily and run 5 experiments per week, that adds up fast.

# Back-of-the-envelope training cost
gpu_price_per_hour = 3.50   # e.g., A100 on AWS
num_gpus = 4
training_hours = 8
experiments_per_week = 5
retrains_per_week = 1

weekly_cost = (experiments_per_week + retrains_per_week) * num_gpus * training_hours * gpu_price_per_hour
monthly_cost = weekly_cost * 4.3
print(f"Monthly training cost: ${monthly_cost:,.0f}")
# Monthly training cost: $2,902

Serving Costs

Serving costs scale with traffic. If your model serves 1 million requests per day and each request takes 50ms of GPU time, you need to provision enough GPU capacity to handle peak traffic (which might be 10× average). Factor in auto-scaling headroom, redundancy, and the cost of keeping model servers warm.

Data and Storage Costs

Large datasets cost money to store and move. Training data, feature tables, model artifacts, logs, and monitoring data all consume storage. Data transfer between regions or clouds incurs egress fees. A common surprise: storing the logs from your model's predictions can cost more than the model serving itself.

Total Cost of Ownership

Add it all up: data pipeline compute, storage, training, serving, monitoring, and — the biggest cost — engineering time. A system that requires a full-time engineer to maintain is costing you $200K+ per year in salary alone. Simpler systems are cheaper systems. Always ask: does the complexity justify the improvement?

Putting It Together: A Design Walkthrough

Let us walk through designing a real system — a fraud detection service for an e-commerce platform.

Requirements: Flag fraudulent transactions in real-time (before payment is authorized). Latency budget: 200ms. The system processes ~10,000 transactions per minute at peak. The fraud rate is 0.3%. A missed fraud costs the company an average of $150. A false positive (blocking a legitimate transaction) costs $5 in customer support and lost goodwill.

Metrics: Offline — precision and recall at a chosen threshold (we want 80%+ recall with at least 50% precision). Online — fraud loss rate (dollar amount of undetected fraud / total transaction volume) and false positive rate.

Architecture: Real-time serving is non-negotiable. A cascade of models: a fast rule-based filter catches obvious fraud (known bad IPs, impossible shipping addresses), a lightweight gradient-boosted tree handles most traffic, and a deep neural network handles edge cases. Features are precomputed in a feature store (user purchase history, device fingerprint, velocity features). The model runs behind an API gateway with circuit-breaker logic — if the model server is unresponsive, fall back to the rule-based filter only (higher false positives, but no missed fraud).

Monitoring: Track prediction distribution (if the model suddenly flags 20% of transactions, something is wrong), feature drift (if the average transaction amount shifts dramatically), and ground-truth performance (fraud reports that come in days or weeks later). Set up alerts for each.

Cost estimate: Model serving on 4 GPUs for low-latency inference: ~$10K/month. Feature pipeline compute: ~$2K/month. Storage and logging: ~$1K/month. Daily retraining: ~$3K/month. Total: ~$16K/month. The fraud the system prevents is estimated at $200K/month. The ROI is clear.

This is what ML system design looks like in practice. It is not glamorous. There is no magic. It is careful thinking about requirements, honest assessment of tradeoffs, and relentless attention to the boring parts — data quality, monitoring, fallback logic — that separate systems that work from systems that work in a demo.
