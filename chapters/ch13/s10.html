Scaling and Hardware

Chapter 13: ML Systems & Production 4 subtopics

TL;DR

Modern ML models are compute monsters. Training GPT-scale models takes thousands of GPUs coordinating for weeks. Understanding GPU architecture, memory hierarchies, and distributed training strategies isn't optional anymore — it's the difference between a model that trains in days and one that never finishes. This section covers the hardware landscape (GPUs, TPUs, custom accelerators), distributed training paradigms (data/model/pipeline parallelism), and the infrastructure decisions that make large-scale ML feasible and cost-effective.

Why Hardware Matters Now More Than Ever

Ten years ago, most ML practitioners never thought about hardware. You trained a random forest on your laptop and called it a day. That world is gone. A single training run for a frontier LLM costs millions of dollars in compute. Even fine-tuning a 7B parameter model needs careful hardware planning. Understanding the metal underneath your code has become a core ML engineering skill.

The compute demands of ML have grown roughly 10x per year since 2012 — far outpacing Moore's Law. This gap is why specialized hardware like GPUs, TPUs, and custom accelerators exist. General-purpose CPUs weren't designed for the massive matrix multiplications that define neural network training.

GPU Computing

GPU Architecture: Why GPUs Dominate ML

A CPU has a handful of powerful cores optimized for sequential tasks. A GPU has thousands of simpler cores optimized for doing the same operation on many data points simultaneously. Neural network training is overwhelmingly matrix multiplications — multiply millions of weights by millions of activations, add them up, repeat. That's exactly the kind of embarrassingly parallel workload GPUs were built for.

NVIDIA GPUs have two types of cores that matter for ML. CUDA cores are general-purpose parallel processors — they handle any floating-point math. Tensor Cores, introduced with the Volta architecture, are specialized hardware for matrix multiply-accumulate operations. A single Tensor Core operation computes a 4×4 matrix multiply in one clock cycle — something that would take many cycles on CUDA cores. This is why Tensor Core utilization is the key metric for ML workload performance.

GPU Memory Management

GPU memory (VRAM) is almost always the bottleneck, not compute. A 7B parameter model in FP32 needs ~28 GB just to store the weights. Add optimizer states (Adam stores two additional copies per parameter), gradients, and activations, and you easily need 4-5x the model size in memory during training.

# Memory estimation for training a model
def estimate_training_memory(param_count_billions, precision="fp32"):
    bytes_per_param = {"fp32": 4, "fp16": 2, "bf16": 2, "int8": 1}
    bp = bytes_per_param[precision]

    model_memory = param_count_billions * 1e9 * bp
    # Adam optimizer: 2 states per param (momentum + variance) in fp32
    optimizer_memory = param_count_billions * 1e9 * 4 * 2
    gradient_memory = param_count_billions * 1e9 * bp
    # Activations depend on batch size and sequence length
    # Rule of thumb: roughly equal to model memory for typical configs
    activation_memory = model_memory

    total_gb = (model_memory + optimizer_memory +
                gradient_memory + activation_memory) / (1024**3)
    return total_gb

print(f"7B model (fp16): ~{estimate_training_memory(7, 'fp16'):.0f} GB")
print(f"7B model (fp32): ~{estimate_training_memory(7, 'fp32'):.0f} GB")
# 7B model (fp16): ~98 GB  →  needs multi-GPU setup
# 7B model (fp32): ~150 GB →  definitely multi-GPU
The memory wall: Even NVIDIA's H100 has 80 GB of HBM3. A 70B parameter model in FP16 needs ~140 GB just for weights. This is why techniques like model parallelism, gradient checkpointing, and quantization aren't just nice-to-haves — they're requirements for training large models.

NVIDIA GPU Ecosystem

The A100 (Ampere architecture, 2020) was the workhorse of the LLM revolution. 80 GB HBM2e, 312 TFLOPS of FP16 Tensor Core performance, and NVLink for multi-GPU communication. The H100 (Hopper, 2023) roughly doubled that with the Transformer Engine — dedicated hardware for the attention mechanisms used in transformers, supporting FP8 precision for even more throughput. The B200 (Blackwell, 2024) pushes further with second-gen Transformer Engine and up to 192 GB of HBM3e.

Mixed Precision Training

Mixed precision training uses FP16 or BF16 for most operations while keeping a master copy of weights in FP32. This nearly halves memory usage and doubles throughput on Tensor Cores, with negligible accuracy loss. The key insight: neural networks are remarkably tolerant of reduced precision during forward and backward passes — you only need full precision for the weight update step.

import torch
from torch.cuda.amp import autocast, GradScaler

model = MyModel().cuda()
optimizer = torch.optim.AdamW(model.parameters(), lr=1e-4)
scaler = GradScaler()  # Handles loss scaling for FP16 stability

for batch in dataloader:
    optimizer.zero_grad()

    # Forward pass in FP16 (faster, less memory)
    with autocast(dtype=torch.bfloat16):
        outputs = model(batch['input'].cuda())
        loss = criterion(outputs, batch['target'].cuda())

    # Backward pass with loss scaling to prevent gradient underflow
    scaler.scale(loss).backward()
    scaler.step(optimizer)
    scaler.update()

BF16 (Brain Float 16), developed by Google Brain, has the same exponent range as FP32 but fewer mantissa bits. This means it handles the same range of numbers (no overflow issues) while being half the size. For ML training, BF16 is often preferred over FP16 because you skip the loss-scaling dance entirely.

GPU Profiling

Use NVIDIA's nsight-systems and nsight-compute to profile your training. Common findings: you're often GPU-bound on large matmuls but CPU-bound on data loading. The fix is usually more data workers, prefetching, or moving preprocessing to the GPU. PyTorch Profiler wraps these tools with a Python-friendly API.

TPU and Custom Hardware

TPU Architecture

Google's Tensor Processing Units (TPUs) take a different approach than GPUs. They use a systolic array architecture — data flows through a grid of processing elements in a regular pattern, like a wave. This is phenomenally efficient for matrix operations but less flexible than GPUs for general computation. TPUs are available exclusively through Google Cloud and Kaggle.

TPU Programming with XLA

XLA (Accelerated Linear Algebra) is the compiler that converts your ML framework operations into optimized TPU instructions. When you use JAX or TensorFlow with TPUs, XLA analyzes your entire computation graph and fuses operations, eliminates redundant computation, and optimizes memory layout. PyTorch supports TPUs via PyTorch/XLA.

Custom AI Accelerators

The hardware landscape is diversifying rapidly. Apple's Neural Engine powers on-device ML for iPhones and Macs — it's why Stable Diffusion can run on a MacBook. Intel Habana Gaudi targets data center training as a cost-effective alternative to NVIDIA. AWS Trainium and Inferentia offer custom silicon optimized for training and inference respectively, with significant cost savings for AWS-native workloads.

Distributed Training

The Fundamental Problem

When a model doesn't fit on one GPU — or when training on one GPU would take months — you distribute the work across multiple GPUs. This introduces a fundamental tension: more GPUs means more compute, but also more communication overhead. The art of distributed training is maximizing the compute-to-communication ratio.

Data Parallelism

Data parallelism is the simplest and most common strategy. Each GPU holds a complete copy of the model. You split each batch across GPUs — GPU 0 gets samples 1-32, GPU 1 gets samples 33-64, etc. Each GPU computes gradients independently, then all GPUs synchronize their gradients (via AllReduce) and update their weights identically.

import torch
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP

def setup(rank, world_size):
    dist.init_process_group("nccl", rank=rank, world_size=world_size)
    torch.cuda.set_device(rank)

def train(rank, world_size):
    setup(rank, world_size)

    model = MyModel().to(rank)
    model = DDP(model, device_ids=[rank])  # Wrap model for distributed
    optimizer = torch.optim.AdamW(model.parameters(), lr=1e-4)

    sampler = torch.utils.data.distributed.DistributedSampler(
        dataset, num_replicas=world_size, rank=rank
    )
    dataloader = DataLoader(dataset, sampler=sampler, batch_size=32)

    for epoch in range(num_epochs):
        sampler.set_epoch(epoch)  # Ensure proper shuffling
        for batch in dataloader:
            loss = model(batch.to(rank)).loss
            loss.backward()        # Gradients auto-synced by DDP
            optimizer.step()
            optimizer.zero_grad()

Data parallelism scales well to dozens of GPUs but has a hard limit: the model must fit on a single GPU. For large models, you need model parallelism.

Model Parallelism: Tensor and Pipeline

Tensor parallelism splits individual layers across GPUs. A linear layer with a weight matrix of shape [4096, 16384] can be split column-wise across 4 GPUs, each holding a [4096, 4096] slice. This requires communication within each layer (AllReduce of partial results) but lets you handle layers too large for one GPU.

Pipeline parallelism assigns different layers to different GPUs. GPU 0 runs layers 1-10, GPU 1 runs layers 11-20, and so on. The challenge: naive pipelining has terrible utilization because each GPU waits for the previous one. The solution is micro-batching — split each batch into micro-batches and pipeline them, so all GPUs stay busy most of the time. GPipe and PipeDream are the key papers here.

Fully Sharded Data Parallel (FSDP)

FSDP is PyTorch's answer to the memory problem of data parallelism. Instead of each GPU holding a full model copy, FSDP shards model parameters, gradients, and optimizer states across all GPUs. Each GPU only holds 1/N of the model. Before a forward pass through a layer, FSDP gathers the full parameters; after, it discards them. This trades communication for memory.

from torch.distributed.fsdp import FullyShardedDataParallel as FSDP
from torch.distributed.fsdp import ShardingStrategy

model = MyLargeModel()
model = FSDP(
    model,
    sharding_strategy=ShardingStrategy.FULL_SHARD,
    mixed_precision=MixedPrecision(
        param_dtype=torch.bfloat16,
        reduce_dtype=torch.bfloat16,
    ),
)
# Now each GPU only holds 1/N of the parameters
# Memory per GPU drops dramatically

DeepSpeed ZeRO

Microsoft's DeepSpeed ZeRO (Zero Redundancy Optimizer) takes the same sharding idea but offers three progressive stages. ZeRO-1 shards optimizer states only (2-4x memory reduction). ZeRO-2 adds gradient sharding (4-8x). ZeRO-3 shards everything — parameters too (linear memory scaling with GPU count). ZeRO-3 with CPU offloading can train models that seem impossibly large for your hardware.

Collective Communication Primitives

All distributed training boils down to a few communication patterns. AllReduce computes a sum (or average) across all GPUs — the core of gradient synchronization. AllGather collects data from all GPUs to every GPU — used by FSDP to reconstruct parameters. ReduceScatter reduces then distributes — used by ZeRO for efficient sharding. Understanding these primitives helps you reason about communication costs.

Gradient Accumulation

Can't afford more GPUs but need a larger effective batch size? Gradient accumulation runs multiple forward-backward passes, accumulates the gradients, then does a single optimizer step. This simulates a larger batch size without needing more memory. The trade-off: training takes proportionally longer.

Scaling efficiency: Doubling your GPUs rarely halves your training time. Communication overhead, synchronization barriers, and reduced per-GPU batch sizes all eat into the speedup. Measure your actual scaling efficiency: if 8 GPUs give you a 6x speedup over 1 GPU, that's 75% efficiency — pretty good for data parallelism.

Infrastructure Scaling

Multi-Node Clusters and Interconnects

Within a single machine, GPUs communicate via NVLink — NVIDIA's high-bandwidth interconnect (900 GB/s on H100). Between machines, you need InfiniBand (400 Gb/s with NDR) or high-speed Ethernet (RoCE). The inter-node bandwidth is typically 10-100x lower than intra-node. This is why smart parallelism strategies put tensor parallelism within a node (high bandwidth needed) and pipeline/data parallelism across nodes (lower bandwidth tolerated).

Spot/Preemptible Instance Strategies

Cloud GPU instances are expensive — an 8xH100 instance can cost $25+/hour. Spot instances offer 60-90% discounts but can be reclaimed with little notice. For training, implement robust checkpointing: save model state every N steps, resume from the last checkpoint when the instance restarts. Tools like Nebius, Lambda Labs, and RunPod specialize in affordable GPU compute for ML workloads.

Training Cost Optimization

Cost optimization is a systems problem. Use mixed precision to get more TFLOPS per dollar. Implement gradient checkpointing to trade compute for memory (use fewer GPUs). Profile your training loop to eliminate data loading bottlenecks. Pre-process data offline rather than on-the-fly. And always calculate: is it cheaper to train on 4 GPUs for 8 hours or 8 GPUs for 3 hours? The answer depends on your instance pricing and scaling efficiency.
