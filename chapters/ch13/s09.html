Monitoring and Observability

Chapter 13: ML Systems & Production 5 subtopics

TL;DR

Deploying a model is the easy part — keeping it honest in production is where the real engineering lives. ML models silently degrade as the world changes around them. Monitoring tracks what you expect to go wrong (metrics, thresholds, dashboards). Observability lets you diagnose what you didn't expect (traces, logs, high-cardinality exploration). Together, they form your safety net against data drift, concept drift, and the slow rot that turns a great model into a liability.

Why Models Go Bad in Production

Here's something that surprises every team the first time: your model worked perfectly at deployment, but three months later it's making terrible predictions — and nobody noticed. Why? Because software bugs crash loudly. ML bugs fail silently. A model returning a wrong prediction looks identical to one returning a right prediction at the API layer. There's no stack trace, no 500 error. The model confidently returns garbage, your service keeps humming along, and your users quietly lose trust.

This is the core argument for ML monitoring. Traditional software monitoring — latency, error rates, uptime — is necessary but nowhere near sufficient. You need to monitor the statistical assumptions your model was built on.

Data Drift Detection

What Drift Actually Means

Data drift (also called covariate shift) happens when the distribution of your input features changes from what the model saw during training. Think of a credit scoring model trained on pre-pandemic spending patterns. In March 2020, everyone's spending patterns changed overnight. The model's inputs now look nothing like training data, so its predictions become unreliable.

Drift detection is about answering one question: "Does the data coming in today look like the data I trained on?"

Statistical Tests for Drift

The Kolmogorov-Smirnov (KS) test is the workhorse for continuous features. It measures the maximum distance between two cumulative distribution functions — your training distribution and your production distribution. A large KS statistic means the distributions have diverged significantly.

from scipy import stats
import numpy as np

# Compare training vs production feature distributions
train_feature = np.random.normal(50, 10, 10000)  # training data
prod_feature = np.random.normal(53, 12, 5000)     # production data (shifted!)

ks_stat, p_value = stats.ks_2samp(train_feature, prod_feature)
print(f"KS statistic: {ks_stat:.4f}, p-value: {p_value:.6f}")

if p_value < 0.05:
    print("ALERT: Significant drift detected!")

For categorical features, use the Chi-Square test. It compares observed frequencies against expected frequencies. If your "country" feature used to be 60% US / 30% UK / 10% other, and now it's 40% US / 20% UK / 40% other, the chi-square test will flag that shift.

Population Stability Index (PSI)

The Population Stability Index (PSI) is a classic metric from credit risk modeling that's become a go-to for ML drift detection. It quantifies how much a distribution has shifted by comparing the percentage of records in each bin between your reference and current datasets.

import numpy as np

def calculate_psi(reference, current, bins=10):
    """Calculate Population Stability Index between two distributions."""
    # Bin the reference distribution
    breakpoints = np.percentile(reference, np.linspace(0, 100, bins + 1))
    breakpoints[-1] = np.inf
    breakpoints[0] = -np.inf

    ref_counts = np.histogram(reference, bins=breakpoints)[0]
    cur_counts = np.histogram(current, bins=breakpoints)[0]

    # Convert to proportions (add small epsilon to avoid division by zero)
    ref_pct = ref_counts / len(reference) + 1e-6
    cur_pct = cur_counts / len(current) + 1e-6

    psi = np.sum((cur_pct - ref_pct) * np.log(cur_pct / ref_pct))
    return psi

# PSI interpretation:
# < 0.1  → No significant drift
# 0.1-0.25 → Moderate drift, investigate
# > 0.25  → Significant drift, action needed
Rule of thumb: PSI below 0.1 means you can sleep well. Between 0.1 and 0.25, something changed — look into it. Above 0.25, your model is operating outside its comfort zone and you likely need to retrain.

Feature Distribution Monitoring & Schema Drift

Beyond statistical tests, monitor for structural changes. Schema drift means the shape of your data changed: a column was renamed, a new category appeared, a field that was never null suddenly has nulls. These are often caused by upstream data pipeline changes — someone refactored the event logging, a third-party API changed its response format, or a new version of the mobile app sends different telemetry.

Set up validation checks that answer: Are all expected columns present? Are data types unchanged? Are value ranges within expected bounds? Are null rates stable? This is the boring stuff that prevents 3 AM pages.

Concept Drift

When the World Changes Under Your Model

Concept drift is sneakier than data drift. Here, it's not the inputs that changed — it's the relationship between inputs and outputs. The function your model learned is no longer the right function. A spam classifier trained in 2020 might see the same email features in 2024, but what constitutes spam has evolved. The concept of "spammy" drifted.

Gradual vs. Sudden Drift

Gradual drift is like fashion — it changes slowly. Your fraud model's accuracy degrades over months as fraudsters evolve their tactics. Sudden drift is an earthquake — a regulatory change, a pandemic, a platform policy update that overnight changes user behavior. Your monitoring system needs to catch both, which means monitoring at different time scales.

Prediction Drift and Label Drift

Prediction drift is a useful proxy when you can't get ground truth quickly. If your model's output distribution shifts — say your approval rate jumps from 70% to 85% — something changed, even if you don't yet know whether the new predictions are right or wrong. Label drift occurs when the distribution of actual outcomes shifts: your e-commerce return rate went from 8% to 15%. Even if your model's inputs and predictions are stable, the world moved.

Windowed Performance Analysis

The standard approach is to compute metrics over rolling time windows. Compare this week's accuracy against last week, last month, and the training evaluation. Use Page-Hinkley or ADWIN (Adaptive Windowing) algorithms for automated change-point detection in streaming metrics.

from collections import deque
import numpy as np

class SimpleWindowedMonitor:
    """Track metric drift using sliding windows."""
    def __init__(self, ref_window=1000, test_window=200, threshold=0.05):
        self.ref_window = deque(maxlen=ref_window)
        self.test_window = deque(maxlen=test_window)
        self.threshold = threshold

    def update(self, actual, predicted):
        correct = int(actual == predicted)
        self.ref_window.append(correct)
        self.test_window.append(correct)

        if len(self.test_window) == self.test_window.maxlen:
            ref_acc = np.mean(list(self.ref_window))
            test_acc = np.mean(list(self.test_window))
            drop = ref_acc - test_acc
            if drop > self.threshold:
                return f"DRIFT ALERT: accuracy dropped {drop:.2%}"
        return None

Drift Adaptation Strategies

Once you detect drift, you have several options. Retrain on recent data — the most common and often most effective. Online learning — continuously update the model with new data. Ensemble with a new model — train a fresh model on recent data and blend predictions. Feature engineering — sometimes adding a time-based feature or recency-weighted features helps the model adapt. The right choice depends on how fast your domain moves and how expensive retraining is.

Model Performance Monitoring

Real-Time Metric Dashboards

Your dashboard should show a hierarchy of metrics. At the top: business KPIs (revenue impact, user engagement). In the middle: model performance metrics (accuracy, precision, recall, AUC). At the bottom: operational metrics (latency, throughput, error rate). When a business KPI dips, you drill down through these layers to find the root cause.

Alerting and Thresholds

Setting alert thresholds is more art than science. Too tight and you'll drown in false alarms — alert fatigue is real and dangerous. Too loose and you'll miss real problems. A good starting point: alert on absolute thresholds (accuracy below 0.85) AND relative thresholds (accuracy dropped 5% from last week). Use different severity levels — an information alert for a 2% drop, a warning for 5%, and a critical page for 10%.

Segment-Level Monitoring

Aggregate metrics hide problems. Your overall accuracy might be 92%, but it's 98% for your majority user segment and 60% for a minority one. Monitor performance across slices: by geography, user demographic, device type, time of day. This is where fairness monitoring meets production monitoring.

The ground truth problem: Many ML systems have delayed labels. A loan default takes months to materialize. A content recommendation's true value unfolds over weeks. You can't compute accuracy if you don't have labels yet. Use proxy metrics (prediction confidence, prediction distribution shifts) to bridge the gap until ground truth arrives.

Feedback Loop Integration & Ground Truth Collection

Design your system to close the feedback loop. Every prediction should eventually be linked to an outcome. In some domains this is automatic (did the user click?). In others, you need explicit labeling workflows (was this flagged transaction actually fraud?). Build the data pipeline for ground truth collection before you deploy the model — retrofitting it later is painful.

Observability Tools

Purpose-Built ML Observability

Evidently AI is the open-source darling. It generates beautiful drift reports, data quality reports, and model performance dashboards. It integrates with pandas DataFrames, making it easy to drop into existing Python pipelines.

from evidently.report import Report
from evidently.metric_preset import DataDriftPreset, DataQualityPreset

# Compare reference (training) vs current (production) data
drift_report = Report(metrics=[
    DataDriftPreset(),
    DataQualityPreset(),
])
drift_report.run(reference_data=train_df, current_data=prod_df)
drift_report.save_html("drift_report.html")

WhyLabs / whylogs takes a different approach. It creates lightweight statistical profiles of your data — sketches of distributions rather than storing raw data. This makes it feasible to monitor high-volume streams without blowing up your storage costs.

Arize AI provides a commercial platform that ties together embeddings, feature drift, and model performance in a unified view. Its embedding drift detection is particularly useful for unstructured data models — tracking when your image or text embeddings shift tells you something changed even when individual features are hard to define.

Prometheus + Grafana for ML

Don't overlook the tools you already have. Prometheus can scrape custom model metrics exposed via your serving endpoint. Grafana can visualize them alongside your infrastructure metrics. The advantage: your ops team already knows these tools.

from prometheus_client import Histogram, Counter, start_http_server

# Define ML-specific metrics
PREDICTION_LATENCY = Histogram(
    'model_prediction_seconds', 'Time spent on prediction',
    buckets=[0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1.0]
)
PREDICTION_COUNTER = Counter(
    'model_predictions_total', 'Total predictions',
    ['model_version', 'predicted_class']
)

@PREDICTION_LATENCY.time()
def predict(features):
    result = model.predict(features)
    PREDICTION_COUNTER.labels(
        model_version='v2.3',
        predicted_class=str(result)
    ).inc()
    return result

start_http_server(8000)  # Expose metrics on /metrics endpoint

Advanced Monitoring Topics

Delayed Ground Truth Handling

When labels arrive days or weeks after predictions, use a two-stage monitoring system. Stage one: monitor proxy signals in real time (prediction distribution, confidence scores, input drift). Stage two: compute true performance metrics when labels arrive and backfill your dashboards. Set up reconciliation jobs that join predictions with outcomes and trigger alerts if the backfilled metrics show degradation.

Embedding-Based Drift Detection for LLMs

For large language models, traditional feature-level drift doesn't work — your "features" are token sequences. Instead, monitor the embedding space. Compute embeddings for incoming queries and track their distribution using distance metrics like Maximum Mean Discrepancy (MMD) or cosine similarity clustering. If your production queries start clustering in regions far from your evaluation set, your LLM is seeing inputs it wasn't tested on.

Fairness Monitoring in Production

Fairness isn't a one-time audit. Monitor demographic parity, equalized odds, and calibration across protected groups continuously. A model that was fair at deployment can become unfair as population distributions shift. Tools like Aequitas and Fairlearn can be integrated into your monitoring pipeline.

Distributed Tracing for ML

OpenTelemetry has become the standard for distributed tracing. Instrument your ML serving pipeline end-to-end: feature retrieval, preprocessing, model inference, postprocessing. When a prediction takes 2 seconds instead of 200ms, traces show you exactly which stage is the bottleneck. Pair with Jaeger for trace visualization.

SLAs and SLOs for ML Serving

Define Service Level Objectives (SLOs) that are ML-specific. Beyond p99 latency and uptime, include: prediction freshness (how stale can cached predictions be?), model version currency (are we running a model trained within the last N days?), and drift thresholds (PSI must stay below 0.2). These SLOs formalize the contract between your ML team and the rest of the organization.

Observability vs. Monitoring: Monitoring is asking predefined questions: "Is accuracy above 90%?" Observability is having enough instrumentation to ask questions you didn't think of in advance: "Why did predictions for users in Brazil change on Tuesday?" You need both. Monitoring catches known failure modes. Observability lets you debug unknown ones.

Monitoring Tools: Alibi Detect & Fiddler

Alibi Detect is an open-source library by Seldon for drift detection, outlier detection, and adversarial detection. It supports both tabular and unstructured data with a clean API for integrating into serving pipelines. Fiddler AI provides a commercial observability platform emphasizing explainability — it doesn't just tell you the model drifted, it helps you understand which features contributed to that drift and why predictions changed.
