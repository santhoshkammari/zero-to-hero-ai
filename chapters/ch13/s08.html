MLOps

Chapter 13: ML Systems & Production 4 subtopics
TL;DR

MLOps is DevOps for machine learning — the practices and tools that let you train, deploy, and maintain models reliably and repeatedly. This section covers CI/CD pipelines for ML (which are fundamentally different from CI/CD for software), model registries for versioning and governance, pipeline orchestration with Kubeflow and Vertex AI, and infrastructure as code to make your ML platform reproducible.

CI/CD for ML

Why ML CI/CD Is Different

In traditional software, CI/CD is straightforward: run tests, build the artifact, deploy it. If the tests pass, the build is good. In ML, the code can be correct while the model is terrible. A training pipeline that runs without errors can produce a model with 50% accuracy because the data distribution shifted. CI/CD for ML must validate not just code but also data and model quality.

This means ML pipelines have three things to test: the code (unit tests, integration tests, linting), the data (schema validation, distribution checks, freshness), and the model (accuracy above a threshold, no performance regression on key slices, latency within budget).

Continuous Training

Continuous training is the ML-specific addition to CI/CD. When new data arrives (daily, weekly, or triggered by a data quality event), a pipeline automatically retrains the model, evaluates it against quality gates, and promotes it to production if it passes. No human intervention required for the happy path — humans only get involved when something fails.

# GitHub Actions workflow for ML CI/CD
# .github/workflows/ml_pipeline.yml
name: ML Pipeline

on:
  push:
    branches: [main]
  schedule:
    - cron: '0 6 * * *'  # Daily at 6 AM for continuous training

jobs:
  test-code:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run unit tests
        run: |
          pip install -r requirements.txt
          pytest tests/ -v --tb=short

  validate-data:
    runs-on: ubuntu-latest
    needs: test-code
    steps:
      - name: Run data validation
        run: |
          python pipelines/validate_data.py \
            --source s3://data-lake/latest/ \
            --expectations expectations/feature_suite.json
          # Fails the pipeline if data quality checks fail

  train-and-evaluate:
    runs-on: [self-hosted, gpu]
    needs: validate-data
    steps:
      - name: Train model
        run: |
          python pipelines/train.py \
            --data-version $(date +%Y-%m-%d) \
            --config configs/production.yaml

      - name: Evaluate model
        run: |
          python pipelines/evaluate.py \
            --model-path outputs/model_latest/ \
            --test-data s3://data-lake/test_set/ \
            --min-f1 0.82 \
            --max-latency-ms 100 \
            --check-slices gender,age_group,region

      - name: Register model
        if: success()
        run: |
          python pipelines/register_model.py \
            --model-path outputs/model_latest/ \
            --stage staging

Model Quality Gates

Quality gates are automated checks that a model must pass before promotion. Typical gates include: overall accuracy above a threshold, no slice of the population has accuracy below a minimum (fairness check), inference latency below the SLA, model size below a deployment limit, and no regression compared to the currently deployed model. If any gate fails, the model is rejected and the team is alerted.

The "no regression" gate is especially important. A new model might have higher overall accuracy but lower accuracy on a critical subgroup. The gate catches this before users are affected.

GitOps for ML

GitOps treats Git as the single source of truth for both code and infrastructure. Every change — code, config, model version, pipeline definition — goes through a Git pull request. Automated systems watch the repository and apply changes to the production environment. This gives you an audit trail (who changed what and when), easy rollbacks (revert a commit), and reproducibility (the state of production is fully described in Git).

For ML, GitOps means: model configurations live in version-controlled YAML files, pipeline definitions are in code (not configured through a UI), and model promotions happen through pull requests that trigger automated validation.

Model Registry

What Is a Model Registry

A model registry is a versioned store for trained models — think of it as Git for models. Every trained model is stored as a versioned artifact with metadata: what data it was trained on, what hyperparameters were used, what metrics it achieved, who trained it, and when. Models progress through stages: development → staging → production → archived.

import mlflow

# Register a model after training
with mlflow.start_run():
    model = train_model(X_train, y_train)
    metrics = evaluate_model(model, X_test, y_test)

    mlflow.log_metrics(metrics)
    mlflow.sklearn.log_model(model, "model",
                              registered_model_name="fraud-detector")

# Promote a model to production (via API or UI)
client = mlflow.tracking.MlflowClient()
client.transition_model_version_stage(
    name="fraud-detector",
    version=5,
    stage="Production",
)

# Load the production model for serving
production_model = mlflow.sklearn.load_model(
    "models:/fraud-detector/Production"
)

Model Lineage and Reproducibility

Model lineage tracks the full provenance of a model: which data version was used, which code commit, which training configuration, which parent experiment. If a model in production starts making bad predictions, lineage lets you trace back to exactly what produced it. This is not just a nice-to-have — in regulated industries (finance, healthcare), auditors require this traceability.

Model Cards

Model cards are structured documentation for models — a standardized format introduced by Google that covers: what the model does, what data it was trained on, how it performs across different subgroups, its intended use cases, and its known limitations. Think of them as nutrition labels for models.

# Model card as structured metadata
model_card = {
    "model_name": "fraud-detector-v5",
    "description": "Binary classifier for payment fraud detection",
    "intended_use": "Real-time scoring of e-commerce transactions",
    "training_data": {
        "source": "transaction_logs_2023",
        "size": "12M transactions",
        "date_range": "2023-01-01 to 2023-12-31",
        "fraud_rate": "0.3%",
    },
    "performance": {
        "overall_f1": 0.87,
        "overall_precision": 0.82,
        "overall_recall": 0.93,
        "slices": {
            "domestic": {"f1": 0.89, "n": 10_000_000},
            "international": {"f1": 0.78, "n": 2_000_000},
            "high_value_gt_500": {"f1": 0.84, "n": 500_000},
        }
    },
    "limitations": [
        "Lower performance on international transactions",
        "Not tested on cryptocurrency transactions",
        "May exhibit bias toward flagging transactions from new accounts",
    ],
    "ethical_considerations": [
        "False positives disproportionately affect international customers",
        "Model should not be sole decision-maker for account freezes",
    ],
}

Every model in your registry should have a model card. It takes 30 minutes to write and saves hours of confusion when someone asks, "What does this model do, and should we trust it?"

Pipeline Orchestration

Kubeflow Pipelines

Kubeflow Pipelines is an ML-specific pipeline orchestration platform that runs on Kubernetes. Unlike Airflow (which orchestrates arbitrary data pipelines), Kubeflow is designed specifically for ML workflows — it understands concepts like experiments, runs, model artifacts, and metrics natively.

Each step in a Kubeflow pipeline runs in its own container. This provides strong isolation (different steps can use different Python versions or GPU types) and reproducibility (the container defines the exact environment). The downside: containerization adds overhead, and debugging is harder than running a local Python script.

# Kubeflow pipeline definition
from kfp import dsl, compiler

@dsl.component(base_image="python:3.11", packages_to_install=["pandas", "scikit-learn"])
def preprocess_data(input_path: str, output_path: dsl.Output[dsl.Dataset]):
    """Clean and prepare training data."""
    import pandas as pd
    df = pd.read_parquet(input_path)
    df = df.dropna(subset=["target"])
    df = df[df["amount"] > 0]
    df.to_parquet(output_path.path)

@dsl.component(base_image="pytorch/pytorch:2.1.0-cuda12.1-cudnn8-runtime")
def train_model(
    dataset: dsl.Input[dsl.Dataset],
    model_artifact: dsl.Output[dsl.Model],
    learning_rate: float = 0.001,
    epochs: int = 50,
):
    """Train a PyTorch model."""
    import torch
    import pandas as pd

    df = pd.read_parquet(dataset.path)
    model = build_model()
    trained = train(model, df, lr=learning_rate, epochs=epochs)
    torch.save(trained.state_dict(), model_artifact.path)

@dsl.component(packages_to_install=["scikit-learn", "mlflow"])
def evaluate_and_register(
    model: dsl.Input[dsl.Model],
    test_data: str,
    min_f1: float = 0.80,
):
    """Evaluate model and register if it passes quality gates."""
    metrics = evaluate(model.path, test_data)
    if metrics["f1"] >= min_f1:
        register_to_mlflow(model.path, metrics)
        print(f"Model registered with F1={metrics['f1']:.3f}")
    else:
        raise ValueError(f"F1={metrics['f1']:.3f} below threshold {min_f1}")

@dsl.pipeline(name="fraud-detection-pipeline")
def fraud_pipeline(data_path: str = "s3://data/transactions/"):
    preprocess_task = preprocess_data(input_path=data_path)
    train_task = train_model(
        dataset=preprocess_task.outputs["output_path"],
        learning_rate=0.001,
        epochs=50,
    )
    evaluate_and_register(
        model=train_task.outputs["model_artifact"],
        test_data="s3://data/test_set/",
    )

compiler.Compiler().compile(fraud_pipeline, "pipeline.yaml")

Vertex AI Pipelines

Vertex AI Pipelines is Google Cloud's managed ML pipeline service. It is built on Kubeflow Pipelines under the hood but removes the operational burden of managing a Kubernetes cluster. You define pipelines using the same KFP SDK, submit them to Vertex AI, and Google handles scheduling, scaling, and monitoring. For teams on Google Cloud, it is the path of least resistance.

Metaflow and ZenML

Metaflow (from Netflix) takes a different philosophy. Instead of containerizing each step, it runs pipelines as regular Python code and handles data passing and versioning transparently. You decorate Python classes with @step and Metaflow manages execution, whether locally, on AWS Batch, or on Kubernetes. It is the most Pythonic orchestrator — your pipeline feels like normal code, not infrastructure configuration.

ZenML is a newer framework that aims to be infrastructure-agnostic. You write your pipeline once and deploy it to any orchestrator (Airflow, Kubeflow, Vertex AI, local). It provides a unified interface for experiment tracking, model registries, and feature stores, connecting to whatever tools you already use.

Infrastructure as Code

Why Infrastructure as Code for ML

ML infrastructure is complex: GPU clusters, model servers, feature stores, monitoring dashboards, data pipelines, networking, storage, and IAM permissions. If you set this up manually through a cloud console, it is unreproducible (nobody remembers every click), fragile (one wrong setting breaks everything), and un-auditable (no record of who changed what).

Infrastructure as Code (IaC) means defining all infrastructure in version-controlled configuration files. Terraform, Pulumi, and AWS CloudFormation are the standard tools. You describe what you want (a Kubernetes cluster with 4 GPU nodes, a Redis instance for the feature store, a PostgreSQL database for the model registry), and the tool creates it. Change the configuration and apply — the tool figures out what needs to change.

# Terraform: ML serving infrastructure
# main.tf

resource "google_container_cluster" "ml_cluster" {
  name     = "ml-serving-cluster"
  location = "us-central1"

  node_pool {
    name       = "gpu-pool"
    node_count = 3

    node_config {
      machine_type = "n1-standard-8"
      guest_accelerator {
        type  = "nvidia-tesla-t4"
        count = 1
      }
    }

    autoscaling {
      min_node_count = 1
      max_node_count = 10
    }
  }

  node_pool {
    name       = "cpu-pool"
    node_count = 2

    node_config {
      machine_type = "n1-standard-4"
    }
  }
}

resource "google_redis_instance" "feature_store" {
  name           = "ml-feature-store"
  tier           = "STANDARD_HA"
  memory_size_gb = 16
  region         = "us-central1"
}

resource "google_sql_database_instance" "mlflow_backend" {
  name             = "mlflow-metadata"
  database_version = "POSTGRES_15"
  region           = "us-central1"

  settings {
    tier = "db-custom-2-4096"
  }
}

Benefits for ML Teams

Reproducibility: Spin up an identical environment for staging, development, or disaster recovery. New team members can create a working ML platform from scratch by running terraform apply.

Version control: Every infrastructure change goes through a pull request. You can see who changed GPU quotas, when, and why. Roll back to a previous configuration if something breaks.

Cost visibility: IaC tools can estimate costs before applying changes. "This change adds 4 GPUs to the serving cluster. Estimated monthly cost increase: $2,400." No surprises on the cloud bill.

Environment parity: Staging and production use the same Terraform modules with different variable values. This eliminates the "works in staging, fails in production" class of bugs caused by environment differences.

IaC in Practice

Start with the most critical infrastructure: the serving cluster and the model registry database. Add the feature store, monitoring, and data pipelines as you grow. Do not try to codify everything at once — it is a migration, not a rewrite. Store Terraform state remotely (S3, GCS) with locking to prevent concurrent modifications. Use modules to encapsulate reusable components (a "model serving endpoint" module that creates the load balancer, auto-scaler, and health checks together). And always, always run terraform plan before terraform apply — review what will change before it changes.

The combination of CI/CD, model registry, pipeline orchestration, and IaC forms the backbone of a mature ML platform. None of these pieces are exciting on their own. Together, they are the difference between a team that deploys models once a quarter after weeks of manual work and a team that ships model updates daily with confidence.
