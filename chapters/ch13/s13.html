AI Engineering with Foundation Models

Chapter 13: ML Systems & Production 4 subtopics

TL;DR

Foundation models — large pre-trained models like GPT-4, Claude, and Llama — have created a new discipline: AI engineering. Instead of training models from scratch, you integrate, fine-tune, and orchestrate these powerful base models into applications. This section covers the full stack: API-based integration, RAG architectures, prompt engineering in production, fine-tuning with LoRA/QLoRA/RLHF, building reliable LLM applications with guardrails and evaluation frameworks, and the operational concerns of cost, quality control, and long document processing.

The Shift to AI Engineering

For years, ML engineering meant: collect data, train model, deploy model. Foundation models changed the game. Now you start with a model that already understands language, images, or code. Your job shifts from training to integration — how do you wrap this powerful but unpredictable model into a reliable application? This is AI engineering, and it requires a different mindset from traditional ML.

The key tension: foundation models are incredibly capable but non-deterministic. They hallucinate, ignore instructions, and cost real money per token. Building production applications means engineering around these limitations while preserving the capability.

LLM Integration Patterns

API-Based Model Access

The simplest integration pattern: call an LLM API. OpenAI, Anthropic, Google, and others expose models via REST APIs. You send a prompt, you get a completion. This is how 90% of LLM applications start.

from openai import OpenAI

client = OpenAI()

def classify_support_ticket(ticket_text: str) -> dict:
    """Classify a support ticket using an LLM."""
    response = client.chat.completions.create(
        model="gpt-4o-mini",
        temperature=0,  # Deterministic for classification
        response_format={"type": "json_object"},
        messages=[
            {"role": "system", "content": """You are a support ticket classifier.
Classify each ticket into exactly one category: billing, technical, account, other.
Also assess priority: low, medium, high, urgent.
Respond in JSON: {"category": "...", "priority": "...", "reasoning": "..."}"""},
            {"role": "user", "content": ticket_text}
        ],
    )
    return json.loads(response.choices[0].message.content)
API reliability matters: LLM APIs have outages, rate limits, and latency spikes. Production code needs retries with exponential backoff, timeout handling, fallback models (try GPT-4o, fall back to GPT-4o-mini), and circuit breakers. Treat LLM APIs with the same care you'd give any external dependency.

Prompt Engineering in Production

Prompt engineering in production is different from prompt engineering in a notebook. In production, prompts are code — they need version control, testing, and monitoring. Structure your prompts as templates with clear separation between system instructions, few-shot examples, and user input. Test prompts against a regression suite of inputs before deploying changes.

Retrieval-Augmented Generation (RAG)

Retrieval-Augmented Generation (RAG) solves the knowledge problem. LLMs have a training cutoff and don't know about your private data. RAG retrieves relevant documents from your knowledge base and injects them into the prompt as context. The LLM then generates answers grounded in your actual documents, dramatically reducing hallucination.

from langchain.vectorstores import Chroma
from langchain.embeddings import OpenAIEmbeddings
from langchain.text_splitter import RecursiveCharacterTextSplitter

# 1. Chunk your documents
splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000, chunk_overlap=200,
    separators=["\n\n", "\n", ". ", " "]
)
chunks = splitter.split_documents(documents)

# 2. Embed and store in a vector database
vectorstore = Chroma.from_documents(
    chunks,
    embedding=OpenAIEmbeddings(model="text-embedding-3-small"),
    persist_directory="./chroma_db"
)

# 3. Retrieve relevant chunks for a query
retriever = vectorstore.as_retriever(
    search_type="mmr",        # Maximal Marginal Relevance (diverse results)
    search_kwargs={"k": 5}    # Top 5 chunks
)

# 4. Generate answer with retrieved context
def answer_question(question: str) -> str:
    relevant_docs = retriever.invoke(question)
    context = "\n\n".join([doc.page_content for doc in relevant_docs])
    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[
            {"role": "system", "content": f"""Answer based ONLY on this context:
{context}
If the context doesn't contain the answer, say "I don't have that information." """},
            {"role": "user", "content": question}
        ]
    )
    return response.choices[0].message.content

Function Calling and Tool Use

Function calling lets LLMs invoke structured tools — query a database, call an API, execute code. The model decides which function to call and with what parameters, based on the user's request. This bridges the gap between natural language and deterministic software systems.

tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "Get current weather for a location",
        "parameters": {
            "type": "object",
            "properties": {
                "location": {"type": "string", "description": "City name"},
                "unit": {"type": "string", "enum": ["celsius", "fahrenheit"]}
            },
            "required": ["location"]
        }
    }
}]

response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "What's the weather in Tokyo?"}],
    tools=tools,
    tool_choice="auto"
)

# Model returns a tool call with structured arguments:
# {"name": "get_weather", "arguments": {"location": "Tokyo", "unit": "celsius"}}

Agents and Orchestration

Agents are LLMs that iteratively decide what action to take, execute it, observe the result, and plan the next step. They combine reasoning with tool use in a loop. LangChain and LlamaIndex are the dominant orchestration frameworks. LangChain provides abstractions for chains (sequential operations), agents (autonomous tool-using loops), and retrieval pipelines. LlamaIndex specializes in data ingestion and querying, with particularly strong support for RAG architectures.

Fine-Tuning Foundation Models

When to Fine-Tune vs. Prompt

Fine-tuning is not always the answer. If your task can be solved with well-crafted prompts and few-shot examples, don't fine-tune. Fine-tune when: you need consistent output formatting, the task requires domain-specific knowledge the model lacks, latency/cost matter (fine-tuned small models beat prompted large models), or prompt engineering has hit a quality ceiling.

Full Fine-Tuning

Full fine-tuning updates every parameter in the model. For a 7B model, that's 7 billion gradient updates per step. It gives the most flexibility but requires the most compute and risks catastrophic forgetting — the model "forgets" its general capabilities while specializing. Full fine-tuning is typically reserved for cases where you have abundant domain data and need maximum performance.

Parameter-Efficient Fine-Tuning: LoRA and QLoRA

LoRA (Low-Rank Adaptation) is the breakthrough that made fine-tuning practical for most teams. Instead of updating all 7B parameters, LoRA freezes the original weights and adds tiny trainable "adapter" matrices to each layer. These adapters have low rank (typically 8-64), meaning you train only 0.1-1% of the model's parameters. The result: fine-tuning a 7B model on a single GPU in a few hours.

from peft import LoraConfig, get_peft_model, TaskType
from transformers import AutoModelForCausalLM, AutoTokenizer

model = AutoModelForCausalLM.from_pretrained("meta-llama/Llama-3-8B")
tokenizer = AutoTokenizer.from_pretrained("meta-llama/Llama-3-8B")

# Configure LoRA: small rank, applied to attention layers
lora_config = LoraConfig(
    task_type=TaskType.CAUSAL_LM,
    r=16,                    # Rank of the adapter matrices
    lora_alpha=32,           # Scaling factor
    lora_dropout=0.05,
    target_modules=["q_proj", "v_proj", "k_proj", "o_proj"],
)

model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
# trainable params: 13,631,488 (0.17% of 8,030,261,248)

QLoRA combines LoRA with quantization. It loads the base model in 4-bit precision (using NF4 quantization), then trains LoRA adapters in BF16. This lets you fine-tune a 70B model on a single 48GB GPU — something that would otherwise require a cluster. QLoRA's key insight: the base weights can be heavily quantized because they're frozen; only the small adapters need full precision.

Instruction Tuning

Instruction tuning fine-tunes a base model on instruction-response pairs to make it follow instructions. This is what turns a raw language model (which only completes text) into a helpful assistant. Datasets like Alpaca, FLAN, and OpenHermes provide diverse instruction-response pairs. The format typically wraps each example in a chat template.

RLHF and DPO: Aligning Models with Human Preferences

Reinforcement Learning from Human Feedback (RLHF) is the technique that turned GPT-3 into ChatGPT. It's a three-stage pipeline:

Stage 1: Supervised Fine-Tuning (SFT). Fine-tune the base model on high-quality demonstration data. This teaches it the basic format and behavior. Think of it as showing the model what good responses look like.

Stage 2: Reward Model Training. Collect human preferences — show annotators two model responses and ask which is better. Train a reward model that predicts these preferences. This model learns to score responses the way humans would.

Stage 3: PPO Fine-Tuning. Use Proximal Policy Optimization to fine-tune the SFT model to maximize the reward model's score. A critical addition: the KL penalty prevents the model from drifting too far from the SFT model, which would lead to reward hacking — generating responses that game the reward model without actually being better.

# Conceptual RLHF training loop
from trl import PPOTrainer, PPOConfig

ppo_config = PPOConfig(
    learning_rate=1e-5,
    batch_size=16,
    kl_penalty="kl",   # KL divergence penalty
    init_kl_coef=0.2,  # How strongly to penalize divergence
)

ppo_trainer = PPOTrainer(
    model=sft_model,
    ref_model=sft_model_frozen,  # Reference model for KL computation
    config=ppo_config,
    tokenizer=tokenizer,
)

for batch in dataloader:
    # Generate responses
    responses = ppo_trainer.generate(batch["input_ids"])
    # Score with reward model
    rewards = reward_model(batch["input_ids"], responses)
    # PPO update with KL penalty
    stats = ppo_trainer.step(batch["input_ids"], responses, rewards)
Reward hacking is real: If the reward model has blind spots, the policy will find and exploit them. Common failure: the model learns that longer responses get higher scores (annotators equate length with quality), so it generates unnecessarily verbose outputs. The KL penalty helps, but reward model quality is the fundamental constraint.

Direct Preference Optimization (DPO) is the simpler alternative to RLHF. It skips the reward model entirely. Given pairs of preferred and rejected responses, DPO directly optimizes the language model using a clever loss function derived from the RLHF objective. Same input data (preference pairs), fewer moving parts, no reward model to train, no PPO instability to debug. DPO has become the default choice for most fine-tuning practitioners.

Other alternatives include RLAIF (using AI feedback instead of human feedback), Constitutional AI (self-improvement through self-critique), and Rejection Sampling Fine-Tuning (generate many responses, keep the best, fine-tune on those).

LLM Application Architecture

Vector Store Integration

RAG applications need a vector store — a database optimized for similarity search over embedding vectors. Popular choices: Pinecone (managed, scalable), Weaviate (open-source, hybrid search), ChromaDB (lightweight, great for prototyping), pgvector (Postgres extension, use your existing database). The choice depends on your scale, latency requirements, and operational preferences.

Embedding Pipelines and Chunking Strategies

Before you can search your documents, you need to split them into chunks and embed each chunk. Chunking strategies dramatically affect retrieval quality. Fixed-size chunking (500 tokens) is simple but splits mid-paragraph. Recursive chunking respects document structure (split on paragraphs, then sentences). Semantic chunking uses embeddings to find natural breakpoints. Chunk overlap (10-20%) ensures context isn't lost at boundaries.

Guardrails and Output Validation

LLMs produce free-form text. Production applications need structured, validated output. Guardrails enforce constraints on model outputs: is the response valid JSON? Does it contain prohibited content? Is it within the expected length? Tools like Guardrails AI, NeMo Guardrails, and Instructor provide frameworks for output validation, re-prompting on failure, and content filtering.

from pydantic import BaseModel, Field
import instructor

client = instructor.from_openai(OpenAI())

class SupportResponse(BaseModel):
    """Structured output for support ticket response."""
    category: str = Field(description="One of: billing, technical, account")
    priority: int = Field(ge=1, le=5, description="1=lowest, 5=highest")
    response: str = Field(max_length=500)
    escalate: bool = Field(description="Whether to escalate to human agent")

# Instructor ensures the LLM output matches the Pydantic model
result = client.chat.completions.create(
    model="gpt-4o-mini",
    response_model=SupportResponse,
    messages=[{"role": "user", "content": "I've been double charged!"}],
    max_retries=3,  # Auto-retry if output doesn't validate
)
# result is a validated SupportResponse object, guaranteed

Caching and Cost Optimization

LLM API calls are expensive. Caching identical or similar queries saves money and reduces latency. Exact caching stores responses keyed by the exact prompt. Semantic caching embeds queries and returns cached responses for semantically similar queries. Tools like GPTCache implement this pattern.

Evaluation Frameworks

How do you know your LLM application works? Traditional accuracy metrics don't apply to open-ended generation. RAGAS evaluates RAG pipelines on faithfulness (does the answer match the retrieved context?), relevance (is the retrieved context relevant?), and answer correctness. LLM-as-Judge uses a strong model (GPT-4) to evaluate a weaker model's outputs on criteria you define. Both approaches have limitations but are better than vibes-based evaluation.

LLM Production Operations

Cost Optimization Deep Dive

Prompt caching (KV cache sharing across requests with the same prefix) can cut costs 50-90% for applications where many queries share a long system prompt. Model routing uses a lightweight classifier to route easy queries to a small, cheap model and hard queries to an expensive one — a model cascade. Semantic caching reuses responses for similar queries. Token budget management enforces limits per user, per request, and per billing period.

class ModelRouter:
    """Route queries to appropriate model based on complexity."""
    def __init__(self):
        self.cheap_model = "gpt-4o-mini"    # ~$0.15/1M input tokens
        self.expensive_model = "gpt-4o"      # ~$2.50/1M input tokens

    def route(self, query: str, context: dict) -> str:
        # Simple heuristic routing
        complexity_signals = [
            len(query) > 500,           # Long queries are often complex
            "explain" in query.lower(),  # Explanation requests need depth
            context.get("retries", 0) > 0,  # Retry → previous model failed
        ]
        if sum(complexity_signals) >= 2:
            return self.expensive_model
        return self.cheap_model  # Default to cheap

Output Quality Control

Structured generation constrains the model to output valid formats (JSON, XML, SQL). Regression testing for prompts maintains a test suite of inputs and expected outputs — run it before deploying prompt changes. Hallucination detection in production cross-references model claims against retrieved documents, flagging responses that assert facts not present in the context.

Long Document Processing

When documents exceed the model's context window, you need a strategy. Map-reduce processes each chunk independently (map), then combines the results (reduce). Hierarchical summarization summarizes chunks, then summarizes the summaries. Long-context models (128K+ tokens) can handle entire documents in one pass but at higher cost and with degraded attention to middle content (the "lost in the middle" problem). For most production use cases, well-implemented RAG with good chunking outperforms stuffing everything into a single long context.

The RAG vs. long-context tradeoff: Long-context models seem simpler (dump everything in), but they're more expensive, slower, and less precise. RAG retrieves only what's relevant, reducing noise and cost. However, RAG requires maintaining an embedding pipeline and vector store. The right choice depends on your document size, query frequency, and latency requirements. Many production systems use both: RAG for retrieval, then a long-context window for reasoning over the retrieved chunks.
