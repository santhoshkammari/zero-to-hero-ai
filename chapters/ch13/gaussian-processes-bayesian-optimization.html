Gaussian Processes give regression and classification with built-in uncertainty estimates — exact tool when data is scarce and every prediction needs error bars. Origin in geostatistics in 1960s under name kriging, after South African mining engineer Danie Krige. Carl Rasmussen and Christopher Williams’ 2006 textbook formalized them for machine learning. GPs also power Bayesian Optimization — most sample-efficient method we have for tuning expensive black-box functions.

The Weather Station Problem

Imagine meteorologist job. Three weather stations scattered across valley. Station A near lake reads 18°C. Station B on ridge reads 12°C. Station C in town center reads 20°C. Farmer calls: “What’s temperature at my field, 3 km east of Station A?”

Could fit straight line. Or parabola. Or wiggly polynomial through all three points. Each choice commits to specific shape, too early with only three data points. Better answer: “Given three readings and assumption that temperature changes smoothly over short distances, field is probably around 17°C — but confidence is low. Want more confidence? Give more stations.”

That’s what Gaussian Process does. No single curve. It considers all smooth curves passing near data and scores which look more plausible. Where data is dense, plausible curves agree and uncertainty stays low. Where data is sparse, curves fan out and uncertainty rises.

We use this valley and these weather stations as running example. Starts with three stations. By end, it teaches everything we need.

What “Distribution Over Functions” Actually Means

Phrase “distribution over functions” sounds dangerous. Here’s useful unpacking.

Start with single random variable. Gaussian distribution gives bell curve: some values more likely, some less. Now take two random variables — temperature at Station A and Station B. Multivariate Gaussian gives two-dimensional bell shape and encodes correlation. If Station A is warm, Station B is probably warm too. Same valley. Multivariate Gaussian captures that.

Now take three random variables — temperatures at A, B, and C. Same story in three dimensions. Multivariate Gaussian still works. Specify it with 3-element mean vector and 3×3 covariance matrix.

Now big leap. Imagine weather station at every point in valley — infinite locations. Then infinite random variables, one temperature per location. Need Gaussian distribution over all of them at once. That’s Gaussian Process. Infinite-dimensional extension of multivariate Gaussian. Pick any finite collection of locations, and temperatures there follow multivariate Gaussian distribution.

GP needs two pieces. Mean function m(x): expected temperature at location x before data (usually zero; let data speak). Kernel function k(x, x′): how correlated temperatures at two locations should be. Kernel holds interesting modeling decisions.

f(x) ~ GP(m(x), k(x, x'))

Pick any finite set of locations x₁, x₂, ..., xₙ:
[f(x₁), f(x₂), ..., f(xₙ)] ~ Normal(μ, K)

where μᵢ = m(xᵢ)  and  Kᵢⱼ = k(xᵢ, xⱼ)

Regular regression says “answer is this line.” GP says “answer sits somewhere in this bundle of curves, with probabilities.” Every curve in bundle is function. GP assigns probabilities across bundle. That’s distribution over functions.

First read felt like clever math trick, not practical tool. Wrong. It works. Reason: kernel.

The Kernel: Your Only Modeling Decision

In most machine learning, you choose architecture: layers, neurons, activation functions. In GP, choose kernel (covariance function). That’s core choice. Kernel encodes beliefs about function before data — smoothness, variation speed, repetition.

Kernel k(x, x′) takes two input locations, returns number for output correlation. If x and x′ are close and kernel returns large value, GP expects nearby points to have similar function values. Smooth functions. If kernel drops fast with distance, GP allows more wiggly behavior.

Back to valley: if temperature changes gradually over kilometers, use kernel where points a few hundred meters apart stay highly correlated, points 10 km apart are weakly correlated, and points 50 km apart are almost independent. That belief, written as function, is kernel.

The RBF (Squared Exponential) Kernel

Common starting point. Formula: k(x, x′) = σ² exp(−||x − x′||² / 2l²). Correlation between two points decays like Gaussian bell curve with distance. Parameter l is lengthscale — it controls how far apart two points can be and still stay correlated. Short l means wiggly functions. Long l means gentle sweeping curves.

Functions from RBF kernel are infinitely differentiable — smoother than real physical processes. Sounds good. Can hurt in practice. RBF assumes world is very smooth; real data often isn’t. RBF-based GPs can produce unrealistically confident predictions between data points because they refuse to wiggle.

The Matérn Kernel

Matérn family fixes over-smoothness by adding parameter ν, which controls how many times function is differentiable. Matérn-3/2 (ν = 3/2) gives once-differentiable functions — smooth, but corners allowed. Matérn-5/2 (ν = 5/2) gives twice-differentiable functions. As ν goes to infinity, Matérn converges to RBF.

In practice, Matérn-5/2 is default recommendation for most applications. Smooth enough for nice predictions. Rough enough to stay realistic. Remember one thing: start with Matérn-5/2, not RBF.

The Periodic Kernel

Some functions repeat. Daily temperature cycles. Weekly sales patterns. Heart rhythms. Periodic kernel has form k(x, x′) = σ² exp(−2 sin²(π|x − x′|/p) / l²), where p is period. Two points exactly one period apart act like neighbors.

Now fun part: kernels combine. Add periodic kernel to Matérn kernel, get function with repeating seasonal pattern plus long-term trend. Multiply periodic kernel by RBF, get pattern that repeats but whose amplitude changes over time. Kernel composition lets GPs express complex structural beliefs — far more interpretable than stacking neural network layers.

Every kernel has hyperparameters: lengthscale, variance, period. Learn them from data by maximizing marginal likelihood — probability of observed data given kernel and hyperparameters, integrated over all possible functions. Elegant part of GP framework: model balances fit and complexity automatically. No separate validation set needed.

This elegance costs. We hit cost soon.

GP Regression, Step by Step

Back to weather valley. Walk through exact GP prediction steps. No hand-waving.

We have three stations. Positions along valley (in km): x = [2, 5, 8]. Temperature readings: y = [18, 12, 20]. Choose Matérn-5/2 kernel with lengthscale l = 3 and variance σ² = 1, plus observation noise σ_n² = 0.1 for thermometer imprecision.

First build kernel matrix K. This 3×3 matrix has entry Kij = k(xi, xj). Diagonal entries are k(x, x) = σ² = 1; each station is perfectly correlated with itself. Off-diagonal entries measure correlation for each station pair — stations 2 km apart get high value, stations 6 km apart get lower value. Add noise: Knoisy = K + σ_n² I.

Now farmer asks about position x* = 5.5, point between Stations B and C. Compute k*, vector of correlations between x* and each training station. Station B at x = 5 is closest, so its entry in k* is largest.

Predictive mean — best temperature guess — comes from this formula:

μ* = k*ᵀ (K + σ²ₙ I)⁻¹ y

Read right to left. Take training temperatures y. Weight them by how “important” each station is; matrix inverse does that. Then combine those weights with how correlated x* is with each station. Stations close to x* and carrying non-redundant information get most weight. Result: single number, predicted temperature.

Predictive variance — uncertainty — comes from this:

σ²* = k** − k*ᵀ (K + σ²ₙ I)⁻¹ k*

Start with prior variance k** — how uncertain we were before data. Then subtract variance reduction from training data. Near station, reduction is large and uncertainty is small. Far from any station, reduction is small and uncertainty is large. That behavior makes GPs useful for decision-making: model tells you where it knows things and where it’s guessing.

Entire posterior — updated GP after data — is itself GP. That means we can predict at any number of new locations and get full multivariate Gaussian distribution over all of them. Predictions at nearby locations are correlated, so uncertainty band stays smooth and continuous, not jagged mess of independent error bars.

import numpy as np
from sklearn.gaussian_process import GaussianProcessRegressor
from sklearn.gaussian_process.kernels import Matern, WhiteKernel

# Our three weather stations
X_train = np.array([[2], [5], [8]])
y_train = np.array([18, 12, 20])

# Matérn-5/2 kernel + noise term
kernel = Matern(nu=2.5, length_scale=3.0) + WhiteKernel(noise_level=0.1)
gp = GaussianProcessRegressor(kernel=kernel, n_restarts_optimizer=5)
gp.fit(X_train, y_train)

# Predict across the valley
X_test = np.linspace(0, 10, 200).reshape(-1, 1)
y_pred, y_std = gp.predict(X_test, return_std=True)

# y_pred[i]  = predicted temperature at X_test[i]
# y_std[i]   = uncertainty (one standard deviation)
# Near stations: y_std is small.  Far from stations: y_std grows.

That code is whole thing. Fit three points. Get predictions with uncertainty everywhere. No architecture selection, no training epochs, no learning rate. Kernel is model.

GP Classification

So far we predict continuous temperatures. What if weather stations recorded something binary — whether it rained (1) or not (0) at each location? We still want probability of rain at farmer’s field, with uncertainty.

Problem: Gaussian distribution lives on real numbers, from −∞ to +∞. Probabilities live between 0 and 1. Need bridge.

GP classification introduces latent function f(x) with GP prior — it can take any real value. Then push f(x) through squashing function (sigmoid or probit, Gaussian CDF) to get probability: p(rain | x) = σ(f(x)). If latent function is large and positive at location x, rain probability is high. If it’s large and negative, probability is low. If it’s near zero, uncertainty stays high.

Catch is real. In GP regression, Gaussian likelihood works nicely with Gaussian prior — posterior is available in closed form. In classification, likelihood is Bernoulli, not Gaussian, and posterior is no longer tractable. Need approximation.

Two main approximation strategies: Laplace approximation (find posterior mode, fit Gaussian around it — fast but sometimes inaccurate) and Expectation Propagation (iteratively refine Gaussian approximation to each likelihood term — slower but more accurate). Variational inference is third option and grows popular because it scales better.

My intuition here is still forming. In practice, Laplace approximation works fine for well-separated classes. If decision boundary is subtle, Expectation Propagation often gives better calibrated uncertainties.

The O(n³) Problem and Sparse GPs

Here elegance of GPs hits reality. Prediction formula has (K + σ²I)&supmin;¹ in it — inverse of n×n matrix, where n is number of training points. Matrix inversion is O(n³) in time and O(n²) in memory. For three weather stations, trivial. For 1,000 stations, few seconds. For 10,000 stations, minutes. For 100,000, impractical on most hardware.

Concrete scale: if CPU can do about 10 billion floating-point operations per second, inverting 10,000×10,000 matrix takes around 100 seconds. 100,000×100,000 matrix? About 100,000 seconds — more than a day. Cubic scaling is brutal.

Main fix is sparse GPs with inducing points. Idea: do not use all n training points to define GP posterior. Pick small set of m points (inducing points or pseudo-inputs, where m is much smaller than n) and approximate full GP with those points. Think of m “summary stations” in valley that capture essential behavior of all n stations.

Inducing points do not have to be actual data points. Their locations and associated function values are learned during optimization, usually by maximizing variational lower bound on marginal likelihood (formalized by Titsias in 2009). Cost drops from O(n³) to O(nm²) — huge improvement when m is 500 and n is 100,000.

Second approach is Random Fourier Features. Insight: certain kernels (RBF, Matérn) can be approximated by random trigonometric features. Once you compute these features, GP reduces to Bayesian linear regression, which is O(nd²) where d is number of features. Trade kernel fidelity for speed.

In practice, GPyTorch (built on PyTorch) and GPflow (built on TensorFlow) make sparse GPs and GPU-accelerated inference accessible. GPyTorch especially pushed GP inference to datasets with hundreds of thousands of points using inducing points, conjugate gradients, and GPU parallelism. Not magic — approximation loses some fidelity, especially in uncertainty estimates — but for most practical work, it works well.

First time I saw sparse GP fit 50,000 points in under minute and produce sensible uncertainty bands, surprise hit. Approximation loss was much smaller than expected. But edge cases exist — multi-modal functions, data with sharp transitions — where inducing point approximation can miss structure. Good engineering, not free lunch.

Bayesian Optimization: Searching Smartly

Here’s where GPs earn their keep in modern ML practice. Not as a standalone regression model (neural networks have largely eaten that market), but as the engine inside Bayesian Optimization.

The problem: you have a function that’s expensive to evaluate. Training a deep neural network with a specific set of hyperparameters takes 4 hours. Running a physics simulation takes 6 hours. Synthesizing a candidate molecule in a wet lab takes a week. You want to find the input that maximizes (or minimizes) this function, and you can’t afford 10,000 evaluations. Maybe you can afford 50.

Grid search divides the space into a regular grid and evaluates every point. For 5 hyperparameters with 10 values each, that’s 100,000 evaluations — at 4 hours each, roughly 45 years. Random search is better (it doesn’t waste evaluations on unimportant dimensions), but it doesn’t learn from its past evaluations. Each new trial is as uninformed as the first.

Bayesian Optimization learns. It maintains a surrogate model — a cheap-to-evaluate approximation of the expensive function — and uses it to decide where to evaluate next. The surrogate is typically a GP, because GPs provide exactly what we need: a prediction of the function value at any point, plus an uncertainty estimate that tells us how much we might learn by evaluating there.

The loop looks like this. We start with a few initial evaluations (often random). We fit a GP to those evaluations. We then ask: “given what I know so far, where should I evaluate next to make the most progress?” That question is answered by an acquisition function, which we maximize over the cheap GP surrogate instead of the expensive real function. We evaluate the real function at the point the acquisition function recommends, add the result to our data, update the GP, and repeat.

Back to our valley analogy: imagine you’re drilling for water, and each bore hole costs $10,000. After three holes (our weather stations), the GP tells you both where water is likely (high predicted value) and where you have the most uncertainty (high variance). The acquisition function combines these signals to pick the fourth drilling location. After 20 strategic holes, you’ve found the water table’s peak — something that random drilling might have needed 200 holes to achieve.

Acquisition Functions: Where to Look Next

The acquisition function is the decision-making engine of Bayesian Optimization. It takes the GP’s posterior (mean prediction μ(x) and uncertainty σ(x) at every point) and produces a single score for each candidate location. The location with the highest score is where we evaluate next.

Every acquisition function faces the same tension: exploitation (evaluate where the GP predicts good values) versus exploration (evaluate where the GP is uncertain, because we might discover something better). Lean too far toward exploitation and you get stuck in a local optimum. Lean too far toward exploration and you waste evaluations on irrelevant regions.

Expected Improvement (EI)

The most popular acquisition function, and a good default. EI asks: “If I evaluate at point x, how much improvement over my current best result do I expect?” The expectation is taken over the GP’s posterior distribution at x.

Consider our drilling analogy. Say our best hole so far hit water at depth 15 meters. At a new candidate location, the GP predicts a mean of 13 meters with standard deviation 5 meters. There’s a reasonable probability the true depth exceeds 15 — the bell curve extends well past it. EI computes the average of all the ways this point could beat 15, weighted by their probabilities. If the mean is below 15 but the uncertainty is large, there’s still significant expected improvement — we might get lucky. That’s how EI naturally balances exploitation and exploration.

EI has a closed-form solution (involving the Gaussian PDF and CDF), which makes it fast to compute. In practice, I’ve found it works well in 80% of tuning problems without any fiddling.

Upper Confidence Bound (UCB)

UCB takes a more direct approach: UCB(x) = μ(x) + κσ(x). The predicted mean plus a multiple of the uncertainty. The parameter κ is a knob you can turn. Set κ high and the acquisition function strongly favors uncertain regions (exploration). Set κ low and it favors regions where the mean is already high (exploitation).

I like UCB because the exploration-exploitation tradeoff is explicit and tunable. If you’re early in the search and want to explore broadly, crank κ up to 2 or 3. If you’re running out of budget and want to refine around the best region, drop it to 0.5. EI doesn’t give you that dial.

Probability of Improvement (PI)

PI answers the question: “What is the probability that this point beats my current best?” It’s the simplest of the three, but it has a well-known flaw: it’s too greedy. PI treats a point that beats the current best by 0.001 the same as a point that beats it by 10. It chases high-probability, low-magnitude improvements and tends to get stuck near the current best. In my experience, it’s rarely the right choice for problems where the budget matters.

For most hyperparameter tuning: start with EI. If you suspect multiple good regions in the search space, switch to UCB with a moderate κ. PI exists for theoretical completeness, but I’d reach for it last.

BO for Hyperparameter Tuning: A Worked Example

Let’s see this in practice. We’ll tune an XGBoost classifier with Bayesian Optimization using Optuna. Optuna’s default surrogate is a Tree-structured Parzen Estimator (TPE), not a GP — TPE handles mixed continuous/categorical spaces better than GPs do, which is why it’s the practical default. The core idea is the same though: a surrogate model guides the search.

import optuna
from sklearn.datasets import make_classification
from sklearn.model_selection import cross_val_score
from xgboost import XGBClassifier

X, y = make_classification(n_samples=1000, n_features=20, random_state=42)

def objective(trial):
    params = {
        'n_estimators': trial.suggest_int('n_estimators', 50, 500),
        'max_depth': trial.suggest_int('max_depth', 3, 10),
        'learning_rate': trial.suggest_float('learning_rate', 0.01, 0.3, log=True),
        'subsample': trial.suggest_float('subsample', 0.5, 1.0),
        'colsample_bytree': trial.suggest_float('colsample_bytree', 0.5, 1.0),
    }
    clf = XGBClassifier(**params, eval_metric='logloss', random_state=42)
    return cross_val_score(clf, X, y, cv=5, scoring='accuracy').mean()

study = optuna.create_study(direction='maximize')
study.optimize(objective, n_trials=50)

Each of those 50 trials is informed by all previous trials. Optuna builds a surrogate, decides which hyperparameter combination is most promising, evaluates it, and updates. Fifty intelligent trials often outperform 500 random ones. That’s the power of Bayesian Optimization — not faster individual evaluations, but far fewer wasted evaluations.

For GP-based BO specifically (say, when your search space is purely continuous), BoTorch (Meta’s library built on GPyTorch) gives you full control over the GP surrogate and acquisition function. It’s more complex to set up, but it’s the tool of choice for research and custom optimization problems.

Multi-Fidelity Bayesian Optimization

There’s one more problem worth confronting. Even with Bayesian Optimization choosing evaluation points intelligently, each evaluation might still take hours. Training a neural network for 100 epochs takes a long time. But training it for 5 epochs? That’s fast, and it gives you a rough estimate of how well those hyperparameters will perform.

Multi-fidelity Bayesian Optimization exploits this idea: instead of always evaluating the full expensive function, use cheap low-fidelity approximations (fewer epochs, less data, lower resolution) to weed out bad candidates quickly, and reserve expensive high-fidelity evaluations for the most promising ones.

The most influential algorithm in this space is Hyperband, which runs multiple rounds of Successive Halving. Start with many hyperparameter configurations, each given a tiny resource budget (say, 1 epoch). Evaluate all of them. Throw away the worst half. Double the budget for the survivors. Repeat until one remains. Hyperband runs several of these races with different starting configurations and budget allocations, automatically balancing between evaluating many configurations cheaply and few configurations expensively.

BOHB (Bayesian Optimization + HyperBand) combines Hyperband’s resource allocation with a TPE surrogate for choosing which configurations to try. Instead of picking configurations randomly (as vanilla Hyperband does), BOHB uses the surrogate model to propose configurations that are more likely to be good. It gets the sample efficiency of Bayesian Optimization and the resource efficiency of early stopping.

This matters because it changes the economics of hyperparameter tuning. Without multi-fidelity, a 50-trial BO search where each trial takes 4 hours costs 200 hours. With BOHB, most configurations are killed after a few minutes, and only the top candidates get the full 4-hour treatment. The total cost might drop to 20–30 hours for the same quality result.

My favorite thing about multi-fidelity BO is that it matches how humans actually tune models. You train for a few epochs, glance at the loss curve, mutter “that’s terrible,” and kill it. BOHB automates that instinct with a principled framework underneath.
