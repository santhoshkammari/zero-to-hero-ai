Data Types and Sources

Chapter 3: Data Fundamentals 4 subtopics
TL;DR

Every ML model is only as good as the data it learns from. This section walks through the kinds of data you'll encounter — structured tables, raw text, images, time series — where that data comes from, how to load it into Python, and which file formats actually matter at scale. If you remember one thing: pick the right format early, or pay for it later.

Types of Data

Structured vs Unstructured vs Semi-Structured

Garbage in, garbage out — so where does good data come from? Before you can answer that, you need to know what kinds of data exist. The first big distinction is how organized the data is.

Structured data lives in rows and columns. Think spreadsheets, SQL tables, anything where every record has the same fields in the same order. A CSV of customer transactions is structured. You know exactly what column 3 means. Machines love this stuff because it's predictable.

Unstructured data has no predefined schema. A folder of photographs, a pile of PDF contracts, a dump of tweets — there's no column header telling you what's inside. Most of the world's data is unstructured, which is exactly why deep learning became so important. Before neural networks got good, unstructured data was almost unusable for modeling.

Semi-structured data sits in between. JSON files, XML documents, and HTML pages have some organization — tags, keys, nesting — but no rigid table schema. A JSON API response might have a "user" object with a "name" field, but some records might include "middle_name" and others won't. You get flexibility at the cost of messiness.

Numerical Data

Numbers split into two camps. Continuous data can take any value within a range — temperature, salary, the probability a user clicks a button. You can always squeeze another decimal place in. Discrete data comes in countable chunks — number of children, number of purchases, number of retries before a server gives up. The distinction matters because different models handle them differently. Linear regression expects continuous targets. A Poisson model expects discrete counts.

Categorical Data

Categorical data represents groups or labels. It comes in two flavors. Nominal data has no natural order: colors (red, green, blue), country codes, or blood types. Ordinal data has a meaningful ranking: education level (high school < bachelor's < master's < PhD) or customer satisfaction (poor < fair < good < excellent). The ranking matters, but the distance between ranks isn't necessarily equal. The gap between "poor" and "fair" might not be the same as between "good" and "excellent." Many beginners encode ordinal data as plain integers (1, 2, 3, 4) and accidentally tell the model that the gaps are uniform. Be careful.

Text, Time Series, and Image Data

Text data is a sequence of characters or tokens. Emails, product reviews, medical notes. It's technically unstructured, but NLP techniques can extract structure from it — sentiment scores, named entities, embeddings. The challenge with text is that the same idea can be expressed a thousand different ways.

Time series data is any sequence of measurements indexed by time. Stock prices sampled every minute, daily temperature readings, server CPU usage every 5 seconds. Order matters here. Shuffling the rows destroys the signal. Models for time series — ARIMA, LSTMs, Transformers — all exploit the temporal structure that plain tabular models ignore.

Image data is a grid of pixel values. A grayscale image is a 2D matrix of intensities. A color image is three stacked matrices (red, green, blue channels). A 224×224 RGB image is really a tensor of shape (224, 224, 3) — that's 150,528 numbers. Convolutional neural networks became dominant precisely because they exploit the spatial structure in this grid: nearby pixels are more related than distant ones.

Tabular data is the workaround term for "data that fits in a table." Each row is an observation, each column is a feature. It's the bread and butter of data science — and despite all the hype around deep learning, gradient-boosted trees (XGBoost, LightGBM) still beat neural networks on most tabular tasks. Don't reach for a Transformer when a decision tree will do.

Data Sources

CSV, TSV, JSON, and XML

CSV (Comma-Separated Values) is the universal lowest-common-denominator format. Every tool reads it. Every tool writes it. It's human-readable. It's also painfully slow for large datasets, stores no type information (is "42" a string or an integer?), and breaks the moment someone puts a comma inside a field without proper quoting. TSV (Tab-Separated Values) is the same idea but uses tabs — slightly less fragile since tabs appear less often in real data.

JSON (JavaScript Object Notation) handles nested, semi-structured data well. API responses almost always come back as JSON. It's readable, flexible, and universally supported. The downside: it's verbose, slow to parse at scale, and terrible for columnar access. If you need the "price" field from a million JSON records, you still have to parse every single record. XML is JSON's older, more verbose cousin. You'll encounter it in legacy enterprise systems and SOAP APIs. Avoid it when you have a choice.

Databases

SQL databases (PostgreSQL, MySQL, SQLite) store structured data with enforced schemas, ACID transactions, and powerful query languages. If your data has clear relationships — users have orders, orders have items — SQL is the right home. NoSQL databases (MongoDB, Cassandra, Redis) trade rigid schemas for flexibility and horizontal scalability. MongoDB stores documents (basically JSON), Cassandra handles massive write-heavy workloads, and Redis keeps data in memory for sub-millisecond reads. The honest tradeoff: SQL gives you correctness guarantees and decades of tooling. NoSQL gives you flexibility and scale. Most ML pipelines read from SQL databases because the data was already cleaned and structured by application engineers.

APIs, Web Scraping, and Public Datasets

Many data sources are accessed through APIs — structured HTTP endpoints that return JSON. Twitter's API, weather services, financial data providers. Web scraping extracts data from HTML pages when no API exists. It works, but it's brittle: one website redesign and your scraper breaks. Always check the terms of service before scraping.

For learning and prototyping, public datasets are gold. Kaggle hosts thousands of competition datasets with community notebooks. The UCI Machine Learning Repository has classic benchmark datasets. Government open-data portals (data.gov, EU Open Data) provide real-world economic, health, and environmental data. When in doubt, start with a public dataset. You'll learn the techniques without the headache of data collection.

Data Warehouses, Data Lakes, and Streaming

In production, data often lives in a data warehouse — a centralized store optimized for analytical queries (Snowflake, BigQuery, Redshift). Warehouses store clean, structured data. A data lake is the wilder counterpart: a massive store (usually on S3 or HDFS) that holds raw data in any format — Parquet files, JSON logs, images, whatever. Warehouses are for analysts. Lakes are for engineers who need everything in one place and will clean it later.

Streaming data sources (Kafka, Kinesis, Pub/Sub) deliver records in real time. Instead of querying a database for yesterday's data, you consume a continuous stream of events — clicks, sensor readings, transactions. ML models that serve real-time predictions often need streaming pipelines to stay current.

Synthetic data generation creates fake but statistically plausible data. Useful when real data is scarce, expensive, or privacy-sensitive. Libraries like Faker generate realistic names and addresses. GANs can generate synthetic images. The risk: synthetic data only captures patterns you already know about. It can't surprise you with edge cases from the real world.

Loading Data

Loading CSV with Pandas

Pandas is the default tool for loading tabular data in Python. For a CSV, it's one line:

import pandas as pd

df = pd.read_csv("customers.csv")
print(df.shape)       # (rows, columns)
print(df.dtypes)      # inferred types for each column
print(df.head())      # first 5 rows

Pandas guesses column types automatically. It usually gets it right, but not always. A column of zip codes like "02134" gets read as the integer 2134 — the leading zero vanishes. Always verify dtypes after loading.

Loading JSON Data

For JSON, Pandas handles flat and nested structures:

# Flat JSON array
df = pd.read_json("events.json")

# Nested JSON — normalize to a flat table
import json
with open("nested.json") as f:
    data = json.load(f)
df = pd.json_normalize(data, record_path="items", meta=["order_id", "customer"])

The json_normalize function flattens nested records into columns. You specify which nested list to explode (record_path) and which parent fields to carry along (meta). It saves hours of manual flattening.

Connecting to SQL Databases

For databases, use SQLAlchemy as the connection layer:

from sqlalchemy import create_engine

engine = create_engine("sqlite:///my_database.db")
df = pd.read_sql("SELECT * FROM orders WHERE total > 100", engine)

Push filtering into the SQL query. Don't load a million rows and then filter in Python — let the database do what it's good at. This is the single biggest performance mistake beginners make with database-backed ML pipelines.

Handling Large Files

When a CSV is too large to fit in memory, read it in chunks:

chunks = pd.read_csv("huge_file.csv", chunksize=50_000)

results = []
for chunk in chunks:
    filtered = chunk[chunk["status"] == "active"]
    results.append(filtered)

df = pd.concat(results, ignore_index=True)

Each chunk is a regular DataFrame. Process it, keep what you need, discard the rest. This way you never load the full file at once. For truly massive datasets (tens of gigabytes), consider Dask or Polars instead of Pandas — they handle out-of-core computation natively.

Data Type Inference and Explicit Casting

Pandas infers types on load, but inference can be wrong or wasteful. A column of integers with a single missing value becomes float64 (because NaN is a float in NumPy). A column of "yes"/"no" strings could be a boolean. Explicit casting fixes both problems:

# Force specific types on load
df = pd.read_csv("data.csv", dtype={"zip_code": str, "quantity": "Int32"})

# Cast after load
df["is_active"] = df["is_active"].map({"yes": True, "no": False})
df["category"] = df["category"].astype("category")  # saves memory

Using the category dtype for low-cardinality string columns (like country codes or product types) can cut memory usage by 90%. It's one of the most underused Pandas optimizations.

Columnar and Binary Data Formats

Apache Parquet

Apache Parquet is the format you should default to for any dataset larger than a few hundred megabytes. It stores data column-by-column instead of row-by-row. Why does that matter? If you only need 3 columns out of 200, Parquet reads just those 3 columns from disk. CSV reads every single byte of every row. Parquet also compresses aggressively — a 10 GB CSV often shrinks to 1-2 GB in Parquet — and preserves type information (no more guessing if "42" is a string).

# Write to Parquet
df.to_parquet("customers.parquet", engine="pyarrow")

# Read only the columns you need
df = pd.read_parquet("customers.parquet", columns=["name", "revenue"])

Apache Arrow and Feather

Apache Arrow defines an in-memory columnar format that avoids serialization overhead when passing data between systems (Python to R, Pandas to Spark). The Feather format is Arrow's on-disk sibling — extremely fast to read and write, zero-copy when possible. Use Feather for intermediate files in a pipeline where speed matters more than compression. Use Parquet for long-term storage and interchange.

# Feather — fastest read/write for intermediate data
df.to_feather("intermediate.feather")
df = pd.read_feather("intermediate.feather")

HDF5

HDF5 (Hierarchical Data Format) stores large numerical arrays efficiently. It's popular in scientific computing and was the default for Keras model weights for years. HDF5 supports partial reads (grab a slice of a 50 GB array without loading all of it) and internal compression. The downside: its Python library (h5py) has a clunkier API than Parquet tooling, and the format is less common in modern data engineering pipelines.

Format Selection Guidelines

Here's the honest decision framework. For quick exploration and sharing with non-technical people, CSV is fine — everyone can open it. For anything going into a data pipeline or ML training loop, switch to Parquet: it's smaller, faster, and type-safe. For intermediate results that stay within a single Python pipeline, Feather is the fastest option. For large multi-dimensional arrays in scientific work, HDF5 still has its place. And if someone suggests storing 50 GB of training data as JSON files, politely suggest they reconsider.

# Quick comparison: file size and read speed
import time, os

# Write the same DataFrame in different formats
df.to_csv("data.csv", index=False)
df.to_parquet("data.parquet")
df.to_feather("data.feather")

for fmt in ["data.csv", "data.parquet", "data.feather"]:
    start = time.time()
    if fmt.endswith(".csv"):     pd.read_csv(fmt)
    elif fmt.endswith(".parquet"): pd.read_parquet(fmt)
    else:                        pd.read_feather(fmt)
    elapsed = time.time() - start
    size_mb = os.path.getsize(fmt) / 1e6
    print(f"{fmt:20s}  {size_mb:6.1f} MB  {elapsed:.3f}s")

Run this on your own data. The numbers will make the argument better than any paragraph can. Parquet is typically 5–10× smaller than CSV and 3–5× faster to read. Feather is even faster but slightly larger. CSV loses on every axis except human readability.
