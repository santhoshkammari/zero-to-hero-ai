Data Quality and Validation

Chapter 3: Data Fundamentals 6 subtopics
TL;DR

Your model is only as good as the data you feed it. Data quality and validation is the discipline of catching bad data before it silently ruins your predictions. This section covers why data quality matters more than model architecture, how to use Great Expectations and Pandera to write declarative checks on your DataFrames, how to enforce schemas with JSON Schema, Avro, and Protobuf, how to detect data drift with statistical tests like KS and PSI, how to treat data tests as first-class citizens alongside your unit tests, and how to monitor data quality in production with alerts, dashboards, and SLAs. If you only remember one thing: a fancy model trained on garbage data will confidently produce garbage predictions.

Why Data Quality Matters More Than Model Complexity

Here's a pattern I've seen play out at least a dozen times. A team spends three months building an elaborate model — ensemble methods, custom architectures, hyperparameter sweeps. They deploy it. Accuracy craters within two weeks. The root cause is never the model. It's always the data. A column silently changed format upstream. A third-party feed started returning nulls. A categorical field grew three new values nobody told the ML team about.

There's a saying in this field: garbage in, garbage out. It sounds obvious, but it is routinely underestimated. Research from Google's famous "Data Cascades" paper showed that 92% of AI practitioners had experienced data quality issues that harmed downstream model performance, and most of these issues were preventable. The painful part is that bad data doesn't throw an error. Your pipeline runs. Your model trains. It just quietly gets worse.

Think about it from a cost perspective. Fixing a data issue at ingestion time costs almost nothing — a schema check, a type assertion, a range validation. Fixing the same issue after a model has been retrained on corrupted data and deployed to production? That costs incident response time, rollback effort, and lost user trust. The earlier you catch it, the cheaper it is. This is the exact same principle as catching bugs early in software development, but applied to data.

The uncomfortable truth is that improving your data quality by even a small margin will almost always beat switching to a fancier model. A logistic regression on clean, well-validated data will outperform a transformer on noisy, drifting data nine times out of ten. So let's talk about the tools that make data quality systematic instead of accidental.

Great Expectations

Great Expectations is a Python framework for validating, documenting, and profiling your data. The core idea is simple: you write declarative "expectations" about what your data should look like, and the framework checks whether reality matches. Think of it as unit tests, but for DataFrames.

Expectations, Suites, and Checkpoints

An expectation is a single assertion about your data. "This column should never be null." "Values in this column should be between 0 and 1." "This table should have between 1,000 and 100,000 rows." Each expectation is a method call that returns a success or failure result along with diagnostic information.

An expectation suite is a named collection of expectations — like a test class in pytest. You might have one suite for your raw ingestion data and a different, stricter suite for your feature-engineered training set. Suites are stored as JSON, so they're version-controllable and shareable.

A checkpoint ties everything together. It says: "Take this data source, validate it against this suite, and do something with the results." That "something" could be generating an HTML report, sending a Slack alert, or failing a pipeline step. Checkpoints are what you actually run in production.

Here's what this looks like in practice:

import great_expectations as gx

# Create a data context — the central config object
context = gx.get_context()

# Connect to your data
datasource = context.sources.add_pandas("my_datasource")
data_asset = datasource.add_dataframe_asset(name="training_data")

# Build a batch request from a DataFrame
import pandas as pd
df = pd.read_csv("features.csv")
batch_request = data_asset.build_batch_request(dataframe=df)

# Create an expectation suite
suite = context.add_expectation_suite("training_data_quality")

# Create a validator — this is what you write expectations against
validator = context.get_validator(
    batch_request=batch_request,
    expectation_suite_name="training_data_quality"
)

# Write expectations
validator.expect_column_to_exist("user_id")
validator.expect_column_values_to_not_be_null("user_id")
validator.expect_column_values_to_be_between("age", min_value=0, max_value=150)
validator.expect_column_values_to_be_in_set(
    "status", ["active", "inactive", "pending"]
)
validator.expect_table_row_count_to_be_between(min_value=1000, max_value=500000)

# Save the suite
validator.save_expectation_suite(discard_failed_expectations=False)

# Run validation and get results
results = validator.validate()
print(f"Success: {results.success}")
print(f"Statistics: {results.statistics}")

The beauty of this approach is that your expectations become living documentation. When a new team member asks "what does valid training data look like?", you point them at the suite. When an upstream change breaks your assumptions, the checkpoint catches it before your model ever sees the bad data.

Pandera

Pandera takes a different angle on the same problem. Instead of writing expectations as method calls, you define a schema as a Python class with type annotations. If you've used Pydantic for API validation, Pandera will feel instantly familiar — it's Pydantic for DataFrames.

The key advantage is that your schema definition lives right next to your code. It's concise, readable, and leverages Python's type system. You can also use it as a decorator to validate function inputs and outputs automatically.

import pandas as pd
import pandera as pa
from pandera.typing import DataFrame, Series

class TrainingDataSchema(pa.DataFrameModel):
    user_id: Series[int] = pa.Field(gt=0, nullable=False)
    age: Series[float] = pa.Field(ge=0, le=150, nullable=False)
    income: Series[float] = pa.Field(ge=0)
    status: Series[str] = pa.Field(isin=["active", "inactive", "pending"])
    signup_date: Series[pa.DateTime] = pa.Field(nullable=False)

    class Config:
        strict = True       # No extra columns allowed
        coerce = True       # Attempt type coercion before validation

# Validate explicitly
df = pd.read_csv("features.csv")
validated_df = TrainingDataSchema.validate(df)

# Or use as a function decorator — validation happens automatically
@pa.check_types
def build_features(raw: DataFrame[TrainingDataSchema]) -> DataFrame[TrainingDataSchema]:
    raw["age_bucket"] = pd.cut(raw["age"], bins=[0, 18, 35, 65, 150])
    return raw

Pandera also supports custom checks, hypothesis testing, and lazy validation (collect all errors instead of failing on the first one). For teams that prefer a code-first approach over configuration files, Pandera tends to be the better fit. Great Expectations shines more in data platform and orchestration contexts where you want rich HTML reports and checkpoint-based workflows.

Schema Validation for ML Data

DataFrames aren't the only data format in ML pipelines. You'll encounter JSON payloads from APIs, serialized messages on Kafka topics, and protocol buffer definitions for model serving. Each of these has its own schema validation story.

JSON Schema

JSON Schema lets you describe the shape of a JSON document — required fields, types, allowed values, nested structures. It's widely supported and language-agnostic. For ML, you'll commonly use it to validate inference request payloads or configuration files.

import jsonschema

feature_schema = {
    "type": "object",
    "required": ["user_id", "age", "income"],
    "properties": {
        "user_id": {"type": "integer", "minimum": 1},
        "age": {"type": "number", "minimum": 0, "maximum": 150},
        "income": {"type": "number", "minimum": 0},
        "status": {"type": "string", "enum": ["active", "inactive", "pending"]}
    },
    "additionalProperties": False
}

# Validate a single record
record = {"user_id": 42, "age": 29, "income": 55000.0, "status": "active"}
jsonschema.validate(instance=record, schema=feature_schema)  # Raises on failure

Avro and Protobuf

Apache Avro and Protocol Buffers (Protobuf) are binary serialization formats with built-in schema enforcement. Avro is popular in Kafka-based pipelines because the schema is stored separately in a schema registry, and producers and consumers negotiate compatibility automatically. Protobuf is Google's format and is commonly used in gRPC-based model serving (like TensorFlow Serving).

The key idea with both is schema evolution — the ability to change your data format over time without breaking existing consumers. You can add optional fields, deprecate old ones, and the serialization layer handles the mismatch. For ML pipelines that run for months or years, this forward and backward compatibility is essential. Without it, every schema change becomes a coordinated deployment across every service that touches the data.

Statistical Tests for Drift

Schema validation catches structural problems — wrong types, missing columns, out-of-range values. But what about statistical problems? Your data can be structurally perfect and still be quietly drifting in ways that degrade your model. Maybe the distribution of ages shifted because you launched in a new country. Maybe income values skewed higher because of a seasonal effect. The schema still passes. The model still runs. But predictions get worse.

Data drift is when the statistical properties of your input data change over time relative to the data your model was trained on. Detecting it requires comparing distributions, not just checking individual values.

The KS Test

The Kolmogorov-Smirnov (KS) test is a non-parametric test that compares two distributions. It measures the maximum distance between their cumulative distribution functions. A small p-value means the two distributions are statistically different. It works for any continuous feature and makes no assumptions about the shape of the distribution.

from scipy import stats
import numpy as np

# Training data distribution
train_ages = np.random.normal(loc=35, scale=10, size=10000)

# Production data — has it shifted?
prod_ages = np.random.normal(loc=38, scale=12, size=5000)

statistic, p_value = stats.ks_2samp(train_ages, prod_ages)
print(f"KS statistic: {statistic:.4f}, p-value: {p_value:.6f}")

if p_value < 0.05:
    print("ALERT: Significant distribution shift detected in 'age'")

Population Stability Index (PSI)

The Population Stability Index (PSI) is a metric borrowed from credit risk modeling that quantifies how much a distribution has shifted. Unlike the KS test, it gives you a single number with well-known thresholds: PSI below 0.1 means little change, 0.1 to 0.2 means moderate shift, and above 0.2 means significant drift that probably warrants investigation or retraining.

import numpy as np

def calculate_psi(expected, actual, bins=10):
    """Calculate Population Stability Index between two distributions."""
    # Bin the distributions using the same breakpoints
    breakpoints = np.linspace(
        min(expected.min(), actual.min()),
        max(expected.max(), actual.max()),
        bins + 1
    )
    expected_counts = np.histogram(expected, breakpoints)[0] / len(expected)
    actual_counts = np.histogram(actual, breakpoints)[0] / len(actual)

    # Avoid division by zero with a small epsilon
    expected_counts = np.clip(expected_counts, 1e-4, None)
    actual_counts = np.clip(actual_counts, 1e-4, None)

    psi = np.sum((actual_counts - expected_counts) *
                  np.log(actual_counts / expected_counts))
    return psi

train_incomes = np.random.lognormal(mean=10.5, sigma=0.8, size=10000)
prod_incomes = np.random.lognormal(mean=10.8, sigma=0.9, size=5000)

psi = calculate_psi(train_incomes, prod_incomes)
print(f"PSI: {psi:.4f}")

if psi > 0.2:
    print("ALERT: Major distribution shift — consider retraining")
elif psi > 0.1:
    print("WARNING: Moderate shift — monitor closely")

A practical approach is to compute PSI for every feature on a daily or weekly cadence, then alert when any feature crosses the 0.2 threshold. The KS test is useful for deeper investigation once an alert fires — it can tell you whether the shift is statistically significant or just noise from a small sample.

Data Unit Testing

You write unit tests for your code. Why not for your data? The idea is simple: treat data validation checks as first-class tests that run in your CI/CD pipeline alongside your code tests. If the data doesn't meet expectations, the pipeline fails, exactly like a broken unit test.

You already know pytest. Let's use it for data:

import pytest
import pandas as pd
from scipy import stats

@pytest.fixture
def training_data():
    return pd.read_csv("features.csv")

@pytest.fixture
def reference_data():
    return pd.read_csv("reference_features.csv")


class TestDataQuality:
    def test_no_nulls_in_required_columns(self, training_data):
        required = ["user_id", "age", "income", "status"]
        for col in required:
            null_count = training_data[col].isnull().sum()
            assert null_count == 0, f"{col} has {null_count} nulls"

    def test_age_range(self, training_data):
        assert training_data["age"].min() >= 0, "Negative ages found"
        assert training_data["age"].max() <= 150, "Unrealistic ages found"

    def test_no_duplicate_ids(self, training_data):
        dupes = training_data["user_id"].duplicated().sum()
        assert dupes == 0, f"Found {dupes} duplicate user_ids"

    def test_row_count_reasonable(self, training_data):
        n = len(training_data)
        assert 1000 <= n <= 500000, f"Row count {n} outside expected range"

    def test_age_distribution_stable(self, training_data, reference_data):
        _, p_value = stats.ks_2samp(
            reference_data["age"].dropna(),
            training_data["age"].dropna()
        )
        assert p_value > 0.01, (
            f"Age distribution has drifted significantly (p={p_value:.4f})"
        )

Run this with pytest test_data_quality.py -v and you get the same familiar output: green checks, red failures, clear error messages. The distribution test at the end is particularly powerful — it catches the subtle drifts that structural checks miss.

A few practical tips. Keep your data tests fast by sampling large datasets instead of scanning every row. Separate your structural tests (nulls, types, ranges) from your statistical tests (distribution shifts) because they fail for different reasons and need different remediation. And tag your tests with pytest markers so you can run quick checks on every commit and slower statistical checks on a nightly schedule.

Monitoring Data Quality in Production

Tests in CI catch problems at build time. But data keeps flowing after deployment. Production monitoring is about catching problems in real time — when a data source goes stale, when completeness drops, when distributions start drifting between retraining cycles.

What to Monitor

Data freshness answers the question: "When was this table last updated?" If your feature store hasn't been refreshed in 48 hours and your SLA says 24 hours, something is broken upstream. Track the timestamp of the most recent record and alert when the gap exceeds your threshold.

Data completeness measures the proportion of non-null values in critical columns. A column that's historically 99.8% complete suddenly dropping to 85% is a strong signal that something changed upstream. Track completeness as a time series and set alerts on percentage drops, not just absolute thresholds.

Schema conformance checks whether the actual data matches the expected schema. Did a new category appear in a categorical column? Did a column change from integer to float? These checks run on every batch or micro-batch.

Distribution stability is your PSI and KS tests running on a schedule. Compare every incoming batch against your training data reference. Log the metrics. Alert on thresholds.

Building the Monitoring Layer

You don't need to build all of this from scratch. Export your metrics to whatever observability stack your team already uses — Prometheus and Grafana, Datadog, CloudWatch, or even a simple database table with a dashboard on top. The key is making data quality metrics as visible as your application metrics. If your team has dashboards for API latency and error rates, data freshness and completeness should be on the same screen.

Define data SLAs — explicit, written agreements about quality expectations. For example: "The features table will be updated within 6 hours of the raw event data landing. Completeness for all required columns will be above 99%. No feature will have a PSI above 0.2 relative to the training reference." Write these down. Put them in your runbook. Alert on violations. Review them quarterly.

The hardest part of production data monitoring isn't the technology. It's the organizational discipline. Someone needs to own data quality. Alerts need to route to a specific team. Incidents need to be tracked and resolved. Without that ownership, even the best monitoring setup will have its alerts silenced within a month. Treat data quality incidents with the same seriousness as production outages — because for an ML system, they are production outages.
